README.cmake
1Experimental cmake-based build support for Apache httpd on Microsoft Windows
2
3Status
4------
5
6This build support is currently intended only for Microsoft Windows.
7
8This build support is experimental.  Specifically,
9
10* It does not support all features of Apache httpd.
11* Some components may not be built correctly and/or in a manner
12  compatible with the previous Windows build support.
13* Build interfaces, such as the mechanisms which are used to enable
14  optional functionality or specify prerequisites, may change from
15  release to release as feedback is received from users and bugs and
16  limitations are resolved.
17
18Important: Refer to the "Known Bugs and Limitations" section for further
19           information.
20
21           It is beyond the scope of this document to document or explain
22           how to utilize the various cmake features, such as different
23           build backends or provisions for finding support libraries.
24
25           Please refer to the cmake documentation for additional information
26           that applies to building any project with cmake.
27
28Prerequisites
29-------------
30
31The following tools must be in PATH:
32
33* cmake, version 2.8 or later
34  cmake version 3.1.3 or later is required to work with current OpenSSL
35  releases.  (OpenSSL is an optional prerequisite of httpd.)
36* Perl
37* If using a command-line compiler: compiler and linker and related tools
38  (Refer to the cmake documentation for more information.)
39
40The following support libraries are mandatory:
41
42* APR, built with cmake
43  + Either APR 2.0-dev (trunk) or APR 1.5.x and APR-Util 1.5.x.
44  + When building APR (but not APR-Util), specify the build option
45    APR_INSTALL_PRIVATE_H so that non-standard files required for building
46    Apache httpd are installed.
47  + Additional APR settings affect httpd but are not mandatory, such as
48    APR_HAVE_IPV6.
49* PCRE
50
51Certain optional features of APR 2.0-dev (trunk) or APR-Util 1.5.x
52allow some optional features of httpd to be enabled.  For example,
53APU_HAVE_CRYPTO is required for mod_session_crypto.
54
55Additional support libraries allow some optional features of httpd to be
56enabled:
57
58* libxml2 (e.g., mod_proxy_html)
59* lua 5.1 (mod_lua)
60* nghttp2 (mod_http2)
61* openssl (mod_ssl and https support for ab)
62* zlib (mod_deflate)
63
64OpenSSL
65-------
66
67If you have a binary install of OpenSSL in a well-known directory (e.g.,
68%HOME%\OpenSSL-Win64) and you wish to build httpd against a different
69install of OpenSSL, the cmake build may unexpectedly select OpenSSL
70libraries in the well-known directory even if the expected include files
71are used.  Check the cmake output from your httpd build to confirm that
72the expected OpenSSL libraries and include files are used.
73
74The cmake FindOpenSSL module searches for OpenSSL libraries in a "VC"
75subdirectory of the OpenSSL install with filenames that indicate the build
76type (e.g., "<PREFIX>/lib/VC/ssleay32MD.lib"); defining CMAKE_PREFIX_PATH
77or OPENSSL_ROOT_DIR or even OPENSSL_LIBRARIES does not circumvent finding
78these libraries.
79
80To work around this issue, rename the well-known OpenSSL directory while
81building httpd.  Let us know if you find a better solution.
82
83nghttp2
84-------
85
86This is required for mod_http2.
87
88cmake-based build support for nghttp2 for Windows can be found at
89https://github.com/trawick/nghttp2-minimal-cmake.  That easily fits into
90a build system that already uses cmake for httpd, apr, and perhaps other
91packages.  A dynamic build of nghttp2 using its normal Windows build
92system should also be usable by nghttp2.
93
94How to build
95------------
96
971. cd to a clean directory for building (i.e., don't build in your
98   source tree)
99
1002. Make sure cmake and Perl are in PATH.  Additionally, some backends 
101   require compile tools in PATH.  (Hint: "Visual Studio Command Prompt")
102
1033. cmake -G "some backend, like 'NMake Makefiles'"
104     -DCMAKE_INSTALL_PREFIX=d:/path/to/httpdinst
105     -DENABLE_foo=A|I|O|a|i
106     -DENABLE_MODULES=A|I|O|a|i
107     d:/path/to/httpdsource
108
109   Alternately, you can use the cmake-gui and update settings in the GUI.
110
111   PCRE_INCLUDE_DIR, PCRE_LIBRARIES, APR_INCLUDE_DIR, APR_LIBRARIES,
112     NGHTTP2_INCLUDE_DIR, NGHTTP2_LIBRARIES:
113
114       cmake doesn't bundle FindXXX for these packages, so the crucial
115       information has to be specified in this manner if they aren't found
116       in their default location.
117
118     -DPCRE_INCLUDE_DIR=d:/path/to/pcreinst/include
119     -DPCRE_LIBRARIES=d:/path/to/pcreinst/lib/pcre[d].lib
120
121       These will have to be specified only if PCRE is installed to a different
122       directory than httpd, or if debug *and* release builds of PCRE were
123       installed there and you want to control which is used.  (Currently the
124       build will use pcred.lib (debug) if it is found in the default location
125       and not overridden with -DPCRE_LIBRARIES.)
126
127     -DAPR_INCLUDE_DIR=d:/path/to/aprinst/include
128     -DAPR_LIBRARIES="d:/path/to/aprinst/lib/libapr-1.lib;d:/path/to/aprinst/lib/libaprutil-1.lib"
129
130       These will have to be specified if APR[-Util] was installed to a
131       different directory than httpd.
132
133       When building with APR trunk (future APR 2.x, with integrated APR-Util),
134       specify just the path to libapr-2.lib:
135
136           -DAPR_LIBRARIES=d:/path/to/aprinst/lib/libapr-2.lib
137
138       APR+APR-Util 1.x vs. APR trunk will be detected automatically if they
139       are installed to the same location as httpd.
140
141       APR-Util 1.x has an optional LDAP library.  If APR-Util has LDAP enabled
142       and httpd's mod_ldap and mod_authnz_ldap are being used, include the
143       path to the LDAP library in the APR_LIBRARIES setting.  (If APR and
144       APR-Util are found in the default location, the LDAP library will be
145       included if it is present.
146
147     -DNGHTTP2_INCLUDE_DIR=d:/path/to/nghttp2inst/include  (which has nghttp2/*.h)
148     -DNGHTTP2_LIBRARIES="d:/path/to/nghttp2inst/lib/nghttp2.lib"
149
150       These will have to be specified if nghttp2 was installed to a different
151       directory than httpd.
152
153   LIBXML2_ICONV_INCLUDE_DIR, LIBXML2_ICONV_LIBRARIES
154
155      If using a module that requires libxml2 *and* the build of libxml2 requires
156      iconv, set these variables to allow iconv includes and libraries to be
157      used.  For example:
158
159      -DLIBXML2_ICONV_INCLUDE_DIR=c:\iconv-1.9.2.win32\include
160      -DLIBXML2_ICONV_LIBRARIES=c:\iconv-1.9.2.win32\lib\iconv.lib
161
162   CMAKE_C_FLAGS_RELEASE, _DEBUG, _RELWITHDEBINFO, _MINSIZEREL
163   CMAKE_BUILD_TYPE
164
165       For NMake Makefiles the choices are at least DEBUG, RELEASE,
166       RELWITHDEBINFO, and MINSIZEREL
167       Other backends may have other selections.
168
169   ENABLE_foo:
170
171       Each module has a default setting which can be overridden with one of
172       the following values:
173           A        build and Activate module
174           a        build and Activate module IFF prereqs are available; if
175                    prereqs are unavailable, don't build it
176           I        build module but leave it Inactive (commented-out
177                    LoadModule directive)
178           i        build module but leave it Inactive IFF prereqs are
179                    available; if prereqs are unavailable, don't build it
180           O        Omit module completely
181
182       Examples: -DENABLE_ACCESS_COMPAT=O
183                 -DENABLE_PROXY_HTML=i
184
185   ENABLE_MODULES:
186
187       This changes the *minimum* enablement of all modules to the specified
188       value (one of A, a, I, i, O, as described under ENABLE_foo above).
189
190       The ranking of enablement from lowest to highest is O, i, I, a, A.
191       If a specific module has a higher rank enablement setting, either from
192       a built-in default or from -DENABLE_foo, ENABLE_MODULES won't affect
193       that module.  However, if a specific module has a lower-rank enablement
194       setting, presumably from a built-in default, the value of ENABLE_MODULES
195       will be used for that module.
196
197       Explanations for possible values:
198
199       -DENABLE_MODULES=a      build and activate all possible modules,
200                               ignoring any with missing prereqs
201                               (doesn't affect modules with A for ENABLE_foo)
202
203       -DENABLE_MODULES=i      build but leave inactive all possible
204                               modules, ignoring any with missing
205                               prereqs
206                               (doesn't affect modules with A, a, or I for 
207                               ENABLE_foo)
208
209       -DENABLE_MODULES=O      no impact, since all modules are either
210                               already disabled or have a higher setting
211
212       -DENABLE_MODULES=A      build and activate all possible modules,
213                               failing the build if any module is missing
214                               a prereq
215
216       -DENABLE_MODULES=I      similar to -DENABLE_MODULES=A
217                               (doesn't affect modules with A or a for
218                               ENABLE_foo)
219
220   WITH_MODULES:
221
222       Comma-separated paths to single file modules to statically linked into
223       the server, like the --with-module=modpath:/path/to/mod_foo.c with
224       the autoconf-based build.  Key differences: The modpath (e.g., 
225       "generators") isn't provided or used, and the copy of the module
226       source being built is automatically updated when it changes.
227       See also EXTRA_COMPILE_FLAGS, EXTRA_INCLUDES, and EXTRA_LIBS.
228
229   EXTRA_COMPILE_FLAGS:
230
231       Space-delimited compile flags to define with the build.
232
233   EXTRA_INCLUDES:
234
235       List of additional directories to search for .h files.  This may
236       be necessary when including third-party modules in the httpd build
237       via WITH_MODULES.
238
239   EXTRA_LIBS:
240
241       List of additional libraries to link with.  This may be necessary when
242       including third-party modules in the httpd build via WITH_MODULES.
243
244   Port and SSLPort:
245
246       Port numbers for substitution into default .conf files.  (The defaults
247       are 80 and 443.)
248
249   INSTALL_PDB:
250
251       If .pdb files are generated for debugging, install them.
252       Default: ON
253
254       The .pdb files are generally needed for debugging low-level code
255       problems.  If they aren't installed, they are still available in the
256       build directory for use by alternate packaging implementations or when
257       debugging on the build machine.
258
259   INSTALL_MANUAL:
260
261       Install the Apache HTTP Server manual.
262       Default: ON
263
264       This could be turned off when developing changes in order to speed up
265       installation time.
266
2674. Build using the chosen generator (e.g., "nmake install" for cmake's "NMake
268   Makefiles" generator).

270Running the server and support programs
271---------------------------------------
272
273This build system does not copy binaries such as dlls from other projects
274into the httpd install location.  Without taking some precautions, httpd
275and support programs can fail to start or modules can fail to load because
276a support library can't be found in PATH or in the directory of the httpd
277binary.
278
279This can be resolved in several different ways:
280
281* Install httpd and the various support libraries to a common install
282  prefix so that support libraries and httpd programs are installed in
283  the same bin directory and are found without setting PATH.
284
285* Update PATH to include the bin directories of all necessary support
286  libraries.
287
288  Depending on where PATH is set, it may not affect starting httpd as
289  a service.
290
291* Maintain a script which combines required binaries into a common 
292  location, such as the httpd installion bin directory, and use that
293  script after building or otherwise installing or updating support
294  libraries.
295
296* AVOID THE USE of any unrepeatable process of copying dll files around
297  from different install locations until something starts working.  The
298  result is that when you later update a support library to pick up a
299  security fix, httpd will likely continue to use the old, vulnerable
300  library file.

302Known Bugs and Limitations
303--------------------------
304
305* no standard script or makefile is provided to tie together the builds
306  of httpd and support libraries in a manner suitable for typical users
307* no logic to find support libraries or otherwise build these modules:
308  + mod_socache_dc (requires distcache), mod_serf (requires serf)
309  + additionally, mod_firehose doesn't compile on Windows anyway
310* buildmark.c isn't necessarily rebuilt when httpd.exe is regenerated
311* ApacheMonitor has a build error and is disabled
312* CGI examples aren't installed
313* dbmmanage.pl and wintty aren't built/installed
314* module enablement defaults are not in sync with the autoconf-based build
315* no support for static support library builds; unclear if that is a
316  requirement; if so: taking PCRE as an example, we'd need to detect that it
317  is static and then turn on PCRE_STATIC for the libhttpd build
318
319Generally:
320
321* Many httpd features have not been tested with this build.
322* Developers need to examine the existing Windows build in great detail and see
323  what is missing from the cmake-based build, whether a feature or some build
324  nuance.
325* Any feedback you can provide on your experiences with this build will be
326  helpful.
