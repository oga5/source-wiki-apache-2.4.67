2Variables
3---------
4
5The CI scripts use the following environment variables:
6
7* APR_VERSION - if set, APR of this version is built and installed in
8  $HOME/root/apr-$APR_VERSION - a value of "trunk" means trunk is
9  used, "*.x" means a branch, otherwise a tagged version is implied.
10
11* APR_CONFIG - arguments to pass when running APR's configure script
12  if APR_VERSION is set
13
14* APU_VERSION - if set, APR-util of this version is built and
15  installed in $HOME/root/apr-util-$APU_VERSION - a value of "*.x"
16  means a branch, otherwise a tagged version is implied.  (Since there
17  is no "trunk" for apr-util, that value cannot be used here.)
18
19* APU_CONFIG - arguments to pass when running APR-util's configure
20  script if APU_VERSION is set
21
22* CONFIG - arguments to pass to httpd's configure script.
23
24* BUILDCONFIG - arguments to pass when running httpd's ./buildconf script
25
26* MFLAGS - arguments to pass when running "make" for httpd.
27
28* SKIP_TESTING - if set, the Perl test framework is not run for the
29  build.
30
31* TEST_UBSAN - set for job using UBSan ("Undefined Behaviour Sanitizer")
32
33* TEST_MALLOC - set for job using enhanced malloc debugging.
34
35* TEST_INSTALL - set for job testing "make install"
36
37* TEST_VPATH - set for job testing srcdir!=builddir 
38
39* TEST_LDAP - set for job with slapd, running LDAP tests
40
41* TEST_SSL - set for job with SSL/TLS testing variants
42
43* TESTS - a list of Perl framework tests to run
44
45* TEST_ARGS - arguments to pass to ./t/TEST in the Perl test framework
46
47* CLEAR_CACHE - if set, the cached $HOME/root is removed before each build
48
49Caching -- NOTE, BROKEN IN GITHUB ACTIONS --
50-------
51
52Perl modules installed in $HOME/perl5 are cached.
53
54Anything installed into the $HOME/root directory is cached - notably,
55versions of APR/APR-util are installed here and cached across httpd
56build jobs without needing to be rebuilt every time.
57
58The cached installs of APR/APR-util are refreshed if the
59last-changed-revision of the build is stale.
60
61If APR_VERSION and APU_VERSION are both set to 1.x versions, then
62CLEAR_CACHE should also be set to disable APR* caching.  APR-util can
63only be rebuilt if an APR checkout is present, so a APR-util cannot be
64built from source alone.  (i.e. the scripts do not handle the case of
65cached, fresh APR plus a cached but stale APR-util)

115Testing from a Feature Branch [*** N/A FOR GITHUB ACTIONS ***]
116-----------------------------
117
118An SVN branch off trunk should be mirrored to github, and will be
119tested in the same way that trunk is in CI, so this workflow is
120available for those familiar with using Subversion and the standard
121ASF/httpd repository layout.
122
123Tested branches are listed at: https://travis-ci.com/github/apache/httpd/branches
124
125Travis will also run the tests for a PR filed against the httpd Github
126repository at https://github.com/apache/httpd or from a fork of this
127repository if enabled for the Travis user.
128
129A workflow to enable testing would be as follows, substituting
130$USERNAME for your github username:
131
132  $ git clone https://github.com/apache/httpd
133  $ cd httpd
134  $ git remote add $USERNAME git@github.com:$USERNAME/httpd.git
135  $ git checkout -b my-feature origin/trunk
136  ... do some work ...
137  $ git commit ...
138  $ git push -u $USERNAME my-feature:my-feature
139
140To enable testing for a fork, visit the settings page at
141https://travis-ci.com/$USERNAME/httpd/settings - you may need to sync
142your account via https://travis-ci.com/account/repositories for a
143freshly created fork.
144
145To create a Pull Request, go to the URL produced in the "git push"
146command output when pushing to your fork, which is something like:
147https://github.com/apache/httpd/compare/trunk...$USERNAME:trunk
148
149Once a PR has been created, travis will run the tests and link the
150results from a PR comment. All tested PRs are listed here:
151https://travis-ci.com/github/apache/httpd/pull_requests
152
153To merge from github back to SVN trunk, create a patch from e.g.:
154
155  $ git diff origin/trunk..my-feature
156
157and then apply it in SVN.  To rebase a feature once trunk has
158diverged, from a feature branch run:
159
160  $ git pull
161  $ git rebase -i origin/trunk
162
163and follow the standard rebase steps.
