66<h2><a name="introduction" id="introduction">Introduction</a></h2>
67<p>A <code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">RewriteRule</a></code> can have
68its behavior modified by one or more flags. Flags are included in
69square brackets at the end of the rule, and multiple flags are separated
70by commas.</p>
71<pre class="prettyprint lang-config">RewriteRule pattern target [Flag1,Flag2,Flag3]</pre>
72
73
74<p>Each flag (with a few exceptions) has a short form, such as
75<code>CO</code>, as well as a longer form, such as <code>cookie</code>.
76While it is most common to use
77the short form, it is recommended that you familiarize yourself with the
78long form, so that you remember what each flag is supposed to do.
79Some flags take one or more arguments. Flags are not case sensitive.</p>
80
81<p>Flags that alter metadata associated with the request (T=, H=, E=)
82have no affect in per-directory and htaccess context, when a substitution
83(other than '-') is performed during the same round of rewrite processing.
84</p>
85
86<p>Presented here are each of the available flags, along with an example
87of how you might use them.</p>
88</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>

90<h2><a name="flag_b" id="flag_b">B (escape backreferences)</a></h2>
91<p>The [B] flag instructs <code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">RewriteRule</a></code> to escape non-alphanumeric
92characters before applying the transformation.</p>
93
94<p><code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code> has to unescape URLs before mapping them,
95so backreferences are unescaped at the time they are applied.
96Using the B flag, non-alphanumeric characters in backreferences
97will be escaped. For example, consider the rule:</p>
98
99<p>For similar escaping of server-variables, see
100    the "escape" <a href="#mapfunc">mapping-function</a></p>
101
102
103<pre class="prettyprint lang-config">RewriteRule "^search/(.*)$" "/search.php?term=$1"</pre>
104
105
106<p>Given a search term of 'x &amp; y/z', a browser will encode it as
107'x%20%26%20y%2Fz', making the request 'search/x%20%26%20y%2Fz'. Without the B
108flag, this rewrite rule will map to 'search.php?term=x &amp; y/z', which
109isn't a valid URL, and so would be encoded as
110<code>search.php?term=x%20&amp;y%2Fz=</code>, which is not what was intended.</p>
111
112<p>With the B flag set on this same rule, the parameters are re-encoded
113before being passed on to the output URL, resulting in a correct mapping to
114<code>/search.php?term=x%20%26%20y%2Fz</code>.</p>
115
116<pre class="prettyprint lang-config">RewriteRule "^search/(.*)$" "/search.php?term=$1" [B,PT]</pre>
117
118
119<p>Note that you may also need to set <code class="directive"><a href="../mod/core.html#allowencodedslashes">AllowEncodedSlashes</a></code> to <code>On</code> to get this
120particular example to work, as httpd does not allow encoded slashes in URLs, and
121returns a 404 if it sees one.</p>
122
123<p>This escaping is particularly necessary in a proxy situation,
124when the backend may break if presented with an unescaped URL.</p>
125
126<p>An alternative to this flag is using a <code class="directive"><a href="../mod/mod_rewrite.html#rewritecond">RewriteCond</a></code> to capture against %{THE_REQUEST} which will capture
127strings in the encoded form.</p>
128
129<p>In 2.4.26 and later, you can limit the escaping to specific characters
130in backreferences by listing them: <code>[B=#?;]</code>. Note: The space
131character can be used in the list of characters to escape, but you must quote
132the entire third argument of <code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">RewriteRule</a></code>
133and the space must not be the last character in the list.</p>
134
135<pre class="prettyprint lang-config"># Escape spaces and question marks.  The quotes around the final argument 
136# are required when a space is included.
137RewriteRule "^search/(.*)$" "/search.php?term=$1" "[B= ?]"</pre>
138
139
140<p>To limit the characters escaped this way, see <a href="#flag_bne">#flag_bne</a>
141        and <a href="#flag_bctls">#flag_bctls</a></p>
142</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>

144<h2><a name="flag_bnp" id="flag_bnp">BNP|backrefnoplus (don't escape space to +)</a></h2>
145<p>The [BNP] flag instructs <code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">RewriteRule</a></code> to escape the space character
146in a backreference to %20 rather than '+'. Useful when the backreference
147will be used in the path component rather than the query string.</p>
148
149<pre class="prettyprint lang-config"># Escape spaces to %20 in the path instead of + as used in form submission via
150# the query string
151RewriteRule "^search/(.*)$" "/search.php/$1" "[B,BNP]"</pre>
152
153
154
155<p>This flag is available in version 2.4.26 and later.</p>
156</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>
157<div class="section">
158<h2><a name="flag_bctls" id="flag_bctls">BCTLS</a></h2>
159<p>The [BCTLS] flag is similar to the [B] flag, but only escapes
160control characters and the space character. This is the same set of
161characters rejected when they are copied into the query string unencoded.
162</p>
163
164<pre class="prettyprint lang-config"># Escape control characters and spaces
165RewriteRule "^search/(.*)$" "/search.php/$1" "[BCTLS]"</pre>
166
167
168<p>This flag is available in version 2.4.57 and later.</p>
169
170</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>
171<div class="section">
172<h2><a name="flag_bne" id="flag_bne">BNE</a></h2>
173<p>The list of characters in [BNE=...] are treated as exclusions to the
174characters of the [B] or [BCTLS] flags. The listed characters will not be
175escaped.
176</p>
177
178<pre class="prettyprint lang-config"># Escape the default characters, but leave /
179RewriteRule "^search/(.*)$" "/search.php?term=$1" "[B,BNE=/]"</pre>
180
181
182<p>This flag is available in version 2.4.57 and later.</p>
183</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>

185<h2><a name="flag_c" id="flag_c">C|chain</a></h2>
186<p>The [C] or [chain] flag indicates that the <code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">RewriteRule</a></code> is chained to the next
187rule. That is, if the rule matches, then it is processed as usual and
188control moves on to the next rule. However, if it does not match, then
189the next rule, and any other rules that are chained together, are
190skipped.</p>
191
192</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>

194<h2><a name="flag_co" id="flag_co">CO|cookie</a></h2>
195<p>The [CO], or [cookie] flag, allows you to set a cookie when a
196particular <code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">RewriteRule</a></code>
197matches. The argument consists of three required fields and five optional
198fields.</p>
199
200<p>The full syntax for the flag, including all attributes, is as
201follows:</p>
202
203<div class="example"><p><code>
204[CO=NAME:VALUE:DOMAIN:lifetime:path:secure:httponly:samesite]
205</code></p></div>
206
207<p>If a literal ':' character is needed in any of the cookie fields, an 
208alternate syntax is available.  To opt-in to the alternate syntax, the cookie 
209"Name" should be preceded with a ';' character, and field separators should be
210specified as ';'.</p>
211
212<div class="example"><p><code>
213[CO=;NAME;VALUE:MOREVALUE;DOMAIN;lifetime;path;secure;httponly;samesite]
214</code></p></div>
215
216<p>You must declare a name, a value, and a domain for the cookie to be set.</p>
217
218<dl>
219<dt>Domain</dt>
220<dd>The domain for which you want the cookie to be valid. This may be a
221hostname, such as <code>www.example.com</code>, or it may be a domain,
222such as <code>.example.com</code>. It must be at least two parts
223separated by a dot. That is, it may not be merely <code>.com</code> or
224<code>.net</code>. Cookies of that kind are forbidden by the cookie
225security model.</dd>
226</dl>
227
228<p>You may optionally also set the following values:</p>
229
230<dl>
231<dt>Lifetime</dt>
232<dd>The time for which the cookie will persist, in minutes.</dd>
233<dd>A value of 0 indicates that the cookie will persist only for the
234current browser session. This is the default value if none is
235specified.</dd>
236
237<dt>Path</dt>
238<dd>The path, on the current website, for which the cookie is valid,
239such as <code>/customers/</code> or <code>/files/download/</code>.</dd>
240<dd>By default, this is set to <code>/</code> - that is, the entire
241website.</dd>
242
243<dt>Secure</dt>
244<dd>If set to <code>secure</code>, <code>true</code>, or <code>1</code>,
245the cookie will only be permitted to be translated via secure (https)
246connections.</dd>
247
248<dt>httponly</dt>
249<dd>If set to <code>HttpOnly</code>, <code>true</code>, or
250<code>1</code>, the cookie will have the <code>HttpOnly</code> flag set,
251which means that the cookie is inaccessible to JavaScript code on
252browsers that support this feature.</dd>
253
254<dt>samesite</dt>
255<dd>If set to anything other than <code>false</code> or <code>0</code>, the <code>SameSite</code>
256attribute is set to the specified value.  Typical values are <code>None</code>,
257<code>Lax</code>, and <code>Strict</code>. Available in 2.4.47 and later.</dd>
258</dl>
259
260
261<p>Consider this example:</p>
262
263<pre class="prettyprint lang-config">RewriteEngine On
264RewriteRule   "^/index\.html"   "-" [CO=frontdoor:yes:.example.com:1440:/]</pre>
265
266
267<p>In the example give, the rule doesn't rewrite the request.
268The "-" rewrite target tells <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code> to pass the request
269through unchanged. Instead, it sets a cookie
270called 'frontdoor' to a value of 'yes'. The cookie is valid for any host
271in the <code>.example.com</code> domain. It is set to expire in 1440
272minutes (24 hours) and is returned for all URIs.</p>
273

276<h2><a name="flag_dpi" id="flag_dpi">DPI|discardpath</a></h2>
277<p>The DPI flag causes the PATH_INFO portion of the rewritten URI to be
278discarded.</p>
279<p>This flag is available in version 2.2.12 and later.</p>
280<p>In per-directory context, the URI each <code class="directive">RewriteRule</code>
281compares against is the concatenation of the current values of the URI
282and PATH_INFO.</p>
283
284<p>The current URI can be the initial URI as requested by the client, the
285result of a previous round of <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code> processing, or the result of
286a prior rule in the current round of <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code> processing.</p>
287
288<p>In contrast, the PATH_INFO that is appended to the URI before each
289rule reflects only the value of PATH_INFO before this round of
290<code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code> processing. As a consequence, if large portions
291of the URI are matched and copied into a substitution in multiple
292<code class="directive">RewriteRule</code> directives, without regard for
293which parts of the URI came from the current PATH_INFO, the final
294URI may have multiple copies of PATH_INFO appended to it.</p>
295
296<p>Use this flag on any substitution where the PATH_INFO that resulted
297from the previous mapping of this request to the filesystem is not of
298interest.  This flag permanently forgets the PATH_INFO established
299before this round of <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code> processing began. PATH_INFO will
300not be recalculated until the current round of <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code> processing
301completes.  Subsequent rules during this round of processing will see
302only the direct result of substitutions, without any PATH_INFO
303appended.</p>
304</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>

306<h2><a name="flag_e" id="flag_e">E|env</a></h2>
307<p>With the [E], or [env] flag, you can set the value of an environment
308variable. Note that some environment variables may be set after the rule
309is run, thus unsetting what you have set. See <a href="../env.html">the
310Environment Variables document</a> for more details on how Environment
311variables work.</p>
312
313<p>The full syntax for this flag is:</p>
314
315<pre class="prettyprint lang-config">[E=VAR:VAL]
316[E=!VAR]</pre>
317
318
319<p><code>VAL</code> may contain backreferences (<code>$N</code> or
320<code>%N</code>) which are expanded.</p>
321
322<p>Using the short form</p>
323
324<div class="example"><p><code>
325[E=VAR]
326</code></p></div>
327
328<p>you can set the environment variable named <code>VAR</code> to an
329empty value.</p>
330
331<p>The form</p>
332
333<div class="example"><p><code>
334[E=!VAR]
335</code></p></div>
336
337<p>allows to unset a previously set environment variable named
338<code>VAR</code>.</p>
339
340<p>Environment variables can then be used in a variety of
341contexts, including CGI programs, other RewriteRule directives, or
342CustomLog directives.</p>
343
344<p>The following example sets an environment variable called 'image' to a
345value of '1' if the requested URI is an image file. Then, that
346environment variable is used to exclude those requests from the access
347log.</p>
348
349<pre class="prettyprint lang-config">RewriteRule "\.(png|gif|jpg)$"   "-" [E=image:1]
350CustomLog   "logs/access_log"    combined env=!image</pre>
351
352
353<p>Note that this same effect can be obtained using <code class="directive"><a href="../mod/mod_setenvif.html#setenvif">SetEnvIf</a></code>. This technique is offered as
354an example, not as a recommendation.</p>
355</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>

357<h2><a name="flag_end" id="flag_end">END</a></h2>
358<p>Using the [END] flag terminates not only the current round of rewrite
359processing (like [L]) but also prevents any subsequent rewrite
360processing from occurring in per-directory (htaccess) context.</p>
361
362<p>This does not apply to new requests resulting from external
363redirects.</p>
364</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>
365<div class="section">
366<h2><a name="flag_f" id="flag_f">F|forbidden</a></h2>
367<p>Using the [F] flag causes the server to return a 403 Forbidden status
368code to the client. While the same behavior can be accomplished using
369the <code class="directive"><a href="../mod/mod_access_compat.html#deny">Deny</a></code> directive, this
370allows more flexibility in assigning a Forbidden status.</p>
371
372<p>The following rule will forbid <code>.exe</code> files from being
373downloaded from your server.</p>
374
375<pre class="prettyprint lang-config">RewriteRule "\.exe"   "-" [F]</pre>
376
377
378<p>This example uses the "-" syntax for the rewrite target, which means
379that the requested URI is not modified. There's no reason to rewrite to
380another URI, if you're going to forbid the request.</p>
381
382<p>When using [F], an [L] is implied - that is, the response is returned
383immediately, and no further rules are evaluated.</p>
384
385</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>
386<div class="section">
387<h2><a name="flag_g" id="flag_g">G|gone</a></h2>
388<p>The [G] flag forces the server to return a 410 Gone status with the
389response. This indicates that a resource used to be available, but is no
390longer available.</p>
391
392<p>As with the [F] flag, you will typically use the "-" syntax for the
393rewrite target when using the [G] flag:</p>
394
395<pre class="prettyprint lang-config">RewriteRule "oldproduct"   "-" [G,NC]</pre>
396
397
398<p>When using [G], an [L] is implied - that is, the response is returned
399immediately, and no further rules are evaluated.</p>
400
401</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>

403<h2><a name="flag_h" id="flag_h">H|handler</a></h2>
404<p>Forces the resulting request to be handled with the specified
405handler. For example, one might use this to force all files without a
406file extension to be parsed by the php handler:</p>
407
408<pre class="prettyprint lang-config">RewriteRule "!\."  "-" [H=application/x-httpd-php]</pre>
409
410
411<p>
412The regular expression above - <code>!\.</code> - will match any request
413that does not contain the literal <code>.</code> character.
414</p>
415
416<p>This can be also used to force the handler based on some conditions.
417For example, the following snippet used in per-server context allows
418<code>.php</code> files to be <em>displayed</em> by <code>mod_php</code>
419if they are requested with the <code>.phps</code> extension:</p>
420
421<pre class="prettyprint lang-config">RewriteRule "^(/source/.+\.php)s$" "$1" [H=application/x-httpd-php-source]</pre>
422
423
424<p>The regular expression above - <code>^(/source/.+\.php)s$</code> - will
425match any request that starts with <code>/source/</code> followed by 1 or
426n characters followed by <code>.phps</code> literally. The backreference
427$1 referrers to the captured match within parenthesis of the regular
428expression.</p>
429</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>
430<div class="section">
431<h2><a name="flag_l" id="flag_l">L|last</a></h2>
432<p>The [L] flag causes <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code> to stop processing
433the rule set. In most contexts, this means that if the rule matches, no
434further rules will be processed. This corresponds to the
435<code>last</code> command in Perl, or the <code>break</code> command in
436C. Use this flag to indicate that the current rule should be applied
437immediately without considering further rules.</p>
438
439<p>If you are using <code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">RewriteRule</a></code> in either
440<code>.htaccess</code> files or in
441<code class="directive"><a href="../mod/core.html#directory">&lt;Directory&gt;</a></code> sections,
442it is important to have some understanding of how the rules are
443processed.  The simplified form of this is that once the rules have been
444processed, the rewritten request is handed back to the URL parsing
445engine to do what it may with it. It is possible that as the rewritten
446request is handled, the <code>.htaccess</code> file or
447<code class="directive"><a href="../mod/core.html#directory">&lt;Directory&gt;</a></code> section
448may be encountered again, and thus the ruleset may be run again from the
449start. Most commonly this will happen if one of the rules causes a
450redirect - either internal or external - causing the request process to
451start over.</p>
452
453<p>It is therefore important, if you are using <code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">RewriteRule</a></code> directives in one of these
454contexts, that you take explicit steps to avoid rules looping, and not
455count solely on the [L] flag to terminate execution of a series of
456rules, as shown below.</p>
457
458<p> An alternative flag, [END], can be used to terminate not only the
459current round of rewrite processing but prevent any subsequent
460rewrite processing from occurring in per-directory (htaccess)
461context. This does not apply to new requests resulting from external
462redirects.</p>
463
464<p>The example given here will rewrite any request to
465<code>index.php</code>, giving the original request as a query string
466argument to <code>index.php</code>, however, the <code class="directive"><a href="../mod/mod_rewrite.html#rewritecond">RewriteCond</a></code> ensures that if the request
467is already for <code>index.php</code>, the <code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">RewriteRule</a></code> will be skipped.</p>
468
469<pre class="prettyprint lang-config">RewriteBase "/"
470RewriteCond "%{REQUEST_URI}" !=/index.php
471RewriteRule "^(.*)"          "/index.php?req=$1" [L,PT]</pre>
472
473</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>
474<div class="section">
475<h2><a name="flag_n" id="flag_n">N|next</a></h2>
476<p>
477The [N] flag causes the ruleset to start over again from the top, using
478the result of the ruleset so far as a starting point. Use
479with extreme caution, as it may result in loop.
480</p>
481<p>
482The [Next] flag could be used, for example, if you wished to replace a
483certain string or letter repeatedly in a request. The example shown here
484will replace A with B everywhere in a request, and will continue doing
485so until there are no more As to be replaced.
486</p>
487<pre class="prettyprint lang-config">RewriteRule "(.*)A(.*)" "$1B$2" [N]</pre>
488
489<p>You can think of this as a <code>while</code> loop: While this
490pattern still matches (i.e., while the URI still contains an
491<code>A</code>), perform this substitution (i.e., replace the
492<code>A</code> with a <code>B</code>).</p>
493
494<p>In 2.4.8 and later, this module returns an error after 10,000 iterations to
495protect against unintended looping.  An alternative maximum number of
496iterations can be specified by adding to the N flag.  </p>
497<pre class="prettyprint lang-config"># Be willing to replace 1 character in each pass of the loop
498RewriteRule "(.+)[&gt;&lt;;]$" "$1" [N=32000]
499# ... or, give up if after 10 loops
500RewriteRule "(.+)[&gt;&lt;;]$" "$1" [N=10]</pre>
501
502
503</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>
504<div class="section">
505<h2><a name="flag_nc" id="flag_nc">NC|nocase</a></h2>
506<p>Use of the [NC] flag causes the <code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">RewriteRule</a></code> to be matched in a
507case-insensitive manner. That is, it doesn't care whether letters appear
508as upper-case or lower-case in the matched URI.</p>
509
510<p>In the example below, any request for an image file will be proxied
511to your dedicated image server. The match is case-insensitive, so that
512<code>.jpg</code> and <code>.JPG</code> files are both acceptable, for
513example.</p>
514
515<pre class="prettyprint lang-config">RewriteRule "(.*\.(jpg|gif|png))$" "http://images.example.com$1" [P,NC]</pre>
516
517</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>
518<div class="section">
519<h2><a name="flag_ne" id="flag_ne">NE|noescape</a></h2>
520<p>By default, when a <code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">RewriteRule</a></code>
521results in an external redirect, any characters in the output that are
522not in the following safe set will be converted to their hexcode
523(percent-encoded) equivalents:</p>
524
525<ul>
526  <li>Alphanumeric characters: <code>A-Z</code>, <code>a-z</code>,
527  <code>0-9</code></li>
528  <li>Special characters: <code>$-_.+!*'(),:;@&amp;=/~</code></li>
529</ul>
530
531<p>For example, <code>#</code> would be converted to <code>%23</code>,
532and <code>?</code> to <code>%3F</code>. The <code>%</code> character
533is also escaped (to <code>%25</code>), which means that any
534percent-encoding already present in the substitution will be
535double-encoded.</p>
536
537<p>Using the [NE] flag prevents this escaping, allowing characters
538such as <code>#</code> and <code>?</code> to pass through to the
539redirect URL unmodified.</p>
540
541<pre class="prettyprint lang-config">RewriteRule "^/anchor/(.+)" "/bigpage.html#$1" [NE,R]</pre>
542
543
544<p>
545The above example will redirect <code>/anchor/xyz</code> to
546<code>/bigpage.html#xyz</code>. Omitting the [NE] will result in the #
547being converted to its hexcode equivalent, <code>%23</code>, which will
548then result in a 404 Not Found error condition.
549</p>
550
551</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>
552<div class="section">
553<h2><a name="flag_ns" id="flag_ns">NS|nosubreq</a></h2>
554<p>Use of the [NS] flag prevents the rule from being used on
555subrequests. For example, a page which is included using an SSI (Server
556Side Include) is a subrequest, and you may want to avoid rewrites
557happening on those subrequests. Also, when <code class="module"><a href="../mod/mod_dir.html">mod_dir</a></code>
558tries to find out information about possible directory default files
559(such as <code>index.html</code> files), this is an internal
560subrequest, and you often want to avoid rewrites on such subrequests.
561On subrequests, it is not always useful, and can even cause errors, if
562the complete set of rules are applied. Use this flag to exclude
563problematic rules.</p>
564
565<p>To decide whether or not to use this rule: if you prefix URLs with
566CGI-scripts, to force them to be processed by the CGI-script, it's
567likely that you will run into problems (or significant overhead)
568on sub-requests. In these cases, use this flag.</p>
569
570<p>
571Images, javascript files, or css files, loaded as part of an HTML page,
572are not subrequests - the browser requests them as separate HTTP
573requests.
574</p>
575</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>
576<div class="section">
577<h2><a name="flag_p" id="flag_p">P|proxy</a></h2>
578<p>Use of the [P] flag causes the request to be handled by
579<code class="module"><a href="../mod/mod_proxy.html">mod_proxy</a></code>, and handled via a proxy request. For
580example, if you wanted all image requests to be handled by a back-end
581image server, you might do something like the following:</p>
582
583<pre class="prettyprint lang-config">RewriteRule "/(.*)\.(jpg|gif|png)$" "http://images.example.com/$1.$2" [P]</pre>
584
585
586<p>Use of the [P] flag implies [L] - that is, the request is immediately
587pushed through the proxy, and any following rules will not be
588considered.</p>
589
590<p>
591You must make sure that the substitution string is a valid URI
592(typically starting with <code>http://</code><em>hostname</em>) which can be
593handled by the <code class="module"><a href="../mod/mod_proxy.html">mod_proxy</a></code>. If not, you will get an
594error from the proxy module. Use this flag to achieve a
595more powerful implementation of the <code class="directive"><a href="../mod/mod_proxy.html#proxypass">ProxyPass</a></code> directive,
596to map remote content into the namespace of the local server.</p>
597
598<div class="warning">
599<h3>Security Warning</h3>
600<p>Take care when constructing the target URL of the rule, considering
601the security impact from allowing the client influence over the set of
602URLs to which your server will act as a proxy.  Ensure that the scheme
603and hostname part of the URL is either fixed, or does not allow the
604client undue influence.</p>
605</div>
606
607<div class="warning">
608<h3>Performance warning</h3>
609<p>Using this flag triggers the use of <code class="module"><a href="../mod/mod_proxy.html">mod_proxy</a></code>, without
610handling of persistent connections as the default worker is used in this case,
611which does not handle connection pooling/reuse.</p>
612<p>In order to use persistent connections you need to setup a
613<code class="directive"><a href="../mod/mod_proxy.html#proxy">Proxy</a></code> block at least for the scheme
614and host part of the target URL containing a
615<code class="directive"><a href="../mod/mod_proxy.html#proxyset">ProxySet</a></code> directive where you e.g. set
616a timeout.</p>
617<p>If you set it up with <code class="directive"><a href="../mod/mod_proxy.html#proxypass">ProxyPass</a></code> or
618<code class="directive"><a href="../mod/mod_proxy.html#proxypassmatch">ProxyPassMatch</a></code> persistent connections
619will be used automatically.</p>
620</div>
621
622<p>Note: <code class="module"><a href="../mod/mod_proxy.html">mod_proxy</a></code> must be enabled in order
623to use this flag.</p>
624
625</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>
626<div class="section">
627<h2><a name="flag_pt" id="flag_pt">PT|passthrough</a></h2>
628
629<p>
630The target (or substitution string) in a RewriteRule is assumed to be a
631file path, by default. The use of the [PT] flag causes it to be treated
632as a URI instead. That is to say, the
633use of the [PT] flag causes the result of the <code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">RewriteRule</a></code> to be passed back through
634URL mapping, so that location-based mappings, such as <code class="directive"><a href="../mod/mod_alias.html#alias">Alias</a></code>, <code class="directive"><a href="../mod/mod_alias.html#redirect">Redirect</a></code>, or <code class="directive"><a href="../mod/mod_alias.html#scriptalias">ScriptAlias</a></code>, for example, might have a
635chance to take effect.
636</p>
637
638<p>
639If, for example, you have an
640<code class="directive"><a href="../mod/mod_alias.html#alias">Alias</a></code>
641for /icons, and have a <code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">RewriteRule</a></code> pointing there, you should
642use the [PT] flag to ensure that the
643<code class="directive"><a href="../mod/mod_alias.html#alias">Alias</a></code> is evaluated.
644</p>
645
646<pre class="prettyprint lang-config">Alias "/icons" "/usr/local/apache/icons"
647RewriteRule "/pics/(.+)\.jpg$" "/icons/$1.gif" [PT]</pre>
648
649
650<p>
651Omission of the [PT] flag in this case will cause the Alias to be
652ignored, resulting in a 'File not found' error being returned.
653</p>
654
655<p>The <code>PT</code> flag implies the <code>L</code> flag:
656rewriting will be stopped in order to pass the request to
657the next phase of processing.</p>
658
659<p>Note that the <code>PT</code> flag is implied in per-directory
660contexts such as
661<code class="directive"><a href="../mod/core.html#directory">&lt;Directory&gt;</a></code> sections
662or in <code>.htaccess</code> files. The only way to circumvent that
663is to rewrite to <code>-</code>.</p>
664
665</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>
666<div class="section">
667<h2><a name="flag_qsa" id="flag_qsa">QSA|qsappend</a></h2>
668<p>
669When the replacement URI contains a query string, the default behavior
670of <code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">RewriteRule</a></code> is to discard
671the existing query string, and replace it with the newly generated one.
672Using the [QSA] flag causes the query strings to be combined.
673</p>
674
675<p>Consider the following rule:</p>
676
677<pre class="prettyprint lang-config">RewriteRule "/pages/(.+)" "/page.php?page=$1" [QSA]</pre>
678
679
680<p>With the [QSA] flag, a request for <code>/pages/123?one=two</code> will be
681mapped to <code>/page.php?page=123&amp;one=two</code>. Without the [QSA]
682flag, that same request will be mapped to
683<code>/page.php?page=123</code> - that is, the existing query string
684will be discarded.
685</p>
686</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>
687<div class="section">
688<h2><a name="flag_qsd" id="flag_qsd">QSD|qsdiscard</a></h2>
689<p>
690When the requested URI contains a query string, and the target URI does
691not, the default behavior of <code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">RewriteRule</a></code> is to copy that query
692string to the target URI. Using the [QSD] flag causes the query string
693to be discarded.
694</p>
695
696<p>This flag is available in version 2.4.0 and later.</p>
697
698<p>
699Using [QSD] and [QSA] together will result in [QSD] taking precedence.
700</p>
701
702<p>
703If the target URI has a query string, the default behavior will be
704observed - that is, the original query string will be discarded and
705replaced with the query string in the <code>RewriteRule</code> target
706URI.
707</p>
708
709</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>
710<div class="section">
711<h2><a name="flag_qsl" id="flag_qsl">QSL|qslast</a></h2>
712<p>
713By default, the first (left-most) question mark in the substitution
714delimits the path from the query string.  Using the [QSL] flag instructs
715<code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">RewriteRule</a></code> to instead split
716the two components using the last (right-most) question mark.  </p>
717
718<p>
719This is useful when mapping to files that have literal question marks in 
720their filename.  If no query string is used in the substitution, 
721a question mark can be appended to it in combination with this flag.  </p>
722
723<p> This flag is available in version 2.4.19 and later.</p>
724
725</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>
726<div class="section">
727<h2><a name="flag_r" id="flag_r">R|redirect</a></h2>
728<p>
729Use of the [R] flag causes a HTTP redirect to be issued to the browser.
730If a fully-qualified URL is specified (that is, including
731<code>http://servername/</code>) then a redirect will be issued to that
732location. Otherwise, the current protocol, servername, and port number
733will be used to generate the URL sent with the redirect.
734</p>
735
736<p>
737<em>Any</em> valid HTTP response  status code may be specified,
738using the syntax [R=305], with a 302 status code being used by
739default if none is specified. The status code specified need not
740necessarily be a redirect (3xx) status code. However,
741if a status code is outside the redirect range (300-399) then the
742substitution string is dropped entirely, and rewriting is stopped as if
743the <code>L</code> were used.</p>
744
745<p>In addition to response status codes, you may also specify redirect
746status using their symbolic names: <code>temp</code> (default),
747<code>permanent</code>, or <code>seeother</code>.</p>
748
749<p>
750You will almost always want to use [R] in conjunction with [L] (that is,
751use [R,L]) because on its own, the [R] flag prepends
752<code>http://thishost[:thisport]</code> to the URI, but then passes this
753on to the next rule in the ruleset, which can often result in 'Invalid
754URI in request' warnings.
755</p>
756
757<p>Note: httpd only supports status codes that are included in the HTTP
758spec.  Using an unrecognized status code will result in a 500 error and
759error log message.</p>
760
761</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>
762<div class="section">
763<h2><a name="flag_s" id="flag_s">S|skip</a></h2>
764<p>The [S] flag is used to skip rules that you don't want to run. The
765syntax of the skip flag is [S=<em>N</em>], where <em>N</em> signifies
766the number of rules to skip (provided the <code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">
767RewriteRule</a></code> and any preceding <code class="directive"><a href="../mod/mod_rewrite.html#rewritecond">
768RewriteCond</a></code> directives match). This can be thought of as a
769<code>goto</code> statement in your rewrite ruleset. In the following
770example, we only want to run the <code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">
771RewriteRule</a></code> if the requested URI doesn't correspond with an
772actual file.</p>
773
774<pre class="prettyprint lang-config"># Is the request for a non-existent file?
775RewriteCond "%{REQUEST_FILENAME}" !-f
776RewriteCond "%{REQUEST_FILENAME}" !-d
777# If so, skip these two RewriteRules
778RewriteRule ".?"                  "-" [S=2]
779
780RewriteRule "(.*\.gif)"           "images.php?$1"
781RewriteRule "(.*\.html)"          "docs.php?$1"</pre>
782
783
784<p>This technique is useful because a <code class="directive"><a href="../mod/mod_rewrite.html#rewritecond">RewriteCond</a></code> only applies to the
785<code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">RewriteRule</a></code> immediately
786following it. Thus, if you want to make a <code>RewriteCond</code> apply
787to several <code>RewriteRule</code>s, one possible technique is to
788negate those conditions and add a <code>RewriteRule</code> with a [Skip] flag. You can
789use this to make pseudo if-then-else constructs: The last rule of
790the then-clause becomes <code>skip=N</code>, where N is the
791number of rules in the else-clause:</p>
792<pre class="prettyprint lang-config"># Does the file exist?
793RewriteCond "%{REQUEST_FILENAME}" !-f
794RewriteCond "%{REQUEST_FILENAME}" !-d
795# Create an if-then-else construct by skipping 3 lines if we meant to go to the "else" stanza.
796RewriteRule ".?"                  "-" [S=3]
797
798# IF the file exists, then:
799    RewriteRule "(.*\.gif)"  "images.php?$1"
800    RewriteRule "(.*\.html)" "docs.php?$1"
801    # Skip past the "else" stanza.
802    RewriteRule ".?"         "-" [S=1]
803# ELSE...
804    RewriteRule "(.*)"       "404.php?file=$1"
805# END</pre>
806
807
808<p>It is probably easier to accomplish this kind of configuration using
809the <code class="directive">&lt;If&gt;</code>, <code class="directive">&lt;ElseIf&gt;</code>, and <code class="directive">&lt;Else&gt;</code> directives instead.</p>
810
811</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>
812<div class="section">
813<h2><a name="flag_t" id="flag_t">T|type</a></h2>
814<p>Sets the MIME type with which the resulting response will be
815sent. This has the same effect as the <code class="directive"><a href="../mod/mod_mime.html#addtype">AddType</a></code> directive.</p>
816
817<p>For example, you might use the following technique to serve Perl
818source code as plain text, if requested in a particular way:</p>
819
820<pre class="prettyprint lang-config"># Serve .pl files as plain text
821RewriteRule "\.pl$"  "-" [T=text/plain]</pre>
822
823
824<p>Or, perhaps, if you have a camera that produces jpeg images without
825file extensions, you could force those images to be served with the
826correct MIME type by virtue of their file names:</p>
827
828<pre class="prettyprint lang-config"># Files with 'IMG' in the name are jpg images.
829RewriteRule "IMG"  "-" [T=image/jpg]</pre>
830
831
832<p>Please note that this is a trivial example, and could be better done
833using <code class="directive"><a href="../mod/core.html#filesmatch">&lt;FilesMatch&gt;</a></code>
834instead. Always consider the alternate
835solutions to a problem before resorting to rewrite, which will
836invariably be a less efficient solution than the alternatives.</p>
837
838<p>
839If used in per-directory context, use only <code>-</code> (dash)
840as the substitution <em>for the entire round of <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code> processing</em>,
841otherwise the MIME-type set with this flag is lost due to an internal
842re-processing (including subsequent rounds of <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code> processing).
843The <code>L</code> flag can be useful in this context to end the
844<em>current</em> round of <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code> processing.</p>
