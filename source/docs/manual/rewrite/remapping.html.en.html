59<h2><a name="old-to-new" id="old-to-new">From Old to New (internal)</a></h2>
60
61  
62
63  <dl>
64    <dt>Description:</dt>
65
66    <dd>
67      <p>Assume we have recently renamed the page
68      <code>foo.html</code> to <code>bar.html</code> and now want
69      to provide the old URL for backward compatibility. However,
70      we want that users of the old URL even not recognize that
71      the pages was renamed - that is, we don't want the address to
72      change in their browser.</p>
73    </dd>
74
75    <dt>Solution:</dt>
76
77    <dd>
78      <p>We rewrite the old URL to the new one internally via the
79      following rule:</p>
80
81<pre class="prettyprint lang-config">RewriteEngine  on
82RewriteRule    "^<strong>/foo</strong>\.html$"  "<strong>/bar</strong>.html" [PT]</pre>
83
84    </dd>
85  </dl>

89<h2><a name="old-to-new-extern" id="old-to-new-extern">Rewriting From Old to New (external)</a></h2>
90
91  
92
93  <dl>
94    <dt>Description:</dt>
95
96    <dd>
97      <p>Assume again that we have recently renamed the page
98      <code>foo.html</code> to <code>bar.html</code> and now want
99      to provide the old URL for backward compatibility. But this
100      time we want that the users of the old URL get hinted to
101      the new one, i.e. their browsers Location field should
102      change, too.</p>
103    </dd>
104
105    <dt>Solution:</dt>
106
107    <dd>
108      <p>We force a HTTP redirect to the new URL which leads to a
109      change of the browsers and thus the users view:</p>
110
111<pre class="prettyprint lang-config">RewriteEngine  on
112RewriteRule    "^<strong>/foo</strong>\.html$"  "<strong>bar</strong>.html"  [<strong>R</strong>]</pre>
113
114</dd>
115
116<dt>Discussion</dt>
117
118    <dd>
119    <p>In this example, as contrasted to the <a href="#old-to-new-intern">internal</a> example above, we can simply
120    use the Redirect directive. <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code> was used in that earlier
121    example in order to hide the redirect from the client:</p>
122
123    <pre class="prettyprint lang-config">Redirect "/foo.html" "/bar.html"</pre>
124
125
126    </dd>
127  </dl>

131<h2><a name="movehomedirs" id="movehomedirs">Resource Moved to Another Server</a></h2>
132
133  
134
135  <dl>
136    <dt>Description:</dt>
137
138    <dd>
139      <p>If a resource has moved to another server, you may wish to have
140      URLs continue to work for a time on the old server while people
141      update their bookmarks.</p>
142    </dd>
143
144    <dt>Solution:</dt>
145
146    <dd>
147      <p>You can use <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code> to redirect these URLs
148      to the new server, but you might also consider using the Redirect
149      or RedirectMatch directive.</p>
150
151<pre class="prettyprint lang-config">#With mod_rewrite
152RewriteEngine on
153RewriteRule   "^/docs/(.+)"  "http://new.example.com/docs/$1"  [R,L]</pre>
154
155
156<pre class="prettyprint lang-config">#With RedirectMatch
157RedirectMatch "^/docs/(.*)" "http://new.example.com/docs/$1"</pre>
158
159
160<pre class="prettyprint lang-config">#With Redirect
161Redirect "/docs/" "http://new.example.com/docs/"</pre>
162
163    </dd>
164  </dl>

168<h2><a name="static-to-dynamic" id="static-to-dynamic">From Static to Dynamic</a></h2>
169
170  
171
172  <dl>
173    <dt>Description:</dt>
174
175    <dd>
176      <p>How can we transform a static page
177      <code>foo.html</code> into a dynamic variant
178      <code>foo.cgi</code> in a seamless way, i.e. without notice
179      by the browser/user.</p>
180    </dd>
181
182    <dt>Solution:</dt>
183
184    <dd>
185      <p>We just rewrite the URL to the CGI-script and force the
186      handler to be <strong>cgi-script</strong> so that it is
187      executed as a CGI program.
188      This way a request to <code>/~quux/foo.html</code>
189      internally leads to the invocation of
190      <code>/~quux/foo.cgi</code>.</p>
191
192<pre class="prettyprint lang-config">RewriteEngine  on
193RewriteBase    "/~quux/"
194RewriteRule    "^foo\.html$"  "foo.cgi"  [H=<strong>cgi-script</strong>]</pre>
195
196    </dd>
197  </dl>

201<h2><a name="backward-compatibility" id="backward-compatibility">Backward Compatibility for file extension change</a></h2>
202
203  
204
205  <dl>
206    <dt>Description:</dt>
207
208    <dd>
209      <p>How can we make URLs backward compatible (still
210      existing virtually) after migrating <code>document.YYYY</code>
211      to <code>document.XXXX</code>, e.g. after translating a
212      bunch of <code>.html</code> files to <code>.php</code>?</p>
213    </dd>
214
215    <dt>Solution:</dt>
216
217    <dd>
218      <p>The URL is rewritten from the old extension to the new
219      one only if the target file with the new extension exists
220      and the original file with the old extension does not.
221      Otherwise, the URL is left unchanged.</p>
222
223<pre class="prettyprint lang-config">#   backward compatibility ruleset for
224#   rewriting document.html to document.php
225#   when and only when document.php exists
226&lt;Directory "/var/www/htdocs"&gt;
227    RewriteEngine on
228    RewriteBase "/var/www/htdocs"
229
230    RewriteCond "$1.php" -f
231    RewriteCond "$1.html" !-f
232    RewriteRule "^(.*).html$" "$1.php"
233&lt;/Directory&gt;</pre>
234
235    </dd>
236
237    <dt>Discussion</dt>
238    <dd>
239    <p>This example uses an often-overlooked feature of <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code>,
240    by taking advantage of the order of execution of the ruleset. In
241    particular, <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code> evaluates the left-hand-side of the
242    RewriteRule before it evaluates the RewriteCond directives.
243    Consequently, $1 is already defined by the time the RewriteCond
244    directives are evaluated. This allows us to test for the existence
245    of the original (<code>document.html</code>) and target
246    (<code>document.php</code>) files using the same base filename.</p>
247
248    <p>This ruleset is designed to use in a per-directory context (In a
249    &lt;Directory&gt; block or in a .htaccess file), so that the
250    <code>-f</code> checks are looking at the correct directory path.
251    You may need to set a <code class="directive"><a href="../mod/mod_rewrite.html#rewritebase">RewriteBase</a></code> directive to specify the
252    directory base that you're working in.</p>
253    </dd>
254  </dl>

258<h2><a name="canonicalhost" id="canonicalhost">Canonical Hostnames</a></h2>
259
260
261
262      <dl>
263        <dt>Description:</dt>
264
265        <dd>The goal of this rule is to force the use of a particular
266        hostname, in preference to other hostnames which may be used to
267        reach the same site. For example, if you wish to force the use
268        of <strong>www.example.com</strong> instead of
269        <strong>example.com</strong>, you might use a variant of the
270        following recipe.</dd>
271
272        <dt>Solution:</dt>
273
274        <dd>
275
276<p>The very best way to solve this doesn't involve <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code> at all,
277but rather uses the <code class="directive"><a href="../mod/mod_alias.html#redirect">Redirect</a></code>
278directive placed in a virtual host for the non-canonical
279hostname(s).</p>
280
281<pre class="prettyprint lang-config">&lt;VirtualHost *:80&gt;
282  ServerName undesired.example.com
283  ServerAlias example.com notthis.example.com
284
285  Redirect "/" "http://www.example.com/"
286&lt;/VirtualHost&gt;
287
288&lt;VirtualHost *:80&gt;
289  ServerName www.example.com
290&lt;/VirtualHost&gt;</pre>
291
292
293<p>You can alternatively accomplish this using the
294<code class="directive"><a href="../mod/core.html#if">&lt;If&gt;</a></code>
295directive:</p>
296
297<pre class="prettyprint lang-config">&lt;If "%{HTTP_HOST} != 'www.example.com'"&gt;
298    Redirect "/" "http://www.example.com/"
299&lt;/If&gt;</pre>
300
301
302<p>Or, for example, to redirect a portion of your site to HTTPS, you
303might do the following:</p>
304
305<pre class="prettyprint lang-config">&lt;If "%{SERVER_PROTOCOL} != 'HTTPS'"&gt;
306    Redirect "/admin/" "https://www.example.com/admin/"
307&lt;/If&gt;</pre>
308
309
310<p>If, for whatever reason, you still want to use <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code>
311- if, for example, you need this to work with a larger set of RewriteRules -
312you might use one of the recipes below.</p>
313
314<p>For sites running on a port other than 80:</p>
315<pre class="prettyprint lang-config">RewriteCond "%{HTTP_HOST}"   "!^www\.example\.com" [NC]
316RewriteCond "%{HTTP_HOST}"   "!^$"
317RewriteCond "%{SERVER_PORT}" "!^80$"
318RewriteRule "^/?(.*)"        "http://www.example.com:%{SERVER_PORT}/$1" [L,R,NE]</pre>
319
320
321<p>And for a site running on port 80</p>
322<pre class="prettyprint lang-config">RewriteCond "%{HTTP_HOST}"   "!^www\.example\.com" [NC]
323RewriteCond "%{HTTP_HOST}"   "!^$"
324RewriteRule "^/?(.*)"        "http://www.example.com/$1" [L,R,NE]</pre>
325
326
327        <p>
328        If you wanted to do this generically for all domain names - that
329        is, if you want to redirect <strong>example.com</strong> to
330        <strong>www.example.com</strong> for all possible values of
331        <strong>example.com</strong>, you could use the following
332        recipe:</p>
333
334<pre class="prettyprint lang-config">RewriteCond "%{HTTP_HOST}" "!^www\." [NC]
335RewriteCond "%{HTTP_HOST}" "!^$"
336RewriteRule "^/?(.*)"      "http://www.%{HTTP_HOST}/$1" [L,R,NE]</pre>
337
338
339    <p>These rulesets will work either in your main server configuration
340    file, or in a <code>.htaccess</code> file placed in the <code class="directive"><a href="../mod/core.html#documentroot">DocumentRoot</a></code> of the server.</p>
341        </dd>
342      </dl>

346<h2><a name="multipledirs" id="multipledirs">Search for pages in more than one directory</a></h2>
347
348  
349
350  <dl>
351    <dt>Description:</dt>
352
353    <dd>
354      <p>A particular resource might exist in one of several places, and
355      we want to look in those places for the resource when it is
356      requested. Perhaps we've recently rearranged our directory
357      structure, dividing content into several locations.</p>
358    </dd>
359
360    <dt>Solution:</dt>
361
362    <dd>
363      <p>The following ruleset searches in two directories to find the
364      resource, and, if not finding it in either place, will attempt to
365      just serve it out of the location requested.</p>
366
367<pre class="prettyprint lang-config">RewriteEngine on
368
369#   first try to find it in dir1/...
370#   ...and if found stop and be happy:
371RewriteCond         "%{DOCUMENT_ROOT}/<strong>dir1</strong>/%{REQUEST_URI}"  -f
372RewriteRule "^(.+)" "%{DOCUMENT_ROOT}/<strong>dir1</strong>/$1"  [L]
373
374#   second try to find it in dir2/...
375#   ...and if found stop and be happy:
376RewriteCond         "%{DOCUMENT_ROOT}/<strong>dir2</strong>/%{REQUEST_URI}"  -f
377RewriteRule "^(.+)" "%{DOCUMENT_ROOT}/<strong>dir2</strong>/$1"  [L]
378
379#   else go on for other Alias or ScriptAlias directives,
380#   etc.
381RewriteRule   "^"  "-"  [PT]</pre>
382
383    </dd>
384  </dl>

388<h2><a name="archive-access-multiplexer" id="archive-access-multiplexer">Redirecting to Geographically Distributed Servers</a></h2>
389
390  
391
392  <dl>
393    <dt>Description:</dt>
394
395    <dd>
396    <p>We have numerous mirrors of our website, and want to redirect
397    people to the one that is located in the country where they are
398    located.</p>
399    </dd>
400
401    <dt>Solution:</dt>
402
403    <dd>
404    <p>Looking at the hostname of the requesting client, we determine
405    which country they are coming from. If we can't do a lookup on their
406    IP address, we fall back to a default server.</p>
407    <p>We'll use a <code class="directive"><a href="../mod/mod_rewrite.html#rewritemap">RewriteMap</a></code>
408    directive to build a list of servers that we wish to use.</p>
409
410<pre class="prettyprint lang-config">HostnameLookups on
411RewriteEngine on
412RewriteMap    multiplex         "txt:/path/to/map.mirrors"
413RewriteCond   "%{REMOTE_HOST}"  "([a-z]+)$" [NC]
414RewriteRule   "^/(.*)$"  "${multiplex:<strong>%1</strong>|http://www.example.com/}$1"  [R,L]</pre>
415
416
417<div class="example"><p><code>
418##  map.mirrors -- Multiplexing Map<br />
419<br />
420de        http://www.example.de/<br />
421uk        http://www.example.uk/<br />
422com       http://www.example.com/<br />
423##EOF##
424</code></p></div>
425    </dd>
426
427    <dt>Discussion</dt>
428    <dd>
429    <div class="warning">This ruleset relies on
430    <code class="directive"><a href="../mod/core.html#hostnamelookups">HostNameLookups</a></code>
431    being set <code>on</code>, which can be
432    a significant performance hit.</div>
433
434    <p>The <code class="directive"><a href="../mod/mod_rewrite.html#rewritecond">RewriteCond</a></code>
435    directive captures the last portion of the hostname of the
436    requesting client - the country code - and the following RewriteRule
437    uses that value to look up the appropriate mirror host in the map
438    file.</p>
439    </dd>
440  </dl>

444<h2><a name="browser-dependent-content" id="browser-dependent-content">Browser Dependent Content</a></h2>
445
446  
447
448  <dl>
449    <dt>Description:</dt>
450
451    <dd>
452      <p>We wish to provide different content based on the browser, or
453      user-agent, which is requesting the content.</p>
454    </dd>
455
456    <dt>Solution:</dt>
457
458    <dd>
459      <p>We have to decide, based on the HTTP header "User-Agent",
460      which content to serve. The following config
461      does the following: If the HTTP header "User-Agent"
462      contains "Mozilla/3", the page <code>foo.html</code>
463      is rewritten to <code>foo.NS.html</code> and the
464      rewriting stops. If the browser is "Lynx" or "Mozilla" of
465      version 1 or 2, the URL becomes <code>foo.20.html</code>.
466      All other browsers receive page <code>foo.32.html</code>.
467      This is done with the following ruleset:</p>
468
469<pre class="prettyprint lang-config">RewriteCond "%{HTTP_USER_AGENT}"  "^<strong>Mozilla/3</strong>.*"
470RewriteRule "^foo\.html$"         "foo.<strong>NS</strong>.html"          [<strong>L</strong>]
471
472RewriteCond "%{HTTP_USER_AGENT}"  "^Lynx/" [OR]
473RewriteCond "%{HTTP_USER_AGENT}"  "^Mozilla/[12]"
474RewriteRule "^foo\.html$"         "foo.<strong>20</strong>.html"          [<strong>L</strong>]
475
476RewriteRule "^foo\.html$"         "foo.<strong>32</strong>.html"          [<strong>L</strong>]</pre>
477
478    </dd>
479  </dl>

482<div class="section">
483<h2><a name="canonicalurl" id="canonicalurl">Canonical URLs</a></h2>
484
485
486
487<dl>
488 <dt>Description:</dt>
489
490   <dd>
491     <p>On some webservers there is more than one URL for a
492     resource. Usually there are canonical URLs (which are be
493     actually used and distributed) and those which are just
494     shortcuts, internal ones, and so on. Independent of which URL the
495     user supplied with the request, they should finally see the
496     canonical one in their browser address bar.</p>
497   </dd>
498
499   <dt>Solution:</dt>
500
501     <dd>
502       <p>We do an external HTTP redirect for all non-canonical
503       URLs to fix them in the location view of the Browser and
504       for all subsequent requests. In the example ruleset below
505       we replace <code>/puppies</code> and <code>/canines</code>
506       by the canonical <code>/dogs</code>.</p>
507
508<pre class="prettyprint lang-config">RewriteRule   "^/(puppies|canines)/(.*)"    "/dogs/$2"  [R]</pre>
509
510        </dd>
511
512     <dt>Discussion:</dt>
513     <dd>
514     This should really be accomplished with Redirect or RedirectMatch
515     directives:
516
517     <pre class="prettyprint lang-config">RedirectMatch "^/(puppies|canines)/(.*)" "/dogs/$2"</pre>
518
519     </dd>
520      </dl>

524<h2><a name="moveddocroot" id="moveddocroot">Moved <code>DocumentRoot</code></a></h2>
525
526  
527
528  <dl>
529    <dt>Description:</dt>
530
531    <dd>
532<p>Usually the <code class="directive"><a href="../mod/core.html#documentroot">DocumentRoot</a></code>
533of the webserver directly relates to the URL "<code>/</code>".
534But often this data is not really of top-level priority. For example,
535you may wish for visitors, on first entering a site, to go to a
536particular subdirectory <code>/about/</code>. This may be accomplished
537using the following ruleset:</p>
538</dd>
539
540    <dt>Solution:</dt>
541
542    <dd>
543      <p>We redirect the URL <code>/</code> to
544      <code>/about/</code>:
545      </p>
546
547<pre class="prettyprint lang-config">RewriteEngine on
548RewriteRule   "^/$"  "/about/"  [<strong>R</strong>]</pre>
549
550
551<p>Note that this can also be handled using the <code class="directive"><a href="../mod/mod_alias.html#redirectmatch">RedirectMatch</a></code> directive:</p>
552
553<pre class="prettyprint lang-config">RedirectMatch "^/$" "http://example.com/about/"</pre>
554
555
556<p>Note also that the example rewrites only the root URL. That is, it
557rewrites a request for <code>http://example.com/</code>, but not a
558request for <code>http://example.com/page.html</code>. If you have in
559fact changed your document root - that is, if <strong>all</strong> of
560your content is in fact in that subdirectory, it is greatly preferable
561to simply change your <code class="directive"><a href="../mod/core.html#documentroot">DocumentRoot</a></code>
562directive, or move all of the content up one directory,
563rather than rewriting URLs.</p>
564</dd>
565</dl>
566
567</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>
568<div class="section">
569<h2><a name="fallback-resource" id="fallback-resource">Fallback Resource</a></h2>
570
571
572<dl>
573<dt>Description:</dt>
574<dd>You want a single resource (say, a certain file, like index.php) to
575handle all requests that come to a particular directory, except those
576that should go to an existing resource such as an image, or a css file.</dd>
577
578<dt>Solution:</dt>
579<dd>
580<p>As of version 2.2.16, you should use the <code class="directive"><a href="../mod/mod_dir.html#fallbackresource">FallbackResource</a></code> directive for this:</p>
581
582<pre class="prettyprint lang-config">&lt;Directory "/var/www/my_blog"&gt;
583  FallbackResource "index.php"
584&lt;/Directory&gt;</pre>
585
586
587<p>However, in earlier versions of Apache, or if your needs are more
588complicated than this, you can use a variation of the following rewrite
589set to accomplish the same thing:</p>
590
591<pre class="prettyprint lang-config">&lt;Directory "/var/www/my_blog"&gt;
592  RewriteBase "/my_blog"
593
594  RewriteCond "/var/www/my_blog/%{REQUEST_FILENAME}" !-f
595  RewriteCond "/var/www/my_blog/%{REQUEST_FILENAME}" !-d
596  RewriteRule "^" "index.php" [PT]
597&lt;/Directory&gt;</pre>
598
599
600<p>If, on the other hand, you wish to pass the requested URI as a query
601string argument to index.php, you can replace that RewriteRule with:</p>
602
603<pre class="prettyprint lang-config">RewriteRule "(.*)" "index.php?$1" [PT,QSA]</pre>
604
605
606<p>Note that these rulesets can be used in a <code>.htaccess</code>
607file, as well as in a &lt;Directory&gt; block.</p>
608
609</dd>
610
611</dl>

614<div class="section">
615<h2><a name="rewrite-query" id="rewrite-query">Rewrite query string</a></h2>
616
617
618<dl>
619<dt>Description:</dt>
620<dd>You want to capture a particular value from a query string
621and either replace it or incorporate it into another component
622of the URL.</dd>
623
624<dt>Solutions:</dt>
625<dd>
626<p> Many of the solutions in this section will all use the same condition,
627which leaves the matched value in the %2 backreference.  %1 is the beginining
628of the query string (up to the key of intererest), and %3 is the remainder. This
629condition is a bit complex for flexibility and to avoid double '&amp;&amp;' in the
630substitutions.</p>
631<ul>
632  <li>This solution removes the matching key and value:
633
634<pre class="prettyprint lang-config"># Remove mykey=???
635RewriteCond "%{QUERY_STRING}" "(.*(?:^|&amp;))mykey=([^&amp;]*)&amp;?(.*)&amp;?$"
636RewriteRule "(.*)" "$1?%1%3"</pre>
637
638  </li>
639
640  <li>This solution uses the captured value in the URL substitution,
641  discarding the rest of the original query by appending a '?':
642
643<pre class="prettyprint lang-config"># Copy from query string to PATH_INFO
644RewriteCond "%{QUERY_STRING}" "(.*(?:^|&amp;))mykey=([^&amp;]*)&amp;?(.*)&amp;?$"
645RewriteRule "(.*)" "$1/products/%2/?" [PT]</pre>
646
647  </li>
648
649  <li>This solution checks the captured value in a subsequent condition:
650
651<pre class="prettyprint lang-config"># Capture the value of mykey in the query string
652RewriteCond "%{QUERY_STRING}" "(.*(?:^|&amp;))mykey=([^&amp;]*)&amp;?(.*)&amp;?$"
653RewriteCond "%2" !=not-so-secret-value 
654RewriteRule "(.*)" - [F]</pre>
655
656  </li>
657
658  <li>This solution shows the reverse of the previous ones, copying
659      path components (perhaps PATH_INFO) from the URL into the query string.
660<pre class="prettyprint lang-config"># The desired URL might be /products/kitchen-sink, and the script expects
661# /path?products=kitchen-sink.
662RewriteRule "^/?path/([^/]+)/([^/]+)" "/path?$1=$2" [PT]</pre>
663
664  </li>
665</ul>
666
667</dd>
668
669</dl>
