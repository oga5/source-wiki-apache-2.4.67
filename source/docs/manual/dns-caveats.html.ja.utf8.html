37    <p>本文書の内容は次の一言に尽きます。「Apache が設定ファイルを読み込むときに
38    DNS を使用する必要がないようにして下さい」。Apache が設定ファイルを
39    読み込むときに DNS を使用する必要がある場合、信頼性の問題
40    (起動しないかもしれません) やサービス拒否や盗用アタック
41    (他のユーザからヒットを盗むことを含みます)
42    の問題に直面するかもしれません。</p>

52<h2><a name="example" id="example">簡単な例</a></h2>
53    
54
55    <div class="example"><p><code>
56      &lt;VirtualHost www.abc.dom&gt; <br />
57      ServerAdmin webgirl@abc.dom <br />
58      DocumentRoot /www/abc <br />
59      &lt;/VirtualHost&gt;
60    </code></p></div>
61
62    <p>Apache が正常に機能するには、バーチャルホスト毎に必ず二つの
63    情報が必要になります。それは、
64    <code class="directive"><a href="./mod/core.html#servername">ServerName</a></code>
65    と、そのサーバが応答するための IP (最低一つ) です。
66    上記例では IP アドレスを含んでいませんので、Apache は DNS
67    を使用して <code>www.abc.dom</code> を見つけなければなりません。
68    何らかの理由で設定ファイルを読み込んでいるときに DNS 
69    が利用できなかった場合、
70    バーチャルホストは<strong>設定されません</strong>。
71    そして、そのバーチャルホストに対するヒットには応答がなされません
72    (Apache 1.2 以前では起動すらしません)。</p>
73
74    <p><code>www.abc.dom</code> のアドレスが 192.0.2.1
75    だとします。では、次の設定について考えてみましょう。</p>
76
77    <div class="example"><p><code>
78      &lt;VirtualHost 192.0.2.1&gt; <br />
79      ServerAdmin webgirl@abc.dom <br />
80      DocumentRoot /www/abc <br />
81      &lt;/VirtualHost&gt;
82    </code></p></div>
83
84    <p>現在のリリースでは Apache は DNS 逆引きを使用して
85    このバーチャルホストの <code class="directive"><a href="./mod/core.html#servername">ServerName</a></code>
86    を見つけます。
87    その逆引きが失敗した場合は部分的にバーチャルホストを無効にします
88    (Apache 1.2 より前では起動すらしません)。
89    バーチャルホストが名前ベースであれば完全に無効になりますが、
90    IP ベースであれば概ね動作します。しかしながら、サーバ名を
91    含む完全な URL を生成しなければならない場合は、正しい URL
92    の生成ができません。</p>
93
94    <p>次の例は上記の問題を解決しています。</p>
95
96    <div class="example"><p><code>
97      &lt;VirtualHost 192.0.2.1&gt; <br />
98      ServerName www.abc.dom <br />
99      ServerAdmin webgirl@abc.dom <br />
100      DocumentRoot /www/abc <br />
101      &lt;/VirtualHost&gt;
102    </code></p></div>

105<h2><a name="denial" id="denial">サービス拒否</a></h2>
106    
107
108    <p>サービス拒否が起こる場合、(少なくとも) 二つのケースがあります。
109    Apache 1.2 より前を実行している場合、バーチャルホストのための
110    上記の二つの DNS 検索のうち一つ失敗すれば起動すらしません。
111    そしてこの DNS 検索が自分の制御下にすらない場合もありえます。
112    例えば、<code>abc.dom</code> が顧客のサーバの一つで、
113    DNS は顧客自身で管理している場合、単に
114    <code>www.abc.dom</code> レコードを削除するだけで、
115    (1.2 より前の) サーバを起動不能にすることができます。</p>
116
117    <p>もう一つのケースは、より気付きにくいものです。
118    次の設定について考えてみましょう。</p>
119
120    <div class="example"><p><code>
121      &lt;VirtualHost www.abc.dom&gt; <br />
122      <span class="indent">
123        ServerAdmin webgirl@abc.dom <br />
124        DocumentRoot /www/abc <br />
125      </span>
126      &lt;/VirtualHost&gt; <br />
127      <br />
128      &lt;VirtualHost www.def.dom&gt; <br />
129      <span class="indent">
130        ServerAdmin webguy@def.dom <br />
131        DocumentRoot /www/def <br />
132      </span>
133      &lt;/VirtualHost&gt;
134    </code></p></div>
135
136    <p>192.0.2.1 を <code>www.abc.dom</code> に、
137    192.0.2.2 を <code>www.def.dom</code> に割り当てているとします。
138    また、<code>def.dom</code> は顧客自身の DNS
139    の制御下にあるとします。この設定で、<code>abc.dom</code>
140    に向けられたトラフィック全てを奪うことができる位置に
141    <code>def.dom</code> を設置できています。後は単に 
142    <code>www.def.dom</code> が 192.0.2.1 を参照するように
143    設定するだけです。DNS は顧客側の DNS でコントロールされているので、
144    <code>www.def.dom</code> レコードが好きな場所を指すように
145    設定できてしまうのを止めさせることができません。</p>
146
147    <p>192.0.2.1 に対するリクエスト
148    (<code>http://www.abc.dom/whatever</code> 形式の URL 
149    を入力したユーザからのもの全てを含みます) 
150    は、<code>def.dom</code> バーチャルホストで応答されます。
151    このようなことが何故起こるかもっと良く知るためには、
152    応答の必要なバーチャルホストへのリクエストに対して、
153    Apache がどのように整合性を確保するかについて、
154    深い議論が必要になります。おおざっぱな説明は<a href="vhosts/details.html">こちら</a>に記述されています。</p>
155  </div><div class="top"><a href="#page-header"><img alt="top" src="./images/up.gif" /></a></div>

157<h2><a name="main" id="main">「主サーバ」アドレス</a></h2>
158    
159
160    <p>Apache 1.1 での <a href="vhost/name-based.html">名前ベースのバーチャルホストのサポート</a> 追加の際に、
161    Apache は <code class="program"><a href="./programs/httpd.html">httpd</a></code> の実行されているホストの IP
162    アドレスを知る必要が出てきました。このアドレスを得るために、
163    (もしあれば) グローバルな
164    <code class="directive"><a href="./mod/core.html#servername">ServerName</a></code> を使用するか、
165    C 言語の関数 <code>gethostname</code> (コマンドプロンプトで
166    <code>hostname</code> とタイプしたときと同じものを返します)
167    を呼び出すかをします。
168    その後、得られたアドレスで DNS 検索を行ないます。
169    現在のところ、この DNS 検索を回避する方法はありません。</p>
170
171    <p>DNS サーバがダウンして、この検索ができない事態が起こることを
172    恐れているのであれば、<code>/etc/hosts</code>
173    にホスト名を記述しておくことができます
174    (マシンが正常に起動するように既に設定されているかもしれません)。
175    その場合、DNS 参照が失敗した場合にマシンが <code>/etc/hosts</code>
176    を使用するように設定していることを確認してください。
177    その方法は、どの OS を使用しているかに依存しますが、
178    <code>/etc/resolv.conf</code> か <code>/etc/nsswitch.conf</code>
179    を編集することで設定できます。</p>
180
181    <p>もし他の理由で DNS を利用する必要がない場合は、
182    <code>HOSTRESORDER</code> 環境変数を「 <code>local</code>
183    」に設定することでそのようにできます。以上これらの事柄は、どんな
184    OS 、レゾルバライブラリを使用しているかに依存します。また、
185    <code class="module"><a href="./mod/mod_env.html">mod_env</a></code> を使用して環境変数を制御しない限り、
186    CGI にも影響を与えます。man ページや使用している OS
187    の FAQ で調べると良いでしょう。</p>

190<h2><a name="tips" id="tips">以上の問題を解決する方法</a></h2>
191    
192
193    <ul>
194      <li>
195        <code class="directive"><a href="./mod/core.html#virtualhost">VirtualHost</a></code>
196        で IP アドレスを使用する。
197      </li>
198
199      <li>
200        <code class="directive"><a href="./mod/mpm_common.html#listen">Listen</a></code>
201        で IP アドレスを使用する。
202      </li>
203
204      <li>
205        全てのバーチャルホストが明示的に
206        <code class="directive"><a href="./mod/core.html#servername">ServerName</a></code>
207        を持つようにする。
208      </li>
209
210      <li>何も応答しない
211      <code>&lt;VirtualHost _default_:*&gt;</code>
212      サーバを作る。</li>
213    </ul>

216<h2><a name="appendix" id="appendix">付録: 将来的な方向性</a></h2>
217    
218
219    <p>DNS に関して、現状は全く宜しくありません。Apache 1.2 で、
220    DNS のイベントが失敗しても少なくとも起動プロセスが続くようにしましたが、
221    これが最高の解決方法ではないでしょう。アドレスの再割り当てが必要不可避
222    となっている今日のインターネットにおいては、
223    設定ファイルの中で明示的な IP アドレスを要求する仕様は、
224    全く宜しくありません。</p>
225
226    <p>盗用のサービスアタックに関して行なうべき事は、
227    DNS 順引きを行なって得られたアドレスに対する DNS 
228    逆引きを行なって、二つの名前を比較することです。
229    この二つが一致しなければバーチャルホストは無効になるようにします。
230    こうするためには逆引き DNS が適切に設定されている必要があります
231    (FTP サーバや TCP ラッパーのおかげで「二重逆引き」DNS は一般的に
232    なっていますので、管理者にはお馴染みものでしょう)。</p>
233
234    <p>IP アドレスが使用されていなくて DNS が失敗した場合は、
235    どうしてもバーチャルホストウェブサーバを信頼性を確保して
236    起動させることは不可能のようです。
237    設定の一部を無効にするというような部分的な解決では、
238    サーバが何をするようにするかにもよりますが、
239    そのサーバが起動しないより確実に悪い状況になるでしょう。</p>
240
241    <p>HTTP/1.1 が開発され、ブラウザやプロキシが <code>Host</code>
242    ヘッダを発行するようになったので、IP ベースのバーチャルホストを
243    全く使用しなくても済むようになるかもしれません。
244    この場合、ウェブサーバは設定中に DNS 参照をしなくても済みます。
245    しかし 1997 年 3 月時点の状況では、
246    商用レベルのウェブサーバで使用できるほどには、
247    これらの機能は広く開発が進んでいません。</p>
