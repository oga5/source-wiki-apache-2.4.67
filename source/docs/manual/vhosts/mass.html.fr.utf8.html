58<h2><a name="motivation" id="motivation">A qui ce document est-il destiné ?</a></h2>
59
60    <p>Les techniques décrites ici vous concernent si votre
61    <code>httpd.conf</code> contient de nombreuses sections
62    <code>&lt;VirtualHost&gt;</code> très semblables,
63    dans le style :</p>
64
65<pre class="prettyprint lang-config">&lt;VirtualHost 111.22.33.44&gt;
66    ServerName                 customer-1.example.com
67    DocumentRoot        "/www/hosts/customer-1.example.com/docs"
68    ScriptAlias  "/cgi-bin/" "/www/hosts/customer-1.example.com/cgi-bin"
69&lt;/VirtualHost&gt;
70
71&lt;VirtualHost 111.22.33.44&gt;
72    ServerName                 customer-2.example.com
73    DocumentRoot        "/www/hosts/customer-2.example.com/docs"
74    ScriptAlias  "/cgi-bin/" "/www/hosts/customer-2.example.com/cgi-bin"
75&lt;/VirtualHost&gt;
76
77&lt;VirtualHost 111.22.33.44&gt;
78    ServerName                 customer-N.example.com
79    DocumentRoot        "/www/hosts/customer-N.example.com/docs"
80    ScriptAlias  "/cgi-bin/" "/www/hosts/customer-N.example.com/cgi-bin"
81&lt;/VirtualHost&gt;</pre>
82
83
84    <p>Nous voulons remplacer toutes les configurations
85    <code>&lt;VirtualHost&gt;</code> par un mécanisme qui les génère
86    dynamiquement. Ceci présente certains avantages :</p>
87
88    <ol>
89      <li>Votre fichier de configuration est plus petit, ainsi Apache
90      démarre plus rapidement et consomme moins de mémoire. Et ce qui
91      est peut-être le plus important, le fichier de configuration plus
92      petit est plus facile à maintenir, et le risque d'erreurs en est
93      diminué d'autant.
94      </li>
95
96      <li>Pour ajouter des serveurs virtuels, il suffit de créer les
97      répertoires appropriés dans le système de fichiers et les entrées
98      dans le DNS - il n'est plus nécessaire de reconfigurer ou de
99      redémarrer Apache.</li>
100    </ol>
101
102    <p>Le principal désavantage réside dans le fait que vous ne pouvez
103    pas définir un fichier journal différent pour chaque serveur
104    virtuel. De toute façon, ce serait une mauvaise idée si vous avez de
105    nombreux serveurs virtuels, car cela nécessiterait un <a href="fd-limits.html">nombre important de descripteurs de
106    fichier</a>. Il est préférable de rediriger <a href="../logs.html#piped">les journaux via un pipe ou
107    une file fifo</a> vers un
108    programme, et faire en sorte que ce dernier éclate les journaux
109    en un journal par serveur virtuel. L'utilitaire <a href="../programs/split-logfile.html">split-logfile</a>
110    constitue un exemple de ce traitement.</p>

114<h2><a name="overview" id="overview">Vue d'ensemble</a></h2>
115
116    <p>Un serveur virtuel peut être défini par deux informations : son
117    adresse IP, et le contenu de l'en-tête <code>Host:</code> de la
118    requête HTTP. La technique d'hébergement virtuel dynamique de masse
119    utilisée ici consiste à insérer automatiquement ces informations
120    dans le chemin du fichier à utiliser pour répondre à la requête. On
121    peut y parvenir assez facilement en utilisant
122    <code class="module"><a href="../mod/mod_vhost_alias.html">mod_vhost_alias</a></code> avec Apache httpd, mais on peut aussi
123    <a href="../rewrite/vhosts.html">utiliser mod_rewrite</a>. </p>
124    <p>Par défaut, ces deux modules
125    sont désactivés ; vous devez activer l'un d'eux lors de la
126    compilation et de la configuration d'Apache httpd si vous voulez utiliser
127    cette technique.</p>
128
129    <p>Certains paramètres doivent être extraits de la requête pour que le serveur
130    dynamique se présente comme un serveur dynamique normal. Le plus
131    important est le nom du serveur, que le serveur utilise pour générer des
132    URLs d'auto-référencement, etc... Il est défini via la directive
133    <code>ServerName</code>, et les CGIs peuvent s'y référer via la
134    variable d'environnement <code>SERVER_NAME</code>. Sa véritable
135    valeur utilisée à l'exécution est contrôlée par la définition de la
136    directive
137    <code class="directive"><a href="../mod/core.html#usecanonicalname">UseCanonicalName</a></code>. Avec
138    <code>UseCanonicalName Off</code>, le nom du serveur correspond au
139    contenu de l'en-tête <code>Host:</code> de la requête. Avec
140    <code>UseCanonicalName DNS</code>, il est extrait d'une recherche
141    DNS inverse sur l'adresse IP du serveur virtuel. La première
142    configuration est utilisée pour l'hébergement virtuel dynamique par
143    nom, et la deuxième pour l'hébergement virtuel dynamique par IP. Si
144    httpd ne peut pas déterminer le nom du serveur, soit parce qu'il
145    n'y a pas d'en-tête <code>Host:</code>, soit parce que la recherche
146    DNS a échoué, il prend en compte la valeur définie par la directive
147    <code>ServerName</code>.</p>
148
149    <p>L'autre paramètre à extraire est la racine des documents (définie
150    via la directive <code>DocumentRoot</code> et disponible pour les
151    scripts CGI via la variable d'environnement <code>DOCUMENT_ROOT</code>).
152    Dans une configuration classique, il est utilisé par le module core
153    pour faire correspondre les URIs aux noms de fichiers, mais lorsque
154    la configuration du serveur comporte des serveurs virtuels, ce
155    traitement doit être pris en charge par un autre module (soit
156    <code class="module"><a href="../mod/mod_vhost_alias.html">mod_vhost_alias</a></code>, soit <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code>), qui
157    utilise un méthode de correspondance différente. Aucun de ces
158    modules ne se chargeant de définir la variable d'environnement
159    <code>DOCUMENT_ROOT</code>, si des CGIs ou des documents SSI
160    doivent en faire usage, ils obtiendront une valeur erronée.</p>

164<h2><a name="simple" id="simple">Hébergement virtuel
165dynamique avec mod_vhost_alias</a></h2>
166
167    <p>Cet extrait de fichier <code>httpd.conf</code> implémente
168    l'hébergement virtuel décrit dans la section <a href="#motivation">À qui ce document est-il destiné ?</a> ci-dessus
169    en utilisant <code class="module"><a href="../mod/mod_vhost_alias.html">mod_vhost_alias</a></code>.</p>
170
171<pre class="prettyprint lang-config"># extrait le nom du serveur de l'en-tête Host:
172UseCanonicalName Off
173
174# ce format de journal peut être éclaté en journaux par serveur virtuel
175# à l'aide du premier champ via l'utilitaire split-logfile
176LogFormat "%V %h %l %u %t \"%r\" %s %b" vcommon
177CustomLog "logs/access_log" vcommon
178
179# inclut le nom du serveur dans les noms de fichiers ressources
180# nécessaires aux traitements des requêtes
181VirtualDocumentRoot "/www/hosts/%0/docs"
182VirtualScriptAlias  "/www/hosts/%0/cgi-bin"</pre>
183
184
185    <p>Pour changer cette configuration en solution de serveur virtuel
186    par IP, il suffit de remplacer <code>UseCanonicalName
187    Off</code> par <code>UseCanonicalName DNS</code>. Le nom du serveur
188    inséré dans le nom de fichier sera alors déduit de l'adresse IP du
189    serveur virtuel. La variable <code>%0</code> fait référence au nom
190    de serveur de la requête, tel qu'il est indiqué dans l'en-tête
191    <code>Host:</code>.</p>
192
193    <p>Voir la documentation du module <code class="module"><a href="../mod/mod_vhost_alias.html">mod_vhost_alias</a></code>
194    pour d'avantages d'exemples d'utilisation.</p>

198<h2><a name="homepages" id="homepages">Système de serveurs virtuels dynamiques
199simplifié</a></h2>
200
201    <p>Il s'agit d'une adaptation du système ci-dessus, ajusté pour un
202    serveur d'hébergement web de FAI. Grâce à la variable
203    <code>%2</code>, on peut extraire des sous-chaînes de caractères du
204    nom du serveur pour les utiliser dans le nom de fichier afin, par
205    exemple, de définir <code>/home/user/www</code> comme emplacement des
206    documents pour <code>www.user.example.com</code>. Un seul répertoire
207    <code>cgi-bin</code> suffit pour l'ensemble des
208    serveurs virtuels.</p>
209
210<pre class="prettyprint lang-config">UseCanonicalName Off
211
212LogFormat "%V %h %l %u %t \"%r\" %s %b" vcommon
213CustomLog "logs/access_log" vcommon
214
215# insertion d'une partie du nom du serveur dans les noms de fichiers
216VirtualDocumentRoot "/home/%2/www"
217
218# répertoire cgi-bin unique
219ScriptAlias  "/cgi-bin/"  "/www/std-cgi/"</pre>
220
221
222    <p>Vous trouverez des exemples plus élaborés d'utilisation de la
223    directive <code>VirtualDocumentRoot</code> dans la documentation du
224    module <code class="module"><a href="../mod/mod_vhost_alias.html">mod_vhost_alias</a></code>.</p>

228<h2><a name="combinations" id="combinations">Utiliser plusieurs systèmes
229d'hébergement virtuel sur le même serveur</a></h2>
230
231    <p>Moyennant une configuration un peu plus compliquée, vous pouvez
232    contrôler la portée des différentes configurations d'hébergement
233    virtuel à l'aide des directives <code>&lt;VirtualHost&gt;</code>
234    normales de httpd. Par exemple, on peut associer une adresse IP pour
235    les pages d'accueil des clients en général, et une autre pour les
236    clients commerciaux avec la configuration suivante. Cette
237    configuration peut être combinée avec les sections
238    <code>&lt;VirtualHost&gt;</code> conventionnelles, comme indiqué
239    plus loin.</p>
240
241<pre class="prettyprint lang-config">UseCanonicalName Off
242
243LogFormat "%V %h %l %u %t \"%r\" %s %b" vcommon
244
245&lt;Directory "/www/commercial"&gt;
246    Options FollowSymLinks
247    AllowOverride All
248&lt;/Directory&gt;
249
250&lt;Directory "/www/homepages"&gt;
251    Options FollowSymLinks
252    AllowOverride None
253&lt;/Directory&gt;
254
255&lt;VirtualHost 111.22.33.44&gt;
256    ServerName www.commercial.example.com
257    
258    CustomLog "logs/access_log.commercial" vcommon
259    
260    VirtualDocumentRoot "/www/commercial/%0/docs"
261    VirtualScriptAlias  "/www/commercial/%0/cgi-bin"
262&lt;/VirtualHost&gt;
263
264&lt;VirtualHost 111.22.33.45&gt;
265    ServerName www.homepages.example.com
266    
267    CustomLog "logs/access_log.homepages" vcommon
268    
269    VirtualDocumentRoot "/www/homepages/%0/docs"
270    ScriptAlias         "/cgi-bin/" "/www/std-cgi/"
271&lt;/VirtualHost&gt;</pre>
272
273
274<div class="note">
275	<h3>Note</h3>
276	<p>Si le premier bloc VirtualHost ne comporte <em>pas</em> de
277	directive <code class="directive"><a href="../mod/core.html#servername">ServerName</a></code>, c'est
278	le nom issu d'une recherche DNS inverse à partir de l'adresse IP
279	du serveur virtuel qui sera utilisé. Si ce nom ne correspond pas
280	à celui que vous voulez utiliser, vous pouvez ajouter une entrée
281	de remplacement (par exemple <code>ServerName
282	none.example.com</code>) pour éviter ce comportement.</p>

286<div class="section">
287<h2><a name="ipbased" id="ipbased">Pour un hébergement virtuel par IP plus
288efficace</a></h2>
289
290    <p>Les changements de configuration suggérés pour transformer <a href="#simple">le premier exemple</a> en hébergement virtuel par IP
291    conduisent à une configuration peu efficace. Chaque requête
292    nécessite une nouvelle recherche DNS. Pour éviter cette surcharge de
293    travail, le système de fichiers peut être organisé pour correspondre
294    aux adresses IP, plutôt qu'aux noms de serveurs, supprimant par
295    la-même la nécessité d'une recherche DNS. La journalisation doit
296    aussi être adaptée pour fonctionner sur un tel système.</p>
297
298<pre class="prettyprint lang-config"># obtention du nom du serveur par recherche DNS inverse
299# sur l'adresse IP
300UseCanonicalName DNS
301
302# insertion de l'adresse IP dans les journaux afin de pouvoir les
303# éclater
304LogFormat "%A %h %l %u %t \"%r\" %s %b" vcommon
305CustomLog "logs/access_log" vcommon
306
307# insertion de l'adresse IP dans les noms de fichiers
308VirtualDocumentRootIP "/www/hosts/%0/docs"
309VirtualScriptAliasIP  "/www/hosts/%0/cgi-bin"</pre>

314<h2><a name="rewrite" id="rewrite">Hébergement virtuel de masse avec
315mod_rewrite</a></h2>
316
317<p>
318L'hébergement virtuel de masse peut aussi être effectué en utilisant
319<code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code>, soit à l'aide de simples directives <code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">RewriteRule</a></code>, soit en utilisant des
320techniques plus compliquées comme le stockage externe des définitions
321des serveurs virtuels, ces dernières étant accessibles via des
322directives <code class="directive"><a href="../mod/mod_rewrite.html#rewritemap">RewriteMap</a></code>. Ces
323techniques sont décrites dans la <a href="../rewrite/vhosts.html">documentation sur la réécriture</a>.</p>

327<h2><a name="macro" id="macro">Hébergement virtuel en masse avec mod_macro</a></h2>
328
329<p>Une autre option pour générer dynamiquement des serveurs virtuels :
