44<h2><a name="explanation" id="explanation">What is IP-based virtual hosting</a></h2>
45<p>IP-based virtual hosting is a method to apply different directives
46based on the IP address and port a request is received on.  Most commonly,
47this is used to serve different websites on different ports or interfaces.</p>
48
49<p>In many cases, <a href="name-based.html">name-based
50virtual hosts</a> are more convenient, because they allow
51many virtual hosts to share a single address/port.
52See <a href="name-based.html#namevip">Name-based vs. IP-based
53Virtual Hosts</a> to help you decide.  </p>
54</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>
55<div class="section">
56<h2><a name="requirements" id="requirements">System requirements</a></h2>
57
58    <p>As the term <cite>IP-based</cite> indicates, the server
59    <strong>must have a different IP address/port combination for each IP-based
60    virtual host</strong>. This can be achieved by the machine
61    having several physical network connections, or by use of
62    virtual interfaces which are supported by most modern operating
63    systems (see system documentation for details, these are
64    frequently called "ip aliases", and the "ifconfig" command is
65    most commonly used to set them up), and/or using multiple
66    port numbers.</p>
67
68    <p> In the terminology of Apache HTTP Server, using a single IP address
69    but multiple TCP ports, is also IP-based virtual hosting.</p>
70
71</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>

73<h2><a name="howto" id="howto">How to set up Apache</a></h2>
74
75    <p>There are two ways of configuring apache to support multiple
76    hosts. Either by running a separate <code class="program"><a href="../programs/httpd.html">httpd</a></code> daemon for
77    each hostname, or by running a single daemon which supports all the
78    virtual hosts.</p>
79
80    <p>Use multiple daemons when:</p>
81
82    <ul>
83      <li>There are security partitioning issues, such as company1
84      does not want anyone at company2 to be able to read their
85      data except via the web. In this case you would need two
86      daemons, each running with different <code class="directive"><a href="../mod/mod_unixd.html#user">User</a></code>, <code class="directive"><a href="../mod/mod_unixd.html#group">Group</a></code>, <code class="directive"><a href="../mod/mpm_common.html#listen">Listen</a></code>, and <code class="directive"><a href="../mod/core.html#serverroot">ServerRoot</a></code> settings.</li>
87
88      <li>You can afford the memory and file descriptor
89      requirements of listening to every IP alias on the
90      machine. It's only possible to <code class="directive"><a href="../mod/mpm_common.html#listen">Listen</a></code> to the "wildcard"
91      address, or to specific addresses. So if you have a need to
92      listen to a specific address for whatever reason, then you
93      will need to listen to all specific addresses. (Although one
94      <code class="program"><a href="../programs/httpd.html">httpd</a></code> could listen to N-1 of the addresses, and another could
95      listen to the remaining address.)</li>
96    </ul>
97
98    <p>Use a single daemon when:</p>
99
100    <ul>
101      <li>Sharing of the httpd configuration between virtual hosts
102      is acceptable.</li>
103
104      <li>The machine services a large number of requests, and so
105      the performance loss in running separate daemons may be
106      significant.</li>
107    </ul>
108

111<h2><a name="multiple" id="multiple">Setting up multiple daemons</a></h2>
112
113    <p>Create a separate <code class="program"><a href="../programs/httpd.html">httpd</a></code> installation for each
114    virtual host. For each installation, use the <code class="directive"><a href="../mod/mpm_common.html#listen">Listen</a></code> directive in the
115    configuration file to select which IP address (or virtual host)
116    that daemon services. e.g.</p>
117
118    <pre class="prettyprint lang-config">Listen 192.0.2.100:80</pre>
119
120
121    <p>It is recommended that you use an IP address instead of a
122    hostname (see <a href="../dns-caveats.html">DNS caveats</a>).</p>
123
124</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>
125<div class="section">
126<h2><a name="single" id="single">Setting up a single daemon
127  with virtual hosts</a></h2>
128
129    <p>For this case, a single <code class="program"><a href="../programs/httpd.html">httpd</a></code> will service
130    requests for the main server and all the virtual hosts. The <code class="directive"><a href="../mod/core.html#virtualhost">VirtualHost</a></code> directive
131    in the configuration file is used to set the values of <code class="directive"><a href="../mod/core.html#serveradmin">ServerAdmin</a></code>, <code class="directive"><a href="../mod/core.html#servername">ServerName</a></code>, <code class="directive"><a href="../mod/core.html#documentroot">DocumentRoot</a></code>, <code class="directive"><a href="../mod/core.html#errorlog">ErrorLog</a></code> and <code class="directive"><a href="../mod/mod_log_config.html#transferlog">TransferLog</a></code>
132    or <code class="directive"><a href="../mod/mod_log_config.html#customlog">CustomLog</a></code>
133    configuration directives to different values for each virtual
134    host. e.g.</p>
135
136    <pre class="prettyprint lang-config">&lt;VirtualHost 172.20.30.40:80&gt;
137    ServerAdmin webmaster@www1.example.com
138    DocumentRoot "/www/vhosts/www1"
139    ServerName www1.example.com
140    ErrorLog "/www/logs/www1/error_log"
141    CustomLog "/www/logs/www1/access_log" combined
142&lt;/VirtualHost&gt;
143
144&lt;VirtualHost 172.20.30.50:80&gt;
145    ServerAdmin webmaster@www2.example.org
146    DocumentRoot "/www/vhosts/www2"
147    ServerName www2.example.org
148    ErrorLog "/www/logs/www2/error_log"
149    CustomLog "/www/logs/www2/access_log" combined
150&lt;/VirtualHost&gt;</pre>
151
152
153    <p>It is recommended that you use an IP address instead of a
154    hostname in the &lt;VirtualHost&gt; directive
155    (see <a href="../dns-caveats.html">DNS caveats</a>).</p>
156
157    <p> Specific IP addresses or ports have precedence over their wildcard
158    equivalents, and any virtual host that matches has precedence over
159    the servers base configuration.</p>
160
161    <p>Almost <strong>any</strong> configuration directive can be
162    put in the VirtualHost directive, with the exception of
163    directives that control process creation and a few other
164    directives. To find out if a directive can be used in the
165    VirtualHost directive, check the <a href="../mod/directive-dict.html#Context">Context</a> using the
166    <a href="../mod/quickreference.html">directive index</a>.</p>
167
168    <p><code class="directive"><a href="../mod/mod_suexec.html#suexecusergroup">SuexecUserGroup</a></code>
169    may be used inside a
170    VirtualHost directive if the <a href="../suexec.html">suEXEC
171    wrapper</a> is used.</p>
172
173    <p><em>SECURITY:</em> When specifying where to write log files,
174    be aware of some security risks which are present if anyone
175    other than the user that starts Apache has write access to the
176    directory where they are written. See the <a href="../misc/security_tips.html">security tips</a> document
177    for details.</p>
