54<h2><a name="motivation" id="motivation">Motivation</a></h2>
55
56    <p>The techniques described here are of interest if your
57    <code>httpd.conf</code> contains many
58    <code>&lt;VirtualHost&gt;</code> sections that are
59    substantially the same, for example:</p>
60
61<pre class="prettyprint lang-config">&lt;VirtualHost 111.22.33.44&gt;
62    ServerName                 customer-1.example.com
63    DocumentRoot        "/www/hosts/customer-1.example.com/docs"
64    ScriptAlias  "/cgi-bin/"  "/www/hosts/customer-1.example.com/cgi-bin"
65&lt;/VirtualHost&gt;
66
67&lt;VirtualHost 111.22.33.44&gt;
68    ServerName                 customer-2.example.com
69    DocumentRoot        "/www/hosts/customer-2.example.com/docs"
70    ScriptAlias  "/cgi-bin/"  "/www/hosts/customer-2.example.com/cgi-bin"
71&lt;/VirtualHost&gt;
72
73&lt;VirtualHost 111.22.33.44&gt;
74    ServerName                 customer-N.example.com
75    DocumentRoot        "/www/hosts/customer-N.example.com/docs"
76    ScriptAlias  "/cgi-bin/"  "/www/hosts/customer-N.example.com/cgi-bin"
77&lt;/VirtualHost&gt;</pre>
78
79
80    <p>We wish to replace these multiple
81    <code>&lt;VirtualHost&gt;</code> blocks with a mechanism
82    that works them out dynamically. This has a number of
83    advantages:</p>
84
85    <ol>
86      <li>Your configuration file is smaller, so Apache starts
87      more quickly and uses less memory. Perhaps more importantly, the
88      smaller configuration is easier to maintain, and leaves less room
89      for errors.</li>
90
91      <li>Adding virtual hosts is simply a matter of creating the
92      appropriate directories in the filesystem and entries in the
93      DNS - you don't need to reconfigure or restart Apache.</li>
94    </ol>
95
96    <p>The main disadvantage is that you cannot have a different log file for
97    each virtual host; however, if you have many virtual hosts, doing
98    this can be a bad idea anyway, because of the <a href="fd-limits.html">number of file descriptors needed</a>.
99    It is better to <a href="../logs.html#piped">log to a pipe or a fifo</a>,
100    and arrange for the process at the other end to split up the log
101    files into one per virtual host. One example of such a process can
102    be found in the <a href="../programs/split-logfile.html">split-logfile</a>
103    utility.</p>
104

107<h2><a name="overview" id="overview">Overview</a></h2>
108
109    <p>A virtual host is defined by two pieces of information: its
110    IP address, and the contents of the <code>Host:</code> header
111    in the HTTP request. The dynamic mass virtual hosting technique
112    used here is based on automatically inserting this information into the
113    pathname of the file that is used to satisfy the request. This
114    can be most easily done by using <code class="module"><a href="../mod/mod_vhost_alias.html">mod_vhost_alias</a></code>
115    with Apache httpd. Alternatively,
116    <a href="../rewrite/vhosts.html">mod_rewrite can
117    be used</a>.</p>
118    <p>Both of these modules are disabled by default; you must enable
119    one of them when configuring and building Apache httpd if you want to
120    use this technique.</p>
121
122    <p>A couple of things need to be determined from the request in
123    order to make the dynamic
124    virtual host look like a normal one. The most important is the
125    server name, which is used by the server to generate
126    self-referential URLs etc. It is configured with the
127    <code>ServerName</code> directive, and it is available to CGIs
128    via the <code>SERVER_NAME</code> environment variable. The
129    actual value used at run time is controlled by the <code class="directive"><a href="../mod/core.html#usecanonicalname">UseCanonicalName</a></code>
130    setting. With <code>UseCanonicalName Off</code>, the server name
131    is taken from the contents of the <code>Host:</code> header in the
132    request. With <code>UseCanonicalName DNS</code>, it is taken from a
133    reverse DNS lookup of the virtual host's IP address. The former
134    setting is used for name-based dynamic virtual hosting, and the
135    latter is used for IP-based hosting. If httpd cannot work out
136    the server name because there is no <code>Host:</code> header,
137    or the DNS lookup fails, then the value configured with
138    <code>ServerName</code> is used instead.</p>
139
140    <p>The other thing to determine is the document root (configured
141    with <code>DocumentRoot</code> and available to CGI scripts via the
142    <code>DOCUMENT_ROOT</code> environment variable). In a normal
143    configuration, this is used by the core module when
144    mapping URIs to filenames, but when the server is configured to
145    do dynamic virtual hosting, that job must be taken over by another
146    module (either <code class="module"><a href="../mod/mod_vhost_alias.html">mod_vhost_alias</a></code> or
147    <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code>), which has a different way of doing
148    the mapping. Neither of these modules is responsible for
149    setting the <code>DOCUMENT_ROOT</code> environment variable so
150    if any CGIs or SSI documents make use of it, they will get a
151    misleading value.</p>

155<h2><a name="simple" id="simple">Dynamic Virtual Hosts with
156mod_vhost_alias</a></h2>
157
158    <p>This extract from <code>httpd.conf</code> implements the
159    virtual host arrangement outlined in the <a href="#motivation">Motivation</a> section above
160    using <code class="module"><a href="../mod/mod_vhost_alias.html">mod_vhost_alias</a></code>.</p>
161
162<pre class="prettyprint lang-config"># get the server name from the Host: header
163UseCanonicalName Off
164
165# this log format can be split per-virtual-host based on the first field
166# using the split-logfile utility.
167LogFormat "%V %h %l %u %t \"%r\" %s %b" vcommon
168CustomLog "logs/access_log" vcommon
169
170# include the server name in the filenames used to satisfy requests
171VirtualDocumentRoot "/www/hosts/%0/docs"
172VirtualScriptAlias  "/www/hosts/%0/cgi-bin"</pre>
173
174
175    <p>This configuration can be changed into an IP-based virtual
176    hosting solution by just turning <code>UseCanonicalName
177    Off</code> into <code>UseCanonicalName DNS</code>. The server
178    name that is inserted into the filename is then derived from
179    the IP address of the virtual host. The variable <code>%0</code>
180    references the requested servername, as indicated in the
181    <code>Host:</code> header.</p>
182
183<p>See the <code class="module"><a href="../mod/mod_vhost_alias.html">mod_vhost_alias</a></code> documentation for more usage
184examples.</p>

188<h2><a name="homepages" id="homepages">Simplified Dynamic Virtual Hosts</a></h2>
189
190    <p>This is an adjustment of the above system, tailored for an
191    ISP's web hosting server. Using <code>%2</code>,
192    we can select substrings of the server name to
193    use in the filename so that, for example, the documents for
194    <code>www.user.example.com</code> are found in
195    <code>/home/user/www</code>. It uses a single <code>cgi-bin</code>
196    directory instead of one per virtual host.</p>
197
198<pre class="prettyprint lang-config">UseCanonicalName Off
199
200LogFormat "%V %h %l %u %t \"%r\" %s %b" vcommon
201CustomLog "logs/access_log" vcommon
202
203# include part of the server name in the filenames
204VirtualDocumentRoot "/home/%2/www"
205
206# single cgi-bin directory
207ScriptAlias  "/cgi-bin/"  "/www/std-cgi/"</pre>
208
209
210    <p>There are examples of more complicated
211    <code>VirtualDocumentRoot</code> settings in the
212    <code class="module"><a href="../mod/mod_vhost_alias.html">mod_vhost_alias</a></code> documentation.</p>

216<h2><a name="combinations" id="combinations">Using Multiple Virtual
217  Hosting Systems on the Same Server</a></h2>
218
219    <p>With more complicated setups, you can use httpd's normal
220    <code>&lt;VirtualHost&gt;</code> directives to control the
221    scope of the various virtual hosting configurations. For
222    example, you could have one IP address for general customers' homepages,
223    and another for commercial customers, with the following setup.
224    This can be combined with conventional
225    <code>&lt;VirtualHost&gt;</code> configuration sections, as shown
226    below.</p>
227
228<pre class="prettyprint lang-config">UseCanonicalName Off
229
230LogFormat "%V %h %l %u %t \"%r\" %s %b" vcommon
231
232&lt;Directory "/www/commercial"&gt;
233    Options FollowSymLinks
234    AllowOverride All
235&lt;/Directory&gt;
236
237&lt;Directory "/www/homepages"&gt;
238    Options FollowSymLinks
239    AllowOverride None
240&lt;/Directory&gt;
241
242&lt;VirtualHost 111.22.33.44&gt;
243    ServerName www.commercial.example.com
244
245    CustomLog "logs/access_log.commercial" vcommon
246
247    VirtualDocumentRoot "/www/commercial/%0/docs"
248    VirtualScriptAlias  "/www/commercial/%0/cgi-bin"
249&lt;/VirtualHost&gt;
250
251&lt;VirtualHost 111.22.33.45&gt;
252    ServerName www.homepages.example.com
253
254    CustomLog "logs/access_log.homepages" vcommon
255
256    VirtualDocumentRoot "/www/homepages/%0/docs"
257    ScriptAlias         "/cgi-bin/" "/www/std-cgi/"
258&lt;/VirtualHost&gt;</pre>
259
260
261<div class="note">
262    <h3>Note</h3>
263    <p>If the first VirtualHost block does <em>not</em> include a
264    <code class="directive"><a href="../mod/core.html#servername">ServerName</a></code> directive, the reverse
265    DNS of the relevant IP will be used instead.
266    If this is not the server name you
267    wish to use, a bogus entry (eg. <code>ServerName
268    none.example.com</code>) can be added to get around this
269    behavior.</p>
270</div>

274<h2><a name="ipbased" id="ipbased">More Efficient IP-Based Virtual Hosting</a></h2>
275
276    <p>The configuration changes suggested to turn <a href="#simple">the first
277    example</a> into an IP-based virtual hosting setup result in
278    a rather inefficient setup. A new DNS lookup is required for every
279    request. To avoid this overhead, the filesystem can be arranged to
280    correspond to the IP addresses, instead of to the host names, thereby
281    negating the need for a DNS lookup. Logging will also have to be adjusted
282    to fit this system.</p>
283
284<pre class="prettyprint lang-config"># get the server name from the reverse DNS of the IP address
285UseCanonicalName DNS
286
287# include the IP address in the logs so they may be split
288LogFormat "%A %h %l %u %t \"%r\" %s %b" vcommon
289CustomLog "logs/access_log" vcommon
290
291# include the IP address in the filenames
292VirtualDocumentRootIP "/www/hosts/%0/docs"
293VirtualScriptAliasIP  "/www/hosts/%0/cgi-bin"</pre>

298<h2><a name="rewrite" id="rewrite">Mass virtual hosts with
299mod_rewrite</a></h2>
300
301<p>
302Mass virtual hosting may also be accomplished using
303<code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code>, either using simple <code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">RewriteRule</a></code> directives, or using more
304complicated techniques such as storing the vhost definitions externally
305and accessing them via <code class="directive"><a href="../mod/mod_rewrite.html#rewritemap">RewriteMap</a></code>. These techniques are
306discussed in the <a href="../rewrite/vhosts.html">rewrite
307documentation</a>.</p>
