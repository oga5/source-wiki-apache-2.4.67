67<h2><a name="purename" id="purename">Fonctionnement de plusieurs serveurs 
68  virtuels par nom sur une seule adresse IP.</a></h2>
69
70    <p>Votre serveur possède plusieurs noms d'hôte qui correspondent à une seule
71    adresse IP, et vous souhaitez des réponses différentes si on demande
72    <code>www.example.com</code> ou <code>www.example.org</code>.</p>
73
74    <div class="note"><h3>Note&nbsp;:</h3><p>La configuration de serveurs virtuels 
75    sous Apache ne provoque pas leur apparition magique dans la 
76    configuration du DNS. Il <em>faut</em> que leurs noms soient 
77    définis dans le DNS, et qu'ils y soient résolus sur l'adresse IP 
78    du serveur, faute de quoi personne ne pourra visiter votre site Web. 
79    Il est possible d'ajouter des entrées dans le fichier 
80    <code>hosts</code> pour tests locaux, mais qui ne fonctionneront 
81    que sur la machine possédant ces entrées.</p>
82    </div>
83
84    <pre class="prettyprint lang-config"># Apache doit écouter sur le port 80
85Listen 80
86&lt;VirtualHost *:80&gt;
87    DocumentRoot "/www/example1"
88    ServerName www.example.com
89  
90    # Autres directives ici
91&lt;/VirtualHost&gt;
92
93&lt;VirtualHost *:80&gt;
94    DocumentRoot "/www/example2"
95    ServerName www.example.org
96
97    # Autres directives ici
98&lt;/VirtualHost&gt;</pre>
99
100   
101
102    <p>Les astérisques correspondent à toutes les adresses, si bien que 
103    le serveur principal ne répondra jamais à aucune requête. Comme le
104    serveur virtuel
105    <code>ServerName www.example.com</code> se trouve en premier dans le fichier 
106    de configuration, il a la plus grande priorité et peut être vu 
107    comme serveur <cite>par défaut</cite> ou <cite>primaire</cite>&nbsp;; 
108    ce qui signifie que toute requête reçue ne correspondant à aucune 
109    des directives <code class="directive"><a href="../mod/core.html#servername">ServerName</a></code> sera servie par ce premier 
110    <code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code>.</p>
111
112    <p>La configuration ci-dessus correspond à ce que l'on souhaite pour
113    la plupart des serveurs virtuels à base de nom. Il faudra cependant
114    utiliser une configuration différente si vous souhaitez servir un
115    contenu différent en fonction de l'adresse IP ou du port.</p>
116
117    <div class="note">
118            <h3>Note&nbsp;:</h3>
119
120            <p>Vous pouvez remplacer <code>*</code> 
121            par une adresse IP du système. Le serveur virtuel concerné
122	    ne sera alors sélectionné que pour les requêtes HTTP vers
123	    cette adresse IP.</p>
124
125           <p>En général, il est commode d'utiliser <code>*</code> sur 
126           les systèmes dont l'adresse IP n'est pas constante - par 
127           exemple, pour des serveurs dont l'adresse IP est attribuée 
128           dynamiquement par le FAI, et où le DNS est géré au moyen 
129           d'un DNS dynamique quelconque. Comme <code>*</code> signifie 
130           <cite>n'importe quelle adresse</cite>, cette configuration 
131           fonctionne sans devoir être modifiée quand l'adresse IP du 
132           système est modifiée.</p>
133    </div>

137<h2><a name="twoips" id="twoips">Serveurs virtuels par nom sur plus 
138    d'une seule adresse IP.</a></h2>
139
140  	<div class="note">
141          <h3>Note&nbsp;:</h3><p>Toutes les techniques présentées ici 
142          peuvent être étendues à un plus grand nombre d'adresses IP.</p>
143    </div>
144
145    <p>Le serveur a deux adresses IP. Sur l'une 
146    (<code>172.20.30.40</code>), le serveur "principal" 
147    <code>server.example.com</code> doit répondre, et sur l'autre 
148    (<code>172.20.30.50</code>), deux serveurs virtuels (ou plus) 
149    répondront.</p>
150
151    <pre class="prettyprint lang-config">Listen 80
152
153# Serveur "principal" sur 172.20.30.40
154ServerName server.example.com
155DocumentRoot "/www/mainserver"
156
157&lt;VirtualHost 172.20.30.50&gt;
158    DocumentRoot "/www/example1"
159    ServerName www.example.com
160    
161    # D'autres directives ici ...
162&lt;/VirtualHost&gt;
163
164&lt;VirtualHost 172.20.30.50&gt;
165    DocumentRoot "/www/example2"
166    ServerName www.example.org
167    
168    # D'autres directives ici ...
169&lt;/VirtualHost&gt;</pre>
170
171
172    <p>Toute requête arrivant sur une autre adresse que 
173    <code>172.20.30.50</code> sera servie par le serveur principal. 
174    Les requêtes vers <code>172.20.30.50</code> avec un nom de serveur 
175    inconnu, ou sans en-tête <code>Host:</code>, seront servies par 
176    <code>www.example.com</code>.</p>

180<h2><a name="intraextra" id="intraextra">Servir le même contenu sur des 
181    adresses IP différentes (telle qu'une adresse interne et une 
182    externe).</a></h2>
183
184    <p>La machine serveur dispose de deux adresses IP 
185    (<code>192.168.1.1</code> et <code>172.20.30.40</code>). Cette 
186    machine est placée à la fois sur le réseau interne (l'Intranet) 
187    et le réseau externe (Internet). Sur Internet, le nom 
188    <code>server.example.com</code> pointe vers l'adresse externe 
189    (<code>172.20.30.40</code>), mais sur le réseau interne, ce même 
190    nom pointe vers l'adresse interne (<code>192.168.1.1</code>).</p>
191
192    <p>Le serveur peut être configuré pour répondre de la même manière 
193    aux requêtes internes et externes, au moyen d'une seule section 
194    <code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code>.</p>
195
196    <pre class="prettyprint lang-config">&lt;VirtualHost 192.168.1.1 172.20.30.40&gt;
197    DocumentRoot "/www/server1"
198    ServerName server.example.com
199    ServerAlias server
200&lt;/VirtualHost&gt;</pre>
201
202
203    <p>Ainsi, les requêtes en provenance de chacun des deux réseaux 
204    seront servies par le même <code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code>.</p>
205
206    <div class="note">
207          <h3>Note&nbsp;:</h3><p>Sur le réseau interne, il est possible 
208          d'utiliser le nom raccourci <code>server</code> au lieu du nom 
209          complet <code>server.example.com</code>.</p>
210
211          <p>Notez également que dans l'exemple précédent, vous pouvez 
212          remplacer la liste des adresses IP par des <code>*</code> afin 
213          que le serveur réponde de la même manière sur toutes ses 
214          adresses.</p>
215    </div>

219<h2><a name="port" id="port">Servir différents sites sur différents 
220    ports.</a></h2>
221
222    <p>Vous disposez de plusieurs domaines pointant sur la même adresse 
223    IP et vous voulez également servir de multiples ports. L'exemple
224    suivant montre que la sélection en fonction du nom intervient après
225    la sélection de la meilleure correspondance du point de vue adresse
226    IP/port.</p>
227
228    <pre class="prettyprint lang-config">Listen 80
229Listen 8080
230
231&lt;VirtualHost 172.20.30.40:80&gt;
232    ServerName www.example.com
233    DocumentRoot "/www/domain-80"
234&lt;/VirtualHost&gt;
235
236&lt;VirtualHost 172.20.30.40:8080&gt;
237    ServerName www.example.com
238    DocumentRoot "/www/domain-8080"
239&lt;/VirtualHost&gt;
240
241&lt;VirtualHost 172.20.30.40:80&gt;
242    ServerName www.example.org
243    DocumentRoot "/www/otherdomain-80"
244&lt;/VirtualHost&gt;
245
246&lt;VirtualHost 172.20.30.40:8080&gt;
247    ServerName www.example.org
248    DocumentRoot "/www/otherdomain-8080"
249&lt;/VirtualHost&gt;</pre>

255
256    <p>Le serveur dispose de deux adresses IP (<code>172.20.30.40</code> 
257    et <code>172.20.30.50</code>) correspondant respectivement aux noms

355<h2><a name="proxy" id="proxy">Utilisation simultanée de 
356    <code>Virtual_host</code> et de mod_proxy</a></h2>
357
358    <p>L'exemple suivant montre comment une machine peut mandater 
359    un serveur virtuel fonctionnant sur le serveur d'une autre machine. 
360    Dans cet exemple, un serveur virtuel de même nom est configuré sur 
361    une machine à l'adresse <code>192.168.111.2</code>. La directive 
362    <code class="directive"><a href="../mod/mod_proxy.html#proxypreservehost">ProxyPreserveHost On</a></code> est
363    employée pour permette au nom de domaine d'être préservé lors du 
364    transfert, au cas où plusieurs noms de domaines cohabitent sur 
365    une même machine.</p>
366
367    <pre class="prettyprint lang-config">&lt;VirtualHost *:*&gt;
368    ProxyPreserveHost On
369    ProxyPass        "/" "http://192.168.111.2/"
370    ProxyPassReverse "/" "http://192.168.111.2/"
371    ServerName hostname.example.com
372&lt;/VirtualHost&gt;</pre>

377<h2><a name="default" id="default">Utilisation de serveurs virtuels 
378    <code>_default_</code></a></h2>
379
380    <h3><a name="defaultallports" id="defaultallports">Serveurs virtuels 
381    <code>_default_</code> pour tous les ports</a></h3>
382
383    <p>Exemple de capture de <em>toutes</em> les requêtes émanant 
384    d'adresses IP ou de ports non connus, <em>c'est-à-dire</em>, d'un 
385    couple adresse/port non traité par aucun autre serveur virtuel.</p>
386
387    <pre class="prettyprint lang-config">&lt;VirtualHost _default_:*&gt;
388    DocumentRoot "/www/default"
389&lt;/VirtualHost&gt;</pre>
390
391
392    <p>L'utilisation d'un tel serveur virtuel avec un joker pour le 
393    port empêche de manière efficace qu'une requête n'atteigne le 
394    serveur principal.</p>
395
396    <p>Un serveur virtuel par défaut ne servira jamais une requête 
397    qui est envoyée vers un couple adresse/port utilisée par un 
398    serveur virtuel par nom. Si la requête contient un en-tête 
399    <code>Host:</code> inconnu, ou si celui-ci est absent, elle 
400    sera toujours servie par le serveur virtuel primaire par nom 
401    (celui correspondant à ce couple adresse/port trouvé en premier 
402    dans le fichier de configuration).</p>
403
404    <p>Vous pouvez utiliser une directive 
405    <code class="directive"><a href="../mod/mod_alias.html#aliasmatch">AliasMatch</a></code> ou 
406    <code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">RewriteRule</a></code> afin de 
407    réécrire une requête pour une unique page d'information (ou pour 
408    un script).</p>
409    
410
411    <h3><a name="defaultdifferentports" id="defaultdifferentports">Serveurs virtuels 
412    <code>_default_</code> pour des ports différents</a></h3>
413
414    <p>La configuration est similaire à l'exemple précédent, mais 
415    le serveur écoute sur plusieurs ports et un second serveur virtuel 
416    <code>_default_</code> pour le port 80 est ajouté.</p>
417
418    <pre class="prettyprint lang-config">&lt;VirtualHost _default_:80&gt;
419    DocumentRoot "/www/default80"
420    # ...
421&lt;/VirtualHost&gt;
422
423&lt;VirtualHost _default_:*&gt;
424    DocumentRoot "/www/default"
425    # ...
426&lt;/VirtualHost&gt;</pre>
427
428
429    <p>Le serveur virtuel par défaut défini pour le port 80 (il doit 
430    impérativement être placé avant un autre serveur virtuel par 
431    défaut traitant tous les ports grâce au joker *) capture toutes 
432    les requêtes envoyées sur une adresse IP non spécifiée. Le 
433    serveur principal n'est jamais utilisé pour servir une requête.</p>
434    
435
436    <h3><a name="defaultoneport" id="defaultoneport">Serveurs virtuels 
437    <code>_default_</code> pour un seul port</a></h3>
438
439    <p>Nous voulons créer un serveur virtuel par défaut seulement 
440    pour le port 80.</p>
441
442    <pre class="prettyprint lang-config">&lt;VirtualHost _default_:80&gt;
443    DocumentRoot "/www/default"
444...
445&lt;/VirtualHost&gt;</pre>
446
447
448    <p>Une requête vers une adresse non spécifiée sur le port 80 
449    sera servie par le serveur virtuel par défaut, et toute autre 
450    requête vers une adresse et un port non spécifiés sera servie 
451    par le serveur principal.</p>
452
453    <p>L'utilisation du caractère générique <code>*</code> dans la
454    déclaration d'un serveur virtuel l'emporte sur
455    <code>_default_</code>.</p>
456

460<h2><a name="migrate" id="migrate">Migration d'un serveur virtuel 
461	par nom en un serveur virtuel par IP</a></h2>
462
463    <p>Le serveur virtuel par nom avec le nom de domaine 
464    <code>www.example.org</code> (de notre <a href="#name">exemple 
465    par nom</a>) devrait obtenir sa propre adresse IP. Pendant la 
466    phase de migration, il est possible d'éviter les problèmes avec 
467    les noms de serveurs et autres serveurs mandataires qui mémorisent 
468    les vielles adresses IP pour les serveurs virtuels par nom.<br />
469    La solution est simple, car il suffit d'ajouter la nouvelle 
470    adresse IP (<code>172.20.30.50</code>) dans la directive 
471    <code>VirtualHost</code>.</p>
472
473    <pre class="prettyprint lang-config">Listen 80
474ServerName www.example.com
475DocumentRoot "/www/example1"
476
477&lt;VirtualHost 172.20.30.40 172.20.30.50&gt;
478    DocumentRoot "/www/example2"
479    ServerName www.example.org
480    # ...
481&lt;/VirtualHost&gt;
482
483&lt;VirtualHost 172.20.30.40&gt;
484    DocumentRoot "/www/example3"
485    ServerName www.example.net
486    ServerAlias *.example.net
487    # ...
488&lt;/VirtualHost&gt;</pre>
489
490
491    <p>Le serveur virtuel peut maintenant être joint par la nouvelle 
492    adresse (comme un serveur virtuel par IP) et par l'ancienne 
493    adresse (comme un serveur virtuel par nom).</p>

496<div class="section">
497<h2><a name="serverpath" id="serverpath">Utilisation de la directive 
498    <code>ServerPath</code></a></h2>
499
500    <p>Dans le cas où vous disposez de deux serveurs virtuels par nom, 
501    le client doit transmettre un en-tête <code>Host:</code> correct 
502    pour déterminer le serveur concerné. Les vieux clients HTTP/1.0 
503    n'envoient pas un tel en-tête et Apache n'a aucun indice pour 
504    connaître le serveur virtuel devant être joint (il sert la 
505    requête à partir d'un serveur virtuel primaire). Dans un soucis 
506    de préserver la compatibilité descendante, il suffit de créer 
507    un serveur virtuel primaire chargé de retourner une page contenant 
508    des liens dont les URLs auront un préfixe identifiant les serveurs 
509    virtuels par nom.</p>
510
511    <pre class="prettyprint lang-config">&lt;VirtualHost 172.20.30.40&gt;
512    # serveur virtuel primaire
513    DocumentRoot "/www/subdomain"
514    RewriteEngine On
515    RewriteRule "." "/www/subdomain/index.html"
516    # ...
517&lt;/VirtualHost&gt;
518
519&lt;VirtualHost 172.20.30.40&gt;
520    DocumentRoot "/www/subdomain/sub1"
521    ServerName www.sub1.domain.tld
522    ServerPath "/sub1/"
523    RewriteEngine On
524    RewriteRule "^(/sub1/.*)" "/www/subdomain$1
525    # ...
526&lt;/VirtualHost&gt;
527
528&lt;VirtualHost 172.20.30.40&gt;
529    DocumentRoot "/www/subdomain/sub2"
530    ServerName www.sub2.domain.tld
531    ServerPath "/sub2/"
532    RewriteEngine On
533    RewriteRule "^(/sub2/.*)" "/www/subdomain$1"
534    # ...
535&lt;/VirtualHost&gt;</pre>
536
537
538    <p>À cause de la directive 
539    <code class="directive"><a href="../mod/core.html#serverpath">ServerPath</a></code>, une requête sur 
540    une URL <code>http://www.sub1.domain.tld/sub1/</code> est 
541    <em>toujours</em> servie par le serveur sub1-vhost.<br />
542    Une requête sur une URL <code>http://www.sub1.domain.tld/</code> n'est 
543    servie par le serveur sub1-vhost que si le client envoie un en-tête 
544    <code>Host:</code> correct. Si aucun en-tête <code>Host:</code> 
545    n'est transmis, le serveur primaire sera utilisé.</p>
546    <p>Notez qu'il y a une singularité&nbsp;: une requête sur 
547    <code>http://www.sub2.domain.tld/sub1/</code> est également servie 
548    par le serveur sub1-vhost si le client n'envoie pas d'en-tête 
549    <code>Host:</code>.</p>
550    <p>Les directives <code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">RewriteRule</a></code> 
551    sont employées pour s'assurer que le client qui envoie un en-tête 
552    <code>Host:</code> correct puisse utiliser d'autres variantes d'URLs, 
553    <em>c'est-à-dire</em> avec ou sans préfixe d'URL.</p>
