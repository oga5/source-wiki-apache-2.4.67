51<h2><a name="namevip" id="namevip">Namensbasierte gegen&#252;ber IP-basierten
52    virtuellen Hosts</a></h2>
53
54  <p>IP-basierte virtuelle Hosts verwenden die IP-Adresse der Verbindung, um den
55    korrekten virtuellen Host zur Bedienung einer Anfrage zu ermitteln. Folglich 
56    ben&#246;tigen Sie eine IP-Adresse f&#252;r jeden virtuellen Host. Bei der 
57    Verwendung von namensbasierten virtuellen Hosts verl&#228;&#223;t sich der 
58    Server darauf, dass der Client den Hostnamen als Bestandteil der HTTP-Header 
59    angibt. Durch Anwendung dieser Technik k&#246;nnen sich mehrere verschiedene 
60    Hosts die gleiche IP-Adresse teilen.</p>
61
62  <p>Die Verwendung von namensbasierten virtuellen Hosts ist gew&#246;hnlich 
63    einfacher. Sie m&#252;ssen lediglich Ihren DNS-Server darauf einstellen, 
64    jeden Hostnamen auf die richtige IP-Adresse abzubilden, und dann den Apache 
65    HTTP Server so konfigurieren, dass er die verschiedenen Hostnamen erkennt.
66    Namensbasierte virtuelle Hosts entsch&#228;rfen auch den Bedarf an 
67    knappen IP-Adressen. Daher sollten Sie namensbasierte virtuelle Hosts 
68    verwenden, sofern kein besonderer Grund daf&#252;r existiert, IP-basierte 
69    virtuelle Hosts zu w&#228;hlen. M&#246;gliche Gr&#252;nde f&#252;r die 
70    Verwendung IP-basierter virtueller Hosts sind:</p>
71
72  <ul>
73    <li>Einige antike Clients sind nicht kompatibel zu namensbasierten
74      virtuellen Hosts. Damit namensbasierte virtuelle Hosts funktionieren,
75      muss der Client den HTTP-Host-Header senden. Dies ist bei HTTP/1.1
76      vorgeschrieben und in allen modernen HTTP/1.0-Browsern als Erweiterung
77      implementiert. Wenn Sie Unterst&#252;tzung f&#252;r veraltete Clients
78      ben&#246;tigen und dennoch namensbasierte virtuelle Hosts verwenden,
79      dann finden Sie eine m&#246;gliche L&#246;sung daf&#252;r am Ende des
80      Dokuments.</li>
81
82    <li>Namensbasierte virtuelle Hosts k&#246;nnen aufgrund der Natur des
83      SSL-Protokolls nicht mit SSL-gesicherten Servern verwendet werden.</li>
84
85    <li>Einige Betriebssysteme und Netzwerkanlagen setzen Techniken zum 
86      Bandbreiten-Management ein, die nicht zwischen Hosts unterscheiden
87      k&#246;nnen, wenn diese nicht auf verschiedenen IP-Adressen liegen.</li>
88    </ul>

92<h2><a name="using" id="using">Die Verwendung von namensbasierten virtuellen Hosts</a></h2>
93
94  <table class="related"><tr><th>Referenzierte Module</th><th>Referenzierte Direktiven</th></tr><tr><td><ul><li><code class="module"><a href="../mod/core.html">core</a></code></li></ul></td><td><ul><li><code class="directive"><a href="../mod/core.html#documentroot">DocumentRoot</a></code></li><li><code class="directive"><a href="../mod/core.html#namevirtualhost">NameVirtualHost</a></code></li><li><code class="directive"><a href="../mod/core.html#serveralias">ServerAlias</a></code></li><li><code class="directive"><a href="../mod/core.html#servername">ServerName</a></code></li><li><code class="directive"><a href="../mod/core.html#serverpath">ServerPath</a></code></li><li><code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code></li></ul></td></tr></table>
95
96  <p>Um namensbasierte virtuelle Hosts zu verwenden, m&#252;ssen Sie die
97    IP-Adresse (und m&#246;glicherweise den Port) des Servers benennen, an
98    der Anfragen f&#252;r die Hosts entgegengenommen werden. Dies wird mit
99    der Direktive <code class="directive"><a href="../mod/core.html#namevirtualhost">NameVirtualHost</a></code>
100    eingestellt. Im Normalfall, wenn alle IP-Adressen des Server verwendet
101    werden sollen, k&#246;nnen Sie <code>*</code> als Argument f&#252;r
102    <code class="directive"><a href="../mod/core.html#namevirtualhost">NameVirtualHost</a></code> verwenden. Wenn Sie
103    vorhaben, mehrere Ports zu nutzen (etwa wenn SSL l&#228;uft), sollten
104    Sie dem Argument einen Port hinzuf&#252;gen, wie zum Beispiel
105    <code>*:80</code>. Beachten Sie,
106    dass die Angabe einer IP-Adresse in einer <code class="directive"><a href="../mod/core.html#namevirtualhost">NameVirtualHost</a></code>-Anweisung den Server nicht
107    automatisch an dieser Adresse lauschen l&#228;&#223;t. Lesen Sie bitte "<a href="../bind.html">Bestimmen der vom Apache verwendeten Adressen und
108    Ports</a>" f&#252;r weitere Details. Zus&#228;tzlich muss jede hier
109    angegebene IP-Adresse einer Netzwerkkarte des Servers zugeordnet sein.</p>
110 
111  <p>Der n&#228;chste Schritt ist die Erstellung eines <code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code>-Blocks f&#252;r jeden einzelnen
112    Host, den Sie bedienen wollen. Das Argument der Direktive <code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code> sollte das gleiche
113    sein wie das Argument der <code class="directive"><a href="../mod/core.html#namevirtualhost">NameVirtualHost</a></code>-Anweisung (d.h. eine IP-Adresse
114    oder <code>*</code> f&#252;r alle Adressen). Innerhalb jedes <code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code>-Blocks ben&#246;tigen
115    Sie zumindestens eine <code class="directive"><a href="../mod/core.html#servername">ServerName</a></code>-Anweisung, um zu bestimmen, welcher
116    Host bedient wird, und eine <code class="directive"><a href="../mod/core.html#documentroot">DocumentRoot</a></code>-Anweisung, um anzugeben, wo im
117    Dateisystem der Inhalt des Hosts abgelegt ist.</p>
118
119  <div class="note"><h3>Der Hauptserver verschwindet</h3>
120    Wenn Sie virtuelle Hosts zu einem bestehenden Webserver hinzuf&#252;gen,
121    m&#252;ssen Sie auch einen <code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code>-Block f&#252;r den bestehenden Host
122    <span class="transnote">(<em>Anm.d.&#220;.:</em> und bisherigen Hauptserver)</span> erstellen. 
123    Die <code class="directive"><a href="../mod/core.html#servername">ServerName</a></code>- und
124    <code class="directive"><a href="../mod/core.html#documentroot">DocumentRoot</a></code>-Anweisungen zu diesem
125    virtuellen Host sollten die gleichen sein wie die globalen <code class="directive"><a href="../mod/core.html#servername">ServerName</a></code>- und <code class="directive"><a href="../mod/core.html#documentroot">DocumentRoot</a></code>-Anweisungen. F&#252;hren Sie diesen
126    virtuellen Host als erstes in der Konfigurationsdatei auf, so dass er als
127    Standard-Host fungiert.
128  </div>
129
130  <p>Vorausgesetzt, Sie bedienen z.B. die Domain
131    <code>www.domain.tld</code> und m&#246;chten den virtuellen Host
132    <code>www.otherdomain.tld</code> hinzuf&#252;gen, welcher auf
133    die gleiche IP-Adresse zeigt. Dann f&#252;gen Sie einfach Folgendes der
134    <code>httpd.conf</code> hinzu:</p>
135
136    <div class="example"><p><code>
137    NameVirtualHost *:80<br />
138    <br />
139    &lt;VirtualHost *:80&gt;<br />
140    <span class="indent">
141    ServerName www.domain.tld<br />
142    ServerAlias domain.tld *.domain.tld<br />
143    DocumentRoot /www/domain<br />
144    </span>
145    &lt;/VirtualHost&gt;<br />
146    <br />
147    &lt;VirtualHost *:80&gt;<br />
148    <span class="indent">ServerName www.otherdomain.tld<br />
149    DocumentRoot /www/otherdomain<br />
150    </span>
151    &lt;/VirtualHost&gt;<br />
152    </code></p></div>
153
154  <p>Sie k&#246;nnen anstelle des <code>*</code> bei den beiden Anweisungen 
155    <code class="directive"><a href="../mod/core.html#namevirtualhost">NameVirtualHost</a></code> und <code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code> alternativ eine 
156    eindeutige IP-Adresse angeben. Das kann man beispielsweise machen, um 
157    einige namensbasierte virtuelle Hosts auf einer IP-Adresse zu betreiben und 
158    entweder IP-basierte oder ein anderes Set von namensbasierten virtuellen 
159    Hosts auf einer anderen Adresse.</p>
160  
161  <p>Viele Server wollen unter mehr als einem Namen erreichbar sein. Die 
162    Direktive <code class="directive"><a href="../mod/core.html#serveralias">ServerAlias</a></code>, die innerhalb 
163    des <code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code>-Abschnittes angegeben wird,
164    erm&#246;glicht dies. Zum Beispiel zeigt die <code class="directive"><a href="../mod/core.html#serveralias">ServerAlias</a></code>-Anweisung in dem ersten <code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code>-Block oben an, dass die
165    aufgef&#252;hrten Namen alternative Namen sind, die man verwenden kann, um
166    das gleiche Webangebot zu erreichen:</p>
167
168    <div class="example"><p><code>
169    ServerAlias domain.tld *.domain.tld
170    </code></p></div>
171
172  <p>Anfragen f&#252;r alle Hosts der Domain <code>domain.tld</code> werden
173    von dem virtuellen Host <code>www.domain.tld</code> bedient. Die
174    Platzhalter <code>*</code> und <code>?</code> k&#246;nnen anstelle
175    entsprechender Namen verwendet werden. Nat&#252;rlich k&#246;nnen Sie nicht
176    einfach Namen erfinden und diese bei <code class="directive"><a href="../mod/core.html#servername">ServerName</a></code> oder <code>ServerAlias</code>
177    angeben, Sie m&#252;ssen zun&#228;chst Ihren DNS Server entsprechend
178    konfigurieren, dass er diese Namen auf die mit Ihrem Server verkn&#252;pfte
179    IP-Adresse abbildet.</p>
180
181  <p>Und schlu&#223;endlich k&#246;nnen Sie die Konfiguration der virtuellen
182    Hosts mittels Angabe weiterer Direktiven innherhalb der <code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code>-Container
183    feineinstellen. Die meisten Direktiven k&#246;nnen in diesen Containern
184    angegeben werden und ver&#228;ndern dann ausschlie&#223;lich die
185    Konfiguration des entsprechenden virtuellen Hosts. Pr&#252;fen Sie den <a href="../mod/directive-dict.html#Context">Kontext</a> einer Direktive, um
186    herauszufinden, ob eine bestimmte Direktive zul&#228;ssig ist.
187    Im <em>Hauptserver-Kontext</em> (au&#223;erhalb der <code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code>-Container) definierte
188    Konfigurationsanweisungen werden nur dann angewendet, wenn sie nicht durch
189    Einstellungen des virtuellen Hosts au&#223;er Kraft gesetzt wurden.</p>
190
191  <p>Wenn nun eine Anfrage eintrifft, pr&#252;ft der Server zuerst, ob sie eine
192    IP-Adresse verwendet, die der <code class="directive"><a href="../mod/core.html#namevirtualhost">NameVirtualHost</a></code>-Anweisung entspricht. Ist dies der
193    Fall, dann sieht er sich jeden <code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code>-Abschnitt mit einer passenden
194    IP-Adresse an und versucht den einen zu finden, dessen <code class="directive"><a href="../mod/core.html#servername">ServerName</a></code>- oder <code class="directive"><a href="../mod/core.html#serveralias">ServerAlias</a></code>-Anweisung mit dem gew&#252;nschten
195    Hostnamen &#252;bereinstimmt. Findet er einen, dann verwendet er die
196    Konfiguration dieses Servers. Wird kein passender virtueller Host gefunden,
197    dann wird <strong>der erste angegeben virtuelle Host</strong> verwendet,
198    dessen IP-Adresse pa&#223;t.</p>
199
200  <p>Die Folge davon ist, dass der erste aufgef&#252;hrte virtuelle Host der
201    <em>Standard</em>-Virtual-Host ist. Die <code class="directive"><a href="../mod/core.html#documentroot">DocumentRoot</a></code>-Anweisung des <em>Hauptservers</em>
202    wird <strong>niemals</strong> verwendet, wenn eine IP-Adresse mit einer 
203    <code class="directive"><a href="../mod/core.html#namevirtualhost">NameVirtualHost</a></code>-Anweisung
204    &#252;bereinstimmt. Wenn Sie eine spezielle Konfiguration f&#252;r Anfragen
205    angeben m&#246;chten, die keinem bestimmten virtuellen Host entsprechen,
206    packen Sie diese Konfiguration einfach in einen <code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code>-Container und f&#252;hren diesen als
207    erstes in der Konfigurationsdatei auf.</p>

210<div class="section">
211<h2><a name="compat" id="compat">Kompatibilit&#228;t mit &#228;lteren Browsern</a></h2>
212  
213  <p>Wie zuvor erw&#228;hnt gibt es einige Clients, die nicht die notwendigen
214    Daten senden, mit denen namensbasierte virtuelle Hosts korrekt
215    funktionieren. Diesen Clients werden stets die Seiten des ersten, f&#252;r
216    diese IP-Adresse aufgef&#252;hrten virtuellen Hosts gesendet werden (des
217    <cite>prim&#228;ren</cite> namensbasierten virtuellen Hosts).</p>
218
219  <div class="note"><h3>Was bedeutet &#228;lter?</h3>
220    <p>Beachten Sie bitte, wenn wir von &#228;lter sprechen, meinen wir auch
221    &#228;lter. Es ist sehr unwahrscheinlich, dass sie einen dieser Browser
222    heutzutage in Verwendung finden werden. Alle aktuellen Browser-Versionen
223    senden den <code>Host</code>-Header, so wie er f&#252;r namensbasierte
224    virtuelle Hosts ben&#228;&#246;tigt wird.</p>
225  </div>
226
227  <p>Mit der Direktive <code class="directive"><a href="../mod/core.html#serverpath">ServerPath</a></code> existiert  
228    eine m&#246;gliche Behelfskonstruktion, obgleich sie etwas schwerf&#228;llig
229    ist:</p>
230
231  <p>Beispielkonfiguration:</p>
232
233  <div class="example"><p><code>
234    NameVirtualHost 111.22.33.44<br />
235    <br />
236    &lt;VirtualHost 111.22.33.44&gt;<br />
237    <span class="indent">
238    ServerName www.domain.tld<br />
239    ServerPath /domain<br />
240    DocumentRoot /web/domain<br />
241    </span>
242    &lt;/VirtualHost&gt;<br />
243  </code></p></div>
244
245  <p>Was bedeutet das? Es bedeutet, dass eine Anfrage f&#252;r eine mit
246    "<code>/domain</code>" beginnende URI von dem virtuellen Host
247    <code>www.domain.tld</code> bedient wird. Dies hei&#223;t, dass die Seiten
248    f&#252;r alle Clients unter <code>http://www.domain.tld/domain/</code>
249    abrufbar sind, wenngleich Clients, die den Header <code>Host:</code>
250    senden, auch &#252;ber <code>http://www.domain.tld/</code> auf sie zugreifen
251    k&#246;nnen.</p>
252
253  <p>Legen Sie einen Link auf der Seite Ihres prim&#228;ren virtuellen Hosts zu 
254    <code>http://www.domain.tld/domain/</code>, um die Behelfsl&#246;sung
255    verf&#252;gbar zu machen. Bei den Seiten der virtuellen Hosts m&#252;ssen
256    Sie dann sicherstellen, entweder au&#223;schlie&#223;lich relative Links
257    (<em>z.B.</em> "<code>file.html</code>" oder
258    "<code>../icons/image.gif</code>") zu verwenden oder Links, die das
259    einleitende <code>/domain/</code> enthalten (<em>z.B.</em>,
260    "<code>http://www.domain.tld/domain/misc/file.html</code>" oder
261    "<code>/domain/misc/file.html</code>").</p>
262
263  <p>Dies erfordert etwas Disziplin, die Befolgung dieser Richtlinien stellt
264    jedoch gr&#246;&#223;tenteils sicher, dass Ihre Seiten mit allen Browsern
265    funktionieren, alten wie neuen.</p>
