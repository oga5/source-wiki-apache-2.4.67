42<h2><a name="namevip" id="namevip">Name-based vs. IP-based Virtual Hosts</a></h2>
43
44    <p><a href="ip-based.html">IP-based virtual hosts</a> use the IP address of the connection to
45    determine the correct virtual host to serve.  Therefore you need to
46    have a separate IP address for each host.</p>
47
48    <p>With name-based virtual hosting, the server relies on the client to
49    report the hostname as part of the HTTP headers.  Using this technique,
50    many different hosts can share the same IP address.</p>
51
52    <p>Name-based virtual hosting is usually simpler, since you need
53    only configure your DNS server to map each hostname to the correct
54    IP address and then configure the Apache HTTP Server to recognize
55    the different hostnames. Name-based virtual hosting also eases
56    the demand for scarce IP addresses. Therefore you should use
57    name-based virtual hosting unless you are using equipment
58    that explicitly demands IP-based hosting.  Historical reasons for
59    IP-based virtual hosting based on client support are no longer
60    applicable to a general-purpose web server.</p>
61
62    <p> Name-based virtual hosting builds off of the IP-based virtual host
63    selection algorithm, meaning that searches for the proper server name
64    occur only between virtual hosts that have the best IP-based address.</p>
65
66</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>

68<h2><a name="alg" id="alg">How the server selects the proper name-based virtual host</a></h2>
69
70    <p>It is important to recognize that the first step in name-based virtual
71    host resolution is IP-based resolution.  Name-based virtual host
72    resolution only chooses the most appropriate name-based virtual host
73    after narrowing down the candidates to the best IP-based match.  Using a wildcard (*)
74    for the IP address in all of the VirtualHost directives makes this
75    IP-based mapping irrelevant.</p>
76
77    <p>When a request arrives, the server will find the best (most specific) matching
78    <code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code> argument based on
79    the IP address and port used by the request.  If there is more than one virtual host
80    containing this best-match address and port combination, Apache will further
81    compare the <code class="directive"><a href="../mod/core.html#servername">ServerName</a></code> and <code class="directive"><a href="../mod/core.html#serveralias">ServerAlias</a></code> directives to the server name
82    present in the request.</p>
83
84    <p>If you omit the <code class="directive"><a href="../mod/core.html#servername">ServerName</a></code> 
85    directive from any name-based virtual host, the server will default
86    to a fully qualified domain name (FQDN) derived from the system hostname.
87    This implicitly set server name can lead to counter-intuitive virtual host
88    matching and is discouraged.</p>
89 
90    <h3><a name="defaultvhost" id="defaultvhost">The default name-based vhost for an IP and port combination </a></h3>
91    <p> If no matching ServerName or ServerAlias is found in the set of
92    virtual hosts containing the most specific matching IP address and port
93    combination, then <strong>the first listed virtual host</strong> that
94    matches that will be used.</p>

97<h2><a name="using" id="using">Using Name-based Virtual Hosts</a></h2>
98
99<table class="related"><tr><th>Related Modules</th><th>Related Directives</th></tr><tr><td><ul><li><code class="module"><a href="../mod/core.html">core</a></code></li></ul></td><td><ul><li><code class="directive"><a href="../mod/core.html#documentroot">DocumentRoot</a></code></li><li><code class="directive"><a href="../mod/core.html#serveralias">ServerAlias</a></code></li><li><code class="directive"><a href="../mod/core.html#servername">ServerName</a></code></li><li><code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code></li></ul></td></tr></table>
100
101    <p>The first step is to create a <code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code> block for
102    each different host that you would like to serve.  Inside each <code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code> block, you will need at minimum a
103    <code class="directive"><a href="../mod/core.html#servername">ServerName</a></code> directive to designate
104    which host is served and a <code class="directive"><a href="../mod/core.html#documentroot">DocumentRoot</a></code>
105    directive to show where in the filesystem the content for that host
106    lives.</p>
107
108    <div class="note"><h3>Main host goes away</h3>
109        <p> Any request that doesn't match an existing <code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code> is handled by the global
110        server configuration, regardless of the hostname or ServerName.</p>
111
112        <p> When you add a name-based virtual host to an existing server, and
113        the virtual host arguments match preexisting IP and port combinations,
114        requests will now be handled by an explicit virtual host.  In this case,
115        it's usually wise to create a <a href="#defaultvhost">default virtual host</a>
116        with a <code class="directive"><a href="../mod/core.html#servername">ServerName</a></code> matching that of
117        the base server.  New domains on the same interface and port, but
118        requiring separate configurations,  can then be added as subsequent (non-default)
119        virtual hosts.</p>
120    </div>
121
122    <div class="note"><h3>ServerName inheritance</h3>
123       <p> It is best to always explicitly list a  <code class="directive"><a href="../mod/core.html#servername">ServerName</a></code> in every name-based virtual host.</p>
124       <p>If a <code class="directive"><a href="../mod/core.html#virtualhost">VirtualHost</a></code> doesn't specify 
125       a <code class="directive"><a href="../mod/core.html#servername">ServerName</a></code>, a server name will be 
126       inherited from the base server configuration.  If no server name was 
127       specified globally, one is detected at startup through reverse DNS resolution
128       of the first listening address.  In either case, this inherited server name
129       will influence name-based virtual host resolution, so it is best to always
130       explicitly list a  <code class="directive"><a href="../mod/core.html#servername">ServerName</a></code> in every
131       name-based virtual host.</p>
132    </div>
133
134    <p>For example, suppose that you are serving the domain
135    <code>www.example.com</code> and you wish to add the virtual host
136    <code>other.example.com</code>, which points at the same IP address.
137    Then you simply add the following to <code>httpd.conf</code>:</p>
138
139    <pre class="prettyprint lang-config">&lt;VirtualHost *:80&gt;
140    # This first-listed virtual host is also the default for *:80
141    ServerName www.example.com
142    ServerAlias example.com 
143    DocumentRoot "/www/domain"
144&lt;/VirtualHost&gt;
145
146&lt;VirtualHost *:80&gt;
147    ServerName other.example.com
148    DocumentRoot "/www/otherdomain"
149&lt;/VirtualHost&gt;</pre>
150
151
152    <p>You can alternatively specify an explicit IP address in place of the
153    <code>*</code> in <code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code> directives. For example, you might want to do this
154    in order to run some name-based virtual hosts on one IP address, and either
155    IP-based, or another set of name-based virtual hosts on another address.</p>
156
157    <p>Many servers want to be accessible by more than one name. This is
158    possible with the <code class="directive"><a href="../mod/core.html#serveralias">ServerAlias</a></code>
159    directive, placed inside the <code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code> section. For example in the first <code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code> block above, the
160    <code class="directive"><a href="../mod/core.html#serveralias">ServerAlias</a></code> directive indicates that
161    the listed names are other names which people can use to see that same
162    web site:</p>
163
164    <pre class="prettyprint lang-config">ServerAlias example.com *.example.com</pre>
165
166
167    <p>then requests for all hosts in the <code>example.com</code> domain will
168    be served by the <code>www.example.com</code> virtual host. The wildcard
169    characters <code>*</code> and <code>?</code> can be used to match names.
170    Of course, you can't just make up names and place them in <code class="directive"><a href="../mod/core.html#servername">ServerName</a></code> or <code>ServerAlias</code>. You must
171    first have your DNS server properly configured to map those names to an IP
172    address associated with your server.</p>
173
174    <p>Name-based virtual hosts for the best-matching set of  <code class="directive"><a href="../mod/core.html#virtualhost">&lt;virtualhost&gt;</a></code>s are processed 
175    in the order they appear in the configuration.  The first matching <code class="directive"><a href="../mod/core.html#servername">ServerName</a></code> or <code class="directive"><a href="../mod/core.html#serveralias">ServerAlias</a></code> is used, with no different precedence for wildcards
176    (nor for ServerName vs. ServerAlias).  </p>
177
178    <p>The complete list of names in the <code class="directive"><a href="../mod/core.html#virtualhost">VirtualHost</a></code>
179    directive are treated just like a (non wildcard) 
180    <code class="directive"><a href="../mod/core.html#serveralias">ServerAlias</a></code>.</p>
181
182    <p>Finally, you can fine-tune the configuration of the virtual hosts
183    by placing other directives inside the <code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code> containers. Most directives can be
184    placed in these containers and will then change the configuration only of
185    the relevant virtual host. To find out if a particular directive is allowed,
186    check the <a href="../mod/directive-dict.html#Context">Context</a> of the
187    directive. Configuration directives set in the <em>main server context</em>
188    (outside any <code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code>
189    container) will be used only if they are not overridden by the virtual host
190    settings.</p>
