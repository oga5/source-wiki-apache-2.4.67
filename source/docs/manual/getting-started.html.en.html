44<h2><a name="clientserver" id="clientserver">Clients, Servers, and URLs</a></h2>
45
46
47<p>
48Addresses on the Web are expressed with URLs - Uniform Resource Locators
49- which specify a protocol (e.g. <code>http</code>), a servername (e.g.
50<code>www.apache.org</code>), a URL-path (e.g.
51<code>/docs/current/getting-started.html</code>), and possibly a query
52string (e.g. <code>?arg=value</code>) used to pass additional
53arguments to the server.
54</p>
55
56<p>A client (e.g., a web browser) connects to a server (e.g., your Apache HTTP Server),
57with the specified protocol, and makes a <strong>request</strong> for a resource using the
58URL-path.</p>
59
60<p>The URL-path may represent any number of things on the server. It may
61be a file (like <code>getting-started.html</code>) a handler (like <a href="mod/mod_status.html">server-status</a>) or some kind of program
62file (like <code>index.php</code>). We'll discuss this more below in
63the <a href="#content">Web Site Content</a> section.</p>
64
65<p>
66The server will send a <strong>response</strong> consisting of a status
67code and, optionally, a response body.
68The status code indicates whether the request was successful, and, if not, what
69kind of error condition there was. This tells the client what it should
70do with the response. You can read about the possible response codes in
71<a href="https://cwiki.apache.org/confluence/display/httpd/CommonHTTPStatusCodes">HTTP Server
72wiki</a>.</p>
73
74<p>Details of the transaction, and any error conditions, are written to
75log files. This is discussed in greater detail below in the <a href="#logs">Logs Files and Troubleshooting</a> section.</p>

79<h2><a name="dns" id="dns">Hostnames and DNS</a></h2>
80
81
82<p>In order to connect to a server, the client will first have to resolve
83the servername to an IP address - the location on the Internet where the
84server resides. Thus, in order for your web server to be reachable, it
85is necessary that the servername be in DNS.</p>
86
87<p>If you don't know how to do this, you'll need to contact your network
88administrator, or Internet service provider, to perform this step for
89you.</p>
90
91<p>More than one hostname may point to the same IP address, and more
92than one IP address can be attached to the same physical server. Thus, you
93can run more than one web site on the same physical server, using a
94feature called <a href="vhosts/">virtual hosts</a>.</p>
95
96<p>If you are testing a server that is not Internet-accessible, you
97can put host names in your hosts file in order to do local resolution.
98For example, you might want to put a record in your hosts file to map a
99request for <code>www.example.com</code> to your local system, for
100testing purposes. This entry would look like:</p>
101
102<div class="example"><p><code>
103127.0.0.1 www.example.com
104</code></p></div>
105
106<p>A hosts file will probably be located at <code>/etc/hosts</code> or
107<code>C:\Windows\system32\drivers\etc\hosts</code>.</p>
108
109<p>You can read more about the hosts file at <a href="http://en.wikipedia.org/wiki/Hosts_(file)">Wikipedia.org/wiki/Hosts_(file)</a>, and
110more about DNS at <a href="http://en.wikipedia.org/wiki/Domain_Name_System">Wikipedia.org/wiki/Domain_Name_System</a>.</p>

113<h2><a name="configuration" id="configuration">Configuration Files and Directives</a></h2>
114
115
116<p>The Apache HTTP Server is configured via simple text files.
117These files may be located any of a variety of places, depending on how
118exactly you installed the server. Common locations for these files may
119be found <a href="https://cwiki.apache.org/confluence/display/httpd/DistrosDefaultLayout">in
120the httpd wiki</a>. If you installed httpd from source, the default
121location of the configuration files is
122<code>/usr/local/apache2/conf</code>. The default configuration file is
123usually called <code>httpd.conf</code>. This, too, can vary in
124third-party distributions of the server.</p>
125
126<p>The configuration is frequently broken into multiple smaller files,
127for ease of management. These files are loaded via the <code class="directive"><a href="./mod/core.html#include">Include</a></code> directive. The names or locations of
128these sub-files are not magical, and may vary greatly from one
129installation to another. Arrange and subdivide these files as
130makes the most sense to <strong>you</strong>. If the file arrangement
131you have by default doesn't make sense to you, feel free to rearrange it.</p>
132
133<p>The server is configured by placing <a href="mod/quickreference.html">configuration directives</a> in these
134configuration files. A directive is a keyword followed by one or more
135arguments that set its value.</p>
136
137<p>The question of "<em>Where should I put that
138directive?</em>" is generally answered by considering where you want a
139directive to be effective. If it is a global setting, it should appear
140in the configuration file, outside of any <code class="directive"><a href="./mod/core.html#directory">&lt;Directory&gt;</a></code>, <code class="directive"><a href="./mod/core.html#location">&lt;Location&gt;</a></code>, <code class="directive"><a href="./mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code>, or other section. If it is to
141apply only to a particular directory, then it should go inside a
142<code class="directive"><a href="./mod/core.html#directory">&lt;Directory&gt;</a></code> section referring to
143that directory, and so on. See the <a href="sections.html">Configuration
144Sections</a> document for further discussion of these sections.</p>
145
146<p>In addition to the main configuration files, certain directives may go in
147<code>.htaccess</code> files located in the content directories.
148<code>.htaccess</code> files are primarily for people who do not have
149access to the main server configuration file(s). You can read more about
150<code>.htaccess</code> files in the <a href="howto/htaccess.html"><code>.htaccess</code> howto</a>.</p>

154<h2><a name="content" id="content">Web Site Content</a></h2>
155
156
157<p>Web site content can take many different forms, but may be broadly
158divided into static and dynamic content.</p>
159
160<p>Static content is things like HTML files, image files, CSS files,
161and other files that reside in the filesystem. The <code class="directive"><a href="./mod/core.html#documentroot">DocumentRoot</a></code> directive specifies where in your
162filesystem you should place these files. This directive is either set
163globally, or per virtual host. Look in your configuration file(s) to
164determine how this is set for your server.</p>
165
166<p>Typically, a document called <code>index.html</code> will be served
167when a directory is requested without a file name being specified. For
168example, if <code>DocumentRoot</code> is set to
169<code>/var/www/html</code> and a request is made for
170<code>http://www.example.com/work/</code>, the file
171<code>/var/www/html/work/index.html</code> will be served to the
172client.</p>
173
174<p>Dynamic content is anything that is generated at request
175time, and may change from one request to another. There are numerous
176ways that dynamic content may be generated. Various <a href="handler.html">handlers</a> are available to generate content. <a href="howto/cgi.html">CGI programs</a> may be written to generate
177content for your site.</p>
178
179<p>Third-party modules like mod_php may be used to write code that does a
180variety of things. Many third-party applications, written using a
181variety of languages and tools, are available for download and
182installation on your Apache HTTP Server. Support of these third-party
183things is beyond the scope of this documentation, and you should find
184their documentation or other support forums to answer your questions
185about them.</p>

188<h2><a name="logs" id="logs">Log Files and Troubleshooting</a></h2>
189
190<p>As an Apache HTTP Server administrator, your most valuable assets are
191the log files, and, in particular, the error log. Troubleshooting any
192problem without the error log is like driving with your eyes closed.</p>
193
194<p>The location of the error log is defined by the <code class="directive"><a href="./mod/core.html#errorlog">ErrorLog</a></code> directive, which may be set globally,
195or per virtual host. Entries in the error log tell you what went wrong,
196and when. They often also tell you how to fix it. Each error log message
197contains an error code, which you can search for online for even more
198detailed descriptions of how to address the problem. You can also
199configure your error log to contain a log ID which you can then
200correlate to an access log entry, so that you can determine what request
201caused the error condition.</p>
202
203<p>You can read more about logging in the <a href="logs.html">logs
204documentation</a>.</p>
