37    <p>Apache HTTP サーバはモジュール化されたプログラムで、
38    管理者がモジュールを選択することでサーバに組み込む機能を選ぶことができます。
39    モジュールはサーバがビルドされるときに <code class="program"><a href="./programs/httpd.html">httpd</a></code> バイナリに
40    静的に組み込むことができます。もしくは、<code class="program"><a href="./programs/httpd.html">httpd</a></code> バイナリとは
41    別に存在する動的共有オブジェクト (訳注: Dynamic Shared Object)
42    (DSO) としてコンパイルすることも
43    できます。DSO モジュールはサーバがビルドされるときにコンパイルしたり、
44    Apache 拡張ツール (<code class="program"><a href="./programs/apxs.html">apxs</a></code>) を
45    使って後でコンパイルして追加したりできます。</p>
46
47    <p>この文書は DSO モジュールの使い方と、仕組みについて
48    説明します。</p>

57<h2><a name="implementation" id="implementation">実装</a></h2>
58
59<table class="related"><tr><th>関連モジュール</th><th>関連ディレクティブ</th></tr><tr><td><ul><li><code class="module"><a href="./mod/mod_so.html">mod_so</a></code></li></ul></td><td><ul><li><code class="directive"><a href="./mod/mod_so.html#loadmodule">LoadModule</a></code></li></ul></td></tr></table>
60
61    <p>個々の Apache モジュールをロードするための DSO サポートは
62    <code class="module"><a href="./mod/mod_so.c.html">mod_so.c</a></code> というモジュールの機能に基づいています。
63    このモジュール は Apache のコアに静的に組み込まれている必要があります。
64    それは <code class="module"><a href="./mod/core.c.html">core.c</a></code> 以外では DSO にできない唯一の
65    モジュールです。事実上、他のすべての Apache のモジュールは、
66    <a href="install.html">インストールの文書</a>で説明されているように、
67    <code class="program"><a href="./programs/configure.html">configure</a></code> の
68    <code>--enable-<em>module</em>=shared</code> オプションでそれぞれを
69    DSO ビルドにすることにより、DSO モジュールにすることができます。
70    <code>mod_foo.so</code> のような DSO にモジュールがコンパイルされれば、
71    <code>httpd.conf</code> ファイル中で <code class="module"><a href="./mod/mod_so.html">mod_so</a></code> の
72    <code class="directive"><a href="./mod/mod_so.html#loadmodule">LoadModule</a></code>
73    ディレクティブを使うことでサーバの起動や再起動時にこのモジュールを
74    ロードするようにできます。</p>
75
76    <p>Apache モジュール用の (特にサードパーティモジュールの) DSO ファイルの
77    作成を簡単にするために、<code class="program"><a href="./programs/apxs.html">apxs</a></code>
78    (<dfn>APache eXtenSion</dfn>) という新しいサポートプログラムがあります。
79    Apache のソースツリーの<em>外で</em> DSO モジュールをビルドするために
80    使うことができます。発想は単純です: Apache のインストール時の
81    <code class="program"><a href="./programs/configure.html">configure</a></code>、<code>make install</code> のときに Apache の
82    C ヘッダをインストールし、DSO ビルド用のプラットフォーム依存の
83    コンパイラとリンカのフラグを <code class="program"><a href="./programs/apxs.html">apxs</a></code> プログラムに追加します。
84    これにより、ユーザが Apache の配布ソースツリーなしで、さらに
85    DSO サポートのためのプラットフォーム依存のコンパイラやリンカの
86    フラグをいじることなく Apache のモジュールのソースをコンパイル
87    できるようになります。</p>

90<h2><a name="usage" id="usage">使用法の概要</a></h2>
91
92    <p>Apache 2.x の DSO 機能の概略を知ることができるための、
93    短く簡潔な概要です:</p>
94
95    <ol>
96      <li>
97        <em>配布されている</em> Apache モジュール、仮に <code>mod_foo.c</code>
98        として、それを DSO <code>mod_foo.so</code> にビルド、インストール: 
99
100<div class="example"><p><code>
101$ ./configure --prefix=/path/to/install --enable-foo=shared<br />
102$ make install
103</code></p></div>
104      </li>
105
106      <li>
107        <em>サードパーティ</em> Apache モジュール、仮に <code>mod_foo.c</code>
108        として、それを DSO <code>mod_foo.so</code> にビルド、インストール:
109
110<div class="example"><p><code>
111$ ./configure --add-module=<var>module_type</var>:/path/to/3rdparty/mod_foo.c \<br />
112<span class="indent">
113  --enable-foo=shared<br />
114</span>
115$ make install
116</code></p></div>
117      </li>
118
119      <li>
120        共有モジュールの <em>後々のインストール</em> のために
121        Apache を設定:
122
123<div class="example"><p><code>
124$ ./configure --enable-so<br />
125$ make install
126</code></p></div>
127      </li>
128
129      <li>
130        <em>サードパーティ</em> Apache モジュール、仮に <code>mod_foo.c</code>
131        として、それを <code class="program"><a href="./programs/apxs.html">apxs</a></code> を使って
132        Apache ソースツリーの<em>外で</em> DSO にビルド、インストール:
133
134<div class="example"><p><code>
135$ cd /path/to/3rdparty<br />
136$ apxs -c mod_foo.c<br />
137$ apxs -i -a -n foo mod_foo.la
138</code></p></div>
139      </li>
140    </ol>
141
142    <p>どの場合においても、共有モジュールをコンパイルした後で、
143    <code>httpd.conf</code> で 
144    <code class="directive"><a href="./mod/mod_so.html#loadmodule">LoadModule</a></code>
145    ディレクティブを使って Apache がモジュールを使用するように
146    しなければなりません。</p>

149<h2><a name="background" id="background">背景</a></h2>
150
151    <p>最近の Unix 系の OS には <em>動的共有オブジェクト</em> (DSO)
152    の動的リンク/ロードという気のきいた機構が
153    存在します。これは、実行時にプログラムのアドレス空間に
154    ロードできるような特別な形式でプログラムをビルドすることを
155    可能にします。</p>
156
157    <p>このロードは二つの方法で行なうことができます: 実行プログラムが
158    起動されたときに <code>ld.so</code> というシステムプログラム
159    により自動的に行なわれる方法と、実行プログラム中から、システムコール
160    <code>dlopen()/dlsym()</code> による Unix ローダへの
161    プログラムシステムのインタフェースを使って手動で行なう方法とが
162    あります。</p>
163
164    <p>最初の方法では DSO は普通は<em>共有ライブラリ</em>や <em>DSO
165    ライブラリ</em> と呼ばれていて、DSO の名前は
166    <code>libfoo.so</code> や <code>libfoo.so.1.2</code> のようになっています。
167    これらはシステムディレクトリ (通常 <code>/usr/lib</code>) に存在し、
168    実行プログラムへのリンクはビルド時に <code>-lfoo</code> をリンカに
169    指定することで確立されます。これによりライブラリへの参照が実行プログラムの
170    ファイルに書き込まれて、起動時に Unix のローダが <code>/usr/lib</code> や、
171    リンカの <code>-R</code> のようなオプションによりハードコードされたパス、
172    環境変数 <code>LD_LIBRARY_PATH</code> により設定されたパス、の中から
173    <code>libfoo.so</code> の場所を見つけることができます。それから、
174    実行プログラム中の (まだ未解決の) シンボルを DSO にあるシンボルで
175    解決します。</p>
176
177    <p>普通は実行プログラム中のシンボルは DSO からは参照されません
178    (DSO は一般的なコードによる再利用可能なライブラリですので)。
179    ですから、さらなるシンボルの解決は必要ありません。
180    シンボルは Unix ローダにより完全な解決が行なわれますので、実行ファイル自身は
181    何もする必要がありません。(実際のところ、静的でない方法でリンクされている
182    すべての実行プログラムに組み込まれている開始用のコードの一部に
183    <code>ld.so</code> を起動するコードが含まれています)。よく使われる
184    ライブラリの動的ロードの利点は明らかです。ライブラリのコードは
185    システムライブラリに <code>libc.so</code> のようにして一度保存するだけでよく、
186    プログラムのために必要なディスクの領域を節約することができます。</p>
187
188    <p>二つめの方法では DSO は普通は<em>共有オブジェクト</em>や
189    <em>DSO ファイル</em>と呼ばれていて、任意の拡張子を付けることができます
190    (ただし、標準的な名前は <code>foo.so</code> です)。
191    これらのファイルは通常はプログラム専用のディレクトリに置かれ、
192    これらを使う実行プログラムへのリンクは自動的にはされません。
193    ですので、実行プログラムは <code>dlopen()</code> を使って
194    実行時に手動で DSO をプログラムのアドレス空間にロードする必要があります。
195    この時点では実行プログラムに対して DSO のシンボルの解決は行なわれません。
196    しかし、その代わりに Unix のローダが DSO の (まだ未解決の) シンボルを
197    実行プログラムによりエクスポートされたシンボルと既にロードされた
198    DSO ライブラリによりエクスポートされたシンボル (特に、どこにでもある
199    <code>libc.so</code> のすべてのシンボル) で自動的に解決します。
200    こうすることで、DSO は最初から静的にリンクされていたかのように、
201    実行プログラムのシンボルを知ることができます。</p>
202
203    <p>最後に、DSO の API を利点を生かすために、プログラムは
204    後でディスパッチテーブル<em>など</em>でシンボルを使うことができるように、
205    <code>dlsym()</code> を使っていくつかのシンボルを解決します。
206    すなわち: 実行プログラムは必要なすべてのシンボルを手動で解決しなければ
207    なりません。この機構の利点はプログラムのオプショナルな部分は
208    必要になるまでロードする必要がない (だからメモリも消費しない)
209    ことです。必要ならば、基本プログラムの機能を拡張するために
210    これらの部分を動的にロードすることができます。</p>
211
212    <p>この DSO 機構は簡単なように見えますが、少なくとも一つ難しい点が
213    あります: プログラムを拡張するために DSO を使っているときに、
214    DSO が実行プログラムからシンボルを解決する点です (二番目の方法)。
215    これはなぜでしょうか。それは、DSO のシンボルを実行プログラムの
216    シンボルから「逆解決」するというのはライブラリの設計
217    (ライブラリはそれを使用するプログラムのことは何も
218    知らない) に反していて、この機能はすべてのプラットフォームに
219    あるわけではなく、標準化もされていないからです。
220    実際には実行プログラムのグローバルなシンボルは再エクスポートされることは
221    あまりなく、DSO から使うことができません。リンカにグローバルシンボルすべてを
222    エクスポートするようにさせる方法を見つけることが、実行時にプログラムを
223    拡張するために DSO を使うときの一番の問題です。</p>
224
225    <p>共有ライブラリのアプローチが普通の方法です。DSO 機構はそのために
226    設計されたものですから。したがって、その方法はオペレーティングシステムが
227    提供するほとんどすべての種類のライブラリで使われています。
228    一方、プログラムの拡張のために共有オブジェクトを使用する、という方は
229    あまり使われていません。</p>
230
231    <p>1998 年の時点で、実行時に実際に機能拡張のために DSO 機構を使っている
232    ソフトウェアパッケージは少しだけでした: Perl 5 (XS 機構と DnaLoader モジュール
233    によるもの)、Netscape サーバ<em>など</em>です。Apache はすでに
234    モジュールの概念を使って機能拡張をしていて、内部的にディスパッチリストに
235    基づいた外部モジュールの Apache コア機能へのリンクを行なっていましたので、
236    バージョン 1.3 から、Apache も DSO 機構を使う仲間になりました。
237    Apache は実行時に DSO を使ってモジュールをロードするようにすでに
238    運命付けられていたのです。</p>

241<h2><a name="advantages" id="advantages">利点と欠点</a></h2>
242
243    <p>上記の DSO に基づいた機能は以下の利点があります:</p>
244
245    <ul>
246      <li>実際のサーバプロセスを組み立てるために、
247      ビルド時に <code>configure</code> のオプションを使う代わりに
248      実行時に <code>httpd.conf</code> の設定用コマンド
249      <code class="directive"><a href="./mod/mod_so.html#loadmodule">LoadModule</a></code>
250      を使うことができますので、サーバパッケージの柔軟性が高まりました。
251      たとえば、一つの Apache のインストールから
252      違う構成のサーバ (標準版と SSL 版、最小構成と拡張版 [mod_perl, PHP3]
253      <em>など</em>) を実行することができます。</li>
254
255      <li>インストールの後であっても、サーバのパッケージをサードパーティ
256      モジュールで簡単に拡張できるようになりました。これは、Apache コア
257      パッケージと、PHP3, mod_perl, mod_fastcgi <em>など</em> の追加の
258      パッケージを作成できるので、少なくともベンダのパッケージ管理者にとって
259      大きな利点があります。</li>
260
261      <li>Apache モジュールの開発が簡単になります。
262      これは DSO と <code class="program"><a href="./programs/apxs.html">apxs</a></code> の組み合わせにより、Apache ソースツリーの
263      外で作業でき、開発中のモジュールの新しいバージョンを
264      実行中の Apache サーバに組み込むために <code>apxs -i</code> と
265      <code>apachectl restart</code> を行なうだけで良くなるからです。</li>
266    </ul>
267
268    <p>DSO には以下の欠点があります:</p>
269
270    <ul>
271      <li>すべてのオペレーティングシステムがプログラムのアドレス空間に
272      コードを動的ロードすることをサポートしているわではないので、
273      プラットフォームによっては DSO 機構は使えません。</li>
274
275      <li>Unix のローダがシンボルの解決をする必要ができたので、
276      そのオーバヘッドによりサーバの起動時間が約 20% 遅くなっています。</li>
277
278      <li>位置非依存コード (PIC) (訳注 position independent code) は
279      相対アドレスのために複雑なアセンブラのトリックが必要なことがあり、
280      それは必ずしも絶対アドレスと同じくらいの速度がでるわけではありませんので、
281      プラットフォームによってはサーバの実行速度が約 5% 遅くなります。</li>
282
283      <li>DSO モジュールはすべてのプラットフォームで他の DSO に基づいた
284      ライブラリに対してリンクできる (<code>ld -lfoo</code>)
285      というわけではありませんので (たとえば、a.out のプラットフォームでは
286      この機能はありませんが、ELF のプラットフォームにはあります)、
287      すべての種類のモジュールに DSO 機構を使えるわけではありません。
288      言い換えると、DSO ファイルとしてコンパイルされたモジュールの
289      使えるシンボルは、
290      Apache のコアのシンボル、C ライブラリ (<code>libc</code>) と
291      Apache コアが使っている他のすべての静的なライブラリと動的ライブラリの
292      シンボル、PIC による静的なライブラリ (<code>libfoo.a</code>) の
293      シンボルのみに制限されます。その他のコードを使う方法は、
294      Apache コア自身がすでにそのコードへの参照があるようにするか、
295      <code>dlopen ()</code> を使ってコードを自分自身でロードするかの
296     どちらかしかありません。</li>
297    </ul>
