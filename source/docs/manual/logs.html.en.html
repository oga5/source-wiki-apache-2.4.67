33    <p>In order to effectively manage a web server, it is necessary
34    to get feedback about the activity and performance of the
35    server as well as any problems that may be occurring. The Apache HTTP Server
36    provides very comprehensive and flexible logging
37    capabilities. This document describes how to configure its
38    logging capabilities, and how to understand what the logs
39    contain.</p>
40  </div>

53<h2><a name="overview" id="overview">Overview</a></h2>
54    
55
56  <table class="related"><tr><th>Related Modules</th><th>Related Directives</th></tr><tr><td><ul><li><code class="module"><a href="./mod/mod_log_config.html">mod_log_config</a></code></li><li><code class="module"><a href="./mod/mod_log_forensic.html">mod_log_forensic</a></code></li><li><code class="module"><a href="./mod/mod_logio.html">mod_logio</a></code></li><li><code class="module"><a href="./mod/mod_cgi.html">mod_cgi</a></code></li></ul></td><td /></tr></table>

58  <p>
59  The Apache HTTP Server provides a variety of different mechanisms for
60  logging everything that happens on your server, from the initial
61  request, through the URL mapping process, to the final resolution of
62  the connection, including any errors that may have occurred in the
63  process. In addition to this, third-party modules may provide logging
64  capabilities, or inject entries into the existing log files, and
65  applications such as CGI programs, or PHP scripts, or other handlers,
66  may send messages to the server error log.
67  </p>
68
69  <p>
70  In this document we discuss the logging modules that are a standard
71  part of the http server.
72  </p>
73
74  </div><div class="top"><a href="#page-header"><img alt="top" src="./images/up.gif" /></a></div>

76<h2><a name="security" id="security">Security Warning</a></h2>
77    
78
79    <p>Anyone who can write to the directory where Apache httpd is
80    writing a log file can almost certainly gain access to the uid
81    that the server is started as, which is normally root. Do
82    <em>NOT</em> give people write access to the directory the logs
83    are stored in without being aware of the consequences; see the
84    <a href="misc/security_tips.html">security tips</a> document
85    for details.</p>
86
87    <p>In addition, log files may contain information supplied
88    directly by the client, without escaping. Therefore, it is
89    possible for malicious clients to insert control-characters in
90    the log files, so care must be taken in dealing with raw
91    logs.</p>
92  </div><div class="top"><a href="#page-header"><img alt="top" src="./images/up.gif" /></a></div>

94<h2><a name="errorlog" id="errorlog">Error Log</a></h2>
95    
96
97    <table class="related"><tr><th>Related Modules</th><th>Related Directives</th></tr><tr><td><ul><li><code class="module"><a href="./mod/core.html">core</a></code></li></ul></td><td><ul><li><code class="directive"><a href="./mod/core.html#errorlog">ErrorLog</a></code></li><li><code class="directive"><a href="./mod/core.html#errorlogformat">ErrorLogFormat</a></code></li><li><code class="directive"><a href="./mod/core.html#loglevel">LogLevel</a></code></li></ul></td></tr></table>
98
99    <p>The server error log, whose name and location is set by the
100    <code class="directive"><a href="./mod/core.html#errorlog">ErrorLog</a></code> directive, is the
101    most important log file. This is the place where Apache httpd
102    will send diagnostic information and record any errors that it
103    encounters in processing requests. It is the first place to
104    look when a problem occurs with starting the server or with the
105    operation of the server, since it will often contain details of
106    what went wrong and how to fix it.</p>
107
108    <p>The error log is usually written to a file (typically
109    <code>error_log</code> on Unix systems and
110    <code>error.log</code> on Windows and OS/2). On Unix systems it
111    is also possible to have the server send errors to
112    <code>syslog</code> or <a href="#piped">pipe them to a
113    program</a>.</p>
114
115    <p>The format of the error log is defined by the <code class="directive"><a href="./mod/core.html#errorlogformat">ErrorLogFormat</a></code> directive, with which you
116    can customize what values are logged. A default is format defined
117    if you don't specify one. A typical log message follows:</p>
118
119    <div class="example"><p><code>
120    [Fri Sep 09 10:42:29.902022 2011] [core:error] [pid 35708:tid 4328636416]
121    [client 72.15.99.187] AH00124: Request exceeded the limit of 10 internal redirects due to probable configuration error
122    </code></p></div>
123
124    <p>The first item in the log entry is the date and time of the
125    message. The next is the module producing the message (core, in this
126    case) and the severity level of that message. This is followed by
127    the process ID and, if appropriate, the thread ID, of the process
128    that experienced the condition. Next, we have the client address
129    that made the request. And finally is the detailed error message.</p>
130
131    <div class="note"><p>Note that some messages that were previously logged at
132    <code>error</code> level have been downgraded to <code>info</code>
133    level in httpd 2.4. For example, "File does not exist" messages
134    for 404 responses are logged at <code>info</code> level and will
135    not appear in the error log with the default
136    <code class="directive"><a href="./mod/core.html#loglevel">LogLevel</a></code> of <code>warn</code>.
137    To restore the previous behavior, use
138    <code>LogLevel warn core:info</code>.</p></div>
139
140    <p>A very wide variety of different messages can appear in the
141    error log. Most look similar to the example above. The error
142    log will also contain debugging output from CGI scripts. Any
143    information written to <code>stderr</code> by a CGI script will
144    be copied directly to the error log.</p>
145
146    <p>Putting a <code>%L</code> token in both the error log and the access
147    log will produce a log entry ID with which you can correlate the entry
148    in the error log with the entry in the access log. If
149    <code class="module"><a href="./mod/mod_unique_id.html">mod_unique_id</a></code> is loaded, its unique request ID will be
150    used as the log entry ID, too.</p>
151
152    <p>During testing, it is often useful to continuously monitor
153    the error log for any problems. On Unix systems, you can
154    accomplish this using:</p>
155
156    <div class="example"><p><code>
157      tail -f error_log
158    </code></p></div>

161<h2><a name="permodule" id="permodule">Per-module logging</a></h2>
162    
163
164    <p>The <code class="directive"><a href="./mod/core.html#loglevel">LogLevel</a></code> directive
165    allows you to specify a log severity level on a per-module basis. In
166    this way, if you are troubleshooting a problem with just one
167    particular module, you can turn up its logging volume without also
168    getting the details of other modules that you're not interested in.
169    This is particularly useful for modules such as
170    <code class="module"><a href="./mod/mod_proxy.html">mod_proxy</a></code> or <code class="module"><a href="./mod/mod_rewrite.html">mod_rewrite</a></code> where you
171    want to know details about what it's trying to do.</p>
172
173    <p>Do this by specifying the name of the module in your
174    <code class="directive">LogLevel</code> directive:</p>
175
176    <pre class="prettyprint lang-config">LogLevel info rewrite:trace5</pre>
177
178
179    <p>This sets the main <code class="directive">LogLevel</code> to info, but
180    turns it up to <code>trace5</code> for
181    <code class="module"><a href="./mod/mod_rewrite.html">mod_rewrite</a></code>.</p>
182
183    <div class="note">This replaces the per-module logging directives, such as
184    <code>RewriteLog</code>, that were present in earlier versions of
185    the server.</div>

188<h2><a name="accesslog" id="accesslog">Access Log</a></h2>
189    
190
191    <table class="related"><tr><th>Related Modules</th><th>Related Directives</th></tr><tr><td><ul><li><code class="module"><a href="./mod/mod_log_config.html">mod_log_config</a></code></li><li><code class="module"><a href="./mod/mod_setenvif.html">mod_setenvif</a></code></li></ul></td><td><ul><li><code class="directive"><a href="./mod/mod_log_config.html#customlog">CustomLog</a></code></li><li><code class="directive"><a href="./mod/mod_log_config.html#logformat">LogFormat</a></code></li><li><code class="directive"><a href="./mod/mod_setenvif.html#setenvif">SetEnvIf</a></code></li></ul></td></tr></table>
192
193    <p>The server access log records all requests processed by the
194    server. The location and content of the access log are
195    controlled by the <code class="directive"><a href="./mod/mod_log_config.html#customlog">CustomLog</a></code>
196    directive. The <code class="directive"><a href="./mod/mod_log_config.html#logformat">LogFormat</a></code>
197    directive can be used to simplify the selection of
198    the contents of the logs. This section describes how to configure the server
199    to record information in the access log.</p>
200
201    <p>Storing the information in the access log is only
202    the start of log management. The next step is to analyze this
203    information to produce useful statistics. Log analysis in
204    general is beyond the scope of this document, and not really
205    part of the job of the web server itself.
206    </p>
207
208    <p>Various versions of Apache httpd have used other modules and
209    directives to control access logging, including
210    mod_log_referer, mod_log_agent, and the
211    <code>TransferLog</code> directive. The <code class="directive"><a href="./mod/mod_log_config.html#customlog">CustomLog</a></code> directive now subsumes
212    the functionality of all the older directives.</p>
213
214    <p>The format of the access log is highly configurable. The format
215    is specified using a format string that looks much like a C-style
216    printf(1) format string. Some examples are presented in the next
217    sections. For a complete list of the possible contents of the
218    format string, see the <code class="module"><a href="./mod/mod_log_config.html">mod_log_config</a></code> <a href="mod/mod_log_config.html#formats">format strings</a>.</p>
219
220    <h3><a name="common" id="common">Common Log Format</a></h3>
221      
222
223      <p>A typical configuration for the access log might look as
224      follows.</p>
225
226      <pre class="prettyprint lang-config">LogFormat "%h %l %u %t \"%r\" %&gt;s %b" common
227CustomLog logs/access_log common</pre>
228
229
230      <p>This defines the <em>nickname</em> <code>common</code> and
231      associates it with a particular log format string. The format
232      string consists of percent directives, each of which tell the
233      server to log a particular piece of information. Literal
234      characters may also be placed in the format string and will be
235      copied directly into the log output. The quote character
236      (<code>"</code>) must be escaped by placing a backslash before
237      it to prevent it from being interpreted as the end of the
238      format string. The format string may also contain the special
239      control characters "<code>\n</code>" for new-line and
240      "<code>\t</code>" for tab.</p>
241
242      <p>The <code class="directive"><a href="./mod/mod_log_config.html#customlog">CustomLog</a></code>
243      directive sets up a new log file using the defined
244      <em>nickname</em>. The filename for the access log is relative to
245      the <code class="directive"><a href="./mod/core.html#serverroot">ServerRoot</a></code> unless it
246      begins with a slash.</p>
247
248      <p>The above configuration will write log entries in a format
249      known as the Common Log Format (CLF). This standard format can
250      be produced by many different web servers and read by many log
251      analysis programs. The log file entries produced in CLF will
252      look something like this:</p>
253
254      <div class="example"><p><code>
255        127.0.0.1 - frank [10/Oct/2000:13:55:36 -0700] "GET
256        /apache_pb.gif HTTP/1.0" 200 2326
257      </code></p></div>
258
259      <p>Each part of this log entry is described below.</p>
260
261      <dl>
262        <dt><code>127.0.0.1</code> (<code>%h</code>)</dt>
263
264        <dd>This is the IP address of the client (remote host) which
265        made the request to the server. If <code class="directive"><a href="./mod/core.html#hostnamelookups">HostnameLookups</a></code> is
266        set to <code>On</code>, then the server will try to determine
267        the hostname and log it in place of the IP address. However,
268        this configuration is not recommended since it can
269        significantly slow the server. Instead, it is best to use a
270        log post-processor such as <code class="program"><a href="./programs/logresolve.html">logresolve</a></code> to determine
271        the hostnames. The IP address reported here is not
272        necessarily the address of the machine at which the user is
273        sitting. If a proxy server exists between the user and the
274        server, this address will be the address of the proxy, rather
275        than the originating machine.</dd>
276
277        <dt><code>-</code> (<code>%l</code>)</dt>
278
279        <dd>The "hyphen" in the output indicates that the requested
280        piece of information is not available. In this case, the
281        information that is not available is the RFC 1413 identity of
282        the client determined by <code>identd</code> on the clients
283        machine. This information is highly unreliable and should
284        almost never be used except on tightly controlled internal
285        networks. Apache httpd will not even attempt to determine
286        this information unless <code class="directive"><a href="./mod/mod_ident.html#identitycheck">IdentityCheck</a></code> is set
287        to <code>On</code>.</dd>
288
289        <dt><code>frank</code> (<code>%u</code>)</dt>
290
291        <dd>This is the userid of the person requesting the document
292        as determined by HTTP authentication. The same value is
293        typically provided to CGI scripts in the
294        <code>REMOTE_USER</code> environment variable. If the status
295        code for the request (see below) is 401, then this value
296        should not be trusted because the user is not yet
297        authenticated. If the document is not password protected,
298        this part will be "<code>-</code>" just like the previous
299        one.</dd>
300
301        <dt><code>[10/Oct/2000:13:55:36 -0700]</code>
302        (<code>%t</code>)</dt>
303
304        <dd>
305          The time that the request was received.
306          The format is:
307
308          <p class="indent">
309            <code>[day/month/year:hour:minute:second zone]<br />
310             day = 2*digit<br />
311             month = 3*letter<br />
312             year = 4*digit<br />
313             hour = 2*digit<br />
314             minute = 2*digit<br />
315             second = 2*digit<br />
316             zone = (`+' | `-') 4*digit</code>
317          </p>
318          <p>It is possible to have the time displayed in another format
319          by specifying <code>%{format}t</code> in the log format
320          string, where <code>format</code> is either as in
321          <code>strftime(3)</code> from the C standard library,
322          or one of the supported special tokens. For details see
323          the <code class="module"><a href="./mod/mod_log_config.html">mod_log_config</a></code> <a href="mod/mod_log_config.html#formats">format strings</a>.</p>
324        </dd>
325
326        <dt><code>"GET /apache_pb.gif HTTP/1.0"</code>
327        (<code>\"%r\"</code>)</dt>
328
329        <dd>The request line from the client is given in double
330        quotes. The request line contains a great deal of useful
331        information. First, the method used by the client is
332        <code>GET</code>. Second, the client requested the resource
333        <code>/apache_pb.gif</code>, and third, the client used the
334        protocol <code>HTTP/1.0</code>. It is also possible to log
335        one or more parts of the request line independently. For
336        example, the format string "<code>%m %U%q %H</code>" will log
337        the method, path, query-string, and protocol, resulting in
338        exactly the same output as "<code>%r</code>".</dd>
339
340        <dt><code>200</code> (<code>%&gt;s</code>)</dt>
341
342        <dd>This is the status code that the server sends back to the
343        client. This information is very valuable, because it reveals
344        whether the request resulted in a successful response (codes
345        beginning in 2), a redirection (codes beginning in 3), an
346        error caused by the client (codes beginning in 4), or an
347        error in the server (codes beginning in 5). The full list of
348        possible status codes can be found in the <a href="http://www.w3.org/Protocols/rfc2616/rfc2616.txt">HTTP
349        specification</a> (RFC2616 section 10).</dd>
350
351        <dt><code>2326</code> (<code>%b</code>)</dt>
352
353        <dd>The last part indicates the size of the object returned
354        to the client, not including the response headers. If no
355        content was returned to the client, this value will be
356        "<code>-</code>". To log "<code>0</code>" for no content, use
357        <code>%B</code> instead.</dd>
358      </dl>
359    
360
361    <h3><a name="combined" id="combined">Combined Log Format</a></h3>
362      
363
364      <p>Another commonly used format string is called the Combined
365      Log Format. It can be used as follows.</p>
366
367      <pre class="prettyprint lang-config">LogFormat "%h %l %u %t \"%r\" %&gt;s %b \"%{Referer}i\" \"%{User-agent}i\"" combined
368CustomLog log/access_log combined</pre>
369
370
371      <p>This format is exactly the same as the Common Log Format,
372      with the addition of two more fields. Each of the additional
373      fields uses the percent-directive
374      <code>%{<em>header</em>}i</code>, where <em>header</em> can be
375      any HTTP request header. The access log under this format will
376      look like:</p>
377
378      <div class="example"><p><code>
379        127.0.0.1 - frank [10/Oct/2000:13:55:36 -0700] "GET
380        /apache_pb.gif HTTP/1.0" 200 2326
381        "http://www.example.com/start.html" "Mozilla/4.08 [en]
382        (Win98; I ;Nav)"
383      </code></p></div>
384
385      <p>The additional fields are:</p>
386
387      <dl>
388        <dt><code>"http://www.example.com/start.html"</code>
389        (<code>\"%{Referer}i\"</code>)</dt>
390
391        <dd>The "Referer" (sic) HTTP request header. This gives the
392        site that the client reports having been referred from. (This
393        should be the page that links to or includes
394        <code>/apache_pb.gif</code>).</dd>
395
396        <dt><code>"Mozilla/4.08 [en] (Win98; I ;Nav)"</code>
397        (<code>\"%{User-agent}i\"</code>)</dt>
398
399        <dd>The User-Agent HTTP request header. This is the
400        identifying information that the client browser reports about
401        itself.</dd>
402      </dl>

405    <h3><a name="multiple" id="multiple">Multiple Access Logs</a></h3>
406      
407
408      <p>Multiple access logs can be created simply by specifying
409      multiple <code class="directive"><a href="./mod/mod_log_config.html#customlog">CustomLog</a></code>
410      directives in the configuration
411      file. For example, the following directives will create three
412      access logs. The first contains the basic CLF information,
413      while the second and third contain referer and browser
414      information. The last two <code class="directive"><a href="./mod/mod_log_config.html#customlog">CustomLog</a></code> lines show how
415      to mimic the effects of the <code>ReferLog</code> and <code>AgentLog</code> directives.</p>
416
417      <pre class="prettyprint lang-config">LogFormat "%h %l %u %t \"%r\" %&gt;s %b" common
418CustomLog logs/access_log common
419CustomLog logs/referer_log "%{Referer}i -&gt; %U"
420CustomLog logs/agent_log "%{User-agent}i"</pre>
421
422
423      <p>This example also shows that it is not necessary to define a
424      nickname with the <code class="directive"><a href="./mod/mod_log_config.html#logformat">LogFormat</a></code> directive. Instead,
425      the log format can be specified directly in the <code class="directive"><a href="./mod/mod_log_config.html#customlog">CustomLog</a></code> directive.</p>

428    <h3><a name="conditional" id="conditional">Conditional Logs</a></h3>
429      
430
431      <p>There are times when it is convenient to exclude certain
432      entries from the access logs based on characteristics of the
433      client request. This is easily accomplished with the help of <a href="env.html">environment variables</a>. First, an
434      environment variable must be set to indicate that the request
435      meets certain conditions. This is usually accomplished with
436      <code class="directive"><a href="./mod/mod_setenvif.html#setenvif">SetEnvIf</a></code>. Then the
437      <code>env=</code> clause of the <code class="directive"><a href="./mod/mod_log_config.html#customlog">CustomLog</a></code> directive is used to
438      include or exclude requests where the environment variable is
439      set. Some examples:</p>
440
441      <pre class="prettyprint lang-config"># Mark requests from the loop-back interface
442SetEnvIf Remote_Addr "127\.0\.0\.1" dontlog
443# Mark requests for the robots.txt file
444SetEnvIf Request_URI "^/robots\.txt$" dontlog
445# Log what remains
446CustomLog logs/access_log common env=!dontlog</pre>
447
448
449      <p>As another example, consider logging requests from
450      english-speakers to one log file, and non-english speakers to a
451      different log file.</p>
452
453      <pre class="prettyprint lang-config">SetEnvIf Accept-Language "en" english
454CustomLog logs/english_log common env=english
455CustomLog logs/non_english_log common env=!english</pre>
456
457
458      <p>In a caching scenario one would want to know about
459      the efficiency of the cache. A very simple method to
460      find this out would be:</p>
461
462      <pre class="prettyprint lang-config">SetEnv CACHE_MISS 1
463LogFormat "%h %l %u %t "%r " %&gt;s %b %{CACHE_MISS}e" common-cache
464CustomLog logs/access_log common-cache</pre>
465
466
467      <p><code class="module"><a href="./mod/mod_cache.html">mod_cache</a></code> will run before
468      <code class="module"><a href="./mod/mod_env.html">mod_env</a></code> and, when successful, will deliver the
469      content without it. In that case a cache hit will log
470      <code>-</code>, while a cache miss will log <code>1</code>.</p>
471
472      <p>In addition to the <code>env=</code> syntax, <code class="directive"><a href="./mod/mod_log_config.html#logformat">LogFormat</a></code> supports logging values
473      conditional upon the HTTP response code:</p>
474
475      <pre class="prettyprint lang-config">LogFormat "%400,501{User-agent}i" browserlog
476LogFormat "%!200,304,302{Referer}i" refererlog</pre>
477
478
479      <p>In the first example, the <code>User-agent</code> will be
480      logged if the HTTP status code is 400 or 501. In other cases, a
481      literal "-" will be logged instead. Likewise, in the second
482      example, the <code>Referer</code> will be logged if the HTTP
483      status code is <strong>not</strong> 200, 304, or 302. (Note the
484      "!" before the status codes.</p>
485
486      <p>Although we have just shown that conditional logging is very
487      powerful and flexible, it is not the only way to control the
488      contents of the logs. Log files are more useful when they
489      contain a complete record of server activity. It is often
490      easier to simply post-process the log files to remove requests
491      that you do not want to consider.</p>
492    
493  </div><div class="top"><a href="#page-header"><img alt="top" src="./images/up.gif" /></a></div>

495<h2><a name="rotation" id="rotation">Log Rotation</a></h2>
496    
497
498    <p>On even a moderately busy server, the quantity of
499    information stored in the log files is very large. The access
500    log file typically grows 1 MB or more per 10,000 requests. It
501    will consequently be necessary to periodically rotate the log
502    files by moving or deleting the existing logs. This cannot be
503    done while the server is running, because Apache httpd will continue
504    writing to the old log file as long as it holds the file open.
505    Instead, the server must be <a href="stopping.html">restarted</a> after the log files are
506    moved or deleted so that it will open new log files.</p>
507
508    <p>By using a <em>graceful</em> restart, the server can be
509    instructed to open new log files without losing any existing or
510    pending connections from clients. However, in order to
511    accomplish this, the server must continue to write to the old
512    log files while it finishes serving old requests. It is
513    therefore necessary to wait for some time after the restart
514    before doing any processing on the log files. A typical
515    scenario that simply rotates the logs and compresses the old
516    logs to save space is:</p>
517
518    <div class="example"><p><code>
519      mv access_log access_log.old<br />
520      mv error_log error_log.old<br />
521      apachectl graceful<br />
522      sleep 600<br />
523      gzip access_log.old error_log.old
524    </code></p></div>
525
526    <p>Another way to perform log rotation is using <a href="#piped">piped logs</a> as discussed in the next
527    section.</p>
528  </div><div class="top"><a href="#page-header"><img alt="top" src="./images/up.gif" /></a></div>

530<h2><a name="piped" id="piped">Piped Logs</a></h2>
531    
532
533    <p>Apache httpd is capable of writing error and access log
534    files through a pipe to another process, rather than directly
535    to a file. This capability dramatically increases the
536    flexibility of logging, without adding code to the main server.
537    In order to write logs to a pipe, simply replace the filename
538    with the pipe character "<code>|</code>", followed by the name
539    of the executable which should accept log entries on its
540    standard input. The server will start the piped-log process when
541    the server starts, and will restart it if it crashes while the
542    server is running. (This last feature is why we can refer to
543    this technique as "reliable piped logging".)</p>
544
545    <p>Piped log processes are spawned by the parent Apache httpd
546    process, and inherit the userid of that process. This means
547    that piped log programs usually run as root. It is therefore
548    very important to keep the programs simple and secure.</p>
549
550    <p>One important use of piped logs is to allow log rotation
551    without having to restart the server. The Apache HTTP Server
552    includes a simple program called <code class="program"><a href="./programs/rotatelogs.html">rotatelogs</a></code>
553    for this purpose. For example, to rotate the logs every 24 hours, you
554    can use:</p>
555
556    <pre class="prettyprint lang-config">CustomLog "|/usr/local/apache/bin/rotatelogs /var/log/access_log 86400" common</pre>
557
558
559    <p>Notice that quotes are used to enclose the entire command
560    that will be called for the pipe. Although these examples are
561    for the access log, the same technique can be used for the
562    error log.</p>
563
564    <p>As with conditional logging, piped logs are a very powerful
565    tool, but they should not be used where a simpler solution like
566    off-line post-processing is available.</p>
567
568    <p>By default the piped log process is spawned without invoking
569    a shell. Use "<code>|$</code>" instead of "<code>|</code>"
570    to spawn using a shell (usually with <code>/bin/sh -c</code>):</p>
571
572    <pre class="prettyprint lang-config"># Invoke "rotatelogs" using a shell
573CustomLog "|$/usr/local/apache/bin/rotatelogs   /var/log/access_log 86400" common</pre>
574
575
576    <p>This was the default behavior for Apache 2.2.
577    Depending on the shell specifics this might lead to
578    an additional shell process for the lifetime of the logging
579    pipe program and signal handling problems during restart.
580    For compatibility reasons with Apache 2.2 the notation
581    "<code>||</code>" is also supported and equivalent to using
582    "<code>|</code>".</p>
583
584    <div class="note"><h3>Windows note</h3>
585    <p>Note that on Windows, you may run into problems when running many piped
586    logger processes, especially when HTTPD is running as a service. This is
587    caused by running out of desktop heap space. The desktop heap space given
588    to each service is specified by the third argument to the
589    <code>SharedSection</code> parameter in the
590    HKEY_LOCAL_MACHINE\System\CurrentControlSet\Control\SessionManager\SubSystems\Windows
591    registry value. <strong>Change this value with care</strong>; the normal
592    caveats for changing the Windows registry apply, but you might also exhaust
593    the desktop heap pool if the number is adjusted too high.</p>
594    </div>

597<h2><a name="virtualhost" id="virtualhost">Virtual Hosts</a></h2>
598    
599
600    <p>When running a server with many <a href="vhosts/">virtual
601    hosts</a>, there are several options for dealing with log
602    files. First, it is possible to use logs exactly as in a
603    single-host server. Simply by placing the logging directives
604    outside the <code class="directive"><a href="./mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code> sections in the
605    main server context, it is possible to log all requests in the
606    same access log and error log. This technique does not allow
607    for easy collection of statistics on individual virtual
608    hosts.</p>
609
610    <p>If <code class="directive"><a href="./mod/mod_log_config.html#customlog">CustomLog</a></code>
611    or <code class="directive"><a href="./mod/core.html#errorlog">ErrorLog</a></code>
612    directives are placed inside a
613    <code class="directive"><a href="./mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code>
614    section, all requests or errors for that virtual host will be
615    logged only to the specified file. Any virtual host which does
616    not have logging directives will still have its requests sent
617    to the main server logs. This technique is very useful for a
618    small number of virtual hosts, but if the number of hosts is
619    very large, it can be complicated to manage. In addition, it
620    can often create problems with <a href="vhosts/fd-limits.html">insufficient file
621    descriptors</a>.</p>
622
623    <p>For the access log, there is a very good compromise. By
624    adding information on the virtual host to the log format
625    string, it is possible to log all hosts to the same log, and
626    later split the log into individual files. For example,
627    consider the following directives.</p>
628
629    <pre class="prettyprint lang-config">LogFormat "%v %l %u %t \"%r\" %&gt;s %b" comonvhost
630CustomLog logs/access_log comonvhost</pre>
631
632
633    <p>The <code>%v</code> is used to log the name of the virtual
634    host that is serving the request. Then a program like <a href="programs/split-logfile.html">split-logfile</a> can be used to
635    post-process the access log in order to split it into one file
636    per virtual host.</p>

653    <h3>Forensic Logging</h3>
654      
655
656      <p><code class="module"><a href="./mod/mod_log_forensic.html">mod_log_forensic</a></code> provides for forensic logging of
657         client requests. Logging is done before and after processing a
658         request, so the forensic log contains two log lines for each
659         request. The forensic logger is very strict with no customizations.
660         It can be an invaluable debugging and security tool.</p>

663    <h3><a name="pidfile" id="pidfile">PID File</a></h3>
664      
665
666      <p>On startup, Apache httpd saves the process id of the parent
667      httpd process to the file <code>logs/httpd.pid</code>. This
668      filename can be changed with the <code class="directive"><a href="./mod/mpm_common.html#pidfile">PidFile</a></code> directive. The
669      process-id is for use by the administrator in restarting and
670      terminating the daemon by sending signals to the parent
671      process; on Windows, use the -k command line option instead.
672      For more information see the <a href="stopping.html">Stopping
673      and Restarting</a> page.</p>
