53<h3><a name="mutex" id="mutex">Why do I get permission errors related to
54        SSLMutex when I start Apache?</a></h3>
55    <p>Errors such as ``<code>mod_ssl: Child could not open
56    SSLMutex lockfile /opt/apache/logs/ssl_mutex.18332 (System error follows)
57    [...] System: Permission denied (errno: 13)</code>'' are usually
58    caused by overly restrictive permissions on the <em>parent</em> directories.
59    Make sure that all parent directories (here <code>/opt</code>,
60    <code>/opt/apache</code> and <code>/opt/apache/logs</code>) have the x-bit
61    set for, at minimum, the UID under which Apache's children are running (see
62    the <code class="directive"><a href="../mod/mod_unixd.html#user">User</a></code> directive).</p>
63

65<h3><a name="entropy" id="entropy">Why does mod_ssl stop with the error
66        "Failed to generate temporary 512 bit RSA private key" when I start
67        Apache?</a></h3>
68    <p>Cryptographic software needs a source of unpredictable data
69    to work correctly. Many open source operating systems provide
70    a "randomness device" that serves this purpose (usually named
71    <code>/dev/random</code>). On other systems, applications have to
72    seed the OpenSSL Pseudo Random Number Generator (PRNG) manually with
73    appropriate data before generating keys or performing public key
74    encryption. As of version 0.9.5, the OpenSSL functions that need
75    randomness report an error if the PRNG has not been seeded with
76    at least 128 bits of randomness.</p>
77    <p>To prevent this error, <code class="module"><a href="../mod/mod_ssl.html">mod_ssl</a></code> has to provide
78    enough entropy to the PRNG to allow it to work correctly. This can
79    be done via the <code class="directive"><a href="../mod/mod_ssl.html#sslrandomseed">SSLRandomSeed</a></code>
80    directive.</p>

101<h3><a name="parallel" id="parallel">Is it possible to provide HTTP and HTTPS
102        from the same server?</a></h3>
103    <p>Yes. HTTP and HTTPS use different server ports (HTTP binds to
104    port 80, HTTPS to port 443), so there is no direct conflict between
105    them. You can either run two separate server instances bound to
106    these ports, or use Apache's elegant virtual hosting facility to
107    create two virtual servers, both served by the same instance of Apache
108    - one responding over HTTP to requests on port 80, and the other
109    responding over HTTPS to requests on port 443.</p>
110
111
112<h3><a name="ports" id="ports">Which port does HTTPS use?</a></h3>
113<p>You can run HTTPS on any port, but the standards specify port 443, which
114    is where any HTTPS compliant browser will look by default. You can force
115    your browser to look on a different port by specifying it in the URL. For
116    example, if your server is set up to serve pages over HTTPS on port 8080,
117    you can access them at <code>https://example.com:8080/</code></p>

120<h3><a name="httpstest" id="httpstest">How do I speak HTTPS manually for testing purposes?</a></h3>
121 <p>While you usually just use</p>
122
123    <div class="example"><p><code>$ telnet localhost 80<br />
124    GET / HTTP/1.0</code></p></div>
125
126    <p>for simple testing of Apache via HTTP, it's not so easy for
127    HTTPS because of the SSL protocol between TCP and HTTP. With the
128    help of OpenSSL's <code>s_client</code> command, however, you can
129    do a similar check via HTTPS:</p>
130
131    <div class="example"><p><code>$ openssl s_client -connect localhost:443 -state -debug<br />
132    GET / HTTP/1.0</code></p></div>
133
134    <p>Before the actual HTTP response you will receive detailed
135    information about the SSL handshake. For a more general command
136    line client which directly understands both HTTP and HTTPS, can
137    perform GET and POST operations, can use a proxy, supports byte
138    ranges, etc. you should have a look at the nifty
139    <a href="http://curl.haxx.se/">cURL</a> tool. Using this, you can
140    check that Apache is responding correctly to requests via HTTP and
141    HTTPS as follows:</p>
142
143    <div class="example"><p><code>$ curl http://localhost/<br />
144    $ curl https://localhost/</code></p></div>

147<h3><a name="hang" id="hang">Why does the connection hang when I connect
148    to my SSL-aware Apache server?</a></h3>
149
150<p>This can happen when you try to connect to a HTTPS server (or virtual
151    server) via HTTP (eg, using <code>http://example.com/</code> instead of
152    <code>https://example.com</code>). It can also happen when trying to
153    connect via HTTPS to a HTTP server (eg, using
154    <code>https://example.com/</code> on a server which doesn't support HTTPS,
155    or which supports it on a non-standard port). Make sure that you're
156    connecting to a (virtual) server that supports SSL.</p>
157
158<h3><a name="refused" id="refused">Why do I get ``Connection Refused'' messages,
159    when trying to access my newly installed Apache+mod_ssl server via HTTPS?</a></h3>
160<p>
161    This error can be caused by an incorrect configuration.
162    Please make sure that your <code class="directive"><a href="../mod/mpm_common.html#listen">Listen</a></code> directives match your
163    <code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code>
164    directives. If all else fails, please start afresh, using the default
165    configuration provided by <code class="module"><a href="../mod/mod_ssl.html">mod_ssl</a></code>.</p>

174<h3><a name="relative" id="relative">How can I switch between HTTP and HTTPS in relative
175    hyperlinks?</a></h3>
176
177<p>Usually, to switch between HTTP and HTTPS, you have to use
178    fully-qualified hyperlinks (because you have to change the URL
179    scheme).  Using <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code> however, you can
180    manipulate relative hyperlinks, to achieve the same effect.</p>
181    <pre class="prettyprint lang-config">RewriteEngine on
182RewriteRule   "^/(.*)_SSL$"   "https://%{SERVER_NAME}/$1" [R,L]
183RewriteRule   "^/(.*)_NOSSL$" "http://%{SERVER_NAME}/$1"  [R,L]</pre>
184
185
186    <p>This rewrite ruleset lets you use hyperlinks of the form
187    <code>&lt;a href="document.html_SSL"&gt;</code>, to switch to HTTPS
188    in a relative link. (Replace SSL with NOSSL to switch to HTTP.)</p>

215<h3><a name="keyscerts" id="keyscerts">What are RSA Private Keys, CSRs and Certificates?</a></h3>
216<p>An RSA private key file is a digital file that you can use to decrypt
217    messages sent to you. It has a public component which you distribute (via
218    your Certificate file) which allows people to encrypt those messages to
219    you.</p>
220    <p>A Certificate Signing Request (CSR) is a digital file which contains
221    your public key and your name. You send the CSR to a Certifying Authority
222    (CA), who will convert it into a real Certificate, by signing it.</p>
223    <p>A Certificate contains your
224    RSA public key, your name, the name of the CA, and is digitally signed by
225    the CA. Browsers that know the CA can verify the signature on that
226    Certificate, thereby obtaining your RSA public key. That enables them to
227    send messages which only you can decrypt.</p>
228    <p>See the <a href="ssl_intro.html">Introduction</a> chapter for a general
229    description of the SSL protocol.</p>
230
231
232<h3><a name="startup" id="startup">Is there a difference on startup between
233    a non-SSL-aware Apache and an SSL-aware Apache?</a></h3>
234<p>Yes. In general, starting Apache with
235    <code class="module"><a href="../mod/mod_ssl.html">mod_ssl</a></code> built-in is just like starting Apache
236    without it. However, if you have a passphrase on your SSL private
237    key file, a startup dialog will pop up which asks you to enter the
238    pass phrase.</p>
239
240    <p>Having to manually enter the passphrase when starting the server
241    can be problematic - for example, when starting the server from the
242    system boot scripts. In this case, you can follow the steps
243    <a href="#removepassphrase">below</a> to remove the passphrase from
244    your private key. Bear in mind that doing so brings additional security
245    risks - proceed with caution!</p>

248<h3><a name="selfcert" id="selfcert">How do I create a self-signed SSL
249Certificate for testing purposes?</a></h3>
250    <ol>
251    <li>Make sure OpenSSL is installed and in your <code>PATH</code>.<br />
252    <br />
253    </li>
254    <li>Run the following command, to create <code>server.key</code> and
255        <code>server.crt</code> files:<br />
256        <code><strong>$ openssl req -new -x509 -nodes -out server.crt
257                        -keyout server.key</strong></code><br />
258        These can be used as follows in your <code>httpd.conf</code>
259        file:
260        <pre class="prettyprint lang-config">SSLCertificateFile    "/path/to/this/server.crt"
261SSLCertificateKeyFile "/path/to/this/server.key"</pre>
262
263    </li>
264    <li>It is important that you are aware that this
265        <code>server.key</code> does <em>not</em> have any passphrase.
266        To add a passphrase to the key, you should run the following
267        command, and enter &amp; verify the passphrase as requested.<br />
268        <p><code><strong>$ openssl rsa -des3 -in server.key -out
269        server.key.new</strong></code><br />
270        <code><strong>$ mv server.key.new server.key</strong></code><br /></p>
271        Please backup the <code>server.key</code> file, and the passphrase
272        you entered, in a secure location.
273    </li>
274    </ol>

277<h3><a name="realcert" id="realcert">How do I create a real SSL Certificate?</a></h3>
278<p>Here is a step-by-step description:</p>
279    <ol>
280    <li>Make sure OpenSSL is installed and in your <code>PATH</code>.
281    <br />
282    <br />
283    </li>
284    <li>Create a RSA private key for your Apache server
285       (will be Triple-DES encrypted and PEM formatted):<br />
286       <br />
287       <code><strong>$ openssl genrsa -des3 -out server.key 2048</strong></code><br />
288       <br />
289       Please backup this <code>server.key</code> file and the
290       pass-phrase you entered in a secure location.
291       You can see the details of this RSA private key by using the command:<br />
292
293       <br />
294       <code><strong>$ openssl rsa -noout -text -in server.key</strong></code><br />
295       <br />
296       If necessary, you can also create a decrypted PEM version (not
297       recommended) of this RSA private key with:<br />
298       <br />
299       <code><strong>$ openssl rsa -in server.key -out server.key.unsecure</strong></code><br />
300       <br />
301
302    </li>
303    <li>Create a Certificate Signing Request (CSR) with the server RSA private
304       key (output will be PEM formatted):<br />
305       <br />
306       <code><strong>$ openssl req -new -key server.key -out server.csr</strong></code><br />
307       <br />
308       Make sure you enter the FQDN ("Fully Qualified Domain Name") of the
309       server when OpenSSL prompts you for the "CommonName", i.e. when you
310       generate a CSR for a website which will be later accessed via
311       <code>https://www.foo.dom/</code>, enter "www.foo.dom" here.
312       You can see the details of this CSR by using<br />
313
314       <br />
315       <code><strong>$ openssl req -noout -text -in server.csr</strong></code><br />
316       <br />
317    </li>
318    <li>You now have to send this Certificate Signing Request (CSR) to
319       a Certifying Authority (CA) to be signed. Once the CSR has been
320       signed, you will have a real Certificate, which can be used by
321       Apache. You can have a CSR signed by a commercial CA, or you can
322       create your own CA to sign it.<br />
323       Commercial CAs usually ask you to post the CSR into a web form,
324       pay for the signing, and then send a signed Certificate, which
325       you can store in a server.crt file.<br />
326
327       For details on how to create your own CA, and use this to sign
328       a CSR, see <a href="#ownca">below</a>.<br />
329
330       Once your CSR has been signed, you can see the details of the
331       Certificate as follows:<br />
332       <br />
333       <code><strong>$ openssl x509 -noout -text -in server.crt</strong></code><br />
334
335    </li>
336    <li>You should now have two files: <code>server.key</code> and
337    <code>server.crt</code>. These can be used as follows in your
338    <code>httpd.conf</code> file:
339       <pre class="prettyprint lang-config">SSLCertificateFile    "/path/to/this/server.crt"
340SSLCertificateKeyFile "/path/to/this/server.key"</pre>
341
342       The <code>server.csr</code> file is no longer needed.
343    </li>
344
345    </ol>

348<h3><a name="ownca" id="ownca">How do I create and use my own Certificate Authority (CA)?</a></h3>
349    <p>The short answer is to use the <code>CA.sh</code> or <code>CA.pl</code>
350    script provided by OpenSSL. Unless you have a good reason not to,
351    you should use these for preference. If you cannot, you can create a
352    self-signed certificate as follows:</p>
353
354    <ol>
355    <li>Create a RSA private key for your server
356       (will be Triple-DES encrypted and PEM formatted):<br />
357       <br />
358       <code><strong>$ openssl genrsa -des3 -out server.key 2048</strong></code><br />
359       <br />
360       Please backup this <code>server.key</code> file and the
361       pass-phrase you entered in a secure location.
362       You can see the details of this RSA private key by using the
363       command:<br />
364       <br />
365       <code><strong>$ openssl rsa -noout -text -in server.key</strong></code><br />
366       <br />
367       If necessary, you can also create a decrypted PEM version (not
368       recommended) of this RSA private key with:<br />
369       <br />
370       <code><strong>$ openssl rsa -in server.key -out server.key.unsecure</strong></code><br />
371       <br />
372    </li>
373    <li>Create a self-signed certificate (X509 structure)
374       with the RSA key you just created (output will be PEM formatted):<br />
375       <br />
376       <code><strong>$ openssl req -new -x509 -nodes -sha1 -days 365
377                       -key server.key -out server.crt -extensions usr_cert</strong></code><br />
378       <br />
379       This signs the server CSR and results in a <code>server.crt</code> file.<br />
380       You can see the details of this Certificate using:<br />
381       <br />
382       <code><strong>$ openssl x509 -noout -text -in server.crt</strong></code><br />
383       <br />
384    </li>
385    </ol>

388<h3><a name="passphrase" id="passphrase">How can I change the pass-phrase on my private key file?</a></h3>
389<p>You simply have to read it with the old pass-phrase and write it again,
390    specifying the new pass-phrase. You can accomplish this with the following
391    commands:</p>
392
393
394    <p><code><strong>$ openssl rsa -des3 -in server.key -out server.key.new</strong></code><br />
395    <code><strong>$ mv server.key.new server.key</strong></code><br /></p>
396
397    <p>The first time you're asked for a PEM pass-phrase, you should
398    enter the old pass-phrase. After that, you'll be asked again to
399    enter a pass-phrase - this time, use the new pass-phrase. If you
400    are asked to verify the pass-phrase, you'll need to enter the new
401    pass-phrase a second time.</p>

404<h3><a name="removepassphrase" id="removepassphrase">How can I get rid of the pass-phrase dialog at Apache startup time?</a></h3>
405<p>The reason this dialog pops up at startup and every re-start
406    is that the RSA private key inside your server.key file is stored in
407    encrypted format for security reasons. The pass-phrase is needed to decrypt
408    this file, so it can be read and parsed. Removing the pass-phrase
409    removes a layer of security from your server - proceed with caution!</p>
410    <ol>
411    <li>Remove the encryption from the RSA private key (while
412       keeping a backup copy of the original file):<br />
413       <br />
414       <code><strong>$ cp server.key server.key.org</strong></code><br />
415       <code><strong>$ openssl rsa -in server.key.org -out server.key</strong></code><br />
416
417       <br />
418    </li>
419    <li>Make sure the server.key file is only readable by root:<br />
420       <br />
421       <code><strong>$ chmod 400 server.key</strong></code><br />
422       <br />
423    </li>
424    </ol>
425
426    <p>Now <code>server.key</code> contains an unencrypted copy of the key.
427    If you point your server at this file, it will not prompt you for a
428    pass-phrase. HOWEVER, if anyone gets this key they will be able to
429    impersonate you on the net. PLEASE make sure that the permissions on this
430    file are such that only root or the web server user can read it
431    (preferably get your web server to start as root but run as another
432    user, and have the key readable only by root).</p>
433
434    <p>As an alternative approach you can use the ``<code>SSLPassPhraseDialog
435    exec:/path/to/program</code>'' facility. Bear in mind that this is
436    neither more nor less secure, of course.</p>

439<h3><a name="verify" id="verify">How do I verify that a private key matches its Certificate?</a></h3>
440<p>A private key contains a series of numbers. Two of these numbers form
441    the "public key", the others are part of the "private key". The "public
442    key" bits are included when you generate a CSR, and subsequently form
443    part of the associated Certificate.</p>
444    <p>To check that the public key in your Certificate matches the public
445    portion of your private key, you simply need to compare these numbers.
446    To view the Certificate and the key run the commands:</p>
447
448    <p><code><strong>$ openssl x509 -noout -text -in server.crt</strong></code><br />
449    <code><strong>$ openssl rsa -noout -text -in server.key</strong></code></p>
450
451    <p>The `modulus' and the `public exponent' portions in the key and the
452    Certificate must match. As the public exponent is usually 65537
453    and it's difficult to visually check that the long modulus numbers
454    are the same, you can use the following approach:</p>
455
456    <p><code><strong>$ openssl x509 -noout -modulus -in server.crt | openssl md5</strong></code><br />
457    <code><strong>$ openssl rsa -noout -modulus -in server.key | openssl md5</strong></code></p>
458
459    <p>This leaves you with two rather shorter numbers to compare. It is,
460    in theory, possible that these numbers may be the same, without the
461    modulus numbers being the same, but the chances of this are
462    overwhelmingly remote.</p>
463    <p>Should you wish to check to which key or certificate a particular
464    CSR belongs you can perform the same calculation on the CSR as
465    follows:</p>
466
467    <p><code><strong>$ openssl req -noout -modulus -in server.csr | openssl md5</strong></code></p>
468
469
470<h3><a name="pemder" id="pemder">How can I convert a certificate from PEM to DER format?</a></h3>
471<p>The default certificate format for OpenSSL is PEM, which is simply
472    Base64 encoded DER, with header and footer lines. For some applications
473    (e.g. Microsoft Internet Explorer) you need the certificate in plain DER
474    format. You can convert a PEM file <code>cert.pem</code> into the
475    corresponding DER file <code>cert.der</code> using the following command:
476    <code><strong>$ openssl x509 -in cert.pem -out cert.der -outform DER</strong></code></p>
477
478
479<h3><a name="gid" id="gid">Why do browsers complain that they cannot verify my server certificate?</a></h3>
480
481    <p>One reason this might happen is because your server certificate is signed
482    by an intermediate CA. Various CAs, such as Verisign or Thawte, have started
483    signing certificates not with their root certificate but with intermediate
484    certificates.</p>
485
486    <p>Intermediate CA certificates lie between the root CA certificate (which is
487    installed in the browsers) and the server certificate (which you installed
488    on the server). In order for the browser to be able to traverse and verify
489    the trust chain from the server certificate to the root certificate it
490    needs need to be given the intermediate certificates. The CAs should
491    be able to provide you such intermediate certificate packages that can be
492    installed on the server.</p>
493
494    <p>You need to include those intermediate certificates with the
495    <code class="directive"><a href="../mod/mod_ssl.html#sslcertificatechainfile">SSLCertificateChainFile</a></code>
496    directive.</p>

528<h3><a name="random" id="random">Why do I get lots of random SSL protocol
529errors under heavy server load?</a></h3>
530<p>There can be a number of reasons for this, but the main one
531    is problems with the SSL session Cache specified by the
532    <code class="directive"><a href="../mod/mod_ssl.html#sslsessioncache">SSLSessionCache</a></code> directive. The DBM session
533    cache is the most likely source of the problem, so using the SHM session cache (or
534    no cache at all) may help.</p>
535
536
537<h3><a name="load" id="load">Why does my webserver have a higher load, now
538that it serves SSL encrypted traffic?</a></h3>
539<p>SSL uses strong cryptographic encryption, which necessitates a lot of
540    number crunching. When you request a webpage via HTTPS, everything (even
541    the images) is encrypted before it is transferred. So increased HTTPS
542    traffic leads to load increases.</p>
543
544
545<h3><a name="establishing" id="establishing">Why do HTTPS connections to my server
546sometimes take up to 30 seconds to establish a connection?</a></h3>
547<p>This is usually caused by a <code>/dev/random</code> device for
548    <code class="directive"><a href="../mod/mod_ssl.html#sslrandomseed">SSLRandomSeed</a></code> which blocks the
549    read(2) call until enough entropy is available to service the
550    request. More information is available in the reference
551    manual for the <code class="directive"><a href="../mod/mod_ssl.html#sslrandomseed">SSLRandomSeed</a></code>
552    directive.</p>
553
554
555<h3><a name="ciphers" id="ciphers">What SSL Ciphers are supported by mod_ssl?</a></h3>
556<p>Usually, any SSL ciphers supported by the version of OpenSSL in use,
557    are also supported by <code class="module"><a href="../mod/mod_ssl.html">mod_ssl</a></code>. Which ciphers are
558    available can depend on the way you built OpenSSL. Typically, at
559    least the following ciphers are supported:</p>
560
561    <ol>
562    <li>RC4 with SHA1</li>
563    <li>AES with SHA1</li>
564    <li>Triple-DES with SHA1</li>
565    </ol>
566
567    <p>To determine the actual list of ciphers available, you should run
568    the following:</p>
569    <div class="example"><p><code>$ openssl ciphers -v</code></p></div>
570

572<h3><a name="adh" id="adh">Why do I get ``no shared cipher'' errors, when
573trying to use Anonymous Diffie-Hellman (ADH) ciphers?</a></h3>
574<p>By default, OpenSSL does <em>not</em> allow ADH ciphers, for security
575    reasons. Please be sure you are aware of the potential side-effects
576    if you choose to enable these ciphers.</p>
577    <p>In order to use Anonymous Diffie-Hellman (ADH) ciphers, you must
578    build OpenSSL with ``<code>-DSSL_ALLOW_ADH</code>'', and then add
579    ``<code>ADH</code>'' into your <code class="directive"><a href="../mod/mod_ssl.html#sslciphersuite">SSLCipherSuite</a></code>.</p>
580
581
582<h3><a name="sharedciphers" id="sharedciphers">Why do I get a 'no shared ciphers'
583error when connecting to my newly installed server?</a></h3>
584<p>Either you have made a mistake with your
585    <code class="directive"><a href="../mod/mod_ssl.html#sslciphersuite">SSLCipherSuite</a></code>
586    directive (compare it with the pre-configured example in
587    <code>extra/httpd-ssl.conf</code>) or you chose to use DSA/DH
588    algorithms instead of RSA when you generated your private key
589    and ignored or overlooked the warnings. If you have chosen
590    DSA/DH, then your server cannot communicate using RSA-based SSL
591    ciphers (at least until you configure an additional RSA-based
592    certificate/key pair). Modern browsers like NS or IE can only
593    communicate over SSL using RSA ciphers. The result is the
594    "no shared ciphers" error. To fix this, regenerate your server
595    certificate/key pair, using the RSA algorithm.</p>
596

598<h3><a name="vhosts" id="vhosts">Why can't I use SSL with name-based/non-IP-based virtual hosts?</a></h3>
599<p>The reason is very technical, and a somewhat "chicken and egg" problem.
600    The SSL protocol layer stays below the HTTP protocol layer and
601    encapsulates HTTP. When an SSL connection (HTTPS) is established
602    Apache/mod_ssl has to negotiate the SSL protocol parameters with the
603    client. For this, mod_ssl has to consult the configuration of the virtual
604    server (for instance it has to look for the cipher suite, the server
605    certificate, etc.). But in order to go to the correct virtual server
606    Apache has to know the <code>Host</code> HTTP header field. To do this, the
607    HTTP request header has to be read. This cannot be done before the SSL
608    handshake is finished, but the information is needed in order to
609    complete the SSL handshake phase. See the next question for how to
610    circumvent this issue.</p>
611    
612    <p>Note that if you have a wildcard SSL certificate, or a
613    certificate that has multiple hostnames on it using subjectAltName
614    fields, you can use SSL on name-based virtual hosts without further
615    workarounds.</p>
616
617
618<h3><a name="vhosts2" id="vhosts2">Is it possible to use Name-Based
619Virtual Hosting to identify different SSL virtual hosts?</a></h3>
620    <p>Name-Based Virtual Hosting is a very popular method of identifying
621    different virtual hosts. It allows you to use the same IP address and
622    the same port number for many different sites. When people move on to
623    SSL, it seems natural to assume that the same method can be used to have
624    lots of different SSL virtual hosts on the same server.</p>
625
626    <p>It is possible, but only if using a 2.2.12 or later web server,
627    built with 0.9.8j or later OpenSSL.  This is because it requires a
628    feature that only the most recent revisions of the SSL
629    specification added, called Server Name Indication (SNI).</p>
630
631    <p>Note that if you have a wildcard SSL certificate, or a
632    certificate that has multiple hostnames on it using subjectAltName
633    fields, you can use SSL on name-based virtual hosts without further
634    workarounds.</p>
635
636    <p>The reason is that the SSL protocol is a separate layer which
637    encapsulates the HTTP protocol. So the SSL session is a separate
638    transaction, that takes place before the HTTP session has begun.
639    The server receives an SSL request on IP address X and port Y
640    (usually 443). Since the SSL request did not contain any Host:
641    field, the server had no way to decide which SSL virtual host to use.
642    Usually, it just used the first one it found which matched the
643    port and IP address specified.</p>
644
645    <p>If you are using a version of the web server and OpenSSL that
646    support SNI, though, and the client's browser also supports SNI,
647    then the hostname is included in the original SSL request, and the
648    web server can select the correct SSL virtual host.</p>
649
650    <p>You can, of course, use Name-Based Virtual Hosting to identify many
651    non-SSL virtual hosts (all on port 80, for example) and then
652    have a single SSL virtual host (on port 443). But if you do this,
653    you must make sure to put the non-SSL port number on the NameVirtualHost
654    directive, e.g.</p>
655
656    <pre class="prettyprint lang-config">NameVirtualHost 192.168.1.1:80</pre>
657
658
659    <p>Other workaround solutions include: </p>
660
661    <p>Using separate IP addresses for different SSL hosts.
662    Using different port numbers for different SSL hosts.</p>

665<h3><a name="comp" id="comp">How do I get SSL compression working?</a></h3>
666<p>Although SSL compression negotiation was defined in the specification
667of SSLv2 and TLS, it took until May 2004 for RFC 3749 to define DEFLATE as
668a negotiable standard compression method.
669</p>
670<p>OpenSSL 0.9.8 started to support this by default when compiled with the
671<code>zlib</code> option. If both the client and the server support compression,
672it will be used. However, most clients still try to initially connect with an
673SSLv2 Hello. As SSLv2 did not include an array of preferred compression algorithms
674in its handshake, compression cannot be negotiated with these clients.
675If the client disables support for SSLv2, either an SSLv3 or TLS Hello
676may be sent, depending on which SSL library is used, and compression may
677be set up. You can verify whether clients make use of SSL compression by
678logging the <code>%{SSL_COMPRESS_METHOD}x</code> variable.
679</p>

682<h3><a name="lockicon" id="lockicon">When I use Basic Authentication over HTTPS
683the lock icon in Netscape browsers stays unlocked when the dialog pops up.
684Does this mean the username/password is being sent unencrypted?</a></h3>
685<p>No, the username/password is transmitted encrypted. The icon in
686    Netscape browsers is not actually synchronized with the SSL/TLS layer.
687    It only toggles to the locked state when the first part of the actual
688    webpage data is transferred, which may confuse people. The Basic
689    Authentication facility is part of the HTTP layer, which is above
690    the SSL/TLS layer in HTTPS. Before any HTTP data communication takes
691    place in HTTPS, the SSL/TLS layer has already completed its handshake
692    phase, and switched to encrypted communication. So don't be
693    confused by this icon.</p>

696<h3><a name="msie" id="msie">Why do I get I/O errors when connecting via
697HTTPS to an Apache+mod_ssl server with older versions of Microsoft Internet
698Explorer (MSIE)?</a></h3>
699<p>The first reason is that the SSL implementation in some MSIE versions has
700    some subtle bugs related to the HTTP keep-alive facility and the SSL close
701    notify alerts on socket connection close. Additionally the interaction
702    between SSL and HTTP/1.1 features are problematic in some MSIE versions.
703    You can work around these problems by forcing Apache not to use HTTP/1.1,
704    keep-alive connections or send the SSL close notify messages to MSIE clients.
705    This can be done by using the following directive in your SSL-aware
706    virtual host section:</p>
707    <pre class="prettyprint lang-config">SetEnvIf User-Agent "MSIE [2-5]" \
708         nokeepalive ssl-unclean-shutdown \
709         downgrade-1.0 force-response-1.0</pre>
710
711    <p>Further, some MSIE versions have problems with particular ciphers.
712    Unfortunately, it is not possible to implement a MSIE-specific
713    workaround for this, because the ciphers are needed as early as the
714    SSL handshake phase. So a MSIE-specific
715    <code class="directive"><a href="../mod/mod_setenvif.html#setenvif">SetEnvIf</a></code> won't solve these
716    problems. Instead, you will have to make more drastic
717    adjustments to the global parameters. Before you decide to do
718    this, make sure your clients really have problems. If not, do not
719    make these changes - they will affect <em>all</em> your clients, MSIE
720    or otherwise.</p>

723<h3><a name="srp" id="srp">How do I enable TLS-SRP?</a></h3>
724    <p>TLS-SRP (Secure Remote Password key exchange for TLS, specified in RFC 5054)
725    can supplement or replace certificates in authenticating an SSL connection.
726    To use TLS-SRP, set the
727    <code class="directive"><a href="../mod/mod_ssl.html#sslsrpverifierfile">SSLSRPVerifierFile</a></code> directive to
728    point to an OpenSSL SRP verifier file. To create the verifier file, use the
729    <code>openssl</code> tool:</p>
730    <div class="example"><p><code>
731    openssl srp -srpvfile passwd.srpv -add username
732    </code></p></div>
733    <p>After creating this file, specify it in the SSL server configuration:</p>
734    <div class="example"><p><code>
735    SSLSRPVerifierFile /path/to/passwd.srpv
736    </code></p></div>
737    <p>To force clients to use non-certificate TLS-SRP cipher suites, use the
738    following directive:</p>
739    <div class="example"><p><code>
740    SSLCipherSuite "!DSS:!aRSA:SRP"
741    </code></p></div>

744<h3><a name="javadh" id="javadh">Why do I get handshake failures with Java-based clients when using a certificate with more than 1024 bits?</a></h3>
745    <p>Beginning with version 2.4.7,
746    <code class="module"><a href="../mod/mod_ssl.html">mod_ssl</a></code> will use DH parameters which include primes
747    with lengths of more than 1024 bits. Java 7 and earlier limit their
748    support for DH prime sizes to a maximum of 1024 bits, however.</p>
749
750    <p>If your Java-based client aborts with exceptions such as
751    <code>java.lang.RuntimeException: Could not generate DH keypair</code> and
752    <code>java.security.InvalidAlgorithmParameterException: Prime size must be
753    multiple of 64, and can only range from 512 to 1024 (inclusive)</code>,
754    and httpd logs <code>tlsv1 alert internal error (SSL alert number 80)</code>
755    (at <code class="directive"><a href="../mod/core.html#loglevel">LogLevel</a></code> <code>info</code>
756    or higher), you can either rearrange mod_ssl's cipher list with
757    <code class="directive"><a href="../mod/mod_ssl.html#sslciphersuite">SSLCipherSuite</a></code>
758    (possibly in conjunction with <code class="directive"><a href="../mod/mod_ssl.html#sslhonorcipherorder">SSLHonorCipherOrder</a></code>),
759    or you can use custom DH parameters with a 1024-bit prime, which
760    will always have precedence over any of the built-in DH parameters.</p>
761
762    <p>To generate custom DH parameters, use the <code>openssl dhparam 1024</code>
763    command. Alternatively, you can use the following standard 1024-bit DH
764    parameters from <a href="http://www.ietf.org/rfc/rfc2409.txt">RFC 2409</a>,
765    section 6.2:</p>
766    <div class="example"><pre>-----BEGIN DH PARAMETERS-----
767MIGHAoGBAP//////////yQ/aoiFowjTExmKLgNwc0SkCTgiKZ8x0Agu+pjsTmyJR
768Sgh5jjQE3e+VGbPNOkMbMCsKbfJfFDdP4TVtbVHCReSFtXZiXn7G9ExC6aY37WsL
769/1y29Aa37e44a/taiZ+lrp8kEXxLH+ZJKGZR7OZTgf//////////AgEC
770-----END DH PARAMETERS-----</pre></div>
771    <p>Add the custom parameters including the "BEGIN DH PARAMETERS" and
772    "END DH PARAMETERS" lines to the end of the first certificate file
773    you have configured using the
774    <code class="directive"><a href="../mod/mod_ssl.html#sslcertificatefile">SSLCertificateFile</a></code> directive.</p>

871<p>In general no, at least not unless you provide more details about the code
872    location where Apache dumped core. What is usually always required in
873    order to help you is a backtrace (see next question). Without this
874    information it is mostly impossible to find the problem and help you in
875    fixing it.</p>

878<h3><a name="backtrace" id="backtrace">How do I get a backtrace, to help find
879the reason for my core dump?</a></h3>
880<p>Following are the steps you will need to complete, to get a backtrace:</p>
881    <ol>
882    <li>Make sure you have debugging symbols available, at least
883        in Apache. On platforms where you use GCC/GDB, you will have to build
884        Apache+mod_ssl with ``<code>OPTIM="-g -ggdb3"</code>'' to get this. On
885        other platforms at least ``<code>OPTIM="-g"</code>'' is needed.
886    </li>
887
888    <li>Start the server and try to reproduce the core-dump. For this you may
889        want to use a directive like ``<code>CoreDumpDirectory /tmp</code>'' to
890        make sure that the core-dump file can be written. This should result
891        in a <code>/tmp/core</code> or <code>/tmp/httpd.core</code> file. If you
892        don't get one of these, try running your server under a non-root UID.
893        Many modern kernels do not allow a process to dump core after it has
894        done a <code>setuid()</code> (unless it does an <code>exec()</code>) for
895        security reasons (there can be privileged information left over in
896        memory). If necessary, you can run <code>/path/to/httpd -X</code>
897        manually to force Apache to not fork.
898    </li>
899
900    <li>Analyze the core-dump. For this, run <code>gdb /path/to/httpd
901        /tmp/httpd.core</code> or a similar command. In GDB, all you
902        have to do then is to enter <code>bt</code>, and voila, you get the
903        backtrace. For other debuggers consult your local debugger manual.
904    </li>
905    </ol>
