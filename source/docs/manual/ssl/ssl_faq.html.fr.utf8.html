53<h3><a name="mutex" id="mutex">Pourquoi le démarrage d'Apache provoque-t-il des
54erreurs de permission en rapport avec SSLMutex ?</a></h3>
55    <p>Des erreurs telles que ``<code>mod_ssl: Child could not open
56    SSLMutex lockfile /opt/apache/logs/ssl_mutex.18332 (avec l'erreur
57    système qui suit) [...] System: Permission denied (errno: 13)</code>''
58    sont souvent provoquées par des permissions trop restrictives sur les
59    répertoires <em>parents</em>. Assurez-vous que tous les répertoires
60    parents (ici <code>/opt</code>, <code>/opt/apache</code> et
61    <code>/opt/apache/logs</code>) ont le bit x positionné au moins pour
62    l'UID sous lequel les processus enfants d'Apache s'exécutent (voir la
63    directive <code class="directive"><a href="../mod/mod_unixd.html#user">User</a></code>).</p>
64
65
66<h3><a name="entropy" id="entropy">Pourquoi mod_ssl s'arrête-t-il avec l'erreur
67"Failed to generate temporary 512 bit RSA private key" au démarrage
68d'Apache ?</a></h3>
69    <p>Pour fonctionner correctement, les logiciels de cryptographie ont
70    besoin d'une source de données aléatoires. De nombreux systèmes
71    d'exploitation libres proposent un "périphérique source d'entropie"
72    qui fournit ce service (il se nomme en général
73    <code>/dev/random</code>). Sur d'autres systèmes, les applications
74    doivent amorcer manuellement
75    le Générateur de Nombres Pseudo-Aléatoires d'OpenSSL
76    (Pseudo Random Number Generator -PRNG) à l'aide de données appropriées
77    avant de générer des clés ou d'effectuer un chiffrement à clé
78    publique. Depuis la version 0.9.5, les fonctions d'OpenSSL qui nécessitent
79    des données aléatoires provoquent une erreur si le PRNG n'a pas été amorcé
80    avec une source de données aléatoires d'au moins 128 bits.</p>
81    <p>Pour éviter cette erreur, <code class="module"><a href="../mod/mod_ssl.html">mod_ssl</a></code> doit fournir
82    suffisamment d'entropie au PRNG pour lui permettre de fonctionner
83    correctement. Ce niveau d'entropie est défini par la directive
84    <code class="directive"><a href="../mod/mod_ssl.html#sslrandomseed">SSLRandomSeed</a></code>.</p>

106<h3><a name="parallel" id="parallel">Peut-on faire cohabiter HTTP et HTTPS sur le même
107serveur ?</a></h3>
108    <p>Oui. HTTP et HTTPS utilisent des ports différents (HTTP écoute le port
109    80 et HTTPS le port 443), si bien qu'il n'y a pas de conflit direct entre
110    les deux. Vous pouvez soit exécuter deux instances séparées du serveur,
111    chacune d'entre elles écoutant l'un de ces ports, soit utiliser l'élégante
112    fonctionnalité d'Apache que constituent les hôtes virtuels pour créer
113    deux serveurs virtuels gérés par la même instance d'Apache - le
114    premier serveur répondant en HTTP aux requêtes sur le port 80,
115    le second répondant en HTTPS aux requêtes sur le port
116    443.</p>
117
118
119<h3><a name="ports" id="ports">Quel port HTTPS utilise-t-il ?</a></h3>
120<p>Vous pouvez associer le protocole HTTPS à n'importe quel port, mais le port
121standard est le port 443, que tout navigateur compatible HTTPS va utiliser par
122défaut. Vous pouvez forcer votre navigateur à utiliser un port différent en le
123précisant dans l'URL. Par exemple, si votre serveur est configuré pour
124servir des pages en HTTPS sur le port 8080, vous pourrez y accéder par
125l'adresse <code>https://example.com:8080/</code>.</p>
126
127
128<h3><a name="httpstest" id="httpstest">Comment s'exprimer en langage HTTPS à des fins
129de test ?</a></h3>
130 <p>Alors que vous utilisez simplement</p>
131
132    <div class="example"><p><code>$ telnet localhost 80<br />
133    GET / HTTP/1.0</code></p></div>
134
135    <p>pour tester facilement Apache via HTTP, les choses ne sont pas si
136    simples pour HTTPS à cause du protocole SSL situé entre TCP et HTTP.
137    La commande OpenSSL <code>s_client</code> vous permet cependant
138    d'effectuer un test similaire via HTTPS :</p>
139
140    <div class="example"><p><code>$ openssl s_client -connect localhost:443 -state -debug<br />
141    GET / HTTP/1.0</code></p></div>
142
143    <p>Avant la véritable réponse HTTP, vous recevrez des informations
144    détaillées à propos de l'établissement de la connexion SSL. Si vous
145    recherchez un client en ligne de commande à usage plus général qui comprend
146    directement HTTP et HTTPS, qui peut effectuer des opérations GET et POST,
147    peut utiliser un mandataire, supporte les requêtes portant sur une partie
148    d'un fichier (byte-range), etc..., vous devriez vous tourner vers
149    l'excellent outil <a href="http://curl.haxx.se/">cURL</a>. Grâce à lui,
150    vous pouvez vérifier si Apache répond correctement aux requêtes via
151    HTTP et HTTPS comme suit :</p>
152
153    <div class="example"><p><code>$ curl http://localhost/<br />
154    $ curl https://localhost/</code></p></div>

157<h3><a name="hang" id="hang">Pourquoi la communication se bloque-t-elle lorsque je
158me connecte à mon serveur Apache configuré pour SSL ?</a></h3>
159<p>Ceci peut arriver si vous vous connectez à un serveur HTTPS (ou à
160un serveur virtuel) via HTTP (par exemple, en utilisant
161<code>http://example.com/</code> au lieu de <code>https://example.com</code>).
162Cela peut aussi arriver en essayant de vous connecter via HTTPS à un
163serveur HTTP (par exemple, en utilisant <code>https://example.com/</code>
164avec un serveur qui ne supporte pas HTTPS, ou le supporte, mais sur un
165port non standard). Assurez-vous que vous vous connectez bien à un
166serveur (virtuel) qui supporte SSL.</p>

169<h3><a name="refused" id="refused">Pourquoi, lorsque je tente d'accéder en HTTPS à mon
170serveur Apache+mod_ssl fraîchement installé, l'erreur ``Connection Refused''
171s'affiche-t-elle ?</a></h3>
172<p>Une configuration incorrecte peut provoquer ce type d'erreur.
173Assurez-vous que vos directives <code class="directive"><a href="../mod/mpm_common.html#listen">Listen</a></code> s'accordent avec vos directives
174    <code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code>. Si
175    l'erreur persiste, recommencez depuis le début en restaurant la
176    configuration par défaut fournie par<code class="module"><a href="../mod/mod_ssl.html">mod_ssl</a></code>.</p>

179<h3><a name="envvars" id="envvars">Pourquoi les variables <code>SSL_XXX</code>
180ne sont-elles pas disponibles dans mes scripts CGI et SSI ?</a></h3>
181<p>Assurez-vous que la directive ``<code>SSLOptions +StdEnvVars</code>'' est
182bien présente dans le contexte de vos requêtes CGI/SSI.</p>

185<h3><a name="relative" id="relative">Comment puis-je basculer entre les protocoles HTTP et
186HTTPS dans les hyperliens relatifs ?</a></h3>
187
188<p>Normalement, pour basculer entre HTTP et HTTPS, vous devez utiliser des
189hyperliens pleinement qualifiés (car vous devez modifier le schéma de l'URL).
190Cependant, à l'aide du module <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code>, vous pouvez
191manipuler des hyperliens relatifs, pour obtenir le même effet.</p>
192    <pre class="prettyprint lang-config">RewriteEngine on
193RewriteRule   "^/(.*)_SSL$"   "https://%{SERVER_NAME}/$1" [R,L]
194RewriteRule   "^/(.*)_NOSSL$" "http://%{SERVER_NAME}/$1"  [R,L]</pre>
195
196
197    <p>Ce jeu de règles rewrite vous permet d'utiliser des hyperliens de la
198    forme <code>&lt;a href="document.html_SSL"&gt;</code> pour passer en HTTPS
199    dans les liens relatifs. (Remplacez SSL par NOSSL pour passer en HTTP.)</p>

226<h3><a name="keyscerts" id="keyscerts">Qu'est-ce qu'un clé privée RSA, un certificat,
227une demande de signature de certificat (CSR) ?</a></h3>
228<p>Un fichier de clé privée RSA est un fichier numérique que vous pouvez
229utiliser pour déchiffrer des messages que l'on vous a envoyés. Il a son
230pendant à caractère public que vous pouvez distribuer (par le biais de votre
231certificat), ce qui permet aux utilisateurs de chiffrer les messages qu'ils
232vous envoient.</p>
233    <p>Une Demande de Signature de Certificat (CSR) est un fichier numérique
234    qui contient votre clé publique et votre nom. La CSR doit être envoyée à
235    une Autorité de Certification (CA), qui va la convertir en vrai certificat
236    en la signant.</p>
237    <p>Un certificat contient votre clé publique RSA, votre nom, le nom
238    de la CA, et est signé numériquement par cette dernière. Les navigateurs
239    qui reconnaissent la CA peuvent vérifier la signature du certificat, et
240    ainsi en extraire votre clé publique RSA. Ceci leur permet de vous envoyer
241    des messages chiffrés que vous seul pourrez déchiffrer.</p>
242    <p>Se référer au chapitre <a href="ssl_intro.html">Introduction</a>
243    pour une description générale du protocole SSL.</p>
244

246<h3><a name="startup" id="startup">Y a-t-il une différence au démarrage entre un serveur
247Apache non SSL et un serveur Apache supportant SSL ?</a></h3>
248<p>Oui. En général, avec ou sans <code class="module"><a href="../mod/mod_ssl.html">mod_ssl</a></code> intégré, le démarrage
249d'Apache ne présente pas de différences. Cependant, si votre fichier de clé
250privée SSL possède un mot de passe, vous devrez le taper au démarrage
251d'Apache.</p>
252
253    <p>Devoir entrer manuellement le mot de passe au démarrage du serveur peut
254    poser quelques problèmes - par exemple, quand le serveur est démarré au
255    moyen de scripts au lancement du système. Dans ce cas, vous pouvez suivre
256    les étapes <a href="#removepassphrase">ci-dessous</a> pour supprimer le
257    mot de passe de votre clé privée. Gardez à l'esprit qu'agir ainsi augmente
258    les risques de sécurité - agissez avec précaution !</p>
259

261<h3><a name="selfcert" id="selfcert">Comment créer un certificat auto-signé SSL à des
262fins de test ?</a></h3>
263    <ol>
264    <li>Vérifiez qu'OpenSSL est installé et l'exécutable openssl dans votre
265    <code>PATH</code>.<br />
266    <br />
267    </li>
268    <li>Exécuter la commande suivante pour créer les fichiers
269    <code>server.key</code> et <code>server.crt</code> :<br />
270	<code><strong>$ openssl req -new -x509 -nodes -out server.crt
271			-keyout server.key</strong></code><br />
272	Ces fichiers seront utilisés comme suit dans votre
273	<code>httpd.conf</code> :
274        <pre class="prettyprint lang-config">SSLCertificateFile    "/path/to/this/server.crt"
275SSLCertificateKeyFile "/path/to/this/server.key"</pre>
276
277    </li>
278    <li>Il est important de savoir que le fichier <code>server.key</code> n'a
279    <em>pas</em> de mot de passe. Pour ajouter un mot de passe à la clé, vous
280    devez exécuter la commande suivante et confirmer le mot de passe comme
281    demandé.<br />
282	<p><code><strong>$ openssl rsa -des3 -in server.key -out
283	server.key.new</strong></code><br />
284	<code><strong>$ mv server.key.new server.key</strong></code><br /></p>
285	Sauvegardez le fichier <code>server.key</code> ainsi que son mot de
286	passe en lieu sûr.
287    </li>
288    </ol>

291<h3><a name="realcert" id="realcert">Comment créer un vrai certificat SSL ?</a></h3>
292<p>Voici la marche à suivre pas à pas :</p>
293    <ol>
294    <li>Assurez-vous qu'OpenSSL est bien installé et dans votre <code>PATH</code>.
295    <br />
296    <br />
297    </li>
298    <li>Créez une clé privée RSA pour votre serveur Apache
299    	(elle sera au format PEM et chiffrée en Triple-DES):<br />
300       <br />
301       <code><strong>$ openssl genrsa -des3 -out server.key 2048</strong></code><br />
302       <br />
303       Enregistrez le fichier <code>server.key</code> et le mot de passe
304       éventuellement défini en lieu sûr.
305       Vous pouvez afficher les détails de cette clé privée RSA à l'aide de la
306       commande :<br />
307
308       <br />
309       <code><strong>$ openssl rsa -noout -text -in server.key</strong></code><br />
310       <br />
311       Si nécessaire, vous pouvez aussi créer une version PEM non chiffrée
312       (non recommandé) de clé privée RSA avec :<br />
313       <br />
314       <code><strong>$ openssl rsa -in server.key -out server.key.unsecure</strong></code><br />
315       <br />
316
317    </li>
318    <li>Créez une Demande de signature de Certificat (CSR) à l'aide de la
319    clé privée précédemment générée (la sortie sera au format PEM):<br />
320       <br />
321       <code><strong>$ openssl req -new -key server.key -out server.csr</strong></code><br />
322       <br />
323       Vous devez entrer le Nom de Domaine Pleinement Qualifié
324       ("Fully Qualified Domain Name" ou FQDN) de votre serveur lorsqu'OpenSSL
325       vous demande le "CommonName", c'est à dire que si vous générez une CSR
326       pour un site web auquel on accèdera par l'URL
327       <code>https://www.foo.dom/</code>, le FQDN sera "www.foo.dom". Vous
328       pouvez afficher les détails de ce CSR avec :<br />
329
330       <br />
331       <code><strong>$ openssl req -noout -text -in server.csr</strong></code><br />
332       <br />
333    </li>
334    <li>Vous devez maintenant envoyer la CSR à une Autorité de Certification
335    (CA), afin que cette dernière puisse la signer. Une fois la CSR signée,
336    vous disposerez d'un véritable certificat que vous pourrez utiliser avec
337    Apache. Vous pouvez faire signer votre CSR par une CA commerciale ou par
338    votre propre CA.<br />
339       Les CAs commerciales vous demandent en général de leur envoyer la CSR
340       par l'intermédiaire d'un formulaire web, de régler le montant de la
341       signature, puis vous envoient un certificat signé que vous pouvez
342       enregistrer dans un fichier server.crt.
343
344       Pour plus de détails sur la manière de créer sa propre CA, et de
345       l'utiliser pour signer une CSR, voir <a href="#ownca">ci-dessous</a>.<br />
346
347       Une fois la CSR signée, vous pouvez afficher les détails du certificat
348       comme suit :<br />
349       <br />
350       <code><strong>$ openssl x509 -noout -text -in server.crt</strong></code><br />
351
352    </li>
353    <li>Vous devez maintenant disposer de deux fichiers :
354    <code>server.key</code> et <code>server.crt</code>. Ils sont précisés dans
355    votre fichier <code>httpd.conf</code> comme suit :
356       <pre class="prettyprint lang-config">SSLCertificateFile    "/path/to/this/server.crt"
357SSLCertificateKeyFile "/path/to/this/server.key"</pre>
358
359       Le fichier <code>server.csr</code> n'est plus nécessaire.
360    </li>
361
362    </ol>

365<h3><a name="ownca" id="ownca">Comment créer et utiliser sa propre Autorité de
366certification (CA) ?</a></h3>
367    <p>La solution la plus simple consiste à utiliser les scripts
368    <code>CA.sh</code> ou <code>CA.pl</code> fournis avec OpenSSL. De
369    préférence, utilisez cette solution, à moins que vous ayez de bonnes
370    raisons de ne pas le faire. Dans ce dernier cas, vous pouvez créer un
371    certificat auto-signé comme suit :</p>
372
373    <ol>
374    <li>Créez une clé privée RSA pour votre serveur
375    	(elle sera au format PEM et chiffrée en Triple-DES) :<br />
376       <br />
377       <code><strong>$ openssl genrsa -des3 -out server.key 2048</strong></code><br />
378       <br />
379       Sauvegardez le fichier <code>server.key</code> et le mot de passe
380       éventuellement défini en lieu sûr.
381       Vous pouvez afficher les détails de cette clé privée RSA à l'aide de la
382       commande :<br />
383       <br />
384       <code><strong>$ openssl rsa -noout -text -in server.key</strong></code><br />
385       <br />
386       Si nécessaire, vous pouvez aussi créer une version PEM non chiffrée
387       (non recommandé) de cette clé privée RSA	 avec :<br />
388       <br />
389       <code><strong>$ openssl rsa -in server.key -out server.key.unsecure</strong></code><br />
390       <br />
391    </li>
392    <li>Créez un certificat auto-signé (structure X509) à l'aide de la clé RSA
393    que vous venez de générer (la sortie sera au format PEM) :<br />
394       <br />
395       <code><strong>$ openssl req -new -x509 -nodes -sha1 -days 365
396		       -key server.key -out server.crt -extensions usr_cert</strong></code><br />
397       <br />
398       Cette commande signe le certificat du serveur et produit un fichier
399       <code>server.crt</code>. Vous pouvez afficher les détails de ce
400       certificat avec :<br />
401       <br />
402       <code><strong>$ openssl x509 -noout -text -in server.crt</strong></code><br />
403       <br />
404    </li>
405    </ol>

408<h3><a name="passphrase" id="passphrase">Comment modifier le mot de passe
409de ma clé privée ?</a></h3>
410<p>Vous devez simplement lire la clé avec l'ancien mot de passe et la
411réécrire en spécifiant le nouveau mot de passe. Pour cela, vous pouvez
412utiliser les commandes suivantes :</p>
413
414
415    <p><code><strong>$ openssl rsa -des3 -in server.key -out server.key.new</strong></code><br />
416    <code><strong>$ mv server.key.new server.key</strong></code><br /></p>
417
418    <p>La première fois qu'il vous est demandé un mot de passe PEM, vous
419    devez entrer l'ancien mot de passe. Ensuite, on vous demandera d'entrer
420    encore un mot de passe - cette fois, entrez le nouveau mot de passe. Si on
421    vous demande de vérifier le mot de passe, vous devrez entrer le nouveau
422    mot de passe une seconde fois.</p>

425<h3><a name="removepassphrase" id="removepassphrase">Comment démarrer Apache sans avoir à entrer de
426mot de passe ?</a></h3>
427<p>L'apparition de ce dialogue au démarrage et à chaque redémarrage provient
428du fait que la clé privée RSA contenue dans votre fichier server.key est
429enregistrée sous forme chiffrée pour des raisons de sécurité. Le
430déchiffrement de ce fichier nécessite un mot de passe, afin de pouvoir être
431lu et interprété. Cependant, La suppression du mot de passe diminue le niveau de
432sécurité du serveur - agissez avec précautions !</p>
433    <ol>
434    <li>Supprimer le chiffrement de la clé privée RSA (tout en conservant une
435    copie de sauvegarde du fichier original) :<br />
436       <br />
437       <code><strong>$ cp server.key server.key.org</strong></code><br />
438       <code><strong>$ openssl rsa -in server.key.org -out server.key</strong></code><br />
439
440       <br />
441    </li>
442    <li>Assurez-vous que le fichier server.key n'est lisible que par root :<br />
443       <br />
444       <code><strong>$ chmod 400 server.key</strong></code><br />
445       <br />
446    </li>
447    </ol>
448
449    <p>Maintenant, <code>server.key</code> contient une copie non chiffrée de
450    la clé. Si vous utilisez ce fichier pour votre serveur, il ne vous
451    demandera plus de mot de passe. CEPENDANT, si quelqu'un arrive à obtenir
452    cette clé, il sera en mesure d'usurper votre identité sur le réseau.
453    Vous DEVEZ par conséquent vous assurer que seuls root ou le serveur web
454    peuvent lire ce fichier (de préférence, démarrez le serveur web sous
455    root et faites le s'exécuter sous un autre utilisateur, en n'autorisant
456    la lecture de la clé que par root).</p>
457
458    <p>Une autre alternative consiste à utiliser la directive
459    ``<code>SSLPassPhraseDialog exec:/chemin/vers/programme</code>''. Gardez
460    cependant à l'esprit que ce n'est bien entendu ni plus ni moins
461    sécurisé.</p>

464<h3><a name="verify" id="verify">Comment vérifier si une clé privée correspond bien
465à son certificat ?</a></h3>
466<p>Une clé privée contient une série de nombres. Deux de ces nombres forment la
467"clé publique", les autres appartiennent à la "clé privée". Les bits de la
468"clé publique" sont inclus quand vous générez une CSR, et font par
469conséquent partie du certificat associé.</p>
470    <p>Pour vérifier que la clé publique contenue dans votre certificat
471    correspond bien à la partie publique de votre clé privée, il vous suffit
472    de comparer ces nombres. Pour afficher le certificat et la clé,
473    utilisez cette commande :</p>
474
475    <p><code><strong>$ openssl x509 -noout -text -in server.crt</strong></code><br />
476    <code><strong>$ openssl rsa -noout -text -in server.key</strong></code></p>
477
478    <p>Les parties `modulus' et `public exponent' doivent être identiques dans
479    la clé et le certificat. Comme le `public exponent' est habituellement
480    65537, et comme il est difficile de vérifier visuellement que les nombreux
481    nombres du `modulus' sont identiques, vous pouvez utiliser l'approche
482    suivante :</p>
483
484    <p><code><strong>$ openssl x509 -noout -modulus -in server.crt | openssl md5</strong></code><br />
485    <code><strong>$ openssl rsa -noout -modulus -in server.key | openssl md5</strong></code></p>
486
487    <p>Il ne vous reste ainsi que deux nombres relativement courts à comparer.
488    Il est possible, en théorie que ces deux nombres soient les mêmes, sans que
489    les nombres du modulus soient identiques, mais les chances en sont infimes.</p>
490    <p>Si vous souhaitez vérifier à quelle clé ou certificat appartient une CSR
491    particulière, vous pouvez effectuer le même calcul
492    sur la CSR comme suit :</p>
493
494    <p><code><strong>$ openssl req -noout -modulus -in server.csr | openssl md5</strong></code></p>

497<h3><a name="pemder" id="pemder">Comment convertir un certificat du format PEM
498au format DER ?</a></h3>
499<p>Le format des certificats par défaut pour OpenSSL est le format PEM,
500qui est tout simplement un format DER codé en Base64, avec des lignes
501d'en-têtes et des annotations. Certaines applications, comme
502Microsoft Internet Explorer, ont besoin d'un certificat au format DER de base.
503Vous pouvez convertir un fichier PEM <code>cert.pem</code> en son équivalent
504au format DER <code>cert.der</code> à l'aide de la commande suivante :
505<code><strong>$ openssl x509 -in cert.pem -out cert.der
506-outform DER</strong></code></p>

509<h3><a name="gid" id="gid">Pourquoi les navigateurs se plaignent-ils de ne pas pouvoir
510vérifier mon certificat de serveur ?</a></h3>
511
512    <p>Ceci peut se produire si votre certificat de serveur est signé
513    par une autorité de certification intermédiaire. Plusieurs CAs,
514    comme Verisign ou Thawte, ont commencé à signer les certificats avec
515    des certificats intermédiaires au lieu de leur certificat racine.</p>
516
517    <p>Les certificats de CA intermédiaires se situe à un niveau
518    intermédiaire entre le certificat racine de la CA (qui est installé dans les
519    navigateurs) et le certificat du serveur (que vous avez installé sur
520    votre serveur). Pour que le navigateur puisse traverser et vérifier
521    la chaîne de confiance depuis le certificat du serveur jusqu'au
522    certificat racine, il faut lui fournir les certificats
523    intermédiaires. Les CAs devraient pouvoir fournir de tels
524    paquetages de certificats intermédiaires à installer sur les
525    serveurs.</p>
526
527    <p>Vous devez inclure ces certificats intermédiaires via la
528    directive <code class="directive"><a href="../mod/mod_ssl.html#sslcertificatechainfile">SSLCertificateChainFile</a></code>.</p>

567<h3><a name="random" id="random">Pourquoi de nombreuses et aléatoires erreurs de
568protocole SSL apparaissent-elles en cas de forte charge du serveur ?</a></h3>
569<p>Ce problème peut avoir plusieurs causes, mais la principale réside dans le
570cache de session SSL défini par la directive
571<code class="directive"><a href="../mod/mod_ssl.html#sslsessioncache">SSLSessionCache</a></code>. Le cache de session
572DBM est souvent à la source du problème qui peut être résolu en utilisant le
573cache de session SHM (ou en n'utilisant tout simplement pas de cache).</p>
574
575
576<h3><a name="load" id="load">Pourquoi la charge de mon serveur est-elle plus
577importante depuis qu'il sert des ressources chiffrées en SSL ?</a></h3>
578<p>SSL utilise un procédé de chiffrement fort qui nécessite la manipulation
579d'une quantité très importante de nombres. Lorsque vous effectuez une requête
580pour une page web via HTTPS, tout (même les images) est chiffré avant d'être
581transmis. C'est pourquoi un accroissement du traffic HTTPS entraîne une
582augmentation de la charge.</p>

585<h3><a name="establishing" id="establishing">Pourquoi les connexions en HTTPS à mon serveur
586prennent-elles parfois jusqu'à 30 secondes pour s'établir ?</a></h3>
587<p>Ce problème provient en général d'un périphérique <code>/dev/random</code>
588qui bloque l'appel système read(2) jusqu'à ce que suffisamment d'entropie
589soit disponible pour servir la requête. Pour plus d'information, se référer au
590manuel de référence de la directive
591<code class="directive"><a href="../mod/mod_ssl.html#sslrandomseed">SSLRandomSeed</a></code>.</p>

594<h3><a name="ciphers" id="ciphers">Quels sont les algorithmes de chiffrement
595supportés par mod_ssl ?</a></h3>
596<p>En général, tous les algorithmes de chiffrement supportés par la version
597d'OpenSSL installée, le sont aussi par <code class="module"><a href="../mod/mod_ssl.html">mod_ssl</a></code>. La liste des
598algorithmes disponibles peut dépendre de la manière dont vous avez installé
599OpenSSL. Typiquement, au moins les algorithmes suivants sont supportés :</p>
600
601    <ol>
602    <li>RC4 avec SHA1</li>
603    <li>AES avec SHA1</li>
604    <li>Triple-DES avec SHA1</li>
605    </ol>
606
607    <p>Pour déterminer la liste réelle des algorithmes disponibles, vous
608    pouvez utiliser la commande suivante :</p>
609    <div class="example"><p><code>$ openssl ciphers -v</code></p></div>

612<h3><a name="adh" id="adh">Pourquoi une erreur ``no shared cipher'' apparaît-elle
613quand j'essaie d'utiliser un algorithme de chiffrement
614Diffie-Hellman anonyme (ADH) ?</a></h3>
615<p>Par défaut et pour des raisons de sécurité, OpenSSl ne permet <em>pas</em>
616l'utilisation des algorithmes de chiffrements ADH. Veuillez vous informer
617sur les effets pervers potentiels si vous choisissez d'activer le support
618de ces algorithmes de chiffrements.</p>
619<p>Pour pouvoir utiliser les algorithmes de chiffrements Diffie-Hellman
620anonymes (ADH), vous devez compiler OpenSSL avec
621``<code>-DSSL_ALLOW_ADH</code>'', puis ajouter ``<code>ADH</code>'' à votre
622directive <code class="directive"><a href="../mod/mod_ssl.html#sslciphersuite">SSLCipherSuite</a></code>.</p>

625<h3><a name="sharedciphers" id="sharedciphers">Pourquoi une erreur ``no shared cipher''
626apparaît-elle lorsqu'on se connecte à mon serveur
627fraîchement installé ?</a></h3>
628<p>Soit vous avez fait une erreur en définissant votre directive
629<code class="directive"><a href="../mod/mod_ssl.html#sslciphersuite">SSLCipherSuite</a></code> (comparez-la avec
630l'exemple préconfiguré dans <code>extra/httpd-ssl.conf</code>), soit vous avez
631choisi d'utiliser des algorithmes DSA/DH au lieu de RSA lorsque vous avez
632généré votre clé privée, et avez ignoré ou êtes passé outre les
633avertissements. Si vous avez choisi DSA/DH, votre serveur est incapable de
634communiquer en utilisant des algorithmes de chiffrements SSL basés sur RSA
635(du moins tant que vous n'aurez pas configuré une paire clé/certificat RSA
636additionnelle). Les navigateurs modernes tels que NS ou IE ne peuvent
637communiquer par SSL qu'avec des algorithmes RSA. C'est ce qui provoque l'erreur
638"no shared ciphers". Pour la corriger, générez une nouvelle paire
639clé/certificat pour le serveur en utilisant un algorithme de chiffrement
640RSA.</p>

643<h3><a name="vhosts" id="vhosts">Pourquoi ne peut-on pas utiliser SSL avec des hôtes
644virtuels identifiés par un nom et non par une adresse IP ?</a></h3>
645<p>La raison est très technique, et s'apparente au problème de la primauté de
646l'oeuf ou de la poule. La couche du protocole SSL se trouve en dessous de la
647couche de protocole HTTP qu'elle encapsule. Lors de l'établissement d'une
648connexion SSL (HTTPS), Apache/mod_ssl doit négocier les paramètres du
649protocole SSL avec le client. Pour cela, mod_ssl doit consulter la
650configuration du serveur virtuel (par exemple, il doit accéder à la suite
651d'algorithmes de chiffrement, au certificat du serveur, etc...). Mais afin de
652sélectionner le bon serveur virtuel, Apache doit connaître le contenu du champ
653d'en-tête HTTP <code>Host</code>. Pour cela, il doit lire l'en-tête de la
654requête HTTP. Mais il ne peut le faire tant que la négociation SSL n'est pas
655terminée, or, la phase de négociation SSL a besoin du nom d'hôte contenu
656dans l'en-tête de la requête. Voir la question suivante pour
657contourner ce problème.</p>
658
659    <p>Notez que si votre certificat comporte un nom de serveur avec
660    caractères génériques, ou des noms de serveurs multiples dans le
661    champ subjectAltName, vous pouvez utiliser SSL avec les serveurs
662    virtuels à base de noms sans avoir à contourner ce problème.</p>
663

665<h3><a name="vhosts2" id="vhosts2">Est-il possible d'utiliser
666l'hébergement virtuel basé sur le nom d'hôte
667pour différencier plusieurs hôtes virtuels ?</a></h3>
668    <p>L'hébergement virtuel basé sur le nom est une méthode très populaire
669    d'identification des différents hôtes virtuels. Il permet d'utiliser la
670    même adresse IP et le même numéro de port pour de nombreux sites
671    différents. Lorsqu'on se tourne vers SSL, il semble tout naturel de penser
672    que l'on peut appliquer la même méthode pour gérer plusieurs hôtes
673    virtuels SSL sur le même serveur.</p>
674
675    <p>C'est possible, mais seulement si on utilise une version 2.2.12
676    ou supérieure du serveur web compilée avec OpenSSL
677    version 0.9.8j ou supérieure. Ceci est du au fait que
678    l'utilisation de l'hébergement virtuel à base de nom
679    avec SSL nécessite une fonctionnalité appelée
680    Indication du Nom de Serveur (Server Name Indication - SNI) que
681    seules les révisions les plus récentes de la
682    spécification SSL supportent.</p>
683
684    <p>Notez que si votre certificat comporte un nom de serveur avec
685    caractères génériques, ou des noms de serveurs multiples dans le
686    champ subjectAltName, vous pouvez utiliser SSL avec les serveurs
687    virtuels à base de noms sans avoir à contourner ce problème.</p>
688
689    <p>La raison en est que le protocole SSL constitue une couche séparée qui
690    encapsule le protocole HTTP. Aini, la session SSL nécessite une
691    transaction séparée qui prend place avant que la session HTTP n'ait débuté.
692    Le serveur reçoit une requête SSL sur l'adresse IP X et le port Y
693    (habituellement 443). Comme la requête SSL ne contenait aucun
694    en-tête Host:, le serveur n'avait aucun moyen de déterminer quel hôte virtuel SSL il
695    devait utiliser. En général, il utilisait le premier
696    qu'il trouvait et qui
697    correspondait à l'adresse IP et au port spécifiés.</p>
698
699    <p>Par contre, si vous utilisez des versions du serveur web et
700    d'OpenSSL qui supportent SNI, et si le navigateur du client le
701    supporte aussi, alors le nom d'hôte sera inclus dans la
702    requête SSL originale, et le serveur web pourra
703    sélectionner le bon serveur virtuel SSL.</p>
704
705    <p>Bien entendu, vous pouvez utiliser l'hébergement virtuel basé sur le nom
706    pour identifier de nombreux hôtes virtuels non-SSL
707    (tous sur le port 80 par exemple), et ne gérer qu'un seul hôte virtuel SSL
708    (sur le port 443). Mais dans ce cas, vous devez définir le numéro de port
709    non-SSL à l'aide de la directive NameVirtualHost dans ce style :</p>
710
711    <pre class="prettyprint lang-config">NameVirtualHost 192.168.1.1:80</pre>
712
713
714    <p>il existe d'autres solutions alternatives comme :</p>
715
716    <p>Utiliser des adresses IP différentes pour chaque hôte SSL.
717    Utiliser des numéros de port différents pour chaque hôte SSL.</p>

720<h3><a name="comp" id="comp">Comment mettre en oeuvre la compression SSL ?</a></h3>
721<p>Bien que la négociation pour la compression SSL ait été définie dans la
722spécification de SSLv2 et TLS, ce n'est qu'en mai 2004 que la RFC 3749 a
723défini DEFLATE comme une méthode de compression standard négociable.
724</p>
725<p>Depuis la version 0.9.8, OpenSSL supporte cette compression par défaut
726lorsqu'il est compilé avec l'option <code>zlib</code>. Si le client et le
727serveur supportent la compression, elle sera utilisée. Cependant, la
728plupart des clients essaient encore de se connecter avec un Hello SSLv2.
729Comme SSLv2 ne comportait pas de table des algorithmes de compression préférés
730dans sa négociation, la compression ne peut pas être négociée avec ces clients.
731Si le client désactive le support SSLv2, un Hello SSLv3 ou TLS peut être
732envoyé, selon la bibliothèque SSL utilisée, et la compression peut être mise
733en oeuvre. Vous pouvez vérifier si un client utilise la compression SSL en
734journalisant la variable <code>%{SSL_COMPRESS_METHOD}x</code>.
735</p>

738<h3><a name="lockicon" id="lockicon">Lorsque j'utilise l'authentification de base sur HTTPS,
739l'icône de verrouillage des navigateurs Netscape reste ouverte quand la boîte
740de dialogue d'authentification apparaît. Cela signifie-t-il que les utilisateur
741et mot de passe sont envoyés en clair ?</a></h3>
742<p>Non, le couple utilisateur/mot de passe est transmis sous forme chiffrée.
743	L'icône de chiffrement dans les navigateurs Netscape n'est pas vraiment
744	synchronisé avec la couche SSL/TLS. Il ne passe à l'état verrouillé
745	qu'au moment où la première partie des données relatives à la page web
746	proprement dite sont transférées, ce qui peut prêter à confusion. Le
747	dispositif d'authentification de base appartient à la couche HTTP, qui
748	est située au dessus de la couche SSL/TLS dans HTTPS. Avant tout
749	transfert de données HTTP sous HTTPS, la couche SSL/TLS a déjà achevé
750	sa phase de négociation et basculé dans le mode de communication
751	chiffrée. Ne vous laissez donc pas abuser par l'état de cet icône.</p>

754<h3><a name="msie" id="msie">Pourquoi des erreurs d'entrée/sortie apparaissent-elles
755lorsqu'on se connecte via HTTPS à un serveur Apache+mod_ssl avec des
756versions anciennes de
757Microsoft Internet Explorer (MSIE) ?</a></h3>
758<p>La première raison en est la présence dans l'implémentation SSL de
759certaines versions de MSIE de bogues subtils en rapport avec le
760dispositif de "maintien en vie" (keep-alive) HTTP, et les alertes de
761notification de fermeture de session SSL en cas de coupure de la
762connexion au point d'entrée (socket). De plus, l'interaction entre
763SSL et les fonctionnalités HTTP/1.1 pose problème avec certaines
764versions de MSIE. Vous pouvez contourner ces problèmes en interdisant
765à Apache l'utilisation de HTTP/1.1, les connexions avec maintien en vie
766ou l'envoi de messages de notification de fermeture de session SSL aux
767clients MSIE. Pour cela, vous pouvez utiliser la directive suivante
768dans votre section d'hôte virtuel avec support SSL :</p>
769    <pre class="prettyprint lang-config">SetEnvIf User-Agent "MSIE [2-5]" \
770         nokeepalive ssl-unclean-shutdown \
771         downgrade-1.0 force-response-1.0</pre>
772
773    <p>En outre, certaines versions de MSIE ont des problèmes avec des
774    algorithmes de chiffrement particuliers. Hélas, il n'est pas
775    possible d'apporter une solution spécifique à MSIE pour ces
776    problèmes, car les algorithmes de chiffrement sont utilisés dès la
777    phase de négociation SSL. Ainsi, une directive
778    <code class="directive"><a href="../mod/mod_setenvif.html#setenvif">SetEnvIf</a></code> spécifique
779    à MSIE ne peut être d'aucun secours. Par contre, vous devrez
780    ajuster les paramètres généraux de manière drastique. Avant de
781    vous décider, soyez sûr que vos clients rencontrent vraiment des
782    problèmes. Dans la négative, n'effectuez pas ces ajustements car
783    ils affecteront <em>tous</em> vos clients, ceux utilisant MSIE,
784    mais aussi les autres.</p>

788<h3><a name="srp" id="srp">Comment activer TLS-SRP ?</a></h3>
789    <p>Le protocole TLS-SRP (Echange de clés sécurisé par mot de passe
790    pour TLS comme spécifié dans la RFC 5054) peut compléter ou même
791    remplacer les certificats lors du processus d'authentification des
792    connexions SSL. Pour utiliser TLS-SRP, spécifiez un fichier de
793    vérification SRP OpenSSL via la directive <code class="directive"><a href="../mod/mod_ssl.html#sslsrpverifierfile">SSLSRPVerifierFile</a></code>. Vous pouvez créer
794    le fichier de vérification via l'utilitaire <code>openssl</code> :</p>
795    <div class="example"><p><code>
796    openssl srp -srpvfile passwd.srpv -add username
797    </code></p></div>
798    <p>Une fois ce fichier créé, vous devez le référencer dans la
799    configuration du serveur SSL :</p>
800    <div class="example"><p><code>
801    SSLSRPVerifierFile /path/to/passwd.srpv
802    </code></p></div>
803    <p>Pour forcer les clients à utiliser des algorithmes de chiffrement
804    basés sur TLS-SRP et s'affranchissant des certificats, utilisez la
805    directive suivante :</p>
806    <div class="example"><p><code>
807    SSLCipherSuite "!DSS:!aRSA:SRP"
808    </code></p></div>

811<h3><a name="javadh" id="javadh">Pourquoi des erreurs de négociation apparaissent
812avec les clients basés sur Java lorsqu'on utilise un certificat de plus
813de 1024 bits ?</a></h3>
814    <p>Depuis la version 2.4.7,
815    <code class="module"><a href="../mod/mod_ssl.html">mod_ssl</a></code> utilise des paramètres DH qui comportent
816    des nombres premiers de plus de 1024 bits. Cependant, java 7 et ses versions
817    antérieures ne supportent que les nombres premiers DH d'une longueur
818    maximale de 1024 bits.</p>
819
820    <p>Si votre client basé sur Java s'arrête avec une exception telle
821    que <code>java.lang.RuntimeException: Could not generate DH
822    keypair</code> et
823    <code>java.security.InvalidAlgorithmParameterException: Prime size
824    must be multiple of 64, and can only range from 512 to 1024
825    (inclusive)</code>, et si httpd enregistre le message <code>tlsv1
826    alert internal error (SSL alert number 80)</code> dans son journal
827    des erreurs (avec un <code class="directive"><a href="../mod/core.html#loglevel">LogLevel</a></code>
828    <code>info</code> ou supérieur), vous pouvez soit réarranger la
829    liste d'algorithmes de mod_ssl via la directive <code class="directive"><a href="../mod/mod_ssl.html#sslciphersuite">SSLCipherSuite</a></code> (éventuellement en
830    conjonction avec la directive <code class="directive"><a href="../mod/mod_ssl.html#sslhonorcipherorder">SSLHonorCipherOrder</a></code>), soit utiliser des
831    paramètres DH personnalisés avec un nombre
832    premier de 1024 bits, paramètres qui seront toujours prioritaires
833    par rapport à tout autre paramètre DH par défaut.</p>
834
835    <p>Pour générer des paramètres DH personnalisés, utilisez la
836    commande <code>openssl dhparam 1024</code>. Vous pouvez aussi
837    utiliser les
838    paramètres DH standards issus de la <a href="http://www.ietf.org/rfc/rfc2409.txt">RFC 2409</a>, section 6.2 :</p>
839    <div class="example"><pre>-----BEGIN DH PARAMETERS-----
840MIGHAoGBAP//////////yQ/aoiFowjTExmKLgNwc0SkCTgiKZ8x0Agu+pjsTmyJR
841Sgh5jjQE3e+VGbPNOkMbMCsKbfJfFDdP4TVtbVHCReSFtXZiXn7G9ExC6aY37WsL
842/1y29Aa37e44a/taiZ+lrp8kEXxLH+ZJKGZR7OZTgf//////////AgEC
843-----END DH PARAMETERS-----</pre></div>
844    <p>Ajoute les paramètres personnalisés incluant les lignes "BEGIN DH
845    PARAMETERS" et "END DH PARAMETERS" à la fin du premier fichier de
846    certificat défini via la directive <code class="directive"><a href="../mod/mod_ssl.html#sslcertificatefile">SSLCertificateFile</a></code>.</p>

915<h3><a name="reportdetails" id="reportdetails">Quelles informations dois-je fournir lors
916de l'écriture d'un rapport de bogue ?</a></h3>
917<p>Vous devez toujours fournir au moins les informations
918suivantes :</p>
919
920    <dl>
921    <dt>Les versions d'Apache httpd et OpenSSL installées</dt>
922    <dd>La version d'Apache peut être déterminée en exécutant
923    <code>httpd -v</code>. La version d'OpenSSL peut être déterminée
924    en exécutant <code>openssl version</code>. Si Lynx est installé,
925    vous pouvez aussi exécuter la commande<code>lynx -mime_header
926    http://localhost/ | grep Server</code> et ainsi obtenir ces
927    informations en une seule fois.
928    </dd>
929
930    <dt>Les détails de votre installation d'Apache httpd et OpenSSL</dt>
931    <dd>A cet effet, vous pouvez fournir un fichier journal de votre
932    session de terminal qui montre les étapes de la configuration et
933    de l'installation. En cas d'impossibilité, vous devez au moins
934    fournir la ligne de commande <code class="program"><a href="../programs/configure.html">configure</a></code> que
935    vous avez utilisée.
936    </dd>
937
938    <dt>En cas de vidage mémoire, inclure une trace de ce qui s'est
939    passé</dt>
940    <dd>Si votre serveur Apache httpd fait un vidage de sa
941    mémoire, merci de fournir en pièce jointe un fichier contenant
942    une trace de la zone dédiée à la pile (voir
943    <a href="#backtrace">ci-dessous</a> pour des informations sur la manière
944    de l'obtenir). Il est nécessaire de disposer de ces informations
945    afin de pouvoir déterminer la raison de votre vidage mémoire.
946    </dd>
947
948    <dt>Une description détaillée de votre problème</dt>
949
950    <dd>Ne riez pas, nous sommes sérieux ! De nombreux rapports
951    n'incluent pas de description de la véritable nature du problème.
952    Sans ces informations, il est très difficile pour quiconque de
953    vous aider. Donc, et c'est votre propre intérêt (vous souhaitez
954    que le problème soit résolu, n'est-ce pas ?), fournissez, s'il vous
955    plait, le maximum de détails possible. Bien entendu, vous devez
956    aussi inclure tout ce qui a été dit précédemment.
957    </dd>
958    </dl>

961<h3><a name="coredumphelp" id="coredumphelp">Un vidage mémoire s'est produit,
962pouvez-vous m'aider ?</a></h3>
963<p>En général non, du moins tant que vous n'aurez pas fourni plus de
964détails à propos de la localisation dans le code où Apache a effectué
965son vidage mémoire. Ce dont nous avons en général besoin pour vous
966aider est une trace de ce qui s'est passé (voir la question suivante).
967Sans cette information, il est pratiquement impossible de déterminer
968la nature du problème et de vous aider à le résoudre.</p>

971<h3><a name="backtrace" id="backtrace">Comment puis-je obtenir une journalisation de
972ce qui s'est passé, pour m'aider à trouver la raison de ce vidage
973mémoire ?</a></h3>
974<p>Vous trouverez ci-dessous les différentes étapes permettant
975d'obtenir une journalisation des évènements (backtrace) :</p>
976    <ol>
977    <li>Assurez-vous que les symboles de débogage sont disponibles, au
978    moins pour Apache. Pour cela, sur les plates-formes où GCC/GDB est
979    utilisé, vous devez compiler Apache+mod_ssl avec l'option
980    ``<code>OPTIM="-g -ggdb3"</code>''. Sur les autres plates-formes,
981    l'option ``<code>OPTIM="-g"</code>'' est un minimum.
982    </li>
983
984    <li>Démarrez le serveur et essayez de reproduire le vidage mémoire.
985    A cet effet, vous pouvez utiliser une directive du style
986    ``<code>CoreDumpDirectory /tmp</code>'' pour être sûr que le
987    fichier de vidage mémoire puisse bien être écrit. Vous devriez
988    obtenir un fichier <code>/tmp/core</code> ou
989    <code>/tmp/httpd.core</code>. Si ce n'est pas le cas, essayez de
990    lancer votre serveur sous un UID autre que root.
991    Pour des raisons de sécurité, de nombreux
992    noyaux modernes de permettent pas à un processus de vider sa
993    mémoire une fois qu'il a accompli un <code>setuid()</code> (à moins
994    qu'il effectue un <code>exec()</code>) car des informations d'un
995    niveau privilégié pourraient être transmises en mémoire. Si
996    nécessaire, vous pouvez exécuter <code>/chemin/vers/httpd -X</code>
997    manuellement afin de ne pas permettre à Apache de se clôner (fork).
998    </li>
999
1000    <li>Analysez le vidage mémoire. Pour cela, exécutez
1001    <code>gdb /path/to/httpd /tmp/httpd.core</code> ou une commande
1002    similaire. Dans GDB, tout ce que vous avez à faire est d'entrer
1003    <code>bt</code>, et voila, vous obtenez la backtrace. Pour les
1004    débogueurs autres que GDB consulter le manuel correspondant.
1005    </li>
1006    </ol>
