47<h2><a name="clientserver" id="clientserver">Clients, serveurs et URLs</a></h2>
48
49
50<p>
51Les adresses des pages web sur la Toile se présentent sous forme d'URLs
52- Uniform Resource Locators - qui comportent un protocole (par
53  exemple <code>http</code>), un nom de serveur (par exemple
54  <code>www.apache.org</code>), un chemin (par exemple
55  <code>/docs/current/getting-started.html</code>), et le cas échéant
56  une chaîne de paramètres (query string) (par exemple <code>?arg=value</code>)
57  permettant de transmettre des informations supplémentaires au serveur.
58</p>
59
60<p>Un client (par exemple un navigateur web) se connecte à un serveur
61(par exemple votre serveur HTTP Apache) avec un protocole spécifique, et
62effectue une <strong>requête</strong> pour une ressource en spécifiant
63son chemin.</p>
64
65<p>Un chemin peut représenter plusieurs types de ressources sur le
66serveur. Il peut s'agir d'un fichier (comme
67<code>getting-started.html</code>), d'un gestionnaire (comme <a href="mod/mod_status.html">server-status</a>) ou d'une sorte quelconque de
68programme (comme <code>index.php</code>). Nous décrirons tout cela plus
69en détails ci-dessous dans la section <a href="#content">Contenu d'un
70site web</a>.</p>
71
72<p>
73Le serveur envoie alors une <strong>réponse</strong> comportant un code
74d'état, et éventuellement un corps de réponse. Le code d'état indique si
75la requête a été traitée avec succès, ou dans la négative, quel type
76d'erreur a été rencontré. Le client est alors censé savoir quoi faire de
77la réponse. Vous pouvez vous familiariser avec les différents codes
78d'état en consultant le <a href="https://cwiki.apache.org/confluence/display/httpd/CommonHTTPStatusCodes">Wiki du
79serveur HTTP Apache</a>.</p>
80
81<p>Les détails de la transaction, ainsi que les erreurs rencontrées,
82sont enregistrés dans des fichiers journaux. Tout cela est décrit en
83détails ci-dessous dans la section <a href="#logs">Débogage et fichiers
84journaux</a>.</p>

88<h2><a name="dns" id="dns">Noms d'hôte et DNS</a></h2>
89
90
91<p>Pour se connecter à un serveur, le client doit tout d'abord traduire
92le nom du serveur en adresse IP, cette dernière permettant de localiser
93le serveur sur Internet. Ainsi, pour que votre serveur web soit
94accessible, son nom doit être enregistré dans le DNS.</p>
95
96<p>Si vous ne savez pas comment effectuer cet enregistrement, vous
97devez contacter votre administrateur réseau ou votre fournisseur
98d'accès à Internet afin qu'il effectue cette opération pour vous.</p>
99
100<p>Plusieurs noms d'hôte peuvent pointer vers la même adresse IP, et
101plusieurs adresses IP peuvent être attachées au même serveur physique.
102Vous pouvez ainsi héberger plusieurs serveurs web sur le même serveur
103physique grâce au mécanisme des <a href="vhosts/">serveurs virtuels</a>.</p>
104
105<p>Pour tester un serveur non encore accessible sur Internet, vous
106pouvez renseigner son nom d'hôte dans votre fichier hosts afin
107d'effectuer une résolution de nom locale. Par exemple, pour tester le
108serveur web <code>www.example.com</code> depuis le serveur physique qui
109l'héberge, vous pouvez ajouter la ligne suivante au fichier hosts de ce
110dernier : </p>
111
112<div class="example"><p><code>
113127.0.0.1 www.example.com
114</code></p></div>
115
116<p>En général, le fichier hosts se trouve dans le répertoire
117<code>/etc</code> sur les systèmes de style Unix, ou
118<code>C:\Windows\system32\drivers\etc</code> sous Windows.</p>
119
120<p>Vous trouverez plus de détails à propos du fichier hosts à <a href="http://en.wikipedia.org/wiki/Hosts_(file)">Wikipedia.org/wiki/Hosts_(file)</a>,
121et à propos du DNS à <a href="http://en.wikipedia.org/wiki/Domain_Name_System">Wikipedia.org/wiki/Domain_Name_System</a>.</p>

124<h2><a name="configuration" id="configuration">Fichiers de configuration et directives</a></h2>
125
126
127<p>La configuration du serveur HTTP Apache s'effectue via de simples
128fichiers texte. Ces fichiers peuvent se trouver dans de nombreux
129endroits différents en fonction du mode d'installation du serveur. Vous
130trouverez les positions courantes de ces fichiers dans le <a href="https://cwiki.apache.org/confluence/display/httpd/DistrosDefaultLayout">wiki httpd</a>.
131Si vous installez httpd depuis le code source, le répertoire par défaut
132des fichiers de configuration est <code>/usr/local/apache2/conf</code>.
133Le nom du fichier de configuration par défaut est en général
134<code>httpd.conf</code>, mais peut aussi varier en fonction des
135distributions tierces du serveur.</p>
136
137<p>L'ensemble de la configuration est en général divisé en plusieurs
138fichiers afin d'en faciliter la gestion. Ces fichiers sont inclus dans
139le fichier de configuration principal via la directive <code class="directive"><a href="./mod/core.html#include">Include</a></code>. Les noms ou positions de ces fichiers
140ne sont pas figés et peuvent varier considérablement d'une distribution
141à l'autre. N'hésitez pas à les arranger et subdiviser selon
142<strong>vos</strong> goûts et besoins, quitte à en modifier
143l'organisation par défaut.</p>
144
145<p>La configuration du serveur s'effectue via des <a href="mod/quickreference.html">directives de configuration</a> que l'on
146insère dans les fichiers de configuration. Une directive se compose d'un
147mot-clé suivi d'un ou plusieurs arguments qui définissent sa valeur.</p>
148
149<p>La réponse à la question "<em>Où dois-je placer cette directive ?</em>"
150dépend en général du niveau auquel cette directive doit être prise en compte.
151S'il s'agit du niveau global, elle doit être placée dans le fichier de
152configuration principal, et en dehors de toute section  <code class="directive"><a href="./mod/core.html#directory">&lt;Directory&gt;</a></code>, <code class="directive"><a href="./mod/core.html#location">&lt;Location&gt;</a></code>, <code class="directive"><a href="./mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code> ou de toute autre section.  Sinon, si par
153exemple elle ne doit s'appliquer qu'à un répertoire particulier, elle doit être
154placée dans la section <code class="directive"><a href="./mod/core.html#directory">&lt;Directory&gt;</a></code> qui fait référence à ce répertoire.  Voir la
155documentation sur les <a href="sections.html">Sections de configuration</a> pour
156plus de détails.</p>
157
158<p>En complément des fichiers de configuration principaux, certaines directives
159peuvent être insérées dans des fichiers <code>.htaccess</code> que l'on place
160directement dans le répertoire concerné. Les fichiers <code>.htaccess</code>
161sont essentiellement destinés aux personnes qui n'ont pas accès aux fichiers de
162configuration du serveur. Vous trouverez plus de détails à propos des fichiers
163<code>.htaccess</code> dans ce <a href="howto/htaccess.html"><code>.htaccess</code>howto</a>.</p>

167<h2><a name="content" id="content">Contenu du site web</a></h2>
168
169
170<p>Si le contenu du site web peut se présenter sous de nombreuses
171formes, il en existe deux principales : les
172contenus statiques et les contenus dynamiques.</p>
173
174<p>Les contenus statiques sont par exemple les fichiers HTML, les
175images, les fichiers CSS et tout autre fichier résidant dans le système
176de fichiers. La directive <code class="directive"><a href="./mod/core.html#documentroot">DocumentRoot</a></code> permet de définir la position
177dans l'arborescence du site où vous devez placer ces fichiers. Cette
178directive peut être définie au niveau global, ou au niveau de chaque
179serveur virtuel. Vous pouvez consulter vos fichiers de configuration
180pour vérifier la manière dont cette directive est définie pour votre
181serveur.</p>
182
183<p>En général, et si aucun nom de fichier n'est spécifié dans la
184requête, c'est une page de nom <code>index.html</code> qui sera
185renvoyée. Par exemple, si la directive <code>DocumentRoot</code> est
186définie à <code>/var/www/html</code>, et si une requête est effectuée
187pour l'adresse <code>http://www.example.com/work/</code>, c'est le
188fichier <code>/var/www/html/work/index.html</code> qui sera envoyé au
189client par le serveur.</p>
190
191<p>Un contenu dynamique est un contenu qui est généré au moment du
192traitement de la requête, et qui peut différer d'une requête à l'autre.
193Ces contenus dynamiques peuvent être générés de nombreuses manières par
194l'intermédiaire de <a href="handler.html">gestionnaires de contenu</a>
195ou "handlers". Il est aussi possible de créer des <a href="howto/cgi.html">programmes CGI</a> pour générer le contenu de
196votre site.</p>
197
198<p>Enfin, on peut utiliser des modules tiers comme mod_php pour écrire
199du code permettant d'effectuer de nombreuses choses. De nombreuses
200applications tierces écrites à partir de divers langages ou outils sont
201disponibles en téléchargement et peuvent être installées sur votre
202serveur HTTP Apache. Le support de ces applications est en dehors du sujet de
203ce document, et nous vous invitons à consulter le site de leur éditeur
204pour accéder à leur documentation.</p>

207<h2><a name="logs" id="logs">Fichiers journaux et résolution des problèmes</a></h2>
208
209<p>En tant qu'administrateur d'un serveur HTTP Apache, vos sources
210d'informations principales sont les fichiers journaux, et en particulier
211le journal des erreurs. Toute tentative de résolution d'un problème sans
212consulter le journal des erreurs revient à essayer de conduire les yeux
213fermés.</p>
214
215<p>La position dans le système de fichiers du journal des erreurs est
216spécifiée par la directive <code class="directive"><a href="./mod/core.html#errorlog">ErrorLog</a></code>
217qui peut être définie au niveau global, ou au niveau de chaque serveur
218virtuel. Chaque entrée du journal des erreurs vous informe sur la nature
219des problèmes et le moment de leur survenue. En outre, elle vous indique
220souvent comment résoudre le problème. Chaque message d'erreur contient
221un code d'erreur que vous pouvez utiliser pour effectuer une recherche
222en ligne afin d'obtenir une description plus détaillée de la manière de
223résoudre le problème. Vous pouvez aussi configurer votre journal des
224erreurs de manière à ce qu'il enregistre un identifiant d'erreur que
225vous pourrez ensuite utiliser pour effectuer une corrélation avec le
226journal des accès afin de déterminer quelle requête est à l'origine de
227l'erreur.</p>
228
229<p>Vous trouverez plus de détails à ce sujet dans la <a href="logs.html">Documentation sur la journalisation</a>.</p>
