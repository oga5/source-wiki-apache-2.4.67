29
30  <p>In order to assist folks upgrading, we maintain a document
31  describing information critical to existing Apache HTTP Server users. These
32  are intended to be brief notes, and you should be able to find
33  more information in either the <a href="new_features_2_4.html">New Features</a> document, or in
34  the <code>src/CHANGES</code> file.  Application and module developers
35  can find a summary of API changes in the <a href="developer/new_api_2_4.html">API updates</a> overview.</p>
36
37  <p>This document describes changes in server behavior that might
38  require you to change your configuration or how you use the server
39  in order to continue using 2.4 as you are currently using 2.2.
40  To take advantage of new features in 2.4, see the New Features
41  document.</p>
42
43  <p>This document describes only the changes from 2.2 to 2.4.  If you
44  are upgrading from version 2.0, you should also consult the <a href="http://httpd.apache.org/docs/2.2/upgrading.html">2.0 to 2.2
45  upgrading document.</a></p>

57<h2><a name="compile-time" id="compile-time">Compile-Time Configuration Changes</a></h2>
58    
59
60    <p>The compilation process is very similar to the one used in
61    version 2.2.  Your old <code>configure</code> command line (as
62    found in <code>build/config.nice</code> in the installed server
63    directory) can be used in most cases.  There are some changes in
64    the default settings.  Some details of changes:</p>
65
66    <ul>
67      <li>These modules have been removed: mod_authn_default,
68      mod_authz_default, mod_mem_cache.  If you were using
69      mod_mem_cache in 2.2, look at <code class="module"><a href="./mod/mod_cache_disk.html">mod_cache_disk</a></code> in
70      2.4.</li>
71
72      <li>All load balancing implementations have been moved to
73      individual, self-contained mod_proxy submodules, e.g.
74      <code class="module"><a href="./mod/mod_lbmethod_bybusyness.html">mod_lbmethod_bybusyness</a></code>.  You might need
75      to build and load any of these that your configuration
76      uses.</li>
77
78      <li>Platform support has been removed for BeOS, TPF, and
79      even older platforms such as A/UX, Next, and Tandem.  These were
80      believed to be broken anyway.</li>
81
82      <li>configure: dynamic modules (DSO) are built by default</li>
83
84      <li>configure: By default, only a basic set of modules is loaded. The
85      other <code class="directive">LoadModule</code> directives are commented
86      out in the configuration file.</li>
87
88      <li>configure: the "most" module set gets built by default</li>
89
90      <li>configure: the "reallyall" module set adds developer modules
91      to the "all" set</li>
92    </ul>

96<h2><a name="run-time" id="run-time">Run-Time Configuration Changes</a></h2>
97    
98    <p>There have been significant changes in authorization configuration,
99    and other minor configuration changes, that could require changes to your 2.2
100    configuration files before using them for 2.4.</p>
101
102    <h3><a name="authz" id="authz">Authorization</a></h3>
103      
104
105      <p>Any configuration file that uses authorization will likely
106      need changes.</p>
107
108    <p>You should review the <a href="howto/auth.html">Authentication,
109    Authorization and Access Control Howto</a>, especially the section
110    <a href="howto/auth.html#beyond">Beyond just authorization</a>
111    which explains the new mechanisms for controlling the order in
112    which the authorization directives are applied.</p>
113
114    <p>Directives that control how authorization modules respond when they don't match
115    the authenticated user have been removed: This includes 
116    AuthzLDAPAuthoritative, AuthzDBDAuthoritative, AuthzDBMAuthoritative, 
117    AuthzGroupFileAuthoritative, AuthzUserAuthoritative,
118    and AuthzOwnerAuthoritative.   These directives have been replaced by the
119    more expressive <code class="directive"><a href="./mod/mod_authz_core.html#requireany">RequireAny</a></code>, 
120    <code class="directive"><a href="./mod/mod_authz_core.html#requirenone">RequireNone</a></code>, and
121    <code class="directive"><a href="./mod/mod_authz_core.html#requireall">RequireAll</a></code>.</p>
122
123    <p>If you use <code class="module"><a href="./mod/mod_authz_dbm.html">mod_authz_dbm</a></code>, you must port your 
124    configuration to use <code>Require dbm-group ...</code> in place
125    of <code>Require group ...</code>.</p>
126
127    <h4><a name="access" id="access">Access control</a></h4>
128      
129
130      <p>In 2.2, access control based on client hostname, IP address,
131      and other characteristics of client requests was done using the
132      directives <code class="directive"><a href="./mod/mod_access_compat.html#order">Order</a></code>, <code class="directive"><a href="./mod/mod_access_compat.html#allow">Allow</a></code>, <code class="directive"><a href="./mod/mod_access_compat.html#deny">Deny</a></code>, and <code class="directive"><a href="./mod/mod_access_compat.html#satisfy">Satisfy</a></code>.</p>
133
134      <p>In 2.4, such access control is done in the same way as other
135      authorization checks, using the new module
136      <code class="module"><a href="./mod/mod_authz_host.html">mod_authz_host</a></code>.  The old access control idioms
137      should be replaced by the new authentication mechanisms,
138      although for compatibility with old configurations, the new
139      module <code class="module"><a href="./mod/mod_access_compat.html">mod_access_compat</a></code> is provided.</p>
140
141      <div class="note"><h3>Mixing old and new directives</h3>
142      <p>Mixing old directives like <code class="directive"><a href="./mod/mod_access_compat.html#order">Order</a></code>, <code class="directive"><a href="./mod/mod_access_compat.html#allow">Allow</a></code> or <code class="directive"><a href="./mod/mod_access_compat.html#deny">Deny</a></code> with new ones like
143      <code class="directive"><a href="./mod/mod_authz_core.html#require">Require</a></code> is technically possible 
144      but discouraged. <code class="module"><a href="./mod/mod_access_compat.html">mod_access_compat</a></code> was created to support 
145      configurations containing only old directives to facilitate the 2.4 upgrade. 
146      Please check the examples below to get a better idea about issues that might arise.
147      </p>
148      </div>
149
150      <p>Here are some examples of old and new ways to do the same
151      access control.</p>
152
153      <p>In this example, there is no authentication and all requests are denied.</p>
154      <div class="example"><h3>2.2 configuration:</h3><pre class="prettyprint lang-config">Order deny,allow
155Deny from all</pre>
156</div>
157      <div class="example"><h3>2.4 configuration:</h3><pre class="prettyprint lang-config">Require all denied</pre>
158</div>
159
160      <p>In this example, there is no authentication and all requests are allowed.</p>
161      <div class="example"><h3>2.2 configuration:</h3><pre class="prettyprint lang-config">Order allow,deny
162Allow from all</pre>
163</div>
164      <div class="example"><h3>2.4 configuration:</h3><pre class="prettyprint lang-config">Require all granted</pre>
165</div>
166
167      <p>In the following example, there is no authentication and all hosts in the example.org domain
168      are allowed access; all other hosts are denied access.</p>
169
170      <div class="example"><h3>2.2 configuration:</h3><pre class="prettyprint lang-config">Order Deny,Allow
171Deny from all
172Allow from example.org</pre>
173</div>
174      <div class="example"><h3>2.4 configuration:</h3><pre class="prettyprint lang-config">Require host example.org</pre>
175</div>
176
177      <p>In the following example, mixing old and new directives leads to 
178      unexpected results.</p>
179 
180      <div class="example"><h3>Mixing old and new directives: NOT WORKING AS EXPECTED</h3><pre class="prettyprint lang-config">DocumentRoot "/var/www/html"
181
182&lt;Directory "/"&gt;
183    AllowOverride None
184    Order deny,allow
185    Deny from all
186&lt;/Directory&gt;
187
188&lt;Location "/server-status"&gt;
189    SetHandler server-status
190    Require local
191&lt;/Location&gt;
192
193access.log - GET /server-status 403 127.0.0.1
194error.log - AH01797: client denied by server configuration: /var/www/html/server-status</pre>
195</div>
196      <p>Why httpd denies access to servers-status even if the configuration seems to allow it?
197        Because <code class="module"><a href="./mod/mod_access_compat.html">mod_access_compat</a></code> directives take precedence
198        over the <code class="module"><a href="./mod/mod_authz_host.html">mod_authz_host</a></code> one in this configuration 
199        <a href="sections.html#merging">merge</a> scenario.</p>
200
201      <p>This example conversely works as expected:</p>
202
203      <div class="example"><h3>Mixing old and new directives: WORKING AS EXPECTED</h3><pre class="prettyprint lang-config">DocumentRoot "/var/www/html"
204
205&lt;Directory "/"&gt;
206    AllowOverride None
207    Require all denied
208&lt;/Directory&gt;
209
210&lt;Location "/server-status"&gt;
211    SetHandler server-status
212    Order deny,allow
213    Deny from all
214    Allow From 127.0.0.1
215&lt;/Location&gt;
216
217access.log - GET /server-status 200 127.0.0.1</pre>
218</div> 
219      <p>So even if mixing configuration is still
220        possible, please try to avoid it when upgrading: either keep old directives and then migrate
221        to the new ones on a later stage or just migrate everything in bulk.  
222      </p>
223    
224
225     <p>In many configurations with authentication, where the value of the
226     <code class="directive">Satisfy</code> was the default of <em>ALL</em>, snippets
227     that simply disabled host-based access control are omitted:</p>
228
229      <div class="example"><h3>2.2 configuration:</h3><pre class="prettyprint lang-config"># 2.2 config that disables host-based access control and uses only authentication
230Order Deny,Allow
231Allow from all
232AuthType Basic
233AuthBasicProvider file
234AuthUserFile /example.com/conf/users.passwd
235AuthName secure
236Require valid-user</pre>
237</div>
238      <div class="example"><h3>2.4 configuration:</h3><pre class="prettyprint lang-config"># No replacement of disabling host-based access control needed
239AuthType Basic
240AuthBasicProvider file
241AuthUserFile /example.com/conf/users.passwd
242AuthName secure
243Require valid-user</pre>
244</div>
245
246     <p>In configurations where both authentication and access control were meaningfully combined, the 
247        access control directives should be migrated. This example allows requests meeting <em>both</em> criteria:</p>
248      <div class="example"><h3>2.2 configuration:</h3><pre class="prettyprint lang-config">Order allow,deny
249Deny from all
250# Satisfy ALL is the default
251Satisfy ALL
252Allow from 127.0.0.1
253AuthType Basic
254AuthBasicProvider file
255AuthUserFile /example.com/conf/users.passwd
256AuthName secure
257Require valid-user</pre>
258</div>
259      <div class="example"><h3>2.4 configuration:</h3><pre class="prettyprint lang-config">AuthType Basic
260AuthBasicProvider file
261AuthUserFile /example.com/conf/users.passwd
262AuthName secure
263&lt;RequireAll&gt;
264  Require valid-user
265  Require ip 127.0.0.1
266&lt;/RequireAll&gt;</pre>
267</div>
268
269     <p>In configurations where both authentication and access control were meaningfully combined, the 
270        access control directives should be migrated. This example allows requests meeting <em>either</em> criteria:</p>
271      <div class="example"><h3>2.2 configuration:</h3><pre class="prettyprint lang-config">Order allow,deny
272Deny from all
273Satisfy any
274Allow from 127.0.0.1
275AuthType Basic
276AuthBasicProvider file
277AuthUserFile /example.com/conf/users.passwd
278AuthName secure
279Require valid-user</pre>
280</div>
281      <div class="example"><h3>2.4 configuration:</h3><pre class="prettyprint lang-config">AuthType Basic
282AuthBasicProvider file
283AuthUserFile /example.com/conf/users.passwd
284AuthName secure
285# Implicitly &lt;RequireAny&gt;
286Require valid-user
287Require ip 127.0.0.1</pre>
288</div>

291
292    <h3><a name="config" id="config">Other configuration changes</a></h3>
293      
294
295      <p>Some other small adjustments may be necessary for particular
296      configurations as discussed below.</p>
297
298      <ul>
299        <li><code class="directive">MaxRequestsPerChild</code> has been renamed to
300        <code class="directive"><a href="./mod/mpm_common.html#maxconnectionsperchild">MaxConnectionsPerChild</a></code>,
301        describes more accurately what it does. The old name is still
302        supported.</li>
303
304        <li><code class="directive">MaxClients</code> has been renamed to
305        <code class="directive"><a href="./mod/mpm_common.html#maxrequestworkers">MaxRequestWorkers</a></code>,
306        which describes more accurately what it does. For async MPMs, like
307        <code class="module"><a href="./mod/event.html">event</a></code>, the maximum number of clients is not
308        equivalent than the number of worker threads. The old name is still
309        supported.</li>
310
311        <li>The <code class="directive"><a href="./mod/core.html#defaulttype">DefaultType</a></code>
312        directive no longer has any effect, other than to emit a
313        warning if it's used with any value other than
314        <code>none</code>.  You need to use other configuration
315        settings to replace it in 2.4.
316        </li>
317
318        <li><code class="directive"><a href="./mod/core.html#allowoverride">AllowOverride</a></code> now
319        defaults to <code>None</code>.</li>
320
321        <li><code class="directive"><a href="./mod/core.html#enablesendfile">EnableSendfile</a></code> now
322        defaults to Off.</li>
323
324        <li><code class="directive"><a href="./mod/core.html#fileetag">FileETag</a></code> now
325        defaults to "MTime Size" (without INode).</li>
326
327        <li><code class="module"><a href="./mod/mod_dav_fs.html">mod_dav_fs</a></code>: The format of the <code class="directive"><a href="./mod/mod_dav_fs.html#davlockdb">DavLockDB</a></code> file has changed for
328        systems with inodes.  The old <code class="directive"><a href="./mod/mod_dav_fs.html#davlockdb">DavLockDB</a></code> file must be deleted on
329        upgrade.
330        </li>
331
332        <li><code class="directive"><a href="./mod/core.html#keepalive">KeepAlive</a></code> only
333        accepts values of <code>On</code> or <code>Off</code>.
334        Previously, any value other than "Off" or "0" was treated as
335        "On".</li>
336
337        <li>Directives AcceptMutex, LockFile, RewriteLock, SSLMutex,
338        SSLStaplingMutex, and WatchdogMutexPath have been replaced
339        with a single <code class="directive"><a href="./mod/core.html#mutex">Mutex</a></code>
340        directive.  You will need to evaluate any use of these removed
341        directives in your 2.2 configuration to determine if they can
342        just be deleted or will need to be replaced using <code class="directive"><a href="./mod/core.html#mutex">Mutex</a></code>.</li>
343
344        <li><code class="module"><a href="./mod/mod_cache.html">mod_cache</a></code>: <code class="directive"><a href="./mod/mod_cache.html#cacheignoreurlsessionidentifiers">CacheIgnoreURLSessionIdentifiers</a></code>
345        now does an exact match against the query string instead of a
346        partial match.  If your configuration was using partial
347        strings, e.g. using <code>sessionid</code> to match
348        <code>/someapplication/image.gif;jsessionid=123456789</code>,
349        then you will need to change to the full string
350        <code>jsessionid</code>.
351        </li>
352
353        <li><code class="module"><a href="./mod/mod_cache.html">mod_cache</a></code>: The second parameter to 
354        <code class="directive"><a href="./mod/mod_cache.html#cacheenable">CacheEnable</a></code> only
355        matches forward proxy content if it begins with the correct
356        protocol. In 2.2 and earlier, a parameter of '/' matched all
357        content.</li>
358
359        <li><code class="module"><a href="./mod/mod_ldap.html">mod_ldap</a></code>: <code class="directive"><a href="./mod/mod_ldap.html#ldaptrustedclientcert">LDAPTrustedClientCert</a></code> is now
360        consistently a per-directory setting only.  If you use this
361        directive, review your configuration to make sure it is
362        present in all the necessary directory contexts.</li>
363
364        <li><code class="module"><a href="./mod/mod_filter.html">mod_filter</a></code>: <code class="directive"><a href="./mod/mod_filter.html#filterprovider">FilterProvider</a></code> syntax has changed and
365        now uses a boolean expression to determine if a filter is applied.
366        </li>
367
368        <li><code class="module"><a href="./mod/mod_include.html">mod_include</a></code>:
369            <ul>
370            <li>The <code>#if expr</code> element now uses the new <a href="expr.html">expression parser</a>. The old syntax can be
371            restored with the new directive <code class="directive"><a href="./mod/mod_include.html#ssilegacyexprparser">SSILegacyExprParser</a></code>.
372            </li>
373            <li>An SSI* config directive in directory scope no longer causes
374            all other per-directory SSI* directives to be reset to their
375            default values.</li>
376            </ul>
377        </li>
378
379        <li><code class="module"><a href="./mod/mod_charset_lite.html">mod_charset_lite</a></code>: The <code>DebugLevel</code>
380        option has been removed in favor of per-module <code class="directive"><a href="./mod/core.html#loglevel">LogLevel</a></code> configuration.
381        </li>
382
383        <li><code class="module"><a href="./mod/mod_ext_filter.html">mod_ext_filter</a></code>: The <code>DebugLevel</code>
384        option has been removed in favor of per-module <code class="directive"><a href="./mod/core.html#loglevel">LogLevel</a></code> configuration.
385        </li>
386
387        <li><code class="module"><a href="./mod/mod_proxy_scgi.html">mod_proxy_scgi</a></code>: The default setting for
388        <code>PATH_INFO</code> has changed from httpd 2.2, and
389        some web applications will no longer operate properly with
390        the new <code>PATH_INFO</code> setting.  The previous setting
391        can be restored by configuring the <code>proxy-scgi-pathinfo</code>
392        variable.</li>
393
394        <li><code class="module"><a href="./mod/mod_ssl.html">mod_ssl</a></code>: CRL based revocation checking
395        now needs to be explicitly configured through <code class="directive"><a href="./mod/mod_ssl.html#sslcarevocationcheck">SSLCARevocationCheck</a></code>.
396        </li>
397
398        <li><code class="module"><a href="./mod/mod_substitute.html">mod_substitute</a></code>: The maximum line length is now
399        limited to 1MB.
400        </li>
401
402        <li><code class="module"><a href="./mod/mod_reqtimeout.html">mod_reqtimeout</a></code>: If the module is loaded, it
403        will now set some default timeouts.</li>
404
405        <li><code class="module"><a href="./mod/mod_dumpio.html">mod_dumpio</a></code>: <code class="directive">DumpIOLogLevel</code>
406        is no longer supported.  Data is always logged at <code class="directive"><a href="./mod/core.html#loglevel">LogLevel</a></code> <code>trace7</code>.</li>
407
408        <li>On Unix platforms, piped logging commands configured using
409        either <code class="directive"><a href="./mod/core.html#errorlog">ErrorLog</a></code> or
410        <code class="directive"><a href="./mod/mod_log_config.html#customlog">CustomLog</a></code> were invoked using
411        <code>/bin/sh -c</code> in 2.2 and earlier.  In 2.4 and later,
412        piped logging commands are executed directly.  To restore the
413        old behavior, see the <a href="logs.html#piped">piped logging
414        documentation</a>.</li>
415
416      </ul>

420<h2><a name="misc" id="misc">Misc Changes</a></h2>
421    
422
423    <ul>
424      <li><code class="module"><a href="./mod/mod_autoindex.html">mod_autoindex</a></code>: will now extract titles and
425      display descriptions for .xhtml files, which were previously
426      ignored.</li>
427
428      <li><code class="module"><a href="./mod/mod_ssl.html">mod_ssl</a></code>: The default format of the <code>*_DN</code>
429      variables has changed. The old format can still be used with the new
430      <code>LegacyDNStringFormat</code> argument to <code class="directive"><a href="./mod/mod_ssl.html#ssloptions">SSLOptions</a></code>. The SSLv2 protocol is
431      no longer supported. <code class="directive"><a href="./mod/mod_ssl.html#sslproxycheckpeercn">SSLProxyCheckPeerCN
432	  </a></code> and <code class="directive"><a href="./mod/mod_ssl.html#sslproxycheckpeerexpire">SSLProxyCheckPeerExpire
433	  </a></code> now default to On, causing proxy requests to HTTPS hosts
434	  with bad or outdated certificates to fail with a 502 status code (Bad 
435	  gateway)</li>
436
437      <li><code class="program"><a href="./programs/htpasswd.html">htpasswd</a></code> now uses MD5 hash by default on
438      all platforms.</li>
439
440      <li>The <code class="directive"><a href="./mod/core.html#namevirtualhost">NameVirtualHost</a></code>
441      directive no longer has any effect, other than to emit a
442      warning.  Any address/port combination appearing in multiple
443      virtual hosts is implicitly treated as a name-based virtual host.
444      </li>
445
446      <li><code class="module"><a href="./mod/mod_deflate.html">mod_deflate</a></code> will now skip compression if it knows
447      that the size overhead added by the compression is larger than the data
448      to be compressed.
449      </li>
450
451      <li>Multi-language error documents from 2.2.x may not work unless
452      they are adjusted to the new syntax of <code class="module"><a href="./mod/mod_include.html">mod_include</a></code>'s
453      <code>#if expr=</code> element or the directive
454      <code class="directive"><a href="./mod/mod_include.html#ssilegacyexprparser">SSILegacyExprParser</a></code> is
455      enabled for the directory containing the error documents.
456      </li>
457
458      <li>The functionality provided by <code>mod_authn_alias</code>
459      in previous versions (i.e., the <code class="directive"><a href="./mod/mod_authn_core.html#authnprovideralias">AuthnProviderAlias</a></code> directive)
460      has been moved into <code class="module"><a href="./mod/mod_authn_core.html">mod_authn_core</a></code>.  
461      </li>
462
463      <li>The RewriteLog and RewriteLogLevel directives have been removed.
464      This functionality is now provided by configuring the appropriate
465      level of logging for the <code class="module"><a href="./mod/mod_rewrite.html">mod_rewrite</a></code> module using
466      the <code class="directive"><a href="./mod/core.html#loglevel">LogLevel</a></code> directive.
467      See also the <a href="mod/mod_rewrite.html#logging">mod_rewrite logging</a>
468      section.</li>
469
470      <li><code class="module"><a href="./mod/mod_include.html">mod_include</a></code>'s <code>&lt;set&gt;</code> tag no 
471      longer performs entity decoding on the value being set.</li>
472    </ul>

476<h2><a name="third-party" id="third-party">Third Party Modules</a></h2>
477    
478    <p>All modules must be recompiled for 2.4 before being loaded.</p>

481    otherwise work unchanged with the Apache HTTP Server version 2.4.

483    update</a> overview.</p>

486<h2><a name="commonproblems" id="commonproblems">Common problems when upgrading</a></h2>
487    
488    <ul><li>Startup errors:
489    <ul>
490      <li><code>Invalid command 'User', perhaps misspelled or defined by a module not included in the server configuration</code> - load module <code class="module"><a href="./mod/mod_unixd.html">mod_unixd</a></code></li>
491      <li><code>Invalid command 'Require', perhaps misspelled or defined by a module not included in the server configuration</code>, or
492<code>Invalid command 'Order', perhaps misspelled or defined by a module not included in the server configuration</code>
493 - load module <code class="module"><a href="./mod/mod_access_compat.html">mod_access_compat</a></code>, or update configuration to 2.4 authorization directives.</li>
494      <li><code>Ignoring deprecated use of DefaultType in line NN of /path/to/httpd.conf</code> - remove <code class="directive"><a href="./mod/core.html#defaulttype">DefaultType</a></code>
495      and replace with other configuration settings.</li>
496      <li><code>Invalid command 'AddOutputFilterByType', perhaps misspelled 
497      or defined by a module not included in the server configuration
498      </code> - <code class="directive"><a href="./mod/mod_filter.html#addoutputfilterbytype">AddOutputFilterByType</a></code> 
499      has moved from the core to mod_filter, which must be loaded.</li>
500    </ul></li>
501    <li>Errors serving requests:
502    <ul>
503      <li><code>configuration error:  couldn't check user: /path</code> -
504      load module <code class="module"><a href="./mod/mod_authn_core.html">mod_authn_core</a></code>.</li>
505      <li><code>.htaccess</code> files aren't being processed - Check for an
506      appropriate <code class="directive"><a href="./mod/core.html#allowoverride">AllowOverride</a></code> directive;
507      the default changed to <code>None</code> in 2.4.</li>
508    </ul>
509    </li>
510</ul>
