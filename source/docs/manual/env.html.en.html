32
33    <p>There are two kinds of environment variables that affect
34    the Apache HTTP Server.</p>
35
36    <p>First, there are the environment variables controlled by
37    the underlying operating system.  These are set before the
38    server starts.  They can be used in expansions in configuration
39    files, and can optionally be passed to CGI scripts and SSI
40    using the PassEnv directive.</p>
41
42    <p>Second, the Apache HTTP Server provides a mechanism for storing
43    information in named variables that are also called <em>environment
44    variables</em>. This information can be used to control various
45    operations such as logging or access control. The variables are
46    also used as a mechanism to communicate with external programs
47    such as CGI scripts. This document discusses different ways to
48    manipulate and use these variables.</p>
49
50    <p>Although these variables are referred to as <em>environment
51    variables</em>, they are not the same as the environment
52    variables controlled by the underlying operating system.
53    Instead, these variables are stored and manipulated in an
54    internal httpd structure. They only become actual operating
55    system environment variables when they are provided to CGI
56    scripts and Server Side Include scripts. If you wish to
57    manipulate the operating system environment under which the
58    server itself runs, you must use the standard environment
59    manipulation mechanisms provided by your operating system
60    shell.</p>

74    <h3><a name="basic-manipulation" id="basic-manipulation">Basic Environment Manipulation</a></h3>
75        
76
77        <p>The most basic way to set an environment variable in Apache
78        is using the unconditional <code class="directive"><a href="./mod/mod_env.html#setenv">SetEnv</a></code> directive. Variables may also be passed from
79        the environment of the shell which started the server using the
80        <code class="directive"><a href="./mod/mod_env.html#passenv">PassEnv</a></code> directive.</p>
81
82    
83    <h3><a name="conditional" id="conditional">Conditional Per-Request Settings</a></h3>
84        
85
86        <p>For additional flexibility, the directives provided by
87        <code class="module"><a href="./mod/mod_setenvif.html">mod_setenvif</a></code> allow environment variables to be set
88        on a per-request basis, conditional on characteristics of particular
89        requests. For example, a variable could be set only when a
90        specific browser (User-Agent) is making a request, or only when
91        a specific Referer [sic] header is found. Even more flexibility
92        is available through the <code class="module"><a href="./mod/mod_rewrite.html">mod_rewrite</a></code>'s <code class="directive"><a href="./mod/mod_rewrite.html#rewriterule">RewriteRule</a></code> which uses the
93        <code>[E=...]</code> option to set environment variables.</p>
94
95    
96    <h3><a name="unique-identifiers" id="unique-identifiers">Unique Identifiers</a></h3>
97        
98
99        <p>Finally, <code class="module"><a href="./mod/mod_unique_id.html">mod_unique_id</a></code> sets the environment
100        variable <code>UNIQUE_ID</code> for each request to a value which is
101        guaranteed to be unique across "all" requests under very
102        specific conditions.</p>
103
104    
105    <h3><a name="standard-cgi" id="standard-cgi">Standard CGI Variables</a></h3>
106        
107
108        <p>In addition to all environment variables set within the
109        Apache configuration and passed from the shell, CGI scripts and
110        SSI pages are provided with a set of environment variables
111        containing meta-information about the request as required by
112	the <a href="http://www.ietf.org/rfc/rfc3875">CGI
113        specification</a>.</p>
114
115    
116    <h3><a name="caveats" id="caveats">Some Caveats</a></h3>
117        
118
119        <ul>
120          <li>It is not possible to override or change the standard CGI
121          variables using the environment manipulation directives.</li>
122
123          <li>When <code class="program"><a href="./programs/suexec.html">suexec</a></code> is used to launch
124          CGI scripts, the environment will be cleaned down to a set of
125          <em>safe</em> variables before CGI scripts are launched. The
126          list of <em>safe</em> variables is defined at compile-time in
127          <code>suexec.c</code>.</li>
128
129          <li>For portability reasons, the names of environment
130          variables may contain only letters, numbers, and the
131          underscore character. In addition, the first character may
132          not be a number. Characters which do not match this
133          restriction will be replaced by an underscore when passed to
134          CGI scripts and SSI pages.</li>
135
136          <li>A special case are HTTP headers which are passed to CGI
137          scripts and the like via environment variables (see below).
138          They are converted to uppercase and only dashes are replaced with
139          underscores; if the header contains any other (invalid) character,
140          the whole header is silently dropped. See <a href="#fixheader">
141          below</a> for a workaround.</li>
142
143          <li>The <code class="directive"><a href="./mod/mod_env.html#setenv">SetEnv</a></code> directive runs
144          late during request processing meaning that directives such as
145          <code class="directive"><a href="./mod/mod_setenvif.html#setenvif">SetEnvIf</a></code> and <code class="directive"><a href="./mod/mod_rewrite.html#rewritecond">RewriteCond</a></code> will not see the
146          variables set with it.</li>
147
148          <li>When the server looks up a path via an internal
149          <a class="glossarylink" href="./glossary.html#subrequest" title="see glossary">subrequest</a> such as looking
150          for a <code class="directive"><a href="./mod/mod_dir.html#directoryindex">DirectoryIndex</a></code>
151          or generating a directory listing with <code class="module"><a href="./mod/mod_autoindex.html">mod_autoindex</a></code>,
152          per-request environment variables are <em>not</em> inherited in the 
153          subrequest. Additionally, 
154          <code class="directive"><a href="./mod/mod_setenvif.html#setenvif">SetEnvIf</a></code> directives
155          are not separately evaluated in the subrequest due to the API phases
156           <code class="module"><a href="./mod/mod_setenvif.html">mod_setenvif</a></code> takes action in.</li>
157        </ul>

166    <h3><a name="cgi-scripts" id="cgi-scripts">CGI Scripts</a></h3>
167        
168
169        <p>One of the primary uses of environment variables is to
170        communicate information to CGI scripts. As discussed above, the
171        environment passed to CGI scripts includes standard
172        meta-information about the request in addition to any variables
173        set within the Apache configuration. For more details, see the
174        <a href="howto/cgi.html">CGI tutorial</a>.</p>
175
176    
177    <h3><a name="ssi-pages" id="ssi-pages">SSI Pages</a></h3>
178        
179
180        <p>Server-parsed (SSI) documents processed by
181        <code class="module"><a href="./mod/mod_include.html">mod_include</a></code>'s
182        <code>INCLUDES</code> filter can print environment variables
183        using the <code>echo</code> element, and can use environment
184        variables in flow control elements to makes parts of a page
185        conditional on characteristics of a request. Apache also
186        provides SSI pages with the standard CGI environment variables
187        as discussed above. For more details, see the <a href="howto/ssi.html">SSI tutorial</a>.</p>
188
189    
190    <h3><a name="access-control" id="access-control">Access Control</a></h3>
191        
192
193        <p>Access to the server can be controlled based on
194        environment variables using the <code>Require env</code>
195        and <code>Require not env</code> directives. In combination with
196        <code class="directive"><a href="./mod/mod_setenvif.html#setenvif">SetEnvIf</a></code>, this
197        allows for flexible control of access to the server based on
198        characteristics of the client. For example, you can use these
199        directives to deny access to a particular browser (User-Agent).
200        </p>
201
202    
203    <h3><a name="logging" id="logging">Conditional Logging</a></h3>
204        
205
206        <p>Environment variables can be logged in the access log using
207        the <code class="directive"><a href="./mod/mod_log_config.html#logformat">LogFormat</a></code>
208        option <code>%e</code>. In addition, the decision on whether
209        or not to log requests can be made based on the status of
210        environment variables using the conditional form of the
211        <code class="directive"><a href="./mod/mod_log_config.html#customlog">CustomLog</a></code>
212        directive. In combination with <code class="directive"><a href="./mod/mod_setenvif.html#setenvif">SetEnvIf</a></code> this allows for flexible control of which
213        requests are logged. For example, you can choose not to log
214        requests for filenames ending in <code>gif</code>, or you can
215        choose to only log requests from clients which are outside your
216        subnet.</p>
217
218    
219    <h3><a name="response-headers" id="response-headers">Conditional Response Headers</a></h3>
220        
221
222        <p>The <code class="directive"><a href="./mod/mod_headers.html#header">Header</a></code>
223        directive can use the presence or
224        absence of an environment variable to determine whether or not
225        a certain HTTP header will be placed in the response to the
226        client. This allows, for example, a certain response header to
227        be sent only if a corresponding header is received in the
228        request from the client.</p>
229
230    
231
232    <h3><a name="external-filter" id="external-filter">External Filter Activation</a></h3>
233        
234
235        <p>External filters configured by <code class="module"><a href="./mod/mod_ext_filter.html">mod_ext_filter</a></code>
236        using the <code class="directive"><a href="./mod/mod_ext_filter.html#extfilterdefine">ExtFilterDefine</a></code> directive can
237        by activated conditional on an environment variable using the
238        <code>disableenv=</code> and <code>enableenv=</code> options.</p>
239    
240
241    <h3><a name="url-rewriting" id="url-rewriting">URL Rewriting</a></h3>
242        
243
244        <p>The <code>%{ENV:<em>variable</em>}</code> form of
245        <em>TestString</em> in the <code class="directive"><a href="./mod/mod_rewrite.html#rewritecond">RewriteCond</a></code> allows <code class="module"><a href="./mod/mod_rewrite.html">mod_rewrite</a></code>'s rewrite
246        engine to make decisions conditional on environment variables.
247        Note that the variables accessible in <code class="module"><a href="./mod/mod_rewrite.html">mod_rewrite</a></code>
248        without the <code>ENV:</code> prefix are not actually environment
249        variables. Rather, they are variables special to
250        <code class="module"><a href="./mod/mod_rewrite.html">mod_rewrite</a></code> which cannot be accessed from other
251        modules.</p>

255<h2><a name="cgi" id="cgi">CGI environment variables</a></h2>
256    
257
258    <p>The <a href="https://datatracker.ietf.org/doc/html/rfc3875#section-4.1">CGI specification</a> defines a number of environment
259    variables that expand on those defined by the HTTP spec.
260    These have been adopted more broadly, and are a standard
261    part of passing information between the browser and the
262    server, and between processes on the server side. Here we
263    discuss a few of these. Refer to the CGI spec for further
264    details.</p>
265
266    <h3><a name="query-string" id="query-string">QUERY_STRING</a></h3>
267        
268        <p>The <code>QUERY_STRING</code> variable is set to
269        anything that appears after a question mark (?) at the
270        end of a requested URL. This is used to pass
271        information to the server, and may be divided into
272        key/value pairs. The string should be
273        URL-encoded.</p>
274
275        <div class="example"><p><code>
276        https://example.com/path/resource.php?key1=value1&amp;key2=another%20value
277        </code></p></div>
278
279        <p>In the example here, the variable
280        <code>QUERY_STRING</code> is set to everything
281        following the ?, and the receiving application (e.g.
282        CGI or a PHP script, perhaps) is responsible for
283        parsing the key/value pairs.</p>
284
285        
286    
287
288    <h3><a name="path-info" id="path-info">PATH_INFO</a></h3>
289        
290
291        <p>The variable <code>PATH_INFO</code> is set to
292        the any value that appears after a matches resource
293        at the end of the URL, not including any
294        <code>QUERY_STRING</code>. This is best shown by
295        an example:</p>
296
297        <div class="example"><p><code>
298        https://example.com/resource.php/additional/data?key1=value1
299        </code></p></div>
300
301        <p>In the example shown,
302        <code>PATH_INFO</code> will be set to
303        <code>/additional/data</code> and
304        <code>QUERY_STRING</code> will be set to
305        <code>key1=value1</code>.</p>
306

310<h2><a name="special" id="special">Special Purpose Environment Variables</a></h2>
311    
312
313        <p>Interoperability problems have led to the introduction of
314        mechanisms to modify the way Apache behaves when talking to
315        particular clients. To make these mechanisms as flexible as
316        possible, they are invoked by defining environment variables,
317        typically with <code class="directive"><a href="./mod/mod_setenvif.html#browsermatch">BrowserMatch</a></code>, though <code class="directive"><a href="./mod/mod_env.html#setenv">SetEnv</a></code> and  <code class="directive"><a href="./mod/mod_env.html#passenv">PassEnv</a></code> could also be used, for example.</p>
318
319    <h3><a name="downgrade" id="downgrade">downgrade-1.0</a></h3>
320        
321
322        <p>This forces the request to be treated as a HTTP/1.0 request
323        even if it was in a later dialect.</p>
324
325    
326    <h3><a name="force-gzip" id="force-gzip">force-gzip</a></h3>
327        
328          <p>If you have the <code>DEFLATE</code> filter activated, this
329          environment variable will ignore the accept-encoding setting of
330          your browser and will send compressed output unconditionally.</p>
331    
332    <h3><a name="force-no-vary" id="force-no-vary">force-no-vary</a></h3>
333        
334
335        <p>This causes any <code>Vary</code> fields to be removed from
336        the response header before it is sent back to the client. Some
337        clients don't interpret this field correctly; setting this
338        variable can work around this problem. Setting this variable
339        also implies <strong>force-response-1.0</strong>.</p>
340
341    
342    <h3><a name="force-response" id="force-response">force-response-1.0</a></h3>
343        
344
345      <p>This forces an HTTP/1.0 response to clients making an HTTP/1.0
346      request. It was originally
347      implemented as a result of a problem with AOL's proxies. Some
348      HTTP/1.0 clients may not behave correctly when given an HTTP/1.1
349      response, and this can be used to interoperate with them.</p>
350
351    
352
353    <h3><a name="gzip-only-text-html" id="gzip-only-text-html">gzip-only-text/html</a></h3>
354        
355
356        <p>When set to a value of "1", this variable disables the
357        <code>DEFLATE</code> output filter provided by
358        <code class="module"><a href="./mod/mod_deflate.html">mod_deflate</a></code> for content-types other than
359        <code>text/html</code>. If you'd rather
360        use statically compressed files, <code class="module"><a href="./mod/mod_negotiation.html">mod_negotiation</a></code>
361        evaluates the variable as well (not only for gzip, but for all
362        encodings that differ from "identity").</p>
363    
364
365    <h3><a name="no-gzip" id="no-gzip">no-gzip</a></h3>
366
367        <p>When set, the <code>DEFLATE</code> filter of
368        <code class="module"><a href="./mod/mod_deflate.html">mod_deflate</a></code> will be turned off and
369        <code class="module"><a href="./mod/mod_negotiation.html">mod_negotiation</a></code> will refuse to deliver encoded
370        resources.</p>
371
372    
373
374    <h3><a name="no-cache" id="no-cache">no-cache</a></h3>
375        <p><em>Available in versions 2.2.12 and later</em></p>
376
377        <p>When set, <code class="module"><a href="./mod/mod_cache.html">mod_cache</a></code> will not save an otherwise
378        cacheable response.  This environment variable does not influence
379        whether a response already in the cache will be served for the current
380        request.</p>
381
382    
383
384    <h3><a name="nokeepalive" id="nokeepalive">nokeepalive</a></h3>
385        
386
387        <p>This disables <code class="directive"><a href="./mod/core.html#keepalive">KeepAlive</a></code>
388        when set.</p>
389
390    
391
392    <h3><a name="prefer-language" id="prefer-language">prefer-language</a></h3>
393
394        <p>This influences <code class="module"><a href="./mod/mod_negotiation.html">mod_negotiation</a></code>'s behavior. If
395        it contains a language tag (such as <code>en</code>, <code>ja</code>
396        or <code>x-klingon</code>), <code class="module"><a href="./mod/mod_negotiation.html">mod_negotiation</a></code> tries
397        to deliver a variant with that language. If there's no such variant,
398        the normal <a href="content-negotiation.html">negotiation</a> process
399        applies.</p>
400
401    
402
403    <h3><a name="redirect-carefully" id="redirect-carefully">redirect-carefully</a></h3>
404        
405
406        <p>This forces the server to be more careful when sending a redirect
407        to the client.  This is typically used when a client has a known
408        problem handling redirects.  This was originally implemented as a
409        result of a problem with Microsoft's WebFolders software which has
410        a problem handling redirects on directory resources via DAV
411        methods.</p>
412
413    
414
415   <h3><a name="suppress-error-charset" id="suppress-error-charset">suppress-error-charset</a></h3>
416       
417
418    <p><em>Available in versions after 2.0.54</em></p>
419
420    <p>When Apache issues a redirect in response to a client request,
421    the response includes some actual text to be displayed in case
422    the client can't (or doesn't) automatically follow the redirection.
423    Apache ordinarily labels this text according to the character set
424    which it uses, which is ISO-8859-1.</p>
425
426    <p> However, if the redirection is to a page that uses a different
427    character set, some broken browser versions will try to use the
428    character set from the redirection text rather than the actual page.
429    This can result in Greek, for instance, being incorrectly rendered.</p>
430
431    <p>Setting this environment variable causes Apache to omit the character
432    set for the redirection text, and these broken browsers will then correctly
433    use that of the destination page.</p>
434
435    <div class="warning">
436      <h3>Security note</h3>
437
438      <p>Sending error pages without a specified character set may
439      allow a cross-site-scripting attack for existing browsers (MSIE)
440      which do not follow the HTTP/1.1 specification and attempt to
441      "guess" the character set from the content.  Such browsers can
442      be easily fooled into using the UTF-7 character set, and UTF-7
443      content from input data (such as the request-URI) will not be
444      escaped by the usual escaping mechanisms designed to prevent
445      cross-site-scripting attacks.</p>
446    </div>

471    <h3><a name="fixheader" id="fixheader">Passing broken headers to CGI scripts</a></h3>
472      
473
474      <p>Starting with version 2.4, Apache is more strict about how HTTP
475      headers are converted to environment variables in <code class="module"><a href="./mod/mod_cgi.html">mod_cgi
476      </a></code> and other modules:  Previously any invalid characters
477      in header names were simply translated to underscores.  This allowed
478      for some potential cross-site-scripting attacks via header injection.</p>
479
480      <p>If you have to support a client which sends broken headers and
481      which can't be fixed, a simple workaround involving <code class="module"><a href="./mod/mod_setenvif.html">mod_setenvif
482      </a></code> and <code class="module"><a href="./mod/mod_headers.html">mod_headers</a></code> allows you to still accept
483      these headers:</p>
484
485<pre class="prettyprint lang-config">#
486# The following works around a client sending a broken Accept_Encoding
487# header.
488#
489SetEnvIfNoCase ^Accept.Encoding$ ^(.*)$ fix_accept_encoding=$1
490RequestHeader set Accept-Encoding %{fix_accept_encoding}e env=fix_accept_encoding</pre>
491

495    <h3><a name="misbehaving" id="misbehaving">Changing protocol behavior with misbehaving clients</a></h3>
496        
497
498        <p>Earlier versions recommended that the following lines be included in
499        httpd.conf to deal with known client problems.  Since the affected clients
500        are no longer seen in the wild, this configuration is likely no-longer
501        necessary.</p>
502<pre class="prettyprint lang-config">#
503# The following directives modify normal HTTP response behavior.
504# The first directive disables keepalive for Netscape 2.x and browsers that
505# spoof it. There are known problems with these browser implementations.
506# The second directive is for Microsoft Internet Explorer 4.0b2
507# which has a broken HTTP/1.1 implementation and does not properly
508# support keepalive when it is used on 301 or 302 (redirect) responses.
509#
510BrowserMatch "Mozilla/2" nokeepalive
511BrowserMatch "MSIE 4\.0b2;" nokeepalive downgrade-1.0 force-response-1.0
512
513#
514# The following directive disables HTTP/1.1 responses to browsers which
515# are in violation of the HTTP/1.0 spec by not being able to understand a
516# basic 1.1 response.
517#
518BrowserMatch "RealPlayer 4\.0" force-response-1.0
519BrowserMatch "Java/1\.0" force-response-1.0
520BrowserMatch "JDK/1\.0" force-response-1.0</pre>

524    <h3><a name="no-img-log" id="no-img-log">Do not log requests for images in the access log</a></h3>
525        
526
527        <p>This example keeps requests for images from appearing in the
528        access log. It can be easily modified to prevent logging of
529        particular directories, or to prevent logging of requests
530        coming from particular hosts.</p>
531
532        <pre class="prettyprint lang-config">SetEnvIf Request_URI \.gif image-request
533SetEnvIf Request_URI \.jpg image-request
534SetEnvIf Request_URI \.png image-request
535CustomLog "logs/access_log" common env=!image-request</pre>

539    <h3><a name="image-theft" id="image-theft">Prevent "Image Theft"</a></h3>
540        
541
542        <p>This example shows how to keep people not on your server
543        from using images on your server as inline-images on their
544        pages. This is not a recommended configuration, but it can work
545        in limited circumstances. We assume that all your images are in
546        a directory called <code>/web/images</code>.</p>
547
548        <pre class="prettyprint lang-config">SetEnvIf Referer "^http://www\.example\.com/" local_referal
549# Allow browsers that do not send Referer info
550SetEnvIf Referer "^$" local_referal
551&lt;Directory "/web/images"&gt;
552    Require env local_referal
553&lt;/Directory&gt;</pre>
554
555
556        <p>For more information about this technique, see the
557        "<a href="https://www.serverwatch.com/guides/keeping-your-images-from-adorning-other-sites/">Keeping Your Images from Adorning Other Sites</a>"
558        tutorial on ServerWatch.</p>
