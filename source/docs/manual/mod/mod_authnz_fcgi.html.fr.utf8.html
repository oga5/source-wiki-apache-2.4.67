39<h3>Sommaire</h3>
40
41    <p>Ce module permet aux applications d'autorisation FastCGI
42    d'authentifier les utilisateurs et de contrôler leur accès aux
43    ressources. Il supporte les systèmes d'autorisation FastCGI
44    génériques qui participent en une seule phase à l'authentification
45    et à l'autorisation, ainsi que les processus d'authentification et
46    d'autorisation spécifiques à Apache httpd qui interviennent en une
47    ou plusieurs phases.</p>
48
49    <p>Les processus d'autorisation FastCGI peuvent authentifier un
50    utilisateur via son identificateur et son mot de passe comme dans le
51    processus d'authentification basique, ou via un mécanisme
52    arbitraire.</p>

75<h2><a name="invocations" id="invocations">Modes d'invocation</a></h2>
76
77    <p>Les modes d'invocation des processus d'autorisation FastCGI que
78    ce module supporte se distinguent par deux caractéristiques : le
79    <em>type</em> et le <em>mécanisme</em> d'authentification.</p>
80
81    <p>Le <em>Type</em> est simplement <code>authn</code> pour
82    l'authentification, <code>authz</code> pour l'autorisation et
83    <code>authnz</code> l'authentification et l'autorisation.</p>
84
85    <p>Le <em>mécanisme</em> d'authentification fait référence aux
86    mécanismes d'authentification et aux phases de traitement de la
87    configuration de Apache httpd, et peut être
88    <code>AuthBasicProvider</code>, <code>Require</code>, ou
89    <code>check_user_id</code>. Les deux premiers mécanismes
90    correspondent aux directives utilisées pour participer aux phases de
91    traitement appropriées.</p>
92
93    <p>Description de chaque mode:</p>
94
95    <dl>
96      <dt><em>Type</em> <code>authn</code>, <em>mechanism</em>
97      <code>AuthBasicProvider</code></dt>
98
99      <dd>Dans ce mode, la variable <code>FCGI_ROLE</code> est définie à
100      <code>AUTHORIZER</code>, et la variable
101      <code>FCGI_APACHE_ROLE</code> à <code>AUTHENTICATOR</code>.
102      L'application doit être spécifiée en tant que fournisseur de type
103      <em>authn</em> via la directive <code class="directive"><a href="#authnzfcgidefineprovider">AuthnzFcgiDefineProvider</a></code>, et
104      activée via la directive <code class="directive"><a href="../mod/mod_auth_basic.html#authbasicprovider">AuthBasicProvider</a></code>. Lorsqu'elle
105      est invoquée, l'application est censée authentifier le client à
106      l'aide de l'identifiant et du mot de passe de l'utilisateur.
107      Exemple d'application :
108
109<pre class="prettyprint lang-perl">#!/usr/bin/perl
110use FCGI;
111my $request = FCGI::Request();
112while ($request-&gt;Accept() &gt;= 0) {
113    die if $ENV{'FCGI_APACHE_ROLE'} ne "AUTHENTICATOR";
114    die if $ENV{'FCGI_ROLE'}        ne "AUTHORIZER";
115    die if !$ENV{'REMOTE_PASSWD'};
116    die if !$ENV{'REMOTE_USER'};
117
118    print STDERR "This text is written to the web server error log.\n";
119
120    if ( ($ENV{'REMOTE_USER' } eq "foo" || $ENV{'REMOTE_USER'} eq "foo1") &amp;&amp;
121        $ENV{'REMOTE_PASSWD'} eq "bar" ) {
122        print "Status: 200\n";
123        print "Variable-AUTHN_1: authn_01\n";
124        print "Variable-AUTHN_2: authn_02\n";
125        print "\n";
126    }
127    else {
128        print "Status: 401\n\n";
129    }
130}</pre>
131
132
133      Exemple de configuration httpd :
134<pre class="prettyprint lang-config">AuthnzFcgiDefineProvider authn FooAuthn fcgi://localhost:10102/
135&lt;Location "/protected/"&gt;
136  AuthType Basic
137  AuthName "Restricted"
138  AuthBasicProvider FooAuthn
139  Require ...
140&lt;/Location&gt;</pre>
141
142      </dd>
143
144      <dt><em>Type</em> <code>authz</code>, <em>mechanism</em>
145      <code>Require</code></dt>
146      <dd>Dans ce mode, la variable <code>FCGI_ROLE</code> est définie à
147      <code>AUTHORIZER</code> et <code>FCGI_APACHE_ROLE</code> à
148      <code>AUTHORIZER</code>. L'application doit être spécifiée en tant
149      que fournisseur de type <em>authz</em> via la directive <code class="directive"><a href="#authnzfcgidefineprovider">AuthnzFcgiDefineProvider</a></code>.
150      Lorsqu'elle est invoquée, l'application est censée contrôler les
151      accès du client à l'aide de l'identifiant utilisateur et d'autres
152      données contenues dans la requête. Exemple d'application :
153<pre class="prettyprint lang-perl">#!/usr/bin/perl
154use FCGI;
155my $request = FCGI::Request();
156while ($request-&gt;Accept() &gt;= 0) {
157    die if $ENV{'FCGI_APACHE_ROLE'} ne "AUTHORIZER";
158    die if $ENV{'FCGI_ROLE'}        ne "AUTHORIZER";
159    die if $ENV{'REMOTE_PASSWD'};
160
161    print STDERR "This text is written to the web server error log.\n";
162
163    if ($ENV{'REMOTE_USER'} eq "foo1") {
164        print "Status: 200\n";
165        print "Variable-AUTHZ_1: authz_01\n";
166        print "Variable-AUTHZ_2: authz_02\n";
167        print "\n";
168    }
169    else {
170        print "Status: 403\n\n";
171    }
172}</pre>
173
174
175      Exemple de configuration httpd :
176<pre class="prettyprint lang-config">AuthnzFcgiDefineProvider authz FooAuthz fcgi://localhost:10103/
177&lt;Location "/protected/"&gt;
178  AuthType ...
179  AuthName ...
180  AuthBasicProvider ...
181  Require FooAuthz
182&lt;/Location&gt;</pre>
183
184      </dd>
185
186      <dt><em>Type</em> <code>authnz</code>, <em>mechanism</em>
187      <code>AuthBasicProvider</code> <em>+</em> <code>Require</code></dt>
188
189      <dd>Dans ce mode qui supporte le protocole d'autorisation web
190      server-agnostic FastCGI, la variable <code>FCGI_ROLE</code> est
191      définie à <code>AUTHORIZER</code> et <code>FCGI_APACHE_ROLE</code>
192      n'est pas définie. L'application doit être spécifiée en tant que
193      fournisseur de type <em>authnz</em> via la directive <code class="directive"><a href="#authnzfcgidefineprovider">AuthnzFcgiDefineProvider</a></code>.
194      L'application est censée assurer l'authentification et
195      l'autorisation au cours d'une même invocation à l'aide de
196      l'identifiant et du mot de passe de l'utilisateur et d'autres
197      données contenues dans la requête. L'invocation de l'application
198      intervient au cours de la phase d'authentification de l'API Apache
199      httpd. Si l'application renvoie le code 200, et si le même
200      fournisseur est invoqué au cours de la phase d'autorisation (via
201      une directive <code class="directive">Require</code>), mod_authnz_fcgi
202      renverra un code de type success pour la phase d'autorisation sans
203      invoquer l'application. Exemple d'application :
204<pre class="prettyprint lang-perl">#!/usr/bin/perl
205use FCGI;
206my $request = FCGI::Request();
207while ($request-&gt;Accept() &gt;= 0) {
208    die if $ENV{'FCGI_APACHE_ROLE'};
209    die if $ENV{'FCGI_ROLE'} ne "AUTHORIZER";
210    die if !$ENV{'REMOTE_PASSWD'};
211    die if !$ENV{'REMOTE_USER'};
212
213    print STDERR "This text is written to the web server error log.\n";
214
215    if ( ($ENV{'REMOTE_USER' } eq "foo" || $ENV{'REMOTE_USER'} eq "foo1") &amp;&amp;
216        $ENV{'REMOTE_PASSWD'} eq "bar" &amp;&amp;
217        $ENV{'REQUEST_URI'} =~ m%/bar/.*%) {
218        print "Status: 200\n";
219        print "Variable-AUTHNZ_1: authnz_01\n";
220        print "Variable-AUTHNZ_2: authnz_02\n";
221        print "\n";
222    }
223    else {
224        print "Status: 401\n\n";
225    }
226}</pre>
227
228
229      Exemple de configuration httpd :
230<pre class="prettyprint lang-config">AuthnzFcgiDefineProvider authnz FooAuthnz fcgi://localhost:10103/
231&lt;Location "/protected/"&gt;
232  AuthType Basic
233  AuthName "Restricted"
234  AuthBasicProvider FooAuthnz
235  Require FooAuthnz
236&lt;/Location&gt;</pre>
237
238      </dd>
239
240      <dt><em>Type</em> <code>authn</code>, <em>mechanism</em>
241      <code>check_user_id</code></dt>
242
243      <dd>Dans ce mode, la variable <code>FCGI_ROLE</code> est définie à
244      <code>AUTHORIZER</code> et <code>FCGI_APACHE_ROLE</code> à
245      <code>AUTHENTICATOR</code>. L'application doit être spécifiée en
246      tant que fournisseur de type <em>authn</em> via une directive
247      <code class="directive"><a href="#authnzfcgidefineprovider">AuthnzFcgiDefineProvider</a></code>. La
248      directive <code class="directive"><a href="#authnzfcgicheckauthnprovider">AuthnzFcgiCheckAuthnProvider</a></code>
249      permet de l'invoquer. Exemple d'application :
250<pre class="prettyprint lang-perl">#!/usr/bin/perl
251use FCGI;
252my $request = FCGI::Request();
253while ($request-&gt;Accept() &gt;= 0) {
254    die if $ENV{'FCGI_APACHE_ROLE'} ne "AUTHENTICATOR";
255    die if $ENV{'FCGI_ROLE'} ne "AUTHORIZER";
256
257    # This authorizer assumes that the RequireBasicAuth option of 
258    # AuthnzFcgiCheckAuthnProvider is On:
259    die if !$ENV{'REMOTE_PASSWD'};
260    die if !$ENV{'REMOTE_USER'};
261
262    print STDERR "This text is written to the web server error log.\n";
263
264    if ( ($ENV{'REMOTE_USER' } eq "foo" || $ENV{'REMOTE_USER'} eq "foo1") &amp;&amp;
265        $ENV{'REMOTE_PASSWD'} eq "bar" ) {
266        print "Status: 200\n";
267        print "Variable-AUTHNZ_1: authnz_01\n";
268        print "Variable-AUTHNZ_2: authnz_02\n";
269        print "\n";
270    }
271    else {
272        print "Status: 401\n\n";
273        # If a response body is written here, it will be returned to
274        # the client.
275    }
276}</pre>
277
278
279      Exemple de configuration httpd :
280<pre class="prettyprint lang-config">AuthnzFcgiDefineProvider authn FooAuthn fcgi://localhost:10103/
281&lt;Location "/protected/"&gt;
282  AuthType ...
283  AuthName ...
284  AuthnzFcgiCheckAuthnProvider FooAuthn \
285                               Authoritative On \
286                               RequireBasicAuth Off \
287                               UserExpr "%{reqenv:REMOTE_USER}"
288  Require ...
289&lt;/Location&gt;</pre>
290
291      </dd>
292
293    </dl>

297<h2><a name="examples" id="examples">Exemples supplémentaires</a></h2>
298
299  <ol>
300    <li>Si votre application supporte séparément les rôles
301    d'authentification et d'autorisation (<code>AUTHENTICATOR</code> et
302    <code>AUTHORIZER</code>), vous pouvez définir des fournisseurs
303    séparés comme suit, même s'ils correspondent à la même application :
304
305<pre class="prettyprint lang-config">AuthnzFcgiDefineProvider authn  FooAuthn  fcgi://localhost:10102/
306AuthnzFcgiDefineProvider authz  FooAuthz  fcgi://localhost:10102/</pre>
307
308
309    Spécifie le fournisseur authn via la directive 
310    <code class="directive"><a href="../mod/mod_auth_basic.html#authbasicprovider">AuthBasicProvider</a></code>
311    et le fournisseur authz via la directive
312    <code class="directive"><a href="../mod/mod_authz_core.html#require">Require</a></code>:
313
314<pre class="prettyprint lang-config">AuthType Basic
315AuthName "Restricted"
316AuthBasicProvider FooAuthn
317Require FooAuthz</pre>
318
319    </li>
320
321    <li>Si votre application supporte le rôle générique
322    <code>AUTHORIZER</code> (authentification et autorisation en une
323    seule invocation), vous pouvez définir un fournisseur unique comme
324    suit :
325
326<pre class="prettyprint lang-config">AuthnzFcgiDefineProvider authnz FooAuthnz fcgi://localhost:10103/</pre>
327
328
329    Spécifie le fournisseur authnz via les directives
330    <code class="directive">AuthBasicProvider</code> et
331    <code class="directive">Require</code> :
332
333<pre class="prettyprint lang-config">AuthType Basic
334AuthName "Restricted"
335AuthBasicProvider FooAuthnz
336Require FooAuthnz</pre>
337
338    </li>
339</ol>

342<h2><a name="limitations" id="limitations">Limitations</a></h2>
343
344    <p>Les fonctionnalités suivantes ne sont pas encore implémentées :</p>
345
346    <dl>
347      <dt>Vérificateur d'accès d'Apache httpd</dt>
348      <dd>La phase <em>access check</em> de l'API Apache httpd est
349      distincte des phases d'authentification et d'autorisation.
350      Certaines autres implémentations de FastCGI supportent cette phase
351      et lorsque c'est le cas, la variable <code>FCGI_APACHE_ROLE</code>
352      est définie à <code>ACCESS_CHECKER</code>.</dd>
353
354      <dt>Redirections (pipes) ou sockets locaux (Unix)</dt>
355      <dd>Seuls les sockets TCP sont actuellement supportés.</dd>
356
357      <dt>Support de mod_authn_socache</dt>
358      <dd>Le support de l'interaction avec mod_authn_socache pour les
359      applications qui interviennent dans le processus
360      d'authentification d'Apache httpd serait souhaitable.</dd>
361
362      <dt>Support de l'authentification de type digest à l'aide de AuthDigestProvider</dt>
363      <dd>Cette limitation ne sera probablement jamais franchie car il
364      n'existe aucun flux de données d'autorisation capable de lire dans
365      un condensé de type hash.</dd>
366
367      <dt>Gestion des processus applicatifs</dt>
368      <dd>Cette fonctionnalité restera probablement hors de portée de ce
369      module. Il faudra donc gérer les processus applicatifs d'une autre
370      manière ; par exemple, <code class="program"><a href="../programs/fcgistarter.html">fcgistarter</a></code> permet de
371      les démarrer.</dd>
372
373      <dt>AP_AUTH_INTERNAL_PER_URI</dt>
374      <dd>Tous les fournisseurs sont actuellement enregistrés en tant
375      que AP_AUTH_INTERNAL_PER_CONF, ce qui signifie que les
376      vérifications ne sont pas effectuées pour les
377      sous-requêtes internes avec la même configuration de contrôle
378      d'accès que la requête initiale.</dd>
379
380      <dt>Conversion du jeu de caractères des données de protocole</dt>
381      <dd>Si mod_authnz_fcgi s'exécute dans un environnement de
382      compilation EBCDIC, toutes les données de protocole FastCGI sont
383      écrites en EBCDIC et doivent être disponibles en EBCDIC.</dd>
384
385      <dt>Plusieurs requêtes pour une connexion</dt>
386      <dd>Actuellement, la connexion au fournisseur d'autorisation
387      FastCGI est fermée après chaque phase de traitement. Par exemple,
388      si le fournisseur d'autorisation gère séparément les phases
389      <em>authn</em> et <em>authz</em>, deux connexions seront
390      nécessaires.</dd>
391
392      <dt>Redirection de certains URIs</dt>
393      <dd>Les URIs en provenance des clients ne peuvent pas être
394      redirigés selon une table de redirection, comme avec la directive
395      <code class="directive">ProxyPass</code> utilisée avec les répondeurs
396      FastCGI.</dd>
397
398    </dl>

402<h2><a name="logging" id="logging">Journalisation</a></h2>
403
404    <ol>
405        <li>Les erreurs de traitement sont journalisées à un niveau
406	<code>error</code> ou supérieur.</li>
407        <li>Les messages envoyés par l'application sont journalisés au
408	niveau <code>warn</code>.</li>
409        <li>Les messages de deboguage à caractère général sont
410	journalisés au niveau <code>debug</code>.</li>
411        <li>Les variables d'environnement transmises à l'application
412	sont journalisées au niveau <code>trace2</code>. La valeur de la
413	variable <code>REMOTE_PASSWD</code> sera occultée, mais
414	<strong>toute autre donnée sensible sera visible dans le
415	journal</strong>.</li>
416        <li>Toutes les entrées/sorties entre le module et l'application
417	FastCGI, y compris les variables d'environnement, seront
418	journalisées au format imprimable et hexadécimal au niveau
419	<code>trace5</code>. <strong>Toutes les données sensibles seront
420	visibles dans le journal.</strong></li>
421    </ol>
422
423    <p>La directive <code class="directive"><a href="../mod/core.html#loglevel">LogLevel</a></code> permet
424    de configurer un niveau de journalisation spécifique à
425    mod_authnz_fcgi. Par exemple :</p>
426
427<pre class="prettyprint lang-config">LogLevel info authnz_fcgi:trace8</pre>

436<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>AuthnzFcgiCheckAuthnProvider <em>provider-name</em>|<code>None</code>
437<em>option</em> ...</code></td></tr>
438<tr><th><a href="directive-dict.html#Default">Défaut:</a></th><td><code>none</code></td></tr>
439<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>répertoire</td></tr>
440<tr><th><a href="directive-dict.html#Status">Statut:</a></th><td>Extension</td></tr>
441<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_authnz_fcgi</td></tr>
442</table>
443    <p>Cette directive permet de confier à une application FastCGI la
444    gestion d'une phase spécifique du processus d'authentification ou
445    d'autorisation.</p>
446
447    <p>Certaines fonctionnalités des fournisseurs d'autorisation FastCGI
448    nécessitent cette directive en lieu et place de
449    <code class="directive">AuthBasicProvider</code> pour pouvoir être activées :</p>
450
451    <ul>
452      <li>L'authentification de type autre que basique ; en général,
453      détermination de l'identifiant utilisateur et renvoi de sa valeur
454      depuis le fournisseur d'autorisation ; voir l'option
455      <code>UserExpr</code> ci-dessous</li>
456      <li>Sélection d'un code de réponse personnalisé ; en cas de
457      code de réponse autre que 200 en provenance du fournisseur
458      d'autorisation, c'est ce code qui sera utilisé comme code d'état
459      de la réponse</li>
460      <li>Définition du corps d'une réponse autre que 200 ; si le
461      fournisseur d'autorisation renvoie un corps de réponse avec un
462      code autre que 200, c'est ce corps de réponse qui sera renvoyé au
463      client ; la longueur du texte est limitée à 8192 octets</li>
464    </ul>
465
466    <dl>
467      <dt><em>provider-name</em></dt>
468      <dd>C'est le nom du fournisseur défini au préalable via la
469      directive <code class="directive">AuthnzFcgiDefineProvider</code>.</dd>
470
471      <dt><code>None</code></dt>
472      <dd>Spécifiez <code>None</code> pour désactiver un fournisseur
473      activé avec cette même directive dans une autre portée, par
474      exemple dans un répertoire parent.</dd>
475
476      <dt><em>option</em></dt>
477      <dd>Les options suivantes sont supportées :
478      
479      <dl>
480         <dt>Authoritative On|Off (par défaut On)</dt>
481         <dd>Cette option permet de définir si l'appel à d'autres
482	 modules est autorisé lorsqu'un fournisseur d'autorisation FastCGI a
483	 été configuré et si la requête échoue.</dd>
484
485         <dt>DefaultUser <em>id utilisateur</em></dt>
486         <dd>Lorsque le fournisseur d'autorisation donne son accord, et
487	 si <code>UserExpr</code> est défini et correspond à une chaîne
488	 vide, (par exemple, si le fournisseur d'autorisation ne renvoie
489	 aucune variable), c'est cette valeur qui sera utilisée comme id
490	 utilisateur par défaut. Cela se produit souvent lorsqu'on se trouve dans
491	 un contexte d'invité, ou d'utilisateur non authentifié ;
492	 les utilisateurs et invités se voient alors attribué un id
493	 utilisateur spécifique qui permettra de se connecter et
494	 d'accéder à certaines ressources.</dd>
495
496         <dt>RequireBasicAuth On|Off (par défaut Off)</dt>
497         <dd>Cette option permet de définir si l'authentification
498	 basique est requise avant de transmettre la requête au
499	 fournisseur d'autorisation. Dans l'affirmative, le fournisseur
500	 d'autorisation ne sera invoqué qu'en présence d'un id
501	 utilisateur et d'un mot de passe ; si ces deux éléments ne sont
502	 pas présents, un code d'erreur 401 sera renvoyé</dd>
503
504         <dt>UserExpr <em>expr</em> (pas de valeur par défaut)</dt>
505         <dd>Lorsque le client ne fournit pas l'authentification basique
506	 et si le fournisseur d'autorisation détermine l'id utilisateur,
507	 cette expression, évaluée après l'appel au fournisseur
508	 d'autorisation, permet de déterminer l'id utilisateur. Cette
509	 expression se conforme à la <a href="../expr.html">syntaxe
510	 ap_expr</a> et doit correspondre à une chaîne de caractères.
511	 Une utilisation courante consiste à référencer la définition
512	 d'une <code>Variable-<em>XXX</em></code> renvoyée par le
513	 fournisseur d'autorisation via une option du style
514	 <code>UserExpr "%{reqenv:<em>XXX</em>}"</code>. Si cette option
515	 est spécifiée, et si l'id utilisateur ne peut pas être définie
516	 via l'expression après une authentification réussie, la requête
517	 sera rejetée avec un code d'erreur 500.</dd>
518
519       </dl>
520      </dd>
521     </dl>
522
523</div>

525<div class="directive-section"><h2><a name="authnzfcgidefineprovider" id="authnzfcgidefineprovider">Directive</a> <a name="AuthnzFcgiDefineProvider" id="AuthnzFcgiDefineProvider">AuthnzFcgiDefineProvider</a></h2>
526<table class="directive">
527<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Définit une application FastCGI en tant que fournisseur
528d'authentification et/ou autorisation</td></tr>
529<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>AuthnzFcgiDefineProvider <em>type</em> <em>provider-name</em>
530<em>backend-address</em></code></td></tr>
531<tr><th><a href="directive-dict.html#Default">Défaut:</a></th><td><code>none</code></td></tr>
532<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>configuration globale</td></tr>
533<tr><th><a href="directive-dict.html#Status">Statut:</a></th><td>Extension</td></tr>
534<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_authnz_fcgi</td></tr>
535</table>
536    <p>Cette directive permet de définir une application FastCGI en tant
537    que fournisseur pour une phase particulière d'authentification ou
538    d'autorisation.</p>
539
540    <dl>
541      <dt><em>type</em></dt>
542      <dd>Les valeurs de ce paramètre sont <em>authn</em> pour
543      l'authentification, <em>authz</em> pour l'autorisation, ou
544      <em>authnz</em> pour un fournisseur d'autorisation générique
545      FastCGI qui effectue les deux vérifications.</dd>
546
547      <dt><em>provider-name</em></dt>
548      <dd>Ce paramètre permet d'associer un nom au fournisseur ; ce nom
549      pourra être utilisé dans des directives comme <code class="directive"><a href="../mod/mod_auth_basic.html#authbasicprovider">AuthBasicProvider</a></code> et
550      <code class="directive"><a href="../mod/mod_authz_core.html#require">Require</a></code>.</dd>
551
552      <dt><em>backend-address</em></dt>
553      <dd>Ce paramètre permet de spécifier l'adresse de l'application
554      sous la forme <em>fcgi://hostname:port/</em>. Le ou les processus
555      de l'application doivent être gérés indépendamment comme avec
556      <code class="program"><a href="../programs/fcgistarter.html">fcgistarter</a></code>.</dd>
557    </dl>
