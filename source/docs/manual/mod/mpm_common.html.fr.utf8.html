71<div class="directive-section"><h2><a name="coredumpdirectory" id="coredumpdirectory">Directive</a> <a name="CoreDumpDirectory" id="CoreDumpDirectory">CoreDumpDirectory</a></h2>
72<table class="directive">
73<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Le répertoire dans lequel le serveur HTTP Apache va tenter de se
74positionner avant d'effectuer un vidage mémoire</td></tr>
75<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>CoreDumpDirectory <var>répertoire</var></code></td></tr>
76<tr><th><a href="directive-dict.html#Default">Défaut:</a></th><td><code>Voir ci-dessous pour le répertoire par défaut</code></td></tr>
77<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>configuration globale</td></tr>
78<tr><th><a href="directive-dict.html#Status">Statut:</a></th><td>MPM</td></tr>
79<tr><th><a href="directive-dict.html#Module">Module:</a></th><td><code class="module"><a href="../mod/event.html">event</a></code>, <code class="module"><a href="../mod/worker.html">worker</a></code>, <code class="module"><a href="../mod/prefork.html">prefork</a></code></td></tr>
80</table>
81    <p>Cette directive permet de définir le répertoire dans lequel
82    Apache httpd va tenter de se positionner avant d'effectuer un vidage
83    mémoire sur disque.
84    Si votre système d'exploitation est configuré pour créer des
85    fichiers de vidage mémoire dans le répertoire de travail des
86    processus qui se sont crashés,
87    <code class="directive">CoreDumpDirectory</code> est nécessaire pour
88    définir un répertoire de travail autre que le répertoire par défaut
89    <code class="directive"><a href="../mod/core.html#serverroot">ServerRoot</a></code>, ce répertoire de
90    travail ne devant pas être accessible en écriture par l'utilisateur sous
91    lequel le serveur s'exécute.</p>
92
93    <p>Si vous avez besoin d'un vidage mémoire pour le débogage, vous
94    pouvez utiliser cette directive pour le placer à un endroit
95    différent. Cette directive n'a aucun effet si votre système
96    d'exploitation n'est pas configuré pour créer des
97    fichiers de vidage mémoire dans le répertoire de travail des
98    processus qui se sont crashés.</p>
99
100    <div class="warning">
101      <h3>Note de sécurité pour les systèmes de type Linux</h3>
102
103      <p>Utiliser cette directive sous Linux peut permettre aux autres processus
104      du système s'exécutant avec les même privilèges (comme les scripts CGI) de
105      se rattacher aux processus httpd enfants via l'appel système
106      <code>ptrace</code>. La protection contre certaines attaques engageant la
107      sécurité peut s'en trouver affectée. Il est par conséquent déconseillé
108      d'utiliser cette directive sur les systèmes en production.</p>
109    </div>
110
111    <div class="note"><h3>Vidages mémoire sous Linux</h3>
112      <p>Si Apache httpd est démarré sous l'utilisateur root puis bascule vers
113      un autre utilisateur, le noyau Linux <em>désactive</em> les
114      vidages mémoire, même si le répertoire est accessible en écriture au
115      processus. Apache httpd (versions 2.0.46 et supérieures) réactive les
116      vidages mémoire sous Linux 2.4 et au delà, mais seulement si vous
117      définissez une directive <code class="directive">CoreDumpDirectory</code>.</p>
118    </div>
119
120    <div class="note">
121    <h3>Vidages mémoire sous BSD</h3>
122    <p>Pour activer le vidage mémoire des exécutables suid sur les
123    systèmes de style BSD (comme FreeBSD), définissez
124    <code>kern.sugid_coredump</code> à 1.
125    </p>
126    </div>
127
128    <div class="note"><h3>Signaux spécifiques</h3>
129      <p><code class="directive">CoreDumpDirectory</code> n'est traité qu'à la
130      reception d'un certain nombre de signaux , SIGFPE, SIGILL, SIGABORT,
131      SIGSEGV, et SIGBUS.</p>
132      <p>
133      Sur certains systèmes d'exploitation, SIGQUIT provoque aussi un
134      vidage mémoire, mais n'est pas traité par les directives
135      <code class="directive">CoreDumpDirectory</code> ou
136      <code class="directive">EnableExceptionHook</code>, si bien que la
137      définition du répertoire d'enregistrement du vidage mémoire est
138      entièrement dévolue au système d'exploitation.</p>
139    </div>

190<div class="directive-section"><h2><a name="listen" id="listen">Directive</a> <a name="Listen" id="Listen">Listen</a></h2>
191<table class="directive">
192<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Les adresses IP et ports sur lesquels le serveur écoute</td></tr>
193<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>Listen [<var>adresse IP</var>:]<var>numéro port</var>
194[<var>protocole</var>]</code></td></tr>
195<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>configuration globale</td></tr>
196<tr><th><a href="directive-dict.html#Status">Statut:</a></th><td>MPM</td></tr>
197<tr><th><a href="directive-dict.html#Module">Module:</a></th><td><code class="module"><a href="../mod/event.html">event</a></code>, <code class="module"><a href="../mod/worker.html">worker</a></code>, <code class="module"><a href="../mod/prefork.html">prefork</a></code>, <code class="module"><a href="../mod/mpm_winnt.html">mpm_winnt</a></code>, <code class="module"><a href="../mod/mpm_netware.html">mpm_netware</a></code>, <code class="module"><a href="../mod/mpmt_os2.html">mpmt_os2</a></code></td></tr>
198<tr><th><a href="directive-dict.html#Compatibility">Compatibilité:</a></th><td>L'argument <var>protocole</var> est supporté depuis la version
1992.1.5</td></tr>
200</table>
201    <p>La directive <code class="directive">Listen</code> permet de signifier à
202    Apache httpd de ne se mettre à l'écoute que sur les adresses IP et ports spécifiés ; par
203    défaut, le serveur répond aux requêtes en provenance de toutes les
204    interfaces réseau. La directive <code class="directive">Listen</code> est
205    dorénavant requise, et si elle est absente du fichier de
206    configuration, le serveur refusera de démarrer. Ceci constitue un
207    changement par rapport aux versions précédentes d'Apache httpd.</p>
208
209    <p>La directive <code class="directive">Listen</code> signifie au serveur de
210    n'accepter les requêtes entrantes que vers le port ou le couple
211    adresse-port spécifié. Si seulement un port est spécifié, le serveur
212    se met à l'écoute sur ce port sur toutes les interfaces réseau. Si une adresse IP
213    et un port sont spécifiés, le serveur va se mettre à l'écoute sur ce port sur
214    l'interface réseau correspondant à l'adresse IP.</p>
215
216    <p>On peut utiliser autant de directives
217    <code class="directive">Listen</code> que nécessaire pour spécifier
218    plusieurs adresses et/ou ports à écouter. Le serveur répondra aux
219    requêtes vers tous les adresses et ports spécifiés.</p>
220
221    <p>Par exemple, pour que le serveur accepte les connexions sur les
222    ports 80 et 8000, utilisez :</p>
223
224    <pre class="prettyprint lang-config">Listen 80
225Listen 8000</pre>
226
227
228    <p>Pour que le serveur accepte les connexions sur deux interfaces et
229    ports particuliers, spécifiez :</p>
230
231    <pre class="prettyprint lang-config">Listen 192.170.2.1:80
232Listen 192.170.2.5:8000</pre>
233
234
235    <p>Les adressee IPv6 doivent être entourées de crochets, comme dans
236    l'exemple suivant :</p>
237
238    <pre class="prettyprint lang-config">Listen [2001:db8::a00:20ff:fea7:ccea]:80</pre>
239
240
241    <p>L'argument optionnel <var>protocole</var> n'est pas nécessaire
242    dans la plupart des configurations. S'il est absent,
243    <code>https</code> est la valeur par défaut pour le port 443 et
244    <code>http</code> l'est pour tous les autres ports. L'argument
245    protocole sert à déterminer quel module doit traiter une requête, et
246    à appliquer des optimisations spécifiques à certains protocoles à
247    l'aide de la directive <code class="directive"><a href="../mod/core.html#acceptfilter">AcceptFilter</a></code>.</p>
248
249    <p>La spécification d'un protocole n'est nécessaire que si vous
250    utilisez des ports non standards. Par exemple, pour configurer un
251    site en <code>https</code> sur le port 8443 :</p>
252
253    <pre class="prettyprint lang-config">Listen 192.170.2.1:8443 https</pre>
254
255
256    <div class="note"><h3>Condition d'erreur</h3>
257      Plusieurs directives <code class="directive">Listen</code> pour les mêmes
258      adresse IP/port vont provoquer l'envoi d'un message d'erreur
259      <code>Address already in use</code>.
260    </div>

294    la valeur spécifiée pour l'argument backlog, mais s'en inspirent
295    seulement (et choisissent en général une valeur supérieure).</p>

317    <code class="directive">ListenCoresBucketsRatio</code> pour spécifier un
318    <var>ratio</var> entre le nombre de coeurs de CPU activés et le nombre de
319    segments d'écoute (listeners' buckets) souhaités ; le serveur HTTP Apache va
320    alors créer<code>num_cpu_cores / ratio</code> segments d'écoute, chacun
321    contenant son propre socket d'écoute <code class="directive"><a href="#listen">Listen</a></code> sur le ou les mêmes ports ; chaque
322    processus enfant sera associé à un seul segment d'écoute (avec une
323    distribution de type round-robin des segments à la création des processus
324    enfants).</p>

326    <div class="note"><h3>Définition du terme coeur de CPU activé ("online")</h3>
327    <p>Sous Linux et BSD, un coeur de CPU peut être activé ou désactivé si <a href="https://www.kernel.org/doc/Documentation/cpu-hotplug.txt">Hotplug</a>
328    a été configuré ; la directive
329    <code class="directive">ListenCoresBucketsRatio</code> doit donc tenir compte de ce
330    paramètre pour calculer le nombre de segments d'écoute à créer.</p>
331    </div>
332
333    <p>La directive <code class="directive">ListenCoresBucketsRatio</code> peut
334    améliorer le support de la montée en charge lorsque l'arrivée de
335    nouvelles connexions est/devient un goulot d'étranglement. Le test
336    de cette fonctionnalité avec des machines possédant un nombre de
337    coeurs de CPU important a permit de constater une amélioration des
338    performances significative et des temps de réponse plus courts.</p>
339
340    <p>Pour que cette fonctionnalité soit activée, le nombre de coeurs
341    de CPU doit être égal au moins au double du <var>ratio</var>
342    spécifié. Si vous spécifiez la valeur recommandée pour
343    <var>ratio</var>, à savoir <code>8</code>, le nombre minimum de
344    coeurs de processeurs disponibles sera alors de <code>16</code>. La valeur
345    optimale de <var>ratio</var> permettant d'obtenir des performances maximales
346    doit être calculée pour chaque système cible, en testant plusieurs valeurs
347    et en observant les résultats.</p>
348
349    <p>Cette directive influence le calcul des valeurs limites inférieures de
350    <code class="directive"><a href="#minsparethreads">MinSpareThreads</a></code> et <code class="directive"><a href="#maxsparethreads">MaxSpareThreads</a></code>. En effet, pour accepter les
351    connexions de manière optimale, le nombre de processus enfants doit être un
352    multiple du nombre de segments d'écoute.</p>

354    <div class="note">
355    <h3>Cas où plusieurs <code class="directive">Listen</code>ers ou serveurs HTTP
356    Apache partagent la même adresse IP et port</h3>
357    <p>La définition de l'option <code>SO_REUSEPORT</code> pour les sockets
358    d'écoute permet à plusieurs processus (partageant le même <code>EUID</code>,
359    par exemple <code>root</code>) de se rattacher à la même adresse IP et port,
360    sans obtenir l'erreur de rattachement que le système génère habituellement
361    lorsque ce cas se produit.</p>
362    <p>Cela signifie aussi que plusieurs instances d'Apache httpd configurées
363    avec le même <code>IP:port</code> et avec une valeur
364    <code class="directive">ListenCoresBucketsRatio</code> positive pourraient démarrer
365    sans erreur, et fonctionner ensuite avec une répartition uniforme des
366    connexions entrantes sur ces différentes instances (ce n'est PAS une
367    recommandation et ne constitue pas un usage approprié à tous les cas, mais
368    juste un avertissement sur le fait qu'un véritable problème de rattachement
369    multiple à un <code>IP:port</code> pourrait alors être occulté).</p>
370    <p>Au sein d'une même instance, Apache httpd vérifie la présence de
371    directives <code class="directive">Listen</code> multiples avec la même adresse IP
372    (ou nom d'hôte) et le même port, et refuse de démarrer si c'est le cas, ce
373    qui permet d'éviter la création de segments d'écoute dupliqués qui seraient
374    du coup inutiles et affecteraient les performances. Cependant, il ne peut
375    pas (et n'essaiera pas de le faire) intercepter tous les cas possibles de
376    recouvrement (comme un nom d'hôte correspondant à une adresse IP utilisée
377    quelque part ailleurs).</p>
378    </div>

869      <p>Il existe une limite de <code>ThreadLimit 20000</code> (ou
870      <code>ThreadLimit 100000</code> avec <code class="module"><a href="../mod/event.html">event</a></code>,
871      <code>ThreadLimit 15000</code> avec <code class="module"><a href="../mod/mpm_winnt.html">mpm_winnt</a></code>)
872      codée en dur dans le serveur. Ceci est censé éviter les effets
873      désastreux que pourrait provoquer une faute de frappe. Pour
874      dépasser cette limite, vous devez modifier la valeur de
875      MAX_THREAD_LIMIT dans le fichier source du mpm et recompiler le
876      serveur.</p>
877    </div>
878
879    <p>La valeur de la directive <code class="directive">ThreadLimit</code> doit
880    être supérieure ou égale à 1. Une valeur inférieure à 1 sera automatiquement
881    augmentée à 1 et un avertissement sera journalisé.</p>

885<div class="directive-section"><h2><a name="threadsperchild" id="threadsperchild">Directive</a> <a name="ThreadsPerChild" id="ThreadsPerChild">ThreadsPerChild</a></h2>
886<table class="directive">
887<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Nombre de threads créés par chaque processus
888enfant</td></tr>
889<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>ThreadsPerChild <var>nombre</var></code></td></tr>
890<tr><th><a href="directive-dict.html#Default">Défaut:</a></th><td><code>Voir ci-dessous pour plus de détails</code></td></tr>
891<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>configuration globale</td></tr>
892<tr><th><a href="directive-dict.html#Status">Statut:</a></th><td>MPM</td></tr>
893<tr><th><a href="directive-dict.html#Module">Module:</a></th><td><code class="module"><a href="../mod/event.html">event</a></code>, <code class="module"><a href="../mod/worker.html">worker</a></code>, <code class="module"><a href="../mod/mpm_winnt.html">mpm_winnt</a></code></td></tr>
894</table>
895    <p>Cette directive permet de définir le nombre de threads que va
896    créer chaque processus enfant. Un processus enfant crée ces threads
897    au démarrage et n'en crée plus d'autres par la suite. Si l'on
898    utilise un MPM comme <code class="module"><a href="../mod/mpm_winnt.html">mpm_winnt</a></code> qui ne lance qu'un
899    processus enfant, ce nombre doit être suffisamment grand pour
900    supporter la charge du serveur. Avec un MPM comme
901    <code class="module"><a href="../mod/worker.html">worker</a></code> qui lance plusieurs processus enfants, c'est
902    le nombre <em>total</em> de threads qui doit être suffisamment grand
903    pour supporter la charge du serveur.</p>
904
905    <p>La valeur par défaut de la directive
906    <code class="directive">ThreadsPerChild</code> est <code>64</code> avec
907    <code class="module"><a href="../mod/mpm_winnt.html">mpm_winnt</a></code>, et <code>25</code> avec les autres
908    MPMs.</p>
909
910    <p>La valeur de la directive <code class="directive">ThreadsPerChild</code> ne peut
911    pas dépasser la valeur de la directive <code class="directive"><a href="#threadlimit">ThreadLimit</a></code>. Si on spécifie une valeur
912    supérieure, elle sera automatiquement réduite au démarrage du serveur et un
913    avertissement sera enregistré dans le journal. La relation entre ces deux
914    directives est expliquée dans la <code class="directive"><a href="#documentation de la directive threadlimit">documentation de la directive ThreadLimit</a></code>.</p>
915
916    <p>La valeur de la directive <code class="directive">ThreadsPerChild</code> doit
917    être supérieure ou égale à 1. Une valeur inférieure à 1 sera automatiquement
918    augmentée à 1 et un avertissement sera journalisé.</p>
919
920</div>

922<div class="directive-section"><h2><a name="threadstacksize" id="threadstacksize">Directive</a> <a name="ThreadStackSize" id="ThreadStackSize">ThreadStackSize</a></h2>
923<table class="directive">
924<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>La taille en octets de la pile qu'utilisent les threads qui
925traitent les connexions clients</td></tr>
926<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>ThreadStackSize <var>taille</var></code></td></tr>
927<tr><th><a href="directive-dict.html#Default">Défaut:</a></th><td><code>65536 sous NetWare; varie en fonction des autres systèmes
928d'exploitation</code></td></tr>
929<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>configuration globale</td></tr>
930<tr><th><a href="directive-dict.html#Status">Statut:</a></th><td>MPM</td></tr>
931<tr><th><a href="directive-dict.html#Module">Module:</a></th><td><code class="module"><a href="../mod/event.html">event</a></code>, <code class="module"><a href="../mod/worker.html">worker</a></code>, <code class="module"><a href="../mod/mpm_winnt.html">mpm_winnt</a></code>, <code class="module"><a href="../mod/mpm_netware.html">mpm_netware</a></code>, <code class="module"><a href="../mod/mpmt_os2.html">mpmt_os2</a></code></td></tr>
932<tr><th><a href="directive-dict.html#Compatibility">Compatibilité:</a></th><td>Disponible dans les versions 2.1 et supérieures
933du serveur HTTP Apache</td></tr>
934</table>
935    <p>La directive <code class="directive">ThreadStackSize</code> permet de
936    définir la taille de la pile (pour les données propres) qu'utilisent
937    les threads qui traitent les connexions clients en faisant appel à
938    des modules. Dans la plupart des cas, la valeur par défaut de la
939    taille de la pile du système d'exploitation convient, mais il existe
940    certaines situations où il peut s'avérer nécessaire de l'ajuster
941    :</p>
942
943    <ul>
944      <li>Sur les plates-formes qui possèdent une valeur par défaut de
945      taille de la pile relativement petite (par exemple HP-UX), Apache
946      httpd peut se crasher si l'on utilise certains modules tiers qui
947      possèdent un quantité de données propres stockées relativement
948      importante. Il se peut que ces mêmes modules fonctionnent
949      correctement sur d'autres plate-formes où la valeur par défaut de
950      la taille de la pile est supérieure. Ce type de crash peut être
951      evité en définissant <code class="directive">ThreadStackSize</code> à une
952      valeur supérieure à la valeur par défaut du système
953      d'exploitation. Ce type d'ajustement n'est nécessaire que si le
954      fournisseur du module tiers en fait mention, ou si le diagnostic
955      d'un crash d'Apache httpd indique que la taille de la pile était trop
956      petite.</li>
957
958      <li>Sur les plates-formes où la taille par défaut de la pile des
959      threads est sensiblement supérieure à la taille nécessaire pour la
960      configuration du serveur web, il est possible de disposer d'un
961      plus grand nombre de threads par processus enfant si la directive
962      <code class="directive">ThreadStackSize</code> est définie à une valeur
963      inférieure à la valeur par défaut du système d'exploitation.
964      Cependant, ce
965      type d'ajustement ne doit être effectué que dans un environnement
966      de test permettant de qualifier le serveur web au maximum de ses
967      possibilités, car il peut arriver, dans de rares cas, que des
968      requêtes nécessitent une taille de pile supérieure pour pouvoir
969      être traitées. La taille minimale requise pour la pile dépend
970      fortement des modules utilisés, mais toute modification dans la
971      configuration du serveur web peut invalider la définition courante
972      de la directive <code class="directive">ThreadStackSize</code>.</li>
973
974      <li>Sous Linux, cette directive ne peut être utilisée que pour
975      augmenter la valeur par defaut de la taille de la pile, car
976      l'appel système sous-jacent utilise cette valeur comme taille de pile
977      <em>minimale</em>. C'est la limite logicielle (souvent élevée)
978      pour <code>ulimit -s</code> (8Mo si aucune limite) qui est
979      utilisée comme taille de pile par défaut.</li>
980    </ul>
981
982    <div class="note">Il est recommandé de ne pas réduire
983    <code class="directive">ThreadStackSize</code>, à moins qu'un grand nombre
984    de threads par processus enfant ne soit nécessaire. Sur certaines
985    plates-formes (y compris Linux), une valeur de 128000 est déjà trop
986    basse et provoque des crashes avec certains modules courants.</div>
