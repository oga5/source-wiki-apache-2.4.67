165<h2><a name="forwardreverse" id="forwardreverse">Mandataires directs et
166    mandataires/passerelles inverses</a></h2>
167      <p>Le serveur HTTP Apache peut être configuré dans les deux modes mandataire
168      <dfn>direct</dfn> et mandataire <dfn>inverse</dfn> (aussi nommé
169      mode <dfn>passerelle</dfn>).</p>
170
171      <p>Un <dfn>mandataire direct</dfn> standard est un serveur
172      intermédiaire qui s'intercale entre le client et le <em>serveur
173      demandé</em>. Pour obtenir un contenu hébergé par
174      le serveur demandé, le client envoie une requête au
175      mandataire en nommant le serveur demandé comme
176      cible. Le mandataire extrait alors le contenu depuis le
177      serveur demandé et le renvoie enfin au client. Le client doit être
178      configuré de manière appropriée pour pouvoir utiliser le mandataire
179      direct afin d'accéder à d'autres sites.</p>
180
181      <p>L'accès à Internet depuis des clients situés derrière un
182      pare-feu est une utilisation typique du mandataire direct. Le
183      mandataire direct peut aussi utiliser la mise en cache (fournie
184      par <code class="module"><a href="../mod/mod_cache.html">mod_cache</a></code>) pour réduire la charge du
185      réseau.</p>
186
187      <p>La fonctionnalité de mandataire direct est activée via la
188      directive <code class="directive"><a href="#proxyrequests">ProxyRequests</a></code>.
189      Comme les mandataires directs permettent aux clients d'accéder à
190      des sites quelconques via votre serveur et de dissimuler leur
191      véritable origine, il est indispensable de <a href="#access">sécuriser votre serveur</a> de façon à ce que seuls
192      les clients autorisés puissent accéder à votre serveur avant
193      d'activer la fonctionnalité de mandataire direct.</p>
194
195      <p>Un <dfn>mandataire inverse</dfn> (ou <dfn>passerelle</dfn>),
196      quant à lui, apparaît au client comme un serveur web standard.
197      Aucune configuration particulière du client n'est nécessaire. Le
198      client adresse ses demandes de contenus ordinaires dans l'espace
199      de nommage du mandataire inverse. Ce dernier décide alors où
200      envoyer ces requêtes, et renvoie le contenu au client comme s'il
201      l'hébergeait lui-même.</p>
202
203      <p>L'accès d'utilisateurs depuis Internet vers un serveur situé
204      derrière un pare-feu est une utilisation typique du mandataire
205      inverse. On peut aussi utiliser les mandataires inverses pour
206      mettre en oeuvre une répartition de charge entre plusieurs
207      serveurs en arrière-plan, ou fournir un cache pour un serveur
208      d'arrière-plan plus lent. Les mandataires inverses peuvent aussi
209      tout simplement servir à rassembler plusieurs serveurs dans le
210      même espace de nommage d'URLs.</p>
211
212      <p>La fonctionnalité de mandataire inverse est activée via la
213      directive <code class="directive"><a href="#proxypass">ProxyPass</a></code> ou
214      le drapeau <code>[P]</code> de la directive <code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">RewriteRule</a></code>. Il n'est
215      <strong>pas</strong> nécessaire de définir <code class="directive"><a href="#proxyrequests">ProxyRequests</a></code> pour configurer
216      un mandataire inverse.</p>

258&lt;/FilesMatch&gt;</pre>

266<h2><a name="workers" id="workers">Workers</a></h2>
267      <p>Le mandataire gère la configuration et les paramètres de
268      communication des serveurs originaux au sein d'objets nommés
269      <dfn>workers</dfn>. Deux types de worker sont fournis : le worker
270      par défaut du mandataire direct et le worker par défaut du
271      mandataire inverse. Il est aussi possible de définir explicitement
272      des workers supplémentaires.</p>
273
274      <p>Les deux workers par défaut possèdent une configuration figée
275      et seront utilisés si aucun autre worker ne correspond à la
276      requête. Ils ne réutilisent pas les connexions et n'utilisent pas les
277      connexions HTTP persistantes (Keep-Alive). En effet, les
278      connexions TCP vers le serveur original sont fermées et ouvertes
279      pour chaque requête.</p>
280
281      <p>Les workers définis explicitement sont identifiés par leur URL.
282      Ils sont en général définis via les directives <code class="directive"><a href="#proxypass">ProxyPass</a></code> ou <code class="directive"><a href="#proxypassmatch">ProxyPassMatch</a></code> lorsqu'on les
283      utilise dans le cadre d'un mandataire inverse :</p>
284
285      <div class="example"><pre class="prettyprint lang-config">ProxyPass "/example" "http://backend.example.com" connectiontimeout=5 timeout=30</pre>
286</div>
287      
288
289      <p>Cette directive va créer un worker associé à l'URL du serveur
290      original <code>http://backend.example.com</code> qui utilisera les
291      valeurs de timeout données. Lorsqu'ils sont utilisés dans le cadre
292      d'un mandataire direct, les workers sont en général définis via la
293      directive <code class="directive"><a href="#proxyset">ProxySet</a></code>,</p>
294
295      <div class="example"><pre class="prettyprint lang-config">ProxySet "http://backend.example.com" connectiontimeout=5 timeout=30</pre>
296</div>
297      
298
299      <p>ou encore via les directives <code class="directive"><a href="#proxy">Proxy</a></code> et <code class="directive"><a href="#proxyset">ProxySet</a></code> :</p>
300
301      <pre class="prettyprint lang-config">&lt;Proxy "http://backend.example.com"&gt;
302  ProxySet connectiontimeout=5 timeout=30
303&lt;/Proxy&gt;</pre>
304
305
306      <p>L'utilisation de workers définis explicitement dans le mode
307      mandataire direct n'est pas très courante, car les mandataires
308      directs communiquent en général avec de nombreux serveurs
309      originaux. La création explicite de workers pour certains serveurs
310      originaux peut cependant s'avérer utile si ces serveurs sont
311      très souvent sollicités. A leur niveau, les workers explicitement
312      définis ne possèdent aucune notion de mandataire direct ou
313      inverse. Ils encapsulent un concept de communication commun avec
314      les serveurs originaux. Un worker créé via la directive <code class="directive"><a href="#proxypass">ProxyPass</a></code> pour être utilisé dans le
315      cadre d'un mandataire inverse sera aussi utilisé dans le cadre
316      d'un mandataire directe chaque fois que l'URL vers le serveur
317      original correspondra à l'URL du worker, et vice versa.</p>
318
319      <p>L'URL qui identifie un worker correspond à l'URL de son serveur
320      original, y compris un éventuel chemin donné :</p>
321
322      <pre class="prettyprint lang-config">ProxyPass "/examples" "http://backend.example.com/examples"
323ProxyPass "/docs" "http://backend.example.com/docs"</pre>
324
325
326      <p>Dans cet exemple, deux workers différents sont définis, chacun
327      d'eux utilisant des configurations et jeux de connexions
328      séparés.</p>
329
330      <div class="warning"><h3>Partage de workers</h3>
331        <p>Le partage de workers intervient lorsque les URLs des workers
332	s'entrecoupent, ce qui arrive lorsque l'URL d'un worker
333	correspond au début de l'URL d'un autre worker défini plus loin
334	dans le fichier de configuration. Dans l'exemple suivant,</p>
335
336        <pre class="prettyprint lang-config">ProxyPass "/apps" "http://backend.example.com/" timeout=60
337ProxyPass "/examples" "http://backend.example.com/examples" timeout=10</pre>
338
339
340        <p>le second worker n'est pas vraiment créé. C'est le premier
341	worker qui est en fait utilisé. L'avantage de ceci réside dans
342	le fait qu'il n'existe qu'un seul jeu de connexions, ces
343	dernières étant donc réutilisées plus souvent. Notez que tous
344	les attributs de configuration définis explicitement pour le
345	deuxième worker seront ignorés, ce qui sera journalisé en tant
346	qu'avertissement. Ainsi, dans l'exemple ci-dessus, la valeur de
347	timeout retenue pour l'URL <code>/exemples</code> sera
348	<code>60</code>, et non <code>10</code> !</p>
349
350        <p>Si vous voulez empêcher le partage de workers, classez vos
351	définitions de workers selon la longueur des URLs, de la plus
352	longue à la plus courte. Si au contraire vous voulez favoriser
353	ce partage, utilisez l'ordre de classement inverse. Voir aussi
354	l'avertissement à propos de l'ordre de classement des directives
355	<code class="directive"><a href="#proxypass">ProxyPass</a></code>.</p>
356
357      </div> 
358
359      <p>Les workers définis explicitement sont de deux sortes :
360      <dfn>workers directs</dfn> et <dfn>workers de répartition (de
361      charge)</dfn>. Ils supportent de nombreux attributs de
362      configuration importants décrits dans la directive <code class="directive"><a href="#proxypass">ProxyPass</a></code>. Ces mêmes attributs
363      peuvent aussi être définis via la directive <code class="directive"><a href="#proxyset">ProxySet</a></code>.</p>
364
365      <p>Le jeu d'options disponibles pour un worker direct dépend du
366      protocole spécifié dans l'URL du serveur original. Les protocoles
367      disponibles comprennent <code>ajp</code>, <code>fcgi</code>,
368      <code>ftp</code>, <code>http</code> et <code>scgi</code>.</p>
369
370      <p>Les workers de répartition sont des workers virtuels qui
371      utilisent les workers directs, connus comme faisant partie de leurs
372      membres, pour le traitement effectif des requêtes. Chaque
373      répartiteur peut comporter plusieurs membres. Lorsqu'il traite une
374      requête, il choisit un de ses membres en fonction de l'algorithme
375      de répartition de charge défini.</p>
376
377      <p>Un worker de répartition est créé si son URL de worker comporte
378      <code>balancer</code> comme indicateur de protocole. L'URL du
379      répartiteur permet d'identifier de manière unique le worker de
380      répartition. La directive <code class="directive"><a href="#balancermember">BalancerMember</a></code> permet d'ajouter des
381      membres au répartiteur.</p>
382
383      <div class="note"><h3>Résolution DNS pour les domaines originaux</h3>
384      <p>La résolution DNS s'effectue lorsque le socket vers le
385      domaine original est créé pour la première fois. Lorsque la réutilisation
386      des connexions est activée, chaque domaine d'arrière-plan n'est résolu qu'une
387      seule fois pour chaque processus enfant, et cette résolution est mise en
388      cache pour toutes les connexions ultérieures jusqu'à ce que le processus enfant
389      soit recyclé. Ce comportement doit être pris en considération lorsqu'on
390      planifie des tâches de maintenance du DNS impactant les domaines
391      d'arrière-plan. Veuillez aussi vous reporter aux paramètres de la
392      directive <code class="directive"><a href="#proxypass">ProxyPass</a></code> pour plus de
393      détails à propos de la réutilisation des connexions.</p>
394      </div>

491<h2><a name="request-bodies" id="request-bodies">Corps de requêtes</a></h2>
492
493    <p>Certaines méthodes de requêtes comme POST comportent un corps de
494    requête. Le protocole HTTP stipule que les requêtes qui comportent
495    un corps doivent soit utiliser un codage de transmission
496    fractionnée (chunked transfer encoding), soit envoyer un en-tête de requête
497    <code>Content-Length</code>. Lorsqu'il fait suivre ce genre de
498    requête vers le serveur demandé, <code class="module"><a href="../mod/mod_proxy_http.html">mod_proxy_http</a></code>
499    s'efforce toujours d'envoyer l'en-tête <code>Content-Length</code>.
500    Par contre, si la taille du corps est importante, et si la requête
501    originale utilise un codage à fractionnement, ce dernier peut aussi
502    être utilisé dans la requête montante. Ce comportement peut être
503    contrôlé à l'aide de <a href="../env.html">variables
504    d'environnement</a>. Ainsi, si elle est définie, la variable
505    <code>proxy-sendcl</code> assure une compatibilité maximale avec les
506    serveurs demandés en imposant l'envoi de l'en-tête
507    <code>Content-Length</code>, alors que
508    <code>proxy-sendchunked</code> diminue la consommation de ressources
509    en imposant l'utilisation d'un codage à fractionnement.</p>
510
511    <p>Dans certaines circonstances, le serveur doit mettre en file
512    d'attente sur disque les corps de requêtes afin de satisfaire le
513    traitement demandé des corps de requêtes. Par exemple, cette mise en
514    file d'attente se produira si le corps original a été envoyé selon un
515    codage morcelé (et possède une taille importante), alors que
516    l'administrateur a demandé que les requêtes du serveur
517    d'arrière-plan soient envoyées avec l'en-tête Content-Length ou en
518    HTTP/1.0. Cette mise en file d'attente se produira aussi si le corps
519    de la requête contient déjà un en-tête Content-Length, alors que le
520    serveur est configuré pour filtrer les corps des requêtes entrantes.</p>

524<h2><a name="x-headers" id="x-headers">En-têtes de requête du mandataire
525    inverse</a></h2>
526
527    <p>Lorsqu'il est configuré en mode mandataire inverse (en utilisant
528    par exemple la directive <code class="directive"><a href="#proxypass">ProxyPass</a></code>),
529    <code class="module"><a href="../mod/mod_proxy_http.html">mod_proxy_http</a></code> ajoute plusieurs en-têtes de requête
530    afin de transmettre des informations au serveur demandé. Ces
531    en-têtes sont les suivants :</p>
532
533    <dl>
534      <dt><code>X-Forwarded-For</code></dt>
535      <dd>L'adresse IP du client.</dd>
536      <dt><code>X-Forwarded-Host</code></dt>
537      <dd>L'hôte d'origine demandé par le client dans l'en-tête de
538      requête HTTP <code>Host</code>.</dd>
539      <dt><code>X-Forwarded-Server</code></dt>
540      <dd>Le nom d'hôte du serveur mandataire.</dd>
541    </dl>
542
543    <p>Ces en-têtes doivent être utilisés avec précautions sur le
544    serveur demandé, car ils contiendront plus d'une valeur (séparées
545    par des virgules) si la requête originale contenait déjà un de ces
546    en-têtes. Par exemple, vous pouvez utiliser
547    <code>%{X-Forwarded-For}i</code> dans la chaîne de format du journal
548    du serveur demandé pour enregistrer les adresses IP des clients
549    originaux, mais il est possible que vous obteniez plusieurs adresses
550    si la requête passe à travers plusieurs mandataires.</p>
551
552    <p>Voir aussi les directives <code class="directive"><a href="#proxypreservehost">ProxyPreserveHost</a></code> et <code class="directive"><a href="#proxyvia">ProxyVia</a></code> directives, qui permettent
553    de contrôler d'autres en-têtes de requête.</p>
554
555    <p>Note : Si vous devez ajouter des en-têtes particuliers à la
556    requête mandatée, utilisez la directive  <code class="directive"><a href="../mod/mod_headers.html#requestheader">RequestHeader</a></code>.</p>

690    <var>domaine</var>).</p>
691
692    <div class="example"><h3>Exemple</h3><p><code>
693      .com .example.org.
694    </code></p></div>
695
696    <p>Pour faire la distinction entre <var>domaine</var>s et <var><a href="#hostname">nom d'hôte</a></var>s (des points de vue à la fois
697    syntaxique et
698    sémantique, un domaine DNS pouvant aussi avoir un enregistrement DNS
699    de type A !), les <var>domaine</var>s sont toujours spécifiés en les
700    préfixant par un point.</p>
701
702    <div class="note"><h3>Note</h3>
703      <p>Les comparaisons de noms de domaines s'effectuent sans tenir
704      compte de la casse, et les parties droites des <var>Domaine</var>s
705      sont toujours censées correspondre à la racine de l'arborescence
706      DNS, si bien que les domaines <code>.ExEmple.com</code> et
707      <code>.exemple.com.</code> (notez le point à la fin du nom) sont
708      considérés comme identiques. Comme une comparaison de domaines ne
709      nécessite pas de recherche DNS, elle est beaucoup plus efficace
710      qu'une comparaison de sous-réseaux.</p>
711    </div></dd>
712
713    
714    <dt><var><a name="subnet" id="subnet">Sous-réseau</a></var></dt>
715    <dd>
716    <p>Un <dfn>Sous-réseau</dfn> est une adresse internet partiellement
717    qualifiée sous forme numérique (quatre nombres séparés par des
718    points), optionnellement suivie d'un slash et du masque de
719    sous-réseau spécifiant le nombre de bits significatifs dans le
720    <var>Sous-réseau</var>. Il représente un sous-réseau de serveurs qui
721    peuvent être atteints depuis la même interface réseau. En l'absence
722    de masque de sous-réseau explicite, il est sous-entendu que les
723    digits manquants (ou caractères 0) de fin spécifient le masque de
724    sous-réseau (Dans ce cas, le masque de sous-réseau ne peut être
725    qu'un multiple de 8). Voici quelques exemples :</p>
726
727    <dl>
728    <dt><code>192.168</code> ou <code>192.168.0.0</code></dt>
729    <dd>le sous-réseau 192.168.0.0 avec un masque de sous-réseau
730    implicite de 16 bits significatifs (parfois exprimé sous la forme
731    <code>255.255.0.0</code>)</dd>
732    <dt><code>192.168.112.0/21</code></dt>
733    <dd>le sous-réseau <code>192.168.112.0/21</code> avec un masque de
734    sous-réseau implicite de 21 bits significatifs (parfois exprimé
735    sous la forme<code>255.255.248.0</code>)</dd>
736    </dl>
737
738    <p>Comme cas extrêmes, un <em>Sous-réseau</em> avec un masque de
739    sous-réseau de 32 bits significatifs est équivalent à une <var><a href="#ipaddr">adresse IP</a></var>, alors qu'un <em>Sous-réseau</em> avec un masque de
740    sous-réseau de 0 bit significatif (c'est à dire 0.0.0.0/0) est
741    identique à la constante <var>_Default_</var>, et peut correspondre
742    à toute adresse IP.</p></dd>
743
744    
745    <dt><var><a name="ipaddr" id="ipaddr">Adresse IP</a></var></dt>
746    <dd>
747    <p>Une <dfn>Adresse IP</dfn> est une adresse internet pleinement
748    qualifiée sous forme numérique (quatre nombres séparés par des
749    points). En général, cette adresse représente un serveur, mais elle
750    ne doit pas nécessairement correspondre à un nom de domaine DNS.</p>
751    <div class="example"><h3>Exemple</h3><p><code>
752      192.168.123.7
753    </code></p></div>
754
755    <div class="note"><h3>Note</h3>
756      <p>Une <dfn>Adresse IP</dfn> ne nécessite pas de résolution DNS,
757      et peut ainsi s'avérer plus efficace quant aux performances
758      d'Apache.</p>
759    </div></dd>
760
761    
762    <dt><var><a name="hostname" id="hostname">Nom de serveur</a></var></dt>
763    <dd>
764    <p>Un <dfn>Nom de serveur</dfn> est un nom de domaine DNS pleinement
765    qualifié qui peut être résolu en une ou plusieurs adresses IP par le
766    service de noms de domaines DNS. Il représente un hôte logique (par
767    opposition aux <var><a href="#domain">Domaine</a></var>s, voir
768    ci-dessus), et doit pouvoir être résolu en une ou plusieurs <var><a href="#ipaddr">adresses IP</a></var> (ou souvent en une liste
769    d'hôtes avec différentes <var><a href="#ipaddr">adresses
770    IP</a></var>).</p>
771
772    <div class="example"><h3>Exemples</h3><p><code>
773      prep.ai.example.edu<br />
774      www.example.org
775    </code></p></div>
776
777    <div class="note"><h3>Note</h3>
778      <p>Dans de nombreuses situations, il est plus efficace de
779      spécifier une <var><a href="#ipaddr">adresse IP</a></var> qu'un
780      <var>Nom de serveur</var> car cela évite d'avoir à effectuer une
781      recherche DNS. La résolution de nom dans Apache httpd peut prendre un
782      temps très long lorsque la connexion avec le serveur de noms
783      utilise une liaison PPP lente.</p>
784      <p>Les comparaisons de <var>Nom de serveur</var> s'effectuent sans tenir
785      compte de la casse, et les parties droites des <var>Noms de serveur</var>
786      sont toujours censées correspondre à la racine de l'arborescence
787      DNS, si bien que les domaines <code>WWW.ExEmple.com</code> et
788      <code>www.example.com.</code> (notez le point à la fin du nom) sont
789      considérés comme identiques.</p>
790     </div></dd>
791    </dl>

831      <p>Une URL d'arrière-plan sera concernée par le conteneur Proxy si

833      dernier segment de chemin de la directive ne correspond qu'à un
834      préfixe de segment dee chemin de l'URL d'arrière-plan. Par exemple, &lt;Proxy
835      "http://example.com/foo"&gt; correspondra entre autres aux URLs
836      http://example.com/foo, http://example.com/foo/bar, et
837      http://example.com/foobar. La correspondance de l'URL finale
838      diffère du comportement de la section <code class="directive"><a href="../mod/core.html#location">&lt;Location&gt;</a></code> qui, pour le cas de cette note,
839      traitera le segment de chemin final comme s'il se terminait par un
840      slash.</p>
841      <p>Pour un contrôle plus fin de la correspondance des URL, voir la
842      directive <code class="directive">&lt;ProxyMatch&gt;</code>.</p>

1048    est de <code>512</code> octets, ou définie à <code>0</code> pour indiquer
1049    que la taille de tampon par défaut du système doit être utilisée.</p>
1050
1051    <p>Dans la plupart des cas, il n'y a aucune raison de modifier cette
1052    valeur.</p>
1053
1054    <p>Si elle est utilisée avec AJP, cette directive permet de définir
1055    la taille maximale du paquet AJP en octets. Si la valeur spécifiée
1056    est supérieure à 65536, elle est corrigée et prend la valeur 65536.
1057    Si vous ne conservez pas
1058    la valeur par défaut, vous devez aussi modifier l'attribut
1059    <code>packetSize</code> de votre connecteur AJP du côté de Tomcat !
1060    L'attribut <code>packetSize</code> n'est disponible que dans Tomcat
1061    <code>5.5.20+</code> et <code>6.0.2+</code>.</p>
1062    <p>Il n'est normalement pas nécessaire de modifier la taille
1063    maximale du paquet. Des problèmes ont cependant été rapportés avec
1064    la valeur par défaut lors de l'envoi de certificats ou de chaînes de
1065    certificats.</p>

1137<div class="directive-section"><h2><a name="proxypass" id="proxypass">Directive</a> <a name="ProxyPass" id="ProxyPass">ProxyPass</a></h2>
1138<table class="directive">
1139<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Référencer des serveurs distants depuis
1140l'espace d'URLs du serveur local</td></tr>
1141<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>ProxyPass [<var>chemin</var>] !|<var>url</var> [<var>clé=valeur</var>
1142  <var>[clé=valeur</var> ...]] [nocanon] [interpolate] [noquery]</code></td></tr>
1143<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>configuration globale, serveur virtuel, répertoire</td></tr>
1144<tr><th><a href="directive-dict.html#Status">Statut:</a></th><td>Extension</td></tr>
1145<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_proxy</td></tr>
1146<tr><th><a href="directive-dict.html#Compatibility">Compatibilité:</a></th><td>Les sockets de style Unix (Unix Domain Socket - UDS)
1147sont supportés à partir de la version 2.4.7 du serveur HTTP Apache</td></tr>
1148</table>
1149    <p>Cette directive permet de référencer des serveurs distants depuis
1150    l'espace d'URLs du serveur local. Le serveur
1151    local n'agit pas en tant que mandataire au sens conventionnel, mais
1152    plutôt comme miroir du serveur distant. Le serveur local est
1153    souvent nommé <dfn>mandataire inverse</dfn> ou
1154    <dfn>passerelle</dfn>. L'argument <var>chemin</var> est le nom d'un
1155    chemin virtuel local ; <var>url</var> est une URL partielle pour le
1156    serveur distant et ne doit pas contenir de chaîne d'arguments.</p>
1157
1158    <div class="note">Il est fortement recommandé de revoir le concept de <a href="#workers">Worker</a> avant d'aller plus loin.</div>
1159
1160    <div class="note">Cette directive n'est pas supportée au sein des sections <code class="directive"><a href="../mod/core.html#directory">&lt;Directory&gt;</a></code>, <code class="directive"><a href="../mod/core.html#if">&lt;If&gt;</a></code> et <code class="directive"><a href="../mod/core.html#files">&lt;Files&gt;</a></code>.</div>
1161
1162    <div class="warning">En général, la directive <code class="directive"><a href="#proxyrequests">ProxyRequests</a></code> doit être définie à
1163    <strong>off</strong> lorsqu'on utilise la directive
1164    <code class="directive">ProxyPass</code>.</div>
1165
1166    <p>Les sockets de style Unix sont supportés à partir de la version
1167    2.4.7 du serveur HTTP Apache ; pour utiliser cette fonctionnalité,
1168    il suffit d'utiliser une URL cible préfixée par
1169    <code>unix:/path/lis.sock|</code>. Par exemple, pour mandater HTTP
1170    et cibler l'UDS /home/www.socket, vous devez utiliser
1171    <code>unix:/home/www.socket|http://localhost/whatever/</code>.</p>
1172
1173    <div class="note"><strong>Note :</strong>Le chemin associé à l'URL
1174    <code>unix:</code> tient compte de la directive
1175    <code class="directive">DefaultRuntimeDir</code>.</div>
1176
1177    <p>Lorsque cette directive est utilisée dans une section <code class="directive"><a href="../mod/core.html#location">&lt;Location&gt;</a></code>, le premier
1178    argument est omis et le répertoire local est obtenu à partir de
1179    l'argument de la directive <code class="directive"><a href="../mod/core.html#location">&lt;Location&gt;</a></code>. Il en est de même à l'intérieur
1180    d'une section <code class="directive"><a href="../mod/core.html#locationmatch">&lt;LocationMatch&gt;</a></code>, mais le résultat ne sera
1181    probablement pas celui attendu car ProxyPassReverse va interpréter
1182    l'expression rationnelle littéralement comme un chemin ; si besoin
1183    est dans ce cas, définissez la directive ProxyPassReverse en dehors
1184    de la section, ou dans une section <code class="directive"><a href="../mod/core.html#location">&lt;Location&gt;</a></code> séparée.</p>
1185
1186    <p>Supposons que le serveur local a pour adresse
1187    <code>http://example.com/</code> ; alors la ligne</p>
1188
1189    <pre class="prettyprint lang-config">&lt;Location "/mirror/foo/"&gt;
1190    ProxyPass "http://backend.example.com/"
1191&lt;/Location&gt;</pre>
1192
1193
1194    <p>va convertir en interne toute requête pour
1195    <code>http://example.com/mirror/foo/bar</code> en une requête
1196    mandatée pour <code>http://backend.example.com/bar</code>.</p>
1197
1198    <p>Si vous avez besoin d'un configuration de mandataire inverse plus
1199    souple, reportez-vous à la documentaion de la directive <code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">RewriteRule</a></code> et son drapeau
1200    <code>[P]</code>.</p>
1201
1202    <p>La syntaxe alternative suivante est valide, bien qu'elle puisse
1203    induire une dégradation des performances lorsqu'elle est
1204    présente en très grand nombre. Elle possède l'avantage de
1205    permettre un contrôle dynamique via l'interface <a href="mod_proxy_balancer.html#balancer_manager">Balancer Manager</a> :</p>
1206
1207    <pre class="prettyprint lang-config">ProxyPass "/mirror/foo/" "http://backend.example.com/"</pre>
1208
1209
1210    <div class="warning">
1211    <p>Si le premier argument se termine par un slash
1212    <strong>/</strong>, il doit en être de même pour le second argument
1213    et vice versa. Dans le cas contraire, il risque de manquer des
1214    slashes nécessaires dans la requête résultante vers le serveur
1215    d'arrière-plan et les résulats ne seront pas ceux attendus.
1216    </p>
1217    </div>
1218
1219    <p>Le drapeau <code>!</code> permet de soustraire un sous-répertoire
1220    du mandat inverse, comme dans l'exemple suivant :</p>
1221
1222    <pre class="prettyprint lang-config">&lt;Location "/mirror/foo/"&gt;
1223    ProxyPass "http://backend.example.com/"
1224&lt;/Location&gt;
1225&lt;Location "/mirror/foo/i"&gt;
1226    ProxyPass "!"
1227&lt;/Location&gt;</pre>
1228
1229
1230    <pre class="prettyprint lang-config">ProxyPass "/mirror/foo/i" "!"
1231ProxyPass "/mirror/foo" "http://backend.example.com"</pre>
1232
1233
1234    <p>va mandater toutes les requêtes pour <code>/mirror/foo</code>
1235    vers <code>backend.example.com</code>, <em>sauf</em> les requêtes
1236    pour <code>/mirror/foo/i</code>.</p>
1237
1238    <p>Mélanger plusieurs configurations ProxyPass dans différents contextes ne
1239    fonctionne pas :</p>
1240    <pre class="prettyprint lang-config">ProxyPass "/mirror/foo/i" "!"
1241&lt;Location "/mirror/foo/"&gt;
1242    ProxyPass "http://backend.example.com/"
1243&lt;/Location&gt;</pre>
1244
1245    <p>Dans ce cas, une requête pour <code>/mirror/foo/i</code> sera tout de
1246    même mandatée car c'est la directive <code class="directive">ProxyPass</code> de la
1247    section Location qui sera évaluée en premier. Le fait que la directive
1248    <code class="directive">ProxyPass</code> supporte les deux contextes serveur
1249    principal et répertoire ne signifie pas que sa portée et sa position dans le
1250    fichier de configuration va garantir une quelconque priorité et/ou
1251    chronologie de prise en compte.</p>
1252
1253    <div class="warning"><h3>Ordre de classement des directives ProxyPass</h3>
1254      <p>Les directives <code class="directive"><a href="#proxypass">ProxyPass</a></code> et <code class="directive"><a href="#proxypassmatch">ProxyPassMatch</a></code> sont évaluées dans
1255      l'ordre de leur apparition dans le fichier de configuration. La
1256      première règle qui correspond s'applique. Vous devez donc en
1257      général classer les règles <code class="directive"><a href="#proxypass">ProxyPass</a></code> qui entrent en conflit de
1258      l'URL la plus longue à la plus courte. Dans le cas contraire, les
1259      règles situées après une règle dont l'URL correspond au début de
1260      leur propre URL seront ignorées. Notez que tout ceci est en
1261      relation avec le partage de workers.</p>
1262      
1263      </div>
1264    <div class="warning"><h3>Chronologie de prise en compte des directives
1265    ProxyPass au sein des sections Locations</h3>
1266      <p>On ne peut placer
1267      qu'une seule directive <code class="directive"><a href="#proxypass">ProxyPass</a></code> dans une section
1268      <code class="directive"><a href="../mod/core.html#location">Location</a></code>, et c'est la section
1269      la plus spécifique qui l'emportera.</p>
1270     </div>
1271     <div class="warning"><h3>Exclusions et variable d'environnement no-proxy</h3>
1272      <p>Les exclusions doivent se situer <em>avant</em>
1273      les directives <code class="directive">ProxyPass</code> générales. A partir de la
1274      version 2.4.26 du serveur HTTP Apache, la variable
1275      d'environnement "no-proxy" est une alternative aux exclusions et constitue
1276      le seul moyen de configurer une exclusion pour une directive
1277      <code class="directive">ProxyPass</code> dans le contexte d'une section <code class="directive"><a href="../mod/core.html#location">Location</a></code>. Cette variable doit être définie via
1278      la directive <code class="directive"><a href="../mod/mod_setenvif.html#setenvif">SetEnvIf</a></code> car la
1279      directive <code class="directive"><a href="../mod/mod_env.html#setenv">SetEnv</a></code> n'est pas évaluée
1280      assez tôt.</p>
1281
1282    </div>

1284    <p><strong>ProxyPass <code>clé=valeur</code> Paramètres</strong></p>
1285
1286    <p>Depuis la version 2.1 du serveur HTTP Apache, mod_proxy supporte
1287    les groupements de connexions vers un serveur d'arrière-plan. Les
1288    connexions créées à la demande peuvent être enregistrées dans un
1289    groupement pour une utilisation ultérieure. La taille du groupe
1290    ainsi que d'autres caractéristiques peuvent être définies via la
1291    directive <code class="directive">ProxyPass</code> au moyen de paramètres
1292    <code>clé=valeur</code> dont la description fait l'objet des
1293    tableaux ci-dessous.</p>
1294
1295    <div class="warning"><h3>Nombre maximum de connexions vers
1296    l'arrière-plan</h3>
1297     <p>Par défaut, mod_proxy permet et met en réserve le
1298    nombre maximum de connexions pouvant être utilisées simultanément par le
1299    processus enfant concerné du serveur web. Le paramètre <code>max</code>
1300    permet de réduire cette valeur par défaut. Le jeu de connexions est maintenu
1301    au niveau de chaque processus enfant du serveur web, <code>max</code> et les
1302    autres réglages n'étant pas coordonnés entre ces différents processus, sauf
1303    bien entendu lorsqu'un seul processus enfant n'est autorisé par la
1304    configuration ou le MPM utilisé.</p> </div>
1305    
1306    <p>Le paramètre <code>ttl</code>,
1307    quant à lui, permet de définir une durée de vie optionnelle ; les
1308    connexions qui n'ont pas été utilisées pendant au moins
1309    <code>ttl</code> secondes seront fermées. <code>ttl</code> permet
1310    aussi d'empêcher l'utilisation d'une connexion susceptible d'être
1311    fermée suite à une fin de vie de connexion persistante sur le
1312    serveur d'arrière-plan.</p>
1313

1787    <p><strong>Mots-clés additionnels de ProxyPass</strong></p>
1788
1789    <p>Normalement, mod_proxy va mettre sous leur forme canonique les
1790    URLs traitées par ProxyPass. Mais ceci peut être incompatible avec
1791    certains serveurs d'arrière-plan, et en particulier avec ceux qui
1792    utilisent <var>PATH_INFO</var>. Le mot-clé optionnel
1793    <var>nocanon</var> modifie ce comportement et permet de transmettre
1794    le chemin d'URL sous sa forme brute au serveur d'arrière-plan. Notez
1795    que ceci peut affecter la sécurité de votre serveur d'arrière-plan,
1796    car la protection limitée contre les attaques à base d'URL que
1797    fournit le mandataire est alors supprimée.</p>
1798
1799    <p>Par défaut, mod_proxy inclut la chaîne de paramètres lors de la
1800    génération de la variable d'environnement
1801    <var>SCRIPT_FILENAME</var>. Le mot-clé optionnel <var>noquery</var>
1802    (disponible à partir de la version 2.4.1) permet d'exclure cette
1803    chaîne.</p>
1804
1805    <p>Lorsque la directive ProxyPass est utilisée à l'intérieur d'une
1806    section <code class="directive"><a href="../mod/core.html#location">&lt;Location&gt;</a></code>, le premier argument est omis et le répertoire
1807    local est obtenu à partir de la section <code class="directive"><a href="../mod/core.html#location">&lt;Location&gt;</a></code>. Il en sera de même dans une

1810    nécessaire d'utiliser la directive
1811    <code class="directive">ProxyPassMatch</code> à la place.</p>

1814    <code class="directive"><a href="../mod/core.html#directory">&lt;Directory&gt;</a></code> ou
1815    <code class="directive"><a href="../mod/core.html#files">&lt;Files&gt;</a></code>.</p>

1818    souple, reportez-vous à la documentaion de la directive <code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">RewriteRule</a></code> et son drapeau

1944<div class="directive-section"><h2><a name="proxypassmatch" id="proxypassmatch">Directive</a> <a name="ProxyPassMatch" id="ProxyPassMatch">ProxyPassMatch</a></h2>
1945<table class="directive">
1946<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Fait correspondre des serveurs distants dans l'espace d'URL
1947du serveur local en utilisant des expressions rationnelles</td></tr>
1948<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>ProxyPassMatch [<var>regex</var>] !|<var>url</var> [<var>key=value</var>
1949	<var>[key=value</var> ...]]</code></td></tr>
1950<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>configuration globale, serveur virtuel, répertoire</td></tr>
1951<tr><th><a href="directive-dict.html#Status">Statut:</a></th><td>Extension</td></tr>
1952<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_proxy</td></tr>
1953<tr><th><a href="directive-dict.html#Compatibility">Compatibilité:</a></th><td>Depuis la version 2.4.47, les paramètres <var>key=value</var>
1954sont pris en compte lorsque le paramètre <var>url</var> contient des références
1955arrières (voir note ci-dessous).
1956</td></tr>
1957</table>
1958    <p>Cette directive est identique à la directive <code class="directive"><a href="#proxypass">ProxyPass</a></code>, mais fait usage des
1959    expressions rationnelles, au lieu d'une simple comparaison de
1960    préfixes. L'expression rationnelle spécifiée est comparée à
1961    l'<var>url</var>, et si elle correspond, le serveur va substituer
1962    toute correspondance entre parenthèses dans la chaîne donnée et
1963    l'utiliser comme nouvelle <var>url</var>.</p>
1964
1965    <div class="note"><strong>Note : </strong>Cette directive ne peut pas être
1966    utilisée dans un contexte de niveau répertoire.</div>
1967
1968    <p>Supposons que le serveur local a pour adresse
1969    <code>http://example.com/</code> ; alors</p>
1970
1971    <pre class="prettyprint lang-config">ProxyPassMatch "^(/.*\.gif)$" "http://backend.example.com/$1"</pre>
1972
1973
1974    <p>va provoquer la conversion interne de la requête locale
1975    <code>http://example.com/foo/bar.gif</code> en une requête mandatée
1976    pour <code>http://backend.example.com/foo/bar.gif</code>.</p>
1977
1978    <p>Le drapeau <code>!</code> vous permet de ne pas mandater un
1979    sous-répertoire donné.</p>
1980
1981    <p>Dans une section <code class="directive"><a href="../mod/core.html#locationmatch">&lt;LocationMatch&gt;</a></code>, le premier argument est
1982    omis et l'expression rationnelle est obtenue à partir de la directive
1983    <code class="directive"><a href="../mod/core.html#locationmatch">&lt;LocationMatch&gt;</a></code>.</p>
1984
1985    <p>Si vous avez besoin d'une configuration du mandataire inverse
1986    plus flexible, voyez la directive <code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">RewriteRule</a></code> avec le drapeau
1987    <code>[P]</code>.</p>
1988

1990 	<h3>Substitution par défaut</h3>
1991 	<p>Lorsque le paramètre URL n'utilise pas de références arrières
1992	dans l'expression rationnelle, l'URL originale sera ajoutée au
1993	paramètre URL.
1994 	</p>

1998        <h3>Paramètres <code>key=value</code> et <var>url</var> avec
1999	références arrières</h3>
2000      <p>Depuis la version 2.4.47, les paramètres <code>key=value</code> ne sont
2001      plus ignorés dans une directive <code class="directive">ProxyPassMatch</code>
2002      lorsqu'on utilise une <var>url</var> contenant des références arrières.
2003      Cependant, pour conserver le comportement précédent relatif à la
2004      réutilisation/conservation des connexions d'arrière-plan (qui n'avaient
2005      jamais été réutilisées auparavant pour ces URLs), les paramètres
2006      <var>enablereuse</var> et <var>disablereuse</var> prendront dans ce cas
2007      respectivement comme valeurs par défaut <code>off</code> et

2009      de réutiliser les connexions, <strong>sauf si</strong> des références
2010      arrières se trouvent dans la partie <code>authority</code> (nom d'hôte
2011      et/ou port) de l'<var>url</var> (cette condition est imposée depuis la
2012      version 2.4.55 du serveur HTTP Apache et provoque un avertissement au
2013      démarrage car ces URLs ne sont pas réutilisables sous cette forme).</p>

2028<div class="directive-section"><h2><a name="proxypassreverse" id="proxypassreverse">Directive</a> <a name="ProxyPassReverse" id="ProxyPassReverse">ProxyPassReverse</a></h2>
2029<table class="directive">
2030<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Ajuste l'URL dans les en-têtes de la réponse HTTP envoyée
2031par un serveur mandaté en inverse</td></tr>
2032<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>ProxyPassReverse [<var>chemin</var>] <var>url</var>
2033[interpolate]</code></td></tr>
2034<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>configuration globale, serveur virtuel, répertoire</td></tr>
2035<tr><th><a href="directive-dict.html#Status">Statut:</a></th><td>Extension</td></tr>
2036<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_proxy</td></tr>
2037</table>
2038    <p>Cette directive permet de faire en sorte qu'Apache httpd ajuste l'URL
2039    dans les en-têtes <code>Location</code>,
2040    <code>Content-Location</code> et <code>URI</code> des réponses de
2041    redirection HTTP. Ceci est essentiel lorsqu'Apache httpd est utilisé en
2042    tant que mandataire inverse (ou passerelle), afin d'éviter de
2043    court-circuiter le mandataire inverse suite aux redirections HTTP
2044    sur le serveur d'arrière-plan qui restent derrière le mandataire
2045    inverse.</p>
2046
2047    <p>Seuls les en-têtes de réponse HTTP spécialement mentionnés
2048    ci-dessus seront réécrits. Apache httpd ne réécrira ni les autres en-têtes
2049    de réponse, ni par défaut les références d'URLs dans les pages HTML. Cela
2050    signifie que dans le cas où un contenu mandaté contient des
2051    références à des URLs absolues, elles court-circuiteront le
2052    mandataire. Pour réécrire un contenu HTML afin qu'il corresponde au
2053    mandataire, vous devez charger et activer le module
2054    <code class="module"><a href="../mod/mod_proxy_html.html">mod_proxy_html</a></code>.
2055    </p>
2056
2057    <p><var>chemin</var> est le nom d'un chemin virtuel local.
2058    <var>url</var> est une URL partielle pour le serveur distant. Ces
2059    paramètres s'utilisent de la même façon qu'avec la
2060    directive <code class="directive"><a href="#proxypass">ProxyPass</a></code>.</p>
2061
2062    <p>Supposons par exemple que le serveur local a pour adresse
2063    <code>http://example.com/</code> ; alors</p>
2064
2065    <pre class="prettyprint lang-config">ProxyPass         "/mirror/foo/" "http://backend.example.com/"
2066ProxyPassReverse  "/mirror/foo/" "http://backend.example.com/"
2067ProxyPassReverseCookieDomain  "backend.example.com" "public.example.com"
2068ProxyPassReverseCookiePath  "/"  "/mirror/foo/"</pre>
2069
2070
2071    <p>ne va pas seulement provoquer la conversion interne d'une requête
2072    locale pour <code>http://example.com/mirror/foo/bar</code> en une
2073    requête mandatée pour <code>http://backend.example.com/bar</code>
2074    (la fonctionnalité fournie par <code>ProxyPass</code>). Il va
2075    aussi s'occuper des redirections que le serveur
2076    <code>backend.example.com</code> envoie lorsqu'il redirige
2077    <code>http://backend.example.com/bar</code> vers
2078    <code>http://backend.example.com/quux</code>. Apache
2079    httpd corrige ceci en <code>http://example.com/mirror/foo/quux</code>
2080    avant de faire suivre la redirection HTTP au client. Notez que le
2081    nom d'hôte utilisé pour construire l'URL est choisi en respectant la
2082    définition de la directive <code class="directive"><a href="../mod/core.html#usecanonicalname">UseCanonicalName</a></code>.</p>
2083
2084    <p>Notez que la directive <code class="directive">ProxyPassReverse</code>
2085    peut aussi être utilisée en conjonction avec la
2086    fonctionnalité de mandataire
2087    (<code>RewriteRule ...  [P]</code>) du module
2088    <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code>, car elle ne dépend pas d'une directive
2089    <code class="directive"><a href="#proxypass">ProxyPass</a></code>
2090    correspondante.</p>
2091
2092    <p>Le mot-clé optionnel <code>interpolate</code>, en combinaison avec la
2093    directive <code class="directive"><a href="#proxypassinterpolateenv">ProxyPassInterpolateEnv</a></code>,
2094    permet l'interpolation des variables d'environnement spécifiées en utilisant
2095    le format <var>${VARNAME}</var> Notez que l'interpolation n'est pas
2096    supportée dans la partie protocole d'une URL.  </p>
2097
2098    <p>Lorsque cette directive est utilisée dans une section <code class="directive"><a href="../mod/core.html#location">&lt;Location&gt;</a></code>, le premier
2099    argument est omis et le répertoire local est obtenu à partir de
2100    l'argument de la directive <code class="directive"><a href="../mod/core.html#location">&lt;Location&gt;</a></code>. Il en est de même à l'intérieur
2101    d'une section <code class="directive"><a href="../mod/core.html#locationmatch">&lt;LocationMatch&gt;</a></code>, mais le résultat ne sera
2102    probablement pas celui attendu car ProxyPassReverse va interpréter
2103    l'expression rationnelle littéralement comme un chemin ; si besoin
2104    est dans ce cas, définissez la directive ProxyPassReverse en dehors
2105    de la section, ou dans une section <code class="directive"><a href="../mod/core.html#location">&lt;Location&gt;</a></code> séparée.</p>
2106
2107    <p>Cette directive ne peut pas être placée dans une section
2108    <code class="directive"><a href="../mod/core.html#directory">&lt;Directory&gt;</a></code> ou
2109    <code class="directive"><a href="../mod/core.html#files">&lt;Files&gt;</a></code>.</p>

2209<div class="directive-section"><h2><a name="proxyremote" id="proxyremote">Directive</a> <a name="ProxyRemote" id="ProxyRemote">ProxyRemote</a></h2>
2210<table class="directive">
2211<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Mandataire distant à utiliser pour traiter certaines
2212requêtes</td></tr>
2213<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>ProxyRemote <var>match</var> <var>remote-server</var> [<var>username:password</var>]</code></td></tr>
2214<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>configuration globale, serveur virtuel</td></tr>
2215<tr><th><a href="directive-dict.html#Status">Statut:</a></th><td>Extension</td></tr>
2216<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_proxy</td></tr>
2217<tr><th><a href="directive-dict.html#Compatibility">Compatibilité:</a></th><td>Le troisième argument facultatif est disponible depuis la version
22182.4.59 du serveur HTTP Apache.</td></tr>
2219</table>
2220    <p>Cette directive permet de définir des mandataires distants pour
2221    ce mandataire. <var>match</var> est soit le nom d'un protocole
2222    que supporte le serveur distant, soit une URL partielle pour
2223    laquelle le serveur distant devra être utilisé, soit <code>*</code>
2224    pour indiquer que le serveur distant doit être utilisé pour toutes
2225    les requêtes. <var>remote-server</var> est une URL partielle
2226    correspondant au serveur distant. Syntaxe : </p>
2227
2228    <div class="example"><p><code>
2229      <dfn>remote-server</dfn> =
2230          <var>scheme</var>://<var>hostname</var>[:<var>port</var>]
2231    </code></p></div>
2232
2233    <p><var>scheme</var> est effectivement le protocole à utiliser
2234    pour communiquer avec le serveur distant ; ce module ne supporte que
2235    <code>http</code> et <code>https</code>. Lorsqu'on utilise
2236    <code>https</code>, les requêtes sont redirigées par le mandataire
2237    distant en utilisant la méthode HTTP CONNECT.</p>
2238
2239    <div class="example"><h3>Exemple</h3><pre class="prettyprint lang-config">ProxyRemote "http://goodguys.example.com/" "http://mirrorguys.example.com:8000"
2240ProxyRemote "*" "http://cleverproxy.localdomain"
2241ProxyRemote "ftp" "http://ftpproxy.mydomain:8080"</pre>
2242</div>
2243
2244    <p>Dans la dernière ligne de l'exemple, le mandataire va faire
2245    suivre les requêtes FTP, encapsulées dans une autre requête mandatée
2246    HTTP, vers un autre mandataire capable de les traiter.</p>
2247
2248    <p>Cette directive supporte aussi les configurations de mandataire
2249    inverse ; un serveur web d'arrière-plan peut être intégré dans
2250    l'espace d'URL d'un serveur virtuel, même si ce serveur est caché
2251    par un autre mandataire direct.</p>
2252
2253    <p>Le troisième argument optionnel <var>username:password</var> permet de
2254    spécifier des données d'authentification basiques à transmettre au mandataire
2255    distant défini. Ces données d'authentification seront toujoujours envoyées
2256    sans attendre que le mandataire distant n'effectue une demande
2257    d'authentification. La variable d'environnement <a href="mod_proxy_http.html#env">Proxy-Chain-Auth</a> n'est plus prise en
2258    compte si cet argument est utilisé.</p>

2448    <li>Si elle est définie à <code>Off</code>, valeur par défaut, cette
2449    directive n'effectue aucun traitement particulier. Si une requête ou
2450    une réponse contient un en-tête <code>Via:</code>, il est transmis
2451    sans modification.</li>
2452
2453    <li>Si elle est définie à <code>On</code>, chaque requête ou réponse
2454    se verra ajouter une ligne d'en-tête <code>Via:</code> pour le
2455    serveur courant.</li>
2456
2457    <li>Si elle est définie à <code>Full</code>, chaque ligne d'en-tête
2458    <code>Via:</code> se verra ajouter la version du serveur Apache
2459    httpd sous la forme d'un champ de commentaire <code>Via:</code>.</li>
2460
2461    <li>Si elle est définie à <code>Block</code>, chaque requête
2462    mandatée verra ses lignes d'en-tête <code>Via:</code> supprimées.
2463    Aucun nouvel en-tête <code>Via:</code> ne sera généré.</li>
2464    </ul>
