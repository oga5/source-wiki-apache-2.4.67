38<h3>Summary</h3>
39
40    <p>This module allows FastCGI authorizer applications to
41    authenticate users and authorize access to resources.  It supports
42    generic FastCGI authorizers which participate in a single phase
43    for authentication and authorization as well as Apache httpd-specific
44    authenticators and authorizors which participate in one or both
45    phases.</p>
46
47    <p>FastCGI authorizers can authenticate using user id and password, 
48    such as for Basic authentication, or can authenticate using arbitrary
49    mechanisms.</p>

72<h2><a name="invocations" id="invocations">Invocation modes</a></h2>
73
74    <p>The invocation modes for FastCGI authorizers supported by this 
75    module are distinguished by two characteristics, <em>type</em> and
76    auth <em>mechanism</em>.</p>
77
78    <p><em>Type</em> is simply <code>authn</code> for authentication,
79    <code>authz</code> for authorization, or <code>authnz</code> for
80    combined authentication and authorization.</p>
81
82    <p>Auth <em>mechanism</em> refers to the Apache httpd configuration
83    mechanisms and processing phases, and can be <code>
84    AuthBasicProvider</code>, <code>Require</code>, or <code>
85    check_user_id</code>.  The first two of these
86    correspond to the directives used to enable participation in the
87    appropriate processing phase.</p>
88
89    <p>Descriptions of each mode:</p>
90
91    <dl>
92      <dt><em>Type</em> <code>authn</code>, <em>mechanism</em>
93      <code>AuthBasicProvider</code></dt>
94
95      <dd>In this mode, 
96      <code>FCGI_ROLE</code> is set to <code>AUTHORIZER</code> and
97      <code>FCGI_APACHE_ROLE</code> is set to <code>AUTHENTICATOR</code>.
98      The application must be defined as provider type <em>authn</em>
99      using <code class="directive"><a href="#authnzfcgidefineprovider">
100      AuthnzFcgiDefineProvider</a></code> and enabled with
101      <code class="directive"><a href="../mod/mod_auth_basic.html#authbasicprovider">AuthBasicProvider</a></code>.
102      When invoked, the application is
103      expected to authenticate the client using the provided user id and
104      password.  Example application:
105
106<pre class="prettyprint lang-perl">#!/usr/bin/perl
107use FCGI;
108my $request = FCGI::Request();
109while ($request-&gt;Accept() &gt;= 0) {
110    die if $ENV{'FCGI_APACHE_ROLE'} ne "AUTHENTICATOR";
111    die if $ENV{'FCGI_ROLE'}        ne "AUTHORIZER";
112    die if !$ENV{'REMOTE_PASSWD'};
113    die if !$ENV{'REMOTE_USER'};
114
115    print STDERR "This text is written to the web server error log.\n";
116
117    if ( ($ENV{'REMOTE_USER' } eq "foo" || $ENV{'REMOTE_USER'} eq "foo1") &amp;&amp;
118        $ENV{'REMOTE_PASSWD'} eq "bar" ) {
119        print "Status: 200\n";
120        print "Variable-AUTHN_1: authn_01\n";
121        print "Variable-AUTHN_2: authn_02\n";
122        print "\n";
123    }
124    else {
125        print "Status: 401\n\n";
126    }
127}</pre>
128
129
130      Example configuration:
131<pre class="prettyprint lang-config">AuthnzFcgiDefineProvider authn FooAuthn fcgi://localhost:10102/
132&lt;Location "/protected/"&gt;
133  AuthType Basic
134  AuthName "Restricted"
135  AuthBasicProvider FooAuthn
136  Require ...
137&lt;/Location&gt;</pre>
138
139      </dd>
140
141      <dt><em>Type</em> <code>authz</code>, <em>mechanism</em>
142      <code>Require</code></dt>
143      <dd>In this mode, <code>FCGI_ROLE</code> is set to <code>
144      AUTHORIZER</code> and <code>FCGI_APACHE_ROLE</code> is set to
145      <code>AUTHORIZER</code>.  The application must be defined as
146      provider type <em>authz</em> using <code class="directive"><a href="#authnzfcgidefineprovider">
147      AuthnzFcgiDefineProvider</a></code>.  When invoked, the application 
148      is expected to authorize the client using the provided user id and other
149      request data.  Example application:
150<pre class="prettyprint lang-perl">#!/usr/bin/perl
151use FCGI;
152my $request = FCGI::Request();
153while ($request-&gt;Accept() &gt;= 0) {
154    die if $ENV{'FCGI_APACHE_ROLE'} ne "AUTHORIZER";
155    die if $ENV{'FCGI_ROLE'}        ne "AUTHORIZER";
156    die if $ENV{'REMOTE_PASSWD'};
157
158    print STDERR "This text is written to the web server error log.\n";
159
160    if ($ENV{'REMOTE_USER'} eq "foo1") {
161        print "Status: 200\n";
162        print "Variable-AUTHZ_1: authz_01\n";
163        print "Variable-AUTHZ_2: authz_02\n";
164        print "\n";
165    }
166    else {
167        print "Status: 403\n\n";
168    }
169}</pre>
170
171
172      Example configuration:
173<pre class="prettyprint lang-config">AuthnzFcgiDefineProvider authz FooAuthz fcgi://localhost:10103/
174&lt;Location "/protected/"&gt;
175  AuthType ...
176  AuthName ...
177  AuthBasicProvider ...
178  Require FooAuthz
179&lt;/Location&gt;</pre>
180
181      </dd>
182
183      <dt><em>Type</em> <code>authnz</code>, <em>mechanism</em>
184      <code>AuthBasicProvider</code> <em>+</em> <code>Require</code></dt>
185
186      <dd>In this mode, which supports the web server-agnostic FastCGI
187      <code>AUTHORIZER</code> protocol, <code>FCGI_ROLE</code> is set to
188      <code>AUTHORIZER</code> and <code>FCGI_APACHE_ROLE</code> is not set.
189      The application must be defined as provider type <em>authnz</em>
190      using <code class="directive"><a href="#authnzfcgidefineprovider">
191      AuthnzFcgiDefineProvider</a></code>.  The application is expected to
192      handle both authentication and authorization in the same invocation
193      using the user id, password, and other request data.  The invocation 
194      occurs during the Apache httpd API authentication phase.  If the 
195      application returns 200 and the same provider is invoked during the
196      authorization phase (via <code class="directive">Require</code>), mod_authnz_fcgi
197      will return success for the authorization phase without invoking the
198      application.  Example application:
199<pre class="prettyprint lang-perl">#!/usr/bin/perl
200use FCGI;
201my $request = FCGI::Request();
202while ($request-&gt;Accept() &gt;= 0) {
203    die if $ENV{'FCGI_APACHE_ROLE'};
204    die if $ENV{'FCGI_ROLE'} ne "AUTHORIZER";
205    die if !$ENV{'REMOTE_PASSWD'};
206    die if !$ENV{'REMOTE_USER'};
207
208    print STDERR "This text is written to the web server error log.\n";
209
210    if ( ($ENV{'REMOTE_USER' } eq "foo" || $ENV{'REMOTE_USER'} eq "foo1") &amp;&amp;
211        $ENV{'REMOTE_PASSWD'} eq "bar" &amp;&amp;
212        $ENV{'REQUEST_URI'} =~ m%/bar/.*%) {
213        print "Status: 200\n";
214        print "Variable-AUTHNZ_1: authnz_01\n";
215        print "Variable-AUTHNZ_2: authnz_02\n";
216        print "\n";
217    }
218    else {
219        print "Status: 401\n\n";
220    }
221}</pre>
222
223
224      Example configuration:
225<pre class="prettyprint lang-config">AuthnzFcgiDefineProvider authnz FooAuthnz fcgi://localhost:10103/
226&lt;Location "/protected/"&gt;
227  AuthType Basic
228  AuthName "Restricted"
229  AuthBasicProvider FooAuthnz
230  Require FooAuthnz
231&lt;/Location&gt;</pre>
232
233      </dd>
234
235      <dt><em>Type</em> <code>authn</code>, <em>mechanism</em>
236      <code>check_user_id</code></dt>
237
238      <dd>In this mode, <code>FCGI_ROLE</code> is set to <code>
239      AUTHORIZER</code> and <code>FCGI_APACHE_ROLE</code> is set to
240      <code>AUTHENTICATOR</code>.  The application must be defined as
241      provider type <em>authn</em> using <code class="directive"><a href="#authnzfcgidefineprovider">
242      AuthnzFcgiDefineProvider</a></code>.  <code class="directive"><a href="#authnzfcgicheckauthnprovider">AuthnzFcgiCheckAuthnProvider</a></code>
243      specifies when it is called.  Example application:
244<pre class="prettyprint lang-perl">#!/usr/bin/perl
245use FCGI;
246my $request = FCGI::Request();
247while ($request-&gt;Accept() &gt;= 0) {
248    die if $ENV{'FCGI_APACHE_ROLE'} ne "AUTHENTICATOR";
249    die if $ENV{'FCGI_ROLE'} ne "AUTHORIZER";
250
251    # This authorizer assumes that the RequireBasicAuth option of 
252    # AuthnzFcgiCheckAuthnProvider is On:
253    die if !$ENV{'REMOTE_PASSWD'};
254    die if !$ENV{'REMOTE_USER'};
255
256    print STDERR "This text is written to the web server error log.\n";
257
258    if ( ($ENV{'REMOTE_USER' } eq "foo" || $ENV{'REMOTE_USER'} eq "foo1") &amp;&amp;
259        $ENV{'REMOTE_PASSWD'} eq "bar" ) {
260        print "Status: 200\n";
261        print "Variable-AUTHNZ_1: authnz_01\n";
262        print "Variable-AUTHNZ_2: authnz_02\n";
263        print "\n";
264    }
265    else {
266        print "Status: 401\n\n";
267        # If a response body is written here, it will be returned to
268        # the client.
269    }
270}</pre>
271
272
273      Example configuration:
274<pre class="prettyprint lang-config">AuthnzFcgiDefineProvider authn FooAuthn fcgi://localhost:10103/
275&lt;Location "/protected/"&gt;
276  AuthType ...
277  AuthName ...
278  AuthnzFcgiCheckAuthnProvider FooAuthn \
279                               Authoritative On \
280                               RequireBasicAuth Off \
281                               UserExpr "%{reqenv:REMOTE_USER}"
282  Require ...
283&lt;/Location&gt;</pre>
284
285      </dd>
286
287    </dl>

291<h2><a name="examples" id="examples">Additional examples</a></h2>
292
293  <ol>
294    <li>If your application supports the separate authentication and
295    authorization roles (<code>AUTHENTICATOR</code> and <code>AUTHORIZER</code>), define 
296    separate providers as follows, even if they map to the same
297    application:
298
299<pre class="prettyprint lang-config">AuthnzFcgiDefineProvider authn  FooAuthn  fcgi://localhost:10102/
300AuthnzFcgiDefineProvider authz  FooAuthz  fcgi://localhost:10102/</pre>
301
302
303    Specify the authn provider on 
304    <code class="directive"><a href="../mod/mod_auth_basic.html#authbasicprovider">AuthBasicProvider</a></code>
305    and the authz provider on
306    <code class="directive"><a href="../mod/mod_authz_core.html#require">Require</a></code>:
307
308<pre class="prettyprint lang-config">AuthType Basic
309AuthName "Restricted"
310AuthBasicProvider FooAuthn
311Require FooAuthz</pre>
312
313    </li>
314
315    <li>If your application supports the generic <code>AUTHORIZER</code> role 
316    (authentication and authorizer in one invocation), define a
317    single provider as follows:
318
319<pre class="prettyprint lang-config">AuthnzFcgiDefineProvider authnz FooAuthnz fcgi://localhost:10103/</pre>
320
321
322    Specify the authnz provider on both <code class="directive">AuthBasicProvider</code>
323    and <code class="directive">Require</code>:
324
325<pre class="prettyprint lang-config">AuthType Basic
326AuthName "Restricted"
327AuthBasicProvider FooAuthnz
328Require FooAuthnz</pre>
329
330    </li>
331</ol>

334<h2><a name="limitations" id="limitations">Limitations</a></h2>
335
336    <p>The following are potential features which are not currently
337    implemented:</p>
338
339    <dl>
340      <dt>Apache httpd access checker</dt>
341      <dd>The Apache httpd API <em>access check</em> phase is a separate
342      phase from authentication and authorization.  Some other FastCGI
343      implementations implement this phase, which is denoted by the 
344      setting of <code>FCGI_APACHE_ROLE</code> to <code>ACCESS_CHECKER</code>.</dd>
345
346      <dt>Local (Unix) sockets or pipes</dt>
347      <dd>Only TCP sockets are currently supported.</dd>
348
349      <dt>Support for mod_authn_socache</dt>
350      <dd>mod_authn_socache interaction should be implemented for
351      applications which participate in Apache httpd-style
352      authentication.</dd>
353
354      <dt>Support for digest authentication using AuthDigestProvider</dt>
355      <dd>This is expected to be a permanent limitation as there is
356      no authorizer flow for retrieving a hash.</dd>
357
358      <dt>Application process management</dt>
359      <dd>This is expected to be permanently out of scope for
360      this module.  Application processes must be controlled by
361      other means.  For example, <code class="program"><a href="../programs/fcgistarter.html">fcgistarter</a></code> can be used to
362      start them.</dd>
363
364      <dt>AP_AUTH_INTERNAL_PER_URI</dt>
365      <dd>All providers are currently registered as 
366      AP_AUTH_INTERNAL_PER_CONF, which means that checks are not
367      performed again for internal subrequests with the same
368      access control configuration as the initial request.</dd>
369
370      <dt>Protocol data charset conversion</dt>
371      <dd>If mod_authnz_fcgi runs in an EBCDIC compilation
372      environment, all FastCGI protocol data is written in EBCDIC
373      and expected to be received in EBCDIC.</dd>
374
375      <dt>Multiple requests per connection</dt>
376      <dd>Currently the connection to the FastCGI authorizer is
377      closed after every phase of processing.  For example, if the
378      authorizer handles separate <em>authn</em> and <em>authz</em>
379      phases then two connections will be used.</dd>
380
381      <dt>URI Mapping</dt>
382      <dd>URIs from clients can't be mapped, such as with the <code class="directive">
383      ProxyPass</code> used with FastCGI responders.</dd>
384
385    </dl>

389<h2><a name="logging" id="logging">Logging</a></h2>
390
391    <ol>
392        <li>Processing errors are logged at log level <code>error</code>
393        and higher.</li>
394        <li>Messages written by the application are logged at log
395        level <code>warn</code>.</li>
396        <li>General messages for debugging are logged at log level
397        <code>debug</code>.</li>
398        <li>Environment variables passed to the application are
399        logged at log level <code>trace2</code>. The value of the 
400        <code>REMOTE_PASSWD</code> variable will be obscured,
401        but <strong>any other sensitive data will be visible in the 
402        log</strong>.</li>
403        <li>All I/O between the module and the FastCGI application,
404        including all environment variables, will be logged in printable
405        and hex format at log level <code>trace5</code>.  <strong>All
406        sensitive data will be visible in the log.</strong></li>
407    </ol>
408
409    <p><code class="directive"><a href="../mod/core.html#loglevel">LogLevel</a></code> can be used
410    to configure a log level specific to mod_authnz_fcgi.  For
411    example:</p>
412
413<pre class="prettyprint lang-config">LogLevel info authnz_fcgi:trace8</pre>

418<div class="directive-section"><h2><a name="AuthnzFcgiCheckAuthnProvider" id="AuthnzFcgiCheckAuthnProvider">AuthnzFcgiCheckAuthnProvider</a> <a name="authnzfcgicheckauthnprovider" id="authnzfcgicheckauthnprovider">Directive</a></h2>
419<table class="directive">
420<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Enables a FastCGI application to handle the check_authn
421authentication hook.</td></tr>
422<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>AuthnzFcgiCheckAuthnProvider <em>provider-name</em>|<code>None</code>
423<em>option</em> ...</code></td></tr>
424<tr><th><a href="directive-dict.html#Default">Default:</a></th><td><code>none</code></td></tr>
425<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>directory</td></tr>
426<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Extension</td></tr>
427<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_authnz_fcgi</td></tr>
428</table>
429    <p>This directive is used to enable a FastCGI authorizer to
430    handle a specific processing phase of authentication or
431    authorization.</p>
432
433    <p>Some capabilities of FastCGI authorizers require enablement
434    using this directive instead of
435    <code class="directive">AuthBasicProvider</code>:</p>
436
437    <ul>
438      <li>Non-Basic authentication; generally, determining the user
439      id of the client and returning it from the authorizer; see the
440      <code>UserExpr</code> option below</li>
441      <li>Selecting a custom response code; for a non-200 response
442      from the authorizer, the code from the authorizer will be the
443      status of the response</li>
444      <li>Setting the body of a non-200 response; if the authorizer
445      provides a response body with a non-200 response, that body
446      will be returned to the client; up to 8192 bytes of text are
447      supported</li>
448    </ul>
449
450    <dl>
451      <dt><em>provider-name</em></dt>
452      <dd>This is the name of a provider defined with <code class="directive">
453      AuthnzFcgiDefineProvider</code>.</dd>
454
455      <dt><code>None</code></dt>
456      <dd>Specify <code>None</code> to disable a provider enabled
457      with this directive in an outer scope, such as in a parent
458      directory.</dd>
459
460      <dt><em>option</em></dt>
461      <dd>The following options are supported:
462      
463      <dl>
464         <dt>Authoritative On|Off (default On)</dt>
465         <dd>This controls whether or not other modules are allowed
466         to run when this module has a FastCGI authorizer configured
467         and it fails the request.</dd>
468
469         <dt>DefaultUser <em>userid</em></dt>
470         <dd>When the authorizer returns success and <code>UserExpr</code>
471         is configured and evaluates to an empty string (e.g., authorizer
472         didn't return a variable), this value will be used as the user
473         id.  This is typically used when the authorizer has a concept of
474         guest, or unauthenticated, users and guest users are mapped to
475         some specific user id for logging and other purposes.</dd>
476
477         <dt>RequireBasicAuth On|Off (default Off)</dt>
478         <dd>This controls whether or not Basic auth is required
479         before passing the request to the authorizer.  If required,
480         the authorizer won't be invoked without a user id and
481         password; 401 will be returned for a request without that.</dd>
482
483         <dt>UserExpr <em>expr</em> (no default)</dt>
484         <dd>When Basic authentication isn't provided by the client
485         and the authorizer determines the user, this expression,
486         evaluated after calling the authorizer, determines the
487         user.  The expression follows <a href="../expr.html">
488         ap_expr syntax</a> and must resolve to a string.  A typical
489         use is to reference a <code>Variable-<em>XXX</em></code> 
490         setting returned by the authorizer using an option like
491         <code>UserExpr "%{reqenv:<em>XXX</em>}"</code>.  If
492         this option is specified and the user id can't be retrieved
493         using the expression after a successful authentication, the
494         request will be rejected with a 500 error.</dd>
495
496       </dl>
497      </dd>
498     </dl>

502<div class="directive-section"><h2><a name="AuthnzFcgiDefineProvider" id="AuthnzFcgiDefineProvider">AuthnzFcgiDefineProvider</a> <a name="authnzfcgidefineprovider" id="authnzfcgidefineprovider">Directive</a></h2>
503<table class="directive">
504<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Defines a FastCGI application as a provider for
505authentication and/or authorization</td></tr>
506<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>AuthnzFcgiDefineProvider <em>type</em> <em>provider-name</em>
507<em>backend-address</em></code></td></tr>
508<tr><th><a href="directive-dict.html#Default">Default:</a></th><td><code>none</code></td></tr>
509<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config</td></tr>
510<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Extension</td></tr>
511<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_authnz_fcgi</td></tr>
512</table>
513    <p>This directive is used to define a FastCGI application as
514    a provider for a particular phase of authentication or
515    authorization.</p>
516
517    <dl>
518      <dt><em>type</em></dt>
519      <dd>This must be set to <em>authn</em> for authentication,
520      <em>authz</em> for authorization, or <em>authnz</em> for
521      a generic FastCGI authorizer which performs both checks.</dd>
522
523      <dt><em>provider-name</em></dt>
524      <dd>This is used to assign a name to the provider which is
525      used in other directives such as 
526      <code class="directive"><a href="../mod/mod_auth_basic.html#authbasicprovider">AuthBasicProvider</a></code>
527      and 
528      <code class="directive"><a href="../mod/mod_authz_core.html#require">Require</a></code>.</dd>
529
530      <dt><em>backend-address</em></dt>
531      <dd>This specifies the address of the application, in the form
532      <em>fcgi://hostname:port/</em>.  The application process(es)
533      must be managed independently, such as with
534      <code class="program"><a href="../programs/fcgistarter.html">fcgistarter</a></code>.</dd>
535    </dl>
