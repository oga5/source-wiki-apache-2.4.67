38<h3>Sommaire</h3>
39
40<p>Ce module permet d'ajouter au serveur des extensions sous forme de
41scripts écrits dans le langage de programmation Lua.
42<code class="module"><a href="../mod/mod_lua.html">mod_lua</a></code> fournit de nombreuses extensions
43(hooks) disponibles avec les modules natifs du serveur HTTP Apache,
44comme les associations de requêtes à des fichiers, la génération de
45réponses dynamiques, le contrôle d'accès, l'authentification et
46l'autorisation.</p>
47
48<p>Vous trouverez davantage d'informations à propos du langage de
49programmation Lua sur <a href="http://www.lua.org/">le site web de
50Lua</a>.</p>
51
52<div class="warning"><h3>Avertissement</h3>
53<p>Ce module possède une grande capacité d'action sur le fonctrionnement
54de httpd, ce qui lui confère une grande puissance, mais peut aussi
55induire un risque de sécurité. Il est déconseillé d'utiliser ce module
56sur un serveur partagé avec des utilisateurs auxquels vous ne pouvez pas
57accorder une confiance absolue, car il peut permettre de modifier le
58fonctionnement interne de httpd.</p>

104<h2><a name="basicconf" id="basicconf">Configuration de base</a></h2>
105
106<p>La directive de base pour le chargement du module est</p>
107
108<pre class="prettyprint lang-config">LoadModule lua_module modules/mod_lua.so</pre>
109
110
111<p>
112<code class="module"><a href="../mod/mod_lua.html">mod_lua</a></code> fournit un gestionnaire nommé
113<code>lua-script</code> qui peut être utilisé avec une directive
114<code class="directive"><a href="../mod/mod_mime.html#addhandler">AddHandler</a></code> ou <code class="directive"><a href="../mod/core.html#sethandler">SetHandler</a></code> :</p>
115
116<pre class="prettyprint lang-config">&lt;Files "*.lua"&gt;
117    SetHandler lua-script
118&lt;/Files&gt;</pre>
119
120
121<p>
122Ceci aura pour effet de faire traiter les requêtes pour les fichiers
123dont l'extension est <code>.lua</code> par <code class="module"><a href="../mod/mod_lua.html">mod_lua</a></code> en
124invoquant cette fonction de <code>gestion</code> de fichier.
125</p>
126
127<p>Pour plus de détails, voir la directive
128<code class="directive">LuaMapHandler</code>.

132<h2><a name="writinghandlers" id="writinghandlers">Ecrire des gestionnaires</a></h2>
133<p>Dans l'API du serveur HTTP Apache, un gestionnaire est une sorte de
134point d'accroche (hook) spécifique responsable de la génération de la
135réponse. <code class="module"><a href="../mod/mod_proxy.html">mod_proxy</a></code>, <code class="module"><a href="../mod/mod_cgi.html">mod_cgi</a></code> et
136<code class="module"><a href="../mod/mod_status.html">mod_status</a></code> sont des exemples de modules comportant un
137gestionnaire.</p>
138
139<p><code class="module"><a href="../mod/mod_lua.html">mod_lua</a></code> cherche toujours à invoquer une fonction Lua pour le
140gestionnaire, plutôt que de simplement évaluer le corps d'un script dans
141le style de CGI. Une fonction de gestionnaire se présente comme suit :</p>
142
143
144<pre class="prettyprint lang-lua">
145<strong>example.lua</strong><br />
146-- exemple de gestionnaire
147
148require "string"
149
150--[[
151     Il s'agit du nom de méthode par défaut pour les gestionnaires Lua ;
152     voir les noms de fonctions optionnels dans la directive
153     LuaMapHandler pour choisir un point d'entrée différent.
154--]]
155function handle(r)
156    r.content_type = "text/plain"
157
158    if r.method == 'GET' then
159    	r:puts("Hello Lua World!\n")
160        for k, v in pairs( r:parseargs() ) do
161            r:puts( string.format("%s: %s\n", k, v) )
162        end
163    elseif r.method == 'POST' then
164    	r:puts("Hello Lua World!\n")
165        for k, v in pairs( r:parsebody() ) do
166            r:puts( string.format("%s: %s\n", k, v) )
167        end
168    else
169    elseif r.method == 'PUT' then
170-- message d'erreur personnalisé
171        r:puts("Unsupported HTTP method " .. r.method)
172	r.status = 405
173        return apache2.OK
174    else
175-- message d'erreur ErrorDocument
176        return 501
177    end
178    return apache2.OK
179end</pre>
180
181
182<p>
183Ce gestionnaire se contente d'afficher les arguments codés d'un uri ou
184d'un formulaire dans un page au format texte.
185</p>
186
187<p>
188Cela signifie que vous pouvez (et êtes encouragé à) avoir plusieurs
189gestionnaires (ou points d'entrée, ou filtres) dans le même script.
190</p>

194<h2><a name="writingauthzproviders" id="writingauthzproviders">Ecriture de fournisseurs d'autorisation</a></h2>
195
196
197<p><code class="module"><a href="../mod/mod_authz_core.html">mod_authz_core</a></code> fournit une interface d'autorisation
198de haut niveau bien plus facile à utiliser que dans les hooks
199correspondants. Le premier argument de la directive <code class="directive"><a href="../mod/mod_authz_core.html#require">Require</a></code> permet de spécifier le
200fournisseur d'autorisation à utiliser. Pour chaque directive <code class="directive"><a href="../mod/mod_authz_core.html#require">Require</a></code>,
201<code class="module"><a href="../mod/mod_authz_core.html">mod_authz_core</a></code> appellera le fournisseur d'autorisation
202spécifié, le reste de la ligne constituant les paramètres. Le
203fournisseur considéré va alors vérifier les autorisations et fournir le
204résultat dans une valeur de retour.</p>
205
206<p>En général, le fournisseur authz est appelé avant l'authentification.
207S'il doit connaître le nom d'utilisateur authentifié (ou si
208l'utilisateur est appelé à être authentifié), le fournisseur doit
209renvoyer <code>apache2.AUTHZ_DENIED_NO_USER</code>, ce qui va
210déclancher le processus d'authentification et un deuxième appel du
211fournisseur authz.</p>
212
213<p>La fonction du fournisseur authz ci-dessous accepte deux arguments,
214une adresse IP et un nom d'utilisateur. Elle autorise l'accès dans le
215cas où la requête provient de l'adresse IP spécifiée, ou si
216l'utilisateur authentifié correspond au second argument :</p>
217
218<pre class="prettyprint lang-lua">
219<strong>authz_provider.lua</strong><br />
220
221require 'apache2'
222
223function authz_check_foo(r, ip, user)
224    if r.useragent_ip == ip then
225        return apache2.AUTHZ_GRANTED
226    elseif r.user == nil then
227        return apache2.AUTHZ_DENIED_NO_USER
228    elseif r.user == user then
229        return apache2.AUTHZ_GRANTED
230    else
231        return apache2.AUTHZ_DENIED
232    end
233end</pre>
234
235
236<p>La configuration suivante enregistre cette fonction en tant que
237fournisseur <code>foo</code>, et la configure por l'URL <code>/</code> :</p>
238<pre class="prettyprint lang-config">LuaAuthzProvider foo authz_provider.lua authz_check_foo
239&lt;Location "/"&gt;
240  Require foo 10.1.2.3 john_doe
241&lt;/Location&gt;</pre>

343<p>Les fonctions d'accroche reçoivent l'objet de la requête comme seul
344argument (sauf LuaAuthzProvider qui reçoit aussi des arguments en
345provenance de la directive Require). Elles peuvent renvoyer une valeur,
346selon la fonction, mais il s'agit en général d'un
347code d'état HTTP ou des valeurs OK, DONE, ou DECLINED,
348que vous pouvez écrire dans Lua sous la forme <code>apache2.OK</code>,
349<code>apache2.DONE</code>, ou <code>apache2.DECLINED</code>.</p>
350
351
352<pre class="prettyprint lang-lua">
353<strong>translate_name.lua</strong><br />
354-- exemple d'accroche qui réécrit un URI en chemin du système de fichiers.
355
356require 'apache2'
357
358function translate_name(r)
359    if r.uri == "/translate-name" then
360        r.filename = r.document_root .. "/find_me.txt"
361        return apache2.OK
362    end
363    -- on ne gère pas cette URL et on donne sa chance à un autre module
364    return apache2.DECLINED
365end</pre>
366
367
368
369<pre class="prettyprint lang-lua">
370<strong>translate_name2.lua</strong><br />
371--[[ exemple d'accroche qui réécrit un URI vers un autre URI. Il renvoie
372	un apache2.DECLINED pour permettre à un autre interpréteur d'URL de
373	travailler sur la substitution, y compris l'accroche translate_name
374	de base dont les tables de correspondances se basent sur DocumentRoot.
375
376     Note: utilisez le drapeau early/late de la directive pour
377     l'exécuter avant ou après mod_alias.
378--]]
379
380require 'apache2'
381
382function translate_name(r)
383    if r.uri == "/translate-name" then
384        r.uri = "/find_me.txt"
385        return apache2.DECLINED
386    end
387    return apache2.DECLINED
388end</pre>

395<dt>request_rec</dt>
396        <dd>
397        <p>request_rec est considérée en tant que donnée utilisateur.
398	Elle possède une métatable qui vous permet d'accomplir des

404	distinctes).</p>
405
406	<table class="bordered"><tr class="header">
407          <th><strong>Nom</strong></th>
408          <th><strong>Type Lua</strong></th>
409          <th><strong>Modifiable</strong></th>
410	  <th><strong>Description</strong></th>
411        </tr>

413          <td><code>allowoverrides</code></td>

1128<h2><a name="modifying_buckets" id="modifying_buckets">Modification de contenu avec les filtres lua</a></h2>
1129    
1130    <p>
1131    Les fonctions de filtrage implémentées via les directives <code class="directive"><a href="#luainputfilter">LuaInputFilter</a></code> ou <code class="directive"><a href="#luaoutputfilter">LuaOutputFilter</a></code> sont conçues comme des
1132    fonctions de 3ème phase non blocantes utilisant des sous-routines
1133    pour suspendre et reprendre l'exécution d'une fonction lorsque des
1134    paquets de données sont envoyés à la chaîne de filtrage. La
1135    structure de base d'une telle fonction est :
1136    </p>
1137    <pre class="prettyprint lang-lua">function filter(r)
1138    -- Nous indiquons tout d'abord que nous sommes prêts à recevoir des
1139    -- blocs de données.
1140    -- Avant ceci, nous pouvons définir notre environnement, tester
1141    -- certaines conditions, et, si nous le jugeons nécessaire, refuser le
1142    -- filtrage d'une requête :
1143    if something_bad then
1144        return -- Le filtrage est sauté
1145    end
1146    -- Sans se préoccuper des données que nous devons éventuellement ajouter, un arrêt est réalisé ici.
1147    -- Noter que les filtres de sortie sont les seuls capables d'ajouter des éléments au début des données.
1148    -- Les filtres en entrée peuvent ajouter des éléments à la fin des données au stade final.
1149
1150    coroutine.yield([optional header to be prepended to the content])
1151
1152    -- Après cet arrêt, nous allons recevoir d'autres blocs de données, un par un ;
1153    -- nous pouvons les traiter comme il nous plaît et procéder à la réponse.
1154    -- Ces blocs sont conservés dans la variable globale 'bucket', nous réalisons donc
1155    -- une boucle pour vérifier que 'bucket' n'est pas vide :
1156    while bucket ~= nil do
1157        local output = mangle(bucket) -- Do some stuff to the content
1158        coroutine.yield(output) -- Return our new content to the filter chain
1159    end
1160
1161    -- Une fois les blocs de données épuisés, 'bucket' est positionné à une valeur vide ('nil'),
1162    -- ce qui va nous faire sortir de cette boucle et nous amener à l'étape suivante.
1163    -- On peut ajouter ce qu'on veut à la fin des données à cette étape, qui constitue le dernier
1164    -- arrêt. Les filtres d'entrée comme de sortie peuvent servir à ajouter des éléments à la fin
1165    --  des données à cette étape.
1166    coroutine.yield([optional footer to be appended to the content])
1167end</pre>

1320pas avoir trop de connexions vers la base de données inutilisées. Les

1333    <h3><a name="database_caveat" id="database_caveat">Précautions à prendre lorsque l'on travaille avec les bases
1334de données</a></h3>
1335    
1336    <p>Bien que les fonctions <code>query</code> et <code>run</code>
1337soient toujours disponibles, il est recommandé d'utiliser des requêtes
1338préparées chaque fois que possible, afin d'une part d'optimiser les
1339performances (si votre connexion reste longtemps en vie), et d'autre part
1340minimiser le risque d'attaques par injection SQL. Les fonctions
1341<code>run</code> et <code>query</code> ne doivent être utilisées que
1342lorsque la requête ne contient pas de variables (requête statique). Dans
1343le cas des requêtes dynamiques, utilisez <code>db:prepare</code> ou
1344<code>db:prepared</code>.
1345    </p>

1765<div class="directive-section"><h2><a name="luainputfilter" id="luainputfilter">Directive</a> <a name="LuaInputFilter" id="LuaInputFilter">LuaInputFilter</a></h2>
1766<table class="directive">
1767<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Fournit une fonction Lua pour le filtrage en entrée</td></tr>
1768<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>LuaInputFilter filter_name /path/to/lua/script.lua function_name</code></td></tr>
1769<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>configuration globale</td></tr>
1770<tr><th><a href="directive-dict.html#Status">Statut:</a></th><td>Extension</td></tr>
1771<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_lua</td></tr>
1772<tr><th><a href="directive-dict.html#Compatibility">Compatibilité:</a></th><td>Disponible depuis la version 2.4.5 du serveur HTTP
1773Apache</td></tr>
1774</table>
1775<p>Cette directive permet d'ajouter un filtre en entrée sous la forme
1776d'une fonction Lua. A l'instar des filtres en sorties, les filtres en
1777entrée fonctionnent comme des sous-routines, intervenant dans un premier
1778temps avant l'envoi du contenu des tampons, puis chaque fois qu'un
1779paquet de données doit être transmis à la chaîne, et éventuellement
1780produisant toute donnée à ajouter aux données en entrée. La variable
1781globale <code>bucket</code> contient les paquets de données tels qu'ils
1782sont transmis au script Lua :
1783</p>
1784
1785<pre class="prettyprint lang-config">LuaInputFilter myInputFilter "/www/filter.lua" input_filter
1786&lt;Files "*.lua"&gt;
1787  SetInputFilter myInputFilter
1788&lt;/Files&gt;</pre>
1789
1790<pre class="prettyprint lang-lua">--[[
1791    Exemple de filtre en entrée qui convertit toutes les données POST en
1792    majuscules.
1793]]--
1794function input_filter(r)
1795    print("luaInputFilter called") -- pour débogage
1796    coroutine.yield() -- attend des paquets de données
1797    while bucket do -- Pour chaque paquet, faire ...
1798        local output = string.upper(bucket) -- Convertit toutes les données POST en majuscules
1799        coroutine.yield(output) -- Envoie les données traitées à la chaîne de filtrage
1800    end
1801    -- plus aucune donnée à traiter.
1802    coroutine.yield("&amp;filterSignature=1234") -- Ajoute une signature à la fin
1803end</pre>
1804
1805<p>
1806Le filtre en entrée peut interdire ou sauter un filtre s'il est
1807considéré comme indésirable :
1808</p>
1809<pre class="prettyprint lang-lua">function input_filter(r)
1810    if not good then
1811        return -- Empêche tout simplement le filtrage et transmet le contenu original
1812    end
1813    coroutine.yield() -- attend des paquets de données
1814    ...               -- insert les filtres ici
1815end</pre>
1816

1824<div class="directive-section"><h2><a name="luamaphandler" id="luamaphandler">Directive</a> <a name="LuaMapHandler" id="LuaMapHandler">LuaMapHandler</a></h2>
1825<table class="directive">
1826<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Met en correspondance un chemin avec un gestionnaire lua</td></tr>
1827<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>LuaMapHandler modele-uri /chemin/vers/lua/script.lua
1828[nom-fonction]</code></td></tr>
1829<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>configuration globale, serveur virtuel, répertoire, .htaccess</td></tr>
1830<tr><th><a href="directive-dict.html#Override">Surcharges autorisées:</a></th><td>All</td></tr>
1831<tr><th><a href="directive-dict.html#Status">Statut:</a></th><td>Extension</td></tr>

1848<pre class="prettyprint lang-config">LuaMapHandler "/bingo" "/scripts/wombat.lua"</pre>
1849

2000<div class="directive-section"><h2><a name="luascope" id="luascope">Directive</a> <a name="LuaScope" id="LuaScope">LuaScope</a></h2>
2001<table class="directive">
2002<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Une valeur parmi once, request, conn, thread -- la valeur par défaut est once</td></tr>
2003<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>LuaScope once|request|conn|thread|server [min] [max]</code></td></tr>
2004<tr><th><a href="directive-dict.html#Default">Défaut:</a></th><td><code>LuaScope once</code></td></tr>
2005<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>configuration globale, serveur virtuel, répertoire, .htaccess</td></tr>
2006<tr><th><a href="directive-dict.html#Override">Surcharges autorisées:</a></th><td>All</td></tr>
2007<tr><th><a href="directive-dict.html#Status">Statut:</a></th><td>Extension</td></tr>
2008<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_lua</td></tr>
2009</table>
2010    <p>Cette directive permet de spécifier la durée de vie de
2011    l'interpréteur Lua qui sera utilisé dans ce "répertoire". La valeur
2012    par défaut est "once".</p>
2013
2014   <dl>
2015    <dt>once:</dt> <dd>utilise l'interpréteur une fois.</dd>
2016
2017    <dt>request:</dt> <dd>utilise l'interpréteur pour traiter tout ce
2018    qui est basé sur le même fichier dans la requête, et qui se trouve
2019    aussi dans la portée de la requête.</dd>
2020
2021    <dt>conn:</dt> <dd>idem request, mais attaché à connection_rec</dd>
2022
2023    <dt>thread:</dt> <dd>Utilise l'interpréteur pendant toute la durée
2024    de vie du thread qui traite la requête (disponible seulement avec
2025    les MPMs threadés).</dd>
2026
2027    <dt>server:</dt>  <dd>Le comportement est ici différent, car la
2028    portée du serveur présente une durée de vie assez longue, et
2029    plusieurs threads vont partager le même server_rec. Pour gérer tout
2030    ceci, les états lua du serveur sont stockés dans une liste de ressources
2031    apr. Les arguments <code>min</code> et <code>max</code> permettent
2032    de spécifier les nombres minimaux et maximaux d'états lua à stocker
2033    dans la liste.</dd>
2034   </dl>
2035   <p>En général, les portées <code>thread</code> et <code>server</code>
2036   sont 2 à 3 fois plus rapides que les autres, car elles n'ont pas besoin
2037   de régénérer de nouveaux états Lua à chaque requête (comme c'est le
2038   cas avec le MPM event, où même les connexions persistantes utilisent un
2039   nouveau thread pour chaque requête). Si vous pensez que vos scripts
2040   n'auront pas de problème s'il réutilisent un état, alors les portées
2041   <code>thread</code> ou <code>server</code> doivent être utilisées car
2042   elles présenteront de meilleures performances. Alors que la portée
2043   <code>thread</code> fournira les réponses les plus rapides, la portée
2044   <code>server</code> utilisera moins de mémoire car les états sont
2045   rassemblés dans des jeux, permettant par exemple à 1000 threads de
2046   partager 100 états Lua, ne nécessitant ainsi que 10% de la mémoire
2047   requise par la portée <code>thread</code>.
2048    </p>
