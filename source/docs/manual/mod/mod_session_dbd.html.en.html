37<h3>Summary</h3>
38
39    <div class="warning"><h3>Warning</h3>
40      <p>The session modules make use of HTTP cookies, and as such can fall
41      victim to Cross Site Scripting attacks, or expose potentially private
42      information to clients. Please ensure that the relevant risks have
43      been taken into account before enabling the session functionality on
44      your server.</p>
45    </div>
46
47    <p>This submodule of <code class="module"><a href="../mod/mod_session.html">mod_session</a></code> provides support for the
48    storage of user sessions within a SQL database using the
49    <code class="module"><a href="../mod/mod_dbd.html">mod_dbd</a></code> module.</p>
50
51    <p>Sessions can either be <strong>anonymous</strong>, where the session is
52    keyed by a unique UUID string stored on the browser in a cookie, or
53    <strong>per user</strong>, where the session is keyed against the userid of
54    the logged in user.</p>
55
56    <p>SQL based sessions are hidden from the browser, and so offer a measure of
57    privacy without the need for encryption.</p>
58
59    <p>Different webservers within a server farm may choose to share a database,
60    and so share sessions with one another.</p>
61
62    <p>For more details on the session interface, see the documentation for
63    the <code class="module"><a href="../mod/mod_session.html">mod_session</a></code> module.</p>

92<h2><a name="dbdconfig" id="dbdconfig">DBD Configuration</a></h2>
93
94      <p>Before the <code class="module"><a href="../mod/mod_session_dbd.html">mod_session_dbd</a></code> module can be configured to maintain a
95      session, the <code class="module"><a href="../mod/mod_dbd.html">mod_dbd</a></code> module must be configured to make the various database queries
96      available to the server.</p>
97
98      <p>There are four queries required to keep a session maintained, to select an existing session,
99      to update an existing session, to insert a new session, and to delete an expired or empty
100      session. These queries are configured as per the example below.</p>
101
102      <div class="example"><h3>Sample DBD configuration</h3><pre class="prettyprint lang-config">DBDriver pgsql
103DBDParams "dbname=apachesession user=apache password=xxxxx host=localhost"
104DBDPrepareSQL "delete from session where key = %s" deletesession
105DBDPrepareSQL "update session set value = %s, expiry = %lld, key = %s where key = %s" updatesession
106DBDPrepareSQL "insert into session (value, expiry, key) values (%s, %lld, %s)" insertsession
107DBDPrepareSQL "select value from session where key = %s and (expiry = 0 or expiry &gt; %lld)" selectsession
108DBDPrepareSQL "delete from session where expiry != 0 and expiry &lt; %lld" cleansession</pre>
109</div>

113<h2><a name="anonymous" id="anonymous">Anonymous Sessions</a></h2>
114
115      <p>Anonymous sessions are keyed against a unique UUID, and stored on the
116      browser within an HTTP cookie. This method is similar to that used by most
117      application servers to store session information.</p>
118
119      <p>To create a simple anonymous session and store it in a postgres database
120      table called <var>apachesession</var>, and save the session ID in a cookie
121      called <var>session</var>, configure the session as follows:</p>
122
123      <div class="example"><h3>SQL based anonymous session</h3><pre class="prettyprint lang-config">Session On
124SessionDBDCookieName session path=/</pre>
125</div>
126
127      <p>For more examples on how the session can be configured to be read
128      from and written to by a CGI application, see the
129      <code class="module"><a href="../mod/mod_session.html">mod_session</a></code> examples section.</p>
130
131      <p>For documentation on how the session can be used to store username
132      and password details, see the <code class="module"><a href="../mod/mod_auth_form.html">mod_auth_form</a></code> module.</p>
133

136<h2><a name="peruser" id="peruser">Per User Sessions</a></h2>
137
138      <p>Per user sessions are keyed against the username of a successfully
139      authenticated user. It offers the most privacy, as no external handle
140      to the session exists outside of the authenticated realm.</p>
141
142      <p>Per user sessions work within a correctly configured authenticated
143      environment, be that using basic authentication, digest authentication
144      or SSL client certificates. Due to the limitations of who came first,
145      the chicken or the egg, per user sessions cannot be used to store
146      authentication credentials from a module like
147      <code class="module"><a href="../mod/mod_auth_form.html">mod_auth_form</a></code>.</p>
148
149      <p>To create a simple per user session and store it in a postgres database
150      table called <var>apachesession</var>, and with the session keyed to the
151      userid, configure the session as follows:</p>
152
153      <div class="example"><h3>SQL based per user session</h3><pre class="prettyprint lang-config">Session On
154SessionDBDPerUser On</pre>
155</div>

159<h2><a name="housekeeping" id="housekeeping">Database Housekeeping</a></h2>
160      <p>Over the course of time, the database can be expected to start accumulating
161      expired sessions. At this point, the <code class="module"><a href="../mod/mod_session_dbd.html">mod_session_dbd</a></code> module
162      is not yet able to handle session expiry automatically.</p>
163
164      <div class="warning"><h3>Warning</h3>
165      <p>The administrator will need to set up an external process via cron to clean
166      out expired sessions.</p>
167      </div>

233    containing the session ID will be removed from the headers during request processing.</p>
234
235    <p>In a reverse proxy situation where the Apache server acts as a server frontend for
236    a backend origin server, revealing the contents of the session ID cookie to the backend
237    could be a potential privacy violation. When set to on, the session ID cookie will be
238    removed from the incoming HTTP headers.</p>
