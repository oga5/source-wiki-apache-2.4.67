37<h3>Summary</h3>
38
39        <p>This module provides HTTP/2 (<a href="https://tools.ietf.org/html/rfc7540">RFC 7540</a>) 
40        support for the Apache HTTP Server.</p>
41        
42        <p>This module relies on <a href="http://nghttp2.org/">libnghttp2</a>
43        to provide the core http/2 engine.</p>
44        
45        <p>You must enable HTTP/2 via <code class="directive"><a href="../mod/core.html#protocols">Protocols</a></code>
46        in order to use the functionality described in this document. The
47        HTTP/2 protocol <a href="https://http2.github.io/faq/#does-http2-require-encryption">does not require</a> the use of encryption so two schemes are available:
48        <code>h2</code> (HTTP/2 over TLS) and <code>h2c</code> (HTTP/2 over TCP).</p>
49
50        <p>Two useful configuration schemes are:</p>
51        
52        <div class="note"><h3>HTTP/2 in a VirtualHost context (TLS only)</h3>
53        <pre class="prettyprint lang-config">Protocols h2 http/1.1</pre>
54
55        <p>Allows HTTP/2 negotiation (h2) via TLS ALPN in a secure
56        <code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code>.
57        HTTP/2 preamble checking (Direct mode, see <code class="directive"><a href="#h2direct">H2Direct</a></code>) is disabled by default for <code>h2</code>.</p>
58        </div>
59
60        <div class="note"><h3>HTTP/2 in a Server context (TLS and cleartext)</h3>
61        <pre class="prettyprint lang-config">Protocols h2 h2c http/1.1</pre>
62
63        <p>Allows HTTP/2 negotiation (h2) via TLS ALPN for secure
64        <code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code>. Allows
65        HTTP/2 cleartext negotiation (h2c) upgrading from an initial HTTP/1.1
66        connection or via HTTP/2 preamble checking (Direct mode, see
67        <code class="directive"><a href="#h2direct">H2Direct</a></code>).</p>
68        </div>
69
70        <p>Refer to the official <a href="https://http2.github.io/faq">HTTP/2 FAQ</a>
71        for any doubt about the protocol.</p>

114    <h3><a name="dimensioning" id="dimensioning">HTTP/2 Dimensioning</a></h3>
115        <p>
116            Enabling HTTP/2 on your Apache Server has impact on the resource
117            consumption and if you have a busy site, you may need to consider
118            carefully the implications.
119        </p>
120        <p>
121            The first noticeable thing after enabling HTTP/2 is that your server
122            processes will start additional threads. The reason for this is that
123            HTTP/2 gives all requests that it receives to its own <em>Worker</em>
124            threads for processing, collects the results and streams them out
125            to the client.
126        </p>
127        <p>
128            In the current implementation, these workers use a separate thread
129            pool from the MPM workers that you might be familiar with. This is
130            just how things are right now and not intended to be like this forever.
131            (It might be forever for the 2.4.x release line, though.) So, HTTP/2
132            workers, or shorter H2Workers, will not show up in <code class="module"><a href="../mod/mod_status.html">mod_status</a></code>. They
133            are also not counted against directives such as <code class="directive"><a href="../mod/mpm_common.html#threadsperchild">ThreadsPerChild</a></code>. However
134            they take <code class="directive"><a href="../mod/mpm_common.html#threadsperchild">ThreadsPerChild</a></code>
135            as default if you have not configured something
136            else via <code class="directive"><a href="#h2minworkers">H2MinWorkers</a></code> and
137            <code class="directive"><a href="#h2maxworkers">H2MaxWorkers</a></code>.
138        </p>
139        <p>
140            Another thing to watch out for is is memory consumption. Since HTTP/2
141            keeps more state on the server to manage all the open request, priorities
142            for and dependencies between them, it will always need more memory
143            than HTTP/1.1 processing. There are three directives which steer the
144            memory footprint of a HTTP/2 connection:
145            <code class="directive"><a href="#h2maxsessionstreams">H2MaxSessionStreams</a></code>,
146            <code class="directive"><a href="#h2windowsize">H2WindowSize</a></code> and
147            <code class="directive"><a href="#h2streammaxmemsize">H2StreamMaxMemSize</a></code>.
148        </p>
149        <p>
150            <code class="directive"><a href="#h2maxsessionstreams">H2MaxSessionStreams</a></code> limits the
151            number of parallel requests that a client can make on a HTTP/2 connection.
152            It depends on your site how many you should allow. The default is 100 which
153            is plenty and unless you run into memory problems, I would keep it this
154            way. Most requests that browsers send are GETs without a body, so they
155            use up only a little bit of memory until the actual processing starts.
156        </p>
157        <p>
158            <code class="directive"><a href="#h2windowsize">H2WindowSize</a></code> controls how much
159            the client is allowed to send as body of a request, before it waits
160            for the server to encourage more. Or, the other way around, it is the
161            amount of request body data the server needs to be able to buffer. This
162            is per request.
163        </p>
164        <p>
165            And last, but not least, <code class="directive"><a href="#h2streammaxmemsize">H2StreamMaxMemSize</a></code>
166            controls how much response data shall be buffered. The request sits in
167            a H2Worker thread and is producing data, the HTTP/2 connection tries
168            to send this to the client. If the client does not read fast enough,
169            the connection will buffer this amount of data and then suspend the
170            H2Worker.
171        </p>

174    <h3><a name="misdirected" id="misdirected">Multiple Hosts and Misdirected Requests</a></h3>
175        <p>
176            Many sites use the same TLS certificate for multiple virtual hosts. The 
177            certificate either has a wildcard name, such as '*.example.org' or carries
178            several alternate names. Browsers using HTTP/2 will recognize that and reuse
179            an already opened connection for such hosts.
180        </p>
181        <p>
182            While this is great for performance, it comes at a price: such vhosts 
183            need more care in their configuration. The problem is that you will have
184            multiple requests for multiple hosts on the same TLS connection. And that
185            makes renegotiation impossible, in face the HTTP/2 standard forbids it.
186        </p>
187        <p>
188            So, if you have several virtual hosts using the same certificate and
189            want to use HTTP/2 for them, you need to make sure that all vhosts have
190            exactly the same SSL configuration. You need the same protocol, 
191            ciphers and settings for client verification.
192        </p>
193        <p>
194            If you mix things, Apache httpd will detect it and return a special
195            response code, 421 Misdirected Request, to the client.
196        </p>

235            <p>
236                This directive influences how file content is handled in
237                responses. When <code>off</code>, which is the default, file handles
238                are passed from the requestion processing down to the main
239                connection, using the usual Apache setaside handling for
240                managing the lifetime of the file.
241            </p>
242            <p>
243                When set to <code>on</code>, file content is copied while the
244                request is still being processed and the buffered data is passed
245                on to the main connection. This is better if a third party
246                module is injecting files with different lifetimes into the response. 
247            </p>
248            <p>
249                An example for such a module is <code>mod_wsgi</code> that may place
250                Python file handles into the response. Those files get close down when
251                Python thinks processing has finished. That may be well before
252                <code class="module"><a href="../mod/mod_http2.html">mod_http2</a></code> is done with them.
253            </p>

257<div class="directive-section"><h2><a name="H2Direct" id="H2Direct">H2Direct</a> <a name="h2direct" id="h2direct">Directive</a></h2>
258<table class="directive">
259<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>H2 Direct Protocol Switch</td></tr>
260<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>H2Direct on|off</code></td></tr>
261<tr><th><a href="directive-dict.html#Default">Default:</a></th><td><code>H2Direct on for h2c, off for h2 protocol</code></td></tr>
262<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config, virtual host</td></tr>
263<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Extension</td></tr>
264<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_http2</td></tr>
265</table>
266            <p>
267                This directive toggles the usage of the HTTP/2 Direct Mode. This
268                should be used inside a 
269                <code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code> 
270                section to enable direct HTTP/2 communication for that virtual host. 
271            </p>
272            <p>
273                Direct communication means that if the first bytes received by the 
274                server on a connection match the HTTP/2 preamble, the HTTP/2
275                protocol is switched to immediately without further negotiation.
276                This mode is defined in RFC 7540 for the cleartext (h2c) case. Its
277                use on TLS connections not mandated by the standard.
278            </p>
279            <p>
280                When a server/vhost does not have h2 or h2c enabled via
281                <code class="directive"><a href="../mod/core.html#protocols">Protocols</a></code>,
282                the connection is never inspected for a HTTP/2 preamble.
283                <code class="directive">H2Direct</code>
284                does not matter then. This is important for connections that
285                use protocols where an initial read might hang indefinitely, such
286                as NNTP.
287            </p>
288            <p>
289                For clients that have out-of-band knowledge about a server
290                supporting h2c, direct HTTP/2 saves the client from having to
291                perform an HTTP/1.1 upgrade, resulting in better performance
292                and avoiding the Upgrade restrictions on request bodies.
293            </p>
294            <p>
295                This makes direct h2c attractive for server to server communication
296                as well, when the connection can be trusted or is secured by other means.
297            </p>

621<div class="directive-section"><h2><a name="H2Push" id="H2Push">H2Push</a> <a name="h2push" id="h2push">Directive</a></h2>
622<table class="directive">
623<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>H2 Server Push Switch</td></tr>
624<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>H2Push on|off</code></td></tr>
625<tr><th><a href="directive-dict.html#Default">Default:</a></th><td><code>H2Push on</code></td></tr>
626<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config, virtual host, directory, .htaccess</td></tr>
627<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Extension</td></tr>
628<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_http2</td></tr>
629<tr><th><a href="directive-dict.html#Compatibility">Compatibility:</a></th><td>Available in version 2.4.18 and later.</td></tr>
630</table>
631            <p>
632                This directive toggles the usage of the HTTP/2 server push 
633                protocol feature.
634            </p>
635            <p>
636                The HTTP/2 protocol allows the server to push other resources to
637                a client when it asked for a particular one. This is helpful
638                if those resources are connected in some way and the client can
639                be expected to ask for it anyway. The pushing then saves the
640                time it takes the client to ask for the resources itself. On the
641                other hand, pushing resources the client never needs or already
642                has is a waste of bandwidth.
643            </p>
644            <p>
645                Server pushes are detected by inspecting the <code>Link</code> headers of
646                responses (see https://tools.ietf.org/html/rfc5988 for the 
647                specification). When a link thus specified has the <code>rel=preload</code>
648                attribute, it is treated as a resource to be pushed.
649            </p>
650            <p> 
651                Link headers in responses are either set by the application or
652                can be configured via <code class="directive"><a href="#h2pushresource">H2PushResource</a></code> or
653                using <code class="module"><a href="../mod/mod_headers.html">mod_headers</a></code> as:
654            </p>
655            <div class="example"><h3>mod_headers example</h3><pre class="prettyprint lang-config">&lt;Location /index.html&gt;
656    Header add Link "&lt;/css/site.css&gt;;rel=preload"
657    Header add Link "&lt;/images/logo.jpg&gt;;rel=preload"
658&lt;/Location&gt;</pre>
659</div>
660            <p>
661                As the example shows, there can be several link headers added
662                to a response, resulting in several pushes being triggered. There
663                are no checks in the module to avoid pushing the same resource
664                twice or more to one client. Use with care.
665            </p>
666            <p> 
667                HTTP/2 server pushes are enabled by default. On a server or virtual host,
668                you may enable/disable this feature for any connection to the host. In addition,
669                you may disable PUSH for a set of resources in a Directory/Location. This controls
670                which resources may cause a PUSH, not which resources may be sent via PUSH.
671            </p>
672            <div class="example"><h3>Example</h3><pre class="prettyprint lang-config">H2Push off</pre>
673</div>
674            <p>
675                Last but not least, pushes happen only when the client signals
676                its willingness to accept those. Most browsers do, some, like Safari 9,
677                do not. Also, pushes also only happen for resources from the same
678                <em>authority</em> as the original response is for.
679            </p>
680

719<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>H2PushPriority <var>mime-type</var> [after|before|interleaved] [<var>weight</var>]</code></td></tr>
720<tr><th><a href="directive-dict.html#Default">Default:</a></th><td><code>H2PushPriority * After 16</code></td></tr>
721<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config, virtual host</td></tr>
722<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Extension</td></tr>
723<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_http2</td></tr>
724<tr><th><a href="directive-dict.html#Compatibility">Compatibility:</a></th><td>Available in version 2.4.18 and later. For having an
725            effect, a nghttp2 library version 1.5.0 or newer is necessary.</td></tr>
726</table>
727            <p>
728                This directive defines the priority handling of pushed responses
729                based on the content-type of the response. This is usually defined
730                per server config, but may also appear in a virtual host. 
731            </p>
732            <p>
733                HTTP/2 server pushes are always related to a client request. Each
734                such request/response pairs, or <em>streams</em> have a dependency
735                and a weight, together defining the <em>priority</em> of a stream. 
736            </p>
737            <p>
738                When a stream <em>depends</em> on another, say X depends on Y,
739                then Y gets all bandwidth before X gets any. Note that this
740                does not mean that Y will block X. If Y has no data to send, 
741                all bandwidth allocated to Y can be used by X.
742            </p>
743            <p>
744                When a stream has more than one dependent, say X1 and X2 both
745                depend on Y, the <em>weight</em> determines the bandwidth
746                allocation. If X1 and X2 have the same weight, they both get
747                half of the available bandwidth. If the weight of X1 is twice
748                as large as that for X2, X1 gets twice the bandwidth of X2.
749            </p>
750            <p> 
751                Ultimately, every stream depends on the <em>root</em> stream which
752                gets all the bandwidth available, but never sends anything. So all
753                its bandwidth is distributed by weight among its children. Which
754                either have data to send or distribute the bandwidth to their
755                own children. And so on. If none of the children have data
756                to send, that bandwidth get distributed somewhere else according
757                to the same rules.
758            </p>
759            <p> 
760                The purpose of this priority system is to always make use of
761                available bandwidth while allowing precedence and weight
762                to be given to specific streams. Since, normally, all streams
763                are initiated by the client, it is also the one that sets
764                these priorities.
765            </p>
766            <p>
767                Only when such a stream results in a PUSH, gets the server to
768                decide what the <em>initial</em> priority of such a pushed
769                stream is. In the examples below, X is the client stream. It
770                depends on Y and the server decides to PUSH streams P1 and P2
771                onto X.
772            </p>
773            <p>
774                The default priority rule is:
775            </p>
776            <div class="example"><h3>Default Priority Rule</h3><pre class="prettyprint lang-config">H2PushPriority * After 16</pre>
777</div>
778            <p>
779                which reads as 'Send a pushed stream of any content-type
780                depending on the client stream with weight 16'. And so P1
781                and P2 will be send after X and, as they have equal weight,
782                share bandwidth equally among themselves.
783            </p>
784            <div class="example"><h3>Interleaved Priority Rule</h3><pre class="prettyprint lang-config">H2PushPriority text/css Interleaved 256</pre>
785</div>
786            <p>
787                which reads as 'Send any CSS resource on the same dependency and
788                weight as the client stream'. If P1 has content-type 'text/css',
789                it will depend on Y (as does X) and its effective weight will be
790                calculated as <code>P1ew = Xw * (P1w / 256)</code>. With P1w being 
791                256, this will make the effective weight the same as the weight
792                of X. If both X and P1 have data to send, bandwidth will be allocated
793                to both equally.
794            </p>
795            <p>
796                With Pw specified as 512, a pushed, interleaved stream would
797                get double the weight of X. With 128 only half as much. Note that
798                effective weights are always capped at 256.
799            </p>
800            <div class="example"><h3>Before Priority Rule</h3><pre class="prettyprint lang-config">H2PushPriority application/json Before</pre>
801</div>
802            <p>
803                This says that any pushed stream of content type 'application/json'
804                should be send out <em>before</em> X. This makes P1 dependent
805                on Y and X dependent on P1. So, X will be stalled as long as
806                P1 has data to send. The effective weight is inherited from the
807                client stream. Specifying a weight is not allowed.
808            </p>
809            <p>
810                Be aware that the effect of priority specifications is limited
811                by the available server resources. If a server does not have
812                workers available for pushed streams, the data for the stream
813                may only ever arrive when other streams have been finished.
814            </p>
815            <p>
816                Last, but not least, there are some specifics of the syntax
817                to be used in this directive:
818            </p>
819            <ol>
820                <li>'*' is the only special content-type that matches all others. 
821                    'image/*' will not work.</li>
822                <li>The default dependency is 'After'. </li>
823                <li>There are also default weights: for 'After' it is 16, 'interleaved' is 256. 
824                </li>
825            </ol>
826            <div class="example"><h3>Shorter Priority Rules</h3><pre class="prettyprint lang-config">H2PushPriority application/json 32         # an After rule
827H2PushPriority image/jpeg before           # weight inherited
828H2PushPriority text/css   interleaved      # weight 256 default</pre>
829</div>
