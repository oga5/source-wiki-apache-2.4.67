60<h2><a name="order" id="order">Order of Processing</a></h2>
61
62    <p>The directives provided by <code class="module"><a href="../mod/mod_headers.html">mod_headers</a></code> can
63    occur almost anywhere within the server configuration, and can be
64    limited in scope by enclosing them in <a href="../sections.html">configuration sections</a>.</p>
65
66    <p>Order of processing is important and is affected both by the
67    order in the configuration file and by placement in <a href="../sections.html#mergin">configuration sections</a>. These
68    two directives have a different effect if reversed:</p>
69
70    <pre class="prettyprint lang-config">RequestHeader append MirrorID "mirror 12"
71RequestHeader unset MirrorID</pre>
72
73
74    <p>This way round, the <code>MirrorID</code> header is not set. If
75    reversed, the MirrorID header is set to "mirror 12".</p>
76</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>

78<h2><a name="early" id="early">Early and Late Processing</a></h2>
79    <p><code class="module"><a href="../mod/mod_headers.html">mod_headers</a></code> can be applied either early or late
80    in the request.  The normal mode is late, when <em>Request</em> Headers are
81    set immediately before running the content generator and <em>Response</em>
82    Headers just as the response is sent down the wire.  Always use
83    Late mode in an operational server.</p>
84
85    <p>Early mode is designed as a test/debugging aid for developers.
86    Directives defined using the <code>early</code> keyword are set
87    right at the beginning of processing the request.  This means
88    they can be used to simulate different requests and set up test
89    cases, but it also means that headers may be changed at any time
90    by other modules before generating a Response.</p>
91
92    <p>Because early directives are processed before the request path's
93    configuration is traversed, early headers can only be set in a
94    main server or virtual host context.  Early directives cannot depend
95    on a request path, so they will fail in contexts such as
96    <code class="directive"><a href="../mod/core.html#directory">&lt;Directory&gt;</a></code> or
97    <code class="directive"><a href="../mod/core.html#location">&lt;Location&gt;</a></code>.</p>
98</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>

100<h2><a name="examples" id="examples">Examples</a></h2>
101
102    <ol>
103      <li>
104        Copy all request headers that begin with "TS" to the
105        response headers:
106
107        <pre class="prettyprint lang-config">Header echo ^TS</pre>
108
109      </li>
110
111      <li>
112        Add a header, <code>MyHeader</code>, to the response including a
113        timestamp for when the request was received and how long it
114        took to begin serving the request. This header can be used by
115        the client to intuit load on the server or in isolating
116        bottlenecks between the client and the server.
117
118        <pre class="prettyprint lang-config">Header set MyHeader "%D %t"</pre>
119
120
121        <p>results in this header being added to the response:</p>
122
123        <div class="example"><p><code>
124          MyHeader: D=3775428 t=991424704447256
125        </code></p></div>
126      </li>
127
128      <li>
129        Say hello to Joe
130
131        <pre class="prettyprint lang-config">Header set MyHeader "Hello Joe. It took %D microseconds for Apache to serve this request."</pre>
132
133
134        <p>results in this header being added to the response:</p>
135
136        <div class="example"><p><code>
137          MyHeader: Hello Joe. It took D=3775428 microseconds for Apache
138          to serve this request.
139        </code></p></div>
140      </li>
141
142      <li>
143        Conditionally send <code>MyHeader</code> on the response if and
144        only if header <code>MyRequestHeader</code> is present on the request.
145        This is useful for constructing headers in response to some client
146        stimulus. Note that this example requires the services of the
147        <code class="module"><a href="../mod/mod_setenvif.html">mod_setenvif</a></code> module.
148
149        <pre class="prettyprint lang-config">SetEnvIf MyRequestHeader myvalue HAVE_MyRequestHeader
150Header set MyHeader "%D %t mytext" env=HAVE_MyRequestHeader</pre>
151
152
153        <p>If the header <code>MyRequestHeader: myvalue</code> is present on
154        the HTTP request, the response will contain the following header:</p>
155
156        <div class="example"><p><code>
157          MyHeader: D=3775428 t=991424704447256 mytext
158        </code></p></div>
159      </li>
160
161      <li>
162        Enable DAV to work with Apache running HTTP through SSL hardware
163        (<a href="http://svn.haxx.se/users/archive-2006-03/0549.shtml">problem
164        description</a>) by replacing <var>https:</var> with
165        <var>http:</var> in the <var>Destination</var> header:
166
167        <pre class="prettyprint lang-config">RequestHeader edit Destination ^https: http: early</pre>
168
169      </li>
170
171      <li>
172        Set the same header value under multiple nonexclusive conditions,
173        but do not duplicate the value in the final header.
174        If all of the following conditions applied to a request (i.e.,
175        if the <code>CGI</code>, <code>NO_CACHE</code> and
176        <code>NO_STORE</code> environment variables all existed for the
177        request):
178
179        <pre class="prettyprint lang-config">Header merge Cache-Control no-cache env=CGI
180Header merge Cache-Control no-cache env=NO_CACHE
181Header merge Cache-Control no-store env=NO_STORE</pre>
182
183
184        <p>then the response would contain the following header:</p>
185
186        <div class="example"><p><code>
187          Cache-Control: no-cache, no-store
188        </code></p></div>
189
190        <p>If <code>append</code> was used instead of <code>merge</code>,
191        then the response would contain the following header:</p>
192
193        <div class="example"><p><code>
194          Cache-Control: no-cache, no-cache, no-store
195        </code></p></div>
196      </li>
197      <li>
198        Set a test cookie if and only if the client didn't send us a cookie
199        <pre class="prettyprint lang-config">Header set Set-Cookie testcookie "expr=-z %{req:Cookie}"</pre>
200
201      </li>
202      <li>
203        Append a Caching header for responses with a HTTP status code of 200
204        <pre class="prettyprint lang-config">Header append Cache-Control s-maxage=600 "expr=%{REQUEST_STATUS} == 200"</pre>
205
206      </li>
207
208    </ol>

225    <p>This directive can replace, merge or remove HTTP response
226    headers. The header is modified just after the content handler
227    and output filters are run, allowing outgoing headers to be
228    modified.</p>
229
230    <p> The optional <var>condition</var> argument determines which internal
231    table of responses headers this directive will operate against:
232    <code>onsuccess</code> (default, can be omitted) or <code>always</code>.
233    The difference between the two lists is that the headers contained in the
234    latter are added to the response even on error, and persisted across
235    internal redirects (for example, ErrorDocument handlers).
236
237    Note also that repeating this directive with both conditions makes sense in
238    some scenarios because <code>always</code> is not a superset of
239    <code>onsuccess</code> with respect to existing headers:</p>
240
241    <ul>
242       <li> You're adding a header to a locally generated non-success (non-2xx) response, such
243            as a redirect, in which case only the table corresponding to
244            <code>always</code> is used in the ultimate response.</li>
245       <li> You're modifying or removing a header generated by a CGI script
246            or by <code class="module"><a href="../mod/mod_proxy_fcgi.html">mod_proxy_fcgi</a></code>,
247            in which case the CGI scripts' headers are in the table corresponding to
248            <code>always</code> and not in the default table.</li>
249       <li> You're modifying or removing a header generated by some piece of
250            the server but that header is not being found by the default
251            <code>onsuccess</code> condition.</li>
252    </ul>
253
254    <p>This difference between <code>onsuccess</code> and <code>always</code> is
255    a feature that resulted as a consequence of how httpd internally stores
256    headers for a HTTP response, since it does not offer any "normalized" single
257    list of headers. The main problem that can arise if the following concept
258    is not kept in mind while writing the configuration is that some HTTP responses
259    might end up with the same header duplicated (confusing users or sometimes even
260    HTTP clients). For example, suppose that you have a simple PHP proxy setup with
261    <code class="module"><a href="../mod/mod_proxy_fcgi.html">mod_proxy_fcgi</a></code> and your backend PHP scripts adds the
262    <code>X-Foo: bar</code> header to each HTTP response. As described above,
263    <code class="module"><a href="../mod/mod_proxy_fcgi.html">mod_proxy_fcgi</a></code> uses the <code>always</code> table to store
264    headers, so a configuration like the following ends up in the wrong result, namely
265    having the header duplicated with both values:</p>
266
267    <pre class="prettyprint lang-config"># X-Foo's value is set in the 'onsuccess' headers table
268Header set X-Foo: baz</pre>
269    
270
271    <p>To circumvent this limitation, there are some known configuration
272    patterns that can help, like the following:</p>
273
274        <pre class="prettyprint lang-config"># 'onsuccess' can be omitted since it is the default
275Header onsuccess unset X-Foo
276Header always set X-Foo "baz"</pre>
277
278
279    <p>Separately from the <var>condition</var> parameter described above, you
280    can limit an action based on HTTP status codes for e.g. proxied or CGI
281    requests. See the example that uses %{REQUEST_STATUS} in the section above.</p>
282
283    <p>The action it performs is determined by the first
284    argument (second argument if a <var>condition</var> is specified).
285    This can be one of the following values:</p>
286
287    <div class="warning"><h3>Warning</h3>
288        <p>Please read the difference between <code>always</code>
289        and <code>onsuccess</code> headers list described above
290        before start reading the actions list, since that important
291        concept still applies. Each action, in fact, works as described
292        but only on the target headers list.</p>
293    </div>
294
295    <dl>
296    <dt><code>add</code></dt>
297    <dd>The response header is added to the existing set of headers,
298    even if this header already exists. This can result in two
299    (or more) headers having the same name. This can lead to
300    unforeseen consequences, and in general <code>set</code>,
301    <code>append</code> or <code>merge</code> should be used instead.</dd>
302
303    <dt><code>append</code></dt>
304    <dd>The response header is appended to any existing header of
305    the same name. When a new value is merged onto an existing
306    header it is separated from the existing header with a comma.
307    This is the HTTP standard way of giving a header multiple values.</dd>
308
309    <dt><code>echo</code></dt>
310    <dd>Request headers with this name are echoed back in the
311    response headers. <var>header</var> may be a
312    <a class="glossarylink" href="../glossary.html#regex" title="see glossary">regular expression</a>.
313    <var>value</var> must be omitted.</dd>
314
315    <dt><code>edit</code></dt>
316    <dt><code>edit*</code></dt>
317    <dd>If this response header exists, its value is transformed according
318    to a <a class="glossarylink" href="../glossary.html#regex" title="see glossary">regular expression</a>
319    search-and-replace.  The <var>value</var> argument is a <a class="glossarylink" href="../glossary.html#regex" title="see glossary">regular expression</a>, and the <var>replacement</var>
320    is a replacement string, which may contain backreferences or format specifiers.
321    The <code>edit</code> form will match and replace exactly once
322    in a header value, whereas the <code>edit*</code> form will replace
323    <em>every</em> instance of the search pattern if it appears more
324    than once.</dd>
325
326    <dt><code>merge</code></dt>
327    <dd>The response header is appended to any existing header of
328    the same name, unless the value to be appended already appears in the
329    header's comma-delimited list of values.  When a new value is merged onto
330    an existing header it is separated from the existing header with a comma.
331    This is the HTTP standard way of giving a header multiple values.
332    Values are compared in a case sensitive manner, and after
333    all format specifiers have been processed.  Values in double quotes
334    are considered different from otherwise identical unquoted values.</dd>
335
336    <dt><code>set</code></dt>
337    <dd>The response header is set, replacing any previous header
338    with this name. The <var>value</var> may be a format string.</dd>
339
340    <dt><code>setifempty</code></dt>
341    <dd>The request header is set, but only if there is no previous header
342    with this name.
343    <div class="note">
344    The Content-Type header is a special use case since there might be
345    the chance that its value have been determined but the header is not part
346    of the response when <code>setifempty</code> is evaluated.
347    It is safer to use <code>set</code> for this use case like in the
348    following example:
349    <pre class="prettyprint lang-config">Header set Content-Type "text/plain" "expr=-z %{CONTENT_TYPE}"</pre>
350
351    </div></dd>
352
353    <dt><code>unset</code></dt>
354    <dd>The response header of this name is removed, if it exists.
355    If there are multiple headers of the same name, all will be
356    removed. <var>value</var> must be omitted.</dd>
357
358    <dt><code>note</code></dt>
359    <dd>The value of the named response <var>header</var> is copied into an
360    internal note whose name is given by <var>value</var>.  This is useful
361    if a header sent by a CGI or proxied resource is configured to be unset
362    but should also be logged.<br />
363    Available in 2.4.7 and later.</dd>
364
365    </dl>
366
367    <p>This argument is followed by a <var>header</var> name, which
368    can include the final colon, but it is not required. Case is
369    ignored for <code>set</code>, <code>append</code>, <code>merge</code>,
370    <code>add</code>, <code>unset</code> and <code>edit</code>.
371    The <var>header</var> name for <code>echo</code>
372    is case sensitive and may be a <a class="glossarylink" href="../glossary.html#regex" title="see glossary">regular
373    expression</a>.</p>
374
375    <p>For <code>set</code>, <code>append</code>, <code>merge</code> and
376    <code>add</code> a <var>value</var> is specified as the next argument.
377    If <var>value</var>
378    contains spaces, it should be surrounded by double quotes.
379    <var>value</var> may be a character string, a string containing
380    <code class="module"><a href="../mod/mod_headers.html">mod_headers</a></code> specific format specifiers (and character
381    literals), or an <a href="../expr.html">ap_expr</a> expression prefixed
382    with <em>expr=</em></p>
383
384    <p> The following format specifiers are supported in <var>value</var>:</p>
385
386    <table class="bordered"><tr class="header"><th>Format</th><th>Description</th></tr>
387<tr><td><code>%%</code></td>
388        <td>The percent sign</td></tr>
389<tr class="odd"><td><code>%t</code></td>
390        <td>The time the request was received in Universal Coordinated Time
391        since the epoch (Jan. 1, 1970) measured in microseconds. The value
392        is preceded by <code>t=</code>.</td></tr>
393<tr><td><code>%D</code></td>
394        <td>The time from when the request was received to the time the
395        headers are sent on the wire. This is a measure of the duration
396        of the request. The value is preceded by <code>D=</code>.
397        The value is measured in microseconds.</td></tr>
398<tr class="odd"><td><code>%l</code></td>
399        <td>The current load averages of the actual server itself. It is
400        designed to expose the values obtained by <code>getloadavg()</code>
401        and this represents the current load average, the 5 minute average, and
402        the 15 minute average. The value is preceded by <code>l=</code> with each
403        average separated by <code>/</code>.<br />
404        Available in 2.4.4 and later.
405        </td></tr>
406<tr><td><code>%i</code></td>
407        <td>The current idle percentage of httpd (0 to 100) based on available
408        processes and threads. The value is preceded by <code>i=</code>.<br />
409        Available in 2.4.4 and later.
410        </td></tr>
411<tr class="odd"><td><code>%b</code></td>
412        <td>The current busy percentage of httpd (0 to 100) based on available
413        processes and threads. The value is preceded by <code>b=</code>.<br />
414        Available in 2.4.4 and later.
415        </td></tr>
416<tr><td><code>%{VARNAME}e</code></td>
417        <td>The contents of the <a href="../env.html">environment
418        variable</a> <code>VARNAME</code>.</td></tr>
419<tr class="odd"><td><code>%{VARNAME}s</code></td>
420        <td>The contents of the <a href="mod_ssl.html#envvars">SSL environment
421        variable</a> <code>VARNAME</code>, if <code class="module"><a href="../mod/mod_ssl.html">mod_ssl</a></code> is enabled.</td></tr>
422</table>

424    <div class="note"><h3>Note</h3>
425      <p>The <code>%s</code> format specifier is only available in
426      Apache 2.1 and later; it can be used instead of <code>%e</code>
427      to avoid the overhead of enabling <code>SSLOptions
428      +StdEnvVars</code>.  If <code>SSLOptions +StdEnvVars</code> must
429      be enabled anyway for some other reason, <code>%e</code> will be
430      more efficient than <code>%s</code>.</p>

433    <div class="note"><h3>Note on expression values</h3>
434    <p> When the value parameter uses the <a href="../expr.html">ap_expr</a>
435    parser, some expression syntax will differ from examples that evaluate
436    <em>boolean</em> expressions such as &lt;If&gt;:</p>
437    <ul>
438      <li>The starting point of the grammar is 'string' rather than 'expr'.</li>
439      <li>Function calls use the %{funcname:arg} syntax rather than
440          funcname(arg).</li>
441      <li>Multi-argument functions are not currently accessible from this
442          starting point</li>
443      <li>Quote the entire parameter, such as
444          <pre class="prettyprint lang-config">Header set foo-checksum "expr=%{md5:foo}"</pre>
445
446       </li>
447
448    </ul>
449    </div>
450
451    <p>For <code>edit</code> there is both a <var>value</var> argument
452    which is a <a class="glossarylink" href="../glossary.html#regex" title="see glossary">regular expression</a>,
453    and an additional <var>replacement</var> string. As of version 2.4.7
454    the replacement string may also contain format specifiers.</p>
455
456    <p>The <code class="directive">Header</code> directive may be followed by
457    an additional argument, which may be any of:</p>
458    <dl>
459    <dt><code>early</code></dt>
460    <dd>Specifies <a href="#early">early processing</a>.</dd>
461    <dt><code>env=[!]<var>varname</var></code></dt>
462    <dd>The directive is applied if and only if the <a href="../env.html">environment variable</a> <code>varname</code> exists.
463        A <code>!</code> in front of <code>varname</code> reverses the test,
464        so the directive applies only if <code>varname</code> is unset.</dd>
465    <dt><code>expr=<var>expression</var></code></dt>
466    <dd>The directive is applied if and only if <var>expression</var>
467        evaluates to true. Details of expression syntax and evaluation are
468        documented in the <a href="../expr.html">ap_expr</a> documentation.
469        <pre class="prettyprint lang-config"># This delays the evaluation of the condition clause compared to &lt;If&gt;
470Header always set CustomHeader my-value "expr=%{REQUEST_URI} =~ m#^/special_path.php$#"</pre>
471
472        </dd>
473    </dl>
474
475    <p>Except in <a href="#early">early</a> mode, the
476    <code class="directive">Header</code> directives are processed just
477    before the response is sent to the network. This means that it is
478    possible to set and/or override most headers, except for some headers
479    added by the HTTP header filter.  Prior to 2.2.12, it was not possible
480    to change the Content-Type header with this directive.</p>
481

485<div class="directive-section"><h2><a name="RequestHeader" id="RequestHeader">RequestHeader</a> <a name="requestheader" id="requestheader">Directive</a></h2>
486<table class="directive">
487<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Configure HTTP request headers</td></tr>
488<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>RequestHeader add|append|edit|edit*|merge|set|setifempty|unset
489<var>header</var> [[expr=]<var>value</var> [<var>replacement</var>]
490[early|env=[!]<var>varname</var>|expr=<var>expression</var>]]
491</code></td></tr>
492<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config, virtual host, directory, .htaccess</td></tr>
493<tr><th><a href="directive-dict.html#Override">Override:</a></th><td>FileInfo</td></tr>
494<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Extension</td></tr>
495<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_headers</td></tr>
496<tr><th><a href="directive-dict.html#Compatibility">Compatibility:</a></th><td>SetIfEmpty available in 2.4.7 and later, expr=value
497available in 2.4.10 and later</td></tr>
498</table>
499    <p>This directive can replace, merge, change or remove HTTP request
500    headers. The header is modified just before the content handler
501    is run, allowing incoming headers to be modified. The action it
502    performs is determined by the first argument. This can be one
503    of the following values:</p>
504
505    <dl>
506
507    <dt><code>add</code></dt>
508    <dd>The request header is added to the existing set of headers,
509    even if this header already exists. This can result in two
510    (or more) headers having the same name. This can lead to
511    unforeseen consequences, and in general <code>set</code>,
512    <code>append</code> or <code>merge</code> should be used instead.</dd>
513
514    <dt><code>append</code></dt>
515    <dd>The request header is appended to any existing header of the
516    same name. When a new value is merged onto an existing header
517    it is separated from the existing header with a comma. This
518    is the HTTP standard way of giving a header multiple
519    values.</dd>
520
521    <dt><code>edit</code></dt>
522    <dt><code>edit*</code></dt>
523    <dd>If this request header exists, its value is transformed according
524    to a <a class="glossarylink" href="../glossary.html#regex" title="see glossary">regular expression</a>
525    search-and-replace.  The <var>value</var> argument is a <a class="glossarylink" href="../glossary.html#regex" title="see glossary">regular expression</a>, and the <var>replacement</var>
526    is a replacement string, which may contain backreferences or format specifiers.
527    The <code>edit</code> form will match and replace exactly once
528    in a header value, whereas the <code>edit*</code> form will replace
529    <em>every</em> instance of the search pattern if it appears more
530    than once.</dd>
531
532    <dt><code>merge</code></dt>
533    <dd>The request header is appended to any existing header of
534    the same name, unless the value to be appended already appears in the
535    existing header's comma-delimited list of values.  When a new value is
536    merged onto an existing header it is separated from the existing header
537    with a comma.  This is the HTTP standard way of giving a header multiple
538    values.  Values are compared in a case sensitive manner, and after
539    all format specifiers have been processed.  Values in double quotes
540    are considered different from otherwise identical unquoted values.</dd>
541
542    <dt><code>set</code></dt>
543    <dd>The request header is set, replacing any previous header
544    with this name</dd>
545
546    <dt><code>setifempty</code></dt>
547    <dd>The request header is set, but only if there is no previous header
548    with this name.<br />
549    Available in 2.4.7 and later.</dd>
550
551    <dt><code>unset</code></dt>
552    <dd>The request header of this name is removed, if it exists. If
553    there are multiple headers of the same name, all will be removed.
554    <var>value</var> must be omitted.</dd>
555    </dl>
556
557    <p>This argument is followed by a header name, which can
558    include the final colon, but it is not required. Case is
559    ignored. For <code>set</code>, <code>append</code>, <code>merge</code> and
560    <code>add</code> a <var>value</var> is given as the third argument. If a
561    <var>value</var> contains spaces, it should be surrounded by double
562    quotes. For <code>unset</code>, no <var>value</var> should be given.
563    <var>value</var> may be a character string, a string containing format
564    specifiers or a combination of both. The supported format specifiers
565    are the same as for the <code class="directive"><a href="#header">Header</a></code>,
566    please have a look there for details.  For <code>edit</code> both
567    a <var>value</var> and a <var>replacement</var> are required, and are
568    a <a class="glossarylink" href="../glossary.html#regex" title="see glossary">regular expression</a> and a
569    replacement string respectively.</p>
570
571    <p>The <code class="directive">RequestHeader</code> directive may be followed by
572    an additional argument, which may be any of:</p>
573    <dl>
574    <dt><code>early</code></dt>
575    <dd>Specifies <a href="#early">early processing</a>.</dd>
576    <dt><code>env=[!]<var>varname</var></code></dt>
577    <dd>The directive is applied if and only if the <a href="../env.html">environment variable</a> <code>varname</code> exists.
578        A <code>!</code> in front of <code>varname</code> reverses the test,
579        so the directive applies only if <code>varname</code> is unset.</dd>
580    <dt><code>expr=<var>expression</var></code></dt>
581    <dd>The directive is applied if and only if <var>expression</var>
582        evaluates to true. Details of expression syntax and evaluation are
583        documented in the <a href="../expr.html">ap_expr</a> documentation.</dd>
584    </dl>
585
586    <p>Except in <a href="#early">early</a> mode, the
587    <code class="directive">RequestHeader</code> directive is processed
588    just before the request is run by its handler in the fixup phase.
589    This should allow headers generated by the browser, or by Apache
590    input filters to be overridden or modified.</p>
