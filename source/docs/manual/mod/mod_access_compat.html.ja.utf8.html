44<h3>概要</h3>
45
46    <p><code class="module"><a href="../mod/mod_access_compat.html">mod_access_compat</a></code> により提供されるディレクティブは
47    サーバの特定の部分への
48    アクセスを制御するために <code class="directive"><a href="../mod/core.html#directory">&lt;Directory&gt;</a></code>, <code class="directive"><a href="../mod/core.html#files">&lt;Files&gt;</a></code>, <code class="directive"><a href="../mod/core.html#location">&lt;Location&gt;</a></code>
49    と <code><a href="core.html#accessfilename">.htaccess</a></code> ファイルで使用されます。クライアントのホスト名、IP
50    アドレスや、<a href="../env.html">環境変数</a>などのリクエストの特徴に基づいて
51    アクセス制御を行なうことができます。<code class="directive"><a href="#allow">Allow</a></code> と
52    <code class="directive"><a href="#deny">Deny</a></code> ディレクティブを使って、
53    どのようなクライアントにアクセスを
54    許可する、しないを指定します。また
55    <code class="directive"><a href="#order">Order</a></code> ディレクティブを使って、
56    デフォルトのアクセス状態と、
57    <code class="directive"><a href="#allow">Allow</a></code> ディレクティブと
58    <code class="directive"><a href="#deny">Deny</a></code>
59    ディレクティブとのお互いへの影響の仕方を設定します。
60    </p>
61
62    <p>ホストによるアクセス制限とパスワードによる認証を、
63    同時に組み合わせて使うこともできます。
64    この場合、その二つの制限の関係を指定するために
65    <code class="directive"><a href="#satisfy">Satisfy</a></code>
66    ディレクティブを使用します。</p>
67
68    <div class="warning"><h3>Note</h3>
69      <p><code class="module"><a href="../mod/mod_access_compat.html">mod_access_compat</a></code>が提供するディレクティブは、
70      承認の仕組みの一新に伴い、非推奨になったものです。
71      <code class="module"><a href="../mod/mod_authz_host.html">mod_authz_host</a></code> も見てください。
72      デフォルトの承認の取り扱い機能を使用するために
73      <code class="module"><a href="../mod/mod_authz_default.html">mod_authz_default</a></code>
74      モジュールもロードされなければなりません。</p>
75    </div>
76
77    <p>一般的には、アクセス制限ディレクティブはすべてのアクセスメソッド
78    (<code>GET</code>, <code>PUT</code>, <code>POST</code> など)
79    に適用されます。そして、ほとんどの場合これが望ましい動作です。
80    しかし、<code class="directive"><a href="../mod/core.html#limit">&lt;Limit&gt;</a></code>
81    セクションの中にディレクティブを書くことで、
82    一部のメソッドにのみ制限をかけることもできます。</p>
83</div>

99<div class="directive-section"><h2><a name="Allow" id="Allow">Allow</a> <a name="allow" id="allow">ディレクティブ</a></h2>
100<table class="directive">
101<tr><th><a href="directive-dict.html#Description">説明:</a></th><td>サーバのある領域にアクセスできるホストを制御する</td></tr>
102<tr><th><a href="directive-dict.html#Syntax">構文:</a></th><td><code> Allow from all|<var>host</var>|env=[!]<var>env-variable</var>
103[<var>host</var>|env=[!]<var>env-variable</var>] ...</code></td></tr>
104<tr><th><a href="directive-dict.html#Context">コンテキスト:</a></th><td>ディレクトリ, .htaccess</td></tr>
105<tr><th><a href="directive-dict.html#Override">上書き:</a></th><td>Limit</td></tr>
106<tr><th><a href="directive-dict.html#Status">ステータス:</a></th><td>Extension</td></tr>
107<tr><th><a href="directive-dict.html#Module">モジュール:</a></th><td>mod_access_compat</td></tr>
108</table>
109    <p><code class="directive">Allow</code> ディレクティブは、どのホストが
110    サーバのある領域にアクセスできるかに影響を与えます。
111    アクセスはホスト名、IP アドレス、IP アドレスの範囲や、
112    環境変数などのクライアントのリクエストの
113    特徴に基づいてアクセス制御することができます。</p> 
114
115    <p>このディレクティブの最初の引数は常に <code>from</code> です。
116    それに続く引数は三つの違った形式があります。<code>Allow from
117    all</code> が指定されていれば、すべてのホストにアクセスを許可し、
118    アクセス制限は下で説明されているように、
119    <code class="directive"><a href="#deny">Deny</a></code>
120    ディレクティブと <code class="directive"><a href="#order">Order</a></code>
121    ディレクティブの設定で決まります。
122    特定のホストやホスト群にのみサーバへのアクセスを許可するためには、
123    以下のどれかの形式で <em>host</em> を指定することができます:</p>
124
125    <dl>
126      <dt>ドメイン名 (の一部)</dt>
127
128      <dd>
129      <div class="example"><h3>例</h3><p><code>
130        Allow from apache.org<br />
131        Allow from .net example.edu
132      </code></p></div>
133      <p>この文字列に合うか、これで終わる名前のホストのアクセスが許可されます。
134      各部分が完全に合うものだけに適用されますので、上の例は
135      <code>foo.apache.org</code> にはマッチしますが、
136      <code>fooapache.org</code> にはマッチしません。
137      この設定をすると、Apache は
138      <code class="directive"><a href="../mod/core.html#hostnamelookups">HostnameLookups</a></code>
139      の設定に関わらず、クライアントの IP アドレスに対して
140      DNS の 2 重逆引きを行ないます。
141      ホスト名からオリジナルの IP アドレスを順引きします。
142      順引きと逆引きが一致し、ホスト名が該当した場合にのみ、
143      アクセスが許可されます。</p></dd>
144
145      <dt>完全な IP アドレス</dt>
146
147      <dd>
148      <div class="example"><h3>例</h3><p><code>
149        Allow from 10.1.2.3<br />
150        Allow from 192.168.1.104 192.168.1.205
151      </code></p></div>
152      <p>アクセスを許可する IP アドレスです。</p></dd>
153
154      <dt>IP アドレスの一部</dt>
155
156      <dd>
157      <div class="example"><h3>例</h3><p><code>
158        Allow from 10.1<br />
159        Allow from 10 172.20 192.168.2
160      </code></p></div>
161      <p>サブネットの制限用の、IP
162      アドレスの最初の一つから三つまでのバイトです。</p></dd>
163
164      <dt>ネットワーク/ネットマスク の対</dt>
165
166      <dd>
167      <div class="example"><h3>例</h3><p><code>
168        Allow from 10.1.0.0/255.255.0.0
169      </code></p></div>
170      <p>ネットワーク a.b.c.d とネットマスク w.x.y.z です。
171      より細粒度のサブネット制限用です。</p></dd>
172
173      <dt>ネットワーク/nnn CIDR 指定</dt>
174
175      <dd>
176      <div class="example"><h3>例</h3><p><code>
177        Allow from 10.1.0.0/16
178      </code></p></div>
179      <p>ネットマスクが nnn の上位ビットが 1
180      となっているものからなること以外は前のものと同じです。</p></dd>
181    </dl>
182
183    <p>注: 最後の三つの例はまったく同じホストに合います。</p>
184
185
186    <p>IPv6 アドレスと IPv6 のサブネットは以下のように指定できます:</p>
187
188    <div class="example"><p><code>
189       Allow from 2001:db8::a00:20ff:fea7:ccea<br />
190       Allow from 2001:db8::a00:20ff:fea7:ccea/10
191    </code></p></div>
192
193    <p><code class="directive">Allow</code> ディレクティブの引数の三つ目の形式は、
194    <a href="../env.html">環境変数</a>
195    の存在によりアクセスの制御を行なえるようにするものです。
196    <code>Allow from env=<var>env-variable</var></code>
197    が指定されていると、環境変数 <var>env-variable</var>
198    が存在した場合にリクエストはアクセスを許可されます。
199    <code>Allow from env=!<var>env-variable</var></code>
200    が指定されていると、環境変数 <var>env-variable</var>
201    が存在しない場合にアクセス許可されます。
202    サーバは <code class="module"><a href="../mod/mod_setenvif.html">mod_setenvif</a></code>
203    のディレクティブにより、クライアントのリクエスト
204    の特徴に基づいて柔軟に環境変数を設定する機能を提供します。
205    ですから、このディレクティブはクライアントの
206    <code>User-Agent</code> (ブラウザの種類)、<code>Referer</code>
207    や他の HTTP リクエストのヘッダフィールドなどに基づいて
208    アクセス許可をするために使うことができます。
209    </p>
210
211    <div class="example"><h3>Example:</h3><p><code>
212      SetEnvIf User-Agent ^KnockKnock/2\.0 let_me_in<br />
213      &lt;Directory /docroot&gt;<br />
214      <span class="indent">
215        Order Deny,Allow<br />
216        Deny from all<br />
217        Allow from env=let_me_in<br />
218      </span>
219      &lt;/Directory&gt;
220    </code></p></div>
221
222    <p>この場合、user-agent の文字列が <code>KnockKnock/2.0</code>
223    で始まるブラウザのみがアクセスが許可され、
224    他のものはアクセスが拒否されます。</p>
225
226
227</div>

240    アドレス、環境変数に基づいてサーバへのアクセスを制限します。
241    <code class="directive">Deny</code> ディレクティブの引数は <code class="directive"><a href="#allow">Allow</a></code>
242    ディレクティブとまったく同じです。</p>

246<div class="directive-section"><h2><a name="Order" id="Order">Order</a> <a name="order" id="order">ディレクティブ</a></h2>
247<table class="directive">
248<tr><th><a href="directive-dict.html#Description">説明:</a></th><td>デフォルトのアクセス可能な状態と、<code class="directive">Allow</code> と
249<code class="directive">Deny</code> が評価される順番を制御する</td></tr>
250<tr><th><a href="directive-dict.html#Syntax">構文:</a></th><td><code> Order <var>ordering</var></code></td></tr>
251<tr><th><a href="directive-dict.html#Default">デフォルト:</a></th><td><code>Order Deny,Allow</code></td></tr>
252<tr><th><a href="directive-dict.html#Context">コンテキスト:</a></th><td>ディレクトリ, .htaccess</td></tr>
253<tr><th><a href="directive-dict.html#Override">上書き:</a></th><td>Limit</td></tr>
254<tr><th><a href="directive-dict.html#Status">ステータス:</a></th><td>Extension</td></tr>
255<tr><th><a href="directive-dict.html#Module">モジュール:</a></th><td>mod_access_compat</td></tr>
256</table>
257
258    <p><code class="directive">Order</code> ディレクティブは <code class="directive"><a href="#allow">Allow</a></code> ディレクティブと<code class="directive"><a href="#deny">Deny</a></code> と共に"3段階アクセス制御システム"
259    を制御します。第1段階目では <code class="directive">Order</code> ディレクティブで1番目に
260    指定したディレクティブ(<code class="directive"><a href="#allow">Allow</a></code> 
261    または <code class="directive"><a href="#deny">Deny</a></code>)を全て処理します。
262    第2段階目で、残りのディレクティブ(<code class="directive"><a href="#deny">Deny</a></code> または <code class="directive"><a href="#allow">Allow</a></code>)
263    を全て処理します。第3段階目で、第1段階目と第2段階目で
264    マッチしなかったリクエストを処理します。</p>
265
266    <p>全ての <code class="directive"><a href="#allow">Allow</a></code> と <code class="directive"><a href="#deny">Deny</a></code> が処理され、結局のところ最後にマッチ
267    した条件が有効となることに注意してください。これは最初にマッチした条件だけが有効
268    となる、典型的なファイアウォールの動作とは異なっています。
269    また、設定ファイルに書く順番には意味はありません。<code class="directive"><a href="#allow">Allow</a></code> 行は全部一つのグループとして扱われ、
270    <code class="directive"><a href="#deny">Deny</a></code> 行はもう一つのグループとみなされます。
271    またデフォルト状態は単独で一つのグループとみなされます。</p>
272
273    <p><em>Order</em> 設定は以下のどれかです。</p>
274
275    <dl>
276      <dt><code>Allow,Deny</code></dt>
277
278      <dd>まず <code class="directive"><a href="#allow">Allow</a></code> 
279      ディレクティブが適用されます。どれにもマッチしなかった場合、この時点で
280      リクエストは拒否されます。次に、全ての <code class="directive"><a href="#deny">
281      Deny</a></code> ディレクティブが適用されます。どれか一つでもマッチした場合は、
282      リクエストは拒否されます。
283      最後に、 <code class="directive"><a href="#allow">Allow</a></code> にも <code class="directive"><a href="#deny">Deny</a></code> にもマッチしなかったリクエストは
284      デフォルト設定が適用されるので拒否されます。</dd>
285
286      <dt><code>Deny,Allow</code></dt>
287
288      <dd>まず <code class="directive"><a href="#deny">Deny</a></code> 
289      ディレクティブが適用されます。どれか一つでもマッチした場合は、
290      <code class="directive"><a href="#allow">Allow</a></code> のどれにも
291      マッチ<strong>しなければ</strong>、アクセスは拒否されます。
292      どの <code class="directive"><a href="#allow">Allow</a></code> にも <code class="directive"><a href="#deny">Deny</a></code> にもマッチしないリクエストは
293      許可されます。</dd>
294
295      <dt><code>Mutual-failure</code></dt>
296
297      <dd>これは <code>Order Allow,Deny</code> と全く同じ効果を持ち、 
298      そのため非推奨となっています。
299      </dd>
300    </dl>
301
302    <p>キーワードの間に置けるのはコンマだけです。
303    間に<em>空白があってはいけません</em>。</p>

309        <th>Deny,Allow 時の結果</th>
310      </tr><tr>
311        <th>Allow だけにマッチ</th>
312        <td>許可</td>
313        <td>許可</td>
314      </tr><tr>
315        <th>Deny だけにマッチ</th>
316        <td>拒否</td>
317        <td>拒否</td>
318      </tr><tr>
319        <th>どちらにもマッチしない</th>
320        <td>2番目のディレクティブがデフォルト: 拒否</td>
321        <td>2番目のディレクティブがデフォルト: 許可</td>
322      </tr><tr>
323        <th>Allow と Deny 両方にマッチ</th>
324        <td>最後にマッチしたほう: 拒否</td>
325        <td>最後にマッチしたほう: 許可</td>
326      </tr>
327    </table>
328
329    <p>以下の例では、apache.org
330    ドメインのすべてのホストはアクセスを許可されます。
331    他のすべてのホストはアクセスを拒否されます。</p>
332
333    <div class="example"><p><code>
334      Order Deny,Allow<br />
335      Deny from all<br />
336      Allow from apache.org
337    </code></p></div>
338
339    <p>次の例では、foo.apache.org サブドメインにあるホスト以外の、
340    apache.org ドメインのすべてのホストがアクセスを許可されます。
341    apache.org
342    ドメインでないホストは、デフォルトの状態が <code class="directive"><a href="#deny">Deny</a></code> のため、
343    サーバへのアクセスを拒否されます。</p>
344
345    <div class="example"><p><code>
346      Order Allow,Deny<br />
347      Allow from apache.org<br />
348      Deny from foo.apache.org
349    </code></p></div>
350
351    <p>一方、上の例の <code class="directive">Order</code> が <code>Deny,Allow</code>
352    に変わっていれば、すべのホストにアクセスが許可されます。
353    これは、設定ファイル中の実際の順番に関わらず、
354    <code>Allow from apache.org</code> が最後に評価されて、
355    <code>Deny from foo.apache.org</code> を上書きするからです。
356    <code>apache.org</code>
357    ドメインにないホストも、デフォルトの状態が <code class="directive"><a href="#allow">Allow</a></code>
358    なので、アクセスを許可されます。
359    </p>
360
361    <p><code class="directive">Order</code>
362    ディレクティブはデフォルトのアクセスの状態に影響を与えるので、
363    <code class="directive"><a href="../mod/mod_authz_host.html#allow">Allow</a></code> ディレクティブと
364    <code class="directive"><a href="../mod/mod_authz_host.html#deny">Deny</a></code>
365    ディレクティブが無くても、サーバのアクセスに影響を与えることができます。
366    たとえば、</p>
367
368    <div class="example"><p><code>
369      &lt;Directory /www&gt;<br />
370      <span class="indent">
371        Order Allow,Deny<br />
372      </span>
373      &lt;/Directory&gt;
374    </code></p></div>
375
376    <p>はデフォルトのアクセス状態が 
377    <code class="directive"><a href="#deny">Deny</a></code> になるため、
378    <code>/www</code> ディレクトリへのすべてのアクセスを拒否します。
379    </p>
380
381    <p><code class="directive">Order</code>
382    ディレクティブはサーバの設定処理の各段階でだけ
383    アクセスディレクティブの処理の順番を変更します。これは、たとえば、
384    <code class="directive">Order</code> ディレクティブの設定に関わらず、
385    <code class="directive"><a href="../mod/core.html#location">&lt;Location&gt;</a></code> セクションの
386    <code class="directive"><a href="#allow">Allow</a></code> ディレクティブや
387    <code class="directive"><a href="#deny">Deny</a></code> ディレクティブは、
388    <code class="directive"><a href="../mod/core.html#directory">Directory</a></code> セクションや
389    <code>.htaccess</code> ファイルの <code class="directive"><a href="#allow">Allow</a></code>
390    ディレクティブや <code class="directive"><a href="#deny">Deny</a></code>
391    ディレクティブよりも常に後に評価されるということを意味します。
392    設定セクションのマージの詳細については、
393    <a href="../sections.html">Directory,Location, Files
394    セクションの動作方法</a> を参照してください。</p>
395
396</div>

398<div class="directive-section"><h2><a name="Satisfy" id="Satisfy">Satisfy</a> <a name="satisfy" id="satisfy">ディレクティブ</a></h2>
399<table class="directive">
400<tr><th><a href="directive-dict.html#Description">説明:</a></th><td>ホストレベルのアクセス制御とユーザ認証との相互作用を指定</td></tr>
401<tr><th><a href="directive-dict.html#Syntax">構文:</a></th><td><code>Satisfy Any|All</code></td></tr>
402<tr><th><a href="directive-dict.html#Default">デフォルト:</a></th><td><code>Satisfy All</code></td></tr>
403<tr><th><a href="directive-dict.html#Context">コンテキスト:</a></th><td>ディレクトリ, .htaccess</td></tr>
404<tr><th><a href="directive-dict.html#Override">上書き:</a></th><td>AuthConfig</td></tr>
405<tr><th><a href="directive-dict.html#Status">ステータス:</a></th><td>Extension</td></tr>
406<tr><th><a href="directive-dict.html#Module">モジュール:</a></th><td>mod_access_compat</td></tr>
407<tr><th><a href="directive-dict.html#Compatibility">互換性:</a></th><td>バージョン 2.0.51 以降では <code class="directive"><a href="../mod/core.html#limit">&lt;Limit&gt;</a></code> ディレクティブと <code class="directive"><a href="../mod/core.html#limitexcept">&lt;LimitExcept&gt;</a></code> ディレクティブの影響を受ける
408</td></tr>
409</table>
410    <p><code class="directive"><a href="#allow">Allow</a></code> と
411    <code class="directive"><a href="../mod/mod_authz_core.html#require">Require</a></code> の両方が使われているときの
412    アクセスポリシーを設定します。パラメータは <code>All</code> か <code>Any</code>
413    です。このディレクティブはある場所へのアクセスがユーザ名/パスワード
414    <em>と</em>クライアントのホストのアドレスで制限されているときにのみ
415    役立ちます。デフォルトの動作 (<code>All</code>) はクライアントがアドレスによる
416    アクセス制限を満たし、<em>かつ</em>正しいユーザ名とパスワードを入力することを
417    要求します。<code>Any</code> では、クライアントはホストの制限を満たすか、
418    正しいユーザ名とパスワードの入力をするかをすればアクセスを許可されます。
419    これは、ある場所をパスワードで保護するけれど、特定のアドレスからの
420    クライアントにはパスワードの入力を要求せずにアクセスを許可する、
421    というようなときに使用できます。</p>
422
423    <p>例えば、同じネットワーク上にいる人にはウェブサイトのある部分について
424    無制限のアクセスを許したいけれど、外のネットワークの人には
425    パスワードを提供させるようにするためには、次のような設定をすることが
426    できます:</p>
427
428    <div class="example"><p><code>
429    Require valid-user<br />
430    Allow from 192.168.1<br />
431    Satisfy Any
432    </code></p></div>
433   
434    <p>バージョン 2.0.51 からは
435    <code class="directive"><a href="../mod/core.html#limit">&lt;Limit&gt;</a></code> セクションと
436    <code class="directive"><a href="../mod/core.html#limitexcept">&lt;LimitExcept&gt;</a></code> セクションを使用することで
437    <code class="directive">Satisfy</code> ディレクティブが
438    適用されるメソッドを制限することが
439    できるようになりました。</p>
