37<h3>Summary</h3>
38
39    <p>This module enables smart, context-sensitive configuration of
40    output content filters.  For example, apache can be configured to
41    process different content-types through different filters, even
42    when the content-type is not known in advance (e.g. in a proxy).</p>
43
44    <p><code class="module"><a href="../mod/mod_filter.html">mod_filter</a></code> works by introducing indirection into
45    the filter chain.  Instead of inserting filters in the chain, we insert
46    a filter harness which in turn dispatches conditionally
47    to a filter provider.  Any content filter may be used as a provider
48    to <code class="module"><a href="../mod/mod_filter.html">mod_filter</a></code>; no change to existing filter modules is
49    required (although it may be possible to simplify them).</p>
50</div>

74<h2><a name="smart" id="smart">Smart Filtering</a></h2>
75    <p>In the traditional filtering model, filters are inserted unconditionally
76    using <code class="directive"><a href="../mod/mod_mime.html#addoutputfilter">AddOutputFilter</a></code> and family.
77    Each filter then needs to determine whether to run, and there is little
78    flexibility available for server admins to allow the chain to be
79    configured dynamically.</p>
80
81    <p><code class="module"><a href="../mod/mod_filter.html">mod_filter</a></code> by contrast gives server administrators a
82    great deal of flexibility in configuring the filter chain.  In fact,
83    filters can be inserted based on complex boolean
84    <a href="../expr.html">expressions</a> This generalises the limited
85    flexibility offered by <code class="directive">AddOutputFilterByType</code>.</p>
86</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>
87<div class="section">
88<h2><a name="terms" id="terms">Filter Declarations, Providers and Chains</a></h2>
89    <p class="figure">
90    <img src="../images/mod_filter_old.gif" width="160" height="310" alt="[This image displays the traditional filter model]" /><br />
91    <dfn>Figure 1:</dfn> The traditional filter model</p>
92
93    <p>In the traditional model, output filters are a simple chain
94    from the content generator (handler) to the client.  This works well
95    provided the filter chain can be correctly configured, but presents
96    problems when the filters need to be configured dynamically based on
97    the outcome of the handler.</p>
98
99    <p class="figure">
100    <img src="../images/mod_filter_new.gif" width="423" height="331" alt="[This image shows the mod_filter model]" /><br />
101    <dfn>Figure 2:</dfn> The <code class="module"><a href="../mod/mod_filter.html">mod_filter</a></code> model</p>
102
103    <p><code class="module"><a href="../mod/mod_filter.html">mod_filter</a></code> works by introducing indirection into
104    the filter chain.  Instead of inserting filters in the chain, we insert
105    a filter harness which in turn dispatches conditionally
106    to a filter provider.  Any content filter may be used as a provider
107    to <code class="module"><a href="../mod/mod_filter.html">mod_filter</a></code>; no change to existing filter modules
108    is required (although it may be possible to simplify them).  There can be
109    multiple providers for one filter, but no more than one provider will
110    run for any single request.</p>
111
112    <p>A filter chain comprises any number of instances of the filter
113    harness, each of which may have any number of providers.  A special
114    case is that of a single provider with unconditional dispatch: this
115    is equivalent to inserting the provider filter directly into the chain.</p>
116</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>

118<h2><a name="config" id="config">Configuring the Chain</a></h2>
119    <p>There are three stages to configuring a filter chain with
120    <code class="module"><a href="../mod/mod_filter.html">mod_filter</a></code>. For details of the directives, see below.</p>
121
122    <dl>
123    <dt>Declare Filters</dt>
124    <dd>The <code class="directive"><a href="#filterdeclare">FilterDeclare</a></code> directive
125    declares a filter, assigning it a name and filter type.  Required
126    only if the filter is not the default type AP_FTYPE_RESOURCE.</dd>
127
128    <dt>Register Providers</dt>
129    <dd>The <code class="directive"><a href="#filterprovider">FilterProvider</a></code>
130    directive registers a provider with a filter. The filter may have
131    been declared with <code class="directive"><a href="#filterdeclare">FilterDeclare</a></code>; if not, FilterProvider will implicitly
132    declare it with the default type AP_FTYPE_RESOURCE. The provider
133    must have been
134    registered with <code>ap_register_output_filter</code> by some module.
135    The final argument to <code class="directive"><a href="#filterprovider">FilterProvider</a></code> is an expression: the provider will be
136    selected to run for a request if and only if the expression evaluates
137    to true.  The expression may evaluate HTTP request or response
138    headers, environment variables, or the Handler used by this request.
139    Unlike earlier versions, mod_filter now supports complex expressions
140    involving multiple criteria with AND / OR logic (&amp;&amp; / ||)
141    and brackets. The details of the expression syntax are described in
142    the <a href="../expr.html">ap_expr documentation</a>.</dd>
143
144    <dt>Configure the Chain</dt>
145    <dd>The above directives build components of a smart filter chain,
146    but do not configure it to run.  The <code class="directive"><a href="#filterchain">FilterChain</a></code> directive builds a filter chain from smart
147    filters declared, offering the flexibility to insert filters at the
148    beginning or end of the chain, remove a filter, or clear the chain.</dd>
149</dl>

152<h2><a name="errordocs" id="errordocs">Filtering and Response Status</a></h2>
153    <p>mod_filter normally only runs filters on responses with
154    HTTP status 200 (OK).  If you want to filter documents with
155    other response statuses, you can set the <var>filter-errordocs</var>
156    environment variable, and it will work on all responses
157    regardless of status.  To refine this further, you can use
158    expression conditions with <code class="directive">FilterProvider</code>.</p>

161<h2><a name="upgrade" id="upgrade">Upgrading from Apache HTTP Server 2.2 Configuration</a></h2>
162    <p>The <code class="directive"><a href="#filterprovider">FilterProvider</a></code>
163    directive has changed from httpd 2.2: the <var>match</var> and
164    <var>dispatch</var> arguments are replaced with a single but
165    more versatile <var>expression</var>.  In general, you can convert
166    a match/dispatch pair to the two sides of an expression, using
167    something like:</p>
168    <div class="example"><p><code>"dispatch = 'match'"</code></p></div>
169    <p>The Request headers, Response headers and Environment variables
170    are now interpreted from syntax <var>%{req:foo}</var>,
171    <var>%{resp:foo}</var> and <var>%{env:foo}</var> respectively.
172    The variables <var>%{HANDLER}</var> and <var>%{CONTENT_TYPE}</var>
173    are also supported.</p>
174    <p>Note that the match no longer support substring matches.  They can be
175    replaced by regular expression matches.</p>
176</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>

178<h2><a name="examples" id="examples">Examples</a></h2>
179    <dl>
180    <dt>Server side Includes (SSI)</dt>
181    <dd>A simple case of replacing <code class="directive">AddOutputFilterByType</code>
182    <pre class="prettyprint lang-config">FilterDeclare SSI
183FilterProvider SSI INCLUDES "%{CONTENT_TYPE} =~ m|^text/html|"
184FilterChain SSI</pre>
185
186    </dd>
187
188    <dt>Server side Includes (SSI)</dt>
189    <dd>The same as the above but dispatching on handler (classic
190    SSI behavior; .shtml files get processed).
191    <pre class="prettyprint lang-config">FilterProvider SSI INCLUDES "%{HANDLER} = 'server-parsed'"
192FilterChain SSI</pre>
193
194    </dd>
195
196    <dt>Emulating mod_gzip with mod_deflate</dt>
197    <dd>Insert INFLATE filter only if "gzip" is NOT in the
198    Accept-Encoding header.  This filter runs with ftype CONTENT_SET.
199    <pre class="prettyprint lang-config">FilterDeclare gzip CONTENT_SET
200FilterProvider gzip inflate "%{req:Accept-Encoding} !~ /gzip/"
201FilterChain gzip</pre>
202
203    </dd>
204
205    <dt>Image Downsampling</dt>
206    <dd>Suppose we want to downsample all web images, and have filters
207    for GIF, JPEG and PNG.
208    <pre class="prettyprint lang-config">FilterProvider unpack jpeg_unpack "%{CONTENT_TYPE} = 'image/jpeg'"
209FilterProvider unpack gif_unpack  "%{CONTENT_TYPE} = 'image/gif'"
210FilterProvider unpack png_unpack  "%{CONTENT_TYPE} = 'image/png'"
211
212FilterProvider downsample downsample_filter "%{CONTENT_TYPE} = m|^image/(jpeg|gif|png)|"
213FilterProtocol downsample "change=yes"
214
215FilterProvider repack jpeg_pack "%{CONTENT_TYPE} = 'image/jpeg'"
216FilterProvider repack gif_pack  "%{CONTENT_TYPE} = 'image/gif'"
217FilterProvider repack png_pack  "%{CONTENT_TYPE} = 'image/png'"
218&lt;Location "/image-filter"&gt;
219    FilterChain unpack downsample repack
220&lt;/Location&gt;</pre>
221
222    </dd>
223    </dl>

226<h2><a name="protocol" id="protocol">Protocol Handling</a></h2>
227    <p>Historically, each filter is responsible for ensuring that whatever
228    changes it makes are correctly represented in the HTTP response headers,
229    and that it does not run when it would make an illegal change.  This
230    imposes a burden on filter authors to re-implement some common
231    functionality in every filter:</p>
232
233    <ul>
234    <li>Many filters will change the content, invalidating existing content
235    tags, checksums, hashes, and lengths.</li>
236
237    <li>Filters that require an entire, unbroken response in input need to
238    ensure they don't get byteranges from a backend.</li>
239
240    <li>Filters that transform output in a filter need to ensure they don't
241    violate a <code>Cache-Control: no-transform</code> header from the
242    backend.</li>
243
244    <li>Filters may make responses uncacheable.</li>
245    </ul>
246
247    <p><code class="module"><a href="../mod/mod_filter.html">mod_filter</a></code> aims to offer generic handling of these
248    details of filter implementation, reducing the complexity required of
249    content filter modules. This is work-in-progress; the
250    <code class="directive"><a href="#filterprotocol">FilterProtocol</a></code> implements
251    some of this functionality for back-compatibility with Apache 2.0
252    modules.  For httpd 2.1 and later, the
253    <code>ap_register_output_filter_protocol</code> and
254    <code>ap_filter_protocol</code> API enables filter modules to
255    declare their own behavior.</p>
256
257    <p>At the same time, <code class="module"><a href="../mod/mod_filter.html">mod_filter</a></code> should not interfere
258    with a filter that wants to handle all aspects of the protocol.  By
259    default (i.e. in the absence of any <code class="directive"><a href="#filterprotocol">FilterProtocol</a></code> directives), <code class="module"><a href="../mod/mod_filter.html">mod_filter</a></code>
260    will leave the headers untouched.</p>
261
262    <p>At the time of writing, this feature is largely untested,
263    as modules in common use are designed to work with 2.0.
264    Modules using it should test it carefully.</p>

279    <p>This directive activates a particular output <a href="../filter.html">filter</a> for a request depending on the
280    response <a class="glossarylink" href="../glossary.html#media-type" title="see glossary">media-type</a>.</p>
281
282    <p>The following example uses the <code>DEFLATE</code> filter, which
283    is provided by <code class="module"><a href="../mod/mod_deflate.html">mod_deflate</a></code>. It will compress all
284    output (either static or dynamic) which is labeled as
285    <code>text/html</code> or <code>text/plain</code> before it is sent
286    to the client.</p>
287
288    <pre class="prettyprint lang-config">AddOutputFilterByType DEFLATE text/html text/plain</pre>
289
290
291    <p>If you want the content to be processed by more than one filter, their
292    names have to be separated by semicolons. It's also possible to use one
293    <code class="directive">AddOutputFilterByType</code> directive for each of
294    these filters.</p>
295
296    <p>The configuration below causes all script output labeled as
297    <code>text/html</code> to be processed at first by the
298    <code>INCLUDES</code> filter and then by the <code>DEFLATE</code>
299    filter.</p>
300
301    <pre class="prettyprint lang-config">&lt;Location "/cgi-bin/"&gt;
302    Options Includes
303    AddOutputFilterByType INCLUDES;DEFLATE text/html
304&lt;/Location&gt;</pre>

379<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>FilterProtocol <var>filter-name</var> [<var>provider-name</var>]
380    <var>proto-flags</var></code></td></tr>
381<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config, virtual host, directory, .htaccess</td></tr>
382<tr><th><a href="directive-dict.html#Override">Override:</a></th><td>Options</td></tr>
383<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Base</td></tr>
384<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_filter</td></tr>
385</table>
386    <p>This directs <code class="module"><a href="../mod/mod_filter.html">mod_filter</a></code> to deal with ensuring the
387    filter doesn't run when it shouldn't, and that the HTTP response
388    headers are correctly set taking into account the effects of the
389    filter.</p>
390
391    <p>There are two forms of this directive.  With three arguments, it
392    applies specifically to a <var>filter-name</var> and a
393    <var>provider-name</var> for that filter.
394    With two arguments it applies to a <var>filter-name</var> whenever the
395    filter runs <em>any</em> provider.</p>
396
397    <p>Flags specified with this directive are merged with the flags 
398    that underlying providers may have registered with 
399    <code class="module"><a href="../mod/mod_filter.html">mod_filter</a></code>. For example, a filter may internally specify
400    the equivalent of <code>change=yes</code>, but a particular 
401    configuration of the module can override with <code>change=no</code>.
402    </p>
403
404    <p><var>proto-flags</var> is one or more of</p>
405
406    <dl>
407    <dt><code>change=yes|no</code></dt>
408    <dd>Specifies whether the filter changes the content, including possibly 
409    the content length.  The "no" argument is supported in 2.4.7 and later.</dd>
410
411    <dt><code>change=1:1</code></dt>
412    <dd>The filter changes the content, but will not change the content
413    length</dd>
414
415    <dt><code>byteranges=no</code></dt>
416    <dd>The filter cannot work on byteranges and requires complete input</dd>
417
418    <dt><code>proxy=no</code></dt>
419    <dd>The filter should not run in a proxy context</dd>
420
421    <dt><code>proxy=transform</code></dt>
422    <dd>The filter transforms the response in a manner incompatible with
423    the HTTP <code>Cache-Control: no-transform</code> header.</dd>
424
425    <dt><code>cache=no</code></dt>
426    <dd>The filter renders the output uncacheable (eg by introducing randomised
427    content changes)</dd>
428    </dl>

432<div class="directive-section"><h2><a name="FilterProvider" id="FilterProvider">FilterProvider</a> <a name="filterprovider" id="filterprovider">Directive</a></h2>
433<table class="directive">
434<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Register a content filter</td></tr>
435<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>FilterProvider <var>filter-name</var> <var>provider-name</var>
436 <var>expression</var></code></td></tr>
437<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config, virtual host, directory, .htaccess</td></tr>
438<tr><th><a href="directive-dict.html#Override">Override:</a></th><td>Options</td></tr>
439<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Base</td></tr>
440<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_filter</td></tr>
441</table>
442    <p>This directive registers a <em>provider</em> for the smart filter.
443    The provider will be called if and only if the <var>expression</var>
444    declared evaluates to true when the harness is first called.</p>
445
446    <p>
447    <var>provider-name</var> must have been registered by loading
448    a module that registers the name with
449    <code>ap_register_output_filter</code>.
450    </p>
451
452    <p><var>expression</var> is an
453    <a href="../expr.html">ap_expr</a>.</p>

464<div class="directive-section"><h2><a name="FilterTrace" id="FilterTrace">FilterTrace</a> <a name="filtertrace" id="filtertrace">Directive</a></h2>
465<table class="directive">
466<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Get debug/diagnostic information from
467    <code class="module"><a href="../mod/mod_filter.html">mod_filter</a></code></td></tr>
468<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>FilterTrace <var>filter-name</var> <var>level</var></code></td></tr>
469<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config, virtual host, directory</td></tr>
470<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Base</td></tr>
471<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_filter</td></tr>
472</table>
473    <p>This directive generates debug information from
474    <code class="module"><a href="../mod/mod_filter.html">mod_filter</a></code>.
475    It is designed to help test and debug providers (filter modules), although
476    it may also help with <code class="module"><a href="../mod/mod_filter.html">mod_filter</a></code> itself.</p>
477
478    <p>The debug output depends on the <var>level</var> set:</p>
479    <dl>
480    <dt><code>0</code> (default)</dt>
481    <dd>No debug information is generated.</dd>
482
483    <dt><code>1</code></dt>
484    <dd><code class="module"><a href="../mod/mod_filter.html">mod_filter</a></code> will record buckets and brigades
485    passing through the filter to the error log, before the provider has
486    processed them. This is similar to the information generated by
487    <a href="http://apache.webthing.com/mod_diagnostics/">mod_diagnostics</a>.
488    </dd>
489
490    <dt><code>2</code> (not yet implemented)</dt>
491    <dd>Will dump the full data passing through to a tempfile before the
492    provider. <strong>For single-user debug only</strong>; this will not
493    support concurrent hits.</dd>
494    </dl>
