37<h3>Summary</h3>
38
39    <p>The <code class="module"><a href="../mod/event.html">event</a></code> Multi-Processing Module (MPM) is
40    designed to allow more requests to be served simultaneously by
41    passing off some processing work to the listeners threads, freeing up
42    the worker threads to serve new requests.</p>
43
44    <p>To use the <code class="module"><a href="../mod/event.html">event</a></code> MPM, add
45      <code>--with-mpm=event</code> to the <code class="program"><a href="../programs/configure.html">configure</a></code>
46      script's arguments when building the <code class="program"><a href="../programs/httpd.html">httpd</a></code>.</p>

83<h2><a name="event-worker-relationship" id="event-worker-relationship">Relationship with the Worker MPM</a></h2>
84<p><code class="module"><a href="../mod/event.html">event</a></code> is based on the <code class="module"><a href="../mod/worker.html">worker</a></code> MPM, which implements a hybrid
85multi-process multi-threaded server. A single control process (the parent) is responsible for launching
86child processes. Each child process creates a fixed number of server
87threads as specified in the <code class="directive"><a href="../mod/mpm_common.html#threadsperchild">ThreadsPerChild</a></code> directive, as well
88as a listener thread which listens for connections and passes them to a worker thread for processing when they arrive.</p>
89
90<p>Run-time configuration directives are identical to those provided by <code class="module"><a href="../mod/worker.html">worker</a></code>, with the only addition
91of the <code class="directive">AsyncRequestWorkerFactor</code>.</p>
92

95<h2><a name="how-it-works" id="how-it-works">How it Works</a></h2>
96    <p>This MPM tries to fix the 'keep alive problem' in HTTP. After a client
97    completes the first request, it can keep the connection
98    open, sending further requests using the same socket and saving
99    significant overhead in creating TCP connections. However,
100    Apache HTTP Server traditionally keeps an entire child
101    process/thread waiting for data from the client, which brings its own disadvantages.
102    To solve this problem, this MPM uses a dedicated listener thread for each process
103    to handle both the Listening sockets, all sockets that are in a Keep Alive state,
104    sockets where the handler and protocol filters have done their work
105    and the ones where the only remaining thing to do is send the data to the client.
106    </p>
107
108    <p>This new architecture, leveraging non-blocking sockets and modern kernel
109       features exposed by <a class="glossarylink" href="../glossary.html#apr" title="see glossary">APR</a> (like Linux's epoll),
110       no longer requires the <code>mpm-accept</code> <code class="directive"><a href="../mod/core.html#mutex">Mutex</a></code>
111       configured to avoid the thundering herd problem.</p>
112
113    <p>The total amount of connections that a single process/threads block can handle is regulated
114        by the <code class="directive">AsyncRequestWorkerFactor</code> directive.</p>
115
116    <h3><a name="async-connections" id="async-connections">Async connections</a></h3>
117        <p>Async connections would need a fixed dedicated worker thread with the previous MPMs but not with event.
118        The status page of <code class="module"><a href="../mod/mod_status.html">mod_status</a></code> shows new columns under the Async connections section:</p>
119        <dl>
120            <dt>Writing</dt>
121	    <dd>While sending the response to the client, it might happen that the TCP write buffer fills up because the connection is too slow.
122	 Usually in this case, a <code>write()</code> to the socket returns <code>EWOULDBLOCK</code> or <code>EAGAIN</code> to become writable again after an idle time.
123	 The worker holding the socket might be able to offload the waiting task to the listener thread, that in turn will re-assign it to the first idle worker thread available once an event will be raised for the socket (for example, "the socket is now writable").
124	 Please check the Limitations section for more information.
125            </dd>
126
127            <dt>Keep-alive</dt>
128            <dd>Keep Alive handling is the most basic improvement from the worker MPM.
129            Once a worker thread finishes to flush the response to the client, it can offload the
130            socket handling to the listener thread, that in turn will wait for any event from the
131            OS, like "the socket is readable". If any new request comes from the client, then the
132            listener will forward it to the first worker thread available. Conversely, if the
133            <code class="directive"><a href="../mod/core.html#keepalivetimeout">KeepAliveTimeout</a></code> occurs then the socket will be
134            closed by the listener. In this way, the worker threads are not responsible for idle
135            sockets, and they can be re-used to serve other requests.</dd>
136
137            <dt>Closing</dt>
138            <dd>Sometimes the MPM needs to perform a lingering close, namely sending back an early error to the client while it is still transmitting data to httpd.
139            Sending the response and then closing the connection immediately is not the correct thing to do since the client (still trying to send the rest of the
140	    request) would get a connection reset and could not read the httpd's response.
141	    The lingering close is time-bounded, but it can take a relatively long
142	    time, so it's offloaded to a worker thread (including the shutdown hooks and real socket close).
143	    From 2.4.28 onward, this is also the
144            case when connections finally timeout (the listener thread never handles connections besides waiting for and dispatching their events).
145            </dd>
146        </dl>
147
148        <p>These improvements are valid for both HTTP/HTTPS connections.</p>
149

152    <h3><a name="graceful-close" id="graceful-close">Graceful process termination and Scoreboard usage</a></h3>
153        <p>This mpm showed some scalability bottlenecks in the past, leading to the following
154        error: "<strong>scoreboard is full, not at MaxRequestWorkers</strong>".
155        <code class="directive"><a href="../mod/mpm_common.html#maxrequestworkers">MaxRequestWorkers</a></code>
156        limits the number of simultaneous requests that will be served at any given time
157        and also the number of allowed processes
158        (<code class="directive"><a href="../mod/mpm_common.html#maxrequestworkers">MaxRequestWorkers</a></code> 
159        / <code class="directive"><a href="../mod/mpm_common.html#threadsperchild">ThreadsPerChild</a></code>); meanwhile,
160        the Scoreboard is a representation of all the running processes and
161        the status of their worker threads. If the scoreboard is full (so all the
162        threads have a state that is not idle) but the number of active requests
163        served is not <code class="directive"><a href="../mod/mpm_common.html#maxrequestworkers">MaxRequestWorkers</a></code>,
164        it means that some of them are blocking new requests that could be served
165        but that are queued instead (up to the limit imposed by
166        <code class="directive"><a href="../mod/mpm_common.html#listenbacklog">ListenBacklog</a></code>). Most of the time,
167        the threads are stuck in the Graceful state, namely they are waiting to
168        finish their work with a TCP connection to safely terminate and free up a
169        scoreboard slot (for example, handling long-running requests, slow clients
170        or connections with keep-alive enabled). Two scenarios are very common:</p>
171        <ul>
172            <li>During a <a href="../stopping.html#graceful">graceful restart</a>,
173            the parent process signals all its children to complete
174            their work and terminate, while it reloads the config and forks new
175            processes. If the old children keep running for a while before stopping,
176            the scoreboard will be partially occupied until their slots are freed.
177            </li>
178            <li>The server load goes down in a way that causes httpd to
179            stop some processes (for example, due to
180            <code class="directive"><a href="../mod/mpm_common.html#maxsparethreads">MaxSpareThreads</a></code>).
181            This is particularly problematic because when the load increases again,
182            httpd will try to start new processes.
183            If the pattern repeats, the number of processes can rise quite a bit,
184            ending up in a mixture of old processes trying to stop and new ones
185            trying to do some work.
186            </li>
187        </ul>
188        <p>From 2.4.24 onward, mpm-event is smarter and it is able to handle
189        graceful terminations in a much better way. Some of the improvements are:</p>
190        <ul>
191            <li>Allow the use of all the scoreboard slots up to
192            <code class="directive"><a href="../mod/mpm_common.html#serverlimit">ServerLimit</a></code>.
193            <code class="directive"><a href="../mod/mpm_common.html#maxrequestworkers">MaxRequestWorkers</a></code> and
194            <code class="directive"><a href="../mod/mpm_common.html#threadsperchild">ThreadsPerChild</a></code> are used
195            to limit the amount of active processes; meanwhile,
196            <code class="directive"><a href="../mod/mpm_common.html#serverlimit">ServerLimit</a></code> 
197            takes also into account the ones doing a graceful
198            close to allow extra slots when needed. The idea is to use
199            <code class="directive"><a href="../mod/mpm_common.html#serverlimit">ServerLimit</a></code> to instruct httpd
200            about how many overall processes are tolerated before impacting
201            the system resources.
202            </li>
203            <li>Force gracefully finishing processes to close their
204            connections in keep-alive state.</li>
205            <li>During graceful shutdown, if there are more running worker threads
206            than open connections for a given process, terminate these threads to
207            free resources faster (which may be needed for new processes).</li>
208            <li>If the scoreboard is full, prevent more processes from finishing
209            gracefully due to reduced load until old processes have terminated
210            (otherwise the situation would get worse once the load increases again).</li>
211        </ul>
212        <p>The behavior described in the last point is completely observable via
213        <code class="module"><a href="../mod/mod_status.html">mod_status</a></code> in the connection summary table through two new
214        columns: "Slot" and "Stopping". The former indicates the PID and
215        the latter if the process is stopping or not; the extra state "Yes (old gen)"
216        indicates a process still running after a graceful restart.</p>
217

219    <h3><a name="limitations" id="limitations">Limitations</a></h3>
220        <p>The improved connection handling may not work for certain connection
221        filters that have declared themselves as incompatible with event. In these
222        cases, this MPM will fall back to the behavior of the
223        <code class="module"><a href="../mod/worker.html">worker</a></code> MPM and reserve one worker thread per connection.
224        All modules shipped with the server are compatible with the event MPM.</p>
225
226        <p>A similar restriction is currently present for requests involving an
227        output filter that needs to read and/or modify the whole response body.
228        If the connection to the client blocks while the filter is processing the
229        data, and the amount of data produced by the filter is too big to be
230        buffered in memory, the thread used for the request is not freed while
231        httpd waits until the pending data is sent to the client.<br />
232        To illustrate this point, we can think about the following two situations:
233        serving a static asset (like a CSS file) versus serving content retrieved from
234        FCGI/CGI or a proxied server. The former is predictable, namely the event MPM
235        has full visibility on the end of the content and it can use events: the worker
236        thread serving the response content can flush the first bytes until <code>EWOULDBLOCK</code>
237        or <code>EAGAIN</code> is returned, delegating the rest to the listener. This one in turn
238        waits for an event on the socket and delegates the work to flush the rest of the content
239        to the first idle worker thread. Meanwhile in the latter example (FCGI/CGI/proxied content),
240        the MPM can't predict the end of the response and a worker thread has to finish its work
241        before returning the control to the listener. The only alternative is to buffer the
242        response in memory, but it wouldn't be the safest option for the sake of the
243        server's stability and memory footprint.
244        </p>
245
246

248    <h3><a name="background" id="background">Background material</a></h3>
249        <p>The event model was made possible by the introduction of new APIs into the supported operating systems:</p>
250        <ul>
251            <li>epoll (Linux) </li>
252            <li>kqueue (BSD) </li>
253            <li>event ports (Solaris) </li>
254        </ul>
255        <p>Before these new APIs where made available, the traditional <code>select</code> and <code>poll</code> APIs had to be used.
256        Those APIs get slow if used to handle many connections or if the set of connections rate of change is high.
257        The new APIs allow to monitor many more connections, and they perform way better when the set of connections to monitor changes frequently. So these APIs made it possible to write the event MPM, that scales much better with the typical HTTP pattern of many idle connections.</p>
258
259        <p>The MPM assumes that the underlying <code>apr_pollset</code>
260        implementation is reasonably threadsafe. This enables the MPM to
261        avoid excessive high level locking, or having to wake up the listener
262        thread in order to send it a keep-alive socket. This is currently
263        only compatible with KQueue and EPoll.</p>

269<h2><a name="requirements" id="requirements">Requirements</a></h2>
270    <p>This MPM depends on <a class="glossarylink" href="../glossary.html#apr" title="see glossary">APR</a>'s atomic
271    compare-and-swap operations for thread synchronization. If you are
272    compiling for an x86 target and you don't need to support 386s, or
273    you are compiling for a SPARC and you don't need to run on
274    pre-UltraSPARC chips, add
275    <code>--enable-nonportable-atomics=yes</code> to the
276    <code class="program"><a href="../programs/configure.html">configure</a></code> script's arguments. This will cause
277    APR to implement atomic operations using efficient opcodes not
278    available in older CPUs.</p>
279
280    <p>This MPM does not perform well on older platforms which lack good
281    threading, but the requirement for EPoll or KQueue makes this
282    moot.</p>
283
284    <ul>
285
286      <li>To use this MPM on FreeBSD, FreeBSD 5.3 or higher is recommended.
287      However, it is possible to run this MPM on FreeBSD 5.2.1 if you
288      use <code>libkse</code> (see <code>man libmap.conf</code>).</li>
289
290      <li>For NetBSD, at least version 2.0 is recommended.</li>
291
292      <li>For Linux, a 2.6 kernel is recommended. It is also necessary to
293      ensure that your version of <code>glibc</code> has been compiled
294      with support for EPoll.</li>
295
296    </ul>

299<div class="directive-section"><h2><a name="AsyncRequestWorkerFactor" id="AsyncRequestWorkerFactor">AsyncRequestWorkerFactor</a> <a name="asyncrequestworkerfactor" id="asyncrequestworkerfactor">Directive</a></h2>
300<table class="directive">
301<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Limit concurrent connections per process</td></tr>
302<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>AsyncRequestWorkerFactor <var>factor</var></code></td></tr>
303<tr><th><a href="directive-dict.html#Default">Default:</a></th><td><code>2</code></td></tr>
304<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config</td></tr>
305<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>MPM</td></tr>
306<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>event</td></tr>
307<tr><th><a href="directive-dict.html#Compatibility">Compatibility:</a></th><td>Available in version 2.3.13 and later</td></tr>
308</table>
309    <p>The event MPM handles some connections in an asynchronous way, where
310    request worker threads are only allocated for short periods of time as
311    needed, and other connections with one request worker thread reserved per
312    connection. This can lead to situations where all workers are tied up and
313    no worker thread is available to handle new work on established async
314    connections.</p>
315
316    <p>To mitigate this problem, the event MPM does two things:</p>
317    <ul>
318        <li>It limits the number of connections accepted per process, depending on the
319            number of idle request workers;</li>
320        <li>If all workers are busy, it will
321            close connections in keep-alive state even if the keep-alive timeout has
322            not expired. This allows the respective clients to reconnect to a
323            different process which may still have worker threads available.</li>
324    </ul>
325
326    <p>This directive can be used to fine-tune the per-process connection
327    limit. A <strong>process</strong> will only accept new connections if the current number of
328    connections (not counting connections in the "closing" state) is lower
329    than:</p>
330
331    <p class="indent"><strong>
332        <code class="directive"><a href="../mod/mpm_common.html#threadsperchild">ThreadsPerChild</a></code> +
333        (<code class="directive">AsyncRequestWorkerFactor</code> *
334        <var>number of idle workers</var>)
335    </strong></p>
336
337    <p>An estimation of the maximum concurrent connections across all the processes given
338        an average value of idle worker threads can be calculated with:
339    </p>
340
341
342    <p class="indent"><strong>
343        (<code class="directive"><a href="../mod/mpm_common.html#threadsperchild">ThreadsPerChild</a></code> +
344        (<code class="directive">AsyncRequestWorkerFactor</code> *
345        <var>number of idle workers</var>)) *
346        <code class="directive"><a href="../mod/mpm_common.html#serverlimit">ServerLimit</a></code>
347    </strong></p>
348
349    <div class="note"><h3>Example</h3>
350    <pre class="prettyprint lang-config">ThreadsPerChild = 10
351ServerLimit = 4
352AsyncRequestWorkerFactor = 2
353MaxRequestWorkers = 40
354
355idle_workers = 4 (average for all the processes to keep it simple)
356
357max_connections = (ThreadsPerChild + (AsyncRequestWorkerFactor * idle_workers)) * ServerLimit
358                = (10 + (2 * 4)) * 4 = 72</pre>
359
360    </div>
361
362    <p>When all the worker threads are idle, then absolute maximum numbers of concurrent
363        connections can be calculared in a simpler way:</p>
364
365    <p class="indent"><strong>
366        (<code class="directive">AsyncRequestWorkerFactor</code> + 1) *
367        <code class="directive"><a href="../mod/mpm_common.html#maxrequestworkers">MaxRequestWorkers</a></code>
368    </strong></p>
369
370
371    <div class="note"><h3>Example</h3>
372    <pre class="prettyprint lang-config">ThreadsPerChild = 10
373ServerLimit = 4
374MaxRequestWorkers = 40
375AsyncRequestWorkerFactor = 2</pre>
376
377
378    <p>If all the processes have all threads idle then: </p>
379
380    <pre class="prettyprint lang-config">idle_workers = 10</pre>
381
382
383    <p>We can calculate the absolute maximum numbers of concurrent connections in two ways:</p>
384
385    <pre class="prettyprint lang-config">max_connections = (ThreadsPerChild + (AsyncRequestWorkerFactor * idle_workers)) * ServerLimit
386                = (10 + (2 * 10)) * 4 = 120
387
388max_connections = (AsyncRequestWorkerFactor + 1) * MaxRequestWorkers
389                = (2 + 1) * 40 = 120</pre>
390
391    </div>
392
393    <p>Tuning <code class="directive">AsyncRequestWorkerFactor</code> requires knowledge about the traffic handled by httpd in each specific use case, so changing the default value requires extensive testing and data gathering from <code class="module"><a href="../mod/mod_status.html">mod_status</a></code>.</p>
394
395    <p><code class="directive"><a href="../mod/mpm_common.html#maxrequestworkers">MaxRequestWorkers</a></code> was called
396    <code class="directive">MaxClients</code> prior to version 2.3.13. The above value
397    shows that the old name did not accurately describe its meaning for the event MPM.</p>
398
399    <p><code class="directive">AsyncRequestWorkerFactor</code> can take non-integer
400    arguments, e.g "1.5".</p>
