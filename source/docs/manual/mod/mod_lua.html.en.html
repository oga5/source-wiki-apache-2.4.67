38<h3>Summary</h3>
39
40<p>This module allows the server to be extended with scripts written in the
41Lua programming language.  The extension points (hooks) available with
42<code class="module"><a href="../mod/mod_lua.html">mod_lua</a></code> include many of the hooks available to
43natively compiled Apache HTTP Server modules, such as mapping requests to
44files, generating dynamic responses, access control, authentication, and
45authorization</p>
46
47<p>More information on the Lua programming language can be found at the
48<a href="http://www.lua.org/">the Lua website</a>.</p>
49
50<div class="warning"><h3>Warning</h3>
51<p>This module holds a great deal of power over httpd, which is both a 
52strength and a potential security risk. It is <strong>not</strong> recommended 
53that you use this module on a server that is shared with users you do not 
54trust, as it can be abused to change the internal workings of httpd.</p>

99<h2><a name="basicconf" id="basicconf">Basic Configuration</a></h2>
100
101<p>The basic module loading directive is</p>
102
103<pre class="prettyprint lang-config">LoadModule lua_module modules/mod_lua.so</pre>
104
105
106<p>
107<code class="module"><a href="../mod/mod_lua.html">mod_lua</a></code> provides a handler named <code>lua-script</code>,
108which can be used with a <code class="directive"><a href="../mod/core.html#sethandler">SetHandler</a></code> or
109<code class="directive"><a href="../mod/mod_mime.html#addhandler">AddHandler</a></code> directive:</p>
110
111<pre class="prettyprint lang-config">&lt;Files "*.lua"&gt;
112    SetHandler lua-script
113&lt;/Files&gt;</pre>
114
115
116<p>
117This will cause <code class="module"><a href="../mod/mod_lua.html">mod_lua</a></code> to handle requests for files
118ending in <code>.lua</code> by invoking that file's
119<code>handle</code> function.
120</p>
121
122<p>For more flexibility, see <code class="directive">LuaMapHandler</code>.
123</p>

127<h2><a name="writinghandlers" id="writinghandlers">Writing Handlers</a></h2>
128<p> In the Apache HTTP Server API, the handler is a specific kind of hook
129responsible for generating the response.  Examples of modules that include a
130handler are <code class="module"><a href="../mod/mod_proxy.html">mod_proxy</a></code>, <code class="module"><a href="../mod/mod_cgi.html">mod_cgi</a></code>,
131and <code class="module"><a href="../mod/mod_status.html">mod_status</a></code>.</p>
132
133<p><code class="module"><a href="../mod/mod_lua.html">mod_lua</a></code> always looks to invoke a Lua function for the handler, rather than
134just evaluating a script body CGI style. A handler function looks
135something like this:</p>
136
137
138<pre class="prettyprint lang-lua">
139<strong>example.lua</strong><br />
140-- example handler
141
142require "string"
143
144--[[
145     This is the default method name for Lua handlers, see the optional
146     function-name in the LuaMapHandler directive to choose a different
147     entry point.
148--]]
149function handle(r)
150    r.content_type = "text/plain"
151
152    if r.method == 'GET' then
153        r:puts("Hello Lua World!\n")
154        for k, v in pairs( r:parseargs() ) do
155            r:puts( string.format("%s: %s\n", k, v) )
156        end
157    elseif r.method == 'POST' then
158        r:puts("Hello Lua World!\n")
159        for k, v in pairs( r:parsebody() ) do
160            r:puts( string.format("%s: %s\n", k, v) )
161        end
162    elseif r.method == 'PUT' then
163-- use our own Error contents
164        r:puts("Unsupported HTTP method " .. r.method)
165        r.status = 405
166        return apache2.OK
167    else
168-- use the ErrorDocument
169        return 501
170    end
171    return apache2.OK
172end</pre>
173
174
175<p>
176This handler function just prints out the uri or form encoded
177arguments to a plaintext page.
178</p>
179
180<p>
181This means (and in fact encourages) that you can have multiple
182handlers (or hooks, or filters) in the same script.
183</p>

187<h2><a name="writingauthzproviders" id="writingauthzproviders">Writing Authorization Providers</a></h2>
188
189
190<p><code class="module"><a href="../mod/mod_authz_core.html">mod_authz_core</a></code> provides a high-level interface to
191authorization that is much easier to use than using into the relevant
192hooks directly. The first argument to the
193<code class="directive"><a href="../mod/mod_authz_core.html#require">Require</a></code> directive gives
194the name of the responsible authorization provider. For any
195<code class="directive"><a href="../mod/mod_authz_core.html#require">Require</a></code> line,
196<code class="module"><a href="../mod/mod_authz_core.html">mod_authz_core</a></code> will call the authorization provider
197of the given name, passing the rest of the line as parameters. The
198provider will then check authorization and pass the result as return
199value.</p>
200
201<p>The authz provider is normally called before authentication. If it needs to
202know the authenticated user name (or if the user will be authenticated at
203all), the provider must return <code>apache2.AUTHZ_DENIED_NO_USER</code>.
204This will cause authentication to proceed and the authz provider to be
205called a second time.</p>
206
207<p>The following authz provider function takes two arguments, one ip
208address and one user name. It will allow access from the given ip address
209without authentication, or if the authenticated user matches the second
210argument:</p>
211
212<pre class="prettyprint lang-lua">
213<strong>authz_provider.lua</strong><br />
214
215require 'apache2'
216
217function authz_check_foo(r, ip, user)
218    if r.useragent_ip == ip then
219        return apache2.AUTHZ_GRANTED
220    elseif r.user == nil then
221        return apache2.AUTHZ_DENIED_NO_USER
222    elseif r.user == user then
223        return apache2.AUTHZ_GRANTED
224    else
225        return apache2.AUTHZ_DENIED
226    end
227end</pre>
228
229
230<p>The following configuration registers this function as provider
231<code>foo</code> and configures it for URL <code>/</code>:</p>
232<pre class="prettyprint lang-config">LuaAuthzProvider foo authz_provider.lua authz_check_foo
233&lt;Location "/"&gt;
234  Require foo 10.1.2.3 john_doe
235&lt;/Location&gt;</pre>

240<h2><a name="writinghooks" id="writinghooks">Writing Hooks</a></h2>
241
242<p>Hook functions are how modules (and Lua scripts) participate in the
243processing of requests. Each type of hook exposed by the server exists for
244a specific purpose, such as mapping requests to the file system,
245performing access control, or setting mime types:</p>
246
247<table class="bordered"><tr class="header">
248        <th>Hook phase</th>
249        <th><code class="module"><a href="../mod/mod_lua.html">mod_lua</a></code> directive</th>
250        <th>Description</th>
251    </tr>
252<tr>
253        <td>Quick handler</td>
254        <td><code class="directive"><a href="#luaquickhandler">LuaQuickHandler</a></code></td>
255        <td>This is the first hook that will be called after a request has 
256            been mapped to a host or virtual host</td>
257    </tr>
258<tr class="odd">
259        <td>Pre-Translate name</td>
260        <td><code class="directive"><a href="#luahookpretranslatename">LuaHookPreTranslateName</a></code></td>
261        <td>This phase translates the requested URI into a filename on the
262            system, before decoding occurs. Modules such as <code class="module"><a href="../mod/mod_proxy.html">mod_proxy</a></code>
263            can operate in this phase.</td>
264    </tr>
265<tr>
266        <td>Translate name</td>
267        <td><code class="directive"><a href="#luahooktranslatename">LuaHookTranslateName</a></code></td>
268        <td>This phase translates the requested URI into a filename on the 
269            system. Modules such as <code class="module"><a href="../mod/mod_alias.html">mod_alias</a></code> and
270            <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code> operate in this phase.</td>
271    </tr>
272<tr class="odd">
273        <td>Map to storage</td>
274        <td><code class="directive"><a href="#luahookmaptostorage">LuaHookMapToStorage</a></code></td>
275        <td>This phase maps files to their physical, cached or external/proxied storage. 
276            It can be used by proxy or caching modules</td>
277    </tr>
278<tr>
279        <td>Check Access</td>
280        <td><code class="directive"><a href="#luahookaccesschecker">LuaHookAccessChecker</a></code></td>
281        <td>This phase checks whether a client has access to a resource. This 
282            phase is run before the user is authenticated, so beware.
283        </td>
284    </tr>
285<tr class="odd">
286        <td>Check User ID</td>
287        <td><code class="directive"><a href="#luahookcheckuserid">LuaHookCheckUserID</a></code></td>
288        <td>This phase it used to check the negotiated user ID</td>
289    </tr>
290<tr>
291        <td>Check Authorization</td>
292        <td><code class="directive"><a href="#luahookauthchecker">LuaHookAuthChecker</a></code> or 
293            <code class="directive"><a href="#luaauthzprovider">LuaAuthzProvider</a></code></td>
294        <td>This phase authorizes a user based on the negotiated credentials, such as 
295            user ID, client certificate etc.
296        </td>
297    </tr>
298<tr class="odd">
299        <td>Check Type</td>
300        <td><code class="directive"><a href="#luahooktypechecker">LuaHookTypeChecker</a></code></td>
301        <td>This phase checks the requested file and assigns a content type and 
302            a handler to it</td>
303    </tr>
304<tr>
305        <td>Fixups</td>
306        <td><code class="directive"><a href="#luahookfixups">LuaHookFixups</a></code></td>
307        <td>This is the final "fix anything" phase before the content handlers 
308            are run. Any last-minute changes to the request should be made here.</td>
309    </tr>
310<tr class="odd">
311        <td>Content handler</td>
312        <td>fx. <code>.lua</code> files or through <code class="directive"><a href="#luamaphandler">LuaMapHandler</a></code></td>
313        <td>This is where the content is handled. Files are read, parsed, some are run, 
314            and the result is sent to the client</td>
315    </tr>
316<tr>
317        <td>Logging</td>
318        <td><code class="directive"><a href="#luahooklog">LuaHookLog</a></code></td>
319        <td>Once a request has been handled, it enters several logging phases, 
320            which logs the request in either the error or access log. Mod_lua
321            is able to hook into the start of this and control logging output.</td>
322    </tr>
323</table>
324
325<p>Hook functions are passed the request object as their only argument 
326(except for LuaAuthzProvider, which also gets passed the arguments from 
327the Require directive).
328They can return any value, depending on the hook, but most commonly
329they'll return OK, DONE, or DECLINED, which you can write in Lua as
330<code>apache2.OK</code>, <code>apache2.DONE</code>, or
331<code>apache2.DECLINED</code>, or else an HTTP status code.</p>
332
333
334<pre class="prettyprint lang-lua">
335<strong>translate_name.lua</strong><br />
336-- example hook that rewrites the URI to a filesystem path.
337
338require 'apache2'
339
340function translate_name(r)
341    if r.uri == "/translate-name" then
342        r.filename = r.document_root .. "/find_me.txt"
343        return apache2.OK
344    end
345    -- we don't care about this URL, give another module a chance
346    return apache2.DECLINED
347end</pre>
348
349
350
351<pre class="prettyprint lang-lua">
352<strong>translate_name2.lua</strong><br />
353--[[ example hook that rewrites one URI to another URI. It returns a
354     apache2.DECLINED to give other URL mappers a chance to work on the
355     substitution, including the core translate_name hook which maps based
356     on the DocumentRoot.
357
358     Note: Use the early/late flags in the directive to make it run before
359           or after mod_alias.
360--]]
361
362require 'apache2'
363
364function translate_name(r)
365    if r.uri == "/translate-name" then
366        r.uri = "/find_me.txt"
367        return apache2.DECLINED
368    end
369    return apache2.DECLINED
370end</pre>

1051<h2><a name="modifying_buckets" id="modifying_buckets">Modifying contents with Lua filters</a></h2>
1052    
1053    <p>
1054    Filter functions implemented via <code class="directive"><a href="#luainputfilter">LuaInputFilter</a></code> 
1055    or <code class="directive"><a href="#luaoutputfilter">LuaOutputFilter</a></code> are designed as 
1056    three-stage non-blocking functions using coroutines to suspend and resume a 
1057    function as buckets are sent down the filter chain. The core structure of 
1058    such a function is:
1059    </p>
1060    <pre class="prettyprint lang-lua">function filter(r)
1061    -- Our first yield is to signal that we are ready to receive buckets.
1062    -- Before this yield, we can set up our environment, check for conditions,
1063    -- and, if we deem it necessary, decline filtering a request altogether:
1064    if something_bad then
1065        return -- This would skip this filter.
1066    end
1067    -- Regardless of whether we have data to prepend, a yield MUST be called here.
1068    -- Note that only output filters can prepend data. Input filters must use the 
1069    -- final stage to append data to the content.
1070    coroutine.yield([optional header to be prepended to the content])
1071    
1072    -- After we have yielded, buckets will be sent to us, one by one, and we can 
1073    -- do whatever we want with them and then pass on the result.
1074    -- Buckets are stored in the global variable 'bucket', so we create a loop
1075    -- that checks if 'bucket' is not nil:
1076    while bucket ~= nil do
1077        local output = mangle(bucket) -- Do some stuff to the content
1078        coroutine.yield(output) -- Return our new content to the filter chain
1079    end
1080
1081    -- Once the buckets are gone, 'bucket' is set to nil, which will exit the 
1082    -- loop and land us here. Anything extra we want to append to the content
1083    -- can be done by doing a final yield here. Both input and output filters 
1084    -- can append data to the content in this phase.
1085    coroutine.yield([optional footer to be appended to the content])
1086end</pre>

1090<h2><a name="databases" id="databases">Database connectivity</a></h2>
1091    
1092    <p>
1093    Mod_lua implements a simple database feature for querying and running commands
1094    on the most popular database engines (mySQL, PostgreSQL, FreeTDS, ODBC, SQLite, Oracle)
1095    as well as mod_dbd.</p>
1096    <p>
1097    The <code>dbType</code> to use as the first parameter of <code>dbacquire</code>
1098    is case sensitive.</p>
1099    <p>
1100    It should be one of <code>mysql</code>, <code>pgsql</code>, <code>freetds</code>,
1101    <code>odbc</code>, <code>sqlite2</code>, <code>sqlite3</code>, <code>oracle</code>
1102    or <code>mod_dbd</code>.
1103    </p>
1104    <p>The example below shows how to acquire a database handle and return information from a table:</p>
1105    <pre class="prettyprint lang-lua">function handle(r)
1106    -- Acquire a database handle
1107    local database, err = r:dbacquire("mysql", "server=localhost,user=someuser,pass=somepass,dbname=mydb")
1108    if not err then
1109        -- Select some information from it
1110        local results, err = database:select(r, "SELECT `name`, `age` FROM `people` WHERE 1")
1111        if not err then
1112            local rows = results(0) -- fetch all rows synchronously
1113            for k, row in pairs(rows) do
1114                r:puts( string.format("Name: %s, Age: %s&lt;br/&gt;", row[1], row[2]) )
1115            end
1116        else
1117            r:puts("Database query error: " .. err)
1118        end
1119        database:close()
1120    else
1121        r:puts("Could not connect to the database: " .. err)
1122    end
1123end</pre>
1124
1125    <p>
1126    To utilize <code class="module"><a href="../mod/mod_dbd.html">mod_dbd</a></code>, specify <code>mod_dbd</code>
1127    as the database type, or leave the field blank:
1128    </p>
1129    <pre class="prettyprint lang-lua">local database = r:dbacquire("mod_dbd")</pre>
1130
1131    <h3><a name="database_object" id="database_object">Database object and contained functions</a></h3>
1132        
1133        <p>The database object returned by <code>dbacquire</code> has the following methods:</p>
1134        <p><strong>Normal select and query from a database:</strong></p>
1135    <pre class="prettyprint lang-lua">-- Run a statement and return the number of rows affected:
1136local affected, errmsg = database:query(r, "DELETE FROM `tbl` WHERE 1")
1137
1138-- Run a statement and return a result set that can be used synchronously or async:
1139local result, errmsg = database:select(r, "SELECT * FROM `people` WHERE 1")</pre>
1140
1141        <p><strong>Using prepared statements (recommended):</strong></p>
1142    <pre class="prettyprint lang-lua">-- Create and run a prepared statement:
1143local statement, errmsg = database:prepare(r, "DELETE FROM `tbl` WHERE `age` &gt; %u")
1144if not errmsg then
1145    local result, errmsg = statement:query(20) -- run the statement with age &gt; 20
1146end
1147
1148-- Fetch a prepared statement from a DBDPrepareSQL directive:
1149local statement, errmsg = database:prepared(r, "someTag")
1150if not errmsg then
1151    local result, errmsg = statement:select("John Doe", 123) -- inject the values "John Doe" and 123 into the statement
1152end</pre>

1154        <p><strong>Escaping values, closing databases etc:</strong></p>
1155    <pre class="prettyprint lang-lua">-- Escape a value for use in a statement:

1159database:close()
1160
1161-- Check whether a database connection is up and running:
1162local connected = database:active()</pre>

1165    <h3><a name="result_sets" id="result_sets">Working with result sets</a></h3>
1166    
1167    <p>The result set returned by <code>db:select</code> or by the prepared statement functions 
1168    created through <code>db:prepare</code> can be used to
1169    fetch rows synchronously or asynchronously, depending on the row number specified:<br />
1170    <code>result(0)</code> fetches all rows in a synchronous manner, returning a table of rows.<br />
1171    <code>result(-1)</code> fetches the next available row in the set, asynchronously.<br />
1172    <code>result(N)</code> fetches row number <code>N</code>, asynchronously:
1173    </p>
1174    <pre class="prettyprint lang-lua">-- fetch a result set using a regular query:
1175local result, err = db:select(r, "SELECT * FROM `tbl` WHERE 1")
1176
1177local rows = result(0) -- Fetch ALL rows synchronously
1178local row = result(-1) -- Fetch the next available row, asynchronously
1179local row = result(1234) -- Fetch row number 1234, asynchronously
1180local row = result(-1, true) -- Fetch the next available row, using row names as key indexes.</pre>
1181
1182    <p>One can construct a function that returns an iterative function to iterate over all rows 
1183    in a synchronous or asynchronous way, depending on the async argument:
1184    </p>
1185    <pre class="prettyprint lang-lua">function rows(resultset, async)
1186    local a = 0
1187    local function getnext()
1188        a = a + 1
1189        local row = resultset(-1)
1190        return row and a or nil, row
1191    end
1192    if not async then
1193        return pairs(resultset(0))
1194    else
1195        return getnext, self
1196    end
1197end
1198
1199local statement, err = db:prepare(r, "SELECT * FROM `tbl` WHERE `age` &gt; %u")
1200if not err then
1201     -- fetch rows asynchronously:
1202    local result, err = statement:select(20)
1203    if not err then
1204        for index, row in rows(result, true) do
1205            ....
1206        end
1207    end
1208
1209     -- fetch rows synchronously:
1210    local result, err = statement:select(20)
1211    if not err then
1212        for index, row in rows(result, false) do
1213            ....
1214        end
1215    end
1216end</pre>

1219    <h3><a name="closing_databases" id="closing_databases">Closing a database connection</a></h3>
1220        
1221
1222    <p>Database handles should be closed using <code>database:close()</code> when they are no longer
1223    needed. If you do not close them manually, they will eventually be garbage collected and 
1224    closed by <code class="module"><a href="../mod/mod_lua.html">mod_lua</a></code>, but you may end up having too many unused connections to the database 
1225    if you leave the closing up to <code class="module"><a href="../mod/mod_lua.html">mod_lua</a></code>. Essentially, the following two measures are
1226    the same:
1227    </p>
1228    <pre class="prettyprint lang-lua">-- Method 1: Manually close a handle
1229local database = r:dbacquire("mod_dbd")
1230database:close() -- All done
1231
1232-- Method 2: Letting the garbage collector close it
1233local database = r:dbacquire("mod_dbd")
1234database = nil -- throw away the reference
1235collectgarbage() -- close the handle via GC</pre>
1236
1237    
1238    <h3><a name="database_caveat" id="database_caveat">Precautions when working with databases</a></h3>
1239    
1240    <p>Although the standard <code>query</code> and <code>run</code> functions are freely 
1241    available, it is recommended that you use prepared statements whenever possible, to 
1242    both optimize performance (if your db handle lives on for a long time) and to minimize 
1243    the risk of SQL injection attacks. <code>run</code> and <code>query</code> should only
1244    be used when there are no variables inserted into a statement (a static statement). 
1245    When using dynamic statements, use <code>db:prepare</code> or <code>db:prepared</code>.
1246    </p>

1823    As this phase is run before anything else, directives such as <code class="directive"><a href="../mod/core.html#location">&lt;Location&gt;</a></code> or <code class="directive"><a href="../mod/core.html#directory">&lt;Directory&gt;</a></code> are void in this phase, just as 
1824    URIs have not been properly parsed yet.
1825    </p>

1827   context.</p></div>

1847<div class="directive-section"><h2><a name="LuaScope" id="LuaScope">LuaScope</a> <a name="luascope" id="luascope">Directive</a></h2>
1848<table class="directive">
1849<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>One of once, request, conn, thread -- default is once</td></tr>
1850<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>LuaScope once|request|conn|thread|server [min] [max]</code></td></tr>
1851<tr><th><a href="directive-dict.html#Default">Default:</a></th><td><code>LuaScope once</code></td></tr>
1852<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config, virtual host, directory, .htaccess</td></tr>
1853<tr><th><a href="directive-dict.html#Override">Override:</a></th><td>All</td></tr>
1854<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Extension</td></tr>
1855<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_lua</td></tr>
1856</table>
1857    <p>Specify the life cycle scope of the Lua interpreter which will
1858    be used by handlers in this "Directory." The default is "once"</p>
1859
1860   <dl>
1861    <dt>once:</dt> <dd>use the interpreter once and throw it away.</dd>
1862
1863    <dt>request:</dt> <dd>use the interpreter to handle anything based on
1864             the same file within this request, which is also
1865             request scoped.</dd>
1866
1867    <dt>conn:</dt> <dd>Same as request but attached to the connection_rec</dd>
1868
1869    <dt>thread:</dt> <dd>Use the interpreter for the lifetime of the thread 
1870            handling the request (only available with threaded MPMs).</dd>
1871
1872    <dt>server:</dt>  <dd>This one is different than others because the
1873            server scope is quite long lived, and multiple threads
1874            will have the same server_rec. To accommodate this,
1875            server scoped Lua states are stored in an apr
1876            resource list. The <code>min</code> and <code>max</code> arguments 
1877            specify the minimum and maximum number of Lua states to keep in the 
1878            pool.</dd>
1879   </dl>
1880    <p>
1881    Generally speaking, the <code>thread</code> and <code>server</code> scopes 
1882    execute roughly 2-3 times faster than the rest, because they don't have to 
1883    spawn new Lua states on every request (especially with the event MPM, as 
1884    even keepalive requests will use a new thread for each request). If you are 
1885    satisfied that your scripts will not have problems reusing a state, then 
1886    the <code>thread</code> or <code>server</code> scopes should be used for 
1887    maximum performance. While the <code>thread</code> scope will provide the 
1888    fastest responses, the <code>server</code> scope will use less memory, as 
1889    states are pooled, allowing f.x. 1000 threads to share only 100 Lua states, 
1890    thus using only 10% of the memory required by the <code>thread</code> scope.
1891    </p>
