42<h3>Sommaire</h3>
43
44        <p>
45	Ce module permet de gérer les propriétés courantes des domaines pour un
46	ou plusieurs serveurs virtuels. Il fournit deux fonctionnalités
47	principales : la première permet la supervision et le renouvellement des
48	certificats TLS via le protocole ACME (<a href="https://tools.ietf.org/html/rfc8555">RFC 8555</a>). Le module
49	effectue le renouvellement des certificats avant leur expiration
50	afin d'éviter une interruption des services internet. Il est possible de
51	monitorer l'état de tous les certificats gérés par mod_md et de configurer
52	le serveur de façon à ce qu'il envoie des notifications de
53	renouvellement, d'expiration ou d'erreur personnalisées.
54	</p><p>
55	La seconde fonctionnalité principale fournit une implémentation
56	alternative de l'agrafage OCSP, et ceci aussi bien pour les certificats
57	gérés par mod_md que pour les certificats que vous gérez vous-même.
58	Composant nécessaire pour tout site https, l'agrafage OCSP influence la
59	vitesse de chargement des pages et suivant la configuration, la
60	disponibilité de ces dernières. Vous trouverez plus de détails dans la section
61	agrafage ci-dessous.
62        </p><p>
63        L'autorité ACME par défaut pour la gestion des certificats est <a href="https://letsencrypt.org/">Let's Encrypt</a>, mais il est possible
64	de configurer une autre CA si cette dernière supporte le protocole.
65        </p>
66        
67        <p>Exemple de configuration simple :</p>
68        
69        <div class="note"><h3>TLS dans un contexte de serveur virtuel</h3>
70        <pre class="prettyprint lang-config">MDomain example.org
71
72&lt;VirtualHost *:443&gt;
73    ServerName example.org
74    DocumentRoot htdocs/a
75
76    SSLEngine on
77    # aucun certificat spécifié
78&lt;/VirtualHost&gt;</pre>
79
80        <p>
81            Au démarrage, un serveur ainsi configuré contactera <a href="https://letsencrypt.org/">Let's Encrypt</a> pour demander un
82	    certificat pour le domaine considéré. Si Let's Encrypt peut vérifier
83	    le propriétaire du domaine, le module obtiendra le certificat et sa
84	    chaîne de certification, le stockera dans son système de fichiers
85	    (voir la directive <code class="directive"><a href="#mdstoredir">MDStoreDir</a></code>) et le proposera au prochain
86	    redémarrage à <code class="module"><a href="../mod/mod_ssl.html">mod_ssl</a></code>.
87        </p><p>
88            Ce processus se déroule pendant l'exécution du serveur. Tous les
89	    autres serveurs virtuels continueront à fonctionner normalement,
90	    mais tant que le certificat ne sera pas disponible, toute requête
91	    pour le domaine considéré génèrera une réponse du type '503 Service
92	    Unavailable'. 
93        </p>
94        </div>
95
96        <div class="note"><h3>Prérequis</h3>
97        <p>
98            Pour pouvoir être utilisé, ce module nécessite le chargement
99	    préalable du module <code class="module"><a href="../mod/mod_watchdog.html">mod_watchdog</a></code>.
100        </p><p>
101            Pour que Let's Encrypt puisse signer et renouveler votre certificat,
102	    votre serveur doit être accessible depuis l'internet public sur le port 80
103	    (http:) et/ou 443 (https:), à moins que votre serveur soit configuré
104	    pour utiliser les vérifications DNS - pour plus de détails, voir
105	    "certificats génériques".
106        </p><p>
107            Le module choisit une des méthodes proposées par Let's Encrypt. En
108	    général, LE propose des méthodes de vérification sur les ports ou le
109	    DNS et Apache choisit une des méthodes disponibles.    
110        </p><p>
111            Pour déterminer quelles méthodes sont disponibles, le module
112	    consulte les ports sur lesquels écoute Apache httpd. Si le port 80 en
113	    fait partie, le module supposera que la vérification http: nommée
114	    http-01 est disponible. Si le port 443 en fait aussi partie, la
115	    vérification https: nommée tls-alpn-01 sera ajoutée à la liste des
116	    méthodes disponibles. Enfin, si la directive <code class="directive"><a href="#mdchallengedns01">MDChallengeDns01</a></code> est définie, la méthode
117	    de vérification dns-01 sera aussi ajoutée. 
118        </p><p>
119            Si votre configuration est plus complexe, deux méthodes permettent
120	    d'orienter ce choix. En premier lieu, voyez du côté de la directive
121	    <code class="directive"><a href="#mdportmap">MDPortMap</a></code> si le serveur se
122	    trouve derrière un redirecteur de port comme un pare-feu. En second
123	    lieu, vous pouvez court-circuiter entièrement le processus de choix
124	    du module en définissant directement la directive <code class="directive"><a href="#mdcachallenges">MDCAChallenges</a></code>.  
125        </p>
126        </div>
127
128        <div class="note"><h3>Vérifications https:</h3>
129        <p>
130            Pour la vérification de domaine via le protocole TLS, le nom de la
131	    méthode correspondante est "tls-alpn-01". Le serveur Apache doit
132	    alors être en écoute sur le port 443 (voir la directive <code class="directive"><a href="#mdportmap">MDPortMap</a></code> si vous redirigez ce port vers
133	    un autre).
134        </p><p>
135            Let's Encrypt ouvrira alors une connexion TLS avec Apache en
136	    utilisant l'indicateur spécial "acme-tls/1" (cette portion
137	    indication de TLS se nomme ALPN, d'où le nom de la méthode de
138	    vérification. ALPN est aussi utilisé par les navigateurs pour ouvrir
139	    une connexion HTTP/2.
140        </p><p>
141            Si vous ne souhaitez cependant qu'aucun de vos sites ne soit
142	    accessible sur le port 80, vous pouvez laiser ce dernier ouvert et
143	    rediriger toutes les requêtes vers vos sites en https:. Pour
144	    ce faire, utilisez la directive <code class="directive"><a href="#mdrequirehttps">MDRequireHttps</a></code> décrite plus loin. Votre
145	    serveur pourra alors continuer à répondre au requêtes en http: en
146	    provenance de Let's Encrypt.
147	    Comme dans le cas du protocole HTTP/2, vous pouvez configurer ceci
148	    de la manière suivante :
149        </p>
150	<pre class="prettyprint lang-config">Protocols h2 http/1.1 acme-tls/1</pre>
151
152        <p>
153            La méthode de vérification "tls-alpn-01" sera alors disponible.
154	</p>
155        </div>
156	<div class="note"><h3>Certificats génériques</h3>
157        <p>
158            Les certificats génériques sont supportés à partir de la version 2.x
159	    de mod_md, mais leur obtention n'est pas triviale. Let's Encrypt
160	    impose pour ces derniers la vérification "dns-01".
161	    Aucune autre n'est considérée comme suffisamment efficace.
162        </p><p>
163	    Apache ne peut cependant pas implémenter cette vérification de
164	    lui-même . Comme son nom l'indique, "dns-01" vous demande de
165	    présenter certains enregistrement DNS spécifiques à votre domaine
166	    qui doivent contenir certaines données de vérification. Vous devez
167	    donc être en mesure d'éditer et modifier les enregistrements DNS de
168	    votre domaine.
169        </p><p>
170            Si c'est le cas, vous pouvez procéder via mod_md. Supposons que vous
171	    disposiez pour cela du script /usr/bin/acme-setup-dns ; vous
172	    configurez alors Apache comme suit :
173        </p>
174        <pre class="prettyprint lang-config">MDChallengeDns01 /usr/bin/acme-setup-dns</pre>
175
176        <p>
177            Apache fera alors appel à ce script lorsqu'il aura besoin de
178	    définir ou détruire un enregistrement DNS de vérification pour le
179	    domaine considéré.
180        </p><p>
181            Supposons ainsi que vous souhaitiez obtenir un certificat pour
182	    *.mydomain.com ;  mod_md va appeler :
183        </p>
184        <pre class="prettyprint lang-config">/usr/bin/acme-setup-dns setup mydomain.com challenge-data
185# ceci nécessite de supprimer tout enregistrement DNS TXT pour
186# _acme-challenge.mydomain.com et d'en créer un nouveau dont le contenu sera
187# "challenge-data"</pre>
188
189        <p>
190            il appellera ensuite :
191        </p>
192        <pre class="prettyprint lang-config">/usr/bin/acme-setup-dns teardown mydomain.com
193# ceci nécessite de supprimer tout enregistrement DNS TXT pour
194# _acme-challenge.mydomain.com</pre>

198        <div class="note"><h3>Monitoring</h3>
199            <p>Apache possède un module de monitoring standard :
200	    <code class="module"><a href="../mod/mod_status.html">mod_status</a></code>. mod_md y ajoute une section et facilite
201	    le monitoring de votre domaine.
202            </p><p>
203                Vous pouvez alors visualiser tous vos domaines gérés par ordre
204		alphabétique, les noms de domaine qu'ils contiennent, un état
205		global, les date d'expiration ainsi que des paramètres
206		spécifiques. Ces derniers comprennent la périodicité de
207		renouvellement que vous avez sélectionnée (ou la valeur par
208		défaut), la CA (autorité de certification) utilisée, etc...
209            </p><p>
210                La colonne "Renewal" montre des rapports d'activité ou d'erreur
211		à propos des renouvellements de certificats, ce qui devrait
212		faciliter la vie des utilisateurs qui souhaitent savoir si tout
213		fonctionne correctement ou si des problèmes se produisent.
214            </p><p>
215                Si un des domaines gérés provoque une erreur, elle apparaîtra
216		aussi ici, ce qui vous permettra de visualiser les éventuels
217		problèmes sans devoir vous plonger dans les journaux du serveur.
218            </p><p>
219                Il existe aussi un nouveau gestionnaire, "md-status", qui peut
220		vous fournir les informations à propos des domaines gérés à
221		partir de "server-status" et au format JSON. Vous pouvez le
222		configurer comme suit sur votre serveur :
223            </p>
224            <pre class="prettyprint lang-config">&lt;Location "/md-status"&gt;
225  SetHandler md-status
226&lt;/Location&gt;</pre>
227
228            <p>
229                Comme pour "server-status", vous devez
230		ajouter les autorisations nécessaires. 
231            </p><p>
232                Si vous ne souhaitez recevoir l'état JSON que pour un domaine
233		spécifique, ajoutez le simplement à votre URL d'état :
234            </p>
235            <pre class="prettyprint lang-config">&gt; curl https://&lt;yourhost&gt;/md-status/another-domain.org
236{
237  "name": "another-domain.org",
238  "domains": [
239    "another-domain.org",
240    "www.another-domain.org"
241  ],
242  ...</pre>
243
244            <p>
245                Cet état JSON montre aussi un journal des renouvellements de
246		certificats :
247            </p>
248            <pre class="prettyprint lang-config">{
249"when": "Wed, 19 Jun 2019 14:45:58 GMT",
250"type": "progress", "detail": "The certificate for the managed domain has been renewed successfully and can be used. A graceful server restart now is recommended."
251},{
252"when": "Wed, 19 Jun 2019 14:45:58 GMT",
253"type": "progress", "detail": "Retrieving certificate chain for test-901-003-1560955549.org"
254},{
255"when": "Wed, 19 Jun 2019 14:45:58 GMT",
256"type": "progress", "detail": "Waiting for finalized order to become valid"
257},{
258"when": "Wed, 19 Jun 2019 14:45:50 GMT",
259"type": "progress", "detail": "Submitting CSR to CA for test-901-003-1560955549.org"
260},
261...</pre>
262
263            <p>
264                Vous trouverez aussi ces informations dans le fichier "job.json"
265		dans votre répertoire de test et, s'il est activé, dans le
266		répertoire des domaines. Vous pourrez ainsi les consulter à tout
267		moment. 
268            </p><p>
269                Enfin, la directive <code class="directive"><a href="#mdcertificatestatus">MDCertificateStatus</a></code> donne accès au
270		informations à propos du certificat spécifié au format JSON.
271            </p>

274	<div class="note"><h3>Agrafage</h3>
275            <p>
276                Si vous voulez commencer par tester l'agrafage pour un seul
277		domaine géré, utilisez cette configuration :
278            </p>
279            <pre class="prettyprint lang-config">&lt;MDomain mydomain.net&gt;
280  MDStapling on
281&lt;/MDomain&gt;</pre>
282
283            <p>
284		et utilisez 'server-status' et/ou <code class="directive"><a href="#mdmessagecmd">MDMessageCmd</a></code> pour voir comment tout
285		cela fonctionne. Vous pourrez alors vérifier si l'information
286		d'agrafage est présente, sa durée de validité, son origine et à
287		quel moment elle sera rafraîchie.
288            </p><p>
289                Si tout fonctionne comme vous le souhaitez, vous pouvez définir
290		cette configuration pour tous les certificats ou seulement vos
291		certificats gérés.
292            </p><p>
293                De nombreux sites utilisent l'implémentation d'agrafage
294		existante de mod_ssl depuis des années. Les implémentations par
295		mod-ssl et mod_md présentent deux différences principales :
296            </p>
297            <ol>
298                <li>Lecture des informations à la demande ou de manière planifiée
299		: mod_ssl extrait les informations d'agrafage lorsque le besoin
300		s'en fait sentir, par exemple lors d'une nouvelle connexion. mod_md
301		quant à lui, extrait ces informations au démarrage du serveur et
302		lorsqu'elles ont atteint les deux tiers de leur durée de vie.</li>
303                <li>Conservation des informations en mémoire ou de manière
304		persistante : mod_ssl <em>peut</em> conserver ces informations
305		de manière persistante, mais la plupart des configurations
306		exemples utilisent un cache en mémoire. mod_md quant à lui,
307		stocke systématiquement les informations dans le système de
308		fichiers.</li>
309            </ol>
310            <p>
311                Si par malchance vous redémarrez votre serveur alors que le
312		service OCSP de votre CA est en panne, les utilisateurs ne
313		pourront plus atteindre vos sites. Sans persistance des
314		informations, votre serveur n'est plus en mesure de fournir au
315		client les données nécessaires, et le navigateur client ne peut
316		pas les obtenir lui-même car le service OCSP ne répond pas. 
317            </p><p>
318                Avec l'implémentation de mod_md, l'information d'agrafage est
319		stockée de manière persistante, et elle peut donc être réchargée
320		au démarrage du serveur et être ainsi disponible pour les
321		connexions entrantes. Un jour ou deux avant expiration des
322		informations, mod_md va les renouveler, ce qui permet de faire
323		face à un temps d'indisponibilité du service OCSP assez long.
324            </p><p>
325                Pour conserver une compatibilité ascendante, l'implémentation de
326		mod_ssl n'a pas pu être modifiée en profondeur. Par exemple,
327		mod_ssl est incapable d'ajouter une dépendance à mod_watchdog
328		sans rendre inutilisables de nombreuses configurations
329		existantes qui ne chargent pas ce module.
330            </p>

1030		qu'il se termine. Un code de retour autre que 0 doit indiquer

1034		renouvellement du certificat est censé se produire suffisammant
1035		tôt pour éviter toute interruption de service. Cet évènement est
1036		signalé au plus une fois par heure.
1037            </p><p>
1038                L'évènement "expiring", quant à lui, doit être pris au sérieux.
1039		Il se produit lorsque la valeur de <code class="directive"><a href="#mdwarnwindow">MDWarnWindow</a></code> est atteinte. Par
1040		défaut, cette valeur correspond à 10% de la durée de vie du
1041		certificat, donc actuellement pour Let's Encrypt, 9 jours avant
1042		expiration du certificat. Le message d'avertissement est répété
1043		au plus une fois par jour.
1044            </p><p>
1045                'renewed' indique qu'un nouveau certificat a été obtenu et
1046		se trouve dans la zone intermédiaire du magasin MD. Il sera
1047		activé au prochain restart/reload du serveur.
1048            </p><p>
1049		'installed' indique qu'un nouveau certificat a été transféré
1050		depuis la zone intermédiaire vers la zone des domaines du
1051		magasin MD. Cet évènement se produit lors d'un restart/reload du
1052		serveur. A la différence des autres commandes,
1053		<code class="directive">MDMessageCmd</code> s'exécute avec les
1054		permissions de root (sur les systèmes *nix) et a donc accès aux
1055		fichiers de certificats (et aux clés). Les certificats
1056		nécessaires à d'autres applications ou possédant des formats
1057		différents peuvent être traités suite à cet évènement.
1058            </p><p>
1059                Un évènement de type 'renewing' est déclenché avant le démarrage
1060		du processus de renouvellement pour le domaine géré. Si dans ce
1061		cas la commande renvoie une valeur non nulle, le renouvellement
1062		sera interrompu et tenté à nouveau au cycle suivant. Certaines
1063		configurations de clusters l'utilisent pour n'effectuer le
1064		renouvellement que sur un seul noeud.
1065            </p><p>
1066                Un évènement de type 'challenge-setup:type:domain' est déclenché
1067		lorsque les données de vérification pour un domaine ont été
1068		créées. Il est invoqué avant qu'il soit demandé au serveur ACME
1069		de les vérifier. type contient une des méthodes de vérification
1070		ACME. Il est invoqué pour chaque nom DNS d'un MDomain. Les
1071		configurations de clusters peuvent utiliser cet évènement pour
1072		distribuer les fichiers de vérification à tous les noeuds.
1073            </p><p>
1074		Un évènement de type ocsp-errored est déclenché lorsque le
1075		<code class="directive"><a href="#mdstapling">MDStapling</a></code> est activé
1076		pour un domaine, et indique qu'une erreur s'est produite en
1077		essayant d'obtenir la réponse OCSP  de l'autorité de
1078		certification. mod_md essaiera à nouveau d'obtenir cette
1079		réponse.

1249<div class="directive-section"><h2><a name="mdportmap" id="mdportmap">Directive</a> <a name="MDPortMap" id="MDPortMap">MDPortMap</a></h2>
1250<table class="directive">
1251<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Mappage des ports externes avec les ports internes pour
1252	vérifier à qui appartient le domaine.</td></tr>
1253<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>MDPortMap <var>map1</var> [ <var>map2</var> ]</code></td></tr>
1254<tr><th><a href="directive-dict.html#Default">Défaut:</a></th><td><code>MDPortMap http:80 https:443</code></td></tr>
1255<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>configuration globale</td></tr>
1256<tr><th><a href="directive-dict.html#Status">Statut:</a></th><td>Expérimental</td></tr>
1257<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_md</td></tr>
1258</table>
1259            <p>
1260                Le protocole ACME propose deux méthodes pour vérifier à qui
1261		appartient le domaine via HTTP : la première utilise les URLs en
1262		"http:" (port 80) et la deuxième les URLs en "https:" (port
1263		443). Si votre serveur n'est accessible sur aucun
1264		de ces ports, ACME ne pourra fonctionner que si vous configurez
1265		votre serveur DNS de manière adéquate (voir la directive <code class="directive"><a href="#mdchallengedns01">MDChallengeDns01</a></code>).
1266            </p><p>
1267                Sur la plupart des serveurs publics, "http:" arrive sur le
1268		port 80 et "https:" sur le port 443. Ce module vérifie les ports
1269		sur lesquels votre serveur Apache est en écoute et suppose
1270		qu'ils sont disponibles. Autrement dit, si votre serveur n'est
1271		pas en écoute sur le port 80, le module suppose que les requêtes
1272		en "http:" en provenance d'internet ne seront pas traitées.
1273            </p><p>
1274		Ce raisonnement est légitime, mais il peut s'avérer faux.
1275		Par exemple, même si votre serveur est effectivement en écoute
1276		sur le port 80, votre pare-feu peut bloquer ce dernier. "http:"
1277		ne sera alors disponible que sur votre intranet. Dans ce cas, le
1278		module va supposer de manière erronée que Let's Encrypt peut
1279		effectuer des vérifications en "http:" avec votre serveur. Ces
1280		dernières échouerons car elles auront été rejetées par votre
1281		pare-feu.
1282            </p>
1283            <div class="example"><h3>Exemple</h3><pre class="prettyprint lang-config">MDPortMap http:- https:8433</pre>
1284</div>
1285            <p>
1286                L'exemple précédent montre comment spécifier que les requêtes en
1287		"http:" en provenance d'internet n'arriveront jamais. En outre,
1288		il indique que les requêtes en "https:" arriveront sur le port
1289		8433.
1290            </p><p>
1291		Cette définition peut s'avérer nécessaire si vous faites de la
1292		redirection de port ; votre serveur peut ainsi être accessible
1293		depuis l' Internet sur le port 443, alors que le port local
1294		utilisé par httpd sera différent. Par exemple, votre serveur
1295		peut n'être en écoute que sur les ports 8443 et 8000, mais
1296		accessible depuis internet sur les ports 443 et 80.
1297            </p>

1313		plusieurs types de clés privées et le module obtiendra un
1314		certificat pour chaque clé.

1317		et une valeur inférieure ne sera pas acceptée. Des valeurs
1318		supérieures offriront une plus grande sécurité mais seront plus

1325		créés pour le domaine concerné. Lors d'une connexion avec un
1326		client, c'est la première clé supportée par ce dernier qui sera
1327		utilisée.
1328            </p><p>
1329                Comme les clés et certificats EC sont plus petits, vous pouvez
1330		les proposer en premier pour tous les clients modernes
1331		compatibles, ce qui peut accélérer la phase de négociation.
1332		Ajoutez tout de même une clé RSA pour supporter les clients plus
1333		anciens.		
1334            </p>
1335            <div class="example"><h3>Exemple</h3><pre class="prettyprint lang-config">MDPrivateKeys secp256r1 rsa3072</pre>
1336</div>

1339		exemple, Let's encrypt supporte les courbes elliptiques
1340		'secp256r1' et 'secp384r1'.

1342                Chaque type de clé et certificat est stocké dans son fichier
1343		propre au sein de l'espace de stockage MD. Le type de clé
1344		constitue une partie du nom de fichier avec une convention de
1345		nommage présentant une compatibilité ascendante avec les
1346		certificats RSA. Vous pouvez ainsi continuer à partager ces
1347		fichiers avec les autres applications.
1348            </p><p>
1349		
1350		Notez que cette directive n'aura d'effet que sur les nouvelles
1351		clés. Toute clé préexistante ne sera pas affectée. En outre,
1352		seules les clés privées générées pour les certificats sont
1353		concernées, les clés de comptes ACME n'étant pas affectées.
1354            </p>

1527            <p>Cette directive facilite la migration de vos domaines gérés de
1528	    http: vers https:. Dans l'exemple suivant,
1529            </p>
1530            <div class="example"><h3>Exemple</h3><pre class="prettyprint lang-config">MDRequireHttps temporary</pre>
1531</div>
1532            <p>vous indiquez que vous désirez que pour l'instant, tout le trafic via des URLs en
1533	    http: doit être redirigé vers des URLs en https:. Cette directive
1534	    est sans risque et vous pouvez la désactiver à tout moment.
1535            </p><p>
1536                <strong>Ce qui suit par contre, a des conséquences : </strong>si
1537		vous souhaitez que les clients <strong>n'utilisent plus</strong>
1538		d'URLs en http:, spécifiez :
1539            </p>
1540            <div class="example"><h3>Permanent (pour au moins 6 mois !)</h3><pre class="prettyprint lang-config">MDRequireHttps permanent</pre>
1541</div>
1542            <p>Cette directive a deux effets :
1543            </p>
1544            <ol>
1545                <li>Toutes les requêtes pour une ressource en <code>http:</code>
1546		sont redirigées vers la même requête en remplaçant le protocole
1547		<code>http:</code> par <code>https:</code> et en renvoyant le code
1548		d'état <code>301</code>. Ce dernier indique aux clients que
1549		cette modification est permanente et qu'ils doivent mettre à
1550		jour leurs liens en conséquence.
1551                </li>
1552		<li>Toutes les réponses aux requêtes en <code>https:</code>
1553		comporteront l'en-tête <code>Strict-Transport-Security</code>
1554		avec une durée de vie de six mois. Cela indique au navigateur
1555		qu'il ne devra <strong>jamais</strong> utiliser
1556		<code>http:</code> (pendant six mois) lorsqu'il formulera une
1557		requête pour le domaine concerné.  Avec cette information, les
1558		navigateurs refuseront de contacter votre site en mode non
1559		chiffré. Ceci interdit à des middlewares malicieux de dégrader
1560		les connexions et d'écouter/manipuler le trafic. C'est une bonne
1561		chose, mais cette configuration ne peut pas être désactivée
1562		aussi simplement que la configuration temporaire ci-dessus.
1563                </li>
1564            </ol>
1565            <p>Vous pouvez obtenir le même résultat de manière simple avec
1566	    <code class="module"><a href="../mod/mod_alias.html">mod_alias</a></code> et une configuration basée sur la
1567	    directive <code class="directive"><a href="../mod/mod_alias.html#redirect">Redirect</a></code>. Si
1568	    vous le faites vous-même, assurez-vous d'exclure les chemins
1569	    /.well-known/* de votre redirection, sinon mod_md
1570	    aura des difficultés pour signer les nouveaux certificats.
1571            </p>
1572            <p>Si vous effectuez cette configuration au niveau global, elle
1573	    s'appliquera à tous les domaines gérés. Si vous souhaitez qu'elle ne
1574	    s'applique qu'à un domaine spécifique, utilisez :
1575            </p>
1576            <div class="example"><h3>Exemple</h3><pre class="prettyprint lang-config">&lt;MDomain xxx.yyy&gt;
1577  MDRequireHttps temporary
1578&lt;/MDomain&gt;</pre>
1579</div>

1720<tr><th><a href="directive-dict.html#Compatibility">Compatibilité:</a></th><td>Disponible à partir de la version 2.4.42 du serveur HTTP
1721	Apache</td></tr>
1722</table>
1723            <p>
1724                Cette directive permet de spécifier la durée au bout de laquelle
1725		les données OCSP utilisées pour l'agrafage doivent être
1726		supprimées du magasin. Par défaut, ces informations sont
1727		supprimées lors d'un restart/reload du serveur si elles ont plus
1728		de sept jours. Ceci permet de limiter la taille du magasin
1729		lorsque les certificats sont renouvelés et/ou reconfigurés
1730		fréquemment.
1731            </p><p>

1736<div class="directive-section"><h2><a name="mdstaplingrenewwindow" id="mdstaplingrenewwindow">Directive</a> <a name="MDStaplingRenewWindow" id="MDStaplingRenewWindow">MDStaplingRenewWindow</a></h2>
1737<table class="directive">
1738<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Contrôle l'ancienneté des réponses OCSP au dela de laquelle
1739	ces dernières seront renouvelées.</td></tr>
1740<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>MDStaplingRenewWindow <var>duration</var></code></td></tr>
1741<tr><th><a href="directive-dict.html#Default">Défaut:</a></th><td><code>MDStaplingRenewWindow 33%</code></td></tr>
1742<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>configuration globale</td></tr>
1743<tr><th><a href="directive-dict.html#Status">Statut:</a></th><td>Expérimental</td></tr>
1744<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_md</td></tr>
1745<tr><th><a href="directive-dict.html#Compatibility">Compatibilité:</a></th><td>Disponible à partir de la version 2.4.42 du serveur HTTP
1746	Apache</td></tr>
1747</table>
1748            <p>
1749		Si la durée de validité d'un réponse OCSP passe en dessous de
1750		<var>duration</var>, <code class="module"><a href="../mod/mod_md.html">mod_md</a></code> va tenter de la
1751		renouveler.
1752            </p><p>
1753                La CA à l'origine du certificat fournit aussi en général le
1754		service de réponse OCSP et détermine la durée de validité de sa
1755		réponse signée à propos de la validité du certificat. Plus
1756		longtemps une réponse sera valide, plus longtemps elle pourra
1757		être mise en cache, ce qui arrange tout le monde en matière de
1758		performances. Plus courte sera la validité d'une réponse, plus
1759		vite seront envoyées des révocations de certificats aux clients.
1760		Il est donc important de prendre en compte la qualité de
1761		service.
1762            </p><p>
1763                En ajustant la durée de validité des réponses vous-même, vous
1764		pouvez contrôler une partie du processus. Si vous spécifiez une
1765		durée de vie importante (autrement dit si vous spécifiez un
1766		petit pourcentage de validité avant que l'information n'expire),
1767		vous assurer un temps de mise en cache maximal, mais une
1768		interruption du service OCSP (par exemple un arrêt pour
1769		maintenance) aura plus de chance de vous affecter. Si vous
1770		spécifiez un pourcentage de temps avant expiration plus
1771		important, les mises à jour seront plus fréquentes, ce qui va
1772		augmenter la charge de l'infrastructure de serveurs du CA et
1773		nécessiter d'avantage de coordination entre les processus
1774		enfants de votre propre serveur.
1775            </p><p>
1776                La valeur par défaut choisie est de 33%, ce qui signifie que la
1777		demande de renouvellement interviendra lorsque la durée de vie
1778		de la réponse OCSP passera en dessous de 33%. Pour une CA qui
1779		fournit des réponses OCSP avec une durée de vie de 3 jours, cela
1780		implique 2 jours de mise en cache et 1 jour pour les tentatives
1781		de renouvellement. Pour affecter votre domaine, une interruption
1782		de service devra donc avoir une durée supérieure à 1 jour.
1783            </p><p>
1784                Vous pouvez aussi définir de manière absolue la durée de vie
1785		restante, par exemple `2d` pour 2 jours. 
1786            </p>

1793	seront stockées les données à propos des domaines gérés.</td></tr>

1801                Cette directive permet de définir le répertoire dans le système
1802		de fichiers local où seront stockées les données à propos des
1803		domaines gérés. Il s'agit d'un chemin absolu ou relatif à la
1804		racine du serveur. Par défaut, le répertoire "md" sera créé à la
1805		racine de votre serveur.
1806            </p><p>
1807		Si vous souhaitez changer de répertoire et si ce dernier
1808		contient déjà des données, copiez tout d'abord les données vers
1809		le nouveau répertoire, puis modifier la configuration et
1810		redémarrez le serveur. Si vous commencez par modifier la
1811		configuration et redémarrer le serveur sans copier les données,
1812		ce dernier croira que les certificats sont absents et il tentera
1813		d'en obtenir de nouveaux.
1814            </p>

1818<div class="directive-section"><h2><a name="mdstorelocks" id="mdstorelocks">Directive</a> <a name="MDStoreLocks" id="MDStoreLocks">MDStoreLocks</a></h2>
1819<table class="directive">
1820<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Configure le verrouillage du magasin pour les mises à jour</td></tr>
1821<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>MDStoreLocks on|off|<var>duration</var></code></td></tr>
1822<tr><th><a href="directive-dict.html#Default">Défaut:</a></th><td><code>MDStoreLocks off</code></td></tr>
1823<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>configuration globale</td></tr>
1824<tr><th><a href="directive-dict.html#Status">Statut:</a></th><td>Expérimental</td></tr>
1825<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_md</td></tr>
1826<tr><th><a href="directive-dict.html#Compatibility">Compatibilité:</a></th><td>Disponible à partir de la version 2.4.55 du serveur HTTP
1827	Apache</td></tr>
1828</table>
1829            <p>
1830                Définissez cette directive pour utiliser un fichier verrou au
1831		démarrage du serveur lorsque <code class="directive">MDStoreDir</code>
1832		est synchronisé avec la configuration du serveur et si les
1833		certificats renouvelés sont activés.
1834            </p><p>
1835                Le verrouillage a été implémenté pour les configurations de
1836		cluster où MDStoreDir appartient à un système de fichiers
1837		partagé. L'activation des certificats renouvelés sera alors
1838		protégée lorsque plusieurs noeuds du cluster sont redémarrés ou
1839		reconfigurés simultanément ; ceci à condition bien entendu que
1840		le système de fichiers partagé prenne en charge le verrouillage
1841		de fichier.
1842            </p><p>
1843                Le temps d'attente par défaut pour obtenir le verrou est de 5
1844		secondes. Si le verrou ne peut pas être obtenu, une erreur est
1845		enregistrée dans le journal et le démarrage du serveur se
1846		poursuit ; de ce fait, un des noeuds du cluster pourra encore
1847		utiliser les anciens certificats par la suite.
1848            </p><p>
1849                Un délai d'attente plus long réduira cette probabilité, mais
1850		pourra aussi retarder les redémarrages et reconfigurations du
1851		serveur dans le cas où les verrous ne sont pas correctement
1852		gérés dans le système de fichiers sous-jacent. Un verrou ne doit
1853		être maintenu par une instance de httpd que pendant une courte
1854		durée.
1855            </p>
