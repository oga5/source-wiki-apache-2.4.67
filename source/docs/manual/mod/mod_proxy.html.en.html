39    <div class="warning"><h3>Warning</h3>
40      <p>Do not enable proxying with <code class="directive"><a href="#proxyrequests">ProxyRequests</a></code> until you have <a href="#access">secured your server</a>. Open proxy servers are dangerous both to your
41      network and to the Internet at large.</p>

44    <p><code class="module"><a href="../mod/mod_proxy.html">mod_proxy</a></code> and related modules implement a
45    proxy/gateway for Apache HTTP Server, supporting a number of popular
46    protocols as well as several different load balancing algorithms.
47    Third-party modules can add support for additional protocols and
48    load balancing algorithms.</p>
49
50    <p>A set of modules must be loaded into the server to provide the
51    necessary features.  These modules can be included statically at
52    build time or dynamically via the
53    <code class="directive"><a href="../mod/mod_so.html#loadmodule">LoadModule</a></code> directive).
54    The set must include:</p>

155<h2><a name="forwardreverse" id="forwardreverse">Forward Proxies and Reverse
156       Proxies/Gateways</a></h2>
157      <p>Apache HTTP Server can be configured in both a <dfn>forward</dfn> and
158      <dfn>reverse</dfn> proxy (also known as <dfn>gateway</dfn>) mode.</p>
159
160      <p>An ordinary <dfn>forward proxy</dfn> is an intermediate
161      server that sits between the client and the <em>origin
162      server</em>.  In order to get content from the origin server,
163      the client sends a request to the proxy naming the origin server
164      as the target. The proxy then requests the content from the
165      origin server and returns it to the client.  The client must be
166      specially configured to use the forward proxy to access other
167      sites.</p>
168
169      <p>A typical usage of a forward proxy is to provide Internet
170      access to internal clients that are otherwise restricted by a
171      firewall.  The forward proxy can also use caching (as provided
172      by <code class="module"><a href="../mod/mod_cache.html">mod_cache</a></code>) to reduce network usage.</p>
173
174      <p>The forward proxy is activated using the <code class="directive"><a href="#proxyrequests">ProxyRequests</a></code> directive.  Because
175      forward proxies allow clients to access arbitrary sites through
176      your server and to hide their true origin, it is essential that
177      you <a href="#access">secure your server</a> so that only
178      authorized clients can access the proxy before activating a
179      forward proxy.</p>
180
181      <p>A <dfn>reverse proxy</dfn> (or <dfn>gateway</dfn>), by
182      contrast, appears to the client just like an ordinary web
183      server.  No special configuration on the client is necessary.
184      The client makes ordinary requests for content in the namespace
185      of the reverse proxy.  The reverse proxy then decides where to
186      send those requests and returns the content as if it were itself
187      the origin.</p>
188
189      <p>A typical usage of a reverse proxy is to provide Internet
190      users access to a server that is behind a firewall.  Reverse
191      proxies can also be used to balance load among several back-end
192      servers or to provide caching for a slower back-end server.
193      In addition, reverse proxies can be used simply to bring
194      several servers into the same URL space.</p>
195
196      <p>A reverse proxy is activated using the <code class="directive"><a href="#proxypass">ProxyPass</a></code> directive or the
197      <code>[P]</code> flag to the <code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">RewriteRule</a></code> directive.  It is
198      <strong>not</strong> necessary to turn <code class="directive"><a href="#proxyrequests">ProxyRequests</a></code> on in order to
199      configure a reverse proxy.</p>

244<h2><a name="workers" id="workers">Workers</a></h2>
245      <p>The proxy manages the configuration of origin servers and their
246      communication parameters in objects called <dfn>workers</dfn>.
247      There are two built-in workers: the default forward proxy worker and the
248      default reverse proxy worker. Additional workers can be configured
249      explicitly.</p>
250
251      <p>The two default workers have a fixed configuration
252      and will be used if no other worker matches the request.
253      They do not use HTTP Keep-Alive or connection reuse.
254      The TCP connections to the origin server will instead be
255      opened and closed for each request.</p>
256
257      <p>Explicitly configured workers are identified by their URL.
258      They are usually created and configured using
259      <code class="directive"><a href="#proxypass">ProxyPass</a></code> or
260      <code class="directive"><a href="#proxypassmatch">ProxyPassMatch</a></code> when used
261      for a reverse proxy:</p>
262
263      <pre class="prettyprint lang-config">ProxyPass "/example" "http://backend.example.com" connectiontimeout=5 timeout=30</pre>
264
265
266      <p>This will create a worker associated with the origin server URL
267      <code>http://backend.example.com</code> that will use the given timeout
268      values. When used in a forward proxy, workers are usually defined
269      via the <code class="directive"><a href="#proxyset">ProxySet</a></code> directive:</p>
270
271      <pre class="prettyprint lang-config">ProxySet "http://backend.example.com" connectiontimeout=5 timeout=30</pre>
272
273
274      <p>or alternatively using <code class="directive"><a href="#proxy">Proxy</a></code>
275      and <code class="directive"><a href="#proxyset">ProxySet</a></code>:</p>
276
277      <pre class="prettyprint lang-config">&lt;Proxy "http://backend.example.com"&gt;
278  ProxySet connectiontimeout=5 timeout=30
279&lt;/Proxy&gt;</pre>
280
281
282      <p>Using explicitly configured workers in the forward mode is
283      not very common, because forward proxies usually communicate with many
284      different origin servers. Creating explicit workers for some of the
285      origin servers can still be useful if they are used very often.
286      Explicitly configured workers have no concept of forward or reverse
287      proxying by themselves. They encapsulate a common concept of
288      communication with origin servers. A worker created by
289      <code class="directive"><a href="#proxypass">ProxyPass</a></code> for use in a
290      reverse proxy will also be used for forward proxy requests whenever
291      the URL to the origin server matches the worker URL, and vice versa.</p>
292
293      <p>The URL identifying a direct worker is the URL of its
294      origin server including any path components given:</p>
295
296     <pre class="prettyprint lang-config">ProxyPass "/examples" "http://backend.example.com/examples"
297ProxyPass "/docs" "http://backend.example.com/docs"</pre>
298
299
300      <p>This example defines two different workers, each using a separate
301      connection pool and configuration.</p>
302
303      <div class="warning"><h3>Worker Sharing</h3>
304        <p>Worker sharing happens if the worker URLs overlap, which occurs when
305        the URL of some worker is a leading substring of the URL of another
306        worker defined later in the configuration file. In the following example</p>
307
308        <pre class="prettyprint lang-config">ProxyPass "/apps" "http://backend.example.com/" timeout=60
309ProxyPass "/examples" "http://backend.example.com/examples" timeout=10</pre>
310
311
312        <p>the second worker isn't actually created. Instead the first
313        worker is used. The benefit is, that there is only one connection pool,
314        so connections are more often reused. Note that all configuration attributes
315        given explicitly for the later worker will be ignored. This will be logged
316        as a warning. In the above example, the resulting timeout value
317        for the URL <code>/examples</code> will be <code>60</code> instead
318        of <code>10</code>!</p>
319
320        <p>If you want to avoid worker sharing, sort your worker definitions
321        by URL length, starting with the longest worker URLs. If you want to maximize
322        worker sharing, use the reverse sort order. See also the related warning about
323        ordering <code class="directive"><a href="#proxypass">ProxyPass</a></code> directives.</p>
324
325      </div> 
326
327      <p>Explicitly configured workers come in two flavors:
328      <dfn>direct workers</dfn> and <dfn>(load) balancer workers</dfn>.
329      They support many important configuration attributes which are
330      described below in the <code class="directive"><a href="#proxypass">ProxyPass</a></code>
331      directive. The same attributes can also be set using
332      <code class="directive"><a href="#proxyset">ProxySet</a></code>.</p>
333
334      <p>The set of options available for a direct worker
335      depends on the protocol which is specified in the origin server URL.
336      Available protocols include <code>ajp</code>, <code>fcgi</code>,
337      <code>ftp</code>, <code>http</code> and <code>scgi</code>.</p>
338
339      <p>Balancer workers are virtual workers that use direct workers known
340      as their members to actually handle the requests. Each balancer can
341      have multiple members. When it handles a request, it chooses a member
342      based on the configured load balancing algorithm.</p>
343
344      <p>A balancer worker is created if its worker URL uses
345      <code>balancer</code> as the protocol scheme.
346      The balancer URL uniquely identifies the balancer worker.
347      Members are added to a balancer using
348      <code class="directive"><a href="#balancermember">BalancerMember</a></code>.</p>
349
350      <div class="note"><h3>DNS resolution for origin domains</h3>
351      <p>DNS resolution happens when the socket to
352        the origin domain is created for the first time.
353        When connection reuse is enabled, each backend domain is resolved 
354        only once per child process, and cached for all further connections 
355        until the child is recycled. This information should to be considered 
356        while planning DNS maintenance tasks involving backend domains. 
357        Please also check <code class="directive"><a href="#proxypass">ProxyPass</a></code>
358        parameters for more details about connection reuse.
359        </p>
360      </div>

364<h2><a name="access" id="access">Controlling Access to Your Proxy</a></h2>
365      <p>You can control who can access your proxy via the <code class="directive"><a href="#proxy">&lt;Proxy&gt;</a></code> control block as in
366      the following example:</p>
367
368      <pre class="prettyprint lang-config">&lt;Proxy "*"&gt;
369  Require ip 192.168.0
370&lt;/Proxy&gt;</pre>
371
372
373      <p>For more information on access control directives, see
374      <code class="module"><a href="../mod/mod_authz_host.html">mod_authz_host</a></code>.</p>
375

382      <code>ProxyRequests Off</code>), access control is less
383      critical because clients can only contact the hosts that you
384      have specifically configured.</p>

397<h2><a name="intranet" id="intranet">Intranet Proxy</a></h2>
398      <p>An Apache httpd proxy server situated in an intranet needs to forward
399      external requests through the company's firewall (for this, configure
400      the <code class="directive"><a href="#proxyremote">ProxyRemote</a></code> directive
401      to forward the respective <var>scheme</var> to the firewall proxy).
402      However, when it has to

404      accessing hosts. The <code class="directive"><a href="#noproxy">NoProxy</a></code>
405      directive is useful for specifying which hosts belong to the intranet and
406      should be accessed directly.</p>
407

409      WWW requests, thus requesting "http://somehost/" instead of
410      <code>http://somehost.example.com/</code>. Some commercial proxy servers
411      let them get away with this and simply serve the request, implying a
412      configured local domain. When the <code class="directive"><a href="#proxydomain">ProxyDomain</a></code> directive is used and the server is <a href="#proxyrequests">configured for proxy service</a>, Apache httpd can return
413      a redirect response and send the client to the correct, fully qualified,
414      server address. This is the preferred method since the user's bookmark
415      files will then contain fully qualified hosts.</p>

442<h2><a name="request-bodies" id="request-bodies">Request Bodies</a></h2>
443
444    <p>Some request methods such as POST include a request body.
445    The HTTP protocol requires that requests which include a body
446    either use chunked transfer encoding or send a
447    <code>Content-Length</code> request header.  When passing these
448    requests on to the origin server, <code class="module"><a href="../mod/mod_proxy_http.html">mod_proxy_http</a></code>
449    will always attempt to send the <code>Content-Length</code>.  But
450    if the body is large and the original request used chunked
451    encoding, then chunked encoding may also be used in the upstream
452    request.  You can control this selection using <a href="../env.html">environment variables</a>.  Setting
453    <code>proxy-sendcl</code> ensures maximum compatibility with
454    upstream servers by always sending the
455    <code>Content-Length</code>, while setting
456    <code>proxy-sendchunked</code> minimizes resource usage by using
457    chunked encoding.</p>
458
459    <p>Under some circumstances, the server must spool request bodies
460    to disk to satisfy the requested handling of request bodies.  For
461    example, this spooling will occur if the original body was sent with
462    chunked encoding (and is large), but the administrator has
463    asked for backend requests to be sent with Content-Length or as HTTP/1.0.
464    This spooling can also occur if the request body already has a
465    Content-Length header, but the server is configured to filter incoming
466    request bodies.</p>

470<h2><a name="x-headers" id="x-headers">Reverse Proxy Request Headers</a></h2>
471
472    <p>When acting in a reverse-proxy mode (using the <code class="directive"><a href="#proxypass">ProxyPass</a></code> directive, for example),
473    <code class="module"><a href="../mod/mod_proxy_http.html">mod_proxy_http</a></code> adds several request headers in
474    order to pass information to the origin server. These headers
475    are:</p>
476
477    <dl>
478      <dt><code>X-Forwarded-For</code></dt>
479      <dd>The IP address of the client.</dd>
480      <dt><code>X-Forwarded-Host</code></dt>
481      <dd>The original host requested by the client in the <code>Host</code>
482       HTTP request header.</dd>
483      <dt><code>X-Forwarded-Server</code></dt>
484      <dd>The hostname of the proxy server.</dd>
485    </dl>
486
487    <p>Be careful when using these headers on the origin server, since
488    they will contain more than one (comma-separated) value if the
489    original request already contained one of these headers. For
490    example, you can use <code>%{X-Forwarded-For}i</code> in the log
491    format string of the origin server to log the original clients IP
492    address, but you may get more than one address if the request
493    passes through several proxies.</p>
494
495    <p>See also the <code class="directive"><a href="#proxypreservehost">ProxyPreserveHost</a></code> and <code class="directive"><a href="#proxyvia">ProxyVia</a></code> directives, which control
496    other request headers.</p>
497
498    <p>Note:  If you need to specify custom request headers to be
499    added to the forwarded request, use the
500    <code class="directive"><a href="../mod/mod_headers.html#requestheader">RequestHeader</a></code>
501    directive.</p>

588<div class="directive-section"><h2><a name="NoProxy" id="NoProxy">NoProxy</a> <a name="noproxy" id="noproxy">Directive</a></h2>
589<table class="directive">
590<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Hosts, domains, or networks that will be connected to
591directly</td></tr>
592<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>NoProxy <var>host</var> [<var>host</var>] ...</code></td></tr>
593<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config, virtual host</td></tr>
594<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Extension</td></tr>
595<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_proxy</td></tr>
596</table>
597    <p>This directive is only useful for Apache httpd proxy servers within
598    intranets.  The <code class="directive">NoProxy</code> directive specifies a
599    list of subnets, IP addresses, hosts and/or domains, separated by
600    spaces. A request to a host which matches one or more of these is
601    always served directly, without forwarding to the configured
602    <code class="directive"><a href="#proxyremote">ProxyRemote</a></code> proxy server(s).</p>
603
604    <div class="example"><h3>Example</h3><pre class="prettyprint lang-config">ProxyRemote  "*"  "http://firewall.example.com:81"
605NoProxy         ".example.com" "192.168.112.0/21"</pre>
606</div>
607
608    <p>The <var>host</var> arguments to the <code class="directive">NoProxy</code>
609    directive are one of the following type list:</p>
610
611    <dl>
612    
613    <dt><var><a name="domain" id="domain">Domain</a></var></dt>
614    <dd>
615    <p>A <dfn>Domain</dfn> is a partially qualified DNS domain name, preceded
616    by a period. It represents a list of hosts which logically belong to the
617    same DNS domain or zone (<em>i.e.</em>, the suffixes of the hostnames are
618    all ending in <var>Domain</var>).</p>
619
620    <div class="example"><h3>Examples</h3><p><code>
621      .com .example.org.
622    </code></p></div>
623
624    <p>To distinguish <var>Domain</var>s from <var><a href="#hostname">Hostname</a></var>s (both syntactically and semantically; a DNS domain can
625    have a DNS A record, too!), <var>Domain</var>s are always written with a
626    leading period.</p>
627
628    <div class="note"><h3>Note</h3>
629      <p>Domain name comparisons are done without regard to the case, and
630      <var>Domain</var>s are always assumed to be anchored in the root of the
631      DNS tree; therefore, the two domains <code>.ExAmple.com</code> and
632      <code>.example.com.</code> (note the trailing period) are considered
633      equal. Since a domain comparison does not involve a DNS lookup, it is much
634      more efficient than subnet comparison.</p>
635    </div></dd>
636
637    
638    <dt><var><a name="subnet" id="subnet">SubNet</a></var></dt>
639    <dd>
640    <p>A <dfn>SubNet</dfn> is a partially qualified internet address in
641    numeric (dotted quad) form, optionally followed by a slash and the netmask,
642    specified as the number of significant bits in the <var>SubNet</var>. It is
643    used to represent a subnet of hosts which can be reached over a common
644    network interface. In the absence of the explicit net mask it is assumed
645    that omitted (or zero valued) trailing digits specify the mask. (In this
646    case, the netmask can only be multiples of 8 bits wide.) Examples:</p>
647
648    <dl>
649    <dt><code>192.168</code> or <code>192.168.0.0</code></dt>
650    <dd>the subnet 192.168.0.0 with an implied netmask of 16 valid bits
651    (sometimes used in the netmask form <code>255.255.0.0</code>)</dd>
652    <dt><code>192.168.112.0/21</code></dt>
653    <dd>the subnet <code>192.168.112.0/21</code> with a netmask of 21
654    valid bits (also used in the form <code>255.255.248.0</code>)</dd>
655    </dl>
656
657    <p>As a degenerate case, a <em>SubNet</em> with 32 valid bits is the
658    equivalent to an <var><a href="#ipaddr">IPAddr</a></var>, while a <var>SubNet</var> with zero
659    valid bits (<em>e.g.</em>, 0.0.0.0/0) is the same as the constant
660    <var>_Default_</var>, matching any IP address.</p></dd>
661
662    
663    <dt><var><a name="ipaddr" id="ipaddr">IPAddr</a></var></dt>
664    <dd>
665    <p>A <dfn>IPAddr</dfn> represents a fully qualified internet address in
666    numeric (dotted quad) form. Usually, this address represents a host, but
667    there need not necessarily be a DNS domain name connected with the
668    address.</p>
669    <div class="example"><h3>Example</h3><p><code>
670      192.168.123.7
671    </code></p></div>
672
673    <div class="note"><h3>Note</h3>
674      <p>An <var>IPAddr</var> does not need to be resolved by the DNS system, so
675      it can result in more effective apache performance.</p>
676    </div></dd>
677
678    
679    <dt><var><a name="hostname" id="hostname">Hostname</a></var></dt>
680    <dd>
681    <p>A <dfn>Hostname</dfn> is a fully qualified DNS domain name which can
682    be resolved to one or more <var><a href="#ipaddr">IPAddrs</a></var> via the
683    DNS domain name service. It represents a logical host (in contrast to
684    <var><a href="#domain">Domain</a></var>s, see above) and must be resolvable
685    to at least one <var><a href="#ipaddr">IPAddr</a></var> (or often to a list
686    of hosts with different <var><a href="#ipaddr">IPAddr</a></var>s).</p>
687
688    <div class="example"><h3>Examples</h3><p><code>
689      prep.ai.example.edu<br />
690      www.example.org
691    </code></p></div>
692
693    <div class="note"><h3>Note</h3>
694      <p>In many situations, it is more effective to specify an <var><a href="#ipaddr">IPAddr</a></var> in place of a <var>Hostname</var> since a
695      DNS lookup can be avoided. Name resolution in Apache httpd can take a remarkable
696      deal of time when the connection to the name server uses a slow PPP
697      link.</p>
698      <p><var>Hostname</var> comparisons are done without regard to the case,
699      and <var>Hostname</var>s are always assumed to be anchored in the root
700      of the DNS tree; therefore, the two hosts <code>WWW.ExAmple.com</code>
701      and <code>www.example.com.</code> (note the trailing period) are
702      considered equal.</p>
703     </div></dd>
704    </dl>

1016<div class="directive-section"><h2><a name="ProxyPass" id="ProxyPass">ProxyPass</a> <a name="proxypass" id="proxypass">Directive</a></h2>
1017<table class="directive">
1018<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Maps remote servers into the local server URL-space</td></tr>
1019<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>ProxyPass [<var>path</var>] !|<var>url</var> [<var>key=value</var>
1020  <var>[key=value</var> ...]] [nocanon] [interpolate] [noquery]</code></td></tr>
1021<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config, virtual host, directory</td></tr>
1022<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Extension</td></tr>
1023<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_proxy</td></tr>
1024<tr><th><a href="directive-dict.html#Compatibility">Compatibility:</a></th><td>Unix Domain Socket (UDS) support added in 2.4.7</td></tr>
1025</table>
1026    <p>This directive allows remote servers to be mapped into the
1027    space of the local server. The local server does not act as a
1028    proxy in the conventional sense but appears to be a mirror of the
1029    remote server. The local server is often called a <dfn>reverse
1030    proxy</dfn> or <dfn>gateway</dfn>. The <var>path</var> is the name of
1031    a local virtual path; <var>url</var> is a partial URL for the
1032    remote server and cannot include a query string.</p>
1033
1034    <div class="note">It is strongly suggested to review the concept of a
1035    <a href="#workers">Worker</a> before proceeding any further
1036    with this section.</div>
1037
1038    <div class="note">This directive is not supported within
1039    <code class="directive"><a href="../mod/core.html#directory">&lt;Directory&gt;</a></code>,
1040    <code class="directive"><a href="../mod/core.html#if">&lt;If&gt;</a></code> and
1041    <code class="directive"><a href="../mod/core.html#files">&lt;Files&gt;</a></code> containers.
1042    </div>
1043
1044    <div class="warning">The <code class="directive"><a href="#proxyrequests">ProxyRequests</a></code> directive should
1045    usually be set <strong>off</strong> when using
1046    <code class="directive">ProxyPass</code>.</div>
1047
1048    <p>In 2.4.7 and later, support for using a Unix Domain Socket is available by using a target
1049    which prepends <code>unix:/path/lis.sock|</code>. For example, to proxy
1050    HTTP and target the UDS at /home/www.socket, you would use
1051    <code>unix:/home/www.socket|http://localhost/whatever/</code>.</p>
1052
1053    <div class="note"><strong>Note:</strong> The path associated with the <code>unix:</code>
1054    URL is <code class="directive">DefaultRuntimeDir</code> aware.</div>
1055
1056    <p>When used inside a <code class="directive"><a href="../mod/core.html#location">&lt;Location&gt;</a></code> section, the first argument is omitted and the local
1057    directory is obtained from the <code class="directive"><a href="../mod/core.html#location">&lt;Location&gt;</a></code>. The same will occur inside a
1058    <code class="directive"><a href="../mod/core.html#locationmatch">&lt;LocationMatch&gt;</a></code> section;
1059    however, ProxyPass does not interpret the regexp as such, so it is necessary
1060    to use <code class="directive">ProxyPassMatch</code> in this situation instead.</p>
1061
1062    <p>Suppose the local server has address <code>http://example.com/</code>;
1063    then</p>
1064
1065    <pre class="prettyprint lang-config">&lt;Location "/mirror/foo/"&gt;
1066    ProxyPass "http://backend.example.com/"
1067&lt;/Location&gt;</pre>
1068
1069
1070    <p>will cause a local request for
1071    <code>http://example.com/mirror/foo/bar</code> to be internally converted
1072    into a proxy request to <code>http://backend.example.com/bar</code>.</p>
1073
1074    <p>If you require a more flexible reverse-proxy configuration, see the
1075    <code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">RewriteRule</a></code> directive with the
1076    <code>[P]</code> flag.</p>
1077
1078    <p>The following alternative syntax is possible; however, it can carry a
1079    performance penalty when present in very large numbers. The advantage of
1080    the below syntax is that it allows for dynamic control via the
1081    <a href="mod_proxy_balancer.html#balancer_manager">Balancer Manager</a> interface:</p>
1082
1083    <pre class="prettyprint lang-config">ProxyPass "/mirror/foo/" "http://backend.example.com/"</pre>
1084
1085
1086    <div class="warning">
1087    <p>If the first argument ends with a trailing <strong>/</strong>, the second
1088       argument should also end with a trailing <strong>/</strong>, and vice
1089       versa. Otherwise, the resulting requests to the backend may miss some
1090       needed slashes and do not deliver the expected results.
1091    </p>
1092    </div>
1093
1094    <p>The <code>!</code> directive is useful in situations where you don't want
1095    to reverse-proxy a subdirectory, <em>e.g.</em></p>
1096
1097    <pre class="prettyprint lang-config">&lt;Location "/mirror/foo/"&gt;
1098    ProxyPass "http://backend.example.com/"
1099&lt;/Location&gt;
1100&lt;Location "/mirror/foo/i"&gt;
1101    ProxyPass "!"
1102&lt;/Location&gt;</pre>
1103
1104
1105    <pre class="prettyprint lang-config">ProxyPass "/mirror/foo/i" "!"
1106ProxyPass "/mirror/foo" "http://backend.example.com"</pre>
1107
1108
1109    <p>will proxy all requests to <code>/mirror/foo</code> to
1110    <code>backend.example.com</code> <em>except</em> requests made to
1111    <code>/mirror/foo/i</code>.</p>
1112
1113    <p>Mixing ProxyPass settings in different contexts does not work:</p>
1114    <pre class="prettyprint lang-config">ProxyPass "/mirror/foo/i" "!"
1115&lt;Location "/mirror/foo/"&gt;
1116    ProxyPass "http://backend.example.com/"
1117&lt;/Location&gt;</pre>
1118
1119    <p>In this case, a request to <code>/mirror/foo/i</code> will get proxied,
1120       because the <code class="directive">ProxyPass</code> directive in the Location block will be evaluated
1121       first. The fact that <code class="directive">ProxyPass</code> supports both server and directory contexts
1122       does not mean that their scope and position in the configuration file will
1123       guarantee any ordering or override.</p>
1124
1125    <div class="warning"><h3>Ordering ProxyPass Directives</h3>
1126      <p>The configured <code class="directive"><a href="#proxypass">ProxyPass</a></code>
1127      and <code class="directive"><a href="#proxypassmatch">ProxyPassMatch</a></code>
1128      rules are checked in the order of configuration. The first rule that
1129      matches wins. So usually you should sort conflicting
1130      <code class="directive"><a href="#proxypass">ProxyPass</a></code> rules starting with the
1131      longest URLs first. Otherwise, later rules for longer URLS will be hidden
1132      by any earlier rule which uses a leading substring of the URL. Note that
1133      there is some relation with worker sharing.</p>
1134    </div>

1152    <p><strong>ProxyPass <code>key=value</code> Parameters</strong></p>
1153
1154    <p>In Apache HTTP Server 2.1 and later, mod_proxy supports pooled
1155    connections to a backend server.  Connections created on demand
1156    can be retained in a pool for future use.  Limits on the pool size
1157    and other settings can be coded on
1158    the <code class="directive">ProxyPass</code> directive
1159    using  <code>key=value</code> parameters, described in the tables
1160    below.</p>
1161
1162    <div class="warning"><h3>Maximum connections to the backend</h3>
1163    <p>By default, mod_proxy will allow and retain the maximum number of
1164    connections that could be used simultaneously by that web server child
1165    process.  Use the <code>max</code> parameter to reduce the number from
1166    the default. The pool of connections is maintained per web server child
1167    process, and <code>max</code> and other settings are not coordinated
1168    among all child processes, except when only one child process is allowed
1169    by configuration or MPM design.</p>
1170    </div>
1171
1172    <p>Use the <code>ttl</code> parameter to set an optional
1173    time to live; connections which have been unused for at least
1174    <code>ttl</code> seconds will be closed.  <code>ttl</code> can be used
1175    to avoid using a connection which is subject to closing because of the
1176    backend server's keep-alive timeout.</p>

1587    provided by the proxy.</p>

1589    <p>Normally, mod_proxy will include the query string when
1590    generating the <var>SCRIPT_FILENAME</var> environment variable.
1591    The optional <var>noquery</var> keyword (available in
1592    httpd 2.4.1 and later) prevents this.</p>

1594    <p>The optional <code>interpolate</code> keyword, in combination with
1595    <code class="directive"><a href="#proxypassinterpolateenv">ProxyPassInterpolateEnv</a></code>, causes the ProxyPass
1596    to interpolate environment variables, using the syntax
1597    <var>${VARNAME}</var>.  Note that many of the standard CGI-derived
1598    environment variables will not exist when this interpolation happens,
1599    so you may still have to resort to <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code>
1600    for complex rules. Also note that interpolation is supported
1601    within the scheme/hostname/port portion of a URL only for variables that
1602    are available when the directive is parsed 
1603    (like <code class="directive"><a href="../mod/core.html#define">Define</a></code>). Dynamic determination of
1604    those fields can be accomplished with <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code>.
1605    The following example describes how to use <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code>
1606    to dynamically set the scheme to http or https:</p>
1607
1608    <pre class="prettyprint lang-config">RewriteEngine On
1609
1610RewriteCond "%{HTTPS}" =off
1611RewriteRule "." "-" [E=protocol:http]
1612RewriteCond "%{HTTPS}" =on
1613RewriteRule "." "-" [E=protocol:https]
1614
1615RewriteRule "^/mirror/foo/(.*)" "%{ENV:protocol}://backend.example.com/$1" [P]
1616ProxyPassReverse  "/mirror/foo/" "http://backend.example.com/"
1617ProxyPassReverse  "/mirror/foo/" "https://backend.example.com/"</pre>

1620    <div class="note"><h3><a id="protoupgrade" name="protoupgrade">Protocol Upgrade</a></h3>
1621      <p>Since Apache HTTP Server 2.4.47, protocol Upgrade (tunneling) can be handled
1622      end-to-end by <code class="module"><a href="../mod/mod_proxy_http.html">mod_proxy_http</a></code> using the <code class="directive">ProxyPass</code>
1623      parameter <var><a href="#upgrade">upgrade</a></var>.</p>

1625      forwarded by <code class="module"><a href="../mod/mod_proxy_http.html">mod_proxy_http</a></code> to the origin server and the connection
1626      will be upgraded (and tunneled by <code class="module"><a href="../mod/mod_proxy_http.html">mod_proxy_http</a></code>) only if the origin
1627      server accepts/initiates the upgrade (HTTP response <code>101 Switching Protocols</code>).
1628      If the origin server responds with anything else <code class="module"><a href="../mod/mod_proxy_http.html">mod_proxy_http</a></code>
1629      will continue forwarding (and enforcing) the HTTP protocol as usual for this
1630      connection.</p>
1631      <p>See <a href="#wsupgrade">Websocket Upgrade (2.4.47 and later)</a> for an example of
1632      configuration using <code class="module"><a href="../mod/mod_proxy_http.html">mod_proxy_http</a></code>.</p>
1633      <p>For Apache HTTP Server 2.4.46 and earlier (or if
1634      <code class="directive"><a href="../mod/mod_proxy_wstunnel.html#proxywebsocketfallbacktoproxyhttp">ProxyWebsocketFallbackToProxyHttp</a></code>
1635      from 2.4.48 and later disables <code class="module"><a href="../mod/mod_proxy_http.html">mod_proxy_http</a></code> handling), see the
1636      documentation of <code class="module"><a href="../mod/mod_proxy_wstunnel.html">mod_proxy_wstunnel</a></code> for how to proxy the WebSocket

1702<table class="directive">
1703<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Maps remote servers into the local server URL-space using regular expressions</td></tr>
1704<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>ProxyPassMatch [<var>regex</var>] !|<var>url</var> [<var>key=value</var>
1705	<var>[key=value</var> ...]]</code></td></tr>
1706<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config, virtual host, directory</td></tr>
1707<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Extension</td></tr>

1709<tr><th><a href="directive-dict.html#Compatibility">Compatibility:</a></th><td>Since 2.4.47 the <var>key=value</var> Parameters are honored

1713    <p>This directive is equivalent to <code class="directive"><a href="#proxypass">ProxyPass</a></code>
1714       but makes use of regular expressions instead of simple prefix matching. The
1715       supplied regular expression is matched against the <var>url</var>, and if it
1716       matches, the server will substitute any parenthesized matches into the given
1717       string and use it as a new <var>url</var>.</p>
1718
1719    <div class="note"><strong>Note: </strong>This directive cannot be used within a
1720    <code>&lt;Directory&gt;</code> context.</div>
1721    
1722    <p>Suppose the local server has address <code>http://example.com/</code>;
1723    then</p>
1724

1726
1727
1728    <p>will cause a local request for
1729    <code>http://example.com/foo/bar.gif</code> to be internally converted
1730    into a proxy request to <code>http://backend.example.com/foo/bar.gif</code>.</p>
1731
1732    <p>The <code>!</code> directive is useful in situations where you don't want
1733    to reverse-proxy a subdirectory.</p>
1734
1735    <p>When used inside a <code class="directive"><a href="../mod/core.html#locationmatch">&lt;LocationMatch&gt;</a></code> section, the first argument is omitted and the
1736    regexp is obtained from the <code class="directive"><a href="../mod/core.html#locationmatch">&lt;LocationMatch&gt;</a></code>.</p>
1737
1738    <p>If you require a more flexible reverse-proxy configuration, see the
1739    <code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">RewriteRule</a></code> directive with the
1740    <code>[P]</code> flag.</p>

1742    <div class="note">
1743      <h3>Default Substitution</h3>
1744      <p>When the URL parameter doesn't use any backreferences into the regular
1745      expression, the original URL will be appended to the URL parameter.
1746      </p>
1747    </div>
1748    <div class="note">
1749        <h3><code>key=value</code> Parameters versus <var>url</var> with backreference(s)</h3>
1750      <p>Since Apache HTTP Server 2.4.47, the <code>key=value</code> Parameters
1751      are no longer ignored in a <code class="directive">ProxyPassMatch</code> using
1752      an <var>url</var> with backreference(s). However to keep the existing
1753      behavior regarding reuse/keepalive of backend connections (which were
1754      never reused before for these URLs), the parameter <var>enablereuse</var>
1755      (or <var>disablereuse</var>) default to <code>off</code> (resp. <code>on</code>)
1756      in this case. Setting <code>enablereuse=on</code> explicitely allows to
1757      reuse connections <strong>unless</strong> some backreference(s) belong in
1758      the <code>authority</code> part (hostname and/or port) of the <var>url</var>
1759      (this condition is enforced since Apache HTTP Server 2.4.55, and produces
1760      a warning at startup because these URLs are not reusable per se).</p>

1774<div class="directive-section"><h2><a name="ProxyPassReverse" id="ProxyPassReverse">ProxyPassReverse</a> <a name="proxypassreverse" id="proxypassreverse">Directive</a></h2>
1775<table class="directive">
1776<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Adjusts the URL in HTTP response headers sent from a reverse
1777proxied server</td></tr>
1778<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>ProxyPassReverse [<var>path</var>] <var>url</var>
1779[interpolate]</code></td></tr>
1780<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config, virtual host, directory</td></tr>
1781<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Extension</td></tr>
1782<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_proxy</td></tr>
1783</table>
1784    <p>This directive lets Apache httpd adjust the URL in the <code>Location</code>,
1785    <code>Content-Location</code> and <code>URI</code> headers on HTTP
1786    redirect responses. This is essential when Apache httpd is used as a
1787    reverse proxy (or gateway) to avoid bypassing the reverse proxy
1788    because of HTTP redirects on the backend servers which stay behind
1789    the reverse proxy.</p>
1790
1791    <p>Only the HTTP response headers specifically mentioned above
1792    will be rewritten. Apache httpd will not rewrite other response
1793    headers, nor will it by default rewrite URL references inside HTML pages.
1794    This means that if the proxied content contains absolute URL
1795    references, they will bypass the proxy. To rewrite HTML content to
1796    match the proxy, you must load and enable <code class="module"><a href="../mod/mod_proxy_html.html">mod_proxy_html</a></code>.
1797    </p>
1798
1799    <p><var>path</var> is the name of a local virtual path; <var>url</var> is a
1800    partial URL for the remote server. 
1801    These parameters are used the same way as for the
1802    <code class="directive"><a href="#proxypass">ProxyPass</a></code> directive.</p>
1803
1804    <p>For example, suppose the local server has address
1805    <code>http://example.com/</code>; then</p>
1806
1807    <pre class="prettyprint lang-config">ProxyPass         "/mirror/foo/" "http://backend.example.com/"
1808ProxyPassReverse  "/mirror/foo/" "http://backend.example.com/"
1809ProxyPassReverseCookieDomain  "backend.example.com"  "public.example.com"
1810ProxyPassReverseCookiePath  "/"  "/mirror/foo/"</pre>
1811
1812
1813    <p>will not only cause a local request for the
1814    <code>http://example.com/mirror/foo/bar</code> to be internally converted
1815    into a proxy request to <code>http://backend.example.com/bar</code>
1816    (the functionality which <code>ProxyPass</code> provides here).
1817    It also takes care of redirects which the server <code>backend.example.com</code>
1818    sends when redirecting <code>http://backend.example.com/bar</code> to
1819    <code>http://backend.example.com/quux</code> . Apache httpd adjusts this to
1820    <code>http://example.com/mirror/foo/quux</code> before forwarding the HTTP
1821    redirect response to the client. Note that the hostname used for
1822    constructing the URL is chosen in respect to the setting of the <code class="directive"><a href="../mod/core.html#usecanonicalname">UseCanonicalName</a></code> directive.</p>
1823
1824    <p>Note that this <code class="directive">ProxyPassReverse</code> directive can
1825    also be used in conjunction with the proxy feature
1826    (<code>RewriteRule ...  [P]</code>) from <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code>
1827    because it doesn't depend on a corresponding <code class="directive"><a href="#proxypass">ProxyPass</a></code> directive.</p>
1828
1829    <p>The optional <code>interpolate</code> keyword, used together with
1830    <code class="directive"><a href="#proxypassinterpolateenv">ProxyPassInterpolateEnv</a></code>, enables interpolation
1831    of environment variables specified using the format <var>${VARNAME}</var>.
1832    Note that interpolation is not supported within the scheme portion of a
1833    URL.</p>
1834
1835    <p>When used inside a <code class="directive"><a href="../mod/core.html#location">&lt;Location&gt;</a></code> section, the first argument is omitted and the local
1836    directory is obtained from the <code class="directive"><a href="../mod/core.html#location">&lt;Location&gt;</a></code>. The same occurs inside a <code class="directive"><a href="../mod/core.html#locationmatch">&lt;LocationMatch&gt;</a></code> section, but will probably not work as
1837    intended, as ProxyPassReverse will interpret the regexp literally as a
1838    path; if needed in this situation, specify the ProxyPassReverse outside
1839    the section or in a separate <code class="directive"><a href="../mod/core.html#location">&lt;Location&gt;</a></code> section.</p>
1840
1841    <p>This directive is not supported in <code class="directive"><a href="../mod/core.html#directory">&lt;Directory&gt;</a></code> or <code class="directive"><a href="../mod/core.html#files">&lt;Files&gt;</a></code> sections.</p>

1937<div class="directive-section"><h2><a name="ProxyRemote" id="ProxyRemote">ProxyRemote</a> <a name="proxyremote" id="proxyremote">Directive</a></h2>
1938<table class="directive">
1939<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Remote proxy used to handle certain requests</td></tr>
1940<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>ProxyRemote <var>match</var> <var>remote-server</var> [<var>username:password</var>]</code></td></tr>
1941<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config, virtual host</td></tr>
1942<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Extension</td></tr>
1943<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_proxy</td></tr>
1944<tr><th><a href="directive-dict.html#Compatibility">Compatibility:</a></th><td>The optional third argument is usable only in httpd 2.4.59 and later.</td></tr>
1945</table>
1946    <p>This defines remote proxies to this proxy. <var>match</var> is either the
1947    name of a URL-scheme that the remote server supports, or a partial URL
1948    for which the remote server should be used, or <code>*</code> to indicate
1949    the server should be contacted for all requests. <var>remote-server</var> is
1950    a partial URL for the remote server. Syntax:</p>
1951
1952    <div class="example"><p><code>
1953      <dfn>remote-server</dfn> =
1954          <var>scheme</var>://<var>hostname</var>[:<var>port</var>]
1955    </code></p></div>
1956
1957    <p><var>scheme</var> is effectively the protocol that should be used to
1958    communicate with the remote server; only <code>http</code> and <code>https</code>
1959    are supported by this module. When using <code>https</code>, the requests
1960    are forwarded through the remote proxy using the HTTP CONNECT method.</p>
1961
1962    <div class="example"><h3>Example</h3><pre class="prettyprint lang-config">ProxyRemote "http://goodguys.example.com/" "http://mirrorguys.example.com:8000"
1963ProxyRemote "*" "http://cleverproxy.localdomain"
1964ProxyRemote "ftp" "http://ftpproxy.mydomain:8080"</pre>
1965</div>
1966
1967    <p>In the last example, the proxy will forward FTP requests, encapsulated
1968    as yet another HTTP proxy request, to another proxy which can handle
1969    them.</p>
1970
1971    <p>This option also supports reverse proxy configuration; a backend
1972    webserver can be embedded within a virtualhost URL space even if that
1973    server is hidden by another forward proxy.</p>
1974
1975    <p>An optional third argument <var>username:password</var> may be
1976    given, which defines the Basic authentication credentials to pass
1977    to the configured remote proxy.  The credentials will always be
1978    sent without first waiting for the remote proxy to send a Basic
1979    authentication challenge.  The <a href="mod_proxy_http.html#env">Proxy-Chain-Auth</a> environment
1980    variable has no effect if this argument is used.</p>
1981

2003<table class="directive">
2004<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Enables forward (standard) proxy requests</td></tr>
2005<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>ProxyRequests On|Off</code></td></tr>

2024    need <code class="module"><a href="../mod/mod_proxy_connect.html">mod_proxy_connect</a></code> enabled in the server.</p>

2038<table class="directive">
2039<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Set various Proxy balancer or member parameters</td></tr>
2040<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>ProxySet <var>url</var> <var>key=value [key=value ...]</var></code></td></tr>
2041<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config, virtual host, directory</td></tr>

2047    <p>This directive is used as an alternate method of setting any of the
2048    parameters available to Proxy balancers and workers normally done via the

2058    BalancerMember "http://www2.example.com:8080" loadfactor=1

2060    ProxySet lbmethod=bytraffic
2061&lt;/Proxy&gt;</pre>
2062</div>
2063
2064    <pre class="prettyprint lang-config">&lt;Proxy "http://backend"&gt;
2065    ProxySet keepalive=On
2066&lt;/Proxy&gt;</pre>
2067
2068
2069    <pre class="prettyprint lang-config">ProxySet "balancer://foo" lbmethod=bytraffic timeout=15</pre>
2070
2071
2072    <pre class="prettyprint lang-config">ProxySet "ajp://backend:7001" timeout=15</pre>
2073
2074
2075   <div class="warning"><h3>Warning</h3>
2076      <p>Keep in mind that the same parameter key can have a different meaning
2077      depending whether it is applied to a balancer or a worker, as shown by
2078      the two examples above regarding timeout.</p>

2134<div class="directive-section"><h2><a name="ProxyVia" id="ProxyVia">ProxyVia</a> <a name="proxyvia" id="proxyvia">Directive</a></h2>
2135<table class="directive">
2136<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Information provided in the <code>Via</code> HTTP response
2137header for proxied requests</td></tr>
2138<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>ProxyVia On|Off|Full|Block</code></td></tr>
2139<tr><th><a href="directive-dict.html#Default">Default:</a></th><td><code>ProxyVia Off</code></td></tr>
2140<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config, virtual host</td></tr>
2141<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Extension</td></tr>
2142<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_proxy</td></tr>
2143</table>
2144    <p>This directive controls the use of the <code>Via:</code> HTTP
2145    header by the proxy. Its intended use is to control the flow of
2146    proxy requests along a chain of proxy servers.  See <a href="http://www.ietf.org/rfc/rfc2616.txt">RFC 2616</a> (HTTP/1.1), section
2147    14.45 for an explanation of <code>Via:</code> header lines.</p>
2148
2149    <ul>
2150    <li>If set to <code>Off</code>, which is the default, no special processing
2151    is performed. If a request or reply contains a <code>Via:</code> header,
2152    it is passed through unchanged.</li>
2153
2154    <li>If set to <code>On</code>, each request and reply will get a
2155    <code>Via:</code> header line added for the current host.</li>
2156
2157    <li>If set to <code>Full</code>, each generated <code>Via:</code> header
2158    line will additionally have the Apache httpd server version shown as a
2159    <code>Via:</code> comment field.</li>
2160
2161    <li>If set to <code>Block</code>, every proxy request will have all its
2162    <code>Via:</code> header lines removed. No new <code>Via:</code> header will
2163    be generated.</li>
2164    </ul>
