39<h3>Summary</h3>
40
41        <p>
42        This module manages common properties of domains for one or more virtual hosts. 
43        Its serves two main purposes: for one, supervise/renew TLS certificates via the 
44        ACME protocol (<a href="https://tools.ietf.org/html/rfc8555">RFC 8555</a>). 
45        Certificates will be renewed by the module ahead of their expiration to account 
46        for disruption in internet services. There are ways to monitor the status of all 
47        certificates managed this way and configurations that will run your own 
48        notification commands on renewal, expiration and errors.
49        </p><p>
50        Second, mod_md offers an alternate OCSP Stapling implementation. This works with
51        managed certificates as well as with certificates you configure yourself. OCSP
52        Stapling is a necessary component for any https: site, influencing page load
53        times and, depending on other setups, page availability. More in the
54        stapling section below.
55        </p><p>
56        The default ACME Authority for managing certificates is 
57        <a href="https://letsencrypt.org/">Let's Encrypt</a>, but it is possible 
58        to configure another CA that supports the protocol.
59        </p>
60        
61        <p>Simple configuration example:</p>
62        
63        <div class="note"><h3>TLS in a VirtualHost context</h3>
64        <pre class="prettyprint lang-config">MDomain example.org
65
66&lt;VirtualHost *:443&gt;
67    ServerName example.org
68    DocumentRoot htdocs/a
69
70    SSLEngine on
71    # no certificates specification
72&lt;/VirtualHost&gt;</pre>
73
74        <p>
75            This setup will, on server start, contact
76            <a href="https://letsencrypt.org/">Let's Encrypt</a>
77            to request a certificate for the domain. If Let's Encrypt can verify the ownership
78            of the domain, the module will retrieve the certificate and its chain, store it
79            in the local file system (see <code class="directive"><a href="#mdstoredir">MDStoreDir</a></code>)
80            and provide it, on next restart, to <code class="module"><a href="../mod/mod_ssl.html">mod_ssl</a></code>.
81        </p><p>
82            This happens while the server is already running. All other hosts will continue
83            to work as before. While a certificate is not available, requests for the managed
84            domain will be answered with a '503 Service Unavailable'. 
85        </p>
86        </div>

88        <div class="note"><h3>Prerequisites</h3>
89        <p>
90            This module requires <code class="module"><a href="../mod/mod_watchdog.html">mod_watchdog</a></code> to be loaded as well.
91        </p><p>
92            Certificate sign-up and renewal with Let's Encrypt requires your server to be
93            reachable on port 80 (http:) and/or port 443 (https:) from the public internet.
94            (Unless your server is configured to use DNS for challenges - more on that under
95             'wildcard certificates')  
96        </p><p>
97            The module will select from the methods offered by Let's Encrypt. Usually LE offers
98            challenges on both ports and DNS and Apache chooses a method available.
99        </p><p>
100            To determine which one is available, the module looks at the ports
101            Apache httpd listens on. If those include port 80, it assumes that the
102            http: challenge (named http-01) is available. If the server listens
103            on port 443, the https: challenge (named tls-alpn-01) is also added to
104            the list. (And if <code class="directive"><a href="#mdchallengedns01">MDChallengeDns01</a></code> 
105            is configured, the challenge dns-01 is added as well.) 
106        </p><p>
107            If your setup is not so straight forward, there are two methods available
108            to influence this. First, look at <code class="directive"><a href="#mdportmap">MDPortMap</a></code>
109            if the server is behind a portmapper, such as a firewall. Second, you may
110            override the module's guesswork completely by configuring
111            <code class="directive"><a href="#mdcachallenges">MDCAChallenges</a></code> directly.  
112        </p>
113        </div>
114
115        <div class="note"><h3>https: Challenges</h3>
116        <p>
117            For domain verification via the TLS protocol `tls-alpn-01` is the name
118            of the challenge type. It requires the Apache server to listen on port 443
119            (see <code class="directive"><a href="#mdportmap">MDPortMap</a></code> if you map that port
120            to something else).
121        </p><p>
122            Let's Encrypt will open a TLS connection to Apache using the special indicator
123            `acme-tls/1` (this indication part of TLS is called ALPN, therefore the name
124            of the challenge. ALPN is also used by browsers to request a HTTP/2 connection).
125        </p><p>
126            As with the HTTP/2 protocol, to allow this, you configure:
127        </p>
128        <pre class="prettyprint lang-config">Protocols h2 http/1.1 acme-tls/1</pre>
129
130        <p>
131            And the `tls-alpn-01` challenge type is available.
132	</p>
133        </div>

135        <div class="note"><h3>Wildcard Certificates</h3>
136        <p>
137            Wildcard certificates are possible, but not straight-forward to use out of
138            the box. Let's Encrypt requires the `dns-01` challenge verification 
139            for those. No other is considered good enough.
140        </p><p>
141            The difficulty here is that Apache cannot do that on its own. As the name implies, `dns-01` 
142            requires you to show some specific DNS records for your domain that contain 
143            some challenge data. So you need to _write_ your domain's DNS records.
144        </p><p>
145            If you know how to do that, you can integrated this with mod_md. Let's 
146            say you have a script for that in `/usr/bin/acme-setup-dns` you configure 
147            Apache with:
148        </p>
149        <pre class="prettyprint lang-config">MDChallengeDns01 /usr/bin/acme-setup-dns</pre>
150
151        <p>
152            and Apache will call this script when it needs to setup/teardown a DNS challenge 
153            record for a domain. 
154        </p><p>
155            Assuming you want a certificate for `*.mydomain.com`, mod_md will call:
156        </p>
157        <pre class="prettyprint lang-config">/usr/bin/acme-setup-dns setup mydomain.com challenge-data
158# this needs to remove all existing DNS TXT records for 
159# _acme-challenge.mydomain.com and create a new one with 
160# content "challenge-data"</pre>
161
162        <p>
163            and afterwards it will call
164        </p>
165        <pre class="prettyprint lang-config">/usr/bin/acme-setup-dns teardown mydomain.com
166# this needs to remove all existing DNS TXT records for 
167# _acme-challenge.mydomain.com</pre>
168
169        </div>

171        <div class="note"><h3>Monitoring</h3>
172            <p>
173                Apache has a standard module for monitoring: <code class="module"><a href="../mod/mod_status.html">mod_status</a></code>.
174                mod_md contributes a section and makes monitoring your 
175                domains easy.
176            </p><p>
177                You see all your MDs listed alphabetically, the domain names they contain, 
178                an overall status, expiration times and specific settings. The settings 
179                show your selection of renewal times (or the default), the CA that is used, 
180                etc.
181            </p><p>
182                The 'Renewal' column will show activity and error descriptions for certificate 
183                renewals. This should make life easier for people to find out if everything 
184                is all right or what went wrong.
185            </p><p>
186                If there is an error with an MD it will be shown here as well. This let's 
187                you assess problems without digging through your server logs.
188            </p><p>
189                There is also a new 'md-status' handler available to give you the MD information 
190                from 'server-status' in JSON format. You configure it as
191            </p>
192            <pre class="prettyprint lang-config">&lt;Location "/md-status"&gt;
193  SetHandler md-status
194&lt;/Location&gt;</pre>
195
196            <p>
197                on your server. As with 'server-status' you will want to add 
198                authorization for this. 
199            </p><p>
200                If you just want to check the JSON status of a specific domain, simply append 
201                that to your status url:
202            </p>
203            <pre class="prettyprint lang-config">&gt; curl https://&lt;yourhost&gt;/md-status/another-domain.org
204{
205  "name": "another-domain.org",
206  "domains": [
207    "another-domain.org",
208    "www.another-domain.org"
209  ],
210  ...</pre>
211
212            <p>
213                This JSON status also shows a log of activities when domains are renewed:
214            </p>
215            <pre class="prettyprint lang-config">{
216"when": "Wed, 19 Jun 2019 14:45:58 GMT",
217"type": "progress", "detail": "The certificate for the managed domain has been renewed successfully and can be used. A graceful server restart now is recommended."
218},{
219"when": "Wed, 19 Jun 2019 14:45:58 GMT",
220"type": "progress", "detail": "Retrieving certificate chain for test-901-003-1560955549.org"
221},{
222"when": "Wed, 19 Jun 2019 14:45:58 GMT",
223"type": "progress", "detail": "Waiting for finalized order to become valid"
224},{
225"when": "Wed, 19 Jun 2019 14:45:50 GMT",
226"type": "progress", "detail": "Submitting CSR to CA for test-901-003-1560955549.org"
227},
228...</pre>
229
230            <p>
231                You will also find this information in the file `job.json` in your staging and, 
232                when activated, domains directory. This allows you to inspect these at
233                any later point in time as well. 
234            </p><p>
235                In addition, there is <code class="directive"><a href="#mdcertificatestatus">MDCertificateStatus</a></code> which
236                gives access to relevant certificate information in JSON format.

240        <div class="note"><h3>Stapling</h3>
241            <p>
242                If you want to try the stapling in one Managed Domain alone at first,
243                configure:
244            </p>
245            <pre class="prettyprint lang-config">&lt;MDomain mydomain.net&gt;
246  MDStapling on
247&lt;/MDomain&gt;</pre>
248
249            <p>
250                and use the 'server-status' and/or <code class="directive"><a href="#mdmessagecmd">MDMessageCmd</a></code> to see how it operates. You will
251                see if Stapling information is there, how long it is valid, from where it came and
252                when it will be refreshed.
253            </p><p>
254                If this all works to your satisfaction, you can switch it on for all your
255                certificates or just your managed ones.
256            </p><p>
257                The existing stapling implementation by mod_ssl is used by many sites
258                for years. There are two main differences between the mod_ssl and mod_md
259                one:
260            </p>
261            <ol>
262                <li>On demand vs. scheduled: mod_ssl retrieves the stapling information
263                when it is requested, e.g. on a new connection. mod_md retrieves it
264                right at server start and after 2/3rds of its lifetime.</li>
265                <li>In memory vs. persisted: mod_ssl <em>can</em> persist this
266                information, but most example configurations use a memory cache. mod_md
267                always stores in the file system.</li>
268            </ol>
269            <p>
270                If you are unlucky and restart your server during an outage of your CA's
271                OCSP service, your users may no longer reach your sites. Without persistence
272                your server cannot provide the client with the data and the client browser
273                cannot get it as well, since the OCSP service is not responding. 
274            </p><p>
275                The implementation in mod_md will have persisted it, load it again after
276                restart and have it available for incoming connections. A day or two before
277                this information expires, it will renew it, making it able to cope with
278                a long OCSP service downtime.
279            </p><p>
280                Due to backward compatibility, the existing implementation in mod_ssl could
281                not be changed drastically. For example, mod_ssl is unable to add a dependency 
282                to mod_watchdog without braking many existing installations (that do not load it).
283            </p>

874<table class="directive">
875<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Handle events for Manage Domains</td></tr>
876<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>MDMessageCmd <var>path-to-cmd</var> <var>optional-args</var></code></td></tr>
877<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config</td></tr>
878<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Experimental</td></tr>
879<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_md</td></tr>
880</table>
881            <p>
882                This command gets called when one of the following events happen for
883                a Managed Domain: "renewed", "installed", "expiring", "errored". The command may
884                be invoked for more than these in the future and ignore events
885                it is not prepared to handle.
886            </p><p>
887                This is the more flexible companion to <code class="directive"><a href="#mdnotifycmd">MDNotifyCmd</a></code>.
888            </p>
889            <div class="example"><h3>Example</h3><pre class="prettyprint lang-config">MDMessageCmd /etc/apache/md-message</pre>
890<p><code>
891
892# will be invoked when a new certificate for mydomain.org is available as:
893/etc/apache/md-message renewed mydomain.com
894            </code></p></div>
895            <p>
896                The program should not block, as the module will wait for it to finish. A
897                return code other than 0 is regarded as an error. 
898            </p><p>
899                'errored' is no immediate cause for concern since renewal is attempted
900                early enough to allow the internet to come back. This is reported at most
901                once per hour. 
902            </p><p>
903                'expiring' should be taken serious. It is issued when the
904                <code class="directive"><a href="#mdwarnwindow">MDWarnWindow</a></code> is reached. By default this is
905                10% of the certificate lifetime, so for Let's Encrypt this currently
906                means 9 days before it expires. The warning is repeated at most once
907                a day. 
908            </p><p>
909                'renewed' means that a new certificate has been obtained and is stored
910                in the 'staging' area in the MD store. It will be activated on the next
911                server restart/reload.
912            </p><p>
913                'installed' is triggered when a new certificate has been transferred from
914                staging into the domains location in MD store. This happens at server
915                startup/reload. Different to all other invocations, <code class="directive">MDMessageCmd</code> is run
916                with root permissions (on *nix systems) and has access to the certificate
917                files (and keys). Certificates needed for other applications or
918                in different formats can be processed on this event.
919            </p><p>
920                'renewing' event is triggered before starting renew process for the managed
921                 domain. Should the command return != 0 for this reason, renew will be
922                 aborted and repeated on next cycle. Some cluster setups use this to
923                 allow renewals to run only on a single node.
924            </p><p>
925                'challenge-setup:type:domain' event is triggered when the challenge data for a domain has
926                been created. This is invoked before the ACME server is told to check for it.
927                The type is one of the ACME challenge types. This is invoked for every
928                DNS name in a MDomain. Cluster setups may use this event to distribute
929                challenge files to all nodes in a cluster.
930            </p><p>
931                ocsp-errored happens when <code class="directive"><a href="#mdstapling">MDStapling</a></code>
932                is enabled for a domain, this indicates
933                that an error was encountered retrieving the OCSP response from the
934                Certificate Authority. mod_md will continue trying.
935            </p>

1088<div class="directive-section"><h2><a name="MDPortMap" id="MDPortMap">MDPortMap</a> <a name="mdportmap" id="mdportmap">Directive</a></h2>
1089<table class="directive">
1090<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Map external to internal ports for domain ownership verification.</td></tr>
1091<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>MDPortMap <var>map1</var> [ <var>map2</var> ]</code></td></tr>
1092<tr><th><a href="directive-dict.html#Default">Default:</a></th><td><code>MDPortMap http:80 https:443</code></td></tr>
1093<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config</td></tr>
1094<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Experimental</td></tr>
1095<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_md</td></tr>
1096</table>
1097            <p>
1098                The ACME protocol provides two methods to verify domain ownership via
1099                HTTP: one that uses 'http:' urls (port 80) and one for 'https:' urls
1100                (port 443). If your server is not reachable by at least one
1101                of the two, ACME may only work by configuring your DNS server,
1102                see <code class="directive"><a href="#mdchallengedns01">MDChallengeDns01</a></code>.
1103            </p><p>
1104                On most public facing servers, 'http:' arrives on port 80 and
1105                'https:' on port 443. The module checks the ports your Apache server
1106                is listening on and assumes those are available. This means that
1107                when your server does not listen on port 80, it assumes that
1108                'http:' requests from the internet will not work.
1109            </p><p>
1110                This is a good guess, but it may be wrong. For example, your Apache
1111                might listen to port 80, but your firewall might block it. 'http:' 
1112                is only available in your intranet. So, the module will falsely assume
1113                that Let's Encrypt can use 'http:' challenges with your server. This 
1114                will then fail, because your firewall will drop those.
1115            </p>

1476            </p><p>
1477                The stapling of <code class="module"><a href="../mod/mod_md.html">mod_md</a></code> will also work for domains where the certificates 
1478                are not managed by this module (see <code class="directive"><a href="#mdstapleothers">MDStapleOthers</a></code> for how to control this). 
1479                This allows use of the new stapling without using any ACME certificate 
1480                management.
1481            </p>

1505<div class="directive-section"><h2><a name="MDStaplingRenewWindow" id="MDStaplingRenewWindow">MDStaplingRenewWindow</a> <a name="mdstaplingrenewwindow" id="mdstaplingrenewwindow">Directive</a></h2>
1506<table class="directive">
1507<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Control when the stapling responses will be renewed.</td></tr>
1508<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>MDStaplingRenewWindow <var>duration</var></code></td></tr>
1509<tr><th><a href="directive-dict.html#Default">Default:</a></th><td><code>MDStaplingRenewWindow 33%</code></td></tr>
1510<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config</td></tr>
1511<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Experimental</td></tr>
1512<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_md</td></tr>
1513<tr><th><a href="directive-dict.html#Compatibility">Compatibility:</a></th><td>Available in version 2.4.42 and later</td></tr>
1514</table>
1515            <p>
1516                If the validity of the OCSP response used in stapling falls below <var>duration</var>, 
1517                <code class="module"><a href="../mod/mod_md.html">mod_md</a></code> will obtain a new OCSP response.
1518            </p><p>
1519                The CA issuing a certificate commonly also operates the OCSP responder 
1520                service and determines how long its signed response about the validity 
1521                of a certificate are itself valid. The longer a response is valid, the longer 
1522                it can be cached which mean better overall performance for everyone. 
1523                The shorter the life time, the more rapidly certificate revocations
1524                spread to clients. Also, service reliability is a consideration.
1525            </p><p>
1526                By adjusting the stapling renew window you can control parts of this yourself. 
1527                If you make the renew time short (e.g. a short time before the current
1528                information expires), you gain maximum cache time. But a service outage
1529                (down for maintenance, for example) will affect you. If you renew a long
1530                time before expiry, updates will be made more frequent, cause more load
1531                on the CA server infrastructure and also more coordination between
1532                the child processes of your server.
1533            </p><p>
1534                The default is chosen as 33%, which means renewal is started when only 
1535                a third of the response lifetime is left. For a CA that issues OCSP 
1536                responses with lifetime of 3 days, this means 2 days of caching and 1 day 
1537                for renewal attempts. A service outage would have to last full 24 hours 
1538                to affect your domains.
1539            </p><p>
1540                Setting an absolute renew window, like `2d` (2 days), is also possible. 
1541            </p>
1542

1567<div class="directive-section"><h2><a name="MDStoreLocks" id="MDStoreLocks">MDStoreLocks</a> <a name="mdstorelocks" id="mdstorelocks">Directive</a></h2>
1568<table class="directive">
1569<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Configure locking of store for updates</td></tr>
1570<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>MDStoreLocks on|off|<var>duration</var></code></td></tr>
1571<tr><th><a href="directive-dict.html#Default">Default:</a></th><td><code>MDStoreLocks off</code></td></tr>
1572<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config</td></tr>
1573<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Experimental</td></tr>
1574<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_md</td></tr>
1575<tr><th><a href="directive-dict.html#Compatibility">Compatibility:</a></th><td>Available in version 2.4.55 and later</td></tr>
1576</table>
1577            <p>
1578                Enable this to use a lock file on server startup when
1579                <code class="directive">MDStoreDir</code> is synchronized with the server
1580                configuration and renewed certificates are activated.
1581            </p><p>
1582                Locking is intended for setups in a cluster that have a shared
1583                file system for MDStoreDir. It will protect the activation of
1584                renewed certificates when cluster nodes are restarted/reloaded
1585                at the same time. Under the condition that the shared file
1586                 system does support file locking.
1587            </p><p>
1588                The default duration to obtain the lock is 5 seconds. If the log
1589                cannot be obtained, an error is logged and the server startup will
1590                continue. This may result in a cluster node to still use the
1591                previous certificate afterwards.
1592            </p><p>
1593                A higher timeout will reduce that likelihood, but may delay server
1594                startups/reloads in case the locks are not properly handled in
1595                the underlying file system. A lock should only be held by a
1596                httpd instance for a short duration.
1597            </p>
1598
