39<h3>Sommaire</h3>
40
41    <p>Le module <code class="module"><a href="../mod/mod_dbd.html">mod_dbd</a></code> gère les connexions
42    à une base de données SQL via <a class="glossarylink" href="../glossary.html#apr" title="voir glossaire">APR</a>. Il permet
43    aux modules qui requièrent des fonctions liées aux bases de données
44    SQL de se connecter à une base de données à la demande, et s'efforce
45    de conférer aux bases de données une efficacité et une
46    évolutivité optimales pour les MPMs threadés ou non threadés. Pour
47    plus de détails, voir le site web <a href="http://apr.apache.org/">APR</a>.
48</p>

76<h2><a name="pooling" id="pooling">Regroupement des connexions</a></h2>
77    <p>Ce module gère de manière optimisée en fonction de la plate-forme
78    les connexions aux bases de données. Sur les plates-formes non
79    threadées, il maintient une connexion persistente à la manière d'un
80    LAMP classique (Linux, Apache, Mysql, Perl/PHP/Python). Sur les
81    plates-formes threadées, il maintient un <em>groupe de
82    connexions</em> à la fois plus évolutif et plus efficace, comme
83    décrit dans <a href="http://www.apachetutor.org/dev/reslist">cet
84    article d'ApacheTutor</a>. Notez que <code class="module"><a href="../mod/mod_dbd.html">mod_dbd</a></code>
85    remplace les modules présentés dans cet article.</p>
86</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>

88<h2><a name="connecting" id="connecting">Connexion</a></h2>
89
90    <p>Pour vous connecter à votre base de données, vous devez spécifier un
91    pilote et des paramètres de connexion qui diffèrent selon le moteur de base
92    de données. Par exemple, pour vous connecter à mysql, spécifiez ce qui suit
93    :</p>
94
95<pre class="prettyprint lang-config">DBDriver mysql
96DBDParams host=localhost,dbname=pony,user=shetland,pass=appaloosa</pre>
97
98
99    <p>Vous pourrez alors utiliser cette connexion dans de nombreux autres
100    modules comme <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code>, <code class="module"><a href="../mod/mod_authn_dbd.html">mod_authn_dbd</a></code>
101    et <code class="module"><a href="../mod/mod_lua.html">mod_lua</a></code>. Vous trouverez des exemples d'utilisation dans
102    la documentation de ces modules.</p>
103
104    <p>Voir la syntaxe de la directive <code class="directive">DBDParams</code> pour les
105    informations à fournir dans la chaîne de connexion en fonction des
106    différents pilotes de base de données supportés.</p>

110<h2><a name="API" id="API">API DBD d'Apache</a></h2>
111    <p><code class="module"><a href="../mod/mod_dbd.html">mod_dbd</a></code> exporte cinq fonctions que d'autres
112    modules pourront utiliser. L'API se présente comme suit :</p>
113
114    <pre class="prettyprint lang-c">typedef struct {
115    apr_dbd_t *handle;
116    apr_dbd_driver_t *driver;
117    apr_hash_t *prepared;
118} ap_dbd_t;
119
120/* Fonctions exportées pour accéder à la base de données */
121
122/* ouvre une connexion qui DOIT avoir été explicitement fermée.
123 * Renvoie NULL en cas d'erreur
124 */
125AP_DECLARE(ap_dbd_t*) ap_dbd_open(apr_pool_t*, server_rec*);
126
127/* ferme une connexion ouverte avec ap_dbd_open */
128AP_DECLARE(void) ap_dbd_close(server_rec*, ap_dbd_t*);
129
130/* acquiert une connexion qui aura la durée de vie de la requête et qui
131 * NE DOIT PAS avoir été explicitement fermée. Renvoie NULL en cas
132 * d'erreur. C'est la fonction recommandée pour la plupart des
133 * applications.
134 */
135AP_DECLARE(ap_dbd_t*) ap_dbd_acquire(request_rec*);
136
137/* acquiert une connexion qui aura la durée de vie d'une connexion et
138 * qui NE DOIT PAS avoir été explicitement fermée. Renvoie NULL en cas
139 * d'erreur.
140 */
141AP_DECLARE(ap_dbd_t*) ap_dbd_cacquire(conn_rec*);
142
143/* Prépare une requête qu'un module client pourra utiliser */
144AP_DECLARE(void) ap_dbd_prepare(server_rec*, const char*, const char*);
145
146/* Exporte aussi ces fonctions à titre optionnel mour les modules qui
147 * péfèreraient les utiliser */
148APR_DECLARE_OPTIONAL_FN(ap_dbd_t*, ap_dbd_open, (apr_pool_t*, server_rec*));
149APR_DECLARE_OPTIONAL_FN(void, ap_dbd_close, (server_rec*, ap_dbd_t*));
150APR_DECLARE_OPTIONAL_FN(ap_dbd_t*, ap_dbd_acquire, (request_rec*));
151APR_DECLARE_OPTIONAL_FN(ap_dbd_t*, ap_dbd_cacquire, (conn_rec*));
152APR_DECLARE_OPTIONAL_FN(void, ap_dbd_prepare, (server_rec*, const char*, const char*));</pre>

156<h2><a name="prepared" id="prepared">Requêtes SQL préparées</a></h2>
157    <p><code class="module"><a href="../mod/mod_dbd.html">mod_dbd</a></code> supporte les requêtes SQL préparées à
158    destination des modules qui pourraient les utiliser. Chaque requête
159    préparée doit posséder un nom (étiquette), et est stockée dans un
160    condensé (hash) : les condensés sont du type
161    <code>apr_dbd_prepared_t</code> et s'utilisent dans toute requête
162    SQL ou commande select préparée par apr_dbd.</p>
163
164    <p>Il est du ressort des modules utilisateurs de dbd d'utiliser les
165    requêtes préparées et de préciser quelles requêtes doivent être
166    spécifiées dans httpd.conf, ou de fournir leurs propres directives
167    et d'utiliser <code>ap_dbd_prepare</code>.</p>
168
169    <div class="warning"><h3>Avertissement</h3>
170	Lorsqu'on utilise des requêtes préparées avec des bases de
171	données MySQL, il est préférable de définir
172	<code>reconnect</code> à 0 dans la chaîne de connexion, afin
173	d'éviter des erreurs provoquées par un client MySQL qui se
174	reconnecterait sans réinitialiser correctement les requêtes
175	préparées. Si <code>reconnect</code> est défini à 1, toute
176	connexion défectueuse sera sensée être réparée, mais comme
177	mod_dbd n'en est pas informé, les requêtes préparées seront
178	invalidées.
179	</div>

182<h2><a name="security" id="security">AVERTISSEMENT DE SECURITE</a></h2>
183
184    <p>Toute application web impliquant une base de données doit se
185    protéger elle-même contre les attaques de type injection SQL. Dans
186    la plupart des cas Apache DBD est sûr, car les applications
187    utilisent des requêtes préparées, et les entrées non sures ne seront
188    utilisées qu'à titre de données. Bien entendu, si vous l'utilisez
189    via un module tiers, vous devez être au fait des précautions à
190    prendre.</p>
191    <p>Cependant, le pilote <var>FreeTDS</var> est <strong>non
192    sûr</strong> de par sa nature-même. Comme la bibliothèque
193    sous-jacente ne supporte pas les requêtes préparées, le pilote en
194    effectue une émulation, et les entrées non sûres sont fusionnées
195    avec la requête SQL.</p>
196    <p>Il peut être sécurisé en <em>décontaminant</em> toutes les
197    entrées : un processus inspiré de la recherche de contaminations
198    (<strong>taint mode</strong>) de
199    Perl. Chaque entrée est comparée à une expression rationnelle, et
200    seules les entrées qui correspondent sont utilisées, en accord avec
201    le langage Perl :</p>
202    <div class="example"><pre><code>  $untrusted =~ /([a-z]+)/;
203  $trusted = $1;</code></pre></div>
204    <p>Pour utiliser ceci, les expressions rationnelles de
205    décontamination doivent être incluses dans les requêtes préparées.
206    L'expression rationnelle doit se situer immédiatement après le
207    caractère % dans la requête préparée, et doit être entourée
208    d'accolades {}. Par exemple, si votre application attend une entrée
209    alphanumérique, vous pouvez utiliser :</p>
210    <div class="example"><p><code>
211       <code>"SELECT foo FROM bar WHERE input = %s"</code>
212    </code></p></div>
213    <p>avec d'autres pilotes, et ne risquer au pire qu'une requête
214    échouée. Mais avec FreeTDS, vous devez utiliser :</p>
215    <div class="example"><p><code>
216       <code>"SELECT foo FROM bar WHERE input = %{([A-Za-z0-9]+)}s"</code>
217    </code></p></div>
218    <p>tout ce qui ne correspond pas à l'expression rationnelle est
219    alors rejeté, et la requête est maintenant sûre.</p>
220    <p>Alternativement, vous pouvez utiliser le pilote ODBC tiers, qui
221    offre la sécurité des requêtes préparées authentiques.</p>
222</div>

298<div class="directive-section"><h2><a name="dbdparams" id="dbdparams">Directive</a> <a name="DBDParams" id="DBDParams">DBDParams</a></h2>
299<table class="directive">
300<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Paramètres de la connexion à la base de
301données</td></tr>
302<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>DBDParams
303<var>param1</var>=<var>valeur1</var>[,<var>param2</var>=<var>valeur2</var>]</code></td></tr>
304<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>configuration globale, serveur virtuel</td></tr>

308    <p>Cette directive permet de spécifier des paramètres selon les
309    besoins du pilote concerné. En général, les paramètres à passer
310    concernent tout ce qui n'a pas de valeur par défaut comme le nom
311    d'utilisateur, le mot de passe, le nom de la base de données, le nom
312    d'hôte et le numéro de port de la connexion.</p>
313    <p>Les paramètres de la chaîne de connexion en fonction des
314    différents pilotes comprennent :</p>
315    <dl>
316    <dt>FreeTDS (pour MSSQL et SyBase)</dt>
317    <dd>username, password, appname, dbname, host, charset, lang, server</dd>
318    <dt>MySQL</dt>
319    <dd>host, port, user, pass, dbname, sock, flags, fldsz, group, reconnect</dd>
320    <dt>Oracle</dt>
321    <dd>user, pass, dbname, server</dd>
322    <dt>PostgreSQL</dt>
323    <dd>La chaîne de connexion est passée directement à <code>PQconnectdb</code></dd>
324    <dt>SQLite2</dt>
325    <dd>La chaîne de connexion est scindée avec comme séparateur le
326    caractère ':', et <code>partie1:partie2</code> est utilisé dans
327    <code>sqlite_open(partie1, atoi(partie2), NULL)</code></dd>
328    <dt>SQLite3</dt>
329    <dd>La chaîne de connexion est passée directement à <code>sqlite3_open</code></dd>
330    <dt>ODBC</dt>
331    <dd>datasource, user, password, connect, ctimeout, stimeout, access, txmode, bufsize</dd>
332    </dl>

336<div class="directive-section"><h2><a name="dbdpersist" id="dbdpersist">Directive</a> <a name="DBDPersist" id="DBDPersist">DBDPersist</a></h2>
337<table class="directive">
338<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Utiliser ou non des connexions persistentes</td></tr>
339<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>DBDPersist On|Off</code></td></tr>
340<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>configuration globale, serveur virtuel</td></tr>
341<tr><th><a href="directive-dict.html#Status">Statut:</a></th><td>Extension</td></tr>
342<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_dbd</td></tr>
343</table>
344    <p>Si cette directive est définie à Off, les connexions persistentes
345    et les connexions groupées sont désactivées. À la demande d'un
346    client, une nouvelle connexion à la base de données est ouverte, et
347    fermée immédiatement à l'issue du traitement. Cette configuration ne
348    doit être utilisée qu'à des fins de débogage, ou sur des serveurs à
349    charge faible.</p>
350
351    <p>Par défaut, les groupes de connexions persistentes sont activés
352    (ou une seule connexion persistente du style LAMP pour les serveurs
353    non threadés), et c'est la configuration qui devrait être utilisée
354    dans la plupart des cas sur un serveur en production.</p>
355
356    <p>Avant la version 2.2.2, cette directive n'acceptait que les
357    valeurs <code>0</code> et <code>1</code> au lieu de <code>Off</code>
358    et <code>On</code>, respectivement.</p>
359
360</div>

362<div class="directive-section"><h2><a name="dbdpreparesql" id="dbdpreparesql">Directive</a> <a name="DBDPrepareSQL" id="DBDPrepareSQL">DBDPrepareSQL</a></h2>
363<table class="directive">
364<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Définit une requête SQL préparée</td></tr>
365<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>DBDPrepareSQL <var>"requête SQL"</var> <var>étiquette</var></code></td></tr>
366<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>configuration globale, serveur virtuel</td></tr>
367<tr><th><a href="directive-dict.html#Status">Statut:</a></th><td>Extension</td></tr>
368<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_dbd</td></tr>
369</table>
370    <p>Pour les modules tels que les modules d'authentification, qui
371    utilisent de manière répétée la même requête SQL, on peut optimiser
372    les performances en préparant la requête une fois pour toutes au
373    démarrage, plutôt qu'à chaque utilisation. Cette directive permet de
374    préparer une requête SQL et de lui assigner une étiquette.</p>

378<div class="directive-section"><h2><a name="dbdriver" id="dbdriver">Directive</a> <a name="DBDriver" id="DBDriver">DBDriver</a></h2>
379<table class="directive">
380<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Spécifie un pilote SQL</td></tr>
381<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>DBDriver <var>nom</var></code></td></tr>
382<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>configuration globale, serveur virtuel</td></tr>
383<tr><th><a href="directive-dict.html#Status">Statut:</a></th><td>Extension</td></tr>
384<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_dbd</td></tr>
385</table>
386    <p>Cette directive permet de spécifier un pilote apr_dbd par son
387    nom. Le pilote doit être installé sur votre système (sur la plupart
388    des systèmes, il s'agit d'un objet partagé ou d'une dll). Par
389    exemple, <code>DBDriver mysql</code> va sélectionner le pilote MySQL
390    dans la bibliothèque apr_dbd_mysql.so.</p>
