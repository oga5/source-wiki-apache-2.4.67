39<h3>Summary</h3>
40
41    <p>This module provides for flexible logging of client
42    requests. Logs are written in a customizable format, and may be
43    written directly to a file, or to an external program.
44    Conditional logging is provided so that individual requests may
45    be included or excluded from the logs based on characteristics
46    of the request.</p>
47
48    <p>Three directives are provided by this module:
49    <code class="directive"><a href="#transferlog">TransferLog</a></code> to create
50    a log file, <code class="directive"><a href="#logformat">LogFormat</a></code>
51    to set a custom format, and <code class="directive"><a href="#customlog">CustomLog</a></code> to define a log file and format in one
52    step. The <code class="directive">TransferLog</code> and <code class="directive">CustomLog</code> directives can be used multiple times in each
53    server to cause each request to be logged to multiple files.</p>

73<h2><a name="formats" id="formats">Custom Log Formats</a></h2>
74
75    <p>The format argument to the <code class="directive"><a href="#logformat">LogFormat</a></code> and <code class="directive"><a href="#customlog">CustomLog</a></code> directives is a string. This string is
76    used to log each request to the log file. It can contain literal
77    characters copied into the log files and the C-style control
78    characters "\n" and "\t" to represent new-lines and tabs.
79    Literal quotes and backslashes should be escaped with
80    backslashes.</p>
81
82    <p>The characteristics of the request itself are logged by
83    placing "<code>%</code>" directives in the format string, which are
84    replaced in the log file by the values as follows:</p>

264    <h3><a name="modifiers" id="modifiers">Modifiers</a></h3>
265
266      <p>Particular items can be restricted to print only for
267      responses with specific HTTP status codes by placing a
268      comma-separated list of status codes immediately following the
269      "%". The status code list may be preceded by a "<code>!</code>" to
270      indicate negation.</p>
271
272    <table class="bordered"><tr class="header"><th>Format String</th>
273    <th>Meaning</th></tr>
274<tr>
275    <td><code>%400,501{User-agent}i</code></td>
276    <td>Logs <code>User-agent</code> on 400 errors and 501 errors only. For
277      other status codes, the literal string <code>"-"</code> will be
278      logged.</td></tr>
279<tr class="odd"><td><code>%!200,304,302{Referer}i</code></td>
280    <td>Logs <code>Referer</code> on all requests that do
281    <em>not</em> return one of the three specified codes,
282    "<code>-</code>" otherwise.
283    </td></tr>
284</table>
285
286      <p>The modifiers "&lt;" and "&gt;" can be used for requests that
287      have been internally redirected to choose whether the original
288      or final (respectively) request should be consulted.  By
289      default, the <code>%</code> directives <code>%s, %U, %T,
290      %D,</code> and <code>%r</code> look at the original request
291      while all others look at the final request.  So for example,
292      <code>%&gt;s</code> can be used to record the final status of
293      the request and <code>%&lt;u</code> can be used to record the
294      original authenticated user on a request that is internally
295      redirected to an unauthenticated resource.</p>

299    <h3><a name="format-notes" id="format-notes">Format Notes</a></h3>
300
301      <p>For security reasons, starting with version 2.0.46,
302      non-printable and other special characters in format strings
303      that may contain arbitrary string data are escaped using
304      <code>\x<var>hh</var></code> sequences, where <var>hh</var>
305      stands for the hexadecimal representation of the raw
306      byte. Exceptions from this rule are <code>"</code> and
307      <code>\</code>, which are escaped by prepending a backslash, and
308      all whitespace characters, which are written in their C-style
309      notation (<code>\n</code>, <code>\t</code>, etc).  In versions
310      prior to 2.0.46, no escaping was performed on these strings so
311      you had to be quite careful when dealing with raw log files.</p>

349      <p>The <code>%b</code> and
350      <code>%B</code> format strings do not represent the number of
351      bytes sent to the client, but simply the size in bytes of the
352      HTTP response (which will differ, for instance, if the
353      connection is aborted, or if SSL is used).  The <code>%O</code>
354      format provided by <code class="module"><a href="../mod/mod_logio.html">mod_logio</a></code> will log the
355      actual number of bytes sent over the network.</p>
356
357      <div class="note">
358      <p>Note: <code class="module"><a href="../mod/mod_cache.html">mod_cache</a></code> is implemented as a
359      quick-handler and not as a standard handler. Therefore, the
360      <code>%R</code> format string will not return any handler
361      information when content caching is involved.</p>
362      </div>
363
364

366    <h3><a name="examples" id="examples">Examples</a></h3>
367
368      <p>Some commonly used log format strings are:</p>
369
370      <dl>
371        <dt>Common Log Format (CLF)</dt>
372        <dd><code>"%h %l %u %t \"%r\" %&gt;s %b"</code></dd>
373
374        <dt>Common Log Format with Virtual Host</dt>
375        <dd><code>"%v %h %l %u %t \"%r\" %&gt;s %b"</code></dd>
376
377        <dt>NCSA extended/combined log format</dt>
378        <dd><code>"%h %l %u %t \"%r\" %&gt;s %b \"%{Referer}i\"
379        \"%{User-agent}i\""</code></dd>
380
381        <dt>Referer log format</dt>
382        <dd><code>"%{Referer}i -&gt; %U"</code></dd>
383
384        <dt>Agent (Browser) log format</dt>
385        <dd><code>"%{User-agent}i"</code></dd>
386      </dl>
387
388      <p>You can use the <code>%{format}t</code> directive multiple
389      times to build up a time format using the extended format tokens
390      like <code>msec_frac</code>:</p>
391      <dl>
392<dt>Timestamp including milliseconds</dt>
393<dd><code>"%{%d/%b/%Y %T}t.%{msec_frac}t %{%z}t"</code></dd>
394
395      </dl>

407<div class="directive-section"><h2><a name="BufferedLogs" id="BufferedLogs">BufferedLogs</a> <a name="bufferedlogs" id="bufferedlogs">Directive</a></h2>
408<table class="directive">
409<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Buffer log entries in memory before writing to disk</td></tr>
410<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>BufferedLogs On|Off</code></td></tr>
411<tr><th><a href="directive-dict.html#Default">Default:</a></th><td><code>BufferedLogs Off</code></td></tr>
412<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config</td></tr>
413<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Base</td></tr>
414<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_log_config</td></tr>
415</table>
416    <p>The <code class="directive">BufferedLogs</code> directive causes
417    <code class="module"><a href="../mod/mod_log_config.html">mod_log_config</a></code> to store several log entries in
418    memory and write them together to disk, rather than writing them
419    after each request.  On some systems, this may result in more
420    efficient disk access and hence higher performance.  It may be
421    set only once for the entire server; it cannot be configured
422    per virtual-host.</p>
423
424    <div class="note">This directive should be used with caution as a crash might
425    cause loss of logging data.</div>
426
427</div>

429<div class="directive-section"><h2><a name="CustomLog" id="CustomLog">CustomLog</a> <a name="customlog" id="customlog">Directive</a></h2>
430<table class="directive">
431<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Sets filename and format of log file</td></tr>
432<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>CustomLog  <var>file</var>|<var>pipe</var>
433<var>format</var>|<var>nickname</var>
434[env=[!]<var>environment-variable</var>|
435expr=<var>expression</var>]</code></td></tr>
436<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config, virtual host</td></tr>
437<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Base</td></tr>
438<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_log_config</td></tr>
439</table>
440    <p>The <code class="directive">CustomLog</code> directive is used to
441    log requests to the server. A log format is specified, and the
442    logging can optionally be made conditional on request
443    characteristics using environment variables.</p>
444
445    <p>The first argument, which specifies the location to which
446    the logs will be written, can take one of the following two
447    types of values:</p>
448
449    <dl>
450      <dt><var>file</var></dt>
451      <dd>A filename, relative to the <code class="directive"><a href="../mod/core.html#serverroot">ServerRoot</a></code>.</dd>
452
453      <dt><var>pipe</var></dt>
454      <dd>The pipe character "<code>|</code>", followed by the path
455      to a program to receive the log information on its standard
456      input. See the notes on <a href="../logs.html#piped">piped logs</a>
457      for more information.
458
459      <div class="warning"><h3>Security:</h3>
460      <p>If a program is used, then it will be run as the user who
461      started <code class="program"><a href="../programs/httpd.html">httpd</a></code>. This will be root if the server was
462      started by root; be sure that the program is secure.</p>
463      </div>
464      <div class="warning"><h3>Note</h3>
465        <p>When entering a file path on non-Unix platforms, care should be taken
466        to make sure that only forward slashed are used even though the platform
467        may allow the use of back slashes. In general it is a good idea to always
468        use forward slashes throughout the configuration files.</p>
469      </div></dd>
470    </dl>
471
472    <p>The second argument specifies what will be written to the
473    log file. It can specify either a <var>nickname</var> defined by
474    a previous <code class="directive"><a href="#logformat">LogFormat</a></code>
475    directive, or it can be an explicit <var>format</var> string as
476    described in the <a href="#formats">log formats</a> section.</p>
477
478    <p>For example, the following two sets of directives have
479    exactly the same effect:</p>
480
481    <pre class="prettyprint lang-config"># CustomLog with format nickname
482LogFormat "%h %l %u %t \"%r\" %&gt;s %b" common
483CustomLog "logs/access_log" common
484
485# CustomLog with explicit format string
486CustomLog "logs/access_log" "%h %l %u %t \"%r\" %&gt;s %b"</pre>
487
488
489    <p>The third argument is optional and controls whether or
490    not to log a particular request. The condition can be the
491    presence or absence (in the case of a '<code>env=!<var>name</var></code>'
492    clause) of a particular variable in the server
493    <a href="../env.html">environment</a>. Alternatively, the condition
494    can be expressed as arbitrary boolean <a href="../expr.html">expression</a>. If the condition is not satisfied, the request
495    will not be logged. References to HTTP headers  in the expression
496    will not cause the header names to be added to the Vary header.</p>
497
498    <p>Environment variables can be set on a per-request
499    basis using the <code class="module"><a href="../mod/mod_setenvif.html">mod_setenvif</a></code>
500    and/or <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code> modules. For
501    example, if you want to record requests for all GIF
502    images on your server in a separate logfile but not in your main
503    log, you can use:</p>
504
505    <pre class="prettyprint lang-config">SetEnvIf Request_URI \.gif$ gif-image
506CustomLog "gif-requests.log" common env=gif-image
507CustomLog "nongif-requests.log" common env=!gif-image</pre>
508
509
510    <p>Or, to reproduce the behavior of the old RefererIgnore
511    directive, you might use the following:</p>
512
513    <pre class="prettyprint lang-config">SetEnvIf Referer example\.com localreferer
514CustomLog "referer.log" referer env=!localreferer</pre>
515
516
517</div>

519<div class="directive-section"><h2><a name="GlobalLog" id="GlobalLog">GlobalLog</a> <a name="globallog" id="globallog">Directive</a></h2>
520<table class="directive">
521<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Sets filename and format of log file</td></tr>
522<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>GlobalLog<var>file</var>|<var>pipe</var>
523<var>format</var>|<var>nickname</var>
524[env=[!]<var>environment-variable</var>|
525expr=<var>expression</var>]</code></td></tr>
526<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config</td></tr>
527<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Base</td></tr>
528<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_log_config</td></tr>
529<tr><th><a href="directive-dict.html#Compatibility">Compatibility:</a></th><td>Available in Apache HTTP Server 2.4.19 and later</td></tr>
530</table>
531
532    <p>The <code class="directive">GlobalLog</code> directive defines a log shared
533       by the main server configuration and all defined virtual hosts.</p>
534
535    <p>The <code class="directive">GlobalLog</code> directive is identical to
536    the <code class="directive">CustomLog</code> directive, apart from the following
537    differences:</p>
538    <ul>
539       <li><code class="directive">GlobalLog</code> is not valid in virtual host
540            context.</li>
541       <li><code class="directive">GlobalLog</code> is used by virtual hosts that
542           define their own <code class="directive">CustomLog</code>, unlike a 
543           globally specified <code class="directive">CustomLog</code>.</li>
544    </ul>
545
546</div>

548<div class="directive-section"><h2><a name="LogFormat" id="LogFormat">LogFormat</a> <a name="logformat" id="logformat">Directive</a></h2>
549<table class="directive">
550<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Describes a format for use in a log file</td></tr>
551<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>LogFormat <var>format</var>|<var>nickname</var>
552[<var>nickname</var>]</code></td></tr>
553<tr><th><a href="directive-dict.html#Default">Default:</a></th><td><code>LogFormat "%h %l %u %t \"%r\" %&gt;s %b"</code></td></tr>
554<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config, virtual host</td></tr>
555<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Base</td></tr>
556<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_log_config</td></tr>
557</table>
558    <p>This directive specifies the format of the access log
559    file.</p>
560
561    <p>The <code class="directive">LogFormat</code> directive can take one of two
562    forms. In the first form, where only one argument is specified,
563    this directive sets the log format which will be used by logs
564    specified in subsequent <code class="directive">TransferLog</code>
565    directives. The single argument can specify an explicit
566    <var>format</var> as discussed in the <a href="#formats">custom log
567    formats</a> section above. Alternatively, it can use a
568    <var>nickname</var> to refer to a log format defined in a
569    previous <code class="directive">LogFormat</code> directive as described
570    below.</p>
571
572    <p>The second form of the <code class="directive">LogFormat</code>
573    directive associates an explicit <var>format</var> with a
574    <var>nickname</var>. This <var>nickname</var> can then be used in
575    subsequent <code class="directive">LogFormat</code> or
576    <code class="directive"><a href="#customlog">CustomLog</a></code> directives
577    rather than repeating the entire format string. A
578    <code class="directive">LogFormat</code> directive that defines a nickname
579    <strong>does nothing else</strong> -- that is, it <em>only</em>
580    defines the nickname, it doesn't actually apply the format and make
581    it the default. Therefore, it will not affect subsequent
582    <code class="directive"><a href="#transferlog">TransferLog</a></code> directives.
583    In addition, <code class="directive">LogFormat</code> cannot use one nickname
584    to define another nickname. Note that the nickname should not contain
585    percent signs (<code>%</code>).</p>
586
587    <div class="example"><h3>Example</h3><pre class="prettyprint lang-config">LogFormat "%v %h %l %u %t \"%r\" %&gt;s %b" vhost_common</pre>
588</div>

593<div class="directive-section"><h2><a name="TransferLog" id="TransferLog">TransferLog</a> <a name="transferlog" id="transferlog">Directive</a></h2>
594<table class="directive">
595<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Specify location of a log file</td></tr>
596<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>TransferLog <var>file</var>|<var>pipe</var></code></td></tr>
597<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config, virtual host</td></tr>
598<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Base</td></tr>
599<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_log_config</td></tr>
600</table>
601    <p>This directive has exactly the same arguments and effect as
602    the <code class="directive"><a href="#customlog">CustomLog</a></code>
603    directive, with the exception that it does not allow the log format
604    to be specified explicitly or for conditional logging of requests.
605    Instead, the log format is determined by the most recently specified
606    <code class="directive"><a href="#logformat">LogFormat</a></code> directive
607    which does not define a nickname. Common Log Format is used if no
608    other format has been specified.</p>
609
610    <div class="example"><h3>Example</h3><pre class="prettyprint lang-config">LogFormat "%h %l %u %t \"%r\" %&gt;s %b \"%{Referer}i\" \"%{User-agent}i\""
611TransferLog logs/access_log</pre>
612</div>
