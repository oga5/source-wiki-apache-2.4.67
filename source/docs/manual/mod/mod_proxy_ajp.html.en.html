39<h3>Summary</h3>
40
41    <p>This module <em>requires</em> the service of <code class="module"><a href="../mod/mod_proxy.html">mod_proxy</a></code>. It provides support for the
42    <code>Apache JServ Protocol version 1.3</code> (hereafter
43    <em>AJP13</em>).</p>
44
45    <p>Thus, in order to get the ability of handling <code>AJP13</code>
46    protocol, <code class="module"><a href="../mod/mod_proxy.html">mod_proxy</a></code> and
47    <code class="module"><a href="../mod/mod_proxy_ajp.html">mod_proxy_ajp</a></code> have to be present in the server.</p>
48
49    <div class="warning"><h3>Warning</h3>
50      <p>Do not enable proxying until you have <a href="mod_proxy.html#access">secured your server</a>. Open proxy
51      servers are dangerous both to your network and to the Internet at
52      large.</p>
53    </div>

73<h2><a name="usage" id="usage">Usage</a></h2>
74    <p>This module is used to reverse proxy to a backend application server
75    (e.g. Apache Tomcat) using the AJP13 protocol. The usage is similar to
76    an HTTP reverse proxy, but uses the <code>ajp://</code> prefix:</p>
77
78    <div class="example"><h3>Simple Reverse Proxy</h3><pre class="prettyprint lang-config">ProxyPass "/app" "ajp://backend.example.com:8009/app"</pre>
79</div>
80
81    <p>Options such as the <code>secret</code> option of Tomcat (required by
82    default since Tomcat 8.5.51 and 9.0.31) can just be added as a separate
83    parameter at the end of <code class="directive"><a href="../mod/mod_proxy.html#proxypass">ProxyPass</a></code>
84    or <code class="directive"><a href="../mod/mod_proxy.html#balancermember">BalancerMember</a></code>. This parameter
85    is available in Apache HTTP Server 2.4.42 and later:</p>
86    <div class="example"><h3>Simple Reverse Proxy with <code>secret</code> option</h3><pre class="prettyprint lang-config">ProxyPass "/app" "ajp://backend.example.com:8009/app" secret=YOUR_AJP_SECRET</pre>
87</div>
88
89    <p>Balancers may also be used:</p>
90    <div class="example"><h3>Balancer Reverse Proxy</h3><pre class="prettyprint lang-config">&lt;Proxy "balancer://cluster"&gt;
91    BalancerMember "ajp://app1.example.com:8009" loadfactor=1
92    BalancerMember "ajp://app2.example.com:8009" loadfactor=2
93    ProxySet lbmethod=bytraffic
94&lt;/Proxy&gt;
95ProxyPass "/app" "balancer://cluster/app"</pre>
96</div>
97
98    <p>Note that usually no
99    <code class="directive"><a href="../mod/mod_proxy.html#proxypassreverse">ProxyPassReverse</a></code>
100    directive is necessary. The AJP request includes the original host
101    header given to the proxy, and the application server can be expected
102    to generate self-referential headers relative to this host, so no
103    rewriting is necessary.</p>
104
105    <p>The main exception is when the URL path on the proxy differs from that
106    on the
107    backend. In this case, a redirect header can be rewritten relative to the
108    original host URL (not the backend <code>ajp://</code> URL), for
109    example:</p>
110    <div class="example"><h3>Rewriting Proxied Path</h3><pre class="prettyprint lang-config">ProxyPass "/apps/foo" "ajp://backend.example.com:8009/foo"
111ProxyPassReverse "/apps/foo" "http://www.example.com/foo"</pre>
112</div>
113    <p>However, it is usually better to deploy the application on the backend
114    server at the same path as the proxy rather than to take this approach.
115    </p>

124<h2><a name="overviewprotocol" id="overviewprotocol">Overview of the protocol</a></h2>
125    <p>The <code>AJP13</code> protocol is packet-oriented.  A binary format
126    was presumably chosen over the more readable plain text for reasons of
127    performance.  The web server communicates with the servlet container over
128    TCP connections.  To cut down on the expensive process of socket creation,
129    the web server will attempt to maintain persistent TCP connections to the
130    servlet container, and to reuse a connection for multiple request/response
131    cycles.</p>
132    <p>Once a connection is assigned to a particular request, it will not be
133    used for any others until the request-handling cycle has terminated.  In
134    other words, requests are not multiplexed over connections.  This makes
135    for much simpler code at either end of the connection, although it does
136    cause more connections to be open at once.</p>
137    <p>Once the web server has opened a connection to the servlet container,
138    the connection can be in one of the following states:</p>
139    <ul>
140    <li> Idle <br /> No request is being handled over this connection. </li>
141    <li> Assigned <br /> The connection is handling a specific request.</li>
142    </ul>
143    <p>Once a connection is assigned to handle a particular request, the basic
144    request information (e.g. HTTP headers, etc) is sent over the connection in
145    a highly condensed form (e.g. common strings are encoded as integers).
146    Details of that format are below in Request Packet Structure. If there is a
147    body to the request <code>(content-length &gt; 0)</code>, that is sent in a
148    separate packet immediately after.</p>
149    <p>At this point, the servlet container is presumably ready to start
150    processing the request.  As it does so, it can send the
151    following messages back to the web server:</p>
152    <ul>
153    <li>SEND_HEADERS <br />Send a set of headers back to the browser.</li>
154    <li>SEND_BODY_CHUNK <br />Send a chunk of body data back to the browser.
155    </li>
156    <li>GET_BODY_CHUNK <br />Get further data from the request if it hasn't all
157    been transferred yet.  This is necessary because the packets have a fixed
158    maximum size and arbitrary amounts of data can be included the body of a
159    request (for uploaded files, for example).  (Note: this is unrelated to
160    HTTP chunked transfer).</li>
161    <li>END_RESPONSE <br /> Finish the request-handling cycle.</li>
162    </ul>
163    <p>Each message is accompanied by a differently formatted packet of data.
164    See Response Packet Structures below for details.</p>

167<h2><a name="basppacketstruct" id="basppacketstruct">Basic Packet Structure</a></h2>
168    <p>There is a bit of an XDR heritage to this protocol, but it differs
169    in lots of ways (no 4 byte alignment, for example).</p>
170    <p>AJP13 uses network byte order for all data types.</p>
171    <p>There are four data types in the protocol: bytes, booleans,
172    integers and strings.</p>
173    <dl>
174    <dt><strong>Byte</strong></dt><dd>A single byte.</dd>
175    <dt><strong>Boolean</strong></dt>
176      <dd>A single byte, <code>1 = true</code>, <code>0 = false</code>.
177      Using other non-zero values as true (i.e. C-style) may work in some places,
178      but it won't in others.</dd>
179    <dt><strong>Integer</strong></dt>
180      <dd>A number in the range of <code>0 to 2^16 (32768)</code>.  Stored in
181      2 bytes with the high-order byte first.</dd>
182    <dt><strong>String</strong></dt>
183      <dd>A variable-sized string (length bounded by 2^16). Encoded with
184      the length packed into two bytes first, followed by the string
185      (including the terminating '\0').  Note that the encoded length does
186      <strong>not</strong> include the trailing '\0' -- it is like
187      <code>strlen</code>.  This is a touch confusing on the Java side, which
188      is littered with odd autoincrement statements to skip over these
189      terminators.  I believe the reason this was done was to allow the C
190      code to be extra efficient when reading strings which the servlet
191      container is sending back -- with the terminating \0 character, the
192      C code can pass around references into a single buffer, without copying.
193      if the \0 was missing, the C code would have to copy things out in order
194      to get its notion of a string.</dd>
195    </dl>
196
197  <h3>Packet Size</h3>
198    <p>According to much of the code, the max packet size is <code>
199    8 * 1024 bytes (8K)</code>.  The actual length of the packet is encoded in
200    the header.</p>
201  
202  <h3>Packet Headers</h3>
203    <p>Packets sent from the server to the container begin with
204    <code>0x1234</code>.  Packets sent from the container to the server
205    begin with <code>AB</code> (that's the ASCII code for A followed by the
206    ASCII code for B).  After those first two bytes, there is an integer
207    (encoded as above) with the length of the payload.  Although this might
208    suggest that the maximum payload could be as large as 2^16, in fact, the
209    code sets the maximum to be 8K.</p>

343<h2><a name="rpacetstruct" id="rpacetstruct">Request Packet Structure</a></h2>
344    <p>For messages from the server to the container of type
345    <em>Forward Request</em>:</p>
346    <div class="example"><pre>AJP13_FORWARD_REQUEST :=
347    prefix_code      (byte) 0x02 = JK_AJP13_FORWARD_REQUEST
348    method           (byte)
349    protocol         (string)
350    req_uri          (string)
351    remote_addr      (string)
352    remote_host      (string)
353    server_name      (string)
354    server_port      (integer)
355    is_ssl           (boolean)
356    num_headers      (integer)
357    request_headers *(req_header_name req_header_value)
358    attributes      *(attribut_name attribute_value)
359    request_terminator (byte) OxFF</pre></div>
360    <p>The <code>request_headers</code> have the following structure:
361    </p><div class="example"><pre>req_header_name :=
362    sc_req_header_name | (string)  [see below for how this is parsed]
363
364sc_req_header_name := 0xA0xx (integer)
365
366req_header_value := (string)</pre></div>
367    <p>The <code>attributes</code> are optional and have the following
368    structure:</p>
369    <div class="example"><pre>attribute_name := sc_a_name | (sc_a_req_attribute string)
370
371attribute_value := (string)</pre></div>
372    <p>Not that the all-important header is <code>content-length</code>,
373    because it determines whether or not the container looks for another
374    packet immediately.</p>

416  <h3>protocol, req_uri, remote_addr, remote_host, server_name,
417  server_port, is_ssl</h3>
418    <p>These are all fairly self-explanatory.  Each of these is required, and
419    will be sent for every request.</p>
420  
421  <h3>Headers</h3>
422    <p>The structure of <code>request_headers</code> is the following:
423    First, the number of headers <code>num_headers</code> is encoded.
424    Then, a series of header name <code>req_header_name</code> / value
425    <code>req_header_value</code> pairs follows.
426    Common header names are encoded as integers,
427    to save space.  If the header name is not in the list of basic headers,
428    it is encoded normally (as a string, with prefixed length).  The list of
429    common headers <code>sc_req_header_name</code>and their codes
430    is as follows (all are case-sensitive):</p>
431    <table>
432      <tr><td>Name</td><td>Code value</td><td>Code name</td></tr>
433      <tr><td>accept</td><td>0xA001</td><td>SC_REQ_ACCEPT</td></tr>
434      <tr><td>accept-charset</td><td>0xA002</td><td>SC_REQ_ACCEPT_CHARSET
435      </td></tr>
436      <tr><td>accept-encoding</td><td>0xA003</td><td>SC_REQ_ACCEPT_ENCODING
437      </td></tr>
438      <tr><td>accept-language</td><td>0xA004</td><td>SC_REQ_ACCEPT_LANGUAGE
439      </td></tr>
440      <tr><td>authorization</td><td>0xA005</td><td>SC_REQ_AUTHORIZATION</td>
441      </tr>
442      <tr><td>connection</td><td>0xA006</td><td>SC_REQ_CONNECTION</td></tr>
443      <tr><td>content-type</td><td>0xA007</td><td>SC_REQ_CONTENT_TYPE</td>
444      </tr>
445      <tr><td>content-length</td><td>0xA008</td><td>SC_REQ_CONTENT_LENGTH</td>
446      </tr>
447      <tr><td>cookie</td><td>0xA009</td><td>SC_REQ_COOKIE</td></tr>
448      <tr><td>cookie2</td><td>0xA00A</td><td>SC_REQ_COOKIE2</td></tr>
449      <tr><td>host</td><td>0xA00B</td><td>SC_REQ_HOST</td></tr>
450      <tr><td>pragma</td><td>0xA00C</td><td>SC_REQ_PRAGMA</td></tr>
451      <tr><td>referer</td><td>0xA00D</td><td>SC_REQ_REFERER</td></tr>
452      <tr><td>user-agent</td><td>0xA00E</td><td>SC_REQ_USER_AGENT</td></tr>
453    </table>
454    <p>The Java code that reads this grabs the first two-byte integer and if
455    it sees an <code>'0xA0'</code> in the most significant
456    byte, it uses the integer in the second byte as an index into an array of
457    header names.  If the first byte is not <code>0xA0</code>, it assumes that
458    the two-byte integer is the length of a string, which is then read in.</p>
459    <p>This works on the assumption that no header names will have length
460    greater than <code>0x9FFF (==0xA000 - 1)</code>, which is perfectly
461    reasonable, though somewhat arbitrary.</p>
462    <div class="note"><h3>Note:</h3>
463    The <code>content-length</code> header is extremely
464    important.  If it is present and non-zero, the container assumes that
465    the request has a body (a POST request, for example), and immediately
466    reads a separate packet off the input stream to get that body.
467    </div>
468  
469  <h3>Attributes</h3>
470    <p>The attributes prefixed with a <code>?</code>
471    (e.g. <code>?context</code>) are all optional.  For each, there is a
472    single byte code to indicate the type of attribute, and then its value
473    (string or integer).  They can be sent in any order (though the C code
474    always sends them in the order listed below).  A special terminating code
475    is sent to signal the end of the list of optional attributes. The list of
476    byte codes is:</p>
477    <table>
478      <tr><td>Information</td><td>Code Value</td><td>Type Of Value</td><td>Note</td></tr>
479      <tr><td>?context</td><td>0x01</td><td>-</td><td>Not currently implemented
480      </td></tr>
481      <tr><td>?servlet_path</td><td>0x02</td><td>-</td><td>Not currently implemented
482      </td></tr>
483      <tr><td>?remote_user</td><td>0x03</td><td>String</td><td /></tr>
484      <tr><td>?auth_type</td><td>0x04</td><td>String</td><td /></tr>
485      <tr><td>?query_string</td><td>0x05</td><td>String</td><td /></tr>
486      <tr><td>?jvm_route</td><td>0x06</td><td>String</td><td /></tr>
487      <tr><td>?ssl_cert</td><td>0x07</td><td>String</td><td /></tr>
488      <tr><td>?ssl_cipher</td><td>0x08</td><td>String</td><td /></tr>
489      <tr><td>?ssl_session</td><td>0x09</td><td>String</td><td /></tr>
490      <tr><td>?req_attribute</td><td>0x0A</td><td>String</td><td>Name (the name of the
491      attribute follows)</td></tr>
492      <tr><td>?ssl_key_size</td><td>0x0B</td><td>Integer</td><td /></tr>
493      <tr><td>?secret</td><td>0x0C</td><td>String</td><td>Supported since 2.4.42</td></tr>
494      <tr><td>are_done</td><td>0xFF</td><td>-</td><td>request_terminator</td></tr>
495    </table>
496    <p>The <code>context</code> and <code>servlet_path</code> are not
497    currently set by the C code, and most of the Java code completely ignores
498    whatever is sent over for those fields (and some of it will actually break
499    if a string is sent along after one of those codes).  I don't know if this
500    is a bug or an unimplemented feature or just vestigial code, but it's
501    missing from both sides of the connection.</p>
502    <p>The <code>remote_user</code> and <code>auth_type</code> presumably
503    refer to HTTP-level authentication, and communicate the remote user's
504    username and the type of authentication used to establish their identity
505    (e.g. Basic, Digest).</p>
506    <p>The <code>query_string</code>, <code>ssl_cert</code>,
507    <code>ssl_cipher</code>, <code>ssl_session</code> and
508    <code>ssl_key_size</code> refer to the
509    corresponding pieces of HTTP and HTTPS.</p>
510    <p>The <code>jvm_route</code>, is used to support sticky
511    sessions -- associating a user's sesson with a particular Tomcat instance
512    in the presence of multiple, load-balancing servers.</p>
513    <p>The <code>secret</code> is sent when the <code>secret=secret_keyword</code>
514    parameter is used in
515    <code class="directive"><a href="../mod/mod_proxy.html#proxypass">ProxyPass</a></code> or
516    <code class="directive"><a href="../mod/mod_proxy.html#balancermember">BalancerMember</a></code> directives.
517    The backend needs to support secret and the values must match.
518    <code>request.secret</code> or <code>requiredSecret</code> are documented in the AJP
519    configuration of the Apache Tomcat.</p>
520    <p>Beyond this list of basic attributes, any number of other attributes
521    can be sent via the <code>req_attribute</code> code <code>0x0A</code>.
522    A pair of strings to represent the attribute name and value are sent
523    immediately after each instance of that code.  Environment values are passed
524    in via this method.</p>
525    <p>Finally, after all the attributes have been sent, the attribute
526    terminator, <code>0xFF</code>, is sent.  This signals both the end of the
527    list of attributes and also then end of the Request Packet.</p>
528

562  <h3>Details:</h3>
563  <h3>Send Body Chunk</h3>
564    <p>The chunk is basically binary data, and is sent directly back to the
565    browser.</p>
566  
567  <h3>Send Headers</h3>
568    <p>The status code and message are the usual HTTP things
569    (e.g. <code>200</code> and <code>OK</code>). The response header names are
570    encoded the same way the request header names are. See header_encoding above
571    for details about how the codes are distinguished from the strings.<br />
572    The codes for common headers are:</p>
573    <table>
574      <tr><td>Name</td><td>Code value</td></tr>
575      <tr><td>Content-Type</td><td>0xA001</td></tr>
576      <tr><td>Content-Language</td><td>0xA002</td></tr>
577      <tr><td>Content-Length</td><td>0xA003</td></tr>
578      <tr><td>Date</td><td>0xA004</td></tr>
579      <tr><td>Last-Modified</td><td>0xA005</td></tr>
580      <tr><td>Location</td><td>0xA006</td></tr>
581      <tr><td>Set-Cookie</td><td>0xA007</td></tr>
582      <tr><td>Set-Cookie2</td><td>0xA008</td></tr>
583      <tr><td>Servlet-Engine</td><td>0xA009</td></tr>
584      <tr><td>Status</td><td>0xA00A</td></tr>
585      <tr><td>WWW-Authenticate</td><td>0xA00B</td></tr>
586    </table>
587    <p> After the code or the string header name, the header value is
588    immediately encoded.</p>
589  
590  <h3>End Response</h3>
591    <p>Signals the end of this request-handling cycle.  If the
592    <code>reuse</code> flag is true <code>(anything other than 0 in the actual
593    C code)</code>, this TCP connection can now be used to handle new incoming
594    requests.  If <code>reuse</code> is false (==0), the connection should
595    be closed.</p>
596  
597  <h3>Get Body Chunk</h3>
598    <p>The container asks for more data from the request (If the body was
599    too large to fit in the first packet sent over or when the request is
600    chunked). The server will send a body packet back with an amount of data
601    which is the minimum of the <code>request_length</code>, the maximum send
602    body size <code>(8186 (8 Kbytes - 6))</code>, and the number of bytes
603    actually left to send from the request body.<br />
604    If there is no more data in the body (i.e. the servlet container is
605    trying to read past the end of the body), the server will send back an
606    <em>empty</em> packet, which is a body packet with a payload length of 0.
607    <code>(0x12,0x34,0x00,0x00)</code></p>
