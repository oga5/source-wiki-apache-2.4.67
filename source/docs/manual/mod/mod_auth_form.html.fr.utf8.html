39<h3>Sommaire</h3>
40
41    <div class="warning"><h3>Avertissement</h3>
42      <p>L'authentification à base de formulaire dépend des modules
43      <code class="module"><a href="../mod/mod_session.html">mod_session</a></code> qui utilisent les cookies HTTP, et en
44      tant que tels s'exposent à des attaques de type Cross Site
45      Scripting, ou risquent de divulguer des informations à caractère
46      privé aux clients. Assurez-vous que ces risques ont bien été pris
47      en compte avant d'activer les sessions sur votre serveur.</p>
48    </div>
49
50    <p>Ce module permet de restreindre l'accès en recherchant les
51    utilisateurs dans les fournisseurs spécifiés à l'aide d'un
52    formulaire de connexion HTML. Les formulaires HTML requièrent
53    davantage de configuration que les méthodes d'authentification
54    alternatives, mais ils peuvent s'avérer beaucoup plus conviviaux
55    pour les utilisateurs.
56    </p>
57
58    <p>L'authentification HTTP de base est fournie par le module
59    <code class="module"><a href="../mod/mod_auth_basic.html">mod_auth_basic</a></code>, et l'authentification HTTP à base
60    de condensé par le module <code class="module"><a href="../mod/mod_auth_digest.html">mod_auth_digest</a></code>. Le
61    module <code class="module"><a href="../mod/mod_auth_form.html">mod_auth_form</a></code> doit être utilisé avec au
62    moins un module d'authentification du style
63    <code class="module"><a href="../mod/mod_authn_file.html">mod_authn_file</a></code> et un module d'autorisation comme
64    <code class="module"><a href="../mod/mod_authz_user.html">mod_authz_user</a></code>.</p>
65
66    <p>Lorsque l'utilisateur a été authentifié avec succès, ses
67    informations de connexion sont stockés dans une session fournie par
68    le module <code class="module"><a href="../mod/mod_session.html">mod_session</a></code>.
69    </p>

111<h2><a name="basicconfig" id="basicconfig">Configuration de base</a></h2>
112
113      <p>Pour protéger une URL particulière avec le module
114      <code class="module"><a href="../mod/mod_auth_form.html">mod_auth_form</a></code>, vous devez déterminer l'endroit où
115      vous allez stocker votre <var>session</var>, ainsi que la méthode
116      d'authentification. Dans cet exemple simple, les informations de
117      connexion sont stockées dans une session à l'aide du module
118      <code class="module"><a href="../mod/mod_session_cookie.html">mod_session_cookie</a></code>, et l'authentification utilise
119      un fichier en s'appuyant sur le module
120      <code class="module"><a href="../mod/mod_authn_file.html">mod_authn_file</a></code>. Si l'authentification échoue,
121      l'utilisateur dera redirigé vers la page du formulaire de
122      connexion.</p>
123
124      <div class="example"><h3>Exemple simple</h3><pre class="prettyprint lang-config">&lt;Location "/admin"&gt;
125    AuthFormProvider file
126    AuthUserFile "conf/passwd"
127    AuthType form
128    AuthName "/admin"
129    AuthFormLoginRequiredLocation "http://example.com/login.html"
130
131    Session On
132    SessionCookieName session path=/
133
134    Require valid-user
135&lt;/Location&gt;</pre>
136</div>
137
138      <p>L'authentification <code class="module"><a href="../mod/mod_auth_form.html">mod_auth_form</a></code> est activée
139      en affectant la valeur <var>form</var> à la directive <code class="directive"><a href="../mod/mod_authn_core.html#authtype">AuthType</a></code>. Les directives
140      <code class="directive"><a href="#authformprovider">AuthFormProvider</a></code> et
141      <code class="directive"><a href="../mod/mod_authn_file.html#authuserfile">AuthUserFile</a></code>
142      spécifient que les noms d'utilisateurs et mots de passe seront
143      vérifiés en utilisant le fichier choisi.</p>
144
145      <p>Les directives <code class="directive"><a href="../mod/mod_session.html#session">Session</a></code> et <code class="directive"><a href="../mod/mod_session_cookie.html#sessioncookiename">SessionCookieName</a></code> créent une
146      session chiffrée stockée dans un cookie HTTP au niveau
147      du navigateur. Pour plus d'informations à propos des différentes
148      options de configuration des sessions, reportez-vous à la
149      documentation du module <code class="module"><a href="../mod/mod_session.html">mod_session</a></code>.</p>
150
151      <p>Vous pouvez éventuellement ajouter une directive <code class="directive"><a href="../mod/mod_session_crypto.html#sessioncryptopassphrase">SessionCryptoPassphrase</a></code> pour créer
152      un cookie de session chiffré. Pour utiliser cette directive, le module
153      <code class="module"><a href="../mod/mod_session_crypto.html">mod_session_crypto</a></code> doit avoir été préalablement chargé.</p>
154
155      <p>Dans l'exemple simple ci-dessus, une URL a été protégée par
156      <code class="module"><a href="../mod/mod_auth_form.html">mod_auth_form</a></code>, mais on doit maintenant fournir
157      à l'utilisateur un moyen d'entrer un nom et un mot de passe. À cet
158      effet, on peut soit écrire une page de connexion indépendante
159      dédiée, soit inclure le formulaire de connexion dans la page
160      courante.</p>
161    </div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>

163<h2><a name="standalone" id="standalone">Page de connexion dédiée</a></h2>
164
165      <p>Le formulaire de connexion peut être contenu dans une page
166      indépendante, ou être inclus dans la page courante. </p>
167
168      <p>Lorsque la connexion s'effectue à partir d'une page
169      indépendante et si la tentative d'authentification échoue,
170      l'utilisateur doit être redirigé vers un formulaire de connexion,
171      créé à cet effet sur le site web, en utilisant la directive
172      <code class="directive"><a href="#authformloginrequiredlocation">AuthFormLoginRequiredLocation</a></code>.
173      En général, la page de connexion contiendra un formulaire HTML
174      demandant à l'utilisateur de fournir un nom et un mot de passe.</p>
175
176      <div class="example"><h3>Exemple de formulaire de connexion</h3><pre class="prettyprint lang-html">&lt;form method="POST" action="/dologin.html"&gt;
177  Username: &lt;input type="text" name="httpd_username" value="" /&gt;
178  Password: &lt;input type="password" name="httpd_password" value="" /&gt;
179  &lt;input type="submit" name="login" value="Login" /&gt;
180&lt;/form&gt;</pre>
181</div>
182
183      <p>La partie où s'effectue la connexion proprement dite est
184      traitée par le gestionnaire <var>form-login-handler</var>.
185      L'action de ce formulaire doit pointer vers ce gestionnaire, ce
186      que l'on configure dans Apache httpd comme suit :</p>
187
188      <div class="example"><h3>Exemple de configuration du gestionnaire de
189      formulaire de connexion</h3><pre class="prettyprint lang-config">&lt;Location "/dologin.html"&gt;
190    SetHandler form-login-handler
191    AuthFormLoginRequiredLocation "http://example.com/login.html"
192    AuthFormLoginSuccessLocation "http://example.com/admin/index.html"
193    AuthFormProvider file
194    AuthUserFile "conf/passwd"
195    AuthType form
196    AuthName /admin
197    Session On
198    SessionCookieName session path=/
199    SessionCryptoPassphrase secret
200&lt;/Location&gt;</pre>
201</div>
202
203      <p>L'URL spécifiée par la directive
204      <code class="directive"><a href="#authformloginrequiredlocation">AuthFormLoginRequiredLocation</a></code>
205      référencera en général une page expliquant à l'utilisateur que sa
206      tentative de connexion a échoué, et qu'il doit la renouveler. La
207      directive <code class="directive"><a href="#authformloginsuccesslocation">AuthFormLoginSuccessLocation</a></code>
208      spécifie l'URL vers laquelle l'utilisateur doit être redirigé s'il
209      s'est authentifié avec succès.</p>
210
211      <p>Alternativement, l'URL vers laquelle doit être redirigé
212      l'utilisateur s'il s'est authentifié avec succès peut être
213      intégrée dans le formulaire de connexion, comme dans l'exemple
214      ci-dessous. Il en découle que le même gestionnaire
215      <var>form-login-handler</var> pourra être utilisé pour différentes
216      zones du site web.</p>
217
218      <div class="example"><h3>Exemple de formulaire d'authentification multizone</h3><pre class="prettyprint lang-html">&lt;form method="POST" action="/dologin.html"&gt;
219  Username: &lt;input type="text" name="httpd_username" value="" /&gt;
220  Password: &lt;input type="password" name="httpd_password" value="" /&gt;
221  &lt;input type="submit" name="login" value="Login" /&gt;
222  &lt;input type="hidden" name="httpd_location" value="http://example.com/success.html" /&gt;
223&lt;/form&gt;</pre>
224</div>

228<h2><a name="inline" id="inline">Connexion à la volée</a></h2>
229
230      <div class="warning"><h3>Avertissement</h3>
231        <p>Il existe un risque, dans certaines circonstances, que le
232	formulaire de connexion configuré pour une connexion à la volée
233	soit soumis plusieurs fois, révélant de ce fait les paramètres
234	de connexion à l'application sous-jacente. L'administrateur doit
235	s'assurer que cette dernière est correctement sécurisée afin
236	d'éviter les éventuels abus. En cas de doute, utilisez une page
237	de connexion indépendante dédiée.</p>
238      </div>
239
240      <p>Comme alternative à la page de connexion dédiée pour un site
241      web, il est possible de configurer <code class="module"><a href="../mod/mod_auth_form.html">mod_auth_form</a></code>
242      pour authentifier les utilisateurs à la volée, sans les rediriger
243      vers une autre page, ce qui permet de conserver l'état de la page
244      courante au cours de la tentative de connexion. Ceci peut s'avérer
245      utile dans le cas d'une session limitée dans le temps, si le délai
246      de la session a expiré pendant la requête de l'utilisateur. Ce
247      dernier peut alors se réauthentifier à la même place, et
248      poursuivre son activité à partir du point où il en était resté.</p>
249
250      <p>Si un utilisateur non authentifié tente d'accéder à une page
251      protégée par <code class="module"><a href="../mod/mod_auth_form.html">mod_auth_form</a></code>, et si ce dernier
252      n'est pas configuré avec une directive <code class="directive"><a href="#authformloginrequiredlocation">AuthFormLoginRequiredLocation</a></code>,
253      un code de statut <var>HTTP_UNAUTHORIZED</var> est renvoyé vers le
254      navigateur, indiquant à l'utilisateur qu'il n'est pas autorisé à
255      accéder à cette page.</p>
256
257      <p>Pour configurer l'authentification à la volée, l'administrateur
258      remplace le message d'erreur renvoyé par le code de statut
259      <var>HTTP_UNAUTHORIZED</var> par un message d'erreur personnalisé
260      contenant le formulaire de connexion comme suit :</p>
261
262      <div class="example"><h3>Exemple simple d'authentification à la volée</h3><pre class="prettyprint lang-config">AuthFormProvider file
263ErrorDocument 401 "/login.shtml"
264AuthUserFile "conf/passwd"
265AuthType form
266AuthName realm
267Session On
268SessionCookieName session path=/</pre>
269</div>
270
271      <p>La page du message d'erreur doit contenir un formulaire de
272      connexion dont la propriété action est vide, comme dans l'exemple
273      ci-dessous. Ceci a pour effet de soumettre le formulaire à l'URL
274      protégée originale, cette dernière n'ayant pas besoin d'être
275      connue de la page en cours.</p>
276
277      <div class="example"><h3>Exemple de formulaire de connexion à la volée</h3><pre class="prettyprint lang-html">&lt;form method="POST" <strong>action=""</strong>&gt;
278  Username: &lt;input type="text" name="httpd_username" value="" /&gt;
279  Password: &lt;input type="password" name="httpd_password" value="" /&gt;
280  &lt;input type="submit" name="login" value="Login" /&gt;
281&lt;/form&gt;</pre>
282</div>
283
284      <p>Lorsque l'utilisateur final a entré ses informations de
285      connexion, le formulaire effectue une requête HTTP POST pour l'URL
286      originale protégée par mot de passe.
287      <code class="module"><a href="../mod/mod_auth_form.html">mod_auth_form</a></code> va alors intercepter cette requête
288      POST, et dans le cas où des champs HTML Utilisateur et Mot de
289      passe corrects sont présents, l'utilisateur sera connecté, et
290      l'URL originale protégée par mot de passe lui sera retournée en
291      tant que requête GET.</p>

294<div class="section">
295<h2><a name="inlinepreservebody" id="inlinepreservebody">Connexion à la volée avec
296    conservation du contenu</a></h2>
297
298      <p>Il existe une limite à la technique de connexion à la volée
299      décrite ci-dessus ; si un formulaire HTML POST entraîne une
300      demande d'authentification ou de réauthentification, le contenu du
301      formulaire original envoyé par le navigateur sera perdu. Cela peut
302      s'avérer plus ou moins gênant pour l'utilisateur final selon la
303      fonction du site web.</p>
304
305      <p>Comme solution à ce problème, <code class="module"><a href="../mod/mod_auth_form.html">mod_auth_form</a></code>
306      permet d'intégrer la méthode et le contenu de la requête originale
307      dans le formulaire de connexion. Si l'authentification réussit,
308      Apache httpd pourra refaire une tentative avec la méthode et le contenu
309      originaux, tout en conservant l'état de la requête originale.</p>
310
311      <p>Pour mettre en oeuvre la conservation du contenu, vous devez
312      ajouter trois champs supplémentaires au formulaire de connexion
313      comme dans l'exemple suivant :</p>
314
315      <div class="example"><h3>Exemple de formulaire avec conservation du
316      contenu</h3><pre class="prettyprint lang-html">&lt;form method="POST" action=""&gt;
317  Username: &lt;input type="text" name="httpd_username" value="" /&gt;
318  Password: &lt;input type="password" name="httpd_password" value="" /&gt;
319  &lt;input type="submit" name="login" value="Login" /&gt;
320  <br />  <strong>&lt;input type="hidden" name="httpd_method" value="POST" /&gt;
321  &lt;input type="hidden" name="httpd_mimetype" value="application/x-www-form-urlencoded" /&gt;
322  &lt;input type="hidden" name="httpd_body" value="name1=value1&amp;name2=value2" /&gt;</strong><br />
323&lt;/form&gt;</pre>
324</div>
325
326      <p>La manière dont la méthode, le type MIME et le contenu de la
327      requête originale seront intégrés dans le formulaire de connexion
328      vont dépendre de la plate-forme et de la technologie utilisées au
329      sein du site web.
330      </p>
331
332      <p>Une option consiste à utiliser le module
333      <code class="module"><a href="../mod/mod_include.html">mod_include</a></code> en association avec la directive
334      <code class="directive"><a href="../mod/mod_request.html#keptbodysize">KeptBodySize</a></code>, ainsi
335      qu'un script CGI adapté pour intégrer les variables dans le
336      formulaire.</p>
337
338      <p>Une autre option consiste à présenter le formulaire de
339      connexion en utilisant un script CGI ou une autre technologie
340      dynamique.</p>
341
342      <div class="example"><h3>Exemple avec script CGI</h3><pre class="prettyprint lang-config">        AuthFormProvider file
343        ErrorDocument 401 "/cgi-bin/login.cgi"
344        ...</pre>
345</div>

348<div class="section">
349<h2><a name="loggingout" id="loggingout">Déconnexion</a></h2>
350
351      <p>Pour permettre à un utilisateur de se déconnecter d'une session
352      particulière, vous devez configurer une page pour qu'elle soit
353      traitée par le gestionnaire <var>form-logout-handler</var>. Tout
354      accès à cette URL va entraîner la suppression de l'Utilisateur et
355      du Mot de passe de la session courante, ce qui aura pour effet de
356      déconnecter l'utilisateur.</p>
357
358      <p>Vous pouvez spécifier une URL vers laquelle le navigateur sera
359      redirigé en cas de déconnection réussie, en définissant la
360      directive <code class="directive"><a href="#authformlogoutlocation">AuthFormLogoutLocation</a></code>. Cette
361      URL devra expliquer à l'utilisateur qu'il a été déconnecté, et lui
362      donner la possibilité de se connecter à nouveau.</p>
363
364      <div class="example"><h3>Exemple simple de configuration de la
365      déconnexion</h3><pre class="prettyprint lang-config">SetHandler form-logout-handler
366AuthName realm
367AuthFormLogoutLocation "http://example.com/loggedout.html"
368Session On
369SessionCookieName session path=/</pre>
370</div>
371
372      <p>Notez que la déconnexion d'un utilisateur ne supprime pas la
373      session ; elle supprime seulement l'utilisateur et le mot de passe
374      de la session. Si la session qui en résulte est vide, elle sera
375      probablement supprimée, mais ce n'est pas garanti. Si vous voulez
376      être sûr que la session sera supprimée, affectez une valeur faible
377      à la directive <code class="directive"><a href="../mod/mod_session.html#sessionmaxage">SessionMaxAge</a></code>, par exemple 1
378      (affecter à cette directive la valeur zéro signifie une session
379      sans limite d'âge).
380      </p>
381
382      <div class="example"><h3>Exemple simple avec durée de validité de session
383      limitée</h3><pre class="prettyprint lang-config">SetHandler form-logout-handler
384AuthFormLogoutLocation "http://example.com/loggedout.html"
385Session On
386SessionMaxAge 1
387SessionCookieName session path=/</pre>
388</div>

425    <code class="directive"><a href="#authformprovider">AuthFormProvider</a></code>.

675<div class="directive-section"><h2><a name="authformprovider" id="authformprovider">Directive</a> <a name="AuthFormProvider" id="AuthFormProvider">AuthFormProvider</a></h2>
676<table class="directive">
677<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Définit le(s) fournisseur(s) d'authentification pour la
678zone concernée</td></tr>
679<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>AuthFormProvider <var>nom fournisseur</var>
680[<var>nom fournisseur</var>] ...</code></td></tr>
681<tr><th><a href="directive-dict.html#Default">Défaut:</a></th><td><code>AuthFormProvider file</code></td></tr>
682<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>répertoire, .htaccess</td></tr>
683<tr><th><a href="directive-dict.html#Override">Surcharges autorisées:</a></th><td>AuthConfig</td></tr>
684<tr><th><a href="directive-dict.html#Status">Statut:</a></th><td>Base</td></tr>
685<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_auth_form</td></tr>
686</table>
687    <p>La directive <code class="directive">AuthFormProvider</code> permet de
688    définir quel fournisseur sera utilisé pour authentifier les
689    utilisateurs pour la zone concernée. Le fournisseur par défaut
690    <code>file</code> est implémenté par le module
691    <code class="module"><a href="../mod/mod_authn_file.html">mod_authn_file</a></code>. Assurez-vous que le fournisseur
692    choisi soit bien présent dans le serveur.</p>
693
694    <div class="example"><h3>Exemple</h3><pre class="prettyprint lang-config">&lt;Location "/secure"&gt;
695    AuthType form
696    AuthName "private area"
697    AuthFormProvider  dbm
698    AuthDBMType        SDBM
699    AuthDBMUserFile    "/www/etc/dbmpasswd"
700    Require            valid-user
701    #...
702&lt;/Location&gt;</pre>
703</div>
704
705    <p>Les différents fournisseurs sont implémentés par les modules
706    <code class="module"><a href="../mod/mod_authn_dbm.html">mod_authn_dbm</a></code>, <code class="module"><a href="../mod/mod_authn_file.html">mod_authn_file</a></code>,
707    <code class="module"><a href="../mod/mod_authn_dbd.html">mod_authn_dbd</a></code> et
708    <code class="module"><a href="../mod/mod_authnz_ldap.html">mod_authnz_ldap</a></code>.</p>

712<div class="directive-section"><h2><a name="authformsitepassphrase" id="authformsitepassphrase">Directive</a> <a name="AuthFormSitePassphrase" id="AuthFormSitePassphrase">AuthFormSitePassphrase</a></h2>
713<table class="directive">
714<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Court-circuite l'authentification pour les sites à fort
715trafic</td></tr>
716<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>AuthFormSitePassphrase <var>secret</var></code></td></tr>
717<tr><th><a href="directive-dict.html#Default">Défaut:</a></th><td><code>none</code></td></tr>
718<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>répertoire</td></tr>
719<tr><th><a href="directive-dict.html#Status">Statut:</a></th><td>Base</td></tr>
720<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_auth_form</td></tr>
721<tr><th><a href="directive-dict.html#Compatibility">Compatibilité:</a></th><td>Disponible depuis la version 2.3.0 du serveur HTTP Apache</td></tr>
722</table>
723    <p>La directive <code class="directive"><a href="#authformsitepassphrase">AuthFormSitePassphrase</a></code>
724    spécifie un mot de passe qui, s'il est présent dans la session
725    utilisateur, indique à Apache httpd de court-circuiter l'authentification
726    pour l'URL considérée. On peut l'utiliser dans le cas de sites web à
727    fort trafic afin de réduire la charge induite sur l'infrastructure
728    d'authentification.</p>
729
730    <p>On peut insérer le mot de passe dans une session utilisateur en
731    ajoutant cette directive à la configuration concernant le
732    gestionnaire <var>form-login-handler</var>. Le gestionnaire
733    <var>form-login-handler</var>, quant à lui, effectuera toujours les
734    vérifications d'authentification, qu'un mot de passe soit spécifié
735    ou non.</p>
736
737    <div class="warning"><h3>Avertissement</h3>
738      <p>Si la session est présentée à l'utilisateur à l'aide du module
739      <code class="module"><a href="../mod/mod_session_cookie.html">mod_session_cookie</a></code>, et si la session n'est pas
740      protégée par le module <code class="module"><a href="../mod/mod_session_crypto.html">mod_session_crypto</a></code>, le mot
741      de passe peut faire l'objet d'une attaque de type dictionnaire.
742      Quelle que soit la configuration de la session, assurez-vous que
743      cette directive n'est pas utilisée dans un espace d'URLs contenant
744      des données privées, ou à partir desquelles des transactions
745      sensibles pourraient être menées. En tout état de cause, vous
746      devez être conscient des risques encourus avant de l'utiliser.</p>
747    </div>

752<div class="directive-section"><h2><a name="authformsize" id="authformsize">Directive</a> <a name="AuthFormSize" id="AuthFormSize">AuthFormSize</a></h2>
753<table class="directive">
754<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>La taille maximale en octets du formulaire dont seront
755extraites les informations de connexion</td></tr>
756<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>AuthFormSize <var>taille</var></code></td></tr>
757<tr><th><a href="directive-dict.html#Default">Défaut:</a></th><td><code>AuthFormSize 8192</code></td></tr>
758<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>répertoire</td></tr>
759<tr><th><a href="directive-dict.html#Status">Statut:</a></th><td>Base</td></tr>
760<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_auth_form</td></tr>
761<tr><th><a href="directive-dict.html#Compatibility">Compatibilité:</a></th><td>Disponible depuis la version 2.3.0 du serveur HTTP Apache</td></tr>
762</table>
763    <p>La directive <code class="directive"><a href="#authformsize">AuthFormSize</a></code> spécifie
764    la taille maximale du corps de la requête qui sera utilisée pour
765    trouver le formulaire de connexion.</p>
766
767    <p>Si une requête de connexion entrante possède une taille
768    supérieure à cette valeur, elle sera rejetée avec le code de réponse
769    HTTP <code>HTTP_REQUEST_TOO_LARGE</code>.</p>
770
771    <p>Si vous avez ajouté au formulaire des champs décrits dans <code class="directive"><a href="#authformmethod">AuthFormMethod</a></code>, <code class="directive"><a href="#authformmimetype">AuthFormMimetype</a></code> et <code class="directive"><a href="#authformbody">AuthFormBody</a></code>, il est recommandé
772    de définir cette directive à une valeur similaire à celle de la
773    directive <code class="directive"><a href="../mod/mod_request.html#keptbodysize">KeptBodySize</a></code>.</p>
