39<h3>Sommaire</h3>
40
41    <p>Ce module a été conçu dans le but d'améliorer les performances
42    des sites web s'appuyant sur des connexions en arrière-plan vers des
43    serveurs LDAP. Il ajoute aux fonctions fournies par les
44    bibliothèques standards LDAP la conservation des connexions LDAP
45    ainsi qu'un cache LDAP partagé en mémoire.</p>
46
47    <p>Pour activer ce module, le support LDAP doit être compilé dans
48    apr-util. Pour ce faire, on ajoute l'option <code>--with-ldap</code>
49    au script <code class="program"><a href="../programs/configure.html">configure</a></code> lorsqu'on construit
50    Apache.</p>
51
52    <p>Le support SSL/TLS est conditionné par le kit de développement
53    LDAP qui a été lié à <a class="glossarylink" href="../glossary.html#apr" title="voir glossaire">APR</a>. Au moment où ces
54    lignes sont écrites, APR-util supporte <a href="http://www.openldap.org/">OpenLDAP SDK</a> (version 2.x ou
55    supérieure), <a href="http://developer.novell.com/ndk/cldap.htm">Novell LDAP
56    SDK</a>, <a href="https://wiki.mozilla.org/LDAP_C_SDK">
57    Mozilla LDAP SDK</a>, le SDK LDAP Solaris natif (basé sur Mozilla)
58    ou le SDK LDAP Microsoft natif. Voir le site web <a href="http://apr.apache.org">APR</a> pour plus de détails.</p>

127<h2><a name="pool" id="pool">Conservation des connexions LDAP</a></h2>
128
129    <p>Les connexions LDAP sont conservées de requête en requête. Ceci
130    permet de rester connecté et identifié au serveur LDAP, ce dernier
131    étant ainsi prêt pour la prochaine requête, sans avoir à se
132    déconnecter, reconnecter et réidentifier. Le gain en performances
133    est similaire à celui des connexions persistantes (keepalives)
134    HTTP.</p>
135
136    <p>Sur un serveur très sollicité, il est possible que de nombreuses
137    requêtes tentent d'accéder simultanément à la même connexion au
138    serveur LDAP. Lorsqu'une connexion LDAP est utilisée, Apache en crée
139    une deuxième en parallèle à la première, ce qui permet d'éviter que
140    le système de conservation des connexions ne devienne un goulot
141    d'étranglement.</p>
142
143    <p>Il n'est pas nécessaire d'activer explicitement la conservation
144    des connexions dans la configuration d'Apache. Tout module utilisant
145    le module ldap pour accéder aux services LDAP partagera le jeu de
146    connexions.</p>
147
148    <p>Les connexions LDAP peuvent garder la trace des données
149    d'identification du client ldap utilisées pour l'identification
150    auprès du serveur LDAP. Ces données peuvent être fournies aux
151    serveurs LDAP qui ne permettent pas les connexions anonymes au cours
152    lors des tentatives de sauts vers des serveurs alternatifs. Pour
153    contrôler cette fonctionnalité, voir les directives <code class="directive"><a href="#ldapreferrals">LDAPReferrals</a></code> et <code class="directive"><a href="#ldapreferralhoplimit">LDAPReferralHopLimit</a></code>. Cette
154    fonctionnalité est activée par défaut.</p>

157<h2><a name="cache" id="cache">Cache LDAP</a></h2>
158
159    <p>Pour améliorer les performances, <code class="module"><a href="../mod/mod_ldap.html">mod_ldap</a></code> met en oeuvre
160    une stratégie de mise en cache agressive visant à minimiser le nombre de
161    fois que le serveur LDAP doit être contacté. La mise en cache peut
162    facilement doubler et même tripler le débit d'Apache lorsqu'il sert des
163    pages protégées par <code class="module"><a href="../mod/mod_authnz_ldap.html">mod_authnz_ldap</a></code>. De plus, le serveur
164    LDAP verra lui-même sa charge sensiblement diminuée.</p>
165
166    <p><code class="module"><a href="../mod/mod_ldap.html">mod_ldap</a></code> supporte deux types de mise en cache
167    LDAP : un <em>cache recherche/identification</em> durant la phase
168    de recherche/identification et deux <em>caches d'opérations</em>
169    durant la phase de comparaison. Chaque URL LDAP utilisée par le
170    serveur a son propre jeu d'instances dans ces trois caches.</p>
171
172    <h3><a name="search-bind" id="search-bind">Le cache
173    recherche/identification</a></h3>
174      <p>Les processus de recherche et d'identification sont les
175      opérations LDAP les plus consommatrices en temps, en particulier
176      si l'annuaire est de grande taille. Le cache de
177      recherche/identification met en cache toutes les recherches qui
178      ont abouti à une identification positive. Les résultats négatifs
179      (c'est à dire les recherches sans succès, ou les recherches qui
180      n'ont pas abouti à une identification positive) ne sont pas mis en
181      cache. La raison de cette décision réside dans le fait que les
182      connexions avec des données d'identification invalides ne
183      représentent qu'un faible pourcentage du nombre total de
184      connexions, et ainsi, le fait de ne pas mettre en cache les
185      données d'identification invalides réduira d'autant la taille du
186      cache.</p>
187
188      <p><code class="module"><a href="../mod/mod_ldap.html">mod_ldap</a></code> met en cache le nom d'utilisateur, le
189      DN extrait, le mot de passe utilisé pour l'identification, ainsi
190      que l'heure de l'identification. Chaque fois qu'une nouvelle
191      connexion est initialisée avec le même nom d'utilisateur,
192      <code class="module"><a href="../mod/mod_ldap.html">mod_ldap</a></code> compare le mot de passe de la nouvelle
193      connexion avec le mot de passe enregistré dans le cache. Si les
194      mots de passe correspondent, et si l'entrée du cache n'est pas
195      trop ancienne, <code class="module"><a href="../mod/mod_ldap.html">mod_ldap</a></code> court-circuite la phase
196      de recherche/identification.</p>
197
198      <p>Le cache de recherche/identification est contrôlé par les
199      directives <code class="directive"><a href="#ldapcacheentries">LDAPCacheEntries</a></code> et <code class="directive"><a href="#ldapcachettl">LDAPCacheTTL</a></code>.</p>
200    
201
202    <h3><a name="opcaches" id="opcaches">Les caches d'opérations</a></h3>
203      <p>Au cours des opérations de comparaison d'attributs et de noms
204      distinctifs (DN), <code class="module"><a href="../mod/mod_ldap.html">mod_ldap</a></code> utilise deux caches
205      d'opérations pour mettre en cache les opérations de comparaison.
206      Le premier cache de comparaison sert à mettre en cache les
207      résultats de comparaisons effectuées pour vérifier l'appartenance
208      à un groupe LDAP. Le second cache de comparaison sert à mettre en
209      cache les résultats de comparaisons entre DNs.</p>
210
211      <p>Notez que, lorsque l'appartenance à un groupe est vérifiée,
212      toute comparaison de sous-groupes est mise en cache afin
213      d'accélérer les comparaisons de sous-groupes ultérieures.</p>
214
215      <p>Le comportement de ces deux caches est contrôlé par les
216      directives <code class="directive"><a href="#ldapopcacheentries">LDAPOpCacheEntries</a></code> et <code class="directive"><a href="#ldapopcachettl">LDAPOpCacheTTL</a></code>.</p>
217    
218
219    <h3><a name="monitoring" id="monitoring">Superviser le cache</a></h3>
220      <p><code class="module"><a href="../mod/mod_ldap.html">mod_ldap</a></code> possède un gestionnaire de contenu
221      qui permet aux administrateurs de superviser les performances du
222      cache. Le nom du gestionnaire de contenu est
223      <code>ldap-status</code>, et on peut utiliser les directives
224      suivantes pour accéder aux informations du cache de
225      <code class="module"><a href="../mod/mod_ldap.html">mod_ldap</a></code> :</p>
226
227      <pre class="prettyprint lang-config">&lt;Location "/server/cache-info"&gt;
228    SetHandler ldap-status
229&lt;/Location&gt;</pre>
230
231
232      <p>En se connectant à l'URL
233      <code>http://nom-serveur/infos-cache</code>, l'administrateur peut
234      obtenir un rapport sur le statut de chaque cache qu'utilise
235      <code class="module"><a href="../mod/mod_ldap.html">mod_ldap</a></code>. Notez que si Apache ne supporte pas la
236      mémoire partagée, chaque instance de <code class="program"><a href="../programs/httpd.html">httpd</a></code>
237      possèdera son propre cache, et chaque fois que l'URL sera
238      rechargée, un résultat différent pourra être affiché, en fonction
239      de l'instance de <code class="program"><a href="../programs/httpd.html">httpd</a></code> qui traitera la
240      requête.</p>

299<h2><a name="settingcerts" id="settingcerts">Certificats SSL/TLS</a></h2>
300
301    <p>Les différents SDKs LDAP disposent de nombreuses méthodes pour
302    définir et gérer les certificats des clients et des autorités de
303    certification (CA).</p>
304
305    <p>Si vous avez l'intention d'utiliser SSL ou TLS, lisez cette
306    section ATTENTIVEMENT de façon à bien comprendre les différences de
307    configurations entre les différents SDKs LDAP supportés.</p>
308
309    <h3><a name="settingcerts-netscape" id="settingcerts-netscape">SDK Netscape/Mozilla/iPlanet</a></h3>
310        <p>Les certificat de CA sont enregistrés dans un fichier nommé
311	cert7.db. Le SDK ne dialoguera avec aucun serveur LDAP dont le
312	certificat n'a pas été signé par une CA spécifiée dans ce
313	fichier. Si des certificats clients sont requis, un fichier
314	key3.db ainsi qu'un mot de passe optionnels peuvent être
315	spécifiés. On peut aussi spécifier le fichier secmod si
316	nécessaire. Ces fichiers sont du même format que celui utilisé
317	par les navigateurs web Netscape Communicator ou Mozilla. Le
318	moyen le plus simple pour obtenir ces fichiers consiste à les
319	extraire de l'installation de votre navigateur.</p>
320
321	<p>Les certificats clients sont spécifiés pour chaque connexion en
322	utilisant la directive <code class="directive"><a href="#ldaptrustedclientcert">LDAPTrustedClientCert</a></code> et en se référant au
323	certificat "nickname". On peut éventuellement spécifier un mot de passe
324	pour déverrouiller la clé privée du certificat.</p>
325
326        <p>Le SDK supporte seulement SSL. Toute tentative d'utilisation
327	de STARTTLS engendrera une erreur lors des tentatives de
328	contacter le serveur LDAP pendant l'exécution.</p>
329
330        <pre class="prettyprint lang-config"># Spécifie un fichier de certificats de CA Netscape
331LDAPTrustedGlobalCert CA_CERT7_DB "/certs/cert7.db"
332# Spécifie un fichier key3db optionnel pour le support des
333# certificats clients
334LDAPTrustedGlobalCert CERT_KEY3_DB "/certs/key3.db"
335# Spécifie le fichier secmod si nécessaire
336LDAPTrustedGlobalCert CA_SECMOD "/certs/secmod"
337&lt;Location "/ldap-status"&gt;
338    SetHandler ldap-status
339
340    Require host yourdomain.example.com
341
342    Satisfy any
343    AuthType Basic
344    AuthName "LDAP Protected"
345    AuthBasicProvider ldap
346    LDAPTrustedClientCert CERT_NICKNAME &lt;nickname&gt; [password]
347    AuthLDAPURL "ldaps://127.0.0.1/dc=example,dc=com?uid?one"
348    Require valid-user
349&lt;/Location&gt;</pre>
350

354    <h3><a name="settingcerts-novell" id="settingcerts-novell">SDK Novell</a></h3>
355
356        <p>Un ou plusieurs certificats de CA doivent être spécifiés pour
357	que le SDK Novell fonctionne correctement. Ces certificats
358	peuvent être spécifiés sous forme de fichiers au format binaire
359	DER ou codés en Base64 (PEM).</p>
360
361	<p>Note: Les certificats clients sont spécifiés globalement plutôt qu'à
362	chaque connexion, et doivent être spécifiés à l'aide de la directive
363	<code class="directive"><a href="#ldaptrustedglobalcert">LDAPTrustedGlobalCert</a></code> comme
364	ci-dessous. Définir des certificats clients via la directive <code class="directive"><a href="#ldaptrustedclientcert">LDAPTrustedClientCert</a></code> engendrera une
365	erreur qui sera journalisée, au moment de la tentative de connexion avec
366	le serveur LDAP.</p>
367
368	<p>Le SDK supporte SSL et STARTTLS, le choix étant défini par le
369	paramètre de la directive <code class="directive"><a href="#ldaptrustedmode">LDAPTrustedMode</a></code>. Si une URL de type
370	ldaps:// est spécifiée, le mode SSL est forcé, et l'emporte sur cette
371	directive.</p>
372
373        <pre class="prettyprint lang-config"># Spécifie deux fichiers contenant des certificats de CA
374LDAPTrustedGlobalCert CA_DER "/certs/cacert1.der"
375LDAPTrustedGlobalCert CA_BASE64 "/certs/cacert2.pem"
376# Spécifie un fichier contenant des certificats clients
377# ainsi qu'une clé
378LDAPTrustedGlobalCert CERT_BASE64 "/certs/cert1.pem"

726<table class="directive">
727<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Définit le temps d'attente avant un autre essai de connexion au
728serveur LDAP.</td></tr>
729<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>LDAPRetryDelay <var>secondes</var></code></td></tr>
730<tr><th><a href="directive-dict.html#Default">Défaut:</a></th><td><code>LDAPRetryDelay 0</code></td></tr>
731<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>configuration globale</td></tr>

735    <p>Si la directive <code class="directive">LDAPRetryDelay</code> est définie

782recherche et d'identification LDAP en secondes</td></tr>
783<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>LDAPTimeout <var>secondes</var></code></td></tr>
784<tr><th><a href="directive-dict.html#Default">Défaut:</a></th><td><code>LDAPTimeout 60</code></td></tr>
785<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>configuration globale</td></tr>

790</table>
791    <p>Cette directive permet de spécifier le délai d'attente pour les
792    opérations de recherche et d'identification, ainsi que l'option
793    LDAP_OPT_TIMEOUT dans la bibliothèque LDAP client sous-jacente,

799    pare-feu est configuré pour envoyer des paquets TCP RST au lieu de
800    rejeter silencieusement les paquets.</p>
801
802    <div class="note">
803    <p>Les délais pour les opérations de comparaison LDAP nécessitent un
804    SDK avec LDAP_OPT_TIMEOUT, comme OpenLDAP &gt;= 2.4.4.</p>

892<table class="directive">
893<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Spécifie le mode (SSL ou TLS) à utiliser lors de la
894connexion à un serveur LDAP.</td></tr>
895<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>LDAPTrustedMode <var>type</var></code></td></tr>
896<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>configuration globale, serveur virtuel</td></tr>
897<tr><th><a href="directive-dict.html#Status">Statut:</a></th><td>Extension</td></tr>
898<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_ldap</td></tr>
899</table>
900    <p>Les modes suivants sont supportés :</p>
901    <ul>
902      <li>NONE - aucun chiffrement</li>
903      <li>SSL - chiffrement ldaps:// sur le port par défaut 636</li>
904      <li>TLS - chiffrement STARTTLS sur le port par défaut 389</li>
905    </ul>
906
907    <p>Les modes ci-dessus ne sont pas supportés par tous les SDK LDAP.
908    Un message d'erreur sera généré à l'exécution si un mode n'est pas
909    supporté, et la connexion au serveur LDAP échouera.
910    </p>
911
912    <p>Si une URL de type ldaps:// est spécifiée, le mode est forcé à SSL et la
913    définition de <code class="directive">LDAPTrustedMode</code> est ignorée.</p>
