38<h3>Summary</h3>
39
40    <p>This module allows authentication front-ends such as
41    <code class="module"><a href="../mod/mod_auth_basic.html">mod_auth_basic</a></code> to authenticate users through
42    an ldap directory.</p>
43
44    <p><code class="module"><a href="../mod/mod_authnz_ldap.html">mod_authnz_ldap</a></code> supports the following features:</p>
45
46    <ul>
47      <li>Known to support the <a href="http://www.openldap.org/">OpenLDAP SDK</a> (both 1.x
48      and 2.x).</li>
49
50      <li>Complex authorization policies can be implemented by
51      representing the policy with LDAP filters.</li>
52
53      <li>Uses extensive caching of LDAP operations via <a href="mod_ldap.html">mod_ldap</a>.</li>
54
55      <li>Support for LDAP over SSL/TLS (requires the OpenLDAP 2.x SDK).</li>
56    </ul>
57
58    <p>When using <code class="module"><a href="../mod/mod_auth_basic.html">mod_auth_basic</a></code>, this module is invoked
59    via the <code class="directive"><a href="../mod/mod_auth_basic.html#authbasicprovider">AuthBasicProvider</a></code>
60    directive with the <code>ldap</code> value.</p>

151<h2><a name="gcaveats" id="gcaveats">General caveats</a></h2>
152<p> This module caches authentication and authorization results based
153on the configuration of <code class="module"><a href="../mod/mod_ldap.html">mod_ldap</a></code>. Changes
154made to the backing LDAP server will not be immediately reflected on the
155HTTP Server, including but not limited to user lockouts/revocations, 
156password changes, or changes to group memberships.  Consult the directives 
157in <code class="module"><a href="../mod/mod_ldap.html">mod_ldap</a></code> for details of the cache tunables.

161<h2><a name="operation" id="operation">Operation</a></h2>
162
163    <p>There are two phases in granting access to a user. The first
164    phase is authentication, in which the <code class="module"><a href="../mod/mod_authnz_ldap.html">mod_authnz_ldap</a></code>
165    authentication provider verifies that the user's credentials are valid.
166    This is also called the <em>search/bind</em> phase. The second phase is
167    authorization, in which <code class="module"><a href="../mod/mod_authnz_ldap.html">mod_authnz_ldap</a></code> determines
168    if the authenticated user is allowed access to the resource in
169    question. This is also known as the <em>compare</em>
170    phase.</p>
171
172    <p><code class="module"><a href="../mod/mod_authnz_ldap.html">mod_authnz_ldap</a></code> registers both an authn_ldap authentication
173    provider and an authz_ldap authorization handler.  The authn_ldap
174    authentication provider can be enabled through the
175    <code class="directive"><a href="../mod/mod_auth_basic.html#authbasicprovider">AuthBasicProvider</a></code> directive
176    using the <code>ldap</code> value. The authz_ldap handler extends the
177    <code class="directive"><a href="../mod/mod_authz_core.html#require">Require</a></code> directive's authorization types
178    by adding <code>ldap-user</code>, <code>ldap-dn</code> and <code>ldap-group</code>
179    values.</p>
180
181<h3><a name="authenphase" id="authenphase">The Authentication
182    Phase</a></h3>
183
184    <p>During the authentication phase, <code class="module"><a href="../mod/mod_authnz_ldap.html">mod_authnz_ldap</a></code>
185    searches for an entry in the directory that matches the username
186    that the HTTP client passes. If a single unique match is found,
187    then <code class="module"><a href="../mod/mod_authnz_ldap.html">mod_authnz_ldap</a></code> attempts to bind to the
188    directory server using the DN of the entry plus the password
189    provided by the HTTP client. Because it does a search, then a
190    bind, it is often referred to as the search/bind phase. Here are
191    the steps taken during the search/bind phase.</p>
192
193    <ol>
194      <li>Generate a search filter by combining the attribute and
195      filter provided in the <code class="directive"><a href="#authldapurl">AuthLDAPURL</a></code> directive with
196      the username passed by the HTTP client.</li>
197
198      <li>Search the directory using the generated filter. If the
199      search does not return exactly one entry, deny or decline
200      access.</li>
201
202      <li>Fetch the distinguished name of the entry retrieved from
203      the search and attempt to bind to the LDAP server using that
204      DN and the password passed by the HTTP client. If the bind is
205      unsuccessful, deny or decline access.</li>
206    </ol>
207
208    <p>The following directives are used during the search/bind
209    phase</p>
210
211    <table>
212      
213      <tr>
214        <td><code class="directive"><a href="#authldapurl">AuthLDAPURL</a></code></td>
215
216        <td>Specifies the LDAP server, the
217        base DN, the attribute to use in the search, as well as the
218        extra search filter to use.</td>
219      </tr>
220
221      <tr>
222        <td><code class="directive"><a href="#authldapbinddn">AuthLDAPBindDN</a></code></td>
223
224        <td>An optional DN to bind with
225        during the search phase.</td>
226      </tr>
227
228      <tr>
229        <td><code class="directive"><a href="#authldapbindpassword">AuthLDAPBindPassword</a></code></td>
230
231        <td>An optional password to bind
232        with during the search phase.</td>
233      </tr>
234    </table>

237<h3><a name="authorphase" id="authorphase">The Authorization Phase</a></h3>
238
239    <p>During the authorization phase, <code class="module"><a href="../mod/mod_authnz_ldap.html">mod_authnz_ldap</a></code>
240    attempts to determine if the user is authorized to access the
241    resource.  Many of these checks require
242    <code class="module"><a href="../mod/mod_authnz_ldap.html">mod_authnz_ldap</a></code> to do a compare operation on the
243    LDAP server. This is why this phase is often referred to as the
244    compare phase. <code class="module"><a href="../mod/mod_authnz_ldap.html">mod_authnz_ldap</a></code> accepts the
245    following <code class="directive"><a href="../mod/mod_authz_core.html#require">Require</a></code>
246    directives to determine if the credentials are acceptable:</p>
247
248    <ul>
249      <li>Grant access if there is a <a href="#reqgroup"><code>Require ldap-user</code></a> directive, and the
250      username in the directive matches the username passed by the
251      client.</li>
252
253      <li>Grant access if there is a <a href="#reqdn"><code>Require
254      ldap-dn</code></a> directive, and the DN in the directive matches
255      the DN fetched from the LDAP directory.</li>
256
257      <li>Grant access if there is a <a href="#reqgroup"><code>Require ldap-group</code></a> directive, and
258      the DN fetched from the LDAP directory (or the username
259      passed by the client) occurs in the LDAP group or, potentially, in
260      one of its sub-groups.</li>
261
262      <li>Grant access if there is a <a href="#reqattribute">
263      <code>Require ldap-attribute</code></a>
264      directive, and the attribute fetched from the LDAP directory
265      matches the given value.</li>
266
267      <li>Grant access if there is a <a href="#reqfilter">
268      <code>Require ldap-filter</code></a>
269      directive, and the search filter successfully finds a single user
270      object that matches the dn of the authenticated user.</li>
271
272      <li>Grant access if there is a <a href="#reqsearch">
273      <code>Require ldap-search</code></a>
274      directive, and the search filter successfully returns a single
275      matching object with any distinguished name.</li>
276
277      <li>otherwise, deny or decline access</li>
278    </ul>
279
280    <p>Other <code class="directive"><a href="../mod/mod_authz_core.html#require">Require</a></code> values may also
281    be used which may require loading additional authorization modules.</p>
282
283    <ul>
284        <li>Grant access to all successfully authenticated users if
285        there is a <a href="#requser"><code>Require valid-user</code></a>
286        directive. (requires <code class="module"><a href="../mod/mod_authz_user.html">mod_authz_user</a></code>)</li>
287
288        <li>Grant access if there is a <a href="#reqgroup"><code>Require group</code></a> directive, and
289        <code class="module"><a href="../mod/mod_authz_groupfile.html">mod_authz_groupfile</a></code> has been loaded with the
290        <code class="directive"><a href="../mod/mod_authz_groupfile.html#authgroupfile">AuthGroupFile</a></code>
291        directive set.</li>
292
293        <li>others...</li>
294     </ul>

359<h2><a name="requiredirectives" id="requiredirectives">The Require Directives</a></h2>
360
361    <p>Apache's <code class="directive"><a href="../mod/mod_authz_core.html#require">Require</a></code>
362    directives are used during the authorization phase to ensure that
363    a user is allowed to access a resource.  mod_authnz_ldap extends the
364    authorization types with <code>ldap-user</code>, <code>ldap-dn</code>,
365    <code>ldap-group</code>, <code>ldap-attribute</code> and
366    <code>ldap-filter</code>.  Other authorization types may also be
367    used but may require that additional authorization modules be loaded.</p>
368
369    <p>Since v2.4.8, <a href="../expr.html">expressions</a> are supported
370    within the LDAP require directives.</p>
371
372<h3><a name="requser" id="requser">Require ldap-user</a></h3>
373
374    <p>The <code>Require ldap-user</code> directive specifies what
375    usernames can access the resource. Once
376    <code class="module"><a href="../mod/mod_authnz_ldap.html">mod_authnz_ldap</a></code> has retrieved a unique DN from the
377    directory, it does an LDAP compare operation using the username
378    specified in the <code>Require ldap-user</code> to see if that username
379    is part of the just-fetched LDAP entry.  Multiple users can be
380    granted access by putting multiple usernames on the line,
381    separated with spaces. If a username has a space in it, then it
382    must be surrounded with double quotes. Multiple users can also be
383    granted access by using multiple <code>Require ldap-user</code>
384    directives, with one user per line. For example, with a <code class="directive"><a href="#authldapurl">AuthLDAPURL</a></code> of
385    <code>ldap://ldap/o=Example?cn</code> (i.e., <code>cn</code> is
386    used for searches), the following Require directives could be used
387    to restrict access:</p>
388<pre class="prettyprint lang-config">Require ldap-user "Barbara Jenson"
389Require ldap-user "Fred User"
390Require ldap-user "Joe Manager"</pre>
391
392
393    <p>Because of the way that <code class="module"><a href="../mod/mod_authnz_ldap.html">mod_authnz_ldap</a></code> handles this
394    directive, Barbara Jenson could sign on as <code>Barbara
395    Jenson</code>, <code>Babs Jenson</code> or any other <code>cn</code> that
396    she has in her LDAP entry. Only the single <code>Require
397    ldap-user</code> line is needed to support all values of the attribute
398    in the user's entry.</p>
399
400    <p>If the <code>uid</code> attribute was used instead of the
401    <code>cn</code> attribute in the URL above, the above three lines
402    could be condensed to</p>
403<pre class="prettyprint lang-config">Require ldap-user bjenson fuser jmanager</pre>
404
405
406
407<h3><a name="reqgroup" id="reqgroup">Require ldap-group</a></h3>
408
409    <p>This directive specifies an LDAP group whose members are
410    allowed access. It takes the distinguished name of the LDAP
411    group. Note: Do not surround the group name with quotes.
412    For example, assume that the following entry existed in
413    the LDAP directory:</p>
414<div class="example"><pre>dn: cn=Administrators, o=Example
415objectClass: groupOfUniqueNames
416uniqueMember: cn=Barbara Jenson, o=Example
417uniqueMember: cn=Fred User, o=Example</pre></div>
418
419    <p>The following directive would grant access to both Fred and

467<h3><a name="reqdn" id="reqdn">Require ldap-dn</a></h3>
468
469    <p>The <code>Require ldap-dn</code> directive allows the administrator
470    to grant access based on distinguished names. It specifies a DN
471    that must match for access to be granted. If the distinguished
472    name that was retrieved from the directory server matches the
473    distinguished name in the <code>Require ldap-dn</code>, then
474    authorization is granted. Note: do not surround the distinguished
475    name with quotes.</p>
476
477    <p>The following directive would grant access to a specific
478    DN:</p>
479<pre class="prettyprint lang-config">Require ldap-dn cn=Barbara Jenson, o=Example</pre>
480
481
482    <p>Behavior of this directive is modified by the <code class="directive"><a href="#authldapcomparednonserver">AuthLDAPCompareDNOnServer</a></code>
483    directive.</p>
484

486<h3><a name="reqattribute" id="reqattribute">Require ldap-attribute</a></h3>
487
488    <p>The <code>Require ldap-attribute</code> directive allows the
489    administrator to grant access based on attributes of the authenticated
490    user in the LDAP directory.  If the attribute in the directory
491    matches the value given in the configuration, access is granted.</p>
492
493    <p>The following directive would grant access to anyone with
494    the attribute employeeType = active</p>
495
496    <pre class="prettyprint lang-config">Require ldap-attribute employeeType="active"</pre>
497
498
499    <p>Multiple attribute/value pairs can be specified on the same line
500    separated by spaces or they can be specified in multiple
501    <code>Require ldap-attribute</code> directives. The effect of listing
502    multiple attribute/values pairs is an OR operation. Access will be
503    granted if any of the listed attribute values match the value of the
504    corresponding attribute in the user object. If the value of the
505    attribute contains a space, only the value must be within double quotes.</p>
506
507    <p>The following directive would grant access to anyone with
508    the city attribute equal to "San Jose" or status equal to "Active"</p>
509
510    <pre class="prettyprint lang-config">Require ldap-attribute city="San Jose" status="active"</pre>
511
512

515<h3><a name="reqfilter" id="reqfilter">Require ldap-filter</a></h3>
516
517    <p>The <code>Require ldap-filter</code> directive allows the
518    administrator to grant access based on a complex LDAP search filter.
519    If the dn returned by the filter search matches the authenticated user
520    dn, access is granted.</p>
521
522    <p>The following directive would grant access to anyone having a cell phone
523    and is in the marketing department</p>
524
525    <pre class="prettyprint lang-config">Require ldap-filter "&amp;(cell=*)(department=marketing)"</pre>
526
527
528    <p>The difference between the <code>Require ldap-filter</code> directive and the
529    <code>Require ldap-attribute</code> directive is that <code>ldap-filter</code>
530    performs a search operation on the LDAP directory using the specified search
531    filter rather than a simple attribute comparison. If a simple attribute
532    comparison is all that is required, the comparison operation performed by
533    <code>ldap-attribute</code> will be faster than the search operation
534    used by <code>ldap-filter</code> especially within a large directory.</p>
535
536    <p>When using an <a href="../expr.html">expression</a> within the filter, care
537    must be taken to ensure that LDAP filters are escaped correctly to guard against
538    LDAP injection. The ldap function can be used for this purpose.</p>
539
540<pre class="prettyprint lang-config">&lt;LocationMatch ^/dav/(?&lt;SITENAME&gt;[^/]+)/&gt;
541  Require ldap-filter (memberOf=cn=%{ldap:%{unescape:%{env:MATCH_SITENAME}},ou=Websites,o=Example)
542&lt;/LocationMatch&gt;</pre>
543
544
545
546
547<h3><a name="reqsearch" id="reqsearch">Require ldap-search</a></h3>
548
549    <p>The <code>Require ldap-search</code> directive allows the
550    administrator to grant access based on a generic LDAP search filter using an
551    <a href="../expr.html">expression</a>. If there is exactly one match to the search filter,
552    regardless of the distinguished name, access is granted.</p>
553
554    <p>The following directive would grant access to URLs that match the given objects in the
555    LDAP server:</p>
556
557<pre class="prettyprint lang-config">&lt;LocationMatch ^/dav/(?&lt;SITENAME&gt;[^/]+)/&gt;
558Require ldap-search (cn=%{ldap:%{unescape:%{env:MATCH_SITENAME}} Website)
559&lt;/LocationMatch&gt;</pre>
560
561
562    <p>Note: care must be taken to ensure that any expressions are properly escaped to guard
563    against LDAP injection. The <strong>ldap</strong> function can be used as per the example
564    above.</p>

571
572    <ul>
573      <li>
574        Grant access to anyone who exists in the LDAP directory,
575        using their UID for searches.
576<pre class="prettyprint lang-config">AuthLDAPURL "ldap://ldap1.example.com:389/ou=People, o=Example?uid?sub?(objectClass=*)"
577Require valid-user</pre>
578
579      </li>
580
581      <li>
582        The next example is the same as above; but with the fields
583        that have useful defaults omitted. Also, note the use of a
584        redundant LDAP server.
585<pre class="prettyprint lang-config">AuthLDAPURL "ldap://ldap1.example.com ldap2.example.com/ou=People, o=Example"
586Require valid-user</pre>
587
588      </li>
589
590      <li>
591        The next example is similar to the previous one, but it
592        uses the common name instead of the UID. Note that this
593        could be problematical if multiple people in the directory
594        share the same <code>cn</code>, because a search on <code>cn</code>
595        <strong>must</strong> return exactly one entry. That's why
596        this approach is not recommended: it's a better idea to
597        choose an attribute that is guaranteed unique in your
598        directory, such as <code>uid</code>.
599<pre class="prettyprint lang-config">AuthLDAPURL "ldap://ldap.example.com/ou=People, o=Example?cn"
600Require valid-user</pre>
601
602      </li>
603
604      <li>
605        Grant access to anybody in the Administrators group. The
606        users must authenticate using their UID.
607<pre class="prettyprint lang-config">AuthLDAPURL ldap://ldap.example.com/o=Example?uid
608Require ldap-group cn=Administrators, o=Example</pre>
609
610      </li>
611
612      <li>
613        Grant access to anybody in the group whose name matches the
614        hostname of the virtual host. In this example an
615        <a href="../expr.html">expression</a> is used to build the filter.
616<pre class="prettyprint lang-config">AuthLDAPURL ldap://ldap.example.com/o=Example?uid
617Require ldap-group cn=%{SERVER_NAME}, o=Example</pre>
618
619      </li>
620
621      <li>
622        The next example assumes that everyone at Example who
623        carries an alphanumeric pager will have an LDAP attribute
624        of <code>qpagePagerID</code>. The example will grant access
625        only to people (authenticated via their UID) who have
626        alphanumeric pagers:
627<pre class="prettyprint lang-config">AuthLDAPURL ldap://ldap.example.com/o=Example?uid??(qpagePagerID=*)
628Require valid-user</pre>
629
630      </li>
631
632      <li>
633        <p>The next example demonstrates the power of using filters
634        to accomplish complicated administrative requirements.
635        Without filters, it would have been necessary to create a
636        new LDAP group and ensure that the group's members remain
637        synchronized with the pager users. This becomes trivial
638        with filters. The goal is to grant access to anyone who has
639        a pager, plus grant access to Joe Manager, who doesn't
640        have a pager, but does need to access the same
641        resource:</p>
642<pre class="prettyprint lang-config">AuthLDAPURL ldap://ldap.example.com/o=Example?uid??(|(qpagePagerID=*)(uid=jmanager))
643Require valid-user</pre>
644
645
646        <p>This last may look confusing at first, so it helps to
647        evaluate what the search filter will look like based on who
648        connects, as shown below.  If
649        Fred User connects as <code>fuser</code>, the filter would look
650        like</p>
651
652        <div class="example"><p><code>(&amp;(|(qpagePagerID=*)(uid=jmanager))(uid=fuser))</code></p></div>
653
654        <p>The above search will only succeed if <code>fuser</code> has a
655        pager. When Joe Manager connects as <code>jmanager</code>, the
656        filter looks like</p>
657
658        <div class="example"><p><code>(&amp;(|(qpagePagerID=*)(uid=jmanager))(uid=jmanager))</code></p></div>
659
660        <p>The above search will succeed whether <code>jmanager</code>
661        has a pager or not.</p>
662      </li>
663    </ul>

706<h2><a name="activedirectory" id="activedirectory">Using Active Directory</a></h2>
707
708    <p>An Active Directory installation may support multiple domains at the
709    same time. To distinguish users between domains, an identifier called
710    a User Principle Name (UPN) can be added to a user's entry in the
711    directory. This UPN usually takes the form of the user's account
712    name, followed by the domain components of the particular domain,
713    for example <code>somebody@nz.example.com</code>.</p>
714
715    <p>You may wish to configure the <code class="module"><a href="../mod/mod_authnz_ldap.html">mod_authnz_ldap</a></code>
716    module to authenticate users present in any of the domains making up
717    the Active Directory forest. In this way both
718    <code>somebody@nz.example.com</code> and <code>someone@au.example.com</code>
719    can be authenticated using the same query at the same time.</p>
720
721    <p>To make this practical, Active Directory supports the concept of
722    a Global Catalog. This Global Catalog is a read only copy of selected
723    attributes of all the Active Directory servers within the Active
724    Directory forest. Querying the Global Catalog allows all the domains
725    to be queried in a single query, without the query spanning servers
726    over potentially slow links.</p>
727
728    <p>If enabled, the Global Catalog is an independent directory server
729    that runs on port 3268 (3269 for SSL). To search for a user, do a
730    subtree search for the attribute <code>userPrincipalName</code>, with
731    an empty search root, like so:</p>
732
733<pre class="prettyprint lang-config">AuthLDAPBindDN apache@example.com
734AuthLDAPBindPassword password
735AuthLDAPURL ldap://10.0.0.1:3268/?userPrincipalName?sub</pre>
736
737
738    <p>Users will need to enter their User Principal Name as a login, in
739    the form <code>somebody@nz.example.com</code>.</p>

743<h2><a name="frontpage" id="frontpage">Using Microsoft
744    FrontPage with mod_authnz_ldap</a></h2>
745
746    <p>Normally, FrontPage uses FrontPage-web-specific user/group
747    files (i.e., the <code class="module"><a href="../mod/mod_authn_file.html">mod_authn_file</a></code> and
748    <code class="module"><a href="../mod/mod_authz_groupfile.html">mod_authz_groupfile</a></code> modules) to handle all
749    authentication. Unfortunately, it is not possible to just
750    change to LDAP authentication by adding the proper directives,
751    because it will break the <em>Permissions</em> forms in
752    the FrontPage client, which attempt to modify the standard
753    text-based authorization files.</p>
754
755    <p>Once a FrontPage web has been created, adding LDAP
756    authentication to it is a matter of adding the following
757    directives to <em>every</em> <code>.htaccess</code> file
758    that gets created in the web</p>
759<pre class="prettyprint lang-config">AuthLDAPURL       "the url"
760AuthGroupFile     "mygroupfile"
761Require group     "mygroupfile"</pre>
762
763
764<h3><a name="howitworks" id="howitworks">How It Works</a></h3>
765
766    <p>FrontPage restricts access to a web by adding the <code>Require
767    valid-user</code> directive to the <code>.htaccess</code>
768    files. The <code>Require valid-user</code> directive will succeed for
769    any user who is valid <em>as far as LDAP is
770    concerned</em>. This means that anybody who has an entry in
771    the LDAP directory is considered a valid user, whereas FrontPage
772    considers only those people in the local user file to be
773    valid. By substituting the ldap-group with group file authorization,
774    Apache is allowed to consult the local user file (which is managed by
775    FrontPage) - instead of LDAP - when handling authorizing the user.</p>
776
777    <p>Once directives have been added as specified above,
778    FrontPage users will be able to perform all management
779    operations from the FrontPage client.</p>
780
781
782<h3><a name="fpcaveats" id="fpcaveats">Caveats</a></h3>
783
784    <ul>
785      <li>When choosing the LDAP URL, the attribute to use for
786      authentication should be something that will also be valid
787      for putting into a <code class="module"><a href="../mod/mod_authn_file.html">mod_authn_file</a></code> user file.
788      The user ID is ideal for this.</li>
789
790      <li>When adding users via FrontPage, FrontPage administrators
791      should choose usernames that already exist in the LDAP
792      directory (for obvious reasons). Also, the password that the
793      administrator enters into the form is ignored, since Apache
794      will actually be authenticating against the password in the
795      LDAP database, and not against the password in the local user
796      file. This could cause confusion for web administrators.</li>
797
798      
799      <li>Apache must be compiled with <code class="module"><a href="../mod/mod_auth_basic.html">mod_auth_basic</a></code>,
800      <code class="module"><a href="../mod/mod_authn_file.html">mod_authn_file</a></code> and
801      <code class="module"><a href="../mod/mod_authz_groupfile.html">mod_authz_groupfile</a></code> in order to
802      use FrontPage support. This is because Apache will still use
803      the <code class="module"><a href="../mod/mod_authz_groupfile.html">mod_authz_groupfile</a></code> group file for determine
804      the extent of a user's access to the FrontPage web.</li>
805
806      <li>The directives must be put in the <code>.htaccess</code>
807      files. Attempting to put them inside <code class="directive"><a href="../mod/core.html#location">&lt;Location&gt;</a></code> or <code class="directive"><a href="../mod/core.html#directory">&lt;Directory&gt;</a></code> directives won't work. This
808      is because <code class="module"><a href="../mod/mod_authnz_ldap.html">mod_authnz_ldap</a></code> has to be able to grab
809      the <code class="directive"><a href="../mod/mod_authz_groupfile.html#authgroupfile">AuthGroupFile</a></code>
810      directive that is found in FrontPage <code>.htaccess</code>
811      files so that it knows where to look for the valid user list. If
812      the <code class="module"><a href="../mod/mod_authnz_ldap.html">mod_authnz_ldap</a></code> directives aren't in the same
813      <code>.htaccess</code> file as the FrontPage directives, then
814      the hack won't work, because <code class="module"><a href="../mod/mod_authnz_ldap.html">mod_authnz_ldap</a></code> will
815      never get a chance to process the <code>.htaccess</code> file,
816      and won't be able to find the FrontPage-managed user file.</li>
817    </ul>

1310<div class="directive-section"><h2><a name="AuthLDAPURL" id="AuthLDAPURL">AuthLDAPURL</a> <a name="authldapurl" id="authldapurl">Directive</a></h2>
1311<table class="directive">
1312<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>URL specifying the LDAP search parameters</td></tr>
1313<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>AuthLDAPURL <var>url</var> [NONE|SSL|TLS|STARTTLS]</code></td></tr>
1314<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>directory, .htaccess</td></tr>
1315<tr><th><a href="directive-dict.html#Override">Override:</a></th><td>AuthConfig</td></tr>
1316<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Extension</td></tr>
1317<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_authnz_ldap</td></tr>
1318</table>
1319    <p>An RFC 2255 URL which specifies the LDAP search parameters
1320    to use. The syntax of the URL is</p>
1321<div class="example"><p><code>ldap://host:port/basedn?attribute?scope?filter</code></p></div>
1322    <p>If you want to specify more than one LDAP URL that Apache should try in turn, the syntax is:</p>
1323<pre class="prettyprint lang-config">AuthLDAPURL "ldap://ldap1.example.com ldap2.example.com/dc=..."</pre>
1324
1325<div class="warning"><h3>Caveat</h3>
1326<p>If you specify multiple servers, you need to enclose the entire URL string in quotes;
1327otherwise you will get an error: "AuthLDAPURL takes one argument, URL to define LDAP connection.."
1328You can of course use search parameters on each of these.</p>
1329</div>
1330
1331<dl>
1332<dt>ldap</dt>
1333
1334        <dd>For regular ldap, use the
1335        string <code>ldap</code>. For secure LDAP, use <code>ldaps</code>
1336        instead. Secure LDAP is only available if Apache was linked
1337        to an LDAP library with SSL support.</dd>
1338
1339<dt>host:port</dt>
1340
1341        <dd>
1342          <p>The name/port of the ldap server (defaults to
1343          <code>localhost:389</code> for <code>ldap</code>, and
1344          <code>localhost:636</code> for <code>ldaps</code>). To
1345          specify multiple, redundant LDAP servers, just list all
1346          servers, separated by spaces. <code class="module"><a href="../mod/mod_authnz_ldap.html">mod_authnz_ldap</a></code>
1347          will try connecting to each server in turn, until it makes a
1348          successful connection. If multiple ldap servers are specified,
1349          then entire LDAP URL must be encapsulated in double quotes.</p>
1350
1351          <p>Once a connection has been made to a server, that
1352          connection remains active for the life of the
1353          <code class="program"><a href="../programs/httpd.html">httpd</a></code> process, or until the LDAP server goes
1354          down.</p>
1355
1356          <p>If the LDAP server goes down and breaks an existing
1357          connection, <code class="module"><a href="../mod/mod_authnz_ldap.html">mod_authnz_ldap</a></code> will attempt to
1358          re-connect, starting with the primary server, and trying
1359          each redundant server in turn. Note that this is different
1360          than a true round-robin search.</p>
1361        </dd>
1362
1363<dt>basedn</dt>
1364
1365        <dd>The DN of the branch of the
1366        directory where all searches should start from. At the very
1367        least, this must be the top of your directory tree, but
1368        could also specify a subtree in the directory.</dd>
1369
1370<dt>attribute</dt>
1371
1372        <dd>The attribute to search for.
1373        Although RFC 2255 allows a comma-separated list of
1374        attributes, only the first attribute will be used, no
1375        matter how many are provided. If no attributes are
1376        provided, the default is to use <code>uid</code>. It's a good
1377        idea to choose an attribute that will be unique across all
1378        entries in the subtree you will be using.  All attributes
1379        listed will be put into the environment with an AUTHENTICATE_ prefix
1380        for use by other modules.</dd>
1381
1382<dt>scope</dt>
1383
1384        <dd>The scope of the search. Can be either <code>one</code> or
1385        <code>sub</code>. Note that a scope of <code>base</code> is
1386        also supported by RFC 2255, but is not supported by this
1387        module. If the scope is not provided, or if <code>base</code> scope
1388        is specified, the default is to use a scope of
1389        <code>sub</code>.</dd>
1390
1391<dt>filter</dt>
1392
1393        <dd>A valid LDAP search filter. If
1394        not provided, defaults to <code>(objectClass=*)</code>, which
1395        will search for all objects in the tree. Filters are
1396        limited to approximately 8000 characters (the definition of
1397        <code>MAX_STRING_LEN</code> in the Apache source code). This
1398        should be more than sufficient for any application. In 2.4.10 and later,
1399        the keyword <code>none</code> disables the use of a filter; this is
1400        required by some primitive LDAP servers.</dd>
1401</dl>
1402
1403    <p>When doing searches, the attribute, filter and username passed
1404    by the HTTP client are combined to create a search filter that
1405    looks like
1406    <code>(&amp;(<var>filter</var>)(<var>attribute</var>=<var>username</var>))</code>.</p>
1407
1408    <p>For example, consider an URL of
1409    <code>ldap://ldap.example.com/o=Example?cn?sub?(posixid=*)</code>. When
1410    a client attempts to connect using a username of <code>Babs
1411    Jenson</code>, the resulting search filter will be
1412    <code>(&amp;(posixid=*)(cn=Babs Jenson))</code>.</p>
1413
1414    <p>An optional parameter can be added to allow the LDAP Url to override
1415    the connection type.  This parameter can be one of the following:</p>
1416
1417<dl>
1418    <dt>NONE</dt>
1419        <dd>Establish an unsecure connection on the default LDAP port. This
1420        is the same as <code>ldap://</code> on port 389.</dd>
1421    <dt>SSL</dt>
1422        <dd>Establish a secure connection on the default secure LDAP port.
1423        This is the same as <code>ldaps://</code></dd>
1424    <dt>TLS | STARTTLS</dt>
1425        <dd>Establish an upgraded secure connection on the default LDAP port.
1426        This connection will be initiated on port 389 by default and then
1427        upgraded to a secure connection on the same port.</dd>
1428</dl>
1429
1430    <p>See above for examples of <code class="directive"><a href="#authldapurl">AuthLDAPURL</a></code> URLs.</p>
