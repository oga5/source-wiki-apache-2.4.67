39<h3>Sommaire</h3>
40
41    <div class="warning"><h3>Avertissement</h3>
42      <p>Le module session fait usage des cookies HTTP, et peut à ce
43      titre être victime d'attaques de type Cross Site Scripting, ou
44      divulguer des informations à caractère privé aux clients. Veuillez
45      vous assurer que les risques ainsi encourus ont été pris en compte
46      avant d'activer le support des sessions sur votre serveur.</p>
47    </div>
48
49    <p>Ce module fournit le support d'une interface de session pour
50    chaque utilisateur au niveau du serveur global. Les sessions
51    permettent de transmettre diverses informations  : l'utilisateur
52    est-il connecté ou non, ou toute autre information qui doit être
53    conservée d'une requête à l'autre.</p>
54
55    <p>Les sessions peuvent être stockées sur le serveur, ou au niveau
56    du navigateur. Les sessions peuvent également être chiffrées pour une
57    sécurité accrue. Ces fonctionnalités sont réparties entre différents
58    modules complémentaires de <code class="module"><a href="../mod/mod_session.html">mod_session</a></code> :
59    <code class="module"><a href="../mod/mod_session_crypto.html">mod_session_crypto</a></code>,
60    <code class="module"><a href="../mod/mod_session_cookie.html">mod_session_cookie</a></code> et
61    <code class="module"><a href="../mod/mod_session_dbd.html">mod_session_dbd</a></code>. Chargez les modules appropriés
62    en fonction des besoins du serveur (soit statiquement à la
63    compilation, soit dynamiquement via la directive <code class="directive"><a href="../mod/mod_so.html#loadmodule">LoadModule</a></code>).</p>
64
65    <p>Les sessions peuvent être manipulées par d'autres modules qui
66    dépendent de la session, ou la session peut être lue et écrite dans
67    des variables d'environnement et des en-têtes HTTP, selon les
68    besoins.</p>

106<h2><a name="whatisasession" id="whatisasession">Qu'est-ce qu'une session ?</a></h2>
107      <p>Au coeur de l'interface de session se trouve une table de
108      paires clé/valeur qui sont accessibles d'une requête du navigateur
109      à l'autre. Les valeurs de clés peuvent se voir affecter toute chaîne
110      de caractères valide, en fonction des besoins de l'application qui
111      fait usage de la session.</p>
112
113      <p>Une "session" est une chaîne
114      <strong>application/x-www-form-urlencoded</strong> qui contient la
115      paire clé/valeur définie par la <a href="http://www.w3.org/TR/html4/">spécification HTML</a>.</p>
116
117      <p>Selon les souhaits de l'administrateur, la session peut être
118      chiffrée et codée en base64 avant d'être soumise au dispositif de
119      stockage.</p>
120
121    </div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>

123<h2><a name="whocanuseasession" id="whocanuseasession">Qui peut utiliser une session
124    ?</a></h2>
125      <p>L'interface de session a été conçue à l'origine pour être
126      utilisée par d'autres modules du serveur comme
127      <code class="module"><a href="../mod/mod_auth_form.html">mod_auth_form</a></code> ; les applications à base de
128      programmes CGI peuvent cependant se voir accorder l'accès au
129      contenu d'une session via la variable d'environnement
130      HTTP_SESSION. Il est possible de modifier et/ou de mettre à jour
131      une session en insérant un en-tête de réponse HTTP contenant les
132      nouveaux paramètres de session.</p>
133
134    </div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>
135<div class="section">
136<h2><a name="serversession" id="serversession">Stockage des sessions sur le
137    serveur</a></h2>
138      <p>Apache peut être configuré pour stocker les sessions
139      utilisateurs sur un serveur particulier ou un groupe de serveurs.
140      Cette fonctionnalité est similaire aux sessions disponibles sur
141      les serveurs d'applications courants.</p>
142
143      <p>Selon la configuration, les sessions sont suivies à
144      partir d'un identifiant de session stocké dans un cookie, ou
145      extraites de la chaîne de paramètres de l'URL, comme dans les
146      requêtes GET courantes.</p>
147
148      <p>Comme le contenu de la session est stocké exclusivement sur le
149      serveur, il est nécessaire de préserver la confidentialité de ce
150      contenu. Ceci a des implications en matière de performance et de
151      consommation de ressources lorsqu'un grand nombre de sessions est
152      stocké, ou lorsqu'un grand nombre de serveurs doivent se partager
153      les sessions entre eux.</p>
154
155      <p>Le module <code class="module"><a href="../mod/mod_session_dbd.html">mod_session_dbd</a></code> permet de stocker
156      les sessions utilisateurs dans une base de données SQL via le
157      module <code class="module"><a href="../mod/mod_dbd.html">mod_dbd</a></code>.</p>
158
159    </div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>
160<div class="section">
161<h2><a name="browsersession" id="browsersession">Stockage des sessions au niveau
162    du navigateur</a></h2>
163      <p>Dans les environnements à haut trafic où le stockage d'une
164      session sur un serveur consomme trop
165      de ressources, il est possible de stocker le contenu de la session
166      dans un cookie au niveau du navigateur client.</p>
167
168      <p>Ceci a pour avantage de ne nécessiter qu'une quantité minimale de
169      ressources sur le serveur pour suivre les sessions, et évite à
170      plusieurs serveurs parmi une forêt de serveurs de devoir partager
171      les informations de session.</p>
172
173      <p>Le contenu de la session est cependant présenté au client, avec
174      pour conséquence un risque de perte de confidentialité. Le module
175      <code class="module"><a href="../mod/mod_session_crypto.html">mod_session_crypto</a></code> peut être configuré pour
176      chiffrer le contenu de la session avant qu'elle soit stockée au
177      niveau du client.</p>
178
179      <p>Le module <code class="module"><a href="../mod/mod_session_cookie.html">mod_session_cookie</a></code> permet de stocker
180      les sessions au niveau du navigateur dans un cookie HTTP.</p>
181

184<h2><a name="basicexamples" id="basicexamples">Exemples simples</a></h2>
185
186      <p>La création d'une session consiste simplement à ouvrir la
187      session, et à décider de l'endroit où elle doit être stockée. Dans
188      l'exemple suivant, la session sera stockée au niveau du
189      navigateur, dans un cookie nommé <code>session</code>.</p>
190
191      <div class="example"><h3>Session stockée au niveau du navigateur</h3><pre class="prettyprint lang-config">Session On
192SessionCookieName session path=/</pre>
193</div>
194
195      <p>Une session est inutile s'il n'est pas possible d'y lire
196      ou d'y écrire. L'exemple suivant montre comment des valeurs
197      peuvent être injectées dans une session à l'aide d'un en-tête de
198      réponse HTTP prédéterminé nommé
199      <code>X-Replace-Session</code>.</p>
200
201      <div class="example"><h3>Ecriture dans une session</h3><pre class="prettyprint lang-config">Session On
202SessionCookieName session path=/
203SessionHeader X-Replace-Session</pre>
204</div>
205
206      <p>L'en-tête doit contenir des paires clé/valeur sous le même
207      format que celui de la chaîne d'argument d'une URL, comme dans
208      l'exemple suivant. Donner pour valeur à une clé la chaîne vide a
209      pour effet de supprimer la clé de la session.</p>
210
211      <div class="example"><h3>Script CGI pour écrire dans une session</h3><pre class="prettyprint lang-sh">#!/bin/bash
212echo "Content-Type: text/plain"
213echo "X-Replace-Session: key1=foo&amp;key2=&amp;key3=bar"
214echo
215env</pre>
216</div>
217
218      <p>Selon la configuration, les informations de la session peuvent
219      être extraites de la variable d'environnement HTTP_SESSION. Par
220      défaut la session est privée, et cette fonctionnalité doit donc
221      être explicitement activée via la directive <code class="directive"><a href="#sessionenv">SessionEnv</a></code>.</p>
222
223      <div class="example"><h3>Lecture depuis une session</h3><pre class="prettyprint lang-config">Session On
224SessionEnv On
225SessionCookieName session path=/
226SessionHeader X-Replace-Session</pre>
227</div>
228
229      <p>Une fois la lecture effectuée, la variable CGI
230      <code>HTTP_SESSION</code> doit contenir la valeur
231      <code>clé1=foo&amp;clé3=bar</code>.</p>
232

235<h2><a name="sessionprivacy" id="sessionprivacy">Confidentialité des
236    sessions</a></h2>
237
238      <p>En utilisant la fonctionnalité de votre navigateur "Afficher
239      les cookies", vous pouvez voir une réprésentation de la session
240      sous forme de texte en clair. Ceci peut poser problème si le
241      contenu de la session doit être dissimulé à l'utilisateur final,
242      ou si un tiers accède sans autorisation aux informations de
243      session.</p>
244
245      <p>À ce titre, le contenu de la session peut être chiffré à l'aide
246      du module <code class="module"><a href="../mod/mod_session_crypto.html">mod_session_crypto</a></code> avant d'être stocké
247      au niveau du navigateur.</p>
248
249      <div class="example"><h3>Session chiffrée avant stockage au niveau du
250      navigateur</h3><pre class="prettyprint lang-config">Session On
251SessionCryptoPassphrase secret
252SessionCookieName session path=/</pre>
253</div>
254
255      <p>La session sera automatiquement déchiffrée à la lecture, et
256      rechiffrée par Apache lors de la sauvegarde, si bien que
257      l'application sous-jacente qui utilise la session n'a pas à se
258      préoccuper de savoir si un chiffrement a été mis en oeuvre ou
259      non.</p>
260
261      <p>Les sessions stockées sur le serveur plutôt qu'au niveau du
262      navigateur peuvent aussi être chiffrées, préservant par là-même la
263      confidentialité lorsque des informations sensibles sont partagées
264      entre les serveurs web d'une forêt de serveurs à l'aide du module
265      <code class="module"><a href="../mod/mod_session_dbd.html">mod_session_dbd</a></code>.</p>
266
267    </div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>

269<h2><a name="cookieprivacy" id="cookieprivacy">Confidentialité du cookie</a></h2>
270
271      <p>Le mécanisme de cookie HTTP offre aussi des fonctionnalités
272      quant à la confidentialité, comme la possibilité de
273      restreindre le transport du cookie aux pages protégées par SSL
274      seulement, ou l'interdiction pour les scripts java qui
275      s'exécutent au niveau du navigateur d'obtenir l'accès au contenu
276      du cookie.</p>
277
278      <div class="warning"><h3>Avertissement</h3>
279      <p>Certaines fonctionnalités de confidentialité du cookie HTTP ne
280      sont pas standardisées, ou ne sont pas toujours implémentées au
281      niveau du navigateur. Les modules de session vous permettent de
282      définir les paramètres du cookie, mais il n'est pas garanti que la
283      confidentialité sera respectée par le navigateur. Si la sécurité
284      est la principale préoccupation, chiffrez le contenu de la session
285      avec le module <code class="module"><a href="../mod/mod_session_crypto.html">mod_session_crypto</a></code>, ou stockez la
286      session sur le serveur avec le module
287      <code class="module"><a href="../mod/mod_session_dbd.html">mod_session_dbd</a></code>.</p>
288      </div>
289
290      <p>Les paramètres standards du cookie peuvent être spécifiés après
291      le nom du cookie comme dans l'exemple suivant :</p>
292
293      <div class="example"><h3>Définition des paramètres du cookie</h3><pre class="prettyprint lang-config">Session On
294SessionCryptoPassphrase secret
295SessionCookieName session path=/private;domain=example.com;httponly;secure;</pre>
296</div>
297
298      <p>Dans les cas où le serveur Apache sert de frontal pour des
299      serveurs d'arrière-plan, il est possible de supprimer les cookies
300      de session des en-têtes HTTP entrants à l'aide de la directive
301      <code class="directive"><a href="../mod/mod_session_cookie.html#sessioncookieremove">SessionCookieRemove</a></code>. Ceci
302      permet d'empêcher les serveurs d'arrière-plan d'accéder au contenu
303      des cookies de session.
304      </p>

308<h2><a name="authentication" id="authentication">Support des sessions pour
309    l'authentification</a></h2>
310
311      <p>Comme il est possible de le faire avec de nombreux serveurs
312      d'applications, les modules d'authentification peuvent utiliser
313      une session pour stocker le nom d'utilisateur et le mot de passe
314      après connexion. Le module <code class="module"><a href="../mod/mod_auth_form.html">mod_auth_form</a></code> par
315      exemple, sauvegarde les nom de connexion et mot de passe de
316      l'utilisateur dans une session.</p>
317
318      <div class="example"><h3>Authentification à base de formulaire</h3><pre class="prettyprint lang-config">Session On
319SessionCryptoPassphrase secret
320SessionCookieName session path=/
321AuthFormProvider file
322AuthUserFile "conf/passwd"
323AuthType form
324AuthName "realm"
325#...</pre>
326</div>
327
328      <p>Pour la documentation et des exemples complets, voir le module
329      <code class="module"><a href="../mod/mod_auth_form.html">mod_auth_form</a></code>.</p>

332<div class="section">
333<h2><a name="integration" id="integration">Intégration des sessions avec les
334    applications externes</a></h2>
335
336      <p>Pour que les sessions soient utiles, leur contenu doit être
337      accessible aux applications externes, et ces dernières doivent
338      elles-mêmes être capables d'écrire une session.</p>
339
340      <p>L'exemple type est une application qui modifie le mot de passe
341      d'un utilisateur défini par <code class="module"><a href="../mod/mod_auth_form.html">mod_auth_form</a></code>. Cette
342      application doit pouvoir extraire les nom d'utilisateur et mot de
343      passe courants de la session, effectuer les modifications
344      demandées, puis écrire le nouveau mot de passe dans la session,
345      afin que la transition vers le nouveau mot de passe soit
346      transparente.</p>
347
348      <p>Un autre exemple met en jeu une application qui enregistre un
349      nouvel utilisateur pour la première fois. Une fois
350      l'enregistrement terminé, le nom d'utilisateur et le mot de passe
351      sont écrits dans la session, fournissant là aussi une transition
352      transparente.</p>
353
354      <dl>
355      <dt>Modules Apache</dt>
356      <dd>Selon les besoins, les modules du serveur peuvent utiliser
357      l'API <strong>mod_session.h</strong> pour lire et écrire dans les
358      sessions. Les modules tels que <code class="module"><a href="../mod/mod_auth_form.html">mod_auth_form</a></code>
359      utilisent ce mécanisme.
360      </dd>
361
362      <dt>Programmes CGI et langages de script</dt>
363      <dd>Les applications qui s'exécutent au sein du serveur web
364      peuvent éventuellement extraire la valeur de la session de la
365      variable d'environnement <strong>HTTP_SESSION</strong>. La session
366      doit être codée sous la forme d'une chaîne
367      <strong>application/x-www-form-urlencoded</strong> selon les
368      préconisations de la <a href="http://www.w3.org/TR/html4/">specification HTML</a>. Cette
369      variable d'environnement est définie via la directive <code class="directive"><a href="#sessionenv">SessionEnv</a></code>. Un script peut écrire
370      dans la session en renvoyant un en-tête de réponse
371      <strong>application/x-www-form-urlencoded</strong> dont le nom est
372      défini via la directive <code class="directive"><a href="#sessionheader">SessionHeader</a></code>. Dans les deux cas,
373      tout chiffrement ou déchiffrement, ainsi que la lecture ou
374      l'écriture de ou vers la session à partir du mécanisme de stockage
375      choisi sont gérés par le module <code class="module"><a href="../mod/mod_session.html">mod_session</a></code> et la
376      configuration correspondante.
377      </dd>
378
379      <dt>Applications situées derrière <code class="module"><a href="../mod/mod_proxy.html">mod_proxy</a></code></dt>
380      <dd>Si la directive <code class="directive"><a href="#sessionheader">SessionHeader</a></code> est utilisée pour
381      définir un en-tête de requête HTTP, la session codée sous la forme
382      d'une chaîne <strong>application/x-www-form-urlencoded</strong>
383      sera accessible pour l'application. Si ce même en-tête est fourni
384      dans la réponse, sa valeur sera utilisée pour remplacer la
385      session. Comme précédemment, tout chiffrement ou déchiffrement,
386      ainsi que la lecture ou
387      l'écriture de ou vers la session à partir du mécanisme de stockage
388      choisi sont gérés par le module <code class="module"><a href="../mod/mod_session.html">mod_session</a></code> et la
389      configuration correspondante.</dd>
390
391      <dt>Applications indépendantes</dt>
392      <dd>Les applications peuvent choisir de manipuler la session en
393      s'affranchissant du contrôle du serveur HTTP Apache. Dans ce cas,
394      c'est l'application qui doit prendre en charge la lecture de la
395      session depuis le mécanisme de stockage choisi, son déchiffrement,
396      sa mise à jour, son chiffrement et sa réécriture vers le mécanisme
397      de stockage choisi de manière appropriée.</dd>
398      </dl>

445<div class="directive-section"><h2><a name="sessionexclude" id="sessionexclude">Directive</a> <a name="SessionExclude" id="SessionExclude">SessionExclude</a></h2>
446<table class="directive">
447<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Définit les préfixes d'URLs pour lesquels une session sera
448ignorée</td></tr>
449<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>SessionExclude <var>chemin</var></code></td></tr>
450<tr><th><a href="directive-dict.html#Default">Défaut:</a></th><td><code>none</code></td></tr>
451<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>configuration globale, serveur virtuel, répertoire, .htaccess</td></tr>
452<tr><th><a href="directive-dict.html#Status">Statut:</a></th><td>Extension</td></tr>
453<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_session</td></tr>
454</table>
455    <p>La directive <code class="directive">SessionExclude</code> permet de
456    définir les préfixes d'URLs pour lesquels la session sera
457    désactivée. Ceci peut améliorer l'efficacité d'un site web, en
458    ciblant de manière plus précise l'espace d'URL pour lequel une
459    session devra être maintenue. Par défaut, toutes les URLs du
460    contexte ou du conteneur courant sont incluses dans la session. La
461    directive <code class="directive"><a href="#sessionexclude">SessionExclude</a></code>
462    l'emporte sur la directive <code class="directive"><a href="#sessioninclude">SessionInclude</a></code>.</p>
463
464    <div class="warning"><h3>Avertissement</h3>
465    <p>Cette directive a un comportement similaire à celui de l'attribut
466    <var>chemin</var> des cookies HTTP, mais ne doit pas être confondue
467    avec cet attribut. En effet, cette directive ne définit pas
468    l'attribut <var>chemin</var>, qui doit être configuré
469    séparément.</p></div>

473<div class="directive-section"><h2><a name="sessionexpiryupdateinterval" id="sessionexpiryupdateinterval">Directive</a> <a name="SessionExpiryUpdateInterval" id="SessionExpiryUpdateInterval">SessionExpiryUpdateInterval</a></h2>
474<table class="directive">
475<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Définit le nombre de secondes dont la durée d'expiration d'une
476session peut changer sans que cette session soit mise à jour</td></tr>
477<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>SessionExpiryUpdateInterval <var>interval</var></code></td></tr>
478<tr><th><a href="directive-dict.html#Default">Défaut:</a></th><td><code>SessionExpiryUpdateInterval 0 (mise à jour systématique)</code></td></tr>
479<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>configuration globale, serveur virtuel, répertoire, .htaccess</td></tr>
480<tr><th><a href="directive-dict.html#Status">Statut:</a></th><td>Extension</td></tr>
481<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_session</td></tr>
482<tr><th><a href="directive-dict.html#Compatibility">Compatibilité:</a></th><td>Disponible à partir de la version 2.4.41 du serveur HTTP Apache</td></tr>
483</table>
484    <p>La directive <code class="directive">SessionExpiryUpdateInterval</code>
485    permet d'éviter le coût de l'écriture d'une session pour chaque
486    requête en n'effectuant cette mise à jour que lorsque la date
487    d'expiration a changé. Ceci permet d'améliorer les performances d'un
488    site web ou de réduire la charge d'une base de données lorsqu'on
489    utilise <code class="module"><a href="../mod/mod_session_dbd.html">mod_session_dbd</a></code>. La session est
490    systématiquement mise à jour si les données stockées dans la session
491    ont été modifiées ou si la durée d'expiration a été modifiée d'une
492    durée supérieure à l'intervalle spécifié.</p>
493
494    <p>Définir l'intervalle à 0 désactive cette directive, et
495    l'expiration de la session sera alors rafraîchie pour chaque requête.</p>
496
497    <p>Cette directive n'a d'effet que si on l'utilise en combinaison
498    avec la directive <code class="directive"><a href="#sessionmaxage">SessionMaxAge</a></code> qui active
499    l'expiration des sessions. Les sessions sans date d'expiration ne
500    sont écrites que lorsque les données qu'elles renferment ont été
501    modifiées.</p>
502
503    <div class="warning"><h3>Avertissement</h3>
504    <p>Comme l'expiration de la session n'est pas systématiquement
505    rafraîchie à chaque requête, une session peut arriver à expiration
506    plus tôt d'un nombre de secondes spécifié dans le paramètre
507    <var>interval</var>. Définir un petit intervalle est en général
508    assez sur, mais en revenche n'a qu'un effet minime sur la prise en
509    compte des durées d'expiration.</p></div>

568<div class="directive-section"><h2><a name="sessionmaxage" id="sessionmaxage">Directive</a> <a name="SessionMaxAge" id="SessionMaxAge">SessionMaxAge</a></h2>
569<table class="directive">
570<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Définit une durée de vie maximale pour la session en
571secondes</td></tr>
572<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>SessionMaxAge <var>durée de vie maximale</var></code></td></tr>
573<tr><th><a href="directive-dict.html#Default">Défaut:</a></th><td><code>SessionMaxAge 0</code></td></tr>
574<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>configuration globale, serveur virtuel, répertoire, .htaccess</td></tr>
575<tr><th><a href="directive-dict.html#Override">Surcharges autorisées:</a></th><td>AuthConfig</td></tr>
576<tr><th><a href="directive-dict.html#Status">Statut:</a></th><td>Extension</td></tr>
577<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_session</td></tr>
578</table>
579    <p>La directive <code class="directive">SessionMaxAge</code> permet de
580    définir la durée maximale pendant laquelle une session restera
581    valide. Lorsqu'une session est sauvegardée, cette durée est
582    réinitialisée et la session peut continuer d'exister. Si la durée
583    d'une session dépasse cette limite sans qu'une requête au serveur ne
584    vienne la rafraîchir, la session va passer hors délai et sera
585    supprimée. Lorsqu'une session est utilisée pour stocker les
586    informations de connexion d'un utilisateur, ceci aura pour effet de
587    le déconnecter automatiquement après le délai spécifié.</p>
588
589    <p>Donner à cette directive la valeur 0 empêche l'expiration de la
590    session.</p>
