63<h2><a name="intro" id="intro">Mise en cache des données d'authentification</a></h2>
64    <p>Certains utilisateurs qui mettent en oeuvre une authentification
65    lourde s'appuyant par exemple sur des requêtes SQL
66    (<code class="module"><a href="../mod/mod_authn_dbd.html">mod_authn_dbd</a></code>) ont signalé une charge induite
67    inacceptable sur leur fournisseur d'authentification. Cela se
68    produit typiquement dans le cas où une page HTML contient des
69    centaines d'objets (images, scripts, pages de styles, media,
70    etc...), et où une requête pour cette page génère des centaines de
71    sous-requêtes à effet immédiat pour des contenus supplémentaires
72    authentifiés.</p>
73    <p>Pour résoudre ce problème, <code class="module"><a href="../mod/mod_authn_socache.html">mod_authn_socache</a></code> fournit une
74    solution qui permet de maintenir un cache des données
75    d'authentification.</p>
76</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>
77<div class="section">
78<h2><a name="usage" id="usage">Utilisation</a></h2>
79    <p>Le cache d'authentification doit être utilisé lorsque les
80    requêtes d'authentification induisent une charge significative sur le
81    serveur, le serveur d'arrière-plan ou le réseau. Cette mise en cache
82    n'apportera probablement aucune amélioration dans le cas d'une
83    authentification à base de fichier (<code class="module"><a href="../mod/mod_authn_file.html">mod_authn_file</a></code>)
84    ou de base de données dbm (<code class="module"><a href="../mod/mod_authn_dbm.html">mod_authn_dbm</a></code>) car ces
85    méthodes sont de par leur conception rapides et légères (la mise en
86    cache peut cependant s'avérer utile dans le cas où le fichier est
87    situé sur un montage réseau). Les fournisseurs d'authentification
88    basés sur SQL ou LDAP ont plus de chances de tirer parti de cette
89    mise en cache, en particulier lorsqu'un problème de performances est
90    détecté. <code class="module"><a href="../mod/mod_authnz_ldap.html">mod_authnz_ldap</a></code> gérant son propre cache,
91    seul <code class="module"><a href="../mod/mod_authn_dbd.html">mod_authn_dbd</a></code> est concerné par notre sujet.</p>
92    <p>Les principales règles à appliquer pour la mise en cache sont :</p>
93    <ol><li>Inclure le fournisseur pour lequel vous voulez effectuer une
94    mise en cache dans une directive
95    <code class="directive"><a href="#authncacheprovidefor">AuthnCacheProvideFor</a></code>.</li>
96        <li>Mettre <var>socache</var> avant le fournisseur pour lequel
97	vous voulez effectuer une mise en cache dans votre directive
98	<code class="directive"><a href="../mod/mod_auth_basic.html#authbasicprovider">AuthBasicProvider</a></code>
99	ou <code class="directive"><a href="../mod/mod_auth_digest.html#authdigestprovider">AuthDigestProvider</a></code>.</li>
100    </ol>
101    <p>Voici un exemple simple permettant d'accélérer
102    <code class="module"><a href="../mod/mod_authn_dbd.html">mod_authn_dbd</a></code> et utilisant dbm comme moteur de la
103    mise en cache :</p>
104    <pre class="prettyprint lang-config">    #AuthnCacheSOCache est optionnel. S'il est défini, il l'est pour
105    #l'ensemble du serveur
106AuthnCacheSOCache dbm
107&lt;Directory "/usr/www/myhost/private"&gt;
108    AuthType Basic
109    AuthName "Cached Authentication Example"
110    AuthBasicProvider socache dbd
111    AuthDBDUserPWQuery "SELECT password FROM authn WHERE user = %s"
112    AuthnCacheProvideFor dbd
113    Require valid-user
114    #Optionnel
115    AuthnCacheContext dbd-authn-example
116&lt;/Directory&gt;</pre>

120<h2><a name="dev" id="dev">La mise en cache avec les modules tiers</a></h2>
121    <p>Les développeurs de modules doivent savoir que la mise en cache
122    avec <code class="module"><a href="../mod/mod_authn_socache.html">mod_authn_socache</a></code> doit être activée dans leurs modules. La
123    fonction de l'API <var>ap_authn_cache_store</var> permet de
124    mettre en cache les données d'authentification qu'un fournisseur
125    vient de rechercher ou de générer. Vous trouverez des exemples
126    d'utilisation à <a href="http://svn.eu.apache.org/viewvc?view=revision&amp;revision=957072">r957072</a>, où trois fournisseurs authn sont activés pour la mise
127    en cache.</p>
128</div>

130<div class="directive-section"><h2><a name="authncachecontext" id="authncachecontext">Directive</a> <a name="AuthnCacheContext" id="AuthnCacheContext">AuthnCacheContext</a></h2>
131<table class="directive">
132<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Spécifie une chaîne de contexte à utiliser dans la clé du
133cache</td></tr>
134<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>AuthnCacheContext directory|server|<var>custom-string</var></code></td></tr>
135<tr><th><a href="directive-dict.html#Default">Défaut:</a></th><td><code>AuthnCacheContext directory</code></td></tr>
136<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>répertoire</td></tr>
137<tr><th><a href="directive-dict.html#Status">Statut:</a></th><td>Base</td></tr>
138<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_authn_socache</td></tr>
139</table>
140    <p>Cette directive permet de spécifier une chaîne à utiliser avec le
141    nom d'utilisateur fourni (et le domaine d'authentification - realm -
142    dans le cas d'une authentification à base de condensés) lors de la
143    construction d'une clé de cache. Ceci permet de lever l'ambiguïté
144    entre plusieurs noms d'utilisateurs identiques servant différentes
145    zones d'authentification sur le serveur.</p>
146    <p>Il y a deux valeurs spéciales pour le paramètre : <code>directory</code>,
147    qui utilise le contexte de répertoire de la requête comme chaîne, et
148    <code>server</code>, qui utilise le nom du serveur virtuel.</p>
149    <p>La valeur par défaut est <code>directory</code>, qui est aussi la
150    définition la plus courante. Ceci est cependant loin d'être optimal,
151    car par exemple, <var>$app-base</var>, <var>$app-base/images</var>,
152    <var>$app-base/scripts</var> et <var>$app-base/media</var>
153    possèderont chacun leur propre clé de cache. Il est préférable
154    d'utiliser le fournisseur de mot de passe : par exemple un fichier
155    <var>htpasswd</var> ou une table de base de données.</p>
156    <p>Les contextes peuvent être partagés entre différentes zones du
157    serveur, où les données d'authentification sont partagées. Ceci est
158    cependant susceptible de créer des trous de sécurité de type
159    cross-site ou cross-application, et cette directive n'est donc pas
160    disponible dans les contextes <var>.htaccess</var>.</p>

164<div class="directive-section"><h2><a name="authncacheenable" id="authncacheenable">Directive</a> <a name="AuthnCacheEnable" id="AuthnCacheEnable">AuthnCacheEnable</a></h2>
165<table class="directive">
166<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Active la mise en cache de l'authentification en tout
167endroit</td></tr>
168<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>AuthnCacheEnable</code></td></tr>
169<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>configuration globale</td></tr>
170<tr><th><a href="directive-dict.html#Status">Statut:</a></th><td>Base</td></tr>
171<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_authn_socache</td></tr>
172</table>
173    <p>Normalement, cette directive n'est pas nécessaire : l'activation
174    est implicite si la mise en cache de l'authentification a été
175    activée en tout autre endroit du fichier <var>httpd.conf</var>. Par
176    contre, si cette mise en cache n'a pas été activée, par défaut, elle
177    ne sera pas initialisée, et ne sera donc pas disponible dans un
178    contexte de fichier <var>.htaccess</var>. Cette directive permet
179    d'être sûr que la mise en cache a bien été activée et pourra
180    donc être utilisée dans les fichiers <var>.htaccess</var>.</p>
181
182</div>

184<div class="directive-section"><h2><a name="authncacheprovidefor" id="authncacheprovidefor">Directive</a> <a name="AuthnCacheProvideFor" id="AuthnCacheProvideFor">AuthnCacheProvideFor</a></h2>
185<table class="directive">
186<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Spécifie le fournisseur pour lequel on veut effectuer une
187mise en cache</td></tr>
188<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>AuthnCacheProvideFor <var>fournisseur-authn</var> [...]</code></td></tr>
189<tr><th><a href="directive-dict.html#Default">Défaut:</a></th><td><code>None</code></td></tr>
190<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>répertoire, .htaccess</td></tr>
191<tr><th><a href="directive-dict.html#Override">Surcharges autorisées:</a></th><td>AuthConfig</td></tr>
192<tr><th><a href="directive-dict.html#Status">Statut:</a></th><td>Base</td></tr>
193<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_authn_socache</td></tr>
194</table>
195    <p>Cette directive permet de spécifier un ou plusieurs fournisseurs pour
196    le(s)quel(s) on veut effectuer une mise en cache. Les données
197    d'authentification trouvées par un fournisseur non spécifié dans une
198    directive <code class="directive">AuthnCacheProvideFor</code> ne seront pas mises en
199    cache.</p>
200
201    <p>Par exemple, pour mettre en cache les données d'authentification
202    trouvées par <code class="module"><a href="../mod/mod_authn_dbd.html">mod_authn_dbd</a></code> ou par un fournisseur
203    personnalisé <var>mon-fournisseur</var>, et ne pas mettre en cache
204    celles trouvées par les fournisseurs légers comme file ou dbm :</p>
205    <pre class="prettyprint lang-config">AuthnCacheProvideFor dbd mon-fournisseur</pre>
206

210<div class="directive-section"><h2><a name="authncachesocache" id="authncachesocache">Directive</a> <a name="AuthnCacheSOCache" id="AuthnCacheSOCache">AuthnCacheSOCache</a></h2>
211<table class="directive">
212<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Sélectionne le fournisseur socache d'arrière-plan à
213utiliser</td></tr>
214<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>AuthnCacheSOCache <var>nom-fournisseur[:arguments-fournisseur]</var></code></td></tr>
215<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>configuration globale</td></tr>
216<tr><th><a href="directive-dict.html#Status">Statut:</a></th><td>Base</td></tr>
217<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_authn_socache</td></tr>
218<tr><th><a href="directive-dict.html#Compatibility">Compatibilité:</a></th><td>Les arguments optionnels du fournisseur sont disponibles
219à partir de la version 2.4.7 du serveur HTTP Apache</td></tr>
220</table>
221    <p>Cette définition s'applique à l'ensemble du serveur et permet de
222    sélectionner un fournisseur pour le <a href="../socache.html">cache
223    d'objets partagés</a>, ainsi que des arguments éventuels pour ce
224    fournisseur. Les fournisseurs disponibles sont, entre autres, "dbm",
225    "dc", "memcache", ou "shmcb", chacun d'entre eux nécessitant le chargement
226    du module approprié. Si elle est
227    absente, c'est la valeur par défaut pour votre plate-forme qui sera
228    utilisée.</p>

232<div class="directive-section"><h2><a name="authncachetimeout" id="authncachetimeout">Directive</a> <a name="AuthnCacheTimeout" id="AuthnCacheTimeout">AuthnCacheTimeout</a></h2>
233<table class="directive">
234<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Définit une durée de vie pour les entrées du cache</td></tr>
235<tr><th><a href="directive-dict.html#Syntax">Syntaxe:</a></th><td><code>AuthnCacheTimeout <var>durée-de-vie</var> (secondes)</code></td></tr>
236<tr><th><a href="directive-dict.html#Default">Défaut:</a></th><td><code>AuthnCacheTimeout 300 (5 minutes)</code></td></tr>
237<tr><th><a href="directive-dict.html#Context">Contexte:</a></th><td>répertoire, .htaccess</td></tr>
238<tr><th><a href="directive-dict.html#Override">Surcharges autorisées:</a></th><td>AuthConfig</td></tr>
239<tr><th><a href="directive-dict.html#Status">Statut:</a></th><td>Base</td></tr>
240<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_authn_socache</td></tr>
241</table>
242    <p>La mise en cache des données d'authentification peut constituer
243    un trou de sécurité, bien qu'un mise en cache de courte durée ne
244    posera probablement pas de problème. En général, il est conseillé de
245    conserver les entrées du cache de façon à ce que la charge du serveur
246    d'arrière-plan reste normale, mais pas plus longtemps ;
247    une durée de vie plus longue peut être paramétrée si les
248    changements d'utilisateurs et de mots de passe sont peu fréquents.
249    La durée de vie par défaut de 300 secondes (5 minutes) est à la fois
250    raisonnable et suffisamment importante pour réduire la charge d'un
251    serveur d'arrière-plan comme dbd (requêtes SQL).</p>
252    <p>Cette durée de vie ne doit pas être confondue avec la durée de
253    vie de session qui est un tout autre sujet. Cependant, vous devez
254    utiliser votre logiciel de gestion de session pour vérifier si les
255    données d'authentification mises en cache peuvent allonger
256    accidentellement une session, et en tenir compte lorsque vous
257    définissez la durée de vie.</p>
