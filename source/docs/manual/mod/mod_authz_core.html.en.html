37<h3>Summary</h3>
38
39    <p>This module provides core authorization capabilities so that
40    authenticated users can be allowed or denied access to portions
41    of the web site. <code class="module"><a href="../mod/mod_authz_core.html">mod_authz_core</a></code> provides the
42    functionality to register various authorization providers. It is
43    usually used in conjunction with an authentication
44    provider module such as <code class="module"><a href="../mod/mod_authn_file.html">mod_authn_file</a></code> and an
45    authorization module such as <code class="module"><a href="../mod/mod_authz_user.html">mod_authz_user</a></code>. It
46    also allows for advanced logic to be applied to the
47    authorization processing.</p>

69<h2><a name="logic" id="logic">Authorization Containers</a></h2>
70
71    <p>The authorization container directives
72    <code class="directive"><a href="#requireall">&lt;RequireAll&gt;</a></code>,
73    <code class="directive"><a href="#requireany">&lt;RequireAny&gt;</a></code>
74    and
75    <code class="directive"><a href="#requirenone">&lt;RequireNone&gt;</a></code>
76    may be combined with each other and with the
77    <code class="directive"><a href="#require">Require</a></code>
78    directive to express complex authorization logic.</p>
79
80    <p>The example below expresses the following authorization logic.
81    In order to access the resource, the user must either be the
82    <code>superadmin</code> user, or belong to both the
83    <code>admins</code> group and the <code>Administrators</code> LDAP
84    group and either belong to the <code>sales</code> group or
85    have the LDAP <code>dept</code> attribute <code>sales</code>.
86    Furthermore, in order to access the resource, the user must
87    not belong to either the <code>temps</code> group or the
88    LDAP group <code>Temporary Employees</code>.</p>
89
90    <pre class="prettyprint lang-config">&lt;Directory "/www/mydocs"&gt;
91    &lt;RequireAll&gt;
92        &lt;RequireAny&gt;
93            Require user superadmin
94            &lt;RequireAll&gt;
95                Require group admins
96                Require ldap-group "cn=Administrators,o=Airius"
97                &lt;RequireAny&gt;
98                    Require group sales
99                    Require ldap-attribute dept="sales"
100                &lt;/RequireAny&gt;
101            &lt;/RequireAll&gt;
102        &lt;/RequireAny&gt;
103        &lt;RequireNone&gt;
104            Require group temps
105            Require ldap-group "cn=Temporary Employees,o=Airius"
106        &lt;/RequireNone&gt;
107    &lt;/RequireAll&gt;
108&lt;/Directory&gt;</pre>

112<h2><a name="requiredirectives" id="requiredirectives">The Require Directives</a></h2>
113
114  <p><code class="module"><a href="../mod/mod_authz_core.html">mod_authz_core</a></code> provides some generic authorization
115  providers which can be used with the
116  <code class="directive"><a href="#require">Require</a></code> directive.</p>
117
118  <h3><a name="reqenv" id="reqenv">Require env</a></h3>
119
120    <p>The <code>env</code> provider allows access to the server
121    to be controlled based on the existence of an <a href="../env.html">environment variable</a>. When <code>Require
122    env <var>env-variable</var></code> is specified, then the request is
123    allowed access if the environment variable <var>env-variable</var>
124    exists. The server provides the ability to set environment
125    variables in a flexible way based on characteristics of the client
126    request using the directives provided by
127    <code class="module"><a href="../mod/mod_setenvif.html">mod_setenvif</a></code>. Therefore, this directive can be
128    used to allow access based on such factors as the clients
129    <code>User-Agent</code> (browser type), <code>Referer</code>, or
130    other HTTP request header fields.</p>
131
132    <pre class="prettyprint lang-config">SetEnvIf User-Agent "^KnockKnock/2\.0" let_me_in
133&lt;Directory "/docroot"&gt;
134    Require env let_me_in
135&lt;/Directory&gt;</pre>
136
137
138    <p>In this case, browsers with a user-agent string beginning
139    with <code>KnockKnock/2.0</code> will be allowed access, and all
140    others will be denied.</p>
141
142    <p>When the server looks up a path via an internal
143    <a class="glossarylink" href="../glossary.html#subrequest" title="see glossary">subrequest</a> such as looking
144    for a <code class="directive"><a href="../mod/mod_dir.html#directoryindex">DirectoryIndex</a></code>
145    or generating a directory listing with <code class="module"><a href="../mod/mod_autoindex.html">mod_autoindex</a></code>,
146    per-request environment variables are <em>not</em> inherited in the
147    subrequest. Additionally,
148    <code class="directive"><a href="../mod/mod_setenvif.html#setenvif">SetEnvIf</a></code> directives
149    are not separately evaluated in the subrequest due to the API phases
150    <code class="module"><a href="../mod/mod_setenvif.html">mod_setenvif</a></code> takes action in.</p>
151
152  
153
154  <h3><a name="reqall" id="reqall">Require all</a></h3>
155
156    <p>The <code>all</code> provider mimics the functionality that
157    was previously provided by the 'Allow from all' and 'Deny from all'
158    directives.  This provider can take one of two arguments which are
159    'granted' or 'denied'.  The following examples will grant or deny
160    access to all requests.</p>
161
162    <pre class="prettyprint lang-config">Require all granted</pre>
163
164
165    <pre class="prettyprint lang-config">Require all denied</pre>
166
167
168  
169
170  <h3><a name="reqmethod" id="reqmethod">Require method</a></h3>
171
172    <p>The <code>method</code> provider allows using the HTTP method in
173    authorization decisions. The GET and HEAD methods are treated as
174    equivalent. The TRACE method is not available to this provider,
175    use <code class="directive"><a href="../mod/core.html#traceenable">TraceEnable</a></code> instead.</p>
176
177    <p>The following example will only allow GET, HEAD, POST, and OPTIONS
178    requests:</p>
179
180    <pre class="prettyprint lang-config">Require method GET POST OPTIONS</pre>
181
182
183    <p>The following example will allow GET, HEAD, POST, and OPTIONS
184    requests without authentication, and require a valid user for all other
185    methods:</p>
186
187    <pre class="prettyprint lang-config">&lt;RequireAny&gt;
188     Require method GET POST OPTIONS
189     Require valid-user
190&lt;/RequireAny&gt;</pre>
191
192
193  
194
195  <h3><a name="reqexpr" id="reqexpr">Require expr</a></h3>
196
197  <p>The <code>expr</code> provider allows basing authorization
198  decisions on arbitrary expressions.</p>
199
200    <pre class="prettyprint lang-config">Require expr "%{TIME_HOUR} -ge 9 &amp;&amp; %{TIME_HOUR} -le 17"</pre>
201
202
203    <pre class="prettyprint lang-config">&lt;RequireAll&gt;
204    Require expr "!(%{QUERY_STRING} =~ /secret/)"
205    Require expr "%{REQUEST_URI} in { '/example.cgi', '/other.cgi' }"
206&lt;/RequireAll&gt;</pre>
207
208
209    <pre class="prettyprint lang-config">Require expr "!(%{QUERY_STRING} =~ /secret/) &amp;&amp; %{REQUEST_URI} in { '/example.cgi', '/other.cgi' }"</pre>
210
211
212  <p>The syntax is described in the <a href="../expr.html">ap_expr</a>
213  documentation. Before httpd 2.4.16, the surrounding double-quotes MUST be
214  omitted.</p>
215
216  <p>Normally, the expression is evaluated before authentication. However, if
217  the expression returns false and references the variable
218  <code>%{REMOTE_USER}</code>, authentication will be performed and
219  the expression will be re-evaluated.</p>
220

226<h2><a name="authzalias" id="authzalias">Creating Authorization Provider Aliases</a></h2>
227
228    <p>Extended authorization providers can be created within the configuration
229    file and assigned an alias name.  The alias providers can then be referenced
230    through the <code class="directive"><a href="#require">Require</a></code> directive
231    in the same way as a base authorization provider.  Besides the ability to
232    create and alias an extended provider, it also allows the same extended
233    authorization provider to be referenced by multiple locations.
234    </p>
235
236    <h3><a name="example" id="example">Example</a></h3>
237        <p>The example below creates two different ldap authorization provider
238        aliases based on the ldap-group authorization provider.  This example
239        allows a single authorization location to check group membership within
240        multiple ldap hosts:
241        </p>
242
243        <pre class="prettyprint lang-config">&lt;AuthzProviderAlias ldap-group ldap-group-alias1 "cn=my-group,o=ctx"&gt;
244    AuthLDAPBindDN "cn=youruser,o=ctx"
245    AuthLDAPBindPassword yourpassword
246    AuthLDAPUrl "ldap://ldap.host/o=ctx"
247&lt;/AuthzProviderAlias&gt;
248
249&lt;AuthzProviderAlias ldap-group ldap-group-alias2 "cn=my-other-group,o=dev"&gt;
250    AuthLDAPBindDN "cn=yourotheruser,o=dev"
251    AuthLDAPBindPassword yourotherpassword
252    AuthLDAPUrl "ldap://other.ldap.host/o=dev?cn"
253&lt;/AuthzProviderAlias&gt;
254
255Alias "/secure" "/webpages/secure"
256&lt;Directory "/webpages/secure"&gt;
257    Require all granted
258
259    AuthBasicProvider file
260
261    AuthType Basic
262    AuthName LDAP_Protected_Place
263
264    #implied OR operation
265    Require ldap-group-alias1
266    Require ldap-group-alias2
267&lt;/Directory&gt;</pre>
268

273<div class="directive-section"><h2><a name="AuthMerging" id="AuthMerging">AuthMerging</a> <a name="authmerging" id="authmerging">Directive</a></h2>
274<table class="directive">
275<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Controls the manner in which each configuration section's
276authorization logic is combined with that of preceding configuration
277sections.</td></tr>
278<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>AuthMerging Off | And | Or</code></td></tr>
279<tr><th><a href="directive-dict.html#Default">Default:</a></th><td><code>AuthMerging Off</code></td></tr>
280<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>directory, .htaccess</td></tr>
281<tr><th><a href="directive-dict.html#Override">Override:</a></th><td>AuthConfig</td></tr>
282<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Base</td></tr>
283<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_authz_core</td></tr>
284</table>
285    <p>When authorization is enabled, it is normally inherited by each
286    subsequent <a href="../sections.html#merging">configuration section</a>,
287    unless a different set of authorization directives is specified.
288    This is the default action, which corresponds to an explicit setting
289    of <code>AuthMerging Off</code>.</p>
290
291    <p>However, there may be circumstances in which it is desirable
292    for a configuration section's authorization to be combined with
293    that of its predecessor while configuration sections are being
294    merged.  Two options are available for this case, <code>And</code>
295    and <code>Or</code>.</p>
296
297    <p>When a configuration section contains <code>AuthMerging And</code>
298    or <code>AuthMerging Or</code>,
299    its authorization logic is combined with that of the nearest
300    predecessor (according to the overall order of configuration sections)
301    which also contains authorization logic as if the two sections
302    were jointly contained within a
303    <code class="directive"><a href="#requireall">&lt;RequireAll&gt;</a></code> or
304    <code class="directive"><a href="#requireany">&lt;RequireAny&gt;</a></code>
305    directive, respectively.</p>
306
307    <div class="note">The setting of <code class="directive">AuthMerging</code> is not
308    inherited outside of the configuration section in which it appears.
309    In the following example, only users belonging to group <code>alpha</code>
310    may access <code>/www/docs</code>.  Users belonging to either
311    groups <code>alpha</code> or <code>beta</code> may access
312    <code>/www/docs/ab</code>.  However, the default <code>Off</code>
313    setting of <code class="directive">AuthMerging</code> applies to the
314    <code class="directive"><a href="../mod/core.html#directory">&lt;Directory&gt;</a></code>
315    configuration section for <code>/www/docs/ab/gamma</code>, so
316    that section's authorization directives override those of the
317    preceding sections.  Thus only users belong to the group
318    <code>gamma</code> may access <code>/www/docs/ab/gamma</code>.</div>
319
320    <pre class="prettyprint lang-config">&lt;Directory "/www/docs"&gt;
321    AuthType Basic
322    AuthName Documents
323    AuthBasicProvider file
324    AuthUserFile "/usr/local/apache/passwd/passwords"
325    Require group alpha
326&lt;/Directory&gt;
327
328&lt;Directory "/www/docs/ab"&gt;
329    AuthMerging Or
330    Require group beta
331&lt;/Directory&gt;
332
333&lt;Directory "/www/docs/ab/gamma"&gt;
334    Require group gamma
335&lt;/Directory&gt;</pre>

343extension of a base authorization provider and referenced by the specified
344alias</td></tr>
345<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>&lt;AuthzProviderAlias <var>baseProvider Alias Require-Parameters</var>&gt;

371&lt;/Directory&gt;</pre>

376<div class="directive-section"><h2><a name="AuthzSendForbiddenOnFailure" id="AuthzSendForbiddenOnFailure">AuthzSendForbiddenOnFailure</a> <a name="authzsendforbiddenonfailure" id="authzsendforbiddenonfailure">Directive</a></h2>
377<table class="directive">
378<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Send '403 FORBIDDEN' instead of '401 UNAUTHORIZED' if
379authentication succeeds but authorization fails
380</td></tr>
381<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>AuthzSendForbiddenOnFailure On|Off</code></td></tr>
382<tr><th><a href="directive-dict.html#Default">Default:</a></th><td><code>AuthzSendForbiddenOnFailure Off</code></td></tr>
383<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>directory, .htaccess</td></tr>
384<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Base</td></tr>
385<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_authz_core</td></tr>
386<tr><th><a href="directive-dict.html#Compatibility">Compatibility:</a></th><td>Available in Apache HTTPD 2.3.11 and later</td></tr>
387</table>
388    <p>If authentication succeeds but authorization fails, Apache HTTPD will
389    respond with an HTTP response code of '401 UNAUTHORIZED' by default. This
390    usually causes browsers to display the password dialogue to the user
391    again, which is not wanted in all situations.
392    <code class="directive">AuthzSendForbiddenOnFailure</code> allows to change the
393    response code to '403 FORBIDDEN'.</p>
394
395    <div class="warning"><h3>Security Warning</h3>
396    <p>Modifying the response in case of missing authorization weakens the
397    security of the password, because it reveals to a possible attacker, that
398    his guessed password was right.</p>
399    </div>

403<div class="directive-section"><h2><a name="Require" id="Require">Require</a> <a name="require" id="require">Directive</a></h2>
404<table class="directive">
405<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Tests whether an authenticated user is authorized by
406an authorization provider.</td></tr>
407<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>Require [not] <var>entity-name</var>
408    [<var>entity-name</var>] ...</code></td></tr>
409<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>directory, .htaccess</td></tr>
410<tr><th><a href="directive-dict.html#Override">Override:</a></th><td>AuthConfig</td></tr>
411<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Base</td></tr>
412<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_authz_core</td></tr>
413</table>
414    <p>This directive tests whether an authenticated user is authorized
415    according to a particular authorization provider and the specified
416    restrictions. <code class="module"><a href="../mod/mod_authz_core.html">mod_authz_core</a></code> provides the following
417    generic authorization providers:</p>
418
419    <dl>
420      <dt><code>Require all granted</code></dt>
421      <dd>Access is allowed unconditionally.</dd>
422
423      <dt><code>Require all denied</code></dt>
424      <dd>Access is denied unconditionally.</dd>
425
426      <dt><code>Require env <var>env-var</var> [<var>env-var</var>]
427      ...</code></dt>
428      <dd>Access is allowed only if one of the given environment variables is
429          set.</dd>
430
431      <dt><code>Require method <var>http-method</var> [<var>http-method</var>]
432      ...</code></dt>
433      <dd>Access is allowed only for the given HTTP methods.</dd>
434
435      <dt><code>Require expr <var>expression</var> </code></dt>
436      <dd>Access is allowed if <var>expression</var> evaluates to true.</dd>
437    </dl>
438
439    <p>Some of the allowed syntaxes provided by <code class="module"><a href="../mod/mod_authz_user.html">mod_authz_user</a></code>,
440       <code class="module"><a href="../mod/mod_authz_host.html">mod_authz_host</a></code>,
441       and <code class="module"><a href="../mod/mod_authz_groupfile.html">mod_authz_groupfile</a></code> are:</p>
442
443    <dl>
444      <dt><code>Require user <var>userid</var> [<var>userid</var>]
445      ...</code></dt>
446      <dd>Only the named users can access the resource.</dd>
447
448      <dt><code>Require group <var>group-name</var> [<var>group-name</var>]
449      ...</code></dt>
450      <dd>Only users in the named groups can access the resource.</dd>
451
452      <dt><code>Require valid-user</code></dt>
453      <dd>All valid users can access the resource.</dd>
454
455      <dt><code>Require ip 10 172.20 192.168.2</code></dt>
456      <dd>Clients in the specified IP address ranges can access the
457      resource.</dd>
458
459      <dt><code>Require forward-dns dynamic.example.org</code></dt>
460      <dd>A client the IP of which is resolved from the name dynamic.example.org will be granted access.
461      </dd>
462
463    </dl>
464
465    <p>Other authorization modules that implement require options
466    include <code class="module"><a href="../mod/mod_authnz_ldap.html">mod_authnz_ldap</a></code>,
467    <code class="module"><a href="../mod/mod_authz_dbm.html">mod_authz_dbm</a></code>, <code class="module"><a href="../mod/mod_authz_dbd.html">mod_authz_dbd</a></code>,
468    <code class="module"><a href="../mod/mod_authz_owner.html">mod_authz_owner</a></code> and <code class="module"><a href="../mod/mod_ssl.html">mod_ssl</a></code>.</p>
469
470    <p>In most cases, for a complete authentication and authorization
471    configuration, <code class="directive">Require</code> must be accompanied by
472    <code class="directive"><a href="../mod/mod_authn_core.html#authname">AuthName</a></code>, <code class="directive"><a href="../mod/mod_authn_core.html#authtype">AuthType</a></code> and
473    <code class="directive"><a href="../mod/mod_auth_basic.html#authbasicprovider">AuthBasicProvider</a></code> or
474    <code class="directive"><a href="../mod/mod_auth_digest.html#authdigestprovider">AuthDigestProvider</a></code>
475    directives, and directives such as
476    <code class="directive"><a href="../mod/mod_authn_file.html#authuserfile">AuthUserFile</a></code>
477    and <code class="directive"><a href="../mod/mod_authz_groupfile.html#authgroupfile">AuthGroupFile</a></code> (to
478    define users and groups) in order to work correctly. Example:</p>
479
480    <pre class="prettyprint lang-config">AuthType Basic
481AuthName "Restricted Resource"
482AuthBasicProvider file
483AuthUserFile "/web/users"
484AuthGroupFile "/web/groups"
485Require group admin</pre>
486
487
488    <p>Access controls which are applied in this way are effective for
489    <strong>all</strong> methods. <strong>This is what is normally
490    desired.</strong> If you wish to apply access controls only to
491    specific methods, while leaving other methods unprotected, then
492    place the <code class="directive">Require</code> statement into a
493    <code class="directive"><a href="../mod/core.html#limit">&lt;Limit&gt;</a></code>
494    section.</p>
495
496    <p>The result of the <code class="directive">Require</code> directive
497    may be negated through the use of the
498    <code>not</code> option.  As with the other negated authorization
499    directive <code class="directive">&lt;RequireNone&gt;</code>,
500    when the <code class="directive">Require</code> directive is negated it can
501    only fail or return a neutral result, and therefore may never
502    independently authorize a request.</p>
503
504    <p>In the following example, all users in the <code>alpha</code>
505    and <code>beta</code> groups are authorized, except for those who
506    are also in the <code>reject</code> group.</p>
507
508    <pre class="prettyprint lang-config">&lt;Directory "/www/docs"&gt;
509    &lt;RequireAll&gt;
510        Require group alpha beta
511        Require not group reject
512    &lt;/RequireAll&gt;
513&lt;/Directory&gt;</pre>
514
515
516    <p>When multiple <code class="directive">Require</code> directives are
517    used in a single
518    <a href="../sections.html#merging">configuration section</a>
519    and are not contained in another authorization directive like
520    <code class="directive"><a href="#requireall">&lt;RequireAll&gt;</a></code>,
521    they are implicitly contained within a
522    <code class="directive"><a href="#requireany">&lt;RequireAny&gt;</a></code>
523    directive.  Thus the first one to authorize a user authorizes the
524    entire request, and subsequent <code class="directive">Require</code> directives
525    are ignored.</p>
526
527    <div class="warning"><h3>Security Warning</h3>
528    <p>Exercise caution when setting authorization directives in
529    <code class="directive"><a href="../mod/core.html#location">Location</a></code> sections
530    that overlap with content served out of the filesystem.
531    By default, these <a href="../sections.html#merging">configuration sections</a> overwrite authorization configuration
532    in <code class="directive"><a href="../mod/core.html#directory">Directory</a></code>,
533    and <code class="directive"><a href="../mod/core.html#files">Files</a></code> sections.</p>
534    <p>The <code class="directive"><a href="#authmerging">AuthMerging</a></code> directive
535    can be used to control how authorization configuration sections are
536    merged.</p>
537    </div>

581<div class="directive-section"><h2><a name="RequireAny" id="RequireAny">&lt;RequireAny&gt;</a> <a name="requireany" id="requireany">Directive</a></h2>
582<table class="directive">
583<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Enclose a group of authorization directives of which one
584must succeed for the enclosing directive to succeed.</td></tr>
585<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>&lt;RequireAny&gt; ... &lt;/RequireAny&gt;</code></td></tr>
586<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>directory, .htaccess</td></tr>
587<tr><th><a href="directive-dict.html#Override">Override:</a></th><td>AuthConfig</td></tr>
588<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Base</td></tr>
589<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_authz_core</td></tr>
590</table>
591    <p><code class="directive">&lt;RequireAny&gt;</code> and
592    <code>&lt;/RequireAny&gt;</code> are used to enclose a group of
593    authorization directives of which one must succeed in order for
594    the <code class="directive">&lt;RequireAny&gt;</code> directive to
595    succeed.</p>
596
597    <p>If one or more of the directives contained within the
598    <code class="directive">&lt;RequireAny&gt;</code> directive succeed,
599    then the <code class="directive">&lt;RequireAny&gt;</code> directive
600    succeeds.  If none succeed and none fail, then it returns a
601    neutral result.  In all other cases, it fails.</p>
602
603    <div class="note">Because negated authorization directives are unable to
604    return a successful result, they can not significantly influence
605    the result of a <code class="directive">&lt;RequireAny&gt;</code>
606    directive.  (At most they could cause the directive to fail in
607    the case where they failed and all other directives returned a
608    neutral value.)  Therefore negated authorization directives
609    are not permitted within a <code class="directive">&lt;RequireAny&gt;</code>
610    directive.</div>

621<table class="directive">
622<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Enclose a group of authorization directives of which none
623must succeed for the enclosing directive to not fail.</td></tr>

630    <p><code class="directive">&lt;RequireNone&gt;</code> and
631    <code>&lt;/RequireNone&gt;</code> are used to enclose a group of
632    authorization directives of which none must succeed
633    in order for the
634    <code class="directive">&lt;RequireNone&gt;</code> directive to
635    not fail.</p>
636
637    <p>If one or more of the directives contained within the
638    <code class="directive">&lt;RequireNone&gt;</code> directive succeed,
639    then the <code class="directive">&lt;RequireNone&gt;</code> directive

645    authorized to access a resource.</p>
