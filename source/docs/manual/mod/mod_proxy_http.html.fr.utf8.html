39<h3>Sommaire</h3>
40
41    <p>Pour pouvoir fonctionner, ce module <em>requiert</em> le
42    chargement de <code class="module"><a href="../mod/mod_proxy.html">mod_proxy</a></code>. Il fournit le support du
43    mandatement des requêtes HTTP et HTTPS. <code class="module"><a href="../mod/mod_proxy_http.html">mod_proxy_http</a></code>
44    supporte HTTP/0.9, HTTP/1.0 et HTTP/1.1. Il ne fournit
45    <em>aucune</em> fonctionnalité de mise en cache. Si vous souhaitez
46    mettre en oeuvre un mandataire qui assure aussi les fonctions de
47    mise en cache, vous devez utiliser les services du module
48    <code class="module"><a href="../mod/mod_cache.html">mod_cache</a></code>.</p>
49
50    <p>Ainsi, pour pouvoir traiter les requêtes HTTP mandatées,
51    <code class="module"><a href="../mod/mod_proxy.html">mod_proxy</a></code>, et <code class="module"><a href="../mod/mod_proxy_http.html">mod_proxy_http</a></code>
52    doivent être chargés dans le serveur.</p>
53
54    <div class="warning"><h3>Avertissement</h3>
55      <p>N'activez pas la fonctionnalité de mandataire avant d'avoir <a href="mod_proxy.html#access">sécurisé votre serveur</a>. Les
56      serveurs mandataires ouverts sont dangereux non seulement pour
57      votre réseau, mais aussi pour l'Internet au sens large.</p>

73<h2><a name="env" id="env">Variables d'environnement</a></h2>
74    <p>Outre les directives de configuration qui contrôlent le
75    comportement de <code class="module"><a href="../mod/mod_proxy.html">mod_proxy</a></code>, plusieurs <dfn>variables
76    d'environnement</dfn> permettent de contrôler le fournisseur du
77    protocole HTTP. Parmi les variables suivantes, celle qui ne
78    nécessitent pas de valeur particulière sont définies quelle que soit
79    la valeur qu'on leur affecte.</p>
80    <dl>
81        <dt>proxy-sendextracrlf</dt>
82        <dd>Provoque l'envoi par le mandataire d'une nouvelle ligne
83	CR-LF supplémentaire à la fin de la requête.
84	Ceci constitue un
85	moyen de contournement d'une bogue de certains
86	navigateurs.</dd>
87        <dt>force-proxy-request-1.0</dt>
88        <dd>Force le mandataire à envoyer des requêtes vers le serveur
89	cible selon le protocole HTTP/1.0 et désactive les
90	fonctionnalités propres à HTTP/1.1.</dd>
91        <dt>proxy-nokeepalive</dt>
92        <dd>Force le mandataire à fermer la connexion avec le serveur
93	cible après chaque requête.</dd>
94        <dt>proxy-chain-auth</dt>
95        <dd>Si le mandataire requiert une authentification, il va lire
96	et exploiter les données d'authentification pour mandataire
97	envoyées par le client. Si <var>proxy-chain-auth</var> est
98	définie, il va <em>aussi</em> faire suivre ces données vers le
99	mandataire suivant dans la chaîne. Ceci peut s'avérer nécessaire
100	si une chaîne de mandataires partagent les informations
101	d'authentification.
102	<strong>Avertissement concernant la sécurité :</strong>
103	Ne définissez cette variable que si vous êtes sûr d'en avoir
104	besoin, car elle peut provoquer la divulgation d'informations
105	sensibles !</dd>
106        <dt>proxy-sendcl</dt>
107        <dd>Avec HTTP/1.0, toutes les requêtes qui possèdent un corps
108	(par exemple les requêtes POST) doivent comporter un en-tête
109	<var>Content-Length</var>. Cette variable d'environnement force
110	le mandataire Apache à envoyer cet en-tête au serveur cible,
111	sans tenir compte de ce que lui a envoyé le client. Ceci permet
112	d'assurer la compatibilité lorsqu'on mandate un serveur cible
113	mettant en oeuvre un protocole de type HTTP/1.0 ou inconnu. Elle
114	peut cependant nécessiter la mise en tampon de l'intégralité de
115	la requête par le mandataire, ce qui s'avère très inefficace
116	pour les requêtes de grande taille.</dd>
117        <dt>proxy-sendchunks ou proxy-sendchunked</dt>
118        <dd>Cette variable constitue l'opposé de
119	<var>proxy-sendcl</var>. Elle permet la transmission des corps
120	de requêtes vers le serveur cible en utilisant un codage de
121	transfert fractionné. Ceci permet une transmission des requêtes
122	plus efficace, mais nécessite que le serveur cible supporte le
123	protocole HTTP/1.1.</dd>
124        <dt>proxy-interim-response</dt>
125        <dd>Cette variable peut prendre les valeurs <code>RFC</code>
126	(valeur par défaut) ou
127	<code>Suppress</code>. Les versions précédentes de httpd
128	supprimaient les réponses intermédiaires HTTP (1xx) envoyées par
129	le serveur cible. En pratique, si un serveur cible envoie une
130	réponse intermédiaire, il se peut qu'il étende lui-même le
131	protocole d'une manière dont nous n'avons pas connaissance, ou
132	tout simplement non conforme. Le comportement du mandataire est
133	donc maintenant configurable : définissez
134	<code>proxy-interim-response RFC</code> pour être totalement
135	compatible avec le protocole, ou <code>proxy-interim-response
136	Suppress</code> pour supprimer les réponses intermédiaires.</dd>
137        <dt>proxy-initial-not-pooled</dt>
138        <dd>Si cette variable est définie, aucune connexion faisant
139	partie d'un jeu ne sera réutilisée si la requête
140	du client est la requête initiale pour une connexion.
141	Ceci permet d'éviter le message d'erreur
142	"proxy: error reading status line from remote server" causé par
143	la situation de compétition au cours de laquelle le serveur
144	cible ferme la connexion du jeu après la vérification de la
145	connexion par le mandataire,  et avant que les données envoyées
146	par le mandataire n'atteignent le serveur cible. Il faut
147	cependant garder à l'esprit que la définition de cette variable
148	dégrade les performances, particulièrement avec les clients
149	HTTP/1.0.
150        </dd>
151    </dl>
