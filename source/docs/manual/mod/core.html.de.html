146<div class="directive-section"><h2><a name="AcceptFilter" id="AcceptFilter">AcceptFilter</a>-<a name="acceptfilter" id="acceptfilter">Direktive</a></h2>
147<table class="directive">
148<tr><th><a href="directive-dict.html#Description">Beschreibung:</a></th><td>Konfiguriert Optimierungen f&#252;r lauschende Sockets bestimmter
149Protokolle</td></tr>
150<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>AcceptFilter <var>Protokoll</var> <var>Filter</var></code></td></tr>
151<tr><th><a href="directive-dict.html#Context">Kontext:</a></th><td>Serverkonfiguration</td></tr>
152<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Core</td></tr>
153<tr><th><a href="directive-dict.html#Module">Modul:</a></th><td>core</td></tr>
154<tr><th><a href="directive-dict.html#Compatibility">Kompatibilit&#228;t:</a></th><td>Verf&#252;gbar ab Apache 2.1.5</td></tr>
155</table>
156    <p>Diese Direktive aktiviert betriebssystemspezifische Optimierungen
157    f&#252;r lauschende Sockets anhand des Protokolltyps. Der grundlegende
158    Ansatz ist, dass der Kernel das Socket nicht an den Serverprozess
159    &#252;bergibt, bis entweder Daten verf&#252;gbar sind oder eine komplette
160    HTTP-Anfrage zwischengespeichert wurde. Derzeit werden
161    ausschlie&#223;lich die <a href="http://www.freebsd.org/cgi/man.cgi?query=accept_filter&amp;sektion=9">Accept-Filter von FreeBSD</a> und das primitivere
162    <code>TCP_DEFER_ACCEPT</code> von Linux unterst&#252;tzt.</p>
163
164    <p>Die Standardeinstellungen f&#252;r FreeBSD sind:</p>
165    <div class="example"><p><code>
166        AcceptFilter http httpready<br />
167        AcceptFilter https dataready
168    </code></p></div>
169
170    <p>Der <code>httpready</code>-Accept-Filter puffert komplette
171    HTTP-Anfragen auf Kernelebene. Sobald eine Anfrage vollst&#228;ndig
172    vorliegt, schickt der Kernel sie an den Server weiter. Bitte schlagen Sie
173    in der <a href="http://www.freebsd.org/cgi/man.cgi?query=accf_http&amp;sektion=9">accf_http(9)</a>-Manpage f&#252;r weitere Details nach. HTTPS-Anfragen
174    sind verschl&#252;sselt. Daher wird daf&#252;r nur der <a href="http://www.freebsd.org/cgi/man.cgi?query=accf_data&amp;sektion=9">accf_data(9)</a>-Filter verwendet.</p>
175
176    <p>Die Standardeinstellungen f&#252;r Linux sind:</p>
177    <div class="example"><p><code>
178        AcceptFilter http data<br />
179        AcceptFilter https data
180    </code></p></div>
181
182    <p><code>TCP_DEFER_ACCEPT</code> unter Linux unterst&#252;tzt keine
183    Zwischenspeicherung von HTTP-Anfragen. Jeder andere Wert als
184    <code>none</code> aktiviert <code>TCP_DEFER_ACCEPT</code> auf dem
185    Lauschsocket. Mehr Details finden Sie in der <a href="http://homepages.cwi.nl/~aeb/linux/man2html/man7/tcp.7.html">tcp(7)</a>-Manpage von Linux.</p>
186
187    <p>Wenn Sie <code>none</code> als Argument verwenden, werden alle
188    Accept-Filter f&#252;r das Protokoll abgeschaltet. Das ist sinnvoll
189    f&#252;r Protokolle, bei denen der Server zuerst Daten senden muss,
190    wie zum Beispiel <code>nntp</code>:</p>
191    <div class="example"><p><code>AcceptFilter nttp none</code></p></div>

196<div class="directive-section"><h2><a name="AcceptPathInfo" id="AcceptPathInfo">AcceptPathInfo</a>-<a name="acceptpathinfo" id="acceptpathinfo">Direktive</a></h2>
197<table class="directive">
198<tr><th><a href="directive-dict.html#Description">Beschreibung:</a></th><td>Ressourcen lassen angeh&#228;ngte Pfadangaben zu</td></tr>
199<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>AcceptPathInfo On|Off|Default</code></td></tr>
200<tr><th><a href="directive-dict.html#Default">Voreinstellung:</a></th><td><code>AcceptPathInfo Default</code></td></tr>
201<tr><th><a href="directive-dict.html#Context">Kontext:</a></th><td>Serverkonfiguration, Virtual Host, Verzeichnis, .htaccess</td></tr>
202<tr><th><a href="directive-dict.html#Override">AllowOverride:</a></th><td>FileInfo</td></tr>
203<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Core</td></tr>
204<tr><th><a href="directive-dict.html#Module">Modul:</a></th><td>core</td></tr>
205<tr><th><a href="directive-dict.html#Compatibility">Kompatibilit&#228;t:</a></th><td>Verf&#252;gbar ab Apache 2.0.30</td></tr>
206</table>
207    <p>Die Direktive steuert, ob Anfragen akzeptiert oder
208    abgewiesen werden, bei denen nach der tats&#228;chlichen
209    Datei (oder einer nicht existierenden Datei in einem existierenden
210    Verzeichnis) zus&#228;tzliche Pfadangaben folgen. Die angeh&#228;ngte
211    Pfadangabe kann Skripten in der Umgebungsvariable <code>PATH_INFO</code>
212    verf&#252;gbar gemacht werden.</p>
213
214    <p>Nehmen wir beispielsweise an, dass <code>/test/</code> auf ein
215    Verzeichnis zeigt, welches lediglich eine Datei <code>here.html</code>
216    enth&#228;lt. Dann wird bei Anfragen nach
217    <code>/test/here.html/more</code> und
218    <code>/test/nothere.html/more</code> beides Mal <code>/more</code>
219    als <code>PATH_INFO</code> ermittelt.</p>
220
221    <p>Die drei m&#246;glichen Argumente f&#252;r die Direktive
222    <code class="directive">AcceptPathInfo</code> sind:</p>
223
224    <dl>
225    <dt><code>Off</code></dt><dd>Eine Anfrage wird nur dann akzeptiert,
226    wenn sie exakt auf ein existierendes Verzeichnis (oder eine Datei)
227    abgebildet werden kann. Daher w&#252;rde eine Anfrage mit einer nach dem
228    tats&#228;chlichen Dateinamen angeh&#228;ngten Pfadangabe, wie
229    <code>/test/here.html/more</code> im obigen Beispiel, den Fehler
230    404 NOT FOUND <span class="transnote">(<em>Anm.d.&#220;.:</em> nicht gefunden)</span>
231    zur&#252;ckgeben.</dd>
232
233    <dt><code>On</code></dt>
234    <dd>Eine Anfrage wird akzeptiert, wenn eine vorangestellte Pfadangabe
235    auf ein existierendes Verzeichnis abgebildet werden kann. Das
236    obige Beispiel <code>/test/here.html/more</code> wird akzeptiert,
237    wenn <code>/test/here.html</code> auf eine g&#252;ltige Datei
238    zeigt.</dd>
239
240    <dt><code>Default</code></dt>
241    <dd>Die Behandlung von Anfragen mit angeh&#228;ngten Pfadangaben
242    wird von dem f&#252;r die Anfrage verantwortlichen <a href="../handler.html">Handler</a> bestimmt. Der Core-Handler
243    f&#252;r gew&#246;hnliche Dateien weist <code>PATH_INFO</code>-Zugriffe
244    standardm&#228;&#223;ig zur&#252;ck. Handler, die Skripte bedienen,
245    wie z.B. <a href="mod_cgi.html">cgi-script</a> und
246    <a href="mod_isapi.html">isapi-handler</a>, sind im Allgemeinen darauf
247    voreingestellt, <code>PATH_INFO</code> zu akzeptieren.</dd>
248    </dl>
249
250    <p>Das eigentliche Ziel von <code>AcceptPathInfo</code> ist es, Ihnen
251    das &#220;berschreiben der Voreinstellung der Handler bez&#252;glich
252    der Akzeptanz oder Ablehnung von <code>PATH_INFO</code> zu erlauben.
253    Eine solche &#196;nderung ist zum Beispiel notwendig, wenn Sie einen
254    <a href="../filter.html">Filter</a> wie <a href="mod_include.html">INCLUDES</a> verwenden, um Inhalte
255    abh&#228;ngig von <code>PATH_INFO</code> zu generieren. Der
256    Core-Handler w&#252;rde die Anfrage normalerweise abweisen. Verwenden
257    Sie die folgende Konfiguration, um dennoch solch ein Skript zu
258    erm&#246;glichen.</p>
259
260    <div class="example"><p><code>
261      &lt;Files "mypaths.shtml"&gt;<br />
262      <span class="indent">
263        Options +Includes<br />
264        SetOutputFilter INCLUDES<br />
265        AcceptPathInfo On<br />
266      </span>
267      &lt;/Files&gt;
268    </code></p></div>

316<div class="directive-section"><h2><a name="AddDefaultCharset" id="AddDefaultCharset">AddDefaultCharset</a>-<a name="adddefaultcharset" id="adddefaultcharset">Direktive</a></h2>
317<table class="directive">
318<tr><th><a href="directive-dict.html#Description">Beschreibung:</a></th><td>Standard-Charset-Parameter, der bei Antworten vom Content-Type 
319  <code>text/plain</code> oder <code>text/html</code> hinzugef&#252;gt wird
320</td></tr>
321<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>AddDefaultCharset On|Off|<var>Zeichenkodierung</var></code></td></tr>
322<tr><th><a href="directive-dict.html#Default">Voreinstellung:</a></th><td><code>AddDefaultCharset Off</code></td></tr>
323<tr><th><a href="directive-dict.html#Context">Kontext:</a></th><td>Serverkonfiguration, Virtual Host, Verzeichnis, .htaccess</td></tr>
324<tr><th><a href="directive-dict.html#Override">AllowOverride:</a></th><td>FileInfo</td></tr>
325<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Core</td></tr>
326<tr><th><a href="directive-dict.html#Module">Modul:</a></th><td>core</td></tr>
327</table>
328  <p>Die Direktive gibt einen Standardwert f&#252;r den Charset-Paramter des
329    Medientyps (den Namen einer Zeichencodierung) an, der einer Antwort
330    genau dann hinzugef&#252;gt wird, wenn der Content-Type der Antwort entweder
331    <code>text/plain</code> oder <code>text/html</code> ist. Dies sollte jedes
332    mittels <code>META</code>-Element im Datenteil der Antwort angegebene
333    Charset &#252;berschreiben. Das genaue Verhalten h&#228;ngt jedoch oft von
334    der Client-Konfiguration des Benutzers ab. Die Einstellung
335    <code>AddDefaultCharset Off</code> deaktiviert diese Funktionalit&#228;t.
336    <code>AddDefaultCharset On</code> aktiviert die Standard-Zeichenkodierung
337    <code>iso-8859-1</code>. Jeder andere Wert wird als die zu verwendende
338    <var>Zeichenkodierung</var> aufgefa&#223;t, die eines der bei <a href="http://www.iana.org/assignments/character-sets">IANA registrierten
339      Charset-Werte</a> zur Verwendung in MIME-Medientypen sein sollte. Zum
340    Beispiel:</p>
341
342    <div class="example"><p><code>
343      AddDefaultCharset utf-8
344    </code></p></div>
345
346    <p><code class="directive">AddDefaultCharset</code> sollte nur verwendet werden,
347      wenn von allen Textressourcen, f&#252;r die es gilt, bekannt ist, dass sie
348      in dieser Zeichkodierung vorliegen, oder wenn es zu unbequem ist, ihre
349      Zeichenkodierung indivuell zu benennen. Ein solches Beispiel ist das
350      Hinzuf&#252;gen des Charset-Parameters zu Ressourcen, die generierte
351      Inhalte enthalten. Ein Beispiel sind CGI-Skript-Altlasten, die aufgrund von
352      in die Ausgabe integrierten Daten, die durch den Benutzer &#252;bermittelt
353      wurden, gegen Cross-Site-Scripting-Angriffe verwundbar sind. Eine bessere
354      L&#246;sung w&#228;re jedoch, diese Skripte zu korrigieren (oder zu
355      l&#246;schen), da die Angabe einer Standard-Zeichencodierung keine
356      Anwender sch&#252;tzt, die in ihrem Browser die Funktion zur
357      automatischen Erkennung der Zeichenkodierung aktiviert haben.</p>

578<tr><th><a href="directive-dict.html#Description">Beschreibung:</a></th><td>Aktiviert die Generierung von <code>Content-MD5</code>
579HTTP-Response-Headern</td></tr>
580<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>ContentDigest On|Off</code></td></tr>
581<tr><th><a href="directive-dict.html#Default">Voreinstellung:</a></th><td><code>ContentDigest Off</code></td></tr>
582<tr><th><a href="directive-dict.html#Context">Kontext:</a></th><td>Serverkonfiguration, Virtual Host, Verzeichnis, .htaccess</td></tr>
583<tr><th><a href="directive-dict.html#Override">AllowOverride:</a></th><td>Options</td></tr>
584<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Core</td></tr>
585<tr><th><a href="directive-dict.html#Module">Modul:</a></th><td>core</td></tr>
586</table>
587    <p>Die Direktive aktiviert die Generierung von
588    <code>Content-MD5</code>-Headern, wie sie in RFC1864 bzw. RFC2616
589    definiert sind.</p>
590
591    <p>MD5 ist ein Algorithmus zur Berechnung eines "Datenextrakts"
592    (zuweilen "Fingerabdruck" genannt) <span class="transnote">(<em>Anm.d.&#220;.:</em> Der "Datenextrakt" wird im
593    Englischen als "message digest" oder "fingerprint" bezeichnet.)</span>
594    aus beliebig langen Daten. Es gilt als zuverl&#228;ssig, dass
595    Ver&#228;nderungen an den Daten sich in Ver&#228;nderungen des
596    Extrakts wiederspiegeln.</p>
597
598    <p>Der <code>Content-MD5</code>-Header bietet eine
599    End-to-End-Integrit&#228;tspr&#252;fung (MIC) <span class="transnote">(<em>Anm.d.&#220;.:</em> MIC steht f&#252;r
600    "message integrity check".)</span> des Daten-Inhalts. Ein Proxy oder
601    Client kann diesen Header pr&#252;fen, um zuf&#228;llige Ver&#228;nderungen
602    des Entity-Inhalts bei der &#220;bertragung festzustellen.
603    Beispielheader:</p>
604
605    <div class="example"><p><code>
606      Content-MD5: AuLb7Dp1rqtRtxz2m9kRpA==
607    </code></p></div>
608
609    <p>Beachten Sie bitte, dass dies Performanceprobleme auf Ihrem
610    System verursachen kann, da der Extrakt bei jeder Anfrage
611    berechnet wird (der Wert wird nicht zwischengespeichert).</p>
612
613    <p><code>Content-MD5</code> wird nur f&#252;r Dokumente gesendet,
614    die von <code class="module"><a href="../mod/core.html">core</a></code> bedient werden, nicht jedoch bei
615    Modulen. SSI-Dokumente, CGI-Skript-Ausgaben und Byte-Range-Antworten
616    besitzen diesen Header beispielsweise nicht.</p>

689<div class="directive-section"><h2><a name="Directory" id="Directory">&lt;Directory&gt;</a>-<a name="directory" id="directory">Direktive</a></h2>
690<table class="directive">
691<tr><th><a href="directive-dict.html#Description">Beschreibung:</a></th><td>Umschlie&#223;t eine Gruppe von Direktiven, die nur auf
692das genannte Verzeichnis des Dateisystems und Unterverzeichnisse angewendet
693werden</td></tr>
694<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>&lt;Directory <var>Verzeichnispfad</var>&gt;
695... &lt;/Directory&gt;</code></td></tr>
696<tr><th><a href="directive-dict.html#Context">Kontext:</a></th><td>Serverkonfiguration, Virtual Host</td></tr>
697<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Core</td></tr>
698<tr><th><a href="directive-dict.html#Module">Modul:</a></th><td>core</td></tr>
699</table>
700    <p><code class="directive">&lt;Directory&gt;</code> und
701    <code>&lt;/Directory&gt;</code> werden dazu verwendet, eine Gruppe
702    von Direktiven zusammenzufassen, die nur f&#252;r das genannte
703    Verzeichnis und dessen Unterverzeichnisse gelten. Jede Direktive,
704    die im Verzeichnis-Kontext erlaubt ist, kann verwendet werden.
705    <var>Verzeichnispfad</var> ist entweder der vollst&#228;ndige Pfad zu
706    einem Verzeichnis oder eine Zeichenkette mit Platzhaltern wie sie von der
707    Unix-Shell zum Abgleich verwendet werden. In einer Zeichenkette
708    mit Platzhaltern <span class="transnote">(<em>Anm.d.&#220;.:</em> sogenannte wild-cards)</span> entspricht
709    <code>?</code> einem einzelnen Zeichen und <code>*</code> einer
710    Zeichenkette beliebiger L&#228;nge. Sie k&#246;nnen auch auch
711    <code>[]</code>-Zeichenbereiche verwenden. Keiner der Platzhalter
712    entspricht dem Zeichen "/". Daher passt <code>&lt;Directory
713    /*/public_html&gt;</code> nicht auf <code>/home/user/public_html</code>,
714    <code>&lt;Directory /home/*/public_html&gt;</code> jedoch tut es.
715    Beispiel:</p>
716
717    <div class="example"><p><code>
718      &lt;Directory /usr/local/httpd/htdocs&gt;<br />
719      <span class="indent">
720        Options Indexes FollowSymLinks<br />
721      </span>
722      &lt;/Directory&gt;
723    </code></p></div>
724
725    <div class="note">
726      <p>Seien Sie vorsichtig mit den <var>Verzeichnispfad</var>-Argumenten.
727      Sie m&#252;ssen buchst&#228;blich mit dem Dateisystempfad
728      &#252;bereinstimmen, den der Apache f&#252;r den Zugriff auf die
729      Dateien verwendet. Direktiven, die f&#252;r ein bestimmtes
730      Verzeichnis gelten, gelten nicht f&#252;r Dateien in dem Verzeichnis,
731      auf die &#252;ber einen anderen Pfad zugegriffen wird, wie z.B.
732      &#252;ber verschiedene symbolische Links.</p>
733    </div>
734
735    <p>Erweiterte <a class="glossarylink" href="../glossary.html#regex" title="siehe Glossar">regul&#228;re Ausdr&#252;cke</a> k&#246;nnen ebenfalls
736    verwendet werden, indem das Zeichen <code>~</code> hinzugef&#252;gt
737    wird. Beispielsweise w&#252;rde</p>
738
739    <div class="example"><p><code>
740      &lt;Directory ~ "^/www/.*/[0-9]{3}"&gt;
741    </code></p></div>
742
743    <p>auf Verzeichnisse in <code>/www/</code> passen, die aus drei
744    Zahlen bestehen.</p>
745
746    <p>Wenn mehrere <code class="directive">&lt;Directory&gt;</code>-Abschnitte
747    (ohne regul&#228;re Ausdr&#252;cke) auf ein Verzeichnis (oder
748    ein ihm &#252;bergeordnetes Verzeichnis) passen, welches ein Dokument
749    enth&#228;lt, dann werden die Direktiven der Reihe nach, angefangen
750    beim k&#252;rzesten passenden Muster, vermischt mit den Direktiven
751    aus den <a href="#accessfilename">.htaccess</a>-Dateien, angewendet.
752    Beispiel:</p>
753
754    <div class="example"><p><code>
755      &lt;Directory /&gt;<br />
756      <span class="indent">
757        AllowOverride None<br />
758      </span>
759      &lt;/Directory&gt;<br />
760      <br />
761      &lt;Directory /home/&gt;<br />
762      <span class="indent">
763        AllowOverride FileInfo<br />
764      </span>
765      &lt;/Directory&gt;
766    </code></p></div>
767
768    <p>Beim Zugriff auf das Dokument <code>/home/web/dir/doc.html</code>
769    sind die einzelnen Schritte:</p>
770
771    <ul>
772      <li>Wende die Direktive <code>AllowOverride None</code> an
773      (deaktiviere <code>.htaccess</code>-Dateien).</li>
774
775      <li>Wende die Direktive <code>AllowOverride FileInfo</code>
776      (auf das Verzeichnis <code>/home</code>) an.</li>
777
778      <li>Wende jede <code>FileInfo</code>-Direktive aus
779      <code>/home/.htaccess</code>, <code>/home/web/.htaccess</code> und
780      <code>/home/web/dir/.htaccess</code> der Reihe nach an.</li>
781    </ul>
782
783    <p>Regul&#228;re Ausdr&#252;cke werden solange nicht ber&#252;cksichtigt,
784    bis alle normalen Abschnitte angewendet wurden. Anschlie&#223;end
785    werden alle regul&#228;ren Ausdr&#252;cke in der Reihenfolge
786    gepr&#252;ft, in der sie in der Konfigurationsdatei auftauchen.
787    Beispielsweise wird bei</p>
788
789    <div class="example"><p><code>
790      &lt;Directory ~ abc$&gt;<br />
791      <span class="indent">
792        # ... hier die Direktiven ...<br />
793      </span>
794      &lt;/Directory&gt;
795    </code></p></div>
796
797    <p>der Abschnitt mit dem regul&#228;ren Ausdruck nicht
798    ber&#252;cksichtigt, bis alle normalen
799    <code class="directive">&lt;Directory&gt;</code>-Abschnitte und
800    <code>.htaccess</code>-Dateien angewendet wurden. Dann erst wird
801    der regul&#228;re Ausdruck mit <code>/home/abc/public_html/abc</code>
802    abgeglichen und der entsprechende <code class="directive">&lt;Directory&gt;</code>-Abschnitt angewendet.</p>
803
804    <p><strong>Beachten Sie bitte, dass der vom Apache voreingestellte
805    Zugriff f&#252;r <code>&lt;Directory /&gt;</code>
806    <code>Allow from All</code> ist. Das bedeutet, dass der Apache
807    jede Datei ausliefert, die durch eine URL abgebildet wird. Es wird
808    empfohlen, dass Sie dies durch einen Block wie</strong></p>
809
810    <div class="example"><p><code>
811      &lt;Directory /&gt;<br />
812      <span class="indent">
813        Order Deny,Allow<br />
814        Deny from All<br />
815      </span>
816      &lt;/Directory&gt;
817    </code></p></div>
818
819    <p><strong>&#228;ndern und anschlie&#223;end f&#252;r
820    Verzeichnisse &#252;berschreiben, die Sie verf&#252;gbar machen
821    <em>wollen</em>. F&#252;r weitere Einzelheiten lesen Sie bitte
822    die Seite zu den <a href="../misc/security_tips.html">Sicherheitshinweisen</a>.</strong></p>
823
824    <p>Die Verzeichnisabschnitte erscheinen in der Datei
825    <code>httpd.conf</code>. <code class="directive">&lt;Directory&gt;</code>-Direktiven d&#252;rfen nicht
826    ineinander verschachtelt werden oder innerhalb von <code class="directive"><a href="#limit">&lt;Limit&gt;</a></code>- oder <code class="directive"><a href="#limitexcept">&lt;LimitExcept&gt;</a></code>-Abschnitten auftauchen.</p>
827

980    verwenden darf, wenn er w&#228;hrend der Auslieferung den Inhalt einer
981    Datei lesen muss. Wenn die Bearbeitung einer Anfrage es erfordert,
982    auf die Daten in einer Datei zuzugreifen -- zum Beispiel bei der
983    Auslieferung einer mittels <code class="module"><a href="../mod/mod_include.html">mod_include</a></code> serverseitig
984    analysierten Datei --, dann verwendet der Apache standardm&#228;&#223;ig
985    Memory-Mapping f&#252;r diese Datei, sofern das Betriebssystem es
986    unterst&#252;tzt.</p>

988    <p>Memory-Mapping bedeutet zuweilen eine Performanceverbesserung.
989    In einigen Umgebungen ist es jedoch besser, Memory-Mapping zu
990    deaktivieren, um Problemen w&#228;hrend des Betriebs vorzubeugen:</p>
991
992    <ul>

994    Performance von <code class="program"><a href="../programs/httpd.html">httpd</a></code> reduzieren.</li>
995    <li>Bei einem per NFS eingebundenen <code class="directive"><a href="#documentroot">DocumentRoot</a></code> kann <code class="program"><a href="../programs/httpd.html">httpd</a></code> mit
996    einem Speicherzugriffsfehler <span class="transnote">(<em>Anm.d.&#220;.:</em> ein so genannter "segmentation
997    fault")</span> abst&#252;rzen, wenn eine Datei gel&#246;scht oder
998    gek&#252;rzt wird, w&#228;hrend <code class="program"><a href="../programs/httpd.html">httpd</a></code> sie im Speicher
999    abbildet.</li>
1000    </ul>

1003    anf&#228;llig sind, sollten Sie das Memory-Mapping f&#252;r
1004    auszuliefernde Dateien deaktivieren, indem Sie schreiben:</p>
1005
1006    <div class="example"><p><code>
1007      EnableMMAP Off
1008    </code></p></div>
1009

1011    explizit f&#252;r die st&#246;renden Dateien deaktiviert werden,
1012    indem Sie angeben:</p>
1013
1014    <div class="example"><p><code>
1015      &lt;Directory "/pfad-zu-den-nfs-dateien"&gt;

1036    <p>Die Direktive steuert, ob <code class="program"><a href="../programs/httpd.html">httpd</a></code> die
1037    sendfile-Unterst&#252;tzung des Kernels verwenden kann, um
1038    Dateiinhalte an den Client zu &#252;bermitteln. Wenn die Bearbeitung
1039    einer Anfrage keinen Zugriff auf die Daten in der Datei erfordert --
1040    zum Beispiel bei der Auslieferung einer statischen Datei -- und das
1041    Betriebssystem es unterst&#252;tzt, verwendet der Apache
1042    standardm&#228;&#223;ig sendfile, um den Dateiinhalt zu
1043    &#252;bertragen, ohne die Datei jemals zu lesen.</p>
1044
1045    <p>Der sendfile-Mechanismus vermeidet getrennte Lese- und
1046    Sendeoperationen sowie Puffer-Zuweisungen. Bei einigen Plattformen bzw.
1047    Dateisystemen deaktivieren Sie diese Funktion jedoch besser, um Probleme
1048    w&#228;hrend des Betriebs zu vermeiden:</p>
1049
1050    <ul>
1051    <li>Einige Plattformen besitzen u.U. eine fehlerhafte
1052    sendfile-Unterst&#252;tzung, die das Erstellungssystem nicht erkennt,
1053    insbesondere wenn die Bin&#228;rdateien auf einem anderen Rechner erstellt
1054    und auf eine solche Maschine mit fehlerhafter sendfile-Unterst&#252;tzung
1055    &#252;bertragen wurden.</li>
1056    <li>Bei einem &#252;ber das Netzwerk eingebundenen <code class="directive"><a href="#documentroot">DocumentRoot</a></code> (z.B. NFS oder SMB) ist der
1057    Kernel m&#246;glicherweise nicht in der Lage, die Netzwerkdatei
1058    &#252;ber seinen eigenen Cache zu bedienen.</li>
1059    <li>Unter Linux l&#246;st die Verwendung von <code>sendfile</code>
1060    in Verbindung mit bestimmten Netzwerkkarten und IPv6
1061    TCP-Checksummenfehler aus.</li>
1062    <li>Unter Linux auf Itanium-Systemen kommt sendfile unter Umst&#228;nden
1063    nicht mit Dateien gr&#246;&#223;er als 2GB klar.</li>
1064    </ul>
1065
1066    <p>Bei Serverkonfigurationen, die f&#252;r dieses Problam
1067    anf&#228;llig sind, sollten die diese Funktion deaktivieren, indem
1068    Sie schreiben:</p>
1069
1070    <div class="example"><p><code>
1071      EnableSendfile Off
1072    </code></p></div>
1073
1074    <p>Bei per NFS oder SMB eingebundenen Dateien kann diese Funktion
1075    explizit f&#252;r die st&#246;renden Dateien deaktiviert werden, indem
1076    Sie angeben:</p>
1077
1078    <div class="example"><p><code>
1079      &lt;Directory "/pfad-zu-den-nfs-dateien"&gt;
1080      <span class="indent">
1081        EnableSendfile Off
1082      </span>
1083      &lt;/Directory&gt;
1084    </code></p></div>
1085    <p>Beachten Sie bitte, dass die verzeichnisbasierte und
1086       .htaccess-Konfiguration von <code class="directive">EnableSendfile</code>
1087       nicht vom <code class="module"><a href="../mod/mod_cache_disk.html">mod_cache_disk</a></code>-Modul unterst&#252;tzt wird.
1088       Nur die globale Konfiguration von <code class="directive">EnableSendfile</code>
1089       wird vom Modul beachtet.
1090    </p>

1106<div class="directive-section"><h2><a name="ErrorDocument" id="ErrorDocument">ErrorDocument</a>-<a name="errordocument" id="errordocument">Direktive</a></h2>
1107<table class="directive">
1108<tr><th><a href="directive-dict.html#Description">Beschreibung:</a></th><td>Das, was der Server im Fehlerfall an den Client
1109zur&#252;ckgibt</td></tr>
1110<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>ErrorDocument <var>Fehlercode</var> <var>Dokument</var></code></td></tr>
1111<tr><th><a href="directive-dict.html#Context">Kontext:</a></th><td>Serverkonfiguration, Virtual Host, Verzeichnis, .htaccess</td></tr>
1112<tr><th><a href="directive-dict.html#Override">AllowOverride:</a></th><td>FileInfo</td></tr>
1113<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Core</td></tr>
1114<tr><th><a href="directive-dict.html#Module">Modul:</a></th><td>core</td></tr>
1115<tr><th><a href="directive-dict.html#Compatibility">Kompatibilit&#228;t:</a></th><td>Die Syntax der Anf&#252;hrungszeichen bei Textnachrichten hat
1116sich im Apache 2.0 ge&#228;ndert</td></tr>
1117</table>
1118    <p>Im Falle eines Problems oder Fehlers kann der Apache
1119    konfiguriert werden, eine der vier Aktionen auszuf&#252;hren:</p>
1120
1121    <ol>
1122      <li>Ausgabe einer einfachen, hartkodierten Fehlermeldung</li>
1123
1124      <li>Ausgabe einer angepassten Meldung</li>
1125
1126      <li>Umleitung zu einem lokalen <var>URL-Pfad</var> der das
1127      Problem bzw. den Fehler behandelt</li>
1128
1129      <li>Umleitung zu einer externen <var>URL</var>, die das Problem
1130      bzw. den Fehler behandelt</li>
1131    </ol>
1132
1133    <p>Die erste Option ist Voreinstellung, w&#228;hrend die Optionen
1134    2 bis 4 &#252;ber die Direktive <code class="directive">ErrorDocument</code>
1135    eingestellt werden, welcher der HTTP-Statuscode und eine
1136    URL oder Nachricht folgen. Abh&#228;ngig vom Problem bzw. Fehler bietet
1137    der Apache manchmal zus&#228;tzliche Informationen an.</p>
1138
1139    <p>URLs k&#246;nnen bei lokalen Webpfaden mit einem Schr&#228;gstrich
1140    (/) beginnen (relativ zum <code class="directive"><a href="#documentroot">DocumentRoot</a></code>-Verzeichnis) oder eine vollst&#228;ndige URL
1141    bilden, die der Client aufl&#246;sen kann. Alternativ kann eine
1142    Nachricht f&#252;r die Anzeige im Browser angeboten werden. Beispiel:</p>
1143
1144    <div class="example"><p><code>
1145      ErrorDocument 500 http://foo.example.com/cgi-bin/tester<br />
1146      ErrorDocument 404 /cgi-bin/falsche_urls.pl<br />
1147      ErrorDocument 401 /info_zur_anmeldung.html<br />
1148      ErrorDocument 403 "Der Zugriff ist nicht erlaubt."
1149    </code></p></div>
1150
1151    <p>Au&#223;erdem kann der spezielle Wert <code>default</code> angegeben
1152      werden, um die schlichte, hartkodierte Nachricht des Apache zu verwenden.
1153      Es wird normalerweise nicht ben&#246;tigt, doch <code>default</code>
1154      stellt die einfach, im Apache hartkodierte Meldung in Konfigurationen
1155      wieder her, die ansonsten von einem existierenden <span class="transnote">(<em>Anm.d.&#220;.:</em> zuvor
1156        konfigurierten)</span> <code class="directive">ErrorDocument</code> erben
1157      w&#252;rden.</p>
1158
1159    <div class="example"><p><code>
1160      ErrorDocument 404 /cgi-bin/bad_urls.pl<br /><br />
1161      &lt;Directory /web/docs&gt;<br />
1162      <span class="indent">
1163        ErrorDocument 404 default<br />
1164      </span>
1165      &lt;/Directory&gt;
1166    </code></p></div>
1167
1168    <p>Wenn Sie eine <code class="directive">ErrorDocument</code>-Anweisung
1169    angeben, die auf eine entfernte URL weist (d.h. irgendetwas mit der
1170    Methode <code>http</code> davor), beachten Sie bitte, dass der Apache
1171    eine Umleitung zum Client sendet, um diesem mitzuteilen, wo das
1172    Dokument zu finden ist, auch wenn das Dokument letztlich wieder zum
1173    gleichen Server f&#252;hrt. Das hat mehrere Auswirkungen. Die
1174    wichtigste ist, dass der Client nicht den Original-Statuscode
1175    erh&#228;lt sondern statt dessen einen Umleitungs-Statuscode. Dies
1176    wiederum kann Web-Robots und andere Clients verwirren, die den
1177    Statuscode dazu verwenden, herauszufinden ob eine URL g&#252;ltig ist.
1178    Wenn Sie eine entfernte URL in einer Anweisung
1179    <code>ErrorDocument 401</code> verwenden, wird der Client
1180    dar&#252;ber hinaus nicht wissen, dass er den Benutzer zur Eingabe
1181    eines Passwortes auffordern muss, da er den Statuscode 401 nicht
1182    erh&#228;lt. <strong>Deshalb m&#252;ssen Sie sich auf ein lokales
1183    Dokument beziehen, wenn Sie eine Anweisung <code>ErrorDocument
1184    401</code> verwenden.</strong></p>
1185
1186    <p>Der Microsoft Internet Explorer (MSIE) ignoriert
1187    standardm&#228;&#223;ig serverseitig generierte Fehlermeldungen, wenn
1188    sie "zu kurz" sind und ersetzt sie durch eigene "freundliche"
1189    Fehlermeldungen. Die Gr&#246;&#223;e variiert abh&#228;ngig von der
1190    Art des Fehlers, im Allgemeinen zeigt der MSIE jedoch den
1191    serverseitig generierten Fehler, anstatt ihn zu verstecken, wenn Ihr
1192    Fehlerdokument gr&#246;&#223;er als 512 Bytes ist. Weitere Informationen
1193    sind im Artikel  <a href="http://support.microsoft.com/default.aspx?scid=kb;en-us;Q294807">Q294807</a> in der Microsoft Knowledgebase verf&#252;gbar.</p>
1194
1195    <p>Obwohl die meisten Fehlermeldungen &#252;berschrieben werden
1196    k&#246;nnen, werden unter bestimmten Umst&#228;nden die internen
1197    Meldungen ungeachtet der Einstellung der <code class="directive"><a href="#errordocument">ErrorDocument</a></code>-Direktive verwendet. Insbesondere bei
1198    einer fehlerhaften Anfrage werden der normale Bearbeitungsprozess sofort
1199    beendet und die interne Meldung zur&#252;ckgegeben. Das ist notwendig, um
1200    Sicherheitsprobleme zu vermeiden, die auf Grund fehlerhafter Anfragen
1201    entstehen.</p>
1202
1203    <p>In Versionen vor 2.0 wurden Meldungen durch ein einzelnes
1204    vorangestelltes Anf&#252;hrungszeichen (") erkannt.</p>

1320    zur Einsparung von Netzwerk-Bandbreite benutzt.) Im Apache 1.3.22 und

1337     gleichbedeutend mit:
1338         <div class="example"><p><code>FileETag INode MTime Size</code></p></div></dd>

1340     <dd>Es wird keine <code>ETag</code>-Angabe in die Antwort eingef&#252;gt,
1341     wenn dem Dokument eine Datei zugrundeliegt.</dd>

1347    Vorgabe erlaubt, die von einem gr&#246;&#223;eren Umfeld
1348    geerbt wurde. Jedes Schl&#252;selwort ohne ein solches Prefix
1349    hebt die ererbte Einstellung sofort und vollst&#228;ndig auf.</p>

1876    die gleich TCP-Verbindung zu senden. In einigen F&#228;llen wurde eine
1877    Beschleunigung der Wartezeiten von beinahe 50% f&#252;r HTML-Dokumente
1878    mit vielen Bildern festgestellt. Um Keep-Alive-Verbindungen zu aktivieren,
1879    setzen Sie <code>KeepAlive On</code>.</p>
1880
1881    <p>Bei HTTP/1.0-Clients werden Keep-Alive-Verbindungen nur dann verwendet,
1882    wenn sie vom Client eigens angefordert werden. Desweiteren k&#246;nnen
1883    Keep-Alive-Verbindungen bei einem HTTP/1.0-Client nur dann verwendet
1884    werden, wenn die L&#228;nge des Inhalts im Voraus bekannt ist. Dies
1885    impliziert, dass dynamische Inhalte wie CGI-Ausgaben, SSI-Seiten und
1886    servergenerierte Verzeichnisauflistungen im Allgemeinen keine
1887    Keep-Alive-Verbindungen mit HTTP/1.0-Clients verwenden. Bei
1888    HTTP/1.1-Clients sind Keep-Alive-Verbindungen Voreinstellung, solange
1889    nichts anderes angegeben ist. Wenn der Client es anfordert, wird
1890    Chunked-Encoding verwendet, um Inhalte mit unbekannter L&#228;nge
1891    &#252;ber persistente Verbindungen zu senden.</p>

2241<div class="directive-section"><h2><a name="Location" id="Location">&lt;Location&gt;</a>-<a name="location" id="location">Direktive</a></h2>
2242<table class="directive">
2243<tr><th><a href="directive-dict.html#Description">Beschreibung:</a></th><td>Wendet die enthaltenen Direktiven nur auf die entsprechenden
2244URLs an</td></tr>
2245<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>&lt;Location
2246    <var>URL-Pfad</var>|<var>URL</var>&gt; ... &lt;/Location&gt;</code></td></tr>
2247<tr><th><a href="directive-dict.html#Context">Kontext:</a></th><td>Serverkonfiguration, Virtual Host</td></tr>
2248<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Core</td></tr>
2249<tr><th><a href="directive-dict.html#Module">Modul:</a></th><td>core</td></tr>
2250</table>
2251    <p>Die Direktive <code class="directive">&lt;Location&gt;</code>
2252    begrenzt die Reichweite der enthaltenen Anweisungen auf URLs.
2253    Sie ist der Direktive <code class="directive"><a href="#directory">&lt;Directory&gt;</a></code> &#228;hnlich und startet einen
2254    Abschnitt, der mit der Anweisung <code>&lt;/Location&gt;</code>
2255    abgeschlossen wird. <code class="directive">&lt;Location&gt;</code>-Container werden, nachdem die
2256    <code class="directive"><a href="#directory">&lt;Directory&gt;</a></code>-Container
2257    und <code>.htaccess</code>-Dateien gelesen wurden, und nach den
2258    <code class="directive"><a href="#files">&lt;Files&gt;</a></code>-Containern, in
2259    der Reihenfolge ausgef&#252;hrt, in der sie in der Konfigurationsdatei
2260    erscheinen.</p>
2261
2262    <p><code class="directive">&lt;Location&gt;</code>-Abschnitte operieren
2263      vollst&#228;ndig au&#223;erhalb des Dateisystems. Dies hat mehrere
2264      Konsequenzen. An Wichtigsten, <code class="directive">&lt;Location&gt;</code>-Anweisungen sollten nicht daf&#252;r
2265      verwendet werden, den Zugriff zu Teilen des Dateisystems zu steuern. Da
2266      mehrere unterschiedliche URLs auf die gleiche Stelle des Dateisystems
2267      zeigen k&#246;nnen, k&#246;nnte eine solche Zugriffskontrolle u.U.
2268      umgangen werden.</p>
2269      
2270    <div class="note"><h3>Wann sollte<code class="directive">&lt;Location&gt;</code> verwendet werden</h3>
2271
2272      <p>Verwenden Sie <code class="directive">&lt;Location&gt;</code>, um
2273        Anweisungen auf Inhalte anzuwenden, die au&#223;erhalb des Dateisystems
2274        abgelegt sind. Benutzen Sie <code class="directive"><a href="#directory">&lt;Directory&gt;</a></code> und <code class="directive"><a href="#files">&lt;Files&gt;</a></code> f&#252;r Inhalte, die
2275        innerhalb des Dateisystems abgelegt sind. Eine Ausnahme bildet 
2276        <code>&lt;Location /&gt;</code>, welches ein einfacher Weg ist, um eine 
2277        Konfiguration auf den gesamten Server anzuwenden.</p>
2278    </div>
2279 
2280    <p>F&#252;r alle nicht-Proxy-Anfragen ist die entsprechende URL
2281    ein URL-Pfad in der Form <code>/path/</code>. Es d&#252;rfen weder ein
2282    Schema, noch ein Hostname, noch ein Port, noch ein Query-String einbezogen
2283    werden. F&#252;r Proxy-Anfragen hat die Vergleichs-URL die Form
2284    <code>schema://servername/path</code>. Das Pr&#228;fix muss angegeben
2285    werden.</p>
2286
2287    <p>Die URL kann Platzhalter verwenden. In einer Zeichenfolge mit
2288    Platzhaltern entspricht <code>?</code> einem einzelnen Zeichen und
2289    <code>*</code>einer beliebigen Zeichenfolge.</p>
2290
2291    <p>Erweiterte <a class="glossarylink" href="../glossary.html#regex" title="siehe Glossar">regul&#228;re
2292    Ausdr&#252;cke</a> k&#246;nnen ebenfalls verwendet werden, indem
2293    das Zeichen <code>~</code> hinzugef&#252;gt wird. Beispielsweise
2294    w&#252;rde</p>
2295
2296    <div class="example"><p><code>
2297      &lt;Location ~ "/(extra|special)/data"&gt;
2298    </code></p></div>
2299
2300    <p>auf URLs passen, welche die Zeichenfolge <code>/extra/data</code>
2301    oder <code>/special/data</code> enthalten. Die Direktive <code class="directive"><a href="#locationmatch">&lt;LocationMatch&gt;</a></code> verh&#228;lt sich
2302    genauso wie <code class="directive">&lt;Location&gt;</code> mit
2303    regul&#228;ren Ausdr&#252;cken.</p>
2304
2305    <p>Die Funktionalit&#228;t von <code class="directive">&lt;Location&gt;</code> ist insbesondere dann n&#252;tzlich,
2306    wenn sie mit der <code class="directive"><a href="#sethandler">SetHandler</a></code>-Direktive
2307    kombiniert wird. Um zum Beispiel Statusabfragen zu aktivieren, sie aber
2308    nur von Browsern aus <code>foo.com</code> zuzulassen, k&#246;nnten Sie
2309    schreiben:</p>
2310
2311    <div class="example"><p><code>
2312      &lt;Location /status&gt;<br />
2313      <span class="indent">
2314        SetHandler server-status<br />
2315        Order Deny,Allow<br />
2316        Deny from all<br />
2317        Allow from .foo.com<br />
2318      </span>
2319      &lt;/Location&gt;
2320    </code></p></div>
2321
2322    <div class="note"><h3>Anmerkung zu / (Schr&#228;gstrich, Slash)</h3>
2323      <p>Das Slash-Zeichen hat eine besondere Bedeutung, je nachdem, wo es
2324      in der URL erscheint. Manche werden sein Verhalten vom Dateisystem
2325      gewohnt sein, wo mehrere aufeinanderfolgende Schr&#228;gstriche
2326      h&#228;ufig zu einem Schr&#228;gstrich zusammengefa&#223;t werden
2327      (<em>d.h.</em> <code>/home///foo</code> ist das gleiche wie
2328      <code>/home/foo</code>). Im URL-Raum ist dies nicht notwendigerweise
2329      genauso. Bei der Direktive  <code class="directive"><a href="#locationmatch">&lt;LocationMatch&gt;</a></code> und der <code class="directive">&lt;Location&gt;</code>-Version mit regul&#228;ren Ausdr&#252;cken
2330      m&#252;ssen Sie explizit mehrere Schr&#228;gstriche angeben, wenn Sie
2331      genau dies beabsichtigen.</p>
2332
2333      <p>Beispielsweise w&#252;rde <code>&lt;LocationMatch ^/abc&gt;</code>
2334      auf die angeforderte URL <code>/abc</code> passen, nicht aber auf
2335      <code>//abc</code>. Die Direktive <code class="directive">&lt;Location&gt;</code> (ohne regul&#228;re Ausdr&#252;cke) verh&#228;lt
2336      sich &#228;hnlich, wenn sie f&#252;r Proxy-Anfragen verwendet wird.
2337      Wenn <code class="directive">&lt;Location&gt;</code> (ohne
2338      regul&#228;re Ausdr&#252;cke) jedoch f&#252;r nicht-Proxy-Anfragen
2339      verwendet wird, werden stillscheigend mehrere Schr&#228;chstriche mit
2340      mit einem einzigen Schr&#228;gstrich gleichgesetzt. Geben Sie
2341      beispielsweise <code>&lt;Location /abc/def&gt;</code> an und die
2342      Anfrage lautet auf <code>/abc//def</code>, dann greift die Anweisung.</p>
2343    </div>

2625<div class="directive-section"><h2><a name="NameVirtualHost" id="NameVirtualHost">NameVirtualHost</a>-<a name="namevirtualhost" id="namevirtualhost">Direktive</a></h2>
2626<table class="directive">
2627<tr><th><a href="directive-dict.html#Description">Beschreibung:</a></th><td>Bestimmt eine IP-Adresse f&#252;r den Betrieb namensbasierter
2628virtueller Hosts</td></tr>
2629<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>NameVirtualHost <var>Adresse</var>[:<var>Port</var>]</code></td></tr>
2630<tr><th><a href="directive-dict.html#Context">Kontext:</a></th><td>Serverkonfiguration</td></tr>
2631<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Core</td></tr>
2632<tr><th><a href="directive-dict.html#Module">Modul:</a></th><td>core</td></tr>
2633</table>
2634    <p>Die Direktive <code class="directive">NameVirtualHost</code> ist erforderlich,
2635    wenn Sie <a href="../vhosts/">namensbasierte virtuelle Hosts</a>
2636    konfigurieren m&#246;chten.</p>
2637
2638    <p>Obwohl <var>Adresse</var> eine Hostname sein kann, wird empfohlen,
2639    dass Sie stets eine IP-Adresse verwenden, z.B.:</p>
2640
2641    <div class="example"><p><code>
2642      NameVirtualHost 111.22.33.44
2643    </code></p></div>
2644
2645    <p>Mit der <code class="directive">NameVirtualHost</code>-Anweisung geben Sie
2646    die IP-Adresse an, unter der der Server Anfragen f&#252;r
2647    namensbasierte virtuelle Hosts entgegennimmt. Das ist &#252;blicherweise
2648    die Adresse, zu der die Namen Ihrer namensbasierten virtuellen Hosts
2649    aufgel&#246;st werden. Falls eine Firewall oder ein anderer Proxy die
2650    Anfrage in Empfang nimmt und Sie zu einer weiteren IP-Adresse des Servers
2651    weiterleitet, m&#252;ssen Sie die IP-Adresse der physikalischen
2652    Schnittstelle der Maschine angeben, welche die Anfragen bedient.
2653    Wenn Sie mehrere namensbasierte Hosts an verschiedenen Adressen
2654    betreiben, wiederholen Sie einfach die Anweisung f&#252;r jede
2655    Adresse.</p>
2656
2657    <div class="note"><h3>Anmerkung</h3>
2658      <p>Beachten Sie, dass der "Hauptserver" und jeder
2659      <code>_default_</code>-Server <strong>niemals</strong> bei einer
2660      Anfrage an einer <code class="directive">NameVirtualHost</code>-IP-Adresse
2661      bedient wird (es sei denn, Sie geben aus irgendwelchen Gr&#252;nden
2662      <code class="directive">NameVirtualHost</code> an, definieren dann aber keine
2663      <code class="directive">VirtualHost</code>s f&#252;r diese Adresse).</p>
2664    </div>
2665
2666    <p>Optional k&#246;nnen Sie die Nummer eines Ports angeben, an dem
2667    namensbasierte virtuelle Hosts verwendet werden sollen. Beispiel:</p>
2668
2669    <div class="example"><p><code>
2670      NameVirtualHost 111.22.33.44:8080
2671    </code></p></div>
2672
2673    <p>IPv6-Adressen m&#252;ssen, wie im folgenden Beispiel angegeben, in
2674    eckige Klammern eingeschlossen werden:</p>
2675
2676    <div class="example"><p><code>
2677      NameVirtualHost [2001:db8::a00:20ff:fea7:ccea]:8080
2678    </code></p></div>
2679
2680    <p>Um an allen Schnittstellen Anfragen zu empfangen, k&#246;nnen Sie
2681    <code>*</code> als Argument verwenden.</p>
2682
2683    <div class="example"><p><code>
2684      NameVirtualHost *
2685    </code></p></div>
2686
2687    <div class="note"><h3>Argument der Direktive <code class="directive">&lt;VirtualHost&gt;</code></h3>
2688      <p>Beachten Sie, dass das Argument der <code class="directive">&lt;VirtualHost&gt;</code>-Anweisung exakt auf das Argument
2689      der <code class="directive">NameVirtualHost</code>-Anweisung passen muss.</p>
2690
2691      <div class="example"><p><code>
2692        NameVirtualHost 1.2.3.4<br />
2693        &lt;VirtualHost 1.2.3.4&gt;<br />
2694        # ...<br />
2695        &lt;/VirtualHost&gt;<br />
2696      </code></p></div>
2697    </div>

2705<div class="directive-section"><h2><a name="Options" id="Options">Options</a>-<a name="options" id="options">Direktive</a></h2>
2706<table class="directive">
2707<tr><th><a href="directive-dict.html#Description">Beschreibung:</a></th><td>Definiert, welche Eigenschaften oder Funktionen in einem
2708bestimmten Verzeichnis verf&#252;gbar sind</td></tr>
2709<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>Options
2710    [+|-]<var>Option</var> [[+|-]<var>Option</var>] ...</code></td></tr>
2711<tr><th><a href="directive-dict.html#Default">Voreinstellung:</a></th><td><code>Options All</code></td></tr>
2712<tr><th><a href="directive-dict.html#Context">Kontext:</a></th><td>Serverkonfiguration, Virtual Host, Verzeichnis, .htaccess</td></tr>
2713<tr><th><a href="directive-dict.html#Override">AllowOverride:</a></th><td>Options</td></tr>
2714<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Core</td></tr>
2715<tr><th><a href="directive-dict.html#Module">Modul:</a></th><td>core</td></tr>
2716</table>
2717    <p>Die Direktive <code class="directive">Options</code> steuert, welche
2718    Eigenschaften bzw. Funktionen in einem bestimmten Verzeichnis
2719    verf&#252;gbar sind.</p>
2720
2721    <p><var>Option</var> kann auf <code>None</code> gesetzt werden, wobei
2722    keine der besonderen Eigenschaften verf&#252;gbar sind, oder auf eines
2723    oder mehrere der folgenden:</p>
2724
2725    <dl>
2726      <dt><code>All</code></dt>
2727
2728      <dd>Alle Optionen au&#223;er <code>MultiViews</code>. Dies ist
2729      die Voreinstellung.</dd>
2730
2731      <dt><code>ExecCGI</code></dt>
2732
2733      <dd>Die Ausf&#252;hrung von CGI-Skripten, welche <code class="module"><a href="../mod/mod_cgi.html">mod_cgi</a></code>
2734        verwenden, ist erlaubt.</dd>
2735
2736      <dt><code>FollowSymLinks</code></dt>
2737
2738      <dd>Der Server folgt symbolischen Links in diesem Verzeichnis.
2739      <div class="note">
2740      <p>Auch wenn der Server symbolischen Links folgt, bedeutet dies
2741      <em>nicht</em>, dass der zum Abgleich gegen <code class="directive"><a href="#directory">&lt;Directory&gt;</a></code>-Abschnitte verwendete Pfadname
2742      wechselt.</p>
2743      <p>Beachten Sie auch, dass diese Option innerhalb eines
2744      <code class="directive"><a href="#location">&lt;Location&gt;</a></code>-Abschnitts
2745      <strong>ignoriert wird</strong>.</p>
2746      </div></dd>
2747
2748      <dt><code>Includes</code></dt>
2749
2750      <dd>
2751      Server Side Includes, die von <code class="module"><a href="../mod/mod_include.html">mod_include</a></code> bereitgestellt
2752      werden, sind erlaubt.</dd>
2753
2754      <dt><code>IncludesNOEXEC</code></dt>
2755
2756      <dd>Server Side Includes sind erlaubt, <code>#exec cmd</code>
2757      und <code>#exec cgi</code> sind jedoch deaktiviert. Es ist aber noch
2758      m&#246;glich, CGI-Skripte aus
2759      <code class="directive"><a href="../mod/mod_alias.html#scriptalias">ScriptAlias</a></code>-Verzeichnissen mittels
2760      <code>#include virtual</code> einzubinden.</dd>
2761
2762      <dt><code>Indexes</code></dt>
2763
2764      <dd>Wenn eine URL, die auf ein Verzeichnis zeigt, in dem sich keine durch
2765        <code class="directive"><a href="../mod/mod_dir.html#directoryindex">DirectoryIndex</a></code> definierte 
2766        Indexdatei (<em>z.B.</em> <code>index.html</code>) befindet, dann liefert 
2767        <code class="module"><a href="../mod/mod_autoindex.html">mod_autoindex</a></code> eine formatierte Auflistung des 
2768        Verzeichnisses zur&#252;ck.</dd>
2769
2770      <dt><code>MultiViews</code></dt>
2771
2772      <dd>"MultiViews" sind bei der Verwendung von 
2773        <code class="module"><a href="../mod/mod_negotiation.html">mod_negotiation</a></code> erlaubt (siehe <a href="../content-negotiation.html">Content-Negotiation</a>).</dd>
2774
2775      <dt><code>SymLinksIfOwnerMatch</code></dt>
2776
2777      <dd>Der Server folgt nur symbolischen Links, bei denen die Zieldatei
2778      bzw. das Zielverzeichnis der gleichen Benutzerkennung geh&#246;rt, wie
2779      der Link.
2780      <div class="note"><h3>Anmerkung</h3> Diese Option wird innerhalb eines
2781      <code class="directive"><a href="#location">&lt;Location&gt;</a></code>-Abschnitts
2782      ignoriert.</div></dd>
2783    </dl>
2784
2785    <p>Wenn mehrere <code class="directive">Options</code> auf ein Verzeichnis
2786    angewandt werden k&#246;nnen, dann wird normalerweise die
2787    spezifischste <span class="transnote">(<em>Anm.d.&#220;.:</em> Gemeint ist die zuletzt
2788    ausgef&#252;hrte Option.)</span> verwendet und alle anderen werden
2789    ignoriert; die Optionen werden nicht vermischt. (Siehe auch <a href="../sections.html#mergin">Wie Abschnitte zusammengef&#252;hrt 
2790    werden.</a>.) Wenn jedoch <em>allen</em> Optionen der 
2791    <code class="directive">Options</code>-Anweisung eines der Zeichen
2792    <code>+</code> oder <code>-</code> vorangestellt wird, werden die Optionen
2793    zusammengemischt. Jede Option mit vorangestelltem <code>+</code> wird
2794    zu den momentan g&#252;ltigen Optionen hinzugef&#252;gt und jede Option
2795    mit vorangestelltem <code>-</code> wird aus den derzeit g&#252;ltigen
2796    Optionen entfernt.</p>
2797
2798    <div class="warning"><h3>Warnung</h3>
2799    <p>Die Vermischung von Optionen mit <code>+</code> oder <code>-</code> mit
2800    Optionen ohne diese (Zeichen) ist keine g&#252;ltige Syntax und f&#252;hrt
2801    mit hoher Wahrscheinlichkeit zu unerwarteten Effekten.</p>
2802    </div>
2803
2804    <p>So wird zum Beispiel ohne die Zeichen <code>+</code> und
2805    <code>-</code></p>
2806
2807    <div class="example"><p><code>
2808      &lt;Directory /web/docs&gt;<br />
2809      <span class="indent">
2810        Options Indexes FollowSymLinks<br />
2811      </span>
2812      &lt;/Directory&gt;<br />
2813      <br />
2814      &lt;Directory /web/docs/spec&gt;<br />
2815      <span class="indent">
2816        Options Includes<br />
2817      </span>
2818      &lt;/Directory&gt;
2819    </code></p></div>
2820
2821    <p>f&#252;r das Verzeichnis <code>/web/docs/spec</code> wird jetzt
2822    lediglich <code>Includes</code> gesetzt. Wenn die zweite
2823    <code class="directive">Options</code>-Anweisung jedoch <code>+</code>-
2824    und <code>-</code>-Zeichen verwenden w&#252;rde,</p>
2825
2826    <div class="example"><p><code>
2827      &lt;Directory /web/docs&gt;<br />
2828      <span class="indent">
2829        Options Indexes FollowSymLinks<br />
2830      </span>
2831      &lt;/Directory&gt;<br />
2832      <br />
2833      &lt;Directory /web/docs/spec&gt;<br />
2834      <span class="indent">
2835        Options +Includes -Indexes<br />
2836      </span>
2837      &lt;/Directory&gt;
2838    </code></p></div>
2839
2840    <p>dann w&#252;rden die Optionen <code>FollowSymLinks</code> und
2841    <code>Includes</code> f&#252;r das Verzeichnis <code>/web/docs/spec</code>
2842    gesetzt.</p>
2843
2844    <div class="note"><h3>Anmerkung</h3>
2845      <p>Die Verwendung von <code>-IncludesNOEXEC</code> oder
2846      <code>-Includes</code> deaktiviert Server Side Includes unabh&#228;ngig
2847      von der vorigen Einstellung vollst&#228;ndig.</p>
2848    </div>
2849
2850    <p>Die Voreinstellung ist <code>All</code>, sofern keine anderen Angaben
2851    gemacht wurden.</p>
2852
2853</div>

3224<table class="directive">

3226selbst zu identifizieren</td></tr>
3227<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>ServerName
3228<var>voll-qualifizierter-Domainname</var>[:<var>port</var>]</code></td></tr>

3235</table>
3236    <p>Die Direktive <code class="directive">ServerName</code> bestimmt den
3237    Rechnernamen und Port, den der Server dazu verwendet, sich selbst
3238    zu identifizieren. Diese werden bei der Erstellung von Umleitungs-URLs
3239    ben&#246;tigt. Wenn beispielsweise der Name der Maschine, die den Webserver
3240    beherbergt, <code>simple.example.com</code> lautet, die Maschine jedoch
3241    auch einen DNS-Alias <code>www.example.com</code> besitzt und Sie den
3242    Webserver so identifizieren m&#246;chten, sollten Sie die folgende
3243    Anweisung verwenden:</p>
3244
3245    <div class="example"><p><code>
3246      ServerName www.example.com:80
3247    </code></p></div>
3248
3249    <p>Wenn kein <code class="directive">ServerName</code> angegeben wurde,
3250    dann versucht der Server den Rechnernamen mittels eines Reverse-Lookup
3251    herzuleiten. Wenn kein Port in der
3252    <code class="directive">ServerName</code>-Anweisung angegeben wurde, dann
3253    verwendet der Server den Port der eingegangenen Anfrage. F&#252;r eine
3254    optimale Zuverl&#228;ssigkeit und Berechenbarkeit sollten Sie einen
3255    eindeutigen Rechnernamen und Port angeben, in dem Sie die Direktive
3256    <code class="directive">ServerName</code> verwenden.</p>
3257
3258    <p>Wenn Sie <a href="../vhosts/name-based.html">namensbasierte
3259    virtuelle Hosts</a> verwenden, gibt <code class="directive">ServerName</code>
3260    innerhalb eines <code class="directive"><a href="#virtualhost">&lt;VirtualHost&gt;</a></code>-Abschnitts an, welcher
3261    Hostname im <code>Host:</code>-Header der Anfrage auftauchen muss,
3262    damit sie diesem virtuellen Host zugeordnet wird.</p>
3263
3264    <p>Lesen Sie bitte die Beschreibung der Direktiven <code class="directive"><a href="#usecanonicalname">UseCanonicalName</a></code> und <code class="directive"><a href="#usecanonicalphysicalport">UseCanonicalPhysicalPort</a></code> f&#252;r Einstellungen, die
3265    bestimmen, ob selbstreferenzierende URLs (z.B. vom Modul
3266    <code class="module"><a href="../mod/mod_dir.html">mod_dir</a></code>) auf den angegebenen Port zeigen oder auf die
3267    Portnummern die in der Anfrage des Clients angegeben ist.</p>

3653<div class="directive-section"><h2><a name="UseCanonicalName" id="UseCanonicalName">UseCanonicalName</a>-<a name="usecanonicalname" id="usecanonicalname">Direktive</a></h2>
3654<table class="directive">
3655<tr><th><a href="directive-dict.html#Description">Beschreibung:</a></th><td>Bestimmt, wie der Server seinen eigenen Namen und Port
3656ermittelt</td></tr>
3657<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>UseCanonicalName On|Off|DNS</code></td></tr>
3658<tr><th><a href="directive-dict.html#Default">Voreinstellung:</a></th><td><code>UseCanonicalName Off</code></td></tr>
3659<tr><th><a href="directive-dict.html#Context">Kontext:</a></th><td>Serverkonfiguration, Virtual Host, Verzeichnis</td></tr>
3660<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Core</td></tr>
3661<tr><th><a href="directive-dict.html#Module">Modul:</a></th><td>core</td></tr>
3662</table>
3663    <p>In vielen Situationen muss der Apache eine
3664    <em>selbstreferenzierende</em> URL -- d.h. eine URL, die auf den selben
3665    Server zur&#252;ck verweist -- zusammenbauen. Bei <code>UseCanonicalName
3666    On</code> verwendet der Apache den Hostnamen und Port, der in der
3667    <code class="directive"><a href="#servername">ServerName</a></code>-Anweisung angegeben ist,
3668    um den kanonischen Namen des Servers zu erstellen. Dieser Name wird in
3669    allen selbstreferenzierenden URLs sowie in CGI-Skripten f&#252;r die
3670    Werte von <code>SERVER_NAME</code> und <code>SERVER_PORT</code>
3671    verwendet.</p>
3672
3673    <p>Bei <code>UseCanonicalName Off</code> bildet der Apache
3674    selbstreferenzierende URLs, indem er den vom Client &#252;bermittelten
3675    Hostnamen und Port verwendet, sofern diese vorhanden sind (andernfalls
3676    wird der kanonische Name, wie oben beschrieben, benutzt). Die Werte
3677    sind die gleichen, die zur Anwendung von <a href="../vhosts/name-based.html">namensbasierten virtuellen Hosts</a>
3678    verwendet werden, und sie sind mit den gleichen Clients verf&#252;gbar
3679    <span class="transnote">(<em>Anm.d.&#220;.:</em> , die auch in der Lage sind, auf namensbasierte virtuelle Hosts
3680    zuzugreifen, d.h. einen <code>Host</code>-Header mitschicken)</span>.
3681    Die CGI-Variablen <code>SERVER_NAME</code> und <code>SERVER_PORT</code>
3682    werden ebenfalls aus den vom Client angeboten Werten erstellt.</p>
3683
3684    <p>Ein Intranet-Server, auf den Anwender mit kurzen Namen wie
3685    <code>www</code> zugreifen, ist ein Beispiel, wo dies sinnvoll sein kann.
3686    Sie werden bemerken, dass der Apache den Benutzer auf
3687    <code>http://www.domain.com/splat/</code> umleitet, wenn dieser einen
3688    Kurznamen und eine URL, die einem Verzeichnis entspricht, ohne
3689    abschlie&#223;enden Schr&#228;gstrich eingibt, wie z.B.
3690    <code>http://www/splat</code>. Wenn Sie Authentisierung aktiviert haben,
3691    bewirkt dies, dass der Benutzer sich zweimal identifizieren muss
3692    (einmal f&#252;r <code>www</code> und noch einmal f&#252;r
3693    <code>www.domain.com</code> -- lesen Sie f&#252;r weitere Informationen <a href="http://httpd.apache.org/docs/misc/FAQ.html#prompted-twice">die
3694    FAQ zu diesem Thema</a>). Wenn <code class="directive">UseCanonicalName</code>
3695    jedoch auf <code>Off</code> gesetzt ist, denn wird der Apache zu
3696    <code>http://www/splat/</code> umleiten.</p>
3697
3698    <p>Es existiert noch eine dritte Option, <code>UseCanonicalName DNS</code>,
3699    die f&#252;r den Betrieb von IP-basierten Massen-Virtual-Hosts gedacht ist,
3700    um antiquierte Clients zu unterst&#252;tzen, die keinen
3701    <code>Host:</code>-Header bereit stellen. Um selbstreferenzierende
3702    URLs zu ermitteln, f&#252;hrt der Apache bei dieser Option ein
3703    Reverse-DNS-Lookup auf die IP-Adresse des Servers aus, zu der der Client
3704    Verbindung aufgenommen hat.</p>
3705
3706    <div class="warning"><h3>Warnung</h3>
3707      <p>Wenn CGI-Skripte Vermutungen aufgrund des Wertes von
3708      <code>SERVER_NAME</code> anstellen, k&#246;nnen sie durch diese
3709      Option fehlschlagen. Clients steht es im Wesentlichen frei, einen Wert
3710      f&#252;r den Hostnamen anzugeben, wie er will. Wenn das
3711      CGI-Skript <code>SERVER_NAME</code> jedoch lediglich dazu verwendet,
3712      selbstreferenzierende URLs zu erstellen, sollte das gerade noch
3713      in Ordnung sein.</p>
3714    </div>

3775<table class="directive">
3776<tr><th><a href="directive-dict.html#Description">Beschreibung:</a></th><td>Enth&#228;lt Direktiven, die nur auf bestimmte Hostnamen oder
3777IP-Adressen angewendet werden</td></tr>

3779    <var>Adresse</var>[:<var>Port</var>] [<var>Adresse</var>[:<var>Port</var>]]

3781<tr><th><a href="directive-dict.html#Context">Kontext:</a></th><td>Serverkonfiguration</td></tr>
3782<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Core</td></tr>

3784</table>
3785    <p><code class="directive">&lt;VirtualHost&gt;</code> und
3786    <code>&lt;/VirtualHost&gt;</code> werden dazu verwendet, eine Gruppe
3787    von Direktiven zusammenzufassen, die nur auf einen bestimmten virtuellen
3788    Host angewendet werden. Jede Direktive, die im Virtual-Host-Kontext
3789    zul&#228;ssig ist, kann verwendet werden. Wenn der Server eine Anfrage
3790    f&#252;r ein bestimmtes Dokument eines bestimmten virtuellen Hosts
3791    empf&#228;ngt, dann benutzt er die im
3792    <code class="directive">&lt;VirtualHost&gt;</code>-Container enthaltenen
3793    Konfigurationsanweisungen. <var>Adresse</var> kann sein:</p>
3794
3795    <ul>
3796      <li>Die IP-Adresse des virtuellen Hosts.</li>
3797
3798      <li>Ein voll qualifizierter Domainname f&#252;r die IP-Adresse des
3799      virtuellen Hosts.</li>
3800
3801      <li>Das Zeichen <code>*</code>, welches nur in Kombination mit
3802      <code>NameVirtualHost *</code> verwendet wird, um allen IP-Adressen
3803      zu entsprechen.</li>
3804
3805      <li>Die Zeichenkette <code>_default_</code>, die nur mit IP-basierten
3806      virtuellen Hosts verwendet wird, um nicht zugewiesene IP-Adressen
3807      aufzufangen.</li>
3808    </ul>
3809
3810    <div class="example"><h3>Beispiel</h3><p><code>
3811      &lt;VirtualHost 10.1.2.3&gt;<br />
3812      <span class="indent">
3813        ServerAdmin webmaster@host.foo.com<br />
3814        DocumentRoot /www/docs/host.foo.com<br />
3815        ServerName host.foo.com<br />
3816        ErrorLog logs/host.foo.com-error_log<br />
3817        TransferLog logs/host.foo.com-access_log<br />
3818      </span>
3819      &lt;/VirtualHost&gt;
3820    </code></p></div>
3821
3822    <p>IPv6-Adressen m&#252;ssen in eckigen Klammern angegeben werden, da die
3823    optionale Portnummer sonst nicht erkannt werden kann. Hier ein
3824    IPv6-Beispiel:</p>
3825
3826    <div class="example"><p><code>
3827      &lt;VirtualHost [2001:db8::a00:20ff:fea7:ccea]&gt;<br />
3828      <span class="indent">
3829        ServerAdmin webmaster@host.example.com<br />
3830        DocumentRoot /www/docs/host.example.com<br />
3831        ServerName host.example.com<br />
3832        ErrorLog logs/host.example.com-error_log<br />
3833        TransferLog logs/host.example.com-access_log<br />
3834      </span>
3835      &lt;/VirtualHost&gt;
3836    </code></p></div>
3837
3838    <p>Jeder virtuelle Host muss einer anderen IP-Adresse, einem anderen Port
3839    oder einem anderen Hostnamen f&#252;r den Server entsprechen. Im ersten
3840    Fall muss die Servermaschine so eingerichtet sein, dass sie IP-Pakete
3841    f&#252;r mehrere Adressen akzeptiert. (Wenn der Rechner nicht mehrere
3842    Netzwerkkarten besitzt, kann dies mit dem Befehl <code>ifconfig
3843    alias</code> durchgef&#252;hrt werden -- sofern Ihr Betriebssystem das
3844    unterst&#252;tzt).</p>
3845
3846    <div class="note"><h3>Anmerkung</h3>
3847      <p>Die Verwendung von <code class="directive">&lt;VirtualHost&gt;</code>
3848      beeinflusst <strong>nicht</strong>, an welchen Adressen der Apache
3849      lauscht. Sie m&#252;ssen mit <code class="directive"><a href="../mod/mpm_common.html#listen">Listen</a></code> sicherstellen, dass der Apache
3850      an der richtigen Adresse lauscht.</p>
3851    </div>
3852
3853    <p>Bei der Verwendung IP-basierter virtuellen Hosts kann der spezielle
3854    Name <code>_default_</code> benutzt werden. In diesem Fall weist
3855    der Apache jede IP-Adresse diesem virtuellen Host zu, die nicht explizit in
3856    einem anderen virtuellen Host angegeben ist. Falls kein virtueller Host
3857    <code>_default_</code> angegeben ist, wird die "Hauptserver"-Konfiguration,
3858    die aus allen Definitionen au&#223;erhalb der Virtual-Host-Abschnitte
3859    besteht, f&#252;r nicht passende IPs verwendet. (Beachten Sie jedoch,
3860    dass eine IP-Adressen die zu einer <code class="directive"><a href="#namevirtualhost">NameVirtualHost</a></code>-Anweisung passt, weder den
3861    "Hauptserver" noch den virtuellen Host <code>_default_</code> verwendet.
3862    Lesen Sie f&#252;r weitere Details die Dokumentation zu <a href="../vhosts/name-based.html">namensbasierten virtuell Hosts</a>.)</p>
3863
3864    <p>Sie k&#246;nnen einen speziellen <code>:Port</code> angeben,
3865    um den entsprechenden Port zu wechseln. Falls nicht angegeben, wird
3866    er auf den gleichen Port voreingestellt, wie die letzte
3867    <code class="directive"><a href="../mod/mpm_common.html#listen">Listen</a></code>-Anweisung des
3868    Hauptservers. Sie k&#246;nnen auch <code>:*</code> angeben, um alle
3869    Ports dieser Adresse zu akzeptieren. (Dies wird zusammen mit
3870    <code>_default_</code> empfohlen.)</p>
