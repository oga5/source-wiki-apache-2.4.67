37<h3>Summary</h3>
38
39<p>This module provides SSL v3 and TLS v1.x support for the Apache
40HTTP Server. SSL v2 is no longer supported.</p>
41
42<p>This module relies on <a href="https://www.openssl.org/">OpenSSL</a>
43to provide the cryptography engine.</p>
44
45<p>Further details, discussion, and examples are provided in the
46<a href="../ssl/">SSL documentation</a>.</p>

201<p><var>x509</var> specifies a component of an X.509 DN; one of
202<code>C,ST,L,O,OU,CN,T,I,G,S,D,UID,Email</code>.  In httpd 2.2.0 and
203later, <var>x509</var> may also include a numeric <code>_n</code>
204suffix.  If the DN in question contains multiple attributes of the
205same name, this suffix is used as a zero-based index to select a
206particular attribute.  For example, where the server certificate
207subject DN included two OU attributes, <code>SSL_SERVER_S_DN_OU_0</code>
208and
209<code>SSL_SERVER_S_DN_OU_1</code> could be used to reference each.  A
210variable name without a <code>_n</code> suffix is equivalent to that
211name with a <code>_0</code> suffix; the first (or only) attribute.
212When the environment table is populated using
213the <code>StdEnvVars</code> option of
214the <code class="directive"><a href="#ssloptions">SSLOptions</a></code> directive, the
215first (or only) attribute of any DN is added only under a non-suffixed
216name; i.e. no <code>_0</code> suffixed entries are added.</p>
217
218<p>In httpd 2.4.32 and later, an optional <var>_RAW</var> suffix may be
219added to <var>x509</var> in a DN component, to suppress conversion of
220the attribute value to UTF-8. This must be placed after the index
221suffix (if any). For example, <code>SSL_SERVER_S_DN_OU_RAW</code> or
222<code>SSL_SERVER_S_DN_OU_0_RAW</code> could be used.</p>
223
224<p>The format of the <var>*_DN</var> variables has changed in Apache HTTPD
2252.3.11. See the <code>LegacyDNStringFormat</code> option for
226<code class="directive"><a href="#ssloptions">SSLOptions</a></code> for details.</p>
227
228<p><code>SSL_CLIENT_V_REMAIN</code> is only available in version 2.1
229and later.</p>

630server certificate chain into <code class="directive"><a href="#sslcacertificatepath">SSLCACertificatePath</a></code> has the same effect
631for the certificate chain construction, it has the side-effect that
632client certificates issued by this same CA certificate are also
633accepted on client authentication.</p>

636<em>single</em> RSA <em>or</em> DSA based server certificate. If you are
637using a coupled RSA+DSA certificate pair, this will work only if actually both
638certificates use the <em>same</em> certificate chain. Else the browsers will be
639confused in this situation.</p>

651<div class="directive-section"><h2><a name="SSLCertificateFile" id="SSLCertificateFile">SSLCertificateFile</a> <a name="sslcertificatefile" id="sslcertificatefile">Directive</a></h2>
652<table class="directive">
653<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Server PEM-encoded X.509 certificate data file or token identifier</td></tr>
654<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>SSLCertificateFile <var>file-path</var>|<var>certid</var></code></td></tr>
655<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config, virtual host</td></tr>

660<p>
661This directive points to a file with certificate data in PEM format, or the certificate identifier through a configured cryptographic token.
662If using a PEM file, at minimum, the file must include an end-entity (leaf) certificate.
663The directive can be used multiple times (referencing different filenames)
664to support multiple algorithms for server authentication - typically
665RSA, DSA, and ECC. The number of supported algorithms depends on the
666OpenSSL version being used for mod_ssl: with version 1.0.0 or later,
667<code>openssl list-public-key-algorithms</code> will output a list
668of supported algorithms, see also the note below about limitations
669of OpenSSL versions prior to 1.0.2 and the ways to work around them.
670</p>
671
672<p>
673The files may also include intermediate CA certificates, sorted from
674leaf to root. This is supported with version 2.4.8 and later,
675and obsoletes <code class="directive"><a href="#sslcertificatechainfile">SSLCertificateChainFile</a></code>.
676When running with OpenSSL 1.0.2 or later, this allows
677to configure the intermediate CA chain on a per-certificate basis.
678</p>
679
680<p>
681Custom DH parameters and an EC curve name for ephemeral keys,
682can also be added to end of the first file configured using
683<code class="directive"><a href="#sslcertificatefile">SSLCertificateFile</a></code>.
684This is supported in version 2.4.7 or later.
685Such parameters can be generated using the commands
686<code>openssl dhparam</code> and <code>openssl ecparam</code>.
687The parameters can be added as-is to the end of the first
688certificate file. Only the first file can be used for custom
689parameters, as they are applied independently of the authentication
690algorithm type.
691</p>
692
693<p>
694Finally the end-entity certificate's private key can also be
695added to the certificate file instead of using a separate
696<code class="directive"><a href="#sslcertificatekeyfile">SSLCertificateKeyFile</a></code>
697directive. This practice is highly discouraged. If it is used,
698the certificate files using such an embedded key must be configured
699after the certificates using a separate key file. If the private
700key is encrypted, the pass phrase dialog is forced at startup time.
701</p>
702
703<p>As an alternative to storing certificates and private keys in
704files, a certificate identifier can be used to identify a certificate
705stored in a token.  Currently, only <a href="https://tools.ietf.org/html/rfc7512">PKCS#11 URIs</a> are
706recognized as certificate identifiers, and can be used in conjunction
707with the OpenSSL <code>pkcs11</code> engine or provider.  If <code class="directive"><a href="#sslcertificatekeyfile">SSLCertificateKeyFile</a></code> is omitted, the
708certificate and private key can be loaded through the single
709identifier specified with <code class="directive"><a href="#sslcertificatefile">SSLCertificateFile</a></code>.</p>

711<div class="note">
712<h3>DH parameter interoperability with primes &gt; 1024 bit</h3>
713<p>
714Beginning with version 2.4.7, mod_ssl makes use of
715standardized DH parameters with prime lengths of 2048, 3072 and 4096 bits
716and with additional prime lengths of 6144 and 8192 bits beginning with
717version 2.4.10
718(from <a href="http://www.ietf.org/rfc/rfc3526.txt">RFC 3526</a>), and hands
719them out to clients based on the length of the certificate's RSA/DSA key.
720With Java-based clients in particular (Java 7 or earlier), this may lead
721to handshake failures - see this
722<a href="../ssl/ssl_faq.html#javadh">FAQ answer</a> for working around
723such issues.
724</p>
725</div>
726
727<div class="note">
728<h3>Default DH parameters when using multiple certificates and OpenSSL
729versions prior to 1.0.2</h3>
730<p>
731When using multiple certificates to support different authentication algorithms
732(like RSA, DSA, but mainly ECC) and OpenSSL prior to 1.0.2, it is recommended
733to either use custom DH parameters (preferably) by adding them to the
734first certificate file (as described above), or to order the
735<code class="directive">SSLCertificateFile</code> directives such that RSA/DSA
736certificates are placed <strong>after</strong> the ECC one.
737</p>
738<p>
739This is due to a limitation in older versions of OpenSSL which don't let the
740Apache HTTP Server determine the currently selected certificate at handshake
741time (when the DH parameters must be sent to the peer) but instead always
742provide the last configured certificate. Consequently, the server may select
743default DH parameters based on the length of the wrong certificate's key (ECC
744keys are much smaller than RSA/DSA ones and their length is not relevant for
745selecting DH primes).
746</p>
747<p>
748Since custom DH parameters always take precedence over the default ones, this
749issue can be avoided by creating and configuring them (as described above),
750thus using a custom/suitable length.
751</p>
752</div>

817<div class="directive-section"><h2><a name="SSLCipherSuite" id="SSLCipherSuite">SSLCipherSuite</a> <a name="sslciphersuite" id="sslciphersuite">Directive</a></h2>
818<table class="directive">
819<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Cipher Suite available for negotiation in SSL
820handshake</td></tr>
821<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>SSLCipherSuite [<var>protocol</var>] <var>cipher-spec</var></code></td></tr>
822<tr><th><a href="directive-dict.html#Default">Default:</a></th><td><code>SSLCipherSuite DEFAULT (depends on OpenSSL version)</code></td></tr>
823<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config, virtual host, directory, .htaccess</td></tr>
824<tr><th><a href="directive-dict.html#Override">Override:</a></th><td>AuthConfig</td></tr>
825<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Extension</td></tr>
826<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_ssl</td></tr>
827</table>
828<p>
829This complex directive uses a colon-separated <var>cipher-spec</var> string
830consisting of OpenSSL cipher specifications to configure the Cipher Suite the
831client is permitted to negotiate in the SSL handshake phase. The optional 
832protocol specifier can configure the Cipher Suite for a specific SSL version.
833Possible values include "SSL" for all SSL Protocols up to and including TLSv1.2.
834</p>
835<p>
836Notice that this
837directive can be used both in per-server and per-directory context. 
838In per-server context it applies to the standard SSL handshake when a connection
839is established. In per-directory context it forces a SSL renegotiation with the
840reconfigured Cipher Suite after the HTTP request was read but before the HTTP
841response is sent.</p>
842<p>
843If the SSL library supports TLSv1.3 (OpenSSL 1.1.1 and later), the protocol 
844specifier "TLSv1.3" can be used to configure the cipher suites for that protocol.
845Since TLSv1.3 does not offer renegotiations, specifying ciphers for it in
846a directory context is not allowed.</p>
847<p>
848For a list of TLSv1.3 cipher names, see 
849<a href="https://docs.openssl.org/master/man3/SSL_CTX_set_ciphersuites/">the OpenSSL
850documentation</a>.</p>

853attributes plus a few extra minor ones:</p>
854<ul>

857</li>
858<li><em>Authentication Algorithm</em>:<br />
859    RSA, Diffie-Hellman, DSS, ECDSA, or none.
860</li>
861<li><em>Cipher/Encryption Algorithm</em>:<br />
862    AES, DES, Triple-DES, RC4, RC2, IDEA, etc.
863</li>
864<li><em>MAC Digest Algorithm</em>:<br />
865    MD5, SHA or SHA1, SHA256, SHA384.
866</li>
867</ul>

1366<div class="directive-section"><h2><a name="SSLOptions" id="SSLOptions">SSLOptions</a> <a name="ssloptions" id="ssloptions">Directive</a></h2>
1367<table class="directive">
1368<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Configure various SSL engine run-time options</td></tr>
1369<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>SSLOptions [+|-]<var>option</var> ...</code></td></tr>
1370<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config, virtual host, directory, .htaccess</td></tr>
1371<tr><th><a href="directive-dict.html#Override">Override:</a></th><td>Options</td></tr>
1372<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Extension</td></tr>
1373<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_ssl</td></tr>
1374</table>
1375<p>
1376This directive can be used to control various run-time options on a
1377per-directory basis. Normally, if multiple <code class="directive">SSLOptions</code>
1378could apply to a directory, then the most specific one is taken
1379completely; the options are not merged. However if <em>all</em> the
1380options on the <code class="directive">SSLOptions</code> directive are preceded by a
1381plus (<code>+</code>) or minus (<code>-</code>) symbol, the options
1382are merged. Any options preceded by a <code>+</code> are added to the
1383options currently in force, and any options preceded by a
1384<code>-</code> are removed from the options currently in force.</p>
1385<p>
1386The available <var>option</var>s are:</p>
1387<ul>
1388<li><code>StdEnvVars</code>
1389    <p>
1390    When this option is enabled, the standard set of SSL related CGI/SSI
1391    environment variables are created. This per default is disabled for
1392    performance reasons, because the information extraction step is a
1393    rather expensive operation. So one usually enables this option for
1394    CGI and SSI requests only.</p>
1395</li>
1396<li><code>ExportCertData</code>
1397    <p>
1398    When this option is enabled, additional CGI/SSI environment variables are
1399    created: <code>SSL_SERVER_CERT</code>, <code>SSL_CLIENT_CERT</code> and
1400    <code>SSL_CLIENT_CERT_CHAIN_</code><var>n</var> (with <var>n</var> = 0,1,2,..).
1401    These contain the PEM-encoded X.509 Certificates of server and client for
1402    the current HTTPS connection and can be used by CGI scripts for deeper
1403    Certificate checking. Additionally all other certificates of the client
1404    certificate chain are provided, too. This bloats up the environment a
1405    little bit which is why you have to use this option to enable it on
1406    demand.</p>
1407</li>
1408<li><code>FakeBasicAuth</code>
1409    <p>
1410    When this option is enabled, the Subject Distinguished Name (DN) of the
1411    Client X509 Certificate is translated into a HTTP Basic Authorization
1412    username. This means that the standard Apache authentication methods can
1413    be used for access control. The user name is just the Subject of the
1414    Client's X509 Certificate (can be determined by running OpenSSL's
1415    <code>openssl x509</code> command: <code>openssl x509 -noout -subject -in
1416    </code><var>certificate</var><code>.crt</code>). Note that no password is
1417    obtained from the user. Every entry in the user file needs this password:
1418    ``<code>xxj31ZMTZzkVA</code>'', which is the DES-encrypted version of the
1419    word `<code>password</code>''. Those who live under MD5-based encryption
1420    (for instance under FreeBSD or BSD/OS, etc.) should use the following MD5
1421    hash of the same word: ``<code>$1$OXLyS...$Owx8s2/m9/gfkcRVXzgoE/</code>''.</p>
1422
1423    <p>Note that the <code class="directive"><a href="../mod/mod_auth_basic.html#authbasicfake">AuthBasicFake</a></code>
1424    directive within <code class="module"><a href="../mod/mod_auth_basic.html">mod_auth_basic</a></code> can be used as a more
1425    general mechanism for faking basic authentication, giving control over the
1426    structure of both the username and password.</p>
1427</li>
1428<li><code>StrictRequire</code>
1429    <p>
1430    This <em>forces</em> forbidden access when <code class="directive"><a href="#sslrequiressl">SSLRequireSSL</a></code> or
1431    <code class="directive"><a href="#sslrequire">SSLRequire</a></code> successfully decided that access should be
1432    forbidden. Usually the default is that in the case where a ``<code>Satisfy
1433    any</code>'' directive is used, and other access restrictions are passed,
1434    denial of access due to <code>SSLRequireSSL</code> or
1435    <code>SSLRequire</code> is overridden (because that's how the Apache
1436    <code>Satisfy</code> mechanism should work.) But for strict access restriction
1437    you can use <code>SSLRequireSSL</code> and/or <code>SSLRequire</code> in
1438    combination with an ``<code>SSLOptions +StrictRequire</code>''. Then an
1439    additional ``<code>Satisfy Any</code>'' has no chance once mod_ssl has
1440    decided to deny access.</p>
1441</li>
1442<li><code>OptRenegotiate</code>
1443    <p>
1444    This enables optimized SSL connection renegotiation handling when SSL
1445    directives are used in per-directory context. By default a strict
1446    scheme is enabled where <em>every</em> per-directory reconfiguration of
1447    SSL parameters causes a <em>full</em> SSL renegotiation handshake. When this
1448    option is used mod_ssl tries to avoid unnecessary handshakes by doing more
1449    granular (but still safe) parameter checks. Nevertheless these granular
1450    checks sometimes may not be what the user expects, so enable this on a
1451    per-directory basis only, please.</p>
1452</li>
1453<li><code>LegacyDNStringFormat</code>
1454    <p>
1455    This option influences how values of the
1456    <code>SSL_{CLIENT,SERVER}_{I,S}_DN</code> variables are formatted. Since
1457    version 2.3.11, Apache HTTPD uses a RFC 2253 compatible format by
1458    default. This uses commas as delimiters between the attributes, allows the
1459    use of non-ASCII characters (which are converted to UTF8), escapes
1460    various special characters with backslashes, and sorts the attributes
1461    with the "C" attribute last.</p>
1462
1463    <p>If <code>LegacyDNStringFormat</code> is set, the old format will be
1464    used which sorts the "C" attribute first, uses slashes as separators, and
1465    does not handle non-ASCII and special characters in any consistent way.
1466    </p>
1467</li>
1468</ul>
1469<div class="example"><h3>Example</h3><pre class="prettyprint lang-config">SSLOptions +FakeBasicAuth -StrictRequire
1470&lt;Files ~ "\.(cgi|shtml)$"&gt;
1471    SSLOptions +StdEnvVars -ExportCertData
1472&lt;/Files&gt;</pre>
1473</div>

1478<table class="directive">
1479<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Type of pass phrase dialog for encrypted private
1480keys</td></tr>
1481<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>SSLPassPhraseDialog <var>type</var></code></td></tr>
1482<tr><th><a href="directive-dict.html#Default">Default:</a></th><td><code>SSLPassPhraseDialog builtin</code></td></tr>
1483<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config</td></tr>
1484<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Extension</td></tr>
1485<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_ssl</td></tr>
1486</table>
1487<p>
1488When Apache starts up it has to read the various Certificate (see
1489<code class="directive"><a href="#sslcertificatefile">SSLCertificateFile</a></code>) and
1490Private Key (see <code class="directive"><a href="#sslcertificatekeyfile">SSLCertificateKeyFile</a></code>) files of the
1491SSL-enabled virtual servers. Because for security reasons the Private
1492Key files are usually encrypted, mod_ssl needs to query the
1493administrator for a Pass Phrase in order to decrypt those files. This
1494query can be done in two ways which can be configured by
1495<var>type</var>:</p>
1496<ul>
1497<li><code>builtin</code>
1498    <p>
1499    This is the default where an interactive terminal dialog occurs at startup
1500    time just before Apache detaches from the terminal. Here the administrator
1501    has to manually enter the Pass Phrase for each encrypted Private Key file.
1502    Because a lot of SSL-enabled virtual hosts can be configured, the
1503    following reuse-scheme is used to minimize the dialog: When a Private Key
1504    file is encrypted, all known Pass Phrases (at the beginning there are
1505    none, of course) are tried. If one of those known Pass Phrases succeeds no
1506    dialog pops up for this particular Private Key file. If none succeeded,
1507    another Pass Phrase is queried on the terminal and remembered for the next
1508    round (where it perhaps can be reused).</p>
1509    <p>
1510    This scheme allows mod_ssl to be maximally flexible (because for N encrypted
1511    Private Key files you <em>can</em> use N different Pass Phrases - but then
1512    you have to enter all of them, of course) while minimizing the terminal
1513    dialog (i.e. when you use a single Pass Phrase for all N Private Key files
1514    this Pass Phrase is queried only once).</p></li>
1515
1516<li><code>|/path/to/program [args...]</code>
1517
1518   <p>This mode allows an external program to be used which acts as a
1519   pipe to a particular input device; the program is sent the standard
1520   prompt text used for the <code>builtin</code> mode on
1521   <code>stdin</code>, and is expected to write password strings on
1522   <code>stdout</code>.  If several passwords are needed (or an
1523   incorrect password is entered), additional prompt text will be
1524   written subsequent to the first password being returned, and more
1525   passwords must then be written back.</p></li>
1526
1527<li><code>exec:/path/to/program</code>
1528    <p>
1529    Here an external program is configured which is called at startup for each
1530    encrypted Private Key file. It is called with two arguments (the first is
1531    of the form ``<code>servername:portnumber</code>'', the second is either
1532    ``<code>RSA</code>'', ``<code>DSA</code>'', ``<code>ECC</code>'' or an
1533    integer index starting at 3 if more than three keys are configured), which
1534    indicate for which server and algorithm it has to print the corresponding
1535    Pass Phrase to <code>stdout</code>. In versions 2.4.8 (unreleased)
1536    and 2.4.9, it is called with one argument, a string of the
1537    form ``<code>servername:portnumber:index</code>'' (with <code>index</code>
1538    being a zero-based integer number), which indicate the server, TCP port
1539    and certificate number.  The intent is that this external
1540    program first runs security checks to make sure that the system is not
1541    compromised by an attacker, and only when these checks were passed
1542    successfully it provides the Pass Phrase.</p>
1543    <p>
1544    Both these security checks, and the way the Pass Phrase is determined, can
1545    be as complex as you like. Mod_ssl just defines the interface: an
1546    executable program which provides the Pass Phrase on <code>stdout</code>.
1547    Nothing more or less! So, if you're really paranoid about security, here
1548    is your interface. Anything else has to be left as an exercise to the
1549    administrator, because local security requirements are so different.</p>
1550    <p>
1551    The reuse-algorithm above is used here, too. In other words: The external
1552    program is called only once per unique Pass Phrase.</p></li>
1553</ul>

1604    <p>
1605    This is a shortcut for ``<code>+SSLv3 +TLSv1</code>'' or
1606    - when using OpenSSL 1.0.1 and later -

1608    (except for OpenSSL versions compiled with the ``no-ssl3'' configuration

1613<div class="note">
1614<h3><code class="directive">SSLProtocol</code> for name-based virtual hosts</h3>
1615<p>
1616Before OpenSSL 1.1.1, even though the Server Name Indication (SNI) allowed to
1617determine the targeted virtual host early in the TLS handshake, it was not
1618possible to switch the TLS protocol version of the connection at this point,
1619and thus the <code class="directive">SSLProtocol</code> negotiated was always based off
1620the one of the <em>base virtual host</em> (first virtual host declared on the
1621listening <code>IP:port</code> of the connection).
1622</p>
1623<p>
1624Beginning with Apache HTTP server version 2.4.42, when built/linked against
1625OpenSSL 1.1.1 or later, and when the SNI is provided by the client in the TLS
1626handshake, the <code class="directive">SSLProtocol</code> of each (name-based) virtual
1627host can and will be honored.
1628</p>
1629<p>
1630For compatibility with previous versions, if no
1631<code class="directive">SSLProtocol</code> is configured in a name-based virtual host,
1632the one from the base virtual host still applies, <strong>unless</strong>
1633<code class="directive">SSLProtocol</code> is configured globally in which case the
1634global value applies (this latter exception is more sensible than compatible,
1635though).
1636</p>

1973additionally to <code>SSLProxyMachineCertificatePath</code>.  The referenced file can contain any number of pairs of client
1974certificate and associated private key.  Each pair can be specified in
1975either (certificate, key) or (key, certificate) order.  Non-leaf (CA) certificates can
1976also be included in the file, and are treated as if configured with  <code class="directive"><a href="#sslproxymachinecertificatechainfile">SSLProxyMachineCertificateChainFile</a></code>.</p>
1977
1978<p>When challenged to provide a client certificate by a remote server,
1979the server should provide a list of <em>acceptable certificate
1980authority names</em> in the challenge.  If such a list is <em>not</em>
1981provided, <code class="module"><a href="../mod/mod_ssl.html">mod_ssl</a></code> will use the first configured
1982client cert/key.  If a list of CA names <em>is</em> provided,
1983<code class="module"><a href="../mod/mod_ssl.html">mod_ssl</a></code> will iterate through that list, and attempt
1984to find a configured client cert which was issued either directly by
1985that CA, or indirectly via any number of intermediary CA certificates.
1986The chain of intermediate CA certificates can be built from those
1987included in the file, or configured with <code class="directive"><a href="#sslproxymachinecertificatechainfile">SSLProxyMachineCertificateChainFile</a></code>.  The
1988first configured matching certificate will then be supplied in
1989response to the challenge.</p>
1990
1991<p>If the list of CA names <em>is</em> provided by the remote server,
1992and <em>no</em> matching client certificate can be found, no client
1993certificate will be provided by <code class="module"><a href="../mod/mod_ssl.html">mod_ssl</a></code>, which will
1994likely fail the SSL/TLS handshake (depending on the remote server
1995configuration).</p>

1999</div>

2003"<code>-----BEGIN PRIVATE KEY-----</code>",
2004must be converted, eg. using
2005"<code>openssl rsa -in private-pkcs8.pem -outform pem</code>".</p>

2131<table class="directive">
2132<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Pseudo Random Number Generator (PRNG) seeding

2134<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>SSLRandomSeed <var>context</var> <var>source</var>

2138<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_ssl</td></tr>
2139</table>
2140<p>
2141This configures one or more sources for seeding the Pseudo Random Number
2142Generator (PRNG) in OpenSSL at startup time (<var>context</var> is
2143<code>startup</code>) and/or just before a new SSL connection is established
2144(<var>context</var> is <code>connect</code>). This directive can only be used
2145in the global server context because the PRNG is a global facility.</p>
2146<p>
2147The following <var>source</var> variants are available:</p>
2148<ul>
2149<li><code>builtin</code>
2150    <p> This is the always available builtin seeding source. Its usage
2151    consumes minimum CPU cycles under runtime and hence can be always used
2152    without drawbacks. The source used for seeding the PRNG contains of the
2153    current time, the current process id and a randomly
2154    chosen 128 bytes extract of the stack.
2155    The drawback is that this is not really a strong source and at startup
2156    time (where the scoreboard is still not available) this source just
2157    produces a few bytes of entropy. So you should always, at least for the
2158    startup, use an additional seeding source.</p></li>
2159<li><code>file:/path/to/source</code>
2160    <p>
2161    This variant uses an external file <code>/path/to/source</code> as the
2162    source for seeding the PRNG. When <var>bytes</var> is specified, only the
2163    first <var>bytes</var> number of bytes of the file form the entropy (and
2164    <var>bytes</var> is given to <code>/path/to/source</code> as the first
2165    argument). When <var>bytes</var> is not specified the whole file forms the
2166    entropy (and <code>0</code> is given to <code>/path/to/source</code> as
2167    the first argument). Use this especially at startup time, for instance
2168    with an available <code>/dev/random</code> and/or
2169    <code>/dev/urandom</code> devices (which usually exist on modern Unix
2170    derivatives like FreeBSD and Linux).</p>
2171    <p>
2172    <em>But be careful</em>: Usually <code>/dev/random</code> provides only as
2173    much entropy data as it actually has, i.e. when you request 512 bytes of
2174    entropy, but the device currently has only 100 bytes available two things
2175    can happen: On some platforms you receive only the 100 bytes while on
2176    other platforms the read blocks until enough bytes are available (which
2177    can take a long time). Here using an existing <code>/dev/urandom</code> is
2178    better, because it never blocks and actually gives the amount of requested
2179    data. The drawback is just that the quality of the received data may not
2180    be the best.</p></li>
2181
2182<li><code>exec:/path/to/program</code>
2183    <p>
2184    This variant uses an external executable
2185    <code>/path/to/program</code> as the source for seeding the
2186    PRNG. When <var>bytes</var> is specified, only the first
2187    <var>bytes</var> number of bytes of its <code>stdout</code> contents
2188    form the entropy. When <var>bytes</var> is not specified, the
2189    entirety of the data produced on <code>stdout</code> form the
2190    entropy. Use this only at startup time when you need a very strong
2191    seeding with the help of an external program (for instance as in
2192    the example above with the <code>truerand</code> utility you can
2193    find in the mod_ssl distribution which is based on the AT&amp;T
2194    <em>truerand</em> library). Using this in the connection context
2195    slows down the server too dramatically, of course.  So usually you
2196    should avoid using external programs in that context.</p></li>
2197<li><code>egd:/path/to/egd-socket</code> (Unix only)
2198    <p>
2199    This variant uses the Unix domain socket of the
2200    external Entropy Gathering Daemon (EGD) (see <a href="http://www.lothar.com/tech/crypto/">http://www.lothar.com/tech
2201    /crypto/</a>) to seed the PRNG. Use this if no random device exists
2202    on your platform.</p></li>
2203</ul>

2208SSLRandomSeed connect builtin

2255<div class="note"><h3>SSLRequire is deprecated</h3>
2256<p><code>SSLRequire</code> is deprecated and should in general be replaced
2257by <a href="mod_authz_core.html#reqexpr">Require expr</a>. The so called
2258<a href="../expr.html">ap_expr</a> syntax of <code>Require expr</code> is
2259a superset of the syntax of <code>SSLRequire</code>, with the following
2260exception:</p>
2261
2262<p>In <code>SSLRequire</code>, the comparison operators <code>&lt;</code>,
2263<code>&lt;=</code>, ... are completely equivalent to the operators
2264<code>lt</code>, <code>le</code>, ... and work in a somewhat peculiar way that
2265first compares the length of two strings and then the lexical order.
2266On the other hand, <a href="../expr.html">ap_expr</a> has two sets of
2267comparison operators: The operators <code>&lt;</code>,
2268<code>&lt;=</code>, ... do lexical string comparison, while the operators
2269<code>-lt</code>, <code>-le</code>, ... do integer comparison.
2270For the latter, there are also aliases without the leading dashes:
2271<code>lt</code>, <code>le</code>, ...
2272</p>
2273
2274</div>

2276<p>
2277This directive specifies a general access requirement which has to be
2278fulfilled in order to allow access. It is a very powerful directive because the
2279requirement specification is an arbitrarily complex boolean expression
2280containing any number of access checks.</p>
2281<p>
2282The <var>expression</var> must match the following syntax (given as a BNF
2283grammar notation):</p>
2284<blockquote>
2285<pre>expr     ::= "<strong>true</strong>" | "<strong>false</strong>"
2286           | "<strong>!</strong>" expr
2287           | expr "<strong>&amp;&amp;</strong>" expr
2288           | expr "<strong>||</strong>" expr
2289           | "<strong>(</strong>" expr "<strong>)</strong>"
2290           | comp
2291
2292comp     ::= word "<strong>==</strong>" word | word "<strong>eq</strong>" word
2293           | word "<strong>!=</strong>" word | word "<strong>ne</strong>" word
2294           | word "<strong>&lt;</strong>"  word | word "<strong>lt</strong>" word
2295           | word "<strong>&lt;=</strong>" word | word "<strong>le</strong>" word
2296           | word "<strong>&gt;</strong>"  word | word "<strong>gt</strong>" word
2297           | word "<strong>&gt;=</strong>" word | word "<strong>ge</strong>" word
2298           | word "<strong>in</strong>" "<strong>{</strong>" wordlist "<strong>}</strong>"
2299           | word "<strong>in</strong>" "<strong>PeerExtList(</strong>" word "<strong>)</strong>"
2300           | word "<strong>=~</strong>" regex
2301           | word "<strong>!~</strong>" regex
2302
2303wordlist ::= word
2304           | wordlist "<strong>,</strong>" word
2305
2306word     ::= digit
2307           | cstring
2308           | variable
2309           | function
2310
2311digit    ::= [0-9]+
2312cstring  ::= "..."
2313variable ::= "<strong>%{</strong>" varname "<strong>}</strong>"
2314function ::= funcname "<strong>(</strong>" funcargs "<strong>)</strong>"</pre>
2315</blockquote>
2316<p>For <code>varname</code> any of the variables described in <a href="#envvars">Environment Variables</a> can be used.  For
2317<code>funcname</code> the available functions are listed in
2318the <a href="../expr.html#functions">ap_expr documentation</a>.</p>

2320<p>The <var>expression</var> is parsed into an internal machine
2321representation when the configuration is loaded, and then evaluated
2322during request processing.  In .htaccess context, the <var>expression</var> is
2323both parsed and executed each time the .htaccess file is encountered during
2324request processing.</p>

2397<div class="directive-section"><h2><a name="SSLSessionCache" id="SSLSessionCache">SSLSessionCache</a> <a name="sslsessioncache" id="sslsessioncache">Directive</a></h2>
2398<table class="directive">
2399<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Type of the global/inter-process SSL Session
2400Cache</td></tr>
2401<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>SSLSessionCache <var>type</var></code></td></tr>
2402<tr><th><a href="directive-dict.html#Default">Default:</a></th><td><code>SSLSessionCache none</code></td></tr>
2403<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config</td></tr>
2404<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Extension</td></tr>
2405<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_ssl</td></tr>
2406</table>
2407<p>
2408This configures the storage type of the global/inter-process SSL Session
2409Cache. This cache is an optional facility which speeds up parallel request
2410processing. For requests to the same server process (via HTTP keep-alive),
2411OpenSSL already caches the SSL session information locally. But because modern
2412clients request inlined images and other data via parallel requests (usually
2413up to four parallel requests are common) those requests are served by
2414<em>different</em> pre-forked server processes. Here an inter-process cache
2415helps to avoid unnecessary session handshakes.</p>
2416<p>
2417The following five storage <var>type</var>s are currently supported:</p>
2418<ul>
2419<li><code>none</code>
2420
2421    <p>This disables the global/inter-process Session Cache.  This
2422    will incur a noticeable speed penalty and may cause problems if
2423    using certain browsers, particularly if client certificates are
2424    enabled.  This setting is not recommended.</p></li>
2425
2426<li><code>nonenotnull</code>
2427
2428    <p>This disables any global/inter-process Session Cache.  However
2429    it does force OpenSSL to send a non-null session ID to
2430    accommodate buggy clients that require one.</p></li>
2431
2432<li><code>dbm:/path/to/datafile</code>
2433
2434    <p>This makes use of a DBM hashfile on the local disk to
2435    synchronize the local OpenSSL memory caches of the server
2436    processes. This session cache may suffer reliability issues under
2437    high load. To use this, ensure that
2438    <code class="module"><a href="../mod/mod_socache_dbm.html">mod_socache_dbm</a></code> is loaded.</p></li>
2439
2440<li><code>shmcb:/path/to/datafile</code>[<code>(</code><var>size</var><code>)</code>]
2441
2442    <p>This makes use of a high-performance cyclic buffer
2443    (approx. <var>size</var> bytes in size) inside a shared memory
2444    segment in RAM (established via <code>/path/to/datafile</code>) to
2445    synchronize the local OpenSSL memory caches of the server
2446    processes.  This is the recommended session cache. To use this,
2447    ensure that <code class="module"><a href="../mod/mod_socache_shmcb.html">mod_socache_shmcb</a></code> is loaded.</p></li>
2448
2449<li><code>dc:UNIX:/path/to/socket</code>
2450
2451    <p>This makes use of the <a href="http://distcache.sourceforge.net/">distcache</a> distributed session
2452    caching libraries.  The argument should specify the location of
2453    the server or proxy to be used using the distcache address syntax;
2454    for example, <code>UNIX:/path/to/socket</code> specifies a UNIX
2455    domain socket (typically a local dc_client proxy);
2456    <code>IP:server.example.com:9001</code> specifies an IP
2457    address. To use this, ensure that
2458    <code class="module"><a href="../mod/mod_socache_dc.html">mod_socache_dc</a></code> is loaded.</p></li>
2459
2460</ul>
2461
2462<div class="example"><h3>Examples</h3><pre class="prettyprint lang-config">SSLSessionCache "dbm:/usr/local/apache/logs/ssl_gcache_data"
2463SSLSessionCache "shmcb:/usr/local/apache/logs/ssl_gcache_data(512000)"</pre>
2464</div>
2465
2466<p>The <code>ssl-cache</code> mutex is used to serialize access to
2467the session cache to prevent corruption.  This mutex can be configured
2468using the <code class="directive"><a href="../mod/core.html#mutex">Mutex</a></code> directive.</p>

2825<div class="directive-section"><h2><a name="SSLUseStapling" id="SSLUseStapling">SSLUseStapling</a> <a name="sslusestapling" id="sslusestapling">Directive</a></h2>
2826<table class="directive">
2827<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Enable stapling of OCSP responses in the TLS handshake</td></tr>
2828<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>SSLUseStapling on|off</code></td></tr>
2829<tr><th><a href="directive-dict.html#Default">Default:</a></th><td><code>SSLUseStapling off</code></td></tr>
2830<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config, virtual host</td></tr>
2831<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Extension</td></tr>
2832<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_ssl</td></tr>
2833<tr><th><a href="directive-dict.html#Compatibility">Compatibility:</a></th><td>Available if using OpenSSL 0.9.8h or later</td></tr>
2834</table>
2835<p>This option enables OCSP stapling, as defined by the "Certificate
2836Status Request" TLS extension specified in RFC 6066. If enabled (and
2837requested by the client), mod_ssl will include an OCSP response
2838for its own certificate in the TLS handshake. Configuring an
2839<code class="directive"><a href="#sslstaplingcache">SSLStaplingCache</a></code> is a
2840prerequisite for enabling OCSP stapling.</p>
2841
2842<p>OCSP stapling relieves the client of querying the OCSP responder
2843on its own, but it should be noted that with the RFC 6066 specification,
2844the server's <code>CertificateStatus</code> reply may only include an
2845OCSP response for a single cert. For server certificates with intermediate
2846CA certificates in their chain (the typical case nowadays),
2847stapling in its current implementation therefore only partially achieves the
2848stated goal of "saving roundtrips and resources" - see also
2849<a href="http://www.ietf.org/rfc/rfc6961.txt">RFC 6961</a>
2850(TLS Multiple Certificate Status Extension).
2851</p>
2852
2853<p>When OCSP stapling is enabled, the <code>ssl-stapling</code> mutex is used
2854to control access to the OCSP stapling cache in order to prevent corruption,
2855and the <code>sss-stapling-refresh</code> mutex is used to control refreshes
2856of OCSP responses.  These mutexes can be configured using the
2857<code class="directive"><a href="../mod/core.html#mutex">Mutex</a></code> directive.
2858</p>
2859

2865<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Type of Client Certificate verification</td></tr>
2866<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>SSLVerifyClient <var>level</var></code></td></tr>

2875Authentication. Notice that this directive can be used both in per-server and

2900<div class="directive-section"><h2><a name="SSLVerifyDepth" id="SSLVerifyDepth">SSLVerifyDepth</a> <a name="sslverifydepth" id="sslverifydepth">Directive</a></h2>
2901<table class="directive">
2902<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Maximum depth of CA Certificates in Client
2903Certificate verification</td></tr>
2904<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>SSLVerifyDepth <var>number</var></code></td></tr>
2905<tr><th><a href="directive-dict.html#Default">Default:</a></th><td><code>SSLVerifyDepth 1</code></td></tr>
2906<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config, virtual host, directory, .htaccess</td></tr>
2907<tr><th><a href="directive-dict.html#Override">Override:</a></th><td>AuthConfig</td></tr>
2908<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Extension</td></tr>
2909<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_ssl</td></tr>

2911<p>
2912This directive sets how deeply mod_ssl should verify before deciding that the
2913clients don't have a valid certificate. Notice that this directive can be
2914used both in per-server and per-directory context. In per-server context it
2915applies to the client authentication process used in the standard SSL
2916handshake when a connection is established. In per-directory context it forces

2924certificate can be self-signed or has to be signed by a CA which is directly
2925known to the server (i.e. the CA's certificate is under
2926<code class="directive"><a href="#sslcacertificatepath">SSLCACertificatePath</a></code>), etc.</p>

2933<table class="directive">
2934<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Set compatibility policy for SNI client access to virtual hosts.</td></tr>
2935<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>SSLVHostSNIPolicy strict|secure|authonly|insecure</code></td></tr>
2936<tr><th><a href="directive-dict.html#Default">Default:</a></th><td><code>SSLVHostSNIPolicy secure</code></td></tr>
2937<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config</td></tr>
2938<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Extension</td></tr>
2939<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_ssl</td></tr>
2940<tr><th><a href="directive-dict.html#Compatibility">Compatibility:</a></th><td>Available in httpd 2.4.66 and later</td></tr>
2941</table><p>This directive sets the policy applied when checking whether the
2942<code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code>
2943identified by the <code>Host</code> request header in an HTTP request
2944is compatible with the <code class="directive"><a href="../mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code> identified from the SNI
2945extension sent during the initial TLS connection handshake. If an HTTP
2946request is associated with a virtual host which has an incompatible
2947SSL/TLS configuration under the policy used, an HTTP error response
2948with status code 421 ("Misdirected Request") will be sent.</p>
2949
2950<p>The policy also applies to TLS connections where an SNI extension
2951is not sent during the handshake, implicitly using the default or
2952first virtual host definition. If the Host header in an HTTP request
2953on such a connection identifies any other non-default virtual host,
2954the compatibility policy is tested.</p>
2955
2956<p>The <code>strict</code> policy blocks all HTTP requests which are
2957identified with a different virtual host to that identifed by SNI.
2958The <code>insecure</code> policy allows all HTTP requests regardless
2959of virtual host identified; such a configuration may be vulnerable to
2960<a href="https://httpd.apache.org/security/vulnerabilities_24.html">CVE-2025-23048</a>.
2961</p>
2962
2963<p>The (default) <code>secure</code>, and <code>authonly</code>
2964policies compare specific aspects of the SSL configuration for the two
2965virtual hosts, which are grouped into two categories:</p>
2966
2967<ul>
2968  <li><strong>server certificate/key, or protocol/cipher
2969  restrictions</strong>: directives which determine the server
2970  certificate or key (<code class="directive"><a href="#sslcertificatekeyfile">SSLCertificateKeyFile</a></code> etc), cipher or
2971  protocol restrictions (<code class="directive"><a href="#sslciphersuite">SSLCipherSuite</a></code> and <code class="directive"><a href="#sslprotocol">SSLProtocol</a></code>)</li>
2972
2973  <li><strong>client vertification and authentication
2974  settings</strong>: directives which affect TLS client certificate
2975  verification or authentication, such as <code class="directive"><a href="#sslverifyclient">SSLVerifyClient</a></code>, <code class="directive"><a href="#sslverifymode">SSLVerifyMode</a></code>, <code class="directive"><a href="#sslcacertificatepath">SSLCACertificatePath</a></code>, <code class="directive"><a href="#sslsrpverifierfile">SSLSRPVerifierFile</a></code>; any use of <code class="directive"><a href="#sslopensslconfcmd">SSLOpenSSLConfCmd</a></code></li>
2976</ul>

2980under each different policy setting:</p>
