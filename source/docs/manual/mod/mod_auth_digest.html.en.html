38<h3>Summary</h3>
39
40    <p>This module implements HTTP Digest Authentication
41    (<a href="http://www.faqs.org/rfcs/rfc2617.html">RFC2617</a>), and
42    provides an alternative to <code class="module"><a href="../mod/mod_auth_basic.html">mod_auth_basic</a></code> where the
43    password is not transmitted as cleartext. However, this does
44    <strong>not</strong> lead to a significant security advantage over
45    basic authentication. On the other hand, the password storage on the
46    server is much less secure with digest authentication than with
47    basic authentication. Therefore, using basic auth and encrypting the
48    whole connection using <code class="module"><a href="../mod/mod_ssl.html">mod_ssl</a></code> is a much better
49    alternative.</p>

72<h2><a name="using" id="using">Using Digest Authentication</a></h2>
73
74    <p>To use MD5 Digest authentication, configure the location to be
75    protected as shown in the below example:</p>
76
77    <div class="example"><h3>Example:</h3><pre class="prettyprint lang-config">&lt;Location "/private/"&gt;
78    AuthType Digest
79    AuthName "private area"
80    AuthDigestDomain "/private/" "http://mirror.my.dom/private2/"
81    
82    AuthDigestProvider file
83    AuthUserFile "/web/auth/.digest_pw"
84    Require valid-user
85&lt;/Location&gt;</pre>
86</div>
87
88    <p><code class="directive"><a href="#authdigestdomain">AuthDigestDomain</a></code>
89    should list the locations that will be protected by this
90    configuration.</p>
91
92    <p>The password file referenced in the <code class="directive"><a href="#authuserfile">AuthUserFile</a></code> directive may be
93    created and managed using the <code class="program"><a href="../programs/htdigest.html">htdigest</a></code> tool.</p>
94
95
96    <div class="note"><h3>Note</h3>
97    <p>Digest authentication was intended to be more secure than basic
98    authentication, but no longer fulfills that design goal. A
99    man-in-the-middle attacker can trivially force the browser to downgrade
100    to basic authentication. And even a passive eavesdropper can brute-force
101    the password using today's graphics hardware, because the hashing
102    algorithm used by digest authentication is too fast. Another problem is
103    that the storage of the passwords on the server is insecure. The contents
104    of a stolen htdigest file can be used directly for digest authentication.
105    Therefore using <code class="module"><a href="../mod/mod_ssl.html">mod_ssl</a></code> to encrypt the whole connection is
106    strongly recommended.</p>
107    <p><code class="module"><a href="../mod/mod_auth_digest.html">mod_auth_digest</a></code> only works properly on platforms
108      where APR supports shared memory.</p>
109    </div>

134<div class="directive-section"><h2><a name="AuthDigestDomain" id="AuthDigestDomain">AuthDigestDomain</a> <a name="authdigestdomain" id="authdigestdomain">Directive</a></h2>
135<table class="directive">
136<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>URIs that are in the same protection space for digest
137authentication</td></tr>
138<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>AuthDigestDomain <var>URI</var> [<var>URI</var>] ...</code></td></tr>
139<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>directory, .htaccess</td></tr>
140<tr><th><a href="directive-dict.html#Override">Override:</a></th><td>AuthConfig</td></tr>
141<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Extension</td></tr>
142<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_auth_digest</td></tr>
143</table>
144    <p>The <code class="directive">AuthDigestDomain</code> directive allows
145    you to specify one or more URIs which are in the same protection
146    space (<em>i.e.</em> use the same realm and username/password info).
147    The specified URIs are prefixes; the client will assume
148    that all URIs "below" these are also protected by the same
149    username/password. The URIs may be either absolute URIs (<em>i.e.</em>
150    including a scheme, host, port, etc.) or relative URIs.</p>
151
152    <p>This directive <em>should</em> always be specified and
153    contain at least the (set of) root URI(s) for this space.
154    Omitting to do so will cause the client to send the
155    Authorization header for <em>every request</em> sent to this
156    server.</p>
157
158    <p>The URIs specified can also point to different servers, in
159    which case clients (which understand this) will then share
160    username/password info across multiple servers without
161    prompting the user each time. </p>
162

165<div class="directive-section"><h2><a name="AuthDigestNonceLifetime" id="AuthDigestNonceLifetime">AuthDigestNonceLifetime</a> <a name="authdigestnoncelifetime" id="authdigestnoncelifetime">Directive</a></h2>
166<table class="directive">
167<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>How long the server nonce is valid</td></tr>
168<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>AuthDigestNonceLifetime <var>seconds</var></code></td></tr>
169<tr><th><a href="directive-dict.html#Default">Default:</a></th><td><code>AuthDigestNonceLifetime 300</code></td></tr>
170<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>directory, .htaccess</td></tr>
171<tr><th><a href="directive-dict.html#Override">Override:</a></th><td>AuthConfig</td></tr>
172<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Extension</td></tr>
173<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_auth_digest</td></tr>
174</table>
175    <p>The <code class="directive">AuthDigestNonceLifetime</code> directive
176    controls how long the server nonce is valid. When the client
177    contacts the server using an expired nonce the server will send
178    back a 401 with <code>stale=true</code>. If <var>seconds</var> is
179    greater than 0 then it specifies the amount of time for which the
180    nonce is valid; this should probably never be set to less than 10
181    seconds. If <var>seconds</var> is less than 0 then the nonce never
182    expires. 
183    </p>

210<div class="directive-section"><h2><a name="AuthDigestQop" id="AuthDigestQop">AuthDigestQop</a> <a name="authdigestqop" id="authdigestqop">Directive</a></h2>
211<table class="directive">
212<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>Determines the quality-of-protection to use in digest
213authentication</td></tr>
214<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>AuthDigestQop none|auth|auth-int [auth|auth-int]</code></td></tr>
215<tr><th><a href="directive-dict.html#Default">Default:</a></th><td><code>AuthDigestQop auth</code></td></tr>
216<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>directory, .htaccess</td></tr>
217<tr><th><a href="directive-dict.html#Override">Override:</a></th><td>AuthConfig</td></tr>
218<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Extension</td></tr>
219<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_auth_digest</td></tr>
220</table>
221    <p>The <code class="directive">AuthDigestQop</code> directive determines
222    the <dfn>quality-of-protection</dfn> to use. <code>auth</code> will
223    only do authentication (username/password); <code>auth-int</code> is
224    authentication plus integrity checking (an MD5 hash of the entity
225    is also computed and checked); <code>none</code> will cause the module
226    to use the old RFC-2069 digest algorithm (which does not include
227    integrity checking). Both <code>auth</code> and <code>auth-int</code> may
228    be specified, in which the case the browser will choose which of
229    these to use. <code>none</code> should only be used if the browser for
230    some reason does not like the challenge it receives otherwise.</p>
231
232    <div class="note">
233      <code>auth-int</code> is not implemented yet.
234    </div>

238<div class="directive-section"><h2><a name="AuthDigestShmemSize" id="AuthDigestShmemSize">AuthDigestShmemSize</a> <a name="authdigestshmemsize" id="authdigestshmemsize">Directive</a></h2>
239<table class="directive">
240<tr><th><a href="directive-dict.html#Description">Description:</a></th><td>The amount of shared memory to allocate for keeping track
241of clients</td></tr>
242<tr><th><a href="directive-dict.html#Syntax">Syntax:</a></th><td><code>AuthDigestShmemSize <var>size</var></code></td></tr>
243<tr><th><a href="directive-dict.html#Default">Default:</a></th><td><code>AuthDigestShmemSize 1000</code></td></tr>
244<tr><th><a href="directive-dict.html#Context">Context:</a></th><td>server config</td></tr>
245<tr><th><a href="directive-dict.html#Status">Status:</a></th><td>Extension</td></tr>
246<tr><th><a href="directive-dict.html#Module">Module:</a></th><td>mod_auth_digest</td></tr>
247</table>
248    <p>The <code class="directive">AuthDigestShmemSize</code> directive defines
249    the amount of shared memory, that will be allocated at the server
250    startup for keeping track of clients. Note that the shared memory
251    segment cannot be set less than the space that is necessary for
252    tracking at least <em>one</em> client. This value is dependent on your
253    system. If you want to find out the exact value, you may simply
254    set <code class="directive">AuthDigestShmemSize</code> to the value of
255    <code>0</code> and read the error message after trying to start the
256    server.</p>
257
258    <p>The <var>size</var> is normally expressed in Bytes, but you
259    may follow the number with a <code>K</code> or an <code>M</code> to
260    express your value as KBytes or MBytes. For example, the following
261    directives are all equivalent:</p>
262
263<pre class="prettyprint lang-config">AuthDigestShmemSize 1048576
264AuthDigestShmemSize 1024K
265AuthDigestShmemSize 1M</pre>
