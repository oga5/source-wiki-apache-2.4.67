41<h3>概要</h3>
42
43    <div class="warning"><h3>警告</h3>
44      <p><a href="#access">サーバを安全にする</a>まで <code class="directive"><a href="#proxyrequests">ProxyRequests</a></code> は有効にしないでください。
45      オープンプロキシサーバはあなた自身のネットワークにとっても、
46      インターネット全体にとっても危険です。</p>
47    </div>
48
49    <p>このモジュールは Apache のプロキシ/ゲートウェイ機能を実装しています。
50    <code>AJP13</code> (Apache JServe Protocol version 1.3),
51    <code>FTP</code>, <code>CONNECT</code> (SSL 用),
52    <code>HTTP/0.9</code>, <code>HTTP/1.0</code>, <code>HTTP/1.1</code>
53    のプロキシ機能を実装しています。これらのプロトコルやその他のプロトコル用の
54    プロキシ機能を持った、他のモジュールに接続するようにも設定できます。</p>
55
56    <p>Apache のプロキシ機能は <code class="module"><a href="../mod/mod_proxy.html">mod_proxy</a></code> の他に、
57    いくつかのモジュールに分割されています:
58    <code class="module"><a href="../mod/mod_proxy_http.html">mod_proxy_http</a></code>, <code class="module"><a href="../mod/mod_proxy_ftp.html">mod_proxy_ftp</a></code>,
59    <code class="module"><a href="../mod/mod_proxy_ajp.html">mod_proxy_ajp</a></code>, <code class="module"><a href="../mod/mod_proxy_balancer.html">mod_proxy_balancer</a></code>, 
60    <code class="module"><a href="../mod/mod_proxy_connect.html">mod_proxy_connect</a></code> です。ですから、
61    特定のプロキシの機能を使いたい場合は、<code class="module"><a href="../mod/mod_proxy.html">mod_proxy</a></code> <em>と</em>
62    該当するモジュールをサーバに (コンパイル時に静的に行なうか
63    <code class="directive"><a href="../mod/mod_so.html#loadmodule">LoadModule</a></code> で動的に読み込むかして) 
64    組み込む必要があります。</p>
65
66    <p>これに加えて、他のモジュールによって拡張機能が提供されています。
67    キャッシュは <code class="module"><a href="../mod/mod_cache.html">mod_cache</a></code> と関連モジュールで
68    提供されています。SSL/TLS で遠隔サーバに接続する機能は
69    <code class="module"><a href="../mod/mod_ssl.html">mod_ssl</a></code> の <code>SSLProxy*</code> ディレクティブで
70    提供されています。これらの機能を利用するためには、該当するモジュールを
71    組み込んで設定しなければなりません。</p>
72</div>

128<h2><a name="forwardreverse" id="forwardreverse">フォワードプロキシとリバースプロキシ</a></h2>
129      <p>Apache は<dfn>フォワード</dfn>プロキシとしても、
130      <dfn>リバース</dfn>プロキシとしても設定することができます。</p>
131
132      <p>通常の<dfn>フォワードプロキシ</dfn>はクライアントと
133      <em>オリジンサーバ</em> <span class="transnote">(<em>訳注:</em> コンテンツ生成元のサーバ)</span>
134      の間に位置する中間サーバです。
135      オリジンサーバからコンテンツを取得する過程では、クライアントは
136      行き先としてオリジンサーバを指定しつつプロキシにリクエストを送り、
137      プロキシはオリジンサーバからコンテンツ取得のリクエストを送り、
138      コンテンツが取得できればそれをクライアントに返します。
139      クライアントが他のサイトにフォワードプロクシ経由でアクセスするには、
140      特別にそれ用の設定をしなければなりません。</p>
141
142      <p>フォワードプロキシの一般的な使用方法は、ファイアウォールによって
143      制限されている内部のクライアントにインターネットへのアクセスを
144      提供するものです。フォワードプロキシはネットワークの使用量を
145      減らすために (<code class="module"><a href="../mod/mod_cache.html">mod_cache</a></code> で提供されている) 
146      キャッシュ機能を用いることもできます。</p>
147
148      <p>フォワードプロキシは <code class="directive"><a href="#proxyrequests">ProxyRequests</a></code> ディレクティブで
149      有効になります。フォワードプロキシでは、クライアントは本当の身元を
150      隠して任意のサイトにアクセスできるようになるため、フォワードプロキシを
151      有効にする前に、承認されたクライアントのみがプロキシにアクセスできるように
152      <a href="#access">サーバを安全にする</a>ことが重要です。</p>
153
154      <p>一方<dfn>リバースプロキシ</dfn>は、クライアントには普通の
155      ウェブサーバのように見えます。クライアント側に特別な設定は必要ありません。
156      クライアントはリバースプロキシの名前空間に対して通常のコンテンツへの
157      リクエストを行ないます。プロキシはリクエストをどこに送れば良いかを判定し、
158      あたかも自分自身がオリジンサーバであったかのようにクライアントに
159      コンテンツを返します。</p>
160
161      <p>リバースプロキシのよくある利用方法は、インターネットユーザに
162      ファイアウォールの中にあるサーバにアクセスを与えるというものです。
163      リバースプロキシは複数のバックエンドサーバへ負荷分散をするために
164      使ったり、遅いバックエンドエンドサーバのためにキャッシュ機能を提供したり
165      するために使えます。また、リバースプロキシは複数のサーバを
166      同じ URL 空間にまとめるために使うこともできます。</p>
167
168      <p>リバースプロキシは <code class="directive"><a href="#proxypass">ProxyPass</a></code> ディレクティブや
169      <code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">RewriteRule</a></code> ディレクティブの
170      <code>[P]</code> フラグを使うことで有効になります。リバースプロキシの
171      設定のために <code class="directive"><a href="#proxyrequests">ProxyRequests</a></code> を設定する必要は
172      <em>ありません</em>。</p>

211<h2><a name="access" id="access">プロキシへのアクセス制御</a></h2>
212      <p>プロキシのアクセスは以下のように <code class="directive"><a href="#proxy">&lt;Proxy&gt;</a></code> コンテナの中に
213      ディレクティブを書くことで制御できます:</p>
214
215      <div class="example"><p><code>
216        &lt;Proxy *&gt;<br />
217        <span class="indent">
218          Order Deny,Allow<br />
219          Deny from all<br />
220          Allow from 192.168.0<br />
221        </span>
222        &lt;/Proxy&gt;
223      </code></p></div>
224
225      <p>アクセス制御のためのディレクティブのより詳しい情報は
226      <code class="module"><a href="../mod/mod_authz_host.html">mod_authz_host</a></code> をお読みください。</p>
227
228      <p>(<code class="directive"><a href="#proxyrequests">ProxyRequests</a></code> ディレクティブを
229      使って) フォワードプロキシを設定している場合は、厳しくアクセス
230      制限を行なうことが非常に大切です。そうしないと、任意のクライアントが
231      身元を明かすことなく任意のホストにアクセスするためにサーバを使うことが
232      できてしまいます。これはあなた自身のネットワークにとっても、インターネット
233      全体にとっても危険なことです。(<code>ProxyRequests Off</code> にして
234      <code class="directive"><a href="#proxypass">ProxyPass</a></code> ディレクティブを使って)
235      リバースプロキシを使っている場合には、クライアントはあなたが明示的に
236      設定したホストにしかアクセスできないため、フォワードプロキシのとき
237      ほどアクセス制御に力を注がなくても大丈夫です。</p>

248<h2><a name="intranet" id="intranet">イントラネットプロキシ</a></h2>
249      <p>イントラネットにある Apache プロキシサーバは外部へのリクエストを
250      会社のファイアウォールを通して送らなければなりません。(このためには
251      個々の <var>scheme</var> についてそれぞれ、ファイアウォールの
252      プロキシにフォワードされるように
253      <code class="directive"><a href="#proxyremote">ProxyRemote</a></code> ディレクティブを
254      設定してください)。しかしイントラネット内のリソースにアクセスするときは、
255      ファイアウォールを通さないでもアクセスできます。
256      どのホストがイントラネットに属し、直接アクセスすべきかを指定するには、
257      <code class="directive"><a href="#noproxy">NoProxy</a></code> ディレクティブが
258      役に立ちます。</p>
259
260      <p>イントラネット内のユーザは WWW のリクエストでローカルドメインを
261      省略することがよくあります。<code>http://somehost.example.com/</code> 
262      というリクエストの代わりに "http://somehost/" をリクエストしたりします。
263      このようなリクエストを受け付け、サーバに設定されているローカルドメインが
264      暗黙のうちに使われていると解釈して、単純にリクエストを処理するものも
265      商用プロキシサーバの中にはあります。
266      サーバが <a href="#proxyrequests">プロキシのサービス用に設定されていて</a>
267      <code class="directive"><a href="#proxydomain">ProxyDomain</a></code> ディレクティブが
268      使用された場合には、Apache はクライアントにリダイレクト応答を送って、
269      正しい、完全な (<span class="transnote">(<em>訳注:</em> fully qualified)</span>) 
270      サーバのアドレスに送ることができます。このように
271      リダイレクトすると、ユーザのブックマークが正しい完全なホスト名を含む
272      ことにもなるため、より好ましい方法と言えるでしょう。</p>

275<h2><a name="envsettings" id="envsettings">プロトコルの調整</a></h2>
276      <p>Keepalive や HTTP/1.1 を適切に実装していないアプリケーションサーバに対して
277      <code class="module"><a href="../mod/mod_proxy.html">mod_proxy</a></code> がリクエストを送信する場合、
278      HTTP/1.0 を使って keepalive を無しにしてリクエストを送るようにする
279      環境変数が二つあります。これらは <code class="directive"><a href="../mod/mod_env.html#setenv">SetEnv</a></code> ディレクティブで設定します。</p>
280
281      <p><code>force-proxy-request-1.0</code> と <code>proxy-nokeepalive</code>
282      がその環境変数です。</p>
283

296<h2><a name="request-bodies" id="request-bodies">リクエストボディ</a></h2>
297
298    <p>POST メソッドなどのリクエストには、リクエストボディがあります。
299    HTTP プロトコル仕様によると、ボディのあるリクエストは chunked 
300    転送を使うか、<code>Content-Length</code>
301    ヘッダを送信しなければなりません。
302    このようなリクエストをオリジンサーバに送信する場合、
303    <code class="module"><a href="../mod/mod_proxy_http.html">mod_proxy_http</a></code> は常に <code>Content-Length</code>
304    を送ろうと試みます。しかし。ボディが大きく、オリジナルのリクエストで
305    chunked 転送が使われている場合、上流へのリクエストに
306    chunked 転送も使われます。
307    この挙動は <a href="../env.html">環境変数</a>で制御できます。
308    <code>proxy-sendcl</code> を設定すると、可能な限り常に 
309    <code>Content-Length</code> を付与して、
310    上流サーバに送信するようになります。
311    逆に <code>proxy-sendchunked</code> を設定すると、リソース消費を抑え、
312    chnked エンコードを使って送信するようになります。</p>

366<div class="directive-section"><h2><a name="NoProxy" id="NoProxy">NoProxy</a> <a name="noproxy" id="noproxy">ディレクティブ</a></h2>
367<table class="directive">
368<tr><th><a href="directive-dict.html#Description">説明:</a></th><td>直接接続する ホスト、ドメイン、ネットワーク</td></tr>
369<tr><th><a href="directive-dict.html#Syntax">構文:</a></th><td><code>NoProxy <var>host</var> [<var>host</var>] ...</code></td></tr>
370<tr><th><a href="directive-dict.html#Context">コンテキスト:</a></th><td>サーバ設定ファイル, バーチャルホスト</td></tr>
371<tr><th><a href="directive-dict.html#Status">ステータス:</a></th><td>Extension</td></tr>
372<tr><th><a href="directive-dict.html#Module">モジュール:</a></th><td>mod_proxy</td></tr>
373</table>
374    <p>このディレクティブはイントラネット中の Apache プロキシサーバにのみ
375    有用です。<code class="directive">NoProxy</code> ディレクティブは空白区切りで、
376    サブネット、IP アドレス、ホスト、ドメインのリストを指定します。
377    これらのどれかにマッチするホストへのリクエストは <code class="directive"><a href="#proxyremote">ProxyRemote</a></code> で設定されたプロキシサーバに
378    フォワードされず、直接処理されます。</p>
379
380    <div class="example"><h3>例</h3><p><code>
381      ProxyRemote  *  http://firewall.mycompany.com:81<br />
382      NoProxy         .mycompany.com 192.168.112.0/21
383    </code></p></div>
384
385    <p><code class="directive">NoProxy</code> ディレクティブの <var>host</var> 引数は
386    以下の種類のどれかです:</p>
387
388    <dl>
389    
390    <dt><var><a name="domain" id="domain">Domain</a></var></dt>
391    <dd>
392    <p><dfn>Domain</dfn> は先頭にピリオドの着いた部分 DNS ドメイン名です。
393    同一 DNS ドメイン及びゾーン (<em>すなわち</em>、ホスト名の末尾がすべて
394    <var>Domain</var> で終わっているということ) に属するホストのリストを
395    表します)。</p>
396
397    <div class="example"><h3>例</h3><p><code>
398      .com .apache.org.
399    </code></p></div>
400
401    <p><var>Domain</var> を <a href="#hostname">Hostname</a> と区別するために (意味的にも構文的にも。DNS ドメインも
402    DNS の A レコードを持つことができるのです!)、<var>Domain</var> は
403    常にピリオドで始まります。</p>
404    
405    <div class="note"><h3>注</h3>
406      <p>ドメイン名の比較は大文字小文字を区別せずに行なわれ、<var>Domain</var>
407      は常に DNS ツリーのルートから始まるものとみなされます。ですから、
408      次の二つのドメイン <code>.MyDomain.com</code> と
409      <code>.mydomain.com.</code> (最後のピリオドに注目) は同一であると
410      みなされます。ドメインの比較は DNS ルックアップなしで行なわれるため、
411      サブネットの比較よりもずっと効率的です。</p>
412    </div></dd>
413
414    
415    <dt><var><a name="subnet" id="subnet">SubNet</a></var></dt>
416    <dd>
417    <p><dfn>SubNet</dfn> は数値形式 (ドットで区切られた四つの数字) の
418    部分インターネットアドレスです。後にスラッシュと <var>Subnet</var>
419    の意味のあるビット数を指定するネットマスクとを続けることができます。
420    共通のネットワークインタフェースを使って到達することのできるサブネットを
421    表すために使われます。明示的にネットマスクを指定しない場合は
422    最後の省略された (もしくは値が 0 の) 数字がマスクを指定します。
423    (この場合は、ネットマスクは 8 ビット単位でしか指定できません。)
424    例:</p>
425
426    <dl>
427    <dt><code>192.168</code> もしくは <code>192.168.0.0</code></dt>
428    <dd>サブネット 192.168.0.0 と暗黙の 16 ビット有効なネットマスク
429    (<code>255.255.0.0</code> というネットマスクの形式で使われることも
430    あります)</dd>
431    <dt><code>192.168.112.0/21</code></dt>
432    <dd>サブネット<code>192.168.112.0/21</code> と 21 ビット有効な
433    ネットマスク (<code>255.255.248.0</code> という形式で使われることも
434    あります)</dd>
435    </dl>
436
437    <p>特別な場合に、32 ビット有効な <em>SubNet</em> は
438    <var><a href="#ipadr">IPAddr</a></var> と同等で、
439    0 ビット有効な <var>SubNet</var> (<em>例えば</em>、0.0.0.0/0) は
440    すべての IP アドレスにマッチする定数 <var>_Default_</var> と同じです。</p>
441    </dd>
442
443    
444    <dt><var><a name="ipaddr" id="ipaddr">IPAddr</a></var></dt>
445    <dd>
446    <p><dfn>IPAddr</dfn> は数値形式 (ドットで区切られた四つの数字) の
447    完全インターネットアドレスです。通常はこのアドレスはホストを
448    表しますが、必ずしもアドレスに対応する DNS ドメイン名があるわけでは
449    ありません。</p>
450
451    <div class="example"><h3>例</h3><p><code>
452      192.168.123.7
453    </code></p></div>
454    
455    <div class="note"><h3>注</h3>
456      <p><var>IPAddr</var> は DNS システムにより解決される必要がないので、
457      apache の性能が向上するかもしれません。</p>
458    </div></dd>
459
460    
461    <dt><var><a name="hostname" id="hostname">Hostname</a></var></dt>
462    <dd>
463    <p><dfn>Hostname</dfn> は DNS ドメインサービスにより一つもしくは
464    複数の <var><a href="#ipaddr">IPAddr</a></var> に解決可能な
465    完全な DNS ドメイン名です。これは (<var><a href="#domain">Domain</a></var>
466    と違って、説明は上記を参照) 論理的なホストを表し、少くとも一つの
467    <var><a href="#ipaddr">IPAddr</a></var> (もしくは違う
468    <var><a href="#ipaddr">IPAddr</a></var> のホストのリスト) に解決
469    されなければなりません)。</p>
470
471    <div class="example"><h3>例</h3><p><code>
472      prep.ai.mit.edu<br />
473      www.apache.org
474    </code></p></div>
475
476    <div class="note"><h3>注</h3>
477      <p>多くの場合、<var>Hostname</var> の代わりに <var><a href="#ipaddr">IPAddr</a></var> を指定した方が、DNS ルックアップを
478      避けることができるため、効率が良くなります。Apache の名前解決は
479      ネームサーバへの接続が遅い PPP 上の場合などにかなり時間を取られる
480      ことがあります。</p>
481      <p><var>Hostname</var> の比較は大文字小文字を区別せずに行なわれ、
482      <var>Hostname</var> は常に DNS ツリーのルートから始まるものとみなされます。
483      ですから、二つのドメイン <code>WWW.MyDomain.com</code> と
484      <code>www.mydomain.com.</code> (最後のピリオドに注目) は同一であると
485      みなされます。</p>
486     </div></dd>
487    </dl>

587    それ以降の残りをボディとして扱う。これはヘッダとボディの間に空行を入れ忘れて

594<div class="directive-section"><h2><a name="ProxyBlock" id="ProxyBlock">ProxyBlock</a> <a name="proxyblock" id="proxyblock">ディレクティブ</a></h2>
595<table class="directive">
596<tr><th><a href="directive-dict.html#Description">説明:</a></th><td>プロキシ接続を禁止する語句、ホスト名、ドメインを指定する</td></tr>
597<tr><th><a href="directive-dict.html#Syntax">構文:</a></th><td><code>ProxyBlock *|<var>word</var>|<var>host</var>|<var>domain</var>
598[<var>word</var>|<var>host</var>|<var>domain</var>] ...</code></td></tr>
599<tr><th><a href="directive-dict.html#Context">コンテキスト:</a></th><td>サーバ設定ファイル, バーチャルホスト</td></tr>

603    <p><code class="directive">ProxyBlock</code> ディレクティブは空白で区切られた
604    語句、ホスト名、ドメインのリストを指定します。サイト名にその語句、ホスト名、
605    ドメインを含むサイトへの HTTP、HTTPS、FTP によるドキュメントのリクエストは
606    プロキシサーバにより<em>ブロックされます</em>。プロキシモジュールは
607    起動時にホスト名と思しき項目の IP アドレスを調べ、後のテストのために
608    キャッシュします。これにより、サーバの起動が少し遅くなるかもしれません。</p>
609
610    <div class="example"><h3>Example</h3><p><code>
611      ProxyBlock joes-garage.com some-host.co.uk rocky.wotsamattau.edu
612    </code></p></div>
613
614    <p><code>rocky.wotsamattau.edu</code> が IP アドレスで参照されたときでも
615    マッチします。</p>
616
617    <p><code>wotsamattau.edu</code> のマッチには <code>wotsamattau</code>
618    だけでも十分です。</p>
619
620    <div class="example"><p><code>
621      ProxyBlock *
622    </code></p></div>
623
624    <p>はすべてのサイトへの接続をブロックすることに注意してください。</p>

722<div class="directive-section"><h2><a name="ProxyPass" id="ProxyPass">ProxyPass</a> <a name="proxypass" id="proxypass">ディレクティブ</a></h2>
723<table class="directive">
724<tr><th><a href="directive-dict.html#Description">説明:</a></th><td>リモートサーバをローカルサーバの URL 空間にマップする</td></tr>
725<tr><th><a href="directive-dict.html#Syntax">構文:</a></th><td><code>ProxyPass [<var>path</var>] !|<var>url</var> [<var>key=value</var> <var>key=value</var> ...]]</code></td></tr>
726<tr><th><a href="directive-dict.html#Context">コンテキスト:</a></th><td>サーバ設定ファイル, バーチャルホスト, ディレクトリ</td></tr>
727<tr><th><a href="directive-dict.html#Status">ステータス:</a></th><td>Extension</td></tr>
728<tr><th><a href="directive-dict.html#Module">モジュール:</a></th><td>mod_proxy</td></tr>
729</table>
730    <p>このディレクティブはリモートサーバをローカルサーバの名前空間に
731    マップできるようにします。ローカルサーバは通常の意味でのプロキシと
732    しては動作せず、リモートサーバのミラーとして振る舞います。
733    <var>path</var> はローカルの仮想パスの名前です。<var>url</var> は
734    リモートサーバの部分 URL になり、クエリー文字列を含むことはできません。</p>
735
736    <div class="warning"><code class="directive">ProxyPass</code> ディレクティブを
737    使っているときは <code class="directive"><a href="#proxyrequests">ProxyRequests</a></code> ディレクティブは通常は
738    <strong>off</strong> に設定されているべきです。</div>
739
740    <p>ローカルサーバのアドレスが <code>http://example.com/</code> であると
741    します。すると、</p>
742
743    <div class="example"><p><code>
744      ProxyPass /mirror/foo/ http://backend.example.com/
745    </code></p></div>
746
747    <p>と設定すると <code>http://example.com/mirror/foo/bar</code> への
748    リクエストが内部的に <code>http://backend.example.com/bar</code> への
749    プロキシリクエストに変換されることになります。</p>
750
751    <p>サブディレクトリをリバースプロキシしたくないときに <code>!</code> は
752    役に立ちます。<em>例えば</em>、</p>
753
754    <div class="example"><p><code>
755      ProxyPass /mirror/foo/i !<br />
756      ProxyPass /mirror/foo http://backend.example.com
757    </code></p></div>
758
759    <p>は <code>/mirror/foo/i</code> を<em>除く</em>
760    <code>/mirror/foo</code> へのすべてのリクエストを
761    <code>backend.example.com</code> にプロキシします。</p>
762
763    <div class="note"><h3>注</h3>
764      <p>順番は重要です。一般的な <code class="directive">ProxyPass</code>
765      ディレクティブの<em>前に</em>
766      除外ディレクティブを置く必要があります。</p>
767    </div>
768
769    <p>2.1 の機能で、バックエンドサーバとの接続にプールされたコネクションを
770    使えるようになりました。<code>key=value</code> 形式のパラメータで
771    このコネクションプーリングの調整ができます。<code>Hard Maximum</code> 
772    のデフォルト値は、有効になっている MPM でのプロセス当たりのスレッド数と
773    同じ数のコネクション数です。prefork MPM では通常は 1 で、worker MPM では
774    <code class="directive">ThreadsPerChild</code> で調整されます。</p>
775
776    <p><code>min</code> の設定で、バックエンドサーバとの間に何本のコネクションを
777    常時開くかが決まります。Soft Maximum <code>smax</code> の数に
778    達するまで必要に応じてコネクションは生成されます。<code>smax</code>
779    を超えた数のコネクションは、生存時間 <code>ttl</code> で切断されます。
780    バックエンドサーバと Hard Maximum <code>max</code> の数以上のコネクションを
781    生成することはありません。</p>
782
783    <div class="example"><p><code>
784        ProxyPass /example http://backend.example.com smax=5 max=20 ttl=120 retry=300
785    </code></p></div>

967<div class="directive-section"><h2><a name="ProxyPassReverse" id="ProxyPassReverse">ProxyPassReverse</a> <a name="proxypassreverse" id="proxypassreverse">ディレクティブ</a></h2>
968<table class="directive">
969<tr><th><a href="directive-dict.html#Description">説明:</a></th><td>リバースプロキシされたサーバから送られた HTTP 応答ヘッダの
970URL を調整する</td></tr>
971<tr><th><a href="directive-dict.html#Syntax">構文:</a></th><td><code>ProxyPassReverse [<var>path</var>] <var>url</var></code></td></tr>
972<tr><th><a href="directive-dict.html#Context">コンテキスト:</a></th><td>サーバ設定ファイル, バーチャルホスト, ディレクトリ</td></tr>
973<tr><th><a href="directive-dict.html#Status">ステータス:</a></th><td>Extension</td></tr>
974<tr><th><a href="directive-dict.html#Module">モジュール:</a></th><td>mod_proxy</td></tr>
975</table>
976    <p>このディレクティブは Apache に HTTP リダイレクト応答の
977    <code>Location</code>, <code>Content-Location</code>, <code>URI</code>
978    ヘッダの調整をさせます。これは、Apache がリバースプロキシとして使われている
979    ときに、リバースプロキシを通さないでアクセスすることを防ぐために
980    重要です。これによりバックエンドサーバの HTTP リダイレクトが
981    リバースプロキシとバックエンドの間で扱われるようになります。</p>
982
983    <p>ディレクティブで明示されている HTTP 応答ヘッダのみが書き換えられます。
984    Apache は他の応答ヘッダを書き換えたり、HTML ページの中の URL 参照を
985    書き換えたりすることはありません。HTML の中を見て、URL 参照を書き換える
986    モジュールに Nick Kew さんの <a href="http://apache.webthing.com/mod_proxy_html/">mod_proxy_html</a> があります。</p>
987
988    <p><var>path</var> はローカル仮想パスの名前です。<var>url</var> は
989    リモートサーバの部分 URL です。これらは <code class="directive"><a href="#proxypass">ProxyPass</a></code> ディレクティブと同様です。</p>
990
991    <p>例えば、ローカルサーバのアドレスが <code>http://example.com/</code>
992    だとします。すると</p>
993
994    <div class="example"><p><code>
995      ProxyPass         /mirror/foo/ http://backend.example.com/<br />
996      ProxyPassReverse  /mirror/foo/ http://backend.example.com/<br />
997      ProxyPassReverseCookieDomain  backend.example.com  public.example.com<br />
998      ProxyPassReverseCookiePath  /  /mirror/foo/
999    </code></p></div>
1000
1001    <p>という設定をすると、<code>http://example.com/mirror/foo/bar</code>
1002    へのローカルリクエストが <code>http://backend.example.com/bar</code>
1003    へのプロキシリクエストに内部でリダイレクトされるだけではありません
1004    (これは <code>ProxyPass</code> の機能です)。<code>backend.example.com</code>
1005    が送るリダイレクトの面倒もみます。<code>http://backend.example.com/bar</code>
1006    が <code>http://backend.example.com/quux</code> にリダイレクトされたとき、
1007    Apache は HTTP リダイレクト応答をクライアントに送る前に、
1008    <code>http://example.com/mirror/foo/quux</code> に変更します。
1009    URL を構成するのに使われるホスト名は <code class="directive"><a href="../mod/core.html#usecanonicalname">UseCanonicalName</a></code> の設定に応じて選択されることに
1010    注意してください。</p>
1011
1012    <p><code class="directive">ProxyPassReverse</code> ディレクティブは
1013    対応する <code class="directive"><a href="#proxypass">ProxyPass</a></code> ディレクティブには依存しないため、
1014    <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code> のプロキシ通過機能
1015    (<code>RewriteRule ...  [P]</code>) と併せて使用することができます。</p>
1016
1017    <p><code class="directive"><a href="../mod/core.html#location">&lt;Location&gt;</a></code> セクションの中で使われた場合は、
1018    最初の引数は省略され、ローカルディレクトリは <code class="directive"><a href="../mod/core.html#location">&lt;Location&gt;</a></code> から取得されます。</p>
1019

1226<div class="directive-section"><h2><a name="ProxyVia" id="ProxyVia">ProxyVia</a> <a name="proxyvia" id="proxyvia">ディレクティブ</a></h2>
1227<table class="directive">
1228<tr><th><a href="directive-dict.html#Description">説明:</a></th><td>プロキシされたリクエストの <code>Via</code> HTTP 応答ヘッダ
1229により提供される情報</td></tr>
1230<tr><th><a href="directive-dict.html#Syntax">構文:</a></th><td><code>ProxyVia On|Off|Full|Block</code></td></tr>
1231<tr><th><a href="directive-dict.html#Default">デフォルト:</a></th><td><code>ProxyVia Off</code></td></tr>
1232<tr><th><a href="directive-dict.html#Context">コンテキスト:</a></th><td>サーバ設定ファイル, バーチャルホスト</td></tr>
1233<tr><th><a href="directive-dict.html#Status">ステータス:</a></th><td>Extension</td></tr>
1234<tr><th><a href="directive-dict.html#Module">モジュール:</a></th><td>mod_proxy</td></tr>
1235</table>
1236    <p>このディレクティブはプロキシの <code>Via:</code> HTTP ヘッダの使用を
1237    制御します。想定されている使い方は、プロキシサーバがいくつも繋がっているときに
1238    プロキシリクエストの流れを制御することです。<code>Via:</code> ヘッダ行の
1239    説明は <a href="http://www.ietf.org/rfc/rfc2616.txt">RFC 2616</a> (HTTP/1.1)
1240    の 14.45 節を読んでください。</p>
1241
1242    <ul>
1243    <li>デフォルトの <code>Off</code> に設定されていると、特別な処理は
1244    行なわれません。リクエストやリプライに <code>Via:</code> ヘッダがあれば、
1245    変更されずにそのまま渡します。</li>
1246
1247    <li><code>On</code> に設定されていれば、各リクエストとリプライに
1248    <code>Via:</code> 行が追加されます。</li>
1249
1250    <li><code>Full</code> に設定されていれば、<code>Via:</code> ヘッダは
1251    コメント部分に Apache サーバのバージョンも含むようになります。</li>
1252
1253    <li><code>Block</code> に設定されていれば、すべてのプロキシリクエストから
1254    <code>Via:</code> ヘッダが取り除かれます。新たに <code>Via:</code> が
1255    生成されることはありません。</li>
1256    </ul>
