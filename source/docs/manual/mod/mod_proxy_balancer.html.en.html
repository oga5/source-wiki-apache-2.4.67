38<h3>Summary</h3>
39
40    <p>This module <em>requires</em> the service of <code class="module"><a href="../mod/mod_proxy.html">mod_proxy</a></code> and it provides load balancing for
41    all the supported protocols. The most important ones are:</p>
42    <ul>
43        <li>HTTP, using <code class="module"><a href="../mod/mod_proxy_http.html">mod_proxy_http</a></code></li>
44        <li>FTP, using <code class="module"><a href="../mod/mod_proxy_ftp.html">mod_proxy_ftp</a></code></li>
45        <li>AJP13, using <code class="module"><a href="../mod/mod_proxy_ajp.html">mod_proxy_ajp</a></code></li>
46        <li>WebSocket, using <code class="module"><a href="../mod/mod_proxy_wstunnel.html">mod_proxy_wstunnel</a></code></li>
47    </ul>
48    
49    <p>The Load balancing scheduler algorithm is not provided by this
50    module but from other ones such as:</p>
51    <ul>
52        <li><code class="module"><a href="../mod/mod_lbmethod_byrequests.html">mod_lbmethod_byrequests</a></code></li>
53        <li><code class="module"><a href="../mod/mod_lbmethod_bytraffic.html">mod_lbmethod_bytraffic</a></code></li>
54        <li><code class="module"><a href="../mod/mod_lbmethod_bybusyness.html">mod_lbmethod_bybusyness</a></code></li>
55        <li><code class="module"><a href="../mod/mod_lbmethod_heartbeat.html">mod_lbmethod_heartbeat</a></code></li>
56    </ul>
57
58    <p>Thus, in order to get the ability of load balancing,
59    <code class="module"><a href="../mod/mod_proxy.html">mod_proxy</a></code>, <code class="module"><a href="../mod/mod_proxy_balancer.html">mod_proxy_balancer</a></code>
60    and at least one of load balancing scheduler algorithm modules have
61    to be present in the server.</p>
62
63    <div class="warning"><h3>Warning</h3>
64      <p>Do not enable proxying until you have <a href="mod_proxy.html#access">secured your server</a>. Open proxy
65      servers are dangerous both to your network and to the Internet at
66      large.</p>
67    </div>

91<h2><a name="scheduler" id="scheduler">Load balancer scheduler algorithm</a></h2>
92    
93    <p>At present, there are 4 load balancer scheduler algorithms available
94    for use: Request Counting (<code class="module"><a href="../mod/mod_lbmethod_byrequests.html">mod_lbmethod_byrequests</a></code>),
95    Weighted Traffic Counting (<code class="module"><a href="../mod/mod_lbmethod_bytraffic.html">mod_lbmethod_bytraffic</a></code>),
96    Pending Request Counting (<code class="module"><a href="../mod/mod_lbmethod_bybusyness.html">mod_lbmethod_bybusyness</a></code>) and
97    Heartbeat Traffic Counting (<code class="module"><a href="../mod/mod_lbmethod_heartbeat.html">mod_lbmethod_heartbeat</a></code>).
98    These are controlled via the <code>lbmethod</code> value of
99    the Balancer definition. See the <code class="directive"><a href="../mod/mod_proxy.html#proxypass">ProxyPass</a></code>
100    directive for more information, especially regarding how to
101    configure the Balancer and BalancerMembers.</p>
102</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>

104<h2><a name="stickyness" id="stickyness">Load balancer stickyness</a></h2>
105    
106    <p>The balancer supports stickyness. When a request is proxied
107    to some back-end, then all following requests from the same user
108    should be proxied to the same back-end. Many load balancers implement
109    this feature via a table that maps client IP addresses to back-ends.
110    This approach is transparent to clients and back-ends, but suffers
111    from some problems: unequal load distribution if clients are themselves
112    hidden behind proxies, stickyness errors when a client uses a dynamic
113    IP address that changes during a session and loss of stickyness, if the
114    mapping table overflows.</p>
115    <p>The module <code class="module"><a href="../mod/mod_proxy_balancer.html">mod_proxy_balancer</a></code> implements stickyness
116    on top of two alternative means: cookies and URL encoding. Providing the
117    cookie can be either done by the back-end or by the Apache web server
118    itself. The URL encoding is usually done on the back-end.</p>
119</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>

121<h2><a name="example" id="example">Examples of a balancer configuration</a></h2>
122    
123    <p>Before we dive into the technical details, here's an example of
124    how you might use <code class="module"><a href="../mod/mod_proxy_balancer.html">mod_proxy_balancer</a></code> to provide
125    load balancing between two back-end servers:
126    </p>
127
128    <pre class="prettyprint lang-config">&lt;Proxy "balancer://mycluster"&gt;
129    BalancerMember "http://192.168.1.50:80"
130    BalancerMember "http://192.168.1.51:80"
131&lt;/Proxy&gt;
132ProxyPass        "/test" "balancer://mycluster"
133ProxyPassReverse "/test" "balancer://mycluster"</pre>
134
135
136    <p>Another example of how to provide load balancing with stickyness
137    using <code class="module"><a href="../mod/mod_headers.html">mod_headers</a></code>, even if the back-end server does
138    not set a suitable session cookie:
139    </p>
140
141    <pre class="prettyprint lang-config">Header add Set-Cookie "ROUTEID=.%{BALANCER_WORKER_ROUTE}e; path=/" env=BALANCER_ROUTE_CHANGED
142&lt;Proxy "balancer://mycluster"&gt;
143    BalancerMember "http://192.168.1.50:80" route=1
144    BalancerMember "http://192.168.1.51:80" route=2
145    ProxySet stickysession=ROUTEID
146&lt;/Proxy&gt;
147ProxyPass        "/test" "balancer://mycluster"
148ProxyPassReverse "/test" "balancer://mycluster"</pre>

152<h2><a name="environment" id="environment">Exported Environment Variables</a></h2>
153    
154    <p>At present there are 6 environment variables exported:</p>
155
156    <dl>
157    
158    <dt><var><a name="balancer_session_sticky" id="balancer_session_sticky">BALANCER_SESSION_STICKY</a></var></dt>
159    <dd>
160    <p>This is assigned the <var>stickysession</var> value used for the current
161    request.  It is the name of the cookie or request parameter used for sticky sessions</p>
162    </dd>
163
164    
165    <dt><var><a name="balancer_session_route" id="balancer_session_route">BALANCER_SESSION_ROUTE</a></var></dt>
166    <dd>
167    <p>This is assigned the <var>route</var> parsed from the current
168    request.</p>
169    </dd>
170
171    
172    <dt><var><a name="balancer_name" id="balancer_name">BALANCER_NAME</a></var></dt>
173    <dd>
174    <p>This is assigned the name of the balancer used for the current
175    request. The value is something like <code>balancer://foo</code>.</p>
176    </dd>
177
178    
179    <dt><var><a name="balancer_worker_name" id="balancer_worker_name">BALANCER_WORKER_NAME</a></var></dt>
180    <dd>
181    <p>This is assigned the name of the worker used for the current request.
182    The value is something like <code>http://hostA:1234</code>.</p>
183    </dd>
184
185    
186    <dt><var><a name="balancer_worker_route" id="balancer_worker_route">BALANCER_WORKER_ROUTE</a></var></dt>
187    <dd>
188    <p>This is assigned the <var>route</var> of the worker that will be
189    used for the current request.</p>
190    </dd>
191
192    
193    <dt><var><a name="balancer_route_changed" id="balancer_route_changed">BALANCER_ROUTE_CHANGED</a></var></dt>
194    <dd>
195    <p>This is set to 1 if the session route does not match the
196    worker route (BALANCER_SESSION_ROUTE != BALANCER_WORKER_ROUTE) or the
197    session does not yet have an established route.  This can be used to
198    determine when/if the client needs to be sent an updated route
199    when sticky sessions are used.</p>
200    </dd>
201    </dl>

205<h2><a name="balancer_manager" id="balancer_manager">Enabling Balancer Manager Support</a></h2>
206    
207    <p>This module <em>requires</em> the service of
208    <code class="module"><a href="../mod/mod_status.html">mod_status</a></code>.
209    Balancer manager enables dynamic update of balancer
210    members. You can use balancer manager to change the balance
211    factor of a particular member, or put it in the off line
212    mode.
213    </p>
214
215    <p>Thus, in order to get the ability of load balancer management,
216    <code class="module"><a href="../mod/mod_status.html">mod_status</a></code> and <code class="module"><a href="../mod/mod_proxy_balancer.html">mod_proxy_balancer</a></code>
217    have to be present in the server.</p>
218
219    <p>To enable load balancer management for browsers from the example.com
220    domain add this code to your <code>httpd.conf</code>
221    configuration file</p>
222<pre class="prettyprint lang-config">&lt;Location "/balancer-manager"&gt;
223    SetHandler balancer-manager
224    Require host example.com
225&lt;/Location&gt;</pre>
226
227
228    <p>You can now access load balancer manager by using a Web browser
229    to access the page
230    <code>http://your.server.name/balancer-manager</code>. Please note
231    that only Balancers defined outside of <code>&lt;Location ...&gt;</code>
232    containers can be dynamically controlled by the Manager.</p>

235<h2><a name="stickyness_implementation" id="stickyness_implementation">Details on load balancer stickyness</a></h2>
236    
237    <p>When using cookie based stickyness, you need to configure the
238    name of the cookie that contains the information about which back-end
239    to use. This is done via the <var>stickysession</var> attribute added
240    to either <code class="directive"><a href="../mod/mod_proxy.html#proxypass">ProxyPass</a></code> or
241    <code class="directive"><a href="../mod/mod_proxy.html#proxyset">ProxySet</a></code>. The name of
242    the cookie is case-sensitive. The balancer extracts the value of the
243    cookie and looks for a member worker with <var>route</var> equal
244    to that value. The <var>route</var> must also be set in either
245    <code class="directive"><a href="../mod/mod_proxy.html#proxypass">ProxyPass</a></code> or
246    <code class="directive"><a href="../mod/mod_proxy.html#proxyset">ProxySet</a></code>. The cookie can either
247    be set by the back-end, or as shown in the above
248    <a href="#example">example</a> by the Apache web server itself.</p>
249    <p>Some back-ends use a slightly different form of stickyness cookie,
250    for instance Apache Tomcat. Tomcat adds the name of the Tomcat instance
251    to the end of its session id cookie, separated with a dot (<code>.</code>)
252    from the session id. Thus if the Apache web server finds a dot in the value
253    of the stickyness cookie, it only uses the part behind the dot to search
254    for the route. In order to let Tomcat know about its instance name, you
255    need to set the attribute <code>jvmRoute</code> inside the Tomcat
256    configuration file <code>conf/server.xml</code> to the value of the
257    <var>route</var> of the worker that connects to the respective Tomcat.
258    The name of the session cookie used by Tomcat (and more generally by Java
259    web applications based on servlets) is <code>JSESSIONID</code>
260    (upper case) but can be configured to something else.</p>
261    <p>The second way of implementing stickyness is URL encoding.
262    The web server searches for a query parameter in the URL of the request.
263    The name of the parameter is specified again using <var>stickysession</var>.
264    The value of the parameter is used to lookup a member worker with <var>route</var>
265    equal to that value. Since it is not easy to extract and manipulate all
266    URL links contained in responses, generally the work of adding the parameters
267    to each link is done by the back-end generating the content.
268    In some cases it might be feasible doing
269    this via the web server using <code class="module"><a href="../mod/mod_substitute.html">mod_substitute</a></code> or
270    <code class="module"><a href="../mod/mod_sed.html">mod_sed</a></code>. This can have negative impact on performance though.</p>
271    <p>The Java standards implement URL encoding slightly different. They use
272    a path info appended to the URL using a semicolon (<code>;</code>)
273    as the separator and add the session id behind. As in the cookie case,
274    Apache Tomcat can include the configured <code>jvmRoute</code> in this path
275    info. To let Apache find this sort of path info, you need to set
276    <code>scolonpathdelim</code> to <code>On</code> in
277    <code class="directive"><a href="../mod/mod_proxy.html#proxypass">ProxyPass</a></code> or
278    <code class="directive"><a href="../mod/mod_proxy.html#proxyset">ProxySet</a></code>.</p>
279    <p>Finally you can support cookies and URL encoding at the same time, by
280    configuring the name of the cookie and the name of the URL parameter
281    separated by a vertical bar (<code>|</code>) as in the following example:</p>
282    <pre class="prettyprint lang-config">ProxyPass "/test" "balancer://mycluster" stickysession=JSESSIONID|jsessionid scolonpathdelim=On
283&lt;Proxy "balancer://mycluster"&gt;
284    BalancerMember "http://192.168.1.50:80" route=node1
285    BalancerMember "http://192.168.1.51:80" route=node2
286&lt;/Proxy&gt;</pre>
287
288    <p>If the cookie and the request parameter both provide routing information
289    for the same request, the information from the request parameter is used.</p>

292<h2><a name="stickyness_troubleshooting" id="stickyness_troubleshooting">Troubleshooting load balancer stickyness</a></h2>
293    
294    <p>If you experience stickyness errors, e.g. users lose their
295    application sessions and need to login again, you first want to
296    check whether this is because the back-ends are sometimes unavailable
297    or whether your configuration is wrong. To find out about possible
298    stability problems with the back-ends, check your Apache error log
299    for proxy error messages.</p>
300    <p>To verify your configuration, first check, whether the stickyness
301    is based on a cookie or on URL encoding. Next step would be logging
302    the appropriate data in the access log by using an enhanced
303    <code class="directive"><a href="../mod/mod_log_config.html#logformat">LogFormat</a></code>.
304    The following fields are useful:</p>
305    <dl>
306    <dt><code>%{MYCOOKIE}C</code></dt>
307    <dd>The value contained in the cookie with name <code>MYCOOKIE</code>.
308    The name should be the same given in the <var>stickysession</var>
309    attribute.</dd>
310    <dt><code>%{Set-Cookie}o</code></dt>
311    <dd>This logs any cookie set by the back-end. You can track,
312    whether the back-end sets the session cookie you expect, and
313    to which value it is set.</dd>
314    <dt><code>%{BALANCER_SESSION_STICKY}e</code></dt>
315    <dd>The name of the cookie or request parameter used
316    to lookup the routing information.</dd>
317    <dt><code>%{BALANCER_SESSION_ROUTE}e</code></dt>
318    <dd>The route information found in the request.</dd>
319    <dt><code>%{BALANCER_WORKER_ROUTE}e</code></dt>
320    <dd>The route of the worker chosen.</dd>
321    <dt><code>%{BALANCER_ROUTE_CHANGED}e</code></dt>
322    <dd>Set to <code>1</code> if the route in the request
323    is different from the route of the worker, i.e.
324    the request couldn't be handled sticky.</dd>
325    </dl>
326    <p>Common reasons for loss of session are session timeouts,
327    which are usually configurable on the back-end server.</p>
328    <p>The balancer also logs detailed information about handling
329    stickyness to the error log, if the log level is set to
330    <code>debug</code> or higher. This is an easy way to
331    troubleshoot stickyness problems, but the log volume might
332    be too high for production servers under high load.</p>
