42<h3>概要</h3>
43
44    <p>本モジュールには <code class="module"><a href="../mod/mod_proxy.html">mod_proxy</a></code> が<em>必要です</em>。
45    <code>Apache JServ Protocol version 1.3</code> (以降 <em>AJP13</em>)
46    をサポートします。</p>
47
48    <p><code>AJP13</code> プロトコルを扱えるようにするには
49    <code class="module"><a href="../mod/mod_proxy.html">mod_proxy</a></code> と <code class="module"><a href="../mod/mod_proxy_ajp.html">mod_proxy_ajp</a></code>
50    をサーバに組み込む必要があります。</p>
51
52    <div class="warning"><h3>警告</h3>
53      <p><a href="#access">安全なサーバにする</a>までプロクシ機能は有効にしないでください。
54      オープンプロキシサーバはあなた自身のネットワークにとっても、
55      インターネット全体にとっても危険です。</p>

72<h2><a name="overviewprotocol" id="overviewprotocol">プロトコルの概要</a></h2>
73    <p><code>AJP13</code> プロトコルはパケット指向です。
74    可読なプレーンテキスト形式ではなくバイナリ形式になったのは、
75    おそらくパフォーマンス上の理由によります。
76    ウェブサーバはサーブレットコンテナと TCP コネクションで通信します。
77    ソケット生成は重い処理なので、負荷を減らすために、サーブレットコンテナとの
78    TCP 接続を維持し、複数のリクエスト・レスポンス処理サイクルに対して一つの
79    コネクションを使いまわすようになっています。</p>
80    <p>あるリクエストにコネクションが割り当てられると、その処理サイクルが
81    完了するまで他のものに使われることはありません。
82    つまりコネクション上では、リクエストの同時処理は行われません。
83    このため、コネクション両端での実行するコードを簡潔にできる一方で、
84    同時に開くコネクションは多くなっています。</p>
85    <p>サーブレットコンテナへのコネクションを開いた後は、コネクションの状態は
86    次のどれかになります:</p>
87    <ul>
88    <li> Idle <br />コネクション上で処理されているリクエストはありません。</li>
89    <li> Assigned <br />コネクションはリクエストを処理中です。</li>
90    </ul>
91    <p>コネクションが特定のリクエストにアサインされると、基本的な情報 (例えば
92    HTTP ヘッダ等) が圧縮された形 (例えば通常の文字列は整数にエンコードされます)
93    で転送されます。詳細は下記の「リクエストパケットの構造」を参照してください。
94    リクエストにボディが存在 <code>(content-length &gt; 0)</code> すれば、
95    基本的な情報の直後に別パケットで転送されます。</p>
96    <p>この時点でおそらく、サーブレットコンテナは処理を開始できるようになります。
97    ですので、次のメッセージをウェブサーバに戻して知らせられるようになります。</p>
98    <ul>
99    <li>SEND_HEADERS <br />ブラウザにヘッダを送信します。</li>
100    <li>SEND_BODY_CHUNK <br />ブラウザにボディデータのチャンクを送ります。
101    </li>
102    <li>GET_BODY_CHUNK <br />リクエストのデータを全て受け取り終わっていないときに、
103    残っているデータを受け取ります。パケットにある定まった最大長があり、任意の
104    大きさのデータがリクエストのボディとして含まれうる場合 
105    (例えばファイルのアップロードの場合) に必要となります。
106    (注: HTTP のチャンク転送とは関連ありません。)</li>
107    <li>END_RESPONSE <br />リクエスト処理サイクルを終了します。</li>
108    </ul>
109    <p>個々のメッセージはそれぞれ異なるデータパケット形式になっています。
110    後述の「レスポンスパケットの構造」を参照してください。</p>

113<h2><a name="basppacketstruct" id="basppacketstruct">基本パケット構造</a></h2>
114    <p>このプロトコルには XDR から受け継いだ部分が少しありますが、多くの点で
115    異なります (例えば 4 バイトアライメントでないことなど) 。</p>
116    <p>バイトオーダー: 個々のバイトのエンディアンがどうなっているかは、
117    私は詳しくないのですが、リトルエンディアンになっていると思います。
118    XDR 仕様でそうなっているのと、素晴らしいことに sys/socket ライブラリが
119    (C で) そういう風にできているのでそうなのだと思いました。
120    ソケット呼び出しの内部についてより詳しい方がいらっしゃいましたら、
121    ご教授ください。</p>
122    <p>プロトコルには 4 つのデータタイプがあります: byte, boolean, 
123    integer, string です。</p>
124    <dl>
125    <dt><strong>Byte</strong></dt><dd>バイト一つです。</dd>
126    <dt><strong>Boolean</strong></dt>
127      <dd>バイト一つで、<code>1 = true</code>, <code>0 = false</code> です。
128      (C のように) 非零を真として扱ってしまうと、ある場合は動くかもしれませんし、
129      動かないかもしれません。</dd>
130    <dt><strong>Integer</strong></dt>
131      <dd><code>0 から 2^16 (32768)</code> の範囲の数字。高次の 2 バイトが
132      先に格納されます。</dd>
133    <dt><strong>String</strong></dt>
134      <dd>可変長の文字列 (2^16 が長さの上限) 。長さ情報のパケット 2 バイトの後に
135      文字列 (終端文字 '\0' を含む) が続く形式でエンコードされます。
136      エンコードされている長さ情報は最後の '\0' を<strong>カウントしない</strong>
137      ことに注意してください――これは <code>strlen</code> と同様です。
138      これらの終端文字をスキップするために、あまり意味の無いインクリメント文
139      をたくさん書かないといけないのは、
140      Java の側から見ると少し紛らわしく感じられるかもしれません。
141      こうなった理由はおそらく、Servlet コンテナから返される文字列を読み出す時に、
142      効率よく C のコードを書けるようにする――サーブレットから返される
143      文字列は \0 文字で終端されているので、C のコードではわざわざコピーをせずに、
144      一つのバッファへのリファレンスを取り回すように書くことができる――
145      ためだと思われます。
146      '\0' 文字がない場合は、C では文字列の規則に合うようにコピーしなければ
147      いけなくなってしまいます。</dd>
148    </dl>
149
150  <h3>パケットサイズ</h3>
151    <p>多くのコードでそうなっているのですが、パケットサイズの最大サイズは
152    <code>8 * 1024 (8K)</code> です。パケットの実際の長さはヘッダに
153    エンコードされて入っています。</p>
154  
155  <h3>パケットヘッダ</h3>

157    コンテナからサーバに送られるパケットは <code>AB</code> (ASCII コード A と
158    ASCII コード B) で始まります。この二バイトの後に、ペイロード長が (上記の形式で) 
159    続きます。このため、ペイロード長の最大値は 2^16 にできるように思えますが、
160    実際にはコードでは最大値は 8K に設定されています。</p>

287<h2><a name="rpacetstruct" id="rpacetstruct">リクエストパケット構造</a></h2>
288    <p>サーバからコンテナへ送られるメッセージが
289    <em>Forward Request</em> 型の場合 :</p>
290    <div class="example"><pre>AJP13_FORWARD_REQUEST :=
291    prefix_code      (byte) 0x02 = JK_AJP13_FORWARD_REQUEST
292    method           (byte)
293    protocol         (string)
294    req_uri          (string)
295    remote_addr      (string)
296    remote_host      (string)
297    server_name      (string)
298    server_port      (integer)
299    is_ssl           (boolean)
300    num_headers      (integer)
301    request_headers *(req_header_name req_header_value)
302    attributes      *(attribut_name attribute_value)
303    request_terminator (byte) OxFF</pre></div>
304    <p><code>request_headers</code> は次のような構造になっています :
305    </p><div class="example"><pre>req_header_name := 
306    sc_req_header_name | (string)  [see below for how this is parsed]
307
308sc_req_header_name := 0xA0xx (integer)
309
310req_header_value := (string)</pre></div>
311    <p><code>属性</code> はオプションで、次のような構造をしています :</p>
312    <div class="example"><pre>attribute_name := sc_a_name | (sc_a_req_attribute string)
313
314attribute_value := (string)</pre></div>
315    <p>もっとも重要なヘッダは <code>content-length</code> だということに
316    注意してください。コンテナは次のパケットを探すかどうかを、
317    それを見て決めるからです。</p>

359  <h3>protocol, req_uri, remote_addr, remote_host, server_name,
360  server_port, is_ssl</h3>
361    <p>これらはまさに文字通りのものです。どれも必要で、リクエストの毎回につき
362    送られます。</p>
363  
364  <h3>Headers</h3>
365    <p><code>request_headers</code> の構造は次のようなものです :
366    まずヘッダの数 <code>num_headers</code> がエンコードされます。
367    次にヘッダ名 <code>req_header_name</code> / 値 <code>req_header_value</code>
368    の組が続きます。効率のため、一般的なヘッダは整数でエンコードして転送します。
369    ヘッダ名が基本ヘッダの一覧に無い場合は、通常通り (文字列として、長さ
370    プレフィックス付きで) 転送されます。一般的なヘッダ 
371    <code>sc_req_header_name</code> の一覧とそのコードは次の通りです 
372    (どれも大文字小文字を区別します) :</p>
373    <table>
374      <tr><td>名前</td><td>コードの値</td><td>コード名</td></tr>
375      <tr><td>accept</td><td>0xA001</td><td>SC_REQ_ACCEPT</td></tr>
376      <tr><td>accept-charset</td><td>0xA002</td><td>SC_REQ_ACCEPT_CHARSET
377      </td></tr>
378      <tr><td>accept-encoding</td><td>0xA003</td><td>SC_REQ_ACCEPT_ENCODING
379      </td></tr>
380      <tr><td>accept-language</td><td>0xA004</td><td>SC_REQ_ACCEPT_LANGUAGE
381      </td></tr>
382      <tr><td>authorization</td><td>0xA005</td><td>SC_REQ_AUTHORIZATION</td>
383      </tr>
384      <tr><td>connection</td><td>0xA006</td><td>SC_REQ_CONNECTION</td></tr>
385      <tr><td>content-type</td><td>0xA007</td><td>SC_REQ_CONTENT_TYPE</td>
386      </tr>
387      <tr><td>content-length</td><td>0xA008</td><td>SC_REQ_CONTENT_LENGTH</td>
388      </tr>
389      <tr><td>cookie</td><td>0xA009</td><td>SC_REQ_COOKIE</td></tr>
390      <tr><td>cookie2</td><td>0xA00A</td><td>SC_REQ_COOKIE2</td></tr>
391      <tr><td>host</td><td>0xA00B</td><td>SC_REQ_HOST</td></tr>
392      <tr><td>pragma</td><td>0xA00C</td><td>SC_REQ_PRAGMA</td></tr>
393      <tr><td>referer</td><td>0xA00D</td><td>SC_REQ_REFERER</td></tr>
394      <tr><td>user-agent</td><td>0xA00E</td><td>SC_REQ_USER_AGENT</td></tr>
395    </table>
396    <p>これを読み込む Java のコードでは、最初の 2 バイト整数を取り込み、
397    目印になるバイト <code>'0xA0'</code> であれば、ヘッダ名の配列の
398    インデックスを使います。先頭バイトが <code>0xA0</code> でない場合は、
399    先頭 2 バイトは文字列長を表す整数であると解釈し、読み込みはじめます。</p>
400    <p>ヘッダ名の長さは <code>0x9999 (==0xA000 -1)</code> 以上にならないという
401    仮定の下に動いていて、少しあいまいですが合理的な挙動になっています。</p>
402    <div class="note"><h3>注:</h3>
403    <code>content-length</code> ヘッダはとても重要です。
404    存在していて非ゼロであれば、リクエストにはボディがある (例えば POST 
405    リクエスト) と推測し、そのボディを取り込むために
406    直後のパケットを入力ストリームから読み込みはじめます。
407    </div>
408  
409  <h3>属性</h3>
410    <p><code>?</code> プレフィックスで始まる属性 (例 <code>?context</code>)
411    は。省略可能です。それぞれ属性の型を示す 1 バイトのコードと、
412    値(文字列か整数)が続きます。
413    これらは順不同で送ることができます (C のコードは常に下の一覧順に
414    送るようですが) 。
415    オプションの属性のリストの最後には、特別な終了コードが送られます。
416    コードの一覧は : </p>
417    <table>
418      <tr><td>Information</td><td>Code Value</td><td>Type Of Value</td><td>Note</td></tr>
419      <tr><td>?context</td><td>0x01</td><td>-</td><td>未実装
420      </td></tr>
421      <tr><td>?servlet_path</td><td>0x02</td><td>-</td><td>未実装
422      </td></tr>
423      <tr><td>?remote_user</td><td>0x03</td><td>String</td><td /></tr>
424      <tr><td>?auth_type</td><td>0x04</td><td>String</td><td /></tr>
425      <tr><td>?query_string</td><td>0x05</td><td>String</td><td /></tr>
426      <tr><td>?jvm_route</td><td>0x06</td><td>String</td><td /></tr>
427      <tr><td>?ssl_cert</td><td>0x07</td><td>String</td><td /></tr>
428      <tr><td>?ssl_cipher</td><td>0x08</td><td>String</td><td /></tr>
429      <tr><td>?ssl_session</td><td>0x09</td><td>String</td><td /></tr>
430      <tr><td>?req_attribute</td><td>0x0A</td><td>String</td><td>Name (the name of the
431      attribute follows)</td></tr>
432      <tr><td>?ssl_key_size</td><td>0x0B</td><td>Integer</td><td /></tr>
433      <tr><td>are_done</td><td>0xFF</td><td>-</td><td>request_terminator</td></tr>
434    </table>
435    <p><code>context</code> と <code>servlet_path</code> は現在の C の
436    コードではセットされていません。また、ほとんどの Java のコードでも、
437    このフィールドで何が送られても無視されます (これらのコードの後に文字列が
438    送られると壊れるものもあります)。
439    これがバグなのか、単に未実装なのか、歴史的経緯で残っているコードなのか
440    分かりませんが、コネクションの両側ともで見当たりません。</p>
441    <p><code>remote_user</code> と <code>auth_type</code> はおそらく
442    HTTP レベルの認証を参照していて、リモートユーザのユーザ名と認証に使用した
443    タイプ (例 Basic, Digest) についてやり取りします。</p>
444    <p><code>query_string</code>, <code>ssl_cert</code>, 
445    <code>ssl_cipher</code>, <code>ssl_session</code> 
446    は HTTP と HTTPS の対応する部分を参照します。</p>
447    <p><code>jvm_route</code> はスティッキーセッションのサポート――
448    ロードバランスしている複数のサーバ中の特定の Tomcat インスタンスと、
449    ユーザのセッションとを紐付ける機能――に使われます。</p>
450    <p>この基本属性一覧に無いものについては、<code>req_attribute</code>
451    コード <code>0x0A</code> 経由で属性を何個でも送ることができます。
452    属性の名前と値の文字列の組を、それぞれこのコードの直後に送ります。
453    環境変数はこの方法で伝えられます。</p>
454    <p>最後に属性が全て送信された後に、属性の終端を示す <code>0xFF</code> 
455    が送出されます。この信号は属性の一覧の終わりを示すと同時に、リクエスト
456    パケットの終端をも示しています。</p>
457

460<h2><a name="resppacketstruct" id="resppacketstruct">レスポンスパケット構造</a></h2>
461    <p>コンテナがサーバに送り返すことのできるメッセージ:</p>
462    <div class="example"><pre>AJP13_SEND_BODY_CHUNK :=
463  prefix_code   3
464  chunk_length  (integer)
465  chunk        *(byte)
466  chunk_terminator (byte) Ox00
467
468AJP13_SEND_HEADERS :=
469  prefix_code       4
470  http_status_code  (integer)
471  http_status_msg   (string)
472  num_headers       (integer)
473  response_headers *(res_header_name header_value)
474
475res_header_name :=
476    sc_res_header_name | (string)   [see below for how this is parsed]
477
478sc_res_header_name := 0xA0 (byte)
479
480header_value := (string)
481
482AJP13_END_RESPONSE :=
483  prefix_code       5
484  reuse             (boolean)
485
486
487AJP13_GET_BODY_CHUNK :=
488  prefix_code       6
489  requested_length  (integer)</pre></div>
490  <h3>詳細 :</h3>
491  <h3>Send Body Chunk</h3>
492    <p>チャンクは基本的にはバイナリデータで、ブラウザに直接送られます。</p>
493  
494  <h3>Send Headers</h3>
495    <p>ステータスコードとメッセージが通常の HTTP の通信にはあります (例 
496    <code>200</code> と <code>OK</code>)。レスポンスヘッダ名は、
497    リクエストヘッダ名と同様の方法でエンコードされます。
498    コードと文字列の判別方法の詳細に関しては、上記の header_encoding 
499    を参照してください。
500    一般的なヘッダのコードは :</p>
501    <table>
502      <tr><td>名前</td><td>コードの値</td></tr>
503      <tr><td>Content-Type</td><td>0xA001</td></tr>
504      <tr><td>Content-Language</td><td>0xA002</td></tr>
505      <tr><td>Content-Length</td><td>0xA003</td></tr>
506      <tr><td>Date</td><td>0xA004</td></tr>
507      <tr><td>Last-Modified</td><td>0xA005</td></tr>
508      <tr><td>Location</td><td>0xA006</td></tr>
509      <tr><td>Set-Cookie</td><td>0xA007</td></tr>
510      <tr><td>Set-Cookie2</td><td>0xA008</td></tr>
511      <tr><td>Servlet-Engine</td><td>0xA009</td></tr>
512      <tr><td>Status</td><td>0xA00A</td></tr>
513      <tr><td>WWW-Authenticate</td><td>0xA00B</td></tr>
514    </table>
515    <p>コードかヘッダ文字列の直後には、ヘッダの値がエンコードされます。</p>
516  
517  <h3>End Response</h3>
518    <p>リクエスト処理サイクルの終了を知らせます。<code>reuse</code> フラグが真
519    <code>(==1)</code> の場合、現在使用している TCP コネクションは次の新しい
520    リクエストに使えるようになります。<code>reuse</code> が偽 (C のコードでは
521    1 以外の全て) の場合は、コネクションを閉じることになります。</p>
522  
523  <h3>Get Body Chunk</h3>
524    <p>(ボディのサイズが大きすぎて最初のパケットに収まらない場合や、
525    リクエストがチャンク転送された場合などには、) コンテナはリクエストからの
526    データ読み込み要求をします。サーバ側はそれに対して、最小 
527    <code>request_length</code> 最大 <code>(8186 (8 Kbytes - 6))</code> 
528    の範囲で、未転送で残っているリクエストボディの大きさのデータを
529    送り返します。<br />
530    ボディにそれ以上データが残っていない場合 (つまりサーブレットが
531    ボディの最後を超えて読み込もうとした場合) 、サーバは
532    ペイロード長 0 の<em>空パケット</em><code>(0x12,0x34,0x00,0x00)</code>
533    を送り返します。</p>
