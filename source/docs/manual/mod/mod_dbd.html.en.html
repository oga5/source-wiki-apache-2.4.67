37<h3>Summary</h3>
38
39    <p><code class="module"><a href="../mod/mod_dbd.html">mod_dbd</a></code> manages SQL database connections using
40    <a class="glossarylink" href="../glossary.html#apr" title="see glossary">APR</a>.  It provides database connections on request
41    to modules requiring SQL database functions, and takes care of
42    managing databases with optimal efficiency and scalability
43    for both threaded and non-threaded MPMs.  For details, see the
44    <a href="http://apr.apache.org/">APR</a> website.</p>

71<h2><a name="pooling" id="pooling">Connection Pooling</a></h2>
72    <p>This module manages database connections, in a manner
73    optimized for the platform.  On non-threaded platforms,
74    it provides a persistent connection in the manner of
75    classic LAMP (Linux, Apache, Mysql, Perl/PHP/Python).
76    On threaded platform, it provides an altogether more
77    scalable and efficient <em>connection pool</em>, as
78    described in <a href="http://www.apachetutor.org/dev/reslist">this
79    article at ApacheTutor</a>.  Note that <code class="module"><a href="../mod/mod_dbd.html">mod_dbd</a></code>
80    supersedes the modules presented in that article.</p>
81</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>

83<h2><a name="connecting" id="connecting">Connecting</a></h2>
84
85    <p>To connect to your database, you'll need to specify
86    a driver, and connection parameters. These vary from
87    one database engine to another. For example, to connect
88    to mysql, do the following:</p>
89
90<pre class="prettyprint lang-config">DBDriver mysql
91DBDParams host=localhost,dbname=pony,user=shetland,pass=appaloosa</pre>
92
93
94    <p>You can then use this connection in a variety of other
95    modules, including <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code>,
96    <code class="module"><a href="../mod/mod_authn_dbd.html">mod_authn_dbd</a></code>, and <code class="module"><a href="../mod/mod_lua.html">mod_lua</a></code>.
97    Further usage examples appear in each of those modules'
98    documentation.</p>
99
100    <p>See <code class="directive">DBDParams</code> for connection string
101    information for each of the supported database drivers.</p>

105<h2><a name="API" id="API">Apache DBD API</a></h2>
106    <p><code class="module"><a href="../mod/mod_dbd.html">mod_dbd</a></code> exports five functions for other modules
107    to use. The API is as follows:</p>
108
109<pre class="prettyprint lang-c">typedef struct {
110    apr_dbd_t *handle;
111    apr_dbd_driver_t *driver;
112    apr_hash_t *prepared;
113} ap_dbd_t;
114
115/* Export functions to access the database */
116
117/* acquire a connection that MUST be explicitly closed.
118 * Returns NULL on error
119 */
120AP_DECLARE(ap_dbd_t*) ap_dbd_open(apr_pool_t*, server_rec*);
121
122/* release a connection acquired with ap_dbd_open */
123AP_DECLARE(void) ap_dbd_close(server_rec*, ap_dbd_t*);
124
125/* acquire a connection that will have the lifetime of a request
126 * and MUST NOT be explicitly closed.  Return NULL on error.
127 * This is the preferred function for most applications.
128 */
129AP_DECLARE(ap_dbd_t*) ap_dbd_acquire(request_rec*);
130
131/* acquire a connection that will have the lifetime of a connection
132 * and MUST NOT be explicitly closed.  Return NULL on error.
133 */
134AP_DECLARE(ap_dbd_t*) ap_dbd_cacquire(conn_rec*);
135
136/* Prepare a statement for use by a client module */
137AP_DECLARE(void) ap_dbd_prepare(server_rec*, const char*, const char*);
138
139/* Also export them as optional functions for modules that prefer it */
140APR_DECLARE_OPTIONAL_FN(ap_dbd_t*, ap_dbd_open, (apr_pool_t*, server_rec*));
141APR_DECLARE_OPTIONAL_FN(void, ap_dbd_close, (server_rec*, ap_dbd_t*));
142APR_DECLARE_OPTIONAL_FN(ap_dbd_t*, ap_dbd_acquire, (request_rec*));
143APR_DECLARE_OPTIONAL_FN(ap_dbd_t*, ap_dbd_cacquire, (conn_rec*));
144APR_DECLARE_OPTIONAL_FN(void, ap_dbd_prepare, (server_rec*, const char*, const char*));</pre>

148<h2><a name="prepared" id="prepared">SQL Prepared Statements</a></h2>
149    <p><code class="module"><a href="../mod/mod_dbd.html">mod_dbd</a></code> supports SQL prepared statements on behalf
150    of modules that may wish to use them.  Each prepared statement
151    must be assigned a name (label), and they are stored in a hash:
152    the <code>prepared</code> field of an <code>ap_dbd_t</code>.
153    Hash entries are of type <code>apr_dbd_prepared_t</code>
154    and can be used in any of the apr_dbd prepared statement
155    SQL query or select commands.</p>
156
157    <p>It is up to dbd user modules to use the prepared statements
158    and document what statements can be specified in httpd.conf,
159    or to provide their own directives and use <code>ap_dbd_prepare</code>.</p>
160	
161	<div class="warning"><h3>Caveat</h3>
162	When using prepared statements with a MySQL database, it is preferred to set
163	<code>reconnect</code> to 0 in the connection string as to avoid errors that
164	arise from the MySQL client reconnecting without properly resetting the
165	prepared statements. If set to 1, any broken connections will be attempted
166	fixed, but as mod_dbd is not informed, the prepared statements will be invalidated.
167	</div>

170<h2><a name="security" id="security">SECURITY WARNING</a></h2>
171
172    <p>Any web/database application needs to secure itself against SQL
173    injection attacks.  In most cases, Apache DBD is safe, because
174    applications use prepared statements, and untrusted inputs are
175    only ever used as data.  Of course, if you use it via third-party
176    modules, you should ascertain what precautions they may require.</p>
177    <p>However, the <var>FreeTDS</var> driver is inherently
178    <strong>unsafe</strong>.  The underlying library doesn't support
179    prepared statements, so the driver emulates them, and the
180    untrusted input is merged into the SQL statement.</p>
181    <p>It can be made safe by <em>untainting</em> all inputs:
182    a process inspired by Perl's taint checking.  Each input
183    is matched against a regexp, and only the match is used,
184    according to the Perl idiom:</p>
185    <div class="example"><pre><code>  $untrusted =~ /([a-z]+)/;
186  $trusted = $1;</code></pre></div>
187    <p>To use this, the untainting regexps must be included in the
188    prepared statements configured.  The regexp follows immediately
189    after the % in the prepared statement, and is enclosed in
190    curly brackets {}.  For example, if your application expects
191    alphanumeric input, you can use:</p>
192    <div class="example"><p><code>
193       <code>"SELECT foo FROM bar WHERE input = %s"</code>
194    </code></p></div>
195    <p>with other drivers, and suffer nothing worse than a failed query.
196    But with FreeTDS you'd need:</p>
197    <div class="example"><p><code>
198       <code>"SELECT foo FROM bar WHERE input = %{([A-Za-z0-9]+)}s"</code>
199    </code></p></div>
200    <p>Now anything that doesn't match the regexp's $1 match is
201    discarded, so the statement is safe.</p>
202    <p>An alternative to this may be the third-party ODBC driver,
203    which offers the security of genuine prepared statements.</p>
204</div>
