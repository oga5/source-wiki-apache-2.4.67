40<h3>Sommaire</h3>
41
42    <p>Ce module fournit un filtre qui va traiter les fichiers avant
43    de les envoyer au client. Le traitement est contrôlé via des
44    commentaires SGML spécialement formatés, aussi nommés
45    <dfn>éléments</dfn>. Ces éléments permettent l'insertion
46    conditionnelle de texte, l'inclusion d'autres fichiers ou
47    programmes, ainsi que la définition et l'affichage de variables
48    d'environnement.</p>

80<h2><a name="enabling" id="enabling">Activation des SSI</a></h2>
81    
82
83    <p>Les SSI sont implémentés par le <a href="../filter.html">filtre</a> <code>INCLUDES</code>. Si des
84    documents contenant des directives SSI possèdent une extension
85    .shtml, les directives suivantes indiqueront à Apache de les
86    interpréter et d'assigner le type MIME
87    <code>text/html</code> au document obtenu :</p>
88
89    <pre class="prettyprint lang-config">AddType text/html .shtml
90AddOutputFilter INCLUDES .shtml</pre>
91
92
93    <p>L'option suivante doit être définie pour les répertoires qui
94    contiennent les fichiers shtml (en général dans une section
95    <code class="directive"><a href="../mod/core.html#directory">&lt;Directory&gt;</a></code>, mais
96    cette option peut également être définie dans un fichier
97    <code>.htaccess</code> si <code><code class="directive"><a href="../mod/core.html#allowoverride">AllowOverride</a></code> Options</code> a été défini pour le
98    répertoire considéré) :</p>
99
100    <pre class="prettyprint lang-config">Options +Includes</pre>
101
102
103    <p>Pour des raisons de compatibilité ascendante, le <a href="../handler.html">gestionnaire</a>  <code>server-parsed</code>
104    peut aussi activer le filtre INCLUDES. Ainsi, Apache va activer le
105    filtre INCLUDES pour tout document de type MIME
106    <code>text/x-server-parsed-html</code> ou
107    <code>text/x-server-parsed-html3</code> (et le document obtenu aura
108    pour type MIME <code>text/html</code>).</p>
109
110    <p>Pour plus d'informations, voyez notre <a href="../howto/ssi.html">Tutoriel SSI</a>.</p>

115

118    relatives au chemin en fin de requête). La directive <code class="directive"><a href="../mod/core.html#acceptpathinfo">AcceptPathInfo</a></code> permet de configurer le
119    serveur de façon à ce qu'il accepte ce genre de requête.</p>

122<h2><a name="elements" id="elements">Eléments disponibles</a></h2>
123    <p>Le document est interprété comme un document HTML, avec des
124    commandes spéciales incluses sous forme de commentaires SGML. La
125    syntaxe d'une commande est la suivante : </p>
126
127    <div class="example"><p><code>
128      &lt;!--#<var>élément</var> <var>attribut</var>=<var>valeur</var>
129      <var>attribut</var>=<var>valeur</var> ... --&gt;
130    </code></p></div>
131
132    <p>Les valeurs sont souvent entourées de guillemets, mais on peut
133    aussi utiliser des apostrophes (<code>'</code>) ou des apostrophes
134    inverses (<code>`</code>). De nombreuses commandes n'acceptent
135    qu'une seule paire attribut-valeur. Notez que le terminateur de
136    commentaire (<code>--&gt;</code>) doit être précédé d'un espace afin
137    d'être sûr qu'il ne soit pas considéré comme un élément de commande
138    SSI. Notez aussi que le délimiteur de début <code>&lt;!--#</code>
139    est <em>un</em> élément de commande et ne doit donc pas contenir
140    d'espace.</p>
141
142    <p>La table suivante contient la liste des éléments autorisés :</p>
143
144    <table class="bordered">
145    <tr><th>Elément</th><th>Description</th></tr>
146    <tr><td><code><a href="#element.comment">comment</a></code></td>
147        <td>commentaire SSI</td></tr>
148    <tr><td><code><a href="#element.config">config</a></code></td>
149        <td>configure les formats de sortie</td></tr>
150    <tr><td><code><a href="#element.echo">echo</a></code></td>
151        <td>affiche le contenu de variables</td></tr>
152    <tr><td><code><a href="#element.exec">exec</a></code></td>
153        <td>exécute des programmes externes</td></tr>
154    <tr><td><code><a href="#element.fsize">fsize</a></code></td>
155        <td>affiche la taille d'un fichier</td></tr>
156    <tr><td><code><a href="#element.flastmod">flastmod</a></code></td>
157        <td>affiche la date de dernière modification d'un fichier</td></tr>
158    <tr><td><code><a href="#element.include">include</a></code></td>
159        <td>inclut un fichier</td></tr>
160    <tr><td><code><a href="#element.printenv">printenv</a></code></td>
161        <td>affiche toutes les variables disponibles</td></tr>
162    <tr><td><code><a href="#element.set">set</a></code></td>
163        <td>définit la valeur d'une variable</td></tr>
164    </table>

166    <p>Les éléments SSI peuvent être définis par d'autres modules que
167    <code class="module"><a href="../mod/mod_include.html">mod_include</a></code>. À ce titre, l'élément <code><a href="#element.exec">exec</a></code> est fourni par
168    <code class="module"><a href="../mod/mod_cgi.html">mod_cgi</a></code>, et ne sera disponible que si ce module est
169    chargé.</p>

171    <h3><a name="element.comment" id="element.comment">L'élément comment</a></h3>
172      <p>Cette commande n'affiche aucune information. Elle n'a pour but que
173      l'ajout de commentaires dans un fichier et ces commentaires ne sont pas
174      affichés.</p>
175
176      <p>Cette syntaxe est disponible à partir de la version 2.4.21 du serveur
177      HTTP Apache.</p>
178
179      <div class="example"><p><code>
180      &lt;!--#comment Blah Blah Blah --&gt;<br />
181      &nbsp;&nbsp;&nbsp;or<br />
182      &lt;!--#comment text="Blah Blah Blah" --&gt;
183      </code></p></div>

186    <h3><a name="element.config" id="element.config">L'élément config</a></h3>
187      <p>Cette commande contrôle divers aspects de l'interprétation. Les
188      attributs valides sont :</p>
189
190      <dl>
191      <dt><code>echomsg</code> (<em>Versions 2.1 et supérieures
192      d'Apache</em>)</dt>
193      <dd><p>La valeur est un message qui sera envoyé au client si
194      l'élément <code><a href="#element.echo">echo</a></code> tente
195      d'afficher le contenu d'une variable non définie. Cet attribut
196      l'emporte sur toute directive <code class="directive"><a href="#ssiundefinedecho">SSIUndefinedEcho</a></code>.</p>
197
198      <div class="example"><p><code>
199        &lt;!--#config echomsg="[Valeur non définie]" --&gt;
200      </code></p></div>
201
202      </dd>
203
204      <dt><code>errmsg</code></dt>
205      <dd><p>La valeur est un message qui sera envoyé au client si une
206      erreur survient lors de l'interprétation du document. Cet attribut
207      l'emporte sur toute directive <code class="directive"><a href="#ssierrormsg">SSIErrorMsg</a></code>.</p>
208
209      <div class="example"><p><code>
210       &lt;!--#config errmsg="[Zut, quelque chose s'est mal passé.]" --&gt;
211      </code></p></div>
212
213      </dd>
214
215      <dt><code>sizefmt</code></dt>
216      <dd><p>La valeur définit l'unité employée lors de l'affichage de la
217      taille d'un fichier. Les valeurs possibles sont <code>bytes</code>
218      pour une taille en octets, ou <code>abbrev</code> pour une taille
219      en Ko ou Mo selon son importance ; par exemple, une taille de 1024
220      octets sera affichée sous la forme "1K".</p>
221
222      <div class="example"><p><code>
223      &lt;!--#config sizefmt="abbrev" --&gt;
224      </code></p></div>
225
226      </dd>
227
228      <dt><code>timefmt</code></dt>
229      <dd><p>La valeur est une chaîne que pourra utiliser la fonction de la
230      bibliothèque standard <code>strftime(3)</code> lors de l'affichage
231      des dates.</p>
232
233      <div class="example"><p><code>
234      &lt;!--#config timefmt=""%R, %B %d, %Y"" --&gt;
235      </code></p></div>
236
237      </dd>
238      
239      </dl>

242    <h3><a name="element.echo" id="element.echo">L'élément echo</a></h3>
243      <p>Cette commande affiche le contenu d'une des <a href="#includevars">variables include</a> définies ci-dessous. Si
244      la variable n'est pas définie, le résultat est déterminé par la
245      valeur de la directive <code class="directive"><a href="#ssiundefinedecho">SSIUndefinedEcho</a></code>. Le format d'affichage des dates est
246      défini par l'attribut <code>timefmt</code> de la commande
247      config.</p>
248
249      <p>Attributs:</p>
250
251      <dl>
252      <dt><code>var</code></dt>
253      <dd>La valeur est le nom de la variable à afficher.</dd>
254
255      <dt><code>decoding</code></dt>
256      <dd><p>Spécifie si Apache doit effectuer un décodage dans la
257      variable avant son traitement ultérieur. La valeur par défaut est
258      <code>none</code>, et dans ce cas, aucun décodage n'est effectué.
259      Si la valeur est <code>url</code>, un décodage de type URL sera
260      effectué (il s'agit du codage de type %-encoding utilisé dans les
261      URLs des liens, etc...). Si la valeur est <code>urlencoded</code>,
262      c'est un décodage des éléments de type
263      application/x-www-form-urlencode (que l'on trouve dans les chaînes
264      de paramètres) qui sera effectué. Si la valeur est
265      <code>base64</code>, un
266      decodage de type base64 sera effectué, et si elle est
267      <code>entity</code>, c'est un décodage des entités HTML qui sera
268      effectué. Ce décodage est effectué avant tout codage ultérieur de
269      la variable. Il est possible d'effectuer plusieurs décodages en
270      spécifiant plusieurs valeurs séparées par des virgules. Les
271      spécifications de décodages restent valables jusqu'au prochain
272      attribut de décodage, ou la fin de l'élément.</p>
273
274      <p>Pour être pris en compte, l'attribut de <code>décodage</code>
275      doit <em>précéder</em> l'attribut <code>var</code> correspondant.</p>
276      </dd>
277
278      <dt><code>encoding</code></dt>
279      <dd><p>Spécifie la manière dont Apache va coder les caractères
280      spéciaux que la variable contient avant leur affichage. S'il est
281      défini à <code>none</code>, aucun codage ne sera effectué. S'il
282      est défini à <code>url</code>, un codage de type URL sera effectué
283      (aussi connu sous le nom de codage avec caractères % , il convient
284      pour les URLS des liens, etc...). S'il est défini à
285      <code>urlencoded</code>, c'est un codage compatible
286      application/x-www-form-urlencoded qui sera effectué (à utiliser
287      dans les chaînes de paramètres). S'il est défini à
288      <code>base64</code>, c'est un encodage de type base64 qui sera
289      effectué. Au début d'un élément
290      <code>echo</code>, la valeur par défaut est définie à
291      <code>entity</code>, ce qui correspond à un codage de type entité
292      (codage qui convient pour un élément HTML de type bloc, comme le
293      paragraphe d'un texte). Cette valeur par défaut peut être modifiée
294      en ajoutant un attribut <code>encoding</code>, qui fera effet
295      jusqu'à la définition d'un nouvel attribut <code>encoding</code>
296      ou la fin de l'élément echo.</p>
297
298      <p>Pour produire son effet, l'attribut <code>encoding</code> doit
299      précéder l'attribut <code>var</code> concerné.</p>
300
301      <div class="warning">
302        Afin de prévenir les attaques de type cross-site scripting, il
303	est recommandé de <em>toujours</em> encoder les données fournies
304	par les utilisateurs.
305      </div>
306
307      <div class="example"><h3>Example</h3><p><code>
308        &lt;!--#echo encoding="entity" var="QUERY_STRING" --&gt;
309      </code></p></div>
310      </dd>
311      </dl>

314    <h3><a name="element.exec" id="element.exec">L'élément exec</a></h3>
315      <p>La commande <code>exec</code> exécute la commande shell ou le
316      script spécifié. Elle nécessite le chargement du module
317      <code class="module"><a href="../mod/mod_cgi.html">mod_cgi</a></code>. Si <code class="directive"><a href="../mod/core.html#options">Options</a></code> <code>IncludesNOEXEC</code> est
318      définie, cette commande est désactivée. Les attributs disponibles
319      sont :</p>
320
321      <dl>
322      <dt><code>cgi</code></dt>
323      <dd><p>La valeur spécifie un chemin URL vers le script CGI (encodé
324      avec caractères %). Si le chemin ne commence pas par un slash (/),
325      il est considéré comme relatif au document courant. Le document
326      référencé par ce chemin est invoqué en tant que script CGI, même
327      s'il n'est pas censé être reconnu comme tel par le serveur. Les
328      scripts CGI doivent cependant être activés dans le répertoire qui
329      contient les scripts (via la directive <code class="directive"><a href="../mod/mod_alias.html#scriptalias">ScriptAlias</a></code> ou l'<code class="directive"><a href="../mod/core.html#options">Options</a></code> <code>ExecCGI</code>).</p>
330
331      <p>Le <code>PATH_INFO</code> et la chaîne d'arguments
332      (<code>QUERY_STRING</code>) de la requête originale du client sont
333      fournis au script CGI ; ils <em>ne peuvent pas</em> être spécifiés
334      dans le chemin de l'URL. Le script disposera des variables include
335      en plus de l'environnement standard <a href="mod_cgi.html">CGI</a>.</p>
336
337      <div class="example"><h3>Exemple</h3><p><code>
338        &lt;!--#exec cgi="/cgi-bin/exemple.cgi" --&gt;
339      </code></p></div>
340
341      <p>Si, à la place d'un flux de sortie, le script renvoie un
342      en-tête <code>Location:</code>, ce dernier sera traduit en ancrage
343      HTML.</p>
344
345      <p>L'élément <code><a href="#includevirtual">include
346      virtual</a></code> doit être préféré à <code>exec cgi</code>. En
347      particulier, si vous devez transmettre des arguments
348      supplémentaires à un programme CGI en utilisant la chaîne
349      d'arguments de la requête, c'est impossible avec <code>exec
350      cgi</code>, mais vous pouvez y parvenir avec <code>include
351      virtual</code> comme suit :</p>
352
353      <div class="example"><p><code>
354        &lt;!--#include virtual="/cgi-bin/exemple.cgi?argument=valeur" --&gt;
355      </code></p></div>
356      </dd>
357
358      <dt><code>cmd</code></dt>
359      <dd><p>Le serveur va exécuter la commande fournie en utilisant
360      <code>/bin/sh</code>. La commande dispose des <a href="#includevars">variables include</a>, en plus du jeu habituel
361      de variables CGI.</p>
362
363      <p>Il est toujours préférable d'utiliser <code><a href="#includevirtual">#include virtual</a></code> à la place de
364      <code>#exec cgi</code> ou <code>#exec cmd</code>. <code>#include
365      virtual</code> utilise le mécanisme standard des sous-requêtes
366      d'Apache pour inclure des fichiers ou des scripts. Il a fait
367      l'objet de tests plus approfondis et sa maintenance est mieux
368      suivie.</p>
369
370      <p>De plus, sur certaines plate-formes, comme Win32, et sous unix,
371      si l'on utilise <a href="../suexec.html">suexec</a>, il est
372      impossible de transmettre des arguments à une commande dans une
373      directive <code>exec</code>, à moins d'insérer des espaces dans la
374      commande. Ainsi, alors que ce qui suit fonctionnera sous unix avec
375      une configuration sans suexec, l'effet produit ne sera pas celui
376      désiré sous Win32, ou dans le cas de l'utilisation de suexec
377      :</p>
378
379      <div class="example"><p><code>
380        &lt;!--#exec cmd="perl /chemin/vers/script_perl arg1 arg2" --&gt;
381      </code></p></div>
382      </dd>
383      </dl>

386    <h3><a name="element.fsize" id="element.fsize">L'élément fsize</a></h3>
387      <p>Cette commande permet d'afficher la taille du fichier spécifié
388      en fonction des spécifications de format de <code>sizefmt</code>.
389      Attributs :</p>
390
391      <dl>
392      <dt><code>file</code></dt>
393      <dd>La valeur est le chemin du fichier, relatif au répertoire
394      contenant le document en cours d'interprétation.
395
396      <div class="example"><p><code>
397        Ce fichier a une taille de &lt;!--#fsize file="mod_include.html"
398	--&gt; octets.
399    </code></p></div>
400
401    La valeur de <code>file</code> ne peut pas faire référence à un
402    fichier situé à un niveau supérieur de l'arborescence du répertoire
403    courant ou en dehors de la racine des documents ; il ne peut donc
404    ni commencer par un slash, ni contenir la séquence de caractères
405    <code>../</code>. Si c'est le cas, le message d'erreur <code>The
406    given path was above the root path</code> sera renvoyé.
407      </dd>
408
409      <dt><code>virtual</code></dt>
410      <dd>La valeur est un chemin URL (codé avec caractères %). S'il ne
411      commence pas par un slash (/), il est considéré comme relatif au
412      document courant. Notez que cette commande n'affiche <em>pas</em>
413      la taille de la sortie d'un programme CGI, mais la taille du
414      programme CGI lui-même.</dd>
415      </dl>
416
417      <div class="example"><p><code>
418        Ce fichier a une taille de &lt;!--#fsize
419	virtual="/docs/mod/mod_include.html" --&gt; octets.
420    </code></p></div>
421
422      <p>Notez que dans la plupart des cas, ces deux attributs sont
423      identiques. Cependant, l'attribut <code>file</code> ne respecte
424      pas les aliases URL-space.</p>
425     
426
427    <h3><a name="element.flastmod" id="element.flastmod">L'élément flastmod</a></h3>
428      <p>Cette commande permet d'afficher la date de dernière
429      modification du fichier spécifié, en fonction des spécifications
430      de format de <code>timefmt</code>. Les attributs sont les mêmes
431      que ceux de la commande <code><a href="#element.fsize">fsize</a></code>.</p>
432     
433
434    <h3><a name="element.include" id="element.include">L'élément include</a></h3>
435      <p>Cette commande permet d'insérer le texte d'un autre document ou
436      fichier dans le fichier en cours d'interprétation. Tout fichier
437      inclus est soumis au contrôle d'accès habituel. Si <a href="core.html#options">Options</a> <code>IncludesNOEXEC</code>
438      est défini pour le répertoire contenant le fichier
439      interprété, seuls les documents possèdant un
440      <a class="glossarylink" href="../glossary.html#type mime" title="voir glossaire">type MIME</a> de type texte
441      (<code>text/plain</code>, <code>text/html</code>, etc...) seront
442      inclus. Les scripts CGI, quant à eux, sont invoqués de manière
443      habituelle en utilisant l'URL complète fournie avec la commande, y
444      compris toute chaîne d'arguments éventuelle.</p>
445
446      <p>Un attribut définit le chemin du document à inclure, et peut
447      apparaître plusieurs fois dans l'élément à inclure ; en retour, pour
448      chaque attribut fourni à la commande include, une inclusion est
449      effectuée. Les attributs disponibles sont :</p>
450
451      <dl>
452      <dt><code>file</code></dt>
453      <dd>La valeur est un chemin relatif au répertoire contenant le
454      fichier en cours d'interprétation. Elle ne peut ni contenir
455      <code>../</code>, ni être un chemin absolu. Ainsi, vous ne pouvez
456      pas inclure de fichiers situés en dehors de l'arborescence du
457      site web ou dans un niveau supérieur à celui du fichier courant
458      dans cette arborescence. Il est toujours préférable d'utiliser
459      l'attribut <code>virtual</code>.</dd>
460
461      <dt><code><a id="includevirtual" name="includevirtual">virtual</a></code></dt>
462      <dd><p>La valeur est un chemin URL (codé avec caractères %). L'URL
463      ne peut contenir qu'un chemin et une chaîne d'arguments
464      éventuelle, à l'exclusion de tout protocole ou nom d'hôte. S'il ne
465      commence pas par un slash (/), il est considéré comme relatif au
466      document courant.</p>
467
468      <p>Une URL est construite à partir de l'attribut, et la sortie que
469      renverrait le serveur si l'URL était accédée par le client est
470      incluse dans la sortie interprétée. Les inclusions de fichiers
471      peuvent ainsi être imbriquées.</p>
472
473      <p>Si l'URL spécifiée correspond à un programme CGI, le programme
474      sera exécuté, et son flux de sortie inséré à la place de la
475      directive dans le fichier interprété. Vous pouvez insérer une
476      chaîne d'arguments dans une URL correspond à un programme CGI
477      :</p>
478
479      <div class="example"><p><code>
480        &lt;!--#include virtual="/cgi-bin/exemple.cgi?argument=valeur" --&gt;
481      </code></p></div>
482
483      <p><code>include virtual</code> doit être préféré à <code>exec
484      cgi</code> pour inclure le flux de sortie d'un programme CGI dans
485      un document HTML.</p>
486
487      <p>Si la directive <code class="directive"><a href="../mod/mod_request.html#keptbodysize">KeptBodySize</a></code> est correctement
488      définie et valide pour le fichier inclus, les tentatives de
489      requêtes POST vers le document HTML qui inclut des fichiers seront
490      transmises aux sous-requêtes en tant que requêtes POST
491      elles-mêmes. Sans cette directive, toutes les sous-requêtes sont
492      traitées en tant que requêtes GET.</p>
493
494      </dd>
495
496      <dt><code>onerror</code></dt>
497      <dd><p>La valeur est un chemin-URL (codé-%) qui est affiché si une
498      tentative précédente d'inclure un fichier ou un attribut virtuel a
499      échoué. Pour produire son effet, cet attribut doit être spécifié
500      après le fichier ou les attributs virtuels concernés. Si la
501      tentative d'inclure le chemin onerror échoue, ou si onerror n'est
502      pas spécifié, c'est le message d'erreur par défaut qui sera
503      inclus.</p>
504
505      <div class="example"><pre class="prettyprint lang-html"># Exemple simple<br />
506&lt;!--#include virtual="/not-exist.html" onerror="/error.html" --&gt;</pre>
507</div>
508
509      <div class="example"><pre class="prettyprint lang-html"># Chemins onerror dédiés<br />
510&lt;!--#include virtual="/path-a.html" onerror="/error-a.html" virtual="/path-b.html" onerror="/error-b.html" --&gt;</pre>
511</div>
512
513      </dd>
514      </dl>

520      d'être affichés (se reporter à l'élément <code><a href="#element.echo">echo</a></code> pour plus de détails). Cette
521      commande ne comporte pas d'attributs.</p>

525          &lt;!--#printenv --&gt;
526        &lt;/pre&gt;

530    <h3><a name="element.set" id="element.set">L'élément set</a></h3>
531      <p>Cette commande permet de définir la valeur d'une variable. Les
532      attributs sont :</p>
533
534      <dl>
535      <dt><code>var</code></dt>
536      <dd>Le nom de la variable à définir.</dd>
537
538      <dt><code>value</code></dt>
539      <dd>La valeur à affecter à la variable.</dd>
540      <dt><code>decoding</code></dt>
541      <dd><p>Spécifie si Apache doit effectuer un décodage dans la
542      variable avant son traitement ultérieur. La valeur par défaut est
543      <code>none</code>, et dans ce cas, aucun décodage n'est effectué.
544      Si la valeur est <code>url</code>, <code>urlencoded</code>,
545      <code>base64</code> ou
546      <code>entity</code>, c'est un décodage de type URL,
547      application/x-www-form-urlencoded, base64 ou
548      entité HTML qui sera respectivement effectué. Il est possible
549      d'effectuer plusieurs décodages en
550      spécifiant plusieurs valeurs séparées par des virgules. Les
551      spécifications de décodages restent valables jusqu'au prochain
552      attribut de décodage, ou la fin de l'élément. Pour être pris en
553      compte, l'attribut de <code>décodage</code>
554      doit <em>précéder</em> l'attribut <code>var</code> correspondant.</p>
555      </dd>
556
557      <dt><code>encoding</code></dt>
558      <dd><p>Spécifie la manière dont Apache va encoder les caractères
559      spéciaux que la variable contient avant leur affichage. S'il est
560      défini à <code>none</code>, aucun encodage ne sera effectué. Si la
561      valeur est <code>url</code>, <code>urlencoding</code>,
562      <code>base64</code> ou
563      <code>entity</code>, c'est un encodage de type URL,
564      application/x-www-form-urlencoded, base64 ou
565      entité HTML qui sera respectivement effectué. Il est possible de
566      spécifier plusieurs types d'encodage en les séparant par des
567      virgules. La spécification du type d'encodage fera effet
568      jusqu'à la définition d'un nouvel attribut encoding
569      ou la fin de l'élément. Pour produire son effet, l'attribut <code>encoding</code> doit
570      précéder l'attribut <code>var</code> concerné. Les encodages sont
571      effectués après les opérations de décodage.</p>
572      </dd>
573
574      </dl>
575
576      <div class="example"><h3>Exemple</h3><p><code>
577        &lt;!--#set var="category" value="help" --&gt;
578      </code></p></div>

641<h2><a name="substitution" id="substitution">Substitution de variable</a></h2>
642
643    <p>Une substitution de variable à l'intérieur d'une chaîne entre
644    guillemets s'effectue dans la plupart des situations où cette
645    dernière peut raisonablement constituer un argument d'une directive
646    SSI. Sont concernées les directives <code>config</code>,
647    <code>exec</code>, <code>flastmod</code>, <code>fsize</code>,
648    <code>include</code>, <code>echo</code>, et <code>set</code>. Si la
649    directive <code class="directive"><a href="#ssilegacyexprparser">SSILegacyExprParser</a></code> est définie à
650    <code>on</code>, la substitution s'effectue aussi dans les arguments
651    des opérateurs conditionnels. Vous pouvez insérer
652    un signe dollar en tant que caractère littéral dans une chaîne en
653    utilisant un anti-slash :</p>
654
655    <div class="example"><p><code>
656      &lt;!--#set var="cur" value="\$test" --&gt;
657    </code></p></div>
658
659    <p>Si une référence de variable doit être substituée au beau milieu
660    d'une séquence de caractères qui pourrait être elle-même considérée
661    comme un identifiant valide, l'ambiguïté peut être levée en
662    entourant la référence d'accolades, à la manière du shell :</p>
663
664    <div class="example"><p><code>
665      &lt;!--#set var="Zed" value="${REMOTE_HOST}_${REQUEST_METHOD}" --&gt;
666    </code></p></div>
667
668    <p>Dans cet exemple, la variable <code>Zed</code> se verra affecter
669    la valeur "<code>X_Y</code>" si <code>REMOTE_HOST</code> et
670    <code>REQUEST_METHOD</code> contiennent respectivement
671    "<code>X</code>" et "<code>Y</code>".</p>
672

674<div class="section">
675<h2><a name="flowctrl" id="flowctrl">Eléments de contrôle d'inclusion conditionnelle</a></h2>
676    
677
678    <p>Les éléments de base du contrôle d'inclusion conditionnelle sont
679    :</p>
680
681    <div class="example"><pre class="prettyprint lang-html">&lt;!--#if expr="<var>test_condition</var>" --&gt;<br />
682&lt;!--#elif expr="<var>test_condition</var>" --&gt;<br />
683&lt;!--#else --&gt;<br />
684&lt;!--#endif --&gt;</pre>
685</div>
686
687    <p>L'élément <code>if</code> fonctionne de la même manière que
688    la directive if d'un langage de programmation. La condition est
689    évaluée et si le résultat est vrai, le texte qui suit jusqu'au
690    prochain élément <code>elif</code>, <code>else</code> ou
691    <code>endif</code> sera inclus dans le flux de sortie.</p>
692
693    <p>Les éléments <code>elif</code> ou <code>else</code> permettent
694    d'insérer du texte dans le flux de sortie si
695    <var>test_condition</var> s'est révélé faux. Ces éléments sont
696    optionnels.</p>
697
698    <p>L'élément <code>endif</code> termine le bloc de traitement
699    conditionnel <code>if</code> et est obligatoire.</p>
700
701    <p><var>test_condition</var> est une expression booléenne qui
702    emprunte la syntaxe <a href="../expr.html">ap_expr</a>. La directive
703    <code class="directive"><a href="#ssilegacyexprparser">SSILegacyExprParser</a></code>
704    permet de modifier cette syntaxe pour la rendre compatible avec
705    Apache HTTPD 2.2.x.</p>
706
707    <p>Le jeu de variables SSI avec l'élément <code>var</code> sont
708    exportées vers l'environnement de la requête et sont accessibles via
709    la fonction <code>reqenv</code>. Pour faire simple, le nom de
710    fonction <code>v</code> est aussi disponible dans le module
711    <code class="module"><a href="../mod/mod_include.html">mod_include</a></code>.</p>
712
713    <p>Dans l'exemple suivant, "depuis le réseau local"  sera affiché si
714    l'adresse IP du client appartient au sous-réseau 10.0.0.0/8.</p>
715
716    <div class="example"><pre class="prettyprint lang-html">&lt;!--#if expr='-R "10.0.0.0/8"' --&gt;<br />
717  depuis le réseau local<br />
718&lt;!--#else --&gt;<br />
719  depuis ailleurs<br />
720&lt;!--#endif --&gt;</pre>
721</div>
722
723    <p>Dans l'exemple suivant, "foo vaut bar" sera affiché si la variable
724    <code>foo</code> contient la valeur "bar".</p>
725
726    <div class="example"><pre class="prettyprint lang-html">&lt;!--#if expr='v("foo") = "bar"' --&gt;<br />
727  foo vaut bar<br />
728&lt;!--#endif --&gt;</pre>
729</div>
730
731    <div class="note"><h3>Documentation de référence</h3>
732    <p>Voir aussi <a href="../expr.html">Les expressions dans le serveur
733    HTTP Apache</a> pour une référence complète et des exemples. Les
734    fonctions <em>restricted</em> ne sont pas disponibles dans
735    <code class="module"><a href="../mod/mod_include.html">mod_include</a></code>.</p>
736    </div>

738<div class="section">
739<h2><a name="legacyexpr" id="legacyexpr">Syntaxe des expressions héritée</a></h2>
740    
741
742    <p>Cette section décrit la syntaxe de l'élément <code>#if
743    expr</code> dans le cas où la directive <code class="directive"><a href="#ssilegacyexprparser">SSILegacyExprParser</a></code> est définie à
744    <code>on</code>.</p>
745
746    <dl>
747      <dt><code><var>chaîne</var></code></dt>
748      <dd>vrai si <var>chaîne</var> n'est pas vide</dd>
749
750      <dt><code><var>-A string</var></code></dt>
751      <dd><p>vrai si l'URL que contient la chaîne est accessible du
752      point de vue de la configuration, faux sinon. Il
753      s'avère utile lorsqu'un lien vers une URL doit être caché aux
754      utilisateurs qui ne sont pas autorisés à voir cette URL. Notez que
755      le test porte sur l'autorisation d'accès à l'URL, et non sur son
756      existence.</p>
757
758      <div class="example"><h3>Exemple</h3><pre class="prettyprint lang-html">&lt;!--#if expr="-A /prive" --&gt;<br />
759  Cliquez &lt;a href="/prive"&gt;ici&lt;/a&gt; pour accéder aux
760  informations privées.<br />
761&lt;!--#endif --&gt;</pre>
762</div>
763      </dd>
764
765      <dt><code><var>chaîne1</var> = <var>chaîne2</var><br />
766      <var>chaîne1</var> == <var>chaîne2</var><br />
767      <var>chaîne1</var> != <var>chaîne2</var></code></dt>
768
769      <dd><p>Compare <var>chaîne1</var> à <var>chaîne2</var>. Si
770      <var>chaîne2</var> est de la forme
771      <code>/<var>chaîne2</var>/</code>, elle est traitée comme une
772      expression rationnelle. Les expressions rationnelles sont
773      implémentées par le moteur <a href="http://www.pcre.org">PCRE</a>
774      et possèdent la même syntaxe que celles de <a href="http://www.perl.com">perl 5</a>. Notez que <code>==</code>
775      n'est qu'un alias pour <code>=</code> et se comporte exactement de
776      la même manière que ce dernier.</p>
777
778      <p>Si vous faites une comparaison directe (<code>=</code> ou
779      <code>==</code>), vous pouvez extraire des parties de l'expression
780      rationnelle. Les parties extraites sont stockées dans les
781      variables spéciales <code>$1</code> .. <code>$9</code>. L'ensemble
782      de la chaîne correspondant à l'expression rationnelle est stocké
783      dans la variable spéciale <code>$0</code>.</p>
784
785      <div class="example"><h3>Exemple</h3><pre class="prettyprint lang-html">&lt;!--#if expr="$QUERY_STRING = /^sid=([a-zA-Z0-9]+)/" --&gt;<br />
786&lt;!--#set var="session" value="$1" --&gt;<br />
787&lt;!--#endif --&gt;</pre>
788</div>
789      </dd>
790
791      <dt><code><var>chaîne1</var> &lt; <var>chaîne2</var><br />
792       <var>chaîne1</var> &lt;= <var>chaîne2</var><br />
793       <var>chaîne1</var> &gt; <var>chaîne2</var><br />
794       <var>chaîne1</var> &gt;= <var>chaîne2</var></code></dt>
795
796      <dd>Compare <var>chaîne1</var> à <var>chaîne2</var>. Notez que les
797      chaînes sont comparées de manière <em>littérale</em> (en utilisant
798      <code>strcmp(3)</code>). Ainsi, la chaîne "100" est inférieure à
799      "20".</dd>
800
801      <dt><code>( <var>test_condition</var> )</code></dt>
802      <dd>vrai si <var>test_condition</var> est vrai</dd>
803
804      <dt><code>! <var>test_condition</var></code></dt>
805      <dd>vrai si <var>test_condition</var> est faux</dd>
806
807      <dt><code><var>test_condition1</var> &amp;&amp;
808        <var>test_condition2</var></code></dt>
809      <dd>vrai si <var>test_condition1</var> et
810      <var>test_condition2</var> sont tous les deux vrais</dd>
811
812      <dt><code><var>test_condition1</var> ||
813        <var>test_condition2</var></code></dt>
814      <dd>vrai si au moins un des tests <var>test_condition1</var> ou
815      <var>test_condition2</var> est vrai</dd>
816    </dl>
817
818    <p>"<code>=</code>" et "<code>!=</code>" ont une priorité supérieure
819    à "<code>&amp;&amp;</code>" et "<code>||</code>". "<code>!</code>" a
820    la priorité la plus haute. Ainsi, les deux directives suivantes sont
821    équivalentes :</p>
822
823    <div class="example"><pre class="prettyprint lang-html">&lt;!--#if expr="$a = test1 &amp;&amp; $b = test2" --&gt;<br />
824&lt;!--#if expr="($a = test1) &amp;&amp; ($b = test2)" --&gt;</pre>
825</div>
826
827    <p>Les opérateurs booléens <code>&amp;&amp;</code> et
828    <code>||</code> ont la même priorité. Ainsi, si vous voulez
829    augmenter la priorité d'un de ces opérateurs, vous devez utiliser
830    des parenthèses.</p>
831
832    <p>Tout ce qui n'est pas reconnu comme variable ou opérateur est
833    traité comme une chaîne. Les chaînes peuvent aussi être entourées
834    d'apostrophes : <code>'chaîne'</code>. Les chaînes sans apostrophe
835    ne peuvent pas contenir d'espaces (espaces ou tabulations) car
836    ceux-ci servent à séparer certains éléments comme les variables. Si
837    plusieurs chaînes se trouvent dans une ligne, elles sont concaténées
838    en utilisant des espaces. Ainsi,</p>
839
840    <div class="example"><p><code><var>chaîne1</var>&nbsp;&nbsp;&nbsp;&nbsp;<var>chaîne2</var></code> devient <code><var>chaîne1</var>&nbsp;<var>chaîne2</var></code><br />
841      <br />
842      et<br />
843      <br />
844      <code>'<var>chaîne1</var>&nbsp;&nbsp;&nbsp;&nbsp;<var>chaîne2</var>'</code> devient <code><var>chaîne1</var>&nbsp;&nbsp;&nbsp;&nbsp;<var>chaîne2</var></code>.</p></div>
845
846    <div class="note"><h3>Optimisation des expressions booléennes</h3>
847      <p>Si les expressions atteignent une complexité suffisante pour
848      ralentir les traitements de manière significative, vous pouvez
849      essayer de les optimiser en fonction des règles d'évaluation :</p>
850      <ul>
851      <li>Les expressions sont évaluées de la gauche vers la droite</li>
852      <li>Les opérateurs booléens binaires (<code>&amp;&amp;</code> et
853      <code>||</code>) font l'objet d'une évaluation abrégée chaque fois
854      que cela est possible. En d'autres termes, et selon la règle
855      ci-dessus, <code class="module"><a href="../mod/mod_include.html">mod_include</a></code> évalue tout d'abord la
856      partie gauche de l'expression. Si le résultat de l'évaluation de
857      cette partie gauche suffit à déterminer le résultat final,
858      l'évaluation s'arrête ici. Dans le cas contraire, la partie droite
859      est évaluée, et le résultat final tient compte des résultats des
860      évaluations des parties gauche et droite.</li>
861      <li>L'évaluation abrégée est désactivée tant qu'il reste des
862      expressions régulières à traiter. Ces dernières doivent être
863      évaluées afin de définir les variables correspondant aux
864      références arrières (<code>$1</code> .. <code>$9</code>).</li>
865      </ul>
866      <p>Si vous voulez déterminer la manière dont une expression est
867      traitée, vous pouvez recompiler <code class="module"><a href="../mod/mod_include.html">mod_include</a></code> en
868      utilisant l'option de compilation <code>-DDEBUG_INCLUDE</code>.
869      Ceci a pour effet d'insérer, pour chaque expression interprétée,
870      des informations étiquetées, l'arbre d'interprétation et la
871      manière dont elle est évaluée au sein du flux de sortie envoyé au
872      client.</p>
873    </div>

876    rationnelles</h3>
877     <p>Tous les caractères slashes qui ne sont pas des séparateurs dans
878     votre expression rationnelle doivent être échappés, et ceci sans
879     tenir compte de leur signification du point de vue du moteur
880     d'expressions rationnelles.</p>

1156    l'interprétation des documents html standards. Elle n'affecte que
1157    les fichiers dont le <a class="glossarylink" href="../glossary.html#type mime" title="voir glossaire">type MIME</a> est
1158    <code>text/html</code>. <code class="directive">XBitHack</code> peut prendre
1159    les valeurs suivantes :</p>
1160
1161    <dl>
1162      <dt><code>off</code></dt>
1163      <dd>Aucun traitement particulier pour les fichiers
1164      exécutables.</dd>
1165
1166      <dt><code>on</code></dt>
1167      <dd>Tout fichier <code>text/html</code> dont le bit d'exécution
1168      est positionné pour le propriétaire sera traité en tant que
1169      document html interprété par le serveur.</dd>
1170
1171      <dt><code>full</code></dt>
1172      <dd>Identique à <code>on</code>, avec test du bit d'exécution pour
1173      le groupe. Si ce dernier est positionné, la date de <code>dernière
1174      modification</code> du fichier renvoyé est définie à la date de
1175      dernière modification du fichier. Dans le cas contraire, aucune
1176      date de dernière modification n'est renvoyée. Le positionnement de
1177      ce bit permet aux clients et aux mandataires de gérer la mise en
1178      cache du résultat de la requête.
1179
1180      <div class="note"><h3>Note</h3>
1181      <p>Il est recommandé de n'utiliser l'option full que dans le cas
1182      où vous êtes certain que le bit d'exécution du groupe est non
1183      positionné pour les scripts SSI qui pourraient effectuer l'<code>#include</code> d'un programme CGI ou bien produire des sorties
1184      différentes à chaque accès (ou seraient susceptibles d'être
1185      modifiées au cours des requêtes ultérieures).</p>
1186
1187      <p>Lorsqu'elle est définie à <code>on</code>, la directive
1188      <code class="directive"><a href="#ssilastmodified">SSILastModified</a></code>
1189      l'emporte sur la directive <code class="directive"><a href="#xbithack">XBitHack</a></code>.</p>
1190      </div>

1192      </dd>
