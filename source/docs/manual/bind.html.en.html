44<h2><a name="overview" id="overview">Overview</a></h2>
45    
46
47    <table class="related"><tr><th>Related Modules</th><th>Related Directives</th></tr><tr><td><ul><li><code class="module"><a href="./mod/core.html">core</a></code></li><li><code class="module"><a href="./mod/mpm_common.html">mpm_common</a></code></li></ul></td><td><ul><li><code class="directive"><a href="./mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code></li><li><code class="directive"><a href="./mod/mpm_common.html#listen">Listen</a></code></li></ul></td></tr></table>
48
49
50    <p>When httpd starts, it binds to some port and address on
51    the local machine and waits for incoming requests. By default,
52    it listens to all addresses on the machine. However, it may need to
53    be told to listen on specific ports, or only on selected
54    addresses, or a combination of both. This is often combined with the
55    <a href="vhosts/">Virtual Host</a> feature, which determines how
56    <code>httpd</code> responds to different IP addresses, hostnames and
57    ports.</p>
58
59    <p>The <code class="directive"><a href="./mod/mpm_common.html#listen">Listen</a></code>
60    directive tells the server to accept
61    incoming requests only on the specified port(s) or
62    address-and-port combinations. If only a port number is
63    specified in the <code class="directive"><a href="./mod/mpm_common.html#listen">Listen</a></code>
64    directive, the server listens to the given port on all interfaces.
65    If an IP address is given as well as a port, the server will listen
66    on the given port and interface. Multiple <code class="directive"><a href="./mod/mpm_common.html#listen">Listen</a></code> directives may be used to
67    specify a number of addresses and ports to listen on. The
68    server will respond to requests from any of the listed
69    addresses and ports.</p>
70
71    <p>For example, to make the server accept connections on both
72    port 80 and port 8000, on all interfaces, use:</p>
73
74    <div class="example"><pre class="prettyprint lang-config">Listen 80
75Listen 8000</pre>
76</div>
77
78    <p>To make the server accept connections on port 80 for one interface,
79       and port 8000 on another, use</p>
80
81    <div class="example"><pre class="prettyprint lang-config">Listen 192.0.2.1:80
82Listen 192.0.2.5:8000</pre>
83</div>
84
85    <p>IPv6 addresses must be enclosed in square brackets, as in the
86    following example:</p>
87
88    <div class="example"><pre class="prettyprint lang-config">Listen [2001:db8::a00:20ff:fea7:ccea]:80</pre>
89</div>
90
91    <div class="warning"><p>Overlapping <code class="directive"><a href="./mod/mpm_common.html#listen">Listen</a></code> directives will result in a
92    fatal error which will prevent the server from starting up.</p>
93
94    <div class="example"><p><code>
95      (48)Address already in use: make_sock: could not bind to address [::]:80
96    </code></p></div>
97
98    <p>See <a href="https://cwiki.apache.org/confluence/display/httpd/CouldNotBindToAddress">the
99    discussion in the wiki</a> for further troubleshooting tips.</p>

105<h2><a name="reload" id="reload">Changing Listen configuration on restart</a></h2>
106    
107
108    <p>When httpd is restarted, special consideration must be made for
109    changes to <code class="directive"><a href="./mod/mpm_common.html#listen">Listen</a></code> directives.  During a restart, httpd keeps ports
110    bound (as in the original configuration) to avoid generating
111    "Connection refused" errors for any new attempts to connect to the
112    server. If changes are made to the set of <code class="directive"><a href="./mod/mpm_common.html#listen">Listen</a></code> directives used
113    which conflict with the old configuration, configuration will fail
114    and the server will terminate.</p>
115
116    <p>For example, changing from configuration:</p>
117    
118    <div class="example"><pre class="prettyprint lang-config">Listen 127.0.0.1:80</pre>
119</div>
120
121    <p>to the following may fail, because binding to port 80 across
122    all addresses conflicts with binding to port 80 on just
123    127.0.0.1.</p>
124    
125    <div class="example"><pre class="prettyprint lang-config">Listen 80</pre>
126</div>
127
128    <p>To have such configuration changes take effect, it is necessary
129    to stop and then start the server.</p>
130

133<h2><a name="ipv6" id="ipv6">Special IPv6 Considerations</a></h2>
134    
135
136    <p>A growing number of platforms implement IPv6, and
137    <a class="glossarylink" href="./glossary.html#apr" title="see glossary">APR</a> supports IPv6 on most of these platforms,
138    allowing httpd to allocate IPv6 sockets, and to handle requests sent
139    over IPv6.</p>
140
141    <p>One complicating factor for httpd administrators is whether or
142    not an IPv6 socket can handle both IPv4 connections and IPv6
143    connections. Handling IPv4 connections with an IPv6 socket uses
144    IPv4-mapped IPv6 addresses, which are allowed by default on most
145    platforms, but are disallowed by default on FreeBSD, NetBSD, and
146    OpenBSD, in order to match the system-wide policy on those
147    platforms. On systems where it is disallowed by default, a
148    special <code class="program"><a href="./programs/configure.html">configure</a></code> parameter can change this behavior
149    for httpd.</p>
150
151    <p>On the other hand, on some platforms, such as Linux and Tru64, the
152    <strong>only</strong> way to handle both IPv6 and IPv4 is to use
153    mapped addresses. If you want <code>httpd</code> to handle IPv4 and IPv6 connections
154    with a minimum of sockets, which requires using IPv4-mapped IPv6
155    addresses, specify the <code>--enable-v4-mapped</code> <code class="program"><a href="./programs/configure.html">configure</a></code> option.</p>
156
157    <p><code>--enable-v4-mapped</code> is the default on all platforms except
158    FreeBSD, NetBSD, and OpenBSD, so this is probably how your httpd was
159    built.</p>
160
161    <p>If you want httpd to handle IPv4 connections only, regardless of
162    what your platform and APR will support, specify an IPv4 address on all
163    <code class="directive"><a href="./mod/mpm_common.html#listen">Listen</a></code> directives, as in the
164    following examples:</p>
165
166    <div class="example"><pre class="prettyprint lang-config">Listen 0.0.0.0:80
167Listen 192.0.2.1:80</pre>
168</div>
169
170    <p>If your platform supports it and you want httpd to handle IPv4 and
171    IPv6 connections on separate sockets (i.e., to disable IPv4-mapped
172    addresses), specify the <code>--disable-v4-mapped</code> <code class="program"><a href="./programs/configure.html">configure</a></code> option. <code>--disable-v4-mapped</code> is the
173    default on FreeBSD, NetBSD, and OpenBSD.</p>

176<h2><a name="protocol" id="protocol">Specifying the protocol with Listen</a></h2>
177    
178    <p>The optional second <var>protocol</var> argument of
179       <code class="directive"><a href="./mod/mpm_common.html#listen">Listen</a></code>
180       is not required for most
181       configurations. If not specified, <code>https</code> is the default for
182       port 443 and <code>http</code> the default for all other ports.  The
183       protocol is used to determine which module should handle a request, and
184       to apply protocol specific optimizations with the
185       <code class="directive"><a href="./mod/core.html#acceptfilter">AcceptFilter</a></code> directive.</p>
186
187    <p>You only need to set the protocol if you are running on non-standard
188       ports.  For example, running an <code>https</code> site on port 8443:</p>
189
190    <div class="example"><pre class="prettyprint lang-config">Listen 192.170.2.1:8443 https</pre>
191</div>

194<h2><a name="virtualhost" id="virtualhost">How This Works With Virtual Hosts</a></h2>
195    
196
197    <p> The <code class="directive"><a href="./mod/mpm_common.html#listen">Listen</a></code> directive does not implement
198    Virtual Hosts - it only tells the
199    main server what addresses and ports to listen on. If no
200    <code class="directive"><a href="./mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code>
201    directives are used, the server will behave
202    in the same way for all accepted requests. However,
203    <code class="directive"><a href="./mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code>
204    can be used to specify a different behavior
205    for one or more of the addresses or ports. To implement a
206    VirtualHost, the server must first be told to listen to the
207    address and port to be used. Then a
208    <code class="directive"><a href="./mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code> section
209    should be created for the specified address and port to set the
210    behavior of this virtual host. Note that if the
211    <code class="directive"><a href="./mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code>
212    is set for an address and port that the
213    server is not listening to, it cannot be accessed.</p>
