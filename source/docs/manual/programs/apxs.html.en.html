32    <p><code class="program"><a href="../programs/apxs.html">apxs</a></code> is a tool for building and installing extension
33    modules for the Apache HyperText Transfer Protocol (HTTP) server. This is
34    achieved by building a dynamic shared object (DSO) from one or more source
35    or object <var>files</var> which then can be loaded into the Apache server
36    under runtime via the <code class="directive"><a href="../mod/mod_so.html#loadmodule">LoadModule</a></code>
37    directive from <code class="module"><a href="../mod/mod_so.html">mod_so</a></code>.</p>
38
39    <p>So to use this extension mechanism your platform has to support the DSO
40    feature and your Apache <code class="program"><a href="../programs/httpd.html">httpd</a></code> binary has to be built with the
41    <code class="module"><a href="../mod/mod_so.html">mod_so</a></code> module. The <code class="program"><a href="../programs/apxs.html">apxs</a></code> tool automatically
42    complains if this is not the case. You can check this yourself by manually
43    running the command</p>
44
45    <div class="example"><pre class="prettyprint lang-sh">$ httpd -l</pre>
46</div>
47
48    <p>The module <code class="module"><a href="../mod/mod_so.html">mod_so</a></code> should be part of the displayed list.
49    If these requirements are fulfilled you can easily extend your Apache
50    server's functionality by installing your own modules with the DSO mechanism
51    by the help of this <code class="program"><a href="../programs/apxs.html">apxs</a></code> tool:</p>
52
53    <div class="example"><pre class="prettyprint lang-sh">$ apxs -i -a -c mod_foo.c
54gcc -fpic -DSHARED_MODULE -I/path/to/apache/include -c mod_foo.c
55ld -Bshareable -o mod_foo.so mod_foo.o
56cp mod_foo.so /path/to/apache/modules/mod_foo.so
57chmod 755 /path/to/apache/modules/mod_foo.so
58[activating module `foo' in /path/to/apache/etc/httpd.conf]
59$ apachectl restart
60/path/to/apache/sbin/apachectl restart: httpd not running, trying to start
61[Tue Mar 31 11:27:55 1998] [debug] mod_so.c(303): loaded module foo_module
62/path/to/apache/sbin/apachectl restart: httpd started
63$ _</pre>
64</div>
65
66    <p>The arguments <var>files</var> can be any C source file (.c), a object
67    file (.o) or even a library archive (.a). The <code class="program"><a href="../programs/apxs.html">apxs</a></code> tool
68    automatically recognizes these extensions and  automatically used the C
69    source files for compilation while just using the object and archive files
70    for the linking phase. But when using such pre-compiled objects make sure
71    they are compiled for position independent code (PIC) to be able to use them
72    for a dynamically loaded shared object. For instance with GCC you always
73    just have to use <code>-fpic</code>. For other C compilers consult its
74    manual page or at watch for the flags <code class="program"><a href="../programs/apxs.html">apxs</a></code> uses to compile the
75    object files.</p>
76
77    <p>For more details about DSO support in Apache read the documentation of
78    <code class="module"><a href="../mod/mod_so.html">mod_so</a></code> or perhaps even read the
79    <code>src/modules/standard/mod_so.c</code> source file.</p>

123<h2><a name="options" id="options">Options</a></h2>
124    <h3><a name="options.common" id="options.common">Common Options</a></h3>
125      <dl>
126      <dt><code>-n <var>modname</var></code></dt>
127      <dd>This explicitly sets the module name for the <code>-i</code> (install)
128      and <code>-g</code> (template generation) option. Use this to explicitly
129      specify the module name. For option <code>-g</code> this is required, for
130      option <code>-i</code> the <code class="program"><a href="../programs/apxs.html">apxs</a></code> tool tries to determine the
131      name from the source or (as a fallback) at least by guessing it from the
132      filename.</dd>
133      </dl>
134    
135
136    <h3><a name="options.query" id="options.query">Query Options</a></h3>
137      <dl>
138      <dt><code>-q</code></dt>
139      <dd>Performs a query for variables and environment settings used to
140      build <code>httpd</code>.  When invoked without <var>query</var> parameters,
141      it prints all known variables and their values.  The optional <code>-v</code>
142      parameter formats the list output.
143
144      <p>Use this to manually determine settings used to build the
145      <code>httpd</code> that will load your module.  For instance use</p>
146      <div class="example"><pre class="prettyprint lang-sh">INC=-I`apxs -q INCLUDEDIR`</pre>
147</div>
148
149      <p>inside your own Makefiles if you need manual access to Apache's C
150      header files.</p></dd>
151      </dl>
152

154    <h3><a name="options.conf" id="options.conf">Configuration Options</a></h3>
155      <dl>
156      <dt><code>-S <var>name</var>=<var>value</var></code></dt>
157      <dd>This option changes the apxs settings described above.</dd>
158      </dl>
159    
160
161    <h3><a name="options.template" id="options.template">Template Generation Options</a></h3>
162      <dl>
163      <dt><code>-g</code></dt>
164      <dd>This generates a subdirectory <var>name</var> (see option
165      <code>-n</code>) and there two files: A sample module source file named
166      <code>mod_<var>name</var>.c</code> which can be used as a template for
167      creating your own modules or as a quick start for playing with the
168      apxs mechanism. And a corresponding <code>Makefile</code> for even easier
169      build and installing of this module.</dd>
170      </dl>
171    
172
173    <h3><a name="options.dso" id="options.dso">DSO Compilation Options</a></h3>
174      <dl>
175      <dt><code>-c</code></dt>
176      <dd>This indicates the compilation operation. It first compiles the C
177      source files (.c) of <var>files</var> into corresponding object files (.o)
178      and then builds a dynamically shared object in <var>dsofile</var> by
179      linking these object files plus the remaining object files (.o and .a) of
180      <var>files</var>. If no <code>-o</code> option is specified the output
181      file is guessed from the first filename in <var>files</var> and thus
182      usually defaults to <code>mod_<var>name</var>.so</code>.</dd>
183
184      <dt><code>-o <var>dsofile</var></code></dt>
185      <dd>Explicitly specifies the filename of the created dynamically shared
186      object. If not specified and the name cannot be guessed from the
187      <var>files</var> list, the fallback name <code>mod_unknown.so</code> is
188      used.</dd>
189
190      <dt><code>-D <var>name</var>=<var>value</var></code></dt>
191      <dd>This option is directly passed through to the compilation command(s).
192      Use this to add your own defines to the build process.</dd>
193
194      <dt><code>-I <var>incdir</var></code></dt>
195      <dd>This option is directly passed through to the compilation command(s).
196      Use this to add your own include directories to search to the build
197      process.</dd>
198
199      <dt><code>-L <var>libdir</var></code></dt>
200      <dd>This option is directly passed through to the linker command. Use this
201      to add your own library directories to search to the build  process.</dd>
202
203      <dt><code>-l <var>libname</var></code></dt>
204      <dd>This option is directly passed through to the linker command. Use this
205      to add your own libraries to search to the build process.</dd>
206
207      <dt><code>-Wc,<var>compiler-flags</var></code></dt>
208      <dd>This option passes <var>compiler-flags</var> as additional flags to
209      the <code>libtool --mode=compile</code> command. Use this to add local
210      compiler-specific options.</dd>
211
212      <dt><code>-Wl,<var>linker-flags</var></code></dt>
213      <dd>This option passes <var>linker-flags</var> as additional
214      flags to the <code>libtool --mode=link</code> command. Use this
215      to add local linker-specific options.</dd>
216
217      <dt><code>-p</code></dt>
218      <dd>This option causes apxs to link against the apr/apr-util libraries.
219      This is useful when compiling helper programs that use the apr/apr-util
220      libraries.</dd>
221      </dl>
222    
223
224    <h3><a name="options.dsoinstall" id="options.dsoinstall">DSO Installation and Configuration Options</a></h3>
225    
226      <dl>
227      <dt><code>-i</code></dt>
228      <dd>This indicates the installation operation and installs one or more
229      dynamically shared objects into the server's <var>modules</var>
230      directory.</dd>
231
232      <dt><code>-a</code></dt>
233      <dd>This activates the module by automatically adding a corresponding
234      <code class="directive"><a href="../mod/mod_so.html#loadmodule">LoadModule</a></code> line to Apache's
235      <code>httpd.conf</code> configuration file, or by enabling it if it
236      already exists.</dd>
237
238      <dt><code>-A</code></dt>
239      <dd>Same as option <code>-a</code> but the created <code class="directive"><a href="../mod/mod_so.html#loadmodule">LoadModule</a></code> directive is prefixed with a hash
240      sign (<code>#</code>), <em>i.e.</em>, the module is just prepared for
241      later activation but initially disabled.</dd>
242
243      <dt><code>-e</code></dt>
244      <dd>This indicates the editing operation, which can be used with the
245      <code>-a</code> and <code>-A</code> options similarly to the
246      <code>-i</code> operation to edit Apache's <code>httpd.conf</code>
247      configuration file without attempting to install the module.</dd>
248      </dl>

252<h2><a name="examples" id="examples">Examples</a></h2>
253    <p>Assume you have an Apache module named <code>mod_foo.c</code> available
254    which should extend Apache's server functionality. To accomplish this you
255    first have to compile the C source into a shared object suitable for loading
256    into the Apache server under runtime via the following command:</p>
257
258    <div class="example"><pre class="prettyprint lang-sh">$ apxs -c mod_foo.c
259/path/to/libtool --mode=compile gcc ... -c mod_foo.c
260/path/to/libtool --mode=link gcc ... -o mod_foo.la mod_foo.slo
261$ _</pre>
262</div>
263
264    <p>Then you have to update the Apache configuration by making sure a
265    <code class="directive"><a href="../mod/mod_so.html#loadmodule">LoadModule</a></code> directive is present to
266    load this shared object. To simplify this step <code class="program"><a href="../programs/apxs.html">apxs</a></code> provides
267    an automatic way to install the shared object in its "modules" directory
268    and updating the <code>httpd.conf</code> file accordingly. This can be
269    achieved by running:</p>
270
271    <div class="example"><pre class="prettyprint lang-sh">$ apxs -i -a mod_foo.la
272/path/to/instdso.sh mod_foo.la /path/to/apache/modules
273/path/to/libtool --mode=install cp mod_foo.la /path/to/apache/modules
274...
275chmod 755 /path/to/apache/modules/mod_foo.so
276[activating module `foo' in /path/to/apache/conf/httpd.conf]
277$ _</pre>
278</div>
279
280    <p>This way a line named</p>
281
282    <div class="example"><pre class="prettyprint lang-config">LoadModule foo_module modules/mod_foo.so</pre>
283</div>
284
285    <p>is added to the configuration file if still not present. If you want to
286    have this disabled per default use the <code>-A</code> option,
287    <em>i.e.</em></p>
288
289    <div class="example"><pre class="prettyprint lang-sh">$ apxs -i -A mod_foo.c</pre>
290</div>
291
292    <p>For a quick test of the apxs mechanism you can create a sample Apache
293    module template plus a corresponding Makefile via:</p>
294
295    <div class="example"><pre class="prettyprint lang-sh">$ apxs -g -n foo
296Creating [DIR]  foo
297Creating [FILE] foo/Makefile
298Creating [FILE] foo/modules.mk
299Creating [FILE] foo/mod_foo.c
300Creating [FILE] foo/.deps
301$ _</pre>
302</div>
303
304    <p>Then you can immediately compile this sample module into a shared object
305    and load it into the Apache server:</p>
306
307    <div class="example"><pre class="prettyprint lang-sh">$ cd foo
308$ make all reload
309apxs -c mod_foo.c
310/path/to/libtool --mode=compile gcc ... -c mod_foo.c
311/path/to/libtool --mode=link gcc ... -o mod_foo.la mod_foo.slo
312apxs -i -a -n "foo" mod_foo.la
313/path/to/instdso.sh mod_foo.la /path/to/apache/modules
314/path/to/libtool --mode=install cp mod_foo.la /path/to/apache/modules
315...
316chmod 755 /path/to/apache/modules/mod_foo.so
317[activating module `foo' in /path/to/apache/conf/httpd.conf]
318apachectl restart
319/path/to/apache/sbin/apachectl restart: httpd not running, trying to start
320[Tue Mar 31 11:27:55 1998] [debug] mod_so.c(303): loaded module foo_module
321/path/to/apache/sbin/apachectl restart: httpd started
322$ _</pre>
323</div>
