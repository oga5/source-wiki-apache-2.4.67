29  <p>This document describes changes to the Apache HTTPD API from
30     version 2.2 to 2.4, that may be of interest to module/application
31     developers and core hacks.  As of the first GA release of the
32     2.4 branch API compatibility is preserved for the life of the
33     2.4 branch.  (The
34     <a href="http://svn.apache.org/repos/asf/httpd/httpd/branches/2.4.x/VERSIONING">VERSIONING</a>
35     description for the 2.4 release provides more information about API
36     compatibility.)</p>
37
38  <p>API changes fall into two categories: APIs that are altogether new,
39     and existing APIs that are expanded or changed.  The latter are
40     further divided into those where all changes are backwards-compatible
41     (so existing modules can ignore them), and those that might
42     require attention by maintainers.  As with the transition from
43     HTTPD 2.0 to 2.2, existing modules and applications will require
44     recompiling and may call for some attention, but most should not
45     require any substantial updating (although some may be able to
46     take advantage of API changes to offer significant improvements).</p>
47  <p>For the purpose of this document, the API is split according
48     to the public header files.  These headers are themselves the
49     reference documentation, and can be used to generate a browsable
50     HTML reference with <code>make docs</code>.</p>

57<h2><a name="api_changes" id="api_changes">Changed APIs</a></h2>
58  
59
60  <h3><a name="ap_expr" id="ap_expr">ap_expr (NEW!)</a></h3>
61    
62    <p>Introduces a new API to parse and evaluate boolean and algebraic
63       expressions, including provision for a standard syntax and
64       customized variants.</p>
65  
66
67  <h3><a name="ap_listen" id="ap_listen">ap_listen (changed; backwards-compatible)</a></h3>
68    
69    <p>Introduces a new API to enable httpd child processes to serve
70       different purposes.</p>
71  
72
73  <h3><a name="ap_mpm" id="ap_mpm">ap_mpm (changed)</a></h3>
74    
75  <p><code>ap_mpm_run</code> is replaced by a new <code>mpm</code> hook.
76  Also <code>ap_graceful_stop_signalled</code> is lost, and
77  <code>ap_mpm_register_timed_callback</code> is new.</p>
78  
79
80  <h3><a name="ap_regex" id="ap_regex">ap_regex (changed)</a></h3>
81    
82  <p>In addition to the existing regexp wrapper, a new higher-level API
83  <code>ap_rxplus</code> is now provided.  This provides the capability to
84  compile Perl-style expressions like <code>s/regexp/replacement/flags</code>
85  and to execute them against arbitrary strings. Support for regexp
86  backreferences is also added.</p>
87  
88
89  <h3><a name="ap_slotmem" id="ap_slotmem">ap_slotmem (NEW!)</a></h3>
90    
91    <p>Introduces an API for modules to allocate and manage memory slots,
92    most commonly for shared memory.</p>
93  
94
95  <h3><a name="ap_socache" id="ap_socache">ap_socache (NEW!)</a></h3>
96    
97    <p>API to manage a shared object cache.</p>
98  
99
100  <h3><a name="heartbeat" id="heartbeat">heartbeat (NEW!)</a></h3>
101    
102    <p>common structures for heartbeat modules</p>
103  
104
105  <h3><a name="ap_parse_htaccess" id="ap_parse_htaccess">ap_parse_htaccess (changed)</a></h3>
106    
107    <p>The function signature for <code>ap_parse_htaccess</code> has been
108    changed. A <code>apr_table_t</code> of individual directives allowed
109    for override must now be passed (override remains).</p>
110  
111
112  <h3><a name="http_config" id="http_config">http_config (changed)</a></h3>
113    
114    <ul>
115      <li>Introduces per-module, per-directory loglevels, including macro wrappers.</li>
116      <li>New <code>AP_DECLARE_MODULE</code> macro to declare all modules.</li>
117      <li>New <code>APLOG_USE_MODULE</code> macro necessary for per-module loglevels in
118          multi-file modules.</li>
119      <li>New API to retain data across module unload/load</li>
120      <li>New <code>check_config</code> hook</li>
121      <li>New <code>ap_process_fnmatch_configs()</code> function to process wildcards</li>
122      <li>Change <code>ap_configfile_t</code>, <code>ap_cfg_getline()</code>,
123          <code>ap_cfg_getc()</code> to return error codes, and add
124          <code>ap_pcfg_strerror()</code> for retrieving an error description.</li>
125      <li>Any config directive permitted in ACCESS_CONF context must now
126          correctly handle being called from an .htaccess file via the new
127          <code class="directive"><a href="../mod/core.html#allowoverridelist">AllowOverrideList</a></code> directive.
128          ap_check_cmd_context() accepts a new flag NOT_IN_HTACCESS to detect
129          this case.</li>
130    </ul>
131  
132
133  <h3><a name="http_core" id="http_core">http_core (changed)</a></h3>
134    
135    <ul>
136      <li>REMOVED <code>ap_default_type</code>, <code>ap_requires</code>, all
137          2.2 authnz API</li>
138      <li>Introduces Optional Functions for logio and authnz</li>
139      <li>New function <code>ap_get_server_name_for_url</code> to support IPv6
140          literals.</li>
141      <li>New function <code>ap_register_errorlog_handler</code> to register error log
142          format string handlers.</li>
143      <li>Arguments of <code>error_log</code> hook have changed. Declaration has moved to
144          <code>http_core.h</code>.</li>
145      <li>New function <code>ap_state_query</code> to determine if the server is in the
146          initial configuration preflight phase or not. This is both easier to
147          use and more correct than the old method of creating a pool userdata
148          entry in the process pool.</li>
149      <li>New function <code>ap_get_conn_socket</code> to get the socket descriptor for a
150          connection. This should be used instead of accessing the core
151          connection config directly.</li>
152    </ul>
153  
154
155  <h3><a name="httpd" id="httpd">httpd (changed)</a></h3>
156    
157    <ul>
158      <li>Introduce per-directory, per-module loglevel</li>
159      <li>New loglevels <code>APLOG_TRACEn</code></li>
160      <li>Introduce errorlog ids for requests and connections</li>
161      <li>Support for mod_request kept_body</li>
162      <li>Support buffering filter data for async requests</li>
163      <li>New <code>CONN_STATE</code> values</li>
164      <li>Function changes: <code>ap_escape_html</code> updated;
165          <code>ap_unescape_all</code>, <code>ap_escape_path_segment_buffer</code></li>
166      <li>Modules that load other modules later than the <code>EXEC_ON_READ</code> config
167          reading stage need to call <code>ap_reserve_module_slots()</code> or
168          <code>ap_reserve_module_slots_directive()</code> in their
169          <code>pre_config hook</code>.</li>
170      <li>The useragent IP address per request can now be tracked
171          independently of the client IP address of the connection, for
172          support of deployments with load balancers.</li>
173    </ul>
174  
175
176  <h3><a name="http_log" id="http_log">http_log (changed)</a></h3>
177    
178    <ul>
179      <li>Introduce per-directory, per-module loglevel</li>
180      <li>New loglevels <code>APLOG_TRACEn</code></li>
181      <li><code>ap_log_*error</code> become macro wrappers (backwards-compatible if
182          <code>APLOG_MARK</code> macro is used, except that is no longer possible to
183          use <code>#ifdef</code> inside the argument list)</li>
184      <li>piped logging revamped</li>
185      <li><code>module_index</code> added to error_log hook</li>
186      <li>new function: <code>ap_log_command_line</code></li>
187    </ul>
188  
189
190  <h3><a name="http_request" id="http_request">http_request (changed)</a></h3>
191    
192    <ul>
193      <li>New auth_internal API and auth_provider API</li>
194      <li>New <code>EOR</code> bucket type</li>
195      <li>New function <code>ap_process_async_request</code></li>
196      <li>New flags <code>AP_AUTH_INTERNAL_PER_CONF</code> and
197          <code>AP_AUTH_INTERNAL_PER_URI</code></li>
198      <li>New <code>access_checker_ex</code> hook to apply additional access control
199          and/or bypass authentication.</li>
200      <li>New functions <code>ap_hook_check_access_ex</code>,
201          <code>ap_hook_check_access</code>, <code>ap_hook_check_authn</code>,
202          <code>ap_hook_check_authz</code> which accept
203          <code>AP_AUTH_INTERNAL_PER_*</code> flags</li>
204      <li>DEPRECATED direct use of <code>ap_hook_access_checker</code>,
205          <code>access_checker_ex</code>, <code>ap_hook_check_user_id</code>,
206          <code>ap_hook_auth_checker</code></li>
207    </ul>
208    <p>When possible, registering all access control hooks (including
209       authentication and authorization hooks) using <code>AP_AUTH_INTERNAL_PER_CONF</code>
210       is recommended.  If all modules' access control hooks are registered
211       with this flag, then whenever the server handles an internal
212       sub-request that matches the same set of access control configuration
213       directives as the initial request (which is the common case), it can
214       avoid invoking the access control hooks another time.</p>
215    <p>If your module requires the old behavior and must perform access
216       control checks on every sub-request with a different URI from the
217       initial request, even if that URI matches the same set of access
218       control configuration directives, then use
219       <code>AP_AUTH_INTERNAL_PER_URI</code>.</p>

227  <h3><a name="mod_cache" id="mod_cache">mod_cache (changed)</a></h3>
228    
229    <p>Introduces a <code>commit_entity()</code> function to the cache provider
230    interface, allowing atomic writes to cache. Add a <code>cache_status()</code>
231    hook to report the cache decision. All private structures and functions were
232    removed.</p>

235  <h3><a name="mod_core" id="mod_core">mod_core (NEW!)</a></h3>
236    
237    <p>This introduces low-level APIs to send arbitrary headers,
238    and exposes functions to handle HTTP OPTIONS and TRACE.</p>
239  
240
241  <h3><a name="mod_cache_disk" id="mod_cache_disk">mod_cache_disk (changed)</a></h3>
242    
243    <p>Changes the disk format of the disk cache to support atomic cache
244    updates without locking. The device/inode pair of the body file is
245    embedded in the header file, allowing confirmation that the header
246    and body belong to one another.</p>
247  
248
249  <h3><a name="mod_disk_cache" id="mod_disk_cache">mod_disk_cache (renamed)</a></h3>
250    
251    <p>The mod_disk_cache module has been renamed to mod_cache_disk in
252    order to be consistent with the naming of other modules within the
253    server.</p>

257    
258    <p>The API for <code class="module"><a href="../mod/mod_request.html">mod_request</a></code>, to make input data
259    available to multiple application/handler modules where required,
260    and to parse HTML form data.</p>
261

314<h2><a name="upgrading" id="upgrading">Specific information on upgrading modules from 2.2</a></h2>
315  
316
317  <h3><a name="upgrading_logging" id="upgrading_logging">Logging</a></h3>
318    
319    <p>In order to take advantage of per-module loglevel configuration, any
320       source file that calls the <code>ap_log_*</code> functions should declare
321       which module it belongs to. If the module's module_struct is called
322       <code>foo_module</code>, the following code can be used to remain
323       backward compatible with HTTPD 2.0 and 2.2:</p>
324    <div class="example"><p><code>
325        #include &lt;http_log.h&gt;<br />
326        <br />
327        #ifdef APLOG_USE_MODULE<br />
328        APLOG_USE_MODULE(foo);<br />
329        #endif
330    </code></p></div>
331    <p>Note: This is absolutely required for C++-language modules.  It
332    can be skipped for C-language modules, though that breaks
333    module-specific log level support for files without it.</p>
334    <p>The number of parameters of the <code>ap_log_*</code> functions and the
335       definition of <code>APLOG_MARK</code> has changed. Normally, the change
336       is completely transparent. However, changes are required if a
337       module uses <code>APLOG_MARK</code> as a parameter to its own functions
338       or if a module calls <code>ap_log_*</code> without passing
339       <code>APLOG_MARK</code>.  A module which uses wrappers
340       around <code>ap_log_*</code> typically uses both of these constructs.</p>
341
342    <p>The easiest way to change code which passes <code>APLOG_MARK</code> to
343       its own functions is to define and use a different macro that expands to
344       the parameters required by those functions, as <code>APLOG_MARK</code>
345       should only be used when calling <code>ap_log_*</code>
346       directly.  In this way, the code will remain compatible with HTTPD 2.0
347       and 2.2.</p>
348
349    <p>Code which calls <code>ap_log_*</code> without passing
350       <code>APLOG_MARK</code> will necessarily differ between 2.4 and earlier
351       releases, as 2.4 requires a new third argument,
352       <code>APLOG_MODULE_INDEX</code>.</p>
353
354    <div class="example"><p><code>
355       /* code for httpd 2.0/2.2 */<br />
356       ap_log_perror(file, line, APLOG_ERR, 0, p, "Failed to allocate dynamic lock structure");<br />
357       <br />
358       /* code for httpd 2.4 */<br />
359       ap_log_perror(file, line, APLOG_MODULE_INDEX, APLOG_ERR, 0, p, "Failed to allocate dynamic lock structure");<br />
360       <br />
361    </code></p></div>
362
363    <p><code>ap_log_*error</code> are now implemented as macros. This means
364       that it is no longer possible to use <code>#ifdef</code> inside the
365       argument list of <code>ap_log_*error</code>, as this would cause
366       undefined behavior according to C99.</p>
367
368    <p>A <code>server_rec</code> pointer must be passed to
369       <code>ap_log_error()</code> when called after startup.  This
370       was always appropriate, but there are even more limitations with
371       a <code>NULL</code> <code>server_rec</code> in 2.4 than in
372       previous releases.  Beginning with 2.3.12, the global variable
373       <code>ap_server_conf</code> can always be used as
374       the <code>server_rec</code> parameter, as it will be
375       <code>NULL</code> only when it is valid to pass <code>NULL</code>
376       to <code>ap_log_error()</code>.  <code>ap_server_conf</code>
377       should be used only when a more appropriate <code>server_rec</code>
378       is not available.</p>
379
380    <p>Consider the following changes to take advantage of the new
381       <code>APLOG_TRACE1..8</code> log levels:</p>
382       <ul>
383         <li>Check current use of <code>APLOG_DEBUG</code> and
384         consider if one of the <code>APLOG_TRACEn</code> levels is
385         more appropriate.</li>
386         <li>If your module currently has a mechanism for configuring
387         the amount of debug logging which is performed, consider
388         eliminating that mechanism and relying on the use of
389         different <code>APLOG_TRACEn</code> levels.  If expensive
390         trace processing needs to be bypassed depending on the
391         configured log level, use the <code>APLOGtrace<em>n</em></code>
392         and <code>APLOGrtrace<em>n</em></code> macros to first check
393         if tracing is enabled.</li>
394       </ul>
395
396    <p>Modules sometimes add process id and/or thread id to their log
397       messages.  These ids are now logged by default, so it may not
398       be necessary for the module to log them explicitly.  (Users may
399       remove them from the error log format, but they can be
400       instructed to add it back if necessary for problem diagnosis.)</p>

403  <h3><a name="upgrading_byfunction" id="upgrading_byfunction">If your module uses these existing APIs...</a></h3>
404    
405
406    <dl>
407      <dt><code>ap_default_type()</code></dt>
408      <dd>This is no longer available; Content-Type must be configured
409          explicitly or added by the application.</dd>
410
411      <dt><code>ap_get_server_name()</code></dt>
412      <dd>If the returned server name is used in a URL,
413      use <code>ap_get_server_name_for_url()</code> instead.  This new
414      function handles the odd case where the server name is an IPv6
415      literal address.</dd>
416
417      <dt><code>ap_get_server_version()</code></dt>
418      <dd>For logging purposes, where detailed information is
419          appropriate, use <code>ap_get_server_description()</code>.
420          When generating output, where the amount of information
421          should be configurable by ServerTokens, use
422          <code>ap_get_server_banner()</code>.</dd>
423
424      <dt><code>ap_graceful_stop_signalled()</code></dt>
425      <dd>Replace with a call
426      to <code>ap_mpm_query(AP_MPMQ_MPM_STATE)</code> and checking for
427      state <code>AP_MPMQ_STOPPING</code>.</dd>
428
429      <dt><code>ap_max_daemons_limit</code>, <code>ap_my_generation</code>,
430          and <code>ap_threads_per_child</code></dt>
431      <dd>Use <code>ap_mpm_query()</code> query codes
432          <code>AP_MPMQ_MAX_DAEMON_USED</code>, <code>AP_MPMQ_GENERATION</code>,
433          and <code>AP_MPMQ_MAX_THREADS</code>, respectively.</dd>
434
435      <dt><code>ap_mpm_query()</code></dt>
436      <dd>Ensure that it is not used until after the register-hooks
437          hook has completed.  Otherwise, an MPM built as a DSO
438          would not have had a chance to enable support for this
439          function.</dd>
440
441      <dt><code>ap_requires()</code></dt>
442      <dd>The core server now provides better infrastructure for handling
443          <code class="directive"><a href="../mod/mod_authz_core.html#require">Require</a></code> configuration.
444          Register an auth provider function for each supported entity using
445          <code>ap_register_auth_provider()</code>.  The function will be
446          called as necessary during <code class="directive">Require</code>
447          processing.  (Consult bundled modules for detailed examples.)</dd>
448
449      <dt><code>ap_server_conf-&gt;process-&gt;pool</code>
450      userdata</dt>
451      <dd>
452        Optional:
453        <ul>
454          <li>If your module uses this to determine which pass of the
455	  startup hooks is being run,
456	  use <code>ap_state_query(AP_SQ_MAIN_STATE)</code>.</li>
457          <li>If your module uses this to maintain data across the
458          unloading and reloading of your module, use
459          <code>ap_retained_data_create()</code> and
460          <code>ap_retained_data_get()</code>.</li>
461        </ul>
462      </dd>
463
464      <dt><code>apr_global_mutex_create()</code>,
465          <code>apr_proc_mutex_create()</code></dt>
466      <dd>Optional: See <code>ap_mutex_register()</code>,
467          <code>ap_global_mutex_create()</code>, and
468          <code>ap_proc_mutex_create()</code>; these allow your
469          mutexes to be configurable with
470          the <code class="directive"><a href="../mod/core.html#mutex">Mutex</a></code> directive;
471          you can also remove any configuration mechanisms in your
472          module for such mutexes
473      </dd>
474
475      <dt><code>CORE_PRIVATE</code></dt>
476      <dd>This is now unnecessary and ignored.</dd>
477
478      <dt><code>dav_new_error()</code>
479      and <code>dav_new_error_tag()</code></dt>
480      <dd>Previously, these assumed that <code>errno</code> contained
481      information describing the failure.  Now,
482      an <code>apr_status_t</code> parameter must be provided.  Pass
483      0/APR_SUCCESS if there is no such error information, or a valid
484      <code>apr_status_t</code> value otherwise.</dd>
485
486      <dt><code>mpm_default.h</code>, <code>DEFAULT_LOCKFILE</code>,
487      <code>DEFAULT_THREAD_LIMIT</code>, <code>DEFAULT_PIDLOG</code>,
488      etc.</dt>
489      <dd>The header file and most of the default configuration
490      values set in it are no longer visible to modules.  (Most can
491      still be overridden at build time.)  <code>DEFAULT_PIDLOG</code>
492      and <code>DEFAULT_REL_RUNTIMEDIR</code> are now universally
493      available via <code>ap_config.h</code>.</dd>
494
495      <dt><code>unixd_config</code></dt>
496      <dd>This has been renamed to ap_unixd_config.</dd>
497
498      <dt><code>unixd_setup_child()</code></dt>
499      <dd>This has been renamed to ap_unixd_setup_child(), but most callers
500          should call the added ap_run_drop_privileges() hook.</dd>
501
502      <dt><code>conn_rec-&gt;remote_ip</code> and
503          <code>conn_rec-&gt;remote_addr</code></dt>
504      <dd>These fields have been renamed in order to distinguish between
505      the client IP address of the connection and the useragent IP address
506      of the request (potentially overridden by a load balancer or proxy).
507      References to either of these fields must be updated with one of the
508      following options, as appropriate for the module:
509      <ul>
510        <li>When you require the IP address of the user agent, which
511        might be connected directly to the server, or might optionally be
512        separated from the server by a transparent load balancer or
513        proxy, use <code>request_rec-&gt;useragent_ip</code> and
514        <code>request_rec-&gt;useragent_addr</code>.</li>
515        <li>When you require the IP address of the client that is
516        connected directly to the server, which might be the useragent or
517        might be the load balancer or proxy itself, use
518        <code>conn_rec-&gt;client_ip</code> and
519        <code>conn_rec-&gt;client_addr</code>.</li>
520      </ul>
521      </dd>
522    </dl>

525  <h3><a name="upgrading_byfeature" id="upgrading_byfeature">If your module interfaces with this feature...</a></h3>
526    
527    <dl>
528      <dt>suEXEC</dt>
529      <dd>Optional: If your module logs an error
530          when <code>ap_unixd_config.suexec_enabled</code> is 0,
531          also log the value of the new
532          field <code>suexec_disabled_reason</code>, which contains an
533          explanation of why it is not available.</dd>
534
535      <dt>Extended status data in the scoreboard</dt>
536      <dd>In previous releases, <code>ExtendedStatus</code> had to be
537          set to <code>On</code>, which in turn required that
538          mod_status was loaded.  In 2.4, just
539          set <code>ap_extended_status</code> to <code>1</code> in a
540          pre-config hook and the extended status data will be
541          available.</dd>
542    </dl>

546    
547    <dl>
548    <dt>Parse query args</dt>
549    <dd>Consider if <code>ap_args_to_table()</code> would be
550    helpful.</dd>

553    <dd>Use <code>ap_parse_form_data()</code>.</dd>

557    specified</dt>

560    <dt>Implement cleanups which clear pointer variables</dt>
561    <dd>Use <code>ap_pool_cleanup_set_null()</code>.</dd>
562
563    <dt>Create run-time files such as shared memory files, pid files,
564    etc.</dt>
565    <dd>Use <code>ap_runtime_dir_relative()</code> so that the global
566    configuration for the location of such files, either by the
567    <code>DEFAULT_REL_RUNTIMEDIR</code> compile setting or the
568    <code class="directive"><a href="../mod/core.html#defaultruntimedir">DefaultRuntimeDir</a></code> directive,
569    will be respected.  <em>Apache httpd 2.4.2 and above.</em></dd>
