43<h2><a name="types" id="types">Filter Types</a></h2>
44    <p>There are three basic filter types (each of these is actually broken
45    down into two categories, but that comes later).</p>
46
47    <dl>
48    <dt><code>CONNECTION</code></dt>
49    <dd>Filters of this type are valid for the lifetime of this connection.
50    (<code>AP_FTYPE_CONNECTION</code>, <code>AP_FTYPE_NETWORK</code>)</dd>
51
52    <dt><code>PROTOCOL</code></dt>
53    <dd>Filters of this type are valid for the lifetime of this request from
54    the point of view of the client, this means that the request is valid
55    from the time that the request is sent until the time that the response
56    is received. (<code>AP_FTYPE_PROTOCOL</code>,
57    <code>AP_FTYPE_TRANSCODE</code>)</dd>
58
59    <dt><code>RESOURCE</code></dt>
60    <dd>Filters of this type are valid for the time that this content is used
61    to satisfy a request.  For simple requests, this is identical to
62    <code>PROTOCOL</code>, but internal redirects and sub-requests can change
63    the content without ending the request. (<code>AP_FTYPE_RESOURCE</code>,
64    <code>AP_FTYPE_CONTENT_SET</code>)</dd>
65    </dl>
66
67    <p>It is important to make the distinction between a protocol and a
68    resource filter.  A resource filter is tied to a specific resource, it
69    may also be tied to header information, but the main binding is to a
70    resource.  If you are writing a filter and you want to know if it is
71    resource or protocol, the correct question to ask is:  "Can this filter
72    be removed if the request is redirected to a different resource?"  If
73    the answer is yes, then it is a resource filter.  If it is no, then it
74    is most likely a protocol or connection filter.  I won't go into
75    connection filters, because they seem to be well understood. With this
76    definition, a few examples might help:</p>
77
78    <dl>
79    <dt>Byterange</dt>
80    <dd>We have coded it to be inserted for all requests, and it is removed
81    if not used.  Because this filter is active at the beginning of all
82    requests, it can not be removed if it is redirected, so this is a
83    protocol filter.</dd>
84
85    <dt>http_header</dt>
86    <dd>This filter actually writes the headers to the network.  This is
87    obviously a required filter (except in the asis case which is special
88    and will be dealt with below) and so it is a protocol filter.</dd>
89
90    <dt>Deflate</dt>
91    <dd>The administrator configures this filter based on which file has been
92    requested.  If we do an internal redirect from an autoindex page to an
93    index.html page, the deflate filter may be added or removed based on
94    config, so this is a resource filter.</dd>
95    </dl>
96
97    <p>The further breakdown of each category into two more filter types is
98    strictly for ordering.  We could remove it, and only allow for one
99    filter type, but the order would tend to be wrong, and we would need to
100    hack things to make it work.  Currently, the <code>RESOURCE</code> filters
101    only have one filter type, but that should change.</p>

104<h2><a name="howinserted" id="howinserted">How are filters inserted?</a></h2>
105    <p>This is actually rather simple in theory, but the code is
106    complex.  First of all, it is important that everybody realize that
107    there are three filter lists for each request, but they are all
108    concatenated together:</p>
109    <ul>
110        <li><code>r-&gt;output_filters</code> (corresponds to RESOURCE)</li>
111        <li><code>r-&gt;proto_output_filters</code> (corresponds to PROTOCOL)</li>
112        <li><code>r-&gt;connection-&gt;output_filters</code> (corresponds to CONNECTION)</li>
113    </ul> 
114    
115    <p>The problem previously, was that we used a singly linked list to create the filter stack, and we
116    started from the "correct" location.  This means that if I had a
117    <code>RESOURCE</code> filter on the stack, and I added a
118    <code>CONNECTION</code> filter, the <code>CONNECTION</code> filter would
119    be ignored. This should make sense, because we would insert the connection
120    filter at the top of the <code>c-&gt;output_filters</code> list, but the end
121    of <code>r-&gt;output_filters</code> pointed to the filter that used to be
122    at the front of <code>c-&gt;output_filters</code>. This is obviously wrong.
123    The new insertion code uses a doubly linked list. This has the advantage
124    that we never lose a filter that has been inserted. Unfortunately, it comes
125    with a separate set of headaches.</p>
126
127    <p>The problem is that we have two different cases were we use subrequests.
128    The first is to insert more data into a response. The second is to
129    replace the existing response with an internal redirect. These are two
130    different cases and need to be treated as such.</p>
131
132    <p>In the first case, we are creating the subrequest from within a handler
133    or filter.  This means that the next filter should be passed to
134    <code>make_sub_request</code> function, and the last resource filter in the
135    sub-request will point to the next filter in the main request.  This
136    makes sense, because the sub-request's data needs to flow through the
137    same set of filters as the main request.  A graphical representation
138    might help:</p>
139
140<div class="example"><pre>Default_handler --&gt; includes_filter --&gt; byterange --&gt; ...</pre></div>
141
142    <p>If the includes filter creates a sub request, then we don't want the
143    data from that sub-request to go through the includes filter, because it
144    might not be SSI data.  So, the subrequest adds the following:</p>
145
146<div class="example"><pre>Default_handler --&gt; includes_filter -/-&gt; byterange --&gt; ...
147                                    /
148Default_handler --&gt; sub_request_core</pre></div>
149
150    <p>What happens if the subrequest is SSI data?  Well, that's easy, the
151    <code>includes_filter</code> is a resource filter, so it will be added to
152    the sub request in between the <code>Default_handler</code> and the
153    <code>sub_request_core</code> filter.</p>
154
155    <p>The second case for sub-requests is when one sub-request is going to
156    become the real request.  This happens whenever a sub-request is created
157    outside of a handler or filter, and NULL is passed as the next filter to
158    the <code>make_sub_request</code> function.</p>
159
160    <p>In this case, the resource filters no longer make sense for the new
161    request, because the resource has changed.  So, instead of starting from
162    scratch, we simply point the front of the resource filters for the
163    sub-request to the front of the protocol filters for the old request.
164    This means that we won't lose any of the protocol filters, neither will
165    we try to send this data through a filter that shouldn't see it.</p>
166
167    <p>The problem is that we are using a doubly-linked list for our filter
168    stacks now. But, you should notice that it is possible for two lists to
169    intersect in this model.  So, you do you handle the previous pointer?
170    This is a very difficult question to answer, because there is no "right"
171    answer, either method is equally valid.  I looked at why we use the
172    previous pointer.  The only reason for it is to allow for easier
173    addition of new servers.  With that being said, the solution I chose was
174    to make the previous pointer always stay on the original request.</p>
175
176    <p>This causes some more complex logic, but it works for all cases.  My
177    concern in having it move to the sub-request, is that for the more
178    common case (where a sub-request is used to add data to a response), the
179    main filter chain would be wrong.  That didn't seem like a good idea to
180    me.</p>

183<h2><a name="asis" id="asis">Asis</a></h2>
184    <p>The final topic.  :-)  Mod_Asis is a bit of a hack, but the
185    handler needs to remove all filters except for connection filters, and
186    send the data.  If you are using <code class="module"><a href="../mod/mod_asis.html">mod_asis</a></code>, all other
187    bets are off.</p>

190<h2><a name="conclusion" id="conclusion">Explanations</a></h2>
191    <p>The absolutely last point is that the reason this code was so hard to
192    get right, was because we had hacked so much to force it to work.  I
193    wrote most of the hacks originally, so I am very much to blame.
194    However, now that the code is right, I have started to remove some
195    hacks.  Most people should have seen that the <code>reset_filters</code>
196    and <code>add_required_filters</code> functions are gone.  Those inserted
197    protocol level filters for error conditions, in fact, both functions did
198    the same thing, one after the other, it was really strange. Because we
199    don't lose protocol filters for error cases any more, those hacks went away.
200    The <code>HTTP_HEADER</code>, <code>Content-length</code>, and
201    <code>Byterange</code> filters are all added in the
202    <code>insert_filters</code> phase, because if they were added earlier, we
203    had some interesting interactions.  Now, those could all be moved to be
204    inserted with the <code>HTTP_IN</code>, <code>CORE</code>, and
205    <code>CORE_IN</code> filters.  That would make the code easier to
206    follow.</p>
