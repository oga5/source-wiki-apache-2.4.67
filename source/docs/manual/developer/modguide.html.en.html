43<h2><a name="introduction" id="introduction">Introduction</a></h2>
44<h3><a name="what" id="what">What we will be discussing in this document</a></h3>
45<p>
46This document will discuss how you can create modules for the Apache 
47HTTP Server 2.4, by exploring an example module called 
48<code>mod_example</code>. In the first part of this document, the purpose 
49of this module will be to calculate and print out various digest values for 
50existing files on your web server, whenever we access the URL <code>
51http://hostname/filename.sum</code>. For instance, if we want to know the 
52MD5 digest value of the file located at <code>
53http://www.example.com/index.html</code>, we would visit <code>
54http://www.example.com/index.html.sum</code>. 
55</p>
56
57<p>
58In the second part of this document, which deals with configuration 
59directive and context awareness, we will be looking at a module that simply 
60writes out its own configuration to the client.
61</p>

92<h2><a name="basics" id="basics">Defining a module</a></h2>
93<p>
94<img src="../images/build_a_mod_3.png" alt="Module name tags" /><br />
95Every module starts with the same declaration, or name tag if you will, 
96that defines a module as <em>a separate entity within Apache</em>:</p>
97
98
99
100<pre class="prettyprint lang-c">module AP_MODULE_DECLARE_DATA   example_module =
101{ 
102    STANDARD20_MODULE_STUFF,
103    create_dir_conf, /* Per-directory configuration handler */
104    merge_dir_conf,  /* Merge handler for per-directory configurations */
105    create_svr_conf, /* Per-server configuration handler */
106    merge_svr_conf,  /* Merge handler for per-server configurations */
107    directives,      /* Any directives we may have for httpd */
108    register_hooks   /* Our hook registering function */
109};</pre>
110
111
112
113<p>
114This bit of code lets the server know that we have now registered a new module 
115in the system, and that its name is <code>example_module</code>. The name 
116of the module is used primarily for two things:<br />
117</p>
118<ul>
119<li>Letting the server know how to load the module using the LoadModule</li>
120<li>Setting up a namespace for the module to use in configurations</li>
121</ul>
122<p>
123For now, we're only concerned with the first purpose of the module name, 
124which comes into play when we need to load the module:
125</p>
126<pre class="prettyprint lang-config">LoadModule example_module modules/mod_example.so</pre>
127
128<p>
129In essence, this tells the server to open up <code>mod_example.so</code> and look for a module 
130called <code>example_module</code>.
131</p>
132<p>
133Within this name tag of ours is also a bunch of references to how we would 
134like to handle things: Which directives do we respond to in a configuration 
135file or .htaccess, how do we operate within specific contexts, and what 
136handlers are we interested in registering with the Apache HTTP service. We'll 
137return to all these elements later in this document.
138</p>

141<h2><a name="hooking" id="hooking">Getting started: Hooking into the server</a></h2>
142<h3><a name="hook_intro" id="hook_intro">An introduction to hooks</a></h3>
143<p>
144When handling requests in Apache HTTP Server 2.4, the first thing you will need to do is 
145create a hook into the request handling process. A hook is essentially a 
146message telling the server that you are willing to either serve or at least 
147take a glance at certain requests given by clients. All handlers, whether 
148it's mod_rewrite, mod_authn_*, mod_proxy and so on, are hooked into 
149specific parts of the request process. As you are probably aware, modules 
150serve different purposes; Some are authentication/authorization handlers, 
151others are file or script handlers while some third modules rewrite URIs or 
152proxies content. Furthermore, in the end, it is up to the user of the server 
153how and when each module will come into place. Thus, the server itself does not 
154presume to know which module is responsible for handling a specific 
155request, and will ask each module whether they have an interest in a given 
156request or not. It is then up to each module to either gently decline 
157serving a request, accept serving it or flat out deny the request from 
158being served, as authentication/authorization modules do: <br />
159<img src="../images/build_a_mod_2.png" alt="Hook handling in httpd" /><br />
160To make it a bit easier for handlers such as our mod_example to know 
161whether the client is requesting content we should handle or not, the server 
162has directives for hinting to modules whether their assistance is needed or 
163not. Two of these are <code class="directive"><a href="../mod/mod_mime.html#addhandler">AddHandler</a></code> 
164and <code class="directive"><a href="../mod/core.html#sethandler">SetHandler</a></code>. Let's take a look at 
165an example using <code class="directive"><a href="../mod/mod_mime.html#addhandler">AddHandler</a></code>. In 
166our example case, we want every request ending with .sum to be served by 
167<code>mod_example</code>, so we'll add a configuration directive that tells 
168the server to do just that:
169</p>
170<pre class="prettyprint lang-config">AddHandler example-handler .sum</pre>
171
172<p>
173What this tells the server is the following: <em>Whenever we receive a request 
174for a URI ending in .sum, we are to let all modules know that we are 
175looking for whoever goes by the name of "example-handler" </em>. 
176Thus, when a request is being served that ends in .sum, the server will let all 
177modules know, that this request should be served by "example-handler
178". As you will see later, when we start building mod_example, we will 
179check for this handler tag relayed by <code>AddHandler</code> and reply to 
180the server based on the value of this tag.
181</p>
182
183<h3><a name="hook_declaration" id="hook_declaration">Hooking into httpd</a></h3>
184<p>
185To begin with, we only want to create a simple handler that replies to the 
186client browser when a specific URL is requested, so we won't bother setting 
187up configuration handlers and directives just yet. Our initial module 
188definition will look like this:</p>
189
190
191
192<pre class="prettyprint lang-c">module AP_MODULE_DECLARE_DATA   example_module =
193{
194    STANDARD20_MODULE_STUFF,
195    NULL,
196    NULL,
197    NULL,
198    NULL,
199    NULL,
200    register_hooks   /* Our hook registering function */
201};</pre>
202
203
204
205
206<p>This lets the server know that we are not interested in anything fancy, we 
207just want to hook onto the requests and possibly handle some of them. </p> 
208
209<p> The reference in our example declaration, <code>register_hooks</code> 
210is the name of a function we will create to manage how we hook onto the 
211request process. In this example module, the function has just one purpose; 
212To create a simple hook that gets called after all the rewrites, access 
213control etc has been handled. Thus, we will let the server know that we want 
214to hook into its process as one of the last modules: 
215</p>
216
217
218<pre class="prettyprint lang-c">static void register_hooks(apr_pool_t *pool)
219{
220    /* Create a hook in the request handler, so we get called when a request arrives */
221    ap_hook_handler(example_handler, NULL, NULL, APR_HOOK_LAST);
222}</pre>
223
224
225
226<p>
227The <code>example_handler</code> reference is the function that will handle 
228the request. We will discuss how to create a handler in the next chapter.
229</p>
230
231<h3><a name="hook_others" id="hook_others">Other useful hooks</a></h3>
232<p>
233Hooking into the request handling phase is but one of many hooks that you 
234can create. Some other ways of hooking are:
235</p>
236<ul>
237<li><code>ap_hook_child_init</code>: Place a hook that executes when a child process is spawned (commonly used for initializing modules after the server has forked)</li>
238<li><code>ap_hook_pre_config</code>: Place a hook that executes before any configuration data has been read (very early hook)</li>
239<li><code>ap_hook_post_config</code>: Place a hook that executes after configuration has been parsed, but before the server has forked</li>
240<li><code>ap_hook_pre_translate_name</code>: Place a hook that executes when a URI needs to be translated into a filename on the server, before decoding</li>
241<li><code>ap_hook_translate_name</code>: Place a hook that executes when a URI needs to be translated into a filename on the server (think <code>mod_rewrite</code>)</li>
242<li><code>ap_hook_quick_handler</code>: Similar to <code>ap_hook_handler</code>, except it is run before any other request hooks (translation, auth, fixups etc)</li>
243<li><code>ap_hook_log_transaction</code>: Place a hook that executes when the server is about to add a log entry of the current request</li>
244</ul>

249<h2><a name="handling" id="handling">Building a handler</a></h2>
250<p>
251A handler is essentially a function that receives a callback when a request 
252to the server is made. It is passed a record of the current request (how it was 
253made, which headers and requests were passed along, who's giving the 
254request and so on), and is put in charge of either telling the server that it's 
255not interested in the request or handle the request with the tools provided.
256</p>
257<h3><a name="simple_handler" id="simple_handler">A simple "Hello, world!" 
258handler</a></h3> 
259<p>Let's start off by making a very simple request handler 
260that does the following:
261</p>
262<ol>
263<li>Check that this is a request that should be served by "example-handler"</li>
264<li>Set the content type of our output to <code>text/html</code></li>
265<li>Write "Hello, world!" back to the client browser</li>
266<li>Let the server know that we took care of this request and everything went fine</li>
267</ol>
268<p>
269In C code, our example handler will now look like this:
270</p>
271
272
273<pre class="prettyprint lang-c">static int example_handler(request_rec *r)
274{
275    /* First off, we need to check if this is a call for the "example-handler" handler.
276     * If it is, we accept it and do our things, if not, we simply return DECLINED,
277     * and the server will try somewhere else.
278     */
279    if (!r-&gt;handler || strcmp(r-&gt;handler, "example-handler")) return (DECLINED);
280    
281    /* Now that we are handling this request, we'll write out "Hello, world!" to the client.
282     * To do so, we must first set the appropriate content type, followed by our output.
283     */
284    ap_set_content_type(r, "text/html");
285    ap_rprintf(r, "Hello, world!");
286    
287    /* Lastly, we must tell the server that we took care of this request and everything went fine.
288     * We do so by simply returning the value OK to the server.
289     */
290    return OK;
291}</pre>
292
293
294
295<p>
296Now, we put all we have learned together and end up with a program that 
297looks like 
298<a href="mod_example_1.c">mod_example_1.c</a>
299. The functions used in this example will be explained later in the section 
300<a href="#functions">"Some useful functions you should know"</a>. 
301</p>

303<h3><a name="request_rec" id="request_rec">The request_rec structure</a></h3> 
304<p>The most essential part of any request is the <em>request record
305</em>. In a call to a handler function, this is represented by the <code>
306request_rec* </code> structure passed along with every call that is made. 
307This struct, typically just referred to as <code>r</code> in modules, 
308contains all the information you need for your module to fully process any 
309HTTP request and respond accordingly.</p> <p>Some key elements of the <code>
310request_rec </code> structure are:
311</p>
312<ul>
313<li><code>r-&gt;handler (char*):</code> Contains the name of the handler the server is currently asking to do the handling of this request</li>
314<li><code>r-&gt;method (char*):</code> Contains the HTTP method being used, f.x. GET or POST</li>
315<li><code>r-&gt;filename (char*):</code> Contains the translated filename the client is requesting</li>
316<li><code>r-&gt;args (char*):</code> Contains the query string of the request, if any</li>
317<li><code>r-&gt;headers_in (apr_table_t*):</code> Contains all the headers sent by the client</li>
318<li><code>r-&gt;connection (conn_rec*):</code> A record containing information about the current connection</li>
319<li><code>r-&gt;user (char*):</code> If the URI requires authentication, this is set to the username provided</li>
320<li><code>r-&gt;useragent_ip (char*):</code> The IP address of the client connecting to us</li>
321<li><code>r-&gt;pool (apr_pool_t*)</code>: The memory pool of this request. We'll discuss this in the 
322"<a href="#memory">Memory management</a>" chapter.</li>
323</ul>
324<p>
325A complete list of all the values contained within the <code>request_rec</code> structure can be found in 
326the <a href="http://svn.apache.org/repos/asf/httpd/httpd/trunk/include/httpd.h"><code>httpd.h</code></a> header 
327file or at <a href="http://ci.apache.org/projects/httpd/trunk/doxygen/structrequest__rec.html">http://ci.apache.org/projects/httpd/trunk/doxygen/structrequest__rec.html</a>.
328</p>
329
330
331<p>
332Let's try out some of these variables in another example handler:<br />
333</p>
334
335
336<pre class="prettyprint lang-c">static int example_handler(request_rec *r)
337{
338    /* Set the appropriate content type */
339    ap_set_content_type(r, "text/html");
340
341    /* Print out the IP address of the client connecting to us: */
342    ap_rprintf(r, "&lt;h2&gt;Hello, %s!&lt;/h2&gt;", r-&gt;useragent_ip);
343    
344    /* If we were reached through a GET or a POST request, be happy, else sad. */
345    if ( !strcmp(r-&gt;method, "POST") || !strcmp(r-&gt;method, "GET") ) {
346        ap_rputs("You used a GET or a POST method, that makes us happy!&lt;br/&gt;", r);
347    }
348    else {
349        ap_rputs("You did not use POST or GET, that makes us sad :(&lt;br/&gt;", r);
350    }
351
352    /* Lastly, if there was a query string, let's print that too! */
353    if (r-&gt;args) {
354        ap_rprintf(r, "Your query string was: %s", r-&gt;args);
355    }
356    return OK;
357}</pre>

363<h3><a name="return_value" id="return_value">Return values</a></h3>
364<p>
365Apache relies on return values from handlers to signify whether a request 
366was handled or not, and if so, whether the request went well or not. If a 
367module is not interested in handling a specific request, it should always 
368return the value <code>DECLINED</code>. If it is handling a request, it 
369should either return the generic value <code>OK</code>, or a specific HTTP 
370status code, for example:
371</p>
372
373
374<pre class="prettyprint lang-c">static int example_handler(request_rec *r)
375{
376    /* Return 404: Not found */
377    return HTTP_NOT_FOUND;
378}</pre>
379
380
381
382<p>
383Returning <code>OK</code> or a HTTP status code does not necessarily mean 
384that the request will end. The server may still have other handlers that are 
385interested in this request, for instance the logging modules which, upon a 
386successful request, will write down a summary of what was requested and how 
387it went. To do a full stop and prevent any further processing after your 
388module is done, you can return the value <code>DONE</code> to let the server 
389know that it should cease all activity on this request and carry on with 
390the next, without informing other handlers.
391<br />
392<strong>General response codes:</strong>
393</p>
394<ul>
395<li><code>DECLINED</code>: We are not handling this request</li>
396<li><code>OK</code>: We handled this request and it went well</li>
397<li><code>DONE</code>: We handled this request and the server should just close this thread without further processing</li>
398</ul>
399<p>
400<strong>HTTP specific return codes (excerpt):</strong>
401</p>
402<ul>
403<li><code>HTTP_OK (200)</code>: Request was okay</li>
404<li><code>HTTP_MOVED_PERMANENTLY (301)</code>: The resource has moved to a new URL</li>
405<li><code>HTTP_UNAUTHORIZED (401)</code>: Client is not authorized to visit this page</li>
406<li><code>HTTP_FORBIDDEN (403)</code>: Permission denied</li>
407<li><code>HTTP_NOT_FOUND (404)</code>: File not found</li>
408<li><code>HTTP_INTERNAL_SERVER_ERROR (500)</code>: Internal server error (self explanatory)</li>
409</ul>

457<h3><a name="memory" id="memory">Memory management</a></h3>
458<p>
459Managing your resources in Apache HTTP Server 2.4 is quite easy, thanks to the memory pool 
460system. In essence, each server, connection and request have their own 
461memory pool that gets cleaned up when its scope ends, e.g. when a request 
462is done or when a server process shuts down. All your module needs to do is 
463latch onto this memory pool, and you won't have to worry about having to 
464clean up after yourself - pretty neat, huh?
465</p>
466
467<p>
468In our module, we will primarily be allocating memory for each request, so 
469it's appropriate to use the <code>r-&gt;pool</code> 
470reference when creating new objects. A few of the functions for allocating 
471memory within a pool are:
472</p>
473<ul>
474<li><code>void* <a href="http://apr.apache.org/docs/apr/1.4/group__apr__pools.html#ga85f1e193c31d109affda72f9a92c6915">apr_palloc</a>(
475apr_pool_t *p, apr_size_t size)</code>: Allocates <code>size</code> number of bytes in the pool for you</li>
476<li><code>void* <a href="http://apr.apache.org/docs/apr/1.4/group__apr__pools.html#gaf61c098ad258069d64cdf8c0a9369f9e">apr_pcalloc</a>(
477apr_pool_t *p, apr_size_t size)</code>: Allocates <code>size</code> number of bytes in the pool for you and sets all bytes to 0</li>
478<li><code>char* <a href="http://apr.apache.org/docs/apr/1.4/group__apr__strings.html#gabc79e99ff19abbd7cfd18308c5f85d47">apr_pstrdup</a>(
479apr_pool_t *p, const char *s)</code>: Creates a duplicate of the string <code>s</code>. This is useful for copying constant values so you can edit them</li>
480<li><code>char* <a href="http://apr.apache.org/docs/apr/1.4/group__apr__strings.html#ga3eca76b8d293c5c3f8021e45eda813d8">apr_psprintf</a>(
481apr_pool_t *p, const char *fmt, ...)</code>: Similar to <code>sprintf</code>, except the server supplies you with an appropriately allocated target variable</li>
482</ul>
483
484<p>Let's put these functions into an example handler:</p>
485
486
487
488<pre class="prettyprint lang-c">static int example_handler(request_rec *r)
489{
490    const char *original = "You can't edit this!";
491    char *copy;
492    int *integers;
493    
494    /* Allocate space for 10 integer values and set them all to zero. */
495    integers = apr_pcalloc(r-&gt;pool, sizeof(int)*10); 
496    
497    /* Create a copy of the 'original' variable that we can edit. */
498    copy = apr_pstrdup(r-&gt;pool, original);
499    return OK;
500}</pre>
501
502
503
504<p>
505This is all well and good for our module, which won't need any 
506pre-initialized variables or structures. However, if we wanted to 
507initialize something early on, before the requests come rolling in, we 
508could simply add a call to a function in our <code>register_hooks</code> 
509function to sort it out:
510</p>
511
512
513<pre class="prettyprint lang-c">static void register_hooks(apr_pool_t *pool)
514{
515    /* Call a function that initializes some stuff */
516    example_init_function(pool);
517    /* Create a hook in the request handler, so we get called when a request arrives */
518    ap_hook_handler(example_handler, NULL, NULL, APR_HOOK_LAST);
519}</pre>
520
521
522
523<p>
524In this pre-request initialization function we would not be using the 
525same pool as we did when allocating resources for request-based functions. 
526Instead, we would use the pool given to us by the server for allocating memory 
527on a per-process based level.
528</p>

531<h3><a name="parsing" id="parsing">Parsing request data</a></h3>
532<p>
533In our example module, we would like to add a feature, that checks which 
534type of digest, MD5 or SHA1 the client would like to see. This could be 
535solved by adding a query string to the request. A query string is typically 
536comprised of several keys and values put together in a string, for instance 
537<code>valueA=yes&amp;valueB=no&amp;valueC=maybe</code>. It is up to the 
538module itself to parse these and get the data it requires. In our example, 
539we'll be looking for a key called <code>digest</code>, and if set to <code>
540md5</code>, we'll produce an MD5 digest, otherwise we'll produce a SHA1 
541digest.
542</p>
543<p>
544Since the introduction of Apache HTTP Server 2.4, parsing request data from GET and 
545POST requests have never been easier. All we require to parse both GET and 
546POST data is four simple lines:
547</p> 
548
549
550
551<pre class="prettyprint lang-c">
552<a href="http://ci.apache.org/projects/httpd/trunk/doxygen/group__apr__tables.html#gad7ea82d6608a4a633fc3775694ab71e4">apr_table_t</a> *GET; <em>
553</em><a href="http://ci.apache.org/projects/httpd/trunk/doxygen/structapr__array__header__t.html">apr_array_header_t</a>*POST; 
554<em>
555</em>
556<a href="http://ci.apache.org/projects/httpd/trunk/doxygen/group__APACHE__CORE__SCRIPT.html#gaed25877b529623a4d8f99f819ba1b7bd">
557ap_args_to_table</a>(r, &amp;GET); <em>
558</em><a href="http://ci.apache.org/projects/httpd/trunk/doxygen/group__APACHE__CORE__DAEMON.html#ga9d426b6382b49754d4f87c55f65af202">
559ap_parse_form_data</a>(r, NULL, &amp;POST, -1, 8192);</pre>
560
561
562
563<p>
564In our specific example module, we're looking for the <code>digest</code> 
565value from the query string, which now resides inside a table called <code>
566GET</code>. To extract this value, we need only perform a simple operation:
567</p>
568
569
570
571<pre class="prettyprint lang-c">/* Get the "digest" key from the query string, if any. */
572const char *digestType = apr_table_get(GET, "digest");
573
574/* If no key was returned, we will set a default value instead. */
575if (!digestType) digestType = "sha1";</pre>
576
577
578
579<p>
580The structures used for the POST and GET data are not exactly the same, so 
581if we were to fetch a value from POST data instead of the query string, we 
582would have to resort to a few more lines, as outlined in <a href="#get_post">this example</a> in the last chapter of this document.
583</p>

586<h3><a name="advanced_handler" id="advanced_handler">Making an advanced handler</a></h3>
587<p>
588Now that we have learned how to parse form data and manage our resources, 
589we can move on to creating an advanced version of our module, that spits 
590out the MD5 or SHA1 digest of files:
591</p>
592
593
594
595<pre class="prettyprint lang-c">static int example_handler(request_rec *r)
596{
597    int rc, exists;
598    apr_finfo_t finfo;
599    apr_file_t *file;
600    char *filename;
601    char buffer[256];
602    apr_size_t readBytes;
603    int n;
604    apr_table_t *GET;
605    apr_array_header_t *POST;
606    const char *digestType;
607    
608    
609    /* Check that the "example-handler" handler is being called. */
610    if (!r-&gt;handler || strcmp(r-&gt;handler, "example-handler")) return (DECLINED);
611    
612    /* Figure out which file is being requested by removing the .sum from it */
613    filename = apr_pstrdup(r-&gt;pool, r-&gt;filename);
614    filename[strlen(filename)-4] = 0; /* Cut off the last 4 characters. */
615    
616    /* Figure out if the file we request a sum on exists and isn't a directory */
617    rc = apr_stat(&amp;finfo, filename, APR_FINFO_MIN, r-&gt;pool);
618    if (rc == APR_SUCCESS) {
619        exists =
620        (
621            (finfo.filetype != APR_NOFILE)
622        &amp;&amp;  !(finfo.filetype &amp; APR_DIR)
623        );
624        if (!exists) return HTTP_NOT_FOUND; /* Return a 404 if not found. */
625    }
626    /* If apr_stat failed, we're probably not allowed to check this file. */
627    else return HTTP_FORBIDDEN;
628    
629    /* Parse the GET and, optionally, the POST data sent to us */
630    
631    ap_args_to_table(r, &amp;GET);
632    ap_parse_form_data(r, NULL, &amp;POST, -1, 8192);
633    
634    /* Set the appropriate content type */
635    ap_set_content_type(r, "text/html");
636    
637    /* Print a title and some general information */
638    ap_rprintf(r, "&lt;h2&gt;Information on %s:&lt;/h2&gt;", filename);
639    ap_rprintf(r, "&lt;b&gt;Size:&lt;/b&gt; %u bytes&lt;br/&gt;", finfo.size);
640    
641    /* Get the digest type the client wants to see */
642    digestType = apr_table_get(GET, "digest");
643    if (!digestType) digestType = "MD5";
644    
645    
646    rc = apr_file_open(&amp;file, filename, APR_READ, APR_OS_DEFAULT, r-&gt;pool);
647    if (rc == APR_SUCCESS) {
648        
649        /* Are we trying to calculate the MD5 or the SHA1 digest? */
650        if (!strcasecmp(digestType, "md5")) {
651            /* Calculate the MD5 sum of the file */
652            union {
653                char      chr[16];
654                uint32_t  num[4];
655            } digest;
656            apr_md5_ctx_t md5;
657            apr_md5_init(&amp;md5);
658            readBytes = 256;
659            while ( apr_file_read(file, buffer, &amp;readBytes) == APR_SUCCESS ) {
660                apr_md5_update(&amp;md5, buffer, readBytes);
661            }
662            apr_md5_final(digest.chr, &amp;md5);
663            
664            /* Print out the MD5 digest */
665            ap_rputs("&lt;b&gt;MD5: &lt;/b&gt;&lt;code&gt;", r);
666            for (n = 0; n &lt; APR_MD5_DIGESTSIZE/4; n++) {
667                ap_rprintf(r, "%08x", digest.num[n]);
668            }
669            ap_rputs("&lt;/code&gt;", r);
670            /* Print a link to the SHA1 version */
671            ap_rputs("&lt;br/&gt;&lt;a href='?digest=sha1'&gt;View the SHA1 hash instead&lt;/a&gt;", r);
672        }
673        else {
674            /* Calculate the SHA1 sum of the file */
675            union {
676                char      chr[20];
677                uint32_t  num[5];
678            } digest;
679            apr_sha1_ctx_t sha1;
680            apr_sha1_init(&amp;sha1);
681            readBytes = 256;
682            while ( apr_file_read(file, buffer, &amp;readBytes) == APR_SUCCESS ) {
683                apr_sha1_update(&amp;sha1, buffer, readBytes);
684            }
685            apr_sha1_final(digest.chr, &amp;sha1);
686            
687            /* Print out the SHA1 digest */
688            ap_rputs("&lt;b&gt;SHA1: &lt;/b&gt;&lt;code&gt;", r);
689            for (n = 0; n &lt; APR_SHA1_DIGESTSIZE/4; n++) {
690                ap_rprintf(r, "%08x", digest.num[n]);
691            }
692            ap_rputs("&lt;/code&gt;", r);
693            
694            /* Print a link to the MD5 version */
695            ap_rputs("&lt;br/&gt;&lt;a href='?digest=md5'&gt;View the MD5 hash instead&lt;/a&gt;", r);
696        }
697        apr_file_close(file);
698        
699    }    
700    /* Let the server know that we responded to this request. */
701    return OK;
702}</pre>
703

707This version in its entirety can be found here: 
708<a href="mod_example_2.c">mod_example_2.c</a>.

714<h2><a name="configuration" id="configuration">Adding configuration options</a></h2>
715<p>
716In this next segment of this document, we will turn our eyes away from the 
717digest module and create a new example module, whose only function is to 
718write out its own configuration. The purpose of this is to examine how 
719the server works with configuration, and what happens when you start writing 
720advanced configurations 
721for your modules.
722</p>
723<h3><a name="config_intro" id="config_intro">An introduction to configuration 
724directives</a></h3>
725<p>
726If you are reading this, then you probably already know 
727what a configuration directive is. Simply put, a directive is a way of 
728telling an individual module (or a set of modules) how to behave, such as 
729these directives control how <code>mod_rewrite</code> works:
730</p>
731<pre class="prettyprint lang-config">RewriteEngine On
732RewriteCond "%{REQUEST_URI}" "^/foo/bar"
733RewriteRule "^/foo/bar/(.*)$" "/foobar?page=$1"</pre>
734
735<p>
736Each of these configuration directives are handled by a separate function, 
737that parses the parameters given and sets up a configuration accordingly.
738</p>
739
740<h3><a name="config_simple" id="config_simple">Making an example configuration</a></h3>
741<p>To begin with, we'll create a basic configuration in C-space:</p>
742
743
744
745<pre class="prettyprint lang-c">typedef struct {
746    int         enabled;      /* Enable or disable our module */
747    const char *path;         /* Some path to...something */
748    int         typeOfAction; /* 1 means action A, 2 means action B and so on */
749} example_config;</pre>
750
751
752
753<p>
754Now, let's put this into perspective by creating a very small module that 
755just prints out a hard-coded configuration. You'll notice that we use the 
756<code>register_hooks</code> function for initializing the configuration 
757values to their defaults:
758</p>
759
760
761<pre class="prettyprint lang-c">typedef struct {
762    int         enabled;      /* Enable or disable our module */
763    const char *path;         /* Some path to...something */
764    int         typeOfAction; /* 1 means action A, 2 means action B and so on */
765} example_config;
766
767static example_config config;
768
769static int example_handler(request_rec *r)
770{
771    if (!r-&gt;handler || strcmp(r-&gt;handler, "example-handler")) return(DECLINED);
772    ap_set_content_type(r, "text/plain");
773    ap_rprintf(r, "Enabled: %u\n", config.enabled);
774    ap_rprintf(r, "Path: %s\n", config.path);
775    ap_rprintf(r, "TypeOfAction: %x\n", config.typeOfAction);
776    return OK;
777}
778
779static void register_hooks(apr_pool_t *pool) 
780{
781    config.enabled = 1;
782    config.path = "/foo/bar";
783    config.typeOfAction = 0x00;
784    ap_hook_handler(example_handler, NULL, NULL, APR_HOOK_LAST);
785}
786
787/* Define our module as an entity and assign a function for registering hooks  */
788
789module AP_MODULE_DECLARE_DATA   example_module =
790{
791    STANDARD20_MODULE_STUFF,
792    NULL,            /* Per-directory configuration handler */
793    NULL,            /* Merge handler for per-directory configurations */
794    NULL,            /* Per-server configuration handler */
795    NULL,            /* Merge handler for per-server configurations */
796    NULL,            /* Any directives we may have for httpd */
797    register_hooks   /* Our hook registering function */
798};</pre>
799
800
801
802<p>
803So far so good. To access our new handler, we could add the following to 
804our configuration:
805</p>
806<pre class="prettyprint lang-config">&lt;Location "/example"&gt;
807    SetHandler example-handler
808&lt;/Location&gt;</pre>

810<p>
811When we visit, we'll see our current configuration being spit out by our 
812module. 
813</p>

816<h3><a name="register_directive" id="register_directive">Registering directives with the server</a></h3>
817<p>
818What if we want to change our configuration, not by hard-coding new values 
819into the module, but by using either the httpd.conf file or possibly a 
820.htaccess file? It's time to let the server know that we want this to be 
821possible. To do so, we must first change our <em>name tag</em> to include a 
822reference to the configuration directives we want to register with the server:
823</p>
824
825
826<pre class="prettyprint lang-c">module AP_MODULE_DECLARE_DATA   example_module =
827{
828    STANDARD20_MODULE_STUFF,
829    NULL,               /* Per-directory configuration handler */
830    NULL,               /* Merge handler for per-directory configurations */
831    NULL,               /* Per-server configuration handler */
832    NULL,               /* Merge handler for per-server configurations */
833    example_directives, /* Any directives we may have for httpd */
834    register_hooks      /* Our hook registering function */
835};</pre>
836
837
838
839<p>
840This will tell the server that we are now accepting directives from the 
841configuration files, and that the structure called <code>example_directives
842</code> holds information on what our directives are and how they work. 
843Since we have three different variables in our module configuration, we 
844will add a structure with three directives and a NULL at the end:
845</p>
846
847
848<pre class="prettyprint lang-c">static const command_rec        example_directives[] =
849{
850    AP_INIT_TAKE1("exampleEnabled", example_set_enabled, NULL, RSRC_CONF, "Enable or disable mod_example"),
851    AP_INIT_TAKE1("examplePath", example_set_path, NULL, RSRC_CONF, "The path to whatever"),
852    AP_INIT_TAKE2("exampleAction", example_set_action, NULL, RSRC_CONF, "Special action value!"),
853    { NULL }
854};</pre>
855
856
857
858<p>
859<img src="../images/build_a_mod_4.png" alt="Directives structure" /><br />
860As you can see, each directive needs at least 5 parameters set:
861</p>
862<ol>
863<li><code><a href="http://ci.apache.org/projects/httpd/trunk/doxygen/group__APACHE__CORE__CONFIG.html#ga07c7d22ae17805e61204463326cf9c34">AP_INIT_TAKE1</a></code>: This is a macro that tells the server that this directive takes one and only one argument. 
864If we required two arguments, we could use the macro <code><a href="http://ci.apache.org/projects/httpd/trunk/doxygen/group__APACHE__CORE__CONFIG.html#gafaec43534fcf200f37d9fecbf9247c21">AP_INIT_TAKE2</a></code> and so on (refer to httpd_conf.h 
865for more macros).</li>
866<li><code>exampleEnabled</code>: This is the name of our directive. More precisely, it is what the user must put in his/her 
867configuration in order to invoke a configuration change in our module.</li>
868<li><code>example_set_enabled</code>: This is a reference to a C function that parses the directive and sets the configuration 
869accordingly. We will discuss how to make this in the following paragraph.</li>
870<li><code>RSRC_CONF</code>: This tells the server where the directive is permitted. We'll go into details on this value in the 
871later chapters, but for now, <code>RSRC_CONF</code> means that the server will only accept these directives in a server context.</li>
872<li><code>"Enable or disable...."</code>: This is simply a brief description of what the directive does.</li>
873</ol>
874<p>
875(<em>The "missing" parameter in our definition, which is usually set to 
876<code>NULL</code>, is an optional function that can be run after the 
877initial function to parse the arguments have been run. This is usually 
878omitted, as the function for verifying arguments might as well be used to 
879set them.</em>)
880</p>

882<h3><a name="directive_handler" id="directive_handler">The directive handler function</a></h3>
883<p>
884Now that we have told the server to expect some directives for our module, it's 
885time to make a few functions for handling these. What the server reads in the 
886configuration file(s) is text, and so naturally, what it passes along to 
887our directive handler is one or more strings, that we ourselves need to 
888recognize and act upon. You'll notice, that since we set our <code>
889exampleAction</code> directive to accept two arguments, its C function also 
890has an additional parameter defined:</p> 
891
892
893<pre class="prettyprint lang-c">/* Handler for the "exampleEnabled" directive */

1066<h2><a name="context" id="context">Context aware configurations</a></h2>
1067<h3><a name="context_intro" id="context_intro">Introduction to context aware configurations</a></h3>
1068<p>
1069In Apache HTTP Server 2.4, different URLs, virtual hosts, directories etc can have very 
1070different meanings to the user of the server, and thus different contexts 
1071within which modules must operate. For example, let's assume you have this 
1072configuration set up for mod_rewrite:
1073</p>
1074<pre class="prettyprint lang-config">&lt;Directory "/var/www"&gt;
1075    RewriteCond "%{HTTP_HOST}" "^example.com$"
1076    RewriteRule "(.*)" "http://www.example.com/$1"
1077&lt;/Directory&gt;
1078&lt;Directory "/var/www/sub"&gt;
1079    RewriteRule "^foobar$" "index.php?foobar=true"
1080&lt;/Directory&gt;</pre>
1081
1082<p>
1083In this example, you will have set up two different contexts for 
1084mod_rewrite:</p>
1085<ol>
1086<li>Inside <code>/var/www</code>, all requests for <code>http://example.com</code> must go to <code>http://www.example.com</code></li>
1087<li>Inside <code>/var/www/sub</code>, all requests for <code>foobar</code> must go to <code>index.php?foobar=true</code></li>
1088</ol>
1089<p>
1090If mod_rewrite (or the entire server for that matter) wasn't context aware, then 
1091these rewrite rules would just apply to every and any request made, 
1092regardless of where and how they were made, but since the module can pull 
1093the context specific configuration straight from the server, it does not need 
1094to know itself, which of the directives are valid in this context, since 
1095the server takes care of this.</p>
1096
1097<p>
1098So how does a module get the specific configuration for the server, 
1099directory or location in question? It does so by making one simple call:
1100</p>
1101
1102
1103<pre class="prettyprint lang-c">example_config *config = (example_config*) <a href="http://ci.apache.org/projects/httpd/trunk/doxygen/group__APACHE__CORE__CONFIG.html#ga1093a5908a384eacc929b028c79f2a02">ap_get_module_config</a>(r-&gt;per_dir_config, &amp;example_module);</pre>
1104
1105
1106
1107<p>
1108That's it! Of course, a whole lot goes on behind the scenes, which we will 
1109discuss in this chapter, starting with how the server came to know what our 
1110configuration looks like, and how it came to be set up as it is in the 
1111specific context.
1112</p>
1113

1151<h3><a name="context_which" id="context_which">Choosing a context</a></h3>
1152<p>
1153Before we can start making our module context aware, we must first define, 
1154which contexts we will accept. As we saw in the previous chapter, defining 
1155a directive required five elements be set:</p>
1156
1157
1158
1159<pre class="prettyprint lang-c">AP_INIT_TAKE1("exampleEnabled", example_set_enabled, NULL, RSRC_CONF, "Enable or disable mod_example"),</pre>
1160
1161
1162
1163
1164<p>The <code>RSRC_CONF</code> definition told the server that we would only allow 
1165this directive in a global server context, but since we are now trying out 
1166a context aware version of our module, we should set this to something 
1167more lenient, namely the value <code>ACCESS_CONF</code>, which lets us use 
1168the directive inside &lt;Directory&gt; and &lt;Location&gt; blocks. For more 
1169control over the placement of your directives, you can combine the following 
1170restrictions together to form a specific rule:
1171</p>
1172<ul>
1173<li><code>RSRC_CONF</code>: Allow in .conf files (not .htaccess) outside &lt;Directory&gt; or &lt;Location&gt;</li>
1174<li><code>ACCESS_CONF</code>: Allow in .conf files (not .htaccess) inside &lt;Directory&gt; or &lt;Location&gt;</li>
1175<li><code>OR_OPTIONS</code>: Allow in .conf files and .htaccess when <code>AllowOverride Options</code> is set</li>
1176<li><code>OR_FILEINFO</code>: Allow in .conf files and .htaccess when <code>AllowOverride FileInfo</code> is set</li>
1177<li><code>OR_AUTHCFG</code>: Allow in .conf files and .htaccess when <code>AllowOverride AuthConfig</code> is set</li>
1178<li><code>OR_INDEXES</code>: Allow in .conf files and .htaccess when <code>AllowOverride Indexes</code> is set</li>
1179<li><code>OR_ALL</code>: Allow anywhere in .conf files and .htaccess</li>
1180</ul>

1234<h3><a name="context_merge" id="context_merge">Merging configurations</a></h3>
1235<p>
1236Our next step in creating a context aware configuration is merging 
1237configurations. This part of the process particularly applies to scenarios 
1238where you have a parent configuration and a child, such as the following: 
1239</p>
1240<pre class="prettyprint lang-config">&lt;Directory "/var/www"&gt;
1241    ExampleEnabled On
1242    ExamplePath "/foo/bar"
1243    ExampleAction file allow
1244&lt;/Directory&gt;
1245&lt;Directory "/var/www/subdir"&gt;
1246    ExampleAction file deny
1247&lt;/Directory&gt;</pre>
1248
1249<p>
1250In this example, it is natural to assume that the directory <code>
1251/var/www/subdir</code> should inherit the values set for the <code>/var/www
1252</code> directory, as we did not specify an <code>ExampleEnabled</code> nor 
1253an <code>ExamplePath</code> for this directory. The server does not presume to 
1254know if this is true, but cleverly does the following:
1255</p>
1256<ol>
1257<li>Creates a new configuration for <code>/var/www</code></li>
1258<li>Sets the configuration values according to the directives given for <code>/var/www</code></li>
1259<li>Creates a new configuration for <code>/var/www/subdir</code></li>
1260<li>Sets the configuration values according to the directives given for <code>/var/www/subdir</code></li>
1261<li><strong>Proposes a merge</strong> of the two configurations into a new configuration for <code>/var/www/subdir</code></li>
1262</ol>
1263<p>
1264This proposal is handled by the <code>merge_dir_conf</code> function we 
1265referenced in our name tag. The purpose of this function is to assess the 
1266two configurations and decide how they are to be merged:</p>
1267
1268
1269
1270<pre class="prettyprint lang-c">void *merge_dir_conf(apr_pool_t *pool, void *BASE, void *ADD) {
1271    example_config *base = (example_config *) BASE ; /* This is what was set in the parent context */
1272    example_config *add = (example_config *) ADD ;   /* This is what is set in the new context */
1273    example_config *conf = (example_config *) create_dir_conf(pool, "Merged configuration"); /* This will be the merged configuration */
1274    
1275    /* Merge configurations */
1276    conf-&gt;enabled = ( add-&gt;enabled == 0 ) ? base-&gt;enabled : add-&gt;enabled ;
1277    conf-&gt;typeOfAction = add-&gt;typeOfAction ? add-&gt;typeOfAction : base-&gt;typeOfAction;
1278    strcpy(conf-&gt;path, strlen(add-&gt;path) ? add-&gt;path : base-&gt;path);
1279    
1280    return conf ;
1281}</pre>
