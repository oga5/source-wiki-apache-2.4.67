36    to deal with, <em>etc.</em> They are not yet nearly complete, but hopefully,
37    they will help you get your bearings. Keep in mind that the API is still
38    subject to change as we gain experience with it. (See the TODO file for
39    what <em>might</em> be coming). However, it will be easy to adapt modules
40    to any changes that are made. (We have more modules to adapt than you
41    do).</p>

116<h2><a name="basics" id="basics">Basic concepts</a></h2>
117    <p>We begin with an overview of the basic concepts behind the API, and how
118    they are manifested in the code.</p>
119
120    <h3><a name="HMR" id="HMR">Handlers, Modules, and Requests</a></h3>
121      <p>Apache breaks down request handling into a series of steps, more or
122      less the same way the Netscape server API does (although this API has a
123      few more stages than NetSite does, as hooks for stuff I thought might be
124      useful in the future). These are:</p>
125
126      <ul>
127      <li>URI -&gt; Filename translation</li>
128      <li>Auth ID checking [is the user who they say they are?]</li>
129      <li>Auth access checking [is the user authorized <em>here</em>?]</li>
130      <li>Access checking other than auth</li>
131      <li>Determining MIME type of the object requested</li>
132      <li>`Fixups' -- there aren't any of these yet, but the phase is intended
133      as a hook for possible extensions like <code class="directive"><a href="../mod/mod_env.html#setenv">SetEnv</a></code>, which don't really fit well elsewhere.</li>
134      <li>Actually sending a response back to the client.</li>
135      <li>Logging the request</li>
136      </ul>
137
138      <p>These phases are handled by looking at each of a succession of
139      <em>modules</em>, looking to see if each of them has a handler for the
140      phase, and attempting invoking it if so. The handler can typically do one
141      of three things:</p>
142
143      <ul>
144      <li><em>Handle</em> the request, and indicate that it has done so by
145      returning the magic constant <code>OK</code>.</li>
146
147      <li><em>Decline</em> to handle the request, by returning the magic integer
148      constant <code>DECLINED</code>. In this case, the server behaves in all
149      respects as if the handler simply hadn't been there.</li>
150
151      <li>Signal an error, by returning one of the HTTP error codes. This
152      terminates normal handling of the request, although an ErrorDocument may
153      be invoked to try to mop up, and it will be logged in any case.</li>
154      </ul>
155
156      <p>Most phases are terminated by the first module that handles them;
157      however, for logging, `fixups', and non-access authentication checking,
158      all handlers always run (barring an error). Also, the response phase is
159      unique in that modules may declare multiple handlers for it, via a
160      dispatch table keyed on the MIME type of the requested object. Modules may
161      declare a response-phase handler which can handle <em>any</em> request,
162      by giving it the key <code>*/*</code> (<em>i.e.</em>, a wildcard MIME type
163      specification). However, wildcard handlers are only invoked if the server
164      has already tried and failed to find a more specific response handler for
165      the MIME type of the requested object (either none existed, or they all
166      declined).</p>
167
168      <p>The handlers themselves are functions of one argument (a
169      <code>request_rec</code> structure. vide infra), which returns an integer,
170      as above.</p>
171    
172
173    <h3><a name="moduletour" id="moduletour">A brief tour of a module</a></h3>
174      <p>At this point, we need to explain the structure of a module. Our
175      candidate will be one of the messier ones, the CGI module -- this handles
176      both CGI scripts and the <code class="directive"><a href="../mod/mod_alias.html#scriptalias">ScriptAlias</a></code> config file command. It's actually a great deal
177      more complicated than most modules, but if we're going to have only one
178      example, it might as well be the one with its fingers in every place.</p>
179
180      <p>Let's begin with handlers. In order to handle the CGI scripts, the
181      module declares a response handler for them. Because of <code class="directive"><a href="../mod/mod_alias.html#scriptalias">ScriptAlias</a></code>, it also has handlers for the
182      name translation phase (to recognize <code class="directive"><a href="../mod/mod_alias.html#scriptalias">ScriptAlias</a></code>ed URIs), the type-checking phase (any
183      <code class="directive"><a href="../mod/mod_alias.html#scriptalias">ScriptAlias</a></code>ed request is typed
184      as a CGI script).</p>
185
186      <p>The module needs to maintain some per (virtual) server information,
187      namely, the <code class="directive"><a href="../mod/mod_alias.html#scriptalias">ScriptAlias</a></code>es in
188      effect; the module structure therefore contains pointers to a functions
189      which builds these structures, and to another which combines two of them
190      (in case the main server and a virtual server both have <code class="directive"><a href="../mod/mod_alias.html#scriptalias">ScriptAlias</a></code>es declared).</p>
191
192      <p>Finally, this module contains code to handle the <code class="directive"><a href="../mod/mod_alias.html#scriptalias">ScriptAlias</a></code> command itself. This particular
193      module only declares one command, but there could be more, so modules have
194      <em>command tables</em> which declare their commands, and describe where
195      they are permitted, and how they are to be invoked.</p>
196
197      <p>A final note on the declared types of the arguments of some of these
198      commands: a <code>pool</code> is a pointer to a <em>resource pool</em>
199      structure; these are used by the server to keep track of the memory which
200      has been allocated, files opened, <em>etc.</em>, either to service a
201      particular request, or to handle the process of configuring itself. That
202      way, when the request is over (or, for the configuration pool, when the
203      server is restarting), the memory can be freed, and the files closed,
204      <em>en masse</em>, without anyone having to write explicit code to track
205      them all down and dispose of them. Also, a <code>cmd_parms</code>
206      structure contains various information about the config file being read,
207      and other status information, which is sometimes of use to the function
208      which processes a config-file command (such as <code class="directive"><a href="../mod/mod_alias.html#scriptalias">ScriptAlias</a></code>). With no further ado, the
209      module itself:</p>
210

272<h2><a name="handlers" id="handlers">How handlers work</a></h2>
273    <p>The sole argument to handlers is a <code>request_rec</code> structure.
274    This structure describes a particular request which has been made to the
275    server, on behalf of a client. In most cases, each connection to the
276    client generates only one <code>request_rec</code> structure.</p>
277
278    <h3><a name="req_tour" id="req_tour">A brief tour of the request_rec</a></h3>
279      <p>The <code>request_rec</code> contains pointers to a resource pool
280      which will be cleared when the server is finished handling the request;
281      to structures containing per-server and per-connection information, and
282      most importantly, information on the request itself.</p>
283
284      <p>The most important such information is a small set of character strings
285      describing attributes of the object being requested, including its URI,
286      filename, content-type and content-encoding (these being filled in by the
287      translation and type-check handlers which handle the request,
288      respectively).</p>
289
290      <p>Other commonly used data items are tables giving the MIME headers on
291      the client's original request, MIME headers to be sent back with the
292      response (which modules can add to at will), and environment variables for
293      any subprocesses which are spawned off in the course of servicing the
294      request. These tables are manipulated using the <code>ap_table_get</code>
295      and <code>ap_table_set</code> routines.</p>
296
297      <div class="note">
298        <p>Note that the <code>Content-type</code> header value <em>cannot</em>
299        be set by module content-handlers using the <code>ap_table_*()</code>
300        routines. Rather, it is set by pointing the <code>content_type</code>
301        field in the <code>request_rec</code> structure to an appropriate
302        string. <em>e.g.</em>,</p>
303        <div class="example"><p><code>
304          r-&gt;content_type = "text/html";
305        </code></p></div>
306      </div>
307
308      <p>Finally, there are pointers to two data structures which, in turn,
309      point to per-module configuration structures. Specifically, these hold
310      pointers to the data structures which the module has built to describe
311      the way it has been configured to operate in a given directory (via
312      <code>.htaccess</code> files or <code class="directive"><a href="../mod/core.html#directory">&lt;Directory&gt;</a></code> sections), for private data it has built in the
313      course of servicing the request (so modules' handlers for one phase can
314      pass `notes' to their handlers for other phases). There is another such
315      configuration vector in the <code>server_rec</code> data structure pointed
316      to by the <code>request_rec</code>, which contains per (virtual) server
317      configuration data.</p>
318
319      <p>Here is an abridged declaration, giving the fields most commonly
320      used:</p>
321

389    <h3><a name="req_orig" id="req_orig">Where request_rec structures come from</a></h3>
390      <p>Most <code>request_rec</code> structures are built by reading an HTTP
391      request from a client, and filling in the fields. However, there are a
392      few exceptions:</p>
393
394      <ul>
395      <li>If the request is to an imagemap, a type map (<em>i.e.</em>, a
396      <code>*.var</code> file), or a CGI script which returned a local
397      `Location:', then the resource which the user requested is going to be
398      ultimately located by some URI other than what the client originally
399      supplied. In this case, the server does an <em>internal redirect</em>,
400      constructing a new <code>request_rec</code> for the new URI, and
401      processing it almost exactly as if the client had requested the new URI
402      directly.</li>
403
404      <li>If some handler signaled an error, and an <code>ErrorDocument</code>
405      is in scope, the same internal redirect machinery comes into play.</li>
406
407      <li><p>Finally, a handler occasionally needs to investigate `what would
408      happen if' some other request were run. For instance, the directory
409      indexing module needs to know what MIME type would be assigned to a
410      request for each directory entry, in order to figure out what icon to
411      use.</p>
412
413      <p>Such handlers can construct a <em>sub-request</em>, using the
414      functions <code>ap_sub_req_lookup_file</code>,
415      <code>ap_sub_req_lookup_uri</code>, and <code>ap_sub_req_method_uri</code>;
416      these construct a new <code>request_rec</code> structure and processes it
417      as you would expect, up to but not including the point of actually sending
418      a response. (These functions skip over the access checks if the
419      sub-request is for a file in the same directory as the original
420      request).</p>
421
422      <p>(Server-side includes work by building sub-requests and then actually
423      invoking the response handler for them, via the function
424      <code>ap_run_sub_req</code>).</p>
425      </li>
426      </ul>
427    
428
429    <h3><a name="req_return" id="req_return">Handling requests, declining, and returning
430    error codes</a></h3>
431      <p>As discussed above, each handler, when invoked to handle a particular
432      <code>request_rec</code>, has to return an <code>int</code> to indicate
433      what happened. That can either be</p>
434
435      <ul>
436      <li><code>OK</code> -- the request was handled successfully. This may or
437      may not terminate the phase.</li>
438
439      <li><code>DECLINED</code> -- no erroneous condition exists, but the module
440      declines to handle the phase; the server tries to find another.</li>
441
442      <li>an HTTP error code, which aborts handling of the request.</li>
443      </ul>
444
445      <p>Note that if the error code returned is <code>REDIRECT</code>, then
446      the module should put a <code>Location</code> in the request's
447      <code>headers_out</code>, to indicate where the client should be
448      redirected <em>to</em>.</p>
449    
450
451    <h3><a name="resp_handlers" id="resp_handlers">Special considerations for response
452    handlers</a></h3>
453      <p>Handlers for most phases do their work by simply setting a few fields
454      in the <code>request_rec</code> structure (or, in the case of access
455      checkers, simply by returning the correct error code). However, response
456      handlers have to actually send a request back to the client.</p>
457
458      <p>They should begin by sending an HTTP response header, using the
459      function <code>ap_send_http_header</code>. (You don't have to do anything
460      special to skip sending the header for HTTP/0.9 requests; the function
461      figures out on its own that it shouldn't do anything). If the request is
462      marked <code>header_only</code>, that's all they should do; they should
463      return after that, without attempting any further output.</p>
464
465      <p>Otherwise, they should produce a request body which responds to the
466      client as appropriate. The primitives for this are <code>ap_rputc</code>
467      and <code>ap_rprintf</code>, for internally generated output, and
468      <code>ap_send_fd</code>, to copy the contents of some <code>FILE *</code>
469      straight to the client.</p>
470
471      <p>At this point, you should more or less understand the following piece
472      of code, which is the handler which handles <code>GET</code> requests
473      which have no more specific handler; it also shows how conditional
474      <code>GET</code>s can be handled, if it's desirable to do so in a
475      particular response handler -- <code>ap_set_last_modified</code> checks
476      against the <code>If-modified-since</code> value supplied by the client,
477      if any, and returns an appropriate code (which will, if nonzero, be
478      USE_LOCAL_COPY). No similar considerations apply for
479      <code>ap_set_content_length</code>, but it returns an error code for
480      symmetry.</p>
481
482      <div class="example"><p><code>
483        int default_handler (request_rec *r)<br />
484        {<br />
485        <span class="indent">
486          int errstatus;<br />
487          FILE *f;<br />
488          <br />
489          if (r-&gt;method_number != M_GET) return DECLINED;<br />
490          if (r-&gt;finfo.st_mode == 0) return NOT_FOUND;<br />
491          <br />
492          if ((errstatus = ap_set_content_length (r, r-&gt;finfo.st_size))<br />
493          &nbsp;&nbsp;&nbsp;&nbsp;||
494             (errstatus = ap_set_last_modified (r, r-&gt;finfo.st_mtime)))<br />
495          return errstatus;<br />
496          <br />
497          f = fopen (r-&gt;filename, "r");<br />
498          <br />
499          if (f == NULL) {<br />
500          <span class="indent">
501            log_reason("file permissions deny server access", r-&gt;filename, r);<br />
502            return FORBIDDEN;<br />
503          </span>
504          }<br />
505          <br />
506          register_timeout ("send", r);<br />
507          ap_send_http_header (r);<br />
508          <br />
509          if (!r-&gt;header_only) send_fd (f, r);<br />
510          ap_pfclose (r-&gt;pool, f);<br />
511          return OK;<br />
512        </span>
513        }
514      </code></p></div>
515
516      <p>Finally, if all of this is too much of a challenge, there are a few
517      ways out of it. First off, as shown above, a response handler which has
518      not yet produced any output can simply return an error code, in which
519      case the server will automatically produce an error response. Secondly,
520      it can punt to some other handler by invoking
521      <code>ap_internal_redirect</code>, which is how the internal redirection
522      machinery discussed above is invoked. A response handler which has
523      internally redirected should always return <code>OK</code>.</p>
524
525      <p>(Invoking <code>ap_internal_redirect</code> from handlers which are
526      <em>not</em> response handlers will lead to serious confusion).</p>

565<h2><a name="pools" id="pools">Resource allocation and resource pools</a></h2>
566    <p>One of the problems of writing and designing a server-pool server is
567    that of preventing leakage, that is, allocating resources (memory, open
568    files, <em>etc.</em>), without subsequently releasing them. The resource
569    pool machinery is designed to make it easy to prevent this from happening,
570    by allowing resource to be allocated in such a way that they are
571    <em>automatically</em> released when the server is done with them.</p>
572
573    <p>The way this works is as follows: the memory which is allocated, file
574    opened, <em>etc.</em>, to deal with a particular request are tied to a
575    <em>resource pool</em> which is allocated for the request. The pool is a
576    data structure which itself tracks the resources in question.</p>
577
578    <p>When the request has been processed, the pool is <em>cleared</em>. At
579    that point, all the memory associated with it is released for reuse, all
580    files associated with it are closed, and any other clean-up functions which
581    are associated with the pool are run. When this is over, we can be confident
582    that all the resource tied to the pool have been released, and that none of
583    them have leaked.</p>
584
585    <p>Server restarts, and allocation of memory and resources for per-server
586    configuration, are handled in a similar way. There is a <em>configuration
587    pool</em>, which keeps track of resources which were allocated while reading
588    the server configuration files, and handling the commands therein (for
589    instance, the memory that was allocated for per-server module configuration,
590    log files and other files that were opened, and so forth). When the server
591    restarts, and has to reread the configuration files, the configuration pool
592    is cleared, and so the memory and file descriptors which were taken up by
593    reading them the last time are made available for reuse.</p>
594
595    <p>It should be noted that use of the pool machinery isn't generally
596    obligatory, except for situations like logging handlers, where you really
597    need to register cleanups to make sure that the log file gets closed when
598    the server restarts (this is most easily done by using the function <code><a href="#pool-files">ap_pfopen</a></code>, which also arranges for the
599    underlying file descriptor to be closed before any child processes, such as
600    for CGI scripts, are <code>exec</code>ed), or in case you are using the
601    timeout machinery (which isn't yet even documented here). However, there are
602    two benefits to using it: resources allocated to a pool never leak (even if
603    you allocate a scratch string, and just forget about it); also, for memory
604    allocation, <code>ap_palloc</code> is generally faster than
605    <code>malloc</code>.</p>
606
607    <p>We begin here by describing how memory is allocated to pools, and then
608    discuss how other resources are tracked by the resource pool machinery.</p>
609
610    <h3>Allocation of memory in pools</h3>
611      <p>Memory is allocated to pools by calling the function
612      <code>ap_palloc</code>, which takes two arguments, one being a pointer to
613      a resource pool structure, and the other being the amount of memory to
614      allocate (in <code>char</code>s). Within handlers for handling requests,
615      the most common way of getting a resource pool structure is by looking at
616      the <code>pool</code> slot of the relevant <code>request_rec</code>; hence
617      the repeated appearance of the following idiom in module code:</p>
618
619      <div class="example"><p><code>
620        int my_handler(request_rec *r)<br />
621        {<br />
622        <span class="indent">
623          struct my_structure *foo;<br />
624          ...<br />
625          <br />
626          foo = (foo *)ap_palloc (r-&gt;pool, sizeof(my_structure));<br />
627        </span>
628        }
629      </code></p></div>
630
631      <p>Note that <em>there is no <code>ap_pfree</code></em> --
632      <code>ap_palloc</code>ed memory is freed only when the associated resource
633      pool is cleared. This means that <code>ap_palloc</code> does not have to
634      do as much accounting as <code>malloc()</code>; all it does in the typical
635      case is to round up the size, bump a pointer, and do a range check.</p>
636
637      <p>(It also raises the possibility that heavy use of
638      <code>ap_palloc</code> could cause a server process to grow excessively
639      large. There are two ways to deal with this, which are dealt with below;
640      briefly, you can use <code>malloc</code>, and try to be sure that all of
641      the memory gets explicitly <code>free</code>d, or you can allocate a
642      sub-pool of the main pool, allocate your memory in the sub-pool, and clear
643      it out periodically. The latter technique is discussed in the section
644      on sub-pools below, and is used in the directory-indexing code, in order
645      to avoid excessive storage allocation when listing directories with
646      thousands of files).</p>
647    
648
649    <h3>Allocating initialized memory</h3>
650      <p>There are functions which allocate initialized memory, and are
651      frequently useful. The function <code>ap_pcalloc</code> has the same
652      interface as <code>ap_palloc</code>, but clears out the memory it
653      allocates before it returns it. The function <code>ap_pstrdup</code>
654      takes a resource pool and a <code>char *</code> as arguments, and
655      allocates memory for a copy of the string the pointer points to, returning
656      a pointer to the copy. Finally <code>ap_pstrcat</code> is a varargs-style
657      function, which takes a pointer to a resource pool, and at least two
658      <code>char *</code> arguments, the last of which must be
659      <code>NULL</code>. It allocates enough memory to fit copies of each of
660      the strings, as a unit; for instance:</p>
661
662      <div class="example"><p><code>
663        ap_pstrcat (r-&gt;pool, "foo", "/", "bar", NULL);
664      </code></p></div>
665
666      <p>returns a pointer to 8 bytes worth of memory, initialized to
667      <code>"foo/bar"</code>.</p>
668    
669
670    <h3><a name="pools-used" id="pools-used">Commonly-used pools in the Apache Web
671    server</a></h3>
672      <p>A pool is really defined by its lifetime more than anything else.
673      There are some static pools in http_main which are passed to various
674      non-http_main functions as arguments at opportune times. Here they
675      are:</p>
676
677      <dl>
678      <dt><code>permanent_pool</code></dt>
679      <dd>never passed to anything else, this is the ancestor of all pools</dd>
680
681      <dt><code>pconf</code></dt>
682      <dd>
683        <ul>
684          <li>subpool of permanent_pool</li>
685
686          <li>created at the beginning of a config "cycle"; exists
687          until the server is terminated or restarts; passed to all
688          config-time routines, either via cmd-&gt;pool, or as the
689          "pool *p" argument on those which don't take pools</li>
690
691          <li>passed to the module init() functions</li>
692        </ul>
693      </dd>
694
695      <dt><code>ptemp</code></dt>
696      <dd>
697        <ul>
698          <li>sorry I lie, this pool isn't called this currently in
699          1.3, I renamed it this in my pthreads development. I'm
700          referring to the use of ptrans in the parent... contrast
701          this with the later definition of ptrans in the
702          child.</li>
703
704          <li>subpool of permanent_pool</li>
705
706          <li>created at the beginning of a config "cycle"; exists
707          until the end of config parsing; passed to config-time
708          routines <em>via</em> cmd-&gt;temp_pool. Somewhat of a
709          "bastard child" because it isn't available everywhere.
710          Used for temporary scratch space which may be needed by
711          some config routines but which is deleted at the end of
712          config.</li>
713        </ul>
714      </dd>
715
716      <dt><code>pchild</code></dt>
717      <dd>
718        <ul>
719          <li>subpool of permanent_pool</li>
720
721          <li>created when a child is spawned (or a thread is
722          created); lives until that child (thread) is
723          destroyed</li>
724
725          <li>passed to the module child_init functions</li>
726
727          <li>destruction happens right after the child_exit
728          functions are called... (which may explain why I think
729          child_exit is redundant and unneeded)</li>
730        </ul>
731      </dd>
732
733      <dt><code>ptrans</code></dt>
734      <dd>
735        <ul>
736          <li>should be a subpool of pchild, but currently is a
737          subpool of permanent_pool, see above</li>
738
739          <li>cleared by the child before going into the accept()
740          loop to receive a connection</li>
741
742          <li>used as connection-&gt;pool</li>
743        </ul>
744      </dd>
745
746      <dt><code>r-&gt;pool</code></dt>
747      <dd>
748        <ul>
749          <li>for the main request this is a subpool of
750          connection-&gt;pool; for subrequests it is a subpool of
751          the parent request's pool.</li>
752
753          <li>exists until the end of the request (<em>i.e.</em>,
754          ap_destroy_sub_req, or in child_main after
755          process_request has finished)</li>
756
757          <li>note that r itself is allocated from r-&gt;pool;
758          <em>i.e.</em>, r-&gt;pool is first created and then r is
759          the first thing palloc()d from it</li>
760        </ul>
761      </dd>
762      </dl>
763
764      <p>For almost everything folks do, <code>r-&gt;pool</code> is the pool to
765      use. But you can see how other lifetimes, such as pchild, are useful to
766      some modules... such as modules that need to open a database connection
767      once per child, and wish to clean it up when the child dies.</p>
768
769      <p>You can also see how some bugs have manifested themself, such as
770      setting <code>connection-&gt;user</code> to a value from
771      <code>r-&gt;pool</code> -- in this case connection exists for the
772      lifetime of <code>ptrans</code>, which is longer than
773      <code>r-&gt;pool</code> (especially if <code>r-&gt;pool</code> is a
774      subrequest!). So the correct thing to do is to allocate from
775      <code>connection-&gt;pool</code>.</p>
776
777      <p>And there was another interesting bug in <code class="module"><a href="../mod/mod_include.html">mod_include</a></code>
778      / <code class="module"><a href="../mod/mod_cgi.html">mod_cgi</a></code>. You'll see in those that they do this test
779      to decide if they should use <code>r-&gt;pool</code> or
780      <code>r-&gt;main-&gt;pool</code>. In this case the resource that they are
781      registering for cleanup is a child process. If it were registered in
782      <code>r-&gt;pool</code>, then the code would <code>wait()</code> for the
783      child when the subrequest finishes. With <code class="module"><a href="../mod/mod_include.html">mod_include</a></code> this
784      could be any old <code>#include</code>, and the delay can be up to 3
785      seconds... and happened quite frequently. Instead the subprocess is
786      registered in <code>r-&gt;main-&gt;pool</code> which causes it to be
787      cleaned up when the entire request is done -- <em>i.e.</em>, after the
788      output has been sent to the client and logging has happened.</p>

790
791    <h3><a name="pool-files" id="pool-files">Tracking open files, etc.</a></h3>
792      <p>As indicated above, resource pools are also used to track other sorts
793      of resources besides memory. The most common are open files. The routine
794      which is typically used for this is <code>ap_pfopen</code>, which takes a
795      resource pool and two strings as arguments; the strings are the same as
796      the typical arguments to <code>fopen</code>, <em>e.g.</em>,</p>
797
798      <div class="example"><p><code>
799        ...<br />
800        FILE *f = ap_pfopen (r-&gt;pool, r-&gt;filename, "r");<br />
801        <br />
802        if (f == NULL) { ... } else { ... }<br />
803      </code></p></div>
804
805      <p>There is also a <code>ap_popenf</code> routine, which parallels the
806      lower-level <code>open</code> system call. Both of these routines arrange
807      for the file to be closed when the resource pool in question is
808      cleared.</p>
809
810      <p>Unlike the case for memory, there <em>are</em> functions to close files
811      allocated with <code>ap_pfopen</code>, and <code>ap_popenf</code>, namely
812      <code>ap_pfclose</code> and <code>ap_pclosef</code>. (This is because, on
813      many systems, the number of files which a single process can have open is
814      quite limited). It is important to use these functions to close files
815      allocated with <code>ap_pfopen</code> and <code>ap_popenf</code>, since to
816      do otherwise could cause fatal errors on systems such as Linux, which
817      react badly if the same <code>FILE*</code> is closed more than once.</p>
818
819      <p>(Using the <code>close</code> functions is not mandatory, since the
820      file will eventually be closed regardless, but you should consider it in
821      cases where your module is opening, or could open, a lot of files).</p>
822    
823
824    <h3>Other sorts of resources -- cleanup functions</h3>
825      <p>More text goes here. Describe the cleanup primitives in terms of
826      which the file stuff is implemented; also, <code>spawn_process</code>.</p>
827
828      <p>Pool cleanups live until <code>clear_pool()</code> is called:
829      <code>clear_pool(a)</code> recursively calls <code>destroy_pool()</code>
830      on all subpools of <code>a</code>; then calls all the cleanups for
831      <code>a</code>; then releases all the memory for <code>a</code>.
832      <code>destroy_pool(a)</code> calls <code>clear_pool(a)</code> and then
833      releases the pool structure itself. <em>i.e.</em>,
834      <code>clear_pool(a)</code> doesn't delete <code>a</code>, it just frees
835      up all the resources and you can start using it again immediately.</p>
836    
837
838    <h3>Fine control -- creating and dealing with sub-pools, with
839    a note on sub-requests</h3>
840      <p>On rare occasions, too-free use of <code>ap_palloc()</code> and the
841      associated primitives may result in undesirably profligate resource
842      allocation. You can deal with such a case by creating a <em>sub-pool</em>,
843      allocating within the sub-pool rather than the main pool, and clearing or
844      destroying the sub-pool, which releases the resources which were
845      associated with it. (This really <em>is</em> a rare situation; the only
846      case in which it comes up in the standard module set is in case of listing
847      directories, and then only with <em>very</em> large directories.
848      Unnecessary use of the primitives discussed here can hair up your code
849      quite a bit, with very little gain).</p>
850
851      <p>The primitive for creating a sub-pool is <code>ap_make_sub_pool</code>,
852      which takes another pool (the parent pool) as an argument. When the main
853      pool is cleared, the sub-pool will be destroyed. The sub-pool may also be
854      cleared or destroyed at any time, by calling the functions
855      <code>ap_clear_pool</code> and <code>ap_destroy_pool</code>, respectively.
856      (The difference is that <code>ap_clear_pool</code> frees resources
857      associated with the pool, while <code>ap_destroy_pool</code> also
858      deallocates the pool itself. In the former case, you can allocate new
859      resources within the pool, and clear it again, and so forth; in the
860      latter case, it is simply gone).</p>
861
862      <p>One final note -- sub-requests have their own resource pools, which are
863      sub-pools of the resource pool for the main request. The polite way to
864      reclaim the resources associated with a sub request which you have
865      allocated (using the <code>ap_sub_req_...</code> functions) is
866      <code>ap_destroy_sub_req</code>, which frees the resource pool. Before
867      calling this function, be sure to copy anything that you care about which
868      might be allocated in the sub-request's resource pool into someplace a
869      little less volatile (for instance, the filename in its
870      <code>request_rec</code> structure).</p>
871
872      <p>(Again, under most circumstances, you shouldn't feel obliged to call
873      this function; only 2K of memory or so are allocated for a typical sub
874      request, and it will be freed anyway when the main request pool is
875      cleared. It is only when you are allocating many, many sub-requests for a
876      single main request that you should seriously consider the
877      <code>ap_destroy_...</code> functions).</p>

881<h2><a name="config" id="config">Configuration, commands and the like</a></h2>
882    <p>One of the design goals for this server was to maintain external
883    compatibility with the NCSA 1.3 server --- that is, to read the same
884    configuration files, to process all the directives therein correctly, and
885    in general to be a drop-in replacement for NCSA. On the other hand, another
886    design goal was to move as much of the server's functionality into modules
887    which have as little as possible to do with the monolithic server core. The
888    only way to reconcile these goals is to move the handling of most commands
889    from the central server into the modules.</p>
890
891    <p>However, just giving the modules command tables is not enough to divorce
892    them completely from the server core. The server has to remember the
893    commands in order to act on them later. That involves maintaining data which
894    is private to the modules, and which can be either per-server, or
895    per-directory. Most things are per-directory, including in particular access
896    control and authorization information, but also information on how to
897    determine file types from suffixes, which can be modified by
898    <code class="directive"><a href="../mod/mod_mime.html#addtype">AddType</a></code> and <code class="directive"><a href="../mod/core.html#forcetype">ForceType</a></code> directives, and so forth. In general,
899    the governing philosophy is that anything which <em>can</em> be made
900    configurable by directory should be; per-server information is generally
901    used in the standard set of modules for information like
902    <code class="directive"><a href="../mod/mod_alias.html#alias">Alias</a></code>es and <code class="directive"><a href="../mod/mod_alias.html#redirect">Redirect</a></code>s which come into play before the
903    request is tied to a particular place in the underlying file system.</p>
904
905    <p>Another requirement for emulating the NCSA server is being able to handle
906    the per-directory configuration files, generally called
907    <code>.htaccess</code> files, though even in the NCSA server they can
908    contain directives which have nothing at all to do with access control.
909    Accordingly, after URI -&gt; filename translation, but before performing any
910    other phase, the server walks down the directory hierarchy of the underlying
911    filesystem, following the translated pathname, to read any
912    <code>.htaccess</code> files which might be present. The information which
913    is read in then has to be <em>merged</em> with the applicable information
914    from the server's own config files (either from the <code class="directive"><a href="../mod/core.html#directory">&lt;Directory&gt;</a></code> sections in
915    <code>access.conf</code>, or from defaults in <code>srm.conf</code>, which
916    actually behaves for most purposes almost exactly like <code>&lt;Directory
917    /&gt;</code>).</p>
918
919    <p>Finally, after having served a request which involved reading
920    <code>.htaccess</code> files, we need to discard the storage allocated for
921    handling them. That is solved the same way it is solved wherever else
922    similar problems come up, by tying those structures to the per-transaction
923    resource pool.</p>
924
925    <h3><a name="per-dir" id="per-dir">Per-directory configuration structures</a></h3>
926      <p>Let's look out how all of this plays out in <code>mod_mime.c</code>,
927      which defines the file typing handler which emulates the NCSA server's
928      behavior of determining file types from suffixes. What we'll be looking
929      at, here, is the code which implements the <code class="directive"><a href="../mod/mod_mime.html#addtype">AddType</a></code> and <code class="directive"><a href="../mod/mod_mime.html#addencoding">AddEncoding</a></code> commands. These commands can appear in
930      <code>.htaccess</code> files, so they must be handled in the module's
931      private per-directory data, which in fact, consists of two separate
932      tables for MIME types and encoding information, and is declared as
933      follows:</p>
934
935      <div class="example"><pre>typedef struct {
936    table *forced_types;      /* Additional AddTyped stuff */
937    table *encoding_types;    /* Added with AddEncoding... */
938} mime_dir_config;</pre></div>
939
940      <p>When the server is reading a configuration file, or <code class="directive"><a href="../mod/core.html#directory">&lt;Directory&gt;</a></code> section, which includes
941      one of the MIME module's commands, it needs to create a
942      <code>mime_dir_config</code> structure, so those commands have something
943      to act on. It does this by invoking the function it finds in the module's
944      `create per-dir config slot', with two arguments: the name of the
945      directory to which this configuration information applies (or
946      <code>NULL</code> for <code>srm.conf</code>), and a pointer to a
947      resource pool in which the allocation should happen.</p>
948
949      <p>(If we are reading a <code>.htaccess</code> file, that resource pool
950      is the per-request resource pool for the request; otherwise it is a
951      resource pool which is used for configuration data, and cleared on
952      restarts. Either way, it is important for the structure being created to
953      vanish when the pool is cleared, by registering a cleanup on the pool if
954      necessary).</p>
955
956      <p>For the MIME module, the per-dir config creation function just
957      <code>ap_palloc</code>s the structure above, and a creates a couple of
958      tables to fill it. That looks like this:</p>
959
960      <div class="example"><p><code>
961        void *create_mime_dir_config (pool *p, char *dummy)<br />
962        {<br />
963        <span class="indent">
964          mime_dir_config *new =<br />
965          <span class="indent">
966           (mime_dir_config *) ap_palloc (p, sizeof(mime_dir_config));<br />
967          </span>
968          <br />
969          new-&gt;forced_types = ap_make_table (p, 4);<br />
970          new-&gt;encoding_types = ap_make_table (p, 4);<br />
971          <br />
972          return new;<br />
973        </span>
974        }
975      </code></p></div>
976
977      <p>Now, suppose we've just read in a <code>.htaccess</code> file. We
978      already have the per-directory configuration structure for the next
979      directory up in the hierarchy. If the <code>.htaccess</code> file we just
980      read in didn't have any <code class="directive"><a href="../mod/mod_mime.html#addtype">AddType</a></code>
981      or <code class="directive"><a href="../mod/mod_mime.html#addencoding">AddEncoding</a></code> commands, its
982      per-directory config structure for the MIME module is still valid, and we
983      can just use it. Otherwise, we need to merge the two structures
984      somehow.</p>
985
986      <p>To do that, the server invokes the module's per-directory config merge
987      function, if one is present. That function takes three arguments: the two
988      structures being merged, and a resource pool in which to allocate the
989      result. For the MIME module, all that needs to be done is overlay the
990      tables from the new per-directory config structure with those from the
991      parent:</p>
992
993      <div class="example"><p><code>
994        void *merge_mime_dir_configs (pool *p, void *parent_dirv, void *subdirv)<br />
995        {<br />
996        <span class="indent">
997          mime_dir_config *parent_dir = (mime_dir_config *)parent_dirv;<br />
998          mime_dir_config *subdir = (mime_dir_config *)subdirv;<br />
999          mime_dir_config *new =<br />
1000          <span class="indent">
1001            (mime_dir_config *)ap_palloc (p, sizeof(mime_dir_config));<br />
1002          </span>
1003          <br />
1004          new-&gt;forced_types = ap_overlay_tables (p, subdir-&gt;forced_types,<br />
1005          <span class="indent">
1006            parent_dir-&gt;forced_types);<br />
1007          </span>
1008          new-&gt;encoding_types = ap_overlay_tables (p, subdir-&gt;encoding_types,<br />
1009          <span class="indent">
1010            parent_dir-&gt;encoding_types);<br />
1011          </span>
1012          <br />
1013          return new;<br />
1014        </span>
1015        }
1016      </code></p></div>
1017
1018      <p>As a note -- if there is no per-directory merge function present, the
1019      server will just use the subdirectory's configuration info, and ignore
1020      the parent's. For some modules, that works just fine (<em>e.g.</em>, for
1021      the includes module, whose per-directory configuration information
1022      consists solely of the state of the <code>XBITHACK</code>), and for those
1023      modules, you can just not declare one, and leave the corresponding
1024      structure slot in the module itself <code>NULL</code>.</p>
1025    
1026
1027    <h3><a name="commands" id="commands">Command handling</a></h3>
1028      <p>Now that we have these structures, we need to be able to figure out how
1029      to fill them. That involves processing the actual <code class="directive"><a href="../mod/mod_mime.html#addtype">AddType</a></code> and <code class="directive"><a href="../mod/mod_mime.html#addencoding">AddEncoding</a></code> commands. To find commands, the server looks in
1030      the module's command table. That table contains information on how many
1031      arguments the commands take, and in what formats, where it is permitted,
1032      and so forth. That information is sufficient to allow the server to invoke
1033      most command-handling functions with pre-parsed arguments. Without further
1034      ado, let's look at the <code class="directive"><a href="../mod/mod_mime.html#addtype">AddType</a></code>
1035      command handler, which looks like this (the <code class="directive"><a href="../mod/mod_mime.html#addencoding">AddEncoding</a></code> command looks basically the same, and won't be
1036      shown here):</p>
1037
1038      <div class="example"><p><code>
1039        char *add_type(cmd_parms *cmd, mime_dir_config *m, char *ct, char *ext)<br />
1040        {<br />
1041        <span class="indent">
1042          if (*ext == '.') ++ext;<br />
1043          ap_table_set (m-&gt;forced_types, ext, ct);<br />
1044          return NULL;<br />
1045        </span>
1046        }
1047      </code></p></div>
1048
1049      <p>This command handler is unusually simple. As you can see, it takes
1050      four arguments, two of which are pre-parsed arguments, the third being the
1051      per-directory configuration structure for the module in question, and the
1052      fourth being a pointer to a <code>cmd_parms</code> structure. That
1053      structure contains a bunch of arguments which are frequently of use to
1054      some, but not all, commands, including a resource pool (from which memory
1055      can be allocated, and to which cleanups should be tied), and the (virtual)
1056      server being configured, from which the module's per-server configuration
1057      data can be obtained if required.</p>
1058
1059      <p>Another way in which this particular command handler is unusually
1060      simple is that there are no error conditions which it can encounter. If
1061      there were, it could return an error message instead of <code>NULL</code>;
1062      this causes an error to be printed out on the server's
1063      <code>stderr</code>, followed by a quick exit, if it is in the main config
1064      files; for a <code>.htaccess</code> file, the syntax error is logged in
1065      the server error log (along with an indication of where it came from), and
1066      the request is bounced with a server error response (HTTP error status,
1067      code 500).</p>
1068
1069      <p>The MIME module's command table has entries for these commands, which
1070      look like this:</p>
1071
1072      <div class="example"><p><code>
1073        command_rec mime_cmds[] = {<br />
1074        <span class="indent">
1075          { "AddType", add_type, NULL, OR_FILEINFO, TAKE2,<br />
1076          <span class="indent">"a mime type followed by a file extension" },<br /></span>
1077          { "AddEncoding", add_encoding, NULL, OR_FILEINFO, TAKE2,<br />
1078          <span class="indent">
1079          "an encoding (<em>e.g.</em>, gzip), followed by a file extension" },<br />
1080          </span>
1081          { NULL }<br />
1082        </span>
1083        };
1084      </code></p></div>
1085
1086      <p>The entries in these tables are:</p>
1087      <ul>
1088      <li>The name of the command</li>
1089      <li>The function which handles it</li>
1090      <li>a <code>(void *)</code> pointer, which is passed in the
1091      <code>cmd_parms</code> structure to the command handler ---
1092      this is useful in case many similar commands are handled by
1093      the same function.</li>
1094
1095      <li>A bit mask indicating where the command may appear. There
1096      are mask bits corresponding to each
1097      <code>AllowOverride</code> option, and an additional mask
1098      bit, <code>RSRC_CONF</code>, indicating that the command may
1099      appear in the server's own config files, but <em>not</em> in
1100      any <code>.htaccess</code> file.</li>
1101
1102      <li>A flag indicating how many arguments the command handler
1103      wants pre-parsed, and how they should be passed in.
1104      <code>TAKE2</code> indicates two pre-parsed arguments. Other
1105      options are <code>TAKE1</code>, which indicates one
1106      pre-parsed argument, <code>FLAG</code>, which indicates that
1107      the argument should be <code>On</code> or <code>Off</code>,
1108      and is passed in as a boolean flag, <code>RAW_ARGS</code>,
1109      which causes the server to give the command the raw, unparsed
1110      arguments (everything but the command name itself). There is
1111      also <code>ITERATE</code>, which means that the handler looks
1112      the same as <code>TAKE1</code>, but that if multiple
1113      arguments are present, it should be called multiple times,
1114      and finally <code>ITERATE2</code>, which indicates that the
1115      command handler looks like a <code>TAKE2</code>, but if more
1116      arguments are present, then it should be called multiple
1117      times, holding the first argument constant.</li>
1118
1119      <li>Finally, we have a string which describes the arguments
1120      that should be present. If the arguments in the actual config
1121      file are not as required, this string will be used to help
1122      give a more specific error message. (You can safely leave
1123      this <code>NULL</code>).</li>
1124      </ul>
1125
1126      <p>Finally, having set this all up, we have to use it. This is ultimately
1127      done in the module's handlers, specifically for its file-typing handler,
1128      which looks more or less like this; note that the per-directory
1129      configuration structure is extracted from the <code>request_rec</code>'s
1130      per-directory configuration vector by using the
1131      <code>ap_get_module_config</code> function.</p>
1132
1133      <div class="example"><p><code>
1134        int find_ct(request_rec *r)<br />
1135        {<br />
1136        <span class="indent">
1137          int i;<br />
1138          char *fn = ap_pstrdup (r-&gt;pool, r-&gt;filename);<br />
1139          mime_dir_config *conf = (mime_dir_config *)<br />
1140          <span class="indent">
1141            ap_get_module_config(r-&gt;per_dir_config, &amp;mime_module);<br />
1142          </span>
1143          char *type;<br />
1144          <br />
1145          if (S_ISDIR(r-&gt;finfo.st_mode)) {<br />
1146          <span class="indent">
1147            r-&gt;content_type = DIR_MAGIC_TYPE;<br />
1148            return OK;<br />
1149          </span>
1150          }<br />
1151          <br />
1152          if((i=ap_rind(fn,'.')) &lt; 0) return DECLINED;<br />
1153          ++i;<br />
1154          <br />
1155          if ((type = ap_table_get (conf-&gt;encoding_types, &amp;fn[i])))<br />
1156          {<br />
1157          <span class="indent">
1158            r-&gt;content_encoding = type;<br />
1159            <br />
1160            /* go back to previous extension to try to use it as a type */<br />
1161            fn[i-1] = '\0';<br />
1162            if((i=ap_rind(fn,'.')) &lt; 0) return OK;<br />
1163            ++i;<br />
1164          </span>
1165          }<br />
1166          <br />
1167          if ((type = ap_table_get (conf-&gt;forced_types, &amp;fn[i])))<br />
1168          {<br />
1169          <span class="indent">
1170            r-&gt;content_type = type;<br />
1171          </span>
1172          }<br />
1173          <br />
1174          return OK;
1175        </span>
1176        }
1177      </code></p></div>

1180    <h3><a name="servconf" id="servconf">Side notes -- per-server configuration,
1181    virtual servers, <em>etc</em>.</a></h3>
1182      <p>The basic ideas behind per-server module configuration are basically
1183      the same as those for per-directory configuration; there is a creation
1184      function and a merge function, the latter being invoked where a virtual
1185      server has partially overridden the base server configuration, and a
1186      combined structure must be computed. (As with per-directory configuration,
1187      the default if no merge function is specified, and a module is configured
1188      in some virtual server, is that the base configuration is simply
1189      ignored).</p>
1190
1191      <p>The only substantial difference is that when a command needs to
1192      configure the per-server private module data, it needs to go to the
1193      <code>cmd_parms</code> data to get at it. Here's an example, from the
1194      alias module, which also indicates how a syntax error can be returned
1195      (note that the per-directory configuration argument to the command
1196      handler is declared as a dummy, since the module doesn't actually have
1197      per-directory config data):</p>
1198
1199      <div class="example"><p><code>
1200        char *add_redirect(cmd_parms *cmd, void *dummy, char *f, char *url)<br />
1201        {<br />
1202        <span class="indent">
1203          server_rec *s = cmd-&gt;server;<br />
1204          alias_server_conf *conf = (alias_server_conf *)<br />
1205          <span class="indent">
1206            ap_get_module_config(s-&gt;module_config,&amp;alias_module);<br />
1207          </span>
1208          alias_entry *new = ap_push_array (conf-&gt;redirects);<br />
1209          <br />
1210          if (!ap_is_url (url)) return "Redirect to non-URL";<br />
1211          <br />
1212          new-&gt;fake = f; new-&gt;real = url;<br />
1213          return NULL;<br />
1214        </span>
1215        }
1216      </code></p></div>
