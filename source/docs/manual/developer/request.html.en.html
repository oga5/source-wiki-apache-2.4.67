29    <div class="warning"><h3>Warning</h3>
30      <p>Warning - this is a first (fast) draft that needs further
31      revision!</p>
32    </div>
33
34    <p>Several changes in 2.0 and above affect the internal request
35    processing mechanics. Module authors need to be aware of these
36    changes so they may take advantage of the optimizations and
37    security enhancements.</p>
38
39    <p>The first major change is to the subrequest and redirect
40    mechanisms. There were a number of different code paths in
41    the Apache HTTP Server 1.3 to attempt to optimize subrequest 
42    or redirect behavior. As patches were introduced to 2.0, these
43    optimizations (and the server behavior) were quickly broken due
44    to this duplication of code. All duplicate code has been folded
45    back into <code>ap_process_request_internal()</code> to prevent
46    the code from falling out of sync again.</p>
47
48    <p>This means that much of the existing code was 'unoptimized'.
49    It is the Apache HTTP Project's first goal to create a robust
50    and correct implementation of the HTTP server RFC. Additional
51    goals include security, scalability and optimization. New
52    methods were sought to optimize the server (beyond the
53    performance of 1.3) without introducing fragile or
54    insecure code.</p>

64<h2><a name="processing" id="processing">The Request Processing Cycle</a></h2>
65    <p>All requests pass through <code>ap_process_request_internal()</code>
66    in <code>server/request.c</code>, including subrequests and redirects. If a module
67    doesn't pass generated requests through this code, the author is cautioned
68    that the module may be broken by future changes to request
69    processing.</p>
70
71    <p>To streamline requests, the module author can take advantage
72    of the <a href="./modguide.html#hooking">hooks offered</a> to drop 
73    out of the request cycle early, or to bypass core hooks which are 
74    irrelevant (and costly in terms of CPU.)</p>
75</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>

77<h2><a name="parsing" id="parsing">The Request Parsing Phase</a></h2>
78    <h3><a name="unescape" id="unescape">Unescapes the URL</a></h3>
79      <p>The request's <code>parsed_uri</code> path is unescaped, once and only
80      once, at the beginning of internal request processing.</p>
81
82      <p>This step is bypassed if the proxyreq flag is set, or the
83      <code>parsed_uri.path</code> element is unset. The module has no further
84      control of this one-time unescape operation, either failing to
85      unescape or multiply unescaping the URL leads to security
86      repercussions.</p>
87    
88
89    <h3><a name="strip" id="strip">Strips Parent and This Elements from the
90    URI</a></h3>
91      <p>All <code>/../</code> and <code>/./</code> elements are
92      removed by <code>ap_getparents()</code>, as well as any trailing 
93      <code>/.</code> or <code>/..</code> element. This helps to ensure
94      the path is (nearly) absolute before the request processing
95      continues. (See RFC 1808 section 4 for further discussion.)</p>
96
97      <p>This step cannot be bypassed.</p>
98    
99
100    <h3><a name="inital-location-walk" id="inital-location-walk">Initial URI Location Walk</a></h3>
101      <p>Every request is subject to an
102      <code>ap_location_walk()</code> call. This ensures that
103      <code class="directive"><a href="../mod/core.html#location">&lt;Location&gt;</a></code> sections
104      are consistently enforced for all requests. If the request is an internal
105      redirect or a sub-request, it may borrow some or all of the processing
106      from the previous or parent request's ap_location_walk, so this step
107      is generally very efficient after processing the main request.</p>
108    
109
110    <h3><a name="translate_name" id="translate_name">translate_name</a></h3>
111      <p>Modules can determine the file name, or alter the given URI
112      in this step. For example, <code class="module"><a href="../mod/mod_vhost_alias.html">mod_vhost_alias</a></code> will
113      translate the URI's path into the configured virtual host,
114      <code class="module"><a href="../mod/mod_alias.html">mod_alias</a></code> will translate the path to an alias path,
115      and if the request falls back on the core, the <code class="directive"><a href="../mod/core.html#documentroot">DocumentRoot</a></code> is prepended to the request resource.</p>
116
117      <p>If all modules <code>DECLINE</code> this phase, an error 500 is
118      returned to the browser, and a "couldn't translate name" error is logged
119      automatically.</p>
120    
121
122    <h3><a name="map_to_storage" id="map_to_storage">Hook: map_to_storage</a></h3>
123      <p>After the file or correct URI was determined, the
124      appropriate per-dir configurations are merged together. For
125      example, <code class="module"><a href="../mod/mod_proxy.html">mod_proxy</a></code> compares and merges the appropriate
126      <code class="directive"><a href="../mod/mod_proxy.html#proxy">&lt;Proxy&gt;</a></code> sections.
127      If the URI is nothing more than a local (non-proxy) <code>TRACE</code>
128      request, the core handles the request and returns <code>DONE</code>.
129      If no module answers this hook with <code>OK</code> or <code>DONE</code>,
130      the core will run the request filename against the <code class="directive"><a href="../mod/core.html#directory">&lt;Directory&gt;</a></code> and <code class="directive"><a href="../mod/core.html#files">&lt;Files&gt;</a></code> sections. If the request
131      'filename' isn't an absolute, legal filename, a note is set for
132      later termination.</p>
133    
134
135    <h3><a name="location-walk" id="location-walk">URI Location Walk</a></h3>
136      <p>Every request is hardened by a second
137      <code>ap_location_walk()</code> call. This reassures that a
138      translated request is still subjected to the configured
139      <code class="directive"><a href="../mod/core.html#location">&lt;Location&gt;</a></code> sections.
140      The request again borrows some or all of the processing from its previous
141      <code>location_walk</code> above, so this step is almost always very
142      efficient unless the translated URI mapped to a substantially different
143      path or Virtual Host.</p>
144    
145
146    <h3><a name="header_parser" id="header_parser">Hook: header_parser</a></h3>
147      <p>The main request then parses the client's headers. This
148      prepares the remaining request processing steps to better serve
149      the client's request.</p>

170<h2><a name="preparation" id="preparation">The Preparation Phase</a></h2>
171    <h3><a name="type_checker" id="type_checker">Hook: type_checker</a></h3>
172      <p>The modules have an opportunity to test the URI or filename
173      against the target resource, and set mime information for the
174      request. Both <code class="module"><a href="../mod/mod_mime.html">mod_mime</a></code> and
175      <code class="module"><a href="../mod/mod_mime_magic.html">mod_mime_magic</a></code> use this phase to compare the file
176      name or contents against the administrator's configuration and set the
177      content type, language, character set and request handler. Some modules
178      may set up their filters or other request handling parameters at this
179      time.</p>
180
181      <p>If all modules <code>DECLINE</code> this phase, an error 500 is
182      returned to the browser, and a "couldn't find types" error is logged
183      automatically.</p>
184    
185
186    <h3><a name="fixups" id="fixups">Hook: fixups</a></h3>
187      <p>Many modules are 'trounced' by some phase above. The fixups
188      phase is used by modules to 'reassert' their ownership or force
189      the request's fields to their appropriate values. It isn't
190      always the cleanest mechanism, but occasionally it's the only
191      option.</p>
192

194<div class="section">
195<h2><a name="handler" id="handler">The Handler Phase</a></h2>
196    <p>This phase is <strong>not</strong> part of the processing in
197    <code>ap_process_request_internal()</code>. Many
198    modules prepare one or more subrequests prior to creating any
199    content at all. After the core, or a module calls
200    <code>ap_process_request_internal()</code> it then calls
201    <code>ap_invoke_handler()</code> to generate the request.</p>
202
203    <h3><a name="insert_filter" id="insert_filter">Hook: insert_filter</a></h3>
204      <p>Modules that transform the content in some way can insert
205      their values and override existing filters, such that if the
206      user configured a more advanced filter out-of-order, then the
207      module can move its order as need be.  There is no result code,
208      so actions in this hook better be trusted to always succeed.</p>
209    
210
211    <h3><a name="hook_handler" id="hook_handler">Hook: handler</a></h3>
212      <p>The module finally has a chance to serve the request in its
213      handler hook. Note that not every prepared request is sent to
214      the handler hook. Many modules, such as <code class="module"><a href="../mod/mod_autoindex.html">mod_autoindex</a></code>,
215      will create subrequests for a given URI, and then never serve the
216      subrequest, but simply lists it for the user. Remember not to
217      put required teardown from the hooks above into this module,
218      but register pool cleanups against the request pool to free
219      resources as required.</p>
