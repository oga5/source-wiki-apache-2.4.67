29    <p>There are a number of common pitfalls encountered when writing
30    output filters; this page aims to document best practice for
31    authors of new or existing filters.</p>
32
33    <p>This document is applicable to both version 2.0 and version 2.2
34    of the Apache HTTP Server; it specifically targets
35    <code>RESOURCE</code>-level or <code>CONTENT_SET</code>-level
36    filters though some advice is generic to all types of filter.</p>

51<h2><a name="basics" id="basics">Filters and bucket brigades</a></h2>
52    
53
54    <p>Each time a filter is invoked, it is passed a <em>bucket
55    brigade</em>, containing a sequence of <em>buckets</em> which
56    represent both data content and metadata.  Every bucket has a
57    <em>bucket type</em>; a number of bucket types are defined and
58    used by the <code>httpd</code> core modules (and the
59    <code>apr-util</code> library which provides the bucket brigade
60    interface), but modules are free to define their own types.</p>
61
62    <div class="note">Output filters must be prepared to process
63    buckets of non-standard types; with a few exceptions, a filter
64    need not care about the types of buckets being filtered.</div>
65
66    <p>A filter can tell whether a bucket represents either data or
67    metadata using the <code>APR_BUCKET_IS_METADATA</code> macro.
68    Generally, all metadata buckets should be passed down the filter
69    chain by an output filter.  Filters may transform, delete, and
70    insert data buckets as appropriate.</p>
71
72    <p>There are two metadata bucket types which all filters must pay
73    attention to: the <code>EOS</code> bucket type, and the
74    <code>FLUSH</code> bucket type.  An <code>EOS</code> bucket
75    indicates that the end of the response has been reached and no
76    further buckets need be processed.  A <code>FLUSH</code> bucket
77    indicates that the filter should flush any buffered buckets (if
78    applicable) down the filter chain immediately.</p>
79
80    <div class="note"><code>FLUSH</code> buckets are sent when the
81    content generator (or an upstream filter) knows that there may be
82    a delay before more content can be sent.  By passing
83    <code>FLUSH</code> buckets down the filter chain immediately,
84    filters ensure that the client is not kept waiting for pending
85    data longer than necessary.</div>
86
87    <p>Filters can create <code>FLUSH</code> buckets and pass these
88    down the filter chain if desired.  Generating <code>FLUSH</code>
89    buckets unnecessarily, or too frequently, can harm network
90    utilisation since it may force large numbers of small packets to
91    be sent, rather than a small number of larger packets.  The
92    section on <a href="#nonblock">Non-blocking bucket reads</a>
93    covers a case where filters are encouraged to generate
94    <code>FLUSH</code> buckets.</p>
95
96    <div class="example"><h3>Example bucket brigade</h3><p><code>
97    HEAP FLUSH FILE EOS</code></p></div>
98
99    <p>This shows a bucket brigade which may be passed to a filter; it
100    contains two metadata buckets (<code>FLUSH</code> and
101    <code>EOS</code>), and two data buckets (<code>HEAP</code> and
102    <code>FILE</code>).</p>
103

106<h2><a name="invocation" id="invocation">Filter invocation</a></h2>
107    
108
109    <p>For any given request, an output filter might be invoked only
110    once and be given a single brigade representing the entire response.
111    It is also possible that the number of times a filter is invoked
112    for a single response is proportional to the size of the content
113    being filtered, with the filter being passed a brigade containing
114    a single bucket each time.  Filters must operate correctly in
115    either case.</p>
116
117    <div class="warning">An output filter which allocates long-lived
118    memory every time it is invoked may consume memory proportional to
119    response size.  Output filters which need to allocate memory
120    should do so once per response; see <a href="#state">Maintaining
121    state</a> below.</div>
122
123    <p>An output filter can distinguish the final invocation for a
124    given response by the presence of an <code>EOS</code> bucket in
125    the brigade.  Any buckets in the brigade after an EOS should be
126    ignored.</p>
127
128    <p>An output filter should never pass an empty brigade down the
129    filter chain.  To be defensive, filters should be prepared to
130    accept an empty brigade, and should return success without passing
131    this brigade on down the filter chain.  The handling of an empty
132    brigade should have no side effects (such as changing any state
133    private to the filter).</p>
134
135    <div class="example"><h3>How to handle an empty brigade</h3><pre class="prettyprint lang-c">apr_status_t dummy_filter(ap_filter_t *f, apr_bucket_brigade *bb)
136{
137    if (APR_BRIGADE_EMPTY(bb)) {
138        return APR_SUCCESS;
139    }
140    ...</pre>
141</div>

145<h2><a name="brigade" id="brigade">Brigade structure</a></h2>
146    
147
148    <p>A bucket brigade is a doubly-linked list of buckets.  The list
149    is terminated (at both ends) by a <em>sentinel</em> which can be
150    distinguished from a normal bucket by comparing it with the
151    pointer returned by <code>APR_BRIGADE_SENTINEL</code>.  The list
152    sentinel is in fact not a valid bucket structure; any attempt to
153    call normal bucket functions (such as
154    <code>apr_bucket_read</code>) on the sentinel will have undefined
155    behavior (i.e. will crash the process).</p>
156
157    <p>There are a variety of functions and macros for traversing and
158    manipulating bucket brigades; see the <a href="http://apr.apache.org/docs/apr-util/trunk/group___a_p_r___util___bucket___brigades.html">apr_buckets.h</a>
159    header for complete coverage.  Commonly used macros include:</p>
160
161    <dl>
162      <dt><code>APR_BRIGADE_FIRST(bb)</code></dt>
163      <dd>returns the first bucket in brigade bb</dd>
164
165      <dt><code>APR_BRIGADE_LAST(bb)</code></dt>
166      <dd>returns the last bucket in brigade bb</dd>
167
168      <dt><code>APR_BUCKET_NEXT(e)</code></dt>
169      <dd>gives the next bucket after bucket e</dd>
170
171      <dt><code>APR_BUCKET_PREV(e)</code></dt>
172      <dd>gives the bucket before bucket e</dd>
173
174    </dl>
175
176    <p>The <code>apr_bucket_brigade</code> structure itself is
177    allocated out of a pool, so if a filter creates a new brigade, it
178    must ensure that memory use is correctly bounded.  A filter which
179    allocates a new brigade out of the request pool
180    (<code>r-&gt;pool</code>) on every invocation, for example, will fall
181    foul of the <a href="#invocation">warning above</a> concerning
182    memory use.  Such a filter should instead create a brigade on the
183    first invocation per request, and store that brigade in its <a href="#state">state structure</a>.</p>
184
185    <div class="warning"><p>It is generally never advisable to use
186    <code>apr_brigade_destroy</code> to "destroy" a brigade unless
187    you know for certain that the brigade will never be used
188    again, even then, it should be used rarely.  The
189    memory used by the brigade structure will not be released by
190    calling this function (since it comes from a pool), but the
191    associated pool cleanup is unregistered.  Using
192    <code>apr_brigade_destroy</code> can in fact cause memory leaks;
193    if a "destroyed" brigade contains buckets when its
194    containing pool is destroyed, those buckets will <em>not</em> be
195    immediately destroyed.</p>
196
197    <p>In general, filters should use <code>apr_brigade_cleanup</code>
198    in preference to <code>apr_brigade_destroy</code>.</p></div>

202<h2><a name="buckets" id="buckets">Processing buckets</a></h2>
203
204    
205
206    <p>When dealing with non-metadata buckets, it is important to
207    understand that the "<code>apr_bucket *</code>" object is an
208    abstract <em>representation</em> of data:</p>
209
210    <ol>
211      <li>The amount of data represented by the bucket may or may not
212      have a determinate length; for a bucket which represents data of
213      indeterminate length, the <code>-&gt;length</code> field is set to
214      the value <code>(apr_size_t)-1</code>.  For example, buckets of
215      the <code>PIPE</code> bucket type have an indeterminate length;
216      they represent the output from a pipe.</li>
217
218      <li>The data represented by a bucket may or may not be mapped
219      into memory.  The <code>FILE</code> bucket type, for example,
220      represents data stored in a file on disk.</li>
221    </ol>
222
223    <p>Filters read the data from a bucket using the
224    <code>apr_bucket_read</code> function.  When this function is
225    invoked, the bucket may <em>morph</em> into a different bucket
226    type, and may also insert a new bucket into the bucket brigade.
227    This must happen for buckets which represent data not mapped into
228    memory.</p>
229
230    <p>To give an example; consider a bucket brigade containing a
231    single <code>FILE</code> bucket representing an entire file, 24
232    kilobytes in size:</p>
233
234    <div class="example"><p><code>FILE(0K-24K)</code></p></div>
235
236    <p>When this bucket is read, it will read a block of data from the
237    file, morph into a <code>HEAP</code> bucket to represent that
238    data, and return the data to the caller.  It also inserts a new
239    <code>FILE</code> bucket representing the remainder of the file;
240    after the <code>apr_bucket_read</code> call, the brigade looks
241    like:</p>
242
243    <div class="example"><p><code>HEAP(8K) FILE(8K-24K)</code></p></div>
244

247<h2><a name="filtering" id="filtering">Filtering brigades</a></h2>
248    
249
250    <p>The basic function of any output filter will be to iterate
251    through the passed-in brigade and transform (or simply examine)
252    the content in some manner.  The implementation of the iteration
253    loop is critical to producing a well-behaved output filter.</p>
254
255    <p>Taking an example which loops through the entire brigade as
256    follows:</p>
257
258    <div class="example"><h3>Bad output filter -- do not imitate!</h3><pre class="prettyprint lang-c">apr_bucket *e = APR_BRIGADE_FIRST(bb);
259const char *data;
260apr_size_t length;
261
262while (e != APR_BRIGADE_SENTINEL(bb)) {
263    apr_bucket_read(e, &amp;data, &amp;length, APR_BLOCK_READ);
264    e = APR_BUCKET_NEXT(e);
265}
266
267return ap_pass_brigade(bb);</pre>
268</div>
269
270    <p>The above implementation would consume memory proportional to
271    content size.  If passed a <code>FILE</code> bucket, for example,
272    the entire file contents would be read into memory as each
273    <code>apr_bucket_read</code> call morphed a <code>FILE</code>
274    bucket into a <code>HEAP</code> bucket.</p>
275
276    <p>In contrast, the implementation below will consume a fixed
277    amount of memory to filter any brigade; a temporary brigade is
278    needed and must be allocated only once per response, see the <a href="#state">Maintaining state</a> section.</p>
279
280    <div class="example"><h3>Better output filter</h3><pre class="prettyprint lang-c">apr_bucket *e;
281const char *data;
282apr_size_t length;
283
284while ((e = APR_BRIGADE_FIRST(bb)) != APR_BRIGADE_SENTINEL(bb)) {
285    rv = apr_bucket_read(e, &amp;data, &amp;length, APR_BLOCK_READ);
286    if (rv) ...;
287    /* Remove bucket e from bb. */
288    APR_BUCKET_REMOVE(e);
289    /* Insert it into  temporary brigade. */
290    APR_BRIGADE_INSERT_HEAD(tmpbb, e);
291    /* Pass brigade downstream. */
292    rv = ap_pass_brigade(f-&gt;next, tmpbb);
293    if (rv) ...;
294    apr_brigade_cleanup(tmpbb);
295}</pre>
296</div>

300<h2><a name="state" id="state">Maintaining state</a></h2>
301
302    
303
304    <p>A filter which needs to maintain state over multiple
305    invocations per response can use the <code>-&gt;ctx</code> field of
306    its <code>ap_filter_t</code> structure.  It is typical to store a
307    temporary brigade in such a structure, to avoid having to allocate
308    a new brigade per invocation as described in the <a href="#brigade">Brigade structure</a> section.</p>
309
310    <div class="example"><h3>Example code to maintain filter state</h3><pre class="prettyprint lang-c">struct dummy_state {
311    apr_bucket_brigade *tmpbb;
312    int filter_state;
313    ...
314};
315
316apr_status_t dummy_filter(ap_filter_t *f, apr_bucket_brigade *bb)
317{
318    struct dummy_state *state;
319
320    state = f-&gt;ctx;
321    if (state == NULL) {
322
323        /* First invocation for this response: initialize state structure.
324         */
325        f-&gt;ctx = state = apr_palloc(f-&gt;r-&gt;pool, sizeof *state);
326
327        state-&gt;tmpbb = apr_brigade_create(f-&gt;r-&gt;pool, f-&gt;c-&gt;bucket_alloc);
328        state-&gt;filter_state = ...;
329    }
330    ...</pre>
331</div>

335<h2><a name="buffer" id="buffer">Buffering buckets</a></h2>
336    
337
338    <p>If a filter decides to store buckets beyond the duration of a
339    single filter function invocation (for example storing them in its
340    <code>-&gt;ctx</code> state structure), those buckets must be <em>set
341    aside</em>.  This is necessary because some bucket types provide
342    buckets which represent temporary resources (such as stack memory)
343    which will fall out of scope as soon as the filter chain completes
344    processing the brigade.</p>
345
346    <p>To setaside a bucket, the <code>apr_bucket_setaside</code>
347    function can be called.  Not all bucket types can be setaside, but
348    if successful, the bucket will have morphed to ensure it has a
349    lifetime at least as long as the pool given as an argument to the
350    <code>apr_bucket_setaside</code> function.</p>
351
352    <p>Alternatively, the <code>ap_save_brigade</code> function can be
353    used, which will move all the buckets into a separate brigade
354    containing buckets with a lifetime as long as the given pool
355    argument.  This function must be used with care, taking into
356    account the following points:</p>
357
358    <ol>
359      <li>On return, <code>ap_save_brigade</code> guarantees that all
360      the buckets in the returned brigade will represent data mapped
361      into memory.  If given an input brigade containing, for example,
362      a <code>PIPE</code> bucket, <code>ap_save_brigade</code> will
363      consume an arbitrary amount of memory to store the entire output
364      of the pipe.</li>
365
366      <li>When <code>ap_save_brigade</code> reads from buckets which
367      cannot be setaside, it will always perform blocking reads,
368      removing the opportunity to use <a href="#nonblock">Non-blocking
369      bucket reads</a>.</li>
370
371      <li>If <code>ap_save_brigade</code> is used without passing a
372      non-NULL "<code>saveto</code>" (destination) brigade parameter,
373      the function will create a new brigade, which may cause memory
374      use to be proportional to content size as described in the <a href="#brigade">Brigade structure</a> section.</li>
375    </ol>
376
377    <div class="warning">Filters must ensure that any buffered data is
378    processed and passed down the filter chain during the last
379    invocation for a given response (a brigade containing an EOS
380    bucket).  Otherwise such data will be lost.</div>
381

384<h2><a name="nonblock" id="nonblock">Non-blocking bucket reads</a></h2>
385    
386
387    <p>The <code>apr_bucket_read</code> function takes an
388    <code>apr_read_type_e</code> argument which determines whether a
389    <em>blocking</em> or <em>non-blocking</em> read will be performed
390    from the data source.  A good filter will first attempt to read
391    from every data bucket using a non-blocking read; if that fails
392    with <code>APR_EAGAIN</code>, then send a <code>FLUSH</code>
393    bucket down the filter chain, and retry using a blocking read.</p>
394
395    <p>This mode of operation ensures that any filters further down the
396    filter chain will flush any buffered buckets if a slow content
397    source is being used.</p>
398
399    <p>A CGI script is an example of a slow content source which is
400    implemented as a bucket type. <code class="module"><a href="../mod/mod_cgi.html">mod_cgi</a></code> will send
401    <code>PIPE</code> buckets which represent the output from a CGI
402    script; reading from such a bucket will block when waiting for the
403    CGI script to produce more output.</p>
404
405    <div class="example"><h3>Example code using non-blocking bucket reads</h3><pre class="prettyprint lang-c">apr_bucket *e;
406apr_read_type_e mode = APR_NONBLOCK_READ;
407
408while ((e = APR_BRIGADE_FIRST(bb)) != APR_BRIGADE_SENTINEL(bb)) {
409    apr_status_t rv;
410
411    rv = apr_bucket_read(e, &amp;data, &amp;length, mode);
412    if (rv == APR_EAGAIN &amp;&amp; mode == APR_NONBLOCK_READ) {
413
414        /* Pass down a brigade containing a flush bucket: */
415        APR_BRIGADE_INSERT_TAIL(tmpbb, apr_bucket_flush_create(...));
416        rv = ap_pass_brigade(f-&gt;next, tmpbb);
417        apr_brigade_cleanup(tmpbb);
418        if (rv != APR_SUCCESS) return rv;
419
420        /* Retry, using a blocking read. */
421        mode = APR_BLOCK_READ;
422        continue;
423    }
424    else if (rv != APR_SUCCESS) {
425        /* handle errors */
426    }
427
428    /* Next time, try a non-blocking read first. */
429    mode = APR_NONBLOCK_READ;
430    ...
431}</pre>
432</div>

436<h2><a name="rules" id="rules">Ten rules for output filters</a></h2>
437    
438
439    <p>In summary, here is a set of rules for all output filters to
440    follow:</p>
441
442    <ol>
443      <li>Output filters should not pass empty brigades down the filter
444      chain, but should be tolerant of being passed empty
445      brigades.</li>
446
447      <li>Output filters must pass all metadata buckets down the filter
448      chain; <code>FLUSH</code> buckets should be respected by passing
449      any pending or buffered buckets down the filter chain.</li>
450
451      <li>Output filters should ignore any buckets following an
452      <code>EOS</code> bucket.</li>
453
454      <li>Output filters must process a fixed amount of data at a
455      time, to ensure that memory consumption is not proportional to
456      the size of the content being filtered.</li>
457
458      <li>Output filters should be agnostic with respect to bucket
459      types, and must be able to process buckets of unfamiliar
460      type.</li>
461
462      <li>After calling <code>ap_pass_brigade</code> to pass a brigade
463      down the filter chain, output filters should call
464      <code>apr_brigade_cleanup</code> to ensure the brigade is empty
465      before reusing that brigade structure; output filters should
466      never use <code>apr_brigade_destroy</code> to "destroy"
467      brigades.</li>
468
469      <li>Output filters must <em>setaside</em> any buckets which are
470      preserved beyond the duration of the filter function.</li>
471
472      <li>Output filters must not ignore the return value of
473      <code>ap_pass_brigade</code>, and must return appropriate errors
474      back up the filter chain.</li>
475
476      <li>Output filters must only create a fixed number of bucket
477      brigades for each response, rather than one per invocation.</li>
478
479      <li>Output filters should first attempt non-blocking reads from
480      each data bucket, and send a <code>FLUSH</code> bucket down the
481      filter chain if the read blocks, before retrying with a blocking
482      read.</li>
483
484    </ol>

488<h2><a name="usecase1" id="usecase1">Use case: buffering in mod_ratelimit</a></h2>
489    
490    <p>The <a href="http://svn.apache.org/r1833875">r1833875</a> change is a good
491    example to show what buffering and keeping state means in the context of an
492    output filter. In this use case, a user asked on the users' mailing list a
493    interesting question about why <code class="module"><a href="../mod/mod_ratelimit.html">mod_ratelimit</a></code> seemed not to
494    honor its setting with proxied content (either rate limiting at a different
495    speed or simply not doing it at all). Before diving deep into the solution,
496    it is better to explain on a high level how <code class="module"><a href="../mod/mod_ratelimit.html">mod_ratelimit</a></code> works.
497    The trick is really simple: take the rate limit settings and calculate a
498    chunk size of data to flush every 200ms to the client. For example, let's imagine
499    that to set <code>rate-limit 60</code> in our config, these are the high level
500    steps to find the chunk size:</p>
501    <pre class="prettyprint lang-c">/* milliseconds to wait between each flush of data */
502RATE_INTERVAL_MS = 200;
503/* rate limit speed in b/s */
504speed = 60 * 1024;
505/* final chunk size is 12228 bytes */
506chunk_size = (speed / (1000 / RATE_INTERVAL_MS));</pre>
507
508    <p>If we apply this calculation to a bucket brigade carrying 38400 bytes, it means
509    that the filter will try to do the following:</p>
510    <ol>
511        <li>Split the 38400 bytes in chunks of maximum 12228 bytes each.</li>
512        <li>Flush the first 12228 chunk of bytes and sleep 200ms.</li>
513        <li>Flush the second 12228 chunk of bytes and sleep 200ms.</li>
514        <li>Flush the third 12228 chunk of bytes and sleep 200ms.</li>
515        <li>Flush the remaining 1716 bytes.</li>
516    </ol>
517    <p>The above pseudo code works fine if the output filter handles only one brigade
518    for each response, but it might happen that it needs to be called multiple times
519    with different brigade sizes as well. The former use case is for example when
520    httpd directly serves some content, like a static file: the bucket brigade
521    abstraction takes care of handling the whole content, and rate limiting
522    works nicely. But if the same static content is served via mod_proxy_http (for
523    example a backend is serving it rather than httpd) then the content generator
524    (in this case mod_proxy_http) may use a maximum buffer size and then send data
525    as bucket brigades to the output filters chain regularly, triggering of course
526    multiple calls to <code class="module"><a href="../mod/mod_ratelimit.html">mod_ratelimit</a></code>. If the reader tries to execute the pseudo code
527    assuming multiple calls to the output filter, each one requiring to process
528    a bucket brigade of 38400 bytes, then it is easy to spot some
529    anomalies:</p>
530    <ol>
531        <li>Between the last flush of a brigade and the first one of the next,
532            there is no sleep.</li>
533        <li>Even if the sleep was forced after the last flush, then that chunk size
534            would not be the ideal size (1716 bytes instead of 12228) and the final client's speed
535            would quickly become different than what set in the httpd's config.</li>
536    </ol>
537    <p>In this case, two things might help:</p>
538    <ol>
539        <li>Use the ctx internal data structure, initialized by <code class="module"><a href="../mod/mod_ratelimit.html">mod_ratelimit</a></code>
540        for each response handling cycle, to "remember" when the last sleep was
541        performed across multiple invocations, and act accordingly.</li>
542        <li>If a bucket brigade is not splittable into a finite number of chunk_size
543        blocks, store the remaining bytes (located in the tail of the bucket brigade) 
544        in a temporary holding area (namely another bucket brigade) and then use
545        <code>ap_save_brigade</code> to set them aside.
546        These bytes will be prepended to the next bucket brigade that will be handled
547        in the subsequent invocation.</li>
548        <li>Avoid the previous logic if the bucket brigade that is currently being
549        processed contains the end of stream bucket (EOS). There is no need to sleep
550        or buffering data if the end of stream is reached.</li>
551    </ol>
552    <p>The commit linked in the beginning of the section contains also a bit of code
553    refactoring so it is not trivial to read during the first pass, but the overall
554    idea is basically what written up to now. The goal of this section is not to
555    cause a headache to the reader trying to read C code, but to put him/her into
556    the right mindset needed to use efficiently the tools offered by the httpd's
557    filter chain toolset.</p>
