30    <p>This is a first attempt at writing the lessons I learned
31    when trying to convert the <code>mod_mmap_static</code> module to Apache
32    2.0. It's by no means definitive and probably won't even be
33    correct in some ways, but it's a start.</p>

40<h2><a name="easy" id="easy">The easier changes ...</a></h2>
41
42    <h3><a name="cleanup" id="cleanup">Cleanup Routines</a></h3>
43      <p>These now need to be of type <code>apr_status_t</code> and return a
44      value of that type. Normally the return value will be
45      <code>APR_SUCCESS</code> unless there is some need to signal an error in
46      the cleanup. Be aware that even though you signal an error not all code
47      yet checks and acts upon the error.</p>
48    
49
50    <h3><a name="init" id="init">Initialisation Routines</a></h3>
51      <p>These should now be renamed to better signify where they sit
52      in the overall process. So the name gets a small change from
53      <code>mmap_init</code> to <code>mmap_post_config</code>. The arguments
54      passed have undergone a radical change and now look like</p>
55
56      <ul>
57        <li><code>apr_pool_t *p</code></li>
58        <li><code>apr_pool_t *plog</code></li>
59        <li><code>apr_pool_t *ptemp</code></li>
60        <li><code>server_rec *s</code></li>
61      </ul>
62    
63
64    <h3><a name="datatypes" id="datatypes">Data Types</a></h3>
65      <p>A lot of the data types have been moved into the <a href="http://apr.apache.org/">APR</a>. This means that some have had
66      a name change, such as the one shown above. The following is a brief
67      list of some of the changes that you are likely to have to make.</p>
68
69      <ul>
70        <li><code>pool</code> becomes <code>apr_pool_t</code></li>
71        <li><code>table</code> becomes <code>apr_table_t</code></li>
72      </ul>

76<h2><a name="messy" id="messy">The messier changes...</a></h2>
77
78    <h3><a name="register-hooks" id="register-hooks">Register Hooks</a></h3>
79      <p>The new architecture uses a series of hooks to provide for
80      calling your functions. These you'll need to add to your module
81      by way of a new function, <code>static void register_hooks(void)</code>.
82      The function is really reasonably straightforward once you
83      understand what needs to be done. Each function that needs
84      calling at some stage in the processing of a request needs to
85      be registered, handlers do not. There are a number of phases
86      where functions can be added, and for each you can specify with
87      a high degree of control the relative order that the function
88      will be called in.</p>
89
90      <p>This is the code that was added to <code>mod_mmap_static</code>:</p>
91      <div class="example"><pre>static void register_hooks(void)
92{
93    static const char * const aszPre[]={ "http_core.c",NULL };
94    ap_hook_post_config(mmap_post_config,NULL,NULL,HOOK_MIDDLE);
95    ap_hook_translate_name(mmap_static_xlat,aszPre,NULL,HOOK_LAST);
96};</pre></div>
97
98      <p>This registers 2 functions that need to be called, one in
99      the <code>post_config</code> stage (virtually every module will need this
100      one) and one for the <code>translate_name</code> phase. note that while
101      there are different function names the format of each is
102      identical. So what is the format?</p>
103
104      <div class="example"><p><code>
105        ap_hook_<var>phase_name</var>(<var>function_name</var>,
106        <var>predecessors</var>, <var>successors</var>, <var>position</var>);
107      </code></p></div>
108
109      <p>There are 3 hook positions defined...</p>
110
111      <ul>
112        <li><code>HOOK_FIRST</code></li>
113        <li><code>HOOK_MIDDLE</code></li>
114        <li><code>HOOK_LAST</code></li>
115      </ul>
116
117      <p>To define the position you use the position and then modify
118      it with the predecessors and successors. Each of the modifiers
119      can be a list of functions that should be called, either before
120      the function is run (predecessors) or after the function has
121      run (successors).</p>
122
123      <p>In the <code>mod_mmap_static</code> case I didn't care about the
124      <code>post_config</code> stage, but the <code>mmap_static_xlat</code>
125      <strong>must</strong> be called after the core module had done its name
126      translation, hence the use of the aszPre to define a modifier to the
127      position <code>HOOK_LAST</code>.</p>
128    
129
130    <h3><a name="moddef" id="moddef">Module Definition</a></h3>
131      <p>There are now a lot fewer stages to worry about when
132      creating your module definition. The old definition looked
133      like</p>
134
135      <div class="example"><pre>module MODULE_VAR_EXPORT <var>module_name</var>_module =
136{
137    STANDARD_MODULE_STUFF,
138    /* initializer */
139    /* dir config creater */
140    /* dir merger --- default is to override */
141    /* server config */
142    /* merge server config */
143    /* command handlers */
144    /* handlers */
145    /* filename translation */
146    /* check_user_id */
147    /* check auth */
148    /* check access */
149    /* type_checker */
150    /* fixups */
151    /* logger */
152    /* header parser */
153    /* child_init */
154    /* child_exit */
155    /* post read-request */
156};</pre></div>
157
158      <p>The new structure is a great deal simpler...</p>
159      <div class="example"><pre>module MODULE_VAR_EXPORT <var>module_name</var>_module =
160{
161    STANDARD20_MODULE_STUFF,
162    /* create per-directory config structures */
163    /* merge per-directory config structures  */
164    /* create per-server config structures    */
165    /* merge per-server config structures     */
166    /* command handlers */
167    /* handlers */
168    /* register hooks */
169};</pre></div>
170
171      <p>Some of these read directly across, some don't. I'll try to
172      summarize what should be done below.</p>
173
174      <p>The stages that read directly across :</p>
175
176      <dl>
177        <dt><code>/* dir config creater */</code></dt>
178        <dd><code>/* create per-directory config structures */</code></dd>
179
180        <dt><code>/* server config */</code></dt>
181        <dd><code>/* create per-server config structures */</code></dd>
182
183        <dt><code>/* dir merger */</code></dt>
184        <dd><code>/* merge per-directory config structures */</code></dd>
185
186        <dt><code>/* merge server config */</code></dt>
187        <dd><code>/* merge per-server config structures */</code></dd>
188
189        <dt><code>/* command table */</code></dt>
190        <dd><code>/* command apr_table_t */</code></dd>
191
192        <dt><code>/* handlers */</code></dt>
193        <dd><code>/* handlers */</code></dd>
194      </dl>
195
196      <p>The remainder of the old functions should be registered as
197      hooks. There are the following hook stages defined so
198      far...</p>
199
200      <dl>
201        <dt><code>ap_hook_pre_config</code></dt>
202        <dd>do any setup required prior to processing configuration
203        directives</dd>
204
205        <dt><code>ap_hook_check_config</code></dt>
206        <dd>review configuration directive interdependencies</dd>
207
208        <dt><code>ap_hook_test_config</code></dt>
209        <dd>executes only with <code>-t</code> option</dd>
210
211        <dt><code>ap_hook_open_logs</code></dt>
212        <dd>open any specified logs</dd>
213
214        <dt><code>ap_hook_post_config</code></dt>
215        <dd>this is where the old <code>_init</code> routines get
216        registered</dd>
217
218        <dt><code>ap_hook_http_method</code></dt>
219        <dd>retrieve the http method from a request. (legacy)</dd>
220
221        <dt><code>ap_hook_auth_checker</code></dt>
222        <dd>check if the resource requires authorization</dd>
223
224        <dt><code>ap_hook_access_checker</code></dt>
225        <dd>check for module-specific restrictions</dd>
226
227        <dt><code>ap_hook_check_user_id</code></dt>
228        <dd>check the user-id and password</dd>
229
230        <dt><code>ap_hook_default_port</code></dt>
231        <dd>retrieve the default port for the server</dd>
232
233        <dt><code>ap_hook_pre_connection</code></dt>
234        <dd>do any setup required just before processing, but after
235        accepting</dd>
236
237        <dt><code>ap_hook_process_connection</code></dt>
238        <dd>run the correct protocol</dd>
239
240        <dt><code>ap_hook_child_init</code></dt>
241        <dd>call as soon as the child is started</dd>
242
243        <dt><code>ap_hook_create_request</code></dt>
244        <dd>??</dd>
245
246        <dt><code>ap_hook_fixups</code></dt>
247        <dd>last chance to modify things before generating content</dd>
248
249        <dt><code>ap_hook_handler</code></dt>
250        <dd>generate the content</dd>
251
252        <dt><code>ap_hook_header_parser</code></dt>
253        <dd>lets modules look at the headers, not used by most modules, because
254        they use <code>post_read_request</code> for this</dd>
255
256        <dt><code>ap_hook_insert_filter</code></dt>
257        <dd>to insert filters into the filter chain</dd>
258
259        <dt><code>ap_hook_log_transaction</code></dt>
260        <dd>log information about the request</dd>
261
262        <dt><code>ap_hook_optional_fn_retrieve</code></dt>
263        <dd>retrieve any functions registered as optional</dd>
264
265        <dt><code>ap_hook_post_read_request</code></dt>
266        <dd>called after reading the request, before any other phase</dd>
267
268        <dt><code>ap_hook_quick_handler</code></dt>
269        <dd>called before any request processing, used by cache modules.</dd>
270
271        <dt><code>ap_hook_translate_name</code></dt>
272        <dd>translate the URI into a filename</dd>
273
274        <dt><code>ap_hook_type_checker</code></dt>
275        <dd>determine and/or set the doc type</dd>
276      </dl>
