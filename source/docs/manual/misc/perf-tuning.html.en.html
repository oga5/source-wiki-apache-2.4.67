33    <p>Apache 2.x is a general-purpose webserver, designed to
34    provide a balance of flexibility, portability, and performance.
35    Although it has not been designed specifically to set benchmark
36    records, Apache 2.x is capable of high performance in many
37    real-world situations.</p>
38
39    <p>Compared to Apache 1.3, release 2.x contains many additional
40    optimizations to increase throughput and scalability. Most of
41    these improvements are enabled by default. However, there are
42    compile-time and run-time configuration choices that can
43    significantly affect performance. This document describes the
44    options that a server administrator can configure to tune the
45    performance of an Apache 2.x installation. Some of these
46    configuration options enable the httpd to better take advantage
47    of the capabilities of the hardware and OS, while others allow
48    the administrator to trade functionality for speed.</p>

58<h2><a name="hardware" id="hardware">Hardware and Operating System Issues</a></h2>
59
60    
61
62    <p>The single biggest hardware issue affecting webserver
63    performance is RAM. A webserver should never ever have to swap,
64    as swapping increases the latency of each request beyond a point
65    that users consider "fast enough". This causes users to hit
66    stop and reload, further increasing the load. You can, and
67    should, control the <code class="directive"><a href="../mod/mpm_common.html#maxrequestworkers">MaxRequestWorkers</a></code> setting so that your server
68    does not spawn so many children that it starts swapping. The procedure
69    for doing this is simple: determine the size of your average Apache
70    process, by looking at your process list via a tool such as
71    <code>top</code>, and divide this into your total available memory,
72    leaving some room for other processes.</p>
73
74    <p>Beyond that the rest is mundane: get a fast enough CPU, a
75    fast enough network card, and fast enough disks, where "fast
76    enough" is something that needs to be determined by
77    experimentation.</p>
78
79    <p>Operating system choice is largely a matter of local
80    concerns. But some guidelines that have proven generally
81    useful are:</p>
82
83    <ul>
84      <li>
85        <p>Run the latest stable release and patch level of the
86        operating system that you choose. Many OS suppliers have
87        introduced significant performance improvements to their
88        TCP stacks and thread libraries in recent years.</p>
89      </li>
90
91      <li>
92        <p>If your OS supports a <code>sendfile(2)</code> system
93        call, make sure you install the release and/or patches
94        needed to enable it. (With Linux, for example, this means
95        using Linux 2.4 or later. For early releases of Solaris 8,
96        you may need to apply a patch.) On systems where it is
97        available, <code>sendfile</code> enables Apache 2 to deliver
98        static content faster and with lower CPU utilization.</p>
99      </li>
100    </ul>

110    <h3><a name="dns" id="dns">HostnameLookups and other DNS considerations</a></h3>
111
112      
113
114      <p>Prior to Apache 1.3, <code class="directive"><a href="../mod/core.html#hostnamelookups">HostnameLookups</a></code> defaulted to <code>On</code>.
115      This adds latency to every request because it requires a
116      DNS lookup to complete before the request is finished. In
117      Apache 1.3 this setting defaults to <code>Off</code>. If you need
118      to have addresses in your log files resolved to hostnames, use the
119      <code class="program"><a href="../programs/logresolve.html">logresolve</a></code>
120      program that comes with Apache, or one of the numerous log
121      reporting packages which are available.</p>
122
123      <p>It is recommended that you do this sort of postprocessing of
124      your log files on some machine other than the production web
125      server machine, in order that this activity not adversely affect
126      server performance.</p>
127
128      <p>If you use any <code><code class="directive"><a href="../mod/mod_access_compat.html#allow">Allow</a></code> from domain</code> or <code><code class="directive"><a href="../mod/mod_access_compat.html#deny">Deny</a></code> from domain</code>
129      directives (i.e., using a hostname, or a domain name, rather than
130      an IP address) then you will pay for
131      two DNS lookups (a reverse, followed by a forward lookup
132      to make sure that the reverse is not being spoofed). For best
133      performance, therefore, use IP addresses, rather than names, when
134      using these directives, if possible.</p>
135
136      <p>Note that it's possible to scope the directives, such as
137      within a <code>&lt;Location "/server-status"&gt;</code> section.
138      In this case the DNS lookups are only performed on requests
139      matching the criteria. Here's an example which disables lookups
140      except for <code>.html</code> and <code>.cgi</code> files:</p>
141
142      <pre class="prettyprint lang-config">HostnameLookups off
143&lt;Files ~ "\.(html|cgi)$"&gt;
144  HostnameLookups on
145&lt;/Files&gt;</pre>
146
147
148      <p>But even still, if you just need DNS names in some CGIs you
149      could consider doing the <code>gethostbyname</code> call in the
150      specific CGIs that need it.</p>

154    <h3><a name="symlinks" id="symlinks">FollowSymLinks and SymLinksIfOwnerMatch</a></h3>
155
156      
157
158      <p>Wherever in your URL-space you do not have an <code>Options
159      FollowSymLinks</code>, or you do have an <code>Options
160      SymLinksIfOwnerMatch</code>, Apache will need to issue extra
161      system calls to check up on symlinks. (One extra call per
162      filename component.) For example, if you had:</p>
163
164      <pre class="prettyprint lang-config">DocumentRoot "/www/htdocs"
165&lt;Directory "/"&gt;
166  Options SymLinksIfOwnerMatch
167&lt;/Directory&gt;</pre>
168
169
170      <p>and a request is made for the URI <code>/index.html</code>,
171      then Apache will perform <code>lstat(2)</code> on
172      <code>/www</code>, <code>/www/htdocs</code>, and
173      <code>/www/htdocs/index.html</code>. The results of these
174      <code>lstats</code> are never cached, so they will occur on
175      every single request. If you really desire the symlinks
176      security checking, you can do something like this:</p>
177
178      <pre class="prettyprint lang-config">DocumentRoot "/www/htdocs"
179&lt;Directory "/"&gt;
180  Options FollowSymLinks
181&lt;/Directory&gt;
182
183&lt;Directory "/www/htdocs"&gt;
184  Options -FollowSymLinks +SymLinksIfOwnerMatch
185&lt;/Directory&gt;</pre>
186
187
188      <p>This at least avoids the extra checks for the
189      <code class="directive"><a href="../mod/core.html#documentroot">DocumentRoot</a></code> path.
190      Note that you'll need to add similar sections if you
191      have any <code class="directive"><a href="../mod/mod_alias.html#alias">Alias</a></code> or
192      <code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">RewriteRule</a></code> paths
193      outside of your document root. For highest performance,
194      and no symlink protection, set <code>FollowSymLinks</code>
195      everywhere, and never set <code>SymLinksIfOwnerMatch</code>.</p>
196

199    <h3><a name="htaccess" id="htaccess">AllowOverride</a></h3>
200
201      
202
203      <p>Wherever in your URL-space you allow overrides (typically
204      <code>.htaccess</code> files), Apache will attempt to open
205      <code>.htaccess</code> for each filename component. For
206      example,</p>
207
208      <pre class="prettyprint lang-config">DocumentRoot "/www/htdocs"
209&lt;Directory "/"&gt;
210  AllowOverride all
211&lt;/Directory&gt;</pre>
212
213
214      <p>and a request is made for the URI <code>/index.html</code>.
215      Then Apache will attempt to open <code>/.htaccess</code>,
216      <code>/www/.htaccess</code>, and
217      <code>/www/htdocs/.htaccess</code>. The solutions are similar
218      to the previous case of <code>Options FollowSymLinks</code>.
219      For highest performance use <code>AllowOverride None</code>
220      everywhere in your filesystem.</p>

224    <h3><a name="negotiation" id="negotiation">Negotiation</a></h3>
225
226      
227
228      <p>If at all possible, avoid content negotiation if you're
229      really interested in every last ounce of performance. In
230      practice the benefits of negotiation outweigh the performance
231      penalties. There's one case where you can speed up the server.
232      Instead of using a wildcard such as:</p>
233
234      <pre class="prettyprint lang-config">DirectoryIndex index</pre>
235
236
237      <p>Use a complete list of options:</p>
238
239      <pre class="prettyprint lang-config">DirectoryIndex index.cgi index.pl index.shtml index.html</pre>
240
241
242      <p>where you list the most common choice first.</p>
243
244      <p>Also note that explicitly creating a <code>type-map</code>
245      file provides better performance than using
246      <code>MultiViews</code>, as the necessary information can be
247      determined by reading this single file, rather than having to
248      scan the directory for files.</p>
249
250    <p>If your site needs content negotiation, consider using
251    <code>type-map</code> files, rather than the <code>Options
252    MultiViews</code> directive to accomplish the negotiation. See the
253    <a href="../content-negotiation.html">Content Negotiation</a>
254    documentation for a full discussion of the methods of negotiation,
255    and instructions for creating <code>type-map</code> files.</p>

259    <h3>Memory-mapping</h3>
260
261      
262
263      <p>In situations where Apache 2.x needs to look at the contents
264      of a file being delivered--for example, when doing server-side-include
265      processing--it normally memory-maps the file if the OS supports
266      some form of <code>mmap(2)</code>.</p>
267
268      <p>On some platforms, this memory-mapping improves performance.
269      However, there are cases where memory-mapping can hurt the performance
270      or even the stability of the httpd:</p>
271
272      <ul>
273        <li>
274          <p>On some operating systems, <code>mmap</code> does not scale
275          as well as <code>read(2)</code> when the number of CPUs increases.
276          On multiprocessor Solaris servers, for example, Apache 2.x sometimes
277          delivers server-parsed files faster when <code>mmap</code> is disabled.</p>
278        </li>
279
280        <li>
281          <p>If you memory-map a file located on an NFS-mounted filesystem
282          and a process on another NFS client machine deletes or truncates
283          the file, your process may get a bus error the next time it tries
284          to access the mapped file content.</p>
285        </li>
286      </ul>
287
288      <p>For installations where either of these factors applies, you
289      should use <code>EnableMMAP off</code> to disable the memory-mapping
290      of delivered files. (Note: This directive can be overridden on
291      a per-directory basis.)</p>
292
293    
294
295    <h3>Sendfile</h3>
296
297      
298
299      <p>In situations where Apache 2.x can ignore the contents of the file
300      to be delivered -- for example, when serving static file content --
301      it normally uses the kernel sendfile support for the file if the OS
302      supports the <code>sendfile(2)</code> operation.</p>
303
304      <p>On most platforms, using sendfile improves performance by eliminating
305      separate read and send mechanics.  However, there are cases where using
306      sendfile can harm the stability of the httpd:</p>
307
308      <ul>
309        <li>
310          <p>Some platforms may have broken sendfile support that the build
311          system did not detect, especially if the binaries were built on
312          another box and moved to such a machine with broken sendfile support.</p>
313        </li>
314        <li>
315          <p>With an NFS-mounted filesystem, the kernel may be unable
316          to reliably serve the network file through its own cache.</p>
317        </li>
318      </ul>
319
320      <p>For installations where either of these factors applies, you
321      should use <code>EnableSendfile off</code> to disable sendfile
322      delivery of file contents. (Note: This directive can be overridden
323      on a per-directory basis.)</p>

327    <h3><a name="process" id="process">Process Creation</a></h3>
328
329      
330
331      <p>Prior to Apache 1.3 the <code class="directive"><a href="../mod/prefork.html#minspareservers">MinSpareServers</a></code>, <code class="directive"><a href="../mod/prefork.html#maxspareservers">MaxSpareServers</a></code>, and <code class="directive"><a href="../mod/mpm_common.html#startservers">StartServers</a></code> settings all had drastic effects on
332      benchmark results. In particular, Apache required a "ramp-up"
333      period in order to reach a number of children sufficient to serve
334      the load being applied. After the initial spawning of
335      <code class="directive"><a href="../mod/mpm_common.html#startservers">StartServers</a></code> children,
336      only one child per second would be created to satisfy the
337      <code class="directive"><a href="../mod/prefork.html#minspareservers">MinSpareServers</a></code>
338      setting. So a server being accessed by 100 simultaneous
339      clients, using the default <code class="directive"><a href="../mod/mpm_common.html#startservers">StartServers</a></code> of <code>5</code> would take on
340      the order of 95 seconds to spawn enough children to handle
341      the load. This works fine in practice on real-life servers
342      because they aren't restarted frequently. But it does really
343      poorly on benchmarks which might only run for ten minutes.</p>
344
345      <p>The one-per-second rule was implemented in an effort to
346      avoid swamping the machine with the startup of new children. If
347      the machine is busy spawning children, it can't service
348      requests. But it has such a drastic effect on the perceived
349      performance of Apache that it had to be replaced. As of Apache
350      1.3, the code will relax the one-per-second rule. It will spawn
351      one, wait a second, then spawn two, wait a second, then spawn
352      four, and it will continue exponentially until it is spawning
353      32 children per second. It will stop whenever it satisfies the
354      <code class="directive"><a href="../mod/prefork.html#minspareservers">MinSpareServers</a></code>
355      setting.</p>
356
357      <p>This appears to be responsive enough that it's almost
358      unnecessary to twiddle the <code class="directive"><a href="../mod/prefork.html#minspareservers">MinSpareServers</a></code>, <code class="directive"><a href="../mod/prefork.html#maxspareservers">MaxSpareServers</a></code> and <code class="directive"><a href="../mod/mpm_common.html#startservers">StartServers</a></code> knobs. When more than 4 children are
359      spawned per second, a message will be emitted to the
360      <code class="directive"><a href="../mod/core.html#errorlog">ErrorLog</a></code>. If you
361      see a lot of these errors, then consider tuning these settings.
362      Use the <code class="module"><a href="../mod/mod_status.html">mod_status</a></code> output as a guide.</p>
363
364    <p>Related to process creation is process death induced by the
365    <code class="directive"><a href="../mod/mpm_common.html#maxconnectionsperchild">MaxConnectionsPerChild</a></code>
366    setting. By default this is <code>0</code>,
367    which means that there is no limit to the number of connections
368    handled per child. If your configuration currently has this set
369    to some very low number, such as <code>30</code>, you may want to bump this
370    up significantly. If you are running SunOS or an old version of
371    Solaris, limit this to <code>10000</code> or so because of memory leaks.</p>
372
373    <p>When keep-alives are in use, children will be kept busy
374    doing nothing waiting for more requests on the already open
375    connection. The default <code class="directive"><a href="../mod/core.html#keepalivetimeout">KeepAliveTimeout</a></code> of <code>5</code>
376    seconds attempts to minimize this effect. The tradeoff here is
377    between network bandwidth and server resources. In no event
378    should you raise this above about <code>60</code> seconds, as <a href="http://www.hpl.hp.com/techreports/Compaq-DEC/WRL-95-4.html">
379    most of the benefits are lost</a>.</p>

388
389    <h3>Choosing an MPM</h3>
390
391      
392
393      <p>Apache 2.x supports pluggable concurrency models, called
394      <a href="../mpm.html">Multi-Processing Modules</a> (MPMs).
395      When building Apache, you must choose an MPM to use.  There
396      are platform-specific MPMs for some platforms:
397      <code class="module"><a href="../mod/mpm_netware.html">mpm_netware</a></code>,
398      <code class="module"><a href="../mod/mpmt_os2.html">mpmt_os2</a></code>, and <code class="module"><a href="../mod/mpm_winnt.html">mpm_winnt</a></code>.  For
399      general Unix-type systems, there are several MPMs from which
400      to choose.  The choice of MPM can affect the speed and scalability
401      of the httpd:</p>
402
403      <ul>
404
405        <li>The <code class="module"><a href="../mod/worker.html">worker</a></code> MPM uses multiple child
406        processes with many threads each.  Each thread handles
407        one connection at a time.  Worker generally is a good
408        choice for high-traffic servers because it has a smaller
409        memory footprint than the prefork MPM.</li>
410        
411        <li>The <code class="module"><a href="../mod/event.html">event</a></code> MPM is threaded like the 
412        Worker MPM, but is designed to allow more requests to be 
413        served simultaneously by passing off some processing work 
414        to supporting threads, freeing up the main threads to work
415        on new requests.</li>
416
417        <li>The <code class="module"><a href="../mod/prefork.html">prefork</a></code> MPM uses multiple child
418        processes with one thread each.  Each process handles
419        one connection at a time.  On many systems, prefork is
420        comparable in speed to worker, but it uses more memory.
421        Prefork's threadless design has advantages over worker
422        in some situations: it can be used with non-thread-safe
423        third-party modules, and it is easier to debug on platforms
424        with poor thread debugging support.</li>
425
426      </ul>
427
428      <p>For more information on these and other MPMs, please
429      see the MPM <a href="../mpm.html">documentation</a>.</p>
430

432
433    <h3><a name="modules" id="modules">Modules</a></h3>
434
435        
436
437        <p>Since memory usage is such an important consideration in
438        performance, you should attempt to eliminate modules that you are
439        not actually using. If you have built the modules as <a href="../dso.html">DSOs</a>, eliminating modules is a simple
440        matter of commenting out the associated <code class="directive"><a href="../mod/mod_so.html#loadmodule">LoadModule</a></code> directive for that module.
441        This allows you to experiment with removing modules and seeing
442        if your site still functions in their absence.</p>
443
444        <p>If, on the other hand, you have modules statically linked
445        into your Apache binary, you will need to recompile Apache in
446        order to remove unwanted modules.</p>
447
448        <p>An associated question that arises here is, of course, what
449        modules you need, and which ones you don't. The answer here
450        will, of course, vary from one web site to another. However, the
451        <em>minimal</em> list of modules which you can get by with tends
452        to include <code class="module"><a href="../mod/mod_mime.html">mod_mime</a></code>, <code class="module"><a href="../mod/mod_dir.html">mod_dir</a></code>,
453        and <code class="module"><a href="../mod/mod_log_config.html">mod_log_config</a></code>. <code>mod_log_config</code> is,
454        of course, optional, as you can run a web site without log
455        files. This is, however, not recommended.</p>
456

458
459    <h3>Atomic Operations</h3>
460
461      
462
463      <p>Some modules, such as <code class="module"><a href="../mod/mod_cache.html">mod_cache</a></code> and
464      recent development builds of the worker MPM, use APR's
465      atomic API.  This API provides atomic operations that can
466      be used for lightweight thread synchronization.</p>
467
468      <p>By default, APR implements these operations using the
469      most efficient mechanism available on each target
470      OS/CPU platform.  Many modern CPUs, for example, have
471      an instruction that does an atomic compare-and-swap (CAS)
472      operation in hardware.  On some platforms, however, APR
473      defaults to a slower, mutex-based implementation of the
474      atomic API in order to ensure compatibility with older
475      CPU models that lack such instructions.  If you are
476      building Apache for one of these platforms, and you plan
477      to run only on newer CPUs, you can select a faster atomic
478      implementation at build time by configuring Apache with
479      the <code>--enable-nonportable-atomics</code> option:</p>
480
481      <div class="example"><p><code>
482        ./buildconf<br />
483        ./configure --with-mpm=worker --enable-nonportable-atomics=yes
484      </code></p></div>
485
486      <p>The <code>--enable-nonportable-atomics</code> option is
487      relevant for the following platforms:</p>
488
489      <ul>
490
491        <li>Solaris on SPARC<br />
492            By default, APR uses mutex-based atomics on Solaris/SPARC.
493            If you configure with <code>--enable-nonportable-atomics</code>,
494            however, APR generates code that uses a SPARC v8plus opcode for
495            fast hardware compare-and-swap.  If you configure Apache with
496            this option, the atomic operations will be more efficient
497            (allowing for lower CPU utilization and higher concurrency),
498            but the resulting executable will run only on UltraSPARC
499            chips.
500        </li>
501
502        <li>Linux on x86<br />
503            By default, APR uses mutex-based atomics on Linux.  If you
504            configure with <code>--enable-nonportable-atomics</code>,
505            however, APR generates code that uses a 486 opcode for fast
506            hardware compare-and-swap.  This will result in more efficient
507            atomic operations, but the resulting executable will run only
508            on 486 and later chips (and not on 386).
509        </li>
510
511      </ul>

516
517      
518
519      <p>If you include <code class="module"><a href="../mod/mod_status.html">mod_status</a></code> and you also set

531    <h3>accept Serialization - Multiple Sockets</h3>
532
533      
534
535    <div class="warning"><h3>Warning:</h3>
536      <p>This section has not been fully updated
537      to take into account changes made in the 2.x version of the
538      Apache HTTP Server. Some of the information may still be
539      relevant, but please use it with care.</p>
540    </div>
541
542      <p>This discusses a shortcoming in the Unix socket API. Suppose
543      your web server uses multiple <code class="directive"><a href="../mod/mpm_common.html#listen">Listen</a></code> statements to listen on either multiple
544      ports or multiple addresses. In order to test each socket
545      to see if a connection is ready, Apache uses
546      <code>select(2)</code>. <code>select(2)</code> indicates that a
547      socket has <em>zero</em> or <em>at least one</em> connection
548      waiting on it. Apache's model includes multiple children, and
549      all the idle ones test for new connections at the same time. A
550      naive implementation looks something like this (these examples
551      do not match the code, they're contrived for pedagogical
552      purposes):</p>
553
554      <pre class="prettyprint lang-c">        for (;;) {
555          for (;;) {
556            fd_set accept_fds;
557
558            FD_ZERO (&amp;accept_fds);
559            for (i = first_socket; i &lt;= last_socket; ++i) {
560              FD_SET (i, &amp;accept_fds);
561            }
562            rc = select (last_socket+1, &amp;accept_fds, NULL, NULL, NULL);
563            if (rc &lt; 1) continue;
564            new_connection = -1;
565            for (i = first_socket; i &lt;= last_socket; ++i) {
566              if (FD_ISSET (i, &amp;accept_fds)) {
567                new_connection = accept (i, NULL, NULL);
568                if (new_connection != -1) break;
569              }
570            }
571            if (new_connection != -1) break;
572          }
573          process_the(new_connection);
574        }</pre>
575
576
577      <p>But this naive implementation has a serious starvation problem.
578      Recall that multiple children execute this loop at the same
579      time, and so multiple children will block at
580      <code>select</code> when they are in between requests. All
581      those blocked children will awaken and return from
582      <code>select</code> when a single request appears on any socket.
583      (The number of children which awaken varies depending on the
584      operating system and timing issues.) They will all then fall
585      down into the loop and try to <code>accept</code> the
586      connection. But only one will succeed (assuming there's still
587      only one connection ready). The rest will be <em>blocked</em>
588      in <code>accept</code>. This effectively locks those children
589      into serving requests from that one socket and no other
590      sockets, and they'll be stuck there until enough new requests
591      appear on that socket to wake them all up. This starvation
592      problem was first documented in <a href="http://bugs.apache.org/index/full/467">PR#467</a>. There
593      are at least two solutions.</p>
594
595      <p>One solution is to make the sockets non-blocking. In this
596      case the <code>accept</code> won't block the children, and they
597      will be allowed to continue immediately. But this wastes CPU
598      time. Suppose you have ten idle children in
599      <code>select</code>, and one connection arrives. Then nine of
600      those children will wake up, try to <code>accept</code> the
601      connection, fail, and loop back into <code>select</code>,
602      accomplishing nothing. Meanwhile none of those children are
603      servicing requests that occurred on other sockets until they
604      get back up to the <code>select</code> again. Overall this
605      solution does not seem very fruitful unless you have as many
606      idle CPUs (in a multiprocessor box) as you have idle children
607      (not a very likely situation).</p>
608
609      <p>Another solution, the one used by Apache, is to serialize
610      entry into the inner loop. The loop looks like this
611      (differences highlighted):</p>
612
613      <pre class="prettyprint lang-c">        for (;;) {
614          <strong>accept_mutex_on ();</strong>
615          for (;;) {
616            fd_set accept_fds;
617            
618            FD_ZERO (&amp;accept_fds);
619            for (i = first_socket; i &lt;= last_socket; ++i) {
620              FD_SET (i, &amp;accept_fds);
621            }
622            rc = select (last_socket+1, &amp;accept_fds, NULL, NULL, NULL);
623            if (rc &lt; 1) continue;
624            new_connection = -1;
625            for (i = first_socket; i &lt;= last_socket; ++i) {
626              if (FD_ISSET (i, &amp;accept_fds)) {
627                new_connection = accept (i, NULL, NULL);
628                if (new_connection != -1) break;
629              }
630            }
631            if (new_connection != -1) break;
632          }
633          <strong>accept_mutex_off ();</strong>
634          process the new_connection;
635        }</pre>
636
637
638      <p><a id="serialize" name="serialize">The functions</a>
639      <code>accept_mutex_on</code> and <code>accept_mutex_off</code>
640      implement a mutual exclusion semaphore. Only one child can have
641      the mutex at any time. There are several choices for
642      implementing these mutexes. The choice is defined in
643      <code>src/conf.h</code> (pre-1.3) or
644      <code>src/include/ap_config.h</code> (1.3 or later). Some
645      architectures do not have any locking choice made, on these
646      architectures it is unsafe to use multiple
647      <code class="directive"><a href="../mod/mpm_common.html#listen">Listen</a></code>
648      directives.</p>
649
650      <p>The <code class="directive"><a href="../mod/core.html#mutex">Mutex</a></code> directive can
651      be used to change the mutex implementation of the
652      <code>mpm-accept</code> mutex at run-time.  Special considerations
653      for different mutex implementations are documented with that
654      directive.</p>
655
656      <p>Another solution that has been considered but never
657      implemented is to partially serialize the loop -- that is, let
658      in a certain number of processes. This would only be of
659      interest on multiprocessor boxes where it's possible that multiple
660      children could run simultaneously, and the serialization
661      actually doesn't take advantage of the full bandwidth. This is
662      a possible area of future investigation, but priority remains
663      low because highly parallel web servers are not the norm.</p>
664
665      <p>Ideally you should run servers without multiple
666      <code class="directive"><a href="../mod/mpm_common.html#listen">Listen</a></code>
667      statements if you want the highest performance.
668      But read on.</p>
669
670    
671
672    <h3>accept Serialization - Single Socket</h3>
673
674      
675
676      <p>The above is fine and dandy for multiple socket servers, but
677      what about single socket servers? In theory they shouldn't
678      experience any of these same problems because all the children
679      can just block in <code>accept(2)</code> until a connection
680      arrives, and no starvation results. In practice this hides
681      almost the same "spinning" behavior discussed above in the
682      non-blocking solution. The way that most TCP stacks are
683      implemented, the kernel actually wakes up all processes blocked
684      in <code>accept</code> when a single connection arrives. One of
685      those processes gets the connection and returns to user-space.
686      The rest spin in the kernel and go back to sleep when they
687      discover there's no connection for them. This spinning is
688      hidden from the user-land code, but it's there nonetheless.
689      This can result in the same load-spiking wasteful behavior
690      that a non-blocking solution to the multiple sockets case
691      can.</p>
692
693      <p>For this reason we have found that many architectures behave
694      more "nicely" if we serialize even the single socket case. So
695      this is actually the default in almost all cases. Crude
696      experiments under Linux (2.0.30 on a dual Pentium pro 166
697      w/128Mb RAM) have shown that the serialization of the single
698      socket case causes less than a 3% decrease in requests per
699      second over unserialized single-socket. But unserialized
700      single-socket showed an extra 100ms latency on each request.
701      This latency is probably a wash on long haul lines, and only an
702      issue on LANs. If you want to override the single socket
703      serialization, you can define
704      <code>SINGLE_LISTEN_UNSERIALIZED_ACCEPT</code>, and then
705      single-socket servers will not serialize at all.</p>
706

709    <h3>Lingering Close</h3>
710
711      
712
713      <p>As discussed in <a href="http://www.ics.uci.edu/pub/ietf/http/draft-ietf-http-connection-00.txt">
714      draft-ietf-http-connection-00.txt</a> section 8, in order for
715      an HTTP server to <strong>reliably</strong> implement the
716      protocol, it needs to shut down each direction of the
717      communication independently. (Recall that a TCP connection is
718      bi-directional. Each half is independent of the other.)</p>
719
720      <p>When this feature was added to Apache, it caused a flurry of
721      problems on various versions of Unix because of shortsightedness.
722      The TCP specification does not state that the <code>FIN_WAIT_2</code> 
723      state has a timeout, but it doesn't prohibit it.
724      On systems without the timeout, Apache 1.2 induces many sockets
725      stuck forever in the <code>FIN_WAIT_2</code> state. In many cases this
726      can be avoided by simply upgrading to the latest TCP/IP patches
727      supplied by the vendor. In cases where the vendor has never
728      released patches (<em>i.e.</em>, SunOS4 -- although folks with
729      a source license can patch it themselves), we have decided to
730      disable this feature.</p>
731
732      <p>There are two ways to accomplish this. One is the socket
733      option <code>SO_LINGER</code>. But as fate would have it, this
734      has never been implemented properly in most TCP/IP stacks. Even
735      on those stacks with a proper implementation (<em>i.e.</em>,
736      Linux 2.0.31), this method proves to be more expensive (cputime)
737      than the next solution.</p>
738
739      <p>For the most part, Apache implements this in a function
740      called <code>lingering_close</code> (in
741      <code>http_main.c</code>). The function looks roughly like
742      this:</p>
743
744      <pre class="prettyprint lang-c">        void lingering_close (int s)
745        {
746          char junk_buffer[2048];
747          
748          /* shutdown the sending side */
749          shutdown (s, 1);
750
751          signal (SIGALRM, lingering_death);
752          alarm (30);
753
754          for (;;) {
755            select (s for reading, 2 second timeout);
756            if (error) break;
757            if (s is ready for reading) {
758              if (read (s, junk_buffer, sizeof (junk_buffer)) &lt;= 0) {
759                break;
760              }
761              /* just toss away whatever is here */
762            }
763          }
764          
765          close (s);
766        }</pre>
767
768
769      <p>This naturally adds some expense at the end of a connection,
770      but it is required for a reliable implementation. As HTTP/1.1
771      becomes more prevalent, and all connections are persistent,
772      this expense will be amortized over more requests. If you want
773      to play with fire and disable this feature, you can define
774      <code>NO_LINGCLOSE</code>, but this is not recommended at all.
775      In particular, as HTTP/1.1 pipelined persistent connections
776      come into use, <code>lingering_close</code> is an absolute
777      necessity (and <a href="http://www.w3.org/Protocols/HTTP/Performance/Pipeline.html">
778      pipelined connections are faster</a>, so you want to support
779      them).</p>
780

782
783    <h3>Scoreboard File</h3>
784
785      
786
787      <p>Apache's parent and children communicate with each other
788      through something called the scoreboard. Ideally this should be
789      implemented in shared memory. For those operating systems that
790      we either have access to, or have been given detailed ports
791      for, it typically is implemented using shared memory. The rest
792      default to using an on-disk file. The on-disk file is not only
793      slow, but it is unreliable (and less featured). Peruse the
794      <code>src/main/conf.h</code> file for your architecture, and
795      look for either <code>USE_MMAP_SCOREBOARD</code> or
796      <code>USE_SHMGET_SCOREBOARD</code>. Defining one of those two
797      (as well as their companions <code>HAVE_MMAP</code> and
798      <code>HAVE_SHMGET</code> respectively) enables the supplied
799      shared memory code. If your system has another type of shared
800      memory, edit the file <code>src/main/http_main.c</code> and add
801      the hooks necessary to use it in Apache. (Send us back a patch
802      too, please.)</p>
803
804      <div class="note">Historical note: The Linux port of Apache didn't start to
805      use shared memory until version 1.2 of Apache. This oversight
806      resulted in really poor and unreliable behavior of earlier
807      versions of Apache on Linux.</div>

811    <h3>DYNAMIC_MODULE_LIMIT</h3>
812
813      
814
815      <p>If you have no intention of using dynamically loaded modules
816      (you probably don't if you're reading this and tuning your
817      server for every last ounce of performance), then you should add
818      <code>-DDYNAMIC_MODULE_LIMIT=0</code> when building your

827
828    
829
830    <p>Here is a system call trace of Apache 2.0.38 with the worker MPM
831    on Solaris 8. This trace was collected using:</p>
832
833    <div class="example"><p><code>
834      truss -l -p <var>httpd_child_pid</var>.
835    </code></p></div>
836
837    <p>The <code>-l</code> option tells truss to log the ID of the
838    LWP (lightweight process--Solaris' form of kernel-level thread)
839    that invokes each system call.</p>
840
841    <p>Other systems may have different system call tracing utilities
842    such as <code>strace</code>, <code>ktrace</code>, or <code>par</code>.
843    They all produce similar output.</p>
844
845    <p>In this trace, a client has requested a 10KB static file
846    from the httpd. Traces of non-static requests or requests
847    with content negotiation look wildly different (and quite ugly
848    in some cases).</p>
849
850    <div class="example"><pre>/67:    accept(3, 0x00200BEC, 0x00200C0C, 1) (sleeping...)
851/67:    accept(3, 0x00200BEC, 0x00200C0C, 1)            = 9</pre></div>
852
853    <p>In this trace, the listener thread is running within LWP #67.</p>
854
855    <div class="note">Note the lack of <code>accept(2)</code> serialization. On this
856    particular platform, the worker MPM uses an unserialized accept by
857    default unless it is listening on multiple ports.</div>
858
859    <div class="example"><pre>/65:    lwp_park(0x00000000, 0)                         = 0
860/67:    lwp_unpark(65, 1)                               = 0</pre></div>
861
862    <p>Upon accepting the connection, the listener thread wakes up
863    a worker thread to do the request processing. In this trace,
864    the worker thread that handles the request is mapped to LWP #65.</p>
865
866    <div class="example"><pre>/65:    getsockname(9, 0x00200BA4, 0x00200BC4, 1)       = 0</pre></div>
867
868    <p>In order to implement virtual hosts, Apache needs to know
869    the local socket address used to accept the connection. It
870    is possible to eliminate this call in many situations (such
871    as when there are no virtual hosts, or when
872    <code class="directive"><a href="../mod/mpm_common.html#listen">Listen</a></code> directives
873    are used which do not have wildcard addresses). But
874    no effort has yet been made to do these optimizations. </p>
875
876    <div class="example"><pre>/65:    brk(0x002170E8)                                 = 0
877/65:    brk(0x002190E8)                                 = 0</pre></div>
878
879    <p>The <code>brk(2)</code> calls allocate memory from the heap.
880    It is rare to see these in a system call trace, because the httpd
881    uses custom memory allocators (<code>apr_pool</code> and
882    <code>apr_bucket_alloc</code>) for most request processing.
883    In this trace, the httpd has just been started, so it must
884    call <code>malloc(3)</code> to get the blocks of raw memory
885    with which to create the custom memory allocators.</p>
886
887    <div class="example"><pre>/65:    fcntl(9, F_GETFL, 0x00000000)                   = 2
888/65:    fstat64(9, 0xFAF7B818)                          = 0
889/65:    getsockopt(9, 65535, 8192, 0xFAF7B918, 0xFAF7B910, 2190656) = 0
890/65:    fstat64(9, 0xFAF7B818)                          = 0
891/65:    getsockopt(9, 65535, 8192, 0xFAF7B918, 0xFAF7B914, 2190656) = 0
892/65:    setsockopt(9, 65535, 8192, 0xFAF7B918, 4, 2190656) = 0
893/65:    fcntl(9, F_SETFL, 0x00000082)                   = 0</pre></div>
894
895    <p>Next, the worker thread puts the connection to the client (file
896    descriptor 9) in non-blocking mode. The <code>setsockopt(2)</code>
897    and <code>getsockopt(2)</code> calls are a side-effect of how
898    Solaris' libc handles <code>fcntl(2)</code> on sockets.</p>
899
900    <div class="example"><pre>/65:    read(9, " G E T   / 1 0 k . h t m".., 8000)     = 97</pre></div>
901
902    <p>The worker thread reads the request from the client.</p>
903
904    <div class="example"><pre>/65:    stat("/var/httpd/apache/httpd-8999/htdocs/10k.html", 0xFAF7B978) = 0
905/65:    open("/var/httpd/apache/httpd-8999/htdocs/10k.html", O_RDONLY) = 10</pre></div>
906
907    <p>This httpd has been configured with <code>Options FollowSymLinks</code>
908    and <code>AllowOverride None</code>.  Thus it doesn't need to
909    <code>lstat(2)</code> each directory in the path leading up to the
910    requested file, nor check for <code>.htaccess</code> files.
911    It simply calls <code>stat(2)</code> to verify that the file:
912    1) exists, and 2) is a regular file, not a directory.</p>
913
914    <div class="example"><pre>/65:    sendfilev(0, 9, 0x00200F90, 2, 0xFAF7B53C)      = 10269</pre></div>
915
916    <p>In this example, the httpd is able to send the HTTP response
917    header and the requested file with a single <code>sendfilev(2)</code>
918    system call. Sendfile semantics vary among operating systems. On some other
919    systems, it is necessary to do a <code>write(2)</code> or
920    <code>writev(2)</code> call to send the headers before calling
921    <code>sendfile(2)</code>.</p>
922
923    <div class="example"><pre>/65:    write(4, " 1 2 7 . 0 . 0 . 1   -  ".., 78)      = 78</pre></div>
924
925    <p>This <code>write(2)</code> call records the request in the
926    access log. Note that one thing missing from this trace is a
927    <code>time(2)</code> call. Unlike Apache 1.3, Apache 2.x uses
928    <code>gettimeofday(3)</code> to look up the time. On some operating
929    systems, like Linux or Solaris, <code>gettimeofday</code> has an
930    optimized implementation that doesn't require as much overhead
931    as a typical system call.</p>
932
933    <div class="example"><pre>/65:    shutdown(9, 1, 1)                               = 0
934/65:    poll(0xFAF7B980, 1, 2000)                       = 1
935/65:    read(9, 0xFAF7BC20, 512)                        = 0
936/65:    close(9)                                        = 0</pre></div>
937
938    <p>The worker thread does a lingering close of the connection.</p>
939
940    <div class="example"><pre>/65:    close(10)                                       = 0
941/65:    lwp_park(0x00000000, 0)         (sleeping...)</pre></div>
942
943    <p>Finally the worker thread closes the file that it has just delivered
944    and blocks until the listener assigns it another connection.</p>
945
946    <div class="example"><pre>/67:    accept(3, 0x001FEB74, 0x001FEB94, 1) (sleeping...)</pre></div>
947
948    <p>Meanwhile, the listener thread is able to accept another connection
949    as soon as it has dispatched this connection to a worker thread (subject
950    to some flow-control logic in the worker MPM that throttles the listener
951    if all the available workers are busy).  Though it isn't apparent from
952    this trace, the next <code>accept(2)</code> can (and usually does, under
953    high load conditions) occur in parallel with the worker thread's handling
954    of the just-accepted connection.</p>
