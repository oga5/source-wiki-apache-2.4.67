30  <p>Afin d'assister les utilisateurs lors de leurs opérations de mise à
31  jour, nous maintenons un document
32  qui comporte des informations critiques à l'attention des personnes qui
33  utilisent déjà le serveur HTTP Apache. Ces informations
34  ne sont que de brèves notes, et vous
35  trouverez plus d'informations dans le document <a href="new_features_2_4.html">Nouvelles fonctionnalités</a>, ou dans
36  le fichier <code>src/CHANGES</code>. Les développeurs d'applications
37  et de modules trouveront un résumé des modifications de l'API dans la
38  vue d'ensemble <a href="developer/new_api_2_4.html">Mises à jour de
39  l'API</a>.</p>
40
41  <p>Ce document présente les changements de comportement du serveur qui
42  peuvent nécessiter une modification de la configuration, et une
43  méthode pour utiliser la version 2.4 du serveur en parallèle avec la
44  version 2.2. Pour tirer parti des nouvelles fonctionnalités de la
45  version 2.4, reportez-vous au document "Nouvelles fonctionnalités".</p>
46
47  <p>Ce document ne décrit que les modifications intervenues entre les versions
48  2.2 et 2.4. Si vous effectuez une mise à jour depuis la version 2.0, vous
49  devez aussi consulter le
50  <a href="http://httpd.apache.org/docs/2.2/upgrading.html">document de mise
51  à jour de 2.0 vers 2.2.</a></p>

63<h2><a name="compile-time" id="compile-time">Modifications des paramètres de compilation</a></h2>
64    
65     <p>Le processus de compilation est très similaire à celui de la
66     version 2.2. Dans la plupart des cas, vous pourrez utiliser votre
67     ancienne ligne de commande <code>configure</code> (telle qu'elle
68     est enregistrée dans le fichier <code>build/config.nice</code>
69     situé dans le répertoire de compilation du serveur). Voici certains
70     changements intervenus dans la configuration par défaut :</p>
71
72    <ul>
73      <li>Les modules suivants ont été supprimés : mod_authn_default,
74      mod_authz_default et mod_mem_cache. Si vous utilisiez
75      mod_mem_cache sous la version 2.2, vous devez maintenant utiliser
76      <code class="module"><a href="./mod/mod_cache_disk.html">mod_cache_disk</a></code> dans la version 2.4.</li>
77
78      <li>Toutes les implémentations de répartition de charge ont été
79      déplacées vers des sous-modules spécifiques de mod_proxy, comme
80      <code class="module"><a href="./mod/mod_lbmethod_bybusyness.html">mod_lbmethod_bybusyness</a></code>. Vous devrez compiler et
81      chargés tous les modules correspondants que votre configuration
82      utilise.</li>
83
84      <li>Le support de BeOS, TPF, et des anciennes plates-formes telles
85      que A/UX, Next, et Tandem a été supprimé, car
86      elles ne sont plus considérées comme maintenues.</li>
87
88      <li>configure: les modules dynamiques (DSO) sont compilés par
89      défaut</li>
90
91      <li>configure: par défaut, seul un jeu de modules de base est
92      chargé. Les autres directives <code class="directive">LoadModule</code>
93      sont mises en commentaires dans le fichier de configuration.</li>
94
95      <li>configure: le jeu de modules "most" est compilé par défaut</li>
96
97      <li>configure: le jeu de modules "reallyall" ajoute les modules de
98      développeur au jeu "all".</li>
99    </ul>

103<h2><a name="run-time" id="run-time">Modifications de la configuration à l'exécution</a></h2>
104    
105<p>Des changements significatifs dans la configuration de
106l'autorisation, ainsi que quelques changements mineurs, peuvent
107nécessiter une mise à jour des fichiers de configuration de la version
1082.2 avant de les utiliser sous la version 2.4.</p>
109
110    <h3><a name="authz" id="authz">Autorisation</a></h3>
111      
112
113      <p>Tout fichier de configuration qui gère des autorisations devra
114      probablement être mis à jour.</p>
115
116    <p>Vous devez vous reporter au document <a href="howto/auth.html">Authentification, autorisation et contrôle
117    d'accès</a>, et plus particulièrement à la section <a href="howto/auth.html#beyond">Pour aller plus loin qu'une simple
118    autorisation</a> qui explique les nouveaux mécanismes permettant de
119    contrôler l'ordre dans lequel les directives d'autorisation sont
120    appliquées.</p>
121
122    <p>Les directives qui contrôlent la manière dont les modules
123    d'autorisation réagissent lorsqu'ils ne reconnaissent pas
124    l'utilisateur authentifié ont été supprimées : elles comprennent les
125    directives AuthzLDAPAuthoritative, AuthzDBDAuthoritative,
126    AuthzDBMAuthoritative, AuthzGroupFileAuthoritative,
127    AuthzUserAuthoritative et AuthzOwnerAuthoritative. Ces directives
128    ont été remplacées par les directives plus explicites <code class="directive"><a href="./mod/mod_authz_core.html#requireany">RequireAny</a></code>, <code class="directive"><a href="./mod/mod_authz_core.html#requirenone">RequireNone</a></code>, et <code class="directive"><a href="./mod/mod_authz_core.html#requireall">RequireAll</a></code>.</p>
129
130    <p>Si vous utilisez <code class="module"><a href="./mod/mod_authz_dbm.html">mod_authz_dbm</a></code>, vous devez
131    mettre à jour votre configuration en remplaçant les directives du
132    style <code>Require group ...</code> par des directives du style
133    <code>Require dbm-group ...</code>.</p>
134
135    <h4><a name="access" id="access">Contrôle d'accès</a></h4>
136      
137
138      <p>Dans la version 2.2, le contrôle d'accès basé sur le nom d'hôte
139      du client, son adresse IP, ou d'autres caractéristiques de la
140      requête était assuré via les directives <code class="directive"><a href="./mod/mod_access_compat.html#order">Order</a></code>, <code class="directive"><a href="./mod/mod_access_compat.html#allow">Allow</a></code>, <code class="directive"><a href="./mod/mod_access_compat.html#deny">Deny</a></code>, et <code class="directive"><a href="./mod/mod_access_compat.html#satisfy">Satisfy</a></code>.</p>
141
142      <p>Dans la version 2.4, ce contrôle d'accès est assuré, comme tout
143      contrôle d'autorisation, par le nouveau module
144      <code class="module"><a href="./mod/mod_authz_host.html">mod_authz_host</a></code>. Bien que le module
145      <code class="module"><a href="./mod/mod_access_compat.html">mod_access_compat</a></code> soit fourni à des fins de
146      compatibilité avec les anciennes configurations, les anciennes
147      directives de contrôle d'accès devront être remplacées par les
148      nouveaux mécanismes d'authentification.</p>
149
150      <div class="note"><h3>Mélanger anciennes et nouvelles directives</h3>
151      <p>Mélanger d'anciennes directives comme <code class="directive"><a href="./mod/mod_access_compat.html#order">Order</a></code>, <code class="directive"><a href="./mod/mod_access_compat.html#allow">Allow</a></code> ou <code class="directive"><a href="./mod/mod_access_compat.html#deny">Deny</a></code> avec des nouvelles comme
152      <code class="directive"><a href="./mod/mod_authz_core.html#require">Require</a></code> est techniquement
153      possible mais déconseillé. En effet, <code class="module"><a href="./mod/mod_access_compat.html">mod_access_compat</a></code> a
154      été conçu pour supporter des configurations ne contenant que des anciennes
155      directives afin de faciliter le passage à la version 2.4. Les
156      exemples ci-dessous vous permettront de vous faire une meilleure idée des
157      problèmes qui peuvent survenir.
158      </p>
159      </div>
160
161      <p>Voici quelques exemples de contrôle d'accès avec l'ancienne et
162      la nouvelle méthode :</p>
163
164      <p>Dans cet exemple, il n'y a pas d'authentification et toutes les requêtes sont rejetées :</p>
165      <div class="example"><h3>version 2.2 :</h3><pre class="prettyprint lang-config">Order deny,allow
166Deny from all</pre>
167</div>
168      <div class="example"><h3>version 2.4 :</h3><pre class="prettyprint lang-config">Require all denied</pre>
169</div>
170
171      <p>Dans cet exemple, il n'y a pas d'authentification et toutes les requêtes sont acceptées :</p>
172      <div class="example"><h3>version 2.2 :</h3><pre class="prettyprint lang-config">Order allow,deny
173Allow from all</pre>
174</div>
175      <div class="example"><h3>version 2.4 :</h3><pre class="prettyprint lang-config">Require all granted</pre>
176</div>
177
178      <p>Dans l'exemple suivant, il n'y a pas d'authentification et tous les
179      hôtes du domaine example.org ont l'autorisation d'accès, tous les autres
180      étant rejetés :</p>
181
182      <div class="example"><h3>version 2.2 :</h3><pre class="prettyprint lang-config">Order Deny,Allow
183Deny from all
184Allow from example.org</pre>
185</div>
186      <div class="example"><h3>version 2.4 :</h3><pre class="prettyprint lang-config">Require host example.org</pre>
187</div>
188    <p>Dans l'exemple suivant, tous les hôtes du domaine example.org
189      ont l'autorisation d'accès, tous les autres sont rejetés :</p>
190
191      <div class="example"><h3>version 2.2 :</h3><pre class="prettyprint lang-config">Order Deny,Allow
192Deny from all
193Allow from example.org</pre>
194</div>
195      <div class="example"><h3>version 2.4 :</h3><pre class="prettyprint lang-config">Require host example.org</pre>
196</div>
197
198      <p>Dans l'exemple suivant, le mélange d'anciennes et de nouvelles
199      directives produit des résultats inattendus.</p>
200 
201      <div class="example"><h3>Mélange d'anciennes et de nouvelles directives : RESULTAT
202	INATTENDU</h3><pre class="prettyprint lang-config">DocumentRoot "/var/www/html"
203
204&lt;Directory "/"&gt;
205    AllowOverride None
206    Order deny,allow
207    Deny from all
208&lt;/Directory&gt;
209
210&lt;Location "/server-status"&gt;
211    SetHandler server-status
212    Require local
213&lt;/Location&gt;
214
215access.log - GET /server-status 403 127.0.0.1
216error.log - AH01797: client denied by server configuration: /var/www/html/server-status</pre>
217</div>
218      <p>Pourquoi httpd interdit l'accès à server-status alors que la
219      configuration semble l'autoriser ? Parce que dans ce scénario de <a href="sections.html#merging">fusion</a> de configuration, les
220      directives de <code class="module"><a href="./mod/mod_access_compat.html">mod_access_compat</a></code> sont prioritaires par
221      rapport à celles de <code class="module"><a href="./mod/mod_authz_host.html">mod_authz_host</a></code>.</p>
222
223      <p>L'exemple suivant quant à lui produit un résultat conforme :</p>
224
225      <div class="example"><h3>Mélange d'anciennes et de nouvelles directives : RESULTAT
226	CONFORME</h3><pre class="prettyprint lang-config">DocumentRoot "/var/www/html"
227
228&lt;Directory "/"&gt;
229    AllowOverride None
230    Require all denied
231&lt;/Directory&gt;
232
233&lt;Location "/server-status"&gt;
234    SetHandler server-status
235    Order deny,allow
236    Deny from all
237    Allow From 127.0.0.1
238&lt;/Location&gt;
239
240access.log - GET /server-status 200 127.0.0.1</pre>
241</div> 
242      <p>En conclusion, même si une configuration hybride peut fonctionner,
243      essayez de l'éviter lors de la mise à jour : soit conservez les anciennes
244      directives, puis migrez-les vers les nouvelles ultérieurement, soit
245      effectuez une migration immédiate de toutes les anciennes directives vers
246      les nouvelles.  
247      </p>      
248    
249
250    <p>Dans de nombreuses configurations avec authentification où la directive
251     <code class="directive">Satisfy</code> était définie à sa valeur par défaut
252     <em>ALL</em>, les lignes de configuration qui désactivent le contrôle
253     d'accès basé sur l'hôte sont maintenant omises :</p>
254
255      <div class="example"><h3>Version 2.2 :</h3><pre class="prettyprint lang-config"># configuration en version 2.2 qui désactive le contrôle d'accès basé sur le nom
256# d'hôte pour n'utiliser que l'authentification
257Order Deny,Allow
258Allow from all
259AuthType Basic
260AuthBasicProvider file
261AuthUserFile /example.com/conf/users.passwd
262AuthName secure
263Require valid-user</pre>
264</div>
265      <div class="example"><h3>Version 2.4 :</h3><pre class="prettyprint lang-config"># Pas besoin de remplacer les directives de contrôle d'accès basées sur le nom
266# d'hôte désactivées
267AuthType Basic
268AuthBasicProvider file
269AuthUserFile /example.com/conf/users.passwd
270AuthName secure
271Require valid-user</pre>
272</div>
273
274     <p>Dans les configurations où l'authentification et le contrôle d'accès se
275     combinaient dans un but précis, les directives de contrôle d'accès doivent
276     être migrées. Dans l'exemple suivant, les requêtes qui correspondent aux
277     <em>deux</em> critères sont acceptées :</p>
278      <div class="example"><h3>Version 2.2 :</h3><pre class="prettyprint lang-config">Order allow,deny
279Deny from all
280# ALL est la valeur par défaut de Satisfy
281Satisfy ALL
282Allow from 127.0.0.1
283AuthType Basic
284AuthBasicProvider file
285AuthUserFile /example.com/conf/users.passwd
286AuthName secure
287Require valid-user</pre>
288</div>
289      <div class="example"><h3>Version 2.4 :</h3><pre class="prettyprint lang-config">AuthType Basic
290AuthBasicProvider file
291AuthUserFile /example.com/conf/users.passwd
292AuthName secure
293&lt;RequireAll&gt;
294  Require valid-user
295  Require ip 127.0.0.1
296&lt;/RequireAll&gt;</pre>
297</div>
298
299     <p>Dans les configurations où l'authentification et le contrôle d'accès se
300     combinaient dans un but précis, les directives de contrôle d'accès doivent
301     être migrées. Dans l'exemple suivant, les requêtes qui correspondent à
302     <em>au moins un</em> critère sont acceptées :</p>
303      <div class="example"><h3>Version 2.2 :</h3><pre class="prettyprint lang-config">Order allow,deny
304Deny from all
305Satisfy any
306Allow from 127.0.0.1
307AuthType Basic
308AuthBasicProvider file
309AuthUserFile /example.com/conf/users.passwd
310AuthName secure
311Require valid-user</pre>
312</div>
313      <div class="example"><h3>Version 2.4 :</h3><pre class="prettyprint lang-config">AuthType Basic
314AuthBasicProvider file
315AuthUserFile /example.com/conf/users.passwd
316AuthName secure
317# Implicite : &lt;RequireAny&gt;
318Require valid-user
319Require ip 127.0.0.1</pre>
320</div>

324    <h3><a name="config" id="config">Autres changements dans la configuration</a></h3>
325      
326
327      <p>D'autres ajustements mineurs peuvent s'avérer nécessaires pour
328      certaines configurations particulières, comme décrit ci-dessous.</p>
329
330      <ul>
331        <li>La directive <code class="directive">MaxRequestsPerChild</code> a été renommée en
332	<code class="directive"><a href="./mod/mpm_common.html#maxconnectionsperchild">MaxConnectionsPerChild</a></code>;
333	ce nouveau nom reflète mieux l'usage de cette directive.
334	L'ancien nom est encore supporté.</li>
335
336	<li>La directive <code class="directive">MaxClients</code> a
337	été renommée en <code class="directive"><a href="./mod/mpm_common.html#maxrequestworkers">MaxRequestWorkers</a></code>; ce nouveau
338	nom reflète mieux l'usage de cette directive. Pour les
339	modules multiprocessus asynchrones, comme <code class="module"><a href="./mod/event.html">event</a></code>, le nombre
340	maximal de clients n'est pas équivalent au nombre de threads du
341	worker. L'ancien nom est encore supporté.</li>
342
343        <li>La directive <code class="directive"><a href="./mod/core.html#defaulttype">DefaultType</a></code> ne produit plus aucun
344	effet, si ce n'est d'émettre un avertissement si elle est
345	définie à une valeur autre que <code>none</code>. D'autres
346	directives de configuration la remplacent dans la version 2.4.
347        </li>
348
349	<li>La valeur par défaut de la directive <code class="directive"><a href="./mod/core.html#allowoverride">AllowOverride</a></code> est maintenant
350	<code>None</code>.</li>
351
352	<li>La valeur par défaut de la directive <code class="directive"><a href="./mod/core.html#enablesendfile">EnableSendfile</a></code> est maintenant Off.</li>
353
354	<li>La valeur par défaut de la directive <code class="directive"><a href="./mod/core.html#fileetag">FileETag</a></code> est maintenant "MTime Size"
355	(sans INode).</li>
356
357        <li><code class="module"><a href="./mod/mod_dav_fs.html">mod_dav_fs</a></code>: le format du fichier <code class="directive"><a href="./mod/mod_dav_fs.html#davlockdb">DavLockDB</a></code> a changé pour les systèmes
358	avec inodes. L'ancien fichier <code class="directive"><a href="./mod/mod_dav_fs.html#davlockdb">DavLockDB</a></code> doit être supprimé dans le
359	cadre de la mise à jour.
360        </li>
361
362        <li>La directive <code class="directive"><a href="./mod/core.html#keepalive">KeepAlive</a></code>
363	n'accepte que les valeurs <code>On</code> ou <code>Off</code>.
364	Avant, toute valeur autre que "Off" ou "0" était traitée comme
365	"On".</li>
366
367        <li>Les directives AcceptMutex, LockFile, RewriteLock, SSLMutex,
368	SSLStaplingMutex et WatchdogMutexPath ont été remplacées par la
369	directive unique <code class="directive"><a href="./mod/core.html#mutex">Mutex</a></code>.
370	Vous devez évaluer l'impact de ces directives obsolètes dans
371	votre configuration version 2.2 afin de déterminer si elles
372	peuvent être simplement supprimées, ou si elles doivent être
373	remplacées par la directive <code class="directive"><a href="./mod/core.html#mutex">Mutex</a></code>.</li>
374
375        <li><code class="module"><a href="./mod/mod_cache.html">mod_cache</a></code>: la directive <code class="directive"><a href="./mod/mod_cache.html#cacheignoreurlsessionidentifiers">CacheIgnoreURLSessionIdentifiers</a></code>
376	effectue maintenant une correspondance exacte dans la chaîne de
377	paramètres au lieu d'une correspondance partielle. Si votre
378	configuration mettait en jeu des sous-chaînes comme
379	<code>sessionid</code> pour correspondre à
380	<code>/une-application/image.gif;jsessionid=123456789</code>,
381	vous devez maintenant utiliser la chaîne de correspondance
382	complète <code>jsessionid</code>.
383        </li>
384
385	<li><code class="module"><a href="./mod/mod_cache.html">mod_cache</a></code>: le second paramètre de la
386	directive <code class="directive"><a href="./mod/mod_cache.html#cacheenable">CacheEnable</a></code>
387	ne concerne les contenus en mandat direct que s'ils débutent par
388	le protocole approprié. Dans les versions 2.2 et antérieures, un
389	paramètre tel que '/' concernait tous les contenus.</li>
390
391        <li><code class="module"><a href="./mod/mod_ldap.html">mod_ldap</a></code>: la directive <code class="directive"><a href="./mod/mod_ldap.html#ldaptrustedclientcert">LDAPTrustedClientCert</a></code> s'utilise
392	maintenant exclusivement au sein d'une configuration de niveau
393	répertoire. Si vous utilisez cette directive, passez en revue
394	votre configuration pour vous assurer qu'elle est bien présente
395	dans tous les contextes de répertoire nécessaires.</li>
396
397	<li><code class="module"><a href="./mod/mod_filter.html">mod_filter</a></code>: la syntaxe de la directive
398	<code class="directive"><a href="./mod/mod_filter.html#filterprovider">FilterProvider</a></code> utilise
399	maintenant une expression booléenne pour déterminer si un filtre
400	s'applique.
401        </li>
402
403	<li><code class="module"><a href="./mod/mod_include.html">mod_include</a></code>:
404            <ul>
405            <li>L'élément <code>#if expr</code> utilise maintenant le
406	    nouvel <a href="expr.html">interpréteur d'expressions</a>.
407	    L'ancienne syntaxe peut être réactivée via la directive
408	    <code class="directive"><a href="./mod/mod_include.html#ssilegacyexprparser">SSILegacyExprParser</a></code>.
409            </li>
410            <li>Dans la portée du répertoire, une directive de
411	    configuration SSI* ne provoque plus la réinitialisation à
412	    leur valeur par défaut de toutes les directives SSI* de
413	    niveau répertoire.</li>
414            </ul>
415        </li>
416
417        <li><code class="module"><a href="./mod/mod_charset_lite.html">mod_charset_lite</a></code> : l'option
418	<code>DebugLevel</code> a été supprimée en faveur d'une
419	configuration de la directive <code class="directive"><a href="./mod/core.html#loglevel">LogLevel</a></code> au niveau répertoire.
420        </li>
421
422        <li><code class="module"><a href="./mod/mod_ext_filter.html">mod_ext_filter</a></code> : l'option
423	<code>DebugLevel</code> a été supprimée en faveur d'une
424	configuration de la directive <code class="directive"><a href="./mod/core.html#loglevel">LogLevel</a></code> au niveau répertoire.
425        </li>
426
427	<li><code class="module"><a href="./mod/mod_proxy_scgi.html">mod_proxy_scgi</a></code>: certaines applications web
428	ne fonctionneront plus correctement avec la nouvelle
429	configuration de <code>PATH_INFO</code> qui est différente de
430	celle de la version 2.2. La configuration
431	précédente peut être
432	restaurée en définissant la variable
433	<code>proxy-scgi-pathinfo</code>.</li>
434
435	<li><code class="module"><a href="./mod/mod_ssl.html">mod_ssl</a></code>: le contrôle de révocation des
436	certificats basé sur les CRL doit être maintenant explicitement
437	configuré via la directive <code class="directive"><a href="./mod/mod_ssl.html#sslcarevocationcheck">SSLCARevocationCheck</a></code>.
438        </li>
439
440        <li><code class="module"><a href="./mod/mod_substitute.html">mod_substitute</a></code>: la taille maximale d'une
441	ligne est maintenant 1Mo.
442        </li>
443
444        <li><code class="module"><a href="./mod/mod_reqtimeout.html">mod_reqtimeout</a></code>: si ce module est chargé, il
445	définit maintenant certains temps d'attente par défaut.</li>
446
447	<li><code class="module"><a href="./mod/mod_dumpio.html">mod_dumpio</a></code>: la directive
448	<code class="directive">DumpIOLogLevel</code> n'est plus supportée. Les
449	données sont toujours enregistrées au niveau <code>trace7</code>
450	de <code class="directive"><a href="./mod/core.html#loglevel">LogLevel</a></code></li>
451
452        <li>Jusqu'à la version 2.2, sur les plateformes de style Unix, 
453	les commandes de redirection des logs définies via <code class="directive"><a href="./mod/core.html#errorlog">ErrorLog</a></code> ou <code class="directive"><a href="./mod/mod_log_config.html#customlog">CustomLog</a></code> étaient invoquées
454	en utilisant <code>/bin/sh -c</code>. A
455	partir de la version 2.4, les commandes de redirection des logs
456	sont exécutées directement. Pour retrouver l'ancien
457	comportement, voir la <a href="logs.html#piped">documentation
458	sur la redirection des logs</a></li>
459
460    </ul>

464<h2><a name="misc" id="misc">Changements divers</a></h2>
465    
466
467    <ul>
468      <li><code class="module"><a href="./mod/mod_auto_index.html">mod_auto_index</a></code>: extrait maintenant les titres
469      et affiche la description pour les fichiers .xhtml qui étaient
470      jusqu'alors ignorés.</li>
471
472      <li><code class="module"><a href="./mod/mod_ssl.html">mod_ssl</a></code> : le format par défaut des variables
473      <code>*_DN</code> a changé. Il est cependant encore possible
474      d'utiliser l'ancien format via la nouvelle option
475      <code>LegacyDNStringFormat</code> de la directive <code class="directive"><a href="./mod/mod_ssl.html#ssloptions">SSLOptions</a></code>. Le protocole SSLv2 n'est
476      plus supporté. Les directives <code class="directive"><a href="./mod/mod_ssl.html#sslproxycheckpeercn">SSLProxyCheckPeerCN</a></code> et
477      <code class="directive"><a href="./mod/mod_ssl.html#sslproxycheckpeerexpire">SSLProxyCheckPeerExpire</a></code>
478      sont maintenant définies par défaut à On, et les requêtes mandatées
479      vers des serveurs HTTPS possèdant des certificats non conformes ou
480      périmés échoueront donc avec un code d'erreur 502 (Bad gateway).</li>
481
482      <li><code class="program"><a href="./programs/htpasswd.html">htpasswd</a></code> utilise maintenant par défaut les
483      condensés MD5 sur toutes les plates-formes.</li>
484
485      <li>La directive <code class="directive"><a href="./mod/core.html#namevirtualhost">NameVirtualHost</a></code> n'a plus aucun effet, si
486      ce n'est l'émission d'un avertissement. Toute combinaison
487      adresse/port apparaissant dans plusieurs serveurs virtuels est
488      traitée implicitement comme un serveur virtuel basé sur le nom.
489      </li>
490
491      <li><code class="module"><a href="./mod/mod_deflate.html">mod_deflate</a></code> n'effectue plus de compression
492      s'il s'aperçoit que la quantité de données ajoutée par la
493      compression est supérieure à la quantité de données à compresser.
494      </li>
495
496      <li>Les pages d'erreur multilingues de la version 2.2.x ne
497      fonctionneront qu'après avoir été corrigées pour
498      respecter la nouvelle syntaxe de l'élément <code>#if expr=</code>
499      du module <code class="module"><a href="./mod/mod_include.html">mod_include</a></code>, ou si la directive
500      <code class="directive"><a href="./mod/mod_include.html#ssilegacyexprparser">SSILegacyExprParser</a></code> a
501      été activée pour le répertoire contenant les pages d'erreur.
502      </li>
503
504      <li>La fonctionnalité fournie par <code>mod_authn_alias</code>
505      dans les précédentes versions (en fait la directive
506      <code class="directive"><a href="./mod/mod_authn_core.html#authnprovideralias">AuthnProviderAlias</a></code>)
507      est maintenant fournie par <code class="module"><a href="./mod/mod_authn_core.html">mod_authn_core</a></code>.  
508      </li>
509
510      <li>Les directives RewriteLog et RewriteLogLevel ont été
511      supprimées. Leur fonctions sont maintenant assurées par la
512      directive <code class="directive"><a href="./mod/core.html#loglevel">LogLevel</a></code> qui permet de définir
513      un niveau de journalisation approprié pour le module
514      <code class="module"><a href="./mod/mod_rewrite.html">mod_rewrite</a></code>. Voir aussi la section <a href="mod/mod_rewrite.html#logging">journalisation de
515      mod_rewrite</a>.</li>
516
517      <li>Le symbole <code>&lt;set&gt;</code> de <code class="module"><a href="./mod/mod_include.html">mod_include</a></code>
518      n’effectue plus le décodage d’entité sur la valeur définie.</li>
519
520    </ul>

528	version 2.4 avant d'être chargés.</p>

530    <p>De nombreux modules tiers conçus pour la version 2.2
531    fonctionneront sans changement avec le serveur HTTP Apache
532    version 2.4. Certains nécessiteront cependant des modifications ; se

536<h2><a name="commonproblems" id="commonproblems">Problèmes de mise à jour courants</a></h2>
537    
538    <ul><li>Erreurs au démarrage :
539    <ul>
540      <li><code>Invalid command 'User', perhaps misspelled or defined by
541      a module not included in the server configuration</code> - chargez
542      le module <code class="module"><a href="./mod/mod_unixd.html">mod_unixd</a></code></li>
543
544      <li><code>Invalid command 'Require', perhaps misspelled or defined
545      by a module not included in the server configuration</code>, ou
546      <code>Invalid command 'Order', perhaps misspelled or defined by a
547      module not included in the server configuration</code> - chargez
548      le module <code class="module"><a href="./mod/mod_access_compat.html">mod_access_compat</a></code>, ou mettez à jour
549      vers la version 2.4 les directives d'autorisation.</li>
550
551      <li><code>Ignoring deprecated use of DefaultType in line NN of
552      /path/to/httpd.conf</code> - supprimez la directive <code class="directive"><a href="./mod/core.html#defaulttype">DefaultType</a></code> et remplacez-la par les
553      directives de configuration appropriées.</li>
554
555      <li><code>Invalid command 'AddOutputFilterByType', perhaps misspelled 
556      or defined by a module not included in the server configuration
557      </code> - la directive <code class="directive"><a href="./mod/mod_filter.html#addoutputfilterbytype">AddOutputFilterByType</a></code> qui était
558      jusqu'alors implémentée par le module core, l'est maintenant par
559      le module mod_filter, qui doit donc être chargé.</li>
560
561    </ul></li>
562    <li>Erreurs de traitement des requêtes :
563    <ul>
564      <li><code>configuration error:  couldn't check user: /path</code> -
565      chargez le module <code class="module"><a href="./mod/mod_authn_core.html">mod_authn_core</a></code>.</li>
566      <li>Les fichiers <code>.htaccess</code> ne sont pas traités -
567      Vérifiez la présence d'une directive <code class="directive"><a href="./mod/core.html#allowoverride">AllowOverride</a></code> appropriée ; sa valeur par
568      défaut est maintenant <code>None</code>.</li>
569    </ul>
570    </li>
571</ul>
