33    <p>The Apache HTTP Server is a modular program where the
34    administrator can choose the functionality to include in the
35    server by selecting a set of modules.
36    Modules will be compiled as Dynamic Shared Objects (DSOs)
37    that exist separately from the main <code class="program"><a href="./programs/httpd.html">httpd</a></code>
38    binary file. DSO modules may be compiled at the time the server
39    is built, or they may be compiled and added at a later time
40    using the Apache Extension Tool (<code class="program"><a href="./programs/apxs.html">apxs</a></code>).</p>
41    <p>Alternatively, the modules can be statically compiled into
42    the <code class="program"><a href="./programs/httpd.html">httpd</a></code> binary when the server is built.</p>
43
44    <p>This document describes how to use DSO modules as well as
45    the theory behind their use.</p>
46  </div>

54<h2><a name="implementation" id="implementation">Implementation</a></h2>
55
56<table class="related"><tr><th>Related Modules</th><th>Related Directives</th></tr><tr><td><ul><li><code class="module"><a href="./mod/mod_so.html">mod_so</a></code></li></ul></td><td><ul><li><code class="directive"><a href="./mod/mod_so.html#loadmodule">LoadModule</a></code></li></ul></td></tr></table>
57
58    <p>The DSO support for loading individual Apache httpd modules is based
59    on a module named <code class="module"><a href="./mod/mod_so.html">mod_so</a></code> which must be statically
60    compiled into the Apache httpd core. It is the only module besides
61    <code class="module"><a href="./mod/core.html">core</a></code> which cannot be put into a DSO
62    itself. Practically all other distributed Apache httpd modules will then
63    be placed into a DSO. After a module is compiled into a DSO named
64    <code>mod_foo.so</code> you can use <code class="module"><a href="./mod/mod_so.html">mod_so</a></code>'s <code class="directive"><a href="./mod/mod_so.html#loadmodule">LoadModule</a></code> directive in your
65    <code>httpd.conf</code> file to load this module at server startup
66    or restart.</p>
67    <p>The DSO builds for individual modules can be disabled via
68    <code class="program"><a href="./programs/configure.html">configure</a></code>'s <code>--enable-mods-static</code>
69    option as discussed in the <a href="install.html">install
70    documentation</a>.</p>
71
72    <p>To simplify this creation of DSO files for Apache httpd modules
73    (especially for third-party modules) a support program
74    named <code class="program"><a href="./programs/apxs.html">apxs</a></code> (<dfn>APache
75    eXtenSion</dfn>) is available. It can be used to build DSO based
76    modules <em>outside of</em> the Apache httpd source tree. The idea is
77    simple: When installing Apache HTTP Server the <code class="program"><a href="./programs/configure.html">configure</a></code>'s
78    <code>make install</code> procedure installs the Apache httpd C
79    header files and puts the platform-dependent compiler and
80    linker flags for building DSO files into the <code class="program"><a href="./programs/apxs.html">apxs</a></code>
81    program. This way the user can use <code class="program"><a href="./programs/apxs.html">apxs</a></code> to compile
82    his Apache httpd module sources without the Apache httpd distribution
83    source tree and without having to fiddle with the
84    platform-dependent compiler and linker flags for DSO
85    support.</p>

88<h2><a name="usage" id="usage">Usage Summary</a></h2>
89
90    <p>To give you an overview of the DSO features of Apache HTTP Server 2.x,
91    here is a short and concise summary:</p>
92
93    <ol>
94      <li>
95        <p>Build and install a <em>distributed</em> Apache httpd module, say
96        <code>mod_foo.c</code>, into its own DSO
97        <code>mod_foo.so</code>:</p>
98
99<div class="example"><pre class="prettyprint lang-sh">$ ./configure --prefix=/path/to/install --enable-foo
100$ make install</pre>
101</div>
102      </li>
103
104      <li>
105      <p>Configure Apache HTTP Server with all modules enabled. Only a basic
106      set will be loaded during server startup. You can change the set of loaded
107      modules by activating or deactivating the <code class="directive"><a href="./mod/mod_so.html#loadmodule">LoadModule</a></code> directives in
108      <code>httpd.conf</code>.</p>
109
110<div class="example"><pre class="prettyprint lang-sh">$ ./configure --enable-mods-shared=all
111$ make install</pre>
112</div>
113      </li>
114
115      <li>
116      <p>Some modules are only useful for developers and will not be build.
117      when using the module set <em>all</em>. To build all available modules
118      including developer modules use <em>reallyall</em>. In addition the
119      <code class="directive"><a href="./mod/mod_so.html#loadmodule">LoadModule</a></code> directives for all
120      built modules can be activated via the configure option
121      <code>--enable-load-all-modules</code>.</p>
122
123<div class="example"><pre class="prettyprint lang-sh">$ ./configure --enable-mods-shared=reallyall --enable-load-all-modules
124$ make install</pre>
125</div>
126      </li>
127
128      <li>
129        Build and install a <em>third-party</em> Apache httpd module, say
130        <code>mod_foo.c</code>, into its own DSO
131        <code>mod_foo.so</code> <em>outside of</em> the Apache httpd
132        source tree using <code class="program"><a href="./programs/apxs.html">apxs</a></code>:
133
134<div class="example"><pre class="prettyprint lang-sh">$ cd /path/to/3rdparty
135$ apxs -cia mod_foo.c</pre>
136</div>
137      </li>
138    </ol>
139
140    <p>In all cases, once the shared module is compiled, you must
141    use a <code class="directive"><a href="./mod/mod_so.html#loadmodule">LoadModule</a></code>
142    directive in <code>httpd.conf</code> to tell Apache httpd to activate
143    the module.</p>
144
145    <p>See the <a href="programs/apxs.html">apxs documentation</a> for more details.</p>

148<h2><a name="background" id="background">Background</a></h2>
149
150    <p>On modern Unix derivatives there exists a mechanism
151    called dynamic linking/loading of <em>Dynamic Shared
152    Objects</em> (DSO) which provides a way to build a piece of
153    program code in a special format for loading it at run-time
154    into the address space of an executable program.</p>
155
156    <p>This loading can usually be done in two ways: automatically
157    by a system program called <code>ld.so</code> when an
158    executable program is started or manually from within the
159    executing program via a programmatic system interface to the
160    Unix loader through the system calls
161    <code>dlopen()/dlsym()</code>.</p>
162
163    <p>In the first way the DSO's are usually called <em>shared
164    libraries</em> or <em>DSO libraries</em> and named
165    <code>libfoo.so</code> or <code>libfoo.so.1.2</code>. They
166    reside in a system directory (usually <code>/usr/lib</code>)
167    and the link to the executable program is established at
168    build-time by specifying <code>-lfoo</code> to the linker
169    command. This hard-codes library references into the executable
170    program file so that at start-time the Unix loader is able to
171    locate <code>libfoo.so</code> in <code>/usr/lib</code>, in
172    paths hard-coded via linker-options like <code>-R</code> or in
173    paths configured via the environment variable
174    <code>LD_LIBRARY_PATH</code>. It then resolves any (yet
175    unresolved) symbols in the executable program which are
176    available in the DSO.</p>
177
178    <p>Symbols in the executable program are usually not referenced
179    by the DSO (because it's a reusable library of general code)
180    and hence no further resolving has to be done. The executable
181    program has no need to do anything on its own to use the
182    symbols from the DSO because the complete resolving is done by
183    the Unix loader. (In fact, the code to invoke
184    <code>ld.so</code> is part of the run-time startup code which
185    is linked into every executable program which has been bound
186    non-static). The advantage of dynamic loading of common library
187    code is obvious: the library code needs to be stored only once,
188    in a system library like <code>libc.so</code>, saving disk
189    space for every program.</p>
190
191    <p>In the second way the DSO's are usually called <em>shared
192    objects</em> or <em>DSO files</em> and can be named with an
193    arbitrary extension (although the canonical name is
194    <code>foo.so</code>). These files usually stay inside a
195    program-specific directory and there is no automatically
196    established link to the executable program where they are used.
197    Instead the executable program manually loads the DSO at
198    run-time into its address space via <code>dlopen()</code>. At
199    this time no resolving of symbols from the DSO for the
200    executable program is done. But instead the Unix loader
201    automatically resolves any (yet unresolved) symbols in the DSO
202    from the set of symbols exported by the executable program and
203    its already loaded DSO libraries (especially all symbols from
204    the ubiquitous <code>libc.so</code>). This way the DSO gets
205    knowledge of the executable program's symbol set as if it had
206    been statically linked with it in the first place.</p>
207
208    <p>Finally, to take advantage of the DSO's API the executable
209    program has to resolve particular symbols from the DSO via
210    <code>dlsym()</code> for later use inside dispatch tables
211    <em>etc.</em> In other words: The executable program has to
212    manually resolve every symbol it needs to be able to use it.
213    The advantage of such a mechanism is that optional program
214    parts need not be loaded (and thus do not spend memory) until
215    they are needed by the program in question. When required,
216    these program parts can be loaded dynamically to extend the
217    base program's functionality.</p>
218
219    <p>Although this DSO mechanism sounds straightforward there is
220    at least one difficult step here: The resolving of symbols from
221    the executable program for the DSO when using a DSO to extend a
222    program (the second way). Why? Because "reverse resolving" DSO
223    symbols from the executable program's symbol set is against the
224    library design (where the library has no knowledge about the
225    programs it is used by) and is neither available under all
226    platforms nor standardized. In practice the executable
227    program's global symbols are often not re-exported and thus not
228    available for use in a DSO. Finding a way to force the linker
229    to export all global symbols is the main problem one has to
230    solve when using DSO for extending a program at run-time.</p>
231
232    <p>The shared library approach is the typical one, because it
233    is what the DSO mechanism was designed for, hence it is used
234    for nearly all types of libraries the operating system
235    provides.</p>

239<h2><a name="advantages" id="advantages">Advantages and Disadvantages</a></h2>
240
241    <p>The above DSO based features have the following
242    advantages:</p>
243
244    <ul>
245      <li>The server package is more flexible at run-time because
246      the server process can be assembled at run-time via
247      <code class="directive"><a href="./mod/mod_so.html#loadmodule">LoadModule</a></code>
248      <code>httpd.conf</code> configuration directives instead of
249      <code class="program"><a href="./programs/configure.html">configure</a></code> options at build-time. For instance,
250      this way one is able to run different server instances
251      (standard &amp; SSL version, minimalistic &amp; dynamic
252      version [mod_perl, mod_php], <em>etc.</em>) with only one Apache httpd
253      installation.</li>
254
255      <li>The server package can be easily extended with
256      third-party modules even after installation. This is
257      a great benefit for vendor package maintainers, who can create
258      an Apache httpd core package and additional packages containing
259      extensions like PHP, mod_perl, mod_security, <em>etc.</em></li>
260
261      <li>Easier Apache httpd module prototyping, because with the
262      DSO/<code class="program"><a href="./programs/apxs.html">apxs</a></code> pair you can both work outside the
263      Apache httpd source tree and only need an <code>apxs -i</code>
264      command followed by an <code>apachectl restart</code> to
265      bring a new version of your currently developed module into
266      the running Apache HTTP Server.</li>
267    </ul>
268
269    <p>DSO has the following disadvantages:</p>
270
271    <ul>
272      <li>The server is approximately 20% slower at startup time
273      because of the symbol resolving overhead the Unix loader now
274      has to do.</li>
275
276      <li>The server is approximately 5% slower at execution time
277      under some platforms, because position independent code (PIC)
278      sometimes needs complicated assembler tricks for relative
279      addressing, which are not necessarily as fast as absolute
280      addressing.</li>
281
282      <li>Because DSO modules cannot be linked against other
283      DSO-based libraries (<code>ld -lfoo</code>) on all platforms
284      (for instance a.out-based platforms usually don't provide
285      this functionality while ELF-based platforms do) you cannot
286      use the DSO mechanism for all types of modules. Or in other
287      words, modules compiled as DSO files are restricted to only
288      use symbols from the Apache httpd core, from the C library
289      (<code>libc</code>) and all other dynamic or static libraries
290      used by the Apache httpd core, or from static library archives
291      (<code>libfoo.a</code>) containing position independent code.
292      The only chances to use other code is to either make sure the
293      httpd core itself already contains a reference to it or
294      loading the code yourself via <code>dlopen()</code>.</li>
295    </ul>
