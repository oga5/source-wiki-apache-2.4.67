37    <p>ウェブサーバを効果的に管理するためには、サーバの活動やパフォーマンス、
38    今発生しているかもしれない問題に関するフィードバックを得ることが必要です。
39    Apache HTTP サーバには非常に包括的で柔軟なロギング機能があります。
40    この文書はロギング機能の設定の仕方と、ログに何が書かれているかを
41    理解するための方法を説明します。</p>

54<h2><a name="security" id="security">
55    セキュリティに関する警告</a></h2>
56
57    <p>Apache がログファイルを書いているディレクトリに書き込める人は、
58    ほぼ確実にサーバが起動された uid へのアクセスを手に入れることができます。
59    そして、それは通常は root ユーザです。
60    ちゃんと結果を考えることなく、そのディレクトリへの
61    書き込み権限を与え<em>ない</em>でください。詳しくは
62    <a href="misc/security_tips.html">セキュリティのこつ</a>の文書を
63    読んでください。</p>
64
65    <p>加えて、ログファイルにはクライアントからの情報がそのまま、
66    エスケープされることなく書かれています。ですから、悪意のある
67    クライアントがログファイルに制御文字を挿入することができます。
68    生のログを扱うときは注意してください。</p>

71<h2><a name="errorlog" id="errorlog">エラーログ</a></h2>
72    
73    <table class="related"><tr><th>関連モジュール</th><th>関連ディレクティブ</th></tr><tr><td /><td><ul><li><code class="directive"><a href="./mod/core.html#errorlog">ErrorLog</a></code></li><li><code class="directive"><a href="./mod/core.html#loglevel">LogLevel</a></code></li></ul></td></tr></table>
74
75    <p><code class="directive"><a href="./mod/core.html#errorlog">ErrorLog</a></code> ディレクティブにより
76    名前と場所が決まるサーバのエラーログは、一番重要なログファイルです。
77    Apache の診断情報はここに送られ、リクエストを処理しているときに
78    発生したエラーはすべてここに記録されます。サーバを起動したときや、
79    サーバの動作に問題が起こったときは、一番最初に調べるべき
80    ところです。間違いの詳細や修正方法がそこに書かれていることが
81    よくあります。</p>
82
83    <p>エラーログは普通はファイルに書かれます (通常 Unix システムでは
84    <code>error_log</code>、Windows と OS/2 では <code>error.log</code>)。
85    Unix システムではエラーを <code>syslog</code> や
86    <a href="#piped">パイプでプログラムに送る</a> ことができます。</p>
87
88    <p>エラーログの書式は比較的自由度の高いもので、説明的に書かれています。
89    ただし、いくつかの情報はほとんどのエラーログのエントリにあります。
90    例えば、代表的なものに次のようなメッセージがあります。</p>
91
92    <div class="example"><p><code>
93      [Wed Oct 11 14:32:52 2000] [error] [client 127.0.0.1]
94      client denied by server configuration:
95      /export/home/live/ap/htdocs/test
96    </code></p></div>
97
98    <p>ログエントリの最初の項目はメッセージの日付と時刻です。
99    二つめの項目は報告されているエラーの重要度です。
100    <code class="directive"><a href="./mod/core.html#loglevel">LogLevel</a></code> で重要度のレベルを
101    制限することによりエラーログに送られるエラーの種類を制御することが
102    できます。三つ目の項目はエラーを発生させたクライアントの IP アドレス
103    です。残りはメッセージで、この場合はサーバがクライアントのアクセスを
104    拒否するように設定されている、ということを示しています。
105    サーバはリクエストされた文書の (ウェブのパスではなく) ファイルシステムの
106    パスを報告します。</p>
107
108    <p>非常に広範囲のメッセージがエラーログに現れます。たいていのものは
109    上の例のような感じです。エラーログには CGI スクリプトのデバッグ
110    出力も書かれます。CGI スクリプトが <code>stderr</code> に書いた
111    すべての情報は直接エラーログにコピーされます。</p>
112
113    <p>情報を追加したり削除したりしてエラーログをカスタマイズすることは
114    できません。しかし、リクエストに対するエラーログのエントリは、
115    対応するエントリが<a href="#accesslog">アクセスログ</a>にあります。
116    例えば、上の例のエントリはアクセスログのステータスコード 403 の
117    エントリに対応します。アクセスログはカスタマイズ可能ですので、
118    そちらを使うことによりエラーの状況に関する情報をより多く
119    手に入れることができます。</p>
120
121    <p>テストの最中は、問題が発生しているかどうかを見るために、
122    常にエラーログを監視するのが役に立つ場合がよくあります。
123    Unix システムでは、次のものを使うことができます。</p>
124
125    <div class="example"><p><code>
126      tail -f error_log
127    </code></p></div>

130<h2><a name="accesslog" id="accesslog">アクセスログ</a></h2>
131    
132
133    <table class="related"><tr><th>関連モジュール</th><th>関連ディレクティブ</th></tr><tr><td><ul><li><code class="module"><a href="./mod/mod_log_config.html">mod_log_config</a></code></li><li><code class="module"><a href="./mod/mod_setenvif.html">mod_setenvif</a></code></li></ul></td><td><ul><li><code class="directive"><a href="./mod/mod_log_config.html#customlog">CustomLog</a></code></li><li><code class="directive"><a href="./mod/mod_log_config.html#logformat">LogFormat</a></code></li><li><code class="directive"><a href="./mod/mod_setenvif.html#setenvif">SetEnvIf</a></code></li></ul></td></tr></table>
134
135    <p>サーバアクセスログはサーバが処理をしたすべてのリクエストを
136    記録します。アクセスログの場所と内容は <code class="directive"><a href="./mod/mod_log_config.html#customlog">CustomLog</a></code>
137    ディレクティブにより決まります。ログの内容の選択を簡潔にするために
138    <code class="directive"><a href="./mod/mod_log_config.html#logformat">LogFormat</a></code>
139    ディレクティブを使用することができます。このセクションはアクセスログに
140    情報を記録するためのサーバの設定方法を説明します。</p>
141
142    <p>もちろん、アクセスログに情報を蓄積することはログ管理の
143    始まりに過ぎません。次の段階は有用な統計を取るためにこの情報を
144    解析することです。一般的なログ解析はこの文書の範囲外で、
145    ウェブサーバ自身の仕事というわけでもありません。この話や、
146    ログ解析を行なうアプリケーションの情報を得るには、<a href="http://dmoz.org/Computers/Software/Internet/Site_Management/Log_Analysis/">
147    Open Directory</a> を調べてください。</p>
148
149    <p>いろんなバージョンの Apache httpd が mod_log_config,
150    mod_log_agent, <code>TransferLog</code> ディレクティブといった、
151    他のモジュールやディレクティブを使ってアクセスのロギングを
152    制御してきました。今では、<code class="directive"><a href="./mod/mod_log_config.html#customlog">CustomLog</a></code> がすべての古い
153    ディレクティブの機能を含むようになっています。</p>
154
155    <p>アクセスログの書式は非常に柔軟な設定が可能です。
156    書式は C の printf(1) フォーマット文字列に非常に似た
157    <code class="directive"><a href="./mod/mod_log_config.html#フォーマット文字列">フォーマット文字列</a></code>
158    により指定されます。いくつか次の節で例を示します。
159    フォーマット文字列に使用できる内容の一覧は <a href="mod/mod_log_config.html">mod_log_config の文書</a>
160    を見てください。</p>
161
162    <h3><a name="common" id="common">Common Log Format</a></h3>
163      
164
165      <p>アクセスログのよくある設定に以下のものがあります。</p>
166
167      <div class="example"><p><code>
168        LogFormat "%h %l %u %t \"%r\" %&gt;s %b" common<br />
169         CustomLog logs/access_log common
170      </code></p></div>
171
172      <p>これは、<em>ニックネーム</em> <code>common</code> を定義し、
173      ログのフォーマット文字列の一つと関連付けます。フォーマット文字列は
174      パーセントディレクティブからなり、それぞれのパーセントディレクティブは
175      サーバにどの情報をロギングするかを指示します。フォーマット文字列に
176      文字をそのまま入れることもでき、それらはログの出力に直接コピーされます。
177      そこに引用文字 (<code>"</code>) を書くときは、
178      フォーマット文字列の最後として解釈
179      されることを防ぐためにバックスラッシュでエスケープする必要があります。
180      フォーマット文字列には改行用の "<code>\n</code>"、タブ用の
181      "<code>\t</code>" という特別な制御文字も含めることができます。</p>
182
183      <p><code class="directive"><a href="./mod/mod_log_config.html#customlog">CustomLog</a></code> ディレクティブは
184      既に定義された
185      <em>ニックネーム</em> を使って新しいログファイルを設定します。
186      アクセスログのファイル名はスラッシュで始まらない限り、
187      <code class="directive"><a href="./mod/core.html#serverroot">ServerRoot</a></code> からの相対パスとして
188      扱われます。</p>
189
190      <p>上の設定は Common Log Format (CLF) と呼ばれる形式で
191      ログエントリを書きます。この標準の形式は異なるウェブサーバの多くが
192      生成することができ、多くのログ解析プログラムが読みこむことができます。
193      CLF により生成されたログファイルのエントリは以下のようになります:</p>
194
195      <div class="example"><p><code>
196        127.0.0.1 - frank [10/Oct/2000:13:55:36 -0700] "GET
197        /apache_pb.gif HTTP/1.0" 200 2326
198      </code></p></div>
199
200      <p>このログエントリのそれぞれの部分の意味は以下で説明します。</p>
201
202      <dl>
203	<dt><code>127.0.0.1</code> (<code>%h</code>)</dt>
204
205	<dd>これはサーバへリクエストをしたクライアント (リモートホスト)
206	の IP アドレスです。<code class="directive"><a href="./mod/core.html#hostnamelookups">HostnameLookups</a></code> が
207	<code>On</code> の場合は、サーバはホスト名を調べて、
208	IP アドレスが書かれているところに記録します。しかし、この設定は
209	サーバをかなり遅くするので、あまりお勧めできません。
210	そうではなく、<code class="program"><a href="./programs/logresolve.html">logresolve</a></code> の
211	ようなログの後処理を行なうプログラムでホスト名を調べるのが良いでしょう。
212	ここに報告される IP アドレスは必ずしもユーザが使っているマシンの
213	ものであるとは限りません。ユーザとサーバの間にプロキシサーバが
214	あれば、このアドレスは元のマシンのものではなく、プロキシの
215	アドレスになります。</dd>
216
217	<dt><code>-</code> (<code>%l</code>)</dt>
218
219	<dd>出力中の「ハイフン」は要求された情報が手に入らなかったということを
220	意味します。この場合、取得できなかった情報はクライアントのマシンの
221	<code>identd</code> により決まる RFC 1413 のクライアントの
222	アイデンティティです。この情報はあまり信用することができず、
223	しっかりと管理された内部ネットワークを除いては使うべきではありません。
224	Apache は <code class="directive"><a href="./mod/core.html#identitycheck">IdentityCheck</a></code> が
225	<code>On</code> になっていない限り、この情報を得ようとすらしません。</dd>
226
227	<dt><code>frank</code> (<code>%u</code>)</dt>
228
229	<dd>これは HTTP 認証による、ドキュメントをリクエストした人の
230	ユーザ ID です。CGI スクリプトには通常同じ値が <code>REMOTE_USER</code>
231	環境変数として与えられます。リクエストのステータスコード
232	(以下を参照) が 401 であった場合は、ユーザは認証に失敗しているので、
233	この値は信用できません。ドキュメントがパスワードで保護されていない
234	場合は、この部分は前のものと同じように "<code>-</code>" に
235	なります。</dd>
236
237	<dt><code>[10/Oct/2000:13:55:36 -0700]</code>
238	(<code>%t</code>)</dt>
239
240	<dd>
241	  サーバがリクエストを受け取った時刻です。書式は:
242
243	    <p class="indent">
244	      <code>[day/month/year:hour:minute:second zone]<br />
245	       day = 2*digit<br />
246	       month = 3*letter<br />
247	       year = 4*digit<br />
248	       hour = 2*digit<br />
249	       minute = 2*digit<br />
250	       second = 2*digit<br />
251	       zone = (`+' | `-') 4*digit</code>
252	    </p>
253	  ログのフォーマット文字列に <code>%{format}t</code> を
254	  指定することで、別の形式で時刻を表示させることもできます。
255	  このとき、<code>format</code> は C の標準ライブラリの
256	  <code>strftime(3)</code> の形式になります。
257	</dd>
258
259	<dt><code>"GET /apache_pb.gif HTTP/1.0"</code>
260	(<code>\"%r\"</code>)</dt>
261
262	<dd>クライアントからのリクエストが二重引用符の中に示されています。
263	リクエストには多くの有用な情報があります。まず、この場合クライアントが
264	使ったメソッドは <code>GET</code> です。次に、クライアントは
265	リソース <code>/apache_pb.gif</code> を要求しました。そして、
266	クライアントはプロトコル <code>HTTP/1.0</code> を使用しました。
267	リクエストの各部分を独立にログ収集することもできます。例えば、
268	フォーマット文字列 "<code>%m %U%q %H</code>" は
269	メソッド、パス、クエリ文字列、プロトコルをログ収集し、
270	結局 "<code>%r</code>" とまったく同じ出力になります。</dd>
271
272	<dt><code>200</code> (<code>%&gt;s</code>)</dt>
273
274	<dd>サーバがクライアントに送り返すステータスコードです。
275	この情報は、リクエストが成功応答 (2 で始まるコード) であったか、
276	リダイレクション (3 で始まるコード) であったか、クライアントによる
277	エラー (4 で始まるコード) であったか、サーバのエラー (5 で始まるコード)
278	であったか、を表すので、非常に大切です。ステータスコードの
279	完全なリストは <a href="http://www.w3.org/Protocols/rfc2616/rfc2616.txt">HTTP
280	規格</a> (RFC2616 第 10 節) にあります。</dd>
281
282	<dt><code>2326</code> (<code>%b</code>)</dt>
283
284	<dd>この最後の部分はクライアントに送信されたオブジェクトの、
285	応答ヘッダを除いたサイズを表します。コンテントがクライアントに送られなかった
286	場合は、この値は "<code>-</code>" になります。コンテントが無い場合に
287	"<code>0</code>" をログ収集するには、<code>%b</code> ではなく
288	<code>%B</code> を使ってください。</dd>
289
290      </dl>

293    <h3><a name="combined" id="combined">Combined Log Format</a></h3>
294      
295
296      <p>もう一つのよく使われる書式は Combined Log Format と呼ばれています。
297      以下のようにして使うことができます。</p>
298
299      <div class="example"><p><code>
300        LogFormat "%h %l %u %t \"%r\" %&gt;s %b \"%{Referer}i\"
301        \"%{User-agent}i\"" combined<br />
302         CustomLog log/access_log combined
303      </code></p></div>
304
305      <p>この書式の最初の方は Common Log Format とまったく同じで、最後に
306      二つ追加のエントリがあります。追加のエントリはパーセントディレクティブ
307      <code>%{<em>header</em>}i</code> を使っています。ここで
308      <em>header</em> は HTTP のリクエストヘッダのどれかです。この書式による
309      アクセスログは以下のような感じになります:</p>
310
311      <div class="example"><p><code>
312        127.0.0.1 - frank [10/Oct/2000:13:55:36 -0700] "GET
313        /apache_pb.gif HTTP/1.0" 200 2326
314        "http://www.example.com/start.html" "Mozilla/4.08 [en]
315        (Win98; I ;Nav)"
316      </code></p></div>
317
318      <p>追加のエントリは:</p>
319
320      <dl>
321	<dt><code>"http://www.example.com/start.html"</code>
322	(<code>\"%{Referer}i\"</code>)</dt>
323
324	<dd>"Referer" (意図的な綴り間違い) HTTP リクエストヘッダです。
325	これはクライアントが報告してくる参照元のサイトを表します。
326	(この場合は、<code>/apache_pb.gif</code> にリンクしているか、
327	それを含んでいるページです)。</dd>
328
329	<dt><code>"Mozilla/4.08 [en] (Win98; I ;Nav)"</code>
330	(<code>\"%{User-agent}i\"</code>)</dt>
331
332	<dd>User-Agent HTTP リクエストヘッダです。これはクライアントのブラウザが
333	自分自身のことを報告してくる情報です。</dd>
334      </dl>
335    
336
337    <h3><a name="multiple" id="multiple">複数のアクセスログ</a></h3>
338      
339
340    <p>複数のアクセスログは単に設定ファイルに複数の <code class="directive"><a href="./mod/mod_log_config.html#customlog">CustomLog</a></code>
341    ディレクティブを書くことで作成されます。例えば、以下のディレクティブは
342    三つのアクセスログを作ります。最初のものは基本的な CLF の情報で、
343    二つ目と三つ目は referer とブラウザの情報です。最後二つの
344    <code class="directive"><a href="./mod/mod_log_config.html#customlog">CustomLog</a></code> は
345    <code>ReferLog</code> ディレクティブと
346    <code>AgentLog</code> ディレクティブの効果をまねる方法を示しています。</p>
347
348      <div class="example"><p><code>
349        LogFormat "%h %l %u %t \"%r\" %&gt;s %b" common<br />
350        CustomLog logs/access_log common<br />
351        CustomLog logs/referer_log "%{Referer}i -&gt; %U"<br />
352        CustomLog logs/agent_log "%{User-agent}i"
353      </code></p></div>
354
355    <p>この例は <code class="directive"><a href="./mod/mod_log_config.html#logformat">LogFormat</a></code> で
356    ニックネームを定義する必要がない、
357    ということも示しています。ニックネームの代わりに、
358    <code class="directive"><a href="./mod/mod_log_config.html#customlog">CustomLog</a></code> ディレクティブに
359    直接ログの書式を指定することができます。</p>
360    
361
362    <h3><a name="conditional" id="conditional">条件付きログ</a></h3>
363      
364
365    <p>クライアントのリクエストの特徴に基づいてアクセスログにエントリの
366    一部をロギングしない方が便利なことがあります。これは <a href="env.html">環境変数</a> の補助により簡単に実現できます。まず、
367    リクエストが何らかの条件に合うということを表すために環境変数が
368    設定される必要があります。これは通常は <code class="directive"><a href="./mod/mod_setenvif.html#setenvif">SetEnvIf</a></code> により
369    行なわれます。そして、<code class="directive"><a href="./mod/mod_log_config.html#customlog">CustomLog</a></code> ディレクティブの
370    <code>env=</code> 節を使って環境変数が設定されているリクエストを
371    含めたり排除したりすることができます。いくつか例を挙げます:</p>
372
373      <div class="example"><p><code>
374        # Mark requests from the loop-back interface<br />
375        SetEnvIf Remote_Addr "127\.0\.0\.1" dontlog<br />
376        # Mark requests for the robots.txt file<br />
377        SetEnvIf Request_URI "^/robots\.txt$" dontlog<br />
378        # Log what remains<br />
379        CustomLog logs/access_log common env=!dontlog
380      </code></p></div>
381
382    <p>他の例として、英語を話す人からのリクエストとそれ以外の人からのリクエストを
383    分けたい、という場合を考えてみてください。</p>
384
385      <div class="example"><p><code>
386        SetEnvIf Accept-Language "en" english<br />
387        CustomLog logs/english_log common env=english<br />
388        CustomLog logs/non_english_log common env=!english
389      </code></p></div>
390
391    <p>ここまででは条件付きロギングが非常に強力で柔軟であることを示してきましたが、
392    それがログの内容を制御する唯一の方法というわけではありません。ログファイルは
393    サーバの活動の完全な記録である方がより役に立ちます。単純にログファイルを
394    後処理して、考慮したくないログを削除する方が簡単であることがよくあります。</p>
395    
396  </div><div class="top"><a href="#page-header"><img alt="top" src="./images/up.gif" /></a></div>

398<h2><a name="rotation" id="rotation">ログの交替</a></h2>
399    
400
401    <p>普通の負荷のサーバでさえ、ログファイルに保存される情報の量は
402    膨大になります。アクセスログのファイルは普通 10,000 リクエスト毎に
403    1 MB 以上増えます。ですから、既存のログを移動したり、削除したりして、
404    定期的にログを交替させることが必要になります。これはサーバの実行中には
405    行なえません。というのは、Apache はファイルが open されている間は
406    ずっと古いログファイルに書き続けるからです。
407    新しいログファイルを open できるように、ログファイルが移動されたり
408    削除された後に、サーバを<a href="stopping.html">再起動</a>する
409    必要があります。</p>
410
411    <p><em>優雅な</em> 再起動を行なうことで、サーバは既存のコネクションや
412    処理待ちのコネクションを失うことなく新しいログファイルを open させる
413    ことができます。しかし、これを実現するために、サーバは古いリクエストを
414    扱っている間は古いログファイルに書き続ける必要があります。
415    ですから、再起動の後ではログファイルの処理を始める前に、しばらく待たなければ
416    なりません。単にログを交替させて、ディスクの節約のために古いログを
417    圧縮する普通のシナリオは:</p>
418
419    <div class="example"><p><code>
420      mv access_log access_log.old<br />
421      mv error_log error_log.old<br />
422      apachectl graceful<br />
423      sleep 600<br />
424      gzip access_log.old error_log.old
425    </code></p></div>
426
427    <p>ログの交替をするもう一つの方法は<a href="#piped">パイプ経由のログ</a>を使うもので、次の節で説明されています。</p>
428  </div><div class="top"><a href="#page-header"><img alt="top" src="./images/up.gif" /></a></div>

430<h2><a name="piped" id="piped">パイプ経由のログ</a></h2>
431    
432
433    <p>Apache httpd はエラーログとアクセスログをファイルに直接書く代わりに、
434    パイプを通して別のプログラムに書き出すことができます。
435    この機能により、主サーバにコードを追加することなく
436    ロギングの柔軟性が非常に高まっています。パイプにログを書くためには、
437    単にファイル名をパイプ文字 "<code>|</code>" に置き換え、その続きに
438    標準入力からログのエントリを受けとる実行プログラムの名前を書くだけです。
439    Apache はパイプ経由のログ用のプロセスをサーバの起動時に実行し、
440    サーバの実行中にそのプログラムがクラッシュしたときはそれを再び
441    実行します。(この最後の機能がこの技術が「信頼性のあるパイプ経由のロギング」
442    と呼ばれている理由です。)</p>
443
444    <p>パイプ経由のログ用のプロセスは Apache httpd の親プロセスから起動され、
445    そのプロセスのユーザ ID を継承します。これは、パイプ経由のログ用の
446    プログラムは普通 root として実行されることを意味します。
447    ですから、プログラムを簡単で安全に保つことが非常に重要です。</p>
448
449    <p>パイプ経由のログの重要な利用法は、サーバの再起動なしでログの交替を
450    することです。Apache HTTP サーバにはこのための  <code class="program"><a href="./programs/rotatelogs.html">rotatelogs</a></code> と呼ばれる簡単な
451    プログラムが付属しています。たとえば、24 時間毎にログを交替させるには、
452    以下のものを使うことができます:</p>
453
454    <div class="example"><p><code>
455      CustomLog "|/usr/local/apache/bin/rotatelogs
456      /var/log/access_log 86400" common
457    </code></p></div>
458
459    <p>パイプの先で呼ばれるコマンド全体が引用符で囲まれていることに注目して
460    ください。この例はアクセスログを使っていますが、エラーログにも同じ技術を
461    使うことができます。</p>
462
463    <p>似ているけれど、よりずっと柔軟な
464    <a href="http://www.cronolog.org/">cronolog</a> というログ交替用の
465    プログラムが外部のサイトにあります。</p>
466
467    <p>条件付きロギングと同様、パイプ経由のログは非常に強力な
468    道具ですが、オフラインの後処理のような、より簡単な解決方法があるときは
469    使わない方が良いでしょう。</p>
470  </div><div class="top"><a href="#page-header"><img alt="top" src="./images/up.gif" /></a></div>

472<h2><a name="virtualhosts" id="virtualhosts">バーチャルホスト</a></h2>
473    
474
475    <p>多くの <a href="vhosts/">バーチャルホスト</a> のあるサーバを実行している
476    ときは、ログファイルの扱い方にいくつかの方法があります。
477    まず、単独のホストのみのサーバとまったく同じようにログを使うことができます。
478    ロギングディレクティブを主サーバのコンテキストの
479    <code class="directive"><a href="./mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code> セクションの外に置くことで、
480    すべてのログを同じアクセスログとエラーログにログ収集することができます。
481    この手法では個々のバーチャルホストの統計を簡単にとることはできません。</p>
482
483    <p><code class="directive"><a href="./mod/mod_log_config.html#customlog">CustomLog</a></code> や
484    <code class="directive"><a href="./mod/mod_log_config.html#errorlog">ErrorLog</a></code> ディレクティブが
485    <code class="directive"><a href="./mod/core.html#virtualhost">&lt;VirtualHost&gt;</a></code> の中に
486    置かれた場合は、そのバーチャル
487    ホストへのすべてのリクエストやエラーがそこで指定されたファイルにのみ
488    ログ収集されます。ロギングディレクティブのないバーチャルホストは
489    依然としてリクエストが主サーバのログに送られます。この手法は少ない
490    バーチャルホストに対しては非常に有用ですが、ホストの数が非常に多くなると
491    管理が大変になります。さらに、<a href="vhosts/fd-limits.html">ファイル記述子の限界</a>の問題を起こすことが
492    あります。</p>
493
494    <p>アクセスログには、非常に良い妥協案があります。バーチャルホストの
495    情報をログのフォーマット文字列に加えることで、すべてのホストへの
496    リクエストを同じログにログ収集して、後でログを個々のファイルに分割することが
497    できます。たとえば、以下のディレクティブを見てください。</p>
498
499    <div class="example"><p><code>
500      LogFormat "%v %l %u %t \"%r\" %&gt;s %b"
501      comonvhost<br />
502      CustomLog logs/access_log comonvhost
503    </code></p></div>
504
505    <p><code>%v</code> がリクエストを扱っているバーチャルホストの名前を
506    ログ収集するために使われています。そして、<a href="programs/other.html">split-logfile</a> のようなプログラムを
507    使ってアクセスログを後処理することで、
508    バーチャルホスト毎のファイルにログを分割することができます。</p>
509
510    <p>残念ながら、エラーログには同様の手法はありません。ですから、
511    すべてのバーチャルホストを同じエラーログの中に混ぜるか、
512    バーチャルホスト毎にエラーログを使うかを選ばなければなりません。</p>

531      
532
533      <p><code class="module"><a href="./mod/mod_log_forensic.html">mod_log_forensic</a></code> はクライアントリクエストの

537         デバッグやセキュリティ用のツールとして有効かもしれません。</p>

540    <h3><a name="pidfile" id="pidfile">PID ファイル</a></h3>
541      
542
543      <p>起動時に、Apache は親 httpd プロセスのプロセス ID を
544      <code>logs/httpd.pid</code> に保存します。この
545      ファイル名は <code class="directive"><a href="./mod/mpm_common.html#pidfile">PidFile</a></code> ディレクティブを使って
546      変更することができます。プロセス ID は管理者が親プロセスに
547      シグナルを送ることでデーモンを再起動したり終了させたりするときに
548      使用します。Windows では、代わりに -k コマンドオプションを
549      使ってください。詳しい情報は <a href="stopping.html">終了と
550      再起動</a> のページを見てください。</p>

555

562    <h3><a name="rewritelog" id="rewritelog">リライトログ</a></h3>
563      
564
565      <p><code class="directive"><a href="./mod/mod_rewrite.html#mod_rewrite">mod_rewrite</a></code> の強力で
566      複雑な機能を
567      使っているときは、ほぼいつもデバッグを簡単にするために
568      <code class="directive"><a href="./mod/mod_rewrite.html#rewritelog">RewriteLog</a></code> の使用が
569      必要でしょう。このログファイルにはリライトエンジンがリクエストを
570      書き換える方法の詳細な解析が出力されます。詳しさの度合は <code class="directive"><a href="./mod/mod_rewrite.html#rewriteloglevel">RewriteLogLevel</a></code>
