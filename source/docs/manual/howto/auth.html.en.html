34    <p>Authentication is any process by which you verify that
35    someone is who they claim they are. Authorization is any
36    process by which someone is allowed to be where they want to
37    go, or to have information that they want to have.</p>
38
39    <p>For general access control, see the <a href="access.html">Access
40    Control How-To</a>.</p>

116<h2><a name="introduction" id="introduction">Introduction</a></h2>
117    <p>If you have information on your web site that is sensitive
118    or intended for only a small group of people, the techniques in
119    this article will help you make sure that the people that see
120    those pages are the people that you wanted to see them.</p>
121
122    <p>This article covers the "standard" way of protecting parts
123    of your web site that most of you are going to use.</p>
124
125    <div class="note"><h3>Note:</h3>
126    <p>If your data really needs to be secure, consider using
127    <code class="module"><a href="../mod/mod_ssl.html">mod_ssl</a></code> in addition to any authentication.</p>
128    </div>
129</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>

131<h2><a name="theprerequisites" id="theprerequisites">The Prerequisites</a></h2>
132    <p>The directives discussed in this article will need to go
133    either in your main server configuration file (typically in a
134    <code class="directive"><a href="../mod/core.html#directory">&lt;Directory&gt;</a></code> section), or
135    in per-directory configuration files (<code>.htaccess</code> files).</p>
136
137    <p>If you plan to use <code>.htaccess</code> files, you will
138    need to have a server configuration that permits putting
139    authentication directives in these files. This is done with the
140    <code class="directive"><a href="../mod/core.html#allowoverride">AllowOverride</a></code> directive, which
141    specifies which directives, if any, may be put in per-directory
142    configuration files.</p>
143
144    <p>Since we're talking here about authentication, you will need
145    an <code class="directive"><a href="../mod/core.html#allowoverride">AllowOverride</a></code> directive like the
146    following:</p>
147
148    <pre class="prettyprint lang-config">AllowOverride AuthConfig</pre>
149
150
151    <p>Or, if you are just going to put the directives directly in
152    your main server configuration file, you will of course need to
153    have write permission to that file.</p>
154
155    <p>And you'll need to know a little bit about the directory
156    structure of your server, in order to know where some files are
157    kept. This should not be terribly difficult, and I'll try to
158    make this clear when we come to that point.</p>
159
160    <p>You will also need to make sure that the modules
161    <code class="module"><a href="../mod/mod_authn_core.html">mod_authn_core</a></code> and <code class="module"><a href="../mod/mod_authz_core.html">mod_authz_core</a></code>
162    have either been built into the httpd binary or loaded by the
163    httpd.conf configuration file. Both of these modules provide core
164    directives and functionality that are critical to the configuration
165    and use of authentication and authorization in the web server.</p>
166</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>

168<h2><a name="gettingitworking" id="gettingitworking">Getting it working</a></h2>
169    <p>Here's the basics of password protecting a directory on your
170    server.</p>
171
172    <p>First, you need to create a password file. Exactly how you do
173    this will vary depending on what authentication provider you have
174    chosen. More on that later. To start with, we'll use a text password
175    file.</p>
176
177    <p>This file should be
178    placed somewhere not accessible from the web. This is so that
179    folks cannot download the password file. For example, if your
180    documents are served out of <code>/usr/local/apache/htdocs</code>, you
181    might want to put the password file(s) in
182    <code>/usr/local/apache/passwd</code>.</p>
183
184    <p>To create the file, use the <code class="program"><a href="../programs/htpasswd.html">htpasswd</a></code> utility that
185    came with Apache. This will be located in the <code>bin</code> directory
186    of wherever you installed Apache. If you have installed Apache from
187    a third-party package, it may be in your execution path.</p>
188
189    <p>To create the file, type:</p>
190
191    <div class="example"><p><code>
192      htpasswd -c /usr/local/apache/passwd/passwords rbowen
193    </code></p></div>
194
195    <p><code class="program"><a href="../programs/htpasswd.html">htpasswd</a></code> will ask you for the password, and
196    then ask you to type it again to confirm it:</p>
197
198    <div class="example"><p><code>
199      # htpasswd -c /usr/local/apache/passwd/passwords rbowen<br />
200      New password: mypassword<br />
201      Re-type new password: mypassword<br />
202      Adding password for user rbowen
203    </code></p></div>
204
205    <p>If <code class="program"><a href="../programs/htpasswd.html">htpasswd</a></code> is not in your path, of course
206    you'll have to type the full path to the file to get it to run.
207    With a default installation, it's located at
208    <code>/usr/local/apache2/bin/htpasswd</code></p>
209
210    <p>Next, you'll need to configure the server to request a
211    password and tell the server which users are allowed access.
212    You can do this either by editing the <code>httpd.conf</code>
213    file or using an <code>.htaccess</code> file. For example, if
214    you wish to protect the directory
215    <code>/usr/local/apache/htdocs/secret</code>, you can use the
216    following directives, either placed in the file
217    <code>/usr/local/apache/htdocs/secret/.htaccess</code>, or
218    placed in <code>httpd.conf</code> inside a &lt;Directory
219    "/usr/local/apache/htdocs/secret"&gt; section.</p>
220
221    <pre class="prettyprint lang-config">AuthType Basic
222AuthName "Restricted Files"
223# (Following line optional)
224AuthBasicProvider file
225AuthUserFile "/usr/local/apache/passwd/passwords"
226Require user rbowen</pre>
227
228
229    <p>Let's examine each of those directives individually. The <code class="directive"><a href="../mod/mod_authn_core.html#authtype">AuthType</a></code> directive selects
230    the method that is used to authenticate the user. The most
231    common method is <code>Basic</code>, and this is the method
232    implemented by <code class="module"><a href="../mod/mod_auth_basic.html">mod_auth_basic</a></code>. It is important to be aware,
233    however, that Basic authentication sends the password from the client to
234    the server unencrypted. This method should therefore not be used for
235    highly sensitive data, unless accompanied by <code class="module"><a href="../mod/mod_ssl.html">mod_ssl</a></code>.
236    Apache supports one other authentication method:
237    <code>AuthType Digest</code>. This method is implemented by <code class="module"><a href="../mod/mod_auth_digest.html">mod_auth_digest</a></code> and was intended to be more secure. This is no
238    longer the case and the connection should be encrypted with <code class="module"><a href="../mod/mod_ssl.html">mod_ssl</a></code> instead.</p>
239
240    <p>The <code class="directive"><a href="../mod/mod_authn_core.html#authname">AuthName</a></code> directive sets
241    the <dfn>Realm</dfn> to be used in the authentication. The realm serves
242    two major functions. First, the client often presents this information to
243    the user as part of the password dialog box. Second, it is used by the
244    client to determine what password to send for a given authenticated
245    area.</p>
246
247    <p>So, for example, once a client has authenticated in the
248    <code>"Restricted Files"</code> area, it will automatically
249    retry the same password for any area on the same server that is
250    marked with the <code>"Restricted Files"</code> Realm.
251    Therefore, you can prevent a user from being prompted more than
252    once for a password by letting multiple restricted areas share
253    the same realm. Of course, for security reasons, the client
254    will always need to ask again for the password whenever the
255    hostname of the server changes.</p>
256
257    <p>The <code class="directive"><a href="../mod/mod_auth_basic.html#authbasicprovider">AuthBasicProvider</a></code> is,
258    in this case, optional, since <code>file</code> is the default value
259    for this directive. You'll need to use this directive if you are
260    choosing a different source for authentication, such as
261    <code class="module"><a href="../mod/mod_authn_dbm.html">mod_authn_dbm</a></code> or <code class="module"><a href="../mod/mod_authn_dbd.html">mod_authn_dbd</a></code>.</p>
262
263    <p>The <code class="directive"><a href="../mod/mod_authn_file.html#authuserfile">AuthUserFile</a></code>
264    directive sets the path to the password file that we just
265    created with <code class="program"><a href="../programs/htpasswd.html">htpasswd</a></code>. If you have a large number
266    of users, it can be quite slow to search through a plain text
267    file to authenticate the user on each request. Apache also has
268    the ability to store user information in fast database files.
269    The <code class="module"><a href="../mod/mod_authn_dbm.html">mod_authn_dbm</a></code> module provides the <code class="directive"><a href="../mod/mod_authn_dbm.html#authdbmuserfile">AuthDBMUserFile</a></code> directive. These
270    files can be created and manipulated with the <code class="program"><a href="../programs/dbmmanage.html">dbmmanage</a></code> and <code class="program"><a href="../programs/htdbm.html">htdbm</a></code> programs. Many
271    other types of authentication options are available from third
272    party modules.</p>
273
274    <p>Finally, the <code class="directive"><a href="../mod/mod_authz_core.html#require">Require</a></code>
275    directive provides the authorization part of the process by
276    setting the user that is allowed to access this region of the
277    server. In the next section, we discuss various ways to use the
278    <code class="directive"><a href="../mod/mod_authz_core.html#require">Require</a></code> directive.</p>
279</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>

281<h2><a name="lettingmorethanonepersonin" id="lettingmorethanonepersonin">Letting more than one
282person in</a></h2>
283    <p>The directives above only let one person (specifically
284    someone with a username of <code>rbowen</code>) into the
285    directory. In most cases, you'll want to let more than one
286    person in. This is where the <code class="directive"><a href="../mod/mod_authz_groupfile.html#authgroupfile">AuthGroupFile</a></code> comes in.</p>
287
288    <p>If you want to let more than one person in, you'll need to
289    create a group file that associates group names with a list of
290    users in that group. The format of this file is pretty simple,
291    and you can create it with your favorite editor. The contents
292    of the file will look like this:</p>
293
294   <div class="example"><p><code>
295     GroupName: rbowen dpitts sungo rshersey
296   </code></p></div>
297
298    <p>That's just a list of the members of the group in a long
299    line separated by spaces.</p>
300
301    <p>To add a user to your already existing password file,
302    type:</p>
303
304    <div class="example"><p><code>
305      htpasswd /usr/local/apache/passwd/passwords dpitts
306    </code></p></div>
307
308    <p>You'll get the same response as before, but it will be
309    appended to the existing file, rather than creating a new file.
310    (It's the <code>-c</code> that makes it create a new password
311    file).</p>
312
313    <p>Now, you need to modify your <code>.htaccess</code> file or
314    <code class="directive"><a href="../mod/core.html#directory">&lt;Directory&gt;</a></code> block
315    to look like the following:</p>
316
317    <pre class="prettyprint lang-config">AuthType Basic
318AuthName "By Invitation Only"
319# Optional line:
320AuthBasicProvider file
321AuthUserFile "/usr/local/apache/passwd/passwords"
322AuthGroupFile "/usr/local/apache/passwd/groups"
323Require group GroupName</pre>
324
325
326    <p>Now, anyone that is listed in the group <code>GroupName</code>,
327    and has an entry in the <code>password</code> file, will be let in, if
328    they type the correct password.</p>
329
330    <p>There's another way to let multiple users in that is less
331    specific. Rather than creating a group file, you can just use
332    the following directive:</p>
333
334    <pre class="prettyprint lang-config">Require valid-user</pre>
335
336
337    <p>Using that rather than the <code>Require user rbowen</code>
338    line will allow anyone in that is listed in the password file,
339    and who correctly enters their password.</p>

342<h2><a name="possibleproblems" id="possibleproblems">Possible problems</a></h2>
343    <p>Because of the way that Basic authentication is specified,
344    your username and password must be verified every time you
345    request a document from the server. This is even if you're
346    reloading the same page, and for every image on the page (if
347    they come from a protected directory). As you can imagine, this
348    slows things down a little. The amount that it slows things
349    down is proportional to the size of the password file, because
350    it has to open up that file, and go down the list of users
351    until it gets to your name. And it has to do this every time a
352    page is loaded.</p>
353
354    <p>A consequence of this is that there's a practical limit to
355    how many users you can put in one password file. This limit
356    will vary depending on the performance of your particular
357    server machine, but you can expect to see slowdowns once you
358    get above a few hundred entries, and may wish to consider a
359    different authentication method at that time.</p>
360</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>

362<h2><a name="dbmdbd" id="dbmdbd">Alternate password storage</a></h2>
363
364    <p>Because storing passwords in plain text files has the above
365    problems, you may wish to store your passwords somewhere else, such
366    as in a database.</p>
367
368    <p><code class="module"><a href="../mod/mod_authn_dbm.html">mod_authn_dbm</a></code> and <code class="module"><a href="../mod/mod_authn_dbd.html">mod_authn_dbd</a></code> are two
369    modules which make this possible. Rather than selecting <code><code class="directive"><a href="../mod/mod_auth_basic.html#authbasicprovider">AuthBasicProvider</a></code> file</code>, instead
370    you can choose <code>dbm</code> or <code>dbd</code> as your storage
371    format.</p>
372
373    <p>To select a dbm file rather than a text file, for example:</p>
374
375    <pre class="prettyprint lang-config">&lt;Directory "/www/docs/private"&gt;
376    AuthName "Private"
377    AuthType Basic
378    AuthBasicProvider dbm
379    AuthDBMUserFile "/www/passwords/passwd.dbm"
380    Require valid-user
381&lt;/Directory&gt;</pre>
382
383
384    <p>Other options are available. Consult the
385    <code class="module"><a href="../mod/mod_authn_dbm.html">mod_authn_dbm</a></code> documentation for more details.</p>

388<h2><a name="multprovider" id="multprovider">Using multiple providers</a></h2>
389
390    <p>With the introduction of the new provider based authentication and
391    authorization architecture, you are no longer locked into a single
392    authentication or authorization method. In fact any number of the
393    providers can be mixed and matched to provide you with exactly the
394    scheme that meets your needs. In the following example, both the
395    file and LDAP based authentication providers are being used.</p>
396
397    <pre class="prettyprint lang-config">&lt;Directory "/www/docs/private"&gt;
398    AuthName "Private"
399    AuthType Basic
400    AuthBasicProvider file ldap
401    AuthUserFile "/usr/local/apache/passwd/passwords"
402    AuthLDAPURL ldap://ldaphost/o=yourorg
403    Require valid-user
404&lt;/Directory&gt;</pre>
405
406
407    <p>In this example the file provider will attempt to authenticate
408    the user first. If it is unable to authenticate the user, the LDAP
409    provider will be called. This allows the scope of authentication
410    to be broadened if your organization implements more than
411    one type of authentication store. Other authentication and authorization
412    scenarios may include mixing one type of authentication with a
413    different type of authorization. For example, authenticating against
414    a password file yet authorizing against an LDAP directory.</p>
415
416    <p>Just as multiple authentication providers can be implemented, multiple
417    authorization methods can also be used. In this example both file group
418    authorization as well as LDAP group authorization is being used.</p>
419
420    <pre class="prettyprint lang-config">&lt;Directory "/www/docs/private"&gt;
421    AuthName "Private"
422    AuthType Basic
423    AuthBasicProvider file
424    AuthUserFile "/usr/local/apache/passwd/passwords"
425    AuthLDAPURL ldap://ldaphost/o=yourorg
426    AuthGroupFile "/usr/local/apache/passwd/groups"
427    Require group GroupName
428    Require ldap-group cn=mygroup,o=yourorg
429&lt;/Directory&gt;</pre>
430
431
432    <p>To take authorization a little further, authorization container
433    directives such as
434    <code class="directive"><a href="../mod/mod_authz_core.html#requireall">&lt;RequireAll&gt;</a></code>
435    and
436    <code class="directive"><a href="../mod/mod_authz_core.html#requireany">&lt;RequireAny&gt;</a></code>
437    allow logic to be applied so that the order in which authorization
438    is handled can be completely controlled through the configuration.
439    See <a href="../mod/mod_authz_core.html#logic">Authorization
440    Containers</a> for an example of how they may be applied.</p>
441
442</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>
443<div class="section">
444<h2><a name="beyond" id="beyond">Beyond just authorization</a></h2>
445
446    <p>The way that authorization can be applied is now much more flexible
447    than just a single check against a single data store. Ordering, logic
448    and choosing how authorization will be done is now possible.</p>
449
450    <h3><a name="authandororder" id="authandororder">Applying logic and ordering</a></h3>
451        <p>Controlling how and in what order authorization will be applied
452        has been a bit of a mystery in the past. In Apache 2.2 a provider-based
453        authentication mechanism was introduced to decouple the actual
454        authentication process from authorization and supporting functionality.
455        One of the side benefits was that authentication providers could be
456        configured and called in a specific order which didn't depend on the
457        load order of the auth module itself. This same provider based mechanism
458        has been brought forward into authorization as well. What this means is
459        that the <code class="directive"><a href="../mod/mod_authz_core.html#require">Require</a></code> directive
460        not only specifies which authorization methods should be used, it also
461        specifies the order in which they are called. Multiple authorization
462        methods are called in the same order in which the
463        <code class="directive"><a href="../mod/mod_authz_core.html#require">Require</a></code> directives
464        appear in the configuration.</p>
465
466        <p>With the introduction of authorization container directives
467        such as
468        <code class="directive"><a href="../mod/mod_authz_core.html#requireall">&lt;RequireAll&gt;</a></code>
469        and
470        <code class="directive"><a href="../mod/mod_authz_core.html#requireany">&lt;RequireAny&gt;</a></code>,
471        the configuration also has control over when the
472        authorization methods are called and what criteria determines when
473        access is granted.  See
474        <a href="../mod/mod_authz_core.html#logic">Authorization Containers</a>
475        for an example of how they may be used to express complex
476        authorization logic.</p>
477
478        <p>By default all
479        <code class="directive"><a href="../mod/mod_authz_core.html#require">Require</a></code>
480        directives are handled as though contained within a
481        <code class="directive"><a href="../mod/mod_authz_core.html#requireany">&lt;RequireAny&gt;</a></code>
482        container directive.  In other words, if
483        any of the specified authorization methods succeed, then authorization
484        is granted.</p>
485
486    
487
488    <h3><a name="reqaccessctrl" id="reqaccessctrl">Using authorization providers for access control</a></h3>
489        <p>Authentication by username and password is only part of the
490        story. Frequently you want to let people in based on something
491        other than who they are. Something such as where they are
492        coming from.</p>
493
494        <p>The authorization providers <code>all</code>,
495        <code>env</code>, <code>host</code> and <code>ip</code> let you
496        allow or deny access based on other host based criteria such as
497        host name or ip address of the machine requesting a
498        document.</p>
499
500        <p>The usage of these providers is specified through the
501        <code class="directive"><a href="../mod/mod_authz_core.html#require">Require</a></code> directive.
502        This directive registers the authorization providers
503        that will be called during the authorization stage of the request
504        processing. For example:</p>
505
506        <pre class="prettyprint lang-config">Require ip <var>address</var>
507        </pre>
508
509
510        <p>where <var>address</var> is an IP address (or a partial IP
511        address) or:</p>
512
513        <pre class="prettyprint lang-config">Require host <var>domain_name</var>
514        </pre>
515
516
517        <p>where <var>domain_name</var> is a fully qualified domain name
518        (or a partial domain name); you may provide multiple addresses or
519        domain names, if desired.</p>
520
521        <p>For example, if you have someone spamming your message
522        board, and you want to keep them out, you could do the
523        following:</p>
524
525        <pre class="prettyprint lang-config">&lt;RequireAll&gt;
526    Require all granted
527    Require not ip 10.252.46.165
528&lt;/RequireAll&gt;</pre>
529
530
531        <p>Visitors coming from that address will not be able to see
532        the content covered by this directive. If, instead, you have a
533        machine name, rather than an IP address, you can use that.</p>
534
535        <pre class="prettyprint lang-config">&lt;RequireAll&gt;
536    Require all granted
537    Require not host host.example.com
538&lt;/RequireAll&gt;</pre>
539
540
541        <p>And, if you'd like to block access from an entire domain,
542        you can specify just part of an address or domain name:</p>
543
544        <pre class="prettyprint lang-config">&lt;RequireAll&gt;
545    Require all granted
546    Require not ip 192.168.205
547    Require not host phishers.example.com moreidiots.example
548    Require not host ke
549&lt;/RequireAll&gt;</pre>
550
551
552        <p>Using <code class="directive"><a href="../mod/mod_authz_core.html#requireall">&lt;RequireAll&gt;</a></code>
553        with multiple <code class="directive"><a href="../mod/mod_authz_core.html#require">&lt;Require&gt;</a></code> directives, each negated with <code>not</code>,
554        will only allow access, if all of negated conditions are true. In other words,
555        access will be blocked, if any of the negated conditions fails.</p>
556
557    
558
559    <h3><a name="filesystem" id="filesystem">Access Control backwards compatibility</a></h3>
560        <p>One of the side effects of adopting a provider based mechanism for
561        authentication is that the previous access control directives
562        <code class="directive"><a href="../mod/mod_access_compat.html#order">Order</a></code>,
563        <code class="directive"><a href="../mod/mod_access_compat.html#allow">Allow</a></code>,
564        <code class="directive"><a href="../mod/mod_access_compat.html#deny">Deny</a></code> and
565        <code class="directive"><a href="../mod/mod_access_compat.html#satisfy">Satisfy</a></code> are no longer needed.
566        However to provide backwards compatibility for older configurations, these
567        directives have been moved to the <code class="module"><a href="../mod/mod_access_compat.html">mod_access_compat</a></code> module.</p>
568
569        <div class="warning"><h3>Note</h3>
570        <p>The directives provided by <code class="module"><a href="../mod/mod_access_compat.html">mod_access_compat</a></code> have
571        been deprecated by <code class="module"><a href="../mod/mod_authz_host.html">mod_authz_host</a></code>.
572        Mixing old directives like <code class="directive"><a href="../mod/mod_access_compat.html#order">Order</a></code>, <code class="directive"><a href="../mod/mod_access_compat.html#allow">Allow</a></code> or <code class="directive"><a href="../mod/mod_access_compat.html#deny">Deny</a></code> with new ones like
573        <code class="directive"><a href="../mod/mod_authz_core.html#require">Require</a></code> is technically possible
574        but discouraged. The <code class="module"><a href="../mod/mod_access_compat.html">mod_access_compat</a></code> module was created to support
575        configurations containing only old directives to facilitate the 2.4 upgrade.
576        Please check the <a href="../upgrading.html">upgrading</a> guide for more
577        information.
578        </p>
579        </div>

584<h2><a name="socache" id="socache">Authentication Caching</a></h2>
585    <p>There may be times when authentication puts an unacceptable load
586    on a provider or on your network.  This is most likely to affect users
587    of <code class="module"><a href="../mod/mod_authn_dbd.html">mod_authn_dbd</a></code> (or third-party/custom providers).
588    To deal with this, HTTPD 2.3/2.4 introduces a new caching provider
589    <code class="module"><a href="../mod/mod_authn_socache.html">mod_authn_socache</a></code> to cache credentials and reduce
590    the load on the origin provider(s).</p>
591    <p>This may offer a substantial performance boost to some users.</p>
