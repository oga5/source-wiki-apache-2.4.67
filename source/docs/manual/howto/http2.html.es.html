53<h2><a name="protocol" id="protocol">El protocolo HTTP/2</a></h2>
54    
55
56    <p>HTTP/2 es la evoluci&#243;n del protocolo de la capa de aplicaci&#243;n con m&#225;s
57    &#233;xito, HTTP. Se centra en hacer un uso m&#225;s eficiente de los recursos de red. 
58    No cambia la caracter&#237;stica fundamental de HTTP, la sem&#225;ntica. Todav&#237;a hay 
59    olicitudes, respuestas, cabeceras y todo los elementos t&#237;picos de HTTP/1. As&#237; 
60    que, si ya conoce HTTP/1, tambi&#233;n conoce el 95% de HTTP/2.</p>
61    
62    <p>Se ha escrito mucho sobre HTTP/2 y de c&#243;mo funciona. La norma m&#225;s
63    est&#225;ndar es, por supuesto, su 
64    <a href="https://tools.ietf.org/html/rfc7540">RFC 7540</a> 
65    (<a href="http://httpwg.org/specs/rfc7540.html"> tambi&#233;n disponible en un
66    formato m&#225;s legible, YMMV</a>). As&#237; que, ah&#237; encontrar&#225; toda la especificaci&#243;n 
67    del protocolo.</p>
68
69    <p>Pero, como con todos los RFC, no es ideal como primera lectura. Es mejor
70    entender primero <em>qu&#233;</em> se quiere hacer y despu&#233;s leer el RFC sobre 
71    <em>c&#243;mo</em> hacerlo. Un documento mucho mejor con el que empezar es
72    <a href="https://daniel.haxx.se/http2/">http2 explicado</a>
73    por Daniel Stenberg, el autor de <a href="https://curl.haxx.se">curl</a>. 
74    &#161;Tambi&#233;n est&#225; disponible cada vez en un mayor n&#250;mero lenguajes!</p>
75    
76    <p>Si le parece demasiado largo, o no lo ha leido, hay algunos t&#233;rminos
77    y elementos a tener en cuenta cuando lea este documento:</p>
78    <ul>
79        <li>HTTP/2 es un <strong>protocolo binario</strong>, al contrario que 
80        HTTP 1.1 que es texto plano. La intenci&#243;n para HTTP 1.1 es que sea 
81        legible (por ejemplo capturando el tr&#225;fico de red) mientras que para 
82        HTTP/2 no. M&#225;s informaci&#243;n en el FAQ oficial 
83        <a href="https://http2.github.io/faq/#why-is-http2-binary">&#191;Por qu&#233; es 
84            binario HTTP/2?</a></li>
85
86        <li><strong>h2</strong> es HTTP/2 sobre TLS (negociaci&#243;n de protocolo a 
87        trav&#233;s de ALPN).</li>
88
89        <li><strong>h2c</strong> es HTTP/2 sobre TCP.</li>
90
91        <li>Un <strong>frame</strong> es la unidad m&#225;s peque&#241;a de comunicaci&#243;n 
92        dentro de una conexi&#243;n HTTP/2, que consiste en una cabecera y una secuencia 
93        de octetos de longitud variable estructurada de acuerdo con el tipo de 
94        frame. M&#225;s informaci&#243;n en la documentaci&#243;n oficial 
95        <a href="http://httpwg.org/specs/rfc7540.html#FramingLayer">Secci&#243;n de 
96            Capa de Frame</a>.</li>
97
98        <li>Un <strong>stream</strong> es un flujo bidireccional de frames dentro 
99        de una conexi&#243;n HTTP/2. El concepto correspondiente en HTTP 1.1 es un 
100        intercambio de mensajes de solicitud/respuesta. M&#225;s informaci&#243;n en la 
101        documentaci&#243;n oficial 
102        <a href="http://httpwg.org/specs/rfc7540.html#StreamsLayer">Secci&#243;n Capa 
103            de Stream</a>.</li>
104
105        <li>
106            HTTP/2 es capaz de llevar <strong>m&#250;ltiples streams</strong> de datos
107            sobre la misma conexi&#243;n TCP, evitando la cl&#225;sica solicitud lenta 
108            "head-of-line blocking" de HTTP 1.1 y evitando generar m&#250;ltiples conexiones
109            TCP para cada solicitud/respuesta (KeepAlive parche&#243; el problema en 
110            HTTP 1.1 pero no lo resolvi&#243; completamente).
111      </li>
112    </ul>

115<h2><a name="implementation" id="implementation">HTTP/2 en Apache httpd</a></h2>
116    
117
118    <p>
119        El protocolo HTTP/2 se implementa con su propio m&#243;dulo httpd, llamado 
120        acertadamente <code class="module"><a href="../mod/mod_http2.html">mod_http2</a></code>. Incluye el set completo de 
121        caracter&#237;sticas descritas por el RFC 7540 y soporta HTTP/2 sobre texto 
122        plano (http:), as&#237; como conexiones seguras (https:). La variante de texto
123        plano se llama '<code>h2c</code>', la segura '<code>h2</code>'. Para 
124        <code>h2c</code> permite el modo <em>direct</em>
125        y el <code>Upgrade:</code> a trav&#233;s de una solicitud inicial HTTP/1.
126    </p>
127    
128    <p>
129        Una caracter&#237;stica de HTTP/2 que ofrece capacidades nuevas para 
130        desarrolladores de web es <a href="#push">Server Push</a>. Vea esa secci&#243;n
131         para saber como su aplicaci&#243;n web puede hacer uso de ella.
132     </p>

135<h2><a name="building" id="building">Compilar httpd con soporte HTTP/2</a></h2>
136    
137
138    <p>
139        <code class="module"><a href="../mod/mod_http2.html">mod_http2</a></code> usa la librer&#237;a <a href="https://nghttp2.org">
140        nghttp2</a>como su implementaci&#243;n base. Para compilar 
141        <code class="module"><a href="../mod/mod_http2.html">mod_http2</a></code> necesita al menos la versi&#243;n 1.2.1 de 
142        <code>libnghttp2</code> instalada en su sistema.
143    </p>
144
145    <p>
146        Cuando usted ejecuta <code>./configure</code> en el c&#243;digo fuente de 
147        Apache HTTPD, necesita indicarle '<code>--enable-http2</code>' como una 
148        opci&#243;n adicional para activar la compilaci&#243;n de este m&#243;dulo. Si su 
149        <code>libnghttp2</code> est&#225; ubicado en una ruta no habitual (cualquiera que 
150        sea en su sistema operativo), puede indicar su ubicaci&#243;n con 
151        '<code>--with-nghttp2=&lt;path&gt;</code>' para <code>./configure</code>.
152    </p>
153
154    <p>Aunque puede que eso sirva para la mayor&#237;a, habr&#225; quien prefiera un <code>nghttp2</code> compilado est&#225;ticamente para este m&#243;dulo. Para ellos existe la opci&#243;n <code>--enable-nghttp2-staticlib-deps</code>. Funciona de manera muy similar a como uno debe enlazar openssl est&#225;ticamente para <code class="module"><a href="../mod/mod_ssl.html">mod_ssl</a></code>.</p>
155
156    <p>Hablando de SSL, necesita estar al tanto de que la mayor&#237;a de los navegadores hablan HTTP/2 solo con URLs <code>https:</code>. As&#237; que necesita un servidor con soporte SSL. Pero no solo eso, necesitar&#225; una librer&#237;a SSL que de soporte a la extensi&#243;n <code>ALPN</code>. Si usa OpenSSL, necesita al menos la versi&#243;n 1.0.2.</p>

159<h2><a name="basic-config" id="basic-config">Configuraci&#243;n b&#225;sica</a></h2>
160    
161
162    <p>Cuando tiene un <code>httpd</code> compilado con <code class="module"><a href="../mod/mod_http2.html">mod_http2</a></code> necesita una configuraci&#243;n b&#225;sica para activarlo. Lo primero, como con cualquier otro m&#243;dulo de Apache, es que necesita cargarlo:</p>
163    
164    <pre class="prettyprint lang-config">LoadModule http2_module modules/mod_http2.so</pre>
165
166    
167    <p>La segunda directiva que necesita a&#241;adir a la configuraci&#243;n de su servidor es:</p>
168
169    <pre class="prettyprint lang-config">Protocols h2 http/1.1</pre>
170
171    
172    <p>Esto permite h2, la variante segura, para ser el protocolo preferido de las conexiones en su servidor. Cuando quiera habilitar todas las variantes de HTTP/2, entonces simplemente configure:</p>
173
174    <pre class="prettyprint lang-config">Protocols h2 h2c http/1.1</pre>
175
176
177    <p>Dependiendo de d&#243;nde pone esta directiva, afecta a todas las conexiones o solo a las de ciertos host virtuales. La puede anidar, como en:</p>
178
179    <pre class="prettyprint lang-config">Protocols http/1.1
180&lt;VirtualHost ...&gt;
181    ServerName test.example.org
182    Protocols h2 http/1.1
183&lt;/VirtualHost&gt;</pre>
184
185
186    <p>Esto solo permite HTTP/1, excepto conexiones SSL hacia <code>test.example.org</code> que ofrecen HTTP/2.</p>
187
188    <div class="note"><h3>Escoger un SSLCipherSuite seguro</h3>
189     <p>Es necesario configurar <code class="directive"><a href="../mod/mod_ssl.html#sslciphersuite">SSLCipherSuite</a></code> con una suite segura de cifrado TLS. La versi&#243;n actual de mod_http2 no fuerza ning&#250;n cifrado pero la mayor&#237;a de los clientes si lo hacen. Encaminar un navegador hacia un servidor con <code>h2</code> activado con una suite inapropiada de cifrados forzar&#225; al navegador a rehusar e intentar conectar por HTTP 1.1. Esto es un error com&#250;n cuando se configura httpd con HTTP/2 por primera vez, &#161;as&#237; que por favor tenga en cuenta que debe evitar largas sesiones de depuraci&#243;n! Si quiere estar seguro de la suite de cifrados que escoja, por favor evite los listados en la <a href="http://httpwg.org/specs/rfc7540.html#BadCipherSuites">Lista Negra de TLS para HTTP/2</a>.</p>
190    </div>
191
192    <p>El orden de los protocolos mencionados tambi&#233;n es relevante. Por defecto, el primero es el protocolo preferido. Cuando un cliente ofrece m&#250;ltiples opciones, la que est&#233; m&#225;s a la izquierda ser&#225; la escogida. En</p>
193    <pre class="prettyprint lang-config">Protocols http/1.1 h2</pre>
194
195    
196    <p>el protocolo preferido es HTTP/1 y siempre ser&#225; seleccionado a menos que el cliente <em>s&#243;lo</em> soporte h2. Puesto que queremos hablar HTTP/2 con clientes que lo soporten, el orden correcto es:</p>
197    
198    <pre class="prettyprint lang-config">Protocols h2 h2c http/1.1</pre>
199
200
201    <p>Hay algo m&#225;s respecto al orden: el cliente tambi&#233;n tiene sus propias preferencias. Si quiere, puede configurar su servidor para seleccionar el protocolo preferido por el cliente:</p>
202
203    <pre class="prettyprint lang-config">ProtocolsHonorOrder Off</pre>
204
205
206    <p>Hace que el orden en que <em>usted</em> escribi&#243; los Protocols sea irrelevante y s&#243;lo el orden de preferencia del cliente ser&#225; decisorio.</p>
207
208    <p>Una &#250;ltima cosa: cuando usted configura los protocolos no se comprueba si son correctos o est&#225;n bien escritos. Puede mencionar protocolos que no existen, as&#237; que no hay necesidad de proteger <code class="directive"><a href="../mod/core.html#protocols">Protocols</a></code> con ning&#250;n <code class="directive"><a href="../mod/core.html#ifmodule">&lt;IfModule&gt;</a></code> de comprobaci&#243;n.</p>
209
210    <p>Para m&#225;s consejos avanzados de configuraci&#243;n, vea la <a href="../mod/mod_http2.html#dimensioning">
211    secci&#243;n de m&#243;dulos sobre dimensionamiento</a> y <a href="../mod/mod_http2.html#misdirected">
212    como gestionar multiples hosts con el mismo certificado</a>.</p>

215<h2><a name="mpm-config" id="mpm-config">Configuraci&#243;n MPM</a></h2>
216    
217    
218    <p>HTTP/2 est&#225; soportado en todos los m&#243;dulos de multi-proceso que se ofrecen con httpd. Aun as&#237;, si usa el mpm <code class="module"><a href="../mod/prefork.html">prefork</a></code>, habr&#225;  restricciones severas.</p>
219
220    <p>En <code class="module"><a href="../mod/prefork.html">prefork</a></code>, <code class="module"><a href="../mod/mod_http2.html">mod_http2</a></code> solo procesar&#225; una solicitud cada vez por conexi&#243;n. Pero los clientes, como los navegadores, enviar&#225;n muchas solicitudes al mismo tiempo. Si una de ellas tarda mucho en procesarse (o hace un sondeo que dura m&#225;s de la cuenta), las otras solicitudes se quedar&#225;n atascadas.</p>
221
222    <p><code class="module"><a href="../mod/mod_http2.html">mod_http2</a></code> no evitar&#225; este l&#237;mite por defecto. El motivo es que <code class="module"><a href="../mod/prefork.html">prefork</a></code> hoy en d&#237;a solo se escoge si ejecuta motores de proceso que no est&#225;n preparados para multi-hilo, p.ej. fallar&#225; con m&#225;s de una solicitud.</p>
223
224    <p>Si su configuraci&#243;n lo soporta, hoy en d&#237;a <code class="module"><a href="../mod/event.html">event</a></code> es el mejor mpm que puede usar.</p>
225    
226    <p>Si realmente est&#225; obligado a usar <code class="module"><a href="../mod/prefork.html">prefork</a></code> y quiere multiples solicitudes, puede configurar la directiva <code class="directive"><a href="../mod/mod_http2.html#h2minworkers">H2MinWorkers</a></code> para hacerlo posible. Sin embargo, si esto falla, es bajo su cuenta y riesgo.</p>

264<h2><a name="push" id="push">Server Push</a></h2>
265    
266    
267    <p>El protocolo HTTP/2 permite al servidor hacer PUSH de respuestas a un cliente que nunca las solicit&#243;. El tono de la conversaci&#243;n es: "Aqu&#237; tiene una solicitud que nunca envi&#243; y la respuesta llegar&#225; pronto..."</p>
268
269    <p>Pero hay restricciones: el cliente puede deshabilitar esta caracter&#237;stica y el servidor entonces solo podr&#225; hacer PUSH en una solicitud que hizo previamente del cliente.</p>
270
271    <p>La intenci&#243;n es permitir al servidor enviar recursos que el cliente seguramente vaya a necesitar, p. ej. un recurso css o javascript que pertenece a una p&#225;gina html que el cliente solicit&#243;, un grupo de im&#225;genes a las que se hace referencia en un css, etc.</p>
272
273    <p>La ventaja para el cliente es que ahorra tiempo para solicitudes que pueden tardar desde unos pocos milisegundos a medio segundo, dependiendo de la distancia entre el cliente y el servidor. La desventaja es que el cliente puede recibir cosas que ya tiene en su cache. Por supuesto que HTTP/2 soporta cancelaci&#243;n previa de tales solicitudes, pero aun as&#237; se malgastan recursos.</p>
274
275    <p>Resumiendo: no hay una estrategia mejor sobre c&#243;mo usar esta caracter&#237;stica de HTTP/2 y todo el mundo est&#225; experimentando con ella. As&#237; que, &#191;c&#243;mo experimenta usted con ella en Apache httpd?</p>
276
277    <p><code class="module"><a href="../mod/mod_http2.html">mod_http2</a></code> busca e inspecciona las cabeceras de respuesta 
278    <code>Link</code> con cierto formato:</p>
279
280    <pre class="prettyprint lang-config">Link &lt;/xxx.css&gt;;rel=preload, &lt;/xxx.js&gt;; rel=preload</pre>
281
282
283    <p>
284        Si la conexi&#243;n soporta PUSH, estos dos recursos se enviar&#225;n al cliente. 
285        Como desarrollador web, puede configurar estas cabeceras o bien 
286        directamente en la respuesta de su aplicaci&#243;n o configurar su servidor con:
287    </p>
288
289    <pre class="prettyprint lang-config">&lt;Location /xxx.html&gt;
290    Header add Link "&lt;/xxx.css&gt;;rel=preload"
291    Header add Link "&lt;/xxx.js&gt;;rel=preload"
292&lt;/Location&gt;</pre>
293
294
295    <p>Si quiere usar enlaces con <code>preload</code> sin activar un PUSH, puede
296    usar el par&#225;metro <code>nopush</code>, como en:</p>
297
298    <pre class="prettyprint lang-config">Link &lt;/xxx.css&gt;;rel=preload;nopush</pre>
299
300
301    <p>o puede desactivar PUSH para su servidor por completo con la directiva </p>
302
303    <pre class="prettyprint lang-config">H2Push Off</pre>
304
305
306    <p>Y hay m&#225;s:</p>
307
308    <p>
309        El m&#243;dulo mantiene un registro de lo que se ha enviado con PUSH para cada
310        conexi&#243;n (hashes de URLs, b&#225;sicamente) y no har&#225; PUSH del mismo recurso dos
311        veces. Cuando la conexi&#243;n se cierra, la informaci&#243;n es descartada.
312    </p>
313
314    <p>
315        Hay gente pensando c&#243;mo un cliente puede decirle al servidor lo que ya
316        tiene, para evitar los PUSH de esos elementos, pero eso algo muy
317        experimental ahora mismo.
318    </p>
319
320    <p>Otro borrador experimental que ha sido implementado en 
321    <code class="module"><a href="../mod/mod_http2.html">mod_http2</a></code> es el <a href="https://tools.ietf.org/html/draft-ruellan-http-accept-push-policy-00"> Campo de Cabecera
322    Accept-Push-Policy</a> en la que un cliente puede, para cada solicitud, definir 
323    qu&#233; tipo de PUSH acepta.</p>
324
325    <p>
326        Puede que PUSH no siempre lance la peticion/respuesta/funcionamiento que
327        uno espera. Hay varios estudios sobre este tema en internet, que explican
328        el beneficio y las debilidades de como diferentes funcionalidades del
329        cliente y de la red influyen en el resultado.
330        Por Ejemplo, que un servidor haga "PUSH" de recursos, no significa que el 
331        navegador vaya a usar dichos datos.
332    </p>
333    <p>
334        Lo m&#225;s importante que influye en la respuesta que se env&#237;a, es la solicitud
335        que se simul&#243;. La url de solicitud de un PUSH es dada por la aplicaci&#243;n,
336        pero &#191;de donde vienen las cabeceras de la petici&#243;n? por ejemplo si el PUSH
337        pide una cabecera <code>accept-language</code> y si es as&#237;, &#191;con qu&#233; valor?
338    </p>
339    <p>Httpd mirar&#225; la petici&#243;n original (la que origin&#243; el PUSH) y copiar&#225; las
340        siguientes cabeceras a las peticiones PUSH:
341        <code>user-agent</code>, <code>accept</code>, <code>accept-encoding</code>,
342        <code>accept-language</code>, <code>cache-control</code>.
343    </p>
344    <p>
345        Todas las otras cabeceras son ignorados. Las cookies tampoco ser&#225;n copiadas.
346        Impulsar los recursos que requieren una cookie para estar presente no
347        funcionar&#225;. Esto puede ser una cuesti&#243;n de debate. Pero a menos que esto se
348        discuta m&#225;s claramente con el navegador, evitemos el exceso de precauci&#243;n y
349        no expongamos las cookies donde podr&#237;an o no ser visibles.
350    </p>

353<div class="section">
354<h2><a name="earlyhints" id="earlyhints">"Early Hints"</a></h2>
355    
356
357    <p>Una alternativa de "Pushear" recursos es mandar una cabecera 
358        <code>Link</code> al cliente antes que la respuesta est&#233; lista. Esto usa
359        una caracteristica de HTTP que se llama  "Early Hints" y est&#225; descrita en
360        la <a href="https://tools.ietf.org/html/rfc8297">RFC 8297</a>.</p>
361    <p>Para poder usar esto, necesita habilitarlo explicitamente en el servidor 
362    via</p>
363    
364    <pre class="prettyprint lang-config">H2EarlyHints on</pre>
365
366    
367    <p>(No est&#225; habilitado por defecto ya q ue algunos navegadores m&#225;s antiguos 
368        se caen con dichas  respuestas.)
369    </p>
370
371    <p>si esta funcionalidad esta activada, puede usar la directiva 
372        <code class="directive"><a href="../mod/mod_http2.html#h2pushresource">H2PushResource</a></code> para que lance 
373        "Early hints" y recursos mediante push:
374    </p>
375    <pre class="prettyprint lang-config">&lt;Location /xxx.html&gt;
376    H2PushResource /xxx.css
377    H2PushResource /xxx.js
378&lt;/Location&gt;</pre>
379
380    <p>
381        Esto lanzar&#225; una respuesta <code>"103 Early Hints"</code> a un cliente 
382        tan pronto como el servidor <em>comience</em> a procesar la solicitud. 
383        Esto puede ser mucho antes que en el momento en que se determinaron los 
384        primeros encabezados de respuesta, dependiendo de su aplicaci&#243;n web.
385    </p>
386
387    <p>
388        Si la directiva <code class="directive"><a href="../mod/mod_http2.html#h2push">H2Push</a></code> est&#225; 
389        habilitada, esto comenzar&#225; el PUSH justo despu&#233;s de la respuesta 103.
390        Sin embargo, si la directiva <code class="directive"><a href="../mod/mod_http2.html#h2push">H2Push</a></code> est&#225; dehabilitada, la respuesta 103 se le enviar&#225; al cliente.
391    </p>
