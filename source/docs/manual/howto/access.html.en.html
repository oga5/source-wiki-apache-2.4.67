31    <p>Access control refers to any means of controlling access to any
32    resource. This is separate from <a href="auth.html">authentication and authorization</a>.</p>

42<h2><a name="related" id="related">Related Modules and Directives</a></h2>
43
44    <p>Access control can be done by several different modules. The most
45    important of these are <code class="module"><a href="../mod/mod_authz_core.html">mod_authz_core</a></code> and
46    <code class="module"><a href="../mod/mod_authz_host.html">mod_authz_host</a></code>. Also discussed in this document
47    is access control using <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code>.</p>
48
49</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>
50<div class="section">
51<h2><a name="host" id="host">Access control by host</a></h2>
52    <p>
53    If you wish to restrict access to portions of your site based on the
54    host address of your visitors, this is most easily done using
55    <code class="module"><a href="../mod/mod_authz_host.html">mod_authz_host</a></code>.
56    </p>
57
58    <p>The <code class="directive"><a href="../mod/mod_authz_core.html#require">Require</a></code>
59    provides a variety of different ways to allow or deny access to
60    resources. In conjunction with the <code class="directive"><a href="../mod/mod_authz_core.html#requireall">RequireAll</a></code>, <code class="directive"><a href="../mod/mod_authz_core.html#requireany">RequireAny</a></code>, and <code class="directive"><a href="../mod/mod_authz_core.html#requirenone">RequireNone</a></code> directives, these
61    requirements may be combined in arbitrarily complex ways, to enforce
62    whatever your access policy happens to be.</p>
63
64    <div class="warning"><p>
65    The <code class="directive"><a href="../mod/mod_access_compat.html#allow">Allow</a></code>,
66    <code class="directive"><a href="../mod/mod_access_compat.html#deny">Deny</a></code>, and
67    <code class="directive"><a href="../mod/mod_access_compat.html#order">Order</a></code> directives,
68    provided by <code class="module"><a href="../mod/mod_access_compat.html">mod_access_compat</a></code>, are deprecated and
69    will go away in a future version. You should avoid using them, and
70    avoid outdated tutorials recommending their use.
71    </p></div>
72
73    <p>The usage of these directives is:</p>
74
75    <pre class="prettyprint lang-config">Require host address
76Require ip ip.address</pre>
77
78
79    <p>In the first form, <var>address</var> is a fully qualified
80    domain name (or a partial domain name); you may provide multiple
81    addresses or domain names, if desired.</p>
82
83    <p>In the second form, <var>ip.address</var> is an IP address, a
84    partial IP address, a network/netmask pair, or a network/nnn CIDR
85    specification. Either IPv4 or IPv6 addresses may be used.</p>
86
87    <p>See <a href="../mod/mod_authz_host.html#requiredirectives">the
88    mod_authz_host documentation</a> for further examples of this
89    syntax.</p>
90
91    <p>You can insert <code>not</code> to negate a particular requirement.
92    Note, that since a <code>not</code> is a negation of a value, it cannot 
93    be used by itself to allow or deny a request, as <em>not true</em>
94    does not constitute <em>false</em>. Thus, to deny a visit using a negation,
95    the block must have one element that evaluates as true or false.
96    For example, if you have someone spamming your message
97    board, and you want to keep them out, you could do the
98    following:</p>
99
100    <pre class="prettyprint lang-config">&lt;RequireAll&gt;
101    Require all granted
102    Require not ip 10.252.46.165
103&lt;/RequireAll&gt;</pre>
104
105
106    <p>Visitors coming from that address (<code>10.252.46.165</code>)
107    will not be able to see the content covered by this directive. If, 
108    instead, you have a machine name, rather than an IP address, you 
109    can use that.</p>
110
111    <pre class="prettyprint lang-config">Require not host <var>host.example.com</var>
112    </pre>
113
114
115    <p>And, if you'd like to block access from an entire domain,
116    you can specify just part of an address or domain name:</p>
117
118    <pre class="prettyprint lang-config">Require not ip 192.168.205
119Require not host phishers.example.com moreidiots.example
120Require not host gov</pre>
121
122
123    <p>Use of the <code class="directive"><a href="../mod/mod_authz_core.html#requireall">RequireAll</a></code>, <code class="directive"><a href="../mod/mod_authz_core.html#requireany">RequireAny</a></code>, and <code class="directive"><a href="../mod/mod_authz_core.html#requirenone">RequireNone</a></code> directives may be
124    used to enforce more complex sets of requirements.</p>

128<h2><a name="env" id="env">Access control by arbitrary variables</a></h2>
129
130    <p>Using the <code class="directive"><a href="../mod/core.html#if">&lt;If&gt;</a></code>,
131    you can allow or deny access based on arbitrary environment
132    variables or request header values. For example, to deny access
133    based on user-agent (the browser type) you might do the
134    following:</p>
135
136    <pre class="prettyprint lang-config">&lt;If "%{HTTP_USER_AGENT} == 'BadBot'"&gt;
137    Require all denied
138&lt;/If&gt;</pre>
139
140
141    <p>Using the <code class="directive"><a href="../mod/mod_authz_core.html#require">Require</a></code>
142    <code>expr</code> syntax, this could also be written as:</p>
143
144
145    <pre class="prettyprint lang-config">Require expr %{HTTP_USER_AGENT} != 'BadBot'</pre>
146
147
148    <div class="note"><h3>Warning:</h3>
149    <p>Access control by <code>User-Agent</code> is an unreliable technique,
150    since the <code>User-Agent</code> header can be set to anything at all,
151    at the whim of the end user.</p>
152    </div>
153
154    <p>See <a href="../expr.html">the expressions document</a> for a
155    further discussion of what expression syntaxes and variables are
156    available to you.</p>

160<h2><a name="rewrite" id="rewrite">Access control with mod_rewrite</a></h2>
161
162    <p>The <code>[F]</code> <code class="directive"><a href="../mod/mod_rewrite.html#rewriterule">RewriteRule</a></code> flag causes a 403 Forbidden
163    response to be sent. Using this, you can deny access to a resource based
164    on arbitrary criteria.</p>
165
166    <p>For example, if you wish to block access to a resource between 8pm
167    and 7am, you can do this using <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code>.</p>
168
169    <pre class="prettyprint lang-config">RewriteEngine On
170RewriteCond "%{TIME_HOUR}" "&gt;=20" [OR]
171RewriteCond "%{TIME_HOUR}" "&lt;07"
172RewriteRule "^/fridge"     "-" [F]</pre>
173
174
175    <p>This will return a 403 Forbidden response for any request after 8pm
176    or before 7am. This technique can be used for any criteria that you wish
177    to check. You can also redirect, or otherwise rewrite these requests, if
178    that approach is preferred.</p>
179
180    <p>The <code class="directive"><a href="../mod/core.html#if">&lt;If&gt;</a></code> directive,
181    added in 2.4, replaces many things that <code class="module"><a href="../mod/mod_rewrite.html">mod_rewrite</a></code> has
182    traditionally been used to do, and you should probably look there first
183    before resorting to mod_rewrite.</p>
