51    <p>This article deals with Server Side Includes, usually called
52    simply SSI. In this article, I'll talk about configuring your
53    server to permit SSI, and introduce some basic SSI techniques
54    for adding dynamic content to your existing HTML pages.</p>
55
56    <p>In the latter part of the article, we'll talk about some of
57    the somewhat more advanced things that can be done with SSI,
58    such as conditional statements in your SSI directives.</p>

62<h2><a name="what" id="what">What are SSI?</a></h2>
63
64    <p>SSI (Server Side Includes) are directives that are placed in
65    HTML pages, and evaluated on the server while the pages are
66    being served. They let you add dynamically generated content to
67    an existing HTML page, without having to serve the entire page
68    via a CGI program, or other dynamic technology.</p>
69
70    <p>For example, you might place a directive into an existing HTML
71    page, such as:</p>
72
73    <div class="example"><p><code>
74    &lt;!--#echo var="DATE_LOCAL" --&gt;
75    </code></p></div>
76
77    <p>And, when the page is served, this fragment will be evaluated and replaced with its value:</p>
78
79    <div class="example"><p><code>
80    Tuesday, 15-Jan-2013 19:28:54 EST
81    </code></p></div>
82
83    <p>The decision of when to use SSI, and when to have your page
84    entirely generated by some program, is usually a matter of how
85    much of the page is static, and how much needs to be
86    recalculated every time the page is served. SSI is a great way
87    to add small pieces of information, such as the current time - shown
88    above.  But if a majority of your page is being generated at the time
89    that it is served, you need to look for some other solution.</p>
90</div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>

92<h2><a name="configuring" id="configuring">Configuring your server to permit SSI</a></h2>
93
94
95    <p>To permit SSI on your server, you must have the following
96    directive either in your <code>httpd.conf</code> file, or in a
97    <code>.htaccess</code> file:</p>
98<pre class="prettyprint lang-config">Options +Includes</pre>
99
100
101    <p>This tells Apache that you want to permit files to be parsed
102    for SSI directives.  Note that most configurations contain
103    multiple <code class="directive"><a href="../mod/core.html#options">Options</a></code> directives
104    that can override each other.  You will probably need to apply the
105    <code>Options</code> to the specific directory where you want SSI
106    enabled in order to assure that it gets evaluated last.</p>
107
108    <p>Not just any file is parsed for SSI directives. You have to
109    tell Apache which files should be parsed. There are two ways to
110    do this. You can tell Apache to parse any file with a
111    particular file extension, such as <code>.shtml</code>, with
112    the following directives:</p>
113<pre class="prettyprint lang-config">AddType text/html .shtml
114AddOutputFilter INCLUDES .shtml</pre>
115
116
117    <p>One disadvantage to this approach is that if you wanted to
118    add SSI directives to an existing page, you would have to
119    change the name of that page, and all links to that page, in
120    order to give it a <code>.shtml</code> extension, so that those
121    directives would be executed.</p>
122
123    <p>The other method is to use the <code class="directive"><a href="../mod/mod_include.html#xbithack">XBitHack</a></code> directive:</p>
124<pre class="prettyprint lang-config">XBitHack on</pre>
125
126
127    <p><code class="directive"><a href="../mod/mod_include.html#xbithack">XBitHack</a></code>
128    tells Apache to parse files for SSI
129    directives if they have the execute bit set. So, to add SSI
130    directives to an existing page, rather than having to change
131    the file name, you would just need to make the file executable
132    using <code>chmod</code>.</p>
133<div class="example"><p><code>
134        chmod +x pagename.html
135</code></p></div>
136
137    <p>A brief comment about what not to do. You'll occasionally
138    see people recommending that you just tell Apache to parse all
139    <code>.html</code> files for SSI, so that you don't have to
140    mess with <code>.shtml</code> file names. These folks have
141    perhaps not heard about <code class="directive"><a href="../mod/mod_include.html#xbithack">XBitHack</a></code>. The thing to
142    keep in mind is that, by doing this, you're requiring that
143    Apache read through every single file that it sends out to
144    clients, even if they don't contain any SSI directives. This
145    can slow things down quite a bit, and is not a good idea.</p>
146
147    <p>Of course, on Windows, there is no such thing as an execute
148    bit to set, so that limits your options a little.</p>
149
150    <p>In its default configuration, Apache does not send the last
151    modified date or content length HTTP headers on SSI pages,
152    because these values are difficult to calculate for dynamic
153    content. This can prevent your document from being cached, and
154    result in slower perceived client performance. There are two
155    ways to solve this:</p>
156
157    <ol>
158      <li>Use the <code>XBitHack Full</code> configuration. This
159      tells Apache to determine the last modified date by looking
160      only at the date of the originally requested file, ignoring
161      the modification date of any included files.</li>
162
163      <li>Use the directives provided by
164      <code class="module"><a href="../mod/mod_expires.html">mod_expires</a></code> to set an explicit expiration
165      time on your files, thereby letting browsers and proxies
166      know that it is acceptable to cache them.</li>
167    </ol>

170<h2><a name="basic" id="basic">Basic SSI directives</a></h2>
171
172    <p>SSI directives have the following syntax:</p>
173<div class="example"><p><code>
174        &lt;!--#function attribute=value attribute=value ... --&gt;
175</code></p></div>
176
177    <p>It is formatted like an HTML comment, so if you don't have
178    SSI correctly enabled, the browser will ignore it, but it will
179    still be visible in the HTML source. If you have SSI correctly
180    configured, the directive will be replaced with its
181    results.</p>
182
183    <p>The function can be one of a number of things, and we'll talk
184    some more about most of these in the next installment of this
185    series. For now, here are some examples of what you can do with
186    SSI</p>
187
188<h3><a name="todaysdate" id="todaysdate">Today's date</a></h3>
189
190<div class="example"><p><code>
191        &lt;!--#echo var="DATE_LOCAL" --&gt;
192</code></p></div>
193
194    <p>The <code>echo</code> function just spits out the value of a
195    variable. There are a number of standard variables, which
196    include the whole set of environment variables that are
197    available to CGI programs. Also, you can define your own
198    variables with the <code>set</code> function.</p>
199
200    <p>If you don't like the format in which the date gets printed,
201    you can use the <code>config</code> function, with a
202    <code>timefmt</code> attribute, to modify that formatting.</p>
203
204<div class="example"><p><code>
205        &lt;!--#config timefmt="%A %B %d, %Y" --&gt;<br />
206        Today is &lt;!--#echo var="DATE_LOCAL" --&gt;
207</code></p></div>
208
209
210<h3><a name="lastmodified" id="lastmodified">Modification date of the file</a></h3>
211
212<div class="example"><p><code>
213        This document last modified &lt;!--#flastmod file="index.html" --&gt;
214</code></p></div>
215
216    <p>This function is also subject to <code>timefmt</code> format
217    configurations.</p>
218

220<h3><a name="cgi" id="cgi">Including the results of a CGI program</a></h3>
221
222    <p>This is one of the more common uses of SSI - to output the
223    results of a CGI program, such as everybody's favorite, a ``hit
224    counter.''</p>
225
226<div class="example"><p><code>
227        &lt;!--#include virtual="/cgi-bin/counter.pl" --&gt;
228</code></p></div>

233<h2><a name="additionalexamples" id="additionalexamples">Additional examples</a></h2>
234
235
236    <p>Following are some specific examples of things you can do in
237    your HTML documents with SSI.</p>
238
239<h3><a name="docmodified" id="docmodified">When was this document
240modified?</a></h3>
241
242    <p>Earlier, we mentioned that you could use SSI to inform the
243    user when the document was most recently modified. However, the
244    actual method for doing that was left somewhat in question. The
245    following code, placed in your HTML document, will put such a
246    time stamp on your page. Of course, you will have to have SSI
247    correctly enabled, as discussed above.</p>
248<div class="example"><p><code>
249        &lt;!--#config timefmt="%A %B %d, %Y" --&gt;<br />
250        This file last modified &lt;!--#flastmod file="ssi.shtml" --&gt;
251</code></p></div>
252
253    <p>Of course, you will need to replace the
254    <code>ssi.shtml</code> with the actual name of the file that
255    you're referring to. This can be inconvenient if you're just
256    looking for a generic piece of code that you can paste into any
257    file, so you probably want to use the
258    <code>LAST_MODIFIED</code> variable instead:</p>
259<div class="example"><p><code>
260        &lt;!--#config timefmt="%D" --&gt;<br />
261        This file last modified &lt;!--#echo var="LAST_MODIFIED" --&gt;
262</code></p></div>
263
264    <p>For more details on the <code>timefmt</code> format, go to
265    your favorite search site and look for <code>strftime</code>. The
266    syntax is the same.</p>
267
268
269<h3><a name="standard-footer" id="standard-footer">Including a standard footer</a></h3>
270
271
272    <p>If you are managing any site that is more than a few pages,
273    you may find that making changes to all those pages can be a
274    real pain, particularly if you are trying to maintain some kind
275    of standard look across all those pages.</p>
276
277    <p>Using an include file for a header and/or a footer can
278    reduce the burden of these updates. You just have to make one
279    footer file, and then include it into each page with the
280    <code>include</code> SSI command. The <code>include</code>
281    function can determine what file to include with either the
282    <code>file</code> attribute, or the <code>virtual</code>
283    attribute. The <code>file</code> attribute is a file path,
284    <em>relative to the current directory</em>. That means that it
285    cannot be an absolute file path (starting with /), nor can it
286    contain ../ as part of that path. The <code>virtual</code>
287    attribute is probably more useful, and should specify a URL
288    relative to the document being served. It can start with a /,
289    but must be on the same server as the file being served.</p>
290<div class="example"><p><code>
291        &lt;!--#include virtual="/footer.html" --&gt;
292</code></p></div>
293
294    <p>I'll frequently combine the last two things, putting a
295    <code>LAST_MODIFIED</code> directive inside a footer file to be
296    included. SSI directives can be contained in the included file,
297    and includes can be nested - that is, the included file can
298    include another file, and so on.</p>

303<h2><a name="config" id="config">What else can I config?</a></h2>
304
305
306    <p>In addition to being able to <code>config</code> the time
307    format, you can also <code>config</code> two other things.</p>
308
309    <p>Usually, when something goes wrong with your SSI directive,
310    you get the message</p>
311<div class="example"><p><code>
312        [an error occurred while processing this directive]
313</code></p></div>
314
315    <p>If you want to change that message to something else, you
316    can do so with the <code>errmsg</code> attribute to the
317    <code>config</code> function:</p>
318<div class="example"><p><code>
319        &lt;!--#config errmsg="[It appears that you don't know how to use SSI]" --&gt;
320</code></p></div>
321
322    <p>Hopefully, end users will never see this message, because
323    you will have resolved all the problems with your SSI
324    directives before your site goes live. (Right?)</p>
325
326    <p>And you can <code>config</code> the format in which file
327    sizes are returned with the <code>sizefmt</code> attribute. You
328    can specify <code>bytes</code> for a full count in bytes, or
329    <code>abbrev</code> for an abbreviated number in Kb or Mb, as
330    appropriate.</p>
331    </div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>

333<h2><a name="exec" id="exec">Executing commands</a></h2>
334    
335
336    <p>Here's something else that you can do with the <code>exec</code>
337    function. You can actually have SSI execute a command using the
338    shell (<code>/bin/sh</code>, to be precise - or the DOS shell,
339    if you're on Win32). The following, for example, will give you
340    a directory listing.</p>
341<div class="example"><p><code>
342        &lt;pre&gt;<br />
343        &lt;!--#exec cmd="ls" --&gt;<br />
344        &lt;/pre&gt;
345</code></p></div>
346
347    <p>or, on Windows</p>
348<div class="example"><p><code>
349        &lt;pre&gt;<br />
350        &lt;!--#exec cmd="dir" --&gt;<br />
351        &lt;/pre&gt;
352</code></p></div>
353
354    <p>You might notice some strange formatting with this directive
355    on Windows, because the output from <code>dir</code> contains
356    the string ``&lt;<code>dir</code>&gt;'' in it, which confuses
357    browsers.</p>
358
359    <p>Note that this feature is exceedingly dangerous, as it will
360    execute whatever code happens to be embedded in the
361    <code>exec</code> tag. If you have any situation where users
362    can edit content on your web pages, such as with a
363    ``guestbook'', for example, make sure that you have this
364    feature disabled. You can allow SSI, but not the
365    <code>exec</code> feature, with the <code>IncludesNOEXEC</code>
366    argument to the <code>Options</code> directive.</p>
367    </div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>

369<h2><a name="advanced" id="advanced">Advanced SSI techniques</a></h2>
370
371
372    <p>In addition to spitting out content, Apache SSI gives you
373    the option of setting variables, and using those variables in
374    comparisons and conditionals.</p>
375
376<h3><a name="variables" id="variables">Setting variables</a></h3>
377
378    <p>Using the <code>set</code> directive, you can set variables
379    for later use. We'll need this later in the discussion, so
380    we'll talk about it here. The syntax of this is as follows:</p>
381<div class="example"><p><code>
382        &lt;!--#set var="name" value="Rich" --&gt;
383</code></p></div>
384
385    <p>In addition to merely setting values literally like that, you
386    can use any other variable, including <a href="../env.html">environment variables</a> or the variables
387    discussed above (like <code>LAST_MODIFIED</code>, for example) to
388    give values to your variables. You will specify that something is
389    a variable, rather than a literal string, by using the dollar sign
390    ($) before the name of the variable.</p>
391
392    <div class="example"><p><code> &lt;!--#set var="modified" value="$LAST_MODIFIED" --&gt;
393    </code></p></div>
394
395    <p>To put a literal dollar sign into the value of your
396    variable, you need to escape the dollar sign with a
397    backslash.</p>
398<div class="example"><p><code>
399        &lt;!--#set var="cost" value="\$100" --&gt;
400</code></p></div>
401
402    <p>Finally, if you want to put a variable in the midst of a
403    longer string, and there's a chance that the name of the
404    variable will run up against some other characters, and thus be
405    confused with those characters, you can place the name of the
406    variable in braces, to remove this confusion. (It's hard to
407    come up with a really good example of this, but hopefully
408    you'll get the point.)</p>
409<div class="example"><p><code>
410        &lt;!--#set var="date" value="${DATE_LOCAL}_${DATE_GMT}" --&gt;
411</code></p></div>
412
413
414<h3><a name="conditional" id="conditional">Conditional expressions</a></h3>
415
416
417    <p>Now that we have variables, and are able to set and compare
418    their values, we can use them to express conditionals. This
419    lets SSI be a tiny programming language of sorts.
420    <code class="module"><a href="../mod/mod_include.html">mod_include</a></code> provides an <code>if</code>,
421    <code>elif</code>, <code>else</code>, <code>endif</code>
422    structure for building conditional statements. This allows you
423    to effectively generate multiple logical pages out of one
424    actual page.</p>
425
426    <p>The structure of this conditional construct is:</p>
427<div class="example"><p><code>
428    &lt;!--#if expr="test_condition" --&gt;<br />
429    &lt;!--#elif expr="test_condition" --&gt;<br />
430    &lt;!--#else --&gt;<br />
431    &lt;!--#endif --&gt;
432</code></p></div>
433
434    <p>A <em>test_condition</em> can be any sort of logical
435    comparison - either comparing values to one another, or testing
436    the ``truth'' of a particular value. (A given string is true if
437    it is nonempty.) For a full list of the comparison operators
438    available to you, see the <code class="module"><a href="../mod/mod_include.html">mod_include</a></code>
439    documentation.</p>
440   
441    <p>For example, if you wish to customize the text on your web page
442    based on the time of day, you could use the following recipe, placed
443    in the HTML page:</p>
444
445    <div class="example"><p><code>
446    Good
447    &lt;!--#if expr="%{TIME_HOUR} &lt;12" --&gt;<br />
448    morning!<br />
449    &lt;!--#else --&gt;<br />
450    afternoon!<br />
451    &lt;!--#endif --&gt;<br />
452    </code></p></div>
453
454    <p>Any other variable (either ones that you define, or normal
455    environment variables) can be used in conditional statements.
456    See <a href="../expr.html">Expressions in Apache HTTP Server</a> for
457    more information on the expression evaluation engine.</p>
458
459    <p>With Apache's ability to set environment variables with the
460    <code>SetEnvIf</code> directives, and other related directives,
461    this functionality can let you do a wide variety of dynamic content
462    on the server side without resorting a full web application.</p>

466<h2><a name="conclusion" id="conclusion">Conclusion</a></h2>
467
468    <p>SSI is certainly not a replacement for CGI, or other
469    technologies used for generating dynamic web pages. But it is a
470    great way to add small amounts of dynamic content to pages,
471    without doing a lot of extra work.</p>
