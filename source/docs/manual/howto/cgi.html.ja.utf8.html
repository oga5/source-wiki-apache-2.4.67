47<h2><a name="intro" id="intro">はじめに</a></h2>
48    
49
50    <table class="related"><tr><th>関連モジュール</th><th>関連ディレクティブ</th></tr><tr><td><ul><li><code class="module"><a href="../mod/mod_alias.html">mod_alias</a></code></li><li><code class="module"><a href="../mod/mod_cgi.html">mod_cgi</a></code></li></ul></td><td><ul><li><code class="directive"><a href="../mod/mod_mime.html#addhandler">AddHandler</a></code></li><li><code class="directive"><a href="../mod/core.html#options">Options</a></code></li><li><code class="directive"><a href="../mod/mod_alias.html#scriptalias">ScriptAlias</a></code></li></ul></td></tr></table>
51
52    <p>CGI (Common Gateway Interface) は、ウェブサーバが
53    コンテンツ生成をする外部プログラムと協調して動作するための方法を
54    定義しています。そのプログラムはしばしば CGI プログラムや
55    CGI スクリプトと呼ばれます。CGI は、ウェブサイトに動的な
56    コンテンツを置くための最も簡単で一般的な方法です。このドキュメントは、
57    Apache ウェブサーバで CGI を設定し、
58    CGI プログラムを書き始めるための入門書となるでしょう。</p>

61<h2><a name="configuring" id="configuring">CGI を許可するように Apache を設定する</a></h2>
62    
63
64    <p>CGI プログラムを正しく動作させるには、CGI を許可するように
65    Apache の設定を行う必要があります。
66    これを行なうための方法がいくつかあります。</p>
67
68    <div class="warning">
69    注: Apache が共有モジュール機能着きでビルドされている場合、
70    モジュールがロードされていることを確認してください。
71    つまり <code>httpd.conf</code> で 
72    <code class="directive"><a href="../mod/mod_so.html#loadmodule">LoadModule</a></code>
73    がコメントアウトされていないことを確認してください。
74    正常に設定されていれば次のようになるはずです:
75
76    <div class="example"><p><code>
77      LoadModule cgi_module modules/mod_cgi.so
78    </code></p></div></div>
79
80    <h3><a name="scriptalias" id="scriptalias">ScriptAlias</a></h3>
81      
82
83      <p><code class="directive"><a href="../mod/mod_alias.html#scriptalias">ScriptAlias</a></code>
84      ディレクティブを使用して、
85      CGI プログラム用の特別な別ディレクトリを Apache に設定します。
86      Apache は、このディレクトリ中の全てのファイルを CGI
87      プログラムであると仮定します。
88      そして、この特別なリソースがクライアントから要求されると、
89      そのプログラムの実行を試みます。</p>
90
91      <p><code class="directive"><a href="../mod/mod_alias.html#scriptalias">ScriptAlias</a></code>
92      ディレクティブは以下のように使用します:</p>
93
94      <div class="example"><p><code>
95        ScriptAlias /cgi-bin/ /usr/local/apache2/cgi-bin/
96      </code></p></div>
97
98      <p>デフォルト位置に Apache をインストールしたならば、
99      この例はデフォルト状態の <code>httpd.conf</code>
100      設定ファイルに含まれています。
101      <code class="directive"><a href="../mod/mod_alias.html#scriptalias">ScriptAlias</a></code>
102      ディレクティブは、URL の前に付加するディレクトリを定義する
103      <code class="directive"><a href="../mod/mod_alias.html#alias">Alias</a></code>
104      ディレクティブとかなり似ています。
105      <code class="directive">Alias</code> と <code class="directive">ScriptAlias</code>
106      は通常、<code class="directive"><a href="../mod/core.html#documentroot">DocumentRoot</a></code>
107      ディレクトリ外のディレクトリのために使用されます。
108      <code class="directive">Alias</code> と <code class="directive">ScriptAlias</code>
109      との差は、<code class="directive">ScriptAlias</code> が接頭辞で始まるすべての
110      URL は CGI プログラムとみなされるという追加の意味を含んでいることです。
111      従って、上記の例では、<code>/cgi-bin/</code>
112      で始まるリソースへのあらゆるリクエストに対して、ディレクトリ
113      <code>/usr/local/apache2/cgi-bin/</code> から提供し、それらを
114      CGI プログラムとして扱うよう Apache に示します。</p>
115
116      <p>例えば、URL <code>http://www.example.com/cgi-bin/test.pl</code>
117      が要求された場合、Apache は ファイル
118      <code>/usr/local/apache2/cgi-bin/test.pl</code>
119      を実行し、その出力を返すことを試みます。
120      もちろん、ファイルが存在し、実行可能であり、決められた方法で出力を返します。
121      そうでなければ、Apache はエラーメッセージを返します。</p>
122    
123
124    <h3><a name="nonscriptalias" id="nonscriptalias">ScriptAlias ディレクトリ外の CGI</a></h3>
125      
126
127      <p>CGI プログラムは、セキュリティ上の理由から 
128      <code class="directive"><a href="../mod/mod_alias.html#scriptalias">ScriptAlias</a></code>
129      されたディレクトリに制限されることがしばしばあります。この方法により、
130      CGI プログラムを使用できるユーザを管理者が厳しく制御することができます。
131      しかしながら、適切なセキュリティ事前対策がとられるならば、CGI
132      プログラムを任意のディレクトリで実行できないようにする理由はありません。
133      例えば、ユーザに <code class="directive"><a href="../mod/mod_userdir.html#userdir">UserDir</a></code>
134      ディレクティブで彼らのホームディレクトリ配下にウェブコンテンツを持たせたいとします。
135      もし、彼らが CGI プログラムを持つことを望んでいても、メインの
136      <code>cgi-bin</code> ディレクトリへのアクセスができない場合、
137      CGI プログラムを実行することができる他の場所が必要になります。</p>
138
139      <p>任意のディレクトリで CGI の実行を許可するには二段階の設定が必要です。
140      まず、<code class="directive"><a href="../mod/mod_mime.html#addhandler">AddHandler</a></code> や <code class="directive"><a href="../mod/core.html#sethandler">SetHandler</a></code> ディレクティブによって
141      <code>cgi-script</code> ハンドラが可能になっている必要があります。
142      次に、<code class="directive"><a href="../mod/core.html#options">Options</a></code> ディレクティブで
143      <code>ExecCGI</code> が指定されていなければなりません。</p>
144    
145
146    <h3><a name="options" id="options">CGI の実行を可能にするために Options を明示的に使用する</a></h3>
147      
148
149      <p>サーバのメインの設定ファイル中で <code class="directive"><a href="../mod/core.html#options">Options</a></code>
150      ディレクティブを明示的に使用することで、特定のディレクトリ配下で
151      CGI の実行を許可するように指定することができます:</p>
152
153      <div class="example"><p><code>
154        &lt;Directory /usr/local/apache2/htdocs/somedir&gt;<br />
155        <span class="indent">
156          Options +ExecCGI<br />
157        </span>
158        &lt;/Directory&gt;
159      </code></p></div>
160
161      <p>上記ディレクティブは、CGI ファイルの実行を可能にするよう
162      Apache に伝えます。また、どのファイルが CGI ファイルかを
163      サーバに伝える必要があります。次の
164      <code class="directive"><a href="../mod/mod_mime.html#addhandler">AddHandler</a></code>
165      ディレクティブの例では、<code>cgi</code> または <code>pl</code>
166      を拡張子に持つすべてのファイルを CGI
167      プログラムとしてみなすことをサーバに伝えます:</p>
168
169      <div class="example"><p><code>
170        AddHandler cgi-script .cgi .pl
171      </code></p></div>

182    <h3><a name="userdir" id="userdir">User ディレクトリ</a></h3>
183      
184
185      <p><code>.cgi</code> で終わるすべてのファイルに対して CGI プログラムの
186      実行を許可するには、以下の設定を使用できます。</p>
187
188      <div class="example"><p><code>
189      &lt;Directory /home/*/public_html&gt;<br />
190      <span class="indent">
191        Options +ExecCGI<br />
192        AddHandler cgi-script .cgi<br />
193      </span>
194      &lt;/Directory&gt;
195      </code></p></div>
196
197      <p>ユーザディレクトリの <code>cgi-bin</code> サブディレクトリの
198      すべてのファイルを CGI プログラムとして指定したい場合には
199      以下のようなものを使います。</p>
200
201      <div class="example"><p><code>
202      &lt;Directory /home/*/public_html/cgi-bin&gt;<br />
203      <span class="indent">
204        Options ExecCGI<br />
205        SetHandler cgi-script<br />
206      </span>
207      &lt;/Directory&gt;
208      </code></p></div>

213<h2><a name="writing" id="writing">CGI プログラムを書く</a></h2>
214    
215
216    <p>「通常の」プログラミングと CGI
217    プログラミングの間には主に二つの違いがあります。</p>
218
219    <p>一つは、CGI プログラムのすべての出力には<a class="glossarylink" href="../glossary.html#mime-type" title="用語集を参照">MIME タイプ</a>
220    ヘッダを付けなければなりません。
221    これはどのような種類のコンテンツを受け取っているかをクライアントに示す
222    HTTP ヘッダです。ほとんどの場合では、次のように出力します:</p>
223
224    <div class="example"><p><code>
225      Content-type: text/html
226    </code></p></div>
227
228    <p>もう一つは、出力を HTML
229    か、ブラウザが表示することができる何か他の形式にする必要があります。
230    大抵の場合は HTML でしょうが、GIF イメージや他の非 HTML
231    コンテンツを出力する CGI プログラムを書くこともあるでしょう。</p>
232
233    <p>これら二点以外では、CGI プログラムを書くことは、
234    あなたが書いている他のプログラムとよく似ているでしょう。</p>
235
236    <h3><a name="firstcgi" id="firstcgi">最初の CGI プログラム</a></h3>
237      
238
239      <p>次に示すのは、ブラウザに 1 行印字する CGI
240      プログラムの例です。以下を入力し、<code>first.pl</code>
241      というファイルに保存し、それを <code>cgi-bin</code>
242      ディレクトリに置いてください。</p>
243
244      <div class="example"><p><code>
245        #!/usr/bin/perl<br />
246        print "Content-type: text/html\n\n";<br />
247        print "Hello, World.";
248      </code></p></div>
249
250      <p>Perl に精通していなくても、
251      何が起こるかを理解することはできるでしょう。1 行目は、
252      <code>/usr/bin/perl</code> で見つけられるインタプリタに
253      このファイルを供給することでこのプログラムが実行されることを
254      Apache に (シェル上で実行しようとしているならば、そのシェルに ) 
255      示します。2 行目は、前述したとおり content-type の定義を印字します。
256      これには復帰改行の二つの組を後に付加します。
257      これにより、ヘッダの終りに空行が置かれ、HTTP
258      ヘッダの終りとボディの始まりを示します。3 行目は、"Hello, World."
259      という文字列を印字し、これで終りとなります。</p>
260
261      <p>好みのブラウザを開き、アドレス</p>
262
263      <div class="example"><p><code>
264        http://www.example.com/cgi-bin/first.pl
265      </code></p></div>
266
267      <p>あるいはファイルを置いたロケーションを指定すると、
268      <code>Hello, World.</code>
269      という 1 行がブラウザウィンドに現れるでしょう。
270      それはあまりエキサイティングなことではありません。
271      しかし、これがうまく動けば、
272      他のどのようなものでも動かすことができるようになります。</p>

276<h2><a name="troubleshoot" id="troubleshoot">しかし、まだ動かない !</a></h2>
277    
278
279    <p>ウェブから CGI プログラムへのアクセスを行なったとき、
280    ブラウザで見る可能性がある四つの基本的なことがあります:</p>
281
282    <dl>
283      <dt>CGI プログラムの出力</dt>
284      <dd>素晴らしい ! それはすべてがうまく動いたことを意味します。
285      出力が正常だけれども、ブラウザが正常に処理してくれない場合は、
286      正しい <code>Content-Type</code> を CGI プログラム内で
287      セットしたかを確認してください。</dd>
288
289      <dt>CGI プログラムのソースコード、または "POST Method Not Allowed"
290      というメッセージ</dt> 
291      <dd>これは、CGI プログラムを処理できるよう Apache
292      を適切に設定していなかったことを意味します。<a href="#configuring">「CGI を許可するように
293      Apache を設定する」</a>の章を読み直し、
294      あなたが何を間違えたかを探してみてください。
295      </dd>
296
297      <dt>メッセージが "Forbidden" で始まっている</dt>
298      <dd>これはパーミッションの問題ということを意味します。
299      <a href="#errorlogs">Apache のエラーログ</a>と、後述の<a href="#permissions">「ファイルのパーミッション」</a>
300      の章をチェックしてください。
301      </dd>
302
303      <dt>"Internal Server Error" というメッセージ</dt>
304      <dd><a href="#errorlogs">Apache
305      のエラーログ</a>をチェックすると、"Premature end of script headers"
306      というログが記録されていると思います。そして、おそらく CGI
307      プログラムによって生成されたエラーメッセージも記録されているでしょう。
308      この場合、CGI プログラムが適切な
309      HTTP ヘッダを出力できない原因を知るために、
310      以下の各章でチェックしてみてください。</dd>
311    </dl>
312
313    <h3><a name="permissions" id="permissions">ファイルのパーミッション</a></h3>
314      
315
316      <p>サーバはあなたの権限で実行されていないのを忘れないように。
317      つまり、起動するとき、サーバは特権をもたないユーザ - 通常 <code>nobody</code>
318      や <code>www</code> の権限で実行されます。したがって、あなたが所有する
319      ファイルを実行するには別のパーミッションが必要となります。
320      通常、<code>nobody</code> が実行するのに十分なパーミッションを与える方法は、
321      ファイルに誰でも実行可能とするパーミッションを与えることです:</p>
322
323      <div class="example"><p><code>
324        chmod a+x first.pl
325      </code></p></div>
326
327      <p>また、もしあなたのプログラムが他のファイルを読み書きするならば、
328      それらのファイルは、これが可能となる正しいパーミッション
329      を持っている必要があります。</p>
330
331    
332
333    <h3><a name="pathinformation" id="pathinformation">パス情報と環境</a></h3>
334      
335
336      <p>コマンドラインからプログラムを実行するとき、
337      意識しなくてもシェルに渡される情報があります。
338      例えば、参照するファイルのためにどこを検索したらよいかを
339      シェルに伝える <code>PATH</code> があります。</p>
340
341      <p>プログラムが CGI プログラムとしてウェブサーバによって実行されるとき、
342      それは同じ <code>PATH</code> ではないかもしれません。
343      CGI プログラム内で呼び出すあらゆるプログラム
344      (例えば、<code>sendmail</code> のようなもの) は、
345      フルパスで指定する必要があるでしょう。それにより、CGI
346      プログラムを実行しようとしたとき、
347      シェルはそのようなプログラムを見つけることができます。</p>
348
349      <p>同様なことは、スクリプトのインタプリタ (しばしば <code>perl</code>)
350      へのパスで、CGI プログラムの 1 行目に次のように示されます:</p>
351
352      <div class="example"><p><code>
353        #!/usr/bin/perl
354      </code></p></div>
355
356      <p>これがインタープリタへの実際のパスであることを確認しておきます。</p>
357    
358
359      <p>また、CGI プログラムが他の<a href="#env">環境変数</a>に依存している場合は、その環境変数が
360      Apache から渡されるようにする必要があります。</p>
361
362    <h3><a name="syntaxerrors" id="syntaxerrors">プログラムエラー</a></h3>
363      
364
365      <p>CGI
366      プログラムが失敗するのは大抵、プログラム自身に問題がある場合です。
367      一度 CGI の使い方を理解し、前述の二つの誤りを犯していないならば、
368      まず間違いなくそうでしょう。ブラウザを使ってテストする前に
369      まず確認することは、コマンドラインからプログラムが実行できることです。
370      例えば、以下を実行してみてください:</p>
371
372      <div class="example"><p><code>
373      cd /usr/local/apache2/cgi-bin<br />
374      ./first.pl
375      </code></p></div>
376
377      <p>(<code>perl</code> インタプリタは呼ばないでください。
378      シェルと Apache がスクリプトの最初の行の <a href="#pathinformation">パス情報</a> を使って見つけます。)</p>
379
380      <p>最初にプログラムから出力されるのは <code>Content-Type</code> を含み、
381      後に空行の続く HTTP ヘッダでなければなりません。他のものが出力されている
382      場合は、Apache はこのプログラムをサーバ経由で実行しようとしたときには
383      <code>Premature end of script headers</code> エラーを出力します。詳細は
384      上記の <a href="#writing">CGI プログラムを書く</a> を読んでください。</p>
385    
386
387    <h3><a name="errorlogs" id="errorlogs">エラーログ</a></h3>
388      
389
390      <p>エラーログは友達です。
391      全てのうまくいかないことは、エラーログにメッセージを生成します。
392      必ずそれを最初に見るべきです。
393      もし、あなたがウェブサイトを主催している場所が
394      エラーログの参照を許していないならば、きっと他のサイトで主催するべきです。
395      エラーログの読み方を学ぶことで、ほとんど全ての問題が迅速に確認され、
396      迅速に解決されるということが分かるでしょう。</p>
397    
398
399    <h3><a name="suexec" id="suexec">Suexec</a></h3>
400      
401
402      <p><a href="../suexec.html">suexec</a> サポートプログラムは
403      バーチャルホストやユーザのホームディレクトリの場所に依って
404      CGI プログラムを違うユーザ権限の下で走らせることを可能にします。
405      Suexec の権限のチェックは非常に厳しく、それを満たさない場合は
406      CGI プログラムが <code>Premature end of script headers</code> エラーで
407      実行されません。</p>
408
409      <p>suexec を使っているかどうかを調べためには <code>apachectl
410      -V</code> を実行して、<code>SUEXEC_BIN</code> の場所を調べてください。
411      Apache がそこに <code class="program"><a href="../programs/suexec.html">suexec</a></code> のバイナリを発見した場合は、suexec が
412      使用されます。</p>
413
414      <p>suexec を完全に理解していない限り、使うべきではありません。
415      suexec を無効にするには、<code>SUEXEC_BIN</code> から指されている
416      <code class="program"><a href="../programs/suexec.html">suexec</a></code> バイナリを削除 (か名前を変更) するだけです。
417      <a href="../suexec.html">suexec</a> を読んだ後で、まだそれを
418      使いたいのであれば、<code>suexec -V</code> を実行して suexec の
419      ログファイルの位置を調べ、そのログファイルを使ってポリシー違反を
420      見つけてください。</p>

424<h2><a name="behindscenes" id="behindscenes">裏で何が起こっているのか?</a></h2>
425    
426
427    <p>CGI プログラミングに習熟すると、
428    裏で起こっていることについて更に理解することの役に立ちます。
429    ブラウザとサーバがどのように相互通信するかについては特にそうです。
430    なぜなら、"Hello, World."
431    を印字するプログラムを書くことはおおいに結構ですが、
432    それは特に有益ではありません。</p>
433
434    <h3><a name="env" id="env">環境変数</a></h3>
435      
436
437      <p>環境変数は、
438      あなたがコンピュータを使うときに辺りに存在している値です。
439      それらは、パス
440      (コマンドをタイプしたときに実行する実際のファイルを探し出すところ)、
441      ユーザ名、端末型などのような便利なものです。
442      通常、普段使用している環境変数の完全なリストを調べるには、
443      コマンドプロンプトで <code>env</code> を入力します。</p>
444
445      <p>CGI の処理中、サーバとブラウザも環境変数を設定し、
446      それにより相互に通信することができるようになります。
447      その環境変数は、ブラウザタイプ (Netscape, IE, Lynx)、サーバタイプ
448      (Apache, IIS, WebSite)、実行されている CGI
449      プログラムの名前などです。</p>
450
451      <p>これらの変数は CGI プログラマが使用できます。
452      そして、それはクライアントとサーバの通信の話の半分です。
453      必要な変数の完全なリストは <a href="http://hoohoo.ncsa.uiuc.edu/cgi/env.html">http://hoohoo.ncsa.uiuc.edu/cgi/env.html</a> にあります。</p>
454
455      <p>以下の単純な Perl CGI
456      プログラムは、渡される全ての環境変数を表示します。同様のプログラムは、
457      Apache ディストリビューションの <code>cgi-bin</code>
458      ディレクトリに二つ含まれています。
459      いくつかの変数が必須であり、いくつかは任意であることに注意してください。
460      そして、公式のリストにはないいくつかの変数が表示されているかもしれません。
461      さらに、Apache はデフォルトで用意されている基本的なものに
462      <a href="../env.html">あなた自身の環境変数を加える</a>ための、
463      多くの異なる方法を用意してします。</p>
464
465      <div class="example"><p><code>
466        #!/usr/bin/perl<br />
467        print "Content-type: text/html\n\n";<br />
468        foreach $key (keys %ENV) {<br />
469        <span class="indent">
470          print "$key --&gt; $ENV{$key}&lt;br&gt;";<br />
471        </span>
472        }
473      </code></p></div>
474    
475
476    <h3><a name="stdin" id="stdin">STDIN と STDOUT</a></h3>
477      
478
479      <p>サーバとクライアント間のもう一つの通信は、標準入力
480      (<code>STDIN</code>)と標準出力 (<code>STDOUT</code>)
481      を通じて行なわれます。通常の文脈において、<code>STDIN</code>
482      はキーボードやプログラムが動作するために与えられるファイルを意味し、
483      <code>STDOUT</code> は通常コンソールまたはスクリーンを意味します。</p>
484
485      <p>ウェブフォームから CGI プログラムへ<code>POST</code>
486      したとき、フォームのデータは特別なフォーマットで束ねられ、
487      <code>STDIN</code> を通して、CGI プログラムに引き渡されます。
488      プログラムはデータがキーボード
489      もしくはファイルから来ていたかのように処理することができます。</p>
490
491      <p>「特別なフォーマット」はとても単純です。フィールド名と値はイコール
492      (=) で結ばれます。そして値の組はアンパサンド (&amp;) で結ばれます。
493      スペース、アンパサンド、イコールのような面倒な文字は、
494      それらが動作を駄目にしないようにその文字に相当する 16 進に変換されます。
495      全データ文字列は、以下のようになります:
496      </p>
497
498      <div class="example"><p><code>
499        name=Rich%20Bowen&amp;city=Lexington&amp;state=KY&amp;sidekick=Squirrel%20Monkey
500      </code></p></div>
501
502      <p>時々、このような文字列が URL
503      に付加されるのを見るでしょう。その場合、サーバは
504      <code>QUERY_STRING</code> という環境変数にその文字列を入れます。それは
505      <code>GET</code> リクエストと呼ばれます。
506      HTML フォームでは、データを渡すために <code>GET</code> と
507      <code>POST</code> のどちらを使用するかを、<code>FORM</code> タグの
508      <code>METHOD</code> 属性の設定で指定します。</p>
509
510      <p>CGI プログラムは、その文字列を役に立つ情報に分割する責任があります。
511      幸いにも、そのデータ処理を助けるライブラリやモジュールが存在します。
512      これらは、CGI プログラムの他の面でも同様に役に立ちます。</p>
513

516<h2><a name="libraries" id="libraries">CGI モジュール/ライブラリ</a></h2>
517    
518
519    <p>CGI プログラムを書くとき、面倒な仕事の大部分をしてくれる
520    コードライブラリまたはモジュールを使うことを検討すべきです。
521    これはエラーを減らし、早い開発につながります。</p>
522
523    <p>Perl で CGI プログラムを書いているなら、モジュールは <a href="http://www.cpan.org/">CPAN</a> で提供されています。
524    この目的のための最も普及しているモジュールは <code>CGI.pm</code> です。
525    <code>CGI::Lite</code> も検討しましょう。これは、ほとんどのプログラム
526    において必要とするすべての機能の最小セットの実装です。</p>
527
528    <p>C で CGI プログラムを書いているなら、いろいろな
529    オプションがあります。これらの内の一つは <a href="http://www.boutell.com/cgic/">http://www.boutell.com/cgic/</a>
530    で提供されている <code>CGIC</code> ライブラリです。</p>

533<h2><a name="moreinfo" id="moreinfo">更なる情報</a></h2>
534    
535
536    <p>CGI に関する情報はウェブで数多く提供されています。CGI
537    の問題については Usenet の <a href="news:comp.infosystems.www.authoring.cgi">comp.infosystems.www.authoring.cgi</a> で、
538    他のユーザと論議することができます。HTML Writers Guide の
539    -servers メーリングリストは、あなたの質問に回答してくれる偉大なリソースです。
540    <a href="http://www.hwg.org/lists/hwg-servers/">http://www.hwg.org/lists/hwg-servers/</a>
541    で更に多くを探し出すことができます。</p>
542
543    <p>そしてもちろん、おそらく CGI
544    プログラムの動作に関する詳細の全てが記述されている
545    CGI の仕様を読むべきです。オリジナルバージョンを
546    <a href="http://hoohoo.ncsa.uiuc.edu/cgi/interface.html">NCSA</a>
547    で、アップデートされたドラフトを
548    <a href="http://web.golux.com/coar/cgi/">Common Gateway Interface RFC
549    プロジェクト</a>で参照することができます。</p>
550
551    <p>CGI の問題について、加わっているメーリングリストまたはニュース
552    グループに質問を送るとき、起こったもの、起こってほしいこと、
553    実際に起こったことがどう違うか、使用しているサーバ、
554    CGI プログラムを記述している言語に関する十分な情報と、
555    可能であれば問題のコードを提供するようにしてください。
556    そうすることで、問題がより間単に見つかるようになります。</p>
557
558    <p>Apache のソースコードにおいて問題を発見したことを確信していない限り、
559    CGI の問題に関する質問を Apache
560    バグデータベースに<strong>送るべきでない</strong>
561    ことに注目してください。</p>
