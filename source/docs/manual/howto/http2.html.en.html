48<h2><a name="protocol" id="protocol">The HTTP/2 protocol</a></h2>
49    
50    <p>HTTP/2 is the evolution of the world's most successful application layer protocol, HTTP.
51    It focuses on making more efficient use of network resources. It does not change the fundamentals
52    of HTTP, the semantics. There are still request and responses and headers and all that. So, if
53    you already know HTTP/1, you know 95% about HTTP/2 as well.</p>
54    <p>There has been a lot written about HTTP/2 and how it works. The most normative is, of course,
55    its <a href="https://tools.ietf.org/html/rfc7540">RFC 7540</a> 
56    (<a href="http://httpwg.org/specs/rfc7540.html">also available in more readable formatting, YMMV</a>).
57    So, there you'll find the nuts and bolts.</p>
58    <p>But, as RFC do, it's not really a good thing to read first. It's better to first understand
59    <em>what</em> a thing wants to do and then read the RFC about <em>how</em> it is done. A much
60    better document to start with is <a href="https://daniel.haxx.se/http2/">http2  explained</a>
61    by Daniel Stenberg, the author of <a href="https://curl.haxx.se">curl</a>. It is available in
62    an ever growing list of languages, too!</p>
63    <p>Too Long, Didn't read: there are some new terms and gotchas that need to be kept in mind while reading this document:</p>
64    <ul>
65        <li>HTTP/2 is a <strong>binary protocol</strong>, as opposed to HTTP 1.1 that is plain text. The latter is meant to be human readable (for example sniffing network traffic) meanwhile the former is not. More info in the official FAQ <a href="https://http2.github.io/faq/#why-is-http2-binary">question</a>.</li>
66        <li><strong>h2</strong> is HTTP/2 over TLS (protocol negotiation via ALPN).</li>
67        <li><strong>h2c</strong> is HTTP/2 over TCP.</li>
68        <li>A <strong>frame</strong> is the smallest unit of communication within an HTTP/2 connection, consisting of a header and a variable-length sequence of octets structured according to the frame type. More info in the official documentation <a href="http://httpwg.org/specs/rfc7540.html#FramingLayer"> section</a>.</li>
69        <li>A <strong>stream</strong> is a bidirectional flow of frames within the HTTP/2 connection. The correspondent concept in HTTP 1.1 is a request/response message exchange. More info in the official documentation <a href="http://httpwg.org/specs/rfc7540.html#StreamsLayer"> section</a>.</li>
70        <li>HTTP/2 is able to run <strong>multiple streams</strong> of data over the same TCP connection, avoiding the classic HTTP 1.1 head of blocking slow request and avoiding to re-instantiate TCP connections for each request/response (KeepAlive patched the problem in HTTP 1.1 but did not fully solve it).</li>
71    </ul>

74<h2><a name="implementation" id="implementation">HTTP/2 in Apache httpd</a></h2>
75    
76    <p>The HTTP/2 protocol is implemented by its own httpd module, aptly named
77    <code class="module"><a href="../mod/mod_http2.html">mod_http2</a></code>. It implements the complete set
78    of features described by RFC 7540 and supports HTTP/2 over cleartext (http:), as
79    well as secure (https:) connections. The cleartext variant is named '<code>h2c</code>', 
80    the secure one '<code>h2</code>'. For <code>h2c</code> it allows the <em>direct</em>
81    mode and the <code>Upgrade:</code> via an initial HTTP/1 request.</p>
82    <p>One feature of HTTP/2 that offers new capabilities for web developers is
83    <a href="#push">Server Push</a>. See that section on how your web application
84    can make use of it.</p>
85  </div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>

87<h2><a name="building" id="building">Build httpd with HTTP/2 support</a></h2>
88    
89    <p><code class="module"><a href="../mod/mod_http2.html">mod_http2</a></code> uses the library of <a href="https://nghttp2.org">nghttp2</a>
90    as its implementation base. In order to build <code class="module"><a href="../mod/mod_http2.html">mod_http2</a></code> you need at least version 1.2.1 of
91    <code>libnghttp2</code> installed on your system.</p>
92    <p>When you <code>./configure</code> your Apache httpd source tree, you need to give it 
93    '<code>--enable-http2</code>' as additional argument to trigger the build of the module.
94    Should your <code>libnghttp2</code> reside in an unusual place (whatever that is on your
95    operating system), you may announce its location with '<code>--with-nghttp2=&lt;path&gt;</code>'
96    to <code>configure</code>.</p>
97    <p>While that should do the trick for most, they are people who might prefer a statically
98    linked <code>nghttp2</code> in this module. For those, the option <code>--enable-nghttp2-staticlib-deps</code>
99    exists. It works quite similar to how one statically links openssl to <code class="module"><a href="../mod/mod_ssl.html">mod_ssl</a></code>.</p>
100    <p>Speaking of SSL, you need to be aware that most browsers will speak HTTP/2 only on <code>https:</code>
101    URLs, so you need a server with SSL support. But not only that, you will need a SSL library
102    that supports the <code>ALPN</code> extension. If OpenSSL is the library you use, you need
103    at least version 1.0.2.</p>

106<h2><a name="basic-config" id="basic-config">Basic Configuration</a></h2>
107    
108
109    <p>When you have a <code>httpd</code> built with <code class="module"><a href="../mod/mod_http2.html">mod_http2</a></code> you need some
110    basic configuration for it becoming active. The first thing, as with every Apache module,
111    is that you need to load it:</p>
112    <pre class="prettyprint lang-config">LoadModule http2_module modules/mod_http2.so</pre>
113
114    
115    <p>The second directive you need to add to your server configuration is</p>
116    <pre class="prettyprint lang-config">Protocols h2 http/1.1</pre>
117
118    <p>This allows h2, the secure variant, to be the preferred protocol on your server
119    connections. When you want to enable all HTTP/2 variants, you simply write:</p>
120    <pre class="prettyprint lang-config">Protocols h2 h2c http/1.1</pre>
121
122    <p>Depending on where you put this directive, it affects all connections or just
123    the ones to a certain virtual host. You can nest it, as in:</p>
124    <pre class="prettyprint lang-config">Protocols http/1.1
125&lt;VirtualHost ...&gt;
126    ServerName test.example.org
127    Protocols h2 http/1.1
128&lt;/VirtualHost&gt;</pre>
129
130
131    <p>This allows only HTTP/1 on connections, except SSL connections to <code>test.example.org</code>
132    which offer HTTP/2.</p>
133    <div class="note"><h3>Choose a strong SSLCipherSuite</h3>
134    <p>The <code class="directive"><a href="../mod/mod_ssl.html#sslciphersuite">SSLCipherSuite</a></code> needs to be configured with
135    a strong TLS cipher suite. The current version of <code class="module"><a href="../mod/mod_http2.html">mod_http2</a></code> does not enforce any cipher but most
136    clients do so. Pointing a browser to a <code>h2</code> enabled server with a inappropriate
137    cipher suite will force it to simply refuse and fall back to HTTP 1.1. This is a common mistake
138    that is done while configuring httpd for HTTP/2 the first time, so please keep it in mind to avoid
139    long debugging sessions! If you want to be sure about the cipher suite to choose please avoid
140    the ones listed in the <a href="http://httpwg.org/specs/rfc7540.html#BadCipherSuites">HTTP/2 TLS reject list</a>.</p>
141    </div>
142    <p>The order of protocols mentioned is also relevant. By default, the first one is the 
143    most preferred protocol. When a client offers multiple choices, the one most to the 
144    left is selected. In</p>
145    <pre class="prettyprint lang-config">Protocols http/1.1 h2</pre>
146
147    <p>the most preferred protocol is HTTP/1 and it will always be selected unless a 
148    client <em>only</em> supports h2. Since we want to talk HTTP/2 to clients that
149    support it, the better order is</p>
150    <pre class="prettyprint lang-config">Protocols h2 h2c http/1.1</pre>
151
152
153    <p>There is one more thing to ordering: the client has its own preferences, too. If
154    you want, you can configure your server to select the protocol most preferred by
155    the client:</p>
156    <pre class="prettyprint lang-config">ProtocolsHonorOrder Off</pre>
157
158    <p>makes the order <em>you</em> wrote the Protocols irrelevant and only the client's
159    ordering will decide.</p>
160    <p>A last thing: the protocols you configure are not checked for correctness
161    or spelling. You can mention protocols that do not exist, so there is no need
162    to guard <code class="directive"><a href="../mod/core.html#protocols">Protocols</a></code> with any
163    <code class="directive"><a href="../mod/core.html#ifmodule">&lt;IfModule&gt;</a></code> checks.</p>
164    <p>For more advanced tips on configuration, see the <a href="../mod/mod_http2.html#dimensioning">
165    modules section about dimensioning</a> and <a href="../mod/mod_http2.html#misdirected">
166    how to manage multiple hosts with the same certificate</a>.</p>

169<h2><a name="mpm-config" id="mpm-config">MPM Configuration</a></h2>
170    
171    
172    <p>HTTP/2 is supported in all multi-processing modules that come with httpd. However, if
173    you use the <code class="module"><a href="../mod/prefork.html">prefork</a></code> mpm, there will be severe restrictions.</p>
174    <p>In <code class="module"><a href="../mod/prefork.html">prefork</a></code>, <code class="module"><a href="../mod/mod_http2.html">mod_http2</a></code> will only process one request at at time
175    per connection. But clients, such as browsers, will send many requests at the same time.
176    If one of these takes long to process (or is a long polling one), the other requests will
177    stall.</p>
178    <p><code class="module"><a href="../mod/mod_http2.html">mod_http2</a></code> will not work around this limit by default. The reason is that
179    <code class="module"><a href="../mod/prefork.html">prefork</a></code> is today only chosen, if you run processing engines that are not
180    prepared for multi-threading, e.g. will crash with more than one request.</p>
181    <p>If your setup can handle it, configuring <code class="module"><a href="../mod/event.html">event</a></code> mpm is nowadays
182    the best one (if supported on your platform).</p>
183    <p>If you are really stuck with <code class="module"><a href="../mod/prefork.html">prefork</a></code> and want multiple requests,
184    you can tweak the <code class="directive"><a href="../mod/mod_http2.html#h2minworkers">H2MinWorkers</a></code> to make
185    that possible. If it breaks, however, you own both parts.</p>

188<h2><a name="clients" id="clients">Clients</a></h2>
189    
190    <p>Almost all modern browsers support HTTP/2, but only over SSL connections: Firefox (v43),
191    Chrome (v45), Safari (since v9), iOS Safari (v9), Opera (v35), Chrome for Android (v49)
192    and Internet Explorer (v11 on Windows10) (<a href="http://caniuse.com/#search=http2">source</a>).</p>
193    <p>Other clients, as well as servers, are listed 
194    <a href="https://github.com/http2/http2-spec/wiki/Implementations">on the Implementations wiki</a>,
195    among them implementations for c, c++, common lisp, dart, erlang, haskell, java, nodejs,  php, 
196    python, perl, ruby, rust, scala and swift.</p>
197    <p>Several of the non-browser client implementations support HTTP/2 over cleartext, h2c. The
198    most versatile being <a href="https://curl.haxx.se">curl</a>.</p>

227<h2><a name="push" id="push">Server Push</a></h2>
228    
229    <p>The HTTP/2 protocol allows the server to PUSH responses to a client it never
230    asked for. The tone of the conversation is: "here is a request that you
231    never sent and the response to it will arrive soon..."</p>
232    <p>But there are restrictions: the client can disable this feature and the
233    server may only ever PUSH on a request that came from the client.</p>
234    <p>The intention is to allow the server to send resources to the client that
235    it will most likely need: a css or javascript resource that belongs to a html
236    page the client requested. A set of images that is referenced by a css, etc.</p>
237    <p>The advantage for the client is that it saves the time to send the request which
238    may range from a few milliseconds to half a second, depending on where on the 
239    globe both are located. The disadvantage is that the client may get sent
240    things it already has in its cache. Sure, HTTP/2 allows for the early cancellation
241    of such requests, but still there are resources wasted.</p>
242    <p>To summarize: there is no one good strategy on how to make best use of this 
243    feature of HTTP/2 and everyone is still experimenting. So, how do you experiment
244    with it in Apache httpd?</p>
245    <p><code class="module"><a href="../mod/mod_http2.html">mod_http2</a></code> inspect response header for <code>Link</code> headers
246    in a certain format:</p>
247    <pre class="prettyprint lang-config">Link &lt;/xxx.css&gt;;rel=preload, &lt;/xxx.js&gt;; rel=preload</pre>
248
249    <p>If the connection supports PUSH, these two resources will be sent to the
250    client. As a web developer, you may set these headers either directly in
251    your application response or you configure the server via</p>
252    <pre class="prettyprint lang-config">&lt;Location /xxx.html&gt;
253    Header add Link "&lt;/xxx.css&gt;;rel=preload"
254    Header add Link "&lt;/xxx.js&gt;;rel=preload"
255&lt;/Location&gt;</pre>
256
257    <p>If you want to use <code>preload</code> links without triggering a PUSH, you
258    can use the <code>nopush</code> parameter, as in</p>
259    <pre class="prettyprint lang-config">Link &lt;/xxx.css&gt;;rel=preload;nopush</pre>
260
261    <p>or you may disable PUSHes for your server entirely with the directive</p>
262    <pre class="prettyprint lang-config">H2Push Off</pre>
263
264    <p>And there is more:</p>
265    <p>The module will keep a diary of what has been PUSHed for each connection
266    (hashes of URLs, basically) and will not PUSH the same resource twice. When
267    the connection closes, this information is discarded.</p>
268    <p>There are people thinking about how a client can tell a server what it
269    already has, so PUSHes for those things can be avoided, but this is all
270    highly experimental right now.</p>
271    <p>Another experimental draft that has been implemented in <code class="module"><a href="../mod/mod_http2.html">mod_http2</a></code>
272    is the <a href="https://tools.ietf.org/html/draft-ruellan-http-accept-push-policy-00">
273    Accept-Push-Policy Header Field</a> where a client can, for each request, define
274    what kind of PUSHes it accepts.</p>
275    <p>
276    PUSH might not always trigger the request/response/performance that one expects or 
277    hopes for. There are various studies on this topic to be found on the web that explain
278    benefits and weaknesses and how different features of client and network influence
279    the outcome. For example: just because the server PUSHes a resource does not mean
280    a browser will actually use the data.</p>
281    <p>The major thing that influences the response being PUSHed is the request that was
282    simulated. The request URL for a PUSH is given by the application, but where do the
283    request headers come from? For example, will the PUSH request a <code>accept-language</code>
284    header and if yes with what value?</p>
285    <p>Apache will look at the original request (the one that triggered the PUSH) and copy the 
286    following headers over to PUSH requests: <code>user-agent</code>, <code>accept</code>, 
287    <code>accept-encoding</code>, <code>accept-language</code>, <code>cache-control</code>.</p>
288    <p>All other headers are ignored. Cookies will also not be copied over. PUSHing resources
289    that require a cookie to be present will not work. This can be a matter of debate. But 
290    unless this is more clearly discussed with browser, let's err on the side of caution and
291    not expose cookie where they might ordinarily not be visible.</p>

294<h2><a name="earlyhints" id="earlyhints">Early Hints</a></h2>
295    
296    <p>An alternative to PUSHing resources is to send <code>Link</code> headers to the
297    client before the response is even ready. This uses the HTTP feature called "Early Hints" and
298    is described in <a href="https://tools.ietf.org/html/rfc8297">RFC 8297</a>.</p>
299    <p>In order to use this, you need to explicitly enable it on the server via</p>
300    <pre class="prettyprint lang-config">H2EarlyHints on</pre>
301
302    <p>(It is not enabled by default since some older browser tripped on such responses.)</p>
303    <p>If this feature is on, you can use the directive <code class="directive"><a href="../mod/mod_http2.html#h2pushresource">H2PushResource</a></code> to 
304    trigger early hints and resource PUSHes:</p>
305    <pre class="prettyprint lang-config">&lt;Location /xxx.html&gt;
306    H2PushResource /xxx.css
307    H2PushResource /xxx.js
308&lt;/Location&gt;</pre>
309
310    <p>This will send out a <code>"103 Early Hints"</code> response to a client as soon
311    as the server <em>starts</em> processing the request. This may be much early than
312    the time the first response headers have been determined, depending on your web
313    application.</p>
314    <p>If <code class="directive"><a href="../mod/mod_http2.html#h2push">H2Push</a></code> is enabled, this will also start the PUSH right after the
315    103 response. If <code class="directive"><a href="../mod/mod_http2.html#h2push">H2Push</a></code> is disabled however, the 103 response will be send
316    nevertheless to the client.</p>
