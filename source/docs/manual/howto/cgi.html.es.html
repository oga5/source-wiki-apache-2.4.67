51    	<p>CGI (Common Gateway Interface) es un m&#233;todo por el cual
52		un servidor web puede interactuar con programas externos de 
53		generaci&#243;n de contenido, a ellos nos referimos com&#250;nmente como 
54		programas CGI o scripts CGI. Es el m&#233;todo m&#225;s com&#250;n y sencillo de
55        mostrar contenido din&#225;mico en su sitio web. Este documento es una 
56		introducci&#243;n para configurar CGI en su servidor web Apache, y de
57		iniciaci&#243;n para escribir programas CGI.</p>

60<h2><a name="configuring" id="configuring">Configurando Apache para permitir CGI</a></h2>
61		
62
63        <p>Para conseguir que sus programas CGI funcionen correctamente,
64	    deber&#225; configurar Apache para que permita la ejecuci&#243;n de CGI. Hay
65	    distintas formas de hacerlo.</p>
66
67        <div class="warning">Nota: Si Apache ha sido compilado con soporte
68        de m&#243;dulos compartidos, necesitar&#225; que el m&#243;dulo de CGI est&#233; cargado;
69        en su <code>httpd.conf</code> tiene que asegurarse de que la directiva
70        <code class="directive"><a href="../mod/mod_so.html#loadmodule">LoadModule</a></code>
71        no ha sido comentada. Una directiva configurada correctamente ser&#237;a as&#237;:
72            
73            <pre class="prettyprint lang-config">LoadModule cgid_module modules/mod_cgid.so</pre>
74
75
76        En Windows, o si usa un mpm que no es multihilo, como prefork, una 
77        directiva configurada correctamente podr&#237;a definirse as&#237;: 
78
79        <pre class="prettyprint lang-config">LoadModule cgi_module modules/mod_cgi.so</pre>
80</div>
81
82        <h3><a name="scriptalias" id="scriptalias">ScriptAlias</a></h3>
83            
84
85            <p>La directiva
86            <code class="directive"><a href="../mod/mod_alias.html#scriptalias">ScriptAlias</a></code>
87            indica a Apache que un directorio se ha configurado espec&#237;ficamente
88            para programas CGI. Apache asumir&#225; que cada fichero en este 
89            directorio es un programa CGI, e intentar&#225; ejecutarlos cuando un
90            cliente solicita este recurso.</p>
91        
92            <p>La directiva 
93            <code class="directive"><a href="../mod/mod_alias.html#scriptalias">ScriptAlias</a></code> se puede 
94            definir as&#237;:</p>
95
96            <pre class="prettyprint lang-config">ScriptAlias "/cgi-bin/" "/usr/local/apache2/cgi-bin/"</pre>
97
98        
99            <p>El ejemplo que se muestra es de un archivo de configuraci&#243;n
100            <code>httpd.conf</code> por defecto si usted instal&#243; Apache
101            en la ubicaci&#243;n por defecto. La directiva
102            <code class="directive"><a href="../mod/mod_alias.html#scriptalias">ScriptAlias</a></code> es muy 
103            parecida a la directiva <code class="directive"><a href="../mod/mod_alias.html#alias">Alias</a></code>,
104            &#233;sta define un prefijo de URL que se enlaza a un directorio 
105            en particular. <code class="directive">Alias</code> y
106            <code class="directive">ScriptAlias</code> se usan generalmente para 
107            directorios que se encuentran fuera del directorio 
108            <code class="directive"><a href="../mod/core.html#documentroot">DocumentRoot</a></code>. La diferencia
109            entre <code class="directive">Alias</code> y <code class="directive">ScriptAlias</code>
110            es que en <code class="directive">ScriptAlias</code> cualquier elemento
111            debajo de ese prefijo de URL ser&#225; considerado un programa CGI. As&#237;, 
112            el ejemplo de m&#225;s arriba le indica a Apache que
113            cualquier solicitud para un recurso que comience con 
114            <code>/cgi-bin/</code> deber&#237;a servirse desde el directorio
115            <code>/usr/local/apache2/cgi-bin/</code>, y deber&#237;a tratarse como un
116            programa CGI.</p>
117
118            <p>Por ejemplo, si se solicita la URL
119            <code>http://www.example.com/cgi-bin/test.pl</code>,
120            Apache intentar&#225; ejecutar el archivo
121            <code>/usr/local/apache2/cgi-bin/test.pl</code> y dar
122            el resultado. Por supuesto el archivo debe existir y ser ejecutable, 
123            y dar el resultado de una manera espec&#237;fica o Apache devolver&#225;
124            un mensaje de error.</p>
125        
126
127        <h3><a name="nonscriptalias" id="nonscriptalias">CGI fuera de directorios ScriptAlias</a></h3>
128            
129
130            <p>Los programas CGI habitualmente se restringen a los directorios de
131            <code class="directive"><a href="../mod/mod_alias.html#scriptalias">ScriptAlias</a></code> por razones de
132            seguridad. De esta manera, los administradores pueden controlar de una
133            manera m&#225;s segura quien puede ejecutar programas CGI. Aun as&#237;, si no
134            se toman suficientes precauciones, no hay ninguna raz&#243;n por la que
135            programas CGI no se puedan ejecutar desde directorios seleccionados de 
136            manera arbitraria. Por ejemplo, quiz&#225;s quiera permitir que usuarios del
137            sistema tengan contenido web en sus directorios home con la directiva
138            <code class="directive"><a href="../mod/mod_userdir.html#userdir">UserDir</a></code>. Si quieren 
139            tener sus propios programas CGI, pero no tienen acceso al directorio 
140            principal <code>cgi-bin</code>, necesitar&#225;n ser capaces de 
141            ejecutar sus scripts CGI en alg&#250;n otro sitio.</p>
142      
143            <p>Hay dos pasos a seguir para permitir la ejecuci&#243;n CGI en directorios
144            seleccionados de manera arbitraria. Primero, el handler 
145            <code>cgi-script</code> debe estar activado usando la directiva 
146            <code class="directive"><a href="../mod/mod_mime.html#addhandler">AddHandler</a></code> o la directiva 
147            <code class="directive"><a href="../mod/core.html#sethandler">SetHandler</a></code>. Segundo, el par&#225;metro
148            <code>ExecCGI</code> debe estar definido en la directiva
149            <code class="directive"><a href="../mod/core.html#options">Options</a></code>.</p>
150        
151
152        <h3><a name="options" id="options">Usando Options de manera expl&#237;cita para permitir ejecuci&#243;n de 
153            CGI</a></h3>
154            
155
156            <p>Puede usar la directiva 
157            <code class="directive"><a href="../mod/core.html#options">Options</a></code>, en el archivo de 
158            configuraci&#243;n principal para especificar que se permite la ejecuci&#243;n 
159            de CGI en un directorio en particular:</p>
160
161            <pre class="prettyprint lang-config">&lt;Directory "/usr/local/apache2/htdocs/somedir"&gt;
162    Options +ExecCGI
163&lt;/Directory&gt;</pre>
164
165            
166            <p>Esta directiva de aqu&#237; arriba le indica a Apache que debe 
167            permitir la ejecuci&#243;n de archivos CGI. Tambi&#233;n necesitar&#225; indicarle 
168            al servidor que los archivos son archivos CGI. La directiva 
169            <code class="directive"><a href="../mod/mod_mime.html#addhandler">AddHandler</a></code> le indica al 
170            servidor que debe tratar a todos los archivos con la extensi&#243;n 
171            <code>cgi</code> o <code>pl</code> como programas CGI:</p>
172
173            <pre class="prettyprint lang-config">AddHandler cgi-script .cgi .pl</pre>
174

185        <h3><a name="userdir" id="userdir">Directorios de Usuario</a></h3>
186            
187
188            <p>Para permitir la ejecuci&#243;n de programas CGI para cualquier 
189            archivo que acabe en <code>.cgi</code> en directorios de usuario, 
190            puedes usar la siguiente configuraci&#243;n:</p>
191
192            <pre class="prettyprint lang-config">&lt;Directory "/home/*/public_html"&gt;
193    Options +ExecCGI
194    AddHandler cgi-script .cgi
195&lt;/Directory&gt;</pre>
196
197
198            <p>Si quiere designar un subdirectorio <code>cgi-bin</code> dentro 
199            de un directorio de usuario en el que todos los ficheros ser&#225;n 
200            tratados como un programa CGI, puede usar lo siguiente:</p>
201
202            <pre class="prettyprint lang-config">&lt;Directory "/home/*/public_html/cgi-bin"&gt;
203    Options ExecCGI
204    SetHandler cgi-script
205&lt;/Directory&gt;</pre>

210<h2><a name="writing" id="writing">Escribiendo un programa CGI</a></h2>
211        
212
213        <p>Hay dos diferencias principales entre programaci&#243;n ``regular'' y 
214        programaci&#243;n en CGI.</p>
215
216        <p>Primera, el resultado al completo de tu programa CGI debe estar 
217        precedido de una cabecera <a class="glossarylink" href="../glossary.html#mime-type" title="ver glosario">MIME-type</a>. Esta
218        cabecera HTTP le indica al cliente que tipo de contenido est&#225;
219        recibiendo. La mayor parte de las veces, &#233;sto ser&#225; algo como:</p>
220
221        <div class="example"><p><code>
222            Content-type: text/html
223        </code></p></div>
224
225        <p>Segunda, el resultado debe estar en formato HTML, o cualquier 
226        otro formato que su navegador sea capaz de mostrar. La mayor
227        parte de las veces, ser&#225; HTML, pero otras escribir&#225; un programa
228        CGI que devuelve una imagen gif, u otro contenido no-HTML.</p>
229
230        <p>Aparte de estas dos cosas, escribir un programa en CGI se 
231        parecer&#225; bastante a cualquier otro programa que vaya a escribir.
232        </p>
233
234
235        <h3><a name="firstcgi" id="firstcgi">Su primer programa CGI</a></h3>
236            
237
238            <p>A continuaci&#243;n podr&#225; ver un ejemplo de programa CGI que muestra
239            una l&#237;nea de texto en su navegador. Escriba lo siguiente, 
240            gu&#225;rdelo en un archivo con el nombre <code>first.pl</code>, y 
241            p&#243;ngalo en su directorio <code>cgi-bin</code>.</p>
242
243            <pre class="prettyprint lang-perl">#!/usr/bin/perl
244print "Content-type: text/html\n\n";
245print "Hola, Mundo.";</pre>
246
247
248            <p>Incluso si Perl no le resulta familiar, podr&#225; ver lo que est&#225;
249            ocurriendo aqu&#237;. La primera l&#237;nea le dice a Apache (o a
250            cualquier shell en la que se est&#233; ejecutando) que este programa
251            puede ejecutarse con el int&#233;rprete en la ubicaci&#243;n
252            <code>/usr/bin/perl</code>. La segunda l&#237;nea imprime la
253            declaraci&#243;n de Content-Type que mencionamos antes, seguida de 
254            dos pares de retornos de carro. Esto pone una l&#237;nea en blanco 
255            despu&#233;s de la cabecera para indicar el final de las cabeceras
256            HTTP, y el comienzo del cuerpo del contenido. La tercera 
257            imprime la cadena de caracteres "Hola, Mundo.". Y ese es el 
258            final del programa.</p>
259
260            <p>Si lo abre con su navegador favorito y le dice que solicite la 
261            direcci&#243;n</p>
262
263            <div class="example"><p><code>
264                http://www.example.com/cgi-bin/first.pl
265            </code></p></div>
266
267            <p>o donde quiera que pusiera el archivo, ver&#225; una l&#237;nea
268            <code>Hola, Mundo.</code> aparecer&#225;n la ventana del navegador. No es 
269            muy emocionante, pero una vez que consiga que funcione podr&#225; hacer 
270            lo mismo con casi cualquier programa.</p>
271

274<h2><a name="troubleshoot" id="troubleshoot">&#161;Pero todav&#237;a no funciona!</a></h2>
275        
276
277        <p>Hay 4 cosas b&#225;sicas que puede llegar a ver en su navegador cuando
278        intenta acceder a un programa CGI desde la web:</p>
279
280        <dl>
281            <dt>El resultado del programa CGI</dt>
282            <dd>&#161;Genial! Esto indica que todo funcion&#243; correctamente. Si el
283            resultado es correcto, pero el navegador no lo procesa
284            correctamente, aseg&#250;rese de que tiene especificado 
285            correctamente el <code>Content-Type</code> en su programa 
286            CGI.</dd>
287
288            <dt>El c&#243;digo fuente de su programa CGI o un mensaje del tipo 
289            "POST Method Not Allowed".</dt>
290
291            <dd>Eso significa que no ha configurado Apache de manera
292            apropiada para interpretar su programa CGI. Relea la secci&#243;n
293            de <a href="#configuring">Configurando Apache</a> e intente
294            encontrar qu&#233; le falta.</dd>
295
296            <dt>Un mensaje que empieza con "Forbidden"</dt>
297            <dd>Eso significa que hay un problema de permisos. Compruebe el
298            <a href="#errorlogs">Log de Errores de Apache</a> y la
299            secci&#243;n de m&#225;s abajo de <a href="#permissions">Permisos de
300            Fichero</a>.</dd>
301
302            <dt>Un mensaje indicando "Internal Server Error"</dt>
303            <dd>Si comprueba el <a href="#errorlogs">Log de errores de
304            Apache</a>, probablemente encontrar&#225; que indica "Premature 
305            end of script headers", posiblemente acompa&#241;ado de otro 
306            mensaje de error generado por su programa CGI. En este caso, 
307            querr&#225; comprobar cada una de las secciones de m&#225;s adelante 
308            para ver qu&#233; impide que su programa CGI genere las cabeceras 
309            HTTP adecuadas.</dd>
310            </dl>
311
312        <h3><a name="permissions" id="permissions">Permisos de Fichero</a></h3>
313            
314
315            <p>Recuerde que el servidor no se ejecuta con su usuario. Es decir,
316            cuando el servidor arranca, est&#225; funcionando con un usuario sin
317            privilegios, generalmente el usuario <code>nobody</code>, o
318            <code>www-data</code>, as&#237; que necesitar&#225; permisos extra para
319            ejecutar los archivos de los que usted es due&#241;o. Generalmente, 
320            el m&#233;todo para dar permisos suficientes para que se pueda 
321            ejecutar con <code>nobody</code> es dar permisos de ejecuci&#243;n a 
322            todo el mundo en el fichero:</p>
323
324            <div class="example"><p><code>
325                chmod a+x first.pl
326            </code></p></div>
327
328            <p>Adem&#225;s, si su programa lee desde o escribe a cualquier otro/s
329            archivo/s, esos archivos necesitar&#225;n tener los permisos correctos
330            para permitir esas acciones.</p>
331
332        
333
334        <h3><a name="pathinformation" id="pathinformation">Informaci&#243;n de Ruta y Entorno</a></h3>
335            
336
337            <p>Cuando ejecuta un programa desde la l&#237;nea de comandos, usted tiene
338            cierta informaci&#243;n que se le pasa a la shell sin que usted se
339            percate de ello. Por ejemplo, usted tiene un <code>PATH</code>,
340            que le indica a la shell d&#243;nde debe buscar archivos a los que usted
341            hace referencia.</p>
342
343            <p>Cuando un programa se ejecuta a trav&#233;s del servidor web como un
344            programa CGI, puede que no tenga el mismo <code>PATH</code>. 
345            Cualquier programa que invoque desde su programa CGI (como por
346            ejemplo <code>sendmail</code>) necesitar&#225; que se le indique la
347            ruta absoluta, as&#237; la shell puede encontrarlos cuando intenta 
348            ejecutar su programa CGI.</p>
349
350            <p>Una manifestaci&#243;n com&#250;n de esto es la ruta del int&#233;rprete del 
351            script (a menudo <code>perl</code>) indicado en la primera l&#237;nea
352            de su programa CGI, que parecer&#225; algo como:</p>
353
354            <pre class="prettyprint lang-perl">#!/usr/bin/perl</pre>
355
356
357            <p>Aseg&#250;rese de que &#233;ste es de hecho el path de su int&#233;rprete.</p>
358            <div class="warning">
359            Cuando edita scripts CGI en Windows, los caracteres de retorno de
360            carro podr&#237;an a&#241;adirse a la l&#237;nea donde se especifica el int&#233;rprete. 
361            Aseg&#250;rese de que los archivos se transfieren al servidor en modo 
362            ASCII. Fallar en esto puede acabar con avisos del tipo "Command not 
363            found" del Sistema Operativo, debido a que &#233;ste no reconoce los 
364            caracteres de final de l&#237;nea interpretados como parte del nombre
365            de fichero del int&#233;rprete.
366            </div>

369        <h3><a name="missingenv" id="missingenv">Faltan Variables de Entorno</a></h3>
370            
371
372            <p>Si su programa CGI depende de <a href="#env">variables de entorno</a> no est&#225;ndar, necesitar&#225;
373            asegurarse de que Apache pasa esas variables.</p>
374
375            <p>Cuando no encuentra ciertas cabeceras HTTP del entorno, aseg&#250;rese 
376            de que est&#225;n formateadas seg&#250;n el 
377            <a href="http://tools.ietf.org/html/rfc2616">RFC 2616</a>, 
378            secci&#243;n 4.2: Nombres de Cabeceras deben empezar con una letra, 
379            seguida solo de letras, n&#250;meros o gui&#243;n. Cualquier cabecera 
380            que no cumpla esta regla ser&#225; ignorada de manera silenciosa.</p>

383
384        <h3><a name="syntaxerrors" id="syntaxerrors">Errores de Programa</a></h3>
385            
386
387            <p>La mayor parte de las veces cuando un programa CGI falla, es por un 
388            problema en el programa mismo. Esto ocurre generalmente cuando se 
389            maneja bien con "esto del CGI", y ya no comete los dos errores
390            mencionados m&#225;s arriba. Lo primero que hay que hacer es asegurarse
391            de que su programa se ejecuta correctamente en l&#237;nea de comandos 
392            antes de probarlo a trav&#233;s del servidor web.  Por ejemplo, 
393            intente:</p>
394
395            <div class="example"><p><code>
396                cd /usr/local/apache2/cgi-bin<br />
397                ./first.pl
398            </code></p></div>
399
400            <p>(No llame al int&#233;rprete de <code>perl</code>. La consola y Apache 
401            tienen que poder encontrar el int&#233;rprete usando l&#237;nea 
402            <a href="#pathinformation">l&#237;nea de informaci&#243;n</a> en la primera 
403            l&#237;nea del script.)</p>
404
405            <p>Lo primero que debe ver escrito por su programa es un conjunto de 
406            cabeceras HTTP, incluyendo el <code>Content-Type</code>,
407            seguido de una l&#237;nea en blanco.  Si ve alguna otra cosa, Apache
408            devolver&#225; el error <code>Premature end of script headers</code> si
409            intenta lanzar el script en el servidor web. Vea 
410            <a href="#writing">Escribiendo un programa CGI</a> m&#225;s arriba para
411            m&#225;s detalle.</p>
412        
413
414        <h3><a name="errorlogs" id="errorlogs">Log de Errores</a></h3>
415            
416
417            <p>El log de errores es su amigo. Cualquier cosa que vaya mal generar&#225; 
418            un mensaje en el log de errores. Deber&#237;a mirar siempre ah&#237; primero. 
419            Si el lugar donde est&#225; alojando su sitio web no permite que acceda
420            al log de errores, probablemente deber&#237;a alojarlo en otro sitio.
421            Aprenda a leer el log de errores y se dar&#225; cuenta de que enseguida
422            averiguar&#225; el motivo del error y lo solucionar&#225; r&#225;pidamente.</p>
423        
424
425        <h3><a name="suexec" id="suexec">Suexec</a></h3>
426            
427
428            <p>El programa de soporte <a href="../suexec.html">suexec</a> permite
429            que programas CGI se ejecuten con permisos de usuario distintos,
430            dependiendo del virtualhost o el directorio home donde se 
431            encuentren. Suexec tiene una comprobaci&#243;n de permisos muy estricta, 
432            y cualquier fallo en esa comprobaci&#243;n dar&#225; como resultado un error
433            con el mensaje <code>Premature end of script headers</code>.</p>
434
435            <p>Para comprobar si est&#225; usando Suexec, ejecute 
436            <code>apachectl -V</code> y compruebe la ubicaci&#243;n de 
437            <code>SUEXEC_BIN</code>. Si Apache encuentra un binario 
438            <code class="program"><a href="../programs/suexec.html">suexec</a></code> al arrancar, suexec se activar&#225;.</p>
439
440            <p>A menos que comprenda suxec perfectamente, no deber&#237;a usarlo.
441            Para desactivar suexec, basta con eliminar el binario 
442            <code class="program"><a href="../programs/suexec.html">suexec</a></code> al que apunta <code>SUEXEC_BIN</code> y 
443            reiniciar el servidor. Si despu&#233;s de leer sobre 
444            <a href="../suexec.html">suexec</a> todav&#237;a quiere usarlo, entonces
445            ejecute <code>suexec -V</code> para encontrar la ubicaci&#243;n del 
446            fichero log de suexec, y use ese log para encontrar que pol&#237;tica no
447            est&#225; cumpliendo.</p>

451<h2><a name="behindscenes" id="behindscenes">&#191;Qu&#233; ocurre entre bastidores?</a></h2>
452        
453
454        <p>En cuanto tenga conocimiento avanzado de programaci&#243;n CGI, le ser&#225; 
455        &#250;til comprender m&#225;s de lo que ocurre entre bastidores. 
456        Espec&#237;ficamente, c&#243;mo el navegador y el servidor se comunican el uno
457        con el otro. Porque aunque est&#233; muy bien escribir un programa que 
458        diga "Hola, Mundo.", no tiene una gran utilidad.</p>
459
460        <h3><a name="env" id="env">Variables de Entorno</a></h3>
461            
462
463            <p>Las variables de entorno son valores que est&#225;n ah&#237; cuando 
464            usa el ordenador. Son cosas &#250;tiles como el path (donde su ordenador
465            busca el archivo espec&#237;fico que se lanza cuando usted escribe un 
466            comando), su nombre de usuario, el tipo de terminal que usa, etc. 
467            Para una lista completa de la variables de entorno normales que se 
468            se usan en su d&#237;a a d&#237;a escriba <code>env</code> en la l&#237;nea de 
469            comandos.</p>
470
471            <p>Durante la transacci&#243;n CGI, el servidor y el navegador tambi&#233;n 
472            configuran variables de entorno, y as&#237; pueden comunicarse entre 
473            ellos. Cosas como el tipo de navegador (Netscape, IE, Lynx), el tipo
474            de servidor (Apache, IIS, WebSite), el nombre del programa CGI que
475            se est&#225; ejecutando, etc.</p>
476
477            <p>Estas variables est&#225;n disponibles para el programador de CGI, y son 
478            la mitad de la historia de la comunicaci&#243;n cliente-servidor. La 
479            lista completa de las variables necesarias se encuentra en 
480            <a href="http://www.ietf.org/rfc/rfc3875">el RFC de Common Gateway
481            Interface</a>.</p>
482
483            <p>Este sencillo programa CGI en Perl mostrar&#225; todas las variables 
484            de entorno que se est&#225;n pasando entre el cliente y el navegador. Dos
485            programas similares est&#225;n incluidos en el directorio 
486            <code>cgi-bin</code> de la distribuci&#243;n de Apache. Tenga en cuenta
487            que algunas variables son necesarias mientras que otras son 
488            opcionales, as&#237; que es posible que vea algunas variables que no 
489            est&#225;n en la lista oficial. Adicionalmente, Apache aporta distintas
490            maneras diferentes para que pueda
491            <a href="../env.html">a&#241;adir sus variables de entorno</a> a las 
492            b&#225;sicas que se proveen por defecto.</p>
493
494            <pre class="prettyprint lang-perl">#!/usr/bin/perl
495use strict;
496use warnings;
497
498print "Content-type: text/html\n\n";
499          
500foreach my $key (keys %ENV) {
501    print "$key --&gt; $ENV{$key}&lt;br&gt;";
502}</pre>

506        <h3><a name="stdin" id="stdin">STDIN y STDOUT</a></h3>
507            
508
509            <p>Otra comunicaci&#243;n entre el servidor y el cliente ocurre en la 
510            entrada est&#225;ndar (<code>STDIN</code>) y la salida est&#225;ndar 
511            (<code>STDOUT</code>). En el contexto normal de cada d&#237;a, 
512            <code>STDIN</code> es la entrada con el teclado, o un fichero que se 
513            le da a un programa para que act&#250;e sobre &#233;l, y <code>STDOUT</code>
514            generalmente es la consola o la pantalla.</p>
515
516            <p>Cuando hace <code>POST</code> con un formulario de web a un programa 
517            CGI, los datos en ese formulario se empaquetan en un formato especial
518            que se entrega a su programa CGI en el <code>STDIN</code>.
519            Entonces el programa puede procesar la informaci&#243;n como si le llegara
520            desde el teclado, o desde un fichero.</p>
521
522            <p>El "formato especial" es muy sencillo. Un nombre de campo y su 
523            valor se asocian juntos con el signo igual (=), y pares de valores 
524            se asocian juntos con el ampersand &#243; et en espa&#241;ol (&amp;). 
525            Caracteres inconvenientes como los espacios, ampersands y signos de 
526            igual, se convierten en su equivalente hexadecimal para no impidan 
527            el funcionamiento correcto del programa. La cadena de datos al 
528            completo ser&#225; algo como:</p>
529
530  <div class="example"><p><code>
531        name=Rich%20Bowen&amp;city=Lexington&amp;state=KY&amp;sidekick=Squirrel%20Monkey
532  </code></p></div>
533
534            <p>A veces tendr&#225; este tipo de cadena de caracteres al final de una 
535            URL. Cuando esto ocurre, el servidor pone esa cadena en una variable 
536            de entorno que se llama <code>QUERY_STRING</code>. Esto se llama 
537            solicitud <code>GET</code>. Su formulario HTML especifica si se usa 
538            un <code>GET</code> o un <code>POST</code> para entregar la 
539            informaci&#243;n, configurando el atributo <code>METHOD</code> en la 
540            etiqueta <code>FORM</code>.</p>
541
542            <p>Su programa es el responsable de convertir esa cadena de 
543            caracteres en informaci&#243;n &#250;til. Afortunadamente, hay librer&#237;as y 
544            m&#243;dulos disponibles que ayudan a procesar la informaci&#243;n, as&#237; como a 
545            gestionar los distintos aspectos de su programa CGI.</p>

549<h2><a name="libraries" id="libraries">M&#243;dulos/librer&#237;as CGI</a></h2>
550        
551
552        <p>Cuando escribe programas CGI, deber&#237;a considerar usar una librer&#237;a de
553        c&#243;digo, o m&#243;dulo, para hacer todo el trabajo m&#225;s arduo por usted.
554        Esto lleva a tener menos errores y un desarrollo de c&#243;digo m&#225;s 
555        r&#225;pido.</p>
556
557        <p>Si est&#225; escribiendo un programa CGI en Perl, existen m&#243;dulos 
558        disponibles en <a href="http://www.cpan.org/">CPAN</a>. El m&#243;dulo m&#225;s
559        conocido para este prop&#243;sito es <code>CGI.pm</code>. Quiz&#225;s quiera
560        considerar <code>CGI::Lite</code>, que implementa una funcionalidad 
561        m&#237;nima, que es todo lo que se necesita en la mayor&#237;a de los programas.</p>
562
563        <p>Si est&#225; escribiendo programas CGI en C, hay varidad de opciones. Una
564        de estas es la librer&#237;a <code>CGIC</code>, de
565        <a href="http://www.boutell.com/cgic/">http://www.boutell.com/cgic/</a>.
566        </p>
