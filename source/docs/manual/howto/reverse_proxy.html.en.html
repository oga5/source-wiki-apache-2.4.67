29
30    <p>In addition to being a "basic" web server, and providing static and
31    dynamic content to end-users, Apache httpd (as well as most other web
32    servers) can also act as a reverse proxy server, also-known-as a
33    "gateway" server.</p>
34
35    <p>In such scenarios, httpd itself does not generate or host the data,
36    but rather the content is obtained by one or several backend servers,
37    which normally have no direct connection to the external network. As
38    httpd receives a request from a client, the request itself is <em>proxied</em>
39    to one of these backend servers, which then handles the request, generates
40    the content and then sends this content back to httpd, which then
41    generates the actual HTTP response back to the client.</p>
42
43    <p>There are numerous reasons for such an implementation, but generally
44    the typical rationales are due to security, high-availability, load-balancing
45    and centralized authentication/authorization. It is critical in these
46    implementations that the layout, design and architecture of the backend
47    infrastructure (those servers which actually handle the requests) are
48    insulated and protected from the outside; as far as the client is concerned,
49    the reverse proxy server <em>is</em> the sole source of all content.</p>
50
51    <p>A typical implementation is below:</p>
52    <p class="centered"><img src="../images/reverse-proxy-arch.png" alt="reverse-proxy-arch" /></p>

71<h2><a name="simple" id="simple">Simple reverse proxying</a></h2>
72    
73
74    <p>
75      The <code class="directive"><a href="../mod/mod_proxy.html#proxypass">ProxyPass</a></code>
76      directive specifies the mapping of incoming requests to the backend
77      server (or a cluster of servers known as a <code>Balancer</code>
78      group). The simplest example proxies all requests (<code>"/"</code>)
79      to a single backend:
80    </p>
81
82    <pre class="prettyprint lang-config">ProxyPass "/"  "http://www.example.com/"</pre>
83
84
85    <p>
86      To ensure that and <code>Location:</code> headers generated from
87      the backend are modified to point to the reverse proxy, instead of
88      back to itself, the <code class="directive"><a href="../mod/mod_proxy.html#proxypassreverse">ProxyPassReverse</a></code>
89      directive is most often required:
90    </p>
91
92    <pre class="prettyprint lang-config">ProxyPass "/"  "http://www.example.com/"
93ProxyPassReverse "/"  "http://www.example.com/"</pre>
94
95
96    <p>Only specific URIs can be proxied, as shown in this example:</p>
97
98    <pre class="prettyprint lang-config">ProxyPass "/images"  "http://www.example.com/"
99ProxyPassReverse "/images"  "http://www.example.com/"</pre>
100
101
102    <p>In the above, any requests which start with the <code>/images</code>
103      path with be proxied to the specified backend, otherwise it will be handled
104      locally.
105    </p>

108<h2><a name="cluster" id="cluster">Clusters and Balancers</a></h2>
109    
110
111    <p>
112      As useful as the above is, it still has the deficiencies that should
113      the (single) backend node go down, or become heavily loaded, that proxying
114      those requests provides no real advantage. What is needed is the ability
115      to define a set or group of backend servers which can handle such
116      requests and for the reverse proxy to load balance and failover among
117      them. This group is sometimes called a <em>cluster</em> but Apache httpd's
118      term is a <em>balancer</em>. One defines a balancer by leveraging the
119      <code class="directive"><a href="../mod/mod_proxy.html#proxy">&lt;Proxy&gt;</a></code> and
120      <code class="directive"><a href="../mod/mod_proxy.html#balancermember">BalancerMember</a></code> directives as
121      shown:
122    </p>
123
124    <pre class="prettyprint lang-config">&lt;Proxy balancer://myset&gt;
125    BalancerMember http://www2.example.com:8080
126    BalancerMember http://www3.example.com:8080
127    ProxySet lbmethod=bytraffic
128&lt;/Proxy&gt;
129
130ProxyPass "/images/"  "balancer://myset/"
131ProxyPassReverse "/images/"  "balancer://myset/"</pre>
132
133
134    <p>
135      The <code>balancer://</code> scheme is what tells httpd that we are creating
136      a balancer set, with the name <em>myset</em>. It includes 2 backend servers,
137      which httpd calls <em>BalancerMembers</em>. In this case, any requests for
138      <code>/images</code> will be proxied to <em>one</em> of the 2 backends.
139      The <code class="directive"><a href="../mod/mod_proxy.html#proxyset">ProxySet</a></code> directive
140      specifies that the <em>myset</em> Balancer use a load balancing algorithm
141      that balances based on I/O bytes.
142    </p>
143
144    <div class="note"><h3>Hint</h3>
145      <p>
146      	<em>BalancerMembers</em> are also sometimes referred to as <em>workers</em>.
147      </p>
148   </div>

152<h2><a name="config" id="config">Balancer and BalancerMember configuration</a></h2>
153    
154
155    <p>
156      You can adjust numerous configuration details of the <em>balancers</em>
157      and the <em>workers</em> via the various parameters defined in
158      <code class="directive"><a href="../mod/mod_proxy.html#proxypass">ProxyPass</a></code>. For example,
159      assuming we would want <code>http://www3.example.com:8080</code> to
160      handle 3x the traffic with a timeout of 1 second, we would adjust the
161      configuration as follows:
162    </p>
163
164    <pre class="prettyprint lang-config">&lt;Proxy balancer://myset&gt;
165    BalancerMember http://www2.example.com:8080
166    BalancerMember http://www3.example.com:8080 loadfactor=3 timeout=1
167    ProxySet lbmethod=bytraffic
168&lt;/Proxy&gt;
169
170ProxyPass "/images"  "balancer://myset/"
171ProxyPassReverse "/images"  "balancer://myset/"</pre>
172

176<h2><a name="failover" id="failover">Failover</a></h2>
177    
178
179    <p>
180      You can also fine-tune various failover scenarios, detailing which workers
181      and even which balancers should be accessed in such cases. For example, the
182      below setup implements three failover cases:
183    </p>
184    <ol>
185      <li>
186        <code>http://spare1.example.com:8080</code> and
187        <code>http://spare2.example.com:8080</code> are only sent traffic if one
188        or both of <code>http://www2.example.com:8080</code> or
189        <code>http://www3.example.com:8080</code> is unavailable. (One spare
190        will be used to replace one unusable member of the same balancer set.)
191      </li>
192      <li>
193        <code>http://hstandby.example.com:8080</code> is only sent traffic if
194        all other workers in balancer set <code>0</code> are not available.
195      </li>
196      <li>
197        If all load balancer set <code>0</code> workers, spares, and the standby
198        are unavailable, only then will the
199        <code>http://bkup1.example.com:8080</code> and
200        <code>http://bkup2.example.com:8080</code> workers from balancer set
201        <code>1</code> be brought into rotation.
202      </li>
203    </ol>
204    <p>
205      Thus, it is possible to have one or more hot spares and hot standbys for
206      each load balancer set.
207    </p>
208
209    <pre class="prettyprint lang-config">&lt;Proxy balancer://myset&gt;
210    BalancerMember http://www2.example.com:8080
211    BalancerMember http://www3.example.com:8080 loadfactor=3 timeout=1
212    BalancerMember http://spare1.example.com:8080 status=+R
213    BalancerMember http://spare2.example.com:8080 status=+R
214    BalancerMember http://hstandby.example.com:8080 status=+H
215    BalancerMember http://bkup1.example.com:8080 lbset=1
216    BalancerMember http://bkup2.example.com:8080 lbset=1
217    ProxySet lbmethod=byrequests
218&lt;/Proxy&gt;
219
220ProxyPass "/images/"  "balancer://myset/"
221ProxyPassReverse "/images/"  "balancer://myset/"</pre>
222
223
224    <p>
225      For failover, hot spares are used as replacements for unusable workers in
226      the same load balancer set. A worker is considered unusable if it is
227      draining, stopped, or otherwise in an error/failed state. Hot standbys are
228      used if all workers and spares in the load balancer set are
229      unavailable. Load balancer sets (with their respective hot spares and
230      standbys) are always tried in order from lowest to highest.
231    </p>

235<h2><a name="manager" id="manager">Balancer Manager</a></h2>
236    
237
238    <p>
239      One of the most unique and useful features of Apache httpd's reverse proxy is
240	  the embedded <em>balancer-manager</em> application. Similar to
241	  <code class="module"><a href="../mod/mod_status.html">mod_status</a></code>, <em>balancer-manager</em> displays
242	  the current working configuration and status of the enabled
243	  balancers and workers currently in use. However, not only does it
244	  display these parameters, it also allows for dynamic, runtime, on-the-fly
245	  reconfiguration of almost all of them, including adding new <em>BalancerMembers</em>
246	  (workers) to an existing balancer. To enable these capability, the following
247	  needs to be added to your configuration:
248    </p>
249
250    <pre class="prettyprint lang-config">&lt;Location "/balancer-manager"&gt;
251    SetHandler balancer-manager
252    Require host localhost
253&lt;/Location&gt;</pre>
254
255
256    <div class="warning"><h3>Warning</h3>
257      <p>Do not enable the <em>balancer-manager</em> until you have <a href="../mod/mod_proxy.html#access">secured your server</a>. In
258      particular, ensure that access to the URL is tightly
259      restricted.</p>
260    </div>
261
262    <p>
263      When the reverse proxy server is accessed at that url
264      (eg: <code>http://rproxy.example.com/balancer-manager/</code>, you will see a
265      page similar to the below:
266    </p>
267    <p class="centered"><img src="../images/bal-man.png" alt="balancer-manager page" /></p>
268
269    <p>
270      This form allows the devops admin to adjust various parameters, take
271      workers offline, change load balancing methods and add new works. For
272      example, clicking on the balancer itself, you will get the following page:
273    </p>
274    <p class="centered"><img src="../images/bal-man-b.png" alt="balancer-manager page" /></p>
275
276    <p>
277      Whereas clicking on a worker, displays this page:
278    </p>
279    <p class="centered"><img src="../images/bal-man-w.png" alt="balancer-manager page" /></p>
280
281    <p>
282      To have these changes persist restarts of the reverse proxy, ensure that
283      <code class="directive"><a href="../mod/mod_proxy.html#balancerpersist">BalancerPersist</a></code> is enabled.
284    </p>

288<h2><a name="health-check" id="health-check">Dynamic Health Checks</a></h2>
289    
290
291    <p>
292      Before httpd proxies a request to a worker, it can <em>"test"</em> if that worker
293      is available via setting the <code>ping</code> parameter for that worker using
294      <code class="directive"><a href="../mod/mod_proxy.html#proxypass">ProxyPass</a></code>. Oftentimes it is
295      more useful to check the health of the workers <em>out of band</em>, in a
296      dynamic fashion. This is achieved in Apache httpd by the
297      <code class="module"><a href="../mod/mod_proxy_hcheck.html">mod_proxy_hcheck</a></code> module.
298    </p>

302<h2><a name="status" id="status">BalancerMember status flags</a></h2>
303    
304
305    <p>
306      In the <em>balancer-manager</em> the current state, or <em>status</em>, of a worker
307      is displayed and can be set/reset. The meanings of these statuses are as follows:
308    </p>
309      <table class="bordered">
310      	<tr><th>Flag</th><th>String</th><th>Description</th></tr>
311      	<tr><td>&nbsp;</td><td><em>Ok</em></td><td>Worker is available</td></tr>
312      	<tr><td>&nbsp;</td><td><em>Init</em></td><td>Worker has been initialized</td></tr>
313        <tr><td><code>D</code></td><td><em>Dis</em></td><td>Worker is disabled and will not accept any requests; will be
314                    automatically retried.</td></tr>
315        <tr><td><code>S</code></td><td><em>Stop</em></td><td>Worker is administratively stopped; will not accept requests
316                    and will not be automatically retried</td></tr>
317        <tr><td><code>I</code></td><td><em>Ign</em></td><td>Worker is in ignore-errors mode and will always be considered available.</td></tr>
318        <tr><td><code>R</code></td><td><em>Spar</em></td><td>Worker is a hot spare. For each worker in a given lbset that is unusable
319                    (draining, stopped, in error, etc.), a usable hot spare with the same lbset will be used in
320                    its place. Hot spares can help ensure that a specific number of workers are always available
321                    for use by a balancer.</td></tr>
322        <tr><td><code>H</code></td><td><em>Stby</em></td><td>Worker is in hot-standby mode and will only be used if no other
323                    viable workers or spares are available in the balancer set.</td></tr>
324        <tr><td><code>E</code></td><td><em>Err</em></td><td>Worker is in an error state, usually due to failing pre-request check;
325                    requests will not be proxied to this worker, but it will be retried depending on
326                    the <code>retry</code> setting of the worker.</td></tr>
327        <tr><td><code>N</code></td><td><em>Drn</em></td><td>Worker is in drain mode and will only accept existing sticky sessions
328                    destined for itself and ignore all other requests.</td></tr>
329        <tr><td><code>C</code></td><td><em>HcFl</em></td><td>Worker has failed dynamic health check and will not be used until it
330                    passes subsequent health checks.</td></tr>
331      </table>
