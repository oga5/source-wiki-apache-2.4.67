33  <p>This document describes some of the major changes between the
34     2.0 and 2.2 versions of the Apache HTTP Server. For new features since
35     version 1.3, see the <a href="new_features_2_0.html">2.0 new features</a>
36     document.</p>

45<h2><a name="core" id="core">Core Enhancements</a></h2>
46    
47    <dl>
48
49      <dt>Authn/Authz</dt>
50      <dd>The bundled authentication and authorization modules have
51          been refactored.  The new mod_authn_alias(already removed from 2.3/2.4)
52          module can greatly simplify certain authentication configurations.
53          See <a href="#module">module name changes</a>, and
54          <a href="#developer">the developer changes</a> for more
55          information about how these changes affects users and module
56          writers.</dd>
57
58      <dt>Caching</dt>
59      <dd><code class="module"><a href="./mod/mod_cache.html">mod_cache</a></code>, <code class="module"><a href="./mod/mod_cache_disk.html">mod_cache_disk</a></code>, and
60          mod_mem_cache(already removed from 2.3/2.4) have undergone a lot of changes, and
61          are now considered production-quality. <code class="program"><a href="./programs/htcacheclean.html">htcacheclean</a></code>
62          has been introduced to clean up <code class="module"><a href="./mod/mod_cache_disk.html">mod_cache_disk</a></code>
63          setups.</dd>
64
65      <dt>Configuration</dt>
66      <dd>The default configuration layout has been simplified and
67          modularised. Configuration snippets which can be used to
68          enable commonly-used features are now bundled with Apache, and
69          can be easily added to the main server config.</dd>
70
71      <dt>Graceful stop</dt>
72      <dd>The <code class="module"><a href="./mod/prefork.html">prefork</a></code>, <code class="module"><a href="./mod/worker.html">worker</a></code> and
73          <code class="module"><a href="./mod/event.html">event</a></code>  MPMs now allow <code class="program"><a href="./programs/httpd.html">httpd</a></code>
74          to be shutdown gracefully via the
75          <a href="stopping.html#gracefulstop"><code>graceful-stop</code></a>
76          signal. The <code class="directive"><a href="./mod/mpm_common.html#gracefulshutdowntimeout">GracefulShutdownTimeout</a></code> directive
77          has been added to specify an optional timeout, after which
78          <code class="program"><a href="./programs/httpd.html">httpd</a></code> will terminate regardless of the status
79          of any requests being served.</dd>
80
81      <dt>Proxying</dt>
82      <dd>The new <code class="module"><a href="./mod/mod_proxy_balancer.html">mod_proxy_balancer</a></code> module provides
83          load balancing services for <code class="module"><a href="./mod/mod_proxy.html">mod_proxy</a></code>.
84          The new <code class="module"><a href="./mod/mod_proxy_ajp.html">mod_proxy_ajp</a></code> module adds support for the
85          <code>Apache JServ Protocol version 1.3</code> used by
86          <a href="http://tomcat.apache.org/">Apache Tomcat</a>.</dd>
87
88      <dt>Regular Expression Library Updated</dt>
89      <dd>Version 5.0 of the
90          <a href="http://www.pcre.org/">Perl Compatible Regular Expression
91          Library</a> (PCRE) is now included.  <code class="program"><a href="./programs/httpd.html">httpd</a></code> can be
92          configured to use a system installation of PCRE by passing the
93          <code>--with-pcre</code> flag to configure.</dd>
94
95      <dt>Smart Filtering</dt>
96      <dd><code class="module"><a href="./mod/mod_filter.html">mod_filter</a></code> introduces dynamic configuration
97          to the output filter chain.  It enables filters to be conditionally
98          inserted, based on any Request or Response header or environment
99          variable, and dispenses with the more problematic dependencies and
100          ordering problems in the 2.0 architecture.</dd>
101
102      <dt>Large File Support</dt>
103      <dd><code class="program"><a href="./programs/httpd.html">httpd</a></code> is now built with support for files larger
104          than 2GB on modern 32-bit Unix systems.  Support for handling
105          &gt;2GB request bodies has also been added.</dd>
106
107      <dt>Event MPM</dt>
108      <dd>The <code class="module"><a href="./mod/event.html">event</a></code> MPM uses a separate thread to handle
109          Keep Alive requests and accepting connections.  Keep Alive requests
110          have traditionally required httpd to dedicate a worker to handle it.
111          This dedicated worker could not be used again until the Keep Alive
112          timeout was reached.</dd>
113
114      <dt>SQL Database Support</dt>
115      <dd><code class="module"><a href="./mod/mod_dbd.html">mod_dbd</a></code>, together with the <code>apr_dbd</code>
116          framework, brings direct SQL support to modules that need it.
117          Supports connection pooling in threaded MPMs.</dd>
118
119    </dl>

122<h2><a name="module" id="module">Module Enhancements</a></h2>
123    
124    <dl>
125      <dt>Authn/Authz</dt>
126      <dd>Modules in the aaa directory have been renamed and offer
127          better support for digest authentication.  For example,
128          <code>mod_auth</code> is now split into
129          <code class="module"><a href="./mod/mod_auth_basic.html">mod_auth_basic</a></code> and
130          <code class="module"><a href="./mod/mod_authn_file.html">mod_authn_file</a></code>; <code>mod_auth_dbm</code> is now
131          called <code class="module"><a href="./mod/mod_authn_dbm.html">mod_authn_dbm</a></code>; <code>mod_access</code> has
132          been renamed <code class="module"><a href="./mod/mod_authz_host.html">mod_authz_host</a></code>.  There is also a new
133          mod_authn_alias(already removed from 2.3/2.4) module for simplifying
134          certain authentication configurations.
135      </dd>
136
137      <dt><code class="module"><a href="./mod/mod_authnz_ldap.html">mod_authnz_ldap</a></code></dt>
138      <dd>This module is a port of the 2.0
139          <code>mod_auth_ldap</code> module to the 2.2 <code>Authn/Authz</code>
140          framework.  New features include using LDAP attribute values and
141          complicated search filters in the
142          <code class="directive"><a href="./mod/mod_authz_core.html#require">Require</a></code> directive.</dd>
143
144      <dt><code class="module"><a href="./mod/mod_authz_owner.html">mod_authz_owner</a></code></dt>
145      <dd>A new module that authorizes access to files based
146          on the owner of the file on the file system</dd>
147
148      <dt><code class="module"><a href="./mod/mod_version.html">mod_version</a></code></dt>
149      <dd>A new module that allows configuration blocks to be enabled based on the
150          version number of the running server.</dd>
151
152      <dt><code class="module"><a href="./mod/mod_info.html">mod_info</a></code></dt>
153      <dd>Added a new <code>?config</code> argument which will show
154          the configuration directives as parsed by Apache, including
155          their file name and line number.  The module also
156          shows the order of all request hooks and additional
157          build information, similar to <code>httpd -V</code>.</dd>
158
159      <dt><code class="module"><a href="./mod/mod_ssl.html">mod_ssl</a></code></dt>
160      
161      <dd>Added a support for
162         <a href="http://www.ietf.org/rfc/rfc2817.txt">RFC 2817</a>, which
163         allows connections to upgrade from clear text to TLS encryption.</dd>
164
165      <dt><code class="module"><a href="./mod/mod_imagemap.html">mod_imagemap</a></code></dt>
166      <dd><code>mod_imap</code> has been renamed to
167          <code class="module"><a href="./mod/mod_imagemap.html">mod_imagemap</a></code> to avoid user confusion.</dd>
168    </dl>

172<h2><a name="programs" id="programs">Program Enhancements</a></h2>
173    
174    <dl>
175      <dt><code class="program"><a href="./programs/httpd.html">httpd</a></code></dt>
176      <dd>A new command line option <code>-M</code> has been added that
177      lists all modules that are loaded based on the current
178      configuration. Unlike the <code>-l</code> option, this list
179      includes DSOs loaded via <code class="module"><a href="./mod/mod_so.html">mod_so</a></code>.</dd>
180
181      <dt><code class="program"><a href="./programs/httxt2dbm.html">httxt2dbm</a></code></dt>
182      <dd>A new program used to generate dbm files from text input,
183          for use in <code class="directive"><a href="./mod/mod_rewrite.html#rewritemap">RewriteMap</a></code>
184          with the <code>dbm</code> map type.</dd>
185    </dl>

188<h2><a name="developer" id="developer">Module Developer Changes</a></h2>
189    
190    <dl>
191      <dt><a class="glossarylink" href="./glossary.html#apr" title="see glossary">APR</a> 1.0 API</dt>
192
193      <dd>Apache 2.2 uses the APR 1.0 API.  All deprecated functions and
194          symbols have been removed from <code>APR</code> and
195          <code>APR-Util</code>. For details, see the
196          <a href="http://apr.apache.org/">APR Website</a>.</dd>
197
198      <dt>Authn/Authz</dt>
199      <dd>The bundled authentication and authorization modules have
200          been renamed along the following lines:
201          <ul>
202          <li><code>mod_auth_*</code>  -&gt; Modules that implement an HTTP
203          authentication mechanism</li>
204          <li><code>mod_authn_*</code> -&gt; Modules that provide a backend
205          authentication provider</li>
206          <li><code>mod_authz_*</code> -&gt; Modules that implement
207          authorization (or access)</li>
208          <li><code>mod_authnz_*</code> -&gt; Module that implements both
209          authentication &amp; authorization</li>
210          </ul>
211          There is a new authentication backend provider
212          scheme which greatly eases the construction of new authentication
213          backends.</dd>
214
215      <dt>Connection Error Logging</dt>
216
217      <dd>A new function, <code>ap_log_cerror</code> has been added to log
218          errors that occur with the client's connection.  When logged,
219          the message includes the client IP address.</dd>
220
221      <dt>Test Configuration Hook Added</dt>
222
223      <dd>A new hook, <code>test_config</code> has been added to aid
224          modules that want to execute special code only when the user passes
225          <code>-t</code> to <code class="program"><a href="./programs/httpd.html">httpd</a></code>.</dd>
226
227      <dt>Set Threaded MPM's Stacksize</dt>
228
229      <dd>A new directive, <code class="directive"><a href="./mod/mpm_common.html#threadstacksize">ThreadStackSize</a></code> has been added to
230          set the stack size on all threaded MPMs.  This is required
231          for some third-party modules on platforms with small default
232          thread stack size.</dd>
233
234      <dt>Protocol handling for output filters</dt>
235
236      <dd>In the past, every filter has been responsible for ensuring
237          that it generates the correct response headers where it affects
238          them.  Filters can now delegate common protocol management to
239          <code class="module"><a href="./mod/mod_filter.html">mod_filter</a></code>, using the
240          <code>ap_register_output_filter_protocol</code> or
241          <code>ap_filter_protocol</code> calls.</dd>
242
243      <dt>Monitor hook added</dt>
244      <dd>Monitor hook enables modules to run regular/scheduled jobs
245          in the parent (root) process.</dd>
246
247      <dt>Regular expression API changes</dt>
248
249      <dd>The <code>pcreposix.h</code> header is no longer available;
250      it is replaced by the new <code>ap_regex.h</code> header.  The
251      POSIX.2 <code>regex.h</code> implementation exposed by the old
252      header is now available under the <code>ap_</code> namespace
253      from <code>ap_regex.h</code>.  Calls to <code>regcomp</code>,
254      <code>regexec</code> and so on can be replaced by calls to
255      <code>ap_regcomp</code>, <code>ap_regexec</code>.</dd>
256
257      <dt>DBD Framework (SQL Database API)</dt>
258
259      <dd><p>With Apache 1.x and 2.0, modules requiring an SQL backend
260      had to take responsibility for managing it themselves.  Apart
261      from reinventing the wheel, this can be very inefficient, for
262      example when several modules each maintain their own connections.</p>
263
264      <p>Apache 2.1 and later provides the <code>ap_dbd</code> API for
265      managing database connections (including optimized strategies
266      for threaded and unthreaded MPMs), while APR 1.2 and later provides
267      the <code>apr_dbd</code> API for interacting with the database.</p>
268
269      <p>New modules SHOULD now use these APIs for all SQL database
270      operations.  Existing applications SHOULD be upgraded to use it
271      where feasible, either transparently or as a recommended option
272      to their users.</p></dd>
273    </dl>
