48<h2><a name="overview" id="overview">Overview of the Apache EBCDIC Port</a></h2>
49
50    
51
52    <p>Version 1.3 of the Apache HTTP Server was the first version
53    which included a port to a (non-ASCII) mainframe machine which
54    uses the EBCDIC character set as its native codeset.</p>
55
56    <p>(It is the SIEMENS family of mainframes running the <a href="http://www.siemens.de/servers/bs2osd/osdbc_us.htm">BS2000/OSD
57    operating system</a>. This mainframe OS nowadays features a
58    SVR4-derived POSIX subsystem).</p>
59
60    <p>The port was started initially to</p>
61
62    <ul>
63      <li>prove the feasibility of porting <a href="http://httpd.apache.org/">the Apache HTTP server</a> to
64      this platform</li>
65
66      <li>find a "worthy and capable" successor for the venerable
67      <a href="http://www.w3.org/Daemon/">CERN-3.0</a> daemon
68      (which was ported a couple of years ago), and to</li>
69
70      <li>prove that Apache's preforking process model can on this
71      platform easily outperform the accept-fork-serve model used
72      by CERN by a factor of 5 or more.</li>
73    </ul>
74
75    <p>This document serves as a rationale to describe some of the
76    design decisions of the port to this machine.</p>
77
78  </div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>
79<div class="section">
80<h2><a name="design" id="design">Design Goals</a></h2>
81
82    
83
84    <p>One objective of the EBCDIC port was to maintain enough
85    backwards compatibility with the (EBCDIC) CERN server to make
86    the transition to the new server attractive and easy. This
87    required the addition of a configurable method to define
88    whether a HTML document was stored in ASCII (the only format
89    accepted by the old server) or in EBCDIC (the native document
90    format in the POSIX subsystem, and therefore the only realistic
91    format in which the other POSIX tools like <code>grep</code> or
92    <code>sed</code> could operate on the documents). The current
93    solution to this is a "pseudo-MIME-format" which is intercepted
94    and interpreted by the Apache server (see below). Future versions
95    might solve the problem by defining an "ebcdic-handler" for all
96    documents which must be converted.</p>
97
98  </div><div class="top"><a href="#page-header"><img alt="top" src="../images/up.gif" /></a></div>

100<h2><a name="technical" id="technical">Technical Solution</a></h2>
101
102    
103
104    <p>Since all Apache input and output is based upon the BUFF
105    data type and its methods, the easiest solution was to add the
106    conversion to the BUFF handling routines. The conversion must
107    be settable at any time, so a BUFF flag was added which defines
108    whether a BUFF object has currently enabled conversion or not.
109    This flag is modified at several points in the HTTP
110    protocol:</p>
111
112    <ul>
113      <li><strong>set</strong> before a request is received
114      (because the request and the request header lines are always
115      in ASCII format)</li>
116
117      <li><strong>set/unset</strong> when the request body is
118      received - depending on the content type of the request body
119      (because the request body may contain ASCII text or a binary
120      file)</li>
121
122      <li><strong>set</strong> before a reply header is sent
123      (because the response header lines are always in ASCII
124      format)</li>
125
126      <li><strong>set/unset</strong> when the response body is sent
127      - depending on the content type of the response body (because
128      the response body may contain text or a binary file)</li>
129    </ul>
130

133<h2><a name="porting" id="porting">Porting Notes</a></h2>
134
135    
136
137    <ol>
138      <li>
139        <p>The relevant changes in the source are <code>#ifdef</code>'ed
140        into two categories:</p>
141
142        <dl>
143          <dt><code><strong>#ifdef
144          CHARSET_EBCDIC</strong></code></dt>
145
146          <dd>
147            <p>Code which is needed for any EBCDIC based machine.
148            This includes character translations, differences in
149            contiguity of the two character sets, flags which
150            indicate which part of the HTTP protocol has to be
151            converted and which part doesn't <em>etc.</em></p>
152          </dd>
153
154          <dt><code><strong>#ifdef _OSD_POSIX</strong></code></dt>
155
156          <dd>
157            <p>Code which is needed for the SIEMENS BS2000/OSD
158            mainframe platform only. This deals with include file
159            differences and socket implementation topics which are
160            only required on the BS2000/OSD platform.</p>
161          </dd>
162        </dl>
163      </li>
164
165      <li>
166        <p>The possibility to translate between ASCII and EBCDIC at
167        the socket level (on BS2000 POSIX, there is a socket option
168        which supports this) was intentionally <em>not</em> chosen,
169        because the byte stream at the HTTP protocol level consists
170        of a mixture of protocol related strings and non-protocol
171        related raw file data. HTTP protocol strings are always
172        encoded in ASCII (the <code>GET</code> request, any Header: lines,
173        the chunking information <em>etc.</em>) whereas the file transfer
174        parts (<em>i.e.</em>, GIF images, CGI output <em>etc.</em>)
175        should usually be just "passed through" by the server. This
176        separation between "protocol string" and "raw data" is
177        reflected in the server code by functions like <code>bgets()</code>
178        or <code>rvputs()</code> for strings, and functions like
179        <code>bwrite()</code> for binary data. A global translation
180        of everything would therefore be inadequate.</p>
181
182        <p>(In the case of text files of course, provisions must be
183        made so that EBCDIC documents are always served in
184        ASCII)</p>
185      </li>
186
187      <li>
188        <p>This port therefore features a built-in protocol level
189        conversion for the server-internal strings (which the
190        compiler translated to EBCDIC strings) and thus for all
191        server-generated documents. The hard coded ASCII escapes
192        <code>\012</code> and <code>\015</code> which are ubiquitous
193        in the server code are an exception: they are already the binary
194        encoding of the ASCII <code>\n</code> and <code>\r</code> and
195        must not be converted to ASCII a second time.
196        This exception is only relevant for server-generated strings;
197        and <em>external</em> EBCDIC documents are not expected to
198        contain ASCII newline characters.</p>
199      </li>
200
201      <li>
202        <p>By examining the call hierarchy for the BUFF management
203        routines, I added an "ebcdic/ascii conversion layer" which
204        would be crossed on every puts/write/get/gets, and a
205        conversion flag which allowed enabling/disabling the
206        conversions on-the-fly. Usually, a document crosses this
207        layer twice from its origin source (a file or CGI output) to
208        its destination (the requesting client): <code>file -&gt;
209        Apache</code>, and <code>Apache -&gt; client</code>.</p>
210
211        <p>The server can now read the header lines of a CGI-script
212        output in EBCDIC format, and then find out that the remainder
213        of the script's output is in ASCII (like in the case of the
214        output of a WWW Counter program: the document body contains a
215        GIF image). All header processing is done in the native
216        EBCDIC format; the server then determines, based on the type
217        of document being served, whether the document body (except
218        for the chunking information, of course) is in ASCII already
219        or must be converted from EBCDIC.</p>
220      </li>
221
222      <li>
223        <p>For Text documents (MIME types text/plain, text/html
224        <em>etc.</em>), an implicit translation to ASCII can be
225        used, or (if the users prefer to store some documents in
226        raw ASCII form for faster serving, or because the files
227        reside on a NFS-mounted directory tree) can be served
228        without conversion.</p>
229
230        <p><strong>Example:</strong></p>
231
232        <p>to serve files with the suffix <code>.ahtml</code> as a
233        raw ASCII <code>text/html</code> document without implicit
234        conversion (and suffix <code>.ascii</code> as ASCII
235        <code>text/plain</code>), use the directives:</p>
236
237        <div class="example"><p><code>
238          AddType  text/x-ascii-html  .ahtml <br />
239          AddType  text/x-ascii-plain .ascii
240        </code></p></div>
241
242        <p>Similarly, any <code>text/foo</code> MIME type can be
243        served as "raw ASCII" by configuring a MIME type
244        "<code>text/x-ascii-foo</code>" for it using
245        <code>AddType</code>.</p>
246      </li>
247
248      <li>
249        <p>Non-text documents are always served "binary" without
250        conversion. This seems to be the most sensible choice for,
251        .<em>e.g.</em>, GIF/ZIP/AU file types. This of course
252        requires the user to copy them to the mainframe host using
253        the "<code>rcp -b</code>" binary switch.</p>
254      </li>
255
256      <li>
257        <p>Server parsed files are always assumed to be in native
258        (<em>i.e.</em>, EBCDIC) format as used on the machine, and
259        are converted after processing.</p>
260      </li>
261
262      <li>
263        <p>For CGI output, the CGI script determines whether a
264        conversion is needed or not: by setting the appropriate
265        Content-Type, text files can be converted, or GIF output can
266        be passed through unmodified. An example for the latter case
267        is the wwwcount program which we ported as well.</p>
268      </li>
269
270    </ol>

274<h2><a name="document" id="document">Document Storage Notes</a></h2>
275
276    
277
278    <h3><a name="binary" id="binary">Binary Files</a></h3>
279
280      
281
282      <p>All files with a <code>Content-Type:</code> which does not
283      start with <code>text/</code> are regarded as <em>binary
284      files</em> by the server and are not subject to any conversion.
285      Examples for binary files are GIF images, gzip-compressed files
286      and the like.</p>
287
288      <p>When exchanging binary files between the mainframe host and
289      a Unix machine or Windows PC, be sure to use the ftp "binary"
290      (<code>TYPE I</code>) command, or use the
291      <code>rcp -b</code> command from the mainframe host (the
292      <code>-b</code> switch is not supported in unix
293      <code>rcp</code>'s).</p>
294
295    
296
297    <h3><a name="text" id="text">Text Documents</a></h3>
298
299      
300
301      <p>The default assumption of the server is that Text Files
302      (<em>i.e.</em>, all files whose <code>Content-Type:</code>
303      starts with <code>text/</code>) are stored in the native
304      character set of the host, EBCDIC.</p>
305
306    
307
308    <h3><a name="ssi" id="ssi">Server Side Included Documents</a></h3>
309
310      
311
312      <p>SSI documents must currently be stored in EBCDIC only.
313      No provision is made to convert it from ASCII before
314      processing.</p>
