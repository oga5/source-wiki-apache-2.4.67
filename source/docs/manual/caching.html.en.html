45<h2><a name="introduction" id="introduction">Introduction</a></h2>
46    
47    
48    <p>The Apache HTTP server offers a range of caching features that
49    are designed to improve the performance of the server in various
50    ways.</p>
51
52    <dl>
53        <dt>Three-state RFC2616 HTTP caching</dt>
54        <dd>
55            <code class="module"><a href="./mod/mod_cache.html">mod_cache</a></code>
56            and its provider modules
57            <code class="module"><a href="./mod/mod_cache_disk.html">mod_cache_disk</a></code>
58            provide intelligent, HTTP-aware caching. The content itself is stored
59            in the cache, and mod_cache aims to honor all of the various HTTP
60            headers and options that control the cacheability of content
61            as described in
62            <a href="http://www.w3.org/Protocols/rfc2616/rfc2616-sec13.html">Section
63            13 of RFC2616</a>.
64            <code class="module"><a href="./mod/mod_cache.html">mod_cache</a></code>
65            is aimed at both simple and complex caching configurations, where
66            you are dealing with proxied content, dynamic local content or
67            have a need to speed up access to local files on a potentially
68            slow disk.
69        </dd>
70
71        <dt>Two-state key/value shared object caching</dt>
72        <dd>
73            The <a href="socache.html">shared object cache API</a> (socache)
74            and its provider modules provide a
75            server wide key/value based shared object cache. These modules
76            are designed to cache low level data such as SSL sessions and
77            authentication credentials. Backends allow the data to be stored
78            server wide in shared memory, or datacenter wide in a cache such
79            as memcache or distcache.
80        </dd>
81
82        <dt>Specialized file caching</dt>
83        <dd>
84            <code class="module"><a href="./mod/mod_file_cache.html">mod_file_cache</a></code>
85            offers the ability to pre-load
86            files into memory on server startup, and can improve access
87            times and save file handles on files that are accessed often,
88            as there is no need to go to disk on each request.
89        </dd>
90    </dl>
91
92    <p>To get the most from this document, you should be familiar with
93    the basics of HTTP, and have read the Users' Guides to
94    <a href="urlmapping.html">Mapping URLs to the Filesystem</a> and
95    <a href="content-negotiation.html">Content negotiation</a>.</p>

99<h2><a name="http-caching" id="http-caching">Three-state RFC2616 HTTP caching</a></h2>
100
101    
102
103    <table class="related"><tr><th>Related Modules</th><th>Related Directives</th></tr><tr><td><ul><li><code class="module"><a href="./mod/mod_cache.html">mod_cache</a></code></li><li><code class="module"><a href="./mod/mod_cache_disk.html">mod_cache_disk</a></code></li></ul></td><td><ul><li><code class="directive"><a href="./mod/mod_cache.html#cacheenable">CacheEnable</a></code></li><li><code class="directive"><a href="./mod/mod_cache.html#cachedisable">CacheDisable</a></code></li><li><code class="directive"><a href="./mod/core.html#usecanonicalname">UseCanonicalName</a></code></li><li><code class="directive"><a href="./mod/mod_negotiation.html#cachenegotiateddocs">CacheNegotiatedDocs</a></code></li></ul></td></tr></table>
104
105    <p>The HTTP protocol contains built in support for an in-line caching
106    mechanism 
107    <a href="http://www.w3.org/Protocols/rfc2616/rfc2616-sec13.html">
108    described by section 13 of RFC2616</a>, and the
109    <code class="module"><a href="./mod/mod_cache.html">mod_cache</a></code> module can be used to take advantage of
110    this.</p>
111
112    <p>Unlike a simple two state key/value cache where the content
113    disappears completely when no longer fresh, an HTTP cache includes
114    a mechanism to retain stale content, and to ask the origin server
115    whether this stale content has changed and if not, make it fresh
116    again.</p>
117
118    <p>An entry in an HTTP cache exists in one of three states:</p>
119
120    <dl>
121    <dt>Fresh</dt>
122    <dd>
123        If the content is new enough (younger than its <strong>freshness
124        lifetime</strong>), it is considered <strong>fresh</strong>. An
125        HTTP cache is free to serve fresh content without making any
126        calls to the origin server at all.
127    </dd>
128    <dt>Stale</dt>
129    <dd>
130        <p>If the content is too old (older than its <strong>freshness
131        lifetime</strong>), it is considered <strong>stale</strong>. An
132        HTTP cache should contact the origin server and check whether
133        the content is still fresh before serving stale content to a
134        client. The origin server will either respond with replacement
135        content if not still valid, or ideally, the origin server will
136        respond with a code to tell the cache the content is still
137        fresh, without the need to generate or send the content again.
138        The content becomes fresh again and the cycle continues.</p>
139
140        <p>The HTTP protocol does allow the cache to serve stale data
141        under certain circumstances, such as when an attempt to freshen
142        the data with an origin server has failed with a 5xx error, or
143        when another request is already in the process of freshening
144        the given entry. In these cases a <code>Warning</code> header
145        is added to the response.</p>
146    </dd>
147    <dt>Non Existent</dt>
148    <dd>
149        If the cache gets full, it reserves the option to delete content
150        from the cache to make space. Content can be deleted at any time,
151        and can be stale or fresh. The <a href="programs/htcacheclean.html">htcacheclean</a> tool can be
152        run on a once off basis, or deployed as a daemon to keep the size
153        of the cache within the given size, or the given number of inodes.
154        The tool attempts to delete stale content before attempting to
155        delete fresh content.
156    </dd>
157    </dl>
158
159    <p>Full details of how HTTP caching works can be found in
160    <a href="http://www.w3.org/Protocols/rfc2616/rfc2616-sec13.html">
161    Section 13 of RFC2616</a>.</p>
162
163    <h3>Interaction with the Server</h3>
164      
165
166      <p>The <code class="module"><a href="./mod/mod_cache.html">mod_cache</a></code> module hooks into the server in two
167      possible places depending on the value of the
168      <code class="directive"><a href="./mod/mod_cache.html#cachequickhandler">CacheQuickHandler</a></code> directive:
169      </p>
170
171      <dl>
172        <dt>Quick handler phase</dt>
173        <dd>
174          <p>This phase happens very early on during the request processing,
175              just after the request has been parsed. If the content is
176              found within the cache, it is served immediately and almost
177              all request processing is bypassed.</p>
178
179              <p>In this scenario, the cache behaves as if it has been "bolted
180              on" to the front of the server.</p>
181              
182              <p>This mode offers the best performance, as the majority of
183              server processing is bypassed. This mode however also bypasses the
184              authentication and authorization phases of server processing, so
185              this mode should be chosen with care when this is important.</p>
186  
187              <p> Requests with an "Authorization" header (for example, HTTP Basic
188              Authentication) are neither cacheable nor served from the cache 
189              when <code class="module"><a href="./mod/mod_cache.html">mod_cache</a></code> is running in this phase.</p>
190          </dd>
191          <dt>Normal handler phase</dt>
192          <dd>
193              <p>This phase happens late in the request processing, after all
194              the request phases have completed.</p>
195
196              <p>In this scenario, the cache behaves as if it has been "bolted
197              on" to the back of the server.</p>
198
199              <p>This mode offers the most flexibility, as the potential exists
200              for caching to occur at a precisely controlled point in the filter
201              chain, and cached content can be filtered or personalized before
202              being sent to the client.</p>
203          </dd>
204        </dl>
205
206        <p>If the URL is not found within the cache, <code class="module"><a href="./mod/mod_cache.html">mod_cache</a></code>
207        will add a <a href="filter.html">filter</a> to the filter stack in order
208        to record the response to the cache, and then stand down, allowing normal
209        request processing to continue. If the content is determined to be
210        cacheable, the content will be saved to the cache for future serving,
211        otherwise the content will be ignored.</p>
212
213        <p>If the content found within the cache is stale, the
214        <code class="module"><a href="./mod/mod_cache.html">mod_cache</a></code> module converts the request into a
215        <strong>conditional request</strong>. If the origin server responds with
216        a normal response, the normal response is cached, replacing the content
217        already cached. If the origin server responds with a 304 Not Modified
218        response, the content is marked as fresh again, and the cached content
219        is served by the filter instead of saving it.</p>

222    <h3>Improving Cache Hits</h3>
223      
224
225      <p>When a virtual host is known by one of many different server aliases,
226      ensuring that <code class="directive"><a href="./mod/core.html#usecanonicalname">UseCanonicalName</a></code> is
227      set to <code>On</code> can dramatically improve the ratio of cache hits.
228      This is because the hostname of the virtual-host serving the content is
229      used within the cache key. With the setting set to <code>On</code>
230      virtual-hosts with multiple server names or aliases will not produce
231      differently cached entities, and instead content will be cached as
232      per the canonical hostname.</p>
233
234    
235
236    <h3>Freshness Lifetime</h3>
237      
238
239      <p>Well formed content that is intended to be cached should declare an
240      explicit freshness lifetime with the <code>Cache-Control</code>
241      header's <code>max-age</code> or <code>s-maxage</code> fields, or
242      by including an <code>Expires</code> header.</p>
243      
244      <p>At the same time, the origin server defined freshness lifetime can
245      be overridden by a client when the client presents their own
246      <code>Cache-Control</code> header within the request. In this case,
247      the lowest freshness lifetime between request and response wins.</p>
248
249      <p>When this freshness lifetime is missing from the request or the
250      response, a default freshness lifetime is applied. The default
251      freshness lifetime for cached entities is one hour, however
252      this can be easily over-ridden by using the <code class="directive"><a href="./mod/mod_cache.html#cachedefaultexpire">CacheDefaultExpire</a></code> directive.</p>
253
254      <p>If a response does not include an <code>Expires</code> header but does
255      include a <code>Last-Modified</code> header, <code class="module"><a href="./mod/mod_cache.html">mod_cache</a></code>
256      can infer a freshness lifetime based on a heuristic, which can be
257      controlled through the use of the <code class="directive"><a href="./mod/mod_cache.html#cachelastmodifiedfactor">CacheLastModifiedFactor</a></code> directive.</p>
258
259      <p>For local content, or for remote content that does not define its own
260      <code>Expires</code> header, <code class="module"><a href="./mod/mod_expires.html">mod_expires</a></code> may be used to
261      fine-tune the freshness lifetime by adding <code>max-age</code> and
262      <code>Expires</code>.</p>
263
264      <p>The maximum freshness lifetime may also be controlled by using the
265      <code class="directive"><a href="./mod/mod_cache.html#cachemaxexpire">CacheMaxExpire</a></code>.</p>
266

269    <h3>A Brief Guide to Conditional Requests</h3>
270      
271
272      <p>When content expires from the cache and becomes stale, rather than
273      pass on the original request, httpd will modify the request to make
274      it conditional instead.</p>
275
276      <p>When an <code>ETag</code> header exists in the original cached
277      response, <code class="module"><a href="./mod/mod_cache.html">mod_cache</a></code> will add an
278      <code>If-None-Match</code> header to the request to the origin server.
279      When a <code>Last-Modified</code> header exists in the original
280      cached response, <code class="module"><a href="./mod/mod_cache.html">mod_cache</a></code> will add an
281      <code>If-Modified-Since</code> header to the request to the origin
282      server. Performing either of these actions makes the request
283      <strong>conditional</strong>.</p>
284
285      <p>When a conditional request is received by an origin server, the
286      origin server should check whether the ETag or the Last-Modified
287      parameter has changed, as appropriate for the request. If not, the
288      origin should respond with a terse "304 Not Modified" response. This
289      signals to the cache that the stale content is still fresh should be
290      used for subsequent requests until the content's new freshness lifetime
291      is reached again.</p>
292
293      <p>If the content has changed, then the content is served as if the
294      request were not conditional to begin with.</p>
295
296      <p>Conditional requests offer two benefits. Firstly, when making such
297      a request to the origin server, if the content from the origin
298      matches the content in the cache, this can be determined easily and
299      without the overhead of transferring the entire resource.</p>
300
301      <p>Secondly, a well designed origin server will be designed in such
302      a way that conditional requests will be significantly cheaper to
303      produce than a full response. For static files, typically all that is
304      involved is a call to <code>stat()</code> or similar system call, to
305      see if the file has changed in size or modification time. As such, even
306      local content may still be served faster from the cache if it has not
307      changed.</p>
308      
309      <p>Origin servers should make every effort to support conditional
310      requests as is practical, however if conditional requests are not
311      supported, the origin will respond as if the request was not
312      conditional, and the cache will respond as if the content had changed
313      and save the new content to the cache. In this case, the cache will
314      behave like a simple two state cache, where content is effectively
315      either fresh or deleted.</p>

318    <h3>What Can be Cached?</h3>
319      
320
321      <p>The full definition of which responses can be cached by an HTTP
322      cache is defined in
323      <a href="http://www.w3.org/Protocols/rfc2616/rfc2616-sec13.html#sec13.4">
324      RFC2616 Section 13.4 Response Cacheability</a>, and can be summed up as
325      follows:</p>
326
327      <ol>
328        <li>Caching must be enabled for this URL. See the <code class="directive"><a href="./mod/mod_cache.html#cacheenable">CacheEnable</a></code> and <code class="directive"><a href="./mod/mod_cache.html#cachedisable">CacheDisable</a></code> directives.</li>
329
330        <li>If the response has an HTTP status code other than 200, 203, 300, 
331        301 or 410 it must also specify an "Expires" or "Cache-Control" header.
332        </li>
333
334        <li>The request must be a HTTP GET request.</li>
335
336        <li>If the response contains an "Authorization:" header, it must
337        also contain an "s-maxage", "must-revalidate" or "public" option
338        in the "Cache-Control:" header, or it won't be cached.</li>
339
340        <li>If the URL included a query string (e.g. from a HTML form GET
341        method) it will not be cached unless the response specifies an
342        explicit expiration by including an "Expires:" header or the max-age
343        or s-maxage directive of the "Cache-Control:" header, as per RFC2616
344        sections 13.9 and 13.2.1.</li>
345
346        <li>If the response has a status of 200 (OK), the response must
347        also include at least one of the "Etag", "Last-Modified" or
348        the "Expires" headers, or the max-age or s-maxage directive of
349        the "Cache-Control:" header, unless the
350        <code class="directive"><a href="./mod/mod_cache.html#cacheignorenolastmod">CacheIgnoreNoLastMod</a></code>
351        directive has been used to require otherwise.</li>
352
353        <li>If the response includes the "private" option in a "Cache-Control:"
354        header, it will not be stored unless the
355        <code class="directive"><a href="./mod/mod_cache.html#cachestoreprivate">CacheStorePrivate</a></code> has been
356        used to require otherwise.</li>
357
358        <li>Likewise, if the response includes the "no-store" option in a
359        "Cache-Control:" header, it will not be stored unless the
360        <code class="directive"><a href="./mod/mod_cache.html#cachestorenostore">CacheStoreNoStore</a></code> has been
361        used.</li>
362
363        <li>A response will not be stored if it includes a "Vary:" header
364        containing the match-all "*".</li>
365      </ol>
366    
367
368    <h3>What Should Not be Cached?</h3>
369      
370
371      <p>It should be up to the client creating the request, or the origin
372      server constructing the response to decide whether or not the content
373      should be cacheable or not by correctly setting the
374      <code>Cache-Control</code> header, and <code class="module"><a href="./mod/mod_cache.html">mod_cache</a></code> should
375      be left alone to honor the wishes of the client or server as appropriate.
376      </p>
377
378      <p>Content that is time sensitive, or which varies depending on the
379      particulars of the request that are not covered by HTTP negotiation,
380      should not be cached. This content should declare itself uncacheable
381      using the <code>Cache-Control</code> header.</p>
382      
383      <p>If content changes often, expressed by a freshness lifetime of minutes
384      or seconds, the content can still be cached, however it is highly
385      desirable that the origin server supports
386      <strong>conditional requests</strong> correctly to ensure that
387      full responses do not have to be generated on a regular basis.</p>
388
389      <p>Content that varies based on client provided request headers can be
390      cached through intelligent use of the <code>Vary</code> response
391      header.</p>
392
393    
394
395    <h3>Variable/Negotiated Content</h3>
396      
397
398      <p>When the origin server is designed to respond with different content
399      based on the value of headers in the request, for example to serve
400      multiple languages at the same URL, HTTP's caching mechanism makes it
401      possible to cache multiple variants of the same page at the same URL.</p>
402      
403      <p>This is done by the origin server adding a <code>Vary</code> header
404      to indicate which headers must be taken into account by a cache when
405      determining whether two variants are different from one another.</p>
406
407      <p>If for example, a response is received with a vary header such as;</p>
408
409      <div class="example"><p><code>
410Vary: negotiate,accept-language,accept-charset
411      </code></p></div>
412
413      <p><code class="module"><a href="./mod/mod_cache.html">mod_cache</a></code> will only serve the cached content to
414      requesters with accept-language and accept-charset headers
415      matching those of the original request.</p>
416      
417      <p>Multiple variants of the content can be cached side by side,
418      <code class="module"><a href="./mod/mod_cache.html">mod_cache</a></code> uses the <code>Vary</code> header and the
419      corresponding values of the request headers listed by <code>Vary</code>
420      to decide on which of many variants to return to the client.</p>

431    <h3><a name="disk" id="disk">Caching to Disk</a></h3>
432      
433
434      <p>The <code class="module"><a href="./mod/mod_cache.html">mod_cache</a></code> module relies on specific backend store
435      implementations in order to manage the cache, and for caching to disk
436      <code class="module"><a href="./mod/mod_cache_disk.html">mod_cache_disk</a></code> is provided to support this.</p>
437
438      <p>Typically the module will be configured as so;</p>
439
440      <pre class="prettyprint lang-config">CacheRoot   "/var/cache/apache/"
441CacheEnable disk /
442CacheDirLevels 2
443CacheDirLength 1</pre>
444
445
446      <p>Importantly, as the cached files are locally stored, operating system
447      in-memory caching will typically be applied to their access also. So
448      although the files are stored on disk, if they are frequently accessed
449      it is likely the operating system will ensure that they are actually
450      served from memory.</p>
451
452    
453
454    <h3>Understanding the Cache-Store</h3>
455      
456
457      <p>To store items in the cache, <code class="module"><a href="./mod/mod_cache_disk.html">mod_cache_disk</a></code> creates
458      a 22 character hash of the URL being requested. This hash incorporates
459      the hostname, protocol, port, path and any CGI arguments to the URL,
460      as well as elements defined by the Vary header to ensure that multiple
461      URLs do not collide with one another.</p>
462
463      <p>Each character may be any one of 64-different characters, which mean
464      that overall there are 64^22 possible hashes. For example, a URL might
465      be hashed to <code>xyTGxSMO2b68mBCykqkp1w</code>. This hash is used
466      as a prefix for the naming of the files specific to that URL within
467      the cache, however first it is split up into directories as per
468      the <code class="directive"><a href="./mod/mod_cache_disk.html#cachedirlevels">CacheDirLevels</a></code> and
469      <code class="directive"><a href="./mod/mod_cache_disk.html#cachedirlength">CacheDirLength</a></code>
470      directives.</p>
471
472      <p><code class="directive"><a href="./mod/mod_cache_disk.html#cachedirlevels">CacheDirLevels</a></code>
473      specifies how many levels of subdirectory there should be, and
474      <code class="directive"><a href="./mod/mod_cache_disk.html#cachedirlength">CacheDirLength</a></code>
475      specifies how many characters should be in each directory. With
476      the example settings given above, the hash would be turned into
477      a filename prefix as
478      <code>/var/cache/apache/x/y/TGxSMO2b68mBCykqkp1w</code>.</p>
479
480      <p>The overall aim of this technique is to reduce the number of
481      subdirectories or files that may be in a particular directory,
482      as most file-systems slow down as this number increases. With
483      setting of "1" for
484      <code class="directive"><a href="./mod/mod_cache_disk.html#cachedirlength">CacheDirLength</a></code>
485      there can at most be 64 subdirectories at any particular level.
486      With a setting of 2 there can be 64 * 64 subdirectories, and so on.
487      Unless you have a good reason not to, using a setting of "1"
488      for <code class="directive"><a href="./mod/mod_cache_disk.html#cachedirlength">CacheDirLength</a></code>
489      is recommended.</p>
490
491      <p>Setting
492      <code class="directive"><a href="./mod/mod_cache_disk.html#cachedirlevels">CacheDirLevels</a></code>
493      depends on how many files you anticipate to store in the cache.
494      With the setting of "2" used in the above example, a grand
495      total of 4096 subdirectories can ultimately be created. With
496      1 million files cached, this works out at roughly 245 cached
497      URLs per directory.</p>
498
499      <p>Each URL uses at least two files in the cache-store. Typically
500      there is a ".header" file, which includes meta-information about
501      the URL, such as when it is due to expire and a ".data" file
502      which is a verbatim copy of the content to be served.</p>
503
504      <p>In the case of a content negotiated via the "Vary" header, a
505      ".vary" directory will be created for the URL in question. This
506      directory will have multiple ".data" files corresponding to the
507      differently negotiated content.</p>
508    
509
510    <h3>Maintaining the Disk Cache</h3>
511      
512
513      <p>The <code class="module"><a href="./mod/mod_cache_disk.html">mod_cache_disk</a></code> module makes no attempt to
514      regulate the amount of disk space used by the cache, although it
515      will gracefully stand down on any disk error and behave as if the
516      cache was never present.</p>
517
518      <p>Instead, provided with httpd is the <a href="programs/htcacheclean.html">htcacheclean</a> tool which allows you
519      to clean the cache periodically. Determining how frequently to run <a href="programs/htcacheclean.html">htcacheclean</a> and what target size to
520      use for the cache is somewhat complex and trial and error may be needed to
521      select optimal values.</p>
522
523      <p><a href="programs/htcacheclean.html">htcacheclean</a> has two modes of
524      operation. It can be run as persistent daemon, or periodically from
525      cron. <a href="programs/htcacheclean.html">htcacheclean</a> can take up to an hour
526      or more to process very large (tens of gigabytes) caches and if you are
527      running it from cron it is recommended that you determine how long a typical
528      run takes, to avoid running more than one instance at a time.</p>
529 
530      <p>It is also recommended that an appropriate "nice" level is chosen for
531      htcacheclean so that the tool does not cause excessive disk io while the
532      server is running.</p>
533
534      <p class="figure">
535      <img src="images/caching_fig1.gif" alt="" width="600" height="406" /><br />
536      <a id="figure1" name="figure1"><dfn>Figure 1</dfn></a>: Typical
537      cache growth / clean sequence.</p>
538
539      <p>Because <code class="module"><a href="./mod/mod_cache_disk.html">mod_cache_disk</a></code> does not itself pay attention
540      to how much space is used you should ensure that
541      <a href="programs/htcacheclean.html">htcacheclean</a> is configured to
542      leave enough "grow room" following a clean.</p>

545    <h3><a name="memcache" id="memcache">Caching to memcached</a></h3>
546      
547
548      <p>Using the <code class="module"><a href="./mod/mod_cache_socache.html">mod_cache_socache</a></code> module, <code class="module"><a href="./mod/mod_cache.html">mod_cache</a></code>
549      can cache data from a variety of implementations (aka: "providers"). Using the
550      <code class="module"><a href="./mod/mod_socache_memcache.html">mod_socache_memcache</a></code> module, for example, one can specify that
551      <a href="http://memcached.org">memcached</a> is to be used as the
552      the backend storage mechanism.</p>
553
554      <p>Typically the module will be configured as so:</p>
555
556      <pre class="prettyprint lang-config">CacheEnable socache /
557CacheSocache memcache:memcd.example.com:11211</pre>
558
559
560      <p>Additional <code>memcached</code> servers can be specified by
561      appending them to the end of the <code>CacheSocache memcache:</code>
562      line separated by commas:</p>
563
564      <pre class="prettyprint lang-config">CacheEnable socache /
565CacheSocache memcache:mem1.example.com:11211,mem2.example.com:11212</pre>
566
567
568      <p>This format is also used with the other various <code class="module"><a href="./mod/mod_cache_socache.html">mod_cache_socache</a></code>
569      providers. For example:</p>
570
571      <pre class="prettyprint lang-config">CacheEnable socache /
572CacheSocache shmcb:/path/to/datafile(512000)</pre>
573
574
575      <pre class="prettyprint lang-config">CacheEnable socache /
576CacheSocache dbm:/path/to/datafile</pre>

629<h2><a name="file-caching" id="file-caching">Specialized File Caching</a></h2>
630
631    
632
633    <table class="related"><tr><th>Related Modules</th><th>Related Directives</th></tr><tr><td><ul><li><code class="module"><a href="./mod/mod_file_cache.html">mod_file_cache</a></code></li></ul></td><td><ul><li><code class="directive"><a href="./mod/mod_file_cache.html#cachefile">CacheFile</a></code></li><li><code class="directive"><a href="./mod/mod_file_cache.html#mmapfile">MMapFile</a></code></li></ul></td></tr></table>
634
635    <p>On platforms where a filesystem might be slow, or where file
636    handles are expensive, the option exists to pre-load files into
637    memory on startup.</p>
638
639    <p>On systems where opening files is slow, the option exists to
640    open the file on startup and cache the file handle. These
641    options can help on systems where access to static files is
642    slow.</p>
643
644    <h3><a name="filehandle" id="filehandle">File-Handle Caching</a></h3>
645      
646
647      <p>The act of opening a file can itself be a source of delay, particularly
648      on network filesystems. By maintaining a cache of open file descriptors
649      for commonly served files, httpd can avoid this delay. Currently httpd
650      provides one implementation of File-Handle Caching.</p>
651
652      <h4>CacheFile</h4>
653        
654
655        <p>The most basic form of caching present in httpd is the file-handle
656        caching provided by <code class="module"><a href="./mod/mod_file_cache.html">mod_file_cache</a></code>. Rather than caching
657        file-contents, this cache maintains a table of open file descriptors. Files
658        to be cached in this manner are specified in the configuration file using
659        the <code class="directive"><a href="./mod/mod_file_cache.html#cachefile">CacheFile</a></code>
660        directive.</p>
661
662        <p>The
663        <code class="directive"><a href="./mod/mod_file_cache.html#cachefile">CacheFile</a></code> directive
664        instructs httpd to open the file when it is started and to re-use
665        this file-handle for all subsequent access to this file.</p>
666
667        <pre class="prettyprint lang-config">CacheFile /usr/local/apache2/htdocs/index.html</pre>
668
669
670        <p>If you intend to cache a large number of files in this manner, you
671        must ensure that your operating system's limit for the number of open
672        files is set appropriately.</p>
673
674        <p>Although using <code class="directive"><a href="./mod/mod_file_cache.html#cachefile">CacheFile</a></code>
675        does not cause the file-contents to be cached per-se, it does mean
676        that if the file changes while httpd is running these changes will
677        not be picked up. The file will be consistently served as it was
678        when httpd was started.</p>
679
680        <p>If the file is removed while httpd is running, it will continue
681        to maintain an open file descriptor and serve the file as it was when
682        httpd was started. This usually also means that although the file
683        will have been deleted, and not show up on the filesystem, extra free
684        space will not be recovered until httpd is stopped and the file
685        descriptor closed.</p>
686      
687
688    
689
690    <h3><a name="inmemory" id="inmemory">In-Memory Caching</a></h3>
691      
692
693      <p>Serving directly from system memory is universally the fastest method
694      of serving content. Reading files from a disk controller or, even worse,
695      from a remote network is orders of magnitude slower. Disk controllers
696      usually involve physical processes, and network access is limited by
697      your available bandwidth. Memory access on the other hand can take mere
698      nano-seconds.</p>
699
700      <p>System memory isn't cheap though, byte for byte it's by far the most
701      expensive type of storage and it's important to ensure that it is used
702      efficiently. By caching files in memory you decrease the amount of
703      memory available on the system. As we'll see, in the case of operating
704      system caching, this is not so much of an issue, but when using
705      httpd's own in-memory caching it is important to make sure that you
706      do not allocate too much memory to a cache. Otherwise the system
707      will be forced to swap out memory, which will likely degrade
708      performance.</p>
709
710      <h4>Operating System Caching</h4>
711        
712
713        <p>Almost all modern operating systems cache file-data in memory managed
714        directly by the kernel. This is a powerful feature, and for the most
715        part operating systems get it right. For example, on Linux, let's look at
716        the difference in the time it takes to read a file for the first time
717        and the second time;</p>
718
719        <div class="example"><pre>colm@coroebus:~$ time cat testfile &gt; /dev/null
720real    0m0.065s
721user    0m0.000s
722sys     0m0.001s
723colm@coroebus:~$ time cat testfile &gt; /dev/null
724real    0m0.003s
725user    0m0.003s
726sys     0m0.000s</pre></div>
727
728        <p>Even for this small file, there is a huge difference in the amount
729        of time it takes to read the file. This is because the kernel has cached
730        the file contents in memory.</p>
731
732        <p>By ensuring there is "spare" memory on your system, you can ensure
733        that more and more file-contents will be stored in this cache. This
734        can be a very efficient means of in-memory caching, and involves no
735        extra configuration of httpd at all.</p>
736
737        <p>Additionally, because the operating system knows when files are
738        deleted or modified, it can automatically remove file contents from the
739        cache when necessary. This is a big advantage over httpd's in-memory
740        caching which has no way of knowing when a file has changed.</p>
741      
742
743      <p>Despite the performance and advantages of automatic operating system
744      caching there are some circumstances in which in-memory caching may be
745      better performed by httpd.</p>
746
747      <h4>MMapFile Caching</h4>
748        
749
750        <p><code class="module"><a href="./mod/mod_file_cache.html">mod_file_cache</a></code> provides the
751        <code class="directive"><a href="./mod/mod_file_cache.html#mmapfile">MMapFile</a></code> directive, which
752        allows you to have httpd map a static file's contents into memory at
753        start time (using the mmap system call). httpd will use the in-memory
754        contents for all subsequent accesses to this file.</p>
755
756        <pre class="prettyprint lang-config">MMapFile /usr/local/apache2/htdocs/index.html</pre>
757
758
759        <p>As with the
760        <code class="directive"><a href="./mod/mod_file_cache.html#cachefile">CacheFile</a></code> directive, any
761        changes in these files will not be picked up by httpd after it has
762        started.</p>
763
764        <p> The <code class="directive"><a href="./mod/mod_file_cache.html#mmapfile">MMapFile</a></code>
765        directive does not keep track of how much memory it allocates, so
766        you must ensure not to over-use the directive. Each httpd child
767        process will replicate this memory, so it is critically important
768        to ensure that the files mapped are not so large as to cause the
769        system to swap memory.</p>

775<h2><a name="security" id="security">Security Considerations</a></h2>
776    
777
778    <h3>Authorization and Access Control</h3>
779      
780
781      <p>Using <code class="module"><a href="./mod/mod_cache.html">mod_cache</a></code> in its default state where
782      <code class="directive"><a href="./mod/mod_cache.html#cachequickhandler">CacheQuickHandler</a></code> is set to
783      <code>On</code> is very much like having a caching reverse-proxy bolted
784      to the front of the server. Requests will be served by the caching module
785      unless it determines that the origin server should be queried just as an
786      external cache would, and this drastically changes the security model of
787      httpd.</p>
788
789      <p>As traversing a filesystem hierarchy to examine potential
790      <code>.htaccess</code> files would be a very expensive operation,
791      partially defeating the point of caching (to speed up requests),
792      <code class="module"><a href="./mod/mod_cache.html">mod_cache</a></code> makes no decision about whether a cached
793      entity is authorized for serving. In other words; if
794      <code class="module"><a href="./mod/mod_cache.html">mod_cache</a></code> has cached some content, it will be served
795      from the cache as long as that content has not expired.</p>
796
797      <p>If, for example, your configuration permits access to a resource by IP
798      address you should ensure that this content is not cached. You can do this
799      by using the <code class="directive"><a href="./mod/mod_cache.html#cachedisable">CacheDisable</a></code>
800      directive, or <code class="module"><a href="./mod/mod_expires.html">mod_expires</a></code>. Left unchecked,
801      <code class="module"><a href="./mod/mod_cache.html">mod_cache</a></code> - very much like a reverse proxy - would cache
802      the content when served and then serve it to any client, on any IP
803      address.</p>
804
805      <p>When the <code class="directive"><a href="./mod/mod_cache.html#cachequickhandler">CacheQuickHandler</a></code>
806      directive is set to <code>Off</code>, the full set of request processing
807      phases are executed and the security model remains unchanged.</p>
808    
809
810    <h3>Local exploits</h3>
811      
812
813      <p>As requests to end-users can be served from the cache, the cache
814      itself can become a target for those wishing to deface or interfere with
815      content. It is important to bear in mind that the cache must at all
816      times be writable by the user which httpd is running as. This is in
817      stark contrast to the usually recommended situation of maintaining
818      all content unwritable by the Apache user.</p>
819
820      <p>If the Apache user is compromised, for example through a flaw in
821      a CGI process, it is possible that the cache may be targeted. When
822      using <code class="module"><a href="./mod/mod_cache_disk.html">mod_cache_disk</a></code>, it is relatively easy to
823      insert or modify a cached entity.</p>
824
825      <p>This presents a somewhat elevated risk in comparison to the other
826      types of attack it is possible to make as the Apache user. If you are
827      using <code class="module"><a href="./mod/mod_cache_disk.html">mod_cache_disk</a></code> you should bear this in mind -
828      ensure you upgrade httpd when security upgrades are announced and
829      run CGI processes as a non-Apache user using <a href="suexec.html">suEXEC</a> if possible.</p>
830
831    
832
833    <h3>Cache Poisoning</h3>
834      
835
836      <p>When running httpd as a caching proxy server, there is also the
837      potential for so-called cache poisoning. Cache Poisoning is a broad
838      term for attacks in which an attacker causes the proxy server to
839      retrieve incorrect (and usually undesirable) content from the origin
840      server.</p>
841
842      <p>For example if the DNS servers used by your system running httpd
843      are vulnerable to DNS cache poisoning, an attacker may be able to control
844      where httpd connects to when requesting content from the origin server.
845      Another example is so-called HTTP request-smuggling attacks.</p>
846
847      <p>This document is not the correct place for an in-depth discussion
848      of HTTP request smuggling (instead, try your favorite search engine)
849      however it is important to be aware that it is possible to make
850      a series of requests, and to exploit a vulnerability on an origin
851      webserver such that the attacker can entirely control the content
852      retrieved by the proxy.</p>
853    
854
855    <h3>Denial of Service / Cachebusting</h3>
856      
857
858      <p>The Vary mechanism allows multiple variants of the same URL to be
859      cached side by side. Depending on header values provided by the client,
860      the cache will select the correct variant to return to the client. This
861      mechanism can become a problem when an attempt is made to vary on a
862      header that is known to contain a wide range of possible values under
863      normal use, for example the <code>User-Agent</code> header. Depending
864      on the popularity of the particular web site thousands or millions of
865      duplicate cache entries could be created for the same URL, crowding
866      out other entries in the cache.</p>
867      
868      <p>In other cases, there may be a need to change the URL of a particular
869      resource on every request, usually by adding a "cachebuster" string to
870      the URL. If this content is declared cacheable by a server for a
871      significant freshness lifetime, these entries can crowd out
872      legitimate entries in a cache. While <code class="module"><a href="./mod/mod_cache.html">mod_cache</a></code>
873      provides a
874      <code class="directive"><a href="./mod/mod_cache.html#cacheignoreurlsessionidentifiers">CacheIgnoreURLSessionIdentifiers</a></code>
875      directive, this directive should be used with care to ensure that
876      downstream proxy or browser caches aren't subjected to the same denial
877      of service issue.</p>
