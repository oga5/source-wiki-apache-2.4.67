README.CHANGES
1Changes can be documented in two ways now: Either by directly editing the
2CHANGES file like it was done until now or by storing each entry for the
3CHANGES file correctly formated in a separate file in the changes-entries
4directory.
5
6The benefit of the single file per change approach is that it eases backporting
7the CHANGES entry to a stable branch as it avoids the frequent merge conflicts
8as changes are merged in different orders or not at all in the stable branch.
9
10In order to keep the current CHANGES file for the users as is there is a new
11make target called 'update-changes'. It merges all change files in the
12changes-entries directory to the top of the CHANGES file and removes them
13afterwards.
14
15This make target can be seen in a similar way as the scripts to update the
16documentation files from its xml sources. It can be executed immediately
17after the new file in the changes-entries directory has been created / merged
18and committed or it can be executed later. It should be executed at least before
19a release gets tagged.
