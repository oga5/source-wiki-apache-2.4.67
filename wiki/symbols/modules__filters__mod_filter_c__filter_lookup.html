filter_lookup

定義

  • 種別: function
  • 定義箇所: modules/filters/mod_filter.c:135

目的

リクエストのコンテキスト（Content-Type または ap_expr 式）に基づき、登録されたフィルタプロバイダの中から適切なものを選択し、実行関数を特定する。

シグネチャまたは形状

static int filter_lookup(ap_filter_t *f, ap_filter_rec_t *filter)

入力と出力

  • **入力**:
  • f: 現在のフィルタコンテキストを含む ap_filter_t 構造体。
  • filter: 検索対象となるフィルタ定義 (ap_filter_rec_t)。
  • **出力**:
  • int: プロバイダが見つかり、適用準備が整った場合は 1、それ以外は 0 を返す。

前提条件と不変条件

  • f->r（リクエストレコード）が有効であること。
  • filter->providers にプロバイダのリストが登録されていること。
  • プロバイダの条件判定には ap_expr または Content-Type が使用される。

アルゴリズムまたは振る舞い

1. 登録されたプロバイダを順次走査する。 2. provider->expr が定義されている場合、ap_expr_exec で評価を行う。 3. expr がない場合、r->content_type と provider->types を比較し、一致を確認する。 4. 条件が一致した場合、プロトコルフラグ (proto_flags) に基づき、必要に応じて Cache-Control や Content-Length などの HTTP ヘッダを調整する。 5. 選択されたプロバイダの実行関数を ctx->func にセットし、1 を返す。

呼び出し元と呼び出し先

  • **呼び出し元**: filter_harness (modules/filters/mod_filter.c:312)
  • **呼び出し先**:
  • ap_expr_exec (条件式評価)
  • ap_log_rerror (ログ出力)
  • apr_table_unset / apr_table_addn / apr_table_setn (HTTP ヘッダ操作)

関連データ構造

  • ap_filter_provider_t: 評価条件とプロバイダのペアを保持する連結リスト。
  • harness_ctx: 実行時のコンテキスト。選択された関数ポインタを保持。
  • mod_filter_ctx: リクエストごとの設定コンテキスト。

失敗モードと境界条件

  • **条件不一致**: どのプロバイダも条件に合致しない場合、0 を返す。
  • **式評価エラー**: ap_expr_exec がエラーを返した場合、ログを出力し、そのプロバイダはスキップされる。
  • **プロキシ環境**: r->proxyreq が真の場合、AP_FILTER_PROTO_NO_PROXY フラグを持つプロバイダは無視される。
  • **Content-Type 不在**: リクエストに Content-Type がない場合、そのプロバイダはマッチしない。

保守担当者向けメモ

  • プロトコルヘッダの調整ロジックは「ハック」であり、フィルタ実装者がヘッダ修正の負担を負わないようにするための暫定的な措置である。
  • filter_lookup 内でのヘッダ操作は、フィルタ適用による副作用（Content-Length の不整合など）を補正するために行われる。
