baton_t

定義

  • 種別: struct
  • 定義箇所: modules/proxy/mod_proxy_hcheck.c:71

目的

ヘルスチェック実行時に、対象となるワーカー情報や実行コンテキストを保持し、各チェック関数（TCP, CPING, HTTP）へ受け渡すためのデータ構造です。

シグネチャまたは形状

typedef struct {
    apr_pool_t *ptemp;         /* チェック実行ごとの一時メモリプール */
    sctx_t *ctx;               /* サーバー単位のモジュール設定 */
    proxy_balancer *balancer;  /* 対象ワーカーが属するバランサー */
    proxy_worker *worker;      /* チェック対象のワーカー */
    proxy_worker *hc;          /* ヘルスチェック用のワーカー設定 */
    apr_time_t *now;           /* チェック実行時刻 */
} baton_t;

入力と出力

  • **入力**: hc_init_baton 等で初期化され、各チェック関数へ引数として渡されます。
  • **出力**: 構造体内の各フィールドは、hc_check 実行時の判定ロジックや状態更新に使用されます。

前提条件と不変条件

  • ptemp はチェック実行ごとに作成・破棄される必要があり、リクエスト処理中のメモリリークを防ぐ不変条件として機能します。
  • worker および hc は、ヘルスチェック対象として有効な proxy_worker を指している必要があります。

アルゴリズムまたは振る舞い

  • mod_watchdog がトリガーするヘルスチェックにおいて、対象ワーカーの情報を集約します。
  • hc_check 関数群がこの構造体を参照し、バックエンドへの接続確立やリクエスト送信を行います。
  • apr_pcalloc を用いて動的に割り当てられ、チェック終了後に ptemp と共に解放されます。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • modules/proxy/mod_proxy_hcheck.c 内の hc_init_baton
  • hc_check 関連の実行フロー（hc_check_cping, hc_check_tcp, hc_check_http）
  • apr_pcalloc を呼び出す箇所（modules/proxy/mod_proxy_hcheck.c:1056）
  • **呼び出し先**:
  • 本構造体自体はデータ保持用であり、関数を呼び出すことはありません。

関連データ構造

  • sctx_t: サーバー単位の設定コンテキスト。
  • proxy_worker: プロキシのバックエンドワーカー定義。
  • proxy_balancer: ロードバランサーの定義。
  • apr_pool_t: メモリ管理用プール。

失敗モードと境界条件

  • ptemp の割り当て失敗時、または baton_t の初期化失敗時にヘルスチェックタスクが中断される可能性があります。
  • worker や balancer が NULL の状態で参照された場合、セグメンテーション違反が発生するリスクがあります。

保守担当者向けメモ

  • HC_USE_THREADS が有効な場合、スレッドプールを介してこの構造体が渡されるため、スレッド間でのメモリ整合性に注意してください。
  • ptemp のライフサイクルは baton_t の生存期間と一致させる必要があります。
