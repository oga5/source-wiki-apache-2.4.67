md_acmev2_drive_renew

定義

  • 種別: function
  • 定義箇所: modules/md/md_acmev2_drive.c:151

目的

ACMEv2 プロトコルを用いた証明書の更新プロセスを統括し、注文の作成から検証、証明書の取得までの一連のライフサイクルを管理する。

シグネチャまたは形状

apr_status_t md_acmev2_drive_renew(md_acme_driver_t *ad, md_proto_driver_t *d, md_result_t *result)

入力と出力

  • **入力**:
  • ad: ACME ドライバのコンテキスト。
  • d: プロトコルドライバのコンテキスト。
  • result: 処理結果およびエラー情報を格納する構造体。
  • **出力**:
  • apr_status_t: 処理の成功 (APR_SUCCESS) または失敗を示すステータスコード。

前提条件と不変条件

  • md_acme_drive_set_acct により、有効な ACME アカウントが設定されている必要がある。
  • 既に証明書チェーンが存在する場合、本関数は処理を中断する。

アルゴリズムまたは振る舞い

1. **アカウント設定**: md_acme_drive_set_acct を呼び出し、使用する ACME アカウントを確定する。 2. **注文の準備**: ad_setup_order を呼び出し、既存の注文をロードするか、新規注文を作成する。 3. **注文の更新と検証**: サーバーから注文状態を取得し、INVALID であればローカルの注文データを破棄して再試行する。 4. **チャレンジ実行**: md_acme_order_start_challenges によりドメイン認証を開始する。EINVAL が発生した場合は注文を破棄して再試行する。 5. **状態監視**:

  • md_acme_order_monitor_authzs: ドメイン検証の完了を待機。
  • md_acme_order_await_ready: 注文が READY 状態になるのを待機。
  • READY 状態であれば md_acme_drive_setup_cred_chain を実行。
  • md_acme_order_await_valid: 注文が VALID 状態になるのを待機。

    • 6. **完了**: 証明書 URL が存在することを確認し、成功を返す。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • modules/md/md_acme_drive.c:805
  • **呼び出し先**:
  • ad_setup_order
  • md_acme_drive_set_acct
  • md_acme_order_update
  • md_acme_order_purge
  • md_acme_order_start_challenges
  • md_acme_order_monitor_authzs
  • md_acme_order_await_ready
  • md_acme_drive_setup_cred_chain
  • md_acme_order_await_valid
  • md_result_set / md_result_log

関連データ構造

  • md_acme_driver_t: ACME 固有のドライバ状態。
  • md_proto_driver_t: プロトコル共通のドライバ状態。
  • md_result_t: 処理結果の記録用。

失敗モードと境界条件

  • **注文無効化**: サーバー側で注文が INVALID と判断された場合、ローカルの注文データがパージされる。
  • **検証失敗**: md_acme_order_monitor_authzs 等の待機関数がタイムアウトまたはエラーを返した場合、処理を中断する。
  • **証明書欠落**: 注文が VALID 状態であっても証明書 URL が取得できない場合は APR_EINVAL を返す。

保守担当者向けメモ

  • 注文の再試行ロジック（retry ラベル）は、ドメイン構成変更などでサーバー側から EINVAL が返された場合の自動復旧を目的としている。
  • デバッグ時は md_result_t に記録されるアクティビティログを確認することで、ステートマシンのどの段階で停止しているか特定可能。
