proxy_wstunnel_canon

定義

  • 種別: function
  • 定義箇所: modules/proxy/mod_proxy_wstunnel.c:61

目的

WebSocket プロキシリクエストの URL を解析・正規化し、バックエンド接続用の内部形式（proxy:ws://... または proxy:wss://...）へ変換します。

シグネチャまたは形状

static int proxy_wstunnel_canon(request_rec *r, char *url)

入力と出力

  • **入力**:
  • r: 現在のリクエスト構造体。
  • url: 正規化対象の URL 文字列。
  • **出力**:
  • OK: 成功。r->filename に正規化されたプロキシ URL が格納される。
  • DECLINED: 処理対象外（スキーム不一致やフォールバック設定時）。
  • HTTP_BAD_REQUEST: URL 解析失敗。
  • HTTP_FORBIDDEN: パスまたはクエリ文字列に不正な制御文字が含まれる場合。

前提条件と不変条件

  • r->per_dir_config から proxyws_dir_conf を取得し、フォールバック設定を確認します。
  • URL が ws: または wss: で始まる必要があります。
  • proxy-nocanon または proxy-noencode ノートの設定に基づき、パスのエンコード処理が分岐します。

アルゴリズムまたは振る舞い

1. **フォールバック判定**: can_fallback_to_proxy_http が有効かつ設定で許可されている場合、DECLINED を返して mod_proxy_http に委譲します。 2. **スキーム解析**: ws: または wss: を識別し、デフォルトポート（80 または 443）を設定します。 3. **ネットワーク位置の解析**: ap_proxy_canon_netloc を使用してホスト名とポートを抽出します。 4. **パスとクエリの正規化**: ap_proxy_canonenc_ex を用いてパスをエンコードし、クエリ文字列を結合します。 5. **バリデーション**: パスおよびクエリ文字列に制御文字やスペースが含まれていないか検証します。 6. **URL 再構築**: IPv6 アドレスの括弧補完やポート番号の付与を行い、proxy: スキームを付与した文字列を r->filename に設定します。

呼び出し元と呼び出し先

  • **呼び出し元**: mod_proxy のフックシステム（proxy_hook_canon_handler に登録）。
  • **呼び出し先**:
  • ap_get_module_config, ap_log_rerror, ap_proxy_canon_netloc, ap_proxy_canonenc_ex, apr_pstrcat 等の Apache/APR API。

関連データ構造

  • request_rec: リクエスト情報および設定保持。
  • proxyws_dir_conf: mod_proxy_wstunnel 固有のディレクトリ設定。

失敗モードと境界条件

  • **不正な URL**: ap_proxy_canon_netloc がエラーを返すと HTTP_BAD_REQUEST となります。
  • **不正な文字**: パスやクエリに制御文字が含まれる場合、セキュリティ上の理由から HTTP_FORBIDDEN を返します。
  • **エンコード失敗**: ap_proxy_canonenc_ex が NULL を返すと HTTP_BAD_REQUEST となります。

保守担当者向けメモ

  • 本関数は proxy_hook_canon_handler を通じて呼び出されるため、他のプロキシモジュールとの優先順位（APR_HOOK_FIRST）に注意してください。
  • r->filename を書き換えることで、後続のプロキシハンドラが接続先を特定できるようにしています。
