systemd_post_config

定義

  • 種別: function
  • 定義箇所: modules/arch/unix/mod_systemd.c:73

目的

Apache HTTP Server の設定読み込み完了後に、systemd に対してサービスが準備完了状態（READY=1）であることを通知します。

シグネチャまたは形状

static int systemd_post_config(apr_pool_t *pconf, apr_pool_t *plog,
                               apr_pool_t *ptemp, server_rec *main_server)

入力と出力

  • **入力**:
  • pconf, plog, ptemp: Apache のメモリプール。
  • main_server: サーバー設定構造体。
  • **出力**:
  • 成功時またはスキップ時に OK を返します。

前提条件と不変条件

  • ap_state_query(AP_SQ_MAIN_STATE) が AP_SQ_MS_CREATE_PRE_CONFIG である場合、通知を行わずに即座に終了します。
  • HAVE_SELINUX が定義されている場合、通知の前に log_selinux_context() が実行されます。

アルゴリズムまたは振る舞い

1. サーバーの状態を確認し、初期設定フェーズであれば処理を中断します。 2. SELinux が有効な環境であれば、現在のコンテキストをログに出力します。 3. sd_notify を呼び出し、systemd に対して以下のステータスを送信します。

  • READY=1: サービスがリクエストを受け付け可能であることを通知。
  • STATUS=Configuration loaded.: サービスの状態文字列を更新。

    • 4. 処理完了後、OK を返します。

呼び出し元と呼び出し先

  • **呼び出し元**: Apache の post_config フック（modules/arch/unix/mod_systemd.c:164 にて ap_hook_post_config に登録）。
  • **呼び出し先**:
  • ap_state_query (Apache API)
  • log_selinux_context (内部ヘルパー、HAVE_SELINUX 定義時)
  • sd_notify (libsystemd)

関連データ構造

  • server_rec: サーバー設定情報を保持する構造体。
  • apr_pool_t: メモリ管理用プール。

失敗モードと境界条件

  • sd_notify の戻り値は確認されておらず、通知の失敗が本関数の戻り値（OK）に影響することはありません。
  • 早期リターン条件（AP_SQ_MS_CREATE_PRE_CONFIG）により、設定の初期段階では通知が抑制されます。

保守担当者向けメモ

  • 本関数は APR_HOOK_REALLY_LAST で登録されており、他の設定完了処理がすべて終了した後に実行されることが保証されています。
  • sd_notify は systemd のライブラリ関数であり、本モジュールはこれを通じてサービスマネージャと通信します。
