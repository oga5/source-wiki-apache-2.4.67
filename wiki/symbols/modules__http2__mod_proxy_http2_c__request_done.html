request_done

定義

  • 種別: function
  • 定義箇所: modules/http2/mod_proxy_http2.c:233

目的

プロキシリクエストの完了時に、コンテキストの状態を更新し、エラー処理および再試行の可否を判定する。

シグネチャまたは形状

static void request_done(h2_proxy_ctx *ctx, request_rec *r,
                         apr_status_t status, int touched, int error_code)

入力と出力

  • **入力**:
  • ctx: 現在のリクエストに関連付けられた h2_proxy_ctx 構造体。
  • r: 完了したリクエストの request_rec。
  • status: リクエスト処理の apr_status_t 結果。
  • touched: リクエストが既に処理を開始（送信）したかを示すフラグ。
  • error_code: 発生したエラーコード。
  • **出力**: なし（ctx 内のメンバ変数を更新する副作用を持つ）。

前提条件と不変条件

  • r == ctx->r である場合にのみ、コンテキストの更新が行われる。

アルゴリズムまたは振る舞い

1. リクエストがコンテキストと一致する場合、ログを出力する。 2. ctx->r_done を 1 に設定し、リクエスト完了をマークする。 3. touched が真であれば、ctx->r_may_retry を 0 に設定し、再試行を禁止する。 4. proxy-error-override が設定されている場合、ctx->r_status にリクエストのステータスを保存し、リクエストのステータスを OK に上書きする。 5. それ以外の場合、error_code があれば HTTP_BAD_GATEWAY を、そうでなければ status に基づいて OK または HTTP_SERVICE_UNAVAILABLE を ctx->r_status に設定する。

呼び出し元と呼び出し先

  • **呼び出し元**: modules/http2/mod_proxy_http2.c:257（session->user_data を経由して呼び出される）。
  • **呼び出し先**:
  • ap_log_cerror (Apache API)
  • apr_table_get (APR API)
  • ap_map_http_request_error (Apache API)

関連データ構造

  • h2_proxy_ctx: リクエスト処理の状態を保持する構造体。
  • request_rec: Apache のリクエストレコード。

失敗モードと境界条件

  • r が ctx->r と一致しない場合、関数は何も行わずに終了する。
  • error_code が 0 以外の場合、ctx->r_status は強制的に HTTP_BAD_GATEWAY となる。

保守担当者向けメモ

  • 本関数はリクエスト完了時の状態遷移を管理する重要なフックである。
  • proxy-error-override の有無によってステータスコードの決定ロジックが分岐するため、プロキシ設定との整合性に注意すること。
