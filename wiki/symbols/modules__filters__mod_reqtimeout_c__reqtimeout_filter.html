reqtimeout_filter

定義

  • 種別: function
  • 定義箇所: modules/filters/mod_reqtimeout.c:172

目的

HTTPリクエストの各段階（ヘッダー読み取り等）において、動的なタイムアウト制限を適用し、低速なクライアントによるリソース枯渇攻撃を防止する入力フィルタ。

シグネチャまたは形状

static apr_status_t reqtimeout_filter(ap_filter_t *f,
                                      apr_bucket_brigade *bb,
                                      ap_input_mode_t mode,
                                      apr_read_type_e block,
                                      apr_off_t readbytes)

入力と出力

  • **入力**:
  • f: フィルタコンテキスト（reqtimeout_con_cfg を保持）。
  • bb: 読み取ったデータを格納するバケットブリゲード。
  • mode, block, readbytes: ap_get_brigade に準ずる読み取りモードとパラメータ。
  • **出力**:
  • apr_status_t: 処理結果（成功時は APR_SUCCESS、タイムアウト時は APR_TIMEUP 等）。

前提条件と不変条件

  • f->ctx に reqtimeout_con_cfg が格納されていること。
  • 接続のソケットが ccfg->socket に設定されていること。
  • タイムアウト発生時、接続は AP_CONN_CLOSE に設定され、short-lingering-close ノートが有効化される。

アルゴリズムまたは振る舞い

1. **Keep-Alive処理**: in_keep_alive が設定されている場合、最初のバイトを待機してから通常のタイムアウト監視へ移行。 2. **タイムアウト設定**: cur_stage に基づき、timeout_at および max_timeout_at を更新。 3. **監視**:

  • AP_MODE_GETLINE の場合、行全体を読み取るために非ブロッキング読み取りと apr_poll（または apr_socket_wait）をループ内で実行し、厳密にタイムアウトを監視。
  • 読み取り成功時、rate_factor が設定されていれば extend_timeout を呼び出し、データ量に応じて期限を延長。

    • 4. **終了処理**: タイムアウト発生時、ログ出力を行い、接続を即時切断対象としてマーク。

呼び出し元と呼び出し先

  • **呼び出し元**: ap_get_brigade を経由して呼び出される（フィルタチェーンの一部）。
  • **呼び出し先**:
  • ap_get_brigade (後続フィルタ)
  • check_time_left
  • extend_timeout
  • have_lf_or_eos
  • brigade_append
  • apr_socket_timeout_get / set

関連データ構造

  • reqtimeout_con_cfg: 接続ごとの状態（期限、ソケット、一時バッファ tmpbb）を保持。
  • reqtimeout_stage_t: 現在のステージのタイムアウト設定。

失敗モードと境界条件

  • **タイムアウト**: check_time_left が APR_TIMEUP を返すと、接続を強制終了する。
  • **読み取りエラー**: ap_get_brigade がエラーを返した場合、そのステータスを伝播。
  • **エッジケース**: AP_MODE_SPECULATIVE モードでは、タイムアウト延長を行わない。

保守担当者向けメモ

  • APR_MAJOR_VERSION < 2 の環境では apr_poll を使用し、それ以降は apr_socket_wait を使用する抽象化が行われている。
  • AP_MODE_GETLINE 以外のモードでは、extend_timeout による動的延長が適用されない設計となっている。
