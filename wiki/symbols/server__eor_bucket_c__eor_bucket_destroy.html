eor_bucket_destroy

定義

  • 種別: function
  • 定義箇所: server/eor_bucket.c:93

目的

EOR (End of Request) バケットの破棄処理を実行し、関連付けられたリクエストプールを解放します。

シグネチャまたは形状

static void eor_bucket_destroy(void *data)

入力と出力

  • **入力**: void *data - 破棄対象の ap_bucket_eor 構造体へのポインタ。
  • **出力**: なし（void）。

前提条件と不変条件

  • data は ap_bucket_eor 型としてキャスト可能であること。
  • バケットが共有されている場合、参照カウントが 0 に達したときのみ破棄処理が実行されること。

アルゴリズムまたは振る舞い

1. apr_bucket_shared_destroy(h) を呼び出し、バケットの参照カウントを減算する。 2. 参照カウントが 0 に達した場合（破棄が許可された場合）:

  • バケット内の request_rec *r が存在するか確認する。
  • r が存在する場合、apr_pool_destroy(r->pool) を呼び出し、リクエストに関連するプールを破棄する。これにより、登録済みの eor_bucket_cleanup が発火し、ログ記録や統計更新が行われる。
  • 最後に apr_bucket_free(h) を呼び出し、バケット構造体自体のメモリを解放する。

呼び出し元と呼び出し先

  • **呼び出し元**: バケットの参照カウントが減少した際に、バケット管理機構から呼び出される（ソースコード内のコメントより、バケット破棄のライフサイクルの一部として言及あり）。
  • **呼び出し先**:
  • apr_bucket_shared_destroy
  • apr_pool_destroy
  • apr_bucket_free

関連データ構造

  • ap_bucket_eor: EOR バケットの内部構造体。
  • request_rec: リクエストの状態を保持する構造体。

失敗モードと境界条件

  • r が NULL の場合、プール破棄はスキップされる。
  • eor_bucket_cleanup との競合を防ぐため、プール破棄の順序には注意が必要（eor_bucket_cleanup 側で二重解放防止策が講じられている）。

保守担当者向けメモ

  • 本関数はバケットの破棄とリクエストプールの破棄を連動させる重要な接点です。
  • eor_bucket_cleanup がプール破棄時に実行されるため、この関数内での処理順序を変更する際は、クリーンアップフックへの影響を考慮してください。
