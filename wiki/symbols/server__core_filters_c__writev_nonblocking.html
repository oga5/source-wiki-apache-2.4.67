writev_nonblocking

定義

  • 種別: function
  • 定義箇所: server/core_filters.c:784

目的

apr_socket_sendv を使用して、バケットブリゲード内のデータをソケットへ非ブロッキングで書き出します。書き込み完了後のバケット削除および部分書き込み時のバケット分割を管理します。

シグネチャまたは形状

static apr_status_t writev_nonblocking(apr_socket_t *s,
                                       apr_bucket_brigade *bb,
                                       core_output_filter_ctx_t *ctx,
                                       apr_size_t bytes_to_write,
                                       apr_size_t nvec,
                                       conn_rec *c)

入力と出力

  • **入力**:
  • s: 書き込み対象のソケット。
  • bb: 書き込むデータを含むバケットブリゲード。
  • ctx: 出力フィルタのコンテキスト（iovec 配列を保持）。
  • bytes_to_write: 書き込み対象の総バイト数。
  • nvec: iovec 配列の要素数。
  • c: 接続情報構造体。
  • **出力**:
  • 戻り値: apr_status_t 型のステータスコード（成功時は APR_SUCCESS）。

前提条件と不変条件

  • ctx->vec には、bb 内のデータに対応する有効な iovec 配列が格納されている必要があります。
  • 書き込み処理中、bb 内のバケットは apr_bucket_delete または apr_bucket_split によって適切に管理されます。

アルゴリズムまたは振る舞い

1. apr_socket_sendv を呼び出し、ソケットへデータを送信します。 2. 送信されたバイト数に基づき、bb 内のバケットを走査します。

  • 長さ 0 のバケットは削除します。
  • 送信済みバイト数がバケット長を超える場合、バケットを削除して次のバケットへ進みます。
  • 部分的に送信された場合、apr_bucket_split を用いてバケットを分割し、残りのデータを保持します。

    • 3. bytes_written < bytes_to_write である限り、上記処理を繰り返します。 4. ap__logio_add_bytes_out が定義されている場合、送信バイト数を記録します。 5. 最終的な送信バイト数を ctx->bytes_written に加算し、トレースログを出力して終了します。

呼び出し元と呼び出し先

  • **呼び出し元**: server/core_filters.c 内の各箇所（行 671, 693, 728, 767, 776 など）。
  • **呼び出し先**:
  • apr_socket_sendv (APR)
  • apr_bucket_delete (APR)
  • apr_bucket_split (APR)
  • ap__logio_add_bytes_out (外部モジュール連携用関数ポインタ)
  • ap_log_cerror (Apache HTTPD ログ関数)

関連データ構造

  • apr_socket_t: ソケット抽象化構造体。
  • apr_bucket_brigade: バケットのリスト構造。
  • core_output_filter_ctx_t: 出力フィルタの永続状態を保持する構造体。
  • conn_rec: 接続ごとの状態を保持する構造体。

失敗モードと境界条件

  • apr_socket_sendv が APR_SUCCESS 以外のステータスを返した場合、ループを中断しそのステータスを返します。
  • bytes_written が bytes_to_write に達しない場合、非ブロッキング動作として処理を終了します。

保守担当者向けメモ

  • 本関数は core_output_filter の一部として、非ブロッキング I/O の再試行ロジックと密接に関連しています。
  • ctx->vec の操作は core_output_filter_ctx_t の状態に依存するため、呼び出し側でのコンテキスト整合性に注意してください。
