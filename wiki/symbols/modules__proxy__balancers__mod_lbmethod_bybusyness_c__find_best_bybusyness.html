find_best_bybusyness

定義

  • 種別: function
  • 定義箇所: modules/proxy/balancers/mod_lbmethod_bybusyness.c:45

目的

mod_proxy のロードバランサーにおいて、現在処理中のリクエスト数（busy数）が最も少ないワーカーを特定し、選定します。

シグネチャまたは形状

static proxy_worker *find_best_bybusyness(proxy_balancer *balancer, request_rec *r)

入力と出力

  • **入力**:
  • balancer: 負荷分散対象のバランサー構造体。
  • r: 現在のリクエスト情報。
  • **出力**:
  • 選定された proxy_worker 構造体へのポインタ。選定可能なワーカーがない場合は NULL。

前提条件と不変条件

  • mod_proxy がロードされており、バランサーおよびワーカーの共有メモリが初期化されていること。
  • ap_proxy_balancer_get_best_worker_fn が利用可能であること。

アルゴリズムまたは振る舞い

1. ap_proxy_balancer_get_best_worker_fn を呼び出し、全ワーカーを走査します。この際、比較関数 is_best_bybusyness を用いて、最も負荷の低いワーカーを特定します。 2. 走査の結果、最適なワーカーが見つかった場合、そのワーカーの lbstatus（重み付け状態）から、走査中に算出された total_factor を減算して更新します。 3. 選定されたワーカーを呼び出し元へ返します。

呼び出し元と呼び出し先

  • **呼び出し元**: modules/proxy/balancers/mod_lbmethod_bybusyness.c:81 にて、bybusyness 構造体のメンバとして参照されています。
  • **呼び出し先**:
  • ap_proxy_balancer_get_best_worker_fn（外部提供関数）
  • is_best_bybusyness（比較関数）

関連データ構造

  • proxy_balancer: ロードバランサーの構成情報を保持。
  • proxy_worker: 各バックエンドサーバーの状態（busy 数や lbstatus を含む）を保持。
  • request_rec: Apache のリクエスト情報。

失敗モードと境界条件

  • 候補となるワーカーが存在しない場合、ap_proxy_balancer_get_best_worker_fn は NULL を返し、本関数もそのまま NULL を返します。
  • total_factor はワーカー選定プロセス内で算出され、ワーカーが見つかった場合にのみ lbstatus の更新に使用されます。

保守担当者向けメモ

  • 本関数は PROXY_LBMETHOD プロバイダーの一部として動作します。
  • busy 数は scoreboard を介して管理されるため、マルチプロセス環境下での整合性が保証されています。
  • lbstatus の更新ロジックは、他の lbmethod モジュール（byrequests 等）と共通の設計思想に基づいています。
