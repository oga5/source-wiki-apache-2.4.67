find_best_byrequests

定義

  • 種別: function
  • 定義箇所: modules/proxy/balancers/mod_lbmethod_byrequests.c:83

目的

リクエスト数ベースのロードバランシングにおいて、各ワーカーの重み（lbfactor）に基づき、次に処理を割り当てるべき最適なワーカーを選出します。

シグネチャまたは形状

static proxy_worker *find_best_byrequests(proxy_balancer *balancer, request_rec *r)

入力と出力

  • **入力**:
  • balancer: 対象となるロードバランサー構造体。
  • r: 現在のリクエスト情報。
  • **出力**:
  • 選出された proxy_worker 構造体へのポインタ。選出可能なワーカーがない場合は NULL を返します。

前提条件と不変条件

  • ap_proxy_balancer_get_best_worker_fn が初期化済みであること。
  • is_best_byrequests コールバック関数が、ワーカーの lbstatus を適切に比較・更新できる状態にあること。

アルゴリズムまたは振る舞い

1. ap_proxy_balancer_get_best_worker_fn を呼び出し、is_best_byrequests を用いて全ワーカーを走査します。この過程で、各ワーカーの lbstatus に lbfactor が加算され、最も値が大きいワーカーが選出されます。 2. 選出されたワーカーが存在する場合、そのワーカーの lbstatus から、走査時に計算された全ワーカーの lbfactor 合計値（total_factor）を減算します。 3. 選出されたワーカーを呼び出し元へ返します。

呼び出し元と呼び出し先

  • **呼び出し元**: mod_proxy のロードバランサー機能（modules/proxy/balancers/mod_lbmethod_byrequests.c:122 にて構造体登録）。
  • **呼び出し先**:
  • ap_proxy_balancer_get_best_worker_fn（mod_proxy が提供するワーカー走査関数）。
  • is_best_byrequests（ワーカー比較・状態更新用コールバック）。

関連データ構造

  • proxy_balancer: バランサーの状態を保持。
  • proxy_worker: 各バックエンドサーバーの状態（lbstatus, lbfactor 等）を保持。
  • request_rec: Apache のリクエスト情報。

失敗モードと境界条件

  • 選出可能なワーカーが存在しない場合、ap_proxy_balancer_get_best_worker_fn は NULL を返し、本関数もそのまま NULL を返します。
  • lbstatus の更新は共有メモリ上で行われるため、競合が発生しないよう mod_proxy 側の排他制御に依存しています。

保守担当者向けメモ

  • 本関数は「重み付きラウンドロビン」を実装しています。lbstatus の減算処理は、累積された負荷バランスを維持するための重要なステップです。
  • 統計情報の追跡やデバッグを行う際は、共有メモリ上の lbstatus 値の変化に注意してください。
