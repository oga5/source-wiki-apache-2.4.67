hb_server_t

定義

  • 種別: struct
  • 定義箇所: modules/proxy/balancers/mod_lbmethod_heartbeat.c:49

目的

外部のハートビート情報に基づいた負荷分散において、各バックエンドサーバーの現在の負荷状態と識別情報を保持するために使用されます。

シグネチャまたは形状

typedef struct hb_server_t {
    const char *ip;         /* サーバーのIPアドレス */
    int busy;               /* 処理中のリクエスト数 */
    int ready;              /* 待機中のスレッド数 */
    int port;               /* サーバーのポート番号 */
    int id;                 /* サーバー識別子 */
    apr_time_t seen;        /* 最後にハートビートを受信した時刻 */
    proxy_worker *worker;   /* 関連付けられたプロキシワーカー */
} hb_server_t;

入力と出力

本構造体はデータ保持用であり、関数ではありません。readfile_heartbeats 等の解析処理によって各フィールドが設定され、find_best_hb 等の選定ロジックにおいて、負荷分散の判断基準として読み取られます。

前提条件と不変条件

  • seen フィールドは、最後にハートビートを受信した時刻を示します。LBM_HEARTBEAT_MAX_LASTSEEN（10秒）を超えて更新されていないサーバーは、生存していないとみなされます。
  • worker フィールドは、mod_proxy の proxy_worker 構造体と紐付けられます。

アルゴリズムまたは振る舞い

本構造体は、ハートビート情報の解析結果を格納するコンテナとして機能します。find_best_hb は、この構造体の ready 値に基づいて確率的に転送先を選択します。また、起動直後で busy=0 かつ ready が大きい場合、過負荷防止のために ready 値を 1/4 に調整するヒューリスティック処理の対象となります。

呼び出し元と呼び出し先

  • **呼び出し元**: readfile_heartbeats および readslot_heartbeats が本構造体のインスタンスを生成・更新します。find_best_hb が選定ロジック内で参照します。
  • **呼び出し先**: 構造体自体はデータ保持用であり、直接的な関数呼び出しは行いません。

関連データ構造

  • proxy_worker: mod_proxy が管理するバックエンドサーバーの定義。
  • lb_hb_ctx_t: モジュール全体のコンテキスト。
  • apr_hash_t: サーバー情報を管理するハッシュテーブルの要素として利用されます。

失敗モードと境界条件

  • ハートビート情報が欠落している場合、seen の値が更新されず、生存判定から除外されます。
  • apr_pcalloc によるメモリ確保が失敗した場合、プログラムは異常終了する可能性があります。

保守担当者向けメモ

  • 本構造体は mod_lbmethod_heartbeat.c 内でのみ使用される内部表現です。
  • サーバーの負荷状態を動的に反映するため、find_best_hb 内でリクエスト毎に生成される一時的なプール（tpool）上で管理されることが一般的です。
