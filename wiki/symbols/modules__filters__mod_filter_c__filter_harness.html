filter_harness

定義

  • 種別: function
  • 定義箇所: modules/filters/mod_filter.c:274

目的

Apache HTTP Server のフィルタチェーンにおいて、実行時に適切なフィルタプロバイダを動的に選択・実行するディスパッチャ（ハーネス）として機能します。

シグネチャまたは形状

static apr_status_t filter_harness(ap_filter_t *f, apr_bucket_brigade *bb)

入力と出力

  • **入力**:
  • f: 現在のフィルタコンテキスト (ap_filter_t)。
  • bb: 処理対象のバケットブリゲード (apr_bucket_brigade)。
  • **出力**:
  • apr_status_t: フィルタ処理の結果（成功時は APR_SUCCESS）。

前提条件と不変条件

  • f->ctx は harness_ctx 型である必要があります。
  • フィルタが一度選択された後は、ctx->func に関数ポインタが保持されます。
  • f->r->status が 200 以外の場合、filter-errordocs 環境変数が設定されていない限り、本フィルタはスタックから削除されます。

アルゴリズムまたは振る舞い

1. **エラー処理**: ステータスコードが 200 以外の場合、フィルタを削除して後続へブリゲードを渡します。 2. **トレース**: filter_trace を呼び出し、デバッグ情報を記録します。 3. **プロバイダ選択**: ctx->func が未設定の場合、プロトコル制約（プロキシ要求や no-transform ヘッダの確認）をチェックし、filter_lookup を実行して適切なプロバイダを特定します。 4. **実行**:

  • 現在のフィルタコンテキストをプロバイダ用の ctx->fctx に切り替えます。
  • ctx->func を実行します。
  • 実行後、コンテキストをハーネスの状態に復元します。

呼び出し元と呼び出し先

  • **呼び出し元**: modules/filters/mod_filter.c:413 にて filter->filter_func.out_func に割り当てられています。
  • **呼び出し先**:
  • filter_trace: デバッグログ出力。
  • filter_lookup: プロバイダの検索と選択。
  • ap_remove_output_filter: フィルタの削除。
  • ap_pass_brigade: 後続フィルタへのデータ転送。
  • ctx->func: 選択された実際のフィルタプロバイダ関数。

関連データ構造

  • harness_ctx: 実行時のコンテキストを保持し、選択されたプロバイダ関数 (func) とそのコンテキスト (fctx) を管理します。
  • ap_filter_rec_t: フィルタの定義情報。

失敗モードと境界条件

  • **プロバイダ未検出**: filter_lookup が失敗した場合、フィルタを削除し、処理を後続へパスします。
  • **プロトコル制約**: プロキシ環境下で AP_FILTER_PROTO_NO_PROXY が設定されている場合、または Cache-Control: no-transform が存在する場合、フィルタは適用されません。

保守担当者向けメモ

  • 本関数は mod_filter の中核であり、フィルタの動的切り替えとコンテキストのスイッチングを担います。
  • ctx->func の呼び出し前後で f->ctx を入れ替えることで、ハーネス自身のコンテキストと個別のプロバイダコンテキストを分離しています。
  • NO_PROTOCOL マクロが定義されていない場合、プロトコル整合性のためのヘッダチェックが有効になります。
