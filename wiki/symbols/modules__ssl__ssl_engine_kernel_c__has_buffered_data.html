has_buffered_data

定義

  • 種別: function
  • 定義箇所: modules/ssl/ssl_engine_kernel.c:44

目的

現在の接続の入力フィルタに、未処理のバッファ済みデータが存在するかを判定する。

シグネチャまたは形状

static int has_buffered_data(request_rec *r)

入力と出力

  • **入力**: request_rec *r （現在のリクエスト構造体）
  • **出力**: int （データが存在する場合は非ゼロ、存在しない場合は 0）

前提条件と不変条件

  • r->connection->bucket_alloc が有効であること。
  • r->connection->input_filters がフィルタチェーンとして初期化されていること。

アルゴリズムまたは振る舞い

1. apr_brigade_create を使用して、リクエストのプールに関連付けられた一時的なバケットブリゲードを作成する。 2. ap_get_brigade を AP_MODE_SPECULATIVE モードかつ APR_NONBLOCK_READ で呼び出し、入力フィルタからデータを読み取らずに確認する。 3. apr_brigade_length を使用して、ブリゲード内のデータ長が 0 より大きいかを確認する。 4. apr_brigade_destroy を呼び出してブリゲードを破棄し、判定結果を返す。

呼び出し元と呼び出し先

  • **呼び出し元**: modules/ssl/ssl_engine_kernel.c:808
  • **呼び出し先**:
  • apr_brigade_create
  • ap_get_brigade
  • apr_brigade_length
  • apr_brigade_destroy

関連データ構造

  • request_rec
  • apr_bucket_brigade

失敗モードと境界条件

  • ap_get_brigade が APR_SUCCESS を返さない場合、またはデータ長が 0 の場合は 0 を返す。
  • メモリ不足によりブリゲードの作成に失敗した場合の挙動は、Apache APR のメモリ割り当てポリシーに依存する。

保守担当者向けメモ

  • 本関数は推測的読み取り（Speculative Read）を行うため、フィルタチェーンの状態を破壊せずにデータの有無を確認できる。
  • TLS 再ネゴシエーションの判断など、入力ストリームの先行読み取りが必要な箇所で使用される。
