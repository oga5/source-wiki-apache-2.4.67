DIRENTRY

定義

  • 種別: struct
  • 定義箇所: support/htcacheclean.c:74

目的

キャッシュディレクトリの走査中に、ファイルまたはファイルセット（ヘッダーとデータのペア）のメタデータを一時的に保持するために使用されます。

シグネチャまたは形状

typedef struct _direntry {
    APR_RING_ENTRY(_direntry) link;
    int type;         /* TEMP, HEADER, DATA, HEADERDATA */
    apr_time_t htime; /* ヘッダーファイルの最終更新日時 */
    apr_time_t dtime; /* ボディまたは一時ファイルの最終更新日時 */
    apr_off_t hsize;  /* ヘッダーファイルのサイズ */
    apr_off_t dsize;  /* ボディまたは一時ファイルのサイズ */
    char *basename;   /* ファイル/ファイルセットのベース名 */
} DIRENTRY;

入力と出力

  • 本構造体は、process_dir() 関数内でディレクトリ走査時に生成されます。
  • apr_pcalloc を介してメモリが割り当てられ、キャッシュエントリの属性情報が格納されます。

前提条件と不変条件

  • link メンバにより、APR_RING 構造の一部として管理されます。
  • type メンバは、キャッシュファイルの種類（ヘッダーのみ、データのみ、またはその両方）を識別するために使用されます。

アルゴリズムまたは振る舞い

  • キャッシュディレクトリ内のファイルを走査する際、各エントリを本構造体にマッピングし、リング構造に連結することで、後続の削除判定処理（purge()）のためのリストを構築します。

呼び出し元と呼び出し先

  • 呼び出し元: support/htcacheclean.c 内の process_dir() 関数が、ディレクトリ走査時に DIRENTRY をインスタンス化します。
  • 呼び出し先: 構築された DIRENTRY のリストは、purge() 関数によって参照され、削除対象の選定に使用されます。

関連データ構造

  • APR_RING: DIRENTRY をリストとして管理するために使用されます。
  • ENTRY: キャッシュエントリの管理に使用される別の構造体であり、DIRENTRY と密接に関連してキャッシュの整合性を維持します。

失敗モードと境界条件

  • メモリ不足時: apr_pcalloc が失敗した場合の挙動は APR のメモリ管理ポリシーに依存します。
  • ファイル名解析: basename の取得に失敗した場合や、ファイルシステム上のメタデータが取得できない場合の境界条件については、APR のステータスコードチェックに依存します。

保守担当者向けメモ

  • 本構造体は mod_cache_disk が生成するファイルフォーマットと密接に結びついています。キャッシュフォーマットに変更があった場合は、type 定数やサイズ計算ロジックの整合性を確認してください。
  • apr_pool_t を使用したメモリ管理が行われているため、リスト構築後のメモリ解放はプール全体の破棄に依存します。
