send_brigade_nonblocking

定義

  • 種別: function
  • 定義箇所: server/core_filters.c:649

目的

apr_bucket_brigade 内のデータをソケットへ非ブロッキングで書き出します。sendfile が利用可能な場合はそれを優先し、それ以外は writev を用いて効率的にデータを転送します。

シグネチャまたは形状

static apr_status_t send_brigade_nonblocking(apr_socket_t *s,
                                             apr_bucket_brigade *bb,
                                             core_output_filter_ctx_t *ctx,
                                             conn_rec *c)

入力と出力

  • **入力**:
  • s: 送信先ソケット。
  • bb: 送信対象のバケットブリゲード。
  • ctx: 出力フィルタの永続コンテキスト（iovec 配列やバッファを保持）。
  • c: 接続レコード。
  • **出力**:
  • apr_status_t: 処理結果（成功時は APR_SUCCESS）。

前提条件と不変条件

  • ctx->vec および ctx->nvec は、writev 実行時に有効な状態である必要があります。
  • APR_HAS_SENDFILE が有効な環境では、can_sendfile_bucket の判定に基づき sendfile が選択されます。
  • メタデータバケット（長さ 0 のバケット）は、先行するデータバケットの送信が完了するまで削除されません。

アルゴリズムまたは振る舞い

1. ブリゲード内の各バケットを順次走査します。 2. sendfile 対応バケットであれば、蓄積中の iovec を先に書き出した後、sendfile_nonblocking を呼び出します。 3. データバケットの場合、apr_bucket_read でデータを取得します。読み込みがブロックされる場合は、蓄積済みの iovec を書き出した後、ブロック読み込みに切り替えて再試行します。 4. 取得したデータを ctx->vec に蓄積し、conf->flush_max_threshold を超えるか、iovec が満杯になった時点で writev_nonblocking を呼び出します。 5. 最後に、残存する iovec があれば書き出しを行い、ソケットの nopush 状態を解除して終了します。

呼び出し元と呼び出し先

  • **呼び出し元**: server/core_filters.c 内の ap_core_output_filter（行 506）から呼び出されます。
  • **呼び出し先**:
  • can_sendfile_bucket
  • sock_nopush
  • writev_nonblocking
  • sendfile_nonblocking
  • apr_bucket_read
  • apr_bucket_delete

関連データ構造

  • core_output_filter_ctx_t: フィルタの永続状態と書き込み用バッファを保持。
  • apr_bucket_brigade: 送信データのキュー。
  • apr_socket_t: 通信に使用するソケット。

失敗モードと境界条件

  • apr_bucket_read や writev_nonblocking がエラーを返した場合、即座に処理を中断し、そのエラーコードを返します。
  • iovec の容量不足時は、動的に ctx->vec を再割り当て（最大 NVEC_MAX まで）します。
  • EAGAIN 発生時は、蓄積済みデータをフラッシュしてからブロック読み込みへ移行するリカバリ処理を行います。

保守担当者向けメモ

  • nvec が 0 の場合のみ空バケットを削除するロジックは、EOR（End of Request）等のメタデータが順序通りに処理されることを保証するためのものです。
  • sock_nopush の制御は、ネットワーク効率を最適化するために重要です。修正時は cleanup ラベルでの解除漏れに注意してください。
