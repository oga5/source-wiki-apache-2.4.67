cache_control_t

定義

  • 種別: struct
  • 定義箇所: modules/cache/cache_common.h:30

目的

HTTP の Cache-Control および Pragma ヘッダーに含まれるキャッシュ制御ディレクティブを解析し、後続のキャッシュ処理（保存・提供・再検証の判断）で利用可能な形式で保持する。

シグネチャまたは形状

typedef struct cache_control {
    unsigned int parsed:1;
    unsigned int cache_control:1;
    unsigned int pragma:1;
    unsigned int no_cache:1;
    unsigned int no_cache_header:1;
    unsigned int no_store:1;
    unsigned int max_age:1;
    unsigned int max_stale:1;
    unsigned int min_fresh:1;
    unsigned int no_transform:1;
    unsigned int only_if_cached:1;
    unsigned int public:1;
    unsigned int private:1;
    unsigned int private_header:1;
    unsigned int must_revalidate:1;
    unsigned int proxy_revalidate:1;
    unsigned int s_maxage:1;
    unsigned int invalidated:1;
    apr_int64_t max_age_value;
    apr_int64_t max_stale_value;
    apr_int64_t min_fresh_value;
    apr_int64_t s_maxage_value;
} cache_control_t;

入力と出力

  • **入力**: HTTP リクエストまたはレスポンスのヘッダー値。これらは ap_cache_control() 関数を通じて解析され、本構造体の各フィールドにマッピングされる。
  • **出力**: キャッシュの可否や有効期限を示すフラグおよび数値。これらは mod_cache や各ストレージプロバイダー（mod_cache_disk 等）のロジックへ渡される。

前提条件と不変条件

  • max_age_value 等の数値フィールドは、対応するフラグ（max_age 等）がセットされており、かつ値が正（positive）である場合に有効とみなされる。
  • parsed フラグは、ヘッダーの解析が完了したことを示すために使用される。

アルゴリズムまたは振る舞い

1. **解析**: ap_cache_control() 関数がヘッダー文字列を走査し、RFC 7234 等の仕様に基づき各ビットフィールドをセットする。 2. **保持**: 解析結果を構造体として保持し、キャッシュのライフサイクル管理（保存・提供判断）において、条件分岐の判定基準として利用される。 3. **無効化**: invalidated フィールドは、対象エンティティが無効化されたかどうかを追跡するために使用される。

呼び出し元と呼び出し先

  • **呼び出し元**: mod_cache.c や cache_util.c 等のキャッシュポリシーエンジンが、リクエスト処理中に本構造体を初期化・利用する。
  • **呼び出し先**: 本構造体自体はデータ保持用であり、直接的な関数呼び出しは行わない。
  • **注記**: ap_cache_control() 関数（modules/cache/cache_util.c）が本構造体を操作する主要なインターフェースである。

関連データ構造

  • request_rec: HTTP リクエスト情報を保持する構造体。
  • cache_handle_t: キャッシュエントリへのアクセスハンドル。
  • disk_cache_info_t / cache_socache_info_t: 各ストレージ固有のメタデータ構造。

失敗モードと境界条件

  • ヘッダーが不正な形式である場合、解析ロジックが適切にフラグをセットできない可能性がある。
  • 数値フィールド（max_age_value 等）が負の値である場合、その設定は無視される。

保守担当者向けメモ

  • 本構造体は modules/cache モジュール全体で共有されるため、フィールドの追加や変更は mod_cache および各ストレージプロバイダーの互換性に影響を与える。
  • cache_util.c 内の解析ロジックが、RFC 仕様の変更に追従しているかを確認すること。
  • invalidated フィールドは 2.4.5-dev で追加された比較的新しい要素である。
