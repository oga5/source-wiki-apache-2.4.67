h2test_delay_handler

定義

  • 種別: function
  • 定義箇所: test/modules/http2/mod_h2test/mod_h2test.c:265

目的

HTTP/2 テスト環境において、指定された遅延時間を挟みながらレスポンスを分割送信し、ネットワーク遅延時の挙動を検証する。

シグネチャまたは形状

static int h2test_delay_handler(request_rec *r)

入力と出力

  • **入力**: request_rec *r（Apache リクエスト構造体）。クエリパラメータ r->args に遅延時間（例: "10s"）を指定可能。
  • **出力**: int（Apache ステータスコード OK または AP_FILTER_ERROR）。

前提条件と不変条件

  • r->handler が "h2test-delay" であること。
  • リクエストメソッドが GET または POST であること。
  • duration_parse が成功し、有効な遅延時間が取得できること。

アルゴリズムまたは振る舞い

1. **初期化**: r->handler を確認し、対象外であれば DECLINED を返す。 2. **パラメータ解析**: r->args を duration_parse で解析し、遅延時間を取得する。失敗時は HTTP_BAD_REQUEST を発行。 3. **レスポンス設定**: Content-Length を無効化し、チャンク転送を有効化。Brotli/Gzip 圧縮を無効化する。 4. **ボディ処理**: リクエストボディが存在する場合は読み捨てて消費する。 5. **分割送信**: 8KB のバッファを3回に分けて送信する。各送信の間に apr_sleep(delay) を実行する。 6. **終了**: EOS（End of Stream）バケットを挿入し、フィルタチェーンへ渡して終了する。

呼び出し元と呼び出し先

  • **呼び出し元**: Apache フックシステム（test/modules/http2/mod_h2test/mod_h2test.c:584 にて ap_hook_handler に登録）。
  • **呼び出し先**: duration_parse, ap_die, ap_log_rerror, ap_setup_client_block, ap_get_client_block, apr_brigade_write, ap_pass_brigade, apr_sleep, apr_bucket_eos_create。

関連データ構造

  • request_rec: リクエスト情報の保持とレスポンス制御に使用。
  • apr_bucket_brigade: レスポンスデータの構築と送信に使用。

失敗モードと境界条件

  • **解析失敗**: duration_parse が失敗した場合、ap_die で 400 Bad Request を返し終了する。
  • **送信エラー**: ap_pass_brigade 等でエラーが発生した場合、cleanup ラベルへジャンプし、AP_FILTER_ERROR を返す。
  • **接続中断**: c->aborted が真の場合、処理を中断し OK を返す。

保守担当者向けメモ

  • 本関数はテスト専用であり、意図的に遅延を発生させるため、本番環境での利用は厳禁。
  • メモリ管理は r->pool に依存しているため、明示的な解放は不要。
  • chunks 変数は現在 3 に固定されている。
