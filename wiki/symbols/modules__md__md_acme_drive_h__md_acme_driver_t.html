md_acme_driver_t

定義

  • 種別: struct
  • 定義箇所: modules/md/md_acme_drive.h:24

目的

ACME プロトコルを用いたマネージドドメイン（MD）の証明書取得・更新プロセスにおける実行コンテキストを保持します。上位のプロトコルドライバと具体的な ACME 処理を橋渡しする役割を担います。

シグネチャまたは形状

typedef struct md_acme_driver_t {
    md_proto_driver_t *driver;
    void *sub_driver;
    md_acme_t *acme;
    md_t *md;
    struct apr_array_header_t *domains;
    apr_array_header_t *ca_challenges;
    const char *profile;
    int profile_mandatory;
    int complete;
    apr_array_header_t *creds;
    struct md_credentials_t *cred;
    const char *chain_up_link;
    struct md_acme_order_t *order;
    apr_interval_time_t authz_monitor_timeout;
    const char *csr_der_64;
    apr_interval_time_t cert_poll_timeout;
} md_acme_driver_t;

入力と出力

  • **入力**: md_proto_driver_t の baton フィールドを介して保持されます。
  • **出力**: 証明書取得のライフサイクル（アカウント設定、注文、認証、証明書取得）に応じて、構造体内の各フィールドが順次更新されます。

前提条件と不変条件

  • 本構造体は md_proto_driver_t の baton フィールドに格納されて使用されます。
  • 証明書取得の各フェーズ（アカウント設定→注文→認証→取得）において、対応するフィールドが適切に初期化・更新されている必要があります。

アルゴリズムまたは振る舞い

証明書取得のライフサイクルを管理します。 1. **アカウント設定**: md_acme_drive_set_acct によりアカウント情報を特定・検証します。 2. **注文処理**: md_acme_order_t を通じて ACME サーバーへの注文を管理します。 3. **ポーリング**: md_acme_drive_cert_poll を定期的に呼び出し、証明書が発行されるまでサーバー上の注文状態を監視します。

呼び出し元と呼び出し先

  • **呼び出し元**: modules/md/md_acme_drive.c 内の各関数（d->baton として参照）。
  • **呼び出し先**: 本構造体自体はデータ構造であり、関数呼び出しの対象ではありません。

関連データ構造

  • md_proto_driver_t: 上位のプロトコルドライバ。
  • md_acme_t: ACME サーバーとの通信セッション状態。
  • md_acme_order_t: ACME 注文処理の管理。
  • md_credentials_t: 処理中の認証情報。
  • apr_array_header_t: ドメインリストや認証情報、チャレンジ情報の保持に使用。

失敗モードと境界条件

  • アカウントのロード失敗時には ENOENT を返す設計となっており、これが新規アカウント作成フローへの移行トリガーとなります。

保守担当者向けメモ

  • 本構造体の各フィールドは、証明書取得のライフサイクルに応じて動的に更新されます。
  • sub_driver フィールドは、ACMEv2 などの具体的なプロトコル実装へのポインタとして利用されます。
