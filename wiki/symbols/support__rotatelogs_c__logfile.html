logfile

定義

  • 種別: struct
  • 定義箇所: support/rotatelogs.c:88

目的

ログファイルのローテーション処理において、現在開いているログファイルの状態（ファイルハンドル、メモリプール、ファイルパス）を保持するために使用されます。

シグネチャまたは形状

struct logfile {
    apr_pool_t *pool;
    apr_file_t *fd;
    char name[APR_PATH_MAX];
};

入力と出力

  • **入力**: ログファイルのパス（name）、ファイルハンドル（fd）、および関連するメモリプール（pool）。
  • **出力**: 構造体メンバとして保持される上記リソース。

前提条件と不変条件

  • pool は、このログファイルに関連するメモリ割り当てを管理するために使用されます。
  • fd は、apr_file_t 型の有効なファイルハンドルを指す必要があります。
  • name は、APR_PATH_MAX バイト以内のパス文字列である必要があります。

アルゴリズムまたは振る舞い

  • rotatelogs ユーティリティの実行中、rotate_status 構造体の一部として現在のログファイルの状態を保持します。
  • ローテーション発生時、doRotate() 関数内でこの構造体の内容が更新され、古いファイルハンドルが閉じられ、新しいファイルがオープンされます。

呼び出し元と呼び出し先

  • **呼び出し元**: support/rotatelogs.c 内の rotate_status 構造体（current メンバとして定義）がこの構造体を使用しています。
  • **呼び出し先**: 構造体自体はデータ保持用であり、直接関数を呼び出すことはありません。APR のファイル I/O 関数（apr_file_open 等）がこの構造体のメンバを引数として受け取ります。

関連データ構造

  • rotate_status_t: logfile 構造体を保持し、ローテーションの実行状態を管理します。
  • rotate_config_t: ローテーションの閾値や設定を保持します。

失敗モードと境界条件

  • name が APR_PATH_MAX を超える場合、パスの切り詰めやバッファオーバーフローのリスクがあります。
  • ファイルオープン失敗時、fd は無効な状態となります。

保守担当者向けメモ

  • 本構造体は support/rotatelogs.c 専用です。
  • ログローテーション時に pool を適切に破棄・再作成することで、メモリリークを防ぐ設計となっています。
  • ツールによる解析結果では、modules/ 配下のファイルで logfile という用語が言及されていますが、これらは本構造体とは異なる文脈（ログフォーマットや設定）であり、直接的なコード上の依存関係ではありません。
