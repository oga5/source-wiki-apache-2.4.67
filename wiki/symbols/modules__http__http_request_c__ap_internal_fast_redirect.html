ap_internal_fast_redirect

定義

  • 種別: function
  • 定義箇所: modules/http/http_request.c:692

目的

既存のリクエスト構造 (r) に対して、サブクエスト (rr) の情報を適用し、リクエストコンテキストを効率的に再利用（高速リダイレクト）します。

シグネチャまたは形状

void ap_internal_fast_redirect(request_rec *rr, request_rec *r)

入力と出力

  • **入力**:
  • rr: リダイレクト元のサブクエスト構造体。
  • r: リダイレクト先となるメインリクエスト構造体。
  • **出力**: なし（r の内部状態が rr の内容で更新される）。

前提条件と不変条件

  • rr->pool と r->pool の生存期間を同期させるため、内部で apr_pool_join が実行されます。
  • フィルタチェーン内の request_rec ポインタが古い rr を指したままにならないよう、呼び出し後にフィルタの更新が必要です。

アルゴリズムまたは振る舞い

1. **メモリ管理**: apr_pool_join により、rr のプールを r のプールに結合します。 2. **属性コピー**: URI、ファイルパス、ハンドラ、コンテンツタイプ、エンコーディング等のメタデータを rr から r へコピーします。 3. **ヘッダー統合**: notes、headers_out、err_headers_out、trailers_out、subprocess_env を apr_table_overlay を用いて統合します。 4. **フィルタ更新**: update_r_in_filters を呼び出し、入出力フィルタが保持する request_rec 参照を rr から r へ差し替えます。 5. **フィルタチェーン調整**:

  • r がメインリクエストの場合、必要に応じて SUBREQ_CORE フィルタを追加します。
  • r がサブクエストでない場合、フィルタチェーンから SUBREQ_CORE フィルタを削除します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • modules/mappers/mod_dir.c
  • modules/mappers/mod_negotiation.c
  • **呼び出し先**:
  • apr_pool_join
  • ap_set_content_type_ex
  • apr_table_overlay
  • update_r_in_filters
  • ap_add_output_filter_handle / ap_remove_output_filter

関連データ構造

  • request_rec: リクエストのコンテキストを保持する主要構造体。
  • ap_filter_t: フィルタチェーンを構成する構造体。

失敗モードと境界条件

  • フィルタチェーンの更新 (update_r_in_filters) が不完全な場合、古い request_rec を参照し続けることによる不正なメモリ参照や不整合が発生する可能性があります。
  • rr の構造が不完全な場合、コピーされたデータが r の動作に悪影響を及ぼす可能性があります。

保守担当者向けメモ

  • 本関数は非常に脆弱であるとされており、使用には細心の注意が必要です。
  • update_r_in_filters によるフィルタのポインタ更新は、リダイレクト後の整合性を保つための必須処理です。
  • SUBREQ_CORE フィルタの着脱ロジックは、リクエストの階層構造に依存しているため、フィルタチェーンの構成変更時には注意してください。
