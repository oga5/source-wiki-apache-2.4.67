ap_check_request_header

定義

  • **種別**: 関数
  • **定義箇所**: server/protocol.c:1000

目的

HTTP リクエストヘッダーの検証および仮想ホスト情報の更新を行い、リクエストがプロトコル仕様（RFC）に準拠しているかを確認します。

シグネチャまたは形状

int ap_check_request_header(request_rec *r)

入力と出力

  • **入力**: request_rec *r （現在のリクエスト構造体）
  • **出力**: int 型。検証が成功した場合は 1 を返し、失敗（エラー）した場合は 0 を返します。失敗時は r->status に適切な HTTP ステータスコード（HTTP_BAD_REQUEST や HTTP_EXPECTATION_FAILED 等）が設定されます。

前提条件と不変条件

  • r は有効な request_rec 構造体である必要があります。
  • 本関数は、リクエストラインおよび MIME ヘッダーが既に読み込まれた後に呼び出されることを前提としています。

アルゴリズムまたは振る舞い

1. **仮想ホストの更新**: ap_update_vhost_from_headers_ex を呼び出し、ヘッダー情報に基づいて仮想ホストを特定・更新します。strict_host_check が有効な場合、不一致があれば警告をログ出力し、エラーを返します。 2. **ホスト名の検証**: HTTP/1.1 以降のリクエストにおいて、ホスト名が不明または Host ヘッダーが存在しない場合、RFC 2616 に基づき HTTP_BAD_REQUEST を設定して終了します。 3. **Expect ヘッダーの処理**: Expect ヘッダーが存在する場合、値が 100-continue であれば r->expecting_100 をセットします。それ以外の値であれば HTTP_EXPECTATION_FAILED を設定して終了します。 4. すべてのチェックを通過した場合に 1 を返します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • modules/http2/h2_request.c
  • server/protocol.c (行 1543)
  • **呼び出し先**:
  • ap_get_core_module_config
  • ap_update_vhost_from_headers_ex
  • ap_log_rerror
  • apr_table_get
  • ap_cstr_casecmp

関連データ構造

  • request_rec: リクエストの全状態を保持する主要構造体。
  • core_server_config: コアモジュールの設定情報（strict_host_check 等）。

失敗モードと境界条件

  • **仮想ホスト不一致**: strict_host_check がオンの状態でホスト名が一致しない場合、警告を記録し失敗します。
  • **HTTP/1.1 ホスト欠落**: Host ヘッダーがない場合、HTTP_BAD_REQUEST となります。
  • **未知の Expect 値**: 100-continue 以外の Expect 値が指定された場合、HTTP_EXPECTATION_FAILED となります。

保守担当者向けメモ

  • 本関数はリクエスト処理の初期段階で重要なバリデーションを担います。RFC 準拠の厳格化や新しいプロトコル拡張を追加する際は、本関数のチェックロジックが影響範囲となります。
  • r->status が HTTP_OK でない場合は即座に処理を中断する設計になっています。
