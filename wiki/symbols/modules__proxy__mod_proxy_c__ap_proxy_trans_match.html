ap_proxy_trans_match

定義

  • 種別: function
  • 定義箇所: modules/proxy/mod_proxy.c:825

目的

リクエストの URI がプロキシ設定（ProxyPass 等）と一致するかを判定し、一致した場合はリクエストをプロキシ処理用に書き換える。

シグネチャまたは形状

int ap_proxy_trans_match(request_rec *r, struct proxy_alias *ent, proxy_dir_conf *dconf)

入力と出力

  • **入力**:
  • r: 現在のリクエスト構造体。
  • ent: マッチング対象となるプロキシエイリアス定義。
  • dconf: ディレクトリ単位のプロキシ設定。
  • **出力**:
  • OK: マッチ成功、リクエストをプロキシ用に設定済み。
  • DECLINED: 明示的にプロキシが無効化されている（! 指定）。
  • HTTP_CONTINUE: マッチせず、処理を継続。
  • HTTP_INTERNAL_SERVER_ERROR: 正規表現置換の失敗等。
  • DONE: マッチ成功、かつエンコード済みリクエストとして処理完了。

前提条件と不変条件

  • ent->fake がリクエスト URI と一致（または正規表現マッチ）する必要がある。
  • real が ! の場合、プロキシ処理は明示的に拒否される。
  • r->filename は proxy: で始まる文字列に書き換えられる。

アルゴリズムまたは振る舞い

1. **設定適用**: dconf に基づき ent->fake と ent->real を補間（interpolate_env が有効な場合）。 2. **マッチング**:

  • 正規表現の場合: ap_regexec で判定し、ap_pregsub で置換を実行。
  • パスプレフィックスの場合: alias_match または alias_match_servlet で判定。

    • 3. **リクエスト書き換え**:

  • r->filename を proxy:<real_path> に設定。
  • r->handler を proxy-server に設定。
  • r->proxyreq を PROXYREQ_REVERSE に設定。

    • 4. **フラグ処理**: nocanon, noquery, encoded 等のフラグに応じ、r->notes に情報を格納。 5. **Servlet マッピング**: PROXYPASS_MAP_SERVLET が有効な場合、r->uri を正規化されたパスに更新。

呼び出し元と呼び出し先

  • **呼び出し元**: modules/proxy/mod_proxy.c 内の proxy_translate_name 等（ソース内の直接的な呼び出し箇所は 1027 行目および 1039 行目）。
  • **呼び出し先**: ap_proxy_interpolate, ap_regexec, ap_pregsub, alias_match, alias_match_servlet, proxy_run_check_trans 等。

関連データ構造

  • request_rec: Apache リクエスト構造体。
  • proxy_alias: プロキシのエイリアス定義（fake, real, flags, regex を保持）。
  • proxy_dir_conf: プロキシのディレクトリ設定。

失敗モードと境界条件

  • **正規表現置換失敗**: ap_pregsub が失敗した場合、HTTP_INTERNAL_SERVER_ERROR を返す。
  • **明示的拒否**: real が ! の場合、DECLINED を返す。
  • **不整合**: nocanon 指定時に URI の正規化前後でマッチ結果が異なる場合、警告をログ出力し nocanon を無視する。

保守担当者向けメモ

  • r->uri の書き換え（Servlet マッピング時）は、r->parsed_uri.path にも影響を与えるため注意が必要。
  • proxy_run_check_trans フックにより、他のプロキシモジュール（mod_proxy_wstunnel 等）が処理に介入する余地がある。
