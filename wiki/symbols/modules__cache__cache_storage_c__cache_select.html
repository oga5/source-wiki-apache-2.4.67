cache_select

定義

  • 種別: function
  • 定義箇所: modules/cache/cache_storage.c:214

目的

キャッシュストレージプロバイダーを順次走査し、リクエストに合致するキャッシュエントリの検索、Vary ヘッダーによるネゴシエーション、および鮮度確認を行う。

シグネチャまたは形状

int cache_select(cache_request_rec *cache, request_rec *r)

入力と出力

  • **入力**:
  • cache: リクエストに関連付けられたキャッシュコンテキスト。
  • r: 現在の HTTP リクエスト構造体。
  • **出力**:
  • OK: キャッシュヒット。cache->handle に有効なエントリが設定される。
  • DECLINED: キャッシュミス、または再検証が必要な状態。
  • HTTP_GATEWAY_TIME_OUT: only-if-cached 指定時にキャッシュが見つからなかった場合。
  • その他: プロバイダーによるエラーコード。

前提条件と不変条件

  • cache ポインタが非 NULL であること。
  • cache->providers リストが初期化されていること。
  • r->pool を通じたメモリ管理が行われるため、呼び出し側での明示的な解放は不要。

アルゴリズムまたは振る舞い

1. **初期チェック**: ap_cache_check_no_cache によりキャッシュ不可のリクエストか判定。 2. **キー生成**: cache->key が未設定の場合、cache_generate_key を呼び出し生成する。 3. **プロバイダー走査**: cache->providers リストを反復し、各プロバイダーの open_entity を試行。 4. **ネゴシエーション**: Vary ヘッダーを解析し、現在のリクエストとキャッシュされたリクエストのヘッダーが一致するか確認。不一致時は次のプロバイダーへ。 5. **条件付き検証**: If-Match 等の条件付きヘッダーとキャッシュの整合性を確認。 6. **鮮度確認**: cache_check_freshness を実行。鮮度が古い場合、条件付きリクエストヘッダーを付与して再検証を試みる（DECLINED を返して処理を継続）。 7. **完了**: 成功時、cache_accept_headers でヘッダーをマージし OK を返す。

呼び出し元と呼び出し先

  • **呼び出し元**: modules/cache/mod_cache.c (クイックハンドラおよびフィルタ内)
  • **呼び出し先**:
  • cache_generate_key
  • list->provider->open_entity
  • list->provider->recall_headers
  • cache_table_getm
  • ap_condition_* 系関数
  • cache_check_freshness
  • cache_accept_headers

関連データ構造

  • cache_request_rec: キャッシュの制御情報とプロバイダーリストを保持。
  • cache_handle_t: 検索されたキャッシュエントリのハンドル。
  • cache_provider_list: 利用可能なストレージプロバイダーの連結リスト。

失敗モードと境界条件

  • cache が NULL の場合、エラーログを出力し DECLINED を返す。
  • only-if-cached が設定されている場合、キャッシュミスは 504 Gateway Timeout となる。
  • Vary ヘッダーの不一致や鮮度切れは、エラーではなく「キャッシュ不適合」として次のプロバイダーへフォールバックする。

保守担当者向けメモ

  • 本関数はストレージ抽象化レイヤーの要であり、各プロバイダーの open_entity 呼び出しを制御する。
  • Vary ヘッダーの解析ロジックは RFC 2616 に準拠している。
  • 再検証が必要な場合、既存の条件付きヘッダーを削除し、キャッシュの ETag や Last-Modified を用いて再構築する処理が含まれる。
