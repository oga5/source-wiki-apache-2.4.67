cache_filter

定義

  • 種別: function
  • 定義箇所: modules/cache/mod_cache.c:1735

目的

キャッシュフィルタチェーンにおいて、冗長または無効なキャッシュ処理を検出し、フィルタ自身を削除して後続のフィルタへ処理を透過させるためのマーカーとして機能します。

シグネチャまたは形状

apr_status_t cache_filter(ap_filter_t *f, apr_bucket_brigade *in)

入力と出力

  • **入力**:
  • f: 現在のフィルタコンテキスト (ap_filter_t)。
  • in: 処理対象のバケットブリゲード (apr_bucket_brigade)。
  • **出力**:
  • apr_status_t: 後続のフィルタ (f->next) の実行結果を返します。

前提条件と不変条件

  • f->r->server->module_config から cache_server_conf を取得可能であること。
  • 本関数は、キャッシュ処理が既に完了しているか、あるいは適用対象外である状況で呼び出されることを前提としています。

アルゴリズムまたは振る舞い

1. cache_server_conf を取得し、クイックハンドラモードが有効かどうかを確認します。 2. クイックハンドラモードが有効な場合は警告ログを、それ以外の場合はデバッグログを出力し、キャッシュフィルタが二重に追加されたか、バイパスされたことを記録します。 3. ap_remove_output_filter(f) を呼び出し、フィルタスタックから自身を削除します。 4. ap_pass_brigade(f->next, in) を呼び出し、入力されたバケットブリゲードを後続のフィルタへ渡して終了します。

呼び出し元と呼び出し先

  • **呼び出し元**: modules/cache/mod_cache.c:2637 にて参照されています。
  • **呼び出し先**: ap_remove_output_filter および ap_pass_brigade を呼び出します。

関連データ構造

  • cache_server_conf: サーバー単位のキャッシュ設定を保持する構造体。
  • ap_filter_t: Apache のフィルタチェーンを構成する構造体。

失敗モードと境界条件

  • フィルタが既に削除されている場合や、f->next が存在しない場合の挙動については、ap_pass_brigade の実装に依存します。
  • 本関数自体はエラーを生成せず、後続のフィルタの戻り値をそのまま返します。

保守担当者向けメモ

  • 本関数は「マーカー」としてのみ機能します。キャッシュの有効化や無効化のロジックは含まれていません。
  • ログ出力には APLOGNO が使用されており、デバッグ時には 00776 または 00777 のエラーコードを確認してください。
