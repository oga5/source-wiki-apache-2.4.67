cache_save_filter

定義

  • 種別: function
  • 定義箇所: modules/cache/mod_cache.c:809

目的

HTTP レスポンスをキャッシュストレージへ保存するためのフィルタ処理を実行します。レスポンスのキャッシュ可能性を判定し、条件を満たす場合にストレージへの書き込みを制御します。

シグネチャまたは形状

apr_status_t cache_save_filter(ap_filter_t *f, apr_bucket_brigade *in)

入力と出力

  • **入力**:
  • f: 現在のフィルタコンテキスト。f->ctx に cache_request_rec が格納されていることを前提とする。
  • in: 送信されるデータを含むバケットブリゲード。
  • **出力**:
  • apr_status_t: 処理結果（APR_SUCCESS またはエラーコード）。

前提条件と不変条件

  • f->ctx が cache_request_rec として初期化されていること。初期化されていない場合、本フィルタは自身を削除し、データをそのまま通過させる。
  • cache->in_checked が真であれば、既にキャッシュ可能性の判定が完了しており、cache_save_store へ処理を委譲する。

アルゴリズムまたは振る舞い

1. **初期化チェック**: cache コンテキストが存在しない場合、フィルタを解除して終了する。 2. **ブロック状態確認**: cache->block_response が真であれば、既にレスポンスが送信済みとみなし、入力を無視して APR_SUCCESS を返す。 3. **キャッシュ可能性判定**:

  • cache->in_checked が偽の場合、レスポンスヘッダー（Expires, Last-Modified, Etag, Cache-Control 等）を解析する。
  • RFC 2616 に基づき、ステータスコードやヘッダー情報からキャッシュの可否を判定する。
  • dconf->stale_on_error が有効な場合、エラー発生時に古いキャッシュを再利用する処理を行う。

    • 4. **ストレージへの書き込み**: 判定の結果、キャッシュ可能と判断された場合、cache_save_store を呼び出してデータをストレージへ流し込む。

呼び出し元と呼び出し先

  • **呼び出し元**: フィルタチェーン（modules/cache/mod_cache.c:2647, 2657 に参照の可能性あり）。
  • **呼び出し先**:
  • cache_save_store（キャッシュ保存処理）
  • ap_pass_brigade（フィルタチェーンの次段へのデータ転送）
  • cache_run_cache_status（キャッシュ状態の通知）

関連データ構造

  • cache_request_rec: リクエストごとのキャッシュ状態を保持。
  • cache_server_conf: サーバー全体の設定。
  • cache_dir_conf: ディレクトリ単位の設定。
  • cache_control_t: Cache-Control ヘッダーの解析結果。

失敗モードと境界条件

  • **キャッシュ無効**: 設定やヘッダー（no-store, private 等）によりキャッシュ不可と判断された場合、フィルタはデータを通過させる。
  • **エラー時**: stale_on_error 設定がない場合、エラーレスポンスはキャッシュされない。
  • **不正なヘッダー**: Expires 等が不正な形式の場合、キャッシュ対象外となる。

保守担当者向けメモ

  • 本関数は CACHE_SAVE フィルタのメインエントリです。
  • キャッシュ可能性の判定ロジックは RFC 2616 に準拠していますが、複雑な条件分岐が含まれるため、修正時は cache_control_t の解析結果と dconf の設定値の整合性に注意してください。
  • cache_save_store への移行タイミング（in_checked フラグ）がキャッシュフローの要となります。
