systemd_pre_mpm

定義

  • 種別: function
  • 定義箇所: modules/arch/unix/mod_systemd.c:88

目的

Apache HTTP Server の MPM（Multi-Processing Module）起動直前に、サーバーの準備完了状態とメインプロセス ID を systemd サービスマネージャへ通知します。

シグネチャまたは形状

static int systemd_pre_mpm(apr_pool_t *p, ap_scoreboard_e sb_type)

入力と出力

  • **入力**:
  • p: メモリ割り当て用のプール（未使用）。
  • sb_type: スコアボードの型（未使用）。
  • **出力**:
  • 常に OK を返します。

前提条件と不変条件

  • sd_notifyf を介して systemd と通信するため、実行環境で libsystemd が利用可能である必要があります。
  • 本関数は MPM の起動フェーズで実行されることを前提としています。

アルゴリズムまたは振る舞い

1. sd_notifyf を呼び出し、以下の情報を systemd に送信します。

  • READY=1: サービスが準備完了状態であることを通知。
  • STATUS=Processing requests...: サービスのステータス文字列を設定。
  • MAINPID=...: 現在のプロセス ID (getpid()) を通知。

    • 2. 処理完了後、OK を返して終了します。

呼び出し元と呼び出し先

  • **呼び出し元**: modules/arch/unix/mod_systemd.c:166 にて ap_hook_pre_mpm を通じて登録されており、MPM 起動時にフックされます。
  • **呼び出し先**:
  • sd_notifyf (systemd ライブラリ関数)
  • getpid (システムコール)

関連データ構造

  • apr_pool_t: Apache のメモリ管理プール。
  • ap_scoreboard_e: スコアボードの列挙型。

失敗モードと境界条件

  • sd_notifyf の実行結果については本関数内でチェックを行わず、常に OK を返します。通信失敗時のエラーハンドリングは実装されていません。

保守担当者向けメモ

  • 本関数は ap_hook_pre_mpm の APR_HOOK_LAST として登録されています。
  • READY=1 の通知は、リスニングソケットの準備が整った段階で行われる設計となっています。
