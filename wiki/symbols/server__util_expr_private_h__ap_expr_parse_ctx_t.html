ap_expr_parse_ctx_t

定義

  • 種別: struct
  • 定義箇所: server/util_expr_private.h:83

目的

Apache HTTP Server の式評価エンジン（ap_expr）において、構文解析および字句解析のプロセス全体を通じて状態を保持・管理するためのコンテキスト構造体です。

シグネチャまたは形状

typedef struct {
    /* スキャナーの内部状態 */
    const char        *inputbuf;
    int                inputlen;
    const char        *inputptr;
    void              *scanner;
    char              *scan_ptr;
    char               scan_buf[MAX_STRING_LEN];
    char               scan_del;
    int                at_start;

    /* メモリ管理用プール */
    apr_pool_t        *pool;
    apr_pool_t        *ptemp;

    /* 生成された抽象構文木 (AST) */
    ap_expr_t         *expr;

    /* エラー情報とフラグ */
    const char        *error;
    const char        *error2;
    unsigned           flags;

    /* 変数・関数ルックアップ用関数ポインタ */
    ap_expr_lookup_fn_t *lookup_fn;
} ap_expr_parse_ctx_t;

入力と出力

  • **入力**: 解析対象の文字列（inputbuf）およびルックアップ関数（lookup_fn）。
  • **出力**: 解析結果として構築された抽象構文木（expr）、および解析中に発生したエラーメッセージ（error, error2）。

前提条件と不変条件

  • pool および ptemp は、解析ライフサイクル全体で有効な APR メモリプールである必要があります。
  • inputbuf は解析終了まで生存している必要があります。
  • scan_buf は MAX_STRING_LEN までの文字列を保持可能です。

アルゴリズムまたは振る舞い

1. **初期化**: 解析開始時にコンテキストが初期化され、入力文字列とメモリプールが設定されます。 2. **解析**: ap_expr_yyparse が呼び出されると、スキャナーが inputbuf を走査し、トークンを生成します。 3. **AST 構築**: スキャナーとパーサーが連携し、ap_expr_make 等を用いて AST を構築し、結果を expr フィールドに格納します。 4. **エラーハンドリング**: 解析中に問題が発生した場合、error または error2 に詳細が記録されます。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • server/util_expr_eval.c: 式の評価や解析の開始時にインスタンス化されます。
  • server/util_expr_parse.c: Bison 生成のパーサーが解析状態を保持するために参照します。
  • **呼び出し先**:
  • lookup_fn: 変数や関数の解決時に動的に呼び出されます。

関連データ構造

  • ap_expr_t: 解析結果として生成される抽象構文木のノード。
  • apr_pool_t: メモリ管理に使用される APR プール。
  • ap_expr_lookup_fn_t: 変数や関数を解決するためのコールバック関数型。

失敗モードと境界条件

  • inputbuf が空、または解析不能な構文が含まれる場合、error フィールドにエラー内容が設定されます。
  • scan_buf のサイズ（MAX_STRING_LEN）を超えるトークンは、バッファオーバーフローの境界条件となります。
  • メモリ不足時には、APR プール操作を通じてエラーが発生する可能性があります。

保守担当者向けメモ

  • 本構造体は util_expr_parse.c（Bison）と util_expr_scan.c（Flex）の間で共有されるため、フィールドの変更は両方の生成コードに影響を与えます。
  • ptemp は一時的なメモリ割り当てに使用されるため、解析終了後に解放されることを前提とした設計になっています。
