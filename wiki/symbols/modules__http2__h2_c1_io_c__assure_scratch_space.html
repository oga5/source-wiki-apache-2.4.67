assure_scratch_space

定義

  • 種別: function
  • 定義箇所: modules/http2/h2_c1_io.c:192

目的

HTTP/2 クライアント接続（C1）の出力処理において、データを一時的に蓄積するためのスクラッチバッファを確保または維持します。

シグネチャまたは形状

static apr_size_t assure_scratch_space(h2_c1_io *io)

入力と出力

  • **入力**: h2_c1_io *io（接続の入出力管理構造体）
  • **出力**: apr_size_t（スクラッチバッファ内の残り書き込み可能バイト数）

前提条件と不変条件

  • io->session->c1->bucket_alloc が有効なバケットアロケータとして存在していること。
  • io->write_size には、TLS 最適化を考慮した適切なバッファサイズが設定されていること。

アルゴリズムまたは振る舞い

1. 現在のバッファの残り容量（remain）を計算します。 2. バッファが既に存在し、かつ満杯（remain == 0）である場合、append_scratch(io) を呼び出して既存のバッファをブリゲードへフラッシュします。 3. スクラッチバッファが未割り当ての場合、io->write_size に基づいて新しいバケットを割り当て、バッファのメタデータ（ssize, slen）を初期化します。 4. 最終的な残り書き込み可能バイト数を返します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • modules/http2/h2_c1_io.c:371
  • modules/http2/h2_c1_io.c:407
  • **呼び出し先**:
  • append_scratch（バッファが満杯の場合）
  • apr_bucket_alloc（バッファの新規確保時）

関連データ構造

  • h2_c1_io: 接続の入出力状態を保持する構造体。
  • apr_bucket: Apache のデータ転送単位。

失敗モードと境界条件

  • メモリ不足時: apr_bucket_alloc が失敗した場合の挙動は、Apache の標準的なメモリ管理に依存します。
  • io->scratch が既に存在する場合、その容量は io->write_size に基づいて管理されます。

保守担当者向けメモ

  • 本関数は、TLS レコードの効率的なパッキングを目的としています。バッファサイズは h2_c1_io のライフサイクルを通じて動的に調整される可能性があるため、write_size の変更が適切に反映されるか注意してください。
