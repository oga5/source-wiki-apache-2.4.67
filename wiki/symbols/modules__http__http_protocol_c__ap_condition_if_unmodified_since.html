ap_condition_if_unmodified_since

定義

  • 種別: function
  • 定義箇所: modules/http/http_protocol.c:349

目的

HTTP リクエストに含まれる If-Unmodified-Since ヘッダーを評価し、リソースが指定時刻以降に変更されていないかを判定する。

シグネチャまたは形状

ap_condition_e ap_condition_if_unmodified_since(request_rec *r, apr_table_t *headers)

入力と出力

  • **入力**:
  • r: 現在のリクエスト構造体。r->headers_in から If-Unmodified-Since を取得し、r->mtime や r->request_time をフォールバックとして使用する。
  • headers: 比較対象のヘッダーテーブル（通常は r->headers_out）。Last-Modified および Date を取得する。
  • **出力**:
  • ap_condition_e 型の列挙値:
  • AP_CONDITION_NONE: ヘッダーが存在しない場合。
  • AP_CONDITION_NOMATCH: 条件を満たさない（リソースが更新されている）場合。
  • AP_CONDITION_WEAK: 条件を満たし、かつ弱一致と判定される場合。
  • AP_CONDITION_STRONG: 条件を満たし、かつ強一致と判定される場合。

前提条件と不変条件

  • 時刻比較はすべて秒単位で行われる（HTTP 仕様の最小解像度に準拠）。
  • Last-Modified が取得できない場合は r->mtime を、r->mtime もない場合は現在時刻を使用する。
  • Date が取得できない場合は r->request_time を使用する。

アルゴリズムまたは振る舞い

1. If-Unmodified-Since ヘッダーを解析し、時刻 ius を取得する。存在しない場合は AP_CONDITION_NONE を返す。 2. headers から Last-Modified (mtime) と Date (reqtime) を取得・解析する。 3. mtime > ius（リソースの更新時刻が指定時刻より後）の場合にのみ条件判定を行う。

  • reqtime < mtime + 60 かつ Range ヘッダーが存在する場合: AP_CONDITION_NOMATCH を返す（Range リクエストでの弱一致禁止）。
  • reqtime < mtime + 60 の場合: AP_CONDITION_WEAK を返す。
  • それ以外の場合: AP_CONDITION_STRONG を返す。

    • 4. mtime <= ius の場合は AP_CONDITION_NOMATCH を返す。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • modules/cache/cache_storage.c:325
  • modules/http/http_protocol.c:581 (ap_meets_conditions 内)
  • **呼び出し先**:
  • apr_table_get (APR)
  • apr_date_parse_http (APR)
  • apr_time_sec (APR)
  • apr_time_now (APR)

関連データ構造

  • request_rec: リクエスト情報を保持する主要構造体。
  • apr_table_t: ヘッダー情報を保持するテーブル構造体。
  • ap_condition_e: 判定結果を格納する列挙型。

失敗モードと境界条件

  • If-Unmodified-Since ヘッダーが不正な形式の場合、apr_date_parse_http は APR_DATE_BAD を返し、比較ロジックにより AP_CONDITION_NOMATCH となる。
  • Range リクエストが存在する場合、弱一致 (AP_CONDITION_WEAK) は許可されず AP_CONDITION_NOMATCH に格下げされる。

保守担当者向けメモ

  • 本関数は ap_meets_conditions による条件評価フローの一部として呼び出される。
  • 比較ロジック内の mtime + 60 という定数は、HTTP 仕様における許容誤差（クロックのずれ）を考慮した実装である。
