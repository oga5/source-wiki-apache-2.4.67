buffer_ctx

定義

  • **種別**: struct
  • **定義箇所**: modules/filters/mod_buffer.c:44

目的

リクエストごとのフィルタ実行時状態を保持し、バッファリング処理（入力および出力フィルタ）のコンテキストを管理するために使用されます。

シグネチャまたは形状

typedef struct buffer_ctx {
    apr_bucket_brigade *bb;      /* メインバッファ */
    apr_bucket_brigade *tmp;     /* 入力用一時バッファ */
    buffer_conf *conf;           /* ディレクトリ単位の設定 */
    apr_off_t remaining;         /* 残り容量 */
    int seen_eos;                /* EOS（End of Stream）到達フラグ */
} buffer_ctx;

入力と出力

  • **入力**: フィルタ関数（buffer_out_filter, buffer_in_filter）において、f->ctx を通じて参照されます。
  • **出力**: フィルタの実行状態（バッファ内のデータ、EOS到達状況、残り容量）を保持し、後続のフィルタ呼び出しへ引き継がれます。

前提条件と不変条件

  • buffer_ctx およびその内部のブリゲードは、リクエストプール（r->pool）に紐付いており、リクエスト終了時に自動的に解放されます。
  • 出力フィルタにおいて、サブクエスト（f->r->main が存在する場合）はバッファリングの対象外となります。

アルゴリズムまたは振る舞い

  • **バッファ管理**: buffer_conf に基づき、データを bb に蓄積します。
  • **出力処理**: バッファサイズが上限に達した場合、またはフラッシュ/EOSバケットを受信した際に、蓄積されたデータを下流へ送信します。
  • **入力処理**: AP_MODE_READBYTES モード時、上流から読み込んだデータを tmp に蓄積し、要求されたサイズ分を切り出して提供します。

呼び出し元と呼び出し先

  • **呼び出し元**: buffer_out_filter および buffer_in_filter が f->ctx を介してこの構造体を取得します。
  • **呼び出し先**: 構造体自体はデータ保持用であり、直接関数を呼び出すことはありません。

関連データ構造

  • buffer_conf: ディレクトリ単位の設定（バッファ上限サイズ等）を保持。
  • apr_bucket_brigade: バケットの連結リスト。bb および tmp として使用。

失敗モードと境界条件

  • **サブクエスト**: 出力フィルタにおいて、サブクエストの場合はバッファリングを行わず透過させます。
  • **入力モード**: 入力フィルタにおいて、AP_MODE_READBYTES 以外のモード（GETLINE 等）ではバッファリングを行わず透過させます。

保守担当者向けメモ

  • 本構造体はリクエストプールに依存しているため、メモリリークの心配は少ないですが、サブクエストや非標準的な入力モードでの挙動に注意してください。
