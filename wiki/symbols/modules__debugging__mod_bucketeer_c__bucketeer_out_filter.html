bucketeer_out_filter

定義

  • 種別: function
  • 定義箇所: modules/debugging/mod_bucketeer.c:61

目的

Apache HTTP Server の出力フィルタとして、データストリーム内の特定の制御文字（デリミタ）を検出し、バケットの分割、フラッシュ、およびブリゲードの送信を強制的に実行することで、フィルタチェーンの挙動をテスト・検証する。

シグネチャまたは形状

static apr_status_t bucketeer_out_filter(ap_filter_t *f, apr_bucket_brigade *bb)

入力と出力

  • **入力**:
  • f: フィルタコンテキストを含むフィルタ構造体。
  • bb: 処理対象のバケットブリゲード。
  • **出力**:
  • apr_status_t: 処理結果（成功時は APR_SUCCESS）。

前提条件と不変条件

  • コンテンツタイプが text/ で始まる場合のみフィルタリングを行う。それ以外はフィルタを解除し、透過的に処理する。
  • フィルタ実行時に Content-Length ヘッダーを削除する（バケット再構成によるデータ長変化のため）。
  • 初回実行時に bucketeer_ctx_t コンテキストを生成し、以降の呼び出しで再利用する。

アルゴリズムまたは振る舞い

1. コンテキスト未生成の場合、対象コンテンツタイプを判定し、非対象ならフィルタを解除して ap_pass_brigade を実行。 2. ブリゲード内の各バケットを走査する。

  • **EOS**: ブリゲードを送信して終了。
  • **FLUSH**: 無視する。
  • **METADATA**: コンテキスト内のブリゲードにコピーを追加。
  • **データバケット**: データを読み込み、設定されたデリミタ（flushdelimiter, bucketdelimiter, passdelimiter）を検索。
  • デリミタ発見時、バケットを分割して新しいバケットを作成し、コンテキスト内のブリゲードに追加。
  • flushdelimiter の場合、フラッシュバケットを挿入。
  • passdelimiter の場合、コンテキスト内のブリゲードを下流へ送信。

    • 3. 走査終了後、残りのデータをコンテキスト内のブリゲードに追加する。

呼び出し元と呼び出し先

  • **呼び出し元**: ap_register_output_filter により登録（modules/debugging/mod_bucketeer.c:171）。
  • **呼び出し先**: ap_pass_brigade（下流フィルタへのデータ送信）、apr_bucket_read、apr_bucket_pool_create 等の APR バケット操作関数。

関連データ構造

  • bucketeer_filter_config_t: デリミタ設定を保持。
  • bucketeer_ctx_t: フィルタ実行中の状態（ブリゲード）を保持。

失敗モードと境界条件

  • ap_pass_brigade がエラーを返した場合、そのステータスを即座に返す。
  • バイナリデータが含まれる場合、意図しないデリミタ検知が発生する可能性がある。
  • 残りのデータ処理において、次の「実」バケットへの追加が不完全であるという既知の課題（XXXコメント）が存在する。

保守担当者向けメモ

  • 本モジュールはデバッグ専用であり、実運用環境での使用は推奨されない。
  • Content-Length ヘッダーの削除は必須の副作用である。
  • フィルタ内の XXX コメントにある通り、残りのデータ処理ロジックに改善の余地がある。
