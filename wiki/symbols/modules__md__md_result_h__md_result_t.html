md_result_t

定義

  • 種別: struct
  • 定義箇所: modules/md/md_result.h:23

目的

mod_md モジュールにおける処理結果、エラー状態、および進行状況を保持・管理するためのコンテナ構造体です。ACME プロトコル等の非同期処理において、ステータスや詳細情報を一元管理します。

シグネチャまたは形状

typedef struct md_result_t md_result_t;

入力と出力

  • **入力**: md_result_make() 関数により生成され、処理対象の md_name 等が設定されます。
  • **出力**: md_result_to_json() を介した JSON 形式へのシリアライズ、または md_result_log() を介したログ出力が可能です。

前提条件と不変条件

  • 本構造体は apr_pool_t に強く依存しており、メモリ管理のために有効なプールが生存している必要があります。

アルゴリズムまたは振る舞い

1. **状態更新**: md_result_activity_set や md_result_problem_set を通じて、処理の進行状況やエラー内容を更新します。 2. **通知**: 更新メソッドの実行に伴い、登録された on_change コールバックが呼び出され、外部の監視者へ状態変化を伝達します。 3. **エラー管理**: ACME プロトコル特有のサブ問題（subproblems）を保持し、詳細なエラーハンドリングをサポートします。

呼び出し元と呼び出し先

  • **呼び出し元（参照箇所）**:
  • modules/md/md.h
  • modules/md/md_acme.c (例: md_acme_report_result, md_acme_setup)
  • modules/md/md_acme.h
  • **呼び出し先**:
  • 本構造体自体はデータ保持用であり、操作には md_result_make, md_result_problem_set 等の API 関数が使用されます。

関連データ構造

  • apr_pool_t: メモリ管理に使用。
  • md_json.h: サブ問題の保持やシリアライズに使用。

失敗モードと境界条件

  • on_change コールバック内での重い処理は避ける必要があります（状態更新のたびに実行されるため）。
  • subproblems フィールドは、サーバーから返される詳細なエラー情報を扱うために不可欠です。

保守担当者向けメモ

  • md_result_t を使用する際は、必ず有効な apr_pool_t が生存していることを確認してください。
  • 状態更新コールバックの設計において、パフォーマンスへの影響を考慮してください。
