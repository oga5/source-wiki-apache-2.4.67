walk_walked_t

定義

  • 種別: struct
  • 定義箇所: server/request.c:436

目的

リクエスト処理における設定マージ（Walk）プロセスにおいて、特定のパスやセクションに対して適用された設定の履歴を保持するために使用されます。

シグネチャまたは形状

typedef struct walk_walked_t {
    ap_conf_vector_t *matched; /* 適合した dir_conf セクション */
    ap_conf_vector_t *merged;  /* マージされた dir_conf の結果 */
} walk_walked_t;

入力と出力

  • 本構造体は walk_cache_t 内の配列 walked の要素として格納されます。
  • matched は、リクエストパスに対して適合した設定ベクトルを指します。
  • merged は、その時点までの設定が統合された結果ベクトルを指します。

前提条件と不変条件

  • walk_cache_t のライフサイクルに依存し、リクエストのプール（r->pool）から割り当てられたメモリ上で管理されます。
  • walked 配列への追加は、apr_array_push を通じて行われます。

アルゴリズムまたは振る舞い

  • リクエスト処理の Walk フェーズ（ap_directory_walk 等）において、設定がマージされるたびに新しい walk_walked_t インスタンスが生成され、キャッシュ配列に記録されます。
  • これにより、後続のサブリクエストやリダイレクト時に、以前のマージ結果を再利用することが可能になります。

呼び出し元と呼び出し先

  • **呼び出し元**: server/request.c 内の ap_directory_walk、ap_location_walk、ap_file_walk、ap_if_walk 等の Walk 関数群。
  • **呼び出し先**: 直接的な関数呼び出しではなく、apr_array_push による配列操作および cache->walked へのアクセスを通じて利用されます。

関連データ構造

  • walk_cache_t: 本構造体のリストを保持する親構造体。
  • ap_conf_vector_t: Apache の設定ベクトル。
  • request_rec: リクエストのコンテキスト。

失敗モードと境界条件

  • メモリ不足: apr_array_make や apr_array_push が失敗した場合、APR のメモリ管理ポリシーに従いプロセスが終了する可能性があります。
  • キャッシュの不整合: walked 配列の要素数と実際のマージ回数が一致しない場合、設定の適用順序や結果に影響を与える可能性があります。

保守担当者向けメモ

  • 本構造体は server/request.c 内のキャッシュ機構の根幹です。Walk ロジックを変更する際は、matched と merged の整合性が維持されていることを確認してください。
  • walked 配列の要素は r->pool に紐付いているため、サブリクエストの破棄時に適切に解放されることを前提としています。
