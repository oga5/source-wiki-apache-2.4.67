ap_condition_if_none_match

定義

  • 種別: function
  • 定義箇所: modules/http/http_protocol.c:397

目的

HTTP リクエストに含まれる If-None-Match ヘッダーを評価し、リソースの ETag との比較に基づいて条件付きリクエストのステータスを判定します。

シグネチャまたは形状

ap_condition_e ap_condition_if_none_match(request_rec *r, apr_table_t *headers)

入力と出力

  • **入力**:
  • r: 現在のリクエストを保持する request_rec 構造体。r->headers_in から If-None-Match ヘッダーを取得し、r->method_number でメソッドを判定します。
  • headers: 比較対象の ETag を含む apr_table_t。
  • **出力**:
  • ap_condition_e 型の列挙値:
  • AP_CONDITION_STRONG: 条件が強力に一致（または * 指定）。
  • AP_CONDITION_WEAK: 条件が弱く一致。
  • AP_CONDITION_NOMATCH: 条件が一致しない。
  • AP_CONDITION_NONE: If-None-Match ヘッダーが存在しない。

前提条件と不変条件

  • r->pool はメモリ確保に使用され、リクエスト終了時に自動解放されます。
  • If-None-Match が * の場合、無条件に AP_CONDITION_STRONG を返します。

アルゴリズムまたは振る舞い

1. r->headers_in から If-None-Match を取得します。存在しない場合は AP_CONDITION_NONE を返します。 2. 値が * であれば AP_CONDITION_STRONG を返します。 3. GET メソッドの場合:

  • Range ヘッダーが存在すれば ap_find_etag_strong で強力な一致を判定します。
  • Range がなければ ap_find_etag_weak で弱い一致を判定します。

    • 4. GET 以外の場合:

  • ap_find_etag_strong を使用して強力な一致のみを判定します。

    • 5. 一致しない場合は AP_CONDITION_NOMATCH を返します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • modules/cache/cache_storage.c:327
  • modules/http/http_protocol.c:601
  • **呼び出し先**:
  • apr_table_get (APR)
  • ap_find_etag_strong / ap_find_etag_weak (内部ユーティリティ)

関連データ構造

  • request_rec: リクエスト情報および入力ヘッダーの保持。
  • apr_table_t: ヘッダー情報の格納。
  • ap_condition_e: 戻り値の列挙型。

失敗モードと境界条件

  • If-None-Match ヘッダーが存在しない場合、判定を行わず AP_CONDITION_NONE を返します。
  • ETag ヘッダーが headers 引数内に存在しない場合、一致判定は行われず AP_CONDITION_NOMATCH となります。

保守担当者向けメモ

  • RFC 2616 (Section 13.3.3) のエンティティタグ比較ルールに基づいています。
  • 弱い比較は GET または HEAD リクエストでのみ許可される仕様ですが、本関数は GET メソッドのみを明示的にチェックしています。
