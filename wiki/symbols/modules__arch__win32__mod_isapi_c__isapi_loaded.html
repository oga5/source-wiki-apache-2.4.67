isapi_loaded

定義

  • 種別: struct
  • 定義箇所: modules/arch/win32/mod_isapi.c:78

目的

Apache HTTP Server 上でロードされた ISAPI 拡張（DLL）の実行状態、関数ポインタ、およびライフサイクル情報を保持するためのキャッシュ構造体です。

シグネチャまたは形状

ソースコード上では struct isapi_loaded として定義されています。具体的なメンバ構成は modules/arch/win32/mod_isapi.c:206 以降に定義されており、主に以下の情報を保持します。

  • ロードされた DLL のハンドル（apr_dso_handle_t）
  • ISAPI 拡張のエントリポイント関数（GetExtensionVersion, HttpExtensionProc, TerminateExtension）
  • ロード状態およびキャッシュ管理用データ

入力と出力

本構造体自体はデータ保持用ですが、以下の操作で利用されます。

  • 入力: isapi_load 関数において、プールおよびサーバー設定と共に初期化されます。
  • 出力: isapi_lookup 等を通じて、リクエスト処理に必要な DLL のエントリポイント情報を提供します。

前提条件と不変条件

  • isapi_loaded 構造体は、apr_thread_rwlock_t を用いて複数スレッドからの同時アクセスに対して保護されています。
  • リクエスト処理中、関連するコンテキスト（isapi_cid）はリクエストプール (r->pool) に紐付けられ、リクエスト終了時に自動的に解放されます。

アルゴリズムまたは振る舞い

  • **ロード管理**: isapi_load 関数により、apr_dso_load を介して DLL がメモリにロードされ、必要な関数ポインタが解決されます。
  • **キャッシュ**: グローバルハッシュテーブルにより管理され、同一の DLL が複数回ロードされることを防ぎます。
  • **アンロード**: isapi_unload 関数により、DLL の解放とリソースのクリーンアップが行われます。

呼び出し元と呼び出し先

  • 呼び出し元: isapi_lookup（DLL 検索時）、isapi_load（DLL ロード時）、isapi_unload（DLL 解放時）、および isapi_handler（リクエスト処理時）。
  • 呼び出し先: 本構造体はデータ構造であるため、直接の呼び出し先はありませんが、保持する関数ポインタを通じて ISAPI 拡張の HttpExtensionProc 等が実行されます。

関連データ構造

  • isapi_cid: リクエスト処理中の接続コンテキスト。
  • isapi_global_conf: ロード済みモジュールを管理するグローバルハッシュテーブル。
  • EXTENSION_CONTROL_BLOCK: ISAPI 拡張と通信するための標準構造体。

失敗モードと境界条件

  • DLL のロード失敗（apr_dso_load のエラー）が発生した場合、isapi_load は適切なエラーコードを返し、リクエスト処理は中断されます。
  • fake_async 設定が有効な場合、非同期 I/O のエミュレーションにおいてミューテックスによる待機が発生し、高負荷時にはスレッド枯渇のリスクがあります。

保守担当者向けメモ

  • 本構造体は mod_isapi.c 内で完結しており、外部モジュールからの直接操作は推奨されません。
  • メモリ管理は Apache のプールシステムに依存しているため、手動でのメモリ解放は行わないでください。
  • isapi_loaded のメンバを変更する場合は、isapi_load および isapi_unload での整合性確保に注意してください。
