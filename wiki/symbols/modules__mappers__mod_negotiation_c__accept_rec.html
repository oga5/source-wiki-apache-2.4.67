accept_rec

定義

  • 種別: struct
  • 定義箇所: modules/mappers/mod_negotiation.c:175

目的

HTTP リクエストに含まれる Accept ヘッダーの各要素（MIMEタイプ、品質値など）を保持し、コンテンツネゴシエーションにおける比較・評価の単位として利用する。

シグネチャまたは形状

typedef struct accept_rec {
    char *name;                 /* MUST be lowercase */
    float quality;
    float level;
    char *charset;              /* for content-type only */
} accept_rec;

入力と出力

  • **入力**: parse_accept_headers() 等の関数により、クライアントから送信されたヘッダー文字列から解析された値が格納される。
  • **出力**: best_match() 等の選択アルゴリズムにおいて、サーバー側のバリアント（var_rec）と比較するためのデータソースとして利用される。

前提条件と不変条件

  • name メンバは必ず小文字で格納されていなければならない。
  • charset メンバは、name が Content-Type を指す場合にのみ使用される。

アルゴリズムまたは振る舞い

本構造体自体はデータ保持用であり、ロジックは含まない。mod_negotiation 内のネゴシエーション処理において、apr_array_header_t に格納された accept_rec の配列を走査し、各バリアントの適合度を計算する際に参照される。

呼び出し元と呼び出し先

  • **呼び出し元（参照・操作箇所）**:
  • set_mime_fields (引数として利用)
  • get_entry (解析結果の格納先として利用)
  • apr_array_make / apr_array_push (配列管理用)
  • **呼び出し先**: 構造体であるため、関数呼び出しは発生しない。

関連データ構造

  • var_rec: サーバー側のバリアント属性を保持する構造体。accept_rec と対比してネゴシエーションの判断材料となる。
  • negotiation_state: リクエストごとの状態を管理する構造体。accept_rec の配列を保持する。

失敗モードと境界条件

  • メモリ割り当て失敗: apr_array_push 等の APR 配列操作時にメモリ不足が発生した場合、APR の規約に従いプロセスが終了する可能性がある。
  • 不正なヘッダー値: name が小文字化されていない場合、比較ロジックが正しく機能しない可能性がある。

保守担当者向けメモ

  • name フィールドの小文字化は、解析処理側（parse_accept_headers 等）で保証されている必要がある。
  • accept_rec の配列は request_rec のプール (r->pool) から割り当てられるため、リクエスト終了時に自動的に解放される。
