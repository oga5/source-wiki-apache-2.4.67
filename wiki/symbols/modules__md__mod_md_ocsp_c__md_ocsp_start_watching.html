md_ocsp_start_watching

定義

  • 種別: function
  • 定義箇所: modules/md/mod_md_ocsp.c:215

目的

OCSP Stapling の定期的な更新および古いレスポンスファイルの削除を行うための mod_watchdog インスタンスを登録・起動する。

シグネチャまたは形状

apr_status_t md_ocsp_start_watching(md_mod_conf_t *mc, server_rec *s, apr_pool_t *p)

入力と出力

  • **入力**:
  • mc: md_mod_conf_t 型のモジュール設定構造体。
  • s: server_rec 型のサーバーレコード。
  • p: 親となる apr_pool_t。
  • **出力**:
  • apr_status_t: 成功時は APR_SUCCESS、失敗時はエラーコードを返す。

前提条件と不変条件

  • mod_watchdog がサーバーにロードされている必要がある。
  • 内部で専用の apr_pool_t と apr_allocator_t を作成し、Watchdog のライフサイクル全体でデータを保持する。

アルゴリズムまたは振る舞い

1. ap_watchdog_get_instance 等のオプション関数を取得し、利用可能か検証する。 2. Watchdog 実行用の専用メモリプールとアロケータを作成する。 3. ocsp_remove_old_responses を呼び出し、古い OCSP レスポンスファイルを削除するハウスキーピングを実行する。 4. MD_OCSP_WATCHDOG_NAME を指定して Watchdog インスタンスを取得する。 5. run_watchdog をコールバックとして登録し、定期実行を開始する。

呼び出し元と呼び出し先

  • **呼び出し元**: modules/md/mod_md.c:1033
  • **呼び出し先**:
  • ap_watchdog_get_instance (オプション関数)
  • ap_watchdog_register_callback (オプション関数)
  • ap_watchdog_set_callback_interval (オプション関数)
  • ocsp_remove_old_responses
  • run_watchdog (コールバックとして登録)

関連データ構造

  • md_ocsp_ctx_t: Watchdog の実行コンテキストを保持する構造体。
  • md_mod_conf_t: モジュール設定を保持する構造体。

失敗モードと境界条件

  • mod_watchdog がロードされていない場合、APR_EGENERAL を返しエラーログを出力する。
  • メモリプールの作成に失敗した場合、そのエラーコードを返す。
  • Watchdog インスタンスの取得またはコールバック登録に失敗した場合、そのエラーコードを返す。

保守担当者向けメモ

  • 本関数はサーバー起動時に一度だけ実行されることを想定している。
  • Watchdog の実行コンテキストは、サーバーのライフサイクルを跨いで生存させるため、専用の apr_pool_t を使用している。
  • ocsp_remove_old_responses によるハウスキーピングは、証明書の変更や削除を明示的に検知できない仕様を補完するためのものである。
