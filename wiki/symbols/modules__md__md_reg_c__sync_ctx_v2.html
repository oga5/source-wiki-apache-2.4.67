sync_ctx_v2

定義

  • 種別: struct
  • 定義箇所: modules/md/md_reg.c:820

目的

md_reg_sync 処理において、マスター設定と永続化ストア間のドメイン情報の同期状態を追跡・管理するためのコンテキスト構造体です。

シグネチャまたは形状

typedef struct {
    apr_pool_t *p;
    apr_array_header_t *master_mds;
    apr_array_header_t *store_names;
    apr_array_header_t *maybe_new_mds;
    apr_array_header_t *new_mds;
    apr_array_header_t *unassigned_mds;
} sync_ctx_v2;

入力と出力

  • 本構造体は関数内のローカル変数として初期化され、同期処理の各フェーズで各配列フィールドにドメイン情報が分類・格納されます。

前提条件と不変条件

  • p は、同期処理の生存期間を管理する apr_pool_t である必要があります。
  • 各 apr_array_header_t フィールドは、同期ロジックの進行に応じて適切に初期化・更新されることが期待されます。

アルゴリズムまたは振る舞い

  • 同期処理において、現在のマスター設定とストア上の情報を比較し、以下の分類を行います。
  • master_mds: 現在のマスター設定に含まれるドメイン群。
  • store_names: ストア上に存在するドメイン名リスト。
  • maybe_new_mds / new_mds: 新規追加候補および確定した新規ドメイン。
  • unassigned_mds: 割り当てられていないドメイン。

呼び出し元と呼び出し先

  • 呼び出し元: modules/md/md_reg.c 内の同期関連関数（md_reg_sync 等の内部処理）。
  • 呼び出し先: 本構造体自体はデータ保持用であり、直接関数を呼び出すことはありません。

関連データ構造

  • apr_array_header_t: APRの配列構造体。各ドメイン情報のリスト保持に使用。
  • apr_pool_t: メモリ管理用プール。

失敗モードと境界条件

  • メモリ不足時: apr_array_header_t の初期化や要素追加時に apr_pool_t 経由でエラーが発生する可能性があります。

保守担当者向けメモ

  • 本構造体は md_reg_sync の内部的な状態保持を目的としています。
  • sync_ctx_v2 という名称から、以前の同期ロジック（v1）からの移行または改善版であることが示唆されます。
  • 構造体の各フィールドは、同期処理の各ステップで整合性を保つ必要があるため、変更時は同期ロジック全体への影響を確認してください。
