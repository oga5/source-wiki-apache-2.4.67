is_best_bybusyness

定義

  • 種別: function
  • 定義箇所: modules/proxy/balancers/mod_lbmethod_bybusyness.c:28

目的

ロードバランサーがリクエストを振り分ける際、複数のワーカーの中から「最も負荷が低い（最適な）」ワーカーを決定するための比較基準を提供します。

シグネチャまたは形状

static int is_best_bybusyness(proxy_worker *current, proxy_worker *prev_best, void *baton)

入力と出力

  • **入力**:
  • current: 現在評価中のワーカー。
  • prev_best: これまでに評価した中で最も適していると判断されたワーカー。
  • baton: int *total_factor としてキャストされ、累積の重み合計を保持するポインタ。
  • **出力**:
  • int: current が prev_best よりも適している場合は非ゼロ（真）、そうでない場合は 0（偽）を返します。

前提条件と不変条件

  • baton は int 型のポインタとして正しく初期化されている必要があります。
  • current および prev_best の s メンバ（proxy_worker_shared）が有効である必要があります。

アルゴリズムまたは振る舞い

1. current ワーカーの lbstatus を lbfactor 分だけ増加させ、total_factor にも加算します。 2. 以下のいずれかの条件を満たす場合、current を「より良い」と判定します。

  • prev_best が NULL である（最初のワーカーである）。
  • current の busy 数が prev_best より小さい。
  • busy 数が同数の場合、current の lbstatus が prev_best より大きい。

呼び出し元と呼び出し先

  • **呼び出し元**: modules/proxy/balancers/mod_lbmethod_bybusyness.c:50 にて、ap_proxy_balancer_get_best_worker_fn の引数として渡されています。
  • **呼び出し先**: 直接的な関数呼び出しはありませんが、current->s および prev_best->s を介して共有メモリ上のワーカー状態を参照・更新します。

関連データ構造

  • proxy_worker: ロードバランサーのバックエンドサーバーを表す構造体。
  • proxy_worker_shared: proxy_worker 内の共有メモリ領域。busy 数や lbstatus を保持します。

失敗モードと境界条件

  • prev_best が NULL の場合、最初のワーカーを無条件で選択します。
  • busy 数がすべて同数の場合、lbstatus（重み）の大きいワーカーが優先されます。

保守担当者向けメモ

  • 本関数は ap_proxy_balancer_get_best_worker_fn による反復処理の中で呼び出されます。
  • lbstatus の更新がこの比較関数内で行われるため、呼び出し順序や反復回数に依存した副作用がある点に注意してください。
