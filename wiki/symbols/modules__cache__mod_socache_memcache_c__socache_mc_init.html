socache_mc_init

定義

  • 種別: function
  • 定義箇所: modules/cache/mod_socache_memcache.c:81

目的

apr_memcache を用いて Memcached サーバー群との接続を確立し、ap_socache_instance_t コンテキストを初期化する。

シグネチャまたは形状

apr_status_t socache_mc_init(ap_socache_instance_t *ctx,
                             const char *namespace,
                             const struct ap_socache_hints *hints,
                             server_rec *s, apr_pool_t *p)

入力と出力

  • **入力**:
  • ctx: 初期化対象の socache インスタンス（servers メンバにカンマ区切りのサーバーリストを含む）。
  • namespace: キャッシュの識別に使用する名前空間文字列。
  • hints: キャッシュのヒント情報（本関数内では未使用）。
  • s: サーバー設定を取得するための server_rec。
  • p: メモリ割り当てに使用するプール。
  • **出力**:
  • apr_status_t: 成功時は APR_SUCCESS、失敗時はエラーコードを返す。

前提条件と不変条件

  • ctx->servers にはカンマ区切りの Memcached サーバーリスト（host:port）が格納されていること。
  • ctx->taglen は 16 バイト以下であること（AP_DEBUG_ASSERT による検証）。

アルゴリズムまたは振る舞い

1. ap_mpm_query を呼び出し、現在の MPM のスレッド制限値を取得する。 2. ctx->servers を解析し、サーバーの総数をカウントする。 3. apr_memcache_create を呼び出し、Memcached オブジェクトを生成する。 4. サーバーリストを再度走査し、各サーバーに対して以下を実行する:

  • apr_parse_addr_port でホストとポートを分離（ポート未指定時は MC_DEFAULT_SERVER_PORT を使用）。
  • apr_memcache_server_create でサーバー接続を生成（スレッド制限値や sconf->ttl を適用）。
  • apr_memcache_add_server で Memcached オブジェクトにサーバーを追加。

    • 5. namespace を基に ctx->tag を生成し、その長さを ctx->taglen に設定する。

呼び出し元と呼び出し先

  • **呼び出し元**: modules/cache/mod_socache_memcache.c:356（socache_mc プロバイダー構造体経由での呼び出しが想定される）。
  • **呼び出し先**: apr_memcache_create, apr_memcache_server_create, apr_memcache_add_server, ap_mpm_query, apr_parse_addr_port 等の APR/Apache API。

関連データ構造

  • ap_socache_instance_t: キャッシュインスタンスのコンテキスト。
  • socache_mc_svr_cfg: モジュールのサーバー設定（TTL 等）。
  • apr_memcache_t: apr_memcache ライブラリのメインハンドル。

失敗モードと境界条件

  • apr_memcache_create 失敗時: APR_CRIT レベルでログ出力しエラーを返す。
  • apr_parse_addr_port 失敗時: ホスト名が不正な場合等にエラーを返す。
  • apr_memcache_server_create または add_server 失敗時: 接続確立失敗としてエラーを返す。
  • ctx->taglen > 16: デバッグビルド時にアサーション違反となる。

保守担当者向けメモ

  • 本関数は ctx->servers 文字列を破壊的に操作するため、apr_pstrdup で複製してから解析を行っている。
  • MC_DEFAULT_SERVER_PORT などの定数は mod_socache_memcache.c 内で定義されている。
  • socache_mc_destroy は現在 no-op であるため、リソース解放はプール p の破棄に依存している。
