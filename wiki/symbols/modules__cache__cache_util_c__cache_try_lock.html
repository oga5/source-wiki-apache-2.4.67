cache_try_lock

定義

  • 種別: function
  • 定義箇所: modules/cache/cache_util.c:279

目的

キャッシュ更新時に「サンダリング・ハード（Thundering Herd）」問題を回避するため、ファイルシステム上に排他ロックを作成する。

シグネチャまたは形状

apr_status_t cache_try_lock(cache_server_conf *conf, cache_request_rec *cache, request_rec *r)

入力と出力

  • **入力**:
  • conf: キャッシュサーバー設定（ロックパスや有効期限設定を含む）。
  • cache: リクエストに関連付けられたキャッシュコンテキスト。
  • r: 現在のリクエストレコード。
  • **出力**:
  • apr_status_t: 成功時は APR_SUCCESS。ロックが既に存在する場合や作成失敗時はエラーコードを返す。

前提条件と不変条件

  • conf が NULL、またはロック設定が無効な場合は即座に APR_SUCCESS を返す。
  • リクエストのプール (r->pool) に既にロックファイルが登録されている場合は、再作成せず成功とみなす。
  • ロックファイルは、キャッシュキーから生成されたハッシュ値に基づき、2階層のディレクトリ構造内に配置される。

アルゴリズムまたは振る舞い

1. **既存チェック**: r->pool 内のユーザーデータを確認し、既にロックが取得済みであれば終了。 2. **キー生成**: cache->key が未設定の場合、既存のハンドルから取得するか、cache_generate_key を呼び出して生成。 3. **ディレクトリ作成**: conf->lockpath 配下に、ハッシュ値に基づくディレクトリを再帰的に作成。 4. **期限切れチェック**: 既存のロックファイルが存在する場合、最終更新時刻 (mtime) を確認。conf->lockmaxage を超えていれば古いロックを削除。 5. **ロック取得**: APR_EXCL および APR_CREATE フラグを指定してファイルを作成。成功時は r->pool にファイルハンドルを保存し、APR_DELONCLOSE により終了時に自動削除されるよう設定。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • modules/cache/cache_util.c:771
  • modules/cache/mod_cache.c:191
  • modules/cache/mod_cache.c:490
  • **呼び出し先**:

  • ap_cache_generate_name

  • apr_stat
  • apr_file_remove
  • apr_file_open
  • ap_log_rerror

関連データ構造

  • cache_server_conf: ロック設定（lockpath, lockmaxage）を保持。
  • cache_request_rec: キャッシュキーおよびハンドル情報を保持。
  • request_rec: リクエストごとのメモリプール (r->pool) を管理。

失敗モードと境界条件

  • **ディレクトリ作成失敗**: apr_dir_make_recursive が失敗した場合、エラーログを出力しそのステータスを返す。
  • **ロック競合**: 他のプロセスが既にロックを作成している場合、apr_file_open が APR_EEXIST 等のステータスを返し、ロック取得に失敗する。
  • **古いロック**: lockmaxage を超えたロックファイルは自動的に削除されるため、デッドロックを防止する設計となっている。

保守担当者向けメモ

  • 本関数は r->pool を使用してメモリ管理を行っているため、リクエスト終了時にロックファイルハンドルやパス情報が適切にクリーンアップされる。
  • ロックファイル名は ap_cache_generate_name によって生成されるため、ハッシュ衝突の可能性については当該関数の実装に依存する。
