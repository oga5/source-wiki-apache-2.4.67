delete_entry

定義

  • 種別: function
  • 定義箇所: support/htcacheclean.c:359

目的

キャッシュディレクトリ内の特定のキャッシュエントリ（ヘッダーファイルおよびデータファイル）を削除し、関連する空のディレクトリ構造を整理する。

シグネチャまたは形状

static void delete_entry(char *path, char *basename, apr_off_t *nodes, apr_pool_t *pool)

入力と出力

  • path: キャッシュエントリが存在するディレクトリパス。
  • basename: 削除対象エントリのベースファイル名。
  • nodes: 削除されたファイル数（inode数）を減算するためのポインタ。
  • pool: メモリ管理用プール。

前提条件と不変条件

  • path と basename は有効なキャッシュファイルパスを構成する必要がある。
  • nodes は有効な apr_off_t 型のポインタであること。
  • dryrun グローバル変数が真の場合、ファイル削除は行われず、apr_stat による存在確認のみが行われる。

アルゴリズムまたは振る舞い

1. 一時的なサブプール p を作成し、メモリ消費を抑制する。 2. CACHE_HEADER_SUFFIX を付加したヘッダーファイルを特定し、dryrun モードに応じて削除または存在確認を行う。成功時は nodes をデクリメントする。 3. 同様に CACHE_DATA_SUFFIX を付加したデータファイルを処理する。 4. サブプール p を破棄する。 5. benice フラグが有効な場合、delcount を更新し、一定数削除するごとに apr_sleep を呼び出してシステム負荷を抑制する。 6. 最後に delete_parent() を呼び出し、空になった親ディレクトリの削除を試みる。

呼び出し元と呼び出し先

  • **呼び出し元**: support/htcacheclean.c 内のキャッシュクリーンアップロジック（purge() 等）から複数箇所で呼び出される。
  • **呼び出し先**: apr_pool_create, apr_pstrcat, apr_stat, apr_file_remove, apr_pool_destroy, apr_sleep, delete_parent。

関連データ構造

  • apr_pool_t: メモリ管理に使用。
  • apr_off_t: inode カウントの追跡に使用。

失敗モードと境界条件

  • apr_file_remove が失敗した場合、そのファイルに対する nodes のデクリメントは行われない。
  • dryrun モードでは、ファイルが実際に削除されることはないが、apr_stat が成功すれば nodes が減算される。
  • benice が有効な場合、削除処理の合間に apr_sleep による待機が発生する。

保守担当者向けメモ

  • 本関数は delete_parent を末尾で呼び出しており、ファイル削除後のディレクトリ階層のクリーンアップまでを責務としている。
  • メモリリークを防ぐため、ファイルパスの構築には必ず一時的なサブプール p を使用すること。
