cache_canonicalise_key

定義

  • 種別: function
  • 定義箇所: modules/cache/cache_storage.c:428

目的

リクエストの URI コンポーネント（ホスト名、スキーム、ポート、パス、クエリ文字列）を正規化し、キャッシュの一貫した識別キーを生成する。

シグネチャまたは形状

static apr_status_t cache_canonicalise_key(request_rec *r, apr_pool_t* p,
                                           const char *path, const char *query,
                                           apr_uri_t *parsed_uri,
                                           const char **key)

入力と出力

  • **入力**:
  • r: 現在のリクエスト構造体。
  • p: メモリ割り当てに使用するプール。
  • path, query: 正規化対象のパスとクエリ文字列。
  • parsed_uri: 解析済みの URI 構造体。
  • **出力**:
  • key: 生成された正規化済みキー文字列（出力引数）。
  • 戻り値: APR_SUCCESS。

前提条件と不変条件

  • *key が既に設定されている場合、関数は即座に APR_SUCCESS を返し、再計算を行わない。
  • リクエストのライフサイクルを通じて一貫したキーを生成するため、リクエスト設定（request_config）にキーを保持することが推奨される。

アルゴリズムまたは振る舞い

1. **ホスト名の決定**: プロキシリクエストか否かに応じて、base_uri または ap_get_server_name を使用して正規ホスト名を決定する。 2. **スキームの決定**: parsed_uri または ap_http_scheme を使用し、小文字に変換して保持する。 3. **ポートの決定**: プロキシ設定やスキームのデフォルトポートに基づき、適切なポート文字列を生成する。 4. **セッション ID の除外**: CacheIgnoreSessionID 設定に基づき、パスやクエリ文字列から指定されたセッション識別子を削除する。 5. **キーの構築**: scheme://hostname:port/path?query の形式で文字列を連結し、*key に格納する。 6. **ログ出力**: 生成されたキーをデバッグログに出力する。

呼び出し元と呼び出し先

  • **呼び出し元**: modules/cache/cache_storage.c 内の関数（cache_generate_key_default 等）から呼び出される。
  • **呼び出し先**: ap_get_module_config, ap_get_server_name, ap_http_scheme, apr_uri_port_of_scheme, ap_strrchr_c, ap_strstr_c, ap_log_rerror 等の Apache/APR API。

関連データ構造

  • request_rec: リクエストコンテキスト。
  • cache_server_conf: キャッシュモジュールの設定（CacheIgnoreQueryString や CacheIgnoreSessionID を保持）。
  • apr_uri_t: URI の各要素を保持する構造体。

失敗モードと境界条件

  • ホスト名やスキームが特定できない場合、_default_ や空文字列を使用してキー生成を継続する。
  • クエリ文字列が空の場合、? 以降の処理を適切にスキップする。
  • セッション ID の削除処理において、クエリ文字列内のサブキーマッチングを避けるため、区切り文字（&, =）を考慮した厳密な検索を行う。

保守担当者向けメモ

  • 本関数はリバースプロキシとフォワードプロキシの両方で一貫したキーを生成するよう設計されている。r->proxyreq の状態によってパスが分岐するため、修正時は両方のケースへの影響を考慮すること。
  • r->parsed_uri はリクエスト処理中に変更される可能性があるため、初回生成時にキーをキャッシュしておくことが重要である。
