alloc_listener

定義

  • 種別: function
  • 定義箇所: server/listen.c:439

目的

設定ファイルで指定された Listen ディレクティブに基づき、新しいリスナー構造体（ap_listen_rec）を生成し、グローバルなリスナーリスト（ap_listeners）に追加する。

シグネチャまたは形状

static const char *alloc_listener(process_rec *process, const char *addr,
                                  apr_port_t port, const char* proto,
                                  const char *scope_id, void *slave,
                                  apr_pool_t *temp_pool)

入力と出力

  • **入力**:
  • process: プロセス管理用構造体（メモリプールを含む）。
  • addr: バインド対象の IP アドレスまたはホスト名。
  • port: バインド対象のポート番号。
  • proto: プロトコル名（例: "http"）。
  • scope_id: IPv6 スコープ ID。
  • slave: スレーブリスナーのポインタ。
  • temp_pool: 一時的なメモリプール。
  • **出力**:
  • 成功時は NULL を返す。
  • 失敗時はエラーメッセージ（const char *）を返す。

前提条件と不変条件

  • ap_listeners はサーバーのライフサイクルを通じて生存する連結リストである。
  • 同一の IP:ポート に対して複数のリスナーを定義することはできない。
  • apr_sockaddr_info_get により取得されたアドレス順序を維持する必要がある。

アルゴリズムまたは振る舞い

1. find_listeners を呼び出し、指定された IP:ポート が既に存在しないか確認する。 2. apr_sockaddr_info_get を使用して、指定されたアドレスとポートからソケットアドレス構造体を取得する。 3. 取得したアドレスリストをループし、各アドレスに対して以下の処理を行う:

  • ap_listen_rec 構造体を process->pool から割り当てる。
  • apr_socket_create でソケットを作成する。
  • IPv6 環境でアドレス解決に失敗した場合、次のアドレスへスキップする。
  • apr_sockaddr_zone_set が利用可能な場合、スコープ ID を適用する。
  • 新しいリスナーを ap_listeners リストの末尾に追加する。

    • 4. 成功した場合は NULL を返す。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • server/listen.c:1057 (設定解析時)
  • **呼び出し先**:
  • find_listeners
  • apr_sockaddr_info_get
  • ap_log_perror
  • apr_palloc
  • apr_pstrdup
  • apr_socket_create
  • apr_sockaddr_zone_set (条件付き)

関連データ構造

  • ap_listen_rec: リスナー情報を保持する構造体。
  • process_rec: プロセス全体の情報を保持する構造体。
  • apr_sockaddr_t: APR のソケットアドレス構造体。

失敗モードと境界条件

  • **重複定義**: 同一の IP:ポート が既に存在する場合、「Cannot define multiple Listeners...」を返す。
  • **スレーブ競合**: 既存のリスナーとスレーブ設定が競合する場合、「Cannot define a slave...」を返す。
  • **アドレス解決失敗**: apr_sockaddr_info_get が失敗した場合、「Listen setup failed」を返す。
  • **ソケット作成失敗**: apr_socket_create が失敗した場合、「Listen setup failed」を返す。
  • **スコープ設定失敗**: apr_sockaddr_zone_set が失敗した場合、「Listen step failed」を返す。

保守担当者向けメモ

  • 本関数は process->pool を使用してメモリを割り当てるため、サーバー再起動時までリスナー情報が保持される。
  • apr_socket_create の呼び出しにおいて、プラットフォーム固有の挙動（特に IPv6）が考慮されている。
  • ap_listeners リストの末尾を探索する処理が含まれており、リストが長くなる場合は計算量に注意が必要。
