ctx_servers_t

定義

  • 種別: struct
  • 定義箇所: modules/proxy/balancers/mod_lbmethod_heartbeat.c:59

目的

ハートビート情報の収集処理において、特定の時点における全バックエンドサーバーの負荷状態を一時的に保持・管理するためのコンテキスト構造体です。

シグネチャまたは形状

typedef struct ctx_servers {
    apr_time_t now;
    apr_hash_t *servers;
} ctx_servers_t;

入力と出力

  • now: 収集処理を実行した時刻（apr_time_t）。
  • servers: サーバー識別子をキーとし、hb_server_t 構造体を値として格納するハッシュテーブル。

前提条件と不変条件

  • servers ハッシュテーブルは、使用前に apr_hash_make 等で初期化されている必要があります。
  • 保持される各サーバー情報は、ハートビートの有効期限内（LBM_HEARTBEAT_MAX_LASTSEEN）であることを前提として処理されます。

アルゴリズムまたは振る舞い

  • ハートビート情報の読み込み処理（readslot_heartbeats 等）において、共有メモリやファイルから抽出したデータをこの構造体に集約します。
  • 収集したデータは、リクエストごとの転送先選定ロジック（find_best_hb）において、各バックエンドの稼働状況を評価するための参照元として利用されます。

呼び出し元と呼び出し先

  • 呼び出し元:
  • readslot_heartbeats (引数として受け取り、データを格納)
  • find_best_hb (ローカル変数として宣言し、read_heartbeats を通じて初期化)
  • 呼び出し先:
  • 本構造体自体はデータ保持用であり、直接的な関数呼び出しは行いません。

関連データ構造

  • hb_server_t: servers ハッシュテーブルの各要素として格納される、個別のサーバー負荷状態を保持する構造体。

失敗モードと境界条件

  • メモリ不足: apr_hash_t の初期化や要素の追加時にメモリ確保が失敗する可能性があります。
  • データ不整合: 読み込み元（共有メモリやファイル）のデータが破損している場合、ハッシュテーブルへの格納時に予期せぬ値が含まれる可能性があります。

保守担当者向けメモ

  • 本構造体はリクエスト処理のたびに生成・破棄される一時的なコンテキストです。メモリリークを防ぐため、必ず適切なプール（tpool）上で管理されていることを確認してください。
