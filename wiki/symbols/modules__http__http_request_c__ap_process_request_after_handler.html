ap_process_request_after_handler

定義

  • 種別: function
  • 定義箇所: modules/http/http_request.c:358

目的

HTTP リクエストのハンドラ実行後の後処理を実行します。EOR (End of Request) バケットの送信によるリクエスト終了処理、パイプライン化されたリクエストの確認、および接続状態の更新を行います。

シグネチャまたは形状

void ap_process_request_after_handler(request_rec *r)

入力と出力

  • **入力**: request_rec *r (処理対象のリクエスト構造体)
  • **出力**: なし（戻り値なし）

前提条件と不変条件

  • r は有効なリクエスト構造体である必要があります。
  • EOR バケットがフィルタチェーンに投入された後は、r および r->pool への参照は安全ではありません（EOR バケットのクリーンアップ関数によってプールが解放される可能性があるため）。

アルゴリズムまたは振る舞い

1. **EOR バケットの生成と送信**: ap_bucket_eor_create を使用して EOR バケットを作成し、出力フィルタチェーン (c->output_filters) に流します。これにより、リクエストのログ記録やプール解放がトリガーされます。 2. **クリーンアップ**: 使用したブリゲード (bb) を apr_brigade_cleanup で初期化します。 3. **パイプライン確認**: ap_check_pipeline を呼び出し、接続上の空行を消費してパイプライン化されたリクエストの有無を確認します。 4. **接続状態の更新**: 接続が中断されている場合は CONN_STATE_LINGER、それ以外は CONN_STATE_WRITE_COMPLETION に状態を遷移させます。 5. **統計更新**: ap_extended_status が有効な場合、ap_time_process_request を呼び出してリクエスト処理時間を記録します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • modules/http/http_request.c:477
  • modules/test/mod_dialup.c:119
  • **呼び出し先**:
  • ap_bucket_eor_create
  • ap_pass_brigade
  • apr_brigade_cleanup
  • ap_check_pipeline
  • ap_time_process_request

関連データ構造

  • request_rec: リクエストのコンテキストを保持。
  • conn_rec: 接続状態を保持。
  • apr_bucket_brigade: フィルタチェーンを流れるデータバケットのコンテナ。

失敗モードと境界条件

  • c->aborted が真の場合、接続は CONN_STATE_LINGER 状態へ移行します。
  • ap_check_pipeline の結果に基づき、c->data_in_input_filters が更新されます。

保守担当者向けメモ

  • 本関数実行後の r へのアクセスは、メモリ破壊を引き起こす可能性があるため厳禁です。
  • ap_process_request_after_handler は mod_serf 等の外部モジュールからも利用される公開インターフェースです。
