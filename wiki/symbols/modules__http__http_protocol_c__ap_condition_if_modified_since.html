ap_condition_if_modified_since

定義

  • 種別: function
  • 定義箇所: modules/http/http_protocol.c:438

目的

HTTP リクエストに含まれる If-Modified-Since ヘッダーを評価し、リソースが条件を満たしているかを判定する。

シグネチャまたは形状

ap_condition_e ap_condition_if_modified_since(request_rec *r, apr_table_t *headers)

入力と出力

  • **入力**:
  • r: 現在のリクエスト構造体 (request_rec)。headers_in から If-Modified-Since を取得し、mtime や request_time を参照する。
  • headers: 比較対象のヘッダーテーブル。Last-Modified および Date を取得するために使用。
  • **出力**:
  • ap_condition_e 型の列挙値:
  • AP_CONDITION_NONE: ヘッダーが存在しない場合。
  • AP_CONDITION_NOMATCH: 条件を満たさない場合。
  • AP_CONDITION_WEAK: 条件を満たすが、弱一致（Range リクエスト等）の場合。
  • AP_CONDITION_STRONG: 条件を満たす強一致の場合。

前提条件と不変条件

  • 時刻比較はすべて秒単位で行われる（HTTP 仕様の最小時間分解能に基づく）。
  • Last-Modified が解析不能な場合は、r->mtime または現在時刻が使用される。
  • Date が解析不能な場合は、r->request_time が使用される。

アルゴリズムまたは振る舞い

1. r->headers_in から If-Modified-Since ヘッダーを取得する。存在しない場合は AP_CONDITION_NONE を返す。 2. headers から Last-Modified (mtime) と Date (reqtime) を解析し、秒単位の数値に変換する。 3. If-Modified-Since の値 (ims) と比較する。 4. ims >= mtime かつ ims <= reqtime の場合:

  • reqtime < mtime + 60 かつ Range ヘッダーが存在する場合は AP_CONDITION_NOMATCH を返す。
  • reqtime < mtime + 60 で Range ヘッダーがない場合は AP_CONDITION_WEAK を返す。
  • それ以外は AP_CONDITION_STRONG を返す。

    • 5. 条件を満たさない場合は AP_CONDITION_NOMATCH を返す。

呼び出し元と呼び出し先

関連データ構造

  • request_rec: リクエスト情報およびヘッダー保持に使用。
  • apr_table_t: ヘッダーの格納および検索に使用。
  • ap_condition_e: 戻り値の列挙型。

失敗モードと境界条件

  • If-Modified-Since ヘッダーがリクエストに含まれない場合、即座に AP_CONDITION_NONE を返す。
  • Range ヘッダーが存在する場合、弱一致 (AP_CONDITION_WEAK) は許可されず AP_CONDITION_NOMATCH となる。

保守担当者向けメモ

  • 本関数は ap_meets_conditions による条件付きリクエスト評価フローの一部として呼び出される。
  • 時刻解析には apr_date_parse_http を使用しているため、RFC 準拠のフォーマット以外は APR_DATE_BAD となり、フォールバック処理が実行される点に注意すること。
