hc_template_t

定義

  • 種別: struct
  • 定義箇所: modules/proxy/mod_proxy_hcheck.c:39

目的

バックエンドサーバーに対するアクティブなヘルスチェックの動作パラメータ（間隔、成功/失敗の閾値、通信メソッド等）を保持するためのテンプレート構造体です。

シグネチャまたは形状

typedef struct {
    char *name;
    hcmethod_t method;
    int passes;
    int fails;
    apr_interval_time_t interval;
    char *hurl;
    char *hcexpr;
} hc_template_t;

入力と出力

本構造体は設定データとして機能します。

  • **入力**: ProxyHCTemplate ディレクティブ等を通じて設定された値が格納されます。
  • **出力**: ヘルスチェック実行時に、監視ロジックが本構造体の各フィールドを参照し、チェックの実行間隔や判定基準を決定します。

前提条件と不変条件

  • mod_proxy_hcheck モジュールの設定コンテキスト（sctx_t）内の templates 配列に格納されて管理されます。
  • 実行時には、設定された interval に基づいて mod_watchdog が定期的にチェックをスケジュールします。

アルゴリズムまたは振る舞い

  • 本構造体自体はデータ保持用であり、ロジックは持ちません。
  • 監視プロセスにおいて、このテンプレートを基に baton_t が初期化され、実際のヘルスチェック（TCP, CPING, HTTP）が実行されます。
  • 成功回数（passes）および失敗回数（fails）の閾値に達した際、ワーカーのステータス更新判定に使用されます。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • modules/proxy/mod_proxy_hcheck.c 内の設定読み込み処理およびワーカー初期化処理から参照されます。
  • **呼び出し先**:
  • 構造体自体はデータ保持用のため、直接的な関数呼び出し先はありません。

関連データ構造

  • sctx_t: 本構造体を保持するサーバー単位のモジュール設定。
  • baton_t: ヘルスチェック実行時に本テンプレートの情報を引き継ぐ実行コンテキスト。
  • proxy_worker: ヘルスチェックの対象となるバックエンドワーカー。

失敗モードと境界条件

  • 設定値が不正な場合（例: interval が 0 以下など）、ヘルスチェックのスケジュールや実行に影響を与える可能性があります。
  • 構造体内の文字列ポインタ（name, hurl, hcexpr）は、設定の生存期間に依存するため、設定破棄後のアクセスは未定義動作となります。

保守担当者向けメモ

  • 本構造体は apr_array_make で作成された配列内に格納されます。配列の拡張や要素へのアクセス時には、apr_array_push 等の APR 配列操作関数を介して安全に行う必要があります。
  • hcexpr は ap_expr で評価されるため、設定変更時の構文チェックについては ap_expr_parse 等の関連処理を確認してください。
