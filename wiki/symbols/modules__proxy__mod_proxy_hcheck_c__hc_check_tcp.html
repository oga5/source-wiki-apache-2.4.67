hc_check_tcp

定義

  • 種別: function
  • 定義箇所: modules/proxy/mod_proxy_hcheck.c:701

目的

バックエンドサーバーに対してTCPレベルの接続確認を行い、ヘルスチェックを実行する。

シグネチャまたは形状

static apr_status_t hc_check_tcp(baton_t *baton)

入力と出力

  • **入力**: baton_t *baton（ヘルスチェック対象のワーカー情報およびコンテキストを保持する構造体）。
  • **出力**: apr_status_t（操作の結果を示すステータスコード）。

前提条件と不変条件

  • baton が有効な sctx_t（サーバーコンテキスト）および proxy_worker を保持していること。
  • hc_get_backend が正常にバックエンド接続オブジェクトを初期化できること。

アルゴリズムまたは振る舞い

1. hc_get_backend を呼び出し、対象ワーカーのバックエンド接続オブジェクトを取得する。 2. 接続取得に成功した場合、ap_proxy_connect_backend を呼び出し、バックエンドへのTCP接続を試行する。 3. ap_proxy_connect_backend は内部で ap_proxy_is_socket_connected() を実行し、接続状態を検証する。 4. 最後に backend_cleanup を呼び出し、接続の解放および最終的なステータスコードの返却を行う。

呼び出し元と呼び出し先

  • **呼び出し元**: modules/proxy/mod_proxy_hcheck.c:935（直接的な呼び出しを確認）。
  • **呼び出し先**:
  • hc_get_backend
  • ap_proxy_connect_backend
  • backend_cleanup

関連データ構造

  • baton_t: ヘルスチェック実行に必要な情報を集約した構造体。
  • proxy_worker: チェック対象のバックエンドワーカー。
  • proxy_conn_rec: バックエンドへの接続状態を管理する構造体。
  • sctx_t: サーバー単位のモジュール設定コンテキスト。

失敗モードと境界条件

  • hc_get_backend が OK 以外のステータスを返した場合、接続処理はスキップされ、エラーが伝播される。
  • ap_proxy_connect_backend が失敗した場合、そのステータスが backend_cleanup を通じて返される。

保守担当者向けメモ

  • 本関数はTCP接続の生存確認のみを行う。HTTPレベルの応答内容やAJPのCPING/CPONG確認は別の関数（hc_check_http, hc_check_cping）が担当する。
  • backend_cleanup が戻り値を決定するため、接続の成否に関わらず適切なクリーンアップが保証される設計となっている。
