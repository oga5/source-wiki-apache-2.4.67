force_recovery

定義

  • 種別: function
  • 定義箇所: modules/proxy/mod_proxy_balancer.c:426

目的

ロードバランサ内の全ワーカーがエラー状態にある場合に、設定に応じて強制的に復旧処理を行い、ワーカーを再利用可能な状態へ戻す。

シグネチャまたは形状

static void force_recovery(proxy_balancer *balancer, server_rec *s)

入力と出力

  • **入力**:
  • balancer: 対象となる proxy_balancer 構造体へのポインタ。
  • s: 現在のサーバー設定を表す server_rec 構造体へのポインタ。
  • **出力**: なし（副作用として共有メモリ上のワーカー状態を更新）。

前提条件と不変条件

  • balancer->workers が有効なワーカーリストを保持していること。
  • balancer->s->forcerecovery が真の場合、すべてのワーカーに対して強制復旧が試行される。

アルゴリズムまたは振る舞い

1. まず、エラー状態（PROXY_WORKER_IN_ERROR）ではないワーカーが存在するかを確認する。 2. エラー状態のワーカーに対しては、ap_proxy_retry_worker_fn を呼び出し、個別の復旧を試みる。 3. 復旧を試みた結果、いずれかのワーカーが正常状態に戻れば処理を終了する。 4. 全ワーカーが依然としてエラー状態であり、かつ balancer->s->forcerecovery が有効な場合：

  • すべてのワーカーの retries カウンタをインクリメントする。
  • すべてのワーカーから PROXY_WORKER_IN_ERROR フラグを除去し、強制的に復旧させる。
  • 復旧処理の実行をデバッグログに出力する。

呼び出し元と呼び出し先

  • **呼び出し元**: modules/proxy/mod_proxy_balancer.c:507（proxy_balancer_pre_request 内から呼び出される）。
  • **呼び出し先**:
  • ap_proxy_retry_worker_fn: ワーカーの個別復旧を試みる関数。
  • ap_log_error: 復旧実行時のログ出力。

関連データ構造

  • proxy_balancer: バランサの設定とワーカーリストを保持。
  • proxy_worker: 個々のバックエンドワーカーの状態（status, retries 等）を保持。
  • server_rec: サーバーのコンテキスト情報。

失敗モードと境界条件

  • ワーカーリストが空の場合、ループは実行されず何も行われない。
  • forcerecovery が無効な場合、すべてのワーカーがエラー状態であっても強制復旧は行われない。

保守担当者向けメモ

  • 本関数は proxy_balancer_pre_request の一部として、リクエスト処理の過程で呼び出される。
  • PROXY_WORKER_IN_ERROR フラグの操作は共有メモリ（slotmem）を介して行われるため、マルチプロセス環境での整合性に注意すること。
