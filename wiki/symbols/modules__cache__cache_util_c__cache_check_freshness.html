cache_check_freshness

定義

  • 種別: function
  • 定義箇所: modules/cache/cache_util.c:533

目的

キャッシュされたエンティティが現在のリクエストに対して「鮮度（fresh）」を保っているかを判定し、鮮度が切れている場合に「サンダリング・ハード（Thundering Herd）」を防ぐための排他ロック制御を行う。

シグネチャまたは形状

int cache_check_freshness(cache_handle_t *h, cache_request_rec *cache, request_rec *r)

入力と出力

  • **入力**:
  • h: キャッシュエントリへのハンドル。
  • cache: リクエストに関連付けられたキャッシュコンテキスト。
  • r: 現在の HTTP リクエスト構造体。
  • **出力**:
  • 1: キャッシュが鮮度を保っている（またはロック競合により鮮度があると見なす）。
  • 0: キャッシュが期限切れ（stale）であり、再検証が必要。

前提条件と不変条件

  • r->pool を使用してメモリ管理を行う。
  • Cache-Control や Pragma ヘッダーの解析結果（cache->control_in）が利用可能であること。
  • h->cache_obj->info にキャッシュのメタデータが格納されていること。

アルゴリズムまたは振る舞い

1. **無効化判定**: リクエストまたはキャッシュされたレスポンスに no-cache が含まれる場合、即座に 0（stale）を返す。 2. **鮮度計算**: ap_cache_current_age を用いて経過時間を算出。max-age、s-maxage、Expires ヘッダーを RFC 2616 に基づいて比較し、鮮度を判定する。 3. **警告付与**: 鮮度が維持されている場合でも、max-stale の使用やヒューリスティックな期限切れ判定に基づき、必要に応じて Warning ヘッダー（110 または 113）をレスポンスに追加する。 4. **ロック制御**: 鮮度が切れている場合、cache_try_lock を試行する。

  • ロック取得成功: 0 を返し、バックエンドへの再検証を許可する。
  • ロック取得失敗（EEXIST）: 他のプロセスが再検証中と見なし、警告を付与した上で 1 を返し、stale なデータを一時的に提供する。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • modules/cache/cache_storage.c:336
  • **呼び出し先**:
  • ap_cache_control
  • ap_cache_current_age
  • cache_try_lock

  • ap_log_rerror

関連データ構造

  • cache_handle_t: キャッシュオブジェクトへのアクセス用。
  • cache_request_rec: リクエストごとのキャッシュ制御情報。
  • cache_server_conf: サーバー設定（ignorecachecontrol 等のフラグを含む）。

失敗モードと境界条件

  • **ロック失敗**: cache_try_lock が予期せぬエラーを返した場合、安全側に倒して 0（stale）を返す。
  • **設定による無視**: ignorecachecontrol が有効な場合、no-cache ヘッダーを無視してキャッシュを返す場合がある。
  • **Age ヘッダーの欠落**: Age ヘッダーがない場合は 0 として計算される。

保守担当者向けメモ

  • 本関数は RFC 2616 の複雑な鮮度計算ロジックを内包している。修正時は s-maxage と max-age の優先順位、および max-stale と must-revalidate の相互作用に注意すること。
  • ロック機構はファイルシステムベースであるため、高負荷環境での I/O 競合がパフォーマンスに影響を与える可能性がある。
