ap_process_request

定義

  • 種別: function
  • 定義箇所: modules/http/http_request.c:480

目的

HTTP リクエスト処理の標準的なエントリポイントとして、リクエストの実行と接続状態の最終的なフラッシュ処理を統括します。

シグネチャまたは形状

void ap_process_request(request_rec *r)

入力と出力

  • **入力**: request_rec *r （処理対象のリクエスト構造体）
  • **出力**: なし（戻り値 void）

前提条件と不変条件

  • r は有効なリクエスト構造体である必要があります。
  • 呼び出し時点で、リクエストの解析およびハンドラの実行準備が整っていることが想定されます。

アルゴリズムまたは振る舞い

1. **リクエスト実行**: ap_process_async_request(r) を呼び出し、リクエストの主要な処理（フック実行、ハンドラ処理等）を実行します。 2. **パイプラインチェック**: 接続に未処理のデータ（c->data_in_input_filters）がない場合、出力フィルタチェーンに対してフラッシュバケットを送信し、バッファを強制的にフラッシュします。 3. **エラーログ**: フラッシュ処理中に APR_STATUS_IS_TIMEUP が発生した場合は、ネットワーク接続の切断や DoS 攻撃の可能性を考慮し、APLOG_INFO レベルでログを出力します。 4. **統計更新**: ap_extended_status が有効な場合、ap_time_process_request を呼び出し、リクエスト処理の終了時刻を記録します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • modules/http/http_core.c
  • modules/http2/h2_c2.c
  • **呼び出し先**:
  • ap_process_async_request
  • ap_pass_brigade
  • ap_log_cerror
  • ap_time_process_request
  • apr_brigade_cleanup

関連データ構造

  • request_rec: リクエストのコンテキストを保持。
  • conn_rec: 接続状態およびフィルタチェーンを保持。
  • apr_bucket_brigade: 出力フィルタへデータを渡すためのバケット管理構造。

失敗モードと境界条件

  • ap_pass_brigade がタイムアウト（APR_TIMEUP）を返した場合、エラーログが出力されますが、処理は継続されます。
  • 接続が既に閉じている場合やフィルタチェーンの状態によっては、フラッシュ処理が影響を受ける可能性があります。

保守担当者向けメモ

  • 本関数はリクエスト処理の完了直後に呼び出されるため、ここでのフラッシュ処理はクライアントへのレスポンス完了を保証する重要な役割を担います。
  • ap_process_async_request の実行結果に関わらず、接続のクリーンアップと統計記録が確実に行われるよう設計されています。
