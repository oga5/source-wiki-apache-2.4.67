read_response

定義

  • 種別: function
  • 定義箇所: modules/ssl/ssl_util_ocsp.c:204

目的

OCSP レスポンダから送信された HTTP レスポンスをソケットから読み取り、OCSP 応答データ（OCSP_RESPONSE）をデコードして抽出する。

シグネチャまたは形状

OCSP_RESPONSE *read_response(apr_socket_t *sd, BIO *bio, conn_rec *c, apr_pool_t *p)

入力と出力

  • **入力**:
  • sd: 通信に使用する apr_socket_t。
  • bio: レスポンスボディを蓄積するためのメモリ BIO。
  • c: 現在の接続情報 conn_rec。
  • p: メモリ割り当てに使用する apr_pool_t。
  • **出力**:
  • 成功時: デコードされた OCSP_RESPONSE オブジェクトへのポインタ。
  • 失敗時: NULL。

前提条件と不変条件

  • apr_bucket_brigade を使用してソケットからデータを読み取る。
  • HTTP/1.0 プロトコルを前提としており、レスポンス終了後の接続切断を期待する。
  • MAX_HEADERS を超えるヘッダー行、または MAX_CONTENT を超えるボディサイズはエラーとして扱われる。

アルゴリズムまたは振る舞い

1. apr_bucket_brigade を作成し、ソケットを挿入する。 2. get_line を用いて HTTP ステータスラインを読み取り、2xx 系コードであることを検証する。 3. get_line を繰り返し、空行が現れるまでヘッダーを読み飛ばす。 4. 残りのボディデータを apr_bucket_read で読み取り、メモリ BIO に書き込む。 5. d2i_OCSP_RESPONSE_bio を呼び出し、BIO に蓄積されたデータから OCSP_RESPONSE を生成する。 6. 使用したブリゲードを破棄し、結果を返す。

呼び出し元と呼び出し先

  • **呼び出し元**: modules/ssl/ssl_util_ocsp.c 内の関数（ソースコード上の直接的な言及として modssl_dispatch_ocsp_request からの呼び出しが確認されている）。
  • **呼び出し先**: get_line, apr_brigade_create, apr_bucket_socket_create, apr_atoi64, ap_log_cerror, apr_bucket_read, BIO_write, apr_bucket_delete, apr_brigade_destroy, d2i_OCSP_RESPONSE_bio, ssl_log_ssl_error。

関連データ構造

  • OCSP_RESPONSE: OpenSSL の OCSP 応答構造体。
  • apr_bucket_brigade: HTTP レスポンスのストリーム処理に使用されるバケットの集合。
  • conn_rec: Apache の接続状態を保持する構造体。

失敗モードと境界条件

  • HTTP ステータスコードが 200〜299 以外の場合、エラーログを出力し NULL を返す。
  • ヘッダー行数が MAX_HEADERS を超過した場合、エラーログを出力し NULL を返す。
  • ボディサイズが MAX_CONTENT を超過した場合、エラーログを出力し NULL を返す。
  • d2i_OCSP_RESPONSE_bio によるデコードに失敗した場合、エラーログを出力し NULL を返す。

保守担当者向けメモ

  • 本関数内の HTTP 実装は簡易的なものであり、将来的な標準 HTTP クライアントライブラリへの移行が推奨されている。
  • MAX_HEADERS および MAX_CONTENT 定数は、不正なレスポンスによるメモリ枯渇を防ぐための重要な保護策である。
