mod_filter_cfg

定義

  • 種別: struct
  • 定義箇所: modules/filters/mod_filter.c:67

目的

mod_filter モジュールの設定情報を保持するための構造体です。登録されたフィルタの管理と、適用順序を定義するチェーンを格納します。

シグネチャまたは形状

typedef struct {
    apr_hash_t *live_filters;
    mod_filter_chain *chain;
} mod_filter_cfg;

入力と出力

  • **入力**: 設定ファイル（httpd.conf 等）のディレクティブ（FilterDeclare, FilterProvider, FilterChain）から読み込まれたデータ。
  • **出力**: モジュール内の各関数（filter_insert 等）が参照する設定コンテキスト。

前提条件と不変条件

  • live_filters は apr_hash_t 型であり、登録済みフィルタを管理します。
  • chain は mod_filter_chain 型であり、フィルタの適用順序を保持します。
  • 設定データは cmd->pool を使用してメモリ管理されます。

アルゴリズムまたは振る舞い

  • 本構造体は、mod_filter の設定フェーズで生成・初期化されます。
  • リクエスト処理時には、ap_get_module_config を通じてディレクトリごとの設定として取得され、フィルタの挿入や条件判定の基準として利用されます。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • modules/filters/mod_filter.c 内の各種設定ハンドラおよびリクエスト処理関数（filter_insert 等）。
  • 設定のマージ処理（BASE, ADD を用いたマージロジック）。
  • **呼び出し先**:
  • 本構造体自体はデータ保持用であり、直接的な関数呼び出しは行いません。

関連データ構造

  • mod_filter_chain: フィルタの適用順序を定義する構造体。
  • apr_hash_t: live_filters を格納するために使用される APR ハッシュテーブル。

失敗モードと境界条件

  • メモリ割り当て失敗時（apr_palloc 失敗時）の挙動は、Apache の標準的なメモリ管理ポリシーに従います。
  • 設定が未定義の場合、各関数内で cfg が NULL になる可能性があるため、呼び出し側での適切なチェックが必要です。

保守担当者向けメモ

  • 本構造体は mod_filter の設定管理の中核です。設定の追加やマージロジックを変更する際は、live_filters と chain の整合性が保たれていることを確認してください。
  • リクエストごとのコンテキスト（mod_filter_ctx）とは異なり、本構造体はディレクトリ単位の設定を保持する点に注意してください。
