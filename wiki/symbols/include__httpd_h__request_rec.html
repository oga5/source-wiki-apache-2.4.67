request_rec

定義

  • 種別: struct
  • 定義箇所: include/httpd.h:822

目的

Apache HTTP Server におけるリクエスト処理の中核となるデータ構造です。単一の HTTP リクエストのライフサイクル全体を通じて、リクエストの状態、ヘッダー、URI、メソッド、フィルタチェーンなどの情報を保持・共有するために使用されます。

シグネチャまたは形状

typedef struct request_rec request_rec;

入力と出力

  • **入力**: リクエスト処理の各フェーズ（翻訳、認証、ハンドラ実行等）において、モジュール関数に引数として渡されます。
  • **出力**: 各フェーズでモジュールが情報を蓄積・更新し、後続のフェーズやフィルタチェーンへ引き継がれます。

前提条件と不変条件

  • **メモリ管理**: 構造体内の多くの文字列やデータは、リクエスト専用のメモリプール（apr_pool_t）に割り当てられています。リクエスト終了時に自動的に解放されるため、手動での解放は不要です。
  • **アクセス規約**: 非常に巨大な構造体であるため、モジュール開発者は必要なフィールドのみにアクセスし、不必要な変更を避けることが推奨されます。

アルゴリズムまたは振る舞い

  • リクエスト処理の各フェーズで受け渡され、per_dir_config や notes を通じてモジュール間で情報を共有します。
  • フィルタチェーン（input_filters, output_filters）を管理し、ストリームデータ処理の基盤となります。

呼び出し元と呼び出し先

  • **呼び出し元（例）**:
  • mod_example_1.c および mod_example_2.c のハンドラ関数。
  • ap_expr_exec 等の式評価関数。
  • **呼び出し先**:
  • 本構造体はデータ構造であり、関数ではありません。ただし、ap_add_*_filter などのフィルタ操作関数や、ap_expr_* 系の関数がこの構造体を引数として受け取ります。

関連データ構造

  • conn_rec: クライアントとの接続状態を保持。
  • server_rec: 仮想ホストごとの設定と状態を保持。
  • apr_pool_t: メモリ管理に使用されるリクエストプール。

失敗モードと境界条件

  • **メモリ不足**: 構造体内の動的メモリ確保において OOM（Out of Memory）が発生した場合、ap_abort_on_oom が呼び出される可能性があります。
  • **API 変更**: include/ap_mmn.h に記録されている通り、過去に content_language の削除や used_path_info の追加など、構造体の定義はバージョンアップに伴い変更されることがあります。

保守担当者向けメモ

  • AP_DECLARE マクロを介して API を利用してください。
  • 構造体のレイアウトは非常に複雑であるため、特定のモジュール固有のロジックについては各モジュールのヘッダーファイルを参照してください。
