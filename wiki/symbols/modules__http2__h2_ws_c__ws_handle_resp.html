ws_handle_resp

定義

  • 種別: function
  • 定義箇所: modules/http2/h2_ws.c:197

目的

HTTP/2 上の WebSocket ハンドシェイクにおいて、バックエンドからのレスポンスを検査し、プロトコル変換の成否を判定してレスポンスバケットを書き換える。

シグネチャまたは形状

static void ws_handle_resp(conn_rec *c2, h2_conn_ctx_t *conn_ctx,
                           struct ws_filter_ctx *ws_ctx, apr_bucket *b)

入力と出力

  • c2: 処理対象の接続レコード。
  • conn_ctx: ストリームごとの接続コンテキスト。
  • ws_ctx: フィルタの実行状態を保持するコンテキスト。
  • b: 検査対象のレスポンスバケット。
  • 出力: 引数 b を介してレスポンスを置換し、ws_ctx および conn_ctx の状態を更新する。

前提条件と不変条件

  • ws_ctx->has_final_response が真の場合、本関数は何もせず終了する。
  • Sec-WebSocket-Accept ヘッダーの検証には、ws_ctx->ws_accept_base64 に格納された期待値を使用する。

アルゴリズムまたは振る舞い

1. **レスポンス検査**: レスポンスステータスを確認する。 2. **101 Switching Protocols の処理**:

  • Sec-WebSocket-Accept ヘッダーが期待値と一致する場合、make_valid_resp を呼び出し 200 OK に変換する。
  • 一致しない場合、make_invalid_resp を呼び出し 502 Bad Gateway に変換する。

    • 3. **エラー処理**: 200 未満のステータスは透過させ、200 以上 300 未満のステータスは 502 Bad Gateway に変換する。 4. **バケット置換**: b_override が生成された場合、元のバケットを削除し、新しいバケットを挿入する。 5. **状態更新**: 最終レスポンスが確定した場合、ws_ctx および conn_ctx の has_final_response フラグを立てる。

呼び出し元と呼び出し先

  • 呼び出し元: modules/http2/h2_ws.c:314 (出力フィルタ h2_c2_ws_filter_out 内)
  • 呼び出し先: make_valid_resp, make_invalid_resp, apr_bucket_delete, APR_BUCKET_INSERT_BEFORE, APR_BUCKET_INSERT_AFTER, apr_bucket_eos_create

関連データ構造

  • conn_rec: Apache HTTPD の接続構造体。
  • h2_conn_ctx_t: HTTP/2 ストリームごとの接続コンテキスト。
  • ws_filter_ctx: WebSocket フィルタの実行状態（Accept キー、完了フラグ等）を保持。

失敗モードと境界条件

  • Sec-WebSocket-Accept ヘッダーが欠落または不一致の場合、502 Bad Gateway を返し、override_body を設定して以降のボディを破棄する。
  • 101 以外の成功ステータス（2xx）が返された場合、WebSocket 接続として不適切とみなし 502 Bad Gateway を返す。

保守担当者向けメモ

  • 本関数は H2_C2_WS_OUT フィルタの一部として動作する。
  • AP_HAS_RESPONSE_BUCKETS の定義有無により、レスポンスヘッダーの取得方法が分岐する。
  • 変換後のレスポンスは c2->pool を使用してメモリ管理される。
