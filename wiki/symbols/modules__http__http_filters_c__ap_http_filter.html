ap_http_filter

定義

  • 種別: function
  • 定義箇所: modules/http/http_filters.c:340

目的

HTTP リクエストボディの入力フィルタリングを行う。チャンク転送のデコード、Content-Length による境界管理、およびリクエストボディのサイズ制限（LimitRequestBody）を強制し、ハンドラが安全にデータを読み取れるようにする。

シグネチャまたは形状

apr_status_t ap_http_filter(ap_filter_t *f, apr_bucket_brigade *b,
                            ap_input_mode_t mode, apr_read_type_e block,
                            apr_off_t readbytes)

入力と出力

  • **入力**:
  • f: フィルタコンテキスト。
  • b: データを格納するバケットブリゲード。
  • mode: 読み込みモード（AP_MODE_READBYTES または AP_MODE_GETLINE）。
  • block: ブロッキングの有無。
  • readbytes: 読み込み要求バイト数。
  • **出力**:
  • apr_status_t: 処理結果（APR_SUCCESS, APR_EOF, APR_EAGAIN, APR_INCOMPLETE 等）。
  • b: フィルタリング済みのデータバケットが格納される。

前提条件と不変条件

  • f->ctx（http_ctx_t）は、初回呼び出し時に初期化される。
  • LimitRequestBody 設定はプロキシリクエストには適用されない。
  • BODY_NONE 状態は、トレーラヘッダ処理を除き、開始状態としては無効。
  • ctx->remaining は、Content-Length が指定されている場合に残りの読み込み可能バイト数を保持する。

アルゴリズムまたは振る舞い

1. **初期化**: ctx が未設定の場合、Transfer-Encoding や Content-Length ヘッダに基づき状態（BODY_CHUNK, BODY_LENGTH 等）を決定する。 2. **100-Continue**: ブロッキング読み込み時、必要に応じて 100 Continue レスポンスを送信する。 3. **データ読み込み**:

  • BODY_CHUNK 系状態: parse_chunk_size を使用してチャンクヘッダを解析し、データを抽出する。終了時に read_chunked_trailers を呼び出す。
  • BODY_LENGTH / BODY_NONE 状態: ap_get_brigade を呼び出し、ctx->remaining を超えない範囲でデータを取得する。

    • 4. **制限チェック**: ctx->limit を超える読み込みが発生した場合、エラーを返す。 5. **終了処理**: ctx->eos_sent フラグを管理し、ストリーム終了時に EOS バケットを挿入する。

呼び出し元と呼び出し先

  • **呼び出し元**: フィルタチェーンを通じて呼び出される（modules/proxy/mod_proxy_http.c 等からの言及あり）。
  • **呼び出し先**: ap_get_brigade（下位フィルタ）、parse_chunk_size、read_chunked_trailers、bail_out_on_error、ap_send_interim_response。

関連データ構造

  • http_ctx_t: 入力フィルタの状態管理（state, remaining, limit, eos_sent 等を保持）。
  • core_server_config: サーバ設定（http_conformance, merge_trailers 等）。
  • apr_bucket_brigade: フィルタ間でデータを渡すためのバケットリスト。

失敗モードと境界条件

  • HTTP_REQUEST_ENTITY_TOO_LARGE: Content-Length が制限を超えている、または読み込み中に制限を超過した場合。
  • HTTP_BAD_REQUEST: 不正な Content-Length 値や、サポートされていない Transfer-Encoding が指定された場合。
  • APR_INCOMPLETE: 読み込み中に接続が切断された場合。
  • APR_EAGAIN: ノンブロッキング読み込みでデータが未到着の場合。

保守担当者向けメモ

  • ctx->state の遷移は RFC 7230 準拠のため非常に複雑である。修正時は http_ctx_t の状態定義を確認すること。
  • チャンク入力時の ctx->limit 計算には注意が必要（チャンク長とデータ長の扱いに曖昧さが残る可能性がある）。
  • メモリ管理は f->r->pool に依存しているため、バケット操作時のリークに注意すること。
