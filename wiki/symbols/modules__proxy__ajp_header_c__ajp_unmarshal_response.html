ajp_unmarshal_response

定義

  • 種別: function
  • 定義箇所: modules/proxy/ajp_header.c:540

目的

バックエンドサーバーから受信した AJP SEND_HEADERS パケットを解析し、その内容を Apache の request_rec 構造体（HTTP ステータス、レスポンスヘッダー、コンテンツタイプ等）へ反映させる。

シグネチャまたは形状

static apr_status_t ajp_unmarshal_response(ajp_msg_t *msg,
                                           request_rec *r,
                                           proxy_dir_conf *dconf)

入力と出力

  • **入力**:
  • msg: 解析対象の AJP メッセージバッファ。
  • r: 処理対象の Apache リクエスト構造体。
  • dconf: プロキシのディレクトリ設定（リバースマップ処理に使用）。
  • **出力**:
  • apr_status_t: 成功時は APR_SUCCESS。解析失敗時はエラーコードを返す。
  • 副作用として r->status、r->status_line、r->headers_out が更新される。

前提条件と不変条件

  • msg は有効な AJP メッセージバッファであること。
  • r->pool を使用してメモリが割り当てられるため、リクエストのライフサイクルに依存する。
  • EBCDIC 環境下では、文字列データに対して適切な文字コード変換が行われる。

アルゴリズムまたは振る舞い

1. **ステータス解析**: メッセージから 16 ビットのステータスコードを読み取り r->status に設定。続いてステータス文字列を読み取り r->status_line を構築。 2. **ヘッダー数取得**: ヘッダー数を取得し、r->headers_out を初期化。既存の Set-Cookie ヘッダーを保持する。 3. **ヘッダー解析ループ**: 指定された数だけヘッダーを反復処理。

  • ヘッダー名が AJP 内部 ID の場合は long_res_header_for_sc で文字列に変換。
  • それ以外は文字列として読み取る。
  • Set-Cookie、Location、Content-Location、URI、Destination に対しては、dconf を用いたリバースマップ変換を適用。
  • Content-Type ヘッダーの場合は ap_set_content_type を呼び出す。

    • 4. **クリーンアップ**: Content-Length および Transfer-Encoding ヘッダーを r->headers_out から削除し、コアによる適切なフレーム制御を許可する。

呼び出し元と呼び出し先

  • **呼び出し元**: modules/proxy/ajp_header.c:800（ajp_parse_header 関数内から呼び出される）。
  • **呼び出し先**:
  • ajp_msg_get_uint16, ajp_msg_get_string, ajp_msg_peek_uint16 (メッセージ操作)
  • long_res_header_for_sc (ヘッダー名変換)
  • ap_proxy_cookie_reverse_map, ap_proxy_location_reverse_map (パス変換)
  • ap_set_content_type (コンテンツタイプ設定)
  • addit_dammit (ヘッダーコピー用コールバック)

関連データ構造

  • ajp_msg_t: AJP 通信のバイナリパケット構造体。
  • request_rec: Apache のリクエスト情報。
  • proxy_dir_conf: プロキシ設定情報。

失敗モードと境界条件

  • ステータスコードやヘッダー数の読み取りに失敗した場合、エラーログを出力して APR_SUCCESS 以外の値を返す。
  • 不明な AJP ヘッダー ID が検出された場合、AJP_EBAD_HEADER を返す。
  • ヘッダー名や値の読み取りで ajp_msg_get_string が失敗した場合、エラーを伝播させる。

保守担当者向けメモ

  • r->headers_out の再構築時に apr_table_make を使用している。既存のヘッダーを完全に破棄するのではなく、Set-Cookie のような特定のヘッダーを維持するロジックが含まれている点に注意。
  • Content-Length の削除ロジックは、AJP 独自のフレーム構造と Apache のフィルタチェーンの整合性を保つための重要な処理である。
