ap_proxy_clean_warnings

定義

  • 種別: function
  • 定義箇所: modules/proxy/mod_proxy_http.c:230

目的

HTTP レスポンスヘッダーに含まれる Warning ヘッダーを検証し、期限切れの警告を削除またはフィルタリングして、有効な警告のみを含むヘッダーテーブルを返します。

シグネチャまたは形状

apr_table_t *ap_proxy_clean_warnings(apr_pool_t *p, apr_table_t *headers)

入力と出力

  • **入力**:
  • p: メモリ割り当てに使用する apr_pool_t。
  • headers: 処理対象の HTTP ヘッダーテーブル。
  • **出力**:
  • 警告が修正された場合は、元のヘッダーから Warning を除外し、フィルタリング済みの警告を追加した新しい apr_table_t を返します。警告に変更がない場合は、元の headers をそのまま返します。

前提条件と不変条件

  • headers テーブル内に Date ヘッダーが存在する場合、その値は apr_date_parse_http で解析可能である必要があります。
  • 内部で clean_warning_headers をコールバックとして使用するため、Warning ヘッダーの各値は clean_warning_headers が期待する形式に従っている必要があります。

アルゴリズムまたは振る舞い

1. Date ヘッダーを解析し、基準時刻を取得します。 2. apr_table_do を使用して、headers 内のすべての Warning ヘッダーに対して clean_warning_headers を適用します。 3. フィルタリングの結果、有効な警告が抽出された場合（x.table != NULL）:

  • 元のテーブルから Warning ヘッダーを削除します。
  • apr_table_overlay を使用して、元のヘッダーと抽出された警告テーブルを結合して返します。

    • 4. 警告に変更がない場合は、入力された headers をそのまま返します。

呼び出し元と呼び出し先

  • **呼び出し元**: modules/proxy/mod_proxy_http.c:1346 にて、レスポンスヘッダーのクリーンアップ処理として呼び出されています。
  • **呼び出し先**:
  • apr_date_parse_http (APR)
  • apr_table_do (APR)
  • clean_warning_headers (同一ファイル内の静的関数)
  • apr_table_unset (APR)
  • apr_table_overlay (APR)

関連データ構造

  • header_dptr: 警告のフィルタリング状態（プール、テーブル、基準時刻）を保持するために使用される構造体。
  • apr_table_t: HTTP ヘッダーの保持に使用される APR テーブル構造。

失敗モードと境界条件

  • Date ヘッダーが存在しない、または解析不能な場合、apr_date_parse_http は 0 を返します。この場合、警告の有効期限判定ロジックは 0 を基準に行われます。
  • Warning ヘッダーが存在しない場合、関数は入力された headers をそのまま返します。

保守担当者向けメモ

  • 本関数は Warning ヘッダーの RFC 準拠のフィルタリングを目的としています。ロジックの詳細は clean_warning_headers に依存しているため、警告の削除条件を変更する場合はそちらを確認してください。
  • apr_table_overlay を使用しているため、メモリ消費量はヘッダーの数に比例します。大量の警告ヘッダーが含まれるレスポンスでは注意が必要です。
