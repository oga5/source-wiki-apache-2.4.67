util_ald_create_cache

定義

  • 種別: function
  • 定義箇所: modules/ldap/util_ldap_cache_mgr.c:338

目的

LDAP キャッシュを管理するための util_ald_cache_t 構造体を初期化し、ハッシュテーブル領域を割り当てる。

シグネチャまたは形状

util_ald_cache_t *util_ald_create_cache(util_ldap_state_t *st,
                                long cache_size,
                                long cache_ttl,
                                unsigned long (*hashfunc)(void *),
                                int (*comparefunc)(void *, void *),
                                void * (*copyfunc)(util_ald_cache_t *cache, void *),
                                void (*freefunc)(util_ald_cache_t *cache, void *),
                                void (*displayfunc)(request_rec *r, util_ald_cache_t *cache, void *))

入力と出力

  • **入力**:
  • st: LDAP 状態管理構造体（共有メモリ管理用）。
  • cache_size: キャッシュ可能な最大エントリ数。
  • cache_ttl: キャッシュの有効期限（秒）。
  • hashfunc, comparefunc, copyfunc, freefunc, displayfunc: キャッシュ操作用コールバック関数群。
  • **出力**:
  • 成功時: 初期化済みの util_ald_cache_t へのポインタ。
  • 失敗時: NULL。

前提条件と不変条件

  • cache_size が 0 以下の場合、関数は即座に NULL を返す。
  • APR_HAS_SHARED_MEMORY が有効な場合、st->cache_rmm を介して共有メモリから割り当てを行う。無効または st->cache_rmm が未設定の場合は calloc を使用する。
  • キャッシュのハッシュテーブルサイズは、指定された cache_size の 1/3 を基準とし、64 以上かつ素数になるよう調整される。

アルゴリズムまたは振る舞い

1. キャッシュ構造体本体をメモリ（共有メモリまたはヒープ）に割り当てる。 2. ハッシュテーブルのバケット数（size）を計算し、素数テーブルに基づいて決定する。 3. util_ald_alloc を使用してハッシュテーブル（nodes）の領域を確保し、各バケットを NULL で初期化する。 4. 引数で渡されたコールバック関数を構造体に登録する。 5. キャッシュのパージ閾値（fullmark）を maxentries の 75% に設定し、統計用カウンタを 0 に初期化して構造体を返す。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • modules/ldap/util_ldap_cache.c:455
  • modules/ldap/util_ldap_cache_mgr.c:289, 297, 305
  • **呼び出し先**:
  • util_ald_alloc (メモリ割り当て)
  • util_ald_free (エラー時のメモリ解放)
  • apr_rmm_calloc / apr_rmm_addr_get (共有メモリ使用時)

関連データ構造

  • util_ald_cache_t: キャッシュのメイン構造体。
  • util_cache_node_t: ハッシュテーブルの各バケットに連結されるノード。
  • util_ldap_state_t: 共有メモリ管理情報を含む状態構造体。

失敗モードと境界条件

  • cache_size <= 0: NULL を返して終了。
  • メモリ割り当て失敗: cache 構造体または cache->nodes の確保に失敗した場合、適切にメモリを解放して NULL を返す。
  • cache_size が小さい場合: ハッシュテーブルサイズは最小 64 に切り上げられる。

保守担当者向けメモ

  • 共有メモリ使用時は apr_rmm を通じたポインタ管理が必要であり、直接的なポインタ代入は避けること。
  • ハッシュテーブルのサイズ決定ロジックは primes 配列（ソースコード内で定義）に依存している。
  • キャッシュのパージ閾値は maxentries の 3/4 に固定されている。
