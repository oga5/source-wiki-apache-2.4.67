http_ctx_t

定義

  • 種別: struct
  • 定義箇所: modules/http/http_filters.c:60

目的

HTTP 入力フィルタにおけるリクエストボディの解析状態およびストリーム処理の進捗を管理するためのコンテキスト構造体です。

シグネチャまたは形状

typedef struct http_filter_ctx {
    apr_off_t remaining;
    apr_off_t limit;
    apr_off_t limit_used;
    apr_int32_t chunk_used;
    apr_int32_t chunk_bws;
    apr_int32_t chunkbits;
    enum { ... } state;
    unsigned int eos_sent :1,
                 seen_data:1;
    apr_bucket_brigade *bb;
} http_ctx_t;

入力と出力

  • **入力**: フィルタチェーンを通じて受信する apr_bucket_brigade を保持し、解析状態に応じてデータを消費します。
  • **出力**: 解析されたリクエストボディの断片を後続のハンドラへ提供し、プロトコル違反や制限超過時にはエラー状態を保持します。

前提条件と不変条件

  • state メンバは、RFC 7230 に準拠したチャンク転送の解析状態（BODY_CHUNK から BODY_CHUNK_TRAILER まで）を厳密に遷移する必要があります。
  • remaining は、Content-Length またはチャンクサイズに基づく残りの読み取りバイト数を保持します。

アルゴリズムまたは振る舞い

  • **状態管理**: state を用いて、ストリームデータが「チャンクヘッダ」「データ本体」「トレーラ」のどの段階にあるかを追跡します。
  • **制限管理**: limit および limit_used を使用して、設定されたリクエストボディの最大サイズ制限を強制します。
  • **バッファリング**: bb（apr_bucket_brigade）を介して、フィルタリング対象のデータを一時的に保持・処理します。

呼び出し元と呼び出し先

  • **呼び出し元**: ap_http_filter がこの構造体をフィルタコンテキストとして利用します。また、bail_out_on_error、parse_chunk_size、read_chunked_trailers が引数として受け取ります。
  • **呼び出し先**: 構造体自体はデータ保持用であり、直接関数を呼び出すことはありませんが、各フィルタ関数が本構造体のメンバを更新・参照します。

関連データ構造

  • apr_bucket_brigade: ストリームデータを保持するバッファ。
  • ap_filter_t: 本構造体がコンテキストとしてアタッチされるフィルタ構造体。

失敗モードと境界条件

  • limit を超えるデータを受信した場合、bail_out_on_error を通じて接続を終了させる必要があります。
  • チャンク解析中に不正なフォーマットやオーバーフローが検出された場合、プロトコルエラーとして処理されます。
  • eos_sent フラグにより、EOS（End of Stream）バケットの二重送信を防止します。

保守担当者向けメモ

  • state の遷移は非常に複雑です。特に BODY_CHUNK_CR や BODY_CHUNK_LF などの細かい状態は、厳密なプロトコル解析のために存在するため、変更時は RFC 7230 を参照してください。
  • メモリ管理は r->pool または f->c->bucket_alloc に依存しているため、バケット操作時のリークに注意してください。
