unload_module

定義

  • 種別: function
  • 定義箇所: modules/core/mod_so.c:129

目的

Apache HTTP Server の終了時や設定の再読み込み時に、動的にロードされたモジュールを内部管理リストから削除し、メモリ上の参照を無効化するクリーンアップ処理を行います。

シグネチャまたは形状

static apr_status_t unload_module(void *data)

入力と出力

  • **入力**: data - ap_module_symbol_t 構造体へのポインタ。アンロード対象のモジュール情報を含みます。
  • **出力**: apr_status_t - 常に APR_SUCCESS を返します。

前提条件と不変条件

  • 引数 data は ap_module_symbol_t* としてキャスト可能である必要があります。
  • modi->modp が NULL の場合、関数は何もせず正常終了します。

アルゴリズムまたは振る舞い

1. 引数として渡された ap_module_symbol_t 構造体内の modp（モジュールポインタ）を確認します。 2. modp が NULL であれば、既にアンロード済みと判断し APR_SUCCESS を返して終了します。 3. ap_remove_loaded_module(modi->modp) を呼び出し、Apache のグローバルなモジュールリストから当該モジュールを除去します。 4. modi->modp および modi->name を NULL に設定し、参照をクリアします。 5. APR_SUCCESS を返します。

呼び出し元と呼び出し先

  • **呼び出し元**: modules/core/mod_so.c:312 において、apr_pool_cleanup_register のコールバック関数として登録されます。
  • **呼び出し先**: ap_remove_loaded_module（Apache Core API）。

関連データ構造

  • ap_module_symbol_t: モジュールの名前とポインタを保持する構造体。
  • apr_pool_t: この関数が登録されるメモリプール。

失敗モードと境界条件

  • 本関数自体はエラーを返さない設計ですが、ap_remove_loaded_module が内部で失敗した場合の挙動は Apache Core の実装に依存します。
  • 二重に呼び出された場合、最初の呼び出しで modp が NULL になるため、二回目以降は安全に無視されます。

保守担当者向けメモ

  • 本関数は apr_pool_cleanup_register を介して呼び出されるため、直接呼び出すのではなく、メモリプールのライフサイクル管理に委ねる必要があります。
  • modi->modp を NULL に設定することで、クリーンアップ後の再実行を防ぐ不変条件を維持しています。
