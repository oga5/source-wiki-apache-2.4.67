proxy_conn_rec

定義

  • 種別: struct
  • 定義箇所: modules/proxy/mod_proxy.h:269

目的

バックエンドサーバーへの接続状態を保持・管理するための構造体です。ソケット、接続プール、SSL 状態、および接続に関連するメモリ管理用プールを統合的に管理します。

シグネチャまたは形状

typedef struct {
    conn_rec     *connection;
    request_rec  *r;
    proxy_worker *worker;
    apr_pool_t   *pool;
    const char   *hostname;
    apr_sockaddr_t *addr;
    apr_pool_t   *scpool;
    apr_socket_t *sock;
    void         *data;
    void         *forward;
    apr_uint32_t flags;
    apr_port_t   port;
    unsigned int is_ssl:1;
    unsigned int close:1;
    unsigned int need_flush:1;
    unsigned int inreslist:1;
    const char   *uds_path;
    const char   *ssl_hostname;
    apr_bucket_brigade *tmp_bb;
    apr_pool_t   *uds_pool;
    apr_pool_t   *fwd_pool;
    struct proxy_address *address;
} proxy_conn_rec;

入力と出力

  • **入力**: 接続確立時に proxy_worker から参照され、apr_pool_t を介してホスト名やアドレス情報が割り当てられます。
  • **出力**: ap_proxy_acquire_connection 等の関数を通じて、確立されたソケットや接続状態がプロキシハンドラへ提供されます。

前提条件と不変条件

  • scpool はソケットおよび接続データ専用のサブプールとして使用されます。
  • tmp_bb は接続および scpool に紐付いて作成され、使用後はクリーンアップされる必要があります。
  • inreslist フラグは、当該接続が apr_reslist（接続プール）によって管理されているかを示します。

アルゴリズムまたは振る舞い

1. **接続管理**: proxy_worker が保持する接続プールから取得・返却されます。 2. **データ転送**: sock を介してバックエンドと通信し、必要に応じて tmp_bb（バケットブリゲード）を使用してデータを一時保持します。 3. **状態制御**: close フラグが設定された場合、接続は再利用されずに破棄されます。is_ssl や ssl_hostname（SNI）により、SSL/TLS 通信の要否とホスト名が制御されます。 4. **UDS 対応**: uds_path が設定されている場合、Unix Domain Socket を介した通信が行われます。

呼び出し元と呼び出し先

  • **呼び出し元**: modules/http2/h2_proxy_session.c において、h2_proxy_session 構造体や接続セッションのコンテキスト内で参照されています。
  • **呼び出し先**: 接続管理のライフサイクルにおいて apr_reslist 関連の APR 関数や、プロキシのユーティリティ関数（ap_proxy_acquire_connection 等）が本構造体を操作します。

関連データ構造

  • proxy_worker: 接続が属する接続プールを管理します。
  • conn_rec: Apache のコア接続構造体。
  • apr_reslist_t: 接続の再利用を管理するリソースリスト。
  • apr_bucket_brigade: データ転送用の一時バッファ。

失敗モードと境界条件

  • scpool のメモリ不足により接続確立が失敗する可能性があります。
  • uds_path が指定されている場合、ファイルシステム上のパス解決に失敗すると接続が確立できません。
  • close フラグが適切に管理されない場合、接続プール内でのリソースリークが発生する可能性があります。

保守担当者向けメモ

  • 本構造体はバイナリ互換性に影響を与えるため、フィールドの追加・削除時は ap_mmn.h（モジュールマジックナンバー）の更新が必要です。
  • uds_path や ssl_hostname などのフィールドは、プロキシ機能の拡張に伴い追加された経緯があります。
  • r（リクエストレコード）フィールドは、過去のバージョンで追加・削除の経緯があるため、現在の実装におけるライフサイクル管理には特に注意してください。
