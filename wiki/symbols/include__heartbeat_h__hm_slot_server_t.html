hm_slot_server_t

定義

  • 種別: struct
  • 定義箇所: include/heartbeat.h:41

目的

Apache HTTP Server のハートビート機能において、バックエンドサーバーの稼働状況を共有メモリやファイル経由で受け渡すためのデータ構造です。

シグネチャまたは形状

typedef struct hm_slot_server_t
{
    char ip[MAXIPSIZE];
    int busy;
    int ready;
    apr_time_t seen;
    int id;
} hm_slot_server_t;

入力と出力

  • **入力**: mod_heartmonitor が収集したサーバーの統計情報（IPアドレス、負荷状態、最終確認時刻など）が各フィールドに書き込まれます。
  • **出力**: mod_lbmethod_heartbeat がこの構造体を読み取り、ロードバランサーの負荷分散先選定ロジックに反映させます。

前提条件と不変条件

  • MAXIPSIZE（46バイト）は、IPv4-mapped IPv6 アドレスを格納するために十分なサイズである必要があります。
  • 共有メモリ（slotmem）を介してやり取りされるため、プラットフォーム間での構造体のパディングやアライメントの一致が前提となります。

アルゴリズムまたは振る舞い

本構造体はデータ保持のみを行い、直接的な制御フローは持ちません。mod_heartmonitor が収集した情報をシリアライズし、mod_lbmethod_heartbeat がデシリアライズすることで、モジュール間でのサーバー状態の共有を実現します。

呼び出し元と呼び出し先

  • **呼び出し元（参照元）**:
  • modules/cluster/mod_heartmonitor.c: 共有メモリの確保およびサーバー情報の書き込みに使用。
  • modules/proxy/balancers/mod_lbmethod_heartbeat.c: 共有メモリからサーバー情報を読み取り、負荷分散判断に使用。
  • **呼び出し先**:
  • 本構造体自体はデータ型であり、関数呼び出しは含みません。

関連データ構造

  • apr_time_t: seen フィールドで使用される時間型。
  • MAXIPSIZE: ip フィールドのサイズを規定する定数。

失敗モードと境界条件

  • 共有メモリが利用できない場合、DEFAULT_HEARTBEAT_STORAGE で定義されたファイルがフォールバックとして使用されます。
  • MAXIPSIZE を超える IP アドレスが渡された場合、バッファオーバーフローのリスクがあるため、書き込み側の実装で適切な切り詰めが必要です。

保守担当者向けメモ

  • 本構造体は mod_heartmonitor と mod_lbmethod_heartbeat 間の「契約」です。フィールドの追加や変更を行う際は、両モジュール間でのバイナリ互換性を維持してください。
  • 共有メモリ（slotmem）を使用しているため、構造体のサイズ変更は共有メモリレイアウトの破壊を招く可能性があります。
