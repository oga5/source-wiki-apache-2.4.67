session_rec

定義

  • 種別: struct
  • 定義箇所: modules/session/mod_session.h:84

目的

Apache HTTP Server のセッション管理において、個々のセッション状態（キー・値ペア、有効期限、識別子、状態フラグ）を保持・管理するための主要なデータ構造です。

シグネチャまたは形状

typedef struct {
    apr_pool_t *pool;
    apr_uuid_t *uuid;
    const char *remote_user;
    apr_table_t *entries;
    const char *encoded;
    apr_time_t expiry;
    long maxage;
    int dirty;
    int cached;
    int written;
} session_rec;

入力と出力

本構造体は関数ではなくデータ構造ですが、ap_session_load フックを通じてリクエストごとに生成・取得され、ap_session_save フックを通じて永続化されます。

前提条件と不変条件

  • entries はセッションのキー・値ペアを保持する apr_table_t です。
  • dirty フラグは、セッション内容が変更された際にセットされ、保存処理のトリガーとなります。
  • expiry が 0 より大きい場合、その値がセッションの有効期限を示します。

アルゴリズムまたは振る舞い

1. **ロード**: リクエスト開始時に session_load フックにより復元されます。 2. **操作**: ap_session_get / set 関数を介して entries が操作されます。 3. **保存**: リクエスト終了時に dirty フラグが確認され、セットされている場合のみ session_save フックが実行されて永続化されます。

呼び出し元と呼び出し先

  • **呼び出し元**: mod_session.c の ap_session_load および ap_session_save、mod_auth_form.c など。
  • **呼び出し先**: 本構造体自体はデータ保持用であり、直接関数を呼び出すことはありません。

関連データ構造

  • apr_table_t: セッションデータの保持に使用。
  • apr_pool_t: メモリ管理に使用。
  • session_dir_conf: ディレクトリ単位の設定を保持する構造体。

失敗モードと境界条件

  • dirty フラグの管理が不適切な場合、セッションの変更が永続化されない可能性があります。
  • expiry や maxage の計算において、システム時刻との整合性が保たれない場合、セッションが早期に無効化される可能性があります。

保守担当者向けメモ

  • session_encode および session_decode フックは RUN_ALL で実行されるため、複数のモジュールが連鎖的にデータを処理する設計になっています。
  • パフォーマンス最適化のため、変更時のみ dirty フラグを立てる運用を徹底してください。
