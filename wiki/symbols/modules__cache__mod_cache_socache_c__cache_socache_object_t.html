cache_socache_object_t

定義

  • **種別**: struct
  • **定義箇所**: modules/cache/mod_cache_socache.c:75

目的

mod_cache_socache モジュールにおいて、キャッシュエントリのメモリ内表現を保持するためのデータ構造です。HTTP レスポンスのヘッダー、ボディ、およびメタデータを統合的に管理します。

シグネチャまたは形状

typedef struct cache_socache_object_t {
    apr_pool_t *pool;
    unsigned char *buffer;
    apr_size_t buffer_len;
    apr_bucket_brigade *body;
    apr_table_t *headers_in;
    apr_table_t *headers_out;
    cache_socache_info_t socache_info;
    apr_size_t body_offset;
    apr_off_t body_length;
    apr_time_t expire;
    const char *name;
    const char *key;
    apr_off_t offset;
    apr_time_t timeout;
    unsigned int newbody :1;
    unsigned int done :1;
} cache_socache_object_t;

入力と出力

  • **入力**: キャッシュ生成時、apr_pcalloc を介してメモリが割り当てられ、リクエスト情報やヘッダー情報が格納されます。
  • **出力**: socache バックエンドへの永続化、またはキャッシュヒット時のレスポンス構築に使用されます。

前提条件と不変条件

  • pool メンバは、キャッシュエントリのライフサイクルを管理します。
  • キャッシュバッファは「フォーマットヘッダー + メタデータ + ヘッダー群 + ボディ」の順でパックされる必要があります。

アルゴリズムまたは振る舞い

1. **初期化**: open_entity 等の処理内で apr_pcalloc を用いてインスタンス化されます。 2. **データ構築**: store_table 等のユーティリティ関数により、ヘッダーテーブルがバッファへシリアライズされます。 3. **ボディ管理**: body ブリゲードは apr_bucket_pool_create を用いてバッファからマッピングされます。 4. **終了処理**: commit_entity 完了時やエラー発生時に apr_pool_destroy を呼び出し、メモリを解放します。

呼び出し元と呼び出し先

  • **呼び出し元**: mod_cache_socache.c 内の create_entity、open_entity、store_body 等のキャッシュプロバイダー関数から参照・操作されます。
  • **呼び出し先**: apr_pool_t や apr_bucket_brigade 等の APR API を利用します。

関連データ構造

  • cache_socache_info_t: キャッシュエントリのメタデータ。
  • apr_bucket_brigade: HTTP レスポンスボディのストリーム表現。
  • apr_table_t: HTTP ヘッダーの格納用。

失敗モードと境界条件

  • **メモリ不足**: apr_pcalloc 失敗時の挙動は呼び出し元に依存します。
  • **範囲リクエスト**: 現在、範囲リクエスト（Range Requests）のキャッシュには対応していません。
  • **排他制御**: バックエンドがスレッドセーフでない場合、socache_mutex による保護が必要です。

保守担当者向けメモ

  • pool の管理はメモリリーク防止の要です。open_entity 失敗時や commit_entity 完了時の解放漏れに注意してください。
  • slider 変数（ソースコード内のロジック）がバッファ内のオフセットを追跡しているため、構造変更時はオフセット計算の整合性を確認してください。
