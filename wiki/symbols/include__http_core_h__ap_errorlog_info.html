ap_errorlog_info

定義

  • 種別: struct
  • 定義箇所: include/http_core.h:890

目的

エラーログ出力処理において、ログメッセージに関連するコンテキスト情報（サーバー、接続、リクエスト、発生場所、エラー状態など）を保持し、error_log フックを通じて各ハンドラへ伝達するために使用されます。

シグネチャまたは形状

typedef struct ap_errorlog_info {
    const server_rec *s;
    const conn_rec *c;
    const request_rec *r;
    const request_rec *rmain;
    apr_pool_t *pool;
    const char *file;
    int line;
    int module_index;
    int level;
    apr_status_t status;
    int using_syslog;
    int startup;
    const char *format;
} ap_errorlog_info;

入力と出力

  • **入力**: ログ生成時に各フィールドが設定されます。s, c, r はそれぞれ関連するサーバー、接続、リクエスト情報を指します。file や line はソースコード上の発生位置を示します。
  • **出力**: 本構造体は error_log フックの引数として渡され、ログフォーマットの解析や出力処理を行う関数（log_remote_address 等）によって参照されます。

前提条件と不変条件

  • s, c, r は、それぞれ c->base_server や r->server、r->connection よりも優先して参照されるべき情報源です。
  • rmain は、リクエスト r がサブクエストである場合は r->main を指し、それ以外の場合は r 自身を指します。
  • pool は ap_log_perror が呼び出された場合にのみ有効であり、それ以外は NULL となる可能性があります。

アルゴリズムまたは振る舞い

本構造体は、ログ出力のライフサイクル全体を通じて、エラーログのフォーマット定義やフィルタリング、および error_log フックに登録された各モジュールのログハンドラに対して、一貫したコンテキストを提供します。

呼び出し元と呼び出し先

  • **呼び出し元**: server/core.c および server/log.c 内のログ出力関数群が本構造体を初期化し、フックを起動します。
  • **呼び出し先**: error_log フックに登録された関数（log_constant_item, log_remote_address, log_local_address, log_pid, log_tid 等）が、本構造体を引数として受け取り情報を抽出します。

関連データ構造

  • server_rec, conn_rec, request_rec: サーバー、接続、リクエストの基本構造体。
  • apr_pool_t: メモリ管理用プール。
  • ap_errorlog_handler_fn_t: 本構造体を受け取るログハンドラの関数型定義。

失敗モードと境界条件

  • file が NULL または line が 0 の場合、発生場所の情報は利用不可（N/A）として扱われます。
  • module_index が APLOG_NO_MODULE の場合、モジュール情報は付与されません。
  • level が -1 の場合、ログレベルは未定義として扱われます。
  • status が 0 の場合、エラー状態は存在しないものとみなされます。

保守担当者向けメモ

  • 2011年（2.3.12-dev）に format フィールドが追加され、2011年（2.4.0-dev）に pool フィールドが追加された経緯があります。
  • error_log フックのシグネチャ変更時には、本構造体のフィールド整合性に注意してください。
