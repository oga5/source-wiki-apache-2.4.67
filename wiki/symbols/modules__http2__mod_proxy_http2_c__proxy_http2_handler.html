proxy_http2_handler

定義

  • 種別: function
  • 定義箇所: modules/http2/mod_proxy_http2.c:318

目的

Apache HTTP Server のプロキシサブシステムにおいて、h2: および h2c: スキームのリクエストを処理し、バックエンドサーバーへの HTTP/2 通信を確立・実行するメインハンドラです。

シグネチャまたは形状

static int proxy_http2_handler(request_rec *r, 
                               proxy_worker *worker,
                               proxy_server_conf *conf,
                               char *url, 
                               const char *proxyname,
                               apr_port_t proxyport)

入力と出力

  • **入力**:
  • r: 現在のクライアントリクエスト情報。
  • worker: プロキシワーカ設定。
  • url: 転送先 URL。
  • **出力**:
  • int: HTTP ステータスコード（成功時は OK、失敗時はエラーコード、対象外スキームは DECLINED）。

前提条件と不変条件

  • url が h2 または h2c で始まる必要があります。
  • リクエストのプール (r->pool) を使用して h2_proxy_ctx が割り当てられ、リクエスト終了時に自動的に解放されます。
  • h2_proxy_ctx は r->connection の conn_config に設定されます。

アルゴリズムまたは振る舞い

1. **スキーム検証**: URL が h2 または h2c かを確認し、プロキシ関数名と SSL 使用有無を決定します。 2. **コンテキスト初期化**: h2_proxy_ctx を作成し、リクエスト情報を格納します。 3. **接続確立ループ (run_connect)**:

  • ap_proxy_acquire_connection でバックエンド接続を取得します。
  • ap_proxy_determine_connection で接続先を確定します。
  • ap_proxy_connect_backend でバックエンドへのソケット接続を確立します。
  • ap_proxy_connection_create_ex で接続用の conn_rec を作成します。

    • 4. **セッション実行**: ctx_run(ctx) を呼び出し、HTTP/2 セッションを開始してリクエストを処理します。 5. **再試行判定**: エラー発生時、proxy-error-override が設定されておらず、かつ再試行可能であれば、最大 1 回まで接続を破棄して再試行します。 6. **終了処理**: 接続の解放、モジュール設定のクリアを行い、最終的なステータスを返します。

呼び出し元と呼び出し先

  • **呼び出し元**: proxy_hook_scheme_handler に登録され、プロキシフレームワークから呼び出されます。
  • **呼び出し先**:
  • ap_proxy_acquire_connection, ap_proxy_determine_connection, ap_proxy_connect_backend, ap_proxy_connection_create_ex (プロキシフレームワーク API)
  • ctx_run (セッション実行)
  • proxy_run_detach_backend (条件付き呼び出し)

関連データ構造

  • h2_proxy_ctx: リクエスト処理中の状態を保持する構造体。
  • request_rec: Apache のリクエスト構造体。
  • proxy_worker, proxy_server_conf: プロキシ設定構造体。

失敗モードと境界条件

  • **スキーム不一致**: h2/h2c 以外の場合は DECLINED を返します。
  • **接続失敗**: ap_proxy_acquire_connection 等が失敗した場合、cleanup へ遷移しエラーを返します。
  • **再試行制限**: reconnects が 2 回に達すると再試行を停止します。
  • **セッション状態**: セッションが「クリーン」でない（オープンなストリームが存在する）場合、接続は再利用されずクローズされます。

保守担当者向けメモ

  • h2_proxy_ctx の r_may_retry フラグが再試行の可否を制御します。
  • ctx_run は h2_proxy_session 関連のロジックへ処理を委譲しています。
  • AP_MODULE_MAGIC_AT_LEAST マクロにより、特定のバージョン以降で proxy_run_detach_backend が呼び出されます。
