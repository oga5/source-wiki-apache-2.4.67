ap_watchdog_t

定義

  • 種別: struct
  • 定義箇所: modules/core/mod_watchdog.c:46

目的

Apache HTTP Server におけるバックグラウンド・タスク監視（Watchdog）フレームワークのメイン構造体であり、特定の監視インスタンスに関連するスレッド、排他制御、およびコールバックリストを保持・管理します。

シグネチャまたは形状

struct ap_watchdog_t {
    apr_uint32_t          thread_started; /* スレッド実行時に1に設定 */
    apr_proc_mutex_t     *mutex;          /* プロセス間排他制御用 */
    const char           *name;           /* インスタンス名 */
    watchdog_list_t      *callbacks;      /* 登録されたコールバックのリスト */
    int                   is_running;     /* 実行状態フラグ */
    int                   singleton;      /* シングルトンモードの有効化フラグ */
    int                   active;         /* アクティブ状態フラグ */
    apr_interval_time_t   step;           /* 実行間隔 */
    apr_thread_t         *thread;         /* ワーカースレッド */
    apr_pool_t           *pool;           /* メモリ管理用プール */
};

入力と出力

  • **入力**: ap_watchdog_get_instance を介して生成・取得されます。
  • **出力**: 内部の thread が wd_worker を実行し、登録されたコールバックを定期的に呼び出します。

前提条件と不変条件

  • pool は登録時に指定されたライフサイクル（通常は pconf）に従います。
  • singleton モードが有効な場合、mutex を使用して全子プロセス中で唯一の実行が保証されます。
  • コールバック関数内での長時間ブロックは、インスタンス全体の遅延を引き起こします。

アルゴリズムまたは振る舞い

  • wd_worker が専用スレッドとして起動し、apr_sleep を用いて AP_WD_TM_SLICE（100ms）ごとにループします。
  • callbacks リストを走査し、経過時間が step を超えた場合にコールバックを実行します。
  • 終了時には wd_worker_cleanup がプール破棄時に呼ばれ、STOPPING 状態を通知してスレッドを join します。

呼び出し元と呼び出し先

  • **呼び出し元**: mod_watchdog.c 内の wd_startup, ap_watchdog_get_instance, ap_watchdog_set_callback_interval, ap_watchdog_register_callback 等がこの構造体を操作します。また、mod_heartmonitor.c 等の外部モジュールから参照されます。
  • **呼び出し先**: 構造体自体はデータ保持用であり、直接的な関数呼び出し先は定義されていません。

関連データ構造

  • watchdog_list_t: 登録されたコールバック関数のリンクリスト。
  • wd_server_conf_t: モジュール全体のサーバー設定。

失敗モードと境界条件

  • singleton モードにおいて、apr_proc_mutex のロック取得に失敗したプロセスは待機状態となります。
  • スレッド生成やメモリ確保（apr_pcalloc）の失敗は、apr_status_t を通じて呼び出し元に伝播されます。

保守担当者向けメモ

  • 本構造体は mod_watchdog.c の内部実装と密接に関連しています。
  • メモリ所有権は pool に依存するため、インスタンスの破棄タイミングには注意が必要です。
  • ap_run_watchdog_* フックの具体的な実装は各利用モジュール側に存在するため、本構造体はあくまで実行基盤の管理に徹しています。
