util_ldap_connection_t

定義

  • 種別: struct
  • 定義箇所: include/util_ldap.h:106

目的

LDAP サーバーへの接続状態、認証情報、および接続プール内での管理情報を保持する構造体です。Apache HTTP Server の mod_ldap において、LDAP 接続のライフサイクルと再利用を管理するために使用されます。

シグネチャまたは形状

typedef struct util_ldap_connection_t {
    LDAP *ldap;
    apr_pool_t *pool;
#if APR_HAS_THREADS
    apr_thread_mutex_t *lock;
#endif
    const char *host;
    int port;
    deref_options deref;
    const char *binddn;
    const char *bindpw;
    int bound;
    int secure;
    apr_array_header_t *client_certs;
    const char *reason;
    struct util_ldap_connection_t *next;
    struct util_ldap_state_t *st;
    int keep;
    int ChaseReferrals;
    int ReferralHopLimit;
    apr_time_t freed;
    apr_pool_t *rebind_pool;
    int must_rebind;
    request_rec *r;
    apr_time_t last_backend_conn;
} util_ldap_connection_t;

入力と出力

本構造体はデータ保持を目的としており、関数ではありません。各フィールドは LDAP 接続の構成設定（ホスト、ポート、認証情報）および実行時の状態（バインド済みフラグ、エラー理由、最終接続時刻）を格納します。

前提条件と不変条件

  • APR_HAS_LDAP が定義されている環境で利用可能です。
  • マルチスレッド環境下では、lock フィールドを用いて接続の排他制御を行う必要があります。
  • st フィールドは、この接続が属する LDAP バーチャルホスト設定（util_ldap_state_t）を指す必要があります。

アルゴリズムまたは振る舞い

  • **接続管理**: uldap_connection_find によってプールから検索され、uldap_connection_close によってプールへ返却されます。
  • **再利用**: keep フラグが有効な場合、接続はクローズされずにプール内に保持されます。
  • **再バインド**: must_rebind フラグが設定されている場合、接続再利用時に再認証（バインド）処理が実行されます。

呼び出し元と呼び出し先

  • **呼び出し元**: mod_ldap の内部関数（uldap_connection_find 等）および mod_authnz_ldap などのクライアントモジュールが本構造体を操作します。
  • **呼び出し先**: 本構造体自体はデータ構造であり、直接的な関数呼び出しは行いません。
  • **注記**: ソースコード内の具体的な関数呼び出し関係の詳細は、本抜粋からは網羅的に特定できません。

関連データ構造

  • util_ldap_state_t: 接続が属するモジュール全体のグローバル設定およびキャッシュ状態。
  • LDAP: APR-util が提供する LDAP 接続ハンドル。
  • request_rec: 接続の検索に使用されたリクエストコンテキスト。

失敗モードと境界条件

  • reason フィールドに接続失敗時のエラー理由が格納されます。
  • bound フラグが 0 の場合、LDAP 操作を実行する前にバインド処理が必要です。
  • サーバーダウン時は AP_LDAP_IS_SERVER_DOWN マクロ等を用いて状態を判定する必要があります。

保守担当者向けメモ

  • 本構造体は mod_ldap の中核であり、フィールドの追加・変更は ap_mmn.h で定義されるモジュールバイナリ互換性（MMN）に影響を与える可能性があります。
  • rebind_pool は再バインドデータを保持するために頻繁にクリアされるため、永続的なデータの格納には使用しないでください。
