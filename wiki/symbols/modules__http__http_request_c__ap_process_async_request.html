ap_process_async_request

定義

  • 種別: function
  • 定義箇所: modules/http/http_request.c:407

目的

非同期リクエスト処理のコアロジックを実行し、リクエストのライフサイクル（フックの実行、ハンドラの呼び出し、後処理）を統括する。

シグネチャまたは形状

void ap_process_async_request(request_rec *r)

入力と出力

  • **入力**: request_rec *r （処理対象のリクエスト構造体）
  • **出力**: なし（戻り値は void）

前提条件と不変条件

  • r は有効な request_rec 構造体を指している必要がある。
  • スレッド環境下では、r->invoke_mtx を用いて排他制御が行われる。
  • 処理の開始と終了時に AP_PROCESS_REQUEST_ENTRY および AP_PROCESS_REQUEST_RETURN マクロが呼び出される。

アルゴリズムまたは振る舞い

1. **初期化**: 拡張ステータスが有効な場合、処理開始時間を記録する。 2. **ログ出力**: トレースレベルが 4 以上の場合、受信した全ヘッダーをログに記録する。 3. **排他制御**: スレッド環境では r->invoke_mtx をロックする。 4. **クイックハンドラ**: ap_run_quick_handler を実行し、成功（OK）または処理不要（DECLINED）を判定する。 5. **内部処理**: クイックハンドラが DECLINED の場合、ap_process_request_internal で設定適用とアクセスチェックを行い、成功すれば ap_invoke_handler でコンテンツハンドラを実行する。 6. **中断処理**: ステータスが SUSPENDED の場合、ステータスを更新して早期リターンする。 7. **終了処理**:

  • ap_die_r を呼び出し、エラー発生時のレスポンス生成を試みる。
  • ap_process_request_after_handler を呼び出し、EOR バケットの送信やパイプラインの確認を行う。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • modules/http/http_core.c:155
  • modules/http/http_request.c:487
  • **呼び出し先**:
  • ap_run_quick_handler
  • ap_process_request_internal
  • ap_invoke_handler
  • ap_die_r
  • ap_process_request_after_handler
  • ap_time_process_request (条件付き)

関連データ構造

  • request_rec: リクエストのコンテキストを保持する主要構造体。
  • conn_rec: 接続状態を保持する構造体。

失敗モードと境界条件

  • ap_run_quick_handler が SUSPENDED を返した場合、処理は中断され、後続のハンドラ実行や ap_die_r はスキップされる。
  • ap_process_request_internal や ap_invoke_handler がエラーコードを返した場合、ap_die_r がそのステータスに基づいたエラーレスポンスを処理する。

保守担当者向けメモ

  • quick_handler フックは、キャッシュや特定のルーティングに利用される強力なフックであるため、使用には細心の注意が必要である。
  • SUSPENDED 状態の処理において、AP_PROCESS_REQUEST_ENTRY を再呼び出しする汎用関数へのリファクタリングが検討されている。
