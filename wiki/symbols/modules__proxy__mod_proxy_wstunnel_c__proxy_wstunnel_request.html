proxy_wstunnel_request

定義

  • 種別: function
  • 定義箇所: modules/proxy/mod_proxy_wstunnel.c:163

目的

クライアントとバックエンドサーバー間での WebSocket トンネリングを確立し、双方向のデータストリームを中継する。

シグネチャまたは形状

static int proxy_wstunnel_request(apr_pool_t *p, request_rec *r,
                                proxy_conn_rec *conn,
                                proxy_worker *worker,
                                proxy_server_conf *conf,
                                apr_uri_t *uri,
                                char *url, char *server_portstr)

入力と出力

  • **入力**: リクエスト情報 (r)、バックエンド接続情報 (conn)、ワーカー設定 (worker)、URI 情報等。
  • **出力**: 処理成功時は OK、バックエンドへの応答がなかった場合は HTTP_BAD_GATEWAY、内部エラー時は HTTP_INTERNAL_SERVER_ERROR を返す。

前提条件と不変条件

  • r->connection および conn->connection が有効であること。
  • ap_proxy_create_hdrbrgd によりヘッダーブリゲードが正常に作成されていること。
  • 本関数が実行されると、当該接続の keepalive は AP_CONN_CLOSE に設定され、リクエスト処理全体を占有する。

アルゴリズムまたは振る舞い

1. **ヘッダー転送**: Upgrade および Connection ヘッダーを構築し、バックエンドへ送信する。 2. **ポーリング設定**: apr_pollset を作成し、クライアントソケットとバックエンドソケットを監視対象として登録する。 3. **トンネリングループ**:

  • apr_pollset_poll を使用してソケットの読み込み可能状態を待機する。
  • どちらかのソケットが読み込み可能になった場合、ap_proxy_transfer_between_connections を呼び出し、データを相互に転送する。
  • 接続が切断されるかエラーが発生するまでループを継続する。

    • 4. **終了処理**: ループ終了後、バックエンドからの応答有無を確認し、適切なステータスコードを返して終了する。

呼び出し元と呼び出し先

  • **呼び出し元**: modules/proxy/mod_proxy_wstunnel.c:433 (proxy_wstunnel_handler 内)
  • **呼び出し先**: ap_proxy_create_hdrbrgd, ap_proxy_pass_brigade, apr_pollset_create, ap_proxy_transfer_between_connections 等。

関連データ構造

  • apr_pollset_t: ソケット監視用。
  • apr_bucket_brigade: データ転送用バッファ。
  • proxy_conn_rec: バックエンド接続状態。

失敗モードと境界条件

  • apr_pollset_create の失敗: HTTP_INTERNAL_SERVER_ERROR を返す。
  • バックエンドまたはクライアントのソケットエラー: done フラグを立ててループを終了する。
  • 応答なし: ループ終了時に replied が偽であれば HTTP_BAD_GATEWAY を返す。

保守担当者向けメモ

  • 本関数はブロッキングループで接続を占有するため、他のリクエスト処理と並行して動作しない点に注意すること。
  • apr_brigade はリクエストプール (r->pool) を使用して管理されている。
  • Upgrade メソッドの挙動は worker->s->upgrade の設定値に依存する。
