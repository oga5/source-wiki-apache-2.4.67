is_best_bytraffic

定義

  • 種別: function
  • 定義箇所: modules/proxy/balancers/mod_lbmethod_bytraffic.c:28

目的

ロードバランサーが管理する全ワーカーの中から、転送バイト数と読み込みバイト数の合計を重み（lbfactor）で正規化した値が最小となるワーカーを特定します。

シグネチャまたは形状

static int is_best_bytraffic(proxy_worker *current, proxy_worker *prev_best, void *baton)

入力と出力

  • **入力**:
  • current: 現在評価中の proxy_worker 構造体。
  • prev_best: これまでに評価された中で最も負荷が低いと判断された proxy_worker（初回は NULL）。
  • baton: 最小負荷値を保持する apr_off_t 型変数へのポインタ。
  • **出力**:
  • TRUE (1): current が現時点で最も負荷が低い場合。
  • FALSE (0): それ以外の場合。

前提条件と不変条件

  • baton は apr_off_t 型の有効なメモリ領域を指している必要があります。
  • current および current->s は有効なワーカー情報を含んでいる必要があります。

アルゴリズムまたは振る舞い

1. (current->s->transferred + current->s->read) / current->s->lbfactor を計算し、現在のワーカーの負荷値を算出します。 2. prev_best が NULL であるか、算出された負荷値が baton が指す最小値（min_traffic）より小さい場合：

  • min_traffic を現在の負荷値で更新します。
  • TRUE を返します。

    • 3. 上記以外の場合は FALSE を返します。

呼び出し元と呼び出し先

  • **呼び出し元**: modules/proxy/balancers/mod_lbmethod_bytraffic.c:65 にて、ap_proxy_balancer_get_best_worker_fn のコールバックとして渡されます。
  • **呼び出し先**: 外部関数呼び出しはありません。

関連データ構造

  • proxy_worker: ワーカーの統計情報（transferred, read, lbfactor）を保持する構造体。

失敗モードと境界条件

  • lbfactor が 0 の場合、除算による例外が発生する可能性があります（mod_proxy の仕様上、通常は 1 以上に設定されます）。
  • baton が指すメモリが不正な場合、セグメンテーションフォールトが発生します。

保守担当者向けメモ

  • 本関数は ap_proxy_balancer_get_best_worker_fn を介して反復的に呼び出されるため、計算コストを最小限に抑える必要があります。
  • 統計情報の更新は呼び出し元の find_best_bytraffic 側で行われるため、本関数は比較と最小値の保持のみに専念しています。
