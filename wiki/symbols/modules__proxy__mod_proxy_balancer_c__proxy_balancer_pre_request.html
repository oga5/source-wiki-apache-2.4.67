proxy_balancer_pre_request

定義

  • 種別: function
  • 定義箇所: modules/proxy/mod_proxy_balancer.c:473

目的

Apache HTTP Server のプロキシリクエスト処理において、ロードバランサ経由のルーティングを決定し、適切なバックエンドワーカーを選択・設定する。

シグネチャまたは形状

int proxy_balancer_pre_request(proxy_worker **worker,
                               proxy_balancer **balancer,
                               request_rec *r,
                               proxy_server_conf *conf, char **url);

入力と出力

  • **入力**:
  • worker: 選択されたワーカーを格納するポインタ。
  • balancer: 対象のバランサ構造体（未指定の場合は URL から検索）。
  • r: 現在のリクエストレコード。
  • conf: プロキシサーバー設定。
  • url: リクエスト対象の URL（balancer:// スキーム）。
  • **出力**:
  • 成功時: OK または balancer_fixup の戻り値。
  • 失敗時: DECLINED（対象外）、HTTP_SERVICE_UNAVAILABLE（ワーカー不在またはエラー）。
  • worker および url は、選択されたワーカーの情報に基づいて更新される。

前提条件と不変条件

  • url が balancer:// で始まる必要がある。
  • マルチスレッド環境下では PROXY_THREAD_LOCK による排他制御が必須。
  • 選択されたワーカーの busy カウントは、リクエスト終了時に decrement_busy_count で自動的に減算されるよう登録される。

アルゴリズムまたは振る舞い

1. **バランサ特定**: balancer が未指定の場合、URL からバランサを検索する。 2. **排他制御**: スレッドセーフなアクセスを確保するため、バランサをロックする。 3. **状態更新**: force_recovery でエラー状態のワーカーを復旧させ、ap_proxy_sync_balancer でメンバーリストを同期する。 4. **ワーカー選択**:

  • find_session_route を呼び出し、スティッキーセッション情報に基づくワーカーを優先的に検索する。
  • セッションルートが見つからない場合、find_best_worker を使用して負荷分散アルゴリズムに基づきワーカーを決定する。

    • 5. **環境変数設定**: BALANCER_NAME, BALANCER_WORKER_NAME 等の情報をリクエスト環境変数に追加する。 6. **URL 書き換え**: balancer_fixup を呼び出し、balancer:// URL を実際のワーカーのスキーム・ホスト・ポートに変換する。 7. **終了処理**: ロックを解除し、結果を返す。

呼び出し元と呼び出し先

  • **呼び出し元**: proxy_hook_pre_request フック（modules/proxy/mod_proxy_balancer.c:2064）。
  • **呼び出し先**: ap_proxy_get_balancer, force_recovery, ap_proxy_sync_balancer, find_session_route, find_best_worker, balancer_fixup, decrement_busy_count。

関連データ構造

  • proxy_balancer: ロードバランサの状態とワーカーリストを保持。
  • proxy_worker: 個別のバックエンドワーカーの状態（lbstatus, lbfactor, busy 等）。
  • request_rec: リクエスト固有のコンテキストと環境変数テーブル。

失敗モードと境界条件

  • **ワーカー不在**: バランサに登録されたワーカーがすべてエラー状態、またはワーカーが存在しない場合 HTTP_SERVICE_UNAVAILABLE を返す。
  • **ロック失敗**: スレッドロックが取得できない場合、DECLINED を返す。
  • **ルート不一致**: スティッキーセッションで指定されたルートが現在のバランサに存在しない場合、エラーログを出力し HTTP_SERVICE_UNAVAILABLE を返す。

保守担当者向けメモ

  • balancer_fixup の戻り値が最終的なリクエストの成否に直結するため、変更時は注意が必要。
  • スレッドロックの取得・解放は APR_HAS_THREADS マクロで囲まれており、非スレッド環境との互換性が考慮されている。
