ajp_msg_append_uint8

定義

  • 種別: function
  • 定義箇所: modules/proxy/ajp_msg.c:308

目的

AJP メッセージバッファの末尾に 1 バイト（uint8）のデータを追加します。

シグネチャまたは形状

apr_status_t ajp_msg_append_uint8(ajp_msg_t *msg, apr_byte_t value)

入力と出力

  • **入力**:
  • msg: 操作対象の ajp_msg_t 構造体へのポインタ。
  • value: 追加する 8 ビットのデータ。
  • **出力**:
  • 成功時は APR_SUCCESS を返します。
  • バッファの容量不足時は ajp_log_overflow の戻り値（エラーコード）を返します。

前提条件と不変条件

  • msg は有効な ajp_msg_t 構造体を指している必要があります。
  • 内部の msg->len は常に現在のバッファ使用量を正しく示している必要があります。

アルゴリズムまたは振る舞い

1. 現在のメッセージ長 msg->len を取得します。 2. len + 1 が msg->max_size を超えるか検証します。 3. 超過している場合、ajp_log_overflow を呼び出し、エラーを返して終了します。 4. 超過していない場合、msg->buf の len 番目の位置に value を書き込みます。 5. msg->len を 1 インクリメントします。 6. APR_SUCCESS を返します。

呼び出し元と呼び出し先

  • **呼び出し元**: modules/proxy/ajp_header.c 内の各関数（リクエストヘッダーや属性の構築時）。
  • **呼び出し先**: ajp_log_overflow（エラー発生時）。

関連データ構造

  • ajp_msg_t: AJP メッセージを保持する構造体。buf（データ領域）、len（現在の長さ）、max_size（最大容量）を含みます。

失敗モードと境界条件

  • **バッファオーバーフロー**: msg->len + 1 > msg->max_size となる場合、データは書き込まれずエラーが返されます。

保守担当者向けメモ

  • 本関数はネットワークバイトオーダーを意識したシリアライズ処理の一部です。1 バイトの追加であるためエンディアン変換は不要ですが、上位の append 系関数と同様に、必ず ajp_log_overflow を介した境界チェックが行われます。
