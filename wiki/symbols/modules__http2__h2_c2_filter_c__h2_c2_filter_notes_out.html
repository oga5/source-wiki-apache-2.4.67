h2_c2_filter_notes_out

定義

  • 種別: function
  • 定義箇所: modules/http2/h2_c2_filter.c:48

目的

HTTP/2 セカンダリコネクション（C2）において、レスポンスバケットを走査し、特定の条件に基づいてレスポンスノート（notes）の更新や設定の適用を行う。

シグネチャまたは形状

apr_status_t h2_c2_filter_notes_out(ap_filter_t *f, apr_bucket_brigade *bb)

入力と出力

  • **入力**:
  • f: フィルタコンテキスト。f->r（リクエスト）および f->c（コネクション）を参照。
  • bb: 処理対象のバケットブリゲード。
  • **出力**:
  • 戻り値: ap_pass_brigade の結果（apr_status_t）。
  • 副作用: ブリゲード内の AP_BUCKET_IS_RESPONSE バケットの notes テーブルを更新、または f->next フィルタへブリゲードを渡す。

前提条件と不変条件

  • f->r が存在しない場合、ノートの更新処理をスキップし、直ちに pass ラベルへ遷移する。
  • AP_BUCKET_IS_RESPONSE バケットが見つかった場合のみ、内部の条件分岐が実行される。

アルゴリズムまたは振る舞い

1. f->r が存在しない場合は、後続のフィルタへブリゲードを渡す。 2. ブリゲード内の各バケットを走査し、AP_BUCKET_IS_RESPONSE バケットを特定する。 3. ステータスコードが 400 以上かつ f->r->prev が存在する場合（内部リダイレクト時）、リクエストチェーンを遡り、ssl-renegotiate-forbidden ノートを現在のレスポンスノートにコピーする。 4. 上記以外で、現在のリクエスト設定で PUSH が無効、かつサーバー設定で PUSH が有効な場合、H2_PUSH_MODE_NOTE を "0" に設定する。 5. 最後に ap_pass_brigade を呼び出し、処理済みのブリゲードを後続のフィルタへ渡す。

呼び出し元と呼び出し先

  • **呼び出し元**: modules/http2/h2_c2.c:638（ap_register_output_filter による登録）。
  • **呼び出し先**:
  • ap_pass_brigade（後続フィルタへの転送）。
  • h2_config_rgeti / h2_config_sgeti（設定値の取得）。
  • ap_log_cerror（ログ出力）。

関連データ構造

  • ap_filter_t: フィルタチェーンのノード。
  • apr_bucket_brigade: バケットのリスト。
  • ap_bucket_response: レスポンスバケットのデータ構造。
  • request_rec: リクエスト情報とノート（notes）を保持。

失敗モードと境界条件

  • f->r が NULL の場合、ノート更新を行わずに正常終了する。
  • AP_BUCKET_IS_RESPONSE バケットが含まれていない場合、ノートの更新は行われず、ブリゲードはそのまま通過する。
  • 内部リダイレクトの連鎖において、r_prev が NULL になるまで遡る。

保守担当者向けメモ

  • 本フィルタは H2_C2_NOTES_OUT として登録されている。
  • エラーレスポンス時のノート引き継ぎロジックは、HTTP/2 ストリーム固有のエラーコード設定と密接に関連している。
  • PUSH モードの制御は、リクエスト単位の設定とサーバー単位の設定の優先順位を考慮している。
