ap_check_pipeline

定義

  • 種別: function
  • 定義箇所: modules/http/http_request.c:237

目的

HTTP Keep-Alive 接続において、パイプライン化されたリクエストの先頭に存在する空行（CRLF または LF）を消費し、接続の継続可否を判定します。

シグネチャまたは形状

apr_status_t ap_check_pipeline(conn_rec *c, apr_bucket_brigade *bb, unsigned int max_blank_lines)

入力と出力

  • **入力**:
  • c: 接続状態を保持する conn_rec 構造体。
  • bb: 入力データを保持する apr_bucket_brigade。
  • max_blank_lines: 許容する空行の最大数。
  • **出力**:
  • apr_status_t: 処理結果。成功時は APR_SUCCESS または APR_EAGAIN、エラー時は APR_EGENERAL や APR_NOTFOUND 等を返します。

前提条件と不変条件

  • 接続 c が生存しており、かつ c->aborted が偽である必要があります。
  • max_blank_lines が 0 の場合、データが存在するかどうかの確認のみを行い、空行の消費は行いません。

アルゴリズムまたは振る舞い

1. AP_MODE_SPECULATIVE を用いて、接続からデータを非破壊的に読み取ります。 2. 読み取ったデータが空行（CRLF または LF）であるかを確認します。 3. 空行であれば num_blank_lines をインクリメントし、AP_MODE_READBYTES を用いて実際にそのデータを消費します。 4. num_blank_lines が max_blank_lines を超えた場合、c->keepalive を AP_CONN_CLOSE に設定し、接続を終了させます。 5. 空行以外のデータが検出された場合、またはエラーが発生した場合はループを終了します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • modules/http/http_request.c: ap_process_request_after_handler 内で呼び出されます。
  • modules/proxy/proxy_util.c: プロキシ接続の管理において呼び出されます。
  • **呼び出し先**:
  • ap_get_brigade: 入力フィルタからデータを取得します。
  • apr_brigade_cleanup, apr_brigade_length, apr_brigade_flatten: バケットブリゲードの操作を行います。
  • ap_log_cerror: エラーログの出力を行います。

関連データ構造

  • conn_rec: 接続状態（Keep-Alive 状態など）を管理します。
  • apr_bucket_brigade: ネットワークから読み込まれたデータストリームを保持します。

失敗モードと境界条件

  • **空行超過**: max_blank_lines を超える空行が続いた場合、APR_NOTFOUND を返し接続を閉じます。
  • **読み取りエラー**: ap_get_brigade が失敗した場合、接続状態に応じて APR_EGENERAL または APR_EAGAIN を返します。
  • **データ不整合**: 期待した形式のデータが読み取れなかった場合、ログを出力しエラーを返します。

保守担当者向けメモ

  • 本関数は http_request.c のリクエスト処理フローにおいて、Keep-Alive 接続の効率的な再利用を制御する重要な役割を担っています。
  • max_blank_lines の制限値は、DoS 攻撃対策として機能します。
