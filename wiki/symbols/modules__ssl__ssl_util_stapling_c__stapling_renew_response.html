stapling_renew_response

定義

  • 種別: function
  • 定義箇所: modules/ssl/ssl_util_stapling.c:479

目的

OCSP レスポンダーに対して証明書の有効性確認を再問い合わせし、取得したレスポンスをキャッシュへ更新する。

シグネチャまたは形状

BOOL stapling_renew_response(server_rec *s, modssl_ctx_t *mctx, SSL *ssl,
                             certinfo *cinf, OCSP_RESPONSE **prsp,
                             BOOL *pok, apr_pool_t *pool)

入力と出力

  • **入力**:
  • s: サーバーレコード。
  • mctx: SSL コンテキスト。
  • ssl: 現在の SSL 接続オブジェクト。
  • cinf: 証明書情報（certinfo 構造体）。
  • pool: キャッシュ保存に使用するメモリプール。
  • **出力**:
  • prsp: 取得した OCSP_RESPONSE へのポインタ。
  • pok: レスポンスが有効かどうかのフラグ。
  • 戻り値: 処理が成功した場合は TRUE、失敗した場合は FALSE。

前提条件と不変条件

  • cinf には有効な証明書情報が含まれている必要がある。
  • modssl_dispatch_ocsp_request を介して HTTP 通信を行うため、OCSP URI が設定されている必要がある。
  • メモリ管理のため、関数内で一時的なサブプールを作成し、終了時に破棄する。

アルゴリズムまたは振る舞い

1. OCSP_REQUEST を生成し、証明書 ID (cinf->cid) および TLS 拡張情報を追加する。 2. 設定された OCSP レスポンダー URI（または mctx->stapling_force_url）を解析する。 3. modssl_dispatch_ocsp_request を呼び出し、OCSP レスポンダーへ HTTP リクエストを送信する。 4. レスポンスが成功した場合、stapling_check_response を実行して内容を検証する。 5. 検証結果（pok）と共に stapling_cache_response を呼び出し、キャッシュを更新する。 6. 最後に一時的なリソース（リクエストオブジェクト等）を解放する。

呼び出し元と呼び出し先

  • **呼び出し元**: modules/ssl/ssl_util_stapling.c:893（文脈上、キャッシュ更新が必要なタイミングで呼び出される）。
  • **呼び出し先**:
  • modssl_dispatch_ocsp_request: OCSP リクエストの送信。
  • stapling_check_response: レスポンスの妥当性検証。
  • stapling_cache_response: キャッシュへの書き込み。
  • OpenSSL API (OCSP_REQUEST_new, OCSP_CERTID_dup, OCSP_response_create 等)。

関連データ構造

  • certinfo: 証明書と OCSP 関連情報を保持する構造体。
  • modssl_ctx_t: SSL モジュールの設定コンテキスト。
  • OCSP_RESPONSE, OCSP_REQUEST: OpenSSL の OCSP プロトコル用構造体。

失敗モードと境界条件

  • **URI 不備**: OCSP URI が存在しない、または http スキーム以外の場合は FALSE を返す。
  • **通信エラー**: レスポンダーからの応答がない場合、mctx->stapling_fake_trylater が有効であれば TRYLATER ステータスを生成し、そうでなければエラーとして終了する。
  • **検証失敗**: stapling_check_response でエラーが検出された場合、pok は FALSE となる。
  • **キャッシュ失敗**: キャッシュへの保存に失敗した場合、ログ出力を行うが関数自体は継続する。

保守担当者向けメモ

  • 本関数は apr_pool_t を使用したメモリ管理を行っているため、サブプールの作成と破棄の整合性に注意すること。
  • MAX_STAPLING_DER（10KB）を超えるレスポンスは切り捨てられる制約がある。
  • OCSP レスポンダーとの通信は同期的に行われるため、タイムアウト設定 (mctx->stapling_responder_timeout) が適切か確認すること。
