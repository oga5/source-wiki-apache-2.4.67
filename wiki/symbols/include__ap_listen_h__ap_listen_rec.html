ap_listen_rec

定義

  • 種別: struct
  • 定義箇所: include/ap_listen.h:39

目的

Apache HTTP Server において、サーバーが接続を待機する各リスナーソケットの情報を保持・管理するための構造体です。

シグネチャまたは形状

struct ap_listen_rec {
    ap_listen_rec *next;
    /* その他、ソケット情報やaccept関数ポインタを含む */
};

※ソース抜粋に基づき、リンクリスト構造の next ポインタの存在が確認されています。

入力と出力

  • 入力: 設定ファイル（Listen ディレクティブ）に基づき生成されます。
  • 出力: MPM（Multi-Processing Module）が接続を受け入れる際の参照先として利用されます。

前提条件と不変条件

  • グローバル変数 ap_listeners を起点とするリンクリストの一部として管理されます。
  • 各インスタンスは、socket(), bind(), listen() を経て有効なリスニング状態にある必要があります。

アルゴリズムまたは振る舞い

1. **初期化**: ap_setup_listeners() により、設定に基づいたソケット生成とバインドが行われます。 2. **実行**: MPM はこの構造体を参照し、accept_func を通じて接続を受け入れます。 3. **複製**: ap_duplicate_listeners() を介して、マルチコア環境向けにリスナーの複製が行われる場合があります。 4. **終了**: ap_close_listeners() または ap_close_listeners_ex() により、保持するソケットがクローズされます。

呼び出し元と呼び出し先

  • 呼び出し元: server/listen.c 内のリスナー管理関数群。
  • 呼び出し先: accept_function 型の関数ポインタ（プロトコルごとの接続処理）。
  • 補足: modules/arch/netware/mod_nw_ssl.c 等のモジュールからも参照されています。

関連データ構造

  • ap_listeners: 全リスナーを保持するグローバルなリンクリスト。
  • accept_function: 接続受け入れ時に呼び出される関数ポインタ型。
  • apr_pool_t: リスナーのライフサイクル管理に使用されるメモリプール。

失敗モードと境界条件

  • ソケットのバインド失敗やリスニング開始の失敗は、サーバーの起動停止要因となります。
  • SO_REUSEPORT のサポート状況により、ap_duplicate_listeners の挙動が異なります。

保守担当者向けメモ

  • ap_listen_rec の accept_func は、プロトコルごとのハンドリングを可能にする重要なフックです。
  • 設定解析時に使用される ap_set_... 系関数は内部的なものであり、外部モジュールから直接呼び出さないでください。
