data_ctx

定義

  • 種別: struct
  • 定義箇所: modules/filters/mod_data.c:40

目的

HTTP レスポンスボディを Base64 エンコードして data URL スキームへ変換する際、フィルタの呼び出し間で状態を保持するために使用されるコンテキスト構造体です。

シグネチャまたは形状

typedef struct data_ctx
{
    unsigned char overflow[3];
    int count;
    apr_bucket_brigade *bb;
} data_ctx;

入力と出力

  • **入力**: フィルタ処理中のレスポンスボディ（バケット単位）。
  • **出力**: Base64 エンコード後のデータが格納される apr_bucket_brigade。

前提条件と不変条件

  • Base64 エンコードは 3 バイト単位で処理されるため、overflow バッファにはエンコード処理しきれなかった端数（1〜2バイト）が保持されます。
  • count は overflow バッファ内に現在保持されている有効なバイト数を表します。

アルゴリズムまたは振る舞い

  • フィルタの初回呼び出し時に生成されます。
  • 入力ブリゲードからデータを読み取り、overflow に残ったデータと連結して 3 バイト単位で Base64 エンコードを行います。
  • 変換後のデータは bb に蓄積され、ap_pass_brigade を介して後続のフィルタへ渡されます。
  • EOS (End of Stream) 到達時に overflow に残存するデータがあれば、それを処理して出力します。

呼び出し元と呼び出し先

  • **呼び出し元**: modules/filters/mod_data.c 内の data_out_filter 関数が f->ctx を通じて本構造体を参照しています。
  • **呼び出し先**: 本構造体自体はデータ保持用であり、関数呼び出しは行いません。

関連データ構造

  • apr_bucket_brigade: 変換後のデータを保持し、フィルタチェーン間で受け渡すために使用されます。

失敗モードと境界条件

  • **サブリクエスト**: 現在の実装ではサブリクエストに対してはフィルタを適用せず、本構造体も使用されません。
  • **メモリ**: APR_BUCKET_BUFF_SIZE を利用したチャンク単位の処理により、メモリ消費を抑制しています。

保守担当者向けメモ

  • 本構造体は mod_data フィルタのステート管理に不可欠です。Base64 変換の境界処理（端数処理）ロジックを変更する際は、overflow と count の整合性に注意してください。
