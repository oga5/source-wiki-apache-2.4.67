request_rec

定義

  • 種別: typedef
  • 定義箇所: modules/lua/lua_request.c:45

目的

Apache HTTP Server における HTTP リクエスト処理の核となるデータ構造であり、mod_lua においては Lua スクリプトから Apache 内部のリクエスト情報へアクセスするためのブリッジとして機能します。

シグネチャまたは形状

Apache HTTP Server のコア構造体であり、リクエストに関連する URI、メソッド、ステータス、ヘッダー、メモリプール（r->pool）などのメタデータを保持します。

入力と出力

  • 入力: Lua スクリプトから Apache2.Request userdata を介して参照されます。
  • 出力: mod_lua の各関数（req_dispatch 等）を通じて、リクエストの属性値や操作結果が Lua スタックへ返されます。

前提条件と不変条件

  • すべてのメモリ割り当ては、当該 request_rec に紐付くプール（r->pool）を使用し、リクエスト終了時に自動的に解放されます。
  • ap_lua_check_request_rec 関数により、Lua スタック上の userdata が有効な request_rec であることが検証される必要があります。

アルゴリズムまたは振る舞い

mod_lua 内では、Lua からのフィールドアクセス（r:field）に対し、req_dispatch メタメソッドがハッシュから対応する C 関数を検索・実行します。これにより、C 言語ベースの request_rec フィールドが Lua のテーブルや型に変換され、読み書きが可能となります。

呼び出し元と呼び出し先

  • 呼び出し元: mod_lua の各 API 関数（lua_read_body, lua_write_body, req_aprtable2luatable 等）が引数として受け取ります。
  • 呼び出し先: Apache コアの各種フィルタやユーティリティ関数（ap_expr_exec, ap_add_*_filter 等）へ渡されます。

関連データ構造

  • conn_rec: 接続情報。
  • server_rec: サーバー設定情報。
  • apr_pool_t: メモリ管理用プール。
  • apr_table_t: ヘッダーや環境変数等のテーブル構造。

失敗モードと境界条件

  • req_newindex を介したフィールド設定には制限があり、すべてのフィールドが書き込み可能ではありません。
  • 正規表現のキャプチャ数には MODLUA_MAX_REG_MATCH による制限が存在します。

保守担当者向けメモ

  • 本構造体は mod_lua だけでなく、Apache 全体のリクエスト処理において中心的な役割を果たすため、変更時は include/httpd.h 等のコア定義との整合性に注意してください。
  • mod_lua におけるメモリ管理は r->pool に完全に依存しているため、Lua 側でのオブジェクト寿命管理と混同しないよう注意が必要です。
