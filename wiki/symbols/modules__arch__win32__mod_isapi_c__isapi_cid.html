isapi_cid

定義

  • 種別: struct
  • 定義箇所: modules/arch/win32/mod_isapi.c:476

目的

ISAPI 拡張のリクエスト処理中における接続状態とコンテキストを保持するための構造体です。Apache のリクエスト情報を ISAPI 形式へマッピングし、リクエストのライフサイクル全体を通じて状態を管理します。

シグネチャまたは形状

struct isapi_cid {
    EXTENSION_CONTROL_BLOCK *ecb;
    isapi_dir_conf           dconf;
    isapi_loaded            *isa;
    request_rec             *r;
    int                      headers_set;
    int                      response_sent;
    PFN_HSE_IO_COMPLETION    completion;
    void                    *completion_arg;
    apr_thread_mutex_t      *completed;
};

入力と出力

  • **入力**: isapi_handler 内で apr_pcalloc を用いてリクエストプール (r->pool) 上に割り当てられます。
  • **出力**: 構造体内の各フィールドは、ISAPI 拡張のコールバック関数（regfnGetServerVariable 等）や、レスポンスヘッダー送信処理（send_response_header）において、リクエストの進捗や状態を追跡するために参照・更新されます。

前提条件と不変条件

  • リクエストプール (r->pool) に紐付いており、リクエスト終了時に自動的に解放されます。
  • isapi_cid は、ISAPI 拡張が実行される間、Apache のリクエスト処理パイプラインと ISAPI 拡張との間の仲介役として一貫性を保つ必要があります。

アルゴリズムまたは振る舞い

  • isapi_handler がリクエストを受け取ると、この構造体を初期化し、Apache の request_rec データを EXTENSION_CONTROL_BLOCK に変換します。
  • headers_set や response_sent フラグにより、レスポンスヘッダーの送信状態を管理します。
  • completed ミューテックスは、非同期エミュレーション（fake_async）において、同期的な待機を実現するために使用されます。

呼び出し元と呼び出し先

  • **呼び出し元**: modules/arch/win32/mod_isapi.c 内の isapi_handler がインスタンスを生成します。
  • **呼び出し先**: 以下の関数が引数として isapi_cid を受け取り、内部状態を操作します。
  • regfnGetServerVariable
  • regfnReadClient
  • send_response_header
  • regfnWriteClient
  • regfnServerSupportFunction

関連データ構造

  • EXTENSION_CONTROL_BLOCK: ISAPI 拡張が期待するリクエスト制御ブロック。
  • isapi_dir_conf: ディレクトリごとの設定情報。
  • isapi_loaded: ロードされた DLL のキャッシュ情報。
  • request_rec: Apache のリクエスト情報。
  • HCONN: isapi_cid へのポインタとして定義された型エイリアス（mod_isapi.h:49）。

失敗モードと境界条件

  • fake_async 設定が有効な場合、高負荷時にミューテックスの待機によりスレッド枯渇が発生する可能性があります。
  • メモリ割り当てはリクエストプールに依存しているため、プールが破棄された後のアクセスは未定義動作となります。

保守担当者向けメモ

  • 本構造体はリクエスト終了時に自動解放されるため、手動でのメモリ管理は不要です。
  • 非同期 I/O 関連のフィールド（completion, completion_arg）は、Apache のアーキテクチャとの不整合により、一部の機能が限定的または擬似的な実装に留まっている点に注意してください。
