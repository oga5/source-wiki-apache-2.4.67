internal_internal_redirect

定義

  • 種別: function
  • 定義箇所: modules/http/http_request.c:528

目的

リダイレクト先 URI に基づき、新しい request_rec 構造体を生成してリクエストコンテキストを再構築する内部ヘルパー関数。

シグネチャまたは形状

static request_rec *internal_internal_redirect(const char *new_uri, request_rec *r);

入力と出力

  • **入力**:
  • new_uri: リダイレクト先の URI 文字列。
  • r: 現在のリクエスト構造体 (request_rec)。
  • **出力**:
  • 成功時: 新しく構築された request_rec 構造体へのポインタ。
  • 失敗時: NULL (再帰制限超過時や ap_post_read_request 失敗時)。

前提条件と不変条件

  • r は有効な request_rec であること。
  • リダイレクトの再帰回数が制限を超えていないこと。
  • 呼び出し前に r->pool が適切に初期化されていること。

アルゴリズムまたは振る舞い

1. **再帰チェック**: ap_is_recursion_limit_exceeded でリダイレクトの深さを確認。 2. **構造体生成**: r->pool を使用して新しい request_rec を割り当て。 3. **コンテキスト継承**: 接続情報、サーバー設定、メソッド、認証情報、プロトコル情報等を r からコピー。 4. **URI 解析**: ap_parse_uri を呼び出し、新しい URI を解析。 5. **フック実行**: ap_run_create_request を実行し、モジュールに新しいリクエストを通知。 6. **フィルタ管理**:

  • サブクエストの場合: 古いリクエスト固有のフィルタを特定し、ap_remove_output_filter で除去。
  • 非サブクエストの場合: リソース固有のフィルタをクリア。
  • update_r_in_filters を呼び出し、フィルタチェーン内の request_rec 参照を新しい構造体に更新。

    • 7. **環境変数設定**: REDIRECT_STATUS を設定。 8. **最終処理**: ap_post_read_request を実行し、成功すれば新しい request_rec を返す。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • modules/http/http_request.c 内の ap_internal_redirect および ap_internal_redirect_handler (ソース内の行番号 777, 803 付近)。
  • **呼び出し先**:
  • ap_is_recursion_limit_exceeded, ap_die, apr_pcalloc, ap_make_method_list, ap_parse_uri, ap_create_request_config, ap_run_create_request, apr_table_make, apr_table_get, apr_table_setn, ap_log_rerror, ap_remove_output_filter, update_r_in_filters, apr_itoa, ap_post_read_request。

関連データ構造

  • request_rec: リクエストのライフサイクルを管理する主要構造体。
  • apr_pool_t: メモリ管理に使用。
  • ap_filter_t: フィルタチェーンの管理に使用。

失敗モードと境界条件

  • **再帰制限**: リダイレクトがループまたは過剰な深さに達した場合、ap_die を経由して HTTP_INTERNAL_SERVER_ERROR を返し NULL を戻す。
  • **リクエスト処理失敗**: ap_post_read_request がエラーを返した場合、そのステータスで ap_die を呼び出し NULL を戻す。
  • **フィルタ不整合**: プロトコルフィルタが誤って除去されないよう、proto_output_filters で停止するガード条件が存在する。

保守担当者向けメモ

  • http_protocol.c との責務分担が曖昧であり、特に header_only 等の設定共有が不十分である。
  • mod_setenvif や mod_unique_id が内部リダイレクトでも動作する必要があるため、この関数名が適切でない可能性があるとの指摘がコード内に存在する。
  • フィルタチェーンの更新 (update_r_in_filters) は、リダイレクト後のリクエストで古い request_rec が参照されないようにするための必須処理である。
