ap_get_client_block

定義

  • 種別: function
  • 定義箇所: modules/http/http_filters.c:1809

目的

リクエストボディを読み取り、指定されたバッファに格納します。ハンドラがリクエストボディを処理するための抽象化レイヤーとして機能します。

シグネチャまたは形状

long ap_get_client_block(request_rec *r, char *buffer, apr_size_t bufsiz)

入力と出力

  • **入力**:
  • r: 現在のリクエスト構造体。
  • buffer: 読み取ったデータを格納するバッファ。
  • bufsiz: バッファのサイズ（読み取り要求バイト数）。
  • **出力**:
  • 成功時: 実際に読み取ったバイト数（long）。
  • 終了時: 0。
  • 失敗時: -1。

前提条件と不変条件

  • r->remaining が負、またはチャンク転送ではない場合に r->remaining が 0 であれば、読み取りは行われず 0 を返します。
  • 内部で apr_bucket_brigade を使用してデータを取得します。
  • 成功時には r->read_length が読み取ったバイト数分増加します。

アルゴリズムまたは振る舞い

1. 読み取り不要な状態か確認し、該当すれば 0 を返します。 2. apr_brigade_create でブリゲードを生成し、ap_get_brigade を呼び出してフィルタチェーンからデータを取得します。 3. AP_FILTER_ERROR またはその他のエラーが発生した場合、接続を AP_CONN_CLOSE に設定し、ブリゲードを破棄して -1 を返します。 4. 取得したブリゲードの末尾に EOS (End of Stream) バケットがある場合、r->remaining を更新して終了状態を記録します。 5. apr_brigade_flatten を使用してブリゲード内のデータを buffer にコピーします。 6. ブリゲードを破棄し、読み取ったバイト数を返します。

呼び出し元と呼び出し先

  • **呼び出し元**: modules/arch/win32/mod_isapi.c、modules/lua/lua_request.c、および modules/http/http_filters.c 内の各所から呼び出されます。
  • **呼び出し先**: ap_get_brigade (フィルタチェーンの実行)、apr_brigade_create、apr_brigade_flatten、apr_brigade_destroy。

関連データ構造

  • request_rec: リクエストの状態（remaining, read_chunked, read_length 等）を保持。
  • apr_bucket_brigade: ストリームデータのバッファリングに使用。

失敗モードと境界条件

  • apr_brigade_create が失敗した場合、接続を閉じて -1 を返します。
  • ap_get_brigade がエラーを返した場合、接続を閉じて -1 を返します。
  • apr_brigade_flatten が失敗した場合、ブリゲードを破棄して -1 を返します。
  • フィルタが正しく実装されていない場合、AP_DEBUG_ASSERT によりデバッグビルドでアサーションエラーが発生する可能性があります。

保守担当者向けメモ

  • ソースコード内のコメントにある通り、本関数はエラーコードを詳細に伝達できないため、使用には注意が必要です。
  • r->remaining の管理は、チャンク転送か否かによってロジックが分岐するため、状態遷移の変更時には注意してください。
