content_type

定義

  • 種別: struct
  • 定義箇所: modules/http/mod_mime.c:102

目的

MIME タイプを構成する主要な要素（メディアタイプ、サブタイプ、および関連するパラメータ）を構造化して保持するために使用されます。

シグネチャまたは形状

typedef struct {
    const char *type;
    apr_size_t type_len;
    const char *subtype;
    apr_size_t subtype_len;
    param *param;
} content_type;

入力と出力

本構造体はデータ保持用であり、関数ではありません。mod_mime 内の解析処理において、文字列から抽出された情報が各フィールドに格納されます。

前提条件と不変条件

  • type および subtype は、メモリプール（apr_pool_t）によって管理された文字列を指すことが想定されます。
  • param フィールドは、関連するパラメータのリストを保持する param 構造体へのポインタです。

アルゴリズムまたは振る舞い

本構造体自体はデータコンテナとして機能します。mod_mime においては、analyze_ct 関数などが Content-Type ヘッダ文字列を解析する際、この構造体に情報を分解・格納することで、後続のパラメータ処理や正規化を容易にします。

呼び出し元と呼び出し先

  • **呼び出し元**: mod_mime.c 内の analyze_ct 関数が、ヘッダ解析の結果を格納するために本構造体を利用する可能性があります。
  • **呼び出し先**: 本構造体自体はデータ型であり、関数呼び出しは行いません。
  • **補足**: ツールによる検索結果（http_protocol.h 等）で言及されている r->content_type は、Apache の request_rec 構造体内の const char * 型フィールドを指しており、本構造体とは直接的な型の一致はありません。

関連データ構造

  • param: 本構造体の param フィールドとして使用される、MIME パラメータを保持する構造体。

失敗モードと境界条件

  • type_len や subtype_len が実際の文字列長と一致しない場合、後続の文字列操作でバッファオーバーランや不正なメモリ参照が発生するリスクがあります。

保守担当者向けメモ

  • 本構造体は mod_mime 内部での解析用中間表現として定義されています。
  • request_rec->content_type と混同しないよう注意してください。あちらは最終的な HTTP ヘッダ値を保持する文字列型です。
  • EBCDIC 環境での動作については、mod_mime の他の箇所と同様に注意が必要です。
