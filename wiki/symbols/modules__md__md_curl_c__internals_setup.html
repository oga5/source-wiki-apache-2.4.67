internals_setup

定義

  • 種別: function
  • 定義箇所: modules/md/md_curl.c:243

目的

md_http_request_t を libcurl の easy_handle に変換し、HTTP リクエスト実行に必要な設定とコンテキストを初期化する。

シグネチャまたは形状

apr_status_t internals_setup(md_http_request_t *req)

入力と出力

  • **入力**: md_http_request_t *req（HTTP リクエスト定義およびメモリプールを含む構造体）
  • **出力**: apr_status_t（成功時は APR_SUCCESS、失敗時はエラーコード）
  • **副作用**: req->internals に md_curl_internals_t 構造体が割り当てられ、libcurl ハンドルが設定される。

前提条件と不変条件

  • req->pool が有効であること。
  • req->http から取得可能な curl インスタンスが存在するか、新規作成可能であること。
  • req->internals は呼び出し前に NULL であることが想定される。

アルゴリズムまたは振る舞い

1. **ハンドル取得**: md_http_get_impl_data を通じて既存の curl ハンドルを再利用するか、curl_easy_init で新規作成する。 2. **コールバック設定**: header_cb、req_data_cb、resp_data_cb を libcurl オプションに登録。 3. **コンテキスト構築**: md_curl_internals_t をプールから確保し、レスポンス用バッファ（apr_brigade）やヘッダーテーブルを初期化。 4. **リクエスト設定**: HTTP メソッド（GET/HEAD/POST等）、タイムアウト、CA 証明書、Unix ソケット、ユーザーエージェント、プロキシ設定を curl_easy_setopt で適用。 5. **ヘッダー変換**: req->headers を curlify_headers を用いて libcurl 形式のリストに変換し、設定。 6. **ログ設定**: デバッグレベルが TRACE4 の場合、curl_debug_log を有効化。 7. **終了処理**: 成功時に req->internals を更新してリターン。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • modules/md/md_curl.c:407
  • modules/md/md_curl.c:532
  • **呼び出し先**:
  • curl_easy_init, curl_easy_reset, curl_easy_setopt (libcurl API)
  • md_http_get_impl_data, md_log_perror, md_log_is_level
  • apr_pcalloc, apr_table_make, apr_brigade_create, apr_table_do
  • header_cb, req_data_cb, resp_data_cb, curl_debug_log, curlify_headers

関連データ構造

  • md_curl_internals_t: リクエストごとの libcurl ハンドルとレスポンス状態を保持。
  • md_http_request_t: HTTP リクエストの抽象定義。
  • curlify_hdrs_ctx: ヘッダー変換時のコンテキスト。

失敗モードと境界条件

  • curl_easy_init が失敗した場合、APR_EGENERAL を返す。
  • req->body_len が負の値の場合、Content-Length は設定されない。
  • ヘッダー変換（curlify_headers）でエラーが発生した場合、リクエスト設定は中断される。

保守担当者向けメモ

  • libcurl ハンドルはスレッドセーフではないため、リクエストごとのプール内で管理される設計となっている。
  • CURLSSLOPT_NO_REVOKE の設定は、プラットフォーム依存のコンパイル時定義に依存する。
  • req->internals の解放は、リクエスト終了時に md_curl_req_cleanup 等で適切に行う必要がある。
