client_entry

定義

  • 種別: struct
  • 定義箇所: modules/aaa/mod_auth_digest.c:121

目的

HTTP Digest 認証におけるクライアントの状態を共有メモリ上で保持するためのデータ構造です。リプレイ攻撃防止のための nonce カウント管理や、クライアントの追跡に使用されます。

シグネチャまたは形状

typedef struct hash_entry {
    unsigned long      key;                     /* エントリのキー */
    struct hash_entry *next;                    /* バケット内の次のエントリ */
    unsigned long      nonce_count;             /* nonce カウントチェック用 */
    char               last_nonce[NONCE_LEN+1]; /* ワンタイム nonce 用 */
} client_entry;

入力と出力

  • 本構造体は client_list（ハッシュテーブル）内の要素として管理されます。
  • nonce_count はリクエストごとに更新され、リプレイ攻撃の検出に使用されます。
  • last_nonce は最後に使用された nonce を保持し、有効性の検証に用いられます。

前提条件と不変条件

  • 共有メモリ（apr_shm）上に配置されるため、アクセスには client_lock によるプロセス間同期が必須です。
  • next ポインタは共有メモリ内の相対アドレスまたは適切なポインタ管理下にある必要があります。

アルゴリズムまたは振る舞い

  • get_client() 関数により、特定のキーに基づいてハッシュテーブルから該当する client_entry が検索されます。
  • add_client() 関数により、新しいクライアント情報がリストに追加されます。
  • gc() 関数（ガベージコレクション）により、メモリ不足時に古いエントリが削除される LRU 的な管理が行われます。

呼び出し元と呼び出し先

  • 呼び出し元: get_client、add_client、および共有メモリの初期化・管理を行う関数群から参照されます。
  • 呼び出し先: 本構造体自体はデータ保持用であり、直接関数を呼び出すことはありません。

関連データ構造

  • client_list: client_entry を要素として持つハッシュテーブル構造。
  • digest_header_rec: パースされた Authorization ヘッダー情報を保持し、本構造体と照合されます。

失敗モードと境界条件

  • 共有メモリが利用できない環境では、nonce カウントチェック等の機能が制限されます。
  • ハッシュテーブルの衝突（HASH_DEPTH）が発生した場合、連結リストを辿る処理が発生します。

保守担当者向けメモ

  • 共有メモリ上のデータであるため、ポインタの扱いやメモリレイアウトには細心の注意を払ってください。
