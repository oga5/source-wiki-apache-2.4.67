ap_process_request_internal

定義

  • 種別: function
  • 定義箇所: server/request.c:187

目的

HTTP リクエスト処理のメインロジックをオーケストレーションし、URI の正規化、翻訳、アクセス制御、型チェック、修正（fixups）を順次実行してリクエストを処理する。

シグネチャまたは形状

int ap_process_request_internal(request_rec *r)

入力と出力

  • **入力**: request_rec *r （処理対象のリクエスト構造体）
  • **出力**: int （処理結果のステータスコード。成功時は OK、エラー時は HTTP_* ステータスコードを返す）

前提条件と不変条件

  • リクエストがファイルサブクエストである場合、一部の処理（translate_name 等）はスキップされる。
  • 認証・認可のキャッシュが有効な場合、親リクエストや先行リクエストの結果を再利用する。
  • セキュリティモデルの根幹であるため、処理順序の変更は認可バイパス等の脆弱性を招く可能性がある。

アルゴリズムまたは振る舞い

1. **正規化**: ap_normalize_path を用いて URI パスを正規化（/./ や /../ の除去等）。 2. **翻訳前処理**: ファイルサブクエスト以外では walk_location_and_if を実行し、pre_translate_name フックを呼び出す。 3. **デコード**: ap_unescape_url_ex を用いて URI をデコード。設定に応じてエンコードされたスラッシュの扱いを制御する。 4. **翻訳とマッピング**: translate_name フックおよび map_to_storage フックを実行。 5. **設定マージ**: walk_location_and_if を再実行し、設定を最新化。 6. **ヘッダー解析**: メインリクエストの場合のみ header_parser フックを実行。 7. **認証・認可**: ap_satisfies に基づき、access_checker、check_user_id、auth_checker フックを順次実行。 8. **最終処理**: type_checker および fixups フックを実行して終了。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • modules/http/http_request.c
  • server/request.c （サブリクエスト生成時）
  • **呼び出し先**:
  • walk_location_and_if, ap_normalize_path, ap_unescape_url_ex, decl_die
  • 各種フック: ap_run_pre_translate_name, ap_run_translate_name, ap_run_map_to_storage, ap_run_post_perdir_config, ap_run_header_parser, ap_run_access_checker, ap_run_access_checker_ex, ap_run_force_authn, ap_run_check_user_id, ap_run_auth_checker, ap_run_type_checker, ap_run_fixups

関連データ構造

  • request_rec: リクエストの状態を保持する主要構造体。
  • core_server_config: サーバー設定。
  • core_dir_config: ディレクトリ単位の設定。

失敗モードと境界条件

  • URI が不正な場合: HTTP_BAD_REQUEST を返す。
  • 認証・認可失敗時: decl_die を経由して適切なエラーコードを返す。
  • 内部エラー: HTTP_INTERNAL_SERVER_ERROR を返す場合がある。
  • ファイルサブクエスト: URI が空であっても許容されるが、通常の翻訳処理はスキップされる。

保守担当者向けメモ

  • サブリクエストのメモリ管理（r->pool）には注意が必要。
  • walk_cache_t による設定マージ結果のキャッシュはパフォーマンスに直結するため、変更時はキャッシュの整合性に留意すること。
  • 認証・認可の順序はセキュリティに直結するため、慎重に扱うこと。
