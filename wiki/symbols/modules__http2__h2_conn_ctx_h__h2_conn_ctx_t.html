h2_conn_ctx_t

定義

  • 種別: struct
  • 定義箇所: modules/http2/h2_conn_ctx.h:40

目的

Apache HTTP/2 モジュールにおいて、クライアントとの直接接続（C1）および内部的なストリーム処理用接続（C2）の両方に付随する状態管理コンテキストを保持します。

シグネチャまたは形状

struct h2_conn_ctx_t {
    const char *id;
    server_rec *server;
    const char *protocol;
    struct h2_session *session;
    struct h2_mplx *mplx;
    struct h2_c2_transit *transit;
#if !AP_HAS_RESPONSE_BUCKETS
    int pre_conn_done;
#endif
    int stream_id;
    apr_pool_t *req_pool;
    const struct h2_request *request;
    struct h2_bucket_beam *beam_out;
    struct h2_bucket_beam *beam_in;
    unsigned input_chunked:1;
    unsigned is_upgrade:1;
    apr_file_t *pipe_in[2];
    apr_pollfd_t pfd;
    int has_final_response;
    apr_status_t last_err;
    apr_off_t bytes_sent;
    apr_uint32_t started;
    apr_time_t started_at;
    apr_uint32_t done;
    apr_time_t done_at;
};

入力と出力

  • **入力**: conn_rec にアタッチされ、h2_conn_ctx_get(c) マクロを通じて取得されます。
  • **出力**: 接続ごとのセッション情報、ストリームID、入出力ビーム（beam_in/beam_out）、および処理状態（started/done）を保持し、モジュール内の各フィルタや処理ルーチンへ提供します。

前提条件と不変条件

  • h2_conn_ctx_get は、HTTP/2 モジュールが有効な接続に対してのみ有効なコンテキストを返すことを前提としています。
  • フィールドには「C1用」「C2用」の用途区分があり、利用箇所に応じて適切に参照する必要があります。

アルゴリズムまたは振る舞い

  • **C1（クライアント接続）**: セッション管理、プロトコル識別、および poll によるソケット監視を行います。
  • **C2（ストリーム処理接続）**: h2_mplx を介した多重化処理、リクエストの処理、および h2_bucket_beam を用いたデータ転送を行います。
  • **状態追跡**: started および done フィールドにより、ストリーム処理のライフサイクルを記録します。

呼び出し元と呼び出し先

  • **呼び出し元**: h2_bucket_beam.c、h2_c1.c、h2_c2.c 等のモジュール内各所から h2_conn_ctx_get を介して参照されます。
  • **呼び出し先**: 本構造体自体はデータ保持用であり、直接的な関数呼び出しは行いません。

関連データ構造

  • conn_rec: 本コンテキストがアタッチされる Apache の接続構造体。
  • h2_session: C1 で管理される HTTP/2 セッション。
  • h2_mplx: C2 で使用される多重化器。
  • h2_bucket_beam: ストリーム間データ転送用バッファ。

失敗モードと境界条件

  • last_err フィールドにより、フィルタ処理中に発生したエラー状態を保持します。
  • pipe_in は C2 接続におけるデータ通知用パイプとして使用されます。

保守担当者向けメモ

  • 本構造体は C1 と C2 で再利用されるため、各フィールドのコメントにある c1 / c2 の注釈を必ず確認してください。
  • C2 のライフサイクルは h2_mplx と密接に関連しているため、デバッグ時は h2_mplx.c を併せて参照してください。
