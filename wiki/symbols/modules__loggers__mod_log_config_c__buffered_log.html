buffered_log

定義

  • 種別: struct
  • 定義箇所: modules/loggers/mod_log_config.c:255

目的

BufferedLogs ディレクティブが有効な場合に、ログファイルの書き込みをバッファリングし、I/O 効率を向上させるための構造体です。

シグネチャまたは形状

typedef struct {
    apr_file_t *handle;
    apr_size_t outcnt;
    char outbuf[LOG_BUFSIZE];
    apr_anylock_t mutex;
} buffered_log;

入力と出力

  • **入力**: ログ書き込み時に outbuf へデータが蓄積されます。
  • **出力**: outcnt が LOG_BUFSIZE に達した際、または flush_log が呼び出された際に、handle を通じてファイルへ書き出されます。

前提条件と不変条件

  • handle は有効な apr_file_t である必要があります。
  • マルチスレッド環境下では、mutex を用いて outbuf へのアクセスとファイル書き込みの排他制御が行われます。

アルゴリズムまたは振る舞い

1. ログ出力要求が発生すると、データが outbuf にコピーされます。 2. outcnt がバッファサイズを超えない限り、メモリ上のバッファに蓄積されます。 3. バッファが満杯になるか、明示的なフラッシュ要求（flush_log）が発生すると、mutex で保護された状態で apr_file_write が実行され、ファイルへ書き出されます。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • modules/loggers/mod_log_config.c 内のログ管理ロジック（all_buffered_logs 配列を介して管理）。
  • apr_pcalloc を用いてインスタンス化されます。
  • **呼び出し先**:
  • flush_log 関数が本構造体を引数として受け取り、バッファのフラッシュを実行します。

関連データ構造

  • apr_file_t: 出力先のファイルハンドル。
  • apr_anylock_t: スレッドセーフな書き込みのためのロック機構。
  • all_buffered_logs: サーバー全体で管理される buffered_log のポインタ配列。

失敗モードと境界条件

  • outbuf が満杯になった際のフラッシュ処理でファイル書き込みエラーが発生した場合、ログデータが消失する可能性があります。
  • mutex の初期化やロック取得に失敗した場合、マルチスレッド環境でのデータ整合性が損なわれるリスクがあります。

保守担当者向けメモ

  • LOG_BUFSIZE の値はコンパイル時に決定されます。
  • all_buffered_logs 配列を介して管理されているため、モジュールの終了処理時にすべてのバッファがフラッシュされていることを確認してください。
