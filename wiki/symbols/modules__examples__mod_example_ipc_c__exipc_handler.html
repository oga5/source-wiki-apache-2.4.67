exipc_handler

定義

  • 種別: function
  • 定義箇所: modules/examples/mod_example_ipc.c:221

目的

HTTP リクエスト（example_ipc ハンドラ）に応答して共有カウンタをインクリメントし、現在のカウンタ値とプロセス ID を含む HTML レスポンスを生成する。

シグネチャまたは形状

static int exipc_handler(request_rec *r)

入力と出力

  • **入力**: request_rec *r（Apache リクエスト構造体）。
  • **出力**: int（Apache のステータスコード OK または DECLINED）。

前提条件と不変条件

  • r->handler が "example_ipc" である必要がある。
  • 共有メモリ exipc_shm およびグローバルミューテックス exipc_mutex が初期化済みであること。
  • ロック取得の成否に関わらず、最終的に OK を返してリクエスト処理を終了する。

アルゴリズムまたは振る舞い

1. **ハンドラ確認**: r->handler が "example_ipc" でない場合は DECLINED を返す。 2. **ロック取得**: apr_global_mutex_trylock を最大 MAXCAMP 回試行する。

  • EBUSY の場合は CAMPOUT 時間待機して再試行。
  • ENOTIMPL の場合は apr_global_mutex_lock を使用してブロックする。

    • 3. **カウンタ更新**: ロック取得に成功した場合、共有メモリ exipc_shm から exipc_data を取得し、カウンタをインクリメントする。 4. **レスポンス生成**: r->header_only が偽の場合、カウンタ値、プロセス ID、ロック待機時間を含む HTML を出力する。 5. **ロック解放**: ロックを取得していた場合は apr_global_mutex_unlock を実行する。

呼び出し元と呼び出し先

  • **呼び出し元**: Apache のハンドラフック（ap_hook_handler により登録）。
  • **呼び出し先**:
  • apr_global_mutex_trylock, apr_global_mutex_lock, apr_global_mutex_unlock
  • apr_shm_baseaddr_get
  • ap_rputs, ap_rprintf
  • apr_sleep
  • ap_log_error

関連データ構造

  • exipc_data: 共有メモリ内に配置されるカウンタ保持用構造体。
  • request_rec: Apache のリクエスト情報構造体。

失敗モードと境界条件

  • **ロック取得失敗**: MAXCAMP 回試行してもロックが取得できない場合、カウンタを更新せずにその旨をレスポンスに含める。
  • **エラーログ**: ロック取得時のエラーは ap_log_error を通じて記録される。

保守担当者向けメモ

  • 本関数は教育用であり、高負荷環境でのパフォーマンス最適化は考慮されていない。
  • ロック取得の試行ロジックは、trylock がサポートされていないプラットフォームへの互換性を維持するために意図的に冗長に記述されている。
