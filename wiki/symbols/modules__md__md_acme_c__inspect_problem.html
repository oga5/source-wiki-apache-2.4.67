inspect_problem

定義

  • 種別: function
  • 定義箇所: modules/md/md_acme.c:170

目的

ACME サーバーから返された HTTP レスポンスを解析し、エラー内容（Problem）を特定して適切な Apache ステータスコードへ変換する。

シグネチャまたは形状

static apr_status_t inspect_problem(md_acme_req_t *req, const md_http_response_t *res)

入力と出力

  • **入力**:
  • req: ACME リクエストのコンテキスト構造体。
  • res: サーバーからの HTTP レスポンス。
  • **出力**:
  • apr_status_t: 判定されたステータスコード。成功時は APR_SUCCESS、エラー時は APR_EAGAIN（再試行推奨）や APR_EINVAL、APR_EACCES 等が返される。

前提条件と不変条件

  • req および res は有効なポインタであること。
  • req->resp_hdrs にレスポンスヘッダーが格納されていること。

アルゴリズムまたは振る舞い

1. レスポンスの Content-Type を確認し、application/problem+json であれば RFC 7807 形式のエラーとして解析する。 2. JSON 内の type フィールドを抽出し、problem_status_get を用いて内部ステータスコードにマッピングする。 3. 解析したエラー情報を req->result に記録し、ログを出力する。 4. JSON 形式でない、または解析できない場合は、HTTP ステータスコード（400, 401, 403, 404）に基づき、対応する apr_status_t を返す。 5. 未知のステータスコードの場合は APR_EGENERAL を返す。

呼び出し元と呼び出し先

  • **呼び出し元**: modules/md/md_acme.c:323（md_acme_req_send 内のレスポンス処理フロー）
  • **呼び出し先**:
  • md_util_parse_ct
  • md_json_read_http
  • problem_status_get
  • md_json_gets
  • md_result_problem_set
  • md_log_perror
  • md_result_printf

関連データ構造

  • md_acme_req_t: リクエストの状態と結果を保持。
  • md_http_response_t: HTTP レスポンスの生データ。
  • md_json_t: 解析されたエラー詳細（Problem JSON）。

失敗モードと境界条件

  • Content-Type が application/problem+json でない場合、HTTP ステータスコードによるフォールバック判定が行われる。
  • problem_status_get が APR_EAGAIN を返す場合、呼び出し元で再試行ロジックがトリガーされる。
  • JSON 解析に失敗した場合や、期待するフィールドが存在しない場合は、HTTP ステータスコードに基づくデフォルトの処理へ移行する。

保守担当者向けメモ

  • 本関数は ACME プロトコルのエラーハンドリングの要であり、サーバー側の仕様変更（新しいエラー型の追加など）に応じて problem_status_get のマッピングを更新する必要がある。
  • req->p（リクエストプール）を使用してメモリを確保しているため、リクエスト終了時のプール破棄によりメモリリークは防止される。
