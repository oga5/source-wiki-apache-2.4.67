test/test_limits.c

概要

サーバーの境界値処理能力を検証するための負荷試験ツールです。HTTPリクエストの各要素（リクエスト行、ヘッダー、ボディ）に対して、意図的に異常な長さのデータを送信し、サーバーの堅牢性をテストします。

モジュール内での役割

test モジュール内で、ネットワークプロトコル解析の限界値を検証するストレステスト用ユーティリティとして機能します。

責務

  • 指定されたターゲット（ホスト・ポート）へのTCP接続の確立。
  • HTTPリクエストの構成要素（リクエスト行、フィールド数、フィールドサイズ、ボディ）を過剰に肥大化させたペイロードの生成。
  • サーバーからの応答を標準出力へ転送し、接続状態を監視する。

重要なシンボル

  • TEST_LONG_REQUEST_LINE: リクエスト行を異常に長くするテストモード。
  • TEST_LONG_REQUEST_FIELDS: ヘッダーフィールドの数を過剰に増やすテストモード。
  • TEST_LONG_REQUEST_FIELDSIZE: 単一のヘッダーフィールド値を過剰に長くするテストモード。
  • TEST_LONG_REQUEST_BODY: ボディサイズを偽装し、過剰なデータを送り込むテストモード。

制御フローとデータフロー

1. **初期化**: getopt を用いてコマンドライン引数からターゲット情報とテストタイプを解析。 2. **接続**: gethostbyname でホストを解決し、socket および connect でサーバーへTCP接続。 3. **攻撃実行**: testtype に応じて、fprintf をループさせ、サーバーに対して異常な長さのHTTPリクエストをストリーム送信。 4. **応答処理**: read を用いてサーバーからの応答を読み取り、標準出力へ書き出すことで、サーバーが接続を維持しているか、あるいは切断したかを確認。

関連コード

  • test/test_parser.c: 本ツールで送信されるような異常なリクエストを、サーバー側でどのように解析・処理するかを検証するコード。
  • test/test-writev.c: ネットワークI/Oの別の側面（ベクトル書き込み）をテストする関連ツール。

読者向けメモ

  • 本コードは1998年当時のDoS攻撃手法を再現した歴史的なテストツールです。
  • 現代のサーバー環境に対して実行すると、セキュリティ保護機能（WAFやレート制限）により即座に遮断される可能性があります。
  • apr_ssize_t が使用されていますが、これはApache Portable Runtime (APR) の型定義に依存しており、コンパイル時には適切な環境設定が必要です。
  • サーバーのバッファオーバーフローやメモリ枯渇に対する脆弱性を確認するための「ネガティブテスト」として利用されます。
