test/modules/proxy/test_02_unix.py

概要

本ファイルは、Apache HTTP Server の mod_proxy における Unix Domain Socket (UDS) 経由のプロキシ機能の動作検証を行うためのテストスイートです。

モジュール内での役割

test/modules/proxy モジュールの一部として、HTTP/HTTPS 経由のプロキシテストに続き、UDS をバックエンドに使用した際のルーティングおよびセキュリティ制約を検証します。

責務

  • **UDS モックサーバーの提供**: UDSFaker クラスにより、UDS をリッスンし固定の HTTP レスポンスを返す簡易サーバーを構築する。
  • **プロキシ設定の検証**: ProxyPass での UDS 指定（unix:...|http://...）が、リバースプロキシおよびフォワードプロキシ環境で正しく機能するかを確認する。
  • **セキュリティ境界のテスト**: 外部からの不正な UDS パス指定が適切に拒否されることを確認する。

重要なシンボル

  • UDSFaker: Unix Domain Socket を使用してリクエストを待ち受け、テスト用の固定レスポンスを返すスレッドベースのモックサーバー。
  • TestProxyUds: テストクラス。pytest フィクスチャを使用して Apache の仮想ホスト（vhost）を構築し、UDS 経由の通信をテストする。
  • test_proxy_02_003: UDS パスを直接リクエストに含める攻撃的シナリオをシミュレートし、エラーコード（400/500）が返ることを検証するテストケース。

制御フローとデータフロー

1. **セットアップ**: _class_scope フィクスチャが UDSFaker を起動し、Apache の設定ファイル（HttpdConf）を生成して UDS をバックエンドに持つ vhost を定義・再起動する。 2. **通信**: 各テストメソッドが curl や openssl を使用して、定義された vhost に対してリクエストを送信する。 3. **検証**: モックサーバーが受け取ったリクエストに対し、期待される JSON レスポンスや HTTP ステータスコードが返るかをアサーションで確認する。 4. **クリーンアップ**: テスト終了時に faker.stop() を呼び出し、ソケットとスレッドを解放する。

直接依存

  • os, re, socket, threading: UDS モックサーバーの実装に使用。
  • pytest: テストフレームワークおよびフィクスチャ管理。
  • pyhttpd.conf.HttpdConf: Apache 設定生成用。
  • pyhttpd.result.ExecResult: コマンド実行結果の解析用。

関連コード

  • test/modules/proxy/env.py: テスト環境の共通設定やディレクトリパスの管理。
  • test/modules/proxy/test_01_http.py: HTTP プロキシの基本動作テスト（比較対象）。

読者向けメモ

  • 本テストは mod_proxy の UDS サポートに依存しています。実行環境で Apache が UDS をサポートしている必要があります。
  • test_proxy_02_003 では、意図的にエラーを発生させるため、env.httpd_error_log.ignore_recent を使用してログのノイズを抑制しています。
  • UDSFaker は非常にシンプルな実装であり、複雑な HTTP プロトコル解析は行いません。特定のテストケースに必要な最小限の応答のみを返します。
