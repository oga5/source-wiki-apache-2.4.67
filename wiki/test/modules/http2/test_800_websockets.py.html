test/modules/http2/test_800_websockets.py

概要

HTTP/2 上での WebSocket プロトコル（RFC 8441）のサポートを検証するための統合テストスイートです。Apache HTTP Server (mod_http2) が CONNECT メソッドを介して WebSocket トンネルを正しく処理できるかをテストします。

モジュール内での役割

test/modules/http2 モジュールの一部として、HTTP/2 の高度な機能である WebSocket プロキシ機能の品質保証を担います。外部の ws_server.py と連携し、クライアント側のテストツール h2ws を駆動して通信を検証します。

責務

  • **プロトコル検証**: CONNECT リクエストの妥当性（ヘッダーの有無、プロトコル指定）の確認。
  • **トンネル機能のテスト**: 正常な WebSocket 通信、PING/PONG、データ転送の整合性確認。
  • **エラーハンドリング**: 不正なリクエストや存在しないパスに対する適切なステータスコード（400, 404, 501, 502）の返却確認。
  • **負荷・タイミングテスト**: フレーム長や遅延を変化させた際のデータ転送スループットとタイムアウト挙動の検証。

重要なシンボル

  • ws_run: テスト用クライアント h2ws を実行し、結果（終了コード、ログ、受信フレーム）を収集するヘルパー関数。
  • TestWebSockets: テストクラス。pytest フィクスチャを用いて Apache の設定と WebSocket サーバーのライフサイクルを管理。
  • ws_server: テストクラス内で起動されるバックエンドの WebSocket サーバー。
  • H2Conf: Apache の設定を動的に生成・適用するためのクラス。

制御フローとデータフロー

1. **セットアップ**: _class_scope フィクスチャが H2Conf を用いて Apache を設定し、H2WebSockets on を有効化して再起動します。 2. **サーバー起動**: ws_server フィクスチャが ws_server.py をサブプロセスとして起動し、テスト対象のバックエンドを提供します。 3. **テスト実行**: 各 test_h2_800_* メソッドが ws_run を呼び出します。

  • ws_run は h2ws をサブプロセスで起動し、標準入力を通じて WebSocket フレームを送信します。
  • 出力は一時ファイルに書き出され、終了後に WsFrameReader が解析して WsFrame オブジェクトのリストを返します。

    • 4. **検証**: 返されたステータスコードやフレームの内容を assert で検証します。

直接依存

  • pytest: テストフレームワーク。
  • websockets: Python の WebSocket ライブラリ（バージョン 10.4 以上が必要）。
  • pyhttpd: Apache テスト用ユーティリティ（ExecResult, WsFrameReader 等）。
  • subprocess: h2ws および ws_server.py の実行に使用。

関連コード

  • test/modules/http2/ws_server.py: テスト用のバックエンドサーバー実装。
  • test/modules/http2/env.py: テスト環境の構成と Apache の制御を行う基盤コード。

読者向けメモ

  • 本テストは httpd 2.4.60 以上を要求します。
  • h2ws クライアントがビルドされていない場合、テストは失敗します。
  • ws_run はデッドロックを避けるため、出力をファイル経由で取得する設計になっています。デバッグ時は pytest -o log_cli=true を使用して詳細なログを確認してください。
