test/modules/http2/mod_h2test/mod_h2test.c

概要

mod_h2test.c は、Apache HTTP Server の HTTP/2 テストスイート用の診断・検証用モジュールです。主に HTTP/2 プロトコルの挙動（ストリーム制御、トレーラー、エラー処理、遅延など）を検証するためのカスタムハンドラを提供します。

モジュール内での役割

本モジュールは、実際の運用環境ではなく、テスト環境において特定のネットワーク挙動をシミュレートする「テスト用エンドポイント」として機能します。mod_h2 の動作を検証するための補助的な役割を担います。

責務

  • **リクエストハンドリング**: 特定のハンドラ名（h2test-echo, delay, trailer, error）に基づき、リクエストを処理する。
  • **挙動シミュレーション**: 意図的な遅延、エラー発生、トレーラーヘッダーの付与、ストリームの強制終了などを実行する。
  • **設定と初期化**: モジュールのフック登録および子プロセス初期化時の状態管理。

重要なシンボル

モジュール定義とフック

  • h2test_hooks: Apache のフックシステムに各ハンドラを登録するエントリーポイント。
  • AP_DECLARE_MODULE(h2test): モジュールのメタデータ定義。

リクエストハンドラ

  • h2test_echo_handler: リクエストボディをそのままレスポンスとして返す（エコー）。遅延や強制終了のパラメータをサポート。
  • h2test_delay_handler: 指定された遅延を挟みながらレスポンスを分割送信する。
  • h2test_trailer_handler: トレーラーヘッダーを付与したレスポンスを生成する。
  • h2test_error_handler: 指定された HTTP ステータスやエラー種別（タイムアウト、リセット等）をシミュレートする。

ユーティリティ

  • duration_parse: 文字列形式の期間（例: "10s", "50ms"）を apr_interval_time_t に変換する。
  • status_from_str: 文字列から apr_status_t エラーコードをマッピングする。

制御フローとデータフロー

1. **リクエスト受信**: 各ハンドラは request_rec を受け取り、r->handler を確認して自身の担当か判定します。 2. **パラメータ解析**: r->args を解析し、テスト用の制御パラメータ（遅延時間、エラーフラグ等）を抽出します。 3. **ブリゲード操作**: apr_bucket_brigade を使用してレスポンスを構築します。ap_pass_brigade を介してフィルタチェーンへデータを流し込みます。 4. **エラー処理**: 処理中にエラーが発生した場合、ap_bucket_error_create を使用してエラーバケットを挿入し、クライアントへ通知します。 5. **クリーンアップ**: apr_pool を利用したメモリ管理により、リクエスト終了時にバケットや一時データが自動的に解放されます。

読者向けメモ

  • **メモリ管理**: すべてのメモリ割り当ては r->pool に紐付いており、リクエスト終了時に自動的に解放されます。明示的な free は不要です。
  • **エラーパス**: goto cleanup パターンが多用されています。これは apr_brigade の操作中にエラーが発生した際、適切にリソースを解放しつつ終了するための定石です。
  • **制約**: 本モジュールはテスト専用です。プロダクション環境での使用は想定されておらず、意図的にエラーを発生させる機能が含まれているため、セキュリティ上のリスクがあります。
  • **省略された詳細**: h2test_post_config 内の mod_h2test_init_counter は、モジュールの二重初期化を防ぐための単純なフラグであり、本質的なロジックはハンドラ群に集中しています。
