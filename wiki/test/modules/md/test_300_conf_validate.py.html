test/modules/md/test_300_conf_validate.py

概要

本ファイルは、Apache HTTP Server の mod_md モジュールにおける設定ディレクティブの妥当性検証および構成ルールのテストスイートです。MDomain 定義、仮想ホスト（vhost）との整合性、証明書関連の設定、および設定ファイルの破損に対する回復挙動を網羅的に検証します。

モジュール内での役割

test/modules/md モジュール群において、設定の静的解析および Apache 起動時の構成チェックを担当する統合テスト層です。MDConf クラスを介して Apache の設定を動的に生成し、期待される成功または失敗（エラーログの確認を含む）を検証します。

責務

  • **設定の妥当性検証**: MDomain 定義の重複、vhost との不整合、無効なパラメータ指定の検出。
  • **構成ルールの検証**: MDMatchNames や MDRequireHttps 等のディレクティブが正しく適用されるかの確認。
  • **エラーハンドリング**: 設定エラー時の Apache 起動失敗と、特定のログコード（AHxxxxx）の出力確認。
  • **自己修復機能のテスト**: 設定ファイル（JSON）が破損した際の Apache の挙動（修復または停止）の検証。

重要なシンボル

テストクラス

  • TestConf: pytest を用いたテストスイートの基底クラス。MDTestEnv を利用して環境を制御します。

設定操作クラス

  • MDConf: Apache 設定を構築するためのヘルパー。add() や add_vhost() で設定を生成し、install() で適用します。

環境制御

  • env: MDTestEnv インスタンス。Apache の起動・停止、設定ストアのパージ、ログの監視（httpd_error_log）を提供します。

制御フローとデータフロー

1. **セットアップ**: _class_scope フィーチャにより、クラス単位で ACME サーバーの起動とストアの初期化が行われます。 2. **設定生成**: 各テストメソッド内で MDConf をインスタンス化し、text 引数や add_vhost() を通じて設定文字列を構築します。 3. **適用と検証**:

  • conf.install() を呼び出し、Apache 設定を書き込みます。
  • env.apache_restart() または env.apache_fail() を呼び出し、戻り値で成否を判定します。
  • env.httpd_error_log.ignore_recent() を使用して、期待されるエラーログを無視（確認）します。

    • 4. **破損テスト**: test_md_300_030/031 では、ファイルシステム上の JSON ファイルを直接書き換えることで、mod_md のロード時の堅牢性を検証します。

直接依存

  • pytest: テストフレームワーク。
  • .md_conf.MDConf: 設定生成用ユーティリティ。
  • .md_env.MDTestEnv: テスト環境の抽象化レイヤー。

関連コード

  • test/modules/md/md_conf.py: MDConf クラスの実装詳細。
  • test/modules/md/md_env.py: テスト環境のセットアップと Apache 制御の実装。
  • test/modules/md/test_310_conf_store.py: 設定の永続化とストア管理に関する追加テスト。

読者向けメモ

  • **ログの無視**: env.httpd_error_log.ignore_recent() は、テストが意図的に発生させたエラーログを「確認済み」として処理するために必須です。これがないと、テスト実行時にログの不一致で警告が出ることがあります。
  • **メモリと永続化**: 本テストはファイルシステム上の設定ファイル（store_dir）を直接操作します。env.purge_store() や env.clear_store() を呼び出さないと、テスト間で設定が汚染される可能性があります。
  • **網羅性**: 本ファイルは mod_md の設定ディレクティブの多くを網羅していますが、ACME プロトコル自体の通信フローは test_502_acmev2_drive.py 等の別ファイルで扱われます。
  • **制約**: test_md_300_031 での「破損したストア」のテストは、Apache の起動失敗を期待するものであり、ファイルシステムの状態を破壊するため、テスト終了後のクリーンアップが重要です。
