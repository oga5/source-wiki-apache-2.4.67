test/modules/md/test_502_acmev2_drive.py

概要

本ファイルは、Apache HTTP Server の mod_md モジュールにおける ACMEv2 プロトコルを用いた証明書取得（drive）プロセスの統合テストスイートです。ACME サーバーとの通信、証明書のライフサイクル管理、設定変更に伴う再取得、およびエラーハンドリングの妥当性を検証します。

モジュール内での役割

test/modules/md モジュール群の一部として、a2md コマンドラインツールを介した mod_md の主要な機能（証明書の自動取得・更新）をブラックボックステストする役割を担います。

責務

  • **プロトコル検証**: ACMEv2 プロトコルを通じた証明書発行フローの正常系・異常系のテスト。
  • **設定整合性**: MDConf を用いた Apache 設定と mod_md の状態管理の同期確認。
  • **証明書ライフサイクル**: 初回取得、更新（renew）、強制更新、および証明書アーカイブの検証。
  • **環境依存テスト**: プロキシ経由の通信、HTTP/HTTPS リダイレクト、およびヘッダー上書きの挙動確認。

重要なシンボル

テストクラス

  • TestDrivev2: ACMEv2 の駆動プロセスを検証するメインクラス。pytest フィクスチャによりクラスおよびメソッド単位で環境を初期化します。

フィクスチャとセットアップ

  • _class_scope: クラス単位で ACME サーバーを起動し、Apache 設定をインストール。
  • _method_scope: メソッド単位で mod_md のストアをクリアし、テスト対象ドメインを再設定。

ヘルパーメソッド

  • _prepare_md: 指定ドメインの MD（Managed Domain）を追加し、連絡先と利用規約（ToS）を設定する共通処理。
  • _check_account_key: アカウント秘密鍵の暗号化状態と整合性を検証。
  • _write_res_file: テスト用のドキュメントルートにリソースファイルを配置。

制御フローとデータフロー

1. **初期化**: _class_scope が ACME サーバーを起動し、MDConf で Apache を構成。 2. **実行**: 各テストケースが env.a2md(["drive", ...]) を呼び出し、mod_md のバックエンド処理をトリガー。 3. **検証**:

  • a2md の終了コードと標準エラー出力を確認。
  • MDCertUtil を用いて生成された証明書のシリアル番号や鍵長を解析。
  • env.get_meta を通じて、Apache が提供する証明書やリダイレクト設定が期待通りかを確認。

    • 4. **クリーンアップ**: 各テスト終了時に env.clear_store() が呼ばれ、次回のテストに影響を与えないよう状態をリセット。

直接依存

  • pytest: テストフレームワーク。
  • pyhttpd.certs.CertificateSpec: 証明書仕様の定義。
  • test.modules.md.md_conf.MDConf: Apache 設定管理。
  • test.modules.md.md_cert_util.MDCertUtil: 証明書解析ユーティリティ。
  • test.modules.md.md_env.MDTestEnv: テスト環境の抽象化レイヤー。

関連コード

  • test/modules/md/test_740_acme_errors.py: ACME エラーハンドリングの詳細な検証。
  • test/modules/md/test_001_store.py: mod_md のストア構造の基礎テスト。

読者向けメモ

  • **前提条件**: MDTestEnv.has_a2md() および has_acme_server() が真である環境でのみ実行されます。
  • **メモリとライフタイム**: a2md は外部プロセスとして実行されるため、Python 側のメモリ管理よりも、ファイルシステム上のストア（store_challenges 等）の整合性が重要です。
  • **エッジケース**: test_md_502_109 や 110 では、mod_md と mod_alias のリダイレクト優先順位という複雑な Apache 設定の相互作用を検証しています。
  • **省略箇所**: 本ファイルは drive プロセスの統合テストに特化しており、個別の ACME チャレンジ（dns-01 等）の内部ロジックについては、それぞれの専用テストファイルに委譲されています。
