httpd-2.4.67

このシステムは何か

httpd-2.4.67 は、世界で最も広く利用されているオープンソースの HTTP サーバーソフトウェアです。モジュール化されたアーキテクチャを持ち、多様なプラットフォーム（Unix, Windows, NetWare, OS/2等）での動作と、MPM（Multi-Processing Module）による柔軟な並行処理モデルの切り替えを特徴とします。

アーキテクチャ概観

本システムは、コアサーバー（server）を基盤とし、リクエスト処理パイプライン（フィルタチェーンとフック）を通じて各モジュールが機能を拡張する疎結合な設計です。

  • **コア**: プロセス/スレッド管理（MPM）、設定解析、リクエストライフサイクル管理。
  • **モジュール**: 認証（aaa）、キャッシュ（cache）、プロキシ（proxy）、WebDAV（dav）、フィルタ（filters）、SSL/TLS（ssl）など、機能ごとにディレクトリ単位で分離されています。
  • **プラットフォーム抽象化**: os ディレクトリ配下で OS 固有の差異（プロセス生成、権限管理、文字コード）を吸収しています。

主要モジュール

  • server: サーバーの心臓部。MPM、設定、プロトコル処理の基盤。
  • modules/http: HTTP プロトコル処理のコア。
  • modules/proxy: プロキシおよびロードバランシング機能。
  • modules/aaa: 認証・認可フレームワーク。
  • modules/cache: 透過的なキャッシュサブシステム。
  • modules/ssl: OpenSSL を統合したセキュア通信機能。
  • os: 各プラットフォーム固有の抽象化レイヤー。

実行時ライフサイクル

1. **起動・初期化**: main が設定を読み込み、register_hooks を通じて各モジュールがフックを登録。 2. **プロセス/スレッド生成**: MPM（event, worker, prefork 等）が make_child や create_listener_thread を通じて処理主体を生成。 3. **リクエスト処理**:

  • process_socket で接続を受け入れ、http_request.c がリクエストを解析。
  • フィルタチェーン（ap_filter_t）を介してデータが処理され、ハンドラがレスポンスを生成。

    • 4. **終了**: シグナル受信により just_die や clean_child_exit が呼び出され、リソースを解放して終了。

中核データ構造

  • request_rec: 単一リクエストの全状態を保持するコア構造体。
  • conn_rec: 接続固有のコンテキスト（IP、フィルタチェーン等）。
  • server_rec: バーチャルホストを含むサーバー設定の基盤。
  • module: モジュールのエントリポイントを定義する構造体。
  • apr_pool_t: リソースのライフサイクルを管理するメモリプール。
  • ap_filter_t: フィルタチェーンの各ノード。

横断的トピック

  • **メモリ管理**: apr_pool_t を用いた階層的なメモリ管理と自動解放。
  • **フックシステム**: ap_hooks.h に基づく、モジュール間連携のための疎結合な拡張基盤。
  • **MPM**: プロセス/スレッドモデルの抽象化（event, worker, prefork 等）。
  • **フィルタチェーン**: apr_bucket_brigade を用いたストリーム処理パイプライン。
  • **設定管理**: ap_conf_vector_t を用いた階層的な設定マージ。

推奨読書順

1. include/httpd.h (基本構造と主要構造体の理解) 2. server/main.c (起動から初期化のフロー) 3. server/request.c (リクエスト処理のライフサイクル) 4. include/ap_hooks.h (モジュール連携の基盤) 5. include/util_filter.h (フィルタチェーンの仕組み) 6. server/mpm/event/event.c (並行処理モデルの例) 7. docs/manual/developer/mod_example_1.c (モジュール開発の最小構成)
