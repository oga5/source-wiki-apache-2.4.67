httpd-2.4.67

このシステムは何か

httpd-2.4.67 は、モジュール拡張可能なアーキテクチャを持つ、世界で最も広く利用されているオープンソースの HTTP サーバーです。本プロジェクトは、リクエスト処理パイプライン、マルチプロセッシングモデル（MPM）、および動的モジュールロード機構を核として構成されています。

アーキテクチャ概観

Apache httpd は、コアサーバーと、その機能を拡張する多数のモジュールから構成されます。

  • **コア**: プロセス/スレッド管理（MPM）、設定解析、リクエスト処理のライフサイクル管理を担当。
  • **フィルタチェーン**: 入出力データ（バケットブリゲード）を順次加工するパイプライン。
  • **フックシステム**: モジュールがサーバーの各処理フェーズ（認証、翻訳、ハンドラ実行など）に介入するための疎結合な連携基盤。
  • **プロバイダーモデル**: 認証、キャッシュ、DB接続などの機能を抽象化し、バックエンド実装を差し替え可能にする機構。

主要モジュール

  • **server/mpm**: プロセス/スレッドモデル（event, worker, prefork等）の管理。
  • **modules/http**: HTTP プロトコル解析およびフィルタリングの基盤。
  • **modules/proxy**: ロードバランシングおよびバックエンド転送機能。
  • **modules/aaa**: 認証・認可フレームワーク。
  • **modules/cache**: キャッシュポリシーエンジンとストレージプロバイダー。
  • **modules/ssl**: OpenSSL を統合したセキュア通信基盤。
  • **modules/md**: ACME プロトコルによる証明書自動管理。

実行時ライフサイクル

1. **初期化**: main が設定ファイルを解析し、モジュールをロード。post_config フックで各モジュールが初期化される。 2. **MPM 起動**: child_main または worker_thread が生成され、リスナーソケットが監視される。 3. **リクエスト処理**:

  • process_connection が接続を受け入れ、http_request.c がリクエストを解析。
  • フィルタチェーン（ap_filter_t）を介してデータが処理される。
  • handler がコンテンツを生成し、レスポンスを返却。

    • 4. **終了**: シグナル受信により shutdown_pending や workers_may_exit がセットされ、クリーンアップ処理を経てプロセスが終了する。

中核データ構造

  • **request_rec**: リクエストの全状態を保持する中心的な構造体。
  • **conn_rec**: 接続固有のコンテキスト（IP、ソケット、フィルタチェーン）。
  • **server_rec**: 仮想ホスト設定を含むサーバー単位のコンテキスト。
  • **apr_bucket_brigade**: フィルタチェーンを流れるデータストリームの連結リスト。
  • **module**: モジュールのエントリポイントとフックを定義する構造体。

横断的トピック

  • **メモリ管理**: apr_pool_t を用いた階層的なメモリ管理。リクエストや接続のライフサイクルに紐付いた自動解放が基本。
  • **フックシステム**: ap_hooks.h に基づく、モジュール間連携の疎結合な基盤。
  • **MPM**: プロセス/スレッドモデルの抽象化。event, worker, prefork 等の切り替えが可能。
  • **フィルタチェーン**: util_filter.h に基づく、データストリームの加工パイプライン。

推奨読書順

1. include/httpd.h (基本構造とデータ構造) 2. server/main.c (サーバーの起動とライフサイクル) 3. server/request.c (リクエスト処理のパイプライン) 4. include/ap_hooks.h (モジュール連携の基盤) 5. include/util_filter.h (フィルタチェーンの仕組み) 6. server/mpm/event/event.c (並行処理モデルの例)

未解決の問い

  • 各 MPM（event, worker, prefork）における child_main の詳細な差異と、シグナルハンドリングの統合的な挙動の精緻化。
  • apr_pool_t の階層構造における、特定のモジュール（mod_lua 等）でのメモリリーク防止策の検証。
