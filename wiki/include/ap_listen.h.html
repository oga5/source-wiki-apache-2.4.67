include/ap_listen.h

概要

ap_listen.h は、Apache HTTP Server におけるネットワークリスナーの管理を定義するヘッダーファイルです。MPM（Multi-Processing Module）が接続を受け付けるためのソケット設定、バインド、およびリスニング状態の管理を行うためのデータ構造と API を提供します。

モジュール内での役割

Apache コアの一部として、設定ファイルで指定された Listen ディレクティブに基づき、サーバーが待機すべき全ソケットのリストを保持・管理します。MPM はこのリストを参照して、各プロセス/スレッドがどのソケットから接続を受け入れるかを決定します。

責務

  • リスナー設定の保持: ap_listen_rec 構造体によるソケット情報の管理。
  • ソケットのライフサイクル管理: ソケットの生成、バインド、リスニングの開始、およびクローズ処理。
  • 設定の統合: Listen 関連のディレクティブ（Listen, ListenBacklog 等）の解析と適用。
  • マルチコア最適化: SO_REUSEPORT を利用したリスナーのバケット分割（ap_duplicate_listeners）。

重要なシンボル

  • ap_listen_rec: リスナーソケットの情報を保持する主要な構造体。
  • ap_listeners: 全リスナーを保持するグローバルなリンクリスト。
  • ap_setup_listeners(): 設定に基づいてソケットを生成し、バインドする関数。
  • LISTEN_COMMANDS: MPM が設定テーブルに含めるべき、リスナー関連ディレクティブの定義マクロ。

制御フローとデータフロー

1. **設定フェーズ**: LISTEN_COMMANDS を通じて ap_set_listener 等が呼び出され、ap_listen_rec リストが構築されます。 2. **初期化フェーズ**: ap_setup_listeners() が呼び出され、リスト内の全エントリに対して socket(), bind(), listen() が実行されます。 3. **実行フェーズ**: 必要に応じて ap_duplicate_listeners() が呼ばれ、マルチコア環境向けにリスナーが複製されます。 4. **終了フェーズ**: ap_close_listeners() により、全ソケットが適切にクローズされます。

直接依存

  • apr_network_io.h: APR レベルのソケット操作。
  • httpd.h: server_rec 等のコアデータ構造。
  • http_config.h: 設定解析用コマンド定義。
  • apr_optional.h: システム固有（systemd 等）のオプション機能の統合。

関連コード

  • server/listen.c: 本ヘッダーで宣言された関数の実装が含まれる。
  • server/mpm/: 各 MPM が ap_listeners をどのように利用して接続を受け入れるかの実装。

読者向けメモ

  • 本ファイルで宣言されている ap_set_... 系関数は、設定解析時にのみ使用される内部的なものです。外部モジュールから直接呼び出すべきではありません。
  • ap_listen_rec の accept_func は、MPM が接続を受け入れる際に呼び出す関数ポインタであり、プロトコルごとのハンドリングを可能にする重要なフックです。
  • ap_duplicate_listeners は、高負荷環境でのスケーラビリティ向上のために使用されます。プラットフォームが SO_REUSEPORT をサポートしているかどうかが動作の鍵となります。
