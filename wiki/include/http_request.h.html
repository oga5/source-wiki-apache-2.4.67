include/http_request.h

概要

Apache HTTP Server のリクエスト処理ライフサイクルを定義する中心的なヘッダーファイルです。クライアントからのリクエスト受信後の「ディレクトリ設定の適用」「モジュールディスパッチ」「サブリクエストの生成」「内部リダイレクト」といった、リクエスト処理のメインラインを制御するための API とフックを提供します。

モジュール内での役割

modules/http/http_request.c と対をなし、Apache のリクエスト処理パイプラインのインターフェースを定義します。httpd.h で定義される request_rec 構造体を操作し、リクエストの各フェーズ（翻訳、認証、認可、修正、コンテンツ生成）をモジュールが介入できるようにします。

責務

  • **サブリクエスト管理**: URI やファイルパスに基づいたサブリクエストの生成、実行、破棄。
  • **内部リダイレクト**: リクエストの転送とハンドラの差し替え。
  • **認証・認可フックの提供**: アクセス制御フェーズにおけるモジュール連携の定義。
  • **リクエスト処理の制御**: メインリクエストの処理開始、終了後の後処理、および EOR (End Of Request) バケットによるリソース解放の管理。
  • **メソッド管理**: リソースに対する許可メソッドの動的な制御。

重要なシンボル

サブリクエスト・リダイレクト

  • ap_sub_req_lookup_uri, ap_sub_req_lookup_file: URI/ファイルからサブリクエストを生成。
  • ap_run_sub_req: サブリクエストの実行。
  • ap_destroy_sub_req: サブリクエストのメモリ解放。
  • ap_internal_redirect: リクエストの内部リダイレクト。

認証・認可フック登録

  • ap_hook_check_access, ap_hook_check_authn, ap_hook_check_authz: 各フェーズのフック登録用。
  • AP_AUTH_INTERNAL_PER_URI, AP_AUTH_INTERNAL_PER_CONF: 内部リクエストにおけるアクセス制御の適用範囲を指定するフラグ。

リクエスト処理制御

  • ap_process_request: メインリクエストの同期処理。
  • ap_process_async_request: 非同期処理を許容するリクエスト処理。
  • ap_die: エラー発生時のリクエスト終了処理。

EOR (End Of Request) バケット

  • ap_bucket_eor_create: リクエスト終了時に request_rec を破棄するための特殊バケット生成。
  • AP_BUCKET_IS_EOR: バケットが EOR かどうかの判定。

制御フローとデータフロー

1. **リクエスト開始**: ap_process_request が呼び出され、リクエストの各フェーズ（translate_name → map_to_storage → check_user_id → fixups → type_checker → access_checker → auth_checker）が順次実行されます。 2. **サブリクエスト**: ap_sub_req_lookup_* で生成された request_rec は、メインリクエストのプールから派生したメモリ空間を持ちます。ap_run_sub_req により、独立したフィルタスタックを通じて処理されます。 3. **終了処理**: 応答の送信完了後、EOR バケットがフィルタチェーンを通過することで、ap_destroy_sub_req に相当するリソース解放が自動的に行われます。

直接依存

  • apr_optional.h: オプション機能の動的解決。
  • util_filter.h: フィルタチェーンの管理。

関連コード

  • modules/http/http_request.c: 本ヘッダーで宣言された関数の実装。
  • include/httpd.h: request_rec 構造体の定義。
  • include/http_config.h: モジュール設定とフックの基盤。

読者向けメモ

  • **メモリ所有権**: サブリクエストは apr_pool_t を使用します。ap_destroy_sub_req を呼び出すと関連メモリが解放されるため、サブリクエストの生存期間中に参照を保持する場合は注意が必要です。
  • **EOR バケット**: リクエストの生存期間を制御する重要な仕組みです。出力フィルタの最後に配置され、すべてのデータが送信された後にリクエストをクリーンアップします。
  • **省略された詳細**: 本ファイルはインターフェース定義に特化しており、具体的なフックの実行順序や、ap_directory_walk などの内部的なパス走査アルゴリズムの詳細は http_request.c に委ねられています。
