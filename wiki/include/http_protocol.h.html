include/http_protocol.h

概要

http_protocol.h は、Apache HTTP Server における HTTP プロトコル処理のコアインターフェースを定義するヘッダーファイルです。クライアントとの直接的な通信、リクエストの解析、レスポンスの生成、およびプロトコルスイッチング（HTTP/1.1 からのアップグレード等）を制御する主要な関数群とデータ構造を提供します。

モジュール内での役割

APACHE_CORE_PROTO グループの一部として、httpd.h で定義された基本構造体（request_rec, conn_rec）を操作し、HTTP プロトコル特有のセマンティクスを実装するための抽象化層を提供します。modules/http/http_protocol.c と対になっており、フィルタチェーンを通じたデータ送受信の制御を担います。

責務

  • **リクエスト処理**: リクエスト行の解析、ヘッダーの読み込み、認証情報の抽出。
  • **レスポンス生成**: ステータスラインの構築、ETag/Last-Modified 等のキャッシュ制御ヘッダーの管理。
  • **データ転送**: フィルタを通じたクライアントへのデータ出力（ap_rputs, ap_rprintf 等）およびクライアントからのボディ読み込み。
  • **プロトコル管理**: プロトコルネゴシエーション（ALPN 等）およびプロトコルスイッチングのフック提供。
  • **エラーハンドリング**: HTTP エラーレスポンスの生成と、フィルタスタック内でのエラーバケット（ap_bucket_error）の管理。

重要なシンボル

リクエスト・レスポンス制御

  • ap_read_request, ap_parse_request_line: リクエストのライフサイクル開始と解析。
  • ap_send_error_response: エラー発生時のレスポンス生成。
  • ap_finalize_request_protocol: リクエスト終了時のプロトコル処理の完了。

データ出力（フィルタ経由）

  • ap_rputs, ap_rprintf, ap_rwrite: クライアントへのデータ書き込み。
  • ap_rflush: バッファのフラッシュ。
  • ap_send_fd, ap_send_mmap: ファイルやメモリマップの効率的な送信。

条件付きリクエストとキャッシュ

  • ap_meets_conditions: If-Match 等の条件付きリクエストの評価。
  • ap_make_etag, ap_set_etag: ETag の生成と設定。
  • etag_rec: ETag 生成に必要なメタデータ構造体。

プロトコルスイッチング

  • ap_protocol_propose, ap_protocol_switch: プロトコル切り替えのためのフック。
  • ap_select_protocol, ap_switch_protocol: プロトコル選択と実行の API。

エラーバケット

  • ap_bucket_error: フィルタスタックでエラーを伝播させるための特殊なバケット型。

制御フローとデータフロー

1. **入力**: ap_rgetline を通じてクライアントからの生データを読み込み、request_rec に格納します。 2. **解析**: ap_parse_request_line や ap_get_mime_headers がリクエストを分解し、内部状態を更新します。 3. **処理**: 各種フック（post_read_request 等）が実行され、リクエストの妥当性が検証されます。 4. **出力**: ap_r* 系関数が apr_bucket_brigade を通じてフィルタスタックにデータを投入し、最終的にネットワークへ送信されます。 5. **エラー**: フィルタ処理中に問題が発生した場合、ap_bucket_error を brigade に挿入することで、後続のフィルタをスキップしエラーレスポンスを生成します。

直接依存

  • httpd.h: 基本的なリクエスト・接続構造体。
  • apr_buckets.h: フィルタチェーンのデータ転送単位。
  • util_filter.h: フィルタスタックの管理。
  • apr_portable.h, apr_mmap.h: プラットフォーム依存のファイル・メモリ操作。

関連コード

  • modules/http/http_protocol.c: 本ヘッダーの関数実装。
  • include/httpd.h: request_rec 等の定義。
  • include/util_filter.h: フィルタの仕組み。

読者向けメモ

  • **メモリ管理**: 多くの関数は request_rec 内のプール（r->pool）を使用してメモリを割り当てます。ライフサイクルはリクエストの終了とともに破棄されるため、永続的なデータ保持には注意が必要です。
  • **スレッド安全性**: 本 API は主にリクエスト処理コンテキストで呼び出されることを想定しており、特定の接続に対する排他制御は呼び出し側（MPM）の責任範囲です。
  • **省略された詳細**: プロトコルスイッチングの具体的な実装や、EBCDIC 環境における文字コード変換の詳細については、実装ファイル（.c）を参照してください。これらはプラットフォーム固有の複雑さを隠蔽するためにマクロやラッパーでラップされています。
