Multi-Processing Modules (MPM) and Process Lifecycle

概念

Multi-Processing Modules (MPM) は、Apache HTTP Server がネットワークポートをバインドし、クライアントからのリクエストを受け付け、それを処理する子プロセスやスレッドを管理するためのコアアーキテクチャです。

Apache は異なる並行処理モデル（プロセスベース、ハイブリッド・マルチスレッド、非同期イベント駆動など）をサポートするために、MPM をモジュール化して提供しています。代表的な実装には、prefork、worker、event があります。

なぜ重要か

MPM は Web サーバーのパフォーマンス、スケーラビリティ、およびリソース消費（メモリと CPU）を決定づける基盤です。

  • **Prefork**: スレッドセーフでないライブラリとの互換性を確保するために、1リクエストを1プロセスで処理します。
  • **Worker**: マルチプロセスとマルチスレッドを組み合わせることで、メモリ消費を抑えつつ高い並行性を実現します。
  • **Event**: Worker モデルをベースに、キープアライブ接続などの非同期処理を専用のリスナースレッドに委ねることで、さらに効率的なリソース利用を可能にします。

主要エンティティ

各 MPM のライフサイクルと動作を定義する主要なソースファイルは以下の通りです。

  • server/mpm/event/event.c: 非同期イベント駆動型モデル（Event MPM）の実装。
  • server/mpm/prefork/prefork.c: スレッドを使用しないプロセス先行生成型モデル（Prefork MPM）の実装。
  • server/mpm/worker/worker.c: マルチプロセス・マルチスレッド型モデル（Worker MPM）の実装。

エンドツーエンドの流れ

親プロセスが起動し、設定に基づいて子プロセス（およびスレッド）を生成・維持し、シャットダウンまたは再起動時にそれらを適切に回収するライフサイクルが存在します。

重要な不変条件

  • 各 MPM は、設定された最小/最大アイドルプロセス数（またはスレッド数）を維持するように、プロセスの生成と破棄を動的に制御する必要があります。
  • prefork では1つのプロセスが同時に1つの接続のみを処理し、worker および event ではスレッド間で安全に接続が分配される必要があります。

※ 具体的な同期プリミティブやスコアボードの更新に関する詳細な不変条件は、現時点では未確定です。

よくある落とし穴

  • **スレッド安全性の欠如**: worker や event などのマルチスレッド MPM を使用する際、スレッドセーフでないサードパーティ製モジュールやライブラリをロードすると、メモリ汚染やクラッシュを引き起こす原因になります。
  • **リソースの枯渇**: prefork において、最大プロセス数（MaxRequestWorkers）を高く設定しすぎると、高負荷時に物理メモリを使い果たし、システムのスワップアウトを招く可能性があります。

※ ソースコードレベルでの具体的なデッドロックや競合状態に関する落とし穴は、現時点では未確定です。
