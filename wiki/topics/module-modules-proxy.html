Module: modules/proxy

概念

modules/proxy は、Apache HTTP Server (httpd) においてプロキシ機能およびロードバランシング機能を提供するモジュール群です。コアとなる共通基盤（mod_proxy）と、特定のプロトコルやロードバランシングアルゴリズムを処理する多数のサブモジュールから構成されています。これにより、httpd をリバースプロキシ、フォワードプロキシ、またはゲートウェイとして動作させることが可能になります。

なぜ重要か

現代のWebアーキテクチャにおいて、アプリケーションサーバー（Tomcat, Node.js, FastCGIアプリケーションなど）の前段に配置されるリバースプロキシやロードバランサの役割は不可欠です。modules/proxy は、多様なバックエンドプロトコルを統一された設定と接続管理機構のもとで効率的にハンドリングし、システムの可用性とスケーラビリティを向上させるための基盤を提供します。

関与するモジュール

  • **module:modules/proxy**: プロキシのコア機能、接続ユーティリティ、および各種プロトコルハンドラ。
  • **module:modules/proxy/balancers**: ロードバランサのスケジューリングアルゴリズム（負荷分散メソッド）。

主要エンティティ

提供されたファイル群から、以下の主要なコンポーネントが識別されます。

  • **コアおよびユーティリティ**:
  • mod_proxy.c / mod_proxy.h: プロキシの基本設定、ディレクティブ、共通APIの定義。
  • proxy_util.c / proxy_util.h: 接続プール（Connection Pool）の管理、ワーカーの初期化、DNS解決などの共通ユーティリティ。
  • **プロトコル固有のハンドラ**:
  • mod_proxy_http.c: HTTP/HTTPS プロキシ。
  • mod_proxy_ajp.c (および ajp_*.c, ajp.h): Apache JServ Protocol (AJP) を用いたアプリケーションサーバー（Tomcat等）との通信。
  • mod_proxy_fcgi.c: FastCGI プロキシ。
  • mod_proxy_scgi.c / scgi.h: SCGI プロキシ。
  • mod_proxy_uwsgi.c: uWSGI プロキシ。
  • mod_proxy_wstunnel.c: WebSocket トンネリング。
  • mod_proxy_connect.c: SSLトンネリング等で用いられる CONNECT メソッドの処理。
  • mod_proxy_ftp.c: FTP プロキシ。
  • **ロードバランシングとヘルスチェック**:
  • mod_proxy_balancer.c: 複数のバックエンドワーカーをグループ化し、リクエストを分散。
  • balancers/mod_lbmethod_*.c:
  • byrequests: リクエスト数に基づく分散。
  • bytraffic: 転送バイト数に基づく分散。
  • bybusyness: アクティブな接続数（ビジー度）に基づく分散。
  • heartbeat: 外部のハートビート監視に基づく分散。
  • mod_proxy_hcheck.c: バックエンドワーカーの動的なヘルスチェック。
  • **その他の特殊モジュール**:
  • mod_proxy_express.c: 大規模なバーチャルホスト向けのシンプルな動的プロキシ。
  • mod_proxy_fdpass.c / mod_proxy_fdpass.h: プロセス間でのファイル記述子（FD）の受け渡し。

エンドツーエンドの流れ

*※提供されたソースコードの具体的な内部関数呼び出しや詳細な実行フローは、現時点では完全には検証されていません。*

一般的な想定フローは以下の通りです： 1. **リクエストの受信**: クライアントからのリクエストを httpd が受信し、mod_proxy のハンドラが介入します。 2. **プロキシターゲットの決定**: 設定（ProxyPass など）に基づき、リクエストを転送するバックエンドのURL（例: http://, ajp://, balancer://）を決定します。 3. **ロードバランシング（必要な場合）**: ターゲットが balancer:// の場合、mod_proxy_balancer が有効なロードバランサメソッド（例: mod_lbmethod_byrequests）を呼び出し、最適なバックエンドワーカーを選択します。 4. **プロトコル変換と転送**: 選択されたワーカーのプロトコルに対応するサブモジュール（例: mod_proxy_http）が、proxy_util の接続管理機能を利用してバックエンドへの接続を確立または再利用し、リクエストを転送します。 5. **レスポンスの返却**: バックエンドからのレスポンスを受信し、クライアントへ中継します。

重要な不変条件

*※提供されたファイル一覧からは、具体的なコードレベルの不変条件（スレッドセーフティ、メモリプールの寿命、ロックの順序など）は直接確認されていません。*

一般的に、プロキシモジュール群においては以下の不変条件が維持される必要があります：

  • **接続プールの整合性**: 各ワーカーの接続プールは、設定された最大接続数（max）を超えてはならず、スレッド間で安全に共有・再利用されなければならない。
  • **スコアボードの同期**: ロードバランサの状態（各ワーカーのアクセス数やエラー状態など）は、マルチプロセス/マルチスレッド環境下で一貫して共有メモリ（スコアボード）に反映されなければならない。

よくある落とし穴

*※提供されたファイル一覧からは、具体的なバグや落とし穴の詳細は直接確認されていません。*

一般的なプロキシ運用における既知の注意点：

  • **タイムアウト設定の不整合**: ProxyTimeout や各ワーカーの timeout 設定がバックエンドの処理時間と乖離している場合、不必要な 504 Gateway Timeout や接続の早期切断が発生する。
  • **接続の枯渇**: バックエンドとのキープアライブ設定（keepalive=On）や接続プールサイズが適切でない場合、一時的な高負荷時に新規接続が作成できなくなる。
  • **DNS解決の遅延**: バックエンドのホスト名解決がリクエストごとに行われる設定になっている場合、パフォーマンスが著しく低下する（proxy_util.c でのキャッシュ制御が重要）。

  • **modules/proxy/mod_proxy_balancer.c**: ロードバランシングの制御ロジック。
