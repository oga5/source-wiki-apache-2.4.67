Module: modules/http2

概念

modules/http2 は、Apache HTTP Server において HTTP/2 プロトコル（RFC 7540 / RFC 9113）および HTTP/2 上での WebSocket（RFC 8441）をサポートするためのモジュール群です。

本モジュールは、クライアントとの物理的な TCP 接続を扱うプライマリ接続（c1）と、その上で多重化された個々の HTTP/2 ストリームを処理する仮想的なセカンダリ接続（c2）を分離して管理するアーキテクチャを採用しています。

なぜ重要か

HTTP/2 は、単一の TCP 接続上で複数のリクエストとレスポンスを並行して多重化（Multiplexing）送信できるため、Web ページの読み込みパフォーマンスを大幅に向上させます。

本モジュール群は、Apache の既存の HTTP/1.1 処理パイプライン（接続、フィルタ、コンテンツ生成）を再利用しつつ、HTTP/2 の多重化ストリーム、サーバープッシュ（Server Push）、およびストリームごとのフロー制御を効率的に実現するために不可欠な役割を担っています。

主要エンティティ

提供されたファイル群から、以下の主要エンティティが定義・利用されていることが確認できます。

  • **c1 (Primary Connection)** (h2_c1.c, h2_c1_io.c): クライアントとの直接的な物理 TCP 接続を管理します。
  • **c2 (Secondary Connection)** (h2_c2.c, h2_c2_filter.c): 各 HTTP/2 ストリームを処理するために仮想的に作成される接続です。
  • **h2_session** (h2_session.c): HTTP/2 接続全体のステートやプロトコルフレーム（HEADERS, DATA など）の送受信を管理するセッションオブジェクトです。
  • **h2_stream** (h2_stream.c): 個々の HTTP/2 ストリーム（リクエストとレスポンスのペア）を表すオブジェクトです。
  • **h2_mplx** (h2_mplx.c): c1 接続と複数の c2 接続（およびワーカー）の間で、データの多重化や同期、タスクのスケジューリングを仲介するコーディネーターです。
  • **h2_bucket_beam** (h2_bucket_beam.c): スレッド境界（c1 接続スレッドと c2 ワーカータスクスレッドの間など）を越えて、バケットブリゲード（データ）を安全かつ高速に転送するための同期バッファ機構です。
  • **h2_workers** (h2_workers.c): c2 接続の処理タスクを非同期に実行するための専用ワーカープールです。

エンドツーエンドの流れ

詳細なエンドツーエンドの制御フロー（例：特定のフレーム受信からレスポンス送信までの厳密な関数コールスタックやエラーハンドリングの全容）は、提供されたファイル構造からは完全には特定されていません。

大まかな処理の流れは以下の通りです：

1. クライアントからの接続要求を mod_http2.c が受け取り、HTTP/2 プロトコルへの切り替え（Upgrade または ALPN）を検出します。 2. h2_session が作成され、c1 接続上で HTTP/2 フレームの解析が開始されます。 3. 新しいストリーム（HEADERS フレームなど）を検知すると、h2_stream が生成され、h2_mplx を介して h2_workers のタスクとしてスケジュールされます。 4. 各ストリームは仮想的な c2 接続として処理され、Apache の標準的なハンドラやフィルタ（h2_c2_filter.c）を通過してコンテンツが生成されます。 5. 生成されたレスポンスデータは h2_bucket_beam を通じて c1 接続側に送り返され、h2_session によって HTTP/2 フレームにカプセル化されてクライアントに送信されます。

重要な不変条件

詳細な内部状態の不変条件（スレッドセーフティの境界条件や、h2_bucket_beam におけるメモリ管理の厳密なライフサイクルルールなど）は、提供されたソースコードのメタデータからは完全には特定されていません。

設計上、以下の不変条件が維持される必要があります：

  • **接続の分離**: クライアントと直接通信する c1 接続スレッドと、個々のストリームを処理する c2 ワーカースレッドは明確に分離され、データのやり取りは h2_mplx および h2_bucket_beam を介してのみ行われなければなりません。
  • **ストリームの多重化**: 単一の c1 接続上で、複数の c2 接続（ストリーム）が並行して進行し、それぞれのフロー制御（Window Size）が独立して維持される必要があります。

よくある落とし穴

詳細な実行時の落とし穴（特定のデッドロックシナリオや、リソースリークの具体的な発生条件など）は、提供されたファイル構造からは完全には特定されていません。

一般的な設計上の注意点として以下が挙げられます：

  • **スレッド競合とデッドロック**: h2_mplx や h2_bucket_beam におけるロックの順序や、c1 と c2 の間でのリソース競合。
  • **メモリ消費**: 多数の並行ストリームが存在する場合、各ストリームのバッファ（Bucket Beam）が消費するメモリの総量が急増する可能性があるため、適切な制限（ウィンドウサイズや最大ストリーム数）の設定が必要です。
  • **プロキシ時の挙動**: mod_proxy_http2 を使用してバックエンドと HTTP/2 で通信する際、フロントエンドとバックエンド間のストリーム状態の同期やタイムアウトの不一致に注意する必要があります。
