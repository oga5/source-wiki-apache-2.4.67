argstr_to_table

概念

argstr_to_table は、key=value&key2=value2 のようなクエリ文字列形式の文字列を解析（パース）し、そのキーと値のペアを Apache Portable Runtime のテーブル構造体である apr_table_t に格納するための関数です。

なぜ重要か

HTTP リクエストのクエリパラメータや、ハートビート監視データなどの設定文字列をパースし、プログラムから扱いやすいキー・バリュー形式（apr_table_t）に変換する処理は、Apache HTTP Server 内の複数のモジュールで必要とされます。本関数は、URLデコード（ap_unescape_url）や、値が省略されたキー（例: key のみの場合に値を "1" とする）のハンドリングなど、クエリ文字列解析の共通的な手続きを提供します。ただし、複数のモジュールで個別に同名関数が定義されているため、コードの重複管理の観点でも注意が必要です。

関与するモジュール

主要エンティティ

  • symbol:modules/proxy/balancers/mod_lbmethod_heartbeat.c#argstr_to_table : mod_lbmethod_heartbeat.c 内で定義されている関数。readfile_heartbeats から呼び出されます。
  • symbol:server/util_script.c#argstr_to_table : server/util_script.c 内に存在する同名の別定義関数。

エンドツーエンドの流れ

mod_lbmethod_heartbeat.c における具体的な処理の流れは以下の通りです：

1. readfile_heartbeats 関数から argstr_to_table が呼び出されます。 2. apr_strtok を使用して、入力文字列 str を & を区切り文字としてトークンに分割します。 3. 各トークンについて、= の位置を検索し、キーと値に分割します。 4. = が見つからない場合、値は "1" として扱われます。 5. ap_unescape_url を呼び出して、キーと値の URL エンコードを解除（デコード）します。 6. apr_table_set を使用して、デコードされたキーと値のペアを apr_table_t に追加します。

※ server/util_script.c 側の argstr_to_table を含む全体的なエンドツーエンドの流れや、両者の統合的な呼び出しフローは現時点では不明です。

重要な不変条件

  • **破壊的変更**: 入力される文字列 str は、内部で apr_strtok によって直接分割・書き換えられるため、呼び出し元で変更されても問題ない（または変更されてもよいバッファである）必要があります。
  • **デフォルト値**: キーに対応する値が存在しない場合（例: key&key2）、値は一貫して "1" として扱われます。

よくある落とし穴

  • **重複実装**: mod_lbmethod_heartbeat.c と server/util_script.c に同名の関数が個別に定義されています。一方の挙動を修正・変更する際、もう一方の挙動との不整合が発生する可能性があります。
  • **入力文字列の破壊**: 渡された str が書き換えられるため、読み取り専用のメモリ領域にある文字列を直接渡すとセグメンテーションフォールトなどの予期しない動作を引き起こします。
  • **不正なURLエンコード**: ap_unescape_url によるデコード処理の成否や、不正なエンコードが含まれていた場合の挙動は、呼び出し先の ap_unescape_url の実装に依存します。
