argstr_to_table

概念

argstr_to_table は、HTTP のクエリ文字列形式（例: key=value&key2=value2）の文字列を解析（パース）し、キーと値のペアを Apache Portable Runtime のテーブル構造体である apr_table_t に格納するための関数です。

なぜ重要か

この関数は、クエリパラメータの解析処理を共通化し、URL デコード（ap_unescape_url）やキー・値の抽出、およびデフォルト値の設定（値がないキーに対して "1" を割り当てるなど）をカプセル化します。これにより、モジュールが受信したパラメータを容易に扱えるようになります。また、server/util_script.c と modules/proxy/balancers/mod_lbmethod_heartbeat.c の複数箇所に同名の関数が存在しており、同様の解析処理がシステム内の異なる文脈で必要とされていることを示しています。

エンドツーエンドの流れ

modules/proxy/balancers/mod_lbmethod_heartbeat.c における具体的な処理の流れは以下の通りです：

1. **呼び出し**: readfile_heartbeats 関数内（179行目付近）から argstr_to_table が呼び出されます。 2. **トークン分割**: apr_strtok を使用して、入力文字列 str を & を区切り文字として分割し、各パラメータを走査します。 3. **キーと値の分離**: 各トークン内で = を検索し、キーと値に分割します。 = が見つからない場合、値は "1" とみなされます。 4. **URL デコード**: ap_unescape_url を呼び出して、分割されたキーと値の URL エンコードを解除します。 5. **テーブルへの格納**: apr_table_set を使用して、解析されたキーと値を apr_table_t に追加します。

なお、server/util_script.c に定義されている argstr_to_table については、現時点では詳細なエンドツーエンドの流れや呼び出し元は判明していません。

重要な不変条件

  • **入力文字列の破壊的変更**: 入力されるクエリ文字列 str は、内部で apr_strtok によって直接分割（ヌル文字の挿入など）されるため、呼び出し元で保持している元の文字列が変更されても問題ない状態（書き込み可能なメモリ領域）である必要があります。
  • **デフォルト値の割り当て**: 値が存在しないキー（例: key&key2）が解析された場合、そのキーに対応する値は一貫して "1" としてテーブルに格納されます。

よくある落とし穴

  • **読み取り専用メモリの書き換え**: 読み取り専用のメモリ領域（文字列リテラルなど）を str 引数として渡すと、apr_strtok による書き換えが発生してセグメンテーションフォールトなどの未定義動作を引き起こします。
  • **重複実装の存在**: modules/proxy/balancers/mod_lbmethod_heartbeat.c と server/util_script.c に同名の関数が別個に定義されています。これらは類似のロジックを持つ可能性がありますが、別定義であるため、一方を変更する際にもう一方との整合性に注意する必要があります。
  • **URL デコードの失敗**: URL エンコードが不正な場合、ap_unescape_url の挙動に依存するため、予期しないデコード結果になる可能性があります。
