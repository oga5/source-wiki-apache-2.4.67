Apache Filter Architecture and Bucket Brigades

概念

Apache HTTP Serverにおけるフィルタアーキテクチャは、リクエストおよびレスポンスのデータをストリームとして段階的に処理するためのパイプライン機構です。データは「バケット（Bucket）」と呼ばれるデータ片に分割され、それらを集めた双方向連結リストである「バケットブリゲード（Bucket Brigade）」として、登録されたフィルタのチェーン（ap_filter_t）を順次通過します。

なぜ重要か

このアーキテクチャにより、データの生成、加工、送信が高度にモジュール化されます。例えば、コンテンツ生成モジュールがデータを出力した後、データの圧縮、文字コード変換、あるいはHTTPチャンクエンコーディング（modules/http/chunk_filter.c）やHTTPヘッダの付与（modules/http/http_filters.c）といった処理を、個別の独立したフィルタとして動的に連結して適用できます。

関与するモジュール

  • module:include （フィルタの基本構造やAPIを定義する include/util_filter.h を提供）
  • module:modules/http （HTTPプロトコル固有のフィルタ処理を行う modules/http/chunk_filter.c や modules/http/http_filters.c を提供）

主要エンティティ

  • **ap_filter_t** (include/util_filter.h):

    • フィルタチェーン内の個々のフィルタインスタンスを表す構造体。次のフィルタへのポインタ（next）や、フィルタ固有の実行コンテキスト（ctx）を保持します。

  • **ap_filter_rec_t** (include/util_filter.h):

    • 登録されたフィルタの定義情報。フィルタ名、フィルタ関数ポインタ、およびフィルタの適用階層を決定するフィルタタイプ（ap_filter_type）を含みます。

  • **apr_bucket_brigade**:

    • 複数のバケットを格納するコンテナ。フィルタ間で受け渡されるデータの最小単位のシーケンスです。

  • **apr_bucket**:

    • 実際のデータ（メモリ、ファイル、ソケットなど）を抽象化したデータ単位。

  • **HTTP関連フィルタ関数** (modules/http/http_filters.c, modules/http/chunk_filter.c):

    • ap_http_header_filter や ap_chunk_filter など、HTTPプロトコル準拠のデータ整形やチャンク化を行う具体的なフィルタ実装。

エンドツーエンドの流れ

提供されたソースコードの範囲内では、フィルタチェーン全体の動的な構築プロセスや、すべてのエラーハンドリングを含む完全なエンドツーエンドのライフサイクルは完全には開示されていませんが、基本的なデータフローは以下の通りです。

1. **フィルタの登録**: モジュールの初期化時に、ap_register_output_filter などのAPIを用いてフィルタ定義（ap_filter_rec_t）が登録されます。 2. **チェーンの構築**: 接続またはリクエストの開始時に、適切なフィルタが ap_filter_t のリンクドリストとして接続されます。 3. **データの書き込みと伝播**: コンテンツジェネレータまたは上流のフィルタが apr_bucket_brigade を作成し、ap_pass_brigade を呼び出して下流のフィルタに渡します。 4. **フィルタ処理の実行**: modules/http/chunk_filter.c や modules/http/http_filters.c に実装されたフィルタ関数が呼び出され、ブリゲード内のバケットを走査・加工（例: データのチャンク化やヘッダの挿入）し、さらに ap_pass_brigade で次のフィルタへ伝播します。

重要な不変条件

提供されたソースコードの範囲内では、すべての厳密な不変条件は明示されていませんが、以下の設計上の原則が求められます。

  • **フィルタタイプの順序維持**:

    • フィルタチェーンは、ap_filter_type（コンテンツフィルタ、プロトコルフィルタ、ネットワークフィルタなど）で定義された優先順位に従って厳密にソートされていなければなりません。

  • **メタデータバケットの伝播**:

    • ストリームの終端を示す EOS（End of Stream）バケットや、データの即時フラッシュを要求する FLUSH バケットなどのメタデータバケットは、各フィルタで無視せず、適切に処理または下流へ伝播させなければなりません。

よくある落とし穴

  • **メモリリークとリソース管理**:

    • バケットブリゲードやバケットの作成・割り当てにおいて、適切なプールやアロケータを使用し、処理後にブリゲードを適切にクリーンアップ（apr_brigade_cleanup）しないと、メモリリークの原因になります。

  • **ブロッキング/ノンブロッキングの誤用**:

    • 入力フィルタ（ap_get_brigade）において、読み込みモード（ブロッキング、ノンブロッキングなど）の指定を誤ると、スレッドのハングアップや不要なスピンリードが発生する可能性があります。

  • **メタデータバケットの消失**:

    • 自作フィルタ内でバケットを再構築する際、EOS バケットを破棄してしまうと、クライアントへのレスポンスが完了せず、接続がタイムアウトするまでハングする原因になります。

次に読む場所

  • **include/util_filter.h**:

    • フィルタの登録、チェーン操作API、および ap_filter_t などの基本構造体の定義を確認できます。

  • **modules/http/chunk_filter.c**:

    • 出力データをHTTPチャンク形式に変換する、具体的かつシンプルな出力フィルタの実装例を学習できます。

  • **modules/http/http_filters.c**:

    • HTTPヘッダの処理や、リクエストボディの読み込みを行う入力/出力フィルタの複雑な実例を確認できます。
