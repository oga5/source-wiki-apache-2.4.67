dav_locktoken

概念

dav_locktoken は、Apache HTTP Server の WebDAV（mod_dav）モジュール群において、WebDAV ロックを一意に識別するためのトークン（UUID）を保持するラッパー構造体です。

本構造体は以下のように定義され、内部に APR（Apache Portable Runtime）の UUID 型である apr_uuid_t を保持します。

struct dav_locktoken {
    apr_uuid_t uuid;
};

なぜ重要か

WebDAV プロトコルにおけるロック操作（リソースの排他制御など）において、各ロックを一意に識別、照合、比較、および永続化するための基盤となる識別子として機能するためです。mod_dav のロックプロバイダインターフェースにおいて、ロックの作成、検索、解決を行う際の共通のキーとして利用されます。

エンドツーエンドの流れ

WebDAV リクエストの受信からロックトークンの生成、検証、レスポンス返却にいたる完全なエンドツーエンドのフローは、提供された情報からは明らかになっていません。

ただし、部分的な処理の流れとして以下が判明しています：

1. **解析と生成**: ロックトークンの生成、解析、比較には、dav_fs_parse_locktoken、dav_fs_format_locktoken、dav_fs_compare_locktoken などの専用関数が使用されます。 2. **ロードと照合**: dav_fs_load_lock_record などのロック管理関数が、ロック情報の構築や比較のために本構造体を参照します。 3. **解決**: ロックの作成、検索、解決の各プロセスにおいて、リソースに関連付けられたロック（dav_lock_discovery や dav_lock_indirect などの関連データ構造）を特定するために参照されます。

重要な不変条件

  • **単一のデータ保持**: dav_locktoken は、apr_uuid_t を保持する単一のフィールドのみで構成されます。
  • **一意性の保証**: ロックの一意性は、内部に保持される apr_uuid_t を介して保証されることが期待されます。
  • **読み取り専用の扱い**: ロック操作の文脈において、この構造体へのポインタは const として扱われることが一般的です。

よくある落とし穴

  • **エラーハンドリングの依存性**: 本構造体自体は単純なデータコンテナであるため、メモリ割り当て失敗時を除き、構造体固有の失敗モードはありません。ただし、無効な UUID が渡された場合の挙動は、利用する関数（dav_fs_compare_locktoken 等）や APR ライブラリ側の実装に依存します。
  • **複数箇所での定義**: modules/dav/fs/lock.c と modules/dav/lock/locks.c の両方に構造体の定義が存在します。構造体の拡張や変更を行う際は、ファイルシステム側のロック実装と汎用ロック管理実装の双方への影響を考慮する必要があります。
