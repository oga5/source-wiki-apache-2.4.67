dav_locktoken

概念

dav_locktoken は、WebDAV リソースのロックを一意に識別するためのトークン（識別子）を保持する構造体です。

mod_dav のメインヘッダー（mod_dav.h）では不透明（opaque）な構造体として宣言されており、具体的なプロバイダー実装（mod_dav_fs や mod_dav_lock など）において、内部に apr_uuid_t 型の UUID を保持する具体的な構造体として定義されています。

なぜ重要か

WebDAV 仕様におけるロック管理において、リソースに対する一意な識別子（ロックトークン）を提供するために不可欠な要素です。 mod_dav のロックプロバイダーインターフェースにおける識別子の基盤として機能し、ロックの作成、検索、比較、解決、および永続化の際に、特定のリソースに関連付けられたロックを特定するためのキーとして利用されます。

エンドツーエンドの流れ

WebDAV リソースのロック要求から解放に至るまでの完全なエンドツーエンドの流れ（どのようにトークンが生成され、クライアントに返され、再検証されるかなど）の全容は、提供された情報からは完全には明らかになっていません。

部分的な流れとして、ロックの作成や検索のプロセスにおいて、dav_fs_parse_locktoken、dav_fs_format_locktoken、dav_fs_compare_locktoken などの専用関数を介してトークンの解析、フォーマット、比較が行われ、dav_lock_discovery や dav_lock_indirect などの関連データ構造に格納されて管理されることが分かっています。

重要な不変条件

  • **一意性の保証**: 内部に保持される apr_uuid_t を介して、ロックトークンの一意性が保証されることが期待されます。
  • **不透明性の維持**: mod_dav.h では不透明な型として宣言されているため、利用側（mod_dav コアなど）からは構造体の内部データ（uuid など）へ直接アクセスせず、提供される API を介して操作することが求められます。
  • **読み取り専用の扱い**: ロック操作の文脈において、この構造体へのポインタは const として扱われることが一般的です。

よくある落とし穴

  • **不透明性の破壊**: mod_dav.h で不透明な型として定義されているため、特定のプロバイダー実装（mod_dav_fs など）に依存した直接的な構造体メンバーへのアクセスを行うと、他のプロバイダー実装との互換性が失われます。
  • **未初期化データでの比較**: 構造体内部のデータ（UUID）が適切に初期化されていない状態で比較処理（dav_fs_compare_locktoken など）を行うと、予期せぬ動作を引き起こす可能性があります。
  • **メモリ割り当てエラー**: 構造体のメモリ割り当てに失敗した場合や、無効なトークンが渡された場合の挙動はプロバイダーの実装に依存するため、エラーハンドリングに注意が必要です。

次に読む場所

  • **modules/dav/main/mod_dav.h**: 不透明な構造体宣言と、ロックインターフェースの定義を確認できます。
  • **modules/dav/fs/lock.c**: ファイルシステムベースのロックプロバイダーにおける具体的な実装と操作関数群（解析、フォーマット、比較など）を確認できます。
  • **modules/dav/lock/locks.c**: 汎用ロックモジュールにおける具体的な実装を確認できます。
