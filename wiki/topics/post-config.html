post_config

概念

post_config は、Apache HTTP Server の設定フェーズの後半（設定ファイルの読み込みと適用が完了した後）に実行される初期化フック関数です。各モジュールはこのフェーズを利用して、依存する他モジュールの存在確認や、実行に必要な内部状態のセットアップを行います。

なぜ重要か

モジュールが正常に動作するための前提条件（他の必須モジュールがロードされているかなど）を、リクエスト処理が始まる前のサーバー起動・再起動フェーズで検証・記録するために重要です。例えば、mod_proxy_express では、基盤となる mod_proxy がロードされているかをこの段階で確認し、その後の処理の可否を決定します。

エンドツーエンドの流れ

Apache HTTP Server の起動または再起動時、設定ファイルの読み込みが完了した後に post_config フックが呼び出されます。

mod_proxy_express における具体的な流れは以下の通りです： 1. Apache Core から post_config が呼び出されます。 2. ap_find_linked_module("mod_proxy.c") を呼び出し、プロキシ機能の基盤となる mod_proxy がロードされているかを判定します。 3. 判定結果（真偽値）を静的変数 proxy_available に格納します。 4. 常に OK を返して処理を終了します。

※ mod_rewrite および mod_slotmem_shm における post_config の具体的なエンドツーエンドの流れや、システム全体での呼び出し順序の制御などの詳細は現時点では不明です。

重要な不変条件

  • mod_proxy_express の post_config は、mod_proxy.c がロードされていない場合でもエラーを返さず、常に OK を返します。この場合、静的変数 proxy_available が 0 (false) に設定されます。
  • 他のモジュール（mod_rewrite、mod_slotmem_shm）における post_config の不変条件は現時点では不明です。

よくある落とし穴

  • mod_proxy_express は mod_proxy がロードされていなくても post_config でエラーを発生させないため、設定エラーが起動時に即座にブロックされない可能性があります（proxy_available が false になるのみ）。
  • 各モジュールの post_config が実行される順序や、依存関係の解決ルールについては、提供された情報からは不明なため、実行順序に依存する処理を記述する際には注意が必要です。
