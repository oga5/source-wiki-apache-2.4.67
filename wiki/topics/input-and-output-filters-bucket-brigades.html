Input and Output Filters (Bucket Brigades)

概念

Apache HTTP Serverにおけるフィルタシステムは、リクエストおよびレスポンスのデータを段階的に処理・加工するための仕組みです。データは「バケットブリゲード（apr_bucket_brigade）」と呼ばれるバケットの連結リスト構造に格納され、フィルタチェーン（登録されたフィルタの連鎖）を介して順次受け渡されます。

なぜ重要か

この仕組みにより、データの生成元（コンテンツハンドラなど）と、データの加工処理（バイトレンジの切り出し、チャンクエンコーディングなど）を独立したモジュールとして分離できます。各フィルタは、バケットブリゲードを受け取り、必要な加工を施して次のフィルタへ渡す（ap_pass_brigade）という単一の責務に集中できるため、柔軟で拡張性の高いHTTP処理パイプラインが実現されます。

関与するモジュール

提供されたソースコード情報に基づき、以下のモジュールが関与しています。

  • module:include (フィルタ定義のヘッダーファイルなど)
  • module:modules/http (HTTPプロトコル固有のフィルタ実装)
  • module:server (フィルタチェーンの基盤管理)

主要エンティティ

  • **symbol:include/util_filter.h#ap_filter_rec_t**: フィルタの定義情報（名前、実行関数、フィルタ型、プロトコル制約など）を保持する構造体。フィルタチェーンの構築や順序付けに使用されます。
  • **symbol:modules/http/byterange_filter.c#ap_byterange_filter**: HTTP Range リクエストを処理し、指定されたバイト範囲のみを切り出して送信する出力フィルタ。
  • **symbol:modules/http/chunk_filter.c#ap_http_chunk_filter**: HTTP/1.1 のチャンク転送エンコーディングを適用し、レスポンスボディをチャンク形式に変換する出力フィルタ。

エンドツーエンドの流れ

1. **フィルタの登録**: 各フィルタ（ap_byterange_filter や ap_http_chunk_filter など）は、ap_filter_rec_t を用いてシステムに登録されます。 2. **フィルタチェーンの構築**: リクエスト処理時に、登録されたフィルタがその型（ap_filter_type）に基づいてソートされ、チェーンが形成されます。 3. **データの処理と伝播**:

  • 上流からバケットブリゲード（apr_bucket_brigade）が渡されます。
  • 各フィルタはブリゲード内のバケットを走査・加工します。
  • 例: ap_byterange_filter は、指定された範囲のデータを抽出し、適切なヘッダーを付与します。
  • 例: ap_http_chunk_filter は、データ長を計算して16進数のサイズヘッダーと CRLF を付与し、チャンク化します。
  • 加工されたブリゲードは ap_pass_brigade を介して次の下流フィルタへ送信されます。

    • 4. **終了処理**: EOS（End of Stream）バケットを検出すると、フィルタは自身の役割を終え、必要に応じてフィルタチェーンから自身を削除（ap_remove_output_filter）します。

*注意: フィルタチェーン全体の初期化から最終的なネットワーク送信に至る完全なエンドツーエンドの制御フローの詳細は、提供されたモジュール情報からはすべて特定されていません。*

重要な不変条件

  • **フィルタの順序**: フィルタは ap_filter_type に基づいて厳密にソートされます。
  • **サブリクエストの制限**: AP_FTYPE_CONNECTION 以上の型を持つフィルタは、サブリクエスト内では使用できません。
  • **バイトレンジ処理の前提**: ap_byterange_filter が動作するためには、ブリゲード内に EOS バケットが存在し、かつすべてのバケットの長さが確定している必要があります。条件を満たさない場合、フィルタは自身を削除して処理をバイパスします。
  • **チャンクフィルタのスタック変数**: ap_http_chunk_filter 内の chunk_hdr はスタック上に確保されるため、関数を抜ける前にデータがコピーされる必要があります。

よくある落とし穴

  • **不適切なフィルタ順序**: フィルタの登録順序や型定義（ap_filter_type）が不適切な場合、意図しない順序でフィルタが実行され、データが破損する可能性があります。
  • **データ長不明時のバイパス**: パイプなどから読み込まれたデータでバケット長が不明（-1）な場合、ap_byterange_filter は安全のために処理をスキップします。これを考慮しないと、レンジリクエストが機能しない原因になります。
  • **エラー時の不完全な終了処理**: ap_http_chunk_filter において、バックエンドエラー（HTTP_BAD_GATEWAY）が発生した場合は、最終チャンクマーカー（0\r\n\r\n）の付与を意図的にスキップします。これを誤って付与すると、クライアントが不完全なレスポンスを正常終了と誤認する落とし穴があります。
