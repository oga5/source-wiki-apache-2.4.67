Multi-Processing Modules (MPM) Architecture

概念

Multi-Processing Modules (MPM) Architecture は、Apache HTTP Server がネットワークポートをバインドし、クライアントからの接続を受け付け、それらを処理するためのプロセスやスレッドを管理する並行処理モデルの基盤です。Apache は、異なるオペレーティングシステムや要件に合わせて、異なる並行処理モデル（Prefork、Worker、Event など）をモジュールとして差し替えることができる設計になっています。

なぜ重要か

Web サーバーのパフォーマンス、スケーラビリティ、およびリソース消費は、並行処理モデルに強く依存します。

  • **Prefork (prefork.c)**: スレッドを使用せず、1 つのリクエストを 1 つのプロセスで処理するため、スレッドセーフでないライブラリとの互換性に優れています。
  • **Worker (worker.c)**: マルチプロセスとマルチスレッドのハイブリッドモデルであり、メモリ消費を抑えつつ高い並行性を実現します。
  • **Event (event.c)**: Worker モデルをベースに、Keep-Alive 接続などの非同期処理をリスナースレッドに委ねることで、さらに効率的なリソース利用を可能にします。

共通の API インターフェース（ap_mpm.h）を介してこれらを提供することで、コアサーバーのコードを変更することなく、最適な並行処理モデルを選択できます。

関与するモジュール

  • module:server/mpm

主要エンティティ

  • **include/ap_mpm.h**: MPM の共通インターフェースを定義します。サーバーコアや他のモジュールが、現在実行中の MPM の特性（スレッド化されているか、動的なプロセス生成を行うかなど）を問い合わせるための ap_mpm_query などの関数を提供します。
  • **server/mpm/event/event.c**: イベント駆動型の非同期処理をサポートする Event MPM の実装ファイル。
  • **server/mpm/worker/worker.c**: マルチプロセス・マルチスレッド型の Worker MPM の実装ファイル。
  • **server/mpm/prefork/prefork.c**: シングルスレッド・マルチプロセス型の Prefork MPM の実装ファイル。

エンドツーエンドの流れ

親プロセスが起動し、設定された MPM（Event、Worker、Prefork のいずれか）を初期化して起動するまでの詳細な制御フロー、および各 MPM 内部での接続の受け入れからワーカースレッド/プロセスへのディスパッチに関する完全なエンドツーエンドのフローは、提供されたソースコードの範囲からは**まだ十分に特定されていません**。

重要な不変条件

各 MPM の状態管理、スレッドプールとプロセスプールの同期、およびシグナルハンドリングにおける厳密な不変条件（Invariants）は、提供された情報からは**まだ十分に特定されていません**。

よくある落とし穴

  • **スレッド安全性の欠如**: スレッドセーフでないサードパーティ製モジュールや PHP 拡張などを、マルチスレッドモデルである Worker や Event MPM で動作させると、メモリ破損や予期しないクラッシュを引き起こす可能性があります。その場合は Prefork MPM を使用する必要があります。
  • **リソースの不適切なチューニング**: 各 MPM の設定値（MaxRequestWorkers、ThreadsPerChild、MinSpareThreads など）の不適切な設定は、プロセスのスラッシングやメモリ枯渇、あるいは接続の拒否を招く原因となります。

  • server/mpm/event/event.c — 非同期イベント駆動モデルの実装詳細。
