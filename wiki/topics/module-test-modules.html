Module: test/modules

概念

test/modules は、Apache HTTP Server (httpd) の各機能モジュール（Core, HTTP/1.x, HTTP/2, Managed Domains (MD), Proxy）に対する自動テストおよび統合テストを提供するテストスイートです。

Pythonのテストフレームワーク（各ディレクトリに conftest.py が存在することから pytest と推測されます）を利用し、実際のサーバー動作、プロトコル準拠、エッジケース、セキュリティ設定などを検証するためのテストスクリプト群（test_*.py）と、テスト用のCGIスクリプトやヘルパーCモジュールで構成されています。

なぜ重要か

httpdのコア機能や拡張モジュール（特に複雑なプロトコルを扱う mod_http2 や、証明書管理を行う mod_md、プロキシを行う mod_proxy）の変更が、既存の動作を破壊しないか（デグレード防止）を保証するために極めて重要です。

また、C言語で書かれたカスタムテストモジュール（mod_h1test.c、mod_h2test.c）を組み合わせて、通常のクライアントからは再現が難しいプロトコルレベルの挙動（トレーラーヘッダー、フレーム長、タイムアウトなど）をシミュレートし、堅牢性を検証できます。

関与するモジュール

提供されたコンテキストに基づき、以下のモジュールが関与しています：

  • module:test/modules: テストスイート全体のルート。
  • module:test/modules/core: httpdコア機能（エンコーディング、再起動など）のテスト。
  • module:test/modules/http1: HTTP/1.xプロトコル、CGI、および mod_h1test を用いたテスト。
  • module:test/modules/http2: HTTP/2プロトコル、接続再利用、SSL再ネゴシエーション、プッシュ、プロキシ、WebSockets、および mod_h2test を用いた広範なテスト。
  • module:test/modules/md: Managed Domains（ACMEプロトコル、証明書管理、自動更新、Staplingなど）のテスト。
  • module:test/modules/proxy: HTTPプロキシおよびUnixドメインソケットプロキシのテスト。

主要エンティティ

提供されたファイル群から識別できる主要なエンティティは以下の通りです：

  • **テスト設定・環境定義 (conftest.py, env.py, md_env.py)**: 各テストモジュールにおけるテスト環境の初期化、httpdの設定、テスト用フィクスチャを定義します。
  • **テストスクリプト (test_*.py)**: 実際のテストケース。例えば、HTTP/2のウィンドウ制御、タイムアウト、プロキシ動作、MDのACME連携などを検証します。
  • **テスト用CGIスクリプト (hello.py, upload.py, echo.py など)**: HTTPリクエストを受け取り、特定のレスポンスやエラーを返すことで、サーバー側のハンドリングをテストするためのスクリプト。
  • **テスト用Cモジュール (mod_h1test.c, mod_h2test.c)**: テスト実行時にhttpdにロードされ、特殊なプロトコル挙動（HTTP/2フレームの直接操作など）をテストするために使用されるヘルパーモジュール。
  • **MDテスト用ヘルパー (dns01.py, md_acme.py, md_certs.py など)**: ACMEチャレンジ（DNS-01など）や証明書検証、通知処理をシミュレート・検証するためのユーティリティ。

エンドツーエンドの流れ

テスト実行の具体的なエンドツーエンドの起動フローや、テスト環境（httpdプロセス）の起動・停止のライフサイクル、テストランナーの詳細な連携シーケンスは、提供されたファイルパスのメタデータのみからは完全には特定されていません。

一般的には、テストランナーが起動されると、各ディレクトリの conftest.py および env.py がテスト用の一時的なhttpdインスタンスを設定・起動し、Pythonのテストコードから curl やカスタムクライアント（ws_server.py など）を介してリクエストを送信し、レスポンスやサーバーログを検証する流れになると推測されます。

重要な不変条件

テスト実行時における具体的な不変条件（例：テスト用ポートの競合回避、テスト用証明書の有効期限、クリーンアップ処理の保証など）の詳細は、提供されたソースコードのメタデータからは直接確認できていません。

ただし、test_910_cleanups.py や conftest.py の存在から、テスト実行後に環境が汚染されないようクリーンアップが行われること、および各テストが独立して実行可能であることが期待されます。

よくある落とし穴

提供されたファイルパス情報のみからは、具体的な実装上の落とし穴（例：特定のPythonライブラリへの依存、テスト実行時の権限問題、ポート競合、外部ACMEサーバーへの依存など）は特定されていません。

しかし、test/modules/md においてACMEエラー（test_740_acme_errors.py）やセットアップエラー（test_741_setup_errors.py）のテストが存在することから、外部APIシミュレーションの成否がテストの安定性に影響しやすいことが推測されます。
