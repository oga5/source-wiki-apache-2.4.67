Connection and Request Filtering via Bucket Brigades

概念

Apache HTTP Serverにおけるフィルタリングメカニズムは、入力および出力データを **Bucket Brigades**（バケットブリゲード：データのコンテナであるバケットの双方向連結リスト）を介して段階的に処理するアーキテクチャです。

この仕組みにより、ネットワークから読み込まれた生データは、接続（Connection）レベルおよびリクエスト（Request）レベルのフィルタチェーンを経由して、プロトコル解析、データの変換（圧縮や暗号化など）、およびコンテンツの生成へとシームレスに渡されます。

なぜ重要か

  • **メモリ効率の最適化**: データを一括でメモリに展開するのではなく、バケット単位でストリーミング処理するため、大容量ファイルの転送時でもメモリ消費量を最小限に抑えられます。
  • **高いモジュール性と柔軟性**: SSL/TLS処理、データの圧縮（DEFLATE）、HTTPプロトコルのフレーミングなどの個々の機能を独立したフィルタとして実装し、動的にチェーンへ挿入・結合できます。

関与するモジュール

  • **module:include** (include/util_filter.h): フィルタの登録、チェーンの構築、およびフィルタ操作のためのコアAPIとデータ構造を定義します。
  • **module:modules/http** (modules/http/http_core.c, modules/http/http_filters.c): HTTPプロトコル固有の入力・出力フィルタ（ヘッダー処理、チャンクエンコーディングなど）を実装し、リクエスト処理ライフサイクルに統合します。

主要エンティティ

  • **ap_filter_t** (include/util_filter.h): 実行中のフィルタインスタンスを表す構造体。チェーン内の次のフィルタ（next）へのポインタや、関連するリクエスト（request_rec）、接続（conn_rec）のコンテキストを保持します。
  • **ap_filter_rec_t** (include/util_filter.h): 登録されたフィルタの静的定義。フィルタ名、処理関数ポインタ、およびフィルタの適用タイプ（プロトコルレベル、コンテンツレベルなど）を保持します。
  • **ap_http_filter / ap_http_header_filter** (modules/http/http_filters.c): HTTPリクエストボディの読み込みや、HTTPレスポンスヘッダーの生成・送信を担う、HTTPプロトコル層の具体的なフィルタ実装です。

エンドツーエンドの流れ

現時点では、ネットワークソケットの初期化から各フィルタの呼び出しに至る完全なエンドツーエンドの実行フローおよびトリガー条件の全容は完全には特定されていません。

一般的な処理の流れは以下の通りです： 1. クライアントからの接続が確立されると、接続レベルのフィルタチェーンが構築されます。 2. リクエストヘッダーが解析された後、リクエストレベルの入力・出力フィルタ（ap_http_filter や ap_http_header_filter など）がチェーンに挿入されます。 3. コンテンツハンドラがデータを生成して出力する際、データは apr_bucket_brigade に格納され、出力フィルタチェーンを順次通過しながら加工され、最終的にネットワークへ書き込まれます。

重要な不変条件

現時点では、ソースコードから抽出された厳密な不変条件の全容は完全には特定されていません。

一般的に維持されるべき不変条件は以下の通りです：

  • **チェーンの伝播**: 各フィルタは、自身が処理した（または処理をスキップした）Bucket Brigadeを、必ず次のフィルタ（f->next）に渡さなければなりません。
  • **メタデータバケットの尊重**: EOS（End of Stream）や FLUSH などのメタデータバケットは、データの終端やフラッシュのタイミングを示す重要なシグナルであり、フィルタ内で不適切に破棄してはなりません。

よくある落とし穴

  • **メタデータバケットのハンドリング漏れ**: APR_BUCKET_IS_METADATA で判定されるバケット（特に EOS）を適切に後続フィルタに転送しないと、接続がハングアップしたり、クライアントへのレスポンスが未完了のまま終了したりする原因になります。
  • **フィルタの挿入順序の誤り**: フィルタのタイプ（ap_filter_type）や登録順序が不適切である場合、例えば圧縮フィルタが暗号化フィルタの後に実行されてしまい、データが正しく処理できなくなるなどの問題が発生します。
  • **ブリゲードのライフサイクル管理不備**: 一時的に作成したバケットやブリゲードのクリーンアップを怠ると、メモリリークや、解放済みメモリへの不正アクセス（Use-After-Free）を引き起こす可能性があります。

次に読む場所

  • **include/util_filter.h**: フィルタAPIの基本構造、フィルタタイプ、およびチェーン操作関数の定義を確認してください。
  • **modules/http/http_filters.c**: HTTPプロトコルにおける具体的な入力・出力フィルタの実装例を読み、Bucket Brigadeがどのように操作されるかを理解してください。
  • **modules/http/http_core.c**: HTTPコアモジュールにおけるフィルタの登録と初期化のプロセスを参照してください。
