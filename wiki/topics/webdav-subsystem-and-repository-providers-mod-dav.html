WebDAV Subsystem and Repository Providers (mod_dav)

概念

mod_dav は、Apache HTTP Server における WebDAV（Web-based Distributed Authoring and Versioning）プロトコルのコア実装です。このサブシステムは、WebDAV プロトコル処理（リクエストのパース、レスポンスの生成、プロパティ管理など）を行うコアレイヤーと、実際のデータストレージを抽象化するリポジトリプロバイダレイヤーに分離されています。

ファイルシステムをバックエンドとする標準的なリポジトリプロバイダ（dav_fs）や、プロパティデータベース、ロック管理機構などが協調して動作します。

なぜ重要か

WebDAV は、HTTP を拡張してクライアントがサーバー上のファイルを直接編集、作成、削除、ロックできるようにするプロトコルです。mod_dav のモジュール化されたアーキテクチャにより、以下のメリットが得られます。

  • **ストレージの抽象化**: コアロジック（mod_dav.c）を変更することなく、ファイルシステム（dav_fs）やその他のカスタムデータベース、バージョン管理システムをリポジトリとして統合できます。
  • **堅牢な並行制御**: 複数クライアントによる同時編集の衝突を防ぐためのロック管理（lock.c）を提供します。
  • **メタデータ管理**: リソースに関連付けられた任意のメタデータ（Dead/Live properties）を効率的に管理（props.c）できます。

関与するモジュール

  • module:modules/dav: WebDAV サブシステム全体を格納する親モジュール。
  • module:modules/dav/main: WebDAV のコアプロトコルハンドラおよびプロパティ管理の実装（mod_dav.c, props.c）。
  • module:modules/dav/fs: ファイルシステムをバックエンドとするリポジトリおよびロックプロバイダの実装（repos.c, lock.c）。

主要エンティティ

  • **dav_provider (構造体)**: リポジトリ操作（作成、削除、移動、コピーなど）を抽象化するインターフェース。modules/dav/fs/repos.c でファイルシステム向けの実装が定義されます。
  • **dav_hooks_repository**: リポジトリプロバイダが実装すべきコールバック関数群。
  • **dav_hooks_locks**: ロックの作成、検証、削除を管理するためのフック群（modules/dav/fs/lock.c で実装）。
  • **dav_hooks_propdb**: プロパティデータベースを操作するためのフック群（modules/dav/main/props.c で使用）。
  • **dav_resource**: 操作対象となる WebDAV リソース（ファイルやコレクション）を表現する構造体。

エンドツーエンドの流れ

WebDAV リクエストが処理される大まかな流れは以下の通りですが、詳細なコールスタックや内部状態遷移の全容は提供されたソースコードの断片からは完全には特定されていません。

1. **リクエスト受信**: クライアントから PROPFIND、PROPPATCH、LOCK などの WebDAV メソッドが送信されると、modules/dav/main/mod_dav.c のハンドラがリクエストをトラップします。 2. **プロバイダの特定**: 設定されたディレクトリやロケーションに対応するリポジトリプロバイダ（dav_provider）が特定されます。 3. **プロパティ・ロック処理**:

  • プロパティ操作の場合、modules/dav/main/props.c を介してプロパティデータベースにアクセスします。
  • ロック操作や書き込み時のロック検証の場合、modules/dav/fs/lock.c などのロックプロバイダが呼び出されます。

    • 4. **リポジトリ操作の実行**: modules/dav/fs/repos.c などのプロバイダが、実際のファイルシステムに対してファイルやディレクトリの作成・変更・削除を実行します。

重要な不変条件

詳細な不変条件（Invariants）は提供されたコンテキストからは完全には特定されていませんが、設計上以下の不変条件が維持される必要があります。

  • **プロバイダインターフェースの一貫性**: すべてのリポジトリプロバイダは、コア（mod_dav）に対して dav_provider 構造体で定義された API を厳密に遵守しなければなりません。
  • **ロックの排他性**: 有効なロックが存在するリソースに対する書き込み操作（PUT, DELETE など）は、適切なロックトークンが提示されない限り、ロックプロバイダ（lock.c）によって拒否されなければなりません。

よくある落とし穴

具体的な落とし穴の詳細は提供されたソースコードからは直接特定されていませんが、一般的な構成において以下の点に注意が必要です。

  • **アクセス権限の不整合**: ファイルシステムプロバイダ（dav_fs）を使用する場合、Web サーバープロセス（httpd）を実行するユーザーが、対象のディレクトリ、ファイル、およびロックデータベースファイルに対して適切な読み書き権限を持っている必要があります。権限が不足していると、WebDAV 操作が 403 Forbidden や 500 Internal Server Error で失敗します。
  • **ロックデータベースの破損**: サーバーの異常終了などにより、ロック情報を管理するデータベースファイルが破損し、リソースが不当にロックされたままになる可能性があります。

次に読む場所

  • modules/dav/main/mod_dav.c: WebDAV プロトコルのメインハンドラとコアロジック。
  • modules/dav/fs/repos.c: ファイルシステムベースのリポジトリプロバイダの具体的な実装。
  • modules/dav/fs/lock.c: ファイルシステムベースのロック管理の実装。
  • modules/dav/main/props.c: WebDAV プロパティ（Dead/Live properties）の管理ロジック。
