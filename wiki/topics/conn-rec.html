conn_rec

概念

conn_rec は、Apache HTTP Server においてクライアントとサーバー間の個別の接続（Connection）状態を保持・管理するための中心的なデータ構造（構造体）です。接続ごとのIPアドレス、ソケット、フィルタチェーン、Keep-Alive 状態などのコンテキストを保持し、接続の確立から終了までのライフサイクルを通じて各処理フェーズに受け渡されます。

なぜ重要か

conn_rec は、Apache HTTP Server の接続処理の基盤として以下の役割を担います。

1. **接続状態の一元管理**: フィルタチェーンやソケット情報、メモリプール（APRプール）などの接続固有のコンテキストを保持し、ap_process_connection などのコア関数を通じて処理フローを制御します。 2. **HTTP/2 ワーカーとの連携**: HTTP/2 モジュール（h2_workers）において、ワーカーが非同期または並行して処理する個別の接続状態を表現し、ストリーム処理やフィルタ処理を実行するために不可欠です。

エンドツーエンドの流れ

提供された情報に基づく、接続処理のおおよその流れは以下の通りです。

1. **接続の確立と初期化**: 接続が確立されると conn_rec が生成され、APRプール管理下で初期化されます。 2. **ワーカーによる取得**: HTTP/2 の文脈では、h2_workers のワーカーが next コールバックを介して conn_rec を取得します。 3. **処理の実行**: ワーカーは取得した conn_rec に対して、HTTP/2 ストリーム処理やフィルタ処理を実行します。コアの ap_process_connection などの接続管理関数もこの構造体を操作します。 4. **処理の完了と解放**: 処理終了後、done コールバック（ap_conn_producer_done など）が呼び出され、接続の解放や後処理が行われます。

※なお、接続の確立から破棄に至る詳細な内部状態遷移や、コアとHTTP/2モジュール間における完全なエンドツーエンドの制御フローの全容は、提供された情報からは未確定です。

重要な不変条件

  • **ライフサイクルとメモリ管理**: conn_rec 内の多くのフィールドは、Apache Portable Runtime (APR) のプール管理下にあり、接続のライフサイクルを通じて維持される必要があります。
  • **排他制御**: 複数のスレッドから同一の conn_rec を同時に操作することは禁止されており、HTTP/2 モジュール内では h2_mplx による排他制御が前提となります。
  • **ワーカー処理中の有効性**: h2_workers の文脈において、ワーカーが取得した conn_rec は、処理が完了するまで有効でなければなりません。

よくある落とし穴

  • **直接的な構造体変更の制限**: conn_rec は非常に広範かつ重要なデータ構造であるため、モジュール開発時に不必要な変更を加えることは避けるべきです。構造体の変更は ap_mmn.h で定義されるモジュールマジックナンバー（MMN）の更新を伴い、バイナリ互換性に影響を与える可能性があります。
  • **スレッドセーフティ**: h2_mplx による適切な排他制御を欠いた状態で、複数スレッドから同一の conn_rec を操作すると、予期せぬ競合状態を引き起こす原因となります。
