conn_rec

概念

conn_rec は、Apache HTTP Server においてクライアントとサーバー間の個別の接続（Connection）状態を保持するための中心的なデータ構造です。コアシステム（include/httpd.h）で定義され、HTTP/2 モジュール（modules/http2/h2_workers.h）などの各モジュールにおいて、ワーカーが処理する個別の接続状態を表現・管理するために広く参照されます。

なぜ重要か

この構造体は、接続のライフサイクルを通じて維持されるコンテキストであり、IPアドレス、ソケット情報、フィルタチェーン、Keep-Alive 状態などの接続ごとの重要な状態を保持します。HTTP/2 などのマルチスレッドや非同期処理を行うモジュールにおいても、ワーカーが適切な接続コンテキストを取得・処理し、ライフサイクルを管理するための基盤として機能するため極めて重要です。

関与するモジュール

  • module:include
  • module:modules/http2

主要エンティティ

  • symbol:include/httpd.h#conn_rec: 接続状態を保持する中心的な構造体定義。
  • symbol:modules/http2/h2_workers.h#conn_rec: HTTP/2 ワーカーが処理する接続コンテキストとしての参照。

エンドツーエンドの流れ

提供された情報に基づく、接続処理の基本的な流れは以下の通りです：

1. **接続の確立と受け渡し**: 接続が確立されると、conn_rec が作成され、ap_process_connection などの関数を通じて接続の処理フローへ渡されます。 2. **ワーカーによる取得**: HTTP/2 モジュール（h2_workers）の文脈では、ワーカーが next コールバックを介して conn_rec を取得します。 3. **処理の実行**: ワーカーは取得した conn_rec に対して、HTTP/2 のストリーム処理やフィルタ処理を実行します。 4. **完了と解放**: 処理終了後、done コールバック（ap_conn_producer_done など）が呼び出され、接続の解放や後処理が行われます。

※なお、接続の確立から破棄に至る詳細なコア側のエンドツーエンドの全フローや、内部の具体的な状態遷移の全容は、提供された情報からは完全には特定されていません。

重要な不変条件

  • **ライフサイクルの維持**: conn_rec は接続のライフサイクルを通じて維持されるコンテキストであり、構造体内の多くのフィールドは Apache Portable Runtime (APR) のプール管理下にあります。
  • **ワーカー処理中の有効性**: h2_workers が取得した conn_rec は、処理が完了するまで有効である必要があります。
  • **排他制御**: 複数のスレッドから同一の conn_rec を操作することは禁止されており、h2_mplx による排他制御が前提となります。

よくある落とし穴

  • **スレッドセーフティ**: 同一の conn_rec を複数スレッドから同時に操作すると競合が発生するため、必ず h2_mplx 等による適切な排他制御を行う必要があります。
  • **構造体の変更と互換性**: conn_rec は非常に広範かつ中心的なデータ構造です。モジュール開発時に不必要な変更を加えることは避け、構造体を変更する場合は ap_mmn.h で定義されるモジュールマジックナンバー（MMN）の更新を伴うバイナリ互換性の影響に注意する必要があります。
  • **メモリ管理**: 接続切断やメモリ不足（OOM）時の挙動は、APR プール管理および ap_abort_on_oom の仕様に依存するため、メモリのライフサイクル管理に注意してください。
