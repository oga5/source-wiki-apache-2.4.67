conn_rec

概念

conn_rec は、Apache HTTP Server においてクライアントとサーバー間の個別の接続（Connection）状態を保持するための中心的なデータ構造（struct / typedef）です。

なぜ重要か

conn_rec は、接続ごとの状態（IPアドレス、ソケット、フィルタチェーン、Keep-Alive 状態など）を管理し、接続の確立から終了までのライフサイクルを通じてコンテキストを維持するために不可欠です。HTTP/2 モジュール（mod_http2）においては、ワーカーが処理する個別の接続状態を表現し、ストリーム処理やフィルタ処理を実行する際にも使用されます。

関与するモジュール

  • module:include
  • module:modules/http2

主要エンティティ

  • symbol:include/httpd.h#conn_rec: コアで定義される、接続状態を保持する中心的な構造体。
  • symbol:modules/http2/h2_workers.h#conn_rec: HTTP/2 のワーカー処理において、処理対象の接続コンテキストとして参照される型。

エンドツーエンドの流れ

提供された情報に基づく、接続処理の基本的な流れは以下の通りです。

1. **接続の確立と管理**: ap_process_connection 等のコア関数を通じて、接続の確立から終了までの処理フローで conn_rec が受け渡されます。 2. **HTTP/2 ワーカーによる取得**: h2_workers のワーカーが next コールバックを介して conn_rec を取得します。 3. **処理の実行**: ワーカーは取得した conn_rec に対して、HTTP/2 のストリーム処理やフィルタ処理を実行します。 4. **完了と解放**: 処理終了後、done コールバック（ap_conn_producer_done など）が呼び出され、接続の解放や後処理が行われます。

※ 接続の完全な生成から破棄に至る全ステップや、内部フィルタチェーンの具体的な遷移など、詳細なエンドツーエンドのフローは提供された情報からは完全には判明していません。

重要な不変条件

  • **メモリ管理**: conn_rec 内の多くのフィールドは、Apache Portable Runtime (APR) のプール管理下にあり、接続のライフサイクルを通じて維持されます。
  • **スレッド安全性**: 複数のスレッドから同一の conn_rec を操作することは禁止されており、HTTP/2 モジュール内では h2_mplx による排他制御が前提となります。
  • **ライフサイクルの整合性**: h2_workers の文脈において、ワーカーが取得した conn_rec は、処理が完了するまで有効である必要があります。

※ その他の詳細な不変条件や、コアと各モジュール間における厳密な状態遷移の制約は、提供された情報からは完全には判明していません。

よくある落とし穴

  • **バイナリ互換性 (MMN)**: conn_rec は非常に広範なデータ構造です。構造体を変更すると、ap_mmn.h で定義されるモジュールマジックナンバー（MMN）の更新を伴う場合があります。モジュール開発時は、必要なフィールドのみにアクセスし、不必要な変更を避けてください。
  • **HTTP/2 モジュール内での変更禁止**: modules/http2/h2_workers.h における conn_rec はコア定義への参照です。HTTP/2 モジュール側で構造体自体を変更してはならず、コア側の API 変更に注意する必要があります。
