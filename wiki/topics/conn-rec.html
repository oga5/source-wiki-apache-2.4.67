conn_rec

概念

conn_rec は、Apache HTTP Server においてクライアントとサーバー間の個別の接続（Connection）状態を保持するための中心的なデータ構造です。コアシステムにおける接続管理だけでなく、HTTP/2 などの個別プロトコルモジュールにおけるワーカー処理のコンテキストとしても使用されます。

なぜ重要か

この構造体は、接続のライフサイクル全体（IPアドレス、ソケット、フィルタチェーン、Keep-Alive 状態など）を管理するための基盤です。HTTP/2 モジュール（mod_http2）においては、マルチスレッド環境下でワーカーが個別の接続を安全かつ効率的に処理するためのコンテキスト表現として機能します。また、この構造体の変更はモジュールマジックナンバー（MMN）の更新を伴うため、バイナリ互換性の観点からも極めて重要です。

エンドツーエンドの流れ

提供された情報に基づく、接続処理のおおよその流れは以下の通りです。ただし、詳細なエンドツーエンドの全フローは完全には開示されていません（未検証の部分を含みます）。

1. **接続の確立と受け渡し**: 接続が確立されると、conn_rec が生成され、ap_process_connection などのコア関数を通じて処理フローに渡されます。 2. **HTTP/2 ワーカーによる取得**: h2_workers のワーカー（slot_run など）が next コールバックを介して conn_rec を取得します。 3. **処理の実行**: ワーカーは取得した conn_rec に対して、HTTP/2 のストリーム処理やフィルタ処理を実行します。 4. **完了と解放**: 処理終了後、done コールバック（ap_conn_producer_done など）が呼び出され、接続の解放や後処理が行われます。

重要な不変条件

  • **ライフサイクルの維持**: 接続のライフサイクルを通じて conn_rec コンテキストは維持される必要があります。また、多くのフィールドは APR（Apache Portable Runtime）のプール管理下にあります。
  • **ワーカー処理中の有効性**: h2_workers の文脈において、ワーカーが取得した conn_rec は処理が完了するまで有効でなければなりません。
  • **排他制御**: 複数のスレッドから同一の conn_rec を同時に操作することは禁止されており、h2_mplx による排他制御が前提となります。

よくある落とし穴

  • **スレッドセーフティの欠如**: 同一の conn_rec を複数スレッドから直接操作すると競合が発生するため、h2_mplx 等による適切な排他制御を怠らないようにする必要があります。
  • **不要な構造体変更**: conn_rec は広範に参照されるため、モジュール開発時に不必要な変更を加えると、ap_mmn.h で管理されるモジュールマジックナンバー（MMN）の互換性に影響を及ぼします。
  • **メモリ管理**: 構造体内の多くのフィールドが APR プールに依存しているため、プール外での不正なライフサイクル管理や、OOM（メモリ不足）発生時の挙動（ap_abort_on_oom 依存）に注意する必要があります。
