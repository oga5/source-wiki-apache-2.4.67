conn_rec

概念

conn_rec は、Apache HTTP Server においてクライアントとサーバー間の個別の接続（Connection）状態を保持・管理するための中心的なデータ構造です。コアシステム（module:include）で定義され、HTTP/2 モジュール（module:modules/http2）などの各モジュールにおいて、接続コンテキストの受け渡しやワーカーによる処理の対象として広く利用されます。

なぜ重要か

conn_rec は、接続のライフサイクルを通じて維持されるコンテキストであり、IPアドレス、ソケット情報、フィルタチェーン、Keep-Alive 状態などの接続固有の情報を一元管理します。HTTP/2 モジュールにおいては、ワーカープール（h2_workers）が個別の接続状態を識別し、ストリーム処理やフィルタ処理を適切に実行するために不可欠なデータ構造となっています。

関与するモジュール

  • module:include
  • module:modules/http2

主要エンティティ

  • symbol:include/httpd.h#conn_rec: コアで定義される接続状態管理用の構造体。
  • symbol:modules/http2/h2_workers.h#conn_rec: HTTP/2 ワーカーの処理対象として参照される接続コンテキストの型定義。

エンドツーエンドの流れ

1. **接続の確立と初期化**: 接続が確立されると conn_rec が生成され、ap_process_connection などの関数を通じて接続処理フローに渡されます。 2. **ワーカーによる取得**: HTTP/2 モジュールにおいて、h2_workers のワーカーが next コールバックを介して conn_rec を取得します。 3. **処理の実行**: ワーカーは取得した conn_rec に対して、HTTP/2 のストリーム処理やフィルタ処理を実行します。 4. **完了と解放**: 処理終了後、ap_conn_producer_done などの done コールバックが呼び出され、接続の解放や後処理が行われます。

※なお、接続の確立から破棄に至る詳細な内部イベントや、APRプールによるメモリ解放の厳密なタイミングなど、完全なエンドツーエンドのフローの全容は現時点の提供情報からは未確定です。

重要な不変条件

  • **スレッド間の排他制御**: 複数のスレッドから同一の conn_rec を同時に操作することは禁止されており、HTTP/2 モジュール内では h2_mplx による排他制御が前提となります。
  • **ライフサイクルの維持**: h2_workers の文脈において、ワーカーが取得した conn_rec は、その処理が完了するまで有効である必要があります。
  • **メモリ管理**: 構造体内の多くのフィールドは、Apache Portable Runtime (APR) のプール管理下にあります。

よくある落とし穴

  • **マルチスレッド競合**: h2_mplx による適切な排他制御を行わずに、複数のスレッドから同一の conn_rec を操作すると、競合状態を引き起こす可能性があります。
  • **バイナリ互換性の破壊**: conn_rec はコアの広範なデータ構造であり、構造体を直接変更するとモジュールマジックナンバー（MMN）の更新が必要になります。HTTP/2 モジュールなどの個別モジュール内からコア側の構造体を直接変更することは避け、APIを介した操作に留める必要があります。
