conn_rec

概念

conn_rec は、Apache HTTP Server においてクライアントとサーバー間の個別の接続（Connection）状態を保持するための中心的なデータ構造です。コアシステム（include/httpd.h）で定義され、HTTP/2 モジュール（modules/http2/h2_workers.h）などの各モジュールにおいて、接続コンテキストやワーカーが処理する個別の接続状態を表現するために広く参照・利用されます。

なぜ重要か

このデータ構造は、接続の確立から終了までのライフサイクル全体にわたって、IPアドレス、ソケット情報、フィルタチェーン、Keep-Alive 状態などの接続固有のコンテキストを維持するために不可欠です。HTTP/2 モジュールにおいては、ワーカープール（h2_workers）が個別の接続を処理する際に対象となる接続状態を管理・追跡するための基盤となります。

関与するモジュール

  • module:include
  • module:modules/http2

主要エンティティ

  • symbol:include/httpd.h#conn_rec: コアで定義される、接続状態を保持する中心的な構造体。
  • symbol:modules/http2/h2_workers.h#conn_rec: HTTP/2 ワーカーの文脈で、処理対象の接続状態を表現するために参照される型定義。

エンドツーエンドの流れ

1. **接続の確立と管理**: 接続が確立されると conn_rec が生成され、ap_process_connection などの関数を通じて接続のライフサイクルが管理されます。 2. **HTTP/2 ワーカーによる取得**: h2_workers 内のワーカーが next コールバックを介して処理対象の conn_rec を取得します。 3. **処理の実行**: ワーカーは取得した conn_rec に対して、HTTP/2 のストリーム処理やフィルタ処理を実行します。 4. **処理の完了**: 処理終了後、done コールバック（ap_conn_producer_done など）が呼び出され、接続の解放や後処理が行われます。

※なお、接続の確立から破棄に至る詳細なコア側の全体フローや、APRプール管理の具体的な内部動作など、一部のエンドツーエンドのフローの詳細は本コンテキストからは完全には明らかになっていません。

重要な不変条件

  • **ライフサイクル中の有効性**: h2_workers の文脈において、ワーカーが取得した conn_rec は、その処理が完了するまで有効でなければなりません。
  • **排他制御**: 複数のスレッドから同一の conn_rec を操作することは禁止されており、HTTP/2 モジュール内では h2_mplx による排他制御が前提となります。
  • **メモリ管理**: 構造体内の多くのフィールドは、Apache Portable Runtime (APR) のプール管理下に置かれます。

よくある落とし穴

  • **不必要な変更とバイナリ互換性**: conn_rec は非常に広範かつ重要なデータ構造です。モジュール開発時に不必要な変更を加えることは避ける必要があります。構造体の変更は ap_mmn.h で定義されるモジュールマジックナンバー（MMN）の更新を伴うため、バイナリ互換性に影響を及ぼします。
  • **スレッドセーフティ**: h2_mplx による適切な排他制御を欠いた状態で、複数スレッドから同一の conn_rec を操作すると、予期しない競合や破損を招く恐れがあります。
