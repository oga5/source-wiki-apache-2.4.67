accept_mutex_on

概念

accept_mutex_on は、Apache HTTP Server のマルチプロセッシングモジュール（MPM）およびテストスイートにおいて、複数のプロセスが同時に accept() を呼び出す際の競合を防ぐために、プロセス間の排他ロック（ミューテックス）を取得する関数です。

本番環境の実装（prefork MPM）と、テスト環境（test/time-sem.c）の双方に同名の関数が存在し、それぞれ異なるロック機構を用いてプロセス間の同期を実現しています。

なぜ重要か

  • **prefork MPM における競合回避**: 複数の子プロセスが同一のリスニングソケットに対して安全に accept() を実行できるようにし、接続処理の衝突やリソースの無駄な消費（サンダリングハード問題など）を防ぐために不可欠です。
  • **テスト環境でのシミュレーション**: test/time-sem.c においては、fcntl などの異なるロック方式を抽象化し、プロセス間同期の挙動や性能を検証するために重要な役割を果たします。

関与するモジュール

  • module:server/mpm/prefork
  • module:test

主要エンティティ

  • **symbol:server/mpm/prefork/prefork.c#accept_mutex_on**: prefork MPM における実装。APR（Apache Portable Runtime）のプロセス間ミューテックス（apr_proc_mutex_lock）を利用します。
  • **symbol:test/time-sem.c#accept_mutex_on**: テスト環境における実装。fcntl システムコール（F_SETLKW）を利用したファイルロックによる排他制御を行います。

エンドツーエンドの流れ

現時点では、本番用の prefork MPM とテスト用コードの間の完全な統合フローや、システム全体におけるエンドツーエンドのライフサイクルは完全には判明していません。判明している各コンテキストにおける個別の流れは以下の通りです。

prefork MPM における流れ

1. 子プロセスから、SAFE_ACCEPT マクロを経由して accept_mutex_on() が呼び出されます。 2. グローバルな my_bucket->mutex に対して apr_proc_mutex_lock を実行し、ロックの取得を試みます。 3. ロック取得に成功した場合は、APR_SUCCESS を返して呼び出し元に復帰し、後続の accept() 処理へ進みます。 4. ロック取得に失敗した場合、現在のプロセス世代（my_generation）とスコアボード上の実行世代を比較します。

  • 世代が異なる場合は、デバッグログを出力した後に clean_child_exit(0) を呼び出して正常終了します。
  • 世代が一致する場合は、緊急ログを出力した後に exit(APEXIT_CHILDFATAL) を呼び出して異常終了します。

テスト環境（test/time-sem.c）における流れ

1. テストプログラム内の処理から accept_mutex_on() が呼び出されます。 2. fcntl システムコールを F_SETLKW コマンドで呼び出し、fcntl_fd に対する排他ロックの獲得を試みます。 3. EINTR（システムコール割り込み）が発生した場合は、ループによって処理を再試行します。 4. ロック獲得に成功した場合はそのまま復帰し、失敗した場合は perror でエラーを出力して exit(1) によりプロセスを即時終了します。

重要な不変条件

現時点でシステム全体にわたるグローバルな不変条件は未確定ですが、各関数レベルで以下の前提条件が定義されています。

  • **prefork MPM 側**:
  • 呼び出し時に my_bucket が適切に初期化されていること。
  • 本関数は子プロセスから呼び出されること。
  • **テスト環境側**:
  • fcntl_fd が有効なファイル記述子として初期化されていること。
  • lock_it 構造体が F_SETLKW 用の適切なロック設定で初期化されていること。

よくある落とし穴

  • **同名関数の混同**: prefork.c と time-sem.c に同名の関数が存在しますが、これらは独立した実装です。コードリーディングやデバッグの際に混同しないよう注意が必要です。
  • **マクロ経由の呼び出し**: prefork.c の実装は SAFE_ACCEPT マクロ経由での呼び出しを前提として設計されているため、直接呼び出すと意図しない挙動を引き起こす可能性があります。
  • **テスト用実装のブロッキングと即時終了**: test/time-sem.c の実装は F_SETLKW によるブロッキング動作を行うため、デッドロック環境下ではプロセスが永久に待機状態になります。また、エラー発生時には exit(1) でプロセスが即時終了するため、そのまま本番環境のコードへ流用することはできません。
