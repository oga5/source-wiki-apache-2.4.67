accept_mutex_on

概念

accept_mutex_on は、複数のプロセスが同時に accept() を呼び出す際の競合を防ぐために、プロセス間の排他ロック（ミューテックス）を取得するための関数です。Apache HTTP Server の本番コード（prefork MPM）とテストコード（time-sem.c）の双方に同名の関数が存在しますが、それぞれ異なる仕組み（APRのプロセス間ミューテックス、および fcntl によるファイルロック）を用いて実装されています。

なぜ重要か

マルチプロセスモデル（MPM）において、複数の子プロセスが同一のリスニングソケットに対して同時に接続を受け入れようとすると、競合や非効率なコンテキストスイッチが発生する可能性があります。accept_mutex_on は、ソケットの受け入れ処理を安全にシリアライズ（直列化）し、一度に1つのプロセスのみが安全に接続を受け入れられるように制御するために重要です。

関与するモジュール

主要エンティティ

  • **symbol:server/mpm/prefork/prefork.c#accept_mutex_on**: prefork MPM において、APR（Apache Portable Runtime）のプロセス間ミューテックス（apr_proc_mutex_lock）を用いてロックを取得する関数。
  • **symbol:test/time-sem.c#accept_mutex_on**: テスト環境において、fcntl システムコール（F_SETLKW）を用いてファイルベースの排他ロックを取得する関数。

エンドツーエンドの流れ

本番環境におけるリクエスト受付からレスポンス返却、およびロック解放に至る完全なエンドツーエンドのライフサイクルやフローの全容は、提供された情報からは明らかになっていません。

現時点で判明している部分的な実行フローは以下の通りです：

1. 本番環境 (prefork.c) での流れ

1. 子プロセスから SAFE_ACCEPT マクロを経由して accept_mutex_on が呼び出されます。 2. グローバルな my_bucket->mutex に対して apr_proc_mutex_lock を実行し、ロックの取得を試みます。 3. ロック取得に成功した場合は APR_SUCCESS を返し、呼び出し元へ復帰します。 4. ロック取得に失敗した場合、現在のプロセス世代（my_generation）とスコアボード上の実行世代を比較します。

  • 世代が異なる場合：デバッグログを出力し、clean_child_exit(0) を呼び出して正常終了します。
  • 世代が一致する場合：緊急ログを出力し、exit(APEXIT_CHILDFATAL) を呼び出して異常終了します。

2. テスト環境 (time-sem.c) での流れ

1. テスト内の複数箇所（532行目、542行目など）から accept_mutex_on が呼び出されます。 2. fcntl システムコールを F_SETLKW（ロック待ち）コマンドで呼び出し、ロックの獲得を試みます。 3. EINTR（システムコール割り込み）が発生した場合は、ループを継続して再試行します。 4. ロック獲得に失敗した場合は、perror でエラーを出力し、exit(1) でプロセスを終了します。

重要な不変条件

  • **本番環境 (prefork.c)**:
  • 呼び出し時に、グローバルな my_bucket が初期化済みである必要があります。
  • 本関数は子プロセスから呼び出されることを前提としています。
  • **テスト環境 (time-sem.c)**:
  • グローバル変数 fcntl_fd が有効なファイル記述子として初期化されている必要があります。
  • lock_it 構造体が適切なロック設定（F_SETLKW 用）で初期化されている必要があります。

よくある落とし穴

  • **テスト用実装との混同**: test/time-sem.c にも同名の accept_mutex_on が定義されていますが、これはテスト用の抽象化レイヤーの一部であり、本番の prefork MPM の実装とは直接の関係がありません。
  • **直接呼び出しの制限**: 本番環境の accept_mutex_on は SAFE_ACCEPT マクロ経由での呼び出しが前提となっているため、直接呼び出す際には注意が必要です。
  • **デッドロックによるハング**: テスト用の fcntl ベースの実装はブロッキング動作を行うため、デッドロックが発生している環境では呼び出し元が永久に待機状態になります。
  • **エラー時の即時終了**: どちらの実装も、ロック取得の致命的な失敗時にはエラーを呼び出し元に返さず、プロセスを即座に終了（exit）させる設計になっています。
