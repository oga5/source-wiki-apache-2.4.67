accept_mutex_on

概念

accept_mutex_on は、複数のプロセスが同時に accept() を呼び出すことによる競合（サンダリングハード問題など）を防ぐために、プロセス間での排他制御（ミューテックスまたはロック）を獲得するための関数です。Apache HTTP Server の prefork MPM（Multi-Processing Module）における本番実装と、テストスイート（test/time-sem.c）におけるシミュレーション/測定用の実装の2種類が存在します。

なぜ重要か

マルチプロセスモデル（特に prefork MPM）において、複数の子プロセスが同一のリスニングソケットに対して同時に accept() を試みると、非効率なリソース競合や予期しない挙動が発生する可能性があります。accept_mutex_on は、プロセス間で安全にシリアライズされた accept() 呼び出しを保証し、システムの安定性とパフォーマンスを維持するために重要です。

関与するモジュール

主要エンティティ

  • symbol:server/mpm/prefork/prefork.c#accept_mutex_on: prefork MPM における本番用の実装。apr_proc_mutex_lock を使用してプロセス間ミューテックスを取得します。
  • symbol:test/time-sem.c#accept_mutex_on: テスト環境用の実装。fcntl システムコール（F_SETLKW）を用いたファイルロックによる排他制御を行います。

エンドツーエンドの流れ

本番環境（prefork MPM）におけるエンドツーエンドの流れは以下の通りです： 1. 子プロセスが SAFE_ACCEPT マクロを経由して accept_mutex_on を呼び出します。 2. accept_mutex_on 内部で、グローバルな my_bucket（prefork_child_bucket 構造体）に保持されている mutex に対して apr_proc_mutex_lock を実行します。 3. ロックの取得に成功した場合、APR_SUCCESS を返して呼び出し元に復帰し、安全に accept() 処理へ移行します。 4. ロックの取得に失敗した場合、現在のプロセス世代（my_generation）とスコアボード上の実行世代を比較します。

  • 世代が異なる（古い世代のプロセスである）場合は、デバッグログを出力した後に clean_child_exit(0) を呼び出して正常終了します。
  • 世代が一致する場合は、重大なエラーとして緊急ログを出力し、exit(APEXIT_CHILDFATAL) を呼び出して異常終了します。

テスト環境（test/time-sem.c）における流れは以下の通りです： 1. テストプログラム内の処理（532行目、542行目など）から accept_mutex_on が呼び出されます。 2. グローバル変数 fcntl_fd および lock_it（struct flock）を使用して、fcntl システムコールを F_SETLKW コマンドで呼び出します。 3. EINTR（システムコール割り込み）が発生した場合は、ループにより自動的に再試行します。 4. ロック取得に成功すればそのまま復帰し、失敗した場合は perror でエラーを出力して exit(1) により即時終了します。

なお、これら2つの実装がシステム全体としてどのように統合され、どのようなトリガーで切り替えられるかといった、完全なエンドツーエンドの統合フローは現時点では未確定です。

重要な不変条件

  • **本番環境 (prefork.c)**:
  • 呼び出し時にグローバルな my_bucket が適切に初期化されている必要があります。
  • 本関数は子プロセスから呼び出されることを前提としています。
  • **テスト環境 (time-sem.c)**:
  • fcntl_fd が有効なファイル記述子として初期化されている必要があります。
  • lock_it 構造体が F_SETLKW 用に適切に設定されている必要があります。

よくある落とし穴

  • **本番環境 (prefork.c)**:
  • 直接呼び出すのではなく、SAFE_ACCEPT マクロを経由して呼び出すことが前提となっています。直接呼び出すと、マクロが提供する安全なラップ処理がバイパスされる恐れがあります。
  • ロック取得失敗時のエラーハンドリングにおいて、世代不一致による正常終了（clean_child_exit）と、世代一致による致命的エラー（APEXIT_CHILDFATAL）の分岐が存在するため、世代管理（prefork_retained_data）が正しく機能していないと、誤ったプロセス終了ステータスを招く可能性があります。
  • **テスト環境 (time-sem.c)**:
  • fcntl によるロック獲得はブロッキング動作（F_SETLKW）を行うため、デッドロックが発生している環境では呼び出し元プロセスが永久に待機状態（ハングアップ）になります。
  • エラー発生時には exit(1) でプロセスが即時終了するため、本番コードへの流用や、エラーからの回復が必要なコンテキストでの使用には適していません。
