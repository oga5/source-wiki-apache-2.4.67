accept_mutex_on

概念

accept_mutex_on は、複数のプロセスが同時に accept() を呼び出すことによる競合を防ぐために、プロセス間ミューテックス（排他ロック）を取得する関数です。Apache HTTP Server の prefork MPM（Multi-Processing Module）の実装、およびテストプログラム test/time-sem.c の両方に同名の関数が存在しますが、それぞれ異なる目的と同期メカニズムを持っています。

なぜ重要か

マルチプロセスモデル（特に prefork MPM）において、複数の子プロセスが同一のリスニングソケットに対して同時に接続を受け付けようとすると、競合が発生します。accept_mutex_on は、プロセス間で確実に排他制御を行うことでこの競合を防ぎ、安定した接続処理を実現するために重要です。また、テスト環境においては、異なる同期メカニズム（fcntl など）の挙動や性能を検証するための抽象化レイヤーとして機能します。

エンドツーエンドの流れ

提供された情報に基づく、それぞれのコンテキストにおける処理の流れは以下の通りです。なお、prefork MPM とテストプログラムを跨ぐ統合されたエンドツーエンドのフローや、システム全体としての詳細なライフサイクルは現時点で不明です。

1. prefork MPM における流れ

1. 子プロセスが接続を待ち受ける際、server/mpm/prefork/prefork.c 内で SAFE_ACCEPT マクロを経由して accept_mutex_on が呼び出されます。 2. グローバルな my_bucket->mutex に対して apr_proc_mutex_lock を実行し、ロックの取得を試みます。 3. ロック取得に成功した場合は APR_SUCCESS を返し、接続処理へ進みます。 4. ロック取得に失敗した場合、現在のプロセス世代（my_generation）とスコアボード上の実行世代を比較します。

  • 世代が異なる場合: デバッグログを出力し、clean_child_exit(0) を呼び出して正常終了します。
  • 世代が一致する場合: 緊急ログを出力し、exit(APEXIT_CHILDFATAL) を呼び出して異常終了します。

2. テストプログラムにおける流れ

1. test/time-sem.c 内の複数箇所から accept_mutex_on が呼び出されます。 2. グローバル変数 fcntl_fd および lock_it 構造体（struct flock）を使用し、fcntl システムコールを F_SETLKW（ロック待ち）コマンドで呼び出します。 3. EINTR（システムコール割り込み）が発生した場合は、ループを継続して再試行します。 4. ロック獲得に失敗（ret < 0）した場合は、perror でエラーを出力し、exit(1) でプロセスを終了します。

重要な不変条件

  • **prefork MPM**:
  • 呼び出し時に、グローバルな my_bucket が適切に初期化されている必要があります。
  • 本関数は子プロセスから呼び出されることを前提としています。
  • **テストプログラム**:
  • fcntl_fd が有効なファイル記述子として初期化されている必要があります。
  • lock_it 構造体が適切なロック設定（F_SETLKW 用）で初期化されている必要があります。

よくある落とし穴

  • **エラー時の即時終了**:

    • 両実装とも、ロック取得に失敗した場合は呼び出し元にエラーを返さず、プロセスを即座に終了（exit または clean_child_exit）させます。そのため、呼び出し元でエラーハンドリングを行うことはできません。

  • **ブロッキングとデッドロック**:

    • テストプログラムの fcntl を用いた実装はブロッキング動作を行うため、デッドロックが発生している環境では呼び出し元が永久に待機状態となります。

  • **名前の重複と無関係性**:

    • test/time-sem.c に存在する accept_mutex_on はテスト用の抽象化レイヤーの一部であり、prefork MPM の本番実装とは直接の関係がありません。コードリーディングや移植の際に混同しないよう注意が必要です。
