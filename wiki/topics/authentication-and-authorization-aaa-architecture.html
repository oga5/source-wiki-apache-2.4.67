Authentication and Authorization (AAA) Architecture

概念

Apache HTTP ServerにおけるAAA（認証・認可）アーキテクチャは、認証（Authentication: authn）と認可（Authorization: authz）の処理を分離し、それぞれを独立したプロバイダーとしてプラグイン可能（疎結合）にする設計パターンです。 認証プロバイダー（authn_provider）がユーザーの身元（資格情報）を検証し、認可プロバイダー（authz_provider）がそのユーザーに対するアクセス権限を判定します。

なぜ重要か

このアーキテクチャにより、認証方式（Basic認証など）と、バックエンドのユーザーデータベース（ファイル、LDAP、データベースなど）、および認可ルール（Require ディレクティブ）を柔軟に組み合わせて構成することが可能になります。コアモジュールが仲介役となることで、個々の認証・認可モジュールは自身の固有ロジックに専念できます。

関与するモジュール

  • module:modules/aaa

主要エンティティ

  • **symbol:include/mod_auth.h#authn_provider**: 認証プロバイダーが実装すべきインターフェース。パスワード検証用の check_password や、レルムハッシュ取得用の get_realm_hash を定義します。
  • **symbol:include/mod_auth.h#authz_provider**: 認可プロバイダーが実装すべきインターフェース。Require 設定行の解析を行う parse_require_line と、アクセス可否を判定する check_authorization を定義します。
  • **symbol:modules/aaa/mod_auth_basic.c#authenticate_basic_user**: HTTP Basic 認証の仲介を行う関数。登録された authn_provider を順次呼び出して資格情報を検証します。
  • **symbol:modules/aaa/mod_authz_core.c#authorize_user**: 認可処理のコア部分で呼び出される関数。

エンドツーエンドの流れ

1. **設定解析時**: サーバー起動時または設定再読み込み時に、authz_provider の parse_require_line が呼び出され、Require ディレクティブの構文チェックと最適化が行われます。 2. **リクエスト受信時（認証フェーズ）**:

  • AuthType Basic などの設定に基づき、authenticate_basic_user などの認証ハンドラが起動します。
  • クライアントから送られた Authorization ヘッダーから資格情報を抽出します。
  • 設定された authn_provider の check_password または get_realm_hash を呼び出して検証します。

    • 3. **リクエスト受信時（認可フェーズ）**:

  • 認証成功後、mod_authz_core などの認可コアモジュールが authorize_user を介して認可処理を進行します。
  • 登録された authz_provider の check_authorization を呼び出し、アクセスを許可（AUTHZ_GRANTED）するかどうかを判定します。

*注意: 認証から認可へ至る詳細な内部遷移や、authorize_user の具体的な内部フローなど、エンドツーエンドの完全な流れは現時点ではすべて解明されていません。*

重要な不変条件

  • **認証プロバイダーの戻り値**: check_password および get_realm_hash は、成否に応じて適切な authn_status を返さなければなりません。
  • **認可設定の事前解析**: parse_require_line は設定読み込み時に一度だけ実行され、リクエスト処理時の負荷を軽減するために解析結果を parsed_require_line に格納して引き渡す必要があります。
  • **権限委譲（Authoritative）**: authenticate_basic_user において、authoritative 設定が有効な場合は認証失敗時に即座に HTTP_UNAUTHORIZED を返しますが、無効な場合は DECLINED を返して後続の認証モジュールへ処理を委譲します。

*注意: モジュール間におけるその他の詳細な不変条件やスレッドセーフティに関する制約は、現時点ではすべて解明されていません。*

よくある落とし穴

  • **AuthNameの未設定**: Basic認証を使用する際、AuthName が設定されていないと authenticate_basic_user は HTTP_INTERNAL_SERVER_ERROR を返します。
  • **プロバイダーの未登録/ロード失敗**: 必要な認証プロバイダーがロードされていない、または登録されていない場合、デフォルトプロバイダーの取得に失敗し、サーバーエラー（HTTP_INTERNAL_SERVER_ERROR）となります。
  • **解析エラーによる起動失敗**: parse_require_line がエラーメッセージ（文字列）を返した場合、サーバーの設定読み込み自体が失敗し、起動できなくなります。
