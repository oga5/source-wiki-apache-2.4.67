AP_INTERNAL_REDIRECT

概念

AP_INTERNAL_REDIRECT は、Apache HTTP Serverにおいてリクエストを新しいURIに内部的にリダイレクトする処理、およびそれに関連するマクロ・関数群の総称です。

具体的には、ap_internal_redirect 関数が新しいURIに基づくリクエストコンテキストの構築と処理フローの再実行を担い、その処理の過程で AP_INTERNAL_REDIRECT マクロが実行されます。

なぜ重要か

クライアントに一度レスポンスを返して再リクエストさせる外部リダイレクトとは異なり、サーバー内部で効率的にリクエストの再ルーティング（URIの書き換え、キャッシュヒット時の処理、コンテントネゴシエーションなど）を行うために重要です。これにより、モジュール間でのシームレスなリクエストの引き渡しや、動的なURI変換が実現されます。

関与するモジュール

提供された情報に基づく関与モジュールは以下の通りです。

  • module:include (ヘッダーファイル apache_noprobes.h を含む)
  • module:modules/http (リダイレクト処理の実装 http_request.c を含む)

また、ap_internal_redirect の呼び出し元として、以下のモジュールが関与しています。

  • modules/arch/win32/mod_isapi.c
  • modules/cache/mod_cache.c
  • modules/mappers/mod_negotiation.c
  • modules/mappers/mod_rewrite.c

主要エンティティ

  • **symbol:include/apache_noprobes.h#AP_INTERNAL_REDIRECT**: 内部リダイレクト処理中に実行されるマクロ。
  • **symbol:modules/http/http_request.c#ap_internal_redirect**: 内部リダイレクトを実行し、新しいリクエストコンテキストの構築と処理フローの再実行を制御する関数。

エンドツーエンドの流れ

1. 呼び出し元（mod_rewrite や mod_cache など）が、新しいURIを指定して ap_internal_redirect(new_uri, r) を呼び出します。 2. ap_internal_redirect 内部で internal_internal_redirect を呼び出し、新しいURI用の request_rec 構造体（new）を生成します。 3. 生成された new が NULL の場合は、処理を中断します。 4. AP_INTERNAL_REDIRECT マクロを実行します。 5. ap_run_quick_handler を呼び出し、クイックハンドラによる処理を試みます。 6. クイックハンドラが DECLINED を返した場合、ap_process_request_internal で設定の適用とアクセスチェックを行い、成功すれば ap_invoke_handler でハンドラを実行します。 7. 最後に ap_die を呼び出し、リクエストの終了処理またはエラーレスポンスの生成を行います。

※なお、AP_INTERNAL_REDIRECT マクロ自体の詳細な内部動作や、internal_internal_redirect 内での具体的なフィルタチェーン更新処理の全容など、一部のエンドツーエンドのフローの詳細は提供されたコンテキストからは未解明です。

重要な不変条件

  • ap_internal_redirect に渡される r は、有効な request_rec 構造体でなければなりません。
  • 内部リダイレクト処理において、新しいリクエストコンテキスト（request_rec）の生成に失敗した場合（internal_internal_redirect が NULL を返した場合）、関数は即座に処理を中断し終了します。

よくある落とし穴

  • internal_internal_redirect が NULL を返して処理が中断された場合、エラー発生時は既に ap_die が呼び出されている前提となっているため、呼び出し元でのエラーハンドリングの不整合に注意する必要があります。
  • フィルタチェーンの更新は internal_internal_redirect 内部の update_r_in_filters で行われるため、リダイレクト後のフィルタの挙動を追う際は、この内部処理を意識する必要があります。
