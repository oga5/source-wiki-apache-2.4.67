AP_INTERNAL_REDIRECT

概念

AP_INTERNAL_REDIRECT は、Apache HTTP Server においてリクエストを内部的に別の URI へリダイレクトする際に実行されるマクロです。このマクロは、ap_internal_redirect 関数による内部リダイレクト処理の過程で呼び出され、新しい URI に基づくリクエストコンテキストの再構築や処理フローの再実行と連動します。

なぜ重要か

内部リダイレクトは、クライアントに一度レスポンスを返すことなく、サーバー内部でリクエストを別のリソースに転送して処理を継続するために不可欠な仕組みです。AP_INTERNAL_REDIRECT マクロは、この内部リダイレクトの実行タイミングをフックまたはトレースするために機能します。これにより、URL書き換え（mod_rewrite）、コンテンツネゴシエーション（mod_negotiation）、キャッシュ処理（mod_cache）などの様々なモジュールが、シームレスにリダイレクト処理を追跡・実行できるようになります。

関与するモジュール

  • module:include（ヘッダー定義）
  • module:modules/http（リクエスト処理コア）

主要エンティティ

  • **symbol:include/apache_noprobes.h#AP_INTERNAL_REDIRECT**

    • 内部リダイレクトの実行時に呼び出されるマクロ。具体的なマクロの定義内容や、これがシステムトレース（DTrace等）のプローブであるかなどの詳細な実装仕様は、提供された情報からは未確定です。

  • **symbol:modules/http/http_request.c#ap_internal_redirect**

    • 新しい URI と現在の request_rec を受け取り、内部リダイレクトのライフサイクル全体を制御する関数。

エンドツーエンドの流れ

内部リダイレクトは以下の流れで実行されます。

1. **呼び出し**: mod_rewrite や mod_cache などのモジュールから ap_internal_redirect(new_uri, r) が呼び出される。 2. **コンテキスト生成**: internal_internal_redirect を呼び出し、新しい URI 用の request_rec（new）を生成する。 3. **検証**: 生成された new が NULL の場合、処理を中断する。 4. **マクロ実行**: AP_INTERNAL_REDIRECT マクロを実行する。 5. **クイックハンドラ実行**: ap_run_quick_handler を呼び出し、クイックハンドラによる処理を試みる。 6. **リクエスト処理の適用**: クイックハンドラが DECLINED を返した場合、ap_process_request_internal で設定の適用とアクセスチェックを行い、成功すれば ap_invoke_handler でハンドラを実行する。 7. **終了処理**: 最後に ap_die を呼び出し、リクエストの終了処理またはエラーレスポンスの生成を行う。

重要な不変条件

  • **リクエスト構造体の有効性**: ap_internal_redirect に渡される r は有効な request_rec 構造体でなければなりません。
  • **コンテキスト生成失敗時の即時終了**: 新しいリクエストコンテキストの生成（internal_internal_redirect）に失敗して NULL が返された場合、関数は即座に処理を終了します。
  • **フィルタチェーンの更新**: フィルタチェーンの更新は、internal_internal_redirect 内部で update_r_in_filters を通じて行われる必要があります。

よくある落とし穴

  • **コンテキスト生成失敗時の暗黙の前提**: internal_internal_redirect が NULL を返して処理が中断される際、エラー発生時には既に ap_die が呼び出されていることが前提となっています。この前提が崩れると、エラーハンドリングが正しく行われない可能性があります。
  • **再帰的制御フロー**: 本処理はリダイレクト後の処理フローを再帰的に制御するため、フィルタやハンドラの状態管理が複雑になりやすい点に注意が必要です。
