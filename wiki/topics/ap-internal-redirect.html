AP_INTERNAL_REDIRECT

概念

AP_INTERNAL_REDIRECT は、Apache HTTP Serverにおいてリクエストを新しいURIに内部的にリダイレクトする際に実行されるマクロです。このマクロは、ap_internal_redirect 関数による内部リダイレクト処理の過程で、新しいリクエストコンテキストが生成された直後に呼び出されます。

なぜ重要か

内部リダイレクトは、クライアントに一度レスポンスを返して再リクエストさせる外部リダイレクトとは異なり、サーバー内部でシームレスに新しいURIへのリクエストコンテキストを再構築して処理を継続するために重要です。AP_INTERNAL_REDIRECT マクロは、この内部リダイレクトの実行プロセスにおいて、処理の追跡やプローブ（計測・監視）を行うために重要な役割を果たしていると考えられます。

関与するモジュール

  • module:include
  • module:modules/http

主要エンティティ

  • symbol:include/apache_noprobes.h#AP_INTERNAL_REDIRECT : 内部リダイレクト時に実行されるマクロ。
  • symbol:modules/http/http_request.c#ap_internal_redirect : 内部リダイレクトを制御・実行する関数。

エンドツーエンドの流れ

1. mod_rewrite や mod_cache などのモジュールから ap_internal_redirect(new_uri, r) が呼び出されます。 2. internal_internal_redirect が呼び出され、新しい URI 用の request_rec が生成されます。 3. 生成された新しい request_rec が NULL の場合、処理は中断されます。 4. **AP_INTERNAL_REDIRECT マクロが実行されます。**（※マクロの具体的な展開内容や、システム全体におけるエンドツーエンドのプローブ動作の詳細は提供された情報からは未解明です） 5. ap_run_quick_handler を呼び出し、クイックハンドラによる処理を試みます。 6. クイックハンドラが DECLINED を返した場合、ap_process_request_internal で設定の適用とアクセスチェックを行い、成功すれば ap_invoke_handler でハンドラを実行します。 7. 最後に ap_die を呼び出し、リクエストの終了処理またはエラーレスポンスの生成を行います。

重要な不変条件

  • ap_internal_redirect に渡される r は有効な request_rec 構造体でなければなりません。
  • 内部リダイレクト処理において、新しいリクエストコンテキストの生成（internal_internal_redirect）に失敗した場合、関数は即座に終了します。
  • ※ AP_INTERNAL_REDIRECT マクロ自体の不変条件や、マクロ展開時の制約については提供された情報からは未解明です。

よくある落とし穴

  • internal_internal_redirect が NULL を返した場合、リダイレクト処理は行われず正常終了します。エラー発生時は既に ap_die が呼び出されている前提であるため、呼び出し元での二重のエラーハンドリングに注意する必要があります。
  • フィルタチェーンの更新は internal_internal_redirect 内部で update_r_in_filters によって行われるため、ap_internal_redirect の呼び出し前後でのフィルタの状態変化に留意する必要があります。
  • ※ AP_INTERNAL_REDIRECT マクロ自体の詳細な挙動や、特定のビルド設定（DTraceなどのプロ―ブの有無など）における落とし穴は現時点では未解明です。
