AP_INTERNAL_REDIRECT

概念

AP_INTERNAL_REDIRECT は、Apache HTTP Server においてクライアントに一度レスポンスを返すことなく、サーバー内部で新しい URI へリクエストを再転送（内部リダイレクト）する際に使用されるマクロ、およびそれに関連する処理フローを指します。

なぜ重要か

内部リダイレクトは、URLの書き換え（mod_rewrite）やコンテンツネゴシエーション（mod_negotiation）、キャッシュ処理（mod_cache）など、多くの主要モジュールでリクエストの再ルーティングを行うために不可欠な仕組みです。AP_INTERNAL_REDIRECT マクロは、この内部リダイレクトの実行プロセス（特に ap_internal_redirect 関数内）において、処理の節目をマークまたはトレースするために重要な役割を果たしています。

関与するモジュール

  • module:include (include/apache_noprobes.h): AP_INTERNAL_REDIRECT マクロが定義されています。
  • module:modules/http (modules/http/http_request.c): 内部リダイレクトのメインフローを制御する ap_internal_redirect 関数が定義されています。

主要エンティティ

  • **symbol:include/apache_noprobes.h#AP_INTERNAL_REDIRECT**: 内部リダイレクトの発生を検知・記録するためのマクロ（プローブ）。
  • **symbol:modules/http/http_request.c#ap_internal_redirect**: 新しい URI への内部リダイレクトを処理し、リクエストコンテキストの再構築とハンドラの再実行を行う関数。

エンドツーエンドの流れ

ap_internal_redirect 関数が呼び出された際の流れは以下の通りです。

1. **リクエストの生成**: internal_internal_redirect を呼び出し、新しい URI に基づく新しい request_rec 構造体を生成します。 2. **生成チェック**: 生成された新しいリクエスト構造体が NULL の場合、処理を即座に中断します。 3. **マクロの実行**: AP_INTERNAL_REDIRECT マクロを実行します。 4. **クイックハンドラの実行**: ap_run_quick_handler を呼び出して、クイックハンドラによる処理を試みます。 5. **リクエスト処理の実行**: クイックハンドラが DECLINED を返した場合、ap_process_request_internal を呼び出して設定の適用とアクセスチェックを行い、成功すれば ap_invoke_handler でハンドラを実行します。 6. **終了処理**: 最後に ap_die を呼び出し、リクエストの終了処理またはエラーレスポンスの生成を行います。

※ 呼び出し元モジュール（mod_rewrite や mod_cache など）から ap_internal_redirect が呼び出されるまでの詳細なエンドツーエンドのトリガー条件や、AP_INTERNAL_REDIRECT マクロ自体の具体的な内部実装（プローブとしての詳細な振る舞い）の全容は、提供された情報からは未解明です。

重要な不変条件

  • **リクエスト構造体の有効性**: ap_internal_redirect に渡される request_rec は有効な構造体でなければなりません。
  • **リダイレクト処理の継続条件**: internal_internal_redirect による新しいリクエストコンテキストの生成に失敗（NULL を返却）した場合、関数はそれ以降の処理（マクロ実行やハンドラ呼び出しなど）を行わずに即座に終了します。
  • **フィルタチェーンの更新**: 内部リダイレクトに伴うフィルタチェーンの更新は、internal_internal_redirect 内部で update_r_in_filters を通じて行われます。

よくある落とし穴

  • **リダイレクト失敗時の挙動**: internal_internal_redirect が NULL を返した場合、関数は何も処理を行わずに正常終了します。このとき、エラーが発生している場合は呼び出し前に既に ap_die が実行されていることが前提となっているため、呼び出し側の文脈に注意する必要があります。
  • **再帰的な制御**: 本関数はリダイレクト後の処理フローを再帰的に制御するため、無限リダイレクトなどのループが発生しないよう、呼び出し元で適切に制御する必要があります（ただし、具体的なループ防止策の詳細は提供された情報からは未解明です）。
