AP_INTERNAL_REDIRECT

概念

AP_INTERNAL_REDIRECT は、Apache HTTP Serverにおいてリクエストを新しいURIに内部的にリダイレクトする際に使用されるマクロ、およびそれに関連する内部リダイレクト処理（ap_internal_redirect）の仕組みです。クライアントにレスポンスを返すことなく、サーバー内部でリクエストの再評価とハンドラの再実行を行います。

なぜ重要か

内部リダイレクトは、URLの書き換え（mod_rewrite）、キャッシュ処理（mod_cache）、コンテントネゴシエーション（mod_negotiation）、ISAPI拡張（mod_isapi）などの多様なモジュールにおいて、リクエストの処理フローを動的に変更するために不可欠です。これにより、クライアントに追加の往復（RTT）を強いることなく、サーバー側で柔軟なリクエスト制御が可能になります。

関与するモジュール

提供された情報に基づき、以下のモジュールが関与しています。

  • module:include
  • module:modules/http

主要エンティティ

  • **symbol:include/apache_noprobes.h#AP_INTERNAL_REDIRECT**: 内部リダイレクト処理の過程（ap_internal_redirect 内）で実行されるマクロ。
  • **symbol:modules/http/http_request.c#ap_internal_redirect**: 内部リダイレクトを制御する中心的な関数。新しいURIと現在の request_rec を受け取り、新しいリクエストコンテキストの構築と処理フローの再実行を行います。

エンドツーエンドの流れ

1. **リダイレクトの開始**: mod_rewrite や mod_cache などの呼び出し元モジュールから ap_internal_redirect(new_uri, r) が呼び出されます。 2. **新規リクエスト構造体の生成**: internal_internal_redirect を呼び出し、新しい URI 用の request_rec（new）を生成します。 3. **マクロの実行**: 生成された new が NULL でない場合、AP_INTERNAL_REDIRECT マクロが実行されます。 4. **クイックハンドラの実行**: ap_run_quick_handler を呼び出し、クイックハンドラによる処理を試みます。 5. **内部リクエスト処理の実行**: クイックハンドラが DECLINED を返した場合、ap_process_request_internal によって設定の適用とアクセスチェックが行われ、成功すれば ap_invoke_handler でハンドラが実行されます。 6. **終了処理**: 最後に ap_die を呼び出し、リクエストの終了処理またはエラーレスポンスの生成を行います。

※なお、AP_INTERNAL_REDIRECT マクロ自体の具体的な定義内容や、internal_internal_redirect 内部の完全な挙動など、一部のエンドツーエンドのフローの詳細は提供されたコンテキストからは明らかになっていません。

重要な不変条件

  • ap_internal_redirect に渡される r は、有効な request_rec 構造体でなければなりません。
  • 内部リダイレクト処理において、新しいリクエストコンテキストの生成（internal_internal_redirect）に失敗して NULL が返された場合、関数は即座に処理を中断し、正常終了します（この際、エラー発生時は既に ap_die が呼び出されていることが前提となります）。

よくある落とし穴

  • internal_internal_redirect が NULL を返した場合、リダイレクト処理は行われずに終了するため、呼び出し元でこの挙動を考慮しておく必要があります。
  • フィルタチェーンの更新は ap_internal_redirect の直接の処理ではなく、internal_internal_redirect 内部で update_r_in_filters を通じて行われるため、フィルタの挙動を追跡する際は注意が必要です。
