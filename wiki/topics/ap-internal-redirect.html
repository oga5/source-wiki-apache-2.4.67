ap_internal_redirect

概念

ap_internal_redirect は、Apache HTTP Server 内でリクエストの内部リダイレクトを実行するための関数です。クライアントにレスポンスを返すことなく、サーバー内部で新しい URI に基づく新しいリクエストコンテキストを構築し、リクエスト処理フローを再実行します。

なぜ重要か

この機能により、各種モジュールはクライアントに再度 HTTP リクエストを要求（外部リダイレクト）することなく、サーバーの内部処理だけでリクエストの宛先を動的に変更できます。これにより、パフォーマンスの向上や、シームレスな URL 書き換え、コンテンツネゴシエーション、キャッシュ制御などが実現されます。

関与するモジュール

  • module:include
  • module:modules/http

主要エンティティ

  • symbol:modules/http/http_request.c#ap_internal_redirect: 内部リダイレクト処理を制御する中心的な関数。
  • symbol:include/apache_noprobes.h#AP_INTERNAL_REDIRECT: ap_internal_redirect の処理中に実行されるマクロ。

エンドツーエンドの流れ

1. **呼び出し**: mod_rewrite、mod_cache、mod_negotiation、mod_isapi などのモジュールから、新しい URI（new_uri）と現在の request_rec（r）を引数として ap_internal_redirect が呼び出されます。 2. **新規リクエストの生成**: internal_internal_redirect を呼び出し、新しい URI 用の request_rec（new）を生成します。 3. **検証とマクロ実行**: 生成された new が NULL の場合は処理を中断します。有効な場合は AP_INTERNAL_REDIRECT マクロを実行します。 4. **クイックハンドラの実行**: ap_run_quick_handler を呼び出し、クイックハンドラによる処理を試みます。 5. **内部プロセスの実行**: クイックハンドラが DECLINED を返した場合、ap_process_request_internal を呼び出して設定の適用とアクセスチェックを行います。これが成功すると、ap_invoke_handler でハンドラを実行します。 6. **終了処理**: 最後に ap_die を呼び出し、リクエストの終了処理またはエラーレスポンスの生成を行います。

※ なお、internal_internal_redirect の内部的な詳細や、呼び出し元モジュールにおける具体的なトリガー条件など、エンドツーエンドの完全なフローの全容は提供された情報からは完全には判明していません。

重要な不変条件

  • **リクエスト構造体の有効性**: 引数として渡される r は、有効な request_rec 構造体でなければなりません。
  • **コンテキスト生成失敗時の即時終了**: 新しいリクエストコンテキストの生成（internal_internal_redirect）に失敗して NULL が返された場合、関数は即座に処理を終了します。
  • **フィルタチェーンの更新**: フィルタチェーンの更新は、internal_internal_redirect 内部の update_r_in_filters によって行われる必要があります。

よくある落とし穴

  • **リダイレクト失敗時の暗黙の正常終了**: internal_internal_redirect が NULL を返した場合、エラーは発生せずに関数は正常終了します。これは、エラー発生時にはすでに ap_die が呼び出されていることを前提としているためです。この前提を誤解すると、エラーハンドリングの追跡が困難になる可能性があります。
  • **フィルタチェーンの同期**: フィルタチェーンの更新（update_r_in_filters）は internal_internal_redirect 内部で自動的に行われますが、これを考慮せずに手動でフィルタを操作しようとすると不整合が発生する恐れがあります。
