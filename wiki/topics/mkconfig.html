mkconfig

概念

mkconfig は、Apache HTTP Serverの特定モジュールにおいて、モジュール固有の設定構造体をメモリ上に確保し、初期デフォルト値を設定するための内部ヘルパー関数です。

なぜ重要か

各モジュール（mod_suexec や mod_speling）が正しく動作するためには、リクエスト処理やモジュール初期化の段階で、設定パラメータが適切に初期化されている必要があります。mkconfig は、APR（Apache Portable Runtime）のメモリプールを利用して安全にメモリを確保し、各設定項目のデフォルト値を保証する役割を担っています。

関与するモジュール

  • module:modules/generators
  • module:modules/mappers

主要エンティティ

  • **symbol:modules/generators/mod_suexec.c#mkconfig**: suexec_config_t 構造体を apr_palloc で確保し、active メンバを 0（無効）に初期化する関数。
  • **symbol:modules/mappers/mod_speling.c#mkconfig**: spconfig 構造体を apr_pcalloc で確保し、enabled を 0、check_case_only を 0、check_basename_match を 1（有効）に初期化する関数。

エンドツーエンドの流れ

各モジュールの初期化または設定ディレクティブの処理時に、Apache コアまたはモジュール構成システムからこれらの関数が呼び出されます。

1. **メモリ確保**: APR メモリプール（apr_pool_t）から、対象となる設定構造体のサイズ分のメモリを確保します。 2. **初期値の設定**:

  • mod_suexec の場合: active フラグを 0 に設定します。
  • mod_speling の場合: enabled と check_case_only を 0 に、check_basename_match を 1 に設定します。

    • 3. **返却**: 初期化された構造体へのポインタを呼び出し元に返します。

※なお、静的解析ツールにより mod_suexec.c と mod_speling.c の間で相互呼び出しがあるように検出されていますが、これらが実際に相互に依存しているか、あるいは単に同名の static 関数が個別に呼び出されているだけなのか、詳細なエンドツーエンドの結合フローは完全には判明していません。

重要な不変条件

  • **メモリ確保の成否**: メモリ確保には APR のプール関数（apr_palloc / apr_pcalloc）が使用されます。メモリ不足が発生した場合、APR の仕様によりプロセス自体が終了する可能性があります。
  • **初期状態の保証**:
  • mod_suexec の suexec_config_t では、active メンバは常に 0 で初期化されます。
  • mod_speling の spconfig では、check_basename_match は常に 1（有効）、それ以外は 0 で初期化されます。

よくある落とし穴

  • **静的スコープと重複検出**: 両関数は static void *mkconfig(apr_pool_t *p) という同一のシグネチャを持ち、それぞれのファイル内で独立して定義されています。ツールによる解析において、mod_suexec.c から mod_speling.c の mkconfig（またはその逆）への呼び出しが検出されていますが、これは静的スコープの境界を越えた誤検出である可能性が高いため、コードの保守やリファクタリング時には実際の依存関係を誤認しないよう注意が必要です。
