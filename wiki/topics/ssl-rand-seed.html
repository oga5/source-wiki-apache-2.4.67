ssl_rand_seed

概念

ssl_rand_seed は、Apache HTTP Server（httpd）およびそのベンチマークツールである ApacheBench（ab）において、OpenSSL の擬似乱数生成器（PRNG）にエントロピー（シード値）を供給して初期化・強化するための関数です。

この関数は、サーバーモジュール（modules/ssl）とサポートツール（support）の双方に同名のシンボルとして存在しますが、それぞれ異なるシグネチャと役割を持っています。

なぜ重要か

暗号通信（SSL/TLS）において、安全な鍵生成やプロトコル処理を行うためには、予測不可能な乱数が必要不可欠です。ssl_rand_seed は、起動時や接続時などの適切なタイミングで、ファイル、外部コマンド、EGD（Entropy Gathering Daemon）、あるいはシステム時刻やプロセスID、スタックメモリなどの多様なソースからエントロピーを収集して OpenSSL に供給し、暗号強度の安全性を担保するために重要な役割を果たします。

関与するモジュール

  • module:modules/ssl (Apache HTTP Server の SSL/TLS モジュール)
  • module:support (ApacheBench などのサポートツール)

主要エンティティ

  • **symbol:modules/ssl/ssl_engine_rand.c#ssl_rand_seed**

    • サーバー用の実装。設定された複数のソースからエントロピーを収集し、OpenSSL の PRNG を初期化します。 *シグネチャ*: int ssl_rand_seed(server_rec *s, apr_pool_t *p, ssl_rsctx_t nCtx, char *prefix)

  • **symbol:support/ab.c#ssl_rand_seed**

    • ab ツール専用の静的関数。実行環境のローカル情報からシード値を生成し、OpenSSL の PRNG を初期化します。 *シグネチャ*: static void ssl_rand_seed(void)

エンドツーエンドの流れ

1. サーバー側 (modules/ssl) での流れ

1. サーバーの起動時（ssl_engine_init.c）または接続時（mod_ssl.c）に ssl_rand_seed が呼び出されます。 2. 設定構造体 SSLModConfigRec に登録された乱数ソースリストを順次走査し、現在のコンテキスト（起動時か接続時か）に合致するソースを処理します。

  • **ファイル (SSL_RSSRC_FILE)**: 指定ファイルを読み込み、ssl_rand_feedfp を介して供給します。
  • **外部コマンド (SSL_RSSRC_EXEC)**: 指定コマンドを実行し、その標準出力を ssl_rand_feedfp を介して供給します。
  • **EGD (SSL_RSSRC_EGD)**: RAND_egd() を呼び出し、EGD からエントロピーを取得します。
  • **組み込み (SSL_RSSRC_BUILTIN)**: 現在時刻、PID、および実行時のスタックメモリの一部を RAND_seed() に渡します。

    • 3. 供給された総バイト数をログに出力し、RAND_status() を用いてエントロピーが充足しているか検証します。

2. クライアント側 (support/ab.c) での流れ

1. ab の実行中（ab.c:1453）に ssl_rand_seed が呼び出されます。 2. time(NULL) による現在時刻、getpid() によるプロセスID、および ssl_rand_choosenum を用いて決定されたスタック上の128バイトのデータを取得します。 3. 取得したデータを RAND_seed() に渡して OpenSSL の内部状態を更新します。

> **注意**: サーバー側とクライアント側（ab）の連携、およびこれら2つのコンテキスト間における詳細なエンドツーエンドの相互作用（例：ab からサーバーへのリクエスト送信時にどのように乱数シードが影響し合うかなど）の全体像は、提供された情報からは完全には明らかになっていません。

重要な不変条件

  • **サーバー側**:
  • SSLModConfigRec に乱数シードソースのリストが適切に設定されている必要があります。
  • 処理完了後、OpenSSL の RAND_status() が呼び出され、エントロピーの充足状態が検証される必要があります（不足している場合は警告ログが出力されます）。
  • **クライアント側 (ab.c)**:
  • ssl_rand_choosenum が返すオフセットは、スタック上に確保された stackdata バッファ（256バイト）の範囲を超えないように制御される必要があります（sizeof(stackdata) - 128 - 1 が上限）。

よくある落とし穴

  • **名前の混同**:

    • modules/ssl の ssl_rand_seed（引数あり）と support/ab.c の ssl_rand_seed（静的関数、引数なし）は同名ですが、シグネチャも実装も異なる別個の関数です。コードリーディングやデバッグの際は混同しないよう注意が必要です。

  • **起動パフォーマンスへの影響**:

    • サーバー側で外部コマンド実行（SSL_RSSRC_EXEC）をシードソースとして設定している場合、コマンドの実行遅延がサーバーの起動時間に直接影響を与える可能性があります。

  • **エントロピーの質**:

    • 組み込みソース（SSL_RSSRC_BUILTIN や ab.c でのスタックデータ利用）は補助的な手法であり、これらのみに依存すると十分なエントロピーが得られない可能性があります。現代の OpenSSL 環境では OS のエントロピー源が自動利用されるため、手動シードの重要性は低下しています。
