Shared Object Cache (socache) and Slotmem Abstractions

概念

Shared Object Cache (socache) と Slotmem は、Apache HTTP Server において共有メモリ領域やオブジェクトキャッシュを抽象化するための仕組みです。

  • **Slotmem (ap_slotmem_instance_t)**: 共有メモリ領域を固定サイズのスロット単位で管理・操作するための不透明な（opaque）ハンドルを提供します。
  • **Shared Object Cache (ap_socache_instance_t)**: SSLセッションや認証情報などの小規模オブジェクトをキャッシュするためのバックエンド（shmcb や dbm など）を抽象化する不透明なハンドルを提供します。

なぜ重要か

これらの抽象化により、具体的な共有メモリの確保方法やキャッシュの保存先（メモリ、ファイル、データベースなど）の実装詳細が隠蔽されます。利用側モジュール（mod_proxy_balancer や mod_authn_socache など）は、特定のバックエンド実装に依存することなく、統一されたプロバイダーAPIを介して安全かつ一貫した方法で共有リソースにアクセスできます。

関与するモジュール

  • module:include (インターフェース定義)
  • module:modules/cache (キャッシュプロバイダー実装)
  • module:modules/slotmem (共有メモリプロバイダー実装)

主要エンティティ

  • **symbol:include/ap_slotmem.h#ap_slotmem_instance_t**: 共有メモリ領域を識別し、スロット操作の起点となる不透明な構造体。
  • **symbol:include/ap_socache.h#ap_socache_instance_t**: 各キャッシュプロバイダーが個別のキャッシュインスタンスの状態（メモリ領域やファイルハンドルなど）を保持するための不透明な構造体。

エンドツーエンドの流れ

提供されたコンテキストから確認できる操作の流れは以下の通りです。

Slotmem の利用フロー

1. **インスタンスの取得**: 利用側モジュールが ap_slotmem_provider_t の create または attach を呼び出し、ap_slotmem_instance_t のポインタを取得します。 2. **データの操作**: 取得したインスタンスを引数として、dptr（直接参照）、get/put（データ転送）、num_slots（スロット数取得）、doall（反復処理）などのAPIを呼び出します。

Socache の利用フロー

1. **インスタンスの生成**: 利用側モジュールが ap_socache_provider_t の create を呼び出し、ap_socache_instance_t のポインタを生成します。 2. **初期化**: キャッシュ操作を行う前に、init メソッドを呼び出してインスタンスを初期化します。 3. **キャッシュ操作**: 初期化完了後、store や retrieve などのメソッドにインスタンスを渡してオブジェクトの格納や取得を行います。

※詳細なプロセス間同期のライフサイクル全体や、具体的なエラーハンドリングの伝播フローは、提供されたコンテキストからは完全には明らかになっていません。

重要な不変条件

  • **不透明性 (Opaqueness)**: ap_slotmem_instance_t および ap_socache_instance_t は不透明な型として定義されています。利用側は構造体のメンバに直接アクセスしてはならず、必ずプロバイダーAPI（関数ポインタ群）を経由して操作しなければなりません。
  • **初期化の強制**:
  • ap_slotmem_instance_t は、create または attach が正常に成功した後にのみ、他のAPI関数に渡すことができます。
  • ap_socache_instance_t は、create で生成された後、init メソッドが正常に実行されるまで、store や retrieve などのキャッシュ操作に使用することはできません。

よくある落とし穴

  • **未初期化インスタンスの参照**: インスタンスが初期化される前、あるいは create/attach が失敗して無効な状態のままAPIに渡した場合、挙動はプロバイダーの実装に依存します（一般的には APR_EGENERAL などのエラーコードが返されるか、未定義の動作を引き起こす可能性があります）。
  • **互換性の破壊**: 構造体の内部定義は各プロバイダー（mod_slotmem_shm.c や mod_socache_shmcb.c など）に委ねられているため、実装固有の内部構造に依存したコードを記述すると、プロバイダー変更時に動作が破綻します。
