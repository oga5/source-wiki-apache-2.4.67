ap_sb_handle_t

概念

ap_sb_handle_t は、Apache HTTP Server において特定のワーカープロセスまたはスレッドのスコアボード（実行時状態）を識別・参照するための不透明な（opaque）ハンドルです。

なぜ重要か

MPM（Multi-Processing Module）やモジュール（mod_echo など）がワーカーの状態を更新する際、共有メモリ上の該当領域（worker_score 構造体）を特定するためのキーとして機能します。これにより、排他制御なしで安全かつ高速に各ワーカーの統計情報や状態を更新・参照することが可能になります。

関与するモジュール

  • module:include
  • module:server

主要エンティティ

  • symbol:include/scoreboard.h#ap_sb_handle_t: 不透明なポインタとしての型定義。
  • symbol:server/scoreboard.c#ap_sb_handle_t: child_num と thread_num を保持する具体的な構造体定義。

エンドツーエンドの流れ

1. **生成**: ap_create_sb_handle などの関数を通じて、特定の child_num および thread_num を持つハンドルが生成されます。 2. **状態更新・参照**: MPM や mod_echo などのモジュールが、ap_update_child_status や ap_get_scoreboard_worker などのスコアボード操作 API にこのハンドルを渡します。 3. **アクセス**: API 内部でハンドルに格納されたインデックス情報（child_num, thread_num）に基づき、共有メモリ上の対応する worker_score 構造体が特定され、統計情報や状態の読み書きが行われます。

※なお、詳細な初期化タイミングや破棄を含む完全なライフサイクル全体のエンドツーエンドの流れは、提供された情報からは完全には判明していません。

重要な不変条件

  • **有効なインデックスの保持**: ハンドル内の child_num および thread_num は、常にスコアボード内の有効なインデックスを指していなければなりません。
  • **単一所有者による書き込み**: 本ハンドルが指すスコアボード上のスロットは、単一の所有者（ワーカー）のみが書き込むことを想定しており、排他制御なしでの更新が保証されます。
  • **不変のインデックス**: メモリレイアウト設計上、ハンドルが指すインデックスは初期化後に変更されない前提で運用されます。

よくある落とし穴

  • **直接的なメンバアクセス**: ap_sb_handle_t は不透明な型として公開されているため、直接メンバ（child_num, thread_num）にアクセスすることは避け、必ず ap_get_scoreboard_worker などの公開 API を経由する必要があります。
  • **未初期化・無効なハンドル**: ハンドルが適切に初期化されていない、または無効なインデックスを保持している場合、共有メモリへのアクセス時に領域外アクセスや未定義動作、統計情報の誤更新を引き起こす危険性があります。
  • **定義の二重管理**: 本構造体は server/scoreboard.c と include/scoreboard.h の両方で定義されているため、変更を加える際は両ファイルの整合性を確認する必要があります。
