ap_sb_handle_t

概念

ap_sb_handle_t は、Apache HTTP Server において特定のワーカープロセスまたはスレッドのスコアボード（実行時状態）を識別・参照するためのハンドルです。

公開ヘッダである include/scoreboard.h では不透明な構造体（Opaque Structure）として宣言されており、実装ファイルである server/scoreboard.c 内部で具体的なメンバが定義されています。

なぜ重要か

MPM（Multi-Processing Module）や各種モジュール（mod_echo など）が、共有メモリ上に配置されたスコアボード内の特定のスロット（worker_score）を特定し、状態の更新や統計情報の読み書きを安全に行うための共通の識別キーとして機能するためです。

  • module:server

主要エンティティ

  • **symbol:include/scoreboard.h#ap_sb_handle_t**

    • 外部モジュール向けに公開されている不透明な型定義。直接メンバにアクセスさせず、API 経由での操作を強制します。

  • **symbol:server/scoreboard.c#ap_sb_handle_t**

    • スコアボード実装内部で定義されている具体的な構造体。以下のメンバを持ち、共有メモリ上のオフセットを特定するために使用されます。 ``c struct ap_sb_handle_t { int child_num; int thread_num; }; ``

エンドツーエンドの流れ

1. **ハンドルの生成**: ap_create_sb_handle などの関数を通じて、特定の child_num および thread_num を保持する ap_sb_handle_t が生成・初期化されます。 2. **状態の更新・参照**: MPM やモジュール（例: mod_echo 内の update_echo_child_status）が、このハンドルを ap_update_child_status や ap_get_scoreboard_worker などのスコアボード操作 API に引数として渡します。 3. **共有メモリへのアクセス**: API 内部でハンドル内のインデックス情報を基に、共有メモリ上の該当する worker_score 構造体を特定し、統計情報や状態の書き換えを行います。

※なお、ハンドルが生成されてから破棄されるまでの完全なライフサイクルや、具体的なメモリ確保・解放の全容は提供された情報からは完全には判明していません。

重要な不変条件

  • **有効なインデックスの保持**: ハンドル内の child_num および thread_num は、常にスコアボードの有効な範囲内のインデックスを指していなければなりません。
  • **単一所有者による書き込み**: 本ハンドルが指すスコアボード上のスロットは、単一の所有者（ワーカー）のみが書き込むことを想定しており、排他制御なしでの更新が可能です。
  • **インデックスの不変性**: メモリレイアウト設計上、このハンドルが指すインデックスは初期化後に変更されない前提で運用されます。

よくある落とし穴

  • **直接アクセスの禁止**: ap_sb_handle_t は不透明な型として設計されているため、外部モジュールから直接メンバにアクセスしてはいけません。必ず ap_get_scoreboard_worker などの公開 API を経由する必要があります。
  • **未初期化・無効なハンドルの使用**: 初期化されていない、または無効なインデックスを保持したハンドルを使用すると、スコアボードのメモリ領域外へのアクセスや、誤った統計情報の更新といった未定義動作を引き起こす原因になります。
  • **定義の不整合**: 本構造体は server/scoreboard.c と include/scoreboard.h の両方に関連定義が存在するため、構造を変更する際は両者の整合性を慎重に確認する必要があります。
