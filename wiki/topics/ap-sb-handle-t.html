ap_sb_handle_t

概念

ap_sb_handle_t は、Apache HTTP Server において特定のワーカープロセスまたはスレッドのスコアボード（実行時状態）を識別・参照するためのハンドルです。

公開ヘッダ（include/scoreboard.h）上では不透明なポインタ（不透明な構造体）として定義されていますが、実装ファイル（server/scoreboard.c）の内部では、子プロセス番号とスレッド番号を保持する具体的な構造体として定義されています。

なぜ重要か

このハンドルは、MPM（Multi-Processing Module）や各種モジュール（mod_echo など）が、共有メモリ上のスコアボード内にある特定のワーカーの統計情報や状態（worker_score）を特定・更新するためのキーとして機能します。不透明なハンドルとしてカプセル化されているため、利用側は直接内部メンバにアクセスすることなく、安全にスコアボード操作 API を呼び出すことができます。

関与するモジュール

  • module:server

主要エンティティ

  • **symbol:include/scoreboard.h#ap_sb_handle_t**

    • 公開ヘッダで定義される不透明な構造体型（typedef struct ap_sb_handle_t ap_sb_handle_t;）。

  • **symbol:server/scoreboard.c#ap_sb_handle_t**

    • 実装ファイルで定義される具体的な構造体。child_num（子プロセス番号）と thread_num（スレッド番号）をメンバに持ちます。

エンドツーエンドの流れ

1. **ハンドルの生成**: ap_create_sb_handle 等の関数を通じて、特定のワーカーに対応する ap_sb_handle_t が生成・初期化されます。 2. **状態の更新・参照**: MPM や mod_echo などのモジュール（例: update_echo_child_status）が、このハンドルを ap_update_child_status や ap_get_scoreboard_worker などのスコアボード操作 API に渡します。 3. **共有メモリへのアクセス**: API 内部では、ハンドル内の child_num および thread_num をオフセットとして使用し、共有メモリ上の該当する worker_score 構造体を特定して、統計情報や状態の読み書きを行います。

※なお、生成から破棄に至る完全なライフサイクルや、すべての呼び出し元における詳細なエンドツーエンドのフローは、提供された情報からは完全には判明していません。

重要な不変条件

  • **インデックスの有効性**: ハンドル内の child_num および thread_num は、スコアボード内の有効なインデックスを指していなければなりません。
  • **インデックスの不変性**: メモリレイアウトの設計上、このハンドルが指すインデックスは初期化後に変更されない前提で運用されます。
  • **単一所有者による更新**: 本ハンドルが指すスコアボード上のスロットは、単一の所有者（ワーカー）によってのみ書き込まれることが想定されており、排他制御なしでの更新が可能です。

よくある落とし穴

  • **直接アクセスの禁止**: ap_sb_handle_t は不透明な型として公開されているため、直接メンバにアクセスしてはなりません。worker_score へのアクセスが必要な場合は、必ず ap_get_scoreboard_worker などの公開 API を経由する必要があります。
  • **未初期化・無効なハンドル**: ハンドルが適切に初期化されていない、または無効なメモリ領域を指している場合、アクセサ関数経由で共有メモリへアクセスした際に、領域外アクセスや誤った統計情報の更新、未定義動作を引き起こす可能性があります。
  • **定義の二重管理**: 本構造体は server/scoreboard.c と include/scoreboard.h の両方で定義（宣言と実体定義）されているため、変更時には両方の整合性を確認する必要があります。
