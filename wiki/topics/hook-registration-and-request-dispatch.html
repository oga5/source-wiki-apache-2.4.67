Hook Registration and Request Dispatch

概念

Apache HTTPDにおける **Hook Registration and Request Dispatch**（フック登録とリクエストディスパッチ）は、サーバーのコア機能を拡張するための中心的なメカニズムです。コアサーバーはリクエスト処理のライフサイクルを複数のフェーズ（設定解析、アクセス制御、コンテンツ生成など）に分割し、各フェーズに「フック」を提供します。各モジュールは、自身が処理を行いたいフェーズのフックにハンドラ関数を登録し、サーバーはリクエストの到着時にこれらを順次呼び出します（ディスパッチ）。

なぜ重要か

この仕組みにより、コアサーバー（module:server）のコードを変更することなく、新しいプロトコル、認証方式、コンテンツフィルタなどの機能をモジュールとして動的に追加できます。モジュール間の結合度を極限まで下げつつ、リクエスト処理の全ライフサイクルにわたる柔軟な制御を可能にするために不可欠なアーキテクチャです。

関与するモジュール

  • **module:include**: フックの宣言、登録、および呼び出しを行うための共通マクロやAPI定義（include/ap_hooks.h など）を提供します。
  • **module:server**: モジュールの初期化、フックの登録処理の統制、および実際のリクエスト処理フェーズにおけるフックのディスパッチ（server/config.c など）を担当します。

主要エンティティ

  • **include/ap_hooks.h**:

    • フックを定義・実装するためのマクロ群（AP_DECLARE_HOOK、AP_IMPLEMENT_HOOK_RUN_ALL、AP_IMPLEMENT_HOOK_VOID など）を提供します。これにより、型安全なフック呼び出しが実現されます。

  • **server/config.c**:

    • モジュールの構造体を管理し、サーバー起動時に各モジュールの register_hooks 関数を呼び出してフックを登録します。また、登録されたフックのソートや、リクエスト処理フェーズの実行制御を行います。

エンドツーエンドの流れ

1. **モジュール初期化と登録**: サーバー起動時、server/config.c はロードされた各モジュールのフック登録関数を呼び出します。モジュールは ap_hook_xxx（例: ap_hook_handler）を使用して、特定のフェーズに自前の関数を登録します。 2. **フックのソート**: 登録完了後、module:server は各フックに登録されたハンドラを、指定された優先度（APR_HOOK_FIRST, APR_HOOK_MIDDLE, APR_HOOK_LAST など）および先行・後行関係に基づいてソートします。 3. **リクエストの受信とディスパッチ**: リクエストが到着すると、サーバーは定義されたフェーズに従って ap_run_xxx（例: ap_run_handler）を呼び出し、ソート済みのハンドラ群を順次実行します。

*注意: 特定のリクエストタイプにおける詳細なフック遷移の全容や、エラーハンドリング時の厳密なロールバックフローは、提供されたモジュールコンテキストからは完全には特定されていません。*

重要な不変条件

  • **登録フェーズの限定**: フックの登録は、サーバーの初期化・設定フェーズ（リクエストの受付が開始される前）に完了していなければなりません。リクエスト処理中に動的にフックを追加することはできません。
  • **順序の固定化**: フックの実行順序は、起動時のソート処理によって決定された後は固定され、リクエストごとに動的に変更することはできません。

よくある落とし穴

  • **戻り値による実行制御の誤解**:

    • フックには「すべてのハンドラを実行するもの（RUN_ALL / VOID）」と「いずれかのハンドラが処理を完了したら終了するもの（RUN_FIRST）」があります。ハンドラが返すステータス値（OK, DECLINED, HTTP_...）が、後続のハンドラの実行にどのように影響するかを正しく理解していないと、意図せず他のモジュールの処理を遮断してしまう原因になります。

  • **依存関係（先行・後行）の指定漏れ**:

    • 他のモジュールが処理した結果に依存している場合、適切な優先度や先行・後行関係（predecessors/successors）を指定しないと、実行順序が不定になり、ランダムな不具合を引き起こす可能性があります。

次に読む場所

  • **include/ap_hooks.h**: フックの宣言方法、登録マクロ、および利用可能な優先度定数の詳細な仕様を理解するために参照してください。
  • **server/config.c**: モジュールがどのようにロードされ、フックがどのように集約・ソートされて実行されるかの具体的な実装フローを確認するために参照してください。
