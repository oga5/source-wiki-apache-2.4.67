log_err

概念

log_err は、Apache HTTP Server（httpd-2.4.67）の異なるモジュールにおいて、エラー情報を出力するために定義されている同名の静的（static）関数です。これらは共通の親クラスや共有ライブラリを持つものではなく、それぞれのファイル内で独立して実装されています。

なぜ重要か

エラーログの記録は、システムのセキュリティ監査やデバッグにおいて極めて重要です。

  • **support/suexec.c における重要性**: 特権実行を伴う suexec のセキュリティ検証（UID/GID不一致、不正コマンド実行、メモリ確保失敗など）におけるエラーをシステムログ（syslog）または指定ファイルに記録し、監査トレールを提供します。
  • **test/clients/h2ws.c における重要性**: テストクライアント実行時のエラーを標準出力やログ出力先に一貫した形式で出力し、テストの失敗原因の特定を容易にします。

関与するモジュール

  • module:support
  • module:test/clients

主要エンティティ

  • **symbol:support/suexec.c#log_err**: suexec 内で可変長引数を受け取り、err_output を介してエラーを記録する関数。
  • **symbol:test/clients/h2ws.c#log_err**: h2ws テストクライアント内で、発生箇所（where）とメッセージ（msg）を受け取り、log_out を介してエラーを記録する関数。

エンドツーエンドの流れ

これらは独立したツール・クライアント内のローカルな静的関数のため、両者を跨ぐ単一のエンドツーエンドの流れは存在しません。それぞれの流れは以下の通りです。

suexec におけるエラー記録の流れ

1. suexec.c 内の検証ロジック（メモリ確保失敗、UID/GID不一致、引数不足、不正なコマンド実行など）でエラーが発生します。 2. log_err(fmt, ...) が呼び出されます。 3. コンパイル時に AP_LOG_EXEC マクロが定義されている場合、va_list を初期化して可変長引数を受け取り、err_output(1, fmt, ap) を呼び出してログ出力処理を委譲します。 4. va_end で引数リストを終了し、呼び出し元へ制御を戻します。

h2ws におけるエラー記録の流れ

1. テストクライアント内でエラーが発生します（ただし、具体的な呼び出し元は提供されたソースコード内からは明示されていません）。 2. log_err(where, msg) が呼び出されます。 3. 引数として受け取った where と msg を、固定のラベル "ERROR" と共に log_out 関数へ渡し、ログ出力処理を委譲します。

重要な不変条件

  • **support/suexec.c#log_err**:
  • コンパイル時に AP_LOG_EXEC マクロが定義されている場合のみ、実際のログ出力処理が実行されます。定義されていない場合、この関数は何も行わず即座に復帰します。
  • 可変長引数は stdarg.h の規約に従う必要があります。
  • **test/clients/h2ws.c#log_err**:
  • 引数 where および msg は、有効なヌル終端文字列でなければなりません。

よくある落とし穴

  • **名前の混同**: support/suexec.c の log_err と test/clients/h2ws.c の log_err は、名前は同一ですがシグネチャ（引数の型や数）も目的も異なる全く別の関数です。これらを混同しないよう注意が必要です。
  • **マクロ定義への依存**: suexec の log_err は AP_LOG_EXEC マクロが定義されていないと動作しないため、ビルド設定によってはログが記録されない落とし穴があります。
  • **出力ストリームの閉鎖**: h2ws の log_err が依存する log_out の出力先ストリームがクローズされている場合、ログ出力がサイレントに失敗する可能性があります。

次に読む場所

  • **support/suexec.c**: log_err から呼び出される実際のログ出力ロジックである err_output 関数の実装を確認してください。
  • **test/clients/h2ws.c**: log_err から呼び出される実際のログ出力ロジックである log_out 関数の実装を確認してください。
