log_err

概念

log_err は、Apache HTTP Server（httpd-2.4.67）のソースツリー内の異なるコンテキストにおいて、エラー情報を記録するために定義されている静的（static）関数です。具体的には、セキュリティ実行ラッパーである suexec と、テストクライアントである h2ws の2つの独立したプログラムで同名の関数として実装されています。

なぜ重要か

エラーの適切な記録は、システムのセキュリティ監査やデバッグにおいて極めて重要です。

  • **suexec における重要性**: 権限昇格を伴う実行時の検証失敗（UID/GID不一致、引数不足、不正なコマンド実行など）をシステムログ（syslog）または指定されたログファイルに記録し、セキュリティ監査の証跡を残します。
  • **h2ws における重要性**: テストクライアントの実行中に発生したエラーを標準出力やログ出力先に一貫した形式で出力し、テスト失敗時の原因特定を容易にします。

関与するモジュール

  • module:support
  • module:test/clients

主要エンティティ

  • symbol:support/suexec.c#log_err: suexec 内でエラーを err_output へ委譲して記録する可変長引数関数。
  • symbol:test/clients/h2ws.c#log_err: h2ws 内でエラー箇所とメッセージを "ERROR" ラベル付きで log_out へ委譲する関数。

エンドツーエンドの流れ

各モジュールにおけるエラー記録の流れは以下の通りです。

suexec における流れ

1. suexec.c 内の各検証ロジック（メモリ確保失敗、UID/GID不一致、引数不足、不正なコマンド実行など）が異常を検知し、log_err を呼び出します。 2. log_err は可変長引数（va_list）を初期化します。 3. err_output(1, fmt, ap) を呼び出し、エラーフラグ（1）を立てて実際のログ出力処理を委譲します。

h2ws における流れ

1. エラーが発生した際、log_err(where, msg) が呼び出されます。 2. log_err は受け取った where と msg を、固定ラベル "ERROR" とともに log_out 関数へ渡します。 3. log_out が実際の出力処理を行います。

※なお、h2ws における具体的な呼び出し元の全容や、err_output および log_out の内部で処理が完結するまでの詳細なエンドツーエンドのフローは、提供された情報からは完全には判明していません。

重要な不変条件

  • **suexec の log_err**:
  • コンパイル時に AP_LOG_EXEC マクロが定義されている場合のみ、実際のログ出力処理が実行されます。定義されていない場合、この関数は何も行わずに即座に復帰します。
  • 可変長引数は stdarg.h の規約に従う必要があります。
  • **h2ws の log_err**:
  • 引数 where および msg は、有効なヌル終端文字列でなければなりません。

よくある落とし穴

  • **同名関数の混同**: support/suexec.c の log_err と test/clients/h2ws.c の log_err は、名前は同じですがシグネチャも目的も異なる完全に独立した関数です。これらを混同しないよう注意が必要です。
  • **suexec での通常ログとの混同**: suexec にはエラーを記録する log_err（内部で is_error = 1 として処理）のほかに、通常のトランザクションを記録する log_no_err が存在します。セキュリティ監査上、適切な関数を選択する必要があります。
  • **出力ストリームの閉塞**: h2ws の log_err が依存する log_out の出力ストリームがクローズされている場合、ログ出力が失敗する可能性があります。

次に読む場所

  • support/suexec.c: err_output の実装および log_no_err との比較を確認してください。
  • test/clients/h2ws.c: log_out の実装および実際の呼び出し箇所を確認してください。
