get_suexec_id_doer

概念

get_suexec_id_doer は、Apache HTTP Server において suEXEC（特定の UNIX ユーザー/グループ権限でのプログラム実行）を適用する際、リクエストに関連付けられた実行ユーザーおよびグループ ID（UID/GID）を解決・取得するための関数です。

この関数は、ディレクトリごとの設定から ID を取得する実装（mod_suexec）と、ユーザーのホームディレクトリ（UserDir）からシステム上の UID/GID を動的に解決する実装（mod_userdir）の2種類が存在します。

なぜ重要か

Apache が CGI などの外部プログラムを実行する際、セキュリティ上の理由から適切な権限に切り替える必要があります。get_suexec_id_doer は、ap_hook_get_suexec_identity フックを介して、リクエストに応じた適切な実行権限情報（ap_unix_identity_t）を動的に提供する役割を担うため重要です。

関与するモジュール

  • module:modules/generators (mod_suexec)
  • module:modules/mappers (mod_userdir)

主要エンティティ

  • **symbol:modules/generators/mod_suexec.c#get_suexec_id_doer**:

    • ディレクトリごとの設定（suexec_config_t）が有効（active）である場合に、設定済みの UID/GID（cfg->ugid）を返します。

  • **symbol:modules/mappers/mod_userdir.c#get_suexec_id_doer**:

    • リクエストに関連付けられたユーザー名から、OS のシステム情報（apr_uid_get）を用いて動的に UID/GID を解決し、userdir フラグを 1 に設定した構造体を返します。

エンドツーエンドの流れ

1. **フックの登録**: mod_suexec および mod_userdir は、初期化時に ap_hook_get_suexec_identity フックに対してそれぞれの get_suexec_id_doer を登録します。 2. **リクエスト処理時の呼び出し**: リクエスト処理中、Apache のフック機構により ap_hook_get_suexec_identity が呼び出されます。 3. **ID の解決**:

  • mod_suexec 側では、ディレクトリ設定の active フラグが真であれば、設定された UID/GID を返します。
  • mod_userdir 側では、リクエスト設定からユーザー名を取得し、apr_uid_get を用いて OS から UID/GID を取得して返します。

*注意: 実際にどのタイミングでフックがトリガーされ、取得された ID がどのように suexec バイナリへ渡されるかという詳細なエンドツーエンドの全体フローは、提供されたモジュール情報からは完全には明らかになっていません。*

重要な不変条件

  • **mod_suexec における不変条件**:
  • r->per_dir_config から suexec_module に関連付けられた設定が取得可能である必要があります。
  • suexec_config_t の active フラグが真である場合のみ、有効な ugid ポインタを返します。偽の場合は NULL を返します。
  • **mod_userdir における不変条件**:
  • APR_HAS_USER マクロが定義されている環境でのみ動作します。未定義の場合は常に NULL を返します。
  • リクエストの request_config に userdir_module キーでユーザー名が格納されている必要があります。
  • 正常に UID/GID が解決された場合、返される ap_unix_identity_t の userdir フラグは必ず 1 に設定されます。

よくある落とし穴

  • **設定の無効化による NULL 返却**:

    • mod_suexec で cfg->active が偽の場合、または mod_userdir でユーザー名が未設定の場合は NULL が返されます。この場合、呼び出し側（mod_cgi など）が適切にハンドリングを行わないと、suEXEC が適用されません。

  • **OS 依存の解決失敗**:

    • mod_userdir の実装は apr_uid_get を介して OS のユーザーデータベースに依存しているため、システム上に該当ユーザーが存在しない場合や、プラットフォーム固有のユーザー管理設定に問題がある場合、UID/GID の解決に失敗して NULL を返します。
