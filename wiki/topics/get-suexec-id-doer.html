get_suexec_id_doer

概念

get_suexec_id_doer は、Apache HTTP Server において suEXEC 実行時のユーザーおよびグループ ID（ap_unix_identity_t）を解決するための関数（フックハンドラ）です。異なるモジュール（mod_suexec と mod_userdir）で同名の関数として定義されており、それぞれ異なるコンテキスト（ディレクトリ設定またはユーザーディレクトリ）から適切なシステム権限情報を取得する役割を持ちます。

なぜ重要か

suEXEC 環境下で CGI などの外部プログラムを実行する際、どのシステムユーザー/グループ権限で実行すべきかを正しく決定する必要があります。get_suexec_id_doer は、ap_hook_get_suexec_identity フックを介して、リクエストに応じた適切な UID/GID 情報を動的に提供するため、セキュリティと権限分離の制御において重要です。

関与するモジュール

  • module:modules/generators
  • module:modules/mappers

主要エンティティ

  • symbol:modules/generators/mod_suexec.c#get_suexec_id_doer: ディレクトリごとの設定（suexec_config_t）から、設定済みの UID/GID を取得して返します。
  • symbol:modules/mappers/mod_userdir.c#get_suexec_id_doer: リクエストに関連付けられたユーザー名を基に、システム（OS）から UID/GID を動的に解決して返します。

エンドツーエンドの流れ

1. Apache がリクエストを処理する際、suEXEC の実行権限を決定するために ap_hook_get_suexec_identity フックを呼び出します。 2. このフックに登録された各モジュールの get_suexec_id_doer が呼び出されます。

  • **mod_suexec の場合**: リクエストのディレクトリ設定から suexec_config_t を取得し、active フラグが真であれば設定済みの ugid を返します。
  • **mod_userdir の場合**: リクエスト設定からユーザー名を取得し、apr_uid_get を用いて OS から UID/GID を取得して返します。

    • 3. 取得された ap_unix_identity_t は、後続の処理（mod_cgi など）で実際の suEXEC バイナリ実行時の権限切り替えに使用されます。

*注意: フック呼び出しの優先順位や、複数のハンドラが競合した場合の最終的な決定ロジックなど、詳細なエンドツーエンドの制御フローは提供された情報からは完全には明らかになっていません。*

重要な不変条件

  • **mod_suexec 側**: suexec_config_t の active フラグが真である場合のみ、有効な ap_unix_identity_t が返されます。偽の場合は NULL を返します。
  • **mod_userdir 側**: APR_HAS_USER が定義されている環境でのみ動作します。未定義の場合や、リクエスト内にユーザー名が設定されていない場合、あるいは apr_uid_get が失敗した場合は NULL を返します。また、成功時には ugid->userdir フラグが 1 に設定されます。

よくある落とし穴

  • **設定の依存関係**: mod_suexec のハンドラは active フラグに依存しており、設定が未初期化または無効な場合は NULL を返します。この場合、呼び出し側（mod_cgi 等）が suEXEC を適用しないなどの判断を行う必要があります。
  • **OS/プラットフォーム依存**: mod_userdir のハンドラは OS のユーザー名前解決（apr_uid_get）に直接依存しているため、プラットフォーム固有のユーザー管理設定の影響を受けます。
  • **メモリ確保**: mod_userdir ではリクエストプールから apr_palloc でメモリを確保するため、メモリ確保に失敗した場合は NULL が返されます。
