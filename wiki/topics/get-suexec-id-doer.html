get_suexec_id_doer

概念

get_suexec_id_doer は、Apache HTTP Server において suEXEC（特定のユーザー/グループ権限でのCGI等の実行）が有効な環境下で、リクエストに関連付けられた実行ユーザー（UID）およびグループ（GID）の識別情報を取得・解決するための関数（フックハンドラ）です。

この関数は、mod_suexec と mod_userdir という2つの異なるモジュールに同名で定義されており、それぞれの文脈（ディレクトリごとの設定、またはユーザーディレクトリの解決）に応じてUID/GID情報を取得します。

なぜ重要か

Apacheがリクエストを処理する際、セキュリティ上の理由から適切な権限でスクリプトを実行する必要があります。get_suexec_id_doer は、ap_hook_get_suexec_identity フックを介して、リクエストに応じた適切な実行権限情報（ap_unix_identity_t）を動的に提供する役割を担います。これにより、モジュール間で協調して安全な権限移行（suEXEC）を実現できます。

関与するモジュール

  • module:modules/generators （mod_suexec.c）
  • module:modules/mappers （mod_userdir.c）

主要エンティティ

  • **symbol:modules/generators/mod_suexec.c#get_suexec_id_doer**
  • ディレクトリごとの設定（suexec_config_t）から、設定済みのUID/GID（ap_unix_identity_t）を取得します。
  • **symbol:modules/mappers/mod_userdir.c#get_suexec_id_doer**
  • リクエストに関連付けられたユーザー名から、OSのシステム情報を参照してUID/GIDを動的に解決します。

エンドツーエンドの流れ

1. Apacheがリクエストを処理する際、suEXECの適用が必要な場面で ap_hook_get_suexec_identity フックが呼び出されます。 2. フックに登録された各 get_suexec_id_doer が呼び出されます。

  • **mod_suexec 側**:

    • 1. ap_get_module_config を用いて、リクエストのディレクトリ設定から suexec_config_t を取得します。 2. 設定が有効（cfg->active が真）であれば、設定されている cfg->ugid のアドレスを返します。

  • **mod_userdir 側**:

    • 1. ap_get_module_config を用いて、リクエストから対象のユーザー名を取得します。 2. ユーザー名が存在する場合、apr_palloc で ap_unix_identity_t のメモリを確保し、apr_uid_get を用いてシステムからUID/GIDを取得します。 3. 取得成功時、ugid->userdir フラグを 1 に設定して構造体を返します。 3. 取得された ap_unix_identity_t は、呼び出し側（mod_cgi 等）に渡され、最終的に外部の suexec バイナリ等を通じて権限切り替えが実行されます。

*注意: 呼び出し側（mod_cgi 等）がこれらの戻り値をどのように処理し、最終的に suexec バイナリへ引き渡すかという詳細なエンドツーエンドの制御フロー全体や、複数のフックハンドラ間の優先順位・競合解決の仕組みは、提供された情報からは完全には明らかになっていません。*

重要な不変条件

  • **mod_suexec における不変条件**:
  • suexec_config_t 構造体の active フラグが真である場合のみ、設定された ugid が有効として返されます。偽の場合は NULL を返します。
  • **mod_userdir における不変条件**:
  • APR_HAS_USER が定義されている環境でのみ動作します（未定義時は常に NULL を返します）。
  • リクエストの request_config に userdir_module キーでユーザー名が格納されている必要があります。
  • システムからのUID/GID取得（apr_uid_get）が成功（APR_SUCCESS）した場合のみ、有効な構造体を返します。

よくある落とし穴

  • **設定の無効化による NULL 返却**:

    • mod_suexec の cfg->active が偽の場合や、mod_userdir でユーザー名が取得できない場合、関数は NULL を返します。この場合、呼び出し側が適切にフォールバック処理を行わないと、suEXECが適用されません。

  • **OS依存のユーザー解決失敗**:

    • mod_userdir の実装は apr_uid_get に依存しているため、OSのユーザー管理設定（/etc/passwd や LDAP 等）に問題がある場合、ユーザー情報の解決に失敗して NULL が返されます。

  • **実際の権限切り替えとの分離**:

    • 本関数はUID/GIDの「特定」を行うだけであり、実際のプロセス権限切り替えは外部の suexec バイナリ等が担当します。本関数の戻り値が正常であっても、バイナリ側の制約（UIDの最小値制限など）により実行が失敗することがあります。
