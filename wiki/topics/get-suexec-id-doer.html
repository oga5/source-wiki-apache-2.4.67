get_suexec_id_doer

概念

get_suexec_id_doer は、Apache HTTP Server においてリクエスト処理時に suEXEC（ユーザー権限での外部プログラム実行）を適用するための UNIX 実行アイデンティティ（UID/GID）を取得・解決する関数です。この関数は、ディレクトリごとの設定を扱う mod_suexec と、ユーザー個人のディレクトリ（~user）をマッピングする mod_userdir の2つの異なるモジュールで同名の関数として実装されており、それぞれ異なるアプローチで実行アイデンティティを解決します。

なぜ重要か

Apache が CGI などの外部スクリプトを実行する際、セキュリティ上の理由から適切なユーザー権限に切り替える必要があります。get_suexec_id_doer は、ap_hook_get_suexec_identity フックを介して、リクエストに応じた適切な UNIX 実行アイデンティティ（ap_unix_identity_t）を動的に提供する役割を担うため、セキュアな権限移行プロセスの基盤として重要です。

関与するモジュール

  • module:modules/generators (mod_suexec.c)
  • module:modules/mappers (mod_userdir.c)

主要エンティティ

  • **symbol:modules/generators/mod_suexec.c#get_suexec_id_doer**

    • ディレクトリごとの設定（suexec_config_t）から、suEXEC が有効化されている場合に設定済みの UID/GID 構造体（ap_unix_identity_t）を返します。

  • **symbol:modules/mappers/mod_userdir.c#get_suexec_id_doer**

    • リクエストに関連付けられたユーザー名から、OS のシステム情報を解決して UID/GID を取得し、ap_unix_identity_t 構造体を生成して返します。

エンドツーエンドの流れ

1. **フックの登録**: 両モジュールは、ap_hook_get_suexec_identity フックに対してそれぞれの get_suexec_id_doer 関数をハンドラとして登録します。 2. **リクエスト処理時の呼び出し**: リクエスト処理中（CGI 実行時など）に、Apache のフック機構を介して登録された get_suexec_id_doer が呼び出されます。 3. **アイデンティティの解決**:

  • **mod_suexec 側**: リクエストのディレクトリ設定から suexec_config_t を取得し、active フラグが真であれば設定済みの ugid へのポインタを返します。
  • **mod_userdir 側**: リクエスト設定から対象のユーザー名を取得し、apr_uid_get を用いてシステムから UID/GID を取得して、新しくメモリ確保した ap_unix_identity_t 構造体を返します。

    • 4. **権限の適用**: 取得された ap_unix_identity_t に基づき、最終的に外部の suexec バイナリ等を通じて権限が切り替えられます。

*注意: 実際に権限切り替えを行う外部 suexec バイナリの呼び出しや、戻り値が NULL の場合の呼び出し側（mod_cgi 等）における詳細なハンドリングなど、エンドツーエンドの全体フローの詳細は提供された情報からは完全には明らかになっていません。*

重要な不変条件

  • **mod_suexec における不変条件**:
  • suexec_config_t の active フラグが真である場合のみ、有効な ugid ポインタが返されます。偽の場合は NULL を返します。
  • **mod_userdir における不変条件**:
  • APR_HAS_USER マクロが定義されている環境でのみ動作します。未定義の場合は常に NULL を返します。
  • リクエストの request_config に userdir_module キーでユーザー名が格納されている必要があります。
  • 解決に成功した場合、返される ap_unix_identity_t の userdir フラグは必ず 1 に設定されます。

よくある落とし穴

  • **NULL 返却時の挙動**: いずれの実装も、条件を満たさない場合（設定が無効、ユーザー名が未設定、システムからの UID 取得失敗など）は NULL を返します。この NULL が返された際のフォールバック挙動は呼び出し側のモジュール（mod_cgi など）の実装に依存するため、呼び出し側の挙動を把握していないと意図しない権限で実行される、または実行が拒否される可能性があります。
  • **OS依存のユーザー解決**: mod_userdir の実装は apr_uid_get を介して OS のユーザー管理に依存しているため、OS 側のユーザー設定やネームサービス（LDAP, NIS 等）の一時的な不具合により、予期せず NULL が返される（解決に失敗する）リスクがあります。
