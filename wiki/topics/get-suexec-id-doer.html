get_suexec_id_doer

概念

get_suexec_id_doer は、Apache HTTP Server（httpd-2.4.67）において、リクエストを処理する際に suEXEC（特定のユーザー/グループ権限でのプログラム実行）を適用するためのユーザーおよびグループID（UID/GID）情報を取得・解決する関数です。この関数は、ディレクトリごとの設定を扱う mod_suexec と、ユーザーのホームディレクトリへのマッピングを行う mod_userdir の両方に同名の関数として存在し、それぞれ異なるアプローチで対象の UID/GID を特定します。

なぜ重要か

Apache が CGI などの外部プログラムを実行する際、セキュリティ上の理由から特定のユーザー権限（suEXEC）で実行する必要があります。get_suexec_id_doer は、ap_hook_get_suexec_identity フックを通じて呼び出され、リクエストのコンテキスト（ディレクトリ設定やユーザーディレクトリの指定）に応じた適切な実行権限情報（ap_unix_identity_t）を動的に提供する役割を担うため重要です。

関与するモジュール

  • module:modules/generators (mod_suexec.c)
  • module:modules/mappers (mod_userdir.c)

主要エンティティ

  • **symbol:modules/generators/mod_suexec.c#get_suexec_id_doer**:

    • ディレクトリごとの設定（suexec_config_t）から、あらかじめ設定された active な UID/GID 情報を取得して返します。

  • **symbol:modules/mappers/mod_userdir.c#get_suexec_id_doer**:

    • リクエストに関連付けられたユーザー名から、OS のシステム情報を基に動的に UID/GID を解決し、userdir フラグを立てた ap_unix_identity_t 構造体を生成して返します。

エンドツーエンドの流れ

1. **フックの登録**: 両モジュールは、ap_hook_get_suexec_identity フックに対してそれぞれの get_suexec_id_doer をハンドラとして登録します。 2. **リクエスト処理時の呼び出し**: リクエスト処理中（例えば CGI 実行前など）、Apache のコアまたは他のモジュールが ap_hook_get_suexec_identity フックを起動します。 3. **UID/GID の解決**:

  • **mod_suexec 側**: リクエストのディレクトリ設定（r->per_dir_config）から suexec_config_t を取得し、cfg->active が真であれば設定済みの cfg->ugid を返します。
  • **mod_userdir 側**: リクエストの設定からユーザー名を取得し、apr_uid_get を用いてシステムから UID/GID を取得して返します。

    • 4. **結果の返却**: 解決された ap_unix_identity_t 構造体が呼び出し元に返され、実際の権限切り替え処理（外部の suexec バイナリなど）に利用されます。

*注意: フックが呼び出される正確なタイミングや、複数の get_suexec_id_doer が登録されている場合の優先順位、および戻り値が NULL だった場合の呼び出し側（mod_cgi 等）の最終的なフォールバック挙動など、詳細なエンドツーエンドの制御フローは完全には判明していません。*

重要な不変条件

  • **mod_suexec における不変条件**:
  • r->per_dir_config から suexec_module に関連付けられた設定が取得可能である必要があります。
  • suexec_config_t の active フラグが真である場合のみ、有効な UID/GID 情報が返されます。
  • **mod_userdir における不変条件**:
  • APR_HAS_USER マクロが定義されている環境でのみ動作します。未定義の場合は常に NULL を返します。
  • リクエストの request_config に userdir_module キーでユーザー名が格納されている必要があります。

よくある落とし穴

  • **設定の有効性**:

    • mod_suexec の cfg->active が偽の場合や、mod_userdir でユーザー名が設定されていない場合は NULL が返されます。この場合、呼び出し元は suEXEC を適用しない（またはデフォルトの権限で実行する）ため、意図しない権限でプログラムが実行される可能性があります。

  • **OS 依存の解決**:

    • mod_userdir の実装は apr_uid_get を介して OS のユーザーデータベースに依存しているため、OS 側のユーザー管理設定（/etc/passwd や LDAP など）に問題がある場合、UID/GID の解決に失敗して NULL を返す原因になります。

  • **メモリ確保**:

    • mod_userdir では apr_palloc を使用してリクエストプールからメモリを確保するため、メモリ不足時には NULL を返します。
