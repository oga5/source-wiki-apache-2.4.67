dav_resource_private

概念

dav_resource_private は、Apache HTTP ServerのWebDAVモジュール（mod_dav）において、WebDAVリソース（dav_resource）に関連付けられたプロバイダー固有のプライベート情報を保持するための構造体です。

modules/dav/main/mod_dav.h においては、具体的な中身を隠蔽した不透明な（opaque）構造体として前方宣言されています。一方、具体的なプロバイダー実装である modules/dav/fs/repos.c（mod_dav_fs）においては、ファイルシステム上のパスやファイル属性などを保持する具体的な構造体として定義されています。

なぜ重要か

mod_dav コアロジックと、具体的なリポジトリ実装（ファイルシステムなど）の結合度を下げる（カプセル化する）ために重要です。コア側は dav_resource_private の内部構造を直接操作せず、プロバイダー側が dav_resource->info を通じて自身に必要なコンテキスト情報を安全に保持・参照できるように設計されています。

関与するモジュール

主要エンティティ

  • **symbol:modules/dav/main/mod_dav.h#dav_resource_private**

    • 不透明な構造体としての宣言。 ``c typedef struct dav_resource_private dav_resource_private; ``

  • **symbol:modules/dav/fs/repos.c#dav_resource_private**

    • mod_dav_fs における具体的な構造体定義。 ``c struct dav_resource_private { apr_pool_t *pool; /* リクエストに関連付けられたメモリプール */ const char *pathname; /* リソースのフルパス */ apr_finfo_t finfo; /* ファイルシステム情報 */ request_rec *r; /* Apache リクエストレコード */ }; ``

エンドツーエンドの流れ

1. **リソースの生成と初期化** mod_dav_fs のリソース取得処理（dav_fs_get_resource など）において、dav_resource_private 構造体がメモリプール（pool）から割り当てられ、初期化されます。 2. **リソースへの紐付け** 生成された dav_resource_private インスタンスは、親となる dav_resource 構造体の info メンバ（ポインタ）に格納されます。 3. **プロバイダー内での利用** mod_dav_fs 内の各関数（ファイルコピー、移動、プロパティ管理、ディレクトリ走査など）は、resource->info を dav_resource_private* にキャストして、ファイルパス（pathname）やファイル情報（finfo）を取得し、実際のファイルシステム操作を実行します。

※なお、mod_dav コアからプロバイダー呼び出しに至る全体的なエンドツーエンドの制御フローの詳細は、提供された情報からは完全には明らかになっていません。

重要な不変条件

  • **生存期間の同期**

    • dav_resource_private は、親となる dav_resource のライフサイクルと同期して管理される必要があります。また、内部の pool はリクエストの生存期間と同期した apr_pool_t である必要があります。

  • **有効なリクエスト参照**

    • メンバ r は、常に有効な Apache の request_rec を指していなければなりません。

  • **非NULLポインタ**

    • プロバイダー固有の処理が実行される際、dav_resource->info は有効な dav_resource_private インスタンスを指している（NULL でない）必要があります。

よくある落とし穴

  • **セグメンテーション違反**

    • dav_resource->info が NULL の状態でプロバイダー固有のデータにアクセスしようとすると、セグメンテーション違反が発生します。

  • **メモリの無効化（未定義動作）**

    • pathname などの文字列は pool から割り当てられているため、明示的な解放は不要ですが、リクエスト終了時にプールが破棄された後に本構造体やそのメンバを参照すると、未定義の動作を引き起こします。

  • **不整合なキャスト**

    • プロバイダーの実装に不整合がある場合、resource->info を誤った型にキャストしてアクセスすると、メモリ破損や未定義動作の原因となります。
