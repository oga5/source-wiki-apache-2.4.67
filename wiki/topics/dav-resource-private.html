dav_resource_private

概念

dav_resource_private は、Apache HTTP Server の WebDAV モジュール（mod_dav）において、WebDAV リソース（dav_resource）に関連付けられたリポジトリ（プロバイダー）固有のプライベート情報を保持するための構造体です。

modules/dav/main/mod_dav.h においては不透明な（opaque）構造体として前方宣言されており、具体的なメンバ構成は各プロバイダーの実装に委ねられています。一方、modules/dav/fs/repos.c では、ファイルシステムをリポジトリとするプロバイダー（mod_dav_fs）向けの具体的な構造体として定義されています。

なぜ重要か

mod_dav のコアロジックと、具体的なリポジトリプロバイダー（mod_dav_fs など）の実装を疎結合に保つために重要です。

mod_dav コアは本構造体の内部構造を直接操作せず、プロバイダー側が dav_resource->info を dav_resource_private* にキャストして自身に必要なメタデータ（ファイルシステム上の絶対パスやリクエストコンテキストなど）を取り出すことで、プロバイダーに依存しない共通の WebDAV 処理を実現しています。

関与するモジュール

  • module:modules/dav/fs (mod_dav_fs プロバイダーの実装)
  • module:modules/dav/main (mod_dav コアモジュール)

主要エンティティ

  • **symbol:modules/dav/main/mod_dav.h#dav_resource_private**

    • 不透明な構造体としての typedef 宣言。

  • **symbol:modules/dav/fs/repos.c#dav_resource_private**

    • mod_dav_fs における具体的な構造体定義。以下のメンバを保持します。

  • apr_pool_t *pool: リクエストに関連付けられたメモリプール。
  • const char *pathname: リソースのフルパス。
  • apr_finfo_t finfo: ファイルシステム情報（サイズ、権限、時刻等）。
  • request_rec *r: Apache のリクエストレコード。

エンドツーエンドの流れ

詳細なエンドツーエンドの全体フローは完全には明らかになっていませんが、提供された情報に基づく部分的な流れは以下の通りです：

1. **リソースの生成と初期化**: mod_dav_fs のリソース取得処理（dav_fs_get_resource 相当の処理など）において、dav_resource_private のメモリが割り当てられ、初期化されます。 2. **親構造体への格納**: 生成された dav_resource_private は、親となる dav_resource 構造体の info メンバに格納されます。 3. **プロバイダー処理での参照**: mod_dav_fs 内の各関数（dav_fs_copymove_file など）は、resource->info から本構造体へのポインタを取得（キャスト）し、保持されている pathname や finfo を使用して、ファイル操作やプロパティ管理、ディレクトリ走査などの実処理を実行します。

重要な不変条件

  • **生存期間の同期**:

    • dav_resource_private は、親となる dav_resource のライフサイクルと同期して管理される必要があります。

  • **有効なポインタの保持**:
  • pool は、リクエストの生存期間と同期した有効な apr_pool_t である必要があります。
  • r は、有効な Apache request_rec を指している必要があります。
  • **メモリ管理**:

    • pathname の文字列は pool から割り当てられているため、明示的な解放は不要ですが、リクエスト終了時のプール破棄に伴い無効化されます。

よくある落とし穴

  • **NULL アクセスによるクラッシュ**:

    • resource->info が NULL の状態でプロバイダー固有のデータにアクセスしようとすると、セグメンテーション違反が発生します。

  • **ダングリングポインタ**:

    • pool が破棄された後に本構造体またはそのメンバ（pathname など）を参照すると、未定義の動作を引き起こします。

  • **不整合なキャスト**:

    • プロバイダーの実装が不整合な場合、キャスト後のデータアクセスで未定義の動作を引き起こす可能性があります。
