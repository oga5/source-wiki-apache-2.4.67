dav_resource_private

概念

dav_resource_private は、Apache HTTP Server の WebDAV モジュール（mod_dav）において、各リポジトリプロバイダーがリソース（dav_resource）に関連付けられた固有のプライベート情報を保持するための構造体です。

modules/dav/main/mod_dav.h においては不透明な（opaque）構造体として前方宣言されており、具体的な構造体の定義は各プロバイダーの実装（例: modules/dav/fs/repos.c）に委ねられています。

なぜ重要か

mod_dav コアモジュールと、具体的なストレージ（ファイルシステムなど）を扱うプロバイダーモジュールとの間で、関心の分離（Separation of Concerns）を実現するために重要です。

コアモジュールは dav_resource_private の内部構造を直接操作せず、プロバイダー側のみがこの構造体を具体的な型としてキャストして利用します。これにより、コアロジックを汚染することなく、プロバイダー固有のメタデータ（ファイルパスやリクエストプール、ファイルシステム情報など）を安全に伝播させることができます。

関与するモジュール

主要エンティティ

  • **symbol:modules/dav/main/mod_dav.h#dav_resource_private**

    • 不透明型（typedef struct dav_resource_private dav_resource_private;）としての前方宣言。dav_resource 構造体の info フィールドを通じて保持されます。

  • **symbol:modules/dav/fs/repos.c#dav_resource_private**

    • mod_dav_fs における具体的な構造体定義。以下のメンバを含み、ファイルシステム上のリソース情報を管理します。

  • apr_pool_t *pool: リクエストに関連付けられたメモリプール。
  • const char *pathname: リソースのファイルシステム上のフルパス。
  • apr_finfo_t finfo: ファイルシステム情報（サイズ、権限、時刻など）。
  • request_rec *r: Apache のリクエストレコード。

エンドツーエンドの流れ

1. **リソースの生成と初期化** クライアントからの WebDAV リクエストに基づき、プロバイダー（mod_dav_fs の dav_fs_get_resource 相当の処理など）が dav_resource を生成する際、本構造体のメモリを割り当てて初期化し、resource->info に格納します。 2. **プロバイダー内での参照** ファイル操作（コピー、移動など）、プロパティ管理、ディレクトリ走査の際、プロバイダー内の各関数は resource->info を dav_resource_private* にキャストして、必要なファイルシステム情報やリクエストコンテキストを取得・利用します。 3. **リソースの破棄** リクエスト終了時に、関連付けられた apr_pool_t が破棄されることで、本構造体のメモリも自動的に解放されます。

※なお、プロバイダーの登録からコアモジュールを経由した呼び出しに至る完全なエンドツーエンドの制御フローの全容は、提供された情報からは完全には特定されていません。

重要な不変条件

  • **生存期間の同期**

    • dav_resource_private のライフサイクルは、親となる dav_resource およびリクエストのメモリプール（pool）の生存期間と同期していなければなりません。

  • **有効なポインタの維持**

    • r は常に有効な Apache の request_rec を指していなければならず、pool はリクエストの生存期間中有効な apr_pool_t である必要があります。

  • **プロバイダー間の一貫性**

    • resource->info に格納されるポインタは、それを生成したプロバイダーが定義する dav_resource_private のメモリレイアウトと完全に一致している必要があります。

よくある落とし穴

  • **NULL ポインタアクセス**

    • resource->info が NULL の状態でプロバイダー固有の処理がアクセスを試みると、セグメンテーション違反が発生します。

  • **メモリの二重解放とダングリングポインタ**

    • pathname などの文字列や構造体自体は pool から割り当てられているため、明示的に解放してはなりません。また、プールが破棄された後に本構造体やそのメンバへアクセスすると、未定義の動作を引き起こします。

  • **不整合なキャスト**

    • 異なるプロバイダー間で resource->info を誤ってキャストすると、メモリレイアウトの不整合により未定義の動作やクラッシュの原因となります。
