dav_resource_private

概念

dav_resource_private は、Apache HTTP Server の WebDAV モジュール（mod_dav）において、各リポジトリプロバイダーがリソース（dav_resource）に関連付けられた固有のプライベート情報を保持するための構造体です。

modules/dav/main/mod_dav.h においては不透明な（opaque）構造体として前方宣言されており、具体的なメンバ構成は各プロバイダーの実装に委ねられています。ファイルシステムを扱うプロバイダーである mod_dav_fs（modules/dav/fs/repos.c）では、ファイルパスやリクエストコンテキストなどを保持する具体的な構造体として定義されています。

なぜ重要か

mod_dav コアロジックと、具体的なリポジトリ実装（mod_dav_fs など）を疎結合にするために重要です。コアロジックは dav_resource_private の内部構造を直接操作せず、プロバイダー側が dav_resource->info を通じて自身に必要なメタデータ（ファイルシステム情報やリクエストレコードなど）を安全に保持・参照できるように設計されています。

関与するモジュール

  • module:modules/dav/main （mod_dav コア）
  • module:modules/dav/fs （ファイルシステムプロバイダー mod_dav_fs）

主要エンティティ

  • **symbol:modules/dav/main/mod_dav.h#dav_resource_private**
  • mod_dav コアで定義される不透明な構造体の前方宣言。dav_resource の info フィールドとして保持されます。
  • **symbol:modules/dav/fs/repos.c#dav_resource_private**
  • mod_dav_fs における具体的な構造体定義。以下のメンバを保持します。
  • apr_pool_t *pool: リクエストに関連付けられたメモリプール。
  • const char *pathname: リソースのフルパス。
  • apr_finfo_t finfo: ファイルシステム情報（サイズ、権限、時刻など）。
  • request_rec *r: Apache のリクエストレコード。

エンドツーエンドの流れ

1. **リソースの生成と初期化** mod_dav_fs のリソース取得処理（dav_fs_get_resource 相当の処理）において、dav_resource_private 構造体がメモリプールから割り当てられ、初期化されます。 2. **リソースへの紐付け** 生成されたプライベート構造体へのポインタが、親となる dav_resource 構造体の info メンバに格納されます。 3. **プロバイダー処理での参照** mod_dav_fs 内の各関数（ファイルコピー、移動、プロパティ管理、ディレクトリ走査など）は、resource->info を dav_resource_private* にキャストして、ファイルシステム上のパス（pathname）やリクエスト情報（r）を取得し、実際の処理を実行します。

※なお、mod_dav コアからプロバイダー呼び出しに至る全体的なエンドツーエンドの制御フローの詳細は、提供されたコンテキストからは完全には明らかになっていません。

重要な不変条件

  • **ライフサイクルの同期**

    • dav_resource_private 内の pool は、リクエストの生存期間と同期した apr_pool_t である必要があります。また、pathname などの文字列データはこのプールから割り当てられるため、リクエスト終了時のプール破棄に伴い無効化されます。

  • **有効なリクエスト参照**

    • r メンバは、常に有効な Apache の request_rec を指している必要があります。

  • **非NULLポインタ**

    • プロバイダー固有のデータにアクセスする際、dav_resource->info が NULL でないことが前提となります。

よくある落とし穴

  • **セグメンテーション違反**

    • dav_resource->info が NULL の状態でプロバイダー固有のデータにアクセスしようとすると、セグメンテーション違反が発生します。

  • **未定義動作（キャスト不整合）**

    • 本構造体は mod_dav.h では不透明型として定義されているため、異なるプロバイダー間で誤ったキャストを行うと、メモリレイアウトの不整合により未定義の動作を引き起こす可能性があります。

  • **メモリの寿命**

    • pool が破棄された後に dav_resource_private やそのメンバ（pathname など）を参照すると、解放済みメモリへのアクセス（Use-After-Free）が発生します。

次に読む場所

  • **modules/dav/main/mod_dav.h**
  • 不透明構造体の前方宣言と、それを保持する dav_resource 構造体の定義を確認できます。
  • **modules/dav/fs/repos.c**
  • mod_dav_fs における具体的な構造体定義と、それを利用したリソース操作の実装を確認できます。
