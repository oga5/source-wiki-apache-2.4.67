dav_resource_private

概念

dav_resource_private は、WebDAV リソース（dav_resource 構造体）に関連付けられた、リポジトリ（プロバイダー）固有のプライベートな情報を保持するための構造体です。

mod_dav のコアモジュール（mod_dav.h）においては、具体的なメンバを持たない不透明な（opaque）構造体として前方宣言されています。一方で、具体的なプロバイダーである mod_dav_fs（repos.c）においては、ファイルシステム上のパスやファイル属性、メモリプール、リクエストレコードなどを保持する具体的な構造体として定義されています。

なぜ重要か

この構造体は、mod_dav コアと具体的なリポジトリプロバイダー（mod_dav_fs など）との間を疎結合に保つために重要です。

mod_dav コアはリソースの内部詳細を直接操作せず、プロバイダー側が dav_resource の info メンバ（dav_resource_private *）を介して自身に必要なコンテキスト情報をカプセル化して保持・利用できるように設計されています。

関与するモジュール

  • module:modules/dav/fs (mod_dav_fs プロバイダーの実装)
  • module:modules/dav/main (mod_dav コアの実装)

主要エンティティ

  • symbol:modules/dav/main/mod_dav.h#dav_resource_private
  • mod_dav コアで定義されている不透明な構造体の前方宣言。
  • symbol:modules/dav/fs/repos.c#dav_resource_private
  • mod_dav_fs で定義されている具体的な構造体。以下のメンバを持ちます。
  • apr_pool_t *pool: リクエストに関連付けられたメモリプール。
  • const char *pathname: リソースのファイルシステム上のフルパス。
  • apr_finfo_t finfo: ファイルシステム情報（サイズ、権限、時刻など）。
  • request_rec *r: Apache のリクエストレコード。

エンドツーエンドの流れ

提供された情報からは、リクエスト受信からレスポンス送信に至る完全なエンドツーエンドの流れは判明していません。ただし、リソース操作における部分的な流れは以下の通りです。

1. **初期化**: mod_dav_fs のリソース取得処理（dav_fs_get_resource 相当）において、dav_resource の生成時に dav_resource_private のメモリが割り当てられ、初期化されます。 2. **保持**: 初期化された構造体は、dav_resource の info メンバに格納されます。 3. **利用**: mod_dav_fs 内の各関数（ファイル操作、プロパティ管理、ディレクトリ走査など）が呼び出された際、resource->info を dav_resource_private* にキャストして、ファイルパス（pathname）やファイル情報（finfo）などのコンテキスト情報を取得・利用します。

重要な不変条件

  • **ライフサイクルの同期**: dav_resource_private は、親となる dav_resource のライフサイクルと同期して管理される必要があります。
  • **メモリプールの有効性**: pool メンバは、リクエストの生存期間と同期した有効な apr_pool_t である必要があります。
  • **リクエストレコードの有効性**: r メンバは、有効な Apache request_rec を指していなければなりません。
  • **パス文字列の生存期間**: pathname の文字列は pool から割り当てられているため、リクエスト終了時のプール破棄に伴い無効化されます。

よくある落とし穴

  • **NULL ポインタアクセス**: resource->info が NULL の状態でプロバイダー固有のデータにアクセスしようとすると、セグメンテーション違反が発生します。
  • **解放後の参照（Use-After-Free）**: pool が破棄された後に dav_resource_private やそのメンバ（pathname など）を参照すると、未定義の動作を引き起こします。
  • **不整合なキャスト**: プロバイダーの実装に不整合がある場合、キャスト後のデータアクセスで未定義の動作を引き起こす可能性があります。
