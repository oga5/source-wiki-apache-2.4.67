Module: test/modules/http2/htdocs/cgi

概念

test/modules/http2/htdocs/cgi は、Apache HTTP Server（httpd）のHTTP/2モジュール（mod_http2）の統合テストにおいて、CGI（Common Gateway Interface）の動作を検証するために用意されたPythonスクリプト群です。これらは、HTTP/2プロトコルを経由したCGIプロセスの起動、データの送受信、環境変数の伝播、およびエラーハンドリングが正しく機能するかをテストするためのモックおよび検証用エンドポイントとして機能します。

なぜ重要か

HTTP/2は、単一のTCP接続上で複数のストリームを多重化するプロトコルです。従来のHTTP/1.1とは異なり、CGIプロセスへのリクエストボディのストリーミングやレスポンスの回収が非同期かつ並行して行われます。このディレクトリ内のスクリプト群は、以下のようなHTTP/2特有の挙動やエッジケースをシミュレートし、サーバーがデッドロックやデータ破損を起こさずにCGIと通信できるかを検証するために不可欠です。

  • 大容量データのアップロードとエコーバック
  • チャンク化されたデータのハンドリング
  • 正確なCGI環境変数のマッピング
  • ヘッダーの送受信処理

関与するモジュール

主要エンティティ

このモジュールを構成する主なPythonスクリプトは以下の通りです。

  • **hello.py**: 最もシンプルなCGIテスト。固定のテキストレスポンスを返し、基本的なCGI実行パスを検証します。
  • **echo.py**: クライアントから送信されたリクエストボディをそのままレスポンスとして送り返し、双方向データ転送を検証します。
  • **echohd.py / hecho.py**: 送信されたHTTPリクエストヘッダーを解析し、レスポンスヘッダーまたはボディとしてエコーバックします。
  • **env.py**: CGIプロセスに渡された環境変数（REQUEST_METHOD や HTTP/2固有の変数など）を出力し、環境変数の整合性を検証します。
  • **upload.py**: ファイルアップロードやマルチパートデータの受信処理をシミュレートします。
  • **necho.py**: 特定のバイト数や特殊なパターンでのエコー処理を行います。
  • **mnot164.py**: 特定のテストシナリオ（RFC関連の検証など）に対応するカスタムレスポンスを生成します。
  • **requestparser.py**: 上記の各CGIスクリプトがリクエストデータやマルチパートデータを解析するために共通で使用する内部ヘルパーライブラリです。

エンドツーエンドの流れ

1. **テストの起動**: テストスイート（通常は pytest 等）が、HTTP/2経由でApacheサーバーに対して特定のCGIスクリプト（例: /cgi/echo.py）をターゲットとするリクエストを送信します。 2. **リクエストのハンドリング**: mod_http2 がリクエストを受信し、内部リダイレクトまたはハンドラを介して mod_cgi（または mod_cgid）に処理を渡します。 3. **CGIの実行**: Apacheが該当するPythonスクリプトを子プロセスとして起動します。 4. **データの解析と処理**: スクリプトは requestparser.py を使用して標準入力からリクエストデータを読み込み、要求された処理（エコーや環境変数の出力など）を実行します。 5. **レスポンスの返却**: スクリプトが標準出力にHTTPステータス、ヘッダー、およびボディを書き込みます。ApacheはこれをHTTP/2フレームに変換してクライアントに返送します。

*注意: テストフレームワーク側での具体的なアサーションロジックや、詳細なプロセス間通信（IPC）のライフサイクルは、提供されたファイル群の範囲外であるため、完全なエンドツーエンドの不変条件は未検証です。*

重要な不変条件

  • **CGIプロトコルの遵守**: すべてのスクリプトは、RFC 3875（CGI/1.1）に準拠した形式で標準入出力を処理する必要があります。
  • **非ブロッキング動作**: HTTP/2のフロー制御下において、CGIスクリプトが標準入力を読み込む際、または標準出力に書き出す際に、バッファリングの不整合によるデッドロックが発生しないこと。

*注意: 内部バッファサイズやタイムアウト値などの具体的な不変条件の詳細は、提供されたメタデータからは特定されていません。*

よくある落とし穴

  • **Pythonのバージョン互換性**: テストが実行される環境のPythonインタープリタのバージョン（Python 2系と3系）によって、文字列とバイナリ（bytes）の扱い（特に requestparser.py や echo.py での入出力）が異なり、テストが破損する可能性があります。
  • **実行権限とシバン（Shebang）**: スクリプトが実行可能（chmod +x）であり、シバン（#!/usr/bin/env python3 など）がテスト環境のパスと一致していない場合、Apacheが 500 Internal Server Error を返してテストが失敗します。
  • **バッファリング問題**: Pythonが標準出力をバッファリングすることにより、ヘッダーやボディの送信タイミングが遅れ、HTTP/2のストリームタイムアウトを引き起こすことがあります。
