Shared Object Cache (socache) and Caching Architecture

概念

Shared Object Cache (socache) および Caching Architecture は、Apache HTTP Server においてデータの再利用性と応答性能を向上させるためのキャッシュ機構です。

  • **Shared Object Cache (socache)**: SSLセッションや認証情報など、サイズが小さく頻繁にアクセスされるオブジェクトを、複数プロセス・スレッド間で共有して保持するための共通抽象インターフェース（include/ap_socache.h）およびその実装群です。
  • **HTTP Caching**: mod_cache や mod_cache_disk に代表される、HTTPレスポンス（コンテンツ）そのものをキャッシュし、バックエンドへの不要なリクエストを削減するためのアーキテクチャです。

なぜ重要か

  • **SSL/TLS接続の高速化**: mod_socache_shmcb などを利用してSSLセッションを共有メモリにキャッシュすることで、再接続時の高負荷なフルハンドシェイクを回避し、接続遅延を大幅に削減します。
  • **バックエンド負荷の軽減と応答性の向上**: mod_cache_disk を用いてコンテンツをディスクにキャッシュすることで、動的コンテンツ生成やバックエンドサーバーへの問い合わせ回数を減らし、クライアントへの応答速度を向上させます。
  • **モジュール性の確保**: ap_socache.h で定義された統一APIにより、キャッシュの具体的な格納先（共有メモリ、DBM、memcachedなど）を意識することなく、利用側モジュール（mod_ssl など）がキャッシュ機能を利用できます。

関与するモジュール

  • module:modules/cache (mod_cache.c, mod_cache_disk.c): HTTPコンテンツキャッシュの制御およびディスクへの永続化を担当します。
  • modules/cache/mod_socache_shmcb.c: 共有メモリ（shmcb: Shared Memory Cyclic Buffer）を用いた、高効率なsocacheプロバイダの実装を提供します。

主要エンティティ

  • ap_socache_provider_t (include/ap_socache.h): socacheの実装（初期化、保存、取得、削除など）が満たすべき関数ポインタ群を定義する構造体。
  • mod_cache (modules/cache/mod_cache.c): HTTPキャッシュのフィルタ処理、有効期限の判定、およびキャッシュ制御ヘッダの処理を行うコアモジュール。
  • mod_cache_disk (modules/cache/mod_cache_disk.c): HTTPレスポンスのヘッダおよびボディをディスク上のファイルとして管理・格納するストレージプロバイダ。
  • shmcb (modules/cache/mod_socache_shmcb.c): 共有メモリを循環バッファとして分割管理し、排他制御の競合を最小限に抑えながら高速にオブジェクトを格納するsocacheプロバイダ。

エンドツーエンドの流れ

現時点では、具体的なリクエスト処理時における mod_cache と socache の詳細な連携や、各API呼び出しの完全なエンドツーエンドのシーケンス（例：mod_ssl から mod_socache_shmcb へのセッション保存・復元の詳細なコールスタックなど）は、提供されたソースコードの範囲内では完全には特定されていません。

一般的には、以下のような流れが想定されます：

1. **HTTPキャッシュの場合**:

  • クライアントからのリクエストを mod_cache がフックします。
  • mod_cache_disk などのプロバイダにキャッシュの有無を問い合わせ、有効なキャッシュが存在すればバックエンドに代わってキャッシュされたコンテンツを返します。
  • キャッシュが存在しない場合はバックエンドからコンテンツを取得し、必要に応じて mod_cache_disk を介してディスクに保存します。

2. **Shared Object Cache (socache) の場合**:

  • mod_ssl などの利用側モジュールが、ap_socache_provider_t インターフェースを介して、mod_socache_shmcb などのプロバイダに対してオブジェクト（SSLセッションなど）の保存（store）や取得（retrieve）を要求します。

重要な不変条件

現時点では、提供されたモジュールやヘッダーファイルから直接導き出せる、システム全体にわたる厳密な不変条件（Invariants）の詳細は完全には特定されていません。 ただし、設計上想定される一般的な制約として以下が挙げられます：

  • ap_socache_provider_t を実装するプロバイダは、マルチプロセス・マルチスレッド環境においてスレッドセーフかつプロセス間で安全に共有データにアクセスできなければなりません（例：mod_socache_shmcb における共有メモリの排他制御）。
  • mod_cache は、HTTP/1.1のキャッシュ仕様（RFC）に準拠し、有効期限（Expires）やキャッシュ制御（Cache-Control）ヘッダのルールに従ってキャッシュの有効性を判定しなければなりません。

よくある落とし穴

  • **共有メモリのサイズ不足**: mod_socache_shmcb で確保した共有メモリサイズが小さすぎる場合、高トラフィック環境下で古いセッションが頻繁に上書き（ローテーション）され、キャッシュミス率が上昇する可能性があります。
  • **ディスクキャッシュのクリーンアップ**: mod_cache_disk を使用する際、ディスク上のキャッシュ容量を適切に管理（htcacheclean などの外部ツールによる定期的な削除）しないと、ディスクフルを引き起こす原因になります。
  • **不適切なキャッシュ制御ヘッダ**: バックエンドが不適切な Cache-Control ヘッダを返すと、機密情報が mod_cache にキャッシュされたり、逆にキャッシュすべきコンテンツがキャッシュされなくなったりします。

次に読む場所

  • include/ap_socache.h: 共有オブジェクトキャッシュプロバイダのAPI定義と、各関数の役割を理解する。
  • modules/cache/mod_socache_shmcb.c: 共有メモリを用いたサイクリックバッファ（shmcb）の具体的な実装と排他制御の仕組みを学ぶ。
  • modules/cache/mod_cache.c: HTTPキャッシュのコアロジック、フィルタ処理、およびキャッシュ判定基準を調査する。
  • modules/cache/mod_cache_disk.c: ディスクへのキャッシュデータの書き込み、読み込み、およびファイル構造を理解する。
