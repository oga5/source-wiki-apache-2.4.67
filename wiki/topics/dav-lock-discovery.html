dav_lock_discovery

概念

dav_lock_discovery は、Apache HTTP Server の WebDAV モジュール群において、リソースに対する「直接ロック（Direct Lock）」の情報をメモリ上で保持するための構造体です。この構造体は、単方向のリンクドリスト（next ポインタを使用）として構築され、同一リソースに対する複数のロック情報を管理します。

なぜ重要か

WebDAV 仕様におけるロック管理（特に直接ロックの検出や永続化）を支える中核的なデータ構造だからです。DBM などの永続ストレージから読み込まれたロック情報のデシリアライズ先、およびストレージへ書き込む際のシリアライズ元として機能し、ロックの有効期限チェックやクライアントへのロック情報返却に利用されます。

関与するモジュール

  • module:modules/dav/fs (ファイルシステムベースの WebDAV ロック管理)
  • module:modules/dav/lock (汎用的な WebDAV ロック管理)

主要エンティティ

  • **symbol:modules/dav/fs/lock.c#dav_lock_discovery**: ファイルシステム層における直接ロック情報の保持用構造体。dav_fs_load_lock_record や dav_fs_save_lock_record などの関数から操作されます。
  • **symbol:modules/dav/lock/locks.c#dav_lock_discovery**: 汎用ロック管理層における直接ロック情報の保持用構造体。dav_generic_load_lock_record や dav_generic_save_lock_record などの関数から操作されます。

エンドツーエンドの流れ

1. **読み込みと構築**:

  • DBM などの永続ストレージからロックレコードが読み込まれます（dav_fs_load_lock_record または dav_generic_load_lock_record）。
  • 読み込みの際、現在時刻と比較して期限切れのロックは除外され、有効なロックのみが dav_lock_discovery 構造体のインスタンスとしてデシリアライズされ、リンクドリストとして再構築されます。

    • 2. **参照と操作**:

  • dav_fs_get_locks などの関数が、構築されたリストを参照してロック状態を判定します。

    • 3. **シリアライズと保存**:

  • ロック情報の更新時、dav_lock_discovery のリストが走査され、各要素がバイナリ形式にシリアライズされて DBM へ書き込まれます（dav_fs_save_lock_record または dav_generic_save_lock_record）。

※なお、WebDAV リクエストの受信からレスポンス送信に至る完全なエンドツーエンドの制御フローの全容は、提供された情報からは完全には特定されていません。

重要な不変条件

  • **メモリ管理**: すべての dav_lock_discovery インスタンスは、関連するリクエスト（request_rec）のメモリプール（r->pool）から割り当てられる必要があります。
  • **リスト構造**: next ポインタを介して単方向リンクドリストを形成します。
  • **固定長データとの分離**: ロックの有効期限やスコープなどの固定長データは、構造体の先頭にある f メンバ（dav_lock_discovery_fixed 構造体）に格納され、可変長データ（owner や auth_user など）と区別されます。
  • **シリアライズの整合性**: 構造体内の各フィールドは、DBM への書き込み・読み込み時に使用されるオフセット計算ロジック（dav_size_direct など）と厳密に整合している必要があります。

よくある落とし穴

  • **シリアライズ不整合によるデータ破損**: modules/dav/lock/locks.c 内のオフセット計算やシリアライズロジックと、構造体の定義が一致しない場合、DBM からの読み込み時にデータ破損が発生する可能性があります。
  • **メモリ消費**: メモリ管理は Apache のプール機構に依存しているため明示的な free は不要ですが、長期間実行されるプロセスや大量のロックを扱う処理では、メモリ消費量の累積に注意する必要があります。
  • **DBM の状態依存**: DBM が読み取り専用で開かれている場合、この構造体を用いた保存操作（永続化）は失敗します。また、DBM 自体の破損に対する保護機能は本構造体にはありません。
  • **モジュール間の密結合**: modules/dav/fs/lock.c と modules/dav/lock/locks.c は密接に関連しているため、一方の構造体やシリアライズロジックを変更する際は、もう一方の参照ロジックに影響がないか確認する必要があります。
