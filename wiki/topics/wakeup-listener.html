wakeup_listener

概念

wakeup_listener は、Apache HTTP Serverのマルチプロセッシングモジュール（MPM）である worker や event において、ブロッキング状態（ソケットの監視やキューの待機など）にあるリスナースレッド（listener_thread）を強制的に起床させ、終了処理へ移行させるための関数です。

なぜ重要か

MPMのシャットダウンや再起動の際、リスナースレッドが接続待ち（poll）などでブロックし続けると、プロセスが正常に終了できなくなります。wakeup_listener は、終了フラグの設定、ポルセットのウェイクアップ、キューの終端処理、およびシグナル送信を組み合わせることで、リスナースレッドを確実に待機状態から脱出させ、安全かつ迅速なプロセス終了を実現するために不可欠な役割を担っています。

エンドツーエンドの流れ

提供された worker.c の情報に基づく、リスナー起床の処理フローは以下の通りです。

1. **終了フラグの設定**: listener_may_exit フラグを 1 に設定し、スレッドに終了を指示します。 2. **ポルセットの解除**: worker_pollset が存在し、かつ listener_is_wakeable が真であれば、apr_pollset_wakeup を実行して poll() によるブロッキングを解除します。 3. **キューの終端化**: ap_queue_info_term を呼び出し、ワーカーを待機しているリスナーのキュー待ち状態を解放します。 4. **シグナル送信**: listener_os_thread が初期化済みであれば、プラットフォームに応じて以下のいずれかを実行し、スレッドへ直接シグナルを送信して起床させます。

  • HAVE_PTHREAD_KILL が定義されている場合: pthread_kill を使用。
  • それ以外の場合: kill(ap_my_pid, LISTENER_SIGNAL) を使用。

※ event MPM における詳細なエンドツーエンドの流れや、呼び出し元から本関数がトリガーされる具体的な全体シーケンスの全容は、提供されたコンテキストからは未だ完全には明らかになっていません。

重要な不変条件

  • **終了フラグの即時反映**: 本関数が呼ばれると、必ず listener_may_exit が 1 に設定されます。
  • **キューの解放保証**: worker_queue_info に対する ap_queue_info_term の呼び出しが確実に行われ、キューに起因するデッドロックを防ぎます。

よくある落とし穴

  • **早期リターンによる終了漏れ**: リスナースレッドの生成直後など、listener_os_thread が未設定（初期化前）の段階で wakeup_listener が呼び出された場合、シグナル送信処理がスキップされます。この際、一部の終了処理が完全に行われない可能性がある旨がソースコード内に注釈されています。
  • **プラットフォーム依存のシグナル挙動**: pthread_kill を使用する環境と、プロセス全体に kill を送る環境で挙動が分かれるため、シグナルハンドリングを変更する際は注意が必要です。
