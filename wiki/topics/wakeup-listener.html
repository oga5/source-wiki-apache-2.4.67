wakeup_listener

概念

wakeup_listener は、Apache HTTP Server（httpd）のマルチプロセッシングモジュール（MPM）である worker や event において、ブロッキング状態にあるリスナースレッド（listener_thread）を強制的に起床させ、終了処理へ移行させるための関数です。

なぜ重要か

MPMの終了シーケンスにおいて、リスナースレッドがソケットの監視（poll）やワーカーの空き待ちなどでブロッキング状態にある場合、プロセスを安全かつ迅速に終了させることができません。wakeup_listener は、シグナル送信やポーリングセットのウェイクアップ、キュー情報の終端処理を組み合わせることで、リスナーを確実に待機状態から脱出させる役割を担うため重要です。

エンドツーエンドの流れ

worker MPMにおける流れは以下の通りです。

1. プロセスの終了や再起動などのイベントに伴い、wakeup_listener が呼び出されます（server/mpm/worker/worker.c 内の複数箇所から呼び出し）。 2. listener_may_exit フラグを 1 に設定し、リスナーに対して終了指示を出します。 3. worker_pollset が有効かつ listener_is_wakeable が真の場合、apr_pollset_wakeup を呼び出して poll() のブロッキングを解除します。 4. ap_queue_info_term を呼び出し、ワーカーを待機しているリスナーを解放します。 5. listener_os_thread が初期化済みであれば、プラットフォームに応じてスレッドへシグナルを送信します。

  • HAVE_PTHREAD_KILL が定義されている場合: pthread_kill を使用。
  • それ以外の場合: kill(ap_my_pid, LISTENER_SIGNAL) を使用。

※ event MPMにおける詳細なエンドツーエンドの流れや、worker MPMとの差異については、提供されたコンテキストからは未解明です。

重要な不変条件

worker MPMにおける不変条件は以下の通りです。

  • 呼び出し後、listener_may_exit フラグが必ず 1 に設定されること。
  • worker_pollset が有効な場合、apr_pollset_wakeup が呼び出されること。
  • worker_queue_info に対して ap_queue_info_term が呼び出され、ワーカー待ち状態が解除されること。

※ event MPMにおける不変条件については、提供されたコンテキストからは未解明です。

よくある落とし穴

  • **初期化前の呼び出しによる不完全な終了**: リスナー生成直後で listener_os_thread が未設定（未初期化）の場合、シグナル送信処理がスキップされて早期リターンします。この際、一部の終了処理が完全に行われない可能性があることがソースコード内に注釈されています。
  • **プラットフォーム依存の挙動**: シグナル送信において、pthread_kill を使用する環境と kill を使用する環境で挙動が分かれるため、シグナルハンドリングを変更する際には注意が必要です。
