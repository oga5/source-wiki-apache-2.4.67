wakeup_listener

概念

wakeup_listener は、Apache HTTP Serverのマルチプロセッシングモジュール（MPM）である event および worker において、ブロッキング状態にあるリスナースレッド（listener_thread）を強制的に起床させ、終了処理へと移行させるための内部関数です。

なぜ重要か

MPMのシャットダウンや再起動の際、リスナースレッドがネットワークイベントの待機（poll など）やワーカーの空き待ちでブロックし続けると、プロセスが正常に終了できなくなります。wakeup_listener は、これらの待機状態を安全かつ確実に解除し、スレッドを終了シーケンスへ導くために不可欠な役割を担っています。

エンドツーエンドの流れ

worker MPMにおける大まかな流れは以下の通りです。なお、event MPMにおける詳細なエンドツーエンドの流れは現時点では完全には解析されていません。

1. **終了トリガーの発生**: join_workers や signal_threads などの終了処理ルーチン（worker.c 内の呼び出し元など）から wakeup_listener が呼び出されます。 2. **終了フラグの設定**: listener_may_exit フラグを 1 に設定し、リスナーに終了を指示します。 3. **ポーリングの解除**: worker_pollset が存在し、かつ listener_is_wakeable が真であれば、apr_pollset_wakeup を実行して poll() によるブロッキングを解除します。 4. **キューの終了処理**: ap_queue_info_term を呼び出して worker_queue_info を終了させ、ワーカーを待機しているリスナーを解放します。 5. **シグナル送信**: listener_os_thread が初期化済みであれば、プラットフォームに応じて pthread_kill または kill(ap_my_pid, LISTENER_SIGNAL) を使用してリスナースレッドにシグナルを送信し、確実に起床させます。

重要な不変条件

  • worker MPMにおいて、wakeup_listener が実行されると、listener_may_exit フラグは必ず 1 に設定されます。
  • worker_pollset が有効な場合、apr_pollset_wakeup が呼び出されます。
  • worker_queue_info に対して ap_queue_info_term が呼び出され、ワーカー待ち状態が解除されます。
  • event MPMにおける不変条件の詳細は現時点では未解析です。

よくある落とし穴

  • **早期リターンのリスク**: リスナースレッドの生成直後など、listener_os_thread がまだ初期化（設定）されていない段階で wakeup_listener が呼び出された場合、シグナル送信処理がスキップされて早期リターンします。この場合、一部の終了処理が完全に実行されない可能性があることがソースコード内の注釈で指摘されています。
  • **プラットフォーム依存の挙動**: シグナル送信において、HAVE_PTHREAD_KILL の有無により pthread_kill と kill のどちらが使用されるかが分かれるため、シグナルハンドリングを変更する際にはプラットフォームごとの挙動の違いに注意する必要があります。
