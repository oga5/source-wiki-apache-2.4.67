wakeup_listener

概念

wakeup_listener は、Apache HTTP Serverのマルチプロセッシングモジュール（MPM）において、ブロッキング状態にあるリスナースレッド（listener_thread）を強制的に起床させ、終了処理へと移行させるためのシグナル送信および状態解除を行う関数です。

なぜ重要か

MPMのシャットダウンや再起動を行う際、リスナースレッドがネットワークイベントの待機（poll）やワーカーの空き待ちなどでブロックし続けていると、プロセスが正常に終了できなくなります。wakeup_listener は、これらのブロッキング状態を安全かつ確実に解除し、迅速な終了シーケンスを保証するために不可欠な役割を担っています。

エンドツーエンドの流れ

worker MPMにおける終了シーケンス中の流れは以下の通りです：

1. シャットダウンや再起動のトリガーに伴い、wakeup_listener が呼び出されます。 2. listener_may_exit フラグを 1 に設定し、リスナーに対して終了指示を明示します。 3. リスナーがソケットを監視している場合（worker_pollset が有効かつ listener_is_wakeable が真）、apr_pollset_wakeup を呼び出して poll() のブロッキングを解除します。 4. ap_queue_info_term を呼び出し、ワーカーの空きを待機しているリスナーを解放します。 5. リスナースレッド（listener_os_thread）が初期化済みであれば、プラットフォームに応じて以下のいずれかの方法でスレッドへシグナルを送信し、確実に起床させます。

  • HAVE_PTHREAD_KILL が定義されている場合: pthread_kill を使用。
  • それ以外の場合: kill(ap_my_pid, LISTENER_SIGNAL) を使用。

※ event MPMにおけるエンドツーエンドの流れの詳細は、提供されたコンテキストからは未だ不明です。

重要な不変条件

  • 呼び出し時、listener_may_exit フラグは必ず 1 に設定されます。
  • worker_pollset が有効な場合、必ず apr_pollset_wakeup が呼び出されます。
  • worker_queue_info に対して必ず ap_queue_info_term が呼び出され、ワーカー待ち状態が解除されます。

よくある落とし穴

  • **早期リターンによる終了不全**: リスナースレッドの生成直後など、listener_os_thread が未設定（未初期化）の状態で wakeup_listener が呼び出された場合、シグナル送信処理がスキップされて早期リターンします。この際、一部の終了処理が完全に行われない可能性があることがソースコード内で指摘されています。
  • **プラットフォーム依存のシグナル送信**: スレッドへのシグナル送信は、HAVE_PTHREAD_KILL の有無によって pthread_kill と kill の間で切り替わります。シグナルハンドリングを変更する際は、プラットフォームごとの挙動の違いに注意する必要があります。
