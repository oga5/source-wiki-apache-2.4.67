wakeup_listener

概念

wakeup_listener は、Apache HTTP Serverのマルチプロセッシングモジュール（MPM）において、ブロッキング状態（待機状態）にあるリスナースレッド（listener_thread）を強制的に解除し、安全な終了処理へ移行させるためのシグナル送信および状態更新を行う関数です。

なぜ重要か

MPM（worker や event）の終了時、リスナースレッドがソケットの監視（poll）やワーカーの空き待ちなどでブロックし続けていると、プロセス全体のクリーンアップや終了処理が滞る原因になります。wakeup_listener は、これらのブロッキング状態を確実に解除し、スレッドを終了フェーズへ進めるために不可欠な役割を担っています。

エンドツーエンドの流れ

1. **終了指示の開始**: MPMの終了シーケンスにおいて、wakeup_listener が呼び出されます。 2. **終了フラグの設定**: listener_may_exit フラグを 1 に設定し、リスナーに対して終了要求を通知します。 3. **ポーリングの解除**: worker_pollset が有効かつ listener_is_wakeable が真である場合、apr_pollset_wakeup を呼び出して poll() によるソケット監視のブロッキングを解除します。 4. **キューの解放**: ap_queue_info_term を呼び出し、ワーカーの空きを待機しているリスナーを解放します。 5. **シグナルの送信**: listener_os_thread が初期化済みであれば、プラットフォームに応じて pthread_kill または kill(ap_my_pid, LISTENER_SIGNAL) を実行し、リスナースレッドへ直接シグナルを送信して待機状態を強制解除します。

*注意: event MPMにおける詳細なエンドツーエンドの流れや、呼び出し元（server/mpm/event/event.c など）との完全な連携フローは、現時点の提供情報からは未確定です。*

重要な不変条件

  • listener_may_exit フラグは必ず 1 に設定されます。
  • worker_pollset が有効な場合は、apr_pollset_wakeup が確実に呼び出されます。
  • worker_queue_info に対して ap_queue_info_term が呼び出され、ワーカー待ち状態が解除されます。

よくある落とし穴

  • **早期リターンのリスク**: リスナースレッドの生成直後など、listener_os_thread がまだ初期化されていない段階で wakeup_listener が呼び出されると、シグナル送信処理がスキップされて早期リターンします。この場合、一部の終了処理が完全に行われない可能性があることがソースコード内で注釈されています。
  • **プラットフォーム依存の挙動**: シグナル送信において、HAVE_PTHREAD_KILL の有無により pthread_kill と kill のどちらが使用されるかが分かれるため、シグナルハンドリングの挙動がプラットフォーム間で異なる点に注意が必要です。
