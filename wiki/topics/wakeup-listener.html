wakeup_listener

概念

wakeup_listener は、Apache HTTP Serverのマルチプロセッシングモジュール（MPM）である worker や event において、ブロッキング状態（待機状態）にあるリスナースレッド（listener_thread）を強制的に解除し、終了処理へと移行させるための関数です。

なぜ重要か

MPMの終了シーケンスにおいて、リスナースレッドがソケットの監視（poll）やワーカーキューの空き待ちなどでブロックし続けるのを防ぎ、安全かつ迅速にプロセスをシャットダウンするために不可欠な役割を担っています。

エンドツーエンドの流れ

worker MPMにおける大まかな流れは以下の通りです。なお、event MPMにおける詳細なエンドツーエンドの流れは提供された情報からは未確定です。

1. **終了指示のセット**: listener_may_exit フラグを 1 に設定し、終了要求を明示します。 2. **Pollsetの解除**: worker_pollset が有効かつ listener_is_wakeable が真の場合、apr_pollset_wakeup を呼び出して poll() によるブロッキングを解除します。 3. **キュー待機の解除**: ap_queue_info_term を呼び出し、ワーカーを待機しているリスナーを解放します。 4. **シグナル送信**: listener_os_thread が初期化済みであれば、プラットフォームに応じて pthread_kill または kill(ap_my_pid, LISTENER_SIGNAL) を用いてスレッドにシグナルを送信し、確実にスレッドを起床させます。

重要な不変条件

  • wakeup_listener が呼び出されると、listener_may_exit フラグは必ず 1 に設定されます。
  • listener_os_thread が初期化されている場合にのみ、スレッドへの直接的なシグナル送信（pthread_kill 等）が実行されます。
  • event MPMにおける不変条件の詳細は提供された情報からは未確定です。

よくある落とし穴

  • **早期リターンの影響**: リスナースレッドの生成直後など、listener_os_thread がまだ設定されていないタイミングで wakeup_listener が呼び出されると、シグナル送信処理がスキップされて早期リターンします。この際、一部の終了処理が完全に行われない可能性があることがソースコード内で注釈されています。
  • **プラットフォーム依存の挙動**: シグナル送信において、HAVE_PTHREAD_KILL の定義有無により pthread_kill と kill のどちらが使用されるかが分かれるため、シグナルハンドリングを変更する際には注意が必要です。
