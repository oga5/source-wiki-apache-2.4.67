dav_lockdb_private

概念

dav_lockdb_private は、Apache HTTP Serverの mod_dav モジュール群において、WebDAVロックデータベース（DBM）へのアクセスや操作に必要なプロバイダー固有のコンテキスト情報を保持するための構造体です。

mod_dav.h においては不透明な構造体（opaque structure）として前方宣言されており、具体的な構造体の定義は各ロックプロバイダーの実装（modules/dav/fs/lock.c や modules/dav/lock/locks.c）に隠蔽されています。

なぜ重要か

この構造体は、mod_dav コアと具体的なロックデータベースプロバイダー実装との間を疎結合に保つために重要です。

mod_dav コア側は dav_lockdb_private の具体的なメンバに依存せず、ポインタとしてのみ扱います。一方で、プロバイダー側はリクエストコンテキスト（request_rec）、メモリプール（apr_pool_t）、データベースパス、およびDBMハンドル（dav_db や apr_dbm_t）などの内部状態をこの構造体の中にカプセル化し、遅延オープンなどの効率的なリソース管理を実現できます。

関与するモジュール

  • module:modules/dav/fs （ファイルシステムベースのロックプロバイダー）
  • module:modules/dav/lock （汎用的なDBMロックプロバイダー）
  • module:modules/dav/main （WebDAVコアモジュール）

主要エンティティ

  • **symbol:modules/dav/main/mod_dav.h#dav_lockdb_private**

    • 不透明な構造体としての前方宣言。dav_lockdb 構造体内の info メンバ（ポインタ）として参照されます。

  • **symbol:modules/dav/fs/lock.c#dav_lockdb_private**

    • ファイルシステムベースのロック実装における具体的な構造体定義。dav_db ハンドルを保持します。

  • **symbol:modules/dav/lock/locks.c#dav_lockdb_private**

    • 汎用DBMロック実装における具体的な構造体定義。apr_dbm_t ハンドルを保持します。

エンドツーエンドの流れ

詳細なエンドツーエンドの呼び出しフローの全容は提供された情報からは完全には明らかになっていませんが、以下の部分的なライフサイクルが判明しています。

1. **初期化と遅延オープン** ロックデータベースのオープン時（例: dav_generic_open_lockdb）、構造体が初期化されてパスやプールが設定されますが、この時点では実際のDBMハンドルは開かれません。 2. **実際のオープン** 実際のI/O処理が発生するタイミングで、dav_generic_really_open_lockdb などの関数を通じて db フィールドが設定され、データベースが実際に開かれます。 3. **リソース解放** リクエスト終了時に、構造体に紐付けられたメモリプール（pool）が破棄されることで、関連するデータベースハンドルやメモリリソースが自動的に解放されます。

重要な不変条件

  • **遅延オープンの整合性**

    • db フィールドが非NULLである場合、データベースは正常に開かれた有効な状態（opened フラグが真）でなければなりません。

  • **メモリ管理の局所性**

    • 構造体およびそのメンバに必要なメモリは、リクエストに関連付けられたプール（pool / request_rec のプール）から割り当てられる必要があります。

  • **プロバイダー間の独立性**

    • mod_dav コアは本構造体の内部メンバに直接アクセスしてはならず、各プロバイダーの実装内でのみ具体的な定義が参照・操作されます。

よくある落とし穴

  • **未オープン状態での操作**

    • db ハンドルが NULL の状態でデータベース操作を試みると、適切なオープン処理（dav_generic_really_open_lockdb など）を経由しない限り操作が失敗します。

  • **書き込み制限**

    • データベースが読み取り専用で開かれている場合、書き込み操作や期限切れロックの自動クリーンアップ処理は失敗するかスキップされます。

  • **定義の不一致**

    • 本構造体は mod_dav.h で前方宣言されているため、新しいプロバイダーを実装する際や構造体を変更する際は、すべてのロックプロバイダー実装との整合性を保つ必要があります。

次に読む場所

  • **modules/dav/main/mod_dav.h**

    • 前方宣言と、dav_lockdb 構造体での info ポインタとしての使われ方を確認できます。

  • **modules/dav/fs/lock.c**

    • ファイルシステムプロバイダーにおける具体的な定義と、dav_fs_open_lockdb などの操作を確認できます。

  • **modules/dav/lock/locks.c**

    • 汎用DBMプロバイダーにおける具体的な定義と、dav_generic_really_open_lockdb などの操作を確認できます。
