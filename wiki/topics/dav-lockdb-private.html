dav_lockdb_private

概念

dav_lockdb_private は、Apache HTTP Serverの mod_dav モジュール群において、WebDAVロックデータベース（DBM）へのアクセスや操作に必要なコンテキスト、状態情報、およびデータベースハンドルを保持するための内部（プライベート）構造体です。この構造体は、ファイルシステムベースのロックプロバイダー（modules/dav/fs）と、ジェネリックなロックプロバイダー（modules/dav/lock）のそれぞれで個別に定義されています。

なぜ重要か

この構造体は、ロックデータベースの「遅延オープン（Lazy Open）」をサポートし、不要なI/O処理を削減するために重要です。データベースのパスやリクエストコンテキスト（request_rec）、メモリプールを保持し、実際にロックの読み書きが発生するまでデータベースのオープン処理を遅延させる状態管理を担います。また、上位の dav_lockdb（または dav_lockdb_combined）構造体の info メンバとしてカプセル化され、プロバイダー固有の実装詳細を隠蔽する役割も果たします。

主要エンティティ

  • symbol:modules/dav/fs/lock.c#dav_lockdb_private: ファイルシステムベースのロック実装で使用される構造体。dav_db ハンドルを保持します。
  • symbol:modules/dav/lock/locks.c#dav_lockdb_private: ジェネリックなロック実装で使用される構造体。apr_dbm_t ハンドルを保持します。
  • symbol:modules/dav/main/mod_dav.h#dav_lockdb_private: dav_lockdb などの上位構造体の info ポインタを介して、これらプライベート構造体を参照します。

エンドツーエンドの流れ

1. **初期化**: ロックデータベースの初期化時（例: dav_generic_open_lockdb など）に dav_lockdb_private 構造体が割り当てられ、request_rec、apr_pool_t、lockdb_path が設定されます。この時点では opened は 0（偽）であり、データベースハンドル db は NULL です。 2. **遅延オープン**: 実際にロックレコードの読み込みや保存（例: dav_generic_load_lock_record など）が行われる際、dav_generic_really_open_lockdb などの関数を通じてデータベースが実際に開かれ、db フィールドに有効なハンドルが設定され、opened が 1（真）になります。 3. **データベース操作**: 開かれた db ハンドル（dav_db または apr_dbm_t）を介して、ロック情報の読み書きが実行されます。 4. **クリーンアップ**: リクエストの終了に伴い、関連付けられた apr_pool_t が破棄されることで、関連するリソースが自動的に解放されます。

*注意: 詳細な関数呼び出しの全容や、エラーハンドリングを含む完全なエンドツーエンドのライフサイクルは、提供されたソースコードの範囲内では一部未確定です。*

重要な不変条件

  • **遅延オープンの整合性**: db が非NULLである場合、opened フラグは真（データベースが正常に開かれている状態）でなければなりません。
  • **メモリ管理の局所性**: 構造体およびそのメンバに必要なメモリは、必ず構造体内の pool（通常は r->pool に由来）から割り当てられなければなりません。
  • **リクエストの有効性**: r メンバは、現在処理中の有効な request_rec を指し示している必要があります。

よくある落とし穴

  • **未オープン状態での操作**: db ハンドルが NULL の状態で直接データベース操作を試みると、セグメンテーションフォールトや操作の失敗を招きます。必ず事前にオープン処理（dav_generic_really_open_lockdb 等）を経由させる必要があります。
  • **読み取り専用時の書き込み**: データベースが読み取り専用で開かれている場合、書き込み操作や期限切れロックの自動クリーンアップ処理は失敗するか、スキップされます。
  • **定義の混同**: modules/dav/fs/lock.c と modules/dav/lock/locks.c で同名の構造体が個別に定義されており、保持する db メンバの型（dav_db * と apr_dbm_t *）が異なります。これらは各モジュールのプライベート定義であるため、相互にキャストしたり混同したりしないよう注意が必要です。
