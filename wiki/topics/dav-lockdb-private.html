dav_lockdb_private

概念

dav_lockdb_private は、Apache HTTP Server の mod_dav モジュールにおいて、WebDAV ロックデータベース（DBM）へのアクセスや操作に必要なコンテキスト、状態情報、およびデータベースハンドルを保持するための内部プライベート構造体です。

この構造体は、ファイルシステムベースのロックプロバイダー（modules/dav/fs/lock.c）と、汎用的なロックプロバイダー（modules/dav/lock/locks.c）のそれぞれで同名の構造体として定義されており、各プロバイダーの内部状態管理に特化したフィールドを持っています。

なぜ重要か

WebDAV のロック管理では、リクエストごとにロックデータベースへの接続や読み書きが発生します。dav_lockdb_private は以下の役割を果たすことで、効率的かつ安全なロック管理を実現します。

  • **遅延オープン（Lazy Open）のサポート**: データベースの実際のオープン処理を初回アクセス時まで遅延させ、不要な I/O 負荷を削減します。
  • **リソース管理の統合**: Apache のリクエストコンテキスト（request_rec）やメモリプール（apr_pool_t）と紐付けることで、リクエスト終了時に自動的かつ安全にリソースを解放します。
  • **プロバイダーごとのカプセル化**: 上位の dav_lockdb 構造体の info メンバ（ポインタ）を介して参照されるため、具体的なデータベース実装（DBM ハンドルの型など）を隠蔽します。

関与するモジュール

  • module:modules/dav/fs (ファイルシステムベースのロックプロバイダー)
  • module:modules/dav/lock (汎用的なロックプロバイダー)
  • module:modules/dav/main (mod_dav のメインヘッダー定義)

主要エンティティ

  • **symbol:modules/dav/fs/lock.c#dav_lockdb_private**: ファイルシステムベースのロックデータベース（dav_db 抽象化ハンドルを使用）の状態を追跡する構造体。
  • **symbol:modules/dav/lock/locks.c#dav_lockdb_private**: 汎用的なロックデータベース（apr_dbm_t ハンドルを使用）の状態を追跡する構造体。
  • **symbol:modules/dav/main/mod_dav.h#dav_lockdb_private**: dav_lockdb 構造体の info メンバとして、各プロバイダーのプライベートデータへのポインタを保持するための前方参照。

エンドツーエンドの流れ

1. **初期化**: ロックデータベースのオープン要求（dav_fs_open_lockdb や dav_generic_open_lockdb など）が発生すると、dav_lockdb_private がメモリプールから割り当てられ、request_rec、apr_pool_t、lockdb_path などの基本情報が設定されます。この時点では、実際のデータベースファイルはまだ開かれません。 2. **遅延オープン**: 実際にロックの読み書きや削除などの I/O 操作が発生したタイミングで、dav_generic_really_open_lockdb などの関数が呼び出され、データベースハンドル（db）が取得され、opened フラグがセットされます。 3. **ロック操作**: dav_generic_load_lock_record や dav_generic_save_lock_record などの関数が、この構造体に保持されている db ハンドルを介して DBM 操作を実行します。 4. **クリーンアップ**: リクエストの終了に伴い、関連付けられたメモリプール（pool）が破棄されることで、データベースハンドルや割り当てられたメモリなどのリソースが自動的に解放されます。

*注意: 上記以外の詳細なエラーハンドリングや、特定のロック競合時における詳細なエンドツーエンドの排他制御フローは、提供された情報からは完全には明らかになっていません。*

重要な不変条件

  • **データベースハンドルの有効性**: db フィールドが非NULLである場合、データベースは正常に開かれており、有効な状態である必要があります。
  • **遅延オープンの整合性**: opened フラグと db ハンドルの状態は常に同期している必要があります（opened が真であれば db は有効、またはその逆）。
  • **メモリプールの整合性**: 構造体およびそのメンバに割り当てられるメモリは、必ず構造体内の pool（通常は r->pool）から割り当てられなければなりません。
  • **書き込み制限**: データベースが読み取り専用モードで開かれている場合、書き込み操作や期限切れロックの自動クリーンアップ処理は実行できません。

*注意: 複数プロセス・スレッド間でのロックデータベースファイル自体の排他制御や一貫性に関する不変条件は、提供された情報からは完全には明らかになっていません。*

よくある落とし穴

  • **未オープン状態での操作**: db ハンドルが NULL の状態で直接データベース操作を試みると失敗します。必ず事前に遅延オープン処理（really_open など）を経由させる必要があります。
  • **読み取り専用時の書き込みエラー**: 読み取り専用で開かれたデータベースに対して書き込みや削除を試みるとエラーが発生します。特に、期限切れロックの自動削除処理などが意図せずスキップされる原因になります。
  • **構造体定義の不一致**: dav_lockdb_private は fs/lock.c と lock/locks.c で個別に定義されているため、一方のフィールドを変更する際、もう一方やそれらを利用するシリアライズロジック（dav_generic_save_lock_record など）との整合性を誤って崩さないよう注意が必要です。
