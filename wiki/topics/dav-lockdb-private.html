dav_lockdb_private

概念

dav_lockdb_private は、Apache HTTP Server の mod_dav モジュール群において、WebDAVロックデータベース（DBM）へのアクセスに必要な状態情報やコンテキストを保持するためのプライベートな構造体です。この構造体は、modules/dav/fs/lock.c と modules/dav/lock/locks.c の2箇所でそれぞれ定義されており、それぞれのロックプロバイダー実装に特化したデータベースハンドル（dav_db または apr_dbm_t）をカプセル化しています。

なぜ重要か

WebDAVのロック管理において、データベースのオープン状態、パス、メモリプール、およびデータベースハンドルをカプセル化し、データベースの実際のオープン処理を遅延させる「遅延オープン（lazy open）」や、リクエストのライフサイクルに合わせたメモリ管理を効率的に行うために重要です。これにより、不要なデータベース接続を避け、リソースの消費を抑えることができます。

関与するモジュール

主要エンティティ

  • **symbol:modules/dav/fs/lock.c#dav_lockdb_private**: ファイルシステムベースのロックデータベース（dav_db）へのアクセス状態を保持する構造体。
  • **symbol:modules/dav/lock/locks.c#dav_lockdb_private**: 汎用的なDBM（apr_dbm_t）ベースのロックデータベースへのアクセス状態を保持する構造体。
  • **symbol:modules/dav/main/mod_dav.h#dav_lockdb_private**: dav_lockdb 構造体などの info メンバとして参照される、不透明なプライベート構造体としての前方宣言。

エンドツーエンドの流れ

提供されたソース情報に基づく、ロックデータベース操作の基本的な流れは以下の通りです。

1. **初期化**: dav_generic_open_lockdb（または dav_fs_open_lockdb）などによって dav_lockdb_private 構造体が初期化されます。この段階では、データベースは即座には開かれません。 2. **遅延オープン**: 実際のI/O操作が発生するタイミングで、dav_generic_really_open_lockdb などを通じてデータベースが実際に開かれ、db フィールドにハンドルが設定され、opened フラグが更新されます。 3. **データベース操作**: dav_generic_load_lock_record などの関数が、この構造体内の db ハンドルを介して DBM 操作を実行します。 4. **リソース解放**: リクエスト終了時に、構造体に関連付けられたメモリプール（pool）が破棄されることで、関連するリソースが自動的に解放されます。

*注意: 詳細なエラーハンドリングパスや、すべての呼び出し元におけるエンドツーエンドの完全な制御フローは、提供されたコンテキストからは完全には特定されていません。*

重要な不変条件

  • **有効なハンドル**: db フィールドが非NULLである場合、データベースは正常に開かれており、有効な状態であるとみなされます。
  • **メモリプールの整合性**: 構造体および関連するメモリ割り当ては、必ず request_rec に関連付けられた pool メンバから行われなければなりません。
  • **遅延状態の同期**: opened フラグと db ハンドルの状態は、遅延初期化の制御において一貫している必要があります。

よくある落とし穴

  • **未オープン状態での操作**: db が NULL の状態で直接データベース操作を試みると、事前に really_open 処理が実行されない限り操作が失敗します。
  • **書き込み権限の欠如**: データベースが読み取り専用で開かれている場合、書き込み操作や期限切れロックの自動削除（クリーンアップ）処理は失敗するか、スキップされます。
  • **シリアライズの不整合**: 本構造体はロックプロバイダーの実装と密結合しているため、フィールドの追加や変更を行う際は、locks.c 内のシリアライズロジック（dav_generic_save_lock_record 等）との整合性に注意する必要があります。
