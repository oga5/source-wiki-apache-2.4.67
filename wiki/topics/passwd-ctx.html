passwd_ctx

概念

passwd_ctx は、Apache HTTP Server（httpd-2.4.67）の複数のモジュールおよびサポートツールにおいて、パスワードやパスフレーズの処理（ハッシュ生成、検証、暗号化操作など）に必要なコンテキスト情報をカプセル化するために定義されている構造体です。

同名の構造体が複数の場所に存在し、それぞれの用途（Luaスクリプト用、ACME/MDモジュールの暗号化用、サポートツール共通のパスワード管理用）に応じて異なるフィールド構成を持っています。

なぜ重要か

パスワード処理には、メモリ管理（APRプール）、入力ソースの切り替え（プロンプト、引数、標準入力）、ハッシュアルゴリズムの選択（MD5, SHA, bcrypt, crypt等）、エラーハンドリングなど、多くの状態管理が伴います。passwd_ctx を用いてこれらの状態をコンテキストとして一元管理することで、ハッシュ生成関数（mk_password_hash や mkhash）や暗号化コールバックに対して、シンプルかつ統一されたインターフェースでデータを引き渡すことが可能になります。

関与するモジュール

  • module:modules/lua (Luaスクリプト連携におけるパスワードハッシュ生成)
  • module:modules/md (Managed Domainsにおける暗号化・パスフレーズ管理)
  • module:support (htpasswd や htdbm などのコマンドラインツール共通のパスワード処理)

主要エンティティ

  • symbol:modules/lua/lua_passwd.h#passwd_ctx: Luaモジュール用のパスワードハッシュ生成コンテキスト。
  • symbol:modules/md/md_crypt.c#passwd_ctx: MDモジュール内の暗号化操作でパスフレーズと長さを保持するシンプルなコンテキスト。
  • symbol:support/passwd_common.h#passwd_ctx: htpasswd や htdbm などのサポートツールで共有される、入力ソースやコスト値、エラー情報を統合管理するコンテキスト。

エンドツーエンドの流れ

各コンテキストにおける具体的な処理の流れは以下の通りです。

1. **support / lua モジュールにおけるハッシュ生成フロー**:

  • 呼び出し元（htpasswd.c や lua_passwd.c など）が passwd_ctx を初期化し、メモリプール（pool）や入力パスワード、アルゴリズム（alg）を設定します。
  • support では、get_password() を介して passwd_src（プロンプト、引数、標準入力など）に応じたパスワードが passwd メンバに格納されます。
  • mkhash() または mk_password_hash() が呼び出され、ctx 内のアルゴリズムやコスト（cost）に基づいてハッシュ（MD5, SHA-1, bcrypt等）を計算します。
  • 生成されたハッシュ文字列が ctx->out に書き込まれ、エラーが発生した場合は ctx->errstr に詳細が記録されます。

2. **modules/md における暗号化フロー**:

  • modules/md/md_crypt.c 内で passwd_ctx が初期化され、パスフレーズ（pass_phrase）とその長さ（pass_len）が格納されます。
  • OpenSSL のコールバック関数やパスワード入力を必要とする暗号化ルーチンに対して、この構造体がコンテキストとして引き渡されます。

なお、これら異なるモジュール間での passwd_ctx の直接的な相互作用や、システム全体における完全なエンドツーエンドの統合フローの詳細は現時点では未解明です。

重要な不変条件

  • **メモリ管理**: pool メンバを持つ構造体（modules/lua および support）において、pool は有効な apr_pool_t を指していなければなりません。
  • **パスワード長制限**: modules/lua において、入力パスワードの長さは MAX_PASSWD_LEN（256バイト）を超えてはなりません。
  • **初期化の強制**: support の共通処理では、エラー出力用の errfile が事前に初期化されている必要があります（未初期化の状態で putline 等が呼ばれるとセグメンテーション違反が発生します）。
  • **データの整合性**: modules/md の passwd_ctx では、pass_phrase が有効なポインタであり、pass_len がその実際の長さを正確に反映している必要があります。

よくある落とし穴

  • **同名構造体の混同**: modules/md/md_crypt.c の passwd_ctx と modules/lua/lua_passwd.h / support/passwd_common.h の passwd_ctx は定義が異なります。これらは相互に参照されている箇所（lua_request.c や htpasswd.c など）があるため、構造体を変更する際はそれぞれの定義の整合性に注意する必要があります。
  • **メモリ不足時の強制終了**: support モジュールでは、メモリ確保失敗時に abort_on_oom が登録されている場合、プロセスが即座に強制終了します。
  • **未サポートアルゴリズム**: 実行環境やOSの制限（CRYPT_ALGO_SUPPORTED 等のフラグ）により、指定された alg がサポートされていない場合は ERR_ALG_NOT_SUPP などのエラーが発生します。
