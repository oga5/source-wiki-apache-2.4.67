passwd_ctx

概念

passwd_ctx は、Apache HTTP Server（httpd）においてパスワードのハッシュ化、検証、暗号化処理、またはパスフレーズの保持を行うためのコンテキスト情報を格納する構造体です。

ただし、この構造体は単一のグローバルな定義ではなく、関与するモジュールやツールごとに異なる役割と定義を持っています。

  • **support および modules/lua**: メモリプール、入力パスワード、ハッシュアルゴリズム、出力バッファなどを管理し、ハッシュ生成・検証プロセス全体の状態を保持します。
  • **modules/md**: OpenSSL等の暗号化操作向けに、パスフレーズとその長さを保持するシンプルなコンテナとして機能します。

なぜ重要か

パスワードやパスフレーズの処理は、Webサーバーの認証機能（htpasswd や htdbm などのユーティリティ、Luaスクリプトによる認証、MDモジュールによる暗号化管理）において中核となるセキュリティ機能です。

passwd_ctx は、これらの処理に必要なパラメータ（入力ソース、アルゴリズム、コスト、出力先、エラー情報など）をカプセル化し、共通のハッシュ生成・検証関数（mk_password_hash や mkhash など）へ安全かつ一貫した方法で引き渡す役割を担っています。

  • **symbol:modules/md/md_crypt.c#passwd_ctx**

    • MD（Managed Domains）モジュール用のパスフレーズ保持コンテキスト。

エンドツーエンドの流れ

各コンテキストにおける処理の流れは以下の通りです。

1. サポートツール（support/passwd_common.h）における流れ

1. 呼び出し元（htpasswd.c の mkrecord や htdbm.c）が passwd_ctx を初期化し、入力ソース（passwd_src）、アルゴリズム（alg）、コスト（cost）などを設定します。 2. get_password() を呼び出し、設定された passwd_src に応じてパスワードを取得し、passwd メンバに格納します。 3. mkhash() を呼び出し、ctx 内のアルゴリズムとコストに基づいてハッシュを計算し、結果を out に書き込みます。 4. 成功時は 0 を返し、失敗時は ERR_* コードを返すと同時に errstr にエラー詳細をセットします。

2. Luaモジュール（modules/lua/lua_passwd.h）における流れ

1. 呼び出し元（lua_passwd.c や lua_request.c）が passwd_ctx を初期化し、メモリプール（pool）と入力パスワード（passwd）を設定します。 2. mk_password_hash(passwd_ctx *ctx) を呼び出します。 3. ctx->alg に基づき、apr_md5_encode、apr_sha1_base64、またはシステム標準の crypt() 等へ処理が分岐します。 4. 生成されたハッシュ文字列が ctx->out に書き込まれます。

3. MDモジュール（modules/md/md_crypt.c）における流れ

1. 呼び出し元が passwd_ctx を初期化し、pass_phrase とその長さ pass_len を設定します。 2. OpenSSL のコールバック関数やパスワード入力を必要とする暗号化ルーチンにおいて、認証情報を一括して保持するコンテナとして受け渡されます。

*※注意: これら3つの異なる passwd_ctx 定義の間で、データを直接相互変換するようなエンドツーエンドの統合フローが存在するかどうか、あるいはそれらの間の完全な相互作用のインバリアントは現時点では不明です。*

重要な不変条件

  • **メモリ管理**: support および lua の passwd_ctx において、pool は有効な apr_pool_t である必要があります。support 側では abort_on_oom() が登録されている場合、メモリ確保失敗時にプロセスが即座に終了します。
  • **パスワード長制限**: modules/lua の実装においては、パスワードの長さは MAX_PASSWD_LEN (256) を超えてはなりません。
  • **初期化要件**:
  • support 側では、処理前に errfile が初期化されている必要があります（未初期化の場合、putline 等でセグメンテーション違反が発生する可能性があります）。
  • modules/md 側では、pass_phrase は有効な文字列を指し、pass_len はその実際の長さを正確に反映している必要があります。

よくある落とし穴

  • **同名構造体の混同**: passwd_ctx という名前の構造体は、modules/lua、modules/md、support の3箇所でそれぞれ独立して（異なるフィールド構成で）定義されています。これらは互換性がないため、ソースコードの変更や移植を行う際に混同しないよう注意が必要です。
  • **初期化漏れによるクラッシュ**: support の passwd_ctx を使用する際、errfile が未初期化の状態でエラーが発生し putline 等が呼び出されると、セグメンテーション違反（クラッシュ）を引き起こします。
  • **プラットフォーム依存のアルゴリズム制限**: modules/lua や support でハッシュ生成を行う際、CRYPT_ALGO_SUPPORTED などのフラグやOSの制限により、指定したアルゴリズム（alg）が利用できず ERR_ALG_NOT_SUPP が発生することがあります。
  • **不正なポインタ/長さ**: modules/md において、pass_phrase が NULL であるか、pass_len に負の値が設定されている場合、それを利用する暗号化関数側で未定義の動作を引き起こす可能性があります。
