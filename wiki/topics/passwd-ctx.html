passwd_ctx

概念

passwd_ctx は、Apache HTTP Server（httpd-2.4.67）の複数モジュールおよびサポートツールにおいて、パスワードハッシュ生成、検証、または暗号化処理（パスフレーズ管理）のコンテキスト情報を保持するために定義されている構造体です。

同名の構造体が複数の場所に定義されており、それぞれ保持するフィールドや役割が異なりますが、パスワードやパスフレーズの処理に必要な状態（メモリプール、入力データ、アルゴリズム、出力先など）をカプセル化する共通の目的を持っています。

なぜ重要か

パスワード処理や暗号化操作は、メモリ管理（APRプール）、多様なハッシュアルゴリズム（MD5, SHA, bcrypt, crypt等）、入力ソース（プロンプト、引数、標準入力等）のハンドリングを伴います。passwd_ctx はこれらの状態を1つのコンテキストに集約することで、ハッシュ生成関数（mk_password_hash や mkhash）や暗号化コールバックへのパラメータ受け渡しを簡素化し、コードの再利用性と安全性を高める役割を果たしています。また、support ツール（htpasswd, htdbm）と各モジュール（mod_lua, mod_md）の間で、パスワード処理のロジックを共通化・連携させるための基盤となっています。

主要エンティティ

  • symbol:modules/lua/lua_passwd.h#passwd_ctx: Luaモジュール用のパスワードハッシュ生成コンテキスト。メモリプール、出力バッファ、パスワード、アルゴリズムを保持します。
  • symbol:modules/md/md_crypt.c#passwd_ctx: MDモジュール用のパスフレーズコンテキスト。パスフレーズ文字列とその長さを保持し、OpenSSL等の暗号化ルーチンに渡されます。
  • symbol:support/passwd_common.h#passwd_ctx: サポートツール共通のパスワード生成・検証コンテキスト。入力ソース（プロンプト、引数、標準入力等）、アルゴリズム、コスト、エラー文字列、出力バッファなどを統合管理します。

エンドツーエンドの流れ

各コンテキストにおける具体的な処理の流れは以下の通りです。ただし、これら3つの異なる passwd_ctx 定義がシステム全体でどのように相互作用し、完全に統合された単一のエンドツーエンドフローを形成しているかの詳細なインバリアントは完全には判明していません。

1. Luaモジュール (modules/lua)

1. 呼び出し元が passwd_ctx を初期化し、メモリプール (pool) と入力パスワード (passwd) を設定します。 2. mk_password_hash(passwd_ctx *ctx) が呼び出されます。 3. ctx->alg に基づき、apr_md5_encode、apr_sha1_base64、またはシステム標準の crypt() 等へ処理が分岐します。 4. 生成されたハッシュ文字列が ctx->out に書き込まれます。

2. MDモジュール (modules/md)

1. 暗号化操作やパスワード認証処理において、pass_phrase と pass_len を保持する passwd_ctx を初期化します。 2. OpenSSL のコールバック関数やパスワード入力を必要とする暗号化ルーチンにおいて、必要な認証情報を一括して保持するコンテナとして機能します。

3. サポートツール (support)

1. passwd_ctx を初期化し、入力ソース passwd_src を設定します。 2. get_password() を通じて、指定されたソース（標準入力、引数、プロンプト等）からパスワードを取得し passwd メンバに格納します。 3. mkhash() が ctx 内の alg および cost を参照してハッシュを計算し、out に書き込みます。 4. 成功時は 0 を返し、失敗時は ERR_* コードを返すと同時に errstr にエラー詳細をセットします。

重要な不変条件

  • **メモリ管理**: modules/lua および support の passwd_ctx において、pool は有効な apr_pool_t である必要があります。
  • **パスワード長制限**: modules/lua において、パスワードの長さは MAX_PASSWD_LEN (256) を超えてはなりません。
  • **データの整合性**: modules/md において、pass_phrase は有効な文字列を指し、pass_len はその実際の長さを正確に反映している必要があります。
  • **初期化要件**: support において、処理を実行する前に errfile が適切に初期化されている必要があります（未初期化の場合、putline 等でセグメンテーション違反が発生する可能性があります）。

よくある落とし穴

  • **名前の衝突と定義の不整合**: passwd_ctx は modules/lua/lua_passwd.h、modules/md/md_crypt.c、support/passwd_common.h の3箇所で個別に定義されています。これらは相互に参照されたり影響を与えたりするため、いずれかの構造体を変更する際は、他のモジュールでの定義や利用箇所（htpasswd.c, htdbm.c, lua_request.c など）との整合性を慎重に確認する必要があります。
  • **初期化漏れによるクラッシュ**: support の共通処理において、errfile が未初期化の状態で putline 等が呼び出されると、セグメンテーション違反（クラッシュ）が発生する危険性があります。
  • **メモリ不足時の強制終了**: support ツール群では abort_on_oom() が登録されている場合、メモリ確保失敗時にプロセスが即座に終了します。
  • **プラットフォーム依存性**: CRYPT_ALGO_SUPPORTED などの定義により、OS環境によって利用可能なハッシュアルゴリズムが制限されるため、未サポートのアルゴリズム（ERR_ALG_NOT_SUPP）への考慮が必要です。
