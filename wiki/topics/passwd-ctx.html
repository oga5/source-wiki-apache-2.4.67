passwd_ctx

概念

passwd_ctx は、Apache HTTP Server (httpd) においてパスワードのハッシュ化、検証、暗号化処理、およびパスフレーズの受け渡しに必要なコンテキスト情報を保持・管理するための構造体です。

この構造体は、使用されるコンテキスト（サポートツール、Luaスクリプト、Managed Domainsモジュール）に応じて個別に定義されており、それぞれ保持するフィールドや目的が異なります。

なぜ重要か

パスワードやパスフレーズの処理は、認証（htpasswd や htdbm などのユーティリティ、Luaスクリプトによる認証）や暗号化（MDモジュールにおける証明書・鍵管理など）のセキュリティ基盤を支えています。

passwd_ctx を用いてメモリプール、アルゴリズム、入力元、エラー状態などのコンテキストをカプセル化することで、パスワードの取得からハッシュ生成（MD5, SHA, bcrypt, crypt等）にいたる一連の処理を安全かつ統一的に扱うことができます。

関与するモジュール

提供された情報に基づき、以下のモジュールが関与しています。

  • module:modules/lua (Luaスクリプトからのパスワードハッシュ生成)
  • module:modules/md (ACME/Managed Domainsにおける暗号化・パスフレーズ処理)
  • module:support (htpasswd や htdbm などのユーティリティツール)

主要エンティティ

passwd_ctx は以下の3つのファイルで個別に定義されています。

  • **symbol:support/passwd_common.h#passwd_ctx**

    • サポートツール用。メモリプール（pool）、エラー文字列（errstr）、出力バッファ（out）、パスワード（passwd）、アルゴリズム（alg）、計算コスト（cost）、入力ソース（passwd_src）などの豊富なフィールドを持ちます。

  • **symbol:modules/lua/lua_passwd.h#passwd_ctx**

    • Luaモジュール用。pool、out、passwd、alg などのフィールドを持ち、パスワードハッシュ生成プロセスをカプセル化します。

  • **symbol:modules/md/md_crypt.c#passwd_ctx**

    • MDモジュール用。pass_phrase（パスフレーズ文字列）と pass_len（長さ）のみを保持するシンプルな構造体です。

エンドツーエンドの流れ

各コンテキストにおける処理の流れは以下の通りです。

1. サポートツールにおける流れ (support/passwd_common.h)

1. 呼び出し元（htpasswd.c の mkrecord や htdbm.c）が passwd_ctx を初期化します。 2. get_password() を呼び出し、passwd_src（プロンプト、引数、標準入力など）に応じたソースからパスワードを取得して passwd メンバに格納します。 3. mkhash() を呼び出し、ctx 内の alg（アルゴリズム）や cost（計算コスト）を参照してハッシュを計算し、結果を out に書き込みます。 4. 成功時は 0 を返し、失敗時は ERR_* コードを返し、errstr にエラー詳細をセットします。

2. Luaモジュールにおける流れ (modules/lua/lua_passwd.h)

1. 呼び出し元（lua_passwd.c や lua_request.c）が passwd_ctx を初期化し、メモリプールと入力パスワードを設定します。 2. mk_password_hash(passwd_ctx *ctx) を呼び出します。 3. ctx->alg に基づき、apr_md5_encode、apr_sha1_base64、システム標準の crypt() などの処理に分岐し、生成されたハッシュを ctx->out に書き込みます。

3. MDモジュールにおける流れ (modules/md/md_crypt.c)

1. md_crypt.c 内で passwd_ctx を初期化し、パスフレーズと長さを設定します。 2. OpenSSL のコールバック関数やパスワード入力を必要とする暗号化ルーチンにコンテキストとして渡され、認証情報を一括して受け渡します。

*注意: これら3つの異なる passwd_ctx 定義の間で、データを直接相互に受け渡すようなグローバルなエンドツーエンドの統合フローが存在するかどうか、あるいはこれらが完全に独立して動作しているかについての詳細は、提供された情報からは不明です。*

重要な不変条件

  • **メモリ管理のライフサイクル**: support および lua の passwd_ctx では、pool（apr_pool_t）が有効でなければなりません。pool のライフサイクル管理は呼び出し元の責任です。
  • **パスワード長制限**: lua_passwd.h において、パスワードの長さは MAX_PASSWD_LEN（256）を超えてはなりません。
  • **データの正確性**: md_crypt.c において、pass_phrase は有効な文字列を指し、pass_len はその実際の長さを正確に反映していなければなりません。
  • **初期化の前提**: support ツール群では、処理前に errfile が初期化されている必要があります。

よくある落とし穴

  • **名前の衝突と定義の不一致**: passwd_ctx という同名の構造体が3箇所で個別に定義されています。これらはフィールド構成や目的が異なるため、混同したり、一方の変更を他方に誤って適用したりしないよう注意が必要です。
  • **未初期化によるクラッシュ**: support ツールにおいて、errfile が未初期化の状態でエラーが発生し putline 等が呼び出されると、セグメンテーション違反（クラッシュ）を引き起こします。
  • **メモリ不足時の強制終了**: support ツールでは abort_on_oom() が登録されている場合、メモリ確保失敗時にプロセスが即座に強制終了します。
  • **アルゴリズムの未サポート**: 指定された alg が実行環境でサポートされていない場合、ERR_ALG_NOT_SUPP などのエラーが発生します。また、OSの制限（CRYPT_ALGO_SUPPORTED 等）により利用可能なアルゴリズムが制限されます。
