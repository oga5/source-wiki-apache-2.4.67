server_rec

概念

server_rec は、Apache HTTP Server において仮想ホスト（バーチャルホスト）ごとの設定情報、サーバーの状態、および実行時情報を保持するための中心的なデータ構造（struct / typedef）です。

なぜ重要か

本構造体は、サーバーのライフサイクル全体およびリクエスト処理の各フェーズにおいて、設定コンテキストとして広範に参照されます。各仮想ホストの設定（ホスト名、ポート、タイムアウト、スキームなど）を分離・管理し、モジュールがサーバー単位で保持するデータ（module_config）の格納場所としても機能するため、Apache のマルチホスト運用やモジュール連携の基盤となっています。

エンドツーエンドの流れ

サーバーの起動からリクエスト処理に至る流れは以下の通りです。

1. **初期化**: サーバー起動時に ap_init_vhost_config によって初期化され、設定ファイル（httpd.conf 等）の内容に基づいて各仮想ホストに対応する server_rec インスタンスが生成されます。 2. **テーブル構築**: 設定読み込み後、ap_fini_vhost_config によって実行時の高速検索用テーブルが構築されます。 3. **接続時の特定**: クライアントからの接続時、ap_update_vhost_given_ip により、接続の IP アドレスとポートからベースとなる server_rec が特定されます。 4. **リクエスト時の切り替え**: リクエスト処理時、ap_update_vhost_from_headers が HTTP の Host ヘッダーを解析し、名前ベースのバーチャルホストに対応する server_rec へ r->server を切り替えます。

また、MPM（マルチプロセッシングモジュール）のフック（mpm や child_status 等）やリスナー設定（ap_setup_listeners 等）の引数としても本構造体が渡され、状態の共有や通知が行われます。

重要な不変条件

  • **生存期間**: サーバーのライフサイクル全体を通じて生存するオブジェクトです。
  • **初期化の確定**: open_logs フック等の初期化フェーズにおいて、サーバー構成が確定している必要があります。
  • **テーブル構築の前提**: 実行時の高速検索を行う前に、ap_init_vhost_config および ap_fini_vhost_config が正しく実行されている必要があります。

よくある落とし穴

  • **設定の不整合**: 設定ファイルに不整合がある場合、ap_fini_vhost_config によるテーブル構築時にエラーが発生する可能性があります。
  • **Host ヘッダーの欠如**: リクエスト時に適切な Host ヘッダーが存在しない場合、デフォルトのバーチャルホスト設定が適用されます。
  • **不適切なフィールド変更**: 構造体は非常に巨大であるため、モジュール開発者は必要なフィールドのみにアクセスし、不必要な変更を避けることが推奨されます。フィールド（タイムアウト値やスキーム等）が不適切に設定されると、予期せぬ動作を引き起こす原因になります。
  • **スレッド同期**: 複数のスレッドから同時にアクセスされる場合、構成情報の変更には適切な同期が必要です。
  • **ABI 互換性（MMN）**: server_rec の構造変更（メンバーの追加など）は、include/ap_mmn.h に定義されているモジュールマジックナンバー（MMN）の更新を伴うため、モジュールの再コンパイルが必要になる場合があります。
