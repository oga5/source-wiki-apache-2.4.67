server_rec

概念

server_rec は、Apache HTTP Server において、メインサーバーおよび各バーチャルホスト（仮想ホスト）の設定情報、状態、および実行時の構成情報を保持するための中心的なデータ構造（構造体）です。

なぜ重要か

  • **バーチャルホストの識別**: サーバーのライフサイクル全体を通じて、接続やリクエストがどのバーチャルホスト設定に属するかを特定・管理するための基点となります。
  • **設定コンテキストの提供**: タイムアウト設定、サーバー名、スキーム、コンテキスト情報、およびモジュール固有の設定（module_config）などを保持し、各モジュールが仮想ホストごとの設定にアクセスするための共通インターフェースを提供します。
  • **広範な連携**: MPM（マルチプロセッシングモジュール）のフック、リスナー設定、ログ出力（mod_log_config）など、コア機能から周辺モジュールに至るまで幅広く参照されます。

エンドツーエンドの流れ

提供された情報に基づく、接続からリクエスト処理、ログ出力に至る大まかな流れは以下の通りです。

1. **初期化フェーズ**:

  • サーバー起動時に ap_init_vhost_config で初期化されます。
  • 設定ファイル（httpd.conf 等）の読み込み後、ap_fini_vhost_config によって実行時の高速検索用テーブルが構築されます。

    • 2. **接続フェーズ**:

  • クライアントからの接続時、ap_update_vhost_given_ip により、接続の IP アドレスとポートに基づいてベースとなる server_rec が特定されます。

    • 3. **リクエスト処理フェーズ**:

  • リクエスト受信時、ap_update_vhost_from_headers が HTTP の Host ヘッダーを解析し、名前ベースのバーチャルホストに対応する server_rec へ r->server を切り替えます。

    • 4. **ログ出力フェーズ**:

  • mod_log_config などのログモジュールが、server_rec に保持されたサーバー設定を参照して、ログの出力先やフォーマットを決定します。

*注意: 各フックにおける詳細な制御フローや、内部マッチングアルゴリズムの完全なエンドツーエンドの全貌は、提供された情報からは完全には判明していません。*

重要な不変条件

  • **生存期間**: サーバーのライフサイクル全体を通じて生存するオブジェクトです。
  • **構成の確定**: open_logs フックなどの初期化フェーズにおいて、サーバー構成が確定している必要があります。
  • **初期化の成否**: サーバー起動時の設定解析が正常に完了し、ap_fini_vhost_config によるテーブル構築がエラーなく行われている必要があります。

よくある落とし穴

  • **設定の不整合**: 設定ファイルに不整合がある場合、起動時のテーブル構築（ap_fini_vhost_config）でエラーが発生する可能性があります。また、タイムアウト値やスキームなどのフィールドが不適切に設定されていると、リクエスト処理や接続管理で予期せぬ動作を引き起こします。
  • **デフォルトホストへのフォールバック**: リクエスト時に適切な Host ヘッダーが存在しない場合、デフォルトのバーチャルホスト設定が適用されます。
  • **構造体変更時の互換性**: server_rec は非常に巨大な構造体であり、メンバーの追加や変更（例: server_scheme や context の追加）は、モジュールマジックナンバー（MMN、include/ap_mmn.h）の更新を伴います。モジュール開発者は不必要な変更を避け、必要なフィールドのみにアクセスすることが推奨されます。
  • **マルチスレッド環境での同期**: 複数のスレッドから同時にアクセスされる場合、構成情報の変更には適切な同期が必要です。
