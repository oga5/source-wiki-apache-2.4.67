Scoreboard and Server Status Monitoring

概念

**Scoreboard** は、Apache HTTP Server において親プロセスと子ワーカープロセス（またはスレッド）の間で、アクティブな接続、リクエスト数、ワーカーの状態などの稼働情報を共有・追跡するために使用される共有メモリ構造です。

**Server Status Monitoring** は、このスコアボードに記録されたリアルタイムの稼働情報を収集し、管理者向けに人間が読める形式（HTML やプレーンテキストなど）で提供する仕組みです。これは主に mod_status モジュールによって実現されます。

なぜ重要か

マルチプロセスやマルチスレッド（MPM）で動作する Web サーバーにおいて、各ワーカーの現在の状態（ビジー、アイドル、キープアライブなど）や処理中のリクエストをリアルタイムに把握することは、システムの監視、パフォーマンスチューニング、および障害発生時のトラブルシューティングにおいて極めて重要です。スコアボードは、これらの情報を低遅延かつ一元的に管理するための基盤を提供します。

関与するモジュール

  • module:include : スコアボードのデータ構造や操作用 API を定義するヘッダーファイルを提供します。
  • module:modules/generators : スコアボードの情報を読み取り、ステータスレポートとして生成・出力するハンドラを提供します。

主要エンティティ

  • **include/scoreboard.h** : スコアボードの全体構造、プロセスごとの情報（process_score）、ワーカーごとの詳細な状態（worker_score）などを定義するヘッダーファイル。
  • **modules/generators/mod_status.c** : スコアボードからデータを読み出し、Web 経由でサーバーの稼働ステータスを配信する mod_status モジュールの実装。

エンドツーエンドの流れ

現時点では、詳細なエンドツーエンドの流れ（親プロセスによる共有メモリの初期化、各ワーカープロセスによるスコアボードの更新タイミング、および mod_status による読み出し時の排他制御の具体的なライフサイクル）は完全には判明していません。

一般的な想定フローは以下の通りです： 1. サーバー起動時に親プロセスがスコアボード用の共有メモリ領域を確保・初期化する。 2. 各ワーカープロセスがリクエストを受信・処理・完了するたびに、include/scoreboard.h で定義された構造体を介して自身のステータスを更新する。 3. 管理者または監視ツールが mod_status で設定されたエンドポイントにアクセスすると、modules/generators/mod_status.c がスコアボードの共有メモリから最新の情報を読み取り、レスポンスとして返却する。

重要な不変条件

現時点では、スコアボードのデータ整合性を維持するための具体的な不変条件（例：マルチプロセス間での書き込み競合を防ぐためのロック要件や、状態遷移の整合性ルール）は完全には判明していません。

よくある落とし穴

現時点では、スコアボードの共有メモリへのアクセスに伴うパフォーマンス上のボトルネックや、特定の MPM 設定下における制限など、具体的な落とし穴は判明していません。

次に読む場所

  • **include/scoreboard.h** : スコアボードを構成するデータ構造（global_score、process_score、worker_score など）の定義を確認する。
  • **modules/generators/mod_status.c** : スコアボードからどのようにデータを抽出し、ステータス画面としてレンダリングしているかの実装を確認する。
