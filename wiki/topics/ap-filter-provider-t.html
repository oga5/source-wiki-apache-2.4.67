ap_filter_provider_t

概念

ap_filter_provider_t は、Apache HTTP Server（httpd）において、特定の条件に基づいて動的に適用される「スマートフィルタ」のプロバイダを管理するための構造体です。

本構造体は、ヘッダーファイル include/util_filter.h においては不完全型（不透明な構造体）として宣言されており、具体的なメンバ定義は modules/filters/mod_filter.c の内部で行われています。

なぜ重要か

静的なフィルタ適用だけでなく、リクエストのコンテキスト（コンテンツタイプや条件式の評価結果など）に応じて、実行時に適切なフィルタ実装（プロバイダ）を柔軟に選択・適用する仕組みを可能にするためです。これにより、柔軟なフィルタチェーンの動的構成が実現されます。

関与するモジュール

  • module:include
  • module:modules/filters

主要エンティティ

  • **symbol:include/util_filter.h#ap_filter_provider_t**

    • 外部公開用の不完全型宣言です。 ``c typedef struct ap_filter_provider_t ap_filter_provider_t; ``

  • **symbol:modules/filters/mod_filter.c#ap_filter_provider_t**

    • mod_filter 内部で定義される具体的な構造体です。 ``c struct ap_filter_provider_t { ap_expr_info_t *expr; /* フィルタを適用するための条件式 */ const char **types; /* 適用対象となるコンテンツタイプのリスト */ ap_filter_rec_t *frec; /* このプロバイダが実装するフィルタレコード */ ap_filter_provider_t *next; /* リスト内の次のプロバイダへのポインタ */ }; ``

エンドツーエンドの流れ

1. **設定フェーズ**: 設定関数（add_filter など）において、設定用プール（cmd->pool）から apr_palloc を用いて ap_filter_provider_t が割り当てられます。条件式、コンテンツタイプ、および対応する ap_filter_rec_t が登録され、next ポインタを介して単方向連結リストが形成されます。 2. **リクエスト処理フェーズ**: mod_filter のディスパッチャ（filter_lookup）が、登録されたプロバイダの連結リストを走査します。 3. **評価と適用**: 各プロバイダの expr の評価結果や types の一致を確認し、リクエストに最適なプロバイダを特定します。マッチしたプロバイダの frec が持つ関数ポインタを通じて、実際のフィルタ処理が呼び出されます。

*注意: 提供された情報からは、filter_lookup の具体的な探索アルゴリズムの詳細や、他のフィルタモジュールとの完全な相互作用など、エンドツーエンドのフローの全容は完全には明らかになっていません。*

重要な不変条件

  • **連結リストの終端**: リストの末尾は必ず NULL で終端されていなければなりません。
  • **優先順位**: 複数のプロバイダが条件に合致する場合、連結リストの先頭に近い（先に登録された）ものが優先して適用されます。
  • **メモリ管理**: 構造体は設定時に cmd->pool を使用して割り当てられるため、リクエスト終了後のライフサイクル管理は自動的に行われます。

よくある落とし穴

  • **不完全型によるカプセル化**: 本構造体は include/util_filter.h で不完全型として宣言されているため、mod_filter.c 以外のモジュールから直接メンバ（expr や next など）にアクセスすることはできません。構造体の拡張や変更を行う場合は、双方のファイルの整合性に注意する必要があります。
  • **条件式のスキップ**: expr が NULL の場合、条件式による評価はスキップされます。
