ap_filter_provider_t

概念

ap_filter_provider_t は、Apache HTTP Server（特に mod_filter）において、特定の条件に基づいて適用されるフィルタプロバイダー（スマートフィルタの実装）を管理するための構造体です。

ヘッダーファイル include/util_filter.h においては不完全型として宣言されており、具体的な構造体のメンバ定義は modules/filters/mod_filter.c 内で実装されています。

なぜ重要か

Apacheの「スマートフィルタ」機能を実現するための中核的なデータ構造だからです。特定のフィルタ名に対して、条件式やコンテンツタイプに基づいた複数のプロバイダー（実装）を紐付け、実行時にリクエストのコンテキストに応じて最適なフィルタを動的に選択・適用することを可能にします。

関与するモジュール

  • module:include
  • module:modules/filters

主要エンティティ

  • **symbol:include/util_filter.h#ap_filter_provider_t**

    • 不完全型としての宣言。 ``c typedef struct ap_filter_provider_t ap_filter_provider_t; ``

  • **symbol:modules/filters/mod_filter.c#ap_filter_provider_t**

    • 構造体の詳細定義。 ``c struct ap_filter_provider_t { ap_expr_info_t *expr; const char **types; ap_filter_rec_t *frec; ap_filter_provider_t *next; }; ``

  • expr: フィルタを適用するための条件式（ap_expr_info_t）。
  • types: 適用対象となるコンテンツタイプのリスト。
  • frec: このプロバイダが実装するフィルタのレコード（ap_filter_rec_t）。
  • next: リスト内の次のプロバイダへのポインタ。

エンドツーエンドの流れ

1. **設定フェーズ**: 設定関数（add_filter など）の実行時に、設定用メモリプール（cmd->pool）から apr_palloc を用いて ap_filter_provider_t のメモリが割り当てられます。 2. **プロバイダーの登録**: 条件式（expr）、コンテンツタイプ（types）、フィルタレコード（frec）が設定され、next ポインタを介して単方向連結リストが形成されます。 3. **リクエスト処理フェーズ**: mod_filter のディスパッチャ（filter_lookup）がこの連結リストを走査します。 4. **評価と適用**: 各プロバイダーの expr の評価結果や types の一致を確認することで、リクエストに最適なプロバイダーを特定します。マッチしたプロバイダーの frec が持つ関数ポインタを通じて、実際のフィルタ処理が呼び出されます。

重要な不変条件

  • **リストの終端**: 連結リストの末尾は next ポインタが NULL であることで終端されなければなりません。
  • **優先順位**: 複数のプロバイダーが条件に合致する場合、リストの先頭に近い（先に登録された）ものが優先して適用されます。
  • **メモリのライフサイクル**: 設定時に cmd->pool を使用して割り当てられるため、リクエスト終了後のライフサイクル管理は自動的に行われます。

よくある落とし穴

  • **不完全型のカプセル化**:

    • 本構造体は include/util_filter.h で不完全型として定義されているため、mod_filter.c 以外の場所から直接メンバにアクセスすることはできません。構造体を拡張・変更する場合は、双方のファイルの整合性に注意する必要があります。

  • **条件式のスキップ**:

    • expr が NULL の場合、条件式による評価はスキップされます。
