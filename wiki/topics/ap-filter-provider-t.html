ap_filter_provider_t

概念

ap_filter_provider_t は、Apache HTTP Server（特に mod_filter）において「スマートフィルタ」のプロバイダを管理するための構造体です。

ヘッダーファイル include/util_filter.h では不完全型として宣言されており、具体的な構造体のメンバ定義は modules/filters/mod_filter.c で隠蔽（カプセル化）されています。この構造体は、特定のフィルタに対して条件式やコンテンツタイプに基づいた複数の実装（プロバイダ）を紐付け、単方向連結リストを形成して管理するために使用されます。

なぜ重要か

リクエストのコンテキスト（コンテンツタイプや動的な条件式の評価結果など）に応じて、実行時に適切なフィルタを動的に選択・適用する「スマートフィルタ」の仕組みを実現するために不可欠なデータ構造だからです。

関与するモジュール

  • module:include
  • module:modules/filters

主要エンティティ

  • symbol:include/util_filter.h#ap_filter_provider_t（不完全型の宣言）
  • symbol:modules/filters/mod_filter.c#ap_filter_provider_t（構造体の実体定義）

エンドツーエンドの流れ

1. **設定フェーズ**: 設定のパース時（add_filter などの設定関数内）に、設定用プール（cmd->pool）から apr_palloc を用いて ap_filter_provider_t のメモリが割り当てられます。 2. **プロバイダの登録**: 条件式（expr）、コンテンツタイプ（types）、および対応するフィルタレコード（frec）が構造体に設定され、next ポインタを介して連結リストに登録されます。 3. **リクエスト処理時の評価**: mod_filter のディスパッチャ（filter_lookup）がこの連結リストを走査し、各プロバイダの expr（条件式）の評価や types（コンテンツタイプ）の一致を確認します。 4. **フィルタの適用**: 条件に合致したプロバイダが特定されると、その frec が指す実際のフィルタ処理が呼び出されます。

重要な不変条件

  • **リストの終端**: 連結リストの末尾の next ポインタは必ず NULL で終端されていなければなりません。
  • **優先順位**: 複数のプロバイダが条件に合致する場合、リストの先頭に近い（先に登録された）プロバイダが優先されます。
  • **メモリのライフサイクル**: 設定時のプール（cmd->pool）から割り当てられるため、設定フェーズ以降のライフサイクル管理はApacheのプール機構によって自動的に行われます。

よくある落とし穴

  • **カプセル化によるアクセス制限**:

    • 本構造体は include/util_filter.h では不完全型として定義されているため、mod_filter.c 以外のモジュールから直接メンバ（expr や next など）を参照・操作することはできません。構造体の定義を変更する場合は、双方のファイルの整合性に注意する必要があります。

  • **条件式の省略**:

    • expr が NULL の場合、条件式による評価はスキップされます。
