Dynamic Module Loading and Configuration Parsing

概念

**Dynamic Module Loading and Configuration Parsing**（動的モジュールロードと設定解析）は、Apache HTTP Server が実行時に動的共有オブジェクト（DSO）をロードし、そのモジュールが提供する設定ディレクティブを解析・統合するための仕組みです。

mod_so が提供するロード機構により、サーバーは必要に応じて新しいモジュールをプロセス空間にマッピングし、各モジュールが定義する設定ディレクティブ（command_rec）をコアの設定解析処理に登録します。これにより、コア機能を肥大化させることなく、柔軟な機能拡張が可能になります。

なぜ重要か

Apache HTTP Server のモジュール式アーキテクチャを支える基盤だからです。 実行時にモジュールを動的にロードできることで、以下のメリットが得られます。

  • **メモリ効率の向上**: 必要な機能（モジュール）のみをロードして実行できます。
  • **運用の柔軟性**: サーバーを再コンパイルすることなく、設定ファイルの変更と再起動（または再ロード）だけで機能の追加・削除が行えます。
  • **拡張性**: サードパーティ製モジュールを、コアコードを変更せずに統合できます。

関与するモジュール

本トピックには以下のモジュールが関与しています。

  • module:include: 設定ディレクティブの定義（command_rec）や、設定解析に必要な共通構造体を提供します。
  • module:modules/core: mod_so.c を含み、DSO のロード、シンボル解決、およびロードされたモジュールの管理を担当します。
  • module:server: config.c を含み、サーバー設定の生成や管理など、コアの設定解析処理を担当します。

主要エンティティ

本トピックを構成する主要なエンティティは以下の通りです。

  • **symbol:include/http_config.h#command_rec** (struct)

    • モジュールが提供する設定ディレクティブの定義（ディレクティブ名、引数の数、設定ハンドラー関数など）を保持する構造体です。

  • **symbol:modules/core/mod_so.c#load_module** (function)

    • LoadModule ディレクティブのハンドラー関数です。指定された DSO ファイルをロードし、モジュール構造体を検証した上で、グローバルなモジュールリストに追加します。

  • **symbol:modules/core/mod_so.c#ap_find_loaded_module_symbol** (function)

    • 指定されたモジュール名に対応する、ロード済みの module 構造体へのポインタをサーバー設定から検索して返します。

  • **symbol:server/config.c#create_server_config** (function)

    • サーバー設定（ap_conf_vector_t）を生成する内部関数です。

エンドツーエンドの流れ

1. **設定ファイルの解析と LoadModule の検出** 設定ファイルの解析中に LoadModule ディレクティブが検出されると、そのハンドラーである load_module が呼び出されます。 2. **DSO のロードと検証** load_module は、指定された共有オブジェクトファイルをプロセス空間にマッピング（dso_load）し、apr_dso_sym を使用してモジュール構造体（module）のシンボルを解決します。 3. **互換性の検証と登録** 取得したモジュールの magic フィールドが MODULE_MAGIC_COOKIE と一致するか検証し、問題がなければ ap_add_loaded_module を介してグローバルモジュールリストに追加します。 4. **設定の初期化と適用** ap_single_module_configure を呼び出してモジュールの設定を初期化します。サーバー設定の生成には create_server_config が関与します。 5. **ディレクティブの検索と実行** 設定解析処理（ap_walk_config など）がディレクティブを走査する際、登録された各モジュールの command_rec テーブルから該当するディレクティブが検索され、定義されたハンドラーが実行されます。

> **注意**: create_server_config の詳細な内部動作や、動的ロード処理と設定ベクター生成の完全なエンドツーエンドの統合フローの全容は、提供された情報からは未確定です。

重要な不変条件

  • **重複の禁止**: ロード対象のモジュールが、既にロードされているモジュールや、静的にリンクされているモジュールと名前が重複していてはなりません（load_module 内でチェックされます）。
  • **マジッククッキーの一致**: ロードされる DSO 内のモジュール構造体は、サーバー本体と互換性のある MODULE_MAGIC_COOKIE を保持している必要があります。
  • **コンテキストの有効性**: ap_find_loaded_module_symbol を呼び出す際、引数 server_rec *s は有効なサーバー設定を指しており、so_server_conf および loaded_modules が初期化済みでなければなりません。

よくある落とし穴

  • **静的モジュールとの衝突**

    • すでに静的にコンパイルされて組み込まれているモジュールと同じ名前のモジュールを LoadModule で動的にロードしようとすると、名前の衝突によりエラーが発生し、ロードに失敗します。

  • **バージョン不整合（Magic Cookie エラー）**

    • サーバー本体と異なるバージョン（または異なるビルドオプション）でコンパイルされた DSO をロードしようとすると、magic 番号の不一致により互換性エラーとなり、ロードが拒否されます。

  • **NULL 引数による検索スキップ**

    • ap_find_loaded_module_symbol に渡す modname が NULL の場合、比較処理は行われずに NULL が返されます。呼び出し側での引数の妥当性検証が不足していると、予期せぬ動作を招く可能性があります。

  • **未解析の初期化処理**

    • create_server_config の前提条件や失敗モードは現時点では未解析であるため、この関数が関与する設定生成フェーズのカスタマイズやデバッグの際は注意が必要です。
