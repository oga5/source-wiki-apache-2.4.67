dav_propdb

概念

dav_propdb は、Apache HTTP Server の mod_dav モジュールにおいて、WebDAV プロパティ（ライブプロパティおよびデッドプロパティ）の操作コンテキストを保持・管理するための構造体、およびそれに関連するサブシステムです。

なぜ重要か

WebDAV プロトコルにおける PROPFIND や PROPPATCH などのメソッド処理において、プロパティの解決、トランザクション処理（検証、実行、コミット/ロールバック）、名前空間の管理、およびデータベースへのアクセスを統合的に行うための中核的な役割を担います。

エンドツーエンドの流れ

提供されたコンテキストからは、完全なエンドツーエンドの制御フローは完全には明らかになっていませんが、以下の部分的な流れが示されています。

1. **初期化**: dav_popen_propdb により、リクエストレコード (request_rec) や対象リソース (dav_resource) を設定して dav_propdb コンテキストが作成されます。 2. **データベースのオープン**: deferred フラグが設定されている場合、データベースのオープンは実際にプロパティ操作が必要になるまで遅延されます。データベースのオープン/クローズ処理には dav_propdb_open や dav_propdb_close が関与します。 3. **プロパティの解決と操作**:

  • dav_find_liveprop を通じてライブプロパティプロバイダーを特定し、見つからない場合はデッドプロパティとして処理します。
  • PROPPATCH 操作時には、validate -> exec -> commit/rollback の順序でトランザクション処理が行われます。

    • 4. **XML 構築**: dav_get_props 等により、名前空間（ns_xlate, mapping）を管理しながら XML 形式のプロパティリストを生成します。 5. **クローズ/解放**: dav_close_propdb を呼び出すことで、リソースの解放および内部で生成されたサブリクエスト (subreq) の破棄を行います。

重要な不変条件

  • **メモリ管理**: dav_propdb はリクエストプール (r->pool) または指定されたプールを使用してメモリ管理を行います。
  • **リソース解放の保証**: サブリクエスト (subreq) を使用した場合は、リソースのリークを防ぐため、必ず dav_close_propdb を呼び出して適切に破棄しなければなりません。
  • **遅延オープン**: deferred フラグが有効な場合、実際にプロパティ操作が必要になるまでデータベースのオープンは行われません。

よくある落とし穴

  • **サブリクエストの破棄漏れ**: subreq を使用した際に dav_close_propdb による適切な破棄を怠ると、リソースリークの原因となります。
  • **名前空間プレフィックスへの依存**: プロパティ値内の名前空間は ns<index> 形式に変換されるため、クライアント側で特定のプレフィックスに依存する実装を行うと正常に動作しない可能性があります。
  • **書き込み制限**: DAV_DISABLE_WRITABLE_PROPS が有効な場合、一部のコアプロパティへの書き込みが制限されます。
  • **エラーハンドリングの不完全性**: PROPPATCH のエラーハンドリングにおいて、複雑なケースではエラー伝播が不完全になる可能性があります。
