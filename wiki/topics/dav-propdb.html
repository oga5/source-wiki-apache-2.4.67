dav_propdb

概念

dav_propdb は、Apache HTTP Server の WebDAV モジュール（mod_dav）において、WebDAV プロパティ（ライブプロパティおよびデッドプロパティ）の操作コンテキストを保持するための構造体、およびそれに関連する管理機構です。プロパティデータベースへのアクセス、トランザクション処理、名前空間の解決などを統合的に管理します。

なぜ重要か

WebDAV プロトコルにおける PROPFIND や PROPPATCH などのプロパティ操作要求を効率的かつ安全に処理するための中核的なデータ構造だからです。データベース接続の遅延オープン（deferred）や、トランザクションのコミット/ロールバック、名前空間の変換、サブリクエストの管理などを一元化することで、プロパティ管理の複雑さを隠蔽します。

エンドツーエンドの流れ

提供されたコンテキストからは、dav_propdb_open および dav_propdb_close の具体的な内部実装や、これらが props.c の dav_propdb とどのように直接連携して呼び出されるかの完全なエンドツーエンドの流れは完全には明らかになっていません。

ただし、一般的な処理の流れとして以下が示されています：

1. **初期化**: mod_dav.c 内の各メソッドハンドラ（PROPFIND, PROPPATCH 等）から、dav_popen_propdb 等を通じて dav_propdb コンテキストが初期化されます。 2. **データベース接続**: deferred フラグが有効な場合は遅延され、実際にプロパティ操作が必要になった段階でデータベース（dav_db）がオープンされます。 3. **プロパティ処理**:

  • **解決と取得**: dav_find_liveprop を用いてライブプロパティを解決し、見つからない場合はデッドプロパティとして処理します。dav_get_props 等により、名前空間を管理しながら XML 形式のプロパティリストを構築します。
  • **トランザクション**: PROPPATCH 操作時には、validate -> exec -> commit/rollback の順序で処理され、失敗時には dav_rollback_item を用いて状態を復旧します。

    • 4. **クローズ**: 処理終了時に dav_close_propdb を呼び出し、リソースの解放やサブリクエスト（subreq）の破棄を行います。

重要な不変条件

  • **メモリ管理**: dav_propdb は、リクエストプール（r->pool）または指定されたプール（p）を使用してメモリ管理を行います。
  • **遅延オープン**: deferred フラグが設定されている場合、実際にプロパティ操作が必要になるまでデータベースのオープンは行われません。
  • **リソースの確実な解放**: 内部でサブリクエスト（subreq）が生成された場合、リークを防ぐために必ず dav_close_propdb を呼び出して適切に破棄しなければなりません。

よくある落とし穴

  • **サブリクエストの破棄漏れ**: subreq を使用した後に dav_close_propdb によるクローズ処理を怠ると、リソースリークが発生する可能性があります。
  • **書き込み制限**: DAV_DISABLE_WRITABLE_PROPS が有効な場合、一部のコアプロパティへの書き込みが制限されるため、書き込み操作が失敗することがあります。
  • **名前空間の依存性**: プロパティ値内の名前空間は ns<index> 形式に変換されるため、クライアント側が特定のプレフィックスに依存した実装になっていると、正しく解析できない落とし穴があります。
  • **エラーハンドリングの複雑さ**: PROPPATCH のエラーハンドリングにおいて、複雑なケースではエラーの伝播が不完全になる可能性があります。
