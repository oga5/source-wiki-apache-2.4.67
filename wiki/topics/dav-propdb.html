dav_propdb

概念

dav_propdb は、Apache HTTP Server の WebDAV モジュール（mod_dav）において、WebDAV プロパティ（ライブプロパティおよびデッドプロパティ）のデータベースへのアクセス、操作、およびトランザクション処理のコンテキストを管理するための構造体、およびそれに関連する操作群です。

なぜ重要か

WebDAV プロトコルでは、リソースに関連付けられたメタデータ（プロパティ）をクライアントが取得（PROPFIND）または更新（PROPPATCH）できます。dav_propdb は、これらのプロパティ操作を抽象化し、バックエンドのデータベース（DBM など）との仲介役として機能します。これにより、メモリ管理、名前空間の解決、トランザクション（validate/exec/commit/rollback）の一貫性を保ちながらプロパティを安全に管理できます。

関与するモジュール

  • module:modules/dav/main: WebDAV のコアロジックおよびプロパティ管理の共通フレームワークを提供します。
  • module:modules/dav/fs: ファイルシステムバックエンドにおける具体的なプロパティデータベース（DBM）のオープン・クローズ処理などを提供します。

主要エンティティ

  • **struct dav_propdb** (modules/dav/main/props.c / modules/dav/main/mod_dav.h):

    • プロパティ操作のコンテキストを保持する中心的な構造体。リクエストレコード (request_rec)、対象リソース (dav_resource)、データベース接続 (dav_db)、名前空間マップ、ロックデータベースなどを保持します。

  • **dav_propdb_open** (modules/dav/fs/dbm.c):

    • 指定されたリソースに対応するプロパティデータベースを開き、dav_db インスタンスを初期化する関数。

  • **dav_propdb_close** (modules/dav/fs/dbm.c):

    • オープンされたプロパティデータベースを閉じ、データの保存やリソースのクリーンアップを行う関数。

エンドツーエンドの流れ

1. **初期化とオープン**: mod_dav のリクエスト処理（PROPFIND や PROPPATCH など）において、dav_popen_propdb 等を介して dav_propdb コンテキストが初期化されます。必要に応じて、バックエンド（modules/dav/fs）の dav_propdb_open が呼び出され、実際のデータベース（dav_db）がオープンされます（deferred フラグが有効な場合は、実際の操作時まで遅延されます）。 2. **プロパティ操作**: props.c 内のロジックにより、ライブプロパティの検索（dav_find_liveprop）やデッドプロパティの処理が行われます。PROPPATCH の場合は、validate -> exec -> commit/rollback のトランザクション処理が実行されます。 3. **クローズと解放**: 操作完了後、dav_close_propdb が呼び出され、内部で生成されたサブリクエストの破棄やメモリの解放が行われます。また、バックエンド側では dav_propdb_close が呼び出され、データベースのクローズとデータの永続化が行われます。

*注意: 提供されたシンボル情報からは、dav_propdb_open や dav_propdb_close が dav_propdb 構造体や dav_hooks_db を介してどのように直接バインドされ呼び出されるか、詳細な呼び出し経路や完全なエンドツーエンドの制御フローは完全には明らかになっていません。*

重要な不変条件

  • **明示的なクローズ**: dav_propdb を使用した操作の終了時には、必ず dav_close_propdb を呼び出して、内部リソース（特にサブリクエスト subreq）を確実に破棄しなければなりません。
  • **遅延オープンの整合性**: deferred フラグが設定されている場合、データベースは実際にプロパティ操作が必要になるまでオープンされませんが、操作実行時には正しくオープンされ、一貫したコンテキストが維持される必要があります。
  • **メモリ管理のライフサイクル**: dav_propdb は Apache の apr_pool_t（通常はリクエストプール r->pool）に依存してメモリ管理を行うため、関連するリクエストのライフサイクルを超えて参照してはなりません。

よくある落とし穴

  • **サブリクエストのリーク**: dav_propdb 内部で生成されたサブリクエスト (subreq) は、dav_close_propdb を呼び出さないと適切に破棄されず、メモリリークやリソースリークの原因になります。
  • **名前空間プレフィックスへの依存**: プロパティ値内の名前空間は ns<index> 形式に変換されるため、クライアント側で特定のプレフィックスに依存した実装を行うと、名前空間の解決に失敗する可能性があります。
  • **書き込み制限の無視**: DAV_DISABLE_WRITABLE_PROPS が有効な場合、一部のコアプロパティへの書き込みが制限されるため、書き込み操作が予期せず失敗することがあります。

次に読む場所

  • modules/dav/main/mod_dav.h: dav_propdb の不透明ポインタ定義や関連するフック構造体の定義を確認できます。
  • modules/dav/main/props.c: struct dav_propdb の詳細なフィールド構成や、プロパティ管理・トランザクション処理の具体的な実装を確認できます。
  • modules/dav/fs/dbm.c: DBM を使用したプロパティデータベースのオープン（dav_propdb_open）およびクローズ（dav_propdb_close）の具体的なバックエンド実装を確認できます。
