dav_propdb

概念

dav_propdb は、Apache HTTP Server の WebDAV モジュール（mod_dav）において、WebDAV プロパティ（ライブプロパティおよびデッドプロパティ）を操作・管理するためのコンテキスト情報を保持する構造体、およびそれに関連するサブシステムです。

なぜ重要か

WebDAV プロトコルでは、リソースに付随するメタデータ（プロパティ）をクライアントから取得・設定（PROPFIND や PROPPATCH）する必要があります。dav_propdb は、リクエスト情報、対象リソース、データベース接続、名前空間マッピング、ロック情報などを一元管理し、プロパティ操作のトランザクション処理や XML 形式への変換を安全かつ効率的に行うための中心的な役割を果たします。また、データベースの遅延オープン（deferred）などの最適化機構も提供します。

エンドツーエンドの流れ

提供された情報から推論される大まかな流れは以下の通りですが、詳細なエンドツーエンドのフロー（特に modules/dav/fs/dbm.c 内の具体的なデータベース操作との連携など）は完全には明らかになっていません。

1. **初期化**: mod_dav.c 内の各メソッドハンドラ（PROPFIND, PROPPATCH 等）から、dav_popen_propdb 等を通じて dav_propdb コンテキストが作成されます。 2. **データベースのオープン**: deferred フラグが有効な場合は遅延され、実際にプロパティ操作が必要になった段階でデータベース（dav_db）がオープンされます（この際、dav_propdb_open が関与すると推測されます）。 3. **プロパティの解決と操作**:

  • dav_find_liveprop を用いてライブプロパティプロバイダーを特定し、存在しない場合はデッドプロパティとして処理します。
  • PROPPATCH 時には、validate -> exec -> commit/rollback の順でトランザクション処理が行われます。

    • 4. **XML 構築**: dav_get_props 等により、名前空間マッピング（ns_xlate, mapping）を解決しながら XML 形式のプロパティリストを生成します。 5. **クローズと解放**: 処理終了時に dav_close_propdb（および dav_propdb_close）を呼び出し、データベースのクローズ、リソースの解放、および生成されたサブリクエスト（subreq）の破棄を行います。

重要な不変条件

  • **メモリ管理**: dav_propdb は、リクエストプール（r->pool）または指定されたプールを使用してメモリ管理を行います。
  • **遅延オープン**: deferred フラグが設定されている場合、実際にプロパティ操作が必要になるまでデータベース接続の確立は遅延されなければなりません。
  • **リソースの確実な解放**: 内部でサブリクエスト（subreq）が生成された場合、メモリリークや予期せぬ挙動を防ぐため、必ず dav_close_propdb を呼び出して適切に破棄する必要があります。

よくある落とし穴

  • **サブリクエストの破棄漏れ**: dav_propdb 内部で生成されたサブリクエスト（subreq）は、dav_close_propdb を呼び出さないと適切に破棄されず、リソースリークの原因となります。
  • **名前空間プレフィックスの依存**: プロパティ値内の名前空間は ns<index> 形式に変換されるため、クライアント側で特定のプレフィックス（例: D: など）に固定的に依存する実装を行っていると、相互運用性の問題が発生する可能性があります。
  • **書き込み制限**: DAV_DISABLE_WRITABLE_PROPS が有効な場合、一部のコアプロパティへの書き込みが制限されるため、書き込み操作が失敗する境界条件に注意が必要です。
  • **エラーハンドリングの複雑さ**: PROPPATCH のエラーハンドリングにおいて、複雑なケースではエラーの伝播が不完全になる可能性があります。
