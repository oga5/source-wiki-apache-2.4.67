Embedded Scripting with mod_lua

概念

mod_lua を使用した組み込みスクリプト（Embedded Scripting）は、Apache HTTP Server のリクエスト処理サイクルに Lua スクリプトをフックするための仕組みです。これにより、C 言語で記述された Apache コアのデータ構造と Lua 実行環境（VM）を相互にブリッジし、柔軟なリクエスト処理やカスタムハンドラの記述が可能になります。

なぜ重要か

Apache のコア構造体である request_rec に Lua から安全にアクセスできるようにすることで、C 言語によるモジュール開発を行うことなく、動的なリクエスト制御やヘッダー操作、コンテンツ生成を迅速に実装できます。また、Lua 仮想マシン（lua_State）のライフサイクル管理やキャッシュポリシーを適切に制御することで、パフォーマンスとリソース消費のバランスを最適化できます。

関与するモジュール

  • module:modules/lua

主要エンティティ

  • **symbol:modules/lua/lua_request.c#request_rec**: Apache のリクエスト処理の核となるデータ構造。Lua 側からは Apache2.Request userdata を介して参照され、req_dispatch メタメソッドを通じて C 言語のフィールドへの読み書きや操作がブリッジされます。
  • **symbol:modules/lua/lua_vmprep.c#ap_lua_get_lua_state**: 指定されたスコープ（サーバー、リクエスト、または1回限り）やキャッシュポリシーに基づき、Lua 仮想マシン（lua_State）を取得または新規生成する関数。
  • **symbol:modules/lua/mod_lua.c#lua_handler**: リクエストを処理するためのハンドラ関数。ap_lua_get_lua_state を呼び出して VM を取得し、Lua スクリプトを実行します。

エンドツーエンドの流れ

提供された情報からは、Apache コアと mod_lua 間の完全なエンドツーエンドのライフサイクルや詳細な呼び出しフローは完全には明らかになっていませんが、判明している範囲の流れは以下の通りです。

1. **ハンドラの起動**: Apache がリクエストを受信すると、lua_handler が起動されます。 2. **VM の取得**: lua_handler は ap_lua_get_lua_state を呼び出し、現在のリクエストコンテキスト（request_rec）と仕様（ap_lua_vm_spec）に基づいて lua_State を取得します。 3. **キャッシュ検証とロード**: ap_lua_get_lua_state 内部では、スコープに応じてキャッシュから既存の VM を検索します。キャッシュミス時やスクリプトの更新検知時（AP_LUA_CACHE_STAT 等による）は、新規に VM を構築し、luaL_loadfile および lua_pcall を用いてスクリプトをロード・実行します。 4. **リクエスト処理**: Lua スクリプト内から request_rec のフィールドへのアクセスが発生すると、req_dispatch メタメソッドを介して C 言語の対応する関数が実行され、結果が Lua スタックへ返されます。 5. **VM の解放**: 処理完了後、lua_handler は ap_lua_release_state を呼び出して VM を解放またはキャッシュに戻します。

重要な不変条件

  • **メモリ管理の不変条件**: mod_lua におけるすべてのメモリ割り当ては、request_rec に紐付くプール（r->pool）を使用します。これにより、リクエスト終了時に自動的にメモリが解放されます。Lua 側でのオブジェクト寿命管理と混同してはなりません。
  • **VM のライフサイクル紐付け**: 生成された VM のライフサイクルは lifecycle_pool に紐付けられ、プール破棄時に自動的に lua_close が実行されるようクリーンアップハンドラが登録されます。
  • **排他制御**: spec->scope が AP_LUA_SCOPE_SERVER の場合、内部で ap_lua_mutex を使用した排他制御が行われます。
  • **リクエスト検証**: Lua スタック上の userdata が有効な request_rec であることは、ap_lua_check_request_rec によって検証される必要があります。

よくある落とし穴

  • **書き込み制限**: req_newindex を介した request_rec のフィールド設定には制限があり、すべてのフィールドが書き込み可能ではありません。
  • **正規表現の制限**: 正規表現のキャプチャ数には MODLUA_MAX_REG_MATCH による制限が存在します。
  • **キャッシュのオーバーヘッド**: AP_LUA_CACHE_STAT を使用している場合、リクエストごとに apr_stat によるファイル更新チェックが発生するため、高負荷環境ではパフォーマンスのボトルネックになる可能性があります。
  • **スレッド競合と VM 取得失敗**: AP_LUA_SCOPE_SERVER においてスレッド競合が発生した場合、apr_reslist_acquire の失敗により ap_lua_get_lua_state が NULL を返す可能性があります。
