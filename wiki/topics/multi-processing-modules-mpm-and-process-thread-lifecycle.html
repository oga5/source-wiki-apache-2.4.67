Multi-Processing Modules (MPM) and Process/Thread Lifecycle

概念

Apache HTTP Serverにおける **Multi-Processing Modules (MPM)** は、ネットワークポートのバインド、クライアントからのリクエストの受け入れ、およびそれらを処理するための子プロセスやスレッドへの割り当て（ディスパッチ）を管理するアーキテクチャです。

httpdは、異なるオペレーティングシステムやパフォーマンス要件に対応するため、以下のような異なる並行処理モデル（MPM）を提供しています。

  • **Prefork (prefork.c)**: スレッドを使用せず、リクエストごとに独立した子プロセスをフォークして処理するモデル。
  • **Worker (worker.c)**: マルチプロセスとマルチスレッドのハイブリッドモデル。各子プロセスが複数のワーカースレッドと1つのリスナースレッドを持ちます。
  • **Event (event.c)**: Workerモデルをベースに、キープアライブ接続などの非同期イベント処理を専用のリスナースレッドで効率的に処理するモデル。

なぜ重要か

MPMは、Webサーバーのスケーラビリティ、メモリ使用量、および安定性に直結します。

  • **リソースの最適化**: プラットフォーム（Unix系、Windowsなど）やハードウェアリソースに応じて、最適なプロセス・スレッドモデルを選択できます。
  • **スレッドセーフティの担保**: スレッドセーフでないモジュール（例: 一部のPHP拡張など）を使用する場合、スレッドを使用しないPrefork MPMを選択することで、プロセスメモリ空間を分離し安全に実行できます。
  • **ライフサイクル管理**: 負荷の変動に応じて動的に子プロセスやスレッドを生成・消滅させ、アイドル状態のプロセス数を適切に維持します。

関与するモジュール

  • module:server/mpm (MPM共通の基盤)
  • module:server/mpm/event (Event MPMの実装)
  • module:server/mpm/prefork (Prefork MPMの実装)
  • module:server/mpm/worker (Worker MPMの実装)

主要エンティティ

  • **include/ap_mpm.h**: MPMの共通インターフェース。サーバーの他の部分が現在のMPMの動作特性（スレッド化されているか、動的プロセス生成を行うかなど）を問い合わせるための ap_mpm_query などのAPIを定義します。
  • **server/mpm/prefork/prefork.c**: Prefork MPMのメインループ、子プロセスのフォーク、およびシグナルハンドリングを実装します。
  • **server/mpm/worker/worker.c**: Worker MPMにおける、子プロセスの管理、リスナースレッドおよびワーカースレッドプールの生成とライフサイクルを実装します。
  • **server/mpm/event/event.c**: Event MPMにおける、非同期接続の監視（epollやkqueueなどのイベント通知機構の利用）と、ワーカースレッドへのタスク割り当てを実装します。

エンドツーエンドの流れ

1. **初期化と起動**: 親プロセスが起動し、設定ファイルを読み込み、リスニングソケットをバインドします。 2. **プロセスの生成**: 各MPM（prefork.c, worker.c, event.c）の親プロセスループが、設定された初期プロセス数（StartServers）に基づいて子プロセスをフォークします。 3. **接続の受け入れと処理**:

  • **Prefork**: 各子プロセスが直接 accept() を呼び出し、接続をシングルスレッドで同期的に処理します。
  • **Worker**: 子プロセス内のリスナースレッドが accept() を行い、接続をキューを介してワーカースレッドに渡します。
  • **Event**: リスナースレッドが非同期イベントを監視し、データが到着したアクティブな接続のみをワーカースレッドにディスパッチします。

    • 4. **プロセスの維持と終了**: 親プロセスは定期的に子プロセスの状態（スコアボード）を監視し、アイドルプロセス数が設定範囲内に収まるよう動的に調整します。シャットダウンや再起動のシグナルを受信すると、クリーンな終了シーケンスを実行します。

*注意: 各MPMにおけるシグナル処理の伝播や、スコアボードの同期に関する具体的な低レベルの関数呼び出しフローは、提供されたソースコードの範囲内では詳細が未検証です。*

重要な不変条件

  • **最大/最小アイドルスレッド・プロセス数の維持**: 各MPMは、急激なアクセス増加に対応しつつリソースを浪費しないよう、設定されたアイドル制限（MinSpareThreads/MaxSpareThreads など）を維持しなければなりません。
  • **MaxRequestWorkers (MaxClients) の遵守**: 同時にアクティブになるプロセスまたはスレッドの総数は、設定された上限を超えてはなりません。

*注意: スコアボードの排他制御や、プロセス生成・消滅時におけるスレッドセーフティに関する具体的な不変条件の検証は、提供されたコンテキストからは未確定です。*

よくある落とし穴

  • **スレッドセーフでないモジュールの混入**: WorkerやEventなどのマルチスレッドMPMを使用している環境で、スレッドセーフでないサードパーティ製モジュールをロードすると、メモリ破損や予期しないクラッシュを引き起こす原因になります。
  • **リソース制限（FD制限など）の枯渇**: 大規模なスレッドやプロセスを生成する設定（ThreadsPerChild や MaxRequestWorkers の過大な設定）に対して、OSのファイル記述子制限（ulimit -n）やプロセス数制限が不十分な場合、プロセスのフォークやスレッドの生成に失敗します。
  • **ゾンビプロセスの発生**: 親プロセスと子プロセスの間のシグナルハンドリング（SIGCHLD など）が正しく処理されない場合、終了した子プロセスがゾンビプロセスとしてシステムに残る可能性があります。

次に読む場所

  • **include/ap_mpm.h**: MPM共通のクエリ機能や、MPMが提供すべき基本機能の定義を理解するために参照してください。
  • **server/mpm/prefork/prefork.c**: 最もシンプルなプロセスベースのライフサイクル管理とシグナル処理の基本パターンを学ぶのに適しています。
  • **server/mpm/worker/worker.c**: スレッドプールとリスナースレッドの協調動作、およびキューイングモデルを理解するために参照してください。
  • **server/mpm/event/event.c**: 高度な非同期イベント駆動型接続処理の実装詳細を追うために参照してください。
