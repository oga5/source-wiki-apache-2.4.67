server_limit

概念

server_limit は、Apache HTTP Server（httpd-2.4.67）において、同時に起動できるマルチプロセッシングモジュール（MPM）の最大プロセス数（サーバー数）を制御・制限するための設定値および変数です。この値は、共有メモリ上のスコアボードのサイズ決定や、各MPMにおけるプロセス管理の境界値として使用されます。

なぜ重要か

server_limit は、サーバーのリソース消費を制限し、過度なプロセス生成によるシステムリソースの枯渇を防ぐために極めて重要です。また、プロセスごとの状態を記録するスコアボード（共有メモリ）の領域サイズを決定する基準となるため、メモリ割り当ての安全性と効率性を担保する役割も担っています。

エンドツーエンドの流れ

1. **設定の読み込みと初期化**: サーバー起動時、設定ファイル（httpd.confなど）の ServerLimit ディレクティブに基づき、有効化されているMPM（prefork, worker, event のいずれか）の set_server_limit 関数が呼び出され、各MPM固有の server_limit 変数に値が格納されます。 2. **スコアボードの構築**: server/scoreboard.c は、設定された server_limit の値を参照して、必要な共有メモリ（スコアボード）のサイズを決定し、初期化を行います。 3. **ステータス監視**: mod_status（mod_status.c）がリクエストを処理する際、server_limit を上限としてスコアボード内の各プロセススロットを走査し、稼働中のプロセス情報を収集・表示します。

*注意: 設定ディレクティブのパースから各MPMへの具体的な値の伝播、および共有メモリ確保の厳密な実行順序の詳細は、提供されたシンボル情報からは完全には特定されていません。*

重要な不変条件

  • **スコアボードサイズとの整合性**: スコアボードに割り当てられたプロセススロットの最大数は、server_limit と一致している必要があります。これを超えてプロセススロットにアクセスすることはできません。
  • **MPMごとの独立性**: 各MPM（prefork, worker, event）はそれぞれ独自に server_limit および set_server_limit を定義しており、有効なMPMの設定値がシステム全体に適用されます。

*注意: コンパイル時のハードリミット（MAX_SERVER_LIMITなど）との比較や、実行時の動的な不変条件の詳細は、提供されたシンボル情報からは直接確認できていません。*

よくある落とし穴

  • **動的変更の制限**: ServerLimit（server_limit）はスコアボードの共有メモリサイズを決定するため、サーバーの稼働中に動的に変更することはできません。設定を反映させるには、サーバーの完全な再起動（graceful restartではなく、stop/start）が必要になる場合があります。
  • **MPM間の設定の不一致**: 使用しているMPM（例: event から prefork への変更など）に応じて、適切な ServerLimit の挙動やデフォルト値が異なるため、MPMの切り替え時には注意が必要です。
