Multi-Processing Modules (MPM) Execution Model

概念

Multi-Processing Modules (MPM) 実行モデルは、Apache HTTPDがクライアントからのリクエストを並行処理するためのプロセスおよびスレッドの管理モデルです。 HTTPDは、異なる並行処理アーキテクチャ（プロセスベースのPrefork、スレッドベースのWorker、非同期イベント駆動のEventなど）をMPMとして抽象化し、プロセスの生成、スレッドの初期化、アイドル状態の監視、およびクリーンな終了処理を制御します。

なぜ重要か

MPM実行モデルは、Webサーバーのパフォーマンス、スケーラビリティ、およびリソース消費を最適化するために不可欠です。 サーバー全体のプロセス数やスレッド数の上限を適切に制限（set_server_limit, set_thread_limit）しつつ、稼働中のアイドルプロセス/スレッドを動的に維持管理（perform_idle_server_maintenance）することで、急激なトラフィック変動への追従とシステムリソースの浪費防止を両立させます。

関与するモジュール

本モデルには、以下のモジュールが関与しています。

  • module:server/mpm
  • module:server/mpm/event
  • module:server/mpm/prefork
  • module:server/mpm/worker

主要エンティティ

本モデルを構成する主要なエンティティ（server/mpm/event/event.c 内のシンボルを中心とする）は以下の通りです。

  • **child_main**: 子プロセスが起動した際に実行されるメイン関数。スレッドの作成（ap_thread_create）やシグナル処理のセットアップ、スコアボードの再オープンなどを行い、プロセスのライフサイクルを管理します。
  • **clean_child_exit**: 子プロセスを安全かつクリーンに終了させるための関数。終了時に event_note_child_stopped を呼び出して状態を通知します。
  • **perform_idle_server_maintenance**: メインループ等から定期的に呼び出され、アイドル状態の子プロセスやスレッドの数を監視し、必要に応じて新しい子プロセスを生成（make_child）したり、不要なプロセスに終了シグナル（ap_mpm_podx_signal）を送ったりしてサーバーの稼働規模を維持します。
  • **set_server_limit**: 起動構成において、生成可能な最大プロセス数（サーバーリミット）を設定するためのコマンドハンドラ。
  • **set_thread_limit**: 起動構成において、プロセスあたりに生成可能な最大スレッド数（スレッドリミット）を設定するためのコマンドハンドラ。

エンドツーエンドの流れ

現時点では、初期化からリクエスト処理、シャットダウンに至る完全なエンドツーエンドの実行フローの全容は完全には解析されていません。

ただし、提供されたシンボル情報から以下の部分的な流れが確認できます。 1. **設定フェーズ**: サーバー起動時に set_server_limit や set_thread_limit を通じて、プロセスおよびスレッドの最大制限値が設定されます。 2. **維持管理ループ**: 親プロセスのメインループ（server_main_loop や prefork_run など）から perform_idle_server_maintenance が定期的に呼び出され、負荷状況に応じて make_child により新しい子プロセスが生成されます。 3. **子プロセスの実行**: 生成された子プロセスは child_main を実行し、内部でワーカースレッドの生成や初期化を行います。 4. **終了処理**: 子プロセスが終了する際は、clean_child_exit を経由してクリーンに終了処理が行われます。

重要な不変条件

現時点では、プロセス間通信やスレッドセーフティに関する詳細な不変条件（インバリアント）は完全には解析されていません。 ただし、set_server_limit や set_thread_limit によって設定される上限値が、実行時におけるプロセス生成（make_child）やスレッド生成の境界条件として厳格に維持される必要があります。

よくある落とし穴

現時点では、具体的な実装コードにおける競合状態やリソースリークなどの「よくある落とし穴」の詳細は完全には解析されていません。 しかし、clean_child_exit が適切に呼び出されない場合、子プロセスがスコアボードやシステムリソースを保持したままゾンビ化する、あるいは異常終了時に親プロセスが状態を検知できなくなるリスクが想定されます。
