ssl_var_lookup

概念

ssl_var_lookup は、Apache HTTP Server において SSL/TLS 接続、リクエスト、またはサーバー設定に関連する環境変数や各種属性（SSLクライアント証明書情報、プロトコルバージョン、暗号スイート、時刻情報など）を動的に解決し、文字列として取得するための関数およびその仕組みです。

なぜ重要か

SSL/TLS 接続のメタデータ（暗号化の強度、クライアント証明書の詳細、プロトコルなど）は、アクセス制御、ログ記録、CGI/SSI への環境変数引き渡し、Lua スクリプトによる制御など、Web サーバーの多様な機能で必要とされます。ssl_var_lookup は、これらの情報を一元的に検索・解決する共通インターフェースを提供します。また、古い API との互換性を維持するための後方互換レイヤー（server/ssl.c）も存在し、システムの安定性と拡張性を両立させるために重要です。

関与するモジュール

提供された情報に基づき、以下のモジュールが関与しています。

  • module:modules/ssl: 実際の SSL 変数解決ロジック（ssl_engine_vars.c）を実装するコアモジュール。
  • module:server: 古い API との互換性を維持するためのブリッジ/ラッパー（ssl.c）を提供するコアサーバー。
  • module:modules/arch/netware: NetWare プラットフォーム向けの SSL 実装（mod_nw_ssl.c）において、本関数を呼び出すか、またはオプション関数として参照します。

主要エンティティ

  • **symbol:modules/ssl/ssl_engine_vars.c#ssl_var_lookup**: SSL 関連変数、環境変数、時刻情報などを実際に解析・解決する実体関数。
  • **symbol:server/ssl.c#ssl_var_lookup**: 古い API を使用するモジュール向けの後方互換性レイヤー。内部で ap_ssl_var_lookup を呼び出して処理を委譲します。
  • **symbol:modules/arch/netware/mod_nw_ssl.c#ssl_var_lookup**: NetWare 環境において、本関数を呼び出すか、またはオプション関数として登録・参照するエンティティ。

エンドツーエンドの流れ

1. **呼び出しの発生**: モジュール（例: mod_lua、mod_nw_ssl、または SSL フィルタ/初期化処理）が、特定の変数名（例: SSL_PROTOCOL や HTTPS）の解決を求めて ssl_var_lookup を呼び出します。 2. **互換レイヤー（古い API の場合）**: server/ssl.c の ssl_var_lookup が呼ばれた場合、処理は即座に ap_ssl_var_lookup へ委譲されます。 3. **実体処理（ssl_engine_vars.c）**:

  • **プール特定**: メモリ割り当て用の apr_pool_t が未指定の場合、リクエストや接続、モジュール設定から自動的に選択します。
  • **リクエスト依存の解決**: リクエストレコード r が存在する場合、HTTP_ ヘッダ、REQUEST_ 属性、REMOTE_ 属性、SERVER_ 属性、および ENV: 変数を順次検索します。
  • **接続依存の解決**: 接続レコード c が存在する場合、SSL_ で始まる変数や HTTPS 変数を ssl_var_lookup_ssl へ委譲して解決します。
  • **独立した解決**: 上記で解決できない場合、SSL_VERSION_、SERVER_SOFTWARE、API_VERSION、および現在時刻（TIME_ 系）を解決します。

    • 4. **結果の返却**: 解決された値は apr_pool_t 上に確保（必要に応じて apr_pstrdup で複製）され、呼び出し元に返されます。

*注意: NetWare 環境（mod_nw_ssl.c）における具体的な呼び出しタイミングや、フックを介した詳細なエンドツーエンドの全フローは、提供された情報からは完全には特定されていません。*

重要な不変条件

  • **メモリ管理のライフサイクル**: 戻り値の文字列は、常に指定された（または自動選択された）apr_pool_t 上に確保されます。呼び出し元は、このプールの生存期間を考慮する必要があります。
  • **フォールバック動作**: 変数名が一致しない、または解決できない場合、ssl_engine_vars.c の実装は NULL ではなく空文字列 "" を返します。
  • **コンテキスト依存性**: r（リクエスト）や c（接続）が NULL の場合、それらに依存する変数（例: SSL_ プレフィックスを持つ変数）は解決されません。

よくある落とし穴

  • **メモリ消費の累積**: ssl_var_lookup は内部で apr_pstrdup や apr_psprintf を多用してプール上にメモリを確保するため、リクエスト処理中に頻繁に呼び出すと、メモリ消費量が累積する可能性があります。
  • **古い API の使用**: server/ssl.c の ssl_var_lookup は後方互換性のためのラッパーです。新規開発やモジュールのメンテナンスにおいては、この互換レイヤーを介さず、直接 ap_ssl_var_lookup を使用することが推奨されます。
  • **コンテキスト不足による解決失敗**: 接続レコード c や SSLConnRec が存在しない、または初期化されていない状態で SSL_ 変数を検索しようとすると、値が解決できず空文字列が返されます。

次に読む場所

  • modules/ssl/ssl_engine_vars.c: 実際の変数解決ロジックと、ssl_var_lookup_ssl などの内部委譲先の実装を確認できます。
  • server/ssl.c: 後方互換性レイヤーと ap_ssl_var_lookup への委譲処理を確認できます。
  • modules/arch/netware/mod_nw_ssl.c: NetWare 環境における SSL 変数参照の具体的なユースケースを確認できます。
