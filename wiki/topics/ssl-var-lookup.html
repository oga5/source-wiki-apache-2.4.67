ssl_var_lookup

概念

ssl_var_lookup は、Apache HTTP Server において、リクエスト、接続、またはサーバー情報に基づいて SSL 関連の変数や環境変数、時刻情報などの値を解決し、文字列として取得するための関数群およびインターフェースです。

なぜ重要か

SSL/TLS 接続に関する詳細情報（暗号スイート、証明書情報、プロトコルバージョンなど）や、サーバー・リクエストのコンテキスト情報を、モジュールや設定、Lua スクリプトなどから動的に参照・利用できるようにするために不可欠です。また、古い API を使用するモジュールとの後方互換性を維持するためのブリッジとしても機能します。

エンドツーエンドの流れ

提供されたコンテキストからは、システム全体における完全なエンドツーエンドの呼び出しフロー（特に NetWare 環境での具体的な呼び出しや、すべての呼び出し元からの遷移）は完全には明らかになっていません。

現在判明している部分的な流れは以下の通りです：

1. **互換レイヤー経由の呼び出し**:

  • 古い API を使用するモジュールが server/ssl.c の ssl_var_lookup を呼び出す。
  • 内部的に ap_ssl_var_lookup へ処理が委譲される。

2. **実体ロジック (ssl_engine_vars.c) での解決**:

  • プールが未指定の場合は、コンテキストから自動的に特定する。
  • リクエスト r が存在する場合、HTTP_ ヘッダ、REQUEST_、REMOTE_、SERVER_ 属性、ENV: 変数を検索する。
  • 接続 c が存在する場合、SSL_ で始まる変数や HTTPS 変数を ssl_var_lookup_ssl へ委譲して解決する。
  • 上記で解決できない場合、SSL_VERSION_、SERVER_SOFTWARE、API_VERSION、現在時刻（TIME_ 系）などを解決する。
  • 解決された値をプールから複製し、結果を返す。

重要な不変条件

  • **メモリ管理**: 戻り値は常に apr_pool_t 上に確保されます。resdup フラグが真の場合、apr_pstrdup によりメモリが複製されます。
  • **フォールバック**: 変数名が一致しない、または解決できない場合、最終的に空文字列 ""（または NULL）が返されます。
  • **コンテキスト依存**: r や c が NULL の場合、それらに依存する変数（例: SSL_ プレフィックスを持つ変数など）は解決されません。

※ NetWare 環境における固有の不変条件や制約については、提供されたコンテキストからは明らかになっていません。

よくある落とし穴

  • **メモリ消費**: apr_pstrdup を多用するため、高頻度で呼び出すとメモリプール内の消費量が増加する可能性があります。
  • **コンテキストの欠落**: 必要なコンテキスト（server_rec, conn_rec, request_rec）が NULL で渡された場合、特定の変数が解決できず空文字列が返されます。
  • **API の混在**: 新規開発においては、互換レイヤーである server/ssl.c の ssl_var_lookup ではなく、直接 ap_ssl_var_lookup を使用することが推奨されます。
