ssl_var_lookup

概念

ssl_var_lookup は、Apache HTTP Server において SSL/TLS 関連の接続情報、リクエスト属性、サーバー設定、環境変数、および時刻情報などの各種変数を解決し、文字列として取得するための関数およびインターフェースです。複数のモジュールや互換レイヤーに同名のシンボルが存在し、役割に応じて実装や委譲が行われます。

なぜ重要か

SSL/TLS 接続のメタデータ（暗号スイート、クライアント証明書、プロトコルバージョンなど）やサーバーの内部状態を、モジュール（mod_ssl、mod_nw_ssl、mod_lua など）やコアシステムから統一的に参照できるようにするために不可欠です。また、古い API を使用するレガシーモジュールとの互換性を維持するためのブリッジとしても機能します。

関与するモジュール

提供されたコンテキストに基づき、以下のモジュールが関与しています。

  • **module:modules/ssl (mod_ssl)**: 実際の変数解決ロジック（ssl_engine_vars.c）を実装します。
  • **module:server (コアサーバー)**: 後方互換性のためのラッパー（server/ssl.c）を提供します。
  • **module:modules/arch/netware (mod_nw_ssl)**: NetWare 環境向けの SSL モジュールであり、本関数を呼び出すか、または関連するシンボルを保持します。

主要エンティティ

  • **symbol:modules/ssl/ssl_engine_vars.c#ssl_var_lookup**:

    • 変数解決の実体となる関数。リクエスト、接続、サーバー情報、時刻情報などを解析し、プール上に確保された文字列を返します。

  • **symbol:server/ssl.c#ssl_var_lookup**:

    • 後方互換性のための静的関数。内部的に推奨 API である ap_ssl_var_lookup へ処理を委譲します。

  • **symbol:modules/arch/netware/mod_nw_ssl.c#ssl_var_lookup**:

    • NetWare 環境において ssl_var_lookup を参照または提供するシンボル。

エンドツーエンドの流れ

システム全体における完全なエンドツーエンドの呼び出しフロー（どのタイミングでどのモジュールが最初に呼び出し、どのように伝播するか）の全容は、提供されたコンテキストからは完全には明らかになっていません。

現在判明している部分的な流れは以下の通りです：

1. **呼び出しの発生**: mod_lua、mod_nw_ssl、または mod_ssl 内部の初期化・IO 処理（ssl_engine_init.c、ssl_engine_io.c）などから ssl_var_lookup が呼び出されます。 2. **互換レイヤー経由の処理**: 古い API を使用する呼び出し元が server/ssl.c の ssl_var_lookup を呼び出した場合、処理は ap_ssl_var_lookup に委譲されます。 3. **変数の解決（ssl_engine_vars.c）**:

  • プール（apr_pool_t）が未指定の場合、リクエストや接続などのコンテキストから自動的にプールを選択します。
  • リクエスト r が存在する場合、HTTP_ ヘッダ、REQUEST_、REMOTE_、SERVER_ 属性、および ENV: 変数を順次検索します。
  • 接続 c が存在する場合、SSL_ で始まる変数や HTTPS 変数を ssl_var_lookup_ssl へ委譲して解決します。
  • 上記で解決できない場合、SSL_VERSION_、SERVER_SOFTWARE、および現在時刻（TIME_ 系）を解決します。
  • 解決された値をプールから複製し、結果を返します。

重要な不変条件

  • **メモリ管理の局所性**:

    • 戻り値の文字列は、常に指定された（または自動選択された）apr_pool_t 上に確保されます。resdup フラグが真の場合、apr_pstrdup によりメモリが複製されます。

  • **フォールバック値**:

    • 変数名が一致しない、または解決できない場合、ssl_engine_vars.c の実装は NULL ではなく空文字列 "" を返します。

  • **コンテキストの依存性**:

    • リクエスト r や接続 c が NULL の場合、それらに依存する変数（特に SSL_ プレフィックスを持つ変数）は解決されません。

よくある落とし穴

  • **メモリ消費の累積**:

    • apr_pstrdup を多用してプール上にメモリを確保するため、同一プール内で本関数を頻繁に呼び出すと、メモリ消費量が累積的に増加する可能性があります。

  • **コンテキスト不足による解決失敗**:

    • 呼び出し時に r や c を適切に渡さないと、特定の変数が解決できず空文字列 "" が返されるため、呼び出し側のコンテキスト管理に注意が必要です。

  • **非推奨 API の使用**:

    • server/ssl.c の ssl_var_lookup は後方互換性のためのブリッジです。新規開発においては、この互換レイヤーを介さず、直接 ap_ssl_var_lookup を使用することが推奨されます。

次に読む場所

  • **modules/ssl/ssl_engine_vars.c**: 実際の変数解決アルゴリズム、ssl_var_lookup_ssl への委譲、および時刻文字列生成マクロ（MKTIMESTR）の実装。
  • **server/ssl.c**: 後方互換レイヤーの実装と ap_ssl_var_lookup への委譲ロジック。
  • **modules/arch/netware/mod_nw_ssl.c**: NetWare プラットフォームにおける SSL 変数参照の統合方法。
