get_line

概念

get_line は、Apache HTTP Server (httpd) の複数の異なるモジュールおよびユーティリティにおいて、入力ソース（バケットブリゲードやファイルストリームなど）から「1行」のデータを読み取るために定義されている、同名かつ独立したヘルパー関数群です。

なぜ重要か

httpd 内では、HTTP/1.1 レスポンスの解析、OCSP レスポンダからの HTTP レスポンス受信、および htdigest ツールによるユーザー認証ファイルの読み込みなど、テキストベースのプロトコルやファイルを1行単位で処理する場面が多々存在します。それぞれのコンテキストに最適化された get_line 実装を用いることで、バッファ管理や改行コード（CRLF/LF）の適切な処理、エラーハンドリングを局所的にカプセル化しています。

関与するモジュール

  • module:modules/http2 (HTTP/2 接続フィルタ内での HTTP/1.1 レスポンス解析)
  • module:modules/ssl (OCSP レスポンス処理における簡易 HTTP クライアント)
  • module:support (htdigest ユーティリティにおけるファイル読み込み)

主要エンティティ

  • symbol:modules/http2/h2_c2_filter.c#get_line: HTTP/1.1 形式のレスポンスデータ（バケットブリゲード）から CRLF で終了する1行を読み取り、バッファにコピーします。
  • symbol:modules/ssl/ssl_util_ocsp.c#get_line: OCSP レスポンダからの HTTP レスポンスストリームから1行を読み取り、改行コードを除去してメモリプール上に割り当てて返します。
  • symbol:support/htdigest.c#get_line: apr_file_t ファイルストリームから1行を文字単位で読み込み、指定バッファに格納します。

エンドツーエンドの流れ

これら3つの get_line 関数は、それぞれ独立したコンテキストで動作しており、これらを横断する単一のエンドツーエンドの流れは存在しません。それぞれの個別フローは以下の通りです。

1. HTTP/2 フィルタ (h2_c2_filter.c)

  • h2_c2_filter_catch_h1_out などの解析ロジックから呼び出されます。
  • apr_brigade_split_line を使用して、入力ブリゲード bb から1行分を parser->tmp へ切り出します。
  • 切り出したデータを apr_brigade_flatten で line バッファにコピーします。
  • 行末に H2_CRLF が存在する場合はこれを除去して APR_SUCCESS を返します。行が不完全な場合は ap_save_brigade を用いて parser->tmp の内容を parser->saveto に退避し、APR_EAGAIN を返して処理を中断します。

2. OCSP ユーティリティ (ssl_util_ocsp.c)

  • OCSP レスポンスの解析処理から呼び出されます。
  • bbout をクリーンアップしたのち、apr_brigade_split_line を使用して bbin から最大 8192 バイトまでを bbout に切り出します。
  • apr_brigade_pflatten を使用して、ブリゲードの内容をメモリプール p 上の文字列にフラット化します。
  • 行末の LF（および直前の CR）をヌル文字に置換して返却します。

3. htdigest ツール (htdigest.c)

  • ファイルから行を読み込むループ内で呼び出されます。
  • apr_file_getc を繰り返し呼び出し、改行文字（\n）に達するかバッファサイズ制限（n - 2）に達するまで文字をバッファ s に格納します。
  • ヌル終端を付与し、成功時は 0、EOF またはエラー時は 1 を返します。

重要な不変条件

  • **HTTP/2**:
  • parser->tmp は一時的な切り出し用、parser->saveto は不完全な行を次回の呼び出しまで保持するために一貫して使用されます。
  • 正常終了時、返される line は必ずヌル終端され、末尾の CRLF は除去されています。
  • **OCSP**:
  • bbout は呼び出しのたびに apr_brigade_cleanup によって初期化されます。
  • 読み取られた行は必ず APR_ASCII_LF で終了している必要があり、末尾の LF（および直前の CR）はヌル文字に置換されます。
  • **htdigest**:
  • バッファ s は、少なくとも n バイトの領域を確保している必要があります。
  • 最大 n - 2 文字までしか読み込まないことで、バッファオーバーフローを防ぎます。

よくある落とし穴

  • **バッファサイズ制限とエラーハンドリング**:
  • HTTP/2 の get_line では、行の長さがバッファサイズ len を超える場合、APR_ENOSPC が発生します。また、LF のみで CRLF が存在しない場合の挙動は未定義です。
  • OCSP の get_line では、8192 バイトを超える行や、行末が LF でない場合はエラーとなり NULL が返されます。
  • htdigest の get_line では、戻り値 1 が EOF とエラーの両方を表すため、呼び出し側で詳細な状態判別を行う際には注意が必要です。また、バッファサイズを超える長い行は途中で切り捨てられます。
  • **名前の衝突と誤解**:
  • これら3つの関数はシグネチャも動作要件も異なるため、静的解析やコードリーディング時に同一の共通関数として混同しないよう注意が必要です。
