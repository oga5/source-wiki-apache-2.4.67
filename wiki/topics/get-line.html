get_line

概念

get_line は、Apache HTTP Server（httpd）の複数の独立したコンテキストにおいて、入力ストリーム（バケットブリゲードまたはファイル）から1行分のデータを読み込み、改行コードの処理やバッファへの格納を行うために定義されているヘルパー関数群です。これらは同名ですが、シグネチャや対象とする入力ソース、エラーハンドリングの挙動が異なり、それぞれのモジュール内で独立して実装されています。

なぜ重要か

各コンテキストにおいて、プロトコル仕様（HTTP/1.1のCRLF、OCSPレスポンスの改行、htdigestファイルの改行）に準拠した行単位のパースを安全かつ効率的に行うために重要です。バケットブリゲードからの部分的な読み込み（ノンブロッキング処理）への対応や、バッファオーバーフローの防止など、堅牢なストリーム処理の基礎を担っています。

関与するモジュール

  • module:modules/http2
  • module:modules/ssl
  • module:support

主要エンティティ

  • **symbol:modules/http2/h2_c2_filter.c#get_line**: HTTP/1.1 形式のレスポンスデータ（ブリゲード）から CRLF で終了する 1 行を読み取り、指定されたバッファへコピーする。
  • **symbol:modules/ssl/ssl_util_ocsp.c#get_line**: OCSP レスポンダから送信された HTTP レスポンスのストリームから 1 行単位で文字列を読み取り、改行コードを除去してメモリプール上に返却する。
  • **symbol:support/htdigest.c#get_line**: ファイルストリームから 1 行を読み込み、指定されたバッファに格納する。

エンドツーエンドの流れ

これら3つの get_line 関数は、それぞれ独立したフローで呼び出されており、システム全体を横断する単一のエンドツーエンドフローは存在しません。それぞれの独立した流れは以下の通りです。

1. **HTTP/2 接続フィルタフロー (modules/http2)**:

  • h2_c2_filter_catch_h1_out などの解析ロジックから呼び出されます。
  • 入力ブリゲード bb から apr_brigade_split_line を使用して1行分を parser->tmp へ切り出します。
  • 切り出したデータを apr_brigade_flatten で line バッファにコピーし、行末の CRLF を除去します。
  • データが不完全な場合は ap_save_brigade を用いて parser->saveto に退避し、次回呼び出し時に処理を再開します。

2. **OCSP クライアントフロー (modules/ssl)**:

  • ssl_util_ocsp.c 内の OCSP レスポンス解析処理から呼び出されます。
  • ソースブリゲード bbin から apr_brigade_split_line で最大 8192 バイトまでを bbout に切り出します。
  • apr_brigade_pflatten を使用してメモリプール p 上にフラット化し、行末の LF（および直前の CR）をヌル文字に置換して返却します。

3. **htdigest ユーティリティフロー (support)**:

  • htdigest のメイン処理ループから呼び出されます。
  • apr_file_getc を用いてファイルから1文字ずつ読み込み、改行文字（\n）に達するか、バッファサイズ制限に達するまでバッファ s に格納します。

重要な不変条件

  • **改行コードの除去と終端**:
  • modules/http2 の実装では、行末の CRLF が除去され、line はヌル終端されます。
  • modules/ssl の実装では、行末の LF（および直前の CR）が除去され、ヌル終端されます。
  • support の実装では、改行文字（\n）自体はバッファに保持されたままヌル終端されます。
  • **バッファ境界の維持**:
  • modules/http2 では、行の長さがバッファサイズ len を超える場合に APR_ENOSPC を返します。
  • modules/ssl では、8192 バイトを超える行はエラー（NULL 返却）となります。
  • support では、バッファサイズ n - 2 を超える文字は読み込まずに切り捨てられます。

よくある落とし穴

  • **共通APIではない点**: 同名ですが、シグネチャや戻り値の型（apr_status_t、char *、int）が全く異なるため、混同しないよう注意が必要です。
  • **不完全なデータの扱い**:
  • modules/http2 の get_line は、ノンブロッキング読み込み時に APR_EAGAIN を返す可能性があり、呼び出し側で退避バッファ（parser->saveto）の適切な管理が必要です。
  • modules/ssl の get_line は、行末が LF で終わらない場合（行が長すぎる場合など）にエラーログを出力して NULL を返します。
  • support の get_line は、EOF とエラーの双方で 1 を返すため、呼び出し側で詳細な状態判別ができません。
