piped_log

概念

piped_log は、Apache HTTP Serverにおいて外部プロセスへログデータを転送するためのパイプ通信を抽象化し、管理するための構造体です。公開ヘッダーである include/http_log.h では不透明（opaque）な構造体として宣言されており、具体的なメンバ定義は server/log.c 内に隠蔽されています。

なぜ重要か

ログのローテーションや外部システムへの転送を安全に行うために、外部プログラム（例: rotatelogs など）と連携する仕組みを提供します。特に、信頼性の高いパイプドログ機能（AP_HAVE_RELIABLE_PIPED_LOGS）が有効な環境では、外部ログプロセスが異常終了した際に自動的に再起動を試みる死活監視機能を提供し、ログ記録の継続性を担保する重要な役割を担っています。

関与するモジュール

  • module:include: 外部モジュール向けのAPI定義および不透明構造体としての宣言を提供します。
  • module:server: 構造体の具体的な定義、プロセスの生成・監視、およびライフサイクル管理の実装を担当します。

主要エンティティ

  • symbol:include/http_log.h#piped_log: 外部モジュールに公開される不透明構造体 piped_log の不完全宣言。
  • symbol:server/log.c#piped_log: 実際のメンバ（メモリプール、ファイル記述子、プロセス情報など）を保持する構造体の定義。

エンドツーエンドの流れ

1. **初期化と生成**: mod_log_config や mod_log_forensic などのログ関連モジュールが、メモリプール（apr_pool_t）と実行プログラム名を指定して ap_open_piped_log（または ap_open_piped_log_ex）を呼び出します。 2. **プロセス起動**: 内部的に piped_log_spawn が呼び出され、外部ログプロセスが生成されます。 3. **ログの書き込み**: ログ出力時、呼び出し元は ap_piped_log_write_fd を介して書き込み用ファイル記述子（write_fd）を取得し、パイプ経由でデータを外部プロセスへ転送します。 4. **死活監視（条件付き）**: AP_HAVE_RELIABLE_PIPED_LOGS が有効な場合、piped_log_maintenance が apr_proc_other_child_register を介して子プロセスを登録し、異常終了時に再起動を試みます。 5. **クローズ**: ap_close_piped_log を呼び出すことで、パイプが閉じられ、関連するリソースが解放されます。

*注意: プラットフォームごとの詳細なプロセス生成処理や、パイプ詰まり発生時の詳細なエラーハンドリングの挙動など、提供されたコンテキストに含まれないエンドツーエンドのフローの詳細は完全には判明していません。*

重要な不変条件

  • **ライフサイクルの紐付け**: piped_log 構造体および関連するファイル記述子やプロセス情報は、生成時に渡された apr_pool_t のライフサイクルに従って管理されます。
  • **カプセル化の維持**: 構造体の実体は server/log.c に隠蔽されているため、外部モジュールは必ず ap_ で始まる公開 API を介して操作しなければならず、直接メンバにアクセスすることはできません。

よくある落とし穴

  • **直接アクセスの試み**: piped_log は不透明型であるため、server/log.c 以外のコードから直接構造体のメンバ（write_fd など）にアクセスしようとするとコンパイルエラーになります。
  • **プラットフォームによる信頼性の差異**: AP_HAVE_RELIABLE_PIPED_LOGS が定義されていない環境では、外部ログプロセスの自動再起動機能が提供されないため、プロセス停止時にログが失われるリスクがあります。
  • **I/O ボトルネック**: 高負荷環境において、外部プロセスへのパイプ書き込みがボトルネックとなり、サーバー全体のパフォーマンスに影響を与える可能性があります。

次に読む場所

  • include/http_log.h: piped_log の不透明型宣言および操作用 API のシグネチャを確認できます。
  • server/log.c: struct piped_log の具体的な定義、およびプロセス生成・監視ロジックの実装を確認できます。
