piped_log

概念

piped_log は、Apache HTTP Serverにおいて外部プロセスへログを転送するためのパイプ処理を抽象化・管理するデータ構造です。 include/http_log.h では不透明（opaque）な構造体として宣言され、server/log.c でその具体的なメンバ（メモリプール、読み書き用のファイル記述子、プロセス情報など）が定義されています。

なぜ重要か

ログの収集やローテーションを外部プログラムに委譲する際、サーバープロセスと外部ログ処理プロセス間のパイプ通信を安全かつ効率的に管理するために不可欠です。特に、信頼性の高いパイプドログ（AP_HAVE_RELIABLE_PIPED_LOGS）が有効な環境では、外部ログプロセスの死活監視や異常終了時の自動再起動をサポートし、ログの損失を防ぐ役割を果たします。

関与するモジュール

  • module:include
  • module:server

主要エンティティ

  • symbol:include/http_log.h#piped_log: 外部モジュールに公開される不透明構造体としての piped_log の宣言。
  • symbol:server/log.c#piped_log: 構造体の具体的なメンバ（apr_pool_t *p, apr_file_t *read_fd, *write_fd, apr_proc_t *pid など）を定義する実体。

エンドツーエンドの流れ

1. **生成**: ログ関連モジュール（mod_log_config.c や mod_log_forensic.c など）が ap_open_piped_log または ap_open_piped_log_ex を呼び出し、メモリプールと実行プログラム名を渡して piped_log 構造体を生成します。 2. **プロセス起動**: 内部的に piped_log_spawn が呼び出され、外部プロセスが起動してパイプが確立されます。 3. **ログ書き込み**: ログ出力時、ap_piped_log_write_fd を介して取得した書き込み用ファイル記述子（write_fd）に対してログデータが書き込まれ、パイプ経由で外部プロセスへ転送されます。 4. **監視と再起動**: AP_HAVE_RELIABLE_PIPED_LOGS が有効な場合、piped_log_maintenance が apr_proc_other_child_register を介して子プロセスを監視し、異常終了を検知した場合は再起動を試みます。 5. **破棄**: ap_close_piped_log が呼び出されることで、パイプがクローズされ、関連するリソースが解放されます。

*注意: 外部プロセス起動の失敗時やパイプ作成失敗時の詳細なエラーハンドリングの全容、および具体的な呼び出し元モジュールにおける詳細な書き込みフローの完全なエンドツーエンドの挙動は、提供されたコンテキストからは一部未確定です。*

重要な不変条件

  • **メモリ管理の依存性**: piped_log 構造体および関連リソースの生存期間は、生成時に渡された apr_pool_t（メモリプール）のライフサイクルに厳密に従わなければなりません。
  • **カプセル化の維持**: 外部モジュールからは piped_log は不透明な型として扱われるため、直接メンバにアクセスせず、必ず ap_piped_log_read_fd や ap_piped_log_write_fd などの公開 API を介して操作しなければなりません。
  • **条件付きプロセス管理**: AP_HAVE_RELIABLE_PIPED_LOGS が定義されている場合のみ、外部プロセスの情報（pid や cmdtype）が保持され、自動再起動機能が有効になります。未定義の環境では自動再起動は提供されません。

よくある落とし穴

  • **直接アクセスによる破損**: piped_log は include/http_log.h で不透明型として定義されていますが、server/log.c でその実体が定義されています。外部モジュールから構造体のメンバへ直接アクセスしようとすると、コンパイルエラーや予期せぬ動作を引き起こすため、必ず提供されている ap_ API を使用する必要があります。
  • **I/O ボトルネック**: 高負荷環境において、パイプ経由のログ転送処理がボトルネックとなり、サーバー全体のパフォーマンスに影響を与える可能性があります。
  • **プラットフォーム/ビルド依存性**: AP_HAVE_RELIABLE_PIPED_LOGS が未定義のビルド環境では、外部ログプロセスが異常終了した際に自動再起動が行われないため、ログの消失に繋がる恐れがあります。
