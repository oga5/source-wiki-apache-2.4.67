piped_log

概念

piped_log は、Apache HTTP Server において外部プロセスへログを転送するためのパイプ通信を抽象化・管理する構造体です。ヘッダーファイル（include/http_log.h）では不透明（opaque）な型として宣言され、実体は server/log.c で定義されています。これにより、外部ログ処理プロセスとのパイプ接続、データの転送、および子プロセスのライフサイクル管理がカプセル化されます。

なぜ重要か

ログ出力を外部の専用プロセス（例: ログローテーションツールなど）にパイプ経由で委譲することで、Webサーバー本体の処理からログ管理の負荷を分離できます。また、特定の環境（AP_HAVE_RELIABLE_PIPED_LOGS が有効な場合）では、ログ転送先の子プロセスが異常終了した際に自動的に再起動を試みる死活監視機能を提供し、ログの損失を防ぐ極めて重要な役割を担っています。

関与するモジュール

  • module:include (include/http_log.h)
  • module:server (server/log.c)
  • 呼び出し元モジュール（modules/loggers/mod_log_config.c、modules/loggers/mod_log_forensic.c など）

主要エンティティ

  • **symbol:include/http_log.h#piped_log**:

    • 外部モジュール向けに公開されている不透明構造体としての定義です。直接メンバにアクセスすることはできず、APIを介して操作します。

  • **symbol:server/log.c#piped_log**:

    • 構造体の具体的な定義です。以下のメンバを含みます。

  • apr_pool_t *p: 構造体の生存期間を管理するメモリプール。
  • apr_file_t *read_fd, *write_fd: パイプの読み取り用および書き込み用のファイル記述子。
  • char *program, apr_proc_t *pid, apr_cmdtype_e cmdtype（AP_HAVE_RELIABLE_PIPED_LOGS 定義時のみ）: 外部プロセスの再起動や管理に必要な情報。

エンドツーエンドの流れ

1. **生成とプロセス起動**: ログ関連モジュールが ap_open_piped_log または ap_open_piped_log_ex を呼び出し、メモリプールと実行プログラム名を渡して piped_log を生成します。内部的に piped_log_spawn が呼び出され、外部プロセスが起動してパイプが確立されます。 2. **ログの書き込み**: サーバーは ap_piped_log_write_fd を通じて書き込み用のファイル記述子（write_fd）を取得し、ログデータをパイプ経由で外部プロセスへ転送します。 3. **プロセスの監視と再起動（条件付き）**: AP_HAVE_RELIABLE_PIPED_LOGS が有効な場合、piped_log_maintenance 関数が apr_proc_other_child_register を介して子プロセスの死活監視を行い、異常終了を検知した場合は再起動を試みます。 4. **破棄**: ap_close_piped_log を呼び出すことで、パイプがクローズされ、関連するリソースが解放されます。

重要な不変条件

  • **不透明性の維持**:

    • piped_log は不透明型であるため、外部モジュールから直接構造体のメンバにアクセスしてはならず、必ず ap_piped_log_read_fd や ap_piped_log_write_fd などの提供された API を介して操作しなければなりません。

  • **ライフサイクルの依存**:

    • piped_log 構造体および関連リソースの生存期間は、生成時に渡された apr_pool_t のライフサイクルに厳密に従います。

  • **プラットフォーム/ビルド依存の監視**:

    • 子プロセスの自動再起動機能は、AP_HAVE_RELIABLE_PIPED_LOGS が定義されている環境でのみ保証されます。

よくある落とし穴

  • **直接アクセスによる破損**:

    • server/log.c 以外から構造体の内部メンバに直接アクセスしようとすると、カプセル化が壊れ、将来のコード変更時に互換性の問題が発生します。

  • **I/O ボトルネック**:

    • 高負荷環境において、外部ログプロセスの処理遅延やパイプのバッファ詰まりが発生すると、サーバー全体の I/O パフォーマンスに悪影響を及ぼす可能性があります。

  • **自動再起動の未サポート環境**:

    • AP_HAVE_RELIABLE_PIPED_LOGS が未定義の環境では、外部プロセスが終了しても自動的に再起動されないため、ログの転送が停止するリスクがあります。
