Module: modules/lua

概念

modules/lua（mod_lua）は、Apache HTTP Server 内で Lua スクリプトを直接実行するための拡張モジュールです。C言語で独自のApacheモジュールを開発することなく、軽量かつ高速なスクリプト言語である Lua を用いて、Webサーバーの挙動を柔軟にカスタマイズ・制御する仕組みを提供します。

なぜ重要か

このモジュールは、Apache のコア機能やリクエスト処理パイプラインに対して、スクリプト層からの動的な介入を可能にします。

  • **柔軟なリクエスト制御**: リクエストの書き換え、ヘッダー操作、アクセス制御、コンテンツ生成などを Lua で記述できます。
  • **高パフォーマンス**: C言語で実装された Apache 内部オブジェクト（request_rec や APR など）への高速なバインディングを提供します。
  • **データベース連携**: APR-util の DBD 機能を介して、Lua から直接データベースにアクセスできます。

主要エンティティ

本モジュールは、役割ごとに分割された以下のファイル群で構成されています。

  • **mod_lua.c / mod_lua.h**: モジュールのエントリポイント。Apache の各種フック（Translate Name, Header Parser, Handler など）の登録や、モジュールの初期化を行います。
  • **lua_vmprep.c / lua_vmprep.h**: Lua 仮想マシン（lua_State）の作成、初期化、キャッシュ、およびライフサイクル管理を担当します。
  • **lua_request.c / lua_request.h**: Apache のリクエスト構造体（request_rec）を Lua 側にエクスポートし、ヘッダーの読み書きやレスポンスの送信を行うインターフェースを提供します。
  • **lua_config.c / lua_config.h**: LuaHook* や LuaMapHandler などの Apache 設定ディレクティブをパースし、設定情報を管理します。
  • **lua_apr.c / lua_apr.h**: APR（Apache Portable Runtime）のユーティリティ（テーブル操作や暗号化など）を Lua から利用可能にします。
  • **lua_dbd.c / lua_dbd.h**: Apache の DBD（データベース接続プール）機能を Lua にバインドします。
  • **lua_passwd.c / lua_passwd.h**: パスワードのハッシュ化や検証機能を提供します。

エンドツーエンドの流れ

1. **設定のロード**: Apache の起動時または再起動時に、lua_config.c が設定ファイル内の Lua 関連ディレクティブを解析し、設定を保持します。 2. **VM の準備**: リクエスト処理時または起動時に、lua_vmprep.c が Lua VM を初期化し、必要な Apache API（lua_request や lua_apr など）を Lua 環境に登録します。 3. **フックの実行**: リクエストが特定のフェーズに達すると、mod_lua.c に登録されたフックが起動し、指定された Lua スクリプトを実行します。 4. **リクエスト操作と応答**: Lua スクリプト内から r:write や r.headers_out などのバインディング（lua_request.c）を介してクライアントへ応答を返し、処理を終了します。

*注意: マルチスレッド環境（Worker/Event MPMなど）における VM の再利用や、接続プールとの詳細な同期フローなど、一部の内部的なエンドツーエンドの挙動は完全には判明していません。*

重要な不変条件

  • **スレッドセーフティ**: Lua VM（lua_State）はスレッドセーフではないため、複数のスレッドで同時に同一の VM インスタンスを共有してはなりません（lua_vmprep.c による適切なスコープ管理と排他制御が必要です）。
  • **リクエストの生存期間**: lua_request.c を通じて Lua 側に渡される request_rec への参照は、そのリクエストのライフサイクル内でのみ有効でなければなりません。

*注意: メモリプール（apr_pool_t）の解放タイミングに伴うオブジェクトの有効性など、厳密な不変条件の全容は現時点では完全には特定されていません。*

よくある落とし穴

  • **VM 生成のオーバーヘッド**: リクエストごとに Lua VM を新規作成するとパフォーマンスが著しく低下するため、LuaScope ディレクティブ等による適切なキャッシュ（Thread/Request/Once）の選択が必要です。
  • **メモリリーク**: Lua 側で確保したリソースや、C言語側からバインドされたオブジェクトが適切にガベージコレクション（GC）されない場合、メモリリークの原因となります。

*注意: 実際の運用環境における詳細なエラーパターンや競合状態の落とし穴については、現時点では完全には特定されていません。*
