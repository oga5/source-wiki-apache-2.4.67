Multi-Processing Modules (MPM) and Worker Lifecycle

概念

**Multi-Processing Modules (MPM)** は、Apache HTTP Server がネットワークポートをバインドし、リクエストを受け付け、それを処理する子プロセスやスレッドに割り当てるためのマルチプロセッシングモデルを定義するコアアーキテクチャです。

**Worker Lifecycle** は、これらリクエストを処理する実行主体（プロセスまたはスレッド）が、親プロセスによって生成（Fork/Spawn）され、リクエストを待ち受けて処理し、アイドル状態を維持し、最終的に終了（Terminate）するまでの一連の生存周期を指します。

主要なMPMモデルには以下の3つがあります：

  • **Prefork (prefork.c)**: スレッドを使用せず、1つの子プロセスが同時に1つの接続を処理するマルチプロセスモデル。
  • **Worker (worker.c)**: マルチプロセスとマルチスレッドのハイブリッドモデル。各子プロセスが複数のワーカースレッドと1つのリスナースレッドを持ちます。
  • **Event (event.c)**: Workerモデルをベースに、Keep-Aliveなどのアイドル接続を専用のリスナースレッドが非同期に監視し、リクエスト発生時のみワーカースレッドを割り当てるイベント駆動型モデル。

なぜ重要か

MPMとワーカーのライフサイクル管理は、Webサーバーの**スケーラビリティ、パフォーマンス、およびリソース消費（メモリ・CPU）**を決定づける最重要コンポーネントです。 トラフィックの特性（同時接続数、Keep-Alive接続の割合、動的/静的コンテンツの比率）や、実行するアプリケーションモジュールのスレッドセーフティ要件に応じて適切なMPMを選択し、そのライフサイクルパラメータをチューニングする必要があります。

関与するモジュール

  • **module:include**: MPMの共通インターフェースやクエリAPI（include/ap_mpm.h）を提供します。
  • **module:server/mpm/event**: Event MPMの実装（server/mpm/event/event.c）を担当します。
  • **module:server/mpm/prefork**: Prefork MPMの実装（server/mpm/prefork/prefork.c）を担当します。
  • **module:server/mpm/worker**: Worker MPMの実装（server/mpm/worker/worker.c）を担当します。

主要エンティティ

  • **include/ap_mpm.h**:

    • 各MPMモジュールが共通で提供すべきインターフェースや、サーバーコアがMPMの状態（稼働中のスレッド数、プロセス制限など）を問い合わせるためのクエリ関数・マクロを定義します。

  • **server/mpm/prefork/prefork.c**:

    • スレッドを使用しないシンプルなプロセスベースのワーカー管理ロジック。シグナルハンドリングや、親プロセスによる子プロセスの定期的なフォーク・回収処理が実装されています。

  • **server/mpm/worker/worker.c**:

    • プロセス生成と、プロセス内でのマルチスレッド（リスナースレッドおよび複数のワーカースレッド）の生成・同期・ライフサイクル管理ロジック。

  • **server/mpm/event/event.c**:

    • 非同期I/O（epollやkqueueなど）を利用して、Keep-Alive状態のソケットをワーカースレッドから分離して管理する、高度なイベント駆動型ライフサイクルロジック。

エンドツーエンドの流れ

1. **初期化と起動**: 親プロセスが起動し、設定ファイルから最大/最小ワーカー数などのパラメータを読み込みます。その後、ap_mpm.h で定義されたインターフェースを介して、各MPM固有の初期化処理が実行されます。 2. **子プロセスの生成 (Spawn)**: 設定された初期プロセス数（StartServers など）に基づき、親プロセスが子プロセスをフォークします。 3. **スレッドの生成 (Worker/Eventのみ)**: worker や event では、起動した子プロセス内でリスナースレッドと複数のワーカースレッドが生成されます。 4. **リクエストの受付と処理**:

  • **Prefork**: 子プロセスが直接ソケットを accept し、リクエストを処理します。
  • **Worker**: リスナースレッドが accept した接続をキューに投入し、空いているワーカースレッドがそれを引き取って処理します。
  • **Event**: リスナースレッドが接続を監視し、アクティブなリクエストがある場合のみワーカースレッドに処理を委譲します。Keep-Alive中はリスナーが監視を継続します。

    • 5. **動的維持と終了**: 親プロセスは定期的に子プロセスの稼働状況を監視し、アイドル状態のワーカー数が設定範囲（MinSpareThreads/MaxSpareThreads など）に収まるよう、プロセスの追加生成や終了シグナルの送信を行います。

*注意: 各MPM内部における詳細なシグナルハンドリング、スコアボード（Scoreboard）の更新タイミング、およびプラットフォーム固有のシステムコール呼び出しの完全なシーケンスは、提供されたソースコードの範囲内では未検証です。*

重要な不変条件

  • **最大リソース制限の遵守**:

    • 生成される総プロセス数および総スレッド数は、設定された上限（MaxRequestWorkers / ServerLimit / ThreadLimit）をいかなる場合も超えてはなりません。

  • **排他制御の維持**:

    • worker や event において、複数のスレッドが同一の接続ソケットや共有リソース（スコアボードなど）に同時にアクセスして競合を起こさないよう、適切なミューテックスやキューによる同期が維持される必要があります。

*注意: 各MPMの実装コード（prefork.c, worker.c, event.c）における具体的なロック順序や、シグナルセーフなコンテキストにおける不変条件の詳細は、提供された情報からは完全には特定されていません。*

よくある落とし穴

  • **Preforkにおけるメモリ枯渇**:

    • Preforkはリクエストごとにプロセスを割り当てるため、高負荷時に MaxRequestWorkers を高く設定しすぎると、物理メモリを使い果たしてスワップアウトやOOM Killerによる強制終了を引き起こす原因になります。

  • **非スレッドセーフなモジュールとの競合**:

    • worker や event などのマルチスレッドMPM環境下で、スレッドセーフでないサードパーティ製モジュール（例: 非スレッドセーフ版のPHPなど）をロードすると、メモリ汚染や予期せぬクラッシュが発生します。

  • **Event MPMにおけるリソースのデッドロック**:

    • 非同期処理の特性上、特定のフィルタやモジュールがブロッキング操作を伴う処理をワーカースレッド上で行うと、Event MPMの非同期的な利点が失われ、スレッドの枯渇を招くことがあります。

次に読む場所

  • **include/ap_mpm.h**: MPMが提供すべき共通APIとクエリインターフェースの定義を確認する。
  • **server/mpm/prefork/prefork.c**: 最もシンプルなプロセスベースのライフサイクル管理とシグナル処理の実装を理解する。
  • **server/mpm/worker/worker.c**: マルチスレッド環境におけるスレッド生成とキューイングによるタスク分散モデルを学習する。
  • **server/mpm/event/event.c**: リスナースレッドによる非同期接続監視と、Keep-Alive処理の最適化ロジックを詳細に分析する。
