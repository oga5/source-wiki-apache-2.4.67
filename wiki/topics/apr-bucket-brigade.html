apr_bucket_brigade

概念

apr_bucket_brigade は、Apache HTTP Server のフィルタチェーンにおいて、データ（バケット）のストリームを保持、管理、および転送するための抽象的なコンテナ（バケットの連結リスト）です。これは Apache Portable Runtime (APR) の基盤構造体であり、フィルタ間でデータをやり取りする際の基本単位として機能します。

なぜ重要か

フィルタチェーン内を通過するデータストリームを効率的に処理するために不可欠です。各フィルタは、このブリゲード内のバケットを順次処理し、データの変換、追加、削除、分割を行うことで、メモリ効率の良いストリーム処理とモジュール間での柔軟なデータ受け渡しを実現します。

関与するモジュール

  • module:include
  • module:modules/filters

主要エンティティ

  • **symbol:include/util_filter.h#apr_bucket_brigade**: フィルタ関数の引数として使用される不透明な構造体。
  • **symbol:modules/filters/mod_include.h#apr_bucket_brigade**: SSI（Server Side Includes）フィルタのハンドラシグネチャ（include_handler_fn_t）の引数として使用される定義。

エンドツーエンドの流れ

1. ap_core_output_filter、ap_byterange_filter、ap_http_header_filter などのフィルタ関数が、引数として apr_bucket_brigade を受け取ります。 2. フィルタはブリゲード内のバケットを順次処理します。例えば、SSI フィルタの実行時には、登録されたハンドラ（include_handler_fn_t）が include_ctx_t を参照して条件分岐や変数展開を行い、その結果を apr_bucket_brigade に挿入します。 3. 処理されたブリゲードは、ap_pass_brigade を通じて後続のフィルタへ渡されます。 4. ストリームの終了時には、各フィルタが内部状態をフラッシュし、リソースを解放します。

※なお、詳細なエンドツーエンドの全体フローや、すべてのフィルタにおける具体的な処理順序の全容は、提供されたコンテキストからは完全には明らかになっていません。

重要な不変条件

  • ap_pass_brigade を介してデータを渡す際、ブリゲードの所有権は呼び出し元に帰属します。
  • フィルタ内でデータを保持する必要がある場合は、ap_save_brigade を使用して所有権を管理する必要があります。
  • フィルタ処理において、バケットの挿入、削除、または分割が行われる可能性があります。

よくある落とし穴

  • **メモリ管理の誤り**: メモリ所有権の管理を誤ると、解放済みメモリへのアクセスやメモリリークが発生する可能性があります。
  • **エラー伝播の欠如**: フィルタチェーンの処理中やバケット読み込み時にエラーが発生した場合、apr_status_t 型の戻り値を通じて適切にエラーを伝播させる必要があります。
  • **終端処理の不備**: ストリームの終端（EOS）に達した際の処理が適切に行われない場合、フィルタチェーンの停止やデータの欠損が発生する可能性があります。

次に読む場所

  • **include/util_filter.h**: フィルタ関数の型定義（ap_filter_func）やブリゲードの基本的な扱い。
  • **modules/filters/mod_include.h**: SSI フィルタハンドラ（include_handler_fn_t）におけるブリゲードの使用方法。
  • **apr_buckets.h** (APRのバケット管理ドキュメント): apr_bucket_brigade の詳細な内部構造や操作関数。
