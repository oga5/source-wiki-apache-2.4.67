apr_bucket_brigade

概念

apr_bucket_brigade は、Apache HTTP Server のフィルタチェーンにおいて、データ（バケット）のストリームを保持・管理・転送するための抽象的なコンテナ（バケットの連結リスト）です。APR（Apache Portable Runtime）が提供する基盤構造体であり、フィルタ間でデータを受け渡すための基本単位として機能します。

なぜ重要か

Apache のフィルタフレームワークにおいて、コンテンツの変換、追加、削除などのストリーム処理を効率的かつ統一的に行うための共通インターフェースを提供するからです。例えば、SSI（Server Side Includes）を処理する mod_include などのフィルタは、このブリゲードを介してデータを受け取り、条件分岐や変数展開の結果を挿入して後続のフィルタへ伝播させます。

関与するモジュール

  • module:include
  • module:modules/filters

主要エンティティ

  • **symbol:include/util_filter.h#apr_bucket_brigade**

    • フィルタ関数（ap_out_filter_func 等）の引数として使用される不透明な構造体としての定義。

  • **symbol:modules/filters/mod_include.h#apr_bucket_brigade**

    • SSI フィルタのハンドラシグネチャ（include_handler_fn_t）の引数として使用される定義。

エンドツーエンドの流れ

1. **入力**: ap_core_output_filter、ap_byterange_filter、ap_http_header_filter などのフィルタ関数が、処理対象のデータストリームとして apr_bucket_brigade を引数に受け取ります。 2. **処理**: フィルタ（例: mod_include）は、ブリゲード内のバケット（apr_bucket）を順次処理します。SSI 実行時には、登録されたハンドラ（include_handler_fn_t）が include_ctx_t を参照して変数展開などを行い、結果をブリゲードに挿入します。 3. **出力**: 処理されたブリゲードは、ap_pass_brigade を介して後続のフィルタへ渡されます。 4. **終端**: ストリームの終了時（EOS）には、各フィルタが内部状態をフラッシュし、リソースを解放します。

*注意: フィルタチェーン全体の実行順序の決定ロジックや、すべてのフィルタ間における具体的な受け渡しフローの全容は、提供されたコンテキストからは完全には特定されていません。*

重要な不変条件

  • **所有権の管理**: ap_pass_brigade を介してデータを渡す際、ブリゲードの所有権は呼び出し元に帰属します。フィルタ内でデータを保持する必要がある場合は、ap_save_brigade を使用して所有権を管理しなければなりません。
  • **データ構造の整合性**: ブリゲードは apr_bucket の連結リストであり、フィルタ処理の過程でバケットの挿入、削除、または分割が安全に行われる必要があります。

よくある落とし穴

  • **メモリ管理の誤り**: メモリ所有権の管理を誤ると、解放済みメモリへのアクセスやメモリリークが発生する原因になります。
  • **エラー伝播の欠如**: フィルタチェーンの処理中やメモリ不足時に発生したエラーを、apr_status_t 型の戻り値を通じて適切に伝播させないと、予期せぬ動作を引き起こします。
  • **終端（EOS）処理の不備**: ストリームの終端に達した際の処理が適切に行われない場合、フィルタチェーンが停止したり、データが欠損したりする可能性があります。

次に読む場所

  • include/util_filter.h : フィルタ関数の定義や apr_bucket_brigade の基本的な使われ方。
  • modules/filters/mod_include.h : SSI ハンドラにおけるブリゲードの利用方法。
  • APR（Apache Portable Runtime）のバケット管理ドキュメント（apr_buckets.h）: ブリゲードの内部構造や詳細な操作関数。
