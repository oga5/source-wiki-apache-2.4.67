apr_bucket_brigade

概念

apr_bucket_brigade は、Apache HTTP Server のフィルタチェーンにおいて、データ（バケット）のストリームを保持・管理・転送するための抽象的なコンテナ（バケットの連結リスト）です。apr_buckets.h で定義される不透明な構造体であり、フィルタ関数やハンドラ関数の引数としてポインタ型で使用されます。

なぜ重要か

Apache のフィルタフレームワークにおけるデータ転送の基本単位だからです。フィルタチェーン内を順次通過するデータストリームとして機能し、各フィルタはこれを通じてバケットを順次処理し、データの変換、追加、削除、あるいは SSI（Server Side Includes）における変数展開などを実行します。

関与するモジュール

  • module:include
  • module:modules/filters

主要エンティティ

  • **symbol:include/util_filter.h#apr_bucket_brigade**

    • フィルタ関数（ap_out_filter_func 等）の引数として渡され、処理対象のデータストリームを提供・管理するための定義です。

  • **symbol:modules/filters/mod_include.h#apr_bucket_brigade**

    • SSI フィルタのハンドラシグネチャ（include_handler_fn_t）の引数として使用され、条件分岐や変数展開の結果を格納・転送するために定義されています。

エンドツーエンドの流れ

1. **入力と受け渡し**: フィルタ関数（ap_core_output_filter、ap_byterange_filter、ap_http_header_filter など）や SSI ハンドラ（include_handler_fn_t）が、引数として apr_bucket_brigade を受け取ります。 2. **バケットの処理**: フィルタはブリゲード内のバケットを順次処理し、必要に応じて変換、追加、削除、分割を行います。SSI フィルタ実行時には、ハンドラが include_ctx_t を参照して展開した結果をブリゲードに挿入します。 3. **後続への転送**: 処理されたブリゲードは、ap_pass_brigade を介して後続のフィルタへ渡されます。 4. **終端と解放**: ストリームの終了時（EOS 検出時など）には、各フィルタが内部状態をフラッシュし、リソースを解放します。

*注意: フィルタチェーン全体の詳細なライフサイクルや、すべての具体的なフィルタ間におけるエンドツーエンドの完全なデータフローは、提供されたモジュール情報の範囲内では完全には網羅されていません。*

重要な不変条件

  • **所有権の管理**: ap_pass_brigade を介してデータを渡す際、ブリゲードの所有権は呼び出し元に帰属します。フィルタ内でデータを保持する必要がある場合は、ap_save_brigade を使用して明示的に所有権を管理しなければなりません。
  • **データ転送の基本単位**: フィルタ処理の過程でバケットの挿入、削除、分割が行われることがありますが、一貫してデータ転送の基本コンテナとして機能し続ける必要があります。

*注意: その他の詳細な内部状態や排他制御に関する不変条件は、提供されたコンテキストからは不明です。*

よくある落とし穴

  • **メモリ管理の誤り**: メモリ所有権の管理（ap_save_brigade の適切な使用など）を誤ると、解放済みメモリへのアクセスやメモリリークが発生する原因になります。
  • **エラー伝播の欠落**: メモリ不足やバケット読み込みエラーが発生した際、apr_status_t 型の戻り値を通じて適切にエラーを伝播させないと、システムの不安定化を招きます。
  • **終端（EOS）処理の不備**: ストリームの終端（EOS）に達した際の処理が適切に行われない場合、フィルタチェーンが停止したり、データが欠損したりする可能性があります。
