apr_bucket_brigade

概念

apr_bucket_brigade は、Apache HTTP Server のフィルタチェーンにおいて、データ（バケット）のストリームを保持・管理・転送するための抽象的なコンテナ（バケットの連結リスト）です。

なぜ重要か

Apache のフィルタフレームワークにおけるデータ転送の基本単位であり、フィルタチェーン内を順次通過するデータストリームとして機能します。各フィルタは、このブリゲード内のバケットを順次処理し、必要に応じてデータの変換、追加、削除を行うことで、効率的なストリーム処理を実現します。

関与するモジュール

  • module:include
  • module:modules/filters

主要エンティティ

  • **symbol:include/util_filter.h#apr_bucket_brigade**: フィルタ関数の引数としてポインタ型で使用される、不透明な構造体としての定義です。
  • **symbol:modules/filters/mod_include.h#apr_bucket_brigade**: SSI（Server Side Includes）フィルタのハンドラシグネチャ（include_handler_fn_t）の引数として使用される定義です。

エンドツーエンドの流れ

1. ap_core_output_filter、ap_byterange_filter、ap_http_header_filter などのフィルタ関数が、引数として apr_bucket_brigade を受け取ります。 2. フィルタはブリゲード内のバケット（apr_bucket）を順次処理します。 3. SSI フィルタ（mod_include）の実行時には、登録されたハンドラ（include_handler_fn_t）が include_ctx_t を参照して条件分岐や変数展開を行い、その結果を apr_bucket_brigade に挿入します。 4. 処理されたブリゲードは、ap_pass_brigade を介して後続のフィルタへ渡されます。 5. ストリームの終了時には、各フィルタが内部状態をフラッシュし、リソースを解放します。

※リクエストの受信から最終的なレスポンス送信に至る、完全なエンドツーエンドの全体フローは、提供された情報からはまだ明らかになっていません。

重要な不変条件

  • **所有権の管理**: ap_pass_brigade を介してデータを渡す際、ブリゲードの所有権は呼び出し元に帰属します。フィルタ内でデータを保持（退避）する必要がある場合は、ap_save_brigade を使用して明示的に所有権を管理しなければなりません。
  • **データ転送の基本単位**: フィルタチェーンを流れるデータは、常にこのブリゲード（およびそれを構成するバケット）の形式を維持する必要があります。

よくある落とし穴

  • **メモリ管理の誤り**: 所有権の管理を誤ると、解放済みメモリへのアクセスやメモリリークが発生する可能性があります。
  • **エラーハンドリングの欠如**: フィルタチェーンの処理中やバケットの読み込みエラー、メモリ不足が発生した場合、apr_status_t 型の戻り値を通じて適切にエラーを伝播させないと、システムの不安定化を招きます。
  • **終端処理の不備**: ストリームの終端（EOS）に達した際の処理が適切に行われない場合、フィルタチェーンの停止やデータの欠損が発生する可能性があります。

次に読む場所

  • include/util_filter.h: フィルタ関数の定義や apr_bucket_brigade の基本的な使われ方。
  • modules/filters/mod_include.h: SSI ハンドラ（include_handler_fn_t）におけるブリゲードの具体的な利用方法。
  • apr_buckets.h (APRのバケット管理ドキュメント): ブリゲードの内部構造や操作関数（バケットの作成、分割、削除など）の詳細。
