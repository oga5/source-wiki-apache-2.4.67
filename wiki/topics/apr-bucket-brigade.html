apr_bucket_brigade

概念

apr_bucket_brigade は、Apache HTTP Server のフィルタチェーンにおいて、データ（バケット）のストリームを保持、管理、および転送するための抽象的なコンテナ（バケットの連結リスト）です。

なぜ重要か

apr_bucket_brigade は、Apache のフィルタフレームワークにおけるデータ転送の基本単位です。各フィルタは、このブリゲードを介してデータを受け取り、変換、追加、削除などの操作を施した上で後続のフィルタへ伝播させます。これにより、効率的かつ柔軟なストリームデータのパイプライン処理が可能になります。

関与するモジュール

  • module:include
  • module:modules/filters

主要エンティティ

  • **symbol:include/util_filter.h#apr_bucket_brigade**: フィルタ関数（ap_out_filter_func など）の引数として使用され、フィルタチェーン内でのデータ操作や所有権管理の基準を定義します。
  • **symbol:modules/filters/mod_include.h#apr_bucket_brigade**: Server Side Includes (SSI) フィルタにおいて、ハンドラ関数（include_handler_fn_t）が条件分岐や変数展開の結果を格納・出力するために使用されます。

エンドツーエンドの流れ

1. **入力と受け渡し**: ap_core_output_filter、ap_byterange_filter、ap_http_header_filter などのフィルタ関数が、引数として apr_bucket_brigade を受け取ります。 2. **フィルタ処理**:

  • 各フィルタはブリゲード内のバケット（apr_bucket）を順次処理します。
  • mod_include では、登録されたハンドラが include_ctx_t を参照して処理を行い、その結果をブリゲードに挿入します。

    • 3. **後続への伝達**: 処理されたブリゲードは、ap_pass_brigade を通じて次のフィルタへと渡されます。 4. **終端処理**: ストリームの終了時（EOS）には、各フィルタが内部状態をフラッシュし、リソースを解放します。

*注意: フィルタチェーン全体における具体的なバケットの生成から最終的なネットワーク送信に至る完全なエンドツーエンドのライフサイクルは、提供されたコンテキストからは一部未検証です。*

重要な不変条件

  • **所有権の帰属**: ap_pass_brigade を介してデータを渡す際、ブリゲードの所有権は呼び出し元に帰属します。
  • **データの保持**: フィルタ内でデータを保持（退避）する必要がある場合は、ap_save_brigade を使用して明示的に所有権を管理しなければなりません。

よくある落とし穴

  • **メモリ管理の誤り**: メモリ所有権の管理を誤ると、解放済みメモリへのアクセスやメモリリークが発生する原因になります。
  • **EOS（終端）処理の不備**: ストリームの終端（EOS）に達した際の処理が適切に行われない場合、フィルタチェーンが停止したり、データが欠損したりする可能性があります。
  • **エラー伝播の無視**: バケットの読み込みエラーやメモリ不足が発生した際、apr_status_t 型の戻り値によるエラー伝播を適切に処理しないと、システムの不安定化を招きます。

次に読む場所

  • **include/util_filter.h**: フィルタ関数のシグネチャや、ブリゲードを扱う基本APIが定義されています。
  • **modules/filters/mod_include.h**: SSI フィルタにおけるブリゲードの具体的な利用方法（include_handler_fn_t）が定義されています。
  • **apr_buckets.h** (外部依存): バケットおよびブリゲードの内部構造や、詳細な操作関数群（作成、分割、削除など）を理解するために参照する必要があります。
