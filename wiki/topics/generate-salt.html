generate_salt

概念

generate_salt は、パスワードハッシュ生成時に使用されるランダムなソルト文字列を生成するためのユーティリティ関数です。APR（Apache Portable Runtime）の乱数生成機能を利用して安全な乱数を取得し、それを特定のBase64風文字セット（itoa64）を用いてエンコードした上で、指定されたバッファに格納します。

なぜ重要か

セキュアなパスワードハッシュを生成するためには、予測不可能なソルトが不可欠です。この関数は、暗号論的に安全な乱数生成（apr_generate_random_bytes）と、ハッシュアルゴリズムと互換性のあるBase64風エンコーディングを組み合わせることで、一貫したソルト生成処理を提供します。

エンドツーエンドの流れ

1. **呼び出し**: パスワードハッシュ生成処理（mk_password_hash や mkhash など）から、必要なソルトの長さ（size）を指定して generate_salt が呼び出されます。 2. **乱数生成**: 関数内部で size に応じた必要バイト数 n を算出し、apr_generate_random_bytes を用いて安全な乱数バイト列を取得します。 3. **エンコード**: 取得した乱数バイト列を、itoa64 テーブル（./0-9A-Za-z...）を用いて6ビットずつBase64風の文字列に変換します。 4. **出力**: 変換された文字列を指定されたバッファ s に格納し、ヌル終端します。

※なお、呼び出し元からハッシュ生成、そして最終的な認証処理に至る完全なエンドツーエンドのフローの全容は、提供されたコンテキストからは完全には明らかになっていません。

重要な不変条件

  • **バッファサイズ制限**: 生成に必要なバイト数 n は、内部で確保されている一時バッファ rnd のサイズ（32バイト）を超えてはなりません。
  • **バッファの確保**: 呼び出し元が渡すバッファ s は、少なくとも size + 1 バイト以上の領域が確保されている必要があります。

よくある落とし穴

  • **エラーハンドリングの不一致**:
  • support/passwd_common.c の実装では、要求された size に基づく必要バイト数が32バイトを超えた場合、エラーを返すのではなく abort() を呼び出してプロセスを強制終了します。
  • 一方、modules/lua/lua_passwd.c の実装では、同様のケースで ERR_RANDOM を返します。
  • **非標準のBase64**: 使用されている itoa64 変換テーブルは標準的なBase64とは異なるため、一般的なBase64デコーダで直接デコードすることはできません。
  • **グローバル変数への依存**: support/passwd_common.c の実装はグローバル変数 errfile に依存しているため、呼び出し前にこれが適切に初期化されている必要があります。
