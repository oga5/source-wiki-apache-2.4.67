content_type

概念

content_type は、Apache HTTP Server において HTTP の Content-Type ヘッダや MIME タイプの情報を表現・処理するために使用される概念です。 具体的には、mod_mime モジュール内部で MIME タイプ（メディアタイプ、サブタイプ、および関連パラメータ）を構造化して保持するための内部データ構造 content_type 構造体として定義されています。また、サポートツールである ab.c にも関連するシンボルが存在します。

なぜ重要か

HTTP 通信において、コンテンツのデータ型を正しく識別・処理することは不可欠です。mod_mime において content_type 構造体を用いることで、Content-Type ヘッダ文字列をメディアタイプ、サブタイプ、パラメータに分解し、個別に正規化やパラメータ処理を行うことが容易になります。これにより、MIME タイプの正確な解析とハンドリングが可能になります。

関与するモジュール

  • module:modules/http
  • module:support

主要エンティティ

  • symbol:modules/http/mod_mime.c#content_type: mod_mime 内部で MIME タイプの要素（type, subtype, param など）を保持するための構造体。
  • symbol:support/ab.c#content_type: ab（ApacheBench）ツール内に存在する関連シンボル（詳細な構造や役割は現時点で未定義）。

エンドツーエンドの流れ

mod_mime における解析の流れは以下の通りですが、システム全体にわたる詳細なエンドツーエンドの流れは現時点では完全に明らかになっていません。

1. **ヘッダの解析**: mod_mime.c 内の analyze_ct などの関数が、Content-Type ヘッダ文字列を解析します。 2. **構造体への格納**: 解析されたメディアタイプ、サブタイプ、およびそれぞれの長さ（type_len, subtype_len）、パラメータへのポインタ（param）が content_type 構造体に格納されます。 3. **パラメータ処理**: 構造体に格納された情報を基に、後続のパラメータ処理や正規化が行われます。

なお、support/ab.c における content_type の処理フローとの連携や、リクエスト処理全体における詳細なライフサイクルは現時点では不明です。

重要な不変条件

  • content_type 構造体の type および subtype フィールドは、メモリプール（apr_pool_t）によって管理された有効な文字列を指している必要があります。
  • type_len および subtype_len は、それぞれ type および subtype が指す文字列の実際の長さと正確に一致していなければなりません。
  • その他のシステム全体における不変条件は現時点では不明です。

よくある落とし穴

  • **request_rec->content_type との混同**: Apache のリクエスト構造体である request_rec には content_type という同名のフィールド（const char * 型）が存在しますが、これは最終的な HTTP ヘッダ値を保持する文字列であり、mod_mime.c で定義されている content_type 構造体とは直接的な型の一致はありません。これらを混同しないよう注意が必要です。
  • **バッファオーバーランのリスク**: type_len や subtype_len が実際の文字列長と一致しない場合、後続の文字列操作においてバッファオーバーランや不正なメモリ参照が発生するリスクがあります。
