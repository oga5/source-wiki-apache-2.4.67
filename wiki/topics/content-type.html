content_type

概念

content_type は、Apache HTTP Server（httpd）において HTTP メッセージのメディアタイプ（MIME タイプ）を表現・解析するために使用される概念およびデータ構造です。

modules/http/mod_mime.c においては、メディアタイプ、サブタイプ、および関連するパラメータを構造化して保持する内部構造体（struct）として定義されています。また、support/ab.c などのサポートツール群にも同名のシンボルが存在します。

なぜ重要か

HTTP 通信において、ペイロードのデータ形式を正しく識別・処理することは不可欠です。mod_mime では、Content-Type ヘッダ文字列を単なる文字列としてではなく、タイプ、サブタイプ、パラメータに分解して構造化（content_type 構造体）することで、正規化や後続のパラメータ処理を効率的かつ正確に行うことができます。

関与するモジュール

  • module:modules/http
  • module:support

主要エンティティ

  • **symbol:modules/http/mod_mime.c#content_type**: mod_mime.c 内で定義されている構造体。type、subtype、および param リストへのポインタを保持し、ヘッダ解析の中間表現として機能します。
  • **symbol:support/ab.c#content_type**: support/ab.c に存在するシンボル（詳細な構造や役割は本コンテキストからは不明）。

エンドツーエンドの流れ

mod_mime における処理の流れは以下の通りです。 1. mod_mime.c 内の解析処理（analyze_ct 関数など）が、対象となる Content-Type ヘッダ文字列を読み込みます。 2. 文字列からメディアタイプ、サブタイプ、およびパラメータが抽出され、content_type 構造体の各フィールド（type, subtype, param など）に分解・格納されます。

なお、support/ab.c における content_type の具体的な利用フローや、これら2つのモジュール間を跨ぐエンドツーエンドの相互作用については、現時点では不明です。

重要な不変条件

  • mod_mime.c の content_type 構造体において、type および subtype フィールドは、メモリプール（apr_pool_t）によって管理された有効な文字列を指している必要があります。
  • type_len および subtype_len は、それぞれ対応する文字列の実際の長さと一致していなければなりません。
  • support/ab.c における content_type の不変条件については、現時点では不明です。

よくある落とし穴

  • **request_rec->content_type との混同**: mod_mime 内部の解析用構造体である content_type と、Apache のリクエスト情報を表す request_rec 構造体内の content_type フィールド（こちらは最終的なヘッダ値を保持する const char * 型の文字列）は、名前が同じですが型も用途も異なります。混同しないよう注意が必要です。
  • **バッファオーバーランのリスク**: type_len や subtype_len が実際の文字列長と一致しない場合、後続の文字列操作においてバッファオーバーランや不正なメモリ参照を引き起こす可能性があります。
