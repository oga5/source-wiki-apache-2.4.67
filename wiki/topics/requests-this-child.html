requests_this_child

概念

requests_this_child は、Apache HTTP Serverのマルチプロセッシングモジュール（MPM）において、個々の子プロセス（child process）が起動してから現在までに処理したリクエストの累計数を保持・追跡するための変数です。

なぜ重要か

この変数は、子プロセスの寿命（ライフサイクル）を管理するために極めて重要です。Apacheでは、メモリリークの蓄積を防ぐなどの目的で、1つの子プロセスが一定数のリクエストを処理した後に自動的に終了・再生成（リサイクル）される仕組み（MaxConnectionsPerChild 設定など）があります。requests_this_child は、その制限値に達したかどうかを判定するための基準値として機能します。

関与するモジュール

  • module:server/mpm/prefork
  • module:server/mpm/worker

主要エンティティ

  • symbol:server/mpm/prefork/prefork.c#requests_this_child: prefork MPMにおいて、各子プロセスが処理したリクエスト数をカウントする変数。
  • symbol:server/mpm/worker/worker.c#requests_this_child: worker MPMにおいて、各子プロセス（およびその配下のスレッド群）が処理したリクエストの総数をカウントする変数。

エンドツーエンドの流れ

1. **初期化**: 子プロセスがフォーク（起動）された際、requests_this_child は 0 に初期化されます。 2. **リクエストの処理とインクリメント**:

  • prefork では、子プロセスがリクエストを1件処理するたびに、この変数がインクリメントされます。
  • worker では、複数のワーカースレッドがリクエストを処理し、共通の requests_this_child カウンタをインクリメントします。

    • 3. **限界値の判定**: 各リクエスト処理の終了時、またはループの評価時に、requests_this_child が設定された最大接続数（MaxConnectionsPerChild）に達したかどうかがチェックされます。 4. **プロセスの終了**: 制限に達した場合、子プロセスは新規接続の受け入れを停止し、現在処理中のリクエストを完了させた後に安全に終了します。

*注意: 具体的な排他制御（アトミック操作やミューテックス）の有無や、シグナルハンドリングを伴う詳細な終了シーケンスのコードレベルでの完全なエンドツーエンドのフローは、提供されたメタデータからは完全には特定されていません。*

重要な不変条件

  • requests_this_child は、子プロセスの生存期間中、単調非減少（減少することはない）です。
  • MaxConnectionsPerChild が 0（無制限）に設定されている場合、この値がいくら増加しても子プロセスがリクエスト数制限によって自動終了することはありません。

*注意: スレッド間での同期やメモリバリアに関する厳密な不変条件は、提供されたメタデータからは完全には特定されていません。*

よくある落とし穴

  • **マルチスレッド環境での競合**: worker MPMなどのマルチスレッド環境では、複数のスレッドが同時に requests_this_child を更新しようとするため、適切なアトミック操作やロックによる保護が行われていない場合、カウントの不整合（ロストアップデート）が発生する可能性があります。
  • **早期終了**: 設定された最大値に達する前に、メモリ不足や親プロセスからのシグナル（Graceful Restartなど）によってプロセスが終了する場合、requests_this_child が最大値に達することはありません。
