requests_this_child

概念

requests_this_child は、Apache HTTP Serverのマルチプロセッシングモジュール（MPM）において、**現在の子プロセスが起動してから処理した累積リクエスト数**を追跡・カウントするための変数（シンボル）です。

なぜ重要か

Apache HTTP Serverでは、長期運用に伴うメモリリークの防止やリソースの健全性を維持するために、1つの子プロセスが処理できる最大リクエスト数（MaxConnectionsPerChild 設定値）に制限を設ける仕組みがあります。requests_this_child は、この制限値に達したかどうかを各子プロセスが自己判定し、適切に引退（終了）プロセスへ移行するために不可欠なカウンター役割を担っています。

関与するモジュール

  • module:server/mpm/worker

主要エンティティ

  • symbol:server/mpm/prefork/prefork.c#requests_this_child
  • シングルスレッド・マルチプロセスモデルである prefork MPM 内で、個別プロセスが処理したリクエスト数を管理します。
  • symbol:server/mpm/worker/worker.c#requests_this_child
  • マルチスレッド・マルチプロセスモデルである worker MPM 内で、プロセス内の全スレッドが共同で処理した合計リクエスト数を管理します。

エンドツーエンドの流れ

1. **初期化**: 子プロセスの起動時、またはメインループの開始時に requests_this_child が 0 に初期化されます。 2. **インクリメント**: 子プロセス（またはその配下のスレッド）が接続を受け入れ、リクエストの処理を完了するたびに、このカウンターがインクリメントされます。 3. **限界値の評価**: インクリメント後、設定された最大リクエスト数（ap_max_requests_per_child など）に達したかどうかが評価されます。 4. **プロセスの終了**: 制限に達した場合、子プロセスは新規接続の受け入れを停止し、既存の処理を終えた後に安全に終了します。

*注意: 提供されたコンテキストからは、具体的なインクリメントのタイミングや、シグナルハンドリングを伴う終了処理の厳密なコードレベルでのエンドツーエンドフローは完全には特定されていません。*

重要な不変条件

  • requests_this_child は、設定された最大接続数（ap_max_requests_per_child）に達するまで単調増加します。
  • prefork MPM ではプロセスごとに独立してカウントされます。

*注意: スレッドセーフなアトミック操作の有無や、特定条件下でのリセット処理など、詳細な不変条件は提供されたコンテキストからは完全には特定されていません。*

よくある落とし穴

  • **マルチスレッド環境での競合**: worker MPM のようなマルチスレッドモデルでは、複数のワーカースレッドが同時に requests_this_child を更新しようとするため、適切な排他制御（アトミック操作やロック）が行われていない場合、カウントの不整合（ロストアップデート）が発生する可能性があります。
  • **設定値 0（無制限）の扱い**: MaxConnectionsPerChild が 0（無制限）に設定されている場合、このカウンターによるプロセス終了のトリガーをバイパスする必要があります。

*注意: 実際のソースコードにおける具体的な競合対策やバイパスロジックの有無は、提供されたコンテキストからは完全には特定されていません。*
