sei_entry

概念

sei_entry は、Apache HTTP Server (httpd) において、特定の条件に基づいて環境変数を動的に設定・操作するためのルールを保持する構造体です。

ただし、この名前の構造体は mod_setenvif モジュールと mod_proxy_fcgi モジュールの2箇所で個別に定義されており、それぞれ異なる役割とデータ構造を持っています。

  • **mod_setenvif.c における sei_entry**: SetEnvIf ディレクティブによる条件ルール（対象ヘッダ、正規表現、環境変数テーブルなど）を保持します。
  • **mod_proxy_fcgi.c における sei_entry**: ProxyFCGISetEnvIf などのディレクティブによる条件付き環境変数設定（条件式、置換式、環境変数名）を管理します。

なぜ重要か

リクエストの属性、ヘッダ、あるいはカスタムの条件式に基づいて環境変数を動的に制御することは、柔軟なリクエスト処理や、バックエンド（FastCGIなど）へ適切なパラメータを伝達するために極めて重要です。sei_entry は、これらの条件評価ルールを設定フェーズから実行フェーズへと受け渡すためのコアデータ構造として機能します。

関与するモジュール

  • module:modules/metadata (mod_setenvif.c)
  • module:modules/proxy (mod_proxy_fcgi.c)

主要エンティティ

  • **symbol:modules/metadata/mod_setenvif.c#sei_entry**:

    • SetEnvIf 用の構造体。ヘッダ名（name）、コンパイル済み正規表現（preg）、高速マッチング用パターン（pattern）、解析済み式（expr）、および設定・解除する環境変数のテーブル（features）などを保持します。

  • **symbol:modules/proxy/mod_proxy_fcgi.c#sei_entry**:

    • ProxyFCGISetEnvIf 用の構造体。条件式（cond）、置換式（subst）、および対象の環境変数名（envname）を保持します。

エンドツーエンドの流れ

両モジュールにおける sei_entry がシステム全体としてどのように相互作用するか、あるいは完全に独立して動作するかといった、統合されたエンドツーエンドのフローの全貌は現時点では明らかになっていません。

ただし、それぞれのモジュール内における個別の処理の流れは以下の通りです。

mod_setenvif における流れ

1. **設定フェーズ**: add_setenvif_core 関数が SetEnvIf ディレクティブを解析し、sei_entry 構造体を生成・構成します。 2. **実行フェーズ**: リクエスト処理の初期段階で match_headers 関数が呼び出され、sei_entry に格納された条件（正規表現や式）を評価します。条件に合致した場合、リクエストの subprocess_env テーブルが更新されます。

mod_proxy_fcgi における流れ

1. **設定フェーズ**: ProxyFCGISetEnvIf 等のディレクティブにより sei_entry が生成され、配列（apr_array_header_t）に格納されます。 2. **実行フェーズ**: リクエスト処理時に保持された sei_entry のリストを順次走査し、条件式（cond）が真と評価された場合に置換式（subst）を評価して環境変数（envname）に値を割り当てます。

重要な不変条件

  • **mod_setenvif.c の不変条件**:
  • 正規表現を使用する場合、preg には ap_pregcomp でコンパイルされた有効なオブジェクトが格納されていなければなりません。
  • 単純文字列マッチングの場合、pattern に apr_strmatch_precompile で生成されたパターンが格納されている必要があります。
  • **mod_proxy_fcgi.c の不変条件**:
  • cond および subst は、Apache の式評価エンジン（ap_expr）によってコンパイルされた有効な式情報（ap_expr_info_t）である必要があります。
  • **モジュール間の不変条件**:
  • 両モジュールで同名の構造体が定義されていますが、これらがバイナリレベルでどのように互換性を保っているか、あるいは完全に独立したメモリ空間として管理されているかの詳細な不変条件は未確定です。

よくある落とし穴

  • **評価タイミングの制限**:

    • mod_setenvif はリクエスト処理の初期段階で実行されるため、後続のモジュールが動的に生成したヘッダを対象に SetEnvIf で評価することはできません。

  • **設定時のエラー**:

    • 正規表現のコンパイルや Apache 式（ap_expr）の解析に失敗した場合、リクエスト処理時ではなく、設定読み込み（サーバー起動・再起動）時にエラーが発生します。

  • **同名構造体の混同**:

    • mod_proxy_fcgi.c と mod_setenvif.c の両方に sei_entry という同名の構造体が存在しますが、メンバ構成が異なります。ソースコードの修正や拡張を行う際、これらを混同するとコンパイルエラーや予期しないメモリ破損を引き起こす可能性があります。
