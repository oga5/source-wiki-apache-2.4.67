Module: modules/md

概念

modules/md (Managed Domains) は、Apache HTTP Server において SSL/TLS 証明書の自動管理（ACME プロトコルによる取得、更新、管理など）を実現するためのモジュールおよび関連ライブラリ群です。

このモジュールは、ACME クライアント機能（md_acme.c）、暗号処理（md_crypt.c）、JSON 処理（md_json.c）、HTTP 通信（md_curl.c, md_http.c）、およびローカルストレージ管理（md_store.c, md_store_fs.c）などのコンポーネントを内包し、Web サーバー自体が自律的に証明書のライフサイクルを管理できるように設計されています。

なぜ重要か

従来の SSL/TLS 証明書の管理では、外部ツール（Certbot など）を用いた手動または半自動の更新設定と、Web サーバーの再起動が必要でした。modules/md を導入することで、以下のメリットが得られます。

  • **自動化されたライフサイクル**: ACME プロトコル（Let's Encrypt など）を介して、証明書の申請、検証、取得、更新を Web サーバープロセス内で完結させます。
  • **運用の堅牢性**: OCSP ステープリング（md_ocsp.c, mod_md_ocsp.c）の自動管理や、証明書の期限切れに伴うサービス停止リスクの排除。
  • **設定の簡素化**: Apache の設定ファイル（mod_md_config.c）内で管理対象ドメイン（Managed Domain）を直接定義できます。

関与するモジュール

  • module:modules/md (Apache HTTP Server の mod_md モジュールおよび関連するコアロジック)

主要エンティティ

提供されたファイル群から、以下の主要なエンティティおよびコンポーネントが定義されていることが確認できます。

  • **Managed Domain (MD) コア**: md.h, md_core.c
  • 管理対象ドメインの基本構造やライフサイクルを定義します。
  • **ACME プロトコルハンドラ**: md_acme.h, md_acme_acct.h, md_acme_authz.h, md_acme_order.h, md_acmev2_drive.h
  • ACME サーバーとのアカウント管理、ドメイン所有権の認可（Authorization）、証明書注文（Order）のフローを制御します。
  • **暗号・署名ユーティリティ**: md_crypt.h, md_jws.h
  • 鍵ペアの生成、証明書署名要求 (CSR) の作成、および ACME サーバーとの通信に必要な JSON Web Signature (JWS) の処理を行います。
  • **ストレージマネージャ**: md_store.h, md_store_fs.h
  • 取得した証明書、秘密鍵、および ACME アカウント情報をファイルシステム上に安全に永続化します。
  • **HTTP クライアント**: md_http.h, md_curl.h
  • libcurl 等を利用して ACME サーバーと通信するための抽象化レイヤーです。
  • **Apache 統合レイヤー**: mod_md.h, mod_md_config.h, mod_md_drive.h
  • Apache の設定ディレクティブのパースや、バックグラウンドでの証明書更新タスクのスケジューリングを行います。

エンドツーエンドの流れ

具体的な内部関数呼び出しや詳細なシーケンスは、提供されたファイル構造からは完全には特定されていませんが、ソースコードの構成から以下の流れが推測されます。

1. **設定のロード**: Apache 起動時に mod_md_config.c が設定ファイルをパースし、管理対象ドメイン（MD）のリストを構築します。 2. **状態検証と監視**: mod_md_drive.c がバックグラウンドで起動し、各 MD の証明書の有効期限やステータスをチェックします。 3. **ACME フローの開始**: 更新が必要な場合、md_acme_drive.c / md_acmev2_drive.c が起動し、md_acme_acct.c を用いてアカウントを確保します。 4. **チャレンジと認可**: md_acme_authz.c を介してドメイン所有権の検証（チャレンジ）を実行し、md_acme_order.c で証明書を要求します。 5. **保存と適用**: 新しい証明書が発行されると、md_store_fs.c を介してディスクに書き込まれ、Apache に動的または再起動を伴って適用されます。

重要な不変条件

詳細なコードレベルの不変条件（Invariants）は、提供されたファイル一覧からは未検証です。設計上、以下の不変条件が維持される必要があります。

  • **秘密鍵の秘匿性**: md_crypt.h および md_store_fs.h によって生成・保存される秘密鍵は、適切なファイルパーミッション（例: 所有者のみ読み書き可能）で厳重に保護されなければなりません。
  • **通信の整合性**: ACME サーバーへのすべての状態変更リクエストは、md_jws.h に基づき、登録されたアカウントキーで正しく署名されていなければなりません。

よくある落とし穴

具体的なランタイムエラーや落とし穴の詳細は、提供された静的ファイル構成からは特定されていません。一般的に以下の点に留意する必要があります。

  • **ネットワーク制限**: md_curl.c を介した外部 ACME サーバーへの通信が、ファイアウォールやプロキシ設定によって遮断されるケース。
  • **書き込み権限の不足**: Apache の実行ユーザーが、md_store_fs.c の指定するストレージディレクトリに対して適切な書き込み権限を持っていない場合、証明書の保存に失敗します。
  • **チャレンジの到達性**: HTTP-01 チャレンジなどの検証時に、外部からの HTTP リクエストが対象の Apache サーバーに正しくルーティングされない場合、認可（Authorization）が失敗します。

次に読む場所

  • modules/md/mod_md.c: Apache モジュールとしてのエントリポイント、フック、および初期化処理。
  • modules/md/md.h: Managed Domain のコアデータ構造と基本 API。
  • modules/md/md_store.h: 証明書や鍵の永続化に関する抽象化インターフェース。
  • modules/md/md_acme.h: ACME プロトコルクライアントの主要なインターフェース定義。
