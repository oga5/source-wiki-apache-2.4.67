Multi-Processing Modules (MPM) Concurrency Model

概念

Multi-Processing Modules (MPM) は、Apache HTTP Server においてネットワークポートのバインド、リクエストの受け付け、およびそれらを処理する子プロセスやスレッドの管理といった並行処理モデル（Concurrency Model）をカプセル化するアーキテクチャです。システム要件やリソース特性に応じて、プロセスベース（prefork）、スレッド/プロセスハイブリッド（worker）、非同期イベント駆動（event）などの異なる並行処理モデルを選択できます。

なぜ重要か

Webサーバーのパフォーマンス、スケーラビリティ、およびリソース消費効率は、採用する並行処理モデルに直接依存します。例えば、接続維持コストを最小化する event MPM や、プロセスの隔離性を重視する prefork MPM など、動作環境に適したリソース管理を行うために、MPM による並行処理の制御は極めて重要です。

主要エンティティ

  • **symbol:server/mpm/event/event.c#child_main**: 子プロセスのメイン処理を担当する関数。スレッドの初期化、リスナースレッドの生成、および終了時のクリーンアップ（clean_child_exit など）を統制します。
  • **symbol:server/mpm/event/event.c#create_listener_thread**: 接続要求を待ち受けるためのリスナースレッドを生成する関数。
  • **symbol:server/mpm/event/event.c#perform_idle_server_maintenance**: アイドル状態のサーバープロセス数を監視し、必要に応じて子プロセスの生成（make_child）や不要なプロセスの終了を指示するメンテナンス関数。
  • **symbol:server/mpm/event/event.c#server_limit**: 起動可能な最大プロセス数を制限する設定値。
  • **symbol:server/mpm/event/event.c#thread_limit**: プロセスあたりに生成可能な最大スレッド数を制限する設定値。

エンドツーエンドの流れ

提供されたシンボル情報から得られる部分的な処理の流れは以下の通りですが、詳細なライフサイクルやリクエスト処理の全体フローは**未解明**です。

1. **プロセスの維持管理**: メインループ等から perform_idle_server_maintenance が定期的に呼び出され、負荷状況に応じて make_child を介して新しい子プロセスをフォークします。 2. **子プロセスの初期化**: 生成された子プロセスは child_main を実行し、シグナル設定やスコアボードの再オープンを行います。 3. **スレッドの起動**: child_main 内から create_listener_thread が呼び出され、クライアントからの接続を監視するリスナースレッドが起動します。 4. **プロセスの終了**: 終了シグナルやメンテナンス要求を検知すると、signal_threads や join_workers を経て、clean_child_exit により安全にプロセスが終了します。

重要な不変条件

現時点では、詳細な不変条件（状態遷移の整合性や排他制御のルールなど）は**未解明**です。ただし、システム全体の安定性を維持するために、server_limit や thread_limit といった上限値を超えてプロセスやスレッドが生成されないように制御する不変条件が存在することが示唆されています。

よくある落とし穴

現時点では、本コードベースにおける具体的な失敗モードや競合状態などの落とし穴は**未解明**です。一般的には、server_limit や thread_limit の設定値の不整合、あるいはシグナルハンドリングの不備による子プロセスのクリーンアップ失敗（ゾンビプロセスの発生やリソースリーク）などが境界条件として懸念されます。
