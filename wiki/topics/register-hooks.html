register_hooks

概念

register_hooks は、Apache HTTP Server (httpd) の各モジュールが起動・初期化フェーズにおいて、自身の機能（リクエストハンドラ、認証・認可プロバイダー、入力/出力フィルタ、特定の処理フェーズで実行されるフック関数、およびオプション関数など）をサーバーコアや他のモジュールに登録するための標準的なエントリーポイント（コールバック関数）です。

なぜ重要か

Apache HTTP Server は極めてモジュール化されたアーキテクチャを採用しており、コア自体は最小限の機能しか提供しません。register_hooks は、各モジュールがサーバーのライフサイクル（設定読み込み、プロセス起動、接続確立、リクエスト解析、コンテンツ生成、ログ記録など）の適切なタイミングに介入するための「窓口」として機能します。この関数を通じてフックやプロバイダーを登録することで、モジュールはサーバー全体の挙動を拡張・カスタマイズすることができます。

エンドツーエンドの流れ

1. **モジュールのロードと初期化**: Apache の起動時、または動的モジュールロード（mod_so）時に、各モジュールの module 構造体（AP_DECLARE_MODULE マクロ等で定義）が読み込まれます。 2. **register_hooks の呼び出し**: Apache コアは、登録された各モジュールの register_hooks 関数を、メモリプール（apr_pool_t *）を引数として呼び出します。 3. **各種機能の登録**:

  • **ハンドラの登録**: ap_hook_handler を使用して、特定のリクエストを処理するハンドラ（例: asis_handler, status_handler）を登録します。
  • **プロバイダーの登録**: ap_register_provider や ap_register_auth_provider を使用して、認証・認可・キャッシュなどのプロバイダー（例: anon, dbm, file, dc）を登録します。
  • **フィルタの登録**: ap_register_output_filter や ap_register_input_filter を使用して、データの入出力を加工するフィルタ（例: buffer_out_filter, rate_limit_filter）を登録します。
  • **ライフサイクルフックの登録**: ap_hook_pre_config, ap_hook_post_config, ap_hook_child_init, ap_hook_translate_name, ap_hook_fixups などの API を使用して、特定の処理フェーズで実行されるコールバック関数を登録します。
  • **オプション関数の登録・取得**: APR_REGISTER_OPTIONAL_FN で自モジュールの関数を公開し、ap_hook_optional_fn_retrieve で他モジュールの関数を取得します。

    • 4. **リクエスト処理時の実行**: 登録されたフックやハンドラは、リクエストが到着した際、指定された優先順位（APR_HOOK_FIRST, APR_HOOK_MIDDLE, APR_HOOK_LAST など）に従って順次実行されます。

重要な不変条件

  • **シグネチャの不変性**: すべての register_hooks 関数は static void register_hooks(apr_pool_t *p) のシグネチャを持ち、戻り値を返しません。
  • **実行タイミングの制限**: register_hooks はサーバーの起動・設定初期化フェーズでのみ呼び出されます。リクエスト処理中に動的にフックを追加することはできません。
  • **優先順位と依存関係の明示**: フック登録時、先行・後続すべきモジュールがある場合は、aszPre や aszSucc 配列（例: mod_userdir.c や mod_vhost_alias.c での mod_alias.c との順序制御）を用いて依存関係を明示する必要があります。

よくある落とし穴

  • **優先順位（APR_HOOK_*）の誤設定**:
  • APR_HOOK_LAST を指定すると、他のすべてのハンドラが処理を辞退（DECLINED）した後にのみ実行されます（例: mod_example_1.c）。意図せず他のモジュールに処理を奪われる原因になります。
  • 逆に APR_HOOK_FIRST を指定すると、他のモジュールの処理を阻害する可能性があります。
  • **他モジュールとのロード順序依存**:
  • オプション関数の取得（ap_hook_optional_fn_retrieve）や、他モジュールが提供するプロバイダーへの依存がある場合、初期化順序が正しく制御されていないと、関数ポインタが NULL になりセグメンテーションフォールトや機能不全を引き起こします（例: mod_authz_groupfile.c が mod_authz_owner に依存するケース）。
  • **プラットフォーム依存の条件分岐**:
  • HAVE_UNIX_SUEXEC（mod_userdir.c）や HAVE_TIMES（mod_status.c）などのプリプロセッサマクロによる条件分岐がある場合、ビルド環境によって登録されるフックが異なるため、ポータビリティに配慮する必要があります。
