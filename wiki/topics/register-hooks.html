register_hooks

概念

register_hooks は、Apache HTTP Server (httpd) の各モジュールにおいて、サーバーの起動・初期化フェーズ中に自身の機能（リクエストハンドラ、フィルタ、認証・認可プロバイダー、オプション関数など）をサーバーのフックシステムやプロバイダーレジストリに登録するためのエントリーポイント（コールバック関数）です。

各モジュールは、module 構造体を通じてこの関数を公開し、Apache コアがモジュールロード時にこれを呼び出します。

なぜ重要か

Apache HTTP Server は極めてモジュール化されたアーキテクチャを採用しており、コア自体は最小限の機能しか提供しません。リクエスト処理パイプライン（名前翻訳、アクセス制御、認証、認可、フィルタ適用、コンテンツ生成、ログ記録など）の各フェーズにモジュールが介入するためには、この register_hooks を介して適切なフック（例: ap_hook_handler, ap_hook_translate_name, ap_hook_fixups など）やプロバイダー（例: ap_register_auth_provider, ap_register_provider）を登録する必要があります。これにより、モジュールはサーバーのライフサイクルやリクエスト処理の特定のタイミングで自身のコードを実行させることができます。

エンドツーエンドの流れ

1. **サーバー起動とモジュールロード** Apache HTTP Server の起動時、または動的モジュールロード（mod_so）時に、各モジュールの module 構造体が読み込まれます。 2. **フック登録フェーズの実行** Apache コアは、ロードされたすべてのモジュールに対して、その module 構造体に定義されている register_hooks 関数ポインタを呼び出します。 3. **フック・プロバイダーの登録** 各モジュールの register_hooks 内部で、以下のような登録処理が実行されます。

  • ap_hook_*（例: ap_hook_handler, ap_hook_translate_name, ap_hook_fixups）を呼び出し、特定の処理フェーズにコールバック関数を登録する。
  • ap_register_provider または ap_register_auth_provider を呼び出し、認証・認可、キャッシュ、プロキシなどのプロバイダーを登録する。
  • ap_register_input_filter / ap_register_output_filter を呼び出し、フィルタを登録する。
  • APR_REGISTER_OPTIONAL_FN を呼び出し、他のモジュールから動的に取得可能なオプション関数を公開する。

    • 4. **リクエスト処理時の実行** 登録されたフックやプロバイダーは、実際のリクエスト処理パイプラインにおいて、指定された優先順位（APR_HOOK_FIRST, APR_HOOK_MIDDLE, APR_HOOK_LAST など）や依存関係（先行・後続モジュールの指定）に従って順次呼び出されます。

重要な不変条件

  • **シグネチャの不変性**

    • すべての register_hooks 関数は static void register_hooks(apr_pool_t *p) という同一のシグネチャを持ちます。

  • **実行フェーズの限定**

    • 本関数はサーバーの初期化・設定フェーズ（リクエスト処理が開始される前）にのみ呼び出され、リクエスト処理中に動的にフックを追加することはできません。

  • **優先順位と依存関係の指定**

    • フック登録時には、実行優先順位（APR_HOOK_FIRST, APR_HOOK_MIDDLE, APR_HOOK_LAST など）や、特定のモジュールの前後に実行されるべきという依存関係（例: mod_userdir.c における mod_alias.c の後、mod_vhost_alias.c の前という指定）が明示的に定義されます。

よくある落とし穴

  • **優先順位の競合**

    • 複数のモジュールが同じフェーズに APR_HOOK_LAST や APR_HOOK_FIRST を指定した場合、それらの間の相対的な実行順序は保証されません。明示的な順序制御が必要な場合は、先行・後続モジュールの配列（aszPre / aszSucc）を適切に設定する必要があります。

  • **オプション関数の取得タイミング**

    • ap_hook_optional_fn_retrieve を使用して他のモジュールが提供するオプション関数を取得する場合、その取得処理はすべてのモジュールの register_hooks が完了した後の適切なフェーズ（通常は optional_fn_retrieve フック内）で行う必要があります。register_hooks 自体の実行中に直接取得しようとすると、相手のモジュールがまだ登録を完了していない可能性があります。

  • **プラットフォーム依存の条件分岐**

    • mod_userdir.c の HAVE_UNIX_SUEXEC や mod_status.c の HAVE_TIMES のように、コンパイル環境やプラットフォームによって登録されるフックが条件分岐する場合があります。これらを考慮せずにフックの存在を前提とすると、特定のプラットフォームで動作不良を起こす原因となります。
