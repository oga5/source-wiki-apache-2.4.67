register_hooks

概念

register_hooks は、Apache HTTP Server (httpd) のモジュールアーキテクチャにおいて、各モジュールが自身の処理関数（ハンドラ、フィルタ、プロバイダー、オプション関数など）をサーバーのライフサイクルやリクエスト処理パイプライン（フックシステム）に登録するための標準的なエントリーポイント（コールバック関数）です。

各モジュールは、モジュール定義構造体（module）を通じてこの関数を公開し、サーバーの起動・初期化フェーズにおいて呼び出されます。

なぜ重要か

Apache HTTP Server は極めてモジュール化された設計を採用しており、コア自体は最小限の機能しか持ちません。リクエストの解析、アクセス制御、認証・認可、コンテンツ生成、フィルタリング、ログ記録などの具体的な処理はすべてモジュールが担当します。

register_hooks は、これらのモジュールが提供する具体的な処理ロジックを、サーバーの適切な処理フェーズ（translate_name, check_access, handler, post_config など）に結びつけるための唯一の標準的な手段であるため、極めて重要です。

関与するモジュール

本プロジェクトにおいて、register_hooks は以下の広範なモジュール群に関与しています。

  • module:docs/manual/developer (開発者向けドキュメント・サンプル)
  • module:modules/aaa (認証・認可・アクセス制御)
  • module:modules/arch/netware (NetWare プラットフォーム固有処理)
  • module:modules/arch/win32 (Windows プラットフォーム固有処理)
  • module:modules/cache (キャッシュ制御・共有キャッシュ)
  • module:modules/core (動的モジュールロードなど)
  • module:modules/dav/fs, module:modules/dav/lock, module:modules/dav/main (WebDAV 関連)
  • module:modules/debugging (デバッグ用フィルタ)
  • module:modules/echo (プロトコルエコー)
  • module:modules/filters (バッファ、データ、外部フィルタ、レート制限、置換などのフィルタ)
  • module:modules/generators (ASIS、ステータスなどのコンテンツ生成)
  • module:modules/http (HTTP プロトコルコア処理)
  • module:modules/loggers (ログ記録)
  • module:modules/mappers (アクション、ディレクトリ、スペル修正、ユーザーディレクトリ、仮想ホストエイリアスなどのマッピング)
  • module:modules/metadata (CERN メタデータ、環境変数、有効期限、識別情報、一意 ID など)
  • module:modules/proxy, module:modules/proxy/balancers (プロキシ、負荷分散、各種プロキシプロトコル)
  • module:modules/session (セッション管理)
  • module:server (コアシステム)

主要エンティティ

各モジュール内に定義されている register_hooks 関数、およびそれらが登録する主要な関数やプロバイダーの例は以下の通りです。

  • **symbol:docs/manual/developer/mod_example_1.c#register_hooks**: example_handler を ap_hook_handler に登録。
  • **symbol:modules/aaa/mod_access_compat.c#register_hooks**: check_dir_access を ap_hook_check_access に登録し、オプション関数 access_compat_ap_satisfies を登録。
  • **symbol:modules/aaa/mod_authn_anon.c#register_hooks**: 匿名認証プロバイダー "anon" を登録。
  • **symbol:modules/aaa/mod_authn_dbm.c#register_hooks**: "dbm" 認証プロバイダーを登録し、オプション関数取得フックを登録。
  • **symbol:modules/aaa/mod_authz_host.c#register_hooks**: ip, host, forward-dns, local 認可プロバイダーを登録。
  • **symbol:modules/cache/mod_file_cache.c#register_hooks**: file_cache_handler, file_cache_post_config, file_cache_xlat をそれぞれのフックに登録。
  • **symbol:modules/core/mod_so.c#register_hooks**: ap_find_loaded_module_symbol をオプション関数として登録し、dump_loaded_modules を設定テストフックに登録。
  • **symbol:modules/filters/mod_buffer.c#register_hooks**: buffer_out_filter, buffer_in_filter を登録。
  • **symbol:modules/http/http_core.c#register_hooks**: HTTP プロトコル処理用の各種フック（ap_process_http_connection など）およびフィルタ（HTTP_IN, HTTP_HEADER など）を登録。

エンドツーエンドの流れ

1. **サーバー起動・モジュールロード**: Apache HTTP Server の起動時、または動的モジュール（DSO）のロード時に、各モジュールの module 構造体が参照されます。 2. **フック登録フェーズ**: サーバーはロードされた各モジュールの register_hooks 関数を順次呼び出します。この際、メモリ管理用の apr_pool_t ポインタが引数として渡されます。 3. **API 呼び出しによる登録**: 各モジュールの register_hooks 内で、以下のいずれか（または複数）の登録 API が呼び出されます。

  • ap_hook_<phase>: 特定の処理フェーズにコールバック関数を登録。
  • ap_register_provider / ap_register_auth_provider: 認証、認可、キャッシュなどのプロバイダーを登録。
  • ap_register_input_filter / ap_register_output_filter: フィルタを登録。
  • APR_REGISTER_OPTIONAL_FN: 他のモジュールから動的に呼び出し可能なオプション関数を公開。
  • ap_hook_optional_fn_retrieve: 他のモジュールが公開したオプション関数を取得するためのフックを登録。

    • 4. **リクエスト処理時の実行**: 登録されたフックやフィルタは、実際のリクエスト処理パイプラインの対応するフェーズにおいて、登録された優先順位に従って順次実行されます。

*注意: サーバーコア（server/core.c など）がどのようにこれらの register_hooks を走査し、内部のフック配列を再構築・ソートするかの詳細なエンドツーエンドの制御フローは、提供されたモジュールコンテキストからは完全には明らかになっていません。*

重要な不変条件

  • **シグネチャの不変性**: すべての register_hooks 関数は static void register_hooks(apr_pool_t *p)（または pool）という同一のシグネチャを持ちます。
  • **実行順序の制御**: フック登録時、優先順位（APR_HOOK_FIRST, APR_HOOK_MIDDLE, APR_HOOK_LAST など）や、先行・後続モジュールの明示的な指定（aszPre, aszSucc 配列など）を用いて、実行順序の不変関係を定義します。
  • 例: mod_userdir.c では、mod_alias.c の後、mod_vhost_alias.c の前に実行されるよう順序が制御されます。
  • 例: mod_expires.c では、キャッシュ保存フィルタよりも先に実行されるよう AP_FTYPE_CONTENT_SET-2 の位置にフィルタを登録します。
  • **スレッド安全性とライフサイクル**: register_hooks はサーバーの初期化フェーズ（シングルスレッド環境）で一度だけ実行され、リクエスト処理中（マルチスレッド/マルチプロセス環境）に動的にフックを追加することは想定されていません。

よくある落とし穴

  • **優先順位の競合**: 複数のモジュールが APR_HOOK_LAST や APR_HOOK_FIRST を指定した場合、それらの間の相対的な実行順序は保証されません。順序依存性がある場合は、先行・後続モジュールを明示的に指定する必要があります。
  • **先行モジュールの未ロード**: ap_hook_optional_fn_retrieve やプロバイダー登録において、依存する他のモジュール（例: mod_authz_owner が提供する関数を必要とする mod_authz_groupfile）がロードされていない場合、オプション関数の取得に失敗し、機能が正常に動作しない可能性があります。
  • **フィルタ位置の誤り**: フィルタを登録する際、適切なフィルタタイプ（AP_FTYPE_RESOURCE, AP_FTYPE_CONTENT_SET, AP_FTYPE_CONNECTION など）やオフセットを指定しないと、他のフィルタ（例: mod_deflate）との実行順序が狂い、データが破損したり処理されなかったりする原因になります。
