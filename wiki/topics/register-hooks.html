register_hooks

概念

register_hooks は、Apache HTTP Server のモジュールアーキテクチャにおける標準的な初期化関数（コールバック）の命名規則、およびその仕組みです。

各モジュールは、自身の module 構造体（AP_DECLARE_MODULE マクロなどを通じて定義される）のフック登録メンバーにこの関数を登録します。サーバーの起動（モジュールロード）時に、Apache コアがこの関数を呼び出すことで、モジュールが提供するハンドラ、フィルタ、プロバイダー、オプション関数などがシステムに登録されます。

なぜ重要か

Apache HTTP Server は極めてモジュール化された設計を採用しており、コア自体は最小限の機能しか持ちません。register_hooks は、各モジュールがサーバーのライフサイクル（設定フェーズ、接続処理、リクエスト処理、ログ出力など）に介入するための公式な窓口です。

この関数を通じて、モジュールは適切なフェーズ（例: translate_name, check_authn, fixups, handler など）に自身の関数をフックとして挿入し、サーバー全体と協調して動作することができます。

主要エンティティ

各モジュールおよびコアシステム内に、それぞれ固有の register_hooks 関数が定義されています。代表的なエンティティは以下の通りです。

  • **コアシステム**:
  • symbol:server/core.c#register_hooks: コアサーバー自体の基本的なフックやフィルタの登録。
  • symbol:modules/http/http_core.c#register_hooks: HTTP プロトコル処理に必要なフック、フィルタ（HTTP_IN, HTTP_HEADER など）、およびメソッドレジストリの初期化。
  • **認証・認可 (AAA)**:
  • symbol:modules/aaa/mod_auth_basic.c#register_hooks / symbol:modules/aaa/mod_authn_file.c#register_hooks など: 認証プロバイダーや認可プロバイダーの登録。
  • **リクエストマッピング**:
  • symbol:modules/mappers/mod_rewrite.c#register_hooks / symbol:modules/mappers/mod_userdir.c#register_hooks: リクエストのURL翻訳や書き換えフックの登録。
  • **フィルタ**:
  • symbol:modules/filters/mod_deflate.c#register_hooks / symbol:modules/filters/mod_expires.c#register_hooks: 出力・入力フィルタの登録。

エンドツーエンドの流れ

1. **モジュールのロード**: Apache の起動時に、各モジュールの共有ライブラリがロードされ、module 構造体が参照されます。 2. **フック登録の実行**: Apache コア（ap_register_hooks）が、ロードされたすべてのモジュールの register_hooks 関数を順次呼び出します。 3. **APIによる登録**: 各モジュールの register_hooks 内部で、以下の登録 API が呼び出されます。

  • **フックの登録**: ap_hook_handler, ap_hook_translate_name, ap_hook_fixups など。
  • **フィルタの登録**: ap_register_output_filter, ap_register_input_filter。
  • **プロバイダーの登録**: ap_register_provider, ap_register_auth_provider。
  • **オプション関数の公開**: APR_REGISTER_OPTIONAL_FN。

    • 4. **リクエスト処理時の実行**: 登録されたフックやフィルタは、実際のリクエスト処理パイプラインにおいて、指定された優先順位に従って実行されます。

*注意: 全モジュールにわたる完全な実行順序や、コア内部の具体的な呼び出しフローの全容は、提供された情報からは完全には特定されていません。*

重要な不変条件

  • **シグネチャの不変性**: ほぼすべての register_hooks は static void register_hooks(apr_pool_t *p) というシグネチャを持ち、メモリ管理用の APR プールを受け取ります。
  • **実行タイミングの不変性**: すべてのフックおよびフィルタの登録は、実際のリクエスト処理が開始される前の、サーバーの初期化・設定フェーズ中に完了していなければなりません。
  • **順序制御の指定**: フック登録時には、APR_HOOK_FIRST, APR_HOOK_MIDDLE, APR_HOOK_LAST などの優先度、あるいは先行・後続モジュールの明示的な指定（例: mod_userdir.c における mod_alias.c の後、mod_vhost_alias.c の前という指定）により、モジュール間の実行順序の不変条件が維持されます。

*注意: メモリプール apr_pool_t の具体的な生存期間やスレッドセーフ性に関する詳細な制約は、提供された情報からは完全には特定されていません。*

よくある落とし穴

  • **優先順位の競合**: 複数のモジュールが同じフェーズに APR_HOOK_LAST や APR_HOOK_FIRST を指定して登録した場合、それらの相対的な実行順序は保証されません。明示的な順序制御（先行・後続リストの指定）を行わないと、意図しない挙動が発生する可能性があります。
  • **オプション関数の解決タイミング**: 他のモジュールが提供するオプション関数を register_hooks 内で直接取得しようとすると、ロード順序によってはまだ登録されていない可能性があります。そのため、ap_hook_optional_fn_retrieve を使用して適切なタイミングで取得する必要があります（例: mod_authn_file.c や mod_authz_groupfile.c での opt_retr の使用）。
  • **エラーハンドリングの欠如**: register_hooks 自体は void を返すため、内部での登録失敗（メモリ不足など）を直接呼び出し元にエラーとして返せません。登録失敗はサーバー起動の失敗や、リクエストが捕捉されないといった暗黙的な不具合として現れます。
