register_hooks

概念

register_hooks は、Apache HTTP Server (httpd) の各モジュールにおいて、サーバーの起動・初期化フェーズ中に自身が提供する機能（リクエストハンドラ、フィルタ、認証・認可プロバイダー、オプション関数など）を、サーバーのコアシステムや他のモジュールに登録するためのエントリーポイントとなる関数です。

なぜ重要か

Apache HTTP Server は高度にモジュール化されたアーキテクチャを採用しており、コアシステム自体は最小限の機能しか持ちません。各モジュールがリクエスト処理パイプライン（URL翻訳、アクセス制御、認証、認可、コンテンツ生成、フィルタリング、ロギングなど）に介入するためには、この register_hooks を通じて適切なフックやプロバイダーを登録する必要があります。これにより、モジュールはサーバーのライフサイクルイベントに動的にバインドされ、協調して動作することが可能になります。

主要エンティティ

各モジュール内で定義されている register_hooks 関数（static void register_hooks(apr_pool_t *p)）が主要なエンティティです。 代表的な登録例：

  • **リクエストハンドラの登録**: symbol:docs/manual/developer/mod_example_1.c#register_hooks や symbol:modules/generators/mod_asis.c#register_hooks などで ap_hook_handler を使用。
  • **認証・認可プロバイダーの登録**: symbol:modules/aaa/mod_authn_anon.c#register_hooks や symbol:modules/aaa/mod_authz_user.c#register_hooks などで ap_register_auth_provider を使用。
  • **フィルタの登録**: symbol:modules/filters/mod_buffer.c#register_hooks や symbol:modules/filters/mod_ratelimit.c#register_hooks で ap_register_output_filter / ap_register_input_filter を使用。
  • **オプション関数の公開**: symbol:modules/aaa/mod_access_compat.c#register_hooks や symbol:modules/metadata/mod_ident.c#register_hooks で APR_REGISTER_OPTIONAL_FN を使用。
  • **ライフサイクルフックの登録**: symbol:modules/metadata/mod_unique_id.c#register_hooks で ap_hook_post_config や ap_hook_child_init などを登録。

エンドツーエンドの流れ

1. **サーバー起動とモジュールロード**: Apache HTTP Server の起動時、設定ファイルに基づいて各モジュールがロードされます。 2. **フック登録の実行**: モジュール構造体（module）の定義を介して、Apache コアが各モジュールの register_hooks 関数を呼び出します。この際、メモリ管理用の apr_pool_t が引数として渡されます。 3. **各種機能の登録**: register_hooks 内部で、ap_hook_*、ap_register_*、APR_REGISTER_OPTIONAL_FN などの API を呼び出し、ハンドラ、フィルタ、プロバイダー、オプション関数を登録します。 4. **リクエスト処理時の実行**: 登録されたフックやフィルタは、実際のリクエスト処理パイプラインの対応するフェーズ（例: translate_name, check_access, fixups, フィルタチェーンの実行など）において、登録された優先順位に従って呼び出されます。

*注意: モジュールロードからフック呼び出しに至る詳細なコア側のディスパッチロジックや、全モジュール間の厳密な実行順序の決定ロジックなど、エンドツーエンドの完全な内部フローは提供された情報からは完全には特定されていません。*

重要な不変条件

  • **シグネチャの不変性**: ほぼすべてのモジュールで static void register_hooks(apr_pool_t *p) という共通のシグネチャを持ち、メモリプール apr_pool_t を受け取ります。
  • **実行フェーズの順序制御**: フック登録時、APR_HOOK_FIRST、APR_HOOK_MIDDLE、APR_HOOK_LAST などの優先度定数、または先行・後続モジュールの明示的な指定（例: symbol:modules/mappers/mod_userdir.c#register_hooks における mod_alias.c の後、mod_vhost_alias.c の前という指定）により、実行順序の不変関係が定義されます。
  • **プロバイダーグループの整合性**: 認証・認可やキャッシュなどのプロバイダー登録（ap_register_provider / ap_register_auth_provider）では、AUTHN_PROVIDER_GROUP や AP_SOCACHE_PROVIDER_GROUP などの定義済みのグループ名とバージョンに準拠する必要があります。

よくある落とし穴

  • **優先順位（順序）の競合**: 複数のモジュールが同じフェーズ（例: fixups や translate_name）に APR_HOOK_LAST や APR_HOOK_FIRST で登録した場合、それらの間の相対的な実行順序は保証されず、予期しない動作を引き起こす可能性があります。
  • **先行モジュールへの依存**: symbol:modules/aaa/mod_authz_groupfile.c#register_hooks のように他のモジュール（mod_authz_owner）が提供するオプション関数に依存している場合、オプション関数の取得（ap_hook_optional_fn_retrieve）が適切なタイミングで行われないと、認可処理が正常に動作しない原因になります。
  • **環境依存の登録漏れ**: symbol:modules/mappers/mod_userdir.c#register_hooks の HAVE_UNIX_SUEXEC や symbol:modules/generators/mod_status.c#register_hooks の HAVE_TIMES のように、コンパイル時マクロによってフック登録が条件分岐している場合、特定のプラットフォームやビルド構成でのみフックが登録されない（または登録される）動作の違いに注意する必要があります。
