append_bucket

概念

append_bucket は、Apache HTTP Server 内の複数のモジュール（mod_sed フィルタおよび HTTP/2 接続制御）において、バケットブリゲード（apr_bucket_brigade）にデータバケットを追加・挿入するために定義されている内部関数です。同名ですが、それぞれのモジュールで独立して実装されています。

なぜ重要か

Apache のバケットブリゲードモデルにおいて、効率的なメモリ管理とデータのストリーミング転送を両立させるために重要です。例えば mod_sed では、フィルタ処理結果を一時バケット（transient bucket）やプールバケットとしてブリゲードに追加し、一定数に達した段階でフラッシュ（後続フィルタへの引き渡しとメモリ解放）を行うことで、メモリ消費の肥大化を防いでいます。

エンドツーエンドの流れ

mod_sed.c における流れ

1. **バケットの生成**: sed フィルタの処理結果バッファを受け取り、ctx->tpool がリクエストプールと一致する場合はプールバケット（apr_bucket_pool_create）を作成します。それ以外の場合は一時バケット（apr_bucket_transient_create）を作成し、一時バケット数をインクリメントします。 2. **ブリゲードへの挿入**: 生成したバケットをブリゲード ctx->bb の末尾に挿入します。 3. **フラッシュ判定**: 一時バケット数が MAX_TRANSIENT_BUCKETS に達した場合、フラッシュバケットを作成してブリゲードの末尾に挿入し、ap_pass_brigade を呼び出して後続フィルタへ渡します。その後、apr_brigade_cleanup でブリゲードを空にし、clear_ctxpool(ctx) でコンテキストのプールを解放します。

h2_bucket_beam.c における流れ

  • 提供されたコンテキストからは、h2_bucket_beam.c における append_bucket の具体的なエンドツーエンドの流れ、および mod_sed.c 側との直接的な関係は**未解明（不明）**です。

重要な不変条件

mod_sed.c における不変条件

  • ctx->bb (バケットブリゲード) が初期化されていること。
  • ctx->r->connection->bucket_alloc が有効であること。

h2_bucket_beam.c における不変条件

  • 提供されたコンテキストからは**未解明（不明）**です。

よくある落とし穴

  • **メモリリークとフラッシュタイミング**: mod_sed.c において、一時バケットの累積によるメモリリークを防ぐためには、MAX_TRANSIENT_BUCKETS に達した際の clear_ctxpool の呼び出しタイミングが極めて重要です。
  • **同名関数の混同**: mod_sed.c と h2_bucket_beam.c に存在する append_bucket は、名前は同一ですが別個の実装であり、直接的な依存関係はありません。
