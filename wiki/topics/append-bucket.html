append_bucket

概念

append_bucket は、Apache HTTP Server (httpd) の複数のモジュール（mod_sed および mod_http2）において、バケットブリゲード（データのストリームを表現するバケットの連結リスト）に新しいデータバケットを追加するために定義されている内部関数です。

これらは共通の共有ライブラリ関数ではなく、各モジュールの文脈（フィルタ処理やHTTP/2のバケット転送）に応じて個別に実装された同名の関数です。

なぜ重要か

バケットブリゲードへのデータの追加は、Apacheのフィルタチェーンやデータ転送において基本となる処理です。

例えば mod_sed においては、sedによるストリーム編集の処理結果を効率的にバケット化し、メモリ消費を抑えつつ後続のフィルタへ受け渡す（フラッシュする）役割を担っています。これにより、大容量データ処理時におけるメモリリークや過度なメモリ消費を防ぐことができます。

主要エンティティ

  • **symbol:modules/filters/mod_sed.c#append_bucket**: sed フィルタの処理結果をバケットブリゲードに追加し、必要に応じてフラッシュとメモリ解放を行う静的関数。
  • **symbol:modules/http2/h2_bucket_beam.c#append_bucket**: h2_bucket_beam.c 内で定義または呼び出されている同名の関数（詳細な内部仕様は本コンテキストからは未詳）。

エンドツーエンドの流れ

mod_sed における流れ

1. **バケットの生成**:

  • コンテキストのプール（ctx->tpool）がリクエストプールと一致する場合は、apr_bucket_pool_create を用いてプールバケットを作成します。
  • それ以外の場合は、apr_bucket_transient_create を用いて一時バケット（transient bucket）を作成し、一時バケットのカウント（ctx->numbuckets）をインクリメントします。

    • 2. **ブリゲードへの挿入**: 生成したバケットを、コンテキストが保持するバケットブリゲード（ctx->bb）の末尾に挿入します。 3. **フラッシュとクリーンアップ**:

  • 一時バケットの数が MAX_TRANSIENT_BUCKETS に達した場合、フラッシュバケットを作成してブリゲードの末尾に挿入します。
  • ap_pass_brigade を呼び出して、ブリゲードを後続のフィルタチェーンに渡します。
  • 送信後、apr_brigade_cleanup でブリゲードをクリアし、clear_ctxpool(ctx) を呼び出してコンテキストプールを解放します。

mod_http2 における流れ

重要な不変条件

  • **mod_sed における前提条件**:
  • append_bucket の呼び出し時、フィルタ実行コンテキスト内のバケットブリゲード（ctx->bb）が適切に初期化されている必要があります。
  • 接続オブジェクトのバケットアロケータ（ctx->r->connection->bucket_alloc）が有効でなければなりません。
  • **メモリ管理の不変条件**:
  • 一時バケットの累積数が MAX_TRANSIENT_BUCKETS に達した際は、必ず後続フィルタへの受け渡し（ap_pass_brigade）と、コンテキストプール（clear_ctxpool）の解放が行われ、メモリ使用量が一定以下に維持される必要があります。
  • **mod_http2 における不変条件**:
  • modules/http2/h2_bucket_beam.c 側の不変条件については、現時点では不明です。

よくある落とし穴

  • **メモリリークとフラッシュタイミング**: mod_sed の append_bucket は clear_ctxpool によるプール管理と密接に結びついています。一時バケットの管理やフラッシュのタイミングを誤ると、メモリ消費が急増したり、解放済みのメモリにアクセスする危険性があります。
  • **名前の衝突と混同**: mod_sed.c と h2_bucket_beam.c の append_bucket は、名前は同じですが別個の実装です。これらを混同して、一方の仕様や前提条件を他方に適用しないよう注意が必要です。
