on_event

概念

on_event は、Apache HTTP Server（httpd）の複数の独立したモジュール（http2 および md）において、特定のイベントが発生した際に呼び出される内部関数またはコールバック関数です。これらは同名ですが、それぞれ異なる目的と文脈で実装されており、直接的な相互作用はありません。

  • **http2 モジュールにおける on_event**: HTTP/2 ストリームの状態遷移を管理するための関数。
  • **md モジュールにおける on_event**: md_event サブシステムからのイベント通知を受け取り、関連するジョブ通知処理（notify）へ転送するためのコールバック関数。

なぜ重要か

各モジュールにおける on_event の重要性は以下の通りです。

  • **modules/http2**: HTTP/2 ストリームの状態遷移ロジックをカプセル化し、プロトコル仕様に準拠した正しい状態管理（trans_on_event テーブルに基づく遷移）を保証するために不可欠です。
  • **modules/md**: ACME などのマネージドドメイン（MD）管理において、非同期に発生するイベントを検知し、適切なジョブ通知（notify）へと橋渡しするイベント駆動アーキテクチャの起点として機能します。

エンドツーエンドの流れ

これら2つの関数は完全に独立して動作しており、システム全体を横断する単一のエンドツーエンドフローは存在しません。それぞれのモジュールにおける処理の流れは以下の通りです。

http2 モジュールでの流れ

1. ストリームの状態遷移を伴う処理（h2_stream.c 内の各所）から on_event が呼び出されます。 2. ストリームに monitor が登録されている場合、stream->monitor->on_event コールバックが実行され、イベントが通知されます。 3. イベント ev が trans_on_event 配列の範囲内であれば、on_map 関数を介して現在の状態（stream->state）とイベントに対応する遷移先を計算し、新しい状態を返します。

md モジュールでの流れ

1. modules/md/mod_md.c 内で md_event_subscribe に登録された本コールバックが、md_event サブシステムからイベント通知を受け取ります。 2. 受け取ったパラメータ（job, event, result, p, baton）をそのまま notify 関数にパススルーで引き渡します。 3. notify 関数の実行結果（apr_status_t）を戻り値として返します。

重要な不変条件

  • **http2 モジュール**:
  • 関数呼び出し時に H2_STRM_ASSERT_MAGIC(stream, H2_STRM_MAGIC_OK) により、ストリーム構造体の整合性が常に検証されます。
  • イベント ev が定義範囲外（trans_on_event の範囲外）の場合、状態遷移は行われず、現在の状態（stream->state）がそのまま維持されます。
  • **md モジュール**:
  • 引数 mdomain は関数内で明示的にキャストされ、使用されずに無視されます。
  • notify 関数が正常に呼び出せる状態である必要があります。

よくある落とし穴

  • **名前の混同**: modules/http2/h2_stream.c の on_event と modules/md/mod_md.c の on_event は、名前が同一であるだけで、設計上および実行時のつながりは一切ありません。これらを混同しないよう注意が必要です。
  • **http2 での状態遷移定義**: 状態遷移ルールを変更する場合、on_event 自体ではなく trans_on_event テーブルの定義を修正する必要があります。また、無効なストリームを渡すとアサーションエラーが発生します。
  • **md でのエラー伝播**: on_event は単なるパススルーとして機能するため、呼び出し先の notify 関数が失敗した場合、そのエラーコードがそのまま呼び出し元に伝播します。
