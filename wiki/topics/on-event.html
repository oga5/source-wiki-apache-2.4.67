on_event

概念

on_event は、Apache HTTP Server（httpd）の複数の異なるモジュール（mod_http2 および mod_md）において、特定のイベントが発生した際に呼び出されるイベント処理関数（コールバック）です。

これらは同名ですが、それぞれ独立したコンテキストと目的を持っています。

  • **mod_http2 における on_event**: HTTP/2 ストリームの状態遷移（ステートマシン）を管理・制御します。
  • **mod_md における on_event**: マネージドドメイン（Managed Domain）のジョブイベントを検知し、通知処理（notify）へ転送します。

なぜ重要か

  • **HTTP/2 ストリームの整合性維持 (mod_http2)**: HTTP/2 プロトコルにおけるストリームのライフサイクルと状態遷移を正確に制御し、不正な状態遷移を防ぐために不可欠です。
  • **ドメイン管理ジョブの追跡 (mod_md)**: 証明書の取得や更新といった非同期ジョブの進捗や結果を、適切な通知システムへ確実に連携するために重要です。

関与するモジュール

  • module:modules/http2
  • module:modules/md

主要エンティティ

  • symbol:modules/http2/h2_stream.c#on_event: HTTP/2 ストリームの状態遷移イベントを処理する静的関数。
  • symbol:modules/md/mod_md.c#on_event: md_event サブシステムからのイベントを notify 関数へ転送する静的コールバック関数。

エンドツーエンドの流れ

これら2つのモジュール間にまたがる単一のエンドツーエンドの流れは存在せず、それぞれ独立して動作します。各モジュール内での流れは以下の通りです。

mod_http2 における流れ

1. ストリームの状態遷移を伴う各処理（h2_stream.c 内の各所）から on_event(stream, ev) が呼び出されます。 2. ストリームに監視用の monitor が登録されている場合、stream->monitor->on_event コールバックが呼び出され、イベントが通知されます。 3. イベント ev が状態遷移テーブル trans_on_event の範囲内であるか確認されます。 4. 範囲内であれば、on_map 関数を呼び出して現在の stream->state とイベントに対応する遷移先（新しい状態）を計算して返します。範囲外の場合は、現在の状態をそのまま維持して返します。

mod_md における流れ

1. modules/md/mod_md.c 内で md_event_subscribe に登録された on_event が、md_event サブシステムからイベント通知を受け取ります。 2. on_event は、受け取った引数のうち mdomain を無視し、それ以外の引数（job、event、result、p、baton）をそのまま notify 関数に引き渡します。 3. notify 関数の実行結果（apr_status_t）がそのまま戻り値として返されます。

重要な不変条件

  • **mod_http2**:
  • H2_STRM_ASSERT_MAGIC(stream, H2_STRM_MAGIC_OK) により、処理対象のストリーム構造体の整合性が常に検証されます。
  • イベント ev が trans_on_event の定義範囲外である場合、状態遷移は行われず、現在の状態が維持されます。
  • **mod_md**:
  • mdomain 引数は関数内で使用されず、明示的にキャストされて無視されます。
  • イベント転送を成立させるため、notify 関数が正常に呼び出せる状態である必要があります。

よくある落とし穴

  • **同名関数の混同**: modules/http2 の on_event と modules/md の on_event は、名前は同一ですが、全く異なるデータ構造と目的を持つ独立した静的関数です。ソースコードの静的解析やデバッグの際にはこれらを混同しないよう注意が必要です。
  • **HTTP/2 状態遷移の変更**: mod_http2 において状態遷移ルールを変更する場合、on_event のコードだけでなく、遷移テーブルである trans_on_event の定義を正しく更新する必要があります。
  • **MD イベントのエラーハンドリング**: mod_md の on_event は単なるパススルーとして実装されているため、転送先の notify 関数が失敗した場合、そのエラーコードがそのまま呼び出し元に返されます。
