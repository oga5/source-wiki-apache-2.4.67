Shared Object Cache (socache) Architecture

概念

Shared Object Cache (socache) は、Apache HTTP Server (httpd) において、SSLセッションや認証情報などのオブジェクトを、複数のプロセスやスレッド間で共有・キャッシュするためのプロバイダーベースの共通アーキテクチャです。共通のインターフェースを介して、異なるキャッシュバックエンド（共有メモリ、memcached、Redisなど）を透過的に切り替えて利用できるように設計されています。

なぜ重要か

Webサーバーのパフォーマンス向上において、SSL/TLSハンドシェイクの再ネゴシエーションコストの削減や、重複する認証処理の回避は極めて重要です。socacheアーキテクチャは、高パフォーマンスなキー・バリューストアへのアクセスを抽象化されたAPIで提供することで、各モジュール（例えば mod_ssl など）がバックエンドの具体的な実装（共有メモリの排他制御や外部キャッシュサーバーとの通信など）を意識することなく、効率的なキャッシュ機構を利用できるようにします。

関与するモジュール

  • module:include
  • module:modules/cache

主要エンティティ

  • **include/ap_socache.h**:

    • socacheプロバイダーが実装すべき共通API（オブジェクトの作成、初期化、保存、取得、削除、破棄など）を定義するヘッダーファイルです。

  • **modules/cache/mod_socache_shmcb.c**:

    • 共有メモリ（Shared Memory）上にサイクリックバッファ（shmcb）を構築してオブジェクトをキャッシュするプロバイダー実装です。外部依存なしに高速な共有キャッシュを実現します。

  • **modules/cache/mod_socache_memcache.c**:

    • 外部の分散メモリキャッシュシステムである memcached をバックエンドとして利用するプロバイダー実装です。

  • **modules/cache/mod_socache_redis.c**:

    • 外部のキー・バリューストアである Redis をバックエンドとして利用するプロバイダー実装です。

エンドツーエンドの流れ

キャッシュを利用するモジュール（呼び出し元）と各socacheプロバイダー間の詳細な呼び出しシーケンスや、エラーハンドリングを含む完全なエンドツーエンドの制御フローは、現時点では十分に解析されておらず未確定です。

一般的な想定フローとしては、サーバー起動時に ap_socache_provider_t インターフェースを介して選択されたプロバイダーが初期化され、リクエスト処理中にキー（SSLセッションIDなど）に基づいたオブジェクトの保存（store）や取得（retrieve）が呼び出されます。

重要な不変条件

  • **インターフェースの整合性**:

    • すべてのsocacheプロバイダー（shmcb, memcache, redis など）は、include/ap_socache.h で定義された共通のAPI規約を満たす必要があります。

  • **排他制御とスレッドセーフティ**:

    • 特に mod_socache_shmcb.c のような共有メモリ実装においては、複数プロセス・スレッドからの同時書き込みに対してデータが破損しないよう、適切なロックおよび排他制御が維持される必要があります。ただし、具体的なロックのライフサイクルや境界条件の詳細は現時点では未確定です。

よくある落とし穴

  • **外部キャッシュへの依存**:

    • mod_socache_memcache や mod_socache_redis を使用する場合、ネットワークの遅延や外部サーバーのダウンタイムがWebサーバー全体の応答性に直接影響を与える可能性があります。

  • **容量制限とデータ消失**:

    • mod_socache_shmcb では、共有メモリのサイズ制限に達した際に古いエントリーがサイクリックに上書きされます。この挙動を考慮した適切なバッファサイズ設計を行わないと、意図しないキャッシュミスが多発する原因となります。

次に読む場所

  • **include/ap_socache.h**:

    • socacheプロバイダーが提供するAPIの全容と、各関数のシグネチャを理解するために最初に参照すべきファイルです。

  • **modules/cache/mod_socache_shmcb.c**:

    • ローカル共有メモリを用いた、最も標準的かつ自己完結したキャッシュ実装の詳細を確認できます。

  • **modules/cache/mod_socache_memcache.c / modules/cache/mod_socache_redis.c**:

    • 外部ネットワークサービスと連携するキャッシュプロバイダーの実装パターンを学ぶのに適しています。
