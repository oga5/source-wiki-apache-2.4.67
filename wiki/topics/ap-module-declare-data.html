AP_MODULE_DECLARE_DATA

概念

AP_MODULE_DECLARE_DATA は、Apache HTTP Server（httpd）のモジュール構造体（module 構造体）を宣言・定義する際に付与するプリプロセッサマクロです。

共有ライブラリ（DSO: Dynamic Shared Object）としてモジュールをビルドする際、Windows（Win32）などの特定のプラットフォームでは、外部からシンボルを参照できるようにするために特別なストレージクラス属性（__declspec(dllexport) や __declspec(dllimport) など）をシンボルに付与する必要があります。AP_MODULE_DECLARE_DATA は、プラットフォームやビルド構成（静的リンクか動的リンクか）の違いを吸収し、適切なエクスポート/インポート属性へと展開される役割を持ちます。

なぜ重要か

  • **クロスプラットフォーム互換性**: Windows、NetWare、Unix系OSなど、異なるシンボルエクスポート機構を持つOS間で、同一のCソースコードを変更せずにコンパイル可能にします。
  • **モジュールの動的ロード**: mod_so などの動的ローダーが、共有ライブラリからモジュールのエントリポイント（module 構造体）を正しく解決してロードするために不可欠です。

関与するモジュール

本マクロは、Apacheのほぼすべての機能モジュールおよびコアシステムに関与しています。

  • module:include (マクロの定義ヘッダー)
  • module:modules/aaa (認証・承認モジュール群)
  • module:modules/cache (キャッシュ管理)
  • module:modules/core (コア機能、mod_so など)
  • module:modules/filters (出力・入力フィルタ)
  • module:modules/generators (コンテンツ生成)
  • module:modules/mappers (URLマッピング、mod_rewrite など)
  • module:modules/proxy (プロキシおよびロードバランサ)
  • module:modules/ssl (mod_ssl)
  • module:server/mpm/event (マルチプロセッシングモジュール)
  • その他、database, dav, debugging, ldap, loggers, lua, session などの各種拡張モジュール

主要エンティティ

  • **include/ap_config.h**: プラットフォームごとの AP_MODULE_DECLARE_DATA マクロの具体的な展開ルールを定義します。
  • **include/httpd.h**: module 構造体の定義、およびコアシステムにおけるモジュール宣言の基礎を提供します。
  • **各モジュールソース（例: modules/ssl/mod_ssl.c）**:

    • ``c module AP_MODULE_DECLARE_DATA ssl_module; ` のように、自身のモジュール定義シンボル（例: ssl_module`）に対してマクロを適用します。

エンドツーエンドの流れ

1. **プリプロセス段階**: コンパイル時に ap_config.h が読み込まれ、ターゲットプラットフォームやコンパイルオプション（DSOビルドか否かなど）に応じて AP_MODULE_DECLARE_DATA が適切な属性（または空）に展開されます。 2. **シンボルのエクスポート**: コンパイラおよびリンカは、このマクロによって修飾された module 構造体シンボル（例: ssl_module）を、生成される共有ライブラリ（.so や .dll）のエクスポートシンボルテーブルに登録します。 3. **モジュールのロード**: Apacheの起動時または再起動時に、設定ファイル（httpd.conf）の LoadModule ディレクティブに基づき、mod_so が共有ライブラリをロードし、エクスポートされた module 構造体のアドレスを取得してApacheコアに登録します。

*注意: プラットフォームごとのコンパイラ固有の展開規則や、リンカフラグとの詳細な相互作用の全容は、提供されたソースコードの範囲内では完全には網羅されていません。*

重要な不変条件

  • **単一のモジュールシンボル**: 各モジュールファイルは、AP_MODULE_DECLARE_DATA で修飾された module 型のグローバル変数を**正確に1つ**定義しなければなりません。
  • **命名規則の一致**: ロード時の混乱を避けるため、定義される構造体名は通常、モジュールのファイル名に対応する名称（例: mod_rewrite.c 内では rewrite_module）にする必要があります。

よくある落とし穴

  • **マクロの付与漏れ**: 新しいモジュールを自作する際、AP_MODULE_DECLARE_DATA を付与せずに module 構造体を定義すると、Windows環境などでシンボルが外部に公開されず、起動時に「シンボルが見つからない（DLLロードエラー）」としてエラーになります。
  • **静的/動的ビルドの不整合**: ビルドシステム側で共有モジュール（DSO）としてビルドする際のプリプロセッサマクロ（SHARED_MODULE など）が正しく定義されていないと、インポートとエクスポートの属性が逆転し、リンクエラーを引き起こす可能性があります。

次に読む場所

  • **include/ap_config.h**: マクロがどのようにプラットフォームごとに分岐定義されているかを確認できます。
  • **include/httpd.h**: module 構造体のメンバ構成や、関連する宣言マクロの全体像を把握できます。
  • **modules/examples/mod_example_hooks.c**: 実際のモジュール実装における標準的な記述例として参考になります。
