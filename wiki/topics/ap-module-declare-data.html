AP_MODULE_DECLARE_DATA

概念

AP_MODULE_DECLARE_DATA は、Apache HTTP Server (httpd) において、各モジュールの実体である module 構造体（例：ssl_module、auth_basic_module など）を宣言・定義する際に付与されるマクロです。

このマクロは include/ap_config.h および include/httpd.h で定義されており、プラットフォーム固有のリンケージ属性（特にWindows環境におけるDLLのインポート/エクスポート指定など）を抽象化し、モジュールシンボルがコアや他の動的ライブラリから正しく参照できるようにします。

なぜ重要か

Apache HTTP Serverは動的共有オブジェクト (DSO) によるモジュールロード機構を採用しています。

Windowsなどの特定のプラットフォームでは、DLLからデータシンボルをエクスポートする際に __declspec(dllexport)、インポートする際に __declspec(dllimport) を明示的に指定する必要があります。AP_MODULE_DECLARE_DATA を用いることで、ビルド構成（コアのビルドか、外部モジュールのビルドか）やターゲットプラットフォームの違いを吸収し、ポータブルにモジュールシンボルを公開・共有することができます。

関与するモジュール

本マクロは、httpdのほぼすべての機能モジュールおよびMPM（Multi-Processing Module）に関与しています。

  • module:include (マクロの定義元)
  • module:modules/aaa (認証・認可関連モジュール群)
  • module:modules/cache (キャッシュ関連モジュール群)
  • module:modules/core (コア機能、mod_so など)
  • module:modules/filters (フィルタ関連モジュール群)
  • module:modules/generators (ハンドラ・ジェネレータ群)
  • module:modules/mappers (URLマッピング関連群)
  • module:modules/metadata (メタデータ・ヘッダー制御群)
  • module:modules/proxy (プロキシ関連モジュール群)
  • module:modules/ssl (mod_ssl)
  • module:server/mpm/event (イベント駆動型MPM)

    • ※その他、modules/arch、modules/database、modules/dav、modules/debugging、modules/echo、modules/examples、modules/ldap、modules/loggers、modules/lua、modules/session、modules/test などの多数のモジュール群が関与しています。

主要エンティティ

  • **symbol:include/ap_config.h#AP_MODULE_DECLARE_DATA**: プラットフォームやビルド環境に応じたマクロの具体的な展開内容（ストレージクラス属性など）を定義するヘッダーファイル。
  • **symbol:include/httpd.h#AP_MODULE_DECLARE_DATA**: モジュール構造体の基本定義とともに、本マクロの利用をサポートするヘッダーファイル。
  • **各モジュール内の AP_MODULE_DECLARE_DATA 適用シンボル**:
  • symbol:modules/aaa/mod_auth_basic.c#AP_MODULE_DECLARE_DATA (auth_basic_module)
  • symbol:modules/ssl/mod_ssl.c#AP_MODULE_DECLARE_DATA (ssl_module)
  • symbol:server/mpm/event/event.c#AP_MODULE_DECLARE_DATA (mpm_event_module)
  • その他、100以上のモジュールソースファイルで定義される module 型の静的/グローバル変数。

エンドツーエンドの流れ

1. **コンパイル時**: 各モジュールのソースコード（例: mod_ssl.c）がコンパイルされる際、AP_MODULE_DECLARE_DATA module ssl_module; のように宣言されます。このとき、ap_config.h の定義に基づき、適切なコンパイラ属性（例: Windows環境でモジュール自体をビルドする際は __declspec(dllexport)）に展開されます。 2. **リンク・ロード時**: mod_so などの動的ローダーが共有ライブラリ（.so や .dll）をロードする際、このエクスポートされたモジュールシンボル（例: ssl_module）の名前をキーにしてシンボルアドレスを解決し、Apacheコアのランタイムに登録します。

なお、ビルドシステム全体におけるマクロ展開の切り替え条件（どのマクロ定義がどのタイミングで有効化されるか）の完全なエンドツーエンドの制御フローは、提供されたソースコードの範囲内では完全には特定されていません。

重要な不変条件

  • **シンボル名の一貫性**: AP_MODULE_DECLARE_DATA で宣言されるモジュール構造体の変数名は、設定ファイル（httpd.conf）の LoadModule ディレクティブで指定されるシンボル名と完全に一致していなければなりません。
  • **データ型の不変性**: 本マクロが修飾する対象は、常に module 構造体（またはそれに準ずるモジュールメタデータ）のインスタンスでなければなりません。

なお、プラットフォームごとのリンケージ境界におけるコンパイラレベルの厳密な不変条件（スレッドローカルストレージやアライメント制約など）の詳細は、提供されたエンティティからは直接確認できません。

よくある落とし穴

  • **マクロの付与漏れ**: 新規にカスタムモジュールを作成する際、AP_MODULE_DECLARE_DATA を付与せずに module 構造体を定義すると、Unix/Linux環境では動作しても、Windows環境（MSVC）でビルド・ロードする際にシンボルがエクスポートされず、リンクエラーやロード時の「Symbol not found」エラーが発生します。
  • **ヘッダーのインクルード順**: ap_config.h や httpd.h が適切にインクルードされる前にモジュール宣言を行うと、マクロが未定義となりコンパイルエラーになります。

次に読む場所

  • include/ap_config.h (マクロのプラットフォーム別定義の確認)
  • include/httpd.h (モジュール構造体 module の定義と宣言方法の確認)
  • modules/core/mod_so.c (DSOロード機構におけるシンボル解決の仕組み)
  • modules/examples/mod_example_hooks.c (標準的なモジュール実装の参照実装)
