destroy_cache

概念

destroy_cache は、Apache HTTP Server（httpd-2.4.67）において、共有オブジェクトキャッシュ（socache）のインスタンスを破棄し、割り当てられたリソースを安全に解放するためのクリーンアップ関数です。複数のモジュールで同様の目的を持つ関数として定義されています。

なぜ重要か

サーバーの停止時や設定の再読み込み（リスタート）の際、古いキャッシュインスタンスや関連リソースを確実に解放し、メモリリークやリソースの競合を防ぐために重要です。設定用プール（pconf）のクリーンアップハンドラとして登録されることで、プロセスのライフサイクルに連動した自動的なリソース管理を実現します。

エンドツーエンドの流れ

mod_authn_socache.c における処理の流れは以下の通りです：

1. **登録**: 設定フェーズにおいて、apr_pool_cleanup_register を介して設定用プール（pconf）のクリーンアップハンドラとして destroy_cache が登録されます（modules/aaa/mod_authn_socache.c:134）。 2. **トリガー**: サーバーの停止または設定の再読み込みが発生すると、プールがクリアされ、本関数が自動的に呼び出されます。 3. **破棄の実行**:

  • 入力された void *data を server_rec* にキャストします。
  • グローバル変数 socache_instance が非 NULL であることを確認します。
  • socache_provider->destroy を呼び出し、キャッシュインスタンスとサーバーレコードを渡して破棄処理を実行します。
  • socache_instance を NULL にリセットします。

    • 4. **完了**: 常に APR_SUCCESS を返します。

※ mod_cache_socache.c における destroy_cache のエンドツーエンドの流れについては、提供された情報に含まれていないため現時点では不明です。

重要な不変条件

  • **二重解放の防止**: socache_instance を破棄した直後に NULL にリセットすることは、二重解放を防ぐための重要な不変条件です。
  • **前提条件**: socache_instance がグローバル変数として存在し、socache_provider が初期化済みである必要があります。

よくある落とし穴

  • **未初期化プロバイダーによるセグメンテーション違反**: socache_provider が未設定（NULL）の状態で本関数が呼び出された場合、socache_provider->destroy の呼び出し時にセグメンテーション違反を引き起こす可能性があります。
