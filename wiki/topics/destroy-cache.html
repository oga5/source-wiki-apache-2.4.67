destroy_cache

概念

destroy_cache は、Apache HTTP Server（httpd-2.4.67）において、共有オブジェクトキャッシュ（socache）のインスタンスを破棄し、割り当てられたリソースを解放するためのクリーンアップ関数です。複数のモジュール（認証用の mod_authn_socache やキャッシュ用の mod_cache_socache）で同様の目的を持つ関数として定義されています。

なぜ重要か

サーバーの停止時や設定の再読み込み（リスタート）時に、古いキャッシュインスタンスや関連リソースを確実に解放し、メモリリークやリソースの競合を防ぐために重要です。特に、設定プール（pconf）のクリーンアップハンドラとして登録されることで、Apacheのライフサイクルに合わせた自動的なリソース管理を実現します。

関与するモジュール

  • module:modules/aaa
  • module:modules/cache

主要エンティティ

  • symbol:modules/aaa/mod_authn_socache.c#destroy_cache: mod_authn_socache モジュールにおけるキャッシュ破棄関数。
  • symbol:modules/cache/mod_cache_socache.c#destroy_cache: mod_cache_socache モジュールにおけるキャッシュ破棄関数（詳細な実装は現時点で未解析）。

エンドツーエンドの流れ

mod_authn_socache における処理の流れは以下の通りです：

1. **登録**: サーバー初期化時に、apr_pool_cleanup_register を介して設定用プール（pconf）のクリーンアップハンドラとして destroy_cache が登録されます。 2. **トリガー**: サーバーの停止または設定の再読み込みが発生すると、プールクリーンアップ処理から本関数が自動的に呼び出されます。 3. **破棄の実行**:

  • 入力された void *data を server_rec* にキャストします。
  • グローバル変数 socache_instance が非 NULL であることを確認し、socache_provider->destroy を呼び出して実際の破棄処理を実行します。

    • 4. **クリーンアップ**: socache_instance を NULL にリセットし、APR_SUCCESS を返します。

※ mod_cache_socache 側におけるエンドツーエンドの流れ、および両モジュールを跨ぐ統合的なフローの詳細は現時点では不明です。

重要な不変条件

  • **二重解放の防止**: mod_authn_socache では、キャッシュ破棄後に socache_instance を必ず NULL にリセットする必要があります。これにより、二重解放（double free）を防ぎます。
  • **前提条件**: socache_instance が存在し、かつ socache_provider が初期化済みである必要があります。
  • ※ mod_cache_socache における不変条件は現時点では不明です。

よくある落とし穴

  • **未初期化プロバイダーによるクラッシュ**: socache_provider が未設定（NULL）の状態で destroy_cache が呼び出された場合、セグメンテーション違反を引き起こす可能性があります。
  • **情報の不足**: mod_cache_socache 側の destroy_cache の具体的な失敗モードや境界条件については、現時点のコンテキストからは不明です。
