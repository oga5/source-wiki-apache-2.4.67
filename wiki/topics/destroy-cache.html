destroy_cache

概念

destroy_cache は、Apache HTTP Server において共有オブジェクトキャッシュ（socache）のインスタンスを破棄し、割り当てられたシステムリソースを解放するためのクリーンアップ関数です。主に設定用プール（pconf）のライフサイクルと連動して実行されます。

なぜ重要か

サーバーの停止時や設定の再読み込み（リスタート）の際に、古いキャッシュインスタンスを安全に破棄し、メモリやファイルディスクリプタなどのリソースリークを防ぐために重要です。また、グローバルなキャッシュインスタンスポインタを適切にクリアすることで、二重解放や無効なポインタ参照を防止します。

エンドツーエンドの流れ

mod_authn_socache.c における処理の流れは以下の通りです。

1. **登録**: モジュールの初期化時に、apr_pool_cleanup_register を介して設定用プール（pconf）のクリーンアップハンドラとして destroy_cache が登録されます（modules/aaa/mod_authn_socache.c:134）。 2. **トリガー**: サーバーの停止や設定の再読み込みが発生すると、設定用プールが破棄され、登録された destroy_cache が自動的に呼び出されます。 3. **破棄処理**:

  • 入力された void *data を server_rec*（サーバーレコード）にキャストします。
  • グローバル変数 socache_instance が非 NULL であるかを確認します。
  • socache_provider->destroy を呼び出し、キャッシュインスタンスとサーバーレコードを渡してバックエンドの破棄処理を実行します。
  • socache_instance を NULL にリセットし、APR_SUCCESS を返します。

重要な不変条件

  • **二重解放の防止**: destroy_cache の実行後、グローバル変数 socache_instance は必ず NULL にリセットされます。これにより、既に破棄済みの状態で再度呼び出された場合でも、安全に早期リターン（何もしない）が行われます。
  • **前提条件**: socache_provider が初期化済みである必要があります。
  • ※ modules/cache/mod_cache_socache.c 側の不変条件については、提供されたコンテキストに情報がないため不明です。

よくある落とし穴

  • **未初期化プロバイダーによるクラッシュ**: socache_provider が未設定（NULL）の状態でこの関数が呼び出された場合、socache_provider->destroy の呼び出し時にセグメンテーション違反（Segmentation Fault）が発生する可能性があります。
