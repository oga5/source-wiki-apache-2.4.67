dav_db

概念

dav_db は、Apache HTTP Server の WebDAV モジュール（特に mod_dav_fs）において、ファイルシステムリポジトリのプロパティ（Dead Properties）を管理するための構造体です。DBM（Database Manager）ファイルへの接続状態、XML 名前空間テーブル、および関連するメタデータを保持します。

なぜ重要か

WebDAV では、リソースに関連付けられた任意のプロパティ（Dead Properties）を永続的に保存・管理する必要があります。dav_db は、DBM データベースへのアクセスを抽象化し、XML 名前空間のインデックス化によるキーの最適化や、プロパティの効率的な検索・更新・反復処理を可能にするため、WebDAV プロパティ管理の中核を担っています。

エンドツーエンドの流れ

1. **初期化とオープン**: dav_propdb_open などにより dav_db が初期化され、DBM ファイルがオープンされます。この際、METADATA キーからバージョン情報と名前空間テーブル（ns_table）が読み込まれ、uri_index ハッシュテーブルが構築されます。 2. **キー生成と操作**: プロパティの読み書き（dav_dbm_fetch, dav_dbm_store など）の際、dav_build_key が uri_index を参照して名前空間 ID とプロパティ名から #:name 形式の DBM キーを生成し、DBM 操作を行います。 3. **反復処理**: iter フィールド（apr_datum_t）を使用して、DBM 内のキーを順次走査（dav_dbm_firstkey, dav_dbm_nextkey）し、メタデータキーをスキップしながらプロパティを抽出します。 4. **クローズ**: dav_dbm_close により、必要に応じて変更された名前空間テーブルがデータベースに書き戻され（ns_table_dirty フラグによる制御）、DBM ファイルが閉じられます。

*注意: WebDAV リクエストの受信からレスポンス送信に至る完全なエンドツーエンドの制御フローは、提供されたコンテキストからは未詳です。*

重要な不変条件

  • **ライフサイクルの一貫性**: pool フィールドは apr_pool_t に紐付いており、dav_db の生存期間はこのプールのライフサイクルと一致している必要があります。
  • **名前空間の同期**: 名前空間テーブルが変更された場合、ns_table_dirty フラグがセットされ、データベースクローズ時等に永続化される必要があります。
  • **キー生成バッファの一時性**: wb_key は dav_build_key で使用される静的バッファ（作業用バッファ）であるため、生成されたキーは呼び出し直後にコピーして使用する必要があります。

*注意: 並行アクセス時やトランザクション境界における厳密な不変条件の詳細は、提供されたコンテキストからは未詳です。*

よくある落とし穴

  • **メモリリーク**: dav_db は apr_pool_t に依存しているため、プールが解放される前に適切にクローズ（dav_dbm_close）されない場合、DBM リソースやメモリのリークが発生します。
  • **並行書き込み時の不整合**: DBM の特性上、複数のプロセスやスレッドから同時に読み書きが発生した際の再試行ロジックが不足しており、データの整合性が損なわれる可能性があります。
  • **バッファの生存期間**: dav_build_key が使用する wb_key は静的バッファであるため、その結果を退避（コピー）せずに次のキー生成を行うと、データが上書きされる危険性があります。
  • **メタデータ更新の不完全さ**: METADATA のマイナーバージョン更新ロジックは現在不完全であるため、将来のスキーマ変更時に注意が必要です。
