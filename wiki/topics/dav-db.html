dav_db

概念

dav_db は、Apache HTTP Server の WebDAV モジュール（mod_dav および mod_dav_fs）において、ファイルシステムリポジトリのプロパティ（Dead Properties）を管理するための DBM データベース接続状態および関連メタデータを保持する構造体です。

なぜ重要か

WebDAV では、リソースに関連付けられた任意のメタデータ（プロパティ）を永続化する必要があります。dav_db は、XML 名前空間テーブルのキャッシュ、URI インデックス、および DBM ファイルハンドル（apr_dbm_t）を統合して管理することで、プロパティの効率的な読み書き、キー生成、および反復処理を実現する役割を担っています。

関与するモジュール

  • module:modules/dav/fs (modules/dav/fs/dbm.c)
  • module:modules/dav/main (modules/dav/main/mod_dav.h)

主要エンティティ

  • symbol:modules/dav/fs/dbm.c#dav_db: DBM 接続状態、名前空間テーブル（ns_table）、URI インデックス（uri_index）、キー生成バッファ（wb_key）などを保持する構造体。
  • symbol:modules/dav/main/mod_dav.h#dav_db: メインモジュール側で参照される dav_db の宣言。

エンドツーエンドの流れ

1. **初期化とオープン**: dav_propdb_open が呼び出されると、dav_db 構造体が初期化され、DBM ファイルがオープンされます。データベース内の METADATA キーからバージョン情報と名前空間テーブルが構築されます。 2. **キー生成と操作**: プロパティの読み書きや削除の際、dav_build_key が uri_index を参照して名前空間 ID とプロパティ名から #:name 形式の DBM キーを生成し、dav_dbm_store や dav_dbm_fetch などの関数を通じて DBM へのアクセスが行われます。 3. **反復処理**: プロパティの走査時には、iter フィールド（apr_datum_t）を使用して DBM 内のキーを順次走査し、メタデータキーをスキップしながらプロパティを抽出します。 4. **クローズ**: 処理終了時、dav_dbm_close により DBM ファイルが閉じられ、リソースが解放されます。

※なお、mod_dav メインモジュール（modules/dav/main）と mod_dav_fs（modules/dav/fs）の間で、この構造体がどのように受け渡され、全体としてどのようなライフサイクルを辿るかの詳細なエンドツーエンドのフローは、提供されたコンテキストからは完全には明らかになっていません。

重要な不変条件

  • **メモリプールとの紐付け**: pool フィールドは、データベース接続のライフサイクルを管理する apr_pool_t に紐付いていなければなりません。
  • **名前空間の同期**: 名前空間テーブルが変更された場合、ns_table_dirty フラグがセットされ、データベースへ書き戻す必要があることを示します。
  • **キー生成バッファの一時性**: wb_key は dav_build_key で使用される静的バッファ（dav_buffer）として機能するため、生成されたキーは呼び出し直後にコピーして使用する必要があります。

よくある落とし穴

  • **メモリリーク**: dav_db は apr_pool_t に依存しているため、プール解放時に適切にクローズ（dav_dbm_close）されない場合、リソースリークが発生します。
  • **並行性問題**: DBM の性質上、読み取りと書き込みが重なる際の再試行ロジックが不足しており、高負荷環境下での並行書き込み時に整合性が損なわれる可能性があります。
  • **バッファの競合**: dav_build_key が使用する wb_key は静的バッファであるため、結果を即座にコピーしないと、後続の呼び出しによってデータが上書きされる危険性があります。
  • **不完全なバージョン管理**: METADATA のマイナーバージョン更新ロジックは現在不完全です。
