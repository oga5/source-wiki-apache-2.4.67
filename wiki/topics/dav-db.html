dav_db

概念

dav_db は、Apache HTTP Server（httpd-2.4.67）の WebDAV モジュール群（特に mod_dav_fs）において、ファイルシステムリポジトリのプロパティ（Dead Properties）を管理するための、DBM データベース接続状態および関連メタデータを保持する構造体です。

なぜ重要か

WebDAV では、リソース（ファイルやディレクトリ）に関連付けられた任意のプロパティ（Dead Properties）を永続化する必要があります。dav_db は、DBM ファイルへのアクセス、XML 名前空間テーブルの管理、および DBM キーの最適化（名前空間 URI からインデックスへのマッピングなど）を一元的に管理し、プロパティの効率的な読み書きや走査を実現するために重要な役割を果たします。

エンドツーエンドの流れ

WebDAV リクエストの受信から DBM への最終的な書き込み、およびレスポンス返却に至る完全なエンドツーエンドのコールスタックやライフサイクルは、提供された情報からはまだ完全には明らかになっていません。

ただし、dbm.c 内における dav_db を介した部分的な処理の流れは以下の通りです： 1. **初期化**: dav_propdb_open により dav_db が初期化され、METADATA キーからバージョン情報と名前空間テーブルが構築されます。 2. **キー生成**: dav_build_key が uri_index を参照し、名前空間 ID とプロパティ名から #:name 形式の DBM キーを生成します。 3. **操作**: dav_dbm_fetch、dav_dbm_store、dav_dbm_delete などの関数を介して、DBM ファイルへの読み書きや削除が行われます。 4. **走査**: iter フィールドを使用し、dav_dbm_firstkey や dav_dbm_nextkey を通じて DBM 内のキーを順次走査し、メタデータキーをスキップしながらプロパティを抽出します。 5. **クローズ**: dav_dbm_close によりデータベース接続が閉じられます。

重要な不変条件

  • **メモリプールとの紐付け**: pool フィールドは、データベース接続のライフサイクルを管理する apr_pool_t に適切に紐付いていなければなりません。
  • **名前空間テーブルの同期**: 名前空間テーブルが変更された場合、ns_table_dirty フラグがセットされ、データベースへ書き戻す必要があります。
  • **キー生成バッファの一時性**: wb_key は dav_build_key で使用される静的バッファ（dav_buffer）として機能するため、呼び出し直後に結果をコピーする必要があります。

よくある落とし穴

  • **リソースリーク**: dav_db は apr_pool_t に依存しているため、プール解放時に適切にクローズ（dav_dbm_close）されない場合、リソースリークが発生します。
  • **並行性問題**: DBM の性質上、読み取りと書き込みが重なる際の再試行ロジックが不足しており、高負荷環境下での並行書き込み時に整合性が損なわれる可能性があります。
  • **バッファの競合**: dav_build_key が使用する wb_key は静的バッファであるため、結果を即座にコピーしないと、後続の呼び出しによってデータが上書きされる危険性があります。
  • **不完全なバージョン管理**: METADATA のマイナーバージョン更新ロジックは現在不完全です。
