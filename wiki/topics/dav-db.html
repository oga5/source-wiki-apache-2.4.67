dav_db

概念

dav_db は、Apache HTTP Server の WebDAV モジュール（特に mod_dav_fs）において、ファイルシステムリポジトリのプロパティ（Dead Properties）を管理するための構造体です。DBM（Database Manager）データベースへの接続状態、および関連するメタデータ（名前空間テーブルやインデックスなど）を保持する役割を担います。

なぜ重要か

WebDAV では、リソースに関連付けられた任意のメタデータ（プロパティ）を永続化する必要があります。dav_db は、これらのプロパティを DBM ファイルに効率的に格納・取得・管理するための中心的なデータ構造です。XML 名前空間のインデックス化や、DBM キーの最適化（#:name 形式のキー生成など）を行うことで、プロパティ管理のパフォーマンスと整合性を支えています。

エンドツーエンドの流れ

詳細なエンドツーエンドの全体フロー（HTTP リクエストの受信からレスポンスの返却まで）は、提供されたコンテキストからは完全には明らかになっていません。しかし、dbm.c 内における部分的な処理の流れは以下の通りです。

1. **初期化とオープン**: dav_propdb_open により dav_db が初期化され、DBM ファイルがオープンされます。この際、データベース内の METADATA キーからバージョン情報や名前空間テーブル（ns_table）が読み込まれ、構築されます。 2. **キーの生成**: プロパティの読み書きの際、dav_build_key が uri_index を参照し、名前空間 ID とプロパティ名から #:name 形式の DBM キーを生成します。 3. **データの操作**: dav_dbm_store や dav_dbm_fetch などの関数を介して、DBM ファイルへの書き込みや読み込みが行われます。名前空間テーブルに変更があった場合は ns_table_dirty フラグが立ち、データベースへ書き戻されます。 4. **反復処理**: iter フィールドを使用して DBM 内のキーを順次走査し、メタデータキーをスキップしながらプロパティを抽出します。 5. **クローズ**: dav_dbm_close により、DBM 接続が閉じられ、リソースが解放されます。

重要な不変条件

  • **ライフサイクルの一致**: pool フィールドは、データベース接続のライフサイクルを管理する apr_pool_t に紐付いており、このプールが有効な間のみ dav_db の操作が保証されます。
  • **名前空間の同期**: ns_table_dirty フラグは、メモリ上の名前空間テーブルが変更された際に、データベース（METADATA）へ書き戻す必要があることを示す不変のシグナルとして機能します。
  • **キー生成バッファの制約**: wb_key は dav_build_key で使用される静的バッファ（dav_buffer）であり、一時的なキー生成にのみ使用されます。

よくある落とし穴

  • **メモリリーク**: dav_db は apr_pool_t に依存しているため、プールが解放される前に適切にクローズ（dav_dbm_close）されない場合、リソースリークが発生する可能性があります。
  • **並行書き込み時の不整合**: DBM の特性上、複数のプロセスやスレッドから同時に読み書きが発生した際の再試行ロジックが不足しており、高負荷環境下でデータの整合性が損なわれるリスクがあります。
  • **静的バッファの再利用**: dav_build_key が使用する wb_key は静的バッファであるため、関数を呼び出した直後にその結果（生成されたキー）を別の場所にコピーしないと、後続の呼び出しによってデータが上書きされる危険性があります。
  • **未完全なバージョン管理**: METADATA のマイナーバージョン更新ロジックは現在不完全であるため、将来的なスキーマ変更の際に注意が必要です。
