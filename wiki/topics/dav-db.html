dav_db

概念

dav_db は、Apache HTTP Server の WebDAV モジュール（mod_dav）において、WebDAV プロパティ（Dead Properties）などのメタデータを永続化するためのデータベース（DBM）接続状態やコンテキストを管理・抽象化する構造体です。

modules/dav/main/mod_dav.h では不透明な構造体（Opaque Structure）として前方宣言されており、具体的なデータ構造の定義は modules/dav/fs/dbm.c などの各ストレージプロバイダーの実装に隠蔽されています。

なぜ重要か

WebDAV プロパティデータベースの具体的な実装詳細（DBM ファイルハンドル、名前空間テーブル、URI インデックスなど）を mod_dav コアから隠蔽するためです。これにより、コアモジュールは具体的なデータベースの仕組みに依存することなく、抽象化されたインターフェースを介してプロパティの読み書きや削除を行うことができます。

関与するモジュール

  • module:modules/dav/fs (DBM を用いた具体的なファイルシステムリポジトリの実装)
  • module:modules/dav/main (WebDAV コアの定義と抽象インターフェース)

主要エンティティ

  • **symbol:modules/dav/main/mod_dav.h#dav_db**
  • typedef struct dav_db dav_db; として前方宣言された不透明な構造体。データベース操作関数（dav_dbm_fetch, dav_dbm_store など）の引数として使用されます。
  • **symbol:modules/dav/fs/dbm.c#dav_db**
  • mod_dav_fs における具体的な構造体定義。以下の主要なフィールドを保持します。
  • apr_pool_t *pool: データベース接続のライフサイクルを管理するメモリプール。
  • apr_dbm_t *file: 基盤となる DBM ファイルハンドル。
  • dav_buffer ns_table / short ns_count / int ns_table_dirty: XML 名前空間 URI を保持・管理し、DBM キーの最適化に使用されるテーブルとフラグ。
  • apr_hash_t *uri_index: URI からテーブルインデックスへのマッピングを保持するハッシュテーブル。
  • dav_buffer wb_key: キー生成用の静的バッファ。
  • apr_datum_t iter: DBM 内のキーを順次走査するためのイテレータ状態。

エンドツーエンドの流れ

提供された情報から確認できる操作の流れは以下の通りです。

1. **初期化とオープン**: dav_propdb_open（または dav_fs_dbm_open）により dav_db が初期化され、DBM ファイルがオープンされます。この際、METADATA キーからバージョン情報や名前空間テーブル（ns_table）が構築されます。 2. **キーの生成**: プロパティの読み書きの際、dav_build_key が uri_index を参照し、名前空間 ID とプロパティ名から #:name 形式の DBM キーを生成します。 3. **データの操作**: dav_dbm_fetch、dav_dbm_store、dav_dbm_delete などの関数に dav_db ポインタを渡すことで、データの読み書きや削除が行われます。 4. **反復処理**: dav_dbm_firstkey および dav_dbm_nextkey を使用し、iter フィールドを介して DBM 内のキーを順次走査します。 5. **クローズ**: dav_dbm_close により DBM ファイルが閉じられ、リソースが解放されます。

※ WebDAV リクエストの受信からレスポンス送信に至る完全なエンドツーエンドの呼び出しフロー全体は、提供されたコンテキストからは未解明です。

重要な不変条件

  • **ライフサイクルの紐付け**: dav_db は apr_pool_t に紐付いており、このプールの生存期間中のみ有効です。
  • **名前空間テーブルの同期**: 名前空間テーブルが変更された場合、ns_table_dirty フラグがセットされ、データベースへ書き戻す必要があることを示します。
  • **キー生成バッファの排他性**: wb_key は dav_build_key で使用される静的バッファとして機能するため、呼び出し側は取得した結果を直ちにコピーする必要があります。

よくある落とし穴

  • **メモリリーク**: dav_db は apr_pool_t に依存しているため、プール解放時に適切にクローズされない場合、DBM リソースのリークが発生します。
  • **並行書き込み時の不整合**: DBM の性質上、読み取りと書き込みが重なる際の再試行ロジックが不足しており、高負荷環境下での並行書き込み時に整合性が損なわれる可能性があります。
  • **バッファの再利用**: dav_build_key が使用する wb_key は静的バッファであるため、連続して呼び出すと以前のデータが上書きされます。
  • **不完全なバージョン管理**: METADATA のマイナーバージョン更新ロジックは現在不完全です。
