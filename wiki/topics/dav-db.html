dav_db

概念

dav_db は、Apache HTTP Server の WebDAV モジュール（特に mod_dav_fs）において、ファイルシステムリポジトリのデッドプロパティ（Dead Properties）を管理するための DBM データベース接続状態および関連するメタデータを保持する構造体です。

なぜ重要か

WebDAV プロトコルでは、リソースに関連付けられた任意のメタデータ（プロパティ）を永続化する必要があります。dav_db は、DBM ファイルへのアクセス、XML 名前空間のインデックス管理、およびプロパティキーの効率的な生成・走査をカプセル化することで、プロパティ管理の基盤を提供します。

エンドツーエンドの流れ

1. **初期化とオープン**: dav_propdb_open により dav_db 構造体が初期化され、DBM ファイルがオープンされます。この際、データベース内の METADATA キーからバージョン情報と名前空間テーブル（ns_table）が構築されます。 2. **キーの生成と操作**: プロパティの読み書き（dav_dbm_fetch, dav_dbm_store など）の際、dav_build_key が uri_index を参照して名前空間 ID とプロパティ名から #:name 形式の DBM キーを生成します。 3. **反復処理**: プロパティの一覧取得時には、iter フィールドを使用して DBM 内のキーを順次走査し、メタデータキーをスキップしながらプロパティを抽出します。 4. **クローズと永続化**: dav_dbm_close により、必要に応じて変更された名前空間テーブルがデータベースに書き戻され、DBM ファイルが閉じられます。

*注意: 上位の WebDAV リクエスト（PROPFIND/PROPPATCH など）からこれらの一連の関数が呼び出される具体的なエンドツーエンドの完全な制御フローは、提供されたコンテキストからは完全には特定されていません。*

重要な不変条件

  • **メモリプールとの紐付け**: pool フィールドは、データベース接続のライフサイクルを管理する apr_pool_t に紐付いていなければなりません。
  • **名前空間の同期**: ns_table_dirty フラグは、メモリ上の名前空間テーブルが変更された際に、データベースの METADATA へ書き戻す必要があることを示す不変のシグナルとして機能します。
  • **キー生成バッファの一時性**: wb_key は dav_build_key で使用される静的バッファ（作業用バッファ）として機能するため、生成されたキーは即座に消費されるか、呼び出し直後にコピーされる必要があります。

よくある落とし穴

  • **リソースリーク**: dav_db は apr_pool_t に依存しているため、プールが解放される前に適切にクローズ（dav_dbm_close）されない場合、DBM ファイルハンドルなどのリソースリークが発生します。
  • **並行書き込み時の不整合**: DBM の性質上、複数のプロセスやスレッドから同時に読み書きが行われる際の再試行ロジックが不足しており、高負荷環境下でデータの整合性が損なわれる可能性があります。
  • **バッファの上書き**: dav_build_key が使用する wb_key は共有のバッファであるため、結果を退避させずに連続して呼び出すと、以前に生成されたキーが上書きされる危険性があります。
  • **不完全なバージョン管理**: METADATA のマイナーバージョン更新ロジックは現在不完全であるため、将来的なスキーマ変更の際に注意が必要です。
