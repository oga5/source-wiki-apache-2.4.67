ap_socache_instance_t

概念

ap_socache_instance_t は、Apache HTTP Server (httpd) の小規模オブジェクトキャッシュ（socache）アーキテクチャにおいて、各キャッシュプロバイダー（バックエンド実装）が個別のキャッシュインスタンスの状態や実行時コンテキストを保持するための構造体です。

include/ap_socache.h においては不透明な（opaque）型として宣言されており、具体的な構造体の定義は各プロバイダー実装（DBM、distcache、Memcached、Redis、SHMCBなど）のソースコード内に隠蔽されています。

なぜ重要か

この型は、上位モジュール（mod_authn_socache や mod_cache_socache など）に対して統一的なキャッシュ操作インターフェース（ap_socache_provider_t）を提供しつつ、バックエンドごとの固有情報（ファイルパス、接続ハンドル、共有メモリセグメント、名前空間タグなど）をカプセル化するために重要です。これにより、上位モジュールは具体的なキャッシュ実装の詳細に依存することなく、同一のAPIを通じて多様なキャッシュバックエンドを透過的に利用できます。

主要エンティティ

  • **symbol:include/ap_socache.h#ap_socache_instance_t**

    • 不透明な型としての共通定義。各プロバイダーメソッドの第一引数として受け渡しされます。

  • **symbol:modules/cache/mod_socache_dbm.c#ap_socache_instance_t**

    • DBMベースの実装。データファイルパス（data_file）、メモリプール、および有効期限管理用の時刻情報を保持します。

  • **symbol:modules/cache/mod_socache_dc.c#ap_socache_instance_t**

    • distcache（分散キャッシュ）ベースの実装。接続先ターゲット文字列と distcache ライブラリのクライアントコンテキスト（DC_CTX）を保持します。

  • **symbol:modules/cache/mod_socache_memcache.c#ap_socache_instance_t**

    • Memcachedベースの実装。サーバーリスト、apr_memcache_t クライアントオブジェクト、およびキー衝突を防ぐための名前空間タグを保持します。

  • **symbol:modules/cache/mod_socache_redis.c#ap_socache_instance_t**

    • Redisベースの実装。サーバーリスト、apr_redis_t クライアントハンドル、および名前空間タグを保持します。

  • **symbol:modules/cache/mod_socache_shmcb.c#ap_socache_instance_t**

    • 共有メモリ（SHMCB）ベースの実装。共有メモリセグメント（apr_shm_t）やメタデータ管理用のヘッダー（SHMCBHeader）へのポインタを保持します。

エンドツーエンドの流れ

1. **生成**: 上位モジュールがプロバイダーの create メソッドを呼び出し、各プロバイダー固有の ap_socache_instance_t インスタンスがメモリプールから確保されます。 2. **初期化**: init メソッドが呼び出され、接続の確立、共有メモリの確保、あるいはファイルのオープンなど、バックエンドに応じた初期化処理が行われます。 3. **操作**: store, retrieve, remove などのAPI呼び出し時に、このインスタンスへのポインタが第一引数として渡され、キャッシュデータの読み書きが行われます。 4. **破棄**: destroy メソッド（またはメモリプールのクリーンアップ）により、インスタンスに関連付けられたリソースが解放されます。

※上位モジュールとプロバイダー間の厳密なインタラクションの全容や、エラーハンドリングを含む完全なエンドツーエンドの制御フローは、提供されたコンテキストからは完全には特定されていません。

重要な不変条件

  • **初期化の必須性**: init メソッドによる初期化が成功するまで、インスタンスをキャッシュ操作に使用してはなりません（例: mod_socache_dc の dc コンテキストや mod_socache_memcache の mc ハンドルは初期化前は無効です）。
  • **スレッドセーフティと排他制御**:
  • mod_socache_dbm では、pool メンバへのアクセスは対応する mutex を保持している間のみ許可されます。
  • mod_socache_shmcb は AP_SOCACHE_FLAG_NOTMPSAFE を指定しており、インスタンス自体にロック機構を持たないため、呼び出し側での排他制御が前提となります。
  • **名前空間タグの制限**:
  • mod_socache_redis では、デバッグビルドにおいて taglen（strlen(tag) + 1）が16バイト以下でなければなりません（AP_DEBUG_ASSERT による検証）。
  • mod_socache_memcache では、タグを含むキー長が MC_KEY_LEN（254バイト）を超えてはなりません。

よくある落とし穴

  • **排他制御の欠如**: mod_socache_dbm や mod_socache_shmcb など、スレッドセーフでないプロバイダーを使用する際、呼び出し側で適切なロックを確保せずにアクセスすると、メモリ競合や未定義動作を引き起こす可能性があります。
  • **リソースリーク**: mod_socache_memcache の socache_mc_destroy は現在 no-op（何もしない）とされていますが、将来的なリソース解放の設計時にはメモリプールとのライフサイクルの不整合に注意する必要があります。
  • **サイズ制限とオーバーフロー**:
  • mod_socache_dbm では、ファイルシステムの制限により約1KB程度のデータサイズ制限を受ける場合があります。
  • mod_socache_shmcb では、shm_size が極端に大きい場合に内部のサイズ計算で unsigned int のオーバーフローが発生するリスクが指摘されています。
