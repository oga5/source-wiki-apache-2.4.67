ap_socache_instance_t

概念

ap_socache_instance_t は、Apache HTTP Server (httpd-2.4.67) の小規模オブジェクトキャッシュ（socache）アーキテクチャにおいて、各キャッシュバックエンド（プロバイダー）が個別のキャッシュインスタンスの状態や実行時コンテキストを保持するための構造体です。

include/ap_socache.h においては不透明な（opaque）型として宣言されており、具体的な構造体のメンバー定義は各プロバイダー実装（DBM、distcache、Memcached、Redis、SHMCBなど）の内部に隠蔽されています。

なぜ重要か

この型があることで、上位のキャッシュ利用モジュール（mod_authn_socache や mod_cache_socache など）は、バックエンドの具体的なデータ構造や接続管理方法（ファイルパス、メモリプール、ソケット、共有メモリセグメントなど）を意識する必要がなくなります。

上位モジュールは ap_socache_provider_t で定義された共通インターフェースを介して、このインスタンスへのポインタを受け渡すだけで、バックエンドに依存しない一貫したキャッシュ操作を行うことができます。

エンドツーエンドの流れ

1. **生成 (Create)**: 上位モジュールがプロバイダーの create メソッドを呼び出し、各プロバイダー固有の ap_socache_instance_t 構造体をメモリプール上に確保します。 2. **初期化 (Init)**: init メソッドを呼び出し、接続の確立（Redis/Memcached/distcache）、共有メモリの確保（SHMCB）、またはファイルパスの設定（DBM）などを行い、インスタンスを実行可能な状態にします。 3. **操作 (Store/Retrieve/Remove)**: キャッシュの読み書きを行う際、上位モジュールは操作対象を特定するために、このインスタンスへのポインタを第一引数として各メソッドに渡します。 4. **破棄 (Destroy)**: キャッシュのライフサイクル終了時に destroy メソッドを呼び出し、リソース（distcache の DC_CTX など）を解放します（プロバイダーによっては no-op の場合もあります）。

※ 上位モジュールにおける具体的な呼び出し順序の完全な制御フローや、スレッド間での同期のライフサイクル全体は、提供されたモジュール情報からは完全には特定されていません。

重要な不変条件

各プロバイダー実装において、以下の不変条件が維持される必要があります。

  • **共通**:
  • create によって生成され、init が正常に完了するまで、キャッシュ操作（store, retrieve 等）にインスタンスを使用してはなりません。
  • **mod_socache_dbm**:
  • pool メンバーは、対応する mutex が保持されている間のみ使用可能です。
  • **mod_socache_memcache**:
  • taglen は常に strlen(tag) + 1 と一致していなければなりません。
  • **mod_socache_redis**:
  • taglen は 16 バイト以下でなければなりません（デバッグビルドでは AP_DEBUG_ASSERT で検証されます）。
  • **mod_socache_shmcb**:
  • shm は apr_shm_t を介して確保された有効な共有メモリ領域を指し、header はその先頭に配置された SHMCBHeader 構造体を指さなければなりません。

よくある落とし穴

  • **排他制御の欠如**:

    • mod_socache_dbm はスレッドセーフではなく、mod_socache_shmcb も AP_SOCACHE_FLAG_NOTMPSAFE を指定しています。これらのインスタンスを操作する際は、呼び出し側で適切な排他制御（ロック）を行わないと、メモリプールの破損や未定義動作を引き起こす原因になります。

  • **サイズ制限とオーバーフロー**:
  • DBM 実装では、ファイルシステム側の制限により、格納できるデータサイズが約 1KB 程度に制限される場合があります。
  • SHMCB 実装では、設定された shm_size が極端に大きい場合、内部のサイズ計算において unsigned int のオーバーフローが発生するリスクが指摘されています。
  • **タグの長さ制限**:

    • Redis 実装では tag の長さが 16 バイトを超えるとアサーションエラーが発生し、Memcached 実装ではキー全体の長さが MC_KEY_LEN（254バイト）を超えるとエラーになります。
