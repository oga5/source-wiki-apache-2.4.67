ap_socache_instance_t

概念

ap_socache_instance_t は、Apache HTTP Server（httpd-2.4.67）の小規模オブジェクトキャッシュ（socache）機構において、各キャッシュバックエンド（プロバイダー）が個別のキャッシュインスタンスの状態や実行時コンテキストを保持するための不透明な（opaque）構造体型です。

ヘッダーファイル include/ap_socache.h で前方宣言されており、具体的な構造体の定義は各プロバイダーモジュールのソースコード内に隠蔽されています。

なぜ重要か

ap_socache_instance_t は、キャッシュの具体的な実装詳細（共有メモリ、DBMファイル、外部キャッシュサーバーへの接続ハンドルなど）を呼び出し側から隠蔽する役割を担います。

mod_authn_socache や mod_cache_socache などの上位モジュールは、この不透明なポインタを保持し、ap_socache_provider_t インターフェースを介して間接的に操作することで、バックエンドの実装に依存しない共通のAPIでキャッシュを制御できます。

関与するモジュール

  • module:include（共通インターフェース定義）
  • module:modules/cache（各プロバイダー実装）

主要エンティティ

  • **symbol:include/ap_socache.h#ap_socache_instance_t**

    • 不透明な構造体としての共通定義。

  • **symbol:modules/cache/mod_socache_dbm.c#ap_socache_instance_t**

    • DBMファイルベースのキャッシュインスタンス。データファイルパス（data_file）、メモリプール（pool）、期限切れ処理の管理情報（last_expiry, expiry_interval）を保持します。

  • **symbol:modules/cache/mod_socache_dc.c#ap_socache_instance_t**

    • distcache（分散キャッシュ）用のインスタンス。接続先ターゲット（target）と distcache クライアントコンテキスト（dc）を保持します。

  • **symbol:modules/cache/mod_socache_memcache.c#ap_socache_instance_t**

    • Memcached 用のインスタンス。サーバーリスト（servers）、apr_memcache_t クライアントオブジェクト（mc）、名前空間分離用のタグ情報（tag, taglen）を保持します。

  • **symbol:modules/cache/mod_socache_redis.c#ap_socache_instance_t**

    • Redis 用のインスタンス。サーバーリスト（servers）、apr_redis_t クライアントオブジェクト（rc）、名前空間分離用のタグ情報（tag, taglen）を保持します。

  • **symbol:modules/cache/mod_socache_shmcb.c#ap_socache_instance_t**

    • 共有メモリ（SHMCB）ベースのキャッシュインスタンス。メモリプール（pool）、データファイル（data_file）、共有メモリサイズ（shm_size）、共有メモリセグメント（shm）、およびヘッダー情報（header）を保持します。

エンドツーエンドの流れ

1. **生成**: 呼び出し側モジュールが、ap_socache_provider_t の create メソッド（例: socache_shmcb_create や socache_dbm_create）を呼び出し、ap_socache_instance_t のインスタンスを生成・確保します。 2. **初期化**: キャッシュ操作を行う前に、init メソッドを呼び出してインスタンスを初期化します。この段階で、共有メモリの確保、ファイルのオープン、または外部キャッシュサーバー（Memcached/Redis/distcache）への接続ハンドル設定などが行われます。 3. **キャッシュ操作**: 初期化されたインスタンスへのポインタを第一引数として、store、retrieve、remove などのAPIメソッドに渡し、キャッシュデータの読み書きや削除を行います。 4. **破棄**: キャッシュの利用終了時、destroy メソッドが呼び出されるか、関連付けられたメモリプール（apr_pool_t）が解放されることで、インスタンスおよび関連リソースが破棄されます。

重要な不変条件

  • **初期化の必須性**: init メソッドによる初期化が成功するまで、対象の ap_socache_instance_t を用いたキャッシュ操作（store, retrieve 等）を実行することはできません。
  • **スレッド安全性と排他制御**:
  • mod_socache_dbm や mod_socache_shmcb（AP_SOCACHE_FLAG_NOTMPSAFE を指定）は、それ自体ではスレッドセーフではありません。呼び出し側が適切に排他制御（mutex の保持など）を行う必要があります。
  • 特に mod_socache_dbm の pool メンバは、対応する mutex が保持されている間のみ使用可能です。
  • **タグの長さ制限**:
  • mod_socache_redis では、taglen が 16 バイト以下でなければなりません（デバッグビルドでは AP_DEBUG_ASSERT で検証されます）。
  • mod_socache_memcache では、taglen は strlen(tag) + 1 と一致している必要があります。

よくある落とし穴

  • **排他制御の欠如**: スレッドセーフでないプロバイダー（DBM や SHMCB）において、呼び出し側が適切なロックを取得せずにインスタンスを操作した場合、メモリ競合や未定義の動作が発生する可能性があります。
  • **初期化前のアクセス**: インスタンスが適切に初期化される前に操作メソッドに渡された場合、各プロバイダーの挙動に依存しますが、一般に APR_EGENERAL などのエラーが返されるか、未定義の動作を引き起こします。
  • **サイズ制限とオーバーフロー**:
  • DBM 実装では、ファイルシステム側の制限により、保存できるデータサイズが約 1KB 程度に制限される場合があります。
  • SHMCB 実装では、指定された shm_size が極端に大きい場合、内部のサイズ計算において unsigned int のオーバーフローが発生するリスクが指摘されています。
  • Redis 実装において、名前空間タグ（tag）の長さが 16 バイトを超えるとアサーションエラーになります。また、Memcached 実装においてタグを含むキーの長さが MC_KEY_LEN（254バイト）を超えるとエラーになります。
