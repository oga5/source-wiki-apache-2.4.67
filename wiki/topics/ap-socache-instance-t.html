ap_socache_instance_t

概念

ap_socache_instance_t は、Apache HTTP Server (httpd) の小規模オブジェクトキャッシュ（socache）において、各バックエンド実装（プロバイダー）が個別のキャッシュインスタンスの状態やコンテキストを保持するための不透明な（opaque）構造体型です。

ヘッダーファイル include/ap_socache.h においては、以下のように不透明な型として定義されています。

typedef struct ap_socache_instance_t ap_socache_instance_t;

具体的な構造体のメンバは公開されておらず、各プロバイダー実装（DBM、distcache、Memcached、Redis、SHMCBなど）のソースコード内で個別に定義されています。

なぜ重要か

この型が存在することにより、上位のキャッシュ利用モジュール（mod_authn_socache や mod_cache_socache など）は、バックエンドの具体的なキャッシュ実装（共有メモリ、ファイル、外部キャッシュサーバーなど）の詳細に依存することなく、統一された ap_socache_provider_t インターフェースを介してキャッシュインスタンスを操作できます。これにより、キャッシュ実装の疎結合性と拡張性が確保されています。

関与するモジュール

主要エンティティ

  • **symbol:include/ap_socache.h#ap_socache_instance_t**: 共通インターフェースで定義される不透明な型。
  • **symbol:modules/cache/mod_socache_dbm.c#ap_socache_instance_t**: DBM実装。データファイルパス、メモリプール、期限切れ管理用の時刻を保持します。
  • **symbol:modules/cache/mod_socache_dc.c#ap_socache_instance_t**: distcache（分散キャッシュ）実装。接続先ターゲットとクライアントコンテキスト（DC_CTX）を保持します。
  • **symbol:modules/cache/mod_socache_memcache.c#ap_socache_instance_t**: Memcached実装。サーバーリスト、apr_memcache_t クライアント、名前空間タグを保持します。
  • **symbol:modules/cache/mod_socache_redis.c#ap_socache_instance_t**: Redis実装。サーバーリスト、apr_redis_t クライアント、名前空間タグを保持します。
  • **symbol:modules/cache/mod_socache_shmcb.c#ap_socache_instance_t**: SHMCB（共有メモリ）実装。共有メモリセグメント（apr_shm_t）やメタデータヘッダー（SHMCBHeader）を保持します。

エンドツーエンドの流れ

詳細なエンドツーエンドのライフサイクル全体のフローは完全には判明していませんが、提供された情報に基づく大まかな流れは以下の通りです。

1. **生成 (Create)**: 上位モジュールがプロバイダーの create メソッド（例: socache_mc_create）を呼び出し、各プロバイダー固有の ap_socache_instance_t 構造体をメモリプールから確保します。 2. **初期化 (Init)**: init メソッド（例: socache_rd_init, socache_shmcb_init）を呼び出し、接続の確立、共有メモリの確保、名前空間タグの設定などを行います。 3. **操作 (Store / Retrieve / Remove)**: store や retrieve などの操作時に、このインスタンスへのポインタを第一引数として渡し、キャッシュデータの読み書きを行います。 4. **破棄 (Destroy)**: destroy メソッドを介して、関連リソース（distcacheのコンテキストなど）を解放します（Memcachedのように no-op の場合もあります）。

重要な不変条件

すべてのプロバイダーに共通するグローバルな不変条件の全容は完全には判明していませんが、各プロバイダー固有の重要な不変条件は以下の通りです。

  • **初期化の必須性**: init メソッドが正常に完了するまで、インスタンスをキャッシュ操作に使用することはできません。
  • **スレッド安全性と排他制御**:
  • mod_socache_dbm では、pool メンバは対応する mutex が保持されている間のみ使用可能であり、スレッドセーフではないため呼び出し側での排他制御が前提となります。
  • mod_socache_shmcb は AP_SOCACHE_FLAG_NOTMPSAFE を指定しており、構造体自体にロック機構を持たず、排他制御は呼び出し側に依存します。
  • **Redisにおけるタグ長制限**: mod_socache_redis では、デバッグビルドにおいて taglen が 16 バイト以下である必要があります（AP_DEBUG_ASSERT による検証）。
  • **Memcachedにおけるタグ長制限**: mod_socache_memcache では、tag の長さが MC_KEY_LEN（254バイト）を超えるとキー生成時にエラーとなります。

よくある落とし穴

  • **不透明な型の二重定義**: ap_socache_instance_t は ap_socache.h で不透明な型として宣言されていますが、各 .c ファイルで具体的な構造体が個別に定義されています。構造体を変更する際は、各プロバイダー実装間での整合性に注意する必要があります。
  • **排他制御の欠如**: DBM や SHMCB などのプロバイダーは自身でロックを管理しない（または呼び出し側の排他制御を前提とする）ため、適切な mutex 制御なしにマルチスレッド環境からアクセスすると、未定義動作や競合が発生します。
  • **メモリ管理の依存**: 多くの実装で Apache のメモリプール（apr_pool_t）に依存しているため、プールが解放されるタイミングや、キャッシュ操作時のメモリ消費量に注意を払わないと、メモリリークや不正アクセスにつながる可能性があります。
