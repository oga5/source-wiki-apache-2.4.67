ap_socache_instance_t

概念

ap_socache_instance_t は、Apache HTTP Server (httpd) の小規模オブジェクトキャッシュ（socache）アーキテクチャにおいて、各キャッシュプロバイダー（バックエンド実装）が個々のキャッシュインスタンスの状態や実行時コンテキストを保持するための構造体です。

include/ap_socache.h においては不透明な（opaque）型として宣言されており、具体的な構造体の定義は各プロバイダー実装（DBM、distcache、Memcached、Redis、SHMCBなど）のソースコード内で個別に定義されています。

なぜ重要か

この型は、上位のキャッシュ利用モジュール（mod_authn_socache や mod_cache_socache など）に対して、バックエンドの具体的な実装詳細（共有メモリ、ファイル、外部キャッシュサーバーへの接続など）を隠蔽する役割を持ちます。

これにより、上位モジュールは共通の ap_socache_provider_t インターフェースを介して、バックエンドの実装に依存することなく統一的にキャッシュ操作（create, init, store, retrieve など）を行うことができます。

主要エンティティ

  • **symbol:include/ap_socache.h#ap_socache_instance_t**

    • 不透明な型としての共通定義。

  • **symbol:modules/cache/mod_socache_dbm.c#ap_socache_instance_t**

    • DBMバックエンド用の実装。データファイルパス（data_file）、メモリプール（pool）、期限切れ管理情報（last_expiry, expiry_interval）を保持します。

  • **symbol:modules/cache/mod_socache_dc.c#ap_socache_instance_t**

    • distcacheバックエンド用の実装。接続先ターゲット（target）、distcacheクライアントコンテキスト（dc）を保持します。

  • **symbol:modules/cache/mod_socache_memcache.c#ap_socache_instance_t**

    • Memcachedバックエンド用の実装。サーバーリスト（servers）、apr_memcache クライアント（mc）、名前空間タグ（tag, taglen）を保持します。

  • **symbol:modules/cache/mod_socache_redis.c#ap_socache_instance_t**

    • Redisバックエンド用の実装。サーバーリスト（servers）、apr_redis クライアント（rc）、名前空間タグ（tag, taglen）を保持します。

  • **symbol:modules/cache/mod_socache_shmcb.c#ap_socache_instance_t**

    • 共有メモリベースのキャッシュ（SHMCB）用の実装。共有メモリセグメント（shm）、メタデータヘッダー（header）、サイズ情報（shm_size）を保持します。

エンドツーエンドの流れ

上位モジュールからプロバイダーを介した一般的な利用フローは以下の通りです。ただし、上位モジュール側における完全なライフサイクル管理の全容や、すべてのエラーハンドリングの不変条件は本コンテキストからは未確定です。

1. **生成 (Create)**: 上位モジュールがプロバイダーの create メソッド（例: socache_mc_create, socache_shmcb_create）を呼び出し、メモリプールから ap_socache_instance_t が確保されます。 2. **初期化 (Init)**: init メソッド（例: socache_shmcb_init, socache_rd_init）が呼び出され、共有メモリの確保、外部サーバーへの接続ハンドル（apr_memcache_t, apr_redis_t, DC_CTX）の作成、あるいはファイルパスの設定など、バックエンド固有の初期化処理が行われます。 3. **操作 (Store / Retrieve / Remove)**: キャッシュの保存（store）や取得（retrieve）、削除（remove）の際、このインスタンスへのポインタが第一引数として各プロバイダー関数に渡され、コンテキスト情報（接続ハンドルや共有メモリのアドレスなど）が参照・更新されます。 4. **破棄 (Destroy)**: destroy メソッド（またはメモリプールのクリーンアップ）により、確保されたリソース（外部接続の解放など）が破棄されます。

重要な不変条件

  • **初期化の必須性**:

    • create メソッドによって生成された後、init メソッドが正常に完了するまでは、キャッシュ操作（store, retrieve 等）に使用することはできません。

  • **スレッドセーフティの欠如（一部実装）**:
  • mod_socache_dbm において、pool メンバは対応する mutex が保持されている間のみ使用可能であり、呼び出し側での排他制御が前提となります。
  • mod_socache_shmcb は AP_SOCACHE_FLAG_NOTMPSAFE を指定しており、構造体自体にロック機構を持たず、排他制御は呼び出し側に依存します。
  • **Redisにおけるタグ長制限**:

    • mod_socache_redis では、デバッグビルドにおいて taglen が 16 バイト以下である必要があります（AP_DEBUG_ASSERT による検証）。

  • **Memcachedにおけるタグ長制限**:

    • mod_socache_memcache では、tag の長さが MC_KEY_LEN（254バイト）の制限を超える場合、キー生成時にエラーが発生します。

よくある落とし穴

  • **不透明な型としての二重定義**:

    • ap_socache_instance_t は ap_socache.h で不透明な型として宣言されている一方で、各 .c ファイルで具体的な構造体として再定義されています。構造体を変更する際は、共通インターフェースとの整合性や、他のプロバイダー実装への影響に注意する必要があります。

  • **排他制御の欠如**:

    • mod_socache_dbm や mod_socache_shmcb など、一部のプロバイダーは自身でスレッドセーフな排他制御を提供していません。呼び出し側（上位モジュール）が適切にロックを管理しない場合、メモリ競合や未定義動作が発生します。

  • **サイズ制限とオーバーフロー**:
  • mod_socache_shmcb では、shm_size が非常に大きい場合に内部のサイズ計算で unsigned int のオーバーフローが発生するリスクが指摘されています。
  • mod_socache_dbm では、ファイルシステムやDBMの制限により、約 1KB 程度のデータサイズ制限を受ける場合があります。
  • **接続の永続性**:

    • mod_socache_dc では、現在 SESSION_CTX_FLAG_PERSISTENT が無効化されており、接続は操作ごとに確立される設計となっています。
