dav_lock_discovery_fixed

概念

dav_lock_discovery_fixed は、WebDAVの直接ロック（direct lock）情報をDBMデータベースへ永続化（シリアライズ・デシリアライズ）する際に、固定長属性を保持するために使用されるデータ構造（struct）です。

この構造体は、ロックのスコープ（scope）、タイプ（type）、深度（depth）、および有効期限（timeout）といった固定長データをバイナリ形式で保存・復元するための共通のデータレイアウトとして機能します。

なぜ重要か

WebDAVのロック情報を効率的に永続化するためには、メモリ上の構造体とDBMファイルなどのバイナリ表現との間で高速かつ正確な変換が必要となります。

dav_lock_discovery_fixed は、可変長データを含むロック情報（dav_lock_discovery）のうち、固定長部分を明確に分離して定義することで、シリアライズ時のオフセット計算やバイナリデータの直接的なキャスト・コピーを容易にし、ロック管理のパフォーマンス向上と実装の簡素化に貢献しています。また、この構造体は複数のモジュールで同様に定義されており、システム全体でのロック情報の整合性を保つための共通のインターフェースとなっています。

エンドツーエンドの流れ

1. **ロック情報の保存（シリアライズ）**: WebDAVのロック保存処理（例: dav_generic_save_lock_record）において、メモリ上のロック構造体から dav_lock_discovery_fixed の各フィールドへ値がコピーされます。 2. **DBMへの書き込み**: dav_size_direct マクロ等を用いて算出されたサイズに基づき、本構造体のバイナリデータがDBMレコードの一部（固定長ヘッダー）として書き込まれます。 3. **ロック情報の読み込み（デシリアライズ）**: DBMからの読み込み処理（例: dav_generic_load_lock_record）において、バイナリデータから本構造体へキャストまたはコピーされ、有効期限やスコープなどの固定長属性が復元されます。

*注意: 詳細な関数呼び出しの順序や、DBM書き込み・読み込みの完全なエンドツーエンドの制御フロー全体は、提供されたコンテキストからは完全には明らかになっていません。*

重要な不変条件

  • **サイズ計算の同期**: dav_size_direct マクロによるサイズ計算（本構造体のサイズに1バイトを加えた値など）は、シリアライズ時のオフセット計算に利用されており、構造体の実際のサイズと常に同期していなければなりません。
  • **構造体の同一性**: modules/dav/fs/lock.c と modules/dav/lock/locks.c の両方に同名の構造体が定義されており、両者のフィールド定義（char scope, char type, int depth, time_t timeout）およびレイアウトは完全に一致している必要があります。

よくある落とし穴

  • **バイナリ互換性の喪失**: 本構造体のサイズやフィールド順序を変更すると、既存のDBMファイルに保存されているバイナリデータとの互換性が失われ、古いロック情報の読み込みに失敗します。
  • **プラットフォーム依存のパディングと型サイズ**: time_t や int のサイズ、およびコンパイラによる構造体のパディング（アライメント）はプラットフォームやアーキテクチャによって異なります。これにより、異なる環境間でDBMファイルを移行した際にオフセットの不整合やデータ破損が発生するリスクがあります。
  • **重複定義の更新漏れ**: 2つのファイル（modules/dav/fs/lock.c と modules/dav/lock/locks.c）に個別に定義されているため、一方のみを変更してもう一方を更新し忘れると、モジュール間でロックデータの解釈に不整合が生じます。
