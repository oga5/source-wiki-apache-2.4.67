set_cookie_name

概念

set_cookie_name は、Apache HTTP Server（httpd）の複数のモジュール（mod_usertrack、mod_session_cookie、および mod_session_dbd）において、Cookieの名前や関連する属性を設定・検証するために使用される設定ディレクティブハンドラ（関数）です。各モジュールで同名の静的（static）関数として定義されており、それぞれのコンテキストに応じたデータ構造の初期化と検証ロジックを実行します。

なぜ重要か

Apacheの設定ファイル（httpd.conf など）で指定される CookieName、SessionCookieName、SessionDBDCookieName といったディレクティブの入力を適切に処理し、システムが正しくCookieを識別・追跡・検証できるようにするために重要です。

  • **mod_usertrack** では、ユーザートラッキング用のCookieを検索するための正規表現を構築・検証します。
  • **mod_session_cookie** では、セッション管理用のCookie名とオプションの属性を解析・検証し、セッション設定構造体に格納します。

関与するモジュール

提供された情報に基づき、以下のモジュールが関与します。

  • module:modules/metadata (mod_usertrack.c)
  • module:modules/session (mod_session_cookie.c, mod_session_dbd.c)

主要エンティティ

  • **symbol:modules/metadata/mod_usertrack.c#set_cookie_name**:
  • CookieName ディレクティブ（AP_INIT_TAKE1）に対応するハンドラ。
  • cookie_dir_rec 構造体の cookie_name を設定し、Cookie検索用の正規表現をコンパイル・検証します。
  • **symbol:modules/session/mod_session_cookie.c#set_cookie_name**:
  • SessionCookieName ディレクティブ（AP_INIT_RAW_ARGS）に対応するハンドラ。
  • session_cookie_dir_conf 構造体にCookie名と属性を格納し、check_string で検証します。
  • **symbol:modules/session/mod_session_dbd.c#set_cookie_name**:
  • SessionDBDCookieName ディレクティブ（AP_INIT_RAW_ARGS）に対応するハンドラ。詳細な実装は提供された情報からは未検証です。

エンドツーエンドの流れ

設定ファイルの解析フェーズにおいて、Apacheのコアが各ディレクティブを検出した際に、対応する set_cookie_name 関数が呼び出されます。

1. mod_usertrack における流れ

1. CookieName ディレクティブが検出されると、mod_usertrack.c の set_cookie_name が呼び出されます。 2. 引数として渡された name を cookie_dir_rec の cookie_name に設定します。 3. set_and_comp_regexp を呼び出して、Cookie検索用の正規表現をコンパイルし、dcfg->regexp に格納します。 4. コンパイルされた正規表現のキャプチャグループ数が NUM_SUBS と一致することを確認します。

2. mod_session_cookie における流れ

1. SessionCookieName ディレクティブが検出されると、mod_session_cookie.c の set_cookie_name が呼び出されます。 2. 引数文字列 args を apr_pstrdup で複製します。 3. apr_strtok を用いて最初の空白までを Cookie 名として抽出し、conf->name に格納して conf->name_set フラグを 1 に設定します。 4. 残りの文字列から先頭の空白を除去し、conf->name_attrs（Cookie属性）として格納します。 5. check_string を呼び出して、Cookie 名に不正な文字が含まれていないか検証します。

3. mod_session_dbd における流れ

  • SessionDBDCookieName ディレクティブが検出された際に呼び出されますが、具体的な内部処理の流れは提供された情報からは未検証です。

重要な不変条件

  • **mod_usertrack における不変条件**:
  • set_and_comp_regexp によって生成される正規表現のキャプチャグループ数（re_nsub + 1）は、必ず NUM_SUBS（値は3）と一致しなければなりません。一致しない場合は設定エラーとなります。
  • **mod_session_cookie における不変条件**:
  • Cookie名には = や & などの不正な文字が含まれてはなりません（check_string による検証）。
  • 抽出されたCookie名は、設定のライフサイクル中保持されるように、Apacheのプールメモリ（apr_pstrdup）に複製されている必要があります。

よくある落とし穴

  • **モジュール間の同名関数の混同**:

    • set_cookie_name という名前の関数は、mod_usertrack.c、mod_session_cookie.c、および mod_session_dbd.c のそれぞれに独立して定義されています。これらは静的（static）関数ですが、保守時にこれらが単一の共有関数であると誤解しないよう注意が必要です。

  • **正規表現の不整合 (mod_usertrack)**:

    • mod_usertrack において、NUM_SUBS は正規表現の構造に直結しているため、正規表現の生成ロジック（set_and_comp_regexp）を変更する際は、NUM_SUBS との整合性を維持しなければ設定エラーを引き起こします。

  • **無効な文字の混入 (mod_session_cookie)**:

    • SessionCookieName に = や & を含む文字列を指定すると、check_string による検証でエラーとなり、起動時または設定再読み込み時に失敗します。
