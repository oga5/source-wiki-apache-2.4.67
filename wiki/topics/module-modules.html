Module: modules

概念

modules ディレクトリは、Apache HTTP Server (httpd) のコア機能を拡張するための多種多様なモジュール群を格納する、サーバーアーキテクチャの中核的なコンポーネントです。

httpd はコア部分を軽量に保ち、認証、キャッシュ、プロトコル処理、フィルタリング、プロキシ、セキュリティなどの具体的な機能のほとんどを独立したモジュールとして実装しています。これらは、静的にビルドされるか、あるいは動的共有オブジェクト (DSO) として実行時にロードされます。

なぜ重要か

httpd の柔軟性と拡張性は、このモジュール構造によって実現されています。 管理者は必要な機能を提供するモジュールのみを選択してロード（またはコンパイル）できるため、サーバーのメモリフットプリントを最小限に抑え、セキュリティ攻撃面を減らすことができます。また、開発者はコアコードを修正することなく、新しいプロトコル（例: HTTP/2）やカスタムの認証方式、フィルタなどを追加できます。

関与するモジュール

提供された構成に基づき、以下のサブモジュール（カテゴリ）が modules 配下に関与しています。

  • **modules/aaa**: 認証 (Authentication)、承認 (Authorization)、アクセス制御 (Access Control) を担当（mod_auth_basic, mod_authz_core など）。
  • **modules/arch**: プラットフォーム固有の統合処理（NetWare, Unix, Windows）。
  • **modules/cache**: コンテンツキャッシュおよびストレージ管理（mod_cache, mod_file_cache など）。
  • **modules/cluster**: クラスタリングと負荷監視（mod_heartbeat, mod_heartmonitor）。
  • **modules/core**: 動的ロード（mod_so）やバックグラウンドタスク（mod_watchdog）などのコア拡張。
  • **modules/database**: SQL データベース接続の管理（mod_dbd）。
  • **modules/dav**: WebDAV プロトコルのサポート（mod_dav, mod_dav_fs など）。
  • **modules/debugging**: データのダンプやデバッグ用フィルタ（mod_dumpio など）。
  • **modules/echo**: 単純なエコープロトコルの実装（mod_echo）。
  • **modules/examples**: 開発者向けのフックや IPC のサンプルコード。
  • **modules/filters**: データの圧縮や書き換えなどのフィルタリング（mod_deflate, mod_filter, mod_substitute など）。
  • **modules/generators**: CGI やステータス表示などのコンテンツ生成（mod_cgi, mod_status など）。
  • **modules/http**: HTTP プロトコルの基本処理（http_protocol.c, http_filters.c など）。
  • **modules/http2**: HTTP/2 プロトコルおよび接続・ストリーム管理（mod_http2）。
  • **modules/ldap**: LDAP 接続とキャッシュの管理（util_ldap）。
  • **modules/loggers**: アクセスログやデバッグログの記録（mod_log_config など）。
  • **modules/lua**: Lua スクリプトによるリクエスト処理の拡張（mod_lua）。
  • **modules/mappers**: URL からファイルパスへのマッピングや書き換え（mod_rewrite, mod_alias など）。
  • **modules/md**: ACME プロトコルを用いた自動証明書管理（mod_md）。
  • **modules/metadata**: ヘッダーや環境変数、クライアント IP の制御（mod_headers, mod_remoteip など）。
  • **modules/proxy**: リバースプロキシ、フォワードプロキシ、およびロードバランサ（mod_proxy, mod_proxy_http など）。
  • **modules/session**: セッション状態の維持と暗号化（mod_session）。
  • **modules/slotmem**: 共有メモリセグメントのプロバイダ（mod_slotmem_shm など）。
  • **modules/ssl**: OpenSSL を用いた SSL/TLS 暗号化通信（mod_ssl）。
  • **modules/test**: テスト用およびオプション機能のエクスポート。

主要エンティティ

各モジュールカテゴリを代表する主要なソースファイルおよびヘッダーは以下の通りです。

  • **modules/core/mod_so.c / mod_so.h**: 動的共有オブジェクト (DSO) を実行時にロードするための基盤。
  • **modules/http/http_protocol.c**: HTTP プロトコルの送受信やステータスライン処理のコア。
  • **modules/proxy/mod_proxy.c / mod_proxy.h**: プロキシおよびロードバランサ機能の共通フレームワーク。
  • **modules/ssl/mod_ssl.c / ssl_private.h**: SSL/TLS 接続、ハンドシェイク、証明書検証の統合。
  • **modules/http2/mod_http2.c**: HTTP/2 接続の多重化ストリーム処理のエントリポイント。
  • **modules/mappers/mod_rewrite.c**: 正規表現に基づく強力な URL 書き換えルールエンジン。

エンドツーエンドの流れ

DSO ロードから各モジュールの初期化、フック（Hook）の登録、そしてリクエスト処理フェーズ（翻訳、認証、コンテンツ生成、フィルタリング）におけるモジュール間の詳細な呼び出しシーケンスや状態遷移といった**具体的なエンドツーエンドの制御フローは、提供されたファイル構成情報のみからは完全には特定されていません。**

一般的な動作モデルとしては、mod_so によってロードされた各モジュールが、初期化フェーズで自らの関数を特定のフック（例: ap_hook_handler）に登録し、リクエスト受信時にコアから順次呼び出される仕組みとなっています。

重要な不変条件

**モジュール全体のメモリプール（apr_pool_t）のライフサイクル境界、スレッドセーフ性の要件、およびフックの実行優先順序に関する厳密な不変条件は、提供されたソースコードの構造のみからは完全には特定されていません。**

よくある落とし穴

  • **モジュールのロード順序と依存関係**:

    • 一部のモジュールは他のモジュールが提供する API やプロバイダに依存しています。例えば、mod_proxy_http や mod_proxy_balancer は基盤となる mod_proxy が先にロードされている必要があります。依存関係を無視してロードすると、未解決シンボルエラーや起動失敗の原因になります。

  • **フィルタチェーンの順序**:

    • modules/filters 配下のモジュール（例: mod_deflate による圧縮と mod_substitute によるコンテンツ書き換え）を組み合わせる際、適用する順序を誤ると、圧縮済みのバイナリデータに対してテキスト置換を試みるなど、レスポンスの破損を招く恐れがあります。

次に読む場所

各機能領域の具体的な実装を理解するための推奨エントリポイント：

  • **モジュールの動的ロードの仕組み**: modules/core/mod_so.c
  • **HTTP プロトコルのハンドリング**: modules/http/http_protocol.c
  • **SSL/TLS 処理の統合**: modules/ssl/mod_ssl.c
  • **プロキシ・バランサの制御**: modules/proxy/mod_proxy.c
  • **HTTP/2 接続とストリーム管理**: modules/http2/mod_http2.c
  • **URL マッピングと書き換えルール**: modules/mappers/mod_rewrite.c
