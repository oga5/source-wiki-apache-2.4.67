thread_limit

概念

thread_limit は、Apache HTTP Serverの各種マルチプロセッシングモジュール（MPM）において、生成可能なスレッド数の上限を制御するための概念です。設定ファイル内の ThreadLimit ディレクティブに対応し、内部の静的変数 thread_limit およびそれを設定する関数 set_thread_limit を通じて管理されます。

なぜ重要か

マルチスレッド動作を行うMPM（winnt、event、worker、netwareなど）において、リソース消費や並行処理能力を適切に制限・管理するために重要です。設定ファイルからグローバルなコンテキストで安全に上限値を読み込み、システムに適用する役割を担っています。

主要エンティティ

  • **symbol:server/mpm/winnt/mpm_winnt.c#set_thread_limit**: ThreadLimit ディレクティブの値を解析し、内部変数 thread_limit に設定する関数。
  • **symbol:server/mpm/event/event.c#set_thread_limit / thread_limit**: event MPMにおける設定関数および変数。
  • **symbol:server/mpm/netware/mpm_netware.c#set_thread_limit**: netware MPMにおける設定関数。
  • **symbol:server/mpm/worker/worker.c#set_thread_limit / thread_limit**: worker MPMにおける設定関数および変数。

エンドツーエンドの流れ

1. 設定ファイルの解析中に ThreadLimit ディレクティブが検出されると、対応するMPMの set_thread_limit 関数が呼び出されます。 2. 関数内部で ap_check_cmd_context を呼び出し、現在の設定コンテキストがグローバル（GLOBAL_ONLY）であるかを検証します。 3. コンテキストが適切であれば、引数として渡された文字列を atoi を用いて整数に変換し、静的変数 thread_limit に代入します。

※ 設定された thread_limit が、その後のスレッド生成やプール初期化の段階でどのように参照・適用されるかといった、実行時における詳細なエンドツーエンドのフローは、提供された情報からは未だ完全には判明していません。

重要な不変条件

  • **グローバルコンテキストの強制**: set_thread_limit は、必ずグローバルな設定コンテキスト（GLOBAL_ONLY）でのみ実行されなければなりません。
  • **入力値の形式**: 設定値として渡される引数は、atoi で変換可能な数値文字列である必要があります。

※ 変換後の整数値に対する最小値・最大値などの範囲制限や、他の設定値との相関関係に関する不変条件は、提供された情報からは未だ完全には判明していません。

よくある落とし穴

  • **コンテキストエラー**: ThreadLimit ディレクティブをバーチャルホストやディレクトリコンテキストなど、グローバル以外の場所に記述した場合、ap_check_cmd_context によりエラーが返され、サーバーの起動や再起動が失敗します。
  • **バリデーションの欠如**: mpm_winnt.c などの実装では、設定値の変換に atoi を使用しており、範囲チェックが明示的に行われていません。不正な数値や極端な値を指定した場合の境界条件の挙動に注意する必要があります。
