Module: test/modules/md

概念

test/modules/md は、Apache HTTP Server (httpd) の mod_md (Managed Domains) モジュールに対する統合テストおよび機能テストを実行するための Python テストスイートです。このモジュールは、ACME プロトコル（Let's Encrypt や ZeroSSL など）を用いた SSL/TLS 証明書の自動取得、更新、DNS-01/HTTP-01 チャレンジの解決、OCSP Stapling、および関連する通知機能などの挙動を検証します。

なぜ重要か

mod_md は Web サーバーのセキュリティと証明書管理の自動化を担う重要なコンポーネントです。このテストスイートがあることで、ACME サーバーとの通信、証明書のローテーション、設定のバリデーション、エラーハンドリング、および通知スクリプトの実行が、Apache のライフサイクル内で期待通りに動作することを継続的に保証できます。

関与するモジュール

  • module:test/modules/md (テストコードおよびテスト支援スクリプト群)

主要エンティティ

提供されたファイル群から、以下の主要なテストコンポーネントおよびカテゴリが識別されます。

  • **テスト環境・設定管理**:
  • md_env.py: テスト実行環境（Apache の起動、ACME テストサーバーとの接続など）を管理します。
  • md_conf.py: テスト用の Apache 設定（MDomain ディレクティブなど）を動的に生成します。
  • conftest.py: pytest の共通フィクスチャを定義します。
  • **ACME・証明書ユーティリティ**:
  • md_acme.py: ACME プロトコル関連のヘルパー。
  • md_certs.py / md_cert_util.py: 証明書の解析や検証を行うユーティリティ。
  • **チャレンジプロバイダー**:
  • dns01.py / dns01_v2.py: DNS-01 チャレンジのシミュレーション。
  • http_challenge_foobar.py: HTTP-01 チャレンジ用のスタブ/ヘルパー。
  • **通知・メッセージング**:
  • notify.py / notifail.py / message.py / msg_fail_on.py: 証明書更新時やエラー時の通知スクリプトの挙動テスト用。
  • **テストケース群 (test_*.py)**:
  • test_001_store.py / test_010_store_migrate.py: 証明書ストアの管理と移行テスト。
  • test_100_reg_add.py 〜 test_202_acmev2_regs.py: ACME アカウント登録関連テスト。
  • test_300_conf_validate.py / test_310_conf_store.py: 設定の検証と保存。
  • test_502_acmev2_drive.py / test_602_roundtrip.py / test_702_auto.py: ACMEv2 による証明書取得の自動化とラウンドトリップテスト。
  • test_720_wildcard.py: ワイルドカード証明書のテスト。
  • test_750_eab.py 〜 test_752_zerossl.py: 外部アカウントバインディング (EAB) や特定プロバイダ (Sectigo, ZeroSSL) のテスト。
  • test_800_must_staple.py / test_801_stapling.py: OCSP Stapling のテスト。
  • test_820_locks.py: ロック機構のテスト。
  • test_900_notify.py / test_901_message.py: 通知機能のテスト。

エンドツーエンドの流れ

テスト実行時の詳細なエンドツーエンドの内部フロー（pytest から Apache プロセスおよび ACME テストサーバーがどのように起動され、どのようにモック DNS が連動するかなど）の完全なシーケンスは、提供されたファイル名の一覧からは直接特定できていません。

一般的な推測される流れは以下の通りです： 1. conftest.py および md_env.py がテスト環境（Apache インスタンス、ACME テストサーバーなど）を初期化します。 2. 各 test_*.py が md_conf.py を使用して特定の mod_md 設定を生成し、Apache を起動または再起動します。 3. テストスクリプトが HTTP リクエストを送信するか、ACME チャレンジ（dns01.py など）をトリガーして、証明書の取得や更新をシミュレートします。 4. md_certs.py などのユーティリティを使用して、取得された証明書やストアの状態が期待通りであることを検証します。

重要な不変条件

テストスイート全体における厳密な不変条件（例：テスト実行前後でのストアのクリーンアップ状態、ACME ディレクトリの整合性など）の詳細は、提供されたファイル名からは直接特定できていません。

ただし、テストの独立性を保つために、各テストケースの実行前後で Apache の設定や証明書ストアが初期化またはクリーンアップされることが期待されます（test_910_cleanups.py などがこれに関与している可能性があります）。

よくある落とし穴

  • **外部依存関係の未解決**: ACME テストサーバー（Pebble など）や DNS ヘルパーが適切に起動・設定されていない場合、ACME チャレンジを伴うテスト（test_702_auto.py や test_720_wildcard.py など）が偽陰性（False Negative）で失敗する可能性があります。
  • **状態の残留**: テスト内で生成された証明書やアカウント登録情報が適切にクリーンアップされない場合、後続のテストでキャッシュが再利用され、期待する新規取得フローが検証できなくなる可能性があります。
