apr_status_t

概念

apr_status_t は、Apache Portable Runtime (APR) における操作結果やエラーコードを表現するための標準的な整数型（通常は int）です。Apache HTTP Server の各モジュールにおいて、関数の実行成否や特定の状態を呼び出し元に伝達するための共通のインターフェースとして機能します。

なぜ重要か

複数の独立したモジュール（mod_proxy_http2 や mod_md など）が、共通のエラーハンドリングおよび状態管理の仕組みとして apr_status_t に依存しているためです。この型が返す値（APR_SUCCESS、APR_EAGAIN、APR_EOF など）は、イベントループの制御フローの決定や、HTTP 通信の成否判定において極めて重要な役割を果たします。

エンドツーエンドの流れ

各モジュールにおける具体的なエンドツーエンドのフロー全体は、提供されたソースコードの抜粋からは完全には明らかになっていません。しかし、部分的な流れとして以下が確認されています。

1. **HTTP/2 プロキシ処理 (mod_proxy_http2)**: h2_proxy_session_process などの関数が apr_status_t を戻り値として返し、イベントループの継続・終了を制御します。また、h2_proxy_request_done コールバック関数の引数としても渡されます。 2. **HTTP 送受信処理 (mod_md)**: md_http_perform などの関数が処理結果として apr_status_t を返し、コールバック関数 md_http_status_cb の引数としてステータスが伝達されます。

重要な不変条件

不変条件の全容は未だ完全には明らかになっていませんが、以下の基本原則が定義されています。

  • **成功の定義**: APR_SUCCESS（通常は 0）は、操作が正常に完了したことを示します。
  • **異常/状態の定義**: 0 以外の値は、エラーまたは特定の非同期状態（再試行が必要な APR_EAGAIN、接続終了を示す APR_EOF など）を示します。

よくある落とし穴

  • **モジュール固有の定義ではない**: apr_status_t は APR のコア定義であり、mod_proxy_http2 や mod_md 固有の型ではありません。そのため、独自のエラーコードを定義・返却することは避け、APR のエラーコード体系に厳密に従う必要があります。
  • **イベントループへの影響**: h2_proxy_session_process などの戻り値として本型を扱う際、エラーコードの判定を誤ると、イベントループが意図せず終了したり、無限ループに陥ったりする危険性があります。
