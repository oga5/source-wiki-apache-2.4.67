apr_status_t

概念

apr_status_t は、Apache Portable Runtime (APR) において、関数の実行結果やエラーコード、および特定の処理状態を表現するための標準的なプラットフォーム依存の整数型（通常は int の typedef）です。

なぜ重要か

Apache HTTP Server の各モジュール（mod_proxy_http2 や mod_md など）において、処理の成功・失敗や、非同期処理における再試行要求（APR_EAGAIN）、接続終了（APR_EOF）などの状態を統一的に伝達するために不可欠です。これにより、一貫したエラーハンドリングやイベントループの制御が可能になります。

エンドツーエンドの流れ

システム全体にわたる詳細なエンドツーエンドの制御フローは完全には判明していませんが、各モジュール内での部分的な利用フローは以下の通りです。

  • **mod_proxy_http2 における流れ**:

    • h2_proxy_session_process() などのセッション管理関数が apr_status_t を戻り値として返します。この値に基づいてイベントループの継続や終了が決定されます。また、リクエスト完了時の h2_proxy_request_done コールバック関数の引数としても渡されます。

  • **mod_md における流れ**:

    • HTTP通信を実行する md_http_perform などの関数が処理結果として apr_status_t を返します。また、HTTPステータスを扱うコールバック関数 md_http_status_cb の引数としても渡され、後続の処理を継続するかエラーハンドリングを行うかの判断材料となります。

重要な不変条件

  • **成功の定義**: APR_SUCCESS（通常は 0）は、操作が正常に完了したことを示します。
  • **エラー・状態の定義**: 0 以外の値は、何らかのエラー（ネットワークエラー、タイムアウト、リソース不足など）または特定の状態（APR_EAGAIN、APR_EOF など）を示します。具体的なエラーコードの意味は APR ライブラリの定義に準拠します。

よくある落とし穴

詳細なシステム全体の落とし穴は完全には判明していませんが、以下の点に注意が必要です。

  • **固有定義ではない点**: apr_status_t は APR のコア型であり、各モジュール固有の型ではありません。そのため、エラーハンドリングを実装する際は、モジュール独自の解釈ではなく、APR のエラーコード体系に厳密に従う必要があります。
  • **イベントループへの影響**: h2_proxy_session_process などの戻り値はイベントループの継続条件に直結しているため、戻り値のハンドリングを誤ると、意図しないループの終了やリソースのリークを引き起こす可能性があります。
