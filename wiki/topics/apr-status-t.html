apr_status_t

概念

apr_status_t は、Apache Portable Runtime (APR) における操作結果やエラーコードを表現するための標準的な整数型（通常は int）です。Apache HTTP Server の各モジュールにおいて、関数の実行成否や特定の状態（接続終了や再試行要求など）を伝達する共通の手段として利用されます。

なぜ重要か

この型は、非同期処理やネットワーク通信、プロキシ処理などの制御フローを決定する上で極めて重要です。例えば、mod_proxy_http2 ではイベントループの継続条件に直結し、mod_md では HTTP 通信の成否判定やエラーハンドリングの基準となります。統一されたステータスコード体系により、モジュールをまたいだ一貫したエラー処理が可能になります。

エンドツーエンドの流れ

システム全体における詳細なエンドツーエンドの呼び出しフローは完全には明らかになっていませんが、各モジュール内での部分的な流れは以下の通りです：

  • **mod_proxy_http2 における流れ**:

    • h2_proxy_session_process() などの関数が apr_status_t を返し、その値（APR_EAGAIN や APR_EOF など）に基づいてイベントループの制御フローが決定されます。また、h2_proxy_request_done コールバック関数の引数としてもこの型が渡されます。

  • **mod_md における流れ**:

    • md_http_perform などの関数が処理結果として apr_status_t を返します。また、コールバック関数 md_http_status_cb の引数としても渡され、呼び出し元は返されたステータス値を確認して後続処理の継続やエラーハンドリングを判断します。

重要な不変条件

  • APR_SUCCESS（通常は 0）は、操作が正常に完了したことを示します。
  • 0 以外の値（負の値や特定の正の値）は、APR ライブラリで定義されたエラー（APR_EAGAIN、APR_EOF など）または特定の状態に対応します。

よくある落とし穴

  • apr_status_t は APR のコア定義であり、各モジュール（mod_proxy_http2 や mod_md）固有の型ではありません。そのため、独自のエラーハンドリングを実装する際にも、APR のエラーコード体系に厳密に従う必要があります。
  • h2_proxy_session_process などの戻り値はイベントループの継続条件に直結しているため、エラーコードの扱いを誤ると、無限ループや意図しない接続切断を引き起こす可能性があります。
