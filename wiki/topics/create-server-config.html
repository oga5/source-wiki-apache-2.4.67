create_server_config

概念

create_server_config は、Apache HTTP Server (httpd) において、各モジュールがサーバー単位（バーチャルホストを含む server_rec 単位）の設定構造体を生成・初期化するためのフック関数、およびコアシステムにおける設定ベクトル生成関数です。

各モジュールは、この共通のシグネチャを持つ関数を実装してモジュール構造体に登録することで、サーバーの起動時や設定読み込み時にフレームワークから呼び出され、自身専用の設定領域を確保します。

なぜ重要か

Apache HTTP Server は、複数のバーチャルホストや多様なモジュールが協調して動作するマルチプロセス/マルチスレッドアーキテクチャを採用しています。各モジュールがサーバーごとに異なる設定（キャッシュサイズ、タイムアウト、圧縮レベルなど）を安全かつ独立して保持するためには、サーバーごとに独立した設定メモリ領域を確保し、適切なデフォルト値で初期化する必要があります。create_server_config は、このサーバー単位の設定ライフサイクルの起点として機能するため極めて重要です。

関与するモジュール

提供された情報に基づき、以下のモジュールがこの処理に関与しています。

  • module:modules/cache (mod_file_cache.c, mod_socache_memcache.c, mod_socache_redis.c)
  • module:modules/filters (mod_brotli.c)
  • module:modules/lua (mod_lua.c)
  • module:server (config.c)

主要エンティティ

  • **モジュール個別実装（関数）**:
  • symbol:modules/cache/mod_file_cache.c#create_server_config: a_server_config 構造体を生成し、キャッシュファイル管理用のハッシュテーブル（fileht）を初期化します。
  • symbol:modules/cache/mod_socache_memcache.c#create_server_config: socache_mc_svr_cfg 構造体を生成し、TTLのデフォルト値（MC_DEFAULT_SERVER_TTL）を設定します。
  • symbol:modules/cache/mod_socache_redis.c#create_server_config: socache_rd_svr_cfg 構造体を生成し、TTL（RD_DEFAULT_SERVER_TTL）やタイムアウト（RD_DEFAULT_SERVER_RWTO）を設定します。
  • symbol:modules/filters/mod_brotli.c#create_server_config: brotli_server_config_t 構造体を生成し、品質（quality）やウィンドウサイズ（lgwin）などのデフォルト値を設定します。
  • symbol:modules/lua/mod_lua.c#create_server_config: Luaモジュール用のサーバー設定を生成します。
  • **コアシステム（関数）**:
  • symbol:server/config.c#create_server_config: 設定ベクトル（ap_conf_vector_t）を生成します。

エンドツーエンドの流れ

1. **設定フェーズの開始**: Apacheの起動時または再起動時に、コア設定処理（server/config.c 内の init_server_config など）が呼び出されます。 2. **設定ベクトルの生成**: コアシステム側の create_server_config が呼び出され、各モジュールの設定構造体へのポインタを格納するためのベクトル（ap_conf_vector_t）が用意されます。 3. **各モジュールのフック呼び出し**: 各モジュールが登録した create_server_config フックが順次呼び出されます。

  • 各モジュールは、渡されたメモリプール（apr_pool_t *p）から apr_palloc または apr_pcalloc を用いて、自身のサーバー設定構造体のメモリを確保します。
  • 確保した構造体にデフォルト値を設定し、ポインタを返します。

    • 4. **設定の格納**: コアシステムは、返された各モジュールの設定構造体ポインタを、該当するサーバーレコード（server_rec）の設定ベクトルに格納します。

*注意: コアシステム（server/config.c）および mod_lua における具体的な内部アルゴリズムや、init_server_config からの完全なエンドツーエンドの呼び出しシーケンスの全容は、提供された情報からは完全には解析されていません。*

重要な不変条件

  • **メモリ管理の局所性**: 設定構造体のメモリ確保には、必ず引数として渡されるメモリプール p（apr_pool_t）を使用しなければなりません。これにより、設定のライフサイクル（設定の再読み込みやサーバー停止）に連動した自動的なメモリ解放が保証されます。
  • **シグネチャの整合性**: 各モジュールの実装は void *create_server_config(apr_pool_t *p, server_rec *s) という共通のシグネチャを持ちます。
  • **未使用引数の維持**: 多くのモジュール実装（mod_file_cache, mod_socache_memcache, mod_socache_redis, mod_brotli）において、引数 server_rec *s は現在使用されていませんが、将来的な拡張やAPI規格適合のためにシグネチャに維持されています。

よくある落とし穴

  • **未初期化領域の残存**: apr_palloc を使用してメモリを確保する場合（mod_file_cache や mod_socache_redis など）、構造体のすべてのメンバを明示的に初期化しないと、未定義の値（ゴミデータ）が残る可能性があります。安全のために apr_pcalloc を使用してゼロクリアするか、すべてのフィールドを確実に初期化する必要があります。
  • **メモリプールの誤用**: 引数 p 以外のライフサイクルが異なるメモリプールから設定構造体を割り当てると、設定の破棄や再読み込み時にメモリリークや不正アクセス（Use-After-Free）を引き起こす原因となります。
