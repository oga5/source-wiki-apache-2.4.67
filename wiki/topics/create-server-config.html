create_server_config

概念

create_server_config は、Apache HTTP Server（httpd）の各モジュールにおいて、サーバー単位（バーチャルホストなどを含む）の設定構造体を生成・初期化するためのコールバック関数（フック）です。各モジュールは、この関数を通じて自身が必要とする設定用のメモリ領域を確保し、デフォルト値を設定します。

なぜ重要か

この関数は、Apache HTTP Server の設定フェーズにおけるライフサイクルの起点となります。モジュールが正しく動作するために必要な設定パラメータ（キャッシュ用のハッシュテーブル、タイムアウト値、圧縮品質など）の初期値を安全に割り当てることで、設定ファイルの解析やその後のリクエスト処理フェーズで参照される設定データの整合性を保証します。

関与するモジュール

提供されたコンテキストに基づき、以下のモジュールが関与しています。

  • module:modules/cache (キャッシュ関連モジュール)
  • module:modules/filters (フィルタ関連モジュール)
  • module:modules/lua (Luaスクリプト関連モジュール)
  • module:server (サーバーコア設定処理)

主要エンティティ

本トピックに関連する主要なソースファイルおよびシンボルは以下の通りです。

  • **modules/cache/mod_file_cache.c**
  • create_server_config: a_server_config 構造体を生成し、キャッシュ対象ファイルを管理するハッシュテーブル fileht を初期化します。
  • **modules/cache/mod_socache_memcache.c**
  • create_server_config: socache_mc_svr_cfg 構造体を生成し、TTL（生存期間）にデフォルト値 MC_DEFAULT_SERVER_TTL を設定します。
  • **modules/cache/mod_socache_redis.c**
  • create_server_config: socache_rd_svr_cfg 構造体を生成し、TTLおよび読み書きタイムアウト（rwto）にデフォルト値を設定します。
  • **modules/filters/mod_brotli.c**
  • create_server_config: brotli_server_config_t 構造体を生成し、圧縮品質（quality）やウィンドウサイズ（lgwin）などのデフォルト値を設定します。
  • **modules/lua/mod_lua.c**
  • create_server_config: モジュール固有のサーバー設定を生成します。
  • **server/config.c**
  • create_server_config: 各モジュールのフックを呼び出すコア側の設定処理です。

エンドツーエンドの流れ

1. **設定フェーズの開始**: Apache HTTP Server の起動時または再起動時に、コア設定処理（server/config.c など）が各モジュールの設定初期化を開始します。 2. **フックの呼び出し**: コアは、登録された各モジュールの create_server_config フックを順次呼び出します。この際、メモリプール（apr_pool_t *p）とサーバーレコード（server_rec *s）が引数として渡されます。 3. **メモリ確保と初期化**:

  • 各モジュールは、apr_palloc または apr_pcalloc を使用して、自身のサーバー設定構造体（例: brotli_server_config_t や socache_rd_svr_cfg）に必要なメモリをプールから確保します。
  • 確保した構造体の各メンバにデフォルト値を代入、あるいはハッシュテーブルなどの関連データ構造を初期化します。

    • 4. **ポインタの返却**: 初期化された構造体へのポインタ（void *）を返却し、コア設定フレームワークに登録されます。

*注意: コア設定処理（server/config.c）がどのようにこれらの設定をマージし、最終的なサーバー設定として確定させるかという詳細なエンドツーエンドの全体フローは、提供された情報からは完全には特定されていません。*

重要な不変条件

  • **メモリ管理の不変条件**: 設定構造体のメモリ確保には、引数として渡されるメモリプール p（apr_pool_t）を使用しなければなりません。これにより、設定のライフサイクルに連動した自動的なメモリ解放が保証されます。
  • **シグネチャの不変条件**: すべてのモジュールで static void *create_server_config(apr_pool_t *p, server_rec *s) という共通のシグネチャを維持する必要があります。
  • **未使用引数**: 現在の実装において、引数 server_rec *s は多くのモジュール（mod_file_cache, mod_socache_memcache, mod_socache_redis, mod_brotli）で未使用ですが、将来の拡張性やフレームワークの規約としてシグネチャに含まれ続けています。

よくある落とし穴

  • **ゼロ初期化の欠如**: apr_palloc を使用してメモリを確保する場合、構造体のすべてのメンバを明示的に初期化する必要があります。初期化を怠ると、未定義の値（ゴミデータ）が残る原因になります。安全のために apr_pcalloc を使用してゼロクリアすることが推奨されるケースもあります（例: mod_socache_memcache や mod_brotli での採用例）。
  • **メモリプール外での確保**: 標準の malloc などを用いてメモリを確保すると、Apache の再起動時や設定再読み込み時にメモリリークを引き起こす原因になります。必ず提供された apr_pool_t を使用する必要があります。

次に読む場所

  • **server/config.c**: コア設定フレームワークがどのように各モジュールの create_server_config を呼び出し、管理しているかを理解するために参照してください。
  • **modules/filters/mod_brotli.c**: apr_pcalloc を用いた安全な構造体確保と、具体的なデフォルト値設定の好例として参照してください。
  • **modules/cache/mod_file_cache.c**: 設定構造体内部でさらに別の動的データ構造（apr_hash_t）を初期化するパターンの実装例として参照してください。
