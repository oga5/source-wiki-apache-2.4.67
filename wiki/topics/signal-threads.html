signal_threads

概念

signal_threads は、Apache HTTP Server のマルチプロセッシングモジュール（MPM）において、子プロセス内のスレッド群に対して終了処理（Graceful または Ungraceful）を通知し、停止プロセスを開始するための関数です。

なぜ重要か

サーバーのシャットダウンや再起動時に、実行中のスレッド（リスナースレッドやワーカースレッド）を適切に制御して終了させるために不可欠です。Graceful（穏やかな）停止では既存の接続処理の完了を待ち、Ungraceful（即時の）停止ではソケットを強制的に閉じて迅速にプロセスを終了させるなど、システムの運用要件に応じた終了制御を可能にします。

関与するモジュール

エンドツーエンドの流れ

worker MPM における signal_threads(int mode) の処理フローは以下の通りです。

1. **状態遷移**: グローバルな terminate_mode を指定された mode（ST_GRACEFUL または ST_UNGRACEFUL）に更新し、MPM の状態（retained->mpm->mpm_state）を AP_MPMQ_STOPPING に設定します。 2. **リスナーの解除**: wakeup_listener() を呼び出し、ブロッキング状態にあるリスナースレッドを解除します。 3. **Ungraceful 停止時の処理**: mode が ST_UNGRACEFUL の場合、以下の処理を強制的に行います。

  • workers_may_exit フラグを 1 に設定。
  • ap_queue_interrupt_all(worker_queue) を呼び出し、キュー待ちしているすべてのワーカーを解放。
  • close_worker_sockets() を呼び出し、現在処理中の全接続ソケットを強制的にクローズ。

    • 4. **停止フックの実行**: ap_run_child_stopping(pchild, ...) を呼び出し、子プロセスの停止処理フックを実行します。

**未解明な点**: worker.c 内における signal_threads の直接的な呼び出し元や、event MPM 側での具体的なエンドツーエンドの呼び出しフローの全容は、提供された情報からは未解明です。

重要な不変条件

  • **重複処理の防止**: 指定された mode が、既に設定されている terminate_mode と一致する場合、関数は何も行わずに即座に終了します。
  • **状態の同期**: 本関数が実行されると、MPM の状態は必ず AP_MPMQ_STOPPING に遷移します。

よくある落とし穴

  • **リクエストの中断**: ST_UNGRACEFUL モードで呼び出された場合、接続のクリーンアップを待たずに close_worker_sockets() によってソケットが強制的に閉じられるため、処理中のアクティブなリクエストは中断されます。
  • **モジュール間のコード共有**: 本関数は worker.c 内で定義されていますが、event.c からの参照も検出されています。これらの MPM 間でのコード共有や依存関係の範囲に注意してメンテナンスを行う必要があります。
