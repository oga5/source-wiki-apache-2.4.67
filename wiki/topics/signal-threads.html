signal_threads

概念

signal_threads は、Apache HTTP Server のマルチプロセッシングモジュール（MPM）において、子プロセス内のスレッド群に対して終了処理（Graceful または Ungraceful）を通知し、安全または強制的に停止プロセスを開始するための関数です。

なぜ重要か

サーバーのシャットダウンや再起動の際、実行中のスレッド（リスナースレッドやワーカースレッド）を適切に制御して停止させる必要があります。signal_threads は、終了モード（優雅な停止 ST_GRACEFUL または強制停止 ST_UNGRACEFUL）に応じて、リスナーのブロック解除、キューの割り込み、ソケットのクローズなどのクリーンアップ処理を統制する役割を担うため重要です。

関与するモジュール

提供された情報に基づき、以下のモジュールが関与しています。

  • module:server/mpm/worker
  • module:server/mpm/event

主要エンティティ

  • **symbol:server/mpm/worker/worker.c#signal_threads**: worker MPM における実装。終了モードに応じたスレッド停止処理をカプセル化しています。
  • **symbol:server/mpm/event/event.c#signal_threads**: event MPM 側でも検出されており、コード共有または同様のシグナル処理構造が存在することを示唆しています（ただし、event.c 側の詳細な実装は提供されていません）。

エンドツーエンドの流れ

提供された情報からは、signal_threads を呼び出す全体的なエンドツーエンドの制御フロー（親プロセスからのシグナル受信から子プロセスへの伝播など）の全容は完全には明らかになっていません。

現時点で判明している子プロセス内での処理の流れは以下の通りです： 1. signal_threads(mode) が呼び出される。 2. terminate_mode が更新され、MPM 状態（retained->mpm->mpm_state）が AP_MPMQ_STOPPING に設定される。 3. wakeup_listener() が実行され、リスナースレッドのブロッキング状態が解除される。 4. mode が ST_UNGRACEFUL（強制停止）の場合：

  • workers_may_exit フラグが 1 に設定される。
  • ap_queue_interrupt_all(worker_queue) により、キュー待ちの全ワーカーが解放される。
  • close_worker_sockets() により、処理中の全接続ソケットが強制的に閉じられる。

    • 5. ap_run_child_stopping フックが実行され、子プロセスの停止処理が通知される。

重要な不変条件

  • **状態の重複防止**: terminate_mode が既に指定された mode と一致する場合、関数は何も処理を行わずに即座に復帰（アーリーリターン）します。
  • **状態遷移**: 呼び出しが行われると、retained->mpm->mpm_state は必ず AP_MPMQ_STOPPING に遷移します。

よくある落とし穴

  • **強制終了時のリクエスト中断**: ST_UNGRACEFUL モードで呼び出された場合、close_worker_sockets() によって現在処理中のアクティブな接続ソケットが強制的に閉じられます。これにより、進行中のリクエストが途中で切断されるため、呼び出し時のモード選択には注意が必要です。
  • **モジュール間のコード共有**: worker.c で定義されている一方で event.c からの参照も検出されており、両 MPM 間でのコード共有や依存関係の境界に注意してメンテナンスを行う必要があります。
