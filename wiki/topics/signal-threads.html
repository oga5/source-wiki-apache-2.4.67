signal_threads

概念

signal_threads は、Apache HTTP Server のマルチプロセッシングモジュール（MPM）において、子プロセス内のスレッド群に対して終了処理の開始を通知し、停止シーケンスを制御するための関数です。

なぜ重要か

サーバーのシャットダウンや再起動の際、実行中のスレッド（リスナースレッドやワーカースレッド）を安全かつ迅速に停止させる必要があります。signal_threads は、優雅な停止（Graceful）と即時停止（Ungraceful）の2つのモードを切り替え、リスナーのブロック解除やソケットの強制クローズ、キューの割り込みなどを一元的に制御することで、リソースのクリーンアップとプロセスの安全な終了を保証します。

関与するモジュール

  • module:server/mpm/event
  • module:server/mpm/worker

主要エンティティ

  • symbol:server/mpm/worker/worker.c#signal_threads: worker MPM におけるスレッド停止処理の実装。
  • symbol:server/mpm/event/event.c#signal_threads: event MPM におけるスレッド停止処理の実装。

エンドツーエンドの流れ

1. **終了モードの判定と状態遷移**: 引数 mode（ST_GRACEFUL または ST_UNGRACEFUL）を受け取り、グローバルな terminate_mode を更新します。また、サーバー状態を示す retained->mpm->mpm_state を AP_MPMQ_STOPPING に設定します。 2. **リスナーの解除**: wakeup_listener() を呼び出し、ブロッキング状態にあるリスナースレッドを解除します。 3. **即時停止（Ungraceful）時の処理**: mode が ST_UNGRACEFUL の場合、以下の処理を追加で実行します。

  • workers_may_exit フラグを 1 に設定します。
  • ap_queue_interrupt_all(worker_queue) を呼び出し、キュー待ち状態の全ワーカーを解放します。
  • close_worker_sockets() を呼び出し、現在処理中の全接続ソケットを強制的にクローズします。

    • 4. **停止フックの実行**: 最後に ap_run_child_stopping(pchild, ...) を呼び出し、子プロセスの停止処理フックを実行します。

*注意: worker.c 内における signal_threads の具体的な直接の呼び出し元や、event.c 側での詳細なエンドツーエンドの呼び出しフローは、提供された情報からは完全には判明していません。*

重要な不変条件

  • **重複処理の防止**:

    • 指定された mode が既に設定されている terminate_mode と一致する場合、関数は何も処理を行わずに即座に終了します。

  • **状態の一貫性**:

    • 本関数が実行されると、必ず retained->mpm->mpm_state が AP_MPMQ_STOPPING に遷移します。

よくある落とし穴

  • **強制切断によるリクエストの中断**:

    • ST_UNGRACEFUL モードで呼び出された場合、接続のクリーンアップを待たずに close_worker_sockets() によってソケットが強制的に閉じられるため、進行中のアクティブなリクエストは中断されます。

  • **モジュール間のコード共有**:

    • 本関数は worker.c で定義されていますが、event.c からの参照も検出されています。MPM 間でのコード共有や依存関係の変更を行う際は、双方の挙動に影響を与えないよう注意が必要です。
