signal_threads

概念

signal_threads は、Apache HTTP Server のマルチプロセッシングモジュール（MPM）において、子プロセス内のスレッド群に対して終了処理（Graceful または Ungraceful）を通知し、停止シーケンスを開始するための関数です。

なぜ重要か

サーバーのシャットダウンや再起動時に、実行中のスレッド（リスナースレッドやワーカースレッド）を安全かつ迅速に停止させるために重要です。Graceful（緩やかな）停止と Ungraceful（強制的な）停止を区別し、それぞれに応じた適切なクリーンアップ処理（ソケットのクローズやキューの割り込みなど）を制御する役割を担います。

関与するモジュール

エンドツーエンドの流れ

シグナル受信などの上位イベントから signal_threads が呼び出されるまでの完全なエンドツーエンドのトリガーフローは、提供されたコンテキストからは判明していません。

worker.c における呼び出し後の内部的な流れは以下の通りです： 1. 引数 mode（ST_GRACEFUL または ST_UNGRACEFUL）を受け取り、グローバルな terminate_mode を更新し、サーバー状態（retained->mpm->mpm_state）を AP_MPMQ_STOPPING に設定します。 2. wakeup_listener() を呼び出して、ブロッキング状態のリスナースレッドを解除します。 3. mode が ST_UNGRACEFUL の場合：

  • workers_may_exit フラグを 1 に設定します。
  • ap_queue_interrupt_all(worker_queue) を呼び出し、接続キュー（worker_queue）で待機中の全ワーカーを解放します。
  • close_worker_sockets() を呼び出し、現在処理中の全接続ソケットを強制的に閉じます。

    • 4. ap_run_child_stopping(pchild, ...) を呼び出し、子プロセスの停止処理フックを実行します。

重要な不変条件

  • terminate_mode が既に指定された mode と一致する場合、関数は何も処理を行わずに即座にリターンします。
  • ap_run_child_stopping の第2引数（Graceful フラグ）は、mode == ST_GRACEFUL の場合に真となります。

よくある落とし穴

  • ST_UNGRACEFUL モードが指定された場合、接続のクリーンアップを待たずにソケットが強制的に閉じられるため、進行中のリクエストは中断されます。
  • 本関数は worker.c 内で定義されていますが、event.c からの参照も検出されています。モジュール間でのコード共有範囲や依存関係に注意する必要があります。
