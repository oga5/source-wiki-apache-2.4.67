signal_threads

概念

signal_threads は、Apache HTTP Server のマルチプロセッシングモジュール（MPM）である event および worker において、子プロセス内のスレッド群に対して終了モード（Graceful または Ungraceful）に応じた通知を行い、停止処理シーケンスを開始するための内部関数です。

なぜ重要か

マルチスレッドで動作する MPM において、子プロセスを安全かつ迅速にシャットダウンするためには、リスナースレッドのブロッキング解除や、ワーカーキューで待機中のスレッドへの割り込み、ソケットのクローズなどを秩序立てて行う必要があります。signal_threads は、指定された終了モード（緩やかな停止か、強制的な即時停止か）に基づいてこれらのクリーンアップ処理を一元的に制御する役割を担います。

関与するモジュール

  • module:server/mpm/event
  • module:server/mpm/worker

主要エンティティ

  • **symbol:server/mpm/event/event.c#signal_threads**: event MPM における実装。
  • **symbol:server/mpm/worker/worker.c#signal_threads**: worker MPM における実装。

エンドツーエンドの流れ

worker MPM における処理フロー

1. signal_threads(mode) が呼び出されます。引数 mode には ST_GRACEFUL または ST_UNGRACEFUL が指定されます。 2. terminate_mode を更新し、サーバー状態（retained->mpm->mpm_state）を AP_MPMQ_STOPPING に遷移させます。 3. wakeup_listener() を呼び出し、リスナースレッドがブロッキング状態（接続待ちなど）であればそれを解除します。 4. mode が ST_UNGRACEFUL（強制停止）の場合、以下の処理を追加で実行します：

  • workers_may_exit フラグを 1 に設定します。
  • ap_queue_interrupt_all(worker_queue) を呼び出し、接続キューで待機しているすべてのワーカースレッドを解放します。
  • close_worker_sockets() を呼び出し、現在処理中のすべての接続ソケットを強制的に閉じます。

    • 5. 最後に ap_run_child_stopping(pchild, ...) を呼び出し、子プロセスの停止処理フックを実行します。

不明な点・未検証の流れ

  • **worker.c 内の呼び出し元**: 提供された情報からは、worker.c 内における signal_threads の直接的な呼び出し元は特定されていません。
  • **event MPM における詳細なフロー**: event.c 内では、accept_mutex_error、check_infinite_requests、child_main、get_worker、listener_thread、process_lingering_close、process_socket、push2worker、worker_thread など多数の箇所から呼び出されることが判明していますが、event 固有の具体的な実行パスや条件分岐の詳細は現時点で解析されていません。

重要な不変条件

  • **重複処理の防止**: terminate_mode が既に指定された mode と一致する場合、関数は何も処理を行わずに即座に終了します。
  • **状態遷移の保証**: 本関数が実行されると、グローバルな terminate_mode が更新され、retained->mpm->mpm_state は必ず AP_MPMQ_STOPPING に設定されます。

よくある落とし穴

  • **強制終了時のリクエスト切断**: ST_UNGRACEFUL モードで呼び出された場合、接続のクリーンアップを待たずに close_worker_sockets() によってソケットが強制的に閉じられるため、処理中のアクティブなリクエストは途中で切断されます。
  • **モジュール間のコード共有と依存**: worker.c 内で定義されている処理と event.c 内の処理は類似していますが、event.c からの参照や静的解析上の重複が検出されているため、コードを変更する際は両 MPM への影響範囲に注意する必要があります。
