dav_stream

概念

dav_stream は、Apache HTTP Server の WebDAV モジュール（特にファイルシステムバックエンドである mod_dav_fs）において、WebDAV リソースに対する読み書き操作のストリーム状態を管理するためのデータ構造です。

なぜ重要か

WebDAV のファイル操作（特に書き込み操作）において、直接ファイルを書き換えるのではなく、一時ファイル（temppath）を用いて安全に書き込みを行い、正常終了時に本来のパス（pathname）へリネーム（コミット）する、あるいはエラー時に一時ファイルを削除（クリーンアップ）するといったトランザクション的なファイル制御を可能にするために重要です。

関与するモジュール

  • module:modules/dav/main

主要エンティティ

  • symbol:modules/dav/fs/repos.c#dav_stream: ファイルシステムベースの WebDAV リソース操作におけるストリーム状態を保持する構造体。
  • symbol:modules/dav/main/mod_dav.h#dav_stream: mod_dav コアと各バックエンドプロバイダ間でストリームを扱うための定義。

エンドツーエンドの流れ

1. **ストリームのオープン**: mod_dav コアからの要求に基づき、dav_fs_open_stream が呼び出され、dav_stream 構造体が初期化されます。書き込みの場合は一時ファイルパス temppath が設定されます。 2. **データの読み書き**: dav_fs_write_stream や dav_fs_seek_stream などの関数を経由して、apr_file_t を通じた実際のファイル I/O 操作が行われます。 3. **ストリームのクローズとコミット**: dav_fs_close_stream が呼び出されます。

  • 正常にコミットされる場合：一時ファイル（temppath）から本来のパス（pathname）へのリネーム処理が行われます。
  • エラーが発生した場合：unlink_on_error フラグが真であれば、一時ファイルが削除されます。

※なお、mod_dav コア（mod_dav.h）側からこれらの関数が呼び出される具体的なエンドツーエンドの制御フローの全容については、提供された情報からは完全には明らかになっていません。

重要な不変条件

  • **メモリプールの同期**: apr_pool_t *p は、関連するリクエストのライフサイクルと同期している必要があります。
  • **一時ファイルのリネーム**: 書き込み操作において temppath が設定されている場合、dav_fs_close_stream でのコミット時に必ずリネーム処理が行われなければなりません。
  • **エラー時のクリーンアップ**: ファイルオープン失敗や書き込みエラーが発生し、かつ unlink_on_error が真である場合、クローズ処理時に一時ファイルが適切に破棄（削除）される必要があります。

よくある落とし穴

  • **クローズ処理への依存**: コミット失敗時やエラー時のクリーンアップ処理（一時ファイルの削除など）は dav_fs_close_stream の呼び出しに依存しています。そのため、この構造体のフィールドを直接操作したり、クローズ処理をバイパスしたりすると、一時ファイルがシステム内に残存するなどの不整合が発生する可能性があります。
  • **構造体の変更影響**: dav_stream は repos.c 内で完結する操作が主ですが、フィールドを変更する際は dav_fs_close_stream 内のクローズ・コミットロジックへの影響を慎重に確認する必要があります。
