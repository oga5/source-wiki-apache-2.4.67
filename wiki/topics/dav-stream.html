dav_stream

概念

dav_stream は、Apache HTTP Server の WebDAV モジュール（特に mod_dav_fs）において、ファイルシステムベースの WebDAV リソースに対する読み書き操作を行う際に、そのストリームの状態を保持するための構造体です。

なぜ重要か

WebDAV リソースへの書き込み処理において、一時ファイル（temppath）を用いた安全な書き込みと、エラー発生時の適切なクリーンアップ（一時ファイルの削除）を制御し、データの整合性を保つために重要な役割を担っています。

エンドツーエンドの流れ

1. **初期化**: mod_dav コアから mod_dav.h で定義されたフックを経由して呼び出しが行われ、dav_fs_open_stream 内で dav_stream が生成・初期化されます。 2. **ストリーム操作**: dav_fs_write_stream や dav_fs_seek_stream などの関数を通じて、APR (Apache Portable Runtime) のファイル I/O 関数群（apr_file_t）を介した実際の読み書きが行われます。 3. **クローズとコミット/破棄**: dav_fs_close_stream が呼び出されます。

  • コミットが指示された場合、temppath が設定されていれば、本来のパス（pathname）へのリネーム処理が行われます。
  • エラーが発生し、かつ unlink_on_error が真である場合は、一時ファイルが削除されます。

※ なお、mod_dav コア側におけるリクエスト受信からフック呼び出しに至るまでの完全なエンドツーエンドの制御フローの全容は、提供された情報からは未詳です。

重要な不変条件

  • **メモリプールの同期**: 構造体内の apr_pool_t *p は、関連するリクエストのライフサイクルと同期している必要があります。
  • **コミット時のリネーム**: 書き込み操作において temppath が設定されている場合、dav_fs_close_stream でのコミット時に必ずリネーム処理が行われなければなりません。
  • **エラー時のクリーンアップ**: エラー発生時、unlink_on_error が真であれば、クローズ処理時に一時ファイルが確実に削除される必要があります。

よくある落とし穴

  • **クローズ処理への依存**: 一時ファイルのクリーンアップやリネームなどの重要な後処理は、すべて dav_fs_close_stream の呼び出しに依存しています。そのため、この構造体のフィールドを直接変更したり、クローズ処理を適切に呼び出さなかったりすると、一時ファイルがファイルシステム上に残存するなどの不整合が発生する原因となります。
