dav_stream

概念

dav_stream は、Apache HTTP Server の WebDAV モジュール（特にファイルシステムプロバイダである mod_dav_fs）において、リソースに対する読み書き操作のストリーム状態を管理するための構造体です。

なぜ重要か

WebDAV リクエスト（PUT など）によるファイル操作において、直接対象ファイルを書き換えるのではなく、一時ファイル（temppath）への書き込みを行い、エラー発生時のクリーンアップや、正常終了時のコミット（リネーム処理）を安全に行うための状態管理を担うため重要です。

関与するモジュール

  • module:modules/dav/main

主要エンティティ

  • symbol:modules/dav/fs/repos.c#dav_stream: repos.c 内で定義されている構造体。メモリプール（apr_pool_t）、ファイルハンドル（apr_file_t）、対象ファイルのパス、一時ファイルパス、エラー時削除フラグを保持します。
  • symbol:modules/dav/main/mod_dav.h#dav_stream: mod_dav コアとプロバイダ間のフック経由で操作されるストリームのインターフェース。

エンドツーエンドの流れ

1. **初期化**: mod_dav コアからフック経由で dav_fs_open_stream が呼び出され、dav_stream 構造体が初期化されます。 2. **書き込み・シーク**: dav_fs_write_stream や dav_fs_seek_stream などの関数を用いて、APR (Apache Portable Runtime) のファイル I/O 関数群を介した実ファイル操作が行われます。 3. **クローズとコミット/破棄**: dav_fs_close_stream が呼び出されます。

  • **正常終了時**: コミット引数に応じて、一時ファイル（temppath）から本来のパス（pathname）へのリネーム処理が行われます。
  • **エラー発生時**: unlink_on_error が真に設定されている場合、一時ファイルが削除されます。

※ mod_dav.h 側における具体的なハンドリングの全容や、他のプロバイダでの利用有無など、エンドツーエンドの完全なフローは提供されたコンテキストからは未詳です。

重要な不変条件

  • **ライフサイクルの同期**: 構造体内の apr_pool_t *p は、関連するリクエストのライフサイクルと同期している必要があります。
  • **コミット時のリネーム**: 書き込み操作において temppath が設定されている場合、dav_fs_close_stream でのコミット時に必ずリネーム処理が行われなければなりません。
  • **エラー時のクリーンアップ**: エラー発生時、unlink_on_error が真であれば、クローズ処理時に一時ファイルが適切に破棄（削除）される必要があります。

よくある落とし穴

  • **クローズ処理への依存**: 一時ファイルの削除やコミット処理は dav_fs_close_stream の呼び出しに依存しています。エラーハンドリングの過程でクローズ処理がバイパスされると、一時ファイルが残存する可能性があります。
  • **内部実装の結合度**: dav_stream は主に repos.c 内で完結する操作を前提として設計されているため、構造体のフィールドを変更する際は、クローズ時のコミット/クリーンアップロジックへの影響を慎重に確認する必要があります。
