dav_stream

概念

dav_stream は、Apache HTTP Server の WebDAV モジュール（mod_dav および mod_dav_fs）において、ファイルシステムベースのリソースに対する読み書き操作を行う際に、そのストリームの状態を保持するためのデータ構造です。

なぜ重要か

WebDAV のリソース操作（特にファイルの書き込みや更新）において、一時ファイルを用いた安全な書き込みと、エラー発生時のクリーンアップ、および正常終了時のコミット（リネーム処理）をカプセル化して管理するために重要です。これにより、不完全な書き込みによるデータの破損を防ぎます。

  • symbol:modules/dav/main/mod_dav.h#dav_stream: mod_dav コアとプロバイダ（mod_dav_fs）間のインターフェースで参照されるストリーム型。

エンドツーエンドの流れ

1. **ストリームのオープン**: mod_dav コアからの要求により、dav_fs_open_stream が呼び出され、dav_stream 構造体が初期化されます。書き込み時には一時ファイルパス temppath が設定されます。 2. **データの操作**: dav_fs_write_stream や dav_fs_seek_stream などの関数を介して、apr_file_t に対する実際のファイル I/O 操作が行われます。 3. **ストリームのクローズ**: 操作完了後、dav_fs_close_stream が呼び出されます。

  • コミットが指示された場合：一時ファイル（temppath）から本来のパス（pathname）へのリネーム処理が行われます。
  • エラーが発生しコミットされない場合：unlink_on_error が真であれば、一時ファイルが削除されます。

※なお、mod_dav コア側（mod_dav.h）における具体的な呼び出しライフサイクルや、他プロバイダでの利用を含む完全なエンドツーエンドのフローの詳細は、提供された情報からは完全には特定されていません。

重要な不変条件

  • apr_pool_t（メモリプール）は、関連する WebDAV リクエストのライフサイクルと同期していなければなりません。
  • 書き込み操作において temppath が設定されている場合、正常なコミット時には必ず本来の pathname へのリネーム処理が行われ、エラー時には unlink_on_error に従って一時ファイルが破棄される必要があります。

よくある落とし穴

  • dav_fs_close_stream によるクローズ処理に依存してクリーンアップ（一時ファイルの削除やリネーム）が行われるため、この関数を適切に呼び出さないと、一時ファイルが残存したり、書き込みが反映されなかったりするリソースリークやデータ不整合が発生します。
  • dav_stream のフィールドを直接変更する場合、dav_fs_close_stream 内のコミット・クリーンアップロジックに予期せぬ影響を与える可能性があります。
