find_chr

概念

find_chr は、HTTP/2 や ACME (Managed Domains) などのモジュールにおいて、Link ヘッダーなどの文字列を解析する際に、特定の文字（クォートや区切り文字など）を探索するために使用されるユーティリティ関数です。

なぜ重要か

HTTP/2 のプッシュ機能（h2_push）やプロキシユーティリティ（h2_proxy_util）、および MD モジュール（md_util）において、ヘッダー値などの文字列をパースする処理は頻出します。find_chr は、解析コンテキスト（link_ctx）の状態を維持しながら、指定された文字の位置を安全に探索するための共通的な手段を提供します。

エンドツーエンドの流れ

システム全体における完全なエンドツーエンドの制御フロー（例えば、HTTP リクエストの受信から Link ヘッダーの解析、そしてプッシュのトリガーに至る詳細なシーケンス）は、提供された情報からは完全には明らかになっていません。

現時点で判明している局所的な流れは以下の通りです： 1. Link ヘッダーなどの解析対象文字列と現在のオフセット（i）、文字列長（slen）を保持する link_ctx 構造体が用意されます。 2. 解析処理の中で、クォートや区切り文字などの特定の文字 c を探す必要が生じた際、find_chr(ctx, c, &pidx) が呼び出されます。 3. find_chr は ctx->i から ctx->slen までの範囲を線形探索します。 4. 文字が見つかった場合、そのインデックスが pidx に格納され、関数は 1 を返します。呼び出し側はこのインデックスを基に次の解析ステップ（トークンの切り出しなど）に進みます。 5. 文字が見つからない場合は 0 を返します。

重要な不変条件

  • **探索範囲の制限**: 探索は常に現在のオフセット ctx->i から開始され、文字列の長さ ctx->slen を超えてアクセスすることはありません。
  • **引数の有効性**: 呼び出し時に ctx は有効な link_ctx 構造体を指し、pidx は書き込み可能なメモリ領域を指している必要があります。

よくある落とし穴

  • **オフセットの更新漏れ**: find_chr 自体は ctx->i（現在のオフセット）を自動的に更新しません。呼び出し側で ctx->i を適切に進めないと、無限ループに陥るか、同じ文字を繰り返し検出する原因になります。
  • **境界条件のハンドリング**: ctx->i がすでに ctx->slen 以上である場合、探索ループは実行されず即座に 0 が返されます。呼び出し側はこの戻り値を適切にハンドリングする必要があります。
