find_chr

概念

find_chr は、Apache HTTP Server の複数モジュール（mod_http2 および mod_md）において、特定の解析コンテキスト（link_ctx）から指定された文字を検索するために使用されるユーティリティ関数です。主に Link ヘッダーなどの文字列をパースする際、クォートや区切り文字（デリミタ）の位置を特定するために利用されます。

なぜ重要か

HTTP/2 のサーバープッシュ（h2_push.c、h2_proxy_util.c）や、Managed Domains（md_util.c）におけるメタデータやヘッダーの解析処理において、文字列内の特定のデリミタを正確に探索することは、プロトコル準拠のパース処理を行う上で不可欠です。この関数は、解析の現在位置（オフセット）を管理する link_ctx 構造体と連携し、共通の探索ロジックを提供します。

エンドツーエンドの流れ

提供された情報からは、各モジュール（http2 および md）における全体的なエンドツーエンドの処理フロー（どのように Link ヘッダーが受信され、パースされ、最終的に処理されるか）の詳細は完全には明らかになっていません。

判明している局所的なパースの流れは以下の通りです：

1. 呼び出し側が link_ctx 構造体を初期化し、解析対象の文字列 s、長さ slen、および現在のオフセット i を設定します。 2. find_chr が呼び出され、ctx->i から ctx->slen までの範囲で指定された文字 c を線形探索します。 3. 文字が見つかった場合、そのインデックスを pidx に格納して 1 を返します。見つからない場合は 0 を返します。 4. 呼び出し側（h2_push.c、h2_proxy_util.c、md_util.c の各パース処理）は、得られたインデックスを基に次のトークンの切り出しやオフセット ctx->i の更新を行います。

重要な不変条件

  • **探索範囲の整合性**: 探索は常に ctx->i（現在のオフセット）から開始され、ctx->slen（文字列長）に達した時点で終了します。
  • **引数の有効性**: ctx は有効な link_ctx 構造体を指し、pidx は結果を書き込み可能なメモリ領域を指している必要があります。
  • **状態の非破壊性**: find_chr 自体は ctx->i（現在のオフセット）を自動的に進めることはせず、呼び出し側が手動で更新する必要があります。

よくある落とし穴

  • **オフセットの更新漏れ**: find_chr は ctx->i を自動的に更新しないため、呼び出し側で適切にオフセットを進めないと、同じ文字を繰り返し検索して無限ループに陥る可能性があります。
  • **境界条件**: ctx->i がすでに ctx->slen 以上である場合、関数は即座に 0 を返します。呼び出し側は、事前にインデックスが範囲内にあることを保証するか、戻り値 0 を適切にハンドリングする必要があります。
