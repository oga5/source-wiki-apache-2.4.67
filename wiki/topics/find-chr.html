find_chr

概念

find_chr は、HTTP/2 や Managed Domains（MD）モジュールにおいて、Link ヘッダーなどの文字列解析を行う際に、指定された解析コンテキスト（link_ctx）の現在位置から特定の文字を検索するためのユーティリティ関数です。

なぜ重要か

HTTP/2 のサーバープッシュ（h2_push.c）やプロキシ（h2_proxy_util.c）、および Managed Domains（md_util.c）の処理において、Link ヘッダーに含まれるクォートや区切り文字を正確にパースすることは、プロトコル準拠および正しいリソース連携を行うために不可欠です。find_chr は、このパース処理における文字探索の共通パターンを提供します。

エンドツーエンドの流れ

1. Link ヘッダーなどの解析対象文字列と現在のオフセット i、文字列長 slen を保持する link_ctx 構造体が用意されます。 2. 各モジュールのパース処理（例：h2_push.c や md_util.c 内の呼び出し元関数）から find_chr が呼び出され、特定の区切り文字（カンマやクォートなど）の位置を探索します。 3. find_chr は ctx->i から ctx->slen まで線形探索を行い、文字が見つかった場合はそのインデックスを pidx に格納して 1 を返します。見つからない場合は 0 を返します。 4. 呼び出し側は、得られたインデックスを基にトークンの切り出しやオフセット ctx->i の更新を行います。

※なお、各モジュール間を跨ぐ全体的なエンドツーエンドの制御フローの全容や、これら3つのファイルにおける find_chr の実装の完全な同一性・共有関係の詳細は現時点では未解明です。

重要な不変条件

  • **探索範囲の制限**: find_chr の探索範囲は、常に現在のオフセット ctx->i から文字列長 ctx->slen までに制限されます。
  • **戻り値と出力の整合性**: 文字が見つかった場合、出力用ポインタ pidx には必ず有効なインデックスが書き込まれ、戻り値は 1 となります。見つからない場合は 0 が返されます。

よくある落とし穴

  • **オフセットの更新漏れ**: 呼び出し側で ctx->i を適切に進めないと、同じ文字を繰り返し検出し続け、無限ループや誤解析の原因になります。
  • **境界条件**: ctx->i がすでに ctx->slen 以上である場合、探索ループは実行されずに即座に 0 を返します。呼び出し側は事前にコンテキストの状態が有効であることを保証する必要があります。
