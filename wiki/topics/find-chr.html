find_chr

概念

find_chr は、Apache HTTP Server の特定モジュール（HTTP/2 や Managed Domains）において、Link ヘッダーなどの文字列解析を行う際に使用されるユーティリティ関数です。指定された解析コンテキスト（link_ctx）の現在位置から、特定の文字（クォートや区切り文字など）を検索する役割を持ちます。

なぜ重要か

HTTP/2 のサーバープッシュ（h2_push）やプロキシユーティリティ（h2_proxy_util）、および Managed Domains（md_util）では、Link ヘッダー等のパース処理が頻繁に行われます。find_chr は、これらのパース処理における「特定の文字を探索する」という共通のパターンをカプセル化し、コードの簡素化に貢献しています。

エンドツーエンドの流れ

現時点では、find_chr を含むヘッダー解析処理全体がどのようにトリガーされ、解析結果がシステム全体でどのように利用されるかという完全なエンドツーエンドの流れは判明していません。

判明している局所的な流れは以下の通りです： 1. Link ヘッダー等の解析中に、特定の区切り文字やクォート文字を探索する必要が生じる。 2. 解析コンテキスト（link_ctx）と探索対象の文字 c を指定して find_chr を呼び出す。 3. find_chr は ctx->i（現在のオフセット）から ctx->slen（文字列長）まで線形探索を行う。 4. 文字が見つかった場合は、そのインデックスを pidx に格納して 1 を返し、呼び出し側で次の解析ステップ（オフセットの更新など）に進む。見つからない場合は 0 を返す。

重要な不変条件

  • **ポインタの有効性**: 引数として渡される ctx は有効な link_ctx 構造体を指し、pidx は書き込み可能なメモリ領域を指している必要があります。
  • **探索範囲の制限**: 探索は常に ctx->i から開始され、ctx->slen を超えることはありません。ctx->i が ctx->slen 以上の場合は、探索を行わずに即座に 0 を返します。

よくある落とし穴

  • **オフセット更新の不備**: find_chr 自体は ctx->i（現在のオフセット）を自動的に進めません。呼び出し側で ctx->i を適切に更新しない場合、同じ位置を繰り返し探索して無限ループに陥るか、解析が進行しなくなる可能性があります。
  • **重複定義の存在**: find_chr は複数のファイル（h2_proxy_util.c、h2_push.c、md_util.c）で static 関数として個別に定義されているため、修正を行う際はそれぞれのファイルにおける定義と振る舞いの整合性に注意する必要があります。
