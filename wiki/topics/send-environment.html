send_environment

概念

send_environment は、Apache HTTP Server（httpd）において、HTTPリクエストに関連付けられた環境変数をFastCGIプロトコルの PARAMS レコードとしてシリアライズし、FastCGIバックエンドへ送信するための処理（関数）です。

なぜ重要か

FastCGIプロトコルでは、Webサーバーからアプリケーション（バックエンド）に対して、CGI環境変数などのリクエスト情報を PARAMS レコードとして渡す必要があります。send_environment は、プロキシ処理（mod_proxy_fcgi）や認証・認可処理（mod_authnz_fcgi）において、この環境変数の構築、エンコード、および送信を正しく行うための中心的な役割を担っています。

エンドツーエンドの流れ

1. **呼び出し**: mod_proxy_fcgi.c または mod_authnz_fcgi.c から send_environment が呼び出されます。 2. **環境変数の準備**: ap_add_common_vars および ap_add_cgi_vars を呼び出してCGI関連の環境変数を構築し、さらに fix_cgivars を用いて設定に基づいた環境変数の書き換え（動的調整）を行います。 3. **シリアライズ**: r->subprocess_env から環境変数を取得し、最大16KB単位の PARAMS レコードに分割してエンコードします。 4. **送信**: send_data を使用して、ヘッダとボディ（環境変数データ）をバックエンドへ送信します。 5. **終了通知**: すべての環境変数の送信が完了した後、データ長が0の PARAMS レコードを送信して、環境変数の送信終了をバックエンドに通知します。

※なお、呼び出し元（mod_proxy_fcgi や mod_authnz_fcgi）の全体的な処理における完全なエンドツーエンドのライフサイクルや、バックエンド側での受信後の詳細なフローは、提供された情報からは完全には明らかになっていません。

重要な不変条件

  • **環境変数の存在**: r->subprocess_env に環境変数が適切に格納されている必要があります。
  • **メモリ管理**: 一時的なメモリ割り当てには呼び出し元から渡される temp_pool が使用され、関数内で apr_pool_clear により適宜解放されます。
  • **単一セッション前提**: FastCGIの多重化（Multiplexing）はサポートされておらず、request_id は単一のセッションを前提として動作します。

よくある落とし穴

  • **パス処理の整合性**: 関数内で r->filename の一時的な書き換えと復元を行っているため、この周辺のコードを修正する際はパス処理の整合性が崩れないよう注意が必要です。
  • **データサイズ制限とエラー**: 環境変数が16KBを超える場合、複数の PARAMS レコードに分割して送信されます。この分割処理やエンコードに失敗した場合、警告ログが出力されて該当の変数がスキップされます。
  • **エラーの伝播**: send_data による送信処理が失敗した場合、エラーは即座に呼び出し元へ伝播されるため、呼び出し元での適切なハンドリングが必要です。
