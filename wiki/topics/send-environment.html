send_environment

概念

send_environment は、Apache HTTP Server（httpd-2.4.67）において、HTTP リクエストに関連付けられた環境変数を FastCGI プロトコルの PARAMS レコードとしてシリアライズし、バックエンド（認証プロバイダやプロキシ先アプリケーション）へ送信する処理、およびそれを実行する関数群を指します。

この処理は、mod_authnz_fcgi.c と mod_proxy_fcgi.c の両モジュールにそれぞれ同名の静的（static）関数として定義されています。

なぜ重要か

FastCGI プロトコルでは、Web サーバーからバックエンドに対して CGI 環境変数などのメタデータを伝達するために PARAMS レコードを使用します。send_environment は、Apache 内部のリクエスト情報（request_rec）から環境変数を抽出し、FastCGI 規格に準拠した形式にエンコードして送信する役割を担うため、FastCGI 連携におけるデータ転送の基盤として極めて重要です。

エンドツーエンドの流れ

現時点で、両モジュールを跨ぐ統一的なエンドツーエンドのフローの全貌や、mod_authnz_fcgi.c 側の詳細な内部アルゴリズムは完全には解析されていません。現在判明している各モジュールでの流れは以下の通りです。

mod_proxy_fcgi.c における流れ

1. **呼び出し**: プロキシ処理の過程（mod_proxy_fcgi.c:1056 など）から呼び出されます。 2. **環境変数の準備**: ap_add_common_vars および ap_add_cgi_vars を呼び出して CGI 関連の環境変数を構築し、fix_cgivars によって動的な書き換えを行います。 3. **シリアライズ**: r->subprocess_env から環境変数を取得し、ap_fcgi_encode_env などのユーティリティを用いて、最大 16KB 単位の PARAMS レコードに分割してエンコードします。 4. **送信**: send_data を使用して、ヘッダとエンコードされた環境変数をバックエンドへ送信します。 5. **終了通知**: すべての環境変数の送信後、データ長 0 の PARAMS レコードを送信して、環境変数送信の終了をバックエンドに通知します。

mod_authnz_fcgi.c における流れ

  • 呼び出し元である fcgi_do_request や req_rsp から呼び出され、ap_fcgi_encode_env や sendv_data などのユーティリティを介して環境変数を送信しますが、詳細な制御フローは未解析です。

重要な不変条件

現時点で判明している不変条件は以下の通りです（一部は mod_proxy_fcgi.c に特化しており、mod_authnz_fcgi.c 側の不変条件は未解析です）。

  • **環境変数の格納先**: mod_proxy_fcgi において、送信対象の環境変数は r->subprocess_env に適切に格納されている必要があります。
  • **メモリ管理**: 一時的なメモリ割り当てには temp_pool が使用され、呼び出し元で管理されます。
  • **多重化の制限**: FastCGI の多重化（Multiplexing）はサポートされておらず、request_id は単一のセッションを前提とします。
  • **終端レコードの送信**: すべての環境変数を送信し終えた後、必ずデータ長 0 の PARAMS レコードを送信して終端を示さなければなりません。

よくある落とし穴

  • **一時的なリクエスト情報の書き換え**: mod_proxy_fcgi.c の send_environment 内部では、処理中に r->filename の一時的な書き換えと復元を行っています。この周辺のコードを修正する際は、パス処理の整合性が崩れないよう注意が必要です。
  • **サイズ制限とスキップ**: 環境変数が 16KB を超える場合、自動的に複数の PARAMS レコードに分割されます。このエンコード処理に失敗した場合、警告ログが出力されて該当の環境変数がスキップされます。
  • **未解決の検討課題**: ソースコード内の XXX コメントが示すように、FastCGI 特有の環境変数追加の必要性については、将来的な検討課題として残されている可能性があります。
