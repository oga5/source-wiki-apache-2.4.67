send_environment

概念

send_environment は、Apache HTTP Server (httpd) において HTTP リクエストに関連付けられた環境変数を FastCGI プロトコルの PARAMS レコードとしてシリアライズし、FastCGI バックエンドへ送信するための処理または関数です。

なぜ重要か

FastCGI プロトコルを介して外部のアプリケーションや認証プロバイダと通信する際、CGI 規格に準拠した環境変数（ヘッダ情報やリクエストメタデータなど）をバックエンドへ正しく伝達するために不可欠な役割を担っています。

エンドツーエンドの流れ

modules/proxy/mod_proxy_fcgi.c における主な処理の流れは以下の通りです。

1. **環境変数の準備**: ap_add_common_vars および ap_add_cgi_vars を呼び出し、CGI 関連の環境変数を構築します。 2. **動的調整**: fix_cgivars を呼び出し、設定に基づいた環境変数の書き換えを行います。 3. **シリアライズ**: r->subprocess_env から環境変数を取得し、最大 16KB 単位の PARAMS レコードに分割してエンコードします。 4. **送信**: send_data を使用して、ヘッダとボディ（環境変数）をバックエンドへ送信します。 5. **終了通知**: 全ての環境変数の送信後、データ長 0 の PARAMS レコードを送信して終了を通知します。

※ modules/aaa/mod_authnz_fcgi.c からの呼び出しを含む、モジュール間を跨ぐ詳細なエンドツーエンドの連携フローの全容は、提供された情報からは完全には明らかになっていません。

重要な不変条件

  • **環境変数の存在**: r->subprocess_env に環境変数が格納されている必要があります。
  • **メモリ管理**: temp_pool は呼び出し元で管理され、関数内で apr_pool_clear により適宜解放されます。
  • **単一セッション前提**: FastCGI の多重化はサポートしておらず、request_id は単一のセッションを前提とします。

よくある落とし穴

  • **エンコード失敗時の挙動**: 環境変数のエンコードに失敗した場合、警告ログを出力して該当変数をスキップします。
  • **送信エラーの伝播**: send_data がエラーを返した場合、即座にそのエラーを呼び出し元へ伝播します。
  • **パス処理の整合性**: 処理中に r->filename の一時的な書き換えと復元を行っているため、コード修正時にはパス処理の整合性に注意する必要があります。
