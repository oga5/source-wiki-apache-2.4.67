dav_lock_indirect

概念

dav_lock_indirect は、WebDAV（RFC 4918）において親コレクションから継承された「間接ロック（Indirect Lock）」の情報を保持・管理するためのデータ構造（struct）です。ファイルシステムや汎用ロックモジュールにおいて、特定のリソースに関連付けられたロック状態を追跡・解決するために使用されます。

なぜ重要か

WebDAVでは、フォルダなどの親コレクションにかけられたロックが、その配下のリソース（ファイルやサブフォルダ）に間接的に適用されます。dav_lock_indirect は、この間接ロックのトークン、対応する直接ロック（Direct Lock）を検索するためのDBMキー、および有効期限（タイムアウト）を保持します。これにより、リソースに対する操作要求があった際に、親から継承されたロックが有効であるかを正しく検証（ロック解決）することが可能になります。

関与するモジュール

  • module:modules/dav/fs (ファイルシステムベースのロックプロバイダ)
  • module:modules/dav/lock (汎用ロックプロバイダ)

主要エンティティ

  • symbol:modules/dav/fs/lock.c#dav_lock_indirect (ファイルシステムロック用の構造体定義)
  • symbol:modules/dav/lock/locks.c#dav_lock_indirect (汎用ロック用の構造体定義)

両モジュールにおいて、構造体は以下の同一の形状で定義されています。

typedef struct dav_lock_indirect
{
    dav_locktoken *locktoken;
    apr_datum_t key;
    struct dav_lock_indirect *next;
    time_t timeout;
} dav_lock_indirect;
  • locktoken: ロックを一意に識別するための UUID ラッパー。
  • key: DBM 内の直接ロック（dav_lock_discovery）を検索するためのキー情報。
  • next: 同一リソースに関連付けられた間接ロックのリストを形成するポインタ。
  • timeout: ロックの有効期限。

エンドツーエンドの流れ

1. **DBMからの読み込みと構築**: DBM（データベース）からシリアライズされたロックデータが読み込まれ、dav_fs_load_lock_record（modules/dav/fs）または dav_generic_load_lock_record（modules/dav/lock）によって、dav_lock_indirect 構造体のリストとしてメモリ上に再構築されます。 2. **有効期限の判定**: 読み込み処理の過程で、各間接ロックの timeout が現在時刻（time(NULL)）と比較され、期限切れのロックはリストから自動的に除外されます。 3. **ロックの解決（検証）**: dav_fs_resolve または dav_generic_resolve が呼び出され、dav_lock_indirect の key を使用して、対応する直接ロック（dav_lock_discovery）をDBMから再帰的に検索・特定し、間接ロックの有効性を検証します。 4. **DBMへの保存**: ロック状態の変更時、dav_generic_save_lock_record 等によって、この構造体の内容がバイナリ形式にシリアライズされてDBMへ書き戻されます。

※ なお、Apache HTTPD のコアや mod_dav のメインハンドラからこれらの関数が呼び出される具体的なトリガーや、リクエスト処理全体にわたる完全なエンドツーエンドの制御フローの詳細は、提供されたコンテキストからは完全には明らかになっていません。

重要な不変条件

  • **単方向リストの形成**: next ポインタにより、同一リソースに関連付けられた間接ロックの単方向リストを形成します。
  • **メモリ管理**: すべてのインスタンスは、リクエスト処理に関連付けられたメモリプール（request_rec に紐付くプール）から割り当てられ、リクエスト終了時に自動的に解放されます。明示的な個別解放は不要です。
  • **DBMキーの整合性**: key（apr_datum_t）は、対応する直接ロックをDBMから一意に検索できる有効な識別子でなければなりません。

よくある落とし穴

  • **バイナリシリアライズとの密結合**: dav_lock_indirect は DBM へのシリアライズ/デシリアライズ処理（dav_generic_save_lock_record / dav_generic_load_lock_record など）と密に結合しています。構造体のメンバを変更・追加する場合、オフセット計算やバイナリ形式の後方互換性を慎重に更新しなければ、DBM読み込み時にデータ破損や不整合を引き起こす原因となります。
  • **キーの無効化**: 構造体内の key が破損または無効になった場合、dav_fs_resolve / dav_generic_resolve による直接ロックの解決に失敗し、ロックの検証が正しく行えなくなります。
