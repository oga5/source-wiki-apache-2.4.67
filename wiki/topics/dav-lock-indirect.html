DAV_LOCK_INDIRECT

概念

DAV_LOCK_INDIRECT（およびそれに対応する dav_lock_indirect 構造体）は、WebDAVにおける「間接ロック（Indirect Lock）」の情報を保持・管理するためのデータ構造および概念です。

WebDAVでは、親コレクション（ディレクトリ）に適用されたロックが、その配下のリソース（ファイルやサブコレクション）に継承されます。この継承されたロックを「間接ロック」と呼び、リソースごとのロック状態の追跡やアクセス制御の検証に使用されます。

なぜ重要か

間接ロックを正しく管理することは、WebDAVの階層的なロックモデルを維持するために不可欠です。dav_lock_indirect 構造体は、親から継承されたロックのトークン、対応する直接ロック（Direct Lock）を検索するためのキー、および有効期限（タイムアウト）を保持します。これにより、リソースへのアクセス要求があった際に、親コレクションのロック状態を再帰的に解決（Resolve）し、アクセスを許可するかどうかを正しく判定できます。

関与するモジュール

提供された情報に基づくと、以下のモジュールが関与しています。

  • module:modules/dav/fs (ファイルシステムベースのDAVロックプロバイダ実装)
  • module:modules/dav/lock (ジェネリックなDAVロック管理実装)

主要エンティティ

  • **struct dav_lock_indirect** (modules/dav/fs/lock.c および modules/dav/lock/locks.c):

    • 間接ロックの情報を保持する構造体であり、以下の形状を持ちます。 ``c typedef struct dav_lock_indirect { dav_locktoken *locktoken; apr_datum_t key; struct dav_lock_indirect *next; time_t timeout; } dav_lock_indirect; ``

  • locktoken: ロックを一意に識別するためのUUIDラッパー。
  • key: 対応する直接ロックをDBMから検索するための識別子。
  • next: 同一リソースに関連付けられた間接ロックのリストを形成するポインタ。
  • timeout: ロックの有効期限。

エンドツーエンドの流れ

1. **DBMからのロードと構築**: リクエスト処理時、dav_fs_load_lock_record（modules/dav/fs）または dav_generic_load_lock_record（modules/dav/lock）が、DBMにシリアライズされて保存されていたバイナリデータから dav_lock_indirect 構造体を構築します。 2. **期限切れの判定**: ロード処理の過程で、構造体内の timeout が現在時刻（time(NULL)）と比較され、期限切れのロックは自動的にリストから除外されます。 3. **ロックの解決（Resolve）**: dav_fs_resolve または dav_generic_resolve が呼び出され、dav_lock_indirect の key を使用して親リソースの直接ロックを特定し、トークンの一致を確認して間接ロックの有効性を検証します。 4. **DBMへの保存（永続化）**: ロック状態の変更時、dav_generic_save_lock_record などの関数により、dav_lock_indirect の内容がバイナリ形式にシリアライズされ、DBMへ書き込まれます。

*注意: modules/dav/fs と modules/dav/lock の間での具体的な呼び出し関係や、詳細なエンドツーエンドの制御フローの全貌は、提供された情報からは完全には特定されていません。*

重要な不変条件

  • **単方向リストの形成**: next ポインタは、同一リソースに関連付けられた間接ロックの有効な単方向リストを形成しなければなりません。
  • **メモリ管理**: すべての dav_lock_indirect インスタンスは、リクエスト処理に関連付けられたメモリプール（request_rec に紐付くプール）から割り当てられ、リクエスト終了時に自動的に解放されます。明示的な解放処理は不要です。
  • **有効期限の整合性**: timeout を過ぎた間接ロックは、ロード処理時に除外され、有効なロックとして解決されてはなりません。

よくある落とし穴

  • **シリアライズ不整合**: dav_lock_indirect 構造体は、DBMへのバイナリシリアライズ/デシリアライズロジック（dav_generic_save_lock_record / dav_generic_load_lock_record など）と密結合しています。構造体のメンバを変更する場合、オフセット計算やシリアライズ処理も同時に更新しなければ、データの破損や不整合を引き起こします。
  • **キー破損による解決失敗**: 構造体内の key が無効または破損している場合、dav_fs_resolve による直接ロックの解決が失敗し、ロックの検証ができなくなります。
  • **DBMデータ破損時のオフセット不整合**: DBMから読み込んだバイナリデータが破損している場合、デシリアライズ時のオフセット計算が狂い、メモリ不整合を引き起こす可能性があります。

  • modules/dav/lock/locks.c: ジェネリックなロック管理における dav_lock_indirect の定義、dav_generic_load_lock_record、dav_generic_save_lock_record、dav_generic_resolve などの実装。
