dav_lock_indirect

概念

dav_lock_indirect は、WebDAVにおいて親コレクションから継承された「間接ロック（Indirect Lock）」の情報を保持するための構造体です。ファイルシステム上のリソースに関連付けられたロック状態を管理・追跡し、解決するために使用されます。

なぜ重要か

WebDAVでは、親ディレクトリ（コレクション）に適用されたロックが、その配下のリソース（ファイルやサブコレクション）に間接的に適用されます。dav_lock_indirect は、この間接ロックを表現し、DBM（データベース）に永続化されたロック情報とメモリ上のロック解決処理を仲介する役割を果たします。これにより、リソースに対するアクセス権限の検証や、ロックの有効期限管理が正確に行われます。

関与するモジュール

  • module:modules/dav/fs（ファイルシステムベースのロックプロバイダ実装）
  • module:modules/dav/lock（汎用的なロック管理実装）

主要エンティティ

  • **symbol:modules/dav/fs/lock.c#dav_lock_indirect (struct)**: modules/dav/fs において、DBMから読み込まれた間接ロック情報を保持し、ロック解決処理（dav_fs_resolve）等で参照されます。
  • **symbol:modules/dav/lock/locks.c#dav_lock_indirect (struct)**: modules/dav/lock において、DBMとのシリアライズ/デシリアライズ（dav_generic_save_lock_record / dav_generic_load_lock_record）や、ロック解決処理（dav_generic_resolve）で参照されます。
  • **symbol:modules/dav/fs/lock.c#DAV_LOCK_INDIRECT** および **symbol:modules/dav/lock/locks.c#DAV_LOCK_INDIRECT**: 関連する定数またはマクロ（詳細な定義内容は提供されたコンテキストからは未詳）。

エンドツーエンドの流れ

1. **DBMからの読み込みと構築**:

  • dav_fs_load_lock_record（modules/dav/fs）または dav_generic_load_lock_record（modules/dav/lock）により、DBMからシリアライズされたバイナリデータが読み込まれます。
  • 読み込まれたデータから dav_lock_indirect 構造体が構築されます。この際、timeout が現在時刻（time(NULL)）と比較され、期限切れのロックはリストから除外されます。

    • 2. **ロックの解決と検証**:

  • dav_fs_resolve または dav_generic_resolve が呼び出され、dav_lock_indirect の key（apr_datum_t）を使用して、対応する直接ロック（Direct Lock）をDBMから再帰的に検索・特定します。
  • 親リソースのロックレコードとトークン（locktoken）の一致を確認し、間接ロックの有効性を検証します。

    • 3. **DBMへの保存**:

  • dav_generic_save_lock_record 等により、dav_lock_indirect 構造体の内容がバイナリ形式にシリアライズされ、DBMへ書き込まれます。

*注意: 詳細な呼び出しシーケンスや、リクエスト処理のライフサイクル全体における完全なエンドツーエンドのフローは、提供されたコンテキストからは完全には明らかになっていません。*

重要な不変条件

  • **単方向リストの形成**: next ポインタは、同一リソースに関連付けられた間接ロックのリスト（単方向リスト）を形成しなければなりません。
  • **直接ロックへの参照**: key メンバは、対応する直接ロック（Direct Lock）をDBMから検索するための有効な識別子（apr_datum_t）を保持していなければなりません。
  • **メモリ管理**: すべてのインスタンスは、リクエスト処理に関連付けられたメモリプール（request_rec に紐付くプール）から割り当てられ、明示的な解放は不要です。
  • **有効期限**: timeout を過ぎたロックは、読み込み処理の過程で自動的に除外されます。

よくある落とし穴

  • **バイナリシリアライズとの密結合**: dav_lock_indirect 構造体は、DBMへのバイナリシリアライズ/デシリアライズロジック（dav_generic_save_lock_record / dav_generic_load_lock_record など）と密に結合しています。構造体のメンバを変更する場合、オフセット計算やシリアライズ処理も同時に更新しなければ、データの不整合や破損を引き起こします。
  • **キーの破損**: 構造体内の key が無効または破損している場合、dav_fs_resolve や dav_generic_resolve による直接ロックの解決が失敗します。
  • **後方互換性**: 将来的な構造体の拡張時には、DBMに保存されている既存のバイナリデータとの後方互換性に注意する必要があります。
