abort_on_oom

概念

abort_on_oom は、Apache HTTP Server (httpd) およびその周辺ツールにおいて、メモリ割り当て失敗（OOM: Out of Memory）が発生した際にプロセスを安全かつ即座に強制終了させるためのコールバック関数（エラーハンドラ）です。

APR (Apache Portable Runtime) のメモリプール管理機能において、apr_pool_abort_set を介して登録され、メモリ不足時に呼び出されます。

なぜ重要か

メモリ不足が発生した際、不完全な状態での処理継続や予期せぬ Null ポインタ参照によるクラッシュを防ぎ、プロセスを即座に終了させることで、システムの整合性を保ち、予測可能なエラーハンドリングを実現するために重要です。

サーバー本体（httpd）だけでなく、HTTP/2 モジュールや各種サポートツール（ab, htpasswd, htdbm など）でも一貫してこの仕組みが導入されています。

主要エンティティ

  • **symbol:modules/http2/h2_mplx.c#abort_on_oom**

    • HTTP/2 接続マルチプレクサ (h2_mplx) 内で apr_pool_abort_set に登録されるコールバック。内部で ap_abort_on_oom() を呼び出します。

  • **symbol:server/main.c#abort_on_oom**

    • サーバー本体の初期化処理等で APR プールのエラーハンドラとして登録される static 関数。内部で ap_abort_on_oom() を呼び出します。

  • **symbol:support/ab.c#abort_on_oom**

    • ベンチマークツール ab 内で定義される static 関数。標準エラー出力に "Could not allocate memory" を出力し、exit(1) で終了します。

  • **symbol:support/passwd_common.c#abort_on_oom**

    • パスワード関連ツール（htpasswd, htdbm 等）で共有される関数。標準エラー出力に "Error: out of memory\n" を write システムコールで書き込み、abort() を呼び出します。

エンドツーエンドの流れ

1. **初期化と登録**: 各コンポーネント（サーバー本体、HTTP/2 モジュール、サポートツール）の初期化フェーズにおいて、APR メモリプール (apr_pool_t) が作成されます。作成されたプールに対して、apr_pool_abort_set を呼び出し、それぞれの abort_on_oom 関数をエラーハンドラとして登録します。 2. **OOMの発生**: APR のメモリ割り当て処理中に物理メモリが不足し、割り当てに失敗します。 3. **ハンドラの呼び出し**: APR のプール管理機能が、登録されている abort_on_oom コールバックを呼び出します。 4. **プロセスの強制終了**:

  • server/main.c や modules/http2/h2_mplx.c では、ap_abort_on_oom() を呼び出してプロセスを終了させます。
  • support/ab.c では、標準エラー出力にメッセージを出力後、exit(1) を呼び出します。
  • support/passwd_common.c では、write() システムコールで標準エラー出力にメッセージを書き込み、abort() を呼び出します。

※注: APR 内部から abort_on_oom が呼び出される正確なトリガー条件や、ap_abort_on_oom() の内部実装の詳細なエンドツーエンドのフローは、提供されたコンテキストからは完全には明らかになっていません。

重要な不変条件

  • **到達不能性 (Non-returning)**: abort_on_oom は呼び出された場合、呼び出し元へ制御を戻してはなりません（プロセスを確実に終了させる）。シグネチャ上は int を返すように定義され、内部で return 文が記述されている場合もありますが、実際には exit(), abort(), または ap_abort_on_oom() によってプロセスが終了するため、そのコードパスは到達不能（NOTREACHED）です。
  • **登録の必須性**: メモリ不足時にこのハンドラが機能するためには、対象となる apr_pool_t に対して apr_pool_abort_set を用いて事前にハンドラが登録されていなければなりません。

よくある落とし穴

  • **同名関数の重複とスコープ**: abort_on_oom という名前の関数は、複数のソースファイル（server/main.c, support/ab.c など）で static として定義されているか、あるいは support/passwd_common.c のようにグローバルスコープで定義されています。リンク時の重複定義や、意図しない実装の呼び出しを防ぐため、スコープ管理に注意する必要があります。
  • **ハンドラ未登録時の挙動**: apr_pool_abort_set による登録が行われていないプールで OOM が発生した場合、本ハンドラは起動しません。
  • **シグナル安全性の考慮**: support/passwd_common.c の実装では、write システムコールをループさせて確実にエラーメッセージを出力しようとしますが、他の実装（ab.c の fprintf など）ではバッファリングを伴う標準入出力関数が使われており、OOM 発生時のコンテキストによっては安全に動作しない可能性があります。
