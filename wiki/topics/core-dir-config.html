core_dir_config

概念

core_dir_config は、Apache HTTP Server においてディレクトリ単位の設定情報を保持するための構造体です。Options、AllowOverride、フィルタ設定、リソース制限、アクセス制御ルールなど、リクエスト処理の基盤となる設定状態を管理します。

なぜ重要か

この構造体は、Apacheがリクエストを処理する際、対象のディレクトリに適用すべき動作（シンボリックリンクの許可、文字セット、リソース制限など）を決定するための設定値を保持します。また、ディレクトリ階層に応じた設定の継承やマージ（directory_walk など）を行うための基盤データ構造として機能します。

関与するモジュール

  • module:include
  • module:server

主要エンティティ

  • symbol:include/http_core.h#core_dir_config (ディレクトリ設定を保持する構造体)
  • symbol:server/core.c#create_core_dir_config (構造体を生成・初期化する関数)

エンドツーエンドの流れ

1. **メモリ確保と初期化**: ディレクトリ単位の設定構造体が必要になると、server/core.c 内の create_core_dir_config が呼び出されます。この関数は apr_pcalloc を使用して core_dir_config のメモリを確保し、各メンバを未設定状態を示す定数（_UNSET 系）やデフォルト値で初期化します。 2. **設定の適用とマージ**: 設定ファイル（httpd.conf など）の解析結果や、親ディレクトリからの継承設定が反映されます。親から子への設定継承は、ディレクトリ単位の設定マージ処理を通じて行われます。 3. **リクエスト処理時の参照**: リクエスト処理の実行時、core モジュールがこの構造体に格納された設定値を参照して動作を決定します。

※設定ファイルの解析から directory_walk を経て最終的にリクエスト処理へ至る具体的な関数呼び出しの完全なエンドツーエンドの流れは、提供されたソースコード抜粋からは特定されていません。

重要な不変条件

  • **オプション追加・削除の排他性**:

    • opts_add と opts_remove は排他的でなければなりません（opts_add & opts_remove == 0）。各ビットはどちらか一方にのみ記録され、相対的なオプション変更の最終状態を保持します。

  • **ディレクトリ有無による初期値の差異**:

    • create_core_dir_config において、dir 引数が指定されている場合、conf->opts は OPT_UNSET に設定されます。指定がない（NULL の）場合は OPT_UNSET|OPT_SYM_LINKS が適用されます。

  • **メモリのゼロクリア**:

    • apr_pcalloc を用いて構造体が確保されるため、明示的に初期化されないメンバは NULL または 0 に初期化されます。

よくある落とし穴

  • **モジュールバイナリ互換性（MMN）**:

    • core_dir_config 構造体に新しいフィールドを追加する場合、include/ap_mmn.h で定義されるMMN（Module Magic Number）のバージョン更新を伴う可能性があります。

  • **新規フィールドの初期化漏れ**:

    • 構造体に新しい設定項目を追加した際、create_core_dir_config で適切な _UNSET 値やデフォルト値による初期化を行わないと、後続のマージ処理（merge_core_dir_configs など）で予期しない動作を引き起こす原因になります。

  • **プラットフォーム依存のリソース制限**:

    • リソース制限（rlimit）は RLIMIT_CPU などのプラットフォーム依存の定義に依存しているため、動作環境によって構造体のサイズや有効性が異なる場合があります。

  • **未使用フィールド**:

    • response_code_strings メンバは定義されていますが、現在は使用されていません。
