core_dir_config

概念

core_dir_config は、Apache HTTP Server におけるディレクトリ単位の設定情報を保持するための構造体です。Options、AllowOverride、フィルタ設定、リソース制限、アクセス制御ルールなど、リクエスト処理の基盤となる設定状態を管理します。

なぜ重要か

この構造体は、ディレクトリごとの動作（シンボリックリンクの許可、オーバーライド、文字セット、リクエストボディ制限など）をきめ細かく制御し、リクエスト処理時に core モジュールが参照する設定値の基盤となります。また、設定の継承やマージ（directory_walk 等）において中心的な役割を果たします。

関与するモジュール

  • module:include
  • module:server

主要エンティティ

  • symbol:include/http_core.h#core_dir_config：ディレクトリ単位の設定情報を保持する構造体定義。
  • symbol:server/core.c#create_core_dir_config：core_dir_config 構造体をメモリ上に確保し、デフォルト値で初期化する関数。

エンドツーエンドの流れ

1. **メモリ確保と初期化**: server/core.c の create_core_dir_config が呼び出され、apr_pcalloc を用いて core_dir_config 構造体のメモリを確保し、デフォルト値や未設定状態を示す定数（_UNSET 系）で初期化します。 2. **設定の適用とマージ**: 設定ファイル（httpd.conf 等）の解析結果や、親ディレクトリからの継承設定が反映されます。ディレクトリ単位の設定マージ（directory_walk 等）において、親から子へ設定が継承されます。 3. **リクエスト処理時の参照**: リクエスト処理時に core モジュールがこの構造体を参照して動作を決定します。

※設定解析の具体的なトリガーや、merge_core_dir_configs などの詳細なマージ処理、および directory_walk の具体的な実行フローは、提供されたソース抜粋からは完全には特定されていません。

重要な不変条件

  • **オプション追加・削除の排他性**:

    • opts_add と opts_remove は排他的である必要があります（opts_add & opts_remove == 0）。各ビットはどちらか一方にのみ記録され、相対的なオプション変更の最終状態を保持します。

  • **初期化時のディレクトリ有無による挙動**:

    • create_core_dir_config において、dir 引数が指定されている場合、conf->opts は OPT_UNSET に設定されます。指定がない（NULL の）場合は OPT_UNSET|OPT_SYM_LINKS が適用されます。

よくある落とし穴

  • **MMN（モジュールバイナリ互換性）の破損**:

    • 本構造体にフィールドを追加すると、include/ap_mmn.h で定義されるモジュールバイナリ互換性（MMN）のバージョン更新が必要になる可能性があります。

  • **初期化漏れ**:

    • 新しい設定項目を core_dir_config に追加した際、create_core_dir_config で適切な _UNSET 値やデフォルト値による初期化を怠ると、後続のマージ処理で予期しない動作を引き起こす可能性があります。

  • **プラットフォーム依存性**:

    • リソース制限（rlimit）はプラットフォーム依存の定義（RLIMIT_CPU 等）に依存するため、環境によって構造体のサイズや有効性が異なります。

  • **未使用フィールド**:

    • response_code_strings は構造体内に定義されていますが、現在は使用されていません。

次に読む場所

  • include/http_core.h：core_dir_config 構造体の詳細なフィールド定義やビットマスク型（allow_options_t など）を確認できます。
  • server/core.c：create_core_dir_config による初期化処理の具体的な実装を確認できます。
