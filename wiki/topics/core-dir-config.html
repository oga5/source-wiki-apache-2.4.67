core_dir_config

概念

core_dir_config は、Apache HTTP Server におけるディレクトリ単位の設定情報を保持するための中心的な構造体です。Options、AllowOverride、フィルタ設定、リソース制限、アクセス制御ルールなど、リクエスト処理の基盤となる設定状態を管理します。この構造体は、設定の初期化関数である create_core_dir_config を通じてメモリ上に確保され、デフォルト値や未設定状態を示す値で初期化されます。

なぜ重要か

この構造体は、Apache のディレクトリ単位の設定マージ（directory_walk など）やリクエスト処理時の挙動を決定する基盤となります。また、この構造体にフィールドを追加・変更することは、モジュールバイナリ互換性（MMN: Module Magic Number）の更新に直結するため、コアシステムおよび外部モジュールの互換性を維持する上で極めて重要なエンティティです。

エンドツーエンドの流れ

1. **初期化**: 設定ファイルの解析時などに、server/core.c 内で create_core_dir_config が呼び出されます。この関数は apr_pool_t からメモリを確保し、構造体の各メンバをデフォルト値や未設定状態（_UNSET 系定数）で初期化します。 2. **マージと適用**: 初期化された設定は、後続の処理（merge_core_dir_configs など）において親ディレクトリの設定とマージされ、最終的なディレクトリ単位の設定が決定されます。 3. **リクエスト処理での参照**: リクエスト処理時に core モジュールがこの設定を参照し、適切なアクセス制御やフィルタの適用を行います。

※なお、設定解析の開始からリクエスト処理における参照に至る完全なエンドツーエンドの制御フローの全容は、提供されたソースコード抜粋からは完全には特定されていません。

重要な不変条件

  • **オプションの排他性**: 相対的なオプション変更を保持する opts_add と opts_remove は、互いに排他的でなければなりません（opts_add & opts_remove == 0）。
  • **メモリ初期化**: apr_pcalloc を使用してメモリを確保するため、明示的に初期化されないメンバは NULL または 0 に初期化されます。
  • **ディレクトリ有無による初期値の差異**: create_core_dir_config において、対象ディレクトリ dir が指定されている場合、opts の初期値は OPT_UNSET に設定されます。指定がない（NULL の）場合は OPT_UNSET|OPT_SYM_LINKS が適用されます。

よくある落とし穴

  • **初期化漏れ**: core_dir_config に新しい設定項目を追加した際、create_core_dir_config で適切な _UNSET 値やデフォルト値による初期化を怠ると、マージ処理で意図しない挙動を引き起こす原因になります。
  • **バイナリ互換性の破壊**: 構造体のレイアウト変更（フィールドの追加・削除など）は、MMN の更新を伴うため、既存のコンパイル済みモジュールとの互換性を損なうリスクがあります。
  • **プラットフォーム依存性**: リソース制限（rlimit）などの一部のメンバは、プラットフォーム依存の定義（RLIMIT_CPU など）に依存するため、ビルド環境や実行環境によって構造体のサイズや有効性が変化する可能性があります。
  • **未使用フィールド**: response_code_strings メンバは定義されていますが、現在は使用されていません。

次に読む場所

  • include/http_core.h: core_dir_config 構造体の詳細なフィールド定義や、関連するビットマスク型（allow_options_t など）を確認できます。
  • server/core.c: create_core_dir_config による初期化ロジックや、後続のマージ処理（merge_core_dir_configs など）の実装を確認できます。
