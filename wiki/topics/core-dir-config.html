core_dir_config

概念

core_dir_config は、Apache HTTP Server においてディレクトリ単位の設定情報を保持するための構造体です。これには、オプション（Options）、上書き許可（AllowOverride）、フィルタ設定、リソース制限、アクセス制御ルールなど、リクエスト処理の基盤となる設定状態が格納されます。

なぜ重要か

この構造体は、リクエスト処理時に core モジュールが参照する設定値の基盤であり、ディレクトリごとの動作（<Directory> や <If> などの条件付きブロック）を制御・マージするために不可欠です。また、構造体の変更はモジュールバイナリ互換性（MMN）のバージョン更新に影響を与えるため、保守開発において極めて重要な位置を占めます。

関与するモジュール

  • module:include
  • module:server

主要エンティティ

  • symbol:include/http_core.h#core_dir_config

    • ディレクトリ単位の設定情報を保持する構造体。

  • symbol:server/core.c#create_core_dir_config

    • core_dir_config 構造体をメモリ上に確保し、デフォルト値で初期化する関数。

エンドツーエンドの流れ

1. **メモリ確保と初期化**: server/core.c の create_core_dir_config が呼び出され、apr_pcalloc を用いて core_dir_config 構造体のメモリがゼロクリアされた状態で確保されます。 2. **デフォルト値の設定**: create_core_dir_config 内で、各種フラグや制限値（opts, override, limit_req_body など）が、未設定状態を示す定数（_UNSET 系）やデフォルト値で初期化されます。 3. **設定の適用とマージ**: ディレクトリ単位の設定マージ（directory_walk 等）において、親ディレクトリから子ディレクトリへ設定が継承・マージされ、最終的な設定がリクエスト処理時に参照されます。

*注意: 設定ファイルの解析からリクエスト処理に至る具体的なエンドツーエンドの完全な制御フローの詳細は、提供されたソース抜粋からは特定されていません。*

重要な不変条件

  • **オプション追加・削除の排他性**:

    • opts_add と opts_remove は排他的でなければなりません（opts_add & opts_remove == 0）。各ビットはどちらか一方にのみ記録され、相対的なオプション変更の最終状態を保持します。

  • **初期化時の opts の状態**:

    • create_core_dir_config において、dir 引数が指定されている場合は conf->opts が OPT_UNSET に設定され、指定がない（NULL の）場合は OPT_UNSET|OPT_SYM_LINKS が適用されます。

よくある落とし穴

  • **MMNの破損**:

    • core_dir_config 構造体にフィールドを追加・変更すると、include/ap_mmn.h で定義されるモジュールバイナリ互換性（MMN）のバージョン更新が必要になる可能性があります。

  • **未設定状態のハンドリング**:

    • max_ranges などの数値設定において、AP_MAXRANGES_UNSET などの定数による未設定状態を適切にハンドリングする必要があります。

  • **プラットフォーム依存性**:

    • リソース制限（rlimit）はプラットフォーム依存の定義（RLIMIT_CPU 等）に依存するため、環境によって構造体のサイズや有効性が異なる場合があります。

  • **マージ前提の設計**:

    • create_core_dir_config で初期化された各メンバは、後続の merge_core_dir_configs でマージされる前提で設計されています。新規項目を追加する際は、適切な初期化（_UNSET 値など）を怠るとマージ時に予期しない挙動を引き起こす原因になります。
