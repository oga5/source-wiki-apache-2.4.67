request_rec

概念

request_rec は、Apache HTTP Server における単一の HTTP リクエストのライフサイクル全体を管理・追跡するための中心的なデータ構造（struct）です。リクエストの状態、URI、メソッド、ヘッダー、フィルタチェーン、メモリプールなどの情報を保持・共有するために使用されます。

なぜ重要か

Apache HTTP Server のリクエスト処理パイプライン（翻訳、認証、アクセス制御、コンテンツ生成、ログ記録など）の各フェーズにおいて、モジュール間でリクエストの状態やコンテキストを共有・伝播するための標準的なインターフェースとして機能するためです。

例えば、mod_log_config はログ記録のために、mod_rewrite はURI書き換えのために、mod_headers はヘッダー操作のために、それぞれ request_rec を参照・更新します。また、mod_lua を通じて Lua スクリプトから Apache 内部のリクエスト情報へアクセスする際のブリッジとしても機能します。

関与するモジュール

提供されたソースコード群に基づき、以下のモジュールが request_rec の定義や利用に関与しています。

  • module:include （コアヘッダー include/httpd.h を提供）
  • module:modules/loggers （mod_log_config.h を提供）
  • module:modules/lua （lua_request.c を提供）
  • module:modules/mappers （mod_rewrite.h を提供）
  • module:modules/metadata （mod_headers.c を提供）
  • module:modules/proxy （mod_proxy_hcheck.c を提供）
  • module:server （config.c を提供）

主要エンティティ

  • **symbol:include/httpd.h#request_rec**: 構造体の実体定義（httpd.h:822）。リクエスト処理の中核となるメンバを保持します。
  • **symbol:modules/loggers/mod_log_config.h#request_rec**: ログ記録モジュールにおけるエイリアス定義。ログハンドラ（ap_log_handler_fn_t）の引数として使用されます。
  • **symbol:modules/lua/lua_request.c#request_rec**: Lua スクリプトから Apache2.Request userdata を介して C 言語の request_rec フィールドを操作するためのブリッジ実装。
  • **symbol:modules/mappers/mod_rewrite.h#request_rec**: 書き換えルールやマップ関数（rewrite_mapfunc_t）がリクエスト属性を評価・操作するためのコンテキスト。
  • **symbol:modules/metadata/mod_headers.c#request_rec**: headers_in、headers_out、err_headers_out などのヘッダーテーブルを直接操作するためのコンテキスト。
  • **symbol:modules/proxy/mod_proxy_hcheck.c#create_request_rec**: プロキシのヘルスチェック等でリクエスト構造体を生成するための関数。
  • **symbol:server/config.c#request_rec**: サーバー設定やハンドラ関数でリクエストコンテキストを扱うためのエイリアス定義。

エンドツーエンドの流れ

リクエストの生成から完全な破棄に至るエンドツーエンドのライフサイクル全体の詳細な流れは、提供された情報からは完全には明らかになっていません。ただし、部分的な処理の流れとして以下が判明しています。

1. **リクエストの伝播**: HTTP リクエスト処理の各フェーズにおいて、Apache コアから各モジュールのフック関数へ request_rec のポインタが引数として渡されます。 2. **情報の参照と書き換え**:

  • mod_rewrite は、リクエストの URI や環境変数を評価し、書き換えを行います。
  • mod_headers（do_headers_fixup）は、構造体内の headers_in や headers_out などのテーブルを直接操作してヘッダーを追加・削除・置換します。
  • mod_lua（req_dispatch）は、Lua からのフィールドアクセス（r:field）に応じて C 言語のフィールド値を Lua の型に変換して返します。

    • 3. **ログ出力**: リクエスト処理の最終段階（log_transaction フック等）で、mod_log_config などのログモジュールが request_rec から必要な情報を抽出してログを構築・出力します。

重要な不変条件

  • **メモリ管理の不変条件**: request_rec 内の多くの動的メモリ割り当ては、リクエスト専用のメモリプール（apr_pool_t 型の r->pool）に紐付いています。リクエスト終了時にこのプールごと自動的に解放されるため、手動で解放してはなりません。
  • **生存期間の不変条件**: リクエストの処理中（ログ記録フェーズを含む）は、常に有効なポインタとして存在し、有効なメモリプールに関連付けられていなければなりません。
  • **アクセス・変更規約**: 非常に巨大な構造体であり、複数モジュール間で共有されるため、各モジュールは自身のコンテキスト領域（request_config など）以外を不用意に変更してはなりません。また、Lua 経由のアクセス（req_newindex）など、すべてのフィールドが書き込み可能ではない点に留意する必要があります。

よくある落とし穴

  • **NULL ポインタ参照**: リクエスト処理の初期段階や異常終了時、あるいは適切に初期化されていない状態で request_rec ポインタにアクセスすると、セグメンテーション違反（セグフォ）を引き起こします。
  • **バイナリ互換性の崩壊**: request_rec の構造体レイアウトは Apache のバージョンアップ（ap_mmn.h に記録される Module Magic Number の変更）に伴い、メンバーの追加や削除（例: content_language の削除や used_path_info の追加）が行われます。そのため、直接メンバーにアクセスするモジュールはバイナリ互換性に注意が必要です。
  • **タイミング依存の未初期化アクセス**: フックの実行順序（例: early 処理と通常のフィルタ処理）やタイミングによっては、特定のフィールドがまだ初期化されていない、あるいは既に解放されている場合があります。
  • **メモリ管理の混同**: mod_lua などのスクリプト環境において、Lua 側のオブジェクト寿命管理と r->pool による C 側のメモリ管理を混同すると、メモリリークや不正アクセスにつながります。

次に読む場所

  • **include/httpd.h**: request_rec 構造体のメンバ定義やメモリプール、関連する conn_rec や server_rec の定義を確認できます。
  • **modules/lua/lua_request.c**: Lua から request_rec の各フィールドへアクセスする際のマッピングロジック（req_dispatch など）が記述されています。
  • **modules/metadata/mod_headers.c**: headers_in や headers_out などのヘッダーテーブルを操作する具体的な実装例（do_headers_fixup）を確認できます。
  • **modules/proxy/mod_proxy_hcheck.c**: create_request_rec によるリクエスト構造体の生成処理を確認できます。
