request_rec

概念

request_rec は、Apache HTTP Server における単一の HTTP リクエストのライフサイクル全体を管理・追跡するための中心的なデータ構造（struct）です。 リクエストの状態、URI、メソッド、ヘッダー（headers_in, headers_out, err_headers_out）、フィルタチェーン、メモリプール（apr_pool_t）などのメタデータを保持し、リクエスト処理の各フェーズ（翻訳、認証、ハンドラ実行、ログ記録など）を通じてモジュール間で情報を共有・伝達する役割を果たします。

なぜ重要か

Apache HTTP Server のリクエスト処理パイプラインにおいて、ほぼすべてのモジュールがこの構造体を介してリクエストの状態を共有・操作するためです。 例えば、mod_headers によるヘッダーの追加・削除、mod_rewrite によるURIの書き換え、mod_log_config によるログ出力、mod_lua によるスクリプトからのリクエスト制御など、サーバーの動的機能の根幹が request_rec に依存しています。

エンドツーエンドの流れ

リクエスト処理の完全なエンドツーエンドの流れ（生成から完全な破棄、および内部の全遷移）の詳細は、提供された情報からは完全には明らかになっていません。ただし、部分的な流れとして以下が判明しています：

1. **生成**: mod_proxy_hcheck の create_request_rec などにより、必要に応じて構造体が生成・初期化されます。 2. **フェーズ処理と伝達**: 解析、翻訳、認証、アクセス制御、コンテンツ生成などの各フェーズにおいて、Apache コアから各モジュールのフック関数（mod_headers の do_headers_fixup や mod_rewrite の書き換え関数など）に request_rec * として渡され、状態が更新されます。 3. **Lua スクリプト連携**: mod_lua では、Apache2.Request userdata を介して Lua 側から request_rec のフィールドが参照・操作され、r->pool に依存したメモリ管理が行われます。 4. **ログ記録**: リクエストの終了フェーズ（log_transaction フック等）において、mod_log_config などのログハンドラに渡され、ログ行の構築に必要な情報が抽出されます。 5. **解放**: リクエスト終了時に、関連付けられたメモリプール（apr_pool_t）とともに自動的に解放されます。

重要な不変条件

  • **メモリ管理の不変条件**: 構造体内の多くの文字列やデータは、リクエスト専用のメモリプール（apr_pool_t / r->pool）に割り当てられており、リクエスト終了時に自動的に一括解放されます。手動での解放は不要であり、Lua 側でのオブジェクト寿命管理と混同してはなりません。
  • **生存期間の不変条件**: リクエストの処理中、常に有効なポインタとして存在し、有効なメモリプールに関連付けられている必要があります。
  • **アクセス規約**: 非常に巨大な構造体であり、複数のモジュール間で共有されるため、各モジュールは自身のコンテキスト領域（request_config）や許可されたヘッダーテーブル（headers_in, headers_out 等）以外を不用意に変更してはなりません。
  • **検証**: mod_lua においては、Lua スタック上の userdata が有効な request_rec であることが ap_lua_check_request_rec により検証されなければなりません。

よくある落とし穴

  • **NULL ポインタ参照**: リクエスト処理の初期段階や異常終了時、またはリクエスト終了後に NULL ポインタや解放済みの領域にアクセスすると、セグメンテーション違反を引き起こします。
  • **バイナリ互換性の崩壊**: request_rec はバージョンアップ（ap_mmn.h に記録される Module Magic Number の変更）に伴い、メンバーの追加・削除・レイアウト変更が行われます。そのため、モジュールをビルドする際は、対象となる Apache バージョンのヘッダー定義と完全に一致させる必要があります。
  • **競合と実行順序依存**: 複数のモジュールが同一のメンバー（例：ヘッダーテーブル）を更新しようとする場合、フックの優先度（実行順序）に依存した予期せぬ動作や競合が発生する可能性があります。
  • **フィールド設定の制限**: mod_lua の req_newindex などを介したフィールド設定において、すべてのフィールドが書き込み可能ではないため、書き込み制限に注意する必要があります。
