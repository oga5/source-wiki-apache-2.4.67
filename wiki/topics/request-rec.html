request_rec

概念

request_rec は、Apache HTTP Server における単一の HTTP リクエストのライフサイクル全体を管理・追跡するための中心的なデータ構造（struct）です。リクエストの状態、ヘッダー、URI、メソッド、フィルタチェーン、メモリプールなどの情報を保持・共有するために使用されます。

なぜ重要か

Apache HTTP Server のリクエスト処理パイプライン（翻訳、認証、アクセス制御、コンテンツ生成、ログ記録など）において、各フェーズのモジュール間でリクエスト固有のコンテキスト情報を共有・伝達するための共通インターフェースとして機能するためです。また、mod_lua などの拡張モジュールを介してスクリプト言語からリクエスト情報を操作する際のブリッジとしても機能します。

エンドツーエンドの流れ

提供された情報からは、リクエストの生成から破棄に至る完全なエンドツーエンドの制御フロー（特に create_request_rec から各フェーズを経て解放されるまでの詳細な順序や、全モジュールを横断する正確なコールスタック）は完全には明らかになっていません。

判明している部分的な流れは以下の通りです：

1. **リクエストの発生・生成**: create_request_rec（mod_proxy_hcheck.c）などにより生成、またはコアで生成され、リクエスト専用のメモリプール（apr_pool_t）が割り当てられます。 2. **各フェーズでの処理と伝播**:

  • 翻訳、認証、書き換え（mod_rewrite）、ヘッダー修正（mod_headers の do_headers_fixup）などの各フェーズで、request_rec ポインタが引数として受け渡されます。
  • 各モジュールは request_rec 内のメンバ（headers_in, headers_out, subprocess_env など）を参照・更新し、per_dir_config や notes を通じて情報を共有します。
  • 式評価エンジン（ap_expr_exec など）にコンテキストとして渡されます。

    • 3. **ログ記録と終了**:

  • リクエストの終了フェーズ（log_transaction フック等）において、mod_log_config などのログモジュールが request_rec から情報を抽出してログを記録します。
  • リクエスト終了時に、紐付くメモリプール（apr_pool_t）が解放され、構造体内の動的メモリも自動的に解放されます。

重要な不変条件

  • **メモリ管理のライフサイクル**: request_rec 内の多くの動的データは、リクエスト専用のメモリプール（apr_pool_t）に割り当てられており、リクエスト終了時に自動的に解放されます。手動での解放は不要です。
  • **有効なポインタの維持**: リクエスト処理中、およびログ記録などの終了処理が完了するまでは、常に有効なポインタとして存在しなければなりません。
  • **モジュール間共有の規約**: 非常に巨大な構造体であり、複数モジュール間で共有されるため、各モジュールは自身のコンテキスト領域（request_config）や許可されたフィールド以外を不用意に変更してはなりません。

よくある落とし穴

  • **NULL ポインタ参照**: リクエスト処理の初期段階や異常終了時、または適切に初期化されていない状態でアクセスすると、セグメンテーション違反を引き起こします。
  • **バイナリ互換性の破壊**: request_rec の構造体レイアウトは複雑であり、Apache のバージョンアップ（ap_mmn.h の Module Magic Number の変更）に伴いメンバーの追加・削除が行われます。直接メンバーにアクセスするコードは、バージョン間のバイナリ互換性に注意が必要です。
  • **タイミング依存の未初期化アクセス**: フックの実行順序（例: early 処理と通常のフィルタ処理）によっては、特定のフィールドがまだ初期化されていない、あるいは既に解放されている場合があります。
  • **書き込み制限**: mod_lua などを介してフィールドを設定する際、すべてのフィールドが書き込み可能ではありません（例: req_newindex による制限）。
