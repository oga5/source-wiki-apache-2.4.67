request_rec

概念

request_rec は、Apache HTTP Server において単一の HTTP リクエストのライフサイクル全体を管理・追跡するための中心的なデータ構造（struct）です。リクエストの URI、メソッド、ステータス、ヘッダー（headers_in, headers_out, err_headers_out）、フィルタチェーン、メモリプール（apr_pool_t）などのメタデータを保持し、リクエスト処理の各フェーズをまたいで情報を共有・伝達する役割を担います。

なぜ重要か

Apache のアーキテクチャはモジュール式であり、リクエスト処理の各段階で異なるモジュールが動作します。request_rec は、これらのモジュール間で一貫したリクエスト状態を共有するための共通コンテキストを提供するため極めて重要です。

  • **モジュール間連携**: mod_rewrite による URI 書き換え、mod_headers によるヘッダー操作、mod_lua によるスクリプトからのリクエスト制御、mod_log_config によるログ記録など、すべての主要機能がこの構造体を参照・更新することで成り立っています。
  • **メモリ管理の統合**: 構造体内に紐付けられたメモリプール（apr_pool_t）により、リクエスト処理中に割り当てられたメモリのライフサイクルが自動管理され、メモリリークを防ぎます。

エンドツーエンドの流れ

ネットワークソケットから request_rec が生成され、最終的に破棄されるまでの完全なエンドツーエンドのライフサイクルは、提供されたモジュール情報からは完全には明らかになっていません。

現時点で判明している部分的な流れは以下の通りです： 1. **生成**: mod_proxy_hcheck の create_request_rec などの関数により、必要に応じて動的に生成されるか、あるいはコアのリクエスト処理エンジンによって生成されます。 2. **フェーズごとの処理と伝播**:

  • **書き換えフェーズ**: mod_rewrite が request_rec の URI や環境変数を評価・変更します。
  • **ヘッダー操作フェーズ**: mod_headers の do_headers_fixup が headers_in や headers_out を直接操作します。
  • **Lua スクリプト実行**: mod_lua を通じて、Lua 側から Apache2.Request として request_rec のフィールドが読み書きされます。

    • 3. **ログ記録と終了**: mod_log_config が request_rec から情報を抽出してログを出力し、リクエスト終了時に紐付けられたメモリプール（apr_pool_t）が解放されます。

重要な不変条件

  • **メモリ管理の不変条件**: request_rec 内の動的メモリ割り当ては、常に構造体に紐付けられたメモリプール（r->pool）を使用しなければなりません。これにより、リクエスト終了時に手動解放なしで自動的にメモリが解放されます。
  • **生存期間の不変条件**: リクエスト処理中、request_rec へのポインタは常に有効でなければならず、リクエスト終了（ログ記録など）まで生存している必要があります。
  • **アクセス規約**: 巨大な構造体であるため、各モジュールは自身のコンテキスト領域（request_config）以外を不用意に変更してはならず、競合を避ける必要があります。

よくある落とし穴

  • **NULL ポインタ参照**: リクエスト処理の初期段階や異常終了時、あるいは適切に初期化されていない状態で request_rec にアクセスすると、セグメンテーション違反を引き起こします。
  • **バイナリ互換性の崩壊**: request_rec の構造体レイアウトは Apache のバージョンアップ（ap_mmn.h の更新）に伴い変更されることがあります。メンバーの追加や削除があるため、モジュール開発時はバイナリ互換性に注意が必要です。
  • **不適切なフィールド変更**: 複数のモジュールが同一のメンバー（URI やヘッダーなど）を競合して変更すると、フックの実行順序に依存した予期せぬ動作が発生します。
  • **Lua での寿命管理の混同**: mod_lua において、Lua 側のオブジェクト寿命管理と C 側の r->pool によるメモリ管理を混同しないようにする必要があります。
