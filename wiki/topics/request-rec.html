request_rec

概念

request_rec は、Apache HTTP Server における単一の HTTP リクエストのライフサイクル全体を管理・追跡するための中核的なデータ構造（struct）です。リクエストの状態、URI、メソッド、ヘッダー、フィルタチェーン、メモリプールなどの情報を保持し、リクエスト処理の各フェーズ（翻訳、認証、アクセス制御、コンテンツ生成、ログ記録など）を通じてモジュール間で共有・更新されます。

なぜ重要か

Apache HTTP Server のアーキテクチャにおいて、リクエスト処理のほぼすべてのフェーズがこの構造体を中心に展開されるためです。

  • **モジュール間連携**: 各モジュール（mod_rewrite、mod_headers、mod_lua など）は、request_rec を介してリクエストの状態を読み取り、書き換え、あるいはレスポンスヘッダーを操作します。
  • **メモリ管理の統合**: リクエスト専用のメモリプール（apr_pool_t）と紐付いており、リクエスト終了時に自動的にメモリが解放される仕組みを提供することで、メモリリークを防ぎます。
  • **拡張性**: per_dir_config や notes、request_config などの領域を通じて、モジュール固有のコンテキストデータをリクエストに紐付けることができます。

エンドツーエンドの流れ

リクエスト処理の完全なエンドツーエンドの流れは、提供された情報からはすべて明らかになっていません。 ただし、関与するモジュールの情報から、以下のようなライフサイクルを辿ることが確認できます：

1. **リクエストの生成**: クライアントからの接続（conn_rec）に基づき、リクエスト専用のメモリプール（apr_pool_t）を伴って request_rec が生成されます（mod_proxy_hcheck では create_request_rec が使用されます）。 2. **フェーズごとの処理と伝播**:

  • **翻訳・書き換え**: mod_rewrite 等がリクエストの URI や環境変数を評価・書き換えます。
  • **メタデータ操作**: mod_headers（do_headers_fixup）が headers_in、headers_out、err_headers_out を操作し、ヘッダーの追加・削除を行います。
  • **スクリプト実行**: mod_lua では、Lua スクリプトから Apache2.Request を介して request_rec のフィールドが読み書きされます。

    • 3. **ログ記録**: リクエスト終了時（log_transaction フック等）に、mod_log_config などのログモジュールが request_rec から必要な情報を抽出してログを出力します。 4. **破棄**: リクエスト終了に伴い、紐付くメモリプール（r->pool）が解放され、request_rec 自体も自動的に解放されます。

重要な不変条件

  • **メモリプールの生存期間**: request_rec 内の動的メモリ確保は、すべて当該リクエストに紐付くプール（r->pool）を使用しなければなりません。このプールはリクエスト終了時に自動的に解放されるため、手動での解放は不要であり、かつリクエスト終了後にプール内のデータにアクセスしてはなりません。
  • **ポインタの有効性**: リクエスト処理中、request_rec へのポインタは常に有効でなければなりません。
  • **モジュール境界の尊重**: 非常に巨大な構造体であり、複数のモジュール間で共有されるため、各モジュールは自身のコンテキスト領域（request_config など）以外を不用意に変更してはなりません。

よくある落とし穴

  • **NULL ポインタ参照**: リクエスト処理の初期段階や異常終了時、あるいは適切に初期化されていない状態で request_rec ポインタにアクセスすると、セグメンテーション違反（セグフォ）を引き起こします。
  • **バイナリ互換性（MMN）の欠如**: request_rec の構造体レイアウトは Apache のバージョンアップに伴い変更されることがあります（例: メンバの追加・削除）。モジュールをビルドする際は、include/ap_mmn.h（Module Magic Number）の変更履歴に注意し、バイナリ互換性の崩壊に備える必要があります。
  • **タイミング依存の未初期化アクセス**: フックの実行順序（例: early 処理と通常のフィルタ処理）によっては、特定のフィールドがまだ初期化されていない、あるいは既に解放されている場合があります。
  • **Lua 側での寿命管理の混同**: mod_lua において、メモリ管理は完全に r->pool に依存しているため、Lua 側のガベージコレクションによるオブジェクト寿命管理と混同してはなりません。また、Lua からの書き込み（req_newindex）はすべてのフィールドで許可されているわけではありません。
