cmds

概念

cmds は、Apache HTTP Server (httpd) の各モジュールにおいて、設定ファイル（httpd.conf など）からカスタム設定ディレクティブを読み込み、パースするための command_rec 構造体の配列です。各モジュールは、自身が提供する設定ディレクティブの名前、引数の数、適用範囲、および対応する設定コールバック関数をこの配列内に定義します。

なぜ重要か

Apache HTTP Serverは高度にモジュール化されたアーキテクチャを採用しています。各モジュールが独自の機能（圧縮、文字コード変換、プロキシ、ロードバランシングなど）を制御するための設定項目を提供するには、この cmds 配列を介してApacheコアにディレクティブを登録する必要があります。これにより、サーバー起動時に設定ファイルを正しくパースし、各モジュールの動作を動的にカスタマイズすることが可能になります。

関与するモジュール

本トピックには、以下のモジュールが関与しています。

  • module:modules/filters (コンテンツのフィルタリングや変換を担当)
  • module:modules/proxy (プロキシ機能全般を担当)
  • module:modules/proxy/balancers (プロキシのロードバランシング手法を担当)

主要エンティティ

各モジュールファイル内で定義されている cmds 配列（command_rec 構造体配列）は以下の通りです。

  • symbol:modules/filters/mod_brotli.c#cmds: Brotli圧縮フィルターに関する設定ディレクティブ（圧縮レベルやバッファサイズなど）を定義します。
  • symbol:modules/filters/mod_charset_lite.c#cmds: 応答やリクエストの文字コード変換に関する設定ディレクティブを定義します。
  • symbol:modules/filters/mod_ext_filter.c#cmds: 外部プログラムをフィルターとして実行するための設定ディレクティブを定義します。
  • symbol:modules/proxy/balancers/mod_lbmethod_heartbeat.c#cmds: ハートビート（生存監視）に基づくロードバランシング手法の設定ディレクティブを定義します。
  • symbol:modules/proxy/mod_proxy_connect.c#cmds: CONNECTメソッドを用いたプロキシ接続に関する設定ディレクティブを定義します。

エンドツーエンドの流れ

1. **モジュールのロードと登録**: サーバー起動時、Apacheコアはロードされた各モジュールの module 構造体を参照します。この構造体の中に cmds 配列へのポインタが含まれています。 2. **ディレクティブの登録**: Apacheコアの設定パーサーは、各モジュールの cmds 配列を走査し、サポートされているディレクティブの一覧を内部テーブルに登録します。 3. **設定ファイルのパース**: httpd.conf などの設定ファイルを読み込む際、登録されたディレクティブに遭遇すると、cmds で指定されたコールバック関数（設定ハンドラ）が呼び出されます。 4. **設定値の格納**: コールバック関数は、パースされた引数をモジュール固有の設定構造体（ディレクトリ設定やサーバー設定）に格納します。 5. **リクエスト処理への適用**: 実際のリクエスト処理フェーズにおいて、モジュールは格納された設定値を参照して動作（フィルタリングの適用やプロキシ先の決定など）を変更します。

*注意: 提供されたソースコードのコンテキストからは、Apacheコアによるパース処理の具体的な内部関数や、詳細なコールバック呼び出しの低レイヤな制御フローの全容は直接示されていません。*

重要な不変条件

  • **センチネル（終端要素）の配置**: cmds 配列は、Apacheコアが配列の終端を正しく検知できるように、必ず NULL またはゼロクリアされた要素（{NULL}）で終端されていなければなりません。
  • **引数指定（args_how）とコールバックの整合性**: AP_INIT_TAKE1 や AP_INIT_FLAG などのマクロで指定する引数のパース方法と、実際にバインドされるコールバック関数の引数シグネチャは厳密に一致している必要があります。

よくある落とし穴

  • **終端マーカーの欠落**: cmds 配列の末尾に {NULL} を置き忘れると、Apacheコアがメモリの範囲外までディレクティブ定義を読み込み続け、起動時にセグメンテーションフォールト（クラッシュ）を引き起こします。
  • **コールバック関数の型不一致**: ディレクティブの引数タイプ（TAKE1, TAKE2, FLAG など）と、コールバック関数が期待する引数の数や型が一致していない場合、実行時に不正なメモリ領域を指すポインタが渡され、予期しない動作やクラッシュの原因になります。
