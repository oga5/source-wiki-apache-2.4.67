check_string

概念

check_string は、Apache HTTP Server の設定解析フェーズにおいて、指定された設定値（文字列）が不正な文字を含んでいないか、または空でないかを検証するためのバリデーション関数です。主にフォーム認証（mod_auth_form）やセッション管理（mod_session_cookie, mod_session_dbd）のモジュールにおいて、クッキーやフォーム処理の仕様上、問題を引き起こす可能性のある特定の禁止文字（=, &, ; など）を検出するために使用されます。

なぜ重要か

この関数は、設定ディレクティブに不適切な文字列（例えば、クッキー名やセッション設定において区切り文字として機能してしまう = や &、; など）が指定されるのを防ぐために重要です。これにより、設定のパースエラーや、実行時におけるセッション・クッキー処理の予期しない挙動やセキュリティ上の問題を未然に防ぐことができます。

関与するモジュール

  • module:modules/aaa
  • module:modules/session

主要エンティティ

  • symbol:modules/aaa/mod_auth_form.c#check_string: フォーム認証の設定（location, username, password, method, mimetype, body など）において、空文字列や =、& の有無を検証します。
  • symbol:modules/session/mod_session_cookie.c#check_string: クッキー名などの設定値において、空文字列や =、& の有無を検証します。
  • symbol:modules/session/mod_session_dbd.c#check_string: DBDセッション設定において、ap_cookie_check_string を用いてクッキー仕様上禁止されている文字（=, ;, &）の有無を検証します。

エンドツーエンドの流れ

1. **設定のロードと解析**: Apache の起動時または再起動時に、各モジュールの設定ディレクティブ（例: mod_auth_form の各種設定、mod_session_cookie の set_cookie_name 等、mod_session_dbd の設定）が読み込まれます。 2. **検証の実行**: 各ディレクティブの解析処理から check_string が呼び出されます。

  • mod_auth_form.c および mod_session_cookie.c では、ap_strchr_c を使用して直接 = や & を検索します。
  • mod_session_dbd.c では、ap_cookie_check_string を呼び出して =, ;, & を検証します。

    • 3. **結果の返却**:

  • 検証を通過した場合、NULL を返して処理を継続します。
  • 検証に失敗した場合、cmd->pool を用いてエラーメッセージを構築して返し、設定エラーとして処理を中断します。

*注意: 提供された情報からは、これら3つの check_string 関数が完全に独立した static 関数として動作しているのか、あるいはモジュール間で直接呼び出し合う具体的なバインディングが存在するのかといった、詳細なエンドツーエンドの相互作用フローの全容は完全には明らかになっていません。*

重要な不変条件

  • **禁止文字の排除**: 検証対象の文字列に、仕様上禁止されている文字（= および &、mod_session_dbd の場合はさらに ;）が含まれていてはなりません。
  • **非空文字列**: 検証対象の文字列は NULL または空文字列であってはなりません（mod_auth_form.c および mod_session_cookie.c における不変条件）。
  • **メモリプールの有効性**: エラーメッセージ生成のために、cmd->pool が有効なメモリプールとして割り当てられている必要があります。

よくある落とし穴

  • **モジュール間の仕様差**: mod_session_dbd の check_string は ap_cookie_check_string を使用して ; も禁止文字として検証しますが、他の2つの実装では主に = と & のみを直接検証しています。モジュール間で検証ルールに僅かな差異があるため、共通のバリデーターとして扱う際には注意が必要です。
  • **静的スコープの混同**: 各ファイルで static 関数として定義されているため、一見すると共通ライブラリ関数のようでありながら、実際にはファイルごとに個別の実装が存在します。一方の挙動を変更しても、他方に自動的には反映されません。
