check_string

概念

check_string は、Apache HTTP Server（httpd-2.4.67）の複数のモジュールにおいて、設定ディレクティブで指定された文字列（Cookie名やフォーム認証の設定値など）の妥当性を検証するために定義されているバリデーション関数です。主に、文字列が空でないこと、およびクッキーやセッション処理において禁止されている特定の文字（=, &, ; など）を含んでいないことを検証します。

なぜ重要か

設定値に = や & などの文字が含まれていると、クッキーやフォームデータのパース処理において予期しない動作やセキュリティ上の問題を引き起こす可能性があります。check_string は、設定のロード（解析）段階でこれらの不適切な文字列を検出し、エラーメッセージを返すことで、実行時の不具合を未然に防ぐ役割を担っています。

関与するモジュール

  • module:modules/aaa
  • module:modules/session

主要エンティティ

  • **symbol:modules/aaa/mod_auth_form.c#check_string**
  • フォーム認証の設定（location, username, password, method, mimetype, body）の解析時に呼び出されます。= および & の有無を検証します。
  • **symbol:modules/session/mod_session_cookie.c#check_string**
  • クッキー設定（set_cookie_name, set_cookie_name2 等）の検証時に呼び出されます。= および & の有無を検証します。
  • **symbol:modules/session/mod_session_dbd.c#check_string**
  • DBDセッション設定の検証時に呼び出されます。ap_cookie_check_string を利用して =, ;, & の有無を検証します。

エンドツーエンドの流れ

設定ファイルのロード時（ディレクティブ解析時）に、各モジュールの設定ハンドラから check_string が呼び出されます。

1. **入力の受け取り**: 設定ディレクティブから渡された文字列（string）と設定パラメータ（cmd）を引数として check_string が実行されます。 2. **文字検証**:

  • mod_auth_form.c および mod_session_cookie.c では、ap_strchr_c を使用して直接 = または & が含まれているかを検索します。
  • mod_session_dbd.c では、外部関数 ap_cookie_check_string を呼び出して =, ;, & が含まれていないかを確認します。

    • 3. **結果の返却**:

  • 検証に失敗した場合、cmd->pool を用いてエラーメッセージを構築（apr_pstrcat を使用）して返します。
  • 検証を通過した場合は NULL を返します。

※ **未検証の事項**: 提供された情報では、各モジュールの check_string が互いに他のモジュールからも呼び出されている旨が記載されていますが、これらは static 関数として定義されています。これらが実際にどのようにモジュール境界を越えて共有・呼び出されているのか、あるいは単に同名の独立したヘルパー関数として機能しているのかの正確な結合関係や詳細な実行フローは完全には判明していません。

重要な不変条件

  • **メモリプールの有効性**: 呼び出し時に cmd->pool が有効なメモリプールとして割り当てられており、エラーメッセージ生成時に参照可能である必要があります。
  • **禁止文字の排除**:
  • mod_auth_form および mod_session_cookie においては、検証対象の文字列に = および & が含まれていないこと。
  • mod_session_dbd においては、検証対象の文字列に =, ;, & が含まれていないこと。
  • **戻り値のセマンティクス**: 検証成功時は NULL を返し、失敗時は非 NULL のエラーメッセージ文字列を返さなければなりません。

よくある落とし穴

  • **モジュール間での検証ルールの不一致**: mod_session_dbd の check_string は ; も禁止文字として検証しますが、他の2つのモジュールでは = と & のみを検証します。モジュール間で禁止文字の基準が微妙に異なるため、共通の文字列を扱う際に混乱が生じる可能性があります。
  • **静的定義による重複**: 各ファイルで static const char *check_string として個別に定義されているため、一箇所のバリデーションロジックを変更しても他のモジュールには自動的に反映されません。保守時にはこれら複数のファイルに定義された同名関数の影響範囲を考慮する必要があります。
