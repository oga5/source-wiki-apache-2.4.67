WebDAV Repository and Locking Engine

概念

WebDAV Repository and Locking Engine は、Apache HTTP Server（mod_dav）において、WebDAVリソースのプロパティ（メタデータ）管理および排他制御（ロック）を司る抽象化レイヤーとバックエンドプロバイダーの仕組みです。

  • **プロパティ管理**: dav_propdb 構造体や dav_open_propdb 関数を介して、WebDAVリソースに紐づくプロパティデータベースの操作を抽象化します。
  • **ロック管理**: dav_lockdb_private や dav_lock_private などの不透明（opaque）なデータ構造を用いて、バックエンド固有のロック情報を隠蔽し、一貫したロック制御を提供します。

なぜ重要か

WebDAVプロトコル（RFC 4918等）では、リソースのプロパティ設定・取得（PROPFIND/PROPPATCH）や、複数クライアント間での編集競合を防ぐためのロック制御（LOCK/UNLOCK）がコア機能として定義されています。

mod_dav は、これらプロパティやロックの具体的な永続化ロジックをコアから分離し、プロバイダー（ファイルシステムをバックエンドとする mod_dav_fs など）が独自の実装を提供できるように抽象化しています。これにより、ストレージの実装に依存しない、柔軟で一貫したWebDAVプロトコル処理が可能になります。

主要エンティティ

  • **symbol:modules/dav/fs/lock.c#dav_fs_open_lockdb**: ファイルシステムバックエンド（mod_dav_fs）において、ロックデータベースを開くための関数。
  • **symbol:modules/dav/main/mod_dav.h#dav_lock_private**: ロックデータベースの実装において、リポジトリ固有のロック情報を保持するための不透明なデータ構造。dav_lock 構造体の info メンバーとして保持されます。
  • **symbol:modules/dav/main/mod_dav.h#dav_lockdb_private**: ロックデータベースプロバイダー固有のプライベートデータを保持するための不透明な構造体。
  • **symbol:modules/dav/main/mod_dav.h#dav_propdb**: WebDAVプロパティデータベースへのアクセスおよび操作を管理するためのコンテキスト構造体。
  • **symbol:modules/dav/main/props.c#dav_open_propdb**: WebDAVプロパティデータベース（dav_propdb）をオープンし、初期化するための関数。

エンドツーエンドの流れ

提供された情報からは、リクエスト受信からプロパティ・ロックデータベースの操作、レスポンス返却に至る完全なエンドツーエンドの制御フローはまだ十分に判明していません。

現時点で判明している部分的な流れは以下の通りです：

1. **プロパティ操作の開始**: dav_method_proppatch や dav_gen_supported_live_props などの呼び出し元から dav_open_propdb が呼び出され、内部で dav_popen_propdb を介してプロパティデータベース（dav_propdb）がオープンされます。 2. **ロックデータベースのオープン**: ファイルシステムバックエンドでは、dav_fs_open_lockdb が呼び出され、内部で dav_fs_really_open_lockdb や dav_get_lockdb_path を用いてロックデータベースが開かれます。

重要な不変条件

提供された情報からは、システム全体の詳細な不変条件は完全には判明していません。現時点で判明している局所的な不変条件は以下の通りです：

  • **メモリ管理のライフサイクル**: dav_lock_private や dav_propdb などの構造体のメモリ管理は、関連付けられたApacheのリクエストプール（apr_pool_t）の生存期間に依存します。
  • **プロパティ操作のクローズ**: dav_open_propdb によってオープンされた dav_propdb は、処理の終了時に dav_close_propdb を通じて明示的にクローズされ、リソースが解放されなければなりません。
  • **不透明ポインタの整合性**: dav_lock_private や dav_lockdb_private は前方宣言のみの不透明ポインタであるため、これらを扱う各プロバイダーの実装側で型安全性を担保する必要があります。

よくある落とし穴

  • **プロバイダー間の型安全性の欠如**: dav_lock_private や dav_lockdb_private は具体的なメンバーが公開されていないため、異なるプロバイダーの実装間で誤ってキャストして使用すると、未定義の動作やメモリ破損を引き起こす可能性があります。
  • **リソースリーク**: プロパティデータベースのオープン後、エラーハンドリングの経路などで dav_close_propdb による明示的なクローズを怠ると、リソースリークに繋がる恐れがあります。
  • **プロバイダー未定義時のエラー**: プロパティデータベースのオープン時（dav_open_propdb）に、対応するプロバイダーが未定義であったり、データベースへの接続に失敗したりすると、後続のプロパティ操作はすべて実行できなくなります。
