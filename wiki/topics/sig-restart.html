sig_restart

概念

sig_restart は、Apache HTTP Server においてサーバーの再起動（通常の再起動およびグレースフル再起動）を要求するシグナルを受信した際に呼び出されるシグナルハンドラ関数です。プラットフォームやMPM（Multi-Processing Module）の実装（Unix系およびOS/2）に応じて、それぞれのソースファイルで定義されています。

なぜ重要か

サーバーの稼働中に設定の再読み込みやプロセスの再起動を安全に行うために不可欠です。この関数は、受信したシグナル（SIGHUP、SIGUSR1、AP_SIG_GRACEFULなど）に応じて、グレースフル（優雅な）再起動か、あるいは強制的な（非優雅な）再起動かを判別し、関連するグローバルな状態フラグを適切に更新することで、メインループに対して再起動処理の開始を促します。

関与するモジュール

主要エンティティ

  • symbol:os/unix/unixd.c#sig_restart: Unix環境における再起動シグナルハンドラ。共有メモリ領域 retained_data（ap_unixd_mpm_retained_data 構造体）の状態を更新します。
  • symbol:server/mpm/mpmt_os2/mpmt_os2.c#sig_restart: OS/2環境における再起動シグナルハンドラ。グローバル変数 restart_pending および is_graceful を更新します。

エンドツーエンドの流れ

1. **シグナルの登録**: サーバー起動時または初期化時に、apr_signal 等を用いて sig_restart が特定の再起動シグナル（SIGHUP、SIGUSR1、AP_SIG_GRACEFULなど）のハンドラとして登録されます。 2. **シグナルの受信**: 外部からプロセスへ再起動シグナルが送信されると、sig_restart がトリガーされます。 3. **状態の更新**:

  • **Unix環境**: retained_data が存在することを確認し、mpm_state を AP_MPMQ_STOPPING に設定します。重複処理を避けるチェックを行った後、restart_pending を 1 に設定します。シグナルが AP_SIG_GRACEFUL でない場合は、is_ungraceful を 1 に設定します。
  • **OS/2環境**: シグナルが SIGUSR1 の場合は is_graceful を 1 に設定し、restart_pending を 1 に設定します。

    • 4. **メインループでの検知と処理**:

  • シグナルハンドラ自体はフラグの更新のみを行い、実際の再起動処理はメインループ側でこれらのフラグを検知して実行されます。ただし、メインループがこれらのフラグをどのように監視し、最終的にプロセスを再起動するかの詳細なエンドツーエンドのフローは、提供された情報からは未だ完全には判明していません。

重要な不変条件

  • **非同期シグナル安全性**: 本関数はシグナルハンドラとして動作するため、非同期シグナル安全な操作のみを行う必要があります。
  • **Unix環境における前提条件**: retained_data（ap_unixd_mpm_retained_data 構造体）が初期化されている必要があります。初期化されていない（NULLである）場合、関数は何もせずに即座に終了します。
  • **OS/2環境における前提条件**: restart_pending および is_graceful グローバル変数が、本関数が呼び出される前に適切に初期化されている必要があります。

よくある落とし穴

  • **重複シグナルの処理**: すでに再起動が保留されている状態でさらにシグナルを受信した場合、Unix環境の実装では状態の上書きを避けるためのガードロジックが存在しますが、OS/2環境の実装では単純にフラグが再設定されるため、プラットフォーム間での挙動の差異に注意が必要です。
  • **非同期安全性**: シグナルハンドラ内で複雑な処理や非シグナル安全な関数を呼び出すと、デッドロックや未定義動作を引き起こす可能性があります。本関数はフラグの更新に特化していますが、保守の際に不要な処理を追加しないよう注意する必要があります。
