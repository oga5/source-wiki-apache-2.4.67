ACME Protocol and Managed Domains (mod_md)

概念

mod_md（Managed Domains）は、Apache HTTP ServerにおいてSSL/TLS証明書の自動管理を実現するためのモジュールです。RFC 8555で定義される **ACME（Automated Certificate Management Environment）プロトコル** を使用して、Let's Encrypt などの認証局（CA）から証明書を自動的に取得、更新、および管理します。

この機能は主に以下の3つのソースファイルによって支えられています。

  • **mod_md.c**: Apache HTTP Serverとの統合、設定ディレクティブの処理、およびライフサイクル管理。
  • **md_acme.c**: ACMEプロトコルに準拠したクライアント機能（アカウント登録、注文、チャレンジの解決など）。
  • **md_store.c**: 取得した証明書、秘密鍵、およびアカウント情報のローカルストレージへの永続化。

なぜ重要か

従来のSSL/TLS証明書の管理は、手動での申請、検証、配置、およびWebサーバーの再起動が必要であり、ヒューマンエラーや更新忘れによるサービス停止のリスクが常に存在していました。

mod_md を導入することで、Webサーバー自体が証明書の有効期限を監視し、ACMEプロトコルを通じてバックグラウンドで自動的に証明書を更新します。これにより、運用の安全性が向上し、管理コストが大幅に削減されます。

関与するモジュール

  • **module:modules/md**
  • modules/md/mod_md.c
  • modules/md/md_acme.c
  • modules/md/md_store.c

主要エンティティ

  • **Managed Domain (MD)**: mod_md.c で定義および管理される、自動証明書管理の対象となるドメイン（およびそのサブドメイン）のグループ。
  • **ACME Client (md_acme.c)**: ACMEサーバーと通信し、ディレクトリの取得、アカウントの作成、証明書の署名要求（CSR）の送信、およびチャレンジ（HTTP-01など）の検証を行うエンティティ。
  • **Store (md_store.c)**: 証明書、秘密鍵、およびACMEアカウントのメタデータをファイルシステム等に安全に保存・ロードするストレージ管理機構。

エンドツーエンドの流れ

提供されたソースコードのメタデータからは、詳細な関数呼び出しシーケンスや完全な状態遷移は完全には特定されていませんが、一般的な処理フローは以下の通りです。

1. **設定のパースと初期化 (mod_md.c)**: Apacheの起動時に設定ファイルから ManagedDomain 定義を読み込み、対象ドメインを登録します。 2. **ストレージの確認 (md_store.c)**: 既存の証明書や秘密鍵がローカルストレージに存在するか、またその有効期限が切れていないかを確認します。 3. **ACMEプロトコルの実行 (md_acme.c)**: 証明書の新規取得または更新が必要な場合、ACMEサーバーに接続してアカウントを登録（またはロード）し、新しい証明書の「注文（Order）」を作成します。 4. **チャレンジの解決**: CAから提示されたチャレンジ（例: HTTP-01）を解決するため、一時的な検証用リソースを配置し、CAによる検証をパスします。 5. **証明書の取得と保存 (md_store.c)**: 検証成功後、CSRを送信して発行された証明書をダウンロードし、ローカルストレージに安全に保存します。

*注意: 内部的なスレッドモデルや、Apacheの親プロセス・子プロセス間での具体的な同期フローの詳細は、現時点では未確定です。*

重要な不変条件

  • **秘密鍵の秘匿性**: md_store.c を介して保存される秘密鍵は、適切なファイルパーミッションで保護され、外部に漏洩してはなりません。
  • **ACMEチャレンジの到達性**: HTTP-01 チャレンジを実行する際、外部のCAから対象ドメインの特定のパス（.well-known/acme-challenge/）へのHTTPリクエストが、mod_md によって適切に処理されなければなりません。

*注意: コードレベルでの厳密な不変条件（状態遷移のガード条件やロック要件など）は、提供されたファイル名のみからは完全には特定されていません。*

よくある落とし穴

  • **ストレージの書き込み権限不足**: md_store.c が証明書や鍵を保存するディレクトリ（通常は state ディレクトリ）に対して、Apacheの実行ユーザーが書き込み権限を持っていない場合、更新処理が失敗します。
  • **DNS/ファイアウォールの制限**: ACMEサーバーからのHTTP-01チャレンジ用リクエストが、ファイアウォールやDNSの誤設定によってブロックされると、証明書の発行が拒否されます。
  • **レートリミット**: ACMEプロトコル（特に Let's Encrypt）のテスト時に本番環境のCAエンドポイントを使用すると、短時間での頻繁なリクエストによりレートリミットに達し、一時的に証明書が取得できなくなることがあります。

次に読む場所

  • **modules/md/mod_md.c**: Apacheモジュールとしての初期化処理、ディレクティブ定義、およびリクエストハンドラとの連携を理解するために参照してください。
  • **modules/md/md_acme.c**: ACMEプロトコルの具体的な通信ロジック、JSON Web Signature (JWS) の構築、およびチャレンジ処理を調査する場合に参照してください。
  • **modules/md/md_store.c**: 証明書や鍵がファイルシステム上でどのように永続化され、管理されるかを把握するために参照してください。
