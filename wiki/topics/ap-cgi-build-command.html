ap_cgi_build_command

概念

ap_cgi_build_command は、Apache HTTP Server (httpd) において、プラットフォーム固有のCGIスクリプトや実行ファイルの起動に必要なコマンドラインおよび引数配列（argv）を構築するための関数です。NetWare環境（mod_netware.c）およびWindows環境（mod_win32.c）でそれぞれ個別に実装されており、OSごとの実行特性の違いを吸収する役割を持っています。

なぜ重要か

CGIの実行プロセスはオペレーティングシステムによって大きく異なります。

  • **Windows** では、ファイルの拡張子判別、レジストリ（ScriptInterpreterSource）の検索、ファイル先頭のシバン（#!）行の解析、およびWindows固有のコマンドラインエスケープ処理が必要です。
  • **NetWare** では、拡張子に基づいたNLM（NetWare Loadable Module）ハンドラの検索や、アドレス空間の分離制御（e_info->addrspace）が必要になります。

ap_cgi_build_command は、これらの複雑なプラットフォーム固有ロジックをカプセル化し、mod_cgi などの汎用CGI実行モジュールに対して統一されたインターフェースを提供するために重要です。

エンドツーエンドの流れ

1. **呼び出し**: modules/generators/mod_cgi.c が APR_RETRIEVE_OPTIONAL_FN を介して、プラットフォーム固有の ap_cgi_build_command を動的に取得して呼び出します。 2. **コマンド構築（プラットフォーム別）**:

  • **NetWare環境**:

    • 1. リクエスト設定（netware_dir_config）から拡張子に対応するハンドラを検索します。 2. ハンドラが見つかった場合、[ハンドラ] [ファイル名] の形式でコマンド文字列を生成します。 3. apr_tokenize_to_argv で引数をトークン化し、ap_server_root_relative で絶対パスを解決します。

  • **Windows環境**:

    • 1. ファイル拡張子（.exe, .com, .bat, .cmd）を判定し、標準的な実行形式であればインタープリタを空に設定します。 2. 必要に応じて、レジストリまたはファイル先頭のシバン（#!）行からインタープリタを特定します。 3. split_argv を用いて引数テンプレートを展開・配列化し、環境変数をWindows向けに調整します。 3. **結果の返却**: 構築されたコマンドパス（cmd）、引数配列（argv）、および実行情報（cgi_exec_info_t）を呼び出し元に返します。

※なお、呼び出し元である mod_cgi 側での具体的なプロセス起動処理や、エラーハンドリングの完全なエンドツーエンドの統合フローの詳細は、提供されたコンテキストからは未詳です。

重要な不変条件

  • **メモリ管理の局所性**: すべての動的メモリ確保は、引数 p で渡される apr_pool_t に依存して行われなければなりません。
  • **エラー時の契約**: コマンドの構築やインタープリタの特定に失敗した場合（ハンドラ未登録、ファイル読み込み失敗、実行形式不明など）、関数は APR_EBADF などのエラー用ステータスコードを返し、処理を中断しなければなりません。
  • **オプショナル関数としての公開**: 本関数は APR_REGISTER_OPTIONAL_FN を介して登録され、外部モジュールから動的に呼び出せる状態になっている必要があります。

よくある落とし穴

  • **Windowsにおける文字コード変換の限界**: prep_string によるUTF-8からWindowsローカルコードページへの変換は限定的な実装に留まっており、特定の環境変数以外で文字化けや予期しない挙動が発生する可能性があります。
  • **Windowsの引数エスケープの複雑さ**: split_argv 内の引数解析は複雑なエスケープ処理を伴うため、Windowsのコマンドライン引数ルールに反する変更を加えると、セキュリティ脆弱性や起動失敗を招く恐れがあります。
  • **NetWareのハンドラ未登録**: 拡張子に対応するコマンドが file_type_handlers に登録されていない場合、即座に APR_EBADF エラーとなるため、設定ファイルの不備が直接CGIの起動失敗につながります。
