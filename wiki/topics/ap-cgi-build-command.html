ap_cgi_build_command

概念

ap_cgi_build_command は、Apache HTTP Server（httpd-2.4.67）において、プラットフォーム固有（NetWare および Windows）の環境でCGIスクリプトや実行ファイルを起動するためのコマンドラインおよび引数（argv）を構築する関数です。

なぜ重要か

CGIの実行方法はOSやプラットフォームによって大きく異なります。例えば、NetWareではNLM（NetWare Loadable Module）の実行やアドレス空間の分離が必要であり、Windowsではファイル拡張子の判定、レジストリ検索、シバン（#!）解析、複雑な引数のエスケープ処理が必要となります。ap_cgi_build_command は、これらのプラットフォーム固有の複雑なコマンド構築ロジックをカプセル化し、mod_cgi などのジェネリックなCGI実行モジュールに対して統一されたインターフェースを提供する役割を持っています。

関与するモジュール

  • symbol:modules/arch/netware/mod_netware.c#ap_cgi_build_command: NetWare環境用のコマンド構築関数。
  • symbol:modules/arch/win32/mod_win32.c#ap_cgi_build_command: Windows環境用のコマンド構築関数。

エンドツーエンドの流れ

1. **呼び出しの準備**: modules/generators/mod_cgi.c が APR_RETRIEVE_OPTIONAL_FN を介して、プラットフォーム固有の ap_cgi_build_command 関数を動的に取得します。 2. **関数の実行**:

  • **NetWareの場合**:

    • 1. リクエスト設定から netware_dir_config を取得します。 2. r->filename の拡張子を基に file_type_handlers からハンドラ（NLM）を検索します。 3. ハンドラが見つかった場合、[ハンドラ] [ファイル名] の形式でコマンド文字列を生成し、apr_tokenize_to_argv で引数配列に変換します。 4. ap_server_root_relative で絶対パスを解決します。

  • **Windowsの場合**:

    • 1. ファイル拡張子（.exe, .com, .bat, .cmd）を判定します。 2. 必要に応じて、設定（ScriptInterpreterSource）に基づくレジストリ検索、またはファイル先頭のシバン（#!）行の解析によりインタープリタを特定します。 3. split_argv を用いて引数テンプレートを展開・配列化し、prep_string で環境変数を調整します。 3. **結果の返却**: 構築されたコマンドパス（cmd）と引数配列（argv）を呼び出し元に返し、CGIプロセスが起動されます。

*※なお、mod_cgi 側でこの関数が呼び出された後の具体的なプロセス起動処理や、エラーハンドリングの完全なエンドツーエンドのフローは、提供された情報からは完全には明らかになっていません。*

重要な不変条件

  • **メモリ管理**: すべての動的メモリ確保は、引数 p で渡される apr_pool_t に依存して行われなければなりません。
  • **インターフェースの統一**: NetWare版とWindows版で、全く同じシグネチャ（apr_status_t ap_cgi_build_command(const char **cmd, const char ***argv, request_rec *r, apr_pool_t *p, cgi_exec_info_t *e_info)）を維持し、mod_cgi からオプショナル関数として透過的に呼び出せるようにする必要があります。
  • **NetWare固有の不変条件**: e_info->process_cgi が真の場合、r->filename をベースにコマンドが初期化されます。また、拡張子に対応するハンドラが d->file_type_handlers に登録されている必要があります。
  • **Windows固有の不変条件**: e_info->cmd_type が APR_SHELLCMD 以外でインタープリタが特定された場合、APR_PROGRAM_PATH に設定されます。また、処理成功時には e_info->detached が 1 に設定されます。

よくある落とし穴

  • **ハンドラ/インタープリタの未定義**:
  • NetWareでは、拡張子に対応するコマンドが file_type_handlers に登録されていない場合、APR_EBADF を返して失敗します。
  • Windowsでは、インタープリタが特定できず、かつシバン行も見つからない場合に APR_EBADF を返して失敗します。
  • **Windowsにおける引数エスケープと文字コード**:
  • split_argv 内の引数解析は複雑なエスケープ処理を含むため、変更時にWindowsのコマンドライン引数ルールを逸脱しやすい傾向にあります。
  • prep_string によるUTF-8からWindowsローカルコードページへの変換は限定的な実装に留まっているため、多言語環境での環境変数処理に注意が必要です。
  • **シバン解析の制限**: Windows環境でのシバン解析時、ファイル読み込み（apr_file_open 等）が失敗するとエラーとなります。

次に読む場所

  • modules/arch/netware/mod_netware.c: NetWareにおける具体的なコマンド構築とアドレス空間制御（e_info->addrspace）の実装。
  • modules/arch/win32/mod_win32.c: Windowsにおけるレジストリ検索、シバン解析、引数トークン化（split_argv）の実装。
