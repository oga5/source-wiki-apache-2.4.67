master_main

概念

master_main は、Apache HTTP Server（httpd-2.4.67）の特定のマルチプロセッシングモジュール（MPM）において、親プロセスのメイン処理を担う関数です。OS/2（mpmt_os2）やWindows（winnt）などのプラットフォーム固有のMPMで定義されており、主に子プロセスの生成、監視、およびライフサイクル管理を制御する役割を持ちます。

なぜ重要か

この関数は、サーバーの親プロセスが起動した後に、リスナーソケットの準備、共有メモリやスコアボードなどの共有リソースの初期化、そして必要な数の子プロセスの起動と維持を行うため、サーバー全体のライフサイクルを統制する上で極めて重要です。特に、子プロセスが異常終了した際の検知や再生成、サーバー停止時のクリーンアップ処理を司ります。

エンドツーエンドの流れ

OS/2（mpmt_os2）における master_main の処理フローは以下の通りです。なお、Windows（winnt）における詳細なエンドツーエンドの流れは提供された情報からは未解明です。

1. **初期化とリソース確保**:

  • mpmt_os2_run から呼び出され、シグナルハンドラを設定し、リスナーソケットを準備します。
  • DosAllocSharedMem を使用してリスナーソケット情報を格納する共有メモリ（parent_info_t）を確保し、OS/2 ミューテックス ap_mpm_accept_mutex を作成します。
  • 未初期化の場合は、スコアボード用共有メモリを確保・初期化します。

    • 2. **メインループ（子プロセスの管理）**:

  • 現在の稼働プロセス数をカウントし、目標数（ap_daemons_to_start）に達するまで spawn_child を呼び出して子プロセスを生成します。
  • DosWaitChild を非ブロックで呼び出し、子プロセスの終了を監視します。
  • 子プロセスが終了した場合はスコアボードを更新します。致命的エラー（APEXIT_CHILDFATAL）を検知した場合はループを抜けます。
  • 終了フラグが立つまで apr_sleep で待機しながらループを繰り返します。

    • 3. **終了処理**:

  • 全ての子プロセスにシグナル（SIGTERM または SIGHUP）を送信して終了を促します。
  • 共有メモリ（DosFreeMem）を解放し、終了ステータス（OK または DONE）を返します。

重要な不変条件

OS/2（mpmt_os2）の実装において、以下の不変条件が維持されます。Windows（winnt）における不変条件は未解明です。

  • **前提条件**: ap_setup_listeners によるリスナーソケットのセットアップ、および共有メモリ（parent_info_t やスコアボード）の確保が正常に完了している必要があります。
  • **プロセス数制限**: 生成される子プロセスの数は、ハードコードされた上限値 HARD_SERVER_LIMIT（10）以下に制限されます。
  • **エラー時の即時停止**: 子プロセスが致命的エラー（APEXIT_CHILDFATAL）で終了した場合、親プロセスはメインループを中断し、サーバーを停止します。

よくある落とし穴

  • **移植性の欠如**: master_main（特に mpmt_os2 版）は、DosAllocSharedMem や DosWaitChild などの OS/2 固有の API に強く依存しているため、他のプラットフォームへの移植性はありません。
  • **ハードコードされた制限**: OS/2 環境では HARD_SERVER_LIMIT が 10 に固定されており、これを超えるプロセス数を設定することはできません。
  • **リソース解放のタイミング**: 共有メモリ（DosFreeMem）の解放は、メインループを完全に抜けた後のクリーンアップフェーズでのみ実行されるため、ループ中に異常終了が発生した場合のリークに注意する必要があります。
