master_main

概念

master_main は、Apache HTTP Server (httpd-2.4.67) の特定のマルチプロセッシングモジュール (MPM) において、親プロセス（マスタープロセス）のメイン処理を担う関数です。OS/2 環境（mpmt_os2）および Windows 環境（winnt）のそれぞれで定義されており、子プロセスの生成、監視、およびライフサイクル管理を制御する役割を持ちます。

なぜ重要か

この関数は、サーバー全体の安定稼働とプロセスのライフサイクル管理において中核的な役割を果たします。OS/2 環境では、リスナーソケットやスコアボードなどの共有リソースを初期化し、必要な数の子プロセスを維持・監視します。Windows 環境でも同様に、シャットダウンや再起動のイベントをハンドリングしながら、子プロセスの起動や停止を制御するために重要です。

関与するモジュール

  • symbol:server/mpm/winnt/mpm_winnt.c#master_main: Windows MPM における親プロセスのメイン処理関数。

エンドツーエンドの流れ

OS/2 環境（mpmt_os2）における流れは以下の通りです： 1. **初期化とリソース確保**: mpmt_os2_run から呼び出され、シグナルハンドラの設定、リスナーソケットの準備、共有メモリ（parent_info_t）の確保、および OS/2 ミューテックス（ap_mpm_accept_mutex）の作成を行います。 2. **スコアボード初期化**: スコアボード用の共有メモリを確保・初期化します。 3. **メインループ**:

  • 稼働プロセス数をカウントし、目標数（ap_daemons_to_start）に達するまで spawn_child を用いて子プロセスを生成します。
  • DosWaitChild を非ブロックで呼び出し、子プロセスの終了を監視します。
  • 子プロセスが終了した場合はスコアボードを更新します。
  • 終了フラグが立つまで apr_sleep で待機します。

    • 4. **終了処理**: 全子プロセスにシグナル（SIGTERM または SIGHUP）を送信し、共有メモリを解放して終了します。

Windows 環境（winnt）における詳細なエンドツーエンドの流れ（シャットダウンや再起動イベントの具体的なハンドリング手順など）は、現時点では十分に解析されておらず未解明です。ただし、winnt_run から呼び出され、create_process や winnt_note_child_started などを介して子プロセスの管理を行うことが分かっています。

重要な不変条件

  • **OS/2 環境 (mpmt_os2)**:
  • ap_setup_listeners が成功し、リスナーソケットが利用可能である必要があります。
  • 共有メモリおよびスコアボード用の共有メモリが正常に確保されている必要があります。
  • プロセス数の上限として HARD_SERVER_LIMIT（10）が適用されます。
  • **Windows 環境 (winnt)**:
  • 重要な不変条件は現時点では解析されておらず未解明です。

よくある落とし穴

  • **OS/2 環境 (mpmt_os2)**:
  • OS/2 の Dos 系 API（DosAllocSharedMem, DosCreateMutexSem, DosWaitChild など）に強く依存しているため、他プラットフォームへの移植性はありません。
  • プロセス数の上限（HARD_SERVER_LIMIT = 10）がハードコードされているため、これを超えるスケーラビリティは確保できません。
  • 子プロセスが致命的エラー（APEXIT_CHILDFATAL）で終了した場合、メインループが中断されサーバー全体が停止します。
  • **Windows 環境 (winnt)**:
  • 詳細な失敗モードや落とし穴は現時点では解析されておらず未解明です。
