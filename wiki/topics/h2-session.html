h2_session

概念

h2_session は、HTTP/2 接続におけるセッションのライフサイクル、状態管理、およびストリームの多重化を制御するための中心的なデータ構造です。

この名称を持つ構造体は、システム内の2つの異なるコンテキストで定義されています。 1. **サーバー側 (mod_http2)**: クライアントとの単一の TCP 接続（conn_rec）を抽象化し、プロトコル層としての役割を担う構造体。 2. **クライアント側 (h2ws.c)**: テストツールにおいて、nghttp2 ライブラリを用いた接続・ストリーム管理および I/O 状態追跡を行うための構造体。

なぜ重要か

  • **サーバー側 (mod_http2)**: HTTP/2 接続（C1層）におけるフレームの解析、イベントのディスパッチ、およびストリームの多重化を制御する司令塔として機能します。nghttp2 ライブラリと Apache の接続状態を仲介し、ワーカースレッド（h2_workers）へのリクエストの振り分けやレスポンスの送出を管理するため、モジュール全体の基盤となります。
  • **クライアント側 (h2ws.c)**: テストクライアントにおいて、非ブロッキングソケットを介した HTTP/2 セッションの維持、I/O イベントの監視（poll ループの制御）、およびストリームのライフサイクル管理を行うために不可欠です。

  • module:test/clients (HTTP/2 および WebSocket のテスト用クライアントツール)

エンドツーエンドの流れ

提供された情報に基づく、それぞれのセッション処理の流れは以下の通りです。

サーバー側 (mod_http2)

1. **初期化**: h2_session_create により h2_session が生成され、クライアント接続（conn_rec）に関連付けられます。 2. **イベント処理**: h2_session_process がメインループとして機能し、nghttp2 を介してフレームを読み込み、イベントをディスパッチします。 3. **多重化と処理**: 受信したリクエストは h2_mplx を通じてワーカースレッド（h2_workers）へ送られ、処理されます。 4. **レスポンス送出**: レスポンスは h2_bucket_beam 等を介してメイン接続（c1）へ戻されます。 5. **クリーンアップ**: 接続切断やプロトコルエラー発生時、h2_session_state を通じて CLEANUP 状態へ遷移し、リソースを解放します。

クライアント側 (h2ws.c)

1. **初期化**: h2_session_open により、非ブロッキングソケット（fd）の作成、nghttp2_session の初期化、および h2_session 構造体のセットアップが行われます。 2. **イベントループ**: h2_session_run 内で poll() ループを回し、want_io フラグに基づいて読み書きの必要性を制御しながら、nghttp2 コールバックや I/O 処理を実行します。 3. **終了**: h2_session_close により、セッションを終了しリソースをクローズします。

*注意: サーバー側とクライアント側の間でのネットワークを跨いだ詳細なエンドツーエンドの相互作用や、サーバー側内部の h2_session_process の詳細なループ不変条件は、提供されたシンボル情報からは完全には網羅されていません。*

重要な不変条件

  • **サーバー側 (mod_http2)**:
  • ngh2 フィールド（内部的な nghttp2 セッション状態）は、Apache 側の状態（h2_session_state）と同期している必要があります。
  • pool（メモリプール）は、セッションの生存期間中、常に有効でなければなりません。
  • セッションは、対応する接続（conn_rec）と密接に紐付いて運用されます。
  • **クライアント側 (h2ws.c)**:
  • fd は非ブロッキングモードで設定された有効な TCP ソケットでなければなりません。
  • ngh2 は nghttp2_session_client_new() 等で適切に初期化されている必要があります。

よくある落とし穴

  • **名前の重複**: modules/http2 の h2_session と test/clients/h2ws.c の h2_session は、名前は同じですが完全に別個の実装であり、直接的な関係性はありません。コードを追う際やデバッグの際には、どちらのコンテキストであるかを混同しないよう注意が必要です。
  • **エラーハンドリングとリソースリーク**:
  • サーバー側では、max_stream_errors を超えるストリームエラーが発生した場合にセッションが異常終了します。
  • クライアント側（h2ws.c）では、終了時のメモリ解放処理が網羅的ではないため、テスト実行後のメモリリークに注意する必要があります。
  • **I/O ブロック**: クライアント側では want_io フラグによる poll() 制御が適切に行われないと、I/O 待ちや処理の遅延が発生する可能性があります。
