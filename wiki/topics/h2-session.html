h2_session

概念

h2_session は、HTTP/2 接続におけるセッションの状態やライフサイクルを管理するためのデータ構造です。この名前を持つ構造体は、Apache HTTP Server のサーバー側実装（mod_http2）と、テスト用クライアントツール（h2ws.c）の2つの異なるコンテキストに存在します。

  • **サーバー側 (mod_http2)**: クライアントとの単一の TCP 接続（conn_rec）を抽象化し、プロトコル層としてフレームの解析、イベントのディスパッチ、およびストリームの多重化を制御します。
  • **クライアント側 (h2ws.c)**: テストクライアントにおける HTTP/2 接続、ストリームのライフサイクル、および非ブロッキング I/O 状態を追跡・管理します。

なぜ重要か

  • **サーバー側**: mod_http2 モジュールの全コンポーネントが依存する中心的な基盤であり、接続（C1層）と密接に紐付いて HTTP/2 プロトコルの制御フローを維持するためです。
  • **クライアント側**: テストツールが nghttp2 ライブラリを介してプロトコルイベントを適切に処理し、接続の確立や終了をシミュレートするための基盤となるためです。

エンドツーエンドの流れ

サーバー側 (mod_http2)

1. **初期化**: h2_c1_setup から h2_session_create が呼び出され、h2_session 構造体が生成されます。この内部で h2_c1_io_init や各種キュー（h2_iq_create）、ストリームの初期化が行われます。 2. **イベント処理と多重化**: h2_session_process がメインループとして機能し、nghttp2 を介してフレームを読み込みます。受信したリクエストは h2_mplx を通じてワーカースレッド（h2_workers）へ送られ、レスポンスはメイン接続（c1）へ送出されます。 3. **終了**: 接続の切断やプロトコルエラー発生時、セッションは CLEANUP 状態へ遷移し、リソースを解放します。

クライアント側 (h2ws.c)

1. **初期化**: main から h2_session_open が呼び出され、ソケットの接続（open_connection）や非ブロッキング設定（sock_nonblock_nodelay）が行われ、h2_session が初期化されます。 2. **実行**: h2_session_run 内のイベントループと nghttp2 コールバックを通じて、I/O 状態（want_io）やストリームのライフサイクルが管理されます。 3. **終了**: main から h2_session_close が呼び出され、セッションがクローズされます。

*注意: h2_session_create、h2_session_open、および h2_session_close の詳細な内部アルゴリズムや一部の呼び出し関係は、現時点では完全には解析されていません。*

重要な不変条件

  • **サーバー側**:
  • ngh2 フィールド（nghttp2_session）が保持するプロトコル状態は、Apache 側のセッション状態（h2_session_state）と同期していなければなりません。
  • pool フィールドは、セッションの生存期間を通じて有効なメモリプールとして維持される必要があります。
  • **クライアント側**:
  • fd は非ブロッキングモードで設定された有効な TCP ソケットでなければなりません。
  • ngh2 は nghttp2 ライブラリによって適切に初期化されている必要があります。

よくある落とし穴

  • **名前の重複**: test/clients/h2ws.c 内の h2_session は、Apache 本体の mod_http2 にある同名の構造体とは完全に別個の実装であり、直接的な関係はありません。コードリーディングの際は混同しないよう注意が必要です。
  • **エラーによるセッション終了**: サーバー側では、max_stream_errors を超えるストリームエラーが発生すると、セッション全体が異常終了します。また、out_c1_blocked などのキュー状態により I/O 待ちや遅延が発生することがあります。
  • **メモリリーク**: クライアント側（h2ws.c）はテスト用ツールであるため、終了時のメモリ解放処理が網羅的ではない可能性があり、メモリリークに注意する必要があります。
