h2_session

概念

h2_session は、HTTP/2 プロトコルにおけるセッション（接続）の状態やライフサイクルを管理するためのデータ構造です。 この名前を持つ構造体は、Apache HTTP Server の mod_http2 モジュール（サーバー側）と、テストクライアントツールである h2ws.c（クライアント側）の2つの異なるコンテキストにおいて、それぞれ独立して定義されています。

  • **サーバー側 (mod_http2)**: クライアントとの単一の TCP 接続（conn_rec）を抽象化し、プロトコル層として HTTP/2 フレームの解析、イベントディスパッチ、およびストリームの多重化を制御します。
  • **クライアント側 (h2ws)**: テストクライアントにおける HTTP/2 セッション状態、ソケット記述子（fd）、および nghttp2 ライブラリとの接続状態を管理します。

なぜ重要か

  • **サーバー側 (mod_http2)**: mod_http2 の全コンポーネントが依存する基盤であり、接続（conn_rec）と密接に紐付いて動作します。nghttp2 ライブラリによるフレーム解析と Apache 側の状態（h2_session_state）を同期させ、ワーカースレッドへのリクエスト振り分け（h2_mplx 経由）やレスポンス送出を統制する司令塔となるため極めて重要です。
  • **クライアント側 (h2ws)**: テストツールにおいて、非ブロッキングソケットを介した I/O 監視（want_io）やストリームのライフサイクル、異常終了（aborted）の追跡を担う中心的な構造体です。

エンドツーエンドの流れ

提供された情報からは、サーバー側およびクライアント側における完全なエンドツーエンドの制御フロー（接続確立から切断にいたる全ステップ）の詳細は完全には明らかになっていません。現時点で判明している部分的な流れは以下の通りです。

サーバー側 (mod_http2)

1. **初期化**: h2_session_create によりセッションが生成され、接続に関連付けられます。 2. **イベント処理**: h2_session_process がメインループとして機能し、nghttp2 を介してフレームを読み込み、イベントをディスパッチします。 3. **多重化**: 受信したリクエストは h2_mplx を通じてワーカースレッドへ送られ、レスポンスはメイン接続へ戻されます。 4. **クリーンアップ**: 接続切断やプロトコルエラー発生時、h2_session_state を通じて CLEANUP 状態へ遷移し、リソースの解放を試みます。

クライアント側 (h2ws)

1. **初期化**: h2_session_open() により構造体が初期化され、非ブロッキングソケット fd や nghttp2 セッションが設定されます。 2. **イベントループ**: h2_session_run() 内において、poll() による I/O 監視（want_io）や nghttp2 コールバックを介して処理が行われます。

重要な不変条件

  • **サーバー側 (mod_http2)**:
  • ngh2 フィールド（nghttp2_session）が保持する内部的なプロトコル状態は、Apache 側のセッション状態（h2_session_state）と常に同期していなければなりません。
  • pool フィールドは、セッションの生存期間中、有効なメモリプールとして維持される必要があります。
  • **クライアント側 (h2ws)**:
  • fd は非ブロッキングモードで設定された有効な TCP ソケットでなければなりません。
  • ngh2 は nghttp2_session_client_new() 等で適切に初期化されていなければなりません。

よくある落とし穴

  • **名前の重複**: modules/http2 の h2_session と test/clients/h2ws.c の h2_session は、名前は同一ですが全く別個の実装であり、直接的な相互作用はありません。コードリーディングやデバッグの際に混同しないよう注意が必要です。
  • **サーバー側のエラーハンドリング**: max_stream_errors を超えるストリームエラーが発生すると、セッションが異常終了します。また、out_c1_blocked や ready_to_process キューの状態によっては I/O 待ちや処理遅延が発生しやすい傾向にあります。
  • **クライアント側のメモリ管理**: h2ws.c における終了時のメモリ解放処理は網羅的ではないため、テスト実行後のメモリリークに注意する必要があります。
