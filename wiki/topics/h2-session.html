h2_session

概念

h2_session は、HTTP/2 接続におけるセッションの状態、ライフサイクル、およびストリームの多重化を管理するためのデータ構造です。

この名前を持つ構造体は、Apache HTTP Server のサーバー側実装（mod_http2）と、テストクライアント（h2ws）の **2つの異なるコンテキストにおいて独立して定義されています**。

  • **サーバー側 (mod_http2)**: クライアントとの単一の TCP 接続（conn_rec）を抽象化し、プロトコル層としての役割を担う巨大な構造体です。
  • **クライアント側 (h2ws)**: テストツールにおいて、nghttp2 ライブラリを用いた接続管理、ストリームのライフサイクル管理、および I/O 状態の追跡を行うための軽量な構造体です。

なぜ重要か

  • **サーバー側 (mod_http2)**: HTTP/2 接続（C1層）におけるフレームの解析、イベントのディスパッチ、およびストリームの多重化を制御する司令塔として機能します。nghttp2 ライブラリと Apache 本体の仲介役として、リクエストを h2_mplx を経由してワーカースレッド（h2_workers）へ振り分け、レスポンスをクライアントへ送出する中心的な役割を担います。
  • **クライアント側 (h2ws)**: テストクライアントにおいて、非ブロッキングソケットを介した HTTP/2 通信イベントループを駆動し、テストシナリオを実行するための基盤となります。

エンドツーエンドの流れ

提供された情報に基づく、それぞれのコンテキストにおける処理の流れは以下の通りです。

サーバー側 (mod_http2)

1. **初期化**: 接続が確立されると、h2_session_create によって h2_session が生成され、接続（conn_rec）に関連付けられます。 2. **イベント処理**: h2_session_process がメインループとして機能し、nghttp2 を介してフレームを読み込み、イベントをディスパッチします。 3. **多重化と処理**: 受信したリクエストは h2_mplx を通じてワーカースレッドへ送られ、処理されます。 4. **レスポンス送出**: ワーカースレッドからのレスポンスは h2_bucket_beam 等を介してメイン接続（C1層）へ戻され、クライアントへ送信されます。 5. **クローズ**: 接続切断やエラー発生時、h2_session_state が CLEANUP 状態へ遷移し、リソースが解放されます。

クライアント側 (h2ws)

1. **初期化**: h2_session_open() により構造体が初期化され、非ブロッキングソケット fd や nghttp2_session がセットアップされます。 2. **イベントループ**: h2_session_run() 内で poll() ループが実行され、want_io フラグに基づいて読み書きを制御しながら nghttp2 コールバックを処理します。 3. **終了**: 接続エラーや処理完了に伴い、aborted フラグが設定され、セッションループが終了して h2_session_close 等でクローズされます。

*注意: h2_session_process や h2_session_run などの具体的な内部処理フローの全容は、提供されたシンボル情報からは完全には明らかになっていません。*

重要な不変条件

  • **サーバー側 (mod_http2)**:
  • ngh2 フィールド（nghttp2_session）が保持する内部的なプロトコル状態は、Apache 側の状態（h2_session_state）と常に同期していなければなりません。
  • pool（メモリプール）は、セッションの生存期間全体にわたって有効でなければなりません。
  • **クライアント側 (h2ws)**:
  • fd は非ブロッキングモードで設定された有効な TCP ソケットでなければなりません。
  • ngh2 は nghttp2_session_client_new() 等で適切に初期化されている必要があります。

よくある落とし穴

  • **名前の衝突**: modules/http2 の h2_session と test/clients/h2ws.c の h2_session は、名前は同じですが**完全に別個の実装**です。コードを追う際やデバッグ時にこれらを混同しないよう注意が必要です。
  • **サーバー側**: h2_session は非常に巨大な構造体であり、多くのサブコンポーネント（h2_mplx, h2_workers など）と密結合しています。構造体の変更はモジュール全体に影響を及ぼします。また、max_stream_errors を超えるストリームエラーが発生すると、セッション全体が異常終了します。
  • **クライアント側**: テストツール内の簡易実装であるため、終了時のメモリ解放処理が網羅的でない可能性があり、テスト実行後のメモリリークに注意する必要があります。
