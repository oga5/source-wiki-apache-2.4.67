h2_session

概念

h2_session は、HTTP/2 プロトコルにおけるセッションの状態、ライフサイクル、およびストリームの多重化を管理するためのデータ構造です。

この名前を持つ構造体は、Apache HTTP Server の mod_http2 モジュール（サーバー側）と、テストクライアントツール h2ws.c（クライアント側）の2つの異なるコンテキストにおいて、それぞれ独立して定義されています。

  • **サーバー側 (mod_http2)**: クライアントとの単一の TCP 接続（conn_rec）を抽象化し、プロトコル層としての役割を担います。
  • **クライアント側 (h2ws.c)**: テストクライアントにおける HTTP/2 接続状態、ソケット、およびストリームのライフサイクルを管理します。

なぜ重要か

HTTP/2 は単一の物理接続上で複数のストリームを多重化するプロトコルであるため、接続状態とストリーム群を統括する管理機構が不可欠です。

  • **サーバー側**では、nghttp2 ライブラリによるプロトコル解析と Apache の接続・リクエスト処理（ワーカースレッドへのディスパッチなど）を仲介する司令塔として機能します。
  • **クライアント側**では、非ブロッキング I/O ループと nghttp2 イベントのディスパッチを管理し、テストシナリオの実行を支えます。

関与するモジュール

  • module:modules/http2 (Apache HTTP Server の HTTP/2 実装)
  • module:test/clients (テスト用クライアントツール)

エンドツーエンドの流れ

サーバー側 (mod_http2)

1. **初期化**: 接続が確立されると、h2_session_create により h2_session が生成され、接続（conn_rec）に関連付けられます。 2. **イベント処理**: h2_session_process がメインループとして機能し、nghttp2 を介して受信フレームを解析します。 3. **多重化とディスパッチ**: 受信したリクエストは h2_mplx を通じてワーカースレッド（h2_workers）へ振り分けられます。 4. **レスポンス送信**: ワーカースレッドからのレスポンスは h2_bucket_beam などを経由してメイン接続（c1）へ送出されます。 5. **クローズ**: 接続切断やプロトコルエラー発生時、セッション状態（h2_session_state）は CLEANUP へ遷移し、リソースが解放されます。

クライアント側 (h2ws.c)

1. **初期化**: h2_session_open により、非ブロッキングソケット fd の設定や nghttp2_session の初期化が行われます。 2. **実行**: h2_session_run 内のイベントループで、poll() を用いて I/O 状態（want_io）を監視しながら、nghttp2 コールバックを介してフレームの送受信を行います。 3. **終了**: h2_session_close によりセッションが閉じられます。

*注意: サーバー側とクライアント側の詳細な相互作用や、サーバー側における一部の具体的なエラーハンドリングおよび I/O 制御の完全なエンドツーエンドフローは、提供されたコンテキストからは完全には特定されていません。*

重要な不変条件

  • **サーバー側 (mod_http2)**:
  • 内部の nghttp2_session (ngh2) の状態は、Apache 側のセッション状態（h2_session_state）と同期している必要があります。
  • セッションのメモリプール（pool）は、セッションの生存期間中、常に有効でなければなりません。
  • ストリームエラーの累積数が max_stream_errors を超えた場合、セッションは異常終了します。
  • **クライアント側 (h2ws.c)**:
  • ソケット fd は、非ブロッキングモードで設定された有効な TCP ソケットでなければなりません。
  • ngh2 は nghttp2 ライブラリによって適切に初期化されている必要があります。

よくある落とし穴

  • **名前の重複**: modules/http2 の h2_session と test/clients/h2ws.c の h2_session は、名前は同じですが**完全に別個の実装**であり、直接的なソースコード上の関係はありません。コードを追う際はコンテキストを混同しないよう注意が必要です。
  • **メモリリーク**: クライアント側（h2ws.c）の終了時におけるメモリ解放処理は網羅的ではないため、テスト実行後のメモリリークに注意する必要があります。
  • **状態同期**: サーバー側では、nghttp2 のプロトコル状態と Apache 側のセッション状態の同期が崩れると、予期しない動作やハングアップを招く可能性があります。
