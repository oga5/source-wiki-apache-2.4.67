core_server_config

概念

core_server_config は、Apache HTTP Server のサーバー単位（グローバルおよび仮想ホスト単位）の設定情報を保持するための構造体です。ドキュメントルート、アクセス制御、リクエスト処理の制限、およびプロトコル関連の動作設定などを格納します。

なぜ重要か

サーバー全体の動作やセキュリティ、リクエスト処理の制限（リダイレクト制限やHTTPプロトコル適合性チェックなど）を決定する中心的な設定データ構造だからです。各モジュール（http_filters.c、http_protocol.c、h2_c2_filter.c、h2_headers.c など）は、ap_get_module_config を介してこの構造体を参照し、リクエスト処理の基準値として利用します。

関与するモジュール

エンドツーエンドの流れ

1. **初期化**: サーバー起動時または仮想ホスト作成時に、create_core_server_config が呼び出され、apr_pcalloc により core_server_config 構造体のメモリが割り当てられます。

  • グローバルサーバーの場合、デフォルトのドキュメントルートやアクセプトフィルタ（accf_map）などが設定されます。
  • 仮想ホストの場合、メインサーバーの設定から一部（error_log_format など）を継承し、未設定状態の初期値が与えられます。

    • 2. **設定の適用**: 設定ファイル（httpd.conf 等）のディレクティブにより、構造体内の各フィールドに値が設定されます（※設定ファイルから各フィールドへの具体的なマージやパースのフローの全体像は、提供された情報からは未検証です）。 3. **参照**: リクエスト処理中、各モジュールが ap_get_module_config(s->module_config, &core_module) を呼び出してこの設定を取得し、動作を制御します。

重要な不変条件

  • trace_enable、http09_enable、http_conformance、http_methods などの設定値は、それぞれ定義された定数（例: AP_TRACE_*）の範囲内で使用される必要があります。
  • sec_dir および sec_url は、apr_array_header_t として適切に初期化されている必要があります。
  • 仮想ホスト設定の初期化時、メインサーバーの設定が適切に初期化されている必要があります（ap_get_core_module_config の結果に依存するため）。

よくある落とし穴

  • **プラットフォーム依存**: Windows環境では unc_list が追加される（#ifdef WIN32）ため、プラットフォーム間の互換性に注意が必要です。また、アクセプトフィルタ（accf_map）の設定もOS固有の挙動に依存します。
  • **バイナリ互換性**: 構造体にメンバーを追加すると、ap_mmn.h で定義されるモジュールバイナリ互換性（MMN）に影響を与える可能性があります。
  • **制限値超過と拒否**: redirect_limit や subreq_limit を超えた場合、リクエスト処理が中断されます。また、http_conformance を AP_HTTP_CONFORMANCE_STRICT に設定すると、非準拠なHTTPリクエストが拒否されるようになります。
