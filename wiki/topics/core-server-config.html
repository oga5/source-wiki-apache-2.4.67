core_server_config

概念

core_server_config は、Apache HTTP Server においてサーバー単位（グローバルおよび仮想ホスト単位）の設定情報を保持するための構造体です。ドキュメントルート、アクセス制御、リクエスト処理の制限、およびプロトコル関連の動作設定などのコアな設定値を格納します。

なぜ重要か

この構造体は、サーバーのライフサイクル全体を通じて、リクエスト処理の再帰制限（リダイレクトやサブリクエストの制限）やHTTPプロトコルの適合性チェック、プラットフォーム固有のアクセプトフィルタ設定などの基準値として機能します。Apacheのコア動作を制御し、各モジュールが共通の設定を参照できるようにするために極めて重要です。

関与するモジュール

  • module:include
  • module:server

主要エンティティ

  • symbol:include/http_core.h#core_server_config：サーバー単位の設定を保持する構造体定義。
  • symbol:server/core.c#create_core_server_config：core_server_config 構造体を初期化し、メモリを割り当てる関数。

エンドツーエンドの流れ

1. **初期化**: サーバーの起動時または仮想ホストの作成時に、server/core.c 内の create_core_server_config が呼び出されます。

  • apr_pcalloc により core_server_config 構造体のメモリが確保されます。
  • グローバルサーバー（!is_virtual）の場合は、デフォルトのドキュメントルートやアクセス名、プラットフォームに応じたアクセプトフィルタ（accf_map）などが設定されます。
  • 仮想ホストの場合は、メインサーバーから error_log_format などの設定を継承し、一部の値を初期化します。

    • 2. **設定の適用**: 設定ファイル（httpd.conf等）のディレクティブにより値が設定されます（※設定ファイルから各フィールドへの具体的なマッピングやパースの全体フローは、提供された情報からは未解明です）。 3. **参照**: リクエスト処理中、modules/http/http_filters.c や modules/http/http_protocol.c などの各モジュールが ap_get_module_config を介して本構造体を取得し、動作の制御（リダイレクト制限のチェックやHTTP適合性検証など）に利用します。

重要な不変条件

  • trace_enable、http09_enable、http_conformance、http_methods などのフィールドは、それぞれ定義された定数（AP_TRACE_* 等）の範囲内で使用される必要があります。
  • sec_dir および sec_url は apr_array_header_t として適切に初期化されている必要があります。
  • 仮想ホスト設定の初期化時、メインサーバーの設定が未初期化である場合は ap_get_core_module_config の結果に依存します。

よくある落とし穴

  • **プラットフォーム依存性**: Windows環境では unc_list が追加される（#ifdef WIN32）ため、プラットフォーム依存のコードを扱う際は注意が必要です。また、OS固有のアクセプトフィルタ設定による挙動の差異にも注意する必要があります。
  • **バイナリ互換性**: 構造体にメンバーを追加すると、ap_mmn.h で定義されるモジュールバイナリ互換性（MMN）に影響を与える可能性があります。
  • **制限値によるリクエスト拒否**: redirect_limit や subreq_limit を超えた場合、リクエスト処理が中断されます。また、http_conformance が AP_HTTP_CONFORMANCE_STRICT に設定されている場合、非準拠なHTTPリクエストが拒否される可能性があります。

次に読む場所

  • include/http_core.h：core_server_config 構造体の詳細なフィールド定義を確認できます。
  • server/core.c：create_core_server_config による初期化処理やデフォルト値の設定ロジックを確認できます。
