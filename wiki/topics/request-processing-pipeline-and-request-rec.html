Request Processing Pipeline and request_rec

概念

Apache HTTP Serverにおけるリクエスト処理パイプラインは、クライアントから受信したHTTPリクエストを処理し、レスポンスを返して接続をクリーンアップするまでの一連のフェーズ（認証、アクセスチェック、ハンドラ実行、フィルタ処理など）を管理するフレームワークです。

このパイプラインの中核をなすのが request_rec 構造体です。これは単一のHTTPリクエストのライフサイクル全体にわたり、リクエストの状態、ヘッダー、URI、メソッド、フィルタチェーンなどの情報を保持・共有するための中心的なデータ構造として機能します。

なぜ重要か

request_rec は、パイプラインを構成する様々なモジュール（認証、URL書き換え、コンテンツ生成など）の間でリクエストのコンテキストを共有するための共通言語を提供します。

また、標準的なリクエスト処理のエントリポイントである ap_process_request や、URIの書き換え等に伴い処理を再実行する ap_internal_redirect などの主要なAPIは、すべてこの request_rec を中心に設計されています。これにより、モジュール開発者は一貫したインターフェースを通じてリクエストの挙動を制御・カスタマイズできます。

関与するモジュール

提供された情報に基づき、以下のモジュールがこのパイプラインとデータ構造に関与しています。

  • module:include （include/httpd.h を含む）
  • module:modules/http （modules/http/http_request.c を含む）
  • module:server （server/config.c を含む）

主要エンティティ

  • **symbol:include/httpd.h#request_rec**:

    • リクエストのライフサイクルとコンテキストを保持する主要な構造体。メモリ管理用のリクエストプール（apr_pool_t）や、フィルタチェーン（input_filters, output_filters）などを管理します。

  • **symbol:modules/http/http_request.c#ap_process_request**:

    • HTTPリクエスト処理の標準的なエントリポイント。リクエストの実行（ap_process_async_request の呼び出し）と、接続状態に応じた出力バッファの最終的なフラッシュ処理を統括します。

  • **symbol:modules/http/http_request.c#ap_internal_redirect**:

    • 新しいURIに基づいてリクエストコンテキストを再構築し、処理フローを再帰的に再実行する内部リダイレクト関数。

エンドツーエンドの流れ

提供された情報から確認できるリクエスト処理の流れは以下の通りです。なお、ソケット受信から最終的な接続クローズに至る完全なエンドツーエンドの流れは、現時点では完全には判明していません。

1. **リクエストの実行開始**: ap_process_request が呼び出され、ap_process_async_request を介して主要な処理（フックの実行やハンドラ処理など）が開始されます。 2. **内部リダイレクトの発生（任意）**: 処理の過程で内部リダイレクトが必要になった場合、ap_internal_redirect が呼び出されます。

  • internal_internal_redirect により、新しいURI用の request_rec が生成されます。
  • クイックハンドラ（ap_run_quick_handler）が試行され、これが DECLINED を返した場合は、ap_process_request_internal によるアクセスチェック等を経て ap_invoke_handler でハンドラが実行されます。
  • 最終的に ap_die が呼び出され、リダイレクト処理の終了またはエラーレスポンスの生成が行われます。

    • 3. **レスポンスのフラッシュと統計記録**: ap_process_request は、接続に未処理の入力データがない場合、出力フィルタチェーンに対してフラッシュバケットを送信し、クライアントへのレスポンスを強制的にフラッシュします。また、ap_extended_status が有効な場合は、ap_time_process_request を呼び出して処理終了時刻を記録します。

重要な不変条件

  • **メモリ管理の自動化**:

    • request_rec 構造体内の多くの文字列や動的データは、リクエスト専用のメモリプール（apr_pool_t）に割り当てられます。このプールはリクエスト終了時に自動的に解放されるため、手動で解放してはならないという不変条件があります。

  • **リクエスト構造体の有効性**:

    • ap_process_request や ap_internal_redirect に渡される request_rec ポインタは、常に有効なインスタンスを指していなければなりません。

  • **内部リダイレクトのフォールバック**:

    • ap_internal_redirect において、新しいリクエストコンテキスト（new）の生成に失敗した場合、リダイレクト処理は行われず即座に処理が中断されます。

よくある落とし穴

  • **メモリの二重解放**:

    • request_rec に紐付くメモリプールで管理されているデータを、モジュール側で誤って手動解放（free など）してしまうと、メモリ破損やクラッシュの原因になります。

  • **構造体のレイアウト変更に伴う互換性問題**:

    • request_rec は非常に複雑な構造体であり、Apacheのバージョンアップ（MMN: Module Magic Number の更新）に伴い、フィールドの追加や削除（例: content_language の削除や used_path_info の追加など）が行われることがあります。直接フィールドにアクセスする際は、対象バージョンのヘッダー定義に注意する必要があります。

  • **タイムアウトエラーの誤認**:

    • ap_process_request 内のフラッシュ処理中に APR_STATUS_IS_TIMEUP が発生した場合、これはネットワーク切断やDoS攻撃の可能性を示すものであり、APLOG_INFO レベルでログ出力されますが、システム全体の致命的なエラーとは異なり、処理自体は継続されます。

次に読む場所

  • **include/httpd.h**: request_rec 構造体の詳細なフィールド定義や、関連するデータ構造（conn_rec, server_rec）を確認できます。
  • **modules/http/http_request.c**: ap_process_request や ap_internal_redirect の具体的な制御フローと、内部リダイレクト時のフィルタ更新処理（update_r_in_filters）の実装を確認できます。
  • **server/config.c**: リクエスト処理パイプラインにおける設定の適用や、モジュールごとのコンテキスト管理の仕組みを理解するのに役立ちます。
