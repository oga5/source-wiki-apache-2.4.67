main

概念

main は、Apache HTTP Server（httpd）プロジェクトにおける様々な実行可能バイナリのメインエントリポイント（起動起点）です。これには、Webサーバー本体（httpd）だけでなく、ビルド時に使用されるコード生成ツール、各種管理・支援用サポートユーティリティ、および動作検証やベンチマークを行うためのテストプログラムが含まれます。

なぜ重要か

各 main 関数は、それぞれのバイナリが独立したプロセスとしてOSから起動される際の初期化、引数解析、コアロジックの実行、および終了処理を統括します。

  • **サーバー本体 (server/main.c)**: サーバー全体のライフサイクル、設定読み込み、MPMの起動を管理する極めて重要なエントリポイントです。
  • **ビルドツール (server/gen_test_char.c)**: コンパイル時に必要な静的ルックアップテーブルを自動生成します。
  • **サポートユーティリティ (support/ 配下)**: パスワード管理（htpasswd, htdigest）、ログ解析（logresolve）、権限制御（suexec, checkgid）など、サーバーの運用・セキュリティを支える独立したツール群を提供します。
  • **テストプログラム (test/ 配下)**: 内部APIの単体テストや、システムコール（writev, select）の挙動、セマフォの性能測定などを個別に検証可能にします。

  • **サポートユーティリティ**:

エンドツーエンドの流れ

これらはそれぞれ独立してビルド・実行される個別のプログラムであるため、プロジェクト全体を通した単一の「エンドツーエンドの流れ」は存在しません。

ただし、代表的な流れとして以下が挙げられます。 1. **サーバー起動フロー (server/main.c)**:

  • apr_app_initialize によるAPR環境の初期化。
  • コマンドライン引数の解析（動作モードの決定）。
  • 設定ファイル（httpd.conf）の読み込みと適用。
  • 各種設定・初期化フック（pre_config, post_config）の実行。
  • MPM（マルチプロセッシングモジュール）のメインループ起動。設定再読み込み時はループ内で再起動。

    • 2. **安全なCGI実行フロー (support/suexec.c)**:

  • 環境変数の初期化（clean_env）。
  • 呼び出し元ユーザーの検証。
  • ターゲットユーザー/グループのUID/GID解決と権限降格（setuid/setgid）。
  • ディレクトリ権限およびドキュメントルート（AP_DOC_ROOT）内のパス検証。
  • execv によるターゲットプログラムの実行。

その他のテストやサポートツールについては、それぞれのコマンドライン引数に基づき、ファイル操作、ネットワーク通信、または内部APIの呼び出しを単発で実行して終了する流れとなります。

重要な不変条件

各 main 関数およびそのプログラムが維持すべき不変条件は以下の通りです。

  • **メモリ管理**: APR（Apache Portable Runtime）を使用するプログラム（server/main.c, support/htpasswd.c, support/logresolve.c など）では、すべての動的メモリは apr_pool_t によって管理され、終了時またはループのクリア時に適切に解放されなければならない。
  • **ファイル操作の原子性**: htpasswd や htdigest などのパスワード更新処理では、既存ファイルを直接書き換えるのではなく、一時ファイル（.tmp）を経由してアトミックに上書きコピーを行う。
  • **セキュリティ境界 (suexec.c)**: 実行対象ファイルやディレクトリがグループまたは他者への書き込み権限（S_IWGRP, S_IWOTH）を持ってはならず、setuid/setgid ビットが設定されていてもならない。また、ターゲットUID/GIDは AP_UID_MIN/AP_GID_MIN 以上でなければならない。
  • **文字種テーブルの参照 (gen_test_char.c)**: 生成された test_char_table を参照する際、符号拡張による不正なインデックス参照を防ぐため、インデックスは必ず unsigned char にキャストしてアクセスしなければならない。

よくある落とし穴

  • **プラットフォーム依存性**: server/main.c や support/win32/wintty.c などは、Windows（WIN32）やNetWareなどのプラットフォーム依存コードが条件付きコンパイルで多用されているため、修正時にクロスプラットフォームへの影響を考慮しないとビルドエラーを引き起こす。
  • **バッファオーバーフロー**: 古いテストプログラム（test/test_find.c, test/test_parser.c）では、標準入力の読み込みに脆弱な gets() が使用されているため、512バイトを超える入力によりバッファオーバーフローが発生する。
  • **排他制御の欠如**: htdigest などのツールでは、ファイル更新時に明示的なファイルロック（排他制御）が実装されていないため、複数プロセスから同時に実行された場合に競合が発生する可能性がある。
  • **実行権限への依存**: checkgid や suexec は、プロセスの権限（スーパーユーザー権限など）に強く依存しており、一般ユーザー権限で実行すると検証や権限降格に失敗する。
