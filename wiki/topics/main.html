main

概念

main は、C言語プログラムにおける標準的な実行エントリーポイント（main 関数）を指します。Apache HTTP Server のコードベースにおいては、サーバー本体の起動だけでなく、ビルド時のコード生成ツール、各種管理・サポート用ユーティリティ、および動作検証用のテストプログラムなど、多岐にわたる独立した実行可能バイナリのエントリーポイントとして定義されています。

なぜ重要か

Apache HTTP Server プロジェクトは、単一の Web サーバーデーモンだけでなく、それを支えるエコシステム（ビルドツール、運用管理ユーティリティ、テストスイート）全体で構成されています。それぞれのバイナリが固有の main 関数を起点として動作するため、システムのライフサイクル管理、セキュリティ境界の確立（例: suexec）、管理タスクの自動化、および機能検証を支える極めて重要な役割を担っています。

関与するモジュール

  • module:server (サーバー本体およびビルド時ツール)
  • module:support (各種管理・サポートユーティリティ)
  • module:support/win32 (Windows サービス環境向けユーティリティ)
  • module:test (各種機能・負荷テストプログラム)
  • module:test/clients (テスト用クライアント)

エンドツーエンドの流れ

各 main 関数はそれぞれ独立したバイナリのエントリーポイントであるため、これらすべてを包括する単一のエンドツーエンドの流れは存在しません。

個々の主要な流れの例は以下の通りです：

  • **サーバー起動 (server/main.c)**:

    • 1. init_process による APR 環境とグローバルプールの構築。 2. apr_getopt によるコマンドライン引数の解析と動作モードの決定。 3. ap_read_config による設定ファイルの読み込みと適用。 4. 設定前後のフック（pre_config, post_config）の実行。 5. メインループ内での MPM の起動と、設定再読み込み時の再起動制御。

  • **CGI 安全実行 (support/suexec.c)**:

    • 1. clean_env() による環境変数の初期化。 2. 呼び出し元ユーザーが AP_HTTPD_USER と一致するかの検証。 3. ターゲットユーザー・グループの UID/GID 解決。 4. setgid(), initgroups(), setuid() による権限降格。 5. 実行対象ディレクトリ・ファイルのパーミッションおよび AP_DOC_ROOT 内に存在することの検証。 6. execv() によるターゲットプログラムの実行。

なお、support/ab.c、support/htcacheclean.c、support/htdbm.c、test/clients/h2ws.c の詳細なエンドツーエンドの流れについては、現時点では解析されておらず不明です。

重要な不変条件

それぞれの main 関数が動作するコンテキストにおいて、以下の不変条件が維持される必要があります：

  • **メモリ管理**: 多くのユーティリティ（server/main.c, support/htpasswd.c, support/httxt2dbm.c, support/logresolve.c など）では、動的メモリは APR プール (apr_pool_t) によって管理され、終了時またはループの各反復時に適切に解放・クリアされる必要があります。
  • **セキュリティ境界 (suexec.c)**: 実行対象ファイルやディレクトリがグループまたは他者への書き込み権限（S_IWGRP, S_IWOTH）を持ってはならず、setuid/setgid ビットが設定されていてはなりません。また、実行対象の UID/GID は AP_UID_MIN/AP_GID_MIN 以上でなければなりません。
  • **アトミックなファイル更新 (htpasswd.c, htdigest.c)**: パスワードファイルの更新は、直接書き換えるのではなく、一時ファイル（htpasswd.tmp.XXXXXX など）を経由して行われ、最後に元のファイルへコピー（上書き）することでアトミック性を保証します。
  • **文字種判定テーブルの安全性 (gen_test_char.c)**: 生成された test_char_table を参照する際、符号拡張による不正な負のインデックス参照を防ぐため、インデックスアクセス時には必ず unsigned char へのキャストを行う必要があります。

よくある落とし穴

  • **プラットフォーム依存性**:
  • support/win32/wintty.c は Windows サービス環境に特化した実装であり、isservice の判定が WindowStation 名（"Service-" で始まるか）に依存しているため、環境によっては誤判定が発生する可能性があります。
  • support/suexec.c には _OSD_POSIX などのプラットフォーム依存コードが含まれており、移植時の挙動に注意が必要です。
  • **古い C 標準関数の使用による脆弱性**:
  • test/test_find.c や test/test_parser.c などの古いテストコードでは、バッファオーバーフローの脆弱性を持つ gets() が使用されており、512バイトを超える入力に対して未定義動作を引き起こします。
  • **排他制御の欠如**:
  • support/htdigest.c などのユーティリティでは、ファイル更新時に明示的なファイルロック（排他制御）が実装されていないため、複数プロセスから同時に実行された場合に競合が発生するリスクがあります。

次に読む場所

  • server/main.c (サーバー本体の起動処理とライフサイクル管理)
  • support/suexec.c (セキュリティ上重要な権限降格ラッパーの実装)
  • support/htpasswd.c (一時ファイルを用いたアトミックなファイル更新処理)
  • server/gen_test_char.c (ビルド時に実行される文字種テーブル生成ロジック)
  • support/win32/wintty.c (Windows サービス環境におけるコンソール制御)
