dav_namespace_map

概念

dav_namespace_map は、WebDAV リクエストやプロパティ処理において、XML 名前空間（Namespace）の URI を内部的な数値 ID（インデックス）にマッピングするためのテーブル構造を保持する構造体です。

modules/dav/main/mod_dav.h で不完全型として宣言され、modules/dav/fs/dbm.c において具体的なメンバを持つ構造体として定義されています。

なぜ重要か

WebDAV プロパティ（特に DBM ベースのプロパティデータベース）の処理において、冗長な XML 名前空間 URI を効率的に管理・解決するために不可欠です。名前空間を数値 ID にマッピングすることで、プロパティの保存や検索（キーの構築など）における整合性と処理効率を担保します。

主要エンティティ

  • **symbol:modules/dav/main/mod_dav.h#dav_namespace_map**

    • プロバイダーインターフェースやプロパティ処理コンテキスト（dav_prop_ctx）などで参照される、構造体の型宣言です。

  • **symbol:modules/dav/fs/dbm.c#dav_namespace_map**

    • int *ns_map をメンバに持つ、構造体の具体的な定義です。名前空間 URI をインデックス化するための整数配列へのポインタを保持します。

エンドツーエンドの流れ

提供されたソースコードの断片からは、詳細なエンドツーエンドの制御フロー全体は完全には明らかになっていません。現時点で判明している部分的な流れは以下の通りです。

1. **初期化とメモリ確保**: DBM プロパティデータベースの操作時などに、関連するメモリプール（apr_pool_t、例えば db->pool）を介して dav_namespace_map のメモリが割り当てられます。 2. **名前空間のマッピング**: dav_propdb_map_namespaces などの関数において、XML 名前空間 URI が内部 ID に変換され、ns_map 配列にマッピング情報が格納・更新されます。 3. **プロパティ処理での参照**: プロパティコンテキスト（dav_prop_ctx）や DBM 操作（dav_db）の中で、このマッピング情報が参照され、名前空間の解決やデータベースキーの構築（dav_build_key）に利用されます。

重要な不変条件

  • **メモリの有効性**:

    • ns_map メンバは、apr_palloc 等によって適切にメモリ確保された整数配列を指していなければなりません。

  • **ライフサイクルの整合性**:

    • 本構造体のメモリ管理は Apache の apr_pool_t に依存しており、割り当てられたプールの寿命と整合している必要があります。

  • **インデックスの整合性**:

    • dav_build_key が使用する名前空間インデックスと、dav_namespace_map 内で管理されるインデックスは常に一致していなければなりません。

よくある落とし穴

  • **NULL ポインタアクセス**:

    • ns_map が NULL の状態でマッピング処理や参照を行うと、セグメンテーションフォールトが発生する可能性があります。

  • **未定義状態での参照**:

    • 名前空間マッピングが未定義または NULL の状態で参照された場合の挙動は、各フック実装側のガード条件に依存します。

  • **プール寿命の超過**:

    • apr_pool_t に紐付いたメモリ管理が行われているため、プールの寿命を超えて構造体やそのメンバへのポインタを保持・参照すると、メモリ破損や不正アクセスを引き起こす恐れがあります。
