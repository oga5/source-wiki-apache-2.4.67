worker_main

概念

worker_main は、Apache HTTP Server（httpd-2.4.67）の特定のマルチプロセッシングモジュール（MPM）において、クライアントからの接続リクエストを処理する個々のワーカースレッドのメインループを実行する関数です。OS/2、NetWare、Windows（winnt）の各プラットフォーム向けMPMで、それぞれ固有のAPIや並行処理モデルを用いて実装されています。

なぜ重要か

各プラットフォームの特性（OS/2のキュー、NetWareのNKSスレッドと select、WindowsのI/O Completion Port（IOCP））に最適化された方法で、接続の受け入れ、接続オブジェクトの作成、リクエストの処理、および接続のクローズというApacheのコアな接続ライフサイクルを駆動するためです。

エンドツーエンドの流れ

各MPMにおけるスレッド起動から終了までの流れは以下の通りです。なお、親プロセスから子プロセス・スレッド起動に至る全体的なエンドツーエンドの統合フローの詳細は、提供された情報からは完全には明らかになっていません。

1. OS/2 (mpmt_os2)

1. **起動**: _beginthread により起動され、例外ハンドラ（DosSetExceptionHandler）を登録してOS/2キューを開きます。 2. **待機**: DosReadQueue を使用して、キューから接続情報（worker_args）が届くのを待機します。 3. **処理**: ap_run_create_connection で接続オブジェクトを作成し、ap_process_connection でリクエストを処理後、ap_lingering_close で接続を閉じます。 4. **終了**: WORKTYPE_EXIT メッセージを受信するかキューの読み取りに失敗した時点で、スコアボードを SERVER_DEAD に更新して終了します。

2. NetWare (netware)

1. **起動**: NXContextAlloc に関数ポインタが渡されて起動し、トランザクション用メモリプール（ptrans）を初期化します。 2. **待機と受付**: select() でリスニングソケットを監視し、接続要求を検知すると apr_socket_accept() でソケットを受け入れます。 3. **処理**: ap_run_create_connection -> ap_process_connection -> ap_lingering_close の順で接続を処理します。 4. **終了**: die_now フラグが真になるか、リクエスト上限に達した場合に clean_child_exit を呼び出して終了します。

3. Windows (winnt)

1. **起動**: 子プロセス起動時にワーカースレッドとして生成されます。 2. **待機**: winnt_get_connection を呼び出し、IOCPを介して接続コンテキスト（winnt_conn_ctx_t）が取得できるのを待機します。 3. **処理**: ap_run_create_connection -> ap_process_connection -> ap_lingering_close の順で接続を処理します。 4. **終了**: 接続コンテキストが取得できなくなった場合やリクエスト上限に達した場合に、スコアボードを SERVER_DEAD に更新して終了します。

重要な不変条件

プラットフォーム全体を跨ぐグローバルな不変条件の全容は完全には既知ではありませんが、各実装において以下の不変条件が維持されます。

  • **スコアボード状態の同期**: 接続処理の開始時や待機時にはスコアボードを SERVER_READY（または相当する状態）に、処理中は SERVER_BUSY 相当に、スレッド終了時には SERVER_DEAD に更新し、現在のスレッド状態を正確に反映します。
  • **リソースの局所化**: 各接続処理において、メモリリークを防ぐために接続用のプール（OS/2の pconn、NetWareの ptrans など）が接続ごとに作成・破棄されます。
  • **終了シグナルの監視**: die_now フラグ（NetWare）や ap_max_requests_per_child によるリクエスト上限（NetWare、Windows）に達した際、スレッドは安全にループを脱出してクリーンアップ処理へ移行しなければなりません。

よくある落とし穴

  • **プラットフォーム依存性**: 各 worker_main はOS固有のAPI（OS/2の Dos*、NetWareのNKS、WindowsのIOCP）に強く依存しているため、コードの移植性はありません。
  • **メモリリーク**: Windows環境における winnt_conn_ctx_t の再利用ロジックなど、エラーパスにおけるメモリプールの解放漏れに注意が必要です。
  • **ブロッキングとハングアップ**: NetWare環境などでは、ブロッキング操作によりスレッドがハングアップするリスクがあります。また、ネットワーク層の停止（ENETDOWN）などの致命的エラー発生時に、適切なエラーハンドリング（clean_child_exit による安全な終了など）を行わないと、サーバー全体が不安定になります。
