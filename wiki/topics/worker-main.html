worker_main

概念

worker_main は、Apache HTTP Server（httpd-2.4.67）の特定のプラットフォーム向けマルチプロセッシングモジュール（MPM）において、クライアントからの接続を受け入れてリクエストを処理する**ワーカースレッドのメインループ**を実装した関数です。OS/2、NetWare、Windows (WinNT) の各MPMで同名の関数が定義されており、それぞれのOS固有の非同期I/Oやスレッドモデルに適合した方法で接続を待ち受け、共通の接続処理APIを呼び出します。

なぜ重要か

この関数は、サーバーがクライアントからのリクエストを実際に処理する実行主体のライフサイクルを直接管理するため極めて重要です。プラットフォームごとの最適なI/Oモデル（OS/2のメッセージキュー、NetWareの select、WindowsのIOCP）を隠蔽しつつ、Apache共通の接続処理フロー（接続作成、リクエスト処理、遅延クローズ）へと橋渡しする役割を担っています。

エンドツーエンドの流れ

各プラットフォームにおける大まかな流れは以下の通りですが、プラットフォーム間の統合的なエンドツーエンドの全体フローや、親プロセスからこれらのスレッドが起動されるまでの完全なライフサイクルは、提供された情報からは完全には明らかになっていません。

1. **初期化**:

  • **OS/2**: 例外ハンドラ（DosSetExceptionHandler）を登録し、OS/2キューを開く。
  • **NetWare**: スレッド固有のメモリプール（ptrans）とバケットアロケータを作成。
  • **Windows**: スレッドローカルストレージを初期化（AP_HAS_THREAD_LOCAL 有効時）。

    • 2. **接続の待機と取得**:

  • **OS/2**: DosReadQueue を使用して、キューから接続情報（worker_args）を読み出す。
  • **NetWare**: select() でリスニングソケットを監視し、apr_socket_accept() で接続を受け入れる。
  • **Windows**: winnt_get_connection を呼び出し、IOCP経由で接続コンテキストを取得する。

    • 3. **共通の接続処理**:

  • スコアボードのステータスを更新（SERVER_READY から SERVER_BUSY 相当へ）。
  • ap_run_create_connection で接続オブジェクトを作成。
  • ap_process_connection でHTTPリクエストを処理。
  • ap_lingering_close で接続を閉じる。

    • 4. **終了処理**:

  • 終了シグナルやリクエスト上限（ap_max_requests_per_child）に達した場合、ループを脱出。
  • スコアボードを SERVER_DEAD に更新し、リソースを解放してスレッドを終了する。

重要な不変条件

プラットフォーム全体にわたる統一的な不変条件は提供された情報からは完全には特定されていませんが、各実装において以下の条件が維持されます：

  • **OS/2**: 例外発生時にプロセス全体のクラッシュを防ぐため、DosSetExceptionHandler による例外ハンドラが登録されていること。
  • **NetWare**: die_now フラグが偽である限りスレッドはループを継続し、requests_this_child が上限を超えると終了すること。
  • **Windows**: pchild プールが子プロセスの生存期間を管理し、winnt_get_connection が NULL を返した場合はループを終了すること。

よくある落とし穴

  • **プラットフォーム依存性**: 同名関数でありながら、OS/2の Dos* API、NetWareのNKSスレッド、WindowsのIOCPなど、実装が完全にプラットフォーム固有のAPIに依存しているため、共通コードとして混同しないよう注意が必要です。
  • **メモリリーク**:
  • OS/2では、メモリリークを防ぐために各接続処理ごとに pconn プールを作成・破棄する設計になっています。
  • Windowsでは、winnt_conn_ctx_t の再利用ロジックにおけるエラーパスでメモリリークが発生する可能性があります。
  • **ブロッキング操作**: NetWare環境ではNKSスレッドとして動作するため、ブロッキング操作によるパフォーマンス低下に注意する必要があります。
