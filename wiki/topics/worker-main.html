worker_main

概念

worker_main は、Apache HTTP Server（httpd-2.4.67）の特定のマルチプロセッシングモジュール（MPM）において、クライアントからの接続リクエストを処理する個別ワーカー（スレッド）のメインループまたはエントリポイントとして機能する関数です。OS/2、NetWare、Windows（winnt）の各プラットフォーム向けMPMで、それぞれ独自の並行処理モデルに合わせて実装されています。

なぜ重要か

この関数は、各プラットフォーム固有の非同期I/Oやスレッドモデル（OS/2のキュー、NetWareの select/accept、WindowsのI/O完了ポート）に適合した形で、Apacheのコアな接続ライフサイクル（接続の受け入れ、接続オブジェクトの作成、リクエスト処理、接続のクローズ）を駆動するために不可欠な役割を担っています。

主要エンティティ

  • **symbol:server/mpm/mpmt_os2/mpmt_os2_child.c#worker_main**

    • OS/2 MPMの子プロセスにおいて、_beginthread により起動されるワーカースレッドのメイン関数。OS/2のシステムキューから接続情報を取得して処理します。

  • **symbol:server/mpm/netware/mpm_netware.c#worker_main**

    • NetWare MPMにおけるワーカースレッドのメインループ。NXContextAlloc に関数ポインタとして渡され、select() と apr_socket_accept() を用いて直接接続を監視・受け入れます。

  • **symbol:server/mpm/winnt/child.c#worker_main**

    • Windows (winnt) MPMにおけるワーカースレッドのメインループ。I/O完了ポート（IOCP）を介して winnt_get_connection から接続コンテキストを取得し、処理を行います。

エンドツーエンドの流れ

プラットフォームごとに接続の取得方法が異なります。なお、親プロセスから子プロセスが起動され、各スレッドへ接続がディスパッチされるシステム全体の完全なエンドツーエンドのフローは、提供されたモジュール情報の範囲内では一部未確定ですが、各 worker_main 内部の処理フローは以下の通りです。

1. **初期化**:

  • **OS/2**: DosSetExceptionHandler によりスレッド内の例外ハンドラを登録し、OS/2キューを開きます。
  • **NetWare**: スレッド固有のメモリプール（ptrans）とバケットアロケータを作成し、スレッドカウントをインクリメントします。
  • **Windows**: スレッドローカルストレージ（AP_HAS_THREAD_LOCAL 有効時）の初期化を行います。

    • 2. **接続の取得と待機**:

  • **OS/2**: DosReadQueue を使用して、キューから worker_args_t（接続情報）が届くのを待機します。
  • **NetWare**: select() で listenfds を監視し、接続要求を検知すると apr_socket_accept() でソケットを受け入れます。
  • **Windows**: winnt_get_connection を呼び出し、接続コンテキスト（winnt_conn_ctx_t）が取得できるまで待機します。

    • 3. **リクエストの処理**:

  • 共通して、ap_run_create_connection で接続オブジェクトを作成し、ap_process_connection でHTTPリクエストを処理し、ap_lingering_close で接続を閉じます。
  • 処理の開始・終了に合わせて、スコアボードのステータス（SERVER_READY や SERVER_BUSY など）を更新します。

    • 4. **終了処理**:

  • **OS/2**: WORKTYPE_EXIT メッセージを受信するか、キューの読み取りに失敗した時点でループを脱出し、スコアボードを SERVER_DEAD に更新して終了します。
  • **NetWare**: die_now フラグが真になるか、リクエスト上限（ap_max_requests_per_child）に達した際に clean_child_exit を呼び出して終了します。
  • **Windows**: winnt_get_connection が NULL を返した場合やリクエスト上限に達した際にループを抜け、スコアボードを SERVER_DEAD に更新して終了します。

重要な不変条件

  • **OS/2**: スレッド内の例外によってプロセス全体がクラッシュするのを防ぐため、DosSetExceptionHandler による例外ハンドラが登録されている必要があります。また、pchild プールが生存している必要があります。
  • **NetWare**: 呼び出し時に pmain プールが初期化されている必要があります。また、die_now フラグが偽である限り、スレッドは生存し続けます。
  • **Windows**: pchild プールが子プロセスの生存期間を管理している必要があります。
  • **共通**: メモリリークを防ぐため、各接続処理のたびにトランザクション用のメモリプール（OS/2の pconn、NetWareの ptrans など）が適切に作成・破棄される必要があります。

よくある落とし穴

  • **OS/2**: OS/2の Dos* APIに強く依存しているため、他プラットフォームへの移植性はありません。キューのオープン（DosOpenQueue）に失敗した場合、該当スロットのスコアボードをクリアして即座に終了します。
  • **NetWare**: NetWareのNKSスレッドとして動作するため、ブロッキング操作を行うとシステム全体のパフォーマンスに影響を及ぼす可能性があります。また、ネットワーク層の停止（ENETDOWN）を検知した場合は APEXIT_CHILDFATAL を指定してサーバー全体を終了させます。
  • **Windows**: winnt_conn_ctx_t の再利用ロジックにおいて、エラーパスを通過する際のメモリリークに注意が必要です。また、AP_HAS_THREAD_LOCAL の定義の有無によって、スレッドローカルストレージの初期化と破棄の挙動が分岐します。
