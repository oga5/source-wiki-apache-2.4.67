Cache Storage and Shared Memory Socache Providers

概念

Apache HTTP Serverにおけるキャッシュ機構は、主に2つの異なる目的とレイヤーに分かれて提供されています。

1. **HTTPレスポンスキャッシュ（RFC準拠）**: クライアントへのHTTPレスポンスをキャッシュするための仕組みです。コアロジックを mod_cache.c が担い、実際の永続化ストレージは mod_cache_disk.c などのストレージプロバイダが担当します。 2. **共有オブジェクトキャッシュ（socache）**: SSLセッションや認証情報など、小さく高頻度でアクセスされるオブジェクトをプロセス間で高速に共有・キャッシュするための仕組みです。ap_socache.h で共通インターフェースが定義され、mod_socache_shmcb.c などのプロバイダが具体的な共有メモリ管理を提供します。

なぜ重要か

  • **パフォーマンスの向上**:

    • 重いバックエンド処理やディスクI/O、SSL/TLSハンドシェイクにおける鍵交換などの高コストな処理を回避し、レスポンスタイムを劇的に削減します。

  • **モジュール化と柔軟性**:

    • キャッシュの制御ロジックとストレージの実装が分離されているため、システムの要件（ディスクベース、共有メモリベースなど）に応じて最適なプロバイダを選択・拡張できます。

関与するモジュール

  • module:include
  • module:modules/cache

主要エンティティ

  • **ap_socache_provider_t** (include/ap_socache.h):

    • 共有オブジェクトキャッシュ（socache）の実装が満たすべきインターフェースを定義する構造体です。init, create, store, retrieve, remove, destroy などの関数ポインタを提供します。

  • **mod_cache** (modules/cache/mod_cache.c):

    • HTTPキャッシュのコアロジックを実装するモジュールです。キャッシュの有効期限（Expiration）や再検証（Validation）などのRFC準拠の制御を行います。

  • **mod_cache_disk** (modules/cache/mod_cache_disk.c):

    • mod_cache のためのディスクベースのストレージプロバイダです。キャッシュされたレスポンスのヘッダーとボディをディスク上のファイルとして管理します。

  • **mod_socache_shmcb** (modules/cache/mod_socache_shmcb.c):

    • 共有メモリ上のサイクリックバッファ（shmcb）を利用した ap_socache_provider_t の具体的な実装です。SSLセッションキャッシュなどで広く利用されます。

エンドツーエンドの流れ

提供されたソースコードの範囲内では、HTTPキャッシュ（mod_cache / mod_cache_disk）と共有オブジェクトキャッシュ（mod_socache_shmcb）の間の直接的な統合や、詳細な相互作用のエンドツーエンドのフローは完全には定義されておらず、これらがどのように連携するか（あるいは独立して動作するか）の全体像は未確定です。

一般的には、以下のように独立したフローが存在します。

HTTPキャッシュのフロー

1. クライアントからリクエストを受信すると、mod_cache がキャッシュの有無を判定します。 2. キャッシュが存在する場合、mod_cache_disk を介してディスクからキャッシュデータを読み出し、クライアントにレスポンスを返します。 3. キャッシュが存在しない、または期限切れの場合は、オリジンサーバーやバックエンドからコンテンツを取得し、mod_cache_disk を介してディスクに書き込みます。

共有オブジェクトキャッシュ（socache）のフロー

1. SSLハンドシェイク時などに、セッションデータの保存要求が発生します。 2. ap_socache_provider_t インターフェースを介して、mod_socache_shmcb の store 関数が呼び出されます。 3. mod_socache_shmcb は、共有メモリ内のサイクリックバッファにデータをシリアライズして格納します。 4. 次回以降の接続時に retrieve 関数が呼び出され、共有メモリから高速にデータが復元されます。

重要な不変条件

  • **API契約の遵守**:

    • すべてのsocacheプロバイダは、include/ap_socache.h で定義されている ap_socache_provider_t のライフサイクルおよびデータ操作のインターフェースを厳格に遵守しなければなりません。

  • **排他制御とスレッドセーフティ**:

    • mod_socache_shmcb などの共有メモリプロバイダは、複数プロセス・複数スレッドからの同時アクセスに対して、データの破損を防ぐための適切なロックおよびバッファ境界の整合性を維持する必要があります。

  • **ディスクキャッシュのメタデータ整合性**:

    • mod_cache_disk は、ディスク上に書き込まれるメタデータファイル（.header）とデータファイル（.data）の対応関係および一貫性を維持しなければなりません。

よくある落とし穴

  • **ディスクI/Oのボトルネック**:

    • mod_cache_disk を使用する場合、高負荷環境下ではディスクの書き込み・読み込み速度がボトルネックとなり、キャッシュによるパフォーマンス向上の効果が相殺されることがあります。

  • **共有メモリのサイズ制限とエビクション**:

    • mod_socache_shmcb は固定サイズの共有メモリセグメントを使用します。キャッシュ対象（SSLセッションなど）が急増してバッファが溢れた場合、古いエントリが強制的に上書き（エビクション）され、キャッシュミス率が上昇する原因になります。

  • **プロバイダの混同**:

    • HTTPレスポンスキャッシュ（mod_cache）と、SSLセッション等に使われるオブジェクトキャッシュ（socache）は、設定方法や目的が全く異なります。これらを混同して設定を適用しないよう注意が必要です。
