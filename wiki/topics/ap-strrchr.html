ap_strrchr

概念

ap_strrchr は、標準 C ライブラリの strrchr 関数をラップし、Apache HTTP Server のビルド環境において一貫した文字列検索インターフェースを提供する関数です。指定された文字列の末尾から特定の文字を検索します。

なぜ重要か

Apache HTTP Server 内の多様なモジュール（mod_deflate、mod_rewrite、mod_proxy_fcgi など）やコアコンポーネント（server/core.c）において、一貫した文字列操作 API を提供するために重要です。また、デバッグビルドや特定のプラットフォームにおけるマクロ定義の競合を回避・制御する役割も担っています。

関与するモジュール

  • module:include （ヘッダー定義の提供）
  • module:server （ユーティリティ実装の提供）

主要エンティティ

  • **symbol:include/httpd.h#ap_strrchr**: 関数宣言、または環境に応じたマクロ定義を提供します。
  • **symbol:server/util_debug.c#ap_strrchr**: 標準ライブラリの strrchr を直接呼び出す実体としてのラッパー関数です。

エンドツーエンドの流れ

1. 呼び出し元（mod_rewrite.c や core.c など）が、検索対象のヌル終端文字列 s と検索文字 c を指定して ap_strrchr(s, c) を呼び出します。 2. server/util_debug.c 内の実装（またはマクロ展開）を経由し、標準ライブラリの strrchr(s, c) が呼び出されます。 3. 文字列内で最後に出現した文字 c へのポインタが返されます。文字が見つからない場合は NULL が返されます。

※ なお、ビルド構成（デバッグモードやマクロの有効化状態）によって、マクロ展開と実関数呼び出しのどちらが優先されるかなど、コンパイル時の完全な解決フローの詳細は現時点では判明していません。

重要な不変条件

  • **有効な入力**: 引数 s は、有効なヌル終端文字列を指すポインタでなければなりません。
  • **メモリ管理の非関与**: 本関数はメモリの割り当てや解放を行わず、渡されたポインタのライフサイクルには一切関与しません。

よくある落とし穴

  • **NULL ポインタの伝播**: s に NULL を渡した場合、標準ライブラリの strrchr の挙動に従うため、未定義動作（一般的にはセグメンテーション違反）を引き起こします。呼び出し前に NULL チェックを行う必要があります。
  • **マクロの競合**: httpd.h 内で strrchr がマクロとして定義されている場合があり、server/util_debug.c では競合を避けるために一度 #undef してから再定義する処理が行われています。ヘッダーのインクルード順序やマクロ定義の干渉に注意が必要です。
  • **const 性の欠如**: 本関数は char * を受け取り char * を返します。const char * を安全に扱う必要がある場合は、類似の型安全なバリアントである ap_strrchr_c を使用してください。

次に読む場所

  • include/httpd.h （マクロ定義および関数宣言の確認）
  • server/util_debug.c （ap_strrchr の実装と #undef 周りの処理の確認）
