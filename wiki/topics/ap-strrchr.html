ap_strrchr

概念

ap_strrchr は、標準Cライブラリの strrchr をラップし、Apache HTTP Server 内で一貫した文字列検索インターフェースを提供するユーティリティ関数です。指定されたヌル終端文字列内を末尾から検索し、指定された文字が最後に出現する位置へのポインタを返します。

なぜ重要か

Apache HTTP Server のビルド環境において、プラットフォーム間の一貫性を確保するために重要です。また、httpd.h 内で strrchr マクロとして定義される場合があり、デバッグや特定のビルド構成におけるシンボルの衝突を防ぐ役割を担っています。類似の型安全なバリアントとして、const char * を扱う ap_strrchr_c も提供されています。

関与するモジュール

  • module:include
  • module:server

主要エンティティ

  • **symbol:include/httpd.h#ap_strrchr**: マクロ定義やプロトタイプ宣言を提供するヘッダー。
  • **symbol:server/util_debug.c#ap_strrchr**: 標準ライブラリの strrchr を直接呼び出す実体定義。

エンドツーエンドの流れ

1. **呼び出し**: modules/filters/mod_deflate.c、modules/mappers/mod_rewrite.c、modules/proxy/mod_proxy_fcgi.c、server/core.c などの各モジュールから ap_strrchr(s, c) が呼び出されます。 2. **ラッパーの実行**: server/util_debug.c 内で定義された ap_strrchr が呼び出されます。このファイル内では、マクロ競合を避けるために一度 strrchr が undef された上で、標準ライブラリの strrchr(s, c) が実行されます。 3. **返却**: 検索文字が見つかった場合はそのポインタを返し、見つからない場合は NULL を返します。

*注意: ビルド構成（デバッグビルドなど）ごとのマクロ展開の優先順位や、すべての呼び出し元における詳細なエラーハンドリングの全容は完全には判明していません。*

重要な不変条件

  • **入力の妥当性**: 引数 s は有効なヌル終端文字列ポインタでなければなりません。
  • **メモリ管理の非関与**: 本関数はメモリの割り当てや解放を行わず、渡されたポインタのライフサイクルには一切関与しません。

よくある落とし穴

  • **NULL ポインタの伝播**: 引数 s に NULL を渡した場合、標準ライブラリの strrchr の挙動に従うため、未定義動作（通常はセグメンテーション違反）を引き起こす可能性があります。
  • **マクロの競合**: httpd.h 内で strrchr がマクロとして再定義されることがあるため、実装ファイル（server/util_debug.c）のように適切に undef 処理を行わないと、コンパイルエラーや意図しない再帰呼び出しが発生する危険性があります。

次に読む場所

  • include/httpd.h (マクロ定義およびプロトタイプ宣言の確認)
  • server/util_debug.c (標準ライブラリ呼び出しの実装と undef 処理の確認)
