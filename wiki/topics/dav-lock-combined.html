dav_lock_combined

概念

dav_lock_combined は、WebDAVのロック情報をメモリ上で一元管理するための統合構造体（コンテナ構造体）です。公開用のロック情報（dav_lock）、内部管理用のプライベート情報（dav_lock_private）、およびロックを一意に識別するトークン（dav_locktoken）を1つのメモリブロックにまとめ、効率的なデータ操作を可能にします。

なぜ重要か

WebDAVのロック管理において、公開インターフェース用のデータと、ファイルシステムやロックプロバイダ固有の内部状態、および一意なUUIDトークンは密接に関連しています。これらを個別に管理するのではなく、dav_lock_combined として1つの構造体に統合することで、メモリ割り当ての効率化や、DBMなどの永続化ストレージから読み込まれたデータのデシリアライズ・マッピング処理を簡素化できます。また、mod_dav のロックプロバイダインターフェースとの整合性を保つ役割も担っています。

両ファイルにおいて、以下の形状を持つ構造体として定義されています。

typedef struct
{
    dav_lock pub;
    dav_lock_private priv;
    dav_locktoken token;
} dav_lock_combined;

エンドツーエンドの流れ

DBMなどの永続化ストレージからロックレコードが読み込まれる際（例: dav_fs_load_lock_record などの関数による処理）、バイナリデータがデシリアライズされ、この dav_lock_combined 構造体にマッピングされます。メモリ上では、ロック情報をリンクドリストとして構築する際のノードなどとして機能し、ロックの解決や検証、生成時に参照されます。

ただし、リクエストの受信からロックの検証、永続化、レスポンス返却に至る完全なエンドツーエンドの制御フローの詳細は、現時点の提供情報からは完全には明らかになっていません。

重要な不変条件

  • pub フィールド（dav_lock 型）は、mod_dav が要求する公開インターフェースと互換性を持つ必要があります。
  • token フィールド（dav_locktoken 型）は、UUIDに基づく一意な識別子である必要があります。
  • 各フィールド（pub, priv, token）が適切に初期化されている必要があります。
  • メモリ割り当ては、関連する request_rec のプールから行われることが想定されています。

よくある落とし穴

  • **メモリレイアウトの不整合**: 本構造体は modules/dav/fs/lock.c と modules/dav/lock/locks.c の両方で個別に定義されています。構造体のメンバやレイアウトを変更する場合、両方のファイルおよび mod_dav 側の期待するデータレイアウトとの整合性を慎重に考慮しないと、メモリ破壊や不正なデータ読み込みが発生する可能性があります。
  • **初期化不足**: 構造体内の各フィールドが正しく初期化されていない場合、ロックの解決や検証時に不整合が生じる可能性があります。
  • **メモリ割り当て失敗**: apr_palloc 等によるメモリ割り当てが失敗した場合、構造体の生成自体が失敗します。
