dav_lock_combined

概念

dav_lock_combined は、WebDAVのロック情報をメモリ上で保持するための統合構造体（コンテナ構造体）です。公開用のロック情報（dav_lock）、内部管理用のプライベート情報（dav_lock_private）、およびロック識別子（dav_locktoken）を1つのメモリブロックにまとめ、効率的なデータ操作を可能にします。

なぜ重要か

この構造体は、WebDAVのロックプロバイダインターフェース（mod_dav）と密接に関連しています。DBMなどの永続ストレージから読み込まれたバイナリデータをデシリアライズしてメモリ上にマッピングする際のターゲットとして機能し、メモリ管理の効率化とロック情報の解決・検証プロセスの簡素化を実現します。

両ファイルにおいて、以下のシグネチャを持つ構造体として定義されています。

typedef struct
{
    dav_lock pub;
    dav_lock_private priv;
    dav_locktoken token;
} dav_lock_combined;

エンドツーエンドの流れ

1. **データの読み込みとマッピング**: dav_fs_load_lock_record などの関数により、DBM等の永続ストレージからロックレコードが読み込まれます。 2. **デシリアライズ**: 読み込まれたバイナリデータがデシリアライズされ、dav_lock_combined 構造体にマッピングされます。 3. **メモリ上での管理**: メモリ上でリンクドリストのノードなどとして構築され、ロックの解決や検証、属性へのアクセスに利用されます。

※ なお、具体的なHTTPリクエストの受信からストレージへの保存・読み込みに至る完全なエンドツーエンドの制御フローの全容は、提供された情報からはまだ完全には判明していません。

重要な不変条件

  • **インターフェース互換性**: pub フィールド（dav_lock）は、mod_dav が要求する公開インターフェースと互換性を持つ必要があります。
  • **一意性**: token フィールド（dav_locktoken）は、UUIDに基づく一意な識別子であることが期待されます。
  • **メモリ管理**: メモリ割り当ては、関連する request_rec のプール（apr_palloc 等）から行われることが想定されています。

よくある落とし穴

  • **重複定義とメモリレイアウトの不整合**: 本構造体は modules/dav/fs/lock.c と modules/dav/lock/locks.c の両方で定義されています。構造を変更する場合は、これら両方のファイルにおけるメモリレイアウトへの影響や、mod_dav 側の期待するデータレイアウトとの整合性を考慮する必要があります。整合性が取れない場合、メモリ破壊や不正なデータ読み込みが発生する可能性があります。
  • **初期化不足**: 構造体内の各フィールドが正しく初期化されていない場合、ロックの解決や検証時に不整合が生じる原因となります。
