read_link

概念

read_link は、Apache HTTP Server（httpd）の複数のモジュール（mod_http2 および mod_md）において、HTTPの Link ヘッダーなどのリンク情報を解析・処理するために定義されている内部的な静的（static）関数です。共通のデータ構造である link_ctx をコンテキストとして受け取り、文字の読み込みやスキップを行いながらリンクのパースを行います。

なぜ重要か

この関数は、HTTP/2におけるサーバープッシュ（Server Push）やプロキシ時のリンクヘッダーの逆マッピング（Reverse Mapping）、およびManaged Domains（ACME等の証明書管理）におけるリンクヘッダーの処理など、プロトコルレベルでのメタデータ解析において重要な役割を果たします。それぞれのモジュールで独立して定義されていますが、同様のシグネチャとパースロジックを共有しています。

関与するモジュール

  • module:modules/http2
  • module:modules/md

主要エンティティ

  • **symbol:modules/http2/h2_proxy_util.c#read_link**: HTTP/2 プロキシユーティリティにおける Link ヘッダー解析用関数。
  • **symbol:modules/http2/h2_push.c#read_link**: HTTP/2 サーバープッシュにおける Link ヘッダー解析用関数。
  • **symbol:modules/md/md_util.c#read_link**: Managed Domains ユーティリティにおける Link ヘッダー解析用関数。

これらの関数はすべて static int read_link(link_ctx *ctx) というシグネチャを持ちます。

エンドツーエンドの流れ

現時点では、詳細なパースアルゴリズムや、呼び出し元から呼び出し先への完全なエンドツーエンドの制御フローは完全には解析されていません。 ただし、静的解析情報から以下の呼び出し関係が判明しています。

1. **呼び出し元**: 各モジュールの find_url、h2_proxy_link_reverse_map、または inspect_link から read_link が呼び出されます。 2. **内部処理（呼び出し先）**: read_link 内部では、コンテキスト link_ctx を走査するために、以下のヘルパー関数が呼び出されます。

  • find_chr (文字の検索)
  • read_chr (文字の読み込み)
  • skip_ws (空白文字のスキップ)
  • mk_str (h2_push.c のみ、文字列の生成)

重要な不変条件

現時点では、link_ctx の状態遷移や、パース処理における具体的な前提条件・不変条件は解析されておらず、未確定です。

よくある落とし穴

各モジュール（h2_proxy_util.c、h2_push.c、md_util.c）に同名かつほぼ同一のシグネチャを持つ static 関数として個別に重複して実装されているため、共通のバグ修正や仕様変更を行う際に、1箇所の修正だけでは他のモジュールに反映されないというメンテナンス上の落とし穴があります。また、具体的なエラーハンドリングや境界条件の挙動の違いについては、現時点では明らかになっていません。
