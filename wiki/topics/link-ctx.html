link_ctx

概念

link_ctx は、Apache HTTP Server（httpd）の複数モジュールにおいて、RFC 5988 に準拠した HTTP Link ヘッダーのパースや処理を行う際に、解析状態や対象文字列、抽出されたパラメータなどを保持するために使用されるコンテキスト構造体です。

同じ link_ctx という名前の構造体が、modules/http2 および modules/md 内の異なるソースファイルで個別に定義されており、それぞれの用途（リバースプロキシでのURI書き換え、HTTP/2サーバープッシュ、MD関連のリンク解析など）に応じた固有のメンバを持っています。

なぜ重要か

HTTP Link ヘッダーは、リバースプロキシ時のURI書き換え（Link ヘッダー内のURLの調整）や、HTTP/2におけるサーバープッシュ（rel=preload の検出）など、プロトコル制御において重要な役割を果たします。 link_ctx は、これらの複雑なヘッダー文字列を再帰なしのステートマシンとして安全かつ効率的に走査・解析するための一時的な状態管理コンテキストとして機能し、パース処理の共通化や状態追跡を容易にします。

関与するモジュール

主要エンティティ

  • **symbol:modules/http2/h2_proxy_util.c#link_ctx**: プロキシ環境下での Link ヘッダーのパースおよびURI書き換え（リバースマッピング）用のコンテキスト。
  • **symbol:modules/http2/h2_push.c#link_ctx**: HTTP/2 サーバープッシュ候補（rel=preload など）を抽出するためのコンテキスト。固定長バッファ b[4096] を内包します。
  • **symbol:modules/md/md_util.c#link_ctx**: Link ヘッダーからリンク要素やパラメータのオフセット情報（開始位置と長さ）を抽出するための軽量なコンテキスト。

エンドツーエンドの流れ

各モジュールにおける具体的な処理フローは以下の通りですが、モジュール間を跨ぐ統一的な単一のフローや、異なる link_ctx 構造体同士が直接メモリ上でキャストされて相互作用する詳細な結合メカニズムの全容は、提供された情報からは完全には明らかではありません。

1. **プロキシURI書き換え（h2_proxy_util.c）**:

  • h2_proxy_link_reverse_map（想定される呼び出し元）から処理が開始されます。
  • link_ctx に対象文字列 s やプロキシ設定（server_uri など）をセットします。
  • skip_ws や read_link などのパース関数群が link_ctx のインデックス i を進めながらトークンを解析し、subst_str を用いてURIを書き換えます。

2. **サーバープッシュ候補の収集（h2_push.c）**:

  • h2_push_collect() 等から呼び出され、link_ctx を初期化します。
  • h2_proxy_util.c 内のユーティリティ関数（skip_ws, read_link 等）を利用して Link ヘッダーを走査します。
  • 抽出されたプッシュ候補は pushes 配列に、パラメータは params テーブルに格納されます。

3. **MDユーティリティでの解析（md_util.c）**:

  • Link ヘッダーを解析し、各要素の開始位置（_start）と長さ（_len）を link_ctx に記録します。
  • h2_proxy_util.c 内の複数の静的関数（skip_ws 等）がこの構造体を引数として受け取り、解析状態を操作します。

重要な不変条件

  • **境界チェックの維持**: すべての link_ctx において、現在の走査位置を示すインデックス i は、常に 0 から対象文字列の長さ（slen）の範囲内でなければなりません。各パース関数は、この範囲を超えたアクセス（バッファオーバーラン）が発生しないように設計されています。
  • **メモリプールの有効性**: pool メンバを持つ構造体（h2_proxy_util.c および h2_push.c の link_ctx）では、メモリ管理のために有効な apr_pool_t が割り当てられている必要があります。

よくある落とし穴

  • **同名構造体の混同**: link_ctx は3つのファイルで個別に定義されており、それぞれメンバ構成が異なります。特に h2_proxy_util.c のパース関数群が h2_push.c や md_util.c で定義された link_ctx を引数として受け取る設計になっているため、構造体の定義変更を行う際は、すべての関連ファイル（h2_proxy_util.c, h2_push.c, md_util.c）における整合性を慎重に確認する必要があります。
  • **固定長バッファの制限**: h2_push.c の link_ctx には char b[4096] という固定長バッファが含まれています。解析対象のトークンや文字列がこのサイズを超える場合の境界条件やエラーハンドリングに注意が必要です。
