link_ctx

概念

link_ctx は、Apache HTTP Server（httpd）の複数モジュールにおいて、RFC 5988 に基づく HTTP Link ヘッダーの構文解析や処理を行う際に、解析状態や対象文字列、抽出されたパラメータなどを保持するために使用されるコンテキスト構造体です。

同名の構造体が複数のファイル（h2_proxy_util.c、h2_push.c、md_util.c）で個別に定義されており、それぞれプロキシの URI 書き換え、HTTP/2 サーバープッシュ、ACME/MD（Managed Domains）関連の処理など、用途に応じた異なるフィールドを持っています。

なぜ重要か

HTTP Link ヘッダーは、リバースプロキシ環境におけるバックエンド URI の書き換え（h2_proxy_util.c）や、HTTP/2 におけるサーバープッシュ（h2_push.c）のトリガー（rel=preload など）、および MD モジュール（md_util.c）でのメタデータ解析など、プロトコル制御において重要な役割を果たします。

link_ctx は、これらの複雑なヘッダー文字列を再帰なしのステートマシンとして安全かつ効率的に走査・パースするための中核的なデータ構造として機能します。

関与するモジュール

主要エンティティ

  • **symbol:modules/http2/h2_proxy_util.c#link_ctx**: プロキシ環境下での Link ヘッダーの URI 書き換え用コンテキスト。リクエスト情報（request_rec）やプロキシ設定（proxy_dir_conf）、書き換え前後の URI 情報を保持します。
  • **symbol:modules/http2/h2_push.c#link_ctx**: HTTP/2 サーバープッシュ候補を抽出するためのコンテキスト。抽出されたプッシュ候補の配列（pushes）や、一時作業用バッファ b[4096] を内包します。
  • **symbol:modules/md/md_util.c#link_ctx**: Link ヘッダーの解析におけるオフセット情報（開始位置や長さ）を保持するための軽量なコンテキスト。

エンドツーエンドの流れ

提供された情報からは、これら 3 つの異なる link_ctx 構造体がシステム全体でどのように相互作用するかの完全なエンドツーエンドのフローは明らかになっていません。ただし、部分的な流れとして以下が確認されています：

1. **プロキシ URI 書き換え**: h2_proxy_link_reverse_map（想定される呼び出し元）から、h2_proxy_util.c の link_ctx を用いて Link ヘッダーがパースされ、subst_str により URI が書き換えられます。 2. **サーバープッシュ**: h2_push_collect() 等から h2_push.c の link_ctx が初期化され、inspect_link などの解析関数を通じてプッシュ候補（pushes 配列）が抽出されます。 3. **共通解析処理**: 提供された情報によると、h2_proxy_util.c 内の静的パース関数群（skip_ws, find_chr, read_chr, read_link 等）が、h2_push.c や md_util.c で定義された link_ctx を引数として受け取り、解析状態を操作する関係にあるとされています。

重要な不変条件

  • **インデックスの境界**: すべての link_ctx において、現在の走査位置を示すインデックス i は、常に 0 から解析対象文字列の長さ slen の範囲内でなければなりません。
  • **メモリ管理**: pool フィールドを持つ構造体（h2_proxy_util.c および h2_push.c の link_ctx）では、メモリ確保のために有効な apr_pool_t が設定されている必要があります。
  • **バッファ制限**: h2_push.c の link_ctx に含まれる一時作業用バッファ b は 4096 バイトの固定長であり、これを超えるトークンの処理には制限が生じる可能性があります。

よくある落とし穴

  • **構造体の混同**: link_ctx という名前の構造体は 3 つの異なるファイルで個別に定義されており、それぞれメンバ構成が異なります。これらを跨ぐ関数呼び出しや参照を行う際は、型定義の不一致やキャストによる不具合に注意する必要があります。
  • **バッファオーバーフロー/境界チェック**: h2_push.c の固定長バッファ b[4096] や、各パース関数におけるインデックス i の境界チェックを誤ると、メモリ破損や不正アクセスの原因となります。
  • **メモリ枯渇時の挙動**: apr_pool_t を用いた動的メモリ確保（subst_str など）において、メモリが枯渇した場合の失敗モードの挙動は、提供された情報からは不明です。

次に読む場所

  • **modules/http2/h2_push.c**: HTTP/2 サーバープッシュにおける Link ヘッダー解析とプッシュ候補抽出の実装。
  • **modules/md/md_util.c**: MD モジュールにおける Link ヘッダー解析用の軽量コンテキスト定義。
