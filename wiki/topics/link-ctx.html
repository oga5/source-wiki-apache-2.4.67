link_ctx

概念

link_ctx は、Apache HTTP Server 内の複数のモジュール（mod_http2 および mod_md）において、RFC 5988 に基づく HTTP Link ヘッダーのパースや解析、およびそれに伴う URI 書き換えやサーバープッシュ候補の抽出を行うために定義されているコンテキスト構造体です。

同名の構造体が複数のファイルで個別に定義されており、それぞれのユースケース（プロキシ書き換え、HTTP/2 サーバープッシュ、ACME/MD 関連のリンク解析など）に応じたフィールドを持っていますが、共通して「解析対象文字列（s）」「文字列長（slen）」「現在の走査位置（i）」などの状態を保持するステートマシン用のコンテキストとして機能します。

なぜ重要か

HTTP Link ヘッダーは、リバースプロキシ環境におけるバックエンド URI の書き換え（mod_proxy_html や mod_http2 のプロキシ連携）や、HTTP/2 における rel=preload を用いたサーバープッシュのトリガー、ACME プロトコル（mod_md）における各種リンク情報の抽出など、多岐にわたる機能で利用されます。 link_ctx は、これらの複雑なヘッダー文字列を安全かつ効率的にトークン化・パースし、状態を維持しながら処理を進めるための共通的なパターンを提供しているため重要です。

関与するモジュール

  • module:modules/http2
  • module:modules/md

主要エンティティ

  • **symbol:modules/http2/h2_proxy_util.c#link_ctx**: プロキシ環境下での Link ヘッダーのパースおよび URI 書き換え（リバースマップ）用のコンテキスト。
  • **symbol:modules/http2/h2_push.c#link_ctx**: HTTP/2 サーバープッシュ（rel=preload）の候補を抽出するためのコンテキスト。一時バッファ b[4096] やプッシュ候補の配列 pushes を保持します。
  • **symbol:modules/md/md_util.c#link_ctx**: Link ヘッダーからリンク要素やパラメータ（名前と値）の開始位置と長さを抽出するための軽量なコンテキスト。

エンドツーエンドの流れ

各モジュールにおける詳細なエンドツーエンドの呼び出しフロー全体は完全には明らかになっていませんが、提供された情報から以下の部分的な流れが判明しています。

1. **プロキシ URI 書き換え (h2_proxy_util.c)**:

  • h2_proxy_link_reverse_map（推測される呼び出し元）等からパース処理が開始されます。
  • link_ctx に対象文字列やプロキシ設定（server_uri、real_backend_uri など）を格納します。
  • skip_ws、read_link、subst_str などのパース関数群がこのコンテキストを更新しながら走査し、URI の置換を行います。

2. **HTTP/2 サーバープッシュ (h2_push.c)**:

  • h2_push_collect() 等の関数により、レスポンスヘッダー内の Link ヘッダー文字列が link_ctx にセットされます。
  • inspect_link などの解析関数が、h2_proxy_util.c 内のユーティリティ関数（skip_ws、read_link など）を呼び出してパースを行います。
  • 抽出されたプッシュ候補が pushes 配列に格納されます。

3. **MD ユーティリティ (md_util.c)**:

  • Link ヘッダーの解析関数（read_link 等）に link_ctx が渡されます。
  • h2_proxy_util.c 内の複数の静的関数（skip_ws、read_chr、skip_param など）がこの構造体を引数として受け取り、解析状態を操作してパラメータ名や値のオフセット（pn_start、pv_start など）を特定します。

重要な不変条件

  • **インデックスの境界条件**: すべての link_ctx において、現在の走査位置を示すインデックス i は、常に 0 から文字列長 slen（または slen に対応する型）の範囲内でなければなりません。
  • **メモリプール**: h2_proxy_util.c および h2_push.c の link_ctx において、pool フィールドは有効な apr_pool_t を指している必要があります。
  • **文字列の有効性**: s は常に解析対象の有効な Link ヘッダー文字列を指している必要があります。

よくある落とし穴

  • **同名構造体の混同と相互依存**: link_ctx は h2_proxy_util.c、h2_push.c、md_util.c のそれぞれで個別に定義されています。しかし、h2_proxy_util.c 内の静的関数群が md_util.c や h2_push.c で定義された link_ctx を引数として受け取って処理する設計になっている箇所があります。構造体の定義を変更する際は、これら複数ファイル間での整合性が崩れないよう極めて慎重に行う必要があります。
  • **固定長バッファの溢れ**: h2_push.c の link_ctx には固定長バッファ b[4096] が定義されています。解析対象のトークンや文字列がこのサイズを超える場合の境界条件やエラーハンドリングに注意が必要です。
