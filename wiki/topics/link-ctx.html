link_ctx

概念

link_ctx は、Apache HTTP Server 内の複数のモジュール（mod_http2 および mod_md）において、HTTP Link ヘッダー（RFC 5988）の構文解析や、それに伴う処理（URIの書き換えやサーバープッシュ候補の抽出など）を行う際に、解析状態や対象文字列のメタデータを保持するために定義されている構造体です。

同名の構造体が複数のソースファイルで個別に定義されていますが、いずれも Link ヘッダー文字列を走査・解析するためのステートフルなコンテキストとして機能します。

なぜ重要か

HTTP Link ヘッダーは、リバースプロキシ環境におけるバックエンドURIからフロントエンドURIへの書き換え（mod_proxy_html や mod_http2 のプロキシ機能）や、HTTP/2 におけるサーバープッシュ（rel=preload の検出）など、複数の重要な機能でパースする必要があります。 link_ctx を用いることで、再帰を使用しない軽量なステートマシンとして文字列を安全かつ効率的に走査し、メモリ管理や境界チェックを一元化できます。

関与するモジュール

  • module:modules/http2

主要エンティティ

  • **symbol:modules/http2/h2_proxy_util.c#link_ctx**: リバースプロキシ環境下での Link ヘッダーのパースおよび URI 書き換え処理用のコンテキスト。
  • **symbol:modules/http2/h2_push.c#link_ctx**: HTTP/2 サーバープッシュ機能において、Link ヘッダーからプッシュ候補（rel=preload など）を抽出するためのコンテキスト。
  • **symbol:modules/md/md_util.c#link_ctx**: Link ヘッダーの解析において、抽出されたリンク要素やパラメータのオフセット情報を保持するための汎用的なコンテキスト。

エンドツーエンドの流れ

詳細なエンドツーエンドの呼び出しフロー全体は完全には判明していませんが、提供された情報から以下の部分的な流れが確認できます。

1. **プロキシ時のURI書き換え (h2_proxy_util.c)**:

  • h2_proxy_link_reverse_map（推測される呼び出し元）等からパース処理が開始されます。
  • link_ctx に対象の Link ヘッダー文字列 s や書き換え用のベース URI 情報（server_uri など）を格納します。
  • skip_ws, read_link, subst_str などのユーティリティ関数群が link_ctx を更新しながら走査し、URI の置換を行います。

2. **サーバープッシュ候補の抽出 (h2_push.c)**:

  • h2_push_collect() 等の関数により、レスポンスヘッダー内の Link 文字列が link_ctx にセットされます。
  • inspect_link などの解析関数が、h2_proxy_util.c 内の共通ユーティリティ（skip_ws, read_link など）を呼び出してパースを進めます。
  • 解析結果として、抽出されたプッシュ候補は pushes 配列に格納され、パラメータは params テーブルに保持されます。

3. **MDユーティリティによる解析 (md_util.c)**:

  • Link ヘッダーの解析関数（read_link 等）が link_ctx を引数として受け取り、文字列内のリンクやパラメータの開始位置・長さをオフセット情報（link_start, link_len など）として書き込みます。
  • h2_proxy_util.c 内の静的関数群（skip_ws, read_link など）がこの構造体を引数として受け取り、解析状態を操作します。

重要な不変条件

  • **インデックスの境界**: 解析位置を示すインデックス（i）は、常に 0 から対象文字列の長さ（slen）の範囲内でなければなりません。
  • **メモリ管理**: pool フィールドが存在する場合、それは有効な apr_pool_t を指している必要があります。
  • **オフセットの整合性**: md_util.c の link_ctx において、各 _start および _len フィールドは、対象文字列 s 内の有効な部分文字列を指す必要があります。

よくある落とし穴

  • **バッファオーバーフローの懸念**: h2_push.c の link_ctx は char b[4096] という固定長の一時作業用バッファを保持しています。解析対象のトークンや文字列がこのサイズを超える場合、境界条件の処理に注意が必要です。
  • **モジュール間の密結合**: modules/md/md_util.c や modules/http2/h2_push.c で定義されている link_ctx は、modules/http2/h2_proxy_util.c 内の関数群からも参照・操作されています。構造体の定義を変更する際は、これら複数のファイルにまたがる解析ロジックの整合性を崩さないよう注意する必要があります。
