link_ctx

概念

link_ctx は、Apache HTTP Server（httpd）の複数モジュール（mod_http2 および mod_md）において、RFC 5988 等に基づく HTTP Link ヘッダーの構文解析（パース）や、それに伴う URI 書き換え、サーバープッシュ（Server Push）候補の抽出を行うための状態（コンテキスト）を保持する構造体です。

同名の構造体が複数のファイルで定義されており、それぞれ用途に応じたフィールド構成を持っていますが、いずれも文字列の走査位置や解析中の一時データを管理するステートマシン用のコンテキストとして機能します。

なぜ重要か

HTTP Link ヘッダーは、リバースプロキシ環境下での URI 書き換え（mod_proxy 連携）や、HTTP/2 における rel=preload を用いたサーバープッシュのトリガー、ACME プロトコル等（mod_md）でのリンク関係の解決など、複数の重要な機能で利用されます。

link_ctx を用いてパース処理の状態をカプセル化することで、再帰呼び出しを避けた安全な逐次走査（ステートマシン）を実現し、バッファオーバーフローなどの脆弱性を防ぎつつ、複雑なヘッダー解析ロジックを共通のユーティリティ関数群（skip_ws, read_link など）で処理できるようにしています。

エンドツーエンドの流れ

各コンテキストにおける具体的なエンドツーエンドの制御フローの全容（特にモジュール間を跨ぐ完全な呼び出しシーケンスや、どのタイミングでどの構造体が相互変換・参照されるか）は、提供された情報からは完全には明らかになっていません。現時点で判明している部分的な流れは以下の通りです。

1. **プロキシ書き換え (h2_proxy_util.c)**: h2_proxy_link_reverse_map（推測される呼び出し元）等からパース処理が開始され、link_ctx に対象文字列 s やプロキシ設定、書き換え用 URI 情報を格納します。その後、skip_ws や read_link などの解析関数群がコンテキスト内のインデックス i を進めながらパースし、subst_str を用いて URI を書き換えます。 2. **サーバープッシュ (h2_push.c)**: h2_push_collect() 等により link_ctx が初期化され、Link ヘッダー文字列がセットされます。h2_proxy_util.c のユーティリティ関数（skip_ws, read_link 等）を呼び出して解析を行い、抽出されたプッシュ候補を pushes 配列に、パラメータを params テーブルに格納します。 3. **MD ユーティリティ (md_util.c)**: Link ヘッダーを解析する際、link_ctx に文字列 s と長さ slen をセットし、read_link 等の関数を介してリンクやパラメータの開始位置（_start）と長さ（_len）を特定します。

重要な不変条件

  • **境界チェックの不変条件**:

    • すべての link_ctx において、現在の走査位置を示すインデックス i は、常に 0 以上かつ解析対象文字列の長さ（slen または slen 相当）以下（0 <= i <= slen）でなければなりません。各パース関数はこの不変条件を維持するようにインデックスをチェックします。

  • **メモリ管理の不変条件**:

    • h2_proxy_util.c および h2_push.c の link_ctx において、pool フィールドは有効な apr_pool_t を指していなければなりません。

  • **バッファ制限**:

    • h2_push.c の link_ctx には固定長バッファ char b[4096] が定義されており、このサイズを超えるトークンや文字列を処理する際の境界条件が維持される必要があります。

よくある落とし穴

  • **定義の重複と相互参照**:

    • link_ctx は h2_push.c、h2_proxy_util.c、md_util.c でそれぞれ個別に定義されていますが、h2_proxy_util.c の解析関数群が他方の構造体（h2_push.c や md_util.c の link_ctx）を引数として受け取り操作するような密結合な関係が見られます。いずれかの構造体の定義やフィールド（特に i や s などの共通的な役割を持つメンバ）を変更する場合、他方のファイルや解析ロジックに予期せぬ影響を与えないよう、両方の整合性を慎重に確認する必要があります。

  • **固定バッファの溢れ**:

    • h2_push.c 内の b[4096] は固定長であるため、極端に長い Link ヘッダーやパラメータが渡された場合にバッファ制限に達する可能性があります。

  • **メモリリーク**:

    • パース中に apr_pool_t から動的にメモリを割り当てる（subst_str など）際、ループ処理や大量のヘッダー処理においてプールメモリが過剰に消費されないよう注意が必要です。

次に読む場所

  • **modules/http2/h2_proxy_util.c**: Link ヘッダーのパースおよび URI 書き換えを行う具体的な静的関数群（skip_ws, read_link, subst_str 等）の実装。
  • **modules/http2/h2_push.c**: HTTP/2 サーバープッシュにおける Link ヘッダー解析とプッシュ候補抽出の実装。
  • **modules/md/md_util.c**: mod_md における Link ヘッダー解析とオフセット管理の実装。
