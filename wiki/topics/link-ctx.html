link_ctx

概念

link_ctx は、Apache HTTP Server（httpd）の複数モジュールにおいて、HTTPの Link ヘッダー（RFC 5988等に基づく）をパース・解析し、その状態を管理するために定義されている構造体です。 同名の構造体が複数のファイル（h2_proxy_util.c、h2_push.c、md_util.c）で個別に定義されており、それぞれ用途に応じた異なるフィールド構成を持っていますが、いずれも Link ヘッダー文字列の走査位置や解析結果を保持するコンテキストとして機能します。

なぜ重要か

HTTP/2におけるサーバープッシュ（rel=preload の抽出など）や、リバースプロキシ環境下での Link ヘッダー内のURI書き換え処理、ACMEプロトコル等を取り扱うManaged Domains（mod_md）でのリンク解析など、プロトコル制御やコンテンツ配信の最適化に直結するヘッダー処理を、状態を保持しながら安全かつ段階的に行うために不可欠なデータ構造です。

主要エンティティ

  • **symbol:modules/http2/h2_proxy_util.c#link_ctx**: リバースプロキシ環境下での Link ヘッダーのURI書き換え処理用コンテキスト。
  • **symbol:modules/http2/h2_push.c#link_ctx**: HTTP/2 サーバープッシュ候補（Link ヘッダーから抽出）を収集・解析するためのコンテキスト。
  • **symbol:modules/md/md_util.c#link_ctx**: Link ヘッダーの要素やパラメータ（名前・値）のオフセット情報を保持する汎用的な解析コンテキスト。

エンドツーエンドの流れ

詳細なエンドツーエンドの全体フローは完全には明らかになっていませんが、提供された情報から以下の部分的な流れが確認できます。

1. **初期化**: 各処理（プロキシ書き換え、プッシュ候補収集、MDユーティリティ処理）の開始時に、対象の Link ヘッダー文字列（s）や長さ（slen）、メモリプール（pool）などを link_ctx にセットします。 2. **パース・走査**: skip_ws、read_chr、read_link などの解析関数群に link_ctx を引き渡し、現在のインデックス i を進めながらトークンを順次解析します。 3. **結果の適用**:

  • h2_proxy_util.c では、特定されたリンク範囲（link_start, link_end）を基に subst_str を用いてURIを書き換えます。
  • h2_push.c では、抽出されたプッシュ候補を pushes 配列に格納し、パラメータを params テーブルに保持します。
  • md_util.c では、パラメータ名や値の開始位置と長さを構造体内のオフセット情報（pn_start, pv_start 等）に書き込みます。

重要な不変条件

  • **インデックスの境界**: 現在の走査位置を示すインデックス i は、常に 0 から解析対象文字列の長さ slen の範囲内でなければなりません。
  • **メモリ管理**: pool フィールドが存在する場合、それは有効な apr_pool_t ポインタでなければなりません。
  • **文字列の整合性**: link_ctx 内の各種オフセット（link_start、link_len、pn_start など）は、常に解析対象文字列 s の有効な範囲内を指す必要があります。

よくある落とし穴

  • **バッファオーバーフローの制約**: h2_push.c の link_ctx には固定長バッファ char b[4096] が定義されており、解析対象のトークンや文字列がこのサイズを超える場合の境界条件に注意が必要です。
  • **モジュール間の依存と重複**: link_ctx は h2_proxy_util.c、h2_push.c、md_util.c でそれぞれ個別に定義されていますが、ソースコードの記述によると、例えば md_util.c の定義が h2_proxy_util.c から参照されている、あるいは h2_push.c の処理が h2_proxy_util.c の関数を呼び出しているなど、相互に関連し合っています。いずれかの構造体定義やパースロジックを変更する際は、他方への影響を慎重に確認する必要があります。
