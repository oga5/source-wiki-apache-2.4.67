max_spare_threads

概念

max_spare_threads は、Apache HTTP Server のマルチプロセッシングモジュール（MPM）において、維持するアイドル状態（スペア）のワーカースレッドの最大数を制御するための設定値および変数です。

なぜ重要か

過剰なアイドルスレッドが維持されると、メモリやシステムリソースが無駄に消費されます。max_spare_threads を適切に設定することで、サーバーの負荷が下がった際に余分なスレッド（またはプロセス）を安全に削減し、システムリソースを効率的に回収することが可能になります。

関与するモジュール

  • module:server/mpm/mpmt_os2

主要エンティティ

  • **symbol:server/mpm/event/event.c#max_spare_threads**: event MPM における最大スペアスレッド数を保持する内部変数。
  • **symbol:server/mpm/event/event.c#set_max_spare_threads**: event MPM 用の設定ディレクティブハンドラ。
  • **symbol:server/mpm/mpmt_os2/mpmt_os2.c#set_max_spare_threads**: mpmt_os2 MPM 用の設定ディレクティブハンドラ。
  • **symbol:server/mpm/worker/worker.c#max_spare_threads**: worker MPM における最大スペアスレッド数を保持する内部変数。
  • **symbol:server/mpm/worker/worker.c#set_max_spare_threads**: worker MPM 用の設定ディレクティブハンドラ。

エンドツーエンドの流れ

1. **設定の解析**: サーバー起動時または再起動時に、設定ファイル中のディレクティブに応じて各 MPM の設定関数（set_max_spare_threads）が呼び出され、設定値が読み込まれます。 2. **変数の更新**: 読み込まれた値は、それぞれの MPM 内の変数（max_spare_threads など）に格納されます。 3. **スレッドの監視と削減**: 実行時、各 MPM の監視スレッドやプロセス管理ループが現在のアイドルスレッド数を監視します。アイドルスレッド数が max_spare_threads を超過した場合、余分なスレッドやプロセスを終了させる処理がトリガーされます。

*注意: アイドルスレッドのカウント方法や、実際にスレッド・プロセスを終了させる具体的なシグナル送信・回収処理の完全なエンドツーエンドのフローは、提供されたモジュール情報からは未解明です。*

重要な不変条件

  • 各 MPM（event, worker）において、max_spare_threads はシステムが維持すべき最小スペアスレッド数（min_spare_threads）以上である必要があります。

*注意: 設定値のバリデーションロジックや、他の設定値（ThreadsPerChild など）との具体的な相関関係・不変条件の詳細は、提供されたシンボル情報からは未解明です。*

よくある落とし穴

  • **MPM 間の挙動の差異**: event、mpmt_os2、worker の各 MPM で個別に設定関数や変数が定義されているため、使用している MPM によってスレッドのライフサイクル管理や削減の挙動が異なる場合があります。
  • **極端な設定値**: 値を小さく設定しすぎると、突発的なアクセス増加時にスレッドの生成が追いつかず、レスポンス遅延を招く可能性があります。逆に大きすぎると、リソースの解放が行われずメモリを圧迫します。
