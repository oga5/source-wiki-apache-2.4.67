Multi-Processing Modules (MPM) Architecture & Lifecycle

概念

**Multi-Processing Modules (MPM)** は、Apache HTTP Server においてネットワークポートのバインド、クライアントからの接続の受け入れ、およびそれらを処理する子プロセスやスレッドの管理を担当するコアアーキテクチャです。システム環境や要求されるパフォーマンス特性に応じて、異なる並行処理モデル（プロセス駆動、スレッド駆動、イベント駆動など）をモジュールとして差し替えることができます。

なぜ重要か

MPM は Web サーバーのスケーラビリティとリソース消費効率を決定づける最重要コンポーネントです。 例えば、event MPM は非同期イベント処理を導入して接続維持（Keep-Alive）のオーバーヘッドを最小化し、prefork はスレッドを使用せずプロセスのみで処理することで非スレッドセーフなライブラリとの安全な連携を可能にし、worker はマルチプロセスとマルチスレッドのハイブリッド構成により高スループットを実現します。これらを適切に制御・維持することは、サーバーの安定稼働に直結します。

関与するモジュール

  • module:server/mpm: MPM 全体の共通インターフェースや基盤。
  • module:server/mpm/event: イベント駆動型の並行処理モデルを実装するモジュール。
  • module:server/mpm/prefork: スレッドを使用せず、プロセスのみで並行処理を行うモデルを実装するモジュール。
  • module:server/mpm/worker: マルチプロセスとマルチスレッドを組み合わせたハイブリッドモデルを実装するモジュール。

主要エンティティ

提供されたソース情報に基づく主要なエンティティは以下の通りです：

  • **symbol:server/mpm/event/event.c#child_main**: 子プロセスのメインループおよび初期化処理を担当する関数。シグナル処理のセットアップ、スコアボードの再オープン、およびワーカースレッドの生成などを行います。
  • **symbol:server/mpm/event/event.c#clean_child_exit**: 子プロセスが安全に終了するためのクリーンアップ処理を行い、終了ステータスを通知する関数。
  • **symbol:server/mpm/event/event.c#perform_idle_server_maintenance**: アイドル状態のサーバープロセスやスレッドの数を監視し、必要に応じて子プロセスの生成（make_child）や終了シグナルの送信（ap_mpm_podx_signal）を行うメンテナンス関数。
  • **symbol:server/mpm/event/event.c#set_server_limit**: 起動可能な最大プロセス数（ServerLimit）を設定するための設定コマンドハンドラ。
  • **symbol:server/mpm/event/event.c#set_thread_limit**: プロセスあたりの最大スレッド数（ThreadLimit）を設定するための設定コマンドハンドラ。

エンドツーエンドの流れ

提供されたシンボル情報からは、詳細なエンドツーエンドのライフサイクルフローの全容は完全には解明されていません。しかし、部分的な流れとして以下が確認できます：

1. **初期設定と制限値の適用**: 設定ファイルの解析時に set_server_limit や set_thread_limit が呼び出され、プロセスやスレッドの上限値が設定されます。 2. **サーバーの維持管理（メインループ）**: 親プロセスのメインループ等から perform_idle_server_maintenance が定期的に呼び出され、現在の負荷状況に応じて make_child を介して新しい子プロセスを動的に生成します。 3. **子プロセスの起動と実行**: 生成された子プロセスは child_main を実行します。ここで ap_reopen_scoreboard によるスコアボードの再オープンや、ap_thread_create によるワーカースレッドの生成が行われ、実際のリクエスト処理を開始します。 4. **プロセスの終了**: エラーの発生やプロセスの寿命、またはメンテナンスによる終了要求の際、clean_child_exit が呼び出されて安全にプロセスが終了します。

重要な不変条件

提供されたシンボル情報からは、MPM 全体における厳密な不変条件（インバリアント）の詳細はまだ完全には解析されていません。

ただし、set_server_limit や set_thread_limit の内部で ap_check_cmd_context が呼び出されていることから、「これらの制限値の設定は特定の構成コンテキスト（例：サーバー全体の設定ファイルレベル）でのみ実行可能であり、動的なリクエストコンテキスト中などで変更されてはならない」という制約が存在することが推測されます。

よくある落とし穴

提供されたソースコードのメタデータからは、具体的なバグや運用上の落とし穴の詳細は直接示されていません。

しかし、一般に ServerLimit や ThreadLimit の設定値がシステムの物理リソース（メモリやファイル記述子など）を超えて不適切に高く設定された場合、リソースの枯渇やプロセスの過剰生成によるスラッシングを引き起こす可能性があります。また、子プロセスの異常終了時に clean_child_exit が適切に呼び出されない場合、共有メモリ（スコアボード）の不整合やゾンビプロセスの発生につながるリスクが考えられます。
