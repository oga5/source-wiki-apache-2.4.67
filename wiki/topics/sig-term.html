sig_term

概念

sig_term は、Apache HTTP Server の複数のプラットフォーム（Unix、OS/2、NetWare）およびマルチプロセッシングモジュール（MPM）において、サーバーの停止シグナル（SIGTERM や AP_SIG_GRACEFUL_STOP など）を受信した際に呼び出されるシグナルハンドラ関数です。プラットフォームやプロセスの役割（親プロセスか子プロセスか）に応じて、それぞれ固有のシャットダウン処理や状態遷移のトリガーとして実装されています。

なぜ重要か

サーバープロセスが外部からの終了要求（シグナル）を安全かつ適切に処理し、リソースの解放やスレッドのクリーンアップ、あるいは即時停止（ungraceful）やグレースフル停止（graceful）の状態遷移を正しく制御するために不可欠な役割を担っています。

エンドツーエンドの流れ

各プラットフォームにおけるシグナル受信からサーバー全体の完全な終了に至るエンドツーエンドの全体フローは、提供された情報からは完全には明らかになっていません。現時点で判明している各環境の処理フローは以下の通りです。

  • **Unix環境**:

    • 1. SIGTERM または AP_SIG_GRACEFUL_STOP シグナルを受信します。 2. retained_data（ap_unixd_mpm_retained_data）が存在することを確認します（存在しない場合は即座に終了）。 3. retained_data->mpm_state を AP_MPMQ_STOPPING に、retained_data->shutdown_pending を 1 に設定します。 4. 受信したシグナルが AP_SIG_GRACEFUL_STOP でない場合、retained_data->is_ungraceful を 1 に設定して強制停止モードにします。

  • **OS/2環境（親プロセス）**:

    • 1. SIGTERM を受信します。 2. グローバル変数 shutdown_pending を 1 に設定します。 3. signal(SIGTERM, SIG_DFL) を呼び出し、以降の SIGTERM ハンドラをシステムのデフォルト動作（プロセス終了）にリセットします。

  • **OS/2環境（子プロセス）**:

    • 1. SIGTERM を受信します。 2. shutdown_pending を 1 に、is_graceful を 0（即時終了）に設定します。 3. signal(SIGTERM, SIG_DFL) を呼び出し、以降の SIGTERM ハンドラをデフォルト動作にリセットします。

  • **NetWare環境**:

    • 1. SIGTERM または SIGABRT を受信します。 2. shutdown_pending がすでに 1 の場合は、二重のシャットダウン要求と判断して即座に終了します。 3. shutdown_pending を 1 に設定します。 4. wait_to_finish が真である間、apr_thread_yield() を呼び出してスレッドの終了を待機するループを実行します。

重要な不変条件

  • **非同期シグナル安全性**: すべてのプラットフォームの sig_term はシグナルハンドラとして実行されるため、非同期シグナル安全（async-signal-safe）な操作のみを行う必要があります。
  • **Unix環境における前提条件**: retained_data が初期化されている必要があります。初期化されていない場合、関数は何も行わずに即座に終了します。
  • **OS/2環境における再入防止**: signal(SIGTERM, SIG_DFL) を呼び出すことで、2回目以降の SIGTERM 受信時はシステムのデフォルト動作（即時プロセス終了）に移行させます。
  • **NetWare環境におけるデッドロック回避**: wait_to_finish の状態管理が適切に行われない場合、スレッド終了待ちのループから抜け出せずデッドロックに陥る可能性があります。

よくある落とし穴

  • **プラットフォーム間の同名関数の混同**: sig_term という名前の関数は複数のプラットフォーム（Unix, OS/2, NetWare）で個別に定義されています。これらはソースコード上で直接的な呼び出し関係はなく、それぞれのビルド環境で排他的にコンパイル・実行されるため、共通化やリファクタリングの際にはプラットフォーム固有の挙動の差異に注意する必要があります。
  • **非同期シグナル安全でない処理の混入**: シグナルハンドラ内で非同期シグナル安全でない関数を呼び出すと、予期しない動作やデッドロックを引き起こす可能性があります。
