sig_term

概念

sig_term は、Apache HTTP Server の複数のプラットフォーム（Unix、OS/2、NetWare）およびマルチプロセッシングモジュール（MPM）において、サーバーの停止シグナル（SIGTERM など）を受信した際に、安全かつ適切なシャットダウンプロセスを開始するためのシグナルハンドラ関数です。

なぜ重要か

サーバープロセスに対して外部から終了要求（シグナル）が送られた際、リソースのクリーンアップやスレッドの安全な終了、あるいはグレースフル（段階的）な停止と即時停止の切り替えを制御し、サーバー全体のライフサイクルを正常に終了させるための起点となるため重要です。

エンドツーエンドの流れ

各プラットフォームにおけるシグナル受信時の処理フローは以下の通りです。ただし、シグナル受信後にメインループやスレッド管理側がこれらのフラグを検知して最終的にプロセスを終了するまでの詳細な全体フロー（エンドツーエンドの完全な流れ）は、提供された情報からは未解明です。

  • **Unix**:

    • 1. SIGTERM または AP_SIG_GRACEFUL_STOP を受信します。 2. retained_data の mpm_state を AP_MPMQ_STOPPING に、shutdown_pending を 1 に設定します。 3. 強制停止（ungraceful）の場合は is_ungraceful を 1 に設定します。

  • **OS/2 (親プロセス)**:

    • 1. SIGTERM を受信します。 2. グローバル変数 shutdown_pending を 1 に設定します。 3. SIGTERM ハンドラをデフォルト（SIG_DFL）にリセットします。

  • **OS/2 (子プロセス)**:

    • 1. SIGTERM を受信します。 2. shutdown_pending を 1、is_graceful を 0（即時終了）に設定します。 3. ハンドラをデフォルトにリセットします。

  • **NetWare**:

    • 1. SIGTERM または SIGABRT を受信します。 2. shutdown_pending を 1 に設定します。 3. wait_to_finish が真である間、apr_thread_yield() を呼び出してスレッドの終了を待機します。

重要な不変条件

  • **Unix**: retained_data が初期化されている必要があります。初期化されていない場合は処理を行わずに即座に終了します。
  • **OS/2**: 2回目以降の SIGTERM 受信時には、ハンドラが SIG_DFL にリセットされているため、システムのデフォルト動作（即時終了）が適用されます。
  • **NetWare**: shutdown_pending がすでに 1 の場合は、二重処理を防ぐため即座に早期リターンします。
  • **共通**: シグナルハンドラとして実行されるため、非同期シグナル安全（async-signal-safe）な操作のみを行う必要があります。

よくある落とし穴

  • **プラットフォーム間の動作差異**: 同名の sig_term 関数であっても、プラットフォームやプロセスの役割（親・子）によって、グレースフル停止の扱い（is_graceful や is_ungraceful の設定）や、ハンドラのリセット有無、スレッド待機処理（apr_thread_yield()）の有無などの実装が大きく異なります。これらを混同して共通化しようとすると、プラットフォーム固有のシグナル処理が破損する恐れがあります。
  • **NetWareでのデッドロック**: NetWare環境において wait_to_finish の状態が他のスレッド管理ロジックによって適切に更新されない場合、sig_term 内の待機ループが終了せず、デッドロックに陥る可能性があります。
