sig_term

概念

sig_term は、Apache HTTP Server の複数のマルチプロセッシングモジュール（MPM）およびプラットフォーム固有のコード（Unix、OS/2、NetWare）において、サーバーやプロセスのシャットダウン要求（SIGTERM などのシグナル）を処理するために実装されているシグナルハンドラ関数です。

なぜ重要か

サーバーの安全かつ確実な停止（シャットダウン）プロセスを開始するためのトリガーとして機能するため重要です。プラットフォームやプロセスの役割（親プロセスか子プロセスか）に応じて、グレースフル（猶予あり）な停止か、即時（強制）停止かを判別し、適切な終了フラグ（shutdown_pending など）をセットしてメインループやスレッドに終了を促します。

関与するモジュール

  • module:os/unix
  • module:server/mpm/mpmt_os2
  • module:server/mpm/netware

主要エンティティ

  • **symbol:os/unix/unixd.c#sig_term**: Unix環境におけるMPM停止シグナル（SIGTERM または AP_SIG_GRACEFUL_STOP）を処理し、状態保持構造体 retained_data の状態を更新します。
  • **symbol:server/mpm/mpmt_os2/mpmt_os2.c#sig_term**: OS/2環境の親プロセスにおいて、SIGTERM 受信時に shutdown_pending を設定し、ハンドラをデフォルトにリセットします。
  • **symbol:server/mpm/mpmt_os2/mpmt_os2_child.c#sig_term**: OS/2環境の子プロセスにおいて、SIGTERM 受信時に即時終了モード（is_graceful = 0）へ移行させます。
  • **symbol:server/mpm/netware/mpm_netware.c#sig_term**: NetWare環境において、SIGTERM や SIGABRT を受信した際に shutdown_pending を設定し、スレッドの終了を待機します。

エンドツーエンドの流れ

各プラットフォームにおけるシグナル受信からシャットダウン完了までの完全なエンドツーエンドの流れ（特にメインループ側がどのようにこれらのフラグを検知してクリーンアップを行うかなど）の全貌は、提供されたモジュール情報からは完全には明らかになっていません。

現時点で判明している部分的な流れは以下の通りです： 1. **シグナル受信**: OSまたはシステムから SIGTERM（または AP_SIG_GRACEFUL_STOP、SIGABRT）が送信され、登録された sig_term ハンドラが起動します。 2. **状態の更新**:

  • **Unix**: retained_data->shutdown_pending を 1 に設定し、必要に応じて is_ungraceful を設定します。
  • **OS/2 (親)**: グローバル変数 shutdown_pending を 1 に設定し、シグナル動作をデフォルト（SIG_DFL）に戻します。
  • **OS/2 (子)**: shutdown_pending を 1、is_graceful を 0 に設定し、シグナル動作をデフォルトに戻します。
  • **NetWare**: shutdown_pending を 1 に設定し、wait_to_finish が真の間 apr_thread_yield() を呼び出してスレッドの終了を待機します。

    • 3. **メインループへの通知**: 各環境のメインループやスレッド管理ロジックが、これらの更新されたフラグを検知して実際のシャットダウン処理を実行します。

重要な不変条件

  • **非同期シグナル安全性**: sig_term はシグナルハンドラとして非同期に呼び出されるため、ハンドラ内では非同期シグナル安全（async-signal-safe）な操作のみを行う必要があります。
  • **再入・重複防止**:
  • Unix環境では、retained_data が初期化されていない場合は即座に終了します。また、すでに停止処理が保留中の場合は重複処理を避けます。
  • NetWare環境では、shutdown_pending がすでに 1 の場合は二重処理を避けるために即座に早期リターンします。
  • OS/2環境では、一度シグナルを処理すると signal(SIGTERM, SIG_DFL) によりハンドラがリセットされ、2回目以降の SIGTERM はシステムのデフォルト動作（即時終了）になります。

よくある落とし穴

  • **デッドロックの危険性 (NetWare)**: NetWare環境の実装では、wait_to_finish が真である間ループしてスレッドの終了を待ちますが、この変数が他のスレッドによって適切に更新されない場合、無限ループ（デッドロック）に陥る可能性があります。
  • **プラットフォーム間の動作差異**: 同名の sig_term 関数であっても、Unix、OS/2、NetWareでシグナルハンドラのリセット挙動や、更新するグローバル変数の種類（retained_data 構造体か、単一のグローバル変数か）が異なります。コードの共通化や移植の際には、これらのプラットフォーム固有の差異に注意する必要があります。
