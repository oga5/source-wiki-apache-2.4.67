sig_term

概念

sig_term は、Apache HTTP Server の各プラットフォーム（Unix、OS/2、NetWare）およびマルチプロセッシングモジュール（MPM）において、サーバーやプロセスの停止シグナル（SIGTERM 等）を受信した際にシャットダウン処理を開始するためのシグナルハンドラ関数です。

なぜ重要か

サーバープロセスが外部（OSや管理者）からの終了要求（SIGTERM や AP_SIG_GRACEFUL_STOP など）を安全かつ適切に処理し、グレースフル（猶予あり）または即時のシャットダウンシーケンスをトリガーするために不可欠な役割を果たします。プラットフォームごとに異なるシグナル処理や状態管理の仕組みを抽象化し、それぞれの環境に適した終了フラグの設定やリソースのクリーンアップへの移行を保証します。

エンドツーエンドの流れ

各プラットフォームにおけるシグナル受信からシャットダウン開始までの流れは以下の通りです。ただし、シグナル受信後のメインループ側での具体的な終了処理の全容など、詳細なエンドツーエンドのフローは完全には判明していません。

  • **Unix環境 (os/unix/unixd.c)**:

    • 1. SIGTERM または AP_SIG_GRACEFUL_STOP を受信します。 2. retained_data（生存状態保持構造体）の mpm_state を AP_MPMQ_STOPPING に設定します。 3. shutdown_pending を 1 に設定します。 4. グレースフル停止でない場合は is_ungraceful を 1 に設定します。

  • **OS/2環境（親プロセス mpmt_os2.c）**:

    • 1. SIGTERM を受信します。 2. グローバル変数 shutdown_pending を 1 に設定します。 3. signal(SIGTERM, SIG_DFL) により、以降の SIGTERM ハンドラをデフォルト動作にリセットします。

  • **OS/2環境（子プロセス mpmt_os2_child.c）**:

    • 1. SIGTERM を受信します。 2. shutdown_pending を 1、is_graceful を 0（即時終了モード）に設定します。 3. ハンドラをデフォルト（SIG_DFL）にリセットします。

  • **NetWare環境 (mpm_netware.c)**:

    • 1. SIGTERM または SIGABRT を受信します。 2. shutdown_pending が既に 1 の場合は即座に終了（重複防止）します。 3. shutdown_pending を 1 に設定します。 4. wait_to_finish が真である間、apr_thread_yield() を呼び出してスレッドの終了を待機します。

重要な不変条件

  • **Unix環境**: retained_data が初期化されている必要があります。初期化されていない場合、関数は何も行わずに即座に終了します。
  • **OS/2環境**: SIGTERM 受信後、ハンドラは即座にデフォルト（SIG_DFL）にリセットされ、2回目以降のシグナル受信ではプロセスが即時終了します。
  • **NetWare環境**: shutdown_pending が既に 1 である場合、二重処理を避けるために関数は副作用なしに早期リターンします。
  • ※プラットフォーム間を跨ぐ統一的な不変条件や、メインループとの厳密な同期メカニズムの全容は完全には判明していません。

よくある落とし穴

  • **非同期シグナル安全性**: sig_term はシグナルハンドラとして実行されるため、非同期シグナル安全（async-signal-safe）な操作のみを行う必要があります。
  • **デッドロックの危険性 (NetWare)**: wait_to_finish の状態が適切に更新されない場合、apr_thread_yield() のループから抜け出せず、デッドロックに陥る可能性があります。
  • **プラットフォーム間の差異**: 同名の sig_term 関数であっても、プラットフォーム（Unix, OS/2, NetWare）やプロセス役割（OS/2の親と子）によって、グレースフル停止の扱い（is_graceful や is_ungraceful の設定）やシグナル再登録の挙動が異なるため、共通化やリファクタリングの際には注意が必要です。
