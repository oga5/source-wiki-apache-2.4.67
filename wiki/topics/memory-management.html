Memory management

概念

Apache HTTP Server（httpd）におけるメモリ管理は、主に Apache Portable Runtime（APR）が提供するメモリプール（apr_pool_t）システムに基づいています。

メモリプールは、リソースの生存期間（ライフサイクル）に応じてメモリを一括して割り当て・解放する仕組みであり、個別の明示的な free 呼び出しを不要にすることでメモリリークを防ぎます。また、プロセス間でのデータ共有のために共有メモリ管理（apr_rmm_t）が使用されるほか、外部ライブラリが割り当てたメモリに対しては明示的な解放処理が行われます。

なぜ重要か

httpd は高並列・高可用性が求められる Web サーバーであり、リクエスト処理ごとにメモリの割り当てと解放が頻繁に発生します。

  • **メモリリークの防止**: リクエストや接続、設定のライフサイクルに紐づいたメモリプールを使用することで、処理終了時に一括してメモリを解放し、長期稼働時のメモリリークを防ぎます。
  • **共有メモリの効率的利用**: 認証情報やキャッシュなどのプロセス間共有データにおいて、共有メモリ（apr_rmm_t）を適切に管理・解放（rmm_free）することで、限られた共有リソースの枯渇を防ぎます。
  • **リソースの確実なクリーンアップ**: プールクリーンアップ（apr_pool_cleanup_register）を利用して、メモリだけでなくファイル記述子やミューテックスなどのシステムリソースを安全に解放します。

関与するモジュール

提供された構成情報に基づき、以下のモジュールがメモリ管理に関与しています。

  • module:include (メモリ割り当て属性マクロ ap_config.h の提供)
  • module:modules/aaa (認証モジュールにおける共有メモリ管理や設定構造体のプール割り当て)
  • module:modules/arch/unix (SELinux コンテキストなどの OS 固有リソースの解放)
  • module:modules/cache (キャッシュ処理におけるリクエストプール r->pool を用いたメモリ管理や文字列複製)
  • その他、module:server や各種 MPM、プロキシ、SSL、LDAP などのモジュール群。

主要エンティティ

提供されたシンボルから、メモリ管理に直接関与する主要なエンティティは以下の通りです。

  • **メモリ割り当て属性マクロ**:
  • symbol:include/ap_config.h#AP_FN_ATTR_ALLOC_SIZE / AP_FN_ATTR_ALLOC_SIZE2: コンパイラに対してメモリ割り当てサイズを通知する属性定義。
  • **共有メモリ管理**:
  • symbol:modules/aaa/mod_auth_digest.c#rmm_free: 共有メモリ管理ハンドル（apr_rmm_t）を介して、割り当てられたメモリ領域を解放するラッパー。
  • **設定・キャッシュのライフサイクル管理**:
  • symbol:modules/aaa/mod_authn_socache.c#authn_cache_dircfg_create: メモリプールから apr_palloc を用いてディレクトリ設定構造体を割り当てる。
  • symbol:modules/aaa/mod_authn_socache.c#authn_cache_dircfg_merge: apr_pmemdup を用いて設定構造体を複製・マージする。
  • symbol:modules/aaa/mod_authn_socache.c#authn_cache_post_config: 設定用プール pconf や一時プール ptmp を使用し、クリーンアップハンドラ（destroy_cache など）を登録する。
  • symbol:modules/aaa/mod_authn_socache.c#destroy_cache: キャッシュインスタンスを破棄し、グローバルポインタを NULL にリセットする。
  • **外部ライブラリリソースの解放**:
  • symbol:modules/arch/unix/mod_systemd.c#log_selinux_context: getcon で取得した SELinux コンテキストを freecon で明示的に解放する。
  • **リクエスト単位のメモリ管理**:
  • symbol:modules/cache/cache_storage.c#cache_create_entity / cache_select / cache_invalidate: リクエストプール r->pool を使用して、キャッシュハンドル（cache_handle_t）や一時的な構造体を割り当てる。
  • symbol:modules/cache/cache_util.c#ap_cache_tokstr: apr_pstrmemdup を使用して、指定されたプールから複製された文字列トークンを割り当てる。
  • symbol:modules/cache/cache_util.c#cache_remove_lock: リクエストプール r->pool に登録されたユーザーデータ（CACHE_LOCKFILE_KEY）からファイルハンドルを取得してクローズする。

エンドツーエンドの流れ

現時点では、システム全体におけるメモリ管理の完全なエンドツーエンドのフロー（すべてのプール階層の生成から消滅までの全プロセス）は完全には判明していません。しかし、提供されたエンティティから以下の部分的な流れが確認できます。

1. **設定フェーズ**:

  • サーバー起動時、設定用プール pconf や一時プール ptmp が提供されます。
  • authn_cache_dircfg_create や authn_cache_dircfg_merge により、プールから設定構造体が割り当てられます。
  • authn_cache_post_config において、apr_pool_cleanup_register を使用して pconf に destroy_cache などのクリーンアップ関数が登録されます。

    • 2. **リクエスト処理フェーズ**:

  • リクエストごとに r->pool が生成されます。
  • cache_select や cache_create_entity などの処理において、r->pool を用いて cache_handle_t などのメモリが割り当てられます。
  • ap_cache_tokstr などの文字列操作において、apr_pstrmemdup により r->pool 上に複製された文字列が生成されます。

    • 3. **クリーンアップフェーズ**:

  • リクエスト終了時、r->pool が破棄され、そのプールから割り当てられたすべてのメモリが一括して解放されます。
  • サーバー停止または再起動時、pconf が破棄され、登録された destroy_cache が実行されて共有キャッシュインスタンスなどのリソースが解放されます。

重要な不変条件

  • **プールの生存期間（ライフサイクル）の遵守**:
  • r->pool から割り当てられたメモリ（例: cache_handle_t や ap_cache_tokstr で生成された文字列）は、リクエストのライフサイクルを超えて生存することはできません。リクエスト終了後にこれらのポインタを参照してはなりません。
  • **共有メモリの二重解放防止と NULL リセット**:
  • destroy_cache において、socache_provider->destroy を呼び出した後、グローバル変数 socache_instance は必ず NULL にリセットされなければなりません。これにより、二重解放やダングリングポインタの参照を防ぎます。
  • **外部リソースの明示的解放**:
  • log_selinux_context のように、APR プール管理外の外部ライブラリ（SELinux など）によって割り当てられたメモリは、関数の終了前に必ず対応する解放関数（freecon）で明示的に解放されなければなりません。

よくある落とし穴

  • **プール生存期間の誤認**:
  • r->pool から割り当てられた文字列や構造体のポインタを、リクエストのライフサイクルを超えて存続するグローバルな構造体や接続プール（c->pool）に格納してしまうと、リクエスト終了後に不正なメモリ参照（Use-After-Free）が発生します。
  • **クリーンアップ登録の漏れ**:
  • post_config などの初期化フェーズで作成したグローバルなリソース（ミューテックスやキャッシュインスタンスなど）について、プールクリーンアップ（apr_pool_cleanup_register）への登録を怠ると、サーバー再起動（graceful restart）時にリソースリークが発生します。
  • **外部メモリの解放漏れ**:
  • APR プールによる自動解放に依存しすぎると、getcon などの外部 API が割り当てたメモリの freecon による解放を忘れ、メモリリークを引き起こす原因になります。
