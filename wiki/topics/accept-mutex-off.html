accept_mutex_off

概念

accept_mutex_off は、Apache HTTP Server (httpd) のマルチプロセッシングモジュール (MPM) である prefork や、関連するテストツールにおいて、接続の受け入れ（accept）処理の直後にプロセス間排他制御（ミューテックスやロック）を解放するために使用される関数です。

なぜ重要か

複数の子プロセスが同時に接続を待ち受けるマルチプロセスモデルにおいて、特定のプロセスが接続を受け入れた後、速やかにロックを解放して他の子プロセスが次の接続を受け入れられるようにするために重要です。これにより、接続処理の競合を防ぎ、効率的なリクエスト処理を実現します。

主要エンティティ

  • **symbol:server/mpm/prefork/prefork.c#accept_mutex_off**

    • prefork MPMにおいて、apr_proc_mutex_unlock を用いてプロセス間ミューテックスを解放する関数。

  • **symbol:test/time-sem.c#accept_mutex_off**

    • テスト環境において、fcntl システムコールを用いてプロセス間排他制御（ロック）を解放する関数。

エンドツーエンドの流れ

本関数は、接続処理のライフサイクルにおいて以下のように呼び出されます。ただし、テスト用実装と本番用実装がビルド時にどのように切り替えられるかなど、システム全体における詳細なエンドツーエンドの連携フローの全容は完全には明らかになっていません。

1. prefork MPMにおける流れ

  • 子プロセスが SAFE_ACCEPT マクロ経由で accept_mutex_off を呼び出します。
  • my_bucket->mutex に対して apr_proc_mutex_unlock を実行し、ミューテックスを解放します。
  • 解放に成功した場合は APR_SUCCESS を返します。
  • 解放に失敗した場合、現在のプロセス世代とスコアボード上の実行世代を比較します。世代が一致していれば緊急ログを出力して APEXIT_CHILDFATAL で強制終了し、世代が異なる（グレースフル再起動中など）場合はデバッグログを出力して処理を継続します。

2. test/time-sem.c における流れ

  • 計測ループまたは SAFE_ACCEPT マクロ経由で呼び出されます。
  • fcntl システムコールを F_SETLKW コマンドで呼び出し、unlock_it 構造体に基づいてロックを解除します。
  • 解放に失敗した場合は、perror でエラーを出力し、exit(1) で即座にプロセスを終了します。

重要な不変条件

  • **prefork MPMの実装**: グローバルな my_bucket が初期化されており、有効なミューテックスを保持している必要があります。また、本関数は accept() 処理の直後に呼び出されることを想定しています。
  • **test/time-sem.c の実装**: fcntl_fd が有効なファイル記述子として初期化されており、unlock_it が struct flock 構造体として適切に設定されている必要があります。

よくある落とし穴

  • **マクロ展開による隠蔽**: prefork MPMでは、本関数は直接呼び出されるのではなく SAFE_ACCEPT マクロを通じて呼び出されるため、ソースコードを静的解析する際に直接の呼び出し箇所を見落とす可能性があります。
  • **エラーハンドリングの差異**: test/time-sem.c はベンチマークやテスト用の設計であるため、ロック解除失敗時にエラー復旧を行わず即座に exit(1) で終了します。本番環境の挙動とは異なる点に注意が必要です。
  • **同名関数の重複**: test/time-sem.c 内には、コンパイルオプションによって異なるロック方式（SysV sem, pthread mutex等）に対応した同名の関数が複数定義されている可能性があります。
