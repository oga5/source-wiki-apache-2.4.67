accept_mutex_off

概念

accept_mutex_off は、Apache HTTP Server (httpd) のマルチプロセッシングモジュール（MPM）である prefork や、関連するテストツールにおいて、接続の受け入れ（accept）時に使用したプロセス間排他制御（ミューテックスまたはロック）を解放するための関数です。

なぜ重要か

複数の子プロセスが同時に accept() を呼び出すことによる競合（サンダリングハード問題など）を防ぐために、プロセス間でミューテックスによる排他制御が行われます。accept_mutex_off は、接続を正常に受け入れた後にこのロックを速やかに解放し、他のプロセスが次の接続を受け入れられる状態にするために極めて重要です。これが正しく機能しないと、デッドロックやプロセスのハングアップ、あるいは致命的なエラーによるプロセスの強制終了が発生します。

関与するモジュール

主要エンティティ

  • symbol:server/mpm/prefork/prefork.c#accept_mutex_off: prefork MPMにおいて、APR（Apache Portable Runtime）の apr_proc_mutex_unlock を用いてプロセス間ミューテックスを解放する関数。
  • symbol:test/time-sem.c#accept_mutex_off: テスト環境において、fcntl システムコールを用いてプロセス間のファイルロックを解放する関数。

エンドツーエンドの流れ

1. **prefork MPMにおける流れ**:

  • 子プロセスが SAFE_ACCEPT マクロを介して accept() 処理を行います。
  • accept() の直後に SAFE_ACCEPT マクロ経由で accept_mutex_off() が呼び出されます。
  • accept_mutex_off() は、グローバルな my_bucket から my_bucket->mutex を取得し、apr_proc_mutex_unlock を呼び出してロックを解放します。
  • 解放に成功した場合は APR_SUCCESS を返します。
  • 解放に失敗した場合、現在のプロセス世代（retained->mpm->my_generation）とスコアボード上の実行世代を比較します。世代が異なる場合はデバッグログを出力して処理を継続し、世代が一致している場合は緊急ログを出力して APEXIT_CHILDFATAL でプロセスを強制終了します。

2. **テスト（time-sem.c）における流れ**:

  • 計測ループ内、または SAFE_ACCEPT マクロ経由で accept_mutex_off() が呼び出されます。
  • グローバル変数 fcntl_fd および unlock_it（struct flock 型）を使用して、fcntl システムコール（F_SETLKW）を呼び出し、ロックを解除します。
  • fcntl がエラー（負の値）を返した場合は、perror でエラーを出力し、exit(1) で即座にプログラムを終了します。

※なお、prefork.c と test/time-sem.c の間での具体的な連携や、テストコードがどのように prefork のマクロと結合して動作するかといった詳細なエンドツーエンドの統合フローは、提供された情報からは完全には明らかになっていません。

重要な不変条件

  • **prefork MPM**:
  • 呼び出し時に my_bucket が初期化されており、有効なミューテックスを保持している必要があります。
  • 本関数は accept() 処理の直後に呼び出されることを想定しています。
  • **テスト（time-sem.c）**:
  • fcntl_fd が有効なファイル記述子として初期化されている必要があります。
  • unlock_it が fcntl の F_SETLKW 操作に適した struct flock 構造体として設定されている必要があります。

よくある落とし穴

  • **マクロ展開の誤認**: prefork.c において、本関数は直接呼び出されるのではなく SAFE_ACCEPT マクロ（591行目、613行目、647行目）を通じて呼び出されるため、ソースコード上で直接の関数呼び出しを検索する際に見落とす可能性があります。
  • **エラー時の致命的終了**: ロック解放失敗時の挙動として、世代が一致している場合は APEXIT_CHILDFATAL による強制終了や、テスト環境での exit(1) による即時終了が発生するため、ミューテックスの管理不全はプロセスの生存に直結します。
  • **同名関数の混同**: test/time-sem.c と prefork.c に同名の accept_mutex_off が存在しますが、これらは別個の実装です。また、テスト環境ではコンパイルオプションによって異なるロック方式（SysV sem, pthread mutex等）の同名関数が有効化される可能性があるため、ビルド構成に注意する必要があります。
