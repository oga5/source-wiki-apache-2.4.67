accept_mutex_off

概念

accept_mutex_off は、Apache HTTP Server (httpd) において、複数の子プロセスが同時に accept() を呼び出す際の競合を防ぐために取得したプロセス間ミューテックス（またはロック）を解放するための関数です。本番用のマルチプロセッシングモジュール（MPM）である prefork と、ロックの性能測定を行うテストプログラム test/time-sem.c の両方に同名の関数が存在し、それぞれ異なるロック解放アプローチ（APRミューテックスと fcntl）を実装しています。

なぜ重要か

マルチプロセスモデル（特に prefork MPM）では、複数の子プロセスが同一のリスニングソケットに対して accept() を呼び出す際、接続処理の直列化（シリアライズ）が必要です。accept_mutex_off は、接続を受け入れた直後にロックを速やかに解放し、待機している他の子プロセスが次の接続を受け入れられるようにするために極めて重要です。これが正しく機能しないと、デッドロックが発生するか、特定のプロセスのみが処理を独占することになります。

関与するモジュール

  • module:server/mpm/prefork
  • module:test

主要エンティティ

  • symbol:server/mpm/prefork/prefork.c#accept_mutex_off: prefork MPM におけるミューテックス解放関数。apr_proc_mutex_unlock を呼び出します。
  • symbol:test/time-sem.c#accept_mutex_off: fcntl を用いたロック解放を行うテスト用関数。

エンドツーエンドの流れ

1. **prefork MPM における流れ**:

  • 子プロセスは SAFE_ACCEPT マクロを介してロックを取得し、accept() で接続を待ち受けます。
  • 接続が確立されると、SAFE_ACCEPT マクロ経由で accept_mutex_off() が呼び出されます。
  • accept_mutex_off() は my_bucket->mutex に対して apr_proc_mutex_unlock を実行し、ロックを解放します。
  • ロック解放に失敗した場合、プロセスの世代（retained->mpm->my_generation）を確認し、現在の世代と一致していれば致命的エラー（APEXIT_CHILDFATAL）としてプロセスを終了します。世代が異なる（グレースフル再起動中など）場合は、デバッグログを出力して処理を継続します。

2. **テスト（time-sem.c）における流れ**:

  • 計測ループ内、または SAFE_ACCEPT マクロ経由で accept_mutex_off() が呼び出されます。
  • fcntl システムコールに F_SETLKW コマンドと unlock_it（struct flock）を渡し、ファイル記述子 fcntl_fd のロックを解除します。
  • 解除に失敗した場合は、perror でエラーを出力し、exit(1) で即座に終了します。

*注意: test/time-sem.c の関数が prefork.c の SAFE_ACCEPT マクロからどのように切り替えられて呼び出されるか、そのビルド時や実行時の詳細な統合フローは現時点では完全には判明していません。*

重要な不変条件

  • **prefork MPM**:
  • 呼び出し時に my_bucket が初期化されており、有効なミューテックスを保持している必要があります。
  • 本関数は accept() 処理の直後に呼び出されることが前提となっています。
  • **テスト（time-sem.c）**:
  • fcntl_fd が有効なファイル記述子として初期化されている必要があります。
  • unlock_it が fcntl の F_SETLKW 操作に適した struct flock 構造体として設定されている必要があります。

よくある落とし穴

  • **マクロ展開の不透明性**: prefork.c 内では、本関数は直接呼び出されず SAFE_ACCEPT マクロ（591行目、613行目、647行目など）を介して呼び出されるため、静的解析やコード検索の際に見落とす可能性があります。
  • **エラーハンドリングの差異**: prefork MPM では世代交代を考慮してエラーをデバッグログに留める緩和策がありますが、テストコード（time-sem.c）ではエラー発生時に即座に exit(1) でプロセスが終了するため、テスト環境と本番環境でのエラー耐性の違いに注意が必要です。
  • **ビルド構成による競合**: test/time-sem.c 内には、コンパイルオプションによって異なるロック方式（SysV sem, pthread mutex等）に対応した同名の関数が複数定義されている可能性があり、ビルド構成の確認を怠ると意図しない実装が評価される恐れがあります。
