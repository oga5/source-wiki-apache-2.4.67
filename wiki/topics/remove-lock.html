remove_lock

概念

remove_lock は、Apache HTTP Server（httpd-2.4.67）の socache（Shared Object Cache）に関連する複数のモジュールにおいて、初期化されたグローバルミューテックスなどのロックリソースを安全に破棄・解放するためのクリーンアップ関数です。

なぜ重要か

モジュールのアンロードやプロセスの終了、プールの破棄に伴い、確保されたシステムリソース（グローバルミューテックスなど）を確実に解放し、リソースリークを防ぐために重要です。主に apr_pool_cleanup_register を介した自動クリーンアップ機構と連携して動作します。

  • symbol:modules/cache/mod_cache_socache.c#remove_lock: mod_cache_socache モジュール内に定義されている同名の関数（詳細な振る舞いは未解析）。

エンドツーエンドの流れ

mod_authn_socache における流れは以下の通りです： 1. モジュールの初期化時（modules/aaa/mod_authn_socache.c:125）に、apr_pool_cleanup_register を介して remove_lock がプール破棄時のクリーンアップハンドラとして登録されます。 2. プールが破棄される際、登録された remove_lock が呼び出されます。 3. remove_lock 内でグローバル変数 authn_cache_mutex が NULL でないことを確認し、apr_global_mutex_destroy を呼び出してミューテックスを破棄します。 4. 破棄後、authn_cache_mutex を NULL に設定し、APR_SUCCESS を返します。

※ mod_cache_socache.c における remove_lock のエンドツーエンドの流れや呼び出し関係については、現時点では解析されておらず不明です。

重要な不変条件

  • **mod_authn_socache における不変条件**:
  • authn_cache_mutex が初期化済みであるか、または NULL であること。
  • 本関数の実行完了後、authn_cache_mutex は必ず NULL にリセットされること。
  • **mod_cache_socache における不変条件**:
  • 現時点では未解析のため不明です。

よくある落とし穴

  • **同名関数の混同**: mod_authn_socache.c と mod_cache_socache.c の両方に static apr_status_t remove_lock(void *data) という同名の静的関数が存在するため、デバッグやスタックトレースの確認時にファイルパスを混同しないよう注意が必要です。
  • **エラーの非伝播**: mod_authn_socache の実装では、apr_global_mutex_destroy がエラーを返した場合でも、関数自体は常に APR_SUCCESS を返します。そのため、ミューテックスの破棄失敗が上位に伝播しない点に留意する必要があります。
