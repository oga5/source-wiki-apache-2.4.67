remove_lock

概念

remove_lock は、Apache HTTP Server（httpd）の共有オブジェクトキャッシュ（socache）を利用するモジュールにおいて、排他制御に使用されるグローバルミューテックスを安全に破棄し、システムリソースを解放するためのクリーンアップ関数です。

なぜ重要か

マルチプロセスやマルチスレッド環境において、キャッシュへのアクセスを同期するためのグローバルミューテックス（例: authn_cache_mutex）は、プロセスの終了時やメモリプールの破棄時に適切に解放される必要があります。remove_lock は、メモリプール（apr_pool_t）のクリーンアップ処理と連動して自動的にミューテックスを破棄することで、リソースリークやデッドロックを防止します。

関与するモジュール

  • module:modules/aaa（認証・認可モジュール群。具体的には mod_authn_socache）
  • module:modules/cache（キャッシュモジュール群。具体的には mod_cache_socache）

主要エンティティ

  • symbol:modules/aaa/mod_authn_socache.c#remove_lock: mod_authn_socache 内で定義されるクリーンアップ関数。グローバル変数 authn_cache_mutex を破棄します。
  • symbol:modules/cache/mod_cache_socache.c#remove_lock: mod_cache_socache 内に存在する同名の関数。

エンドツーエンドの流れ

1. **初期化と登録**: mod_authn_socache の初期化時において、apr_pool_cleanup_register を介して remove_lock がプール破棄時のクリーンアップハンドラとして登録されます。 2. **プールの破棄**: Apache のライフサイクルに伴いプールが破棄される際、登録された remove_lock が自動的に呼び出されます。 3. **ミューテックスの破棄**:

  • グローバル変数 authn_cache_mutex が NULL でないことを確認します。
  • apr_global_mutex_destroy を呼び出して、実際のミューテックスを破棄します。
  • authn_cache_mutex を NULL に設定します。
  • 常に APR_SUCCESS を返します。

※ mod_cache_socache.c 側の remove_lock における具体的なエンドツーエンドの流れや、両モジュール間を跨ぐ統合的な制御フローは現時点では不明です。

重要な不変条件

  • **冪等性（安全な複数回実行）**: authn_cache_mutex が既に NULL である場合、関数は何も行わずに正常終了します。
  • **実行後の状態**: 本関数の実行完了後、対象のミューテックスポインタ（authn_cache_mutex）は必ず NULL にリセットされます。
  • **戻り値の保証**: apr_global_mutex_destroy がエラーを返した場合でも、本関数は常に APR_SUCCESS を返します。
  • ※ mod_cache_socache.c 側の不変条件については、詳細な情報が提供されていないため不明です。

よくある落とし穴

  • **同名関数の混同**: mod_authn_socache.c と mod_cache_socache.c の両方に remove_lock という同名の static 関数が存在します。これらは別個の実装であるため、デバッグやスタックトレースの確認時にはファイルパスを混同しないよう注意が必要です。
  • **直接呼び出しの回避**: 本関数は apr_pool_cleanup_register を通じたプールクリーンアップ処理から呼び出されることを前提としています。手動で直接呼び出すと、ライフサイクル管理に不整合が生じる可能性があります。
