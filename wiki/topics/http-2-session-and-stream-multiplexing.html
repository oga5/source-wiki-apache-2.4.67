HTTP/2 Session and Stream Multiplexing

概念

HTTP/2 におけるセッションとストリームの多重化（Multiplexing）とは、単一の TCP 接続（メイン接続 c1）上で、複数の独立した双方向リクエスト/レスポンス（ストリーム）を並行して処理する仕組みです。

mod_http2 においては、接続全体のプロトコル状態やライフサイクルを管理する **セッション（h2_session）** と、個々の論理的なリクエスト/レスポンスを管理する **ストリーム（h2_stream）** が明確に分離されています。これらは多重化器（h2_mplx）を介して、メイン接続を処理するスレッドと、実際の処理を非同期に行うワーカースレッド（h2_workers）の間で安全に仲介されます。

なぜ重要か

HTTP/1.x では、1つの接続で同時に処理できるリクエストは1つのみであり、ヘッドオブラインブロッキング（HOLB）が発生していました。

HTTP/2 の多重化により、単一の接続上で複数のストリームを同時に並行処理できるようになります。これにより、接続確立のオーバーヘッドが削減され、ネットワークリソースの利用効率が大幅に向上します。Apache mod_http2 では、メイン接続（c1）を制御するスレッドと、個々のストリームを処理するワーカースレッド（c2）を分離し、データをスレッド間で安全かつ効率的に受け渡すことで、この多重化を実現しています。

関与するモジュール

  • module:modules/http2

主要エンティティ

  • **symbol:modules/http2/h2_session.h#h2_session**: HTTP/2 セッションのライフサイクル、状態管理、およびストリームの多重化を制御するメイン構造体。クライアントとの単一の TCP 接続（conn_rec）を抽象化します。
  • **symbol:modules/http2/h2_mplx.h#h2_stream**: セッション内における個別のリクエストおよびレスポンスのライフサイクルと状態を管理するコンテキスト構造体。
  • **symbol:modules/http2/h2_session.c#h2_session_create**: h2_session を生成し、必要な関連コンポーネント（h2_mplx など）を初期化する関数。
  • **symbol:modules/http2/h2_stream.c#h2_stream_prepare_processing**: ストリームの処理開始直前に、入力データの受け渡しに必要な「入力ビーム（h2_bucket_beam）」をセットアップする関数。
  • **symbol:modules/http2/h2_workers.c#h2_workers_create**: リクエスト処理用のワーカースレッド管理機構（h2_workers）を生成する関数。

エンドツーエンドの流れ

1. **初期化**:

  • 子プロセスの初期化時（h2_c1_child_init）に h2_workers_create が呼び出され、ワーカースレッドプール（h2_workers）が生成されます。
  • クライアントからの接続確立時（h2_c1_setup）に h2_session_create が呼び出され、h2_session が生成されます。この内部で h2_mplx_c1_create が呼ばれ、多重化のハブとなる h2_mplx が作成されます。

    • 2. **フレームの受信とストリーム生成**:

  • クライアントから受信した HTTP/2 フレームは、nghttp2 ライブラリを介して解析され、h2_session のメインループ（h2_session_process）でイベントとして処理されます。
  • 新しいリクエストに対して h2_stream が作成され、h2_mplx に登録されます。

    • 3. **ストリーム処理の準備**:

  • ストリームの処理が開始される直前に、h2_mplx.c から h2_stream_prepare_processing が呼び出されます。
  • 入力がまだ閉じられていない場合、stream_setup_input を介して、メイン接続（c1）とワーカースレッド（c2）間でデータを転送するためのバッファ（入力ビーム）がセットアップされます。

    • 4. **非同期実行とレスポンス**:

  • 準備が整ったストリームは、h2_workers によって管理されるワーカースレッドに送られ、非同期に処理されます。
  • ワーカースレッドで生成されたレスポンスデータは、ビームを介してメイン接続（c1）に送り返され、クライアントへ送信されます。

    • 5. **クリーンアップ**:

  • ストリームの処理が完了すると、h2_mplx_c1_stream_cleanup を経て破棄キューへ移動し、リソースが解放されます。

※ なお、nghttp2 と Apache 側の状態遷移の完全な同期アルゴリズムや、スレッド間での詳細な排他制御の全容など、一部のエンドツーエンドのフローは提供されたコンテキストからは未解明です。

重要な不変条件

  • **スレッド間アクセスの同期**: h2_stream はメイン接続（c1）を扱うスレッドと、処理を行うワーカースレッド（c2）の間で共有されるため、アクセスには h2_mplx を介した適切な同期が必要です。
  • **状態の同期**: h2_session 内の nghttp2_session（ngh2）が保持するプロトコル解析状態と、Apache 側のセッション状態（h2_session_state）は常に同期していなければなりません。
  • **メモリプールの生存期間**: h2_session の pool は、セッションの生存期間中常に有効でなければなりません。

よくある落とし穴

  • **直接アクセスの禁止**: h2_stream は c1 と c2 のコンテキストを跨ぐため、構造体のメンバへ直接アクセスすることは避け、必ず提供されている API を介して操作する必要があります。これを怠ると、デッドロックや競合状態を引き起こす原因になります。
  • **ストリームエラーの累積**: セッション内で発生したストリームエラーの数（stream_errors）が max_stream_errors を超えると、セッション全体が異常終了します。
  • **I/O ブロック**: out_c1_blocked や ready_to_process キューの状態管理に不備があると、I/O 待ちや処理の遅延が発生する可能性があります。
