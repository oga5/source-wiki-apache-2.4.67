Authentication and Authorization Provider Architecture

概念

Apache HTTP Server（httpd）における **Authentication and Authorization Provider Architecture**（認証・認可プロバイダアーキテクチャ）は、ユーザーの本人確認を行う「認証（Authentication / authn）」と、リソースへのアクセス権限を検証する「認可（Authorization / authz）」の機能を分離し、それぞれをプラグイン可能なプロバイダとしてモジュール化する共通フレームワークです。

このアーキテクチャにより、コアとなるアクセス制御ロジックと、具体的なバックエンド（テキストファイル、データベース、LDAPなど）への問い合わせロジックが疎結合に保たれます。

なぜ重要か

1. **柔軟な拡張性**: 新しい認証方式や認可ルールを導入する際、コアモジュールを書き換えることなく、独自のプロバイダモジュールを追加するだけで対応できます。 2. **設定の共通化と組み合わせ**: mod_authn_core および mod_authz_core が提供する共通ディレクティブ（AuthBasicProvider や <RequireAll> などの論理コンテナ）を使用することで、複数のプロバイダを組み合わせた複雑なアクセス制御を統一された方法で記述できます。

関与するモジュール

  • module:include
  • module:modules/aaa

主要エンティティ

  • **include/mod_auth.h**:

    • 認証および認可プロバイダが実装すべきインターフェース（構造体定義や共通の戻り値ステータスなど）を定義します。プロバイダ登録のための共通APIやマクロが提供されます。

  • **modules/aaa/mod_authn_core.c**:

    • 認証（authn）のコアフレームワークです。認証プロバイダ（例: file, dbm）の登録管理、および AuthType や AuthName などの認証ディレクティブの処理を担当します。

  • **modules/aaa/mod_authz_core.c**:

    • 認可（authz）のコアフレームワークです。認可プロバイダ（例: user, group, ip）の登録管理、および Require ディレクティブや、それらをネストして論理評価するコンテナ（<RequireAll>, <RequireAny>, <RequireNone>）の処理を担当します。

エンドツーエンドの流れ

1. **リクエストの受信とフェーズの開始**: クライアントからのリクエストがアクセス制限されたディレクトリに到達すると、Apacheのフック機構を介してAAA処理が開始されます。 2. **認証（Authentication）フェーズ**:

  • mod_authn_core が、設定された認証プロバイダ（例: AuthBasicProvider file であれば mod_authn_file）を呼び出します。
  • プロバイダは include/mod_auth.h で定義されたインターフェースに従い、クライアントから送られた資格情報（パスワードなど）を検証します。

    • 3. **認可（Authorization）フェーズ**:

  • 認証が成功すると、mod_authz_core が Require ディレクティブに指定された認可プロバイダを呼び出します。
  • 認可プロバイダは、認証されたユーザー名やクライアントのIPアドレスなどの属性を基に、アクセスを許可するかどうかを判定します。

*注意: 提供されたソースコード（mod_auth.h, mod_authn_core.c, mod_authz_core.c）の範囲外にある具体的なフックの呼び出し順序や、他モジュールとの詳細な内部遷移フローの全容は、本コンテキストからは完全には特定されていません。*

重要な不変条件

  • **認証と認可の分離**: 認可処理（authz）は、認証処理（authn）によってユーザーの識別子が確定しているか、あるいはアノニマスアクセスが明示的に許可されていることを前提として動作します。
  • **プロバイダ登録のライフサイクル**: すべての認証・認可プロバイダは、Apacheの起動（初期化）フェーズにおいて、ap_register_provider を用いて対応するコアモジュール（authn または authz グループ）に登録されていなければなりません。

*注意: スレッドセーフティやメモリ管理に関する詳細な不変条件は、提供されたコンテキストからは完全には特定されていません。*

よくある落とし穴

  • **プロバイダの未ロード**: 設定ファイル（httpd.conf）で AuthBasicProvider file のようにプロバイダを指定していても、該当する実体モジュール（mod_authn_file など）がロードされていない場合、実行時にプロバイダが見つからずエラーとなります。
  • **認可コンテナの評価順序の誤解**: <RequireAll> や <RequireAny> などのコンテナをネストさせる際、評価の優先順位や否定条件（Require not）の挙動を誤解すると、意図しないアクセス許可や拒否が発生しやすくなります。

次に読む場所

  • include/mod_auth.h (プロバイダ構造体定義と共通APIの確認)
  • modules/aaa/mod_authn_core.c (認証プロバイダの登録とディレクティブ処理の実装)
  • modules/aaa/mod_authz_core.c (認可プロバイダの登録とRequireコンテナの評価ロジックの実装)
