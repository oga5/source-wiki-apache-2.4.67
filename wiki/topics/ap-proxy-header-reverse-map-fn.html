ap_proxy_header_reverse_map_fn

概念

ap_proxy_header_reverse_map_fn は、Apache HTTP Server（httpd）のプロキシ機能において、バックエンドサーバーから受信したレスポンスヘッダー（Location、Set-Cookie など）に含まれるURLやパスを、クライアント向けに適切に書き換える（リバースマッピングする）ためのコールバック関数を定義する関数ポインタ型（typedef）です。

なぜ重要か

リバースプロキシ構成（ProxyPassReverse など）において、バックエンドサーバーが返した内部向けのURLやCookieのドメイン・パス情報をそのままクライアントに送信すると、クライアントが正しくアクセスできなくなります。この関数ポインタ型を介して、Location や Set-Cookie などのヘッダー値を動的に書き換えることで、プロキシの透過性を維持し、クライアントがシームレスに通信できるようにします。

エンドツーエンドの流れ

1. **レスポンス受信**: プロキシモジュール（mod_proxy_http や mod_http2 のプロキシセッション）がバックエンドサーバーからレスポンスヘッダーを受信します。 2. **ヘッダーの走査**: 受信したヘッダーを処理する際（mod_proxy_http.c の process_proxy_header や h2_proxy_session.c のヘッダー処理ロジックなど）、定義された transform_hdrs テーブルを走査します。 3. **マッピング関数の適用**: Location、Content-Location、URI、Destination、Set-Cookie などの対象ヘッダーが見つかった場合、対応する ap_proxy_header_reverse_map_fn 型の関数（ap_proxy_location_reverse_map や ap_proxy_cookie_reverse_map）を呼び出します。 4. **ヘッダーの書き換え**: 関数によって書き換えられた新しいヘッダー値が、クライアントへのレスポンスに適用されます。

*注意: HTTP/2 プロキシセッション（h2_proxy_session.c）における直接的な呼び出し箇所など、一部の具体的な制御フローの詳細は提供されたソース抜粋外であるため、完全なエンドツーエンドの実行経路は未解明です。*

重要な不変条件

  • **生存期間の保証**: 呼び出し側は、この関数が返す変換後の文字列が、現在のリクエスト（request_rec *r）の生存期間中有効であることを期待します。
  • **テーブル管理**: transform_hdrs 配列において、対象となる特定のヘッダー名と、この関数型を実装する関数が必ずペアで登録されている必要があります。

よくある落とし穴

  • **シグネチャの不一致**: 提供された情報において、h2_proxy_session.c 側と mod_proxy_http.c 側で参照されている ap_proxy_header_reverse_map_fn のシグネチャ（引数の数や役割：url のみか、rem と loc の両方か）に差異がみられます。プラットフォームやバージョン、呼び出し規約（__stdcall など）の指定によるバイナリ互換性の問題に注意する必要があります。
  • **変換対象外の挙動**: 変換対象のヘッダー値が NULL や空文字列の場合、または変換ルールに一致しない場合の挙動は各実装関数に依存します（通常は元の値をそのまま返すか、変換を行いません）。
