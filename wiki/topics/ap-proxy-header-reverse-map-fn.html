ap_proxy_header_reverse_map_fn

概念

ap_proxy_header_reverse_map_fn は、Apache HTTP Server（httpd）のプロキシ機能において、バックエンドサーバーから返された特定の HTTP レスポンスヘッダー（Location、Set-Cookie、Content-Location など）の値を、クライアントから見て適切な形式（プロキシの外部公開URLなど）に逆マッピング（書き換え）するためのコールバック関数を定義する関数ポインタ型（typedef）です。

なぜ重要か

プロキシ環境において、バックエンドサーバーが自身の内部URLやドメインをレスポンスヘッダー（リダイレクト先を示す Location やクッキーの Set-Cookie など）に含めて返してしまうと、クライアントが直接バックエンドにアクセスしようとしてエラーになります。この関数型を介して適切なマッピング処理（ProxyPassReverse 設定に基づく変換など）を動的に適用することで、プロキシの透過性を維持し、クライアントとバックエンド間のシームレスな通信を保証します。

主要エンティティ

  • **ap_proxy_header_reverse_map_fn** (modules/proxy/mod_proxy.h:733 で定義される関数ポインタ型)
  • **symbol:modules/proxy/mod_proxy_http.c#ap_proxy_header_reverse_map_fn**: HTTP/1.1 プロキシ処理において、レスポンスヘッダーを書き換えるために参照されます。
  • **symbol:modules/http2/h2_proxy_session.c#ap_proxy_header_reverse_map_fn**: HTTP/2 プロキシセッションにおいて、同様のヘッダー書き換えを行うために参照されます。
  • **transform_hdrs**: mod_proxy_http.c および h2_proxy_session.c 内で定義されている静的構造体配列。ヘッダー名（Location、Set-Cookie など）と、この関数ポインタ型に適合する具体的な変換関数をペアで管理します。
  • **適合する実装関数**: ap_proxy_location_reverse_map や ap_proxy_cookie_reverse_map など。

エンドツーエンドの流れ

1. **レスポンス受信**: プロキシモジュール（mod_proxy_http または h2_proxy_session）がバックエンドサーバーからレスポンスヘッダーを受信します。 2. **ヘッダーの走査**: 受信したヘッダーを処理する際、各モジュール内に定義された transform_hdrs テーブルを走査し、変換対象のヘッダー名（Location, Content-Location, URI, Destination, Set-Cookie など）と一致するか確認します。 3. **コールバックの実行**: 一致するヘッダーが見つかった場合、対応する ap_proxy_header_reverse_map_fn 型の関数（ap_proxy_location_reverse_map など）を呼び出します。 4. **ヘッダー値の書き換え**: 関数は、リクエスト情報（request_rec）やプロキシ設定に基づき、バックエンドのURL/パスをクライアント向けの適切な値に書き換えて返します。 5. **クライアントへの送信**: 書き換えられたヘッダーがクライアントへ送信されます。

*注意: 提供されたソースコードの抜粋外にある詳細な制御フロー（具体的な呼び出しタイミングや、変換後のヘッダーがどのようにクライアントへ送出されるかの全体像）の完全なエンドツーエンドの流れは未確定です。*

重要な不変条件

  • **メモリの有効性**: この関数が返す変換後の文字列（const char *）は、該当するリクエスト（request_rec）の生存期間中、有効であり続けなければなりません。
  • **マッピングの対応関係**: transform_hdrs 配列において、対象となるヘッダー名と、この関数型に適合する実装関数が正しくペアとして登録されている必要があります。

よくある落とし穴

  • **NULL や空文字列の扱い**: 変換対象のヘッダー値が NULL または空文字列の場合の挙動は、呼び出される具体的な実装関数（ap_proxy_location_reverse_map など）に依存するため、呼び出し側での事前チェックや実装側での安全なハンドリングが必要です。
  • **呼び出し規約の不一致**: プラットフォームによっては __stdcall などの特定の呼び出し規約が指定されている定義が存在するため、関数を実装・登録する際はバイナリ互換性に注意する必要があります。
  • **変換ルールの不一致**: 変換ルールに一致しない場合、関数は元の値をそのまま返すか、変換を行わない設計となっているため、意図しないヘッダーがそのままクライアントに露出する可能性があります。
