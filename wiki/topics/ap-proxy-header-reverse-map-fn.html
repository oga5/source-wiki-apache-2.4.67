ap_proxy_header_reverse_map_fn

概念

ap_proxy_header_reverse_map_fn は、Apache HTTP Server（httpd）のプロキシ機能において、バックエンドサーバーから受信した HTTP レスポンスヘッダー（Location、Set-Cookie など）に含まれる URL やパスを、クライアント向けの適切な形式に書き換える（リバースマッピングする）ためのコールバック関数を定義する関数ポインタ型（typedef）です。

なぜ重要か

プロキシ環境（ProxyPassReverse などを使用する環境）において、バックエンドサーバーが返した内部的な URL や Cookie のパスをそのままクライアントに返すと、クライアントが正しくアクセスできなくなります。この関数ポインタ型を介して適切な変換ロジック（ap_proxy_location_reverse_map や ap_proxy_cookie_reverse_map など）を動的に適用することで、プロキシの透過性を維持し、一貫したクライアント通信を保証するために重要です。

エンドツーエンドの流れ

1. プロキシモジュール（mod_proxy_http や mod_http2 のプロキシセッション）がバックエンドサーバーからレスポンスヘッダーを受信します。 2. 受信したヘッダーを走査し、各モジュール内の transform_hdrs 配列に定義された対象ヘッダー（Location, Content-Location, URI, Destination, Set-Cookie など）と照合します。 3. 一致するヘッダーが存在する場合、登録されている ap_proxy_header_reverse_map_fn 型のコールバック関数（ap_proxy_location_reverse_map や ap_proxy_cookie_reverse_map）を呼び出し、ヘッダー値を書き換えます。

※ なお、詳細な制御フローの全体像や直接的な呼び出し箇所の完全なコードは、提供されたソース抜粋の範囲外であるため、エンドツーエンドのフローの詳細は完全には判明していません。

重要な不変条件

  • **生存期間の保証**: 呼び出し側は、この関数が返す変換後の文字列が、対象リクエスト（request_rec *r）の生存期間中、有効に保持されることを期待します。
  • **シグネチャの適合**: mod_proxy.h で定義されたインターフェースに適合している必要があります。

よくある落とし穴

  • **シグネチャの不一致**: 提供された情報において、h2_proxy_session.c 側と mod_proxy_http.c 側で関数の引数構成（url のみか、rem と loc の両方か）に関する記述に差異が見られます。プラットフォームやバージョンごとの正確な定義に注意する必要があります。
  • **未変換時の挙動**: 変換ルールに一致しない場合や、変換対象のヘッダー値が NULL または空文字列の場合の挙動は、各実装関数（ap_proxy_location_reverse_map 等）に依存します。
  • **呼び出し規約**: プラットフォームによっては __stdcall などの呼び出し規約が指定されている場合があり、バイナリ互換性に注意が必要です。
