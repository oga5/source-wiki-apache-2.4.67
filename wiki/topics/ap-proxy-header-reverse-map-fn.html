ap_proxy_header_reverse_map_fn

概念

ap_proxy_header_reverse_map_fn は、Apache HTTP Server (httpd) のプロキシ機能において、バックエンドサーバーから返された特定のレスポンスヘッダー（Location、Set-Cookie など）に含まれるURLやパスを、クライアントから見て適切な形式（プロキシの外部公開URLなど）に書き換える（リバースマップする）ためのコールバック関数を定義する関数ポインタ型（typedef）です。

なぜ重要か

プロキシ環境において、バックエンドサーバーが自身の内部URL（例: http://internal-backend/login）をレスポンスヘッダー（Location など）に含めて返した場合、クライアントはそのURLに直接アクセスできません。

この関数ポインタを介して、ProxyPassReverse などの設定に基づき、クライアントがアクセス可能なプロキシのURL（例: http://public-proxy/login）へ適切に書き換えることで、プロキシの透過性を維持します。また、HTTP/1.x プロキシ (mod_proxy_http) と HTTP/2 プロキシ (mod_http2) の両方で共通のインターフェースとして利用され、ヘッダー書き換えロジックを抽象化・共通化する役割を担っています。

    • HTTP/2 プロキシセッション処理において、レスポンスヘッダーの書き換えを行うために参照されます。

  • **symbol:modules/proxy/mod_proxy_http.c#ap_proxy_header_reverse_map_fn**

    • HTTP/1.x プロキシ処理において、バックエンドからのレスポンスヘッダーを書き換えるために参照されます。

  • **transform_hdrs**

    • h2_proxy_session.c および mod_proxy_http.c のそれぞれで定義されている静的構造体配列です。書き換え対象となるヘッダー名（Location, Content-Location, URI, Destination, Set-Cookie など）と、本関数ポインタ型に適合する具体的な変換関数（ap_proxy_location_reverse_map や ap_proxy_cookie_reverse_map）のペアを管理します。

エンドツーエンドの流れ

1. **レスポンスヘッダーの受信**: プロキシモジュール（mod_proxy_http や h2_proxy_session）がバックエンドサーバーからレスポンスヘッダーを受信します。 2. **ヘッダーの走査**: 受信したヘッダーを処理する際、各モジュール内の transform_hdrs テーブルを走査し、一致するヘッダー名があるか確認します。 3. **コールバックの実行**: 一致するヘッダーが見つかった場合、対応する ap_proxy_header_reverse_map_fn 型の関数ポインタ（例: ap_proxy_location_reverse_map）を呼び出します。 4. **ヘッダー値の書き換え**: 呼び出された関数は、リクエスト情報（request_rec *r）やマッピングルールに基づき、ヘッダー値（str）を書き換えて新しい文字列を返します。 5. **クライアントへの送信**: 書き換えられたヘッダーがクライアントへのレスポンスに適用されます。

※ なお、具体的な呼び出し元の完全なコードパスや、HTTP/2における詳細な呼び出しコンテキストの全容は、提供されたソース抜粋の範囲外であるため、一部未確定な部分があります。

重要な不変条件

  • **生存期間の保証**: この関数が返す変換後の文字列（const char *）は、リクエストの生存期間中（request_rec のプール内など）有効でなければなりません。
  • **シグネチャの一致**: プラットフォームによっては __stdcall などの呼び出し規約が指定されている定義も存在するため、バイナリ互換性を維持するために、実装する関数は mod_proxy.h で定義されたシグネチャと完全に一致している必要があります。

よくある落とし穴

  • **シグネチャの不一致**: 新しいマッピング関数を実装する際、引数の数や型、呼び出し規約が ap_proxy_header_reverse_map_fn の定義と異なると、コンパイルエラーや未定義動作の原因になります。
  • **テーブルへの登録漏れ**: 新しいヘッダー変換ルールを追加する場合、transform_hdrs 配列に新しいエントリと適切な関数ポインタを正しく追加しなければ、変換処理が実行されません。
  • **NULL/空文字のハンドリング**: 変換対象のヘッダー値が NULL や空文字列の場合の挙動は各実装関数に依存するため、呼び出し側および実装側での適切な境界条件の考慮が必要です。
