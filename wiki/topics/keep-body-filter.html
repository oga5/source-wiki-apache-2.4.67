keep_body_filter

概念

keep_body_filter は、HTTP リクエストボディをメモリ（r->kept_body）に一時的にコピーして保持するための入力フィルタです。これにより、通常は一度読み取ると消費されてしまうリクエストボディを、後続のサブリクエストや他のフィルタが再利用できるようになります。

なぜ重要か

Apache HTTP Server 内で、認証、リダイレクト、あるいはサブリクエストの処理など、同一リクエストのコンテキストにおいてリクエストボディを複数回参照・再利用したいユースケースが存在します。keep_body_filter は、設定されたサイズ制限の範囲内でボディデータを安全にキャッシュし、この再利用性を実現するために不可欠な役割を担っています。

関与するモジュール

  • module:include （ヘッダー定義 include/mod_request.h を提供）
  • module:modules/filters （フィルタの実装 modules/filters/mod_request.c を提供）

主要エンティティ

  • symbol:include/mod_request.h#KEEP_BODY_FILTER: フィルタを識別または制御するための定義。
  • symbol:modules/filters/mod_request.c#keep_body_filter: リクエストボディを複製・保持する入力フィルタの本体関数。

エンドツーエンドの流れ

1. **呼び出し**: Apache のフィルタチェーン（ap_get_brigade 経由）から keep_body_filter が呼び出されます。 2. **初期化**: 初回呼び出し時に keep_body_ctx_t を作成し、Content-Length を検証します。 3. **データ取得**: ap_get_brigade を通じて上流フィルタからデータを読み込みます。 4. **サイズ検証と複製**:

  • 読み込んだデータの合計が、設定された保持サイズ制限（dconf->keep_body）を超えていないか検証します。
  • 制限内であれば、apr_bucket_copy を使用してバケットを複製し、r->kept_body ブリゲードの末尾に追加します。

    • 5. **透過**: 取得した元のブリゲード b は、そのまま後続のフィルタへ透過的に渡されます。

*注意: フィルタがチェーンに挿入される具体的なトリガーや、保持された r->kept_body が後続のどのフェーズでどのように引き出されて再利用されるかという、完全なエンドツーエンドの統合フローは提供されたコンテキストからは明らかになっていません。*

重要な不変条件

  • **メモリのライフサイクル**: r->kept_body は r->pool（リクエストプール）に紐付けられており、リクエスト終了時に自動的に解放されます。
  • **サイズ制限の遵守**: 保持されるデータの合計サイズは、request_dir_conf で設定された keep_body の制限を超えてはなりません。超過した場合は、保持データが破棄されエラーが返されます。
  • **自己削除条件**: 既に r->kept_body が存在する場合、フィルタは自身をチェーンから削除し、以降は透過的に動作します。

よくある落とし穴

  • **メモリ消費の増大**: リクエストボディをメモリ上にコピーするため、KeptBodySize の設定値が大きすぎると、高並行環境下でメモリを過度に消費する可能性があります。
  • **サイズ制限超過によるエラー**: Content-Length が設定値を超えている場合、または読み込み中に制限を超えた場合、リクエストは即座に HTTP_REQUEST_ENTITY_TOO_LARGE で失敗します。
