KEEP_BODY_FILTER

概念

KEEP_BODY_FILTER は、HTTP リクエストボディをメモリ上に一時的にバッファリング（コピー）し、後続のサブリクエストや他のフィルタがリクエストボディを再利用できるようにするための入力フィルタです。

通常、HTTP リクエストボディは一度読み込むと消費されてしまいますが、このフィルタを適用することで、データを r->kept_body に保持し、複数回アクセスすることが可能になります。

なぜ重要か

Apache HTTP Server において、特定のモジュールやサブリクエストが元のリクエストボディを再度読み込む必要がある場合、標準の入力ストリームはすでに消費されているため読み込めません。KEEP_BODY_FILTER は、リクエストボディをメモリにキャッシュすることで、リクエストデータの「巻き戻し」や「再読み込み」を可能にし、モジュール間での柔軟なデータ共有を実現します。

関与するモジュール

  • module:include
  • module:modules/filters

主要エンティティ

  • **symbol:include/mod_request.h#KEEP_BODY_FILTER**: フィルタを識別するための定義。
  • **symbol:modules/filters/mod_request.c#keep_body_filter**: リクエストボディを r->kept_body にコピーする、実際の入力フィルタ関数。

エンドツーエンドの流れ

1. **フィルタの呼び出し**: Apache のフィルタチェーンから ap_get_brigade を経由して keep_body_filter が呼び出されます。 2. **初期化と検証**: 初回呼び出し時、フィルタは keep_body_ctx_t を作成し、Content-Length ヘッダーを検証します。 3. **データの取得**: 上流フィルタから ap_get_brigade を通じてデータを読み込みます。 4. **サイズ制限のチェック**: 読み込んだデータの合計サイズが、設定された制限（dconf->keep_body）を超えていないか検証します。 5. **データの複製と保持**: 読み込んだバケットを apr_bucket_copy で複製し、r->kept_body ブリゲードの末尾に追加します。 6. **データの透過**: 読み込んだオリジナルのブリゲード b は、そのまま下流のフィルタへ透過的に渡されます。

*注意: フィルタがどのような条件でフィルタチェーンに挿入されるか、またサブリクエスト側がどのように r->kept_body を再利用するかという全体的なエンドツーエンドの統合フローは、提供された情報からは完全には明らかになっていません。*

重要な不変条件

  • **メモリの生存期間**: r->kept_body はリクエストプール（r->pool）に紐付けられており、リクエスト終了時に自動的に解放されます。
  • **サイズ制限の遵守**: 読み込んだデータサイズが設定された制限（dconf->keep_body）を超過した場合、フィルタは即座にエラー（HTTP_REQUEST_ENTITY_TOO_LARGE）を返し、それまでに保持されたデータは破棄されます。
  • **重複動作の防止**: すでに r->kept_body が存在する場合、フィルタは自身をフィルタチェーンから削除し、以降は透過的に動作します。

よくある落とし穴

  • **メモリ消費の増大**: リクエストボディをメモリ上にコピーするため、大きなリクエストボディを扱うとメモリ消費が急増します。これを防ぐために KeptBodySize ディレクティブによる適切なサイズ制限の設定が必須です。
  • **エラー時のデータ消失**: データの読み込み中にサイズ制限を超過した場合、それまでにバッファリングされていたデータはすべてクリーンアップされ、破棄されます。
