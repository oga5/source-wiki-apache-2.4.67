ap_os_create_privileged_process

概念

ap_os_create_privileged_process は、Apache HTTP Server において、プラットフォーム固有のプロセス生成モデルの違いを抽象化し、特権プロセス（または標準プロセス）を生成するための共通インターフェースを提供する関数です。Unix、Windows、NetWare、OS/2 などの異なるオペレーティングシステム（OS）環境において、一貫したシグネチャでプロセス生成を呼び出せるように設計されています。

なぜ重要か

Apache HTTP Server はマルチプラットフォームで動作するため、CGI などの外部プロセスを起動する際に、OS ごとの権限管理やプロセス生成 API の違いを吸収する必要があります。

特に Unix 環境では、suexec によるユーザー/グループ権限の切り替え（特権プロセス生成）が必要となる一方、他の OS（Windows, NetWare, OS/2）では現時点で標準的な apr_proc_create の単純なラッパーとして実装されています。この関数が存在することで、mod_cgi や mod_cgid などの呼び出し元モジュールは、実行プラットフォームの差異を意識することなく安全にプロセスを生成できます。

エンドツーエンドの流れ

1. **呼び出しの発生**: modules/generators/mod_cgi.c や modules/generators/mod_cgid.c から、CGI スクリプトなどの外部プロセスを起動するために ap_os_create_privileged_process が呼び出されます。 2. **プラットフォームごとの分岐**:

  • **Unix 環境 (os/unix/unixd.c)**:

    • 1. ap_run_get_suexec_identity(r) を呼び出し、リクエストに関連付けられた suexec 用のユーザー/グループ識別子（ugid）を取得します。 2. ugid が取得できた場合は、ap_unix_create_privileged_process を呼び出して suexec 経由で特権プロセスを生成します。 3. ugid が NULL の場合は、標準の apr_proc_create を直接呼び出します。

  • **その他の環境 (Windows, NetWare, OS/2)**:

    • 1. 受け取った引数（newproc, progname, args, env, attr, p）をそのまま apr_proc_create に転送してプロセスを生成します。 3. **結果の返却**: プロセス生成の成否を示す apr_status_t ステータスコードが呼び出し元に返されます。

*注意: ap_unix_create_privileged_process の内部的な詳細な挙動や、suexec 実行時の OS レベルでの完全なエンドツーエンドの特権昇格フローの詳細は、提供された情報からは特定されていません。*

重要な不変条件

  • **引数の有効性**: 呼び出し元は、newproc（apr_proc_t）および attr（apr_procattr_t）に対して、適切に初期化または割り当てられた有効なメモリ領域を渡す必要があります。
  • **Unix 環境におけるフォールバック**: Unix 環境において ap_run_get_suexec_identity が NULL を返した場合、または suexec が適用不可な場合は、必ず通常の apr_proc_create によるプロセス生成にフォールバックします。
  • **非Unix環境における透過性**: Windows、NetWare、OS/2 環境においては、現時点で追加の特権処理を行わず、apr_proc_create の戻り値をそのまま透過的に返却します。

よくある落とし穴

  • **プラットフォーム間の機能差**: Unix 環境では suexec による実質的な権限変更（特権プロセス生成）が行われますが、Windows、NetWare、OS/2 環境では現時点で単なる apr_proc_create のラッパーに過ぎません。プラットフォーム間で「特権」のセマンティクスやセキュリティ境界が異なる点に注意が必要です。
  • **エラーハンドリングの依存**: 本関数自体は独自の詳細なエラーハンドリングを持たず、下位の apr_proc_create や ap_unix_create_privileged_process の戻り値（apr_status_t）に依存しているため、エラー原因の特定には APR 側のエラーコードを正しく解釈する必要があります。
