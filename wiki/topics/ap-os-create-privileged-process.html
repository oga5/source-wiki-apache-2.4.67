ap_os_create_privileged_process

概念

ap_os_create_privileged_process は、Apache HTTP Serverにおいて、プラットフォーム固有のセキュリティモデルやプロセス生成機構を抽象化し、特権プロセス（CGIスクリプトなど）を生成するための共通インターフェースです。プラットフォームごとの権限管理の差異を吸収し、統一的なAPIを提供します。

なぜ重要か

CGIなどの外部プロセスを起動する際、Unix系OSでは suexec を用いたユーザー/グループ権限の切り替えが必要となる一方、WindowsやNetWare、OS/2などの他プラットフォームでは異なるプロセス生成モデルが使用されます。この関数は、mod_cgi や mod_cgid などの呼び出し元モジュールに対してプラットフォーム固有の差異を隠蔽し、安全かつ一貫したプロセス生成を可能にするために重要です。

エンドツーエンドの流れ

1. **プロセスの起動要求**: modules/generators/mod_cgi.c または modules/generators/mod_cgid.c が、CGIプロセスの生成時に ap_os_create_privileged_process を呼び出します。 2. **プラットフォーム固有の処理**:

  • **Unix環境 (os/unix/unixd.c)**:

    • 1. ap_run_get_suexec_identity(r) を呼び出し、対象リクエストに対する suexec 用のユーザー/グループ識別子（ugid）を取得します。 2. ugid が取得できた場合、ap_unix_create_privileged_process を呼び出して suexec を経由した特権プロセス生成を行います。 3. ugid が NULL の場合（suexec が適用不可の場合）は、通常の apr_proc_create を直接呼び出します。

  • **Windows, NetWare, OS/2環境 (os/win32/util_win32.c, os/netware/util_nw.c, os/os2/util_os2.c)**:
  • 現時点の実装では、特権昇格などの追加処理は行わず、受け取った引数をそのまま apr_proc_create に転送してプロセスを生成します。

    • 3. **結果の返却**: 生成されたプロセス情報（apr_proc_t）と、実行結果を示すステータスコード（apr_status_t）を呼び出し元に返します。

*注意: 各プラットフォームにおける詳細なエラーハンドリングや、Unix以外のOSで将来的に特権制御がどのように拡張されるかといった、完全なエンドツーエンドのライフサイクルは現時点では完全には判明していません。*

重要な不変条件

  • **引数の有効性**: 呼び出し元は、有効な newproc（apr_proc_t）および attr（apr_procattr_t）を渡す必要があります。
  • **Unix環境でのフォールバック**: suexec の識別子（ugid）が取得できない場合は、必ず通常の apr_proc_create によるプロセス生成にフォールバックします。
  • **非Unix環境での透過性**: 現行の実装において、Unix以外のプラットフォームでは apr_proc_create の単純なラッパーとして機能し、追加のオーバーヘッドや独自の権限変更を行いません。

よくある落とし穴

  • **プラットフォーム間の挙動の不一致**: Unix環境では suexec による厳格な権限分離が適用される可能性がある一方、Windows、NetWare、OS/2環境では単なる apr_proc_create のラッパーであるため、プラットフォーム固有の特権昇格やセキュリティ制限が自動的に適用されるわけではありません。
  • **エラーコードの透過性**: 内部で発生したエラー（apr_proc_create や ap_unix_create_privileged_process の失敗）はそのまま呼び出し元に返されるため、呼び出し元で適切にハンドリングする必要があります。
