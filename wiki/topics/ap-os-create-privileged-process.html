ap_os_create_privileged_process

概念

ap_os_create_privileged_process は、Apache HTTP Server において、プラットフォーム固有のプロセス生成や特権管理の差異を抽象化し、リクエストに関連付けられた権限情報に基づいて外部プロセス（CGI など）を生成するための共通インターフェースです。

なぜ重要か

Apache HTTP Server はマルチプラットフォームで動作するため、CGI などの外部プロセスを起動する際に、プラットフォームごとに異なるセキュリティモデルやプロセス生成 API を扱う必要があります。たとえば、Unix 環境では suexec を用いた権限の切り替えが必要ですが、Windows や OS/2、NetWare ではそれぞれの OS に適したプロセス生成方法が存在します。

この関数は、それらのプラットフォーム固有の実装を隠蔽し、mod_cgi や mod_cgid などのジェネレータモジュールに対して統一されたプロセス生成 API を提供する役割を持つため重要です。

エンドツーエンドの流れ

1. **呼び出し元からの要求**: modules/generators/mod_cgi.c または modules/generators/mod_cgid.c から、リクエスト情報（request_rec）、プログラム名、引数、環境変数、プロセス属性（apr_procattr_t）などを伴って ap_os_create_privileged_process が呼び出されます。

2. **プラットフォームごとの分岐と実行**:

  • **Unix 環境 (os/unix/unixd.c)**:

    • 1. ap_run_get_suexec_identity(r) を呼び出し、対象リクエストに対する suexec 用のユーザー/グループ識別子（ugid）の取得を試みます。 2. ugid が取得できた場合、ap_unix_create_privileged_process を呼び出して suexec を経由した特権プロセス生成を行います。 3. ugid が NULL の場合、apr_proc_create を直接呼び出して標準的なプロセス生成を行います。

  • **NetWare (os/netware/util_nw.c) / OS/2 (os/os2/util_os2.c) / Windows (os/win32/util_win32.c) 環境**:
  • 現時点の実装では、受け取った引数をそのまま APR（Apache Portable Runtime）の apr_proc_create に転送してプロセスを生成する単純なラッパーとして動作します。

3. **結果の返却**: プロセス生成の結果（apr_status_t）が呼び出し元に返されます。

*注意: ap_unix_create_privileged_process の内部的な詳細や、各プラットフォームにおける完全なエラーハンドリングの全容は、提供された情報からは完全には明らかになっていません。*

重要な不変条件

  • **引数の有効性**:

    • 呼び出し元は、newproc（apr_proc_t）および attr（apr_procattr_t）に対して、適切に初期化または割り当てられた有効なポインタを渡さなければなりません。

  • **Unix 環境における suexec 判定**:

    • Unix 環境では、ap_run_get_suexec_identity フックの返却値（ugid）の有無によって、suexec 経由の特権生成か標準生成（apr_proc_create）かが厳密に分岐されます。

  • **非 Unix 環境におけるフォールバック**:

    • NetWare、OS/2、Windows 環境では、現時点で特権昇格や固有のプロセス生成ロジックは実装されておらず、APR の標準的なプロセス生成（apr_proc_create）に依存します。

よくある落とし穴

  • **プラットフォーム間のセキュリティ挙動の差異**:

    • Unix 環境では suexec による詳細な権限制御が行われますが、Windows、OS/2、NetWare 環境では単なる apr_proc_create のラッパーであるため、プラットフォーム間でセキュリティや実行権限の挙動が異なる点に注意が必要です。

  • **エラーコードの透過性**:

    • プロセス生成に失敗した場合、内部で呼び出される apr_proc_create や ap_unix_create_privileged_process のエラーコードがそのまま返されるため、呼び出し元はプラットフォーム依存のエラーを適切に解釈する必要があります。
