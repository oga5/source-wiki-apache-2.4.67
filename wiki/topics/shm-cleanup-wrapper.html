shm_cleanup_wrapper

概念

shm_cleanup_wrapper は、Apache HTTP Server のモジュールにおいて、APR (Apache Portable Runtime) プールのライフサイクルに合わせて共有メモリセグメントを安全に破棄するために用意されたクリーンアップ用コールバック関数です。mod_example_ipc と mod_lua の両モジュールで同名の関数が定義されています。

なぜ重要か

Apache HTTP Server のプロセス終了時やプールの破棄時に、割り当てられた共有メモリ（exipc_shm や lua_ivm_shm）が解放されずにシステムに残る（リソースリーク）のを防ぐために重要です。APR のプールクリーンアップ機構（apr_pool_cleanup_register）にこのラッパー関数を登録しておくことで、プール破棄時に自動的かつ安全に apr_shm_destroy が呼び出されます。

関与するモジュール

  • module:modules/examples
  • module:modules/lua

主要エンティティ

  • **symbol:modules/examples/mod_example_ipc.c#shm_cleanup_wrapper**: mod_example_ipc.c 内で定義されている静的（static）関数。グローバル変数 exipc_shm が指す共有メモリを破棄します。
  • **symbol:modules/lua/mod_lua.c#shm_cleanup_wrapper**: mod_lua.c 内で定義されている関数。グローバル変数 lua_ivm_shm が指す共有メモリを破棄します。

エンドツーエンドの流れ

1. **登録**: 各モジュールの初期化時等に、apr_pool_cleanup_register を用いて、対象のプールに shm_cleanup_wrapper をクリーンアップハンドラとして登録します。

  • mod_example_ipc.c では mod_example_ipc.c:186 付近で登録されます。
  • mod_lua.c では mod_lua.c:2059 付近で登録されます。

    • 2. **実行**: プールが破棄される際、APR プール管理機構から shm_cleanup_wrapper が呼び出されます。 3. **破棄**:

  • mod_example_ipc 側では、exipc_shm が非 NULL の場合に apr_shm_destroy を呼び出して共有メモリを破棄します。
  • mod_lua 側では、lua_ivm_shm が非 NULL の場合に apr_shm_destroy を呼び出して共有メモリを破棄します。
  • いずれの場合も、ポインタが NULL であれば何もせず OK を返します。

※なお、静的解析上、両モジュール間での相互参照（例: mod_lua.c から mod_example_ipc.c の同名関数への参照など）が検出されていますが、これが意図的なクロスモジュール呼び出しなのか、単なる同名シンボルによる誤検出（あるいはコードのコピー＆ペーストによるもの）なのか、詳細なエンドツーエンドの結合フローは完全には判明していません。

重要な不変条件

  • **ポインタの安全性**: 共有メモリのグローバルポインタ（exipc_shm または lua_ivm_shm）が NULL である場合は、apr_shm_destroy を呼び出さずに安全に OK を返さなければなりません。
  • **ライフサイクルの一貫性**: 本関数は直接手動で呼び出すのではなく、必ず APR プールのクリーンアップ登録を介して、プールの破棄に伴い自動的に実行される必要があります。

よくある落とし穴

  • **シンボル名の衝突と誤参照**: shm_cleanup_wrapper という名前が複数のモジュールで重複して定義されています。特に mod_lua.c の関数は static 宣言されていないため、ビルド環境や静的解析ツールにおいて意図しないモジュールの関数が参照されたり、シンボル衝突を引き起こしたりする可能性があります。
  • **二重解放の危険性**: グローバルポインタが指す共有メモリが、プールのクリーンアップ以外で手動で破棄され、かつポインタが NULL にクリアされていない場合、本関数内で apr_shm_destroy が再度呼び出され、不正なポインタアクセスやエラーが発生する恐れがあります。
