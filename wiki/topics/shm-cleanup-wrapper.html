shm_cleanup_wrapper

概念

shm_cleanup_wrapper は、Apache HTTP Server のモジュールにおいて、APR（Apache Portable Runtime）のプールライフサイクルに合わせて共有メモリセグメントを安全に破棄するためのクリーンアップ用ラッパー関数です。

なぜ重要か

Apache HTTP Server では、プロセスの終了時やプールの破棄時にリソースを適切に解放する必要があります。shm_cleanup_wrapper は、APR のクリーンアップ機構（apr_pool_cleanup_register）に登録されるコールバック関数として機能し、共有メモリの解放漏れを防ぐ役割を果たします。

関与するモジュール

  • module:modules/examples
  • module:modules/lua

主要エンティティ

  • **symbol:modules/examples/mod_example_ipc.c#shm_cleanup_wrapper**: mod_example_ipc 内のグローバル共有メモリ exipc_shm を破棄するための静的（static）関数。
  • **symbol:modules/lua/mod_lua.c#shm_cleanup_wrapper**: mod_lua 内のグローバル共有メモリ lua_ivm_shm を破棄するための関数。

エンドツーエンドの流れ

1. **登録**: 各モジュールの初期化時等に、apr_pool_cleanup_register を用いて、対象のプールに shm_cleanup_wrapper を登録します。

  • mod_example_ipc.c では modules/examples/mod_example_ipc.c:186 にて登録されます。
  • mod_lua.c では modules/lua/mod_lua.c:2059 にて登録されます。

    • 2. **実行**: プールが破棄される際、APR のクリーンアップ処理から本関数が呼び出されます。 3. **破棄**:

  • mod_example_ipc.c の実装では、グローバル変数 exipc_shm が非 NULL の場合に apr_shm_destroy を呼び出して共有メモリを破棄します。
  • mod_lua.c の実装では、グローバル変数 lua_ivm_shm が非 NULL の場合に apr_shm_destroy を呼び出して共有メモリを破棄します。

なお、モジュール間を跨ぐような直接的な呼び出しや、これら以外の詳細なエンドツーエンドの制御フローは現時点では未確定です。

重要な不変条件

  • **グローバルポインタの検証**: 共有メモリを破棄する前に、対象のグローバル変数（exipc_shm または lua_ivm_shm）が NULL でないことを確認する必要があります。NULL の場合は何もせず OK を返さなければなりません。
  • **シグネチャの一致**: APR のクリーンアップコールバックとして登録するため、引数は void * を受け取り、戻り値は apr_status_t である必要があります。

よくある落とし穴

  • **シンボル名の衝突と誤参照**: mod_example_ipc.c と mod_lua.c の両方に同名の shm_cleanup_wrapper が定義されています。静的解析において、mod_example_ipc.c から mod_lua.c のシンボルへの参照、あるいはその逆の参照が検出されることがありますが、これが意図的な依存関係なのか、単なるコードのコピー＆ペーストによる誤検出なのかは注意深く検証する必要があります。
  • **直接呼び出しの回避**: 本関数は APR プールのライフサイクル管理に委ねるコールバックとして設計されているため、手動で直接呼び出すべきではありません。
