shm_cleanup_wrapper

概念

shm_cleanup_wrapper は、Apache HTTP Server のモジュールにおいて、APR（Apache Portable Runtime）のプールクリーンアップ機構を介して共有メモリセグメントを安全に破棄するためのコールバック関数です。複数のモジュール（mod_example_ipc および mod_lua）で同名の関数が定義されており、それぞれのモジュールが管理するグローバルな共有メモリポインタを対象にクリーンアップ処理を行います。

なぜ重要か

Apache HTTP Server のプロセス終了時やプールの破棄時に、割り当てられた共有メモリリソースが解放されずに残る（リソースリークが発生する）のを防ぐために重要です。APR プールのライフサイクルにクリーンアップ処理を登録することで、明示的な解放処理を自動化し、安全なリソース管理を実現します。

関与するモジュール

  • module:modules/examples (mod_example_ipc.c)
  • module:modules/lua (mod_lua.c)

主要エンティティ

  • **symbol:modules/examples/mod_example_ipc.c#shm_cleanup_wrapper**: mod_example_ipc 内で定義されている静的（static）関数。グローバル変数 exipc_shm を破棄します。
  • **symbol:modules/lua/mod_lua.c#shm_cleanup_wrapper**: mod_lua 内で定義されている関数。グローバル変数 lua_ivm_shm を破棄します。

エンドツーエンドの流れ

1. **登録**: 各モジュールの初期化時等に、apr_pool_cleanup_register を使用して、対象の shm_cleanup_wrapper を APR プールのクリーンアップハンドラとして登録します。

  • mod_example_ipc.c では mod_example_ipc.c:186 付近で登録されます。
  • mod_lua.c では mod_lua.c:2059 付近で登録されます。

    • 2. **実行**: プールが破棄される際、APR のライフサイクル管理によって shm_cleanup_wrapper が自動的に呼び出されます。 3. **破棄**:

  • mod_example_ipc の場合: exipc_shm が非 NULL であれば apr_shm_destroy を呼び出して共有メモリを破棄します。
  • mod_lua の場合: lua_ivm_shm が非 NULL であれば apr_shm_destroy を呼び出して共有メモリを破棄します。
  • いずれの場合も、ポインタが NULL の場合は何もせず OK を返します。

※なお、両モジュール間での具体的な連携や、静的解析で検出された相互参照（例: mod_lua.c から mod_example_ipc.c のシンボルへの参照など）が、意図的なものであるか、あるいは単なるコードのコピー＆ペーストや誤検出によるものかといった詳細なエンドツーエンドの相互作用は完全には解明されていません。

重要な不変条件

  • **対象ポインタの検証**: 共有メモリの破棄を実行する前に、対象となるグローバルポインタ（exipc_shm または lua_ivm_shm）が非 NULL であることを確認しなければなりません。
  • **APR クリーンアップのシグネチャ適合**: APR プールのクリーンアップハンドラとして登録するため、引数として void * を受け取り、apr_status_t を返すシグネチャを維持する必要があります（渡される引数は関数内では使用されません）。

よくある落とし穴

  • **シンボルの衝突と誤参照**: mod_lua.c の shm_cleanup_wrapper には static 修飾子が付与されていないため、グローバルスコープでシンボルが衝突したり、他モジュールから誤って参照されたりするリスクがあります。
  • **二重解放の危険性**: 共有メモリが既に破棄されている、または不正なポインタである状態で apr_shm_destroy が呼び出されると、エラーや予期しない動作を引き起こす可能性があります。ただし、ポインタが適切に管理（NULL 初期化など）されていれば、安全に OK が返されます。
