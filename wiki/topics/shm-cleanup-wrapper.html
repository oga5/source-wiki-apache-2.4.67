shm_cleanup_wrapper

概念

shm_cleanup_wrapper は、Apache HTTP Server のモジュールにおいて、割り当てられた共有メモリ（SHM）セグメントを安全に破棄するために用意されたクリーンアップ用ラッパー関数です。APR（Apache Portable Runtime）のプール管理機構と連携し、プールの破棄やサーバー終了のタイミングで自動的に共有メモリを解放するコールバック関数として機能します。

なぜ重要か

Apache HTTP Server では、プロセスやスレッド間でデータを共有するために共有メモリが使用されます。プロセスの終了時やモジュールのアンロード時に共有メモリが適切に解放されないと、システムリソースのリークが発生します。shm_cleanup_wrapper は、APR プールのライフサイクル（apr_pool_cleanup_register）に共有メモリの破棄処理（apr_shm_destroy）を登録することで、リソースの確実な自動解放を保証するために重要です。

関与するモジュール

本機能には以下のモジュールが関与しています。

  • module:modules/examples (modules/examples/mod_example_ipc.c)
  • module:modules/lua (modules/lua/mod_lua.c)

主要エンティティ

本トピックに関連する主要なエンティティは以下の通りです。

  • **symbol:modules/examples/mod_example_ipc.c#shm_cleanup_wrapper**: mod_example_ipc モジュール内のクリーンアップ関数。グローバル変数 exipc_shm で管理される共有メモリセグメントを対象とします。
  • **symbol:modules/lua/mod_lua.c#shm_cleanup_wrapper**: mod_lua モジュール内のクリーンアップ関数。グローバル変数 lua_ivm_shm で管理される共有メモリセグメントを対象とします。

エンドツーエンドの流れ

1. **登録**: 各モジュールの初期化フェーズにおいて、apr_pool_cleanup_register を呼び出し、プールに対して shm_cleanup_wrapper をクリーンアップハンドラとして登録します。

  • mod_example_ipc.c では modules/examples/mod_example_ipc.c:186 にて登録されます。
  • mod_lua.c では modules/lua/mod_lua.c:2059 にて登録されます。

    • 2. **実行**: 登録先のプールが破棄される際（サーバーの終了時や再起動時など）、APR プール機構から shm_cleanup_wrapper が呼び出されます。 3. **破棄**:

  • mod_example_ipc.c の場合、グローバル変数 exipc_shm が非 NULL であれば apr_shm_destroy を呼び出して共有メモリを破棄します。
  • mod_lua.c の場合、グローバル変数 lua_ivm_shm が非 NULL であれば apr_shm_destroy を呼び出して共有メモリを破棄します。

    • 4. **完了**: 共有メモリの破棄に成功した場合は apr_shm_destroy の戻り値を返し、対象の共有メモリが存在しない（NULL である）場合は OK を返します。

*注意: 両モジュール間での相互参照（例: mod_example_ipc.c から mod_lua.c のシンボルへの参照、またはその逆）が一部のインデックスで検出されていますが、これが意図的な共有利用であるか、あるいは単なるシンボル名の衝突やコードのコピー＆ペーストによるものかなど、詳細なエンドツーエンドの依存関係は完全には判明していません。*

重要な不変条件

  • **ポインタの有効性**: クリーンアップ実行時に、対象となるグローバル変数（exipc_shm または lua_ivm_shm）は、有効な共有メモリセグメントを指しているか、あるいは NULL でなければなりません。
  • **ライフサイクルの一貫性**: 本関数は直接呼び出されるのではなく、必ず apr_pool_cleanup_register を介して登録され、APR プールのライフサイクル管理下で呼び出される必要があります。

よくある落とし穴

  • **二重解放や不正ポインタ**: 対象の共有メモリが既に他の処理によって破棄されている、あるいは初期化されていない不正なポインタを保持している場合、apr_shm_destroy の呼び出しによりエラーや予期せぬ動作が発生する可能性があります。
  • **シンボル名の衝突と誤参照**: mod_example_ipc.c と mod_lua.c の両方で同名の shm_cleanup_wrapper が定義されています。静的解析やビルドの設定によっては、意図しないモジュール側の関数が参照されたり、コードのコピー＆ペーストによる誤った依存関係が生じたりするリスクがあります。変更を加える際は、影響範囲に注意が必要です。
