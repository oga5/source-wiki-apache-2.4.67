shm_cleanup_wrapper

概念

shm_cleanup_wrapper は、Apache HTTP Server のモジュールにおいて、APR（Apache Portable Runtime）プールの破棄時に共有メモリセグメントを安全に解放するためのクリーンアップ用ラッパー関数です。

なぜ重要か

Apache のプロセス終了時やプールのライフサイクル終了時に、割り当てられた共有メモリリソースが解放されずにシステムに残存する（リソースリーク）のを防ぐために重要です。APR プールのクリーンアップ機構と連携することで、共有メモリの自動かつ確実な破棄を保証します。

関与するモジュール

  • module:modules/examples (mod_example_ipc.c)
  • module:modules/lua (mod_lua.c)

主要エンティティ

  • **symbol:modules/examples/mod_example_ipc.c#shm_cleanup_wrapper**
  • mod_example_ipc モジュールで定義されているクリーンアップ関数。グローバル変数 exipc_shm が指す共有メモリセグメントを対象とします。
  • **symbol:modules/lua/mod_lua.c#shm_cleanup_wrapper**
  • mod_lua モジュールで定義されているクリーンアップ関数。グローバル変数 lua_ivm_shm が指す共有メモリセグメントを対象とします。

エンドツーエンドの流れ

1. **登録**: モジュールの初期化時等に、apr_pool_cleanup_register を用いて、対象の APR プールに shm_cleanup_wrapper をクリーンアップコールバックとして登録します。

  • mod_example_ipc.c 内の登録箇所: mod_example_ipc.c:186
  • mod_lua.c 内の登録箇所: mod_lua.c:2059

    • 2. **実行**: プールが破棄される際、APR クリーンアップ機構から本関数が呼び出されます。 3. **判定と破棄**:

  • 関数内部で、対象のグローバルポインタ（exipc_shm または lua_ivm_shm）が非 NULL であるかを確認します。
  • ポインタが存在する場合、apr_shm_destroy を呼び出して共有メモリを破棄し、そのステータスを返します。
  • ポインタが NULL の場合は、何も行わずに OK を返します。

*注意: 提供されたインデックス情報において、mod_example_ipc.c から mod_lua.c の同名関数への参照、あるいはその逆の参照が検出されています。これが意図的なモジュール間共有なのか、あるいは単なる静的解析上の誤検出（同名シンボルの混同）なのか、正確なエンドツーエンドの依存関係は完全には確定していません。*

重要な不変条件

  • **ポインタの有効性**: 対象となるグローバル変数（exipc_shm または lua_ivm_shm）は、有効な共有メモリセグメントを指しているか、あるいは NULL でなければなりません。
  • **ライフサイクル管理**: 本関数は直接手動で呼び出すのではなく、登録された APR プールの破棄プロセスを通じてのみ呼び出される必要があります。

よくある落とし穴

  • **シンボル名の衝突と誤参照**:

    • 両モジュールで全く同じ shm_cleanup_wrapper という名前の関数が定義されています。これにより、ビルド構成やリンクの順序によっては、意図しないモジュールの関数が呼び出されたり、シンボルの衝突が発生したりするリスクがあります。

  • **二重解放と不正ポインタ**:

    • 共有メモリが既に別の場所で破棄されている、あるいはポインタが不正な状態のまま apr_shm_destroy が呼び出されると、エラーが発生する可能性があります。
