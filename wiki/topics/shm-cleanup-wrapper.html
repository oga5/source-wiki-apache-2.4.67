shm_cleanup_wrapper

概念

shm_cleanup_wrapper は、Apache HTTP Server のモジュールにおいて、APR（Apache Portable Runtime）のプールライフサイクルに合わせて共有メモリセグメントを安全に破棄するためのクリーンアップ用ラッパー関数です。

なぜ重要か

Apache HTTP Server のモジュール（mod_example_ipc や mod_lua など）が共有メモリを使用する場合、プロセスの終了時やプールの破棄時に共有メモリを確実に解放する必要があります。この関数を APR のクリーンアップ処理（apr_pool_cleanup_register）に登録しておくことで、メモリリークや共有メモリセグメントの残留を防ぐことができます。

関与するモジュール

  • module:modules/examples
  • module:modules/lua

主要エンティティ

  • **symbol:modules/examples/mod_example_ipc.c#shm_cleanup_wrapper**: mod_example_ipc モジュール内のクリーンアップ関数。グローバル変数 exipc_shm（共有メモリブロックへのポインタ）を対象とします。
  • **symbol:modules/lua/mod_lua.c#shm_cleanup_wrapper**: mod_lua モジュール内のクリーンアップ関数。グローバル変数 lua_ivm_shm（Lua IVM 用の共有メモリセグメントへのポインタ）を対象とします。

エンドツーエンドの流れ

1. **登録**: 各モジュールの初期化時などに、apr_pool_cleanup_register を用いて shm_cleanup_wrapper を APR プールのクリーンアップハンドラとして登録します。

  • mod_example_ipc.c では mod_example_ipc.c:186 付近で登録されます。
  • mod_lua.c では mod_lua.c:2059 付近で登録されます。

    • 2. **実行**: 登録された APR プールが破棄される際、APR 内部から shm_cleanup_wrapper が自動的に呼び出されます。 3. **破棄処理**:

  • 関数内部で、対象となるグローバルな共有メモリポインタ（exipc_shm または lua_ivm_shm）が非 NULL であるか確認します。
  • ポインタが存在する場合、apr_shm_destroy を呼び出して共有メモリセグメントを破棄し、そのステータスを返します。
  • ポインタが NULL の場合は、何も行わずに OK を返します。

なお、モジュール間を跨ぐ詳細な呼び出し関係や、両モジュール間でのシンボルの相互参照（例: mod_lua.c から mod_example_ipc.c のシンボルへの参照、またはその逆）が意図的なものであるか、あるいは単なる静的解析上の名前衝突やコードのコピー＆ペーストによるものかなど、完全なエンドツーエンドの結合フローは現時点では確定していません。

重要な不変条件

  • **ポインタの安全性**: 対象となるグローバル変数（exipc_shm または lua_ivm_shm）は、有効な共有メモリセグメントを指しているか、あるいは NULL である必要があります。
  • **ライフサイクル管理**: 本関数は直接呼び出すのではなく、APR プールのクリーンアップ機構を介して呼び出される必要があります。

よくある落とし穴

  • **シンボル名の衝突と誤参照**: mod_example_ipc.c と mod_lua.c の両方に同名の shm_cleanup_wrapper が定義されています。静的解析やビルドの設定によっては、これらが意図せず相互に参照されているように見える、あるいは衝突する可能性があります。コードの変更やリファクタリングを行う際は、影響範囲に注意が必要です。
  • **二重解放の危険性**: グローバルポインタが指す共有メモリが、APR プールのクリーンアップ以外の場所で手動で破棄され、かつポインタが NULL にクリアされていない場合、apr_shm_destroy が不正なポインタに対して呼び出されエラーやクラッシュを引き起こす可能性があります。
