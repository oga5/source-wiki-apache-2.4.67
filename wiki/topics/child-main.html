child_main

概念

child_main は、Apache HTTP Serverのマルチプロセッシングモジュール（MPM）において、新しく生成された子プロセス（またはWindows環境における子プロセススレッド）のメイン処理を実行する関数です。各MPM（event, prefork, winnt, worker）の並行処理モデルに応じた初期化、スレッド生成、接続の受け入れ、および終了処理を管理します。

なぜ重要か

child_main は、クライアントからのリクエストを実際に処理する実行コンテキスト（プロセスやスレッド）のライフサイクルを制御するエントリーポイントであるため重要です。スコアボードの再オープン、シグナル処理のセットアップ、リスナーやワーカースレッドの生成、プロセスのクリーンアップなど、サーバーの安定稼働に直結する基盤処理を担っています。

エンドツーエンドの流れ

各MPMにおいて、child_main は make_child または winnt_run から呼び出されます。しかし、プロセス生成から child_main への遷移、およびリクエスト処理の具体的なループからプロセス終了に至る詳細なエンドツーエンドの制御フローは、現時点では完全には判明していません。

呼び出し先シンボルから推測される各MPMの主な処理の流れは以下の通りです：

  • **event / worker (スレッド駆動モデル)**:

    • 1. child_init による初期化、ap_reopen_scoreboard によるスコアボードの再オープン。 2. ap_fatal_signal_child_setup による致命的シグナルの設定。 3. ap_thread_create や setup_threads_runtime を使用したワーカースレッドの生成と実行。 4. join_workers によるスレッドの待機と、clean_child_exit による終了処理。

  • **prefork (プロセス駆動モデル)**:

    • 1. child_init およびスコアボードの再オープン。 2. accept_mutex_on / accept_mutex_off による排他制御を用いた接続の受け入れ。 3. clean_child_exit による終了処理。

  • **winnt (Windows環境)**:

    • 1. create_listener_thread によるリスナースレッドの作成。 2. ap_signal_parent による親プロセスへの通知。

重要な不変条件

現時点では、各MPMにおける child_main 実行中の具体的な前提条件や不変条件（例：排他制御のロック状態、スレッド数の上限、スコアボードの整合性など）は解析されておらず、明らかになっていません。

よくある落とし穴

現時点では、具体的なソースコードのロジックやエラーハンドリングの解析が行われていないため、開発や運用における具体的な落とし穴（リソースリーク、デッドロック、シグナルハンドリングの不整合など）は特定されていません。
