just_die

概念

just_die は、Apache HTTP Server (httpd) のマルチプロセッシングモジュール (MPM) において、シグナル（SIGHUP や SIGTERM など）を受信した際に子プロセスを安全かつ正常に終了させるためのシグナルハンドラ関数です。

なぜ重要か

子プロセスがシグナルを受信した際、単に強制終了するのではなく、clean_child_exit(0) を呼び出してクリーンアップ処理を適切に行い、正常終了（ステータス0）させるために重要です。これにより、リソースのリークを防ぎ、安全なプロセスのライフサイクル管理を実現します。

エンドツーエンドの流れ

1. 子プロセスの初期化時、apr_signal を介して SIGHUP や SIGTERM などのシグナルハンドラとして just_die が登録されます。 2. 子プロセスが対象のシグナルを受信すると、シグナルハンドラとして just_die(int sig) が呼び出されます。 3. just_die は受信したシグナル番号を無視し、内部で clean_child_exit(0) を呼び出します。 4. clean_child_exit により子プロセスのクリーンアップ処理が実行され、プロセスが終了します。

なお、event および worker MPM における具体的な登録・呼び出しのライフサイクルや詳細なエンドツーエンドの流れは、現時点では完全に明らかになっていません。

重要な不変条件

  • **制御の非返却**: just_die が呼び出された後、プロセスは clean_child_exit を経由して終了するため、呼び出し元へ制御は戻りません。
  • **正常終了の保証**: 関数名に die と含まれていますが、終了ステータスは 0（正常終了）として扱われます。

よくある落とし穴

  • **名称による誤解**: just_die という名称から異常終了や即時強制終了を連想しがちですが、実際には clean_child_exit(0) を呼び出すため、正常終了として処理されます。
  • **MPM間の挙動の差異**: prefork、worker、event の各 MPM で同様のシグナルハンドラとして定義・参照されていますが、それぞれの MPM における clean_child_exit の具体的なクリーンアップ挙動や影響範囲の違いに注意する必要があります。
