Request Processing Pipeline & Hook Registration

概念

Request Processing Pipeline & Hook Registration は、Apache HTTP Server においてクライアントからのリクエストを処理するための一連のフェーズ（パイプライン）と、各モジュールがそのフェーズに独自の処理を挿入するためのフック登録メカニズムを指します。

モジュールはフックを介してリクエストの解析、アクセス制御、コンテンツ生成などの各段階に介入し、コアエンジン（server）およびプロトコルハンドラ（modules/http）が全体の実行フローとエラーハンドリングを統括します。

なぜ重要か

Apache の極めて高いモジュール性と拡張性は、このパイプラインとフック機構によって実現されています。リクエストのライフサイクルが明確なフェーズに分割され、各モジュールが標準化されたフック（AP_IMPLEMENT_HOOK_RUN_ALL など）を介して協調動作することで、安全かつ一貫したリクエスト処理と、エラー発生時の適切なハンドリング（ap_die などによるエラーレスポンス生成）が可能になります。

関与するモジュール

  • module:modules/http
  • module:server

主要エンティティ

  • symbol:include/ap_hooks.h#AP_IMPLEMENT_HOOK_RUN_ALL: 登録されたすべてのフックハンドラを順次実行するためのフックマクロ。
  • symbol:modules/http/http_request.c#ap_process_request: HTTP リクエスト処理の標準的なエントリポイント。主要な処理の実行と、最終的な接続バッファのフラッシュを担います。
  • symbol:modules/http/http_request.c#ap_die: リクエスト処理中に発生したエラーを処理し、適切なエラーレスポンスを生成するための汎用ハンドラ。
  • symbol:server/config.c#create_server_config: サーバー設定（ap_conf_vector_t）を生成する内部関数。

エンドツーエンドの流れ

提供された情報からは、リクエストの受信からフックの完全な実行順序、およびレスポンス送信にいたるエンドツーエンドの全容は未だ完全には解明されていません。現時点で判明している部分的な流れは以下の通りです。

1. **設定の初期化**: サーバーの起動または設定読み込み時に、init_server_config から create_server_config が呼び出され、サーバー設定ベクトルが構築されます。 2. **リクエストの受付と実行**: modules/http/http_core.c などの呼び出し元から ap_process_request が呼び出され、リクエスト処理が開始されます。 3. **非同期処理の駆動**: ap_process_request は内部で ap_process_async_request を呼び出し、登録されたフック（AP_IMPLEMENT_HOOK_RUN_ALL 等で実装されたものを含む）やハンドラを実行します。 4. **バッファの強制フラッシュ**: 接続に未処理の入力データがない場合、出力フィルタチェーンに対してフラッシュバケットを送信し、レスポンスデータをクライアントへ強制的にフラッシュします。 5. **エラーハンドリング**: パイプラインの実行中にエラーが発生した場合、ap_die が呼び出され、内部関数 ap_die_r を介して適切なエラーレスポンスが生成・送信されます。

重要な不変条件

  • **リクエスト構造体の有効性**: ap_process_request および ap_die を呼び出す際、引数として渡される request_rec *r は有効な構造体でなければなりません。
  • **ステータスコードの事前設定**: ap_die を呼び出す前に、r->status に適切なエラーレスポンスコードが設定されている必要があります（ap_die は内部で r->status を参照して ap_die_r を呼び出すため）。
  • ※ フックの登録順序やマルチスレッド環境における排他制御などの詳細な不変条件は、現時点では未解明です。

よくある落とし穴

  • **再帰的なエラー処理の発生**: ap_die はエラー処理中にさらにエラーが発生した場合に再帰的に呼び出されるリスクがあります。呼び出し元のフィルタやハンドラでは、無限再帰を避けるための制御が必要です。
  • **フィルタチェーンの状態変化**: エラー発生時にはフィルタチェーンの状態が変更される可能性があるため、ap_die 呼び出し後のフロー制御やリソース解放には注意が必要です。
  • **フラッシュ時のタイムアウト**: ap_process_request 内でのフラッシュ処理（ap_pass_brigade）がタイムアウト（APR_TIMEUP）を返した場合、情報ログは出力されますが、処理自体はそのまま継続されます。クライアントへの到達が保証されない境界条件に留意してください。

次に読む場所

  • include/ap_hooks.h: フックマクロ（AP_IMPLEMENT_HOOK_RUN_ALL など）の定義と、フック登録システムの詳細。
  • include/http_request.h: リクエスト処理パイプラインに関連する公開APIおよび構造体の宣言。
  • modules/http/http_request.c: ap_process_request や ap_die などの具体的なパイプライン制御・エラー処理の実装。
  • server/config.c: create_server_config を含む、サーバー設定の生成と初期化プロセス。
