Authentication and Authorization (AAA) Framework

概念

Apache HTTP ServerにおけるAAA（Authentication, Authorization, Accounting）フレームワークは、認証（Authentication: ユーザーが誰であるかの確認）と認可（Authorization: ユーザーが特定のリソースにアクセスできるかの判定）を分離・協調させるためのモジュール式アーキテクチャです。

認証プロバイダー（authn_provider）と認可プロバイダー（authz_provider）という共通のインターフェースを定義することで、Basic認証などの具体的な認証方式や、様々なアクセス制御ルール（Require ディレクティブなど）をプラグイン可能な形で統合します。

なぜ重要か

AAAフレームワークが重要な理由は、認証・認可のロジックをコアシステムから分離し、疎結合なプロバイダーモデルを提供することにあります。これにより、以下が可能になります： 1. **拡張性**: 開発者は authn_provider や authz_provider インターフェースを実装するだけで、新しい認証ソース（LDAP、データベース、カスタムファイル等）や認可ルールを容易に追加できます。 2. **柔軟な設定**: 管理者は、Basic認証（mod_auth_basic）などの認証仲介モジュールを介して、複数の認証プロバイダーをチェーン化したり、認可ルールを柔軟に組み合わせたりできます。

関与するモジュール

提供された情報に基づくと、以下のモジュールが関与しています。

  • module:include: 共通のプロバイダーインターフェース（mod_auth.h）を定義します。
  • module:modules/aaa: 認証・認可のコアロジックおよび具体的なプロバイダー（mod_auth_basic.c, mod_authn_core.c, mod_authz_core.c など）を実装します。

主要エンティティ

  • **symbol:include/mod_auth.h#authn_provider (struct)**: 認証プロバイダーが実装すべきインターフェース。パスワード検証を行う check_password や、レルムベースのハッシュを取得する get_realm_hash などの関数ポインタを定義します。
  • **symbol:include/mod_auth.h#authz_provider (struct)**: 認可プロバイダーが実装すべきインターフェース。Require 設定行を解析する parse_require_line と、アクセス可否を判定する check_authorization を定義します。
  • **symbol:modules/aaa/mod_auth_basic.c#authenticate_basic_user (function)**: HTTP Basic認証において、クライアントから送られた資格情報を抽出し、登録された authn_provider リストを順次呼び出して認証を仲介する関数。
  • **symbol:modules/aaa/mod_authz_core.c#authorize_user_core (function)**: ユーザーの認可処理を司るコア関数（詳細な内部ロジックは未解析）。

エンドツーエンドの流れ

提供された情報から推測される大まかな流れは以下の通りですが、**詳細なエンドツーエンドの制御フロー（特に認証から認可への完全な遷移や、mod_authz_core の詳細な内部処理）は完全には明らかになっていません。**

1. **設定解析時**:

  • サーバー起動時または設定再読み込み時に、認可プロバイダー（authz_provider）の parse_require_line が呼び出され、Require ディレクティブの構文チェックと最適化が行われます。

    • 2. **リクエスト受信時（認証フェーズ）**:

  • ap_hook_check_authn フックにより、authenticate_basic_user などの認証仲介関数が呼び出されます。
  • authenticate_basic_user は、リクエストからユーザー名とパスワードを抽出し、設定された authn_provider（例: check_password）を順次呼び出して資格情報を検証します。

    • 3. **リクエスト受信時（認可フェーズ）**:

  • 認証成功後、authorize_user_core などを通じて認可処理が実行されます。
  • 登録された authz_provider の check_authorization が呼び出され、リクエストが認可基準を満たしているか（AUTHZ_GRANTED など）が判定されます。

重要な不変条件

提供された情報に基づく不変条件は以下の通りですが、**フレームワーク全体におけるすべての不変条件や詳細な状態遷移ルールは完全には明らかになっていません。**

  • **認証プロバイダーの応答**: authn_provider の各関数は、成否に応じて適切な authn_status を返す必要があります。
  • **Basic認証の前提条件**: authenticate_basic_user が実行される際、AuthType が Basic であり、かつ AuthName が設定されていなければなりません（AuthName が未設定の場合は HTTP_INTERNAL_SERVER_ERROR となります）。
  • **認可設定の事前解析**: authz_provider の parse_require_line は設定読み込み時に一度だけ実行され、リクエスト処理時の check_authorization ではその解析済みデータ（parsed_require_line）が再利用されます。

よくある落とし穴

  • **AuthName の設定漏れ**: Basic認証を使用する際、AuthName（レルム名）が設定されていないと、authenticate_basic_user は HTTP_INTERNAL_SERVER_ERROR を返して処理を中断します。
  • **権限委譲（Authoritative）の設定**: authoritative 設定が有効な場合、プロバイダーでの認証失敗は即座に HTTP_UNAUTHORIZED となり、後続のプロバイダーへ処理が回りません。意図せず他の認証手段へのフォールバックがブロックされる原因となります。
  • **関数ポインタの NULL ハンドリング**: authn_provider や authz_provider を実装する際、オプションの関数ポインタ（parse_require_line など）が NULL の場合、呼び出し側が適切にハンドリングする必要がありますが、実装漏れがあると予期せぬ動作を引き起こす可能性があります。

次に読む場所

  • include/mod_auth.h: authn_provider および authz_provider のインターフェース定義を確認する。
  • modules/aaa/mod_auth_basic.c: authenticate_basic_user によるBasic認証の仲介処理とプロバイダー呼び出しの具体例を理解する。
  • modules/aaa/mod_authn_core.c: 認証プロバイダーの登録と管理の仕組みを調べる。
  • modules/aaa/mod_authz_core.c: authorize_user_core を含む認可処理のコアロジックと、Require ディレクティブの評価フローを追跡する。
