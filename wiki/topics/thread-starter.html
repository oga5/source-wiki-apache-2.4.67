thread_starter

概念

thread_starter は、Apache HTTP Server のマルチプロセッシングモジュール（MPM）である Event MPM および Worker MPM において、子プロセス内のスレッド（リスナーおよびワーカー）を生成・管理するための情報を保持するコンテナ構造体（struct）です。

なぜ重要か

この構造体は、子プロセス起動時にスレッドプールを初期化・管理するための共通コンテキストとして機能します。リスナーおよびワーカー用スレッドの生成、管理、および終了処理に必要な情報（スレッドハンドル、属性、プロセス識別子など）を1つにまとめ、スレッド生成関数（create_listener_thread など）へ安全かつ一貫して引き渡す役割を担うため重要です。

関与するモジュール

  • module:server/mpm/event
  • module:server/mpm/worker

主要エンティティ

  • **symbol:server/mpm/event/event.c#thread_starter**: Event MPM におけるスレッド生成・管理用構造体。接続管理スレッドのライフサイクルと密接に関連します。
  • **symbol:server/mpm/worker/worker.c#thread_starter**: Worker MPM におけるスレッド生成・管理用構造体。child_main などでインスタンス化されます。

両構造体は以下の同一のシグネチャを持ちます：

typedef struct {
    apr_thread_t **threads;
    apr_thread_t *listener;
    int child_num_arg;
    apr_threadattr_t *threadattr;
} thread_starter;

エンドツーエンドの流れ

1. **初期化**: 子プロセスのメイン処理（Worker MPM では child_main など）において、apr_palloc 等を用いて thread_starter 構造体のメモリが割り当てられ、各フィールドが初期化されます。 2. **スレッド生成の準備**: ワーカー数に応じた threads 配列のメモリ確保や、APRのスレッド属性 threadattr の設定が行われます。 3. **スレッドの起動**: create_listener_thread などの関数にこの構造体が引数として渡されます。 4. **スレッド生成**: 構造体内の情報を元に、APRのスレッド生成関数（apr_thread_create 等）が呼び出され、リスナーおよびワーカー用スレッドが生成されます。 5. **管理と終了**: 生成されたスレッドのハンドル（threads や listener）を通じて、スレッドの管理や終了処理が制御されます。

※詳細なスレッド生成の内部アルゴリズムや、エラー発生時の詳細なリカバリーフローなど、エンドツーエンドの完全なライフサイクルは提供された情報からは未解明です。

重要な不変条件

  • **メモリ割り当て**: 構造体自体および threads 配列は、スレッド生成処理に渡される前に apr_palloc 等によって適切にメモリが確保されていなければなりません。
  • **属性の初期化**: threadattr は、APR（Apache Portable Runtime）のスレッド属性として適切に初期化されている必要があります。
  • **識別子の保持**: child_num_arg には、対象となる子プロセスの識別番号が正しく設定されている必要があります。

よくある落とし穴

  • **メモリ割り当て失敗**: apr_palloc によるメモリ確保に失敗した場合、構造体へのアクセスは未定義動作を引き起こします。
  • **初期化不足**: threadattr や threads 配列が正しく設定されないままスレッド生成処理に渡されると、スレッドの起動に失敗する可能性があります。
  • **モジュール間の混同**: Event MPM と Worker MPM で同名の構造体が定義されており、設計思想は共通していますが、それぞれのモジュール内でのメモリ管理やライフサイクル（特に Event MPM における接続管理スレッドのライフサイクル）が異なるため、変更時にはそれぞれのモジュールへの影響を個別に確認する必要があります。
