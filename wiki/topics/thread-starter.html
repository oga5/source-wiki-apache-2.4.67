thread_starter

概念

thread_starter は、Apache HTTP Server (httpd) のマルチプロセッシングモジュール (MPM) である **Event MPM** および **Worker MPM** において、子プロセス内のスレッド（リスナーおよびワーカー）を生成・管理するためのコンテキスト情報を保持する構造体（struct）です。

なぜ重要か

この構造体は、子プロセスの初期化時にリスナースレッドやワーカースレッドを生成・制御するために必要な情報を一元管理します。スレッド属性（threadattr）やスレッドハンドル（threads, listener）、子プロセスの識別子（child_num_arg）をまとめてスレッド生成関数（create_listener_thread など）に渡すことで、各MPMにおける並行処理の基盤となるスレッドプールの構築を可能にします。

関与するモジュール

  • module:server/mpm/event
  • module:server/mpm/worker

主要エンティティ

  • symbol:server/mpm/event/event.c#thread_starter
  • symbol:server/mpm/worker/worker.c#thread_starter

両モジュールにおいて、以下の同一のシグネチャを持つ構造体として定義されています。

typedef struct {
    apr_thread_t **threads;
    apr_thread_t *listener;
    int child_num_arg;
    apr_threadattr_t *threadattr;
} thread_starter;

エンドツーエンドの流れ

1. **初期化とインスタンス化**

  • Worker MPM では child_main 関数などで thread_starter がインスタンス化され、apr_palloc 等を用いてメモリが割り当てられます。
  • Event MPM でも同様に、子プロセス起動時にスレッドプールを初期化する際のコンテキストとして使用されます。

    • 2. **スレッド生成の準備**

  • 構造体内の threadattr（スレッド属性）や threads（ワーカー用スレッド配列）が適切に初期化されます。
  • この構造体が create_listener_thread などの関数に引数として渡されます。

    • 3. **スレッドの起動**

  • create_listener_thread 等の内部で、本構造体の情報を元に apr_thread_create などの APR 関数が呼び出され、リスナーおよびワーカースレッドが生成・管理されます。

※ 詳細なスレッドのライフサイクルや終了処理の全体的なエンドツーエンドの流れは、提供されたコンテキストからは完全には明らかになっていません。

重要な不変条件

  • **メモリ確保**: threads 配列は、生成されるワーカー数に応じたメモリ領域が事前に確保されている必要があります。また、構造体自体も apr_palloc 等により適切にメモリ割り当てされている必要があります。
  • **属性の初期化**: threadattr は、APR のスレッド属性として適切に初期化されている必要があります。
  • **スレッド生成前の初期化**: スレッド生成処理（create_listener_thread 等）に渡される前に、構造体のすべてのフィールドが適切に初期化されている必要があります。

よくある落とし穴

  • **メモリ割り当ての失敗**: apr_palloc 等によるメモリ割り当てが失敗した場合、構造体へのアクセスは未定義動作となります。
  • **不適切な初期化によるスレッド起動失敗**: threadattr や threads 配列が正しく設定されていない場合、スレッドの起動（apr_thread_create 等）に失敗する可能性があります。
  • **モジュール間の独立性**: event MPM と worker MPM で同名の構造体が定義されていますが、それぞれ独立したファイル（event.c と worker.c）で定義されており、各モジュール内でのメモリ管理やライフサイクル、接続管理ロジック（特に Event MPM 特有の接続管理）に注意して個別に保守する必要があります。
