thread_starter

概念

thread_starter は、Apache HTTP Server (httpd) のマルチプロセッシングモジュール (MPM) である event および worker において、子プロセス内のスレッド（リスナーおよびワーカー）を生成・管理するためのコンテキスト情報を保持する構造体（struct）です。

なぜ重要か

この構造体は、子プロセスの起動時にスレッドプールを初期化する際の中心的なデータコンテナとして機能します。リスナースレッドやワーカースレッドの生成に必要な属性（threadattr）や、生成されたスレッドのハンドル（threads、listener）、プロセス固有の識別子（child_num_arg）を1つの構造体にまとめて管理・共有することで、スレッド生成フロー（create_listener_thread などの関数への引数渡し）を簡素化します。

関与するモジュール

  • module:server/mpm/event
  • module:server/mpm/worker

主要エンティティ

  • symbol:server/mpm/event/event.c#thread_starter
  • symbol:server/mpm/worker/worker.c#thread_starter

両モジュールにおいて、以下の同一のシグネチャを持つ構造体として定義されています。

typedef struct {
    apr_thread_t **threads;
    apr_thread_t *listener;
    int child_num_arg;
    apr_threadattr_t *threadattr;
} thread_starter;

エンドツーエンドの流れ

1. **インスタンス化と初期化**: 子プロセスのメイン処理（worker MPM の child_main など）において、apr_palloc 等を用いて本構造体のメモリが割り当てられ、インスタンス化されます。 2. **コンテキストの設定**: 生成されるワーカー数に応じた threads 配列の確保、APR のスレッド属性 threadattr の初期化、および子プロセス識別子 child_num_arg の設定が行われます。 3. **スレッド生成の実行**: 本構造体は create_listener_thread などの関数に引数として渡されます。内部で apr_thread_create などの APR 関数が呼び出され、リスナーおよびワーカースレッドが起動されます。

※スレッド起動後の詳細なライフサイクル管理や、エラー発生時の詳細なリカバリーなど、エンドツーエンドの完全なフローは現時点では完全には判明していません。

重要な不変条件

  • **メモリの事前確保**: threads 配列および構造体自体は、スレッド生成処理に渡される前に適切にメモリが割り当てられ、初期化されていなければなりません。
  • **属性の有効性**: threadattr は、APR のスレッド属性として適切に初期化されている必要があります。

※スレッド間の同期や排他制御に関する詳細な不変条件は、現時点では完全には判明していません。

よくある落とし穴

  • **メモリ割り当て失敗**: apr_palloc 等によるメモリ確保が失敗した場合、構造体へのアクセスは未定義動作となります。
  • **初期化不足による起動失敗**: threadattr や threads 配列が正しく設定されないままスレッド生成処理に渡されると、スレッドの起動に失敗する可能性があります。
  • **モジュール間の混同**: event MPM と worker MPM で同名の構造体が定義されています。設計思想は共通していますが、各モジュール内でのメモリ管理やライフサイクル（特に event MPM 特有の接続管理スレッドのライフサイクル）が異なるため、コード変更時にはそれぞれのモジュールへの影響を個別に確認する必要があります。
