cmd_parms

概念

cmd_parms は、Apache HTTP Server の設定ファイル解析フェーズにおいて、各設定ディレクティブのハンドラー関数（command_rec 内で定義される関数）へ、現在の解析状態や設定コンテキストを伝達するための構造体（cmd_parms_struct のエイリアス）です。

なぜ重要か

設定解析エンジンと各モジュール固有の設定ハンドラーとの間で、コンテキスト情報を安全かつ統一的に受け渡す役割を担うためです。これにより、ハンドラーは現在の設定ファイル上の位置や、適切なメモリプール（pconf や temp_pool）にアクセスし、設定値をモジュール固有の構造体に正しく反映させることができます。

関与するモジュール

  • module:include
  • module:server

主要エンティティ

  • symbol:include/http_config.h#cmd_parms: 構造体のエイリアス定義。設定解析のライフサイクルにおける一時的なコンテキストを表現します。
  • symbol:server/core.c#cmd_parms: http_config.h の定義を参照し、コア設定や各ディレクティブハンドラーとの連携に利用されます。

エンドツーエンドの流れ

1. 設定ファイル解析エンジン（ap_walk_config や server/config.c など）が設定ツリーを走査します。 2. 各ディレクティブに対応するハンドラーを呼び出す際、現在の解析状態、設定ファイル位置、メモリプールなどのコンテキスト情報を格納した cmd_parms 構造体が第一引数として引き渡されます。 3. ハンドラー（例: ap_set_listenbacklog などの設定関数や ap_expr_parse_cmd_mi などの式評価関数）は、cmd_parms を通じて設定コンテキストを読み取り、モジュール固有の設定構造体（core_dir_config や core_server_config など）に設定値を適用します。 4. 解析に失敗した場合、ハンドラーはエラーメッセージを返し、cmd_parms を介して設定プロセスに通知されます。

※なお、呼び出し元と呼び出し先の詳細なエラーハンドリングの境界条件など、一部のエンドツーエンドのフローの詳細は提供されたソース情報からは完全には判明していません。

重要な不変条件

  • **メモリ管理のライフサイクル**: 設定データは主に設定プール（pconf）に割り当てられます。解析中に一時的に使用される temp_pool は、リクエスト処理が開始される前に破棄される必要があります。
  • **有効なコンテキストの保証**: ディレクティブハンドラーに渡される際、cmd_parms は有効な設定コンテキスト（プール、サーバー設定、ディレクトリ設定など）を指している必要があります。

よくある落とし穴

  • **ライフサイクルの誤用**: cmd_parms は設定解析のライフサイクルにおいて一時的なコンテキストを保持するための構造体です。そのため、ハンドラーの外部でこの構造体を保持したり、再利用したりすることは避ける必要があります。
  • **ポインタの検証不足**: 構造体内のポインタが NULL である場合、設定適用時にセグメンテーション違反が発生する可能性があります。また、デバッグビルド以外では、構造体内のポインタやインデックスに対する範囲外アクセス保護マクロが省略される場合があります。
  • **構造体変更の影響範囲**: cmd_parms のメンバを変更すると、この構造体を参照しているすべてのモジュールの設定ハンドラーに影響が及ぶため、変更の際は注意が必要です。
