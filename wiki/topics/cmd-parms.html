cmd_parms

概念

cmd_parms は、Apache HTTP Server の設定ファイル解析プロセスにおいて、各設定ディレクティブのハンドラー関数（command_rec 内で定義）に現在の解析状態やコンテキスト情報を伝達するための構造体（cmd_parms_struct のエイリアス）です。

なぜ重要か

設定ファイルの解析時に、現在の設定ファイルの位置、プール情報、サーバー設定、ディレクトリ設定などのコンテキスト情報を一元的に保持し、各モジュールの設定ハンドラーへ安全かつ一貫した方法で引き渡す役割を担うためです。これにより、モジュール固有の構造体への設定値の反映や、メモリ管理、エラーハンドリングが可能になります。

関与するモジュール

  • module:include
  • module:server

主要エンティティ

  • symbol:include/http_config.h#cmd_parms : cmd_parms_struct のエイリアスとしての定義。
  • symbol:server/core.c#cmd_parms : http_config.h の定義を参照し、設定ディレクティブハンドラーの引数として利用。

エンドツーエンドの流れ

1. ap_walk_config などの設定ファイル解析エンジンが設定ツリーを走査します。 2. 各ディレクティブに対応するハンドラーを呼び出す際、現在の解析状態や設定コンテキストを保持する cmd_parms が第一引数として引き渡されます。 3. ハンドラー（例: ap_set_listenbacklog などの設定ハンドラー関数や、ap_expr_parse_cmd_mi などの式評価関数）は、cmd_parms を通じて現在の設定状態やプール情報にアクセスし、設定値をモジュール固有の構造体（core_dir_config や core_server_config など）に反映します。 4. 解析結果やエラーメッセージが戻り値として返され、エラーが発生した場合は cmd_parms を通じて設定プロセスに通知されます。

※なお、呼び出し元と呼び出し先の詳細なエラーハンドリングの境界条件など、エンドツーエンドの完全なフローの全容は提供された情報からは完全には判明していません。

重要な不変条件

  • 設定データは主に pconf（設定プール）に割り当てられます。
  • 解析中に使用される temp_pool は、リクエスト処理開始前に破棄されます。
  • ディレクティブハンドラーに渡される際、cmd_parms は有効な設定コンテキストを指している必要があります。
  • 構造体内のポインタやインデックスに対する範囲外アクセス保護は、デバッグビルド以外ではマクロ実装により省略される場合があります。

よくある落とし穴

  • cmd_parms は設定解析のライフサイクルにおいて一時的なコンテキストを保持するためのものであるため、ハンドラー外での保持や再利用は避ける必要があります。
  • 構造体内のポインタが NULL である場合、設定の適用時にセグメンテーション違反が発生する可能性があります。
  • cmd_parms の構造体メンバを変更すると、全モジュールの設定ハンドラーに影響を与えるため、保守・変更時には注意が必要です。
