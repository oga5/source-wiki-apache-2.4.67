cmd_parms

概念

cmd_parms は、Apache HTTP Serverの設定ファイル解析（ディレクティブ解析）時に、各ディレクティブのハンドラー関数へ現在の解析状態や設定コンテキストを伝達するための構造体（struct cmd_parms_struct の typedef エイリアス）です。

なぜ重要か

設定ファイルの解析プロセスにおいて、各モジュールの設定ハンドラーが現在の設定ファイル位置、メモリプール、およびモジュール固有の設定データ（ap_conf_vector_t など）にアクセスし、設定値を適切に反映させるための共通のコンテキスト情報を提供する役割を担うため重要です。

関与するモジュール

  • module:include
  • module:server

主要エンティティ

  • symbol:include/http_config.h#cmd_parms: struct cmd_parms_struct のエイリアス定義。設定解析のコンテキストを保持します。
  • symbol:server/core.c#cmd_parms: http_config.h の定義を参照し、設定ディレクティブを処理する関数（command_rec のハンドラ）の第一引数として使用されます。

エンドツーエンドの流れ

1. 設定ファイル解析エンジン（ap_walk_config など）が設定ツリーを走査します。 2. 各ディレクティブに対応するハンドラー（command_rec で定義された関数。例: ap_set_listener や ap_set_listenbacklog など）を呼び出す際、現在の解析状態や設定コンテキストを保持する cmd_parms が第一引数として引き渡されます。 3. ハンドラーは cmd_parms を通じて現在の設定状態やプール情報にアクセスし、設定値をモジュール固有の構造体（core_dir_config や core_server_config など）に反映させます。 4. 必要に応じて、ap_expr_parse_cmd_mi などの式評価関数にも cmd_parms が渡されます。

※解析エンジン内部の完全な遷移や、呼び出し元と呼び出し先の詳細なエラーハンドリングの境界条件など、エンドツーエンドのすべてのフローは提供された情報からは完全には判明していません。

重要な不変条件

  • 設定データは主に pconf（設定プール）に割り当てられます。
  • 解析中に使用される temp_pool は、リクエスト処理開始前に破棄されます。
  • ディレクティブハンドラに渡される際、cmd_parms は有効な設定コンテキストを指している必要があります。
  • 設定解析のライフサイクルにおいて、この構造体は一時的なコンテキストを保持するため、ハンドラー外での保持や再利用は避ける必要があります。

よくある落とし穴

  • 構造体内のポインタが NULL である場合、設定の適用時にセグメンテーション違反が発生する可能性があります。
  • 構造体内のポインタやインデックスに対する範囲外アクセス保護は、デバッグビルド以外ではマクロ実装により省略される場合があります。
  • 構造体のメンバ変更は、全モジュールの設定ハンドラに影響を与えるため、変更の際は注意が必要です。
