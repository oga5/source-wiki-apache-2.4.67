cmd_parms

概念

cmd_parms は、Apache HTTP Server の設定ファイル（ディレクティブ）解析時に、各ディレクティブのハンドラー関数へコンテキスト情報を提供するための構造体（struct cmd_parms_struct の typedef エイリアス）です。

なぜ重要か

設定ディレクティブを処理する関数（command_rec 内で定義されたハンドラー）が、現在の設定コンテキスト（メモリプール、サーバー設定、ディレクトリ設定など）や解析状態を読み取り、モジュール固有の設定構造体に値を安全かつ一貫して反映させるための共通インターフェースとして機能するためです。

関与するモジュール

  • module:include
  • module:server

主要エンティティ

  • **symbol:include/http_config.h#cmd_parms**: 構造体のエイリアス定義。設定解析エンジンや各モジュールのハンドラーで共通して利用されます。
  • **symbol:server/core.c#cmd_parms**: 設定ファイル解析中にディレクティブの引数やコンテキスト情報を保持・伝達するために参照されます。

エンドツーエンドの流れ

1. 設定ファイル解析エンジン（ap_walk_config など）が設定ツリーを走査します。 2. 各ディレクティブに対応するハンドラーを呼び出す際、現在の解析状態や設定コンテキストを保持する cmd_parms が第一引数として引き渡されます。 3. ハンドラー（例: ap_set_listener や ap_expr_parse_cmd_mi など）は、この構造体を通じて現在の設定ファイル位置やプール情報にアクセスし、設定値をモジュール固有の構造体に反映させます。 4. ハンドラーは解析結果やエラーメッセージを戻り値として返します。

※なお、呼び出し元と呼び出し先の詳細なエラーハンドリングの境界条件など、完全なエンドツーエンドのフローの詳細は本ソース情報からは完全には判明していません。

重要な不変条件

  • 設定データは主に pconf（設定プール）に割り当てられます。
  • 解析中に使用される temp_pool は、リクエスト処理開始前に破棄されます。
  • ディレクティブハンドラーに渡される際、cmd_parms は有効な設定コンテキストを指している必要があります。

よくある落とし穴

  • **一時的なライフサイクル**: cmd_parms は設定解析のライフサイクルにおいて一時的なコンテキストを保持するため、ハンドラー外での保持や再利用は避ける必要があります。
  • **NULLポインタ**: 構造体内のポインタが NULL である場合、設定の適用時にセグメンテーション違反が発生する可能性があります。
  • **範囲外アクセス**: 構造体内のポインタやインデックスに対する範囲外アクセス保護は、デバッグビルド以外ではマクロ実装により省略される場合があります。
  • **波及影響**: cmd_parms の構造体メンバを変更すると、全モジュールの設定ハンドラーに影響を与えるため、変更には注意が必要です。
