cmd_parms

概念

cmd_parms は、Apache HTTP Server の設定ファイル解析（ディレクティブ解析）時に、各ディレクティブのハンドラー関数へ現在の解析状態や設定コンテキストを伝達するための構造体（cmd_parms_struct のエイリアス）です。

なぜ重要か

設定解析エンジンと、各モジュールで定義された個別の設定ハンドラー関数（例: ap_set_listener や ap_expr_parse_cmd_mi など）との間で、設定ファイルの位置、メモリプール、設定データへのアクセスベクトルなどのコンテキスト情報を統一的に受け渡す役割を担うためです。この構造体の定義は、全モジュールの設定ハンドラーのインターフェースに直接影響を与えます。

関与するモジュール

  • module:include
  • module:server

主要エンティティ

  • symbol:include/http_config.h#cmd_parms : cmd_parms_struct のエイリアスとしての構造体定義。
  • symbol:server/core.c#cmd_parms : コア設定ディレクティブの処理や、core_dir_config / core_server_config への設定適用時に参照される定義。

エンドツーエンドの流れ

1. ap_walk_config などの設定解析エンジンが設定ツリーを走査します。 2. 各ディレクティブに対応するハンドラーを呼び出す際、現在の解析状態や設定コンテキストを保持する cmd_parms 構造体が第一引数としてハンドラーに引き渡されます。 3. ハンドラー（例: ap_set_listenbacklog などの設定ハンドラー関数や ap_expr_parse_cmd_mi などの式評価関数）は、cmd_parms を通じて現在の設定状態やプール情報にアクセスし、モジュール固有の設定構造体に値を反映させます。 4. ハンドラーは解析結果やエラーメッセージを戻り値として返します。

※なお、設定解析エンジン（server/config.c など）の内部実装や、呼び出し元と呼び出し先の間における詳細なエラーハンドリングの境界条件など、エンドツーエンドの完全なフローの詳細は提供された情報からは未確認です。

重要な不変条件

  • 設定データは主に pconf（設定プール）に割り当てられます。
  • 解析中に使用される temp_pool は、リクエスト処理開始前に破棄されます。
  • ディレクティブハンドラーに渡される際、cmd_parms は有効な設定コンテキストを指している必要があります。

よくある落とし穴

  • **一時的なライフサイクル**: cmd_parms は設定解析のライフサイクルにおいて一時的なコンテキストを保持するものであるため、ハンドラー外での保持や再利用は避ける必要があります。
  • **NULLポインタ**: 構造体内のポインタが NULL である場合、設定の適用時にセグメンテーション違反が発生する可能性があります。
  • **範囲外アクセス**: 構造体内のポインタやインデックスに対する範囲外アクセス保護は、デバッグビルド以外ではマクロ実装により省略される場合があります。
  • **広範な影響**: 構造体のメンバを変更すると、すべてのモジュールの設定ハンドラーに影響が及ぶため、変更の際は注意が必要です。
