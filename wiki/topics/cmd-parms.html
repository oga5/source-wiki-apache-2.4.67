cmd_parms

概念

cmd_parms（実体は struct cmd_parms_struct の typedef エイリアス）は、Apache HTTP Server の設定ファイル解析プロセスにおいて、各設定ディレクティブのハンドラー関数へ現在の解析状態やコンテキスト情報を伝達するための構造体です。

なぜ重要か

設定ディレクティブを処理するハンドラー関数（command_rec 内で定義）が、現在の設定ファイルの位置、メモリプール（pconf や temp_pool）、サーバー設定、ディレクトリ設定などのコンテキスト情報にアクセスし、モジュール固有の構造体に設定値を正しく反映させるための共通インターフェースとして機能するためです。

関与するモジュール

  • module:include
  • module:server

主要エンティティ

  • symbol:include/http_config.h#cmd_parms：構造体のエイリアス定義。
  • symbol:server/core.c#cmd_parms：設定ディレクティブの処理やコンテキスト保持における利用。

エンドツーエンドの流れ

1. **設定解析の開始**: ap_walk_config などの設定解析エンジンが設定ツリーを走査します。 2. **ハンドラーの呼び出し**: 各ディレクティブに対応するハンドラー関数を呼び出す際、現在の解析状態や設定コンテキストを保持する cmd_parms が第一引数として引き渡されます。 3. **設定の適用**: ハンドラー（例: ap_set_listenbacklog などの設定関数や ap_expr_parse_cmd_mi などの式評価関数）は、cmd_parms を通じてメモリプールや設定状態を読み取り、設定値を適用します。 4. **結果の返却**: ハンドラーは解析結果やエラーメッセージを戻り値として返します。

*注意: 呼び出し元と呼び出し先の間における詳細なエラーハンドリングの境界条件など、完全なエンドツーエンドのフローの詳細は本ソース情報からは未確定です。*

重要な不変条件

  • **メモリ管理のライフサイクル**: 設定データは主に pconf（設定プール）に割り当てられます。解析中に使用される temp_pool は、リクエスト処理開始前に破棄されます。
  • **コンテキストの有効性**: ディレクティブハンドラーに渡される際、cmd_parms は有効な設定コンテキストを指している必要があります。
  • **ポインタの妥当性**: 構造体内のポインタが NULL である場合、設定適用時にセグメンテーション違反が発生する可能性があります。

よくある落とし穴

  • **一時的なライフサイクル**: cmd_parms は設定解析フェーズの一時的なコンテキストを保持するものであるため、ハンドラーのスコープ外で保持したり再利用したりすることは避ける必要があります。
  • **構造体変更の影響範囲**: cmd_parms のメンバを変更すると、全モジュールの設定ハンドラーに影響を与えるため、定義の変更には注意が必要です。
  • **範囲外アクセスのリスク**: 構造体内のポインタやインデックスに対する範囲外アクセス保護は、デバッグビルド以外ではマクロ実装により省略される場合があります。
