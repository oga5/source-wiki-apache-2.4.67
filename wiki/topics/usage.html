usage

概念

usage は、Apache HTTP Server（httpd）本体、各種サポートユーティリティ、およびテストツールにおいて、コマンドライン引数の誤りやヘルプオプションの指定を検知した際に呼び出される共通の関数パターンです。正しいコマンド構文や利用可能なオプション一覧（ヘルプメッセージ）を標準エラー出力やログシステムに表示し、プログラムを終了させる役割を持ちます。

なぜ重要か

各実行バイナリが提供するコマンドラインインターフェース（CLI）の仕様をユーザーに提示し、不正な引数による誤動作を防ぐために重要です。また、ビルド時のマクロ定義（例: APR_FILES_AS_SOCKETS や APU_HAVE_*）やプラットフォーム（WIN32, NETWARE 等）に応じた動的なオプションの有効・無効状態をユーザーに正しく伝える役割も担っています。

エンドツーエンドの流れ

各バイナリにおける一般的な実行フローは以下の通りです。

1. **引数の解析**: main 関数や引数チェック関数（check_args など）で、コマンドライン引数（argv）を解析します。 2. **異常またはヘルプ要求の検知**: 不正なオプション、必須引数の不足、またはヘルプオプション（-h など）が指定されたことを検知します。 3. **usage の呼び出し**: 各バイナリ固有の usage 関数が呼び出されます。 4. **情報の出力**:

  • server/main.c では ap_log_error（APLOG_STARTUP）を使用し、プラットフォーム固有のオプションを含めて出力します。
  • サポートツールやテストツールでは、apr_file_printf（errfile 宛て）や標準ライブラリの fprintf（stderr 宛て）を使用して、オプション一覧やビルド時のサポート状況（例: httxt2dbm での DBM バックエンドの有効状態）を出力します。

    • 5. **プロセスの終了**: exit(1)、exit(ERR_SYNTAX)、または destroy_and_exit_process を呼び出し、呼び出し元に制御を戻さずにプロセスを終了させます。

*注意: 個々のツールにおける詳細な呼び出し条件や、すべてのツールにおける完全なエンドツーエンドのフローは、提供された情報からは一部未解明です。*

重要な不変条件

  • **プロセスの強制終了**: usage 関数が呼び出された場合、プログラムは必ず終了し、呼び出し元の処理へ制御が戻ることはありません。
  • **出力先の確保**: 出力に使用されるファイルストリーム（stderr、errfile、またはログシステム）が有効に初期化されている必要があります。例えば server/main.c では process_rec が NULL でないこと、support ツール群では errfile が初期化済みであることが前提となります。

よくある落とし穴

  • **手動更新の漏れ**: コマンドラインオプションを追加・変更した際、対応する usage 関数内のヘルプメッセージやフォーマット文字列を手動で更新しないと、実際の動作とヘルプ表示に乖離が生じます。
  • **プラットフォーム/ビルド依存の不整合**: rotatelogs の APR_FILES_AS_SOCKETS や httxt2dbm の APU_HAVE_* マクロ、server/main.c の WIN32/NETWARE 定義など、ビルド環境によって表示されるオプションが変化するため、条件付きコンパイルの整合性に注意する必要があります。
  • **セグメンテーション違反**: server/main.c#usage において、引数の process_rec が NULL の場合、process->argv[0] へのアクセスによりセグメンテーション違反が発生する危険性があります。
