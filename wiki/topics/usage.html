usage

概念

usage は、Apache HTTP Server（httpd）本体、およびその周辺のサポートツールやテストユーティリティにおいて、コマンドライン引数のヘルプメッセージ（使用方法）を出力するための関数群です。

共通のライブラリ関数として一元化されているわけではなく、各プログラム（httpd 本体、fcgistarter、htpasswd、rotatelogs など）のソースコード内に個別の usage 関数として定義されています。それぞれのプログラムがサポートする独自のコマンドラインオプションや動作モードに応じたヘルプメッセージを出力するように実装されています。

なぜ重要か

ユーザーや管理者がコマンドラインからツールを誤った引数で実行した際、または -h などのヘルプオプションを指定した際に、適切な使用方法を提示して安全かつ即座にプロセスを終了させるために重要です。

また、新しいコマンドラインオプションが追加された場合には、実際の引数解析ロジックと usage 内のヘルプメッセージの整合性を手動で維持する必要があるため、メンテナンスの観点からも重要な変更対象となります。

主要エンティティ

各モジュールおよびファイルにおける usage 関数の定義は以下の通りです。

  • **symbol:server/main.c#usage**: httpd 本体のヘルプメッセージを APLOG_STARTUP（標準エラー出力）に表示し、destroy_and_exit_process を呼び出して終了します。
  • **symbol:support/fcgistarter.c#usage**: fcgistarter ユーティリティの使用方法を stderr に出力し、exit(EXIT_FAILURE) で終了します。
  • **symbol:support/htdigest.c#usage**: htdigest コマンドの使用方法を errfile に出力し、exit(1) で終了します。
  • **symbol:support/htpasswd.c#usage**: htpasswd コマンドの使用方法を errfile に出力し、exit(ERR_SYNTAX) で終了します。
  • **symbol:support/httxt2dbm.c#usage**: httxt2dbm コマンドの使用方法と、ビルド時に有効な DBM フォーマット（SDBM, GDBM 等）のサポート状況を errfile に出力します。
  • **symbol:support/logresolve.c#usage**: logresolve コマンドの使用方法を errfile に出力し、exit(1) で終了します。
  • **symbol:support/rotatelogs.c#usage**: rotatelogs ユーティリティの使用方法を stderr に出力し、exit(1) で終了します。
  • **symbol:test/test_limits.c#usage**: test_limits ユーティリティの使用方法を stderr に出力し、exit(1) で終了します。

※ 以下のエンティティについては、ソース参照に含まれているものの、詳細なコンテキストが提供されていないため、具体的な振る舞いは未確定です。

エンドツーエンドの流れ

1. **起動と引数解析**: 各プログラムの main 関数が起動し、コマンドライン引数（argv）の解析を開始します。 2. **エラーまたはヘルプの検出**: 引数の解析中に不正なオプションが指定された場合、またはヘルプオプション（-h や --help など）が指定された場合に、各プログラム固有の usage 関数が呼び出されます。 3. **メッセージ出力**: usage 関数内部で、標準エラー出力（stderr）、APR のファイルハンドル（errfile）、または ap_log_error を介してヘルプメッセージが出力されます。 4. **プロセスの終了**: メッセージ出力後、exit や destroy_and_exit_process などの終了処理を呼び出し、呼び出し元に制御を戻さずにプロセスを終了します。

※ support/ab.c、support/htcacheclean.c、test/clients/h2ws.c における詳細な呼び出しフローやエンドツーエンドの挙動は、提供された情報からは特定されていません。

重要な不変条件

  • **プロセスの強制終了**: usage 関数が呼び出された場合、プログラムは必ず制御を呼び出し元に戻さず、プロセスを終了（exit またはそれに準ずる終了処理）させなければなりません。
  • **引数の有効性**: server/main.c の usage においては、引数として渡される process_rec *process が NULL でないことが前提となります（process->argv[0] を参照するため）。
  • **出力ストリームの初期化**: support 配下の多くのツール（htdigest, htpasswd, httxt2dbm, logresolve）では、出力先となる errfile（apr_file_t）が有効に初期化されている必要があります。

よくある落とし穴

  • **ヘルプメッセージの更新漏れ**: 新しいコマンドラインオプションを追加した際、対応する usage 関数の出力メッセージを手動で更新し忘れると、ユーザーに正しい使い方が伝わらなくなります。
  • **プラットフォーム固有の条件付きコンパイル**: server/main.c や support/rotatelogs.c などでは、WIN32 や NETWARE、あるいは APR_FILES_AS_SOCKETS などのマクロ定義によって表示されるオプションが切り替わります。プラットフォームごとの差異を考慮せずにメッセージを変更すると、特定の環境でビルドエラーや表示の不整合が発生する原因になります。
  • **NULL ポインタアクセス**: server/main.c の usage において、process 構造体が NULL の状態で呼び出されると、process->argv[0] へのアクセスによりセグメンテーション違反が発生する可能性があります。
