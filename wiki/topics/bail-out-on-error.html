bail_out_on_error

概念

bail_out_on_error は、Apache HTTP Server（httpd）のリクエスト処理やHTTPフィルタ処理において、プロトコル違反、リソース制限超過（リクエストボディサイズ制限超過など）、またはその他のエラーが発生した際に、安全に処理を中断してエラーレスポンスを生成・送信するための内部ヘルパー関数です。

なぜ重要か

エラー発生時に、既存のバケットブリゲードの内容をクリアし、適切なエラーバケット（ap_bucket_error_create）とストリーム終了（EOS）バケット（apr_bucket_eos_create）を挿入して下位フィルタへ渡すことで、クライアントに対して一貫したエラーレスポンスを安全に返却できます。また、HTTPフィルタ側では接続の Keep-Alive を無効化（AP_CONN_CLOSE）することで、不整合なデータが後続のリクエストに混入するのを防ぎ、プロトコルの整合性と安全性を維持します。

主要エンティティ

  • **symbol:modules/filters/mod_request.c#bail_out_on_error**: リクエストボディの保持フィルタ（keep_body_filter）などでサイズ制限超過などのエラーが発生した際に、バケットブリゲードを初期化してエラーバケットとEOSバケットを挿入し、クライアントへ送信します。
  • **symbol:modules/http/http_filters.c#bail_out_on_error**: HTTP入力フィルタ処理（ap_http_filter など）において、プロトコル違反や制限超過が発生した際に、エラーバケットの挿入、接続のクローズ（Keep-Alive無効化）、およびプロキシリクエスト時のエラーコード返却を行います。

エンドツーエンドの流れ

1. **エラーの検知**: keep_body_filter（mod_request.c）でのリクエストボディサイズ制限超過や、HTTPフィルタ（http_filters.c）でのプロトコル違反・制限超過が発生します。 2. **bail_out_on_error の呼び出し**: mod_request.c 内の関数、または http_filters.c 内の各所から bail_out_on_error が呼び出されます。 3. **ブリゲードのクリーンアップ**: 対象のバケットブリゲード（bb または ctx->bb）が apr_brigade_cleanup によって初期化され、既存のバケットが破棄されます。 4. **レスポンスの構築と送信**:

  • **通常リクエストの場合**: ap_bucket_error_create で指定された HTTP エラーコードを含むエラーバケットを作成し、EOSバケットとともにブリゲードの末尾に挿入します。その後、ap_pass_brigade を呼び出して下位フィルタチェーンへ送信します。
  • **プロキシリクエスト（PROXYREQ_RESPONSE）の場合**（http_filters.c のみ）: エラーコードに対応する apr_status_t（APR_ENOSPC、APR_INCOMPLETE など）を即座に返します。

    • 5. **接続状態の更新**: http_filters.c の実装では、ctx->eos_sent を 1 に設定し、接続の keepalive を AP_CONN_CLOSE に設定して接続を閉じる準備をします。

※なお、フィルタチェーン全体における詳細なエラー伝播や、後続の出力フィルタがエラーバケットを処理する具体的なエンドツーエンドの全容は、提供された情報からは完全には特定されていません。

重要な不変条件

  • **ブリゲードの初期化**: bail_out_on_error が呼び出されると、対象のバケットブリゲードは必ず apr_brigade_cleanup によって既存の内容が破棄されます。
  • **フィルタコンテキストの有効性**: 呼び出し時に f および f->r（リクエストオブジェクト）が有効なオブジェクトを指している必要があります。
  • **Keep-Alive の無効化（http_filters.c）**: 通常リクエストのエラー処理時には、後続リクエストへのデータ混入を防ぐため、接続の keepalive 属性が必ず AP_CONN_CLOSE に設定されます。

よくある落とし穴

  • **プロキシリクエストと通常リクエストの挙動の違い**: modules/http/http_filters.c の bail_out_on_error は、プロキシリクエスト（PROXYREQ_RESPONSE）の場合にエラーバケットを送信せず、即座に apr_status_t エラーコードを返却します。このため、呼び出し元はリクエストの種類に応じた戻り値のハンドリングに注意する必要があります。
  • **メモリ確保の失敗**: f->r->pool や f->c->bucket_alloc が無効な場合、バケット作成時のメモリ確保で異常終了する可能性があります。
  • **呼び出し後の処理中断**: 本関数はエラー発生時に接続を閉じる、またはエラーを伝播させるためのものであるため、呼び出し元は本関数を実行した後、速やかにフィルタチェーンの処理を終了（ベイルアウト）しなければなりません。
