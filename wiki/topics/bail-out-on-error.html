bail_out_on_error

概念

bail_out_on_error は、Apache HTTP Server（httpd）の mod_request.c および http_filters.c に定義されている内部関数です。リクエスト処理中やHTTPフィルタ処理中に、リクエストボディの制限超過やプロトコル違反などのエラーが発生した際、現在処理中のバケットブリゲードをクリーンアップし、エラーバケットとEOS（End of Stream）バケットを挿入して安全に処理を中断・エラーレスポンスを送信するためのエラーハンドリング機構を提供します。

なぜ重要か

エラー発生時に、既存のバケットブリゲードのデータを破棄し、適切なエラーバケットとEOSバケットを構築して下位フィルタへ渡すことで、クライアントへ適切なHTTPエラーレスポンス（例: HTTP_REQUEST_ENTITY_TOO_LARGE）を安全に返却できます。

特にHTTPフィルタ（http_filters.c）においては、チャンクエンコーディングや Content-Length の不整合が発生した際に接続の Keep-Alive を強制的に無効化（AP_CONN_CLOSE）することで、後続のリクエストへのデータ混入を防ぎ、プロトコルの整合性とセキュリティを維持する重要な役割を担っています。

関与するモジュール

主要エンティティ

  • **symbol:modules/filters/mod_request.c#bail_out_on_error**
  • **シグネチャ**: static apr_status_t bail_out_on_error(apr_bucket_brigade *bb, ap_filter_t *f, int http_error)
  • **役割**: keep_body_filter 内でのリクエストボディサイズ制限超過時などに呼び出され、指定されたブリゲード bb をクリーンアップしてエラーバケットとEOSバケットを挿入し、ap_pass_brigade で送信します。
  • **symbol:modules/http/http_filters.c#bail_out_on_error**
  • **シグネチャ**: static apr_status_t bail_out_on_error(http_ctx_t *ctx, ap_filter_t *f, int http_error)
  • **役割**: HTTP入力フィルタ処理（ap_http_filter 等）でのプロトコル違反や制限超過時に呼び出されます。プロキシリクエストと通常リクエストで処理を分岐し、通常リクエストの場合はエラーバケットとEOSバケットを挿入して Keep-Alive を無効化し、ap_pass_brigade で送信します。

エンドツーエンドの流れ

1. **エラーの検知**:

  • mod_request.c の keep_body_filter や、http_filters.c の ap_http_filter などの処理中に、リクエストボディサイズ制限超過やプロトコル違反などのエラーが発生します。

    • 2. **bail_out_on_error の呼び出し**:

  • 発生した HTTP エラーコード（例: HTTP_REQUEST_ENTITY_TOO_LARGE）を指定して bail_out_on_error が呼び出されます。

    • 3. **ブリゲードのクリーンアップ**:

  • 対象のバケットブリゲード（bb または ctx->bb）に対して apr_brigade_cleanup を実行し、既存のバケットをすべて破棄します。

    • 4. **プロキシリクエストの分岐（http_filters.c のみ）**:

  • プロキシリクエスト（PROXYREQ_RESPONSE）の場合、エラーコードに応じて APR_ENOSPC、APR_INCOMPLETE、APR_ENOTIMPL、または APR_EGENERAL を即座に呼び出し元に返します。

    • 5. **エラーバケットとEOSバケットの構築（通常リクエスト）**:

  • ap_bucket_error_create を使用して指定された HTTP エラーを含むエラーバケットを作成し、ブリゲードの末尾に挿入します。
  • apr_bucket_eos_create を使用して EOS バケットを作成し、ブリゲードの末尾に挿入します。

    • 6. **接続状態の更新（http_filters.c のみ）**:

  • ctx->eos_sent を 1 に設定します。
  • 接続の keepalive を AP_CONN_CLOSE に設定し、Keep-Alive を無効化します。

    • 7. **下位フィルタへの送信**:

  • ap_pass_brigade を呼び出し、構築したブリゲードを出力フィルタチェーンへ渡します。

※なお、これら2つの bail_out_on_error 関数がシステム全体でどのように相互作用するか、またフィルタチェーン全体における完全なエンドツーエンドの伝播フローの詳細は、提供されたコンテキストからは完全には明らかになっていません。

重要な不変条件

  • **有効なコンテキストの存在**:
  • フィルタコンテキスト f およびリクエストオブジェクト f->r が有効でなければなりません。
  • http_filters.c の場合、ctx->bb が有効なバケットブリゲードとして初期化されている必要があります。
  • **ブリゲードの初期化**:
  • bail_out_on_error 処理の開始時に、対象のバケットブリゲードは必ず apr_brigade_cleanup によって既存の内容が完全に破棄されます。
  • **接続の切断（通常リクエスト時）**:
  • http_filters.c の通常リクエスト処理においては、エラー発生時に Keep-Alive が強制的に AP_CONN_CLOSE に設定され、後続リクエストへのデータ混入を防ぐ不変条件が維持されます。

よくある落とし穴

  • **プロキシリクエストと通常リクエストの挙動の違い**:
  • http_filters.c の bail_out_on_error は、プロキシリクエストの場合にエラーバケットを送信せず、即座に APR_ENOSPC などのエラーコードを返します。この挙動の違いを意識しないと、プロキシ時のエラーハンドリングで意図しない動作を招く可能性があります。
  • **メモリ確保の失敗**:
  • f->r->pool や f->c->bucket_alloc が無効な場合、バケット作成時のメモリ確保で異常終了する可能性があります。
  • **呼び出し後の処理継続**:
  • 本関数は接続を閉じるか、エラーを伝播させて処理を中断することを目的としているため、呼び出し元は速やかにフィルタチェーンの処理を終了させる必要があります。
