bail_out_on_error

概念

bail_out_on_error は、Apache HTTP Server (httpd) のリクエスト処理やHTTPフィルタ処理において、プロトコル違反、リソース制限超過（例: リクエストボディサイズ制限超過）、その他のエラーが発生した際に、安全に処理を中断してエラーレスポンスの送信手続きを行うための静的（static）なヘルパー関数です。

なぜ重要か

エラー発生時に、バケットブリゲード（データの送受信単位）に残っている未処理のデータをクリアし、適切なエラーバケットとストリーム終了（EOS）バケットを挿入して下位フィルタに渡すことで、クライアントに対して一貫したエラーレスポンスを返却できます。また、HTTPフィルタ処理においては、接続の Keep-Alive を無効化（AP_CONN_CLOSE）することで、不整合なデータが後続のリクエストに混入するのを防ぎ、プロトコルの整合性を維持する役割を果たします。

主要エンティティ

  • **symbol:modules/filters/mod_request.c#bail_out_on_error**: リクエストフィルタ（keep_body_filter など）の処理中にエラーが発生した際、バケットブリゲードをクリーンアップし、エラーバケットとEOSバケットを挿入して出力フィルタチェーンへ渡します。
  • **symbol:modules/http/http_filters.c#bail_out_on_error**: HTTP入力フィルタ（ap_http_filter など）の処理中にプロトコル違反や制限超過が発生した際、プロキシリクエストと通常リクエストを判別し、適切にエラーバケットの送信や接続のクローズ（Keep-Aliveの無効化）を行います。

エンドツーエンドの流れ

エラー発生からレスポンス送信までの完全なエンドツーエンドの制御フロー（特に、下位フィルタチェーンに渡されたエラーバケットが最終的にどのようにクライアントへ送信されるか、およびエラーハンドリング全体のライフサイクル）の詳細は、提供されたモジュール情報だけでは完全には明らかになっていません。

現時点で判明している局所的な流れは以下の通りです： 1. **エラー検知**: keep_body_filter や ap_http_filter 内で、リクエストボディのサイズ制限超過やプロトコル違反などのエラーを検知します。 2. **関数の呼び出し**: 各モジュール内の bail_out_on_error が呼び出されます。 3. **ブリゲードの初期化**: 対象となるバケットブリゲード（bb または ctx->bb）を apr_brigade_cleanup でクリアします。 4. **プロキシの個別処理（http_filters.c のみ）**: プロキシリクエスト（PROXYREQ_RESPONSE）の場合は、HTTPエラーコードに対応する apr_status_t（APR_ENOSPC、APR_INCOMPLETE など）を即座に返します。 5. **バケットの構築と送信（通常リクエスト）**:

  • ap_bucket_error_create を用いて、指定された http_error コードを含むエラーバケットを作成し、ブリゲードの末尾に挿入します。
  • apr_bucket_eos_create を用いて EOS バケットを作成し、ブリゲードの末尾に挿入します。
  • http_filters.c では、ctx->eos_sent を 1 に設定し、接続の keepalive 状態を AP_CONN_CLOSE に設定します。
  • ap_pass_brigade を呼び出して、構築したブリゲードを下位フィルタチェーンへ渡します。

重要な不変条件

  • **ブリゲードのクリーンアップ**: bail_out_on_error が呼び出されると、対象のバケットブリゲードにそれまで格納されていたデータは apr_brigade_cleanup によってすべて破棄されます。
  • **フィルタコンテキストの有効性**: 呼び出し時に、フィルタコンテキスト f およびリクエストオブジェクト f->r が有効なポインタである必要があります。
  • **接続の切断（HTTPフィルタ）**: http_filters.c の実装では、通常リクエストのエラー時に Keep-Alive が強制的に無効化（AP_CONN_CLOSE）され、接続が閉じられることが保証されます。

よくある落とし穴

  • **プロキシリクエストと通常リクエストの挙動の違い**: modules/http/http_filters.c の bail_out_on_error は、プロキシリクエストの場合にエラーバケットを送信せず、即座に apr_status_t のエラーコードを返却します。このため、呼び出し元はコンテキストに応じた戻り値のハンドリングを行う必要があります。
  • **メモリ確保の失敗**: f->r->pool や f->c->bucket_alloc が無効な状態で呼び出された場合、バケット作成時のメモリ確保において異常終了（セグメンテーションフォールトなど）を引き起こす可能性があります。
  • **呼び出し後の処理継続**: 本関数はエラー発生時に接続を閉じる、または処理を中断するための定型処理を行うため、呼び出し元は速やかにフィルタチェーンの処理を終了させる必要があります。
