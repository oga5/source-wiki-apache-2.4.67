bail_out_on_error

概念

bail_out_on_error は、Apache HTTP Server（httpd-2.4.67）のリクエスト処理やHTTPフィルタ処理中にエラーが発生した際、安全に処理を中断（ベイルアウト）し、エラーレスポンスを生成・送信するための一連の内部エラーハンドリング処理です。

なぜ重要か

リクエストボディの制限超過やプロトコル違反などのエラーが発生した際、不整合な状態のまま処理を継続すると、データの混入や予期しない動作を引き起こす可能性があります。bail_out_on_error は、既存のバケットブリゲードを安全にクリーンアップし、エラーバケットとストリーム終了（EOS）バケットを挿入して下位フィルタへ渡すことで、クライアントに対して一貫したエラーレスポンスを返し、必要に応じて接続をクローズする役割を担います。

主要エンティティ

  • **symbol:modules/filters/mod_request.c#bail_out_on_error**

    • リクエストボディの保持（keep_body_filter）などの処理中に、サイズ制限超過などのエラーが発生した際に呼び出されます。バケットブリゲードを初期化し、エラーバケットとEOSバケットを挿入して出力フィルタチェーンへ渡します。

  • **symbol:modules/http/http_filters.c#bail_out_on_error**

    • HTTP入力フィルタ（ap_http_filter など）において、プロトコル違反やリソース制限超過が発生した際に呼び出されます。エラーバケットの挿入とEOSの送信に加え、接続のクローズ（Keep-Aliveの無効化）を行います。

エンドツーエンドの流れ

1. **エラーの検知**: keep_body_filter（mod_request.c）や ap_http_filter（http_filters.c）の処理中に、リクエストサイズ超過やプロトコル違反などのエラーが発生します。 2. **関数の呼び出し**: 発生したHTTPエラーコード（例: HTTP_REQUEST_ENTITY_TOO_LARGE）を引数として bail_out_on_error が呼び出されます。 3. **ブリゲードのクリーンアップ**: 対象となるバケットブリゲード（mod_request.c では引数の bb、http_filters.c では ctx->bb）に対して apr_brigade_cleanup を実行し、既存のバケットをすべて破棄します。 4. **エラー/EOSバケットの構築と送信**:

  • **通常リクエストの場合**: ap_bucket_error_create でエラーバケットを作成し、apr_bucket_eos_create でEOSバケットを作成してブリゲードの末尾に挿入します。その後、ap_pass_brigade を呼び出して下位フィルタへ送信します。http_filters.c では、接続の keepalive を AP_CONN_CLOSE に設定してKeep-Aliveを無効化します。
  • **プロキシリクエスト（PROXYREQ_RESPONSE）の場合（http_filters.c のみ）**: エラーバケットは作成せず、HTTPエラーコードに応じた apr_status_t（APR_ENOSPC、APR_INCOMPLETE、APR_ENOTIMPL、または APR_EGENERAL）を即座に返します。

    • 5. **結果の伝播**: ap_pass_brigade の実行結果（またはプロキシ用のエラーコード）が呼び出し元に返されます。

*注意: 送信されたエラーバケットが後続の出力フィルタチェーンでどのように処理され、最終的にクライアントへ届くかという詳細なエンドツーエンドの全体フローは、提供されたコンテキストからは完全には明らかになっていません。*

重要な不変条件

  • **ブリゲードの初期化**: 本関数が呼び出されると、対象のバケットブリゲード内の既存コンテンツは必ず apr_brigade_cleanup によって破棄されます。
  • **有効なコンテキスト**: フィルタコンテキスト f およびリクエストオブジェクト f->r が有効なオブジェクトを指している必要があります。
  • **Keep-Aliveの無効化（http_filters.c）**: 通常リクエストのエラー処理時、チャンクエンコーディングや Content-Length の不整合による後続リクエストへのデータ混入を防ぐため、接続の keepalive 状態は必ず AP_CONN_CLOSE に設定されます。

よくある落とし穴

  • **プロキシリクエスト時の挙動の違い**: http_filters.c の bail_out_on_error は、プロキシリクエストの場合にエラーバケットを送信せず、特定の apr_status_t エラーコードを即座に返します。このパスの違いを意識しないと、プロキシ環境下でのエラーハンドリングで意図しない動作を引き起こす可能性があります。
  • **メモリ確保の失敗**: f->r->pool や f->c->bucket_alloc が無効な場合、バケット作成時のメモリ確保で異常終了するリスクがあります。
  • **呼び出し後の処理継続**: 本関数はエラー発生時に接続を閉じる、またはエラーを伝播させるためのものであるため、呼び出し元は本関数の実行後、速やかにフィルタチェーンの処理を終了させる必要があります。
