storage

概念

storage は、Apache HTTP Server（httpd）の複数のモジュールにおいて、動的なデータ（ハートビート情報、バランサーの設定や状態、スロットメモリなど）を保持・管理するためのストレージ機構またはプロバイダを指します。具体的には、共有メモリ（SHM）やプレーンメモリなどを介してデータを永続化、またはプロセス間で共有するために利用されます。

なぜ重要か

マルチプロセス・マルチスレッドで動作するApacheにおいて、ロードバランサの状態（mod_proxy_balancer）やクラスタの生存監視情報（mod_heartmonitor、mod_lbmethod_heartbeat）を正確に共有・同期することは、高可用性と適切な負荷分散を実現するために不可欠です。また、mod_slotmem_shm や mod_slotmem_plain などのスロットメモリ機構は、これらの一貫したデータ格納領域（storage）を抽象化されたAPIとして提供する役割を担っています。

主要エンティティ

  • **modules/cluster/mod_heartmonitor.c の storage**: ハートビート監視データを保存するためのストレージプロバイダ。
  • **modules/proxy/balancers/mod_lbmethod_heartbeat.c の storage**: ハートビートベースの負荷分散アルゴリズムが参照するストレージ。
  • **modules/proxy/mod_proxy_balancer.c の storage**: バランサーのメンバーや状態を管理するためのストレージ。
  • **modules/slotmem/mod_slotmem_plain.c の storage**: 通常のプレーンメモリを使用したスロットメモリの実装。
  • **modules/slotmem/mod_slotmem_shm.c の storage**: 共有メモリ（Shared Memory）を使用したスロットメモリの実装。

エンドツーエンドの流れ

現時点では、これらの storage シンボル間における具体的な呼び出しシーケンスや、初期化からデータ更新・参照に至る詳細なエンドツーエンドの制御フローは完全には判明していません。一般的には、mod_slotmem などのプロバイダが提供する storage 領域に対して、mod_proxy_balancer や mod_heartmonitor がAPIを介してデータの読み書きを行う流れが想定されます。

重要な不変条件

現時点では、各モジュールの storage 実装間、あるいはメモリ確保・排他制御において維持されるべき具体的な不変条件（インバリアント）は特定されていません。

よくある落とし穴

共有メモリ（mod_slotmem_shm）を使用する場合、プロセス間での排他制御（ロック）が適切に行われないとデータの競合や破損が発生する可能性があります。また、プレーンメモリ（mod_slotmem_plain）はプロセス間での共有ができないため、マルチプロセスモデル（Prefork MPMなど）での利用時には意図したデータ共有が行われないといった制約に注意する必要があります。ただし、具体的な不具合や落とし穴の詳細は現時点では特定されていません。
