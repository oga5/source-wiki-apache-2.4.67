ssl_proxy_enable

概念

ssl_proxy_enable は、Apache HTTP Server においてプロキシ接続などのアウトバウンド（送信）接続に対して SSL/TLS 暗号化機能を有効化するためのインターフェースです。主にオプション関数（Optional Function）として登録され、外部モジュールから動的に呼び出せる仕組みを提供します。

なぜ重要か

プロキシ経由のバックエンド通信など、アウトバウンド接続のセキュリティを確保するために不可欠です。プラットフォーム固有のモジュール（NetWare など）やコアサーバーの SSL 実装が、共通のインターフェースを介して接続（conn_rec）に SSL/TLS をバインドできるようにします。

関与するモジュール

  • module:modules/arch/netware：NetWare プラットフォーム向けの実装（mod_nw_ssl.c）を提供します。
  • module:server：コアサーバーにおける SSL 制御（ssl.c）を提供します。

主要エンティティ

  • symbol:server/ssl.c#ssl_proxy_enable：接続に対して ap_ssl_bind_outgoing を呼び出し、SSL を強制するコア側の関数。
  • symbol:modules/arch/netware/mod_nw_ssl.c#ssl_proxy_enable：NetWare 環境における同等の機能を提供するエンティティ。

エンドツーエンドの流れ

1. **オプション関数の登録**: server/ssl.c などのモジュールが起動時に ssl_proxy_enable をオプション関数として登録します。 2. **呼び出し**: mod_proxy などのプロキシモジュールが、バックエンドへの接続を確立する際にこのオプション関数を取得して呼び出します。 3. **SSL のバインド**: server/ssl.c の実装では、渡された conn_rec *c に対して ap_ssl_bind_outgoing(c, NULL, 1) を実行し、SSL 必須フラグ（1）を指定してアウトバウンド接続に SSL をバインドします。

※ NetWare 環境（mod_nw_ssl.c）における具体的な内部処理や、コア側と NetWare 側がどのように協調・排他制御されるかといった詳細なエンドツーエンドのフローは、提供されたコンテキストからは不明です。

重要な不変条件

  • **有効な接続オブジェクト**: 引数として渡される conn_rec *c は、有効な接続構造体でなければなりません。
  • **SSL の強制**: server/ssl.c の実装においては、ap_ssl_bind_outgoing の第3引数（SSL 必須フラグ）に常に 1 が指定され、SSL 接続が強制されます。

よくある落とし穴

  • **未登録時のエラー**: SSL モジュールがロードされていない、または初期化されていない場合、オプション関数が未登録となり、呼び出し側で適切にハンドリングしないと実行時エラー（ヌルポインタ参照など）を引き起こす可能性があります。
  • **シグネチャ変更の影響**: 本関数は APR_DECLARE_OPTIONAL_FN を介して公開されているため、シグネチャを変更すると mod_proxy などの依存モジュールとの互換性が失われます。

次に読む場所

  • server/ssl.c：コアにおける ssl_proxy_enable の実装と ap_ssl_bind_outgoing の呼び出し。
  • modules/arch/netware/mod_nw_ssl.c：NetWare 環境におけるプラットフォーム固有の実装。
