send_begin_request

概念

send_begin_request は、Apache HTTP Server (httpd-2.4.67) において、FastCGI プロトコルに基づきバックエンドサーバーへ BEGIN_REQUEST（または FCGI_BEGIN_REQUEST）レコードを構築・送信し、新しい FastCGI リクエストセッションの開始を通知するための関数です。この処理は、認証・認可を行うモジュール（mod_authnz_fcgi）と、プロキシ転送を行うモジュール（mod_proxy_fcgi）のそれぞれで個別に実装されています。

なぜ重要か

FastCGI プロトコルを用いた通信を開始する際の最初のステップであり、バックエンドに対してリクエストの役割（ロール）や接続の維持（Keep-Alive）に関するポリシーを伝達するために不可欠です。認証（AAA）処理やプロキシ処理を FastCGI バックエンドに委譲する基盤となっています。

関与するモジュール

  • module:modules/aaa
  • module:modules/proxy

主要エンティティ

  • symbol:modules/aaa/mod_authnz_fcgi.c#send_begin_request: 認証・認可プロバイダー用の FastCGI 開始リクエスト送信関数。
  • symbol:modules/proxy/mod_proxy_fcgi.c#send_begin_request: プロキシ接続用の FastCGI 開始リクエスト送信関数。

エンドツーエンドの流れ

各モジュールにおけるリクエスト送信の具体的な流れは以下の通りですが、システム全体における完全なエンドツーエンドのライフサイクルや、呼び出し元から呼び出し先への全遷移の詳細は完全には明らかになっていません。

1. **ヘッダーの構築**: ap_fcgi_fill_in_header を使用して AP_FCGI_BEGIN_REQUEST 型の FastCGI ヘッダーを生成します。 2. **ボディの構築**:

  • mod_authnz_fcgi では、AP_FCGI_KEEP_CONN フラグを設定せずに（接続維持を行わない前提で）ap_fcgi_fill_in_request_body を呼び出します。
  • mod_proxy_fcgi では、ap_proxy_connection_reusable を確認し、接続が再利用可能であれば AP_FCGI_KEEP_CONN フラグを付与して ap_fcgi_fill_in_request_body を呼び出します。

    • 3. **配列変換と結合**: ヘッダーとボディをそれぞれバイト配列（ap_fcgi_header_to_array、ap_fcgi_begin_request_body_to_array）に変換し、iovec 構造体を用いて結合します。 4. **送信**:

  • mod_authnz_fcgi では、sendv_data を介してソケット経由で送信します。
  • mod_proxy_fcgi では、send_data を介してバックエンドへ送信します。

重要な不変条件

  • **接続の有効性**: 送信対象となるソケット（mod_authnz_fcgi の apr_socket_t）または接続（mod_proxy_fcgi の proxy_conn_rec）は、呼び出し時点で有効かつ接続済みである必要があります。
  • **接続維持フラグの差異**:
  • mod_authnz_fcgi の実装では、AP_FCGI_KEEP_CONN は常に設定されず、接続の維持は行われません。
  • mod_proxy_fcgi の実装では、接続が再利用可能である場合にのみ AP_FCGI_KEEP_CONN が設定されます。

よくある落とし穴

  • **多重化の非サポート**: mod_proxy_fcgi の send_begin_request は FastCGI の多重化（Multiplexing）をサポートしておらず、単一の接続に対してリクエストを開始する設計となっています。
  • **ブロッキング I/O**: mod_proxy_fcgi で使用される send_data はブロッキング I/O を含む可能性があるため、呼び出し時のコンテキストに注意が必要です。
  • **エラーハンドリング**: 送信処理（sendv_data または send_data）が失敗した場合、そのエラーコード（apr_status_t）がそのまま返されるため、呼び出し元で適切にハンドリングする必要があります。
