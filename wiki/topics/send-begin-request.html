send_begin_request

概念

send_begin_request は、Apache HTTP Server（httpd）において FastCGI バックエンドサーバーとの通信を開始する際に、FastCGI プロトコルの BEGIN_REQUEST（または FCGI_BEGIN_REQUEST）レコードを構築して送信するための関数です。この処理により、バックエンドに対して新しいリクエストセッションの開始が通知されます。

なぜ重要か

FastCGI プロトコルに準拠した通信を行うためには、リクエストの最初に必ず BEGIN_REQUEST レコードを送信する必要があります。httpd では、認証・認可を行うモジュール（mod_authnz_fcgi）と、リクエストをプロキシするモジュール（mod_proxy_fcgi）のそれぞれにおいて、バックエンドとのセッションを正しく開始するためにこの処理が不可欠です。

関与するモジュール

  • module:modules/aaa (mod_authnz_fcgi.c)
  • module:modules/proxy (mod_proxy_fcgi.c)

主要エンティティ

  • symbol:modules/aaa/mod_authnz_fcgi.c#send_begin_request: 認証・認可用の FastCGI プロバイダーにおいて、指定されたロール（例: AP_FCGI_AUTHORIZER）で BEGIN_REQUEST を送信します。
  • symbol:modules/proxy/mod_proxy_fcgi.c#send_begin_request: プロキシ接続において、新しいリクエストセッションを開始するために BEGIN_REQUEST を送信します。

エンドツーエンドの流れ

各モジュールにおける個別の送信フローは以下の通りですが、システム全体としての完全なエンドツーエンドのライフサイクルや、これら2つのモジュール間の直接的な連携フロー（例えば、認証後にプロキシへ処理が遷移する際の一連のシグナル伝達など）の詳細は、提供されたコンテキストからは未確定です。

個別モジュール内での送信フロー

1. **ヘッダーの構築**: ap_fcgi_fill_in_header を使用して AP_FCGI_BEGIN_REQUEST 型のヘッダーを生成します。 2. **ボディの構築**: ap_fcgi_fill_in_request_body を使用してボディを生成します。

  • mod_authnz_fcgi では、接続維持を行わない前提（AP_FCGI_KEEP_CONN は設定しない）で構築されます。
  • mod_proxy_fcgi では、ap_proxy_connection_reusable により接続が再利用可能と判断された場合に AP_FCGI_KEEP_CONN フラグが付与されます。

    • 3. **バイト配列への変換**: ap_fcgi_header_to_array および ap_fcgi_begin_request_body_to_array を用いて、ヘッダーとボディをバイト配列に変換します。 4. **送信**: iovec 構造体を用いてヘッダーとボディを結合し、ソケット経由で送信します。

  • mod_authnz_fcgi では sendv_data を使用して送信します。
  • mod_proxy_fcgi では send_data を使用して送信します。

重要な不変条件

  • **接続の有効性**: 送信対象となるソケット（mod_authnz_fcgi の apr_socket_t *s）または接続（mod_proxy_fcgi の proxy_conn_rec *conn）は、呼び出し時点で有効かつ接続済みでなければなりません。
  • **接続維持フラグの制御**:
  • mod_authnz_fcgi の実装では、AP_FCGI_KEEP_CONN フラグは常に設定されず、接続の維持は行われません。
  • mod_proxy_fcgi の実装では、接続の再利用可能性（ap_proxy_connection_reusable）に依存して AP_FCGI_KEEP_CONN の有無が決定されます。

よくある落とし穴

  • **多重化の非サポート**: mod_proxy_fcgi の send_begin_request は FastCGI の多重化（Multiplexing）をサポートしておらず、単一の接続に対してリクエストを開始する設計となっています。
  • **ブロッキング I/O**: mod_proxy_fcgi で使用される send_data はブロッキング I/O を含む可能性があるため、呼び出し時のコンテキストやスレッドのブロックに注意が必要です。
  • **エラーの伝播**: sendv_data や send_data による送信処理が失敗した場合、そのエラー（apr_status_t）がそのまま返されるため、呼び出し元で適切にハンドリングする必要があります。
