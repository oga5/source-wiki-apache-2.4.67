send_begin_request

概念

send_begin_request は、Apache HTTP Server（httpd）において FastCGI プロトコルを用いたバックエンド通信を開始する際に、FCGI_BEGIN_REQUEST（または AP_FCGI_BEGIN_REQUEST）レコードを構築して送信するための関数です。

この処理は、認証・認可を行うモジュール（mod_authnz_fcgi）と、プロキシ転送を行うモジュール（mod_proxy_fcgi）のそれぞれに同名の関数として定義されており、それぞれの役割に応じた FastCGI セッションの開始処理を担っています。

なぜ重要か

FastCGI プロトコルでは、バックエンドとの通信を開始するにあたり、最初に BEGIN_REQUEST レコードを送信してリクエストの役割（Role）や接続維持（Keep-Alive）の意図を伝える必要があります。

send_begin_request は、このプロトコルハンドシェイクの最初のステップをカプセル化しており、認証（Authorizer）やプロキシ（Responder等）といった異なるコンテキストにおいて、正しくバックエンドとのセッションを確立するために不可欠な役割を果たします。

関与するモジュール

  • module:modules/aaa (mod_authnz_fcgi.c)
  • module:modules/proxy (mod_proxy_fcgi.c)

主要エンティティ

  • symbol:modules/aaa/mod_authnz_fcgi.c#send_begin_request: 認証・認可プロバイダー用の FastCGI 接続開始関数。
  • symbol:modules/proxy/mod_proxy_fcgi.c#send_begin_request: プロキシ経由の FastCGI 接続開始関数。

エンドツーエンドの流れ

各モジュールにおける具体的なエンドツーエンドの呼び出しフローの全貌（例えば、リクエストを受信してからこれらの関数が呼び出され、レスポンスを返却するまでの完全なライフサイクルや、上位のハンドラとの詳細な相互作用）は、提供されたコンテキストからは完全には明らかになっていません。

現時点で判明している部分的な流れは以下の通りです。

1. 認証・認可時 (mod_authnz_fcgi.c)

1. modules/aaa/mod_authnz_fcgi.c:734 付近から send_begin_request が呼び出されます。 2. ap_fcgi_fill_in_header を用いて AP_FCGI_BEGIN_REQUEST ヘッダーを生成します。 3. ap_fcgi_fill_in_request_body を用いて、指定されたロール（例: AP_FCGI_AUTHORIZER）を含むボディを生成します。この際、AP_FCGI_KEEP_CONN フラグは設定されません。 4. ヘッダーとボディをバイト配列に変換し、iovec を用いて結合した上で、sendv_data を介してソケット s からバックエンドへ送信します。

2. プロキシ転送時 (mod_proxy_fcgi.c)

1. modules/proxy/mod_proxy_fcgi.c:1044 付近から send_begin_request が呼び出されます。 2. ap_fcgi_fill_in_header を用いて AP_FCGI_BEGIN_REQUEST ヘッダーを生成します。 3. ap_proxy_connection_reusable を確認し、接続が再利用可能であれば AP_FCGI_KEEP_CONN フラグを付与して ap_fcgi_fill_in_request_body を呼び出し、ボディを生成します。 4. ヘッダーとボディをバイト配列に変換し、iovec を用いて結合した上で、send_data を介して proxy_conn_rec の接続からバックエンドへ送信します。

重要な不変条件

  • **接続の有効性**: 呼び出し時に、対象となるソケット（mod_authnz_fcgi では apr_socket_t *s、mod_proxy_fcgi では proxy_conn_rec *conn）が有効な接続状態である必要があります。
  • **接続維持フラグの制御**:
  • mod_authnz_fcgi の実装では、AP_FCGI_KEEP_CONN フラグは設定されず、接続の維持を行わない前提でリクエストボディが構築されます。
  • mod_proxy_fcgi の実装では、ap_proxy_connection_reusable の判定結果に基づいて AP_FCGI_KEEP_CONN フラグの有無が決定されます。

よくある落とし穴

  • **多重化の非サポート**: mod_proxy_fcgi 側の実装は FastCGI の多重化（Multiplexing）をサポートしておらず、単一の接続に対してリクエストを開始する設計となっています。
  • **ブロッキング I/O**: send_data や sendv_data はブロッキング I/O を含む可能性があるため、呼び出し時のコンテキストやスレッドのブロックに注意する必要があります。
  • **エラーハンドリング**: 送信処理（send_data / sendv_data）が失敗した場合、そのエラーコード（apr_status_t）がそのまま返されるため、呼び出し元で適切にハンドリングされないと、不完全な接続状態のまま処理が継続する恐れがあります。
