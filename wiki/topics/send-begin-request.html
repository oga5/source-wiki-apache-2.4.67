send_begin_request

概念

send_begin_request は、Apache HTTP Server（httpd）において FastCGI プロトコルを用いたバックエンドサーバーとの通信を開始する際に、FCGI_BEGIN_REQUEST レコードを構築して送信するための関数です。この処理により、バックエンドに対して新しいリクエストセッションの開始が通知されます。

なぜ重要か

FastCGI プロトコルに準拠した通信を行うための最初の手続きであり、認証・認可（AAA）モジュールやプロキシモジュールがバックエンドの FastCGI アプリケーションと正しくセッションを確立するために不可欠な役割を果たします。接続の再利用（Keep-Alive）の制御や、適切な役割（ロール）の指定もこの段階で行われます。

関与するモジュール

  • module:modules/aaa
  • module:modules/proxy

主要エンティティ

  • symbol:modules/aaa/mod_authnz_fcgi.c#send_begin_request: 認証・認可用の FastCGI プロバイダーにおいて、AP_FCGI_AUTHORIZER などのロールを指定してリクエストを開始します。
  • symbol:modules/proxy/mod_proxy_fcgi.c#send_begin_request: プロキシ経由の FastCGI 通信において、接続の再利用可能性に応じて AP_FCGI_KEEP_CONN フラグを制御しながらリクエストを開始します。

エンドツーエンドの流れ

モジュールごとに個別のアプローチが取られますが、共通して以下のステップを踏みます。

1. **ヘッダーの構築**: ap_fcgi_fill_in_header を使用して AP_FCGI_BEGIN_REQUEST 型の FastCGI ヘッダーを生成します。 2. **ボディの構築**: ap_fcgi_fill_in_request_body を使用して、指定されたロールや接続維持フラグを含むリクエストボディを生成します。

  • mod_authnz_fcgi.c では、接続維持を行わない前提（AP_FCGI_KEEP_CONN を設定しない）で構築されます。
  • mod_proxy_fcgi.c では、ap_proxy_connection_reusable の結果に基づき、再利用可能であれば AP_FCGI_KEEP_CONN フラグを付与します。

    • 3. **配列変換と結合**: ヘッダーとボディをそれぞれバイト配列（ap_fcgi_header_to_array、ap_fcgi_begin_request_body_to_array）に変換し、iovec 構造体を用いて結合します。 4. **送信**: ソケットを介してデータを送信します（mod_authnz_fcgi.c では sendv_data、mod_proxy_fcgi.c では send_data を使用）。

なお、呼び出し元からバックエンドの応答処理に至る完全なエンドツーエンドの全体フローは、提供された情報からは完全には明らかになっていません。

重要な不変条件

  • **有効な接続の存在**: 送信対象となるソケット（mod_authnz_fcgi.c の apr_socket_t *s）または接続レコード（mod_proxy_fcgi.c の proxy_conn_rec *conn）が、有効かつ接続済みの状態である必要があります。
  • **接続維持フラグの不変性（mod_authnz_fcgi）**: mod_authnz_fcgi.c の実装においては、AP_FCGI_KEEP_CONN フラグは明示的に設定されず、接続の維持を行わない前提でリクエストボディが構築されます。

よくある落とし穴

  • **多重化の非サポート**: mod_proxy_fcgi.c の実装は FastCGI の多重化（Multiplexing）をサポートしておらず、単一の接続に対してリクエストを開始する設計となっています。
  • **ブロッキング I/O**: mod_proxy_fcgi.c で使用される send_data はブロッキング I/O を含む可能性があるため、呼び出し時のコンテキストに注意が必要です。
  • **エラーの伝播**: 送信処理（sendv_data または send_data）が失敗した場合、そのエラーコードがそのまま返されるため、呼び出し元で適切にハンドリングする必要があります。
