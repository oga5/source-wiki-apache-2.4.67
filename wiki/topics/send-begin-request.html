send_begin_request

概念

send_begin_request は、Apache HTTP Server（httpd）において FastCGI プロトコルを用いたバックエンドサーバーとの通信を開始する際に、FCGI_BEGIN_REQUEST（または BEGIN_REQUEST）レコードを構築・送信するための関数です。この処理は、認証・認可を行うモジュール（mod_authnz_fcgi）と、プロキシ転送を行うモジュール（mod_proxy_fcgi）のそれぞれに同名の関数として実装されています。

なぜ重要か

FastCGI プロトコルでは、バックエンドとのセッションを開始するために最初に BEGIN_REQUEST レコードを送信する必要があります。この関数は、リクエストの役割（ロール）や接続の再利用性（Keep-Alive）などのメタデータを適切にカプセル化し、バックエンドとの通信の起点となる重要な役割を担っています。

関与するモジュール

  • module:modules/aaa (mod_authnz_fcgi.c)
  • module:modules/proxy (mod_proxy_fcgi.c)

主要エンティティ

  • **symbol:modules/aaa/mod_authnz_fcgi.c#send_begin_request**

    • 認証・認可用。接続の維持（AP_FCGI_KEEP_CONN）を行わない前提でリクエストを構築し、sendv_data を用いて送信します。

  • **symbol:modules/proxy/mod_proxy_fcgi.c#send_begin_request**

    • プロキシ用。接続が再利用可能か（ap_proxy_connection_reusable）を判定し、可能であれば AP_FCGI_KEEP_CONN フラグを付与して send_data を用いて送信します。

エンドツーエンドの流れ

現時点では、HTTP リクエストの受信からバックエンドへの送信、およびレスポンス処理に至る完全なエンドツーエンドの制御フローは提供された情報からは網羅されていません。

ただし、各関数内における送信処理の流れは以下の通りです： 1. **ヘッダーの生成**: ap_fcgi_fill_in_header を使用して AP_FCGI_BEGIN_REQUEST 型の FastCGI ヘッダーを構築します。 2. **ボディの生成**: ap_fcgi_fill_in_request_body を使用して、指定されたロールや接続維持フラグ（AP_FCGI_KEEP_CONN）を含むボディを構築します。 3. **配列への変換**: ap_fcgi_header_to_array および ap_fcgi_begin_request_body_to_array を用いて、ヘッダーとボディをバイト配列に変換します。 4. **送信**: iovec 構造体を用いてヘッダーとボディを結合し、ソケット経由（sendv_data または send_data）でバックエンドへ送信します。

重要な不変条件

  • **接続の有効性**: 送信先となるソケット（mod_authnz_fcgi では apr_socket_t *s、mod_proxy_fcgi では proxy_conn_rec *conn）は、呼び出し時点で有効かつ接続済みでなければなりません。
  • **接続維持フラグの制御**:
  • mod_authnz_fcgi 版では、AP_FCGI_KEEP_CONN フラグは常に設定されず、接続維持を行わない不変条件があります。
  • mod_proxy_fcgi 版では、ap_proxy_connection_reusable の評価結果に基づいて AP_FCGI_KEEP_CONN の有無が決定されます。

よくある落とし穴

  • **多重化の非サポート**: mod_proxy_fcgi 版の send_begin_request は FastCGI の多重化（Multiplexing）をサポートしておらず、単一の接続に対してリクエストを開始する設計になっています。
  • **ブロッキング I/O**: 送信関数（send_data / sendv_data）はブロッキング I/O を伴う可能性があるため、呼び出し時のコンテキストやスレッドのブロックに注意が必要です。
  • **接続再利用の制限**: mod_authnz_fcgi 版は AP_FCGI_KEEP_CONN を明示的に無効化しているため、認証処理において接続を再利用したい場合には、このロジックを修正する必要があります。
