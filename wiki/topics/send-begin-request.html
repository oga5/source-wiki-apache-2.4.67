send_begin_request

概念

send_begin_request は、Apache HTTP Server (httpd) において FastCGI プロトコルを用いたバックエンドサーバーとの通信を開始する際に、FCGI_BEGIN_REQUEST（または BEGIN_REQUEST）レコードを構築・送信するための関数です。この処理は、認証・認可を行うモジュール（mod_authnz_fcgi）と、プロキシ転送を行うモジュール（mod_proxy_fcgi）のそれぞれで個別に同名の関数として実装されています。

なぜ重要か

FastCGI プロトコルでは、バックエンドとのセッションを開始するために最初に BEGIN_REQUEST レコードを送信する必要があります。この関数は、リクエストの役割（ロール）や接続の維持（Keep-Alive）に関する設定を適切にヘッダーおよびボディに反映し、バックエンドとの正しいハンドシェイクを確立するために不可欠な役割を担っています。

関与するモジュール

主要エンティティ

  • symbol:modules/aaa/mod_authnz_fcgi.c#send_begin_request
  • 認証・認可プロバイダー用の FastCGI 接続開始関数。AP_FCGI_KEEP_CONN フラグは設定せず、接続の維持を行わない前提で動作します。
  • symbol:modules/proxy/mod_proxy_fcgi.c#send_begin_request
  • プロキシ転送用の FastCGI 接続開始関数。接続が再利用可能（ap_proxy_connection_reusable）な場合は AP_FCGI_KEEP_CONN フラグを付与します。

エンドツーエンドの流れ

各モジュールにおける具体的なエンドツーエンドの呼び出しフローの全体像（リクエストの受信からバックエンドの応答処理までの一連の流れ）は、提供された情報からは完全には明らかになっていません。

現時点で判明している各関数内部の送信フローは以下の通りです： 1. **ヘッダーの構築**: ap_fcgi_fill_in_header を呼び出し、AP_FCGI_BEGIN_REQUEST 型の FastCGI ヘッダーを生成します。 2. **ボディの構築**: ap_fcgi_fill_in_request_body を呼び出し、ロールや接続維持フラグ（AP_FCGI_KEEP_CONN）を設定したリクエストボディを生成します。 3. **配列変換**: ヘッダーとボディをそれぞれ ap_fcgi_header_to_array および ap_fcgi_begin_request_body_to_array を用いてバイト配列に変換します。 4. **送信**: iovec 構造体を用いてヘッダーとボディを連結し、ソケット経由で送信します。

  • mod_authnz_fcgi では sendv_data を使用します。
  • mod_proxy_fcgi では send_data を使用します。

重要な不変条件

  • **ソケット/接続の有効性**: 送信対象となるソケット（mod_authnz_fcgi の apr_socket_t *s）または接続（mod_proxy_fcgi の proxy_conn_rec *conn）は、呼び出し時点で有効かつ接続済みである必要があります。
  • **接続維持フラグの制御**:
  • mod_authnz_fcgi の実装では、AP_FCGI_KEEP_CONN フラグは設定されず、接続維持は行われません。
  • mod_proxy_fcgi の実装では、ap_proxy_connection_reusable の判定結果に基づいて AP_FCGI_KEEP_CONN フラグの有無が決定されます。

よくある落とし穴

  • **多重化の非サポート**: mod_proxy_fcgi の send_begin_request は FastCGI の多重化（Multiplexing）をサポートしておらず、単一の接続に対してリクエストを開始する設計となっています。
  • **ブロッキング I/O**: mod_proxy_fcgi で使用される send_data はブロッキング I/O を含む可能性があるため、呼び出し時のコンテキストやスレッドのブロックに注意が必要です。
  • **エラーハンドリング**: 送信処理（sendv_data または send_data）が失敗した場合、そのエラーコード（apr_status_t）がそのまま返されるため、呼び出し元で適切にハンドリングする必要があります。
