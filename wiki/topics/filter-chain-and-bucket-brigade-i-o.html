Filter Chain and Bucket Brigade I/O

概念

Filter Chain and Bucket Brigade I/O は、Apache HTTP Server がリクエストおよびレスポンスデータを処理するためのストリーミング制御メカニズムです。データは「バケット（Bucket）」およびそれを連結した「バケットブリゲード（Bucket Brigade: apr_bucket_brigade）」という単位で管理され、個々の処理を担当する「フィルタ（Filter）」が数珠つなぎになった「フィルタチェーン（Filter Chain）」を通過しながら、順次変換・転送されます。

なぜ重要か

Apache は、静的ファイルの読み込み、プロキシ転送、動的コンテンツ生成など、多種多様なデータソースとプロトコルを扱います。フィルタチェーンとバケットブリゲードを採用することで、以下のメリットが得られます。

  • **モジュール化と再利用性**: HTTP ヘッダの生成、チャンクエンコーディングのデコード、データの制限チェックなどの処理を独立したフィルタとして実装し、必要に応じて動的にチェーンへ挿入できます。
  • **高効率なI/O**: データを一括でメモリに読み込むことなく、バケット単位でパイプライン処理するため、メモリ消費量を抑えながら大容量データを扱えます。

関与するモジュール

提供された情報に基づき、以下のモジュールがこのメカニズムに関与しています。

  • module:include (フィルタの基本構造体定義を提供)
  • module:modules/http (HTTP プロトコル固有の入力・出力フィルタを実装)
  • module:server (コアフィルタの実装を担当)

主要エンティティ

本メカニズムを構成する主要なエンティティは以下の通りです。

  • **symbol:include/util_filter.h#ap_filter_rec_t** (構造体)

    • フィルタの定義情報を保持するメタデータ構造体です。フィルタ名、実行関数ポインタ、フィルタの型（優先順位）などを管理し、next ポインタを介して登録済みフィルタのリストを形成します。

  • **symbol:include/util_filter.h#ap_filter_t** (構造体)

    • フィルタチェーン内に実際にインスタンス化された個々のフィルタを表現します。ap_pass_brigade や ap_get_brigade を介して、隣接するフィルタとバケットブリゲードの受け渡しを行います。

  • **symbol:modules/http/http_filters.c#ap_http_filter** (関数)

    • HTTP リクエストボディの入力処理を担当するフィルタ関数です。Transfer-Encoding: chunked のデコード、Content-Length に基づく境界制御、および LimitRequestBody に基づくサイズ制限の強制を行います。

  • **symbol:modules/http/http_filters.c#ap_http_header_filter** (関数)

    • HTTP レスポンスヘッダの出力処理を担当するフィルタ関数です。

エンドツーエンドの流れ

提供された情報から確認できるデータの流れは以下の通りです。

1. **入力データの要求と取得**:

  • ハンドラや上位フィルタが ap_get_brigade を呼び出してデータを要求します。
  • これにより、入力フィルタチェーンが順次呼び出されます。

    • 2. **HTTPリクエストの解析とフィルタリング (ap_http_filter)**:

  • 初回呼び出し時、ap_http_filter は Transfer-Encoding や Content-Length ヘッダを解析し、内部状態（BODY_CHUNK や BODY_LENGTH など）を決定します。
  • 必要に応じて 100 Continue レスポンスを送信します。
  • 下位フィルタからデータを取得し、チャンクサイズ（parse_chunk_size）の解析や、Content-Length に基づく残りバイト数（ctx->remaining）の減算を行いながら、安全なデータバケットを構築して呼び出し元に返します。

    • 3. **出力データの送信 (ap_http_header_filter)**:

  • レスポンス送信時、ap_http_header_filter が呼び出され、ヘッダフィールドの結合や送信処理、および後続の出力フィルタへのブリゲードの受け渡し（ap_pass_brigade）を行います。

※ **注意**: フィルタチェーン全体の構築タイミングや、server/core_filters.c におけるネットワークソケットとの直接的なI/Oを伴う最終的なエンドツーエンドの制御フローの詳細は、提供された情報からは完全には特定されていません。

重要な不変条件

  • **フィルタの順序決定**: フィルタは ap_filter_type に基づいてチェーン内で厳密にソートされます。
  • **サブリクエストの制限**: AP_FTYPE_CONNECTION 以上の型を持つフィルタは、サブリクエスト内では使用できません。
  • **入力フィルタの開始状態**: ap_http_filter において、BODY_NONE 状態はトレーラヘッダ処理を除き、開始状態としては無効です。
  • **残りバイト数の追跡**: Content-Length が指定されている場合、ap_http_filter のコンテキスト（ctx->remaining）は、残りの読み込み可能バイト数を正確に保持しなければなりません。

※ **注意**: バケットの寿命管理や、エラー発生時におけるブリゲードのクリーンアップに関する詳細な不変条件は、提供された情報からは完全には特定されていません。

よくある落とし穴

  • **フィルタの登録順序の誤り**: フィルタの型（ap_filter_type）や登録順序が不適切な場合、チェーン内の実行順序が意図しないものになり、データ破損やプロトコル違反を引き起こす可能性があります。
  • **バケットブリゲードの所有権と退避**: ap_pass_brigade に渡されるブリゲードの所有権は呼び出し元にあります。フィルタ内部でデータを一時的に保持（遅延処理など）する場合は、ap_save_brigade を使用してデータを安全に退避させないと、後続の処理でデータが上書き・消失する危険があります。
  • **複雑な状態遷移のハンドリング**: ap_http_filter における ctx->state の遷移は RFC 7230 に準拠しており非常に複雑です。チャンクサイズや制限値（ctx->limit）の計算ロジックを修正する際は、状態遷移の不整合やバッファオーバーフローを誘発しやすいため、細心の注意が必要です。
  • **メモリリーク**: フィルタ処理中のメモリ管理はリクエストプール（f->r->pool）に依存しているため、ループ内でのバケット操作に伴う一時メモリの累積やリークに注意する必要があります。

次に読む場所

  • **include/util_filter.h**: フィルタの登録、チェーン構築、および ap_filter_t / ap_filter_rec_t の構造体定義を理解するために参照してください。
  • **modules/http/http_filters.c**: ap_http_filter や ap_http_header_filter の具体的な実装、および HTTP プロトコル制約（チャンク、Content-Length、ヘッダ処理）の強制方法を学ぶために参照してください。
  • **server/core_filters.c**: コアなネットワークI/Oとフィルタチェーンの接続部分を理解するために参照してください。
