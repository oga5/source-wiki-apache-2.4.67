handle_headers

概念

handle_headers は、FastCGI バックエンドから受信したストリームデータから HTTP ヘッダの終端（空行）を検出するためのステートマシンとして機能する関数です。

なぜ重要か

FastCGI プロトコルにおいて、バックエンドからのレスポンスヘッダは複数回に分割されて（断片化して）届く可能性があります。この関数は、呼び出し間で解析状態（state）を維持しながらストリームをバイト単位で走査し、\r\n\r\n または \n\n といったヘッダの終端シーケンスを正確に検出することで、ヘッダ部とボディ部の境界を識別する役割を担っています。

関与するモジュール

  • module:modules/aaa
  • module:modules/proxy

主要エンティティ

  • symbol:modules/proxy/mod_proxy_fcgi.c#handle_headers: 実際のヘッダ終端検出ロジックが実装されている関数。
  • symbol:modules/aaa/mod_authnz_fcgi.c#handle_headers: mod_proxy_fcgi.c 内の実装を呼び出す、または同様の処理を行う関連エンティティ。

エンドツーエンドの流れ

1. FastCGI バックエンドからデータを受信します。 2. 受信したデータバッファ（readbuf）と、現在の解析状態を保持する変数（state）を handle_headers に渡します。 3. handle_headers はバッファを走査し、改行コード（\r や \n）の出現パターンに応じて state を更新します。 4. ヘッダの終端（HDR_STATE_DONE_WITH_HEADERS）を検出した場合は 1 を返し、それ以外の場合は 0 を返して次のデータ受信を待機します。

※なお、FastCGI レスポンス全体の処理や、ヘッダ検出後の具体的な転送処理など、詳細なエンドツーエンドの全体フローは提供された情報からは完全には判明していません。

重要な不変条件

  • **状態の維持**: データの断片化に対応するため、呼び出し側は state 変数を初期化後、ヘッダ解析が完了するまで一貫して保持・更新し続ける必要があります。
  • **バッファの整合性**: 入力される readbuf は、指定された readlen 分の有効なデータを含んでいなければなりません。

よくある落とし穴

  • **状態のリセット**: 改行シーケンスの途中でそれ以外の文字が出現した場合、状態は HDR_STATE_READING_HEADERS にリセットされます。
  • **断片化の考慮**: 1回の呼び出しでヘッダ終端に達しない場合（戻り値が 0 の場合）、状態を破棄せずに次のデータ受信時にも同じ state を引き継いで呼び出す必要があります。
