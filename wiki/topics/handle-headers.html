handle_headers

概念

handle_headers は、FastCGI バックエンドから受信したストリームデータから HTTP ヘッダの終端（空行）を検出するためのステートマシンとして機能する関数です。

なぜ重要か

FastCGI プロトコルにおいて、バックエンドから返されるレスポンスのヘッダ部とボディ部を正しく分離するために不可欠です。ネットワークから受信するデータが断片化している場合でも、呼び出し間で解析状態を維持することで、正確にヘッダの終端（\r\n\r\n または \n\n）を検出できます。

エンドツーエンドの流れ

FastCGI 処理における完全なエンドツーエンドのライフサイクル（リクエストの受信からバックエンドへの転送、最終的なレスポンス返却まで）の全容は、提供された情報からは完全には明らかになっていません。

現時点で判明している部分的な流れは以下の通りです： 1. **データ受信**: FastCGI バックエンドからレスポンスデータを受信します。 2. **ヘッダ解析の実行**: 受信したバッファ（readbuf）と現在の解析状態（state）を引数として handle_headers を呼び出します。 3. **状態遷移**: handle_headers はバッファをバイト単位で走査し、\r または \n の出現パターンに応じて state を更新します。 4. **終端検出**: \r\n\r\n または \n\n に相当するシーケンスを検出すると、状態が HDR_STATE_DONE_WITH_HEADERS に移行し、関数は 1 を返します。終端に達しない場合は 0 を返します。 5. **後続処理**: 呼び出し元（mod_proxy_fcgi.c や mod_authnz_fcgi.c）は、戻り値が 1 になるまでデータの受信と解析を繰り返します。

重要な不変条件

  • **状態の一貫性**: 解析状態を示す state 変数は、ヘッダ解析の進行状況を追跡するために、呼び出し間で一貫して保持・維持されなければなりません。
  • **バッファの有効性**: 入力される readbuf は、readlen で指定されたサイズ分の有効なデータを含んでいる必要があります。
  • その他のシステムレベルの不変条件については、提供された情報からは未検証です。

よくある落とし穴

  • **状態のリセット**: データの断片化により複数回に分けて handle_headers が呼び出される際、呼び出し側で誤って state を初期化してしまうと、ヘッダの終端を正しく検出できなくなります。
  • **状態定義の不整合**: ヘッダの解析ロジックや状態遷移を変更する場合、mod_proxy_fcgi.c 内で定義されている HDR_STATE 定数の遷移図とコードの実装を正確に同期させる必要があります。
