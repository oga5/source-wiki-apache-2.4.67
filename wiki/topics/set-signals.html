set_signals

概念

set_signals は、Apache HTTP Serverの特定のマルチプロセッシングモジュール（MPM）において、プロセスの終了、再起動、割り込みなどのシグナルに対するハンドラを登録するための関数です。プラットフォーム（OS/2やNetWare）ごとに個別に実装されており、それぞれの環境に適したシグナルハンドリングをセットアップする役割を持ちます。

なぜ重要か

サーバープロセスが外部からの終了要求（SIGTERMなど）や再起動要求（SIGHUPなど）を適切に検知し、安全なシャットダウン処理や設定の再読み込みといったライフサイクル管理を正しく行うために不可欠です。これが正しく機能しない場合、プロセスがハングアップしたり、リソースが適切に解放されずに終了したりする可能性があります。

エンドツーエンドの流れ

各プラットフォームにおける詳細なエンドツーエンドの呼び出しフロー全体や、親プロセスから子プロセスへのシグナル伝播の完全なライフサイクルは、提供された情報からは完全には明らかになっていません。

ただし、各ファイル内での部分的な流れは以下の通りです。

OS/2 環境における流れ

  • **親プロセス (mpmt_os2.c)**:

    • 1. set_signals が呼び出される。 2. struct sigaction 構造体を初期化し、マスクを空に設定する。 3. SIGTERM および SIGINT に対して sig_term ハンドラを割り当てる。 4. SIGHUP および SIGUSR1 に対して sig_restart ハンドラを割り当てる。 5. sigaction システムコールを呼び出し、失敗した場合は ap_log_error を使用して警告ログを出力する。

  • **子プロセス (mpmt_os2_child.c)**:

    • 1. set_signals が呼び出される。 2. struct sigaction 構造体を初期化し、マスクを空に設定する。 3. SIGTERM に対して sig_term ハンドラを、SIGHUP に対して sig_hup ハンドラを割り当てる。 4. sigaction を呼び出し、失敗した場合は ap_log_error を使用して警告ログを出力する。

NetWare 環境における流れ (mpm_netware.c)

1. set_signals が呼び出される。 2. apr_signal を使用して、SIGTERM シグナルを sig_term 関数にバインドする。 3. apr_signal を使用して、SIGABRT シグナルを sig_term 関数にバインドする。

重要な不変条件

提供された情報から得られる不変条件は以下の通りです。

  • **OS/2 子プロセス (mpmt_os2_child.c)**: set_signals が呼び出される時点で、エラーログ出力のために ap_server_conf が初期化済みでなければなりません。
  • **NetWare (mpm_netware.c)**: apr_signal および sig_term 関数が利用可能でなければなりません。
  • **OS/2 共通**: sigaction システムコールが利用可能であり、対応するハンドラ関数（sig_term, sig_restart, sig_hup）が同一スコープまたはリンク可能な範囲で定義されている必要があります。

※システム全体におけるシグナル受信時の厳密な状態遷移や、プロセス間の同期に関するグローバルな不変条件は、提供された情報からは完全には明らかになっていません。

よくある落とし穴

  • **プラットフォーム依存性**: set_signals は同名で複数のMPMに存在しますが、OS/2では標準の sigaction システムコールを使用し、NetWareでは APR ライブラリの apr_signal を使用するなど、実装や対象シグナルが大きく異なります。共通コードとして扱おうとすると移植性の問題が生じます。
  • **エラーハンドリングの不完全さ**:
  • OS/2の実装では、sigaction が失敗した際に警告ログを出力しますが、プロセスの実行はそのまま継続されます。この場合、シグナルによる終了や再起動が期待通りに動作しない可能性があります。
  • NetWareの実装では、apr_signal が内部的にエラーを返した場合の明示的なエラーハンドリングが行われていません。
