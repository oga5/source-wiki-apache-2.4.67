set_signals

概念

set_signals は、特定のマルチプロセッシングモジュール（MPM）において、プロセスの終了、再起動、割り込み、あるいは異常終了といったライフサイクルイベントを制御するために、OSレベルのシグナルハンドラを登録するプラットフォーム依存の内部関数です。

なぜ重要か

OS/2 や NetWare などの特定のオペレーティングシステム環境において、Apache HTTP Server の親プロセスおよび子プロセスが、外部からの終了要求（SIGTERM など）や再起動要求（SIGHUP など）を適切に検知し、安全なシャットダウンや設定の再読み込みを協調して行うために不可欠な役割を担っています。

関与するモジュール

  • module:server/mpm/mpmt_os2
  • module:server/mpm/netware

主要エンティティ

  • **symbol:server/mpm/mpmt_os2/mpmt_os2.c#set_signals**

    • OS/2 環境の親プロセスにおいて、SIGTERM/SIGINT（終了処理用ハンドラ sig_term）および SIGHUP/SIGUSR1（再起動処理用ハンドラ sig_restart）を登録します。

  • **symbol:server/mpm/mpmt_os2/mpmt_os2_child.c#set_signals**

    • OS/2 環境の子プロセスにおいて、SIGTERM（終了用ハンドラ sig_term）および SIGHUP（設定再読み込み用ハンドラ sig_hup）を登録します。

  • **symbol:server/mpm/netware/mpm_netware.c#set_signals**

    • NetWare 環境において、SIGTERM および SIGABRT を終了処理用ハンドラ sig_term にバインドします。

エンドツーエンドの流れ

1. **初期化と呼び出し**: 各プラットフォームの MPM 起動時（親プロセスまたは子プロセスの初期化フェーズ）において、それぞれの set_signals が呼び出されます。 2. **ハンドラの登録**:

  • **OS/2 (mpmt_os2)**: struct sigaction 構造体を初期化し、sigaction システムコールを呼び出してシグナルとハンドラ関数（sig_term, sig_restart, sig_hup）をマッピングします。
  • **NetWare (mpm_netware)**: APR（Apache Portable Runtime）ライブラリの apr_signal を使用して、SIGTERM および SIGABRT を sig_term にバインドします。

    • 3. **シグナル受信時**: 登録されたシグナルを受信すると、対応するハンドラが起動し、プロセスの安全な終了や再起動処理へ移行します。

*注意: ツールによる静的解析の推測に基づくと、mpmt_os2.c、mpmt_os2_child.c、および mpm_netware.c の間で呼び出し元が一部重複して検出されていますが、プラットフォームを跨いだ詳細な実行時制御フローの全容は完全には判明していません。*

重要な不変条件

  • **サーバー設定の初期化**: mpmt_os2_child.c の set_signals では、エラーログ出力（ap_log_error）に ap_server_conf を使用するため、本関数が呼び出される時点でサーバー設定が初期化済みでなければなりません。
  • **ハンドラ関数の存在**: 登録対象となるハンドラ関数（sig_term, sig_restart, sig_hup）が、同一スコープまたはリンク可能な範囲で定義されている必要があります。

よくある落とし穴

  • **プラットフォーム間の実装差異**:

    • 同名の関数であっても、OS/2 では標準の sigaction システムコールを使用するのに対し、NetWare では APR の apr_signal を使用するなど、実装がプラットフォームに強く依存しています。

  • **エラーハンドリングの不完全さ**:
  • OS/2 環境では、sigaction が失敗した場合に警告ログを出力しますが、プロセスの実行自体は継続されます。この場合、シグナルによる終了や再起動が機能しなくなる可能性があります。
  • NetWare 環境では、apr_signal が内部的に失敗した場合の明示的なエラーハンドリングは行われません。
