pass_response

概念

pass_response は、Apache HTTP Server (httpd) の複数のモジュール（mod_http2 および mod_proxy_scgi）において、バックエンドや前段の解析処理から受け取ったレスポンスデータを処理・変換し、フィルタチェーンやクライアントへ引き渡す（passする）役割を持つ静的関数の名前です。

これらは同名ですが、それぞれのモジュール内で独立して定義されており、異なるプロトコル（HTTP/2、SCGI）のコンテキストでレスポンス処理の最終ステップやヘッダー解析・転送を担当します。

なぜ重要か

レスポンスデータの適切な変換とフィルタチェーンへの受け渡しは、Webサーバーの正確なプロトコル変換と効率的なデータ転送において不可欠です。

  • **mod_http2 において**: 解析された HTTP/1.1 形式のレスポンスを HTTP/2 構造（ヘッダーバケット）へ変換して後続フィルタに流すことで、HTTP/2 通信を実現します。
  • **mod_proxy_scgi において**: SCGI バックエンドからのレスポンスヘッダーを解析し、X-Sendfile や内部リダイレクトといった特殊な処理の判定、および不要なヘッダーの調整を行った上で、レスポンスボディをクライアントへ転送します。

エンドツーエンドの流れ

これら2つの関数は異なるコンテキストで動作するため、それぞれの流れは独立しています。また、システム全体における完全なエンドツーエンドの統合フローは、提供された情報からは完全には明らかになっていません。判明している各モジュール内での流れは以下の通りです。

HTTP/2 フィルタにおける流れ (h2_c2_filter.c)

1. 解析ロジック（parse_response 等）から呼び出されます。 2. h2_response_parser の情報から h2_headers オブジェクトを生成します。 3. parser->tmp ブリゲードを初期化し、生成したヘッダーバケットとフラッシュバケットを挿入します。 4. ap_pass_brigade を呼び出して後続のフィルタへデータを渡します。 5. 処理完了後、parser の状態を H2_RP_STATUS_LINE にリセットし、ヘッダー行配列をクリアして次のレスポンス解析に備えます。 6. レスポンスステータスが 200 以上の場合、conn_ctx->has_final_response を 1 に設定します。

SCGI プロキシにおける流れ (mod_proxy_scgi.c)

1. SCGIハンドラのメインフロー（scgi_handler）から呼び出されます。 2. SOCKET_EX バケットを作成し、バックエンドソケットと統計カウンタを紐付けてブリゲードに挿入します。 3. ap_scan_script_header_err_brigade_ex を呼び出して、バックエンドからのレスポンスヘッダーを解析します。 4. Content-Length および Transfer-Encoding ヘッダーを削除します。 5. X-Sendfile や内部リダイレクトの設定がある場合、該当するヘッダーを検索し、存在すれば scgi_request_config を設定してサブリクエストや内部リダイレクトへ移行します。 6. 上記の特殊処理に該当しない場合、ap_pass_brigade を介してレスポンスボディをクライアントへ転送します。

重要な不変条件

各関数の動作において、以下の不変条件が維持される必要があります。

  • **HTTP/2 フィルタ (h2_c2_filter.c)**:
  • parser 内の http_status および hlines が、有効な HTTP/1.1 レスポンスとして解析済みである必要があります。
  • parser->tmp ブリゲードが利用可能である必要があります。
  • 同一コネクション上で複数のレスポンスが連続して発生する HTTP/2 の特性をサポートするため、処理完了後に parser の状態が必ずリセットされる必要があります。
  • **SCGI プロキシ (mod_proxy_scgi.c)**:
  • r->pool はリクエストの生存期間中有効である必要があります。
  • conn->sock は有効なバックエンドソケットである必要があります。
  • コアによる自動決定を優先するため、SCGI プロトコル特有のボディフレーミングに関連する Content-Length や Transfer-Encoding ヘッダーは削除される必要があります。

よくある落とし穴

  • **名前の混同**: modules/http2/h2_c2_filter.c と modules/proxy/mod_proxy_scgi.c の pass_response は、名前は同一ですがシグネチャも役割も異なる別個の静的関数です。これらを混同して同一のグローバルなフローとして扱わないよう注意が必要です。
  • **エラー伝播時のリソース管理**: いずれの関数も ap_pass_brigade やヘッダー解析関数の戻り値をそのまま呼び出し元へ伝播させますが、エラー発生時に適切にブリゲードの破棄や状態管理が行われないと、メモリリークや不正な状態遷移を引き起こす可能性があります。
  • **ヘッダー走査の範囲**: SCGI における X-Sendfile や内部リダイレクトの判定ロジックは、r->err_headers_out と r->headers_out の両方を走査する必要があり、片方のみの確認では意図した動作にならない落とし穴があります。
