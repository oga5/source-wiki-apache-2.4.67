pass_response

概念

pass_response は、Apache HTTP Server（httpd-2.4.67）の複数のモジュールにおいて、バックエンドや前段の処理から受け取ったレスポンスデータを解析・変換し、フィルタチェーンを介して後続の処理やクライアントへ送信（パス）する役割を持つ、同名の内部関数の総称です。

具体的には、以下の2つの異なるコンテキストで同名の関数が定義されています。

  • **HTTP/2 フィルタ (mod_http2)**: HTTP/1.1 形式で解析されたレスポンスデータを HTTP/2 構造（ヘッダーバケット）へ変換して送信します。
  • **SCGI プロキシ (mod_proxy_scgi)**: SCGI バックエンドからのレスポンスを読み込み、ヘッダー解析や X-Sendfile / 内部リダイレクトの判定を行い、レスポンスを転送します。

なぜ重要か

各プロトコル（HTTP/2 や SCGI）の境界において、レスポンスデータのフォーマット変換、ヘッダーの調整（不要なヘッダーの削除や特殊ヘッダー X-Sendfile などの解釈）、およびフィルタチェーン（ap_pass_brigade）への適切な受け渡しを担うため、レスポンス処理の正常な完了とプロトコル適合性の維持に不可欠です。

エンドツーエンドの流れ

これら2つの関数は、それぞれのモジュール内で独立して動作しており、両者を直接繋ぐ単一のエンドツーエンドのフローは存在しないか、または現時点では不明です。それぞれのモジュール内での処理の流れは以下の通りです。

HTTP/2 フィルタにおける流れ (h2_c2_filter.c)

1. 解析済みのレスポンス状態（h2_response_parser）から h2_headers オブジェクトを生成します。 2. parser->tmp ブリゲードを初期化し、生成したヘッダーバケットとフラッシュバケットを挿入します。 3. ap_pass_brigade を呼び出して後続のフィルタへデータを渡します。 4. 処理完了後、parser の状態を H2_RP_STATUS_LINE にリセットし、ヘッダー行配列をクリアして次のレスポンス解析に備えます。 5. レスポンスステータスが 200 以上の場合、conn_ctx->has_final_response を 1 に設定します。

SCGI プロキシにおける流れ (mod_proxy_scgi.c)

1. scgi_handler のメインフローから呼び出されます。 2. バックエンドソケットと統計カウンタを紐付けた SOCKET_EX バケットを作成し、ブリゲードに挿入します。 3. ap_scan_script_header_err_brigade_ex を呼び出し、バックエンドからのレスポンスヘッダーを解析します。 4. コアによる自動決定を優先するため、Content-Length および Transfer-Encoding をヘッダーから削除します。 5. X-Sendfile や内部リダイレクトの設定が有効な場合、指定されたヘッダー（Location など）を検索し、条件に合致すれば scgi_request_config を設定してサブリクエストや内部リダイレクトへ移行します。 6. 特殊処理に該当しない場合、ap_pass_brigade を介してレスポンスボディをクライアントへ転送します。

重要な不変条件

  • **HTTP/2 (h2_c2_filter.c)**:
  • parser 内の http_status および hlines が、有効な HTTP/1.1 レスポンスとして解析済みである必要があります。
  • parser->tmp ブリゲードが利用可能である必要があります。
  • 同一コネクション上で複数のレスポンスが連続して発生する HTTP/2 の特性をサポートするため、処理完了後に parser の状態リセット（H2_RP_STATUS_LINE への移行とヘッダー行配列のクリア）が確実に行われなければなりません。
  • **SCGI (mod_proxy_scgi.c)**:
  • r->pool はリクエストの生存期間中有効である必要があります。
  • conn->sock は有効なバックエンドソケットである必要があります。
  • SCGI プロトコル特有のボディフレーミングが使用されるため、Content-Length や Transfer-Encoding ヘッダーは削除されていなければなりません。

よくある落とし穴

  • **共通のエラー伝播**:
  • ap_pass_brigade がエラーを返した場合、そのステータスコード（apr_status_t または AP_FILTER_ERROR）がそのまま呼び出し元へ伝播されるため、下流のフィルタでの失敗が適切にハンドリングされる必要があります。
  • **HTTP/2 における状態管理**:
  • parser の状態リセットを怠ると、同一コネクション上で連続する後続レスポンスの解析が破損します。
  • **SCGI におけるヘッダー走査**:
  • X-Sendfile や内部リダイレクトの判定ロジックは、r->err_headers_out と r->headers_out の両方を走査する必要があるため、一方のみを参照すると判定漏れが発生する可能性があります。
