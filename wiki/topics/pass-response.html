pass_response

概念

pass_response は、Apache HTTP Server（httpd-2.4.67）の複数のモジュールにおいて、受信したレスポンスデータを解析・変換し、次のフィルタチェーンやクライアントへ転送（パス）する処理を担当する関数の名称です。

これらは同名の static 関数ですが、HTTP/2 接続フィルタ（mod_http2）と SCGI プロキシ（mod_proxy_scgi）という、全く異なる文脈と役割を持つ2つのモジュールで個別に定義されています。

なぜ重要か

それぞれのモジュールにおける pass_response の重要性は以下の通りです。

  • **HTTP/2 (modules/http2)**: HTTP/1.1 形式で解析されたレスポンスデータを HTTP/2 のヘッダー構造（h2_headers）へ変換し、フィルタチェーンへ流すための最終ステップを担います。これにより、HTTP/2 コネクション上での正しいレスポンス送信と、複数レスポンスの連続処理が可能になります。
  • **SCGI プロキシ (modules/proxy)**: SCGI バックエンドから受信したレスポンスのヘッダーを解析し、Content-Length や Transfer-Encoding の調整、X-Sendfile や内部リダイレクトの判定といった高度なレスポンス制御を適用した上で、クライアントへデータを転送する役割を持ちます。

エンドツーエンドの流れ

これら2つの関数は独立したモジュールに存在し、直接的な連携を行うエンドツーエンドのフローは存在しないか、または現時点では未検証です。それぞれのモジュール内における個別の処理フローは以下の通りです。

1. HTTP/2 フィルタにおける流れ (h2_c2_filter.c)

1. 解析ロジック（parse_response 等のフロー内）から呼び出されます。 2. h2_response_parser の情報を用いて h2_headers オブジェクトを生成します。 3. 一時ブリゲード（parser->tmp）を初期化し、生成したヘッダーバケットとフラッシュバケットを挿入します。 4. ap_pass_brigade を呼び出し、後続のフィルタへデータを渡します。 5. 処理完了後、parser の状態を H2_RP_STATUS_LINE にリセットし、ヘッダー行配列をクリアして次のレスポンス解析に備えます。 6. レスポンスステータスが 200 以上の場合、conn_ctx->has_final_response を 1 に設定します。

2. SCGI プロキシにおける流れ (mod_proxy_scgi.c)

1. SCGI ハンドラのメインフロー（scgi_handler）から呼び出されます。 2. バックエンドソケットと統計カウンタを紐付けた SOCKET_EX バケットを作成し、ブリゲードに挿入します。 3. ap_scan_script_header_err_brigade_ex を呼び出し、バックエンドからのレスポンスヘッダーを解析します。 4. Content-Length および Transfer-Encoding をヘッダーから削除します。 5. X-Sendfile や内部リダイレクトの設定がある場合、r->err_headers_out と r->headers_out を走査して該当ヘッダーを検索し、存在すれば scgi_request_config を設定してサブリクエストや内部リダイレクトへ移行します。 6. 上記の特殊処理に該当しない場合、ap_pass_brigade を介してレスポンスボディをクライアントへ転送します。

重要な不変条件

それぞれの関数において以下の不変条件が求められますが、システム全体を横断する共通の不変条件は存在しません。

  • **HTTP/2 フィルタ (h2_c2_filter.c)**:
  • parser 内の http_status および hlines が、有効な HTTP/1.1 レスポンスとして解析済みであること。
  • parser->tmp ブリゲードが利用可能であること。
  • 同一コネクション上で複数のレスポンスが連続して発生する HTTP/2 の特性をサポートするため、処理完了後に parser の状態リセット（H2_RP_STATUS_LINE への移行、ヘッダー行配列のクリア）が確実に行われること。
  • **SCGI プロキシ (mod_proxy_scgi.c)**:
  • r->pool はリクエストの生存期間中有効であること。
  • conn->sock は有効なバックエンドソケットであること。
  • コアによる自動決定を優先するため、SCGI プロトコル特有のボディフレーミングに関連する Content-Length や Transfer-Encoding ヘッダーは削除されること。

よくある落とし穴

  • **名前の混同**: h2_c2_filter.c と mod_proxy_scgi.c の pass_response は、名前は同一ですが、シグネチャも役割も異なる全く別の静的（static）関数です。これらが直接相互作用することはありません。
  • **エラーハンドリングの違い**:
  • HTTP/2 側では、ap_pass_brigade がエラーを返した場合、そのステータスコードがそのまま呼び出し元へ伝播されます。
  • SCGI 側では、ヘッダー解析失敗時に ap_scan_script_header_err_brigade_ex のエラーコードを返し、フィルタリング失敗（ap_pass_brigade のエラー）時には AP_FILTER_ERROR を返します。
  • **SCGI のヘッダー走査**: X-Sendfile や内部リダイレクトの判定において、r->err_headers_out と r->headers_out の両方を走査する必要があるため、どちらか一方のみを対象とすると判定漏れが生じる可能性があります。
