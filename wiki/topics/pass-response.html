pass_response

概念

pass_response は、Apache HTTP Server（httpd-2.4.67）の複数のモジュールにおいて、バックエンドや解析器から受け取ったレスポンスデータを処理・変換し、フィルタチェーンを通じて後続の処理へ引き渡す（pass する）ために定義されている静的関数です。

この名称を持つ関数は、以下の2つの異なる文脈で独立して存在しています。 1. **HTTP/2 接続フィルタ (h2_c2_filter.c)**: HTTP/1.1 形式で解析されたレスポンスデータを HTTP/2 構造（ヘッダーバケット）へ変換し、フィルタチェーンへ送信します。 2. **SCGI プロキシ (mod_proxy_scgi.c)**: SCGI バックエンドからのレスポンスを読み込み、ヘッダー解析、X-Sendfile や内部リダイレクトの判定、およびクライアントへの転送を行います。

なぜ重要か

各モジュールにおいて、レスポンス処理の最終段階や、プロトコル変換・ヘッダー調整を伴うデータ転送の要となるためです。

  • **HTTP/2 フィルタ**では、HTTP/1.1 形式のレスポンスを HTTP/2 のヘッダーバケットに変換して正しく送信し、次のレスポンス解析に備えて状態をリセットする役割を持ちます。
  • **SCGI プロキシ**では、バックエンドからのレスポンスヘッダーを解析し、Content-Length や Transfer-Encoding の調整、X-Sendfile や内部リダイレクトといった特殊な処理を適切にハンドリングしてクライアントへデータを流す役割を持ちます。

エンドツーエンドの流れ

これら2つの関数は、それぞれ独立したモジュールで定義されており、直接的な連携関係はありません。それぞれの文脈における流れは以下の通りです。

HTTP/2 フィルタ (h2_c2_filter.c) の場合

1. 解析済みのレスポンス状態（h2_response_parser）から h2_headers オブジェクトを生成します。 2. parser->tmp ブリゲードを初期化し、生成したヘッダーバケットとフラッシュバケットを挿入します。 3. ap_pass_brigade を呼び出し、後続のフィルタへデータを渡します。 4. 処理完了後、parser の状態を H2_RP_STATUS_LINE にリセットし、ヘッダー行配列をクリアして次のレスポンス解析に備えます。 5. レスポンスステータスが 200 以上の場合、conn_ctx->has_final_response を 1 に設定します。

SCGI プロキシ (mod_proxy_scgi.c) の場合

1. SOCKET_EX バケットを作成し、バックエンドソケットと統計カウンタを紐付けてブリゲードに挿入します。 2. ap_scan_script_header_err_brigade_ex を呼び出し、バックエンドからのレスポンスヘッダーを解析します。 3. Content-Length および Transfer-Encoding をヘッダーから削除します。 4. X-Sendfile や内部リダイレクトの設定がある場合、指定されたヘッダーを検索し、存在すれば scgi_request_config を設定してサブリクエストや内部リダイレクトへ移行します。 5. 特殊処理に該当しない場合、ap_pass_brigade を介してレスポンスボディをクライアントへ転送します。

※ システム全体におけるこれら2つの処理の統合的なエンドツーエンドの相互作用や、詳細な呼び出し元からの完全な制御フローは、現時点では完全には判明していません。

重要な不変条件

  • **HTTP/2 フィルタ (h2_c2_filter.c)**:
  • parser 内の http_status および hlines が、有効な HTTP/1.1 レスポンスとして解析済みである必要があります。
  • parser->tmp ブリゲードが利用可能でなければなりません。
  • 同一コネクション上での連続するレスポンス解析をサポートするため、処理完了後に parser の状態が H2_RP_STATUS_LINE にリセットされ、ヘッダー行配列がクリアされる必要があります。
  • **SCGI プロキシ (mod_proxy_scgi.c)**:
  • r->pool はリクエストの生存期間中有効である必要があります。
  • conn->sock は有効なバックエンドソケットでなければなりません。
  • コアによる自動決定を優先するため、SCGI プロトコル特有のボディフレーミングにおいて Content-Length や Transfer-Encoding ヘッダーが削除される必要があります。

よくある落とし穴

  • **HTTP/2 フィルタ (h2_c2_filter.c)**:
  • ap_pass_brigade がエラーを返した場合、そのステータスコードがそのまま呼び出し元へ伝播されるため、エラーハンドリングが不適切だと接続の切断やリソースリークに繋がる可能性があります。
  • parser の状態リセットが正しく行われないと、後続のレスポンス解析が破損します。
  • **SCGI プロキシ (mod_proxy_scgi.c)**:
  • ヘッダー解析に失敗した場合、ap_scan_script_header_err_brigade_ex がエラーを返し、ブリゲードが破棄されて処理が中断されます。
  • X-Sendfile や内部リダイレクトの判定において、r->err_headers_out と r->headers_out の両方を走査する必要があるため、どちらか一方のみを参照すると意図したリダイレクトやファイル送信が行われません。
  • **共通**:
  • 同一名称の関数ですが、モジュール間で直接的な関係はなく、シグネチャも振る舞いも異なるため、混同しないよう注意が必要です。
