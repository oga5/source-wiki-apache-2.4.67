pass_response

概念

pass_response は、Apache HTTP Server（httpd-2.4.67）の複数モジュールにおいて、バックエンドや前段の処理から受け取ったレスポンスデータを解析・調整し、フィルタチェーンを介して後続の処理やクライアントへ送信（パス）する役割を持つ、同名の独立した内部関数の総称です。

なぜ重要か

各プロトコルやハンドラの境界において、レスポンスデータを適切な形式に変換・フィルタリングして後続の処理へ引き渡すために重要です。

  • **HTTP/2 (mod_http2)**: HTTP/1.1 形式で解析されたレスポンスデータを HTTP/2 のヘッダー構造（h2_headers）に変換し、フィルタチェーンへ送信することで、HTTP/2 接続上での正しいレスポンス返却を可能にします。
  • **SCGI プロキシ (mod_proxy_scgi)**: バックエンドの SCGI サーバーから受信したレスポンスのヘッダー解析、不要なヘッダーの調整、X-Sendfile や内部リダイレクトの判定を行い、適切にクライアントへデータを転送します。

関与するモジュール

  • module:modules/http2
  • module:modules/proxy

主要エンティティ

  • symbol:modules/http2/h2_c2_filter.c#pass_response: HTTP/1.1 形式のレスポンスを HTTP/2 構造（ヘッダーバケット）へ変換し、フィルタチェーンへ送信する関数。
  • symbol:modules/proxy/mod_proxy_scgi.c#pass_response: SCGI バックエンドからのレスポンスを読み込み、ヘッダー解析、調整、および X-Sendfile や内部リダイレクトの判定を行う関数。

エンドツーエンドの流れ

これら2つの pass_response 関数は、それぞれのモジュール内で独立して動作しており、両者を直接繋ぐ単一のエンドツーエンドのフローは存在しません。それぞれの独立した流れは以下の通りです。

HTTP/2 における流れ (h2_c2_filter.c)

1. **呼び出し**: parse_response などの解析ロジックから呼び出されます。 2. **ヘッダー生成**: h2_response_parser の情報を用いて h2_headers オブジェクトを生成します。 3. **ブリゲード構築**: parser->tmp ブリゲードを初期化し、生成したヘッダーバケットとフラッシュバケットを挿入します。 4. **送信**: ap_pass_brigade を呼び出し、後続のフィルタへデータを渡します。 5. **状態リセット**: parser の状態を H2_RP_STATUS_LINE にリセットし、ヘッダー行配列をクリアして次のレスポンス解析に備えます。

SCGI プロキシにおける流れ (mod_proxy_scgi.c)

1. **呼び出し**: SCGI ハンドラのメインフロー（scgi_handler）から呼び出されます。 2. **バケット作成**: SOCKET_EX バケットを作成し、バックエンドソケットと統計カウンタを紐付けてブリゲードに挿入します。 3. **ヘッダー解析**: ap_scan_script_header_err_brigade_ex を呼び出し、バックエンドからのレスポンスヘッダーを解析します。 4. **ヘッダー調整**: コアによる自動決定を優先するため、Content-Length および Transfer-Encoding をヘッダーから削除します。 5. **特殊処理の判定**: X-Sendfile や内部リダイレクトの設定がある場合、対象ヘッダーを検索し、存在すれば scgi_request_config を設定してサブリクエストや内部リダイレクトへ移行します。 6. **転送**: 特殊処理に該当しない場合、ap_pass_brigade を介してレスポンスボディをクライアントへ転送します。

重要な不変条件

  • **HTTP/2 (h2_c2_filter.c)**:
  • parser 内の http_status および hlines が、有効な HTTP/1.1 レスポンスとして解析済みである必要があります。
  • レスポンスステータスが 200 以上の場合、conn_ctx->has_final_response が必ず 1 に設定されます。
  • **SCGI プロキシ (mod_proxy_scgi.c)**:
  • r->pool はリクエストの生存期間中有効でなければなりません。
  • conn->sock は有効なバックエンドソケットである必要があります。
  • SCGI プロトコル特有のボディフレーミングが使用されるため、Content-Length や Transfer-Encoding ヘッダーはコアによる自動決定を優先するために削除されます。

よくある落とし穴

  • **同名関数の混同**: modules/http2/h2_c2_filter.c と modules/proxy/mod_proxy_scgi.c に存在する pass_response は、名前は同一ですが、シグネチャも役割も異なる完全に独立した静的（static）関数です。これらを混同しないよう注意が必要です。
  • **エラーの伝播**: 両関数とも ap_pass_brigade を呼び出しますが、フィルタリングや送信の失敗が発生した場合は、そのステータス（または AP_FILTER_ERROR）をそのまま呼び出し元へ伝播させるため、呼び出し元での適切なエラーハンドリングが求められます。
  • **状態リセットの欠如**: HTTP/2 側では、同一コネクション上で複数のレスポンスが連続して発生するため、処理完了後に parser の状態（H2_RP_STATUS_LINE へのリセットやヘッダー行配列のクリア）を正しく初期化しないと、後続のレスポンス解析が破損します。
