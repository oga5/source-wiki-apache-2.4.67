pass_response

概念

pass_response は、Apache HTTP Server（httpd）の複数のモジュール（mod_http2 および mod_proxy_scgi）において、バックエンドや前段の解析処理から受け取ったレスポンスデータを処理し、フィルタチェーンやクライアントへ引き渡す（passする）ために定義されている静的（static）関数です。同名ですが、それぞれのモジュールで異なる役割を持ち、直接的な関係はありません。

なぜ重要か

各モジュールにおけるレスポンス処理の境界（インターフェース）として機能するため重要です。

  • **HTTP/2（mod_http2）**: HTTP/1.1形式で解析されたレスポンスをHTTP/2のヘッダー構造に変換して後続のフィルタに流す役割を持ち、HTTP/2通信の整合性を維持します。
  • **SCGIプロキシ（mod_proxy_scgi）**: SCGIバックエンドからのレスポンスヘッダーを解析し、X-Sendfile や内部リダイレクトなどの特殊な処理を判定・適用した上でクライアントにデータを転送する役割を持ちます。

エンドツーエンドの流れ

これら2つの関数は独立したモジュールに存在し、直接的な連携を行うエンドツーエンドの流れは存在しません。それぞれのモジュール内における個別の流れは以下の通りです。

HTTP/2 (h2_c2_filter.c) 内の流れ

1. 解析ロジック（parse_response 等）から呼び出されます。 2. h2_response_parser の情報を用いて h2_headers オブジェクトを生成します。 3. parser->tmp ブリゲードを初期化し、生成したヘッダーバケットとフラッシュバケットを挿入します。 4. ap_pass_brigade を呼び出し、後続のフィルタへデータを渡します。 5. 処理完了後、parser の状態を H2_RP_STATUS_LINE にリセットし、ヘッダー行配列をクリアして次のレスポンス解析に備えます。

SCGIプロキシ (mod_proxy_scgi.c) 内の流れ

1. SCGIハンドラのメインフロー（scgi_handler）から呼び出されます。 2. SOCKET_EX バケットを作成し、バックエンドソケットと統計カウンタを紐付けてブリゲードに挿入します。 3. ap_scan_script_header_err_brigade_ex を呼び出し、バックエンドからのレスポンスヘッダーを解析します。 4. Content-Length および Transfer-Encoding をヘッダーから削除します。 5. X-Sendfile や内部リダイレクトの設定がある場合、指定されたヘッダー（Location など）を検索し、条件に合致すれば scgi_request_config を設定してサブリクエストや内部リダイレクトへ移行します。 6. 上記の特殊処理に該当しない場合、ap_pass_brigade を介してレスポンスボディをクライアントへ転送します。

重要な不変条件

HTTP/2 (h2_c2_filter.c)

  • parser 内の http_status および hlines が、有効な HTTP/1.1 レスポンスとして解析済みである必要があります。
  • parser->tmp ブリゲードが利用可能である必要があります。
  • レスポンスステータスが 200 以上の場合、conn_ctx->has_final_response が 1 に設定されます。

SCGIプロキシ (mod_proxy_scgi.c)

  • r->pool はリクエストの生存期間中有効である必要があります。
  • conn->sock は有効なバックエンドソケットである必要があります。
  • コアによる自動決定を優先するため、Content-Length や Transfer-Encoding ヘッダーは削除されます。

よくある落とし穴

  • **共通**: 同名の関数ですが、シグネチャも目的も全く異なるため、混同しないよう注意が必要です。
  • **HTTP/2 (h2_c2_filter.c)**:
  • ap_pass_brigade がエラーを返した場合、そのステータスコードがそのまま呼び出し元へ伝播されます。
  • 同一コネクション上で複数のレスポンスが連続して発生する HTTP/2 の特性上、処理完了後に parser の状態リセット（H2_RP_STATUS_LINE へのリセットおよびヘッダー行配列のクリア）が確実に行われなければなりません。
  • **SCGIプロキシ (mod_proxy_scgi.c)**:
  • ヘッダー解析に失敗した場合、ap_scan_script_header_err_brigade_ex がエラーを返し、ログ出力後にブリゲードが破棄されます。
  • X-Sendfile や内部リダイレクトの判定ロジックは、r->err_headers_out と r->headers_out の両方を走査する必要があります。
