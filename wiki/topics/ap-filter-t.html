ap_filter_t

概念

ap_filter_t は、Apache HTTP Server のフィルタチェーンにおいて、インストールされた個々のフィルタインスタンスを表現する構造体です（include/util_filter.h）。接続（conn_rec）やリクエスト（request_rec）に関連付けられ、パイプラインを流れるデータを処理する役割を持ちます。

なぜ重要か

Apache のモジュールアーキテクチャにおいて、データの入力・出力をモジュール化し、段階的に加工・制御するための基盤となるからです。ap_filter_t を介して、複数のフィルタが apr_bucket_brigade 形式のデータストリームを順次処理（データの変形、ヘッダーの付与、コンテンツ長の計算など）していくことができます。

エンドツーエンドの流れ

提供されたコンテキストからは、フィルタチェーンの構築から破棄にいたる完全なエンドツーエンドの流れは定義されていません。

部分的に判明している流れは以下の通りです： 1. フィルタは、ap_pass_brigade や ap_get_brigade を介して、apr_bucket_brigade データストリームを順次受け取ります。 2. 各フィルタインスタンスは、ctx ポインタ（状態保持用）や ap_save_brigade を用いてデータを一時保存しながら処理を行います。 3. 処理されたデータは、チェーン内の次のフィルタへと引き渡されます。

重要な不変条件

  • **フィルタのソート順**: フィルタは ap_filter_type に基づいてチェーン内でソートされます。
  • **サブリクエストの制限**: AP_FTYPE_CONNECTION 以上のフィルタは、サブリクエスト内では使用できません。

よくある落とし穴

  • **ブリゲードの所有権**: ap_pass_brigade に渡されるブリゲードの所有権は呼び出し元にあります。そのため、フィルタ内でデータを保持・退避させる場合は、必ず ap_save_brigade を使用する必要があります。
  • **リソースの解放**: ストリーム終了マーカー（EOSなど）の到達時に、各フィルタは内部状態を適切にフラッシュし、確保したリソースを解放しなければなりません。
  • **不透明な構造体**: ap_filter_t は不透明ポインタとして扱われることが多いため、詳細なメンバへアクセスする際は include/util_filter.h の完全な定義を参照する必要があります。
