ap_filter_t

概念

ap_filter_t は、Apache HTTP Server のフィルタチェーンにおいて、インストールされた個々のフィルタインスタンスを表現する構造体です。

なぜ重要か

フィルタチェーンを通過するデータストリーム（apr_bucket_brigade）の入出力を仲介し、各フィルタが状態を保持しながら順次処理を行うための基盤となるためです。これにより、モジュール化されたデータの加工やプロトコル処理が可能になります。

関与するモジュール

  • module:include
  • module:modules/filters

主要エンティティ

  • symbol:include/util_filter.h#ap_filter_t：構造体の定義およびフィルタ操作のインターフェース。
  • symbol:modules/filters/mod_include.c#ap_filter_t：フィルタインスタンスを利用する具体的なフィルタモジュールの実装例。

エンドツーエンドの流れ

提供された情報からは、フィルタチェーン全体の初期化から破棄に至る完全なエンドツーエンドの制御フローはまだ判明していません。

現時点で判明している部分的な流れは以下の通りです： 1. フィルタは、ap_pass_brigade や ap_get_brigade を介して apr_bucket_brigade データを順次処理します。 2. 各フィルタは、構造体メンバと推測される ctx ポインタを使用して自身の状態を保持します。 3. ストリーム終了マーカーに到達した際、各フィルタは内部状態をフラッシュし、リソースを解放します。

重要な不変条件

  • フィルタは ap_filter_type に基づいて、チェーン内で適切にソートされていなければなりません。
  • AP_FTYPE_CONNECTION 以上のフィルタは、サブリクエスト内では使用できません。

よくある落とし穴

  • **ブリゲードの所有権**: ap_pass_brigade に渡されるブリゲードの所有権は呼び出し元にあります。フィルタ内でデータを保持・退避させる必要がある場合は、ap_save_brigade を使用してデータを一時保存しなければなりません。
  • **不透明ポインタとしての扱い**: 本構造体は不透明（Opaque）ポインタとして扱われることが多いため、詳細なメンバへ直接アクセスする際は include/util_filter.h の完全な定義を参照する必要があります。

次に読む場所

  • include/util_filter.h：ap_filter_t の定義や関連する API。
  • modules/filters/mod_include.c：フィルタの具体的な利用例。
