ap_filter_t

概念

ap_filter_t は、Apache HTTP Serverのフィルタチェーンにおいて、インストールされた個々のフィルタインスタンスを表現する構造体です（include/util_filter.h）。 また、modules/filters/mod_include.c においては、フィルタ処理中に現在のフィルタ状態とバケットブリゲードを保持し追跡するための、ローカルなデバッグ用コンテキスト構造体のメンバとしても利用されています。

なぜ重要か

Apacheのフィルタシステムにおいて、入力・出力データをストリームとして処理するための基盤となるため重要です。 ap_filter_t は、コアの入力・出力フィルタ（ap_core_input_filter, ap_core_output_filter）や、プロトコル処理（ap_http_header_filter など）、サブリクエスト処理（ap_sub_req_output_filter）などの間で、apr_bucket_brigade を介したデータの受け渡しと順次処理を可能にします。

関与するモジュール

  • module:include
  • module:modules/filters

主要エンティティ

  • symbol:include/util_filter.h#ap_filter_t: フィルタチェーンの個別インスタンスを表現する基本構造体。
  • symbol:modules/filters/mod_include.c#ap_filter_t: mod_include.c 内でデバッグ用コンテキストとして定義されている、フィルタポインタ f とバケットブリゲード bb を保持する構造体。

エンドツーエンドの流れ

フィルタチェーン全体の詳細な順序制御やエラー発生時の挙動など、完全なエンドツーエンドのフローは提供されたソース抜粋からは明らかになっていません。

現時点で判明している部分的な流れは以下の通りです： 1. フィルタは、ap_pass_brigade や ap_get_brigade を介して apr_bucket_brigade データストリームを順次処理します。 2. フィルタは ctx ポインタを使用して状態を保持し、必要に応じて ap_save_brigade を用いてデータを一時保存します。 3. ストリーム終了マーカー到達時に、各フィルタは内部状態をフラッシュし、リソースを解放します。

重要な不変条件

  • フィルタは ap_filter_type に基づいてチェーン内でソートされる必要があります。
  • AP_FTYPE_CONNECTION 以上のフィルタは、サブリクエスト内では使用できません。
  • ap_pass_brigade に渡されるブリゲードの所有権は呼び出し元にあります。
  • mod_include.c のデバッグ用構造体において、f は有効な ap_filter_t インスタンスを指し、bb は処理対象のバケットブリゲードを指している必要があります。

よくある落とし穴

  • **所有権の誤認**: ap_pass_brigade に渡されるブリゲードの所有権は呼び出し元にあるため、フィルタ内でデータを保持する場合は ap_save_brigade を使用しなければなりません。
  • **構造体の混同**: mod_include.c 内のローカルなデバッグ用定義（debug 構造体）と、util_filter.h で定義されている本来の ap_filter_t 構造体を混同しないよう注意が必要です。
  • **NULLアクセス**: mod_include.c のデバッグ用構造体において、f または bb が NULL の状態でアクセスした場合、セグメンテーション違反が発生する可能性があります。

次に読む場所

  • include/util_filter.h: 本来の ap_filter_t の定義やフィルタのライフサイクル管理について記述されています。
  • modules/filters/mod_include.c: デバッグ用コンテキストや、mod_include におけるフィルタの実行状態を保持する include_ctx_t の実装が記述されています。
