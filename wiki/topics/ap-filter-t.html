ap_filter_t

概念

ap_filter_t は、Apache HTTP Server のフィルタチェーンにおいて、インストールされた個々のフィルタインスタンスを表現する構造体です。フィルタ間で受け渡されるデータストリーム（apr_bucket_brigade）を順次処理する役割を持ちます。

なぜ重要か

Apache のモジュールアーキテクチャにおいて、リクエストやレスポンスのデータを段階的に加工・制御するための基盤となるためです。各フィルタインスタンスは ctx ポインタを介して個別の状態を保持し、データの加工や一時保存（ap_save_brigade）を行うことができます。

エンドツーエンドの流れ

提供された情報からは、フィルタの登録から破棄に至る完全なライフサイクルや、server/util_filter.c における具体的な順序制御の全容など、詳細なエンドツーエンドのフローは完全には明らかになっていません。

現時点で判明している部分的なデータフローは以下の通りです： 1. フィルタは ap_filter_type に基づいてフィルタチェーン内でソートされます。 2. フィルタチェーンを通過する apr_bucket_brigade データが入力されます。 3. フィルタは ap_pass_brigade や ap_get_brigade を介してデータを順次処理します。 4. 処理済み、または次段へ渡される apr_bucket_brigade データを出力します。

重要な不変条件

  • フィルタは ap_filter_type に基づいてチェーン内でソートされていなければなりません。
  • AP_FTYPE_CONNECTION 以上のフィルタは、サブリクエスト内では使用できません。

よくある落とし穴

  • **ブリゲードの所有権**: ap_pass_brigade に渡されるブリゲードの所有権は呼び出し元にあるため、フィルタ内でデータを保持する場合は ap_save_brigade を使用してデータを一時保存する必要があります。
  • **リソースの解放**: ストリーム終了マーカー（EOS）到達時に、各フィルタは内部状態をフラッシュし、リソースを適切に解放する必要があります。
  • **不透明ポインタとしての扱い**: 本構造体は不透明ポインタとして扱われることが多いため、詳細なメンバへのアクセスが必要な場合は include/util_filter.h の完全な定義を参照する必要があります。
