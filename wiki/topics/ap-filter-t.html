ap_filter_t

概念

ap_filter_t は、Apache HTTP Server のフィルタチェーンにおいて、インストールされた個々のフィルタインスタンスを表現する構造体です（include/util_filter.h）。これは、リクエストやレスポンスのデータストリーム（apr_bucket_brigade）を処理・変換するためのフィルタ機構の基盤を形成します。

なぜ重要か

Apache のモジュールアーキテクチャにおいて、データの入力および出力処理はフィルタチェーンを介して行われます。ap_filter_t は、各フィルタが自身の状態（ctx ポインタなど）を保持し、ap_pass_brigade や ap_get_brigade を通じて次段のフィルタへデータを順次受け渡すためのコンテキストを提供するため、データ処理パイプラインの制御に不可欠です。

関与するモジュール

エンドツーエンドの流れ

提供された情報からは、フィルタチェーン全体の初期化から破棄に至る完全なエンドツーエンドの制御フローは未だ完全には明らかになっていません。

現時点で判明している部分的な流れは以下の通りです： 1. フィルタは ap_filter_type に基づいてチェーン内でソートされ、配置されます。 2. データ（apr_bucket_brigade）がフィルタチェーンを通過する際、各フィルタは ap_pass_brigade（出力）や ap_get_brigade（入力）を介してデータを順次処理します。 3. フィルタは必要に応じて ctx ポインタを用いて状態を保持し、データを一時保存する場合は ap_save_brigade を使用します。 4. ストリームの終了マーカー（EOS）に到達した際、各フィルタは内部状態をフラッシュし、リソースを解放します。

重要な不変条件

  • **フィルタのソート順**: フィルタは ap_filter_type に基づいてチェーン内でソートされる必要があります。
  • **サブリクエストの制限**: AP_FTYPE_CONNECTION 以上のタイプを持つフィルタは、サブリクエスト内では使用できません。
  • **ブリゲードの所有権**: ap_pass_brigade に渡されるブリゲードの所有権は呼び出し元にあります。そのため、フィルタ内でデータを保持し続ける必要がある場合は、ap_save_brigade を使用してデータを退避させなければなりません。

よくある落とし穴

  • **不透明ポインタとしての扱い**: ap_filter_t は不透明（opaque）ポインタとして扱われることが多いため、詳細な構造体メンバに直接アクセスする際は、include/util_filter.h の完全な定義を参照する必要があります。
  • **メモリ管理と所有権**: ap_pass_brigade に渡されたブリゲードのデータを、適切に退避（ap_save_brigade を使用）させずに保持し続けると、メモリの破損や予期しない挙動を引き起こす可能性があります。
  • **エラーハンドリング**: ストリーム終了時やエラー発生時における、各フィルタ内での内部状態のフラッシュやリソース解放の不備は、リソースリークや不整合の原因となります。
