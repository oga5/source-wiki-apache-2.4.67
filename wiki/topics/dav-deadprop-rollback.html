dav_deadprop_rollback

概念

dav_deadprop_rollback は、Apache HTTP Server の WebDAV モジュール（mod_dav）において、デッドプロパティ（Dead Properties：サーバーが解釈しない、クライアント定義のプロパティ）の更新処理が失敗した際に、元のプロパティ値を保持・復元（ロールバック）するためのデータ構造です。

なぜ重要か

WebDAV プロパティの更新はトランザクション的な整合性が求められます。更新処理の途中でエラーが発生した場合、この構造体に退避しておいた変更前の DBM キーと値（apr_datum_t）を使用することで、プロパティデータベース（DBM）の状態を安全に元の状態へ差し戻すことができます。

関与するモジュール

  • module:modules/dav/fs
  • module:modules/dav/main

主要エンティティ

  • symbol:modules/dav/fs/dbm.c#dav_deadprop_rollback: DBM プロパティストアにおけるロールバック用構造体の定義と利用。
  • symbol:modules/dav/main/mod_dav.h#dav_deadprop_rollback: mod_dav の共通ヘッダーにおける定義。

エンドツーエンドの流れ

1. **プロパティ更新の開始**: デッドプロパティの更新処理がトリガーされます。 2. **元の値の退避**: 更新を実行する前に、現在の DBM キーと値が apr_datum_t 型として dav_deadprop_rollback 構造体に格納されます。 3. **更新処理の実行**: 新しいプロパティ値の書き込みが試行されます。 4. **ロールバック（失敗時）**: 更新処理が失敗した場合、退避されていた dav_deadprop_rollback の key と value を用いて、DBM 上のデータが元の状態に復元されます。

*注意: 具体的な関数呼び出しの連鎖や、上位レイヤー（modules/dav/main/props.c など）からプロバイダーインターフェースを介した詳細な制御フローは、提供された情報からは完全には特定されていません。*

重要な不変条件

  • **メモリ管理の依存性**: apr_datum_t 型を使用しているため、本構造体のライフサイクルおよびメモリ管理は apr_pool_t に依存します。
  • **プロバイダー固有の閉鎖性**: 本構造体は基本的にはプロバイダー固有（provider-private）のデータ構造として扱われます。

よくある落とし穴

  • **メモリ不足**: apr_pcalloc 等による構造体自体のメモリ確保に失敗した場合、ロールバック情報を生成できず、安全な復元が行えなくなる可能性があります。
  • **DBM の破損**: ロールバック処理を実行する時点で DBM 自体が破損している場合、復元操作そのものが失敗する可能性があります。
  • **定義の二重管理**: modules/dav/fs/dbm.c と modules/dav/main/mod_dav.h の両方に定義が存在するため、構造体を変更する際はインターフェースの整合性に注意する必要があります。

次に読む場所

  • modules/dav/fs/dbm.c: DBM を用いたデッドプロパティの具体的なロールバック処理の実装。
  • modules/dav/main/mod_dav.h: mod_dav における共通定義およびプロバイダーインターフェース。
