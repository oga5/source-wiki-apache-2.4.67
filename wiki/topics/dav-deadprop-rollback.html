dav_deadprop_rollback

概念

dav_deadprop_rollback は、Apache HTTP Server の WebDAV（mod_dav）モジュールにおいて、WebDAV プロパティ（Dead Properties）の更新処理中にエラーが発生した場合や、トランザクション的なロールバックが必要になった際に、変更前のプロパティ値を保持・復元するためのデータ構造です。

なぜ重要か

WebDAV プロパティの更新は、検証（validate）、実行（exec）、確定（commit）、ロールバック（rollback）といったフェーズを経て行われます。この一連の処理において、更新が失敗した際にプロパティデータベース（propdb）の整合性を保ち、変更前の状態に安全に復元するために、この構造体が重要な役割を果たします。

関与するモジュール

  • module:modules/dav/fs (DBM を用いたファイルシステムプロバイダー実装)
  • module:modules/dav/main (WebDAV コアモジュール)

主要エンティティ

    • プロバイダー固有のプライベート構造体として定義された、不透明な（opaque）型定義です。 ``c typedef struct dav_deadprop_rollback dav_deadprop_rollback; ``

  • symbol:modules/dav/fs/dbm.c#dav_deadprop_rollback

エンドツーエンドの流れ

1. **ロールバック情報の生成**: プロパティの更新処理中に、変更前の DBM キーと値（apr_datum_t）が dav_deadprop_rollback 構造体に退避されます。 2. **エラー発生時の復元**: 更新処理が失敗するか、明示的にロールバックが要求された場合、退避されていた key と value を用いて DBM 上のデータが元の状態に復元されます。

*注意*: 提供されたソース情報からは、具体的な関数呼び出しの連鎖や、上位レイヤー（modules/dav/main/props.c など）からプロバイダーインターフェースを介して本構造体が操作される詳細なエンドツーエンドの制御フローは完全には特定されていません。

重要な不変条件

  • **プロバイダー固有の隠蔽**: 本構造体はプロバイダー固有（provider-private）のプライベートな構造体であり、mod_dav コアなどの外部モジュールがその内部構造（key や value など）を直接操作することは想定されていません。
  • **メモリ管理の依存**: apr_datum_t を使用しているため、メモリ管理は apr_pool_t に依存します。

よくある落とし穴

  • **メモリ不足**: apr_pcalloc 等によるメモリ確保に失敗した場合、ロールバック情報を生成できず、エラー発生時に元の状態へ復元できなくなる可能性があります。
  • **DBM の破損**: ロールバック実行時に DBM 自体が破損している場合、復元操作そのものが失敗し、プロパティデータベースの整合性が失われる恐れがあります。
  • **定義の二重性**: mod_dav.h での不透明な型定義と dbm.c での具体的な構造体定義の整合性を維持する必要があります。
