register_hook

概念

register_hook は、Apache HTTP Server の各モジュールにおいて、モジュールのロード時にフック関数やプロバイダーを登録するための標準的な初期化関数（エントリポイントの一部）です。各モジュールが独自に同名の static void register_hook(apr_pool_t *p) 関数を定義し、Apache のコアシステムや他のモジュール（mod_proxy など）に対して、自身のハンドラやプロバイダー構造体を登録します。

なぜ重要か

Apache HTTP Server はモジュール式のアーキテクチャを採用しており、リクエスト処理や設定読み込みなどの各フェーズで特定の処理を挿入するためにフック機構を使用します。register_hook は、モジュールがロードされた直後に呼び出され、以下のような重要な登録処理を行います。

  • **設定完了後（post_config）の初期化関数の登録**: サーバー設定完了後に必要な初期化処理をフックします。
  • **プロキシハンドラの登録**: HTTP/2 プロキシなどのスキームハンドラや URL 正規化ハンドラを登録します（例: mod_proxy_http2）。
  • **ロードバランシングアルゴリズムの登録**: PROXY_LBMETHOD プロバイダーとして、特定のバランシング手法を登録します（例: mod_lbmethod_bybusyness, mod_lbmethod_byrequests, mod_lbmethod_bytraffic）。

これにより、モジュールが提供する機能が Apache のライフサイクルやプロキシシステムに正しく統合されます。

エンドツーエンドの流れ

Apache HTTP Server の起動およびモジュールロードプロセスにおける大まかな流れは以下の通りですが、詳細なエンドツーエンドの実行フロー（Apache コアがどのようにこれらの関数を探索・呼び出すか）は提供された情報からは完全には明らかになっていません。

既知の範囲での流れ

1. **モジュールのロード**: Apache のモジュールロード機構により、各モジュールが読み込まれます。 2. **フックの登録**: 各モジュール内の register_hook が呼び出されます。

  • **mod_proxy_http2**: ap_hook_post_config で h2_proxy_post_config を登録し、proxy_hook_scheme_handler および proxy_hook_canon_handler を用いて HTTP/2 プロキシハンドラを登録します。
  • **ロードバランサーモジュール群 (bybusyness, byrequests, bytraffic)**: ap_register_provider を使用して、それぞれのバランシング構造体（bybusyness, byrequests, bytraffic）を PROXY_LBMETHOD プロバイダーとして登録します。また、ap_hook_post_config を用いて、設定完了後の初期化関数（lbmethod_*_post_config）を登録します。

    • 3. **設定完了後の初期化**: Apache の設定読み込みが完了すると、登録された post_config フック（h2_proxy_post_config や lbmethod_*_post_config など）が呼び出され、モジュールの初期化が完了します。

重要な不変条件

  • **シグネチャの一致**: すべての register_hook 関数は static void register_hook(apr_pool_t *p) という共通のシグネチャを持ち、メモリプール p を引数として受け取る必要があります。
  • **依存関係の前提**:
  • ロードバランサーモジュール（bybusyness, byrequests, bytraffic）が正しく機能するためには、mod_proxy がロードされている環境である必要があります。
  • 登録される各ハンドラやプロバイダー構造体（bybusyness 構造体など）は、登録時点で定義済みでなければなりません。
  • **優先度の設定**: mod_proxy_http2 では、他のプロキシモジュールより優先して実行されるよう、ハンドラ登録時に APR_HOOK_FIRST を指定しています。一方、ロードバランサーモジュール群の post_config フックは APR_HOOK_MIDDLE で登録されます。

※なお、mod_cgid.c における register_hook の前提条件や不変条件は、提供された情報からは未解析です。

よくある落とし穴

  • **同名関数の混同**: register_hook は各モジュール内で static 関数として個別に定義されているため、名前は同じですが実体は完全に独立しています。静的解析ツールやデバッガを使用する際、例えば mod_cgid.c の register_hook と mod_lbmethod_byrequests.c の register_hook を混同しないよう注意が必要です。
  • **登録名の競合**: ap_register_provider を使用してプロバイダーを登録する際、他のモジュールが既に同じ名称（例: "byrequests"）で登録していると、登録が競合し意図しない挙動やエラーを引き起こす可能性があります。
  • **初期化順序の依存**: post_config フック内での依存関係チェック（例: mod_proxy のコア関数取得など）があるため、フックの登録順序や優先度（APR_HOOK_MIDDLE など）を誤って変更すると、初期化に失敗する恐れがあります。
