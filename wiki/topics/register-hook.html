register_hook

概念

register_hook は、Apache HTTP Server の各モジュールが起動・初期化フェーズにおいて、自身のハンドラ関数やプロバイダーを Apache のフックシステムやプロバイダー機構に登録するためのエントリポイント（コールバック関数）です。各モジュールファイル内で同名の static void register_hook(apr_pool_t *p) として定義され、モジュールのロード時に呼び出されます。

なぜ重要か

Apache は極めてモジュール化されたアーキテクチャを採用しており、コア機能と各拡張モジュール（HTTP/2 プロキシ、各種ロードバランシングアルゴリズムなど）の結合を疎結合に保つ必要があります。register_hook は、各モジュールが提供する特定の処理（設定完了後の初期化、リクエストハンドラ、URL正規化、ロードバランシングメソッドの提供など）を Apache コアや他の親モジュール（mod_proxy など）に認識・登録させるための必須の仕組みです。

  • symbol:modules/proxy/balancers/mod_lbmethod_bytraffic.c#register_hook: ロードバランシング手法 "bytraffic" を PROXY_LBMETHOD プロバイダーとして登録し、設定後初期化フックを設定します。

エンドツーエンドの流れ

1. **モジュールのロード**: Apache HTTP Server の起動プロセスにおいて、各モジュールがロードされます。 2. **フック登録の実行**: Apache のモジュール初期化機構が各モジュールの register_hook 関数を呼び出します。この際、メモリプール（apr_pool_t *p）が渡されます。 3. **プロバイダーおよびフックの登録**:

  • **HTTP/2 プロキシ (mod_proxy_http2.c)**: ap_hook_post_config で設定後フックを登録し、proxy_hook_scheme_handler および proxy_hook_canon_handler を用いて、それぞれ proxy_http2_handler と proxy_http2_canon を優先度 APR_HOOK_FIRST で登録します。
  • **ロードバランサー (mod_lbmethod_*.c)**: ap_register_provider を使用して、それぞれのバランシング構造体（bybusyness, byrequests, bytraffic）を PROXY_LBMETHOD プロバイダーとして登録します。また、ap_hook_post_config を用いて、設定後初期化関数（lbmethod_*_post_config）を優先度 APR_HOOK_MIDDLE で登録します。

    • 4. **設定完了後の初期化**: Apache の設定読み込みが完了すると、登録された post_config フック（h2_proxy_post_config や各 lbmethod_*_post_config）が呼び出され、実際の初期化処理や依存関係の解決が行われます。

※なお、mod_cgid.c における register_hook の具体的な登録内容や、Apache コアがこれらを呼び出す全体的な制御フローの完全な詳細は提供された情報からは不明です。

重要な不変条件

  • **シグネチャの一致**: すべての register_hook は static void register_hook(apr_pool_t *p) というシグネチャを持ち、戻り値はありません。
  • **実行タイミング**: 本関数は Apache のモジュールロードプロセス中に呼び出される必要があります。
  • **前提モジュールの存在**: ロードバランサーモジュール（mod_lbmethod_*.c）の登録が正常に機能するためには、前提として mod_proxy がロードされている環境である必要があります。

よくある落とし穴

  • **同名関数の混同**: register_hook は各モジュールファイル内で static 関数として定義されている同名の別関数です。例えば、mod_cgid.c の register_hook と mod_proxy_http2.c や mod_lbmethod_*.c の register_hook は、名前は同じですが全く異なるコンテキストで動作します。静的解析ツールやコード検索の際にこれらを混同しないよう注意が必要です。
  • **優先度の競合**: mod_proxy_http2.c では APR_HOOK_FIRST を指定して他のプロキシモジュールより優先して実行されるようにしていますが、優先度の設定を誤ると意図しないハンドラの実行順序になり、リクエスト処理が失敗する原因になります。
  • **登録名の競合**: ap_register_provider で登録するプロバイダー名（"byrequests" など）が他のモジュールと重複した場合、登録の競合が発生する可能性があります。
