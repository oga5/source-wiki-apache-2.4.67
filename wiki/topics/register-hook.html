register_hook

概念

register_hook は、Apache HTTP Server の各モジュールにおいて、モジュールのロード時に自身の機能（ハンドラ、プロバイダー、初期化関数など）を Apache のフックシステムやプロバイダー機構に登録するためのエントリポイントとなる関数です。

各モジュールで同名の static void register_hook(apr_pool_t *p) として定義されていますが、これらはそれぞれ独立した静的関数であり、モジュールごとに登録する具体的な内容（HTTP/2 プロキシハンドラ、各種ロードバランシングアルゴリズムなど）は異なります。

なぜ重要か

Apache HTTP Server はモジュール式のアーキテクチャを採用しており、コアシステムや他のモジュール（mod_proxy など）と連携するためにフックやプロバイダーの登録が必要です。register_hook は、モジュールがロードされた際に最初に呼び出され、適切なタイミング（設定完了後やリクエスト処理時など）で自モジュールの関数が実行されるようにシステムへ登録を行う極めて重要な役割を担います。

関与するモジュール

提供されたコンテキストに基づき、以下のモジュールが関与します。

  • module:modules/generators （mod_cgid.c 内に同名関数が存在することが言及されています）
  • module:modules/http2
  • module:modules/proxy/balancers

主要エンティティ

各モジュールにおける register_hook の役割は以下の通りです。

  • **symbol:modules/http2/mod_proxy_http2.c#register_hook**

    • HTTP/2 プロキシ関連のハンドラ（スキームハンドラ proxy_http2_handler、URL正規化ハンドラ proxy_http2_canon）および設定完了後フック（h2_proxy_post_config）を登録します。

  • **symbol:modules/proxy/balancers/mod_lbmethod_bybusyness.c#register_hook**

    • "bybusyness" ロードバランシングプロバイダーを登録し、設定完了後の初期化フック（lbmethod_bybusyness_post_config）を構成します。

  • **symbol:modules/proxy/balancers/mod_lbmethod_byrequests.c#register_hook**

    • "byrequests" ロードバランシングプロバイダーを登録し、設定完了後の初期化フック（lbmethod_byrequests_post_config）を構成します。

  • **symbol:modules/proxy/balancers/mod_lbmethod_bytraffic.c#register_hook**

    • "bytraffic" ロードバランシングプロバイダーを登録し、設定完了後の初期化フック（lbmethod_bytraffic_post_config）を構成します。

エンドツーエンドの流れ

Apache HTTP Server の起動およびモジュールロードプロセスにおける一般的な流れは以下の通りですが、Apache コア側からの詳細な呼び出しシーケンス全体は提供されたコンテキストからは完全には明らかになっていません。

1. **モジュールのロード**: Apache の起動時に各モジュールがロードされます。 2. **register_hook の呼び出し**: Apache のフック機構により、各モジュールに定義された register_hook が呼び出されます。 3. **フック・プロバイダーの登録**:

  • **mod_proxy_http2 の場合**: ap_hook_post_config で設定完了後フックを、proxy_hook_scheme_handler および proxy_hook_canon_handler で HTTP/2 プロキシハンドラを登録します。
  • **ロードバランサーモジュール（bybusyness, byrequests, bytraffic）の場合**: ap_register_provider を用いて PROXY_LBMETHOD プロバイダーとして自身を登録し、ap_hook_post_config で設定完了後の初期化関数を登録します。

    • 4. **実行**: 登録されたフックやプロバイダーは、設定ファイルの読み込み完了後（post_config）や、実際のリクエスト処理時に Apache コアまたは mod_proxy から呼び出されます。

重要な不変条件

  • **シグネチャの一致**: すべての register_hook は static void register_hook(apr_pool_t *p) というシグネチャを持ち、戻り値を持ちません。
  • **実行タイミング**: 本関数は Apache のモジュールロードプロセス中に呼び出される必要があります。
  • **前提条件**:
  • ロードバランサーモジュール（bybusyness, byrequests, bytraffic）は、mod_proxy がロードされている環境で実行される必要があります。
  • 登録対象となる各ハンドラや構造体（例: byrequests 構造体、proxy_http2_handler など）が事前に定義済みでなければなりません。

よくある落とし穴

  • **同名関数の混同**: register_hook は複数のモジュールで同名の static 関数として定義されているため、ソースコード検索や静的解析の際に、異なるモジュールの関数（例: mod_cgid.c と mod_proxy_http2.c）を混同しないよう注意が必要です。
  • **優先度の競合**: mod_proxy_http2 では APR_HOOK_FIRST を指定して優先的に実行されるようにしている一方、ロードバランサーモジュールでは APR_HOOK_MIDDLE が指定されています。これらの優先度や登録順序を変更すると、他のモジュールとの依存関係や実行順序に予期せぬ影響を与える可能性があります。
  • **名前の競合**: ap_register_provider でプロバイダーを登録する際、他のモジュールが既に同じ名前（例: "byrequests"）で登録していると、競合が発生する可能性があります。
