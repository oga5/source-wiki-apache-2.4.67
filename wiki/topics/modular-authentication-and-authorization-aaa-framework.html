Modular Authentication and Authorization (AAA) Framework

概念

**Modular Authentication and Authorization (AAA) Framework** は、Apache HTTP Server において認証（Authentication: authn）と認可（Authorization: authz）の処理を分離し、それぞれを独立した「プロバイダ」としてプラグイン可能にするアーキテクチャです。

このフレームワークにより、Basic認証などの認証方式（プロトコルハンドラ）と、ファイルやデータベースといった実際の認証情報の格納先（バックエンドプロバイダ）が疎結合になります。また、認可処理においては、複数の認可条件を論理的に結合する柔軟なコンテナ構造を提供します。

なぜ重要か

  • **関心の分離 (Separation of Concerns)**: 認証方式（例: mod_auth_basic）は、ユーザー資格情報の検証方法（ファイル、DBM、SQLなど）の具体的な実装を知る必要がありません。
  • **柔軟なアクセス制御**: mod_authz_core が提供する論理コンテナ（<RequireAll>, <RequireAny>, <RequireNone>）により、複雑な認可ルールを直感的に組み合わせることができます。
  • **拡張性**: include/mod_auth.h で定義された共通のプロバイダインターフェースに従うことで、サードパーティ製の新しい認証・認可バックエンドを容易に追加できます。

関与するモジュール

  • module:modules/aaa: AAA機能を担うモジュール群が配置されるディレクトリ。
  • modules/aaa/mod_auth_basic.c: Basic認証プロトコルを処理し、認証プロバイダを呼び出すモジュール。
  • modules/aaa/mod_authn_core.c: 認証プロバイダの登録や共通設定を管理するコアモジュール。
  • modules/aaa/mod_authz_core.c: 認可プロバイダの登録や、Require ディレクティブによる認可ロジックの評価を管理するコアモジュール。

主要エンティティ

  • **authn_provider** (include/mod_auth.h): 認証バックエンド（例: file, dbm）が実装すべきインターフェースを定義する構造体。パスワードの検証関数（check_password）などの関数ポインタを持ちます。
  • **authz_provider** (include/mod_auth.h): 認可バックエンド（例: user, group, ip）が実装すべきインターフェースを定義する構造体。アクセス権限を検証する関数（check_authorization）などを含みます。
  • **mod_auth_basic** (modules/aaa/mod_auth_basic.c): クライアントから送信された Authorization: Basic ヘッダを解析し、適切な authn_provider を呼び出して認証を行います。
  • **mod_authn_core** (modules/aaa/mod_authn_core.c): 認証プロバイダの登録（ap_register_provider）や、ディレクトリコンテキストごとの認証設定を管理します。
  • **mod_authz_core** (modules/aaa/mod_authz_core.c): Require ディレクティブをパースし、それらを組み合わせた認可ツリーを構築・評価します。

エンドツーエンドの流れ

1. **認証フェーズ**:

  • クライアントからリクエストが届くと、mod_auth_basic などの認証ハンドラが起動します。
  • mod_auth_basic は HTTP リクエストヘッダからユーザー名とパスワードを抽出します。
  • 設定された認証プロバイダ（mod_authn_core を介して登録された authn_provider）を呼び出し、資格情報の検証を委譲します。

    • 2. **認可フェーズ**:

  • 認証が成功すると、mod_authz_core が認可フェーズを処理します。
  • mod_authz_core は、設定された Require ディレクティブのツリー（RequireAll, RequireAny など）を走査します。
  • 各 Require 行に対応する認可プロバイダ（authz_provider）を呼び出し、アクセスを許可するかどうかを判定します。

    • 3. **アクセス可否の決定**:

  • すべての認可条件が満たされればリクエストの処理を継続し、満たされない場合は HTTP_UNAUTHORIZED (401) または HTTP_FORBIDDEN (403) を返却します。

*注意: 詳細な内部関数呼び出しの順序や、エラーハンドリングの厳密な遷移など、具体的なエンドツーエンドのフローの全容は、提供されたソースコードの静的参照のみからは完全には特定されていません。*

重要な不変条件

  • **認証と認可の分離**: 認証プロバイダ（authn）はユーザーの身元確認のみを行い、そのユーザーが特定のリソースにアクセスできるかどうかの決定（認可）は認可プロバイダ（authz）に委譲されなければなりません。
  • **プロバイダ登録のライフサイクル**: すべての authn_provider および authz_provider は、サーバーの初期化フェーズにおいて、APプロバイダフレームワークを介して一意の名前で登録されていなければなりません。
  • **認可コンテナの評価規則**: mod_authz_core における <RequireAll>、<RequireAny>、<RequireNone> の論理評価は、ネストされた構造であっても一貫した論理演算（AND, OR, NOT）として評価されなければなりません。

*注意: 実行時におけるスレッド安全性や、メモリプール（pool）の割り当てに関する具体的な不変条件は、提供されたコンテキストからは完全には特定されていません。*

よくある落とし穴

  • **プロバイダ名の競合**: 異なるモジュールが同じ名前で authn_provider または authz_provider を登録しようとすると、登録が上書きされるか、予期しない動作を引き起こす可能性があります。
  • **Requireコンテナの誤設定**: <RequireAll> や <RequireAny> のネストを誤ると、意図しないアクセス許可（または拒否）が発生しやすいです。特に Require ディレクティブを単独で記述した場合のデフォルトの評価挙動を誤解することによるセキュリティホールの発生に注意が必要です。
  • **認証プロバイダの未ロード**: mod_auth_basic を有効にしているにもかかわらず、実際の認証データを処理するプロバイダ（例: mod_authn_file）がロードされていない場合、認証処理時にプロバイダが見つからずエラーとなります。

次に読む場所

  • include/mod_auth.h: 認証・認可プロバイダの構造体定義（authn_provider, authz_provider）と共通マクロ。
  • modules/aaa/mod_authn_core.c: 認証プロバイダの登録機構とコアディレクティブの実装。
  • modules/aaa/mod_authz_core.c: 認可コンテナ（Require）のパースと評価ロジックの実装。
  • modules/aaa/mod_auth_basic.c: Basic認証プロトコルのハンドリングと、認証プロバイダの呼び出し例。
