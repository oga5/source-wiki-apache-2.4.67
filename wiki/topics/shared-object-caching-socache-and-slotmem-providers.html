Shared Object Caching (socache) and Slotmem Providers

概念

Apache HTTP Serverにおける共有オブジェクトキャッシュ（socache）とスロットメモリ（slotmem）プロバイダーは、プロセス間またはスレッド間でデータを共有・管理するための抽象化レイヤーです。

  • **socache**: SSLセッションや認証情報などの小規模なオブジェクトをキャッシュするための仕組みです。バックエンドの実装（SHMCBやDBMなど）を隠蔽し、統一されたインターフェースを提供します。
  • **slotmem**: 共有メモリ領域を「スロット」と呼ばれる固定サイズのセグメントに分割して管理する仕組みです。ロードバランサーのステータス管理などで利用されます。

なぜ重要か

マルチプロセス（MPM）モデルを採用するApache HTTP Serverにおいて、プロセス境界を越えてデータを効率的かつ安全に共有することは、パフォーマンス向上（例：SSLハンドシェイクの高速化のためのセッション再利用）や、複数プロセス間での動的な状態同期に不可欠です。socacheとslotmemは、具体的な共有メモリ（apr_shm）や排他制御の実装詳細を隠蔽し、モジュール開発者に一貫したAPIを提供します。

関与するモジュール

  • module:include (共通ヘッダー定義)
  • module:modules/cache (キャッシュ実装、例: mod_socache_shmcb)
  • module:modules/slotmem (共有メモリ管理実装、例: mod_slotmem_shm)

主要エンティティ

  • **symbol:include/ap_slotmem.h#ap_slotmem_instance_t**: 共有メモリ領域を管理するための不透明な（opaque）ハンドル。直接メンバにアクセスすることはできず、ap_slotmem_provider_t のAPIを介して操作します。
  • **symbol:include/ap_socache.h#ap_socache_instance_t**: 各socacheバックエンド実装において、個別のキャッシュインスタンスの状態を保持するための不透明な型。
  • **symbol:modules/cache/mod_socache_shmcb.c#socache_shmcb_store**: 共有メモリベースの循環バッファ（SHMCB）キャッシュにデータを保存する具体的な関数。
  • **symbol:modules/slotmem/mod_slotmem_shm.c#storage**: mod_slotmem_shm における共有メモリのストレージ管理に関連するエンティティ（詳細な内部構造は提供されたコンテキストからは未詳）。

エンドツーエンドの流れ

提供されたコンテキストからは、socacheとslotmemが直接連携するエンドツーエンドの全体フローは完全には明らかになっていません。しかし、それぞれの個別フローは以下のように定義されています。

socacheのデータ保存フロー（SHMCBの場合）

1. 呼び出し側が socache_shmcb_store を呼び出します。 2. キー（id）に基づき、ハッシュマスク（SHMCB_MASK）を用いて対象のサブキャッシュを特定します。 3. 既存の同一IDエントリが存在する場合は、置換準備として削除（shmcb_subcache_remove）します。 4. 新しいデータをサブキャッシュに書き込み（shmcb_subcache_store）、統計カウンタ（stat_replaced または stat_stores）を更新します。

slotmemの操作フロー

1. 利用側モジュールが ap_slotmem_provider_t の create または attach を呼び出し、ap_slotmem_instance_t を取得します。 2. 取得したインスタンスを介して、get/put による安全なデータ転送や、dptr によるメモリへの直接参照を行います。

重要な不変条件

  • **不透明性の維持**: ap_slotmem_instance_t および ap_socache_instance_t は不透明な構造体であり、利用側モジュールは直接メンバにアクセスせず、必ず提供されたAPI（プロバイダーの関数ポインタ）を経由しなければなりません。
  • **socacheの初期化**: ap_socache_instance_t は、create メソッドで生成された後、init メソッドが呼び出されるまでキャッシュ操作に使用してはなりません。
  • **SHMCBのID長制限**: socache_shmcb_store において、キーとなる idlen は必ず4以上でなければなりません（4未満の場合は APR_EINVAL となります）。
  • **排他制御の責任**: socache_shmcb_store 自体は排他制御（スレッドセーフティ）を提供しません（AP_SOCACHE_FLAG_NOTMPSAFE）。呼び出し側で適切なロック管理を行う必要があります。

よくある落とし穴

  • **スレッドセーフティの欠如**: mod_socache_shmcb などの一部のプロバイダー実装はスレッドセーフではないため、呼び出し側で適切なロック（グローバルミューテックスなど）を確保せずに呼び出すと、データの破損や競合状態を引き起こす可能性があります。
  • **初期化前のアクセス**: ap_socache_instance_t や ap_slotmem_instance_t が適切に初期化（init や create/attach の成功）される前にAPIに渡すと、未定義動作やエラー（APR_EGENERALなど）の原因となります。
  • **サイズ計算のオーバーフロー**: socache_shmcb_store などの内部処理では unsigned int を使用したサイズ計算が行われているため、極端に大きなキャッシュサイズを設定した場合にオーバーフローのリスクがあります。
