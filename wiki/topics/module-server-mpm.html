Module: server/mpm

概念

server/mpm（Multi-Processing Module）は、Apache HTTP Serverにおけるリクエスト処理の並行実行モデル（プロセスやスレッドの管理、ネットワーク接続の受け入れなど）を抽象化・実装するモジュール群です。オペレーティングシステム（OS）の特性やシステムの要求仕様に応じて、最適なマルチプロセッシングアーキテクチャを選択できるように設計されています。

提供されている主なMPM実装には以下があります：

  • **event** (server/mpm/event): キープアライブ接続などの非同期イベント駆動処理に特化したモデル。
  • **prefork** (server/mpm/prefork): スレッドを使用せず、リクエストごとに独立した子プロセスを事前に生成して処理する安全性の高いモデル。
  • **worker** (server/mpm/worker): マルチプロセスとマルチスレッドを組み合わせたハイブリッドモデル。
  • **winnt** (server/mpm/winnt): Windows NT系OS向けに最適化された、サービス管理やイベントログ連携を含むモデル。
  • **mpmt_os2** (server/mpm/mpmt_os2): OS/2向けのマルチプロセス・マルチスレッドモデル。
  • **netware** (server/mpm/netware): NetWare向けに最適化されたモデル。

なぜ重要か

Webサーバーのパフォーマンス、スケーラビリティ、リソース消費（メモリおよびCPU）は、採用する並行処理モデルに直接依存します。 また、Windowsのサービス登録（service.c）やイベントログ（nt_eventlog.c）のように、各OS固有のシステムAPIやライフサイクル管理に適合した実装を提供することで、クロスプラットフォーム環境における高い安定性と親和性を実現しています。

エンドツーエンドの流れ

各MPMにおける親プロセスの起動から、子プロセス・スレッドの生成、ネットワークソケットのリスン、リクエストのディスパッチ、およびシャットダウンに至る具体的なエンドツーエンドの制御フローは、提供されたファイル構成情報からは詳細が完全には特定されていません。

重要な不変条件

各MPMが維持すべきスレッドセーフティ、プロセス間通信（IPC）の整合性、シグナルハンドリング、またはリソース制限（最大同時接続数やプロセス数など）に関する具体的な不変条件は、提供されたファイル構成情報からは完全には特定されていません。

よくある落とし穴

  • **プラットフォームの不一致**: winnt や mpmt_os2、netware などのプラットフォーム専用MPMは、対象外のOS環境では動作しません。
  • **デフォルト設定の不適合**: 各MPMの mpm_default.h に定義されている初期値が、実際のサーバーのハードウェアスペックやトラフィックパターンに対して最適化されていない場合、リソースの枯渇やパフォーマンスの低下を招く可能性があります。
  • **モジュール互換性の問題**: スレッドを使用するMPM（worker や event）を使用する場合、ロードするサードパーティ製モジュール（例: PHPなど）がスレッドセーフである必要があります。スレッドセーフでないモジュールを使用する場合は prefork を選択する必要があります。
