Managed Domains (mod_md) and ACME Integration

概念

mod_md（Managed Domains）は、Apache HTTP Server において SSL/TLS 証明書の管理を自動化するためのサブシステムです。ACME（Automated Certificate Management Environment）プロトコルを統合することで、外部の認証局（CA）と連携し、ドメインごとの証明書の取得、更新、およびローカルストレージへの安全な保存を自動的に行います。

なぜ重要か

手動での SSL/TLS 証明書の更新作業は、期限切れによるサービス停止や設定ミスのリスクを伴います。mod_md と ACME の統合により、以下の重要性がもたらされます。

  • **運用の自動化**: 証明書のライフサイクル（取得・更新・適用）を完全に自動化し、管理コストを削減します。
  • **安全なストレージ管理**: 秘密鍵や証明書、アカウント情報を抽象化されたストレージ層を介して一貫した方法で永続化します。
  • **ポリシーの統合**: ドメインごとの更新ウィンドウや再試行ポリシーをレジストリを介して一元管理します。

関与するモジュール

  • module:modules/md

主要エンティティ

本統合において、以下のシンボルが中核的な役割を果たします。

  • **symbol:modules/md/md_acme.c#md_acme_init**: ACME モジュールの初期化関数。製品名（base_product）の設定や、必要に応じた暗号化ライブラリ（md_crypt）の初期化を担当します。
  • **symbol:modules/md/md_reg.c#md_reg_create**: 管理対象ドメイン（MD）のレジストリインスタンス（md_reg_t）を生成します。永続化層である md_store_t や再試行ポリシー、更新ウィンドウなどをバインドし、ACME プロトコルハンドラを登録します。
  • **symbol:modules/md/md_store.c#md_store_load**: ストレージ層から特定のドメイン関連データ（証明書、秘密鍵、JSON設定、アカウント情報など）を読み込むための抽象化された汎用インターフェースです。

エンドツーエンドの流れ

提供されたソースコードの範囲からは、ACME サーバーとの通信、チャレンジの検証、証明書の最終的なインストールに至る完全なエンドツーエンドのプロトコルフローの全容は完全には明らかになっていません。

現時点で判明している初期化およびセットアップ段階の流れは以下の通りです。

1. **モジュールの初期化**: modules/md/mod_md.c から md_acme_init が呼び出され、製品識別子が設定され、暗号化サブシステムが初期化されます。 2. **レジストリの生成**: modules/md/mod_md.c から md_reg_create が呼び出され、md_reg_t が初期化されます。この過程で、利用可能な ACME プロトコルハンドラが登録され、既存のプロパティがストレージからロードされます。 3. **データのロード**: 証明書の更新やチャレンジの検証の際、md_store_load を介して、アカウント鍵（md_acme_acct.c）、チャレンジデータ（md_acme_authz.c）、または設定 JSON（md_reg.c）がストレージからメモリ上にロードされます。

重要な不変条件

  • **ストレージの有効性**: md_reg_create を呼び出す際、渡される store（md_store_t）は事前に適切に初期化されていなければなりません。また、初期化時の既存プロパティのロード（load_props）が失敗した場合、レジストリの生成自体が失敗します。
  • **メモリプールの生存期間**: md_store_load によってロードされたデータの生存期間は、引数として渡された apr_pool_t の生存期間に拘束されます。
  • **暗号化の初期化**: md_acme_init で init_ssl が真に設定されている場合、内部の md_crypt_init が正常に完了（APR_SUCCESS を返却）しなければ、ACME モジュールの初期化は成功しません。

よくある落とし穴

  • **ストレージ関数の未初期化**: md_store_load は実際の I/O 処理を store->load 関数ポインタに委譲するラッパーです。store やその関数ポインタが NULL の状態で呼び出すと、セグメンテーションフォールトを引き起こします。
  • **メモリの早期解放**: md_store_load でロードした証明書や鍵データを保持するメモリプール（apr_pool_t）を誤って早期にクリアまたは破棄すると、データへのアクセス時に未定義動作が発生します。
  • **CAファイルパスの特殊値**: md_reg_create において、ca_file に "none" という文字列が指定された場合、内部的には NULL として扱われます。これにより、証明書検証の挙動が意図せず変化する可能性があります。

次に読む場所

  • modules/md/md_acme.c: ACME 統合の初期化およびプロトコルハンドラの管理。
  • modules/md/md_reg.c: 管理対象ドメインのレジストリ（md_reg_t）の構築とポリシー管理。
  • modules/md/md_store.c: 証明書やアカウント情報の永続化を担うストレージ抽象化レイヤーの実装。
  • modules/md/mod_md.c: mod_md のメインモジュールコードであり、初期化フローの起点。
