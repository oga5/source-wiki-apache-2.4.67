read_chr

概念

read_chr は、HTTP/2 や MD (Managed Domains) モジュールにおいて、Link ヘッダーなどの文字列を構文解析（パース）する際に使用されるヘルパー関数です。解析対象の文字列を保持するコンテキスト（link_ctx）の現在位置にある文字が、指定された文字と一致するかどうかを判定し、一致した場合は解析位置を1文字分進める役割を持ちます。

なぜ重要か

HTTP/2 のサーバープッシュ機能（h2_push）やプロキシユーティリティ（h2_proxy_util）、および MD モジュール（md_util）では、HTTP ヘッダーのパース処理が必要となります。read_chr は、バッファオーバーランなどの境界外アクセスを防ぎつつ、特定の区切り文字や記号を安全かつ簡潔に消費（スキップ）するための共通的なユーティリティとして機能し、パーサーの実装をシンプルに保つために重要です。

エンドツーエンドの流れ

現時点では、read_chr を呼び出す上位のパース処理全体（例えば Link ヘッダー全体のパースからプッシュリクエストの生成や MD の処理に至るまで）の具体的なエンドツーエンドの制御フローの詳細は完全には判明していません。

判明している局所的な処理の流れは以下の通りです： 1. 呼び出し元が link_ctx 構造体に解析対象の文字列、その長さ、および初期インデックスを設定する。 2. パーサーが特定の文字（区切り文字など）を期待する箇所で read_chr(ctx, 'x') を呼び出す。 3. read_chr 内部で、現在のインデックス ctx->i が文字列長 ctx->slen 未満であり、かつ現在位置の文字 ctx->s[ctx->i] が指定された文字 c と一致するかを比較する。 4. 一致する場合、インデックス ctx->i をインクリメントして 1 を返す。一致しない場合はインデックスを変更せずに 0 を返す。 5. 呼び出し元は、戻り値（1 または 0）に基づいてパース処理を分岐または継続する。

重要な不変条件

  • **境界外アクセスの防止**: ctx->i が ctx->slen 未満である場合のみ文字の比較が行われ、インデックスが文字列の長さを超えて前進することはありません。
  • **状態の非破壊性（不一致時）**: 指定された文字と一致しない場合、または終端に達している場合は、インデックス ctx->i は変更されず 0 が返されます。

よくある落とし穴

  • read_chr は一致した場合に自動的にインデックスを前進させるため、文字の存在確認だけを行いたい（インデックスを進めたくない）場合に誤って使用すると、パース位置が意図せずずれる原因になります。
  • 空白文字のスキップなど、複数文字や任意の文字種を読み飛ばす場合は、read_chr ではなく skip_ws などの他の専用ユーティリティと適切に組み合わせる必要があります。
