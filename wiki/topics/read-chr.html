read_chr

概念

read_chr は、HTTP/2 や Managed Domains（mod_md）などのモジュールにおいて、Link ヘッダーなどの文字列を構文解析（パース）する際に使用されるヘルパー関数です。解析対象の文字列を保持するコンテキスト（link_ctx）の現在位置にある文字が、指定された文字と一致するかどうかを判定し、一致した場合は解析位置を1文字進める役割を持ちます。

なぜ重要か

HTTP/2 のサーバープッシュ（h2_push.c、h2_proxy_util.c）や、Managed Domains（md_util.c）の処理において、特定のフォーマットを持つ文字列を安全かつ効率的にパースするために重要です。境界外アクセスを防ぎつつ、特定の区切り文字や記号を消費する共通のパターンを提供します。

エンドツーエンドの流れ

1. **解析の開始**: link_ctx 構造体に解析対象の文字列 s、その長さ slen、初期インデックス i = 0 を設定してパース処理を開始します。 2. **文字の判定と消費**: パース処理の過程で、特定の文字（区切り文字など）を期待する箇所で read_chr(ctx, c) を呼び出します。 3. **インデックスの更新**:

  • 現在のインデックス ctx->i が ctx->slen 未満であり、かつ ctx->s[ctx->i] が指定された文字 c と一致する場合、ctx->i をインクリメントして 1 を返します。
  • 一致しない場合、または終端に達している場合は、インデックスを更新せずに 0 を返します。

    • 4. **後続の処理**: 呼び出し元は戻り値（1 または 0）に応じて、パースを継続するか、エラー処理や分岐を行います。

※なお、各モジュールにおける具体的な呼び出しシーケンス全体や、パースされた結果がどのようにシステム全体に伝播するかの詳細なエンドツーエンドの流れは、提供された情報からは完全には明らかになっていません。

重要な不変条件

  • **境界外アクセスの防止**: ctx->i が ctx->slen 以上の場合、配列アクセスを行わずに即座に 0 を返します。これにより、バッファオーバーランを防ぐ不変条件が維持されます。
  • **状態の非破壊性（不一致時）**: 文字が一致しない場合、インデックス ctx->i は変更されず、解析状態は呼び出し前の位置に維持されます。

よくある落とし穴

  • **インデックスの自動前進**: read_chr は一致した場合に**自動的にインデックスを前進（消費）させる**ため、文字の存在確認（Lookahead）のみを行いたい場合に誤って呼び出すと、解析位置が意図せず進んでしまう原因になります。
  • **コンテキストの初期化**: link_ctx の slen や i が正しく初期化されていない場合、未定義動作や意図しないパースエラーを引き起こす可能性があります。
