read_chr

概念

read_chr は、HTTPの Link ヘッダーなどの文字列を構文解析（パース）する際に使用されるヘルパー関数です。解析状態を管理するコンテキスト構造体（link_ctx）を受け取り、現在の解析位置にある文字が指定された文字と一致するかどうかを判定します。一致した場合は、解析位置を1文字分進める役割を持ちます。

なぜ重要か

この関数は、HTTP/2のサーバープッシュ機能（h2_push.c や h2_proxy_util.c）や、ACME/Managed Domains（md_util.c）における Link ヘッダーのパース処理において、特定の区切り文字や記号（カンマ、セミコロン、イコールなど）を安全かつ簡潔に消費するための共通パターンとして機能します。各ファイルに static 関数として個別に定義されていますが、同様の構文解析ロジックを支える基礎的な部品となっています。

関与するモジュール

主要エンティティ

  • **symbol:modules/http2/h2_push.c#read_chr**: link_ctx を用いて1文字の比較とインデックスの前進を行う実体。
  • **symbol:modules/http2/h2_proxy_util.c#read_chr**: h2_proxy_util.c 内で定義されている同様の static 関数。
  • **symbol:modules/md/md_util.c#read_chr**: md_util.c 内で定義されている同様の static 関数。
  • **link_ctx 構造体**: 解析対象の文字列（s）、その長さ（slen）、および現在の解析インデックス（i）を保持するデータ構造。

エンドツーエンドの流れ

Link ヘッダーなどの解析処理（read_link や read_param など）が開始されると、以下の流れで read_chr が呼び出されます。

1. 解析処理の過程で、特定の文字（例: ; や =）が出現することを期待する箇所に到達します。 2. read_chr(ctx, ';') のように呼び出します。 3. read_chr 内部で、現在のインデックス ctx->i が文字列の長さ ctx->slen 未満であり、かつ ctx->s[ctx->i] が指定された文字と一致するかを判定します。 4. 一致する場合、ctx->i をインクリメントして 1 を返します。一致しない場合は 0 を返します。 5. 呼び出し元は戻り値（1 または 0）に応じて、パースを継続するか、あるいはパースエラーとして処理するかを決定します。

なお、呼び出し元である read_link や read_param などの全体的なエンドツーエンドの制御フローや、パース完了後の詳細な処理フローについては、現時点の解析範囲では完全には明らかになっていません。

重要な不変条件

  • **境界外アクセスの防止**: ctx->i が ctx->slen 未満である場合のみ文字の比較が行われ、インデックスが文字列の長さを超えて前進することはありません。
  • **状態の非破壊性（不一致時）**: 指定された文字と一致しない場合、ctx->i は変更されず、解析状態は呼び出し前のまま維持されます。

よくある落とし穴

  • **static 定義による重複**: read_chr は h2_push.c、h2_proxy_util.c、md_util.c のそれぞれに static 関数として個別に定義されています。これらは名前とシグネチャ、役割が同一ですが、モジュール間で共有された共通ライブラリ関数ではなく、各ファイル内に閉じられた実装です。そのため、一部のファイルを修正する際に、他のファイルの実装と乖離が生じる可能性があります。
  • **空白文字の扱い**: read_chr は厳密に指定された1文字のみを比較するため、前後に空白文字（スペースやタブ）が存在する可能性がある場合は、事前に skip_ws などの別のヘルパー関数を呼び出して空白を消費しておく必要があります。
