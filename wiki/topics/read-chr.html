read_chr

概念

read_chr は、HTTP/2 や ACME (Managed Domains) などのモジュールにおいて、特定のヘッダー（例: Link ヘッダー）や文字列を解析する際に使用される、文字消費型のユーティリティ関数です。解析対象のコンテキストにおいて、現在のインデックスが指す文字が期待する文字と一致する場合に、インデックスを1つ進めて真（1）を返します。

なぜ重要か

この関数は、境界外アクセスを防ぎながら、文字列パーサーが特定の区切り文字や記号（カンマ、セミコロン、等号など）を安全かつ簡潔に消費できるようにします。これにより、複雑なヘッダー解析コードの可読性と堅牢性が向上します。

エンドツーエンドの流れ

各モジュール（http2 や md）における具体的なエンドツーエンドの解析フロー（どのタイミングでどのヘッダーが渡され、どのように全体が処理されるか）の詳細は、提供された情報からはまだ完全には明らかになっていません。

ただし、個別関数レベルの動作フローは以下の通りです： 1. 解析状態を保持する link_ctx 構造体（文字列 s、長さ slen、現在のインデックス i）と、比較対象の文字 c を受け取ります。 2. 現在のインデックス ctx->i が文字列の長さ ctx->slen 未満であることを確認します。 3. 現在位置の文字 ctx->s[ctx->i] が c と一致する場合、ctx->i をインクリメントして 1 を返します。 4. 一致しない場合、または終端に達している場合は、状態を変更せずに 0 を返します。

重要な不変条件

  • **境界外アクセスの防止**: ctx->i が ctx->slen 以上の場合、文字の比較は行われず、必ず 0 が返されます。これにより、バッファオーバーランが発生しないことが保証されます。
  • **状態の非破壊性（不一致時）**: 文字が一致しない場合、インデックス ctx->i は変更されません。

よくある落とし穴

  • read_chr は文字が一致したときのみインデックスを進めるため、一致しなかった場合に呼び出し側で適切にエラーハンドリングや別の解析ロジック（例: skip_ws などの他のユーティリティの適用）を行わないと、解析がスタックする可能性があります。
  • 複数モジュールに同名の read_chr 関数が存在しますが、これらは各ファイル内で static 関数として定義されており、それぞれのコンテキスト（link_ctx などの構造体定義）に依存しています。
