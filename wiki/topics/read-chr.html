read_chr

概念

read_chr は、Apache HTTP Server（httpd-2.4.67）の複数モジュールで定義・利用されている、文字列解析用のユーティリティ関数です。 解析対象のコンテキスト（link_ctx 構造体）において、現在のインデックス位置にある文字が指定された文字と一致するかを判定し、一致した場合は解析位置を1文字分前進（消費）させる役割を持ちます。

なぜ重要か

HTTP/2の Link ヘッダーなどの構文解析において、特定の区切り文字や記号（カンマ、セミコロン、等号など）を安全かつ効率的に消費するために重要です。 境界外アクセスを防ぎながら、文字列パーサーのコードを簡潔に保つための共通パターンとして機能しています。

エンドツーエンドの流れ

各モジュール（h2_proxy_util.c、h2_push.c、md_util.c）における具体的なエンドツーエンドの呼び出しフローや、これらがどのように連携して動作するかについての全体像は、現時点では完全には判明していません。

ただし、個別の解析処理においては以下の流れで動作します： 1. link_ctx 構造体に解析対象の文字列、その長さ、現在のインデックスをセットして初期化します。 2. read_chr(ctx, 'x') を呼び出し、現在の文字が期待する文字（例: '=' や ';'）であるか判定します。 3. 一致した場合はインデックスが前進し、次の文字の解析へ進みます。

重要な不変条件

  • **境界外アクセスの防止**: ctx->i（現在のインデックス）が ctx->slen（文字列の長さ）未満である場合のみ文字の比較が行われます。インデックスが文字列長に達している場合は、安全に 0 を返します。
  • **状態の非破壊性（不一致時）**: 指定された文字と一致しない場合、ctx->i はインクリメントされず、解析位置は変更されません。

よくある落とし穴

  • read_chr は一致した場合にのみ解析位置を前進させるため、一致しなかった場合に適切にエラーハンドリングや別の解析ロジック（skip_ws など）へのフォールバックを行わないと、無限ループや解析漏れが発生する可能性があります。
  • 各ファイル（h2_proxy_util.c、h2_push.c、md_util.c）で個別に static 関数として定義されているため、ファイル間でシグネチャや link_ctx の定義に微妙な差異がある可能性に注意が必要です。
