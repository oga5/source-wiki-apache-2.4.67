read_chr

概念

read_chr は、HTTP/2 や ACME (mod_md) などのモジュールにおいて、文字列（例えば Link ヘッダーなど）の構文解析を行うための軽量なユーティリティ関数です。解析対象のコンテキスト内で、現在のインデックス位置にある文字が指定された文字と一致するかを判定し、一致した場合は解析位置を1文字分進める役割を持ちます。

なぜ重要か

HTTP/2 のサーバープッシュ（h2_push.c）やプロキシ（h2_proxy_util.c）、および管理ドメイン（md_util.c）の処理において、ヘッダー値や特定の文字列フォーマットを安全かつ効率的にパースするために重要です。境界外アクセスを防ぎながら、特定の区切り文字や記号を確実に消費する役割を担っています。

エンドツーエンドの流れ

各モジュールにおける具体的なエンドツーエンドの呼び出しフロー（どのタイミングでどのヘッダーをパースし、どのように read_chr が呼び出されるかの全体像）は、提供された情報からは完全には明らかになっていません。

ただし、h2_push.c における単一の文字消費の流れは以下の通りです： 1. 解析処理の中で、特定の文字（カンマやセミコロンなど）を期待する箇所で read_chr(ctx, c) が呼び出される。 2. ctx->i が ctx->slen 未満であることを確認する。 3. ctx->s[ctx->i] が指定された文字 c と一致する場合、ctx->i をインクリメントして 1 を返す。 4. 一致しない、または終端に達している場合は、インデックスを進めずに 0 を返す。

重要な不変条件

  • **境界外アクセスの防止**: ctx->i が ctx->slen 以上の場合、文字の比較は行われず、常に 0 が返されます。これにより、バッファオーバーランが防止されます。
  • **状態の非破壊性（不一致時）**: 指定された文字 c と現在の文字が一致しない場合、インデックス ctx->i は変更されません。

よくある落とし穴

  • read_chr は文字が一致したときに**自動的にインデックスを進める（消費する）**ため、文字の存在確認だけを行いたい（先読みしたい）場合に誤って呼び出すと、解析位置が意図せず進んでしまう原因になります。
  • 各ファイル（h2_push.c、h2_proxy_util.c、md_util.c）に同名の read_chr が定義されているため、モジュール間でコードが重複しているか、あるいはファイルローカルな static 関数として独立して定義されている点に注意が必要です。
