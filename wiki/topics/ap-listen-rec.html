ap_listen_rec

概念

ap_listen_rec は、Apache HTTP Server においてサーバーが接続を待機する各リスナーソケットの情報を保持・管理するための構造体（struct）です。この構造体は next ポインタを保持しており、グローバルなリンクリスト（ap_listeners）を構成して複数のリスナーソケットを管理します。

なぜ重要か

この構造体は、設定ファイル（Listen ディレクティブ）に基づいて生成されたリスニングソケットの情報を一元管理し、MPM（Multi-Processing Module）が接続を受け入れる（accept）際の参照先として機能します。また、プロトコルごとの接続処理を行うための関数ポインタ（accept_func）を保持しており、柔軟なプロトコルハンドリングを実現するために不可欠な役割を担っています。

関与するモジュール

  • module:include
  • module:server

主要エンティティ

  • symbol:include/ap_listen.h#ap_listen_rec：構造体の定義（include/ap_listen.h）
  • symbol:server/listen.c#ap_listen_rec：リスナー管理関数群の実装（server/listen.c）

エンドツーエンドの流れ

1. **初期化**: 設定ファイルの解析に基づき、ap_setup_listeners() によってソケットの生成、バインド（bind）、およびリスニング（listen）が行われ、ap_listen_rec 構造体が生成されます。 2. **実行**: MPM はグローバルな ap_listeners リンクリストから各 ap_listen_rec を参照し、設定された accept_func（接続受け入れ関数ポインタ）を介してクライアントからの接続を受け入れます。 3. **複製**: マルチコア環境などの要件に応じて、ap_duplicate_listeners() によりリスナーが複製される場合があります（SO_REUSEPORT のサポート状況に依存します）。 4. **終了**: サーバーの終了時または再起動時に、ap_close_listeners() または ap_close_listeners_ex() が呼び出され、保持されているソケットがクローズされます。

重要な不変条件

  • すべての ap_listen_rec インスタンスは、グローバル変数 ap_listeners を起点とするリンクリストの一部として正しく管理されている必要があります。
  • リスト内の各インスタンスは、socket(), bind(), listen() の一連の処理を経て、有効なリスニング状態にある必要があります。
  • メモリおよびソケットのライフサイクルは、関連付けられた apr_pool_t（メモリプール）によって管理されます。

よくある落とし穴

  • ソケットのバインド失敗やリスニング開始の失敗は、サーバー全体の起動停止（致命的エラー）に直結します。
  • SO_REUSEPORT のサポート状況によって ap_duplicate_listeners() の挙動が変化するため、プラットフォームごとの差異に注意が必要です。
  • 設定解析時に使用される ap_set_... 系の関数は内部的なものであり、外部モジュールから直接呼び出してはなりません。

次に読む場所

  • include/ap_listen.h：ap_listen_rec 構造体および関連する関数・型の定義。
  • server/listen.c：リスナーのセットアップ、複製、クローズなどの具体的な管理ロジックの実装。
