Module: test/modules/http2

概念

test/modules/http2 は、Apache HTTP Server（httpd）におけるHTTP/2プロトコル実装（mod_http2 および関連するプロキシモジュールなど）の機能、パフォーマンス、および仕様準拠性を検証するための統合テストスイートです。

Pythonのテストフレームワーク（pytest）を利用し、実際にテスト用の httpd サーバーを起動して、クライアント（curl やカスタムスクリプト）からリクエストを送信することで、エンドツーエンドの動作を検証します。

なぜ重要か

HTTP/2は、ストリームの多重化、ヘッダー圧縮、サーバープッシュ、接続の再利用など、複雑な状態遷移とフレーム処理を伴います。このテストスイートは、以下の点を保証するために極めて重要です。

1. **仕様準拠性の維持**: RFC 9113などの標準仕様への準拠（test_203_rfc9113.py）。 2. **堅牢性の確保**: 無効なヘッダー（test_200_header_invalid.py）やタイムアウト（test_105_timeout.py）などの異常系シナリオにおける安全なエラーハンドリング。 3. **機能のデグレ防止**: プロキシ（test_500_proxy.py）、WebSockets（test_800_websockets.py）、Early Hints（test_401_early_hints.py）など、多岐にわたる機能の継続的な検証。

関与するモジュール

提供された構成において、以下のモジュールが関与しています。

  • module:test/modules/http2: テストスクリプト本体と環境設定。
  • module:test/modules/http2/htdocs: テスト用の静的ファイルやCGIスクリプト。
  • module:test/modules/http2/mod_h2test: テスト専用のApacheカスタムモジュール。

主要エンティティ

  • **テスト環境・設定**:
  • conftest.py: pytestの設定、フィクスチャの定義。
  • env.py: テスト実行環境（httpdのパス、ポート、SSL設定など）の構築。
  • **テストケース (test_*.py)**:
  • test_001_httpd_alive.py 〜 test_800_websockets.py: 基本的なGET/POSTから、SSL再交渉、ウィンドウサイズ、プロキシ、負荷テスト、WebSocketまで、個別の機能を検証するテスト群。
  • **テスト用CGIスクリプト (test/modules/http2/htdocs/cgi/)**:
  • echo.py, upload.py, hecho.py など: リクエストデータやヘッダーをエコーバック、またはアップロード処理をシミュレートするPythonスクリプト。
  • **テスト用ヘルパーモジュール (test/modules/http2/mod_h2test/)**:
  • mod_h2test.c, mod_h2test.h: 標準のhttpdモジュールだけではテストが難しい、HTTP/2固有の低レベルな挙動やフレーム処理をシミュレート・検証するためのカスタムCモジュール。
  • **WebSocketサーバー**:
  • ws_server.py: WebSocketテスト（test_800_websockets.py）用のバックエンドサーバー。

エンドツーエンドの流れ

1. **初期化**: pytestが起動し、conftest.py および env.py を介してテスト環境がセットアップされます。必要に応じて、テスト用のApache設定ファイルが生成され、テスト用モジュール（mod_h2test）がビルド・ロードされます。 2. **サーバー起動**: テスト対象の httpd インスタンスがバックグラウンドで起動します。 3. **テスト実行**: 各 test_*.py が順次実行されます。テストコードは、curl コマンドやPythonのHTTPクライアントを使用して、起動したhttpdサーバーに対してHTTP/2リクエストを送信します。 4. **検証**: サーバーからのレスポンス（ステータスコード、ヘッダー、ボディ、フレーム構造など）をアサートします。CGIや mod_h2test を経由するリクエストでは、バックエンドの挙動も合わせて検証されます。 5. **クリーンアップ**: テスト終了後、httpdサーバーが停止され、一時ファイルがクリーンアップされます。

*※注意: テスト実行時の詳細なプロセス間通信や、特定のテストケースにおける厳密なパケットレベルのフローは、提供されたファイル一覧からは完全には特定されていません。*

重要な不変条件

  • **プロトコルの厳密性**: RFC 9113などの仕様に基づき、無効なヘッダーや不正なフレームシーケンスに対しては、プロトコルエラー（GOAWAYやRST_STREAM）が適切に返されなければなりません（test_200_header_invalid.py, test_203_rfc9113.py）。
  • **テストの独立性**: 各テストケースは、httpdサーバーの状態（接続の再利用やシャットダウンなど）を検証しつつも、他のテストケースの実行を妨げないように制御される必要があります。

*※注意: テストスイート全体で維持されるべき内部状態や、スレッド安全性に関する詳細な不変条件は、提供されたソースコードのメタデータからは完全には特定されていません。*

よくある落とし穴

  • **外部依存関係の欠如**: テストの実行には、curl（HTTP/2サポート付き）や、特定のPythonライブラリ、およびテスト対象の httpd（mod_http2 が有効化されていること）が必要です。これらが環境に存在しない場合、テストがスキップされるか失敗します。
  • **ポート競合**: テスト用httpdが使用するポート（env.py で定義）が既に他のプロセスで使用されている場合、サーバーの起動に失敗します。
  • **CGI実行環境の差異**: htdocs/cgi/ 配下のPythonスクリプトが、テスト実行環境のPythonインタープリタのパスやパーミッションの違いにより実行できない場合、CGI関連のテスト（test_711_load_post_cgi.py など）が失敗します。

次に読む場所

  • test/modules/http2/env.py: テスト環境の構築方法と設定パラメータを理解する。
  • test/modules/http2/conftest.py: pytestのフィクスチャとテスト全体のライフサイクル管理を把握する。
  • test/modules/http2/test_002_curl_basics.py: 最も基本的なHTTP/2テストケースの実装方法を確認する。
  • test/modules/http2/mod_h2test/mod_h2test.c: テスト用カスタムモジュールがどのようにhttpdの挙動を拡張しているかを調べる。
