h2_stream

概念

h2_stream は、HTTP/2 プロトコルにおける個別のリクエストとレスポンスのペアを管理するためのコンテキスト構造体です。Apache HTTP Server（httpd）の mod_http2 モジュール内、およびテスト用クライアントツール（h2ws.c）の双方において、単一の HTTP/2 セッション内で多重化される各ストリームのライフサイクル、状態遷移、およびデータ転送を制御するために定義されています。

なぜ重要か

HTTP/2 は単一の TCP 接続上で複数のリクエスト/レスポンスを多重化して処理します。h2_stream は、この多重化された各通信チャネルの独立した状態を追跡し、メイン接続（c1）とワーカースレッド（c2）の間で安全かつ効率的にデータを転送するための基盤となるため極めて重要です。また、テストクライアント側でも WebSocket（CONNECT メソッド）などのストリーム状態をシミュレート・検証するために不可欠な役割を果たします。

エンドツーエンドの流れ

サーバーサイドにおける h2_stream のライフサイクルおよびデータ転送の流れは以下の通りです。

1. **生成**: h2_session に紐付けられ、h2_stream_create() によって初期化されます。 2. **リクエスト受信と蓄積**: h2_stream_add_header() や h2_stream_recv_DATA() を介してフレームデータが蓄積されます。 3. **処理の準備と実行**: h2_stream_prepare_processing により準備が整うと、ワーカースレッド（c2）がリクエストを処理します。 4. **データ転送**: メイン接続（c1）とワーカースレッド（c2）の間で、h2_bucket_beam を介してレスポンスデータがスレッド安全に転送されます。 5. **レスポンス送出**: h2_stream_read_to() を介して処理済みのレスポンスデータが送出されます。 6. **クリーンアップと破棄**: 処理完了後、h2_mplx_c1_stream_cleanup を経て破棄キューへ移動し、最終的に h2_stream_destroy() によって関連するメモリプールと共に破棄されます。

テストクライアント（h2ws.c）側では、ws_stream_submit() を通じて CONNECT メソッドを用いた WebSocket ストリームを開始し、nghttp2 のイベントループ（h2_session_run）内でフレーム受信やヘッダー処理に応じて状態を更新し、終了時に h2_stream_closed_cb などのコールバックを実行します。

*注意: サーバーサイドとテストクライアント側の詳細な相互作用や、すべてのエラーハンドリングにおけるエンドツーエンドの完全なフローは、提供されたソース範囲からは完全には特定されていません。*

重要な不変条件

  • **RFC 7540 への準拠**: ストリームの状態遷移（IDLE, OPEN, CLOSED 等）は、RFC 7540 に準拠して厳格に管理されなければなりません。
  • **スレッド間の同期**: h2_stream はメイン接続（c1）とワーカースレッド（c2）の間で共有されるため、h2_mplx や h2_bucket_beam を介したアクセスには適切な同期が必要です。
  • **メモリ整合性**: AP_DEBUG が有効な場合、メモリ整合性チェックのためにマジックナンバーが使用され、破損が検出された場合はアサーション等により停止します。

よくある落とし穴

  • **スレッド間アクセスの競合**: メイン接続（c1）とワーカースレッド（c2）のコンテキストを跨ぐため、提供されている API を介さずに h2_stream のメンバへ直接アクセスすると、デッドロックや競合状態を引き起こす原因になります。
  • **状態遷移の不整合**: h2_stream_dispatch() 呼び出し時に、現在のストリーム状態が期待値と異なる場合、予期しないエラーが発生する可能性があります。
  • **同名構造体の混同**: modules/http2/ 配下のサーバー実装としての h2_stream と、test/clients/h2ws.c 内のテストクライアント用 h2_stream は、名前は同じですが役割や依存するライブラリ（後者は nghttp2 に強く依存）が異なるため、混同しないよう注意が必要です。
