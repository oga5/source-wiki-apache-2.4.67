h2_stream

概念

h2_stream は、HTTP/2 プロトコルにおける個別のリクエストとレスポンスのペア（ライフサイクル、状態遷移、およびデータ転送）を管理するためのコンテキスト構造体です。

Apache HTTP Server のサーバー側実装（modules/http2）においては、単一の h2_session 内で多重化される各ストリームを表現し、メイン接続（c1）とワーカースレッド（c2）の間でデータを仲介する役割を担います。また、テスト用クライアント（test/clients/h2ws.c）内にも同名の struct h2_stream が存在し、こちらはクライアント側での WebSocket ストリーム等の状態管理に用いられます。

なぜ重要か

HTTP/2 は単一の物理接続上で複数の双方向ストリームを多重化します。h2_stream は、この多重化された各通信の独立した状態、メモリプール、およびスレッド間データ転送バッファ（h2_bucket_beam）を管理する中核的なデータ構造です。そのため、HTTP/2 接続における並行処理の正確性とパフォーマンスに直接影響を与えます。

関与するモジュール

  • module:modules/http2 (サーバー側の HTTP/2 プロトコル実装)
  • module:test/clients (テスト用クライアントツール)

  • symbol:modules/http2/h2_mplx.h#h2_stream: 多重化ハブである h2_mplx から参照される不完全型（opaque struct）としての h2_stream。

エンドツーエンドの流れ

提供された情報から確認できるサーバー側の大まかな処理の流れは以下の通りです：

1. **生成**: h2_session_create や h2_session_open_stream を契機に h2_stream_create が呼び出され、h2_stream が生成・初期化されます。 2. **状態遷移と処理**: h2_stream_dispatch を通じてフレーム受信などのイベントを処理し、状態（IDLE, OPEN, CLOSED 等）を遷移させます。h2_stream_prepare_processing により準備が完了すると、ワーカーがリクエストを処理します。 3. **データ転送**: メイン接続（c1）とワーカースレッド（c2）の間で、h2_bucket_beam を介してデータが転送されます。 4. **クリーンアップと破棄**: 処理完了後、h2_mplx_c1_stream_cleanup を経て破棄キューへ移動し、最終的に h2_stream_destroy によってメモリプールとともに破棄されます。

*注記: 詳細なエンドツーエンドの制御フロー全体や、test/clients/h2ws.c における nghttp2 コールバックとの完全な連携フローの全容は、提供された情報からは完全には明らかになっていません。*

重要な不変条件

  • **RFC 7540 準拠**: ストリームの状態遷移は RFC 7540 に準拠し、厳格に管理される必要があります。
  • **スレッド間の同期**: h2_mplx によって管理され、メイン接続（c1）とワーカースレッド（c2）の間で共有されるため、h2_bucket_beam を介したスレッドセーフなデータ転送と適切な同期が必要です。
  • **メモリ整合性**: AP_DEBUG が有効な場合、メモリ破壊や整合性エラーを検出するためにマジックナンバーが使用されます。

よくある落とし穴

  • **直接アクセスの禁止**: h2_stream は c1（メイン接続）と c2（ワーカースレッド）のコンテキストを跨ぐため、メンバへの直接アクセスは避け、提供されている API を介して操作する必要があります。不適切なアクセスはデッドロックや競合状態を引き起こす可能性があります。
  • **状態遷移の不整合**: h2_stream_dispatch 呼び出し時に現在の状態が期待値と異なる場合、エラーが発生する可能性があります。
  • **同名構造体の混同**: test/clients/h2ws.c に定義されている h2_stream は、サーバー側の modules/http2/ 配下のものと同名ですが、テストクライアント固有の別定義です。これらが同一の定義を指しているか、あるいは同名の別構造体であるかの断定は、提供されたソースの範囲内では不可能です。
