h2_stream

概念

h2_stream は、HTTP/2 セッション内における個別のリクエストおよびレスポンスのペア（ストリーム）のライフサイクル、状態遷移、およびデータ転送を管理するためのコンテキスト構造体です。

Apache HTTP Server のサーバー側実装（modules/http2）と、テストクライアントツール（test/clients/h2ws.c）の双方において、それぞれの文脈に応じた同名の構造体 h2_stream が定義されています。

なぜ重要か

HTTP/2 プロトコルは単一の接続（セッション）上で複数のストリームを多重化して処理します。h2_stream は、この多重化された各ストリームの状態（RFC 7540 準拠の IDLE, OPEN, CLOSED など）を厳格に管理し、メイン接続（c1）とワーカースレッド（c2）の間で安全かつ効率的にデータを転送するための中心的な役割を担うため重要です。

エンドツーエンドの流れ

提供された情報に基づく、modules/http2 における h2_stream のライフサイクルおよびデータの流れは以下の通りです。

1. **生成**: h2_session に紐付けられ、h2_stream_create() によって初期化されます。 2. **データの蓄積**: h2_stream_add_header() や h2_stream_recv_DATA() を介して、受信したフレームデータが蓄積されます。 3. **処理の準備と実行**: h2_stream_prepare_processing により準備が整うと、ワーカースレッド（c2）がリクエストを処理しレスポンスを生成します。 4. **データ転送**: メイン接続（c1）とワーカースレッド（c2）の間で、h2_bucket_beam を介してスレッドセーフにデータや状態が転送されます。 5. **レスポンス送出**: h2_stream_read_to() を介して処理済みのレスポンスデータが送出されます。 6. **クリーンアップと破棄**: 処理完了後、h2_mplx_c1_stream_cleanup を経て破棄キューへ移動し、最終的に h2_stream_destroy() によって関連するメモリプールと共にリソースが解放されます。

なお、テストクライアント（h2ws.c）においては、ws_stream_submit() を通じて CONNECT メソッドを用いた WebSocket ストリームが開始され、nghttp2 のイベントループ（h2_session_run）内で状態が更新され、終了時に h2_stream_closed_cb などのコールバックが実行される流れとなります。

重要な不変条件

  • **状態遷移の準拠**: ストリームの状態遷移は RFC 7540 に準拠し、厳格に管理される必要があります。
  • **スレッド間同期**: メイン接続（c1）とワーカースレッド（c2）の間で共有されるため、h2_bucket_beam を介したスレッドセーフなデータ転送と、適切な同期アクセスが保証されなければなりません。
  • **メモリ整合性**: AP_DEBUG が有効な場合、メモリ整合性チェックのためにマジックナンバーが維持されます。

よくある落とし穴

  • **直接アクセスの禁止**: h2_stream は c1 と c2 のコンテキストを跨いで使用されるため、メンバへの直接アクセスは避け、提供されている API を介して操作する必要があります。不適切なアクセスはデッドロックや競合状態を引き起こす可能性があります。
  • **状態不整合**: h2_stream_dispatch 呼び出し時に現在の状態が期待値と異なる場合、エラーが発生する可能性があります。
  • **同名構造体の混同**: test/clients/h2ws.c 内の h2_stream と modules/http2/ 内の h2_stream は、同名ですが別個の定義である可能性があり、混同しないよう注意が必要です。
