h2_stream

概念

h2_stream は、HTTP/2 プロトコルにおける個別のリクエストとレスポンスのペア（ストリーム）を管理するためのコンテキスト構造体です。単一の HTTP/2 セッション（h2_session）内で多重化される各ストリームのライフサイクル、状態遷移、およびデータ転送を制御します。

なぜ重要か

HTTP/2 では、単一のTCP接続上で複数のリクエストとレスポンスが並行して処理（多重化）されます。h2_stream は、メイン接続（c1）とワーカースレッド（c2）の間で、リクエストのスケジューリング、処理、およびレスポンスデータの転送（h2_bucket_beam を介したスレッド間データ転送）を安全かつ効率的に行うための中心的な役割を果たします。また、テストクライアント（h2ws.c）においても、WebSocket（CONNECT メソッド）などの個別ストリーム状態を管理するために同名の構造体が使用されます。

エンドツーエンドの流れ

サーバー側における h2_stream のライフサイクルは以下の通りです：

1. **生成**: h2_session に紐付けられ、h2_stream_create() によって初期化されます。 2. **データ蓄積**: h2_stream_add_header() や h2_stream_recv_DATA() を介して、受信したフレームデータが蓄積されます。 3. **処理準備と実行**: h2_stream_prepare_processing により準備が整うと、ワーカースレッド（c2）がリクエストを処理します。メイン接続（c1）とワーカースレッド（c2）の間のデータ転送には h2_bucket_beam が使用されます。 4. **レスポンス送出**: h2_stream_read_to() を介して、処理済みのレスポンスデータが送出されます。 5. **クリーンアップと破棄**: 処理完了後、h2_mplx_c1_stream_cleanup を経て破棄キューへ移動し、最終的に h2_stream_destroy() によって関連するメモリプールと共に破棄されます。

テストクライアント（h2ws.c）側では、ws_stream_submit() を通じて CONNECT メソッドを用いた WebSocket ストリームが開始され、nghttp2 のイベントループ（h2_session_run）内でフレーム受信やヘッダー処理に応じて状態が更新され、終了時には h2_stream_closed_cb などのコールバックが実行されます。

※なお、サーバー側（modules/http2）とテストクライアント側（test/clients）の h2_stream 構造体が同一の定義を指しているか、あるいは同名の別構造体であるかの詳細な関係性については、提供されたソースの範囲内では完全には判明していません。

重要な不変条件

  • **RFC 7540 への準拠**: ストリームの状態遷移（IDLE, OPEN, CLOSED 等）は、RFC 7540 に準拠して厳格に管理される必要があります。
  • **スレッドセーフなデータ転送**: ワーカースレッド（c2）とメイン接続（c1）の間で共有されるため、h2_bucket_beam を介した安全な同期とデータ転送が保証されなければなりません。
  • **メモリ整合性**: AP_DEBUG が有効な場合、メモリ整合性チェックのためにマジックナンバーが使用され、メモリ破壊が検出された場合はアサーション等により停止します。

よくある落とし穴

  • **直接アクセスの禁止**: h2_stream は c1（メイン接続）と c2（ワーカースレッド）のコンテキストを跨ぐため、メンバへの直接アクセスは避け、提供されている API を介して操作する必要があります。不適切なアクセスはデッドロックや競合状態を引き起こす可能性があります。
  • **状態遷移の不整合**: h2_stream_dispatch 呼び出し時に現在の状態が期待値と異なる場合、エラーが発生する可能性があります。
  • **名前の混同**: test/clients/h2ws.c 内の h2_stream と modules/http2/ 内の同名構造体は、役割は類似していますが、依存関係や定義が異なる可能性があるため、混同しないよう注意が必要です。
