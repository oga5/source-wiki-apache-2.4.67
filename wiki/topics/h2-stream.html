h2_stream

概念

h2_stream は、HTTP/2 セッション内における個別のリクエストおよびレスポンスのペア（ストリーム）のライフサイクル、状態遷移、およびデータ転送を管理するためのコンテキスト構造体です。

この構造体は、サーバーの実装である modules/http2 モジュールと、テストクライアントである test/clients モジュールの両方に現れます。サーバー側ではメイン接続（c1）とワーカースレッド（c2）の間で多重化されるリソースとして扱われ、テストクライアント側では WebSocket ストリームなどの個別ストリームの状態管理に用いられます。

なぜ重要か

HTTP/2 プロトコルは、単一の接続上で複数のリクエストとレスポンスを多重化して処理します。h2_stream は、この多重化された各ストリームの独立した状態（IDLE, OPEN, CLOSED 等）を追跡し、スレッド間での安全なデータ転送や、適切なリソース解放を保証するための中核的な役割を担っています。

関与するモジュール

主要エンティティ

  • **symbol:modules/http2/h2_mplx.h#h2_stream**: modules/http2 における不完全型（opaque struct）としての宣言。メイン接続（c1）とワーカースレッド（c2）間の多重化管理を行う h2_mplx から参照されます。
  • **symbol:modules/http2/h2_stream.h#h2_stream**: modules/http2 における h2_stream 構造体の具体的な定義。ストリームのライフサイクル、状態遷移、およびデータ転送用のメンバを保持します。
  • **symbol:modules/http2/h2_stream.c#h2_stream_create**: サーバー側で h2_stream を生成・初期化するための関数。
  • **symbol:test/clients/h2ws.c#h2_stream**: テストクライアント h2ws 内で定義されている、nghttp2 コールバックと連携してストリーム状態を管理するための構造体。

エンドツーエンドの流れ

サーバー側（modules/http2）における一般的なライフサイクルの流れは以下の通りです。

1. **生成**: h2_session に紐付けられ、h2_stream_create によって h2_stream が生成・初期化されます。 2. **データ蓄積と状態遷移**: h2_stream_add_header や h2_stream_recv_DATA を介してフレームデータが蓄積され、h2_stream_dispatch を通じて状態（IDLE, OPEN, CLOSED 等）が遷移します。 3. **処理と転送**: h2_stream_prepare_processing により準備が整うと、ワーカースレッドが処理を開始します。メイン接続（c1）とワーカースレッド（c2）間のデータ転送には h2_bucket_beam が使用されます。 4. **クリーンアップと破棄**: 処理完了後、h2_mplx_c1_stream_cleanup を経て破棄キューへ移動し、最終的に h2_stream_destroy によって関連するメモリプールと共に破棄されます。

テストクライアント（test/clients/h2ws.c）においては、以下の流れで処理されます。

1. **開始**: ws_stream_submit を通じて CONNECT メソッドを用いた WebSocket ストリームが開始されます。 2. **イベントループ**: nghttp2 のイベントループ（h2_session_run）内で、フレーム受信やヘッダー処理に応じてストリームの状態が更新されます。 3. **終了**: ストリーム終了時に、h2_stream_closed_cb などの関連するコールバック関数が実行されます。

※なお、これら2つのモジュール間（サーバー実装とテストクライアント）における詳細な相互作用や、すべての境界条件におけるエンドツーエンドの完全なフローは、提供されたソース範囲からは未解明です。

重要な不変条件

  • **RFC 7540 への準拠**: サーバー側でのストリームの状態遷移は RFC 7540 に準拠し、厳格に管理される必要があります。
  • **スレッド間の同期**: h2_mplx によって管理されるストリームは、メイン接続（c1）とワーカースレッド（c2）の間で共有されるため、アクセスには適切な同期が必要です。スレッド間のデータ転送には h2_bucket_beam が使用され、スレッドセーフな転送が保証されます。
  • **メモリ整合性**: AP_DEBUG が有効な場合、メモリ整合性チェックのためにマジックナンバーが使用され、メモリ破壊が検出された場合はアサーション等により停止します。
  • **テストクライアントにおける依存**: test/clients/h2ws.c の h2_stream は、nghttp2 のコールバック API に強く依存しており、そのライフサイクルは nghttp2 のイベントループと連動します。

よくある落とし穴

  • **直接アクセスの回避**: サーバー側において、h2_stream は c1（メイン接続）と c2（ワーカースレッド）のコンテキストを跨ぐため、メンバへの直接アクセスは避け、提供されている API を介して操作する必要があります。不適切なアクセスはデッドロックや競合状態を引き起こす可能性があります。
  • **状態遷移の不整合**: h2_stream_dispatch 呼び出し時に、現在の状態が期待値と異なる場合、エラーが発生する可能性があります。
  • **同名構造体の混同**: test/clients/h2ws.c 内の h2_stream と modules/http2/ 内の h2_stream は、名前は同じですが、前者はテストクライアント用の独自実装（または nghttp2 連携用）であり、後者はサーバーのコア実装です。これらを混同しないよう注意が必要です。
  • **非ブロッキング I/O のハンドリング**: テストクライアントでは非ブロッキングソケットを使用しているため、nghttp2 のコールバック内で EWOULDBLOCK などの I/O 状態を適切に処理しないと、通信がハングする恐れがあります。
