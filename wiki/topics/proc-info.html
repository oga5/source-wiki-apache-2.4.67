proc_info

概念

proc_info は、Apache HTTP Server のマルチプロセッシングモジュール（MPM）である event および worker において、プロセスやスレッドの識別情報、および関連するコンテキスト（ソケット記述子など）を保持し、スレッド生成時に起動関数へ情報を引き渡すために使用される構造体です。

なぜ重要か

マルチスレッドモデルを採用するMPMにおいて、新しく生成されるスレッド（ワーカー等）に対して、自身がどのプロセススロットやスレッドスロットに属しているか、あるいはどのプロセスID、スレッドID、ソケット記述子を扱うべきかというコンテキストを安全かつ動的に引き渡す役割を担うため重要です。

関与するモジュール

  • module:server/mpm/event
  • module:server/mpm/worker

主要エンティティ

  • symbol:server/mpm/event/event.c#proc_info : プロセススロット（pslot）およびワーカー用スレッドスロット（tslot）を保持する構造体。
  • symbol:server/mpm/worker/worker.c#proc_info : プロセスID（pid）、スレッドID（tid）、およびソケット記述子（sd）を保持する構造体。

エンドツーエンドの流れ

1. **メモリ確保と初期化**: スレッドを生成する側（event.c の thread_starter や、worker.c のスレッド初期化処理など）が、ap_malloc を用いて proc_info 構造体のメモリをヒープ上に動的に確保し、必要な識別情報（スロット番号、PID、TID、ソケット記述子など）を設定します。 2. **コンテキストの引き渡し**: スレッド生成時に、この構造体へのポインタを void *dummy 引数としてスレッド起動関数に渡します。 3. **スレッド内での利用**: 起動されたスレッド関数内において、受け取った void * ポインタを proc_info * にキャストし、自身の識別情報やソケット記述子を参照して処理を行います。

※なお、メモリの解放タイミングなど、詳細なライフサイクル管理を含む完全なエンドツーエンドの流れは、提供されたコンテキストからは完全には判明していません。

重要な不変条件

  • event.c において、pslot および tslot は、Apache MPM のプロセス管理およびスレッド管理におけるインデックスとして有効な値でなければなりません。
  • worker.c において、ap_malloc を用いて動的にメモリ確保された場合は、呼び出し側で適切にメモリ管理を行う必要があります。
  • ※その他の詳細な不変条件や、OSの型定義（pid_t 等）との厳密な整合性に関する不変条件は、現時点では完全には判明していません。

よくある落とし穴

  • **同名異構造**: event.c と worker.c の両方に proc_info という同名の構造体が存在しますが、メンバ構成（pslot/tslot と pid/tid/sd）が異なります。一方を変更する際、もう一方のMPMへの影響やコードの混同に注意する必要があります。
  • **メモリ確保の失敗**: ap_malloc によるメモリ確保が失敗した場合、スレッド起動プロセスが中断されたり、プログラムが終了したりする可能性があります。
  • **キャストの安全性**: スレッド関数内で void *dummy ポインタから proc_info * へキャストする際、ポインタが正しく初期化されていない場合に未定義の動作を引き起こすリスクがあります。
