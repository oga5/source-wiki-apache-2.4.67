proc_info

概念

proc_info は、Apache HTTP Server のマルチプロセッシングモジュール（MPM）である event および worker において、スレッド生成時にプロセスやスレッドの識別情報（および関連するソケット記述子など）をスレッド関数へ引き渡すために使用される構造体です。

なぜ重要か

マルチスレッドモデルを採用する MPM（event や worker）では、新しく起動されるスレッドが自身がどのプロセススロットやスレッドスロットに属しているか、あるいはどのプロセスIDやスレッドID、ソケット記述子を扱うべきかを把握する必要があります。proc_info は、スレッド起動関数の void * 引数を介して、これらのコンテキスト情報を安全かつ動的に伝達する役割を担っています。

関与するモジュール

  • module:server/mpm/event
  • module:server/mpm/worker

主要エンティティ

  • **symbol:server/mpm/event/event.c#proc_info**: pslot（プロセススロット）と tslot（ワーカー用スレッドスロット）を保持する構造体。
  • **symbol:server/mpm/worker/worker.c#proc_info**: pid（プロセスID）、tid（スレッドID）、sd（ソケット記述子）を保持する構造体。

エンドツーエンドの流れ

1. **メモリ確保と初期化**: スレッドを生成する側（例: event.c 内の thread_starter など）が、ap_malloc を使用して proc_info 構造体のインスタンスをヒープ上に動的に確保し、必要な識別情報を設定します。 2. **スレッドへの引き渡し**: スレッド生成時に、この構造体へのポインタをスレッド関数の引数（void *dummy）として渡します。 3. **コンテキストの参照**: 起動されたスレッド関数内で、受け取った void * ポインタを proc_info * にキャストし、自身のスロット情報や識別情報を読み取って処理に利用します。

※なお、スレッド起動後のメモリ解放の具体的なタイミングや、エラー発生時の詳細なクリーンアップフローなど、エンドツーエンドの完全なライフサイクル管理の詳細は現時点では完全には判明していません。

重要な不変条件

  • **有効なインデックス/識別子**: proc_info に格納される pslot、tslot、pid、tid などの値は、MPM のプロセス・スレッド管理において有効な範囲のインデックスまたは識別子でなければなりません。
  • **動的確保の必要性**: スレッドが非同期に起動してパラメータを参照するため、渡される proc_info はスタック領域ではなく、ap_malloc 等で動的に確保されたメモリ領域を指している必要があります。

よくある落とし穴

  • **構造体の定義の不一致**: event.c と worker.c で同名の proc_info 構造体が定義されていますが、保持するメンバ（pslot/tslot と pid/tid/sd）が異なります。一方の MPM での変更が他方に影響を与えないよう、それぞれの定義箇所を意識して保守する必要があります。
  • **メモリリークとライフサイクル管理**: ap_malloc で動的に確保された proc_info インスタンスは、スレッド側で適切に解放されない場合、メモリリークの原因となります。
  • **キャストの安全性**: スレッド関数内で void * から proc_info * へキャストする際、ポインタが正しく初期化されていない、あるいは生存期間が切れている（ダングリングポインタ）場合、未定義動作やクラッシュを引き起こす可能性があります。
