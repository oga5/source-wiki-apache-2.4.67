proc_info

概念

proc_info は、Apache HTTP Server のマルチプロセッシングモジュール（MPM）である event および worker において、新しく生成されるスレッドにプロセスやスレッドの識別情報（およびソケット記述子などのコンテキスト）を安全に引き渡すために使用される構造体です。

なぜ重要か

マルチスレッドモデルを採用する MPM（event や worker）では、スレッド起動時に各スレッドが自身が担当するプロセススロットやスレッドスロット、あるいはプロセスIDやスレッドID、ソケット記述子を正しく認識する必要があります。proc_info は、スレッド生成関数（pthread_create 等に渡される起動ルーチン）に対して、void * 型の汎用引数を経由してこれらのコンテキスト情報を安全に受け渡すためのデータコンテナとして機能します。

関与するモジュール

  • module:server/mpm/event
  • module:server/mpm/worker

主要エンティティ

  • **symbol:server/mpm/event/event.c#proc_info**: event MPM 内で定義される構造体。プロセススロット（pslot）およびワーカー用スレッドスロット（tslot）を保持します。
  • **symbol:server/mpm/worker/worker.c#proc_info**: worker MPM 内で定義される構造体。プロセスID（pid）、スレッドID（tid）、およびソケット記述子（sd）を保持します。

エンドツーエンドの流れ

1. **メモリ確保と初期化**: スレッドを生成する側（例: event.c の thread_starter や、worker.c のスレッド生成処理）が、ap_malloc を使用して proc_info 構造体のインスタンスをヒープ上に動的に割り当てます。 2. **コンテキストの設定**: 割り当てられた構造体に、対象スレッドが使用すべき識別情報（pslot/tslot または pid/tid/sd）を設定します。 3. **スレッドへの引き渡し**: スレッド生成時に、この構造体へのポインタをスレッド関数の引数（void *dummy など）として渡します。 4. **キャストと利用**: 起動したスレッド関数内部で、受け取った void * ポインタを proc_info * にキャストし、格納されている識別情報を読み取って自身のコンテキストを初期化します。

※なお、スレッド起動後の proc_info インスタンスの正確な解放タイミングや、エラー発生時の詳細なクリーンアップフローなど、完全なエンドツーエンドのライフサイクル管理の詳細は提供された情報からは未確定です。

重要な不変条件

  • **有効なインデックス/識別子**: proc_info に格納される pslot、tslot、pid、tid などの値は、MPM のプロセスおよびスレッド管理において常に有効な範囲のインデックスまたは識別子でなければなりません。
  • **メモリ管理の整合性**: ap_malloc で動的に確保された proc_info インスタンスは、受け手側のスレッドまたは生成側によって適切に解放される必要があります（具体的な解放の不変条件は未確定です）。

よくある落とし穴

  • **同名構造体の定義の相違**: proc_info は event.c と worker.c の両方で定義されていますが、そのメンバ構成（pslot/tslot と pid/tid/sd）は異なります。一方の MPM での変更が他方に影響を与えないよう、それぞれのファイルにおける定義と利用箇所を慎重に区別する必要があります。
  • **メモリ確保失敗時のハンドリング**: ap_malloc によるメモリ確保が失敗した場合、スレッドの起動処理が中断され、システム全体の初期化やリクエスト処理に影響を及ぼす可能性があります。
  • **キャスト時の安全性**: スレッド関数内で void * から proc_info * へキャストする際、ポインタが正しく初期化されていない場合や、不正なアドレスが渡された場合に未定義の動作を引き起こす危険性があります。
