proc_info

概念

proc_info は、Apache HTTP Server のマルチプロセッシングモジュール（MPM）である event および worker において、プロセスやスレッドの識別情報、および関連するコンテキスト（ソケット記述子など）をスレッド起動時や初期化時に引き渡すために使用される構造体です。

同名の構造体が event.c と worker.c の両方に定義されていますが、それぞれ保持するメンバや役割が異なります。

なぜ重要か

マルチスレッドモデルを採用する MPM（event および worker）において、新しく生成されるスレッドに対して、自身がどのプロセススロットやスレッドスロットに属しているか、あるいはどのプロセスID（PID）、スレッドID（TID）、ソケット記述子（sd）を扱うべきかというコンテキスト情報を安全かつ動的に引き渡すために不可欠なデータ構造だからです。

関与するモジュール

  • module:server/mpm/event
  • module:server/mpm/worker

主要エンティティ

  • **symbol:server/mpm/event/event.c#proc_info**:

    • pslot（プロセススロット）および tslot（ワーカー用スレッドスロット）を保持する構造体。

  • **symbol:server/mpm/worker/worker.c#proc_info**:

    • pid（プロセスID）、tid（スレッドID）、sd（ソケット記述子）を保持する構造体。

エンドツーエンドの流れ

1. **メモリ確保と初期化**: スレッドを生成する側（例: event.c の thread_starter や worker.c のスレッド生成処理）が、ap_malloc を使用してヒープ上に proc_info 構造体のメモリを動的に割り当て、必要な識別情報を設定します。 2. **スレッドへの引き渡し**: 生成されたスレッドの起動関数に対して、void *dummy 引数（または汎用ポインタ）を経由して proc_info へのポインタが渡されます。 3. **コンテキストの復元**: スレッド関数内において、受け取った void * ポインタを proc_info * にキャストし、自身の所属するプロセスやスレッドの識別子、あるいはソケット記述子を参照して処理を行います。

※なお、スレッド終了時におけるメモリ解放の具体的なタイミングや、エラーハンドリングを含む完全なエンドツーエンドのライフサイクル管理フローの詳細は、提供されたコンテキストからは完全には明らかになっていません。

重要な不変条件

  • **有効なインデックス/識別子**: event.c における pslot および tslot は、MPM のプロセスおよびスレッド管理において有効な範囲のインデックス値でなければなりません。
  • **メモリ管理の責務**: ap_malloc を用いて動的にメモリ確保されるケース（worker.c:852 など）があり、その場合は呼び出し側で適切にメモリ管理を行う必要があります。

よくある落とし穴

  • **同名構造体の混同**: event.c と worker.c で定義されている proc_info は、名前は同じですがメンバ構成（pslot/tslot と pid/tid/sd）が異なります。一方の MPM での変更が他方に影響を与える可能性があるため、保守の際は注意が必要です。
  • **キャストの安全性**: スレッド起動時に関数の引数（void *dummy）を経由してポインタを受け渡すため、ポインタが正しく初期化されていない状態でキャストして参照すると、未定義の動作を引き起こす危険性があります。
  • **メモリ不足時の挙動**: ap_malloc によるメモリ確保が失敗した場合、スレッドの起動プロセスが中断され、プログラムが終了する可能性があります。
