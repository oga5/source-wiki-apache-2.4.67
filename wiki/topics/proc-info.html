proc_info

概念

proc_info は、Apache HTTP Server のマルチプロセッシングモジュール（MPM）である event および worker において、プロセスやスレッドの識別情報、および関連するコンテキスト（ソケット記述子など）を保持し、スレッド起動時や初期化時にスレッド関数へ引き渡すために使用される構造体です。

なぜ重要か

マルチスレッドモデルを採用する MPM（event や worker）では、新しく生成されたスレッドが自身を識別し、適切なリソース（スロットやソケット）にアクセスする必要があります。proc_info は、スレッド生成関数（pthread_create 等に渡されるスレッド関数）の汎用的な void * 引数を介して、スレッド固有のコンテキスト（プロセス/スレッドスロット、PID、TID、ソケット記述子など）を安全かつ確実に引き渡すためのデータキャリアとして機能します。

関与するモジュール

  • module:server/mpm/event
  • module:server/mpm/worker

主要エンティティ

  • **symbol:server/mpm/event/event.c#proc_info**: pslot（プロセススロット）および tslot（ワーカー用スレッドスロット）をメンバに持つ構造体。
  • **symbol:server/mpm/worker/worker.c#proc_info**: pid（プロセスID）、tid（スレッドID）、および sd（ソケット記述子）をメンバに持つ構造体。

エンドツーエンドの流れ

1. **メモリ確保と初期化**: スレッドを生成する側（例: event.c の thread_starter や worker.c のスレッド生成処理）が、ap_malloc を使用して proc_info 構造体のメモリをヒープ上に動的に確保します。 2. **情報の格納**: 確保した構造体に、プロセススロット（pslot）、スレッドスロット（tslot）、あるいはプロセスID（pid）、スレッドID（tid）、ソケット記述子（sd）などの識別情報を設定します。 3. **スレッドへの引き渡し**: スレッド生成時に、この構造体へのポインタをスレッド関数の引数（void *dummy）として渡します。 4. **情報の復元と利用**: 起動したスレッド関数内で、受け取った void * ポインタを proc_info * にキャストし、自身のコンテキスト情報を読み取って処理を開始します。

*注意: メモリの解放タイミングを含む詳細なライフサイクルや、エラー発生時のクリーンアップ処理など、エンドツーエンドの完全なフローは現時点では完全に判明していません。*

重要な不変条件

  • **有効なインデックス/識別子**: event.c における pslot および tslot は、MPM のプロセスおよびスレッド管理において有効なインデックス範囲内でなければなりません。
  • **メモリ管理の責務**: ap_malloc を用いて動的に確保された proc_info インスタンスは、スレッド起動後に適切にメモリ管理（解放など）が行われる必要があります。
  • *注意: スレッド間での排他制御の有無や、具体的なメモリ解放のタイミングに関する不変条件は現時点では完全に判明していません。*

よくある落とし穴

  • **定義の不一致**: proc_info 構造体は event.c と worker.c の両方で同名で定義されていますが、保持するメンバ（pslot/tslot と pid/tid/sd）が異なります。一方の MPM での変更がもう一方に影響を与えないよう、それぞれの定義と使われ方に注意する必要があります。
  • **メモリ確保の失敗**: ap_malloc によるメモリ確保が失敗した場合、スレッドの起動プロセスが中断され、サーバーの動作に影響を与える可能性があります。
  • **キャストの安全性**: スレッド関数内で void * から proc_info * へキャストする際、ポインタが正しく初期化されていない、あるいは生存期間が切れている場合、未定義の動作を引き起こす危険性があります。
