proc_info

概念

proc_info は、Apache HTTP Server のマルチプロセッシングモジュール（MPM）である event および worker において、プロセスやスレッドの識別情報、および関連するコンテキスト（ソケット記述子など）をスレッド起動時や初期化時に引き渡すために使用される構造体です。

なぜ重要か

マルチスレッドモデルを採用する event および worker MPM において、新しく生成されるスレッドに対して、自身がどのプロセス（またはスロット）に属し、どのスレッド（またはスロット）として動作すべきかというコンテキスト情報を安全かつ確実に伝達するために不可欠なデータ構造だからです。

関与するモジュール

  • module:server/mpm/event
  • module:server/mpm/worker

主要エンティティ

  • **symbol:server/mpm/event/event.c#proc_info**
  • メンバとして pslot（プロセススロット）および tslot（ワーカー用スレッドスロット）を持ちます。
  • **symbol:server/mpm/worker/worker.c#proc_info**
  • メンバとして pid（プロセスID）、tid（スレッドID）、および sd（ソケット記述子）を持ちます。

エンドツーエンドの流れ

1. **メモリ確保と初期化**: スレッドを生成する側（event.c の thread_starter や、worker.c のスレッド生成処理）が、ap_malloc を用いてヒープ上に proc_info 構造体の領域を動的に確保し、識別情報を設定します。 2. **コンテキストの引き渡し**: スレッド生成関数の引数（void *dummy など）として、この構造体へのポインタが渡されます。 3. **情報の参照**: 起動したスレッド関数内部で、受け取った void * ポインタを proc_info * にキャストし、自身のプロセス/スレッド識別子やソケット記述子を取得して処理に利用します。

※なお、スレッド生成から破棄に至るライフサイクル全体におけるメモリ解放の厳密なタイミングなど、詳細なエンドツーエンドのフローの全容は、提供されたコンテキストからは完全には判明していません。

重要な不変条件

  • event.c において、pslot および tslot は、MPM のプロセスおよびスレッド管理におけるインデックスとして有効な値でなければなりません。
  • ap_malloc を用いて動的にメモリ確保されるため、呼び出し側または受け取り側で適切にメモリ管理（解放）を行う必要があります。

※その他の詳細な不変条件や、OSの型定義（pid_t 等）との厳密な整合性ルールについては、提供された情報からは完全には判明していません。

よくある落とし穴

  • **同名異構造の混同**: proc_info は event.c と worker.c の両方で定義されていますが、保持するメンバ（pslot/tslot と pid/tid/sd）が異なります。一方のMPMを変更する際、もう一方の定義と混同しないよう注意が必要です。
  • **メモリリークと確保失敗**: ap_malloc によるメモリ確保が失敗した場合、スレッド起動プロセスが中断されたり、メモリ不足によりプログラムが終了したりする可能性があります。また、ライフサイクル管理を誤るとメモリリークの原因になります。
  • **キャストの安全性**: スレッド関数内で void * ポインタから proc_info * へキャストする際、ポインタが正しく初期化されていないと未定義の動作を引き起こす危険性があります。
