cache

概念

Apache HTTP Server（httpd-2.4.67）における cache は、複数のモジュールで定義または利用されるキャッシュ機構を指します。

特に modules/mappers に属する mod_rewrite（mod_rewrite.c）においては、外部マップ（DBM、DBD、PRG 等）のルックアップ結果を保持し、リクエストをまたいで再利用するための専用のキャッシュ管理構造体（struct cache）として定義されています。また、modules/cache モジュール（mod_cache.c）にも cache に関連するシンボルが存在しますが、その詳細な構造や役割は提供された情報からは未解明です。

なぜ重要か

外部マップ（データベースや外部プログラムなど）への問い合わせは高コストな処理であるため、ルックアップ結果をキャッシュすることで、リクエスト処理のパフォーマンスを大幅に向上させることができます。

mod_rewrite のキャッシュは、リクエストごとのメモリプールとは独立した生存期間を持つ独自のプールを使用することで、リクエスト終了後もデータを保持し、効率的な再利用を可能にしています。

エンドツーエンドの流れ

提供された情報からは、キャッシュ全体の詳細なエンドツーエンドの制御フロー（キャッシュの生成、ルックアップ、更新、破棄の具体的な呼び出しシーケンスなど）は完全には明らかになっていません。

判明している範囲では、mod_rewrite において外部マップの検索が行われる際、検索結果が cache 構造体内のハッシュテーブル maps に格納され、スレッド環境下では lock（apr_thread_mutex_t）を介して排他制御を行いながら読み書きが行われます。

重要な不変条件

  • **生存期間の独立性**: mod_rewrite のキャッシュは、リクエストの生存期間（r->pool）とは独立した専用のメモリプール（cache->pool）を使用して管理されなければなりません。
  • **スレッド安全性**: スレッド環境（APR_HAS_THREADS が有効な場合）においては、ハッシュテーブル maps へのアクセス整合性を保証するため、必ず lock を用いて排他制御を行う必要があります。

よくある落とし穴

  • **メモリリーク**: mod_rewrite のキャッシュはリクエストの生存期間を超えてデータを保持するため、適切なメモリ管理を行わないとメモリリークを引き起こすリスクがあります。
  • **名前の混同**: mod_rewrite.c 内部の struct cache は専用のプライベートな構造体であり、include/ap_socache.h などの他のキャッシュAPIや mod_cache とは直接的な関係がありません。混同しないよう注意が必要です。
  • **初期化失敗**: スレッド環境において lock の初期化や取得に失敗した場合、キャッシュへの安全なアクセスが不可能になります。
