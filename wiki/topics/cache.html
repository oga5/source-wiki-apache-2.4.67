cache

概念

Apache HTTP Server における cache は、複数のモジュールで異なる文脈で使用されるキャッシュ機構を指します。

特に mod_rewrite（modules/mappers）においては、外部マップ（DBM、DBD、PRG など）のルックアップ結果をリクエスト間で保持し、再利用するための専用のキャッシュ構造体（struct cache）として定義されています。なお、mod_cache（modules/cache）にも同名のシンボルが存在しますが、その詳細な役割や構造は現時点では提供されていません。

なぜ重要か

mod_rewrite における外部マップの検索処理はコストが高いため、ルックアップ結果をキャッシュすることで、リクエスト処理のパフォーマンスを大幅に向上させることができます。リクエストの生存期間を超えてデータを保持するために、リクエストプールとは独立したメモリプールを管理し、マルチスレッド環境下での安全なアクセスを保証する仕組みが重要となります。

エンドツーエンドの流れ

mod_rewrite におけるキャッシュの利用フローは以下の通りです。

1. **ルックアップ要求**: 外部マップ（DBM, DBD, PRG 等）の検索要求が発生します。 2. **キャッシュ検索**: キャッシュ構造体 cache 内のハッシュテーブル maps から、該当するエントリ（cachedmap）を検索します。 3. **キャッシュミス時の登録**: キャッシュに存在しない場合、外部ソースから結果を取得し、cache->pool を用いてハッシュテーブル maps に登録します。スレッド環境下では、この書き込み操作は lock を介して排他制御されます。 4. **結果の返却**: キャッシュされた結果を返却します。

※ mod_cache 側のエンドツーエンドの流れ、およびシステム全体におけるキャッシュ機構間の統合的なフローは現時点では不明です。

重要な不変条件

  • **ライフサイクルの独立性**: mod_rewrite の cache は、個々のリクエストの生存期間（r->pool）とは独立して生存しなければなりません。そのため、キャッシュデータの割り当てには必ず cache->pool が使用されます。
  • **スレッドセーフティ**: APR_HAS_THREADS が有効な環境では、ハッシュテーブル maps へのアクセスは必ず lock（apr_thread_mutex_t）を介して保護されなければなりません。
  • ※ mod_cache 側の不変条件は現時点では不明です。

よくある落とし穴

  • **メモリリークとライフサイクル管理の誤り**: mod_rewrite のキャッシュはリクエストの生存期間を超えて保持されるため、誤って r->pool からメモリを割り当てると、リクエスト終了時にデータが消失し不正アクセスを招きます。逆に、不要になったキャッシュデータを適切に管理しないとメモリリークの原因になります。
  • **ロックの初期化・取得失敗**: スレッド環境において lock の初期化や取得に失敗した場合、キャッシュへのアクセスが安全に行えなくなります。
  • **名前の混同**: mod_rewrite.c 内の cache 構造体は、include/ap_socache.h などのグローバルな共有キャッシュAPIとは直接の関係がない、モジュール専用のクローズドな実装です。
