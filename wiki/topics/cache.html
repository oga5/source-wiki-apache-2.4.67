cache

概念

Apache HTTP Server（httpd-2.4.67）における「cache」は、複数のモジュールで異なる目的で使用されるキャッシュ機構を指します。

特に modules/mappers/mod_rewrite.c においては、外部マップ（DBM、DBD、PRGなど）のルックアップ結果を保持し、リクエスト処理のパフォーマンスを向上させるための専用のキャッシュ構造体（struct cache）として定義されています。

なお、modules/cache/mod_cache.c にも cache という名前のシンボルが存在しますが、その詳細な仕様や mod_rewrite のキャッシュとの直接的な関係は、提供された情報からは不明です。

なぜ重要か

mod_rewrite における cache は、外部マップの検索結果をメモリ上に保持することで、リクエストごとに発生する外部リソースへのアクセスオーバーヘッドを削減し、ルーティングや書き換え処理のパフォーマンスを大幅に向上させるために重要です。また、リクエストの生存期間を超えてキャッシュを維持する独自のライフサイクル管理を実現しています。

エンドツーエンドの流れ

提供された情報からは、キャッシュ全体の初期化からルックアップ、更新、破棄に至る具体的なエンドツーエンドの流れや、具体的な呼び出し元関数は特定されておらず、未詳です。

判明している部分的な振る舞いは以下の通りです： 1. 外部マップ（DBM, DBD, PRG等）の検索結果がハッシュテーブル maps に格納されます。 2. リクエストごとのメモリプールとは独立した pool を使用して、リクエスト終了後もキャッシュデータを生存させます。 3. マルチスレッド環境下（APR_HAS_THREADS 有効時）では、lock を介してハッシュテーブルへのアクセスを排他制御します。

重要な不変条件

  • **独立したライフサイクル**: キャッシュの生存期間を管理する pool は、個々のリクエストの生存期間（r->pool）とは独立して維持されなければなりません。
  • **スレッドセーフティ**: マルチスレッド環境（APR_HAS_THREADS）においては、ハッシュテーブル maps への読み書き時の整合性を保証するため、必ず lock を用いた排他制御が行われる必要があります。

よくある落とし穴

  • **メモリリーク**: mod_rewrite のキャッシュはリクエストの生存期間を超えてデータを保持するため、r->pool ではなく cachep->pool を使用したメモリ管理において、ライフサイクル設計を誤るとメモリリークが発生しやすくなります。
  • **ロックの初期化失敗**: スレッド環境において lock の初期化や取得に失敗した場合、キャッシュへの安全なアクセスが行えなくなります。
  • **名前の混同**: include/ap_socache.h などの共通キャッシュAPIや mod_cache と名前が類似していますが、mod_rewrite.c の cache は内部専用の独立した構造体であり、直接的な関係はありません。
