cache

概念

Apache HTTP Server（httpd-2.4.67）における「cache」は、複数のモジュールで異なる目的で使用されるキャッシュ機構を指します。

提供された情報に基づくと、mod_rewrite においては外部マップ（DBM、DBD、PRGなど）のルックアップ結果を保持し、リクエストをまたいで再利用するための専用のキャッシュ管理構造体（struct cache）として定義されています。なお、mod_cache にも同名のシンボルが存在しますが、その具体的な詳細情報は提供されていません。

なぜ重要か

外部マップ（データベースや外部プログラムなど）へのルックアップは高コストな処理です。mod_rewrite においてルックアップ結果をキャッシュすることで、リクエストごとの処理パフォーマンスを大幅に向上させることができます。また、リクエストの生存期間（r->pool）とは独立した独自のメモリプールを管理することで、リクエスト終了後もデータを永続化させることが可能になります。

エンドツーエンドの流れ

提供された情報からは、キャッシュ全体の詳細なエンドツーエンドの流れ（ルックアップの発生からキャッシュへの格納、ヒット時の返却、エントリの無効化など）や、mod_cache 側でのキャッシュ処理の具体的な流れは判明していません。

mod_rewrite における部分的な流れとしては以下の通りです：

1. 外部マップ（DBM, DBD, PRG 等）の検索が発生する。 2. 検索結果が cache 構造体内のハッシュテーブル maps に格納される。 3. スレッド環境下（APR_HAS_THREADS 有効時）では、ハッシュテーブルへのアクセス時に lock（ミューテックス）を介して排他制御が行われる。

重要な不変条件

  • **生存期間の独立性**: mod_rewrite の cache は、個々のリクエストのプール（r->pool）ではなく、キャッシュ専用のプール（cachep->pool）を使用し、リクエスト終了後も生存し続けなければなりません。
  • **スレッド安全性**: APR_HAS_THREADS が有効な環境においては、ハッシュテーブル maps へのアクセス（読み書き）は必ず lock（apr_thread_mutex_t）を介して排他制御され、整合性が保証されている必要があります。
  • mod_cache 側の不変条件については、提供された情報からは未解明です。

よくある落とし穴

  • **メモリリーク**: mod_rewrite のキャッシュはリクエストの生存期間を超えてデータを保持するため、メモリ管理（cachep->pool の使用方法）を誤るとメモリリークを引き起こすリスクがあります。
  • **初期化失敗時の安全性**: スレッド環境において lock の初期化や取得に失敗した場合、キャッシュへの安全なアクセスが行えなくなります。また、メモリ不足により pool の割り当てやハッシュテーブルの初期化が失敗する境界条件が存在します。
  • **名前の混同**: mod_rewrite.c 内部の struct cache は専用のプライベートな構造体であり、include/ap_socache.h などの他のキャッシュ機構とは直接的な関係はありません。
