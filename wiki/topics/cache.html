cache

概念

Apache HTTP Server（httpd-2.4.67）における cache は、複数のモジュールにまたがって存在するキャッシュ機構やその管理構造を指します。

具体的には、modules/mappers モジュールの mod_rewrite.c において、外部マップ（DBM、DBD、PRG など）のルックアップ結果を保持し、リクエスト間で再利用するための専用のキャッシュ管理構造体（struct cache）が定義されています。また、modules/cache モジュールの mod_cache.c にも関連するシンボルが存在しますが、その詳細な構造や役割は提供された情報からは不明です。

なぜ重要か

外部マップへのルックアップ処理は、リクエストごとに毎回実行するとシステムのパフォーマンスを著しく低下させる可能性があります。mod_rewrite における cache 構造体は、ルックアップ結果をハッシュテーブルに格納し、リクエストの生存期間を超えてデータを維持することで、不要な再ルックアップのオーバーヘッドを削減し、ルーティングおよびマッピング処理の高速化を実現するために重要です。

エンドツーエンドの流れ

提供された情報からは、キャッシュの初期化から破棄に至る完全なエンドツーエンドの流れ、および mod_cache.c と mod_rewrite.c の間におけるキャッシュ処理の連携の有無は判明していません。

mod_rewrite 内部における部分的な流れは以下の通りです： 1. 外部マップ（DBM, DBD, PRG 等）の検索が発生した際、その結果がハッシュテーブル maps に格納されます。 2. このデータは、リクエストごとのメモリプール（r->pool）とは独立して割り当てられた pool を使用して保持されるため、リクエストが終了した後も生存します。 3. マルチスレッド環境下（APR_HAS_THREADS が有効な場合）では、ハッシュテーブルへのアクセス時に lock を介して排他制御が行われます。

重要な不変条件

  • **ライフサイクルの独立性**: mod_rewrite の cache は、個々のリクエストの生存期間（r->pool）とは独立した専用のメモリプール（cache->pool）を使用し、リクエスト終了後もキャッシュデータを生存させなければなりません。
  • **スレッドセーフティ**: APR_HAS_THREADS が有効な環境においては、ハッシュテーブル maps への読み書き時に必ず lock（apr_thread_mutex_t）を介して排他制御を行い、データの整合性を保証する必要があります。
  • mod_cache.c 側の不変条件については、提供された情報からは不明です。

よくある落とし穴

  • **メモリリーク**: mod_rewrite のキャッシュはリクエストの生存期間を超えてデータを保持する設計（cachep->pool の使用）になっているため、メモリの割り当てやライフサイクル管理を誤ると、メモリリークを引き起こす原因になります。
  • **排他制御の失敗**: スレッド環境において lock の初期化や取得に失敗した場合、キャッシュへのアクセスが安全に行えず、競合状態（レースコンディション）やデータの破損を招く可能性があります。
  • **名前の混同**: mod_rewrite.c 内部の struct cache は同ファイル専用の構造体であり、include/ap_socache.h などの他の共通キャッシュ API とは直接的な関係はありません。
