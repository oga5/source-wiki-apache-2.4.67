cache

概念

Apache HTTP Serverにおける「cache」は、複数のモジュールで異なる目的のために定義・使用されるキャッシュ機構を指します。

提供された情報に基づくと、mod_rewrite モジュール（modules/mappers/mod_rewrite.c）内部の cache 構造体は、外部マップ（DBM、DBD、PRGなど）のルックアップ結果をメモリ上に保持するための専用キャッシュ管理構造体です。

なお、modules/cache/mod_cache.c にも同名の cache シンボルが存在しますが、その具体的な概念や役割は提供された情報からは未解明です。

なぜ重要か

mod_rewrite における cache 構造体は、外部マップの検索結果をキャッシュすることで、リクエストごとの外部ルックアップに伴うオーバーヘッドを削減し、ルーティングおよび書き換え処理のパフォーマンスを向上させるために重要です。リクエストの生存期間（r->pool）とは独立した独自のメモリプールを使用することで、リクエスト終了後もデータを永続化させることができます。

重要な不変条件

  • **生存期間の独立性**: mod_rewrite の cache は、個々のリクエストの生存期間（r->pool）とは独立した専用のメモリプール（cache->pool）を保持し、リクエスト終了後もキャッシュが生存し続ける必要があります。
  • **スレッド安全性**: スレッド環境（APR_HAS_THREADS が有効な場合）においては、ハッシュテーブル maps へのアクセス時の整合性を保証するため、lock（apr_thread_mutex_t）を用いた排他制御が行われなければなりません。

よくある落とし穴

  • **メモリリーク**: mod_rewrite のキャッシュはリクエストの生存期間を超えてデータを保持するため、メモリ管理（cache->pool の使用）を誤るとメモリリークを引き起こすリスクがあります。
  • **スレッド競合**: スレッド環境下で lock の初期化や取得に失敗した場合、キャッシュへの安全なアクセスが妨げられ、データの不整合やクラッシュにつながる可能性があります。
  • **名前の混同**: mod_rewrite.c 内の cache 構造体は内部専用の定義であり、include/ap_socache.h などの他のグローバルなキャッシュAPIとは直接的な関係がありません。
