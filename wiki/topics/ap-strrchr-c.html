ap_strrchr_c

概念

ap_strrchr_c は、標準Cライブラリの strrchr 関数をラップし、const char * 型の文字列に対して型安全な文字検索インターフェースを提供する関数、およびマクロです。

なぜ重要か

C言語の標準 strrchr は const char * を引数に取りながら char * を返却するため、意図せず const 属性が失われる（キャストアウトされる）危険性があります。ap_strrchr_c は、入力と同じ const char * 型のポインタを返却することで型安全性を保証します。また、ヘッダーファイルでのマクロ定義とソースファイルでの実体定義を併用することで、デバッグの容易性やバイナリ互換性を維持しています。

関与するモジュール

  • module:include
  • module:server

主要エンティティ

  • **symbol:include/httpd.h#ap_strrchr_c**: マクロ定義を提供し、コンパイル時の型安全な置換やインライン化を支援します。
  • **symbol:server/util_debug.c#ap_strrchr_c**: デバッグやバイナリ互換性のために提供される、実体としてのラッパー関数定義です。

エンドツーエンドの流れ

各種モジュール（mod_authn_core.c や mod_mime.c など）から ap_strrchr_c(s, c) が呼び出されると、最終的に標準ライブラリの strrchr(s, c) が実行され、文字 c が最後に出現する位置を指すポインタが返されます。

なお、コンパイル時にマクロ定義と server/util_debug.c の実体関数のどちらがどのような条件で優先して選択されるかといった、詳細なビルド時・実行時の制御フローの全貌は、提供された情報からは完全には特定されていません。

重要な不変条件

  • 入力引数 s は、有効なヌル終端文字列である必要があります。
  • 戻り値として返されるポインタは、必ず入力文字列 s のメモリ範囲内を指すか、あるいは NULL になります。

よくある落とし穴

  • 入力 s に NULL を渡した場合の挙動は、ラップしている標準ライブラリ strrchr の実装に依存するため、未定義動作（セグメンテーションフォールトなど）を引き起こす可能性があります。
  • 検索対象の文字 c が文字列内に存在しない場合は NULL が返されるため、呼び出し側で戻り値のヌルチェックを怠ると、ヌルポインタ参照によるクラッシュの原因になります。

次に読む場所

  • include/httpd.h : マクロ定義およびインターフェース宣言の確認。
  • server/util_debug.c : デバッグ用ラッパー関数の実装詳細の確認。
