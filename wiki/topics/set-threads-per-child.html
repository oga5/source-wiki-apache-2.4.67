set_threads_per_child

概念

set_threads_per_child は、Apache HTTP Serverのマルチプロセッシングモジュール（MPM）において、設定ディレクティブ ThreadsPerChild の値を処理するための設定ハンドラ関数です。この関数は、子プロセスごとに生成されるスレッド数を設定ファイルから読み込み、システムに適用する役割を持ちます。

なぜ重要か

Apache HTTP Serverの並行処理能力を決定する重要なパラメータである ThreadsPerChild ディレクティブを解析し、内部のグローバル変数（ap_threads_per_child）に反映するために必要不可欠です。複数の主要なMPM（event、winnt、worker）でそれぞれ同名の関数が定義されており、各MPMの特性に合わせた設定処理を担っています。

エンドツーエンドの流れ

設定ファイルの解析中に ThreadsPerChild ディレクティブが検出されると、AP_INIT_TAKE1 マクロを介して登録された各MPMの set_threads_per_child 関数が呼び出されます。

1. **コンテキストの検証**: ap_check_cmd_context を呼び出し、設定が適切なコンテキスト（グローバルスコープ）で行われているか確認します。 2. **値の変換と代入**: server/mpm/winnt/mpm_winnt.c の実装では、引数として渡された設定値（文字列）を atoi を用いて整数に変換し、グローバル変数 ap_threads_per_child に代入します。

なお、event および worker モジュールにおける具体的な変換処理や代入の内部実装、および設定された値がその後のプロセス/スレッド生成ライフサイクルでどのように使用されるかという詳細なエンドツーエンドのフローは、現時点では完全に解析されていません。

重要な不変条件

  • **グローバルスコープの要求**: 本関数は、設定コンテキストがグローバルスコープであることを要求します。これは ap_check_cmd_context による検証によって保証されます。
  • その他の詳細な不変条件（設定値の最小値・最大値の制限など）については、現時点では未解析です。

よくある落とし穴

  • **不適切な設定コンテキスト**: ThreadsPerChild ディレクティブがグローバルスコープ以外の不適切な場所（例えば <Directory> や <Location> セクションなど）で定義された場合、ap_check_cmd_context による検証が失敗し、エラーメッセージが返されます。
  • **非数値の入力**: winnt モジュールにおける実装では、設定値の変換に atoi を使用しているため、数値として解釈できない文字列が指定された場合は 0 が設定されます。この挙動に対するバリデーションやエラーハンドリングの詳細は現時点では不明です。
