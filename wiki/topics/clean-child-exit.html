clean_child_exit

概念

clean_child_exit は、Apache HTTP Server（httpd-2.4.67）のマルチプロセッシングモジュール（MPM）群において、子プロセスまたはワーカースレッドを安全かつクリーンに終了させるための内部関数です。各MPMのアーキテクチャ（プロセス駆動、スレッド駆動、あるいは特定のOS向け実装）に合わせて個別に定義されており、リソースの解放や状態の同期、プロセスの最終的な停止処理を担います。

なぜ重要か

サーバーの安定稼働とリソース管理において、子プロセスやスレッドの終了処理を適切に行うことは極めて重要です。

  • **リソースの確実な解放**: 子プロセスやスレッドに割り当てられたメモリプール（pchild や ptrans）やバケットアロケータを破棄し、メモリリークを防ぎます。
  • **ステータスの同期**: スコアボード（scoreboard）の状態を更新し、親プロセスや管理機構が子プロセスの停止を正しく検知できるようにします。
  • **安全なシグナル制御**: 終了処理中に不要なシグナル（SIGHUP, SIGTERMなど）を無視するように設定し、二重終了や不整合を防ぎます。

エンドツーエンドの流れ

各MPMにおける具体的なエンドツーエンドの呼び出しフローの全容は完全には解明されていませんが、提供された情報から以下の個別の流れが確認できます。

  • **OS/2 MPM (mpmt_os2_child.c)**:

    • 1. 子プロセスの生存期間を管理するメモリプール pchild が存在する場合、apr_pool_destroy で破棄します。 2. exit(code) を呼び出してプロセスを終了します。

  • **NetWare MPM (mpm_netware.c)**:

    • 1. バケットアロケータを破棄します。 2. サーバーが停止中でない場合、トランザクションプール ptrans を破棄します。 3. アトミック変数 worker_thread_count を減算します。 4. worker_num が 0 以上であれば、スコアボード上の当該スレッド状態を WORKER_DEAD に更新します。 5. NXThreadExit を呼び出してスレッドを終了します。

  • **Prefork MPM (prefork.c)**:

    • 1. MPMの状態を AP_MPMQ_STOPPING に設定し、SIGHUP および SIGTERM シグナルを無視するように設定します。 2. 正常終了（code == 0）の場合、ap_run_child_stopping フックを実行します。 3. メモリプール pchild が存在すれば破棄します。 4. one_process モードの場合、prefork_note_child_killed を呼び出して終了を記録します。 5. POD（Pipe of Death）を閉じ、プロファイリング用のディレクトリ移動を行った後、exit(code) でプロセスを終了します。

  • **Event MPM (event.c) / Worker MPM (worker.c)**:
  • 詳細な内部アルゴリズムは未解析ですが、それぞれ event_note_child_stopped や worker_note_child_killed を呼び出すことが分かっています。
  • 呼び出し元として、accept_mutex_error, accept_mutex_on, ap_mpm_child_main, child_main, worker_main などから呼び出されます。

重要な不変条件

  • **非復帰性**: 本関数は exit() や NXThreadExit を呼び出すため、呼び出し後に制御が戻ることはありません。
  • **メモリプールの存在チェック**: pchild などのメモリプールが NULL でない場合のみ破棄処理が実行されます。
  • **NetWareにおける条件**: worker_num が 0 以上の場合のみスコアボードが更新され、サーバー停止中（shutdown_pending が真）の場合は ptrans の破棄がスキップされます。

よくある落とし穴

  • **同名関数の混同**: clean_child_exit は複数のMPMファイルで同名で定義されていますが、それぞれプラットフォームや並行処理モデル（プロセスベース、スレッドベース、OS/2、NetWareなど）に特化した独立した実装です。静的解析ツールなどの呼び出し元情報が混同して表示されることがあるため、ソースコードを読む際は対象のMPMファイルを正確に区別する必要があります。
  • **リソースの解放漏れ**: shutdown_pending などの状態フラグや、プールが NULL であるかどうかのチェックを誤ると、リソースの解放漏れや不正アクセスが発生する可能性があります。
