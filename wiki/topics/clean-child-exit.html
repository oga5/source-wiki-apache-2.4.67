clean_child_exit

概念

clean_child_exit は、Apache HTTP Serverの各種マルチプロセッシングモジュール（MPM）において、子プロセスまたはワーカースレッドが終了する際に、リソースのクリーンアップ、メモリプールの破棄、スコアボードの更新、および安全なプロセス/スレッドの停止を行うための関数です。

この関数は、各MPMのアーキテクチャ（プロセスモデル、スレッドモデル、OS依存の実装）に合わせて個別に実装されています。

なぜ重要か

子プロセスやスレッドが終了する際、メモリリークの防止（メモリプールの破棄）、アロケータの解放、スコアボードのステータス更新（親プロセスや監視ツールへの状態通知）、およびシグナルハンドリングの適切な制御を行うことで、サーバー全体の安定性とリソース管理の健全性を維持するために極めて重要です。

エンドツーエンドの流れ

各MPMにおける具体的な呼び出し元や、システム全体における統一的なエンドツーエンドの制御フローは、提供された情報からは**完全には明らかになっていません**。

ただし、個別のMPM内部における処理の流れは以下の通りです。

OS/2 MPM (mpmt_os2_child.c)

1. 子プロセスの生存期間を管理するメモリプール pchild が存在する場合、apr_pool_destroy を呼び出して破棄します。 2. exit(code) を呼び出し、指定された終了コードでプロセスを終了します。

NetWare MPM (mpm_netware.c)

1. apr_bucket_alloc_destroy を呼び出し、バケットアロケータを破棄します。 2. サーバーが停止中でない場合（shutdown_pending が偽）、apr_pool_destroy を呼び出してトランザクションプール ptrans を解放します。 3. アトミック変数 worker_thread_count を減算します。 4. worker_num が 0 以上であれば、ap_update_child_status_from_indexes を呼び出してスコアボード上のスレッド状態を WORKER_DEAD に更新します。 5. NXThreadExit を呼び出し、スレッドを終了します。

Prefork MPM (prefork.c)

1. MPMの状態を AP_MPMQ_STOPPING に設定します。 2. SIGHUP および SIGTERM シグナルを無視するように設定します。 3. 正常終了（code が 0）の場合、ap_run_child_stopping フックを実行します。 4. 子プロセス固有のメモリプール pchild が存在する場合、これを破棄します。 5. one_process モードの場合、prefork_note_child_killed を呼び出して終了を記録します。 6. ap_mpm_pod_close を呼び出し、POD（Pipe of Death）を閉じます。 7. chdir_for_gprof を呼び出し、プロファイリング用のディレクトリ移動を行います。 8. exit(code) を呼び出し、プロセスを終了します。

重要な不変条件

すべてのMPMに共通するグローバルな不変条件は**完全には明らかになっていません**が、個別実装において以下の不変条件が維持されます。

  • **OS/2**: メモリプール pchild が NULL の場合、メモリプールの破棄はスキップされ、即座に exit(code) が実行されます。
  • **NetWare**: worker_num が 0 以上の場合のみスコアボードの更新が試行されます。また、shutdown_pending が真（サーバー停止中）の場合、ptrans プールの破棄はスキップされます。
  • **Prefork**: pchild が NULL の場合、メモリプールの破棄はスキップされます。また、one_process が真でない場合、prefork_note_child_killed は呼び出されません。

よくある落とし穴

  • **MPM間の混同**: clean_child_exit は複数のMPMで同名の関数として定義されていますが、シグネチャや引数、対象（プロセス終了かスレッド終了か）が大きく異なります。例えば、NetWareではスレッド終了（NXThreadExit）を扱い、PreforkやOS/2ではプロセス終了（exit）を扱います。これらを混同して共通のAPIのように扱うと、ビルドエラーや予期しない動作を引き起こします。
  • **制御の非返却**: 本関数は内部で exit() や NXThreadExit を呼び出すため、呼び出し後は制御が戻りません。関数の呼び出し後にコードを記述しても実行されない点に注意してください。
  • **リソースの二重解放やリーク**: shutdown_pending などの状態フラグによってプールの破棄がスキップされる場合があり、呼び出し側の状態管理と整合性が取れていないとリソースリークや不正アクセスにつながる可能性があります。
