clean_child_exit

概念

clean_child_exit は、Apache HTTP Server（httpd）の各種 MPM（Multi-Processing Module）において、子プロセスまたはワーカースレッドが終了する際に呼び出される内部関数です。この関数は、割り当てられたメモリプールやバケットアロケータなどのリソースを安全に解放し、プロセスやスレッドを適切に停止させる役割を持ちます。

なぜ重要か

子プロセスやスレッドが終了する際、確保したシステムリソース（メモリプール pchild や ptrans など）を確実にクリーンアップしないと、メモリリークやリソースの枯渇を引き起こす可能性があります。また、スコアボードのステータス更新やシグナルハンドリングの変更、POD（Pipe of Death）のクローズなどを同期的に行うことで、サーバー全体の動作の整合性を保ちながら安全にシャットダウンするために不可欠です。

エンドツーエンドの流れ

各 MPM における具体的な呼び出し元や、システム全体におけるエンドツーエンドの完全な制御フローは、提供された情報からは**未だ完全には判明していません**。

ただし、判明している各 MPM 内部での局所的な終了フローは以下の通りです。

OS/2 (mpmt_os2_child.c)

1. 子プロセスの生存期間を管理するメモリプール pchild が存在する場合、apr_pool_destroy で破棄します。 2. exit(code) を呼び出してプロセスを終了します。

NetWare (mpm_netware.c)

1. apr_bucket_alloc_destroy を呼び出し、バケットアロケータを破棄します。 2. サーバーが停止中でない場合（shutdown_pending が偽）、トランザクションプール ptrans を解放します。 3. アトミック変数 worker_thread_count を減算します。 4. worker_num が 0 以上であれば、スコアボード上のスレッド状態を WORKER_DEAD に更新します。 5. NXThreadExit を呼び出してスレッドを終了します。

Prefork (prefork.c)

1. MPM の状態を AP_MPMQ_STOPPING に設定します。 2. SIGHUP および SIGTERM シグナルを無視するように設定します。 3. 終了コードが 0 の場合、ap_run_child_stopping フックを実行します。 4. 子プロセス固有のメモリプール pchild が存在すれば破棄します。 5. one_process モードの場合は prefork_note_child_killed を呼び出して終了を記録します。 6. POD（Pipe of Death）を閉じます。 7. プロファイリング用のディレクトリ移動（chdir_for_gprof）を行い、exit(code) でプロセスを終了します。

重要な不変条件

提供された情報から特定できる不変条件は以下の通りです。

  • **制御の非返還**: 本関数は最終的に exit または NXThreadExit を呼び出すため、呼び出し元へ制御が戻ることはありません。
  • **メモリプールの存在確認**: OS/2 および Prefork 実装では、pchild が NULL でない場合のみ破棄を試みます。
  • **NetWare における条件付き解放**: shutdown_pending が真の場合、ptrans プールの破棄は本関数内ではスキップされます。また、worker_num が負の値の場合はスコアボードの更新がスキップされます。

よくある落とし穴

  • **MPM 間のシグネチャの不一致**: clean_child_exit は複数の MPM で同名の関数として定義されていますが、引数の数や型が異なります（例: OS/2 や Prefork では int code のみですが、NetWare では int code, int worker_num, apr_pool_t *ptrans, apr_bucket_alloc_t *bucket_alloc を受け取ります）。これらを混同して共通の API のように呼び出すことはできません。
  • **呼び出し後のコード実行**: 本関数を呼び出した後に記述されたコードは実行されません。クリーンアップ処理は必ず本関数の内部、または呼び出し前に完了している必要があります。
