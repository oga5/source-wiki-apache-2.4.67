dav_lock_private

概念

dav_lock_private は、Apache HTTP Server の WebDAV モジュール（mod_dav）において、ロックデータベース（lockdb）の実装がリポジトリ固有のロック情報を保持するためのデータ構造です。

mod_dav.h においては、具体的なメンバを隠蔽した不透明な（opaque）構造体として前方宣言（typedef struct dav_lock_private dav_lock_private;）されています。一方で、具体的なロックプロバイダの実装（modules/dav/fs/lock.c および modules/dav/lock/locks.c）においては、ロックデータベース（DBM）内の特定のレコードを識別するためのキー情報（apr_datum_t key）を保持する構造体として定義されています。

なぜ重要か

mod_dav のコアモジュールと、具体的なロック管理を行う各プロバイダ（ファイルシステムプロバイダや汎用ロックプロバイダ）との間で、実装の詳細をカプセル化しつつ、ロックデータベースのキー情報を安全に受け渡すために重要です。

上位の共通構造体である dav_lock は、この dav_lock_private を info メンバ（ポインタ）として保持することで、プロバイダ固有のデータ（DBMのキーなど）を透過的に管理できます。

エンドツーエンドの流れ

提供された情報からは、具体的な関数呼び出しの連鎖や、ロックの取得・解放・検証における詳細なエンドツーエンドの制御フローは完全には明らかになっていません。

判明している範囲では、以下の流れで利用されます： 1. mod_dav コアがロック操作を行う際、上位の dav_lock 構造体を介してプロバイダを呼び出します。 2. 各プロバイダ（fs や lock）は、dav_lock 内の info ポインタを dav_lock_private 構造体にキャストして使用します。 3. dav_lock_private 内の key（apr_datum_t 型）を用いて、ロックデータベース（DBM）から該当するロックレコードの検索、保存、削除などの物理的な操作を行います。

重要な不変条件

  • **不透明性の維持**: mod_dav コア側からは dav_lock_private の内部構造（key メンバなど）に直接アクセスしてはならず、常に不透明なポインタとして扱う必要があります。
  • **メモリの生存期間**: dav_lock_private 構造体のメモリ管理は、関連付けられたリクエストプール（apr_pool_t）の生存期間に依存します。
  • **キーの有効性**: key フィールド（apr_datum_t）は、DBM 操作に適した形式で適切に初期化されていなければなりません。

よくある落とし穴

  • **型安全性の欠如**: dav_lock_private は不透明なポインタとして扱われるため、異なるプロバイダ間で誤ってキャストして使用した場合に、型安全性が確保できずメモリ破損やクラッシュを引き起こす可能性があります。
  • **DBMの破損や不整合**: key が指す DBM 内のデータが破損している場合、またはデータベースが読み取り専用で開かれている場合に書き込み操作を行うと、後続のデータベース操作が失敗する可能性があります。
  • **互換性の維持**: 構造体の定義を変更する場合、mod_dav コアと各プロバイダ（fs、lock）のすべての実装において整合性を保つ必要があります。
