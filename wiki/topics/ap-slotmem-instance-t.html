ap_slotmem_instance_t

概念

ap_slotmem_instance_t は、Apache HTTP Server において固定サイズメモリ領域（スロット）を管理するためのデータ構造です。

ヘッダーファイル（include/ap_slotmem.h）においては、具体的な構造を隠蔽した不透明な（opaque）ハンドルとして定義されています。一方で、具体的なメモリ管理を行うプロバイダーモジュール（mod_slotmem_plain.c や mod_slotmem_shm.c）の内部では、それぞれの管理方式（プレーンメモリまたは共有メモリ）に適した異なるメンバを持つ構造体として定義されています。

なぜ重要か

この構造体は、メモリ管理の具体的な実装（単一プロセス内でのプレーンメモリ管理か、複数プロセス間での共有メモリ管理か）を抽象化するために重要です。

mod_proxy_balancer などの呼び出し元モジュールは、具体的なメモリレイアウトや排他制御の詳細を意識することなく、ap_slotmem_provider_t インターフェースを介して一貫した方法でスロットメモリの確保、アタッチ、読み書き（get/put/dptr）を行うことができます。

関与するモジュール

  • module:include : 抽象インターフェースおよび不透明なハンドルを定義します。
  • module:modules/slotmem : プレーンメモリ（mod_slotmem_plain.c）および共有メモリ（mod_slotmem_shm.c）による具体的な管理実装を提供します。

主要エンティティ

  • **symbol:include/ap_slotmem.h#ap_slotmem_instance_t**

    • 外部モジュール向けに公開されている不透明な構造体宣言。

  • **symbol:modules/slotmem/mod_slotmem_plain.c#ap_slotmem_instance_t**

    • 単一プロセス内でのスロット管理用構造体。データ領域の開始アドレス（base）や利用状況フラグ（inuse）、セグメントリスト（next）などを保持します。

  • **symbol:modules/slotmem/mod_slotmem_shm.c#ap_slotmem_instance_t**

    • 共有メモリ（SHM）ベースのスロット管理用構造体。apr_shm_t へのポインタ（shm）、永続化ファイルパス（pname）、空きスロット数（num_free）などを保持します。

エンドツーエンドの流れ

1. **初期化と生成**: 呼び出し元モジュールが ap_slotmem_provider_t の create または attach を呼び出します。 2. **インスタンス構築**: 各プロバイダー（plain または shm）は、それぞれの方式でメモリ領域を確保し、ap_slotmem_instance_t を初期化してそのポインタを返します。 3. **データ操作**: 呼び出し元は取得したインスタンスポインタを引数として get、put、dptr などのAPIに渡し、スロットデータの読み書きを行います。

*注意: 提供されたコンテキストからは、呼び出し元モジュール（mod_proxy_balancer 等）における具体的なライフサイクル遷移や、詳細なエンドツーエンドの完全な制御フローは未だ完全には明らかになっていません。*

重要な不変条件

  • **不透明性の維持**: 呼び出し元は ap_slotmem_instance_t の内部メンバに直接アクセスしてはならず、必ず ap_slotmem_provider_t の API を経由する必要があります。
  • **プールの生存期間**: インスタンス内の gpool（セグメントごとのプール）は、インスタンスの生存期間全体にわたって維持され、クリーンアップや再起動時のメモリ解放を制御します。
  • **共有メモリのレイアウト（mod_slotmem_shm）**: 共有メモリ領域は、[desc] [num_free] [data_slots] [inuse_array] の順序で厳密にアライメントされて配置される必要があります。

よくある落とし穴

  • **排他制御の欠如（plain 実装）**: mod_slotmem_plain.c による実装は単一プロセス内での利用を想定しており、マルチスレッド環境で安全に使用するためには呼び出し側での排他制御が必須です。
  • **境界チェックの委譲**: slotmem_get や slotmem_put において、転送元・転送先のバッファサイズ（dest_len / src_len）がスロットサイズ（size）を超えないように保護する責任は、呼び出し側に委ねられています。
  • **プラットフォーム固有の挙動（shm 実装）**: Windows 環境などのプロセス継承時において、is_child_process フラグとの相互作用により共有メモリの再アタッチ挙動が変化するため、プラットフォーム固有の考慮が必要です。
  • **データの破損**: mod_slotmem_shm の永続化データ復元（restore_slotmem）の際、MD5チェックサムが一致しない場合はデータが破損しているとみなされ、初期化が失敗します。

次に読む場所

  • include/ap_slotmem.h : API 定義と不透明なハンドルの宣言。
  • modules/slotmem/mod_slotmem_plain.c : プロセス内メモリ管理の具体的なデータ構造と実装。
  • modules/slotmem/mod_slotmem_shm.c : 共有メモリおよび永続化を伴う管理の具体的なデータ構造と実装。
