ap_slotmem_instance_t

概念

ap_slotmem_instance_t は、Apache HTTP Server において固定サイズメモリ領域（スロット）の集合を管理するための構造体です。

公開ヘッダー（ap_slotmem.h）においては、具体的なメンバを隠蔽した不透明なハンドル（opaque handle）として定義されています。一方で、具体的なプロバイダー実装である単一プロセス向けの mod_slotmem_plain.c や、共有メモリベースでプロセス間通信に対応する mod_slotmem_shm.c の内部において、それぞれの用途に応じた具体的な構造体メンバが定義されています。

なぜ重要か

この構造体は、mod_proxy_balancer などのモジュールが、動的な状態管理やプロセス間でのデータ共有を行うための共通インターフェースを提供します。

利用側モジュールは、具体的なメモリ確保方式（プレーンなメモリか、共有メモリか）の詳細を知ることなく、この不透明ハンドルを介して安全かつ統一的にスロットデータの取得（get）や更新（put）を行うことができます。

関与するモジュール

  • module:include (インターフェース定義: ap_slotmem.h)
  • module:modules/slotmem (プロバイダー実装: mod_slotmem_plain.c, mod_slotmem_shm.c)

主要エンティティ

  • symbol:include/ap_slotmem.h#ap_slotmem_instance_t
  • 外部モジュールに公開される不透明なハンドル定義。
  • symbol:modules/slotmem/mod_slotmem_plain.c#ap_slotmem_instance_t

  • symbol:modules/slotmem/mod_slotmem_shm.c#ap_slotmem_instance_t
  • 共有メモリ（SHM）および永続化ファイルを伴うプロセス間共有用の具体的な構造体定義。

エンドツーエンドの流れ

1. **初期化と作成/アタッチ**: 利用側モジュール（例: mod_proxy_balancer）が、ap_slotmem_provider_t の create または attach API を呼び出します。これにより、プロバイダー実装（mod_slotmem_plain または mod_slotmem_shm）が内部で ap_slotmem_instance_t を確保・初期化し、そのポインタを返します。 2. **データ操作**: 利用側は取得した ap_slotmem_instance_t のポインタを引数として、get、put、dptr などの API を呼び出します。

  • mod_slotmem_plain では、base アドレスからスロットIDに応じたオフセットを計算し、メモリ操作を行います。
  • mod_slotmem_shm では、共有メモリセグメント（apr_shm_t）の base アドレスを基準にオフセット計算を行い、プロセス間で安全にデータを読み書きします。

    • 3. **ライフサイクル管理**: 各インスタンスに関連付けられた gpool（メモリプール）を通じて、サーバーの再起動やMPMの世代交代に伴うメモリの解放・再確保が制御されます。

*注意: 利用側モジュール（mod_proxy_balancer 等）の具体的な呼び出しシーケンスの全体像や、詳細なエラーハンドリングの伝播など、エンドツーエンドの完全なフローは提供されたソースコードの範囲内では完全には網羅されていません。*

重要な不変条件

  • **カプセル化の維持**:

    • 外部モジュールからは直接メンバにアクセスせず、必ず ap_slotmem_provider_t の API を通じて操作しなければなりません。

  • **メモリプールの生存期間**:

    • 構造体内の gpool は、インスタンスのライフサイクル全体を通じて生存し、適切なクリーンアップを保証する必要があります。

  • **共有メモリのレイアウト（mod_slotmem_shm）**:

    • 共有メモリ領域は、[desc] [num_free] [data_slots] [inuse_array] の順序で厳密に配置され、AP_SLOTMEM_OFFSET 等の定数によるアライメント管理が維持される必要があります。

  • **スロット領域の確保**:

    • base ポインタは、常に num 個のスロット（各 size バイト）を格納するのに十分な領域を指していなければなりません。

よくある落とし穴

  • **境界チェックの欠如**:

    • mod_slotmem_plain における slotmem_get や slotmem_put では、コピー先・コピー元のバッファサイズ（dest_len, src_len）がスロットサイズ（size）を超えないようにする保護が呼び出し側に委ねられています。

  • **排他制御の欠如（プレーン実装）**:

    • mod_slotmem_plain は単一プロセス内での利用を想定しており、マルチスレッド環境で安全に使用するためには、呼び出し側で明示的な排他制御を行う必要があります。

  • **プラットフォーム固有の挙動**:

    • mod_slotmem_shm では、Windows環境などの is_child_process フラグの有無により、プロセス継承時の共有メモリへの再アタッチ挙動が変化するため、移植性の問題が発生しやすいです。

  • **データの破損**:

    • mod_slotmem_shm の永続化データ復元時（restore_slotmem）、MD5チェックサムが一致しない場合はデータ破損とみなされ、初期化が失敗します。
