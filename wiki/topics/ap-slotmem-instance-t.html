ap_slotmem_instance_t

概念

ap_slotmem_instance_t は、Apache HTTP Server (httpd) において固定サイズメモリ領域（スロット）を管理するための構造体です。

公開ヘッダーである include/ap_slotmem.h においては、具体的な内部構造を隠蔽した不透明なハンドル（Opaque Handle）として定義されています。一方で、具体的なメモリ管理方式を提供するプロバイダーモジュール（mod_slotmem_plain.c や mod_slotmem_shm.c）の内部において、それぞれの用途に応じた具体的な構造体メンバが定義されます。これにより、メモリの確保、アタッチ、およびデータ操作のライフサイクルが抽象化されています。

なぜ重要か

  • **実装の隠蔽と抽象化**: 利用者側（mod_proxy_balancer など）に対して、メモリの実態が単一プロセス内のプレーンメモリ（mod_slotmem_plain）であるか、プロセス間で共有される共有メモリ（mod_slotmem_shm）であるかを意識させず、統一された API を通じた操作を可能にします。
  • **プロセス間・スレッド間通信の基盤**: 特に mod_slotmem_shm においては、共有メモリ（SHM）のメタデータ、利用状況フラグ、データ領域へのポインタを統合管理し、複数プロセス間での安全な状態共有を支える中心的な役割を果たします。

関与するモジュール

  • module:include （インターフェース定義）
  • module:modules/slotmem （具体的なプロバイダー実装）

主要エンティティ

  • **symbol:include/ap_slotmem.h#ap_slotmem_instance_t**

    • 不透明なハンドルとしての不完全型定義。API 利用者はこのポインタを介して操作を行います。

  • **symbol:modules/slotmem/mod_slotmem_plain.c#ap_slotmem_instance_t**

    • 単一プロセス内でのスロット管理用構造体。セグメント名、データ開始アドレス（base）、スロットサイズ、スロット総数、利用状況フラグ（inuse）などを保持します。

  • **symbol:modules/slotmem/mod_slotmem_shm.c#ap_slotmem_instance_t**

    • 共有メモリ（SHM）および永続化ファイルをベースとしたスロット管理用構造体。apr_shm_t へのポインタ、空きスロット数、永続化データへのポインタ、スロット記述子（sharedslotdesc_t）などを保持します。

エンドツーエンドの流れ

1. **生成・アタッチ**: 利用者モジュールが ap_slotmem_provider_t の create または attach を呼び出します。

  • mod_slotmem_plain では、セグメントごとのプール（gpool）からメモリを確保し、インスタンスを初期化して globallistmem リストに登録します。
  • mod_slotmem_shm では、apr_shm_create 等を用いて共有メモリセグメントを確保・アタッチし、必要に応じて永続化ファイルからデータを復元します。

    • 2. **データアクセス**: get、put、dptr などの API を介して、インスタンス内の base アドレスとスロットID（item_id）からオフセットを計算し、対象スロットへのデータの読み書きや直接参照を行います。 3. **クリーンアップ**: サーバーの再起動や MPM の世代交代に伴い、gpool を通じてメモリの解放や再確保が制御されます。

*注意: 提供されたソースコードの範囲外にある、呼び出し元モジュール（mod_proxy_balancer など）の具体的なコードや、詳細なエラーハンドリングの全容など、エンドツーエンドの完全なシーケンスは未検証の部分があります。*

重要な不変条件

  • **直接アクセスの禁止**: 本構造体は不透明（opaque）であり、利用者は直接メンバにアクセスせず、必ず ap_slotmem_provider_t で定義された API を通じて操作しなければなりません。
  • **プールの生存期間**: インスタンス内の gpool は、インスタンスのライフサイクル全体を通じて生存し、クリーンアップを保証する必要があります。
  • **SHM のメモリレイアウト**: mod_slotmem_shm において、共有メモリは [desc] [num_free] [data_slots] [inuse_array] の順で厳密にレイアウトされ、アライメントが維持されている必要があります。

よくある落とし穴

  • **排他制御の欠如（Plain実装）**: mod_slotmem_plain は単一プロセス内での利用を想定しており、マルチスレッド環境における排他制御は呼び出し側に委ねられています。
  • **メモリ境界チェックの委譲**: slotmem_get や slotmem_put において、コピー先・コピー元のバッファサイズ（dest_len/src_len）がスロットサイズ（size）を超えないようにする保護は、呼び出し側の責任となります。
  • **データの破損（SHM）**: 永続化ファイルからの復元時（restore_slotmem）、MD5チェックサムが一致しない場合はデータが破損しているとみなされ、復元に失敗します。
  • **プラットフォーム固有の挙動**: Windows環境などにおける子プロセスへのメモリ再アタッチ（is_child_process フラグ等）の挙動に留意する必要があります。
