ap_slotmem_instance_t

概念

ap_slotmem_instance_t は、Apache HTTP Server において固定サイズメモリ領域（スロット）を管理するための構造体です。

ヘッダーファイル include/ap_slotmem.h においては、詳細が隠蔽された不透明なハンドル（opaque handle）として定義されています。一方で、具体的なメモリ管理方式を提供するプロバイダーモジュール（単一プロセス向けの mod_slotmem_plain.c や、共有メモリを用いる mod_slotmem_shm.c）の内部において、それぞれの用途に応じた具体的な構造体メンバが定義されています。

なぜ重要か

この構造体は、スロットベースのメモリ管理における「インスタンス」を抽象化します。

mod_proxy_balancer などの上位モジュールは、具体的なメモリ確保の仕組み（単一プロセス内のローカルメモリか、複数プロセス間で共有される共有メモリか）や排他制御の詳細を意識することなく、このハンドルを介して一貫した API（ap_slotmem_provider_t）から安全にデータ領域を操作できます。

関与するモジュール

  • module:include （インターフェース定義）
  • module:modules/slotmem （プロバイダー実装）

エンドツーエンドの流れ

提供された情報からは、システム全体における完全なエンドツーエンドのライフサイクルや詳細な呼び出しフローの全容は完全には明らかになっていません。判明している部分的な流れは以下の通りです。

1. **初期化と生成**: 利用側モジュールが ap_slotmem_provider_t の create または attach を呼び出します。プロバイダー側（mod_slotmem_plain または mod_slotmem_shm）は、それぞれの方式でメモリ領域を確保・アタッチし、ap_slotmem_instance_t 構造体を初期化してそのポインタを返します。 2. **データ操作**: 利用側は、取得した ap_slotmem_instance_t のポインタを引数 s または inst として get、put、dptr などの API に渡します。プロバイダーはインスタンス内の base アドレスとスロットID（item_id）からオフセットを計算し、対象メモリ領域への読み書きを実行します。 3. **クリーンアップ**: インスタンスに紐付けられたプール（gpool）を介して、サーバー再起動やMPMの世代交代に伴うメモリの解放・再確保が制御されます。

重要な不変条件

  • **不透明性の維持**: 利用側モジュールは本構造体のメンバに直接アクセスしてはならず、必ず ap_slotmem_provider_t の関数ポインタを介して操作する必要があります。
  • **プールの生存期間**: 構造体内の gpool（セグメントごとのプール）は、インスタンスのライフサイクル全体を通じて生存し、クリーンアップ処理を保証する必要があります。
  • **共有メモリのレイアウト（SHMの場合）**: mod_slotmem_shm において、共有メモリは [desc] [num_free] [data_slots] [inuse_array] の順で厳密にアライメント管理されて配置される必要があります。
  • **有効なインスタンスの前提**: create または attach が正常に成功した後にのみ、他の API 操作の引数として有効に機能します。

よくある落とし穴

  • **排他制御の欠如（Plain実装）**:

    • mod_slotmem_plain.c による実装は単一プロセス内での利用を想定しており、マルチスレッド環境におけるスレッド間の排他制御は呼び出し側に委ねられています。

  • **境界チェックの呼び出し側依存**:

    • slotmem_get や slotmem_put において、転送元・転送先のバッファサイズ（dest_len や src_len）がスロットサイズ（size）を超えないように保護する責任は呼び出し側にあります。

  • **プラットフォーム固有の挙動（SHM実装）**:

    • Windows環境などにおける is_child_process フラグとの相互作用により、子プロセス継承時の共有メモリ再アタッチ挙動が変化するため、プラットフォームごとの差異に留意する必要があります。

  • **定義の二重管理**:

    • 本構造体はヘッダーファイルと各実装ファイルの両方で定義されています。構造体のレイアウトや定義を変更する場合は、すべてのプロバイダー実装との整合性を維持する必要があります。
