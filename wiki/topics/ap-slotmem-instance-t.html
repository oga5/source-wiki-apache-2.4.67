ap_slotmem_instance_t

概念

ap_slotmem_instance_t は、Apache HTTP Server（httpd-2.4.67）において、固定サイズメモリ領域（スロット）を管理するための構造体です。

ヘッダーファイル include/ap_slotmem.h においては、具体的な内部構造を隠蔽した不透明なハンドル（Opaque Handle）として定義されています。その具体的な実装は、メモリの確保・管理方式を提供する各プロバイダーモジュール（mod_slotmem_plain.c や mod_slotmem_shm.c）ごとに定義されています。

なぜ重要か

この構造体は、共有メモリやローカルメモリのライフサイクル（確保、アタッチ、解放）を抽象化します。

mod_proxy_balancer などの呼び出し元モジュールは、具体的なメモリ管理手法（単一プロセス内のプレーンメモリか、複数プロセス間で共有する共有メモリか）を意識することなく、ap_slotmem_provider_t が提供する一貫したAPIを介してスロットデータの操作（取得、更新、直接参照など）を行うことができます。

関与するモジュール

  • module:include （インターフェース定義）
  • module:modules/slotmem （具体的なプロバイダー実装）

主要エンティティ

  • **symbol:include/ap_slotmem.h#ap_slotmem_instance_t**

    • 不透明なハンドルとしての宣言。APIの引数や戻り値として使用されます。

  • **symbol:modules/slotmem/mod_slotmem_plain.c#ap_slotmem_instance_t**

    • 単一プロセス内での固定サイズメモリ領域を管理するための具体的な構造体定義。

  • **symbol:modules/slotmem/mod_slotmem_shm.c#ap_slotmem_instance_t**

    • 共有メモリ（SHM）をベースとした、プロセス間で共有可能な固定長スロットを管理するための具体的な構造体定義。

エンドツーエンドの流れ

1. **作成・アタッチ**: 呼び出し元モジュールが ap_slotmem_provider_t の create または attach を呼び出します。

  • mod_slotmem_plain では、セグメントごとのグローバルプール（gpool）からメモリを確保し、インスタンスを初期化して静的リスト globallistmem に追加します。
  • mod_slotmem_shm では、apr_shm_t を用いて共有メモリセグメントを確保・アタッチし、メタデータや利用状況フラグテーブル（inuse）を初期化します。

    • 2. **データ操作**: 呼び出し元は、取得した ap_slotmem_instance_t のポインタを get、put、dptr などのAPIに渡します。プロバイダーは指定されたスロットID（item_id）からオフセットを計算し、対象のメモリ領域に対してデータの読み書きや直接参照を提供します。 3. **クリーンアップ**: サーバーの再起動やMPMの世代交代に伴い、gpool を通じてメモリの解放や再確保が制御されます。

*注意: 呼び出し元モジュール（例: mod_proxy_balancer）における具体的な呼び出しシーケンスや、詳細なプロセス間同期（apr_global_mutex 等）のライフサイクル全体の流れは、提供されたコンテキストからは完全には特定されていません。*

重要な不変条件

  • **カプセル化の維持**:

    • 呼び出し元は ap_slotmem_instance_t の内部メンバに直接アクセスしてはならず、必ず ap_slotmem_provider_t の関数ポインタを介して操作しなければなりません。

  • **プールの生存期間**:

    • 構造体内の gpool（プール）は、インスタンスのライフサイクル全体を通じて生存し、クリーンアップを保証する必要があります。

  • **共有メモリのレイアウト（SHMの場合）**:

    • 共有メモリ領域は、[desc] [num_free] [data_slots] [inuse_array] の順で厳密にアライメントされて配置される必要があります。

よくある落とし穴

  • **境界チェックの欠如**:

    • slotmem_get や slotmem_put において、転送元・転送先のバッファサイズ（dest_len や src_len）がスロットサイズ（size）を超えないようにする保護は、呼び出し側に委ねられています。

  • **スレッド安全性の欠如（Plain実装）**:

    • mod_slotmem_plain は単一プロセス内での利用を想定しており、マルチスレッド環境で利用する場合は呼び出し側で明示的な排他制御を行う必要があります。

  • **プラットフォーム固有の挙動（SHM実装）**:

    • Windows環境などにおける is_child_process フラグの有無により、子プロセスへの継承時のメモリ再アタッチ挙動が変化するため、プラットフォームごとの差異に留意する必要があります。

  • **定義の二重管理**:

    • 本構造体はヘッダーと各Cファイルの両方で定義されているため、構造を変更する際はすべてのプロバイダー実装との整合性を維持する必要があります。
