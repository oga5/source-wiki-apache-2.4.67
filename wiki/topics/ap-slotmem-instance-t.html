ap_slotmem_instance_t

概念

ap_slotmem_instance_t は、Apache HTTP Server（httpd）において、固定サイズに分割されたメモリ領域（スロット）を管理するためのデータ構造です。

公開ヘッダー（include/ap_slotmem.h）においては、具体的な内部構造を隠蔽した不透明なハンドル（Opaque Handle）として定義されています。一方で、その具体的な実装はプロバイダーモジュールごとに異なり、単一プロセス内でのメモリ管理を行う mod_slotmem_plain.c や、プロセス間共有メモリを管理する mod_slotmem_shm.c において、それぞれ異なるメンバを持つ構造体として定義されています。

なぜ重要か

この構造体は、mod_proxy_balancer などの他のモジュールが、プロセス間またはプロセス内でデータを安全かつ効率的に共有・管理するための共通インターフェースを提供します。

不透明なハンドルとして抽象化されているため、呼び出し側はメモリが共有メモリ（SHM）として確保されているか、あるいは単一プロセス内のプレーンなメモリとして確保されているかを意識することなく、同一の API（ap_slotmem_provider_t）を介してスロットの取得（get）、保存（put）、直接参照（dptr）などの操作を行うことができます。

エンドツーエンドの流れ

1. **初期化と生成**: mod_proxy_balancer などの呼び出し元モジュールが、ap_slotmem_provider_t の create または attach を呼び出します。 2. **インスタンスの構築**:

  • **Plain実装**: gpool からメモリを確保し、base アドレスやスロット数 num、利用状況フラグ inuse を初期化して globallistmem リストに登録します。
  • **SHM実装**: apr_shm_create 等を用いて共有メモリセグメントを確保し、[desc] [num_free] [data_slots] [inuse_array] の順でレイアウトされた共有メモリ領域へのポインタを base に設定します。必要に応じて永続化ファイルからデータを復元（MD5チェックサム検証を伴う）します。

    • 3. **データ操作**: 呼び出し元は、取得した ap_slotmem_instance_t のポインタを引数として get、put、dptr などの API を呼び出し、スロットデータの読み書きを行います。 4. **クリーンアップ**: サーバーの再起動やMPMの世代交代、プロセス終了時に、gpool を通じてメモリ領域や共有メモリセグメントが解放されます。

*注意: 呼び出し元モジュール（mod_proxy_balancer 等）における具体的な利用シナリオや、スレッド間・プロセス間での完全な同期・排他制御のライフサイクル全体（エンドツーエンドのフロー）の詳細は、提供されたコンテキストからは完全には特定されていません。*

重要な不変条件

  • **不透明性の維持**: 呼び出し側からは ap_slotmem_instance_t の内部メンバに直接アクセスしてはならず、必ず ap_slotmem_provider_t の API を経由しなければなりません。
  • **有効なライフサイクル**: インスタンスは create または attach が正常に成功した後にのみ、他の API の引数として使用可能です。
  • **メモリレイアウト（SHM）**: mod_slotmem_shm において、共有メモリは [desc] [num_free] [data_slots] [inuse_array] の順で厳密にレイアウトされ、AP_SLOTMEM_OFFSET 等の定数によるアライメントが維持される必要があります。
  • **プールの生存期間**: gpool（セグメントごとのプール）は、インスタンスの生存期間全体にわたって有効であり、適切なタイミングでのクリーンアップを保証します。

よくある落とし穴

  • **排他制御の欠如（Plain実装）**: mod_slotmem_plain は単一プロセス内での利用を想定しており、マルチスレッド環境で利用する場合は、呼び出し側で適切な排他制御を行う必要があります。
  • **メモリ境界チェックの欠如**: slotmem_get や slotmem_put において、転送元・転送先のバッファサイズ（dest_len や src_len）がスロットサイズ（size）を超えないようにする保護は、呼び出し側の責任に委ねられています。
  • **データの破損（SHM実装）**: mod_slotmem_shm の起動時（restore_slotmem 実行時）に、永続化データのMD5チェックサムが一致しない場合、データ破損とみなされて復元に失敗します。
  • **ヘッダーと実装の二重定義**: 本構造体は ap_slotmem.h と各実装ファイル（mod_slotmem_plain.c、mod_slotmem_shm.c）の両方で定義されています。構造体や関連する型を変更する際は、すべての実装間での整合性に細心の注意を払う必要があります。
