Authentication and Authorization Provider Framework

概念

Authentication and Authorization Provider Framework は、Apache HTTP Server における認証（Authentication: authn）および認可（Authorization: authz）の仕組みをモジュール化し、コアサーバーと個々の認証・認可方式を疎結合に連携させるための共通フレームワークです。

このフレームワークは、共通のインターフェース構造体（authn_provider および authz_provider）を定義し、各プロバイダーモジュールがこれらを実装・登録することで、Basic認証やDigest認証、各種アクセス制御（Require ディレクティブなど）を統一的な方法で処理できるようにします。

なぜ重要か

このフレームワークにより、Apache HTTP Server は特定の認証・認可方式に依存することなく、多様な認証ソースや認可ルールをプラグイン形式で柔軟に切り替えることができます。

例えば、Basic認証の仲介を行う authenticate_basic_user は、具体的なパスワード検証ロジック（ファイル、データベース、LDAPなど）を直接知る必要がありません。登録された authn_provider の check_password や get_realm_hash を呼び出すだけで検証を完了できるため、システムの拡張性と保守性が大幅に向上します。

関与するモジュール

提供された情報に基づき、以下のモジュールがこのフレームワークに関与しています。

  • module:include : 共通のプロバイダーインターフェース（mod_auth.h）を定義します。
  • module:modules/aaa : 認証・認可のコアロジック（mod_authn_core.c, mod_authz_core.c）や、具体的な認証方式（mod_auth_basic.c）を実装します。

主要エンティティ

本フレームワークを構成する主要なエンティティは以下の通りです。

  • **symbol:include/mod_auth.h#authn_provider** (struct)

    • 認証プロバイダーが実装すべきインターフェース。パスワード検証を行う check_password や、レルムベースのハッシュを取得する get_realm_hash などの関数ポインタを定義します。

  • **symbol:include/mod_auth.h#authz_provider** (struct)

    • 認可プロバイダーが実装すべきインターフェース。Require 設定行を解析する parse_require_line や、アクセス可否を判定する check_authorization などの関数ポインタを定義します。

  • **symbol:modules/aaa/mod_auth_basic.c#authenticate_basic_user** (function)

    • HTTP Basic 認証において、設定された認証プロバイダーを順次呼び出してユーザー資格情報の検証を仲介する関数。

  • **symbol:modules/aaa/mod_authn_core.c#create_authn_core_dir_config** (function)

    • ディレクトリ単位の認証設定（authn_core_dir_conf）をメモリ上に確保し、初期化する関数。

  • **symbol:modules/aaa/mod_authz_core.c#authorize_user** (function)

    • ユーザーの認可処理を行う関数。

エンドツーエンドの流れ

提供された情報から確認できる部分的な処理の流れは以下の通りです。

1. **設定・初期化フェーズ**

  • サーバー起動時または設定再読み込み時に、create_authn_core_dir_config が呼び出され、ディレクトリ単位の認証設定構造体がメモリプールから確保・初期化されます。
  • 認可プロバイダーの parse_require_line が呼び出され、設定ファイル内の Require ディレクティブの構文チェックと事前解析が行われます。

    • 2. **認証フェーズ**

  • クライアントからリクエストを受信した際、AuthType が Basic に設定されていれば、認証フック経由で authenticate_basic_user が呼び出されます。
  • authenticate_basic_user はリクエストヘッダーからユーザー名とパスワードを抽出し、登録されている authn_provider の check_password または get_realm_hash を呼び出して資格情報を検証します。

    • 3. **認可フェーズ**

  • 認証成功後、authorize_user を介して認可処理が実行され、対応する authz_provider の check_authorization が呼び出されてアクセス可否が判定されます。

*注意: 認証・認可の完全なライフサイクルや、authorize_user の詳細な内部アルゴリズム、および他プロバイダーとの統合フローの全容は、提供された情報からは完全には明らかになっていません。*

重要な不変条件

  • **プロバイダーの戻り値**: authn_provider の各関数は、認証の成否に応じて必ず適切な authn_status 型の値を返す必要があります。
  • **Basic認証の必須設定**: authenticate_basic_user が動作するためには、AuthType が Basic であり、かつ AuthName が設定されている必要があります。
  • **設定解析のタイミング**: authz_provider の parse_require_line は設定読み込み時に一度だけ実行され、リクエスト処理時の負荷を軽減するために解析結果を parsed_require_line に格納します。

*注意: フレームワーク全体におけるスレッドセーフ性や、メモリ管理に関するすべての不変条件は、提供された情報からは完全には明らかになっていません。*

よくある落とし穴

  • **AuthName の設定漏れ**

    • Basic認証を使用する際、設定ファイル内で AuthName が指定されていないと、authenticate_basic_user は HTTP_INTERNAL_SERVER_ERROR を返して処理を中断します。

  • **authoritative 設定による意図しない拒否または委譲**

    • authenticate_basic_user において、authoritative（権限委譲）設定が有効な場合は認証失敗時に即座に HTTP_UNAUTHORIZED を返しますが、無効な場合は DECLINED を返して後続の認証モジュールへ処理を委譲します。この挙動を誤解すると、セキュリティホールの原因や意図しないアクセス拒否につながります。

  • **関数ポインタの NULL ハンドリング**

    • authz_provider の parse_require_line など、一部の関数ポインタは NULL を許容するため、呼び出し側で適切に NULL チェックを行わずに呼び出すとセグメンテーションフォールトを引き起こす可能性があります。

次に読む場所

  • **include/mod_auth.h**

    • authn_provider および authz_provider の構造体定義と、提供すべきインターフェースの詳細。

  • **modules/aaa/mod_auth_basic.c**

    • Basic認証におけるプロバイダーの呼び出しと仲介処理（authenticate_basic_user）の実装。

  • **modules/aaa/mod_authn_core.c**

    • 認証コアの設定管理（create_authn_core_dir_config）の実装。

  • **modules/aaa/mod_authz_core.c**

    • 認可コアの処理（authorize_user）の実装。
