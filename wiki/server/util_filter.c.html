server/util_filter.c

概要

Apache HTTP Server のフィルタチェーン機構を実装するコアモジュールです。HTTP リクエストおよびレスポンスの処理過程において、データを変換・加工するフィルタの登録、管理、およびチェーンへの挿入・実行を担います。

モジュール内での役割

server モジュールの一部として、include/util_filter.h で定義されたフィルタインターフェースの実装を提供します。フィルタは request_rec や conn_rec に紐付き、データ（バケットブリゲード）を順次処理するパイプラインを形成します。

責務

  • **フィルタ登録**: フィルタ名とコールバック関数をトライ木構造で管理し、名前解決を高速化する。
  • **チェーン管理**: フィルタの優先度（ftype）に基づき、適切な位置へフィルタを挿入・削除する。
  • **データ転送**: ap_get_brigade および ap_pass_brigade を通じて、フィルタ間でバケットブリゲードを伝播させる。
  • **ユーティリティ**: フィルタ内での文字列出力、フラッシュ処理、ブリゲードの退避（setaside）などの補助機能を提供。

重要なシンボル

フィルタ管理・登録

  • registered_output_filters / registered_input_filters: フィルタ名を保持するトライ木のルート。
  • ap_register_output_filter / ap_register_input_filter: フィルタをシステムに登録する。
  • ap_get_output_filter_handle / ap_get_input_filter_handle: 名前からフィルタ定義（ap_filter_rec_t）を取得する。

チェーン操作

  • ap_add_output_filter / ap_add_input_filter: フィルタをチェーンに追加する。
  • ap_remove_output_filter / ap_remove_input_filter: チェーンからフィルタを削除する。
  • add_any_filter_handle: フィルタの型（ftype）に基づき、適切な挿入位置を決定する内部関数。

データ処理

  • ap_get_brigade: 入力フィルタチェーンからデータを読み込む。
  • ap_pass_brigade: 出力フィルタチェーンへデータを渡す。
  • ap_save_brigade: フィルタがデータを保持するためにブリゲードをコピー・退避させる。

制御フローとデータフロー

1. **登録**: register_filter がフィルタ名をトライ木に格納し、ap_filter_rec_t を生成します。 2. **挿入**: add_any_filter_handle が INSERT_BEFORE マクロを使用して、フィルタの ftype に基づく順序でチェーンを構築します。 3. **実行**:

  • 出力の場合、ap_pass_brigade が現在のフィルタの out_func を呼び出し、結果を次のフィルタへ渡します。
  • 入力の場合、ap_get_brigade が in_func を呼び出し、下位フィルタからデータを吸い上げます。

    • 4. **エラー**: ap_pass_brigade_fchk は、フィルタエラーを検知し、ログ出力と HTTP ステータスコードへの変換を行います。

読者向けメモ

  • **メモリ管理**: フィルタのコンテキストや構造体は、ftype に応じてリクエストプール（r->pool）またはコネクションプール（c->pool）に割り当てられます。
  • **トライ木**: フィルタ名の検索はトライ木で行われ、TRIE_INITIAL_SIZE で初期化されます。
  • **制約**: INSERT_BEFORE マクロは、フィルタの型（ftype）が小さいほどチェーンの先頭に近いという前提で順序を決定します。
  • **省略**: 本ファイルはフィルタの「管理」と「伝播」に特化しており、個別のフィルタロジック（圧縮、チャンク化など）は含まれていません。
  • **注意点**: ap_pass_brigade 内で EOS（End of Stream）バケットを処理する際、リクエストの eos_sent フラグを更新する副作用があるため、フィルタの順序には注意が必要です。
