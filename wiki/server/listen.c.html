server/listen.c

概要

Apache HTTP Server におけるネットワークリスナーの管理、ソケットのバインド、および設定を行うモジュールです。サーバー起動時のソケット作成から、設定ファイルで指定された Listen ディレクティブの処理、マルチコア環境でのリスナーバケットの複製までを担います。

モジュール内での役割

server モジュールのサブコンポーネントとして、MPM（Multi-Processing Module）が接続を受け入れるための「入り口」を準備します。http_main.c や各 MPM から呼び出され、OS レベルのソケット状態と Apache の内部的なリスナー構造体 (ap_listen_rec) を同期させます。

責務

  • **リスナー管理**: Listen ディレクティブによるリスナーの登録と、サーバー再起動時の古いリスナーのクリーンアップ。
  • **ソケット設定**: SO_REUSEADDR, SO_KEEPALIVE, TCP_NODELAY, SO_SNDBUF, SO_RCVBUF などのソケットオプションの適用。
  • **マルチコア最適化**: SO_REUSEPORT を利用したリスナーバケットの複製による負荷分散。
  • **システム統合**: systemd ソケットアクティベーションのサポート。
  • **フィルタ適用**: AcceptFilter の設定と適用。

重要なシンボル

リスナー管理

  • ap_listeners: 現在アクティブなリスナーの連結リスト。
  • ap_listen_rec: リスナー情報を保持する構造体。ソケット、バインド先アドレス、プロトコル、次へのポインタを含む。
  • ap_setup_listeners(): 設定に基づき全リスナーを初期化・バインドするメイン関数。
  • ap_duplicate_listeners(): マルチコア環境用にリスナーを複製する。

設定・状態変数

  • ap_listenbacklog, ap_listencbratio: 接続待ち行列やコア比率の設定値。
  • ap_have_so_reuseport: プラットフォームが SO_REUSEPORT をサポートしているかのフラグ。

内部ヘルパー

  • make_sock(): ソケットの作成、オプション設定、バインド、リスンを行う低レベル関数。
  • alloc_listener(): 設定ファイルからリスナーを解析・生成する。
  • ap_apply_accept_filter(): OS 固有のアクセプトフィルタ（defer_accept 等）を適用する。

制御フローとデータフロー

1. **設定フェーズ**: ap_set_listener() が設定ファイルを読み込み、alloc_listener() を通じて ap_listeners リストを構築します。 2. **初期化フェーズ**: ap_setup_listeners() が呼ばれ、open_listeners() が各リスナーに対して make_sock() を実行します。 3. **複製フェーズ**: ap_duplicate_listeners() が呼ばれると、ap_listen_buckets にリスナーのコピーが作成され、マルチプロセス/スレッド間での負荷分散が準備されます。 4. **終了フェーズ**: ap_close_listeners() がプロセス終了時や再起動時にソケットをクローズします。

読者向けメモ

  • **メモリ所有権**: リスナー構造体は process->pool に割り当てられ、サーバーのライフサイクル全体で生存します。
  • **不変条件**: ap_listeners は常に Listen ディレクティブの順序を維持しようとしますが、IPv6/IPv4 のバインド順序調整のためにリストの並び替えが行われることがあります。
  • **エッジケース**: SO_REUSEPORT の挙動は Linux と BSD で異なるため、コード内でプラットフォーム判定を行っています。
  • **省略箇所**: systemd 関連のコードは HAVE_SYSTEMD マクロで囲まれており、環境依存のロジックが多いため、詳細なシステムコールフローは省略しています。
  • **注意点**: make_sock は APR 抽象化が不十分な箇所があり、OS 固有のソケットオプション設定が直接記述されている箇所があります。
