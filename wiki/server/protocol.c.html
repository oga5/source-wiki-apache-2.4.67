server/protocol.c

概要

HTTP プロトコルの解析、リクエストの構築、およびクライアントとの直接的な通信を担う中核ファイルです。リクエストラインの読み取り、MIME ヘッダーの解析、コンテンツタイプの生成、認証の補助、およびプロトコルスイッチングの制御を行います。

モジュール内での役割

module:server における通信層の最上位に位置します。apr_bucket_brigade を介してフィルタチェーンと連携し、低レベルなバイトストリームを HTTP リクエスト構造体 (request_rec) に変換します。

責務

  • **リクエスト解析**: リクエストラインおよび MIME ヘッダーの読み取りと検証。
  • **リクエスト構築**: request_rec の初期化と、サブリクエスト用のプロトコル状態の管理。
  • **レスポンス補助**: コンテンツタイプの設定、最終更新時刻の合理化、認証失敗の通知。
  • **プロトコル管理**: プロトコルの選択、アップグレード、およびスイッチングのフック管理。
  • **出力フィルタリング**: コンテンツ長計算フィルタ (ap_content_length_filter) および古い書き込みインターフェースのバッファリング。

重要なシンボル

リクエスト処理

  • ap_read_request: リクエストのライフサイクルを開始するメインエントリポイント。
  • ap_create_request: request_rec のメモリ確保と初期化。
  • ap_parse_request_line: リクエストラインの解析と検証。
  • ap_get_mime_headers_core: MIME ヘッダーの読み取りと headers_in テーブルへの格納。

プロトコル・ユーティリティ

  • ap_rgetline_core: フィルタから 1 行ずつ読み取る低レベル関数。
  • ap_make_content_type: Content-Type ヘッダーの生成と charset 付与。
  • ap_rationalize_mtime: Last-Modified 時刻の妥当性チェック。

認証・認可

  • ap_note_auth_failure: 認証失敗時のフック実行。
  • ap_get_basic_auth_pw: Basic 認証情報の抽出とデコード。

プロトコルスイッチング

  • ap_select_protocol, ap_switch_protocol: ALPN 等によるプロトコルネゴシエーションと切り替え。

制御フローとデータフロー

1. **リクエスト開始**: ap_read_request が ap_create_request で構造体を生成し、read_request_line で入力を待ちます。 2. **入力フィルタ**: ap_rgetline_core が r->proto_input_filters を通じて apr_bucket_brigade からデータを取得します。 3. **ヘッダー解析**: ap_get_mime_headers_core がヘッダーを読み取り、apr_table_addn で headers_in に格納します。 4. **出力フィルタ**: ap_content_length_filter がレスポンスボディの長さを計算し、必要に応じて Content-Length ヘッダーを挿入します。 5. **終了**: ap_finalize_request_protocol が EOS (End of Stream) バケットを挿入し、リクエストを完了させます。

読者向けメモ

  • **メモリ管理**: ほとんどの動的メモリは request_rec に紐付いた r->pool から確保されます。リクエスト終了時に自動的に解放されます。
  • **EBCDIC 対応**: ap_rgetline は EBCDIC 環境下で自動的に ASCII 変換を行うラッパーとして機能します。
  • **厳密な解析**: http_conformance 設定により、RFC 7230 に準拠した厳格な解析と、レガシーな「緩い」解析が切り替わります。
  • **省略された詳細**: 本ファイルは非常に広範なため、特定の認証スキーム（Digest 等）の詳細なロジックや、各 MPM 固有のフィルタ挙動については、関連するモジュール側の実装を参照してください。
