server/connection.c

概要

HTTP 接続のライフサイクル管理を担うファイルです。接続の確立から、プロトコル処理の実行、そして終了時の「リンガリング・クローズ（lingering close）」処理までを定義しています。

モジュール内での役割

server モジュールにおいて、ネットワーク接続と Apache の内部表現（conn_rec）を橋渡しする中心的な役割を果たします。MPM（Multi-Processing Module）が受け取ったソケットを、Apache のフィルタチェーンやプロトコルハンドラへ引き渡すためのインターフェースを提供します。

責務

  • **接続フックの管理**: 接続の生成、処理、終了前後のフックポイントを提供し、モジュールが接続状態に介入できるようにする。
  • **リンガリング・クローズの実装**: クライアントとの通信終了時に、TCP の RST パケットによるデータ欠損を防ぐため、残存データの読み込みと正常な終了処理を行う。
  • **接続のフラッシュとシャットダウン**: EOC（End Of Connection）バケットを挿入し、フィルタチェーンを介して接続を安全に閉じる。

重要なシンボル

  • ap_process_connection: 接続処理のメインエントリポイント。バーチャルホストの特定と接続フックの実行を行う。
  • ap_lingering_close: 接続を閉じる際の最終処理。タイムアウトを設けつつ、クライアントからの残存データを読み捨てることで TCP の正常終了を保証する。
  • ap_shutdown_conn: 接続に対してフラッシュと EOC バケットを送信し、出力フィルタをクリーンアップする。
  • APR_HOOK_...: 接続の各フェーズ（pre_connection, process_connection 等）を拡張可能にするためのフック定義。

制御フローとデータフロー

1. **開始**: ap_process_connection が呼ばれ、ap_pre_connection フックを経て ap_run_process_connection が実行されます。 2. **処理**: プロトコルハンドラがリクエストを処理します。 3. **終了**: 接続終了時、ap_lingering_close が呼び出されます。 4. **リンガリング**:

  • ap_start_lingering_close で書き込み側をシャットダウン（FIN 送信）。
  • do-while ループでソケットから残存データを読み込みます。
  • MAX_SECS_TO_LINGER（デフォルト30秒）または SECONDS_TO_LINGER（2秒）のタイムアウトまで待機し、最終的に apr_socket_close でソケットを解放します。

読者向けメモ

  • **リンガリングの重要性**: このファイルで最も複雑なのは ap_lingering_close です。これは HTTP の仕様上、サーバーが先に接続を閉じるとクライアントがまだ送信中のデータを破棄してしまうリスクがあるため、TCP レベルでの整合性を保つための「お作法」です。
  • **フックの活用**: 独自のプロトコルや接続監視モジュールを作成する場合、ここで定義されているフック（pre_connection 等）が主要な拡張ポイントとなります。
  • **デバッグ**: 接続が途中で切れる、あるいは終了処理が遅い場合は、ap_lingering_close 内のタイムアウト設定や short-lingering-close ノートの有無を確認してください。
