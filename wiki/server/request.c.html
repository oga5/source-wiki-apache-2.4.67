server/request.c

概要

server/request.c は、Apache HTTP Server におけるリクエスト処理のライフサイクルと、ディレクトリ/ファイル単位の設定マージ（Walk）を管理する中核ファイルです。リクエストの正規化、翻訳、認証、認可、およびサブリクエストの生成といった一連のフェーズを制御します。

モジュール内での役割

module:server の一部として、HTTP リクエストが接続からハンドラへ渡されるまでの「前処理」を担当します。core モジュールの設定をリクエストのコンテキストに適用し、リクエストのパスに基づいた設定の階層的なマージを最適化する役割を担います。

責務

  • **リクエスト処理のオーケストレーション**: ap_process_request_internal を通じて、URI の正規化から認可までの各フェーズを順次実行する。
  • **設定の階層的マージ**: <Directory>, <Location>, <Files>, <If> セクションの設定をリクエストパスに従ってマージする。
  • **サブリクエスト管理**: サブリクエストの生成、実行、およびメモリ管理（プール破棄）を行う。
  • **認証・認可の最適化**: 設定の変更がない場合に、以前の計算結果を再利用するためのキャッシュ機構を提供。
  • **シンボリックリンクの制御**: FollowSymLinks や SymLinksIfOwnerMatch に基づくファイルシステムアクセスの検証。

重要なシンボル

リクエスト処理

  • ap_process_request_internal: リクエスト処理のメインロジック。URI 正規化、翻訳、アクセス制御、型チェック、修正（fixups）を順次実行。
  • ap_some_authn_required: 認証が必要かどうかを判定する。

設定マージ（Walk）

  • ap_directory_walk: ファイルシステムパスに基づき <Directory> 設定をマージ。
  • ap_location_walk: URI パスに基づき <Location> 設定をマージ。
  • ap_file_walk: ファイル名に基づき <Files> 設定をマージ。
  • ap_if_walk: <If> 条件式を評価し設定をマージ。
  • walk_cache_t: マージ結果をキャッシュし、サブ/リダイレクト時の再計算を回避する構造体。

サブリクエスト

  • make_sub_request: サブリクエスト用の request_rec を初期化。
  • ap_sub_req_lookup_uri / ap_sub_req_lookup_file: URI またはファイルパスからサブリクエストを生成。
  • ap_run_sub_req: サブリクエストを実行し、ハンドラを呼び出す。

制御フローとデータフロー

1. **初期化**: ap_process_request_internal が呼ばれると、URI の正規化が行われる。 2. **Walk フェーズ**: ap_directory_walk 等が呼ばれ、walk_cache_t を介して設定がマージされる。キャッシュが存在すれば、dir_conf_tested を比較して再利用する。 3. **認証・認可**: ap_satisfies に基づき、access_checker や check_user_id などのフックが実行される。 4. **サブリクエスト**: make_sub_request で親リクエストのプールから派生した新しいプールを持つ request_rec が作成され、独立した処理フローが実行される。

読者向けメモ

  • **メモリ管理**: サブリクエストは r->pool を使用して作成され、ap_destroy_sub_req で明示的に破棄する必要があります。メモリリークを防ぐため、サブリクエスト生成後は必ず破棄してください。
  • **キャッシュの重要性**: walk_cache_t はパフォーマンスに直結します。設定の変更頻度が高い環境では、このキャッシュのヒット率が重要になります。
  • **注意点**: ap_process_request_internal 内のロジックはセキュリティモデルの根幹です。この関数内の順序を変更すると、認可バイパスなどの脆弱性を生む可能性があるため、慎重な変更が求められます。
  • **省略された詳細**: 正規表現のコンパイルや、特定の MPM 固有の最適化については、このファイルでは抽象化されており、各モジュール側の実装に依存しています。
