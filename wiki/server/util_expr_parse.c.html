server/util_expr_parse.c

概要

Apache HTTP Server の式評価エンジン（ap_expr）における構文解析器の実装です。GNU Bison 2.5 によって生成された LALR(1) パーサーであり、util_expr_parse.y から生成されています。式言語の文法定義に基づき、トークン列を抽象構文木（AST）へと変換する役割を担います。

モジュール内での役割

server モジュール内の式評価サブシステムの一部です。util_expr_scan.c（字句解析器）からトークンを受け取り、util_expr_eval.c で実行される AST を構築します。

責務

  • **構文解析**: ap_expr_yyparse を通じて、入力文字列を文法規則に従って解析する。
  • **AST 構築**: ap_expr_make などのヘルパー関数を呼び出し、ノードを生成してツリーを組み立てる。
  • **エラーハンドリング**: 構文エラー発生時に yyerror を呼び出し、コンテキスト構造体にエラーメッセージを格納する。
  • **リソース管理**: 解析中に使用するメモリを apr_pool_t を介して管理する。

重要なシンボル

解析制御

  • ap_expr_yyparse: 解析のメインエントリポイント。
  • ap_expr_yylex: 字句解析器を呼び出すインターフェース。
  • ap_expr_yyerror: 構文エラー発生時に呼び出されるコールバック。

データ構造

  • YYSTYPE: パーサーのスタックで使用される共用体。cpVal（文字列）、exVal（AST ノード）、num（数値）を保持。
  • ap_expr_parse_ctx_t: 解析コンテキスト。AST のルート、エラーメッセージ、メモリプール、スキャナー状態を保持する。

トークン定義

  • yytokentype: 式言語の演算子（T_OP_EQ 等）、変数（T_VAR_BEGIN 等）、リテラル（T_STRING 等）を定義する列挙型。

制御フローとデータフロー

1. **初期化**: 呼び出し元が ap_expr_parse_ctx_t を準備し、ap_expr_yyparse を呼び出す。 2. **トークン取得**: yyparse が yylex を呼び出し、入力からトークンを順次消費する。 3. **還元 (Reduction)**: 文法規則に一致すると、対応するアクションが実行される。ここで ap_expr_make 等が呼ばれ、AST ノードが生成される。 4. **AST 構築**: 生成されたノードは YYSTYPE を通じて上位の規則へ渡され、最終的に ctx->expr にルートノードが格納される。 5. **終了**: 解析成功時は 0、失敗時は 1 または 2（メモリ不足等）を返す。

直接依存

  • util_expr_private.h: 内部的なデータ構造定義と ap_expr 関連の API 定義。
  • apr_pool.h (間接): メモリ管理に使用。

関連コード

  • util_expr_parse.y: このファイルのソースとなる文法定義ファイル。
  • util_expr_scan.c: 字句解析器。
  • util_expr_eval.c: 構築された AST を評価するエンジン。

読者向けメモ

  • **メモリ管理**: すべての AST ノードとエラーメッセージは、コンテキストに含まれる apr_pool_t から割り当てられます。解析終了後にプールを破棄することでメモリリークを防ぎます。
  • **自動生成コード**: このファイルは Bison によって自動生成されています。直接編集せず、必ず util_expr_parse.y を修正してください。
  • **エラー処理**: YYERROR マクロが呼び出されると、パーサーはエラー回復ルーチンへ移行します。この際、ctx->error に詳細が設定されるため、呼び出し元は必ず確認してください。
  • **省略された詳細**: Bison の内部スタック管理や yyoverflow などの低レベルなスタック操作ロジックは、標準的な Bison スケルトンコードであるため、本ページでは詳細な解説を省略しています。
