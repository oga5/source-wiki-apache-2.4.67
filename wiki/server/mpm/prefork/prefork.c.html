server/mpm/prefork/prefork.c

概要

prefork.c は、Apache HTTP Server のマルチプロセッシングモジュール（MPM）の一つである mpm_prefork の実装です。各リクエストを単一のプロセスで処理する非スレッド型のモデルを提供します。スレッドセーフではないライブラリとの互換性を維持しつつ、リクエストの急増に備えてあらかじめ子プロセスを生成（プリフォーク）しておくことで応答性を高めます。

モジュール内での役割

Apache の MPM サブシステムにおいて、プロセス管理のライフサイクルを担います。親プロセスが子プロセスの生成、監視、終了、および負荷に応じた動的なスケーリングを管理します。

責務

  • **プロセス管理**: 子プロセスの生成 (make_child)、死活監視、およびシグナルによる制御。
  • **リソース最適化**: MinSpareServers / MaxSpareServers に基づく動的なプロセス数調整。
  • **接続処理**: accept() ミューテックスによる排他制御と、リスニングソケットのポーリング。
  • **設定管理**: ServerLimit や MaxRequestWorkers などのグローバル設定の検証と適用。
  • **ライフサイクル管理**: 設定の読み込みから、正常終了、グレースフル再起動までの状態遷移。

重要なシンボル

設定・状態管理

  • prefork_retained_data: 再起動を跨いで保持されるデータ構造。世代管理やスケーリング状態を格納。
  • ap_daemons_limit / server_limit: 同時接続数およびプロセス数の上限設定。
  • retained: モジュール全体で共有される prefork_retained_data のインスタンス。

プロセス制御

  • child_main(): 子プロセスのメインループ。接続の待機、処理、終了条件のチェックを行う。
  • make_child(): 新規子プロセスの fork() と初期化。
  • perform_idle_server_maintenance(): アイドルプロセス数を監視し、必要に応じて増減させる。

同期・通信

  • prefork_child_bucket: リスナー、パイプ（POD: Pipe of Death）、ミューテックスを保持する構造体。
  • accept_mutex_on() / accept_mutex_off(): accept() 呼び出し時の競合を防ぐためのプロセス間ミューテックス操作。

制御フローとデータフロー

1. **初期化**: prefork_hooks が各フェーズのフックを登録。prefork_pre_config で設定の初期化と retained データの確保を行う。 2. **起動**: prefork_run がメインループを開始。startup_children で初期プロセスを生成。 3. **監視**: 親プロセスは ap_wait_or_timeout で子プロセスの終了を待ち、perform_idle_server_maintenance で負荷に応じたプロセス増減を行う。 4. **接続処理**: 子プロセスは child_main 内で accept() を実行。SAFE_ACCEPT マクロにより、必要に応じてミューテックスで保護された接続受け入れを行う。 5. **終了**: シグナル（SIGTERM, AP_SIG_GRACEFUL）を受け取ると、POD を通じて子プロセスに通知し、クリーンアップ後に終了する。

読者向けメモ

  • **メモリ所有権**: pconf は設定用、pchild は子プロセス固有の生存期間を持つメモリプールです。child_main 内で pchild を適切に破棄することでメモリリークを防いでいます。
  • **シグナル処理**: die_now フラグはシグナルハンドラから更新されるため volatile です。
  • **制約**: ServerLimit は再起動時に変更できません。これは共有メモリのレイアウトが固定されるためです。
  • **省略された詳細**: GPROF 関連のコードはプロファイリング用であり、通常の運用では無視して差し支えありません。また、APR_HAS_THREADS 関連の分岐は、この MPM が非スレッド型であっても、APR の基盤としてスレッドサポートが有効な環境を考慮したものです。
