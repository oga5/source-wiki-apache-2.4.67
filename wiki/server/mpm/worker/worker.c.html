server/mpm/worker/worker.c

概要

worker.c は、Apache HTTP Server のマルチプロセッシングモジュール（MPM）である worker の実装です。各子プロセスが複数のスレッドを生成し、各スレッドが単一の接続を処理するハイブリッドなマルチプロセス・マルチスレッドモデルを提供します。従来のプロセスベースの MPM が抱えていた、スレッド間での accept 競合やクリーンな終了処理の困難さを解決するために設計されています。

モジュール内での役割

server/mpm/worker モジュールの中核であり、プロセス管理、スレッド生成、リスナーとワーカー間の接続受け渡し、およびシグナル処理を担います。mpm_common.h や mpm_fdqueue.h などの共通ライブラリを利用し、OS 固有のプロセス・スレッド制御を抽象化します。

責務

  • **プロセス管理**: fork による子プロセスの生成と、スコアボードを用いた死活監視。
  • **スレッド管理**: リスナー用スレッドと、リクエスト処理用ワーカープールスレッドの生成・管理。
  • **接続の受け渡し**: リスナーが受け取ったソケットを fd_queue を介してワーカーへ引き渡す。
  • **ライフサイクル制御**: 設定の検証、 graceful/ungraceful な再起動・終了処理、およびリソースのクリーンアップ。

重要なシンボル

設定・状態管理

  • threads_per_child, server_limit, thread_limit: MPM の動作を規定する主要な制限値。
  • retained: モジュール再読み込み時にも保持されるデータ構造（世代管理や spawn レートなど）。
  • worker_queue, worker_queue_info: リスナーとワーカー間でソケットをやり取りするためのキュー。

スレッド・プロセス制御

  • listener_thread: リスナー専用スレッド。apr_pollset を用いて複数のリスニングソケットを監視。
  • worker_thread: ワーカー用スレッド。キューからソケットを取り出し、process_socket を呼び出す。
  • make_child: 新規子プロセスを生成し、child_main を実行する。
  • signal_threads: 終了モード（Graceful/Ungraceful）に応じてスレッドへシグナルを送信。

ユーティリティ

  • wakeup_listener: リスナーをブロッキングから解除するためのシグナル送信。
  • close_worker_sockets: 強制終了時に全ワーカーのソケットを閉じる。

制御フローとデータフロー

1. **初期化**: worker_run がメインループを開始。startup_children が初期プロセスを生成。 2. **リスナー**: listener_thread が apr_pollset で接続を待機。接続を受け取ると ap_queue_push_socket でキューに投入。 3. **ワーカー**: worker_thread が ap_queue_pop_socket で接続を取得し、process_socket で ap_process_connection を実行。 4. **終了処理**: signal_threads が呼ばれると、リスナーとワーカーが終了フラグを検知し、join_workers でスレッドが回収される。

読者向けメモ

  • **メモリ所有権**: pconf（設定用）、pchild（子プロセス生存期間）、pruntime（スレッド生存期間）の各プールが厳密に使い分けられています。特に pruntime は、リスナーとワーカーが共有するリソースの生存期間を管理するために重要です。
  • **エッジケース**: worker_thread 内で apr_queue_pop_socket が中断されるケース（EINTR）や、listener_thread が poll 中にシグナルで起こされるケースなど、非同期的な終了処理が複雑に実装されています。
  • **制限事項**: ServerLimit や ThreadLimit は再起動を跨いで変更できません。これはスコアボードのメモリレイアウトが固定されるためです。
  • **省略箇所**: ap_mpm_podx 関連のパイプ通信の詳細や、scoreboard の詳細な更新ロジックは、MPM 共通ライブラリに依存しているため、本ファイルでは呼び出し側に焦点を当てています。
