modules/md/md_store.h

概要

md_store.h は、Apache HTTP Server のマネージド・ドメイン（MD）機能におけるデータ永続化層の抽象インターフェースを定義します。ドメイン設定、証明書、秘密鍵、ACME アカウント情報などのリソースを、グループ・ドメイン名・アスペクトの3階層キーで管理するための仕組みを提供します。

モジュール内での役割

modules/md サブシステムにおけるデータアクセスの「抽象レイヤー」です。具体的なストレージ実装（ファイルシステムなど）からロジックを分離し、上位モジュールがストレージの物理的な場所や形式を意識せずにデータを操作できるようにします。

責務

  • **抽象化**: md_store_t 構造体による関数ポインタの定義を通じ、ストレージ操作のインターフェースを統一する。
  • **データ管理**: 証明書や鍵などの型（md_store_vtype_t）と、用途別のグループ（md_store_group_t）を定義し、一貫したアクセスルールを強制する。
  • **ライフサイクル管理**: データのロード、保存、削除、移動、および更新時刻の比較といった基本操作の提供。
  • **排他制御**: グローバルロック機能による、複数プロセス間でのデータ整合性の保護。

重要なシンボル

  • md_store_t: ストレージ操作を抽象化した構造体。実装（md_store_fs.c など）はこの構造体を初期化して使用する。
  • md_store_group_t: データの用途区分（MD_SG_DOMAINS, MD_SG_ACCOUNTS など）。セキュリティやアクセス制限の境界となる。
  • md_store_vtype_t: データの型定義。JSON、PEM証明書、秘密鍵などを区別し、適切なシリアライズ/デシリアライズを支援する。
  • md_store_load / md_store_save: 汎用的なデータ入出力関数。

制御フローとデータフロー

1. **初期化**: 実装側が md_store_t の各関数ポインタを実装し、システム起動時に登録する。 2. **アクセス**: 上位モジュールが md_store_load などのラッパー関数を呼び出す。 3. **ディスパッチ**: ラッパー関数は md_store_t 内の関数ポインタを経由して、具体的なストレージ実装（例：ファイルシステム操作）を呼び出す。 4. **データ変換**: vtype に基づき、メモリ上の構造体とストレージ上の形式（JSONやPEM）の相互変換が行われる。

直接依存

  • apr_lib.h, apr_hash.h 等の Apache Portable Runtime (APR) ライブラリ。
  • md.h: 基本的なドメイン定義。
  • md_json.h: JSON 形式のデータ操作。
  • md_crypt.h: 暗号化関連の定義。

関連コード

  • md_store.c: md_store.h で定義されたインターフェースのラッパー実装。
  • md_store_fs.c: ファイルシステムを用いた具体的なストレージ実装。
  • md_reg.c: md_store を利用してドメイン登録情報を管理する上位モジュール。

読者向けメモ

  • **グループの重要性**: MD_SG_DOMAINS や MD_SG_ARCHIVE などのグループは、httpd の起動フェーズによってアクセス権限が異なる場合があります。実装や利用時には注意してください。
  • **拡張性**: 新しいストレージバックエンド（データベースなど）を追加する場合は、md_store_t の関数ポインタを実装するだけで済みます。
  • **メモリ管理**: すべてのロード操作は apr_pool_t を介して行われます。メモリリークを防ぐため、適切なプールライフサイクルを維持してください。
