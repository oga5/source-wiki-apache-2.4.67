modules/md/mod_md_ocsp.c

概要

本ファイルは、Apache HTTP Server における OCSP Stapling の統合管理を担当します。mod_md の一部として、証明書のステータス取得、キャッシュ、および定期的な更新（Watchdog 経由）を自動化します。

モジュール内での役割

mod_md サブシステムにおいて、OCSP レスポンスのライフサイクルを管理するレイヤーです。mod_ssl が提供するフックと連携し、証明書の検証に必要な OCSP ステータスを透過的に提供します。

責務

  • **ステータス提供**: mod_ssl からの要求に対し、キャッシュされた OCSP レスポンスを供給する。
  • **初期化と検証**: 証明書チェーンを解析し、OCSP Stapling の対象となる証明書を特定・準備する。
  • **バックグラウンド更新**: mod_watchdog を利用し、定期的に OCSP レスポンスを更新する。
  • **ハウスキーピング**: 古くなった OCSP レスポンスファイルを定期的に削除し、ストレージの肥大化を防止する。

重要なシンボル

ステータス管理 (Public API)

  • md_ocsp_prime_status: 証明書チェーンを解析し、OCSP ステータスの取得準備を行う。
  • md_ocsp_provide_status: mod_ssl から呼ばれるコールバック関数。キャッシュされた OCSP レスポンスを供給する。

Watchdog 制御

  • md_ocsp_start_watching: OCSP 更新用 Watchdog を登録・起動する。
  • run_watchdog: Watchdog のメインループ。md_ocsp_renew を呼び出し、次回の実行タイミングを計算する。

内部構造体

  • md_ocsp_ctx_t: Watchdog の実行コンテキスト。サーバー設定と mod_watchdog インスタンスを保持する。

制御フローとデータフロー

1. **初期化**: md_ocsp_start_watching が呼ばれ、専用のメモリプールと mod_watchdog インスタンスが作成される。 2. **ステータス更新**: run_watchdog が定期的に起動し、md_ocsp_renew を通じて各証明書の OCSP レスポンスを更新する。 3. **リクエスト処理**:

  • mod_ssl が md_ocsp_provide_status を呼び出す。
  • staple_here で設定を確認し、md_ocsp_get_status を経由してストレージから DER 形式のレスポンスを取得する。

    • 4. **クリーンアップ**: ocsp_remove_old_responses が古いレスポンスファイルを削除し、ストレージの整合性を維持する。

直接依存

  • mod_watchdog.h: バックグラウンドタスクの実行基盤。
  • md_ocsp.h: OCSP ロジックのコア実装（md_ocsp_renew 等）。
  • mod_md_config.h: MD_CONFIG_STAPLING 等の設定値取得。
  • http_ssl.h: ap_ssl_ocsp_copy_resp 等の mod_ssl インターフェース。

関連コード

  • md_ocsp.c: OCSP レスポンスの取得・保存・解析を行う低レイヤー実装。
  • mod_md_config.c: OCSP 関連の設定ディレクティブの定義。

読者向けメモ

  • **メモリ管理**: Watchdog 用のコンテキストは、サーバーのライフサイクルを跨いで生存させるため、専用の apr_pool_t と apr_allocator_t を使用しています。
  • **エラーパス**: staple_here が偽を返す場合、OCSP Stapling はスキップされ、DECLINED が返されます。これは設定ミスや無効な証明書チェーンの場合の標準的な挙動です。
  • **制約**: mod_watchdog が必須です。これがロードされていない場合、OCSP Stapling 機能は初期化時にエラーとなります。
  • **省略箇所**: 本ファイルは mod_watchdog とのインターフェースに特化しており、実際の OCSP 通信プロトコル（ASN.1 解析や HTTP 通信）の詳細は md_ocsp.c に委譲されています。
