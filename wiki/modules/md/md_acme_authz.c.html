modules/md/md_acme_authz.c

概要

ACME プロトコルにおけるドメイン認証（Authorization）プロセスを管理するモジュールです。ACME サーバーから提示されたチャレンジ（HTTP-01, TLS-ALPN-01, DNS-01）の取得、検証準備、およびサーバーへの応答通知を担当します。

モジュール内での役割

modules/md サブシステムにおいて、ACME 注文（Order）プロセスの一部として機能します。md_acme_order.c 等から呼び出され、ドメインの所有権証明に必要な環境構築とサーバーとの通信を仲介します。

責務

  • **認証状態の管理**: ACME サーバーから認証リソースを取得し、状態（Pending, Valid, Invalid）を解析・保持する。
  • **チャレンジの解決**: サーバーが要求するチャレンジタイプを特定し、対応するセットアップ処理（ファイル配置、証明書生成、外部コマンド実行）を実行する。
  • **ライフサイクル管理**: 認証完了後のクリーンアップ（teardown）およびエラー発生時の詳細情報の抽出。

重要なシンボル

認証コンテキスト

  • md_acme_authz_t: 認証リソースの構造体。ドメイン名、URL、現在の状態、エラー情報、JSON 表現を保持する。
  • md_acme_authz_create(): 認証オブジェクトの初期化。

チャレンジ処理

  • md_acme_authz_respond(): 認証のメインエントリポイント。サポートするチャレンジタイプを順次試行し、セットアップを試みる。
  • cha_type: チャレンジタイプごとの setup および teardown 関数を定義するディスパッチテーブル。
  • CHA_TYPES: HTTP-01, TLS-ALPN-01, DNS-01 の実装を束ねる静的配列。

サーバー通信

  • md_acme_authz_retrieve(): サーバーから認証リソースをフェッチする。
  • md_acme_authz_update(): 既存の認証リソースを最新の状態に更新する。

制御フローとデータフロー

1. **取得**: md_acme_authz_retrieve が ACME サーバーから JSON を取得し、md_acme_authz_t を構築する。 2. **選択**: md_acme_authz_respond が、設定された優先順位に従い、サーバーが提示したチャレンジとローカルでサポート可能なチャレンジを照合する。 3. **セットアップ**: 適合する cha_setup 関数（cha_http_01_setup 等）が呼び出される。

  • 内部で setup_key_authz を呼び出し、鍵認証トークンを生成。
  • 必要に応じて md_store を使用してファイルシステムへデータを保存、または外部コマンドを実行。
  • 成功時、md_acme_POST を介してサーバーへ応答を送信。

    • 4. **終了**: md_acme_authz_teardown が setup_token を解析し、対応する teardown 関数を呼び出してリソースを解放する。

読者向けメモ

  • **メモリ管理**: すべてのオブジェクトは apr_pool_t に紐付いており、明示的な free は不要です。
  • **エラー処理**: md_result_t を使用して、認証失敗時の詳細な診断情報を上位層へ伝播させます。
  • **設計の意図**: チャレンジのセットアップは cha_type 構造体によるディスパッチ方式を採用しており、新しいチャレンジタイプ（例: 今後のプロトコル拡張）の追加が容易な設計になっています。
  • **制約**: md_acme_authz_t の resource フィールドは、サーバーから取得した生の JSON へのポインタを保持します。このライフサイクルは authz オブジェクトと同期している必要があります。
