modules/md/md_acmev2_drive.c

概要

ACMEv2 プロトコルを用いた証明書の更新プロセスを管理するドライバ実装です。ACME サーバーとの対話を通じて、注文（Order）の作成から検証、証明書の取得までの一連のライフサイクルを制御します。

モジュール内での役割

modules/md サブシステムにおいて、ACMEv2 固有の更新ロジックをカプセル化しています。md_acme_drive.c が提供する汎用的な ACME ドライバ機能を利用しつつ、ACMEv2 特有の注文状態遷移（Pending, Ready, Valid 等）を処理する上位レイヤーとして機能します。

責務

  • **注文のライフサイクル管理**: ステージング領域からの注文読み込み、またはサーバーへの新規注文登録。
  • **状態遷移の監視**: 注文のステータスを定期的に確認し、検証（Challenge）の進行や証明書発行の準備状況を追跡。
  • **エラーリカバリ**: 注文が無効化された場合やドメイン構成の不一致が発生した際の、注文の破棄と再試行。
  • **証明書取得の調整**: サーバー側で注文が READY 状態になった際の証明書チェーン構築のトリガー。

重要なシンボル

  • md_acmev2_drive_renew: 本ファイルのメインエントリーポイント。証明書更新の全プロセスを統括します。
  • ad_setup_order: 既存の注文をロードするか、サーバーへ新規注文を登録する内部関数。

制御フローとデータフロー

1. **初期化**: md_acmev2_drive_renew が呼び出され、アカウント設定を確認します。 2. **注文の準備**: ad_setup_order により、ローカルの STAGING 領域から注文を復元するか、サーバーへ新規作成します。 3. **検証フェーズ**: md_acme_order_start_challenges および md_acme_order_monitor_authzs を通じて、ドメインの所有権確認（HTTP/TLS-ALPN 等）を進行させます。 4. **完了フェーズ**: 注文が READY になると証明書チェーンを構築し、最終的に VALID 状態になるまで待機して証明書を取得します。 5. **例外処理**: 途中で注文が INVALID になった場合は、ローカルの注文データをパージし、必要に応じて再試行します。

関連コード

  • md_acme_order.c: 注文の具体的な API 操作（登録、更新、監視）を実装。
  • md_acme_drive.c: アカウント管理や資格情報チェーンの構築など、ドライバ共通の基盤機能。
  • mod_md_drive.c: 本モジュールを呼び出す上位の Apache モジュールインターフェース。

読者向けメモ

  • 本ファイルは ACMEv2 のステートマシンを実装しています。デバッグ時は md_result_t に記録されるアクティビティログを確認すると、どの段階で停止しているか把握しやすいです。
  • ad_setup_order 内の ari_cert_id 処理は、ACME Renewal Information (ARI) に対応しており、証明書更新の最適化を試みます。
  • 注文の再試行ロジック（retry ラベル）は、ドメイン構成変更などでサーバー側から EINVAL が返された場合の自動復旧を目的としています。
