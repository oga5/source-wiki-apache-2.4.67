modules/md/md_acme_order.c

概要

本ファイルは、ACME プロトコルにおける「Order（注文）」オブジェクトのライフサイクル管理と、ACMEv2 サーバーとの通信フローを定義します。ACME Order は証明書発行プロセスの中核であり、ドメイン認証（Authz）の集約、チャレンジの開始、および証明書発行の準備状態を追跡します。

モジュール内での役割

modules/md サブシステムにおいて、ACME サーバーとの対話層を担当します。md_acme_authz.c が個別のドメイン認証を扱うのに対し、本ファイルはそれらを束ねる Order 全体の状態遷移を管理し、md_store.c を介した永続化を行います。

責務

  • **ライフサイクル管理**: Order オブジェクトの生成、JSON 形式へのシリアライズ/デシリアライズ、およびディスクへの保存・削除。
  • **状態追跡**: ACME サーバーから取得した Order のステータス（pending, ready, valid 等）の管理。
  • **ACMEv2 通信**: newOrder リクエストの送信、Order の更新取得、およびステータス遷移のポーリング。
  • **チャレンジ調整**: 複数のドメイン認証に対するチャレンジの開始と、完了までの監視。

重要なシンボル

データ構造

  • md_acme_order_t: Order の状態を保持するメイン構造体。URL、ステータス、認証 URL リスト、チャレンジセットアップトークンを保持。
  • order_ctx_t: ACME リクエスト実行時のコンテキスト。リクエストの引数や結果を保持する一時的な構造体。

ライフサイクル・変換

  • md_acme_order_create(): 新規 Order オブジェクトのメモリ確保と初期化。
  • md_acme_order_to_json() / md_acme_order_from_json(): JSON 永続化のための変換関数。
  • md_acme_order_load() / md_acme_order_save(): md_store を介したディスク I/O。

ACMEv2 操作

  • md_acme_order_register(): サーバーに対して新規 Order を作成（POST）。
  • md_acme_order_update(): サーバーから最新の Order 情報を取得（GET）。
  • md_acme_order_await_ready() / md_acme_order_await_valid(): 特定の状態になるまでポーリングを行う待機関数。

制御フローとデータフロー

1. **登録**: md_acme_order_register が order_ctx_t を作成し、md_acme_POST を呼び出す。on_init_order_register で JSON ペイロードを構築し、on_order_upd でサーバーからの応答を解析して md_acme_order_t を更新する。 2. **監視**: md_util_try を利用したポーリングループ（await_ready 等）により、サーバー側のステータスが valid になるまで md_acme_order_update を繰り返す。 3. **チャレンジ**: md_acme_order_start_challenges が各認証 URL を巡回し、md_acme_authz_retrieve で状態を確認。pending であれば md_acme_authz_respond を呼び出し、結果を order->challenge_setups に保存して永続化する。

読者向けメモ

  • **メモリ管理**: すべての関数は apr_pool_t を受け取り、オブジェクトの生存期間はプールに依存します。
  • **永続化**: md_acme_order_save は md_util_pool_vdo を介して実行され、一時的なプールと永続的なプールを分離して安全に書き込みを行います。
  • **エラーパス**: md_result_t を通じてエラー詳細が伝播されます。特に await_ 系関数は、タイムアウトやサーバー側のエラーを APR_EAGAIN や APR_EINVAL として返します。
  • **制約**: md_acme_order_purge は、Order 削除時に紐付いているチャレンジのクリーンアップ（md_acme_authz_teardown）を自動的に実行する設計になっています。
