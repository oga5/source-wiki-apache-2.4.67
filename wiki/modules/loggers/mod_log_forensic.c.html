modules/loggers/mod_log_forensic.c

概要

mod_log_forensic.c は、Apache HTTP Server におけるリクエストの「フォレンジック（証拠保全）」ログを記録するためのモジュールです。リクエストの開始時と終了時に、リクエストラインやヘッダーを含む詳細な情報を一意の ID と共に記録し、リクエストの追跡可能性を提供します。

モジュール内での役割

modules/loggers ディレクトリ内の他のログモジュールが主にアクセスログのカスタマイズやデバッグを目的とするのに対し、本モジュールはセキュリティや監査を主眼に置いています。mod_unique_id と連携し、リクエストのライフサイクル全体を識別可能な形式で出力します。

責務

  • **設定管理**: ForensicLog ディレクティブによるログ出力先（ファイルまたはパイプ）の管理。
  • **ライフサイクル追跡**: リクエスト開始時（post_read_request）に詳細なヘッダーを記録し、終了時（log_transaction）に完了フラグを記録。
  • **データエスケープ**: ログ出力時に制御文字や特殊文字を安全な形式（%XX）に変換。
  • **一意性保証**: mod_unique_id が利用できない環境でも、PID、時刻、アトミックカウンタを用いたフォレンジック ID の生成。

重要なシンボル

設定構造体

  • fcfg: サーバーごとの設定を保持。ログファイル名とファイルディスクリプタ（apr_file_t）を管理。

ライフサイクル・フック

  • log_init: サーバー起動時にログファイルを開く。
  • log_before: リクエスト開始時に実行。ヘッダーを解析し、+<id>|... 形式でログを書き込む。
  • log_after: リクエスト終了時に実行。-<id> 形式で完了ログを書き込む。

ユーティリティ

  • log_escape: 文字列内の特殊文字をエスケープし、バッファに書き込む。
  • count_string / count_headers: ログ書き込みに必要なバッファサイズを事前に計算する。

制御フローとデータフロー

1. **初期化**: log_init が各サーバー設定を走査し、open_log を呼び出してログファイルを開く。 2. **開始時**: log_before がリクエストヘッダーを走査。count_headers で必要なメモリ量を算出し、apr_palloc で確保。log_escape を用いてヘッダーをシリアライズし、ファイルへ書き込む。 3. **終了時**: log_after がリクエストコンテキストから ID を取得し、終了を示す -<id> を書き込む。 4. **エスケープ**: log_escape は TEST_CHAR マクロ（test_char.h）を使用して、エスケープ対象文字を判定し、ap_bin2hex で変換を行う。

直接依存

  • httpd.h, http_config.h, http_log.h: Apache コア API。
  • apr_strings.h, apr_atomic.h: 文字列操作およびアトミックな ID 生成。
  • test_char.h: エスケープ対象文字の判定用。
  • http_protocol.h: リクエスト処理関連。
  • unistd.h

関連コード

  • mod_unique_id.c: 本モジュールが依存する一意 ID 生成モジュール。
  • support/check_forensic: 本モジュールが出力したログを解析・検証するための補助ツール。

読者向けメモ

  • **メモリ管理**: リクエストごとのログ生成には r->pool を使用しており、リクエスト終了時に自動的に解放されます。
  • **アトミック性**: next_id のインクリメントには apr_atomic_inc32 を使用しており、マルチスレッド環境での ID の一意性を保証しています。
  • **エスケープの不変条件**: log_escape は ap_assert を多用しており、バッファオーバーフローを厳格に防いでいます。
  • **制限事項**: log_before は r->prev が存在する場合（内部リダイレクト等）はログ出力をスキップします。これは同一リクエストの重複記録を防ぐためです。
  • **パフォーマンス**: ログ出力ごとにヘッダーの走査とメモリ確保が発生するため、高負荷環境では I/O およびメモリ消費に注意が必要です。
