modules/filters/mod_buffer.c

概要

mod_buffer.c は、Apache HTTP Server の入力および出力フィルタスタックにおいて、複数の小さなバケットを大きなバケットに集約（collapse）するためのフィルタモジュールです。ネットワーク I/O の効率化を目的としており、設定されたサイズに達するまでデータをバッファリングします。

モジュール内での役割

フィルタチェーンの AP_FTYPE_CONTENT_SET 階層に位置し、コンテンツ生成とネットワーク送信の間でバッファリングを行います。主に小さなバケットが頻繁に生成される環境において、システムコール回数を削減し、効率的なデータ転送を支援します。

責務

  • **バッファ管理**: BufferSize ディレクティブに基づき、リクエストごとのバッファサイズを管理。
  • **出力フィルタリング**: buffer_out_filter を通じて、下流へ渡されるバケットを蓄積し、サイズ超過時またはフラッシュ/EOS時に一括送信。
  • **入力フィルタリング**: buffer_in_filter を通じて、上流から読み込んだデータをバッファし、要求されたサイズ分を切り出して提供。
  • **設定継承**: ディレクトリ単位の設定（buffer_conf）の作成およびマージ。

重要なシンボル

設定管理

  • buffer_conf: ディレクトリ単位の設定構造体。size（バッファ上限）と size_set（設定有無）を保持。
  • create_buffer_config / merge_buffer_config: 設定の初期化および継承ロジック。
  • set_buffer_size: BufferSize ディレクティブのバリデーションと設定。

フィルタコンテキスト

  • buffer_ctx: リクエストごとの実行時状態。bb（メインバッファ）、tmp（入力用一時バッファ）、remaining（残り容量）を保持。

フィルタ関数

  • buffer_out_filter: 出力チェーン用。f->r->main（サブクエスト）の場合はバッファリングをスキップして透過させる。
  • buffer_in_filter: 入力チェーン用。AP_MODE_READBYTES モード時のみバッファリングを行い、それ以外は透過。

制御フローとデータフロー

1. **初期化**: フィルタが初めて呼ばれた際、リクエストプールから buffer_ctx を確保し、設定を読み込む。 2. **出力フロー**:

  • バケットを順次処理し、メタデータ（EOS/Flush）は即座に下流へ渡す。
  • データバケットは apr_brigade_write でバッファへコピー。
  • バッファが size を超えた場合、ap_pass_brigade で下流へフラッシュ。
  • 最適化として、既存のヒープバケットであればコピーせずに移動（move）を試みる。

    • 3. **入力フロー**:

  • バッファが空の場合、ap_get_brigade で上流からデータを読み込み、バッファが満杯になるまで蓄積。
  • 呼び出し元が要求したサイズ分を apr_brigade_partition で切り出し、呼び出し元へ渡す。

読者向けメモ

  • **メモリ所有権**: buffer_ctx およびバッファ内のブリゲードはリクエストプール（r->pool）に紐付いており、リクエスト終了時に自動解放されます。
  • **サブクエストの制約**: 出力フィルタはサブクエストに対してはバッファリングを行いません（EOSの判定が困難なため）。
  • **パフォーマンス**: ヒープバケットの移動最適化は、バケットサイズが APR_BUCKET_BUFF_SIZE と一致する場合にのみ有効です。
  • **未実装/浅い部分**: buffer_out_filter 内の ETag 生成ロジックはコメントアウトされており、現時点では実装されていません。また、入力フィルタは AP_MODE_READBYTES 以外のモード（GETLINE 等）ではバッファリングを行わず透過します。
