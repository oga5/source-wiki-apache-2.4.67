modules/filters/mod_deflate.c

概要

mod_deflate は、Apache HTTP Server において動的なコンテンツ圧縮（Gzip）および伸長（Inflate）を提供するフィルタモジュールです。zlib ライブラリをラップし、HTTP ストリームの入出力に対して透過的なデータ変換を行います。

モジュール内での役割

Apache のフィルタチェーンの一部として動作します。出力フィルタ（DEFLATE, INFLATE）および入力フィルタ（DEFLATE）を登録し、リクエストの Content-Encoding ヘッダーや Accept-Encoding リクエストヘッダーに基づいて、動的に圧縮・伸長を適用します。

責務

  • **圧縮/伸長処理**: zlib を使用したストリームの変換。
  • **HTTP プロトコル整合性**: Content-Encoding の管理、Vary: Accept-Encoding の付与、ETag の調整。
  • **リソース保護**: 圧縮比や伸長後のサイズ制限による DoS 攻撃対策。
  • **設定管理**: 圧縮レベル、ウィンドウサイズ、メモリ使用量などのチューニング。

重要なシンボル

設定構造体

  • deflate_filter_config: サーバー全体の設定（圧縮レベル、バッファサイズ等）。
  • deflate_dirconf_t: ディレクトリ単位の設定（伸長サイズ制限、圧縮比制限）。

コンテキスト

  • deflate_ctx: フィルタのライフサイクル中に保持される状態。zlib の z_stream、バッファ、CRC、ヘッダー情報、処理済みバイト数などを管理。

フィルタ関数

  • deflate_out_filter: 出力データを圧縮するフィルタ。
  • inflate_out_filter: 出力データを伸長するフィルタ。
  • deflate_in_filter: 入力データ（リクエストボディ）を伸長するフィルタ。

ユーティリティ

  • flush_libz_buffer: zlib の内部バッファをフラッシュし、ブリゲードへ書き出す。
  • check_gzip: リクエストが Gzip 圧縮されているか判定し、ヘッダーを調整する。
  • consume_buffer: 圧縮/伸長されたデータを apr_bucket_brigade に追加する。

制御フローとデータフロー

1. **初期化**: フィルタが初めて呼ばれた際、deflate_ctx を生成し deflateInit2 / inflateInit2 で zlib を初期化。 2. **データ処理**:

  • apr_bucket_brigade からバケットを読み取り、zlib に渡す。
  • zlib の出力バッファが満杯になるか、フラッシュ要求（FLUSH バケット）が来た時点で consume_buffer を呼び出し、次のフィルタへデータを渡す。

    • 3. **終了処理**: EOS (End of Stream) バケットを検知すると、zlib の残存データをフラッシュし、Gzip フッター（CRC/サイズ）を付与して deflateEnd / inflateEnd を呼び出す。

読者向けメモ

  • **メモリ管理**: deflate_ctx はリクエストプールに紐付いており、リクエスト終了時に apr_pool_cleanup_register を通じて zlib リソースが解放されます。
  • **メモリレイアウト**: deflate_ctx 内の buffer は設定された bufferSize に基づいて動的に確保されます。
  • **エッジケース**:
  • HEAD リクエストや 204 No Content、既に圧縮済みのコンテンツはフィルタをスキップします。
  • 伸長時の ratio_limit や inflate_limit を超えた場合、APR_EINVAL や APR_ENOSPC を返して処理を中断します。
  • **省略箇所**: zlib の内部的なハフマン符号化や LZ77 圧縮アルゴリズムの詳細は zlib ライブラリ側に依存するため、本ファイルでは触れていません。
