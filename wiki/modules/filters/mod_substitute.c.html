modules/filters/mod_substitute.c

概要

mod_substitute は、Apache HTTP Server の出力フィルタモジュールであり、レスポンスボディに対して正規表現または文字列置換をオンザフライで実行します。メモリ使用量を予測可能にするため、行単位の処理と最大行長制限（デフォルト 1MB）を強制する設計となっています。

モジュール内での役割

modules/filters サブシステムの一部として、コンテンツ変換フィルタを提供します。mod_sed と類似の目的を持ちますが、より軽量な置換処理に特化しています。

責務

  • **設定管理**: Substitute ディレクティブによる置換パターンの定義と、SubstituteMaxLineLength によるメモリ保護設定の管理。
  • **ストリーム処理**: apr_bucket_brigade を介してレスポンスデータを読み込み、行単位（改行区切り）でバッファリングして置換を実行。
  • **メモリ安全性**: ap_varbuf を活用した動的バッファ管理と、制限を超えた場合の APR_ENOMEM エラーによる保護。

重要なシンボル

設定構造体

  • subst_dir_conf: ディレクトリごとの設定。置換パターンの配列、最大行長、継承順序を保持。
  • subst_pattern_t: 個別の置換ルール。apr_strmatch（高速文字列検索）または ap_regex_t（正規表現）のいずれかを保持。

フィルタコンテキスト

  • substitute_module_ctx: フィルタ実行中の状態保持用。linebb（未完の行）、passbb（出力用）、tpool（一時メモリ用）を管理。

主要関数

  • substitute_filter: フィルタのメインエントリポイント。バケットの読み込み、行の分割、置換処理の呼び出し、次フィルタへの転送を担当。
  • do_pattmatch: 特定の行に対して全パターンを適用するコアロジック。flatten モード（バッファコピー）と高速なバケット分割モードを切り替える。
  • set_pattern: Substitute ディレクティブの構文解析とコンパイル。

制御フローとデータフロー

1. **初期化**: substitute_filter が初回呼び出し時に substitute_module_ctx を生成し、一時プールを確保。 2. **バッファリング**: apr_bucket_read でデータを読み込み、memchr で改行を探す。改行が見つかるまで ctx->linebb にバケットを蓄積。 3. **置換実行**: 1行が完成すると do_pattmatch を呼び出す。

  • flatten モード: ap_varbuf を使用して文字列を結合・置換し、新しいバケットを作成。
  • 高速モード: SEDRMPATBCKT マクロでバケットを分割し、置換文字列を挿入。

    • 4. **転送**: 処理済みバケットを ctx->passbb に移し、ap_pass_brigade で下流フィルタへ送出。 5. **クリーンアップ**: ctx->tpool を定期的にクリアし、メモリリークを防止。

読者向けメモ

  • **メモリ所有権**: ctx->tpool は substitute_filter の各ループ終了時に apr_pool_clear され、メモリが再利用されます。
  • **制限事項**: AP_MAX_BUCKETS を超えるバケットが蓄積された場合、強制的にフラッシュしてメモリを解放する安全装置があります。
  • **設計の意図**: flatten モードはメモリ消費が激しいため、デフォルトではバケット分割によるインプレース置換が優先されます。
  • **未実装/浅い部分**: AP_MAX_BUCKETS のチェックは安全策として存在しますが、極端な断片化が発生するエッジケースではパフォーマンスが低下する可能性があります。
