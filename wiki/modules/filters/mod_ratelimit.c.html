modules/filters/mod_ratelimit.c

概要

mod_ratelimit は、Apache HTTP Server の出力フィルタとして機能し、レスポンスの転送速度を制限するモジュールです。環境変数 rate-limit を通じて動的に帯域制限を適用し、必要に応じて rate-initial-burst によるバースト転送を許可します。

モジュール内での役割

本モジュールは AP_FTYPE_CONNECTION - 1 レベルで動作する出力フィルタです。他のフィルタ（mod_deflate 等）よりも後段で動作し、最終的なネットワークへのデータ送出速度を制御します。

責務

  • **帯域制限の適用**: 指定された速度（KB/s）に基づき、データをチャンク単位で分割して送信する。
  • **バースト制御**: 初期転送時に指定されたバースト量を許可し、その後制限速度へ移行する。
  • **状態管理**: rl_ctx_t を用いて、制限モード（RATE_LIMIT）と全速モード（RATE_FULLSPEED）の状態を保持する。
  • **特殊バケットの提供**: RL_START / RL_END バケットを定義し、ストリーム内の特定区間のみを全速で送信する制御を可能にする。

重要なシンボル

状態管理とコンテキスト

  • rl_ctx_t: フィルタのコンテキスト。速度、バースト量、現在の状態、および処理待ちのブリゲード（tmpbb, holdingbb）を保持。
  • rl_state_e: フィルタの動作モード（RATE_LIMIT または RATE_FULLSPEED）。

フィルタ処理

  • rate_limit_filter: メインのフィルタ関数。環境変数の読み込み、コンテキストの初期化、およびブリゲードの分割・送信を行う。

カスタムバケット（制御用）

  • ap_rl_bucket_type_start / ap_rl_bucket_type_end: 転送速度制御の切り替えポイントを示すメタデータバケット。
  • ap_rl_start_create / ap_rl_end_create: 上記バケットを生成する公開関数。

制御フローとデータフロー

1. **初期化**: 初回呼び出し時に rl_ctx_t を生成し、環境変数から制限値を取得。無効な設定の場合はフィルタを解除。 2. **ブリゲード処理**:

  • RATE_FULLSPEED モード: RL_END バケットが見つかるまでデータをそのまま通過させる。
  • RATE_LIMIT モード: RL_START バケットが見つかるまでデータをチャンク単位（chunk_size）に分割し、apr_sleep を挟みながら ap_pass_brigade で送信する。

    • 3. **データ保持**: 処理しきれなかったデータは ap_save_brigade を用いて holdingbb に保存され、次回の呼び出しに持ち越される。

直接依存

  • httpd.h, http_config.h, http_log.h: Apache コア機能。
  • util_filter.h: フィルタチェーン操作用。
  • mod_ratelimit.h: 公開バケット型定義および生成関数の宣言。

関連コード

  • modules/filters/mod_ratelimit.h: 公開インターフェース。
  • util_filter.c: フィルタチェーンの仕組み。
  • apr_buckets.h: ブリゲードおよびバケット操作の基盤。

読者向けメモ

  • **メモリ管理**: コンテキストおよびブリゲードは f->r->pool に紐付いており、リクエスト終了時に自動的に解放されます。
  • **制限事項**: mod_ratelimit はサブリクエストに対しては自動的にフィルタを解除し、制限を適用しません。
  • **パフォーマンス**: RATE_INTERVAL_MS (200ms) ごとに apr_sleep を呼び出すため、高負荷時にはスレッドのブロッキングが発生する可能性があります。
  • **未実装**: TODO にある通り、現在はモジュールのロード順序に依存する可能性があるため、将来的に Optional Functions への移行が検討されています。
