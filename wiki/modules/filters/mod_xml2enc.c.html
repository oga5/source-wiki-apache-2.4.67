modules/filters/mod_xml2enc.c

概要

mod_xml2enc は、Apache HTTP Server において XML/HTML コンテンツの文字エンコーディングを自動検出し、必要に応じて UTF-8 へ変換する出力フィルタモジュールです。libxml2 を活用してエンコーディングを推論し、apr_xlate を用いて変換を行います。

モジュール内での役割

modules/filters サブシステムにおいて、コンテンツの文字コード正規化を担当するフィルタです。主に mod_proxy_html などの XML/HTML 処理モジュールが、後続の処理で一貫したエンコーディング（UTF-8）を扱えるようにするための前処理として機能します。

責務

  • **エンコーディング検出**: HTTP ヘッダー、BOM、HTML <meta> タグを解析し、コンテンツの文字コードを特定する。
  • **文字コード変換**: 特定されたエンコーディングから UTF-8 への変換をストリーム処理で行う。
  • **コンテンツの正規化**: 検出された <meta> タグの除去や、指定された要素までコンテンツをスキップする処理。
  • **設定管理**: デフォルトの文字コード設定や、エンコーディングのエイリアス定義を管理する。

重要なシンボル

コンテキストと設定

  • xml2ctx: フィルタの実行状態（変換器、バッファ、フラグ）を保持するリクエストごとの構造体。
  • xml2cfg: モジュールのディレクトリごとの設定（デフォルト文字コード、スキップ対象要素）。

フィルタ処理

  • xml2enc_ffunc: フィルタのメインループ。バッファリング、検出、変換、後続フィルタへのパスを行う。
  • xml2enc_filter: フィルタを動的に挿入するための関数。
  • sniff_encoding: コンテンツをスキャンしてエンコーディングを特定するロジック。
  • fix_skipto: 設定された要素までコンテンツをスキップする補助関数。

フックと登録

  • xml2enc_hooks: フィルタの登録と正規表現のコンパイルを行う。
  • xml2enc_module: モジュールのメイン構造体。

制御フローとデータフロー

1. **初期化**: xml2enc_filter_init でコンテキストを生成。 2. **バッファリング**: xml2enc_ffunc は、エンコーディングを特定するために十分なデータ（BUF_MIN）が溜まるまでバッファリングを行う。 3. **検出**: sniff_encoding が呼ばれ、HTTP ヘッダー、BOM、<meta> タグを順に検査。特定できない場合は設定値を使用。 4. **変換**: apr_xlate_conv_buffer を使用し、バッファ内のデータを UTF-8 に変換して後続フィルタへ渡す。 5. **エラー処理**: 不正なバイト列に遭遇した場合は APR_EINVAL を返し、該当バイトをスキップして処理を継続する。

直接依存

  • libxml/encoding.h: エンコーディング検出と名称解決。
  • apr_xlate.h: 文字コード変換の抽象化レイヤー。
  • http_protocol.h, http_config.h: Apache のリクエスト処理と設定管理。
  • mod_xml2enc.h: 公開インターフェースとオプションフックの定義。
  • apr_optional.h
  • apr_strings.h
  • ctype.h
  • http_log.h

関連コード

  • mod_proxy_html.c: 本モジュールを最も頻繁に利用する消費者。
  • mod_charset_lite.c: 同様の文字コード変換を行うが、こちらはより汎用的なフィルタ。

読者向けメモ

  • **メモリ管理**: request_rec のプールを使用してメモリを確保しており、リクエスト終了時に自動解放されます。
  • **不完全なデータ**: APR_EINCOMPLETE が発生した場合、バッファを保持して次回の ffunc 呼び出しを待ちます。
  • **制約**: libxml2 の検出ロジックに依存しているため、サポートされていないエンコーディングの場合は apr_xlate での変換を試みますが、最終的に失敗した場合は設定されたデフォルト値にフォールバックします。
  • **省略された詳細**: libxml2 内部の複雑なエンコーディングテーブルの詳細は本ファイルには含まれず、ライブラリ側に委譲されています。また、ap_assert の使用箇所については、将来的にエラーハンドリングへの置き換えが推奨される旨のコメントがソース内に存在します。
