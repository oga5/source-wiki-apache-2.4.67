modules/metadata/mod_headers.c

概要

mod_headers.c は、HTTP リクエストおよびレスポンスヘッダーの追加、変更、削除、置換を行う Apache HTTP Server モジュールです。Header および RequestHeader ディレクティブを提供し、正規表現による置換や、環境変数・式（ap_expr）に基づいた条件付き処理をサポートします。

モジュール内での役割

modules/metadata サブシステムにおいて、HTTP メッセージのメタデータ（ヘッダー）を動的に操作する中心的な役割を担います。リクエスト処理の初期段階（post_read_request）からレスポンス送信直前まで、複数のフックポイントでヘッダーを制御します。

責務

  • **設定管理**: Header / RequestHeader ディレクティブの解析と、ディレクトリ単位の構成保持。
  • **ヘッダー操作**: set, add, append, merge, unset, edit などのアクションの実装。
  • **動的フォーマット**: %D (duration), %t (time), %e (env) などのタグを解析し、実行時に値を生成。
  • **条件付き処理**: env=... や expr=... による、特定の条件下でのみヘッダーを適用するロジックの提供。
  • **フィルタリング**: 出力フィルタ（FIXUP_HEADERS_OUT/ERR）を介したレスポンスヘッダーの最終調整。

重要なシンボル

設定・データ構造

  • headers_conf: ディレクトリごとの設定を保持。fixup_in, fixup_out, fixup_err の各配列に header_entry を格納。
  • header_entry: 個別のヘッダー操作ルール。アクション、対象ヘッダー名、条件式、正規表現、フォーマットタグ配列を保持。
  • format_tag: ヘッダー値内の動的要素を処理するための関数ポインタと引数のペア。

コマンド・解析

  • header_cmd: Header / RequestHeader ディレクティブのメインエントリーポイント。
  • header_inout_cmd: ディレクティブの引数を解析し、header_entry を構築する内部ロジック。
  • parse_format_string: ヘッダー値内のフォーマットタグを分解し、format_tag 配列を生成。

実行・処理

  • do_headers_fixup: ヘッダー操作のコアエンジン。apr_table_t に対して指定されたアクションを適用。
  • process_tags: format_tag 配列を走査し、動的な値を結合して文字列を生成。
  • process_regexp: edit アクションのための正規表現置換処理。

制御フローとデータフロー

1. **設定時**: header_cmd がディレクティブを読み取り、header_entry を headers_conf に蓄積。 2. **リクエスト処理**:

  • ap_headers_early: post_read_request フックで、early 指定されたヘッダーを即座に処理。
  • ap_headers_fixup: fixups フックで、リクエストヘッダーを処理。
  • ap_headers_output_filter: 出力フィルタとして、レスポンスヘッダーを最終調整。

    • 3. **データフロー**: do_headers_fixup が request_rec 内の headers_in / headers_out / err_headers_out を直接操作します。process_tags は request_rec の情報を参照して値を生成します。

読者向けメモ

  • **メモリ管理**: すべての動的メモリ割り当ては request_rec->pool または cmd->pool を使用しており、リクエスト終了時に自動解放されます。
  • **順序の重要性**: Header ディレクティブは設定ファイル内の出現順序に従って適用されます。append や unset の順序には注意が必要です。
  • **パフォーマンス**: edit アクション（正規表現）はリクエストごとに実行されるため、多用すると負荷が高まります。
  • **制限事項**: header_request_ssl_var など一部のタグは、SSL モジュールが有効でない場合や特定のタイミングでは期待した値を返さない可能性があります。
  • **未解決の深掘り**: format_tag_hash の初期化タイミングや、early 処理と通常のフィルタ処理の競合については、httpd のフック実行順序の理解が不可欠です。
