modules/ssl/ssl_engine_kernel.c

概要

ssl_engine_kernel.c は、Apache HTTP Server の SSL/TLS 処理における「カーネル」として機能する中核ファイルです。リクエスト処理の各フェーズ（ReadReq, Access, Auth, Fixup）における SSL 固有のロジック、再ネゴシエーションの制御、および OpenSSL とのコールバック連携を担います。

モジュール内での役割

mod_ssl サブシステムにおいて、HTTP リクエストのライフサイクルと SSL 状態を橋渡しする役割を持ちます。ssl_engine_io.c が低レベルな I/O フィルタを扱うのに対し、本ファイルはポリシー適用や認証、環境変数の設定といった上位のビジネスロジックを処理します。

責務

  • **リクエストフックの処理**: ReadReq から Fixup まで、各フェーズで必要な SSL 状態の検証と環境設定を行う。
  • **再ネゴシエーション制御**: ディレクトリ単位の SSL 設定変更に伴う再ネゴシエーションの強制、およびそれに伴うリクエストボディのバッファリング。
  • **OpenSSL コールバック**: 証明書検証、セッションキャッシュ、SNI、ALPN、SRP、TLS チケット等の OpenSSL イベントに対するハンドラを提供。
  • **ポリシー適用**: SNI ポリシーのチェックや、SSL 必須設定の強制。

重要なシンボル

リクエストフック（公開 API）

  • ssl_hook_ReadReq: リクエスト開始時の SSL 接続確認と SNI ポリシーチェック。
  • ssl_hook_Access: アクセス制御フェーズ。再ネゴシエーションの要否判断と実行。
  • ssl_hook_UserCheck: クライアント証明書を用いた FakeBasicAuth の処理。
  • ssl_hook_Fixup: CGI/SSI 環境変数への SSL 情報の注入。

OpenSSL コールバック

  • ssl_callback_SSLVerify: 証明書検証プロセス。OCSP チェックや CRL 検証を含む。
  • ssl_callback_NewSessionCacheEntry / GetSessionCacheEntry: セッションキャッシュの永続化と取得。
  • ssl_callback_ServerNameIndication: SNI 拡張によるバーチャルホストの動的切り替え。
  • ssl_callback_alpn_select: ALPN によるプロトコルネゴシエーション。

内部ヘルパー

  • fill_reneg_buffer: 再ネゴシエーション時にリクエストボディを一時退避する。
  • ssl_check_vhost_sni_policy: SNI による vhost 切り替えの妥当性を検証する。

制御フローとデータフロー

1. **リクエスト処理**: ssl_hook_Access が呼び出されると、現在の SSL 設定とディレクトリ設定を比較し、再ネゴシエーションが必要か判断します。 2. **再ネゴシエーション**: 必要であれば fill_reneg_buffer でボディをバッファし、SSL_renegotiate を実行します。 3. **コールバック**: OpenSSL が検証や SNI 処理を行う際、SSL_get_app_data を通じて Apache の conn_rec や request_rec を取得し、コンテキストに応じたポリシーを適用します。 4. **環境設定**: ssl_hook_Fixup が subprocess_env に SSL 関連のメタデータを書き込み、後続の CGI 等で利用可能にします。

直接依存

  • ssl_private.h: モジュール内部のデータ構造（SSLConnRec, SSLSrvConfigRec 等）の定義。
  • mod_ssl.h: 公開インターフェース。
  • util_md5.h: セッション ID コンテキストの生成に使用。
  • scoreboard.h: プロセス状態の更新に使用。

関連コード

  • ssl_engine_io.c: SSL フィルタの入出力処理。
  • ssl_engine_config.c: 設定構造体の管理。
  • ssl_scache.c: セッションキャッシュのバックエンド実装。

読者向けメモ

  • **メモリ管理**: apr_pool_t を介したメモリ管理が徹底されています。コールバック内で一時的に生成されるオブジェクト（証明書チェーン等）の解放には注意が必要です。
  • **再ネゴシエーションの複雑性**: TLS 1.3 では再ネゴシエーションが廃止され、Post-Handshake Authentication に置き換わっています。本ファイルでは ssl_hook_Access_classic と ssl_hook_Access_modern にロジックが分離されています。
  • **制約**: ssl_hook_Access はサブリクエストやスレーブ接続において、再ネゴシエーションを禁止するガードロジックが含まれています。これは並行リクエストとの競合を防ぐための重要な不変条件です。
  • **省略**: SRP や ALPN 等の特定の拡張機能は、ビルド時の定義（HAVE_SRP 等）によって有効・無効が切り替わるため、すべてのコードパスが常に実行されるわけではありません。
