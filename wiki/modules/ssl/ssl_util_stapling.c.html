modules/ssl/ssl_util_stapling.c

概要

本ファイルは、Apache HTTP Server の mod_ssl における **OCSP Stapling**（RFC 6066）の実装を提供します。SSL/TLS ハンドシェイク中に、サーバーが証明書の有効性確認結果（OCSP レスポンス）をクライアントへ直接提供することで、クライアント側の OCSP クエリを不要にし、プライバシーとパフォーマンスを向上させます。

モジュール内での役割

mod_ssl の一部として、OpenSSL の TLS 拡張機能（Certificate Status Request）を処理します。ssl_engine_init.c 等から初期化され、ハンドシェイク時にコールバックとして動作します。

責務

  • **OCSP レスポンスのキャッシュ管理**: socache を利用したレスポンスの保存と取得。
  • **レスポンスの更新**: キャッシュ切れや無効なレスポンスに対する OCSP レスポンダーへの再問い合わせ。
  • **同期制御**: 複数のプロセス/スレッド環境におけるキャッシュアクセスおよび更新処理の排他制御。
  • **証明書情報の管理**: 証明書と OCSP レスポンダー URI の紐付け管理。

重要なシンボル

キャッシュ・状態管理

  • certinfo: 証明書ハッシュ、OCSP CERTID、レスポンダー URI を保持する構造体。
  • stapling_certinfo: 証明書情報を保持するグローバルハッシュテーブル。
  • stapling_cache_response / stapling_get_cached_response: キャッシュへの読み書きを行うラッパー。

コールバック・フック

  • stapling_cb: OpenSSL の SSL_CTX_set_tlsext_status_cb に登録されるメインのコールバック関数。
  • ssl_run_init_stapling_status / ssl_run_get_stapling_status: 他モジュールが OCSP ステータスを提供するためのフック。

同期制御

  • ssl_stapling_mutex_init: 実行時に必要なグローバルミューテックスの初期化。
  • stapling_refresh_mutex_on/off: レスポンス更新時の競合を防ぐための排他制御。

制御フローとデータフロー

1. **初期化**: modssl_init_stapling が呼ばれ、コールバック stapling_cb を登録。 2. **ハンドシェイク時**: クライアントがステータス拡張を要求すると stapling_cb が起動。 3. **キャッシュ検索**: stapling_get_cached_response でキャッシュを検索。 4. **更新判断**: キャッシュがない、または期限切れの場合、stapling_refresh_mutex を取得し、stapling_renew_response で OCSP レスポンダーへ HTTP リクエストを送信。 5. **レスポンス返却**: stapling_set_response を通じて OpenSSL に OCSP レスポンスを渡す。

直接依存

  • ssl_private.h: mod_ssl 内部構造体定義。
  • ap_mpm.h: マルチプロセス管理用。
  • apr_thread_mutex.h: APR レベルの排他制御。

関連コード

  • ssl_engine_ocsp.c: OCSP リクエストの送信やレスポンスの解析など、低レベルな OCSP 通信ロジック。
  • ssl_scache.c: セッションキャッシュの抽象化レイヤー。

読者向けメモ

  • **メモリ管理**: certinfo や OCSP_CERTID は APR プールで管理されます。stapling_renew_response 内では一時的なサブプールを作成し、メモリリークを防いでいます。
  • **排他制御**: AP_SOCACHE_FLAG_NOTMPSAFE が設定されているキャッシュバックエンドを使用する場合、明示的なミューテックス操作が必須となります。
  • **制約**: MAX_STAPLING_DER（10KB）を超える OCSP レスポンスは切り捨てられます。
  • **省略箇所**: 本ファイルは OCSP プロトコルの詳細なパース（ASN.1 構造など）については OpenSSL API に依存しており、その内部実装は含まれていません。
