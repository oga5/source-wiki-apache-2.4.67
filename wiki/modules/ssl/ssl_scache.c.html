modules/ssl/ssl_scache.c

概要

本ファイルは、mod_ssl における SSL セッションキャッシュおよび OCSP ステープリングキャッシュの抽象化レイヤーを提供します。Apache HTTP Server の socache (Shared Object Cache) プロバイダーをラップし、SSL セッションの永続化と取得を統一的なインターフェースで管理します。

モジュール内での役割

mod_ssl のエンジンがセッションを保存・復元する際の仲介役です。具体的なキャッシュ実装（shmcb, dbm, memcache 等）に依存せず、SSLModConfigRec に格納されたプロバイダーを介して操作を委譲します。

責務

  • **キャッシュのライフサイクル管理**: サーバー起動時の初期化 (init) および終了時の破棄 (kill)。
  • **セッションのシリアライズ/デシリアライズ**: OpenSSL の SSL_SESSION 構造体と、キャッシュに保存可能なバイナリ形式（DER）の相互変換。
  • **スレッドセーフティの保証**: プロバイダーがスレッドセーフでない場合に、ssl_mutex を用いた排他制御を自動的に適用。
  • **ステータス表示**: mod_status と連携し、キャッシュの利用状況をサーバー管理画面に出力。

重要なシンボル

  • ssl_scache_init / ssl_scache_kill: キャッシュプロバイダーの初期化とクリーンアップ。
  • ssl_scache_store / ssl_scache_retrieve: セッションの保存と取得。内部で i2d_SSL_SESSION / d2i_SSL_SESSION を使用。
  • ssl_scache_remove: キャッシュからのセッション削除。
  • ssl_ext_status_hook: mod_status 用のフック関数。

制御フローとデータフロー

1. **初期化**: post_config 時に呼ばれ、設定された socache プロバイダーを init します。 2. **保存**: ssl_scache_store が呼ばれると、i2d_SSL_SESSION でセッションを DER 形式に変換し、プロバイダーの store メソッドを呼び出します。 3. **取得**: ssl_scache_retrieve が呼ばれると、プロバイダーからバイナリを取得し、d2i_SSL_SESSION で OpenSSL 構造体に復元します。 4. **排他制御**: AP_SOCACHE_FLAG_NOTMPSAFE フラグが立っている場合、操作の前後で ssl_mutex_on/off が実行されます。

直接依存

  • ssl_private.h: mod_ssl 内部の共有構造体およびマクロ定義。
  • mod_status.h: mod_status へのフック登録用。

関連コード

  • modules/ssl/ssl_engine_init.c: ssl_scache_init が呼び出されるコンテキスト。
  • modules/ssl/ssl_engine_mutex.c: 排他制御の実装詳細。
  • include/ap_socache.h: socache プロバイダーのインターフェース定義（Apache HTTP Server コア）。

読者向けメモ

  • 本ファイルはキャッシュの「抽象化」のみを行っており、実際の保存先（共有メモリやディスク）のロジックは socache プロバイダー側にあります。
  • ssl_scache_store でのシリアライズ時に MODSSL_SESSION_MAX_DER を超えるセッションはエラーとして破棄されます。
  • ssl_scache_init 内の AP_SQ_MS_CREATE_PRE_CONFIG チェックは、設定読み込みの二重実行を回避するための定型的なガードです。
