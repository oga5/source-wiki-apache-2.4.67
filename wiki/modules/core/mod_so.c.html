modules/core/mod_so.c

概要

mod_so.c は、Apache HTTP Server の実行時に共有オブジェクト（DSO: Dynamic Shared Object）を動的にロードおよびアンロードするためのコアモジュールです。サーバーを再起動することなく、LoadModule および LoadFile ディレクティブを通じて機能拡張を可能にします。

モジュール内での役割

Apache の設定フェーズにおいて、外部バイナリをプロセス空間にマッピングし、Apache のモジュール構造として統合するブリッジの役割を果たします。modules/core 内で DSO 関連の唯一の実行エンジンとして機能します。

責務

  • **DSO 管理**: apr_dso を利用した共有ライブラリのロードとシンボル解決。
  • **モジュール統合**: ロードした DSO を Apache の内部モジュールリストへ登録し、設定を適用。
  • **ライフサイクル管理**: 設定プール（pconf）に関連付けられたクリーンアップ処理による、サーバー終了時のモジュールアンロード。
  • **検証**: ロード対象のモジュールが Apache のバイナリ互換性（MODULE_MAGIC_COOKIE）を満たしているかの検証。

重要なシンボル

設定と状態管理

  • so_module: モジュールのメイン構造体。so_cmds と so_sconf_create を保持。
  • so_server_conf: ロードされたモジュールを追跡するための配列（loaded_modules）を保持するサーバー設定構造体。

コマンドハンドラ

  • load_module: LoadModule ディレクティブを処理。DSO のロード、シンボル解決、Apache への登録、設定の初期化を行う。
  • load_file: LoadFile ディレクティブを処理。任意の共有ライブラリをロードする。

内部ユーティリティ

  • dso_load: apr_dso_load をラップし、サーバールートからの相対パス解決やエラーハンドリングを行う。
  • unload_module: apr_pool_cleanup から呼び出され、モジュールをアンロードする。
  • ap_find_loaded_module_symbol: ロード済みモジュールの構造体ポインタを名前から検索する。

制御フローとデータフロー

1. **初期化**: so_sconf_create がサーバー設定を初期化し、loaded_modules 配列を確保。 2. **ロード処理**: load_module が呼び出されると、まず重複ロードをチェックし、次に dso_load でファイルをロード。 3. **シンボル解決**: apr_dso_sym を使用して、共有オブジェクト内の module 構造体シンボルを取得。 4. **統合**: ap_add_loaded_module で Apache のグローバルモジュールリストに追加し、ap_single_module_configure で設定を適用。 5. **クリーンアップ**: apr_pool_cleanup_register により、サーバー終了時に unload_module が実行されるよう予約。

直接依存

  • apr_dso.h: 動的ロードの抽象化レイヤー。

  • mod_so.h: オプション関数（ap_find_loaded_module_symbol）の宣言。

読者向けメモ

  • **メモリ所有権**: ロードされたモジュール情報は cmd->pool（通常は pconf）に紐付けられており、サーバーの再起動や終了時に自動的に解放されます。
  • **プラットフォーム依存**: NO_DLOPEN が定義されている環境では、動的ロード機能は無効化され、警告ログを出力するスタブに置き換わります。
  • **制約**: 組み込み済みの静的モジュールと名前が衝突する場合、ロードは拒否されます。
  • **未実装/浅い部分**: load_file は単にライブラリをロードするだけで、Apache モジュールとしての登録は行いません。また、エラーハンドリングは apr_dso_error を通じてプラットフォーム固有のメッセージに依存しています。
