modules/session/mod_session.c

概要

mod_session.c は、Apache HTTP Server におけるセッション管理の基盤を提供するコアモジュールです。セッションのロード、保存、有効期限管理、およびセッションデータのシリアライズ/デシリアライズの抽象化を担当します。具体的なストレージ（Cookie、DBD、暗号化など）はフックを通じて外部モジュールに委譲する設計となっています。

モジュール内での役割

本モジュールは、セッション機能の「オーケストレーター」です。リクエスト処理のライフサイクル（Fixups、Output Filter）に介入し、セッションデータの永続化とリクエスト間での共有を管理します。

責務

  • **セッションライフサイクル管理**: session_rec の生成、ロード、保存の制御。
  • **フックの提供**: ストレージ実装（session_load, session_save）およびエンコード/デコード処理の拡張ポイントを提供。
  • **設定管理**: ディレクトリ単位でのセッション有効化、有効期限（MaxAge）、パス制限（Include/Exclude）の設定。
  • **出力フィルタリング**: レスポンス送信直前にセッションを永続化するフィルタの提供。
  • **環境変数連携**: セッション情報を CGI 環境変数（HTTP_SESSION）へ注入する機能。

重要なシンボル

セッション操作（API）

  • ap_session_load: セッションをロードし、必要に応じて空の session_rec を作成。
  • ap_session_save: セッションの dirty フラグを確認し、変更がある場合のみ保存フックを呼び出す。
  • ap_session_get / ap_session_set: セッションデータへのアクセス用アクセサ。

フック（外部拡張用）

  • session_load / session_save: ストレージバックエンド（Cookie/DBD等）が実装。
  • session_encode / session_decode: データのシリアライズ形式を定義。

内部処理・ユーティリティ

  • session_identity_encode / session_identity_decode: デフォルトの URL エンコード形式によるシリアライズ処理。
  • session_included: 設定されたパス制限に基づき、セッション処理の対象か判定。

制御フローとデータフロー

1. **ロード**: ap_session_load が呼ばれると、session_included で対象パスか確認し、ap_run_session_load でバックエンドからデータを取得。その後 ap_run_session_decode で session_rec->entries を構築。 2. **変更**: ap_session_set が呼ばれると dirty フラグが立てられ、データが更新される。 3. **保存**: session_output_filter がリクエスト終了時に起動。ap_run_session_encode でシリアライズし、ap_run_session_save でバックエンドへ書き出す。 4. **データ構造**: session_rec は request_rec のプールに関連付けられ、リクエスト終了時に自動的に解放される。

直接依存

  • mod_session.h: セッション構造体定義とマクロ。
  • apr_lib.h, apr_strings.h: 文字列操作およびメモリ管理。
  • util_filter.h: 出力フィルタ処理。
  • http_log.h, http_request.h, http_protocol.h: Apache HTTPD コア API。

関連コード

  • mod_session.h: 全モジュールで共有されるデータ構造定義。
  • mod_session_cookie.c: Cookie を用いたセッションストレージ実装。
  • mod_session_dbd.c: データベースを用いたセッションストレージ実装。
  • mod_session_crypto.c: セッションデータの暗号化/復号処理。

読者向けメモ

  • **メモリ管理**: session_rec はリクエストプール（r->pool）を使用するため、明示的な解放は不要です。
  • **不変条件**: dirty フラグが立っていない場合、ap_session_save は保存処理をスキップしてパフォーマンスを最適化します。
  • **エッジケース**: 内部リダイレクトが発生した場合、session_output_filter は r->next を辿り、連鎖するすべてのリクエストのセッションを保存しようと試みます。
  • **省略された詳細**: 本ファイルはセッションの「管理」に特化しており、具体的な暗号化アルゴリズムやデータベースクエリの詳細は各サブモジュールに委譲されているため、このファイルには含まれていません。
