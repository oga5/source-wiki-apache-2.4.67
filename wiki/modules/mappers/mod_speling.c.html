modules/mappers/mod_speling.c

概要

mod_speling は、リクエストされた URL がサーバー上に存在しない場合に、スペルミスや大文字小文字の不一致を自動的に修正・補完する Apache モジュールです。ファイルシステムをスキャンし、類似した名前のファイルが見つかった場合にリダイレクト（301）または選択肢の提示（300）を行います。

モジュール内での役割

modules/mappers サブシステムの一部として、リクエスト処理の「修正フェーズ（Fixups）」に介入します。他のマッパーモジュールが URL をファイルパスに変換した後に動作し、ファイルが見つからない場合にのみスペルチェックのロジックを適用します。

責務

  • **設定管理**: CheckSpelling などのディレクティブを通じた動作制御。
  • **スペル判定**: 編集距離（挿入・削除・置換・転置）に基づく類似度計算。
  • **ファイル探索**: ディレクトリを走査し、候補となるファイル名リストを生成。
  • **レスポンス生成**: 修正候補が単一なら自動リダイレクト、複数なら 300 Multiple Choices を返却。

重要なシンボル

設定管理

  • spconfig: モジュールの動作設定（有効化、大文字小文字のみ、ベース名マッチング）を保持する構造体。
  • speling_cmds: CheckSpelling 等のディレクティブを定義する command_rec 配列。

スペル判定ロジック

  • sp_reason: 判定結果の品質（一致、大文字小文字違い、転置、欠落、余分、単純ミス、その他）。
  • spdist(): 2つの文字列を比較し、sp_reason を返す関数。tcsh のスペルチェックルーチンをベースに、大文字小文字を無視するように拡張されています。

リクエスト処理

  • check_speling(): fixups フックに登録されるメイン関数。リクエストの妥当性検証、ディレクトリ走査、候補のソート、レスポンスの決定を行う。
  • misspelled_file: 候補ファイル名とその品質を保持する構造体。

制御フローとデータフロー

1. **フック**: register_hooks により ap_hook_fixups に check_speling が登録されます。 2. **検証**: check_speling は GET メソッドかつファイルが存在しないリクエストのみを処理対象とします。 3. **探索**: リクエストされたパスの親ディレクトリを apr_dir_open で開き、apr_dir_read で全ファイルを走査します。 4. **評価**: 各ファイル名に対し spdist を適用し、類似度を判定。候補を apr_array_header_t に蓄積します。 5. **決定**:

  • 候補が 1 つかつ確実性が高い場合: Location ヘッダを付与し HTTP_MOVED_PERMANENTLY を返却。
  • 候補が複数ある場合: notes テーブルに候補リストを格納し HTTP_MULTIPLE_CHOICES を返却。

読者向けメモ

  • **メモリ管理**: r->pool を使用してメモリを割り当てており、リクエスト終了時に自動解放されます。sub_pool はレスポンス生成時の一時的なメモリ確保に使用されます。
  • **パフォーマンス**: ディレクトリ内の全ファイルを走査するため、ファイル数が多いディレクトリではパフォーマンスに影響を与える可能性があります。
  • **制約**: spdist は「1文字の誤り」までしか許容しません。また、シンボリックリンクのループを避けるため、リクエストと一致するファイルが見つかった場合は即座に OK を返して処理を終了します。
  • **省略された詳細**: HTML 生成部分の文字列構築ロジックは冗長ですが、これは当時の Apache の慣習に従ったものであり、本質的なロジックではありません。
