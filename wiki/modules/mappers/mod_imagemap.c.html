modules/mappers/mod_imagemap.c

概要

mod_imagemap.c は、クライアントから送信された座標情報に基づき、画像マップ（.map ファイル）内の定義に従ってリダイレクト先を決定する Apache HTTP Server モジュールです。NCSA の imagemap.c を起源とし、幾何学的な判定ロジックと、非グラフィカルな代替メニュー生成機能を備えています。

モジュール内での役割

modules/mappers サブシステムの一部として、リクエストの URI マッピングとハンドリングを担当します。IMAP_MAGIC_TYPE (application/x-httpd-imap) をハンドラとして認識し、リクエストされたリソースが画像マップファイルである場合に起動します。

責務

  • **設定管理**: ImapMenu, ImapDefault, ImapBase ディレクティブの解析とディレクトリ単位の設定保持。
  • **幾何学的判定**: poly (多角形), circle (円), rect (矩形), point (最近傍点) の各領域判定アルゴリズムの実行。
  • **URL 解決**: 相対パスや referer を含む複雑な URL の構築と正規化。
  • **メニュー生成**: 座標が指定されない場合や設定に応じた、HTML 形式の代替メニューの動的生成。

重要なシンボル

設定管理

  • imap_conf_rec: ディレクトリごとの設定を保持する構造体。
  • imap_cmds: Apache 設定ディレクティブの定義テーブル。
  • create_imap_dir_config / merge_imap_dir_configs: 設定のライフサイクル管理とマージロジック。

幾何学判定ロジック

  • pointinrect, pointincircle, pointinpoly: 座標が各図形に含まれるかを判定する関数。
  • is_closer: point 指令において、現在の座標が以前の候補より近いかを判定し、最小距離を更新する。

URL 処理・メニュー生成

  • imap_url: マップファイル内の値を絶対 URL に変換する。
  • menu_header, menu_directive, menu_footer: HTML メニュー生成用ヘルパー。

制御フローとデータフロー

1. **ハンドラ起動**: imap_handler がリクエストを検証し、imap_handler_internal へ委譲。 2. **設定読み込み**: ap_pcfg_openfile でマップファイルを開き、行単位で解析。 3. **座標取得**: get_x_coord / get_y_coord でクエリ文字列から座標を抽出。 4. **ループ処理**:

  • 各行を解析し、directive と value を分離。
  • showmenu フラグが有効な場合、HTML メニューを構築。
  • 無効な場合、幾何学判定関数を順次呼び出し、ヒットした時点で imap_reply を経由してリダイレクトを返す。

    • 5. **終了処理**: ap_cfg_closefile でファイルを閉じ、必要に応じてメニューのフッターを出力。

直接依存

  • apr.h, apr_strings.h: メモリ管理および文字列操作。
  • httpd.h, http_config.h, http_request.h: Apache モジュール基盤。
  • http_core.h, http_protocol.h: リクエスト処理とレスポンス生成。

関連コード

  • modules/mappers/mod_rewrite.c: より高度な URI 書き換えを行うモジュール。
  • util_script.h: CGI 関連のユーティリティ（本モジュールでも一部利用）。

読者向けメモ

  • **メモリ管理**: r->pool を使用してリクエスト単位でメモリを割り当てています。imap_url 等で生成される文字列はすべてプールに依存するため、明示的な解放は不要です。
  • **制約**: MAXVERTS (100) 以上の頂点を持つ多角形は処理できません。
  • **エッジケース**: 座標が指定されない場合、デフォルトでメニューを表示する挙動をとります。ImapMenu none を設定することでこの挙動を抑制可能です。
  • **設計の注意点**: imap_handler_internal はスタック上に大きな配列 (pointarray) を確保するため、リクエストごとにスタック消費量が増える点に注意してください。
  • **省略箇所**: 本ドキュメントでは、各幾何学アルゴリズムの数学的詳細（pointinpoly の交差判定ロジック等）については、ソースコードのコメントが十分に詳細であるため省略しています。
