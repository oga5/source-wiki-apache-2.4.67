modules/mappers/mod_negotiation.c

概要

mod_negotiation.c は、HTTP コンテンツネゴシエーション（多言語、多形式対応）を実装する Apache HTTP Server モジュールです。クライアントの Accept-* ヘッダーとサーバー側のリソース（ファイルまたはタイプマップ）を照合し、最適なバリアントを選択します。

モジュール内での役割

modules/mappers サブシステムにおいて、リクエストされた URI が複数のバリアントを持つ場合に、どのバリアントを返すかを決定する「タイプアービトレーション（調停）」を担当します。

責務

  • **設定管理**: LanguagePriority や ForceLanguagePriority などのディレクティブを処理し、ディレクトリ単位のネゴシエーション設定を管理する。
  • **ヘッダー解析**: Accept, Accept-Language, Accept-Encoding, Accept-Charset ヘッダーを解析し、品質値（q-value）を抽出する。
  • **バリアント収集**: type-map ファイルの解析、またはディレクトリ内のファイルリスト（MultiViews）から利用可能なバリアントを列挙する。
  • **選択アルゴリズム**: RFC 2296 (RVSA/1.0) または Apache 独自のアルゴリズムを用いて、最適なバリアントを決定する。
  • **レスポンス生成**: Vary, Alternates, TCN ヘッダーの付与、および 300 (Multiple Choices) や 406 (Not Acceptable) のレスポンスを生成する。

重要なシンボル

設定構造体

  • neg_dir_config: ディレクトリごとのネゴシエーション設定（言語優先度など）。
  • negotiation_state: リクエストごとのネゴシエーション状態を保持するコンテキスト。

データレコード

  • accept_rec: クライアントが提示した Accept ヘッダーの各要素（MIMEタイプ、品質値など）。
  • var_rec: サーバー側で利用可能な各バリアントの属性（ファイル名、言語、エンコーディング、品質値など）。

フック関数

  • handle_multi: type_checker フック。MultiViews 処理の起点。
  • handle_map_file: handler フック。application/x-type-map ファイルの処理。
  • fix_encoding: fixups フック。エンコーディング名の正規化（x- プレフィックスの調整）。

制御フローとデータフロー

1. **初期化**: parse_accept_headers() がリクエストヘッダーを解析し、negotiation_state を構築。 2. **バリアント列挙**:

  • read_type_map(): type-map ファイルを読み込み、var_rec 配列を生成。
  • read_types_multi(): ディレクトリをスキャンし、一致するファイルを sub_req として検証・列挙。

    • 3. **評価**: best_match() が各バリアントの品質値を計算し、is_variant_better() (または is_variant_better_rvsa()) を用いて比較。 4. **決定**: 選択されたバリアントに対し、setup_choice_response() がヘッダーを調整し、ap_internal_fast_redirect() でリクエストを転送。

直接依存

  • httpd.h, http_config.h, http_request.h: Apache コア API。
  • apr_*.h: APR (Apache Portable Runtime) によるメモリ管理、ファイル I/O、文字列操作。

関連コード

  • mod_mime.c: ファイル拡張子から MIME タイプや言語を決定するモジュール。mod_negotiation はその結果（ap-mime-exceptions-list など）に依存する。

読者向けメモ

  • **メモリ管理**: すべての動的メモリは request_rec のプール (r->pool) から割り当てられます。
  • **不変条件**: var_rec 内の MIME タイプや言語タグは、解析時に小文字に変換されることが前提です。
  • **エッジケース**: ForceLanguagePriority が設定されている場合、標準の品質値計算を上書きするロジックが働きます。
  • **注意点**: type-map ファイル内のコメント処理や、Body: タグによるファイル内コンテンツの埋め込み処理は、ファイルポインタのシークを伴うため、複雑なエラーパスが存在します。
  • **省略**: 本ドキュメントでは、RFC 2296 の詳細な仕様や、各ヘッダーの品質値計算における細かい「フィドル（調整）」ロジックの詳細は省略しています。これらは主にブラウザの不完全な実装を補完するためのものです。
