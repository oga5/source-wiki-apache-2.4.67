modules/mappers/mod_alias.c

概要

mod_alias.c は、Apache HTTP Server において URI をファイルシステム上のパスや別の URL へマッピングする機能を提供します。Alias、ScriptAlias、Redirect などのディレクティブを処理し、リクエストの翻訳フェーズおよび修正フェーズで動作します。

モジュール内での役割

modules/mappers サブシステムの一部として、リクエストの URI を物理的なファイルパス（Alias）または外部リダイレクト（Redirect）へ変換する役割を担います。他のマッパーモジュール（mod_rewrite や mod_userdir）と連携し、translate_name フックの順序を制御することで、適切な優先順位でマッピングを実行します。

責務

  • **設定管理**: サーバー全体およびディレクトリごとの設定構造体（alias_server_conf, alias_dir_conf）の生成とマージ。
  • **ディレクティブ処理**: Alias や Redirect などの設定を解析し、内部の配列や式（ap_expr）として保持。
  • **リクエスト翻訳**: translate_name フックにて URI をファイルパスへ変換。
  • **リダイレクト処理**: fixups フックにてリダイレクト先を決定し、HTTP レスポンスヘッダを構築。
  • **正規表現サポート**: AliasMatch 等による柔軟な URI マッチングと置換。

重要なシンボル

設定構造体

  • alias_entry: 個別のエイリアス/リダイレクト定義（正規表現、パス、ハンドラ情報）。
  • alias_server_conf: サーバー全体の設定（Alias と Redirect のリスト）。
  • alias_dir_conf: ディレクトリごとの設定（式ベースのエイリアス、リダイレクト設定）。

設定・登録関数

  • create_alias_config / create_alias_dir_config: 設定構造体の初期化。
  • merge_alias_config / merge_alias_dir_config: 親コンテキストの設定を継承・マージ。
  • register_hooks: translate_name および fixups フックの登録。

処理ロジック

  • add_alias_internal / add_redirect_internal: ディレクティブ解析の共通コア。
  • try_alias / try_redirect: ディレクトリ設定に基づく解決。
  • try_alias_list: 設定リストを走査し、マッチするエイリアスを検索。
  • translate_alias_redir: translate_name フックのメインエントリポイント。

制御フローとデータフロー

1. **設定フェーズ**: alias_cmds で定義されたディレクティブが add_alias 等を呼び出し、alias_entry を apr_array_header_t に追加します。 2. **翻訳フェーズ (translate_alias_redir)**:

  • まず try_redirect でリダイレクトを試行。
  • 次に try_alias_list でサーバーレベルのエイリアスを走査。
  • マッチした場合、r->filename を更新し OK を返します。

    • 3. **修正フェーズ (fixup_redir)**:

  • fixups フックにて、ディレクトリ単位で定義されたリダイレクトを処理し、Location ヘッダを生成します。

直接依存

  • apr_strings.h, apr_lib.h: 文字列操作およびメモリ管理。
  • httpd.h, http_config.h, http_request.h: Apache のコア API。
  • ap_expr.h: 式評価（ap_expr）による動的なエイリアス/リダイレクトのサポート。

関連コード

  • modules/mappers/mod_rewrite.c: より高度な URI 書き換え機能を提供。
  • modules/mappers/mod_userdir.c: ユーザーディレクトリへのマッピング。
  • modules/mappers/mod_vhost_alias.c: バーチャルホストベースのエイリアス。

読者向けメモ

  • **メモリ管理**: すべての設定およびリクエスト処理は apr_pool_t を使用しており、リクエスト終了時に自動的に解放されます。
  • **パフォーマンス**: try_alias_list はリストを線形探索するため、エイリアスの数が多い場合はパフォーマンスに影響する可能性があります。
  • **制約**: Alias ディレクティブは Directory コンテキスト内では使用できません（add_alias_internal 内の ap_check_cmd_context でチェック）。
  • **未実装/浅い部分**: コード内のコメントにある通り、ap_server_root_relative の使用やパスの正規化において、一部の古い実装が残っており、最適化の余地があります。また、PREGSUB_ERROR はエラー発生時のマジック値として使用されており、呼び出し元でのチェックが必須です。
