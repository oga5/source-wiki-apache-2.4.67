modules/aaa/mod_authnz_fcgi.c

概要

mod_authnz_fcgi.c は、FastCGI プロトコルを介して外部の認証（Authn）および認可（Authz）サーバーと通信するための Apache HTTP Server モジュールです。このモジュールは、Apache の認証・認可フレームワークを FastCGI Authorizer ロールにブリッジし、外部プロセスがリクエストのアクセス制御を決定できるようにします。

モジュール内での役割

modules/aaa サブシステムにおいて、本モジュールは「外部プロバイダー連携」を担当します。標準的なファイルベースやデータベースベースの認証とは異なり、FastCGI プロトコルを用いて動的な外部サービスへ判定を委譲する役割を担います。

責務

  • **プロバイダー定義**: AuthnzFcgiDefineProvider を通じた FastCGI バックエンドの登録と管理。
  • **リクエストの仲介**: Apache の認証・認可フェーズを FastCGI プロトコル（FCGI_AUTHORIZER ロール）に変換。
  • **環境変数の受け渡し**: CGI 変数および認証情報を FastCGI プロトコル経由でバックエンドへ転送。
  • **レスポンス解析**: バックエンドからの FastCGI レスポンスを解析し、HTTP ステータスやヘッダーを Apache のリクエスト処理に反映。

重要なシンボル

設定構造体

  • fcgi_provider_conf: バックエンドの接続情報（ホスト、ポート、アドレス）と役割（Authn/Authz）を保持。
  • fcgi_dir_conf: ディレクトリごとの設定（プロバイダー名、ユーザー式、権限設定）。

認証・認可プロバイダー

  • fcgi_authn_provider: check_password を実装し、認証フェーズを処理。
  • fcgi_authz_provider: check_authz を実装し、認可フェーズを処理。

主要な処理関数

  • req_rsp: FastCGI 通信のメインループ。接続、環境変数送信、レスポンス受信を統括。
  • handle_response: FastCGI レコード（STDOUT, STDERR, END_REQUEST）を解析し、スクリプトヘッダーを抽出。
  • connect_to_peer: バックエンドへの TCP ソケット接続を確立。

制御フローとデータフロー

1. **初期化**: fcgi_pre_config でプロバイダー格納用のハッシュテーブルを作成。 2. **フック**: fcgi_check_authn が認証フェーズで呼び出され、設定されたプロバイダーを検索。 3. **通信**: req_rsp が呼び出されると、setupenv で環境変数を準備し、send_begin_request と send_environment で FastCGI リクエストを送信。 4. **解析**: handle_response がバックエンドからのストリームを読み込み、handle_headers で HTTP ヘッダーの終端を特定。 5. **反映**: 成功時は r->user の設定や subprocess_env へのヘッダー反映を行い、失敗時はバックエンドのレスポンスをクライアントへ転送。

直接依存

  • util_fcgi.h: FastCGI プロトコルヘッダーおよびエンコード/デコード用ユーティリティ。
  • mod_auth.h: Apache 認証フレームワークのインターフェース。
  • apr_hash.h, apr_socket.h: APR によるハッシュ管理とネットワーク通信。

関連コード

  • modules/proxy/mod_proxy_fcgi.c: FastCGI 通信のヘッダー解析ロジックの参照元。
  • modules/aaa/mod_auth_basic.c: 認証フェーズの順序制御で依存関係がある。

読者向けメモ

  • **メモリ管理**: req_rsp 内では temp_pool を使用し、リクエスト処理ごとの一時的なメモリ消費を抑えています。
  • **制約**: 本モジュールは AF_UNIX ソケットを現時点ではサポートしておらず、TCP のみに対応しています。
  • **エラーパス**: バックエンドがダウンしている場合や、ヘッダー解析に失敗した場合は HTTP_INTERNAL_SERVER_ERROR を返します。
  • **設計の注意点**: handle_response 内の goto recv_again は、FastCGI レコードが分割されて送信された場合を考慮した再帰的な読み込み処理です。
  • **省略箇所**: ap_log_rdata などのトレースログ出力は、デバッグビルド時のみ有効であり、本質的なロジックからは除外して解釈可能です。
