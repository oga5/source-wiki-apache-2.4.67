modules/aaa/mod_access_compat.c

概要

mod_access_compat は、Apache HTTP Server 2.2 以前のアクセス制御構文（Order, Allow, Deny, Satisfy）を 2.4 以降の環境で利用可能にするための互換性モジュールです。2.4 で導入された mod_authz_host の新しい構文への移行を支援する役割を持ちます。

モジュール内での役割

modules/aaa サブシステムにおいて、レガシーな設定ディレクティブを現在の ap_hook_check_access フックに変換するブリッジとして機能します。

責務

  • **設定管理**: Order, Allow, Deny, Satisfy ディレクティブの解析と、ディレクトリごとの設定構造体への格納。
  • **アクセス判定**: リクエスト時に、設定された順序（allow,deny 等）に基づき、環境変数、IPアドレス、ホスト名によるアクセス可否を評価。
  • **互換性提供**: access_compat_ap_satisfies をオプション関数として公開し、他のモジュールが本モジュールの Satisfy 設定を参照できるようにする。

重要なシンボル

設定構造体

  • access_compat_dir_conf: ディレクトリごとの設定を保持。order（判定順序）、allows/denys（ルール配列）、satisfy（ポリシー）を管理。

コマンドハンドラ

  • order, allow_cmd, satisfy: 設定ファイル内のディレクティブを解析し、access_compat_dir_conf を構築するハンドラ。

判定ロジック

  • check_dir_access: ap_hook_check_access に登録されるメインのエントリポイント。設定された order に基づき find_allowdeny を呼び出す。
  • find_allowdeny: 特定のメソッドに対するルール（allow または deny）を走査し、条件（IP, ホスト, 環境変数）に一致するか判定。
  • in_domain: ホスト名が指定ドメインに属するかを検証する補助関数。

制御フローとデータフロー

1. **設定フェーズ**: create_access_compat_dir_config でメモリを確保。allow_cmd が apr_array_push を用いて allows または denys 配列にルールを追加。 2. **リクエスト処理**:

  • check_dir_access がリクエストごとに呼び出される。
  • order 設定（DENY_THEN_ALLOW 等）に従い、find_allowdeny を適切な順序で実行。
  • find_allowdeny は r->useragent_addr（IP）や ap_get_useragent_host（ホスト名）を使い、ルール配列を評価。

    • 3. **終了処理**: 判定結果が HTTP_FORBIDDEN の場合、ap_log_rerror でログを出力。

読者向けメモ

  • **メモリ管理**: 設定構造体は apr_pool_t を介して管理され、リクエスト終了時に自動的に解放されます。
  • **制約**: Order の設定はメソッド単位で行われます。Satisfy は access_compat_ap_satisfies を通じて他のモジュールから参照可能ですが、本モジュール自体は check_dir_access で完結するロジックを持っています。
  • **注意点**: in_domain は単純な文字列比較に基づいているため、DNS 逆引きの結果に依存します。REMOTE_DOUBLE_REV を使用しており、セキュリティ上の考慮が必要です。
  • **省略箇所**: 本ドキュメントでは、apr_ipsubnet_t の内部構造や、ap_get_useragent_host の詳細な DNS 解決プロセスについては、Apache コアの責務であるため省略しています。
