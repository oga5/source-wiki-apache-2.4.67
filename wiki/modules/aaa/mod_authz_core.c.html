modules/aaa/mod_authz_core.c

概要

mod_authz_core.c は、Apache HTTP Server の認可（Authorization）サブシステムの中心となるモジュールです。個別の認可ロジック（mod_authz_user 等）を統合し、Require ディレクティブによる複雑な論理構造（AND/OR/NOT）を管理します。

モジュール内での役割

認可プロバイダーの登録・管理および、リクエスト処理時の認可フックの実行を担当します。他の認可モジュールが提供する機能を、設定ファイル上の階層構造に従って実行する「オーケストレーター」として機能します。

責務

  • **設定管理**: Require や <RequireAll> などのディレクティブを解析し、ディレクトリごとの設定構造（authz_core_dir_conf）を構築・マージする。
  • **プロバイダー管理**: 認可プロバイダーの登録、および AuthzProviderAlias によるプロバイダーのエイリアス化と再設定。
  • **認可実行**: リクエスト処理時に、設定された論理ツリーを再帰的に評価し、最終的な認可結果（Granted/Denied/Neutral）を決定する。
  • **コアプロバイダーの提供**: env, all, method, expr といった基本的な認可プロバイダーの実装。

重要なシンボル

設定構造体

  • authz_core_dir_conf: ディレクトリごとの認可設定を保持。論理演算子（authz_logic_op）と認可セクションのツリーを持つ。
  • authz_section_conf: 認可の論理単位。プロバイダー、論理演算子、否定フラグ、子セクションへのポインタを保持。
  • provider_alias_rec: AuthzProviderAlias で定義されたエイリアスの設定を保持。

認可ロジック・実行

  • apply_authz_sections: 認可ツリーを再帰的に走査し、各プロバイダーの check_authorization を呼び出して結果を統合するコア関数。
  • authorize_user_core: 認可フックのメインエントリーポイント。認可結果に基づき HTTP ステータスコード（200, 401, 403）を決定する。

プロバイダー定義

  • authz_provider: 認可プロバイダーのインターフェース構造体。check_authorization 関数ポインタを保持。
  • authz_env_provider, authz_all_provider, authz_method_provider, authz_expr_provider: 本モジュールが提供する標準プロバイダー。

制御フローとデータフロー

1. **設定フェーズ**: add_authz_provider や add_authz_section が設定ファイルを読み込み、authz_section_conf のツリーを構築する。 2. **検証フェーズ**: authz_core_check_config が起動時に呼び出され、論理的に矛盾した設定（例: 否定のみのセクション）がないか検証する。 3. **リクエスト処理**:

  • authorize_user フックがリクエストを受け取る。
  • apply_authz_sections がツリーを再帰的に走査。
  • section->provider->check_authorization を呼び出し、結果を AUTHZ_GRANTED 等で受け取る。
  • section->op（AND/OR）に従って結果をマージし、最終的な authz_status を算出する。

読者向けメモ

  • **メモリ管理**: 設定構造体は apr_pool_t を使用して割り当てられ、リクエスト処理中のデータは request_rec->pool を使用します。
  • **再帰的評価**: apply_authz_sections は再帰関数です。深いネストを持つ設定はスタック消費に注意が必要です。
  • **エラーパス**: AUTHZ_DENIED_NO_USER は、認証がまだ完了していない場合に認証プロセスへフォールバックさせるための重要なステータスです。
  • **省略された詳細**: 本ファイルは認可の「枠組み」に特化しており、個別の認証方式（LDAP, DB等）の詳細は各サブモジュールに委譲されています。
