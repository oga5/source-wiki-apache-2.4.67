modules/aaa/mod_auth_form.c

概要

mod_auth_form.c は、Apache HTTP Server におけるフォームベース認証（ログインフォームを使用した認証）を実装するモジュールです。ユーザーがログインフォームに入力した認証情報をセッション（mod_session 経由）に保持し、以降のアクセスで認証状態を維持します。

モジュール内での役割

modules/aaa サブシステムにおいて、従来の Basic/Digest 認証とは異なり、アプリケーション層のフォーム入力に基づいた認証フローを提供します。mod_session および mod_request に強く依存しており、これらと連携して認証情報の抽出、保持、およびリクエストの再構築を行います。

責務

  • **設定管理**: ディレクトリ単位での認証プロバイダー、フォームフィールド名、リダイレクト先などの設定保持とマージ。
  • **認証フローの実行**: check_authn を通じた認証プロバイダーの呼び出しと、結果に応じたアクセス制御。
  • **セッション管理**: 認証情報のセッションへの保存・取得（mod_session 依存）。
  • **フォーム解析**: POST リクエストからユーザー名・パスワードを抽出するためのサブリクエスト処理。
  • **ハンドラ提供**: ログイン、ログアウト、および認証成功後のリダイレクト処理を行うためのハンドラ実装。

重要なシンボル

設定構造体

  • auth_form_config_rec: ディレクトリごとの認証設定（プロバイダーリスト、フォームフィールド名、リダイレクト先など）を保持。

設定・初期化

  • create_auth_form_dir_config: 設定構造体の初期化（デフォルト値の設定）。
  • merge_auth_form_dir_config: 親ディレクトリの設定と現在の設定をマージ。
  • add_authn_provider: AuthFormProvider ディレクティブの解析とプロバイダーの登録。

認証・セッション操作

  • get_form_auth: POST リクエストからフォームデータを解析し、認証情報を抽出。
  • check_authn: 抽出された認証情報を基に、登録された認証プロバイダーを順次実行。
  • set_session_auth / get_session_auth: セッションへの認証情報の永続化と取得。

ハンドラ

  • authenticate_form_login_handler: ログインフォームの送信先ハンドラ。
  • authenticate_form_logout_handler: ログアウト処理ハンドラ。
  • authenticate_form_redirect_handler: 認証成功後のリダイレクトおよびリクエスト再構築ハンドラ。

制御フローとデータフロー

1. **認証チェック**: authenticate_form_authn がリクエストごとに呼ばれ、セッションまたはリクエストノートから認証情報を確認します。 2. **フォーム解析**: 認証情報がない場合、POST リクエストであれば get_form_auth を呼び出します。ここではサブリクエストを作成し、mod_request のフィルタを使用してフォームボディを安全に解析します。 3. **認証実行**: check_authn が設定された認証プロバイダーをループで呼び出し、認証結果（AUTH_GRANTED 等）を判定します。 4. **状態維持**: 認証成功時、set_session_auth でセッションに情報を書き込みます。 5. **リダイレクト**: ログイン成功後、設定に応じて指定された URL へリダイレクトします。

直接依存

  • mod_auth.h: 認証プロバイダーインターフェース。
  • mod_session.h: セッション管理機能（ap_session_load, get, set）。
  • mod_request.h: フォーム解析およびフィルタ操作。
  • ap_expr.h: リダイレクト先などの動的式評価。

関連コード

  • modules/session/mod_session.c: セッションデータの保存先。
  • modules/filters/mod_request.c: フォーム解析の基盤。
  • modules/aaa/mod_authn_core.c: 認証プロバイダーの管理。

読者向けメモ

  • **メモリ管理**: request_rec のプールを使用してメモリを割り当てており、サブリクエストのライフサイクルはメインリクエストに依存します。
  • **セキュリティ**: プロキシリクエストでの使用は XSS リスクのため禁止されています。また、AuthFormDisableNoStore を有効にする際は、ブラウザのキャッシュによる認証情報漏洩のリスクを考慮する必要があります。
  • **制約**: フォーム解析のためにサブリクエストを作成するため、リクエストボディの読み取りには一定のオーバーヘッドが発生します。
  • **省略**: ap_session_* などのオプション関数は実行時に APR_RETRIEVE_OPTIONAL_FN で解決されるため、mod_session がロードされていない場合はエラーとなります。
