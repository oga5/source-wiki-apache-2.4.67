modules/http/http_filters.c

概要

HTTP プロトコルにおける入力フィルタリング（リクエストボディの解析）および出力フィルタリング（レスポンスヘッダの構築・送信）を担う中核ファイルです。Apache HTTP Server のフィルタチェーンにおいて、HTTP 固有のセマンティクス（チャンク転送、Content-Length 制限、ヘッダの検証）を適用します。

モジュール内での役割

modules/http モジュールにおいて、プロトコルレベルのデータ変換と検証を行うゲートキーパーとして機能します。http_protocol.c がリクエストの初期解析を行うのに対し、本ファイルはストリーム処理中のフィルタとして、データがクライアントとハンドラの間を流れる際の整合性を保証します。

責務

  • **入力フィルタリング**: ap_http_filter を通じて、チャンク転送のデコード、Content-Length による境界管理、およびリクエストボディのサイズ制限（LimitRequestBody）を強制する。
  • **出力ヘッダ処理**: ap_http_header_filter を通じて、レスポンスヘッダの結合、Vary フィールドの正規化、HTTP ステータスラインの検証、およびプロトコルバージョンの決定を行う。
  • **クライアントブロック管理**: ap_setup_client_block 等の API を提供し、ハンドラがリクエストボディを安全に読み取れるよう抽象化する。
  • **エラーハンドリング**: プロトコル違反やリソース制限超過時に、適切な HTTP エラーレスポンスを生成し、接続を閉じる（Keep-Alive の無効化）。

重要なシンボル

コンテキスト構造体

  • http_ctx_t: 入力フィルタの状態管理用。現在のボディ解析状態（state）、残りのバイト数、チャンク解析の進捗を保持。
  • header_struct: ヘッダ送信時に apr_table_do へ渡すためのバッファとプールを保持。
  • header_filter_ctx: 出力フィルタのヘッダ送信済みフラグを保持。

入力フィルタ系

  • ap_http_filter: メインの入力フィルタ。apr_bucket_brigade を介してリクエストボディを処理。
  • parse_chunk_size: チャンクヘッダを解析し、オーバーフローや不正なフォーマットを検出。
  • read_chunked_trailers: チャンク終了後のトレーラヘッダを読み取り、リクエストヘッダにマージ。

出力フィルタ系

  • ap_http_header_filter: レスポンスヘッダを構築し、フィルタチェーンの先頭で送信する。
  • merge_response_headers: headers_out と err_headers_out を統合し、キャッシュ制御や Vary の修正を行う。
  • basic_http_header: ステータスライン、Date、Server ヘッダを生成。

制御フローとデータフロー

1. **入力**: ap_http_filter が ap_get_brigade を呼び出し、下位フィルタからデータを取得。ctx->state に基づき、チャンク解析または長さ制限付き読み込みを実行。エラー時は bail_out_on_error で接続を終了。 2. **出力**: ap_http_header_filter が呼び出されると、まずヘッダを統合・検証。ctx->headers_sent が未設定であれば、basic_http_header でヘッダを構築し、ap_pass_brigade で下位へ渡す。その後、必要に応じて CHUNK フィルタを挿入する。

読者向けメモ

  • **メモリ管理**: apr_bucket_brigade はリクエストプール（r->pool）または接続プール（f->c->bucket_alloc）を使用します。フィルタ内でバケットを削除・挿入する際は、メモリリークや二重解放に注意してください。
  • **状態遷移**: http_ctx_t の state は非常に複雑です。特にチャンク解析中の BODY_CHUNK_CR や BODY_CHUNK_LF などの細かい状態は、RFC 7230 準拠の厳密な解析のために存在します。
  • **省略された詳細**: 本ファイルはフィルタのロジックに集中しており、mod_proxy 固有の複雑な挙動や、EBCDIC 環境特有の文字変換処理は詳細を省略しています。これらは apr_xlate 関連の関数呼び出しとして現れます。
