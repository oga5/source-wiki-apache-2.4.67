modules/http/byterange_filter.c

概要

HTTP プロトコルにおける Range リクエストを処理し、レスポンスを適切なバイト範囲に切り出す出力フィルタを実装します。Apache HTTP Server のフィルタチェーンの一部として動作し、クライアントが要求した特定のデータ範囲のみを抽出して送信します。

モジュール内での役割

modules/http サブシステムにおいて、コンテンツの断片化（Byte-range）を担うフィルタです。http_filters.c などで構築されるフィルタチェーンに挿入され、レスポンスボディが送信される直前に介入します。

責務

  • **範囲解析**: Range ヘッダーを解析し、有効なバイト範囲を特定する。
  • **バリデーション**: RFC 2616 に基づき、不正な範囲や重複、過剰な範囲指定をチェックし、必要に応じて 416 (Range Not Satisfiable) を返す。
  • **コンテンツ変換**:
  • 単一範囲の場合: Content-Range ヘッダーを付与してデータを送信。
  • 複数範囲の場合: multipart/byteranges 形式に変換し、境界文字列（boundary）を挿入して送信。
  • **フィルタ管理**: 処理対象外のレスポンスやエラー時には、フィルタチェーンから自身を削除して透過的に振る舞う。

重要なシンボル

フィルタ処理

  • ap_byterange_filter: フィルタのメインエントリーポイント。ap_filter_t を通じて呼び出され、バケットブリゲードを操作する。

範囲解析・管理

  • ap_set_byterange: Range ヘッダーを解析し、indexes_t 配列を生成する。重複や逆転のカウントを行い、制限値を超えた場合は処理を中断する。
  • indexes_t: 範囲の開始位置と終了位置を保持する構造体。

バケット操作

  • copy_brigade_range: 指定された範囲のバケットを元のブリゲードからコピーし、新しいブリゲードに格納する。バケットの分割（apr_bucket_split）を伴う。
  • send_416: 範囲指定が不適切な場合に、416 エラーバケットを作成して送信する。

制御フローとデータフロー

1. **初期化**: ap_byterange_filter が呼び出され、ブリゲード内の全バケットの長さを計算。EOS（End of Stream）が存在しない場合や長さが不明な場合は、フィルタを削除して終了。 2. **解析**: ap_set_byterange が Range ヘッダーを解析。設定された制限値（max_ranges 等）を超えた場合は、元のブリゲードをそのまま通過させる。 3. **抽出**: copy_brigade_range をループで呼び出し、各範囲に対応するバケットを tmpbb にコピー。 4. **構築**:

  • 単一範囲なら Content-Range ヘッダーを更新。
  • 複数範囲なら multipart/byteranges ヘッダーを構築し、各範囲の間に境界文字列を挿入。

    • 5. **送信**: ap_pass_brigade を通じてフィルタチェーンの下流へデータを渡す。

読者向けメモ

  • **メモリ管理**: r->pool を使用してメモリを確保しており、リクエスト終了時に自動解放されます。
  • **パフォーマンス**: 範囲が非常に多い場合、copy_brigade_range によるバケットのコピーと分割が発生するため、メモリと CPU のオーバーヘッドが増加します。そのため AP_DEFAULT_MAX_RANGES 等の制限が設けられています。
  • **制約**: バケットの長さが不明（-1）な場合、このフィルタは安全のために処理をスキップします。
  • **省略箇所**: EBCDIC 変換（util_ebcdic.h）に関連する処理は、プラットフォーム依存の文字コード変換として実装されていますが、本ページでは主要なロジックに焦点を当てています。
