modules/http/mod_mime.c

概要

mod_mime は、ファイル名拡張子に基づいて MIME タイプ、エンコーディング、言語、ハンドラ、およびフィルタを決定する Apache HTTP Server モジュールです。リクエスト処理の type_checker フェーズで動作し、ファイルシステム上のリソースに適切なメタデータを付与します。

モジュール内での役割

modules/http サブシステムにおいて、リクエストされたリソースの「正体」を定義する役割を担います。http_protocol.c などがリクエストを処理する前に、このモジュールが request_rec 構造体のメタデータフィールドを適切に設定することで、後続のフィルタやハンドラが正しい処理を選択できるようにします。

責務

  • **設定管理**: ディレクトリ単位の MIME マッピング（AddType, AddEncoding 等）の保持とマージ。
  • **メタデータ解決**: ファイル名拡張子を解析し、content_type, content_encoding, content_languages などを設定。
  • **フィルタ/ハンドラ登録**: 拡張子に基づいた入力/出力フィルタの動的追加。
  • **設定ファイル読み込み**: TypesConfig ディレクティブで指定された MIME 定義ファイルのパース。

重要なシンボル

設定構造体

  • mime_dir_config: ディレクトリごとの設定を保持。拡張子マッピングのハッシュテーブルと、継承/削除ルールを管理。
  • extension_info: 特定の拡張子に関連付けられたメタデータ（タイプ、言語、フィルタ等）の集合。

コマンドハンドラ

  • add_extension_info: AddType 等のディレクティブを処理し、extension_info を更新。
  • remove_extension_info: 特定の拡張子の設定を無効化する予約エントリを作成。
  • multiviews_match: MultiviewsMatch ディレクティブの解析。

処理ロジック

  • find_ct: type_checker フックの本体。ファイル名から拡張子を抽出し、マッピングを適用。
  • analyze_ct: Content-Type ヘッダ文字列を解析し、パラメータ（charset等）を抽出・再構築。
  • mime_post_config: サーバー起動時に mime.types ファイルを読み込み、グローバルなマッピングを構築。

制御フローとデータフロー

1. **初期化**: mime_post_config が mime.types を読み込み、グローバルハッシュ mime_type_extensions を作成。 2. **リクエスト時**: find_ct が呼び出される。

  • r->filename から拡張子を順次抽出。
  • mime_dir_config のハッシュテーブルを検索し、一致する extension_info を取得。
  • 見つかったメタデータを request_rec に適用。

    • 3. **パラメータ処理**: analyze_ct が Content-Type を解析し、charset などのパラメータを正規化して再構築。 4. **終了**: 最終的に r->content_type 等が設定され、OK を返して終了。

読者向けメモ

  • **メモリ管理**: apr_pool_t を使用。設定の継承時には apr_hash_merge や apr_pmemdup を用いて、親設定を破壊しないよう深いコピーが行われます。
  • **エッジケース**: 拡張子が複数ある場合（例: file.tar.gz）、右から左へ順次解析されます。RemoveType 等の削除系ディレクティブは、マージ時に remove_mappings 配列を走査して適用されるため、設定の順序に依存します。
  • **制限事項**: analyze_ct は非常に詳細なパーサーですが、複雑な MIME パラメータのパースはエラーパスが多く、不正なヘッダに対してはログを出力してフォールバックします。
  • **未実装/浅い部分**: XXXX コメントにある通り、EBCDIC 環境での文字コード処理には一部課題が残っています。また、フィルタの重複チェックは行われていないため、設定次第で同じフィルタが複数回追加される可能性があります。
