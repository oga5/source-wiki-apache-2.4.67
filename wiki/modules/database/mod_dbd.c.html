modules/database/mod_dbd.c

概要

mod_dbd.c は、Apache HTTP Server におけるデータベース接続管理の基盤を提供するモジュールです。APR (Apache Portable Runtime) の apr_dbd フレームワークをラップし、接続プーリング、SQL 文の事前準備、およびリクエスト/コネクション単位のライフサイクル管理を実装します。

モジュール内での役割

Apache の他のモジュール（mod_authn_dbd など）がデータベースへアクセスするための共通インターフェースを提供します。設定フェーズで定義されたデータベース接続設定を保持し、実行時にスレッドセーフな接続プール（apr_reslist）を介して接続を提供します。

責務

  • **設定管理**: DBDriver や DBDParams などのディレクティブを解析し、サーバーごとの設定（svr_cfg）を構築・マージする。
  • **接続プーリング**: スレッド環境下では apr_reslist を使用し、非スレッド環境では単一の接続を管理する。
  • **ライフサイクル管理**: 接続のオープン、クローズ、およびリクエスト/コネクション終了時の自動解放（プールへの返却）を管理する。
  • **SQL 準備**: サーバー起動時に SQL 文をプリペアし、各接続にキャッシュする。

重要なシンボル

設定・構造体

  • svr_cfg: サーバーごとの設定コンテナ。dbd_cfg_t（設定値）と dbd_group_t（接続グループ）を保持。
  • dbd_group_t: 接続プール（apr_reslist）または単一接続を管理するグループ構造体。
  • ap_dbd_t: データベース接続ハンドル。ドライバ、接続状態、プリペア済みステートメントのハッシュを保持。

公開 API (エクスポート関数)

  • ap_dbd_open: 接続を明示的に取得する。呼び出し側で ap_dbd_close が必須。
  • ap_dbd_acquire: リクエスト単位で接続を取得。自動的にリクエスト終了時に解放される。
  • ap_dbd_cacquire: コネクション単位で接続を取得。
  • ap_dbd_prepare: SQL 文をプリペアする。

内部フック・ハンドラ

  • dbd_cmds: 設定ディレクティブの定義。
  • dbd_post_config: 設定の検証と接続グループの初期化。
  • dbd_child_init: 子プロセス生成時の接続プール初期化。

制御フローとデータフロー

1. **設定フェーズ**: dbd_cmds がディレクティブを読み込み、create_dbd_config で設定を初期化。dbd_post_config で設定を統合し、共有可能な dbd_group_t を構築する。 2. **接続取得**: ap_dbd_acquire が呼ばれると、リクエストのプールに接続が既に存在するか確認。なければ ap_dbd_open を呼び出し、dbd_construct で新規接続を作成。 3. **接続構築**: dbd_construct は apr_dbd_open_ex で接続を確立し、dbd_prepared_init で SQL 文をプリペアし、dbd_run_post_connect で初期化 SQL を実行する。 4. **解放**: リクエスト終了時、apr_pool_cleanup_register により登録された dbd_release が呼ばれ、接続がプールへ返却される。

読者向けメモ

  • **メモリ管理**: ap_dbd_t は接続ごとに専用のサブプールを持ち、プリペア済みステートメントもそのサブプールで管理されます。これにより、接続終了時にステートメントも自動的にクリーンアップされます。
  • **スレッドセーフ**: APR_HAS_THREADS が定義されている場合、apr_reslist を使用してスレッドセーフなプーリングが行われます。非スレッド環境では単純なポインタ管理となります。
  • **注意点**: ap_dbd_acquire はリクエストのライフサイクルに紐付くため、明示的な ap_dbd_close は不要です。逆に ap_dbd_open を使用した場合は、必ず ap_dbd_close を呼び出す必要があります。
  • **省略箇所**: ドライバ固有の挙動や、apr_dbd 内部の具体的な SQL 実行ロジックについては、apr_dbd ライブラリ側に依存しているため本ファイルでは詳細を扱っていません。
