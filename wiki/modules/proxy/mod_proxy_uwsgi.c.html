modules/proxy/mod_proxy_uwsgi.c

概要

mod_proxy_uwsgi.c は、Apache HTTP Server を uWSGI プロトコル対応のゲートウェイとして機能させるためのプロキシモジュールです。uWSGI バックエンドへのリクエスト転送、ヘッダーの変換、およびレスポンスのストリーミング処理を担います。

モジュール内での役割

mod_proxy サブシステムの一部として、uwsgi:// スキームを処理するハンドラを提供します。mod_proxy_http や mod_proxy_fcgi と同様の役割を担い、Apache のリクエスト処理パイプラインと外部の uWSGI アプリケーションサーバーを仲介します。

責務

  • **URL 正規化**: uwsgi:// スキームの解析と、バックエンドへ渡すパスの正規化。
  • **プロトコル変換**: Apache の環境変数（CGI 変数）を uWSGI プロトコル形式のバイナリヘッダーへ変換。
  • **通信管理**: バックエンドへの接続確立、リクエストボディの転送、およびレスポンスの受信。
  • **レスポンス処理**: バックエンドからのレスポンスを解析し、Apache のフィルタチェーンへ戻す。

重要なシンボル

ハンドラとフック

  • uwsgi_handler: mod_proxy から呼び出されるメインのエントリポイント。接続の確立からレスポンス処理までを統括。
  • uwsgi_canon: URL の正規化を行うフック関数。
  • register_hooks: モジュールのフックを Apache に登録。

通信・変換ユーティリティ

  • uwsgi_send: ソケットへのデータ書き込みをラップし、統計情報を更新。
  • uwsgi_send_headers: subprocess_env から環境変数を抽出し、uWSGI プロトコル仕様に従ったバイナリヘッダーを構築・送信。
  • uwsgi_send_body: クライアントからのリクエストボディを読み取り、バックエンドへ転送。
  • uwsgi_response: バックエンドからのレスポンスを解析し、HTTP ステータスとヘッダーを再構築してクライアントへ返す。

補助構造

  • make_fake_req: バックエンドからのレスポンスを解析するために、一時的な request_rec を生成。

制御フローとデータフロー

1. **正規化**: uwsgi_canon が URL を解析し、proxy:uwsgi://... 形式の内部表現に変換。 2. **接続**: uwsgi_handler が ap_proxy_acquire_connection を呼び出し、バックエンドとの接続を確保。 3. **ヘッダー送信**: uwsgi_send_headers が環境変数をバイナリ形式にシリアライズして送信。 4. **ボディ転送**: uwsgi_send_body が ap_get_client_block を使用してリクエストボディをストリーミング。 5. **レスポンス受信**: uwsgi_response がバックエンドからのストリームを読み取り、ap_getline でヘッダーを解析。その後、apr_bucket_brigade を介してレスポンスをクライアントへ転送。

読者向けメモ

  • **メモリ管理**: r->pool を多用しており、リクエスト終了時に自動解放されます。make_fake_req で作成される一時的な request_rec は、バックエンドのレスポンス解析専用の独立したプールを持ちます。
  • **接続のライフタイム**: uwsgi_handler の最後で backend->close = 1 が設定されており、現在の実装では接続の再利用（Keep-Alive）は行わず、リクエストごとに切断されます。
  • **制約**: uwsgi_send_headers はパケットサイズが APR_UINT16_MAX を超える場合にエラーを返します。
  • **省略された詳細**: 本ファイルはプロトコルの詳細なバイナリ仕様（uWSGI のヘッダー構造など）を直接実装しているため、仕様変更時には uwsgi_send_headers のビット操作部分を注意深く確認する必要があります。
