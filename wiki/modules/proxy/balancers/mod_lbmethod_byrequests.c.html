modules/proxy/balancers/mod_lbmethod_byrequests.c

概要

Apache HTTP Server の mod_proxy におけるロードバランシング手法の一つ、「リクエスト数ベース（byrequests）」の実装です。各ワーカーに設定された lbfactor（重み）に基づき、リクエストを動的に分配します。

モジュール内での役割

mod_proxy のロードバランサー機能に対し、proxy_balancer_method プロバイダーとして機能を提供します。リクエストの振り分け先を決定するアルゴリズムをカプセル化し、他のバランサーモジュールと並列してプラグイン可能なアーキテクチャを構成します。

責務

  • **負荷分散アルゴリズムの提供**: lbfactor を基準とした「最も緊急に処理が必要なワーカー」の選出。
  • **状態管理**: 各ワーカーの lbstatus（累積負荷状況）の更新と、必要に応じたリセット。
  • **プロバイダー登録**: PROXY_LBMETHOD インターフェースを通じて、バランサーが利用可能なメソッドとして自身を登録。

重要なシンボル

  • byrequests: このモジュールが提供する proxy_balancer_method 構造体。アルゴリズムの各フック（find, reset 等）を定義。
  • find_best_byrequests: リクエストごとに呼び出されるメインの選出関数。
  • is_best_byrequests: 各ワーカーの lbstatus を更新し、比較を行うためのコールバック関数。
  • ap_proxy_balancer_get_best_worker_fn: mod_proxy から提供される、ワーカー走査のためのオプション関数。

制御フローとデータフロー

1. **初期化**: post_config フックで mod_proxy から proxy_balancer_get_best_worker 関数を取得します。 2. **選出**: find_best_byrequests が呼ばれると、is_best_byrequests を用いて全ワーカーを走査します。

  • 各ワーカーの lbstatus に lbfactor を加算します。
  • lbstatus が最大であるワーカーを「次回の処理対象」として選出します。

    • 3. **更新**: 選出されたワーカーの lbstatus から、全ワーカーの lbfactor 合計値を減算し、バランスを維持します。

直接依存

  • mod_proxy.h: プロキシ機能の基本構造体（proxy_worker, proxy_balancer 等）の定義。
  • ap_hooks.h, ap_mpm.h: Apache のフック機構および MPM 関連の定義。
  • scoreboard.h: 共有メモリ上のワーカー状態管理。
  • apr_version.h

関連コード

  • mod_proxy.c: ロードバランサーの基盤となるメインモジュール。
  • mod_lbmethod_bybusyness.c / mod_lbmethod_bytraffic.c: 他のロードバランシング手法の実装例。これらと比較することで、アルゴリズムの差異を理解できます。

読者向けメモ

  • **アルゴリズムの特性**: この手法は「重み付きラウンドロビン」に近い挙動をします。lbfactor が大きいほど頻繁に選出されます。
  • **スレッドセーフ**: reset 関数などは呼び出し元でミューテックス保護されている前提で実装されています。
  • **デバッグ**: lbstatus は共有メモリ上に保持されるため、ワーカーの再起動や設定変更時に値がどのように変化するかを追跡する際は、scoreboard の状態に注意してください。
