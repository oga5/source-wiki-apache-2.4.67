modules/proxy/balancers/mod_lbmethod_bytraffic.c

概要

Apache HTTP Server の mod_proxy_balancer における負荷分散アルゴリズムの一つ、「トラフィック量ベース（bytraffic）」を実装するモジュールです。各ワーカーが処理したバイト数（送受信量）を重み（lbfactor）で正規化し、負荷が最も低いワーカーを選択します。

モジュール内での役割

mod_proxy のロードバランサー拡張機能として動作します。PROXY_LBMETHOD プロバイダーとして登録され、バランサーがリクエストを振り分ける際の「最適なワーカーの選定」ロジックを提供します。

責務

  • **負荷計算**: 各ワーカーの累積転送量と読み込み量を lbfactor で除算し、相対的な負荷を算出する。
  • **ワーカー選定**: 算出された負荷が最小のワーカーを特定する。
  • **統計リセット**: バランサーの統計情報（転送量・読み込み量）をゼロクリアする。
  • **プロバイダー登録**: bytraffic という名前でバランサーメソッドをシステムに公開する。

重要なシンボル

  • bytraffic: このモジュールのコアとなる proxy_balancer_method 構造体。アルゴリズムの各関数へのポインタを保持。
  • find_best_bytraffic: バランサーから呼び出されるメインの選定関数。
  • is_best_bytraffic: ap_proxy_balancer_get_best_worker に渡されるコールバック。個々のワーカーの負荷を比較する。
  • ap_proxy_balancer_get_best_worker_fn: mod_proxy から提供される、ワーカー反復処理用のオプション関数。

制御フローとデータフロー

1. **初期化**: register_hook が bytraffic プロバイダーを登録し、post_config で mod_proxy のオプション関数を取得します。 2. **選定フロー**:

  • バランサーがリクエストを受け取ると find_best_bytraffic を呼び出します。
  • find_best_bytraffic は ap_proxy_balancer_get_best_worker_fn を経由して、全ワーカーに対して is_best_bytraffic を実行します。
  • is_best_bytraffic は (transferred + read) / lbfactor を計算し、最小値を持つワーカーを特定します。

    • 3. **リセット**: 管理者が統計をリセットする際、reset 関数が各ワーカーのカウンタを 0 に初期化します。

直接依存

  • mod_proxy.h: プロキシ機能およびバランサー構造体の定義。
  • scoreboard.h: ワーカーのステータス管理。
  • ap_mpm.h: MPM（マルチプロセッシングモジュール）の状態確認。
  • ap_hooks.h: Apache のフックシステム。
  • apr_version.h

関連コード

  • mod_proxy_balancer.c: このモジュールが提供するメソッドを利用するメインのバランサー実装。
  • mod_lbmethod_byrequests.c: リクエスト数ベースで負荷分散を行う類似モジュール。比較対象として有用です。

読者向けメモ

  • **計算式**: (transferred + read) / lbfactor という単純な式に基づいています。リクエスト数ではなく「バイト数」を重視するため、リクエストごとのデータサイズが大きく異なる環境に適しています。
  • **スレッドセーフ**: reset 関数は呼び出し元でミューテックス保護されている前提で実装されています。
  • **初期化の注意**: lbmethod_bytraffic_post_config は起動時に2回呼ばれるため、AP_SQ_MS_CREATE_PRE_CONFIG チェックによるガードが行われています。
