modules/proxy/balancers/mod_lbmethod_bybusyness.c

概要

Apache HTTP Server の mod_proxy におけるロードバランシング手法の一つ、「bybusyness（負荷ベース）」を実装するモジュールです。各ワーカーの現在の「忙しさ（busy数）」を基準に、最も負荷の低いワーカーへリクエストを振り分けます。

モジュール内での役割

mod_proxy のロードバランサー機能に対するプラグインとして動作します。PROXY_LBMETHOD プロバイダーとして登録され、mod_proxy がリクエストを転送する際の「最適なワーカーを選択するアルゴリズム」を提供します。

責務

  • **ワーカー選定ロジックの提供**: 現在の busy 数が最小のワーカーを特定する。
  • **負荷状態の管理**: ワーカーごとの lbstatus（重み付け）と busy カウントの更新。
  • **ライフサイクル管理**: ロードバランサーの初期化時における mod_proxy 依存関係の解決と、プロバイダーとしての登録。

重要なシンボル

  • is_best_bybusyness: 比較関数。busy 数を最優先し、同数の場合は lbstatus（重み）を考慮して比較します。
  • find_best_bybusyness: mod_proxy から呼び出されるメインの選定関数。
  • bybusyness: proxy_balancer_method 構造体。このアルゴリズムの各フック（選定、リセット等）を定義します。
  • ap_proxy_balancer_get_best_worker_fn: mod_proxy から提供される、ワーカー反復処理用のオプション関数。

制御フローとデータフロー

1. **初期化**: register_hook により bybusyness メソッドが登録されます。post_config フックで mod_proxy のコア関数を取得します。 2. **選定フロー**:

  • find_best_bybusyness が呼び出されると、ap_proxy_balancer_get_best_worker_fn を介して全ワーカーを走査します。
  • 各ワーカーに対し is_best_bybusyness が適用され、busy 数が最小のワーカーが選ばれます。
  • 選定後、lbstatus が更新され、次の選定に備えます。

    • 3. **リセット**: reset 関数により、全ワーカーの lbstatus と busy カウントがゼロに初期化されます。

直接依存

  • mod_proxy.h: プロキシ機能の基本構造体（proxy_worker, proxy_balancer 等）。
  • scoreboard.h: サーバーの稼働状況管理。
  • ap_mpm.h: MPM（マルチプロセッシングモジュール）の状態確認。
  • ap_hooks.h: Apache のフックシステム。
  • apr_version.h

関連コード

  • mod_proxy.c: ロードバランシングの基盤となるメインモジュール。
  • mod_lbmethod_byrequests.c / mod_lbmethod_bytraffic.c: 他のロードバランシングアルゴリズムの実装例。これらと比較することで、選定ロジックの差異を理解できます。

読者向けメモ

  • 本モジュールは mod_proxy がロードされていることを前提としています。post_config でのチェックは、モジュール読み込み順序に依存するエラーを防ぐための重要なガードです。
  • busy 数は scoreboard を介して管理されるため、マルチプロセス環境下での正確な負荷分散が可能です。
  • age 関数は現在空実装ですが、将来的な拡張や特定の統計処理のために予約されています。
