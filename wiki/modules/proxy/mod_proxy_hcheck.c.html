modules/proxy/mod_proxy_hcheck.c

概要

mod_proxy_hcheck は、Apache HTTP Server のプロキシワーカーに対するアクティブなヘルスチェック機能を提供するモジュールです。mod_watchdog を利用してバックグラウンドで定期的にバックエンドサーバーへリクエスト（TCP, CPING, HTTP）を送信し、その応答に基づいてワーカーの有効/無効を自動制御します。

モジュール内での役割

mod_proxy サブシステムの一部として、ロードバランサー（mod_proxy_balancer）と連携します。ワーカーのステータスを共有メモリ経由で監視・更新し、障害発生時にトラフィックを自動的に切り離すための「監視者」として機能します。

責務

  • **ヘルスチェック設定の管理**: ProxyHCTemplate および ProxyHCExpr によるチェック条件の定義。
  • **監視タスクの実行**: mod_watchdog を介した定期的なヘルスチェックのスケジューリング。
  • **バックエンド通信**: TCP接続確認、AJP CPING/CPONG、および HTTP リクエストによる生存確認。
  • **ワーカー状態の更新**: 成功/失敗回数に基づくワーカーのステータス（PROXY_WORKER_HC_FAIL_FLAG 等）の操作。
  • **式評価**: ap_expr を利用した柔軟なヘルスチェック判定ルールの提供。

重要なシンボル

設定・コンテキスト

  • sctx_t: サーバー単位のモジュール設定（テンプレート、条件式、ワーカーハッシュを保持）。
  • hc_template_t: ヘルスチェックのパラメータ（間隔、成功/失敗閾値、メソッド等）を保持するテンプレート構造体。
  • wctx_t: 個別のヘルスチェックワーカーのコンテキスト（リクエスト内容や対象ワーカーへの参照）。

監視・実行

  • hc_watchdog_callback: mod_watchdog から呼び出されるメインループ。全ワーカーを走査し、チェック間隔を過ぎたワーカーをキューイングする。
  • hc_check: ヘルスチェックを実行するスレッド関数。baton_t を介してチェック対象を受け取る。

判定ロジック

  • hc_check_tcp, hc_check_cping, hc_check_http: 各プロトコルに応じたチェックの実装。
  • hc_expr_lookup: ap_expr 拡張用フック。HC() 関数や HC_BODY 変数を提供。

制御フローとデータフロー

1. **初期化**: hc_post_config で mod_watchdog インスタンスを取得し、コールバックを登録。 2. **定期実行**: hc_watchdog_callback が定期的に起動し、proxy_server_conf から全バランサーとワーカーを走査。 3. **タスク投入**: チェックが必要なワーカーに対し、baton_t を作成し、スレッドプール（hctp）または直接 hc_check を呼び出す。 4. **チェック実行**: hc_check がバックエンドへの接続を確立し、リクエストを送信。 5. **判定と更新**: 応答を解析し、成功/失敗回数を更新。閾値を超えた場合、ap_proxy_set_wstatus でワーカーの状態を更新。

直接依存

  • mod_proxy.h: プロキシワーカーおよびバランサーのデータ構造。
  • mod_watchdog.h: 定期実行のためのバックグラウンドタスク管理。
  • ap_expr.h: ヘルスチェック条件の動的評価。
  • apr_thread_pool.h: 並列ヘルスチェック実行用。
  • ap_slotmem.h: 共有メモリ管理（ワーカー状態の永続化）。
  • http_ssl.h

関連コード

  • mod_proxy_balancer.c: バランサーの管理とワーカーの同期。
  • proxy_util.c: ap_proxy_acquire_connection 等の接続管理ユーティリティ。
  • ajp_link.c: AJP プロトコル関連の低レイヤー処理。

読者向けメモ

  • **メモリ管理**: baton_t に紐づく ptemp プールは、チェック実行ごとに作成・破棄され、リクエスト処理中のメモリリークを防いでいます。
  • **スレッド安全性**: HC_USE_THREADS が有効な場合、apr_thread_pool を使用します。スレッドプールサイズは ProxyHCTPsize で制御可能です。
  • **制約**: ProxyHCTemplate は mod_proxy のワーカー設定と密接に結合しており、設定変更時は hc_get_hcworker 内で動的に再構築されます。
  • **省略箇所**: 本ドキュメントでは、ap_expr の詳細な構文解析ロジックや、各プロトコル（HTTP/AJP）のパケット構築の詳細については、コードの複雑性を考慮し省略しています。
