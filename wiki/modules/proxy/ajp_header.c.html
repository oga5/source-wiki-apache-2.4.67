modules/proxy/ajp_header.c

概要

AJP (Apache JServ Protocol) v13 プロトコルにおけるリクエスト・レスポンスのマーシャリング（直列化）およびアンマーシャリング（非直列化）を担う実装ファイルです。HTTP リクエストを AJP メッセージ形式に変換し、バックエンドからの応答を解析して Apache の request_rec 構造体に反映させるための低レベルな変換ロジックを提供します。

モジュール内での役割

mod_proxy_ajp の中核となるデータ変換層です。ajp_msg_t 構造体（ajp_msg.c で定義）を操作し、ネットワーク経由で送受信されるバイナリパケットと、Apache 内部の HTTP 抽象化レイヤーとの間の橋渡しを行います。

責務

  • **リクエストのマーシャリング**: Apache の request_rec から AJP FORWARD_REQUEST パケットを構築。
  • **レスポンスのアンマーシャリング**: AJP SEND_HEADERS パケットを解析し、headers_out 等へマッピング。
  • **プロトコル変換**: HTTP ヘッダー名やメソッドを AJP 固有の数値 ID に変換、またはその逆を行う。
  • **データパケットの管理**: ボディチャンクの送受信におけるメッセージ構築と長さの検証。

重要なシンボル

ヘッダー・メソッド変換

  • sc_for_req_header: HTTP ヘッダー名を AJP 内部 ID に変換（高速化のため文字列比較を最適化）。
  • sc_for_req_method_by_id: Apache のメソッド ID を AJP 固有のメソッドコードに変換。
  • long_res_header_for_sc: AJP 応答ヘッダー ID を文字列名に逆変換。

メッセージ処理 (Marshal/Unmarshal)

  • ajp_marshal_into_msgb: リクエスト情報を ajp_msg_t に詰め込むメイン関数。SSL 情報や環境変数もここで処理。
  • ajp_unmarshal_response: バックエンドからの応答パケットを解析し、request_rec を更新。

送受信インターフェース

  • ajp_send_header: メッセージ作成から送信までの一連の処理を実行。
  • ajp_read_header: ソケットからパケットを読み込み、メッセージの型を特定。
  • ajp_parse_data: ボディチャンクの長さを検証し、データポインタを抽出。

制御フローとデータフロー

1. **送信フロー**: ajp_send_header が ajp_msg_create でバッファを確保し、ajp_marshal_into_msgb でヘッダー、環境変数、SSL 情報を順次 ajp_msg_append_* 経由でバイナリ化します。最後に ajp_ilink_send で送信します。 2. **受信フロー**: ajp_read_header がパケットを受信し、ajp_parse_header が ajp_unmarshal_response を呼び出します。ここで headers_out が再構築され、Set-Cookie や Location 等のヘッダーに対して ap_proxy_cookie_reverse_map 等の変換が適用されます。

直接依存

  • ajp_header.h: AJP プロトコルの定数定義。
  • ajp.h: ajp_msg_t 構造体およびリンク層の関数定義。
  • util_script.h: Apache のスクリプトユーティリティ。

関連コード

  • ajp_msg.c: メッセージバッファのメモリ管理とバイナリ操作の基盤。
  • ajp_link.c: ソケットを通じた低レベルなパケット送受信。
  • mod_proxy_ajp.c: 本ファイルを呼び出し、プロキシ処理のメインループを制御。

読者向けメモ

  • **メモリ管理**: ajp_msg_t は request_rec のプール (r->pool) を使用して割り当てられます。ライフサイクルはリクエストの終了と共に終了します。
  • **EBCDIC 対応**: APR_CHARSET_EBCDIC が定義されている環境では、文字列の変換処理が挿入されます。
  • **制約**: sc_for_req_header は固定長のバッファと memcmp を使用して最適化されており、新しいヘッダーを追加する際は注意が必要です。
  • **未実装/省略**: 本ファイルはプロトコルのパースに特化しており、コネクションプーリングや負荷分散のロジックは含みません。これらは mod_proxy 側の責務です。
