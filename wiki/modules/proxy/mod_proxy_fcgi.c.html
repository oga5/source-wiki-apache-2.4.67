modules/proxy/mod_proxy_fcgi.c

概要

mod_proxy_fcgi.c は、Apache HTTP Server のプロキシサブシステムにおいて、FastCGI プロトコルを介したバックエンド通信を担当するモジュールです。HTTP リクエストを FastCGI レコードに変換し、バックエンドとの間で環境変数の受け渡し、リクエストボディの転送、およびレスポンスヘッダの解析を行います。

モジュール内での役割

mod_proxy のスキームハンドラとして動作します。mod_proxy が提供するコネクション管理やワーカ管理のインフラを利用し、FastCGI 特有のプロトコルスタック（util_fcgi.h）を適用してバックエンドとの通信を完結させます。

責務

  • **リクエストの正規化**: proxy_fcgi_canon を通じて、fcgi:// URL の解析とパスの正規化を行う。
  • **環境変数の制御**: ProxyFCGISetEnvIf ディレクティブにより、リクエストに応じた環境変数の動的な書き換え（fix_cgivars）をサポートする。
  • **FastCGI プロトコル処理**: BEGIN_REQUEST の送信、PARAMS レコードによる環境変数のシリアライズ、STDIN によるボディ転送、およびレスポンスのストリーム解析を行う。
  • **レスポンスのフィルタリング**: バックエンドからのレスポンスヘッダを解析し、Apache のフィルタチェーンへ適切に引き渡す。

重要なシンボル

設定構造体

  • fcgi_dirconf_t: ディレクトリ単位の設定。バックエンドタイプ（FPM/Generic）や環境変数の修正ルールを保持。
  • fcgi_req_config_t: リクエスト単位の設定。PATH_INFO の計算に使用されるパス情報を保持。

プロトコルハンドラ

  • proxy_fcgi_handler: mod_proxy から呼び出されるメインのエントリポイント。接続の確立とリクエスト処理の開始を行う。
  • fcgi_do_request: FastCGI の各フェーズ（BEGIN, PARAMS, DISPATCH）を順次実行する制御関数。

ユーティリティ・ヘルパー

  • send_data / get_data: apr_socket をラップし、ワーカの統計情報を更新する送受信関数。
  • handle_headers: バックエンドからのレスポンスストリームから HTTP ヘッダの終端を検出するステートマシン。
  • dispatch: poll を使用してバックエンドとの双方向通信（STDIN 送信と STDOUT/STDERR 受信）を非同期的に処理する。

制御フローとデータフロー

1. **初期化**: proxy_fcgi_handler が接続を確保し、ap_proxy_prefetch_input でリクエストボディをバッファリングする。 2. **リクエスト送信**: fcgi_do_request が send_begin_request でセッションを開始し、send_environment で環境変数を PARAMS レコードとして送信する。 3. **双方向通信**: dispatch 関数が apr_poll を使用してループし、クライアントからの入力を STDIN としてバックエンドへ送りつつ、バックエンドからの STDOUT を解析する。 4. **ヘッダ解析**: handle_headers が STDOUT ストリームからヘッダの終わりを検出し、ap_scan_script_header_err_brigade_ex で Apache のレスポンスヘッダを構築する。 5. **終了**: AP_FCGI_END_REQUEST を受信するか、エラー発生時に接続を解放する。

直接依存

  • mod_proxy.h: プロキシサブシステムの基盤 API。
  • util_fcgi.h: FastCGI プロトコルヘッダの構築・解析用ユーティリティ。
  • util_script.h: CGI 関連の環境変数処理。
  • ap_expr.h: ProxyFCGISetEnvIf で使用される式評価エンジン。

関連コード

  • modules/proxy/proxy_util.c: コネクション管理やワーカの統計管理など、プロキシ共通機能。
  • modules/proxy/mod_proxy_http.c: HTTP プロキシの挙動との比較対象。

読者向けメモ

  • **メモリ管理**: fcgi_do_request 内では temp_pool を使用して、リクエスト処理中の大量のメモリ割り当てを効率的に解放しています。
  • **パフォーマンス**: dispatch 関数は poll を使用して効率的な I/O を行いますが、ヘッダ解析まではバッファリングが発生します。
  • **制約**: 本モジュールは FastCGI の多重化（Multiplexing）をサポートしておらず、1 コネクションにつき 1 リクエスト（request_id = 1 固定）を前提としています。
  • **未実装/浅い部分**: get_data などの I/O 処理は、将来的な完全非同期 I/O への移行を見据えた設計になっていますが、現時点ではブロッキング I/O が混在しています。
