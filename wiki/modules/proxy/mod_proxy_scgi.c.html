modules/proxy/mod_proxy_scgi.c

概要

SCGI (Simple Common Gateway Interface) プロトコルを Apache HTTP Server のプロキシサブシステムに統合するバックエンドモジュールです。SCGI プロトコルに基づき、リクエストヘッダーをネットストリング形式でエンコードしてバックエンドへ送信し、レスポンスを処理します。

モジュール内での役割

mod_proxy のスキームハンドラとして動作します。scgi:// で始まる URL を検知し、バックエンドとの接続確立、プロトコル変換、およびレスポンスのフィルタリングを担います。

責務

  • **プロトコル変換**: CGI 環境変数を SCGI ネットストリング形式に変換して送信。
  • **接続管理**: mod_proxy のコネクションプールを利用したバックエンドとの通信。
  • **レスポンス処理**: バックエンドからのレスポンスヘッダー解析と、X-Sendfile や内部リダイレクトのサポート。
  • **カスタムバケット**: ソケット読み込みを抽象化する SOCKET_EX バケット型の実装。

重要なシンボル

設定と状態管理

  • proxy_scgi_module: モジュールのメイン構造体。
  • scgi_config: ディレクトリ単位の設定（ProxySCGISendfile, ProxySCGIInternalRedirect）。
  • scgi_request_config: リクエスト処理中の内部リダイレクトやファイル送信の指示を保持。

カスタムバケット (SOCKET_EX)

  • bucket_type_socket_ex: ソケットからの読み込みを統計情報（counter）と連動させるためのカスタムバケット型。
  • bucket_socket_ex_read: ソケットからデータを読み込み、ヒープバケットへ変換する関数。

ハンドラ関数

  • scgi_handler: プロキシリクエストのメインエントリポイント。
  • scgi_canon: URL の正規化（scgi:// スキームの処理）。
  • scgi_request_status: 内部リダイレクトやファイル送信のフック処理。

制御フローとデータフロー

1. **正規化**: scgi_canon が URL を解析し、プロキシ内部形式に変換。 2. **接続**: scgi_handler が ap_proxy_acquire_connection でコネクションを取得し、ap_proxy_connect_backend で接続。 3. **送信**: send_headers が環境変数をネットストリング形式にシリアライズし、sendall を経由して送信。続いて send_request_body がリクエストボディを転送。 4. **受信**: pass_response が SOCKET_EX バケットを使用してバックエンドからのレスポンスを読み込み、ap_scan_script_header_err_brigade_ex でヘッダーを解析。 5. **後処理**: 設定に応じて X-Sendfile や内部リダイレクトを判定し、必要であれば scgi_request_status フックを介してサブリクエストを実行。

読者向けメモ

  • **メモリ管理**: すべての動的メモリ割り当ては request_rec のプール (r->pool) に紐付いており、リクエスト終了時に自動解放されます。
  • **ソケットの生存期間**: scgi_handler 内で backend->close = 1 が設定されており、SCGI 通信後は常に接続を閉じる設計です。
  • **バケットの抽象化**: SOCKET_EX バケットは、ソケット読み込み時に統計カウンタを更新するために実装されています。これは apr_bucket_socket をラップする形をとっています。
  • **未実装/浅い部分**: SCGI プロトコル自体の詳細な仕様（ネットストリングの厳密なバリデーションなど）は、このモジュール内では最小限の処理に留まっています。また、sendfile の実装は X-Sendfile ヘッダーの存在に依存しており、バックエンドの挙動に強く依存します。
