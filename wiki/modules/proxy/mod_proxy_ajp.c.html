modules/proxy/mod_proxy_ajp.c

概要

Apache HTTP Server のプロキシサブシステムにおいて、AJP (Apache JServ Protocol) v1.3 を用いたバックエンド通信を実装するモジュールです。mod_proxy の拡張として、AJP プロトコル特有のヘッダー処理、リクエスト転送、レスポンス解析、およびコネクション管理を担当します。

モジュール内での役割

mod_proxy のハンドラおよびカノニカライザ（URL正規化）として機能します。proxy_hook_scheme_handler および proxy_hook_canon_handler を通じて、ajp:// スキームのリクエストを処理する主要なエントリーポイントを提供します。

責務

  • **URL 正規化**: ajp:// スキームの構文解析、ホスト/ポートの抽出、パスのエンコーディング処理。
  • **リクエスト転送**: AJP プロトコルに基づいたヘッダーおよびボディデータのシリアライズと送信。
  • **レスポンス処理**: バックエンドからの AJP メッセージ（ヘッダー、ボディチャンク、終了通知）の解析と、Apache のフィルタチェーンへの変換。
  • **コネクション管理**: バックエンドとの接続確立、CPING/CPONG によるヘルスチェック、再利用の判断。

重要なシンボル

ハンドラ・フック

  • proxy_ajp_handler: ajp:// スキームのメインハンドラ。コネクションの取得からリクエスト処理までを統括。
  • proxy_ajp_canon: URL を正規化し、内部的な proxy:ajp:// 形式へ変換。
  • ap_proxy_http_register_hook: モジュールのフックを登録。

リクエスト処理

  • ap_proxy_ajp_request: AJP プロトコルによる通信の核心。ヘッダー送信、ボディのストリーミング、レスポンスのループ処理を行う。
  • is_idempotent: リクエストが冪等かどうかを判定し、エラー時のリトライ可否を決定。

ユーティリティ

  • get_content_length: リクエストの Content-Length を取得。
  • METHOD_IDEMPOTENT 系マクロ: 冪等性の判定に使用。

制御フローとデータフロー

1. **初期化**: proxy_ajp_handler が呼び出され、ap_proxy_acquire_connection でバックエンド接続を確保。 2. **接続確立**: ap_proxy_connect_backend でソケットを接続し、必要に応じて ajp_handle_cping_cpong で疎通確認。 3. **リクエスト送信**: ap_proxy_ajp_request が ajp_send_header でヘッダーを送信し、apr_brigade_flatten で入力データを AJP メッセージとしてバックエンドへ転送。 4. **レスポンスループ**: ajp_read_header でバックエンドからのメッセージを読み取り、ajp_parse_type でコマンド種別（ヘッダー、ボディ、終了）を判定。 5. **データ変換**: CMD_AJP13_SEND_BODY_CHUNK を受け取ると、AJP データを Apache の apr_bucket に変換し、ap_pass_brigade でクライアントへ送出。 6. **終了**: CMD_AJP13_END_RESPONSE でリクエストを完了し、コネクションを解放または再利用。

直接依存

  • mod_proxy.h: プロキシ共通のデータ構造と API。
  • ajp.h: AJP プロトコル定義、メッセージ構造体、および低レベル通信関数。

関連コード

  • ajp_header.c, ajp_link.c, ajp_msg.c: AJP プロトコルの詳細なパケット操作やリンク層の実装。
  • proxy_util.c: コネクションプール管理や共通のプロキシユーティリティ。

読者向けメモ

  • **メモリ管理**: request_rec のプール (r->pool) を多用しています。リクエスト終了時に自動解放されるため、長期的なメモリ保持には注意が必要です。
  • **エラー処理**: backend_failed や client_failed フラグにより、エラー発生時にコネクションを強制クローズ (conn->close = 1) して再利用を防ぐ設計になっています。
  • **AJP の制約**: AJP プロトコルには明示的なフラッシュメッセージが存在しないため、CMD_AJP13_SEND_BODY_CHUNK 時に apr_poll を用いたタイムアウトベースのフラッシュ（バンドエイド）が実装されています。
  • **省略箇所**: 本ファイルはプロトコルの高レベルな制御に集中しており、低レベルなバイトストリーム操作（ajp_msg_t の内部構造など）は ajp_msg.c 等に委譲されています。
