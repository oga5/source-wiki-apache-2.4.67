modules/proxy/mod_proxy_http.c

概要

Apache HTTP Server のプロキシサブシステムにおける HTTP/HTTPS/WS/WSS プロトコルハンドラです。バックエンドサーバーへのリクエスト転送、レスポンスの受信、およびプロキシ特有のヘッダー変換（Via, Warning, Connection 等）を担います。

モジュール内での役割

mod_proxy のスキームハンドラとして登録され、proxy_http_handler を通じてリクエストを処理します。proxy_util.c が提供する汎用的な接続管理やバケット操作ユーティリティに強く依存しており、HTTP 固有のロジック（チャンク転送、ヘッダー正規化、100-Continue 処理）をカプセル化しています。

責務

  • **リクエスト正規化**: URL のスキーム判定とホスト/ポート/パスの正規化。
  • **リクエスト転送**: クライアントからのリクエストボディをバックエンドへストリーミングまたはスプールして転送。
  • **レスポンス処理**: バックエンドからのレスポンスヘッダー解析、ホップバイホップヘッダーの除去、およびレスポンスボディのフィルタリングと転送。
  • **プロトコル変換**: HTTP/1.1 へのアップグレードや WebSocket トンネリングのサポート。

重要なシンボル

リクエスト状態管理

  • proxy_http_req_t: リクエスト処理中の状態（バックエンド接続、バケットブリゲード、転送メソッド等）を保持する中心的な構造体。
  • rb_methods: リクエストボディの転送方式（ストリーム、チャンク、スプール）を定義する列挙型。

ハンドラとフック

  • proxy_http_handler: mod_proxy から呼び出されるメインのエントリポイント。
  • proxy_http_canon: URL の正規化を行うハンドラ。
  • ap_proxy_http_register_hook: モジュールの初期化とフック登録。

ヘッダー・ボディ処理

  • ap_proxy_read_headers: バックエンドからのレスポンスヘッダーを読み取り、r->headers_out に格納。
  • stream_reqbody: リクエストボディをバックエンドへ転送するメインループ。
  • process_proxy_header: ProxyPassReverse 等のルールに基づきヘッダーを変換。

制御フローとデータフロー

1. **初期化**: proxy_http_handler が接続を確保し、proxy_http_req_t を初期化。 2. **プリフェッチ**: ap_proxy_http_prefetch がリクエストボディを先読みし、転送方式（rb_method）を決定。 3. **接続と送信**: ap_proxy_http_request が stream_reqbody を呼び出し、ヘッダーとボディをバックエンドへ送信。 4. **レスポンス受信**: ap_proxy_http_process_response がバックエンドからのレスポンスを読み取り、フィルタチェーンを通じてクライアントへ転送。 5. **クリーンアップ**: ap_proxy_http_cleanup により接続を解放。

直接依存

  • mod_proxy.h: プロキシサブシステムのコア API。
  • ap_regex.h: Warning ヘッダーの解析用正規表現エンジン。
  • httpd.h, http_core.h, apr_buckets.h: Apache/APR の基本構造とバケット処理。

関連コード

  • modules/proxy/proxy_util.c: 接続管理、バケット操作、設定解析の共通基盤。
  • modules/proxy/mod_proxy.c: プロキシモジュールのメインエントリ。

読者向けメモ

  • **メモリ管理**: r->pool を多用しますが、proxy_http_process_response 内では一時的なバッファやブリゲードのクリーンアップが頻繁に行われます。特に apr_brigade_cleanup を忘れるとメモリリークの原因となります。
  • **エッジケース**: 100-Continue 処理や Upgrade ヘッダーの扱いは複雑です。特にバックエンドが予期せぬレスポンスを返した場合の backend->close = 1 による接続切断ロジックが重要です。
  • **制限事項**: 本ファイルは HTTP 関連のロジックに特化しており、AJP や FCGI 等の他プロトコルは別モジュール（mod_proxy_ajp.c 等）で実装されています。
  • **安全性**: process_proxy_header はヘッダーの正規化を行いますが、不正なヘッダーに対する badopt 設定（bad_error, bad_body, bad_ignore）の挙動に注意してください。
