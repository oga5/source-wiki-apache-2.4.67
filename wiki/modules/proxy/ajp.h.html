modules/proxy/ajp.h

概要

ajp.h は、Apache HTTP Server とサーブレットコンテナ（Tomcat 等）間で通信を行うための **AJP (Apache Jserv Protocol) v13** の定義および API を提供するヘッダーファイルです。プロトコルの定数、エラーコード、メッセージ構造体、およびメッセージのシリアライズ・デシリアライズ・送受信を行うための関数群を定義しています。

モジュール内での役割

mod_proxy_ajp の中核となるデータ構造と通信インターフェースを定義しています。mod_proxy サブシステムにおいて、バックエンドとのバイナリ通信プロトコルを抽象化する役割を担います。

責務

  • **プロトコル定義**: AJP13 のヘッダー、コマンドタイプ、デフォルトポート等の定数管理。
  • **メッセージ構造の定義**: ajp_msg_t による AJP メッセージのメモリレイアウトと状態管理。
  • **シリアライズ/デシリアライズ**: メッセージへのデータ追加（append）および抽出（get/peek）API の提供。
  • **通信抽象化**: ソケットを介したメッセージの送受信およびヘッダー解析のインターフェース提供。

重要なシンボル

メッセージ構造体

  • ajp_msg_t: AJP メッセージを保持する構造体。バッファ、現在の読み取り位置、メッセージ長、サーバーサイドフラグを管理します。

プロトコル定数

  • AJP13_WS_HEADER: AJP メッセージの開始シグネチャ。
  • CMD_AJP13_*: FORWARD_REQUEST, SEND_BODY_CHUNK, END_RESPONSE 等の AJP13 コマンドタイプ。

メッセージ操作 API

  • **構築・管理**: ajp_msg_create, ajp_msg_reset, ajp_msg_reuse, ajp_msg_end。
  • **シリアライズ (Append)**: ajp_msg_append_uint32/16/8, ajp_msg_append_string_ex, ajp_msg_append_bytes。
  • **デシリアライズ (Get/Peek)**: ajp_msg_get_uint32/16/8, ajp_msg_peek_uint16/8, ajp_msg_get_string, ajp_msg_get_bytes。

通信・解析 API

  • **送受信**: ajp_ilink_send, ajp_ilink_receive。
  • **高レベル解析**: ajp_send_header, ajp_read_header, ajp_parse_header, ajp_parse_data。

制御フローとデータフロー

1. **構築**: ajp_msg_create でプールからメモリを確保し、ajp_msg_t を初期化します。 2. **シリアライズ**: ajp_msg_append_* 関数群を用いて、リクエストヘッダーやデータをバッファに書き込みます。 3. **送信**: ajp_ilink_send がバッファの内容をソケットへ書き出します。 4. **受信・解析**: ajp_ilink_receive でソケットから読み込み、ajp_parse_* 関数群を用いてメッセージタイプやヘッダーを抽出します。 5. **状態管理**: ajp_msg_t 内の pos が読み取り位置を追跡し、len がメッセージ全体のサイズを保持します。

読者向けメモ

  • **メモリ管理**: ajp_msg_t は apr_pool_t を介して割り当てられます。ライフサイクルはリクエストの生存期間に依存します。
  • **EBCDIC 対応**: ajp_msg_append_string_ex は EBCDIC 環境での ASCII 変換フラグをサポートしています。
  • **制約**: AJP_MSG_BUFFER_SZ (8KB) および AJP_MAX_BUFFER_SZ (64KB) がメッセージサイズの物理的な上限となります。
  • **エラーハンドリング**: AJP_E* で始まる独自のエラーコードが定義されており、APR のエラーコードと組み合わせて使用されます。
  • **省略**: 本ヘッダーはインターフェース定義に特化しており、具体的なソケットのタイムアウト処理やバッファの再配置ロジックは実装ファイル (.c) に委譲されています。
