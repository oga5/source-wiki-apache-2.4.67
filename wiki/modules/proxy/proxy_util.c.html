modules/proxy/proxy_util.c

概要

proxy_util.c は、Apache HTTP Server の mod_proxy サブシステムにおける中核的なユーティリティ群を提供します。URL の正規化、プロキシワーカーの管理、ロードバランサーのロジック、バックエンド接続のライフサイクル管理、およびプロキシ特有のネットワーク操作（UDS、HTTP CONNECT 等）を担う、サブシステム内で最も広範な機能を持つファイルです。

モジュール内での役割

mod_proxy の各プロトコルハンドラ（mod_proxy_http, mod_proxy_ajp 等）が共通して利用する基盤機能を提供します。設定の解析、ワーカーの初期化、接続プールの管理といった「プロキシとしての共通動作」を抽象化し、各ハンドラがプロトコル固有のロジックに集中できるようにします。

責務

  • **URL/文字列処理**: URL の正規化、エンコード/デコード、ホスト名/IP アドレスの照合。
  • **ワーカー管理**: プロキシワーカーの定義、共有メモリ（SHM）への同期、ヘルスチェック、エラー状態の管理。
  • **ロードバランサー**: バランサーの初期化、ワーカー選択アルゴリズムの実行、ステータス管理。
  • **接続管理**: バックエンド接続のプール（reslist）、再利用判定、SSL 接続の初期化、UDS（Unix Domain Socket）接続の確立。
  • **トンネリング**: ap_proxy_tunnel_run による双方向ストリームの非同期転送。

重要なシンボル

ワーカー・バランサー管理

  • ap_proxy_define_worker_ex: ワーカーの定義と初期設定。
  • ap_proxy_initialize_worker: ワーカーの接続プール（reslist）とミューテックスの初期化。
  • ap_proxy_get_worker_ex: URL に一致するワーカーの検索（longest match）。
  • ap_proxy_sync_balancer: 共有メモリからバランサーとワーカーの構成を同期。

接続・通信制御

  • ap_proxy_connect_backend: バックエンドへのソケット接続確立と初期化。
  • ap_proxy_acquire_connection / release_connection: 接続プールからの接続取得と返却。
  • ap_proxy_transfer_between_connections: 接続間でのデータ転送（トンネリング基盤）。
  • ap_proxy_tunnel_run: 双方向トンネルのメインループ（非同期ポーリング）。

ユーティリティ

  • ap_proxy_canonenc_ex: URL の正規化とエンコーディング。
  • ap_proxy_canon_netloc: ネットワークロケーション（ホスト/ポート/認証情報）の解析。

制御フローとデータフロー

1. **リクエスト処理**: ap_proxy_canon_url で URL を正規化し、ap_proxy_pre_request で適切なワーカーまたはバランサーを特定します。 2. **接続確立**: ap_proxy_acquire_connection が接続プールから proxy_conn_rec を取得し、ap_proxy_connect_backend がソケットを生成してバックエンドへ接続します。 3. **データ転送**: ap_proxy_transfer_between_connections が apr_bucket_brigade を介してクライアントとバックエンド間のデータをストリーミングします。 4. **ライフサイクル**: 接続は proxy_conn_rec 内のサブプール（scpool）で管理され、ワーカーの reslist を通じて再利用されます。

読者向けメモ

  • **メモリ管理**: 接続やワーカーのデータは、apr_pool_t の階層構造によって管理されています。特に scpool は接続のライフサイクルに紐付いており、再利用時に適切にクリアされます。
  • **スレッドセーフティ**: proxy_mutex を使用して、マルチプロセス/マルチスレッド環境での共有メモリ（バランサー/ワーカー状態）へのアクセスを保護しています。
  • **未実装/浅い部分**: proxy_match_ipaddr 等の IP アドレス照合ロジックは IPv6 対応が限定的である可能性があるため、IPv6 環境での利用時は注意が必要です。
  • **トンネリング**: ap_proxy_tunnel_run は非常に複雑な非同期ポーリングロジックを含んでおり、デバッグ時は APLOG_TRACE レベルのログが不可欠です。
