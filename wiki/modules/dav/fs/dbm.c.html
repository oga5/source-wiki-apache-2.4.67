modules/dav/fs/dbm.c

概要

dbm.c は、mod_dav_fs におけるファイルシステムリポジトリのプロパティ（Dead Properties）を永続化するための DBM データベース層です。各ファイルおよびディレクトリに対応する .DAV サブディレクトリ内に SDBM 形式のデータベースを作成し、メタデータとプロパティ値を管理します。

モジュール内での役割

mod_dav のプロパティ管理インターフェース（dav_hooks_db）を実装するバックエンドです。repos.c がファイルシステム操作を担うのに対し、本ファイルはそれらのファイルに付随する WebDAV プロパティの読み書きを抽象化します。

責務

  • **DBM 抽象化**: APR DBM API をラップし、エラーハンドリングとリソース管理を提供。
  • **プロパティ永続化**: XML 形式のプロパティを DBM キー/値ペアに変換して保存。
  • **名前空間管理**: XML 名前空間 URI を内部インデックス（ns_table）にマッピングし、DBM キーのサイズを最適化。
  • **メタデータ管理**: データベースのバージョン管理とスキーマ整合性の維持。

重要なシンボル

データベース管理

  • struct dav_db: データベース接続状態、名前空間テーブル、反復子を保持する主要構造体。
  • dav_dbm_open_direct: DBM ファイルを直接開く低レベル関数。
  • dav_propdb_open: プロパティデータベースを開き、メタデータを検証・初期化する。

プロパティ操作

  • dav_propdb_store: XML 要素を DBM 値に変換して保存。
  • dav_propdb_output_value: DBM から値を読み込み、XML テキストとして出力。
  • dav_propdb_remove: 特定のプロパティを削除。

名前空間・キー生成

  • dav_build_key: 名前空間 ID とプロパティ名から DBM キーを生成。
  • dav_propdb_map_namespaces: URI を内部 ID に変換し、必要に応じて ns_table を更新。

制御フローとデータフロー

1. **初期化**: dav_propdb_open が呼ばれると、METADATA キーを読み込み、バージョンチェックと名前空間テーブルの構築を行う。 2. **キー生成**: dav_build_key は uri_index ハッシュを使用して URI を数値 ID に変換し、#:name 形式の文字列を生成する。 3. **保存**: dav_propdb_store は apr_xml_to_text を使用して XML をシリアライズし、dav_dbm_store を経由して DBM に書き込む。 4. **反復**: dav_propdb_first_name / next_name は DBM のキーを順次走査し、METADATA キーをスキップしつつ dav_set_name で名前空間を復元する。

読者向けメモ

  • **メモリ管理**: dav_db 構造体は apr_pool_t に紐付いており、データベースのクローズ時に適切に解放される必要があります。
  • **並行性**: DBM の性質上、書き込み時のロック制御は APR DBM に依存します。コメントには「読み取り中に書き込みが重なる場合の再試行ロジック」の欠如が示唆されています。
  • **制約**: dav_build_key は静的バッファを使用するため、呼び出し直後に結果をコピーする必要があります。
  • **未実装/浅い部分**: DBM のエラーハンドリングや、データベースのバージョンアップグレードパスは限定的です。特に METADATA のマイナーバージョン更新は現在の実装では不完全な箇所があります。
