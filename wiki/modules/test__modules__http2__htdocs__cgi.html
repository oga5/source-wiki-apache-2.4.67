test/modules/http2/htdocs/cgi

システム内での役割

本ディレクトリは、HTTP/2 テストスイートにおける CGI 動作検証用の Python スクリプト群を格納しています。サーバーがリクエストを適切にパースし、環境変数を引き渡し、ストリーム制御やファイルアップロードを正しく処理できるかを検証するためのバックエンドとして機能します。

責務

  • **リクエスト解析の抽象化**: 共通の requestparser.py を用いて、クエリ文字列やマルチパートフォームデータを一貫して処理する。
  • **動的レスポンス生成**: エコー、ヘッダー情報のダンプ、指定回数のデータ出力など、HTTP/2 のプロトコル検証に必要な動的コンテンツを提供する。
  • **ファイル操作の検証**: multipart/form-data を用いたファイルアップロードおよび削除機能の提供。
  • **タイミング制御**: sleep 等を用いた、HTTP/2 ストリームのタイムアウトやフロー制御の検証。

このモジュールのファイル

  • requestparser.py: リクエスト解析用ヘルパーライブラリ。
  • upload.py: ファイルアップロード・削除機能。
  • necho.py: 指定回数エコーおよび待機処理。
  • hecho.py: 指定ヘッダーの値をエコー。
  • env.py: サーバー環境変数のダンプ。
  • hello.py: サーバー状態の JSON 出力。
  • echohd.py: HTTP ヘッダー由来の環境変数取得。
  • mnot164.py: 大量データ転送によるフロー制御検証。
  • echo.py: リクエストボディの単純エコー。

主要なデータ構造

  • requestparser.FileItem: マルチパート解析で得られるファイルオブジェクトのラッパー。
  • forms (dict): requestparser を通じて取得される、解析済みのフォームパラメータ。

主なフロー

1. **初期化**: 各 CGI スクリプトが requestparser.get_request_params() を呼び出し、入力を解析。 2. **条件分岐**: パラメータ（name, count, file 等）に基づき、レスポンス内容を決定。 3. **レスポンス生成**: HTTP ヘッダー（Status, Content-Type 等）を標準出力へ書き出し、続いてボディを出力。 4. **終了**: 必要に応じて sys.stdout.flush() を行い、ストリームを確定させる。

外部依存

  • multipart (python3-multipart): マルチパート形式のストリーム解析に必須。
  • 標準ライブラリ: os, sys, time, json, urllib.parse 等。

内部構成

ディレクトリ境界に基づき構成されており、requestparser.py が他のすべてのスクリプトからインポートされる依存関係の中心となっています。

推奨読書順

1. requestparser.py (基盤となる解析ロジック) 2. echo.py (最も単純な動作確認) 3. upload.py (複雑なリクエスト処理の例) 4. necho.py / mnot164.py (ストリーム制御の検証) 5. hecho.py / env.py / hello.py (環境変数・ヘッダー情報の確認)

未解決の問い

  • echo.py 等に含まれる未使用のインポート文の整理。
