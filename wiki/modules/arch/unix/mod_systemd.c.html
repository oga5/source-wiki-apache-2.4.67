modules/arch/unix/mod_systemd.c

概要

mod_systemd.c は、Apache HTTP Server を systemd 管理下のサービスとして統合するためのモジュールです。サーバーの起動状態、リロード、および稼働状況（統計情報）を sd_notify プロトコルを通じて systemd に通知し、ソケットの引き継ぎ（Socket Activation）をサポートします。

モジュール内での役割

modules/arch/unix ディレクトリ内で、OS 固有のプロセス管理機能を提供する役割を担います。他の Unix 系モジュール（mod_unixd 等）が権限管理やプロセス制御を行うのに対し、本モジュールは外部のサービスマネージャ（systemd）との通信インターフェースに特化しています。

責務

  • **ライフサイクル通知**: サーバーの起動、設定読み込み、リロード完了を systemd に通知。
  • **稼働状況の可視化**: ExtendedStatus が有効な場合、リクエスト数や負荷状況を systemd のステータス文字列として定期的に更新。
  • **ソケット管理**: sd_listen_fds を介したソケット引き継ぎのサポート。
  • **セキュリティ情報**: SELinux が有効な場合、起動時に現在のコンテキストをログ出力。

重要なシンボル

フック関数（ライフサイクル管理）

  • systemd_pre_config: 設定読み込み開始を通知し、ap_extended_status を強制有効化。
  • systemd_post_config: 設定完了とサービス準備完了（READY=1）を通知。
  • systemd_pre_mpm: MPM 起動直前に PID を通知。
  • systemd_monitor: 定期的に呼び出され、現在の負荷状況を systemd に送信。

公開インターフェース（オプション関数）

  • ap_systemd_listen_fds: sd_listen_fds のラッパー。
  • ap_find_systemd_socket: 環境変数から特定のポートに対応するファイルディスクリプタを検索。

内部ヘルパー

  • log_selinux_context: SELinux コンテキストをログ出力（HAVE_SELINUX 定義時）。

制御フローとデータフロー

1. **初期化**: systemd_register_hooks が各フックを登録。 2. **起動フロー**: pre_config → post_config の順で sd_notify を呼び出し、systemd に状態を伝達。 3. **監視フロー**: ap_hook_monitor により systemd_monitor が定期実行される。ap_get_sload でスコアボードから統計を取得し、sd_notifyf で systemd に文字列として送信。 4. **ソケット検索**: ap_find_systemd_socket は getenv("LISTEN_FDS") を参照し、sd_is_socket_inet を用いて指定ポートと一致する FD を特定する。

読者向けメモ

  • **メモリ管理**: sd_notify 関連の文字列はスタックまたは一時的なバッファで処理されます。log_selinux_context 内の con は freecon で解放されます。
  • **制約**: systemd_monitor は ap_extended_status が無効な場合は何も行わず DECLINED を返します。
  • **設計上の注意**: post_config で READY=1 を送信していますが、これは listen() 完了後に行われるため、接続が拒否されるリスクを最小化しています。
  • **省略箇所**: sd_notify の内部実装や、systemd 側のソケットアクティベーションの詳細は本ファイルには含まれず、外部ライブラリ（libsystemd）に依存しています。
