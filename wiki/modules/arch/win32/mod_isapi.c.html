modules/arch/win32/mod_isapi.c

概要

mod_isapi.c は、Apache HTTP Server 上で Microsoft ISAPI (Internet Server Application) 拡張を動作させるための実装です。ISAPI 2.0 仕様の大部分をサポートし、IIS 5.0 相当の拡張機能の一部も提供します。このモジュールは、Windows 環境におけるレガシーな動的コンテンツ実行環境を Apache に統合する役割を担います。

モジュール内での役割

modules/arch/win32 サブシステムの一部として、Apache のリクエスト処理パイプラインに「ISAPI ハンドラ」を挿入します。外部の .dll ファイルを動的にロードし、Apache の request_rec を ISAPI の EXTENSION_CONTROL_BLOCK (ECB) に変換して、ISAPI 拡張が期待するインターフェースをエミュレートします。

責務

  • **ISAPI ライフサイクル管理**: DLL のロード、キャッシュ、アンロード、および GetExtensionVersion / HttpExtensionProc の呼び出し。
  • **リクエスト変換**: Apache の環境変数やリクエストデータを ISAPI 形式へマッピング。
  • **コールバック実装**: GetServerVariable, WriteClient, ReadClient, ServerSupportFunction の提供。
  • **非同期エミュレーション**: fake_async 設定による、同期的な Apache パイプライン上での非同期 I/O の擬似的な実現。

重要なシンボル

設定・管理構造体

  • isapi_dir_conf: ディレクトリごとの設定（バッファサイズ、ログ設定、非同期エミュレーションの有効化など）。
  • isapi_loaded: ロードされた DLL のハンドル、関数ポインタ、およびロード状態を保持するキャッシュ構造体。
  • isapi_global_conf: ロード済みモジュールを管理するグローバルハッシュテーブルと排他制御用ミューテックス。

リクエスト処理

  • isapi_cid: 「Connection ID」。リクエスト処理中の状態（ECB、リクエスト情報、完了通知用ミューテックス）を保持するコンテキスト。
  • isapi_handler: Apache のハンドラフック。リクエストを受け取り、ISAPI 拡張を呼び出すメインエントリーポイント。

コールバック関数群

  • regfnGetServerVariable: 環境変数の取得。
  • regfnReadClient: クライアントからのデータ読み込み。
  • regfnWriteClient: クライアントへのデータ書き込み（ヘッダー解析を含む）。
  • regfnServerSupportFunction: ISAPI 拡張からの制御リクエスト（リダイレクト、パス変換、ログ出力など）の処理。

制御フローとデータフロー

1. **初期化**: isapi_pre_config でグローバルキャッシュプールとハッシュテーブルを初期化。 2. **ロード**: isapi_lookup が DLL を検索。未ロードの場合、isapi_load で apr_dso_load を行い、必要なエントリポイントを解決。 3. **リクエスト処理**:

  • isapi_handler が isapi_cid を作成し、Apache のリクエストデータを ECB にコピー。
  • HttpExtensionProc を呼び出し、拡張に制御を渡す。
  • 拡張が ServerSupportFunction を呼び出すと、regfnServerSupportFunction がリクエストを解析し、Apache の API（ap_pass_brigade 等）へ変換。

    • 4. **終了**: HSE_REQ_DONE_WITH_SESSION または HttpExtensionProc の戻り値により、リクエストの完了を判定。

読者向けメモ

  • **メモリ管理**: isapi_cid や EXTENSION_CONTROL_BLOCK はリクエストプール (r->pool) 上に割り当てられ、リクエスト終了時に自動解放されます。
  • **非同期エミュレーション**: fake_async は実験的な機能です。本来非同期であるべき ISAPI の呼び出しを、ミューテックスを用いて同期的に待機させるため、高負荷時にはスレッド枯渇のリスクがあります。
  • **未実装**: Microsoft 固有の非同期 I/O 拡張の多くは、Apache のアーキテクチャと一対一で対応しないため、意図的に未実装または警告ログ出力のみとなっています。
  • **スレッド安全性**: isapi_loaded 構造体は apr_thread_rwlock_t を使用して、複数スレッドからの同時ロードを制御しています。
