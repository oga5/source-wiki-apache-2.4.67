modules/cluster/mod_heartmonitor.c

概要

mod_heartmonitor は、バックエンドサーバーから送信されるハートビート信号（UDP マルチキャストまたは HTTP POST）を監視し、サーバーの稼働状況（busy/ready 状態）を収集・集約する Apache HTTP Server モジュールです。収集したデータは、共有メモリ（slotmem）またはファイルシステム上のファイルに永続化され、ロードバランサー等の他モジュールが参照可能な状態を提供します。

モジュール内での役割

modules/cluster サブシステムにおいて、本モジュールは「監視者」として機能します。mod_heartbeat が送信する情報を集約し、mod_lbmethod_heartbeat 等のロードバランサーが動的にバックエンドの負荷を判断するためのデータソースを構築します。

責務

  • **監視と収集**: UDP マルチキャストソケットのリスニング、または HTTP ハンドラ経由でのハートビート情報の受信。
  • **データ永続化**: 収集したサーバー状態を slotmem（共有メモリ）またはファイルに書き出し、更新する。
  • **ライフサイクル管理**: mod_watchdog を利用したバックグラウンドでの監視タスクの実行と、サーバー起動・停止時のリソース管理。
  • **設定管理**: HeartbeatListen 等のディレクティブを通じた監視対象アドレスや保存先の設定。

重要なシンボル

コンテキストと状態

  • hm_ctx_t: モジュールのサーバー単位の設定と実行時状態を保持する構造体。
  • hm_server_t: 個々のバックエンドサーバーの稼働状態（IP, ポート, busy/ready 数, 最終確認時刻）。
  • slotmem / storage: 共有メモリプロバイダーへの参照。

ライフサイクルとフック

  • hm_post_config: モジュールの初期化、mod_watchdog の登録、共有メモリの確保を行う。
  • hm_watchdog_callback: mod_watchdog から呼び出されるメインループ。ソケットのポーリングと統計情報の定期更新を担当。
  • hm_handler: HTTP POST 経由でハートビートを受け取るためのハンドラ。

データ更新・変換

  • hm_update_stats / hm_update_stat: 共有メモリまたはファイルへの書き込みを抽象化するディスパッチャ。
  • qs_to_table: クエリ文字列形式のハートビートメッセージを apr_table_t に変換するユーティリティ。

制御フローとデータフロー

1. **初期化**: hm_post_config が mod_watchdog を取得し、hm_watchdog_callback を登録します。 2. **監視ループ**: hm_watchdog_callback 内で apr_poll を使用し、UDP ソケットからの入力を待機します。 3. **メッセージ処理**: hm_recv がパケットを受信し、hm_processmsg が内容を解析して hm_server_t を更新（または新規作成）します。 4. **永続化**: hm_watchdog_callback の RUNNING 状態において、hm_update_stats が定期的に呼び出され、ハッシュテーブル内の全サーバー情報を slotmem またはファイルへフラッシュします。

読者向けメモ

  • **メモリ管理**: hm_ctx_t はサーバーごとの設定を保持し、apr_pool_t を使用してライフサイクルを管理しています。hm_get_server で生成されるサーバー情報は ctx->p プールに割り当てられます。
  • **排他制御**: slotmem の操作において、grab/put を使用した簡易的な更新が行われていますが、複雑なロック機構は slotmem プロバイダーに依存しています。
  • **制限事項**: HeartbeatMaxServers が 0 の場合はファイルベースのストレージが使用されますが、ファイル更新時は一時ファイルを作成してリネームするアトミックな更新を行っています。
  • **未実装・課題**: HM_UPDATE_SEC は現在マクロ定義ですが、実行時設定への移行が TODO とされています。また、一部のロギングやエラー処理は開発用コードが残っている箇所があります。
