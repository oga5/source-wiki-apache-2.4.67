modules/cluster/mod_heartbeat.c

概要

mod_heartbeat は、Apache HTTP Server の現在の負荷状況（Ready/Busy 状態）を定期的に UDP マルチキャストで外部へ通知するためのモジュールです。mod_watchdog を利用してバックグラウンドで動作し、サーバーのスコアボード情報を集計して送信します。

モジュール内での役割

modules/cluster サブシステムにおいて、本モジュールは「情報発信側」の役割を担います。対となる mod_heartmonitor が受信した情報を収集・管理するのに対し、本モジュールは自身の稼働状況をネットワーク上にブロードキャストします。

責務

  • **負荷情報の集計**: スコアボードを走査し、現在のプロセス/スレッドの稼働状態（Ready/Busy）をカウントする。
  • **定期通知の実行**: mod_watchdog のフックを利用し、一定間隔で UDP パケットを送信する。
  • **設定管理**: HeartbeatAddress ディレクティブによるマルチキャスト先アドレスの定義と、モジュール設定の保持。

重要なシンボル

設定・状態管理

  • hb_ctx_t: モジュールのサーバー設定構造体。マルチキャスト先アドレス、スレッド/サーバー制限値、アクティブフラグを保持。
  • heartbeat_module: Apache モジュール定義構造体。

Watchdog フック（ライフサイクル）

  • hb_watchdog_init: MPM の制限値（スレッド/サーバー数）を初期化時に取得。
  • hb_watchdog_step: メインのループ処理。hb_monitor を呼び出し、負荷情報を送信する。
  • hb_watchdog_need: AP_WATCHDOG_SINGLETON を要求し、単一プロセスでの実行を保証。

処理・設定コマンド

  • hb_monitor: スコアボードを走査し、UDP ソケット経由で v=1&ready=X&busy=Y 形式の文字列を送信する。
  • cmd_hb_address: HeartbeatAddress ディレクティブのハンドラ。アドレスの解析と hb_ctx_t への格納を行う。

制御フローとデータフロー

1. **初期化**: hb_register_hooks が mod_watchdog の各フェーズを登録。 2. **設定**: cmd_hb_address が設定を解析し、hb_ctx_t に apr_sockaddr_t を生成・保持。 3. **定期実行**: mod_watchdog が hb_watchdog_step を呼び出す。 4. **集計と送信**:

  • hb_monitor が ap_scoreboard_image を走査。
  • mpm_generation を確認し、現在の世代のプロセスのみをカウント対象とする。
  • apr_socket_sendto を使用して UDP パケットを送信。
  • 送信後、ソケットは即座にクローズされる。

直接依存

  • mod_watchdog.h: 定期実行のトリガー。
  • scoreboard.h: サーバーの稼働状況取得。
  • ap_mpm.h: MPM の制限値および世代情報の取得。
  • apr_network_io.h (間接的): ソケット通信。
  • apr_strings.h
  • http_config.h
  • http_log.h
  • httpd.h

関連コード

  • modules/cluster/mod_heartmonitor.c: 本モジュールが送信したパケットを受信し、クラスター全体の負荷状況を追跡する対となるモジュール。

読者向けメモ

  • **メモリ管理**: hb_ctx_t はサーバープール (p) 上に割り当てられます。ソケット通信に使用するプールは hb_monitor に渡される一時的なプールであり、送信ごとにソケットの生成・破棄が行われます。
  • **制約**: HeartbeatAddress はグローバルコンテキストでのみ指定可能です。
  • **エラー処理**: ソケット作成や送信失敗時は ap_log_error で警告が出力されますが、処理は継続されます。
  • **未実装/浅い部分**: 本モジュールは送信のみに特化しており、受信側の確認やパケットの暗号化・認証機能は含まれていません。また、HEARTBEAT_INTERVAL はコンパイル時定数として固定されており、実行時の動的な変更はできません。
