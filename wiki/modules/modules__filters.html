modules/filters

システム内での役割

modules/filters は、Apache HTTP Server のフィルタチェーンにおいて、コンテンツの変換、圧縮、解析、およびリクエスト制御を行うモジュール群を格納するディレクトリです。本モジュールは、ディレクトリ境界に基づき、ストリーム処理を行う多様なフィルタ実装を統合しています。

責務

  • **コンテンツ変換**: HTML/XML の置換、文字コード変換、外部プロセスによるフィルタリング。
  • **データ圧縮**: Gzip (mod_deflate) や Brotli (mod_brotli) によるレスポンスの最適化。
  • **リクエスト制御**: SSI (mod_include) による動的生成、レート制限 (mod_ratelimit)、タイムアウト監視 (mod_reqtimeout)。
  • **ストリーム処理基盤**: libsed を用いた正規表現ベースのストリーム編集機能の提供。

このモジュールのファイル

  • libsed.h, sed.h, sed0.c, sed1.c, regexp.c, regexp.h (sed エンジン)
  • mod_brotli.c, mod_deflate.c (圧縮フィルタ)
  • mod_include.c, mod_include.h (SSI フィルタ)
  • mod_xml2enc.c, mod_xml2enc.h (文字コード変換)
  • mod_charset_lite.c (軽量文字コード変換)
  • mod_proxy_html.c (HTML URL 置換)
  • mod_substitute.c (正規表現置換)
  • mod_sed.c (sed フィルタ)
  • mod_filter.c (フィルタディスパッチャ)
  • mod_ext_filter.c (外部プロセスフィルタ)
  • mod_ratelimit.c, mod_ratelimit.h (レート制限)
  • mod_reqtimeout.c (タイムアウト制御)
  • mod_request.c (リクエストボディ保持)
  • mod_data.c, mod_buffer.c, mod_reflector.c (その他ユーティリティ)

主要なデータ構造

  • sed_commands_t, sed_reptr_t: libsed のコンパイル済みコマンド構造。
  • sed_eval_t: libsed の実行時評価コンテキスト。
  • include_ctx_t: SSI 処理の状態管理構造体。
  • brotli_ctx_t / deflate_ctx: 圧縮フィルタのエンコーダ状態。
  • subst_dir_conf: mod_substitute の設定構造体。

主なフロー

1. **フィルタ登録**: mod_filter 等がフックを通じてフィルタをチェーンに挿入。 2. **データ読み込み**: ap_get_brigade を通じてバケットブリゲードを順次取得。 3. **変換処理**: 各モジュールがバケットを走査・改変（圧縮、置換、解析）。 4. **ストリーム出力**: ap_pass_brigade を通じて後続のフィルタへデータを渡す。 5. **終了処理**: EOS (End of Stream) バケット受信時に各モジュールのコンテキストをクリーンアップ。

外部依存

  • apr_buckets.h, util_filter.h: Apache フィルタフレームワークの基盤。
  • zlib.h (mod_deflate), brotli/encode.h (mod_brotli): 圧縮ライブラリ。
  • libxml/HTMLparser.h (mod_proxy_html, mod_xml2enc): XML/HTML 解析。
  • ap_expr.h: 条件式評価エンジン。

内部構成

本モジュールは、個別の機能を持つ独立したフィルタモジュール群と、それらを支える共通ライブラリ（libsed）で構成されています。

推奨読書順

1. mod_filter.c (フィルタチェーンの仕組みを理解) 2. mod_deflate.c (標準的な圧縮フィルタの構成) 3. libsed.h / sed0.c / sed1.c (sed エンジンの基盤) 4. mod_include.c (複雑なステートマシンを持つフィルタの例) 5. mod_proxy_html.c (外部ライブラリ連携の例)

未解決の問い

  • 各フィルタモジュール間でのメモリ管理ポリシーの統一（apr_pool_t の使い分けの最適化）。
  • libsed の現代的な正規表現エンジン（PCRE 等）への移行可能性。
