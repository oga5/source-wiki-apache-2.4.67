modules/lua/mod_lua.c

概要

mod_lua.c は Apache HTTP Server の Lua モジュールのメインエントリポイントです。HTTP リクエスト処理の各フェーズ（ハンドラ、フィルタ、認証、フック）を Lua スクリプトに委譲するためのブリッジ機能を提供します。

モジュール内での役割

modules/lua サブシステムにおける「司令塔」です。設定のパース、Lua VM のライフサイクル管理、Apache の各フックと Lua 関数を結びつけるハーネス（Harness）の実装を担います。

責務

  • **設定管理**: LuaRoot, LuaScope, LuaHook* などのディレクティブを処理し、ディレクトリ単位の設定をマージする。
  • **フックハーネス**: Apache の各フェーズ（translate_name, fixups, authz 等）を Lua 関数へルーティングする。
  • **フィルタ管理**: 入出力フィルタの登録と、Lua コルーチンを用いたストリーム処理の制御。
  • **VM ライフサイクル**: create_vm_spec を通じた VM の生成と、ap_lua_release_state によるリソース解放。
  • **共有メモリ管理**: マルチプロセス環境下での VM 共有のためのグローバルミューテックスと共有メモリの初期化。

重要なシンボル

設定・構造体

  • ap_lua_dir_cfg: ディレクトリ単位の Lua 設定（パス、ハンドラ、フィルタ、VM スコープ）。
  • lua_filter_ctx: フィルタ処理中のコンテキスト。Lua コルーチン（lua_State）とバケットブリゲードを保持。
  • lua_authz_provider_spec: 認証プロバイダとして登録された Lua 関数のメタデータ。

フック・ハンドラ

  • lua_handler: lua-script ハンドラを処理するメイン関数。
  • lua_request_rec_hook_harness: 各種フックを Lua に委譲する汎用ハーネス。
  • lua_output_filter_handle / lua_input_filter_handle: Lua を用いたストリームフィルタリングの実装。

ライフサイクル・ユーティリティ

  • create_vm_spec: リクエスト情報に基づき、VM の実行環境（スコープ、プール）を構築する。
  • ap_lua_release_state: VM の状態を解放、またはリソースリストへ返却する。
  • report_lua_error: Lua スタックからエラーを抽出し、HTTP 500 レスポンスとして出力する。

制御フローとデータフロー

1. **初期化**: lua_register_hooks が Apache の各フックポイントにハーネス関数を登録。 2. **リクエスト処理**:

  • 各フックが呼ばれると、lua_request_rec_hook_harness が設定から該当する Lua スクリプトを特定。
  • create_vm_spec が VM のスコープ（Once, Request, Conn, Server）を決定し、ap_lua_get_lua_state で VM を取得。
  • lua_pcall で Lua 関数を実行し、戻り値を Apache のステータスコードに変換。

    • 3. **フィルタ処理**:

  • lua_setup_filter_ctx で Lua コルーチンを初期化。
  • lua_resume を使用して、バケットブリゲードのデータが渡されるたびに Lua 側で処理（Yield/Resume）を行う。

読者向けメモ

  • **メモリ管理**: apr_pool_t を多用しています。VM のスコープ（vm_scope）によって、VM がどのプールに紐付くかが決まるため、メモリリークを防ぐには ap_lua_release_state の呼び出しが必須です。
  • **コルーチン**: フィルタ処理では lua_resume を使用しており、Lua 側で coroutine.yield() を呼び出すことで Apache のフィルタチェーンと協調動作します。
  • **未実装**: LuaHookInsertFilter はソース内で未実装としてマークされています。
  • **ByteCodeHack**: 設定パース時に Lua コードをバイトコード化して保持する「ハック」が含まれており、複雑なディレクティブ処理を行っています。
