modules/http2/h2_workers.c

概要

HTTP/2 サブシステムにおける「セカンダリ接続」を処理するための専用スレッドプールを管理するモジュールです。動的なスレッド生成・破棄、接続プロデューサー（ap_conn_producer_t）の管理、およびスレッド間での作業割り当てを担います。

モジュール内での役割

h2_mplx や h2_c2 と連携し、HTTP/2 ストリームの並列処理を支える実行基盤を提供します。MPM（Multi-Processing Module）が管理するメインスレッドとは別に、HTTP/2 特有の多重化された接続を処理するワーカー群を制御します。

責務

  • **スレッドプール管理**: min_active から max_slots の範囲でワーカーを動的に増減させ、アイドルタイムアウトによるスレッド削減を行う。
  • **接続プロデューサー管理**: 接続の供給元（プロデューサー）を登録・アクティブ化し、ワーカーが接続をプルする仕組みを提供。
  • **ライフサイクル制御**: ワーカーの起動、終了、およびシャットダウン時のクリーンアップ。
  • **同期**: スレッドセーフなリングバッファ（apr_ring）を用いたタスクキューイングと条件変数によるスレッドのウェイクアップ。

重要なシンボル

構造体

  • h2_workers: プール全体の管理構造体。スレッド属性、スロット配列、および状態別（free, idle, busy, zombie）のリングバッファを保持。
  • h2_slot: 個々のワーカーの状態を保持。スレッドハンドル、条件変数、および現在処理中のプロデューサーへの参照を持つ。
  • ap_conn_producer_t: 接続の供給元。fn_next（接続取得）、fn_done（完了通知）等のコールバックを保持。

主要関数

  • **ライフサイクル**: h2_workers_create（初期化）、h2_workers_shutdown（停止）、workers_pool_cleanup（破棄）。
  • **プロデューサー操作**: h2_workers_register（登録）、h2_workers_activate（処理開始）、h2_workers_join（完了待ち）。
  • **内部制御**: slot_run（ワーカーのメインループ）、activate_slot（新規スレッド生成）、get_next（プロデューサーからのタスク取得）。

制御フローとデータフロー

1. **タスク投入**: h2_workers_activate がプロデューサーを prod_active リングに追加し、wake_idle_worker を呼び出す。 2. **ワーカー起動**: wake_idle_worker はアイドル中のスロットを検索し、条件変数 more_work をシグナルする。スロットが足りない場合は activate_slot で新規スレッドを生成する。 3. **タスク実行**: slot_run 内のループで get_next を呼び出し、プロデューサーから conn_rec を取得。ap_process_connection または h2_c2_process を実行する。 4. **完了処理**: 接続処理終了後、fn_done を呼び出し、prod_done 条件変数をブロードキャストして h2_workers_join を待機しているスレッドに通知する。

読者向けメモ

  • **メモリ所有権**: h2_workers は独自の apr_pool_t を持ち、スレッド生成時に各スロットへサブプールを割り当てます。workers_pool_cleanup は MPM シャットダウン時に呼ばれ、全スレッドの終了を待機します。
  • **スレッド安全性**: すべてのリング操作と状態遷移は workers->lock によって保護されています。
  • **エッジケース**: slot_run 内の接続 ID 生成ロジック（c->id = (c->master->id << 8)^slot->id）は、mod_unique_id との互換性を保つための苦肉の策であり、非常に高い並列度では衝突の可能性があることに注意してください。
  • **省略箇所**: AP_HAS_THREAD_LOCAL 定義時の挙動や、特定のプラットフォーム依存のスタックサイズ設定については、標準的な MPM の挙動に準拠しているため詳細を省略しています。
