modules/http2/h2_push.c

概要

HTTP/2 のサーバープッシュ機能（Server Push）を管理するモジュールです。主に、レスポンスヘッダーに含まれる Link ヘッダーの解析によるプッシュ候補の抽出と、重複プッシュを防ぐための「プッシュ日記（Push Diary）」の実装を担当します。

モジュール内での役割

mod_http2 の一部として、HTTP/2 ストリームのライフサイクル中にプッシュすべきリソースを特定し、クライアントへ重複したプッシュを送信しないよう制御します。

責務

  • **Link ヘッダー解析**: Link ヘッダー（rel=preload）を構文解析し、プッシュ対象の URI やパラメータを抽出する。
  • **プッシュ候補の生成**: 抽出された情報に基づき、h2_request オブジェクトを構築する。
  • **プッシュ日記（Push Diary）の管理**: 過去にプッシュしたリソースのハッシュ値を保持し、重複プッシュを抑制する。
  • **キャッシュダイジェストのエンコード**: Golomb Coded Set (GCS) を用いて、プッシュ日記の内容をバイナリ形式に圧縮・変換する。

重要なシンボル

Link ヘッダー解析

  • link_ctx: 解析状態を保持するコンテキスト構造体。
  • inspect_link(): Link ヘッダー文字列を走査し、各リンクを解析するエントリポイント。
  • add_push(): 解析結果からプッシュ条件（preload 等）を検証し、h2_push を生成する。

プッシュ日記（Push Diary）

  • h2_push_diary: プッシュ済みリソースのハッシュ履歴を保持する構造体。
  • h2_push_diary_create(): 日記インスタンスの初期化。
  • h2_push_diary_update(): 新たなプッシュ候補を日記に登録し、重複を確認する。
  • h2_push_diary_digest_get(): 日記の内容を GCS 形式でシリアライズする。

ハッシュ計算

  • calc_sha256_hash(): OpenSSL 使用時のハッシュ生成（SHA256）。
  • calc_apr_hash(): OpenSSL 未使用時のフォールバック（apr_hashfunc_default）。

制御フローとデータフロー

1. **収集**: h2_push_collect() がレスポンスヘッダーを走査し、link_ctx を用いて Link ヘッダーを解析。 2. **検証**: add_push() が rel=preload を確認し、同一権限（authority）内であることを検証して h2_push を作成。 3. **更新**: h2_push_diary_update() が、生成されたプッシュ候補のハッシュを計算し、日記に存在するか確認。

  • 存在する場合: move_to_last() で最近使用済みリストの末尾へ移動。
  • 存在しない場合: h2_push_diary_append() で追加。

    • 4. **圧縮**: h2_push_diary_digest_get() が日記内のハッシュをソートし、gset_encoder を用いてビットストリームとしてエンコード。

読者向けメモ

  • **メモリ管理**: link_ctx や h2_push_diary はすべて apr_pool_t を介して管理されます。
  • **キャッシュダイジェスト**: 本実装は IETF で廃案となった Cache-Digest ドラフトの概念を継承していますが、現在はクライアントからの値を受け取るのではなく、サーバー側のプッシュ履歴を管理する目的で利用されています。
  • **制約**: h2_push_diary は最大サイズ（N）に達すると古いエントリから破棄する LRU 的な挙動をします。
  • **未実装/浅い部分**: h2_push_collect 内の TODO にある通り、現在は Link ヘッダーのみがプッシュのソースですが、将来的にフックによる拡張が想定されています。
