modules/http2/h2_ws.c

概要

HTTP/2 上での WebSocket (RFC 8441) をサポートするための変換レイヤーです。HTTP/2 の CONNECT メソッドを内部的な HTTP/1.1 形式の GET リクエストに書き換え、WebSocket ハンドシェイクの整合性を検証するフィルタを提供します。

モジュール内での役割

modules/http2 サブシステムにおいて、HTTP/2 ストリームを WebSocket 接続として扱うためのブリッジとして機能します。主に h2_c2（子接続）のコンテキストで動作し、プロトコル変換とレスポンスのインターセプトを担当します。

責務

  • **リクエスト変換**: HTTP/2 の CONNECT メソッドを、バックエンドが処理可能な GET + Upgrade ヘッダー付きリクエストへ変換。
  • **ハンドシェイク検証**: Sec-WebSocket-Accept ヘッダーを生成・検証し、WebSocket 接続の成否を判定。
  • **レスポンスフィルタリング**: H2_C2_WS_OUT フィルタを通じて、不正なレスポンスを 502 Bad Gateway に変換し、成功時には 101 Switching Protocols を 200 OK に変換して透過させる。

重要なシンボル

リクエスト変換・フック

  • h2_ws_rewrite_request: CONNECT リクエストを WebSocket 用の GET リクエストに変換し、出力フィルタを挿入するメイン関数。
  • h2_ws_register_hooks: post_read_request フックおよび出力フィルタ H2_C2_WS_OUT を登録。

フィルタコンテキストと状態

  • struct ws_filter_ctx: フィルタの実行状態（Accept キー、レスポンス完了フラグ、ボディ上書きフラグ）を保持する構造体。
  • h2_c2_ws_filter_out: レスポンスバケットを検査し、WebSocket ハンドシェイクの結果に応じてレスポンスを書き換える出力フィルタ。

ユーティリティ

  • gen_ws_accept: RFC 6455 に基づき、Sec-WebSocket-Key から Sec-WebSocket-Accept を生成する SHA-1 ハッシュ関数。
  • make_valid_resp / make_invalid_resp: フィルタ内でレスポンスを差し替える際に使用するバケット生成関数。

制御フローとデータフロー

1. **リクエスト書き換え**: h2_ws_rewrite_request が呼び出されると、リクエストをクローンし、Upgrade ヘッダーを追加。Sec-WebSocket-Key を生成し、出力フィルタ H2_C2_WS_OUT を H2_C2_NET_OUT の直前に挿入。 2. **レスポンス検査**: H2_C2_WS_OUT フィルタがレスポンスバケットを捕捉。ws_handle_resp が Sec-WebSocket-Accept を検証。 3. **レスポンス変換**:

  • 検証成功時: 101 を 200 に変換。
  • 検証失敗時: 502 を生成し、以降のボディを破棄（override_body フラグをセット）。

    • 4. **終了処理**: ws_post_read フックが、アップグレードされた接続が WebSocket をサポートしているか確認し、未サポートなら 501 Not Implemented を返す。

読者向けメモ

  • **メモリ管理**: フィルタコンテキストや変換後のリクエストは、すべて c2->pool（子接続のプール）を使用して割り当てられており、接続終了時に自動解放されます。
  • **制約**: H2_USE_WEBSOCKETS マクロが定義されていない場合、本モジュールは NOP（何もしない）として動作します。
  • **エッジケース**: 101 レスポンスが返された際、Sec-WebSocket-Accept が欠落している場合や不一致の場合は、セキュリティ上の理由から 502 を返して接続を終了させます。
  • **省略箇所**: 本ファイルは WebSocket のデータフレーム転送そのものではなく、あくまで「ハンドシェイクの変換」に特化しています。実際のデータ転送は HTTP/2 のストリームフレームとして透過的に処理されます。
