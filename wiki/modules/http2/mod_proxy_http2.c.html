modules/http2/mod_proxy_http2.c

概要

mod_proxy_http2.c は、Apache HTTP Server のプロキシサブシステムにおいて、バックエンドへの HTTP/2 通信を担うプロキシモジュールです。mod_proxy のスキームハンドラとして機能し、クライアントからのリクエストを HTTP/2 プロトコルを用いてバックエンドサーバーへ転送・処理します。

モジュール内での役割

mod_proxy フレームワークと mod_http2 コア機能の橋渡しを行います。h2_proxy_session などのユーティリティを呼び出し、HTTP/2 セッションのライフサイクル管理とリクエストの多重化を抽象化します。

責務

  • **プロキシハンドリング**: h2: および h2c: スキームのリクエストを捕捉し、バックエンド接続を確立する。
  • **セッション管理**: h2_proxy_ctx を通じて、バックエンドとの HTTP/2 セッションの再利用性、接続状態、リクエストの完了状態を追跡する。
  • **リクエストの正規化**: proxy_http2_canon により、プロキシリクエストの URL を標準化し、転送可能な形式に変換する。
  • **エラーハンドリングと再試行**: 接続エラーやセッション切断時の再試行ロジック、および proxy-error-override に基づくステータスコードの制御。

重要なシンボル

コンテキスト管理

  • h2_proxy_ctx: リクエスト処理中の状態を保持する構造体。接続情報、リクエストの状態、セッションの再利用フラグを含む。

フック関数

  • proxy_http2_handler: プロキシのメインハンドラ。接続の取得からセッション実行までを統括する。
  • proxy_http2_canon: URL の正規化を行うハンドラ。
  • h2_proxy_post_config: モジュールの初期化と nghttp2 バージョンのログ出力を行う。

セッション操作

  • ctx_run: HTTP/2 セッションをセットアップし、リクエストをサブミットして完了までループする。
  • add_request: セッションに対して新しいリクエストを投入する。
  • request_done / session_req_done: リクエスト完了時のコールバック。ステータスの更新と再試行判定を行う。

制御フローとデータフロー

1. **初期化**: register_hook により proxy_hook_scheme_handler と canon_handler が登録される。 2. **正規化**: proxy_http2_canon が URL を解析し、proxy: スキームの内部表現に変換する。 3. **接続確立**: proxy_http2_handler が ap_proxy_acquire_connection を呼び出し、バックエンドへの接続を確保する。 4. **セッション実行**: ctx_run が h2_proxy_session_setup を経由してセッションを開始し、h2_proxy_session_process で I/O を駆動する。 5. **終了処理**: リクエスト完了後、セッションが再利用可能か判定し、ap_proxy_release_connection で接続を解放する。

読者向けメモ

  • **メモリ管理**: h2_proxy_ctx はリクエストのプール (r->pool) に割り当てられており、リクエスト終了時に自動的に解放されます。
  • **再試行ロジック**: reconnects 変数により、最大 1 回の再試行が許可されています。
  • **セッションの再利用**: has_reusable_session フラグが重要です。セッションがクリーン（オープンなストリームがない状態）であれば、次のリクエストのために接続を維持します。
  • **省略された詳細**: nghttp2 の詳細なフレーム処理や、h2_proxy_session 内部のバッファリングアルゴリズムについては本ファイルでは抽象化されており、h2_proxy_session.c に委譲されています。
