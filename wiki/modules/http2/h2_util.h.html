modules/http2/h2_util.h

概要

h2_util.h は、Apache HTTP/2 モジュール全体で使用される汎用的なユーティリティ関数、データ構造、およびマクロを定義するヘッダーファイルです。HTTP/2 プロトコル特有の処理（ヘッダー変換、Base64URL エンコード、nghttp2 連携）から、低レベルなデータ構造（ハッシュ、キュー、バケット操作）までを網羅しています。

モジュール内での役割

modules/http2 サブシステムにおける「共通基盤」として機能します。特定のビジネスロジックを持たず、ストリーム管理、セッション制御、プロトコル変換など、他のすべてのコンポーネントが依存する低レイヤーの操作を提供します。

責務

  • **データ構造の提供**: 整数 ID をキーとするハッシュ (h2_ihash)、ソート可能な整数キュー (h2_iqueue)、およびスレッドセーフな FIFO キュー (h2_fifo, h2_ififo) の管理。
  • **HTTP/2 プロトコル支援**: ヘッダーの正規化、Push Policy の判定、Base64URL エンコード/デコード。
  • **nghttp2 連携**: nghttp2_nv 構造体への変換や、リクエスト/レスポンスヘッダーの構築支援。
  • **APR Brigade 操作**: バケットブリゲードの連結、コピー、メモリ使用量の見積もり、およびデバッグ用のダンプ出力。

重要なシンボル

データ構造

  • h2_ihash_t: 整数 ID をキーとするハッシュテーブル。h2_ihash_create で生成し、h2_ihash_add/get/remove で操作。
  • h2_iqueue: ストリーム ID 等を管理するためのソート可能な整数キュー。
  • h2_fifo / h2_ififo: スレッドセーフな FIFO キュー。h2_fifo_push/pull によるブロッキング操作や、h2_fifo_peek による非破壊的な先頭要素操作が可能。

ヘッダー・プロトコル変換

  • h2_ngheader: nghttp2 が要求するヘッダー形式への変換結果を保持する構造体。
  • h2_req_add_header: リクエストヘッダーをテーブルに追加する際のバリデーションと変換。
  • h2_push_policy_determine: RFC に基づく Push Policy の決定。

デバッグ・ユーティリティ

  • h2_util_bb_log: バケットブリゲードの内容をログ出力するマクロ。デバッグ時に非常に有用。
  • h2_util_hex_dump: バイナリデータの 16 進ダンプ生成。

制御フローとデータフロー

  • **キュー操作**: h2_fifo_peek は、キューのロックを保持したままコールバック関数を実行する設計となっており、競合を防ぎつつ先頭要素の安全な検査を可能にします。
  • **ブリゲード操作**: h2_append_brigade は、h2_bucket_gate コールバックを使用して、条件に応じたバケットの転送制御を行います。
  • **メモリ管理**: ほとんどの構造体は apr_pool_t を引数に取り、APR のメモリ管理ライフサイクルに従います。

直接依存

  • nghttp2/nghttp2.h: HTTP/2 プロトコルスタック。
  • http_protocol.h: Apache HTTPD のプロトコル処理基盤。
  • h2.h, h2_headers.h: モジュール内の定数定義やヘッダー構造体。

関連コード

  • h2_util.c: 本ヘッダーで宣言された関数の実装。
  • h2_bucket_beam.c: ブリゲードの転送（Beam）に関連するより高度なバケット操作。
  • h2_headers.c: HTTP/2 ヘッダーの具体的な処理ロジック。

読者向けメモ

  • **メモリ所有権**: h2_ihash や h2_iqueue で管理される要素のメモリ所有権は、呼び出し元が管理する場合と、コンテナが管理する場合があるため、各関数のドキュメントを確認してください。
  • **スレッド安全性**: h2_fifo はスレッドセーフですが、h2_iqueue や h2_ihash は明示的なロックを必要とする場合があります。
  • **省略された詳細**: 本ファイルはユーティリティの宣言に特化しており、具体的なアルゴリズム（Base64URL の変換テーブル等）は h2_util.c に隠蔽されています。
  • **パフォーマンス**: h2_util_bb_log は APLOG_C_IS_LEVEL で囲まれており、ログレベルが低い場合はオーバーヘッドが最小化されるよう設計されています。
