test/modules/proxy

システム内での役割

Apache HTTP Server の mod_proxy モジュールおよび関連機能（proxy_http, proxy_balancer 等）の動作を検証するための統合テストスイートです。ディレクトリ境界に基づき、プロキシ機能のテスト環境構築から実行までをカプセル化しています。

責務

  • **テスト環境の構築**: mod_proxy 関連モジュールをロードした Apache HTTP Server のライフサイクル管理。
  • **プロキシ動作の検証**: リバースプロキシ、フォワードプロキシ、ロードバランサ、および Unix Domain Socket (UDS) 経由の通信の整合性確認。
  • **セキュリティ制約のテスト**: 不正なパス指定に対するエラーハンドリングの検証。

このモジュールのファイル

  • test/modules/proxy/__init__.py: パッケージ定義。
  • test/modules/proxy/conftest.py: pytest フィクスチャおよび環境のライフサイクル管理。
  • test/modules/proxy/env.py: テスト環境（ProxyTestEnv）の定義と設定管理。
  • test/modules/proxy/test_01_http.py: HTTP/HTTPS プロキシおよびロードバランサのテスト。
  • test/modules/proxy/test_02_unix.py: UDS 経由のプロキシ機能およびセキュリティテスト。

主要なデータ構造

  • ProxyTestEnv: テスト対象の Apache 設定、ドメイン定義、証明書スペックを保持する環境クラス。
  • UDSFaker: test_02_unix.py で使用される、UDS をリッスンし固定レスポンスを返すスレッドベースのモックサーバー。

主なフロー

1. **セットアップ**: conftest.py が ProxyTestEnv を初期化し、Apache を起動。env.py が必要なモジュールをロードし、仮想ホスト設定を生成。 2. **テスト実行**: 各テストケースが curl 等を用いてプロキシ経由でリクエストを送信。 3. **検証**: レスポンスのステータスコードや JSON レスポンス内のホスト情報をアサートし、ルーティングの正当性を確認。 4. **クリーンアップ**: 全テスト終了後、conftest.py のフィクスチャが Apache プロセスを停止。

外部依存

  • pytest: テストフレームワーク。
  • pyhttpd: Apache HTTP Server のテスト用フレームワーク（設定生成 HttpdConf や環境基底クラス HttpdTestEnv を提供）。

内部構成

本モジュールは pyhttpd フレームワークを拡張し、プロキシ特有のドメイン管理やモジュールロードを抽象化しています。子モジュールは存在せず、フラットな構成です。

推奨読書順

1. test/modules/proxy/env.py (環境定義の理解) 2. test/modules/proxy/conftest.py (フィクスチャとライフサイクルの理解) 3. test/modules/proxy/test_01_http.py (基本機能のテスト) 4. test/modules/proxy/test_02_unix.py (応用機能のテスト) 5. test/modules/proxy/__init__.py

未解決の問い

  • 現在の UDSFaker は簡易的な実装であるため、より複雑な HTTP プロトコル解析が必要になった場合の拡張性。
  • テスト実行時のログ出力が冗長になる傾向があるため、デバッグ効率を考慮したロギング設定の最適化。
