Overview

このシステムは何か

httpd-2.4.67 は、モジュール拡張可能なアーキテクチャを持つ、業界標準のオープンソース HTTP サーバーです。615 個のソースファイルと 72 個の論理モジュールで構成され、HTTP/1.1 および HTTP/2 プロトコル、動的コンテンツ生成、認証・認可、プロキシ、キャッシュ、および証明書管理（Managed Domain）を統合的に提供します。

アーキテクチャ概観

本システムは、コアサーバー（server/）がリクエストのライフサイクルを管理し、フックシステムを通じて各モジュール（modules/）が特定の処理フェーズを拡張する疎結合な設計です。プラットフォーム固有の差異は os/ ディレクトリで抽象化され、マルチプロセッシングモデル（MPM）は server/mpm/ に分離されています。

実行時ライフサイクル

現時点では、サーバー起動時の設定解析、MPM によるプロセス/スレッド生成、リクエスト受信時のフックチェーン実行という大枠のみが特定されています。各フェーズ（post_read_request, translate_name, fixups, handler 等）の詳細な遷移と、モジュール間での request_rec の受け渡しフローは、今後精緻化が必要です。

LLM Call Flows

  1. Module: docs/manual/developer主なフロー

    1. **フック登録**: register_hooks 関数が Apache 起動時に呼び出され、ap_hook_handler を通じてハンドラを登録する。 2. **リクエスト判定**: 各ハンドラ関数内で r->handler を確認し、対象外であれば DECLINED を返して処理を委譲する。 3. **処理実行**: 対象リクエストの場合、ap_rprintf 等を用いてレスポンスを生成し、OK を返して処理を完了する。 4. **リソース管理**: apr_pool_t を活用し、リクエスト終了時に自動的なメモリ解放を行う。

  2. Module: include主なフロー

    本モジュールはインターフェース定義群であるため、特定の単一フローを持ちませんが、以下のライフサイクルを支えています。 1. **設定読み込み**: http_config.h を通じてディレクティブを解析し、設定ツリーを構築。 2. **リクエスト処理**: httpd.h の構造体を受け渡し、util_filter.h のフィルタチェーンを通じてデータを処理。 3. **MPM 制御**: ap_mpm.h と scoreboard.h を用いてプロセス/スレッドの状態を管理。

  3. Module: modules/aaa主なフロー

    1. **認証フェーズ**: mod_authn_core が設定に基づき適切なプロバイダーを呼び出し、ユーザー資格情報を検証。成功時に r->user を設定。 2. **認可フェーズ**: mod_authz_core が Require ディレクティブに従い、登録された認可プロバイダーを再帰的に評価。 3. **キャッシュ連携**: mod_authn_socache 等が認証結果を一時保存し、次回以降の認証処理を高速化。

  4. Module: modules/arch/netware主なフロー

    1. **初期化**: libprews.c が NLM としての基盤を構築し、mod_nw_ssl.c が SecureListen ディレクティブに基づき WinSock を介して SSL ソケットを生成・登録します。 2. **リクエスト処理**: mod_netware.c が CGI リクエストの拡張子を解析し、対応する NLM を特定して実行コマンドを構築します。 3. **SSL アップグレード**: mod_nw_ssl.c が Upgrade ヘッダーを検知すると、ソケットを暗号化状態へ遷移させます。 4. **終了**: libprews.c が NLM アンロード時にネットワークリソースを解放します。

  5. Module: modules/arch/unix主なフロー

    1. **起動時**: mod_unixd が設定を読み込み、mod_systemd が起動完了をサービスマネージャへ通知。 2. **プロセス生成時**: ap_unixd_setup_child() が呼び出され、グループ権限の確定、chroot の実行、最終的な setuid による権限降格が行われる。 3. **リクエスト処理時**: mod_privileges が有効な場合、リクエストごとに権限セットの動的変更やプロセス分離（fork）が実行される。 4. **監視時**: mod_systemd が定期的にスコアボードから統計を読み取り、systemd へステータスを送信。

  6. Module: modules/arch/win32主なフロー

    1. **ISAPI リクエスト**: isapi_handler が DLL をロードし、Apache のリクエスト情報を EXTENSION_CONTROL_BLOCK に変換して HttpExtensionProc を呼び出します。 2. **CGI 実行**: ap_cgi_build_command フックが呼び出されると、レジストリまたはシバン行から適切な実行バイナリを特定し、Windows 形式のコマンドラインを構築します。

  7. Module: modules/cache主なフロー

    1. **判定**: cache_quick_handler がリクエストをインターセプトし、キャッシュ可能か判定。 2. **検索**: cache_select を通じて各プロバイダーを走査し、既存のキャッシュエントリを検索。 3. **提供/保存**: ヒット時は CACHE_OUT フィルタでレスポンスを構築し、ミス時は CACHE_SAVE フィルタでバックエンドからのレスポンスをストレージへ永続化。 4. **無効化**: PUT/POST/DELETE メソッド時に cache_invalidate_filter が関連キャッシュを削除。

  8. Module: modules/cluster主なフロー

    1. **送信**: mod_heartbeat が mod_watchdog を介して定期的にスコアボードを走査し、UDP マルチキャストで負荷情報を送信する。 2. **受信・集約**: mod_heartmonitor が mod_watchdog を介してソケットをポーリングし、パケットを受信・解析する。 3. **永続化**: mod_heartmonitor が収集した情報を slotmem（共有メモリ）またはファイルへ書き出し、ロードバランサーが参照可能な状態にする。
